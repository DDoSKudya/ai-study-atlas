# Соглашение об именовании

Цель: единые и предсказуемые имена файлов/директорий, чтобы структура базы оставалась стабильной при длительном развитии.

## 1) Общий стиль имен

Зафиксированный стиль slug: **lowercase latin + snake_case + digits**.

Разрешено:

- латиница в нижнем регистре: `a-z`;
- цифры: `0-9`;
- разделитель слов: `_`.

Запрещено:

- пробелы;
- кириллица;
- смешение `snake_case` и `kebab-case` в одном типе сущностей;
- специальные символы (кроме стандартных имени файла вроде точки перед `.md`).

## 2) Направления и треки

Структура:

- `directions/<direction_slug>/...`
- `directions/<direction_slug>/<track_slug>/...`

Примеры:

- `directions/architecture/`
- `directions/backend/python/`
- `directions/cs_fundamentals/db/`

Правило: каждое звено пути — короткое, предметное и читаемое без расшифровки в контексте соседних папок.

## 3) Файл плана трека

Для каждого трека обязательный единый файл:

- `mastery_plan.md`

Не использовать альтернативы вроде `plan.md`, `roadmap.md`, `global_plan.md` внутри трека.

## 4) Файлы контента

Допустимы два формата:

1. Один файл на часть:
   - `NN_topic_slug.md`
2. Папка на часть + входной файл:
   - `NN_topic_slug/index.md`

Где:

- `NN` — двухзначный порядковый номер (`01`, `02`, ...);
- `topic_slug` — snake_case slug темы.

Для вложенных файлов внутри папки части рекомендуется префикс с номером подблока:

- `01_03_name.md`, `01_04_name.md` и т.д.

## 5) Служебные файлы

Фиксированные имена:

- корень проекта: `README.md`, `INDEX.md`;
- документация правил: `docs/*.md` (имена в snake_case);
- промпты: `prompts/**/*.md`;
- инструкция по промптам: `prompts/HOW_TO_USE_PROMPTS.md` (сохраняется как исторически зафиксированное имя в upper snake style).

## 6) Практические правила консистентности

- Не переименовывать массово без необходимости: приоритет у стабильных ссылок.
- При создании новых файлов сверяться с уже используемым паттерном в конкретном треке.
- Если в треке выбран формат "папка + `index.md`", придерживаться его для новых частей этого же трека.
- Любое отклонение от правил фиксировать отдельной заметкой в `CHANGELOG.md`.
