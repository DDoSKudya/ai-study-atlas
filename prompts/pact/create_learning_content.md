# Create Learning Content

## Входные данные
- Тема: <указанная пользователем тема>
- Глобальный план: <полный путь к файлу глобального плана>
- Папка с учебными частями (pact): <полный путь к папке pact>
- Файл структуры оформления: <полный путь к README_STRUKTURA_IZUCHENIYA.md, если существует>

Твоя задача: создать новый учебный Markdown-файл по указанной теме и заполнить его максимально подробно, понятно и методически правильно.

Входные данные (из начала чата или из предыдущего сообщения пользователя):

Глобальный план: указанный пользователем путь к глобальному плану
Папка с готовыми частями: указанная папка (pact)
Файл со структурой оформления: указанный файл структуры
Тема: указанная тема
Что нужно сделать:

Сначала внимательно изучи:

глобальный план по указанному пути;
файл со структурой оформления;
несколько существующих файлов из указанной папки pact, чтобы сохранить единый стиль, глубину, формат подачи и схему именования.
На основе указанной темы определи:

к какой части, шагу и подтемам глобального плана она относится;
какой файл (или папку с index.md и подфайлами) нужно создать в указанной папке;
как он должен называться по образцу уже существующих файлов в этой папке.
Имя файла (или папки) определи самостоятельно, строго в стиле существующих в папке pact: та же схема нумерации, именования и slug-формата, чтобы новый файл естественно вписывался в систему.

Создай новый Markdown-файл в указанной папке и полностью заполни его как полноценную учебную часть, а не как черновик.

Пиши материал максимально подробно, но простым и ясным языком.

Сохраняй профессиональную терминологию, но каждый сложный термин обязательно объясняй человеческим языком.

Не сокращай материал ради краткости. Лучше подробно, последовательно и с повторением ключевых идей в разных формах.

Если в плане тема указана кратко, раскрой её глубоко: теория, внутренняя механика, практические сценарии, ограничения, типичные ошибки, диагностика, сравнения, best practices.

Не добавляй заглушки вида «TODO», «дописать позже», «при необходимости можно расширить».

Итог должен быть таким, чтобы файл можно было использовать и как учебник для последовательного изучения, и как справочник для повторения.

Обязательные требования к содержанию:

Материал должен строго соответствовать глобальному плану.
Нельзя уходить в сторону от выбранной темы.
Нужно покрыть все подтемы, которые относятся к ней по плану.
Если уместно для темы, показывай сравнения, альтернативы, варианты (например: для тем по БД — различия между СУБД; для тем по коду — стили кодирования; для других тем — релевантные аналоги).
Обязательно добавляй:
много теории;
понятные объяснения «простыми словами»;
аналогии и «картинку в голове»;
пошаговые разборы;
примеры (код, команды, конфигурации, SQL, диаграммы и т.д. — в зависимости от темы);
практические кейсы;
типичные ошибки;
граничные случаи;
последствия неправильного использования;
рекомендации по продакшн-практике, где применимо;
блоки самопроверки с ответами.
Требования к визуализации (графики и диаграммы):

В тех местах, где структура или процесс сложно представить только текстом (слои архитектуры, потоки данных, взаимодействие сервисов, жизненный цикл запроса и т.п.), обязательно добавляй визуальные схемы:
текстовые схемы или ASCII‑диаграммы;
либо mermaid‑диаграммы (flowchart, sequence, class/er и т.д.), если формат файла это поддерживает.
Диаграммы должны:
помогать «увидеть» связь между компонентами, модулями, слоями, сервисами, данными и потоками;
быть максимально понятными, без визуального шума: минимум лишних стрелок, чёткие подписи, логичная группировка;
дополнять текст (не дублировать его дословно), подчёркивая ключевые мысли.
Важные архитектурные фрагменты (например, «клиент → BFF → сервисы → БД», «границы сервиса и контракты», «слои приложения») желательно сопровождать отдельной диаграммой, чтобы у читателя формировалось устойчивое визуальное представление.
Требования к стилю:

Пиши на русском языке.
Стиль: учебный, ясный, дружелюбный, но профессиональный.
Объясняй так, как будто читатель хочет стать очень сильным специалистом, но не должен спотыкаться об излишне сухую подачу.
Сначала давай интуицию, потом точную формулировку, потом практический пример.
Не перегружай абзацы без необходимости: лучше хорошо структурированный длинный материал, чем сплошная стена текста.
Термины используй корректно и последовательно.
Когда термин важный, сначала дай формальное название, затем объясни простыми словами.
Если есть распространённая путаница между похожими понятиями, обязательно отдельно разъясни различия.
Структура файла должна соответствовать принятому шаблону в указанной папке pact и файлу структуры оформления:

Заголовок части.
Краткая цель части.
Блок «После этой части ты сможешь».
Что желательно знать заранее.
Краткая шпаргалка по терминам.
Маршрут изучения простыми словами.
Структура материала.
Короткое объяснение, как устроены разделы и как пользоваться оглавлением.
Оглавление по этапам изучения.
Основные разделы по теме.
В конце: справочник по части; частые сценарии; вопросы для самопроверки; типичные ошибки; резюме части.
Для каждого основного раздела по возможности используй внутренний шаблон:

Цель раздела;
В этом разделе главное;
Термины;
Теория и правила;
Пошагово;
Простыми словами;
Картинка в голове;
Как запомнить;
Примеры;
Практика / реальные сценарии;
Типичные ошибки;
Что будет если…;
Проверь себя;
Запомните.
Требования к примерам:

Примеры должны быть обучающими, с понятными сценариями из реальной практики.
В зависимости от темы: примеры кода, команд, конфигураций, SQL, диаграмм и т.д.
Для технических тем раскрывай: как это работает, зачем нужно, плюсы/минусы/компромиссы, где ломается, как диагностировать, как применять на практике.
Требования к самопроверке:

В каждом крупном разделе 2–3 вопроса для самопроверки.
Ответы оформляй через <details><summary>Ответ</summary>...</details>.
Вопросы должны проверять понимание, а не механическое запоминание.
Критерии качества:

Файл очень подробный; читается как глава учебника; цельная ментальная модель; понятно в сложных местах; нет сухого конспекта; каждая важная идея названа, объяснена, показана на примере, закреплена выводом.
Важно:

Не пиши коротко; не экономь на объяснениях; не пропускай промежуточные шаги.
Сохраняй единый стиль существующих файлов в указанной папке pact.
Сравни структуру и именование с уже созданными файлами в этой папке.
Финальный результат:

Самостоятельно определи правильное имя (и при необходимости структуру папки) по образцу существующих в pact.
Создай файл в указанной папке.
Контент завершённый, цельный, готовый к сохранению без дополнительной доработки.

Очень важно: материал объясняй предельно понятно, медленно, последовательно, с повторением ключевых идей в разных формах. Не пропускай логические шаги; каждую важную идею — интуиция → формулировка → пример → закрепление. Раскрывай «что это», «зачем», «как работает», «когда применять/не применять», «где ошибаются», «как запомнить». После каждого крупного блока проверяй: не коротко ли, не сухо ли; при необходимости расширяй, добавляй примеры и пояснения. Цель — чтобы даже читатель с низким порогом входа мог читать, понимать и усваивать материал.

При этом старайся максимально подробно и детально раскрывать даже самое маленькое уточнение и т.д.
