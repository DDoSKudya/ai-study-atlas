# Analyze And Improve Latest Learning File

## Входные данные
- GLOBAL_PLAN: <полный путь к файлу глобального плана, например: /path/to/training/topic/mastery_plan.md>

Обозначения (ниже в тексте используй только их, а не явные пути):
- ROOT_DIR  = каталог, в котором лежит GLOBAL_PLAN
- PARTS_DIR = ROOT_DIR/pact
- STRUCT_MD = ROOT_DIR/pact/README_STRUKTURA_IZUCHENIYA.md (если существует)


## Промпт
Проанализируй учебный Markdown-файл, который был создан или заполнен в текущем чате последним (по текущей теме), и не просто дай ответ в чате, а доведи его до состояния, в котором все относящиеся к теме пункты будут раскрыты полностью, глубоко, понятно и без очевидных пробелов.

Исходные данные:
- Глобальный план курса: GLOBAL_PLAN
- Корень курса: ROOT_DIR
- Папка с частями курса: PARTS_DIR
- Структурный эталон для частей (если есть): STRUCT_MD
- Целевой файл для анализа и доработки: последний учебный .md-файл, который был создан или отредактирован в текущем диалоге (название не спрашивай, опирайся на контекст чата).

Главная цель:
- проверить, насколько целевой файл реально покрывает свою тему в рамках плана GLOBAL_PLAN;
- найти все места, где тема раскрыта слабо, поверхностно, сухо, неполно или непонятно;
- автоматически доработать и усилить файл:
  - по содержанию (теория, примеры, практика);
  - по объяснению (простым языком);
  - по структуре (если нужно);
- повторять цикл «анализ → улучшение → повторная проверка» до тех пор, пока файл не станет максимально полезным и цельным для обучения.

Важные принципы:
- целевой файл нужно рассматривать не в вакууме, а в контексте плана из GLOBAL_PLAN и соседних файлов из PARTS_DIR;
- при доработке нельзя ломать общую структуру курса и его стиль;
- цель не переписать всё «как хочется», а довести файл до состояния:
  - «по этой теме теперь действительно можно учиться».

Шаги работы:

1. Определить место файла в плане.

1.1. Выясни, к какой части/разделу GLOBAL_PLAN относится этот файл.
- Пройди по GLOBAL_PLAN:
  - найди раздел(ы), тематика которых совпадает с содержимым файла;
  - зафиксируй:
    - номер шага/части;
    - список подтем (пунктов), которые этот файл должен покрывать.

1.2. Если файл явно привязан к части (например, через ссылку «Навигация по части …» или указания вида «§N»):
- используй эту привязку;
- сопоставь её с секциями GLOBAL_PLAN.

2. Собрать ожидания по содержанию.

2.1. Из GLOBAL_PLAN выпиши для связанной части:
- какие темы и подтемы должны быть раскрыты;
- какие ключевые понятия должны быть объяснены;
- какие сценарии/аспекты должны быть затронуты (если они явно указаны в плане).

2.2. Если STRUCT_MD описывает структуру разделов:
- посмотри, какие блоки ожидаются внутри разделов:
  - цель;
  - термины;
  - теория;
  - «пошагово»;
  - «простыми словами»;
  - «картинка в голове»;
  - «как запомнить»;
  - «проверь себя»;
  - «типичные ошибки»;
  - и т.п.;
- это твой чек-лист по форме.

3. Тщательно проанализировать сам файл.

Пройди целевой файл от начала до конца и оцени:

3.1. Покрытие тем (по плану):
- все ли пункты плана, относящиеся к этой части, вообще встречаются в файле;
- есть ли темы, которые:
  - только упомянуты;
  - раскрыты в 1–2 фразах без примеров и пояснений;
  - описаны так, что новичок не поймёт;
- есть ли темы, которые вообще отсутствуют, хотя GLOBAL_PLAN их требует.

3.2. Глубина и понятность:
- даётся ли интуитивное объяснение перед формальной теорией;
- есть ли пояснения простым языком;
- есть ли примеры (кодовые, текстовые, жизненные) для ключевых идей;
- есть ли переходы между разделами (не выглядит ли как набор несвязанных абзацев);
- можно ли по этому файлу:
  - понять, «что это»;
  - понять, «зачем это»;
  - понять, «как это применять».

3.3. Практическая часть:
- есть ли в файле реальные или хотя бы приближённые к реальности сценарии;
- показано ли, как применить материал из файла:
  - в коде / в настройках / в архитектуре — по контексту курса;
- есть ли разделы вида «типичные ошибки», «что будет, если…», если для темы это критично.

3.4. Самопроверка:
- есть ли вопросы по материалу (2–3 на крупный раздел);
- есть ли ответы под спойлерами (`<details><summary>Ответ</summary>...</details>`);
- вопросы проверяют понимание, а не только запоминание формулировок.

3.5. Структура и формат:
- есть ли у файла:
  - цель / вводная (что в этом файле и зачем);
  - логически сформулированные подзаголовки;
  - финальный блок «соберём всё вместе» (резюме, основные тезисы) — если уместно;
- не нарушает ли файл общую структуру, ожидаемую STRUCT_MD;
- нет ли признаков черновика:
  - TODO;
  - «дописать позже»;
  - обрывки мыслей.

4. Автоматически доработать файл по результатам анализа.

4.1. Темы из GLOBAL_PLAN, которые:
- отсутствуют — добавь как полноценные разделы/подразделы;
- присутствуют, но раскрыты слабо — расширь:
  - добавь теорию, объяснения, примеры и практику;
- присутствуют, но непонятны — перепиши объяснения более простым языком.

4.2. Добавить/усилить блоки структуры (если уместно и согласовано со STRUCT_MD):
- цель раздела / файла;
- краткая «в этом разделе главное»;
- список терминов с краткими определениями;
- блок «теория и правила»;
- блок «пошагово» (если тема процедурная);
- блок «простыми словами»;
- блок «картинка в голове» (метафора);
- блок «как запомнить»;
- блок «проверь себя» (2–3 вопроса);
- блок «типичные ошибки»;
- блок «что будет, если…»;
- блок «запомните» (сжатые выводы).

4.3. Примеры и практика:
- для каждой важной идеи добавь хотя бы один пример;
- привяжи пример к реальному сценарию использования;
- если есть код:
  - сделай его осмысленным, не искусственным;
  - добавь комментарии/пояснения (но без избыточного «комментарии ради комментариев»).

4.4. Самопроверка:
- добавь в конце крупных разделов 2–3 вопроса;
- ответы оформи под `<details><summary>Ответ</summary>...</details>`;
- убедись, что вопросы проверяют именно понимание сути.

4.5. Понятность:
- перепиши трудные места:
  - длинные, перегруженные абзацы — разбей и упростить формулировки;
  - жаргон — либо объясни, либо смягчи;
  - неочевидные шаги в рассуждениях — сделай более явными.

5. Повторная проверка после правок.

5.1. Снова сопоставь файл с GLOBAL_PLAN:
- теперь ли все темы и подтемы покрыты;
- есть ли ещё пункты плана, которые остались только «по названию» и не раскрыты по смыслу.

5.2. Снова оцени файл как учебный текст:
- можно ли по нему:
  - понять тему;
  - объяснить её другим;
  - применить на практике (в простых сценариях);
- остались ли места, которые:
  - путают;
  - перегружают;
  - очевидно недоработаны.

5.3. При необходимости сделай ещё один цикл:
- доработай и снова перепроверь;
- цель — приблизиться к состоянию, когда:
  - файл не оставляет грубых дыр;
  - понятно, как его читать и что он даёт.

## Формат итогового ответа (в чате)
1. Общий итог:
   - удалось ли довести файл до состояния «полноценная учебная часть по своей теме»;
   - если нет — какие ограничения остались (например: «нужны ещё большие примеры», «эту тему лучше выносить в отдельный файл»).

2. Что именно было усилено или добавлено:
   - перечисли группы изменений:
     - «добавлены недостающие подтемы»;
     - «переписаны сложные объяснения проще»;
     - «добавлены примеры»;
     - «добавлены/усилены блоки самопроверки»;
     - и т.д.

3. Какие слабые места были исправлены:
   - по пунктам:
     - «X был только упомянут — теперь есть отдельный раздел с примером»;
     - «Y был без примеров — теперь есть 2 практических сценария»;
     - «структура раздела Z была кривая — выровнял заголовки и порядок».

4. Остались ли ещё вещи, которые всё ещё требуют расширения:
   - честно укажи:
     - «эту тему лучше вынести в отдельный файл/часть»;
     - «нужен отдельный большой пример/кейс, который сейчас не поместился»;
     - «этот блок логичнее будет доработать вместе с соседним файлом».

5. Итоговая оценка этого файла по полноте раскрытия темы (по смыслу) в процентах:
   - от 1 до 100%;
   - с коротким обоснованием:
     - «80% — покрыты все темы, но мало реальных кейсов»;
     - «95% — всё есть, но можно улучшить формулировки» и т.п.

Очень важно:
- не отвечай формально;
- не ограничивайся оценкой «нормально/ненормально»;
- если видишь проблему — исправляй её в файле, а не просто комментируй;
- не проси пользователя дополнительно указать имя файла:
  - ориентируйся на тот .md-файл, который был создан/правился последним в текущем диалоге;
- если в текущем чате было создано/правилось несколько файлов и из контекста неясно, о каком речь, тогда:
  - спроси кратко: «Какой из этих файлов сейчас анализируем?» и только после этого продолжай.
