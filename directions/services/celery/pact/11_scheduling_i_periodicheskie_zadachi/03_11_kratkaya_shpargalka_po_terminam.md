## Краткая шпаргалка по терминам

| Термин | Суть простыми словами |
| --- | --- |
| **Celery Beat** | Процесс-планировщик, который по расписанию **кладёт задачи в очередь** (отправляет сообщения в broker). |
| **Worker** | Процесс, который **забирает** задачи из очереди и **выполняет** код задач. |
| **Centralized scheduler** | Один «источник правды» о том, *когда* должны стартовать периодические задачи (обычно один beat или координируемый планировщик). |
| **`beat_schedule`** | Словарь/конфигурация расписания в коде или настройках приложения Celery. |
| **`PeriodicTask` (концепт)** | Запись «какая задача, как часто, с какими аргументами» — в БД (часто через django-celery-beat) или в памяти beat. |
| **`crontab`** | Расписание в стиле cron: минута/час/день месяца/месяц/день недели + опции вроде часового пояса. |
| **`schedule` (interval)** | Периодичность «каждые N секунд» (интервал), с возможностью первого старта `now` или отложенного. |
| **`solar`** | Расписание относительно **восхода/заката** (зависит от географии и времени года). |
| **Overlap** | Новый запуск начался, пока **предыдущий ещё не закончился**. |
| **Misfire (в бытовом смысле для Celery)** | Beat «проспал» момент или инфраструктура была недоступна: нужна политика **догнать / пропустить / сжать окно**. |
| **Single-run гарантия** | Желание, чтобы в данный момент времени **работал не более одного** экземпляра логического периодического процесса (часто реализуется locks). |
| **Advisory lock** | Блокировка на уровне СУБД «мягко», без жёсткой привязки к строке таблицы (PostgreSQL: `pg_advisory_lock`). |
| **Lease** | Аренда блокировки с **временем жизни**: если worker умер, lock истечёт и система сможет продолжить. |
| **Fencing token** | Возрастающий «номер выпуска» lock: старый исполнитель не должен завершать побочные эффекты после потери аренды. |
| **Redis lock (NX+EX)** | `SET key value NX EX ttl` — попытка взять ключ атомарно с автоистечением; снятие — только владельцу (часто через Lua). |
| **Catch-up** | После простоя **догнать** пропущенные окна (часто с **лимитом** и порциями). |
| **Skip** | **Не** воспроизводить прошлое; актуализировать «с текущего момента» или через watermark. |
| **Watermark** | Курсор «до куда обработано»; основа batch window и устойчивого catch-up. |
| **DatabaseScheduler** | Реализация beat, читающая расписание из БД (типично django-celery-beat). |

#### Проверь себя: шпаргалка

1. Чем **`beat_schedule`** концептуально отличается от строки **`PeriodicTask`** в django-celery-beat?

<details><summary>Ответ</summary>

`beat_schedule` — расписание в **коде/конфиге** приложения Celery, меняется с **деплоем** и ревью в git. `PeriodicTask` — запись в **БД**, которую читает `DatabaseScheduler`; её можно менять **операционно** через админку/API, с отдельными рисками доступа и расхождения окружений.

</details>

2. Почему **fencing token** упоминают рядом с **lease TTL**, хотя это разные абстракции?

<details><summary>Ответ</summary>

TTL lease ограничивает время владения lock; при истечении **другой** воркер может начать работу, пока **старый** ещё пишет во внешние системы. **Fencing token** — способ заставить внешний мир **отвергать** устаревшие операции старого владельца после потери lease.

</details>

3. В какой паре терминов легко ошибиться: оба про «не делать дважды», но механизмы разные?

<details><summary>Ответ</summary>

**Single-run / overlap prevention** (lock, lease) и **`expires`** у сообщения: первое ограничивает **параллельное или повторное исполнение логики**, второе может **отбросить просроченное сообщение** в очереди без гарантии mutex. Их смешивают, ожидая от `expires` защиты от overlap.

</details>
