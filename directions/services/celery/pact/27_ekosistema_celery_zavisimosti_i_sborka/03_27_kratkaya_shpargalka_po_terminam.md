## Краткая шпаргалка по терминам

| Термин | Что это значит простыми словами |
|---|---|
| **Celery core** | Основной фреймворк задач: registry, worker runtime, конфигурация. |
| **Kombu** | Транспортный уровень сообщений (подключения, producer/consumer, abstraction над брокерами). |
| **Billiard** | Процессы и пулы исполнения (fork/exec-поведение для worker-процессов). |
| **Vine** | Вспомогательные примитивы promise/callback, используемые в Celery-экосистеме. |
| **py-amqp / amqp** | Низкоуровневый AMQP-клиент для работы через RabbitMQ/AMQP-транспорт. |
| **Optional extras** | Наборы зависимостей, подключаемые через `celery[...]`. |
| **Pinning** | Фиксация версий зависимостей по выбранной политике. |
| **Reproducible build** | Повторяемая сборка, дающая одинаковый результат при одинаковом вводе. |
| **SBOM** | Список компонентов ПО внутри артефакта (образа/бинаря) для аудита. |
| **DX (Developer Experience)** | Удобство и предсказуемость работы разработчика с проектом. |

#### Проверь себя: термины

1. Чем `Kombu` концептуально отличается от `Celery core`?

<details><summary>Ответ</summary>

Celery core управляет задачами и исполнением на уровне фреймворка, а Kombu — это транспортная "логистика" сообщений между приложением и брокером.

</details>

2. Почему SBOM относится к эксплуатации Celery, а не только к безопасности?

<details><summary>Ответ</summary>

SBOM помогает быстро понять состав зависимостей при инцидентах, апгрейдах и несовместимостях, а не только при security-аудите.

</details>
