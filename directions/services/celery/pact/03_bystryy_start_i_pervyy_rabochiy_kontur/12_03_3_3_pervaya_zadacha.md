## 3.3. Первая задача

### Цель раздела

Ты создашь первую задачу Celery, научишься понимать разницу между локальным вызовом функции и публикацией задачи, и отправишь её через `delay()` и `apply_async()`.

### В этом разделе главное

- `@app.task` — это не “асинхронность”, а объявление распределённой единицы работы (и контракт сериализации аргументов).
- `delay()` и `apply_async()` — разные уровни управления публикацией.
- Аргументы должны быть сериализуемыми (простые типы, словари/списки), иначе задача не сможет корректно доехать до worker.

### Термины

- **сериализуемые аргументы** — такие, которые Celery может упаковать в формат сообщения (обычно JSON) и затем распаковать обратно.
- **`task_id`** — идентификатор конкретного запуска задачи.
- **signature** — объект Celery, который описывает вызов задачи (имя + аргументы/kwargs + options) так, чтобы его можно было сериализовать в сообщение и отправить в broker. В твоих примерах это “под капотом” собирают `delay()`/`apply_async()`.

### Теория и правила

Локальный вызов функции:

- выполняется сразу в текущем процессе;
- не создаёт сообщения;
- не требует broker.

Публикация задачи (например через `delay()`):

- создаёт задачу как сообщение;
- публикует её в broker;
- возвращает объект для отслеживания (`AsyncResult` или его обёртка).

Это фундаментальная разница “что ты запускаешь”: **не функцию, а распределённую публикацию**.

#### Что реально уезжает в broker: payload, сериализация и signature

Когда ты вызываешь задачу через `delay()` или `apply_async()`, Celery делает примерно следующее:

1) берёт задачу по её имени (`tasks.add` в примере);
2) упаковывает аргументы/kwargs в формат сообщения (payload);
3) добавляет служебные метаданные (task id и служебные заголовки);
4) отправляет всё это в broker.

Поэтому **аргументы должны быть сериализуемыми**. В типичном учебном сетапе Celery использует JSON: числа, строки, булевы, списки/словари, `None` обычно проходят. А вот “живые” объекты Python (соединения БД, файлы, дескрипторы, лямбды, экземпляры произвольных классов, coroutine/async primitives) сериализоваться не должны.

Пример того, что часто ломается при сериализации:

```python
# условная несерилизуемая штука (как минимум для JSON)
class Obj:
    pass

add.delay(Obj(), 1)  # ожидай ошибку сериализации на стороне producer (или раннее падение)
```

`signature` — это как “машинка для упаковки вызова”: структурированное описание того, **что именно** будет вызвано и **с какими аргументами**, чтобы это описание можно было сериализовать и доставить worker-у.

Картинка в голове (упрощённый поток):

```text
producer code
    |
    | delay() / apply_async()
    v
[signature] + args/kwargs
    |
    | serialize (json)
    v
message payload + headers (task_id, route)
    |
    v
broker / queue
    |
    v
worker executes Python function
```

### Пошагово

#### Шаг 1. Декларируй задачу через `@app.task`

```python
# tasks.py
from celery_app import app

@app.task(name="tasks.add")
def add(x, y):
    return x + y
```

##### Именованные и безымянные задачи

Декоратор `@app.task` можно использовать в двух вариантах:

- **именованная задача** — ты явно задаёшь `name="..."`:

```python
@app.task(name="tasks.add")
def add(x, y):
    return x + y
```

Такое имя становится стабильным адресом задачи: по нему удобнее маршрутизировать (`task_routes`) и проще понимать, что именно опубликовано/принято (в логах и при диагностике).

- **безымянная задача** — ты не передаёшь `name`:

```python
@app.task
def add(x, y):
    return x + y
```

Тогда Celery сгенерирует имя автоматически (обычно из модуля и функции). Для учебного старта это нормально, но в более “живых” системах явное `name` помогает снизить сюрпризы при рефакторинге (переезд функции между модулями влияет на auto-generated имя).

Правило для старта: если ты хочешь меньше загадок в production-подобных настройках — выбирай **именованные задачи** и согласуй их с routing/очередями.

#### Шаг 2. Отправь через `delay()`

```python
from tasks import add

result = add.delay(2, 3)
print(result.id)  # task_id
```

`delay()` — это shorthand для `apply_async()` с “публикуй прямо сейчас” и без лишних параметров.

#### Шаг 3. Отправь через `apply_async()` (разница в смыслe)

```python
result = add.apply_async(args=[2, 3])
print(result.id)
```

В `apply_async()` ты чаще управляешь:

- countdown/ETA (когда публиковать);
- routing/options (в какую очередь/обменник отправлять);
- headers (для бизнес-корреляции и трассировки).

В этой части мы делаем акцент на старте: “полная публикация” vs “быстрый shorthand”.

### Простыми словами

`delay()` — это как поставить задачу “в очередь”. Ты не ждёшь выполнение, ты получаешь номер квитанции (`task_id`), по которой дальше читаешь статус.

### Картинка в голове

Представь, что задача — это “посылка с накладной”. Локальный вызов — “открыть посылку самому прямо сейчас”. Celery — “передать посылку в логистику”, а затем проверить трекинг.

### Как запомнить

Если ты передумал: не пытайся искать “где функция выполняется” — ищи “где сообщение ушло и кто забрал”.

### Примеры

Мини-скрипт отправки:

```python
# main_send.py
from tasks import add

result = add.delay(10, 20)
print("task_id:", result.id)
print("initial state:", result.state)
```

### Практика / реальные сценарии

- Сложение `add` — учебная задача. Но она моделирует главный шаблон: **публикация + последующее чтение статуса**.
- В production чаще всего задачи вызываются из request/response контура, и важно отделять “ответ пользователю” от «фоновая побочка».

### Типичные ошибки

- Передавать в задачу сложные объекты: ORM instance, файлы, соединения, корутину и т.п. Они могут не сериализоваться.
- Часто ошибка проявляется как “невозможно сериализовать в JSON” (TypeError / kombu serialization error): это означает, что сообщение не доехало до broker/worker, потому что упаковка payload не прошла.
- Путать `delay()` с “моментальным выполнением”. `delay()` публикует, а исполнение происходит позже в worker.
- Неправильные ожидания по `task_id`: “пока вижу `PENDING` значит задача точно в очереди” — это не всегда верно (3.5).

### Что будет если…

… аргументы не сериализуются?

- Сообщение не сможет корректно быть опубликовано/восстановлено. На старте это будет видно как ошибки сериализации или как странное отсутствие эффекта.

… ты вызываешь задачу локально (без delay)?

- Это будет обычная функция, и ты потеряешь именно то свойство, ради которого Celery используют: разделение времени и процессов.

### Проверь себя

1. В чём смысл `task_id`, если ты “и так увидел результат печатью”?

<details><summary>Ответ</summary>

`task_id` нужен для трекинга распределённого исполнения. Если ты локально вызвал функцию, `task_id` может вообще не быть нужен — потому что выполнение уже произошло синхронно.

</details>

2. Почему Celery требует сериализуемости аргументов?

<details><summary>Ответ</summary>

Потому что Celery отправляет сообщение через broker. Это означает, что аргументы нужно упаковать в payload сообщения и затем распаковать на другой стороне worker’а.

</details>

3. Когда разумно выбирать `apply_async()`, а не `delay()`?

<details><summary>Ответ</summary>

Когда нужно явно управлять options публикации: countdown/ETA, routing в конкретную очередь, заголовки, expiration и другие параметры. `delay()` подходит, когда публикация “прямо сейчас” и без дополнительных требований.

</details>

4. Чем локальный вызов функции (без `delay`) принципиально отличается от публикации задачи?
<details><summary>Ответ</summary>
Локальный вызов — это выполнение “прямо сейчас” в текущем процессе: нет сообщения в broker, нет task_id как идентификатора распределённого исполнения и нет стека “producer → broker → worker”. Поэтому ты теряешь ключевое свойство Celery: отделение request/response от фоновой работы.
</details>

5. Что такое `signature` в практическом смысле и почему это важно для message flow?
<details><summary>Ответ</summary>
`signature` — это структурированное описание вызова задачи: имя + аргументы/kwargs + options. Оно нужно затем, чтобы Celery мог сериализовать это описание в сообщение (payload/headers) и доставить worker-у, который потом выполнит именно ту функцию с нужными параметрами. Без корректной “упаковки” сообщение не будет интерпретировано ожидаемо.
</details>

6. Почему “сериализуемость аргументов” — это часть доставки, а не просто “требование библиотечки”?
<details><summary>Ответ</summary>
Потому что Celery отправляет аргументы через broker как часть message payload. Если аргументы не сериализуются, сообщение не может быть опубликовано корректно (или распаковано worker-ом), значит delivery ломается раньше, чем начнётся выполнение бизнес-кода. Это типичный источник kombu/json serialization errors.
</details>

7. Как именованные задачи (`name="..."`) помогают на практике по сравнению с безымянными?
<details><summary>Ответ</summary>
Именованная задача даёт стабильный адрес задачи: её проще маршрутизировать через `task_routes`, проще читать в логах и проще сопоставлять “какую задачу мы ожидали” с “какую реально опубликовали/получили”. При рефакторинге безымянные задачи могут менять auto-generated имя.
</details>

8. В каком случае `delay()` может оказаться недостаточной абстракцией для контролируемой доставки?
<details><summary>Ответ</summary>
Когда тебе нужно управлять параметрами публикации: например, отложить старт (`countdown`/`ETA`), выбрать конкретную очередь/routing, добавить `headers` для корреляции, ограничить expiration или задать другие options. Тогда правильнее использовать `apply_async()`.
</details>

9. Что будет, если ты используешь `apply_async(queue="...")`, но worker слушает другую очередь?
<details><summary>Ответ</summary>
Сообщение будет опубликовано в указанную очередь, но consumer (worker) её не обслуживает. Снаружи это выглядит как “задача опубликована, но не выполняется”: backend может долго показывать `PENDING`, а в логах worker не будет evidence получения этого `task_id`.
</details>

10. Если сообщение не дошло до worker из‑за сериализации, где ты чаще увидишь причину?
<details><summary>Ответ</summary>
Обычно на стороне producer при публикации: Celery пытается упаковать payload, и если сериализация аргументов не проходит, возникает ошибка публикации. Иногда симптом проявляется чуть позже, но логика остаётся той же: без корректного payload delivery невозможна.
</details>

11. Почему важно понимать связь “task_id → AsyncResult” даже если ты пока просто печатаешь результат?
<details><summary>Ответ</summary>
Потому что в реальном приложении ты почти никогда не можешь получить результат “из того же места, где вызвал”. `task_id` — это ключ к дальнейшему трекингу: по нему ты спрашиваешь backend через `AsyncResult`. Если не понимать этот мост, легко перепутать “мы запустили” и “мы наблюдаем статус”.
</details>

12. Какую практическую диагностику ты сделаешь, если publish прошёл, а выполнение не видно?
<details><summary>Ответ</summary>
Сначала проверь: worker зарегистрировал задачу с тем именем, которое ты публикуешь (именование/`name` и module import). Затем проверь маршрутизацию: что worker слушает нужную очередь (и при необходимости `-Q`) и что routing правилен. Если выполнение всё равно не видно, проверь доставку broker/queue и только потом возвращайся к business-коду.
</details>

### Запомните

Task в Celery — это контракт сообщения. Ты управляешь публикацией и отслеживанием, а код задачи исполняет worker позже.
