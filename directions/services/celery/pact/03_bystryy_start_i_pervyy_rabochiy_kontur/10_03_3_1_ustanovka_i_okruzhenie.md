## 3.1. Установка и окружение

### Цель раздела

Ты научишься поставить Celery “правильно для старта”: изолировать окружение, выбрать нужные extras под выбранный broker, подготовить локальный broker (Redis/RabbitMQ) и завести базовые переменные окружения для broker/result backend.

### В этом разделе главное

- Celery зависит не только от `celery`, но и от “подкапотного” messaging layer + transport (например, Redis/RabbitMQ).
- На старте важнее всего воспроизводимость: одинаковые версии + изоляция окружения.
- Переменные окружения (broker URL, backend URL) позволяют избежать “магии” и ускоряют диагностику.

### Термины

- **extras** — дополнительные зависимости, которые позволяют подключить конкретный broker/транспорт без ручного гадания.
- **broker URL** — строка, по которой Celery подключится к очередям.
- **result backend URL** — строка, где Celery будет хранить статусы/результаты.

### Теория и правила

Celery — это “оркестратор” фоновых задач вокруг брокера. Но брокер и backend — это отдельные компоненты. Из-за этого почти любые проблемы на старте делятся на:

1) “Celery не может подключиться к брокеру” (network/DNS/порты/URL).

2) “Celery подключился, но задачи не регистрируются/не маршрутизируются” (неверный `-A`, PYTHONPATH, vhost/queue, serializer).

3) “Задача исполнилась, но ты не видишь статусы/результат” (backend выключен/URL неверный/результаты не игнорируются).

В этом разделе мы создадим базу, чтобы эти категории можно было быстро проверить.

### Пошагово

#### Шаг 1. Создай venv и установи Celery

Выбери Python (например, 3.10+). Внутри проекта:

```bash
python -m venv .venv
source .venv/bin/activate
python --version
pip install --upgrade pip
pip install celery
python -c "import celery; print(celery.__version__)"
pip freeze > requirements.txt
```

Дальше (важно) добавь extras под брокер. Типовые варианты:

```bash
# для Redis чаще всего не требует отдельного пакета extras,
# но некоторые окружения/сборки любят явное перечисление.
pip install "celery[redis]"

# для RabbitMQ:
pip install "celery[rabbitmq]"
```

Если `pip install "celery[redis]"` недоступен в твоём окружении (зависимости зависят от версии), можно начать с `pip install celery` и затем поставить транспорт-пакет, который требует Celery/RQ. На практике проще ориентироваться по документации твоей версии celery и ошибкам импорта.

#### Шаг 2. Поднимем локальный broker

Для обучения локально удобно:

- Redis — быстрый старт.
- RabbitMQ — более “классический” AMQP.

В docker-compose это часто выглядит так (пример для Redis):

```yaml
# docker-compose.yml (фрагмент)
services:
  redis:
    image: redis:7
    ports:
      - "6379:6379"
    command: ["redis-server", "--appendonly", "yes"]
```

Пример для RabbitMQ:

```yaml
# docker-compose.yml (фрагмент)
services:
  rabbitmq:
    image: rabbitmq:3-management
    ports:
      - "5672:5672"   # AMQP
      - "15672:15672" # UI
```

Запуск:

```bash
docker compose up -d
```

Это помогает сделать обучение воспроизводимым: ты фиксируешь версию и поведение broker’а (Redis/RabbitMQ) через контейнеры и меньше страдаешь от “на моей машине работает по-другому”. Для дальнейшей диагностики это важно, потому что большинство ранних проблем — именно про connectivity и URL к брокеру.

#### Шаг 3. Заведи переменные окружения

Выбери, что ты хочешь хранить как result backend. На старте можно использовать тот же broker (что часто удобно), но в проде лучше рассматривать отдельно (этот разговор будет глубже в следующих частях).

Пример для Redis:

```bash
export CELERY_BROKER_URL="redis://localhost:6379/0"
export CELERY_RESULT_BACKEND="redis://localhost:6379/1"
export CELERY_QUEUE_NAME="default"
```

Для RabbitMQ (пример, логин/пароль зависят от образа):

```bash
export CELERY_BROKER_URL="amqp://guest:guest@localhost:5672//"
export CELERY_RESULT_BACKEND="rpc://"
export CELERY_QUEUE_NAME="celery"
```

Важно: в RabbitMQ “/” — это vhost. В дальнейшем это будет критично для диагностики matching (часть 3.6).

#### См. 3.2: auto-discovery задач (как масштабировать код)

Кратко: идея будет разобрана подробно в разделе `3.2`. В минимальном проекте достаточно “простого импорта” модуля с задачами (`tasks.py`). Но когда кодов становится много (несколько пакетов, доменные модули), удобно включить **auto-discovery задач**: Celery сам находит модули, где объявлены задачи, и тем самым запускает регистрацию через декораторы `@app.task`.

Смысл механизма:

1. Ты говоришь Celery app, **в каких пакетах искать задачи**.
2. При старте worker Celery **импортирует** эти пакеты/модули.
3. Декораторы `@app.task` выполняются, и задачи регистрируются в app.

Схематично:

```text
# celery_app.py (фрагмент)
app.autodiscover_tasks(
    packages=["myproject"],  # поменяй на корневой пакет твоего проекта
)
```

Главное: `auto-discovery` в Celery — это по сути обычный `import` пакетов при старте worker. Поэтому если задачи “не находятся”, дальше диагностика почти всегда сводится к `-A` и `PYTHONPATH`.

Важное правило диагностики: auto-discovery — это не “магия”, а механизм import. Если задачи не находятся, обычно виноваты:

- неправильный `-A` (worker импортирует не тот Celery app);
- неправильные пакеты для поиска;
- проблемы с PYTHONPATH (не тот working directory / окружение);
- ошибки импорта внутри модулей задач (тогда декораторы не успевают зарегистрировать задачи).

#### См. 3.2: разнесение settings по окружениям

Кратко: подробный разбор — в разделе `3.2`. Celery проще и безопаснее обучать, когда ты держишь broker/backend/очереди как конфиг, а не как “вшитые константы”.

Практичный подход:

- для dev/test/prod использовать разные broker URL / result backend URL / (часто) разные очереди;
- переключение делать через env vars или через отдельные config-модули.

Пример на `CELERY_ENV`:

```python
# celery_app.py (фрагмент)
import os

env = os.getenv("CELERY_ENV", "development")

if env == "development":
    broker_url = os.getenv("CELERY_BROKER_URL", "redis://localhost:6379/0")
    backend_url = os.getenv("CELERY_RESULT_BACKEND", "redis://localhost:6379/1")
elif env == "test":
    broker_url = os.getenv("CELERY_BROKER_URL", "redis://localhost:6379/10")
    backend_url = os.getenv("CELERY_RESULT_BACKEND", "redis://localhost:6379/11")
else:  # production
    broker_url = os.getenv("CELERY_BROKER_URL")
    backend_url = os.getenv("CELERY_RESULT_BACKEND")
```

Почему это важно уже на старте: если ты поменял env vars на стороне producer, но не перезапустил worker, то “задача вроде отправилась, но статус не обновляется”.

### Простыми словами

Установка Celery — это не “поставить пакет”. Это “собрать понятную пару”: **как Celery подключается к очередям (broker URL)** и **где он хранит статусы/результаты (result backend)**. Если эти строки неверные — worker и задачи могут существовать, но вы будете видеть тишину или странные статусы.

### Картинка в голове

Представь Celery как почтовый сервис:

- broker URL — адрес почтового отделения, куда кладут письма (сообщения);
- result backend URL — где хранится отслеживание отправления (статусы/результаты);
- worker — курьер, который забирает письма и выполняет работу.

### Как запомнить

Всегда держи в голове правило: “**на старте сначала проверяем соединение и URL**”.

### Примеры

Мини-команда проверки окружения (чтобы видеть версии):

```bash
python -c "import sys, celery; print('python', sys.version.split()[0]); print('celery', celery.__version__)"
```

### Практика / реальные сценарии

- Если у тебя в команде есть dev/stage/prod, одинаковые переменные окружения (через env vars) помогают избежать ситуации “на локали работает, а в тесте нет” из-за URL/очередей/vhost.

### Типичные ошибки

- Поставить `celery`, но не установить нужный transport/extra — получаешь ImportError на старте worker’а.
- Ошибиться в broker URL (например, порт, vhost, scheme `amqp` vs `redis`).
- Включить result backend как строку, но не поднять backend-сервис (тогда `AsyncResult` будет “молчать” или показывать `PENDING`).

### Что будет если…

… ты указал broker URL, но broker недоступен?

- Producer сможет “опубликовать” только до момента ошибки соединения; задачи могут не доехать до очереди, и ты будешь диагностировать “задача не дошла”.

… backend URL неверный?

- Задача может исполниться, но статус/результат вы не увидите корректно. Это самый частый источник ложного ощущения “ничего не случилось”.

### Проверь себя

1. Почему на старте важно хранить broker URL в переменных окружения, а не в коде?

<details><summary>Ответ</summary>

Потому что это ускоряет диагностику и делает конфигурацию воспроизводимой между окружениями. При ошибках ты быстро меняешь URL без перекомпиляции/перезапуска лишней логики и понимаешь, где расходится среда.

</details>

2. Чем отличается “не дошло до очереди” от “дошло, но статус не виден”?

<details><summary>Ответ</summary>

“Не дошло до очереди” означает проблему в producer->broker или маршрутизации (URL/очередь/vhost/serializer). “Дошло, но статус не виден” часто означает проблему в result backend (URL неверный/backend выключен) или в механике записи статусов.

</details>

3. Какой главный признак того, что проблема может быть в backend, а не в выполнении задачи?

<details><summary>Ответ</summary>

Если worker логирует выполнение (видно, что задача стартовала/завершилась), но через `AsyncResult` ты постоянно видишь `PENDING`/нет результата — это сильный сигнал: backend настроен не так или отключён/не доступен.

</details>

4. Как проявится ситуация “ты не поставил нужный transport/extra для брокера”?
<details><summary>Ответ</summary>
Обычно это видно сразу при старте worker (и иногда при импорте app): будут ошибки наподобие ImportError/ошибки подключения транспортного слоя (transport) или невозможность инициализировать соответствующий драйвер сообщений. До фактической доставки дело часто не доходит, потому что worker не может корректно подготовить соединение/consumer.
</details>

5. Почему в учебных условиях лучше поднимать broker через Docker (docker-compose), а не полагаться на “локальные пакеты”?
<details><summary>Ответ</summary>
Потому что это делает обучение воспроизводимым: ты фиксируешь версию и базовое поведение брокера (Redis/RabbitMQ), а значит ранние сбои почти всегда сводятся к твоей конфигурации и к коду, а не к отличиям среды “на разных машинах”. Это резко снижает стоимость диагностики.
</details>

6. В чём практическая разница между `CELERY_BROKER_URL` и `CELERY_RESULT_BACKEND`?
<details><summary>Ответ</summary>
`CELERY_BROKER_URL` отвечает за доставку сообщений: куда producer публикует задачу и откуда worker её забирает. `CELERY_RESULT_BACKEND` отвечает за хранение статусов/результата по `task_id`. Задача может быть успешно исполнена даже при проблемах с backend, но тогда ты не увидишь корректный трекинг через `AsyncResult`.
</details>

7. Почему важно понимать “на что именно влияет vhost в RabbitMQ”?
<details><summary>Ответ</summary>
Потому что vhost — это часть адреса broker’а в AMQP. Если producer и worker используют разные vhost, они оказываются в разных “пространствах” очередей: сообщения могут публиковаться, но worker никогда их не увидит, и будет эффект “тишины” без ошибок выполнения задачи.
</details>

8. Если ты поменял env vars в producer, но не перезапустил worker — что ты ожидаешь увидеть?
<details><summary>Ответ</summary>
Worker всё продолжит работать со старой конфигурацией (старым `broker/result backend/очередями`). Поэтому может появиться несоответствие: сообщение публикуется в одно место, а читается из другого, либо backend обновляется “в другую таблицу”. В статитсах это часто проявляется как “долгий PENDING” или странная разница между логами worker и тем, что ты видишь в `AsyncResult`.
</details>

9. Если backend URL неверный, но задача по логам worker исполняется — как это обычно выглядит?
<details><summary>Ответ</summary>
Ты почти наверняка увидишь успешное выполнение в worker logs (или хотя бы переходы выполнения), но `AsyncResult` будет не содержать ожидаемого результата/финального состояния. Типичный симптом — `PENDING` дольше нормы или отсутствие корректного результата.
</details>

10. Как использовать формулировку “на старте сначала проверяем URL” так, чтобы она стала действием, а не лозунгом?
<details><summary>Ответ</summary>
Нужно сделать два быстрых шага: (1) убедиться, что worker подключается к broker по тому же URL (и именно с тем transport-драйвером, который ты установил), и (2) проверить, что по `task_id` ты можешь читать статусы/результат в backend (если он включён). Лишь затем переходить к сомнениям в логике задачи.
</details>

11. Что лучше считать “побочным эффектом” на старте, а что — “центральной проблемой”?
<details><summary>Ответ</summary>
Побочный эффект — это то, что задача ещё не видна или статусы не меняются из-за неправильного окружения. Центральная проблема — это неверный broker/backend URL, невозможность подключения или несоответствие конфигурации producer и worker. Из-за этого процесс “уточняется по границам”, а не по догадкам.
</details>

12. Почему `PENDING` иногда “обманчивый”, даже когда задача уже близка к выполнению?
<details><summary>Ответ</summary>
Потому что `PENDING` означает: backend пока не показал следующую информацию по `task_id`. Это может быть как нормальная задержка доставки, так и проблема в backend-обновлениях/доступности backend. Поэтому нужен “двухканальный” взгляд: worker logs + `AsyncResult.state`.
</details>

### Запомните

Собери “минимальный контур” из трёх вещей: broker доступен, Celery знает broker URL, worker запущен и видит задачи, и (если ты хочешь статусы) backend доступен.
