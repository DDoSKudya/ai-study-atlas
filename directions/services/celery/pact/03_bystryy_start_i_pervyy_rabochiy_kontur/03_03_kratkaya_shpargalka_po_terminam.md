## Краткая шпаргалка по терминам

| Понятие | Суть простыми словами |
|---|---|
| **Celery app** | Объект, который объединяет настройки, регистрацию задач и связь с broker/result backend. |
| **Broker URL** | Строка подключения к брокеру сообщений (например, `redis://...` или AMQP URL). |
| **Result backend** | Хранилище статусов/результатов задач, которое позволяет читать `AsyncResult`. |
| **Worker** | Процесс (или несколько процессов), который забирает задачи из очередей и исполняет их. |
| **`-A` (worker)** | Флаг worker’а, который говорит: “откуда импортировать объект Celery app и зарегистрированные задачи”. |
| **`delay()`** | Быстрый способ “опубликовать задачу прямо сейчас” без ручных параметров публикации. |
| **`apply_async()`** | “Полная версия” публикации: можно явно указать routing/options, countdown/ETA, заголовки и т.д. |
| **Task id** | Уникальный идентификатор конкретного запуска задачи (то, по чему ты читаешь статус). |
| **`AsyncResult`** | Объект-клиент, который по `task_id` запрашивает состояние/результат из result backend (если он используется). |
| **`PENDING`** | Состояние, которое часто означает “статус неизвестен/ещё не обновлён в backend”. |
| **`STARTED`** | На практике — “worker начал обработку” (обычно требует включить `task_track_started` и при доступности result backend). |
| **`RETRY`** | Задача ошиблась, Celery планирует повтор (а не “успешно остановилась навсегда”). |
| **`FAILURE`** | Задача завершилась ошибкой (после финального провала/лимитов). |
| **`SUCCESS`** | Задача завершилась успешно и (если включено) результат записан. |
| **`REVOKED`** | Задача отменена (запретили выполнение или она отозвана до старта/в момент обработки). |
