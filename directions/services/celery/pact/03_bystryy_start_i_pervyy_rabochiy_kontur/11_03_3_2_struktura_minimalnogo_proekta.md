## 3.2. Структура минимального проекта

### Цель раздела

Ты научишься организовать минимальный проект Celery так, чтобы:

- Celery app был единственным источником правды для конфигурации;
- задачи были в отдельном модуле;
- worker мог импортировать app через `-A`;
- не возникали циклические зависимости между модулями.

### В этом разделе главное

- Worker с флагом `-A` должен импортировать нужный объект Celery app (и зарегистрированные задачи).
- Циклические импорты появляются, когда `tasks.py` импортирует app слишком “наивно” и обратно.
- Разнесение settings на dev/test/prod — практичность: оно позволяет менять URL/очереди без переписывания логики.

### Термины

- **namespace конфигурации** — соглашение, в каком модуле/объекте держатся настройки и как они подключаются к app.
- **auto-discovery задач** — механизм, который позволяет Celery самому найти задачи по пакетам (удобен для проектов, но на старте можно начинать проще).
- **PYTHONPATH** — переменная/механизм, от которого зависит, какие модули импортируются.

### Теория и правила

Самая частая причина “worker запустился, но задачи не видит” — несоответствие Python-import’ов. В Celery worker импортируется как обычный Python-импорт:

- флаг `-A` определяет, **что импортировать**;
- далее Celery ищет объект app и зарегистрированные задачи.

Если `-A` указывает не туда, worker будет работать, но “пусто”.

#### Почему так важны импорты (и как избежать циклических зависимостей)

Celery worker стартует как обычный Python-процесс: сначала он импортирует твоё `Celery`-приложение (через `-A`), а дальше — через импорты модулей с задачами и декораторы — регистрирует задачи.

Циклические зависимости обычно появляются, когда ты пытаешься “зарегистрировать задачи” из модуля с app:

- `celery_app.py` импортирует `tasks.py`;
- а `tasks.py` импортирует `app` обратно.

Правильный паттерн для старта (и самый устойчивый при росте проекта):

1) `celery_app.py` содержит только объект `app` и конфиг.
2) `tasks.py` импортирует `app` и объявляет `@app.task(...)`.
3) `main.py` импортирует задачи/функции (чтобы отправлять сообщения), но не участвует в сборке app.

Плохая схема (намекает на цикл):

```python
# celery_app.py (плохо)
from tasks import some_task  # celery_app начинает тянуть tasks

app = Celery(...)
```

Хорошая схема (устойчивый поток импортов):

```python
# celery_app.py (хорошо)
app = Celery(...)

# tasks.py
from celery_app import app

@app.task(...)
def some_task(...):
    ...
```

#### namespace конфигурации (как это ощущается в твоём проекте)

Под “namespace конфигурации” мы понимаем практическое соглашение: где и как лежат настройки.

В твоём учебном проекте это выглядит как env vars с префиксом `CELERY_`:

- `CELERY_BROKER_URL`
- `CELERY_RESULT_BACKEND`
- `CELERY_QUEUE_NAME`

Именно их `celery_app.py` читает и передаёт в `Celery(...)`.

### Пошагово

#### Шаг 1. Представь структуру проекта

Для учебного старта достаточно такой структуры:

```text
project/
  celery_app.py
  tasks.py
  main.py
  .env (опционально)
```

Где:

- `celery_app.py` содержит объект `app` (Celery instance) и настройки;
- `tasks.py` импортирует `app` и декорирует задачи через `@app.task`;
- `main.py` отправляет задачу и читает статус.

#### Шаг 2. Создай `celery_app.py`

Пример:

```python
# celery_app.py
import os
from celery import Celery

CELERY_BROKER_URL = os.getenv("CELERY_BROKER_URL", "redis://localhost:6379/0")
CELERY_RESULT_BACKEND = os.getenv("CELERY_RESULT_BACKEND", "redis://localhost:6379/1")
CELERY_QUEUE_NAME = os.getenv("CELERY_QUEUE_NAME", "default")

app = Celery(
    "demo_app",
    broker=CELERY_BROKER_URL,
    backend=CELERY_RESULT_BACKEND,
)

# Пример базового маршрута задач:
app.conf.task_default_queue = CELERY_QUEUE_NAME
app.conf.task_routes = {
    "tasks.add": {"queue": CELERY_QUEUE_NAME},
}
```

Важное замечание: имя `"demo_app"` — это логическое имя приложения Celery. Оно используется для отображения/идентификации в некоторых местах и может влиять на то, как Celery воспринимает конфигурацию. В проде ты обычно держишь это аккуратно.

#### Автообнаружение задач

Если твоя codebase растёт и задачи лежат не только в одном `tasks.py`, можно включить **auto-discovery задач**. В этом режиме worker при старте импортирует указанные пакеты, и декораторы `@app.task` выполняются, регистрируя задачи в Celery app.

Концептуальный пример:

```python
# celery_app.py (фрагмент)
app.autodiscover_tasks(
    packages=["myproject"],  # корневой пакет проекта
)
```

Правильная форма (для копирования в настоящий код):

```text
app.autodiscover_tasks(
    packages=["myproject"],
)
```

Ключевое правило диагностики: auto-discovery — это import-процесс, а не “магия”. Если задачи не видны, первыми проверяй:

- правильность `-A` (worker импортирует тот app, который ты ожидаешь);
- корректность корневого пакета `packages=[...]`;
- что Python реально видит пакет (working directory / PYTHONPATH);
- отсутствие ошибок импорта внутри модулей задач.

#### Разделение settings по окружениям

Для development/test/production полезно иметь разные значения для:

- `broker URL`;
- `result backend URL`;
- имён очередей (часто).

Стартовый подход: переключать конфиг через env vars.

Например:

```python
# celery_app.py (фрагмент)
import os

env = os.getenv("CELERY_ENV", "development")

broker_url = os.getenv("CELERY_BROKER_URL")
backend_url = os.getenv("CELERY_RESULT_BACKEND")

# В реальном проекте можно расширить до набора env vars по env, но идея одна:
# "один код" + "разные конфиги" между окружениями.
```

Практическая ловушка на старте: поменял env vars в producer, но не перезапустил worker. Тогда обе стороны “думают разное”, и ты будешь видеть странные статусы/отсутствие доставки.

#### Шаг 3. Создай `tasks.py`

```python
# tasks.py
from celery_app import app


@app.task(name="tasks.add")
def add(x, y):
    return x + y
```

Заметь: аргументы `x` и `y` — обычные числа. Они сериализуются без проблем (обычно JSON).

#### Шаг 4. Подготовь `main.py` для отправки и чтения статусов

```python
# main.py
import time
from tasks import add  # чтобы было понятно, где задача
from celery.result import AsyncResult

if __name__ == "__main__":
    result = add.delay(2, 3)
    print("task_id:", result.id)

    # Дадим worker-у шанс выполнить
    for _ in range(10):
        print("state:", result.state)
        if result.successful() or result.failed():
            break
        time.sleep(0.5)

    if result.successful():
        print("result:", result.get())
    else:
        # AsyncResult может пригодиться и отдельно по task_id:
        ar = AsyncResult(result.id)
        print("backend state via AsyncResult:", ar.state)
```

### Простыми словами

Структура проекта — это “правильная упаковка” для worker’а. Если worker не может импортировать app и найти задачи, он будет работать “вхолостую”.

### Картинка в голове

Представь, что worker по `-A` открывает “коробку” с app. Внутри коробки есть “каталог задач” — декораторы уже зарегистрировали функции. Если worker открыл не ту коробку — каталога не существует.

### Как запомнить

Если что-то не видит worker: сначала проверь `-A` и импорты, и только потом — broker.

### Примеры

Когда запускаешь worker, ты будешь писать что-то вроде:

```bash
celery -A celery_app worker -l info
```

Это означает: “импортируй модуль `celery_app`, возьми объект `app` из него и регистрируй задачи из импортируемых модулей”.

### Практика / реальные сценарии

- В небольших проектах достаточно простой структуры (как выше).
- В больших — обычно добавляют auto-discovery и разделяют packages по domain, но логика `-A` остаётся центральной для диагностики.

### Типичные ошибки

- Цикл импортов: `tasks.py` импортирует `app`, а `celery_app.py` импортирует `tasks` слишком рано (или наоборот).
- Неправильный модуль для `-A` (например, `-A app` вместо `-A celery_app`).
- Перемещение файлов без изменения `-A` и/или текущего рабочего каталога (cwd) при запуске worker’а.

### Что будет если…

… в `tasks.py` декоратор без `name=...`?

- Он всё равно будет зарегистрирован, но имя будет сгенерировано по модулю/функции. На старте лучше управлять именем явно, чтобы легче было диагностировать и делать `task_routes`.

… ты передаешь несерилизуемый объект?

- Задача может не публиковаться корректно или worker упадёт на сериализации/распаковке. В этой части мы ещё сделаем акцент на том, какие аргументы должны быть сериализуемыми.

### Проверь себя

1. Почему `-A` — это не “про очередь”, а про Python-импорт?

<details><summary>Ответ</summary>

Потому что worker импортирует модуль/объект приложения, из которого Celery собирает настройки и регистрацию задач. Без правильного импорта worker не знает, какие задачи существуют.

</details>

2. Почему “единый модуль с Celery app” полезнее, чем дублирование настроек в нескольких файлах?

<details><summary>Ответ</summary>

Потому что настройки (broker URL, backend URL, очереди) должны совпадать у producer и worker. Если дублировать — легко допустить расхождение, особенно между dev/test.

</details>

3. Как диагностически отличить проблему `-A` от проблемы подключения к broker?

<details><summary>Ответ</summary>

Проблема `-A` проявится в том, что worker не регистрирует задачу/не видит её (и в логах будет ощущение “задачи отсутствуют”). Проблема broker проявится в ошибках соединения и невозможности доставки в очередь.

</details>

4. Какая схема импорта в мини-проекте снижает риск циклических зависимостей?
<details><summary>Ответ</summary>
Пригодный паттерн такой: `celery_app.py` создаёт один объект `app` (и только он), а `tasks.py` импортирует этот `app` и регистрирует задачи через `@app.task`. Тогда `app` не пытается заранее тянуть `tasks`, и получается направленный поток зависимостей без цикла.
</details>

5. Что случится, если получился цикл импорта `celery_app.py -> tasks.py -> celery_app.py`?
<details><summary>Ответ</summary>
Обычно worker упадёт/не завершит импорт модуля корректно: задачи могут не зарегистрироваться, появятся ошибки импорта в логах или worker будет “частично живым” без ожидаемой регистрации. На уровне симптомов это похоже на проблему `-A`, но первопричина в том, что импорты не разрулились.
</details>

6. Что именно означает `namespace конфигурации` в твоём учебном коде?
<details><summary>Ответ</summary>
Это соглашение: где лежат настройки и как их читают в одном месте. В файле заложена идея env vars с префиксом `CELERY_`: `CELERY_BROKER_URL`, `CELERY_RESULT_BACKEND`, `CELERY_QUEUE_NAME`. Именно их `celery_app.py` считывает и прокидывает в `Celery(...)` и `app.conf`.
</details>

7. Как понять, что auto-discovery не сработал в твоём проекте?
<details><summary>Ответ</summary>
Симптом: worker стартует, но в “Registered tasks …” нет ожидаемых имён задач, либо задачи вообще не перечисляются. Причины обычно последовательные: неверный `-A` (worker загрузил не тот app), неправильный корневой пакет в `packages=[...]`, или Python не видит пакет (cwd/PYTHONPATH), либо внутри модулей задач есть ошибка импорта.
</details>

8. Почему auto-discovery полезен именно при росте codebase, а не на самом старте?
<details><summary>Ответ</summary>
Когда задач становится много и они раскладываются по модулям/пакетам, ручное “подключить каждый tasks-файл” превращается в поддерживаемость-кошмар. Auto-discovery делает регистрацию зависимой от структуры пакетов: worker при старте импортирует нужные пакеты, и декораторы регистрируют задачи автоматически.
</details>

9. Что даёт разделение settings на development/test/production уже на этапе обучения?
<details><summary>Ответ</summary>
Оно устраняет класс проблем “на локали работает, а на тесте нет” из-за разных broker URL/result backend/очередей. Когда у producer и worker одинаковые env vars для выбранного окружения, диагностика быстрее: ты уверен, что проблема не в непреднамеренном расхождении конфигурации.
</details>

10. Если ты меняешь env vars в producer, но не перезапускаешь worker, какой эффект обычно увидишь?
<details><summary>Ответ</summary>
Worker будет использовать старую конфигурацию (старые URLs и/или очереди). Поэтому сообщение может публиковаться “куда-то туда”, а worker слушает по старым правилам, либо backend-статусы пишутся/читаются “из другой таблицы”. На практике это часто выглядит как “вечно PENDING” или несоответствие ожиданий.
</details>

11. Практический вопрос: worker видит `celery_app`, но `tasks.add` не зарегистрирована. Что проверяешь в каком порядке?
<details><summary>Ответ</summary>
В первую очередь — `-A` и рабочий import (проверить, что worker действительно импортирует тот же модуль app). Затем — структуру пакетов и корректность `packages=[...]` для auto-discovery (или наличие нужного импорта `tasks.py`, если auto-discovery не включён). После этого — доступность пакета из Python (cwd/PYTHONPATH) и отсутствие ошибок импорта внутри модулей задач.
</details>

12. Почему `main.py` в учебном проекте не должен “собирать app”?
<details><summary>Ответ</summary>
Потому что сборка/конфигурация `app` должна быть однозначной и одинаковой для producer и worker. Если main начнёт “творить app” по-своему, легко получить несоответствие настроек: разные URLs, разные очереди, разные routing правила. Тогда ты диагностику превращаешь в лотерею вместо проверки границ.
</details>

### Запомните

Мини-проект держится на одном “источнике app” (`celery_app.py`) и на том, что worker через `-A` попадает в правильную точку импорта.
