## Сравнение подходов в спорных местах (быстрый выбор)

| Ситуация | Подход A | Подход B | Когда выбирать |
|---|---|---|---|
| Долгая обработка | один большой task | pipeline из этапов + checkpoints | почти всегда B, если цена перезапуска заметна |
| Большие данные | payload в broker | pointer в object storage | почти всегда B для больших/повторяемых задач |
| Публикация событий | publish в рамках бизнес-кода | transactional outbox relay | B для надежности и воспроизводимости |
| Падение внешнего API | агрессивные retries | backoff+jitter+circuit breaker | B для защиты зависимости и очереди |
| Hybrid маршрутизация | один глобальный broker | zone-aware routing + fallback | B при multi-region и чувствительности к latency |
| Time scheduling | локальное время везде | UTC-first + явная DST policy | B для предсказуемости и дебага |

### Простыми словами

Если сомневаешься, выбирай подход, который:

1. уменьшает blast radius;
2. делает поведение наблюдаемым;
3. упрощает восстановление после сбоя.

#### Проверь себя: сравнение подходов

1. Почему подход B в таблице чаще выбран как предпочтительный, хотя он обычно сложнее внедрения?

<details><summary>Ответ</summary>

Потому что он снижает риск каскадных отказов и стоимость восстановления. Дополнительная инженерная сложность окупается устойчивостью в редких, но дорогих инцидентах.

</details>

2. В каких случаях подход A может быть временно приемлем?

<details><summary>Ответ</summary>

На раннем этапе проекта или в некритичном контуре при малой нагрузке, но с четким пониманием ограничений и планом миграции к более устойчивой схеме.

</details>

### Операционные метрики: минимальный набор сигналов

| Подтема | Что мерить | Тревожный сигнал |
|---|---|---|
| `24.1` Long-running | `task_runtime_p95`, `progress_stall_count` | рост runtime без роста прогресса |
| `24.2` Big payload | `payload_size_p95`, `broker_mem_usage` | скачок размеров + рост queue latency |
| `24.3` Version drift | `deserialize_error_rate`, `redelivery_rate` | всплеск ошибок после rollout |
| `24.4/24.9` Time | `clock_skew_ms`, `eta_start_delta_ms` | системный сдвиг ETA относительно плана |
| `24.5` Retry storm | `retry_rate`, `external_5xx_rate`, `429_rate` | retry растет быстрее recovery зависимости |
| `24.6` Race | `not_found_on_consume`, `outbox_lag_seconds` | consumer часто не видит ожидаемое состояние |
| `24.7` Hybrid | `publish_to_consume_ms_by_route` | деградация конкретного межзонного маршрута |
| `24.8` OOM | `oom_kill_count`, `container_memory_working_set` | повторные OOM на одной группе воркеров |

#### Проверь себя: метрики

1. Почему одной «общей» метрики здоровья очередей недостаточно для edge-case инженерии?

<details><summary>Ответ</summary>

Потому что классы отказов разные: время, память, совместимость, гонки данных, внешние зависимости. Одна метрика не даст причинно-следственной локализации.

</details>

2. Какая метрика полезна для triage, а какая просто «интересная»?

<details><summary>Ответ</summary>

Полезная метрика сразу указывает на следующий шаг: например, рост `retry_rate` ведет к breaker/degrade действиям, а рост `outbox_lag_seconds` — к проверке relay и DB bottleneck.

</details>

### Пример стартовых порогов (как отправная точка, не догма)

| Метрика | Стартовый порог | Что сделать при срабатывании |
|---|---|---|
| `clock_skew_ms` | > 500 ms (10+ минут) | проверить NTP/chrony и временно ограничить ETA-чувствительные задачи |
| `oom_kill_count` | >= 1 за 15 минут на worker-group | снизить concurrency/prefetch, изолировать heavy очередь |
| `deserialize_error_rate` | > 1% после rollout | остановить rollout, проверить compatibility matrix/legacy replay |
| `retry_rate` | рост > 2x baseline за 5 минут | включить breaker/degrade, пересмотреть backoff+jitter |
| `outbox_lag_seconds` | устойчиво > SLA доставки | масштабировать relay и проверить блокировки/cleanup outbox |

Важно: пороги обязательно калибруются под вашу нагрузку и SLO. Эти значения нужны как стартовая рамка для первой рабочей версии мониторинга.

#### Проверь себя: пороги

1. Почему нельзя бездумно переносить пороги из учебного примера в production?

<details><summary>Ответ</summary>

Потому что у разных систем разный baseline, сезонность и SLA. Неподходящие пороги дают либо шум, либо позднее обнаружение реальной деградации.

</details>

2. Как понять, что порог настроен удачно?

<details><summary>Ответ</summary>

Он стабильно срабатывает до каскадной деградации, имеет приемлемый уровень ложных срабатываний и для него есть понятный runbook с владельцем.

</details>
