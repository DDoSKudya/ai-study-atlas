## 24.1 Долгоживущие задачи

### Цель раздела

Понять, почему длинные задачи часто становятся операционно опасными, и научиться проектировать их так, чтобы они были управляемыми, наблюдаемыми и корректно останавливались.

### В этом разделе главное

- долгоживущая задача — это не просто «долго работает», а «долго удерживает риск»;
- heartbeat и фактический прогресс — разные вещи;
- graceful shutdown для длинных задач требует отдельной логики;
- чем длиннее задача, тем важнее checkpointing и декомпозиция на этапы.

### Термины

| Термин | Формально | Простыми словами |
|---|---|---|
| **Long-running task** | Задача с длительностью исполнения, заметно превышающей обычный SLA worker-пула | Задача, которая «сидит» в воркере долго и мешает гибкому управлению |
| **Heartbeat** | Периодический сигнал «процесс жив» | Пульс: процесс есть, но это не гарантия, что он продвигается |
| **Liveness vs progress** | Живость процесса и движение по полезной работе — разные метрики | «Жив» не равно «полезно работает» |
| **Graceful shutdown** | Корректная остановка с сохранением целостности | Остановить без дубликатов, потерь и порчи состояния |
| **Checkpoint** | Сохраненная точка прогресса | Место, с которого можно продолжить после сбоя |

### Теория и правила

1. **Не путай выживание процесса с выполнением бизнес-цели.**  
   Worker может быть жив, heartbeat может идти, а задача фактически застряла на внешнем API или в медленном I/O.

2. **Длинная задача повышает blast radius.**  
   Чем дольше задача живет, тем выше вероятность, что за это время произойдет deploy, рестарт, network glitch, revoke или деградация зависимости.

3. **Graceful shutdown должен быть спроектирован, а не «надеяться на удачу».**  
   Если задача не умеет корректно завершаться, rolling update легко превращается в дубли или в «брошенную» работу.

4. **Checkpointing снижает цену сбоя.**  
   Без checkpoint long task падает близко к финалу и начинает весь путь заново; с checkpoint — продолжает с последнего устойчивого этапа.

5. **Разделяй этапы, если есть естественные границы.**  
   Один гигантский task хуже, чем chain/group из этапов с явными входами/выходами.

### Пошагово: как проектировать долгоживущую задачу

1. Оцени, действительно ли это должна быть одна задача, а не pipeline.
2. Определи checkpoints: где можно гарантированно сохранить прогресс.
3. Добавь idempotency key для каждого этапа.
4. Реализуй мягкую остановку: периодически проверяй флаг отмены/дедлайн.
5. Внеси метрики прогресса (процент, этап, время с последнего прогресса).
6. Протестируй: kill child process, rolling restart, network timeout в середине выполнения.

### Граничные случаи и компромиссы

- **Слишком частый checkpoint:** снижает потери прогресса, но увеличивает I/O и нагрузку на хранилище.
- **Слишком редкий checkpoint:** дешевле в моменте, но дорогой recovery после падения.
- **Одна сверхдлинная задача:** проще «в одном файле», но хуже по управляемости и blast radius.

### Простыми словами

Долгоживущая задача похожа на дальний перелет с пересадками. Если нет отметок «долетели до промежуточного аэропорта», то при любой проблеме придется снова лететь с самого начала.

### Как запомнить

**LPPG:** `Liveness -> Progress -> Pause/Shutdown -> Checkpoint`.  
Всегда проверяй эти 4 пункта у длинной задачи.

### Пример: этапная обработка большого файла

```python
from celery import shared_task

@shared_task(bind=True, acks_late=True, autoretry_for=(TimeoutError,), retry_backoff=True)
def process_report(self, report_id: str):
    state = load_checkpoint(report_id) or {"stage": "start", "offset": 0}

    if state["stage"] == "start":
        prepare_context(report_id)
        save_checkpoint(report_id, {"stage": "chunks", "offset": 0})

    state = load_checkpoint(report_id)
    for chunk_idx, chunk in iter_chunks(report_id, start=state["offset"]):
        ensure_not_cancelled(report_id)  # graceful shutdown/revoke contract
        process_chunk(chunk)
        if chunk_idx % 20 == 0:
            save_checkpoint(report_id, {"stage": "chunks", "offset": chunk_idx})

    finalize_report(report_id)
    save_checkpoint(report_id, {"stage": "done", "offset": -1})
```

### Практика / реальные сценарии

- **Видео-транскодинг:** вместо одного 2-часового task — pipeline из сегментов + merge.
- **ETL большого объема:** chunk-based обработка с checkpoint на пачке.
- **ML инференс batch-job:** этапы `prepare -> infer -> persist -> notify`, каждый с отдельной задачей.

### Что будет, если не учитывать этот блок

- rolling deploy будет обрывать важные задачи;
- очередь «задеревенеет» от висящих worker child-процессов;
- SLA станет непредсказуемым (задача «жива», но результата нет);
- recovery после сбоя будет слишком долгим и дорогим.

### Типичные ошибки

- ориентироваться только на heartbeat без метрик прогресса;
- не иметь checkpoint и начинать все с нуля;
- игнорировать revoke/shutdown сценарии;
- хранить большой mutable state только в памяти child-процесса.

### Проверь себя

1. Почему heartbeat недостаточен как единственный индикатор здоровья long task?

<details><summary>Ответ</summary>

Heartbeat показывает, что процесс не умер, но не подтверждает движение бизнес-работы. Задача может быть «живой» и полностью заблокированной.

</details>

2. Когда стоит разрезать одну задачу на несколько этапов?

<details><summary>Ответ</summary>

Когда есть естественные границы прогресса, можно сохранять промежуточный результат и когда цена повтора «с нуля» слишком высока.

</details>

3. Что главное дает checkpointing?

<details><summary>Ответ</summary>

Уменьшает стоимость сбоев и время восстановления: продолжение с последней устойчивой точки вместо полного перезапуска.

</details>

### Запомните

Долгоживущая задача без checkpoint и плана graceful shutdown — это отложенный production-риск.
