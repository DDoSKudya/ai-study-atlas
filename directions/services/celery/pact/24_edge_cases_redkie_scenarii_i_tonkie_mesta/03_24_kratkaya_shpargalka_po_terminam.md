## Краткая шпаргалка по терминам

| Термин | Что это значит простыми словами |
|---|---|
| **Edge case** | Редкий режим, который редко возникает, но может сильно ломать систему. |
| **Perception of liveness** | «Кажется, что задача умерла/зависла», хотя это может быть неверной интерпретацией. |
| **Checkpointing** | Сохранение промежуточного прогресса, чтобы продолжить после сбоя, а не начинать заново. |
| **Retry storm** | Массовые повторы задач, которые дополнительно добивают и так падающую зависимость. |
| **Jitter** | Случайная «размазка» времени повторов, чтобы не ударить в одну секунду всем пулом. |
| **Feature drift** | Расхождение возможностей между версиями broker/клиентов/библиотек. |
| **DST (daylight saving time)** | Перевод часов, из-за которого «локальное время» прыгает вперед/назад. |
| **Stale read** | Чтение устаревших данных, когда задача стартует раньше фиксации нужного состояния. |
| **Geo-latency** | Задержка между географически удаленными узлами. |
| **cgroups memory limit** | Жесткий лимит памяти контейнера; при превышении процесс убивается. |
| **Clock drift** | Расхождение часов между машинами; ломает ETA/сроки/окна выполнения. |
| **UTC-first strategy** | Хранить и планировать в UTC, локальные зоны использовать только для отображения. |

#### Проверь себя: термины

1. Чем `clock drift` отличается от DST-эффекта и почему эти проблемы нельзя лечить одинаково?

<details><summary>Ответ</summary>

DST — календарное правило локального времени, а drift — рассинхронизация «текущего сейчас» между узлами. Для DST нужны календарные правила и timezone-политика, для drift — NTP/chrony контроль и мониторинг skew.

</details>

2. Почему `perception of liveness` может быть опаснее явного падения процесса?

<details><summary>Ответ</summary>

Команда видит «процесс жив» и откладывает реакцию, тогда как полезная работа может стоять. Это создает скрытую деградацию, которая обнаруживается слишком поздно.

</details>
