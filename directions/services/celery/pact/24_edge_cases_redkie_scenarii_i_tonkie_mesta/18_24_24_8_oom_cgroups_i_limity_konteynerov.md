## 24.8 OOM, cgroups и лимиты контейнеров

### Цель раздела

Разобраться, как container runtime и лимиты памяти влияют на поведение Celery worker, и как избежать внезапных `SIGKILL` и потери прогресса.

### В этом разделе главное

- `max_memory_per_child` не всегда спасает от cgroup OOM kill;
- Kubernetes может убивать процесс раньше, чем Celery успеет «самоочиститься»;
- memory governance должна включать лимиты, профилирование и контроль размера задач;
- swap/disk pressure на ноде влияют на latency и устойчивость.

### Термины

| Термин | Формально | Простыми словами |
|---|---|---|
| **OOM kill** | Принудительное завершение процесса при нехватке памяти | Процесс убивают без graceful cleanup |
| **cgroups limit** | Ограничение ресурсов контейнера | Жесткий «потолок» памяти/CPU |
| **`max_memory_per_child`** | Celery-порог перезапуска child-процесса | Механизм «подчистить распухший child» |
| **Memory leak** | Нерасторгаемый рост памяти | Память заняли и не вернули |

### Теория и правила

1. **Celery memory controls и Kubernetes memory controls — разные уровни.**
2. **Если cgroup limit слишком близко к рабочему пику, OOM будет «случайным».**
3. **Большой payload + высокая concurrency = ускоренный OOM-риск.**
4. **Нужны отдельные очереди для memory-heavy задач.**
5. **Важен budget memory на процесс и на pod в целом.**

### Разбор тонкого места из плана: OOM в Kubernetes vs `max_memory_per_child`

`max_memory_per_child` — это профилактика на уровне Celery child-процессов.  
OOM в Kubernetes — это принудительное действие ядра/контейнерного рантайма на уровне cgroup.

**Важно:** когда pod близок к лимиту, Kubernetes может убить процесс быстро (`SIGKILL`), и Celery не успеет выполнить «аккуратную» ротацию child-процесса.

Практический вывод:  
`max_memory_per_child` нужен, но его недостаточно без корректного memory budget и workload-изоляции.

### Отдельно: swap и disk pressure на ноде

Даже если формально OOM нет, нода может деградировать из-за:

- активного swap (если он разрешен в вашей среде);
- disk pressure (node condition), когда система начинает агрессивнее вытеснять/ограничивать процессы;
- резкого роста latency I/O, из-за чего long-running задачи выглядят «зависшими».

**Что делать:**
- мониторить node-level метрики (`memory pressure`, `disk pressure`, I/O latency);
- не держать тяжелые временные файлы задач на медленном/переполненном диске;
- заранее планировать ephemeral storage budget для worker pod.

### Пошагово: анти-OOM контур

1. Замерь реальный memory profile по типам задач.
2. Раздели очереди: normal / heavy.
3. Для heavy очереди снизь concurrency и prefetched объем.
4. Настрой `max_memory_per_child` как дополнительную страховку.
5. Установи alerting на `container_memory_working_set_bytes` и OOM events.
6. Отработай chaos-сценарий: искусственный memory spike.
7. Добавь node-level алерты на disk pressure и рост swap/I/O wait.
8. Документируй, какие очереди первыми ограничиваются при memory emergency.

### Пример фрагмента конфигурации worker-пула

```bash
celery -A proj worker \
  --queues=heavy_tasks \
  --concurrency=2 \
  --prefetch-multiplier=1 \
  --max-tasks-per-child=20
```

```python
# settings.py
worker_max_memory_per_child = 512000  # KB
task_acks_late = True
```

### Простыми словами

Контейнер похож на комнату с низким потолком: если заносить туда слишком крупную мебель (payload + concurrency), потолок «срежет» процесс внезапно.

### Картинка в голове

`max_memory_per_child` — это уборщик в комнате, который периодически выносит лишнее.  
OOM cgroup — это автоматический рубильник здания: если перегрузили ввод, выключат весь этаж, не дожидаясь уборщика.

### Как запомнить

**BLIM:** `Budget -> Limit -> Isolate heavy queues -> Monitor node pressure`.

### Типичные ошибки

- считать `max_memory_per_child` заменой контейнерным лимитам;
- давать одинаковую concurrency всем типам задач;
- игнорировать всплески памяти на сериализации/десериализации;
- не разделять heavy и normal workloads;
- не смотреть на node-level disk pressure и влияние временных файлов.

### Что будет, если…

`SIGKILL` приведет к резкому росту redelivery/дубликатов, хвостам в очередях и длинному восстановлению после пиков.

### Проверь себя

1. Почему worker может быть убит, даже если настроен `max_memory_per_child`?

<details><summary>Ответ</summary>

Потому что cgroup лимит применяется жестко на уровне контейнера, и OOM kill может произойти раньше, чем Celery успеет перезапустить child по своему внутреннему порогу.

</details>

2. Зачем выносить memory-heavy задачи в отдельную очередь?

<details><summary>Ответ</summary>

Чтобы ограничить blast radius: тяжелые задачи не должны вытеснять и дестабилизировать обычные потоки.

</details>

3. Как связаны prefetch и OOM?

<details><summary>Ответ</summary>

Большой prefetch увеличивает объем одновременно удерживаемых задач и данных в памяти, что повышает риск переполнения.

</details>

### Запомните

OOM-устойчивость — это комбинация архитектуры задач, лимитов контейнера и параметров worker-а, а не одна «волшебная настройка».
