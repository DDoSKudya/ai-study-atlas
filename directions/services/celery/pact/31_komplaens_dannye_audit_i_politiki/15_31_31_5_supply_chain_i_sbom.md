## 31.5 Supply chain и SBOM

### Цель раздела

Закрепить дисциплину управления зависимостями Celery-стека (Celery, Kombu, Billiard и др.) через SBOM, контроль уязвимостей и предсказуемую политику обновлений.

### В этом разделе главное

- Комплаенс включает не только данные, но и "из чего состоит ПО".
- SBOM нужен для быстрой реакции на CVE и для доказуемого процесса обновлений.
- Обновления без политики дают или застой с уязвимостями, или хаос несовместимости.

### Термины

| Термин | Суть |
|---|---|
| **Supply chain security** | Безопасность цепочки поставки ПО: зависимости, артефакты, CI/CD. |
| **SBOM** | Машиночитаемый список компонентов и версий в поставляемом ПО. |
| **CVE** | Идентификатор публично известной уязвимости. |
| **SCA (Software Composition Analysis)** | Автопроверка зависимостей на уязвимости и лицензии. |

### Теория и правила

1. Ведите фиксированный список зависимостей и диапазонов версий.
2. Генерируйте SBOM на каждый релиз артефакта.
3. Настройте SCA-проверки в CI для критичных CVE.
4. Обновляйте зависимости по календарю и по внеплановым security-событиям.
5. Проверяйте совместимость не только Celery, но и соседей (`kombu`, `billiard`, transport libs).
6. Контролируйте не только Python-пакеты, но и базовые контейнерные образы.
7. Для критичных уязвимостей определите SLA реакции (например, 24-48 часов).

### Пошагово

1. Зафиксируй policy обновлений: cadence, owner, критерии срочного патча.
2. Подключи генерацию SBOM в pipeline сборки.
3. Включи security gates по severity уязвимостей.
4. Для каждой волны апдейтов проводи тестовый прогон на staging и canary в production.
5. Документируй решение: почему обновили/отложили, какие риски приняли.

### Простыми словами

Supply chain — это "продуктовый состав" вашей системы. Если вы не знаете, из каких библиотек собран worker и какие в них уязвимости, вы не управляете риском, а просто надеетесь на удачу.

### Картинка в голове

```text
Исходники -> Сборка -> SBOM -> Security scan -> Релиз
                         \-> отчет для аудита/разбора CVE
```

### Как запомнить

**"Нет SBOM — нет прозрачности; нет прозрачности — нет управляемого риска."**

### Примеры

Пример policy-фрагмента (концептуально):

```yaml
dependency_policy:
  scheduled_update: monthly
  emergency_patch:
    cve_severity: [critical, high]
    max_delay_hours: 48
  required_artifacts:
    - sbom.json
    - vulnerability_report.json
  rollout:
    - staging
    - canary_10_percent
    - full_prod
```

Пример "связки с частью 27" на практике:

```text
deps.lock + compatibility matrix + SBOM + CVE scan
=> решение release board: deploy / hold / emergency patch
```

### Практика / реальные сценарии

- **Всплывает критичный CVE в зависимости транспортного слоя:** команда поднимает emergency-процесс патча по формализованному runbook.
- **Аудит поставки ПО:** нужно показать, какие версии библиотек работали в конкретную дату и на каком основании.
- **Миграция major-версии Celery:** SBOM + compatibility matrix + canary дают контролируемый переход.

### Типичные ошибки

- "заморозить версии навсегда" и игнорировать security-апдейты;
- обновлять сразу все без staging/canary;
- не хранить артефакты проверки зависимостей;
- считать, что проверка только `requirements.txt` закрывает тему (пропуская контейнерный базовый образ).

### Что будет если...

- **Если нет контроля зависимостей:** критичные CVE могут жить в production месяцами.
- **Если обновлять хаотично:** растет риск несовместимости и инцидентов после релиза.
- **Если нет SBOM-артефактов:** невозможно быстро ответить "затронуты ли мы уязвимостью X?".
- **Если не учтен базовый образ контейнера:** можно "закрыть" Python-зависимости и пропустить критичный CVE в OS-layer.

### Проверь себя

1. Почему SBOM особенно важен для Celery-контуров, а не только для web-сервиса?

<details><summary>Ответ</summary>

Потому что Celery-стек включает несколько рантаймов и транспортных зависимостей, часто с долгоживущими worker-ами. Уязвимости могут проявляться в фоновом контуре, который не всегда виден через обычный HTTP-периметр.

</details>

2. Что должно быть обязательным выходом security-aware CI для фоновых worker-образов?

<details><summary>Ответ</summary>

Как минимум SBOM и отчет по уязвимостям (SCA/scan), а также решение по блокировке/допуску релиза на основе политики severity.

</details>

### Запомните

Комплаенс по цепочке поставки — это повторяемый процесс обновлений, а не разовый security-спринт.
