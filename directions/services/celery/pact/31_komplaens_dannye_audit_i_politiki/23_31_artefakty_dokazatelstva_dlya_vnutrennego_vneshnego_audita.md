## Артефакты-доказательства для внутреннего/внешнего аудита

Этот раздел отвечает на практический вопрос: "чем именно мы докажем, что требования части 31 выполняются?"

| Область | Минимальный набор артефактов | Кто обычно предоставляет |
|---|---|---|
| Классификация данных | Data classification matrix + task compliance cards | Backend/архитектор |
| Payload hygiene | Отчеты negative tests + примеры заблокированных payload | Backend/QA |
| Retention/удаление | Политика retention + лог выполнения cleanup + отчеты по deletion requests | Platform/Data ops |
| Legal hold | Реестр hold-записей + журнал решений по исключениям | Compliance/Legal ops |
| Аудит и корреляция | Схема audit event + e2e-трейс по `request_id/task_id` | Backend/SRE |
| Резидентность | Карта регионов инфраструктуры + policy роутинга + логи блокировок cross-border | Platform/SRE |
| Supply chain | SBOM, SCA-отчет, CVE triage-решения, release approval | Security/Platform |

Минимальный принцип: если требование нельзя подтвердить артефактом, требование считается не внедренным.

#### Проверь себя: артефакты-доказательства

1. Почему "документ есть" не равно "требование выполнено"?

<details><summary>Ответ</summary>

Документ фиксирует намерение, но не доказывает исполнение. Нужны живые артефакты: отчеты, логи, трейсы, журналы решений, результаты проверок.

</details>

2. Какой артефакт особенно важен при внешнем аудите удалений?

<details><summary>Ответ</summary>

Трассируемый отчет по deletion workflow с отметками исключений/hold и подтверждением фактического выполнения в системах хранения.

</details>
