## 31.1 Классификация данных в сообщениях

### Цель раздела

Научиться проектировать payload задач так, чтобы Celery-очередь не становилась хранилищем чувствительных данных и источником регуляторных рисков.

### В этом разделе главное

- Не любая "удобная структура JSON" допустима для очереди.
- Золотое правило: **передаем ссылку/идентификатор, а не исходный чувствительный объект**.
- Риски есть не только в `args/kwargs`, но и в `headers`, traceback, результатах и логах.

### Термины

| Термин | Суть |
|---|---|
| **Sensitive payload** | Сообщение, содержащее чувствительные поля (паспорт, карта, диагноз и т.д.). |
| **Data classification** | Процесс маркировки данных по уровням чувствительности. |
| **Redaction** | Маскирование или удаление чувствительных фрагментов из лога/ошибки. |
| **Need-to-know** | Доступ и передача только того, что реально нужно исполнителю задачи. |

### Теория и правила

1. **Классифицируй данные до публикации задачи.**  
   Producer должен знать, какие поля относятся к PII/PHI/PCI.
2. **Минимизируй payload.**  
   Передавай `order_id`, `document_ref`, `token`, а не полный пользовательский профиль.
3. **Разделяй бизнес-данные и служебную мета-информацию.**  
   Для трассировки достаточно `request_id`, `actor_id`, `tenant_id` без чувствительных полей.
4. **Ограничивай traceback и result metadata.**  
   Ошибки часто случайно включают сырые аргументы или фрагменты ответа внешнего API.
5. **Отдельно проверь сериализацию.**  
   Нельзя "прятать" риск в бинарной сериализации: риск остается, просто хуже наблюдаем.

### Пошагово

1. Введи таблицу классификации полей для каждой ключевой задачи.
2. На уровне producer добавь валидатор payload с deny-list и allow-list.
3. Замени чувствительные данные ссылками на защищенное хранилище.
4. Настрой логирование с redaction (`email`, `phone`, `card`, `token` и т.д.).
5. Прогони "compliance smoke test": публикация -> выполнение -> ошибка -> ретрай -> аудит.

### Простыми словами

Очередь Celery — это как почтовый конверт, который проходит через несколько сортировочных центров. Если положить в конверт "все личное дело клиента", риск утечки станет системным. Правильнее положить номер заявки, а детали держать в специально защищенном архиве.

### Картинка в голове

`payload = указатель`,  
`безопасное хранилище = сейф`,  
`worker = сотрудник с допуском, который открывает сейф только по необходимости`.

### Как запомнить

Формула: **"ID вместо объекта, контекст вместо секрета, маскировка вместо сырых логов"**.

### Примеры

Плохо:

```python
process_payment.delay({
    "card_number": "4111111111111111",
    "cvv": "123",
    "user_email": "user@example.com",
    "amount": 1200
})
```

Лучше:

```python
process_payment.delay(
    payment_token="pay_tok_8f2a",
    order_id="ORD-98231",
    actor_id="svc-checkout",
    request_id="req-4c7f0"
)
```

Идея: чувствительные реквизиты уже лежат в отдельном PCI-контуре, Celery получает только ссылку.

Celery-специфичный прием для маскировки представления аргументов в событиях/логах:

```python
from celery import shared_task

@shared_task(bind=True, name="billing.capture_payment")
def capture_payment(self, payment_token: str, order_id: str):
    # В задаче храним только токен/идентификатор.
    ...


# При публикации можно явно скрыть подробности аргументов для наблюдаемости:
capture_payment.apply_async(
    kwargs={"payment_token": "pay_tok_real_value", "order_id": "ORD-98231"},
    kwargsrepr="{'payment_token': '***', 'order_id': 'ORD-98231'}",
)
```

Где обычно течет чувствительная информация, даже если payload уже "чистый":

| Участок | Типичная утечка | Что сделать |
|---|---|---|
| `kwargs` задачи | email/телефон/карточные данные | передавать только `*_id`/`*_token` |
| `headers` | сырые токены доступа | не передавать секреты через headers |
| traceback | фрагменты запросов/ответов | sanitize исключения перед логированием |
| result backend | избыточные debug-данные | хранить только технический минимум |

### Практика / реальные сценарии

- **Checkout-сервис:** в очередь идет только `payment_token`, а фактические реквизиты карты не покидают платежный контур.
- **Медицинские уведомления:** в задачу передается `patient_ref`, а доступ к диагнозам worker получает только через сервис с аудитом доступа.
- **Маркетинговая рассылка:** вместо email-адреса передается `user_id`, а резолвинг адреса идет в момент отправки с учетом consent.

### Типичные ошибки

- сохранять весь request body в `kwargs` "для удобства дебага";
- логировать исключения с сырыми аргументами задачи;
- считать, что "внутренний брокер" автоматически безопасен;
- передавать JWT/refresh token напрямую в payload;
- хранить чувствительные поля в результате задачи "чтобы потом смотреть в Flower".

### Что будет если...

- **Если передавать PII в payload массово:** рост поверхности утечки и тяжесть инцидента.
- **Если не маскировать traceback:** чувствительные поля попадут в логи, алерты, тикеты.
- **Если игнорировать классификацию:** сложно доказать контролируемость данных при аудите.

### Проверь себя

1. Почему "удобно для разработчика" не равно "безопасно для комплаенса" при проектировании payload?

<details><summary>Ответ</summary>

Потому что удобство обычно ведет к передаче избыточных данных. Комплаенс требует минимизации и контролируемости: что передано, зачем, где хранится и когда удаляется.

</details>

2. Какие поля чаще всего случайно утекают через Celery-контур, даже если payload "чистый"?

<details><summary>Ответ</summary>

Traceback, текст исключений, служебные логи, result metadata, custom headers и отладочные дампы.

</details>

3. Почему токенизация снижает риск сильнее, чем просто шифрование payload?

<details><summary>Ответ</summary>

Токенизация уменьшает сам объем чувствительных данных в очереди. Шифрование полезно, но данные все равно присутствуют и потенциально могут быть раскрыты при неправильном управлении ключами/доступом.

</details>

### Запомните

Celery-задача должна нести минимум данных, достаточный для выполнения работы. Все чувствительное — за пределами очереди, в контролируемом хранилище с отдельным доступом и аудитом.
