## Быстрый выбор паттерна (decision guide)

Если нужно быстро выбрать архитектурный шаблон, используй дерево решений:

```text
Нужен ли пользователю отдельный конечный результат операции?
|
+-- Нет --> Fire-and-forget (20.1)
|
+-- Да --> Операция длится дольше обычного HTTP timeout?
          |
          +-- Да --> Async request/reply (20.2)
          |         + при большом объеме данных -> Batch (20.3)
          |
          +-- Нет --> Нужен ли многошаговый конвейер?
                    |
                    +-- Да --> Pipeline (20.5)
                    +-- Нет --> Рассмотреть синхронный API без Celery
```

Дополнительно:

- если источник изменений — доменные события между сервисами, добавляй `Event-driven handlers` (`20.4`);
- если в системе есть конфликтующие по приоритету нагрузки, обязательно добавляй `Bulkhead` (`20.6`);
- на любом варианте проверяй себя по `Anti-patterns` (`20.7`).

#### Проверь себя

1. Почему decision guide начинается с вопроса о пользовательском результате, а не с технологических деталей?

<details><summary>Ответ</summary>

Потому что архитектурный паттерн выбирается от продуктового контракта. Технология вторична: сначала определяем, что именно обещаем пользователю и когда.

</details>

2. В каком случае из дерева следует выйти в синхронный API без Celery?

<details><summary>Ответ</summary>

Когда операция быстрая, не требует асинхронного жизненного цикла и не имеет смысла усложнять систему отдельным фоновым контуром.

</details>
