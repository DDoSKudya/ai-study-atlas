#### Проверь себя

1. Почему без понимания eventual consistency опасно проектировать async request/reply?

<details><summary>Ответ</summary>

Потому что пользовательский результат появляется не мгновенно, и между публикацией задачи и финальным статусом есть временной зазор. Если это не учитывать, API и UI начинают обещать то, чего система физически не может гарантировать сразу.

</details>

2. Какую минимальную базу нужно знать до этой части, чтобы не свести всё к "копипасте конфигов"?

<details><summary>Ответ</summary>

Нужно понимать жизненный цикл задачи, модель доставки (повторы/сбои), идемпотентность, а также базовую наблюдаемость. Тогда паттерны выбираются по смыслу, а не по случайным примерам.

</details>
