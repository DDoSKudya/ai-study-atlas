## Краткая шпаргалка по терминам

| Понятие | Суть простыми словами |
| --- | --- |
| **Модуль конфигурации Celery** | Отдельный файл/пакет, где описаны все важные настройки Celery: URL брокера, result backend, очереди, сериализация, таймзона и т.д. |
| **Env‑specific настройки** | Параметры, которые отличаются между окружениями (local/test/stage/prod): URL‑ы, креденшелы, topology очередей, лимиты. |
| **Config through env vars** | Подход, когда ключевые настройки читаются из **переменных окружения**, а не зашиваются в код. |
| **Namespace (Django)** | Пространство имён конфигурации Celery внутри `settings.py` / `CELERY_...`‑prefixed переменных. |
| **Immutable configuration** | Конфигурация, которая **не меняется динамически** в рантайме произвольно: изменения проходят через релиз/деплой. |
| **Broker URL (`broker_url`)** | Строка подключения к брокеру (Redis/RabbitMQ/SQS и т.д.), определяет транспорт и поведение доставки. |
| **Result backend (`result_backend`)** | Хранилище для статусов и результатов задач (Redis, DB, RPC, custom). |
| **Task routes (`task_routes`)** | Карта соответствия задач и очередей: какая задача в какую очередь отправляется. |
| **Default queue (`task_default_queue`)** | Очередь по умолчанию, куда попадают задачи без явного маршрута. |
| **Prefetch (worker_prefetch_multiplier)** | Сколько сообщений worker заранее резервирует из брокера. Влияет на латентность и fairness. |
| **Ack semantics (`task_acks_late` и др.)** | Правила, когда worker подтверждает обработку сообщения брокеру. |
| **Configuration by environment** | Явная схема: чем конфиг на dev отличается от stage/prod (и почему). |
| **Infrastructure‑as‑code** | Подход, когда инфраструктура (очереди, пользователи, мониторинг) описана в коде/манифестах, а не создана вручную. |
| **Versioned config** | Версионирование конфигурации: изменения проходят через VCS и релизы, а не «вручную на проде». |
| **Feature flags** | Флаги/переключатели, позволяющие постепенно включать новые очереди, маршруты, типы задач. |

#### Проверь себя: шпаргалка

1. Почему `broker_url` и `result_backend` логично называть **инфраструктурными**, а `task_routes` — **архитектурным** параметром?

<details><summary>Ответ</summary>

`broker_url`/`result_backend` описывают **конкретные внешние сервисы** (инфраструктуру), к которым подключается Celery. `task_routes` описывает **как домены задач раскладываются по очередям**, то есть структуру приложения и контракты между частями системы — это уже архитектура.

</details>

2. Что делает конфигурацию «immutable» и зачем это нужно?

<details><summary>Ответ</summary>

Immutable‑конфигурация не меняется «на лету» в произвольных местах кода и через ручные правки; любые изменения проходят через VCS, код‑ревью, деплой. Это снижает риск «невоспроизводимых багов» и расхождений между средами.

</details>

3. В чём плюс чтения настроек из `env` перед «жёстким захардкоживанием» в код?

<details><summary>Ответ</summary>

Настройки можно **разводить по средам** без изменения кода, не хранить секреты в репозитории и проще воспроизводить окружения через набор переменных/манифестов.

</details>
