## Анти-путаница: что с чем не перепутать

| Что сравниваем | Правильное различие | Частая путаница |
|---|---|---|
| Signal vs Event | Signal — callback внутри приложения; Event — телеметрическое сообщение для наблюдаемости | считать event «таким же хуком», как signal |
| Signal vs Task subclass | Signal глобально реагирует на lifecycle, `Task` subclass задаёт контракт для семейства задач | реализовывать всю прикладную логику в глобальных signal-handlers |
| Bootstep vs Signal | Bootstep влияет на инфраструктурный lifecycle worker/consumer, signal обычно реактивен | использовать bootstep там, где нужен простой callback |
| `task_annotations` vs routing policy | `task_annotations` меняет task options на уровне политики; routing управляет топологией очередей | пытаться решить архитектуру очередей только annotations |
| Custom serializer vs schema evolution | Serializer определяет формат передачи; эволюция схемы требует миграционного контракта producer/worker | менять формат «одним релизом» без compatibility window |

### Как запомнить различие в одной фразе

Если нужно **наблюдать** — смотри в `signals/events`;  
если нужно **управлять поведением конкретных задач** — смотри в `Task`/`task_annotations`;  
если нужно **расширять инфраструктуру worker** — смотри в `bootsteps`;  
если нужно **менять транспорт формата/хранилища** — смотри в serializer/backend/loader.

#### Проверь себя: анти-путаница

1. Почему `Event` нельзя считать заменой `Signal`, даже если оба описывают lifecycle?
2. Какой риск возникает, если routing-проблему лечить только через `task_annotations`?
3. По какому признаку понять, что задача требует `Task subclass`, а не глобального signal?

<details><summary>Ответ</summary>

1) Event — это телеметрический факт для наблюдения, signal — кодовая точка реакции внутри приложения; у них разная роль и гарантия использования.  
2) Можно замаскировать архитектурную проблему очередей и получить нестабильный SLA при росте нагрузки.  
3) Если нужно устойчивое и повторяемое поведение именно для семейства задач (с тестируемым контрактом), обычно лучше `Task subclass`.

</details>
