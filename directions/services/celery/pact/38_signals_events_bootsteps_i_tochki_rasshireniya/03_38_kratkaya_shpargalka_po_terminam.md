## Краткая шпаргалка по терминам

| Термин | Простыми словами |
|---|---|
| **Signal** | Точка уведомления о событии в жизненном цикле Celery (публикация, исполнение, worker lifecycle). |
| **Publish signals** | Сигналы до/после публикации задачи в брокер (`before_task_publish`, `after_task_publish`). |
| **Task signals** | Сигналы вокруг исполнения задачи (`task_prerun`, `task_postrun`, `task_success`, `task_failure` и др.). |
| **Worker signals** | Сигналы старта/готовности/остановки worker-процессов. |
| **Event protocol** | Поток событий Celery для мониторинга (`worker-*`, `task-*` события). |
| **Bootsteps** | Механизм поэтапной сборки компонентов worker/consumer. |
| **Consumer step** | Кастомный шаг, который встраивается в consumer-пайплайн worker-а. |
| **Task annotations** | Динамическое изменение свойств задач без прямого редактирования кода задач. |
| **Custom Task class** | Базовый класс задач с переопределенными lifecycle-методами. |
| **Extension boundary** | Граница, где расширение безопасно и не нарушает семантику Celery. |

#### Проверь себя: термины и границы

1. Чем `Task signals` отличаются от `Event protocol` на уровне назначения?
2. Почему понятие `Extension boundary` критично в production, а не только «для красоты архитектуры»?
3. В каком случае `Consumer step` уместен, а в каком будет оверинжинирингом?

<details><summary>Ответ</summary>

1) Signals — это callback-механизм внутри runtime Celery, а events — телеметрический поток для наблюдаемости и внешних потребителей.  
2) Потому что выход за границы расширения ломает предсказуемость: растут скрытые зависимости, риск регрессий и стоимость поддержки.  
3) Consumer step уместен, когда нужно инфраструктурное расширение lifecycle consumer-а; если задача решается signal/Task-hook, bootstep обычно лишний.

</details>
