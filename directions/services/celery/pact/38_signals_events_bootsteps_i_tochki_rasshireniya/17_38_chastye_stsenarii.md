## Частые сценарии

### Сценарий 1. Нужно добавить trace correlation в задачи

**Решение:**

1. Добавить корреляционный id в `before_task_publish`.
2. Подхватывать id в `task_prerun`.
3. Логировать id в `on_failure/after_return`.

**Почему так:** минимальное вмешательство, прозрачный путь от producer до worker.

### Сценарий 2. Flower «тяжелит» брокер на пике

**Решение:**

1. Оценить текущий event volume.
2. Ограничить постоянный event stream, оставить incident profile.
3. Перенести регулярные дашборды на агрегированные метрики.

**Почему так:** observability должна быть управляемой по стоимости.

### Сценарий 3. Нужна сложная интеграция в lifecycle consumer

**Решение:**

1. Проверить, можно ли обойтись signal или `Task` subclass.
2. Если нет — добавить минимальный `ConsumerStep` с `requires`.
3. Ввести feature flag + canary rollout + быстрый rollback.

**Почему так:** bootsteps — high-impact зона, её нельзя внедрять без safety контуров.

### Сценарий 4. Централизованная политика rate limits без правки десятков задач

**Решение:**

1. Использовать `task_annotations` как функцию.
2. Сегментировать задачи по namespace/паттерну имени.
3. Проверить конфликт с явными параметрами `apply_async`.

### Сценарий 5. После релиза появились `task_unknown`

**Решение:**

1. Проверить, одинаковый ли код/версия задач на всех worker-нодах.
2. Проверить import/discovery и имя task (`name=` у декоратора).
3. Сверить очереди/routing key и фактический consumer mapping.
4. До фикса отключить публикацию новых сообщений проблемного типа или направить в quarantine queue.

**Почему так:** `task_unknown` обычно указывает на рассинхронизацию контракта producer/worker, а не «случайный сбой сети».

#### Проверь себя: частые сценарии

1. Почему в сценарии с `task_unknown` первым шагом проверяют version/code drift, а не сеть?
2. Чем сценарий с canary bootstep принципиально отличается от сценария с `task_annotations` по риску воздействия?
3. В каком сценарии целесообразно временно ограничить event stream, а в каком — наоборот расширить?

<details><summary>Ответ</summary>

1) Потому что `task_unknown` чаще всего связан с несовпадением имен/регистрации задач между producer и worker, а не с транспортной недоступностью.  
2) Bootstep меняет инфраструктурный lifecycle worker-а (blast radius выше), а `task_annotations` обычно воздействует на policy задач и проще откатывается.  
3) Ограничить — при cost pressure/пике нагрузки; расширить — в окне инцидентного расследования для детальной диагностики.

</details>
