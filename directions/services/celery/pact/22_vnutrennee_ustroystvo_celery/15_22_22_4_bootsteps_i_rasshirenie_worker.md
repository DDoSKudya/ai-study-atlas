## 22.4 Bootsteps и расширение worker

### Цель раздела

Понять, как worker собирается из шагов и как расширять его без неявной магии.

### В этом разделе главное

- bootsteps задают зависимый граф инициализации;
- расширение worker-а должно быть детерминированным и наблюдаемым;
- bootsteps подходят для инфраструктурных задач, не для бизнес-логики.

### Термины

| Термин | Смысл |
|---|---|
| **Blueprint** | План сборки worker-компонентов. |
| **Step** | Отдельный этап инициализации/остановки. |
| **requires** | Зависимости bootstep от других шагов. |

### Теория и правила

1. Bootstep должен иметь явную ответственность.
2. Любая инициализация должна быть обратимой при shutdown.
3. Ошибки в bootsteps должны логироваться с понятным контекстом.

### Когда bootsteps нужен, а когда нет

| Ситуация | Подходит bootsteps? | Почему |
|---|---|---|
| Инициализация telemetry/tracing клиента | Да | Инфраструктурный cross-cutting слой |
| Подключение health/readiness hooks | Да | Это часть lifecycle worker-а |
| Бизнес-компенсации и доменная оркестрация | Нет | Это должна делать явная логика задач/canvas |
| Разовая миграция данных при старте | Скорее нет | Риск долгого старта и плохой управляемости rollout |

### Bootsteps vs Signals: что выбирать

| Критерий | Bootsteps | Signals |
|---|---|---|
| Основная задача | Сборка и lifecycle worker-компонентов | Реакция на события lifecycle задач/worker |
| Подходящий уровень | Инфраструктурный, системный | Наблюдаемость, легкие cross-cutting hooks |
| Риск неправильного применения | Нестабильные старты/утечки ресурсов | Неявная бизнес-логика и хрупкие побочные эффекты |
| Когда точно не использовать | Для доменных действий и workflow-решений | Для критичных бизнес-компенсаций |

#### Проверь себя по выбору точки расширения

1. Когда корректнее выбрать bootsteps вместо signals?
2. Почему бизнес-компенсации в signals считаются архитектурным риском?

<details><summary>Ответ</summary>

1) Когда нужно управлять жизненным циклом самого worker-а: инициализация инфраструктурных компонентов, readiness hooks, controlled startup/shutdown.  
2) Потому что логика становится неявной, хуже тестируется и может срабатывать в неожиданных условиях повторов/отказов.

</details>

### Пошагово

1. Определи цель шага (например, init telemetry client).
2. Укажи зависимости `requires`.
3. Реализуй `start`/`stop`.
4. Проверь graceful shutdown и повторный restart.

### Пример (упрощенно)

```python
from celery import bootsteps

class MetricsStep(bootsteps.StartStopStep):
    requires = {"celery.worker.components:Pool"}

    def start(self, worker):
        worker.my_metrics = {"enabled": True}

    def stop(self, worker):
        worker.my_metrics = None

app.steps["worker"].add(MetricsStep)
```

### Простыми словами

Bootsteps — это "чеклист запуска самолета". Если этапы идут в неправильном порядке, полет возможен, но риск аварии резко растет.

### ASCII-диаграмма порядка запуска

```text
Worker Start
  -> Load config
  -> Build blueprint
  -> Start transport/consumer
  -> Start pool
  -> Start optional custom steps
  -> Mark worker ready
```

### Типичные ошибки

- скрытая тяжелая инициализация в каждом child процессе;
- зависимости шага не объявлены явно;
- отсутствие cleanup при stop.

### Что будет, если...

**...добавлять произвольные side-effects в bootsteps без контроля?**  
Появятся нестабильные старты, утечки ресурсов и сложная диагностика restart-проблем.

### Проверь себя

1. Для чего используется `requires`?
2. Почему `stop` так же важен, как `start`?

<details><summary>Ответ</summary>

1) Чтобы обеспечить корректный порядок инициализации зависимых подсистем.  
2) Без cleanup остаются dangling connections/threads/handlers, что ломает рестарты и rollout.

</details>

### Запомните

Bootsteps — инструмент инженерной сборки worker-а. Пользуйся им как конструктором с явными зависимостями.
