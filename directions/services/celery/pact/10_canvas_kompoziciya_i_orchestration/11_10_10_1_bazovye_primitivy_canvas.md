## 10.1. Базовые примитивы Canvas

### Цель раздела

Сформировать точную картину, из каких примитивов собирается workflow в Celery и какой тип зависимости описывает каждый примитив.

### В этом разделе главное

- `signature` - базовая единица композиции, не "запуск", а "план запуска".
- `chain` - последовательность.
- `group` - параллельность без обязательной агрегации в конце.
- `chord` - параллельность с обязательной синхронизацией и callback.
- `map/starmap/chunks` - инструменты массового запуска с разным уровнем пакетирования.

### Термины

| Термин | Кратко |
| --- | --- |
| **`.s()`** | Создание signature с аргументами. |
| **`.si()`** | Immutable signature: игнорирует автоподстановку результата предыдущего шага. |
| **`.set()`** | Добавление execution-опций к signature (queue, countdown и др.). |
| **`clone()`** | Копия сигнатуры с модификацией аргументов/опций. |
| **Canvas graph** | Граф зависимостей задач, который Celery разворачивает в запуск сообщений. |

### Теория и правила

Интуиция: Canvas похож на "конструктор pipeline".  
Каждый примитив отвечает за тип связи:

- "сначала A, потом B" -> `chain`;
- "A1..An одновременно" -> `group`;
- "A1..An одновременно, потом агрегировать" -> `chord`;
- "одна и та же функция на списке входов" -> `map/starmap/chunks`.

Формально:

1. Любой Canvas-объект в итоге превращается в набор task messages.
2. Оркестрация логически описывается в producer-коде, физически исполняется распределённо.
3. Ошибки, retry и state propagation зависят не только от задач, но и от топологии графа.

Практическая матрица выбора примитива:

| Примитив | Когда применять | Главный плюс | Главный риск |
| --- | --- | --- | --- |
| `chain` | Есть строгая зависимость шага B от результата шага A | Прозрачный последовательный поток | Один сбой останавливает хвост |
| `group` | Много независимых одинаковых/похожих операций | Хороший fan-out и масштабирование | Burst-нагрузка на broker/backend |
| `chord` | Нужен fan-out и затем единый fan-in итог | Естественная модель "параллельно -> агрегировать" | Чувствительность к backend и callback |
| `map/starmap` | Массовый запуск одной функции на коллекции | Компактная запись dispatch | Много сообщений при больших объёмах |
| `chunks` | Нужен баланс между fan-out и overhead | Снижает число сообщений | Неправильный batch size ухудшает retries |

Граничный случай: иногда команда пытается "решить всё одним примитивом".  
Например, строит очень длинный `chain`, хотя этапы независимы и лучше ложатся в `group` + явную агрегацию. Такое решение обычно ухудшает и скорость, и сопровождаемость.

#### Проверь себя по матрице выбора примитивов

1. Почему выбор примитива должен делаться по типу зависимости между шагами, а не по привычке команды?

<details><summary>Ответ</summary>

Потому что примитив задаёт модель исполнения: последовательную, параллельную или fan-in. Ошибка выбора ломает не только производительность, но и корректность бизнес-логики (например, потеря обязательной агрегации или лишняя сериализация длинного chain).

</details>

2. В каком случае `chord` избыточен по сравнению с `group`?

<details><summary>Ответ</summary>

Когда не нужен обязательный финальный шаг агрегации. Если достаточно обработать независимые задачи и получить их результаты отдельно, `group` проще и дешевле по orchestration.

</details>

### Пошагово

1. Определи бизнес-цель logical job.
2. Раздели её на шаги с явными входом/выходом.
3. Выбери связь между шагами (sequence, parallel, barrier).
4. Опиши граф Canvas.
5. Добавь correlation ID в заголовки и логи.
6. Проверь, что каждый шаг идемпотентен для возможного повторного запуска.

### Простыми словами

Canvas - это способ заранее описать "карту маршрута" задач.  
Как в навигаторе: ты задаёшь точки и правила перехода между ними, а не едешь вручную по каждому перекрёстку.

### Картинка в голове

Представь производство:

- `chain` - одна сборочная линия;
- `group` - несколько параллельных линий на разных станциях;
- `chord` - все линии должны завершиться, чтобы контролёр подписал итоговый акт;
- `chunks` - поставка деталей не поштучно, а ящиками.

### Как запомнить

Формула: **Signature -> Связь -> Наблюдаемость -> Надёжность**.  
Если забыть хотя бы один элемент, граф становится хрупким.

### Примеры

```python
from celery import chain, group, chord
from app.celery_app import celery_app

@celery_app.task
def parse_file(file_id: str) -> list[str]:
    return ["row1", "row2"]

@celery_app.task
def enrich_row(row: str) -> dict:
    return {"row": row, "ok": True}

@celery_app.task
def build_report(items: list[dict]) -> str:
    return f"processed={len(items)}"

workflow = chord(
    group(enrich_row.s("row1"), enrich_row.s("row2")),
    build_report.s()
)
```

Пример `partial signature` и `immutable signature`:

```python
@celery_app.task
def add(a: int, b: int) -> int:
    return a + b

@celery_app.task
def format_sum(value: int, prefix: str = "sum") -> str:
    return f"{prefix}:{value}"

# partial: фиксируем первый аргумент заранее
add_ten = add.s(10)

# immutable: не принимаем результат предыдущего шага автоматически
final_log = format_sum.si(prefix="final")

wf = chain(
    add_ten.s(5),              # -> 15
    format_sum.s(prefix="tmp"),# -> "tmp:15"
    final_log                  # immutable: не получит "tmp:15" как value
)
```

Почему это важно:

- `partial` помогает переиспользовать сигнатуры как заготовки;
- `immutable` защищает контракт шага от неявной подстановки результата chain.

#### Проверь себя по partial/immutable signatures

1. Что именно решает `partial signature` в проектировании workflow?

<details><summary>Ответ</summary>

Она позволяет подготовить переиспользуемую "заготовку вызова": часть аргументов фиксируется заранее, остальные подставляются в конкретном сценарии. Это снижает дублирование и делает композицию более управляемой.

</details>

2. Почему `.si()` важна в длинных цепочках?

<details><summary>Ответ</summary>

Потому что `.si()` запрещает неявную передачу результата предыдущего шага. Это защищает от случайной поломки контракта, когда шаг не должен принимать upstream output.

</details>

### Практика / реальные сценарии

- ETL: extract -> transform(parallel) -> load/report.
- Media: decode -> process chunks -> merge metadata.
- Финансы: вычисления по портфелю в параллели -> агрегирующий риск-отчёт.

### Типичные ошибки

- Путать `signature` с немедленным запуском.
- Не фиксировать единый correlation ID на весь graph.
- Смешивать в одной `group` задачи с радикально разным временем выполнения.

### Что будет, если...

- **...не учитывать result backend в сложных `chord`?**  
  Получишь задержки callback, пропуски агрегации или перегруз backend-хранилища.

### Проверь себя

1. Чем `signature` отличается от вызова `delay()`?

<details><summary>Ответ</summary>

`signature` описывает будущий вызов и может быть включён в граф; `delay()` инициирует немедленную отправку задачи.

</details>

2. Зачем нужен `chord`, если есть `group`?

<details><summary>Ответ</summary>

`group` запускает параллельно, но не даёт встроенного барьера "все завершились -> запусти финальный callback". `chord` добавляет этот fan-in шаг.

</details>

3. Почему `chunks` иногда лучше, чем огромный `map`?

<details><summary>Ответ</summary>

Потому что `chunks` уменьшают количество сообщений и снижают orchestration overhead при больших наборах входных данных.

</details>

### Запомните

- Canvas задаёт **архитектуру исполнения**, а не только удобный синтаксис.
- Выбор примитива влияет на надёжность, цену и диагностику.
