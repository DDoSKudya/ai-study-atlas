## 10.2. Chain

### Цель раздела

Научиться строить предсказуемые последовательные workflow через `chain`, корректно передавать данные между шагами и обрабатывать сбои в середине цепочки.

### В этом разделе главное

- В `chain` результат шага автоматически передаётся в следующий.
- Не каждый следующий шаг должен принимать этот результат: для этого есть immutable signature (`.si()`).
- Падение одного звена обычно останавливает цепочку.
- Диагностика цепочки требует сквозной корреляции, а не только отдельных `task_id`.

### Термины

| Термин | Кратко |
| --- | --- |
| **Result propagation** | Автоматическая передача результата по цепочке вперёд. |
| **Immutable signature (`.si`)** | Сигнатура без автоподстановки результата предыдущего шага. |
| **Chain short-circuit** | Остановка дальнейших шагов при ошибке в одном из звеньев. |
| **Compensating action** | Действие для восстановления бизнес-состояния после частично успешной цепочки. |

### Теория и правила

Интуиция: `chain` - это конвейер. Если станция N сломалась, станция N+1 не сможет продолжить.

Формально:

- `chain(a.s(), b.s(), c.s())` означает, что `b` получит результат `a`, а `c` - результат `b`;
- если `b` падает и исключение не обработано, `c` не стартует;
- если шаг не должен получать предыдущий результат, используй `b.si(...)`.

Ключевое правило: проектируй каждый шаг как **контракт**:

1. входные данные (тип/структура),
2. ожидаемый выход,
3. доменный эффект,
4. поведение при повторном запуске.

Граничный случай для длинных `chain`: рост payload между шагами.  
Если каждый шаг "наращивает" результат тяжёлыми данными, система начинает тратить заметную долю времени на сериализацию и транспорт, а не на полезную логику.

Практический паттерн:

- передавать между шагами компактные идентификаторы/ссылки;
- крупные артефакты хранить во внешнем storage;
- в `chain` передавать pointer и метаданные вместо больших blob-объектов.

#### Проверь себя по payload в chain

1. Почему "богатый" результат каждого шага может замедлить всю цепочку даже при быстрых функциях?

<details><summary>Ответ</summary>

Потому что растёт стоимость сериализации, передачи через broker и десериализации на каждом шаге. В итоге система тратит ресурсы не на вычисление, а на транспорт данных.

</details>

2. Какой безопасный паттерн для больших данных между шагами?

<details><summary>Ответ</summary>

Сохранять тяжёлый артефакт во внешнем хранилище, а между шагами передавать только ссылку/ID и минимальные метаданные. Это уменьшает размер сообщений и делает retry дешевле.

</details>

### Пошагово

1. Определи минимальный полезный набор шагов.
2. Зафиксируй схему данных между шагами.
3. Для служебных или независимых шагов используй `.si()`.
4. Добавь обработку ошибок: retry только для transient-классов.
5. Добавь журнал шагов по `logical_job_id`.

#### Диагностика упавшей цепочки (runbook)

Когда `chain` "оборвалась", важно не гадать, а идти по воспроизводимому сценарию:

1. Найди `logical_job_id` и список `task_id` по шагам цепочки.
2. Определи первое упавшее звено (а не "последнюю видимую ошибку").
3. Раздели причину на класс: transient / permanent / data / code.
4. Проверь, какие шаги точно применили побочный эффект.
5. Реши, что безопаснее: replay конкретного шага или компенсация.

Минимальные команды для triage:

```bash
celery -A app inspect active
celery -A app inspect reserved
celery -A app inspect revoked
```

И программная проверка состояния звена:

```python
from celery.result import AsyncResult

res = AsyncResult(task_id, app=celery_app)
print(res.state, res.result)
```

Смысл этого runbook: сначала восстановить точную картину фактов, и только потом "чинить".  
Иначе высокий риск повторно применить уже выполненный шаг.

#### Проверь себя по runbook chain

1. Почему важно находить первое упавшее звено, а не последнее сообщение об ошибке?

<details><summary>Ответ</summary>

Потому что последняя ошибка часто является следствием. Источник проблемы находится в первом нарушении контракта/данных/доступности, и именно его надо чинить, чтобы не получить повторный сбой.

</details>

2. Почему перед replay нужно проверять уже применённые побочные эффекты?

<details><summary>Ответ</summary>

Чтобы не продублировать бизнес-эффект (например, платёж, уведомление, запись во внешний сервис). Replay без проверки состояния может усугубить инцидент.

</details>

### Простыми словами

`chain` хорош там, где "следующий шаг бессмысленен без предыдущего".  
Если шаги не зависят друг от друга - вероятно, нужен `group`, а не длинная цепочка.

### Картинка в голове

ASCII-модель:

```text
[Step A] --result--> [Step B] --result--> [Step C]
    |                    |
  fail? stop here      fail? stop here
```

### Как запомнить

**Chain = "передай дальше или остановись".**

### Примеры

```python
from celery import chain

@celery_app.task
def load_order(order_id: str) -> dict:
    return {"order_id": order_id, "amount": 1200}

@celery_app.task
def validate_order(order: dict) -> dict:
    if order["amount"] <= 0:
        raise ValueError("amount must be > 0")
    return order

@celery_app.task
def persist_order(order: dict) -> str:
    # идемпотентная запись по unique(order_id)
    return f"saved:{order['order_id']}"

@celery_app.task
def notify_ops() -> None:
    # не нужен результат предыдущего шага
    return None

wf = chain(
    load_order.s("ord-1"),
    validate_order.s(),
    persist_order.s(),
    notify_ops.si()  # immutable: не принимает "saved:ord-1"
)
wf.apply_async()
```

### Практика / реальные сценарии

- Обработка заказа: проверить -> сохранить -> инициировать доставку.
- Документы: распарсить -> валидировать -> загрузить в storage -> обновить индекс.
- Интеграция: забрать данные из API -> нормализовать -> записать в БД.

### Типичные ошибки

- Длинные `chain` из десятков шагов без компенсаций и без чекпоинтов.
- Неявные контракты входа/выхода между шагами.
- Смешивание "тяжёлых" и "лёгких" шагов без разделения очередей.

### Что будет, если...

- **...в середине цепочки шаг неидемпотентен и ретраится?**  
  Возможны дубли бизнес-эффекта в этом шаге, даже если вся цепочка логически "одна".

### Проверь себя

1. Когда вместо `.s()` в `chain` уместнее `.si()`?

<details><summary>Ответ</summary>

Когда шаг логически независим от результата предыдущего шага и не должен получить его автоматически.

</details>

2. Почему длинная цепочка без checkpoint-state опасна?

<details><summary>Ответ</summary>

При падении в позднем шаге трудно понять, что уже применено и что можно безопасно переиграть. Растёт риск неправильных компенсаций.

</details>

3. Как диагностировать "где именно упала цепочка"?

<details><summary>Ответ</summary>

Использовать сквозной `logical_job_id`, структурированные логи по шагам и статусы задач по каждому звену, а не только один финальный статус.

</details>

### Запомните

- `chain` - лучший выбор для последовательных зависимостей.
- Контракты данных и идемпотентность шагов критичнее "красоты" цепочки.
