## 10.5. Map, starmap, chunks

### Цель раздела

Научиться использовать batch-dispatch примитивы так, чтобы уменьшать технический overhead и сохранять управляемый уровень параллелизма.

### В этом разделе главное

- `map` - передаёт по одному аргументу из коллекции;
- `starmap` - передаёт кортежи аргументов;
- `chunks` - группирует элементы в партии, уменьшая число сообщений;
- компромисс всегда между "мелко и гибко" vs "крупно и дёшево по orchestration".

### Термины

| Термин | Кратко |
| --- | --- |
| **Dispatch overhead** | Цена одной отправки задачи: сериализация, broker IO, bookkeeping. |
| **Batch size** | Размер партии элементов на одну задачу. |
| **Tail latency** | Время "дотягивания хвоста" последних задач. |
| **Work unit** | Минимальная полезная работа одной задачи. |

### Теория и правила

Интуиция: если полезной работы мало, а задач слишком много, система тратит ресурсы на "доставить и учесть", а не на бизнес-логику.

Правила:

1. Для лёгких операций увеличивай batch size (`chunks`).
2. Для тяжёлых независимых операций делай меньшие партии.
3. Подбирай размер партии экспериментально по метрикам latency/throughput/cost.

Ориентир по выбору `map/starmap/chunks`:

| Условие | Предпочтительнее | Почему |
| --- | --- | --- |
| Один позиционный аргумент, умеренный объём | `map` | Самый простой и читаемый вариант |
| Несколько аргументов на элемент | `starmap` | Явно выражает структуру входа |
| Очень большой объём и мелкая работа | `chunks` | Снижает число сообщений и overhead |
| Payload на элемент крупный | Меньшие `chunks` или обычные задачи | Иначе риск больших сообщений и тяжёлых повторов |

#### Проверь себя по map/starmap/chunks

1. В чём основной критерий выбора между `map` и `chunks`?

<details><summary>Ответ</summary>

Стоимость orchestration на один элемент. Если работа на элемент очень лёгкая и overhead высок, `chunks` обычно выгоднее; если работа существенная и нужна гибкость, `map` может быть удобнее.

</details>

2. Почему слишком крупный `chunks` увеличивает риск дорогого retry?

<details><summary>Ответ</summary>

При ошибке переигрывается большая порция данных за один раз. Это увеличивает время повторного выполнения, вероятность таймаутов и нагрузку на зависимости.

</details>

### Пошагово

1. Измерь среднее время одной операции.
2. Оцени стоимость dispatch на твоём стеке.
3. Выбери 2-3 кандидата размера batch.
4. Прогоняй нагрузочный тест на реальном broker/backend.
5. Зафиксируй дефолт и guardrails в конфиге.

### Простыми словами

`map/starmap/chunks` - это про "как упаковать поток работ".  
Если упаковать плохо, потеряешь время на логистике.

### Картинка в голове

```text
Мелкая упаковка: [a][b][c][d][e][f] -> много маленьких доставок
Крупная упаковка: [a b c][d e f]     -> меньше доставок, но крупнее задачи
```

### Как запомнить

**Не оптимизируй "задачи", оптимизируй "единицу полезной работы на одно сообщение".**

### Примеры

```python
@celery_app.task
def score_one(user_id: int) -> int:
    return user_id * 2

# map: один аргумент
score_one.map([1, 2, 3, 4]).apply_async()

@celery_app.task
def combine(a: int, b: int) -> int:
    return a + b

# starmap: набор кортежей
combine.starmap([(1, 2), (10, 20), (3, 7)]).apply_async()

# chunks: упаковать по 100 элементов в задачу
score_one.chunks(zip(range(1, 1001)), 100).apply_async()
```

### Практика / реальные сценарии

- Массовая рекалькуляция признаков для пользователей.
- Проверка валидности пакета документов.
- Переобогащение данных после миграции справочника.

### Типичные ошибки

- `chunk` слишком большой -> долгие задачи, тяжёлые retries.
- `chunk` слишком маленький -> broker/backend перегружены overhead.
- Пытаться выбрать размер партии "на глаз", без замеров.

### Что будет, если...

- **...в chunk из 1000 элементов ошибка на 999-м?**  
  Ретрай может повторить почти весь объём работы, если нет внутреннего чекпоинта/идемпотентности по элементам.

### Проверь себя

1. Когда `chunks` обычно полезнее `map`?

<details><summary>Ответ</summary>

Когда операция на элемент очень лёгкая и overhead отправки/учёта отдельной задачи становится существенным.

</details>

2. Почему слишком крупный batch может ухудшить надёжность?

<details><summary>Ответ</summary>

При сбое приходится переигрывать большую порцию работы, растёт длительность выполнения и риск таймаутов.

</details>

3. Что именно нужно измерять при подборе batch size?

<details><summary>Ответ</summary>

Throughput, latency (включая хвост), долю ошибок/ретраев, нагрузку на broker/backend и стоимость ресурсов.

</details>

### Запомните

- `map/starmap/chunks` - это инструменты упаковки, а не "магическое ускорение".
- Балансируй batch size по данным и замерам.
