## Краткая шпаргалка по терминам

| Термин | Суть простыми словами |
|--------|------------------------|
| **Поверхность атаки** | Все места, куда **можно подать ввод** или **достучаться до системы** (сеть, API, очередь, логи). |
| **Threat model** | Явное описание: **кто** атакует, **что** защищаем, **какие** сценарии реалистичны. |
| **Treating the broker as trusted LAN** | Анти-паттерн: «раз у нас VPC, можно pickle». **Сеть — не замена** политикам. |
| **Pickle** | Бинарный формат Python, который при загрузке **может выполнить произвольный код** — **недопустим** для недоверенных сообщений. |
| **JSON serializer** | Текстовый обмен структурами; **не** исполняет код при парсинге, но **не отменяет** бизнес-логических атак. |
| **`accept_content`** | Whitelist типов контента, которые worker **согласен** принимать — страховка от **смешения** форматов. |
| **`result_serializer`** | Формат **значения** в result backend (return, ошибка); отдельно от тела задачи в брокере — см. **§17.2г**. |
| **`result_accept_content`** | Whitelist типов при **чтении** результата из backend (в ряде версий Celery — отдельно от `accept_content`; сверьте доку). |
| **Schema / contract** | Соглашение о полях payload, версиях и допустимых значениях — **контракт** между producer и consumer. |
| **TLS** | Шифрование и аутентификация транспорта «по проводу» между клиентами и брокером/backend. |
| **Least privilege** | Учётка брокера имеет **минимально необходимые** права (очереди, exchange, не админ). |
| **vhost / namespace** | Логический изолятор в RabbitMQ; в Redis — **отдельные instance/DB/префиксы** по политике. |
| **PII** | Персональные данные; в Celery часто **утекают** через аргументы, результаты и traceback. |
| **Redaction** | Удаление/замена чувствительных полей **до** записи в лог или метрики. |
| **Retention** | Как долго **хранятся** результаты/метаданные; влияет на **риск утечки** и объём. |
| **Noisy neighbor** | Один тенант/класс задач **забивает** очереди и worker-ы, ухудшая SLO соседей. |
| **Ingress rate limit** | Ограничение **скорости постановки** задач с публичного API — против **флуда** и DoS. |
| **Chord bomb** | Запрос пользователя порождает **огромный** chord/group и **взрыв** нагрузки на брокер/backend. |

#### Проверь себя: шпаргалка

1. Чем **`accept_content`** принципиально отличается от **валидации** полей payload в задаче?

<details><summary>Ответ</summary>

`accept_content` — **whitelist типа сериализации** на уровне протокола Celery/Kombu (какие `content-type` worker **вообще** согласен десериализовать). Валидация — **бизнес-проверка** уже распарсенных данных (идентификаторы, права, диапазоны). Можно отвергнуть pickle на входе и всё равно принять **опасный** JSON без проверки смысла.

</details>

2. Почему **TLS** в шпаргалке стоит рядом с **брокером и backend**, а не только с «внешним интернетом»?

<details><summary>Ответ</summary>

Потому что оба URL (`CELERY_BROKER_URL` и `CELERY_RESULT_BACKEND`) — **отдельные** сетевые цели; трафик к ним может пересекать **сегменты**, зеркала, скомпрометированные поды. Шифрование и проверка сертификата нужны **по пути к каждому** сервису, а не «раз включили где-то».

</details>

3. Как **Chord bomb** связан с **ingress rate limit**, если взрыв идёт через **один** HTTP-запрос?

<details><summary>Ответ</summary>

Один запрос может инициировать **публикацию тысяч** сообщений или тяжёлый canvas на стороне сервера. Rate limit по **числу запросов** не спасает, если не ограничены **fan-out**, размер batch и **серверные** потолки на N (**§17.6**). Нужны **квоты на продуктовую операцию**, а не только на «запросов в секунду».

</details>
