## Справочник по части

| Команда/группа | Когда использовать | Главный риск | Базовая проверка |
|---|---|---|---|
| `worker` | запуск/тюнинг исполняющего контура | неверный app path и перегрузка пула | `inspect registered`, `inspect stats` |
| `beat` | периодические публикации | дубли из-за multiple schedulers | контроль single-active-instance |
| `inspect` | диагностика состояния | неверная интерпретация данных | сравнить `active/reserved/scheduled` в динамике |
| `control` | управляемое вмешательство | массовый побочный эффект | scope по `-d`, post-check |
| `events` | наблюдение event-потока | шум/нагрузка без цели | ограничить использование под сценарий |
| `revoke` | точечная отмена задач | отмена не того task_id | сверка task_id и состояния |
| `terminate` | аварийное прерывание running-задачи | частичные side effects | компенсационный план |
| `purge` | массовая очистка очередей | потеря задач | бизнес-аппрув + инцидентный протокол |
| `report` | сбор среды/версий | утечка чувствительных данных в логах | sanitize вывод при публикации |

### Быстрый runbook-индекс: команда -> что ожидать -> если не так

| Команда | Что ожидаем увидеть | Если не так, куда смотреть первым |
|---|---|---|
| `inspect stats` | ответы от целевых worker-ов | сеть, broker-доступ, TLS, worker hostname |
| `inspect registered` | нужная задача есть в списке | импорты задач, app init, autodiscovery |
| `inspect active/reserved` | адекватное распределение очереди | `-Q`, pool/concurrency, prefetch/QoS |
| `control rate_limit ... -d ...` | изменение лимита на целевом worker-е | корректность `-d`, права control, канал broadcast |
| `beat` со state file | стабильные периодические публикации | права к файлу, целостность файла, single-instance |
| `report` | фактические версии и конфиг | несовпадение окружений, неверный app target |

### Preflight и postflight для операционных команд

| Тип операции | Preflight (до команды) | Postflight (после команды) |
|---|---|---|
| `inspect` | проверить целевой app/окружение | сохранить вывод для сравнения в динамике |
| `control` scoped | проверить `-d`, убедиться в допустимости по runbook | повторный `inspect` + проверка метрик/очереди |
| `revoke/terminate` | проверить точный `task_id` и состояние задачи | подтвердить остановку, оценить side effects |
| `purge` | бизнес-аппрув, оценка объема потерь, план replay | подтвердить пустую очередь, зафиксировать инцидент |

#### Проверь себя: справочник и runbook-индекс

1. Зачем держать отдельно и «быстрый runbook-индекс», и таблицу preflight/postflight?

<details><summary>Ответ</summary>

Индекс отвечает «куда смотреть при симптоме», а preflight/postflight — «как безопасно выполнить действие». Вместе они закрывают и диагностику, и дисциплину выполнения.

</details>

2. Какой самый опасный анти-паттерн при использовании справочника?

</details>
