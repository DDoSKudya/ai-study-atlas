## Краткая шпаргалка по терминам

| Термин                                | Простыми словами                                                                                                                          |
| ------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------- |
| **Execution environment**             | Конкретные условия запуска задач: ОС, Python, контейнер, лимиты, часы.                                                                    |
| **PID 1 problem**                     | Процесс в контейнере как PID 1 иначе обрабатывает сигналы и reap дочерних процессов.                                                      |
| **cgroups**                           | Механизм Linux для ограничения CPU/памяти/процессов контейнера.                                                                           |
| **OOMKill**                           | Принудительное завершение процесса ядром при нехватке памяти.                                                                             |
| **CPU throttling**                    | Искусственное замедление процесса из-за исчерпания CPU quota.                                                                             |
| **Fork safety**                       | Корректность работы библиотек после `fork` worker-процесса.                                                                               |
| **ABI compatibility**                 | Совместимость бинарных пакетов и интерпретатора/ОС.                                                                                       |
| **Locale**                            | Набор региональных правил: кодировки, формат дат/чисел, сортировка.                                                                       |
| **TZ / timezone**                     | Правила локального времени поверх UTC.                                                                                                    |
| **DST drift**                         | Сдвиги поведения расписаний при переходах летнего/зимнего времени.                                                                        |
| **Monotonic clock**                   | Часы, которые только растут и не зависят от корректировки системного времени.                                                             |
| **Graceful shutdown**                 | Мягкая остановка с завершением/перепостановкой задач без потерь.                                                                          |
| **`worker_max_memory_per_child`**     | Лимит памяти на дочерний pool-процесс (Celery), чтобы ротировать "тяжелые" процессы.                                                      |
| **`statedb` / `celerybeat-schedule`** | Файл/хранилище состояния beat-расписания (нужен writable path в контейнере).                                                              |
| **Read-only rootfs**                  | Контейнерный режим, когда образ монтируется read-only, а запись идет только в volume/tmp.                                                 |
| **WSL2**                              | Windows Subsystem for Linux: Linux-ядро внутри Windows (часто удобнее, чем "нативный" Windows).                                           |
| **NUMA**                              | Non-Uniform Memory Access: у CPU может быть "своя" память; важно на больших серверах.                                                     |
| **CPU pinning**                       | Закрепление процесса/потока на конкретных CPU для стабильного perf.                                                                       |
| **NTP / chrony**                      | Синхронизация wall clock на хосте; влияет на корреляцию логов и иногда на тайминг.                                                        |
| **`TZ` (env var)**                    | Быстрый способ задать timezone для многих Unix-программ; легко "перебить" настройки приложения.                                           |
| **`zoneinfo` / IANA**                 | Канонические имена зон (например, `Europe/Moscow`); осторожно с "смещениями" типа `UTC+3`.                                                |
| **`runAsNonRoot` / `fsGroup`**        | В k8s: процесс не root → запись в volume должна быть разрешена через `fsGroup`/права; иначе beat `-s` и `/tmp` дадут `Permission denied`. |
| **DB-backed schedule**                | Расписание/last-run в БД (часто `django-celery-beat`, части `11/18`) вместо файла `celerybeat-schedule` на диске.                         |

#### Проверь себя: шпаргалка

1. Чем **Execution environment** шире, чем «версия Python в образе», и почему в одной таблице рядом стоят **cgroups** и **`worker_max_memory_per_child`**?
2. Почему `OOMKill` почти всегда выглядит как "случайная нестабильность Celery", если не смотреть системные события?
3. В чем смысл monotonic clock для retries/таймаутов; и чем **read-only rootfs** для beat отличается от **OOMKill** на worker-е по симптомам в очереди?

<details><summary>Ответ</summary>

1. Execution environment включает ОС, лимиты, оркестратор, часы и локаль — всё, что меняет поведение без смены исходника задачи. В таблице рядом стоят внешний предел cgroup и внутренняя ротация child в Celery; путать слои — типичный неверный triage. Отдельно: **`TZ` (env)** задаёт окружение быстро, но легко конфликтует с приложением; **`zoneinfo`/IANA** — канон для бизнес-времени и beat, смещениями «в голове» не заменяются.
2. Потому что процесс исчезает внезапно, а в логах задачи это часто выглядит как обрыв исполнения без очевидного стека.
3. Monotonic clock не «отматывается» при NTP/DST и подходит для интервалов ожидания. Read-only чаще даёт детерминированные ошибки записи при старте/сохранении state beat; OOMKill — внезапная смерть под нагрузкой (часто exit 137), ближе к `WorkerLost`/лавине retries.

</details>
