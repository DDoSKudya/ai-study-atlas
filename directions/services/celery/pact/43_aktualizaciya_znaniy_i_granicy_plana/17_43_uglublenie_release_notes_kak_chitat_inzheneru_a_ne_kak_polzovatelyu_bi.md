## Углубление: release notes — как читать инженеру, а не «как пользователю библиотеки»

### Цель блока

Сделать из чтения changelog **управление риском**: классифицировать изменения, связать с вашими used features, вынести в тикет **наблюдаемые** проверки.

### Теория: типы записей в release notes

| Тип записи | Что это значит | Действие команды |
| ---------- | -------------- | ---------------- |
| **Breaking** | Старое поведение сознательно сломано | План миграции, dual-read, feature flag |
| **Deprecated** | Пока работает, но будет удалено | Техдолг с дедлайном, поиск в коде |
| **Security** | Уязвимость или усиление модели доверия | Приоритетный патч, возможно вне мажорного цикла |
| **Performance** | Быстрее/меньше аллокаций | Проверить, не изменилась ли косвенно **семантика** (редко, но бывает) |
| **Bugfix** | Исправление краевого случая | Оценить: ваш сценарий совпадает? добавить регрессионный тест |

### Пошагово: разбор одного релиза за 30 минут

1. **Скопировать** список изменений в заметку (удобно подсвечивать).
2. **Отфильтровать** по ключевым словам вашего стека: `redis`, `amqp`, `sqs`, `chord`, `beat`, `serialization`, `json`, `pickle`, `security`, `fork`, `billiard`.
3. Для каждого совпадения поставить метку: **нас касается / не касается / не уверены**.
4. Для «касается» написать **наблюдаемый критерий**: метрика, тест, ручной сценарий.
5. Для «не уверены» назначить **владельца расследования** на 1–2 часа, иначе пункт зависнет вечно в сером статусе.

### Типичные ошибки

- Читать только заголовки релиза без **подпунктов**.
- Не открывать связанный PR/issue — там часто **объяснение мотивации**, без которого непонятно, опасно ли вам.

### Проверь себя

1. Чем **security**-запись отличается по процессу от обычного **bugfix**?
2. Зачем в шаге 3 вводить явный статус **«не уверены»**, а не молча относить в «не касается»?
3. Почему «performance» иногда требует **функциональной** проверки, а не только бенчмарка throughput?

<details><summary>Ответ</summary>

1. Security обычно требует **ускоренного** патча и иногда внепланового релиза; bugfix можно отнести к обычному циклу, если не попадает в ваш сценарий.
2. Потому что «не уверены» — главный источник **скрытого** риска; без владельца и таймбокса он превращается в долг.
3. Потому что оптимизации иногда меняют порядок операций, тайминги, flush буферов — и проявляются как **редкие гонки**, которые throughput-тест не ловит.

</details>
