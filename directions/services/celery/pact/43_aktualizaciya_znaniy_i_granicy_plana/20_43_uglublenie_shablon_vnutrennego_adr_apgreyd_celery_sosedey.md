## Углубление: шаблон внутреннего ADR «Апгрейд Celery / соседей»

**ADR** (Architecture Decision Record) — короткая запись «мы решили X, потому что Y, цена Z». Ниже — **скелет**, который команда копирует в свой Confluence/Markdown.

### Поля шаблона

1. **Контекст:** текущие версии, зачем апгрейд (security, фича, EOL Python).
2. **Used features:** список из 43.1/43.2.
3. **Анализ рисков:** таблица «изменение → касается нас → как проверим».
4. **План rollout:** canary, порядок обновления producer/worker/beat, drain очередей.
5. **План отката:** версии образов, совместимость сообщений, что делаем с «застрявшими» новыми сообщениями.
6. **Наблюдаемость:** какие метрики/логи — **до/после**.
7. **Решение:** принято/отложено, дата пересмотра.

### Пример заполнения (вымышленный, краткий)

- **Контекст:** поднять Celery из-за security advisory; Python остаётся тем же.
- **Used features:** Redis broker + JSON serializer + chord + beat crontab.
- **Риск:** изменения в обработке chord на конкретной паре backend/redis-py → интеграционный тест + метрика latency backend.
- **Rollout:** 10% worker pool canary 24 часа, затем 50/100.
- **Откат:** вернуть образ + **freeze** publish новых сообщений нового формата (если менялся payload) — отсылка к части **30**.

### Проверь себя

1. Зачем в ADR явно писать **used features**, если «всё и так в коде видно»?
2. Почему **план отката** без пункта про сообщения в очереди часто неполон?

<details><summary>Ответ</summary>

1. Потому что код читают долго, а ADR — **сжатая карта риска** для ревью и для людей вне сервиса; used features задают фокус чтения changelog.
2. Потому что откат кода без стратегии сообщений оставляет в брокере **несовместимые** payload для старых воркеров — классический провал миграции.

</details>
