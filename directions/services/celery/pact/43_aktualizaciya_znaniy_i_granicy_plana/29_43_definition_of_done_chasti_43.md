## Definition of Done части 43

Этот блок нужен, чтобы понять: часть 43 **действительно внедрена** в командную работу, а не просто прочитана.

| Критерий | Как проверить | Статус |
| -------- | ------------- | ------ |
| Есть документ «Источники истины» (43.1) | В нём есть ссылки на docs/release notes/broker docs/issues | `yes/no` |
| Есть минимум один заполненный `Issue digest` | В тикете апгрейда лежит разбор хотя бы 1 релевантного issue | `yes/no` |
| Есть актуальный runbook triage (43.2) | Прописан маршрут «минимальный клиент брокера → класс причины» | `yes/no` |
| Есть хотя бы один свежий drift log (43.3) | Дата не старше квартала, есть delta по 36/37 | `yes/no` |
| Есть ADR по последнему апгрейду | Зафиксированы риски, rollout, rollback, дата решения | `yes/no` |
| Есть назначенные роли (RACI) | Видно, кто R/A/C/I для сверки и апгрейда | `yes/no` |
| Есть формулировка границ для стейкхолдеров (43.4) | Используется в архитектурном обзоре/процессе релиза | `yes/no` |

### Быстрая самодиагностика зрелости

- **0–2 `yes`:** знания в основном теоретические, высок риск повторяемых инцидентов.
- **3–5 `yes`:** процесс частично работает, но хрупок при смене людей/приоритетов.
- **6–7 `yes`:** часть 43 внедрена как эксплуатационная практика, а не как учебный конспект.

#### Проверь себя: шкала зрелости DoD

1. Почему переход с уровня `3–5 yes` к `6–7 yes` обычно требует не новых документов, а дисциплины исполнения?
2. Что означает ситуация «все пункты yes», но инциденты всё равно происходят?
3. Какой один индикатор показывает, что DoD поддерживается, а не был заполнен один раз?

<details><summary>Ответ</summary>

1. Потому что базовые артефакты уже есть; сложность в регулярном обновлении, связи с релизами и реальном использовании в решениях.
2. Это нормальная реальность распределённых систем: DoD снижает хаос и MTTR, но не отменяет сам факт сбоев.
3. Свежие даты/ссылки в drift log, ADR и runbook после каждого значимого изменения, а не статичные записи годовой давности.

</details>

#### Проверь себя: DoD

1. Почему наличие только ADR без drift log недостаточно для DoD?
2. Какой пункт DoD обычно самый «дешёвый» для старта и почему?
3. Если все пункты `yes`, означает ли это «инцидентов не будет»?

<details><summary>Ответ</summary>

1. ADR закрывает событие апгрейда, а drift log ловит медленный регулярный дрейф; без него знание всё равно устаревает между релизами.
2. Часто это «документ источников истины» + базовый triage runbook: быстро сделать, сразу уменьшает хаос в диагностике.
3. Нет. Это означает высокую **управляемость** рисков и быструю реакцию, а не магическую гарантию отсутствия сбоев.

</details>
