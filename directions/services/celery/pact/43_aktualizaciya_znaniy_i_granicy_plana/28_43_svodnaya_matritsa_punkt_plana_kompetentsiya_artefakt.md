## Сводная матрица: пункт плана → компетенция → артефакт

| Пункт плана | Какую компетенцию закрывает процесс | Что положить в репозиторий/тикет |
| ----------- | ----------------------------------- | -------------------------------- |
| **43.1** официальная дока | Умение **ориентироваться** в актуальной модели продукта | Список разделов docs + дата прогона |
| **43.1** release notes | Умение **классифицировать риск** апгрейда | Таблица «запись → касается нас → критерий проверки» |
| **43.1** дока брокера | Понимание **семантики доставки** на вашем транспорте | Список used features брокера |
| **43.1** issues | Умение работать с **неопределённостью** | Список ссылок + workaround + версия фикса |
| **43.2** смежный контур | **Граница ответственности** и быстрый triage | Runbook «сеть/TLS/IAM» |
| **43.3** квартал | **Анти-дрейф** имён настроек и CLI | Drift log |
| **43.3** мажор | **Анти-ломание контракта** сообщений/hooks | ADR + интеграционные тесты + canary |
| **43.4** полнота | **Коммуникация** ожиданий | Формулировка в архитектурном обзоре сервиса |

#### Проверь себя: матрица

1. Зачем в одной строке **и** компетенция, **и** артефакт — разве не дублируют друг друга?
2. Какой артефакт из таблицы **самый дешёвый** по времени внедрения, но даёт высокий эффект?
3. Что будет, если есть только ADR без drift log?

<details><summary>Ответ</summary>

1. Компетенция — *что умеет человек*; артефакт — *доказательство, что умение не утонуло в голове одного человека*; для команды важны оба слоя.
2. Часто это **список used features брокера** и минимальный **runbook triage**: дёшево и резко снижает ложную локализацию проблем.
3. ADR обычно **событийный** (вокруг апгрейда), а drift ловит **медленный** дрейф между событиями; без drift знания про CLI/имена всё равно «ползут» незаметно.

</details>
