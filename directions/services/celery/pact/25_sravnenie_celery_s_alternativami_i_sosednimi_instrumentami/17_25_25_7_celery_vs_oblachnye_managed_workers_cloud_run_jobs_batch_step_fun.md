## 25.7 Celery vs облачные managed workers (Cloud Run Jobs, Batch, Step Functions)

### Цель раздела

Научиться считать не только фичи, но и экономику/операции managed-подхода.

### Термины

| Термин | Коротко |
|---|---|
| **Managed workers** | Платформенные механизмы запуска фоновых задач с меньшей self-hosted эксплуатацией. |
| **Cold start** | Задержка запуска, особенно при scale-to-zero. |
| **Legacy network reachability** | Доступ облака к внутренним/унаследованным on-prem ресурсам. |

### Теория и правила

Managed-решения дают:

- меньше инфраструктурной рутины;
- быстрый запуск для cloud-native проектов;
- понятные сервисные границы.

Но часто есть компромиссы:

- cold start;
- ограничения сети и интеграции с legacy;
- vendor lock-in и особенности тарификации.

Мини-модель стоимости для сравнения:

`Итоговая стоимость = compute + сеть + storage/logging + операционное сопровождение команды + стоимость инцидентов`

Ошибка многих команд: считать только compute.

### Mermaid-диаграмма гибридного сценария

```mermaid
flowchart LR
    A[On-prem core system] --> B[Celery workers on-prem]
    C[Cloud trigger/event] --> D[Managed job runner]
    D -->|API call| A
    B -->|status/events| E[Unified observability]
    D -->|logs/metrics| E
```

### Практический кейс

**Сценарий:** часть задач должна работать рядом с legacy БД в закрытой сети, часть — эпизодические compute-heavy операции в облаке.  
**Решение:** гибрид: Celery для on-prem критичных операций + managed jobs для burst-нагрузки.

Дополнительный кейс:

**Сценарий:** продукт полностью cloud-native, задачи короткие, редкие, без жесткого low-latency SLA.  
**Решение:** managed jobs может быть экономически проще, чем отдельный Celery-контур.

### Что будет если...

Если полностью мигрировать в managed-подход без учета сетевых ограничений:

- часть задач может стать недоступной к нужным ресурсам;
- latency/cold start ударит по SLA;
- стоимость может неожиданно вырасти при burst-шаблонах.

### Production-рекомендации

- проведи пилот с реальным профилем нагрузки (а не синтетикой);
- измерь p50/p95/p99 запусков (включая cold start);
- проверь сетевую достижимость legacy-систем и требования комплаенса до миграции.

### Проверь себя

1. Почему managed не всегда дешевле self-hosted Celery?

<details><summary>Ответ</summary>

Потому что стоимость зависит от профиля нагрузки, частоты запусков, холодных стартов, сетевых требований и сопутствующих облачных сервисов.

</details>

2. Когда гибрид "Celery + managed" оправдан?

<details><summary>Ответ</summary>

Когда часть workload требует локальной близости к legacy/закрытым ресурсам, а часть выгодно запускать эластично в облаке.

</details>
