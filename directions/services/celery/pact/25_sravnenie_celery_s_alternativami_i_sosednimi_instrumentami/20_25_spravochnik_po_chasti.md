## Справочник по части

| Тема | Ключевая мысль | Практический вывод |
|---|---|---|
| Celery vs cron | Планировщик не равен распределенной очереди | Для локальных задач хватит cron, для распределенных — Celery |
| Celery vs Python-alternatives | Простота и функциональность балансируются | Выбирай по зрелости требований, а не по моде |
| Celery vs orchestration engines | Разные классы problem-space | Для DAG/workflow может быть нужен отдельный движок |
| Celery vs stream processing | Task queue != event stream log | Потоковые задачи требуют stream-инструментов |
| Celery vs managed cloud workers | Меньше self-hosted рутины, но есть компромиссы | Считай стоимость, cold start, сеть и lock-in |
| Когда Celery лучший | Python-фоновые задачи + умеренная orchestration | Отличный рабочий выбор при правильной эксплуатации |
| Когда Celery спорный | Чужой класс задачи и жесткие требования | Лучше сменить инструмент, чем наращивать костыли |

#### Проверь себя: справочник

1. Как использовать справочник в ежедневной работе архитектора?

<details><summary>Ответ</summary>

Как быструю карту решения: сначала определить класс задачи и риск, затем перейти к соответствующему разделу за подробным обоснованием и практикой.

</details>

2. Почему справочник не должен заменять полный текст разделов?

<details><summary>Ответ</summary>

Потому что он дает только сжатые ориентиры. Для корректного решения нужны детали: ограничения, граничные случаи, ошибки и сценарии применения.

</details>
