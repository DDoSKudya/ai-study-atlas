## 25.8 Когда Celery - лучший выбор

### Цель раздела

Сформулировать позитивные критерии выбора Celery.

### В этом разделе главное

Celery особенно хорош, когда нужно надежно и гибко исполнять фоновые Python-задачи в production.

### Чеклист "да, Celery подходит"

- Python-heavy стек и команда, уверенная в backend-практиках;
- много фоновых задач: уведомления, отчеты, интеграции, batch;
- нужны retries, очереди по SLA, роутинг и умеренная orchestration;
- есть готовность к операционной дисциплине (наблюдаемость, runbooks, настройка воркеров);
- требования ближе к at-least-once + идемпотентности, чем к "строгому exactly-once".

### Расширенный чеклист готовности команды

| Область | Вопрос | Минимум для "готовы" |
|---|---|---|
| Разработка | Умеем проектировать идемпотентные задачи? | Да, есть code review критерии |
| Эксплуатация | Есть метрики очередей/ошибок/лагов? | Да, с алертами |
| Инциденты | Есть runbook на retry storm/backlog? | Да, отработан на учениях |
| Данные | Есть политика payload и PII? | Да, задокументирована |
| Релизы | Есть стратегия изменений контрактов задач? | Да, backward-compatible подход |

### Пошагово: решение за 5 вопросов

1. Задача класса task queue?  
2. Python — основной язык бизнес-логики?  
3. Нужен гибкий runtime контроля задач?  
4. Команда готова сопровождать worker/broker контур?  
5. Нет жесткого требования durable workflow уровня Temporal?  

Если 4-5 ответов "да" — Celery обычно сильный кандидат.

### Практика

Типовые домены:

- e-commerce: обработка заказов, уведомления, пост-обработка;
- fintech adjunct workloads: отчеты, синхронизации, проверочные задачи;
- B2B SaaS: интеграции с внешними API и asynchronous processing.

### Граничные случаи

- Если домен уже multi-language и Python не центральный, Celery может быть не лучшим "платформенным стандартом".
- Если критична сквозная replay-история процессов, лучше заранее рассмотреть workflow engine.

### Проверь себя

1. Почему "у нас много фоновых задач" еще не гарантирует выбор Celery?

<details><summary>Ответ</summary>

Нужно проверить требования к надежности и типу оркестрации: возможно, нужен workflow engine или stream platform, если класс задачи другой.

</details>

2. Какая организационная готовность важна для Celery?

<details><summary>Ответ</summary>

Готовность поддерживать операционный контур: мониторинг, алерты, практики retries, runbooks, контроль деградаций и регулярную инженерную гигиену.

</details>
