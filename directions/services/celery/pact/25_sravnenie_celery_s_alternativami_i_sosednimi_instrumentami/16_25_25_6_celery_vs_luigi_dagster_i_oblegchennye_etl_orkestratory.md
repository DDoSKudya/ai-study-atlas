## 25.6 Celery vs Luigi/Dagster и облегчённые ETL-оркестраторы

### Цель раздела

Понять границу между "композицией задач" и "данными как главным объектом оркестрации".

### Теория и правила

Celery canvas хорош для:

- умеренной orchestration внутри backend-системы;
- operational batch задач;
- workflow, где lineage и data asset модель не являются центральным требованием.

Luigi/Dagster (и похожие ETL-оркестраторы) сильны там, где:

- важен граф данных и зависимости датасетов;
- нужен lineage, data quality и богатая observability data pipelines.

Практический критерий:

- если главный вопрос команды "выполняется ли задача?" — Celery-подход часто достаточен;
- если главный вопрос "из какого источника и по каким трансформациям получился датасет?" — чаще нужен Dagster/Luigi-класс.

### Пошагово

1. Нужно ли видеть "какой датасет от какого шага получен"?
2. Нужны ли data-oriented артефакты и lineage?
3. Если да — ETL-оркестратор обычно предпочтительнее.

### Простыми словами

Celery фокусируется на "выполни задачу".  
Dagster/Luigi фокусируются на "продвинь данные по графу и сохрани прозрачность происхождения".

### Пример структуры data-lineage

```text
Raw source -> Normalize -> Enrich -> Feature table -> BI report
   |            |            |             |             |
 asset A      asset B      asset C       asset D       asset E
```

Когда такая цепочка становится критичной для аудита и диагностики, ETL-оркестратор обычно выигрывает.

### Проверь себя

1. Когда canvas Celery может быть достаточным для ETL?

<details><summary>Ответ</summary>

Когда pipeline относительно прост, объем умеренный, требования к lineage минимальны, а основной контур уже живет в Python backend с Celery.

</details>

2. Какой риск у "сложного ETL на чистом Celery"?

<details><summary>Ответ</summary>

Рост самописной оркестрации, слабая прозрачность данных и сложность поддержки при масштабировании команд и пайплайнов.

</details>
