## Мини scorecard для архитектурного выбора

Используй простую шкалу 1-5 (5 = лучшее соответствие требованию) и веса критериев.

| Критерий | Вес | Celery | Альтернатива A | Альтернатива B |
|---|---:|---:|---:|---:|
| Соответствие классу задачи | 30% |  |  |  |
| Надежность/гарантии | 20% |  |  |  |
| Операционная сложность | 15% |  |  |  |
| Time-to-market | 15% |  |  |  |
| Стоимость владения (12 мес) | 10% |  |  |  |
| Цена миграции/отката | 10% |  |  |  |

`Итог = сумма(оценка * вес)`  
Такой формат дисциплинирует обсуждение и снижает влияние субъективной "технологической симпатии".

### Проверь себя: scorecard

1. Почему в scorecard нельзя ставить одинаковые веса "по умолчанию" для всех проектов?

<details><summary>Ответ</summary>

Потому что контекст разный: где-то критичнее надежность, где-то скорость вывода на рынок, где-то операционная стоимость. Веса должны отражать реальные приоритеты.

</details>

2. Что опасно делать после получения итогового балла?

<details><summary>Ответ</summary>

Слепо принимать решение только по числу. Балл нужно проверять качественным анализом рисков, пилотом и ограничениями платформы.

</details>

### Шаблон решения для архитектурного RFC

Используй этот шаблон, чтобы выбор инструмента был воспроизводимым:

1. **Контекст:** какие задачи, какие SLA, какие ограничения.
2. **Класс проблемы:** task queue / workflow / stream / scheduler / managed.
3. **Кандидаты:** 2-4 реалистичных варианта.
4. **Сравнение:** scorecard + качественные риски.
5. **Пилот:** что проверяем на реальной нагрузке и за какой срок.
6. **Решение:** выбранный вариант + почему не выбраны альтернативы.
7. **Границы применимости:** где решение перестает быть оптимальным.
8. **План пересмотра:** триггеры повторной оценки (рост нагрузки, новые требования).

### Проверь себя: RFC-шаблон

1. Зачем в RFC явно писать "почему не выбраны альтернативы"?

<details><summary>Ответ</summary>

Это фиксирует рациональность решения, упрощает аудит и будущий пересмотр: команда видит, какие компромиссы были приняты осознанно.

</details>

2. Почему пункт "границы применимости" критичен, а не факультативен?

<details><summary>Ответ</summary>

Без него решение воспринимается как вечное. На практике система растет, и заранее зафиксированные границы помогают вовремя переоценить инструмент.

</details>
