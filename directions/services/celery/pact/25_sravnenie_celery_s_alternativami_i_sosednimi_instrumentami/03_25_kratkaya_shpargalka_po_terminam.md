## Краткая шпаргалка по терминам

| Термин | Что это значит простыми словами |
|---|---|
| **Task Queue** | Очередь заданий "сделай работу позже/в фоне". |
| **Scheduler** | Планировщик "запусти это в такое-то время". |
| **Workflow Engine** | Движок долгоживущих процессов со состоянием шагов и историей. |
| **Stream Processing** | Непрерывная обработка потока событий, часто с порядком и replay. |
| **Durable Workflow** | Оркестрация, где состояние workflow хранится надежно и воспроизводимо. |
| **Replay model** | Повторное воспроизведение истории выполнения для восстановления логики. |
| **Lineage** | Видимость происхождения данных: откуда взялось и через какие шаги прошло. |
| **Managed service** | Облачный сервис, где часть эксплуатации берет на себя провайдер. |
| **Cold start** | Задержка первого запуска после простоя или масштабирования с нуля. |
| **Exactly-once (ожидание)** | Ожидание "ровно один раз" (в реальности сложно и дорого). |
| **At-least-once** | Гарантия "минимум один раз", значит возможны дубли. |
| **Control plane vs data plane** | Плоскость управления и плоскость фактической обработки данных/задач. |

#### Проверь себя: термины

1. Чем `Workflow Engine` концептуально отличается от `Task Queue`?

<details><summary>Ответ</summary>

Task Queue фокусируется на выполнении отдельных задач и их доставке, а Workflow Engine — на управлении длительным процессом целиком: шагами, состоянием, переходами и восстановлением.

</details>

2. Почему `Cold start` важен именно при сравнении managed-подхода с self-hosted Celery?

<details><summary>Ответ</summary>

Cold start добавляет задержку запуска и может повлиять на SLA. В self-hosted контуре задержки другого характера, поэтому сравнение стоимости/надежности без учета cold start будет искаженным.

</details>
