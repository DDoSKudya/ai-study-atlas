## 25.1 Celery vs cron/systemd timers

### Цель раздела

Отделить локальное расписание от распределенной обработки задач.

### Термины

| Термин | Коротко |
|---|---|
| **cron** | Планировщик запуска команд по расписанию на хосте. |
| **systemd timer** | Таймеры systemd как более управляемый вариант расписания в Linux. |
| **Distributed queue semantics** | Семантика распределенной доставки/повторов/масштабирования через очередь. |

### Теория и правила

`cron/systemd timers` хороши, когда:

- задача проста и локальна;
- не нужна сложная логика retry/оркестрации;
- допустимо, что расписание привязано к конкретному узлу.

Celery нужен, когда:

- задачи должны распределяться между воркерами;
- нужна очередь с контролем нагрузки;
- требуются retry policy, роутинг, выделенные очереди.

Важно не путать **периодичность** и **очередность**:

- `cron/systemd` отвечают за "когда стартовать";
- Celery отвечает за "как распределить, повторить, ограничить и наблюдать выполнение".

### Пошаговый разбор выбора

1. Есть ли распределенная нагрузка и несколько исполнителей?
2. Нужны ли гибкие повторы и контроль деградации?
3. Важна ли централизованная наблюдаемость фоновой работы?
4. Если на 1-2 вопроса ответ "нет" — возможно `cron` достаточно.

### Простыми словами

`cron` — как будильник на одном телефоне.  
Celery — как диспетчерская служба, распределяющая заказы по бригаде.

### Пример

```bash
# cron: просто запусти скрипт в 03:00
0 3 * * * /opt/app/scripts/rebuild_cache.sh
```

```python
# Celery beat + worker: планирование + распределенное выполнение
app.conf.beat_schedule = {
    "rebuild-cache-every-night": {
        "task": "tasks.rebuild_cache",
        "schedule": crontab(hour=3, minute=0),
    }
}
```

### Что будет если...

Если ты используешь `cron` для распределенного тяжелого workload, быстро появятся:

- конкуренция заданий на одном узле;
- слабая масштабируемость;
- сложная диагностика сбоев и повторов.

Граничный случай: иногда команды делают "distributed cron" через единый cron на одном хосте, который дергает API нескольких сервисов. Это работает как временный мост, но при росте нагрузки превращается в single point of failure.

### Production-рекомендации

- если остаешься на `cron/systemd`, добавь lock-механику (`flock`, distributed lock, уникальные маркеры), чтобы избежать overlap;
- если переходишь в Celery, начинай с 1-2 очередей и простого routing, не усложняй topology преждевременно;
- всегда документируй критерий эскалации "когда cron больше не подходит".

### Проверь себя

1. Почему cron не заменяет Celery для multi-worker сценариев?

<details><summary>Ответ</summary>

Потому что cron решает только "когда запустить", но не дает распределенную очередь, семантику повторов и управление нагрузкой между воркерами.

</details>

2. Когда systemd timers могут быть предпочтительнее Celery?

<details><summary>Ответ</summary>

Когда задача локальна, проста, редка и не требует распределенной обработки/сложной надежности. Например, техническое обслуживание на одном хосте.

</details>
