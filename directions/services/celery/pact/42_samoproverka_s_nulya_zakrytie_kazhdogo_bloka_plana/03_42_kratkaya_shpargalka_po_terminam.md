## Краткая шпаргалка по терминам

| Термин | Простыми словами |
| ------ | ---------------- |
| **Микро-чеклист** | Короткий набор шагов после главы: термины + схема + «нельзя в проде» + edge case + связка с 36/37. |
| **Микролаба** | Упражнение на ≤30 мин с чётким «definition of done». |
| **Definition of Done (DoD)** | Наблюдаемый критерий завершения (что увидели в логах/брокере/метриках). |
| **Сквозной экзамен** | Один проект из модулей, где темы соединены, а не изолированы. |
| **Ложное знание** | Уверенность без проверки: узнаём текст, но не воспроизводим поведение. |
| **Message flow** | Путь: producer → broker → consumer/worker → ack/nack → (опционально) result backend. |
| **Идемпотентность** | Повтор того же действия не ломает инварианты (важно при at-least-once). |
| **Prefetch** | Сколько сообщений worker «забирает вперёд» от брокера относительно готовности исполнять. |
| **Ack / acks_late** | Подтверждение обработки; «поздний ack» меняет момент подтверждения относительно выполнения задачи. |
| **Correlation / trace id** | Идентификатор, связывающий HTTP-запрос, лог и фоновую задачу в одну цепочку расследования. |

#### Проверь себя: шпаргалка

1. Почему **prefetch** относится и к «скорости», и к «надёжности»?
2. Чем **микролаба** ближе к работе инженера, чем контрольный вопрос «на определения»?
3. Зачем в чеклисте явный пункт **«3 вещи нельзя в проде»**, а не просто «best practices»?

<details><summary>Ответ</summary>

1. Prefetch влияет на распределение нагрузки и на то, **кто удерживает** сообщение при сбое worker — это и throughput, и риск «потерять видимость» задачи при неверных ack/timeouts.
2. Лаба проверяет **процедуру** (повторяемость), а определения без практики часто не переносятся в инцидент.
3. Запреты резче формируют инженерную дисциплину: мозг запоминает красные линии, которые экономят ночи on-call.

</details>
