## Краткая шпаргалка по терминам

| Термин | Простыми словами |
|---|---|
| **GPU-bound task** | Задача, ограниченная ресурсом GPU (VRAM/compute), а не CPU. |
| **Warm worker** | Worker с заранее загруженными моделями/ресурсами для сокращения cold start. |
| **Ad-hoc execution** | Разовый запуск "здесь и сейчас", часто из интерактивной среды. |
| **Multi-region topology** | Архитектура с несколькими географическими регионами выполнения. |
| **Data gravity** | Эффект, когда вычисления выгоднее двигать к данным, а не данные к вычислениям. |
| **Enterprise broker** | Брокер в корпоративном ландшафте (MQ/JMS-обертки, прокси, политики безопасности). |
| **Platform parity** | Согласованность поведения приложения на разных ОС/средах. |
| **Control plane / Data plane** | Управление системой vs фактический поток задач/данных. |
| **Affinity routing** | Маршрутизация задач к конкретному типу worker-ов по признакам задачи. |
| **Blast radius** | Радиус воздействия сбоя: сколько подсистем затрагивает отказ. |

#### Проверь себя: базовые термины

1. Чем `Affinity routing` отличается от "просто роутинга по имени задачи"?

<details><summary>Ответ</summary>

Обычный routing по имени задачи чаще статичен. Affinity routing учитывает свойства нагрузки и среды исполнения (регион, тип GPU, требования к безопасности), то есть маршрутизирует по контексту, а не только по названию функции.

</details>

2. Почему для части 32 важно мыслить через `blast radius`, а не только через "успешность одной задачи"?

<details><summary>Ответ</summary>

Потому что нишевые контуры часто связаны с дорогими или критичными ресурсами. Даже если отдельная задача проходит, архитектурная ошибка может масштабно задеть соседние очереди, регионы или платформы.

</details>
