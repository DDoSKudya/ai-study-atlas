## Расширенные частые сценарии (рабочий блок)

### Сценарий 1. GPU-очередь начала расти, latency выросла в 3 раза

**Как рассуждать:**

1. Проверить VRAM saturation и OOM rate.
2. Сравнить долю cold start vs warm execution.
3. Временно снизить parallelism и включить admission control.
4. Разделить "короткие" и "длинные" задачи по разным очередям.

### Сценарий 2. Ad-hoc notebook запуск испортил данные

**Как рассуждать:**

1. Выяснить режим запуска (dry-run/боевой), инициатора и ticket.
2. Остановить новые ad-hoc публикации, включить sandbox-only.
3. Подготовить compensating tasks и верификацию результата.
4. Зафиксировать guardrails в CI/policy, чтобы предотвратить повтор.

### Сценарий 3. Multi-region failover не сработал как ожидалось

**Как рассуждать:**

1. Проверить актуальность routing policy и доступность target region.
2. Валидировать ограничения data residency.
3. Отработать ручной план деградации (какие задачи можно отложить).
4. Обновить game day сценарий и критерии возврата.

### Сценарий 4. Bridge к enterprise broker "иногда теряет" части metadata

**Как рассуждать:**

1. Проверить, какие поля headers обязательны для downstream-логики.
2. Сравнить raw-сообщения до и после bridge.
3. Добавить валидацию metadata на входе worker-а.
4. Зафиксировать контрактный тест на перенос headers в CI.

### Сценарий 5. Локальная разработка на Windows "зелёная", а Linux staging падает

**Как рассуждать:**

1. Проверить расхождения pool/process model между средами.
2. Воспроизвести запуск в Linux-контейнере локально.
3. Убрать platform-specific хак из бизнес-кода.
4. Обновить support matrix и developer guide.

#### Проверь себя: частые сценарии

1. Что объединяет все пять сценариев с точки зрения инженерной дисциплины?

<details><summary>Ответ</summary>

Во всех случаях нужен не "разовый фикс", а управляемый цикл: сигнал -> гипотеза -> обратимое действие -> проверка эффекта -> фиксация в policy/runbook.

</details>

2. Почему сценарии оформлены как "как рассуждать", а не "единственно верные команды"?

<details><summary>Ответ</summary>

Потому что инфраструктура у команд разная, а причинно-следственный процесс диагностики универсален. Команды меняются, метод рассуждения остается.

</details>
