## 32.2 Jupyter, notebooks, ad-hoc

### Цель раздела

Научиться безопасно использовать Celery в интерактивных и исследовательских средах, не смешивая экспериментальный контур с production-контуром.

### В этом разделе главное

- notebook удобен для прототипа, но не должен быть источником production-истины;
- ad-hoc запуск требует контроля версий кода и зависимостей;
- нужна граница ответственности: "исследование" vs "эксплуатация".

### Термины

| Термин | Определение |
|---|---|
| **Notebook kernel drift** | Расхождение окружения notebook с production runtime. |
| **Execution provenance** | Доказуемая история: кто, чем, когда, зачем запустил задачу. |
| **Sandbox queue** | Очередь, где разрешены только безопасные/ограниченные операции. |

### Теория и правила

1. **Notebook — клиент, не платформа оркестрации.**  
   Допустимо инициировать задачу из notebook, но определение задачи и правила маршрутизации должны жить в версионируемом коде.

2. **Репродуцируемость — обязательна.**  
   Запуск в notebook должен ссылаться на фиксированный образ/версию пакета, иначе отладка становится невозможной.

3. **Минимизируй side effects в ad-hoc задачах.**  
   Любой эксперимент, пишущий в production-БД, должен иметь guardrails (dry-run, sandbox, подтверждение).

4. **Изоляция окружений обязательна.**  
   У notebook должен быть отдельный runtime-профиль (зависимости, секреты, сеть), а не "тот же доступ, что у production service account".

### Пошагово: безопасный ad-hoc workflow

1. Создай отдельный namespace очередей `adhoc.*`.
2. Разреши запуск только в sandbox/readonly режиме по умолчанию.
3. Добавь TTL/автоочистку результатов ad-hoc задач.
4. Логируй инициатора, git SHA и notebook/session id.
5. Для "боевых" действий требуй явное подтверждение и change ticket.

### Мини-чеклист "можно ли запускать ad-hoc в production"

- есть ли task в репозитории, а не только в notebook;
- есть ли dry-run path и верификация эффекта;
- есть ли owner и ticket;
- есть ли возможность остановки/rollback;
- есть ли ограничение blast radius (batch size/rate limit).

#### Проверь себя: ad-hoc чеклист

1. Почему пункт про rollback обязателен даже для "маленькой правки"?

<details><summary>Ответ</summary>

Потому что маленькая правка может затронуть много записей из-за неверного фильтра или повторного запуска. Без rollback/compensation ошибка становится дорогой и долгой в исправлении.

</details>

2. Что означает "ограничить blast radius" в ad-hoc контуре?

<details><summary>Ответ</summary>

Это технически уменьшить возможный ущерб: уменьшить batch, поставить rate limit, запускать на выборке, добавить промежуточные точки проверки результата.

</details>

### Простыми словами

Notebook — это лабораторный стол. На нем удобно пробовать идеи, но нельзя хранить единственный экземпляр "правильной" логики. Все важное должно быть перенесено в кодовую базу и CI.

### Картинка в голове

Исследовательский контур — как черновик в инженерной тетради; production-контур — как подписанный чертеж для цеха. Путать их опасно.

### Пример: защитный запуск ad-hoc задачи

```python
def run_adhoc(task_sig, *, dry_run=True, ticket_id=None):
    if not dry_run and not ticket_id:
        raise ValueError("Для боевого ad-hoc запуска нужен ticket_id")
    headers = {"mode": "adhoc", "dry_run": str(dry_run), "ticket_id": ticket_id or "none"}
    return task_sig.apply_async(queue="adhoc.controlled", headers=headers)
```

### Практика / реальные сценарии

- **Data fix после инцидента:** ad-hoc задача исправляет данные пакетами по 500 записей, сначала `dry_run=True`.
- **Разовый backfill:** запускается из notebook, но фактическая задача определена в репозитории и протестирована в CI.

### Типичные ошибки

- определять task-функции прямо в notebook;
- запускать ad-hoc задачи на production очереди без маркировки;
- отсутствие audit trail "кто и зачем запустил".

### Что будет, если...

- **...эксперимент писать в production без dry-run?**  
  Риск массовой порчи данных и долгого ручного восстановления.
- **...не фиксировать версию кода?**  
  Невозможно воспроизвести результат или объяснить причину отличий.

### Проверь себя

1. Почему определение задач в notebook — плохая долгосрочная практика?

<details><summary>Ответ</summary>

Потому что notebook плохо контролируется как production-код: сложнее versioning, code review, повторяемость и автоматические проверки.

</details>

2. Какой минимум metadata нужен для ad-hoc запуска?

<details><summary>Ответ</summary>

Инициатор, цель запуска, git SHA/версия кода, режим (`dry_run`/боевой), идентификатор change ticket.

</details>

3. Как изоляция окружений снижает риски в notebook-сценариях?

<details><summary>Ответ</summary>

Она ограничивает возможный ущерб: даже если ошибка в notebook-коде, он не получает автоматически полный production-доступ к данным, очередям и секретам.

</details>

### Запомните

Notebook отлично подходит для исследования, но production-надежность начинается там, где есть версионируемый код, проверяемость и эксплуатационная дисциплина.
