## 37.3 `celery beat` и `celery events`

### Цель раздела

Разобрать запуск планировщика и event-контура так, чтобы периодика была предсказуемой, а поток событий — полезным и безопасным.

### В этом разделе главное

- `beat` отвечает за *когда публиковать*, но не за *как исполнять*.
- Для одной логической группы periodic jobs обычно нужен один активный scheduler.
- `events`/camera/dump — инструмент наблюдения и расследования, а не "всегда включенный режим ради интереса".
- Неправильные pid/log/schedule-path в `beat` быстро приводят к дублям и пропускам.

### Термины

| Термин | Что это | Простыми словами |
|---|---|---|
| `celery beat` | Планировщик периодических задач | "Будильник публикации задач" |
| `--scheduler` | Класс scheduler | "Как именно beat хранит и читает расписание" |
| `--schedule` | Файл локального состояния beat | "Где beat помнит прошлые срабатывания" |
| `celery events` | Просмотр/стрим событий worker/task | "Лента телеметрии Celery" |
| `camera` | Плагин обработки event stream | "Куда и как сохранять события" |
| `dump` | Вывод событий в консоль | "Быстрый live-режим для диагностики" |
| `--pidfile`/`--logfile` (beat) | Файлы процесса beat | "Операционные якоря планировщика" |

### Теория и правила

#### 1) Single scheduler principle

Если два `beat` публикуют одну и ту же periodic-задачу, дубли неизбежны.  
Поэтому:

- держи один активный scheduler на домен расписаний;
- fallback/HA делай через явные механизмы, а не "параллельный запуск на всякий случай";
- периодические задачи проектируй идемпотентными.

#### 2) Scheduler class и источник истины

Выбор `--scheduler` определяет, где живет состояние:

- в локальном файле;
- в БД (например, расширения для Django);
- в кастомном backend.

Это влияет на эксплуатацию, миграции и recovery.

Также важны параметры цикла scheduler-а:

- `--max-interval` (или аналог конфигурации) влияет на "частоту пробуждения";
- слишком большое значение повышает риск задержки реакции на изменения расписания;
- слишком маленькое создает лишний I/O и шум.

#### 3) Events и безопасность

Event stream может содержать чувствительные метаданные.  
Важно:

- ограничивать доступ к tooling;
- контролировать объем событий;
- не использовать dump в production как постоянный режим.

Для `celery events` полезно заранее решить:

- кто потребитель event stream (оператор, камера, внешний мониторинг);
- сколько данных реально нужно хранить;
- как удаляются или архивируются исторические event-данные.

### Пошагово

1. Определи источник расписаний и scheduler class.
2. Зафиксируй путь к schedule-файлу (или хранилищу), права доступа и бэкап-политику.
3. Проверь, что одновременно активен только один scheduler-instance.
4. Для событий определи цель: live-debug, аудит, метрики, постмортем.
5. Ограничь доступ и retention для event-данных.

### Простыми словами

`beat` — это диспетчер, который вовремя отправляет задания в очередь.  
`events` — это видеонаблюдение за тем, как эта система живет в реальности.

### Картинка в голове

```text
[beat] --publish by schedule--> [broker] --> [worker]
   |
   +-> state store (schedule file/db)

[events stream] <--- worker/task lifecycle telemetry
```

### Примеры

```bash
# Запуск beat с явным scheduler и файлами
celery -A proj.celery_app beat \
  --loglevel=INFO \
  --scheduler=celery.beat:PersistentScheduler \
  --schedule=/var/lib/celery/beat-schedule.db \
  --pidfile=/var/run/celery/beat.pid \
  --logfile=/var/log/celery/beat.log
```

```bash
# Быстрый просмотр событий
celery -A proj.celery_app events --dump
```

```bash
# Пример events с camera (если камера поддерживается в вашей версии/плагине)
celery -A proj.celery_app events --camera=myproject.monitoring.Camera
```

```bash
# Запуск beat с явным max loop interval
celery -A proj.celery_app beat \
  --scheduler=celery.beat:PersistentScheduler \
  --schedule=/var/lib/celery/beat-schedule.db \
  --max-interval=10 \
  --loglevel=INFO
```

### Практика / реальные сценарии

1. **После деплоя periodic задачи дублируются** — случайно поднято два `beat`.
2. **Пропуски запусков после рестарта** — потерян или поврежден schedule state.
3. **Слишком дорогая наблюдаемость** — events включены максимально подробно без цели и ретенции.

### Типичные ошибки

- запускать `beat` в нескольких репликах без coordination;
- хранить schedule state в ephemeral storage без понимания последствий;
- путать мониторинг событий с полноценной системой аудита;
- открывать events-интерфейсы без контроля доступа.

### Что будет, если...

- **...доверить периодику без идемпотентности:** любой дубль scheduler-а даст реальные дубли side effects.
- **...держать dump-events постоянно в проде:** лишняя нагрузка, информационный шум, риски утечки метаданных.

### Проверь себя

1. Почему `beat` и `worker` нужно рассматривать как разные operational-компоненты?

<details><summary>Ответ</summary>

`Beat` генерирует публикацию по времени, а `worker` исполняет задачи. У них разные точки отказа, метрики, конфигурации и процедуры восстановления.

</details>

2. Чем опасен "невидимый второй beat"?

<details><summary>Ответ</summary>

Он не всегда заметен сразу, но приводит к системным дублям periodic-задач, и ущерб часто проявляется уже в бизнес-эффектах.

</details>

3. Когда `events --dump` действительно уместен?

<details><summary>Ответ</summary>

Для короткой точечной диагностики, когда нужно оперативно увидеть поток событий, а не как постоянный production-режим.

</details>

### Дополнительная самопроверка по подпунктам 37.3

#### К подпункту 37.3.1 (single scheduler)

1. Почему даже при одном `beat` периодические задачи все равно должны быть идемпотентными?

<details><summary>Ответ</summary>

Потому что дубли возможны из-за рестартов, сетевых сбоев, ручных операций и edge-case календаря. Single scheduler снижает риск, но не делает его нулевым.

</details>

2. Как operationally доказать, что дубль идет от второго scheduler-а, а не от retry задачи?

<details><summary>Ответ</summary>

Сопоставить логи/метрики публикации из beat-процессов, PID/hostname scheduler-ов и события публикации; retry обычно виден в lifecycle конкретной задачи, а второй beat — в параллельной публикации.

</details>

#### К подпункту 37.3.2 (scheduler class и loop)

1. Чем рискован слишком большой `--max-interval`?

<details><summary>Ответ</summary>

Планировщик реже "просыпается", из-за чего растет задержка реакции на изменения расписаний и может ухудшиться точность запуска задач.

</details>

2. Почему миграция scheduler backend требует отдельного recovery-плана?

<details><summary>Ответ</summary>

Меняется источник состояния расписаний. Без плана можно получить пропуски/дубли из-за рассинхронизации старого и нового state.

</details>

#### К подпункту 37.3.3 (events и безопасность)

1. Почему `events` нельзя оставлять "как есть" после инцидентной отладки?

<details><summary>Ответ</summary>

Включенный подробный поток событий увеличивает стоимость и шум, а также может расширять поверхность утечки метаданных.

</details>

2. Какой критерий помогает понять, что event-поток полезен, а не "технический шум"?

<details><summary>Ответ</summary>

Наличие ясного потребителя (алертинг, forensic, дашборд), ограниченного retention и проверяемой ценности для SLO/SLA диагностики.

</details>

### Запомните

- `beat` отвечает за время публикации, не за исполнение.
- Один активный scheduler обычно является обязательным инвариантом.
- События полезны, когда есть явная цель и контроль объема/доступа.
