## Частые сценарии

### Сценарий 1. "Worker стартует, но нужные задачи не исполняются"

**Пошагово:**

1. Проверить `-A` в фактической команде запуска.
2. Выполнить `inspect registered` и убедиться, что нужные task names загружены.
3. Проверить `-Q/--queues` и `inspect active_queues`.
4. Сверить env-переопределения (`CELERY_CONFIG_MODULE`, broker/backend).

### Сценарий 2. "После перезапуска появились дубли периодических задач"

**Пошагово:**

1. Проверить, не запущено ли два `beat`.
2. Убедиться в корректности `--schedule` state path и прав доступа.
3. Проверить idempotency periodic task.
4. Зафиксировать policy single scheduler в runbook.

### Сценарий 3. "Инцидент: очередь растет, downstream падает"

**Пошагово:**

1. `inspect` для диагностики (`active/reserved/scheduled`).
2. Временная адресная `control rate_limit` мера.
3. Проверка влияния на backlog и критичные очереди.
4. Пост-инцидентно: постоянный фикс в retry/backoff/routing.

### Сценарий 4. "В контейнере работает иначе, чем на VM"

**Пошагово:**

1. Сверить entrypoint и рабочую директорию.
2. Сравнить env-набор и секреты.
3. Проверить сигналы остановки и grace period оркестратора.
4. Зафиксировать единую каноническую run-команду для всех сред.

### Сценарий 5. "Команда `purge` случайно удаляет полезные задачи"

**Пошагово:**

1. Немедленно зафиксировать инцидент и оценить, какие очереди затронуты.
2. Поднять аудит команд: кто, когда, с какими аргументами.
3. Оценить возможность восстановления через исходные источники событий.
4. Внедрить guardrail: запрет `purge` без явного подтверждения владельца домена.

### Сценарий 6. "После смены env в Kubernetes worker запустился, но работает не так"

**Пошагово:**

1. Снять фактический env из pod и сравнить с ожидаемым.
2. Проверить приоритеты CLI/env/config и дубли параметров.
3. Выполнить `inspect conf`/`report` для подтверждения runtime-параметров.
4. Закрепить preflight-check на обязательные переменные.
