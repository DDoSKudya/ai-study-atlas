## Краткая шпаргалка по терминам

| Термин | Что это значит |
|---|---|
| **ASGI** | Интерфейс для асинхронных Python web-приложений (FastAPI/Starlette/Uvicorn). |
| **WSGI** | Классический синхронный интерфейс web-приложений (Flask/Gunicorn sync workers). |
| **Request context** | Данные текущего HTTP-запроса (headers, auth, trace-id), живущие только в web-процессе. |
| **Task payload** | Сериализованные данные, которые уходят в broker как аргументы задачи. |
| **Correlation ID** | Идентификатор, связывающий событие в API и дальнейшие действия worker-а. |
| **Outbox pattern** | Паттерн надёжной публикации событий через запись в БД и последующую доставку в broker. |
| **Sync/async boundary** | Граница, где async-код API встречается с синхронным task-кодом и внешними клиентами. |
| **Worker-local state** | Локальное состояние процесса worker-а; не то же самое, что состояние web-инстанса. |
| **Task contract version** | Явная версия схемы payload, чтобы безопасно менять формат аргументов. |
| **Eager mode** | Режим тестирования, когда задачи исполняются сразу в том же процессе. |

### Частая путаница терминов (развести в голове заранее)

| Путают | Правильное различие |
|---|---|
| `202 Accepted` и `success` | `202` означает "приняли в асинхронную обработку", а не "бизнес-операция завершена". |
| `task_id` и бизнес-идентификатор операции | `task_id` относится к инфраструктурной задаче; бизнес-сущность (например, `order_id`) может жить отдельно. |
| retry и надежность бизнес-эффекта | Retry повышает шанс исполнения, но не гарантирует идемпотентность side effects. |
| `BackgroundTasks` и Celery | `BackgroundTasks` локален процессу API, Celery — распределённая очередь с worker-ами. |
| publish success и конечный результат | Успешная публикация — только начало жизненного цикла, не финальный outcome. |

#### Проверь себя по путанице терминов

1. Почему `task_id` нельзя показывать как "номер готового результата" без статусов?

<details><summary>Ответ</summary>

`task_id` — это идентификатор инфраструктурной попытки обработки. Без статусной модели и доменного результата пользователь не понимает, завершена ли операция, в ретрае ли она, или уже упала.

</details>

2. Как связаны retry и идемпотентность, и почему их часто путают?

<details><summary>Ответ</summary>

Retry отвечает за повтор попытки исполнения, а идемпотентность — за безопасный итог при повторе. Retry без идемпотентности может усиливать ущерб (дубли платежей, уведомлений и т.п.).

</details>
