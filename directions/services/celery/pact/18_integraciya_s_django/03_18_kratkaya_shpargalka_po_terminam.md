## Краткая шпаргалка по терминам

| Термин | Суть простыми словами |
|--------|------------------------|
| **`celery.py`** | Модуль, где создаётся объект **`Celery`** и задаётся **имя** приложения; точка сборки интеграции. |
| **`app.autodiscover_tasks()`** | Автопоиск `tasks.py` в **установленных** Django‑приложениях (`INSTALLED_APPS`). |
| **`transaction.on_commit(hook)`** | «Вызови **после** успешного коммита текущей транзакции»; страховка от **rollback** и **невидимых строк**. |
| **Гонка «worker раньше коммита»** | Worker читает БД **до** того, как web‑транзакция **зафиксировала** изменения — типичный баг без `on_commit`. |
| **Outbox pattern** | В той же БД пишется строка «нужно отправить сообщение»; **отдельный процесс** читает outbox и публикует в брокер — сильнее связка **БД↔очередь**. |
| **Передавать `pk`, не `instance`** | В сообщение кладут **лёгкий идентификатор**, а объект в worker‑е **перечитывают** — меньше сериализации, меньше **stale** состояния. |
| **Stale data** | Worker работает с **устаревшим снимком** логики/полей, потому что модель изменилась после постановки задачи или читается без нужных `select_related`. |
| **`close_old_connections()`** | Django‑хелпер: закрыть «старые»/протухшие соединения; важно в **долгих** задачах и вокруг **fork**. |
| **`CONN_MAX_AGE`** | Сколько секунд **переиспользовать** TCP‑соединение к БД; в worker‑ах часто **0** или осторожное значение из‑за **prefork**. |
| **`django-celery-beat`** | Хранение расписания periodic tasks в **БД Django** вместо статического `beat_schedule` (по необходимости). |
| **`django-celery-results`** | Хранение **результатов** задач в таблицах Django (альтернатива Redis‑backend и т.д.). |
| **Audit trail** | Неизменяемая **история**: кто инициировал фоновую операцию, когда, с какими параметрами — для расследований и compliance. |
| **`CACHES` (LocMem vs Redis)** | В worker **LocMem** не разделяется с web; для инвалидации между процессами нужен **внешний** кэш. |
| **`MEDIA_ROOT` / storage** | Worker должен **видеть** те же файлы/ключи S3, что и web, иначе `FileField` «есть в админке, нет в задаче». |
| **ASGI + `async` view** | ORM и **`on_commit`** — из **sync** слоя; типично **`sync_to_async`** вокруг сервиса с **`atomic` + delay** (§18.7). |
| **Redelivery / `acks_late`** | Сообщение может прийти **дважды** после краша worker; нужны **идемпотентность** и учёт **миграций** (§18.6, часть 9). |
| **`get_task_logger`** | Логгер задач Celery; проще **фильтровать** worker‑логи и согласовать с `LOGGING` (§18.5). |
| **`CELERY_TASK_ROUTES` / `queue=`** | Направить тяжёлые задачи в **отдельную** очередь; worker должен слушать **`-Q`** с этим именем (§18.1). |
| **`ignore_result`** | Не писать результат в **result backend**, если статус в **модели Job** и `AsyncResult` не используется (§18.1, ч. 6). |
| **`django.setup()`** | Обязателен в **скриптах вне** `manage.py`/worker перед импортом моделей и **`delay`** (§18.1). |

#### Проверь себя: шпаргалка

1. Чем **`on_commit`** отличается от **«после `save()`»** в том же view?

<details><summary>Ответ</summary>

`save()` **не** означает коммит: в транзакции `atomic()` несколько `save` могут откатиться **позже**. `on_commit` привязан к **фактическому commit** на соединении. Код «после save» может выполниться при **ещё не зафиксированных** данных.

</details>

2. Почему **outbox** иногда нужен, если уже есть **`on_commit`**?

<details><summary>Ответ</summary>

`on_commit` не делает публикацию в брокер **транзакционной** с брокером: между коммитом и успешным **publish** возможен сбой → **потеря события**. Outbox в **той же** транзакции даёт **запись‑источник правды**, которую можно **дослать** позже.

</details>

3. Задачи уходят в очередь **`exports`**, но worker их не берёт — что проверить по шпаргалке?

<details><summary>Ответ</summary>

Соответствие **`CELERY_TASK_ROUTES` / `queue=`** при постановке и аргумента **`celery worker -Q ...`**: процесс должен **подписаться** на имя очереди, иначе сообщения **висят** в брокере.

</details>
