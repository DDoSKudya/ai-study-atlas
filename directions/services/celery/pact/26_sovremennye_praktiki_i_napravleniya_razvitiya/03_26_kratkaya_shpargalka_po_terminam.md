## Краткая шпаргалка по терминам

| Термин | Что это значит простыми словами |
|---|---|
| **IaC (Infrastructure as Code)** | Инфраструктура описана в коде (Terraform/Helm/Ansible), а не ручными кликами. |
| **KEDA/HPA** | Механизмы autoscaling в Kubernetes по метрикам очередей и нагрузки. |
| **OpenTelemetry** | Стандарт сбора трасс, метрик и логов для end-to-end наблюдаемости. |
| **SLO для background systems** | Формальные цели качества для фоновых задач (время старта, успех, lag). |
| **Failure budget burn** | Скорость, с которой система "сжигает" допустимый бюджет ошибок. |
| **Quorum queue** | RabbitMQ-очередь с репликацией и устойчивостью через quorum-модель. |
| **Probe design** | Как правильно проверять готовность/живость worker-подов. |
| **Scaling per queue** | Масштабирование разных worker-групп отдельно по конкретным очередям. |
| **Execution layer** | Слой исполнения задач, которым может управлять внешний оркестратор. |
| **Transactional outbox** | Паттерн надежной публикации задач/событий после commit в БД. |

#### Проверь себя: термины

1. Чем `Execution layer` концептуально отличается от `Workflow engine`?

<details><summary>Ответ</summary>

Execution layer исполняет отдельные шаги/задачи, а workflow engine управляет процессом целиком: состоянием, переходами и длительной логикой оркестрации.

</details>

2. Почему термин `Failure budget burn` важен именно для modern-практик, а не только для SRE-теории?

<details><summary>Ответ</summary>

Потому что он связывает алертинг с реальным риском нарушения SLO и помогает приоритизировать действия в фоне задач, где деградации часто накапливаются постепенно.

</details>
