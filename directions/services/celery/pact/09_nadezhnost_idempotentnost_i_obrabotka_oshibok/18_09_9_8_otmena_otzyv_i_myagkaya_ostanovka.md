## 9.8. Отмена, отзыв и мягкая остановка

### Цель раздела

Понять реальные границы отмены задач и научиться строить безопасную "кооперативную отмену" для долгих процессов.

### В этом разделе главное

- `revoke` не равен "гарантированно остановить всё немедленно".
- `terminate` может создать partial side effects.
- Для долгих задач нужна cooperative cancellation.

### Термины

| Термин | Кратко |
| --- | --- |
| **revoke** | Отзыв задачи: запрет на запуск, если она ещё не стартовала. |
| **terminate** | Принудительное завершение уже исполняемого worker-процесса. |
| **Cancel token** | Флаг отмены, который задача проверяет в безопасных точках. |
| **Signal (`SIGTERM`, `SIGKILL`)** | Сигналы ОС, влияющие на остановку процесса worker/child с разной "жёсткостью". |

### Теория и правила

Практическая семантика:

- `revoke` хорошо работает до фактического старта execution.
- Если задача уже внутри тяжёлого блока, "мгновенная отмена без последствий" невозможна.
- `terminate` полезен как аварийный инструмент, но увеличивает риск полуприменённых действий.
- `signal` в `terminate` задаёт "насколько жёстко" прерываем процесс: мягкий сигнал может дать шанс cleanup, жёсткий - почти всегда рвёт выполнение.

#### Важная граница: revoke/terminate не заменяют дизайн задачи

Даже при наличии remote control команд надёжная отмена достигается только тогда, когда код задачи:

- имеет checkpoint-ы;
- умеет читать cancel flag;
- корректно фиксирует промежуточное состояние;
- не держит бесконечные неблокируемые операции без timeout.

#### Проверь себя по границе revoke/terminate

1. Почему даже "идеально работающий" `revoke` не решает проблему отмены уже запущенной задачи?

<details><summary>Ответ</summary>

`revoke` управляет допуском к старту, но не может безопасно "отмотать" код, который уже выполняется. Для этого нужна кооперативная логика внутри самой задачи.

</details>

2. Какой минимальный дизайн делает отмену предсказуемой?

<details><summary>Ответ</summary>

Шаги/checkpoint-ы, cancel flag, таймауты внешних операций и фиксация промежуточного статуса, чтобы задача могла корректно остановиться и не оставить неясное состояние.

</details>

### Пошагово

1. Для long-running задач введи `job_status` в БД (`running/cancel_requested/cancelled`).
2. Разбей задачу на шаги с checkpoint-ами.
3. Перед каждым шагом проверяй cancel flag.
4. При cancel выполняй аккуратный cleanup и фиксируй итог.
5. `terminate` оставь как крайний аварийный путь.

### Простыми словами

Отмена долгой задачи похожа на остановку поезда: безопасно тормозить на станции (checkpoint), а не рвать стоп-кран посреди моста.

### Картинка в голове

```text
run step 1 -> check cancel -> run step 2 -> check cancel -> run step 3 -> complete
```

### Как запомнить

**Надёжная отмена = кооперация кода задачи, а не только команда снаружи.**

### Примеры

```python
@shared_task(bind=True)
def build_report(self, report_id: int):
    for chunk in iterate_chunks(report_id):
        if is_cancel_requested(report_id):
            mark_cancelled(report_id)
            return "cancelled"
        process_chunk(chunk)
    mark_done(report_id)
    return "done"
```

### Практика / реальные сценарии

- **Генерация отчёта на 30 минут:** пользователь нажал "отменить" в UI.
- **ML/ETL batch:** нужно остановить джобу без потери already committed частей.
- **Инцидентный stop:** оператор отзывает задачи при ошибочном rollout.

#### Сравнение стратегий остановки

| Стратегия | Что делает | Плюсы | Риски/ограничения | Когда применять |
| --- | --- | --- | --- | --- |
| `revoke` (до старта) | Не даёт задаче стартовать | Безопасно для side effects | Не помогает, если задача уже исполняется | Отзыв очереди до фактического исполнения |
| Cooperative cancel | Задача сама завершает работу по cancel flag | Наиболее предсказуемо и безопасно | Требует архитектуры шагов/checkpoint-ов | Long-running задачи с пользовательской отменой |
| `terminate` + signal | Прерывает процесс принудительно | Быстро останавливает runaway task | Возможны partial effects и сложное восстановление | Аварийный stop при инциденте |

#### Проверь себя по стратегиям остановки

1. Почему cooperative cancel считается приоритетной стратегией для long-running задач?

<details><summary>Ответ</summary>

Потому что позволяет завершать работу в контролируемых точках, сохранять консистентное состояние и минимизировать риск partial effects, в отличие от жёсткого прерывания процесса.

</details>

2. Когда `terminate` действительно оправдан, несмотря на риски?

<details><summary>Ответ</summary>

Когда задача ведёт себя как runaway и наносит системный ущерб (занимает ресурсы, блокирует очередь), а продолжение исполнения опаснее последствий аварийной остановки.

</details>

### Типичные ошибки

- обещать пользователю "мгновенную отмену" без архитектурной поддержки;
- не хранить отмену как доменное событие;
- использовать `terminate` как обычный механизм контроля потока.

### Что будет, если...

- **...полагаться только на `revoke`?** Уже запущенные задачи продолжат работу.
- **...везде делать `terminate`?** Растёт риск partial effects и сложных восстановлений.
- **...ввести cooperative cancellation?** Отмена станет предсказуемой и управляемой.

### Проверь себя

1. Почему revoke не гарантирует остановку уже выполняемой задачи?

<details><summary>Ответ</summary>

Потому что revoke влияет на решение "запускать/не запускать" задачу. Если исполнение уже началось, нужен кооперативный механизм внутри задачи или аварийный terminate.

</details>

2. Зачем задачу разбивать на шаги для отмены?

<details><summary>Ответ</summary>

Чтобы иметь безопасные точки проверки cancel-флага и минимизировать размер "неделимого" участка, который нельзя аккуратно прервать.

</details>

3. Когда terminate оправдан?

<details><summary>Ответ</summary>

В аварийных ситуациях (например, runaway task), когда риск продолжения выполнения выше риска partial effects и дальнейшего ручного восстановления.

</details>

### Запомните

- Отмена в распределённых системах всегда компромисс.
- Лучшее качество отмены достигается в коде задачи, а не только через внешнюю команду.
- `terminate` - аварийный инструмент, не повседневный.
