## Краткая шпаргалка по терминам

| Термин | Суть простыми словами |
| --- | --- |
| **Идемпотентность** | Повторный запуск не должен менять итог сверх первого корректного результата. |
| **Redelivery** | Повторная доставка того же сообщения (например, ack не успел дойти). |
| **Poison task** | Задача, которая гарантированно падает на данном payload и не "вылечивается" ретраями. |
| **Retry policy** | Набор правил: какие ошибки повторять, сколько раз, с какими паузами. |
| **Backoff** | Увеличение паузы между повторными попытками. |
| **Jitter** | Случайный разброс паузы, чтобы не вызвать лавину одновременных повторов. |
| **Transactional outbox** | Паттерн "сначала фиксируем событие в БД, потом публикуем асинхронно" для согласованности. |
| **Compensation** | Компенсирующее действие при частичном успехе (откат/обратная операция на бизнес-уровне). |
| **DLQ / parking queue** | Очередь "в сторону" для задач, которые нельзя бесконечно ретраить. |
| **Logical job** | Бизнес-работа верхнего уровня (например, "обработать заказ"), которая может включать несколько task execution. |
| **Cooperative cancel** | Кооперативная отмена: задача сама регулярно проверяет флаг отмены и корректно завершает работу. |
