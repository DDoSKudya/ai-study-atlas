## 9.4. Таймауты и ограничения выполнения

### Цель раздела

Научиться задавать таймауты на разных уровнях так, чтобы они дополняли друг друга и не создавали хаос при зависаниях.

### В этом разделе главное

- `soft_time_limit` и `hard_time_limit` решают разные задачи.
- Таймаут HTTP/DB клиента обязателен даже при наличии Celery limits.
- Hard kill может оставить частично выполненный эффект.

### Термины

| Термин | Кратко |
| --- | --- |
| **soft time limit** | Мягкий лимит: задача получает шанс завершиться корректно. |
| **hard time limit** | Жёсткий лимит: процесс задачи принудительно останавливается. |
| **hung task** | "Зависшая" задача, которая не завершает работу в разумный срок. |
| **timeout layering** | Иерархия таймаутов на разных слоях (клиент, задача, очередь). |
| **Runaway task** | Задача, которая неконтролируемо потребляет время/ресурсы и мешает соседним workload. |

### Теория и правила

Правило слоёв:

1. Внешние вызовы (HTTP, DB, RPC) должны иметь собственные таймауты.
2. `soft_time_limit` чуть больше суммы ожидаемых шагов.
3. `hard_time_limit` - аварийный стоп выше soft.
4. Ретраи после таймаута допустимы только при идемпотентности.

### Пошагово

1. Разбей задачу на этапы (fetch, transform, write).
2. Оцени worst-case по каждому этапу.
3. Поставь таймауты клиентам внешних систем.
4. Настрой soft/hard limits на задаче.
5. Добавь обработку `SoftTimeLimitExceeded` для graceful cleanup.

#### Диагностика зависших задач (чек-лист)

1. Проверить, есть ли прогресс по метрикам (`tasks_started` растёт, а `tasks_succeeded/failed` нет).
2. Сверить среднюю и p95/p99 длительность по типам задач.
3. Посмотреть, не исчерпаны ли worker slots долгими вызовами без timeout.
4. Проверить prefetch и `acks_late` (не "зажаты" ли задачи у worker-а).
5. Снять выборочную диагностику stack/trace работающих процессов.

#### Проверь себя по диагностике hung tasks

1. Почему рост `tasks_started` при стабильном `tasks_succeeded/failed` - тревожный сигнал?

<details><summary>Ответ</summary>

Это указывает, что задачи запускаются, но не завершаются, то есть worker slots заняты долгими/зависшими операциями. Без вмешательства система теряет пропускную способность.

</details>

2. Как prefetch может визуально "маскировать" зависания?

<details><summary>Ответ</summary>

Worker может заранее забрать много задач, и снаружи будет казаться, что очередь уменьшается, хотя фактическое завершение не растёт. В результате проблемы исполнения скрываются за буфером prefetch.

</details>

### Простыми словами

Таймауты - как предохранители в электросхеме: один общий рубильник недостаточен, нужны локальные.

### Картинка в голове

```text
[HTTP timeout 3s] < [DB timeout 5s] < [soft limit 20s] < [hard limit 30s]
```

### Как запомнить

**Сначала таймауты на внешних вызовах, потом на задаче в целом.**

### Примеры

```python
from celery.exceptions import SoftTimeLimitExceeded

@shared_task(bind=True, soft_time_limit=20, time_limit=30, max_retries=3)
def enrich_profile(self, profile_id: int):
    try:
        data = fetch_remote_profile(profile_id, timeout=3.0)
        save_profile_data(profile_id, data, db_timeout=5.0)
        return "ok"
    except SoftTimeLimitExceeded as exc:
        mark_profile_as_partial(profile_id)
        raise self.retry(exc=exc, countdown=30)
```

### Практика / реальные сценарии

- **Длинные I/O задачи:** важно не держать бесконечные сокеты.
- **Batch с большими payload:** ограничивать время на элемент и на batch в целом.
- **Сторонний API иногда "висит":** лучше короткий клиентский timeout + retry, чем один длинный.

### Типичные ошибки

- надеяться только на `time_limit` без таймаутов в клиентах;
- ставить слишком маленький hard limit и убивать почти завершённые операции;
- не учитывать cleanup при soft timeout.

### Что будет, если...

- **...не поставить таймауты внешним вызовам?** Зависания съедят worker slots.
- **...полагаться только на hard limit?** Получишь больше partial effects и труднее восстановление.
- **...проектировать timeout layering?** Поведение станет предсказуемым и управляемым.

### Проверь себя

1. Почему `hard time limit` опаснее для бизнес-инвариантов, чем `soft`?

<details><summary>Ответ</summary>

Hard limit может прервать задачу в любой точке, включая момент после внешнего эффекта, но до фиксации внутреннего статуса. Это увеличивает риск "полудоставленного" состояния.

</details>

2. Зачем нужны таймауты HTTP-клиента, если уже есть `soft_time_limit`?

<details><summary>Ответ</summary>

`soft_time_limit` действует на задачу целиком. Без клиентского timeout один подвисший вызов может занять почти весь бюджет времени и ухудшить контроль над шагами внутри задачи.

</details>

3. В каком случае timeout стоит сочетать с circuit breaker?

<details><summary>Ответ</summary>

Когда внешний сервис длительно деградирует: timeout ограничивает ожидание одного вызова, а circuit breaker временно прекращает новые запросы и защищает систему от каскадной перегрузки.

</details>

### Запомните

- Таймауты должны быть многослойными.
- Soft и hard limits нужны вместе, но с разной ролью.
- Таймаут без идемпотентности не делает повторы безопасными.
