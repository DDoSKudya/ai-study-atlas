## Краткая шпаргалка по терминам

| Термин | Что это значит простыми словами |
|---|---|
| **Kombu** | Транспортная библиотека Celery: унифицирует работу с разными брокерами. |
| **Transport** | Реализация протокола/драйвера обмена сообщениями (Redis, AMQP, SQS и др.). |
| **Broker URL** | Адрес и параметры подключения Celery к брокеру. |
| **Dialect transport** | Специфическое поведение конкретного транспорта, не всегда эквивалентное другим. |
| **Visibility timeout (SQS)** | Время, на которое сообщение «прячется» после получения consumer-ом. |
| **Heartbeat (AMQP)** | Периодический сигнал «я жив», чтобы вовремя замечать мертвые соединения. |
| **Requeue** | Возврат сообщения обратно в очередь для повторной доставки. |
| **DLQ (Dead Letter Queue)** | Очередь «мертвых» сообщений, не обработанных основным контуром. |
| **Failover URL list** | Список брокеров/endpoint-ов для переключения при недоступности текущего. |
| **Ephemeral transport** | Непостоянный транспорт для тестов, без гарантий production-уровня. |

#### Проверь себя: термины

1. Чем transport отличается от broker как продукта?

<details><summary>Ответ</summary>

Broker — это конкретная система (RabbitMQ, Redis, AWS SQS). Transport — это способ/драйвер, через который Celery/Kombu с этой системой взаимодействует.

</details>

2. Почему «диалект транспорта» важно помнить при миграции Redis -> RabbitMQ?

<details><summary>Ответ</summary>

Потому что одинаковый Celery-код может менять фактическую семантику доставки и подтверждений. То, что работало на Redis, может иначе вести себя на AMQP из-за другой модели очередей и подтверждений.

</details>
