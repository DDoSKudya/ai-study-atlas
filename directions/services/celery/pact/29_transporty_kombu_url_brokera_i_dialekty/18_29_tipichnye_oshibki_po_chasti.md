## Типичные ошибки по части

- **Ошибка:** выбирать брокер «по привычке команды».  
  **Как правильно:** сначала формализовать требования и failure modes.

- **Ошибка:** оставлять `broker_url` без явных timeout/failover/TLS-параметров.  
  **Как правильно:** делать подключение частью архитектурного контракта.

- **Ошибка:** переносить конфиг между transport-ами без ревизии semantics.  
  **Как правильно:** проводить parity-анализ и адаптацию retry/ack-модели.

- **Ошибка:** не тестировать аварийные сценарии публикации и re-delivery.  
  **Как правильно:** включить fault-injection тесты и сценарии worker crash.

- **Ошибка:** переоценивать результаты memory-тестов.  
  **Как правильно:** держать второй слой на real broker.

### Антипаттерн -> корректный подход (быстрый ориентир)

| Антипаттерн | Чем опасен | Корректный подход |
|---|---|---|
| «Одинаковый конфиг для всех transport-ов» | скрытые несовместимости и неверные ожидания от доставки | отдельный transport-profile + parity-тесты |
| «`apply_async` прошел, значит задача гарантированно в broker» | ложная уверенность, труднообъяснимые потери | контроль publish path + broker ingress метрик |
| «Тестов на memory достаточно» | production-регрессии в сбоях и re-delivery | двухслойная стратегия: fast tests + real-broker failure tests |
| «Failover проверим потом» | аварийное переключение не срабатывает в инциденте | регулярный controlled-failure drill с измерением Recovery Time |

### Что особенно часто путают (финальная памятка)

1. Одинаковый Celery API не означает одинаковую transport-семантику.
2. Валидный `broker_url` не означает устойчивость в аварийном режиме.
3. Успешный `apply_async` не означает, что сообщение уже устойчиво зафиксировано broker-ом.
4. Успех на `memory/file` не означает production-надежность на реальном broker.
