## Краткая шпаргалка по терминам

| Понятие | Суть простыми словами |
| --- | --- |
| **Producer** | Тот, кто «публикует» сообщение/задачу в очередь. Обычно это веб-сервис или другой контур. |
| **Consumer** | Тот, кто «забирает» сообщения из очереди и выполняет работу. Для Celery это worker. |
| **Broker** | Посредник между producer и consumer: хранит, маршрутизирует и/или выдаёт сообщения. |
| **Queue** | Место (логическое), где сообщения ждут исполнения. |
| **Message** | Единица работы в очередях: тело (payload) + метаданные доставки. |
| **Queue depth** | «Сколько сообщений сейчас лежит в очереди» — грубый индикатор перегрузки. |
| **Lag** | «Насколько сильно сообщение отстаёт по времени» (например, по времени отправки). |
| **Throughput** | Сколько сообщений/работ в единицу времени реально обрабатывается. |
| **Service time** | Среднее время обработки одного сообщения consumer-ом. |
| **At-most-once** | Теоретическая семантика «сообщение может выполниться 0 или 1 раз». На практике её сложно обеспечить без потери надёжности. |
| **At-least-once** | Реалистичная семантика «сообщение будет выполнено хотя бы раз, но возможно повторно». |
| **Ack (acknowledgment)** | Подтверждение broker-у: «я обработал и можно считать сообщение выполненным». |
| **Redelivery** | Повторная доставка сообщения, если ack не дошёл/не сработал или истёк visibility/ack timeout. |
| **Visibility timeout** | В managed-очередях (например, SQS) время, в течение которого сообщение «считается невидимым», пока consumer его обрабатывает. |
| **Persistence** | Устойчивое хранение сообщений, которое уменьшает риск потери при перезапусках. |
| **Ordering** | Возможность/гарантии упорядочивания сообщений. Часто они слабые и зависят от топологии. |
| **Priority** | Поддержка приоритетов сообщений (и ограничения на то, насколько реально соблюдается порядок по приоритетам). |
| **Idempotency key** | Ключ идемпотентности: уникальный идентификатор бизнес-операции, по которому dedup контролирует повтор. |
| **De-duplication** | Устранение дублей на уровне приложения (обычно через уникальные ограничения, lock-ы, таблицы «обработано/статус»). |
| **Transactional outbox** | Паттерн: гарантируем публикацию события в очередь «после commit транзакции» через промежуточную таблицу outbox. |
| **Saga / compensation** | Паттерн распределённых процессов без строгих транзакций: шаги плюс компенсации при частичных сбоях. |
| **Retry with backoff** | Повтор с паузами (часто экспоненциальный backoff + jitter), чтобы не убить downstream. |
| **Circuit breaker** | Механизм остановки/ограничения вызовов до восстановления внешней системы. |
| **Rate limiting** | Ограничение скорости входящих/повторных операций, чтобы контролировать нагрузку. |
| **Backpressure** | Механизм «торможения» продюсинга/потока при перегрузке consumer-а/downsream-а. |
