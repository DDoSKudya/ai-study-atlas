## Краткая шпаргалка по терминам

| Термин | Простыми словами |
|---|---|
| **Tooling catalog** | Карта инструментов вокруг Celery: что за что отвечает и где границы. |
| **Flower** | Web UI для наблюдения и базового управления Celery-кластером. |
| **Broker metrics** | Метрики очередей/брокера (глубина очередей, соединения, throughput брокера). |
| **App metrics** | Метрики бизнес-логики задач (успехи, ошибки, latency по типам задач). |
| **Infra metrics** | Метрики инфраструктуры хостов/контейнеров (CPU, RAM, сеть, диск). |
| **Structured logging** | Логи в машиночитаемом формате (обычно JSON) для поиска и корреляции. |
| **High cardinality** | Слишком много уникальных меток в метриках, что резко увеличивает стоимость хранения и запросов. |
| **Auto-instrumentation** | Автоматическое подключение APM/OTel без ручной обвязки каждого вызова. |
| **Best effort integration** | Интеграция "работает пока работает", без строгих гарантий поддержки. |
| **Compatibility matrix** | Таблица совместимости: версия Celery x версия интегратора x версия брокера. |

#### Проверь себя: термины

1. Чем `App metrics` отличаются от `Infra metrics` в диагностике инцидента?

<details><summary>Ответ</summary>

`App metrics` описывают поведение задач и бизнес-логики (успехи, ошибки, latency), а `Infra metrics` — ресурсы и состояние среды (CPU, память, сеть). Первые отвечают за "качество работы задачи", вторые — за "возможность среды выполнять задачу".

</details>

2. Почему `Compatibility matrix` — это не формальность документации?

<details><summary>Ответ</summary>

Потому что без неё обновления проводятся вслепую: после апгрейда может «тихо» отвалиться часть telemetry или интеграций, и это выяснится только в инциденте.

</details>
