## Справочник по части

| Тема | Ключевая мысль |
|---|---|
| Flower | удобно для runtime-обзора, но не замена полноценной observability-платформы |
| Метрики | разделяй broker/app/infra, иначе алерты будут ложными |
| APM/OTel | начинай с критичных задач, контролируй cardinality и стоимость |
| Логи | structured JSON + correlation-id = быстрая диагностика |
| Django-интеграции | удобство админки должно быть под контролем прав и retention |
| Тестирование | eager + real broker tests + матрица версий |
| Каталог | фиксируй support-level, совместимость, owner, fallback |

### Быстрый decision-guide по выбору набора инструментов

```text
Если команда маленькая, SLA умеренный, бюджет ограничен:
  Flower + broker exporter + базовые app-metrics + JSON logs

Если SLA высокий и много критичных задач:
  + трассировка (OTel/APM) + расширенный alerting + compatibility matrix + fallback runbook

Если строгий комплаенс/безопасность:
  минимальный доступ к control plane, RBAC, audit trail, policy на изменение расписаний
```

#### Проверь себя: decision-guide

1. Почему нельзя копировать одинаковый стек observability между командами без адаптации?

<details><summary>Ответ</summary>

У команд различаются SLA, риск-профиль и бюджет. Один и тот же стек может быть либо избыточным, либо недостаточным в другом контексте.

</details>

2. Что является первым сигналом, что пора переходить от базового набора к расширенному?

<details><summary>Ответ</summary>

Регулярные инциденты, которые нельзя быстро диагностировать базовыми метриками и логами, а также рост критичности фоновых задач.

</details>

### Production-ready чеклист для части 33

- [ ] Flower/операторский UI защищен (auth, TLS, network policy, audit).
- [ ] Разделены broker/app/infra метрики и настроены алерты по слоям.
- [ ] В логах есть `task_id` + `correlation_id` + `task_name` + `retry_count`.
- [ ] Для APM/OTel задана стратегия sampling и лимиты cardinality.
- [ ] Для OTel/APM проверена совместимость с используемым pool (`prefork`/`threads`/`gevent`).
- [ ] Для Django-пакетов определены границы использования и retention.
- [ ] Тесты проходят на eager и real broker-контуре, есть tox/nox матрица.
- [ ] Для каждого интегратора есть owner, support-level, версия, fallback.

### Release quality gate для изменений в tooling-каталоге

Перед релизом считается "готово", если одновременно выполнено:

1. обновлена compatibility matrix по затронутым интеграциям;
2. в staging пройден smoke сценарий "публикация -> выполнение -> лог/метрика/трейс";
3. on-call runbook обновлен и содержит fallback;
4. изменения в control-plane (Flower/admin) прошли security-review;
5. алерты проверены на отсутствие дубликатов и ложных срабатываний.

#### Проверь себя: quality gate

1. Почему quality gate должен проверять и telemetry, и процессы (runbook/security review)?

<details><summary>Ответ</summary>

Потому что техническая работоспособность без операционной готовности дает уязвимую систему: данные есть, а действовать по ним команда не умеет безопасно и воспроизводимо.

</details>

2. Что означает «релиз не готов», даже если код интеграции проходит тесты?

<details><summary>Ответ</summary>

Если не выполнены процессные условия gate: нет обновленной матрицы совместимости, нет fallback-runbook, нет валидации алертов и security-review.

</details>
