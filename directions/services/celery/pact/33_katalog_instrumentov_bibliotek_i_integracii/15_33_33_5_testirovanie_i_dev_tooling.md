## 33.5 Тестирование и dev tooling

### Цель раздела

Собрать воспроизводимый тестовый контур, который проверяет Celery-логику до релиза: локально, в CI и на разных брокерах.

### Теория и правила

- Не ограничивайся `task_always_eager`: он полезен, но не моделирует реальные очереди и сетевые сбои.
- Для важных сценариев проверяй поведение с "настоящим" брокером (например, Redis/RabbitMQ в CI).
- Держи матрицу тестов: Python-version x Celery-version x Broker.

### Что включить в pytest-контур

| Тип теста | Что проверяем | Пример риска |
|---|---|---|
| **Unit** | бизнес-ветки задачи без брокера | неверная логика ретрая/валидации |
| **Integration** | публикация/доставка/исполнение через broker | проблемы сериализации, роутинга, ack |
| **Failure mode** | таймауты, внешние ошибки, повторная доставка | бесконечные retries, дубли эффектов |
| **Compatibility** | поведение на нескольких брокерах/версиях | регресс после апгрейда Celery или драйвера |

### pytest plugins и test utilities (ориентир по экосистеме)

- `celery.contrib.pytest` — встроенные pytest-фикстуры Celery (удобны для управляемого запуска worker в тестах);
- `pytest` + docker-контейнеры брокера — практический базовый набор для интеграционных тестов;
- `tox`/`nox` — матрица сред и брокеров;
- внутренние helper-утилиты проекта (factory задач, фикстуры retry/timeout) часто полезнее "универсального" community-плагина.

Правило: **сначала проверяй, какие инструменты поддерживаются вашей версией Celery**, затем фиксируй это в compatibility matrix части `33.6`.

#### Проверь себя: test utilities

1. Почему встроенные фикстуры Celery часто безопаснее случайного community-плагина?

<details><summary>Ответ</summary>

Они ближе к официальной экосистеме и обычно лучше синхронизированы с версионными изменениями Celery.

</details>

2. Когда внутренние helper-утилиты проекта особенно полезны?

<details><summary>Ответ</summary>

Когда нужно стабильно тестировать специфичные доменные сценарии (retry-политики, контрактные ошибки, внешние API), которые generic-пакеты покрывают слабо.

</details>

### Мини-runbook для local dev

1. Подними локально `redis` и `rabbitmq` (через docker compose).
2. Прогоняй smoke-тесты Celery на обоих транспортах.
3. Отдельно прогоняй сценарии retry/time-limit.
4. Сохраняй артефакты логов при падении тестов (для диагностики flaky поведения).

### Пошагово

1. Юнит-тесты бизнес-логики задач отдельно от транспорта.
2. Интеграционные тесты с живым broker-контейнером.
3. Проверка retry/backoff/time-limit сценариев.
4. Матрица в `tox`/`nox` для ключевых комбинаций.
5. Негативные тесты на ошибки сериализации и неизвестные task names.

### Пример CI-матрицы (упрощенно)

```yaml
strategy:
  matrix:
    python: [3.11, 3.12]
    broker: [redis, rabbitmq]
steps:
  - run: nox -s tests -- --broker=${{ matrix.broker }}
```

### Мини-шаблон `tox`/`nox`-подхода для двух брокеров

```text
envlist = py311-redis, py311-rabbitmq, py312-redis, py312-rabbitmq

run:
  - unit tests (быстро, без брокера)
  - integration tests (с выбранным брокером)
  - failure-mode tests (retry/timeout/serialization)
```

#### Проверь себя: матрица окружений

1. Зачем одновременно держать `redis` и `rabbitmq` в тестовой матрице?

<details><summary>Ответ</summary>

Чтобы заранее увидеть транспорт-специфичные расхождения поведения и не обнаружить их только в production.

</details>

2. Почему failure-mode тесты стоит запускать отдельно от unit?

<details><summary>Ответ</summary>

Они тяжелее, зависят от инфраструктуры и проверяют другие риски (доставка, таймауты, сериализация), чем быстрые бизнес-unit тесты.

</details>

### Типичные ошибки

- тестировать только eager-режим;
- не проверять ретраи и дедупликацию;
- не фиксировать версии брокера в CI.
- смешивать бизнес-тесты и инфраструктурные тесты в один хрупкий "интеграционный мегатест".

### Проверь себя

1. Почему `task_always_eager` недостаточен как единственный вид тестов?

<details><summary>Ответ</summary>

Он обходит реальные сетевые и брокерные сценарии, не воспроизводит конкурентное исполнение, подтверждение сообщений и часть проблем сериализации/маршрутизации.

</details>

2. Что обязательно добавить в тестовый контур для production-готовности?

<details><summary>Ответ</summary>

Интеграционные тесты с реальным брокером, негативные сценарии, проверку retry/time-limit и матрицу совместимости версий.

</details>
