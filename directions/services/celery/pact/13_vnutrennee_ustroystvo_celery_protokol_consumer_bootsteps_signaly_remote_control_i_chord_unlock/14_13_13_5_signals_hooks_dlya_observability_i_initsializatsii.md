## 13.5. Signals: hooks для observability и инициализации

### Цель раздела

Научиться использовать сигналы Celery правильно: как точки расширения для телеметрии, логирования, инициализации, диагностики; и отдельно — понять, почему сигналы опасны как место бизнес-логики.

### В этом разделе главное

- Signals — это **hooks** вокруг жизненного цикла задач и worker-а.
- Сигналы отлично подходят для **наблюдаемости** (логи/метрики/трейсы) и “сквозных” тех.политик.
- Сигналы плохо подходят для бизнес-логики: неявность, сложный порядок, риск дублирования.
- Различай “до публикации задачи” и “вокруг исполнения задачи”: это разные места во времени.

### Термины

| Термин | Определение |
|---|---|
| **task_prerun / task_postrun** | События до/после исполнения задачи (в worker-е). |
| **task_failure** | Сигнал при падении задачи (в worker-е). |
| **worker_ready / worker_shutdown** | События готовности/остановки worker-а. |
| **before_task_publish / after_task_publish** | События вокруг публикации сообщения в broker (на стороне producer). |
| **after_setup_task_logger** | Хук настройки логгера задач: место для форматтеров, correlation id, структурированных полей. |

### Теория и правила

#### 1) “Сигнал для наблюдаемости” vs “сигнал для бизнес-эффекта”

Хорошее использование:

- добавить correlation id в логи;
- измерить duration задачи;
- отправить метрику “task_failed_total”;
- добавить/закрыть span трейсинга.

Плохое использование:

- “если задача такая-то — то сделай дополнительный side effect”;
- “при ошибке в сигнале — отменить задачу”.

Почему плохо:

- порядок срабатывания зависит от модели исполнения и версии;
- сигнал может не сработать при падении процесса;
- трудно тестировать как бизнес-правило;
- легко получить дублирование побочного эффекта.

#### 2) Где сигналы живут по времени

Упрощённо:

- **before/after publish** — это producer-side: “сообщение отправляется”.
- **pre/post run** — worker-side: “код реально исполняется”.

Это очень важно: если ты хочешь “прикрепить метаданные к сообщению”, делай это до publish; если хочешь измерить выполнение — вокруг run.

#### 3) Роль `after_setup_task_logger` в production

Этот сигнал нужен, когда ты хочешь централизованно привести task-логи к одному формату:

- добавить обязательные поля (`task_id`, `root_id`, queue, retries);
- включить JSON-формат;
- проставить correlation id для сквозной связи с HTTP/API логами.

Почему это важно: без единого формата “внутренности” Celery плохо диагностируются — у тебя есть события, но нет нормальной корреляции между producer и worker.

### Пошагово

Рецепт безопасного использования signals:

1. Определи цель: наблюдаемость/инициализация/диагностика.
2. Выбери минимальный набор сигналов.
3. Сделай обработчик **быстрым** и **безопасным**: без тяжёлых вызовов и без новых сетевых зависимостей.
4. Не допускай, чтобы ошибка в обработчике сигнала ломала обработку задач.
5. Логи/метрики должны содержать `task_id/root_id` или correlation id.

### Простыми словами

Signals — это “кнопки уведомлений” на заводе:

- “станок начал работу”,
- “станок закончил”,
- “станок сломался”.

Их полезно подключить к панели мониторинга. Но если ты начнёшь принимать бизнес-решения по “мигающей лампочке”, завод станет непредсказуемым.

### Картинка в голове

```text
Producer side: before_publish -> publish -> after_publish
Worker side:   task_prerun   -> execute -> task_postrun / task_failure
```

### Как запомнить

**Signals = наблюдаемость и инфраструктурные hooks.** Бизнес-логика должна быть явной в коде задачи/сервисов.

### Примеры

#### Пример: измерить длительность задачи и логировать связь root_id

```python
import time
from celery import signals

@signals.task_prerun.connect
def on_task_prerun(task_id=None, task=None, args=None, kwargs=None, **extras):
    # Минимальный пример: сохраняем старт в thread-local или в request context.
    # Реально лучше использовать систему трейсинга/контекстов, но принцип такой.
    if task and hasattr(task, "request"):
        task.request.__dict__["_t0"] = time.time()

@signals.task_postrun.connect
def on_task_postrun(task_id=None, task=None, state=None, **extras):
    if task and hasattr(task, "request"):
        t0 = task.request.__dict__.get("_t0")
        root_id = getattr(task.request, "root_id", None)
        if t0 is not None:
            duration = time.time() - t0
            # Здесь должен быть ваш нормальный логгер/метрики
            print("task_done", task.name, task_id, "root_id=", root_id, "sec=", round(duration, 3))
```

Обрати внимание: обработчики короткие, не делают сетевых вызовов и не влияют на бизнес-результат.

#### Пример: добавить correlation id при публикации

Идея: на стороне producer добавлять метаданные (например, из HTTP-request).

```python
from celery import signals

@signals.before_task_publish.connect
def before_publish(headers=None, body=None, properties=None, **kwargs):
    # headers — место, куда можно положить корреляцию
    if headers is not None:
        headers.setdefault("x_correlation_id", "your-correlation-id-here")
```

#### Пример: централизованная настройка task-логгера

```python
import logging
from celery import signals

@signals.after_setup_task_logger.connect
def setup_task_logger(logger=None, **kwargs):
    if logger is None:
        return
    logger.setLevel(logging.INFO)
    # В реальном проекте здесь ставят JSON formatter и фильтр,
    # который прокидывает task_id/root_id/correlation_id.
```

### Практика / реальные сценарии

- **Distributed tracing**: открыть span на producer-side и продолжить на worker-side по trace context.
- **Единая нормализация логов**: логировать `task_id/root_id`, имя очереди, retries.
- **Детектор “подозрительных задач”**: метрика “task runtime > threshold”.

### Типичные ошибки

- Длинные/блокирующие обработчики сигналов → деградация worker-а.
- Сетевые вызовы из signals без timeout → сигнал превращается в “критическую зависимость”.
- Бизнес-логика в signals → невидимые side effects и сложная отладка.

### Что будет если…

- Если телеметрия в signals ломается и падает с исключением, можно повредить стабильность воркера (в зависимости от реализации/версии).
- Если “инфраструктурный сигнал” содержит бизнес-эффект, ты получишь “распределённую неявность” — самый дорогой тип сложности.

### Проверь себя

1. Почему signals не лучший инструмент для бизнес-логики?

<details><summary>Ответ</summary>

Потому что они неявные, зависят от порядка/версии/модели исполнения, плохо тестируются как бизнес-правила и могут срабатывать/не срабатывать при падениях процессов. Бизнес-логика должна быть явной и детерминированной.

</details>

2. Где правильнее добавлять метаданные в “контракт сообщения”: на producer или на worker?

<details><summary>Ответ</summary>

На producer, до публикации, потому что метаданные должны ехать вместе с сообщением через broker и быть доступны worker-у и мониторингу.

</details>

3. Какой главный принцип обработчика сигнала?

<details><summary>Ответ</summary>

Быстро, безопасно, без побочных эффектов: минимум логики, минимум зависимостей, ошибки не должны ломать обработку задач.

</details>

### Запомните

**Signals — идеальный инструмент для observability и инфраструктурных hooks.** Для бизнес-логики — почти всегда ловушка.
