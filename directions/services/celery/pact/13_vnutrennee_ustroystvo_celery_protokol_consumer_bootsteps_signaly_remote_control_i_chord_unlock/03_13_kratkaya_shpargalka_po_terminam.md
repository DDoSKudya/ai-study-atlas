## Краткая шпаргалка по терминам

| Термин | Суть простыми словами |
|---|---|
| **Celery internals** | “Как это устроено внутри”: подсистемы, pipeline, точки расширения. |
| **App (Celery app object)** | Центральный объект конфигурации/регистрации задач и связей с transport/backend. |
| **Task registry** | Реестр “имя задачи → callable”, по которому worker понимает, что запускать. |
| **Loader** | Механизм, который импортирует задачи/модули и связывает их с app. |
| **Consumer blueprint** | Внутренняя схема того, как worker потребляет сообщения и превращает их в исполнение задач. |
| **Task message protocol** | Структура сообщения, в котором передаётся “что выполнить” и метаданные (id, retries, ETA…). |
| **Headers / properties / body** | “Шапка” и метаданные vs “полезная нагрузка” (аргументы/kwargs) сообщения. |
| **Content type / serializer** | Формат и способ кодирования тела (json/msgpack/pickle…). |
| **Bootsteps** | Пошаговая цепочка инициализации worker-а; как “конструктор” из модулей. |
| **Signals** | Hooks: события жизненного цикла worker-а/задачи, на которые можно подписаться. |
| **Remote control** | Механизм команд и инспекции worker-ов (“скажи активные задачи”, “перезагрузи пул”). |
| **Broadcast** | “Рассылка” команды всем worker-ам; зависит от транспорта. |
| **Chord unlock** | Внутренняя логика “разблокировки chord”: когда все задачи группы готовы и можно запускать callback. |
