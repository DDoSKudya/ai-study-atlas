## 13.1. Как Celery устроен “под капотом”

### Цель раздела

Собрать внутреннюю карту Celery: какие подсистемы реально существуют, что они делают, и где проходит граница ответственности. Это “скелет”, без которого всё остальное превращается в набор несвязанных фактов.

### В этом разделе главное

- Celery — это не “декоратор”, а набор подсистем: app/registry/loader/transport/backend/worker-consumer/pool/control/events.
- `Celery app` — центр конфигурации и “контейнер” для задач.
- Worker — это **master процесс** + (часто) **child процессы** пула + consumer-подсистема.
- Внутренний pipeline построен так, чтобы отделять: **получение сообщения** (I/O) → **решение “что делать”** (strategy) → **исполнение** (pool) → **фиксацию результата** (backend/events) → **подтверждение доставки** (ack/reject).
- Большая часть “странностей” в поведении объясняется тем, в каком слое случилась проблема: transport/broker, consumer, pool, backend или код задачи.

### Термины

| Термин | Определение |
|---|---|
| **App** | Объект Celery, который хранит конфигурацию, регистрирует задачи и создаёт связи с transport/backend. |
| **Registry** | Словарь “имя задачи → task object”, который используется при получении сообщения. |
| **Loader** | Механизм импорта задач и автопоиска модулей (зависит от интеграции, например Django loader). |
| **Worker consumer** | Подсистема worker-а, которая читает сообщения из broker и превращает их в исполнение. |
| **Pool** | Исполнитель задач: prefork (процессы), threads, gevent/eventlet, solo. |
| **Events / event dispatcher** | Система событий worker-а (например, для мониторинга), не путать с result backend. |
| **Control / remote control** | Команды “инспектировать/управлять” worker-ом удалённо. |

### Теория и правила

#### 1) Модель “слоёв” внутри Celery

Самый полезный способ помнить internals — слой за слоем:

1. **Transport layer**: соединение с broker, подписка на очереди, получение сообщений.
2. **Consumer layer**: буферизация/prefetch, QoS, выбор стратегии обработки сообщения.
3. **Dispatch layer**: преобразование сообщения в вызов task-а, подготовка контекста, события.
4. **Execution layer (pool)**: реальное исполнение Python-кода (в процессе/потоке/greenlet).
5. **Result/events layer**: запись состояния/результата, публикация событий мониторинга.
6. **Acknowledgment layer**: ack/reject/requeue — подтверждение, что “сообщение можно считать обработанным” (или нет).

Важно: эти слои не всегда “чётко последовательно” выглядят в коде, но для диагностики это лучшая ментальная модель.

#### 2) Почему worker — это не один процесс

В prefork-модели worker обычно состоит из:

- **Main (master) process**: consumer, связь с broker, управление пулом, dispatch, ack/reject.
- **Child processes**: исполняют задачи.

В потоковом/greenlet pool часть может происходить “в одном процессе”, но логика разделения на consumer vs execution сохраняется.

### Пошагово

Мини-алгоритм “как думать про internals, когда что-то не так”:

1. Определи симптом: задача не приходит? приходит, но не стартует? стартует, но не завершает? завершает, но статус не виден? chord завис?
2. Сопоставь симптом слою: transport/broker → consumer → pool → backend → orchestration.
3. Выбери инструменты слоя: broker metrics/logs, `celery inspect`, worker logs, backend queries, Flower/APM.

### Простыми словами

Представь ресторан:

- **Broker** — стойка, где лежат заказы.
- **Consumer** — официант, который забирает пачку заказов со стойки.
- **Strategy/dispatch** — “переводчик заказа”: понять, какое блюдо и кому на кухню.
- **Pool** — кухня, где реально готовят.
- **Backend/events** — касса/табло, где отмечают “готово/не готово”.
- **Ack** — официант ставит штамп “заказ принят/выполнен”, чтобы он не остался на стойке.

Если проблема — важно понимать: это стойка пустая, официант не ходит, кухня перегружена, или касса не отмечает?

### Картинка в голове

```text
             (сообщение)
Producer  ---------------->  Broker/Transport  ----->  Worker Consumer  ----->  Pool (execute)
   |                               |                       |                        |
   |                               |                       |                        |
   +---- (AsyncResult polling) ----+----- Result backend <-+---- (state/result) ----+
                                     \
                                      +--> Events/Monitoring (не результат!)
```

### Как запомнить

- **Celery = pipeline**, а не “один вызов”.
- **Слой = инструмент диагностики**: сначала найди слой, потом смотри метрики/логи/CLI.

### Примеры

#### Пример 1: где “живёт” ошибка

Если ты видишь:

- в broker растёт `queue depth`, но worker “живой” → это **capacity/consumer/prefetch** слой;
- worker пишет “received” но не пишет “started” → это **pool/execution** слой (или starvation);
- задача “finished” в логах, но `AsyncResult` не меняет state → это **backend** слой;
- chord header tasks “готовы”, но callback не запускается → это **chord unlock/backend** слой.

### Практика / реальные сценарии

- **Диагностика прод-симптома**: “всё в порядке, ошибок нет, но очередь растёт”.
  - Привяжи это к слою consumer/pool.
  - Проверь prefetch, concurrency, время исполнения, external bottlenecks.

### Типичные ошибки

- Пытаться лечить всё “увеличением concurrency”.
- Не различать “message received” и “task executed”.
- Смешивать события мониторинга (events) с backend-результатами: это разные системы.

### Что будет если…

- Если ты не понимаешь слои, то любая странность выглядит одинаково: “Celery сломался”.
- Это приводит к неправильным действиям: purge очередей, рестарты без drain, “добавим ретраи”.

### Проверь себя

1. Какая разница между “consumer получил сообщение” и “task реально исполнилась”?

<details><summary>Ответ</summary>

Получение сообщения — это I/O и логика consumer-а: он увидел сообщение в broker и решил “взять в работу”. Исполнение — это фактический запуск Python-кода в pool. Между ними может быть очередь внутри worker-а (prefetch), ожидание свободного slot-а в pool, блокировки и др. Поэтому “received” не гарантирует “started”.

</details>

2. Почему события (events) — не то же самое, что result backend?

<details><summary>Ответ</summary>

Events — поток телеметрии (для мониторинга, Flower и т.п.). Backend — хранилище состояния/результата (для AsyncResult, chord и т.п.). Events могут быть отключены/потеряны и не обязаны быть консистентными; backend нужен для функциональной логики orchestration и статусов.

</details>

3. Какой самый быстрый способ “локализовать слой”, если задачи “не стартуют”?

<details><summary>Ответ</summary>

Сравнить: есть ли сообщения в очереди (broker), видит ли worker “received” (consumer), есть ли свободные слоты в pool/не завис ли пул, и меняется ли state/result (backend). То есть короткая проверка слоёв: broker → consumer → pool → backend.

</details>

### Запомните

**Internals полезны не ради любопытства, а ради локализации проблем.** Сначала слой, потом диагностика.
