## 14.3. Логирование задач: поля и корреляция

### Цель раздела

Научиться писать логи так, чтобы они:

- были **машиночитаемыми** (структурированными),
- позволяли делать **корреляцию** (связь HTTP → task → подзадачи),
- не превращались в помойку чувствительных данных.

### В этом разделе главное

- Логи задач должны быть **структурированными** (JSON), иначе поиск/агрегация не масштабируется.
- Для расследований критично иметь идентификаторы: `task_id`, `root_id`, `parent_id`, `correlation_id`.
- Для multi-tenant добавляется `tenant_id`.
- Для ретраев обязательно логировать `retries`, `eta/countdown`, причину retry.
- Нельзя логировать секреты и “сырые payload” без фильтрации.
- Хорошая практика: логировать **события жизненного цикла** задачи: published → received → started → succeeded/failed/retried.

### Термины

| Термин | Определение |
|---|---|
| **Structured logs** | Логи в формате ключ‑значение (обычно JSON), а не “строка”. |
| **Correlation** | Возможность связать записи разных сервисов и слоёв одной операцией. |
| **Root/parent id** | Идентификаторы графа задач (корень и родитель). |

### Теория и правила

#### Какие поля логировать (рекомендуемый “контракт логов”)

Минимальный набор:

- `timestamp`
- `level`
- `service` / `component` (producer/worker)
- `environment` (prod/stage)
- `task_name`
- `task_id`
- `root_id` / `parent_id` (если есть)
- `correlation_id` (если есть)
- `tenant_id` (если есть)
- `queue` / `routing_key` (если известно)
- `retries`
- `duration_ms` (на завершении)
- `state` / `outcome` (success/failure/retry)
- `error_type` / `error_message` (для ошибок; осторожно с PII)

**Важно:** не все поля доступны в каждом месте. Поэтому удобнее думать не “всё всегда”, а “в каждом лог‑событии — максимум релевантного”.

#### Как связывать HTTP и Celery

Ключевой паттерн: **создать correlation id** на входе HTTP и прокинуть его в:

- заголовки HTTP логов/трейсов,
- параметры/headers Celery сообщения (или аргументы),
- контекст логов задачи.

Корреляция работает, если:

- id стабилен,
- id попадает во все события,
- id индексируется в лог‑хранилище.

### Пошагово

Чек‑лист внедрения:

1. В веб‑приложении создавай `correlation_id` (если клиент не прислал).
2. При публикации задачи передавай `correlation_id` в headers (или аргументы, если так проще).
3. В worker-е при старте задачи поднимай этот id в контекст логов (logger context).
4. Логируй “received/start/end” хотя бы для критичных задач.
5. Добавь фильтры: не логировать payload целиком, не логировать секреты.

### Простыми словами

Без корреляции логи — как коробка с разрозненными квитанциями. С корреляцией — как папка “дело №123”, где собраны все бумаги по одному случаю.

### Картинка в голове

`correlation_id` — это “номер заказа”, который пишут на всех бумажках: на чеке, на коробке, на пакете доставки.

### Как запомнить

Если хочешь расследовать “почему пользователь ждёт”, тебе нужны:

- **кто** (tenant/user),
- **какая операция** (correlation_id),
- **какая задача** (task_id + task_name),
- **где в графе** (root/parent),
- **как долго** (duration + timestamps),
- **почему не получилось** (error_type, retry info).

### Примеры

#### Пример 1. JSON‑лог завершения задачи

```json
{
  "timestamp": "2026-03-30T10:15:20.123Z",
  "level": "INFO",
  "service": "billing-worker",
  "environment": "prod",
  "event": "task_succeeded",
  "task_name": "tasks.generate_invoice",
  "task_id": "b9e1c8c2-...",
  "root_id": "a12f3d9e-...",
  "parent_id": null,
  "correlation_id": "req-7f0c...",
  "tenant_id": "tenant_42",
  "queue": "billing",
  "retries": 1,
  "duration_ms": 842,
  "result_size_bytes": 128
}
```

**Что важно:** мы не кладём в лог “весь invoice”, а кладём измерения и идентификаторы.

#### Пример 2. Лог retry

```json
{
  "timestamp": "2026-03-30T10:15:19.010Z",
  "level": "WARN",
  "service": "billing-worker",
  "environment": "prod",
  "event": "task_retry",
  "task_name": "tasks.generate_invoice",
  "task_id": "b9e1c8c2-...",
  "correlation_id": "req-7f0c...",
  "retries": 1,
  "retry_in_seconds": 60,
  "error_type": "TimeoutError",
  "error_message": "external API timeout"
}
```

### Практика / реальные сценарии

- **Сценарий**: “пользователь жалуется: инвойс не пришёл”.
  - Ищешь по `correlation_id` → видишь publish → retries → succeeded.
- **Сценарий**: “инцидент: выросли ретраи”.
  - По логам строишь топ ошибок по `error_type`, группируешь по внешней зависимости.

### Типичные ошибки

- Логировать в одну строку без структуры: потом невозможно агрегировать.
- Логировать **task_id** но не логировать **root_id** — теряется граф.
- Логировать payload целиком (PII/секреты/кардинальность/стоимость).

### Что будет если…

- **…логировать task_id в метриках как label?** Получишь взрыв кардинальности и убьёшь метрики (это относится к 14.8, но тут важно помнить).
- **…не логировать retries?** Ты не увидишь деградацию как “рост количества попыток”, увидишь только “поздние ошибки”.

### Проверь себя

1. Почему structured logs почти всегда обязательны в production?

<details><summary>Ответ</summary>

Потому что без структуры нельзя надёжно фильтровать, группировать, строить дашборды по логам и искать корреляции. Строка не масштабируется на тысячи воркеров и миллионы задач.

</details>

2. Какие 4 идентификатора чаще всего дают максимальную ценность для корреляции?

<details><summary>Ответ</summary>

`correlation_id` (бизнес‑операция), `task_id` (конкретная задача), `root_id` (граф задач), `tenant_id` (разделение клиентов, если есть).

</details>

3. Почему нельзя “просто логировать все аргументы задачи”?

<details><summary>Ответ</summary>

Потому что там могут быть PII/секреты, большие объёмы данных, и это увеличит стоимость хранения/передачи логов и риск утечек. Лучше логировать идентификаторы и размеры/классы данных, а не сами данные.

</details>

### Запомните

- Логи должны отвечать на вопрос “что случилось” и “как это связано с бизнес‑операцией”.
- Корреляция — это не роскошь, а способ чинить быстро.
