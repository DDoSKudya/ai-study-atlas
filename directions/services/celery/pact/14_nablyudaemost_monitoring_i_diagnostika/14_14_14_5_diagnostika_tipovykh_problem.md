## 14.5. Диагностика типовых проблем

### Цель раздела

Научиться чинить частые “симптомы” Celery по алгоритму: **симптом → слой → проверка → действие**, а не прыгать хаотично между настройками и перезапусками.

### В этом разделе главное

- Почти любой инцидент можно локализовать по слоям: broker/consumer/pool/backend/beat/orchestration.
- Диагностика — это не “перезапусти всё”, а “проверь минимально достаточные факты”.
- Многие проблемы выглядят одинаково (`PENDING`), но причины разные: очередь, backend, несовпадение ожиданий, неправильный мониторинг.
- Для каждого симптома нужен **runbook** (чек‑лист), иначе ночью всё забудется.

### Термины

| Термин | Определение |
|---|---|
| **Runbook** | Пошаговая инструкция “как действовать при симптоме”. |
| **Backlog** | Накопление задач в очереди. |
| **Retry storm** | Взрыв повторов, который сам ухудшает систему. |
| **Poison message** | Сообщение, которое всегда ломает обработку и “крутится” бесконечно. |

### Теория и правила

#### Базовый принцип: сначала “что видно точно”

В инциденте важно отделять:

- **факт**: очередь растёт, consume rate упал, воркеров стало меньше,
- от **интерпретации**: “Celery сломался”.

#### Универсальная таблица “симптом → слой”

| Симптом | Частые причины (слои) |
|---|---|
| `PENDING` “навсегда” | backend не настроен/не доступен, задача не была запущена, ожидание статуса не то, что ты думаешь |
| Задачи долго висят в очереди | backlog, worker мало, consume rate упал, broker тормозит |
| Worker жив, но не потребляет | consumer завис, соединение к broker потеряно, prefetch/QoS забит, event loop/greenlet проблемы |
| Много `RETRY` | деградация внешнего API/БД, слишком агрессивный retry policy |
| Memory leak | утечки в коде задачи, большие результаты, prefork процессы не перезапускаются |
| Дубликаты задач | at-least-once, ранний ack/late ack, worker lost, повторная публикация |
| Chord не завершается | backend не поддерживает/сломался, race conditions, heavy polling |
| Beat публикует дубликаты | несколько beat, clock/DST, overlap prevention отсутствует |

### Пошагово (runbooks)

Ниже — практические чек‑листы. Они написаны “по слоям”: сначала дешёвые проверки, потом дорогие.

#### 14.5.1 Задача застряла в `PENDING`

**Интуиция:** `PENDING` часто означает не “задача ждёт”, а “мы не знаем про неё ничего (ещё)”.

**Проверки:**

1. Это точно Celery‑статус, а не статус бизнес‑системы?
2. Result backend включён? (Если backend отключён, многие клиенты всегда будут видеть `PENDING`.)
3. Есть ли подтверждение, что задача **опубликована** (лог producer / метрика publish)?
4. Есть ли подтверждение, что задача **потреблена** (лог worker received / consume rate)?
5. Нет ли ситуации “сообщение в очереди, но воркеры не слушают эту очередь” (routing mismatch)?

**Действия:**

- Если publish нет → проблема на стороне producer (или broker publish).
- Если publish есть, consume нет → проблема в broker/consumer/routing.
- Если consume есть, но status не меняется → проблема в execution/pool или backend writes.

#### 14.5.2 Задача висит в очереди слишком долго (lag растёт)

**Проверки:**

1. Publish rate вырос или consume rate упал?
2. Воркеров стало меньше (crashes/restarts)?
3. Worker‑ы загружены CPU/IO? (utilization/saturation)
4. Появились “супермедленные” задачи (p99 вырос)?
5. Не изменился routing (задачи попали не в ту очередь)?

**Действия:**

- Если задач стало больше → scale out workers или ограничить producer/backpressure.
- Если задачи стали медленнее → профилировать задачу/внешние зависимости.
- Если broker тормозит → лечить broker (ресурсы, alarms, сеть).

#### 14.5.3 Worker жив, но не потребляет

**Проверки:**

1. Есть ли соединение worker → broker? (broker‑native UI)
2. Consume rate = 0? (метрики)
3. Worker слушает нужные очереди? (routing/queues)
4. Prefetch/QoS не “забит” резервированными задачами? (много reserved, мало active)

**Действия:**

- Если consumer завис — возможно перезапуск worker (но важно понять причину).
- Если routing mismatch — исправить маршрутизацию/запуск с нужными очередями.
- Если prefetch слишком большой — снизить, чтобы не “запирать” задачи на одном воркере.

#### 14.5.4 Много `RETRY`

**Проверки:**

1. Какая причина retry? (error_type)
2. Какая внешняя зависимость деградирует?
3. Backoff/джиттер есть? Или ретраи синхронизированы и создают волну?

**Действия:**

- Включить jitter/backoff.
- Добавить circuit breaker/rate limit.
- Разделить очереди, чтобы ретраи не блокировали важные задачи.

#### 14.5.5 Memory leak / рост памяти

**Проверки:**

1. Растёт ли память по одному child‑процессу (prefork)? Или это общий рост?
2. Есть ли большие payload/results?
3. Есть ли утечки соединений/клиентов (HTTP sessions, DB connections)?

**Действия:**

- Настроить `max_tasks_per_child` (перезапуск процессов после N задач) как “предохранитель”.
- Уменьшить размеры payload/result, хранить данные в объектном хранилище, а не в backend.
- Профилировать память задач.

#### 14.5.6 Дубликаты задач

**Проверки:**

1. Это реальные дубликаты publish или redelivery?
2. Как настроены ack (`acks_late`) и `task_reject_on_worker_lost`?
3. Есть ли idempotency ключ на стороне бизнеса?

**Действия:**

- Принять at-least-once как норму: сделать задачи идемпотентными.
- Для особо критичных операций — dedup/locks/unique constraints.

#### 14.5.7 Chord не завершается / callback не стартует

**Проверки:**

1. Backend поддерживает chord корректно?
2. Видны ли результаты всех header‑задач? (часть может “не записать результат”)
3. Нет ли timeouts/TTL на результаты группы?

**Действия:**

- Проверить backend настройки TTL/expiration.
- Пересмотреть orchestration: возможно, нужен другой паттерн (event-driven или explicit barrier).

#### 14.5.8 Beat публикует дубликаты или не публикует

**Проверки:**

1. Не запущено ли несколько beat инстансов?
2. Нет ли DST/clock drift эффектов?
3. Есть ли overlap prevention для задач расписания?

**Действия:**

- Сделать beat “single leader” или использовать distributed scheduler.
- Добавить lock/dedup на уровне задачи.

#### 14.5.9 Lag растёт только на одной очереди или одном tenant

**Интуиция:** проблема локализована — значит, её причина почти всегда **в конкретном типе задач, tenant-е или очереди**, а не “во всём Celery”.

**Проверки:**

1. Какая это очередь и какие типы задач туда попадают? (есть ли “тяжёлые” задачи?)
2. Только ли один tenant страдает (по `tenant_id` в логах/метриках)?
3. Различается ли профиль нагрузки этого tenant-а (пики, объёмы данных)?
4. Нет ли специфических ограничений/лимитов для этого tenant-а (rate limit, quotas)?
5. Не изменился ли недавно код/конфигурация задач именно для этого tenant-а/очереди?

**Действия:**

- Если лаг локализован по tenant-у:
  - проверь, не “перегружает” ли этот tenant систему (burst, abuse, новая фича),
  - рассмотри выделение отдельной очереди/воркер‑пула под него,
  - введи rate limits/quotas для избыточного load-а.
- Если лаг локализован по очереди:
  - проверь состав задач в очереди (нет ли одной новой “тяжёлой” задачи),
  - пересмотри routing — возможно, часть задач должна идти в отдельный пул,
  - добавь специализованные worker-ы именно под этот тип задач.

### Простыми словами

Диагностика — как поиск течи воды:

- если воды нет в кране — проблема до крана (подача),
- если вода есть, но не доходит до душа — проблема в трубе (маршрут),
- если доходит, но еле течёт — засор или давление (производительность),
- если течёт, но не нагревается — проблема в бойлере (внешняя зависимость).

### Картинка в голове

Симптом — это “красная лампочка”, слой — “какая система самолёта”, runbook — “что пилот делает по чек‑листу”.

### Как запомнить

Формула расследования:

**Где? (слой)** → **Что измерить? (сигнал)** → **Как проверить? (инструмент)** → **Что сделать? (действие)**.

### Примеры

#### Пример. Почему `PENDING` может быть нормой

Если ты отключил result backend (или не сохраняешь результаты), Celery не будет хранить состояние, и клиент может видеть `PENDING` всегда. Это не “задача не выполняется”, это “статус не наблюдаем”.

### Практика / реальные сценарии

- Составь runbook для 3 самых частых симптомов в твоём проекте и проверь, что все шаги могут быть выполнены человеком без контекста (ночью).

### Типичные ошибки

- Перезапускать воркеров без локализации слоя → временно лечит, но не устраняет причину.
- Считать, что “PENDING = в очереди”.
- Диагностировать только по логам задач, не смотря на брокер и инфраструктуру.

### Что будет если…

- **…не иметь runbooks?** Инциденты будут решаться медленнее и по‑разному каждым инженером, ошибки будут повторяться.

### Проверь себя

1. Почему `PENDING` не всегда означает “задача ждёт в очереди”?

<details><summary>Ответ</summary>

Потому что `PENDING` может означать “нет записи о статусе” (нет result backend или запись не создана), либо “мы не нашли результат”. Очередь и backend — разные слои.

</details>

2. Как отличить “задачи стали медленнее” от “воркеров стало меньше”?

<details><summary>Ответ</summary>

Смотри распределение execution time (p95/p99) и utilization: если execution вырос — задачи/зависимости; если execution примерно прежний, но lag растёт и consume rate упал — воркеров стало меньше или они не потребляют.

</details>

3. Почему retry storm опасен сам по себе?

<details><summary>Ответ</summary>

Потому что ретраи увеличивают нагрузку на брокер, воркеры и внешние зависимости, ухудшая ситуацию и создавая положительную обратную связь: деградация → больше ретраев → ещё сильнее деградация.

</details>

### Запомните

- Симптомы похожи, причины разные — лечим по слоям.
- Runbook — это “инженерная память” для инцидентов.
