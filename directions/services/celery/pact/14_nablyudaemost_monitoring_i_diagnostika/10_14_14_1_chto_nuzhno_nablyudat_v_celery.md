## 14.1. Что нужно наблюдать в Celery

### Цель раздела

Научиться выбирать **правильные сигналы** (метрики/события), чтобы отличать “медленно потому что очереди” от “медленно потому что задачи” и от “медленно потому что инфраструктура”, не смешивая разные причины в одну цифру.

### В этом разделе главное

- **Наблюдаемость начинается с вопросов**, а не с инструментов: что мы хотим уметь объяснять?
- У Celery есть минимум **5 слоёв**, и на каждом нужен свой набор сигналов: publish → broker → worker consume → execution → backend/orchestration.
- **Queue lag** обычно важнее, чем queue depth.
- **Execution time** и **end-to-end latency** — разные сущности, и их нужно мерить отдельно.
- Любая “сводная метрика” без разбиения по **queue / task name / tenant / environment** часто бесполезна.
- Нужны сигналы не только “успех/ошибка”, но и **reties**, **timeouts**, **worker restarts**, **broker/backends health**.
- Модель USE/RED помогает не забыть важные категории метрик.

### Термины

| Термин | Определение |
|---|---|
| **Queue lag** | Время ожидания сообщения в очереди до начала обработки. |
| **End-to-end latency** | Время от publish (или логической постановки) до завершения. |
| **Throughput** | Сколько задач/сек выполняется (или публикуется). |
| **Utilization / saturation** | Насколько ресурсы заняты / насколько близко к “упору”. |

### Теория и правила

#### Слои и “что болит” на каждом слое

Полезно держать в голове структуру как “конвейер”:

```text
------------------+     +----------------+     +-------------------+     +----------------+
| Producer / App   | --> | Broker (queue) | --> | Worker consumer    | --> | Pool execution |
| publish tasks    |     | route/store    |     | reserve/dispatch   |     | run Python     |
------------------+     +----------------+     +-------------------+     +----------------+
                                                                       |
                                                                       v
                                                             +------------------+
                                                             | Result backend   |
                                                             | state/result     |
                                                             +------------------+
```

**Ключевая идея:** симптомы часто похожи (“пользователь ждёт”), но причины живут в разных местах.

#### Минимальный набор сигналов (MVP observability)

Если у тебя совсем мало времени, но хочешь “не быть слепым”, минимум такой:

- **Broker / queues**
  - queue depth по очередям;
  - publish rate и consume rate (хотя бы приблизительно);
  - broker health (errors/latency).
- **Workers**
  - количество живых worker-ов и их concurrency (эффективная);
  - worker restarts / crashes;
  - memory/CPU базово.
- **Tasks**
  - success/failure/retry rate по основным задачам;
  - execution time distribution (p50/p95/p99);
  - end-to-end latency distribution (если можешь измерять).
- **Result backend**
  - ошибки записи/чтения;
  - задержки/таймауты.

#### “Инцидент‑готовый” набор сигналов (когда уже серьёзно)

Добавляются:

- **Queue lag** (временной), а не только depth.
- **Сегментация** по:
  - queue,
  - task name,
  - tenant_id (если multi-tenant),
  - environment (prod/stage),
  - worker pool (CPU vs IO pools).
- **Retry rate** как ранний сигнал деградации внешней зависимости.
- **Worker utilization** и saturation по типу нагрузок:
  - CPU‑bound (CPU saturation),
  - I/O‑bound (connections/latency),
  - внешние API (rate limit / timeout).
- **DLQ/poison messages** (если брокерный паттерн применим).
 - Метрики в духе **USE/RED**:
   - Utilization/Saturation/Errors для воркеров и брокера;
   - Rate/Errors/Duration для задач и бизнес‑операций.

### Пошагово

Чек‑лист “что измерять”:

1. Опиши 3–5 **самых важных** классов задач (по бизнес‑ценности) и очереди, в которых они живут.
2. Для каждого класса задач зафиксируй:
   - какой у него целевой SLO по end-to-end,
   - какие внешние зависимости (БД, API),
   - допускается ли retry (и сколько).
3. Для каждой очереди добавь метрики:
   - depth,
   - lag (если доступно),
   - publish/consume rate.
4. Для worker-ов добавь:
   - количество активных процессов/потоков,
   - restarts,
   - memory per child (важно для prefork).
5. Для задач добавь:
   - duration (execution),
   - outcome (success/failure/retry),
   - попытка (attempt).
6. Проверь, что метрики позволяют ответить на вопрос:
   - “почему вырос end-to-end latency?” (очередь? выполнение? ретраи?)

### Простыми словами

Представь очередь в поликлинике:

- **queue depth** — сколько людей сидит в коридоре;
- **queue lag** — сколько времени последний человек уже сидит;
- **execution time** — сколько врач тратит на одного пациента;
- **end-to-end latency** — сколько времени пациент проводит от входа до выхода.

Если врач начал тратить больше времени на каждого — execution растёт. Если врач тот же, но людей больше — растёт lag. Если регистратура перестала выдавать талоны — publish падает. И “пациенту плохо” во всех случаях, но лечить нужно разное.

### Картинка в голове

Наблюдаемость — это “панель приборов” самолёта:

- отдельные датчики (метрики) по системам,
- журнал событий (логи),
- запись траектории полёта (трейсинг),
а не один единственный “скоростомер”.

### Как запомнить

Запомни три пары:

- **Depth vs lag**: “сколько лежит” vs “как давно ждёт”.
- **Execution vs end-to-end**: “как долго выполняли” vs “как долго пользователь ждал”.
- **Errors vs retries**: “сломалось” vs “деградирует”.

### Примеры

#### Пример 1. Почему “очередь 10k” может быть нормой

Если задачи микроскопические и у тебя стабильно высокая публикация, очередь может “жить” около некоторого уровня, при этом лаг будет маленьким. Поэтому глубину нужно читать вместе с:

- consume rate,
- lag,
- end-to-end latency.

#### Пример 2. Мини‑формула для burn‑down backlog

Если publish rate \(P\) и consume rate \(C\), то backlog изменяется как \(dB/dt = P - C\).

- Если \(C > P\), backlog “сгорает” (burn‑down).
- Если \(C < P\), backlog растёт (burn‑up).

### Практика / реальные сценарии

- **Сценарий**: “всё медленно после релиза”.
  - Смотри: retry rate вырос? execution p95 вырос? broker latency вырос? lag вырос?
- **Сценарий**: “внезапно выросли затраты”.
  - Смотри: частота задач (publish rate) и размер payload/результата, backend writes, high cardinality metrics.

### Типичные ошибки

- Пытаться мониторить Celery только по **количеству активных задач**.
- Не различать “задачи медленные” и “задачи ждут”.
- Делать метрики “в среднем по всем задачам”, не выделяя критичные.

### Что будет если…

- **…алёртить по queue depth?** Получишь ложные срабатывания при пиковой нагрузке или при “нормально высокой” очереди.
- **…не измерять retry rate?** Ты пропустишь раннюю деградацию внешнего API, пока ещё нет явных ошибок.

### Проверь себя

1. Почему метрика “среднее время выполнения задачи” часто опасна?

<details><summary>Ответ</summary>

Потому что среднее скрывает хвост распределения: p95/p99 могут резко вырасти, а среднее почти не изменится. В Celery именно хвост (редкие медленные выполнения) часто формирует end-to-end и ломает SLO.

</details>

2. Какая метрика быстрее всего покажет деградацию внешнего API, если у нас настроены ретраи?

<details><summary>Ответ</summary>

Рост **retry rate** и рост распределения длительности попыток (timeouts), иногда при этом общий failure rate может оставаться низким — ретраи “маскируют” падение до момента исчерпания лимитов.

</details>

3. Почему важно разделять метрики по queue и task name?

<details><summary>Ответ</summary>

Потому что разные очереди/типы задач имеют разные SLO, ресурсы и причины деградации. “Суммарная метрика” не покажет, что одна очередь горит, а остальные в норме.

</details>

### Запомните

- Наблюдаемость — это **вопросы → сигналы → инструменты**, а не наоборот.
- Depth без lag — почти всегда “полуправда”.
- Разделяй execution и end-to-end, иначе будешь лечить не то.
