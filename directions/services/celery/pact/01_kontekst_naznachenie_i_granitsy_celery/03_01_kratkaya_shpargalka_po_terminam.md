## Краткая шпаргалка по терминам

| Понятие                    | Суть простыми словами                                                                                         |
| -------------------------- | ------------------------------------------------------------------------------------------------------------- |
| **Celery**                 | Python-фреймворк для распределённого выполнения фоновых задач через broker и worker-ы.                        |
| **Task**                   | Описание единицы фоновой работы: что именно нужно выполнить.                                                  |
| **Producer**               | Код, который отправляет задачу в broker. Часто это веб-приложение, CLI или другая задача.                     |
| **Broker**                 | Посредник между отправителем и исполнителем. Хранит и маршрутизирует сообщения.                               |
| **Worker**                 | Процесс, который забирает задачи из очередей и исполняет их.                                                  |
| **Beat**                   | Планировщик периодических задач. Говоря просто: "кто и когда должен регулярно класть задачи в очередь".       |
| **Result backend**         | Подсистема, где можно хранить статусы и результаты задач. Нужна не всегда.                                    |
| **Queue**                  | Очередь сообщений, из которой worker получает задачи.                                                         |
| **Exchange / routing key** | Механизм маршрутизации сообщения в нужную очередь в broker-ах вроде RabbitMQ.                                 |
| **Signature**              | Объект Celery, который описывает задачу и её параметры как переиспользуемую "заготовку вызова".               |
| **ETA / countdown**        | Запланировать выполнение не сейчас, а позже: в конкретный момент времени или через задержку.                  |
| **Retry**                  | Повтор задачи после ошибки, вручную или автоматически.                                                        |
| **Revoke**                 | Попытка отменить задачу до исполнения или запретить её запуск, если она ещё не началась.                      |
| **Acknowledgment (ack)**   | Подтверждение broker-у, что сообщение получено и считается обработанным или взятым в работу.                  |
| **Prefetch**               | Сколько сообщений worker может заранее забрать себе, ещё не завершив исполнение.                              |
| **Worker pool**            | Модель конкурентного исполнения в worker-е: prefork, threads, gevent, solo и т.д.                             |
| **At-least-once delivery** | Реалистичная семантика: задача будет доставлена как минимум один раз, но возможны повторы.                    |
| **Idempotency**            | Свойство безопасно переживать повторное исполнение без удвоения эффекта.                                      |
| **Poison message**         | Плохое сообщение, которое стабильно ломает обработчик и отравляет очередь повторами.                          |
| **Chord unlock**           | Внутренняя механика Celery для завершения `chord`: когда группа задач закончилась и можно запускать callback. |

#### Проверь себя по шпаргалке

1. Почему `result backend` не является обязательной частью любой установки Celery?

<details><summary>Ответ</summary>

Потому что не всем задачам нужен возвращаемый результат. Очень часто важно лишь "выполни побочный эффект", а хранить статус каждой задачи дорого и иногда вредно. Result backend нужен по сценарию, а не по привычке.

</details>

2. Зачем вообще нужен broker, если producer и worker написаны на одном языке и находятся в одной системе?

<details><summary>Ответ</summary>

Broker даёт буфер, развязку по времени, маршрутизацию, возможность масштабировать producer-ы и worker-ы независимо и переживать кратковременные скачки нагрузки. Без него пришлось бы строить свой транспорт и свою очередь.

</details>

3. Почему знание терминов `ack`, `retry`, `prefetch` важно уже на старте, а не только на проде?

<details><summary>Ответ</summary>

Потому что именно эти термины определяют поведение системы при сбоях и под нагрузкой. Если их не понимать в начале, система может казаться "рабочей" до первого реального инцидента.

</details>
