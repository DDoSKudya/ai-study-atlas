## Справочник по части 1

| Тема                       | Ключевые пункты                                                                                                                     |
| -------------------------- | ----------------------------------------------------------------------------------------------------------------------------------- |
| **Что такое Celery**       | Distributed task queue; отдельный контур из producer, broker, worker; не просто декоратор.                                          |
| **Роль Celery**            | Вынесение и распределение фоновой работы, retries, delayed/periodic execution, масштабирование слоя обработки.                      |
| **Где Celery хорош**       | Интеграции, фоновые побочные эффекты, batch jobs, fan-out, периодические задачи, независимые worker-ы.                              |
| **Где Celery плох**        | Ultra-low latency, strict exactly-once, сложные workflow, слишком простые периодические задачи, локальный параллелизм без очередей. |
| **Главное ограничение**    | Практический мир at-least-once, а значит нужны идемпотентность и понимание failure modes.                                           |
| **Цена асинхронности**     | Eventual consistency, сложнее диагностика, новые точки отказа, зависимость от broker/backend.                                       |
| **Ключевые термины**       | task, signature, broker, queue, routing key, exchange, retry, revoke, ETA, ack, prefetch, worker pool.                              |
| **Главные мифы**           | "очередь гарантирует exactly-once", "backend нужен всегда", "можно передавать любые объекты", "retry всё чинит".                    |
| **Исторический вывод**     | Celery зрелый и широко используемый, но интернет полон legacy-паттернов.                                                            |
| **Главный принцип выбора** | Нужна ли именно queue-driven distributed processing, а не executor, cron, async I/O или workflow engine.                            |
