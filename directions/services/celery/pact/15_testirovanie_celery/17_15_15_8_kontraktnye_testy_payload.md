## 15.8 Контрактные тесты payload

### Цель раздела

Научиться фиксировать и тестировать формат аргументов задач (payload) так, чтобы изменения были управляемыми, совместимость сохранялась, а “старые сообщения в очереди” не ломали новые воркеры после деплоя.

### В этом разделе главное

- Payload задачи — это **внешний вход**, иногда даже “недоверенный”.
- Контракт payload нужно:
  - описывать (schema),
  - валидировать (на входе задачи),
  - тестировать (compatibility),
  - версионировать (version field).
- Самый частый “взрыв” после деплоя — несовместимый payload при непустой очереди.
- Контракт payload нужен не только для сериализации, но и для **отказоустойчивости**: неверные данные должны превращаться в понятный non‑retryable failure, иначе появится “шум” и backlog.

### Термины

| Термин | Определение |
|---|---|
| **Schema discipline** | Дисциплина схемы: явно описываем формат данных и эволюцию. |
| **Backward compatibility** | Совместимость назад: новый воркер понимает старый payload. |
| **Version field** | Поле версии в payload, чтобы безопасно эволюционировать формат. |

### Теория и правила

#### 1) Зачем schema/контракты, если “мы же в одном репозитории”

Потому что:

- деплой бывает rolling: часть воркеров старая, часть новая;
- очередь может быть непустой;
- сообщения могут “жить” дольше, чем одна версия кода;
- иногда задачи публикуют разные сервисы/команды.

#### 2) Минимальный контракт для payload

Хороший минимальный контракт:

- строго определённые поля и типы,
- “дополнительные поля допускаются/не допускаются”,
- наличие `version`,
- правила эволюции: “добавлять поля можно, удалять — только после периода совместимости”.

#### 2.1) Практическая эволюция: “добавляем поле” vs “меняем смысл поля”

Есть две очень разные операции:

- **добавить поле** (обычно безопаснее): старые воркеры могут игнорировать, новые — использовать;
- **изменить смысл/тип существующего поля** (опаснее): старые сообщения становятся “ядовитыми”.

Правило для реальных систем:

- добавления делаем через default‑значения и совместимые валидаторы,
- изменения смысла/типа — через **новое поле** или новую **версию payload**.

#### 3) Что тестировать

- payload версии v1 валиден по схеме v1;
- payload v1 принимается новой схемой/валидатором (backward compatibility), если так задумано;
- payload v2, наоборот, может требовать новой логики — это тоже должно быть явно.

#### 4) Что считать “несовместимой версией” и как это тестировать

Несовместимость — это не только “другое имя поля”. Частые варианты:

- поле стало обязательным, а в старых сообщениях его нет;
- тип поля изменился (`int` -> `str`) без адаптера;
- изменился смысл значения (раньше секунды, теперь миллисекунды);
- удалили поле, которое старая логика ещё использует.

Хороший contract‑тест на несовместимость:

- берёт “реальный старый payload” (фикстура/снэпшот),
- прогоняет через новую валидацию/декодирование,
- проверяет один из двух явных исходов:
  1) payload принят корректно (есть backward compatibility),
  2) payload отклонён контролируемо (понятная ошибка, без retry storm).

#### Проверь себя по несовместимым версиям payload

1. Почему изменение “единиц измерения” поля (секунды -> миллисекунды) считается несовместимостью даже без смены имени поля?

<details><summary>Ответ</summary>

Потому что меняется смысл значения, а не только форма. Старая логика интерпретирует данные иначе и может работать “формально успешно”, но бизнес‑результат будет неверным. Это один из самых опасных типов несовместимости.

</details>

2. Какие два корректных исхода должны быть у новой версии декодера для старого payload?

<details><summary>Ответ</summary>

Либо payload принимается и обрабатывается в режиме backward compatibility, либо отклоняется быстро и контролируемо (понятная ошибка без бессмысленных ретраев и шторма нагрузки).

</details>

### Пошагово: как внедрить contract‑tests для payload

1. Опиши schema для аргументов (например, JSON Schema или Pydantic model).
2. Внутри задачи валидируй вход на раннем этапе:
   - неверный payload → не ретраить (это non‑retryable).
3. Добавь тесты:
   - валидный payload проходит,
   - невалидный падает “правильно”,
   - старые версии payload остаются совместимыми (если это требование).
4. Включи в процесс деплоя: если меняется schema — должен быть план совместимости.

### Простыми словами

Контракт payload — это как “формат бланка”. Если завтра поменять бланк, а на складе лежат старые бланки, система должна уметь их читать (или хотя бы чётко и безопасно отбрасывать).

### Картинка в голове

```text
Непустая очередь при деплое:

  Сообщения v1 (в очереди)  ──►  Новый воркер (v2)
                                 │
                                 ├─ если совместимость есть → обработает
                                 └─ если совместимости нет → массовые падения/ретраи
```

### Как запомнить

**Очередь — это “время между версиями”. Контракт payload — мост через это время.**

### Примеры

#### Пример: версионность payload (идея)

```python
payload = {
    "version": 1,
    "invoice_id": 123,
    "force": False,
}
```

При переходе к v2:

- добавляй поля, сохраняя обработку v1,
- удаляй/меняй поля только после периода совместимости (dual‑read).

#### Пример: Pydantic‑модель как контракт (учебный)

Идея: модель явно задаёт типы и defaults, а тесты фиксируют, что v1/v2 работают так, как вы обещаете.

```python
from typing import Literal, Optional
from pydantic import BaseModel, Field

class InvoicePayloadV1(BaseModel):
    version: Literal[1] = 1
    invoice_id: int
    force: bool = False

class InvoicePayloadV2(BaseModel):
    version: Literal[2] = 2
    invoice_id: int
    force: bool = False
    reason: Optional[str] = Field(default=None, max_length=128)
```

Что тестировать:

- v1 валиден и не требует новых полей,
- v2 валиден, новые поля ограничены (например, длиной),
- если бизнес‑логика “обещает” backward compatibility, то новая задача принимает и v1, и v2 (через `Union`/discriminated union или через ручной switch по `version`).

Пример тестов “на смысл”:

```python
def test_v1_still_valid():
    p = InvoicePayloadV1.model_validate({"invoice_id": 10})
    assert p.version == 1
    assert p.force is False

def test_v2_optional_field():
    p = InvoicePayloadV2.model_validate({"invoice_id": 10, "reason": "manual"})
    assert p.reason == "manual"
```

Смысл этих тестов — не “проверить Pydantic”, а зафиксировать контракт вашей команды: что именно считается валидным payload.

#### Пример: тест на несовместимую версию payload (учебный)

```python
import pytest
from pydantic import ValidationError

def decode_payload(data: dict):
    version = data.get("version")
    if version == 1:
        return InvoicePayloadV1.model_validate(data)
    if version == 2:
        return InvoicePayloadV2.model_validate(data)
    raise ValueError(f"Unsupported payload version: {version}")

def test_unknown_version_is_rejected_fast():
    with pytest.raises(ValueError):
        decode_payload({"version": 999, "invoice_id": 1})

def test_incompatible_shape_fails_controlled():
    # пример: invoice_id пришёл строкой, а мы ожидаем int
    with pytest.raises(ValidationError):
        decode_payload({"version": 2, "invoice_id": "oops"})
```

Практический смысл: “непонятные/битые” payload должны отваливаться **быстро и предсказуемо**, а не уходить в бесконечные повторы.

### Практика / реальные сценарии

- Если задачи публикуются из разных сервисов, контракт‑тесты становятся критически важными.
- Если payload содержит PII, контракт должен явно запрещать/ограничивать такие поля (и тестами это фиксировать).

### Типичные ошибки

- “У нас только Python, значит schema не нужна” → потом ломается на rolling deploy.
- “Поменяли поле, но не подумали про старые сообщения” → очередь превращается в “poison messages”.
- Ретраить ошибки валидации payload → бесконечный шум.

### Что будет если…

- Если не версионировать payload, любая эволюция формата становится рискованной и дорогой.

### Проверь себя

1. Почему “непустая очередь” делает несовместимые изменения payload особенно опасными?

<details><summary>Ответ</summary>

Потому что новые воркеры после деплоя начнут обрабатывать старые сообщения, которые уже лежат в очереди. Если формат изменился несовместимо, воркеры будут падать массово на этих сообщениях, часто с ретраями, создавая шторм и простой.

</details>

2. Что лучше: ретраить ошибку валидации payload или быстро фейлить без retry? Почему?

<details><summary>Ответ</summary>

Быстро фейлить без retry (и, возможно, отправлять в “parking queue”/логировать). Ошибка валидации обычно не лечится повтором: данные неправильные, и повтор только создаст шум и нагрузку.

</details>

### Запомните

- Contract‑тесты payload защищают от самых неприятных регрессий после деплоя.
