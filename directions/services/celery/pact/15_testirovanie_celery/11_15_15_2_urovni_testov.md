## 15.2 Уровни тестов

### Цель раздела

Понять, какие уровни тестов нужны для Celery‑системы, какие гарантии даёт каждый уровень и как построить “пирамиду” так, чтобы тесты были быстрыми, полезными и не flaky.

### В этом разделе главное

- **Большинство тестов должны быть быстрыми (unit/contract).**
- Интеграционные и e2e — меньше по числу, но выше по ценности для критичных путей.
- Chaos/failure‑тесты — точечно и осознанно: их цель — проверить **устойчивость**, а не “покрытие”.
- Главный враг — **flaky**: тесты, завязанные на тайминги и случайность.

### Термины

| Термин | Определение |
|---|---|
| **Test pyramid** | Идея, что внизу много unit‑тестов, сверху меньше интеграционных и ещё меньше e2e. |
| **Test seam** | “Шов” в коде, где можно подменить внешний компонент (HTTP клиент, clock, random). |
| **Flake budget** | Допустимый уровень нестабильности (в идеале — ноль) и меры по его снижению. |

### Теория и правила

#### “Пирамида” для Celery‑задач

```text
                ┌──────────────────────────┐
                │ chaos / failure tests    │  (редко, но очень ценно)
           ┌────┴──────────────────────────┴────┐
           │ end-to-end with worker + broker     │  (мало, но реалистично)
     ┌─────┴────────────────────────────────────┴─────┐
     │ integration (broker/backend/DB)                 │  (умеренно)
┌────┴─────────────────────────────────────────────────┴────┐
│ unit + contract + wrapper (eager)                          │  (много, быстро)
└────────────────────────────────────────────────────────────┘
```

#### Что гарантирует каждый уровень (и что НЕ гарантирует)

| Уровень | Что проверяет | Что не проверяет |
|---|---|---|
| **Unit** | логика, инварианты, ветвления | транспорт, сериализацию “как в проде”, конкуренцию |
| **Contract** | формат payload, версии, валидация | доставка, ack, retry‑механику |
| **Wrapper (eager)** | параметры task, базовую связку вызова, часть исключений | реальный worker/pool, конкуренцию, broker/backend |
| **Integration** | реальные компоненты (broker/backend/DB), сериализация на деле | прод‑нагрузку, сложные гонки, редкие сбои |
| **E2E** | реальный worker исполняет, сообщения проходят как в проде | всё многообразие прод‑сбоев (нужны failure tests) |
| **Chaos/failure** | поведение при отказах, redelivery, деградациях | “покрытие логики”, скорость выполнения |

### Пошагово: как собрать тест‑стратегию для проекта

1. Для каждой важной задачи составь карточку:
   - что делает,
   - какие side effects,
   - какие ошибки возможны,
   - какая критичность.
2. Назначь “обязательный минимум”:
   - unit тесты на логику,
   - contract тесты на payload,
   - тест на идемпотентность (хотя бы “повторный запуск не ломает”).
3. Для критичных задач добавь:
   - integration тест с реальным broker/backend,
   - e2e с worker (хотя бы один сценарий “happy path”).
4. Для самых критичных добавь 1–2 failure‑сценария:
   - временная ошибка внешней зависимости → retry,
   - worker crash/timeout/redelivery → задача не делает двойной эффект.

### Простыми словами

Не пытайся “всё тестировать e2e”. Это как ездить на полигон каждый раз, когда меняешь болтик. Нужны разные проверки: быстрые — каждый раз, тяжёлые — реже, но обязательно для ключевых рисков.

### Картинка в голове

Пирамида — это экономия времени команды. Чем выше тест, тем он:

- дороже (поднять инфраструктуру),
- медленнее,
- более хрупкий к окружению,
- но и ближе к прод‑реальности.

### Как запомнить

**Снизу — уверенность в логике, сверху — уверенность в реальности.**

### Примеры

#### Пример набора тестов для одной задачи “отправить webhook”

- Unit:
  - строится корректный payload,
  - правильно выбирается endpoint,
  - корректно обрабатываются коды ответов (200/429/500).
- Contract:
  - schema payload версии v1 валидируется,
  - при добавлении поля v2 старая версия остаётся валидной.
- Wrapper:
  - задача ретраится только на `TimeoutError`/сетевые ошибки,
  - max_retries = N.
- Integration:
  - реальный Redis/RabbitMQ + реальный backend,
  - мок HTTP‑сервиса через локальный тест‑сервер.
- Failure:
  - временный 500 → backoff → успех,
  - “повторный publish того же logical job” не создаёт дубль side effect (например, по idempotency key).

### Практика / реальные сценарии

- Для workflow (chord/group) делай хотя бы один e2e тест на “partial failure”.
- Для multi‑tenant систем тестируй изоляцию: “tenant A не влияет на tenant B” (например, раздельные очереди/префиксы).

### Что включать в failure/chaos‑слой на практике (минимум)

Чтобы раздел 15.2 не оставался абстракцией, вот минимальный “боевой” набор chaos/failure‑проверок для критичных задач:

1. **Downstream timeout/503**:
   - задача ретраится по политике,
   - нет дубля side effect.
2. **Broker кратковременно недоступен**:
   - producer/consumer не “тихо теряют” сообщения,
   - наблюдаемое состояние ошибки есть в логах/метриках.
3. **Worker restart в середине выполнения**:
   - задача либо корректно повторяется, либо корректно завершается по дизайну,
   - бизнес‑эффект не дублируется.
4. **Result backend деградирует** (если важны статусы/chord):
   - система не зависает “тихо”,
   - есть понятная деградация/эскалация.

Смысл: chaos‑слой не “для красоты”, а для самых дорогих failure modes.

#### Проверь себя по блоку failure/chaos

1. Почему для критичных задач важно отдельно проверять `worker restart` в середине выполнения, даже если все unit/integration тесты зелёные?

<details><summary>Ответ</summary>

Потому что restart проверяет именно поведение в условиях прерывания рантайма: redelivery, повтор попытки, частично выполненные side effects и корректность идемпотентности. Unit и базовые integration чаще подтверждают корректность логики/интеграции, но не моделируют этот класс сбоев полноценно.

</details>

2. Чем полезна проверка деградации result backend для workflow, если сам broker работает стабильно?

<details><summary>Ответ</summary>

Для chain/group/chord backend часто нужен как место хранения промежуточных результатов и статусов. Даже при живом broker workflow может “повисать”, не собирать результат или не запускать callback при проблемах backend. Поэтому тест backend‑деградации проверяет отдельный критичный риск.

</details>

### Типичные ошибки

- Делать только e2e: тесты медленные → их перестают запускать → уверенность падает.
- Делать только unit: всё зелёное, но в проде ломается сериализация/оркестрация.
- Писать интеграционные тесты без изоляции: тесты начинают конфликтовать и флейкать.

### Что будет если…

- Если “пирамиду” перевернуть (много e2e) — CI станет медленным, команда начнёт пропускать тесты, а ошибки уйдут в прод.

### Проверь себя

1. Какие два уровня тестов чаще всего дают максимальную пользу за минимальную стоимость?

<details><summary>Ответ</summary>

Unit‑тесты (чистая логика) и contract‑тесты (payload/схема). Они быстрые, детерминированные и ловят большой класс проблем заранее.

</details>

2. Что именно даёт e2e тест с реальным worker такого, чего почти невозможно гарантировать unit‑тестами?

<details><summary>Ответ</summary>

Реальную доставку/десериализацию через broker, запуск в рантайме worker (процессы/пулы), взаимодействие с backend, и реальную “динамику” выполнения задачи как отдельного процесса/потока исполнения.

</details>

3. Почему chaos/failure‑тесты не должны быть “многочисленными”?

<details><summary>Ответ</summary>

Потому что они дорогие и часто менее детерминированные. Их цель — проверить устойчивость по ключевым failure modes, а не покрыть все ветки логики.

</details>

### Запомните

- Строй тест‑контур слоями.
- Не путай “всё зелёное” с “устойчиво в проде”.
