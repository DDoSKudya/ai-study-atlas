## 15.5 Тестирование идемпотентности

### Цель раздела

Научиться тестировать ключевое для Celery свойство production‑устойчивости: задача может быть выполнена повторно (redelivery/повторная публикация/конкуренция), и система всё равно не должна делать неправильный бизнес‑эффект.

### В этом разделе главное

- Идемпотентность — это про **эффект**, а не про “код не падает”.
- Нужно тестировать:
  - повторный запуск одной и той же задачи,
  - искусственную эмуляцию redelivery,
  - конкурентный запуск близких задач,
  - повторный publish одного logical job.
- Идемпотентность почти всегда требует **ключа** (idempotency key) и/или **инварианта в хранилище** (unique constraint, state machine, lock с TTL, outbox).

### Термины

| Термин | Определение |
|---|---|
| **Idempotency key** | Ключ, который связывает “логическую работу” и защищает от дублей эффекта. |
| **Deduplication** | Дедупликация: распознавание повторов и “мягкое” игнорирование. |
| **Concurrency race** | Гонка: два воркера одновременно пытаются сделать один эффект. |

### Теория и правила

#### 1) Какие “повторы” бывают в реальности

- **Redelivery**: сообщение доставили повторно (ack не дошёл, worker умер).
- **Повторная публикация producer‑ом**: клиент/сервис не получил подтверждение и отправил ещё раз.
- **Retry**: Celery повторяет после ошибки.
- **Конкурентный запуск**: две похожие задачи стартуют почти одновременно (например, два события на один объект).

Важно: эти сценарии выглядят по‑разному, но эффект один — **логическая работа может выполниться больше одного раза**.

#### 2) Что тестировать, чтобы действительно проверить идемпотентность

Тест должен утверждать не “не упало”, а:

- side effect **в БД** создан ровно один раз,
- внешний вызов **не сделан дважды** (или сделан безопасно с idempotency key),
- событие/уведомление не отправлено повторно,
- состояние доменной сущности корректно (state machine не перескочила).

### Пошагово: тест‑набор для идемпотентности одной задачи

1. Выбери “ключ логической работы” (например, `payment_id`, `invoice_id`, `event_id`).
2. Сформулируй эффект: “после выполнения есть запись/статус/событие”.
3. Напиши тест повторного запуска:
   - выполнить задачу дважды с одинаковым logical key,
   - убедиться, что эффект один.
4. Напиши конкурентный тест (если это реально важно):
   - два параллельных запуска,
   - эффект один.
5. Напиши тест “повторная публикация”:
   - смоделируй, что producer отправил одинаковое сообщение дважды,
   - эффект один.

### Пошагово: как искусственно эмулировать redelivery (практично)

Частая проблема: redelivery “сложно воспроизвести” в тесте. Рабочий учебный подход:

1. Считай, что redelivery эквивалентен сценарию “та же logical работа пришла повторно”.
2. В integration/e2e‑тесте запусти задачу с одним `logical_id`.
3. Затем повторно отправь задачу с тем же `logical_id` (это эмуляция redelivery/повторного publish).
4. Утверди:
   - side effect один,
   - статус/лог корректно отражает “дедупликацию” (если это часть контракта).

Почему это корректно: для бизнес‑эффекта неважно, *почему* пришёл повтор (retry/redelivery/повторная публикация), важно, что эффект защищён.

#### Проверь себя по эмуляции redelivery

1. Почему в учебных и прикладных тестах допустимо моделировать redelivery как повторную отправку того же `logical_id`?

<details><summary>Ответ</summary>

Потому что ключевой инвариант — устойчивость бизнес‑эффекта к повторному исполнению. Для проверки этого инварианта источник повтора (retry, redelivery, повторная публикация) вторичен. Если один и тот же logical key безопасен к повтору, главный риск дублей уже контролируется.

</details>

2. Какие два утверждения обязательны в тесте эмуляции redelivery, кроме “задача не упала”?

<details><summary>Ответ</summary>

Во‑первых, side effect должен быть единственным (без дубля). Во‑вторых, система должна наблюдаемо и корректно отражать повтор (например, через статус/лог дедупликации), если это часть контракта команды.

</details>

### Простыми словами

Идемпотентность — это “если я нажму кнопку два раза, система не сделает два списания”.

### Картинка в голове

```text
Повтор исполнения задачи (норма)  ─────►  Защита эффекта (обязательна)

  execute #1  ──► эффект применён ┐
                                  ├──► итог: эффект один (dedup)
  execute #2  ──► попытка повторить┘
```

### Как запомнить

**Celery живёт в at‑least‑once. Значит “повтор” — это не баг, а режим работы.**

### Примеры

#### Пример: тест повторного исполнения (идея)

```python
def test_idempotency_double_run(db, run_task):
    # run_task — условная helper-функция, которая запускает задачу и ждёт завершения
    logical_id = "order:123"

    run_task("process_order", logical_id=logical_id)
    run_task("process_order", logical_id=logical_id)

    # утверждаем эффект: в таблице ровно одна запись “processed”
    assert db.count_processed_orders(logical_id) == 1
```

Ключевой момент: assert должен быть про **эффект**, а не про “task returned ok”.

#### Пример: конкурентный запуск “почти одновременно” (учебная идея)

Даже простой concurrent‑тест полезен, если в нём проверяется именно инвариант:

```python
from concurrent.futures import ThreadPoolExecutor

def test_idempotency_concurrent(db, run_task):
    logical_id = "invoice:42"

    with ThreadPoolExecutor(max_workers=2) as ex:
        f1 = ex.submit(run_task, "process_invoice", logical_id=logical_id)
        f2 = ex.submit(run_task, "process_invoice", logical_id=logical_id)
        f1.result()
        f2.result()

    assert db.count_processed_invoices(logical_id) == 1
```

Важно: этот тест не “доказывает отсутствие всех гонок”, но отлично ловит базовые нарушения инварианта.

#### Проверь себя по конкурентному запуску

1. Почему успешный concurrent‑тест с двумя потоками не доказывает, что гонок “больше нет”?

<details><summary>Ответ</summary>

Потому что гонки зависят от множества факторов: таймингов, нагрузочного профиля, числа воркеров, очередей, внешних зависимостей и редких межпоточных/межпроцессных интерливингов. Один тест закрывает базовый сценарий, но не полное пространство состояний.

</details>

2. Что в таком тесте должно быть главным assert: результат функции или состояние внешнего эффекта? Почему?

<details><summary>Ответ</summary>

Главный assert — состояние внешнего эффекта (БД/события/уведомления), потому что именно там проявляется нарушение идемпотентности. Return value может быть одинаковым при дублированном бизнес‑эффекте.

</details>

### Практика / реальные сценарии

- Для внешних HTTP интеграций используй idempotency key (если API поддерживает).
- Для внутренних эффектов используй уникальные ограничения или таблицу дедупликации (outbox/inbox).
- Если задача обновляет состояние доменной сущности, добавляй тест “state transition не прыгает назад/через шаг при повторе”.

### Типичные ошибки

- “Мы считаем, что задача выполнится ровно один раз” → потом redelivery.
- Дедупликация “в памяти процесса” → не работает при рестартах/скейле.
- Тестировать идемпотентность только “последовательными” запусками и забыть конкуренцию.

### Что будет если…

- Если не тестировать конкуренцию, можно получить rare‑bug: “раз в неделю двойная отправка письма/двойная запись”.

### Проверь себя

1. Чем отличается redelivery от retry, и почему для идемпотентности это почти неважно?

<details><summary>Ответ</summary>

Retry — повтор по инициативе Celery после ошибки; redelivery — повторная доставка сообщения из‑за транспортных/ack причин. Для идемпотентности важно одно: логическая работа может быть выполнена повторно, поэтому эффект должен быть защищён.

</details>

2. Почему дедупликация “в памяти воркера” не является надёжной?

<details><summary>Ответ</summary>

Потому что воркер может перезапуститься, может быть несколько воркеров, может быть параллельность. Память одного процесса не является общим и долговечным источником истины.

</details>

3. Что должен проверять тест идемпотентности: return value задачи или side effects? Почему?

<details><summary>Ответ</summary>

Side effects. Именно они создают бизнес‑эффект (запись в БД, событие, письмо). Return value может быть одинаковым, а эффект — продублированным.

</details>

### Запомните

- Идемпотентность проверяется по эффекту, и требует тестов повторов и конкуренции.
