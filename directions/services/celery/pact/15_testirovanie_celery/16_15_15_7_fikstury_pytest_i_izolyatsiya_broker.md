## 15.7 Фикстуры, pytest и изоляция broker

### Цель раздела

Научиться строить тестовую инфраструктуру так, чтобы Celery‑тесты были воспроизводимыми: общий способ создавать app, поднимать broker/backend в CI, изолировать очереди, не конфликтовать при параллельных тестах и корректно чистить состояние.

### В этом разделе главное

- Самая частая причина flaky тестов — **shared окружение** (одна очередь на всех, один namespace, отсутствие очистки).
- Лучше “изолировать по имени” (префиксы/уникальные очереди), чем “глобально чистить всё”.
- Очистка очередей опасна в shared env — можно снести чужие сообщения.
- Для Celery‑интеграционных тестов важно заранее решить: **мы используем Redis/RabbitMQ как брокер** и **какой backend**. Для chord/group backend нужен практически всегда.

### Термины

| Термин | Определение |
|---|---|
| **Namespace/prefix** | Префикс имён очередей/ключей, чтобы тесты разных запусков не пересекались. |
| **Ephemeral broker** | Брокер, поднятый на время тестов (например, в контейнере). |
| **Parallel tests** | Параллельный запуск тестов (pytest-xdist), создающий гонки за общие ресурсы. |

### Теория и правила

#### 1) Три подхода к изоляции broker в тестах

1) **Самый надёжный**: отдельный broker на запуск (контейнер на CI job).  
2) **Компромисс**: общий broker, но разные **vhost/DB/index** или разные credentials/namespace.  
3) **Рискованный**: общий broker и “мы чистим очереди перед тестами” (может снести чужое).

#### 1.1) Минимальный “тестовый контур” для разных типов задач

| Что вы тестируете | Нужно поднять | Почему |
|---|---|---|
| только unit/contract | ничего | тесты без инфраструктуры |
| “простая задача через брокер” | broker | проверяем publish/consume |
| задача + результат (`AsyncResult.get`) | broker + backend | иначе результат неоткуда читать |
| chord/group/chain сложнее | broker + backend (обязательно) | оркестрации нужна “точка сборки результатов” |
| overlap lock на Redis | broker + Redis (как lock store) | иначе lock не проверить |

Это помогает не “перетестировать” (поднимать лишнее) и не “недотестировать” (думать, что проверили workflow без backend).

#### 2) Что сделать, чтобы параллельные тесты не конфликтовали

- Генерировать уникальный префикс (например, `TEST_RUN_ID` = uuid).
- Конструировать имена очередей/route key с этим префиксом.
- Не использовать глобальные фиксированные имена очередей для тестов.

#### 2.1) Про практический “namespace”: что именно префиксить

Хорошее правило: если ресурс может быть общим — он должен получать префикс `TEST_RUN_ID`.

Обычно префиксуют:

- имена очередей,
- routing key,
- exchange (если вы явно задаёте),
- ключи в Redis (если Redis используется как lock/result store),
- имена “парковочных” очередей (DLQ‑паттерны).

Идея: даже если тесты падают и не успели “прибраться”, следующий прогон не столкнётся с “мусором”.

#### 3) Почему “очистка очередей” — это не универсальная таблетка

Потому что:

- может быть shared брокер (dev‑окружение) и ты снесёшь чужие задачи;
- не всегда ясно, какие очереди надо чистить (routing);
- очистка сама по себе может быть гонкой: кто‑то уже опубликовал новые сообщения.

#### 4) Полезный контракт для CI: быстрый и полный профили

Чтобы тестовая стратегия реально жила в команде, обычно фиксируют два профиля:

- **fast profile** (PR по умолчанию):
  - unit + contract + часть wrapper‑тестов,
  - без тяжёлой инфраструктуры.
- **full profile** (merge/nightly/release):
  - integration/e2e/failure для критичных задач,
  - с реальным broker/backend и worker.

Иначе команда либо “тонет” в медленных тестах, либо перестаёт проверять важные сценарии.

#### Проверь себя по fast/full профилям

1. Что обычно должно входить в fast profile и почему?

<details><summary>Ответ</summary>

Unit, contract и часть wrapper‑тестов. Они быстрые, детерминированные и дают быстрый feedback в PR, не перегружая разработческий цикл тяжёлой инфраструктурой.

</details>

2. Почему full profile лучше запускать на merge/nightly/release, а не на каждый локальный запуск?

<details><summary>Ответ</summary>

Потому что full profile дорогой по времени и окружению (broker/backend/worker/failure‑тесты). Если требовать его в каждом локальном цикле, команда начнёт избегать запусков. Разделение профилей сохраняет и скорость, и надёжность.

</details>

### Пошагово: базовый каркас pytest‑фикстур для Celery проекта (идея)

1. Сделай “фабрику” приложения Celery, куда можно передать broker/backend URL.
2. В тестах передавай URL из окружения (CI) или используй локальные значения.
3. Добавь фикстуру `test_run_id`.
4. Делай очереди/ключи с этим префиксом.
5. Для integration/e2e тестов: отдельный маркер `@pytest.mark.integration`, чтобы локально можно было гонять быстрый слой.

### Примеры: минимальный учебный каркас фикстур (без привязки к конкретному репозиторию)

Ниже — “скелет”, который показывает, как обычно делают:

- фикстуру `celery_app` для тестов,
- уникальные очереди по `test_run_id`,
- отдельные настройки для integration‑тестов через env.

```python
import os
import uuid

import pytest
from celery import Celery

@pytest.fixture(scope="session")
def test_run_id() -> str:
    return os.environ.get("TEST_RUN_ID") or uuid.uuid4().hex

@pytest.fixture(scope="session")
def celery_broker_url() -> str:
    # В CI обычно задаётся через переменные окружения
    return os.environ.get("CELERY_BROKER_URL", "redis://localhost:6379/0")

@pytest.fixture(scope="session")
def celery_result_backend() -> str:
    return os.environ.get("CELERY_RESULT_BACKEND", "redis://localhost:6379/1")

@pytest.fixture()
def celery_app(test_run_id: str, celery_broker_url: str, celery_result_backend: str) -> Celery:
    app = Celery("tests", broker=celery_broker_url, backend=celery_result_backend)
    app.conf.update(
        task_default_queue=f"{test_run_id}.default",
        task_default_exchange=f"{test_run_id}.exchange",
        task_default_routing_key=f"{test_run_id}.default",
        task_always_eager=False,  # integration слой: через broker
        task_ignore_result=False, # если нужно читать результат
    )
    return app
```

Это не “универсальная истина”, а понятная база: видно, где включается изоляция и как CI подсовывает URLs.

### Примеры: docker compose для брокера в CI (учебный)

Если CI позволяет поднимать сервисы, самый простой способ уменьшить флейки — сделать брокер “эпемерным” на job.

Пример для Redis (как broker и/или backend):

```yaml
services:
  redis:
    image: redis:7
    ports:
      - "6379:6379"
```

Ключевая идея: тесты получают `CELERY_BROKER_URL`/`CELERY_RESULT_BACKEND` на этот Redis.

Если вы используете RabbitMQ — концепция та же: отдельный контейнер на job и изоляция по vhost/namespace.

#### Проверь себя по CI-брокеру

1. Почему “эпемерный broker на job” обычно снижает flaky?

<details><summary>Ответ</summary>

Потому что у каждого запуска тестов своё изолированное состояние: нет конфликтов с соседними джобами, нет остатков чужих сообщений и меньше скрытых гонок между независимыми процессами CI.

</details>

2. Что важнее при настройке CI: “сделать как можно проще” или “сделать как можно ближе к прод‑брокеру”?

<details><summary>Ответ</summary>

Нужен баланс: fast‑слой можно делать проще, но критичные integration/e2e сценарии должны быть максимально близки к прод‑контексту, иначе самые дорогие ошибки будут обнаруживаться слишком поздно.

</details>

### Формула раздела

**Изоляция — это не “уборка после теста”, а “архитектура тестового окружения до запуска”.**

### Простыми словами

Тесты любят жить “в своей песочнице”. Если все играют в одной песочнице, они мешают друг другу и ломают замки (flaky).

### Картинка в голове

```text
Shared broker без изоляции
  test A ─┐
          ├──► очередь "default" ◄──┤  test B
  test C ─┘

Изоляция префиксом
  test run id = "r1" → очередь "r1.default"
  test run id = "r2" → очередь "r2.default"
```

### Как запомнить

**Изолируй именами, а чисти осторожно.**

### Примеры

#### Пример: маркировка тестов (идея)

```python
import pytest

@pytest.mark.integration
def test_task_through_broker():
    ...
```

В CI можно запускать:

- быстрые: `pytest -m "not integration"`
- полные: `pytest -m "integration"`

### Практика / реальные сценарии

- В больших репозиториях обязательно делай “контракт”: integration тесты не должны запускаться по умолчанию локально, иначе их перестанут запускать вообще.
- В командах с общими dev‑окружениями строго запрети “purge shared queue” из тестов.

### Типичные ошибки

- Один broker URL и одна очередь на весь CI → непредсказуемые падения.
- Тесты зависят от порядка запуска.
- Тесты “чистят всё” и ломают соседние job‑ы.
- Использовать один Redis DB и для broker, и для backend, и для lock‑ов без дисциплины ключей → сложнее изолировать и диагностировать.

### Что будет если…

- Если тесты flaky, команда начнёт их игнорировать. Дальше ошибки снова начнут попадать в прод.

### Проверь себя

1. Что безопаснее: чистить общую очередь перед тестом или сделать уникальную очередь на тест‑ран? Почему?

<details><summary>Ответ</summary>

Уникальная очередь/namespace безопаснее, потому что не затрагивает чужие сообщения и устраняет конфликты параллельных тестов. Очистка общей очереди может снести чужие задачи и сама по себе создаёт гонки.

</details>

2. Почему интеграционные тесты стоит маркировать отдельно?

<details><summary>Ответ</summary>

Потому что они медленнее, требуют инфраструктуры и иногда менее детерминированы. Отдельный маркер позволяет быстро гонять unit/contract слой локально и запускать полный набор в CI.

</details>

### Запомните

- Хорошие Celery‑тесты — это не только код, но и правильная изоляция окружения.
