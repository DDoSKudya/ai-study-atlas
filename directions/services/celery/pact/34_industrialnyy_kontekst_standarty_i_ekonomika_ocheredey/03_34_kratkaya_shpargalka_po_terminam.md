## Краткая шпаргалка по терминам

| Термин | Простыми словами |
|---|---|
| **AMQP 0-9-1** | Протокол обмена сообщениями (очереди, exchange, routing) в основе RabbitMQ-сценариев. |
| **Task message (Celery)** | Сообщение определенного формата, которое описывает вызов задачи Celery. |
| **Managed queue** | Очередь как облачный сервис, где инфраструктуру поддерживает провайдер. |
| **Serverless compute** | Выполнение кода без управления серверами, обычно с оплатой за вызов/время. |
| **FinOps** | Практика управления облачными затратами как инженерным процессом. |
| **Cost driver** | Фактор, который толкает стоимость вверх или вниз (CPU, память, хранение, egress, retries). |
| **Unit economics** | Экономика на единицу работы, например стоимость обработки одной задачи. |
| **Ownership model** | Модель ответственности: кто владеет очередью, задачами, алертами и runbook. |
| **Error budget** | Допустимый объем деградации сервиса в рамках SLA/SLO. |
| **Internal API contract** | Формализованные правила по payload, retries, versioning и совместимости задач. |

#### Проверь себя: термины

1. Чем отличается `Task message` от "любого сообщения в брокере"?

<details><summary>Ответ</summary>

Task message Celery имеет структуру и семантику, ожидаемую worker-ом Celery (идентификаторы, имя задачи, аргументы, headers, options). Обычное сообщение в брокере может быть чем угодно и не обязано быть исполнимым Celery.

</details>

2. Что дает метрика "стоимость одной задачи" на практике?

<details><summary>Ответ</summary>

Она позволяет сравнивать альтернативы (например, Celery vs managed workflow), оценивать окупаемость оптимизаций и обосновывать технические решения бизнесу на языке денег, а не только latency.

</details>
