## Краткая шпаргалка по терминам

| Термин | Суть простыми словами |
|---|---|
| **Production-эксплуатация (ops)** | Практика “как это живёт”: деплой, наблюдаемость, инциденты, восстановление. |
| **Rolling update** | Обновление без полного остановa: часть worker-ов уже новая, часть ещё старая. |
| **Graceful shutdown** | Остановка процесса так, чтобы он корректно завершил работу (или корректно “отпустил” её). |
| **Drain** | “Слить” нагрузку: перестать брать новые задачи и дождаться завершения текущих. |
| **SLO фона** | Цель по качеству фоновой обработки: например, “95% задач завершены < 2 минут”. |
| **Queue depth** | Сколько сообщений лежит в очереди сейчас. |
| **Lag** | Насколько “отстаёт” обработка: грубо, как долго сообщение ждёт до начала выполнения. |
| **Throughput** | Сколько задач обрабатывается в единицу времени. |
| **Autoscaling thrashing** | Дёргание масштаба вверх-вниз из-за плохого сигнала/порогов. |
| **Runbook** | Пошаговая инструкция, что делать при конкретном инциденте. |
| **DR (Disaster Recovery)** | План восстановления после аварии: что сохраняем, что пересчитываем, как воспроизводим. |
| **Backward compatibility (совместимость назад)** | Новая версия worker умеет обработать старый формат сообщений. |
| **Poison message** | Сообщение, которое стабильно ломает обработчик и вызывает бесконечные повторы. |
| **OOM / SIGKILL** | Процесс убит из-за нехватки памяти (в контейнерах часто без graceful shutdown). |
| **PDB (PodDisruptionBudget)** | Ограничение в Kubernetes: сколько pod можно одновременно “выселить”, чтобы сервис не умер. |
| **KEDA** | Автоскейлер, который умеет масштабировать по сигналам очередей (queue length/lag). |
| **Correlation ID** | Идентификатор для связывания логов/трейсов запроса и фоновой задачи. |

#### Проверь себя: термины

1. Чем “queue depth” отличается от “lag”, и почему depth сам по себе может обманывать?

<details><summary>Ответ</summary>

Depth — количество сообщений в очереди сейчас. Lag — “время ожидания” до начала обработки. Depth может быть небольшим, но lag огромным (например, если сообщения приходят редко, но worker их долго не берёт или есть hot partition). И наоборот: depth может быть большим, но lag приемлемым, если throughput высокий и backlog быстро убывает.

</details>

2. Почему “graceful shutdown” может не сработать в Kubernetes, даже если код/worker умеет?

<details><summary>Ответ</summary>

Потому что Kubernetes может убить pod не через SIGTERM+grace period, а через OOMKill (SIGKILL), node eviction, или слишком маленький `terminationGracePeriodSeconds`. В этих случаях процесс не успевает корректно “drain” текущие задачи.

</details>

3. Почему runbook — это не “бумажка для галочки”, а часть архитектуры?

<details><summary>Ответ</summary>

Потому что фоновые контуры почти неизбежно попадают в сбои. Если нет заранее описанной процедуры действий и метрик, команда будет действовать на ощупь, что увеличивает downtime, риск потери данных и вероятность “лечить симптомы” (например, просто перезапускать всё, создавая ещё больше дубликатов).

</details>
