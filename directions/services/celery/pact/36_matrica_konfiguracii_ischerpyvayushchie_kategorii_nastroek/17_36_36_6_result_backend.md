## 36.6 Result backend

### Цель раздела

Понять, как настроить backend результатов так, чтобы он помогал эксплуатации, но не превращался в дорогое и хрупкое "второе хранилище бизнеса".

### В этом разделе главное

- backend и broker выполняют разные роли;
- `result_expires` и cleanup strategy обязательны;
- chord/group сценарии чувствительны к надежности backend.

### Термины

| Термин | Формально | Простыми словами |
|---|---|---|
| `result_backend` | Хранилище состояний/результатов задач | Где хранится статус выполнения |
| `result_expires` | TTL результатов | Как долго хранить результат |
| `result_extended` | Расширенные метаданные результата | Больше деталей для диагностики |
| `result_backend_always_retry` | Retry для операций backend | Повтор записи/чтения при временных сбоях |
| `result_chord_join_timeout` | Таймаут ожидания завершения chord header | Сколько ждать группу перед callback |

### Теория и правила

1. **Backend должен быть "операционным", а не бизнес-источником истины.**
2. **TTL обязателен.**  
   Без `result_expires` объем данных растет бесконтрольно.
3. **`result_extended` полезен для расследований, но увеличивает нагрузку на storage.**
4. **Chord-сценарии чувствительны к consistency backend.**  
   Потеря/задержка результата может ломать fan-in логику.

### Пошагово

1. Реши, действительно ли нужны результаты для всех задач.
2. Настрой `ignore_result` для fire-and-forget задач.
3. Выстави разумный `result_expires` и задачу очистки.
4. Для chord/group протестируй деградацию backend.
5. Настрой retry-политику backend под инфраструктуру.

### Простыми словами

Result backend — это "журнал статусов", а не "архив бизнес-документов". Он нужен для контроля исполнения, не для долгого хранения доменных данных.

### Картинка в голове

Это табло трекинга доставки: полезно знать, где посылка сейчас, но хранить в табло полный бухгалтерский архив неразумно.

### Как запомнить

**Result backend = операционный след, не бизнес-источник.**

### Примеры

```python
result_backend = "redis://redis:6379/1"
result_backend_always_retry = True
result_backend_transport_options = {"retry_policy": {"timeout": 5.0}}

result_expires = 86400  # 1 day
result_extended = True
result_cache_max = 10000
result_chord_join_timeout = 15.0
result_chord_retry_interval = 1.0
```

### Практика / реальные сценарии

- **UI прогресса задач:** хранить только необходимый набор полей и ограничивать TTL.
- **Heavy batch:** отключать результаты для промежуточных шагов, оставлять только контрольные.
- **Chord-heavy workflow:** усиливать мониторинг backend latency/error.

### Типичные ошибки

- хранить "вечные" результаты без политики очистки;
- использовать backend как аудит-лог бизнес-событий;
- не учитывать backend как отдельную зону отказа.

### Что будет, если...

- **...backend деградирует:** задачи могут выполняться, но клиенты увидят ложные `PENDING`/ошибки чтения состояния.
- **...результаты хранятся слишком долго:** рост стоимости и замедление операций чтения/очистки.

### Проверь себя

1. Почему `result_expires` важен даже при небольшом потоке задач?
2. Когда стоит использовать `task_ignore_result=True`?
3. Почему chord-сценарии часто "проваливают" слабые backend-конфигурации?

<details><summary>Ответ</summary>

1) Потому что накопление идет постоянно, и через недели/месяцы даже небольшой поток становится значимым по объему и стоимости.  
2) Для задач, где результат не нужен ни пользователю, ни следующему шагу workflow.  
3) Потому что chord требует согласованного учета завершения множества задач; задержки/потери в backend бьют по синхронизации callback.

</details>

### Запомните

Result backend должен быть минимально достаточным: полезен для управления, но ограничен по сроку и объему.

### Граничные случаи для backend (часто недооцениваются)

- **Большой `result_extended` + длинный TTL**: диагностически удобно, но стоимость хранения и латентность чтения растут нелинейно.
- **`task_ignore_result=True` в задачах, участвующих в chord/group**: можно потерять корректную синхронизацию fan-in.
- **Слишком маленький `result_cache_max` при high-QPS чтении статусов**: растет давление на backend.
- **Слишком большой `result_cache_max` при ограниченной памяти worker/client процесса**: риск memory pressure.

#### Проверь себя: edge cases backend

1. Почему “длинный TTL + много метаданных” почти всегда бьет по стоимости и latency?
2. В каком случае `task_ignore_result=True` становится ошибкой архитектуры?

<details><summary>Ответ</summary>

1) Накопление результатов и расширенных полей со временем раздувает storage и замедляет операции чтения/очистки.  
2) Когда результат нужен для orchestration (chord/group) или пользовательского прогресса.

</details>
