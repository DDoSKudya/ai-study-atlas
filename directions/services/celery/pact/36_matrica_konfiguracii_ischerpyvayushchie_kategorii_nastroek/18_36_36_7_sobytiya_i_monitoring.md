## 36.7 События и мониторинг

### Цель раздела

Понять, как event-параметры влияют на наблюдаемость и инфраструктурную стоимость.

### В этом разделе главное

- `worker_send_task_events` и `task_send_sent_event` дают видимость lifecycle;
- очереди событий имеют TTL/expiration и не должны бесконтрольно расти;
- monitoring без governance может стоить дороже, чем ожидается.

### Термины

| Термин | Формально | Простыми словами |
|---|---|---|
| `worker_send_task_events` | Публикация событий исполнения задач worker-ом | "Рассказывай мониторингу, что происходит" |
| `task_send_sent_event` | Событие "задача отправлена" на этапе publish | Видимость ранней стадии пути |
| `event_queue_ttl` / `event_queue_expires` | Время жизни event-очередей | Как долго держать служебные события |
| `event_queue_prefix` | Префикс имен event-очередей | Удобство сегментации/изоляции |
| `event_serializer` | Формат событий | В каком формате шлется telemetry |

### Теория и правила

1. **События важны для диагностики, но это дополнительный поток данных.**
2. **TTL для event queues обязателен.**  
   Иначе monitoring-поток может стать отдельной проблемой.
3. **Уровень детализации должен соответствовать целям observability.**  
   Нельзя "включить все навсегда" без контроля cost.

### Пошагово

1. Зафиксируй, какие вопросы мониторинг должен отвечать.
2. Включи нужные события, но ограничь их TTL.
3. Настрой дашборды и алерты на ключевые переходы состояний.
4. Регулярно пересматривай объем event-трафика.

### Простыми словами

Event stream — это "черный ящик самолета": бесценен в расследовании, но его нужно грамотно хранить и обслуживать.

### Картинка в голове

Если писать все подряд без фильтра и сроков хранения, "черный ящик" превращается в свалку, где трудно найти нужный сигнал.

### Как запомнить

**Наблюдаемость = полезные события + контролируемый объем.**

### Примеры

```python
worker_send_task_events = True
task_send_sent_event = True

event_queue_prefix = "celeryev"
event_queue_ttl = 60
event_queue_expires = 120
event_queue_persistent = False
event_serializer = "json"
```

### Практика / реальные сценарии

- **Incident response:** включенные events ускоряют root cause analysis.
- **High throughput контур:** часть low-value событий может быть выключена ради стоимости.
- **Мультисервисная среда:** префиксы event queue помогают изолировать потоки команд.

### Типичные ошибки

- включить события, но не добавить алерты/дашборды;
- не ограничить TTL и получить рост служебных очередей;
- считать Flower "достаточным monitoring", игнорируя системные метрики.

### Что будет, если...

- **...events выключены:** меньше нагрузки, но в инциденте почти нет временной линии.
- **...events слишком много без фильтра:** рост стоимости и шум, затрудняющий диагностику.

### Проверь себя

1. Почему `worker_send_task_events` полезен даже без Flower?
2. Зачем нужны `event_queue_ttl` и `event_queue_expires` одновременно?
3. Как понять, что event-уровень детализации избыточен?

<details><summary>Ответ</summary>

1) Потому что события можно использовать в любом мониторинговом пайплайне, не только в одном UI.  
2) TTL ограничивает жизнь сообщений, expires — жизнь очередей/ресурсов; вместе они удерживают служебный слой под контролем.  
3) Когда объем/стоимость телеметрии растут, а практическая ценность в инцидентах не увеличивается.

</details>

### Запомните

Наблюдаемость должна быть достаточной для расследования, но экономной для повседневной работы.

### Минимальный набор operational-метрик, привязанный к конфигу

| Метрика | Почему нужна | С какими опциями связана |
|---|---|---|
| Queue lag / queue depth | ранний индикатор перегруза | routing, prefetch, concurrency, rate limit |
| Task success/failure/retry rate | здоровье task policy | retry settings, time limits, serializer issues |
| Task duration p95/p99 | SLA и capacity planning | pool/concurrency/prefetch/time limits |
| Broker reconnect count | сетевые/infra проблемы | broker retry/heartbeat/SSL |
| Result backend latency/error | корректность статусов/chord | backend retry/transport options/result settings |
| Event throughput volume | стоимость observability | event flags, TTL/expires/persistence |

#### Проверь себя: operational-метрики

1. Почему нельзя опираться только на `success rate`?
2. Какие две метрики дают ранний сигнал о деградации очередного контура?

<details><summary>Ответ</summary>

1) Можно иметь высокий success rate при критическом росте задержек, дублей и стоимости.  
2) Обычно `queue lag/depth` и `retry rate` (в связке с p95/p99 latency).

</details>
