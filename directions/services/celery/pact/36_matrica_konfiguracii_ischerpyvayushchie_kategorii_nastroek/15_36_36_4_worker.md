## 36.4 Worker

### Цель раздела

Понять, как runtime-настройки worker-а влияют на throughput, fairness, memory profile и управляемость в эксплуатации.

### В этом разделе главное

- `worker_pool`, `worker_concurrency`, `worker_prefetch_multiplier` определяют динамику выполнения;
- `worker_max_tasks_per_child` и `worker_max_memory_per_child` — практические инструменты борьбы с утечками;
- логирование, events и remote control — часть control plane, а не "декорация".

### Термины

| Термин | Формально | Простыми словами |
|---|---|---|
| `worker_pool` | Тип пула (`prefork`, `threads`, `gevent`, `eventlet`, `solo`) | Модель параллелизма |
| `worker_concurrency` | Количество параллельных единиц выполнения | Сколько задач делаем одновременно |
| `worker_prefetch_multiplier` | Сколько задач worker резервирует заранее | Насколько "жадно" забираем очередь |
| `worker_max_tasks_per_child` | Перезапуск child после N задач | Профилактика накопленных утечек |
| `worker_max_memory_per_child` | Перезапуск child при лимите памяти | Страховка от runaway памяти |
| `worker_enable_remote_control` | Разрешение broadcast-команд управления | Можно ли удаленно управлять worker |

### Теория и правила

1. **Prefetch управляет балансом latency vs fairness.**  
   Большой prefetch повышает локальный throughput, но может зажимать другие worker-ы.
2. **Pool выбирается по природе workload.**  
   CPU-bound часто лучше в `prefork`, I/O-bound может выигрывать от green threads.
3. **Memory limits и max tasks per child — практический must-have для долгоживущих процессов.**
4. **Remote control полезен, но увеличивает поверхность управления.**  
   В изолированных контурах его часто ограничивают сетевыми правилами или выключают.

### Пошагово

1. Классифицируй workload (CPU/I-O/mixed).
2. Подбери pool и concurrency через нагрузочные тесты, а не "по числу CPU".
3. Отстрой prefetch под нужный баланс latency/fairness.
4. Введи memory guardrails (`max_tasks_per_child`, `max_memory_per_child`).
5. Зафиксируй logging/event policy и доступ к remote control.

### Простыми словами

Worker — это фабрика. Pool — тип станков, concurrency — сколько станков, prefetch — сколько сырья станки забирают к себе заранее.

### Картинка в голове

Если один цех забрал слишком много заготовок (высокий prefetch), соседние цеха простаивают. Склад вроде полон, а общая линия работает неэффективно.

### Как запомнить

**Pool выбирает форму параллелизма, prefetch выбирает форму справедливости.**

### Примеры

```python
worker_pool = "prefork"
worker_concurrency = 8
worker_prefetch_multiplier = 1

worker_max_tasks_per_child = 1000
worker_max_memory_per_child = 512000  # KB

worker_send_task_events = True
worker_enable_remote_control = True
worker_state_db = "/var/run/celery/worker.state"
worker_timer_precision = 1.0
```

```python
worker_log_format = "[%(asctime)s:%(levelname)s/%(processName)s] %(message)s"
worker_task_log_format = (
    "[%(asctime)s:%(levelname)s/%(processName)s] "
    "%(task_name)s[%(task_id)s]: %(message)s"
)
worker_hijack_root_logger = False
worker_redirect_stdouts = True
worker_redirect_stdouts_level = "INFO"
```

### Практика / реальные сценарии

- **Высокая кардинальность коротких задач:** prefetch может быть >1, если при этом не страдает fairness.
- **Долгие неравномерные задачи:** prefetch=1 почти всегда снижает starvation.
- **Интеграционный сервис с memory spikes:** `max_memory_per_child` обязателен.

### Типичные ошибки

- выбирать concurrency по принципу "чем больше, тем лучше";
- держать prefetch высоким на длинных задачах и получать queue starvation;
- включить events без планирования объема и стоимости хранения метрик/логов.

### Что будет, если...

- **...отключить limits и memory guardrails:** через часы/дни uptime worker станет нестабилен из-за накопленных утечек или фрагментации.
- **...оставить remote control открытым без сетевой сегментации:** увеличится риск нежелательных операционных воздействий.

### Проверь себя

1. Почему `worker_prefetch_multiplier` часто ставят в `1` для "тяжелых" задач?
2. Зачем одновременно использовать и `max_tasks_per_child`, и `max_memory_per_child`?
3. Когда имеет смысл выключить `worker_enable_remote_control`?

<details><summary>Ответ</summary>

1) Чтобы worker не "захватывал" пачки задач и не блокировал справедливое распределение при длинных исполнениях.  
2) Один лимит срабатывает по счетчику задач, второй по фактическому росту памяти; вместе они устойчивее к разным профилям утечек.  
3) В жестко изолированных или high-security контурах, где управление проводится только через контролируемые механизмы деплоя/рестарта.

</details>

### Запомните

Worker-параметры — это управление очередью в реальном времени: скоростью, справедливостью и стабильностью.

### Визуал: как prefetch и время задач влияют на fairness

```text
Очередь: [L][L][S][S][S][S]   (L=long, S=short)

Случай A: prefetch=4, два worker
W1 забрал [L][L][S][S]
W2 забрал [S][S]
=> short-задачи у W1 ждут завершения long-задач (локальная "пробка")

Случай B: prefetch=1
W1 берет [L]
W2 берет [L]
далее оба по очереди получают [S][S][S][S]
=> более справедливое распределение, меньше tail latency для short
```

#### Проверь себя: prefetch и fairness

1. Почему высокий prefetch может ухудшать p99 при внешне нормальном throughput?
2. Когда prefetch > 1 может быть оправдан?

<details><summary>Ответ</summary>

1) Короткие задачи оказываются заблокированы за длинными внутри одного worker и растет хвостовая задержка.  
2) Когда workload однородный и задачи в среднем короткие.

</details>
