## Краткая шпаргалка по терминам

| Термин | Простыми словами |
|---|---|
| **Configuration matrix** | Карта всех категорий настроек Celery с зависимостями, а не просто список ключей. |
| **Task policy** | Набор правил, как задачи сериализуются, подтверждаются, ретраятся и ограничиваются. |
| **Transport options** | Специфичные ключи поведения для конкретного брокера (Redis/Rabbit/SQS). |
| **Worker runtime profile** | Параметры, которые задают поведение процесса/пула worker в рантайме. |
| **Result consistency** | Насколько корректно и вовремя сохраняются состояния/результаты задач в backend. |
| **Control plane** | Каналы и команды управления (`inspect`, `control`, event stream). |
| **Config drift** | Расхождение фактической конфигурации между окружениями или между узлами. |
| **Safe rollout** | Контролируемое внедрение изменений конфигурации с метриками и откатом. |
| **Deprecated option** | Устаревшая настройка, которая может исчезнуть в следующем релизе Celery. |
| **Blast radius** | Масштаб последствий ошибочного изменения параметров. |

#### Проверь себя: шпаргалка терминов

1. Чем `config drift` отличается от обычной "неаккуратной настройки"?
2. Почему `blast radius` нужно оценивать до изменения, а не после?

<details><summary>Ответ</summary>

1) `Config drift` — это системное расхождение фактических настроек между окружениями/узлами, а не разовая ошибка.  
2) Потому что он определяет стратегию canary и отката до того, как произойдет ущерб.

</details>
