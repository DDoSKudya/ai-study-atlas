## Каталог анти-паттернов конфигурации (с последствиями и исправлением)

Этот раздел нужен, чтобы читатель не только знал “как правильно”, но и сразу узнавал типовые разрушительные конфигурации.

| Анти-паттерн | Почему возникает | Реальное последствие | Как исправлять |
|---|---|---|---|
| `task_always_eager=True` в production | переменная окружения “утекла” из тестов | исчезает реальная semantics очереди, ложная стабильность, деградация latency API | жесткое разделение env-профилей + CI policy на запрет eager в prod |
| единая очередь для всех классов задач | “так проще на старте” | starvation, tail latency, приоритеты не работают | `task_routes` + выделенные очереди и worker-пулы |
| высокий prefetch при длинных задачах | попытка поднять throughput без теста fairness | короткие задачи ждут за длинными, растет p99 | prefetch=1 для long-running классов, очереди по workload |
| бесконтрольный retry | вера, что retry “лечит всё” | retry storm, перегрев broker/downstream | bounded retry, jitter/backoff, классификация ошибок |
| широкое `accept_content` | совместимость “на будущее” | рост рисков небезопасной десериализации | JSON-only baseline, исключения только по бизнес-обоснованию |
| вечное хранение результатов | желание “не потерять ничего” | рост стоимости, замедление backend, сложная очистка | `result_expires`, cleanup policy, `ignore_result` где возможно |
| включенные events без TTL | “пусть будет больше наблюдаемости” | шум и стоимость telemetry, служебные очереди разрастаются | `event_queue_ttl`/`event_queue_expires`, оставлять только полезные события |
| апгрейд без ревизии deprecated | нехватка времени перед релизом | silent ignore ключей, непредсказуемое поведение после релиза | обязательный migration checklist и staged rollout |

### Проверь себя: анти-паттерны

1. Почему “единая очередь для всего” особенно опасна при росте команды?
2. Что хуже: отсутствие retry или неконтролируемый retry storm?
3. Почему анти-паттерны чаще всего выявляются только в инцидентах?

<details><summary>Ответ</summary>

1) Потому что разные команды добавляют задачи с разными SLA в один контур, и конфликт приоритетов становится системным.  
2) Обычно хуже retry storm: он одновременно увеличивает нагрузку, скрывает первопричину и ломает соседние системы.  
3) В спокойном режиме многие дефекты скрыты; они проявляются под нагрузкой, отказами и сетевыми флуктуациями.

</details>
