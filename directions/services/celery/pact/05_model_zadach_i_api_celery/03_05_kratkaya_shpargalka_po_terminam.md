## Краткая шпаргалка по терминам

| Понятие | Суть простыми словами |
| --- | --- |
| **Task (задача)** | Именованная единица фоновой работы: что выполнить и с какими аргументами. |
| **`@app.task`** | Декларация: «эта Python-функция — Celery-задача» + опции по умолчанию. |
| **`bind=True`** | Вызов задачи как метода: первый аргумент — экземпляр `Task` (`self`), доступен `self.request`. |
| **`delay()` / `apply_async()`** | Способы **опубликовать** задачу (отправить сообщение), а не выполнить локально. |
| **`send_task()`** | Вызов задачи **по строковому имени**, без импорта функции из модуля worker. |
| **ETA / countdown / expires** | Когда исполнить и до какого момента работа ещё имеет смысл (осторожно с TZ и семантикой брокера). |
| **Signature (`s`, `si`, `signature`) + `.set()`** | «Заготовка вызова»: имя задачи + аргументы + опции; **`.set(queue=…, headers=…)`** накапливает опции доставки до `apply_async` или canvas. |
| **Immutable signature (`si`)** | Сигнатура, которую **нельзя случайно дополнять** аргументами при композиции — важно для canvas. |
| **`self.request`** | Контекст исполнения: id, попытки, метаданные доставки, hostname, заголовки. |
| **`Ignore` / `Reject` / `Retry`** | Способы **закончить** обработку сообщения с разной семантикой для брокера/worker. |
| **`task_id`** | Идентификатор **конкретного запуска**; им можно управлять для дедупликации (осторожно). |
| **`acks_late`** | Ack **после** выполнения (обычно): лучше для потери работы при падении, но выше риск повторов. |
| **`autoretry_for` + backoff + jitter** | Автоматические повторы при классах ошибок + экспоненциальная задержка + случайный разброс. |
| **`time_limit` / `soft_time_limit`** | Жёсткий и «мягкий» лимит времени исполнения. |
| **`ignore_result`** | Не писать результат/метастатус в result backend (если он настроен) — экономия и меньше шума. |
| **`headers` / `priority` / `routing_key`** | Тонкие параметры **сообщения** и маршрутизации; часть возможностей зависит от брокера. |
| **`signature()`** | Фабричная форма: имя задачи строкой + args/kwargs/options — удобно без прямой ссылки на объект задачи. |
| **`Reject`** | Отказ обработать сообщение с передачей решения брокеру (requeue/delete — зависит от брокера и кода). |
| **`TaskRevokedError`** | Исключение/сигнал: задача **отозвана** (revoke), исполнение прервано по политике worker. |
| **`AsyncResult`** | Ручка к result backend по `task_id`; не путать с фактом исполнения; `get()` в веб — только с таймаутом. |
| **`@shared_task`** | Регистрация на «дефолтном» app (типично Django); отличие от `@app.task` — см. § 5.1. |

#### Проверь себя: шпаргалка

1. Чем **`si()`** принципиально отличается от **`s()`** в одном предложении про композицию canvas?

<details><summary>Ответ</summary>

**`si` (immutable)** не должен неявно накапливать аргументы при сборке цепочек; **`s`** допускает последующее добавление args/kwargs при композиции — удобно для ветвления, но рискованно для колбэков.

</details>

2. Почему **`send_task`** важен не только «динамическим именем», но и **границе микросервисов**?

<details><summary>Ответ</summary>

Producer может **не импортировать** Python-модуль worker: достаточно **контракта строкового имени** и сериализуемого payload — это типичная граница между сервисами с разным деплоем.

</details>

3. Свяжи в цепочку причину–следствие: **`acks_late`** → какой продуктовый риск без других мер?

<details><summary>Ответ</summary>

Поздний ack повышает шанс **повторной доставки** после сбоя → без **идемпотентности** возможны **двойные** побочные эффекты; `acks_late` меняет компромисс «потеря работы vs дубль».

</details>
