## Частые сценарии

| Сценарий | Как рассуждать |
| --- | --- |
| «Задача выполнилась дважды» | at-least-once + отсутствие идемпотентности; проверь `acks_late`, crash worker, retry |
| «В логах нет correlation» | добавь headers на публикации + лог на worker |
| «В AsyncResult вечно PENDING» | backend выключен, `ignore_result`, или задача не та, что думаешь |
| «Ретраи убивают API» | backoff+jitter+rate_limit+узкие `autoretry_for` |
| «Нужно вызвать из другого сервиса» | контракт имён + `send_task` + мониторинг версий |
| «Сообщение не десериализуется на worker» | проверь `serializer`/`accept_content`, версии кода, pickle vs json; отлавливай на границе и решай reject/DLQ, а не бесконечный retry |
| «Очередь крутит одно и то же ядовитое сообщение» | классификация ошибок: `Reject(requeue=False)` или DLQ, алерт по метрике, не `autoretry_for(Exception)` |
| «После `revoke` статус в БД не совпал с реальностью» | помни: отзыв и бизнес-флаг не синхронизированы автоматически; синхронизируй политикой и логами |
| «Дважды нажали «отправить» — в очереди одна работа?» | два `apply_async` = две публикации; одинаковый `task_id` **не** заменяет дедуп: нужен claim/уникальный ключ на API или в БД (§ 5.8) |
