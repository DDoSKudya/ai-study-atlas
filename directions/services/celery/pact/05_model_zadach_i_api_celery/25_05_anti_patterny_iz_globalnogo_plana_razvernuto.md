## Анти-паттерны из глобального плана (развёрнуто)

**Задача читает ORM-транзакцию «родителя».** Родитель — HTTP-процесс. Worker — другой процесс. Транзакционные границы не переносятся. Решение: закоммитьте и передайте id; в worker откройте свою сессию.

**Сетевой fan-out без таймаутов.** Одна задача дергает 50 сервисов синхронно → один зависший вызов держит воркер. Решение: таймауты, ограничение параллелизма, отдельные очереди.

**`except Exception: pass` как ретрай-политика.** Ты теряешь сигнал и ломаешь метрики/алерты. Решение: логируй, классифицируй, используй структурированные ретраи.

**Почему это особенно коварно в Celery:** внешне задача может завершиться «без ошибки», а бизнес-эффект **не произойдёт** или произойдёт частично; в очереди не сработают ни retry, ни DLQ, ни алерт по exception rate.

```python
# Плохо: молчаливое проглатывание
try:
    charge(order_id)
except Exception:
    pass  # «потом разберёмся» — не разберётесь без логов и метрик

# Лучше: классификация и наблюдаемость
try:
    charge(order_id)
except TransientPartnerError:
    raise self.retry(countdown=30)
except PermanentDataError:
    logger.exception("charge.invalid", extra={"order_id": order_id})
    raise
```
