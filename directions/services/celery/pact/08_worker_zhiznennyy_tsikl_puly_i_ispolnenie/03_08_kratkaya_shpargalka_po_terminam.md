## Краткая шпаргалка по терминам

| Понятие | Суть простыми словами |
| --- | --- |
| **Worker** | Процесс (или группа процессов), который получает задачи из брокера и исполняет их. |
| **Bootsteps** | Внутренние шаги инициализации worker-а перед началом потребления задач. |
| **Consumer loop** | Основной цикл получения и обработки сообщений из брокера. |
| **Warm shutdown** | "Мягкая" остановка: новые задачи не берем, текущие завершаем по возможности корректно. |
| **Cold shutdown** | "Жесткая" остановка: процессы прерываются, часть задач может оборваться на середине. |
| **Pool** | Модель конкуренции worker-а: процессы, потоки или кооперативные greenlet-ы. |
| **Prefork** | Pool на процессах: основной путь для CPU-bound и изоляции исполнения. |
| **Threads pool** | Исполнение задач в потоках одного процесса; ограничено GIL для Python-кода. |
| **Gevent/Eventlet** | Кооперативная конкуренция для I/O-нагрузки с monkey patching. |
| **Prefetch** | Сколько задач worker резервирует заранее, до фактического исполнения. |
| **Ack** | Подтверждение брокеру, что задача обработана. |
| **`acks_late`** | Ack после выполнения задачи (меньше потерь, больше вероятности дублей). |
| **`task_reject_on_worker_lost`** | Поведение при потере worker-процесса: сообщение может быть возвращено для redelivery. |
| **`max_tasks_per_child`** | Перезапуск child-процессов после N задач для борьбы с утечками/фрагментацией. |
| **`max_memory_per_child`** | Перезапуск child-процесса при превышении лимита памяти. |
| **Queue affinity** | Привязка worker-а к конкретным очередям/типам задач. |
| **Canary worker** | Небольшой контур worker-ов для безопасной проверки новых изменений. |
