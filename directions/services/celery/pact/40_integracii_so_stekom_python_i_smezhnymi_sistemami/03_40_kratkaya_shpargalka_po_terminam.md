## Краткая шпаргалка по терминам

| Термин | Простыми словами |
|---|---|
| **`transaction.on_commit`** | Запускает публикацию задачи только после успешного commit в БД. |
| **Session scope** | Границы жизни ORM-сессии: когда создается, где используется, когда закрывается. |
| **Fork safety** | Гарантия, что клиент/соединение корректно переживает `fork` процесса. |
| **Payload contract** | Формальное описание, какие поля и типы можно передавать в задачу. |
| **Pydantic `TypeAdapter`** | Инструмент валидации/сериализации произвольных типов в Pydantic v2. |
| **Idempotent side effect** | Побочный эффект, который безопасно повторить без двойного результата. |
| **Connection pool per process** | Пул соединений должен принадлежать конкретному worker-процессу. |
| **Dependency Injection (DI)** | Явная передача зависимостей вместо скрытых глобальных объектов. |
| **Checkpointing** | Сохранение прогресса batch-задачи так, чтобы можно было продолжить с места сбоя. |
| **Warm model** | ML-модель уже загружена в память worker-процесса до первой задачи. |

#### Проверь себя: шпаргалка

1. Почему payload contract важнее "удобства быстро передать объект"?
2. Чем DI полезен именно для тестирования задач Celery?

<details><summary>Ответ</summary>

1) Он защищает совместимость и предсказуемость сообщений между версиями и сервисами.  
2) Позволяет подменять внешние зависимости (клиенты, репозитории, настройки) без хрупких monkeypatch-хакав.

</details>
