## Production-чеклист интеграции (быстрый аудит перед релизом)

- [ ] Публикация задач, зависящих от БД, идет после commit (`on_commit`/аналог).
- [ ] В payload нет ORM-инстансов, файловых дескрипторов и несериализуемых объектов.
- [ ] Для HTTP/gRPC вызовов заданы timeout и согласованный retry budget.
- [ ] Проверена fork-safety клиентов и стратегия per-process lifecycle.
- [ ] Выбранный pool соответствует фактическому профилю нагрузки (CPU/I/O/async).
- [ ] Зависимости задачи подменяемы в тестах через явный test seam.
- [ ] Для ETL/ML есть checkpoint-store и policy повторного запуска.
- [ ] Добавлены базовые метрики: queue latency, task duration, retry count, external error rate.

#### Проверь себя: production-чеклист

1. Какие 2 пункта чеклиста дадут максимальный эффект снижения инцидентов в первую очередь?
2. Что произойдет, если пройти чеклист только “по ощущениям”, без тестового подтверждения?

<details><summary>Ответ</summary>

1) Обычно publish-after-commit и timeout/retry budget, потому что они закрывают самые частые и дорогие классы сбоев.  
2) Возникнет ложное чувство готовности, и риски проявятся в проде при реальной нагрузке.

</details>
