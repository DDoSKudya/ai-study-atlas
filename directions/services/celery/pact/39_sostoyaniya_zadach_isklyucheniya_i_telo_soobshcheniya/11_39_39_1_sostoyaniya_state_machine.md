## 39.1 Состояния (state machine)

### Цель раздела

Собрать точную модель состояний Celery, чтобы ты уверенно отвечал на вопросы "что происходит с задачей сейчас", "это уже финал или еще переход", и "почему `PENDING` не всегда значит одно и то же".

### В этом разделе главное

- Состояние задачи — это не только "label", а договор о стадии жизненного цикла.
- Одно и то же визуальное состояние может иметь разный смысл в разных конфигурациях.
- `PENDING` особенно коварен: он часто означает "у меня нет достоверных данных", а не "задача точно еще не начата".
- Переходы зависят от result backend, `task_track_started`, eager-режима и retry-политики.

### Термины

| Термин | Формально | Простыми словами |
|---|---|---|
| **Ready states** | Состояния, после которых задача считается завершенной логически (`SUCCESS`, `FAILURE`, обычно `REVOKED`) | Задача уже "дошла до итога", больше не исполняется |
| **Unready states** | Состояния ожидания/промежуточной обработки (`PENDING`, `STARTED`, `RETRY`) | Работа либо не началась, либо еще идет/перезапланирована |
| **Terminal state** | Конечный статус (с учетом политики вашей системы) | Финальная точка в вашем мониторинге |

### Теория и правила

1. **`PENDING`**  
   Формально: backend не знает о результате задачи.  
   Простыми словами: "информация о задаче отсутствует или еще не записана".  
   Важное следствие: при отключенном result backend почти все задачи со стороны клиента будут выглядеть как `PENDING`.

2. **`STARTED`**  
   Появляется, если включено отслеживание старта (`task_track_started`).  
   Без этой настройки задача может перескочить визуально из `PENDING` сразу в `SUCCESS/FAILURE`.

3. **`SUCCESS` / `FAILURE`**  
   Финальные статусы обычного исполнения.  
   `FAILURE` хранит traceback/exception metadata (если backend это поддерживает и хранение не отключено политикой).

4. **`RETRY`**  
   Не финал. Это "контролируемое решение попробовать позже".  
   Частая ошибка: считать `RETRY` сразу инцидентом. На самом деле это может быть нормальная политика resilience.

5. **`REVOKED`**  
   Административная/политическая отмена задачи.  
   Интерпретация зависит от момента отмены: до старта, во время старта, после получения worker-ом.

6. **`REJECTED`**  
   Признак, что сообщение отвергнуто на пути потребления/валидации.  
   Поведение после reject зависит от transport/queue настроек (`requeue` и брокерной политики).

   Важный нюанс наблюдаемости: в разных версиях/конфигурациях `REJECTED` чаще виден как событие и лог-факт, чем как "устойчивый state" в result backend через `AsyncResult`.
   Поэтому для анализа `REJECTED` всегда нужны минимум два источника: event/log + broker/runtime-контекст.

7. **Переходы в eager-режиме (`task_always_eager=True`)**  
   Задача исполняется в процессе вызывающего кода, без реального прохождения через broker/worker lifecycle.  
   Практический риск: разработчик привык к "мгновенным" и линейным переходам, а в production получает другую динамику и другие точки отказа.

8. **Переходы при отключенном backend**  
   На стороне клиента state-модель становится неполной: формально задача может уже завершиться, но внешний наблюдатель все еще видит `PENDING`.  
   Поэтому для production без backend обязательны альтернативные источники истины: структурированные логи, event stream, бизнес-метрики факта выполнения.

### Пошагово: как читать состояние задачи без самообмана

1. Проверяй, есть ли result backend и каков его retention policy.
2. Уточняй, включен ли `task_track_started`.
3. Для `PENDING` всегда проверяй: задача не отправлена, отправлена но не потреблена, или статус не записан.
4. Для `RETRY` смотри причину retry и максимальное число попыток.
5. Для `FAILURE` разделяй бизнес-исключение и инфраструктурный сбой исполнения.
6. Для `REVOKED/REJECTED` анализируй, кто инициировал действие и что произошло с очередью.
7. Для спорных случаев сверяй "триаду источников истины": `AsyncResult` + worker events/logs + broker telemetry.

### Простыми словами

Состояния Celery похожи на статусы посылки, но с оговоркой: иногда трек-номер есть, а логистическая система временно не дает полную картину. `PENDING` в таком случае означает "данные о движении не подтверждены", а не "посылка точно лежит на складе".

### Картинка в голове

Представь табло в аэропорту:

- `PENDING` — рейс в системе есть, но статус не обновился;
- `STARTED` — самолет начал руление;
- `RETRY` — вылет перенесен на позже;
- `SUCCESS` — прилет состоялся;
- `FAILURE` — рейс отменен по технической причине;
- `REVOKED` — рейс снят административно.

### Как запомнить

Формула: **"Статус без контекста backend и runtime = опасная полуправда"**.

### Примеры

```python
from celery import Celery

app = Celery("demo")
app.conf.update(
    task_track_started=True,  # иначе STARTED обычно не увидишь
    task_ignore_result=False, # если True, клиенту будет сложнее читать финал
)

@app.task(bind=True, autoretry_for=(ConnectionError,), retry_backoff=True, max_retries=5)
def sync_invoice(self, invoice_id: str):
    # бизнес-логика
    return {"invoice_id": invoice_id, "status": "synced"}
```

Пример интерпретации результата:

```python
result = sync_invoice.delay("INV-1001")
print(result.state)  # может быть PENDING сразу после отправки

# через некоторое время:
print(result.state)  # STARTED / RETRY / SUCCESS / FAILURE
```

Сравнение поведения в eager и async-режиме:

```python
# eager: задача выполняется в том же процессе
app.conf.task_always_eager = True
res = sync_invoice.delay("INV-2001")
print(res.state)  # обычно сразу SUCCESS/FAILURE (зависит от eager settings)

# async: реальный broker/worker путь
app.conf.task_always_eager = False
res = sync_invoice.delay("INV-2002")
print(res.state)  # сначала чаще PENDING, затем переходы по runtime
```

### Практика / реальные сценарии

- **Сценарий 1: "Все задачи висят в PENDING"**  
  Проверка часто показывает: backend отключен или неправильно настроен URL backend-а.
- **Сценарий 2: "STARTED никогда не видно"**  
  Причина: не включен `task_track_started`; это не обязательно ошибка исполнения.
- **Сценарий 3: "RETRY выглядит как падение"**  
  На деле это штатная реакция на временную ошибку внешнего API.

### Типичные ошибки

- читать `PENDING` как "точно не запустилось";
- считать `RETRY` финальным провалом;
- не фиксировать, какие состояния команда считает terminal для алертинга;
- смешивать данные из Flower, backend и логов без приоритета источников истины.

### Что будет, если...

- **...игнорировать `PENDING`-семантику без backend:**  
  команда начнет дублировать задачи "на всякий случай", повышая риск двойных побочных эффектов.
- **...включить `task_track_started` без оценки нагрузки:**  
  возрастет поток status updates; на больших объемах это влияет на backend/storage.

### Мини-runbook: сложный случай `REVOKED` vs фактический side effect

1. Определи момент revoke относительно фактического старта задачи (по логам worker и событиям).
2. Проверь, был ли выполнен внешний side effect до сигнала отмены.
3. Если side effect уже произошел, применяй компенсационную операцию по бизнес-процедуре.
4. Обнови policy: где revoke допустим, а где нужны только "мягкие" механизмы остановки.
5. Зафиксируй в runbook, что `REVOKED` не всегда означает "ничего не произошло".

### Проверь себя

1. Почему `PENDING` не равно "очередь точно не доставила задачу"?
2. В чем принципиальная разница между `RETRY` и `FAILURE`?
3. Почему для SLO важно заранее договориться, какие статусы считаются "финальными"?

<details><summary>Ответ</summary>

1) Потому что `PENDING` говорит об отсутствии подтвержденного состояния в backend, а не о фактическом ходе доставки/исполнения.  
2) `RETRY` — переход к новой попытке (не финал), `FAILURE` — финальный неуспех текущей задачи без дальнейших попыток (если не переотправить заново).  
3) Иначе метрики и алерты будут противоречивыми: одна команда считает задачу завершенной, другая — еще в процессе.

</details>

### Запомните

- Состояния читаются только вместе с конфигурационным контекстом.
- `PENDING` — самый частый источник ошибочной диагностики.
- `RETRY` — инструмент надежности, а не автоматический признак "все плохо".
- Eager-режим полезен для локальной проверки, но искажает production-картину переходов.
