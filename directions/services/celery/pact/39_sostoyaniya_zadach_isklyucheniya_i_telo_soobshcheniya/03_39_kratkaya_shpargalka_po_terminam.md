## Краткая шпаргалка по терминам

| Термин | Простыми словами |
|---|---|
| **Task state machine** | Набор состояний и допустимых переходов задачи в жизненном цикле. |
| **`PENDING`** | Статус "результат неизвестен" (не обязательно "задача не запускалась"). |
| **`RETRY`** | Задача специально уходит на повтор с задержкой/политикой. |
| **`REVOKED`** | Задача отменена до/во время исполнения по административной команде/политике. |
| **`REJECTED`** | Worker отклонил сообщение (обычно на уровне потребления/валидации). |
| **`Retry` exception** | Специальное исключение Celery, которое инициирует retry-переход. |
| **`Ignore` exception** | Инструкция Celery "не сохранять обычный результат/состояние как при стандартном success". |
| **`Reject` exception** | Явное отклонение текущего сообщения с управлением requeue-поведением. |
| **`SoftTimeLimitExceeded`** | Мягкий таймаут: задаче дают шанс корректно завершиться/почиститься. |
| **`TimeLimitExceeded`** | Жесткий таймаут: выполнение прерывается принудительно. |
| **`WorkerLostError`** | Процесс исполнения пропал/умер раньше, чем Celery получил финал задачи. |
| **Message payload** | Данные сообщения: метаинформация (`headers`) и аргументы задачи (`body`). |
| **`task_protocol`** | Версия структуры сообщения Celery; влияет на совместимость producer/consumer. |
| **Idempotency key** | Внешний ключ, который помогает не выполнить один и тот же бизнес-эффект дважды. |

#### Проверь себя: шпаргалка терминов

1. Почему `Retry` (исключение) и `RETRY` (состояние) нельзя путать в диагностике?
2. Чем `task_protocol` принципиально отличается от `content-type`?

<details><summary>Ответ</summary>

1) `Retry` — это управляющее действие в коде задачи, а `RETRY` — наблюдаемый статус в жизненном цикле выполнения.  
2) `task_protocol` задает структуру task-сообщения между producer/worker, а `content-type` описывает формат сериализации payload.

</details>
