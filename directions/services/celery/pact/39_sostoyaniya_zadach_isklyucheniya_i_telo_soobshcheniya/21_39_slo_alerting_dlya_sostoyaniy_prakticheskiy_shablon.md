## SLO/алертинг для состояний: практический шаблон

Базовый принцип: алертить не "сырой статус сам по себе", а связку **статус + тренд + контекст**.

| Сигнал | Почему опасно алертить "в лоб" | Лучше алертить так |
|---|---|---|
| `PENDING` count | без backend это почти всегда шум | `PENDING` + рост очереди + падение consume rate |
| `RETRY` count | retry может быть нормальной защитой | `RETRY` rate + исчерпание max retries + рост latency |
| `FAILURE` count | часть failure локальна и некритична | failure по критичным task names + error budget burn rate |
| `REJECTED` count | может быть штатный reject-policy | всплеск reject + deserialization/contract errors |
| `REVOKED` count | часть revoke ожидаема (операции) | неожиданный рост revoke вне maintenance окна |

Короткий operational вывод: у каждого статуса должен быть "партнерский" контекстный сигнал, иначе алертинг превращается в шум.

#### Проверь себя: SLO/алертинг

1. Почему алерт по одному статусу без контекста часто бесполезен?
2. Какой контекстный сигнал логично добавить к `REJECTED`, чтобы отличить политику от регрессии?

<details><summary>Ответ</summary>

1) Потому что статус может быть нормальным промежуточным состоянием; без тренда и соседних метрик сложно отличить штатное поведение от инцидента.  
2) Ошибки десериализации/контракта и изменения в `accept_content`/serializer после релиза.

</details>
