## Краткая шпаргалка по терминам

| Понятие | Суть простыми словами |
|---|---|
| **Broker (transport)** | “Доставка сообщений”: где задача ждёт и как worker её получает. |
| **Delivery semantics** | Реалистичные гарантии доставки при сбоях: что произойдёт при рестартах и таймаутах. |
| **Result backend** | “Табло по `task_id`”: где клиент читает статусы и результаты. |
| **Durable queue** | Очередь, которая переживает перезапуск broker’а (при правильной конфигурации). |
| **Persistent message** | Сообщение, которое broker пытается не потерять при рестарте/сбоях. |
| **Ack / redelivery** | Подтверждение “сообщение обработано” и повторная доставка при отсутствии ack. |
| **Prefetch (QoS)** | Сколько сообщений worker забирает заранее: влияет на latency и “запас” дубликатов при падениях. |
| **Priority** | Механизм “кому достаться очереди первым”, но с ограничениями в реальности. |
| **DLQ / dead-letter** | “Больничная” очередь для сообщений, которые не смогли обработаться и должны отойти в сторону (или ждать retry-план). |
| **Quorum queues vs classic** | Два подхода к репликации/устойчивости в RabbitMQ. Quorum обычно про более предсказуемую надежность ценой некоторой стоимости. |
| **Publisher confirms** | Подтверждение от broker’а: “сообщение действительно принято к хранению/обработке”. |
| **Queue TTL / message TTL** | Временная жизнь очереди/сообщения: после истечения работа может быть неинтересна и должна быть отброшена или переведена. |
| **Visibility timeout (SQS)** | Время, после которого сообщение снова становится доступным для доставки, если оно “в полёте” слишком долго. |
| **Cache-like backend** | Backend, который хранит результаты “как кэш”: TTL/очистка, высокая скорость, но меньше гарантий как у полноценной БД. |
