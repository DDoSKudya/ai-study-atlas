## Краткая шпаргалка по терминам

| Понятие | Суть простыми словами |
|---|---|
| **Producer** | Компонент, который формирует задачу и публикует сообщение в broker. |
| **Broker** | “Почта и склад”: хранит и маршрутизирует сообщения до worker’а. |
| **Worker** | Процесс(ы), которые забирают сообщения и исполняют задачу. |
| **Result backend** | Подсистема, где хранятся статусы и результаты задач, чтобы приложение могло читать их по `task_id`. |
| **Message flow** | Полный путь задачи от публикации до записи состояния/результата. |
| **Headers / body / properties** | Техническая структура сообщения: метаданные, полезная нагрузка и transport-уровень параметры. |
| **Routing** | Принятие решения broker’ом: в какую очередь попадет сообщение. |
| **Reservation / reserve** | Шаг, когда worker забирает сообщение из очереди “в обработку”. |
| **Acknowledgment (ack)** | Подтверждение broker’у: сообщение успешно взято/обработано (с семантикой “когда именно считаем выполненным”). |
| **Visibility / visibility timeout** | “Окно видимости”: после `reserve` сообщение считается “в обработке” и не должно переотдаваться, пока worker не подтвердит (`ack`) или пока не истечет таймаут повторной выдачи (механика зависит от transport). |
| **Ack point** | Точка жизненного цикла сообщения, в которой Celery отправляет `ack` (раньше/позже относительно execute). Она влияет на то, сколько и какие дубликаты возможны при сбоях. |
| **Kombu** | Messaging layer Celery: абстракция транспортов и протоколов, через которую Celery общается с брокером. |
| **Events** | Поток событий от worker/task, который потребляют наблюдатели (например, Flower). |
| **Remote control** | Канал команд управления/inspect (revoke, rate limit, контроль worker’ов). |
| **Failure domain** | “Зона отказа”: отдельный участок системы, падение которого влияет на разные части поведения. |
