## Частые сценарии

### Сценарий 1. "Нужен единый retry policy, задачи разрознены"

**Решение:**
- выносим retry helper в BaseTask;
- для экстренных случаев используем `task_annotations`;
- фиксируем метрики retry-volume и retry-success-rate.

### Сценарий 2. "После релиза часть задач падает на decode payload"

**Решение:**
- проверяем `schema_version`;
- включаем backward-compatible decoding;
- временно поддерживаем обе версии до полного rollout.

### Сценарий 3. "Нужно быстро подключить телеметрию исполнения задач"

**Решение:**
- используем `task_prerun/task_postrun/task_failure`;
- добавляем кореляционные поля (`task_id`, `trace_id`, `tenant`);
- убеждаемся, что handlers не тяжелые и не блокирующие.

### Сценарий 4. "Команда предлагает глубокий хак internals Celery"

**Решение:**
- запускаем decision-matrix из раздела `23.5`;
- сравниваем с альтернативой через дизайн задач;
- если нет owner/tests/runbook — отклоняем кастомизацию.

### Сценарий 5. "После внедрения signals throughput просел"

**Решение:**
- измеряем длительность каждого handler-а;
- выносим тяжелые операции из signal в отдельные неблокирующие механизмы;
- вводим таймауты и fail-open поведение для не критичных хуков.

### Сценарий 6. "При апгрейде Celery сломались кастомные расширения"

**Решение:**
- проверяем, использовались ли private internals;
- добавляем контрактные тесты на extension points;
- переписываем хрупкие места на публичные API и фиксируем upgrade checklist.

### Сценарий 7. "Нужно внедрить кастомизацию без риска для SLA"

**Решение (пошаговый roadmap):**
1. Формулируем измеримую цель (например, снизить p95 latency retry-задач на 20%).
2. Выбираем минимальный extension point (`BaseTask` или `annotations`, а не сразу всё).
3. Готовим canary-rollout на ограниченной группе worker-ов.
4. До включения фиксируем baseline-метрики (latency, failure-rate, retry-rate).
5. Включаем extension с feature flag и наблюдаем 24-48 часов.
6. При деградации — откат по заранее описанному rollback-плану.
7. При успехе — расширяем rollout, обновляем runbook и policy-инвентарь.

### Диагностический атлас по части 23

| Где проблема | Симптом | Первый шаг диагностики | Наиболее частая первопричина | Первое безопасное действие |
|---|---|---|---|---|
| **23.1 BaseTask** | Неожиданные ретраи/логи | Проверить хелперы base class и их тесты | "Умный" base class с побочными эффектами | Отключить/обойти проблемный helper |
| **23.2 Payload/schema** | Decode errors после релиза | Сверить `schema_version` и `accept_content` | Несовместимая схема между producer/worker | Временно вернуть producer на старую схему |
| **23.3 Signals/bootsteps** | Просадка throughput | Посмотреть длительность handlers/bootsteps | Блокирующий код в signal | Fail-open + перенос тяжелой логики |
| **23.4 Annotations** | Поведение не совпадает с кодом | Сравнить effective policy и wildcard overrides | Скрытый config drift | Убрать конфликтующий override |
| **23.5 Decision layer** | Инциденты после апгрейда | Проверить зависимость от private internals | Over-customization без owner/tests | Заморозить фичу и перейти на публичные API |

#### Как пользоваться атласом

1. Найди строку, которая похожа на текущий симптом.
2. Сделай **первый шаг диагностики** до любых глубоких изменений.
3. Выполни **первое безопасное действие**, чтобы ограничить blast radius.
4. Только после стабилизации переходи к root-cause и рефакторингу.

#### Вопросы по диагностическому атласу

1. Почему в атласе есть колонка "первое безопасное действие", а не сразу "финальное исправление"?

<details><summary>Ответ</summary>

Потому что в проде сначала нужно ограничить blast radius и стабилизировать систему. Полный рефакторинг без стабилизации может усилить инцидент.

</details>

2. Как атлас помогает избегать ложной диагностики между `23.1` и `23.4`?

<details><summary>Ответ</summary>

Он разделяет симптомы по слоям: если поведение не совпадает с кодом, это часто `annotations/policy` (`23.4`), а не ошибка BaseTask (`23.1`). Это экономит время расследования.

</details>

3. Когда атлас нужно расширять локальными кейсами команды?

<details><summary>Ответ</summary>

Когда появляются повторяющиеся внутренние инциденты с уникальной инфраструктурной спецификой: собственные policy-паттерны, нестандартные bootsteps, фирменные сериализаторы или особые release-процессы.

</details>

### Практикум: 3 мини-инцидента с эталонным разбором

#### Инцидент A. После релиза выросли `DecodeError` в одной очереди

**Дано:**
- producer обновили на новую схему payload;
- часть worker-ов осталась на старой версии;
- в логах видно `Unsupported schema_version=3`.

**Как рассуждать:**
1. Это зона `23.2` (payload/schema), а не проблема retry в задачах.
2. Проверяем compatibility-ветки decoder-а и `accept_content`.
3. Оцениваем, есть ли в очереди хвост старых/новых сообщений.

**Эталонное решение:**
- временно вернуть producer к совместимой версии payload;
- включить backward-compatible parser для `v2/v3`;
- после выравнивания версий удалить legacy-ветку по плану cleanup.

#### Инцидент B. Throughput упал после "безобидной" телеметрии

**Дано:**
- добавили метрики через `task_prerun`;
- p95 latency вырос, очереди начали расти;
- CPU worker-ов высокий, внешняя метрика-система периодически отвечает медленно.

**Как рассуждать:**
1. Это зона `23.3` (signals), вероятно блокирующий handler.
2. Проверяем длительность signal-хендлеров и таймауты.
3. Смотрим, есть ли fail-open при недоступности telemetry backend.

**Эталонное решение:**
- отключить проблемный handler флагом;
- вынести отправку метрик в неблокирующий буфер/батч;
- добавить таймаут и fallback (не ломать pipeline выполнения задач).

#### Инцидент C. Задача в коде имеет `max_retries=3`, а фактически ретраится 8 раз

**Дано:**
- код задачи не меняли;
- инцидент начался после аварийного "временного" конфига;
- разница только в проде.

**Как рассуждать:**
1. Это зона `23.4` (annotations/policy drift).
2. Сравниваем effective runtime policy в prod и stage.
3. Проверяем wildcard-override и срок действия временных правил.

**Эталонное решение:**
- найти и убрать конфликтующий override;
- зафиксировать TTL для аварийных annotations;
- добавить регулярный policy-diff между окружениями.

#### Проверь себя по практикуму

1. Почему в инциденте A первым шагом не стоит менять BaseTask?

<details><summary>Ответ</summary>

Потому что симптом указывает на несовместимость схемы payload при декодировании, а это уровень контракта сообщений (`23.2`), а не поведения базового класса задач.

</details>

2. Какой общий принцип объединяет решения для инцидентов B и C?

<details><summary>Ответ</summary>

Сначала ограничить blast radius безопасным действием (отключение хука/удаление конфликтного override), затем делать root-cause и устойчивый рефакторинг.

</details>

3. Что должно быть общим требованием к любому "временному" production-override?

<details><summary>Ответ</summary>

Явный срок действия (TTL), владелец, критерии снятия и запись в инвентаре policy-изменений.

</details>
