### 1.1. God Object

#### Цель раздела

Научиться распознавать ситуацию, когда «один класс/модуль делает всё» (God Object), понимать, **почему это почти всегда ведёт к боли**, и как **пошагово и без страха** разбивать такой объект на более мелкие, понятные части.

Если объяснить совсем по‑простому:  
God Object — это как один человек в компании, на котором «всё держится». Он:

- и договора подписывает,
- и зарплату считает,
- и серверы настраивает,
- и баги чинит,
- и работу менеджеров проверяет.

Пока компания маленькая — это «удобный герой». Но дальше:

- он не успевает,
- никто кроме него не понимает, как всё устроено,
- любая его ошибка бьёт по всей компании.

В коде ровно то же самое.

#### Термины

| Термин | Определение простыми словами |
|--------|------------------------------|
| **God Object** | Класс или модуль, который «знает всё и делает всё»: содержит много полей и методов, управляет разными аспектами системы. |
| **Single Responsibility Principle (SRP)** | Принцип единственной ответственности: у сущности должна быть **одна причина для изменения**. «Если завтра поменялось X, мне нужно править только Y». |
| **Cohesion (связанность внутри)** | Насколько сильно части модуля/класса связаны по смыслу. Высокая связанность = в файле/классе всё про одну тему. |
| **Coupling (сцепление между частями)** | Насколько сильно модули/классы зависят друг от друга. Низкое сцепление = модули можно менять по отдельности. |

#### Как выглядит God Object (чек‑лист)

Признаки, по которым вы можете **на глаз** понять, что перед вами God Object:

- файл/класс на **сотни строк**, в котором сложно быстро найти нужный метод;
- в одном классе:
  - и работа с БД,
  - и HTTP‑запросы,
  - и бизнес‑правила,
  - и логирование,
  - и кеширование,
  - и отправка писем/нотификаций;
- на ревью вы часто видите фразу «опять правим этот класс»;
- почти в каждом модуле есть импорт этого класса: «без него ничего не работает»;
- при попытке поправить маленькую фичу в одном месте, **отваливается что‑то в другом**.

Упрощённый пример:

```python
class Application:
    def __init__(self, db_url):
        self.db_url = db_url
        self.cache = {}

    # Работа с пользователями
    def create_user(self, name, email):
        ...

    def delete_user(self, user_id):
        ...

    # Работа с товарами
    def add_product(self, name, price):
        ...

    def get_product(self, product_id):
        ...

    # Отчёты
    def generate_sales_report(self, start_date, end_date):
        ...

    # Логирование
    def log_event(self, message):
        ...
```

Сначала кажется удобно: «всё в одном месте». Потом:

- увеличивается количество методов;
- появляются сильные зависимости (БД, кэш, логирование, отчёты);
- становится страшно трогать этот класс;
- новый разработчик **теряется**, глядя в этот файл: «куда тут вообще смотреть?».

Ментальное правило:  
если, открыв файл, вы **не можете одной фразой описать, «про что этот класс»**, это кандидат на God Object.

#### Как God Object ломает проект (по шагам)

1. **Сначала удобно.**  
   Один файл, один класс, всё рядом — скорость разработки высокая.
2. **Появляются первые «костыли».**  
   «Ну ладно, добавлю ещё один метод сюда, чтобы не создавать новый класс».  
   Так класс разрастается.
3. **Рост скрытой связи.**  
   Методы начинают использовать одни и те же поля, завязываться на общий кэш, на приватные вспомогательные функции.
4. **Непредсказуемые побочные эффекты.**  
   Меняете одно место — ломается другое, даже в другой подсистеме (например, отчёты).
5. **Страх изменений.**  
   Команда боится трогать этот класс: «если сюда полезем, придётся неделю всё чинить».
6. **Технический долг.**  
   Любая новая фича стоит дороже: её труднее внедрять, дольше тестировать, больнее сопровождать.

#### Как рефакторить God Object (подробный план)

Главный страх: «если начну рефакторить, всё сломаю».  
Поэтому действуем **маленькими шагами**, а не «переписываем всё с нуля».

Общий план:

1. **Поставить защиту — тесты.**  
   Не обязательно «идеальные». Достаточно:
   - несколько интеграционных тестов на ключевые сценарии;
   - или хотя бы скрипт, который прогонит основные операции (регистрация, заказ, отчёт и т.п.).
   Цель: после каждого шага **проверить, что всё работает как раньше**.
2. **Нарисовать карту ответственности.**  
   Прямо на бумаге или в комментарии:
   - «эти методы — работа с пользователями»;
   - «эти — с товарами»;
   - «эти — с отчётами»;
   - «эти — утилиты (кэш, логирование)».
3. **Выделить самый самостоятельный кусок.**  
   Например, работу с пользователями:
   - `create_user`,
   - `delete_user`,
   - `get_user`,
   - может быть, `block_user`.
4. **Создать новый класс/сервис под этот кусок.**

```python
class UserService:
    def create_user(self, name, email):
        ...

    def delete_user(self, user_id):
        ...

    def block_user(self, user_id):
        ...
```

5. **Перенести код, но оставить старый интерфейс.**  
   То есть:
   - внутри `Application.create_user` вы просто делаете делегирование в новый сервис;
   - снаружи пока ничего не меняется.

```python
class Application:
    def __init__(self, db_url):
        self.db_url = db_url
        self.cache = {}
        self.user_service = UserService()

    def create_user(self, name, email):
        # временная «прокладка» — чтобы не менять сразу весь код
        return self.user_service.create_user(name, email)
```

6. **Постепенно переводить код на новый сервис.**  
   В остальных частях системы, где вызывали `app.create_user`, начинаете вызывать `user_service.create_user`.  
   Делаете это **по чуть‑чуть**, каждый раз прогоняя тесты.
7. **Когда старый метод стал ненужен — удалить прокладку.**

Тот же подход применяется к другим областям:

- `ProductService` для товаров;
- `ReportService` для отчётов;
- `Logger`/`AuditService` для логирования;
- и т.д.

В итоге у вас появляется небольшая «композиция» сервисов:

```python
class ProductService:
    def add_product(self, name, price):
        ...

    def get_product(self, product_id):
        ...


class ReportService:
    def generate_sales_report(self, start_date, end_date):
        ...


class App:
    def __init__(self, user_service, product_service, report_service):
        self.users = user_service
        self.products = product_service
        self.reports = report_service
```

Теперь:

- ответственность разбита;
- проще тестировать отдельные части;
- проще читать.

#### Типичные вопросы («я тупой, объясни ещё раз»)

- **«Почему нельзя всё держать в одном классе, если мне так легче найти?»**  
  Пока система маленькая — можно. Проблема в том, что **система вырастет**, а структура останется «детской».  
  Как если бы вы всё взрослую жизнь жили в комнате, где все вещи в одной куче. Пока вещей мало — терпимо. Через 5 лет — хаос.

- **«А если я не знаю, как правильно поделить на части?»**  
  Ориентир простой: **задайте одному классу один «глагол»**.  
  Например:
  - `UserService` — отвечает за управление пользователями;
  - `BillingService` — за оплату;
  - `NotificationService` — за отправку уведомлений.
  Если в классе появляются чужие глаголы — это сигнал, что он разрастается.

- **«Это же больше файлов и классов — не усложняю ли я?»**  
  Больше файлов — не проблема, если каждый из них **маленький и про одно**.  
  Проблема — когда мало файлов, но каждый гигантский и «про всё».

#### Типичные ошибки и граничные случаи

- **«Но у нас маленький проект, это не проблема».**  
  Так думают почти в каждом проекте. God Object редко мешает в самом начале.  
  Беда в том, что когда он начнёт мешать, его уже будет сложно распилить.
- **«Давайте сделаем один `utils.py` на все случаи».**  
  Это тот же God Object, только на уровне модуля.
- **«Мы сделаем `CoreManager`, а внутри у него будет всё‑всё‑всё».**  
  Название не спасает. Если по сути это God Object — от этого не легче.

#### Запомните

- God Object = удобно в начале, но болезненно в развитии решение.
- Чем раньше начнёте **делить по ответственности**, тем проще будет поддерживать проект.
- Лечение — **композиция** и **маленькие специализованные классы**, а не «ещё один метод в большой класс».

### 1.2. Spaghetti Code

#### Цель раздела

Понять, как выглядит «лапша» из условных операторов, циклов и вызовов, почему такой код трудно изменять, и как превращать его в **последовательность понятных шагов**, где логика читается «как рассказ».

#### Термины

| Термин | Определение |
|--------|-------------|
| **Spaghetti Code** | Код, в котором поток исполнения запутан: много вложенных `if/else`, переходов, зависимостей между частями. Читателю тяжело понять, «что за чем идёт». |
| **Control flow** | Поток управления: порядок, в котором выполняется код программы. |
| **Refactoring** | Изменение внутренней структуры кода **без изменения внешнего поведения**. |

#### Как выглядит Spaghetti Code

Представьте макароны в тарелке: они переплетены, вы не видите, где начало, где конец.  
Spaghetti Code — это такой же клубок, только из `if`, `for` и вызовов функций.

Пример:

```python
def process_order(order):
    if order.status == "new":
        if order.user.is_blocked:
            send_email(order.user.email, "Ваш аккаунт заблокирован")
            order.status = "blocked"
        else:
            if order.total > 1000:
                if order.user.is_vip:
                    apply_discount(order, 0.1)
                else:
                    apply_discount(order, 0.05)
            else:
                if order.payment_method == "card":
                    if not check_3ds(order):
                        cancel_order(order)
                        return
                elif order.payment_method == "cash":
                    if not check_cash_limit(order):
                        cancel_order(order)
                        return
            if not reserve_items(order):
                cancel_order(order)
                return
            order.status = "processed"
            send_email(order.user.email, "Заказ принят")
    else:
        # ещё несколько уровней условий ...
        ...
```

Что чувствует читатель такого кода:

- «Где тут вообще основной путь?»
- «Что происходит, если пользователь заблокирован? А если нет?»
- «Если что‑то пошло не так при оплате, где это обрабатывается?»

Чтобы ответить на простой вопрос, нужно глазами «прыгать» по коду и держать в голове множество флагов.

#### Почему так писать опасно

- **Сложно добавлять новые условия.**  
  Любой новый `if` добавляет ещё один уровень вложенности.
- **Сложно тестировать.**  
  Нужно продумывать огромное количество комбинаций: заблокирован/нет, VIP/обычный, метод оплаты, результат проверки, наличие товара и т.д.
- **Ошибки прячутся.**  
  Легко пропустить редкую ветку (например, «VIP, но заблокирован, и платит наличными»).

#### Как упростить: идея «лестницы» и «шага за шагом»

Цель — чтобы основная функция читалась как **последовательность шагов**:

1. Проверить статус заказа.
2. Проверить, не заблокирован ли пользователь.
3. Применить скидки.
4. Проверить оплату.
5. Зарезервировать товары.
6. Завершить заказ.

Код до рефакторинга — это всё сразу в одной куче.  
Код после рефакторинга — **каждый шаг выделен в отдельную функцию**.

Переписываем:

```python
def is_user_blocked(order):
    return order.user.is_blocked


def apply_order_discounts(order):
    if order.total > 1000:
        discount = 0.1 if order.user.is_vip else 0.05
        apply_discount(order, discount)


def validate_payment(order):
    if order.payment_method == "card":
        return check_3ds(order)
    if order.payment_method == "cash":
        return check_cash_limit(order)
    return True


def process_order(order):
    if order.status != "new":
        return  # или другая логика

    if is_user_blocked(order):
        send_email(order.user.email, "Ваш аккаунт заблокирован")
        order.status = "blocked"
        return

    apply_order_discounts(order)

    if not validate_payment(order):
        cancel_order(order)
        return

    if not reserve_items(order):
        cancel_order(order)
        return

    order.status = "processed"
    send_email(order.user.email, "Заказ принят")
```

Теперь «скелет» функции:

1. Проверить статус.
2. Проверить блокировку пользователя.
3. Применить скидки.
4. Проверить оплату.
5. Зарезервировать товары.
6. Установить статус и отправить письмо.

Такой код читается как **история**:

> «Если заказ новый, а пользователь не заблокирован, применяем скидки, проверяем оплату, резервируем товар и подтверждаем заказ».

#### Guard‑clauses (ранний выход) против глубокой вложенности

Один из главных инструментов борьбы с «лапшой» — **ранний выход** из функции, когда что‑то пошло не так.

Сравните:

```python
def f(x):
    if cond1:
        # много кода
        if cond2:
            # ещё много кода
            if cond3:
                # и так далее
                ...
```

и:

```python
def f(x):
    if not cond1:
        return

    # много кода

    if not cond2:
        return

    # ещё код

    if not cond3:
        return

    # основной happy-path
```

Во втором варианте **нет глубокой вложенности**: мы обрабатываем негативные случаи сразу и выходим, а основной сценарий идёт по прямой линии.

#### Алгоритм разматывания «лапши»

1. Найдите функцию, которая вызывает у вас **ощущение хаоса**.
2. Попробуйте **словами** (без кода) описать, что она делает, в 3–7 шагах.
3. Для каждого шага создайте **отдельную функцию/метод** с понятным названием.
4. Замените куски внутри основной функции на вызовы этих функций.
5. Убедитесь (тестами или ручной проверкой), что поведение не изменилось.

Так вы постепенно превращаете «макароны» в **лестницу понятных шагов**.

#### Запомните

- Если вы видите **много вложенных `if` и `else`**, почти всегда нужен рефакторинг.
- Основная функция должна читаться как **последовательность шагов**, а не как головоломка.
- Ранние `return` (guard‑clauses) помогают **избавиться от глубокой вложенности** и сделать код линейнее.

### 1.3. Copy‑Paste Programming

#### Цель раздела

Научиться замечать дублирование кода и понимать:

- **почему** копипаста опасна в долгую;
- **когда** с ней нужно бороться сразу;
- **когда** копипаста допустима (иногда это нормально);
- как **пошагово превращать** несколько почти одинаковых кусков кода в один аккуратный, понятный и гибкий.

#### Интуитивное объяснение

Представь, что ты написал сочинение, а потом учитель попросил то же самое сочинение, но:

- один раз — от лица другого персонажа;
- второй раз — с чуть изменённой концовкой.

Ты можешь:

- **вариант А**: три раза переписать всё сочинение полностью;
- **вариант Б**: сделать «шаблон» и менять только отличающиеся части.

Вариант А — это Copy‑Paste Programming в коде:

- ошибки/опечатки повторяются в трёх местах;
- если захочешь исправить текст — нужно помнить, что его **три копии**.

С кодом так же:

- ты копируешь функцию,
- чуть‑чуть меняешь,
- спустя время не помнишь, **где именно какая версия** и чем они отличаются.

#### Суть Copy‑Paste Programming

Copy‑Paste Programming — это когда мы:

- копируем кусок кода;
- немного его меняем;
- продолжаем жить с несколькими версиями одного и того же алгоритма.

Проблема: **ошибку нужно исправлять в нескольких местах**, и легко забыть одно из них.

Простой пример:

```python
def calculate_discount_for_regular(price):
    if price > 1000:
        return price * 0.9
    return price


def calculate_discount_for_vip(price):
    if price > 1000:
        return price * 0.85
    return price
```

Логика почти одна и та же, различается только коэффициент.

Воображаем баг:  
заказчик попросил: «делать скидку не с 1000, а с 1500».

Если ты забудешь поменять порог **в обеих функциях**, получится:

- VIP работает по новым правилам,
- обычные клиенты — по старым,
- дальше — путаница, жалобы, отладка.

#### Как искать дублирование (чек‑лист)

Признаки, что у тебя Copy‑Paste:

- ты видишь два очень похожих кусочка кода рядом/в соседних файлах;
- при добавлении фичи ты **интуитивно берёшь код выше, копируешь и вносишь мелкие правки**;
- на ревью часто звучит фраза: «а разве у нас уже не было такого кода выше/в другом модуле?»;
- при исправлении бага тебе приходится открывать **несколько файлов** и в каждом править одно и то же условие/формулу.

#### Как обобщать: пошаговый рецепт

Идея: **выделить, что общее, и что различается**.

1. Положи два куска кода рядом (можно прямо в редакторе или в голове).
2. Отметь:
   - строки, которые полностью совпадают;
   - места, где код отличается (например, коэффициент скидки).
3. Общее преврати в **функцию/метод**.
4. Отличия преврати в **параметры** или внешние зависимости.

Вернёмся к примеру:

```python
def calculate_discount_for_regular(price):
    if price > 1000:
        return price * 0.9
    return price


def calculate_discount_for_vip(price):
    if price > 1000:
        return price * 0.85
    return price
```

Общее:

- проверка `if price > 1000`;
- формула `price * (1 - rate)`;
- `return price`, если чуть ниже порога.

Отличается только **коэффициент**.

Общий вариант:

```python
def calculate_discount(price, rate):
    if price > 1000:
        return price * (1 - rate)
    return price
```

Используем:

```python
regular_price = calculate_discount(price, rate=0.1)
vip_price = calculate_discount(price, rate=0.15)
```

Теперь, если изменится порог или формула, мы меняем **одно место**, а не две почти идентичные функции.

#### Чуть сложнее пример

До:

```python
def send_welcome_email(user):
    subject = "Добро пожаловать!"
    body = f"Привет, {user.name}! Рады вас видеть."
    send_email(user.email, subject, body)


def send_password_reset_email(user, token):
    subject = "Сброс пароля"
    body = f"Привет, {user.name}! Для сброса пароля перейдите по ссылке: {token}"
    send_email(user.email, subject, body)
```

Общее:

- выбор `user.email`;
- вызов `send_email`;
- формат «Привет, {имя}! ...».

Вариант с обобщением:

```python
def send_user_email(user, subject, body_template, **kwargs):
    body = body_template.format(name=user.name, **kwargs)
    send_email(user.email, subject, body)


def send_welcome_email(user):
    send_user_email(
        user,
        subject="Добро пожаловать!",
        body_template="Привет, {name}! Рады вас видеть.",
    )


def send_password_reset_email(user, token):
    send_user_email(
        user,
        subject="Сброс пароля",
        body_template="Привет, {name}! Для сброса пароля перейдите по ссылке: {token}",
        token=token,
    )
```

Теперь:

- вся общая логика в одном месте (`send_user_email`);
- конкретные сценарии читаются как короткие вызовы.

#### Где осторожность: когда НЕ надо обобщать

Слишком агрессивное обобщение — тоже анти‑паттерн.  
Иногда разработчик видит два похожих куска и пытается сделать «гениально универсальную» функцию, которую потом никто не понимает.

Правила здравого смысла:

- **Обобщать только то, что реально повторяется** (не одну‑две строки, а куски логики).
- Если после обобщения код стал **сложнее читать**, чем две отдельных функции — возможно, это преждевременная оптимизация.
- Не бойся **немного дублирования** там, где сценарии всё‑таки различаются по смыслу.

Пример, где обобщение может быть лишним:

- расчёт скидки для совершенно разных типов клиентов с разными правилами;
- сложные формы в UI, которые похожи только визуально, но сильно отличаются по логике.

#### Запомните

- Дублирование = **сигнал** для рефакторинга, а не приговор.
- Ищите **одинаковые ветки кода** и задавайте себе вопрос:  
  «что здесь по сути одинаково, а что отличается?»
- Обобщайте только тогда, когда это действительно упрощает жизнь:
  - меньше мест для правок,
  - понятнее структура,
  - легче тестировать.

### 1.4. Magic Numbers и Magic Strings

#### Цель раздела

Понять, почему «магические» числа/строки в коде делают его хрупким и непонятным, и как заменить их на **говорящие константы**, чтобы:

- код читался как «рассказ», а не как ребус;
- изменения требований не превращались в охоту за «скрытыми числами по всему проекту».

#### Термины

| Термин | Определение |
|--------|-------------|
| **Magic number** | Число в коде, значение которого **не очевидно без комментариев** («почему именно 18, а не 21?»). |
| **Named constant** | Константа с осмысленным именем (обычно UPPER_CASE), которое объясняет, «что это за число/строка». |

#### История из жизни кода

Кусок кода:

```python
if user.age >= 18:
    allow_access()
```

Когда ты это пишешь, тебе **кажется очевидным**, что:

- 18 — это возраст совершеннолетия;
- «мы делаем сервис для страны, где 18 — порог».

Через год в проект приходит новый разработчик. Он видит:

```python
if user.age >= 18:
    allow_access()
```

и задаётся вопросами:

- «Почему 18? А если нам нужно будет менять это на 21 для другой страны?»
- «Есть ли ещё места в коде, где используется 18?»

Если таких «магических» чисел много, любой бизнес‑изменение превращается в:

- поиск по коду `18`, `21`, `0.1`, `"ok"`, `"error"` и т.п.;
- риск пропустить где‑то один из случаев.

#### Как дать числу имя

Решение простое: **вынести число/строку в константу с понятным именем**.

```python
ADULT_AGE = 18

if user.age >= ADULT_AGE:
    allow_access()
```

Теперь:

- понятно, что 18 — это «возраст совершеннолетия»;
- если бизнес скажет «18 → 21», ты меняешь **одно место**;
- поиск по коду становится осмысленным: ищешь `ADULT_AGE`, а не все `18`.

#### Ещё пример: HTTP‑статусы

```python
if status_code in (200, 201, 204):
    ...
```

Что такое 200, 201, 204?  
Если ты не помнишь наизусть HTTP‑статусы, нужно лезть в документацию.

Более говорящий вариант:

```python
from http import HTTPStatus

if status_code in (
    HTTPStatus.OK,
    HTTPStatus.CREATED,
    HTTPStatus.NO_CONTENT,
):
    ...
```

Теперь даже без знаний HTTP видно:

- `OK` — всё хорошо;
- `CREATED` — создан ресурс;
- `NO_CONTENT` — нет тела в ответе.

#### Когда «магическое» число допустимо

Не каждое число в коде нужно выносить в константу.

Обычно **нормально оставить как есть**:

- `for i in range(3):` — очевидный короткий цикл;
- `index = -1` — последний элемент;
- `percent / 100` — если контекст ясен.

Хорошее правило:

- если число/строка **влияет на бизнес‑логику** (порог скидки, возраст, лимит запросов, время тайм‑аута) — почти всегда стоит вынести в константу;
- если это чисто техническая деталь, понятная из контекста — можно оставить.

#### Магические строки

Проблема не только в числах:

```python
if status == "ok":
    ...
elif status == "error":
    ...
```

Опасности:

- опечатка: `"erorr"` vs `"error"`;
- строка `"ok"` может использоваться в другом контексте с другим смыслом;
- нет единого места, где видно все возможные значения.

Лучше:

```python
STATUS_OK = "ok"
STATUS_ERROR = "error"

if status == STATUS_OK:
    ...
elif status == STATUS_ERROR:
    ...
```

Ещё лучше — `Enum`, если значений много и они логически связаны.

#### Запомните

- «Магическое» число/строка — это то, про что хочется спросить:  
  **«А что это вообще такое?»**
- Если значение **может поменяться по бизнес‑причине** — почти всегда выносим в константу.
- Константы делают код **самодокументируемым**: имя объясняет суть без комментариев.

### 1.5. Анти‑паттерны обработки ошибок

#### Цель раздела

Понять, почему опасно «ловить всё подряд» и игнорировать исключения, и как строить **прозрачную** и **предсказуемую** обработку ошибок:

- чтобы ошибки **не прятались**;
- чтобы при отладке было понятно, **что и где сломалось**;
- чтобы программа вела себя ожидаемо при сбоях.

#### Термины

| Термин | Определение |
|--------|-------------|
| **Bare `except`** | Конструкция `except:` без указания типа исключения. Ловит вообще всё. |
| **Exception swallowing** | «Проглатывание» ошибки — когда исключение перехватывают и никак не обрабатывают (нет логов, нет реакции). |

#### Плохой пример №1: `except: pass`

```python
try:
    do_something()
except:
    pass
```

Что происходит:

- внутри `do_something()` может случиться **что угодно**:
  - ошибка в твоём коде (`ValueError`, `TypeError`, `IndexError`),
  - системная ошибка,
  - `KeyboardInterrupt` (пользователь нажал Ctrl+C),
  - `SystemExit` (программа хочет завершиться);
- все эти исключения **ловятся**;
- дальше с ними **ничего не происходит**: ни логов, ни сообщений, ни остановки программы.

Снаружи это выглядит как:

- «программа ничего не делает и молчит»;
- «у нас ничего не падает, но и не работает».

Это худший вариант: **ошибка есть, но её не видно**.

#### Почему bare `except` особенно опасен

Конструкция `except:` (без указания типа) ловит **все исключения, включая системные**:

- `KeyboardInterrupt` — пользователь пытается остановить программу вручную, а ты это игнорируешь;
- `SystemExit` — код хочет корректно завершить работу (например, через `sys.exit()`), а ты это блокируешь;
- любые другие исключения, включая те, которые ты вообще не ожидал.

Обычно bare `except` **никогда не нужен**.  
Почти всегда правильнее написать:

```python
try:
    do_something()
except SomeSpecificError:
    handle_error()
```

#### Минимум — логировать

Если даже ты не знаешь, что делать с ошибкой прямо сейчас, **минимум** — записать её в лог.

```python
import logging

logger = logging.getLogger(__name__)

try:
    do_something()
except Exception:
    logger.exception("Ошибка при выполнении do_something")
    raise  # пробросить дальше
```

Здесь:

- `logger.exception` напечатает стек вызовов (traceback) — ты увидишь, где именно упало;
- `raise` после логирования **снова выбрасывает** то же исключение — программа не делает вид, что «ничего не случилось».

Даже `except Exception` — уже довольно широкий перехват.  
Лучше, когда можно, ловить конкретные типы:

```python
try:
    data = json.loads(text)
except json.JSONDecodeError as exc:
    logger.warning("Некорректный JSON: %s", exc)
    return None
```

#### Когда игнорировать исключение допустимо

Иногда **легально** сказать: «если случилась вот такая ошибка — просто тихо пропусти».

Пример: удалить временный файл, который мог уже исчезнуть:

```python
from contextlib import suppress
import os

with suppress(FileNotFoundError):
    os.remove("temp.txt")
```

Здесь мы явно говорим:

- «если файл уже удалён — это не ошибка, можно молча продолжить».

Это **не** анти‑паттерн, потому что:

- мы ловим **конкретный** тип (`FileNotFoundError`);
- поведение осознанно и ожидаемо.

#### Как правильно думать про обработку ошибок

Простой алгоритм:

1. Спросить себя: **«какие ошибки тут могут произойти?»**
2. Решить для каждой:
   - надо ли что‑то делать (повторить операцию, показать сообщение, вернуть значение по умолчанию);
   - или нужно просто залогировать и упасть.
3. Ловить **конкретные** исключения под осознанные сценарии.

Обработка ошибок — это часть дизайна, а не «мусорный бак для всего, что упало».

#### Запомните

- Никогда не пишите `except: pass`.
- Чем более широкий у вас `except`, тем больше риск **спрятать настоящую проблему**.
- Если вы ловите исключение, задайте себе вопрос:  
  **«Я точно понимаю, какие ситуации я сейчас обрабатываю и чего хочу добиться?»**
