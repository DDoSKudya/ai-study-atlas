### 2.1. Mutable default argument

Здесь повторяется и закрепляется материал из части о тонкостях, но в контексте анти‑паттернов: не просто «ловушка языка», а **плохой дизайн интерфейса функции**.

#### Цель раздела

Понять:

- **что именно** делает Python с аргументами по умолчанию;
- **почему** мутабельный default (`[]`, `{}`, `set()`) почти всегда ошибка;
- как увидеть этот анти‑паттерн в чужом коде и **механически переписать** на безопасный вариант.

#### Напоминание: как это выглядит

```python
def append_to_list(item, target=[]):  # ПЛОХО
    target.append(item)
    return target
```

Снаружи хочется верить, что при каждом вызове без `target` будет создаваться **новый список**.  
На самом деле:

- при определении функции создаётся **один** список `[]`;
- он живёт в `append_to_list.__defaults__`;
- все вызовы без второго аргумента используют **один и тот же объект**.

Проверка:

```python
print(append_to_list(1))  # [1]
print(append_to_list(2))  # [1, 2]
print(append_to_list(3))  # [1, 2, 3]
```

#### Что именно делает Python (по шагам)

1. Когда интерпретатор встречает `def append_to_list(item, target=[])`, он:
   - создаёт объект функции;
   - **один раз** создаёт объект списка `[]`;
   - кладёт этот список в `append_to_list.__defaults__`.
2. При вызове `append_to_list(1)`:
   - аргумент `target` не передан;
   - Python берёт значение по умолчанию из `__defaults__` — тот самый список;
   - `append` меняет этот список «на месте».
3. При следующем вызове `append_to_list(2)`:
   - снова используется тот же объект списка;
   - он уже содержит `[1]`;
   - после `append` становится `[1, 2]`.

То есть «коробка» со списком одна, и все, кто пришёл без своей коробки, пользуются **общей**.

Показать это можно так:

```python
def f(a=[]):
    a.append(1)
    print(a)

print(f.__defaults__)        # ([],)
f()                          # [1]
print(f.__defaults__)        # ([1],) — список в default тоже изменился
f()                          # [1, 1]
print(f.__defaults__)        # ([1, 1],)
```

#### Почему это именно анти‑паттерн (а не просто «особенность»)

Важно понять не только «как это работает», но и **почему это плохой дизайн**:

- Функция начинает **запоминать состояние между вызовами** там, где это не ожидается.
- Это **скрытое состояние**: по сигнатуре `def f(a=[])` трудно догадаться, что `a` шарится между вызовами.
- Тесты могут случайно проходить/падать в зависимости от **порядка вызовов**.

То есть функция ведёт себя не как обычная «чистая» операция, а как **маленький кэш** или «копилка», но об этом нигде честно не написано.

#### Правильный паттерн: `None` как маркер

Идиоматичный вариант:

```python
def append_to_list(item, target=None):
    if target is None:
        target = []  # создаём новый список при каждом вызове без target
    target.append(item)
    return target
```

Что тут меняется:

- по умолчанию `target` — это `None` (иммутабельный singleton);
- внутри функции мы **каждый раз** создаём новый список, если `target is None`;
- состояние между вызовами **не накапливается**, если явно не передали список.

Такой код соответствует здравому ожиданию:

```python
print(append_to_list(1))  # [1]
print(append_to_list(2))  # [2]
print(append_to_list(3))  # [3]
```

#### Типичные ошибки и вопросы

- **«Почему нельзя сделать `target = target or []`?»**  
  Потому что `0`, `''`, `False`, пустой список/dict/set — тоже «ложные» значения.  
  Вы можете перепутать ситуацию «аргумент не передали» и «аргумент явно передали, но он пустой/ложный».

- **«А если я хочу кэшировать результат между вызовами?»**  
  Тогда используйте **явный кэш**:

  ```python
  _cache = {}

  def get_data(key):
      if key in _cache:
          return _cache[key]
      data = load_data(key)
      _cache[key] = data
      return data
  ```

  или `functools.lru_cache`. Не прячьте кэш в аргументе по умолчанию.

- **«С dict и set такая же проблема?»**  
  Да. Любой мутабельный объект в default (`[]`, `{}`, `set()`, свой класс) ведёт себя так же.

#### Мини‑упражнения по 2.1 (лучше делать в REPL)

1. Возьми функцию:

```python
def add_tag(tag, tags=[]):
    tags.append(tag)
    return tags
```

   а) Вызови её несколько раз без второго аргумента и посмотри на результат.  
   б) Перепиши её на вариант с `None` по образцу из раздела и проверь, что каждый вызов даёт независимый список.

2. Представь функцию:

```python

    users.append(user)
    return users
```

   а) Объясни словами, почему это опасно в веб‑приложении (подумай про разных пользователей/запросы).  
   б) Перепиши её на безопасный вариант, где список передаётся явно или создаётся внутри при `None`.

3. Найди в своём или чужом коде **любой** `def f(arg=[])` / `def f(arg={})` / `def f(arg=set())`.  
   Просто пометь комментариями: «мутабельный default, надо переписать по шаблону из 2.1».

#### Шаблон для переписывания кода

Если видите:

```python
def f(arg=DEFAULT_MUTABLE):
    ...
```

шаблон исправления:

```python
def f(arg=None):
    if arg is None:
        arg = DEFAULT_FACTORY()  # [], {}, set() и т.п.
    ...
```

Где `DEFAULT_FACTORY` — это либо литерал (в простом случае), либо функция/класс.

#### Запомните

- Мутабельный аргумент по умолчанию почти всегда = **анти‑паттерн**.
- Default должен быть **иммутабельным** значением или `None`.
- Если нужно состояние между вызовами — делайте его **явным** (атрибут объекта, модульная переменная, кэш), а не спрятанным в default.

### 2.2. Truthiness и `len(x) > 0`

#### Цель раздела

Понять:

- что такое **truthiness** (правдивость объектов) в Python;
- почему `if len(x) > 0:` — шумный и «непитоничный» способ проверить непустоту;
- когда стоит использовать `if x:`, а когда **нужно именно количество**.

#### Что такое truthiness

В Python **любой объект** можно использовать там, где ожидается `True/False`:

```python
if value:
    ...
```

Правило по умолчанию:

- пустые контейнеры (`[]`, `{}`, `set()`, `''`, `()`, `range(0)`) → `False`;
- `0`, `0.0`, `0j` → `False`;
- `None` → `False`;
- почти всё остальное → `True`.

Это задаётся методами `__bool__` или `__len__` у объекта.

Проверка:

```python
bool([])        # False
bool([1, 2, 3]) # True
bool("")        # False
bool("abc")     # True
bool(0)         # False
bool(10)        # True
```

#### Анти‑паттерн: `len(x) > 0` вместо `if x`

Код:

```python
if len(items) > 0:
    process(items)
```

делает ровно то же, что:

```python
if items:
    process(items)
```

Но:

- длиннее;
- загромождает код;
- показывает, что автор **думает в терминах другого языка**, а не Python‑способом.

В большом коде такие мелочи накапливаются и делают его «шумным».

Идиоматично:

```python
if items:          # есть ли элементы?
    ...

if not items:      # пусто?
    ...
```

#### Когда `len()` нужно по делу

Есть ситуации, где **важно именно количество**, а не просто «пусто/не пусто»:

- нужно проверить, что количество элементов в пределах лимита;
- хочется вывести число в лог/интерфейс.

Примеры, где `len()` уместен:

```python
if len(users) > MAX_USERS_PER_PAGE:
    raise ValueError("Слишком много пользователей для одной страницы")

print(f"Найдено {len(results)} результатов")
```

То есть:

- `if items:` / `if not items:` — когда нужен просто факт пустоты/непустоты;
- `len(items)` — когда нужно само число.

#### Truthiness у своих классов

Вы можете управлять тем, как ведёт себя ваш объект в `if obj:`:

```python
class Bucket:
    def __init__(self):
        self.items = []

    def __bool__(self):
        # "Пустое ведро" считается ложным
        return bool(self.items)
```

Теперь:

```python
bucket = Bucket()
if not bucket:
    print("Ведро пустое")
```
Это иллюстрирует, что `if x:` — **язык соглашений**: объект сам может сказать, «считаюсь ли я пустым».

#### Мини‑упражнения по 2.2

1. Перепиши:

```python
if len(users) > 0:
    send_digest(users)
```

   в идиоматичный вариант с `if users:`. Объясни себе вслух, что логика не изменилась.

2. Найди в своём коде место с `len(x) > 0` или `len(x) == 0`.  
   Перепиши на `if x:` / `if not x:`. Убедись, что тесты (или ручная проверка) дают тот же результат.

3. Придумай пример, где `len()` нужен именно как число  
   (например, проверка «не больше N элементов») и запиши рядом комментарий, зачем тебе длина.

#### Запомните

- `if len(x) > 0:` — шумный и «непитоничный» способ сказать «если не пусто».
- Используйте `if x:` / `if not x:` для проверки пустоты, а `len(x)` — когда нужно **конкретное число**.

### 2.3. Сравнение с `None`

#### Цель раздела

Понять:

- чем отличается `x is None` от `x == None`;
- почему первое — правильно и надёжно, а второе — потенциальная ловушка;
- как связано это с понятием **identity** (тождество) и **equality** (равенство).

#### None как объект‑одиночка (singleton)

В Python `None` — это **один конкретный объект** в памяти:

```python
id(None) == id(None)  # True
```

Когда вы пишете:

```python
x = None
```

вы говорите: «переменная `x` **указывает на тот самый объект** `None`».

Поэтому естественный вопрос:

- «является ли `x` **тем самым** объектом `None`?» → `x is None`.

#### `is` против `==`

- `is` проверяет **тождество**: это **один и тот же объект** или нет?
- `==` проверяет **равенство**, которое может быть переопределено в `__eq__`.

Пример:

```python
a = [1, 2, 3]
b = [1, 2, 3]

print(a == b)  # True  — списки равны по содержимому
print(a is b)  # False — это два разных объекта
```

С `None` нам важно именно тождество:  
«эта переменная сейчас **пустая (None)** или там уже какое‑то значение?».

Правильно:

```python
if x is None:
    ...

if x is not None:
    ...
```

#### Почему `x == None` — ловушка

Потому что любой класс может переопределить `__eq__`:

```python
class Weird:
    def __eq__(self, other):
        # Логика, которая считает None "равным" чему угодно
        return True

obj = Weird()
print(obj == None)    # True
print(obj is None)    # False
```

Если вы в таком случае напишете `if obj == None:`,  
код **может вести себя странно**, не так как вы ожидаете.

#### Паттерн: `None` — через `is`, значения — через `==`

Хорошее правило:

- для `None` → **всегда** `is` / `is not`;
- для обычных значений (`0`, строки, объекты) → `==` / `!=`.

Примеры:

```python
if result is None:         # не было результата
    ...

if status == "ok":         # статус равен строке "ok"
    ...
```

#### Мини‑упражнения по 2.3

1. Найди в своём коде (или придумай) пример с `if x == None:` и перепиши его на `if x is None:`.  
   Проговори вслух: «я проверяю, что тут **нет значения** (None), а не сравниваю по смыслу».

2. Придумай класс, у которого `__eq__` считает `None` «равным» чему угодно  
   (по аналогии с примером `Weird` выше), и посмотри, как по‑разному работают `== None` и `is None`.

3. В любом месте, где ты сравниваешь со строками/числами (`status == "ok"`, `code == 200`),  
   убедись, что ты не используешь `is` — именно `==` здесь правильный инструмент.

### 2.4. Конкатенация строк в цикле

#### Цель раздела

Понять:

- как устроены строки в Python (иммутабельность);
- почему `s += piece` в большом цикле — скрытый расход памяти и времени;
- как правильно собирать большие строки.

#### Иммутабельность строк

Строка в Python — **неизменяемый объект**:

- при любом «изменении» создаётся **новая** строка;
- старая может быть удалена сборщиком мусора позже.

Когда вы пишете:

```python
s = ""
for piece in pieces:
    s += piece  # ПЛОХО
```

под капотом происходит примерно следующее:

1. Берётся текущая строка `s`.
2. Создаётся **новая** строка, в которой содержимое `s` + `piece`.
3. Старая строка помечается как неиспользуемая (если на неё нет других ссылок).

Если у вас 10_000 кусков, вы создадите тысячи временных строк, многие из которых живут всего одну итерацию цикла.

#### Правильный приём: собирать части и делать `join`

Идиоматичный способ:

```python
result = "".join(pieces)
```

Если элементы приходят постепенно:

```python
parts = []
for piece in pieces:
    parts.append(piece)

result = "".join(parts)
```

Почему это лучше:

- список `parts` можно **менять на месте** (append не создаёт новый список каждый раз);
- `str.join` заранее знает суммарную длину и создаёт **один буфер**, заполняя его частями.

#### Когда `s += piece` допустим

Если у вас:

- **маленький** цикл (несколько итераций),
- или строки **короткие**,

то разницы вы можете не почувствовать.  
Анти‑паттерн проявляется, когда:

- много итераций (сотни, тысячи);
- большие строки;
- операция часто повторяется в горячем участке кода.

Практическое правило:

- если вы видите `s += ...` **внутри цикла** в коде, который может работать с большими объёмами — это сигнал переписать на `join`.

### 2.5. Широкие `except Exception` и `from module import *`

Здесь два родственных анти‑паттерна:

- **слишком широкие `except Exception:`**;
- **импорты через `from module import *`**.

Они похожи тем, что оба создают **неявность**:

- вы не знаете, какие именно ошибки вы обрабатываете;
- вы не знаете, какие именно имена у вас есть в пространстве имён.

#### Широкий `except Exception`

```python
try:
    do_something()
except Exception:
    handle_error()
```

Проблема:

- вы ловите **всё, что является `Exception` или его подклассом**;
- в эту категорию попадает куча ошибок, о которых вы **не думали**:
  - баги в логике (`TypeError`, `IndexError`);
  - ошибки конфигурации;
  - неожиданные исключения из сторонних библиотек.

В результате:

- ваш код «героически всё ловит»;
- но вы не отличаете «ожидаемую ситуацию» (например, нет файла) от **реального бага** (опечатка, неверный тип).

Лучше:

```python
try:
    data = load_config(path)
except FileNotFoundError:
    create_default_config()
```

Здесь вы явно говорите:

- «если **нет файла**, я знаю, что делать»;
- всё остальное пусть пробивается наверх.

Иногда `except Exception` всё же нужен (например, на верхнем уровне веб‑приложения, чтобы логировать все необработанные ошибки), но и там:

- его используют 1–2 раза в проекте;
- он почти всегда сопровождается **логированием и пробросом/ответом с ошибкой**, а не молчаливым игнорированием.

#### `from module import *`

```python
from my_module import *
```

Проблемы:

- вы не видите в коде, **какие именно имена приехали** из `my_module`;
- если в другом модуле есть такое же имя, оно может тебя **переопределить**;
- по имени функции/класса непонятно, откуда оно (`from` какого модуля).

Пример конфликта:

```python
from math import *
from my_stats import *

result = mean(values)  # какая именно mean? из math? из my_stats?
```

Правильный подход — **явные импорты**:

```python
from my_module import SomeClass, some_function
```

или:

```python
import my_module

my_module.some_function()
```

Плюсы:

- сразу видно, какие именно имена вы используете;
- меньше шансов случайно затереть что‑то важное;
- IDE и линтеры лучше помогают (подсказки, переходы к определению).

#### Запомните

- `except Exception` без чёткого понимания, **что именно вы хотите обработать**, — почти всегда анти‑паттерн.
- `from module import *` делает код **менее предсказуемым и менее читаемым**.
- Предпочитайте:
  - **конкретные исключения**;
  - **явные импорты** с перечислением имён или префиксом модуля.

### 2.6. Опасный `eval` и модификация коллекций при итерации

#### `eval(input())` как анти‑паттерн безопасности

```python
data = eval(input("Введите выражение: "))  # ОЧЕНЬ ПЛОХО
```

Пользователь может ввести **любой Python‑код**, который выполнится с вашими правами.

Примеры того, что может ввести злоумышленник:

```python
__import__("os").remove("важный_файл.txt")
```

или:

```python
__import__("subprocess").run(["rm", "-rf", "/"], check=False)
```

Если этот код выполнится на сервере:

- он будет выполняться **с теми же правами**, что и ваша программа;
- то есть сможет удалять файлы, читать секреты, ходить в сеть и т.п.

Поэтому `eval` на пользовательском вводе — почти всегда **анти‑паттерн безопасности**.

Если вам нужно распарсить **простой литерал** (список, словарь, число, строку), используйте:

```python
import ast

data = ast.literal_eval(user_input)
```

`ast.literal_eval`:

- понимает только литералы (`[]`, `{}`, числа, строки, кортежи);
- не выполняет произвольный код;
- если ввод странный — просто бросит исключение, а не сделает что‑то опасное.

#### Модификация коллекций при итерации

```python
items = [1, 2, 3, 4]
for x in items:
    if x % 2 == 0:
        items.remove(x)  # анти‑паттерн
```

Вы изменяете список, по которому итерируетесь; индексы «съезжают», элементы пропускаются.

Пример проблемы:

```python
items = [1, 2, 3, 4]
for x in items:
    if x % 2 == 0:
        items.remove(x)
    print("x =", x, "items =", items)
```

Вывод может быть неожиданным: не все чётные удалятся, порядок изменится и т.п.

Правильные варианты:

```python
items = [1, 2, 3, 4]
items = [x for x in items if x % 2 != 0]
```

или:

```python
items = [1, 2, 3, 4]

for x in items[:]:  # копия списка
    if x % 2 == 0:
        items.remove(x)
```

Второй вариант делает копию (`items[:]`) и итерируется по ней,  
а **исходный список** можно безопасно изменять.

С `dict` и `set` Python в большинстве случаев честно бросает `RuntimeError: dictionary changed size during iteration` — это тоже защита от этого анти‑паттерна.

Общее правило:

- если нужно изменить коллекцию во время обхода — **итерируйтесь по копии** или используйте **comprehension** для создания новой коллекции.

или:

```python
for x in items[:]:  # копия
    if x % 2 == 0:
        items.remove(x)
```
