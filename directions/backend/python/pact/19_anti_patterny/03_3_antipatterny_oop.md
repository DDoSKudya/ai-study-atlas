### 3.1. Наследование ради переиспользования

#### Цель раздела

Понять, **когда наследование уместно**, а когда мы используем его просто «чтобы не дублировать код» — и попадаем в анти‑паттерн. Научиться видеть такие случаи и заменять их на **композицию** или обычные функции.

Если по‑простому:

- наследование = «X является разновидностью Y»;
- композиция = «X использует Y внутри».

Анти‑паттерн — когда мы делаем вид «является», хотя по смыслу это «использует».

#### Суть

Анти‑паттерн: использовать наследование **только ради переиспользования кода**, когда достаточно композиции или функции.

```python
class FileLogger:
    def log(self, message):
        ...


class HttpClient(FileLogger):  # ПЛОХО: клиент «является» логгером?
    def get(self, url):
        ...
```

Проблема:

- `HttpClient` **не является** логгером по смыслу;
- но из‑за наследования он унаследовал от `FileLogger` весь его интерфейс;
- любой код, который ожидает «логгер», может случайно получить `HttpClient`.

Это нарушение здравого смысла и принципа подстановки Лисков (LSP):  
подтип (`HttpClient`) не должен ломать ожидания базового типа (`FileLogger`).

#### Правильная модель: композиция

Композиция = «один объект **содержит** другой и использует его».

```python
class HttpClient:
    def __init__(self, logger):
        self.logger = logger

    def get(self, url):
        self.logger.log(f"GET {url}")
        ...
```

Теперь:

- `HttpClient` **использует** логгер, но не притворяется, что он сам логгер;
- вы можете подменить логгер (например, на заглушку в тестах);
- интерфейс `HttpClient` не загрязнён методами логгера.

#### Как понять, когда наследование — это анти‑паттерн

Задай себе несколько вопросов:

1. **Могу ли я честно сказать «X является Y»?**
   - «Круг — фигура» → да, наследование нормально.
   - «HttpClient — логгер» → нет, это странно.
2. **Использую ли я только часть методов родителя, чисто ради куска логики?**
   - если да, скорее всего, ты хочешь не наследование, а функцию/композицию.
3. **Что будет, если где‑то в коде тип `FileLogger` случайно заменится на `HttpClient`?**
   - если это звучит пугающе — лучше композиция.

Ещё один пример анти‑паттерна:

```python
class CsvExporter:
    def export(self, data):
        ...


class Report(CsvExporter):  # "Отчёт является CSV-экспортёром"? Странно.
    def generate(self):
        data = ...
        return self.export(data)
```

Лучше:

```python
class CsvExporter:
    def export(self, data):
        ...


class Report:
    def __init__(self, exporter):
        self.exporter = exporter

    def generate(self):
        data = ...
        return self.exporter.export(data)
```

#### Запомните

- Наследование — когда **объект «является» более общим типом**.
- Композиция — когда **объект «использует» другой объект**.
- Наследование «ради переиспользования кода» без смысла в домене → почти всегда анти‑паттерн.

### 3.2. Глубокие иерархии

Глубокое наследование (3+ уровней) усложняет понимание, **где что реализовано**, как работает MRO, какие методы переопределены.

#### Как это выглядит

Цепочка:

```text
Base -> Intermediate -> MoreSpecific -> Concrete
```

и где‑то в `Concrete` переопределены одни методы, в `MoreSpecific` — другие, в `Intermediate` — третьи.

Чтобы понять, что делает `obj.do_something()`, вам приходится:

- открывать несколько файлов;
- смотреть на порядок классов в MRO;
- разбираться, какой именно метод вызовется.

#### Признаки глубокой иерархии как анти‑паттерна

- длинные цепочки `Base -> Level1 -> Level2 -> Level3 -> ...`;
- классы с абстрактными названиями типа `BaseManager`, `AbstractBaseManager`, `AdvancedBaseManager`;
- новые фичи добавляются через очередное «прокладывание» ещё одного уровня.

#### Почему это плохо

- **Сложно понимать поведение.**  
  Метод может быть определён в одном классе, переопределён в другом, частично реализован в третьем.
- **Сложно менять.**  
  Любое изменение может затронуть несколько уровней наследования.
- **Сложно тестировать.**  
  Непонятно, где именно тестировать ту или иную часть логики.

#### Стратегия упрощения

- по возможности **сокращать глубину**: лучше 1–2 уровня, чем 4–5;
- использовать mixin‑классы только для **узкого, понятного поведения** (например, «умеет сохраняться в БД», «умеет логироваться»);
- максимально использовать **композицию вместо наследования**: передавать объекты как зависимости, а не растить иерархию.

Хорошая проверка:  
если вы не можете устно описать структуру вида  
«у нас есть базовый класс X, от него наследуются Y и Z, и это нужно потому что…» — иерархия, скорее всего, переросла разумные пределы.

### 3.3. Нарушение LSP

**Принцип подстановки Лисков (LSP):**  
объекты подкласса должны быть взаимозаменяемы с объектами базового класса **без изменения ожидаемого поведения**.

Анти‑паттерн:

```python
class Bird:
    def fly(self):
        ...


class Penguin(Bird):
    def fly(self):
        raise NotImplementedError("Пингвины не летают")  # нарушение LSP
```

Код, который ожидает, что любая `Bird` умеет `fly()`, сломается на `Penguin`.

#### Как думать про LSP простыми словами

Если где‑то написано, что функция принимает `Bird`, вы должны иметь возможность передать **любой** её подтип:

```python
def let_bird_fly(bird: Bird):
    bird.fly()
```

Ожидание: любой `Bird` умеет летать.  
Если вы подсунете `Penguin`, который падает с `NotImplementedError`, это нарушение LSP.

То есть:

- LSP про **обещания интерфейса**: если базовый класс обещает метод `fly`, подтипы должны уметь делать **что‑то осмысленное**, а не ломаться.

#### Как исправлять нарушения

- Делать **более точные абстракции**:

  ```python
  class Bird:
      ...

  class FlyingBird(Bird):
      def fly(self):
          ...

  class Penguin(Bird):
      ...
  ```

  Теперь `let_bird_fly` принимает `FlyingBird`, а не любой `Bird`.

- Использовать **протоколы/интерфейсы**:

  ```python
  from typing import Protocol

  class CanFly(Protocol):
      def fly(self) -> None:
          ...
  ```

  Тогда объекты, которые реализуют `fly`, явно помечены как такие.

#### Запомните

- Если подтип **ломает** ожидания базового типа (вызывает ошибку, делает совсем другое) — это нарушение LSP.
- Лучше уточнить абстракцию (разделить на несколько интерфейсов), чем впихивать всех в один базовый класс.

### 3.4. Anemic Domain Model

Анти‑паттерн: классы содержат только поля, а **вся логика лежит снаружи**, в «сервисах» или функциях.

```python
class Order:
    def __init__(self, items, status):
        self.items = items
        self.status = status


def cancel_order(order):
    if order.status != "new":
        raise ValueError("Нельзя отменить")
    order.status = "cancelled"
```

Лучше инкапсулировать логику в классе:

```python
class Order:
    def __init__(self, items, status):
        self.items = items
        self.status = status

    def cancel(self):
        if self.status != "new":
            raise ValueError("Нельзя отменить")
        self.status = "cancelled"
```

#### Почему «анемичная» модель — это проблема

- Логика рассредоточена по сервисам/функциям, которые манипулируют «голыми» данными.
- Инварианты (правила, которые всегда должны соблюдаться) **не закреплены** в одном месте.
- Чтобы понять, что значит «валидный заказ», нужно прочитать десяток функций, а не один класс.

В примере выше правило «заказ можно отменить только из статуса `new`» живёт в функции `cancel_order`.  
Если кто‑то напишет ещё одну функцию, меняющую статус, и забудет это правило — модель ломается.

Когда логика живёт **в самом объекте** (`order.cancel()`):

- становится ясно, **кто за что отвечает**;
- легче переиспользовать поведение;
- проще тестировать отдельные методы доменных объектов.

#### Запомните

- Если класс — это просто «мешок полей», а вся логика в сервисах — признак anemic domain model.
- Доменные правила лучше держать **внутри объектов**, которые эти правила отражают.
