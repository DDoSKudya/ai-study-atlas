### 5.1. Отсутствие и «символические» тесты

#### Проблема №1: вообще нет тестов

Любое изменение — риск сломать старое поведение, при этом:

- вы можете об этом **сразу не узнать**;
- баг вылезет на проде, у пользователя, а не в момент разработки.

Это как чинить проводку вслепую: без индикатора, который показывает «есть напряжение/нет».

#### Проблема №2: тесты «для галочки»

Пример бессмысленного теста:

```python
def test_something():
    do_something()
    assert True
```

Или:

```python
def test_something():
    result = do_something()
    # нет ни одного assert
```

Такие тесты:

- не проверяют **ничего конкретного**;
- дают ложное чувство безопасности: «у нас есть тесты».

#### Минимальный полезный уровень

- писать тесты на **ключевые сценарии** (основной happy‑path и важные граничные случаи);
- проверять **результат поведения**, а не просто отсутствие исключений.

Пример:

```python
def test_adds_two_numbers():
    assert add(2, 3) == 5
```

Даже такой простой тест уже:

- фиксирует ожидаемое поведение;
- сломается, если кто‑то случайно поменяет логику.

### 5.2. God test и тесты реализации

**God test** — один большой тест, который:

- проверяет сразу много сценариев;
- трудно читать и поддерживать;
- падает «по любой причине».

Пример god‑теста (упрощённо):

```python
def test_everything():
    # создаём пользователя
    # логинимся
    # создаём заказ
    # отменяем заказ
    # меняем адрес доставки
    # проверяем отчёт
    ...
```

Если такой тест падает:

- непонятно, **на каком шаге** проблема;
- править и поддерживать его тяжело.

Лучше:

- разделить на **несколько маленьких** тестов;
- каждый тест → **один сценарий**, который можно описать коротким названием.

**Тесты реализации** — тесты, которые завязаны на внутренние детали (структуру классов, приватные методы).  
При рефакторинге они ломаются, хотя поведение не изменилось.

Правило: тестируйте **внешнее поведение и API**, а не внутренности.

### 5.3. Flaky‑тесты

Flaky‑тест — тест, который иногда проходит, а иногда падает **без изменений кода**.

Причины:

- зависимость от времени (`sleep`, точное сравнение со временем);
- зависимость от порядка запуска тестов;
- использование внешних ресурсов (сеть, реальные БД) без изоляции;
- случайность (`random`) без фиксированного seed.

Типичный пример:

```python
def test_something_time_based():
    start = time.time()
    do_something()
    end = time.time()
    assert end - start < 1  # "ожидаем, что будет быстро"
```

На медленном CI такой тест будет падать без реальной причины.

Лечение:

- убирать `sleep`, использовать явные ожидания;
- моки и фиктивные реализации для внешних сервисов;
- фиксированный seed для случайности;
- очищать/изолировать состояние между тестами.
