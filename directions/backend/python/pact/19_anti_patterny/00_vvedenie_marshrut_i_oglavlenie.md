# Часть XIX. Анти-паттерны

> **Цель:** научиться **распознавать и избегать типичных анти‑паттернов** в Python‑коде (и в архитектуре), понимать **почему они возникают**, какие **риски** несут (поддерживаемость, баги, производительность, безопасность) и чем их **заменять на практике** — с примерами и простыми, но точными объяснениями.
>
> После изучения вы:
>
> - будете узнавать в чужом и своём коде **God Object, Spaghetti Code, Copy‑Paste Programming, Magic Numbers** и другие распространённые анти‑паттерны;
> - поймёте, почему опасны **bare `except`, `pass` в `except`, `eval`, `from module import *`**, мутабельные аргументы по умолчанию и модификация коллекций во время итерации;
> - научитесь отличать **плохой дизайн ООП** (слишком глубокое наследование, «анемичная» модель, нарушение подстановки Лисков) от здоровых подходов (композиция, явные интерфейсы, протоколы);
> - сможете находить **узкие места производительности** (N+1 запросы, лишние копирования, создание объектов в циклах) и переписывать код проще и быстрее;
> - поймёте, как строить **тесты**, которые помогают развивать код, а не мешают (борьба с flaky‑тестами, «god‑тестами» и тестами реализации).
>
> Всё — **простым языком**, но с сохранением терминологии (anti‑pattern, God Object, N+1 queries, Liskov Substitution Principle и т.д.).
>
> Важно: эта часть специально написана так, чтобы её мог понять человек, который считает себя «совсем тупым».  
> Мы будем объяснять **медленно, по шагам, с аналогиями**, показывать «плохо/лучше» и давать маленькие упражнения.

## 0.0. Как читать эту часть, чтобы не утонуть

Анти‑паттерны — тема тяжёлая морально: вы постоянно видите «как делать не надо».
Поэтому важно читать так, чтобы:

- не перегореть;
- не чувствовать себя «идиотом»;
- при этом **реально запомнить** и начать видеть эти вещи в коде.

Рекомендованный режим:

1. **Берите один небольшой раздел за раз.**  
   Например, только 1.1 God Object или только 2.1 Mutable default.
2. **Читайте по схеме:** цель → термины → «как выглядит» → пример «плохо» → пример «лучше» → «Запомните».
3. **После чтения найдите 1–2 похожих примера у себя/в чужом коде.**  
   Даже если это будет просто воспоминание «где‑то такое видел».
4. **Сделайте мини‑упражнение** (многие разделы подскажут, что именно переписать).
5. **На следующий день** быстро пробегитесь по шпаргалке и разделу «Запомните» без примеров.

Если чувствуете, что «голова кипит» — это не значит, что вы «тупой».  
Это значит, что мозг перестраивает ментальные модели.  
Лучше читать **короткими подходами**, чем «геройски» пытаться проглотить всё за вечер.

## Общая картина: зачем знать анти‑паттерны

Анти‑паттерн — это **типичное неудачное решение**, которое кажется удобным или «коротким», но в реальности:

- усложняет код (его трудно читать, изменять, тестировать);
- скрывает ошибки (исключения глотаются, данные меняются «где‑то там»);
- ухудшает производительность (лишние запросы, ненужные копии структур);
- создаёт риски безопасности.

Важно: анти‑паттерн — это **не просто «плохой код»**, а **устойчивый шаблон**.  
Если вы его узнаёте, вы можете:

- быстро понять, **почему проект тяжело поддерживать**;
- предложить **конкретный план рефакторинга**;
- **не повторять** те же ошибки в новом коде.

Удобно думать так:

- **Паттерн** — повторяемое удачное решение.
- **Анти‑паттерн** — повторяемое неудачное решение.

Эта часть связана с разделами глобального плана:

- §104 — общие анти‑паттерны;
- §105 — Python‑специфичные анти‑паттерны;
- §106 — анти‑паттерны ООП;
- §107 — анти‑паттерны производительности;
- §108 — анти‑паттерны тестирования.

## 0.05. Мини‑словарь терминов (простыми словами)

Чтобы дальше текст не пугал словами, зафиксируем базовые термины:

- **паттерн (pattern)** — *удачное* повторяющееся решение проблемы. Пример: использовать логгер вместо `print` в продакшене.
- **анти‑паттерн (anti‑pattern)** — *неудачное* повторяющееся решение, которое кажется удобным, но вредит (поддерживаемости, скорости, безопасности).
- **code smell (запах в коде)** — признак, что «что‑то не так», но это ещё не обязательно полноценный анти‑паттерн.
- **SRP (Single Responsibility Principle)** — принцип единственной ответственности: у сущности должна быть одна причина для изменения.
- **cohesion (связанность)** — насколько сильно части одного модуля/класса связаны по смыслу. Высокая связанность = файл/класс «про одно».
- **coupling (сцепление)** — насколько сильно модули/классы зависят друг от друга. Низкое сцепление = их можно менять по отдельности.
- **LSP (Liskov Substitution Principle)** — принцип подстановки Лисков: там, где ожидается базовый тип, должен без сюрпризов работать любой его подтип.
- **N+1 queries** — вместо одного умного запроса: один запрос за списком + ещё N отдельных запросов в цикле.
- **flaky‑тест** — тест, который на одном и том же коде то проходит, то падает (из‑за времени, порядка, окружения).

К этим словам мы будем возвращаться много раз дальше по тексту.

## Краткая шпаргалка

| Анти‑паттерн | Суть простыми словами | Чем заменить / что делать |
|-------------|------------------------|----------------------------|
| **God Object** | Один «большой класс», который «делает всё» | Разбивать на небольшие классы/модули по ответственности (SRP), использовать композицию |
| **Spaghetti Code** | Запутанный поток управления, много взаимозависимостей | Ясные функции, слои, явные зависимости, избегать глобального состояния |
| **Copy‑Paste Programming** | Дублирование кода вместо обобщения | Выделять общие функции/классы, применять DRY |
| **Magic Numbers / Strings** | Необъяснимые числа/строки посреди кода | Вынести в именованные константы, Enum, конфиг |
| **Bare `except:`** | Перехватывать «всё подряд» | Ловить конкретные исключения, логировать, по возможности пробрасывать дальше |
| **`except: pass`** | Игнорировать ошибки | Минимум — логировать; лучше обрабатывать или пробрасывать |
| **Mutable default argument** | `def f(a=[])` — один список на все вызовы | Использовать `None` и создавать коллекцию внутри |
| **`from module import *`** | Загрязнение пространства имён, неясно, откуда что | Явно импортировать имена (`import` / `from ... import name`) |
| **Модификация коллекций при итерации** | Удаление/добавление во время обхода ломает логику | Итерироваться по копии (`lst[:]`, `list(d)`), использовать comprehension |
| **Глубокое наследование** | Большие иерархии классов, сложно понять поведение | Плоское наследование, композиция, протоколы/абстракции |
| **Anemic Domain Model** | Классы только с данными, логика «где‑то вовне» | Переносить поведение в сами объекты, продумывать инварианты |
| **N+1 queries** | Запрос в БД/сеть в каждом шаге цикла | Жадная загрузка, join, batch‑запросы |
| **Ненужные копии структур** | `list(list_obj)`, `data[:]` без повода | Работать с исходной коллекцией, когда безопасно |
| **God test** | Один тест проверяет слишком много | Делить на мелкие, проверяющие один сценарий |
| **Flaky tests** | Тесты, которые иногда падают «случайно» | Убирать недетерминизм, стабилизировать окружение |

## Маршрут изучения

1. **Этап 1 — Общие анти‑паттерны (§104).**  
   Узнаете God Object, Spaghetti Code, Copy‑Paste Programming, Magic Numbers, bare `except`, `except: pass`. Научитесь видеть их «вживую» и заменять на здоровые решения.
2. **Этап 2 — Python‑специфичные анти‑паттерны (§105).**  
   Разберёте mutable default arguments, `len(x) > 0` вместо `if x`, сравнение с `None` через `==`, конкатенацию строк в цикле, широкие `except Exception`, `from module import *`, `eval`, изменение списка в цикле.
3. **Этап 3 — Анти‑паттерны ООП (§106).**  
   Поймёте, чем плохи наследование «ради переиспользования», глубокие иерархии, нарушение принципа подстановки Лисков и «анемичная» модель.
4. **Этап 4 — Анти‑паттерны производительности (§107).**  
   Научитесь замечать N+1 запросы, лишние аллокации, повторное создание тяжёлых объектов, глобальные лукапы в горячих циклах.
5. **Этап 5 — Анти‑паттерны тестирования (§108).**  
   Разберёте, почему отсутствие тестов и плохие тесты — разные проблемы; как бороться с flaky‑тестами и «god‑тестами», как тестировать поведение, а не реализацию.

**Желательно знать заранее:** базовый синтаксис Python, функции и области видимости, основы ООП, базовое представление о тестах и работе с БД/сетями.

## 1. Общие анти‑паттерны (§104)
