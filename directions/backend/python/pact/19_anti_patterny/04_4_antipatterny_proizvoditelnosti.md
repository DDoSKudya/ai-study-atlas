### 4.1. N+1 queries

#### Цель раздела

Научиться:

- **узнавать** паттерн N+1 в своём и чужом коде;
- понимать, **почему он медленный** (и где станет особенно плохо);
- знать **конкретные приёмы**, как переписать код так, чтобы вместо сотен/тысяч мелких запросов было несколько крупных.

#### Термины

| Термин | Простое объяснение |
|--------|--------------------|
| **N+1 queries** | Сначала 1 запрос за «списком», потом N запросов — по одному на каждый элемент этого списка. |
| **Eager loading (жадная загрузка)** | Заранее загрузить связанные данные в одном (или нескольких крупных) запросах. |
| **Batch‑запрос** | Один запрос, который обрабатывает сразу множество сущностей (например, `WHERE id IN (...)`). |

#### Суть анти‑паттерна

Классический пример:

```python
orders = get_orders()          # 1 запрос: взять список заказов
for order in orders:
    user = get_user(order.user_id)  # ещё 1 запрос на КАЖДЫЙ заказ
```

Если заказов 1000 — будет:

- 1 запрос за заказами;
- 1000 запросов за пользователями;
- итого **1001 запрос**, хотя можно обойтись парой.

Интуитивно:

- вы идёте в магазин за каждым продуктом **по отдельности**;
- вместо того чтобы взять список и сходить **один раз**.

#### Почему это плохо (чуть глубже)

- Каждый запрос в БД/по сети — это:
  - сетевой круг (latency),
  - нагрузка на сервер;
- при росте количества элементов N общее время часто растёт **почти линейно от N**, а с учётом оверхеда может ощущаться чуть ли не как квадрат.

Если сегодня у вас 10 заказов — вы этого не заметите.  
Но когда станет 10 000 — система начнёт «таять».

#### Как это выглядит в типичном ORM‑коде

Пример на ORM (Django‑подобный код):

```python
posts = Post.objects.all()   # один запрос за постами
for post in posts:
    author = post.author     # если не настроен select_related — отдельный запрос за автором
```

Под капотом:

- первый запрос: `SELECT * FROM posts;`
- потом для каждого поста: `SELECT * FROM users WHERE id = <author_id>;`

Итого: 1 + N запросов.

Для HTTP‑клиента (микросервисы) аналогично:

```python
user_ids = get_user_ids()  # один запрос
for user_id in user_ids:
    details = fetch_user_details(user_id)  # N HTTP-запросов по одному
```

#### Как найти N+1 в проекте

Признаки:

- есть цикл `for ... in ...`, внутри которого вызывается:
  - метод ORM, который ходит в базу (`.get()`, `.filter()`, `.all()` и т.д.);
  - или сетевой вызов/HTTP‑клиент;
- профайлер/логи показывают **очень много коротких похожих запросов** подряд.

Практический приём:

- включить логирование SQL/HTTP;
- запустить «подозричный» сценарий;
- посмотреть, нет ли последовательностей вида:

```text
SELECT ... WHERE id = 1
SELECT ... WHERE id = 2
SELECT ... WHERE id = 3
...
```

или однотипных HTTP‑запросов.

#### Стратегии исправления

1. **Жадная загрузка (eager loading) в ORM.**

   Вместо:

   ```python
   posts = Post.objects.all()
   for post in posts:
       author = post.author
   ```

   используем:

   ```python
   posts = Post.objects.select_related("author")
   for post in posts:
       author = post.author  # автор уже подгружен
   ```

   Теперь ORM делает:

   - один `JOIN`‑запрос, который сразу приносит и посты, и авторов.

2. **Batch‑запросы / `IN` в SQL.**

   Вместо:

   ```python
   users = []
   for user_id in user_ids:
       users.append(get_user(user_id))
   ```

   делаем один запрос:

   ```sql
   SELECT * FROM users WHERE id IN (...список всех user_id...);
   ```

   В ORM это обычно выглядит как:

   ```python
   users = User.objects.filter(id__in=user_ids)
   users_by_id = {u.id: u for u in users}

   for order in orders:
       user = users_by_id[order.user_id]
   ```

3. **Batch для HTTP.**

   Вместо:

   ```python
   for user_id in user_ids:
       fetch_user_details(user_id)
   ```

   если сервер поддерживает пакетный запрос — один вызов:

   ```python
   details_by_id = fetch_many_user_details(user_ids)
   ```

   Если пакетного API нет — хотя бы ограничить параллелизм/частоту, но это уже архитектурная задача.

#### Запомните

- N+1 — это про **структуру кода**: цикл + запрос внутри.
- Лечение почти всегда сводится к тому, чтобы:
  - заранее узнать все нужные идентификаторы;
  - сделать **1–2 крупных запроса**;
  - разложить результат в структуру (`dict` по id) и использовать в цикле **без дополнительных запросов**.

### 4.2. Лишние аллокации и копии

#### Цель раздела

Понять:

- что такое «аллокация» и «копия» на практике;
- какие привычки приводят к лишнему расходу памяти и времени;
- как распознать и убрать «копирование на всякий случай», не ломая поведение.

#### Примеры анти‑паттерна

- `list(list_obj)` без необходимости;
- `data[:]` только «на всякий случай».

Такие копии увеличивают расход памяти и время выполнения.

Пример:

```python
def process(users):
    users = list(users)  # "на всякий случай"
    ...
```

Если `users` и так уже список, вы просто создали **второй такой же**.

Другой пример:

```python
data = load_data()
copy1 = data[:]    # на всякий случай
copy2 = list(data) # ещё раз на всякий случай
```

В итоге в памяти лежит **несколько одинаковых структур**, и каждая занимает место.

#### Когда копия действительно нужна

Копирование оправдано, если:

- вы собираетесь **изменять структуру отдельно от исходной** (и не хотите трогать оригинал);
- вы хотите «зафиксировать снимок» данных на конкретный момент времени;
- вам нужно работать с объектом в другом потоке/процессе, не меняя общий.

Примеры, где копия уместна:

```python
def process_and_keep_original(items):
    items_copy = list(items)  # будем изменять только копию
    items_copy.sort()
    return items_copy
```

или:

```python
def safe_iterate(mapping):
    # dict может меняться снаружи, поэтому берём список ключей
    for key in list(mapping.keys()):
        ...
```

#### Как искать лишние копии

Чек‑лист:

- Ищите по коду `list(`, `[:]`, `copy()`, `deepcopy()`.
- Спрашивайте себя для каждого случая:
  - «Я дальше **изменяю** эту копию?»
  - «Мне правда нужно два (три, четыре) экземпляра этих данных?»

Если ответ «нет» — скорее всего, копия лишняя.

#### Правило

Делайте копию, **только если она действительно нужна**:

- вы явным текстом можете объяснить, **зачем** она нужна (отделить состояние, зафиксировать снимок и т.д.);
- если объяснить нечего, это, скорее всего, «копия на всякий случай» → анти‑паттерн.

### 4.3. Создание объектов в горячем цикле

Пример:

```python
for _ in range(10_000):
    pattern = re.compile(r"\d+")  # компиляция на каждой итерации
    ...
```

Правильно:

```python
pattern = re.compile(r"\d+")
for _ in range(10_000):
    ...
```

Идея: выносить **тяжёлые операции** из циклов.

Тяжёлые = то, что:

- делает работу с внешними ресурсами (БД, сеть, диск);
- много аллоцирует (создаёт большие структуры, компилирует шаблоны/регекспы);
- инициализирует сложные объекты.

### 4.4. Глобальные лукапы в циклах

В Python доступ к локальной переменной быстрее, чем к глобальной или атрибуту.

Пример оптимизации:

```python
from math import sqrt

def process(nums):
    local_sqrt = sqrt
    result = []
    for n in nums:
        result.append(local_sqrt(n))
    return result
```

Это особенно заметно в очень горячих циклах.

Важно: это **микро‑оптимизация**.  
Сначала стоит исправить более крупные анти‑паттерны (N+1, лишние копии, тяжёлые операции в циклах), а уже потом — такие тонкости.
