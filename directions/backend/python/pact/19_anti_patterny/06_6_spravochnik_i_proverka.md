### 6.1. Сценарии «анти‑паттерн → рефакторинг»

| Анти‑паттерн | Как выглядит | Что делать |
|--------------|--------------|-----------|
| God Object | Огромный класс/модуль «про всё» | Разбить по ответственности, ввести сервисы/слои, использовать композицию |
| Spaghetti Code | Много вложенных `if/else` и переходов | Выделить функции по шагам процесса, использовать guard‑clauses |
| Copy‑Paste Programming | Один и тот же код в нескольких местах | Выделить общую функцию/класс, параметризовать различия |
| Magic Numbers | Необъяснимые числа/строки | Вынести в константы, Enum |
| Bare `except`, `except: pass` | Ловля всего и игнор исключений | Ловить конкретные исключения, логировать, при необходимости пробрасывать |
| Mutable default | `def f(a=[])` | Заменить на `a=None` и создавать список внутри |
| Конкатенация строк в цикле | `s += part` в большом цикле | Собрать части в список и `''.join(parts)` |
| N+1 queries | Запросы в БД в каждом шаге цикла | Eager loading, `JOIN`, batch‑запросы |
| God test | Огромный тест на много сценариев | Разбить на простые, независимые тесты |
| Flaky‑тест | Иногда проходит, иногда падает | Убрать недетерминизм, стабилизировать окружение |

---

### 6.2. Терминология

Краткий словарь:

- **Anti‑pattern** — устойчивый шаблон плохого решения.
- **God Object** — объект, делающий слишком много.
- **Spaghetti Code** — запутанный код с неясным потоком управления.
- **Magic number/string** — «магическое» значение без объяснения.
- **Mutable default argument** — мутабельный аргумент по умолчанию.
- **N+1 queries** — множество мелких запросов вместо нескольких крупных.
- **Flaky test** — нестабильный тест.

### 6.3. Вопросы для самопроверки

Ниже — вопросы по **каждому разделу** части. Ответы спрятаны под `<details>`, чтобы можно было сначала подумать самому.

#### 1.1. God Object

<details>
<summary>1.1.1. Как по признакам понять, что перед вами God Object?</summary>

- Класс/модуль огромный, содержит кучу полей и методов.
- Внутри вперемешку: работа с БД, сетью, бизнес‑логикой, логированием, кешированием, отчётами.
- На него завязано «пол‑проекта»: почти везде он импортируется и используется.
- Любое изменение требует лезть в этот класс и бояться сломать что‑то ещё.
</details>

<details>
<summary>1.1.2. Почему God Object опасен в долгосрочной перспективе?</summary>

Потому что:

- все изменения концентрируются в одном месте → растущая сложность и риск конфликтов;
- сильное сцепление: разные подсистемы зависят от одного и того же куска;
- трудно тестировать отдельные части поведения;
- команда начинает бояться менять этот класс, технический долг растёт.
</details>

<details>
<summary>1.1.3. Опишите пошагово план рефакторинга God Object на более мелкие сервисы.</summary>

Краткий план:

1. Закрыть базовыми тестами или хотя бы скриптом «основной сценарий» (чтобы видеть, что ничего не сломали).
2. Выписать области ответственности (пользователи, товары, отчёты, логирование и т.д.).
3. Сгруппировать методы по этим областям.
4. Создать отдельные классы/сервисы под каждую область.
5. В God Object временно оставить методы‑прокладки, которые делегируют в новые сервисы.
6. Постепенно переводить вызовы на новые сервисы.
7. Удалить ненужные прокладки, когда внешний код больше не использует старые методы.
</details>

#### 1.2. Spaghetti Code

<details>
<summary>1.2.1. Какие признаки указывают, что функция превратилась в Spaghetti Code?</summary>

- Много вложенных `if/elif/else`, иногда на 3–4 уровня.
- Внутри функции инициализируется куча переменных, которые живут весь её объём.
- Трудно словами описать «главный путь» выполнения.
- Чтобы понять один сценарий, нужно глазами прыгать вверх‑вниз по коду.
</details>

<details>
<summary>1.2.2. Как с помощью guard‑clauses (ранних return) можно упростить такую функцию?</summary>

Идея:

- вместо вложенных `if` писать проверки на «плохие случаи» в начале и сразу `return`;
- тем самым happy‑path идёт «по прямой» без глубокой вложенности.

Примерно так:

- было: `if cond1: ... if cond2: ... if cond3: ...`;
- стало: `if not cond1: return`, `if not cond2: return`, `if not cond3: return`, дальше — основной сценарий.
</details>

<details>
<summary>1.2.3. Опишите алгоритм «разматывания лапши» для длинной функции.</summary>

1. Найти функцию, которую тяжело читать.
2. Сформулировать словами 3–7 шагов, что она делает.
3. Для каждого шага выделить отдельную функцию/метод с понятным названием.
4. В исходной функции заменить куски логики вызовами этих функций.
5. Убедиться тестами или ручной проверкой, что поведение не изменилось.
</details>

#### 1.3. Copy‑Paste Programming

<details>
<summary>1.3.1. Почему дублирование кода опасно в долгой перспективе?</summary>

Потому что одну и ту же ошибку придётся исправлять в нескольких местах, и легко забыть про одну из копий. В итоге логика «расходится»: в одном месте поведение уже исправлено, в другом — нет, а код выглядит очень похожим.
</details>

<details>
<summary>1.3.2. Какой общий приём используется, чтобы убрать дублирование в двух почти одинаковых функциях?</summary>

Нужно:

- положить две функции рядом;
- найти, что у них общее, а что различается;
- общую часть превратить в одну функцию/метод;
- различающиеся куски вынести в параметры (например, коэффициент скидки или шаблон строки).
</details>

<details>
<summary>1.3.3. Почему «слишком агрессивное» обобщение тоже может быть анти‑паттерном?</summary>

Если радикально обобщить код:

- итоговая функция может стать слишком абстрактной и трудночитаемой;
- смысл домена теряется за уровнем параметров и флагов;
- поддержка сложнее, чем у пары понятных, чуть‑чуть дублирующих функций.

Поэтому обобщать стоит только реально повторяющиеся куски логики, когда это действительно упрощает жизнь.
</details>

#### 1.4. Magic Numbers и Magic Strings

<details>
<summary>1.4.1. Что такое magic number и почему он мешает поддержке кода?</summary>

Magic number — это «голое» число в коде без объяснения (например, `18`, `0.1`, `42`), значение которого неочевидно без контекста.  
Проблема: непонятно, что оно значит, и если бизнес‑правило меняется (18 → 21), нужно искать все вхождения этого числа, рискуя что‑то пропустить.
</details>

<details>
<summary>1.4.2. Как правильно заменить magic number на говорящую константу?</summary>

Нужно:

1. Придумать осмысленное имя, отражающее смысл числа (например, `ADULT_AGE`, `MAX_RETRY_COUNT`).
2. Объявить константу в подходящем месте (вверху модуля, в конфиге).
3. Заменить в коде «голое» число на имя константы.
</details>

<details>
<summary>1.4.3. Приведите пример магической строки и её замены на константы.</summary>

Плохо:

```python
if status == "ok":
    ...
elif status == "error":
    ...
```

Лучше:

```python
STATUS_OK = "ok"
STATUS_ERROR = "error"

if status == STATUS_OK:
    ...
elif status == STATUS_ERROR:
    ...
```
</details>

#### 1.5. Анти‑паттерны обработки ошибок

<details>
<summary>1.5.1. Чем опасен bare <code>except:</code>, особенно с <code>pass</code> внутри?</summary>

Опасности:

- он ловит вообще все исключения, включая системные (`KeyboardInterrupt`, `SystemExit`);
- полностью прячет ошибку: нет логов, нет сообщения, программа просто «молча не делает то, что должна»;
- отладка усложняется, потому что не видно, где и что сломалось.
</details>

<details>
<summary>1.5.2. В чём разница между <code>except Exception:</code> и ловлей конкретного типа исключения?</summary>

- `except Exception:` ловит **все пользовательские исключения**, в том числе те, которые вы не ожидали (ошибки в логике, конфигурации и т.д.).
- Ловля конкретного типа (`FileNotFoundError`, `ValueError`) явно документирует, какую именно ситуацию вы обрабатываете и почему.
</details>

<details>
<summary>1.5.3. Когда игнорирование исключения (через suppress) допустимо, и почему это не считается анти‑паттерном?</summary>

Например:

```python
from contextlib import suppress
import os

with suppress(FileNotFoundError):
    os.remove("temp.txt")
```

Допустимо, потому что:

- ловится **конкретный** тип ошибки (`FileNotFoundError`);
- для данной ситуации поведение «если файла нет — это не ошибка» осмысленно и ожидаемо.
</details>

#### 2.1. Mutable default argument

<details>
<summary>2.1.1. Почему <code>def f(a=[])</code> ведёт себя неожиданно для многих разработчиков?</summary>

Потому что список `[]` создаётся **один раз** при определении функции и кладётся в `__defaults__`.  
Все вызовы без аргумента используют один и тот же объект списка, который накапливает изменения между вызовами.
</details>

<details>
<summary>2.1.2. Как переписать функцию с мутабельным аргументом по умолчанию на безопасный вариант?</summary>

Шаблон:

```python
def f(a=None):
    if a is None:
        a = []
    ...
```

Мы используем `None` как маркер «аргумент не передали», а список создаём заново в каждом таком вызове.
</details>

<details>
<summary>2.1.3. Почему нельзя просто писать <code>a = a or []</code> вместо проверки на <code>None</code>?</summary>

Потому что:

- `0`, `''`, `False`, пустой список/dict/set — все считаются `False` в булевом смысле;
- `a or []` не различает «аргумент не передали» и «передали, но он пустой/ложный»;
- вы можете потерять переданное значение, подменив его новым списком.
</details>

#### 2.2. Truthiness и `len(x) > 0`

<details>
<summary>2.2.1. Почему в Python проверка на непустоту через <code>if x:</code> считается более идиоматичной, чем <code>if len(x) &gt; 0:</code>?</summary>

Потому что:

- у контейнеров булевое значение по умолчанию зависит от пустоты (`bool([]) == False`, `bool([1]) == True`);
- `if x:` короче и отражает идею «пусто/непусто» напрямую;
- `len(x) > 0` добавляет лишний шум и тянет стиль из других языков.
</details>

<details>
<summary>2.2.2. Когда использование <code>len(x)</code> оправдано, а не считается стилистическим анти‑паттерном?</summary>

Когда вам нужно именно **количество** элементов, а не просто факт пустоты.  
Например:

- проверка на превышение лимита (`len(items) > MAX_COUNT`);
- вывод числа в лог или интерфейс (`print(f"Найдено {len(results)} результатов")`).
</details>

<details>
<summary>2.2.3. Как можно задать «правдивость» (truthiness) для собственного класса?</summary>

Реализовать метод `__bool__`:

```python
class Bucket:
    def __init__(self):
        self.items = []

    def __bool__(self):
        return bool(self.items)
```

Теперь `if bucket:` будет считаться `True`, когда в `items` что‑то есть.
</details>

#### 2.3. Сравнение с `None`

<details>
<summary>2.3.1. Почему сравнение с <code>None</code> нужно делать через <code>is</code>, а не через <code>==</code>?</summary>

Потому что:

- `None` — это singleton‑объект, нас интересует, **является ли переменная именно им**;
- оператор `==` может быть переопределён (`__eq__`), и пользовательский объект может вернуть неожиданный результат при сравнении с `None`.
</details>

<details>
<summary>2.3.2. Приведите пример, где <code>x == None</code> и <code>x is None</code> дают разный результат.</summary>

```python
class Weird:
    def __eq__(self, other):
        return True

obj = Weird()
obj == None   # True
obj is None   # False
```

Здесь `__eq__` заставляет `== None` возвращать `True`, хотя объект никакого отношения к `None` не имеет.
</details>

<details>
<summary>2.3.3. Сформулируйте правило: когда использовать <code>is</code>, а когда <code>==</code>.</summary>

- Для `None` → всегда `is` / `is not`.
- Для обычных значений (`0`, строки, объекты) → `==` / `!=` (проверяем равенство содержимого).
</details>

#### 2.4. Конкатенация строк в цикле

<details>
<summary>2.4.1. Почему <code>s += part</code> в большом цикле считается анти‑паттерном по производительности?</summary>

Потому что строки в Python **иммутабельны**:

- каждое `+=` создаёт новую строку, копируя содержимое старой плюс новую часть;
- при большом числе итераций вы создаёте много временных строк и тратите время и память.
</details>

<details>
<summary>2.4.2. Какой шаблон рекомендуется использовать для эффективной сборки строк из частей?</summary>

Собирать части в список, а потом делать `''.join(parts)`:

```python
parts = []
for piece in pieces:
    parts.append(piece)

result = "".join(parts)
```

Или, если части уже лежат в коллекции, сразу `result = "".join(pieces)`.
</details>

<details>
<summary>2.4.3. В каких случаях использование <code>s += part</code> допустимо и не приведёт к заметным проблемам?</summary>

Когда:

- количество итераций невелико (несколько штук);
- строки короткие;
- код не находится в «горячем» участке, который выполняется много раз.

То есть это вопрос **контекста и масштаба**, но по умолчанию в больших циклах лучше избегать `+=` для строк.
</details>

#### 2.5. Широкие `except Exception` и `from module import *`

<details>
<summary>2.5.1. Почему <code>except Exception:</code> без ясной цели считается анти‑паттерном?</summary>

Потому что:

- ловит слишком широкий диапазон ошибок, в том числе неожиданные баги;
- прячет разницу между «ожидаемой ситуацией» (нет файла, таймаут) и реальной ошибкой логики;
- усложняет отладку: непонятно, что именно вы хотели обрабатывать.
</details>

<details>
<summary>2.5.2. Какие проблемы создаёт использование <code>from module import *</code>?</summary>

- Непонятно, какие именно имена были импортированы — это скрытые зависимости.
- Возможны конфликты имён между разными модулями.
- При чтении кода трудно понять, из какого модуля взята та или иная функция/класс.
</details>

<details>
<summary>2.5.3. Какой стиль импорта считается более безопасным и читаемым?</summary>

Два варианта:

- Явный импорт имён:

  ```python
  from my_module import SomeClass, some_function
  ```

- Импорт модуля целиком:

  ```python
  import my_module
  my_module.some_function()
  ```
</details>

#### 2.6. Опасный `eval` и модификация коллекций при итерации

<details>
<summary>2.6.1. Почему <code>eval(input())</code> на пользовательском вводе — серьёзный анти‑паттерн безопасности?</summary>

Потому что пользователь может ввести **любой Python‑код**, который:

- выполнится с правами вашего процесса;
- может читать/удалять файлы, запускать команды, ходить в сеть и т.д.

То есть вы открываете прямой канал для произвольного выполнения кода (code injection).
</details>

<details>
<summary>2.6.2. Чем <code>ast.literal_eval</code> безопаснее обычного <code>eval</code>?</summary>

`ast.literal_eval`:

- умеет парсить только простые литералы (`[]`, `{}`, числа, строки, кортежи);
- **не выполняет** произвольный код;
- при странном вводе просто бросит исключение, а не выполнит потенциально опасную операцию.
</details>

<details>
<summary>2.6.3. Почему прямое изменение списка при итерации по нему считается анти‑паттерном?</summary>

Потому что:

- вы меняете структуру (длину, индексы) в тот момент, когда по ней уже идёт итерация;
- индексы «съезжают», часть элементов может быть пропущена;
- поведение становится неочевидным и зависит от внутренней реализации цикла.

Правильный подход — итерироваться по копии (`items[:]`) или использовать comprehension/новую коллекцию.
</details>

#### 3.1. Наследование ради переиспользования

<details>
<summary>3.1.1. В чём отличие композиции от наследования «ради переиспользования кода»?</summary>

- Наследование говорит «X **является** Y».
- Композиция говорит «X **использует** Y внутри».

Когда мы наследуемся только ради того, чтобы «взять пару методов», часто нарушается здравый смысл и принципы дизайна.
</details>

<details>
<summary>3.1.2. Почему <code>class HttpClient(FileLogger)</code> — пример плохого наследования?</summary>

Потому что по смыслу:

- HTTP‑клиент **не является** логгером;
- он просто использует логгер, чтобы писать сообщения.

Такое наследование ломает ожидания: код, который ждёт логгер, внезапно может получить HTTP‑клиент.
</details>

<details>
<summary>3.1.3. Как переписать такой пример на композицию?</summary>

```python
class HttpClient:
    def __init__(self, logger):
        self.logger = logger

    def get(self, url):
        self.logger.log(f"GET {url}")
        ...
```

Теперь:

- `HttpClient` использует логгер, но не выдаёт себя за него;
- интерфейсы разделены по смыслу.
</details>

#### 3.2. Глубокие иерархии

<details>
<summary>3.2.1. Какие признаки говорят о том, что иерархия наследования слишком глубокая?</summary>

- Длинные цепочки `Base -> Level1 -> Level2 -> Level3 -> ...`.
- Много абстрактных классов с размытыми названиями (`AbstractBaseManager`, `AdvancedBaseClass` и т.п.).
- Чтобы понять поведение метода, приходится открывать несколько файлов и отслеживать переопределения по всей цепочке.
</details>

<details>
<summary>3.2.2. Почему глубокая иерархия ухудшает поддерживаемость кода?</summary>

Потому что:

- усложняется ментальная модель: трудно понять, откуда именно берётся поведение;
- любые изменения могут неожиданно повлиять на подклассы или базовые классы;
- тестирование усложняется: нужно учитывать взаимодействие между уровнями.
</details>

<details>
<summary>3.2.3. Какие шаги можно предпринять, чтобы упростить слишком глубокую иерархию?</summary>

- Сократить количество уровней (объединить уровни, если они мало что добавляют).
- Вынести повторяющееся поведение в mixin‑классы или отдельные вспомогательные объекты.
- Чаще использовать композицию: передавать зависимости как поля, а не через наследование.
</details>

#### 3.3. Нарушение LSP

<details>
<summary>3.3.1. Сформулируйте принцип подстановки Лисков (LSP) простыми словами.</summary>

Если код работает с объектами типа `Base`, то он должен так же корректно работать с любыми объектами типа `Sub`, которые являются подтипами `Base`, **без изменения ожидаемого поведения**.  
Подтип не должен ломать обещания базового типа.
</details>

<details>
<summary>3.3.2. Почему пример с <code>Penguin(Bird)</code>, у которого <code>fly()</code> выбрасывает исключение, нарушает LSP?</summary>

Потому что:

- базовый класс `Bird` своим интерфейсом подразумевает, что любая птица умеет `fly()`;
- подтип `Penguin` нарушает это обещание, выбрасывая `NotImplementedError`.

Код, который ожидает, что `Bird.fly()` работает для любой птицы, ломается при передаче `Penguin`.
</details>

<details>
<summary>3.3.3. Как можно переработать иерархию, чтобы соблюсти LSP в этом примере?</summary>

Например:

- ввести `FlyingBird` как подтип `Bird`, у которого есть метод `fly`;
- не наследовать `Penguin` от `FlyingBird`, а только от `Bird`;
- функции, которым нужны «летающие птицы», принимать `FlyingBird`, а не любой `Bird`.
</details>

#### 3.4. Anemic Domain Model

<details>
<summary>3.4.1. В чём суть анемичной доменной модели (anemic domain model)?</summary>

В том, что:

- классы домена содержат только поля (данные);
- вся логика живёт в отдельных сервисах/функциях;
- объекты становятся просто «пакетами данных», а не носителями поведения и правил.
</details>

<details>
<summary>3.4.2. Почему анемичная модель затрудняет поддержку и развитие домена?</summary>

Потому что:

- инварианты (правила, которые всегда должны выполняться) размазаны по коду;
- чтобы понять, что значит «валидное состояние объекта», нужно искать по многим функциям;
- легко нарушить правила (написать новый сервис, который меняет данные, но не соблюдает все проверки).
</details>

<details>
<summary>3.4.3. Как перенести часть логики из сервисов в сами доменные объекты?</summary>

Нужно:

- найти функции, которые работают с объектом, используя его поля;
- перенести их внутрь класса в виде методов (например, `order.cancel()` вместо `cancel_order(order)`);
- следить, чтобы инварианты проверялись внутри методов доменных объектов, а не только снаружи.
</details>

#### 4.1. N+1 queries

<details>
<summary>4.1.1. Как по коду распознать потенциальный N+1 анти‑паттерн?</summary>

Признаки:

- есть цикл `for ... in ...`;
- внутри цикла вызывается метод, который ходит в БД (`.get()`, `.filter()`, `.all()` и т.п.) или делает HTTP‑запрос;
- при логировании видно много однотипных запросов подряд (`SELECT ... WHERE id = ...` для разных id).
</details>

<details>
<summary>4.1.2. Почему N+1 может долго оставаться незаметным, а потом вдруг «выстрелить» по производительности?</summary>

Потому что:

- на маленьких объёмах (10–50 элементов) задержка от лишних запросов мала;
- когда объём данных растёт до сотен/тысяч, общее время начинает расти почти линейно (а с учётом сетевых накладных расходов — ощущается как очень медленное поведение);
- проблема проявляется уже в проде, а не на маленьких тестовых данных.
</details>

<details>
<summary>4.1.3. Назовите три подхода к устранению N+1‑запросов.</summary>

- Жадная загрузка (eager loading) в ORM (`select_related`, `prefetch_related`).
- `JOIN` и выборка связанных данных одним SQL‑запросом.
- Batch‑запросы (например, `WHERE id IN (...)` или HTTP‑эндпоинт, принимающий список `id`).
</details>

#### 4.2. Лишние аллокации и копии

<details>
<summary>4.2.1. Чем опасно копирование коллекций «на всякий случай», без явной причины?</summary>

Тем, что:

- создаются дополнительные структуры, которые занимают память;
- тратится время на копирование;
- код засоряется ненужными действиями, усложняя понимание.

В итоге производительность падает, а пользы нет.
</details>

<details>
<summary>4.2.2. В каких ситуациях создание копии коллекции оправдано?</summary>

Например, когда:

- нужно менять коллекцию независимо от оригинала (не затрагивая исходные данные);
- нужно зафиксировать снимок коллекции на момент времени;
- внешний код может менять коллекцию во время итерации, и мы хотим итерироваться по стабильному набору (`list(mapping.keys())`).
</details>

<details>
<summary>4.2.3. Как практически искать в коде потенциально лишние копии коллекций?</summary>

Искать по:

- `list(`,
- `[:]`,
- `copy()`,
- `deepcopy()`.

И затем для каждого случая задавать себе вопрос: «я дальше реально **использую** независимую копию или просто прогоняю по ней тот же сценарий?».
</details>

#### 4.3. Создание объектов в горячем цикле

<details>
<summary>4.3.1. Приведите пример тяжёлой операции, которую не стоит выполнять в теле горячего цикла.</summary>

Например:

- регулярное выражение с `re.compile` внутри цикла;
- установление соединения с базой/внешним сервисом;
- чтение больших конфигураций/файлов;
- создание больших структур данных (словарей, списков) заново на каждой итерации.
</details>

<details>
<summary>4.3.2. Как вынести тяжёлую операцию из цикла, не меняя поведение кода?</summary>

Общий приём:

- выполнить тяжёлую операцию **до цикла** (один раз);
- внутри цикла использовать уже готовый объект.

Пример с `re.compile`:

```python
pattern = re.compile(r"\d+")
for line in lines:
    match = pattern.search(line)
```
</details>

<details>
<summary>4.3.3. Почему важно сначала исправлять крупные анти‑паттерны, а потом уже думать о таких оптимизациях?</summary>

Потому что крупные анти‑паттерны (N+1, лишние копии, неоптимальные алгоритмы) дают **на порядки больший** выигрыш, чем мелкие оптимизации.  
Если тратить усилия только на микро‑оптимизации, можно «полировать» код, который всё равно медленный из‑за архитектурных проблем.
</details>

#### 4.4. Глобальные лукапы в циклах

<details>
<summary>4.4.1. Почему доступ к локальной переменной в Python быстрее, чем к глобальной или атрибуту?</summary>

Потому что:

- локальные переменные хранятся в структуре, которая оптимизирована под быстрый доступ по индексу;
- глобальные/атрибуты требуют дополнительных шагов поиска (по словарям пространств имён, по MRO и т.п.).
</details>

<details>
<summary>4.4.2. Приведите пример, как уменьшить количество глобальных лукапов внутри горячего цикла.</summary>

```python
from math import sqrt

def process(nums):
    local_sqrt = sqrt  # кэшируем ссылку на функцию
    result = []
    for n in nums:
        result.append(local_sqrt(n))
    return result
```

Мы делаем один глобальный лукап (`sqrt`) перед циклом и работаем с локальной ссылкой.
</details>

<details>
<summary>4.4.3. Почему такие оптимизации важно применять только после более крупных улучшений?</summary>

Это **микро‑оптимизации**:

- выигрыш обычно измеряется в процентах или долях процента;
- если в коде есть N+1, неоптимальные алгоритмы, лишние копии — эффект от правки lookup‑ов затеряется;
- сначала чинят архитектуру и крупные проблемы, потом — шлифуют горячие места тонкими оптимизациями.
</details>

#### 5.1. Отсутствие и «символические» тесты

<details>
<summary>5.1.1. Чем наличие «символических» тестов (которые ничего не проверяют) опаснее, чем полное отсутствие тестов?</summary>

Потому что создаётся **ложное ощущение безопасности**:

- команда верит, что «у нас есть тесты»;
- на деле тесты не ловят реальные ошибки;
- проблемы вылезают в проде, хотя отчёты по тестам зелёные.
</details>

<details>
<summary>5.1.2. Как минимально полезный тест отличается от бессмысленного?</summary>

Минимально полезный тест:

- проверяет конкретное исход–результат поведение (`assert add(2, 3) == 5`);
- падает, если поведение меняется.

Бессмысленный тест:

- не содержит assert‑ов вообще;
- или проверяет `assert True`;
- или просто вызывает функцию, надеясь, что «если бы было плохо, упало бы исключение».
</details>

<details>
<summary>5.1.3. Какие сценарии в первую очередь нужно покрывать тестами в реальном проекте?</summary>

- Ключевые бизнес‑сценарии (основные пользовательские потоки).
- Критичные граничные случаи (пустые вводы, большие значения, нетипичные комбинации параметров).
- Участки, где уже были баги (регрессионные тесты).
</details>

#### 5.2. God test и тесты реализации

<details>
<summary>5.2.1. Почему god‑тесты затрудняют отладку и развитие тестов?</summary>

Потому что:

- в одном тесте смешано много разных сценариев;
- при падении непонятно, на каком шаге возникла проблема;
- небольшие изменения в логике ломают один большой тест вместо пары маленьких.
</details>

<details>
<summary>5.2.2. Чем тесты реализации отличаются от тестов поведения?</summary>

- Тесты реализации завязаны на внутренние детали: структуру классов, приватные методы, конкретные алгоритмы.
- Тесты поведения проверяют **внешний результат** и контракт API, не полагаясь на внутреннее устройство.

Тесты реализации ломаются при рефакторинге даже тогда, когда поведение не меняется.
</details>

<details>
<summary>5.2.3. Как переписать god‑тест на набор маленьких, поддерживаемых тестов?</summary>

Нужно:

- выделить отдельные сценарии, которые сейчас проверяются в одном тесте;
- для каждого сценария написать отдельный тест с понятным названием;
- по возможности использовать общие фикстуры/подготовку данных, чтобы не дублировать setup.
</details>

#### 5.3. Flaky‑тесты

<details>
<summary>5.3.1. Какие типичные источники нестабильности (flakiness) в тестах?</summary>

- Зависимость от времени (жёсткие `sleep`, точное сравнение времени).
- Зависимость от порядка запуска тестов (общие глобальные состояния).
- Использование реальных внешних сервисов/сетей без изоляции.
- Использование случайности (`random`) без фиксированного seed.
</details>

<details>
<summary>5.3.2. Почему flaky‑тесты особенно вредны для команды разработки?</summary>

Потому что:

- разработчики перестают доверять результатам тестов;
- «красный билд» может означать случайный сбой, а не реальную проблему;
- на поиск причин уходит много времени, а мотивация чинить реальные баги падает.
</details>

<details>
<summary>5.3.3. Назовите несколько приёмов, которые помогают сделать тесты стабильными.</summary>

- Вместо `sleep` использовать явные условия ожидания (ожидать событие/состояние).
- Мокать внешние сервисы (HTTP, БД), не ходить в реальные в тестах.
- Фиксировать seed для генераторов случайных чисел.
- Изолировать состояние между тестами (очищать БД/кэш, не делиться глобальными переменными).
</details>
