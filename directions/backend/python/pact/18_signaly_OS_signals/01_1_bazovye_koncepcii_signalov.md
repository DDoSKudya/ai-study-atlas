### 1.1. Что такое сигнал и как он доставляется (99)

#### Цель раздела

Понять **интуитивную модель сигналов**: что такое сигнал, кто его посылает и кому, что означает `signum`, как сигнал связан с процессом и почему он приходит «вне очереди» по отношению к обычному коду.  
Мы будем объяснять **очень медленно и по шагам**, с примерами и аналогиями, чтобы можно было понять даже при нулевом опыте с Unix.

#### Термины

| Термин | Определение простыми словами |
|--------|------------------------------|
| **Процесс** | Запущенная программа. У каждого процесса есть свои переменные, память и время на CPU. Можно думать так: вы запустили `python script.py` — появился **отдельный «мир»**, в котором крутится этот скрипт. |
| **PID (Process ID)** | Номер процесса (целое число), который выдаёт ОС. Как номер заказа или талончик в очереди: по PID ОС понимает, **к какому именно процессу обращаться**. |
| **Сигнал (signal)** | Короткое сообщение от ОС или другого процесса: «С тобой произошло X». Это не длинный текст, а просто **код события** (например: «пользователь нажал Ctrl+C»). |
| **`signum`** | Числовой код сигнала (`int`). Например, на большинстве Linux: `2` — это `SIGINT` (прерывание с клавиатуры). В Python вы почти всегда используете **имена сигналов** (`signal.SIGINT`), а не сами числа. |
| **Обработчик сигнала (signal handler)** | Функция, которая должна выполняться, когда приходит определённый сигнал. Это как «инструкция», что делать при звонке телефона: сбросить, взять трубку, записать в лог и т.п. |
| **Действие по умолчанию** | Что делает ОС, если обработчик **не задан**. Например, для многих сигналов действие по умолчанию — просто **убить процесс**. |

#### Правила и базовая модель

На уровне POSIX модель примерно такая (опускаем детали, только суть):

1. **У процесса есть «табличка сигналов».**  
   Для каждого типа сигнала (например, `SIGINT`, `SIGTERM`) там записано:  
   «что делать, если такой сигнал придёт» — убить процесс, проигнорировать или вызвать вашу функцию‑обработчик.
2. **Сигнал не исполняется мгновенно, он сначала «приходит в процесс».**  
   Когда кто‑то посылает сигнал (пользователь, другая программа, ядро ОС), система просто говорит:  
   «У процесса с PID=1234 появился сигнал типа `SIGTERM`».  
   Можно думать, что у процесса есть **флажок** «SIGTERM пришёл».
3. **Доставка**.  
   Когда процесс возвращается из системного вызова или получает управление, ОС смотрит:  
   «Есть ли у этого процесса непрочитанные сигналы?» Если есть — ОС:
   - либо сразу применяет действие по умолчанию (убить процесс, игнорировать и т.п.),
   - либо вызывает ваш обработчик (если вы его зарегистрировали).

##### Та же схема ещё раз, но как «история»

Представьте:

1. У вас есть разработчик Петя (процесс). У него есть табличка:  
   - «Если звонит мама — взять трубку».  
   - «Если звонит спам‑номер — сбросить».  
   - «Если звонит начальник — встать и подойти к начальнику».
2. Сейчас Петя пишет код (выполняет обычные инструкции).  
3. Вдруг звонит телефон — **сигнал пришёл**. Но Петя может быть:
   - в туалете;
   - в кухне за кофе;
   - глубоко в мыслях.  
   С точки зрения ОС — Петя занят в системном вызове.
4. Как только Петя «возвращается к столу» (выполнил системный вызов, вернулся в пользовательский код),  
   он видит мигающий индикатор: «Звонил начальник» → смотрит в табличку: **что делать при таком сигнале**.

Это и есть идея: **сигнал фиксируется, а потом доставляется, когда есть возможность**.

##### Как это выглядит внутри Python (упрощённо)

Python добавляет над этим свою прослойку:

- в низкоуровневом обработчике C (который привязан к сигналу на уровне ОС) Python **не запускает ваш Python‑код**;
- он лишь отмечает: «пришёл сигнал `N`» — выставляет специальный флаг и сохраняет номер сигнала;
- при выполнении байткодов интерпретатор периодически делает:  
  «Флаг сигналов поднят? Если да — исполнить соответствующий Python‑обработчик».

Поэтому:

- ваш обработчик **не врывается посреди любой C‑функции**, а запускается между нормальными шагами интерпретатора;
- он выполняется в условиях обычного Python‑кода: есть GIL, объекты в целостном состоянии.

Но с точки зрения вашей программы событие всё равно выглядит **асинхронно**:  
вы не знаете, на каком именно шаге интерпретатор вскроет флаг и запустит обработчик.

#### Пример: как выглядит сигнал в Python

Простейший пример реакции на сигнал `SIGINT` (обычно `Ctrl+C`).

Сначала **читаем код, не запуская**, и проговариваем, что он делает шаг за шагом:

```python
import signal
import time

def handler(signum, frame):
    print(f"Пойман сигнал {signum}, аккуратно завершаемся...")
    raise SystemExit(0)  # или можно просто выставить флаг и выйти из цикла

signal.signal(signal.SIGINT, handler)

print("Нажмите Ctrl+C для завершения")
while True:
    time.sleep(1)
    print("Работаю...")
```

Разбираем:

1. `import signal`, `import time` — подключаем два модуля:
   - `signal` — работа с сигналами;
   - `time` — для `sleep`, чтобы имитировать «работу».
2. Определяем функцию `handler(signum, frame)`:
   - `signum` — номер сигнала (например, 2 для `SIGINT`);
   - `frame` — объект кадров стека (нам сейчас не важен, его можно игнорировать; нужен для продвинутой отладки, чтобы понять, *где именно* в коде мы были).
   - внутри обработчика мы **печатаем сообщение** и вызываем `SystemExit(0)` — это стандартный способ аккуратно завершить программу.
3. `signal.signal(signal.SIGINT, handler)`:
   - говорим Python: «если придёт сигнал `SIGINT` (обычно `Ctrl+C`), вызывай именно эту функцию `handler`»;
   - до вызова этой строки при `Ctrl+C` Python просто кидал бы `KeyboardInterrupt` с поведением по умолчанию.
4. Потом идёт бесконечный цикл:
   - спим 1 секунду;
   - печатаем `"Работаю..."`;
   - снова спим и т.д.

Что происходит при `Ctrl+C`:

1. Вы жмёте `Ctrl+C` в терминале.
2. Терминал посылает вашему процессу `SIGINT`.
3. ОС ставит пометку: «процессу `PID` пришёл SIGINT».
4. Python, увидев этот сигнал, поднимает внутренний флаг.
5. Между двумя шагами цикла интерпретатор замечает: «флаг сигналов поднят, есть обработчик для SIGINT».
6. Вызывает `handler(signum=signal.SIGINT.value, frame=<текущий кадр>)`.
7. Внутри `handler` выполняется `raise SystemExit(0)`, и программа аккуратно завершается.

#### Граничные случаи и типичные ошибки

- **Путаница сигналов и исключений.** Исключения — механизм **внутри программы**; сигналы — механизм **между ОС и программой** (или между процессами). В Python `KeyboardInterrupt` — это исключение, которое возникает в ответ на `SIGINT`, но это уже обёртка.
- **Ожидание жёсткой реального‑времени семантики.** Сигнал не приходит «моментально». ОС доставляет его, когда процесс будет в состоянии его принять (выйдет из определённых системных вызовов и т.п.).

Ещё несколько моментов, которые часто путают:

- **«Если я послал сигнал, код *обязан* отреагировать прямо сейчас» — нет.**  
  Реакция может произойти через долю секунды (или чуть больше), в зависимости от того, где сейчас застрял процесс (в системном вызове, в блокирующем вводе‑выводе и т.п.).
- **«Сигнал — это как обычная функция, которая просто вызывается» — нет.**  
  Функция‑обработчик вызывается **по особым правилам**, которые завязаны на работу ОС и интерпретатора. Поэтому мы будем осторожны с тем, что внутри неё делать.

#### Запомните

- Сигнал — это **внешнее асинхронное событие**, а не «ещё одно исключение».
- Каждый тип сигнала имеет **номер и имя** (`signal.SIGINT` и т.п.).
- Без вашего вмешательства работает **действие по умолчанию** (чаще всего — завершение процесса).

### 1.2. Основные сигналы, которые нужно знать (99)

#### Цель раздела

Разобраться, какие сигналы вы будете видеть чаще всего в практической работе и какое у них поведение по умолчанию.  
Мы разберём их **по одному**, с короткими историями «из жизни процесса», чтобы ассоциация закрепилась.

#### Термины

| Сигнал | Суть простыми словами |
|--------|------------------------|
| **`SIGINT`** | «Прерывание с клавиатуры». Обычно возникает, когда пользователь нажимает `Ctrl+C` в терминале. По умолчанию говорит: «Прекрати работу прямо сейчас» → процесс завершается. |
| **`SIGTERM`** | «Вежливое завершение». Стандартный способ **попросить процесс аккуратно завершиться**: закрыть файлы, остановить воркеры и т.п. По умолчанию тоже завершает процесс. |
| **`SIGKILL`** | «Немедленное убийство». Процесс *вообще* не может ничего предпринять: ОС просто выкидывает его из памяти. Никаких обработчиков, никакой очистки. |
| **`SIGSTOP`** | «Поставить на паузу». Процесс приостанавливается (как `Ctrl+Z` в терминале). Тоже не перехватывается. |
| **`SIGUSR1`, `SIGUSR2`** | «Пользовательские» сигналы. У них нет фиксированного значения в стандарте: вы сами решаете, что они значат в вашем приложении (например, «сделать дамп статистики», «перечитать конфиг частично» и т.п.). |
| **`SIGHUP`** | Исторически — «отвалился терминал» (hang up). Сейчас часто используется как сигнал «перечитай конфигурацию», особенно в демонах. |
| **`SIGALRM`** | Сигнал от таймера, который вы ставите через `alarm()`. Грубо: «прошло N секунд, пора проснуться/прервать операцию». |

#### Образные объяснения по каждому сигналу

Представьте, что процесс — это тот же Петя‑разработчик.

- **`SIGINT`** — вы сами подходите к Пете и нажимаете у него на клавиатуре `Ctrl+C`:
  - Петя: «А, мне сказали остановиться. Сейчас кину исключение `KeyboardInterrupt` или вызову обработчик и завершу работу».
  - Используется, когда **пользователь сам решил** остановить программу.
- **`SIGTERM`** — начальник Пети подходит и спокойно говорит:  
  «Петя, закончи текущие дела, сохрани файлы и иди домой».
  - Петя: «Ок, я сейчас допишу строку, закрою всё аккуратно и только потом уйду».
  - Используется системами вроде `systemd`, `supervisord`, Kubernetes, когда нужно **аккуратно перезапустить** сервис.
- **`SIGKILL`** — начальник не выдержал, подошёл и **выдернул вилку** из розетки у Пети:
  - Петя не успевает ничего: ни сохранить работу, ни закрыть файлы.
  - Это **крайняя мера**, когда процесс не реагирует ни на какие вежливые просьбы (`SIGTERM`) и приходится его насильно убить.
- **`SIGSTOP`** — начальник говорит: «Замри!» и ставит Пети **жёсткую паузу**:
  - Петя перестаёт печатать, но остаётся в том же состоянии (все файлы открыты, ничего не изменено).
  - Позже начальник может дать `SIGCONT` («продолжай») — и Петя просто продолжит с того же места.
- **`SIGUSR1` / `SIGUSR2`** — это как если начальник и Петя заранее договорились:  
  - «Если я подмигиваю один раз (SIGUSR1) — сделай дамп статистики в лог».  
  - «Если два раза (SIGUSR2) — перечитай конфиг и примени новые настройки».
- **`SIGHUP`** — начальник уходит из офиса и выключает свет/закрывает терминал:
  - В старые времена это означало: «терминал отвалился, можно завершать».
  - Современные демоны часто трактуют `SIGHUP` как команду: «перечитай конфигурацию, но не умирай».
- **`SIGALRM`** — у Пети на столе таймер‑яйцо. Он ставит его сам:
  - «Если через 5 минут я не закончу задачу, позвони мне (SIGALRM), чтобы я не засиделся».
  - Так реализуют **таймауты**: операция должна завершиться за N секунд, иначе мы её прерываем.

#### Правила и поведение по умолчанию

На Unix обычно:

- `SIGINT` → завершение процесса (в Python → `KeyboardInterrupt`, если обработчик по умолчанию);
- `SIGTERM` → завершение процесса (используется `kill`, `systemd`, `supervisord` и т.п.);
- `SIGKILL` → немедленное завершение, не даёт выполнить ни одного байта пользовательского кода;
- `SIGHUP` → завершение или «перезапуск» демона (часто перехватывают и используют как сигнал «перечитай конфиг»).

**Ключевой момент:** `SIGKILL` и `SIGSTOP` **нельзя перехватить или игнорировать** — это гарантия ОС, чтобы администратор мог всегда остановить процесс.

Это значит:

- как бы вы ни старались, вы **не можете** написать обработчик `SIGKILL` или `SIGSTOP` в своём коде — ядро даже не даст вам такого шанса;
- если процесс завис и не реагирует на `SIGTERM`, у системного администратора всегда есть «красная кнопка» (`kill -9 PID`), которая сработает.

#### Пример: логирование сигналов

```python
import os
import signal
import time

def log_signal(signum, frame):
    print(f"[PID={os.getpid()}] Пойман сигнал: {signal.Signals(signum).name}")

for sig in (signal.SIGINT, signal.SIGTERM, signal.SIGHUP, signal.SIGUSR1, signal.SIGUSR2):
    signal.signal(sig, log_signal)

print(f"PID процесса: {os.getpid()}")
print("Посылайте сигналы через `kill -SIGUSR1 PID` и т.п.")

while True:
    time.sleep(5)
    print("Жду сигналы...")
```

Разбор этого примера по шагам:

1. Импортируем модули:
   - `os` — чтобы узнать PID процесса;
   - `signal` — чтобы работать с сигналами;
   - `time` — чтобы сделать паузу между сообщениями.
2. Определяем обработчик `log_signal(signum, frame)`:
   - при любом из выбранных сигналов он просто печатает строку с PID и именем сигнала;
   - `signal.Signals(signum).name` — превращает номер сигнала обратно в удобное имя (`SIGINT`, `SIGTERM` и т.д.).
3. В цикле `for sig in (...)` мы:
   - перебираем несколько сигналов (`SIGINT`, `SIGTERM`, `SIGHUP`, `SIGUSR1`, `SIGUSR2`);
   - для каждого вызываем `signal.signal(sig, log_signal)` — говорим Python, что **при этом сигнале нужно вызвать `log_signal`**.
4. Печатаем свой `PID`, чтобы вы могли из другого окна терминала выполнить, например:  
   `kill -SIGUSR1 <этот_PID>`.
5. В бесконечном цикле:
   - спим 5 секунд;
   - печатаем «Жду сигналы...».

Что произойдёт, если:

- вы нажмёте `Ctrl+C`:
  - придёт `SIGINT`, сработает `log_signal`, вы увидите сообщение;
  - если не менять поведение дальше, программа затем завершится (если вы не переопределили `SIGINT` полностью);
- вы выполните `kill -SIGUSR1 PID`:
  - придёт `SIGUSR1`, сработает `log_signal`, но программа продолжит жить (если вы в обработчике не завершаете её самостоятельно).

#### Запомните

- На практике чаще всего вы увидите **`SIGINT`, `SIGTERM`, реже `SIGHUP`, `SIGUSR1/2` и `SIGALRM`**.
- `SIGKILL` и `SIGSTOP` невозможно обработать — они нужны как «последнее средство».

### 1.3. Отличия Unix и Windows (99)

#### Цель раздела

Понимать, почему код работы с сигналами, написанный под Linux, **может не работать** или работать иначе под Windows.  
Здесь важно не заучить все детали Windows‑API, а запомнить простое правило: **большинство материалов про сигналы подразумевают Unix‑мир**.

#### Ключевые моменты

- Модель сигналов — **POSIX‑концепция**; Windows её поддерживает **частично**.
- Под Windows:
  - нет большинства Unix‑сигналов (`SIGKILL`, `SIGUSR1`, `SIGALRM` и т.п.);
  - `os.kill` реализован иначе (чаще всего → `TerminateProcess`, а не «настоящий» сигнал).
- Модуль `signal` на Windows поддерживает только небольшой поднабор (например, `SIGINT`, `SIGBREAK`, `SIGABRT` и т.п.), и поведение может отличаться.

Если упростить до уровня «инструкция для мозга»:

- **Linux / macOS / BSD** — это мир, где сигналы — «родные граждане». Там всё, что мы обсуждаем в этой части, действительно работает (за редкими оговорками).
- **Windows** — это мир, где сигналы — «гости». Они как бы есть, но сильно ограничены, и часть возможностей просто не доступна.

#### Практический вывод

Если вам нужна **переносимость** между Unix и Windows:

- избегайте завязки на конкретные Unix‑сигналы (`SIGUSR1`, `SIGALRM`) в кроссплатформенном коде;
- используйте **абстракции более высокого уровня**: события (`threading.Event`), очереди, IPC (`multiprocessing.Pipe/Queue`), а сигналы — только там, где вы уверены, что платформа Unix‑подобная.

Ещё один способ запомнить:

- если вы пишете **инфраструктурный код** (демоны, сервисы, воркеры) и ясно, что он будет работать **на Linux‑сервере** — сигналы можно (и нужно) использовать активно;
- если вы пишете **десктопное кроссплатформенное приложение** для конечных пользователей (Windows + Linux + macOS), то:
  - сигналы лучше оставить для Linux‑части,
  - а общую логику управления жизненным циклом строить на других механизмах.

## 2. Модуль `signal` (§100)
