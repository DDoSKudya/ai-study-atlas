### 6.1. Сценарии «задача → решение»

| Задача | Решение (ключевые шаги) |
|--------|-------------------------|
| **Аккуратно завершить CLI‑утилиту по `Ctrl+C`** | Поставить обработчик `SIGINT` через `signal.signal`, внутри — выставить флаг `stop`, в основном цикле проверять `stop` и при `True` аккуратно завершать (закрыть файлы, соединения). |
| **Организовать таймаут для потенциально зависающей операции** | На Unix: обработчик `SIGALRM` + `signal.alarm(N)` перед вызовом; в обработчике — поднять своё исключение и перехватить его снаружи. Или использовать более высокоуровневые механизмы (`asyncio.wait_for`, процессы). |
| **Игнорировать `SIGINT` в критическом участке** | Сохранить старый обработчик `old = signal.signal(SIGINT, SIG_IGN)`, выполнить критический участок, затем восстановить `signal.signal(SIGINT, old)`. |
| **Отправить `SIGTERM` другому процессу по PID** | `os.kill(pid, signal.SIGTERM)`. Подождать завершения (через `waitpid` или API фреймворка). Если не завершился — по необходимости `SIGKILL`. |
| **Грейсфул‑шатдаун асинхронного сервиса на `asyncio`** | В `main` получить loop, зарегистрировать `loop.add_signal_handler(SIGINT/SIGTERM, callback)`, в `callback` устанавливать `asyncio.Event`, корутины в основном цикле ждут/проверяют событие и завершаются. |
| **Синхронизировать завершение `multiprocessing`‑воркеров по сигналу** | В родителе зарегистрировать обработчики сигналов; внутри обработчика выставить `multiprocessing.Event`; воркеры периодически проверяют этот Event и завершаются. |

### 6.2. Терминология по части

Ниже — словарь терминов **в формате “что это → как представить → где встретишь”**.

| Термин | Что это (простыми словами) | Как представить в голове | Где встретишь в жизни |
|-------|-----------------------------|--------------------------|------------------------|
| **Сигнал (signal)** | Короткое внешнее событие для процесса: «остановись», «заверши работу», «таймер истёк». | «ОС сказала процессу одну команду» | `Ctrl+C`, остановка сервиса `systemd`, `kill PID` |
| **`signum`** | Числовой код сигнала. | «ID команды» | В обработчике `handler(signum, frame)` |
| **Обработчик сигнала (handler)** | Функция, которую вызывает Python при сигнале. | «Инструкция: что делать, если пришёл сигнал» | `signal.signal(SIGTERM, handler)` |
| **Действие по умолчанию (`SIG_DFL`)** | Если обработчика нет — ОС/интерпретатор применит стандартное поведение (часто завершение процесса). | «Если не настроил — будет как принято» | Процесс завершается от `SIGTERM` |
| **Игнорирование (`SIG_IGN`)** | Сигнал пришёл, но процесс делает вид, что ничего не было. | «Не слышу» | Временное отключение реакции на `SIGINT` |
| **`SIGINT`** | Сигнал от терминала при `Ctrl+C`. | «Пользователь остановил руками» | Скрипты, CLI |
| **`SIGTERM`** | Вежливое завершение процесса. | «Заканчивай и закрывайся» | `systemctl stop`, Kubernetes termination |
| **`SIGKILL`** | Жёсткое убийство, не перехватывается. | «Выдернули вилку» | `kill -9 PID` |
| **`SIGHUP`** | Исторически: терминал отвалился; часто: “перечитай конфиг”. | «Обнови настройки без перезапуска» | Демоны, конфиги |
| **`SIGUSR1/2`** | Пользовательские сигналы: значение вы задаёте сами. | «Договорной знак» | “дамп/статистика”, “перечитай часть” |
| **`SIGALRM`** | Сигнал таймера `alarm`. | «Будильник» | Таймауты в Unix‑скриптах |
| **Race condition** | Итог зависит от случайного порядка событий/потоков. | «Иногда работает, иногда ломается» | Сложная логика в обработчике |
| **Graceful shutdown** | Мягкое завершение: закрыть ресурсы, доработать очередь, сохранить состояние. | «Уйти домой, закрыв дверь и свет» | Сервисы, воркеры |
| **Process group** | Группа процессов, которой можно послать сигнал всем разом. | «Остановить сразу всю команду» | `os.killpg`, дерево `subprocess` |

Мини‑проверка, чтобы закрепить:

- Если вы хотите **вежливо попросить** процесс завершиться — это обычно `SIGTERM`.
- Если вы хотите **прибить насильно** — это `SIGKILL` (`kill -9`).
- Если пользователь жмёт `Ctrl+C` — обычно это `SIGINT`.

### 6.3. Вопросы для самопроверки

1. Чем сигнал **отличается** от исключения в Python?

<details><summary>Ответ</summary>
Исключение — механизм **внутри программы**, запускаемый самим кодом (через `raise`) или средой выполнения (например, `ZeroDivisionError`), и обрабатывается конструкциями `try/except`.  
Сигнал — это **внешнее асинхронное событие**, приходящее от ОС (или другого процесса) к процессу; в Python оно может быть отображено в исключение (`KeyboardInterrupt`), но само по себе живёт вне модели исключений.
</details>

2. Почему `SIGKILL` и `SIGSTOP` нельзя перехватить или проигнорировать?

<details><summary>Ответ</summary>
Потому что ОС должна иметь **гарантированный способ остановить или приостановить любой процесс**, даже если он ведёт себя неправильно или вредоносно. Эти сигналы реализованы на уровне ядра так, что пользовательский код просто не получает шанса вмешаться.
</details>

3. Почему в обработчике сигнала в Python **не рекомендуется** выполнять долгие операции (сетевые запросы, большие вычисления)?

<details><summary>Ответ</summary>
Потому что обработчик выполняется **в главном потоке** поверх основной логики. Длительная операция в обработчике:
- блокирует реакцию на другие сигналы;
- может привести к сложным для отладки состояниям (нестандартный порядок вызовов, неожиданные блокировки).  
Безопаснее выставить флаг и выполнить тяжёлую работу в контролируемом месте основного кода.
</details>

4. Почему код, использующий `loop.add_signal_handler`, не будет работать под Windows?

<details><summary>Ответ</summary>
Потому что `loop.add_signal_handler` реализован только для **Unix‑подобных систем**, где модель сигналов POSIX‑совместима. На Windows многие сигналы отсутствуют или реализованы иначе, и `asyncio` не поддерживает этот API.
</details>

5. Чем отличается `Popen.terminate()` от `Popen.kill()` на Unix?

<details><summary>Ответ</summary>
Обычно `terminate()` отправляет дочернему процессу `SIGTERM` (вежливое завершение), позволяя ему выполнить очистку.  
`kill()` отправляет `SIGKILL`, который немедленно убивает процесс без шанса на обработку.
</details>

6. Почему в CPython обработчик сигнала выполняется в **главном потоке**, а не “в том, куда прилетело”?

<details><summary>Ответ</summary>
Это сделано ради предсказуемости и безопасности. Если бы обработчик мог выполняться в произвольном рабочем потоке, который держит локи/ресурсы, стало бы легко получить deadlock или поломанное состояние. CPython упрощает модель: сигнал фиксируется, а Python‑обработчик исполняется в главном потоке при проверке флага сигналов.
</details>

7. Почему в обработчике сигнала лучше ставить флаг (`Event`), а “cleanup” делать в основном коде?

<details><summary>Ответ</summary>
Потому что обработчик может прийти в неожиданный момент. Если внутри обработчика делать много действий (закрывать сокеты, писать файлы, трогать сложные структуры), повышается риск гонок и зависаний. Флаговый паттерн делает обработчик быстрым и предсказуемым, а cleanup переносит в контролируемую точку кода.
</details>

8. Что будет, если вы забудете сделать `signal.alarm(0)` после удачной операции?

<details><summary>Ответ</summary>
Таймер останется активным, и `SIGALRM` может прилететь позже “внезапно”, уже во время другой логики, и прервать не ту операцию. Поэтому `alarm(0)` в `finally` — важная дисциплина.
</details>

9. Зачем в `subprocess` важно вызывать `wait()` после `terminate()`/`kill()`?

<details><summary>Ответ</summary>
Чтобы родитель “собрал” статус завершения дочернего процесса. Если этого не сделать на Unix, дочерний может стать зомби: процесс завершился, но запись о нём ещё висит, пока родитель не прочитает статус.
</details>

10. Почему иногда недостаточно `proc.terminate()`, и нужно работать с **группой процессов**?

<details><summary>Ответ</summary>
Потому что дочерняя программа может породить другие процессы. Если послать сигнал только одному PID, его “внуки” могут остаться жить. Запуск в новой сессии/группе и `os.killpg` позволяют остановить весь набор процессов.
</details>

11. Чем отличается “мягко завершить” от “жёстко завершить” в терминах сигналов?

<details><summary>Ответ</summary>
Мягко — это обычно `SIGTERM`/`SIGINT`: процесс может выполнить cleanup и корректно выйти.  
Жёстко — `SIGKILL`: процесс не может выполнить никакой код, ОС завершает его немедленно.
</details>

12. Если ваш код “залип” в длинной операции без точек проверки, почему `stop_event.set()` может не помочь быстро?

<details><summary>Ответ</summary>
Потому что флаг проверяется только там, где ваш код до него доходит. Если поток/процесс завис в блокирующем вызове без таймаута или в бесконечном вычислении без проверки, он может долго не вернуться к месту, где вы смотрите `is_set()`. Тогда нужен таймаут/другая архитектура/жёсткое завершение.
</details>

### 6.4. Типичные ошибки и анти‑паттерны

1. **Сложная логика внутри обработчика сигнала.**
   - Плохо: выполнять там длительные операции, сетевые запросы, сложные манипуляции с состоянием.
   - Лучше: выставить флаг/событие и обрабатывать его в основном цикле.

2. **Расчёт на переносимость `SIGUSR1`/`SIGUSR2` и `SIGALRM` на Windows.**
   - На Windows этих сигналов нет или они работают иначе.
   - Для кроссплатформенного кода лучше использовать другие механизмы IPC и таймаутов.

3. **Неожиданные зомби‑процессы.**
   - Отправить сигнал дочернему процессу, но не вызвать `wait()`/`waitpid()` → дочерний переходит в состояние зомби.
   - Всегда дожидайтесь завершения дочерних процессов.

4. **Отсутствие graceful shutdown.**
   - Программа просто завершается по `SIGTERM` без очистки ресурсов (файлы, соединения, временные файлы).
   - Правильный подход — ловить сигнал, аккуратно завершать, логировать.

5. **Ставить обработчик и “забыть восстановить” (ломать чужое приложение).**
   - Плохо: библиотека ставит `signal.signal(SIGINT, ...)` и оставляет это навсегда.
   - Лучше: сохранять старый обработчик и восстанавливать его (или документировать поведение).

6. **Путать `SIGTERM` и `SIGKILL`.**
   - Плохо: всегда использовать `kill -9` “на всякий случай”.
   - Лучше: сначала `SIGTERM` (дать шанс корректно завершиться), и только потом `SIGKILL`, если процесс не реагирует.

7. **Делать много логики/ввода‑вывода в обработчике.**
   - Плохо: долго печатать, ходить в сеть, долго “чиститься” прямо в handler.
   - Лучше: в handler поставить флаг, а дальше cleanup в обычном коде.

8. **Не думать про “дерево процессов” в `subprocess`.**
   - Плохо: убить только `proc.pid`, а “внуки” останутся жить.
   - Лучше: при необходимости использовать группы процессов (`start_new_session=True` + `os.killpg`).

#### Мини‑чеклист “я сделал правильно?”

- [ ] Обработчик сигнала делает **минимум** (ставит `Event`/флаг).
- [ ] Cleanup выполняется в **обычном коде** (после выхода из цикла / перед `return`).
- [ ] Для дочерних процессов я делаю `wait()`/`join()`.
- [ ] Если есть риск “дерева процессов” — я думаю про **группы процессов**.
- [ ] Я понимаю, что Windows может не поддерживать нужные сигналы.

### 6.5. Мини‑практикум (упражнения)

Короткие задания, чтобы **закрепить руками**. Если не хотите запускать — хотя бы мысленно проговорите: «что произойдёт при Ctrl+C?».

#### Упражнение 1. Мягкое завершение по `Ctrl+C`

**Задача:** цикл печатает число каждую секунду и завершается по `Ctrl+C` аккуратно, печатая в конце `cleanup done`.

<details><summary>Возможное решение</summary>

```python
import signal
import threading
import time

stop_event = threading.Event()

def handle_sigint(signum, frame):
    stop_event.set()

signal.signal(signal.SIGINT, handle_sigint)

i = 0
while not stop_event.is_set():
    print("i =", i)
    i += 1
    time.sleep(1)

print("cleanup done")
```

</details>

#### Упражнение 2. “Первый Ctrl+C — мягко, второй — жёстко”

**Задача:** первый `Ctrl+C` ставит флаг остановки, второй — завершает немедленно (`SystemExit`).

Подсказка: счётчик сигналов.

#### Упражнение 3. Таймаут через `SIGALRM` (Unix)

**Задача:** оберните `time.sleep(10)` таймаутом 2 секунды.

Подсказка: `signal.signal(SIGALRM, ...)`, `alarm(2)`, `alarm(0)` в `finally`.

#### Упражнение 4. `subprocess`: terminate → kill → wait

**Задача:** запустите процесс, который “спит” 60 секунд, и остановите его по схеме:

- `terminate()`
- `wait(timeout=...)`
- если не вышел → `kill()` и `wait()`

#### Упражнение 5. `multiprocessing`: мягко → terminate

**Задача:** 2 воркера работают в цикле. По `Ctrl+C`:

- родитель ставит `mp.Event`;
- воркеры выходят;
- если кто‑то не вышел за 5 секунд — родитель делает `terminate()`.
