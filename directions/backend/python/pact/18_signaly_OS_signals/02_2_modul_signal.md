### 2.1. Базовое использование `signal.signal` (100)

#### Цель раздела

Научиться **устанавливать и снимать обработчики сигналов** в Python, понимать сигнатуру обработчика и значения `SIG_DFL` и `SIG_IGN`.  
Мы будем рассматривать `signal.signal` как «редактор таблички сигналов» внутри процесса.

#### Термины

| Термин | Определение |
|--------|-------------|
| **`signal.signal(signum, handler)`** | Функция, которая **записывает в табличку сигналов**: «для сигнала `signum` теперь нужно делать `handler`». Возвращает старое значение (что было до этого). |
| **`handler(signum, frame)`** | Функция‑обработчик: всегда принимает два аргумента — номер сигнала и объект фрейма (где мы были в коде). Даже если `frame` вам не нужен, он всё равно будет передан. |
| **`signal.SIG_DFL`** | Специальное значение: «вернуть действие по умолчанию». Не функция. |
| **`signal.SIG_IGN`** | Специальное значение: «игнорировать этот сигнал» (ничего не делать). |
| **`signal.getsignal(signum)`** | Возвращает, что сейчас записано в табличке для сигнала `signum`: функция, `SIG_DFL` или `SIG_IGN`. |

Можно думать, что у процесса есть словарь (это образ, не реальный объект):

```python
signal_table = {
    signal.SIGINT: SIG_DFL,
    signal.SIGTERM: SIG_DFL,
    # ...
}
```

Вызов `signal.signal(signal.SIGINT, handler)` делает примерно так:

```python
old = signal_table[signal.SIGINT]
signal_table[signal.SIGINT] = handler
return old
```

В реальности там больше логики и проверок, но для понимания достаточно такой ментальной модели.

#### Примеры (очень подробно, по шагам)

**1. Перехват `Ctrl+C` с аккуратным завершением**

```python
import signal
import time

stop = False

def handle_sigint(signum, frame):
    global stop
    print("Получен SIGINT, установим флаг остановки.")
    stop = True

signal.signal(signal.SIGINT, handle_sigint)

print("Нажмите Ctrl+C для аккуратного завершения")
while not stop:
    print("Работаем...")
    time.sleep(1)

print("Остановлено по сигналу.")
```

Разбор:

1. `stop = False` — глобальная переменная‑флаг: пока он `False`, программа считает, что можно продолжать работу.
2. В `handle_sigint` мы:
   - объявляем `global stop`, чтобы изменять именно глобальную переменную, а не создавать локальную;
   - печатаем сообщение, чтобы было видно, что сигнал реально пришёл;
   - ставим `stop = True` — тем самым просим основной цикл «остановиться при следующей проверке».
3. `signal.signal(signal.SIGINT, handle_sigint)`:
   - говорит: «при `Ctrl+C` (часто это `SIGINT`) вызывай `handle_sigint` вместо поведения по умолчанию».
4. В основном цикле:
   - `while not stop:` — пока флаг не установлен, работаем;
   - внутри цикла пишем «Работаем...» и спим 1 секунду.
5. Когда вы нажмёте `Ctrl+C`:
   - `handle_sigint` установит `stop = True`;
   - цикл увидит, что `not stop` → `False`, и выйдет;
   - исполнится `print("Остановлено по сигналу.")`.

Ключевая идея: **обработчик делает минимум (ставит флаг), а основная логика реагирует на флаг**.

**2. Игнорировать `SIGINT` (не выходить по Ctrl+C)**

```python
import signal
import time

signal.signal(signal.SIGINT, signal.SIG_IGN)

print("SIGINT игнорируется, Ctrl+C не завершит процесс (попробуйте).")
time.sleep(10)
print("Готово.")
```

Разбор:

1. `signal.signal(signal.SIGINT, signal.SIG_IGN)`:
   - вместо функции‑обработчика мы передаём специальную константу `SIG_IGN`;
   - это означает: «при приходе `SIGINT` **ничего не делать**».
2. В результате нажатие `Ctrl+C` **не прерывает** `time.sleep(10)` и не завершает процесс.

Это полезно, когда:

- у вас есть критический участок, который **нельзя оборвать** посередине (например, вы пишете важный файл на диск);
- вы временно отключаете реакцию на `Ctrl+C`, а потом восстанавливаете её (см. следующий пример).

**3. Временно переопределить обработчик**

```python
import signal
import time

def handler(signum, frame):
    print("Временный обработчик SIGINT")

old = signal.signal(signal.SIGINT, handler)
try:
    time.sleep(5)
finally:
    signal.signal(signal.SIGINT, old)  # восстановить старый обработчик
```

Идея:

- иногда нужно **временно** изменить поведение на сигнал (`SIGINT`), а затем вернуть то, что было раньше;
- `signal.signal` возвращает **старый обработчик** — мы его сохраняем в `old`;
- в `finally` мы гарантированно (даже при ошибках) возвращаем прежнее поведение.

Это особенно важно в библиотеках: вы не хотите «сломать» поведение сигналов для всего приложения, изменив его навсегда.

#### Частые вопросы по `handler(signum, frame)`

- **Вопрос:** «Мне `frame` вообще не нужен, можно сделать обработчик с одним аргументом?»  
  **Ответ:** Нет, сигнатура обработчика должна быть **ровно `(signum, frame)`**. Если вы напишете `def handler(signum): ...`, Python пожалуется.  
  Но `frame` можно просто не использовать:

  ```python
  def handler(signum, frame):
      del frame  # чтобы линтер не ругался
      print("Сигнал:", signum)
  ```

- **Вопрос:** «Откуда берётся `signum`, если я использую имена типа `signal.SIGINT`?»  
  **Ответ:** Внутри Python всё равно хранятся **числа**, но вы почти всегда работаете с именованными константами. В обработчик приходит именно число (например, `2`), но вы можете получить имя обратно: `signal.Signals(signum).name`.

#### Граничные случаи

- На Unix установить обработчик сигналов можно **только в главном потоке**; попытка сделать это из другого потока приведёт к `ValueError`.
- Некоторые сигналы **нельзя переназначить** (например, `SIGKILL`, `SIGSTOP`) — попытка приведёт к ошибке.

#### Запомните

- Обработчик всегда имеет сигнатуру `handler(signum, frame)` — даже если вы не используете `frame`.
- `signal.signal` возвращает предыдущий обработчик — удобно для временной замены.
- `SIG_DFL` и `SIG_IGN` — специальные значения, а не функции.

### 2.2. Таймеры и ожидание сигналов (100)

#### Цель раздела

Научиться использовать **таймеры** (`SIGALRM`) и функцию `signal.pause()` для блокирующего ожидания сигнала.  
Мы будем относиться к `SIGALRM` как к «таймеру‑будильнику», а к `signal.pause()` — как к кнопке «ждать, пока не позовут».

#### Термины

| Термин | Определение |
|--------|-------------|
| **`signal.alarm(seconds)`** | Попросить ОС: «через `seconds` секунд пошли мне сигнал `SIGALRM`». Если до этого был уже поставлен будильник, функция вернёт, сколько секунд оставалось до него. |
| **`signal.alarm(0)`** | Снять все ранее поставленные будильники `SIGALRM` (отменить таймер). |
| **`signal.pause()`** | Заморозить выполнение программы до тех пор, пока не придёт **любой** сигнал (не только `SIGALRM`). |

Можно думать, что:

- `signal.alarm(5)` — это как поставить таймер‑яйцо на 5 минут;
- `signal.pause()` — лечь спать и ждать, пока кто‑то не разбудит (любой сигнал).

#### Пример: простой таймаут (понятие «операция с ограничением по времени»)

```python
import signal
import time

class Timeout(Exception):
    pass

def handle_alarm(signum, frame):
    raise Timeout("Операция превысила лимит времени")

signal.signal(signal.SIGALRM, handle_alarm)

def long_operation():
    time.sleep(10)  # имитируем долгую работу, которая может «зависнуть»

signal.alarm(3)  # через 3 секунды придёт SIGALRM
try:
    long_operation()
except Timeout as e:
    print("Прервано по таймауту:", e)
finally:
    signal.alarm(0)  # обязательно отменить, если операция завершилась раньше
```

Разбор по шагам:

1. Мы создаём **своё собственное исключение** `Timeout`, чтобы по нему отличать «вышло время» от других проблем.
2. В обработчике `handle_alarm` мы:
   - вообще не думаем, чем занята сейчас программа;
   - просто поднимаем `Timeout` — «прервать всё, что сейчас делается, и сообщить, что время вышло».
3. `signal.signal(signal.SIGALRM, handle_alarm)`:
   - говорит Python: «если придёт сигнал `SIGALRM`, вызывай `handle_alarm`».
4. Функция `long_operation` имитирует работу, которая может зависнуть (например, сетевой запрос, внешний процесс, тяжёлые вычисления) — здесь это просто `time.sleep(10)`.
5. `signal.alarm(3)`:
   - ставит таймер: «через 3 секунды пришли `SIGALRM` этому процессу».
6. В блоке `try` мы вызываем `long_operation()`:
   - если она завершится **быстрее, чем за 3 секунды**, сигнал ещё не придёт;
   - если она «зависнет» дольше — сработает `SIGALRM`, вызовется `handle_alarm`, поднимется `Timeout`.
7. В `except Timeout` мы аккуратно ловим это исключение и печатаем сообщение.
8. В `finally` мы вызываем `signal.alarm(0)`:
   - это очень важный шаг: если операция **завершилась раньше, чем за 3 секунды**, таймер всё ещё поставлен;
   - `alarm(0)` гарантирует, что никакой отложенный `SIGALRM` не прилетит потом «внезапно», когда вы уже занимаетесь другой задачей.

Итого: мы обернули «любую операцию» в **защиту по времени** — если она не уложилась в N секунд, мы её прерываем.

**Комментарии и ограничения:**

- Такой подход работает **только на Unix** и только в главном потоке.
- В реальном коде часто удобнее использовать более высокоуровневые механизмы (например, `asyncio.wait_for`, потоки, процессы), но `SIGALRM` всё ещё полезен для простых скриптов.

#### Пример: ждать, пока не придёт «любой» сигнал (`signal.pause`)

```python
import signal
import os

received = False

def handler(signum, frame):
    global received
    print("Сигнал получен, продолжаем:", signum)
    received = True

signal.signal(signal.SIGUSR1, handler)

print(f"PID: {os.getpid()}, посылайте `kill -SIGUSR1 {os.getpid()}`")
print("Ждём сигнала...")
signal.pause()  # блокируемся до прихода любого сигнала

print("После signal.pause, received =", received)
```

Разберём:

1. Глобальная переменная `received = False` — по ней мы будем понимать, что сигнал действительно был обработан.
2. Обработчик `handler`:
   - печатает сообщение;
   - ставит `received = True`.
3. Мы регистрируем обработчик для `SIGUSR1`: `signal.signal(signal.SIGUSR1, handler)`.
4. Печатаем свой `PID` и сообщение «Ждём сигнала...».
5. Вызов `signal.pause()`:
   - говорит интерпретатору: «ничего не делай, пока не придёт *какой‑нибудь* сигнал»;
   - программа как бы «замирает» и ждёт внешнего события.
6. Когда вы из другого терминала выполните `kill -SIGUSR1 PID`:
   - придёт `SIGUSR1`;
   - Python вызовет `handler`;
   - `handler` установит `received = True`;
   - `signal.pause()` «просыпается», выполнение идёт дальше, и мы печатаем итог.

Таким образом, `signal.pause()` — это способ сказать: «Я сейчас ничего активного не делаю, просто жду, пока мир меня позовёт».

#### Когда `alarm` и `pause` уместны, а когда нет

**Уместно:**

- небольшие утилиты и скрипты под Unix, где:
  - нужно быстро сделать таймаут вокруг одной‑двух операций;
  - или просто подождать внешнего события (сигнал от другого процесса).

**Менее уместно:**

- большие приложения, особенно кроссплатформенные;
- сложные асинхронные сервисы — там обычно используют `asyncio`, очереди, события и другие механизмы.

#### Запомните

- `signal.alarm` — это **грубый, но простой** способ реализовать таймаут.
- `signal.pause` удобен для демонстраций и простых утилит, но в сложных приложениях лучше использовать свой цикл или event loop.

### 2.3. `signal.siginterrupt` и системные вызовы (100)

#### Цель раздела

Понять, что происходит с блокирующими системными вызовами (`read`, `select`, сетевые операции), когда приходит сигнал, и как влияет `signal.siginterrupt`.  
Здесь мы чуть‑чуть приоткроем «низкий уровень», но аккуратно и с примерами.

#### Термины

| Термин | Определение |
|--------|-------------|
| **`signal.siginterrupt(signum, flag)`** | Управляет тем, будут ли системные вызовы **прерываться** сигналом (`flag=True`) или **перезапускаться** автоматически (`flag=False`). |
| **`EINTR`** | Код ошибки «системный вызов прерван сигналом» (Interrupted system call). |

#### Суть

В классической POSIX‑модели:

- если во время блокирующего системного вызова приходит сигнал и есть обработчик, вызов может завершиться с ошибкой `EINTR`;
- приложение должно само решить — повторять вызов или нет.

Функция `siginterrupt` позволяет управлять этим поведением для конкретного сигнала:

- `signal.siginterrupt(signum, True)` → **прерывать** системные вызовы (получаете `EINTR`);
- `signal.siginterrupt(signum, False)` → **перезапускать** системные вызовы автоматически (грубо как `SA_RESTART`).

В Python многие детали за вас обёрнуты, но важно понимать, что сигнал может «выдернуть» программу из ожидания и привести, например, к `OSError: [Errno 4] Interrupted system call`.

Чтобы запомнить:

- `flag=True` — «будь чувствителен, прерывай ожидание»;
- `flag=False` — «будь терпелив, пытайся продолжать ждать».

#### Мини‑пример (упрощённый сценарий)

Большинство повседневных Python‑разработчиков напрямую `siginterrupt` не трогают, но полезно понимать типичный сценарий:

1. Есть программа, которая ждёт данные из сети через низкоуровневый `select`/`poll`.
2. В это время приходит сигнал (например, `SIGINT` или `SIGTERM`).
3. Если `siginterrupt` настроен так, чтобы **прерывать** вызов, `select`/`poll` вернётся с ошибкой `EINTR`, и код должен:
   - понять, что произошло прерывание;
   - решить: повторять ожидание или аккуратно завершаться.
4. Если `siginterrupt` настроен на **перезапуск**, то ядро попробует повторить ожидание за вас, и ваш код может даже не заметить, что был сигнал (за исключением вызова обработчика).

В чистом Python, особенно при использовании высокоуровневых библиотек (sockets, `selectors`, фреймворки), за вас уже написан код, который правильно реагирует на такие ситуации. Но если вы:

- пишете **свой низкоуровневый event loop**;
- или оборачиваете системные вызовы вручную через `ctypes`/`cffi`;

— знание про `siginterrupt` и `EINTR` становится важным.

#### Запомните

- Если вы работаете с низкоуровневым I/O, будьте готовы к `EINTR`.
- Для большинства высокоуровневых библиотек Python эти детали уже сглажены, но при отладке странных ошибок полезно понимать, откуда берётся `Interrupted system call`.

## 3. Безопасность при работе с сигналами (§101)
