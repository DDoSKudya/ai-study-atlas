### 3.1. Async-signal-safe и ограничения обработчиков (101)

#### Цель раздела

Понять, **почему в обработчике сигнала нельзя делать «всё подряд»**, откуда берутся ограничения и какие действия считаются безопасными.  
Это один из ключевых разделов: если его хорошо понять, вы избежите очень неприятных и сложных багов.

#### Термины

| Термин | Определение |
|--------|-------------|
| **Async-signal-safe** | Свойство функций/операций: их можно безопасно вызывать из обработчика сигнала, даже если сигнал пришёл «в самый неподходящий момент». Такие функции не ломают внутренние структуры, не оставляют систему в полусделанном состоянии. |
| **Race condition** | Ситуация, когда итог работы зависит от того, **кто успел сделать действие первым** (какой поток/обработчик). Если порядок случайный, поведение программы становится непредсказуемым. |

#### Классическая POSIX‑модель

На уровне C/Unix есть список функций, которые считаются **безопасными в обработчике сигнала**: по сути, это очень ограниченный набор (`write` в файловый дескриптор, `_exit`, кое‑что ещё).  
Нельзя вызывать, например, `malloc`, `printf` и большую часть стандартной библиотеки C.

Причина: обработчик может прийти **в середине выполнения** другой функции, и если эта функция неделима и не рассчитана на асинхронный вызов, её внутренние инварианты могут быть нарушены.

Аналогия:

- Представьте, что вы разбираете сложный шкаф: уже наполовину выкрутили болты, детали держатся «на честном слове».  
- В этот момент кто‑то внезапно дёргает шкаф (сигнал) и просит вас **одновременно** начать разбирать ещё один шкаф (внутри обработчика вы вызываете кучу логики).  
- В итоге оба шкафа могут оказаться в полуразобранном, нестабильном состоянии.

Поэтому в мире C говорят: «В обработчике сигнала **можно делать только очень ограниченный набор действий**».

#### Как адаптирован CPython

CPython **не вызывает ваш Python‑обработчик прямо из обработчика ядра ОС**. Вместо этого:

1. В C‑уровневом обработчике ядра выставляется **флаг**, что пришёл сигнал.
2. Главный поток интерпретатора периодически проверяет этот флаг (между байткодами).
3. Когда флаг замечен, вызывается Python‑функция‑обработчик.

Это означает:

- вы **не находитесь посреди произвольного C‑кода**, а исполняете нормальный Python‑код с GIL;
- но сигнал всё равно приходит **в произвольный момент** с точки зрения вашей логики, и обработчик выполняется «поверх» основного потока.

**Практический вывод:** в обработчике Python‑сигнала вы формально можете делать больше, чем в чистом C‑обработчике POSIX, но **паттерн «делаем минимум, ставим флаг» остаётся лучшей практикой**.

Здесь важно понять баланс:

- **хорошая новость:** Python берёт на себя часть сложностей, ваш обработчик работает в «более безопасной зоне»;
- **плохая новость:** если вы начнёте в обработчике:
  - вызывать долгоиграющие операции;
  - работать с глобальными структурами данных;
  - делать сложные переходы состояний,

  то вы всё равно можете словить те самые race condition и «нестабильное состояние программы».

#### Плохой и хороший пример

**Плохо: много логики в обработчике**

```python
import signal
import time

def handler(signum, frame):
    # ПЛОХО: сложная логика, работа с сетью, файлами и т.п.
    print("Получен сигнал, начинаем долгая_очистка()")
    time.sleep(5)  # долгая операция
    print("Готово, выходим...")
    raise SystemExit(0)

signal.signal(signal.SIGTERM, handler)
```

**Лучше: флаг + обработка в основном цикле**

```python
import signal
import time

should_stop = False

def handler(signum, frame):
    global should_stop
    should_stop = True  # только пометили, что надо остановиться

signal.signal(signal.SIGTERM, handler)

while not should_stop:
    # Основная работа
    print("Работаю...")
    time.sleep(1)

print("Очищаем ресурсы...")
# Здесь уже можно аккуратно завершаться: закрыть файлы, соединения и т.п.
```

Разберём, **чем именно плох первый вариант**:

- обработчик `handler`:
  - печатает сообщения;
  - делает `time.sleep(5)` — то есть на 5 секунд «захватывает» главный поток;
  - только потом завершает программу.
- если за это время придут другие сигналы или произойдут другие события, программа может вести себя неожиданно:
  - например, вы отправили `SIGTERM`, ожидаете быстрого завершения, а программа «зависла» на 5 секунд внутри обработчика;
  - или в это время другие части кода ждут, что главный поток будет обрабатывать их задачи.

Во втором варианте:

- обработчик только быстро ставит `should_stop = True`;
- основной цикл, увидев этот флаг, **заканчивает текущую итерацию** и выходит;
- уже за пределами цикла вы можете:
  - записать логи;
  - закрыть соединения;
  - сохранить данные.

То есть мы разделили:

- **реакцию на сигнал** (минимальная и быстрая);
- **сложную логику завершения** (в контролируемой части программы, не в обработчике).

#### Запомните

- Обработчик сигнала должен быть **максимально простым**.
- Лучший шаблон: «выставить флаг → основная логика реагирует на флаг».

### 3.2. Сигналы, потоки и GIL в CPython (101)

#### Цель раздела

Понять, почему обработчики сигналов в Python:

- **должны устанавливаться из главного потока**;
- фактически **выполняются в главном потоке**.

#### Ключевые моменты

- На Unix **сигналы доставляются процессу**, а внутри процесса — одному из потоков (обычно тому, который не маскирует этот сигнал).
- В CPython реализации сделано упрощение:
  - только **главный поток** может устанавливать обработчики сигналов (`signal.signal`), иначе `ValueError`;
  - сам обработчик (Python‑функция) тоже вызывается **в главном потоке**, когда интерпретатор проверяет флаг «пришёл сигнал».
- Это сильно упрощает модель: вам не нужно думать о том, что обработчик выполнится в каком‑то произвольном рабочем потоке.

#### Практический вывод

- Если вы используете `threading`, а сигналы ловите для управления всем приложением, ваш обработчик будет работать **в главном потоке**, в то время как рабочие потоки продолжают свою работу.
- Для координации с потоками удобно использовать **потокобезопасные примитивы** — `threading.Event`, `queue.Queue` и т.п.

#### Очень простая картинка в голове (как это запомнить)

У вас есть:

- **главный поток** — «дирижёр» (он запускает программу, держит главный цикл, часто именно он читает ввод/вывод);
- **рабочие потоки** — «исполнители» (они делают часть работы параллельно).

В CPython сделано так, что:

- «дирижёр» единственный, кто может:
  - назначать обработчики сигналов;
  - реально исполнять Python‑обработчики сигналов.

Почему так сделано? Потому что иначе было бы очень легко получить хаос: сигнал пришёл, обработчик внезапно выполнился в рабочем потоке, который держит какой‑то лок/ресурс — и вы получили deadlock или поломанное состояние. CPython выбирает более предсказуемую модель: **сигналы управляют всей программой через главный поток**.

#### Мини‑демонстрация (проверяем, в каком потоке срабатывает обработчик)

Этот пример можно просто прочитать, а если захотите — запустить на Linux/macOS.

```python
import signal
import threading
import time

def handler(signum, frame):
    # Печатаем, какой поток исполняет обработчик
    print("handler: signum =", signum)
    print("handler: thread name =", threading.current_thread().name)

def worker():
    while True:
        time.sleep(0.5)
        # Рабочий поток живёт отдельно
        # (он не должен устанавливать обработчики сигналов)
        # он просто работает

signal.signal(signal.SIGINT, handler)

t = threading.Thread(target=worker, name="worker-thread", daemon=True)
t.start()

print("Main thread name =", threading.current_thread().name)
print("Нажмите Ctrl+C")
while True:
    time.sleep(1)
```

Ожидаемое поведение:

- `Ctrl+C` приведёт к вызову `handler`;
- `threading.current_thread().name` внутри `handler` будет показывать **главный поток**, а не `worker-thread`.

#### Почему нельзя вызывать `signal.signal(...)` из рабочего потока

Потому что в CPython это намеренно запрещено (обычно вы получите `ValueError`).  
Правильная архитектура такая:

- главный поток ловит сигнал;
- главный поток **сообщает** рабочим потокам «пора завершаться» через `threading.Event` / очередь;
- рабочие потоки периодически проверяют событие и выходят.

Мы ровно это сделаем в следующем разделе (3.3).

### 3.3. Паттерн «флаг и проверка» (101)

#### Цель раздела

Освоить универсальный шаблон реакции на сигналы, который:

- безопасен;
- удобен для тестирования;
- легко переносится на разные типы приложений.

#### Пример: CLI‑утилита с graceful shutdown

```python
import signal
import time

stop = False

def handle_term(signum, frame):
    global stop
    stop = True

signal.signal(signal.SIGINT, handle_term)   # Ctrl+C
signal.signal(signal.SIGTERM, handle_term)  # Вежливое завершение

def main_loop():
    while not stop:
        # Имитация полезной работы
        print("Обрабатываем задачу...")
        time.sleep(1)
    print("Получен сигнал остановки, завершаем...")

if __name__ == "__main__":
    main_loop()
```

Разбор по шагам «как для первого раза»:

1. `stop = False` — флаг «работаем/останавливаемся».
2. `handle_term` — обработчик сигналов:
   - он *ничего умного не делает*;
   - просто ставит `stop = True`.
3. `signal.signal(SIGINT, handle_term)` и `signal.signal(SIGTERM, handle_term)`:
   - говорят Python: «если пришёл `Ctrl+C` или `SIGTERM`, то вызывай `handle_term`».
4. `main_loop`:
   - пока `stop` ещё `False` — мы делаем работу;
   - как только пришёл сигнал и `stop` стал `True` — мы выходим из цикла и печатаем финальное сообщение.

Самое важное, что нужно запомнить:

- обработчик сигнала — это не место для «делать уборку», а место для **сообщить** программе «пора завершаться»;
- уборка и закрытие ресурсов делается **после** выхода из основного цикла, в обычном коде.

#### Улучшение №1: вместо `stop = True` использовать `threading.Event`

Глобальная переменная‑флаг работает, но `threading.Event` обычно удобнее и безопаснее:

- это готовый потокобезопасный флаг;
- его можно передавать в функции/классы;
- он хорошо читается: `stop_event.is_set()` явно говорит, что происходит.

```python
import signal
import threading
import time

stop_event = threading.Event()

def handle_stop(signum, frame):
    # Ставим событие — просим основную логику остановиться
    stop_event.set()

signal.signal(signal.SIGINT, handle_stop)
signal.signal(signal.SIGTERM, handle_stop)

while not stop_event.is_set():
    print("Работаю...")
    time.sleep(1)

print("Останавливаюсь аккуратно: закрываю ресурсы...")
```

#### Улучшение №2: «первый сигнал — мягко, второй — жёстко»

Очень практичный шаблон для сервисов:

- первый `SIGTERM`/`SIGINT` → мягко просим завершиться (`stop_event.set()`), даём время;
- второй сигнал (если вы раздражены и нажали `Ctrl+C` ещё раз) → выходим сразу.

```python
import signal
import threading
import time

stop_event = threading.Event()
signals_count = 0

def handle_stop(signum, frame):
    global signals_count
    signals_count += 1
    if signals_count == 1:
        print("1-й сигнал: начинаем мягкое завершение...")
        stop_event.set()
    else:
        print("2-й сигнал: выходим немедленно.")
        raise SystemExit(1)

signal.signal(signal.SIGINT, handle_stop)
signal.signal(signal.SIGTERM, handle_stop)

while not stop_event.is_set():
    # имитируем полезную работу
    time.sleep(1)
    print("Работаю...")

print("Здесь делаем cleanup (закрыть файлы/сокеты/соединения).")
```

#### Важная ремарка: обработчик вызывается, но поток/процесс может быть «занят»

Флаговый паттерн не делает магии:

- если ваш код прямо сейчас «завис» в длинной операции без точек проверки (например, бесконечное вычисление в C‑расширении),
  то он может не успеть быстро проверить `stop_event` и остановиться.

Но в обычном Python‑цикле с `sleep`, `await`, чтением очереди и т.п. этот паттерн работает очень хорошо.

Этот шаблон легко расширить:

- вместо глобальной переменной можно использовать `threading.Event`;
- можно отслеживать количество сигналов (`SIGTERM` → мягкое завершение, второй `SIGTERM` → немедленное).
