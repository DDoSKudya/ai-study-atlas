## 0.1 История решений и дизайн-философия Python

Чтобы писать на Python осознанно, важно понимать не только синтаксис, но и **почему** язык устроен именно так: какие принципы заложены в дизайн, как принимаются изменения и какие компромиссы были сделаны. Этот раздел даёт подробную основу: от философии (Zen) до механизмов принятия решений (PEP, BDFL, Steering Council) и ключевых технических решений (GIL, Py2→Py3).

### 0.1.1 Zen of Python (PEP 20)

**Zen of Python** — короткий манифест принципов дизайна языка, сформулированный Тимом Петерсом. Его можно вывести в интерпретаторе командой:

```python
import this
```

Вывод — 19 афоризмов в стиле «The Zen of Python» (по одному на строку). Ниже каждый принцип разобран подробно: что он значит на практике, как применяется в коде и какие типичные ошибки он помогает избежать.

#### Разбор принципов с примерами

**1. Beautiful is better than ugly.**

Код пишется для людей: его будут читать и менять. «Красота» здесь — не декоративность, а ясность структуры и имен. Один и тот же алгоритм можно записать так, что его трудно читать, или так, что намерение видно сразу.

Плохо (неочевидная логика, «магия»):

```python
# Неочевидная однострочная «хитрость»
x = [l for s in data for l in s if l and l[0] not in 'aeiou'][:10]
```

Лучше (понятные имена, пошаговая логика):

```python
# Явная цель: первые 10 непустых строк, не начинающихся с гласной
lines = []
for block in data:
    for line in block:
        if line and line[0].lower() not in 'aeiou':
            lines.append(line)
            if len(lines) >= 10:
                break
    if len(lines) >= 10:
        break
result = lines[:10]
```

Читаемость важнее краткости: код живёт долго, его будут править другие (или вы через полгода).

---

**2. Explicit is better than implicit.**

Явное указание намерения уменьшает сюрпризы. Импорты, типы, источники данных, условия — лучше называть их прямо, чем полагаться на глобальное состояние или неочевидные соглашения.

Плохо (неявный источник данных, неявное поведение):

```python
from module import *  # Какие имена попали в текущий namespace — неочевидно
def save(data):       # Куда сохраняем? В какой формат? Не сказано
    open('out').write(data)
```

Лучше:

```python
from module import func_a, func_b   # Явно перечислены используемые имена
def save(data: bytes, path: str = 'out.txt') -> None:
    with open(path, 'wb') as f:
        f.write(data)
```

Явность распространяется на аннотации типов (`def f(x: int) -> str`), явные аргументы функций вместо глобальных переменных, явную обработку ошибок вместо «молчаливого» проглатывания.

---

**3. Simple is better than complex.**

Простое решение легче понимать, тестировать и менять. «Простое» не значит примитивное — оно значит: минимум сущностей и ветвлений при достижении той же цели.

Плохо (лишняя абстракция для простой задачи):

```python
class Handler:
    def handle(self, x): return self._process(x)
class Processor(Handler):
    def _process(self, x): return x * 2
Processor().handle(5)  # 10
```

Лучше для такой задачи:

```python
def double(x):
    return x * 2
double(5)  # 10
```

Введение классов, декораторов и паттернов оправдано, когда они реально упрощают систему (например, убирают дублирование или делают расширение предсказуемым), а не «на будущее».

---

**4. Complex is better than complicated.**

«Complex» — неизбежная сложность предметной области: много правил, состояний, граничных случаев; её нельзя убрать без потери функциональности. «Complicated» — запутанность из-за неудачного дизайна: неочевидные зависимости, скрытые побочные эффекты, дублирование, «магические» связи между частями кода. Цель — оставить только первую и убрать вторую: сложность должна быть структурированной и предсказуемой, а не размазанной по проекту.

**Пример: разбор конфигурации с валидацией.** Сама по себе задача сложная (много полей, типы, обязательность, значения по умолчанию). Решать её через десять глобальных флагов и разбросанные по коду проверки — **complicated**: непонятно, кто где что читает и когда оно актуально. Вынести в один модуль с явной схемой (например, dataclass + валидатор или Pydantic-модель) — **complex**, но структурированно: все правила конфигурации в одном месте, поток данных явный.

Ещё пример: «сложный» алгоритм с несколькими фазами лучше оформить в виде явных шагов (отдельные функции или классы с понятными именами), чем спрятать логику в одну гигантскую функцию с десятками веток — последнее как раз complicated.

---

**5. Flat is better than nested.**

Глубокая вложенность (много уровней `if`/`for`/`try` внутри друг друга) затрудняет чтение и увеличивает когнитивную нагрузку. Плоский код — ранний выход из функции, вынос вложенных циклов в отдельные функции или comprehensions, guard-условия в начале.

Плохо (глубокая вложенность):

```python
def process(user):
    if user:
        if user.is_active:
            if user.subscription:
                if user.subscription.is_valid():
                    return do_work(user)
                else:
                    return None
            else:
                return None
        else:
            return None
    return None
```

Лучше (плоско, ранние выходы):

```python
def process(user):
    if not user or not user.is_active:
        return None
    if not user.subscription or not user.subscription.is_valid():
        return None
    return do_work(user)
```

---

**6. Sparse is better than dense.**

Одна строка с десятком операций хуже нескольких строк с одной логической операцией. Плотный код трудно отлаживать и ревьюить.

Плохо:

```python
return {k: v for k, v in d.items() if v is not None and (not callable(v) or getattr(v, '__name__', '').startswith('_'))}
```

Лучше:

```python
def keep_key(k, v):
    if v is None:
        return False
    if callable(v):
        name = getattr(v, '__name__', '')
        return name.startswith('_')
    return True
return {k: v for k, v in d.items() if keep_key(k, v)}
```

---

**7. Readability counts.**

Читаемость — главный критерий качества кода. Код читают гораздо чаще, чем пишут: при отладке, ревью, рефакторинге, онбординге новых разработчиков. Поэтому имена переменных и функций должны однозначно отражать назначение; структура кода — облегчать понимание потока управления и данных, а не скрывать его за «хитрыми» конструкциями.

**Что способствует читаемости:**

- **Именование:** `user_ids` вместо `u` или `data2`; `def calculate_total_price(...)` вместо `def calc(...)` без контекста. Имена должны отвечать на вопрос «что это?» или «что делает эта функция?».
- **Длина блоков:** короткие функции и циклы легче охватить взглядом; если функция разрослась, её стоит разбить на шаги с говорящими именами.
- **Единообразие:** один и тот же стиль отступов, кавычек, переносов строк во всём проекте (PEP 8 и форматеры вроде Black как раз фиксируют это).
- **Комментарии и докстринги:** там, где *зачем* неочевидно из кода; не дублировать очевидное («увеличиваем счётчик» над `i += 1`).

PEP 8 (Style Guide for Python Code) и рекомендации по именованию (snake_case для функций и переменных, CapWords для классов и т.д.) служат именно читаемости и единообразию в сообществе.

---

**8. Special cases aren't special enough to break the rules.**

Не вводить исключения из общих правил «просто потому что тут особый случай». Если в проекте или в языке принято правило (например, «все публичные методы документируются», «ошибки не глотаются без явной обработки», «импорты — только явные»), то и особый случай должен ему следовать. Иначе накапливаются «исключения из исключений», код перестаёт быть предсказуемым.

**Практика:** если вы ловите себя на мысли «здесь мы нарушим правило, потому что это особый случай» — стоит либо пересмотреть правило для всего кода (и зафиксировать это в стиле/документации), либо найти способ решить задачу в рамках правила (например, вынести особую логику в отдельную функцию с явным именем и документировать её). «Особый случай» редко настолько особенный, чтобы оправдать разрыв консистентности.

---

**9. Although practicality beats purity.**

Теоретическая чистота дизайна уступает практической пользе. Решения принимаются с учётом реального использования, поддержки и стоимости изменений — а не ради «идеальной» архитектуры в вакууме.

**Примеры:**

- **Дублирование кода:** правило «DRY» (Don't Repeat Yourself) не означает «любое повторение — зло». Небольшое дублирование двух похожих кусков иногда читается проще, чем одна абстракция с кучей параметров и веток. Стоит выносить в общую функцию, когда дублирование реально мешает (изменения приходится править в нескольких местах и легко забыть одно).
- **Идеальная иерархия классов:** «правильная» с точки зрения ООП-теории глубокая иерархия может быть переусложнением для простой задачи; плоский набор функций или один класс с ясными методами часто практичнее.
- **«Чистый» функциональный стиль:** отказ от любых побочных эффектов ради чистоты может привести к нечитаемому коду; на практике разумный баланс между чистотой и удобством важнее догмы.

Итог: оптимизируйте под понятность и поддерживаемость в вашем контексте, а не под абстрактную «правильность».

---

**10–11. Errors should never pass silently. Unless explicitly silenced.**

Ошибки не должны исчезать без следа: это маскирует баги и усложняет отладку. Если исключение обрабатывается намеренно (например, ожидаемый `KeyError` или «файл может отсутствовать»), это должно быть **явно**: отдельный `except` с логированием или осмысленной обработкой, а не голый `except: pass`.

Плохо:

```python
try:
    risky()
except:
    pass   # Ошибка «проглочена», непонятно что произошло
```

Лучше:

```python
try:
    risky()
except ExpectedError as e:
    logger.debug("Expected situation: %s", e)   # Явно подавлено с фиксацией
```

---

**12. In the face of ambiguity, refuse the temptation to guess.**

При неоднозначности (в API, в данных, в требованиях) не угадывать — требовать **явного указания**. Угадывание ведёт к скрытым багам и несовместимости между окружениями или версиями.

**Примеры:**

- **Кодировка файла:** не полагаться на умолчание системы; явно указывать `encoding='utf-8'` при открытии текстовых файлов. Иначе на другой машине или в другой локали поведение может измениться.
- **Тип аргумента:** аннотировать сигнатуру (`def f(x: int) -> str`) или явно проверять/документировать ожидаемый тип; не предполагать «здесь всегда приходит список» без контракта.
- **Граничные значения:** поведение при пустой коллекции, при `None`, при нуле — описать в докстринге или в коде (assert, проверки); не оставлять «как получится».
- **Версии зависимостей:** фиксировать версии в `requirements.txt` или `pyproject.toml`; не полагаться на «последнюю версию» при сборке — иначе сборка может сломаться после обновления пакета.

Итог: при любом «а что если…?» — зафиксировать решение явно (в коде, в конфиге или в документации), а не полагаться на неочевидное поведение.

---

**13–14. There should be one—and preferably only one—obvious way to do it. Although that way may not be obvious at first unless you're Dutch.**

Язык подталкивает к **одному очевидному способу** решить типичную задачу. Это снижает разброс стилей, облегчает чтение чужого кода и онбординг новых разработчиков: все используют одни и те же конструкции для одних и тех же целей.

**Примеры «одного очевидного способа»:**

- **Обход коллекции** — `for item in seq` (итерация по итерабельным объектам), а не ручной индекс `for i in range(len(seq))`, если индекс не нужен.
- **Чтение файла** — `with open(path, encoding='utf-8') as f: ...` (контекстный менеджер гарантирует закрытие), а не открытие без `with` и надежда на закрытие вручную.
- **Проверка наличия элемента** — `if x in container`; для словаря — `d.get(key)` или `if key in d`.
- **Обработка ошибок** — EAFP: `try/except` вокруг основной операции, а не длинные цепочки проверок «а что если нет?».

«Obvious» может быть неочевиден новичку (например, итераторы, контекстные менеджеры, генераторы). В таком случае **стоит изучить** принятый в языке способ, а не изобретать свой (например, ручной счётчик вместо `for item in seq`). Шутка «unless you're Dutch» отсылает к Гвидо ван Россуму: автор языка считал «очевидный способ» естественным, хотя новичкам он может потребовать объяснения.

---

**15–16. Now is better than never. Although never is often better than *right* now.**

Два дополняющих друг друга принципа про темп и качество решений.

**«Now is better than never»:** лучше сделать итеративно (написать рабочий вариант, потом улучшить), чем откладывать на «идеальный» дизайн. Бесконечное проектирование без кода ведёт к тому, что требования меняются, а продукта нет. Рабочий прототип или минимальная версия дают обратную связь и позволяют уточнять требования.

**«Although never is often better than *right* now»:** но «сделать прямо сейчас» без понимания последствий часто хуже, чем короткая пауза. Хаки и костыли без плана рефакторинга накапливаются; поспешные изменения в общих модулях ломают зависимый код. Перед крупным решением или «быстрым фиксом» в критичном месте стоит хотя бы кратко обдумать: что сломается, как это будет поддерживаться.

**Баланс:** итерируйте быстро, но не отключайте голову. Фиксируйте технический долг (TODO, тикеты) и выделяйте время на рефакторинг, вместо того чтобы только наращивать «временные» решения.

---

**17–18. If the implementation is hard to explain, it's a bad idea. If the implementation is easy to explain, it may be a good idea.**

Код должен **поддаваться объяснению**: «здесь мы делаем X, потому что Y». Если вы не можете за пару предложений объяснить коллеге (или себе через полгода), что делает участок кода и зачем — высока вероятность, что идея переусложнена или скрывает баг.

**Практика:**

- Если при ревью или при отладке вы ловите себя на мысли «это сложно объяснить» — стоит упростить: разбить на шаги с говорящими именами, вынести логику в отдельные функции, убрать «магию».
- Обратное тоже полезно: если реализация легко объясняется («мы кэшируем результат, потому что функция тяжёлая и вызывается с одними и теми же аргументами»), это признак удачного решения.
- Сложность предметной области — не оправдание нечитаемого кода; сложную логику можно оформить так, чтобы каждый кусок был объясним по отдельности.

Итог: используйте «объяснимость» как тест на качество дизайна.

---

**19. Namespaces are one honking great idea—let's do more of those!**

**Пространства имён** (namespaces) изолируют имена и уменьшают конфликты и неожиданные побочные эффекты. В Python пространства имён есть у модулей, классов, функций, пакетов; при обращении к имени интерпретатор ищет его в локальной области, затем в enclosing, глобальной и встроенной (LEGB).

**Почему это важно:**

- **Модули:** каждая `.py`-файл — свой namespace; импорт `import module` или `from module import a, b` создаёт явную границу: видно, откуда взялось имя. Избегание `from module import *` в библиотеках и больших проектах — следствие этого принципа: иначе в текущем namespace попадают десятки имён из модуля, и источник имени перестаёт быть очевидным.
- **Классы:** атрибуты класса и экземпляра изолированы от глобальных имён; метод получает свой контекст (self, атрибуты экземпляра).
- **Функции:** локальные переменные функции не видны снаружи и не засоряют глобальную область; повторное использование имени в другой функции не конфликтует.
- **Пакеты:** иерархия каталогов с `__init__.py` (или без них в namespace packages) даёт логическую структуру и позволяет иметь одинаковые имена модулей в разных пакетах.

Итог: используйте пространства имён осознанно — не тащите всё в глобальную область, предпочитайте явный импорт и короткие области видимости.

---

**Практический вывод:** при сомнениях в стиле кода или выборе конструкции сверяйтесь с Zen. Они не жёсткие законы, а ориентиры; чаще всего помогают «Explicit is better than implicit» и «Readability counts».

### 0.1.2 Эволюция через PEP: консенсус, обратная совместимость, deprecation period

Изменения в языке и стандартной библиотеке оформляются в виде **PEP** (Python Enhancement Proposal) — предложений по улучшению Python. Понимание того, как устроена эволюция языка, помогает ориентироваться в документации и предсказывать, как будут развиваться возможности и что считается устаревшим.

---

#### Что такое PEP и зачем он нужен

**PEP** (Python Enhancement Proposal) — это нумерованный документ, который описывает:

- **Проблему** или возможность (зачем менять язык или библиотеку).
- **Предлагаемое решение** (синтаксис, API, поведение).
- **Альтернативы** и причины их отклонения.
- **Влияние на обратную совместимость** и план миграции (в т.ч. deprecation).

PEP служат **единым источником истины**: по ним можно понять не только *что* изменилось, но и *почему* и *как* это согласуется с остальным дизайном. Все актуальные PEP публикуются на [peps.python.org](https://peps.python.org/). Формат и назначение PEP описаны в **PEP 1** (PEP Purpose and Guidelines); при первом знакомстве с процессом развития Python полезно прочитать именно его.

**Типичная структура PEP (Standards Track):**

- **Abstract** — краткое изложение предложения (несколько предложений).
- **Motivation** — зачем нужны изменения; какую проблему решаем или какую возможность даём.
- **Rationale** — почему выбран именно такой вариант; связь с Zen и другими PEP.
- **Specification** — техническое описание: синтаксис, API, поведение по шагам.
- **Backwards Compatibility** — что ломается, план миграции, deprecation (версии, предупреждения).
- **Rejected Ideas** — что рассматривали и почему отклонили.

При изучении новой возможности (например, type hints, async/await, match/case) достаточно прочитать **Abstract**, **Motivation** и **Rationale** — этого хватает, чтобы понять контекст и причины решений. **Specification** нужна при реализации или при глубоком разборе деталей.

---

#### Типы PEP

- **Standards Track** — изменения в языке или стандартной библиотеке (новый синтаксис, новые модули, изменение поведения). Именно их чаще всего имеют в виду, когда говорят «приняли PEP».
- **Informational** — рекомендации и описания без изменения кода (например, PEP 20 — Zen of Python, PEP 8 — стиль кода).
- **Process** — процесс разработки Python (например, как избирается Steering Council, как принимаются PEP).

Номер PEP присваивается при создании черновика и не меняется; по нему удобно ссылаться в обсуждениях и в коде.

---

#### Жизненный цикл PEP (упрощённо)

1. **Draft** — автор формулирует идею и описание; PEP получает номер и публикуется в репозитории (например, на GitHub или в официальном репозитории PEP).
2. **Обсуждение** — список рассылки python-dev, дискуссии на Discourse или в issue-трекере, правки текста по замечаниям. Обсуждение может длиться недели или месяцы; крупные изменения (например, PEP 634 match/case) обсуждаются долго.
3. **Accepted / Rejected / Withdrawn** — решение принимает (исторически) BDFL или (сейчас) Steering Council после обсуждения. Accepted — предложение принято к реализации; Rejected — отклонено с обоснованием; Withdrawn — автор отзывает предложение.
4. **Реализация** — патчи в репозиторий CPython (или другой реализации); версия, в которой изменение войдёт, указывается в PEP. Реализация может делать сам автор или другие разработчики; ревью кода — отдельный процесс.
5. **Final** — изменение включено в релиз; статус PEP переводится в Final. После этого PEP остаётся неизменным документом; исправления опечаток иногда допускаются по отдельному процессу.

Обратная совместимость учитывается на этапе обсуждения: ломающие изменения требуют веской мотивации и, как правило, периода deprecation (предупреждение в одной или нескольких версиях перед удалением или изменением поведения).

---

#### Обратная совместимость и deprecation

**Обратная совместимость** — старый корректный код продолжает работать в новых версиях. В Python к ней относятся серьёзно: поломка без веской причины считается ошибкой дизайна. Поэтому новые возможности часто добавляют так, чтобы старый код не трогал (новые функции, новые аргументы с умолчаниями, новый синтаксис рядом со старым).

**Deprecation (устаревание)** — объявление какой-то возможности устаревшей с предупреждением. Обычно при первом использовании устаревшей конструкции выдаётся `DeprecationWarning` (в тестах и скриптах его часто включают через `-W default` или `warnings.filterwarnings`). Цель — дать время обновить код и зависимости до момента **removal** (удаления).

**Пример из практики:** в Python 3.10 устарел альтернативный синтаксис для типов в аннотациях (`List[str]` из `typing` в пользу встроенного `list[str]`). Сначала добавлено предупреждение, позже (в следующих мажорных версиях) устаревшие формы могут быть удалены или оставлены только для совместимости. Конкретные версии всегда указаны в соответствующем PEP.

**Как выглядит deprecation в коде:** при первом использовании устаревшей конструкции интерпретатор может выдать `DeprecationWarning`. По умолчанию в скриптах предупреждения из категории DeprecationWarning иногда не показываются (зависит от версии и способа запуска). Чтобы увидеть их и не пропустить переход на новый API, запускают с флагом `-W default` или включают в коде:

```python
import warnings
warnings.filterwarnings("default", category=DeprecationWarning)
```

В тестах и CI часто включают `-W error::DeprecationWarning`, чтобы использование устаревшего API приводило к падению теста и вынуждало обновить код до снятия поддержки старого API.

**Терминология:**

- **PEP** — Python Enhancement Proposal; документ с предложением и обоснованием.
- **Deprecation** — объявление устаревшим + предупреждение в коде.
- **Removal** — фактическое удаление после периода deprecation (обычно в новой мажорной версии).
- **BDFL** (см. ниже) раньше принимал финальное решение по PEP; сейчас это делает **Steering Council**.

---

#### Примеры важных PEP (для ориентира)

| PEP | Тема | Назначение |
|-----|------|------------|
| PEP 8 | Style Guide for Python Code | Стиль кода (отступы, именование, длина строк). |
| PEP 20 | Zen of Python | Философия дизайна (`import this`). |
| PEP 484 | Type Hints | Аннотации типов (typing). |
| PEP 492 | async/await | Корутины и синтаксис async/await. |
| PEP 634 | Structural Pattern Matching | Синтаксис match/case. |
| PEP 703 | Making the Global Interpreter Lock Optional | Free-threaded Python (экспериментально). |

При изучении новой возможности (типы, async, match) полезно открыть соответствующий PEP и прочитать разделы **Motivation** и **Rationale**. Найти PEP можно по номеру: например, [peps.python.org/pep-0484](https://peps.python.org/pep-0484) — Type Hints, [peps.python.org/pep-0492](https://peps.python.org/pep-0492) — async/await, [peps.python.org/pep-0634](https://peps.python.org/pep-0634) — match/case. В каждом PEP в начале указаны статус (Final, Accepted, Draft и т.д.) и версия Python, к которой относится изменение.

### 0.1.3 Ключевые решения: GIL, «batteries included», «one obvious way»

#### GIL (Global Interpreter Lock) — подробно

**GIL** — глобальная блокировка (мьютекс) в реализации CPython. В каждый момент времени только **один** поток может выполнять байткод Python. То есть несколько потоков **не** ускоряют чисто вычислительные (CPU-bound) задачи: они по очереди получают GIL, остальные ждут.

**Ментальная модель (упрощённо):** представьте одну очередь в кассу (интерпретатор). Только один человек (поток) в момент времени обслуживается (выполняет байткод). Остальные стоят в очереди. Добавление новых потоков не ускоряет «обслуживание» — касса одна. Зато пока один поток «ушёл попить кофе» (ждёт I/O — чтение файла, ответ сети), касса может обслужить следующего в очереди. Поэтому много потоков помогают, когда основное время уходит на ожидание I/O, а не на вычисления в самом Python.

---

**Зачем GIL был введён**

В CPython управление памятью объектов построено на **счётчике ссылок** (reference counting). Операции вроде присваивания или удаления ссылки меняют этот счётчик. Без общей блокировки два потока могли бы одновременно менять ссылки на один и тот же объект, что ведёт к гонкам и порче данных. GIL упростил реализацию: только один поток в данный момент выполняет байткод и меняет объекты, поэтому счётчики ссылок остаются согласованными без тонкой блокировки каждого объекта. Дополнительно GIL защищает внутренние структуры интерпретатора от одновременного доступа.

---

**Когда GIL освобождается**

GIL не держится непрерывно. Он освобождается, в частности:

- При **I/O-операциях**: чтение/запись файла, сетевые вызовы (внутри реализованы через системные вызовы, которые блокируют поток; CPython на время блокировки отдаёт GIL, чтобы другие потоки могли работать).
- В **C-расширениях**: код на C может явно освободить GIL (например, через `Py_BEGIN_ALLOW_THREADS` / `Py_END_ALLOW_THREADS`), выполнить тяжёлую работу без GIL и снова захватить его. Так делают NumPy и другие библиотеки.

Поэтому **I/O-bound** задачи (много ожидания диска или сети) выигрывают от потоков: пока один поток ждёт ответа, другой выполняет Python-код. **CPU-bound** задачи (тяжёлые вычисления в чистом Python) от добавления потоков почти не ускоряются — потоки по очереди держат GIL.

---

**Практические следствия**

- **Многопоточность (threading)** в CPython полезна для I/O-bound (сеть, диск, ожидание ответов), а не для распараллеливания «числодробилок» на чистом Python. Пример: веб-сервер, обрабатывающий много одновременных запросов, — пока один поток ждёт ответа БД или другого сервиса, другой может обрабатывать новый запрос.
- **Параллельные вычисления (CPU-bound)** делают через **multiprocessing** (отдельные процессы, у каждого свой интерпретатор и свой GIL; процессы не делят память, данные передаются через очереди или общую память) или через вынос горячих участков в C/Cython/NumPy с освобождением GIL. Пример: перебор комбинаций, тяжёлые численные циклы — либо вынести в отдельный процесс (или пул процессов), либо переписать горячий цикл на Cython/NumPy.
- В будущем возможен вариант без GIL (PEP 703 — free-threaded Python, экспериментальный build); на этапе изучения достаточно понимать текущую модель: один поток выполняет байткод, I/O и C-расширения могут отдавать GIL.

**Краткое сравнение:** при **I/O-bound** (много ожидания) — `threading` или `asyncio` дают выигрыш; при **CPU-bound** (много вычислений в чистом Python) — `threading` почти не ускоряет, нужен `multiprocessing` или нативный код. **Практическая иллюстрация:** четыре потока с CPU-bound задачей в чистом Python (например, тяжёлый цикл в каждом потоке) **не дадут** примерно четырёхкратного ускорения по сравнению с одним потоком — потоки по очереди держат GIL, поэтому суммарное время выполнения остаётся того же порядка. Чтобы реально ускорить такие вычисления, используют несколько процессов (multiprocessing) или вынос горячих участков в C/Cython/NumPy.

**Термины:**

- **GIL** — Global Interpreter Lock.
- **CPU-bound** — нагрузка на процессор (вычисления); в чистом Python потоки её не распараллеливают из-за GIL.
- **I/O-bound** — нагрузка на ввод-вывод (диск, сеть); потоки здесь полезны, т.к. во время ожидания I/O GIL отдаётся.

---

#### «Batteries included»

Под этим подразумевается **богатая стандартная библиотека (stdlib)**: с интерпретатором уже поставляются модули для типичных задач, без необходимости сразу ставить сторонние пакеты — «батарейки в комплекте».

**Что входит в stdlib (по категориям, неполный список):**

- **Файлы и пути:** `os`, `os.path`, `pathlib`, `tempfile`, `shutil`, `glob`.
- **Сеть:** `socket`, `urllib.request`, `http.server`, `email`, `json`, `xml.etree`.
- **Данные и сериализация:** `json`, `pickle`, `csv`, `configparser`, `xml.etree`, `sqlite3`.
- **Дата и время:** `datetime`, `time`, `calendar`; в 3.9+ — `zoneinfo`.
- **Текст и кодировки:** `str`, `bytes`, `re`, `difflib`, `unicodedata`.
- **Структуры данных и алгоритмы:** `collections`, `heapq`, `bisect`, `array`, `copy`.
- **Система и процессы:** `subprocess`, `multiprocessing`, `threading`, `signal`, `sys`, `platform`.
- **Безопасность и криптография:** `hashlib`, `secrets`, `ssl`, `hmac`.
- **Инструменты разработки:** `unittest`, `doctest`, `pdb`, `logging`, `argparse`, `typing`.

**Зачем это нужно:** снижается зависимость от внешнего репозитория (PyPI), упрощается развёртывание (достаточно установить Python), многие задачи решаются без дополнительных зависимостей. Полный список модулей — в документации Python (Library Reference); в плане изучения им посвящён справочник §7a и соответствующие части (§8 pathlib, §10 dataclasses, §11 collections, §15 модули stdlib и т.д.).

**Когда stdlib недостаточна:** продвинутый HTTP-клиент с пулом соединений, ORM, тяжёлые численные вычисления, специализированные форматы — тогда подключают пакеты с PyPI (requests, SQLAlchemy, NumPy и т.д.). Выбор «stdlib vs пакет» делают по задаче: не тянуть тяжёлые зависимости «на всякий случай» и не изобретать велосипед там, где пакет уже решает задачу.

---

#### «One obvious way» vs «multiple ways»

В Zen сказано: «There should be one—and preferably only one—obvious way to do it.» Язык подталкивает к **одному очевидному способу** решить типичную задачу; это снижает разброс стилей и облегчает чтение чужого кода.

**Примеры «одного очевидного способа»:**

| Задача | Очевидный способ | Альтернативы (для особых случаев) |
|--------|-------------------|-----------------------------------|
| Обход коллекции | `for item in seq:` | `while` + индекс, ручной итератор |
| Открытие файла с гарантией закрытия | `with open(path, encoding='utf-8') as f:` | `try/finally`, ручной `f.close()` |
| Проверка наличия ключа в словаре | `d.get(key)` или `if key in d` | `try/except KeyError` (EAFP) |
| Повторение N раз | `for _ in range(N):` | `while` с счётчиком |
| Чтение файла построчно | `for line in f:` | `f.readline()` в цикле, `f.read().splitlines()` |
| Обработка ошибок | EAFP: `try/except` вокруг операции | LBYL: проверки перед действием |

«Очевидный путь» — приоритет для типичных задач; гибкость не отменяется: для особых случаев остаются другие инструменты (`while`, ручные итераторы, LBYL и т.д.). Не поощряется избыточное разнообразие способов для одной и той же задачи без веской причины.

---

### 0.1.4 BDFL и переход к Steering Council; роль PEP в архитектуре языка

#### BDFL (Benevolent Dictator For Life)

**BDFL** — «доброжелательный диктатор на всю жизнь». Так называли **Гвидо ван Россума** (Guido van Rossum): он создал Python в конце 1980-х и долгое время имел **последнее слово** по спорным решениям — синтаксис, стандартная библиотека, принятие или отклонение PEP, приоритеты развития языка.

**Идея модели BDFL:**

- **Один человек с ясным видением** может быстрее разрешать тупиковые споры и сохранять целостность языка: не размывать дизайн комитетом с противоречивыми интересами.
- **«Доброжелательность»** подразумевает учёт мнений сообщества: обсуждения на python-dev, учёт аргументов в PEP, обоснование решений. BDFL не «диктует» произвольно, а ведёт процесс и принимает итоговые вердикты после обсуждения.

**Примеры решений эпохи BDFL:** введение декораторов, контекстных менеджеров, аннотаций типов (PEP 484), async/await (PEP 492), решение о переходе на Python 3 с контролируемым разрывом совместимости — всё это проходило через обсуждение и финальное решение Гвидо.

---

#### Переход к Steering Council (2018)

В **июле 2018 года** Гвидо объявил, что отходит от роли BDFL (сообщение «Transfer of power»). Причины он не детализировал публично; сообщество приняло решение перейти к **коллегиальному управлению**.

**Steering Council** — совет из нескольких разработчиков (обычно 5 человек), избираемый сообществом на ограниченный срок (примерно два года). Детали избрания, полномочий и процесса описаны в **PEP 13** (Python Language Governance) и связанных governance-документах.

**Как сейчас принимаются решения:**

- PEP по-прежнему пишутся, обсуждаются на python-dev и в issue-трекере.
- **Принятие или отклонение** PEP и стратегических вопросов — за Steering Council. Council не «диктует» идеи сверху: он рассматривает предложения, учитывает обсуждение и принимает итоговые вердикты (Accept, Reject, Defer).
- Гвидо остаётся в сообществе как разработчик; он может участвовать в обсуждениях и вносить предложения, но последнее слово — за Council.

**Практический вывод:** при изучении «почему язык устроен так» полезно смотреть историю PEP и обсуждения; для текущих решений — документы Steering Council и актуальные PEP.

#### Роль PEP в архитектуре языка

PEP фиксируют не только **новую возможность**, но и:

- **Мотивацию** (Motivation): зачем менять язык или библиотеку.
- **Обоснование** (Rationale): почему выбран именно такой вариант.
- **Альтернативы** (Alternatives): что рассматривали и почему отклонили.
- **Обратную совместимость** (Backwards Compatibility): что ломается, план миграции, deprecation.

Поэтому PEP — это и **архитектурная документация** решений: по ним можно понять, почему язык устроен так, а не иначе, и как планируется эволюция. При изучении новой фичи (типы, async, match и т.д.) полезно открыть соответствующий PEP и прочитать разделы Motivation, Rationale и Backwards Compatibility.

### 0.1.5 Python 2 vs 3: причины раскола, уроки

Раскол между Python 2 и 3 возник из-за **намеренно несовместимых изменений** в Python 3: исправление накопленных проблем (строки, деление, print, исключения и т.д.) потребовало поломать часть старого кода. Ниже — ключевые отличия с примерами, хронологией и выводами для перехода.

---

#### Краткая хронология

- **Python 2.0** (2000) — введён тип `unicode`, списковые comprehensions, сборщик мусора.
- **Python 3.0** (2008) — намеренно несовместимые изменения: str/unicode/bytes, print как функция, деление, исключения, удаление `xrange`, `long` и др.
- **Python 2.7** (2010) — последняя мажорная версия ветки 2.x; в неё перенесли часть фич из 3.x для облегчения миграции.
- **Конец поддержки Python 2.7** — январь 2020; с тех пор только исправления безопасности не выпускаются.

**Итог:** для нового кода и обучения ориентируются на Python 3; миграция существующего кода с 2.7 требует правок и инструментов (2to3, six, `future` — см. план §35–37).

---

#### Строки: str/unicode/bytes

В **Python 2** тип `str` — байтовая строка (последовательность байт); тип `unicode` — строка символов (Unicode). Литерал `"hello"` — это `str` (байты); литерал `u"hello"` — `unicode`. В коде часто смешивали оба подхода, что вело к ошибкам при работе с не-ASCII и разной кодировкой (например, конкатенация str и unicode вызывала неявное преобразование и могла падать или давать неверный результат).

В **Python 3** `str` — **всегда** последовательность символов Unicode; `bytes` — неизменяемая последовательность байт (0–255); тип `unicode` удалён, его роль занял `str`. Литерал `"hello"` — это `str`; литерал `b"hello"` — `bytes`.

**Пример (концептуально):**

```python
# Python 2:
# type("привет") зависит от кодировки исходного файла; type(u"привет") — unicode
# s + u — неявное преобразование, возможны UnicodeDecodeError

# Python 3:
s = "привет"       # str (Unicode)
b = s.encode()     # bytes в UTF-8 по умолчанию
s2 = b.decode()    # снова str
# type(s) — str, type(b) — bytes; смешивать без явного encode/decode нельзя
```

**Урок:** в Python 3 по умолчанию работаем с текстом как с `str` (Unicode); с «сырыми» байтами — как с `bytes`/`bytearray`. Явное разделение устраняет путаницу «это байты или символы» и проблемы с кодировками (подробнее — Часть I, §6, §15d).

---

#### print — инструкция vs функция

В **Python 2** `print` — инструкция: `print "hello"` или `print "a", "b"`. В **Python 3** `print` — встроенная функция: `print("hello")`, `print("a", "b", sep=", ")`.

**Плюсы функции:** единообразие с остальным API; возможность передавать аргументы (`sep`, `end`, `file`, `flush`); в тестах или при логировании можно подменить `print` своей функцией; можно передать `print` как аргумент (например, в функцию, которая выводит отладочную информацию).

**Миграция:** автоматическая замена `print x` на `print(x)` делается утилитой `2to3`; для совместимости с обоими версиями в Py2 в начале файла добавляют `from __future__ import print_function`.

---

#### Деление

В **Python 2** для целых чисел оператор `/` — целочисленное деление: `3 / 2 == 1`. В **Python 3** `/` — всегда вещественное деление: `3 / 2 == 1.5`; для целочисленного — явный оператор `//`: `3 // 2 == 1`.

**Урок:** устраняется частый источник ошибок у новичков (ожидание 1.5, получение 1); намерение делается явным: деление «с дробной частью» — `/`, «целочисленное» — `//`.

**Миграция:** в Py2 для поведения Py3 в начале файла добавляют `from __future__ import division`; тогда `3 / 2 == 1.5`, а целочисленное — `3 // 2`.

---

#### Целые числа: long и int

В **Python 2** были типы `int` (ограниченный диапазон, обычно 32 или 64 бита) и `long` (произвольная точность); при переполнении `int` автоматически превращался в `long`. В **Python 3** остался один тип `int` неограниченной точности; имя `long` удалено.

**Миграция:** код, проверяющий `type(x) is long` или использующий имя `long`, нужно менять на `int`.

---

#### Итерация по словарю

В **Python 2** методы `.keys()`, `.values()`, `.items()` возвращали **списки** (копии ключей/значений/пар). В **Python 3** они возвращают **view-объекты** (dict_keys, dict_values, dict_items): ленивые, не копируют данные и **отражают текущее состояние словаря**. Если во время итерации по view изменить словарь (добавить/удалить ключ), поведение может быть неопределённым в Py2 (итерация по копии списка); в Py3 view привязан к словарю и отражает изменения.

**Пример:**

```python
# Python 2: d.keys() — list, можно d.keys()[0]; в Py3 — view, индексации нет
# Python 3:
d = {"a": 1, "b": 2}
keys = d.keys()
d["c"] = 3
# keys теперь отражает и "c"; итерация по keys — три элемента
```

**Урок:** экономия памяти; единая модель «вид на текущее состояние». Подробнее — Часть I, §2.

---

#### Другие важные изменения (кратко)

- **Исключения:** в Py3 только `raise Exception(...)` (синтаксис `raise Exception, args` удалён); цепочка исключений через `raise ... from ...`; переменная в `except E as e:` удаляется в конце блока (экономия памяти); иерархия BaseException/Exception и т.д. (Часть I, §1.4).
- **Модули:** переименования (`urllib` → `urllib.request`, `ConfigParser` → `configparser`, `Queue` → `queue`, `thread` → `_thread` и др.); часть модулей удалена или объединена (справочник §7a).
- **range:** в Py3 `range` — отдельный тип с ленивым вычислением (аналог старого `xrange`); старого `range`, возвращающего список, нет. Код вида `range(10)[5]` в Py3 работает (range поддерживает индексацию); но `range(10).index(5)` и т.д. — через методы range.
- **input:** в Py2 была функция `raw_input()` (возвращает строку) и `input()` (выполняет ввод как выражение); в Py3 `input()` всегда возвращает строку, старого `input()` нет.

**Причины раскола:** накопленный технический долг (строки, деление, print, исключения, дублирование типов). Полная обратная совместимость сделала бы исправления невозможными, поэтому Python 3 пошёл на контролируемый разрыв: миграция требует правок кода и использования инструментов (2to3, six, `future` — см. план §35–37).

**Практический вывод:** план изучения (Часть I и далее) закрывает эти темы: строки/байты, print, деление, исключения, типы, итераторы, модули. Имеет смысл воспринимать всё уже в «мире Python 3» и при необходимости сверяться с PEP и гайдами по миграции.

### 0.1.6 Typing, async/await, match/case: почему так сделано

#### Типизация (typing) опциональна

В Python **аннотации типов** (**type hints**, **PEP 484**) не меняют поведение интерпретатора: он **не** проверяет типы во время выполнения. Проверка выполняется отдельными инструментами (mypy, pyright и т.д.) на этапе статического анализа и в IDE (автодополнение, подсказки).

**Следствия:**

- Существующий код без аннотаций продолжает работать без изменений; интерпретатор игнорирует аннотации (они доступны через `__annotations__`, но не влияют на выполнение).
- Типы можно вводить **постепенно**: файл за файлом, модуль за модулем; смешивание аннотированного и неаннотированного кода допустимо.
- Язык остаётся **динамическим**: типизация — инструмент для человека (документация, автодополнение) и для статического анализа (поиск ошибок до запуска), а не обязательная часть рантайма.

**Пример аннотаций:**

```python
def greet(name: str) -> str:
    return f"Hello, {name}"

def process(items: list[int], default: int | None = None) -> dict[str, int]:
    ...
```

Такой дизайн согласуется с Zen: **явность** (контракт функции виден в сигнатуре) без принудительного включения проверки для всего кода. Подробнее — в плане §9, §9a.

---

#### async/await вместо greenlets

Для асинхронности в стандартной библиотеке выбран синтаксис **async/await** (**PEP 492**) и единый цикл событий (**event loop**, модуль **asyncio**), а не библиотеки вроде **greenlets** (gevent и др.). Причины:

- **Явная маркировка:** `async def` и `await` показывают, что функция/вызов асинхронные; не нужно угадывать по контексту («explicit is better than implicit»). В greenlets любая функция может «переключиться» на другой greenlet неявно — отладка и чтение кода сложнее.
- **Один стандартный механизм** в stdlib (asyncio) — «one obvious way» для асинхронного I/O; не зависимость от сторонней библиотеки с собственной моделью.
- **Совместимость** с другими языками (C#, JavaScript, Rust и др.) и с существующими PEP; проще документировать и обучать.

**Пример (концептуально):**

```python
async def fetch(url: str) -> str:
    async with aiohttp.ClientSession() as session:
        async with session.get(url) as resp:
            return await resp.text()
```

Greenlets и другие корутин-библиотеки остаются возможностью для особых случаев (например, совместимость с блокирующим кодом без переписывания), но не входят в ядро языка. Подробнее — в плане §21 (asyncio).

---

#### Мотивация match/case (PEP 634)

Конструкция **match/case** (структурное сопоставление, **PEP 634**) добавлена для:

- **Читаемого разбора** сложных структур: вложенные структуры данных, типы, литералы, последовательности — без длинных цепочек `if/elif` и вложенного кода.
- **Плоского кода:** одна конструкция `match` заменяет несколько уровней вложенности; «flat is better than nested».
- **Единого места** для проверки и извлечения данных (pattern matching): и условие, и связывание переменных в одном месте; «explicit is better than implicit».

**Пример (концептуально):**

```python
match response:
    case {"status": 200, "data": data}:
        return data
    case {"status": 404}:
        raise NotFound()
    case _:
        raise Error("Unexpected response")
```

То есть и typing, и async/await, и match/case согласуются с Zen: явность, один очевидный способ, читаемость. Детали синтаксиса match/case (литералы, последовательности, guard, `as`, `|`) — в плане §18.
