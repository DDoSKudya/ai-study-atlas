## Краткое повторение терминологии

| Термин | Краткое определение |
|--------|----------------------|
| **Zen of Python** | Набор принципов дизайна языка (PEP 20); выводятся командой `import this`. |
| **PEP** | Python Enhancement Proposal — документ с предложением по изменению языка или stdlib; описывает мотивацию, альтернативы и последствия. |
| **GIL** | Global Interpreter Lock — блокировка в CPython; в каждый момент только один поток выполняет байткод. I/O и C-расширения могут освобождать GIL. |
| **CPU-bound / I/O-bound** | Нагрузка на процессор (вычисления) / на ввод-вывод (диск, сеть). Потоки в CPython полезны для I/O-bound, не ускоряют CPU-bound. |
| **BDFL** | Benevolent Dictator For Life — историческая роль Гвидо ван Россума; сейчас решения принимает Steering Council. |
| **Deprecation / removal** | Объявление устаревшим с предупреждением / фактическое удаление после периода deprecation. |
| **Name binding** | Модель «имена — ярлыки объектов»; присваивание привязывает имя к объекту, не копирует. Передача в функцию — передача ссылки. |
| **Тождество / равенство** | Тождество — «тот же объект» (`is`, `id()`). Равенство — «значения равны» (`==`, `__eq__`). Для `None` используют `is`. |
| **EAFP / LBYL** | EAFP: сначала действие, при ошибке — обработка исключения (try/except). LBYL: сначала проверка условия, потом действие. В Python чаще EAFP. |
| **TOCTOU** | Time Of Check To Time Of Use — гонка между проверкой и действием; EAFP уменьшает риск. |
| **Duck typing** | Использование объекта по наличию нужных методов/поведения, без проверки конкретного типа. |
| **Протокол** | Набор методов/поведения, достаточный для использования в определённом контексте (итерация, контекстный менеджер и т.д.). |
| **Structural subtyping** | Совместимость по структуре (наличие методов), а не по явному наследованию. |
| **Mutable / immutable** | Изменяемый / неизменяемый объект. Мутабельные: list, dict, set, bytearray. Неизменяемые: tuple, str, bytes, frozenset, int, float, bool. |
| **Hashable** | Объект с стабильным хешом, не меняется так, чтобы менялось равенство; пригоден как ключ словаря и элемент множества. |
| **Shallow copy / deep copy** | Поверхностная копия (новый объект верхнего уровня, элементы — те же ссылки) / глубокая копия (рекурсивно копируются вложенные объекты). |
| **Glue-код** | Код, соединяющий уже существующие компоненты (библиотеки, сервисы, скрипты на других языках), а не реализующий сложные алгоритмы с нуля. |

### 0.1 История решений и дизайн-философия

#### Zen of Python (PEP 20)

1. Как вывести Zen of Python в интерпретаторе? Назовите три принципа Zen и поясните их смысл на примере кода.  
   См. [0.1.1 Zen of Python (PEP 20)](#011-zen-of-python-pep-20).

   <details>
   <summary>Показать ответ</summary>

   В интерпретаторе: `import this`. Три принципа (примеры): Explicit is better than implicit — явные импорты, явный encoding при открытии файла; Readability counts — осмысленные имена переменных и функций; Flat is better than nested — ранний выход, comprehensions вместо глубокой вложенности.
   </details>

2. Какой принцип Zen выражается фразой «Explicit is better than implicit»? Приведите пример из кода: неявный vs явный вариант.  
   См. [0.1.1 Zen of Python (PEP 20)](#011-zen-of-python-pep-20).

   <details>
   <summary>Показать ответ</summary>

   «Явное лучше неявного». Пример: неявно — `open("file.txt")` (кодировка по умолчанию); явно — `open("file.txt", encoding="utf-8")`. Или явные импорты вместо `from module import *`.
   </details>

3. Как «Readability counts» связан с именованием переменных и функций? Приведите пример «нечитаемого» и «читаемого» фрагмента.  
   См. [0.1.1 Zen of Python (PEP 20)](#011-zen-of-python-pep-20).

   <details>
   <summary>Показать ответ</summary>

   Читаемость важна: имена должны объяснять назначение. Нечитаемо: `x`, `d`, `f(a,b)`. Читаемо: `user_count`, `config_dict`, `def parse_config(path):`.
   </details>

4. Что значит «Flat is better than nested»? Как достичь «плоского» кода (ранний выход, comprehensions, вынос в функции)?  
   См. [0.1.1 Zen of Python (PEP 20)](#011-zen-of-python-pep-20).

   <details>
   <summary>Показать ответ</summary>

   Плоская структура предпочтительнее глубокой вложенности. Способы: ранний return при ошибках/краевых случаях; списковые/генераторные comprehensions вместо вложенных циклов; вынос логики во вспомогательные функции.
   </details>

5. Как принцип «Errors should never pass silently. Unless explicitly silenced» применяется к обработке исключений? Почему голый `except: pass` — нарушение этого принципа?  
   См. [0.1.1 Zen of Python (PEP 20)](#011-zen-of-python-pep-20).

   <details>
   <summary>Показать ответ</summary>

   Ошибки не должны игнорироваться молча, если только явно не подавлены. Голый `except: pass` скрывает любую ошибку без логирования и без явного решения — нарушение принципа. Правильно: ловить конкретные исключения или явно логировать/перебрасывать.
   </details>

6. Что значит «There should be one—and preferably only one—obvious way to do it»? Приведите два примера «очевидного способа» в Python.  
   См. [0.1.1 Zen of Python (PEP 20)](#011-zen-of-python-pep-20).

   <details>
   <summary>Показать ответ</summary>

   Должен быть один (и желательно только один) очевидный способ сделать что-то. Примеры: обход коллекции — `for x in seq:`; значение по ключу с умолчанием — `d.get(key, default)`; открытие файла — `with open(...) as f:`.
   </details>

7. Как принцип «Namespaces are one honking great idea» связан с импортами и с избеганием `from module import *`?  
   См. [0.1.1 Zen of Python (PEP 20)](#011-zen-of-python-pep-20).

   <details>
   <summary>Показать ответ</summary>

   Пространства имён уменьшают конфликты и делают происхождение имён явным. `from module import *` засоряет текущее пространство имён и скрывает, откуда имя пришло; предпочтительны `import module` или `from module import name`.
   </details>

#### Эволюция через PEP

8. Что такое PEP? Какие типы PEP бывают (Standards Track, Informational, Process) и чем они отличаются?  
   См. [0.1.2 Эволюция через PEP](#012-эволюция-через-pep-консенсус-обратная-совместимость-deprecation-period).

   <details>
   <summary>Показать ответ</summary>

   PEP (Python Enhancement Proposal) — предложение по изменению или документированию Python. Типы: Standards Track (изменения языка/stdlib), Informational (руководства, без изменений), Process (процессы сообщества). Отличаются целью и процедурой принятия.
   </details>

9. Опишите упрощённо жизненный цикл PEP: от черновика до Final.  
   См. [0.1.2 Эволюция через PEP](#012-эволюция-через-pep-консенсус-обратная-совместимость-deprecation-period).

   <details>
   <summary>Показать ответ</summary>

   Draft → обсуждение → Accepted/Rejected/Withdrawn. Принятый PEP проходит стадии реализации; после включения в релиз статус может стать Final (для стандартов). Упрощённо: черновик → ревью → решение → реализация → фиксация.
   </details>

10. Зачем нужен период deprecation перед удалением возможности? Как в коде проявляется deprecation (предупреждение, флаги)?  
   См. [0.1.2 Эволюция через PEP](#012-эволюция-через-pep-консенсус-обратная-совместимость-deprecation-period).

   <details>
   <summary>Показать ответ</summary>

   Deprecation даёт время мигрировать код и не ломает существующие проекты сразу. В коде: warnings.warn(DeprecationWarning) при использовании устаревшего API; иногда флаги или переменные окружения для включения нового поведения.
   </details>

11. Что такое обратная совместимость и почему в Python к ней относятся серьёзно?  
   См. [0.1.2 Эволюция через PEP](#012-эволюция-через-pep-консенсус-обратная-совместимость-deprecation-period).

   <details>
   <summary>Показать ответ</summary>

   Старый код должен продолжать работать при обновлении версии. Серьёзное отношение сохраняет доверие экосистемы, позволяет постепенную миграцию и уменьшает издержки на обновление.
   </details>

12. Какие разделы PEP особенно полезно читать при изучении новой возможности (Motivation, Rationale и др.)?  
   См. [0.1.2 Эволюция через PEP](#012-эволюция-через-pep-консенсус-обратная-совместимость-deprecation-period).

   <details>
   <summary>Показать ответ</summary>

   Motivation — зачем нужно; Rationale — почему выбрано именно так; Specification — что именно меняется; Backwards Compatibility — влияние на старый код; иногда Rejected Ideas — почему не другие варианты.
   </details>

#### GIL, batteries included, one obvious way

13. Что такое GIL и почему в каждый момент только один поток выполняет байткод Python?  
   См. [0.1.3 Ключевые решения: GIL, batteries included](#013-ключевые-решения-gil-batteries-included-one-obvious-way).

   <details>
   <summary>Показать ответ</summary>

   GIL (Global Interpreter Lock) — глобальная блокировка в CPython: только один поток в данный момент выполняет байткод Python. Нужна для простоты и целостности внутренних структур (счётчики ссылок и т.д.).
   </details>

14. Почему четыре потока с CPU-bound задачей в чистом Python не дают примерно четырёхкратного ускорения?  
   См. [0.1.3 Ключевые решения: GIL, batteries included](#013-ключевые-решения-gil-batteries-included-one-obvious-way).

   <details>
   <summary>Показать ответ</summary>

   Из-за GIL в каждый момент выполняется только один поток; остальные ждут. CPU-bound код почти не освобождает GIL, поэтому ускорения нет. Для ускорения CPU-bound — multiprocessing или C-расширения, отпускающие GIL.
   </details>

15. Когда GIL освобождается (I/O, C-расширения)? Почему потоки полезны для I/O-bound задач?  
   См. [0.1.3 Ключевые решения: GIL, batteries included](#013-ключевые-решения-gil-batteries-included-one-obvious-way).

   <details>
   <summary>Показать ответ</summary>

   GIL освобождается при ожидании I/O (файлы, сеть) и в C-расширениях при явном отпускании. Потоки полезны для I/O-bound: пока один поток ждёт I/O, другой может выполнять код — реальный параллелизм по времени ожидания.
   </details>

16. Что такое «batteries included»? Назовите не менее пяти категорий модулей стандартной библиотеки.  
   См. [0.1.3 Ключевые решения: GIL, batteries included](#013-ключевые-решения-gil-batteries-included-one-obvious-way).

   <details>
   <summary>Показать ответ</summary>

   «Батарейки в комплекте» — богатая стандартная библиотека. Категории: io, os, pathlib; re, json, csv; urllib, http; sqlite3; threading, multiprocessing; unittest, doctest; logging; datetime, collections, itertools и др.
   </details>

17. Когда stdlib недостаточна и нужно подключать пакеты с PyPI? Приведите примеры.  
   См. [0.1.3 Ключевые решения: GIL, batteries included](#013-ключевые-решения-gil-batteries-included-one-obvious-way).

   <details>
   <summary>Показать ответ</summary>

   Когда нужна специализированная или более продвинутая функциональность: веб-фреймворки (Django, FastAPI), научные библиотеки (numpy, pandas), продвинутый HTTP (requests), ORM (SQLAlchemy), тесты (pytest) и т.д.
   </details>

18. Приведите примеры «одного очевидного способа» в Python: обход коллекции, открытие файла, проверка ключа в словаре, обработка ошибок.  
   См. [0.1.3 Ключевые решения: GIL, batteries included](#013-ключевые-решения-gil-batteries-included-one-obvious-way).

   <details>
   <summary>Показать ответ</summary>

   Обход — `for x in seq:`; файл — `with open(path) as f:`; значение по ключу с умолчанием — `d.get(key, default)`; обработка ошибок — try/except с конкретными исключениями, не голый `except: pass`.
   </details>

#### BDFL и Steering Council

19. Что такое BDFL и кого так называли? Как устроено принятие решений в Python сейчас (Steering Council)?  
   См. [0.1.4 BDFL и Steering Council](#014-bdfl-и-steering-council-роль-pep-в-архитектуре-языка).

   <details>
   <summary>Показать ответ</summary>

   BDFL (Benevolent Dictator For Life) — «доброжелательный диктатор на всю жизнь»; так называли Гвидо ван Россума. С 2018 года решения принимает Steering Council (несколько избранных разработчиков), ключевые изменения по-прежнему оформляются через PEP.
   </details>

20. Какую роль играют PEP в архитектуре языка (мотивация, альтернативы, обратная совместимость)?  
   См. [0.1.4 BDFL и Steering Council](#014-bdfl-и-steering-council-роль-pep-в-архитектуре-языка).

   <details>
   <summary>Показать ответ</summary>

   PEP фиксируют мотивацию, альтернативы и последствия; обеспечивают обратную совместимость через обсуждение и deprecation; служат единственным источником решений по языку и процессу.
   </details>

#### Python 2 vs 3

21. Чем в Python 3 тип `str` отличается от `str` в Python 2? Что такое `bytes` в Py3 и зачем разделение str/bytes?  
   См. [0.1.5 Python 2 vs 3](#015-python-2-vs-3-причины-раскола-уроки).

   <details>
   <summary>Показать ответ</summary>

   В Py2 str — байты, unicode — текст. В Py3 str — всегда Unicode (текст), bytes — неизменяемые байты. Разделение устраняет путаницу «байты vs текст» и явно задаёт кодировку на границах I/O.
   </details>

22. Почему в Python 3 `print` — функция, а не инструкция? Какие преимущества это даёт?  
   См. [0.1.5 Python 2 vs 3](#015-python-2-vs-3-причины-раскола-уроки).

   <details>
   <summary>Показать ответ</summary>

   Единообразие с остальным языком; можно передать print как аргумент, переопределить в тестах; удобнее вывод в файл: print(..., file=f); один способ вывода вместо специального синтаксиса.
   </details>

23. Как изменилось деление `/` и `//` при переходе с Python 2 на Python 3? Зачем явное целочисленное деление?  
   См. [0.1.5 Python 2 vs 3](#015-python-2-vs-3-причины-раскола-уроки).

   <details>
   <summary>Показать ответ</summary>

   В Py2 / — целочисленное для int. В Py3 / — всегда вещественное, // — явное целочисленное. Так поведение предсказуемо и не зависит от типов операндов; целочисленное деление явно через //.
   </details>

24. Что произошло с типами `int` и `long` в Python 3?  
   См. [0.1.5 Python 2 vs 3](#015-python-2-vs-3-причины-раскола-уроки).

   <details>
   <summary>Показать ответ</summary>

   В Py3 типа long нет: int объединяет прежние int и long, целые не ограничены по величине (ограничена только память).
   </details>

25. Чем в Python 3 отличаются методы `.keys()`, `.values()`, `.items()` словаря от Python 2 (списки vs view)?  
   См. [0.1.5 Python 2 vs 3](#015-python-2-вs-3-причины-раскола-уроки).

   <details>
   <summary>Показать ответ</summary>

   В Py2 они возвращали списки (копии). В Py3 — view-объекты, отражающие текущее состояние словаря; итерация без копирования, изменение словаря во время итерации по view может влиять на обход.
   </details>

26. Назовите другие важные изменения при переходе на Python 3 (исключения, модули, range, input).  
   См. [0.1.5 Python 2 vs 3](#015-python-2-vs-3-причины-раскола-уроки).

   <details>
   <summary>Показать ответ</summary>

   Синтаксис except: `except E as e:` вместо `except E, e:`; переименования модулей (urllib, tkinter); range — как прежний xrange; input() — как прежний raw_input(); все строки Unicode по умолчанию.
   </details>

#### Typing, async/await, match/case

27. Почему аннотации типов (type hints) в Python опциональны и не проверяются интерпретатором во время выполнения?  
   См. [0.1.6 Typing, async/await, match/case](#016-typing-asyncawait-matchcase-почему-так-сделано).

   <details>
   <summary>Показать ответ</summary>

   Чтобы не ломать существующий код и не принуждать к типизации; проверка типов вынесена в отдельные инструменты (mypy, pyright). «Duck typing» и динамика остаются; аннотации — для людей и статического анализа.
   </details>

28. Почему для асинхронности в stdlib выбран async/await, а не greenlets? Какие преимущества явной маркировки async/await?  
   См. [0.1.6 Typing, async/await, match/case](#016-typing-asyncawait-matchcase-почему-так-сделано).

   <details>
   <summary>Показать ответ</summary>

   Явная маркировка: видно, где возможна приостановка; легче отлаживать и избегать случайного вызова корутины без await. Интеграция с типами и единый стиль с другими языками.
   </details>

29. Для чего добавлена конструкция match/case (PEP 634)? Как она связана с принципами Zen (плоский код, явность)?  
   См. [0.1.6 Typing, async/await, match/case](#016-typing-asyncawait-matchcase-почему-так-сделано).

   <details>
   <summary>Показать ответ</summary>

   Сопоставление с образцом: разбор структур данных, ветвление по форме значения. Плоский код — цепочка case вместо вложенных if; явность — паттерны показывают, что именно проверяется.
   </details>

30. В каких областях Python обычно подходит? Перечислите не менее пяти и кратко поясните «почему».

31. Почему Python удобен для прототипирования? Что обычно делают с прототипом после проверки идеи?  
   См. [0.2.1 Когда Python подходит](#021-когда-python-подходит).

37. В каких ситуациях Python — не лучший выбор? Перечислите не менее четырёх и поясните ограничения.

   </details>

#### Второй язык в стеке, экосистема, ограничения

42. Чем subprocess отличается от FFI? Когда что использовать?

   <summary>Показать ответ</summary>

   Subprocess — запуск отдельного процесса (программа, скрипт), обмен через stdin/stdout/файлы. FFI — вызов кода из того же процесса (C-библиотека). Subprocess для утилит и изоляции; FFI для плотной интеграции и скорости.
   </details>

43. Когда уместны ctypes, cffi, pybind11? В чём разница между ними?  
   См. [0.2.3 Python как «второй язык» в стеке](#023-python-как-второй-язык-в-стеке-subprocess-ffi-cython).

   </details>

44. Для чего используется Cython? В чём типичная схема: основная логика на Python, горячие участки на Cython?  
   См. [0.2.3 Python как «второй язык» в стеке](#023-python-как-второй-язык-в-стеке-subprocess-ffi-cython).

   Ускорение горячих циклов и обёртки над C/C++. Схема: большая часть кода на Python, критичные циклы переписаны на Cython (типы, отпускание GIL при необходимости), компиляция в расширение и использование из Python.
   </details>

46. Что регламентирует PEP 668 в отношении системного Python и pip?

### 0.3 Ментальные модели

#### Everything is an object, тождество и равенство

54. В чём разница между тождеством (`is`) и равенством (`==`)? Что возвращает `id()`?  
   См. [0.3.1 «Everything is an object»](#031-everything-is-an-object-ссылки-тождество-is-vs-равенство-id).

   <details>
   <summary>Показать ответ</summary>

   `is` — один и тот же объект в памяти (тождество); `==` — равные значения (равенство). `id()` возвращает идентификатор объекта (в CPython — адрес в памяти). Два разных объекта могут быть равны по `==`, но не по `is`.
   </details>

55. Для чего принято использовать `is` (приведите пример)? Почему для сравнения чисел не используют `is`?  
   См. [0.3.1 «Everything is an object»](#031-everything-is-an-object-ссылки-тождество-is-vs-равенство-id).

   <details>
   <summary>Показать ответ</summary>

   `is` уместен для синглтонов: `if x is None:`, `if obj is sentinel:`. Для чисел не используют: маленькие целые кэшируются, но полагаться на это — деталь реализации; семантика «равны по значению» выражается через `==`.
   </details>

56. Что такое кэширование маленьких целых в CPython и почему из-за него нельзя полагаться на `is` для чисел?  
   См. [0.3.1 «Everything is an object»](#031-everything-is-an-object-ссылки-тождество-is-vs-равенство-id).

   <details>
   <summary>Показать ответ</summary>

   CPython переиспользует объекты для небольших целых (например, от -5 до 256). Один и тот же объект — один id. Полагаться нельзя: это деталь реализации; числа сравнивают по значению через `==`.
   </details>

57. Что такое интернирование строк и почему для строк тоже предпочтительно `==`?  
   См. [0.3.1 «Everything is an object»](#031-everything-is-an-object-ссылки-тождество-is-vs-равенство-id).

   <details>
   <summary>Показать ответ</summary>

   Интернирование — хранение одной копии строковых литералов и некоторых строк. Одинаковые строки могут быть одним объектом, но не обязаны. Сравнивать строки по смыслу нужно через `==`, не через `is`.
   </details>

58. Приведите пример: функция и класс как объекты (присвоение имени, передача в другую функцию, хранение в словаре).  
   См. [0.3.1 «Everything is an object»](#031-everything-is-an-object-ссылки-тождество-is-vs-равенство-id).

   <details>
   <summary>Показать ответ</summary>

   `f = my_func`; `callable(f)`; передача в map: `map(str, items)`; хранение в словаре: `handlers = {"get": get_handler}`; класс как объект: `C = MyClass`, `obj = C()`, передача класса в функцию.
   </details>

#### Name binding

59. Что такое name binding? Почему в Python говорят «имя», а не «переменная»?  
   См. [0.3.2 Name binding](#032-name-binding-имена--ярлыки-объектов-присваивание-не-копирует-a--b--не-переменная).

   <details>
   <summary>Показать ответ</summary>

   Name binding — связывание имени с объектом (присваивание, параметры, импорт). Говорят «имя», потому что имя — ярлык объекта, а не «ящик с значением»; один объект может иметь несколько имён.
   </details>

60. Почему присваивание `a = b` не копирует объект? Что произойдёт с объектом, если через одно из имён его изменить?  
   См. [0.3.2 Name binding](#032-name-binding-имена--ярлыки-объектов-присваивание-не-копирует-a--b--не-переменная).

   <details>
   <summary>Показать ответ</summary>

   Присваивание привязывает имя к тому же объекту. Копирования нет. Если объект мутабельный и изменить его через одно имя (например, a.append(1)), изменение видно по всем именам (b тоже «увидит» изменение).
   </details>

61. К чему приводит передача мутабельного аргумента в функцию (список, словарь)? Как избежать неожиданного изменения снаружи?  
   См. [0.3.2 Name binding](#032-name-binding-имена--ярлыки-объектов-присваивание-не-копирует-a--b--не-переменная).

   <details>
   <summary>Показать ответ</summary>

   В функцию передаётся ссылка на тот же объект. Изменение содержимого (append, pop, ключи словаря) меняет объект снаружи. Избежать: не мутировать входящий аргумент; при необходимости копировать (copy/deepcopy) или документировать побочный эффект.
   </details>

62. Почему присваивание параметру нового объекта внутри функции не меняет внешнюю переменную?  
   См. [0.3.2 Name binding](#032-name-binding-имена--ярлыки-объектов-присваивание-не-копирует-a--b--не-переменная).

   <details>
   <summary>Показать ответ</summary>

   Присваивание перепривязывает локальное имя параметра к новому объекту; связь с тем объектом, на который указывала внешняя переменная, не меняется. Внешнее имя по-прежнему привязано к старому объекту.
   </details>

63. В чём опасность изменяемого аргумента по умолчанию (`def f(x=[])`)? Как правильно оформить такую функцию?  
   См. [0.3.2 Name binding](#032-name-binding-имена--ярлыки-объектов-присваивание-не-копирует-a--b--не-переменная).

   <details>
   <summary>Показать ответ</summary>

   Объект по умолчанию создаётся один раз при определении функции; все вызовы без аргумента используют один и тот же объект. Мутация накапливается между вызовами. Правильно: `def f(x=None): x = [] если x is None else x` и затем работать с этим списком.
   </details>

#### Мутабельность и неизменяемость

64. Какие встроенные типы мутабельны, какие неизменяемы? Приведите таблицу или перечислите.  
   См. [0.3.3 Мутабельность vs неизменяемость](#033-мутабельность-vs-неизменяемость-listdictset-vs-tuplestrfrozenset-последствия-для-передачи-и-сравнения).

   <details>
   <summary>Показать ответ</summary>

   Мутабельные: list, dict, set, bytearray. Неизменяемые: tuple, str, bytes, frozenset, int, float, bool. Неизменяемые можно использовать как ключи и элементы множества (hashable).
   </details>

65. Почему ключ словаря и элемент множества должны быть hashable? Какие типы hashable, какие нет?  
   См. [0.3.3 Мутабельность vs неизменяемость](#033-мутабельность-vs-неизменяемость-listdictset-vs-tuplestrfrozenset-последствия-для-передачи-и-сравнения).

   <details>
   <summary>Показать ответ</summary>

   Словарь и множество опираются на хеш для быстрого поиска; если объект изменится после добавления, хеш и равенство станут некорректными. Hashable: неизменяемые (str, int, tuple, frozenset). Не hashable: list, dict, set.
   </details>

66. Можно ли изменить элемент кортежа, если этот элемент — список? Поясните на примере.  
   См. [0.3.3 Мутабельность vs неизменяемость](#033-мутабельность-vs-неизменяемость-listdictset-vs-tuplestrfrozenset-последствия-для-передачи-и-сравнения).

   <details>
   <summary>Показать ответ</summary>

   Сам кортеж неизменяем (нельзя присвоить t[0] = что-то другое). Но если t[0] — список, содержимое этого списка менять можно: t[0].append(1). Неизменяемость кортежа — только «набор ссылок», не глубже.
   </details>

67. В чём разница между shallow copy и deep copy? Когда нужен каждый из них? Приведите пример с вложенным списком.  
   См. [0.3.3 Мутабельность vs неизменяемость](#033-мутабельность-vs-неизменяемость-listdictset-vs-tuplestrfrozenset-последствия-для-передачи-и-сравнения).

   <details>
   <summary>Показать ответ</summary>

   Shallow copy — новый объект верхнего уровня, элементы — те же ссылки. Deep copy — рекурсивно копируются вложенные объекты. Shallow: когда вложенные объекты не нужно копировать. Deep: когда нужна полная независимая копия (вложенный список — те же объекты при shallow, при изменении b[0].append(1) изменится и a[0]).
   </details>

68. Какие последствия мутабельности/неизменяемости для передачи в функции и для потокобезопасности?  
   См. [0.3.3 Мутабельность vs неизменяемость](#033-мутабельность-vs-неизменяемость-listdictset-vs-tuplestrfrozenset-последствия-для-передачи-и-сравнения).

   <details>
   <summary>Показать ответ</summary>

   Мутабельный объект, переданный в функцию, может быть изменён внутри — побочный эффект снаружи. Общий мутабельный объект между потоками — гонки, нужна синхронизация. Неизменяемые объекты безопаснее передавать и использовать в многопоточности (нет изменения «на месте»).
   </details>

#### EAFP vs LBYL

69. Что такое EAFP и LBYL? Расшифруйте аббревиатуры и поясните смысл.  
   См. [0.3.4 EAFP vs LBYL](#034-eafp-vs-lbyl-easier-to-ask-forgiveness-than-permission).

   <details>
   <summary>Показать ответ</summary>

   EAFP — Easier to Ask Forgiveness than Permission: сначала действие, при ошибке — обработка. LBYL — Look Before You Leap: сначала проверка, потом действие. В Python чаще EAFP: меньше гонок, один доступ к ресурсу.
   </details>

70. Почему в Python чаще предпочитают EAFP? Приведите пример (доступ к ключу словаря, открытие файла).  
   См. [0.3.4 EAFP vs LBYL](#034-eafp-vs-lbyl-easier-to-ask-forgiveness-than-permission).

   <details>
   <summary>Показать ответ</summary>

   Меньше TOCTOU (между проверкой и действием состояние может измениться); один доступ к данным. Пример: `try: x = d[key]; except KeyError: x = default` или `d.get(key, default)`; файл — `try: f = open(path); except FileNotFoundError: ...`.
   </details>

71. Что такое гонка TOCTOU? Почему EAFP уменьшает этот риск? Приведите пример с словарём или файлом.  
   См. [0.3.4 EAFP vs LBYL](#034-eafp-vs-lbyl-easier-to-ask-forgiveness-than-permission).

   <details>
   <summary>Показать ответ</summary>

   TOCTOU — Time Of Check To Time Of Use: между проверкой («ключ есть») и использованием другой поток может удалить ключ. При EAFP проверка и использование совмещены (одно обращение); при LBYL — два шага, окно для гонки.
   </details>

72. В каких случаях уместен LBYL (дёшевая проверка, явное разделение веток)?  
   См. [0.3.4 EAFP vs LBYL](#034-eafp-vs-lbyl-easier-to-ask-forgiveness-than-permission).

   <details>
   <summary>Показать ответ</summary>

   Когда проверка дёшевая и явное разделение веток улучшает читаемость; когда исключение ожидаемо и часто (и проверка быстрее); когда нужна разная логика «есть/нет» без try/except.
   </details>

73. Почему голый `except: pass` — плохая практика? Как обрабатывать исключения правильно?  
   См. [0.3.4 EAFP vs LBYL](#034-eafp-vs-lbyl-easier-to-ask-forgiveness-than-permission).

   <details>
   <summary>Показать ответ</summary>

   Скрывает любые ошибки, включая KeyboardInterrupt и SystemExit; затрудняет отладку. Правильно: ловить конкретные исключения (`except KeyError:`), при необходимости логировать или перебрасывать; не подавлять молча без явного решения.
   </details>

#### Протоколы и duck typing

74. Что такое duck typing? Сформулируйте «принцип утки».  
   См. [0.3.5 Протоколы и утиная типизация](#035-протоколы-и-утиная-типизация-duck-typing).

   <details>
   <summary>Показать ответ</summary>

   «If it walks like a duck and quacks like a duck, it’s a duck» — важнее поведение (наличие методов/возможностей), а не объявленный тип. Не проверяем isinstance, а пробуем использовать и обрабатываем исключения при необходимости.
   </details>

75. Что такое протокол в Python? Должен ли класс наследоваться от протокола, чтобы его использовать?  
   См. [0.3.5 Протоколы и утиная типизация](#035-протоколы-и-утиная-типизация-duck-typing).

   <details>
   <summary>Показать ответ</summary>

   Протокол — неформальный или формальный (typing.Protocol) набор методов/поведения. Наследоваться не обязательно: достаточно реализовать нужные методы (structural subtyping). Пример: итерабельный — __iter__ или __getitem__.
   </details>

76. Перечислите не менее четырёх встроенных протоколов (итерация, длина, контекстный менеджер и т.д.).  
   См. [0.3.5 Протоколы и утиная типизация](#035-протоколы-и-утиная-типизация-duck-typing).

   <details>
   <summary>Показать ответ</summary>

   Итерация (__iter__, __next__ или __getitem__), длина (__len__), контекстный менеджер (__enter__, __exit__), контейнер (__getitem__, __contains__), сравнение (__eq__, __lt__ и др.), вызов (__call__).
   </details>

77. Приведите пример функции, работающей с любым итерабельным объектом без проверки типа. С какими типами она будет работать (list, tuple, str, range, файл)?  
   См. [0.3.5 Протоколы и утиная типизация](#035-протоколы-и-утиная-типизация-duck-typing).

   <details>
   <summary>Показать ответ</summary>

   `def sum_len(seq): return sum(1 for _ in seq)`. Работает с list, tuple, str, range, открытым файлом, множеством, словарём (итерация по ключам) — со всем, у чего есть __iter__ или __getitem__.
   </details>

78. Что такое typing.Protocol и зачем он нужен (аннотации, structural subtyping)?  
   См. [0.3.5 Протоколы и утиная типизация](#035-протоколы-и-утиная-типизация-duck-typing).

   <details>
   <summary>Показать ответ</summary>

   Protocol описывает интерфейс (методы с сигнатурами) для статической проверки типов. Structural subtyping: класс совместим с протоколом, если реализует нужные методы, без явного наследования. Нужен для аннотаций и mypy/pyright.
   </details>

#### Явное лучше неявного, плоское лучше вложенного, один способ

79. Как принцип «явное лучше неявного» проявляется в импортах? Почему избегают `from module import *`?

80. Как «явное лучше неявного» связано с аннотациями типов и с явными аргументами функций?

   Аннотации типов явно задают ожидаемые типы; явный return None вместо «ничего не вернуть»; явные аргументы по смыслу (не полагаться на позицию по умолчанию, когда важна ясность).

81. Как достичь «плоского» кода вместо глубокой вложенности (ранний выход, comprehensions, вынос в функции)?

82. Когда comprehensions могут навредить читаемости? Что делать в таком случае?

   </details>

83. Приведите примеры «одного очевидного способа» в Python для обхода коллекции, открытия файла, проверки ключа, обработки ошибок. Когда допустимы альтернативы (while, ручной итератор, LBYL)?

```python
def double(iterable):

```

**Задание 20.** Перепишите доступ к ключу словаря в стиле EAFP: если ключа нет — вернуть None. Затем перепишите то же самое одним вызовом метода словаря (один очевидный способ).

**Задание 21.** Исправьте функцию так, чтобы она не имела побочного эффекта «накопления» в списке по умолчанию:  
См. [0.3.2 Name binding, изменяемый аргумент по умолчанию](#032-name-binding-имена--ярлыки-объектов-присваивание-не-копирует-a--b--не-переменная).

<details>
<summary>Показать ответ</summary>

```python
def add_value(value, data=None):
```

**Задание 22.** Напишите фрагмент кода, который создаёт shallow copy и deep copy вложенного списка `[[1, 2], [3, 4]]`, затем изменяет один из внутренних списков в копии. Покажите, что в одном случае исходный список изменился, в другом — нет.  
См. [0.3.3 Мутабельность, shallow vs deep copy](#033-мутабельность-vs-неизменяемость-listdictset-vs-tuplestrfrozenset-последствия-для-передачи-и-сравнения).

**Задание 23.** Приведите пример кода, где проверка «тот же объект» (is) уместна, и пример, где нужно сравнивать значения (==). В комментарии укажите, почему выбран именно такой способ.
