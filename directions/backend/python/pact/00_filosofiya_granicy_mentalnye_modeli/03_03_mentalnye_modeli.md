## 0.3 Ментальные модели: как думать на Python

Ментальные модели — это способы мысленно представлять, как устроен язык. Они помогают предсказывать поведение кода и избегать типичных ошибок. В этом разделе каждая модель разобрана подробно: определения, примеры, краевые случаи и связь с другими идеями.

---

### 0.3.1 «Everything is an object»: ссылки, тождество (`is`) vs равенство (`==`), `id()`

В Python всё, с чем вы работаете (числа, строки, функции, классы, модули, сам тип `type`), — **объекты**. «Переменные» в привычном смысле (ячейки памяти с значением) отсутствуют: есть **имена**, которые **ссылаются на объекты**. Присваивание не «записывает значение в ячейку», а **привязывает имя к объекту**.

---

#### Следствия модели «всё — объект»

1. При присваивании `a = b` имя `a` начинает ссылаться на **тот же объект**, на который ссылается `b`. Объект не копируется; копий может быть несколько имён, ссылающихся на один объект.
2. **Тождество** — «это один и тот же объект в памяти?» Проверяется оператором **`is`** и функцией **`id()`** (уникальный идентификатор объекта; в CPython это обычно адрес в памяти). Два объекта с одинаковым содержимым могут быть разными объектами (`a is not b`), но при этом равными по значению (`a == b`).
3. **Равенство** — «значения считаются равными?» Проверяется оператором **`==`** (вызывается метод `__eq__` у объекта). Классы могут переопределять равенство; по умолчанию для своих типов сравниваются по содержимому или по идентичности (зависит от типа).

---

#### Примеры: тождество и равенство

```python
# Тождество (is) vs равенство (==)
a = [1, 2, 3]
b = [1, 2, 3]
print(a == b)   # True  — содержимое одинаковое
print(a is b)   # False — это два разных объекта в памяти

c = a
print(a is c)   # True  — c ссылается на тот же список, что и a

# id() — уникальный идентификатор объекта (в CPython — адрес в памяти)
print(id(a))
print(id(c))    # тот же id, что и у a
print(id(b))    # другой id
```

**Правило для синглтонов:** для `None`, `True`, `False` принято использовать **`is`**, а не `==`:

```python
if value is None:
    ...
if result is not None:
    ...
```

Причины: (1) явно подчёркивается проверка «именно этот единственный объект»; (2) для синглтонов `is` быстрее и не зависит от переопределения `__eq__`; (3) так принято в сообществе и в стиле (PEP 8).

---

#### Что именно является объектом

- **Данные:** числа, строки, списки, словари, множества, кортежи, bytes, range и т.д. — всё объекты.
- **Функции и методы:** функция, определённая через `def`, — объект (её можно присвоить другому имени, передать в другую функцию, положить в список).
- **Классы и типы:** класс, определённый через `class`, — объект (тип); сам `type` — тоже объект (метакласс).
- **Модули:** результат `import module` — объект модуля; у него есть атрибуты (имена, определённые в модуле).
- **Даже «синтаксис»:** `None`, `True`, `False`, `Ellipsis` (`...`) — объекты-синглтоны; декораторы `@staticmethod`, `@property` и т.д. — вызываемые объекты.

Понимание «всё — объект» помогает не удивляться тому, что функции можно передавать в другие функции, классы можно хранить в словаре, а модуль — это просто объект с атрибутами.

**Пример: функция и класс как объекты**

```python
# Функция — объект: можно присвоить другому имени, передать в другую функцию
def greet(name):
    return f"Hello, {name}"

say_hi = greet          # say_hi ссылается на ту же функцию
print(say_hi("World"))  # Hello, World

# Функция как аргумент (например, key для sorted)
items = [("b", 2), ("a", 1)]
sorted_by_letter = sorted(items, key=lambda x: x[0])

# Класс — объект (тип): можно хранить в словаре, вызывать динамически
handlers = {"json": json.loads, "str": str}
result = handlers["json"]('{"a": 1}')
```

#### Краевой случай: кэширование маленьких целых и интернирование строк

В CPython для оптимизации **маленькие целые числа** (в типичном диапазоне примерно от -5 до 256) **кэшируются**: один и тот же объект `int` переиспользуется. Поэтому для таких чисел `a is b` может быть `True`, даже если вы получили их разными путями.

**Пример (интерактивно проверить в Python):**

```python
a = 100
b = 50 + 50
print(a is b)   # True — один и тот же кэшированный объект
print(id(a), id(b))  # одинаковые id

x = 1000
y = 1000
print(x is y)   # Может быть True или False в зависимости от контекста и версии!
# В интерактивной сессии часто True (один литерал 1000); в цикле или разных строках — может быть False.
# Вывод: для чисел никогда не полагайтесь на is — только ==.
```

**Не полагайтесь на `is` для сравнения чисел** — для значений всегда используйте `==`; `is` — только для синглтонов (`None`, иногда булевы) и для проверки «тот же объект» (например, при кэшировании по объекту). 

**Интернирование строк** (interning): некоторые строки (например, короткие строковые литералы, совпадающие с идентификаторами) могут храниться в одном экземпляре. Поведение зависит от реализации и версии; не стоит полагаться на `is` для строк — для сравнения содержимого используйте `==`.

**Вывод:** используйте `is` для `None`, `True`, `False` и когда действительно нужна проверка «тот же объект» (например, при кэшировании по объекту). Для сравнения значений — всегда `==`.

---

**Терминология:**

- **Объект** — сущность в памяти с типом, идентичностью и опционально изменяемым состоянием.
- **Ссылка** — привязка имени (или элемента структуры, например элемента списка) к объекту.
- **Тождество** — совпадение идентификатора объекта (`is`, `id()`).
- **Равенство** — семантическое равенство значений (`==`, метод `__eq__`).

### 0.3.2 Name binding: имена — ярлыки объектов; присваивание не копирует; `a = b` — не «переменная»

**Name binding** (привязка имени) — модель, в которой **имена** — это ярлыки, привязанные к объектам. Присваивание `a = b` не «копирует значение в переменную», а **привязывает имя `a` к тому же объекту**, на который ссылается `b`. В Python часто говорят не «переменная», а **«имя» (name)** или **«именованная ссылка»**, чтобы не путать с «ячейкой памяти с значением».

---

#### Цепочка присваиваний и несколько имён на один объект

Несколько имён могут ссылаться на один и тот же объект. Изменение объекта через одно имя видно через все остальные имена:

```python
x = [1, 2]
y = x          # y привязан к тому же списку
y.append(3)
print(x)       # [1, 2, 3] — изменился «тот же» список
print(x is y)  # True
```

Чтобы получить **отдельный** список, нужно создать новый объект (копию):

```python
x = [1, 2]
y = list(x)    # или x.copy(), или x[:]
y.append(3)
print(x)       # [1, 2] — x не изменился
print(y)       # [1, 2, 3]
print(x is y)  # False
```

---

#### Передача в функции: передаётся ссылка на объект

При вызове функции аргументы не копируются: в параметр попадает **ссылка на тот же объект**. Поэтому если функция изменяет мутабельный объект (список, словарь и т.д.), изменения видны снаружи:

```python
def append_zero(lst):
    lst.append(0)   # меняем тот же объект

items = [1, 2, 3]
append_zero(items)
print(items)   # [1, 2, 3, 0]
```

Если нужно не менять исходный список, внутри функции можно работать с копией (`lst = list(lst)` или `lst = lst.copy()`) или явно документировать, что функция изменяет аргумент.

**Присваивание параметру другого объекта** не меняет внешнюю переменную: вы просто привязываете локальное имя к новому объекту:

```python
def replace(lst):
    lst = [0, 0]   # lst теперь ссылается на новый список; внешний items не меняется

items = [1, 2, 3]
replace(items)
print(items)   # [1, 2, 3]
```

---

#### Ловушка: изменяемый аргумент по умолчанию

Из-за того что присваивание не копирует, **объект по умолчанию** в `def f(x=[])` создаётся один раз при определении функции и переиспользуется при каждом вызове без аргумента. Поэтому изменяемый аргумент по умолчанию — частая ошибка:

```python
def add_item(item, lst=[]):   # плохо: один и тот же список для всех вызовов
    lst.append(item)
    return lst

print(add_item(1))   # [1]
print(add_item(2))   # [1, 2] — тот же список!
```

Правильно — использовать неизменяемое значение по умолчанию и создавать новый список внутри функции:

```python
def add_item(item, lst=None):
    if lst is None:
        lst = []
    lst.append(item)
    return lst
```

Это следствие модели name binding: значение по умолчанию — это один объект, привязанный к параметру при определении функции; он не «пересоздаётся» при каждом вызове.

---

**Практический вывод:** при присваивании копирования нет. Если нужна копия изменяемого объекта — создавайте её явно (`list()`, `dict.copy()`, `copy.copy()`, `copy.deepcopy()` в зависимости от задачи). При передаче в функцию передаётся ссылка; мутабельные аргументы по умолчанию — антипаттерн.

### 0.3.3 Мутабельность vs неизменяемость: list/dict/set vs tuple/str/frozenset; последствия для передачи и сравнения

**Мутабельный (mutable)** объект можно изменить «на месте» (добавить/удалить элементы, изменить содержимое) без создания нового объекта.  
**Неизменяемый (immutable)** объект после создания не меняется; операции, которые «меняют» значение (например, конкатенация строки), **возвращают новый объект**; старый остаётся как был.

---

#### Таблица: мутабельные и неизменяемые типы

| Мутабельные | Неизменяемые |
|-------------|--------------|
| `list`, `dict`, `set`, `bytearray` | `tuple`, `str`, `bytes`, `frozenset`, `int`, `float`, `bool`, `range` |

Правило большого пальца: если тип позволяет изменить содержимое «на месте» (append, pop, присваивание по индексу/ключу без создания нового объекта), он мутабельный. Если таких операций нет или они возвращают новый объект — неизменяемый.

---

#### Примеры: список (мутабельный) и строка (неизменяемая)

```python
# Список — мутабельный
lst = [1, 2, 3]
lst.append(4)      # изменили тот же объект
lst[0] = 0         # тот же объект
# id(lst) не меняется после этих операций

# Строка — неизменяемая
s = "hello"
s.upper()          # возвращает новую строку "HELLO", s не меняется
# s[0] = "H"       # TypeError: 'str' object does not support item assignment
s = s + "!"        # создаётся новая строка, имя s привязывается к ней; старый объект "hello" больше не используется (если на него нет других ссылок)
```

Кортеж (`tuple`) неизменяем: нельзя сделать `t[0] = 1` или `t.append(1)`. Но если элемент кортежа сам мутабельный (например, список), содержимое этого элемента менять можно:

```python
t = (1, [2, 3])
# t[0] = 0        # TypeError
t[1].append(4)     # можно: меняем внутренний список
# t теперь (1, [2, 3, 4])
```

Неизменяемость кортежа касается только «верхнего уровня»: сам кортеж не заменяет свои элементы на другие объекты, но объекты внутри могут быть мутабельными.

---

#### Копирование: shallow vs deep

**Shallow copy** (поверхностная копия) создаёт новый объект верхнего уровня, но элементы остаются теми же (те же ссылки). Для списка: `list(orig)`, `orig[:]`, `orig.copy()`.

**Deep copy** (глубокая копия) рекурсивно копирует все вложенные объекты; в результате получаются полностью независимые копии. В стандартной библиотеке: `copy.deepcopy(obj)`.

Разница важна при вложенных структурах:

```python
import copy
a = [[1, 2], [3, 4]]
b = a.copy()           # shallow
b[0].append(99)
print(a[0])            # [1, 2, 99] — внутренний список тот же!

c = copy.deepcopy(a)
c[0].append(100)
print(a[0])            # [1, 2, 99] — a не изменился
```

Если нужна полная независимость вложенных структур — используйте `copy.deepcopy()`. Для «плоских» списков/словарей достаточно shallow copy.

---

#### Hashable: ключи словаря и элементы множества

Ключи словаря и элементы множества должны быть **hashable**: объект должен иметь стабильный хеш (`hash(obj)`), и он не должен меняться за время жизни (иначе нарушится инвариант структуры данных). Неизменяемые встроенные типы (`str`, `int`, `float`, `tuple`, `frozenset`, `bytes`) обычно hashable. Мутабельные (`list`, `dict`, `set`) — не hashable: их нельзя использовать как ключ словаря или положить в set.

```python
d = {}
d[[1, 2]] = 3   # TypeError: unhashable type: 'list'
d[(1, 2)] = 3   # OK: tuple неизменяем и hashable
s = {[1, 2]}    # TypeError: unhashable type: 'list'
s = {(1, 2)}    # OK
```

Пользовательские классы по умолчанию hashable (хеш по id), но при переопределении `__eq__` нужно согласованно переопределять `__hash__` (и объект не должен меняться способом, влияющим на равенство). Подробнее — в плане (§52–57, хеширование).

---

#### Последствия для передачи в функции и потоков

1. **Передача в функции.** В функцию передаётся ссылка. Мутабельный объект функция может изменить — это будет видно снаружи. Неизменяемый объект изменить нельзя; можно только привязать локальное имя к новому объекту, не затрагивая внешнее имя. Поэтому если функция не должна менять переданный список/словарь, внутри нужно работать с копией или явно документировать, что функция изменяет аргумент.
2. **Потокобезопасность.** Неизменяемые объекты безопаснее передавать между потоками: никто не может изменить их «сбоку»; чтение из нескольких потоков безопасно. Мутабельные объекты при совместном доступе из нескольких потоков требуют **синхронизации** (блокировки, очереди, потокобезопасные структуры), иначе возможны гонки и порча данных. Пример: общий словарь, в который пишут два потока без блокировки — классический источник неопределённого поведения; решение — использовать `threading.Lock` или потокобезопасную структуру (например, `queue.Queue` для обмена данными).

**Термины:**

- **Hashable** — объект имеет стабильный хеш (`hash()`), не меняется так, чтобы менялось равенство, и может использоваться как ключ словаря и элемент множества.
- **Mutable / immutable** — изменяемый / неизменяемый.
- **Shallow copy / deep copy** — поверхностная / глубокая копия.

### 0.3.4 EAFP vs LBYL: «Easier to Ask Forgiveness than Permission»

**EAFP** (Easier to Ask Forgiveness than Permission) — «проще попросить прощения, чем разрешения»: сначала выполняем действие, при ошибке обрабатываем исключение (`try/except`).  
**LBYL** (Look Before You Leap) — «сначала посмотри, потом прыгай»: сначала проверяем условия (есть ли ключ, подходящий ли тип), потом действуем (`if` проверки).

В Python принято предпочитать **EAFP**: операции выполняются в предположении, что условия выполнены, а нарушение обрабатывается через `try/except`. Причины: (1) меньше гонок — между проверкой и действием состояние может измениться (TOCTOU); (2) код часто короче и яснее; (3) одна точка входа в «нормальный» путь и одна — в обработку ошибки.

---

#### Примеры: доступ к словарю

**LBYL:**

```python
if "key" in d:
    value = d["key"]
else:
    value = None
```

**EAFP:**

```python
try:
    value = d["key"]
except KeyError:
    value = None
```

Для словаря ещё проще встроенный метод: `value = d.get("key", None)` или `value = d.get("key")`. Но для более сложных операций (открытие файла, парсинг, вызов API) типичный питоновский стиль — `try/except` вокруг основной операции:

```python
try:
    with open(path, encoding="utf-8") as f:
        data = f.read()
except FileNotFoundError:
    data = ""
except OSError as e:
    logger.error("Cannot read %s: %s", path, e)
    raise
```

---

#### Гонка TOCTOU (Time Of Check To Time Of Use)

При LBYL между **временем проверки** (Time Of Check) и **временем использования** (Time Of Use) состояние может измениться: другой поток удалил ключ из словаря, файл переименовали или удалили, сокет закрыли и т.д. Тогда проверка «прошла», а действие падает — и вы обрабатываете не ту ситуацию или получаете неожиданное исключение. При EAFP вы сразу делаете действие; если оно не удалось — обрабатываете конкретное исключение. Поэтому для конкурентного кода и для операций над внешними ресурсами EAFP часто надёжнее.

**Пример TOCTOU (концептуально):**

```python
# LBYL — уязвимость к гонке: между "key in d" и "d[key]" другой поток может удалить key
if "key" in d:
    value = d["key"]   # здесь может возникнуть KeyError, если ключ успели удалить!

# EAFP — атомарно: либо получаем значение, либо ловим KeyError
try:
    value = d["key"]
except KeyError:
    value = None
```

**Пример с файлом:** проверка `if os.path.exists(path):` и затем `open(path)` — между ними файл могут удалить или переименовать. Надёжнее сразу открывать и ловить `FileNotFoundError` (и при необходимости другие `OSError`).

---

#### Проверка типа vs попытка использовать (duck typing)

**LBYL:** явная проверка типа перед использованием:

```python
if isinstance(x, list):
    x.append(1)
```

**EAFP:** пробуем выполнить операцию, при ошибке обрабатываем:

```python
try:
    x.append(1)
except AttributeError:
    # x не поддерживает append (не список или другой тип без append)
    ...
```

В духе duck typing важнее «есть ли нужное поведение», а не «правильный ли тип». EAFP как раз проверяет поведение действием: если `append` есть и сработал — хорошо; если нет — обрабатываем исключение. Так код работает с любым объектом, у которого есть `append`, а не только с `list`.

---

#### Когда уместен LBYL

- Проверка **дёшевая** и явно улучшает читаемость (например, «если путь — директория, иначе файл»).
- Исключения **ожидаемы и часты**, и вы хотите явно разделить ветки (например, «если ключ есть — одна логика, если нет — другая»); иногда `if key in d` читается проще, чем `try/except KeyError`.
- Нужно **различать несколько типов ошибок** без исключений (например, проверка типа для разной сериализации). В таких случаях LBYL и EAFP можно комбинировать.

**Итог:** в Python чаще используют EAFP; LBYL — когда он явно упрощает код или когда проверка дёшевая и гонка несущественна. Избегайте голого `except: pass`: обрабатывайте конкретные исключения и явно логируйте или перебрасывайте при необходимости.

### 0.3.5 Протоколы и утиная типизация: duck typing

**Duck typing** (утиная типизация): «Если это ходит как утка и крякает как утка — это утка.» То есть важна не declared-тип объекта, а то, **есть ли у него нужные методы/поведение**. Если у объекта есть нужный метод — его можно использовать, наследование от общего базового класса не обязательно.

**Протокол** в Python — неформальный или формальный (через `typing.Protocol`) контракт: «объект, у которого есть такие-то методы, можно использовать так-то». Класс не обязан наследоваться от протокола — достаточно реализовать методы с нужными именами и сигнатурами. Интерпретатор не проверяет «реализует ли объект протокол»; проверка происходит в рантайме при вызове метода (или при итерации, вызове `len()` и т.д.).

**Примеры встроенных протоколов:**

- **Итерация (iterable):** объект с `__iter__` (возвращает итератор) или с `__getitem__` (с целочисленными индексами с 0) можно использовать в `for item in obj`. Функция `iter(obj)` вызывает `__iter__`; цикл `for` использует итератор.
- **Итератор (iterator):** объект с `__iter__` (возвращает себя) и `__next__` (следующий элемент или `StopIteration`). Один раз «прочитанный» итератор не перематывается.
- **Длина:** объект с `__len__` можно передать в `len(obj)`.
- **Контекстный менеджер:** объект с `__enter__` и `__exit__` можно использовать в `with obj:`.
- **Вызов:** объект с `__call__` можно вызывать как функцию: `obj(...)`.
- **Сравнение:** объекты с `__lt__`, `__le__`, `__eq__` и т.д. поддерживают операторы `<`, `<=`, `==`.
- **Строковое представление:** `__str__` для `str(obj)`, `__repr__` для `repr(obj)`.

Подробнее про магические методы и протоколы — в плане (§38–51 и др.).

**Пример утиной типизации:**

```python
def process_items(container):
    # Нам не важен тип container — важно, что по нему можно итерироваться
    for item in container:
        print(item)

process_items([1, 2, 3])           # list
process_items((1, 2, 3))           # tuple
process_items({1, 2, 3})          # set
process_items("abc")               # str
process_items(range(5))           # range
process_items(open("file"))       # файл — итератор строк
```

Ни один из этих типов не объявлен в сигнатуре — но у всех есть поддержка итерации (`__iter__` или `__getitem__`), поэтому функция работает. Это и есть duck typing: мы опираемся на **наличие поведения**, а не на имя типа.

**Пример: свой класс без наследования от общего базового класса.** Достаточно реализовать нужные методы — интерпретатор не проверяет «реализует ли класс протокол», проверка происходит при вызове метода:

```python
class MyRange:
    """Минимальный итерабельный объект: есть __iter__ и __next__."""
    def __init__(self, n):
        self.n = n
        self.i = 0
    def __iter__(self):
        return self
    def __next__(self):
        if self.i >= self.n:
            raise StopIteration
        self.i += 1
        return self.i - 1

for x in MyRange(3):
    print(x)  # 0, 1, 2
# MyRange не наследуется от abc.Iterable — но работает в for, т.к. есть __iter__ и __next__.
```

**typing.Protocol:** в модуле `typing` есть **Protocol** — абстрактное описание набора методов для **аннотаций**. Статический анализатор (mypy, pyright) может проверять совместимость без наследования в рантайме: класс не обязан наследоваться от Protocol — достаточно иметь нужные методы (structural subtyping). Пример аннотации с протоколом:

```python
from typing import Protocol

class SupportsLen(Protocol):
    def __len__(self) -> int: ...

def print_len(obj: SupportsLen) -> None:
    print(len(obj))  # mypy знает, что у obj есть __len__
```

Подробнее — в плане §9, §9a.

**Термины:**

- **Duck typing** — стиль, при котором тип не проверяется явно, важна поддержка нужных операций (методов).
- **Протокол** — набор методов/поведения, достаточный для использования в определённом контексте (итерация, контекстный менеджер, сравнение и т.д.).
- **Structural subtyping** — совместимость по структуре (наличие методов), а не по явному наследованию (nominal subtyping).

### 0.3.6 Явное лучше неявного: import, return None, аннотации типов

Принцип «explicit is better than implicit» из Zen применяется к импортам, возвращаемым значениям и контрактам функций. Ниже — подробнее по каждому аспекту.

---

#### Импорт

Лучше **явно** указывать, что импортируете: `from module import name` или `import module`. Тогда в коде видно, **откуда** взялось имя (какой модуль), и не возникает конфликтов имён и «магического» появления переменных.

**Избегайте** `from module import *` в коде библиотек и больших проектов: это засоряет текущий namespace десятками имён из модуля, затрудняет поиск определения имени (grep по имени находит много мест) и создаёт риск перезаписать локальные имена (например, если в модуле есть имя `open` или `list`). Исключение — иногда используется в **интерактивной консоли** для удобства (меньше печатать).

**Антипаттерн:** импорт внутри функции или внутри условия «для оптимизации» без веской причины — усложняет чтение и затрудняет статический анализ; обычно импорты в начале файла явнее.

**Пример явного импорта:**

```python
from pathlib import Path
import json
# В коде видно: Path — из pathlib, json — модуль json
# Конфликт имён маловероятен: имена из разных модулей явно указаны
```

---

#### return None

В Python функция без `return` или с `return` без значения возвращает `None`. Писать `return None` **не обязательно** — но иногда это делают для ясности («эта ветка явно возвращает отсутствие значения»), особенно в длинных функциях с несколькими выходами.

**Явность здесь — на усмотрение стиля проекта:** главное — не полагаться на неочевидное «ничего не вернула» там, где читатель ожидает значение (например, функция с именем `get_value()` в какой-то ветке не возвращает ничего — лучше явно `return None` или переформулировать логику). В коротких функциях с одним выходом «просто выйти без return» часто достаточно.

---

#### Аннотации типов

Аннотации типов (`def f(x: int) -> str:`) делают **контракт функции явным**: какие типы ожидаются на входе и что возвращается. Интерпретатор их **не проверяет** — проверка делается статическими анализаторами (mypy, pyright) и IDE (автодополнение, подсказки при наведении).

Это согласуется с принципом «explicit is better than implicit»: контракт виден в коде, а не только в документации или в голове автора. Даже без запуска mypy аннотации служат документацией для читателя и для IDE.

---

### 0.3.7 Плоский лучше вложенного: списки, генераторы, comprehensions

Глубокая вложенность (много уровней `if`/`for`/`try` внутри друг друга) усложняет чтение и отладку: трудно отследить, на каком уровне мы находимся, и какие условия привели к текущей ветке. Вместо этого часто используют:

- **Списковые/словарные/множественные включения (comprehensions)** — один уровень конструкции вместо нескольких строк циклов; условие и преобразование в одном месте; цикл «плоский» визуально.
- **Генераторы** и **генераторные выражения** — ленивая последовательность без построения всего списка в памяти; подходят для длинных или бесконечных последовательностей; синтаксис тот же «плоский» стиль.
- **Ранний выход из функции** (`if not ok: return`) и **guard-условия** в начале — уменьшение вложенности вместо длинной цепочки `if/else` в глубину; «нормальный» путь идёт вглубь по коду, а не вправо от вложенных условий.
- **Вынос вложенных циклов во вспомогательную функцию** — имя функции описывает шаг; основной код остаётся плоским (вызовы функций вместо вложенных блоков).

**Пример: глубоко вложенный код vs плоский с comprehensions**

```python
# Глубоко вложенный вариант
result = []
for row in table:
    row_ok = []
    for cell in row:
        if cell is not None and cell > 0:
            row_ok.append(cell * 2)
    if row_ok:
        result.append(row_ok)

# Более «плоский» вариант с comprehensions
result = [[cell * 2 for cell in row if cell is not None and cell > 0]
          for row in table]
result = [row for row in result if row]
```

Второй вариант короче и часто легче читается, когда привыкаешь к синтаксису comprehensions.

**Когда comprehensions вредят читаемости:** если выражение внутри comprehension длинное или логика сложная (несколько условий, вызовы нескольких функций), плоская запись может стать нечитаемой. Тогда лучше вынести во **вспомогательную функцию** или вернуться к **циклу с ранним выходом** — «плоскость» сохраняется за счёт коротких шагов с говорящими именами, а не за счёт одной гигантской строки.

### 0.3.8 Один способ vs множество: «There should be one obvious way» — но с гибкостью

В Zen говорится: «There should be one—and preferably only one—obvious way to do it.» На практике это значит, что для **типичных** задач язык подталкивает к одному очевидному способу; это снижает разброс стилей и облегчает чтение чужого кода.

**Примеры «одного очевидного способа» для типичных задач:**

- **Обход коллекции** — в первую очередь `for item in seq:` (итерация по итерабельным объектам). Не изобретать ручной счётчик `for i in range(len(seq))`, если индекс не нужен; для индекса — `for i, item in enumerate(seq):`.
- **Повторение N раз** — `for _ in range(N):` или явный перебор итератора; не писать `while` с ручным счётчиком без веской причины.
- **Чтение файла построчно** — `with open(path, encoding='utf-8') as f:` и `for line in f:` (файл — итератор строк). Не открывать файл без `with` и не читать весь файл в память (`f.read().splitlines()`) без необходимости.
- **Проверка наличия элемента** — `if x in container`; для словаря — `if key in d` или `d.get(key)`; для получения значения по умолчанию — `d.get(key, default)`.
- **Обработка ошибок** — EAFP: `try/except` вокруг основной операции; обрабатывать конкретные исключения, не голый `except: pass`.

**Когда допустимы другие способы (особые случаи):**

- **Цикл `while`** — когда условие выхода не «обойти последовательность» или «повторить N раз», а что-то более сложное (ожидание события, парсинг с ручным указателем и т.д.).
- **Ручной итератор** (`iter()`, `next()`) — когда нужен тонкий контроль (пропуск элементов, объединение нескольких итераторов с особой логикой); для простого обхода — `for`.
- **LBYL** — когда проверка дёшевая и явно улучшает читаемость или когда исключения ожидаемы и часты и вы хотите явно разделить ветки.

Итог: «один очевидный способ» — для типичных задач; в нестандартных ситуациях допустимо использовать более подходящий вариант, не нарушая общих принципов (читаемость, явность). Не поощряется избыточное разнообразие способов для одной и той же простой задачи без веской причины.
