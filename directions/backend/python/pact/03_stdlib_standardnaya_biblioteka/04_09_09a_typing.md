## 9. typing

Модуль **typing** предоставляет аннотации типов и конструкции для **статической проверки** (mypy, pyright, PyCharm) и **документации** кода. Аннотации не выполняются в runtime по умолчанию (кроме тех, что используются библиотеками вроде pydantic или FastAPI); их главная роль — сообщить type checker и человеку, какие типы ожидаются.

### Что такое typing и зачем он нужен

**Проблема без типов:** функции принимают и возвращают «что угодно»; ошибки (передали список вместо словаря, забыли обработать None) обнаруживаются только в runtime или в тестах. Код сложнее рефакторить: непонятно, какие аргументы допустимы.

**Решение:** аннотировать параметры и возвращаемые значения типами из **typing** (или встроенными **list**, **dict** с параметрами в 3.9+). Type checker статически проверяет вызовы и присваивания; IDE даёт подсказки и подсветку ошибок. **typing** — стандартный набор конструкций (Generic, Union, Callable, Protocol и т.д.), без которого выразить «список строк», «функция от int к str» или «объект с методом read» было бы невозможно в едином виде.

**Теоретическое обоснование:** Аннотации типов — это форма **контрактов**: функция объявляет, что принимает и возвращает. Проверка может быть статической (до запуска) и динамической (в runtime, если библиотека это использует). PEP 484 заложил базовый синтаксис и семантику; последующие PEP (544 Protocol, 585 builtin generics, 593 Annotated) расширили выразительность. **from **future** import annotations** (PEP 563) откладывает вычисление аннотаций, что устраняет проблемы с forward references и циклическими импортами.

### Основные типы и синонимы

- [ ] **typing.List[T]**, **Dict[K, V]**, **Tuple[T, ...]**, **Set[T]** — синонимы параметризованных типов. В **Python 3.9+** (PEP 585) встроенные типы поддерживают подскобки напрямую: **list[str]**, **dict[str, int]**, **tuple[int, ...]**, **set[str]** — предпочтительно в новом коде.
- [ ] **Optional[T]** = **Union[T, None]** — значение типа T или None. Параметр «может быть не передан» или «отсутствует».
- [ ] **Union[A, B, ...]** — значение одного из типов. В **3.10+** можно писать **A | B** вместо **Union[A, B]**.
- [ ] **Callable[[Arg1, Arg2], Ret]** — тип вызываемого объекта (функция, метод) с двумя аргументами и возвратом **Ret**. **Callable[..., Ret]** — любые аргументы (Ellipsis **...**); когда сигнатура аргументов не важна или переменная.

**Примеры**

```python
from typing import Optional, Union, Callable

def greet(name: str) -> str:
    return f"Hello, {name}"

def parse_id(s: Optional[str]) -> int:
    return int(s) if s is not None else 0

def handler(f: Callable[[int], str]) -> None:
    result = f(42)
```

### Переменные типов и дженерики

- [ ] **TypeVar('T')** — переменная типа: при вызове функции или создании класса **T** заменяется на конкретный тип. **TypeVar('T', bound=Base)** — T может быть только подтипом **Base**. **TypeVar('T', covariant=True)** или **contravariant=True** — для variance (см. §9a).
- [ ] **Generic[T]** — базовый класс для дженерик-классов: класс объявляет параметр типа, подкласс или экземпляр указывает конкретный тип (например **Stack[int]**).

**Пример**

```python
from typing import TypeVar, Generic

T = TypeVar('T')

class Stack(Generic[T]):
    def __init__(self) -> None:
        self._items: list[T] = []
    def push(self, x: T) -> None:
        self._items.append(x)
    def pop(self) -> T:
        return self._items.pop()

s: Stack[int] = Stack()
s.push(1)
```

### Протоколы и структурная типизация

- [ ] **Protocol** (PEP 544) — **структурное** подтипирование: класс совместим с протоколом, если у него есть нужные методы с подходящими сигнатурами, **без явного наследования**. Утиная типизация («если ходит как утка…») получает формальное выражение для type checker. **@runtime_checkable** — тогда **isinstance(x, MyProtocol)** работает в runtime (проверка наличия методов).

**Пример**

```python
from typing import Protocol, runtime_checkable

@runtime_checkable
class Readable(Protocol):
    def read(self, size: int = -1) -> str: ...

def use(r: Readable) -> str:
    return r.read(10)

# use(open('f.txt'))  # OK: file has read()
```

### Специальные аннотации

- [ ] **Literal[1, 2, "a"]** — значение ровно одно из перечисленных литералов. **Final** — константа (переприсваивание запрещено для type checker). **TypedDict** — тип словаря с заданными ключами и типами значений.
- [ ] **NewType('UserId', int)** — семантически отличный тип: на runtime это функция-обёртка (без накладных расходов), для type checker — отдельный тип, не смешиваемый с **int**.
- [ ] **ClassVar[T]** — атрибут класса (общий для всех экземпляров). **ParamSpec**, **Concatenate** — для аннотации декораторов с сохранением сигнатур.
- [ ] **@overload** — несколько сигнатур одной функции; реализация одна (обычно с **...**), аннотации перечисляют варианты вызова.
- [ ] **Annotated[T, x, y]** (PEP 593) — метаданные к типу; сам тип — **T**; **x**, **y** — произвольные метаданные (используются FastAPI, pydantic для валидации и сериализации).
- [ ] **Type[T]** — тип **класса** T; для фабрик: **def factory(cls: Type[T]) -> T** — возвращает экземпляр класса T. **isinstance(x, (int, str))** — кортеж типов допустим.
- [ ] **Any** — «любой тип»; отключает проверку. Использовать минимально, где тип действительно неизвестен или динамичен.
- [ ] **from **future** import annotations** (PEP 563, 3.7+) — все аннотации хранятся как **строки** и не вычисляются при определении класса/функции. Упрощает forward references (ссылки на класс, объявленный ниже) и уменьшает проблемы с циклическими импортами.

## 9a. Теория типизации (variance, type guards, narrowing)

### Variance (ковариантность, контравариантность, инвариантность)

**Зачем нужна variance:** при подстановке типов (например, передача **list[Derived]** туда, где ожидается **list[Base]**) нужно гарантировать типобезопасность. В зависимости от того, **читаем** мы из контейнера (producer) или **пишем** в него (consumer), допустимая подстановка разная. Variance формализует эти правила.

- [ ] **Covariant (Producer[T]):** если **A <: B** (A подтип B), то **Producer[A] <: Producer[B]**. То есть «производитель» типа A можно подставить вместо производителя типа B: оттуда мы только **читаем**, и получим значение типа A, которое совместимо с B. Пример: **тип возвращаемого значения** функции; **tuple** в Python ковариантен по типам элементов (только чтение по индексу не меняет тип).
- [ ] **Contravariant (Consumer[T]):** если **A <: B**, то **Consumer[B] <: Consumer[A]**. То есть «потребитель» типа B можно подставить вместо потребителя типа A: мы будем **передавать** в него значения; функция, принимающая B, может принять и A (всё, что проходит для A, проходит для B). Пример: **тип аргумента** функции. **Callable[[B], Ret]** совместим с **Callable[[A], Ret]** при A <: B — потому что вызывающий код будет передавать аргумент типа A, а функция готова принять любой B, включая A.
- [ ] **Invariant (Container[T]):** ни covariant, ни contravariant. **list**, **dict** инвариантны: **list[Derived]** не подтип **list[Base]**. Причина: в список можно **записать** элемент; если бы list был ковариантен, в **list[Base]** можно было бы подставить **list[Derived]**, и тогда в этот список кто-то мог бы записать другой **Base** (не Derived), нарушив тип. Поэтому мутабельные контейнеры — invariant.
- [ ] **TypeVar('T', covariant=True)** или **contravariant=True** — в typing объявляем переменную типа с заданной variance. **Callable[[Arg], Ret]** в typing контравариантен по **Arg** и ковариантен по **Ret**.

**Пример на Callable**

```python
from typing import Callable

class Animal: pass
class Dog(Animal): pass

def feed_animal(f: Callable[[Animal], None]) -> None:
    f(Animal())

def dog_handler(d: Dog) -> None:
    pass

# feed_animal ожидает Callable[[Animal], None]
# dog_handler имеет тип Callable[[Dog], None]
# Contravariance: Dog <: Animal => Callable[[Animal], None] <: Callable[[Dog], None]
# То есть функция, принимающая Animal, может быть передана туда, где ожидают функцию от Dog:
feed_animal(dog_handler)  # type checker: OK (передаём функцию, которая принимает более узкий тип)
# На самом деле передать dog_handler в feed_animal нельзя: feed_animal вызовет f(Animal()),
# а dog_handler ожидает Dog. Так что здесь контравариантность по аргументу: ожидается
# Callable[[Animal], None], а мы можем передать функцию с более широким аргументом (принимающую Animal)?
# Нет: мы передаём функцию с более узким аргументом (Dog). Правило: если ожидается f: (Animal)->None,
# то можно передать g: (Dog)->None? Нет — f будет вызвана с Animal(), а g принимает только Dog.
# Правильно: если ожидается f: (Animal)->None, то можно передать h: (Animal)->None или функцию,
# принимающую супертип Animal. То есть контравариантность: ожидаем Consumer[Animal],
# можно передать Consumer[Base] при Base :> Animal. В коде выше feed_animal(dog_handler) — ошибка типов:
# dog_handler принимает Dog, а feed_animal вызовет с Animal. Итог: Callable контравариантен по аргументу.
```

### Type guards и narrowing

- [ ] **Type narrowing** — сужение типа после проверки: после `if isinstance(x, int):` в блоке тип x — int.
- [ ] **TypeGuard[T]** (PEP 647): функция, возвращающая `TypeGuard[str]`, сообщает type checker, что при True аргумент имеет тип str. Пример: `def is_str(x: Any) -> TypeGuard[str]: return isinstance(x, str)`.
- [ ] **typing.assert_type(x, T)** — проверка на этапе type check; не выполняется в runtime. **typing.cast(T, x)** — явное приведение для type checker; в runtime просто возвращает x.
- [ ] Narrowing достигается также через `if x is None:`, `if hasattr(x, 'attr'):`. **assert_never(x)** в else ветке — исчерпывающая проверка (exhaustive check) для match/union.

### Structural vs nominal subtyping

- [ ] **Nominal** — подтип по явному наследованию: `class B(A):`.
- [ ] **Structural** — подтип по структуре (наличие методов/полей); в Python — **Protocol**. Утиная типизация («если ходит как утка…») — по сути структурная. **@runtime_checkable** позволяет проверять совместимость с Protocol через `isinstance`.
