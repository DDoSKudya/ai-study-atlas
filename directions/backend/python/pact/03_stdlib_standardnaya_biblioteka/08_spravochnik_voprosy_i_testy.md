## Вопросы на проверку

Дополнительные вопросы для самопроверки. Ответы в раскрывающихся блоках.

**1.** Чем в Py3 заменить `d.iteritems()` и почему `d.items()[0]` может не сработать?

<details>
<summary>Показать ответ</summary>
Замена: **d.items()**. В Py3 **d.items()** возвращает **view**, а не список. View не поддерживает индексацию **[0]**. Нужно **next(iter(d.items()))** или **list(d.items())[0]**.
</details>

**2.** Что вернёт `(Path('/etc') / 'hosts').read_text()` и при каком условии это вызовет исключение?

<details>
<summary>Показать ответ</summary>
Вернёт содержимое файла **/etc/hosts** как строку (str). Исключение: **FileNotFoundError** если файл не существует, **PermissionError** при отсутствии прав, **IsADirectoryError** если путь — каталог (в части версий).
</details>

**3.** Зачем в dataclass для поля `tags: list[str]` писать `field(default_factory=list)`, а не `default=[]`?

<details>
<summary>Показать ответ</summary>
**default=[]** даёт **один и тот же** список для всех экземпляров (мутабельный объект по умолчанию создаётся один раз при определении класса). Все экземпляры будут разделять один список — изменение в одном затронет остальные. **default_factory=list** создаёт **новый** список при создании каждого экземпляра.
</details>

**4.** Почему перед `itertools.groupby` обычно нужна сортировка?

<details>
<summary>Показать ответ</summary>
**groupby** объединяет только **подряд идущие** элементы с одинаковым ключом. Без сортировки элементы с одним ключом, разнесённые по итератору, попадут в разные группы. Сортировка по тому же key собирает одинаковые ключи подряд.
</details>

**5.** В чём разница между `functools.lru_cache(maxsize=128)` и `functools.cache`?

<details>
<summary>Показать ответ</summary>
**lru_cache(maxsize=128)** — кеш не более 128 записей; при переполнении вытесняется наименее недавно использованная. **cache** — то же, что **lru_cache(maxsize=None)**: неограниченный кеш, вытеснения нет. **cache** проще, но при неограниченном росте числа вызовов память может расти.
</details>

**6.** Что делает `contextlib.nullcontext()` и когда его удобно использовать?

<details>
<summary>Показать ответ</summary>
**nullcontext()** — контекстный менеджер, который при входе ничего не делает и возвращает **None** (или переданный **enter_result**); при выходе тоже ничего не делает. Удобно, когда контекст нужен только в одной ветке: **ctx = open(f) if f else nullcontext(); with ctx: ...**
</details>

**7.** Почему `list[Derived]` не подтип `list[Base]` при наследовании Derived от Base?

<details>
<summary>Показать ответ</summary>
**list** мутабелен: в него можно **записать** элемент. Если бы list[Derived] подходил вместо list[Base], в этот список можно было бы записать другой **Base** (не Derived), нарушив тип элементов. Поэтому мутабельные контейнеры **инвариантны**.
</details>

**8.** Чем `runpy.run_path('script.py')` лучше `exec(open('script.py').read())`?

<details>
<summary>Показать ответ</summary>
**run_path** устанавливает **__name__ == '__main__'**, добавляет каталог скрипта в **sys.path[0]**, задаёт **__file__**, обрабатывает **__package__** — поведение максимально близко к запуску **python script.py**. **exec(open(...).read())** выполнит код в текущем контексте без смены __name__ и пути; в traceback будет **&lt;string&gt;** вместо пути к файлу.
</details>

## Вопросы по модулям stdlib (с скрытыми ответами)

Вопросы сгруппированы по модулям/темам §7a. Ответы в раскрывающихся блоках. По 2–3 вопроса на каждый модуль/группу.

### __future__, __main__, builtins

**9.** Зачем писать **from __future__ import annotations** в начале файла?

<details>
<summary>Показать ответ</summary>
**annotations** (PEP 563) хранит аннотации типов как **строки** и не вычисляет их при загрузке модуля. Позволяет использовать типы, объявленные позже (forward references), без кавычек; уменьшает время старта и циклические зависимости. С 3.11 при этом импорте все аннотации становятся отложенными.
</details>

**10.** В каком случае блок **if __name__ == '__main__':** не выполнится?

<details>
<summary>Показать ответ</summary>
Блок не выполнится при **импорте** модуля (**import mymodule**). **__name__** равен **'__main__'** только при прямом запуске (**python mymodule.py** или **python -m mymodule**). При импорте **__name__** — имя модуля (например **'mymodule'**).
</details>

**11.** Зачем в тестах подменять **builtins.print** или **builtins.input**?

<details>
<summary>Показать ответ</summary>
Чтобы проверить логику без реального вывода в консоль или ввода пользователя: **builtins.print = lambda *a, **k: None** — заглушка; **builtins.input = lambda: 'test'** — фиксированный ввод. После теста нужно восстановить исходные значения. Лучше использовать **unittest.mock.patch('builtins.open', ...)** с ограниченной областью.
</details>

### abc — abstract base classes

**12.** Почему нельзя создать экземпляр класса, наследующего **ABC** с нереализованным **@abstractmethod**?

<details>
<summary>Показать ответ</summary>
Абстрактный метод задаёт **контракт**: подкласс обязан реализовать метод. При создании экземпляра интерпретатор проверяет наличие всех абстрактных методов; если хотя бы один не реализован — **TypeError: Can't instantiate abstract class**. Это гарантирует, что экземпляр всегда имеет полный интерфейс.
</details>

**13.** Чем **abc.ABCMeta** отличается от наследования от **ABC**?

<details>
<summary>Показать ответ</summary>
**ABC** — удобный класс, наследующий от **object** с метаклассом **ABCMeta**. Наследование от **ABC** достаточно для объявления абстрактных методов через **@abstractmethod**. **ABCMeta** нужен, когда класс уже имеет другой метакласс — тогда указывают **class C(MyBase, metaclass=ABCMeta)** или **class C(MyBase, ABC)** (ABC подставляет метакласс).
</details>

### argparse, optparse, getopt

**14.** Чем **argparse** предпочтительнее **getopt** для разбора аргументов CLI?

<details>
<summary>Показать ответ</summary>
**argparse** даёт автоматическую **--help**, подкоманды (**add_subparsers**), **nargs** (списки, опциональные аргументы), **type** и **choices**, единый стиль и сообщения об ошибках. **getopt** — минимальный разбор в стиле C (**getopt(args, shortopts, longopts)**); для сложного CLI удобнее **argparse**. **optparse** устарел (удалён в 3.12).
</details>

**15.** Что вернёт **parser.parse_args()** при вызове скрипта с **--help**?

<details>
<summary>Показать ответ</summary>
**ArgumentParser** по умолчанию при **--help** или **-h** выводит справку и вызывает **sys.exit()** — до **parse_args()** выполнение не дойдёт. Чтобы обрабатывать **--help** самому, можно отключить: **add_argument** с **help** не создаёт **-h** при **add_argument('--help', action='store_true')** вручную; обычно оставляют встроенное поведение.
</details>

**16.** Для чего в **argparse** используют **nargs='?'** и **nargs='*'**?

<details>
<summary>Показать ответ</summary>
**nargs='?'** — ноль или один аргумент: опция может быть без значения (вернётся **default**) или с одним значением. **nargs='*'** — ноль или более аргументов, собираются в **list** (например **--files a b c** → **args.files == ['a','b','c']**). **nargs='+'** — один или более (пустой список — ошибка).
</details>

### array, struct

**17.** Когда использовать **array.array('i', ...)** вместо **list** из целых?

<details>
<summary>Показать ответ</summary>
**array** хранит элементы **компактно** в памяти (фиксированные байты на элемент, без накладных на объекты Python). Для больших числовых массивов — меньше памяти и быстрее обмен с C (буферный протокол). **.tobytes()** / **.frombytes()** — прямой обмен байтами. Для небольших списков и когда нужны разнотипные объекты — **list**.
</details>

**18.** Что означает первый символ в формате **struct.pack** — **>**, **<**, **=**, **!**?

<details>
<summary>Показать ответ</summary>
**Порядок байт (endianness):** **>** — big-endian (старший байт первым, сетевой порядок); **<** — little-endian; **=** — native порядок; **!** — network (= big-endian). От выбора зависит, как многобайтовые числа упаковываются в байты; при обмене по сети или с файлами формата нужно согласовать порядок.
</details>

**19.** Чем **struct.unpack_from(fmt, buffer, offset)** отличается от **struct.unpack(fmt, buffer)**?

<details>
<summary>Показать ответ</summary>
**unpack** читает с **начала** buffer. **unpack_from** читает с **смещения offset** (в байтах) — удобно для разбора бинарного формата по частям (заголовок, затем блоки по смещениям). **pack_into** записывает по смещению в buffer.
</details>

### ast, dis, tokenize, symtable

**20.** Для чего используют **dis.dis(func)** и что выводит **dis.get_instructions(func)**?

<details>
<summary>Показать ответ</summary>
**dis.dis(func)** — дизассемблирует байткод функции в читаемый вид (инструкции виртуальной машины Python). **dis.get_instructions(func)** — итератор объектов **Instruction** с полями **opname**, **argval**, **offset** и т.д.; удобно для программного анализа байткода (оптимизаторы, анализаторы).
</details>

**21.** Чем **tokenize.generate_tokens(readline)** отличается от разбора строки вручную по пробелам?

<details>
<summary>Показать ответ</summary>
**tokenize** разбивает исходный код на **токены** по правилам Python: ключевые слова, идентификаторы, литералы, операторы, скобки, отступы. Учитывает строки в кавычках, комментарии, переносы строк. Разбор по пробелам не различает **if** и идентификатор, не обрабатывает строки и комментарии. **tokenize** нужен для линтеров, рефакторинга, подсветки синтаксиса.
</details>

**22.** Когда использовать **ast.literal_eval(s)** вместо **eval(s)**?

<details>
<summary>Показать ответ</summary>
**ast.literal_eval** безопасно вычисляет **только литералы**: числа, строки, bytes, кортежи, списки, dict, set, frozenset, True/False/None. **Не выполняет** вызовы функций, доступ к атрибутам, импорты — риск инъекции при **eval** от ненадёжного ввода отсутствует. Для конфигов/данных в виде Python-литералов — **literal_eval**; для произвольного кода — **eval** только с доверенным вводом.
</details>

**23.** Для чего используют **symtable.symtable(code, name, 'exec')** и что даёт **lookup(name).is_free()**?

<details>
<summary>Показать ответ</summary>
**symtable** строит **таблицу символов** скомпилированного кода без выполнения: какие имена объявлены, локальные/глобальные/параметры. **lookup(name)** возвращает **Symbol**; **.is_free()** — **True**, если имя **свободная переменная замыкания** (используется во вложенной функции, но объявлено во внешней). Нужно для статического анализа (линтеры, неиспользуемые переменные, области видимости).
</details>

### asyncio, concurrent.futures

**24.** Когда использовать **ThreadPoolExecutor**, а когда **ProcessPoolExecutor**?

<details>
<summary>Показать ответ</summary>
**ThreadPoolExecutor** — для I/O-задач (сеть, диск, ожидание ответов): мало накладных, GIL отпускается при ожидании. **ProcessPoolExecutor** — для CPU-задач (числа, парсинг, тяжёлые вычисления): обход GIL, изоляция по памяти; накладные на создание процессов выше. Для параллельных HTTP-запросов — ThreadPoolExecutor; для параллельного расчёта по ядрам — ProcessPoolExecutor.
</details>

**25.** Почему в корутине **asyncio** нельзя вызывать **time.sleep(5)**? Чем заменить?

<details>
<summary>Показать ответ</summary>
**time.sleep(5)** блокирует весь поток и цикл событий — остальные корутины не выполняются. Заменить на **await asyncio.sleep(5)** — корутина приостанавливается, событийный цикл передаёт управление другим задачам; через 5 секунд корутина продолжит выполнение.
</details>

**26.** Зачем в **asyncio** использовать **loop.run_in_executor(None, blocking_func)**?

<details>
<summary>Показать ответ</summary>
**run_in_executor** выполняет **блокирующую** функцию (time.sleep, синхронный requests.get) в пуле потоков (по умолчанию **ThreadPoolExecutor**), не блокируя событийный цикл. Возвращает **Future**, можно **await**. Без этого блокирующий вызов внутри корутины остановит весь loop — остальные корутины не выполнятся.
</details>

### base64, binascii, codecs

**27.** В чём разница между **base64.b64encode(data)** и **binascii.b2a_base64(data)**?

<details>
<summary>Показать ответ</summary>
**base64.b64encode(data)** возвращает **bytes**; **base64** обрабатывает переносы строк при декодировании, есть **urlsafe_b64encode** (алфавит без **+** и **/**). **binascii.b2a_base64(data)** — низкоуровневый вариант: возвращает bytes с переводом строки в конце; не добавляет переносы по 76 символов. Для большинства задач удобнее **base64**; **binascii** — когда нужен точный формат без обёрток.
</details>

**28.** Для чего в **codecs** используют **codecs.open(path, encoding='utf-8')** вместо **open(path)**?

<details>
<summary>Показать ответ</summary>
**codecs.open** открывает файл в заданной **кодировке** и возвращает объект, читающий/пишущий **строки** (str) — при чтении байты декодируются, при записи строки кодируются. В Py3 **open(path, encoding='utf-8')** уже делает то же; **codecs.open** полезен для **errors** (например **errors='replace'**) и для совместимости. В Py3 чаще используют **open(..., encoding=..., errors=...)**.
</details>

**29.** Чем **codecs.decode(b, 'hex')** отличается от **bytes.fromhex(s)**?

<details>
<summary>Показать ответ</summary>
**bytes.fromhex(s)** принимает **строку** из шестнадцатеричных символов (пробелы игнорируются) и возвращает **bytes**. **codecs.decode(b, 'hex')** принимает **bytes** (или строку) и декодирует из hex в bytes. Для строки **'deadbeef'** — **bytes.fromhex('deadbeef')**; для байт **b'deadbeef'** — **codecs.decode(b'deadbeef', 'hex')**. Оба дают байты; входной тип разный.
</details>

### collections, collections.abc

**30.** Когда использовать **ChainMap**, а когда объединение словарей через **{**a, **b}**?

<details>
<summary>Показать ответ</summary>
**ChainMap** — когда нужен **вид** на несколько словарей как на один: поиск по цепочке (первый найденный ключ), запись только в первый словарь, исходные словари не копируются. Объединение **{**a, **b}** создаёт **новый** словарь; изменения в **a**/ **b** не видны в результате. **ChainMap** удобен для конфига с переопределениями (базовый + локальный); объединение — когда нужна независимая копия.
</details>

**31.** Зачем в **collections.abc** проверять **isinstance(obj, Sequence)** вместо **isinstance(obj, list)**?

<details>
<summary>Показать ответ</summary>
**Sequence** — протокол (доступ по индексу, **len**, **__contains__**, итерация). **list**, **tuple**, **range**, **str**, **bytes** и пользовательские классы, реализующие протокол, подходят под **Sequence**. Проверка по **list** отклонит tuple, range, str. **isinstance(obj, Sequence)** принимает любой последовательный тип — код работает с разными реализациями без привязки к конкретному классу.
</details>

**32.** Когда использовать **collections.deque** вместо **list** для очереди (добавление/удаление с концов)?

<details>
<summary>Показать ответ</summary>
**deque** поддерживает **O(1)** добавление и удаление с обоих концов (**append**, **appendleft**, **pop**, **popleft**). **list** при **pop(0)** или **insert(0, x)** сдвигает элементы — **O(n)**. Для очереди (FIFO) или стека с операциями с начала — **deque**; для произвольного доступа по индексу и срезов — **list**.
</details>

### bisect, heapq

**33.** Чем **bisect.insort(a, x)** отличается от **a.append(x); a.sort()**?

<details>
<summary>Показать ответ</summary>
**bisect.insort** вставляет **x** в уже отсортированный список **a**, сохраняя порядок, за **O(log n)** поиск + **O(n)** сдвиг. **append** + **sort** каждый раз пересортировывает весь список — **O(n log n)**. Для поддержания отсортированного списка при последовательных вставках — **insort**; для однократной сортировки — **sort**.
</details>

**34.** Что возвращает **heapq.heappop(heap)** и какое свойство кучи гарантируется?

<details>
<summary>Показать ответ</summary>
**heappop** удаляет и возвращает **минимальный** элемент кучи (корень). **heap** после этого снова является мин-кучей. Элементы должны быть сравнимы; для объектов с приоритетом хранят кортежи **(priority, item)**. **heappush(heap, x)** добавляет элемент, сохраняя свойство кучи.
</details>

**35.** Зачем использовать **heapq.nlargest(n, iterable)** вместо **sorted(iterable)[-n:]**?

<details>
<summary>Показать ответ</summary>
**nlargest** использует **O(n)** памяти и за один проход по итератору находит n наибольших элементов — подходит для **очень больших** или бесконечных итераторов. **sorted(iterable)[-n:]** загружает весь итератор в память — **O(len(iterable))**. Для малых n и больших данных — **nlargest**; для полной сортировки — **sorted**.
</details>

### gzip, bz2, lzma, zipfile, tarfile

**36.** Чем **gzip.open(path)** отличается от **open(path)** при чтении файла?

<details>
<summary>Показать ответ</summary>
**gzip.open** читает **сжатые** данные (формат gzip) и возвращает объект, ведущий себя как файл — при **read()** данные автоматически распаковываются. **open(path)** читает сырые байты. Для файлов **.gz** — **gzip.open**; для обычных — **open**. Аналогично **bz2.open**, **lzma.open** для bz2 и xz.
</details>

**37.** Когда использовать **zipfile.ZipFile** вместо **tarfile.open**?

<details>
<summary>Показать ответ</summary>
**ZipFile** — формат **ZIP** (распространён на Windows, Java, веб). **tarfile** — форматы **tar**, **tar.gz**, **tar.bz2**, **tar.xz** (Unix, архивы исходников). Выбор по формату данных: если файл **.zip** — **ZipFile**; если **.tar.gz** и т.д. — **tarfile**. **tarfile** не сжимает сам по себе — используется с **gzip**/ **bz2**/ **lzma**.
</details>

**38.** Как добавить файл в существующий ZIP-архив без перезаписи всего архива?

<details>
<summary>Показать ответ</summary>
**ZipFile(path, 'a')** — режим **append**: открыть архив для добавления. **zipfile.write(file_path, arcname)** — добавить файл по имени в архиве. Полная перезапись не требуется; новые записи добавляются в конец. Для удаления или изменения существующих записей ZIP не поддерживает in-place — нужно создавать новый архив.
</details>

### concurrent (futures, interpreters)

**39.** Чем **concurrent.futures.Future** отличается от **asyncio.Future**?

<details>
<summary>Показать ответ</summary>
**concurrent.futures.Future** — результат **ThreadPoolExecutor** или **ProcessPoolExecutor**; блокирующий **future.result(timeout)** или **concurrent.futures.wait()**. **asyncio.Future** — результат асинхронной операции; **await future** или **asyncio.gather()**; используется в event loop. Смешивать нельзя; **loop.run_in_executor** возвращает **asyncio.Future**, оборачивающий concurrent Future.
</details>

**40.** Зачем в **concurrent.futures** использовать **executor.submit(func, *args)** вместо **executor.map(func, iterable)**?

<details>
<summary>Показать ответ</summary>
**submit** возвращает **Future** на один вызов — можно получать результаты по мере готовности (**as_completed**), отменять отдельные задачи, обрабатывать исключения по одной. **map** возвращает итератор результатов **в порядке** итератора — проще, но блокируется на первом неготовом. Для параллельной обработки с гибким контролем — **submit**; для простого «применить func ко всем» — **map**.
</details>

### configparser, json, csv, tomllib

**41.** Как сериализовать в JSON объект **datetime** или свой класс?

<details>
<summary>Показать ответ</summary>
**json.dumps** по умолчанию не умеет сериализовать **datetime** и произвольные классы. Варианты: (1) передать **default=функция**: функция получает объект и возвращает сериализуемое значение (например **datetime.isoformat()**) или выбрасывает **TypeError**; (2) наследовать **json.JSONEncoder** и переопределить **default**. При загрузке использовать **object_hook** или **cls** с кастомным декодером для восстановления объектов.
</details>

**42.** Зачем в **csv** использовать **csv.DictReader** вместо **csv.reader**?

<details>
<summary>Показать ответ</summary>
**DictReader** возвращает для каждой строки **словарь**: ключи — имена колонок (из первой строки или из **fieldnames**), значения — ячейки строки. Удобно обращаться по имени поля (**row['name']**), а не по индексу (**row[0]**). **reader** возвращает список значений — проще, но при изменении порядка колонок индексы ломаются.
</details>

**43.** Чем **tomllib.loads(data)** (3.11+) отличается от **json.loads(data)**?

<details>
<summary>Показать ответ</summary>
**tomllib** парсит формат **TOML** (конфиги: секции **[section]**, ключи **key = value**, типы число/строка/массив/таблица). **tomllib.loads** принимает **str** (bytes в 3.11 не поддерживается; в 3.12 — поддерживается). **json.loads** парсит **JSON** (объекты, массивы, примитивы). TOML — человекочитаемый конфиг; JSON — обмен данными и вложенные структуры.
</details>

### contextlib, contextvars

**44.** Зачем использовать **@contextmanager** вместо класса с **__enter__**/ **__exit__**?

<details>
<summary>Показать ответ</summary>
**@contextmanager** позволяет описать контекст **одной функцией** с **yield**: код до **yield** — вход (**__enter__**), после **yield** — выход (**__exit__**). Меньше кода и нагляднее для простых сценариев (временное изменение состояния, блокировка). Класс с **__enter__**/ **__exit__** нужен при сложной логике или переиспользовании состояния.
</details>

**45.** Что делает **contextlib.suppress(ValueError)** и когда его удобно использовать?

<details>
<summary>Показать ответ</summary>
**with suppress(ValueError):** блок, внутри которого **ValueError** **не** пробрасывается наружу — подавляется. Удобно для «попытаться сделать, при ошибке — пропустить» без **try/except** (например удалить файл: **with suppress(FileNotFoundError): os.remove(path)**). Для других исключений или логирования — **try/except**.
</details>

**46.** Для чего в **contextvars** используют **ContextVar**?

<details>
<summary>Показать ответ</summary>
**ContextVar** хранит значение, **привязанное к контексту выполнения** (например к цепочке async-вызовов или к потоку). В отличие от глобальной переменной, значение не «протекает» между разными задачами или потоками. Удобно для передачи «текущего запроса», «текущего пользователя» в асинхронном коде без явной передачи аргументом.
</details>

### copy, pickle, marshal

**47.** Почему **copy.copy(nested_list)** может дать неожиданный результат при изменении вложенного элемента?

<details>
<summary>Показать ответ</summary>
**copy.copy** — **поверхностная** копия: создаётся новый список, но элементы (в том числе вложенные списки) остаются **теми же ссылками**. Изменение **copy[0][0] = 99** затронет и оригинал, т.к. **copy[0]** и **original[0]** — один и тот же объект. Для полной независимости использовать **copy.deepcopy(nested_list)**.
</details>

**48.** Почему не следует загружать pickle-файлы из ненадёжного источника?

<details>
<summary>Показать ответ</summary>
**pickle.load** при десериализации **выполняет код** — воссоздаёт объекты через вызовы классов и атрибутов. Злоумышленник может подложить данные, при разборе которых выполнится произвольный код (создание файлов, запуск команд). Для ненадёжных данных использовать **json**, **csv** или форматы без восстановления произвольных объектов.
</details>

**49.** Почему **marshal** не рекомендуют для сериализации пользовательских данных?

<details>
<summary>Показать ответ</summary>
**marshal** — внутренний формат Python для байткода (.pyc); формат **не гарантирован** между версиями и платформами. Документация не рекомендует использовать для персистентного хранения; нет защиты от повреждённых данных. Для обмена данными — **pickle** (внутри процесса) или **json**/ **csv** (межплатформенно).
</details>

### datetime, time, zoneinfo, calendar

**50.** Почему для измерения таймаута используют **time.monotonic()**, а не **time.time()**?

<details>
<summary>Показать ответ</summary>
**time.time()** — системные часы (секунды с эпохи); их могут перевести назад при NTP или вручную — интервал «сейчас минус старт» станет неверным или отрицательным. **time.monotonic()** — монотонные часы, никогда не идут назад; разница двух вызовов даёт корректную длительность. Для таймаутов и замеров длительности — только **monotonic**.
</details>

**51.** Что такое **naive** и **aware** datetime? Почему их нельзя сравнивать?

<details>
<summary>Показать ответ</summary>
**Naive** — **datetime** без **tzinfo** (без привязки к часовому поясу). **Aware** — с заданным **tzinfo** (например **ZoneInfo('Europe/Moscow')**), однозначно соответствует моменту в UTC. Сравнение naive с aware в Python 3 вызывает **TypeError**: неясно, как интерпретировать naive (какая зона?). Сравнивать можно два naive или два aware; при необходимости перевести в общую зону (**astimezone**).
</details>

**52.** Для чего используют **calendar** (модуль **calendar**)?

<details>
<summary>Показать ответ</summary>
**calendar** — работа с **календарём**: **calendar.monthrange(year, month)** — число дней в месяце и день недели первого дня; **calendar.isleap(year)** — високосный год; **calendar.weekday(year, month, day)** — день недели (0=понедельник); **calendar.monthcalendar** — матрица недель месяца; форматирование календаря в строку. Для расчётов по датам удобнее **datetime**; **calendar** — для отображения и календарной логики.
</details>

### dataclasses, enum

**53.** Зачем в **dataclass** для поля **tags: list[str]** писать **field(default_factory=list)**, а не **default=[]**?

<details>
<summary>Показать ответ</summary>
**default=[]** — один и тот же **объект списка** будет использоваться для **всех** экземпляров класса (мутабельные значения по умолчанию общие). Изменение **obj1.tags.append(1)** затронет **obj2.tags**. **default_factory=list** — при создании экземпляра вызывается **list()**, у каждого экземпляра свой список. Для мутабельных полей (list, dict) — только **default_factory**.
</details>

**54.** Чем **enum.Enum** отличается от набора констант (**RED = 1; GREEN = 2**)?

<details>
<summary>Показать ответ</summary>
**Enum** даёт **тип** и **namespace**: элементы **Color.RED**, **Color.GREEN**; итерация **for c in Color**; сравнение по идентичности (**x is Color.RED**); нельзя подменить значением другого типа. Константы — просто числа/строки; путаница с другими числами. **Enum** удобен для ограниченного набора именованных значений и автодополнения.
</details>

**55.** Когда использовать **enum.auto()** в **Enum**?

<details>
<summary>Показать ответ</summary>
**auto()** автоматически присваивает **уникальные** значения элементам (целые по возрастанию). Удобно, когда важны только **имена** элементов, а не конкретные числа — не нужно вручную задавать 1, 2, 3. Для сериализации в числа или совместимости с внешним API задают значения явно.
</details>

### decimal, fractions, statistics, math

**56.** Когда использовать **decimal.Decimal** вместо **float**?

<details>
<summary>Показать ответ</summary>
**Decimal** — **десятичная** арифметика с заданной точностью; нет двоичных погрешностей float (0.1 + 0.2 != 0.3 в float). Для финансов, налогов, валют — **Decimal**. **float** — быстрее и меньше памяти; для научных расчётов и когда погрешность допустима — **float**.
</details>

**57.** Чем **fractions.Fraction** отличается от **decimal.Decimal**?

<details>
<summary>Показать ответ</summary>
**Fraction** хранит число как **рациональную дробь** (числитель/знаменатель) — точное представление для дробей вроде 1/3. **Decimal** — десятичное с фиксированной точностью. **Fraction** удобен для символьной точности дробей; **Decimal** — для десятичных денежных сумм и округления по правилам.
</details>

**58.** Для чего используют **statistics** (mean, median, stdev)?

<details>
<summary>Показать ответ</summary>
**statistics** — базовые **статистические** функции: **mean**, **median**, **mode**, **stdev**, **variance**, **quantiles** и т.д. Работают с итераторами и списками чисел. Для простой описательной статистики — **statistics**; для тяжёлой аналитики и машинного обучения — **numpy**/ **scipy** (вне stdlib).
</details>

### difflib, filecmp

**59.** Для чего используют **difflib.unified_diff(a, b)** и **difflib.SequenceMatcher**?

<details>
<summary>Показать ответ</summary>
**unified_diff** генерирует **unified diff** (как в git) между двумя последовательностями строк — для отображения изменений в файлах или тексте. **SequenceMatcher** находит **наибольшие совпадающие блоки** между двумя последовательностями; **ratio()** — мера похожести (0–1). Удобно для сравнения текстов, поиска плагиата, слияния.
</details>

**60.** Чем **filecmp.cmp(f1, f2)** отличается от сравнения **open(f1).read() == open(f2).read()**?

<details>
<summary>Показать ответ</summary>
**filecmp.cmp** может сравнивать файлы **по содержимому** (по умолчанию) или **поверхностно** (по **os.stat** — размер, mtime). Для больших файлов **cmp(..., shallow=False)** читает по кускам и может остановиться при первом отличии. Сравнение через **read()** загружает оба файла в память целиком — для больших файлов неэффективно.
</details>

**61.** Когда использовать **filecmp.dircmp(dir1, dir2)**?

<details>
<summary>Показать ответ</summary>
**dircmp** сравнивает **два каталога** рекурсивно: общие файлы, только в одном, только в другом, различающиеся файлы, общие подкаталоги. Методы **report()**, **report_full_closure()** выводят отчёт. Удобно для синхронизации каталогов, бэкапов, поиска расхождений без ручного обхода.
</details>

### email, mimetypes, html

**62.** Для чего в **email** используют **email.message_from_string(s)** или **email.message_from_bytes(b)**?

<details>
<summary>Показать ответ</summary>
Парсинг **MIME-сообщения** (заголовки, тело, вложения) из строки или байт. Возвращает объект **Message** с методами **get**, **get_all**, **get_payload**, **walk** (по частям). Удобно разбирать письма из файла или сети без ручного разбора заголовков и границ multipart.
</details>

**63.** Чем **mimetypes.guess_type(path)** отличается от проверки расширения вручную?

<details>
<summary>Показать ответ</summary>
**guess_type** использует **таблицу** расширений и (на части платформ) **magic** (содержимое файла). Возвращает **(type, encoding)** (например **('text/html', None)**). Ручная проверка по расширению не учитывает стандартные типы и кодировки; **mimetypes** — единый источник по MIME-типам.

**64.** Для чего используют **html.escape(s)** перед вставкой строки в HTML?

<details>
<summary>Показать ответ</summary>
**html.escape** заменяет **&**, **<**, **>** на **&amp;**, **&lt;**, **&gt;** — защита от **XSS**: пользовательский ввод не интерпретируется как теги или скрипты. При вставке в HTML (**<div>** + user_input + **</div>**) без экранирования возможна инъекция. **quote=True** (по умолчанию) экранирует и кавычки.
</details>

### errno, faulthandler, traceback, warnings

**65.** Для чего используют **errno** после **OSError** (например **except OSError as e: e.errno**)?

<details>
<summary>Показать ответ</summary>
**e.errno** — код ошибки (**errno.ENOENT**, **errno.EACCES** и т.д.); по коду можно различить «файл не найден», «доступ запрещён», «прервано» и т.д. **errno** содержит константы для этих кодов. Удобно для разной обработки по типу ошибки (повторить при EINTR, сообщить пользователю при ENOENT).
</details>

**66.** Зачем включать **faulthandler.enable()** в продакшене при отладке падений?

<details>
<summary>Показать ответ</summary>
**faulthandler** выводит **traceback** при фатальных сигналах (SIGSEGV, SIGABRT и т.д.) — когда интерпретатор падает без нормального исключения (например segfault в C-расширении). Без **faulthandler** в логе только «Segmentation fault»; с ним — последний Python traceback, что помогает локализовать место сбоя.
</details>

**67.** Чем **warnings.warn(message, UserWarning)** отличается от **raise Exception(message)**?

<details>
<summary>Показать ответ</summary>
**warn** выдаёт **предупреждение** — выполнение продолжается; можно фильтровать через **warnings.filterwarnings**, превращать в исключение (**warnings.simplefilter('error')**). **raise** прерывает выполнение. Предупреждения — для устаревшего API, сомнительных значений; исключения — для ошибок, после которых продолжать нельзя.
</details>

### functools, operator

**68.** Чем **functools.lru_cache(maxsize=128)** отличается от **functools.cache**?

<details>
<summary>Показать ответ</summary>
**lru_cache** — кеш с **ограничением размера** (maxsize); при переполнении удаляется наименее недавно использованный элемент. **cache** (3.9+) — неограниченный кеш (**lru_cache(maxsize=None)**); все результаты сохраняются. Для функций с ограниченным множеством аргументов — **cache**; для больших или неограниченных — **lru_cache(maxsize=N)**.
</details>

**69.** Для чего используют **functools.wraps(f)** в декораторе?

<details>
<summary>Показать ответ</summary>
**wraps(f)** копирует **__name__**, **__doc__**, **__module__** и др. с обёрнутой функции **f** на обёртку — иначе **help(decorated)** и **decorated.__name__** будут указывать на обёртку, а не на исходную функцию. В декораторе: **def decorator(f): @wraps(f); def inner(...): ...; return inner**.
</details>

**70.** Когда использовать **operator.itemgetter('key')** вместо **lambda x: x['key']**?

<details>
<summary>Показать ответ</summary>
**itemgetter('key')** возвращает функцию, эквивалентную **lambda x: x['key']**; реализована в C — быстрее. Удобно для **sorted(items, key=itemgetter('name'))**, **map(itemgetter(0), pairs)**. Для простых выражений разница мала; для горячих циклов и сортировки больших данных — **itemgetter** или **attrgetter**.
</details>

### gc, sys, types, copyreg

**71.** Зачем вызывать **gc.collect()** вручную?

<details>
<summary>Показать ответ</summary>
**gc.collect()** принудительно запускает **полный** цикл сборки мусора. Обычно GC вызывается автоматически. Вручную полезно: после освобождения большого объёма объектов (освободить память сразу), в тестах (проверить, что объекты удаляются), при измерении памяти. Не злоупотреблять — лишние вызовы замедляют.
</details>

**72.** Что хранит **sys.modules** и зачем его проверять?

<details>
<summary>Показать ответ</summary>
**sys.modules** — словарь **загруженных** модулей: ключ — имя модуля, значение — объект модуля. При **import foo** проверяется **'foo' in sys.modules**; если есть — повторная загрузка не выполняется. Проверять полезно для отладки циклических импортов и для подмены модулей в тестах.
</details>

**73.** Для чего используют **types.SimpleNamespace**?

<details>
<summary>Показать ответ</summary>
**SimpleNamespace** — объект с **атрибутами** по ключевым словам: **ns = SimpleNamespace(a=1, b=2)**; **ns.a**, **ns.b**. Удобная замена словаря, когда нужен доступ через точку (**ns.key** вместо **d['key']**). Легковеснее, чем класс; для конфигов и результатов парсинга — **SimpleNamespace** или **Namespace** (argparse).
</details>

### getpass, getopt, gettext, locale

**74.** Зачем использовать **getpass.getpass(prompt)** вместо **input(prompt)** для ввода пароля?

**getpass** не выводит введённые символы на экран (или подменяет на **\*** в зависимости от платформы) — пароль не виден окружающим. **input** печатает всё, что вводит пользователь. Для паролей, токенов, секретов — **getpass**; для обычного ввода — **input**.

**75.** Чем **gettext** отличается от хранения строк в словаре по языку?

Словарь по языку — ручной выбор; **gettext** интегрируется с **locale** и инструментами (xgettext, msgfmt).
</details>

**76.** Для чего используют **locale.setlocale(locale.LC_ALL, '')**?

<details>

**setlocale(LC_ALL, '')** устанавливает локаль по **переменным окружения** (LANG, LC_*). Влияет на форматирование чисел, дат, валют (**locale.format_string**, **locale.str**), сортировку строк. Пустая строка **''** — «пользовательская локаль». Для консистентного вывода в формате пользователя — **setlocale**; в серверных приложениях часто фиксируют локаль (например **'C'**).
</details>

### glob, fnmatch, pathlib, os, shutil, tempfile, fileinput

**77.** Чем **Path('/a/b').resolve()** отличается от **Path('/a/b').absolute()**?

<details>
<summary>Показать ответ</summary>
**resolve()** — **канонический** путь: разрешаются символические ссылки, убираются **.** и **..**, путь приводится к одному виду. **absolute()** — делает путь абсолютным, присоединяя при необходимости текущий каталог, но **не** разрешает symlink и не обязательно нормализует **..**. Для «реального» пути на диске — **resolve()**; для простого абсолютного представления — **absolute()**.
</details>

**78.** Когда использовать **shutil.copy**, а когда **shutil.copy2**?

<details>
<summary>Показать ответ</summary>
**shutil.copy** копирует содержимое файла; метаданные (время модификации, права) могут не сохраниться на части платформ. **shutil.copy2** копирует и метаданные (по возможности **st_mode**, **st_mtime** и т.д.). Для бэкапов и когда важна дата файла — **copy2**; когда нужна только копия содержимого — **copy**.
</details>

**79.** Чем **glob.glob('**/*.py')** отличается от **Path.rglob('*.py')**?

<details>
<summary>Показать ответ</summary>
**glob.glob** возвращает **список строк** (путей); **Path.rglob** возвращает **итератор** объектов **Path**. **rglob** — рекурсивный поиск (все подкаталоги); **glob** с **'**/*.py'** тоже рекурсивный (в текущей версии). **pathlib** удобен для цепочек методов (**Path(...).rglob('*.py').read_text()**); **glob** — для простого списка путей-строк.
</details>

**80.** Зачем использовать **tempfile.NamedTemporaryFile(delete=False)** вместо **open('/tmp/x', 'w')**?

<details>
<summary>Показать ответ</summary>
**NamedTemporaryFile** создаёт файл с **уникальным** именем в системной временной директории; коллизии с другими процессами исключены. **delete=False** — файл не удаляется при закрытии (удобно передать путь другой программе). Ручной **/tmp/x** — риск перезаписи и небезопасен (симлинк-атаки); **tempfile** использует **os.getpid()** и случайность в имени.
</details>

**81.** Для чего используют **fileinput** (модуль **fileinput**)?

<details>
<summary>Показать ответ</summary>
**fileinput** — итерация по **строкам** нескольких файлов или stdin как по одному потоку: **for line in fileinput.input(files)**. Поддержка **inplace** — перезапись файла «на месте» (резервная копия, замена строк). Удобно для скриптов в стиле sed/awk: обработать список файлов из аргументов или stdin.
</details>

### hashlib, hmac, secrets

**82.** Почему для хранения паролей нельзя использовать «сырой» SHA-256 от пароля?

<details>
<summary>Показать ответ</summary>
Одинаковые пароли дают одинаковый хеш — по утёкшей базе хешей можно подбирать пароли по словарю и радужным таблицам. Нет **соли** (уникальной на пользователя) и **итераций** (замедление перебора). Нужно **hashlib.pbkdf2_hmac** (или **scrypt**) с **солью** (например **secrets.token_bytes(16)**) и большим числом итераций; проверку делать через **secrets.compare_digest**.
</details>

**83.** Для чего используют **hmac.new(key, msg, digest_mod)**?

<details>
<summary>Показать ответ</summary>
**HMAC** — **ключевая** хеш-функция: подпись сообщения **msg** секретным ключом **key**; подделка без ключа невозможна. **hmac.new(key, msg, digest_mod).hexdigest()** — подпись в hex. Используется для проверки целостности и подлинности (API-токены, cookies). Сравнивать подписи через **hmac.compare_digest** — защита от timing attacks.
</details>

**84.** Зачем использовать **secrets.compare_digest(a, b)** вместо **a == b** при сравнении токенов или хешей?

<details>
<summary>Показать ответ</summary>
**compare_digest** выполняется за **константное** время — защита от **timing attack**: по времени сравнения нельзя подобрать символы по одному. **a == b** для строк может завершаться раньше при первом несовпадении — утечка информации. Для паролей, токенов, подписей — **secrets.compare_digest**.
</details>

### http, urllib, socket, ssl

**85.** Чем **urllib.request.urlopen(url)** отличается от **http.client.HTTPSConnection** + **request** + **getresponse**?

<details>
<summary>Показать ответ</summary>
**urlopen** — **высокоуровневый** вызов: один URL, автоматическое следование редиректам (опционально), контекстный менеджер. **http.client** — низкоуровневый: ручное соединение, запрос, чтение ответа, повторные запросы (keep-alive). Для простого GET/POST — **urlopen**; для полного контроля над заголовками, телом и несколькими запросами в одном соединении — **http.client**.
</details>

**86.** Для чего используют **urllib.parse.urlparse(url)** и **urllib.parse.urljoin(base, url)**?

<details>
<summary>Показать ответ</summary>
**urlparse(url)** разбивает URL на **схему**, **netloc**, **path**, **query**, **fragment** — для извлечения хоста, пути, параметров. **urljoin(base, url)** собирает **абсолютный** URL из базового и относительного (учёт **../** и корня). Удобно для парсинга ссылок и формирования запросов без ручной склейки строк.
</details>

**87.** Когда использовать **ssl.create_default_context()** при работе с **socket** или **http.client.HTTPSConnection**?

<details>
<summary>Показать ответ</summary>
**create_default_context()** возвращает **SSL-контекст** с настройками по умолчанию: проверка сертификата, подходящие протоколы. Без контекста или с пустым контекстом проверка сервера может быть отключена — уязвимость. Для **HTTPSConnection(host, context=ssl.create_default_context())** или **socket** с **wrap_socket** — передавать этот контекст для безопасного TLS.
</details>

### subprocess, multiprocessing, threading

**88.** Почему для вызова внешней команды с пользовательским вводом предпочитают **subprocess.run([cmd, arg], shell=False)** вместо **shell=True**?

<details>
<summary>Показать ответ</summary>
При **shell=True** строка передаётся в shell (**/bin/sh** или **cmd.exe**); если в строку попадёт пользовательский ввод, возможна **инъекция команд** (например **; rm -rf /**). При **shell=False** и списке аргументов **[cmd, arg]** аргументы передаются в программу как есть, без интерпретации shell — подстановка безопасна. Для пайпов и перенаправлений — несколько **Popen** и связывание потоков вручную.
</details>

**89.** Зачем в скрипте с **multiprocessing.Process** писать **if __name__ == '__main__':** перед созданием процессов?

<details>
<summary>Показать ответ</summary>
При **spawn** (по умолчанию на Windows) дочерний процесс **запускает модуль заново** и выполняет весь код на верхнем уровне. Без проверки **__name__ == '__main__'** код создания процессов выполнится и в дочернем процессе — получится рекурсивное порождение процессов. В блоке **if __name__ == '__main__':** этот код выполняется только в главном процессе.
</details>

**90.** Чем **subprocess.run(...)** отличается от **subprocess.Popen(...)**?

<details>
<summary>Показать ответ</summary>
**run** — **высокоуровневый** вызов: ждёт завершения процесса, возвращает **CompletedProcess** с **returncode**, **stdout**, **stderr**. Удобно для простых вызовов с захватом вывода. **Popen** — низкоуровневый: создаёт процесс, не ждёт; нужно самому вызывать **communicate()** или **wait()**. Для пайпов между процессами и пошагового ввода/вывода — **Popen**; для «вызвать и получить результат» — **run**.
</details>

### re, string, textwrap, unicodedata

**91.** Чем **re.match(pattern, s)** отличается от **re.search(pattern, s)**?

<details>
<summary>Показать ответ</summary>
**re.match** ищет совпадение **с начала строки** (как если бы паттерн был с **^**). **re.search** ищет первое совпадение **в любом месте** строки. **match('a', 'xab')** — None; **search('a', 'xab')** — Match на **'a'**. Для проверки «вся строка подходит под паттерн» используют **re.fullmatch(pattern, s)**.
</details>

**92.** Зачем нормализовать Unicode-строки (**unicodedata.normalize**) перед сравнением?

<details>
<summary>Показать ответ</summary>
Один и тот же символ может быть представлен по-разному: например «é» как один код (U+00E9) или как «e» + комбинирующий знак (U+0065 U+0301). Строки выглядят одинаково, но **'é' == 'e\u0301'** — False. **normalize('NFC', s)** (или **NFD**) приводит к одной форме — сравнение и поиск становятся корректными. Для хранения и сравнения обычно используют **NFC**.
</details>

**93.** Для чего используют **textwrap.fill(text, width=70)**?

<details>
<summary>Показать ответ</summary>
**textwrap.fill** разбивает **текст** на строки заданной **ширины** (по умолчанию 70 символов), перенося по границам слов. Удобно для форматирования длинного текста в консоль или в отчёт. **textwrap.wrap** возвращает список строк; **fill** — одну строку с переводами строк между ними. **shorten** — обрезка с многоточием.
</details>

### logging, unittest, doctest

**94.** Почему сообщения **logger.debug()** не выводятся после **logging.basicConfig(level=logging.INFO)**?

<details>
<summary>Показать ответ</summary>
**basicConfig** задаёт уровень **корневого** логгера (и при необходимости создаёт handler). Уровень **DEBUG** (10) ниже **INFO** (20) — сообщения с уровнем DEBUG отфильтровываются и не передаются в handler. Чтобы видеть DEBUG, вызвать **logging.basicConfig(level=logging.DEBUG)** или установить **logger.setLevel(logging.DEBUG)** для нужного логгера.
</details>

**95.** Зачем в **logging** использовать **logger.exception()** вместо **logger.error()** в блоке **except**?

<details>
<summary>Показать ответ</summary>
**exception()** записывает сообщение с уровнем ERROR и **добавляет traceback** текущего исключения в лог. **error()** только сообщение. В **except** блоке **logger.exception(...)** даёт полный traceback без ручного **traceback.format_exc()** — удобно для отладки и логов ошибок.
</details>

**96.** Чем **unittest.TestCase** отличается от запуска тестов вручную (**if __name__ == '__main__': test_one()**)?

<details>
<summary>Показать ответ</summary>
**unittest** даёт **набор** утверждений (**assertEqual**, **assertRaises**, **assertIn** и т.д.), **фикстуры** (**setUp**/ **tearDown**), **обнаружение** тестов по имени (**test_***), запуск из командной строки (**python -m unittest**), счётчик успехов/ошибок. Ручной запуск — гибко, но без стандартной отчётности и изоляции. **doctest** — тесты в docstring; **unittest** — отдельные методы тестов.
</details>

**97.** Для чего используют **doctest** (модуль **doctest**)?

<details>
<summary>Показать ответ</summary>
**doctest** извлекает **примеры** из docstring (строки, выглядящие как интерактивная сессия Python) и проверяет, что выполнение кода даёт ожидаемый вывод. Удобно для **документации с примерами** и простых тестов в одном месте. **doctest.testmod()** — проверить текущий модуль; **doctest.run_docstring_examples** — проверить конкретную функцию.
</details>

### sched, shelve, sqlite3, dbm

**98.** В чём риск при **shelve** с **writeback=False** и выражении вида **shelf['key'].append(1)**?

<details>
<summary>Показать ответ</summary>
При **writeback=False** изменение объекта «на месте» (**shelf['key'].append(1)**) **не** записывается обратно в хранилище — shelf хранит ссылку на загруженный объект, но при **sync()**/ **close()** перезаписываются только ключи, значение которых было присвоено (**shelf['key'] = ...**). Нужно **shelf['key'] = shelf['key'] + [1]** (или переприсвоить) и затем **sync()**; либо открывать с **writeback=True** (тогда все изменения в памяти сбрасываются при **sync**/ **close**).
</details>

**99.** Для чего используют **sched.scheduler**?

<details>
<summary>Показать ответ</summary>
**sched.scheduler** — **планировщик** событий по времени: **enter(delay, priority, func, args)** — запланировать вызов **func(*args)** через **delay** секунд; **run()** — обрабатывать очередь. Один поток; для конкурентности — **asyncio** или **threading.Timer**. Удобно для простых «выполнить через N секунд» без внешних зависимостей.
</details>

**100.** Чем **sqlite3.connect(':memory:')** отличается от **sqlite3.connect('file.db')**?

<details>
<summary>Показать ответ</summary>
**:memory:** — база в **оперативной памяти**; данные теряются после закрытия соединения. Удобно для тестов и временных расчётов без создания файлов. **file.db** — база в **файле**; данные сохраняются. Для персистентного хранения — файл; для изоляции тестов и кеша в памяти — **:memory:**.
</details>

**101.** Когда использовать **dbm** (модуль **dbm**) вместо **dict** или **sqlite3**?

<details>
<summary>Показать ответ</summary>
**dbm** — **ключ–значение** хранилище на диске (как словарь, но персистентный); ключи и значения — bytes. Проще, чем sqlite3, для простого кеша или индекса. **sqlite3** — запросы, транзакции, типы. **dict** — только в памяти. Для большого кеша, не помещающегося в RAM — **dbm**; для сложных запросов — **sqlite3**.
</details>

### signal, atexit, errno

**102.** Что делать в обработчике сигнала (**signal.signal**), если нужно «мягко» завершить приложение?

<details>
<summary>Показать ответ</summary>
Обработчик должен выполнять только **async-signal-safe** операции (минимум логики). Обычно устанавливают флаг (например глобальную переменную **shutdown = True**) и в основном цикле проверяют его; либо вызывают **sys.exit(code)**. Нельзя вызывать **malloc**, сложный **print**, работу с файлами — возможен deadlock. Для «мягкого» завершения: в обработчике установить флаг, в main loop при флаге закрыть ресурсы и выйти.
</details>

**103.** Для чего используют **atexit.register(func)**?

<details>
<summary>Показать ответ</summary>
**atexit.register(func)** регистрирует функцию для вызова при **нормальном** завершении интерпретатора (не при kill -9). Удобно для очистки временных файлов, закрытия соединений, сохранения состояния. Вызовы выполняются в порядке, обратном регистрации. При аварийном завершении (сигнал, segfault) **atexit** не вызывается.
</details>

**104.** Чем **signal.SIGTERM** отличается от **signal.SIGKILL** с точки зрения обработки в Python?

<details>
<summary>Показать ответ</summary>
**SIGTERM** можно **перехватить** (**signal.signal(signal.SIGTERM, handler)**); процесс может выполнить очистку и выйти. **SIGKILL** **не перехватывается** — ядро сразу завершает процесс; обработчик в Python установить нельзя. Для «мягкого» завершения посылают **SIGTERM**; **SIGKILL** — когда процесс не реагирует.
</details>

### Платформо-зависимые, curses

**105.** Зачем использовать **fcntl.flock(fd, LOCK_EX | LOCK_NB)** вместо просто **LOCK_EX**?

<details>
<summary>Показать ответ</summary>
**LOCK_EX** — эксклюзивная блокировка; при занятом файле вызов **блокируется** до освобождения. **LOCK_NB** (non-blocking) — не ждать: если файл уже заблокирован, **flock** сразу вернёт ошибку (**BlockingIOError** / **PermissionError** в зависимости от платформы). Удобно для «попытаться захватить lock, если не получилось — выйти» без зависания процесса.
</details>

**106.** Почему при работе с **curses** рекомендуют вызывать код через **curses.wrapper(main)**?

<details>
<summary>Показать ответ</summary>
**curses** переводит терминал в особый режим (raw, noecho). При исключении или выходе из программы терминал может остаться в этом режиме — символы не отображаются, Ctrl+C не работает. **curses.wrapper(main)** вызывает **main(stdscr)** и в **finally** всегда выполняет **curses.endwin()**, восстанавливая терминал даже при ошибке или **sys.exit()**.
</details>

**107.** Для чего используют **pwd.getpwuid(uid)** и **grp.getgrgid(gid)** (Unix)?

<details>
<summary>Показать ответ</summary>
**pwd.getpwuid(uid)** возвращает запись о **пользователе** по числовому UID (имя, домашний каталог, shell и т.д.). **grp.getgrgid(gid)** — запись о **группе** по GID. Удобно при работе с **os.stat()**: **st_uid**/ **st_gid** — числа; для отображения имени пользователя/группы файла — **pwd.getpwuid(st.st_uid).pw_name**.
</details>

**108.** Чем **winreg** (Windows) отличается от хранения настроек в файле?

<details>
<summary>Показать ответ</summary>
**winreg** — доступ к **реестру Windows**: ключи **HKEY_LOCAL_MACHINE**, **HKEY_CURRENT_USER** и т.д. Настройки приложений и системы хранятся в реестре; другие программы и ОС ожидают их там. Файл — переносимо между ОС; реестр — интеграция с Windows (ассоциации файлов, автозапуск, настройки системы).
</details>

Удобно для автоматической аутентификации без хранения паролей в коде. Файл должен быть недоступен другим (chmod 600).

Ручной разбор строкой хрупок (вложенность, атрибуты, экранирование). **ElementTree** — стандартный способ работы с XML в stdlib; для больших файлов — **iterparse** для потоковой обработки.

### io, select, selectors, mmap

**120.** Чем **io.StringIO()** отличается от **io.BytesIO()**?

<details>
<summary>Показать ответ</summary>
**StringIO** — поток в памяти для **строк** (str); **read**/ **write** работают с текстом. **BytesIO** — поток для **байт** (bytes); бинарные данные. Оба реализуют интерфейс файла (read, write, seek). **StringIO** удобен для подстановки «файла» в API, ожидающее текстовый поток; **BytesIO** — для бинарных форматов (zip, изображения) без создания файла на диске.
</details>

**121.** Когда использовать **select.select(rlist, wlist, xlist, timeout)**?

<details>
<summary>Показать ответ</summary>
**select** — **мультиплексирование** I/O: ждёт, пока хотя бы один из файловых дескрипторов станет готов к чтению/записи/исключению, или истечёт **timeout**. Удобно для серверов с несколькими сокетами в одном потоке (без блокировки на одном **recv**). На Windows работает только с сокетами; **selectors** — высокоуровневая обёртка с автоматическим выбором **select**/ **epoll**/ **kqueue**.
</details>

**122.** Для чего используют **mmap.mmap(fileno, length, access=ACCESS_WRITE)**?

<details>
<summary>Показать ответ</summary>
**mmap** отображает **файл** (или часть) в память процесса: доступ по срезу **m[i:j]** как к байтам, изменения (при **ACCESS_WRITE**) записываются в файл. Удобно для больших файлов без загрузки целиком, совместного доступа между процессами (shared memory). Для случайного доступа к большим файлам — **mmap**; для последовательного чтения — обычный **read**.
</details>

**123.** Чем **ipaddress.ip_address('192.168.1.1')** отличается от хранения IP как строки?

<details>
<summary>Показать ответ</summary>
**ip_address** создаёт объект с проверкой **валидности** и типом (IPv4 или IPv6); атрибуты **version**, **packed** (bytes); сравнение, проверка вхождения в сеть (**in ip_network**). Строка не проверяется на валидность; арифметика сетей (подсети, хосты) — через **ipaddress**. Для парсинга и логики сетей — **ipaddress**; для простой передачи — строка.
</details>

**124.** Почему перед **itertools.groupby(iterable, key)** обычно нужна сортировка по тому же **key**?

<details>
<summary>Показать ответ</summary>
**groupby** объединяет только **подряд идущие** элементы с одинаковым ключом. Без сортировки элементы с одним ключом, разнесённые по итератору, попадут в разные группы. Сортировка по тому же key собирает одинаковые ключи подряд: **sorted(data, key=key); groupby(data, key)**.
</details>

**125.** Чем **queue.Queue** (модуль **queue**) отличается от **list** для обмена данными между потоками?

<details>
<summary>Показать ответ</summary>
**Queue** — **потокобезопасная** очередь: **put**/ **get** блокируются при переполнении/пустоте, поддерживают таймаут. **list** не потокобезопасен: append/pop из разных потоков без блокировок приводят к гонкам. Для producer–consumer между потоками — **queue.Queue**; для multiprocessing — **multiprocessing.Queue**.
</details>

**126.** Когда использовать **_thread.start_new_thread(func, (args,))** вместо **threading.Thread**?

<details>
<summary>Показать ответ</summary>
**_thread** — низкоуровневый API: нет объекта потока, нет **join**, нет исключений в главном потоке при падении дочернего. **threading.Thread** — объект потока, **join()**, возможность передать исключение. **_thread** — для минимальных сценариев или встроенных скриптов; в обычном коде — **threading.Thread** (или **concurrent.futures**).
</details>

## Отладка и диагностика stdlib

- **argparse:** неверные аргументы — **ArgumentError** или **SystemExit** (parse_args выходит при ошибке). Проверить имена опций (с **--** или без), **nargs**, **choices**. Для отладки смотреть **args** после **parse_args()**.
- **json:** при невалидном JSON — **json.JSONDecodeError** (атрибуты **lineno**, **colno**, **msg**). При сериализации несериализуемого объекта — **TypeError**; использовать **default** в **dumps** или свой **JSONEncoder**.
- **subprocess:** при ненулевом коде и **check=True** — **subprocess.CalledProcessError** (атрибуты **returncode**, **cmd**, **output**, **stderr**). При таймауте — **subprocess.TimeoutExpired**. При **shell=False** и передаче строки вместо списка на части платформ команда не найдена; передавать **list** аргументов.
- **pathlib:** **FileNotFoundError** при чтении несуществующего пути; **PermissionError** при отсутствии прав. Проверить **.exists()**, **.is_file()**; для записи — **.parent.mkdir(parents=True, exist_ok=True)**.
- **logging:** сообщения не выводятся — проверить уровень логгера и корневого (**logging.basicConfig(level=logging.DEBUG)**); проверить, что handlers добавлены к нужному логгеру. Дублирование сообщений — несколько handlers у одного логгера или у корневого.
- **typing:** аннотации не проверяются в runtime (кроме pydantic и т.п.). Запускать **mypy** или **pyright** для статической проверки. **get_type_hints()** из **typing** возвращает аннотации (с **from **future** import annotations** они могут быть строками).
- **dataclass:** **TypeError** при создании с пропущенным обязательным полем или при неверном порядке полей с default. Проверить порядок полей (сначала без default, потом с default); для мутабельных default — **field(default_factory=...)**.
- **itertools.groupby:** «пустые» или неожиданные группы — часто из‑за отсутствия сортировки. **groupby** объединяет только **подряд идущие** элементы с одинаковым key; перед **groupby** отсортировать по тому же key.

## Краткое повторение терминологии

| Термин               | Определение                                                                                                                                |
| -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------ |
| **View (dict)**      | Динамический вид на ключи/значения/пары словаря; не копия, изменения в dict видны в view; в Py3 возвращают .items(), .keys(), .values()    |
| **PurePath / Path**  | PurePath — манипуляции путями без I/O; Path — с операциями чтения/записи и ФС; Path = PosixPath или WindowsPath в зависимости от ОС        |
| **Type variance**    | Covariant (return type), contravariant (argument type), invariant (mutable container); важно для типобезопасности при подстановке типов    |
| **Type narrowing**   | Сужение типа после isinstance, is None, hasattr и т.п.; type checker учитывает это в ветках                                                |
| **TypeGuard**        | Функция-предикат с возвращаемым типом TypeGuard[T], сообщающая type checker о сужении типа аргумента до T                                  |
| **Protocol**         | Структурный подтип (PEP 544): совместимость по наличию методов, без явного наследования; @runtime_checkable для isinstance                 |
| **default_factory**  | Вызываемый объект без аргументов для значения по умолчанию в defaultdict/dataclass; избегает мутабельного default (один общий список/dict) |
| **Context manager**  | Объект с **enter** и **exit** или функция, обёрнутая в @contextmanager; используется в with для гарантированного входа/выхода              |
| **LRU cache**        | Кеш с вытеснением наименее недавно использованных записей при переполнении; lru_cache(maxsize=128)                                         |
| **Key function**     | Функция одного аргумента, возвращающая ключ для сравнения/сортировки; sorted(iterable, key=...)                                            |
| **cmp_to_key**       | Преобразует функцию сравнения двух аргументов (cmp) в key function для sorted/min/max                                                      |
| **Grapheme cluster** | Видимая единица текста для пользователя; может состоять из нескольких Unicode code points (эмодзи с модификаторами, буква + диакритика)    |
