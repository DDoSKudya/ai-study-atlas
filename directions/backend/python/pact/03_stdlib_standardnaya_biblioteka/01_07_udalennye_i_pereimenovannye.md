## 7. Удалённые и переименованные модули (Py2 → Py3)

При переходе с Python 2 на Python 3 ряд конструкций и модулей был удалён или переименован. Цель раздела — дать **полное понимание**: что именно изменилось, почему, как мигрировать и какие подводные камни возможны. Каждый пункт разобран с примерами, пошаговым объяснением и теоретическим обоснованием.

### Синтаксис и builtins

#### exec — функция, не инструкция

**Что такое exec и зачем он нужен**

- [ ] **exec** выполняет переданный код (строку или скомпилированный code object) в заданном или текущем контексте (глобальные и локальные переменные). Используется для динамического выполнения кода: плагины, конфигурация в виде кода, REPL-подобные сценарии. **Опасность:** выполнение произвольной строки от пользователя — риск инъекции; использовать только с доверенным вводом или в изолированном контексте.

**В Py2 vs Py3**

- [ ] **В Python 2** `exec` — **инструкция** (statement). Синтаксис: `exec code` или `exec code in globals, locals`. Скобки не использовались; передать exec как аргумент функции или переопределить в модуле было невозможно без манипуляций с компиляцией.
- [ ] **В Python 3** `exec` — **встроенная функция**. Сигнатура: `exec(object, globals=None, locals=None)`. Первый аргумент — строка с кодом или объект типа `code` (результат `compile()`). Второй и третий — словари контекста; если оба `None`, выполнение идёт в текущем глобальном и локальном контексте вызывающего кода.

**Пошагово: что происходит при вызове exec("x = 1")**

1. Интерпретатор получает строку `"x = 1"`.
2. При необходимости строка компилируется в bytecode (если передан code object, шаг компиляции пропускается).
3. Выполнение идёт в контексте: если переданы `globals`/`locals`, код видит и изменяет эти словари; иначе используются глобальные и локальные переменные текущего scope. Присваивание `x = 1` создаёт или обновляет имя `x` в соответствующем словаре.
4. Возвращаемое значение — `None` (exec не возвращает результат выражения, в отличие от `eval()`).

**Примеры**

```python
# Выполнение в текущем контексте
exec("x = 10")
print(x)  # 10

# Изолированный контекст: только переданные словари
g = {'__builtins__': __builtins__}
l = {}
exec("y = 2 + 2", g, l)
print(l['y'])   # 4
print('y' in dir())  # False — в текущем scope y нет

# Code object — один раз скомпилировать, много раз выполнить
code = compile("z = z + 1", "<inline>", "exec")
exec(code, {'z': 0}, {})  # в локальном l будет z=1; для повторного использования нужен общий dict
```

**Граничные случаи**

- Пустая строка: `exec("")` — допустимо; ничего не выполняется.
- Синтаксическая ошибка в строке: при выполнении выбрасывается `SyntaxError` (или при компиляции, если строка компилируется отдельно).
- Если передать только `globals` и не передать `locals`, по документации `locals` совпадает с `globals`; на практике для изоляции часто передают оба явно.

**Миграция с Python 2**

- Заменить `exec code` на `exec(code)`.
- Заменить `exec code in g, l` на `exec(code, g, l)`.
- Инструмент **2to3** выполняет эту замену автоматически.

#### print — функция (подробнее)

- [ ] **В Py2:** `print` — инструкция: `print x`, `print "a", "b"`, `print >> sys.stderr, "err"`. Нельзя передать как аргумент, переопределить в области видимости.
- [ ] **В Py3:** `print(*objects, sep=' ', end='\n', file=sys.stdout, flush=False)` — встроенная функция. Подробно: Часть I (§1.1), план §1.

**Пошагово: что происходит при print("A", "B", sep="; ", end="!")**

1. Позиционные аргументы **"A"** и **"B"** преобразуются в строки через **str()** (уже строки — без изменений).
2. Строки объединяются с разделителем **sep**: **"A" + "; " + "B"** → **"A; B"**.
3. В конец добавляется **end**: **"A; B" + "!"** → **"A; B!"**.
4. Результат записывается в **file.write("A; B!")** (по умолчанию **sys.stdout**).
5. Если **flush=True**, вызывается **file.flush()**. Функция возвращает **None**.

**Почему это важно:** единообразие с остальными builtins, возможность подмены в тестах (**builtins.print = mock_print**), явные параметры **file** и **flush** без специального синтаксиса **>>**.

#### <> удалён; только !=

- [ ] **В Py2:** оператор **`<>`** означал «не равно» (синоним **`!=`**). Оба допустимы.
- [ ] **В Py3:** **`<>`** удалён как избыточный; остаётся только **`!=`**. Причина: уменьшение дублирования; один явный способ выразить «не равно».

**Миграция:** заменить все вхождения `<>` на `!=` (2to3 делает это автоматически).

---

#### cmp(a, b) удалён

**Что такое cmp и зачем она была**

- [ ] **cmp(a, b)** в Py2 возвращала отрицательное число, если a < b, ноль при равенстве, положительное если a > b. Использовалась для сортировки и сравнений в стиле C (qsort). В Py3 встроенная функция **cmp** удалена: сравнения реализуются через богатые методы (****lt****, ****eq**** и т.д.), а сортировка — через **key** function, что эффективнее и явнее.

**Замена на уровне выражения**

- [ ] Выражение **`(a > b) - (a < b)`** даёт **-1**, **0** или **1**: булевы `a > b` и `a < b` в арифметике приводятся к 0/1, разность даёт нужный знак. Работает для любых типов, поддерживающих **<** и **>**.

**Замена для sorted() и других функций, принимающих key**

- [ ] **functools.cmp_to_key(cmp_func)** принимает функцию двух аргументов (как старая cmp) и возвращает key function: при сравнении двух элементов сортировка вызовет key(a) и key(b), а внутренне сравнивает их через переданную cmp-функцию. Таким образом старый код с **sorted(items, cmp=my_cmp)** переписывается в **sorted(items, key=functools.cmp_to_key(my_cmp))**; параметр **cmp** в **sorted** в Py3 удалён.

**Примеры**

```python
# Эмуляция cmp для двух чисел
def py2_cmp(a, b):
    return (a > b) - (a < b)
py2_cmp(1, 2)   # -1
py2_cmp(2, 2)   # 0
py2_cmp(3, 2)   # 1

# Сортировка по сложному сравнению (например, сначала по длине, потом по строке)
import functools
def my_cmp(a, b):
    if len(a) != len(b):
        return (len(a) > len(b)) - (len(a) < len(b))
    return (a > b) - (a < b)
sorted(["aa", "b", "c"], key=functools.cmp_to_key(my_cmp))  # ['b', 'c', 'aa']
```

**Почему это важно**

- В Py2 код вида **sorted(list, cmp=func)** был частым; в Py3 такой сигнатуры нет. Переход на **key=functools.cmp_to_key(func)** сохраняет поведение. Кроме того, key-based сортировка обычно быстрее (key вычисляется один раз на элемент), поэтому при возможности лучше переписать логику на **key=lambda x: ...**.

#### apply(f, args, kwargs) удалён

- [ ] **В Py2:** **apply(f, args, kwargs)** вызывала функцию **f** с позиционными аргументами из последовательности **args** и именованными из словаря **kwargs**. Удобно было для динамического вызова, когда аргументы собраны в списке и словаре.
- [ ] **В Py3:** **apply** удалена. Замена — прямой вызов с распаковкой: **f(\*args, **kwargs)**. Синтаксис **\*** и **\*\*** в вызовах функций как раз и предназначен для этого; **apply\*\* была избыточной обёрткой.

**Пример**

```python
# Py2: apply(f, (1, 2), {'x': 3})
# Py3:
def f(a, b, x=None): ...
f(*(1, 2), **{'x': 3})
# или f(1, 2, x=3)
```

---

#### обратные кавычки `x` (repr) удалены

- [ ] **В Py2:** обратные кавычки **\`x\`** были синонимом **repr(x)** — строковое представление объекта для отладки.
- [ ] **В Py3:** синтаксис удалён (редко использовался, усложнял парсер). Замена: везде писать **repr(x)**.

#### execfile('script.py') удалён

**Что делал execfile**

- [ ] **execfile(path)** в Py2 читал файл по пути и выполнял его код в текущем глобальном контексте. Удобно для загрузки конфигов в виде кода или скриптов без запуска отдельного процесса. Не устанавливал ****name**** в **'**main**'** и не добавлял путь скрипта в **sys.path** так, как это делает запуск **python script.py**.

**Замена в Py3**

1. **exec(open(path).read())** — минимальная замена: читаем файл, выполняем в текущем контексте. Не меняет ****name****, не добавляет каталог в **sys.path**; путь в ошибках будет **'<string>'**.
2. **exec(open(path, encoding='utf-8').read(), {'**name**': '**main**', ...})** — можно вручную задать контекст.
3. **runpy.run_path(path)** — рекомендуемый способ: устанавливает ****name**** в **'**main**'**, добавляет каталог файла в **sys.path**[0], обрабатывает ****file****, то есть поведение максимально близко к **python script.py**.

**Пример**

```python
import runpy
# Выполняет script.py как если бы он был запущен напрямую
runpy.run_path('script.py')
# С дополнительными глобальными переменными
runpy.run_path('script.py', init_globals={'config': my_config})
```

#### xrange → range; range теперь «ленивый»

**Что такое range и xrange в Py2**

- [ ] В **Python 2** **range(n)** возвращал **список** целых **[0, 1, ..., n-1]** — все числа создавались сразу, что расходовало память при больших n. **xrange(n)** возвращал объект **xrange** — ленивый итератор: числа порождались по одному при итерации, память O(1) по n.

**Что изменилось в Py3**

- [ ] Тип **xrange** удалён. Тип **range** в Py3 ведёт себя как старый xrange: это **не список**, а объект **range**, поддерживающий итерацию, **len()**, индексацию, срезы и проверку **in**; числа не хранятся в памяти целиком. Таким образом **range** в Py3 — это и «ленивый диапазон», и единственный способ задать диапазон индексов.

**Пошагово: что происходит при for i in range(10)**

1. **range(10)** создаёт объект range(0, 10) — границы 0 и 10 хранятся, элементы не создаются.
2. Итерация по range при каждом запросе следующего элемента вычисляет следующее целое (0, 1, ..., 9) и отдаёт его.
3. Память не зависит от размера диапазона (кроме самого объекта range и текущего значения итератора).

**Когда нужен именно список**

- [ ] **list(range(n))** — явно создать список из диапазона (например, чтобы менять элементы по индексу или многократно итерировать по одному и тому же списку).

**Миграция**

- Заменить **xrange(...)** на **range(...)**.
- Код, ожидающий от range список (например, присваивание **r[0] = 1**), в Py3 сломается — range неизменяем и не поддерживает присваивание по индексу. Нужно использовать **list(range(...))** или другую структуру данных.

#### dict.iteritems(), .iterkeys(), .itervalues() удалены; .items(), .keys(), .values() — view

**Что такое view в Py3**

- [ ] В **Python 3** методы **d.items()**, **d.keys()**, **d.values()** возвращают **view** объекты: динамический «вид» на текущее состояние словаря. Они не копируют данные: при изменении словаря (добавление/удаление ключей) вид сразу отражает изменения. View итерируемые; **keys view** ведёт себя как set по операциям (**&**, **|**, **-**, **^**); **items view** поддерживает проверку вхождения пары **(key, value)**.

**В Py2**

- [ ] В Py2 **d.items()**, **d.keys()**, **d.values()** возвращали **списки** (копии) — расход памяти и разрыв с текущим состоянием словаря после изменений. Для экономии памяти использовали **d.iteritems()**, **d.iterkeys()**, **d.itervalues()** — итераторы. В Py3 отдельные «iter»-методы убраны: один набор методов, возвращающий view, совмещает итерацию без копирования, и set-операции для keys/items.

**Примеры**

```python
d = {'a': 1, 'b': 2}
keys = d.keys()
values = d.values()
items = d.items()
# Итерация без копирования
list(keys)   # ['a', 'b']
d['c'] = 3
list(keys)   # ['a', 'b', 'c'] — вид обновился
# Keys view — set-подобные операции
d.keys() & {'a', 'x'}   # {'a'}
```

**Миграция**

- Заменить **d.iteritems()** на **d.items()**, **d.iterkeys()** на **d.keys()**, **d.itervalues()** на **d.values()**. Если код ожидал список (например, индексация **d.items()[0]**), в Py3 нужно явно: **list(d.items())[0]** или **next(iter(d.items()))**.

---

#### unicode удалён; str всегда Unicode

- [ ] **В Py2:** тип **unicode** — строка Unicode (последовательность code points); тип **str** — байты (или ASCII). Два типа строк, неявные преобразования и путаница при смешении.
- [ ] **В Py3:** тип **unicode** удалён; единственный тип «текстовой» строки — **str**, и он всегда представляет последовательность Unicode code points. Байты вынесены в тип **bytes**. Подробно модель текста и байт — Часть II (§6), план §6.

---

#### long удалён; int неограниченной точности

- [ ] **В Py2:** **int** — машинное целое (обычно 32 или 64 бита); при переполнении автоматически создавался **long** (произвольная точность). Два типа целых, суффикс **L** у литералов long.
- [ ] **В Py3:** тип **long** удалён; **int** — единственный целочисленный тип и он **неограниченной точности**. Переполнения нет; суффикс **L** у литералов не поддерживается. Подробно — Часть I (§1.3), план §2.

#### raw_input → input

**Что такое input и raw_input в Py2**

- [ ] В **Python 2** **raw_input(prompt)** читал строку с stdin (до перевода строки) и возвращал её как **str** (в Py2 — байты в системной кодировке). **input(prompt)** вызывал **eval(raw_input(prompt))** — то есть введённая строка **выполнялась как код**. Это создавало риск инъекции: пользователь мог ввести ****import**('os').system('rm -rf /')** и т.п. Для безопасного ввода числа в Py2 использовали **int(raw_input())** или **float(raw_input())**.

**Что изменилось в Py3**

- [ ] **raw_input** удалён. **input(prompt)** в Py3 ведёт себя как старый **raw_input**: читает строку с stdin и возвращает её **без eval**. То есть **input()** в Py3 — безопасный ввод строки. Для числа нужно явно **int(input())** или **float(input())**; при неверном вводе — **ValueError**.

**Пошагово: что происходит при x = input("N: ")**

1. Выводится строка **"N: "** (без перевода строки по умолчанию).
2. Чтение с **sys.stdin** до перевода строки (включённый перевод строки отбрасывается).
3. Возвращается строка (тип **str**). Присваивание **x = ...** сохраняет эту строку в переменную **x**. Если нужен int: **x = int(input("N: "))** — тогда **int()** преобразует строку в число; при вводе **"abc"** будет **ValueError**.

**Примеры**

```python
# Py3: ввод строки
name = input("Name: ")

# Py3: ввод числа (с обработкой ошибки)
try:
    n = int(input("N: "))
except ValueError:
    n = 0
```

**Миграция:** заменить **raw_input(...)** на **input(...)**. Код, использовавший **input()** для ввода числа (в Py2 **input()** возвращал результат eval), в Py3 нужно заменить на **int(input())** или **float(input())**; иначе арифметика со строкой даст TypeError или неверный результат.

#### Частые сценарии миграции Py2 → Py3

**Сценарий 1: скрипт загружает конфиг через execfile**

В Py2: `execfile('config.py')` — в config.py задаются переменные, которые потом читает основной код. В Py3: использовать **runpy.run_path('config.py', init_globals=globals())** если нужно пополнить текущий глобальный контекст, или **runpy.run_path('config.py')** и брать переменные из возвращённого словаря (run_path возвращает глобальные после выполнения). Альтернатива — явно импортировать модуль конфига: **import importlib.util; spec = importlib.util.spec_from_file_location("config", "config.py"); config = importlib.util.module_from_spec(spec); spec.loader.exec_module(config)** — тогда настройки в **config.var_name**.

**Сценарий 2: сортировка по сложному сравнению**

В Py2: `sorted(items, cmp=lambda a, b: ...)`. В Py3: `sorted(items, key=functools.cmp_to_key(lambda a, b: ...))`. Если логику можно выразить через ключ (например, сортировка по полю или по кортежу полей), предпочтительно **key=lambda x: (x.a, x.b)** — быстрее и понятнее.

**Сценарий 3: итерация по ключам/значениям словаря без копирования**

В Py2 использовали **d.iterkeys()**, **d.itervalues()**, **d.iteritems()** чтобы не создавать списки. В Py3 **d.keys()**, **d.values()**, **d.items()** уже возвращают view — копия не создаётся, итерация такая же эффективная. Замена: просто заменить **iterkeys** на **keys**, **itervalues** на **values**, **iteritems** на **items**. Если код делал **d.items()[0]** (индексацию), в Py3 нужно **list(d.items())[0]** или **next(iter(d.items()))**, так как view не поддерживает индексацию.

**Сценарий 4: проверка «это строка (str или bytes)»**

В Py2: `isinstance(x, basestring)`. В Py3: `isinstance(x, (str, bytes))`. Тип **basestring** удалён.

**Сценарий 5: чтение HTTP-страницы**

В Py2: `import urllib2; urllib2.urlopen(url).read()`. В Py3: `from urllib.request import urlopen; urlopen(url).read()`. Исключения: **urllib.error.URLError**, **urllib.error.HTTPError**; в Py2 были **urllib2.URLError**, **urllib2.HTTPError**.

**Сценарий 6: конфиг в формате INI**

В Py2: `from ConfigParser import ConfigParser; c = ConfigParser(); c.read('config.ini')`. В Py3: `import configparser; c = configparser.ConfigParser(); c.read('config.ini')`. Имена методов и семантика секций/опций совместимы.

#### Типичные ошибки при переходе Py2 → Py3

- **Забыть скобки у print:** в Py3 **print "x"** даёт **SyntaxError**. Нужно **print("x")**. Инструмент **2to3** исправляет автоматически.
- **Ожидать от range список:** в Py3 **range(10)** — не список; **r[0] = 1** вызовет **TypeError** (range не поддерживает присваивание). Нужно **list(range(10))** если нужен изменяемый список.
- **Индексировать d.items():** в Py3 **d.items()** возвращает view; **d.items()[0]** даёт **TypeError** (view не поддерживает индексацию). Нужно **list(d.items())[0]** или **next(iter(d.items()))**.
- **Использовать input() для числа:** в Py3 **input()** возвращает строку (как старый **raw_input**). Для числа нужно **int(input())** или **float(input())**. В Py2 **input()** делал **eval(raw_input())** — в Py3 такого нет.
- **Проверять тип файла через type(f) == file:** тип **file** в Py3 удалён. Проверять **isinstance(f, io.IOBase)** или наличие методов **read**/ **write**.
- **Оставить cmp в sorted:** в Py3 параметр **cmp** у **sorted()** удалён. Нужно **key=functools.cmp_to_key(my_cmp)** или переписать на **key=lambda x: ...**.
- **Импорт ConfigParser:** в Py3 модуль называется **configparser** (lowercase); класс **configparser.ConfigParser**. Импорт **from ConfigParser import ConfigParser** даёт **ModuleNotFoundError**.

#### Теоретическое обоснование миграции Py2 → Py3

- **Унификация exec и print как функций (PEP 3105):** в Python «действия» выражаются вызовами; инструкции остаются только для того, что нельзя выразить вызовом (присваивание, ветвление, циклы, объявление классов и функций). Это даёт переопределение в тестах, передачу как аргумента и единообразие с остальными builtins.
- **Один тип текста (str) и один тип целых (int):** уменьшение дублирования API и устранение типичных ошибок кодировок и переполнения целых.
- **View вместо копий и iter-методов:** экономия памяти и согласованность с протоколом итерации; один набор методов вместо двух (items/iteritems).
- **PEP 8 и структура пакетов:** имена модулей в lowercase (configparser, queue); разбиение urllib на request/parse/error улучшает явность импортов и зоны ответственности.
