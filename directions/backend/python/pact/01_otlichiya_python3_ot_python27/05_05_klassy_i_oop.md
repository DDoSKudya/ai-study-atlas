## 5. Классы и ООП

В Python 3 все классы неявно наследуют от **object**; синтаксис **class** и базовая модель (атрибуты, методы, наследование) совпадают с Python 2. Главные отличия: **super()** без аргументов, объявление метакласса через **class C(metaclass=Meta):**, порядок разрешения методов (MRO) по C3. Ниже каждый подраздел разобран подробно — с примерами, «как это работает внутри», типичными ошибками и миграцией с Py2.

---

### 5.1 Базовые механизмы

#### В чём отличие от Python 2.7

В **Python 2** существовали «старые» (classic) и «новые» (new-style) классы. Класс был new-style только если явно наследовал от **object** (или от другого new-style класса): **class C(object):**. У classic-классов не было \***\*new\*\***, **super()** работал иначе, MRO был другой. В **Python 3** все классы по умолчанию new-style: **class C:** эквивалентно **class C(object):** — наследование от **object** неявное. Тип **object** удалять нельзя; он корень иерархии.

#### Все классы наследуют от object по умолчанию

Любое объявление **class Имя:** в Py3 создаёт класс, наследующий от **object**. Проверка: **class A: pass** → **A.**bases\***\* равно **(object,)**. Это даёт каждому экземпляру стандартные методы (\*\***str\***\*, \*\***repr\***\*, \*\***eq\***\* и т.д.) и корректную работу **super()\*\* и MRO.

#### MRO (Method Resolution Order) — порядок поиска метода

При вызове **obj.method()** интерпретатор ищет **method** в классе объекта, затем в базовых классах в определённом порядке. Этот порядок называется **MRO**. В Python используется алгоритм **C3 linearization**: он строит один линейный порядок для всей иерархии так, что если класс A левее класса B в списке баз, то A и будет проверяться раньше B; при этом соблюдаются требования «локальный приоритет» и «монотонность» (родитель не идёт перед потомком в порядке, где потомок уже учтён). Результат MRO можно посмотреть: **C.mro()** или **C.**mro\***\* — это кортеж классов от самого класса до **object\*\*.

**Пример:**

```python
class A: pass
class B(A): pass
class C(A): pass
class D(B, C): pass

print(D.mro())
# [D, B, C, A, object]
```

Поиск метода у экземпляра **D()** идёт в порядке: **D** → **B** → **C** → **A** → **object**. Поэтому при множественном наследовании порядок баз в объявлении **class D(B, C):** важен: от него зависит, какой метод будет найден первым, если он определён и в **B**, и в **C**.

#### Как это работает внутри: поиск метода и вызов super()

**Пошагово при вызове obj.method():**

1. Интерпретатор берёт класс объекта: **cls = type(obj)** (например, **D**).
2. Ищет имя **method** в **cls.**dict\***\*; если найдено и это вызываемый объект (функция, дескриптор с \*\***get\*\*\*\* и т.д.) — используется он. Иначе переходит к следующему шагу.
3. Обходит классы в **MRO** (начиная со следующего после **cls**): для каждого класса в **cls.**mro**[1:]** проверяет **class.**dict**.get('method')**; как только находит непустое значение — использует его (с подстановкой **self** при вызове).
4. Если нигде не найдено — **AttributeError**.

**Пошагово при вызове super().method() внутри метода класса C:**

1. **super()** без аргументов в контексте метода экземпляра эквивалентен **super(**class**, self)**. \***\*class\*\*** — класс, в теле которого определена текущая функция (например, **C**); **self** — экземпляр (тип которого может быть подклассом **C**, например **D**).
2. **super(C, self)** возвращает **proxy-объект**, который при обращении к атрибуту (например, **method**) ищет его в **MRO** объекта **self**, но **начиная с класса, следующего после C** в MRO. То есть для **self** типа **D** с MRO **[D, B, C, A, object]** вызов **super(C, self).method()** ищет **method** в **A**, затем в **object** (пропуская **D**, **B**, **C**).
3. Найденный метод вызывается с тем же **self**. Поэтому цепочка **super().method()** из каждого класса в иерархии вызывает метод **ровно одного** родителя по MRO, без дублирования и без пропусков — это критично при множественном наследовании («ромб»).

**Пример «ромба» наследования (super() и порядок вызовов):**

```python
class A:
    def __init__(self):
        print("A.__init__")

class B(A):
    def __init__(self):
        print("B.__init__")
        super().__init__()

class C(A):
    def __init__(self):
        print("C.__init__")
        super().__init__()

class D(B, C):
    def __init__(self):
        print("D.__init__")
        super().__init__()

D()
# Выведет: D.__init__ → B.__init__ → C.__init__ → A.__init__
# MRO для D: [D, B, C, A, object]
```

Без **super()** при множественном наследовании пришлось бы вручную вызывать **A.**init**(self)** из **B** и **C**, и тогда **A.**init\***\* вызвался бы **дважды**. Благодаря **super()** каждый класс в MRO вызывается ровно один раз: из **D** вызывается **B.**init\*\***, из **B** — **C.**init\***\*, из **C** — **A.**init\*\***. В Py2 с явным **super(B, self).**init**()** и т.д. достичь того же порядка сложнее; в Py3 **super()** без аргументов даёт правильный порядок автоматически.

#### super() без аргументов (Python 3)

В **Python 2** для вызова метода родителя писали **super(CurrentClass, self).method(...)** — явно указывали текущий класс и экземпляр (или класс для classmethod). В **Python 3** внутри метода экземпляра или метода класса можно писать просто **super().method(...)** — интерпретатор сам подставляет текущий класс и **self** (или **cls** для classmethod). Это возможно благодаря тому, что при компиляции функции интерпретатор сохраняет ссылку на класс, в котором она определена (\***\*class\*\***), и при вызове **super()** без аргументов используется **super(**class**, self)** (в контексте метода экземпляра).

**Пример:**

```python
class B:
    def greet(self):
        return "B"

class C(B):
    def greet(self):
        return super().greet() + " -> C"

print(C().greet())   # "B -> C"
```

В Py2 пришлось бы писать **super(C, self).greet()**. При переименовании класса **C** в Py2 нужно было бы менять и вызов **super**; в Py3 имя класса в **super()** не фигурирует, поэтому рефакторинг проще.

#### super() и **class** в замыкании

Внутри вложенной функции или lambda, определённой в методе, **super()** без аргументов в Py3 всё равно может корректно работать: \***\*class\*\*** — это свободная переменная, которую компилятор подставляет в замыкание (cell), и она ссылается на класс, в теле которого определена функция. Поэтому даже при множественном наследовании и вызове через **super()** из вложенной функции порядок MRO соблюдается без явной передачи класса. В Py2 такой трюк с **super(CurrentClass, self)** во вложенной функции требовал явно передавать класс и экземпляр.

#### Миграция с Python 2 на Python 3

- Заменить **class C:** на **class C:** (без изменений) или оставить **class C(object):** — в Py3 это то же самое.
- Заменить все вызовы **super(CurrentClass, self).method(...)** на **super().method(...)**. Аналогично для classmethod: **super(CurrentClass, cls).method(...)** → **super().method(...)**.
- Проверить код на явное использование \***\*bases\*\*** или предположения о «classic» классах — в Py3 их нет.

#### Типичные ошибки

- **Забыть вызвать super().**init**()** в подклассе: если родитель инициализирует состояние в \***\*init\*\***, подкласс должен вызвать **super().**init**(...)** (с нужными аргументами), иначе часть состояния родителя не будет создана.
- **Перепутать порядок баз при множественном наследовании:** от порядка зависит MRO; если нужен метод из конкретного «ветки», проверьте **C.mro()** и при необходимости измените порядок баз или явно вызывайте метод нужного класса.

#### Наследование: одиночное и множественное (резюме)

**Наследование** — механизм, при котором подкласс получает атрибуты и методы базового класса и может добавлять свои или **переопределять** (override) существующие. В Python наследование задаётся в объявлении класса: **class B(A):** — **B** наследует от **A**; **class D(B, C):** — **D** наследует от **B** и **C** (множественное наследование).

- **Одиночное наследование:** один базовый класс; поиск метода идёт вверх по цепочке (**B** → **A** → **object**). Подкласс может переопределить метод родителя и при необходимости вызвать **super().method()** для доступа к реализации родителя.
- **Множественное наследование:** несколько базовых классов; порядок поиска задаётся **MRO** (C3). Порядок баз в объявлении **class D(B, C):** важен — от него зависит, какой метод будет найден первым при совпадении имён в **B** и **C**. Используйте **super()** без аргументов, чтобы цепочка вызовов (например, \***\*init\*\***) проходила по MRO ровно один раз по каждому классу (см. пример «ромба» выше).

**Когда наследовать:** когда отношение между классами — «**B** является разновидностью **A**» (is-a): подкласс расширяет или уточняет поведение базового. Не злоупотребляйте множественным наследованием: оно усложняет MRO и отладку; предпочитайте **композицию** (см. ниже), когда один объект **содержит** другой, а не «является» им.

#### Инкапсуляция: «приватные» атрибуты (\_name и \_\_name)

**Инкапсуляция** — сокрытие внутренней реализации объекта от внешнего кода; снаружи доступен только **публичный интерфейс**. В Python нет жёсткой приватности (как в C++ или Java); используются **соглашения** и **name mangling**.

- **Один подчёркивание \_name** — соглашение: атрибут считается «внутренним» (internal), не частью публичного API. Импорт **from M import \*** не подхватывает имена, начинающиеся с **\_**. Обращение **obj.\_name** технически возможно; смысл — «не используйте снаружи, может измениться».
- **Два подчёркивания \_\_name** — **name mangling**: интерпретатор переименовывает атрибут в **\_ИмяКласса\_\_name** (например, **\_C\_\_name**). Это защищает от случайного переопределения в подклассе: если в подклассе **D** определить **\_\_name**, будет создан **\_D\_\_name**, а не перезапись **\_C\_\_name**. Прямой доступ **obj.\_\_name** снаружи не сработает (атрибут хранится под другим именем); доступ изнутри класса — **self.\_\_name** — работает, так как компилятор подставляет переименованное имя. **Не путайте с дубль-подчёркиванием с обеих сторон** (\***\*init\*\***, \***\*str\*\***): это магические методы, их **не** переименовывают.

**Пошагово: как именно выполняется name mangling при наследовании.** (1) В классе **C** атрибут **\_\_name** при компиляции переименовывается в **\_C\_\_name** — имя берётся из имени класса, в теле которого объявлен атрибут. В **C.**dict\***\* хранится ключ **'\_C\_\_name'**, а не **'\_\_name'**. (2) В подклассе **D(C)** атрибут **\_\_name**, объявленный **в теле D**, переименовывается в **\_D\_\_name** — имя класса **D**. Поэтому в **D.**dict\*\*** будет **'\_D\_\_name'**, а в **C.**dict\***\* — **'\_C\_\_name'**; это **разные** ключи, переопределения по имени **\_\_name** не происходит — каждый класс имеет свой «приватный» слот. (3) При обращении **self.\_\_name** внутри метода класса **C** компилятор подставляет **self.\_C\_\_name**; внутри метода класса **D** — **self.\_D\_\_name**. Снаружи **obj.\_\_name** ищется как буквальное имя **'\_\_name'** — в **obj.**dict\*\*** и в классах по MRO такого ключа нет (там **'\_C\_\_name'** и т.д.), поэтому **AttributeError**. Итог: name mangling привязан к **имени класса**, в котором атрибут объявлен; подкласс получает свой слот с другим внутренним именем.

**Когда что использовать:** для «внутренних» полей и вспомогательных методов — **\_name**; для атрибутов, которые не должны случайно конфликтовать с именами в подклассах (например, миксины), — **\_\_name**. Публичный интерфейс — имена без подчёркивания; для контроля доступа на чтение/запись используйте **@property** (см. §5.2).

```python
class C:
    def __init__(self):
        self._internal = 1      # соглашение: «внутренний»
        self.__private = 2      # name mangling → _C__private

    def get_private(self):
        return self.__private   # внутри класса доступ по self.__private

c = C()
c._internal   # 1 — возможно, но не рекомендуется снаружи
# c.__private   # AttributeError — атрибут хранится как _C__private
c.get_private()   # 2
c._C__private     # 2 — доступ «в обход», возможен, но нарушает инкапсуляцию
```

#### Полиморфизм и переопределение методов

**Полиморфизм** — возможность использовать объекты разных классов через один и тот же интерфейс; один и тот же вызов метода (**obj.method()**) приводит к разному поведению в зависимости от типа **obj**. В Python полиморфизм достигается за счёт **переопределения методов** (override) в подклассах и за счёт **duck typing** («если объект крякает как утка — это утка»): код часто не проверяет тип, а полагается на наличие нужных методов.

- **Переопределение (override):** в подклассе объявляется метод с тем же именем, что и в базовом классе; при вызове **obj.method()** для экземпляра подкласса используется реализация подкласса. Вызвать реализацию родителя можно через **super().method()**.
- **Duck typing:** вместо **isinstance(obj, Duck)** часто пишут код, который просто вызывает **obj.quack()**; если у объекта есть нужный метод — код работает. Это упрощает подстановку разных типов (моков, адаптеров) без общей иерархии наследования. При необходимости контракта (обязательные методы) используют **abc** (abstract base classes) или протоколы (**typing.Protocol**).

**Пример полиморфизма через переопределение:**

```python
class Shape:
    def area(self):
        raise NotImplementedError

class Rectangle(Shape):
    def __init__(self, w, h):
        self.w, self.h = w, h
    def area(self):
        return self.w * self.h

class Circle(Shape):
    def __init__(self, r):
        self.r = r
    def area(self):
        return 3.14159 * self.r ** 2

def total_area(shapes):
    return sum(s.area() for s in shapes)   # полиморфизм: s.area() зависит от типа s

total_area([Rectangle(2, 3), Circle(1)])   # 6 + π ≈ 9.14
```

#### Композиция и наследование: когда что использовать

**Композиция** — включение одного объекта в другой как атрибут («**A** имеет **B**», has-a). **Наследование** — подкласс наследует от базового («**A** является **B**», is-a). Оба способа позволяют переиспользовать код; выбор зависит от семантики.

- **Наследование** уместно, когда подкласс — **разновидность** базового класса: **Dog(Animal)**, **SortedList(list)** (если список с гарантией порядка). Риски: глубокая иерархия и множественное наследование усложняют MRO; изменение базового класса влияет на все подклассы.
- **Композиция** уместна, когда объект **содержит** другой объект или **использует** его поведение: **class Car:** с атрибутом **self.engine = Engine()**; **class Report:** с **self.formatter = Formatter()**. Преимущества: можно менять «части» (другой двигатель, другой форматтер) без новой иерархии классов; тестировать проще (подставлять мок-объекты).

**Правило большого пальца:** если сомневаетесь, «наследование или композиция» — часто лучше **композиция**: включить объект как атрибут и вызывать его методы. Наследование используйте, когда есть явное отношение «is-a» и полиморфизм через переопределение методов.

```python
# Композиция: Car «имеет» Engine
class Engine:
    def start(self):
        return "Engine started"

class Car:
    def __init__(self):
        self.engine = Engine()   # композиция
    def start(self):
        return self.engine.start()

# Наследование: ElectricCar «является» разновидностью Car (если так задумано)
# Или снова композиция: Car имеет Battery вместо Engine — зависит от модели.
```

#### Атрибуты класса и атрибуты экземпляра

**Атрибут класса** — имя, определённое в теле класса (присваиванием **x = 1** или методом) или унаследованное от базового класса; хранится в **C.**dict\***\* (словарь класса). **Атрибут экземпляра** — имя, присвоенное экземпляру (**obj.x = 1**); хранится в **obj.**dict\*\*** (словарь экземпляра). При обращении **obj.x** интерпретатор **сначала** ищет **x** в **obj.**dict\***\*; если не найдено — ищет в **type(obj).**dict\*\*** и по MRO в базовых классах. При **присваивании** **obj.x = value** имя **x** всегда записывается в **obj.**dict\***\* (если в классе нет данных-дескриптора с \*\***set\***\*); при присваивании **C.x = value** — в **C.**dict\*\*** (меняется атрибут класса для всех экземпляров, у которых нет своего **x**).

**Пошагово: что происходит при чтении obj.x и при присваивании obj.x = value.**

1. **Чтение obj.x:** интерпретатор смотрит в **obj.**dict\***\*; если там есть ключ **'x'**, возвращается **obj.**dict**['x']**. Если нет — поиск идёт в **type(obj)** и по MRO; если в классе найден **дескриптор** с \*\***get\***\*, вызывается его \*\***get\***\*; иначе возвращается значение из **C.**dict**['x']**. Поэтому **атрибут класса** «виден» всем экземплярам, пока у экземпляра нет своего **x**; как только сделали **obj.x = 1**, у экземпляра появляется свой **x\*\*, и он «затеняет» атрибут класса для этого объекта.
2. **Присваивание obj.x = value:** если в классе (по MRO) есть **данные-дескриптор** с именем **x** и методом \***\*set\*\***, вызывается \***\*set**(obj, value)**; запись в **obj.**dict\*\*** не выполняется. Иначе значение записывается в **obj.**dict**['x']** — создаётся или обновляется **атрибут экземпляра**. Присваивание **obj.x = value** **никогда** не меняет **C.**dict\*\*\*\* — оно влияет только на данный экземпляр (или на дескриптор).

**Типичная ошибка:** изменить **изменяемый** атрибут класса (например, список), думая, что меняете только один экземпляр. Если **C.items = []** в классе, то **obj.items.append(1)** изменит **C.items** — один и тот же список у всех экземпляров. Нужно создавать атрибут экземпляра: в \***\*init\*\*** писать **self.items = []**, тогда у каждого экземпляра свой список.

```python
class C:
    shared = []           # атрибут класса — один список на весь класс
    def __init__(self):
        self.own = []     # атрибут экземпляра — свой список у каждого экземпляра

a, b = C(), C()
a.shared.append(1)
b.shared   # [1] — тот же список!
a.own.append(1)
b.own      # [] — свой список у b
```

**Граничные случаи: присваивание классу, удаление атрибута экземпляра.**

- **Присваивание C.x = value:** меняет **C.**dict**['x']** — атрибут класса. Все экземпляры, у которых **нет своего x** в **obj.**dict\***\*, при следующем обращении **obj.x** увидят новое значение. Экземпляры, у которых уже есть **obj.**dict**['x']** (созданный ранее **obj.x = ...\*\*), не изменятся — атрибут экземпляра «затеняет» атрибут класса.
- **Удаление атрибута экземпляра del obj.x:** удаляет ключ **'x'** из **obj.**dict\***\*. После этого при обращении **obj.x** снова будет использоваться атрибут класса (если он есть в **C** или в MRO). Поэтому **del obj.x\*\* не удаляет атрибут класса — только атрибут данного экземпляра.
- **Присваивание obj.x = value после C.x = value:** создаёт или обновляет **obj.**dict**['x']** — у этого экземпляра появляется свой **x**, который «затеняет» атрибут класса. **C.x** при этом не меняется — другие экземпляры по-прежнему видят **C.x**.

```python
class C:
    x = 10

a, b = C(), C()
a.x, b.x   # 10, 10 — оба видят атрибут класса
C.x = 20
a.x, b.x   # 20, 20 — оба видят обновлённый атрибут класса
a.x = 5    # у a появился свой атрибут экземпляра
a.x, b.x   # 5, 20 — у a свой x, у b по-прежнему C.x
del a.x    # удалили атрибут экземпляра у a
a.x        # 20 — снова виден атрибут класса
```

#### **bases**, **mro**, **subclasses** — интроспекция иерархии

- **C.**bases\***\* — кортеж **прямых** базовых классов класса **C**. Например, **class D(B, A): pass** → **D.**bases\*\*** равно **(B, A)**. Порядок совпадает с объявлением.
- **C.**mro\***\* — кортеж классов в порядке **MRO** (Method Resolution Order): сам класс, затем все предки в порядке C3 linearization до **object**. Используется при поиске атрибутов и при вызове **super()**. **C.mro()\*\* — метод, возвращающий тот же список (можно переопределить в метаклассе, но редко).
- **C.**subclasses**()** — метод класса, возвращающий **список прямых подклассов** класса **C** (те классы, у которых **C** в \***\*bases\*\***). Не возвращает «внуков»; рекурсивно обойти иерархию можно самому. Полезно для регистрации плагинов или автодокументации.

**Пример:**

```python
class A: pass
class B(A): pass
class C(A): pass
class D(B, C): pass

D.__bases__    # (B, C)
D.__mro__      # (D, B, C, A, object)
A.__subclasses__()   # [B, C] — прямые подклассы A; D не в списке (D — подкласс B и C)
```

#### Abstract Base Classes (abc) — контракт без реализации

Модуль **abc** позволяет объявлять **абстрактные** методы: метод без реализации, который **обязаны** реализовать подклассы. Класс, содержащий хотя бы один абстрактный метод (декорированный **@abstractmethod**), нельзя инстанцировать — попытка **C()** вызовет **TypeError**. Это задаёт **контракт**: все конкретные подклассы должны реализовать указанные методы.

**Использование:** наследуйте от **abc.ABC** (или задайте метакласс **abc.ABCMeta**) и помечайте методы декоратором **@abstractmethod**. Подкласс должен переопределить все абстрактные методы, иначе он тоже останется абстрактным и его нельзя будет создать.

```python
from abc import ABC, abstractmethod

class Shape(ABC):
    @abstractmethod
    def area(self):
        pass

    @abstractmethod
    def perimeter(self):
        pass

class Rectangle(Shape):
    def __init__(self, w, h):
        self.w, self.h = w, h
    def area(self):
        return self.w * self.h
    def perimeter(self):
        return 2 * (self.w + self.h)

# Shape()   # TypeError: Can't instantiate abstract class Shape
r = Rectangle(2, 3)   # OK — все абстрактные методы реализованы
```

**Когда использовать:** когда нужно явно зафиксировать интерфейс (набор методов) и гарантировать, что все «конкретные» классы его реализуют. Альтернатива — **typing.Protocol** (структурная подтипизация без наследования).

**Как это работает внутри: почему нельзя инстанцировать абстрактный класс.** При вызове **C()** (создание экземпляра) интерпретатор в итоге вызывает **type.**call**(C)** (или метакласс **C**). Метакласс **ABCMeta** перехватывает создание экземпляра: перед созданием объекта он проверяет, остались ли у класса **нереализованные** абстрактные методы (методы, помеченные **@abstractmethod** и не переопределённые в **C** или его предках по MRO). Проверка идёт по **C.**dict\***\* и MRO: если у метода есть атрибут \*\***isabstractmethod\***\* равный **True** и он не переопределён в подклассе — класс считается абстрактным. Если класс абстрактный, **ABCMeta** выбрасывает **TypeError: Can't instantiate abstract class ... with abstract method(s) ...**. Поэтому **Shape()** не сработает — у **Shape** есть абстрактные **area** и **perimeter**; **Rectangle(2, 3)** сработает — в **Rectangle\*\* оба метода переопределены.

**Пошагово: что проверяется при создании подкласса абстрактного класса.** (1) При определении **class Rectangle(Shape):** создаётся класс **Rectangle**; **Rectangle** наследует от **Shape**. (2) При первом создании экземпляра **Rectangle(2, 3)** метакласс (или механизм ABC) проверяет: для каждого абстрактного метода **Shape** (area, perimeter) ищется реализация в **Rectangle** по MRO — т.е. есть ли в **Rectangle.**dict\***\* или у предков метод с тем же именем, у которого \*\***isabstractmethod\***\* не **True**. (3) Если все абстрактные методы реализованы (переопределены и не помечены как abstract), создание разрешено. (4) Если подкласс **не** реализовал один из абстрактных методов, он сам остаётся абстрактным — попытка **SubClass()** вызовет **TypeError\*\* с перечислением недостающих методов.

**Плохо vs лучше и типичные ошибки (ABC):**

- **Плохо:** объявить абстрактный метод с телом **pass** и забыть переопределить в подклассе — подкласс тоже будет абстрактным, **SubClass()** вызовет **TypeError**. **Лучше:** явно реализовать все абстрактные методы в каждом конкретном подклассе.
- **Путать ABC с duck typing:** ABC требует **наследования** от базового класса и явной реализации методов; duck typing не требует наследования — достаточно наличия метода. Для «контракта без наследования» используйте **typing.Protocol**.
- **Добавлять абстрактный метод в уже используемый ABC:** все существующие подклассы станут абстрактными, пока не реализуют новый метод — ломает обратную совместимость. Вводите новые абстрактные методы осторожно или с дефолтной реализацией.

#### Миксины (mixins)

**Миксин** — класс, предназначённый **не** для использования сам по себе, а для **множественного наследования**: он добавляет одно или несколько методов (поведение) другим классам. Обычно миксин не наследуется от «бизнес»-класса, а только от **object** (или другого миксина); в объявлении подкласса его ставят **не первым** в списке баз, чтобы MRO был предсказуемым. Имя часто заканчивают на **Mixin** (**LoggingMixin**, **SerializableMixin**).

**Зачем:** переиспользовать поведение (логирование, сериализация, сравнение) без дублирования кода; один миксин можно «подмешать» к разным базовым классам. **Осторожность:** при множественном наследовании порядок баз определяет MRO; миксины не должны предполагать, что они «единственный» предок; используйте **super()**, чтобы цепочка вызовов проходила по MRO.

```python
class JsonMixin:
    def to_json(self):
        import json
        return json.dumps(self.__dict__)

class LoggingMixin:
    def log(self, msg):
        print(f"{self.__class__.__name__}: {msg}")

class Person(LoggingMixin, JsonMixin):
    def __init__(self, name):
        self.name = name

p = Person("Alice")
p.log("hello")      # Person: hello
p.to_json()         # '{"name": "Alice"}'
```

**Пошагово: порядок баз и MRO при использовании миксинов.** При объявлении **class Person(LoggingMixin, JsonMixin):** порядок баз определяет MRO: **Person** → **LoggingMixin** → **JsonMixin** → **object**. Поиск метода **p.log** идёт: сначала **Person** (нет **log**), затем **LoggingMixin** (есть **log**) — используется он. Поиск **p.to_json**: **Person** → **LoggingMixin** (нет **to_json**) → **JsonMixin** (есть **to_json**). Миксин обычно ставят **после** основного базового класса, если он есть: **class Person(Base, LoggingMixin):** — тогда MRO **Person** → **Base** → **LoggingMixin** → ...; методы из **Base** имеют приоритет над миксином. Если миксин должен вызывать **super()** и передавать управление дальше по MRO, порядок баз критичен: следующий класс в MRO должен быть тем, кто обработает вызов (другой миксин или базовый класс).

**Типичные ошибки (миксины):**

- **Забыть super() в миксине, когда он часть цепочки инициализации:** если миксин определяет \***\*init\*\*** и класс наследует ещё один класс с \***\*init\*\***, нужно вызывать **super().**init**(**kwargs)** и передавать **kwargs**, иначе \*\***init\***\* следующего класса в MRO не вызовется. **Лучше:** в миксине писать **def **init**(self, **kwargs): ...; super().**init**(**kwargs)\*\*.
- **Полагаться на порядок атрибутов между миксинами:** миксины не должны предполагать, что у **self** уже есть атрибуты, установленные «другим» миксином, если порядок MRO не зафиксирован. Инициализируйте нужные атрибуты в своём \***\*init\*\*** или документируйте требуемый порядок баз.
- **Делать миксин единственным базовым классом с тяжёлой логикой:** миксин лучше оставлять «тонким» (несколько методов); основную иерархию наследования задаёт «главный» базовый класс.

### 5.2 Декораторы методов

#### В чём отличие

В Py2 и Py3 декораторы **@staticmethod**, **@classmethod**, **@property** работают одинаково. Важно понимать, когда какой использовать и как они влияют на первый аргумент метода.

#### @staticmethod — метод без self и без cls

**@staticmethod** применяется к методу класса так, что при вызове метод **не получает** ни **self** (экземпляр), ни **cls** (класс) первым аргументом. Вызывать можно и как **C.f()**, и как **obj.f()** — в метод передаются только те аргументы, которые вы явно указали. Удобно для функций, которые логически относятся к классу (например, фабрика или хелпер), но не обращаются к экземпляру или классу.

**Пример:**

```python
class MyClass:
    @staticmethod
    def create_from_pair(a, b):
        return MyClass(a + b)   # или другой конструктор

    def __init__(self, value):
        self.value = value

obj = MyClass.create_from_pair(1, 2)   # вызов без экземпляра
```

Если бы **create_from_pair** был обычным методом, при вызове **MyClass.create_from_pair(1, 2)** первый аргумент попал бы в **self**, и **a** получило бы 1, **b** — 2 только при вызове от экземпляра с тремя аргументами. Со **staticmethod** аргументы (1, 2) попадают в (a, b).

#### @classmethod — первый аргумент cls (класс)

**@classmethod** делает так, что при вызове метода первым аргументом передаётся **класс** (обычно именуют **cls**), а не экземпляр. Вызывать можно как **C.f()** или **obj.f()**; в обоих случаях в **cls** попадёт класс **C** (или класс объекта **obj**). Используется для **фабричных методов** и **альтернативных конструкторов**: внутри метода можно вызывать **cls(...)** и создавать экземпляр текущего класса (или подкласса), не привязываясь к имени класса.

**Пример:**

```python
class Point:
    def __init__(self, x, y):
        self.x, self.y = x, y

    @classmethod
    def from_polar(cls, r, phi):
        import math
        return cls(r * math.cos(phi), r * math.sin(phi))

p = Point.from_polar(1, 0)   # Point(1.0, 0.0)
```

Если у вас есть подкласс **Point3D(Point)** и вы вызовете **Point3D.from_polar(1, 0)**, то **cls** будет **Point3D**, и **cls(...)** создаст именно **Point3D**, а не **Point**. С **@staticmethod** пришлось бы явно писать **Point3D(...)** и терять полиморфизм.

#### @property — доступ к атрибуту как к полю с логикой

**@property** превращает метод в **геттер**: обращение к **obj.x** вызывает метод (например, **x(self)**), а не прямой доступ к \***\*dict\*\***. Дополнительно можно задать **setter** и **deleter**: **x.setter**, **x.deleter** — так при присваивании **obj.x = value** или **del obj.x** вызываются заданные методы. Это позволяет проверять и преобразовывать значения, считать производные атрибуты или скрывать внутреннее представление.

**Пример:**

```python
class Circle:
    def __init__(self, radius):
        self._radius = radius

    @property
    def radius(self):
        return self._radius

    @radius.setter
    def radius(self, value):
        if value < 0:
            raise ValueError("radius must be non-negative")
        self._radius = value

    @property
    def area(self):
        return 3.14159 * self._radius ** 2   # только чтение, setter не задан

c = Circle(5)
print(c.radius, c.area)   # 5 и площадь
c.radius = 10
# c.area = 100   # AttributeError: can't set attribute
```

**Плохо vs лучше:** хранить «сырое» значение в приватном атрибуте (**self.\_radius**) и наружу выдавать через **@property** с проверками — лучше, чем давать прямой доступ к **self.radius** и проверять в каждом месте присваивания.

#### Как это работает внутри: вызов property и декораторов

**При обращении obj.x, если x — property:**

1. Интерпретатор ищет **x** в классе объекта; находит дескриптор **property**.
2. Вызывается **property.**get**(obj, type(obj))** — первый аргумент **instance** = **obj**, второй **owner** = класс.
3. Внутри **property.**get\***\* вызывается ваша функция-геттер, переданная в **property(getter)**, с аргументом **obj**; результат возвращается как значение **obj.x\*\*.

**При присваивании obj.x = value, если x — property с сеттером:**

1. Интерпретатор находит в классе дескриптор **property** с методом \***\*set\*\*** (данные-дескриптор).
2. Вызывается **property.**set**(obj, value)**; внутри вызывается ваш сеттер с аргументами **obj** и **value**.
3. Запись в **obj.**dict**['x']** при этом **не** выполняется — данные-дескриптор имеет приоритет над \***\*dict\*\*** экземпляра.

**Плохо vs лучше (staticmethod и имя класса):** если фабричный метод создаёт экземпляр **жёстко по имени класса** (**return MyClass(...)**), при наследовании подкласс получит экземпляр базового класса, а не свой. Лучше использовать **@classmethod** и **return cls(...)** — тогда из подкласса вызовется **SubClass.factory()** и **cls** будет **SubClass**, экземпляр создастся правильного типа.

```python
# Плохо: staticmethod с жёстким именем класса
class Point:
    @staticmethod
    def from_polar(r, phi):
        import math
        return Point(r * math.cos(phi), r * math.sin(phi))   # всегда Point

class Point3D(Point):
    pass
p = Point3D.from_polar(1, 0)   # p — Point, а не Point3D!

# Лучше: classmethod с cls
class Point:
    @classmethod
    def from_polar(cls, r, phi):
        import math
        return cls(r * math.cos(phi), r * math.sin(phi))
p = Point3D.from_polar(1, 0)   # p — Point3D
```

#### Когда что использовать

| Декоратор         | Первый аргумент      | Типичное использование                                |
| ----------------- | -------------------- | ----------------------------------------------------- |
| без декоратора    | **self** (экземпляр) | обычный метод экземпляра                              |
| **@staticmethod** | нет                  | хелперы, фабрики без полиморфизма по классу           |
| **@classmethod**  | **cls** (класс)      | фабрики, альтернативные конструкторы, **cls(...)**    |
| **@property**     | **self** (в геттере) | вычисляемый/проверяемый атрибут, доступ как **obj.x** |

#### Типичные ошибки

- **Путать classmethod и staticmethod:** если метод создаёт экземпляр и должен поддерживать подклассы, используйте **@classmethod** и **cls(...)**; **@staticmethod** не знает о подклассах.
- **В property делать тяжёлые вычисления или I/O при каждом обращении:** геттер вызывается при каждом **obj.x**; при необходимости кешируйте результат (например, в приватном атрибуте при первом обращении) или явно сделайте метод **get_x()**, чтобы вызывающий понимал стоимость.

### 5.3 Магические методы (dunder)

**Магические методы** (dunder — double underscore) — методы с именами вида \***\*имя\*\***; интерпретатор вызывает их в определённых ситуациях (операторы, **len()**, **str()**, циклы и т.д.). В Py2 и Py3 набор и семантика совпадают; ниже — ключевые для обучения и миграции.

#### **init** vs **new** — создание объекта

\***\*new**(cls, ...)** вызывается **первым** при создании экземпляра; он должен **создать и вернуть** объект (обычно экземпляр **cls**). \*\***init**(self, ...)** вызывается **после** \***\*new\*\*** и получает уже созданный экземпляр **self**; он инициализирует атрибуты и не возвращает значение. По умолчанию **object.**new**(cls)** создаёт «пустой» экземпляр; **object.**init**(self)** ничего не делает.

**Когда переопределять **new**:** для **неизменяемых** типов (например, подкласс **str** или **int**), когда нужно контролировать сам факт создания объекта (синглтон, пул объектов), или когда конструктор должен возвращать объект другого типа. В большинстве классов достаточно только \***\*init\*\***.

**Пример синглтона:**

```python
class Singleton:
    _instance = None
    def __new__(cls):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
        return cls._instance
```

#### **str** vs **repr** — строковое представление

**str(obj)** вызывает **obj.**str**()** — представление для **пользователя** (читаемое, может быть неоднозначным). **repr(obj)** вызывает **obj.**repr**()** — представление для **отладки**; по соглашению должно быть однозначным и по возможности таким, что **eval(repr(obj))** воспроизводит объект. Если \***\*str\*\*** не определён, используется \***\*repr\*\***. В интерактивной среде при выводе объекта по умолчанию вызывается **repr**.

**Пример:**

```python
class Point:
    def __init__(self, x, y):
        self.x, self.y = x, y
    def __repr__(self):
        return f"Point({self.x}, {self.y})"
    def __str__(self):
        return f"({self.x}, {self.y})"

p = Point(1, 2)
print(p)        # (1, 2)   — __str__
print(repr(p))  # Point(1, 2)   — __repr__
```

**Плохо vs лучше:** задавать \***\*repr\*\*** так, чтобы по нему можно было понять тип и ключевые данные; \***\*str\*\*** — коротко для вывода пользователю. Не оставлять только \***\*str\*\*** без \***\*repr\*\*** — в трассировках и отладке будет использоваться наследованный **object.**repr\*\*\*\*, часто бесполезный.

#### **eq** и **hash** — контракт hashable

**obj1 == obj2** вызывает **obj1.**eq**(obj2)** (если не переопределён — сравнение по идентичности **is**). Если вы переопределяете \***\*eq\*\***, то по контракту **hashable** объекты, для которых \***\*eq\*\*** возвращает **True**, должны иметь **одинаковый** \***\*hash\*\***. Иначе такой объект нельзя безопасно использовать как ключ **dict** или элемент **set**. По умолчанию **object** реализует \***\*eq\*\*** как сравнение по **id** и \***\*hash\*\*** по **id**; если вы переопределяете \***\*eq\*\*** и хотите, чтобы экземпляры были hashable, переопределите \***\*hash\*\*** так, чтобы равные объекты давали один хеш. Для **неизменяемых** типов (значение не меняется после создания) это естественно; для изменяемых типов обычно **не** определяют \***\*hash\*\*** (или явно задают \***\*hash** = None\*\*), чтобы объект не был hashable.

**Пример:**

```python
class Key:
    def __init__(self, name):
        self.name = name
    def __eq__(self, other):
        return isinstance(other, Key) and self.name == other.name
    def __hash__(self):
        return hash(self.name)

# Теперь Key можно использовать как ключ dict
d = {Key("a"): 1}
```

#### Как это работает внутри: **eq** и **hash** при добавлении в set/dict

**Пошагово при добавлении obj в множество (s.add(obj)) или при использовании obj как ключа (d[obj] = value):**

1. Интерпретатор вызывает **hash(obj)** → **obj.**hash**()**. Хеш должен быть целым числом; объекты, сравниваемые как равные (**obj1 == obj2** → **True**), должны иметь **одинаковый** хеш. Если \***\*hash\*\*** не определён или равен **None**, объект считается неhashable — добавление в **set** или использование как ключа **dict** вызовет **TypeError: unhashable type**.
2. По хешу находится «ведро» (bucket) в хеш-таблице; при коллизии (одинаковый хеш у разных объектов) для проверки «уже есть такой элемент» вызывается **obj.**eq**(other)** для сравнения с уже хранящимися в этом ведре объектами.
3. Если \***\*eq\*\*** возвращает **True** для какого-то элемента — дубликат не добавляется (в set) или перезаписывается значение (в dict). Поэтому контракт: **если a == b, то hash(a) == hash(b)**. Иначе равные объекты могут оказаться в разных ведрах, и множество/dict «увидит» их как разные.

**Для изменяемых типов (list, dict, свой класс с изменяемыми полями) обычно не определяют **hash**:** если объект изменится после добавления в set, его хеш не должен меняться (иначе он «потеряется» в множестве). Поэтому для таких классов задают \***\*hash** = None** — тогда объект явно неhashable, и попытка **set.add(obj)** или **d[obj] = ...** вызовет **TypeError\*\*.

```python
class MutableKey:
    def __init__(self, name):
        self.name = name
    def __eq__(self, other):
        return isinstance(other, MutableKey) and self.name == other.name
    __hash__ = None   # явно неhashable

# {MutableKey("a"): 1}   # TypeError: unhashable type: 'MutableKey'
```

#### **len**, **getitem**, **iter** — протокол контейнера

**len(obj)** вызывает **obj.**len**()**; должен возвращать неотрицательное целое. **obj[i]** и **obj[i:j]** вызывают **obj.**getitem**(key)** (key — индекс или slice). **for x in obj** сначала вызывает **iter(obj)** → **obj.**iter**()**; итератор должен реализовать \***\*next\*\***. Если \***\*iter\*\*** не определён, используется fallback: вызов \***\*getitem**(0)**, \*\***getitem**(1)**, ... до **IndexError**. Этого достаточно для простых последовательностей.

**Пример минимального контейнера:**

```python
class MyList:
    def __init__(self, data):
        self._data = list(data)
    def __len__(self):
        return len(self._data)
    def __getitem__(self, key):
        return self._data[key]
    def __iter__(self):
        return iter(self._data)
```

#### **enter** и **exit** — контекстный менеджер

**with obj:** вызывает **obj.**enter**()** при входе и **obj.**exit**(exc_type, exc_val, exc_tb)** при выходе (при выходе по исключению аргументы содержат данные об исключении; при нормальном выходе все три **None**). \***\*exit\*\*** может вернуть **True**, чтобы «проглотить» исключение; иначе исключение пробрасывается дальше. Так реализованы **with open(...)** и **with lock:**.

**Пример:**

```python
class Timer:
    def __enter__(self):
        self.start = time.time()
        return self
    def __exit__(self, et, ev, tb):
        self.elapsed = time.time() - self.start
        return False
```

#### **call** — callable объект

**obj(...)** вызывает **obj.**call**(\*args, **kwargs)**. Объект с \*\***call\***\* можно вызывать как функцию; так работают декораторы без скобок (объект с \*\***call\***\*), **functools.partial\*\* и т.д.

**Пример:**

```python
class Multiplier:
    def __init__(self, factor):
        self.factor = factor
    def __call__(self, x):
        return x * self.factor

double = Multiplier(2)
print(double(5))   # 10
```

#### **getattr**, **getattribute**, **setattr**, **delattr** — перехват доступа к атрибутам

- \***\*getattr**(self, name)** — вызывается **только когда** атрибут **name\*\* **не найден** обычным поиском (ни в **obj.**dict\***\*, ни в классе по MRO). Используется как **fallback**: «виртуальные» атрибуты, прокси, ленивая загрузка. Если атрибут найден в \*\***dict\***\* или в классе, \*\***getattr\*\*\*\* не вызывается.
- \***\*getattribute**(self, name)** — вызывается **при каждом** обращении к атрибуту **obj.name** (чтение). Если здесь не вернуть значение и не пробросить исключение, поиск не продолжается; внутри \*\***getattribute\***\* обращение к **self.attr** снова вызовет \*\***getattribute\***\* — риск **бесконечной рекурсии**. Поэтому внутри обычно вызывают **object.**getattribute**(self, name)** или **super().**getattribute**(name)\*\*. Используется редко: для прокси, логирования доступа, запрета части атрибутов.
- \***\*setattr**(self, name, value)** — вызывается **при каждом** присваивании **obj.name = value**. Аналогично риск рекурсии: присваивание **self.x = value** внутри \*\***setattr\***\* снова вызовет \*\***setattr\***\*. Присваивание делают через **object.**setattr**(self, name, value)** или **self.**dict**[name] = value\*\*.
- \***\*delattr**(self, name)** — вызывается при **del obj.name**. Те же правила: внутри использовать **object.**delattr**(self, name)** или **del self.**dict**[name]\*\*.

**Различие **getattr** и **getattribute**:** \***\*getattr\*\*** — только когда атрибут **не найден**; \***\*getattribute\*\*** — **всегда** при чтении. Для «дополнительных» атрибутов по умолчанию удобнее \***\*getattr\*\***; \***\*getattribute\*\*** нужен, когда нужно перехватывать **все** обращения (в том числе к существующим атрибутам).

**Пример: fallback через **getattr\*\*\*\*

```python
class Lazy:
    def __getattr__(self, name):
        if name == "heavy":
            value = expensive_computation()
            object.__setattr__(self, "heavy", value)   # кеш в экземпляре
            return value
        raise AttributeError(f"{type(self).__name__!r} object has no attribute {name!r}")
```

**Пример: **setattr** с проверкой (избегаем рекурсии)**

```python
class Guarded:
    def __setattr__(self, name, value):
        if name == "forbidden":
            raise AttributeError("cannot set 'forbidden'")
        object.__setattr__(self, name, value)   # иначе рекурсия при self.x = value
```

#### Типичные ошибки (магические методы)

- **В **eq** не учитывать тип:** сравнивайте **isinstance(other, MyClass)** и только потом поля; иначе **MyClass() == None** может вернуть **False** без ошибки, но **None == MyClass()** вызовет **None.**eq**(MyClass())**, который может вернуть **NotImplemented** или некорректный результат. Лучше явно проверять тип.
- **Переопределить **eq** и забыть **hash**:** если объекты используются как ключи или в set, задайте \***\*hash\*\*** или явно \***\*hash** = None\*\* (тогда объект не hashable).
- **В **getattribute** или **setattr** обращаться к self без object.**getattribute**/**setattr**:** приведёт к бесконечной рекурсии. Всегда используйте **object.**getattribute**(self, name)** или **self.**dict**[name]** для чтения и **object.**setattr**(self, name, value)** или **self.**dict**[name] = value** для записи внутри этих методов.

### 5.4 Дескрипторы

**Дескриптор** — объект класса, реализующий хотя бы один из методов \***\*get\*\***, \***\*set\*\***, \***\*delete\*\***. Когда атрибут класса является дескриптором, доступ к **одноимённому** атрибуту **экземпляра** (чтение, запись, удаление) перехватывается дескриптором: вызывается \***\*get\*\***, \***\*set\*\*** или \***\*delete\*\*** с соответствующими аргументами. Так реализованы **property**, **classmethod**, **staticmethod** и многие механизмы в стандартной библиотеке.

#### Как это работает внутри

При обращении **obj.x** (чтение) интерпретатор ищет **x** в классе объекта; если найден дескриптор с \***\*get\*\***, вызывается **type(obj).**dict**['x'].**get**(obj, type(obj))**. Первый аргумент — **instance** (экземпляр или **None** при доступе через класс), второй — **owner** (класс). При присваивании **obj.x = value** вызывается \***\*set**(obj, value)**; при **del obj.x** — \*\***delete**(obj)**. **Данные-дескриптор** (дескриптор с \***\*set\*\***) имеет приоритет над записью в **obj.**dict\***\*: даже если у экземпляра уже есть **obj.**dict**['x']**, при чтении **obj.x** выигрывает дескриптор в классе. **Не-данные-дескриптор** (только \*\***get\***\*) проигрывает атрибуту экземпляра: если есть **obj.**dict**['x']\*\*, вернётся он.

**Пошагово: приоритет дескриптора vs **dict** экземпляра**

1. **Чтение obj.x:** интерпретатор сначала смотрит в **type(obj).**dict\***\* (и в базовых классах по MRO). Если **x** найден и у него есть \*\***get\***\* (дескриптор), вызывается **x.**get**(obj, type(obj))** — результат и есть значение **obj.x**. Атрибут **obj.**dict**['x']** при этом **не** проверяется для данных-дескрипторов: дескриптор в классе имеет приоритет. Если в классе **x** — не-данные-дескриптор (только \*\***get\***\*, без \*\***set\***\*), то сначала проверяется **obj.**dict\*\***: если там есть **'x'**, возвращается **obj.**dict**['x']**; иначе вызывается **x.**get**(obj, type(obj))**.
2. **Запись obj.x = value:** если в классе **x** — данные-дескриптор (есть \***\*set\*\***), вызывается **x.**set**(obj, value)**; запись в **obj.**dict\***\* не выполняется. Если дескриптора нет или у него нет \*\***set\***\*, значение записывается в **obj.**dict**['x']\*\*.
3. **Итог:** данные-дескриптор (property с сеттером, наш **Positive**) всегда «перехватывает» чтение и запись; атрибут экземпляра с тем же именем не создаётся при присваивании через дескриптор (значение хранится в **obj.**dict\***\* под другим ключом, как в примере **Positive**, где имя слота задаётся в **self.name\*\*).

**Пример: данные-дескриптор vs атрибут экземпляра**

```python
class Desc:
    def __get__(self, obj, owner):
        return obj.__dict__.get('_x', 0)
    def __set__(self, obj, value):
        obj.__dict__['_x'] = value

class C:
    x = Desc()

o = C()
o.x = 5        # вызывается Desc.__set__(o, 5) → o.__dict__['_x'] = 5
print(o.x)    # вызывается Desc.__get__(o, C) → 5
o.__dict__['x'] = 99   # записали в __dict__ напрямую
print(o.x)    # всё равно 5 — при чтении выигрывает дескриптор (данные-дескриптор)
```

#### property, classmethod, staticmethod — реализованы как дескрипторы

**property** — это дескриптор: при **obj.x** вызывается **property.**get**(obj, type(obj))**, который внутри вызывает вашу функцию-геттер. **classmethod** и **staticmethod** при **obj.f** или **C.f** вызывают \***\*get\*\***; **classmethod** подставляет класс в первый аргумент, **staticmethod** не подставляет ничего. Поэтому вы можете писать **obj.x** и получать вызов метода — это не «магия», а протокол дескрипторов.

#### Пример простого дескриптора (валидация)

```python
class Positive:
    def __init__(self, name):
        self.name = name
    def __get__(self, obj, owner):
        if obj is None:
            return self
        return obj.__dict__.get(self.name, 0)
    def __set__(self, obj, value):
        if value < 0:
            raise ValueError("must be >= 0")
        obj.__dict__[self.name] = value

class Box:
    width = Positive("width")
    height = Positive("height")
    def __init__(self, w, h):
        self.width = w
        self.height = h

b = Box(2, 3)
# b.width = -1   # ValueError
```

Здесь **Positive** хранит имя атрибута (**"width"**), а значение — в **obj.**dict\***\*; \*\***set\*\*\*\* проверяет значение перед записью. Так можно переиспользовать одну и ту же логику для нескольких атрибутов.

#### Типичные ошибки (дескрипторы)

- **Хранить значение в дескрипторе и забыть про экземпляр:** если хранить в **self.value** внутри дескриптора, все экземпляры класса разделяют одно значение. Обычно значение хранят в **obj.**dict**[self.name]** или в отдельном словаре по **id(obj)**.
- **Путать вызов **get** при доступе через класс:** при **C.x** в \***\*get\*\*** передаётся **instance=None**; часто в этом случае возвращают сам дескриптор (**return self**), чтобы не ломать интроспекцию.

### 5.5 **slots**

\***\*slots\*\*** — атрибут класса, задающий **фиксированный набор имён атрибутов экземпляра**. Интерпретатор не создаёт \***\*dict\*\*** для экземпляров такого класса; вместо этого резервируется место только для перечисленных имён. Это **экономит память** (особенно при большом числе мелких объектов) и **запрещает** добавлять атрибуты, не указанные в \***\*slots\*\*** (присваивание **obj.new_attr = ...** вызовет **AttributeError**).

#### Как это работает внутри: создание экземпляра и доступ при **slots**

**Пошагово при создании экземпляра класса со **slots**:** (1) При вызове **C()** под экземпляр выделяется структура с **фиксированным числом слотов** — по одному на каждое имя из \***\*slots\*\***; \***\*dict\*\*** не создаётся (если только **'**dict**'** не включён в \***\*slots\*\***). (2) При присваивании **obj.x = value** интерпретатор проверяет, есть ли **x** в \***\*slots\*\*** класса; если да — значение записывается в соответствующий слот; если нет — **AttributeError**. (3) При чтении **obj.x** значение берётся из слота; имена в \***\*slots\*\*** реализованы как дескрипторы на уровне класса. **Почему экономится память:** у обычного экземпляра \***\*dict\*\*** — это словарь (десятки байт на объект); при миллионе мелких объектов (например, точек **Point(x, y)**) слоты под два поля дают существенную экономию по сравнению с миллионом словарей.

#### Как это работает (синтаксис и наследование)

\***\*slots\*\*** задаётся как строка с одним именем или как итерабельный объект строк: \***\*slots** = ('x', 'y')** или \*\***slots** = ['x', 'y']**. Для каждого имени интерпретатор создаёт **дескриптор** (по сути, слот в структуре экземпляра). При обращении **obj.x** значение берётся из этого слота; \***\*dict\*\*** у экземпляра не создаётся, поэтому **obj.**dict\***\* вызовет **AttributeError**, если только вы явно не добавили **'**dict**'** в \*\***slots\***\* (тогда атрибуты из слотов и \*\***dict\*\*\*\* сосуществуют).

**Пример:**

```python
class Point:
    __slots__ = ('x', 'y')
    def __init__(self, x, y):
        self.x = x
        self.y = y

p = Point(1, 2)
print(p.x, p.y)   # 1 2
# p.z = 3   # AttributeError: 'Point' has no attribute 'z'
# print(p.__dict__)   # AttributeError (если __dict__ не в __slots__)
```

#### Наследование и **slots**

Если **базовый класс** имеет \***\*slots\*\***, подкласс **наследует** это ограничение: у экземпляров подкласса тоже не будет \***\*dict\*\*** (если только подкласс не определит свой \***\*slots\*\*** и не включит в него **'**dict**'**). Если подкласс задаёт **свой** \***\*slots\*\***, в нём обычно перечисляют **только дополнительные** имена (свои поля); интерпретатор не объединяет автоматически \***\*slots\*\*** родителя и подкласса в один список — наследуется механизм (слоты родителя плюс слоты подкласса). Если у одного из родителей есть \***\*dict\*\*** (нет \***\*slots\*\*** или **'**dict**'** в \***\*slots\*\***), то у экземпляра подкласса будет \***\*dict\*\***, и можно добавлять произвольные атрибуты.

**Пример наследования:**

```python
class A:
    __slots__ = ('a',)
class B(A):
    __slots__ = ('b',)   # только своё поле; 'a' наследуется от A

o = B()
o.a = 1
o.b = 2
# o.c = 3   # AttributeError
```

#### Когда использовать **slots**

- **Много мелких объектов** с фиксированным набором полей (точки, события, записи) — экономия памяти за счёт отсутствия \***\*dict\*\***.
- **Жёсткий контракт:** запрет на добавление «случайных» атрибутов уменьшает опечатки и неявные зависимости.

**Плохо vs лучше:** не использовать \***\*slots\*\*** «на всякий случай» — это усложняет наследование (например, миксины с \***\*dict\*\***) и мешает динамическому добавлению атрибутов. Использовать только когда есть измеримая польза (память, контракт).

#### Типичные ошибки (**slots**)

- **Добавить в **slots** имя, которое уже есть в классе как метод или свойство:** конфликт имён; слот «затенит» метод. Не включайте в \***\*slots\*\*** имена методов.
- **Ожидать **dict** у экземпляра:** код, который полагается на **getattr(obj, name)** или **obj.**dict\***\*, сломается для классов со \*\***slots\***\* без **'**dict**'\*\*.

### 5.6 Метаклассы

**Метакласс** — это класс, экземплярами которого являются **классы**. То есть класс в Python — это объект, и у него есть тип; этот тип и есть метакласс. По умолчанию тип любого класса — **type**: **type(C)** возвращает **type** (или подкласс **type**, если класс создан через свой метакласс).

#### type как фабрика классов

**type(name, bases, dict)** — встроенная функция, создающая новый класс. **name** — строка (имя класса), **bases** — кортеж базовых классов, **dict** — словарь атрибутов класса (методы, поля класса). Вызов **type('MyClass', (Base,), {'x': 1, 'f': lambda self: self.x})** эквивалентен определению **class MyClass(Base): x = 1; def f(self): return self.x**. Классы, объявленные через **class**, в итоге создаются через **type** (или через указанный метакласс).

#### **metaclass** в Py2 vs class C(metaclass=Meta) в Py3

В **Python 2** метакласс задавали атрибутом \***\*metaclass\*\*** в теле класса или в модуле. В **Python 3** метакласс указывается в списке наследования: **class C(metaclass=Meta):** — ключевое слово **metaclass=Meta** передаётся интерпретатору при создании класса, и класс **C** создаётся вызовом **Meta('C', bases, dict)** вместо **type(...)**. Если метакласс не указан, используется метакласс базового класса или **type**.

#### Как это работает внутри: что происходит при выполнении class C(metaclass=Meta):

**Пошагово при встрече определения класса:** (1) Интерпретатор собирает имя класса (**C**), кортеж базовых классов (**bases**) и словарь атрибутов тела класса (**dict** — методы, поля, вложенные классы). (2) Определяется метакласс: если указан **metaclass=Meta**, используется **Meta**; иначе берётся метакласс первой базы или **type**. (3) Создание класса выполняется вызовом **metaclass(name, bases, dict)** — то есть **Meta('C', bases, dict)**. Обычно вызывается **Meta.**new**(Meta, 'C', bases, dict)** (и при необходимости **Meta.**init\***\*), результат — объект класса **C**. (4) Имя **C** в текущем пространстве имён связывается с этим объектом. Таким образом, **class C(metaclass=Meta): ...** не создаёт класс «напрямую», а передаёт управление метаклассу; **type** — это метакласс по умолчанию, делающий то же самое через свой \*\***new\*\*\*\*.

**Пример:**

```python
class Meta(type):
    def __new__(mcs, name, bases, dct):
        print("Creating class", name)
        return super().__new__(mcs, name, bases, dct)

class C(metaclass=Meta):
    pass
# Выведет: Creating class C
```

#### Когда метакласс реально нужен

- **Изменение создания класса:** проверка или изменение атрибутов, баз, имени перед созданием класса (валидация, регистрация, автогенерация методов).
- **Регистрация классов:** метакласс может при создании класса добавлять его в реестр (плагины, сериализаторы).
- **API фреймворков:** ORM, валидаторы, декларативные конфиги часто используют метаклассы для преобразования объявления класса в поведение.

В **большинстве прикладного кода** метакласс не нужен: достаточно обычных классов, **@property**, дескрипторов и наследования. Метаклассы усложняют отладку и понимание кода; используйте их только когда другой способ (декораторы классов, \***\*init_subclass\*\***) не подходит.

#### **init_subclass** как альтернатива метаклассу

\***\*init_subclass**(cls, **kwargs)** (Python 3.6+) — метод **базового** класса, вызываемый **при создании подкласса** этого класса. Сигнатура: **cls** — только что созданный подкласс; **kwargs** — любые именованные аргументы, переданные в объявлении **class Sub(Base, x=1)** (Python 3.6+). Позволяет реагировать на появление подкласса (регистрация, проверка атрибутов, подстановка значений) **без метакласса**. Вызывается после того, как класс уже создан (метакласс уже отработал); внутри можно проверять **cls.**dict\***\*, **cls.**bases**\*\*, добавлять класс в реестр и т.д.

**Пошагово: когда вызывается **init_subclass**.** (1) Интерпретатор встречает **class A(Base): ...**. (2) Создаётся класс **A** (через **type** или метакласс). (3) После создания класса вызывается **Base.**init_subclass**(A)** — в **cls** передаётся класс **A**. (4) Если у **Base** есть родитель **object**, **object.**init_subclass\***\* по умолчанию ничего не делает. Итог: \*\***init_subclass\***\* вызывается **один раз** при определении каждого подкласса; удобно для регистрации плагинов, проверки обязательных атрибутов подкласса, передачи параметров от базового класса к подклассу через **kwargs\*\*.

**Пример с проверкой атрибута подкласса:**

```python
class Base:
    _registry = []
    def __init_subclass__(cls, **kwargs):
        super().__init_subclass__(**kwargs)
        if not hasattr(cls, "name"):
            raise TypeError(f"{cls.__name__} must define class attribute 'name'")
        Base._registry.append(cls)

class A(Base):
    name = "A"
class B(Base):
    name = "B"
# class C(Base): pass   # TypeError: C must define class attribute 'name'
print(Base._registry)   # [A, B]
```

#### dataclasses (Python 3.7+) — классы-данные без boilerplate

Модуль **dataclasses** и декоратор **@dataclass** упрощают объявление классов, которые в основном хранят данные (поля с аннотациями типов). Автоматически генерируются \***\*init\*\***, \***\*repr\*\***, при необходимости \***\*eq\*\*** и другие методы; можно добавить **frozen=True** для неизменяемых экземпляров. Подходит для DTO, конфигов, записей — когда не нужна сложная логика создания или дескрипторы.

```python
from dataclasses import dataclass

@dataclass
class Point:
    x: float
    y: float

p = Point(1.0, 2.0)   # __init__ сгенерирован
print(p)              # Point(x=1.0, y=2.0) — __repr__ сгенерирован
```

**Когда использовать:** классы с несколькими полями и минимумом логики; замена **namedtuple** с мутабельностью и наследованием. Для сложной инициализации, дескрипторов или метаклассов по-прежнему пишут обычный класс.

#### Типичные ошибки (метаклассы)

- **Использовать метакласс там, где достаточно **init_subclass** или декоратора класса:** метакласс сложнее и менее очевиден для читателя.
- **Путать **new** и **init** метакласса:** \***\*new**(mcs, name, bases, dct)** создаёт класс (объект типа **mcs**); \*\***init**(cls, name, bases, dct)** инициализирует уже созданный класс. Часто достаточно \***\*new\*\*** или \***\*init\*\*** одного из них.

#### Практический вывод по §5 (Классы и ООП)

- В Py3 всегда используйте **super()** без аргументов при вызове методов родителя; проверяйте MRO при множественном наследовании (**C.mro()**).
- **Наследование** — когда отношение «is-a» (подкласс — разновидность базового); **композиция** — когда «has-a» (объект содержит другой); при сомнении предпочитайте композицию.
- **Инкапсуляция:** **\_name** — соглашение «внутренний»; **\_\_name** — name mangling для защиты от конфликтов в подклассах; публичный интерфейс — без подчёркивания; контроль доступа — **@property**.
- **Полиморфизм** — переопределение методов в подклассах и duck typing; один вызов **obj.method()** даёт разное поведение в зависимости от типа **obj**.
- **@classmethod** для фабрик и альтернативных конструкторов (**cls(...)**); **@staticmethod** для хелперов без доступа к классу/экземпляру; **@property** для вычисляемых и проверяемых атрибутов.
- Переопределяя \***\*eq\*\***, учитывайте \***\*hash\*\*** (или явно \***\*hash** = None**); задавайте \*\***repr\*\*\*\* для отладки.
- Дескрипторы позволяют переиспользовать логику доступа к атрибутам (property, classmethod, staticmethod — частные случаи).
- **Атрибуты класса** хранятся в **C.**dict\***\*; **атрибуты экземпляра** — в **obj.**dict\*\***; присваивание **obj.x = v** создаёт атрибут экземпляра и не меняет класс; изменяемый атрибут класса (список и т.д.) общий у всех экземпляров — создавайте атрибуты экземпляра в \***\*init\*\***.
- **ABC** — контракт без реализации (**@abstractmethod**); **миксины** — классы для множественного наследования, добавляющие поведение; \***\*bases\*\***, \***\*mro\*\***, \***\*subclasses**()\*\* — интроспекция иерархии.
- \***\*getattr\*\*** — fallback при отсутствии атрибута; \***\*getattribute\*\*** — перехват каждого чтения; внутри \***\*getattribute\*\*** и \***\*setattr\*\*** используйте **object.**getattribute\***\* / **object.**setattr\*\***, чтобы избежать рекурсии.
- \***\*init_subclass\*\*** — реакция на создание подкласса без метакласса; **@dataclass** — классы-данные с автогенерацией \***\*init\*\***, \***\*repr\*\*** и т.д.
- \***\*slots\*\*** — только при обоснованной экономии памяти или жёстком контракте; метаклассы — только когда \***\*init_subclass\*\*** или декораторы не подходят.
