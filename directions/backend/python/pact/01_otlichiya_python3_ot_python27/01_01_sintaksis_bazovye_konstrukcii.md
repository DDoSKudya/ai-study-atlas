## 1. Синтаксис и базовые конструкции

В Python 3 изменились не только отдельные конструкции, но и сама философия: **явное лучше неявного**, **текст по умолчанию — Unicode**, **целочисленное и вещественное деление разведены**. Ниже разобраны главные синтаксические отличия.

---

### 1.1 print

#### В чём отличие

> Этот подпункт специально расписан очень подробно и «по шагам». Если сейчас вы чувствуете себя «совсем тупым» — это нормально: просто читайте медленно и возвращайтесь к примерам столько раз, сколько нужно. Важно не скорость, а то, чтобы мозг действительно **понял** каждое маленькое действие.

В **Python 2.7** `print` — это **инструкция** (statement). Её нельзя передать как аргумент функции, переопределить в области видимости или вызвать с именованными параметрами.

Чтобы вообще почувствовать разницу, удобно держать в голове очень простую картинку:

- В Py2 `print` — это **кусок синтаксиса языка**, как `if`, `for`, `while`. У него свои правила записи, он живёт «особняком».
- В Py3 `print` — это **обычная функция**, такая же, как `len` или `sum`: у неё есть круглые скобки, параметры по имени, её можно передавать в другие функции, сохранять в переменные и подменять.

Дальше весь раздел по сути отвечает на один вопрос:  
**«Что нам даёт то, что `print` теперь функция, а не синтаксис?»** — мы разберём это на десятках маленьких шагов.

```python
# Python 2.7
print "Hello"
print "A", "B"           # пробел между аргументами по умолчанию
print "No newline",      # запятая в конце подавляет перевод строки
print >> sys.stderr, "err"
```

В **Python 3** `print` — **встроенная функция** (built-in function). У неё есть сигнатура, именованные аргументы и единообразное поведение.

#### Сигнатура

```python
print(*objects, sep=' ', end='\n', file=sys.stdout, flush=False)
```

- **objects** — произвольное число позиционных аргументов; выводятся через `str()`.
- **sep** — строка между объектами (по умолчанию пробел `' '`).
- **end** — строка в конце вывода (по умолчанию `'\n'`).
- **file** — объект с методом `write()` (по умолчанию `sys.stdout`).
- **flush** — сбрасывать ли буфер после вывода (по умолчанию `False`).

#### Примеры

```python
# Простой вывод
print("Hello")                    # Hello\n

# Несколько аргументов
print("A", "B", "C")              # A B C\n

# Без перевода строки в конце
print("Hello", end='')            # Hello
print("World", end='')            # World

# Свой разделитель
print(1, 2, 3, sep='; ')          # 1; 2; 3\n

# Вывод в stderr
import sys
print("Error message", file=sys.stderr)

# Принудительный сброс буфера (логи, реальном времени)
print("Progress...", flush=True)
```

**Пошагово: что происходит при вызове print(1, 2, 3, sep='; ', end='' ).** (1) Интерпретатор получает позиционные аргументы **1**, **2**, **3** и именованные **sep='; '**, **end=''**. (2) Каждый аргумент приводится к строке через **str()**: **str(1)** → **'1'**, **str(2)** → **'2'**, **str(3)** → **'3'**. (3) Строки объединяются в одну: между элементами вставляется **sep** (по умолчанию пробел, здесь — **'; '**): **'1' + '; ' + '2' + '; ' + '3'** → **'1; 2; 3'**. (4) В конец добавляется **end** (по умолчанию **'\\n'**, здесь — пустая строка **''**): итоговая строка **'1; 2; 3'**. (5) Строка записывается в объект **file** (по умолчанию **sys.stdout**) методом **file.write(result)**. (6) Если **flush=True**, вызывается **file.flush()** — буфер сбрасывается в устройство вывода. Функция **print** возвращает **None**. Итог: **print** — это последовательность **str()** → объединение через **sep** → добавление **end** → запись в **file** → опционально **flush**.

#### Теоретическое обоснование

- **Единообразие:** в Python всё, что «делает действие», вызывается как функция; `print` перестал быть исключением. Инструкции (statement) в языке — это только те конструкции, которые нельзя выразить вызовом функции (присваивание, циклы, условия, объявление классов и функций). Сделав `print` функцией, язык устранил лишнее исключение из этого правила.
- **Гибкость:** можно подставить другой `file`, изменить `sep`/`end`, передать `print` как аргумент (например, в `map` или тестах). В Python 2 для перенаправления в файл использовался специальный синтаксис **`print >> sys.stderr, "msg"`**; в Python 3 это выражается единообразно: **`print("msg", file=sys.stderr)`**.
- **PEP 3105** — «Make print a function»; мотивация: упрощение языка, единообразие с другими встроенными функциями, возможность переопределения в модуле (например, для тестов или логирования) без манипуляций с `sys.stdout`.

#### Почему это важно на практике

1. **Переопределение в тестах и подмена вывода.** В Python 2 нельзя было «подменить» `print` в области видимости модуля, не трогая `sys.stdout`. В Python 3 можно передать в тестируемый модуль свою функцию вместо `print` (через патчи или инъекцию) или временно подменить **`builtins.print`**, чтобы перехватывать вывод без перенаправления всего stdout.
2. **Передача как аргумент.** В Python 3 допустимо **`map(print, sequence)`** или **`[print(x) for x in items]`** (с осторожностью: list comprehension создаёт список из `None`). В Python 2 `print` нельзя было передать как объект функции.
3. **Буферизация и flush.** Параметр **`flush=True`** даёт явный контроль над сбросом буфера — важно для логов в реальном времени и для отладочного вывода, который должен сразу появляться в консоли или в файле.

#### Миграция с Python 2 на Python 3

- Заменить **`print "x"`** на **`print("x")`**.
- Заменить **`print "a", "b"`** на **`print("a", "b")`** (разделитель по умолчанию — пробел, как и раньше).
- Заменить **`print "x",`** (без перевода строки) на **`print("x", end='')`**.
- Заменить **`print >> sys.stderr, "err"`** на **`print("err", file=sys.stderr)`**.
- Инструмент **2to3** может выполнять часть этих замен автоматически; для сложных случаев нужен ручной ревью.

#### Типичные ошибки при переходе

- **Забыть скобки:** **`print "Hello"`** в Python 3 даёт **SyntaxError**. Скобки обязательны.
- **Путать `end` и перевод строки:** в Python 2 запятая в конце подавляла перевод строки; в Python 3 для этого явно указывают **`end=''`** или другую строку. Если не указать **end**, после каждого вывода будет перевод строки — это может изменить поведение скриптов, которые собирали вывод в одну строку.

#### Как это работает внутри (упрощённо)

При вызове **`print(a, b, sep=';', end='!')`** интерпретатор:

1. Преобразует каждый позиционный аргумент в строку через **`str(obj)`** (если объект не строка).
2. Собирает результат: между объектами вставляет **sep**, в конец — **end**.
3. Вызывает **`file.write(result)`** у объекта **file** (по умолчанию **sys.stdout**).
4. Если **flush=True**, вызывает **`file.flush()`**.

Поэтому **file** может быть любым объектом с методом **write** (принимающим одну строку): файл, **io.StringIO** (буфер в памяти), кастомный класс для логирования. Это и даёт гибкость: подмена **file** позволяет перехватывать весь вывод без замены **sys.stdout**.

**Пошагово: что именно передаётся в file.write.** Метод **file.write** принимает **один аргумент — одну строку**. Интерпретатор **не** вызывает **write** по одному разу на каждый объект: сначала строится **единая** строка **result**. Алгоритм: (1) **parts = [str(obj) for obj in objects]** — все позиционные аргументы в строки. (2) **result = sep.join(parts) + end** — между строками вставляется **sep**, в конец дописывается **end**. Например, **print(1, 2, 3, sep='; ', end='!')** даёт **result = "1; 2; 3!"**; вызывается **file.write("1; 2; 3!")** — один вызов с одной строкой. (3) Если позиционных аргументов нет (**print()**), то **result = end** (по умолчанию **"\n"**). Итог: **write** всегда вызывается ровно один раз за один вызов **print**; в **write** попадает уже собранная строка с учётом **sep** и **end**.

#### Пример: перехват print в тестах (полный код)

В Python 3 можно перехватить вывод модуля, не трогая **sys.stdout**, — передав в код объект с методом **write** или временно подменив **print**:

```python
# Вариант 1: подмена sys.stdout на буфер в памяти
import sys
from io import StringIO

def test_print_capture():
    old_stdout = sys.stdout
    sys.stdout = buf = StringIO()
    try:
        print("Hello", "World", sep=", ")
        output = buf.getvalue()
        assert output == "Hello, World\n"
    finally:
        sys.stdout = old_stdout

# Вариант 2: в тестируемом модуле вызывается print(...) — мы подменяем builtins.print
import builtins
original_print = builtins.print
captured = []
def mock_print(*args, **kwargs):
    captured.append((args, kwargs))
builtins.print = mock_print
try:
    my_module.do_something()   # внутри вызывается print(...)
    assert len(captured) == 1
    assert captured[0][0] == ("expected",)
finally:
    builtins.print = original_print
```

В Python 2 **print** — инструкция, её нельзя подменить через **builtins**: синтаксис **print x** не вызывает функцию. Поэтому в Py2 для перехвата вывода обычно подменяли **sys.stdout**. В Py3 оба варианта (подмена **sys.stdout** или **print**) возможны; подмена **print** даёт точечный контроль над вызовом.

#### Граничные случаи и тонкости print

- **Вызов без аргументов:** **`print()`** выводит только **end** (по умолчанию перевод строки **`\n`**). То есть пустая строка. Удобно для вставки пустой строки между блоками вывода.
- **Аргументы-нестроки:** **`print(1, 2, 3)`** — каждый аргумент приводится к строке через **str()**; **str(1)** → **'1'**. Объекты с переопределённым \***\*str\*\*** выводятся через него; иначе используется \***\*repr\*\***. Для отладки **print(repr(x))** или **print(f"{x=}")** (Py3.8+) даёт более информативный вывод.
- **Один аргумент:** **`print(x)`** выводит **str(x)** и затем **end**. Разделитель **sep** не используется (между объектами нет ничего).
- **Пустой разделитель:** **`print("a", "b", "c", sep='')`** выводит **abc** без пробелов между элементами. Удобно для склейки фрагментов в одну строку без **''.join()**.
- **end и sep — любые строки:** **end** может быть **''**, **' '**, **'\\n---\\n'** и т.д.; **sep** — тоже. Например, **print(1, 2, 3, sep=' -> ', end='.')** даёт **1 -> 2 -> 3.**.
- **Тип аргументов:** каждый позиционный аргумент приводится к строке через **str(obj)**. Если у объекта определён \***\*str\*\***, будет вызван он; иначе используется \***\*repr\*\***. Для отладочного вывода иногда удобнее **repr(obj)** вручную: **print("x =", repr(x))**.
- **Порядок аргументов:** сначала все позиционные (objects), затем именованные. Именованные можно задавать в любом порядке: **print("a", "b", end='!', sep=';')** — допустимо.

#### Практический вывод по print

- Всегда используйте **print** как функцию с скобками: **`print(...)`**.
- Для вывода в stderr: **`print(..., file=sys.stderr)`**.
- Для отладки «без буферизации»: **`print(..., flush=True)`**.
- При миграции с Py2 проверьте все места, где вывод собирался в одну строку (запятая в конце **print**): замените на **`end=''`**.
- Для пустой строки между блоками используйте **print()**; для склейки нескольких значений в одну строку без пробелов — **print(a, b, c, sep='', end='')**.

### 1.2 Строки (str / bytes / unicode)

Это одно из самых важных отличий Python 3 от 2.7: **модель текста и байт разведена явно**. В Python 2 строка **`str`** могла быть и «текстом», и «байтами»; смешение приводило к ошибкам кодировок и путанице. В Python 3 **текст** (Unicode) и **байты** (сырые байты) — разные типы; это устраняет целый класс багов и соответствует принципу «явное лучше неявного» (PEP 20). Ключевые PEP: **PEP 393** (flexible string representation), **PEP 3100** (Unicode literals), **PEP 358** (bytes type).

#### Краткая таблица

| Концепция           | Python 2.7  | Python 3    |
| ------------------- | ----------- | ----------- |
| Текст (Unicode)     | `unicode`   | `str`       |
| Байты (сырые байты) | `str`       | `bytes`     |
| Изменяемые байты    | `bytearray` | `bytearray` |

В Python 3 **`str`** — это последовательность **Unicode-символов** (code points). Тип **`unicode`** удалён: «строка по умолчанию» теперь всегда текст в Unicode.

**`bytes`** — неизменяемая последовательность байт (целые числа 0–255). **`bytearray`** — изменяемая последовательность байт (те же значения 0–255, но элементы можно менять: **`ba[0] = 65`**). Оба типа поддерживают большинство методов строк (**decode** у bytes/bytearray, **encode** у str; **split**, **strip**, **find** и т.д. — с учётом того, что аргументы и результаты для bytes/bytearray в байтах).

**Unicode и code points (кратко):** строка **str** в Python 3 — это последовательность **Unicode code points** (целых чисел, идентифицирующих символы). Один «символ» с точки зрения пользователя может занимать один code point (например, латинская **A**) или несколько (например, буква с акцентом может быть одной литерой **é** или комбинацией **e** + combining accent). Функции **ord(c)** и **chr(code)** переводят символ в код и обратно: **ord('A')** → **65**, **chr(65)** → **'A'**. Кодировка (UTF-8, CP1251 и т.д.) определяет, как набор code points представляется в виде байт при записи в файл или передаче по сети; **encode**/ **decode** выполняют этот переход.

#### Строковые литералы

```python
# Обычные строки — тип str (Unicode)
s1 = 'одинарные кавычки'
s2 = "двойные кавычки"
s3 = '''многострочные
три пары кавычек'''

# Сырые строки (без экранирования \)
path = r'C:\Users\name\file.txt'

# Байтовые литералы — тип bytes
b = b'hello'
b = b"\x00\x01\x02"

# Форматированные строки (f-strings, Python 3.6+)
name = "World"
f = f"Hello, {name}!"
# Формат-спецификатор: выравнивание, ширина, точность
f"{3.14159:.2f}"      # '3.14'
f"{42:>5}"             # '   42' (ширина 5, выравнивание вправо)
f"{'hi':<5}"           # 'hi   ' (выравнивание влево)
f"{1000:,}"            # '1,000' (разделитель тысяч)
# Выражения внутри {}
f"{2 + 3}"             # '5'
f"{name.upper()}"      # 'WORLD'
# Спецификатор {expr=} (Py3.8+) — для отладки, выводит имя и значение
x, y = 10, 20
f"{x=}, {y=}"          # 'x=10, y=20'
```

**Пошагово: что происходит при вычислении f-строки f"{expr} more".** (1) Интерпретатор встречает f-строку; все выражения внутри **{}** вычисляются **в момент выполнения** (при каждом вызове строки). (2) Для каждого **{expr}** вызывается **format(expr)** или **str(expr)** (в зависимости от наличия спецификатора); результат подставляется вместо **{expr}**. (3) Спецификатор формата (**:width.prec**, **:d**, **:f** и т.д.) задаёт, как отображать значение: **f"{x:.2f}"** — число **x** как float с двумя знаками после запятой; **f"{x:>5}"** — выравнивание вправо шириной 5; **f"{x=}"** (Py3.8+) — вывод имени переменной и её значения. (4) Собранная строка возвращается как результат. Важно: выражения внутри **{}** выполняются в **контексте** вызывающего кода — имеют доступ к локальным и глобальным переменным; нельзя использовать обратный слэш внутри **{}** (синтаксическая ошибка).

**Сырые строки (r'...'):** обратный слэш **\\** не экранируется — **r'\\n'** это два символа (обратный слэш и **n**), а не перевод строки. Удобно для путей Windows (**r'C:\\Users\\file.txt'**) и регулярных выражений. В Python 3.12+ допустимо **rf"..."** или **fr"..."** — сырая f-строка; внутри **f"{path}"** переменная **path** подставляется как есть, экранирование в литерале пути задаётся префиксом **r** у строки с путём.

#### Кодировка исходного кода

В **Python 3** по умолчанию исходный код считается в **UTF-8** (PEP 3120). Объявление кодировки в начале файла допустимо, но не обязательно для UTF-8:

```python
# -*- coding: utf-8 -*-
# Теперь можно писать, например:
text = "Привет, мир!"
```

**Допустимые формы объявления кодировки:** **`# -*- coding: utf-8 -*-`**, **`# coding: utf-8`**, **`# coding=utf-8`** — интерпретатор ищет в первых двух строках файла подстроку **coding** и имя кодировки. Без объявления в Py3 используется UTF-8; в Py2 использовалась ASCII — любой не-ASCII символ в исходнике (например, комментарий или строка с кириллицей) без объявления кодировки вызывал **SyntaxError** в Py2.

**Практический вывод:** для переносимого кода с не-ASCII символами в Py2 объявление кодировки в начале файла было обязательно; в Py3 при UTF-8 объявление не обязательно, но допустимо для ясности и совместимости с инструментами (редакторы, линтеры).

#### str и bytes: основные методы и преобразования

- **Текст → байты:** `str.encode(encoding='utf-8', errors='strict')` → `bytes`
- **Байты → текст:** `bytes.decode(encoding='utf-8', errors='strict')` → `str`

**Пошагово: что происходит при encode.** Строка **str** — последовательность Unicode code points (целых чисел). Кодировка (например, UTF-8) задаёт правило: каждому code point соответствует одна или несколько байт. Метод **encode** проходит по символам строки и для каждого получает соответствующие байты по таблице кодировки; результат собирается в объект **bytes**. Если символ нельзя представить в выбранной кодировке (например, кириллица в Latin-1), поведение задаётся параметром **errors** (выбросить исключение, заменить, пропустить и т.д.). **Пример UTF-8:** символ **'П'** (U+041F) в UTF-8 кодируется как **0xD0 0x9F** — два байта; **'A'** (U+0041) — один байт **0x41**. Метод **encode** вызывает внутренний кодер для каждого символа, получает байты и присоединяет их к буферу; в конце возвращается объект **bytes** с собранными байтами.

**Пошагово: что происходит при decode.** Объект **bytes** — последовательность байт (0–255). Декодер читает байты по правилам кодировки (в UTF-8 это может быть 1–4 байта на символ) и собирает соответствующие code points в строку **str**. Если последовательность байт невалидна для данной кодировки (например, обрезанная многобайтовая последовательность), срабатывает **errors**. **Пример UTF-8:** байты **0xD0 0x9F** в UTF-8 образуют двухбайтовую последовательность для символа U+041F (**'П'**); первый байт **0xD0** (110xxxxx) указывает на двухбайтовый символ, второй **0x9F** (10xxxxxx) — continuation byte. Декодер извлекает биты из обоих байт, собирает code point и добавляет символ в результат. При ошибке (например, **0xFF** — невалидный первый байт в UTF-8): **errors='strict'** → **UnicodeDecodeError**; **errors='replace'** → вставляется символ замены **'\ufffd'** (U+FFFD).

**Когда какую кодировку выбирать:** для обмена с внешним миром (файлы, сеть, API) используйте кодировку, которую ожидает другая сторона: **UTF-8** — универсальный выбор для новых систем; **CP1251**, **CP866** — для совместимости с русской Windows/консолью; **Latin-1** — когда нужен round-trip любых байт 0–255 в символы без потерь. Для внутренней обработки в Python 3 удобнее везде держать **str** (Unicode) и кодировать/декодировать только на границах.

```python
# Текст в байты
"Привет".encode('utf-8')   # b'\xd0\x9f\xd1\x80\xd0\xb8\xd0\xb2\xd0\xb5\xd1\x82'

# Байты в текст
b'\xd0\x9f\xd1\x80'.decode('utf-8')   # 'Пр'

# Ошибка декодирования
b'\xff\xfe'.decode('utf-8')            # UnicodeDecodeError (при errors='strict')
b'\xff\xfe'.decode('utf-8', errors='replace')  # '\ufffd\ufffd'
```

#### Важная тонкость: индексация bytes

В Python 3 **`b'text'[0]`** возвращает **целое число** (код байта), а не однобайтовую строку, как в Python 2:

```python
b = b'ABC'
b[0]    # 65 (int)
b[1]    # 66 (int)

# Срез по-прежнему даёт bytes
b[0:1]  # b'A'
```

#### Методы str: removeprefix / removesuffix (Python 3.9+)

Удобно убирать фиксированный префикс или суффикс **без лишних проверок**: если строка начинается (заканчивается) на указанный префикс (суффикс), он удаляется; иначе возвращается копия строки без изменений. Не нужно писать **`s.startswith(prefix) and s[len(prefix):]`** — один вызов заменяет проверку и срез.

```python
"test_file.py".removeprefix("test_")   # 'file.py'
"file.py".removesuffix(".py")          # 'file'
"file.py".removesuffix(".txt")         # 'file.py' — суффикс не совпал, строка без изменений
```

#### Другие важные методы str (напоминание)

- **`str.join(iterable)`** — склейка строк из итерабельного объекта с разделителем: **`';'.join(['a', 'b', 'c'])`** → **`'a;b;c'`**. Эффективнее конкатенации в цикле (**`''.join(parts)`** вместо **`s = ''; for x in parts: s += x`**).
- **`str.strip([chars])`**, **`str.lstrip([chars])`**, **`str.rstrip([chars])`** — удаление символов с краёв (по умолчанию пробельные символы).
- **`str.format(\*args, **kwargs)`** — форматирование строк; в Python 3.6+ часто удобнее **f-strings**: **`f'{name}: {value}'`\*\*.
- **`str.startswith(prefix[, start[, end]])`**, **`str.endswith(suffix[, start[, end]])`** — проверка начала/конца; **prefix** и **suffix** могут быть **tuple** строк (проверка «начинается с одного из»).
- **`str.find(sub[, start[, end]])`** — возвращает индекс первого вхождения **sub** в срезе, или **-1** при отсутствии. **`str.index(sub[, start[, end]])`** — то же, но при отсутствии выбрасывает **ValueError**. Для проверки «есть ли подстрока» достаточно **sub in s**; для индекса предпочтите **find**, если отсутствие допустимо (чтобы не обрабатывать исключение), или **index**, если отсутствие — ошибка.

#### split, partition, rpartition

- **`str.split(sep=None, maxsplit=-1)`** — разбиение по разделителю; **maxsplit** ограничивает число разбиений (удобно для парсинга вида **key=value**). Если **sep** не передан или **None**, строка разбивается по **любому** подряд идущему пробельному символу (пробел, табуляция, перевод строки и т.д.), при этом ведущие и завершающие пробелы игнорируются. Поэтому **" a b c ".split()** даёт **['a', 'b', 'c']** — удобно для разбора строки из слов. При заданном **sep** разбиение идёт строго по этому разделителю; пустые фрагменты сохраняются (например, **"a||b".split("|")** → **['a', '', 'b']**).
- **`str.partition(sep)`** — разбивает на **`(before, sep, after)`**; **`rpartition(sep)`** — то же, но разбиение справа.

```python
"a=b=c".split("=", maxsplit=1)   # ['a', 'b=c']
"key=value".partition("=")       # ('key', '=', 'value')
"path/to/file".rpartition("/")   # ('path/to', '/', 'file')
```

**Пошагово: что происходит при s.partition(sep).** (1) Интерпретатор ищет **первое** вхождение подстроки **sep** в **s** (слева направо). (2) Если **sep** найден на позиции **i**: результат — кортеж **(s[:i], sep, s[i+len(sep):])** — часть до разделителя, сам разделитель, часть после. (3) Если **sep** не найден: результат — **(s, '', '')** — вся строка в первом элементе, два пустых во втором и третьем. **rpartition(sep)** — то же, но ищет **последнее** вхождение (справа); если не найден — **('', '', s)**. Важно: **partition** всегда возвращает **ровно три** элемента — распаковка **a, sep, b = s.partition("=")** всегда безопасна (три имени — три значения).

**Когда использовать partition вместо split(maxsplit=1):** **partition(sep)** всегда возвращает **три** элемента: **(before, sep, after)**. Если **sep** не найден, результатом будет **(s, '', '')** — вся строка в первом элементе, два пустых во втором и третьем. Удобно, когда нужен фиксированный «тройной» формат и проверка **if sep in s** не нужна. **split(sep, maxsplit=1)** возвращает список из одного или двух элементов; если разделитель не найден — список из одного элемента (вся строка). Для парсинга вида **key=value** оба подходят; **partition** даёт явный доступ к разделителю и одинаковую длину результата.

**Почему split() без аргумента удобен для «слов»:** при **sep=None** интерпретатор рассматривает любую непрерывную последовательность пробельных символов (пробел, табуляция, перевод строки, и т.д.) как один разделитель и отбрасывает пустые фрагменты по краям. Поэтому **" one two \n three ".split()** даёт **['one', 'two', 'three']** — ровно список «слов» без ручной обработки пробелов. При **sep=" "** разбиение идёт строго по одному пробелу: **" a b ".split(" ")** даёт **['', '', 'a', '', 'b', '', '']** — много пустых строк. Для разбора строки из слов, разделённых произвольными пробелами, используйте **split()** без аргумента.

**Пошагово: что происходит при s.split(sep, maxsplit).** (1) Если **sep** не передан (None): строка разбивается по **любому** пробельному символу (пробел, табуляция, перевод строки, и т.д.); ведущие и завершающие пробелы **отбрасываются**; между словами может быть любое количество пробелов — все они считаются одним разделителем. Результат — список «слов» без пустых элементов. (2) Если **sep** задан: разбиение идёт **строго** по подстроке **sep**; каждый разделитель даёт границу; пустые фрагменты между разделителями **сохраняются**. (3) **maxsplit** — максимальное число разбиений; при **maxsplit=1** выполняется **одно** разбиение — получается максимум два элемента. Пример: **"a=b=c".split("=", 1)** → **['a', 'b=c']**; **"a=b=c".split("=")** → **['a', 'b', 'c']**. Пустая строка **""**: **"".split()** → **[]**; **"".split("x")** → **['']**.

**Граничные случаи:**

```python
"aaa".split("b")           # ['aaa'] — разделитель не найден
"aaa".partition("b")       # ('aaa', '', '')
"".split("x")              # ['']
"a".split("a")             # ['', ''] — два пустых по краям
"a".partition("a")         # ('', 'a', '')
"path/to/file".rpartition("/")   # ('path/to', '/', 'file') — разбиение справа
```

#### Файлы и кодировка (open)

В **Python 3** при открытии файла в **текстовом** режиме (**`'r'`**, **`'w'`**, **`'a'`** и т.д.) по умолчанию используется **encoding='utf-8'**. Явно указать кодировку можно так: **`open(path, encoding='utf-8')`** или **`open(path, encoding='cp1251')`**. В **Python 2.7** по умолчанию использовались системная кодировка и байтовые строки; для Unicode в Py2 использовали **`codecs.open`** или **`io.open`** с указанием кодировки. В Py3 **`open`** — это **`io.open`**; явная кодировка устраняет путаницу между текстом и байтами при чтении/записи файлов.

**Текстовый режим vs бинарный:** в **текстовом** режиме (**`'r'`**, **`'w'`**, **`'a'`**) **open** возвращает объект, работающий со **строками** (str); при чтении байты из файла **декодируются** в строки по указанной кодировке; при записи строки **кодируются** в байты. В **бинарном** режиме (**`'rb'`**, **`'wb'`**, **`'ab'`**) **open** возвращает объект, работающий с **bytes**; чтение/запись без преобразования кодировки. Параметр **encoding** имеет смысл только в текстовом режиме; в бинарном его указывать нельзя (будет ошибка).

**Пример чтения и записи текста с явной кодировкой:**

```python
# Чтение текстового файла (UTF-8 по умолчанию)
with open("file.txt", encoding="utf-8") as f:
    text = f.read()   # str

# Запись в другой кодировке (например, для совместимости со старыми системами)
with open("out.txt", "w", encoding="cp1251") as f:
    f.write("Привет")

# Бинарный режим — без encoding, работа с bytes
with open("file.bin", "rb") as f:
    data = f.read()   # bytes
```

**Ошибки при чтении/записи:** при неверной кодировке или «битых» байтах в текстовом режиме может возникнуть **UnicodeDecodeError** (при чтении) или **UnicodeEncodeError** (при записи). Можно передать **errors='replace'** или **errors='ignore'** в **open**: **open(path, encoding='utf-8', errors='replace')** — некорректные байты при чтении заменяются на символ замены.

**Плохо vs лучше: кодировка на границах.**

```python
# Плохо: не указывать кодировку при открытии файла — в разных ОС разное умолчание
with open("data.txt") as f:
    text = f.read()   # на одной машине UTF-8, на другой CP1251 — нестабильно

# Лучше: явно задать кодировку (и при чтении, и при записи)
with open("data.txt", encoding="utf-8") as f:
    text = f.read()
with open("out.txt", "w", encoding="utf-8") as f:
    f.write(text)

# Плохо: склеивать str и bytes без явного преобразования
msg = "Hello " + response.raw_bytes   # TypeError в Py3

# Лучше: явно decode на границе (сеть/файл → приложение)
msg = "Hello " + response.raw_bytes.decode("utf-8")
```

#### Параметр errors при encode/decode

При преобразовании текста в байты (**encode**) или байтов в текст (**decode**) можно указать, как обрабатывать символы, которые нельзя представить в целевой кодировке:

| Значение **errors**           | Поведение при ошибке                                                                                  |
| ----------------------------- | ----------------------------------------------------------------------------------------------------- |
| **`'strict'`** (по умолчанию) | Выбросить **UnicodeDecodeError** / **UnicodeEncodeError**                                             |
| **`'ignore'`**                | Пропустить некодируемый символ (тихо)                                                                 |
| **`'replace'`**               | Заменить на заменяющий символ (например, `'\ufffd'` при decode, `'?'` при encode в ряд кодировок)     |
| **`'backslashreplace'`**      | Заменить на escape-последовательность (например, `\xNN`)                                              |
| **`'surrogateescape'`**       | При decode хранить «битые» байты в виде суррогатных кодов (для последующего round-trip в те же байты) |
| **`'xmlcharrefreplace'`**     | При encode заменить на XML-сущность (например, `&#1234;`)                                             |
| **`'namereplace'`**           | При encode заменить на `\N{name}` (если есть имя у символа)                                           |

Примеры:

```python
# Строка с символом, которого нет в Latin-1
"Привет".encode("latin-1")   # UnicodeEncodeError
"Привет".encode("latin-1", errors="replace")   # b'??????'
"Привет".encode("latin-1", errors="ignore")    # b''

# Байты, не являющиеся валидным UTF-8
b"\xff\xfe".decode("utf-8")   # UnicodeDecodeError
b"\xff\xfe".decode("utf-8", errors="replace")  # '\ufffd\ufffd'
b"\xff\xfe".decode("utf-8", errors="surrogateescape")  # '\udcff\udcfe' (суррогаты)
```

#### Почему разведение str и bytes критично

В Python 2 одна и та же конструкция **`str`** использовалась и для «текста», и для «байт». Это приводило к типичным багам:

1. **Смешение кодировок:** склеивали строки из разных источников (файл в UTF-8, пользовательский ввод в CP1251) без явного decode/encode — получали «кракозябры» или падения в неожиданных местах.
2. **Неявные предположения:** код предполагал, что «строка» — это байты, или наоборот; при смене окружения или данных предположение ломалось.
3. **API файлов и сети:** в Py2 файлы по умолчанию возвращали байты; в одном месте код обрабатывал байты, в другом — Unicode; при передаче между слоями возникали ошибки типов или кодировок.

В Python 3 **тип явно отражает природу данных:** **str** — всегда Unicode (текст), **bytes** — всегда последовательность байт. Нельзя «случайно» передать байты туда, где ожидается текст, и наоборот: операции смешивания (например, **`str + bytes`**) вызывают **TypeError**, что заставляет программиста явно выбрать кодировку и вызвать **encode** или **decode**. Это увеличивает объём кода в точках границы (файл, сеть, внешние API), но устраняет целый класс трудноуловимых багов.

#### Миграция строк с Python 2 на Python 3

- **Байтовые литералы:** в Py2 **`'hello'`** — байты (или «строка» в системной кодировке). В Py3 для байт используйте **`b'hello'`**; **`'hello'`** — это **str** (Unicode).
- **Unicode-литералы:** в Py2 писали **`u'Привет'`** для Unicode. В Py3 все обычные строковые литералы **уже** Unicode; префикс **`u`** допустим, но не обязателен. Тип **unicode** в Py3 удалён — везде используется **str**.
- **Проверки типов:** заменить **`isinstance(s, unicode)`** на **`isinstance(s, str)`**; **`isinstance(s, str)`** в Py2 означало «байты», в Py3 — «Unicode». Для байт в Py3: **`isinstance(s, bytes)`**.
- **Файлы:** везде, где читаете/пишете текст, открывайте файл с **`encoding='...'`** и работайте с **str**; для бинарных данных используйте режим **`'rb'`** / **`'wb'`** и тип **bytes**.

#### Типичные ошибки при работе со строками в Python 3

- **Сложение str и bytes:** **`"x" + b"y"`** даёт **TypeError**. Нужно либо оба типа **str**, либо явно **`"x".encode() + b"y"`** или **`"x" + b"y".decode()`** в зависимости от желаемого результата.
- **Индексация bytes:** **`b'AB'[0]`** в Py3 возвращает **65** (int), а не **b'A'**. Чтобы получить однобайтовый объект, используйте срез: **`b'AB'[0:1]`** → **b'A'**.
- **Полагаться на кодировку по умолчанию:** в разных ОС и локалях «умолчание» может отличаться. Для переносимого кода явно указывайте **encoding** при **open** и при **encode**/ **decode**.

#### bytearray — изменяемые байты (подробнее)

**bytearray** — изменяемая последовательность байт (целые 0–255). Создаётся конструктором **bytearray()**, **bytearray(iterable_of_ints)**, **bytearray(bytes_or_buffer)** или литералом в Py3 нет (литералы только **b'...'** → **bytes**). Методы изменения: **append(x)**, **extend(iterable)**, **insert(i, x)**, **pop([i])**, **remove(x)**, **clear()**, **reverse()** (in-place), а также присваивание по индексу: **ba[i] = n** (n — целое 0–255). Индексация **ba[i]** возвращает **int** (как у **bytes**); срез **ba[i:j]** возвращает **bytearray**. Тип **bytearray** удобен, когда нужно пошагово собирать или менять бинарные данные без создания нового объекта при каждой операции.

**Пошагово: что происходит при ba.append(x) и ba[i] = n.** (1) **ba.append(x)**: интерпретатор проверяет, что **x** — целое в диапазоне 0–255 (иначе **ValueError**); затем добавляет байт **x** в конец **bytearray**; длина **len(ba)** увеличивается на 1. Объект **ba** тот же (мутация на месте); **append** возвращает **None**. (2) **ba[i] = n**: если индекс **i** в допустимом диапазоне (**-len(ba) <= i < len(ba)**), байт в позиции **i** заменяется на **n** (0–255); иначе **IndexError**. При присваивании срезу **ba[i:j] = iterable** длина может измениться: элементы итерабельного (целые 0–255) подставляются вместо среза.

**Когда использовать bytearray vs bytes.** **bytes** — неизменяемый; удобен для константных данных, ключей, аргументов функций, когда изменение не нужно. **bytearray** — изменяемый; удобен, когда нужно собирать бинарные данные по частям (**append**, **extend**), менять отдельные байты (**ba[i] = n**), буферизовать ввод-вывод, протоколы с изменяемыми буферами. Если в итоге нужны неизменяемые **bytes**, вызовите **bytes(ba)** — создастся копия.

Пример:

```python
ba = bytearray(b"hello")
ba[0] = 72    # b"Hello"
ba.append(33) # b"Hello!"
ba.extend(b"??")  # b"Hello!??"
```

#### Сравнение Py2 и Py3: строки (сводная таблица)

| Аспект                           | Python 2.7                                     | Python 3                                   |
| -------------------------------- | ---------------------------------------------- | ------------------------------------------ |
| Литерал `"x"`                    | **str** (байты или «строка» в кодировке файла) | **str** (Unicode)                          |
| Литерал `b"x"`                   | **str** (то же, что `"x"`)                     | **bytes**                                  |
| Литерал `u"x"`                   | **unicode**                                    | **str** (синоним)                          |
| Тип для текста                   | **unicode**                                    | **str**                                    |
| Тип для байт                     | **str**                                        | **bytes**                                  |
| `s[0]` для bytes                 | однобайтовая строка (тип str)                  | **int** (код байта)                        |
| Кодировка исходника по умолчанию | ASCII                                          | UTF-8 (PEP 3120)                           |
| `open()` для текста              | байты / системная кодировка                    | **str**, по умолчанию **encoding='utf-8'** |

#### Практический вывод по str/bytes

- В Py3 по умолчанию работайте с **str** (Unicode); с «сырыми» данными (файл в бинарном режиме, сеть) — с **bytes** или **bytearray**.
- На границах (файл, сеть, внешний API) явно вызывайте **encode** или **decode** и указывайте **encoding**; не полагайтесь на умолчания системы.
- Не смешивайте **str** и **bytes** в одной операции (сложение, сравнение) без явного преобразования — Py3 выдаст **TypeError**, что помогает находить ошибки на этапе разработки.

### 1.3 Деление и числовые операции

#### Деление

В **Python 2.7** поведение оператора **/** зависело от типов операндов:

- `int / int` → целочисленное деление (округление вниз);
- если хотя бы один операнд `float` → вещественное деление.

В **Python 3**:

- **`/`** — всегда **вещественное деление** (даже для двух `int`);
- **`//`** — **целочисленное (floor) деление**.

```python
# Python 3
7 / 2      # 3.5
7 // 2     # 3
-7 // 2    # -4 (округление вниз, не к нулю)
```

**Пошагово: что происходит при вычислении `/` и `//` в Python 3.** (1) Оператор **/** всегда выполняет **вещественное деление**: оба операнда приводятся к **float** (если были int), затем выполняется деление; результат — **float** (даже если математически целое: **4 / 2** → **2.0**). (2) Оператор **//** выполняет **floor division**: сначала вычисляется вещественное частное, затем к нему применяется **math.floor** (округление вниз к ближайшему целому в сторону минус бесконечности); тип результата — **int**, если оба операнда int, иначе **float**. Поэтому **7 / 2** → **3.5**, **7 // 2** → **3**; для отрицательных **-7 / 2** → **-3.5**, **-7 // 2** → **-4** (floor от -3.5 в сторону минус бесконечности). Инвариант: **a == (a // b) \* b + (a % b)** при **b != 0**.

Импорт `from __future__ import division` в Python 2 даёт такое же поведение `/`, как в Python 3.

#### Тип int

В Python 2 был отдельный тип **`long`** для длинных целых (с суффиксом `L`). В Python 3 тип **`long`** удалён: **`int`** имеет неограниченную точность (как прежний `long`).

```python
# Python 3
2**1000    # Очень большое число, тип int
int('1010', 2)   # 10 — второй аргумент: основание (2–36)
```

**int(x, base) — граничные случаи и ошибки:** первый аргумент **x** должен быть строка (str), bytes или bytearray, представляющие целое число в указанной системе счисления (при **base**); либо один аргумент **int(x)** — тогда **x** может быть числом (float округляется в сторону нуля) или строкой в десятичной системе. При **int(string, base)** строка не должна содержать пробелы или знак числа в неподходящем месте; допустимы префиксы **0x**, **0o**, **0b** при **base=0** (автоопределение). **int(" 10 ")** даёт **ValueError** в Py3; **int("10", 2)** даёт **ValueError** (символ '1' допустим в двоичной, но "10" — это 2 в десятичной, в двоичной допустимы только "0" и "1"). **int("ff", 16)** → **255**; **int("0xff", 0)** → **255**. Для разбора пользовательского ввода с возможными пробелами используйте **int(s.strip())** или проверку формата.

#### Подчёркивания в числовых литералах (Python 3.6+)

Для удобства чтения в числах можно ставить подчёркивания (игнорируются интерпретатором). Подчёркивания нельзя ставить в начале и в конце числа, сразу после точки в float, сразу после префикса **0x**, **0o**, **0b** или перед ним.

**Допустимо:**

```python
1_000_000
0x_FF_FF
0o_77
0b_1010
1_234_567.89
1_2_3_4.5_6
0_0_1
```

**Не допускается:** **\_100**, **100\_**, **1\_.0**, **1.\_0**, **0_xFF** — такие варианты вызывают **SyntaxError**. Цель — улучшить читаемость длинных чисел (например, **1_000_000** вместо **1000000**) без изменения значения.

#### int.bit_length(), int.bit_count() (3.10+), int.to_bytes(), int.from_bytes()

**int.bit_length()** — возвращает **число бит**, необходимых для представления числа в двоичном виде **без ведущих нулей** и без знака. Для **n >= 0** это **n.bit_length() == floor(log2(n)) + 1** при **n > 0**, и **0** при **n == 0**. Для отрицательных — число бит для представления числа в дополнительном коде (то есть **(-n).bit_length()** для **n < 0**). Пример: **(0).bit_length()** → **0**; **(1).bit_length()** → **1**; **(255).bit_length()** → **8**; **(256).bit_length()** → **9**. Удобно для выделения буфера под байты: **((n).bit_length() + 7) // 8** — число байт для неотрицательного **n**.

**int.bit_count()** (Python 3.10+) — возвращает **число единичных бит** в двоичном представлении числа (popcount). Для неотрицательных — количество единиц; для отрицательных — количество единиц в представлении числа в дополнительном коде (бесконечное число ведущих единиц). Пример: **(0).bit_count()** → **0**; **(7).bit_count()** → **3** (7 = 0b111); **(255).bit_count()** → **8**. Удобно для подсчёта включённых флагов или хеммингового веса.

**int.to_bytes(length, byteorder='big', \*, signed=False)** — преобразует целое в **bytes** заданной длины. **length** — число байт (должно быть достаточно для представления числа; иначе **OverflowError**). **byteorder** — **'big'** (старший байт первым) или **'little'** (младший первым). **signed=True** — для отрицательных чисел используется дополнительный код. Пример: **(255).to_bytes(1, 'big')** → **b'\\xff'**; **(1024).to_bytes(2, 'big')** → **b'\\x04\\x00'**; **(-1).to_bytes(2, 'big', signed=True)** → **b'\\xff\\xff'**. Длину можно вычислить: **length = (n.bit_length() + 7) // 8** для неотрицательного **n** (для нуля **bit_length()** даёт 0 — нужна проверка **max(1, ...)** или **length=1** для 0).

**int.from_bytes(bytes, byteorder='big', \*, signed=False)** — классовый метод; преобразует **bytes** в целое. **byteorder** и **signed** — как в **to_bytes**. Пример: **int.from_bytes(b'\\xff\\xff', 'big', signed=True)** → **-1**; **int.from_bytes(b'\\x04\\x00', 'big')** → **1024**. Удобно для парсинга бинарных протоколов и файлов.

**Граничные случаи:** **(0).to_bytes(0, 'big')** → **b''** (длина 0 допустима только для числа 0); **(0).to_bytes(1, 'big')** → **b'\\x00'**. Для отрицательного числа **length** должна быть достаточной для представления в дополнительном коде: **(-1).to_bytes(1, 'big', signed=True)** → **OverflowError** (один байт не хватает); **(-1).to_bytes(2, 'big', signed=True)** → **b'\\xff\\xff'**. **from_bytes(b'', 'big')** → **0**.

**byteorder 'big' vs 'little':** в **big-endian** (сетевой порядок) старший байт идёт первым: **(0x1234).to_bytes(2, 'big')** → **b'\\x12\\x34'**. В **little-endian** (порядок x86) младший байт первым: **(0x1234).to_bytes(2, 'little')** → **b'\\x34\\x12'**. При чтении бинарных файлов или протоколов нужно знать порядок байт в формате; **int.from_bytes(data, 'little')** даёт правильное число при том же порядке, что и при записи.

**Практический пример: число байт для неотрицательного int.** Для **n >= 0** минимальная длина в байтах: **length = (n.bit_length() + 7) // 8**; для **n == 0** получается **0** — тогда **(0).to_bytes(0, 'big')** даёт **b''**. Если нужен хотя бы один байт для нуля: **length = max(1, (n.bit_length() + 7) // 8)**. Для отрицательных при **signed=True** длина должна покрывать дополнительный код: например, **-1** в одном байте — это **0xff** (один байт достаточен); **-256** в одном байте не помещается (нужно 2 байта).

**Плохо vs лучше (to_bytes/from_bytes):**

```python
# Плохо: фиксированная длина без проверки — OverflowError для больших чисел
n = 2**20
n.to_bytes(2, 'big')   # OverflowError

# Лучше: вычислить длину или обработать OverflowError
length = (n.bit_length() + 7) // 8
n.to_bytes(length, 'big')
# или: n.to_bytes(length or 1, 'big')  для n >= 0 с минимум 1 байт для 0

# Парсинг бинарного протокола: 2 байта big-endian → int
data = b'\\x04\\x00'
value = int.from_bytes(data, 'big')   # 1024
```

**Типичные ошибки:** передать в **to_bytes** длину меньше необходимой — **OverflowError**. Перепутать **byteorder** при чтении файла — получится неверное число. Использовать **signed=False** при разборе отрицательных (доп. код) — для **b'\\xff\\xff'** с **signed=False** получится 65535, с **signed=True** — -1.

**Откуда формула числа байт: (n.bit_length() + 7) // 8.** Для **n >= 1** число бит без ведущих нулей — **k = bit_length(n)**; то есть **2^(k-1) <= n < 2^k**. Число байт для представления **k** бит: **ceil(k / 8) = (k + 7) // 8**. Для **n == 0** **bit_length(0) == 0**, получается **0** байт — и **(0).to_bytes(0, 'big')** действительно даёт **b''**. Для отрицательных при **signed=True** в дополнительном коде нужно место для знакового бита: **-1** в одном байте — **0xff** (1 байт), **-256** — уже 2 байта (**0xff00** в big-endian).

**Пример: bit_count для набора флагов (битовая маска).**

```python
# Флаги как биты в одном числе
READ, WRITE, EXEC = 1, 2, 4
flags = READ | WRITE   # 3 = 0b11
flags.bit_count()      # 2 — включено два флага
(READ | WRITE | EXEC).bit_count()   # 3

# Отрицательные: количество единиц в доп. коде (бесконечные ведущие единицы)
(-1).bit_count()   # 2 на типичной 64-битной платформе (интерпретация ограниченного числа бит)
```

**Пример: roundtrip to_bytes → from_bytes и парсинг заголовка.**

```python
# Запись числа в файл/сокет и чтение обратно
n = 0x1234
data = n.to_bytes(2, 'big')           # b'\x12\x34'
restored = int.from_bytes(data, 'big')  # 0x1234
assert n == restored

# Парсинг: первые 2 байта — длина, следующие N байт — полезная нагрузка
payload = b"hello"
header = len(payload).to_bytes(2, 'big')
message = header + payload
length = int.from_bytes(message[:2], 'big')
body = message[2:2+length]   # b'hello'
```

<details>
<summary>Вопрос: в каком случае <code>n.to_bytes(1, 'big', signed=True)</code> выдаёт OverflowError?</summary>

**OverflowError** возникает, когда число **n** не помещается в **length** байт в дополнительном коде. Для **length=1** и **signed=True** диапазон: **-128 <= n <= 127**. Поэтому **(-129).to_bytes(1, 'big', signed=True)** даёт **OverflowError** (нужно минимум 2 байта). **(-1).to_bytes(1, 'big', signed=True)** успешно возвращает **b'\\xff'**. Для **signed=False** и **length=1** допустимы только **0 <= n <= 255**; **256.to_bytes(1, 'big')** — **OverflowError**. См. [§1.3](#13-целые-числа-int).

</details>

#### Почему изменили поведение деления

В Python 2 оператор **`/`** для двух целых чисел выполнял **целочисленное** деление (округление вниз). Это было источником частых ошибок: новички ожидали вещественный результат (**`7/2 == 3.5`**), а получали **3**. Кроме того, один и тот же оператор вёл себя по-разному в зависимости от типов операндов — нарушение принципа «явное лучше неявного». В Python 3 **`/`** всегда даёт вещественное деление; для целочисленного деления используется явный оператор **`//`** (floor division). Так устраняется неоднозначность и уменьшается число типичных ошибок.

#### Совместимость с Python 2: **future**.division

В **Python 2** можно включить поведение деления «как в Python 3», поместив в начало файла:

```python
from __future__ import division
```

После этого **`/`** выполняет вещественное деление, **`//`** — целочисленное. Это рекомендуется при написании кода, который планируется переносить в Python 3.

#### Целочисленное деление и отрицательные числа

Оператор **`//`** выполняет **floor division** — округление вниз к ближайшему целому (в сторону минус бесконечности), а не к нулю. Это важно для отрицательных чисел:

```python
7 // 2      # 3
-7 // 2     # -4 (не -3!)
7 // -2     # -4
-7 // -2    # 3
```

Функция **`math.floor()`** даёт тот же результат для положительных и отрицательных чисел: **math.floor(-7/2)** в Py3 равно **-4**, так как **-7/2 == -3.5**, а **floor(-3.5) == -4**.

#### Остаток от деления (%) и знак

В Python остаток **a % b** всегда удовлетворяет инварианту: **`(a // b) * b + (a % b) == a`** при **b != 0**. Знак результата **a % b** совпадает со знаком **делителя b**, а не делимого. Это отличается от поведения в C и части других языков (где знак остатка может совпадать с **a**).

```python
-7 % 2      # 1  (не -1): (-7 // 2) * 2 + (-7 % 2) = -4*2 + 1 = -7
7 % -2      # -1: (7 // -2) * (-2) + (7 % -2) = -4*(-2) + (-1) = 7
-7 % -2     # -1
```

**Практический вывод:** при работе с отрицательными числами не полагайтесь на интуицию из других языков; проверяйте формулу **a == (a // b) \* b + (a % b)**. Для «положительного остатка» при отрицательном **a** можно использовать **a % b** (в Python он уже неотрицателен при **b > 0**) или **((a % b) + b) % b** при **b < 0** для приведения к диапазону **[0, |b|)**.

**Граничные случаи деления и остатка:** (1) **Деление на ноль:** **a / 0**, **a // 0**, **a % 0** при **b == 0** вызывают **ZeroDivisionError** для любых **a** (и int, и float). (2) **float infinity:** **1.0 / 0.0** → **inf** (бесконечность, float); **-1.0 / 0.0** → **-inf**; **0.0 / 0.0** → **nan** (not a number). Для **int** деление на 0 всегда **ZeroDivisionError**; для **float** — **inf** или **nan** в зависимости от операндов. (3) **float // int:** **7.0 // 2** → **3.0** (тип результата float, так как один операнд float). (4) **Округление float:** **int(3.7)** → **3** (к нулю); **int(-3.7)** → **-3**; **math.floor(3.7)** → **3**, **math.floor(-3.7)** → **-4**.

**Плохо vs лучше при делении и остатке:**

```python
# Плохо: ожидать, что остаток при отрицательном a будет отрицательным (как в C)
r = -7 % 2   # в Python r == 1; в C-подобных языках может быть -1

# Лучше: явно опираться на инвариант (a // b) * b + (a % b) == a
q, r = divmod(-7, 2)   # q == -4, r == 1; проверка: -4*2 + 1 == -7

# Плохо: писать два вызова, когда нужны и частное, и остаток
q, r = a // b, a % b   # два обращения к a, b

# Лучше: один вызов divmod — и читаемо, и один проход
q, r = divmod(a, b)
```

#### divmod(a, b)

Встроенная функция **divmod(a, b)** возвращает кортеж **`(a // b, a % b)`** — целочисленное частное и остаток за один вызов. Удобно, когда нужны оба значения (например, разбиение секунд на минуты и секунды, или конвертация в систему счисления). Тип элементов кортежа совпадает с типом **a** и **b** (при целых — **int**; при **float** — **float**).

**Пример: разбиение общего количества секунд на часы, минуты, секунды.** Используем **divmod** дважды: сначала делим на 60 — получаем минуты и секунды; затем минуты делим на 60 — получаем часы и оставшиеся минуты.

```python
total_seconds = 3661
minutes, seconds = divmod(total_seconds, 60)   # (61, 1)
hours, minutes = divmod(minutes, 60)           # (1, 1)
# Итого: 1 час 1 мин 1 сек
```

**Пример: конвертация числа в строку в заданной системе счисления (цифры 0–9, A–Z).** Повторно применяем **divmod(n, base)** и собираем остатки (цифры) в обратном порядке. Для основания 2–36 удобно использовать **int** и строку цифр; **divmod** даёт одну «цифру» за итерацию.

```python
def to_base(n, base=10):
    if n == 0:
        return "0"
    digits = []
    while n:
        n, r = divmod(n, base)
        digits.append("0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZ"[r])
    return "".join(reversed(digits))
to_base(255, 16)   # 'FF'
to_base(10, 2)     # '1010'
```

```python
divmod(17, 5)    # (3, 2)
divmod(-7, 2)    # (-4, 1)
divmod(100, 60)  # (1, 40) — 100 секунд = 1 мин 40 сек
```

**Граничные случаи divmod:** (1) **divmod(0, 0)** — **ZeroDivisionError**: деление на ноль не определено. (2) **divmod(a, 0)** при **b == 0** — **ZeroDivisionError** для любых **a**. (3) **divmod** с **float**: **divmod(7.5, 2.5)** возвращает **(3.0, 0.0)** — частное и остаток как **float**; **divmod(7.5, 2)** — **a** приводится к **float**, результат **float**. (4) Инвариант: **a == q \* b + r** при **divmod(a, b) == (q, r)**; для отрицательных **b** знак остатка совпадает со знаком **b**.

#### Тип результата / и //

- Оператор **/** в Python 3 всегда возвращает **float**, даже при двух целых операндах: **7 / 2** → **3.5** (тип **float**).
- Оператор **//** возвращает **int**, если оба операнда **int**; иначе тип совпадает с типом операндов (при **float** — **float**): **7 // 2** → **3** (int); **7.0 // 2** → **3.0** (float).

#### Тип int: неограниченная точность

В Python 2 тип **long** использовался для целых, превышающих диапазон **int** (на платформе это могло быть, например, 2^31 или 2^63). Литералы длинных чисел заканчивались суффиксом **L** (например, **`123L`**). В Python 3 тип **long** удалён; **int** имеет **неограниченную точность** — можно оперировать сколь угодно большими целыми без переполнения. Конструктор **`int(x, base)`** принимает второй аргумент — основание системы счисления (2–36); удобно для парсинга двоичных, восьмеричных, шестнадцатеричных строк:

```python
int('1010', 2)    # 10
int('FF', 16)     # 255
int('77', 8)      # 63
```

### 1.4 raise и исключения

#### Синтаксис raise

В **Python 2** допускались формы **`raise Exception, args`** и **`raise Exception(args).traceback`**. В **Python 3** эти варианты **удалены**; используется только форма **`raise Exception(...)`** (или **`raise`** без аргументов — см. ниже). Это устраняет неоднозначность и упрощает парсер и обработку исключений.

**raise без аргументов:** внутри блока **except** можно написать просто **`raise`** — тогда будет повторно выброшено **текущее** исключение (то, которое перехвачено). Это полезно для логирования или частичной обработки с последующей передачей исключения выше. При этом **сохраняется оригинальный traceback**; если бы вы написали **`raise e`**, в некоторых случаях traceback мог бы указывать на строку с **raise**, а не на исходное место ошибки. В Python 3 **`raise`** без аргументов **вне** блока **except** вызывает **RuntimeError: No active exception to reraise**.

**Пример: логирование и повторный выброс:**

```python
try:
    risky_operation()
except ValueError as e:
    logger.exception("Ошибка при выполнении: %s", e)   # логируем с traceback
    raise   # пробрасываем то же исключение с оригинальным traceback
# Если бы написали raise e, в старых версиях traceback мог бы указывать на эту строку
```

#### Цепочки исключений: raise ... from ...

Явная цепочка причин задаётся конструкцией **`raise NewException(...) from orig_exception`** и сохраняется в атрибуте **`__cause__`** нового исключения. При перехвате исключения и выбросе нового полезно указывать причину, чтобы в traceback было видно и исходную ошибку, и следствие:

```python
try:
    int("not a number")
except ValueError as e:
    raise RuntimeError("Invalid config") from e
```

**Пошагово: что происходит при raise NewError(...) from e.** (1) В блоке **except** перехвачено исключение **e** (например, **ValueError**). (2) Выполняется **raise RuntimeError("Invalid config") from e**. Интерпретатор создаёт новый объект исключения **RuntimeError("Invalid config")** и присваивает его атрибуту \***\*cause\*\*** значение **e** — явная причина. (3) Атрибут \***\*context\*\*** нового исключения тоже может быть установлен автоматически (контекст — то исключение, при обработке которого произошло новое); при наличии **from e** в выводе traceback показывается и \***\*cause\*\*** (явная причина), и при необходимости \***\*context\*\***. (4) Исключение **RuntimeError** выбрасывается вверх по стеку; при печати traceback интерпретатор выводит сначала **исходное** исключение (**e**), затем строку «The above exception was the direct cause of the following exception», затем **новое** исключение. Итог: **from e** сохраняет явную связь «новое исключение из-за e» и делает отладку предсказуемой.

В traceback будет и **исходное** исключение (cause), и **новое** (effect). Чтобы **скрыть** цепочку из вывода (но сохранить её в коде), можно писать **`raise ... from None`** — тогда у нового исключения **`__suppress_context__`** станет **True**, и интерпретатор не будет печатать \***\*context\*\*** (исходное исключение из блока except). Исключение \***\*cause\*\*** при этом остаётся доступным в коде.

#### Неявная цепочка (**context**)

Если внутри блока **`except`** выбросить **другое** исключение **без** **`from`**, то исходное исключение сохраняется в атрибуте **`__context__`** нового (implicit chaining). Интерпретатор по умолчанию выводит и \***\*context\*\***, и \***\*cause\*\***, если они есть. Различие: \***\*cause\*\*** — явная причина (задаётся только через **`raise ... from ...`**); \***\*context\*\*** — контекст, в котором произошло новое исключение (задаётся автоматически при raise внутри except).

#### except E as e: очистка имени

В Python 3 после выхода из блока `except E as e:` имя **`e`** удаляется (чтобы не держать ссылку на traceback и объект исключения). Это уменьшает риск циклических ссылок и утечек памяти.

#### Иерархия исключений

- **BaseException** — корень иерархии.
  - **Exception** — то, что обычно перехватывают в коде (ValueError, TypeError, KeyError и т.д.).
  - **SystemExit**, **KeyboardInterrupt**, **GeneratorExit** наследуют **BaseException**, но **не** Exception.

**Почему так сделано:** эти три типа — не «ошибки приложения», а механизмы завершения или прерывания. Если поймать их общим `except Exception:`, можно случайно подавить выход из программы (`SystemExit`), прерывание пользователем (`KeyboardInterrupt`) или корректное закрытие генератора (`GeneratorExit`). Поэтому их перехватывают только при явной необходимости.

**Пример «плохо» — перехват SystemExit:**

```python
# Плохо: общий except Exception перехватывает ВСЕ исключения, наследующие Exception
# SystemExit, KeyboardInterrupt, GeneratorExit НЕ наследуют Exception — их не поймает except Exception
# Но если бы вы написали except BaseException: или except:, то:
try:
    sys.exit(0)   # выбрасывает SystemExit(0)
except BaseException:
    pass   # «проглотили» выход — программа не завершится, выполнение продолжится!
# Правило: в прикладном коде ловите только Exception (или конкретные типы), не BaseException и не голый except:
try:
    do_work()
except Exception as e:
    log(e)
    raise
```

#### Таблица иерархии встроенных исключений (упрощённо)

| Класс                                             | Наследует     | Назначение                                                      |
| ------------------------------------------------- | ------------- | --------------------------------------------------------------- |
| **BaseException**                                 | —             | Корень иерархии; перехватывать в общем коде не рекомендуется    |
| **Exception**                                     | BaseException | Базовый класс для «нормальных» исключений приложения            |
| **SystemExit**                                    | BaseException | Выход из программы (sys.exit); не перехватывать общим except    |
| **KeyboardInterrupt**                             | BaseException | Прерывание пользователем (Ctrl+C)                               |
| **GeneratorExit**                                 | BaseException | Закрытие генератора (close); не перехватывать в прикладном коде |
| **ValueError**, **TypeError**, **KeyError** и др. | Exception     | Конкретные типы ошибок; их перехватывают в коде                 |
| **OSError** и подклассы                           | Exception     | Ошибки ввода-вывода (FileNotFoundError, PermissionError и т.д.) |

**Подклассы OSError (часто используемые):** **FileNotFoundError** — файл или каталог не найден; **PermissionError** — доступ запрещён; **TimeoutError** — истекло время ожидания; **ConnectionError** и подклассы (**ConnectionRefusedError**, **ConnectionResetError**) — ошибки сетевого соединения; **IsADirectoryError**, **NotADirectoryError** — ожидался файл/каталог, а не наоборот. Перехват **OSError** ловит все эти подклассы; перехват **FileNotFoundError** — только «файл не найден». Рекомендуется перехватывать конкретный подкласс, когда известен тип ошибки (например, **FileNotFoundError** при открытии файла), и **OSError** для общего случая I/O.

**Правило:** в прикладном коде перехватывайте **Exception** (или конкретные подклассы), а не **BaseException**. Иначе вы рискуете «проглотить» **SystemExit** или **KeyboardInterrupt** и сломать ожидаемое поведение программы или оболочки.

**Граничные случаи raise и except:** (1) **raise** без аргументов — допустимо **только** внутри блока **except**; повторно выбрасывает текущее исключение (re-raise). Вне **except** — **RuntimeError: No active exception to re-raise**. (2) **except E as e:** — имя **e** удаляется в конце блока **except**; после блока обратиться к **e** нельзя. Если нужно сохранить информацию — сохраните **e.args**, **str(e)** или **type(e)** в переменную до выхода. (3) **except Exception:** без **as e** — исключение перехвачено, но к объекту обратиться нельзя; для логирования или **from e** нужен **as e**. (4) **try** без **except** — синтаксически допустим с **finally**; **try:** ... **finally:** ... выполняет **finally** при любом выходе (нормальном, исключении, **return**), но исключение не перехватывается и продолжает распространяться. (5) Пустой **except:** — ловит все исключения (в т.ч. **KeyboardInterrupt**); в Py3 рекомендуется явно указывать **except Exception:**, чтобы не перехватывать **SystemExit** и **KeyboardInterrupt**.

#### Перехват нескольких типов: except (A, B) и порядок except

Перехватить несколько типов исключений одним блоком можно, указав **кортеж** типов: **except (ValueError, TypeError) as e:**. Тогда **e** — экземпляр того исключения, которое было выброшено (одного из перечисленных). **Порядок блоков except важен:** интерпретатор проверяет их сверху вниз; первый подходящий блок выполняется, остальные не проверяются. Поэтому более **специфичные** типы (подклассы) должны идти **выше** более общих (родительских). Иначе блок с родительским типом перехватит исключение раньше, и блок с подклассом никогда не выполнится.

```python
try:
    ...
except (ValueError, TypeError) as e:
    # обрабатываем оба типа
    pass
except OSError as e:
    pass
# Плохо: except Exception перед except ValueError — ValueError никогда не будет пойман отдельно
try:
    ...
except Exception:
    pass
except ValueError:   # недостижимый код! Exception уже поймал ValueError
    pass
```

**Правило:** располагать **except** от более конкретного к более общему: сначала **ValueError**, **KeyError** и т.д., затем **Exception**, при необходимости в конце **BaseException** (обычно не рекомендуется).

#### ExceptionGroup и except \* (PEP 654, Python 3.11+)

В asyncio и других контекстах (например, **TaskGroup**, параллельные задачи) несколько исключений могут возникнуть **одновременно**. Обычный **except** перехватывает одно исключение; если нужно обработать несколько и не терять остальные, в Python 3.11+ используется **ExceptionGroup**.

**Что такое ExceptionGroup:** объект **ExceptionGroup(msg, exceptions)** оборачивает **список** исключений (или вложенных **ExceptionGroup**). Атрибут **.exceptions** — кортеж вложенных исключений; **.message** — строка-описание. При печати traceback выводятся все вложенные исключения. Создать группу вручную: **raise ExceptionGroup("несколько ошибок", [ValueError("a"), TypeError("b")])**.

**Синтаксис except \*Type:** перехват **`except *Type:`** (звезда перед типом) обрабатывает **все** исключения в группе, которые являются экземплярами **Type** (или подкласса). Обработанные исключения из группы «вынимаются»; необработанные остаются и пробрасываются в новом **ExceptionGroup**. Можно указать несколько блоков **except \*Type1:**, **except \*Type2:** — каждый обрабатывает свой тип из группы. Если после обработки в группе не осталось исключений, исключение считается перехваченным; если остались — пробрасывается новый **ExceptionGroup** с оставшимися.

**Пошагово: что происходит при except \*ValueError.** (1) Выброшено **ExceptionGroup("err", [ValueError(1), TypeError(2), ValueError(3)])**. (2) Блок **except \*ValueError:** срабатывает: интерпретатор перебирает **group.exceptions** и отбирает все экземпляры **ValueError** (здесь **ValueError(1)** и **ValueError(3)**). (3) Для отобранных выполняется тело блока **except \*ValueError:** (например, логирование). (4) Исключения **TypeError(2)** не подходят под **ValueError** — они остаются в группе. (5) После выхода из блока формируется новый **ExceptionGroup** только из оставшихся исключений (**TypeError(2)**) и он пробрасывается дальше. Если бы обработали и **except \*TypeError:**, группа могла бы быть полностью обработана.

**Пример: обработка группы и повторный выброс остатка.**

```python
# Python 3.11+
def main():
    try:
        raise ExceptionGroup("проверка",
            [ValueError("v1"), TypeError("t1"), ValueError("v2")])
    except *ValueError as e:
        print("Пойманы ValueError:", e.exceptions)   # (ValueError('v1'), ValueError('v2'))
        # TypeError('t1') остаётся в группе и пробрасывается
    except *TypeError as e:
        print("Пойманы TypeError:", e.exceptions)

# Сначала сработает except *ValueError; затем пробросится ExceptionGroup с [TypeError('t1')]
# Если добавить except *TypeError — оба типа будут обработаны по очереди
```

**Когда использовать:** при параллельном выполнении (asyncio **TaskGroup**, **asyncio.gather** с **return_exceptions=False** и последующим сбором исключений), когда нужно за один проход обработать все ошибки одного типа из группы и пробросить остальные. Для обычного последовательного кода достаточно **except Type:**.

**Граничные случаи:** вложенные **ExceptionGroup** (в **exceptions** могут быть снова группы); **except \*BaseException** перехватывает всё из группы (осторожно). **group.subgroup(condition)** — создать подгруппу из исключений, удовлетворяющих условию; **group.split(condition)** — пара (matched, rest) для разбиения группы.

**subgroup(condition) и split(condition):** метод **group.subgroup(condition)** возвращает новый **ExceptionGroup** только из тех исключений в **group.exceptions**, для которых **condition(e)** истинно (или **type(e)** совпадает с **condition**, если **condition** — класс). **group.split(condition)** возвращает пару **(matched, rest)**: **matched** — подгруппа «подходящих», **rest** — подгруппа «остальных»; если одна из частей пуста, возвращается **None** вместо пустой группы. Удобно для ручной фильтрации перед обработкой или логированием. Пример: **eg = ExceptionGroup("e", [ValueError(1), TypeError(2)]); v, t = eg.split(ValueError)** → **v** — группа с **ValueError(1)**, **t** — с **TypeError(2)**.

**Пошагово: как asyncio собирает ExceptionGroup (концептуально).** При **asyncio.gather(\*tasks, return_exceptions=False)** несколько задач выполняются «параллельно». Если несколько из них выбрасывают исключения, asyncio **собирает** их в один **ExceptionGroup** и пробрасывает его. Вызывающий код может обработать группу через **except \*ValueError:** и т.д., обработать все **ValueError** из всех упавших задач, а остальные типы пробросить дальше. Без **ExceptionGroup** пришлось бы ждать первое исключение и терять остальные; с группой видна полная картина сбоев.

**Плохо vs лучше (ExceptionGroup):**

```python
# Плохо: ловить ExceptionGroup как обычное исключение — теряется возможность обработать по типам
try:
    asyncio.run(main_with_tasks())
except ExceptionGroup as e:
    log(e)   # залогировали, но не разобрали по типам

# Лучше: обработать по типам и при необходимости пробросить остальное
try:
    asyncio.run(main_with_tasks())
except *ValueError as e:
    for ex in e.exceptions:
        log("ValueError", ex)
except *OSError as e:
    for ex in e.exceptions:
        log("OSError", ex)
# Непойманные типы из группы пробросятся в новом ExceptionGroup
```

**Типичные ошибки:** путать **except \*Type:** с **except (Type1, Type2):** — во втором случае перехватывается **одно** исключение типа Type1 или Type2; в первом — **все** исключения указанного типа **из группы**. В **except \*Type as e** переменная **e** — это **тот же ExceptionGroup** (или подгруппа); **e.exceptions** — кортеж отобранных исключений. Не забывать, что после **except \*ValueError** группа может **проброситься** с оставшимися исключениями — нужна обработка выше или ещё один **except \*Type2**.

**Полный пример: asyncio.gather и ExceptionGroup (Python 3.11+).**

```python
# Python 3.11+
import asyncio

async def task_ok():
    return 1

async def task_value_error():
    raise ValueError("ошибка в задаче 1")

async def task_type_error():
    raise TypeError("ошибка в задаче 2")

async def main():
    # Несколько задач; две из них выбрасывают исключения
    results = await asyncio.gather(
        task_ok(),
        task_value_error(),
        task_type_error(),
        return_exceptions=False   # исключения не возвращаются как значения, а пробрасываются
    )
    # Сюда не попадём: gather соберёт ValueError и TypeError в ExceptionGroup

async def main_with_handling():
    try:
        await asyncio.gather(
            task_ok(),
            task_value_error(),
            task_type_error(),
            return_exceptions=False
        )
    except *ValueError as e:
        print("Пойманы ValueError:", [ex.args for ex in e.exceptions])
        # e — подгруппа только из ValueError; остальное пробросится
    except *TypeError as e:
        print("Пойманы TypeError:", [ex.args for ex in e.exceptions])

# asyncio.run(main())   # ExceptionGroup с [ValueError(...), TypeError(...)]
asyncio.run(main_with_handling())   # сначала ValueError, затем пробросится ExceptionGroup с [TypeError(...)]
```

**Пошагово: порядок выполнения.** (1) **asyncio.gather** запускает все корутины «параллельно». (2) **task_ok()** завершается успешно; **task_value_error()** и **task_type_error()** выбрасывают исключения. (3) asyncio **собирает** все исключения в один **ExceptionGroup("...", [ValueError(...), TypeError(...)])** и пробрасывает его. (4) В **main_with_handling** сначала срабатывает **except \*ValueError** — обрабатываются все **ValueError** из группы; после выхода из блока формируется **новый** ExceptionGroup только из оставшихся (**TypeError**) и он пробрасывается. (5) Внешний обработчик или **except \*TypeError** может поймать оставшуюся группу. Если добавить **except \*TypeError** в тот же **try**, оба типа будут обработаны по очереди и группа будет полностью перехвачена.

**Пример: subgroup() и split() в коде.**

```python
# Python 3.11+
eg = ExceptionGroup("ошибки", [
    ValueError("v1"),
    TypeError("t1"),
    ValueError("v2"),
    OSError("os1"),
])

# Подгруппа только из ValueError
vals = eg.subgroup(ValueError)
# vals.exceptions == (ValueError('v1'), ValueError('v2'))

# Разбить на «подходящие» и «остальные»
matched, rest = eg.split(lambda e: isinstance(e, (ValueError, TypeError)))
# matched.exceptions — (ValueError('v1'), TypeError('t1'), ValueError('v2'))
# rest.exceptions — (OSError('os1'),)

# Обработка: залогировать ValueError, пробросить остальное
vals, rest = eg.split(ValueError)
if vals:
    for ex in vals.exceptions:
        log("ValueError", ex)
if rest:
    raise rest
```

**Граничные случаи (ExceptionGroup):** пустая группа **ExceptionGroup("msg", [])** — допустима; **except \*Type** для неё не отбирает ничего, группа считается «обработанной» и не пробрасывается. Вложенные группы: **exceptions** могут содержать другие **ExceptionGroup**; **except \*ValueError** отбирает только «прямые» экземпляры **ValueError**, не рекурсивно из вложенных групп (для рекурсивного обхода используют **exceptions** и проверку **isinstance(e, ExceptionGroup)**). **except \*BaseException** перехватывает все исключения в группе — осторожно, можно скрыть **KeyboardInterrupt** и т.п.

**Практический вывод:** при использовании **asyncio.TaskGroup** или **asyncio.gather** с несколькими задачами всегда учитывайте, что может прилететь **ExceptionGroup**. Обрабатывайте по типам через **except \*Type:** и при необходимости пробрасывайте остаток; для ручной фильтрации используйте **split(condition)** или **subgroup(condition)**.

<details>
<summary>Вопрос: чем отличается <code>except (ValueError, TypeError):</code> от <code>except *ValueError:</code> и <code>except *TypeError:</code>?</summary>

**except (ValueError, TypeError):** перехватывает **одно** исключение, которое является экземпляром **ValueError** или **TypeError**. Если выброшено **ExceptionGroup** с несколькими исключениями, этот блок перехватит **саму группу** (она наследует **BaseException**), но не разберёт её по типам — внутри **e** будет **ExceptionGroup**, у которого **e.exceptions** — кортеж всех вложенных исключений.

**except \*ValueError:** и **except \*TypeError:** работают **только** с **ExceptionGroup**. Они перебирают **group.exceptions** и отбирают исключения по типу; обработанные «вынимаются», необработанные пробрасываются в новом **ExceptionGroup**. Для одиночного исключения (не группы) **except \*Type** не используется — одиночное исключение ловится обычным **except Type:**. См. [§1.4](#14-исключения-и-контекстный-менеджер).

</details>

#### Когда использовать raise ... from None

Иногда вы перехватываете исключение и выбрасываете своё (более высокоуровневое), но не хотите засорять traceback **исходным** исключением — например, когда детали реализации не должны утекать наружу или когда контекст уже не актуален. В таком случае пишут **`raise NewError("message") from None`**. Тогда у нового исключения \***\*suppress_context\*\*** устанавливается в **True**, и интерпретатор **не печатает** цепочку \***\*context\*\*** (исходное исключение из except). При этом \***\*cause\*\*** вы не задаёте (остаётся **None**), так что явная цепочка причин не показывается. Использовать **from None** стоит осознанно: вы теряете информацию для отладки; в логах или при отладке иногда полезнее сохранять цепочку через **raise ... from e**.

**Плохо vs лучше: from e и from None.**

```python
# Лучше: явная причина — в логах видна полная цепочка (для отладки и поддержки)
try:
    parse_config(path)
except OSError as e:
    raise ConfigError(f"Не удалось прочитать {path}") from e

# Приемлемо from None: когда низкоуровневая ошибка не нужна пользователю API
try:
    internal_operation()
except ValueError as e:
    raise UserFacingError("Некорректный ввод") from None   # детали e скрыты

# Плохо: перехватить и выбросить новое без from — контекст будет в __context__, но формулировка «During handling...» менее явная, чем «direct cause»
```

#### sys.exc_info() и sys.exception() (Python 3.11+)

- **`sys.exc_info()`** возвращает кортеж **`(type, value, traceback)`** для **текущего** обрабатываемого исключения (внутри блока **except**). **type** — класс исключения, **value** — экземпляр, **traceback** — объект traceback. Вне блока **except** возвращает **(None, None, None)**. В асинхронном коде (корутины) «текущее» исключение может относиться к другой корутине, поэтому полагаться на **exc_info()** в **finally** или в отложенном коде в asyncio небезопасно.
- **`sys.exception()`** (Python 3.11+) возвращает **только** текущее исключение (эквивалент **sys.exc_info()[1]**); в корутинах и **finally** безопаснее использовать **sys.exception()**, так как не возвращает traceback и не создаёт ссылок на кадры. Рекомендуется в новом коде вместо **exc_info()[1]**.

**Пример использования в except:**

```python
try:
    int("x")
except ValueError:
    typ, val, tb = sys.exc_info()
    print(typ, val)   # <class 'ValueError'> invalid literal for int()
    # val — тот же объект, что и в except ValueError as e: e
```

#### exc.add_note(note) (PEP 678, Python 3.11+)

К объекту исключения можно добавить заметку; она попадёт в traceback:

```python
e = ValueError("bad value")
e.add_note("Context: config file line 10")
raise e
```

Заметки хранятся в **`__notes__`** и выводятся при печати traceback. Это удобно для добавления контекста (например, путь к файлу, номер строки конфига, значение переменной) без изменения типа исключения и без потери информации при цепочке исключений.

Пример с несколькими заметками:

```python
try:
    parse_config("config.ini")
except ValueError as e:
    e.add_note("File: config.ini")
    e.add_note("Section: [database]")
    raise
```

При выводе traceback все заметки будут показаны после блока с исключением.

#### Как traceback показывает **cause** и **context**

При выводе исключения интерпретатор печатает:

1. **Traceback** для **текущего** исключения (то, которое выброшено последним).
2. Если у исключения задан \***\*cause\*\*** (через **raise ... from e**), выводится блок **The above exception was the direct cause of the following exception:** и traceback для \***\*cause\*\***.
3. Если у исключения задан \***\*context\*\*** (исключение выброшено внутри **except** без **from**), и при этом \***\*suppress_context\*\*** не **True**, выводится блок **During handling of the above exception, another exception occurred:** и traceback для \***\*context\*\***.

Таким образом, при **raise New from e** вы видите и исходную ошибку (cause), и новую (effect); при **raise New** внутри **except** без **from** вы видите контекст (какое исключение обрабатывали) и новое исключение. При **raise New from None** контекст не печатается — \***\*suppress_context\*\*** установлен в **True**.

**Пошаговый сценарий: различие from и без from.** Допустим, в блоке **except ValueError** вы делаете **raise RuntimeError("ошибка") from e**. В traceback сначала будет показан **RuntimeError** (текущее исключение), затем строка «The above exception was the direct cause...», затем traceback **ValueError** (e). Если бы вы написали просто **raise RuntimeError("ошибка")** (без **from**), то **ValueError** попал бы в \***\*context\*\***, и в выводе было бы «During handling of the above exception, another exception occurred» и traceback **ValueError**. То есть и в том и в другом случае исходная ошибка видна; разница в формулировке: «прямая причина» (from) vs «во время обработки произошло другое» (context). Для отладки предпочтительнее явно указывать **from e**, когда новое исключение логически вытекает из перехваченного.

#### except E as e: почему имя e удаляется после блока

В Python 2 имя **e** в **except E as e:** оставалось в области видимости после выхода из блока **except**. Объект исключения хранит ссылку на **traceback**, а traceback ссылается на кадры стека, которые могут ссылаться на объекты в той же области видимости. Это создавало **циклические ссылки** (exception → traceback → frame → ... → exception) и мешало сборщику мусора освобождать память до явного удаления или выхода из функции. В Python 3 имя **e** удаляется в конце блока **except**, что разрывает цикл и позволяет быстрее освобождать память. Если нужно использовать объект исключения после блока (например, передать в логгер), сохраните нужные атрибуты (например, **e.args**, **str(e)**) в переменную до выхода из блока.

#### Практический вывод по исключениям

- Используйте только **raise Exception(...)**; формы **raise E, args** в Py3 нет.
- При перехвате и выбросе нового исключения указывайте причину: **raise New(...) from e** — в traceback будет видна полная цепочка.
- Перехватывайте **Exception** (или конкретные типы), а не **BaseException**, чтобы не «проглотить» **SystemExit** или **KeyboardInterrupt**.
- Для добавления контекста к исключению без смены типа используйте **e.add_note("...")** (Python 3.11+).
