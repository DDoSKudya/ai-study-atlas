# Часть XIII. Паттерны проектирования

> **Цель:** понять, **зачем реально нужны паттерны проектирования** в Python‑коде, научиться **узнавать типичные ситуации**, где они помогают:
>
> - осмысленно применять **создающие паттерны** (Singleton, Factory Method, Abstract Factory, Builder, Prototype);
> - использовать **структурные паттерны** (Adapter, Bridge, Composite, Decorator, Facade, Flyweight, Proxy) для организации связей между классами;
> - управлять поведением системы через **поведенческие паттерны** (Chain of Responsibility, Command, Iterator, Mediator, Memento, Observer, State, Strategy, Template Method, Visitor);
> - видеть и применять **“pythonic‑паттерны”**: менеджеры контекста, декораторы, генераторы, дескрипторы, EAFP и др.;
> - при этом **не превращать код в “каталог паттернов”** и понимать, когда паттерн — помощь, а когда оверинжениринг.
>
> Всё — **простым языком**, но с корректной терминологией и акцентом на Python.

---

## 1. Общая картина: зачем нужны паттерны

Паттерны проектирования часто воспринимают как «академическую теорию» из книги **GoF** (Gang of Four). На практике это:

- **общий язык** между разработчиками:
  - вместо «сделай тут несколько if/else и таблицу функций» — «давай применим Strategy»;
  - вместо «обертка, которая добавляет логирование и кеширование» — «Decorator / Proxy».
- **шаблоны решений** типичных проблем:
  - «как создавать сложные объекты» (Builder, Factory Method);
  - «как адаптировать старый код под новый интерфейс» (Adapter);
  - «как уведомлять много объектов об изменении одного» (Observer).
- **способ организовать зависимости и расширяемость**:
  - добавить новый вариант поведения **без** изменений старого кода (OCP);
  - управлять изменениями через абстракции (DIP, из Часть XII).

Важно:

- паттерн — это **не цель**, а **средство**;
- в Python многие идеи паттернов уже **встроены в язык** (итераторы, контекст‑менеджеры, декораторы, функции первого класса), поэтому реализации проще, чем в Java/C++.

## 2. Краткая шпаргалка (паттерны этой части)

| Группа | Паттерн | Вопрос, который решает |
|--------|---------|------------------------|

| | Factory Method | Как делегировать создание подклассам? |
| | Abstract Factory | Как создавать целые **семейства** связанных объектов? |
| | Builder | Как по шагам собирать сложный объект? |
| | Prototype | Как быстро клонировать объекты? |
| **Структурные** | Adapter | Как **подогнать старый интерфейс** под новый? |
| | Bridge | Как разделить **абстракцию и реализацию**? |
| | Composite | Как представить **дерево объектов** единым интерфейсом? |
| | Decorator | Как добавлять поведение без наследования / правки класса? |
| | Facade | Как спрятать сложную подсистему за простым API? |
| | Flyweight | Как экономить память за счёт **общего состояния**? |
| | Proxy | Как добавить контроль доступа / кеш / lazy‑loading? |
| **Поведенческие** | Chain of Responsibility | Как передавать запрос по цепочке обработчиков? |
| | Command | Как представить запрос объектом (отменять, логировать)? |
| | Iterator | Как правильно итерироваться по коллекциям? |
| | Mediator | Как снизить количество **прямых связей** между объектами? |
| | Memento | Как сохранять и откатывать состояние объекта? |
| | Observer | Как уведомлять подписчиков об изменениях? |
| | State | Как менять поведение объекта в зависимости от состояния? |
| | Strategy | Как подменять алгоритм «на лету»? |
| | Template Method | Как задать скелет алгоритма, а детали — в подклассах? |
| | Visitor | Как добавлять операции над деревом классов, не меняя классы? |
| **Pythonic‑паттерны** | Context manager | Управление ресурсами через `with`. |
| | Iterator / Generator | Ленивая работа с последовательностями. |
| | Decorator (функции) | Обёртки: кеширование, логирование, валидация. |
| | Descriptor | Поведение атрибутов (property и не только). |
| | Monkey patching | Замена реализации «на лету» (осторожно). |
| | EAFP vs LBYL | Стиль «сначала делай, потом извиняйся» вместо «проверяй всё заранее». |

## 3. Как изучать эту часть

Рекомендуемый маршрут (соответствует §66–69 глобального плана):

1. **Создающие паттерны (§66)**  
   Понять, как управлять **жизненным циклом объектов** и изолировать логику создания.
2. **Структурные паттерны (§67)**  
   Научиться строить **гибкие связи** между частями системы.
3. **Поведенческие паттерны (§68)**  
   Организовать **сложное поведение** и потоки событий в системе.
4. **Pythonic‑паттерны (§69)**  
   Увидеть, как многие «классические» паттерны **выглядят по‑питоничному**.

Практическая рекомендация:

- не пытаться «натянуть» паттерн на каждое место;
- идти от задачи: «у меня вот такая боль», — и узнавать, **какой паттерн это уже решает**.

## 4. Создающие паттерны (§66)
