### 8.1. Не начинать с паттерна

Плохой подход:

- «хочу применить Visitor» → ищу, куда бы его воткнуть.

Лучше:

1. Сформулировать **конкретную боль**:
   - «если добавляется новый формат отчёта, мне приходится править кучу `if/elif`» → это намёк на Strategy / Factory.
   - «у меня по коду размазаны `try/finally`» → контекст‑менеджеры / Facade.
2. Узнать, **есть ли уже имя и шаблон** для такой ситуации.

### 8.2. Python упрощает многие паттерны

За счёт:

- функций первого класса (Strategy, Command);
- генераторов (Iterator);
- `with` (Resource management);
- декораторов (Decorator / Proxy);
- дескрипторов (часть из паттернов GoF реализуется через них).

Задача — **не механически переписывать Java‑реализации**, а использовать **встроенные возможности языка**.

### 8.3. Когда паттерн — оверинжениринг

Признаки:

- куча мелких классов ради пары строк логики;
- непонятно, почему код сложнее простого `if/else`;
- команда не понимает, что за паттерн используется, и не выигрывает от этого.

Правило:

- если **простое решение** (2–3 функции) решает задачу — начните с него;
- паттерн имеет смысл, когда **сложность уже появилась** (много вариантов, сильные связи, неудобство расширения).

## 10. Дополнительные архитектурные и enterprise‑паттерны

В рамках глобального плана Часть XIII фокус был на GoF‑паттернах и pythonic‑паттернах.  
Ниже — краткий обзор более «крупных» архитектурных и enterprise‑паттернов, а также паттернов конкурентности/надёжности. Здесь цель — **дать карту местности**: понимать, что это за слова, где они применяются, и как они соотносятся с уже изученными паттернами.

### 10.1. Архитектурные паттерны (на уровне приложения)

#### 10.1.1. Layered Architecture (многослойная архитектура)

- Идея: разделить систему на **слои**, каждый из которых:
  - решает свою задачу;
  - общается только с соседними слоями.
- Классическая схема:
  - **Presentation (UI)** — веб‑слой, API‑эндпоинты, CLI;
  - **Application / Service** — сценарии использования, координация операций;
  - **Domain / Model** — бизнес‑правила, сущности, неизменяемые инварианты;
  - **Infrastructure** — БД, очереди, внешние сервисы.

В Python часто реализуется:

- через разнесение модулей/пакетов (`api/`, `services/`, `domain/`, `infra/`);
- с использованием уже знакомых паттернов:
  - `Service Layer`, `Repository`, `Unit of Work`, `Domain Event`.

Главная цель — **отделить бизнес‑логику от деталей инфраструктуры**.

#### 10.1.2. Hexagonal Architecture (Ports & Adapters)

- Известна как:
  - Hexagonal Architecture;
  - Ports & Adapters.
- Основная идея:
  - ядро системы (domain + application) не должно знать о конкретных технологиях ввода/вывода;
  - всё внешнее (БД, HTTP, UI, очереди) — это **адаптеры** к абстрактным «портам».

Термины:

- **Port (порт)** — абстракция взаимодействия:
  - входные порты: сценарии использования (команды/запросы);
  - выходные порты: интерфейсы для БД, внешних сервисов и т.д.
- **Adapter (адаптер)** — конкретная реализация порта:
  - REST‑адаптер, CLI‑адаптер, gRPC‑адаптер;
  - репозиторий для Postgres, для MongoDB и т.д.

Связь с паттернами:

- входные порты часто реализуются через **Command**, **Service Layer**;
- выходные — через **Repository**, **Adapter**, **Abstract Factory**.

#### 10.1.3. Clean Architecture

Clean Architecture развивает идеи слоёв/портов и адаптеров:

- слои располагаются **концентрическими кругами**:
  - в центре — Entities (домен);
  - вокруг — Use Cases;
  - дальше — Interface Adapters;
  - снаружи — Frameworks & Drivers (Django, Flask, БД, брокеры сообщений).
- Все зависимости направлены **внутрь**:
  - inner слои не знают о внешних;
  - outer слои зависят от абстракций внутреннего круга (DIP).

Для Python:

- полезно думать не столько о «строгих кругах», сколько о:
  - разделении **чистой доменной логики** (не завязанной на фреймворки);
  - изоляции инфраструктуры (БД, HTTP) в отдельных пакетах.

#### 10.1.4. MVC/MVP/MVVM (слои представления)

- Эти паттерны относятся в основном к **слою UI**.

Коротко:

- **MVC (Model–View–Controller)**:
  - Model — данные/правила;
  - View — отображение;
  - Controller — обработка ввода, координация.
- **MVP (Model–View–Presenter)**:
  - Presenter знает и о View, и о Model, но View «глупый»;
  - удобно тестировать Presenter отдельно.
- **MVVM (Model–View–ViewModel)**:
  - ViewModel предоставляет данные и команды для View;
  - View связывается с ViewModel через data binding.

В Python:

- в чистом виде чаще используются в десктоп/GUI‑фреймворках;
- в веб‑фреймворках часто микс:
  - Django «views» выполняют роль контроллеров;
  - сериализаторы/формы выступают ролью ViewModel/Presenter.

Главная идея — **отделить представление от логики и модели**.

### 10.2. Enterprise‑паттерны (бизнес‑логика и данные)

Эти паттерны подробно описаны в «Patterns of Enterprise Application Architecture» (Martin Fowler). В Python они появляются почти в любом серьёзном backend‑проекте.

#### 10.2.1. Repository

- Абстракция над источником данных (БД, API, файл):
  - предоставляет методы вроде `add`, `get`, `list`, `remove`;
  - скрывает детали SQL/ORM/HTTP.
- Связь с GoF:
  - часто используется вместе с **Factory Method**, **Abstract Factory**, **Unit of Work**.

В Python:

- обычно реализуется как класс, принимающий `Session`/`Client` в конструктор;
- интерфейс репозитория — хороший кандидат на Protocol/ABC (для тестов и подмены реализации).

#### 10.2.2. Unit of Work

- Координатор изменений в бизнес‑объектах и общем коммите в хранилище:
  - отслеживает новые/изменённые/удалённые сущности;
  - применяет все изменения **как единую транзакцию**.

Типичный интерфейс:

```python
class UnitOfWork(ABC):
    users: UserRepository
    orders: OrderRepository

    @abstractmethod
    def __enter__(self) -> "UnitOfWork": ...

    @abstractmethod
    def __exit__(self, exc_type, exc, tb) -> None: ...

    @abstractmethod
    def commit(self) -> None: ...

    @abstractmethod
    def rollback(self) -> None: ...
```

Использование:

```python
def place_order(uow: UnitOfWork, user_id: int, items: list[int]) -> None:
    with uow:
        user = uow.users.get(user_id)
        order = user.create_order(items)
        uow.orders.add(order)
        uow.commit()
```

Здесь **менеджер контекста** и Unit of Work тесно связаны.

#### 10.2.3. Service Layer

- Отдельный слой «сценариев использования»:
  - собирает together вызовы репозиториев, доменных методов, внешних сервисов;
  - реализует команды вроде «оформить заказ», «зарегистрировать пользователя».
- Позволяет:
  - держать web/API слой тонким (контроллеры просто вызывают сервисы);
  - легко писать юнит‑тесты для сценариев бизнеса без HTTP и БД.

#### 10.2.4. Specification

- Описывает **критерий (условие)** как объект:
  - можно комбинировать спеки `And`, `Or`, `Not`;
  - можно использовать одну и ту же спецификацию:
    - для фильтрации в памяти;
    - и для генерации WHERE‑условий в SQL.

В Python:

- часто реализуется через классы/функции/лямбды;
- можно оформить как `Protocol` с методом `is_satisfied_by(entity) -> bool` и/или методом генерации выражения для ORM.

#### 10.2.5. Domain Event

- Представляет **важное событие домена**:
  - `OrderPlaced`, `UserRegistered`, `PaymentFailed`.
- Служит для:
  - асинхронной реакции (отправка email, обновление проекций, интеграции);
  - отслеживания истории изменений (audit log).

В Python:

- можно реализовать простыми dataclass’ами + шиной событий:
  - синхронной (список обработчиков);
  - асинхронной (очередь сообщений).

#### 10.2.6. Event Sourcing, CQRS, Saga (очень кратко)

- **Event Sourcing**:
  - состояние системы = сумма **событий**, а не текущая запись в БД;
  - для получения состояния сущности нужно «применить» все её события.
- **CQRS (Command Query Responsibility Segregation)**:
  - разделение моделей чтения и записи;
  - команды меняют состояние, запросы читают его через оптимизированные проекции.
- **Saga**:
  - паттерн координации **распределённых транзакций** через последовательность локальных транзакций и компенсационных действий.

Это крупные темы, которые обычно обсуждаются в контексте распределённых систем, но полезно знать их названия и общую идею.

### 10.3. Паттерны конкурентности и отказоустойчивости

Здесь — несколько часто встречающихся идей, особенно в микросервисах и асинхронных системах.

#### 10.3.1. Producer–Consumer

- Производители кладут задачи/сообщения в очередь;
- потребители забирают и обрабатывают.

В Python:

- `queue.Queue` + потоки (`threading.Thread`);
- `asyncio.Queue` + корутины;
- брокеры сообщений (RabbitMQ, Kafka) как внешняя реализация очередей.

#### 10.3.2. Reactor

- Сердце многих event‑loop систем:
  - регистрирует интерес к событиям (I/O ready, таймеры);
  - в цикле опрашивает/ждёт события и диспатчит их обработчикам.

В Python:

- `asyncio` (event loop) реализует по сути Reactor;
- низкоуровневые части — `selectors`, `select`, `epoll`, `kqueue`.

#### 10.3.3. Actor

- Модель конкурентности, где:
  - **акторы** — объекты с собственным состоянием и входной очередью сообщений;
  - общение идёт через **асинхронную отправку сообщений**, без разделяемой памяти.

В Python:

- библиотеки вроде `thespian`, `pykka` реализуют actor‑модель;
- можно набросать упрощённый actor‑runtime на базе `asyncio` + очередей.

#### 10.3.4. Circuit Breaker и Bulkhead

- **Circuit Breaker**:
  - если внешний сервис часто падает/тормозит, «автоматический выключатель» временно блокирует запросы к нему;
  - предотвращает каскадные отказы.

- **Bulkhead (перегородки)**:
  - изоляция ресурсов (thread pool’ы, отдельные лимиты) для разных подсистем;
  - отказ/перегрузка одной части не «заливает» всё приложение.

В Python:

- реализуются библиотеками (`pybreaker`, собственные обёртки):
  - вокруг клиентов БД, HTTP‑клиентов, брокеров сообщений.

## 11. Вопросы для самопроверки по Части XIII

Ниже — по 2–3 вопроса для каждого ключевого паттерна/блока. Ответы спрятаны под спойлерами.

### 11.1. Создающие паттерны

#### 11.1.1. Singleton

1. В чём разница между модулем как Singleton и классическим Singleton‑классом? Какие преимущества даёт модуль в Python?

<details><summary>Показать ответ</summary>

Модуль загружается один раз и затем переиспользуется всеми импортами, поэтому его состояние (`config.DEBUG` и т.п.) уже общее — это «естественный» Singleton. При этом не нужно писать метаклассы/`__new__`, код проще и прозрачнее. Классический Singleton‑класс требует явной логики контроля экземпляра (в т.ч. потокобезопасности) и чаще создаёт скрытое глобальное состояние, которое сложнее тестировать и подменять.

</details>

2. Почему Singleton создаёт проблемы для тестирования и внедрения зависимостей? Как это связано с DIP?

<details><summary>Показать ответ</summary>

Потому что зависимости начинают «подтягиваться» из глобального состояния (Singleton) внутри классов, а не передаваться явно через конструктор. Это нарушает DIP: высокоуровневый код зависит от конкретной реализации (глобального синглтона), а не от абстракции. В тестах сложно подменить такую зависимость, приходится менять глобал или monkey‑patchить модуль.

</details>

#### 11.1.2. Factory Method

1. Чем Factory Method отличается от простого `@classmethod`‑фабрики вроде `User.from_row`?

<details><summary>Показать ответ</summary>

`User.from_row` — фабричный метод на уровне одного класса: он всегда создаёт один и тот же тип (`User`). Factory Method в GoF‑смысле — абстрактный метод в базовом классе, который подклассы реализуют по‑разному, чтобы создавать разные конкретные типы («создание через наследование»). Оба — фабрики, но Factory Method работает на уровне иерархии и полиморфизма.

</details>

2. Почему в Python часто выгоднее передать «фабрику» (класс/функцию) как зависимость, чем вводить отдельную иерархию фабрик?

<details><summary>Показать ответ</summary>

Потому что функции и классы — объекты первого класса: можно просто передать `button_factory=WindowsButton` или `client_factory=make_postgres_client`. Это даёт ту же гибкость выбора реализации, но без лишней иерархии ABC/подклассов. Иерархия фабрик оправдана, когда у вас сложный базовый класс с Template Method и много разных вариантов создания.

</details>

#### 11.1.3. Abstract Factory

1. В чём ключевая идея Abstract Factory по сравнению с Factory Method?

<details><summary>Показать ответ</summary>

Factory Method отвечает за создание **одного** продукта в иерархии (перегружаемый метод базового класса). Abstract Factory — это отдельный объект, который предоставляет интерфейс для создания **семейства** связанных продуктов (кнопка, чекбокс, окно и т.п.), гарантируя, что эти продукты сочетаются друг с другом (одно «семейство»). То есть Abstract Factory → «набор фабрик для связанных объектов».

</details>

2. Как Abstract Factory помогает переключать окружения (например, Postgres vs InMemory) в приложении?

<details><summary>Показать ответ</summary>

Вы создаёте интерфейс фабрики (`RepositoryFactory`) и две реализации: `PostgresRepoFactory` и `InMemoryRepoFactory`. Сервис в конструкторе получает объект `RepositoryFactory` и использует его для создания репозиториев. На продакшене вы передаёте `PostgresRepoFactory`, в тестах — `InMemoryRepoFactory`. Бизнес‑код при этом остаётся неизменным и зависит только от абстракции фабрики и репозиториев.

</details>

#### 11.1.4. Builder

1. В каких ситуациях Builder даёт реальные преимущества перед конструктором с keyword‑аргументами?

<details><summary>Показать ответ</summary>

Когда объект:
— строится в несколько шагов, часто в разных местах кода;  
— имеет сложные инварианты (некоторые комбинации параметров недопустимы) и их удобно проверять в одном месте (`build()`);  
— требует удобного DSL‑подобного интерфейса (`for_user(...).between(...).with_charts()`).  
Простой конструктор с kwargs не даёт поэтапной сборки и централизованной проверки перед «финализацией».

</details>

2. Почему нежелательно, чтобы Builder хранил «полуготовый» объект продукта и модифицировал его напрямую?

<details><summary>Показать ответ</summary>

Потому что такой продукт может «утечь» в остальной код в неконсистентном состоянии (часть полей не инициализирована). Это превращает Builder в набор сеттеров над мутабельным объектом и усложняет гарантии. Чище хранить в Builder только данные конфигурации и создавать новый, полностью инициализированный продукт в `build()`.

</details>

#### 11.1.5. Prototype

1. В чём разница между поверхностной (`copy.copy`) и глубокой (`copy.deepcopy`) копией в контексте Prototype?

<details><summary>Показать ответ</summary>

Поверхностная копия создаёт новый объект, но оставляет ссылки на те же вложенные объекты (списки, словари и т.п.). Изменение вложенных структур будет видно и в прототипе, и в копии. Глубокая копия рекурсивно копирует и вложенные объекты, делая прототип и клон полностью независимыми. В Prototype важно явно решить, какие части состояния должны быть общими, а какие — независимыми.

</details>

2. Зачем может понадобиться реестр прототипов (Prototype Registry)?

<details><summary>Показать ответ</summary>

Реестр прототипов хранит «шаблоны» объектов (конфигурации, типовые сущности) по ключу и позволяет быстро получать новые экземпляры через клонирование с возможностью лёгких модификаций (`clone("default_user", role="admin")`). Это удобно для шаблонов конфигурации, GUI‑элементов, игровых сущностей и т.п., когда нужно множество похожих объектов на основе ограниченного набора эталонов.

</details>

### 11.2. Структурные паттерны

#### 11.2.1. Adapter

1. Как отличить задачу для Adapter от задачи для Facade?

<details><summary>Показать ответ</summary>

Adapter нужен, когда есть **чужой объект с «неправильным» интерфейсом**, и его нужно подогнать под уже существующий ожидаемый интерфейс (совместимость API). Facade нужен, когда внутри есть много мелких компонентов/сложная подсистема, и вы хотите дать **упрощённый вход** — новый удобный интерфейс для клиентов, без акцента на совместимости с конкретным существующим API.

</details>

2. Почему в Python чаще используют объектный Adapter (через композицию), а не классовый (через множественное наследование)?

<details><summary>Показать ответ</summary>

Потому что множественное наследование в Python делает адаптер тем же типом, что и адаптируемый класс, и создаёт жёсткую связь и сложности с MRO. Композиция (`self._wrapped = adaptee`) даёт более слабую связь: адаптер реализует целевой интерфейс, но может адаптировать любой объект, не вмешиваясь в его иерархию наследования, и легче тестируется/подменяется.

</details>

#### 11.2.2. Bridge

1. В чём основная мотивация для использования Bridge вместо простой иерархии наследования?

<details><summary>Показать ответ</summary>

Когда у вас есть **два измерения изменчивости** (например, разные фигуры и разные API отрисовки), наследование по всем комбинациям приводит к взрыву классов (`CircleOpenGL`, `CircleCairo`, `SquareOpenGL`, и т.д.). Bridge разделяет иерархии «фигур» и «API» и соединяет их композицией (`Shape` держит `DrawingAPI`), позволяя независимо добавлять новые фигуры и новые реализации.

</details>

2. Чем Bridge отличается от Strategy, если оба используют композицию?

<details><summary>Показать ответ</summary>

Strategy фокусируется на **подмене алгоритма** (узкого поведения) в рантайме (`DiscountStrategy`, `SortStrategy`) и обычно описывает одну операцию. Bridge — **структурный приём разделения иерархий** абстракций и реализаций; он проектируется заранее, чтобы избежать комбинационного наследования, и часто затрагивает целые семейства методов (например, весь интерфейс `DrawingAPI`, а не одну функцию).

</details>

#### 11.2.3. Composite

1. Как Composite помогает работать с деревом объектов так же, как с одиночным объектом?

<details><summary>Показать ответ</summary>

Composite вводит общий интерфейс `Component` для листьев и узлов‑контейнеров. Клиент работает с `Component` и может вызвать, например, `render()` как на `Text` (лист), так и на `Container` (узел) — последний делегирует вызов детям. Это позволяет писать универсальный код, который не различает одиночный элемент и дерево элементов.

</details>

2. Приведи пример, где Composite уместен в Python‑коде.

<details><summary>Показать ответ</summary>

Типичные примеры: древовидные меню/навигация в UI, представление файловой системы (`File`/`Directory`), дерево синтаксиса (AST) для парсера/интерпретатора, иерархия виджетов в GUI‑фреймворке. Во всех случаях Composite позволяет отобразить структуру «часть‑целое» и работать с поддеревьями как с единым компонентом.

</details>

#### 11.2.4. Decorator (структурный)

1. В чём разница между структурным Decorator и Python‑декораторами функций?

<details><summary>Показать ответ</summary>

Структурный Decorator — объект‑обёртка, реализующий тот же интерфейс, что и «оборачиваемый» объект, и добавляющий поведение (логирование, кеш, безопасность). Python‑декоратор функций — синтаксический сахар над функцией, которая принимает функцию и возвращает обёртку. Идея одна (обёртка), но в структурном случае — уровень объектов/классов, в Python‑декораторах — уровень функций/методов.

</details>

2. Почему Decorator предпочтительнее наследования, когда нужно добавить к классу одно‑два дополнительных поведения?

<details><summary>Показать ответ</summary>

Наследование фиксирует поведение навсегда и плохо масштабируется при комбинации фич (нужно создавать отдельный подкласс под каждую комбинацию). Decorator позволяет «сложить» несколько обёрток вокруг одного объекта динамически (логирование + ретраи + метрики) без изменения исходного класса и без взрыва количества подклассов.

</details>

#### 11.2.5. Facade

1. Как понять, что нужна Facade, а не просто несколько утилитных функций?

<details><summary>Показать ответ</summary>

Если внешний код постоянно должен выполнять **сложную последовательность шагов** (создать X, вызвать Y, обработать Z, учесть ошибки) для типичного сценария работы с подсистемой — это сигнал к введению Facade. Он инкапсулирует эту последовательность в одном понятном методе (`login_and_pay`, `sync_user`), скрывая детали реализации. Набор утилитных функций не предлагает единой, устойчивой точки входа.

</details>

2. Может ли Facade внутри себя использовать Adapter, Repository, Builder и другие паттерны?

<details><summary>Показать ответ</summary>

Да, Facade — это уровень **оркестрации**: он вполне может использовать адаптеры внешних API, репозитории, билдеры запросов, кэш и т.п. Паттерны нижних уровней скрываются за фасадом, а клиент видит только простой высокоуровневый интерфейс. Это типичная комбинация в реальных системах.

</details>

#### 11.2.6. Flyweight

1. Как Flyweight экономит память и за счёт чего это достигается?

<details><summary>Показать ответ</summary>

Flyweight разделяет состояние объекта на **intrinsic** (общее, разделяемое многими экземплярами) и **extrinsic** (уникальное, передаваемое извне при вызове). Общая часть хранится в одном объекте (или в кеширующей фабрике), а уникальные параметры передаются как аргументы методов (`draw(x, y)`). Это уменьшает количество реальных объектов и расход памяти.

</details>

2. Приведи пример, где в Python можно увидеть идею Flyweight.

<details><summary>Показать ответ</summary>

Примеры: кеширование глифов текста (один объект `Glyph('A')` для множества символов A в документе), интернирование строк (`sys.intern`), кеши объектов в ORM, кеширующие фабрики (`lru_cache` для объектов с одинаковой конфигурацией). Во всех случаях общее состояние переиспользуется, а меняются только внешние параметры.

</details>

#### 11.2.7. Proxy

1. Какие виды Proxy ты знаешь и чем они отличаются по целям?

<details><summary>Показать ответ</summary>

Типичные виды:  
— виртуальный / lazy proxy: откладывает создание реального объекта до первого обращения;  
— protection proxy: проверяет права доступа перед делегированием;  
— remote proxy: представляет объект в другом процессе/на другом сервере;  
— cache proxy: кеширует результаты вызовов.  
Все реализуют тот же интерфейс, что и реальный объект, но добавляют дополнительный контроль или оптимизацию.

</details>

2. Чем Proxy отличается от Decorator в практическом коде?

<details><summary>Показать ответ</summary>

Граница размыта, но обычно: Decorator акцентирован на **добавлении поведения** (логирование, метрики, ретраи) к уже удобному интерфейсу. Proxy акцентирован на **контроле доступа и управлении жизненным циклом/ресурсами** (ленивая инициализация, сетевые вызовы, кеширование, права). С точки зрения реализации — оба обёртки, но мотивация немного различается.

</details>

### 11.3. Поведенческие паттерны

#### 11.3.1. Chain of Responsibility

1. Как цепочка обязанностей помогает убрать жёсткий `if/elif` по типу запроса?

<details><summary>Показать ответ</summary>

Вместо одного метода с большим `if/elif`, который решает, кто обрабатывает запрос, вы создаёте набор обработчиков, каждый отвечает за свой случай (`AuthHandler`, `LoggingHandler`, `ValidationHandler`) и передаёт управление дальше, если не может обработать. Добавление нового обработчика не требует менять существующий код, достаточно вставить новый элемент в цепочку.

</details>

2. Когда Chain of Responsibility может быть избыточным усложнением?

<details><summary>Показать ответ</summary>

Когда возможных сценариев мало и они редко меняются — простой `if/elif` читается легче и понятнее. Цепочка даёт выигрыш, когда типов запросов/обработчиков много, они часто меняются/расширяются, а точки включения/отключения отдельных обработчиков должны быть гибкими (например, middleware в web‑фреймворке).

</details>

#### 11.3.2. Command

1. Какие преимущества даёт представление операции как объекта Command?

<details><summary>Показать ответ</summary>

Можно:  
— сохранять историю команд (для undo/redo);  
— ставить команды в очередь или выполнять отложенно;  
— логировать, ретраить, распределять между воркерами;  
— передавать команды между слоями/процессами как данные.  
Команда инкапсулирует всё, что нужно для выполнения операции (получатель + параметры), и её можно обрабатывать как обычный объект.

</details>

2. Как в Python можно реализовать «команду» без явного класса Command?

<details><summary>Показать ответ</summary>

Использовать функцию или любой вызываемый объект (`callable`) как команду:  
— хранить в списке/очереди ссылки на функции и их аргументы;  
— оформлять команды как dataclass с методом `__call__`;  
— использовать замыкания, захватывающие нужный контекст.  
Это даёт те же свойства (отложенное выполнение, очередь), но с более лёгким синтаксисом.

</details>

#### 11.3.3. Iterator

1. Почему в Python паттерн Iterator во многом «встроен в язык»?

<details><summary>Показать ответ</summary>

Потому что поддержка протокола итерации (`__iter__`, `__next__`), генераторов (`yield`), comprehensions, встроенных функций (`iter`, `next`, `sum`, `any`, `all`) реализует почти всю механику паттерна Iterator на уровне языка. Вам достаточно реализовать `__iter__` (или генераторную функцию), и объект автоматически интегрируется со всем итерационным API.

</details>

2. В чём преимущество генераторов по сравнению с реализацией итератора «вручную» через класс с `__next__`?

<details><summary>Показать ответ</summary>

Генераторы позволяют писать логику итерации в виде обычной функции с `yield`, без явного хранения состояния в полях и реализации `__next__`. Это короче, понятнее и менее подвержено ошибкам (не нужно явно бросать `StopIteration`, отслеживать индексы и т.п.), а Python сам создаёт объект‑итератор.

</details>

#### 11.3.4. Mediator

1. Как Mediator уменьшает связность между объектами?

<details><summary>Показать ответ</summary>

Вместо того, чтобы объекты ссылались друг на друга напрямую и вызывали методы друг друга, они взаимодействуют через посредника (Mediator). Например, `User` отправляет сообщение не напрямую другому пользователю, а в `ChatRoom`, который уже знает обо всех пользователях и решает, кому что доставлять. Это уменьшает число прямых зависимостей и делает логику взаимодействия централизованной и более управляемой.

</details>

2. Приведи пример, где Mediator может быть полезен в серверном Python‑коде.

<details><summary>Показать ответ</summary>

Например, в системе обработки команд/событий: `CommandBus` или `EventBus` играют роль Mediator между отправителями и обработчиками команд/событий. Отправитель публикует команду/событие на «шину», шина находит нужные обработчики и вызывает их — отправителю не нужно знать о конкретных обработчиках и их расположении.

</details>

#### 11.3.5. Memento

1. Почему Memento полезен, когда нужно реализовать undo/redo?

<details><summary>Показать ответ</summary>

Потому что он инкапсулирует состояние объекта в отдельном снимке (memento), не раскрывая его внутренней структуры наружу. `Originator` (объект) знает, как создать memento и восстановиться из него, а внешний код оперирует только абстрактными снимками (стеком истории). Это позволяет хранить историю состояний и откатывать изменения безопасно.

</details>

2. В чём отличие Memento от простого «сохранить dict с атрибутами объекта»?

<details><summary>Показать ответ</summary>

Сохранение `obj.__dict__` подходит только для простых случаев и «пробивает» инкапсуляцию. Memento даёт объект с чётко определённым контрактом и структурой, может включать только нужные поля, учитывать инварианты, версии формата, совместимость и т.п. Кроме того, `__dict__` не всегда отражает полное состояние (свойства, скрытые поля, внешние ресурсы).

</details>

#### 11.3.6. Observer

1. В чём преимущество Observer по сравнению с прямым вызовом обработчиков из издателя?

<details><summary>Показать ответ</summary>

Observer отделяет издателя (subject) от подписчиков (observers). Издатель не знает, кто именно подписался, он просто вызывает `_notify`. Подписчики могут добавляться/удаляться динамически, а один и тот же издатель может иметь разных наблюдателей в разных конфигурациях (логирование, отправка email, обновление UI) без изменения своего кода.

</details>

2. Как реализовать простую событийную шину (event bus) на основе Observer в Python?

<details><summary>Показать ответ</summary>

Можно создать класс `EventBus` с методами `subscribe(event_type, handler)` и `publish(event)`, где `event_type` определяется по классу/строке. Внутри хранить словарь `event_type -> list[handler]` и при публикации события вызывать всех подписчиков. Это по сути реализация Observer c несколькими типами событий.

</details>

#### 11.3.7. State

1. Как паттерн State помогает избавиться от множества `if status == ...` в коде?

<details><summary>Показать ответ</summary>

State выносит поведение для каждого состояния в отдельный класс (`ConcreteStateA`, `ConcreteStateB`), а контекст (`Context`) делегирует ему обработку запросов. Вместо большого `if` по статусу вызывается `self.state.handle(self)`, и каждое состояние само решает, что делать и в какое следующее состояние перейти. При добавлении нового состояния не нужно трогать старый `if`, достаточно добавить новый класс.

</details>

2. Когда проще использовать словарь «состояние → функция», а не полноценный State?

<details><summary>Показать ответ</summary>

Когда состояний немного, логика в каждом состоянии простая и не требует собственного внутреннего состояния или сложной композиции. Тогда достаточно словаря `{state: handler_function}` и простого диспетчера. Полноценный State имеет смысл, когда каждое состояние содержит значимое поведение и/или собственное состояние, и количество состояний/переходов растёт.

</details>

#### 11.3.8. Strategy

1. В чём плюс Strategy по сравнению с `if/elif` по режиму работы функции?

<details><summary>Показать ответ</summary>

Strategy выносит разные алгоритмы в отдельные объекты/функции, а контекст (`Stats`, `Sorter`, `DiscountCalculator`) зависит только от абстракции стратегии. Это упрощает добавление новых алгоритмов (без изменения контекста), позволяет подменять стратегию в рантайме, тестировать каждый алгоритм отдельно и комбинировать стратегии.

</details>

2. Почему в Python Strategy часто реализуется просто передачей функции/лямбды?

<details><summary>Показать ответ</summary>

Потому что функции — объекты первого класса: их можно передавать как аргументы, хранить в структурах данных, возвращать из других функций. Вместо класса с методом `apply()` можно передать `sum`, `max`, `lambda xs: sorted(xs)` и т.п., получая ту же гибкость и полиморфизм с гораздо меньшим количеством кода.

</details>

#### 11.3.9. Template Method

1. Когда Template Method предпочтительнее Strategy?

<details><summary>Показать ответ</summary>

Когда нужен **фиксированный «скелет» алгоритма**, а вариации затрагивают только отдельные его шаги. Template Method реализует последовательность шагов в базовом классе (`process()`), а подклассы переопределяют только отдельные методы (`load`, `transform`, `save`). Strategy обычно заменяет целиком один из шагов или всю логику, не диктуя порядок шагов.

</details>

2. Как Template Method можно выразить в Python без наследования?

<details><summary>Показать ответ</summary>

Через функции/композицию: например, функция `process(load, transform, save)` принимает три вызываемых объекта и внутри вызывает их в нужной последовательности. Тогда разные реализации `load/transform/save` можно передавать как стратегии без иерархии наследования. Это более «функциональный» взгляд на Template Method.

</details>

#### 11.3.10. Interpreter

1. Какой тип задач подсказывает, что можно применить паттерн Interpreter?

<details><summary>Показать ответ</summary>

Наличие **простого мини‑языка** (формулы, логические выражения, фильтры, правила), который нужно безопасно парсить и выполнять. Особенно когда нельзя использовать произвольный Python (`eval`) и хочется иметь явную модель языка (AST) и контролируемую семантику.

</details>

2. Чем подход «классы с методом interpret» отличается от подхода «dataclass’ы + одна функция eval_expr»?

<details><summary>Показать ответ</summary>

В классовом подходе логика интерпретации распределена по методам узлов (`node.interpret()`), каждый узел знает, как себя вычислить. В AST + `eval_expr` узлы — простые контейнеры данных, а вся логика находится в одной (или нескольких) функциях, которые делают `isinstance`‑диспетчеризацию. Для Python второй подход часто проще и лучше сочетается с типизацией/Visitor’ами.

</details>

#### 11.3.11. Visitor

1. В чём основное преимущество Visitor при работе с большой иерархией узлов (например, AST)?

<details><summary>Показать ответ</summary>

Visitor позволяет **добавлять новые операции над уже существующей иерархией** без изменения самих узлов: вы пишете новый Visitor (`EvalVisitor`, `PrintVisitor`, `OptimizeVisitor`), а узлы лишь реализуют метод `accept(self, visitor)`. Это особенно удобно, когда структура узлов стабильна, а операций над ними становится всё больше.

</details>

2. В чём недостаток Visitor с точки зрения расширения самой иерархии узлов?

<details><summary>Показать ответ</summary>

Если добавить новый тип узла (`Subtract`, `Multiply`), нужно изменить **все Visitor’ы**, добавив в них метод `visit_subtract`/`visit_multiply`. То есть Visitor облегчает добавление новых операций, но усложняет добавление новых типов узлов. Это компромисс: удобно, когда иерархия узлов редко меняется, а операций много.

</details>

### 11.4. Pythonic‑паттерны

#### 11.4.1. Context Manager

1. Почему менеджеры контекста считаются «питоничным» вариантом паттерна Resource Management/RAII?

<details><summary>Показать ответ</summary>

Они инкапсулируют логику получения и освобождения ресурса в `__enter__`/`__exit__`, гарантируя освобождение даже при исключениях (`with open(...) as f`). Это эквивалентно RAII в C++ (ресурс привязан к области видимости), но выражено через явный синтаксис `with` и протокол контекста, а не только через деструкторы.

</details>

2. Какие ресурсы кроме файлов логично оборачивать в менеджеры контекста?

<details><summary>Показать ответ</summary>

Подключения к БД, транзакции, locks (`threading.Lock`), сетевые соединения, временные изменения конфигурации/окружения, профилировщики, временное изменение лог‑уровня, открытые сокеты и любые объекты, требующие явного закрытия/освобождения или отката.

</details>

#### 11.4.2. Generators / Iterator

1. Какие преимущества дают генераторы при обработке больших объёмов данных?

<details><summary>Показать ответ</summary>

Они позволяют обрабатывать данные **лениво**, по одному элементу за раз, не загружая весь набор в память. Это особенно важно для больших файлов, потоков данных, бесконечных или очень длинных последовательностей. Генераторы также позволяют комбинировать операции (filter/map/reduce) без промежуточных списков.

</details>

2. Чем генераторные выражения отличаются от list comprehension?

<details><summary>Показать ответ</summary>

List comprehension сразу создаёт список в памяти, генераторное выражение возвращает ленивый генератор, который вычисляет элементы по мере запроса. `[...]` → список, `(...)` → генератор. В первом случае сразу расходуется память на все элементы, во втором — только на итератор и текущий элемент.

</details>

#### 11.4.3. Decorators (функции)

1. В чём основная польза функциональных декораторов в Python?

<details><summary>Показать ответ</summary>

Они позволяют **повторно использовать обёртки** вокруг функций/методов: логирование, измерение времени, кеширование, ретраи, проверка прав, валидация аргументов. Вместо дублирования кода в каждой функции вы один раз пишете декоратор и применяете его с помощью `@decorator` к множеству функций.

</details>

2. Почему важно использовать `functools.wraps` при написании декоратора?

<details><summary>Показать ответ</summary>

`wraps` копирует метаданные исходной функции (`__name__`, `__doc__`, `__module__`, `__annotations__`, `__wrapped__`) в обёртку. Без него декорированная функция «теряет лицо» (имя, докстринг, сигнатуру), из-за чего страдают отладка, документация, introspection и инструменты типа `inspect`, `help`, Sphinx.

</details>

#### 11.4.4. Descriptors

1. В чём сила дескрипторов по сравнению с простыми свойствами `property`?

<details><summary>Показать ответ</summary>

`property` — частный случай дескриптора для одного атрибута в одном классе. Явный дескриптор (`__get__`, `__set__`, `__set_name__`) можно переиспользовать **между многими классами и атрибутами**, реализуя общие политики доступа (валидация, lazy‑loading, логирование, вычисляемые поля) и избегая дублирования кода.

</details>

2. Почему дескрипторы активно используются в ORM/фреймворках?

<details><summary>Показать ответ</summary>

Потому что они позволяют перехватывать доступ к полям моделей (`user.name`, `post.author`) и подставлять туда динамическую логику: ленивую загрузку из БД, валидацию, преобразование типов, трекинг изменений и т.п. Объявление поля `name = StringField(...)` создаёт дескриптор, который управляет чтением/записью реального значения.

</details>

#### 11.4.5. EAFP vs LBYL

1. Почему EAFP‑подход предпочтительнее LBYL в многопоточной/асинхронной среде?

<details><summary>Показать ответ</summary>

При LBYL вы сначала проверяете условие, а потом выполняете действие; между проверкой и действием состояние может измениться (другой поток/корутина изменил объект/файл/запись). EAFP исходит из того, что вы просто выполняете действие и, если произошла ошибка, ловите исключение. Это устойчивее к гонкам, потому что не полагается на «свежесть» предварительной проверки.

</details>

2. Когда LBYL всё же оправдан в Python?

<details><summary>Показать ответ</summary>

Когда проверка дешёвая и исключение слишком тяжёлое/шумное (например, в горячем цикле), или когда нужно явно отличить разные исходы до действия (например, проверить флаг конфигурации перед запуском дорогой операции). Также LBYL уместен, когда вы проверяете статическое условие (тип, конфигурацию, права), которое не меняется между проверкой и действием.

</details>

### 11.5. Архитектурные, enterprise‑ и конкурентные паттерны

#### 11.5.1. Архитектурные (Layered, Hexagonal, Clean, MVC/MVP/MVVM)

1. В чём главное различие между «наивной» многослойной архитектурой и Hexagonal/Clean Architecture?

<details><summary>Показать ответ</summary>

В наивной многослойной архитектуре часто слои всё равно зависят от фреймворков и конкретных технологий (например, доменные модели импортируют ORM‑классы, репозитории тянут HTTP‑клиентов и т.д.). В Hexagonal/Clean чётко соблюдается **направление зависимостей внутрь**: домен и use case’ы ничего не знают о фреймворках/БД, а все внешние детали подключаются через абстракции (порты/интерфейсы) и адаптеры.

</details>

2. Зачем разделять codebase на `domain/`, `application/` (services), `infrastructure/` и `api/` в Python‑проекте?

<details><summary>Показать ответ</summary>

Чтобы:  
— изолировать бизнес‑логику от фреймворков и инфраструктуры;  
— облегчить тестирование домена в отрыве от БД/HTTP;  
— снизить связанность (смена ORM или фреймворка не должна переписывать домен);  
— сделать структуру проекта читаемой и предсказуемой для команды (где искать бизнес‑правила, где HTTP‑ручки, где интеграции).

</details>

#### 11.5.2. Enterprise‑паттерны (Repository, Unit of Work, Service Layer, Specification, Domain Event, Event Sourcing, CQRS, Saga)

1. Как Repository помогает реализовать DIP в доступе к данным?

<details><summary>Показать ответ</summary>

Repository предоставляет **абстрактный интерфейс** (Protocol/ABC), скрывающий конкретную технологию хранения (ORM, SQL, NoSQL, HTTP‑API). Бизнес‑код зависит от интерфейса `UserRepository`, а не от `sqlalchemy.Session` или `pymongo.Collection`. Конкретная реализация репозитория подставляется снаружи (DI), что соответствует принципу инверсии зависимостей.

</details>

2. В чём роль Unit of Work и как он сочетается с менеджером контекста?

<details><summary>Показать ответ</summary>

Unit of Work координирует изменения в нескольких репозиториях/агрегатах в рамках одной логической операции: отслеживает новые/изменённые/удалённые сущности и коммитит/откатывает всё одномоментно. Менеджер контекста (`with uow:`) гарантирует, что при исключении будет сделан откат (rollback), а при успешном завершении — `commit`. Это связывает бизнес‑операцию с транзакцией в хранилище.

</details>

3. Что даёт CQRS по сравнению с «одной моделью для чтения и записи»?

<details><summary>Показать ответ</summary>

CQRS разделяет модели чтения и записи: команда изменяет состояние через «write‑модель», а запрос читает данные из оптимизированной «read‑модели» (проекций). Это позволяет:  
— по‑разному масштабировать чтение и запись;  
— делать быстрые, специализированные представления для чтения;  
— убрать лишние зависимости между логикой изменений и логикой отображения/отчётов.  
Цена — усложнение архитектуры и необходимость синхронизации между write‑ и read‑моделями (часто через события).

</details>

#### 11.5.3. Конкурентность и надёжность (Producer–Consumer, Reactor, Actor, Circuit Breaker, Bulkhead)

1. Как паттерн Producer–Consumer помогает разгрузить основной поток исполнения?

<details><summary>Показать ответ</summary>

Производители кладут задачи в очередь, а потребители обрабатывают их асинхронно (в других потоках/процессах/корутинах). Основной поток может быстро реагировать на внешние события (HTTP‑запросы, сообщения), не блокируясь на долгой обработке. Это повышает отзывчивость и позволяет масштабировать количество потребителей по нагрузке.

</details>

2. В чём суть Circuit Breaker и как он защищает систему?

<details><summary>Показать ответ</summary>

Circuit Breaker отслеживает ошибки/таймауты при обращении к внешнему ресурсу. Если доля неудач становится слишком большой, он «размыкает цепь» и на время перестаёт делать реальные запросы (сразу возвращая ошибку илиfallback), периодически проверяя, восстановился ли сервис. Это предотвращает каскадные отказы и излишнюю нагрузку на уже «лежачий» сервис.

</details>

3. Когда Actor‑модель может быть удобнее классического многопоточного кода с lock’ами?

<details><summary>Показать ответ</summary>

Когда есть множество независимых сущностей с собственным состоянием, которые взаимодействуют через обмен сообщениями (например, пользователи в чате, игровые объекты, отдельные подсистемы). Actor‑модель убирает необходимость ручных lock’ов: каждый актор обрабатывает сообщения последовательно, гарантируя целостность своего состояния, а взаимодействие между ними идёт через очереди сообщений.

</details>
