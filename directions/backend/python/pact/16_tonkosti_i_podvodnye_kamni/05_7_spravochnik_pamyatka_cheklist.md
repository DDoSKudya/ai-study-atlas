### 7.1. Практический сценарий: «У меня проблема X — что делать?»

| Ситуация | Что проверить / что сделать |
|----------|-----------------------------|
| Функция «помнит» данные между вызовами | Mutable default — заменить на `None` и создавать объект внутри |
| В цикле создаю обработчики, все срабатывают с последним значением | Late binding — использовать `lambda x=x: ...` |
| При удалении в цикле часть элементов пропускается | list: `lst[:]` или list comprehension; dict/set: `list(d)` / `list(s)` или comprehension |
| Сравнение работает не так, как ожидал | Проверка на None — `is None`; значения — `==` |
| Программа тормозит, не знаю где | cProfile — найти функцию с наибольшим cumtime |
| Нужно сравнить скорость двух вариантов | timeit |
| Программа жрёт много памяти | tracemalloc — найти строки с наибольшим выделением |
| Есть баг, хочу пошагово пройти код | `breakpoint()` + pdb (`n`, `s`, `p`, `l`) |
| Нужно разобрать строку как список/словарь от пользователя | `ast.literal_eval` — не eval! |
| Генерирую пароль или токен | `secrets.token_hex()` или `secrets.token_urlsafe()` |
| Регулярное выражение «зависает» на длинном вводе | ReDoS — упростить паттерн, ограничить длину, использовать regex с timeout |

### 7.2. Частые сценарии (краткая таблица)

| Задача | Решение |
|--------|---------|
| Избежать mutable default | `def f(a=None): a = a or []` или `if a is None: a = []` |
| Захват переменной в замыкании по значению | `lambda x=x: ...` |
| Проверка на None | `x is None` |
| Безопасная итерация с удалением | `for x in lst[:]:` или создание нового списка |
| Профилирование времени | `cProfile.run(...)` или `python -m cProfile script.py` |
| Микробенчмарк | `timeit.Timer(...).timeit(number=N)` |
| Профилирование памяти | `tracemalloc` |
| Точечная отладка | `breakpoint()` или `import pdb; pdb.set_trace()` |
| Безопасный разбор литерала | `ast.literal_eval(s)` |
| Токен/пароль | `secrets.token_hex()`, `secrets.token_urlsafe()` |
| Проверка зависимостей | `pip-audit`, `bandit` |

### 7.3. Терминология

| Термин | Значение |
|--------|----------|
| **Mutable default** | Мутабельный аргумент по умолчанию; создаётся один раз при определении функции и хранится в `__defaults__` |
| **Late binding** | Привязка переменной в замыкании по имени — значение берётся в момент вызова, не при создании |
| **Мутабельный / иммутабельный** | Мутабельный — можно изменить «на месте» (list, dict, set). Иммутабельный — нельзя (str, tuple, int) |
| **Замыкание** | Функция, которая «видит» переменные из окружающей области видимости (Enclosing) |
| **LEGB** | Порядок поиска переменной: Local → Enclosing → Global → Built-in |
| **Интернирование** | Хранение одинаковых неизменяемых объектов (мелкие int, короткие строки) в одном экземпляре |
| **Hashable** | Объект, который можно использовать как ключ dict или элемент set: неизменяемый, с устойчивым `hash()`. str, int, tuple, frozenset — да; list, dict, set — нет |
| **Shallow copy** | Копия, где элементы копируются по ссылке; вложенные объекты общие. `lst[:]` — shallow copy списка |
| **id(obj)** | Уникальный идентификатор объекта в памяти; два объекта с одинаковым id — один и тот же объект |
| **MRO** | Method Resolution Order — порядок обхода базовых классов при поиске атрибута/метода |
| **Truthiness** | Поведение объекта в булевом контексте (True/False); пустые коллекции — False |
| **Bound method** | Метод, привязанный к экземпляру; self подставляется при вызове |
| **RecursionError** | Исключение при превышении лимита стека вызовов; Python не оптимизирует хвостовую рекурсию (TCO) |
| **RCE** | Remote Code Execution — выполнение произвольного кода на машине жертвы |
| **CVE** | Common Vulnerabilities and Exposures — каталог известных уязвимостей |
| **ReDoS** | Regular expression Denial of Service — экспоненциальный backtracking на злонамеренном вводе |
| **Bottleneck** | «Узкое место» — участок кода, на который уходит основная доля времени; оптимизация даёт наибольший эффект |
| **cProfile** | Стандартный профилировщик Python (на C) |
| **tracemalloc** | Модуль профилирования памяти в stdlib |

### 7.4. Типичные ошибки

| Ошибка | Как избежать |
|--------|--------------|
| `def f(a=[])` | `def f(a=None): ... if a is None: a = []` |
| `lambda: x` в цикле | `lambda x=x: ...` |
| `x is 5`, `x is "hi"` | `x == 5`, `x == "hi"` |
| Изменение коллекции в цикле | list: `lst[:]`; dict/set: `list(d)` / `list(s)` или comprehension |
| `x or default` при допустимом 0/'' | `x if x is not None else default` или явная проверка |
| `assert` для валидации ввода | Валидация через if/raise; assert только для инвариантов |
| pickle.loads(untrusted) | Не использовать; json, ast.literal_eval и т.п. |
| Конкатенация в SQL | Параметризованные запросы |
| random для паролей | secrets |

### 7.5. Вопросы для самопроверки

#### 3.1 Mutable default и late binding

<details>
<summary>1. Почему <code>def f(a=[]): a.append(1); return a</code> возвращает накопленный список при повторных вызовах?</summary>

Значение по умолчанию <code>[]</code> создаётся один раз при определении функции и хранится в <code>f.__defaults__</code>. Все вызовы без аргумента используют один и тот же объект списка. При append мы меняем этот общий список.
</details>

<details>
<summary>2. Почему <code>if a is None: a = []</code> безопаснее, чем <code>a = a or []</code>?</summary>

<code>a or []</code> заменит на пустой список любые falsy-значения: <code>0</code>, <code>''</code>, <code>[]</code>, <code>False</code>. Если вызывающий передал <code>0</code> как осмысленное значение — оно потеряется. <code>if a is None: a = []</code> заменяет только <code>None</code>.
</details>

<details>
<summary>3. Как исправить замыкание в цикле: <code>funcs = [lambda: i for i in range(3)]</code>, чтобы каждая функция возвращала свой индекс?</summary>

Использовать привязку по значению через аргумент по умолчанию: <code>funcs = [lambda i=i: i for i in range(3)]</code>. При создании каждой лямбды значение <code>i</code> «замораживается» в параметре. Без этого все лямбды смотрят на переменную <code>i</code>, которая после цикла равна 2.
</details>

#### 3.2 Изменение коллекций, is vs ==, интернирование

<details>
<summary>4. Почему нельзя менять список во время итерации <code>for x in lst</code>?</summary>

Итератор хранит индекс. При remove/pop элементы сдвигаются, индекс перестаёт соответствовать — часть элементов пропускается. Итерация по копии <code>lst[:]</code> или создание нового списка решает проблему.
</details>

<details>
<summary>5. Чем отличается изменение dict/set в цикле от списка?</summary>

У dict и set при изменении размера во время итерации возникает <code>RuntimeError</code>. У списка — «молча» пропускаются элементы. Безопасно: <code>for k in list(d):</code> — итерация по копии ключей.
</details>

<details>
<summary>6. Почему <code>for k in list(d):</code> безопасно, а <code>for k in d.keys():</code> — нет?</summary>

<code>d.keys()</code> возвращает view — он отражает текущее состояние словаря. При изменении <code>d</code> во время итерации по view — RuntimeError. <code>list(d)</code> создаёт снимок ключей на момент вызова — обычный список, не связанный с dict.
</details>

<details>
<summary>7. Когда использовать <code>is</code>, а когда <code>==</code>?</summary>

<code>is</code> — только для <code>None</code> (и иногда <code>True</code>/<code>False</code>). Для значений — всегда <code>==</code>. <code>is</code> сравнивает объекты в памяти; для чисел/строк ненадёжно из-за интернирования.
</details>

<details>
<summary>8. Что такое интернирование и почему на него нельзя полагаться?</summary>

Интернирование — когда Python переиспользует один объект для одинаковых значений (например, мелкие int −5..256, некоторые строки). Экономия памяти. Но это деталь реализации CPython — в PyPy, Jython или в другом контексте правила другие. Всегда сравнивайте значения через <code>==</code>.
</details>

#### 3.3 or/and, del, assert, рекурсия, классовые переменные

<details>
<summary>9. Что вернёт <code>0 or []</code> и почему это важно?</summary>

Вернёт <code>[]</code>. <code>or</code> возвращает первый truthy-операнд или последний, если оба falsy. <code>0</code> — falsy, значит вернётся <code>[]</code>. Важно: если 0 — легитимное значение (например, «показать 0 элементов»), <code>x or default</code> его заменит. Нужна проверка <code>if x is None</code>.
</details>

<details>
<summary>10. Что делает <code>del x</code> — удаляет объект или имя?</summary>

Удаляет только **имя** (ссылку) из текущей области видимости. Объект остаётся, пока на него есть другие ссылки. Объект удалится сборщиком мусора только когда ссылок не останется.
</details>

<details>
<summary>11. Почему assert не подходит для валидации пользовательского ввода?</summary>

При запуске с флагом <code>-O</code> все assert удаляются из байткода. Злоумышленник может запустить скрипт с <code>python -O</code> и обойти проверку. Assert — только для внутренних инвариантов (отладка). Для валидации — <code>if ...: raise ValueError(...)</code>.
</details>

<details>
<summary>12. Почему factorial(n) — не хвостовая рекурсия и Python её не оптимизирует?</summary>

В <code>return n * factorial(n - 1)</code> после вызова factorial нужно выполнить умножение — «работа после возврата». Хвостовая рекурсия — когда последнее действие только вызов себя. В Python TCO нет. Решение — переписать на цикл.
</details>

<details>
<summary>13. Почему <code>class Bad: items = []; def __init__(self, x): self.items.append(x)</code> даёт один список на все экземпляры?</summary>

<code>items = []</code> — атрибут **класса**, не экземпляра. При <code>self.items.append(x)</code> Python ищет items — в экземпляре нет, берёт из класса. append меняет тот же классовый список. Правильно: <code>self.items = []</code> в <code>__init__</code> — создать свой список у каждого экземпляра.
</details>

#### 3.3.7 float, hashable

<details>
<summary>14. Почему нельзя сравнивать float через <code>==</code>?</summary>

Числа с плавающей точкой хранятся приближённо. <code>0.1 + 0.2 == 0.3</code> может быть False. Используйте <code>math.isclose(a, b)</code> или <code>abs(a - b) < epsilon</code>.
</details>

<details>
<summary>15. Почему <code>d[[1, 2]] = 3</code> вызывает TypeError?</summary>

Ключи dict должны быть **hashable** — неизменяемыми, с устойчивым хешем. Список — мутабельный, его нельзя использовать как ключ. Используйте кортеж: <code>d[(1, 2)] = 3</code> (если элементы кортежа тоже hashable).
</details>

<details>
<summary>16. Можно ли использовать кортеж <code>(1, [2, 3])</code> как ключ dict?</summary>

Нет. Кортеж hashable только если все его элементы hashable. Список <code>[2, 3]</code> — не hashable. <code>hash((1, [2, 3]))</code> вызовет TypeError.
</details>

#### 4.1 Производительность

<details>
<summary>17. В чём разница между tottime и cumtime в cProfile?</summary>

<strong>tottime</strong> — время внутри самой функции (без вызванных). <strong>cumtime</strong> — время вместе с вызванными функциями. Для поиска bottleneck смотрите cumtime — она показывает, куда уходит время с учётом всего, что вызывается.
</details>

<details>
<summary>18. Зачем в timeit нужен параметр setup?</summary>

Setup — код, который выполняется один раз в начале, чтобы подготовить окружение (например, <code>lst = []</code>). Время setup не включается в замер. Без setup переменные (lst и т.д.) не существовали бы при выполнении stmt.
</details>

<details>
<summary>19. Когда использовать tracemalloc, а когда memory_profiler?</summary>

<strong>tracemalloc</strong> — встроенный, показывает выделения по строкам кода. <strong>memory_profiler</strong> — декоратор <code>@profile</code>, детальный отчёт по строкам внутри функции. tracemalloc — для «где вообще жрёт», memory_profiler — для детального разбора конкретной функции.
</details>

<details>
<summary>20. Когда использовать py-spy вместо cProfile?</summary>

<strong>py-spy</strong> подключается к уже запущенному процессу — без изменения кода, без перезапуска. Удобно для production. <strong>cProfile</strong> требует запуска с профилированием — меняете код или команду. Для «подглядеть» живой процесс — py-spy.
</details>

<details>
<summary>21. Когда использовать memray, а когда scalene?</summary>

<strong>memray</strong> — глубокий анализ памяти, утечки, flamegraph, allocation timeline. Для расследования «кто держит память». <strong>scalene</strong> — быстрый профиль CPU + памяти по строкам, без настройки. Для первого прогона «где тормозит/жрёт».
</details>

#### 4.2 Отладка

<details>
<summary>22. В чём разница между командами <code>n</code> и <code>s</code> в pdb?</summary>

<strong>n</strong> (next) — выполнить текущую строку и перейти к следующей, не заходя в вызовы функций. <strong>s</strong> (step) — шаг с заходом в функцию. На строке <code>x = len(lst)</code>: n — len выполнится, вы останетесь на следующей строке; s — зайдёте внутрь len.
</details>

<details>
<summary>23. Как отключить все breakpoint() в production без изменения кода?</summary>

Переменная окружения <code>PYTHONBREAKPOINT=0</code>. При таком значении breakpoint() ничего не делает. Запуск: <code>PYTHONBREAKPOINT=0 python script.py</code>.
</details>

<details>
<summary>24. Что такое frame в settrace и зачем он нужен?</summary>

<strong>Frame</strong> — объект, представляющий кадр выполнения (текущая функция, локальные переменные). В settrace callback получает frame при каждом событии. <code>frame.f_code.co_name</code> — имя текущей функции. Используется для трассировки и кастомного профилирования.
</details>

#### 5.1 Безопасность

<details>
<summary>25. Как защититься от SQL injection?</summary>

Использовать параметризованные запросы: передавать значения отдельно от текста запроса. Плейсхолдеры <code>?</code>, <code>%s</code>, <code>:name</code>; значения — вторым аргументом (кортеж или словарь). Никогда не склеивать пользовательский ввод в SQL через конкатенацию или f-строки.
</details>

<details>
<summary>26. Почему кортеж <code>(name,)</code> с запятой нужен в cursor.execute?</summary>

В Python <code>(name)</code> — это просто name в скобках. <code>(name,)</code> — кортеж из одного элемента. Запятая говорит интерпретатору: «это кортеж». Второй аргумент execute должен быть iterable (кортеж, список).
</details>

<details>
<summary>27. Чем опасен pickle и когда его можно использовать?</summary>

pickle при загрузке может выполнять произвольный код. Злоумышленник может создать вредоносный файл → RCE. Использовать только для данных из **полностью доверенного** источника (свои файлы, свой сервер). Никогда — из пользовательского ввода, интернета.
</details>

<details>
<summary>28. Чем marshal и shelve отличаются от pickle по безопасности?</summary>

<strong>marshal</strong> — низкоуровневая сериализация, не для произвольных данных; загрузка из ненадёжного источника тоже опасна. <strong>shelve</strong> — построен на pickle, те же риски. Правило то же: только доверенные данные.
</details>

<details>
<summary>29. Чем ast.literal_eval безопаснее eval?</summary>

<code>ast.literal_eval</code> принимает только литералы — числа, строки, списки, словари, кортежи, True/False/None. Не выполняет вызовы функций, импорты, присваивания. eval выполняет любой код. Для разбора строки вида <code>"[1, 2, 3]"</code> — literal_eval.
</details>

<details>
<summary>30. Почему для паролей и токенов нужен secrets, а не random?</summary>

<code>random</code> — псевдослучайный (Mersenne Twister), предсказуем при известном seed. <code>secrets</code> использует криптографически стойкий источник энтропии ОС. Для паролей, токенов, API-ключей — только secrets.
</details>

<details>
<summary>31. Чем pip-audit отличается от Bandit?</summary>

<strong>pip-audit</strong> проверяет установленные пакеты на известные CVE. «Есть ли уязвимость в requests 2.25.0?» <strong>Bandit</strong> проверяет ваш код — ищет опасные паттерны: eval, pickle, SQL injection, хардкод паролей.
</details>

#### 5.2 sys.audit, hmac, ReDoS

<details>
<summary>32. Для чего нужен sys.addaudithook?</summary>

Позволяет перехватывать «опасные» операции (open, subprocess, ctypes и т.д.). При каждом таком событии вызывается ваша функция. Можно логировать, блокировать, проверять. Используется для песочниц и аудита.
</details>

<details>
<summary>33. Когда использовать hashlib, а когда hmac?</summary>

<strong>hashlib</strong> — хеш от данных, без секрета. Для проверки целостности («файл не изменился?»). <strong>hmac</strong> — подпись с секретным ключом. Для «данные пришли от того, кто знает ключ, и не подменены» — cookie, JWT, API-запросы.
</details>

<details>
<summary>34. Почему паттерн <code>(a+)+</code> опасен (ReDoS)?</summary>

Вложенные квантификаторы. Движок regex при несовпадении откатывается (backtrack) и перебирает варианты. Число вариантов растёт экспоненциально с длиной строки. На злонамеренном вводе (много <code>a</code> и в конце не <code>b</code>) — «зависание» на секунды.
</details>

<details>
<summary>35. Как защититься от ReDoS?</summary>

Избегать вложенных квантификаторов на перекрывающихся группах. Ограничивать длину пользовательского ввода. Использовать модуль <code>regex</code> с параметром <code>timeout</code>. Там, где возможно — обходиться без regex: <code>str.startswith</code>, <code>in</code>.
</details>

#### 6.1–6.2 Дополнительные тонкости

<details>
<summary>36. Чем bound method отличается от unbound?</summary>

<strong>Bound method</strong> — <code>obj.method</code>, при вызове <code>self</code> уже подставлен. <strong>Unbound</strong> — <code>Class.method</code>, при вызове нужно передать экземпляр: <code>Class.method(obj)</code>. При передаче callback удобнее bound: <code>button.on_click(obj.handle_click)</code>.
</details>

<details>
<summary>37. Что считается falsy в Python?</summary>

<code>None</code>, <code>False</code>, <code>0</code>, <code>0.0</code>, <code>0j</code>, пустая строка <code>''</code>, пустые коллекции <code>[]</code>, <code>()</code>, <code>{}</code>, <code>set()</code>, <code>range(0)</code>. Для своих классов — <code>__bool__</code> или <code>__len__</code>.
</details>

<details>
<summary>38. Почему <code>a < b < c</code> лучше, чем <code>(a < b) and (b < c)</code>?</summary>

В цепочке сравнений <code>b</code> вычисляется **один раз**. В <code>(a < b) and (b < c)</code> — два раза. Если <code>b</code> — вызов функции с побочным эффектом, разница важна. Также семантика чище: «x в диапазоне».
</details>

<details>
<summary>39. Когда выполняется else у цикла for/while?</summary>

Только если цикл завершился **нормально** — не по <code>break</code>. Если был break — else не выполняется. Используется для поиска: «не нашли» — цикл прошёл все итерации без break, срабатывает else.
</details>

<details>
<summary>40. Когда выполняется else у try?</summary>

Только если в блоке try **не было исключения**. Порядок: try → (если ок) else → finally. Исключения из else не ловятся этим except. Удобно для кода «только при успехе», который не должен перехватываться except.
</details>

<details>
<summary>41. Утекает ли переменная цикла из comprehension в Python 3?</summary>

Нет. В Python 3 у comprehension своя область видимости. После <code>[x for x in range(5)]</code> снаружи <code>x</code> остаётся прежним (или NameError, если не был определён). В Python 2 переменная «утекала» — имела последнее значение.
</details>

### 7.6. Частые вопросы и что может быть непонятно

**«Почему в Python сделано так с mutable default?»**  
Значения по умолчанию вычисляются при определении функции — это удобно для производительности и для тех случаев, когда default — константа. Для мутабельных объектов это даёт неожиданный эффект. Решение — использовать None и создавать объект внутри функции.

**«Я использовал <code>is</code> для чисел, и оно работало. Почему это плохо?»**  
Для мелких чисел (−5..256) CPython переиспользует объекты, и <code>is</code> может случайно сработать. Но это деталь реализации; в другом контексте или другой версии Python результат может измениться. Для значений всегда <code>==</code>.

**«Нужно ли помнить все команды pdb?»**  
Достаточно знать: <code>n</code> (шаг), <code>s</code> (шаг с заходом), <code>c</code> (продолжить), <code>p x</code> (вывести x), <code>l</code> (код), <code>q</code> (выход). Остальное — по необходимости. Можно набирать <code>help</code> в pdb — покажет список команд.

**«Как не забыть материал?»**  
Возвращайтесь к «Памятке на одну страницу» раз в неделю. Делайте мини-упражнения через пару дней после прочтения. При реальном баге — ищите в справочнике «У меня проблема X» и проверяйте по чек-листу. Повторение в контексте — лучший способ закрепления.

**«Можно ли использовать `a or []` вместо `if a is None: a = []`?»**  
Только если вы уверены, что вызывающий **никогда** не передаст пустой список `[]`, 0 или `''` как осмысленное значение. В противном случае `a or []` заменит их на default. Для «опциональный аргумент, default — пустой список» идиоматично `if a is None: a = []`.

## Памятка на одну страницу (для быстрого повторения)

**Mutable default:** `def f(a=None): a = a or []` — никогда `def f(a=[])`.

**Late binding:** В цикле `lambda: i` даёт последнее значение; пишите `lambda i=i: ...`.

**Изменение в цикле:** list — `for x in lst[:]:` или новый список; dict/set — `for k in list(d):` или comprehension. Dict/set при изменении дают RuntimeError.

**is vs ==:** `is` — только для `None`. Для значений — всегда `==`.

**or:** `x or default` заменяет 0, '', False, пустой список. Если 0 или '' — легитимные значения, пишите `if x is None: x = default`.

**assert:** Отключен при `-O`. Не для валидации ввода.

**Профилирование:** cProfile — где время; timeit — микробенчмарк; tracemalloc — память.

**pdb:** `breakpoint()`, затем `n`/`s`/`c`/`p`/`l`/`q`.

**SQL injection:** Параметризованные запросы — `?`, `%s`, `:name`. Никогда не склеивать строки.

**pickle:** Только из доверенного источника. Для ненадёжных данных — json, ast.literal_eval.

**eval/exec:** Опасно. Для литералов — `ast.literal_eval`.

**Токены и пароли:** `secrets.token_hex()`, `secrets.token_urlsafe()` — не `random`.

**ReDoS:** Избегать вложенных квантификаторов; ограничивать длину ввода.

**else в цикле:** Выполняется, если цикл закончился без `break`.

**else в try:** Выполняется, если не было исключения.

**float и ==:** Использовать `math.isclose(a, b)` или `abs(a - b) < epsilon`. Не полагаться на `==` из-за погрешности округления.

**Ключи dict:** Только hashable — str, int, tuple (с hashable элементами), frozenset. Не list, dict, set.

## Закрепи: мини-упражнения (проверь себя руками)

**Почему это важно.**  
Чтение — половина дела. Когда вы сами вводите код и видите результат, связи «причина → следствие» запоминаются крепче. Мозг лучше удерживает то, что связано с действием. Выполни каждое упражнение, даже если кажется простым — «руки помнят» дольше, чем глаза.

Чтобы лучше понять и запомнить, попробуй выполнить в интерпретаторе:

1. **Mutable default:** Напиши `def f(a=[]): a.append(1); return a`. Вызови `f()`, `f()`, `f()`. **Ожидание:** `[1]`, `[1, 1]`, `[1, 1, 1]` — один список накапливает. Потом исправь на `def f(a=None): a = a or []; a.append(1); return a` и повтори — теперь каждый вызов даёт `[1]` (новый список).
2. **Late binding:** `funcs = [lambda: i for i in range(3)]`. Вызови `funcs[0](), funcs[1](), funcs[2]()`. **Ожидание:** все три вернут 2 (последнее значение i). Исправь на `funcs = [lambda i=i: i for i in range(3)]` — теперь 0, 1, 2.
3. **Изменение в цикле:** `lst = [1,2,3,4,5]`, цикл `for x in lst: if x%2==0: lst.remove(x)`. **Ожидание:** останется `[1, 3, 5]`? Нет — часть чётных может остаться (элемент 4 часто пропускается из-за сдвига индексов). С `for x in lst[:]:` получится корректно `[1, 3, 5]`.
4. **is vs ==:** `a=1000; b=1000; print(a is b)` — может быть True или False. `print(a == b)` — всегда True.
5. **or:** `def f(n=None): return n or 10`. Вызови `f(0)` — получишь 10, хотя 0 может быть легитимным (например, «показать 0 элементов»). **Ожидание:** `f(None)` → 10, `f(0)` → 10 (ошибка!), `f(5)` → 5. Исправь: `def f(n=None): return 10 if n is None else n` или `if n is None: n=10; return n`.

6. **dict в цикле:** `d = {"a": 1, "b": 2}; for k in d: del d[k]` — RuntimeError. Исправь: `for k in list(d): del d[k]` — итерация по копии ключей.

## Чек-лист: что проверить в своём коде

Перед коммитом или code review полезно пробежаться:

- [ ] Нет ли `def f(a=[])`, `def f(x={})` — мутабельные default?
- [ ] Нет ли `lambda: x` в цикле по `x` — late binding?
- [ ] Нет ли изменения коллекции в `for x in lst` / `for k in d` / `for x in s`? Используйте копию: `lst[:]`, `list(d)`, `list(s)`.
- [ ] Проверка на None через `is None`, не через `== None`?
- [ ] Сравнение значений через `==`, не через `is` (кроме None)?
- [ ] Нет ли `assert` для проверки пользовательского ввода?
- [ ] SQL-запросы — параметризованные, не конкатенация?
- [ ] pickle — только из доверенного источника?
- [ ] eval/exec — нет с пользовательским вводом? Есть ast.literal_eval где нужно?
- [ ] Токены и пароли — через `secrets`, не `random`?
