### 3.1. Mutable default argument и late binding в замыканиях

#### 3.1.1. Цель раздела

Понять, **почему изменяемые аргументы по умолчанию — ловушка** и **как замыкания захватывают переменные** — чтобы избежать неожиданного поведения. Эти две ошибки очень часто встречаются даже у опытных разработчиков, поэтому стоит разобрать их максимально подробно.

#### 3.1.2. Термины

| Термин | Определение простыми словами |
|--------|------------------------------|
| **Mutable default** | Аргумент по умолчанию — мутабельный объект (list, dict, set). **Мутабельный** = можно изменить «на месте» (append, pop, `dict[key]=...`) без создания нового объекта. **Иммутабельный** = нельзя; любое «изменение» создаёт новый объект (например, `s = s + "x"` — новая строка). В Python такой объект создаётся **один раз** при определении функции и «живёт» внутри функции навсегда. Не пересоздаётся при каждом вызове. |
| **Late binding** | Замыкание «захватывает» переменную **по имени** (как ярлык на коробку), а не по значению (как копию содержимого). Когда вы вызываете замыкание, оно смотрит: «а что сейчас лежит в этой коробке?» — и берёт актуальное значение, а не то, что было в момент создания. |

#### 3.1.3. Mutable default argument — как это устроено (максимально подробно)

**Суть в одном предложении:** аргумент по умолчанию `[]` создаётся один раз при определении функции; все вызовы без аргумента используют один и тот же список — он накапливает данные между вызовами.

**Аналогия.**  
Представьте: вы написали на стене инструкцию: «Если тебе не дали контейнер — возьми вот эту коробку». Коробка одна и висит на стене. Каждый раз, когда кто-то приходит без контейнера, он берёт **ту же самую коробку** и кладёт в неё что-то. Следующий человек увидит в ней то, что оставил предыдущий. Так и работает `def f(a=[])` — коробка (список) одна на всех.

**Пошагово: что происходит в Python.**

1. Вы пишете `def append_to_list(item, target=[]):` и нажимаете Enter.
2. Python **сразу** выполняет эту строку. Он создаёт объект функции и объект пустого списка `[]`.
3. Ссылка на этот список сохраняется в `append_to_list.__defaults__` — это «склад значений по умолчанию» у функции.
4. Когда вы вызываете `append_to_list(1)` **без второго аргумента**, Python смотрит: «Второй аргумент не передан? Беру из __defaults__». И подставляет **тот самый** список.
5. Вы делаете `target.append(1)` — список изменяется. Это тот же объект, что лежит в `__defaults__`.
6. При следующем вызове `append_to_list(2)` снова берётся тот же список из `__defaults__`. В нём уже [1]. Добавляется 2 → [1, 2].
7. И так далее — список накапливает данные между вызовами.

**Код с подробными комментариями:**

```python
def append_to_list(item, target=[]):
    # target — это ссылка на список. При первом вызове без аргумента
    # это ссылка на тот список, который создан при определении функции.
    target.append(item)
    return target

# Вызов 1: target не передан → берётся список из __defaults__
# Этот список был создан ОДИН РАЗ при def. Сейчас он пустой [].
print(append_to_list(1))   # [1]
# Мы добавили 1. Список в __defaults__ теперь [1].

# Вызов 2: target снова не передан → снова тот же список!
# Он уже [1]. Добавляем 2.
print(append_to_list(2))   # [1, 2]

# Вызов 3: тот же список, он [1, 2]. Добавляем 3.
print(append_to_list(3))   # [1, 2, 3]
```

**Как убедиться, что это один и тот же объект:**

```python
def f(a=[]):
    return a

# До вызова — смотрим, что лежит в «складе» по умолчанию
print(f.__defaults__)           # ([],)  — кортеж с одним элементом: пустой список
print(id(f.__defaults__[0]))    # например, 140234567890 — уникальный «адрес» объекта в памяти (id)

f().append(1)                   # изменили список
print(f.__defaults__)           # ([1],) — тот же список, но уже с 1!
print(id(f.__defaults__[0]))    # 140234567890 — тот же id! Один объект.
```

**То же самое с `dict` и `set`.**  
Проблема не только у `[]`. Мутабельные по умолчанию — это list, dict, set. Все они создаются один раз:

```python
def add_to_cache(key, value, cache={}):  # ПЛОХО — тот же dict на всех!
    cache[key] = value
    return cache
```

Правило то же: `def add_to_cache(key, value, cache=None):` и внутри `if cache is None: cache = {}`. Каждый вызов без `cache` создаст **новый** пустой словарь — потому что проверка выполняется при каждом вызове, и `cache = {}` выполняется каждый раз, когда `cache is None`.

**Частая путаница.**  
Многие думают: «При каждом вызове без аргумента Python создаёт новый `[]`». Это не так. Python создаёт `[]` **один раз** — при разборе определения функции. Все вызовы без аргумента получают ссылку на этот же объект.

**Что может быть непонятно (mutable default).**

- «Почему один список на всех?» Потому что default вычисляется **один раз** при определении функции — в момент создания объекта функции. У функции есть атрибут `__defaults__`. Список `[]` создаётся один раз и кладётся в `__defaults__`. Каждый вызов без аргумента берёт ссылку на этот же объект.
- «Почему `a or []` не всегда ок?» Если вызывающий передал `f([])` — пустой список — то `a or []` даст новый `[]`, а не тот же объект. Поведение может отличаться. И если вы хотите различать «не передали» и «передали пустой список», `or` не поможет — оба дадут `[]`. Поэтому идиоматично: `if a is None: a = []`.

**Правило — идиоматичное решение (почему оно работает):**

```python
def append_to_list(item, target=None):
    if target is None:
        target = []   # ← КЛЮЧЕВОЕ: новый список создаётся ЗДЕСЬ, при КАЖДОМ вызове
    target.append(item)
    return target

print(append_to_list(1))   # [1]   — target был None, создали [], добавили 1
print(append_to_list(2))   # [2]   — target снова None, создали НОВЫЙ [], добавили 2
print(append_to_list(3))   # [3]   — и снова новый список
```

Почему `None` безопасен? Потому что `None` — **неизменяемый** объект. Даже если бы он «шарился» между вызовами (а он и шарится — один объект None на весь интерпретатор), его нельзя изменить. Мы не делаем `None.append(...)`, мы делаем `target = []` — создаём **новый** список и присваиваем его в локальную переменную `target`. Этот новый список живёт только внутри одного вызова.

**Почему не `target = target or []`?**  
Потому что `0 or []` даст `[]`, `'' or []` даст `[]`. Если вы хотите передать пустой список явно — `append_to_list(1, [])` — то `target` будет `[]`, и `[] or []` вернёт первый операнд `[]`, всё ок. Но если вызывающий передаст `append_to_list(1, 0)` (допустим, 0 означает «сбросить счётчик» в вашей логике), то `0 or []` подставит `[]` — и вы потеряете переданное значение 0. Лучше явно проверять `if target is None`.

#### 3.1.4. Late binding в замыканиях (максимально подробно)

**Суть в одном предложении:** замыкание запоминает **имя** переменной, а не её значение; при вызове берётся актуальное значение — к тому моменту цикл может уже закончиться.

**Аналогия.**  
Представьте: вы развесили в комнате три листа с надписью «Посмотри, что сейчас написано на доске, и скажи это». На доске пишут числа 0, потом 1, потом 2. Когда вы развешивали листы, на доске было 0, потом 1, потом 2. Но когда кто-то зайдёт и прочитает листы — он посмотрит на доску **сейчас**. К тому моменту на доске написано 2 (последнее значение). Все три листа «смотрят» на одну и ту же доску — переменную `i`. Это и есть late binding: привязка к **месту** (имени переменной), а не к значению на момент создания.

**Пошагово: что происходит.**

1. `i = 0`. Создаётся лямбда `lambda: i`. Она «запоминает»: «когда меня вызовут, я возьму значение переменной `i`». Не «0», а именно «переменной i».
2. `i = 1`. Создаётся вторая лямбда. Она тоже смотрит на переменную `i`.
3. `i = 2`. Создаётся третья лямбда. Тоже смотрит на `i`.
4. Цикл закончился. `i` равно 2.
5. Вы вызываете `functions[0]()`. Лямбда говорит: «Мне нужно значение `i`». Она смотрит в окружающую область — `i` там 2. Возвращает 2.
6. То же для `functions[1]()` и `functions[2]()` — все смотрят на одну переменную `i`, в которой 2.

**Код с разбором:**

```python
functions = []
for i in range(3):      # i = 0, 1, 2
    # lambda: i — «функция без аргументов, которая вернёт то, что в i»
    # Но «что в i» не фиксируется сейчас! Фиксируется только: «ищи переменную i»
    functions.append(lambda: i)

# Цикл кончился. i = 2.
print(functions[0]())  # Лямбда ищет i. i = 2. Выводит 2.
print(functions[1]())  # То же. 2.
print(functions[2]())  # То же. 2.
```

**Правило — привязка по значению через аргумент по умолчанию.**

Аргумент по умолчанию в Python вычисляется **в момент создания функции**. Когда вы пишете `lambda x=i: x`, при каждой итерации цикла:
- `i` имеет конкретное значение (0, 1, 2);
- Python создаёт новую лямбду и вычисляет «чему равен i прямо сейчас»;
- Это значение подставляется в default: `lambda x=0: x`, `lambda x=1: x`, `lambda x=2: x`;
- Каждая лямбда «замораживает» своё значение в параметре по умолчанию.

```python
functions = []
for i in range(3):
    # При i=0: создаётся lambda x=0: x  — значение 0 «запечатано» в default
    # При i=1: создаётся lambda x=1: x
    # При i=2: создаётся lambda x=2: x
    functions.append(lambda x=i: x)

print(functions[0]())  # вызов без аргумента → x берётся из default = 0
print(functions[1]())  # default = 1
print(functions[2]())  # default = 2
```

**Реальный пример — кнопки в GUI:**

```python
# ПЛОХО: создаём 5 кнопок, при нажатии каждая должна выводить свой номер
buttons = []
for i in range(5):
    btn = Button(text=f"Кнопка {i}")  # текст будет правильный: "Кнопка 0", "Кнопка 1"...
    btn.on_click(lambda: print(f"Нажата кнопка {i}"))  # но при клике все выведут "Кнопка 4"!
    buttons.append(btn)
# Почему? Потому что lambda захватывает переменную i. Когда пользователь кликнет —
# цикл давно закончен, i = 4. Все лямбды смотрят на одну переменную.
#
# Почему текст кнопки правильный ("Кнопка 0", "Кнопка 1"...)?
# Потому что f"Кнопка {i}" вычисляется СРАЗУ при создании кнопки — в этот момент i
# имеет нужное значение. А lambda — не вызывается сразу, она вызовется при клике —
# и тогда уже возьмёт актуальное значение i (4).

# ХОРОШО:
for i in range(5):
    btn = Button(text=f"Кнопка {i}")
    btn.on_click(lambda num=i: print(f"Нажата кнопка {num}"))  # num «заморожен» на 0, 1, 2, 3, 4
    buttons.append(btn)
```

**Запомните разницу.**

| Написание | Что захватывается | Результат при вызове после цикла |
|-----------|-------------------|----------------------------------|
| `lambda: i` | Ссылка на переменную `i` | Все вернут последнее значение `i` |
| `lambda x=i: x` | Значение `i` на момент создания | Каждая вернёт своё замороженное значение |

**Что может быть непонятно (late binding).**

- «Почему имя, а не значение?» Функция (в т.ч. lambda) не «фотографирует» переменные при создании. Она хранит ссылку на окружение (scope). При вызове она смотрит: «где переменная i? какое у неё сейчас значение?» — и берёт актуальное. Это и есть «поздняя привязка» (late binding).
- «Как запомнить?» Фраза: «lambda без аргумента по умолчанию — смотрит на переменную. С аргументом `x=i` — замораживает значение i на момент создания».

#### 3.1.5. Запомните

| Плохо | Лучше |
|-------|-------|
| `def f(a=[]):` | `def f(a=None): a = a or []` или `if a is None: a = []` |
| `lambda: x` в цикле по `x` | `lambda x=x: x` — привязка по значению через default |

### 3.2. Изменение коллекций во время итерации, `is` vs `==`, интернирование

#### 3.2.1. Изменение списка во время итерации (пошагово)

**Суть в одном предложении:** итератор хранит индекс; при remove/pop элементы сдвигаются — индекс «съезжает», часть элементов пропускается. Итерируйте по копии `lst[:]` или создавайте новый список.

**Почему это опасно — пошаговый разбор.**

Когда вы пишете `for x in lst:`, Python создаёт **итератор** для списка. Итератор — объект, который «выдаёт» элементы по одному при каждом вызове `next()`. У списка итератор хранит **индекс текущей позиции**. На каждой итерации он возвращает `lst[index]` и увеличивает index на 1.

Допустим, `lst = [1, 2, 3, 4, 5]`. Индексы: 0→1, 1→2, 2→3, 3→4, 4→5.

| Шаг | index | x = lst[index] | Действие | lst после |
|-----|-------|----------------|----------|-----------|
| 1 | 0 | 1 | 1 нечётное, не удаляем | [1, 2, 3, 4, 5] |
| 2 | 1 | 2 | 2 чётное, remove(2) | [1, 3, 4, 5] ← элементы сдвинулись! |
| 3 | 2 | 4 | Индекс 2 теперь указывает на 4 (а не на 3!) | 3 **пропущен** |
| 4 | 3 | 5 | ... | ... |

После `remove(2)` список стал `[1, 3, 4, 5]`. Элемент 3 «переехал» с индекса 2 на индекс 1. Но итератор уже перешёл на index=2 и возвращает то, что на позиции 2 — а это теперь 4. Элемент 3 мы **пропустили**.

**Добавление в цикле — бесконечный цикл.**  
Если в `for x in lst:` вы делаете `lst.append(something)`, список растёт. Итератор обходит список по индексам и «увидит» новые элементы. Если логика такова, что новые элементы снова приводят к append (например, `for x in lst: lst.append(x*2)` при lst=[1]) — цикл может не закончиться никогда. Итератор дойдёт до нового элемента, добавит ещё, и так далее.

В других случаях — пропуск элементов, дублирование, неопределённое поведение. **Не гарантировано** — не полагайтесь на это.

**Почему `lst[:]` решает проблему.**  
`lst[:]` — это **срез от начала до конца**. Он создаёт **новый список** с теми же элементами (shallow copy — «мелкая копия»). Итератор создаётся для этого **нового** списка. Вы удаляете из **оригинала** `lst`. Копия не меняется, итератор по копии не «съезжает» — он спокойно обходит все элементы копии. Оригинал вы меняете снаружи цикла — это безопасно.

**Что такое shallow copy.**  
Shallow copy («мелкая копия») — копируются сами элементы, но не их «содержимое». Если в списке лежат другие списки или объекты, копируются **ссылки** на них — вложенные списки остаются общими. Изменение `lst_copy[0].append(1)` повлияет и на оригинал `lst[0]`. Для удаления элементов из **плоского** списка (числа, строки) в цикле shallow copy достаточно — мы не меняем элементы, а удаляем из списка.

**Shallow vs deep copy (когда нужно помнить):**  
`copy.deepcopy(lst)` создаёт полную копию со всеми вложенными объектами. Если в списке вложенные списки и вы их изменяете — нужен deepcopy. Для простого «удалить чётные из плоского списка» — достаточно `lst[:]`.

**Правило — итерация по копии или создание нового списка.**

```python
# Вариант 1: итерация по копии lst[:]
# lst[:] — это НОВЫЙ список с теми же элементами. Итератор ходит по копии.
# Удаляем из ОРИГИНАЛА lst. Копия не меняется.
for x in lst[:]:
    if x % 2 == 0:
        lst.remove(x)

# Вариант 2: list comprehension — создать новый список без чётных (ПРЕДПОЧТИТЕЛЬНО)
# Мы не меняем lst «на лету», а строим новый список. Старый lst заменяется новым.
lst = [x for x in lst if x % 2 != 0]

# Вариант 3: идти с конца — при удалении с конца индексы в начале не сдвигаются
# range(len(lst)-1, -1, -1): старт=последний индекс, стоп=-1 (не включая), шаг=-1
# range(4, -1, -1) для lst из 5 элементов даёт: 4, 3, 2, 1, 0 — с конца к началу
# range(start, stop, step): при step=-1 идём вниз. stop=-1 — «до -1 не включая».
# После 0 следующее было бы -1, но -1 не включается — останавливаемся. 0 входит!
for i in range(len(lst) - 1, -1, -1):
    if lst[i] % 2 == 0:
        lst.pop(i)
```

**Когда какой вариант выбирать.**  
- Вариант 2 (list comprehension) — самый простой и читаемый. Используйте по умолчанию.  
- Вариант 1 — когда нужно удалять из списка «на месте» и не создавать новый объект (например, список очень большой).  
- Вариант 3 — когда нужен именно индекс (например, для удаления по условию, зависящему от индекса).

**Зачем идти с конца?** При удалении `lst.pop(i)` все элементы после `i` сдвигаются влево. Если идти с начала — сдвиг «портит» индексы для следующих элементов. Если идти с конца — удаление элемента не влияет на индексы элементов до него (они остаются на своих местах). Поэтому `range(len(lst)-1, -1, -1)` — безопасно.

**Что может быть непонятно (изменение в цикле).**

- «Почему индекс «съезжает»?» Итератор хранит число — «сейчас я на позиции 2». Когда вы удаляете элемент на позиции 1, все элементы после него сдвигаются: то, что было на 2, оказывается на 1. Итератор же увеличивает своё число и идёт на 2 — но на позиции 2 теперь уже другой элемент (тот, что был на 3). Предыдущий элемент (тот, что был на 2) мы пропустили.
- «Почему remove(2) удаляет значение 2, а не индекс?» `list.remove(x)` удаляет **первый элемент со значением** x. `list.pop(i)` удаляет элемент по **индексу** i.
- «Почему `lst[:]` решает проблему?» Потому что мы итерируем по **копии**. Копия не меняется, когда вы делаете `lst.remove(x)` — вы удаляете из **оригинала**. Итератор ходит по копии и не «съезжает».

#### 3.2.1a. Изменение dict и set во время итерации (подробно)

**Суть в одном предложении:** для dict и set в Python 3 при изменении размера во время итерации сразу возникает **RuntimeError** — в отличие от списка, где итератор «молча» пропускает элементы. Поведение разное, но правило одно: не изменяйте коллекцию во время итерации по ней.

**Почему dict и set ведут себя иначе.**  
У списка итератор хранит **индекс**. При удалении элемента индексы «съезжают» — результат неочевидный (пропуск элементов), но исключения нет. У dict и set внутренняя структура — хеш-таблица. При изменении во время итерации Python явно проверяет: «словарь/множество изменились с начала итерации» — и выбрасывает `RuntimeError: dictionary/set changed size during iteration`. Это **защита** от неопределённого поведения.

**Пример — что происходит:**

```python
d = {"a": 1, "b": 2, "c": 3}
for k in d:
    del d[k]   # RuntimeError: dictionary changed size during iteration

s = {1, 2, 3}
for x in s:
    s.remove(x)  # RuntimeError: Set changed size during iteration
```

**Как безопасно удалять из dict во время обхода.**

```python
# Вариант 1: итерация по копии ключей — list(d) или list(d.keys())
# list(d) создаёт список ключей на момент вызова. Итератор ходит по этому списку.
# Удаляем из оригинального d — копия ключей не меняется.
for k in list(d):
    if some_condition(k):
        del d[k]

# Вариант 2: dict comprehension — создать новый словарь
d = {k: v for k, v in d.items() if some_condition(k)}

# Вариант 3: собрать ключи для удаления, потом удалить (если много условий)
to_remove = [k for k in d if some_condition(k)]
for k in to_remove:
    del d[k]
```

**Как безопасно удалять из set во время обхода.**

```python
# list(s) создаёт копию элементов. Итератор ходит по ней.
for x in list(s):
    if some_condition(x):
        s.remove(x)

# Или set comprehension — новый set
s = {x for x in s if some_condition(x)}
```

**Что может быть непонятно (dict/set).**

- «Почему `list(d)` а не `d.keys()`?» `d.keys()` возвращает **view** — объект, который отражает текущее состояние словаря. Если вы итерируете по `d.keys()` и меняете `d`, view «видит» изменения — и получите RuntimeError. `list(d)` создаёт **снимок** ключей на момент вызова — это обычный список, он не связан с словарём. Итерация по нему безопасна.
- «Можно ли добавлять в dict во время итерации?» Добавление тоже меняет размер — будет RuntimeError. Правило: не изменяйте размер коллекции (добавление, удаление) во время итерации.

**Сводка: список vs dict vs set.**

| Коллекция | При изменении во время итерации | Безопасный способ |
|-----------|----------------------------------|-------------------|
| list | Пропуск элементов, неочевидное поведение | `for x in lst[:]:` или новый список |
| dict | RuntimeError | `for k in list(d):` или dict comprehension |
| set | RuntimeError | `for x in list(s):` или set comprehension |

#### 3.2.2. `is` vs `==` (разница простыми словами)

**Суть в одном предложении:** `is` — «один и тот же объект в памяти»; используйте только для `None`. Для сравнения значений (числа, строки, списки) — всегда `==`.

**Аналогия.**  
`==` — «эти два яблока одинаковые по весу и цвету?» (сравнение **значений**).  
`is` — «это одно и то же яблоко, которое я держу в руке?» (сравнение **объектов в памяти**).

| Оператор | Вопрос | Когда использовать |
|----------|--------|--------------------|
| `==` | «Равны ли значения?» | Числа, строки, списки, свои объекты — почти всегда |
| `is` | «Это один и тот же объект в памяти?» | Только `None`, `True`, `False` — и то с оговорками |

**Почему для `None` нужен `is`?**  
Потому что в мире только **один** объект `None`. Все переменные, равные None, ссылаются на него. `x == None` тоже работает, но `x is None` — идиоматичнее и чуть быстрее. Никакой объект не должен переопределять «равенство с None» — `is` это гарантирует.

**Почему `is` опасен для чисел и строк?**  
CPython (стандартный интерпретатор) для экономии памяти **интернирует** мелкие целые числа и некоторые строки. То есть объекты 0, 1, 2, ..., 256 создаются один раз и переиспользуются. Когда вы пишете `a = 100` и `b = 100`, оба могут ссылаться на один объект. Поэтому `a is b` может быть `True`. Но для 1000, 10000 и т.д. — уже не гарантировано. И в других реализациях Python (PyPy, Jython) правила другие. **Вывод:** для значений всегда `==`.

```python
# ПРАВИЛЬНО
if x is None:
    ...
if items == []:       # или if not items (пустой список — falsy)
    ...

# ОПАСНО — не делайте так
if x is 5:            # может «случайно» работать для маленьких чисел
if name is "admin":   # может «случайно» работать для некоторых строк
```

#### 3.2.3. Интернирование (interning) — что это и почему важно знать

**Простыми словами:** интернирование — когда Python вместо создания нового объекта «такого же» возвращает ссылку на уже существующий. Экономится память. Но на это **нельзя полагаться** — это деталь реализации.

**Целые числа (−5..256) в CPython:**  
Эти числа создаются при старте интерпретатора и переиспользуются. Поэтому `a = 100; b = 100; a is b` — `True`. Для чисел вне этого диапазона — как повезёт.

```python
a = 100
b = 100
print(a is b)   # True — один объект

a = 1000
b = 1000
# В интерактивном режиме — может быть True (одна строка — один объект)
# В скрипте с разными строками — может быть False
print(a is b)   # НЕ ГАРАНТИРОВАНО
```

**Строки:**  
**Литерал** — значение, записанное прямо в коде (например, `"hello"` в кавычках). Литералы строк в одном модуле часто интернируются. **Динамически собранные** строки (`"".join(...)`, `str(x)`) создаются в рантайме — как правило, новый объект.

```python
a = "hello"
b = "hello"
print(a is b)   # часто True для литералов

a = "hel" + "lo"   # компилятор при разборе кода может «свернуть» в "hello" (константное сложение)
b = "hello"
print(a is b)   # может быть True

a = "".join(["hel", "lo"])  # выполняется в рантайме (во время выполнения программы, а не при компиляции)
b = "hello"
print(a is b)   # обычно False — разные объекты
```

**Главное:** не используйте `is` для сравнения значений. Только для `None` (и по желанию для `True`/`False`).

**Что может быть непонятно (интернирование).**  
«Зачем это нужно?» Чтобы сэкономить память. Если в программе сотни раз встречается строка `"error"`, хранить один объект выгоднее, чем сотни копий. Но на это **нельзя полагаться** — это внутренняя оптимизация CPython, не часть языка. Пишите код так, будто интернирования нет — сравнивайте через `==`.

**Что может быть непонятно (`is` vs `==`).**

- «Почему `is` иногда «работает» для чисел?» CPython ради экономии памяти кеширует маленькие целые (−5..256). Когда вы пишете `a = 100; b = 100`, оба могут указывать на один и тот же объект. `a is b` будет True. Но это **деталь реализации** — в другом Python или для 1000 это может быть False. Не полагайтесь.
- «Когда точно нужен `is`?» Только для `None`. Для `True`/`False` тоже можно, но `==` обычно достаточно. Для своих классов `is` почти никогда не нужен — сравнивайте по смыслу через `==` (и определите `__eq__`).

#### 3.2.4. Запомните

| Ситуация | Правило |
|----------|---------|
| Проверка на `None` | Только `x is None` |
| Сравнение чисел, строк, коллекций | Всегда `==` |
| Изменение списка в цикле | Итерация по `lst[:]` или новый список |
| Изменение dict/set в цикле | Итерация по `list(d)` / `list(s)` или comprehension |

### 3.3. Цепочки or/and, del, assert, рекурсия, классовые переменные

#### 3.3.1. Цепочка `or` и проблема для 0, '', False

**Как работает `or`.**  
`a or b` возвращает первый операнд, который в булевом контексте даёт `True` (такие объекты называют **truthy**). Если оба **falsy** — возвращается последний.

```python
print(None or 42)   # 42   — None falsy, 42 truthy → 42
print(0 or 100)     # 100  — 0 falsy, 100 truthy → 100
print('' or 'hi')   # 'hi' — пустая строка falsy
print([] or [1])    # [1]  — пустой список falsy
print(0 or [])      # []   — оба falsy, вернётся последний
```

**Подводный камень.**  
Если вы хотите «подставить default только когда None», а легитимные значения — `0`, `''`, `False`, то `x or default` их **заменит**:

```python
def set_page_size(size=None):
    size = size or 10   # ОШИБКА!
    return size

set_page_size(10)   # 10 — ок
set_page_size(None) # 10 — ок
set_page_size(0)    # 10 — НЕПРАВИЛЬНО! Пользователь явно сказал «показывать 0 элементов на странице» (отключить пагинацию).
```

**Правило — явная проверка на None:**

```python
def set_page_size(size=None):
    if size is None:
        size = 10
    return size
```

#### 3.3.2. Short-circuit evaluation в `and`

`a and b`: если `a` falsy, `b` **вообще не вычисляется**. Это называется «short-circuit» (короткое замыкание). Python не «доходит» до `b` — возвращается результат `a`.

```python
lst = []
x = lst and lst[0]   # lst пустой → falsy. lst[0] не вызывается → IndexError не будет
# x = [] — вернулся первый falsy операнд

lst = [10]
x = lst and lst[0]   # lst непустой → truthy. Вычисляется lst[0] → 10
# x = 10
```

Используют для безопасного доступа: `obj and obj.method()` — если obj None или пустой, `method` не вызовется. Если obj есть — вызовется и вернётся результат.

**Что может быть непонятно (or/and).**  
- «Почему `0 or 100` даёт 100?» `or` возвращает **первый truthy** операнд или последний, если оба falsy. `0` — falsy, проверяем второй — `100` truthy → возвращаем 100. Не `True`/`False`, а само значение!  
- «Почему short-circuit важен?» В `lst and lst[0]` если `lst` пустой — `lst[0]` вызовет IndexError. Но `and` не доходит до `lst[0]` — первый операнд `[]` falsy, сразу возвращает `[]`. Второй операнд **не вычисляется**.

**Таблица: что считается truthy и falsy (для or/and):**

| falsy (False в булевом контексте) | truthy (True) |
|-----------------------------------|---------------|
| `None` | Любое число кроме 0 |
| `False` | Непустая строка `"hello"` |
| `0`, `0.0`, `0j` | Непустой список `[1, 2]` |
| `''` (пустая строка) | Непустой словарь `{1: 2}` |
| `[]`, `()`, `{}`, `set()` | Объекты с `__bool__` возвращающим True |
| `range(0)` | |

#### 3.3.3. `del` — удаляет ссылку, не объект

**Аналогия.**  
Представьте объект как коробку, а переменные — как наклейки с именами на ней. `del x` срывает наклейку «x» с коробки. Коробка никуда не делась — она на месте, если на ней есть другие наклейки (другие переменные ссылаются на неё). Объект удалится из памяти только когда на него не останется ни одной ссылки — тогда сборщик мусора освободит память.

**Важно:** `del x` не «удаляет объект». Оно удаляет **имя** `x` из текущей области видимости. Объект остаётся, пока на него есть хотя бы одна ссылка.

```python
a = [1, 2, 3]
b = a
del a       # имя a удалено. Список [1,2,3] жив — на него ссылается b
print(b)    # [1, 2, 3]
del b       # теперь ссылок на список нет → память освободится
```

**Частая путаница:** «Я удалил a, почему b всё ещё работает?» Потому что вы удалили только имя `a`. Список остался, и `b` на него ссылается.

**Что может быть непонятно (del).**  
- «Когда объект реально удаляется?» Когда на него не остаётся **ни одной** ссылки. Сборщик мусора (gc) позже освободит память — не обязательно сразу после последнего del.  
- «del lst[i] — это то же самое?» Нет. `del x` удаляет имя (переменную). `del lst[i]` удаляет элемент из списка — это вызов метода списка, список изменяется.

#### 3.3.4. `assert` — отключен при `python -O`

`assert cond, msg` — это проверка «так и должно быть» для отладки. При запуске с флагом оптимизации `-O` все assert **выбрасываются** из байткода и не выполняются. (Python считает, что в «оптимизированном» режиме вы уже уверены в коде и лишние проверки не нужны.) Поэтому:

- **Не** используйте assert для валидации пользовательского ввода — злоумышленник может запустить с `-O` и обойти проверку.
- Используйте только для **внутренних инвариантов**: «эта переменная не должна быть None здесь», «список не пустой» и т.п.

**Пример: assert для инварианта (правильно):**

```python
def process(items):
    assert len(items) > 0, "вызвали process с пустым списком — это баг в коде"
    # мы уверены, что items не пустой — это наша внутренняя договорённость
```

**Пример: assert для пользовательского ввода (неправильно):**

```python
def set_age(age):
    assert 0 <= age <= 150, "неверный возраст"  # ПЛОХО!
    # Пользователь может передать -5, а скрипт запустить с python -O — проверка пропустится
```

**Правильно для пользовательского ввода:**

```python
def set_age(age):
    if not (0 <= age <= 150):
        raise ValueError("неверный возраст")
```

```bash
python script.py      # assert выполняется
python -O script.py   # assert пропускается (оптимизация)
```

#### 3.3.5. Рекурсия: лимит стека и отсутствие TCO

**Почему рекурсия может упасть.**  
При каждом вызове функции Python кладёт в **стек** (LIFO — «последний вошёл, первый вышел») информацию о вызове: локальные переменные, куда вернуться после return и т.д. Это называется **кадр** (frame). Стек ограничен — обычно примерно 1000 кадров (зависит от системы: `sys.getrecursionlimit()`). При `factorial(10000)` будет 10000 вложенных вызовов — 10000 кадров в стеке. Стек переполнится, получите `RecursionError`.

**TCO (Tail Call Optimization).**  
В некоторых языках (Scheme, Erlang) компилятор оптимизирует **хвостовую рекурсию** — когда последнее действие функции — вызов самой себя **без последующих операций**. Тогда новый вызов может заменить текущий кадр в стеке, и глубина не растёт. В Python такой оптимизации **нет**.

**Почему `factorial` не хвостовая рекурсия.** В `return n * factorial(n - 1)` после вызова `factorial(n - 1)` ещё нужно выполнить умножение `n * ...` — это «работа после возврата». Значит, текущий вызов не может быть «выброшен» из стека, ему нужно дождаться результата и умножить. Хвостовая рекурсия была бы: `return factorial(n - 1, n * acc)` — последнее действие только вызов, ничего после.

**Как обойти — переписать на цикл:**

```python
def factorial(n):
    if n <= 1:
        return 1
    return n * factorial(n - 1)  # RecursionError при большом n

# Вариант с циклом — без ограничения стека:
def factorial_iter(n):
    result = 1
    for i in range(2, n + 1):
        result *= i
    return result

factorial_iter(10000)  # сработает
```

**Что может быть непонятно (рекурсия).**  
«Почему 1000 кадров?» Ограничение задано в `sys.getrecursionlimit()` — можно увеличить `sys.setrecursionlimit(5000)`, но это не решит проблему, только отодвинет. При слишком глубокой рекурсии стек переполнится в любом случае. Лучше переписать на цикл. «RecursionError» — исключение, которое выбрасывается при переполнении стека вызовов.

Для более сложной рекурсии можно использовать **явный стек** — список `stack = []`, в который вы сами кладёте и забираете данные (`stack.append(...)`, `stack.pop()`). Тогда глубина рекурсии не ограничена стеком вызовов — ограничена только памятью. Это называется «переписать рекурсию на цикл со стеком».

#### 3.3.6. Классовая vs экземплярная переменная (подробно)

**Ментальная модель.**  
- **Атрибут класса** — как общая доска в классе. Все ученики (экземпляры) её видят. Изменили надпись на доске — видят все.  
- **Атрибут экземпляра** — как личная тетрадь у каждого ученика. У каждого своя, меняешь свою — чужие не трогаются.

**Разница простыми словами.**  
- **Атрибут класса** — общий для всех экземпляров. Хранится в объекте класса.  
- **Атрибут экземпляра** — свой у каждого экземпляра. Хранится в `obj.__dict__`.

При обращении `obj.attr` Python сначала ищет в экземпляре (`obj.__dict__`), потом в классе и по **MRO** (Method Resolution Order — порядок обхода базовых классов при множественном наследовании). При присваивании `obj.attr = value` **всегда** создаётся/обновляется атрибут экземпляра — даже если до этого значение бралось из класса. При присваивании `Class.attr = value` меняется атрибут класса — это влияет на все экземпляры, у которых нет своего `attr`.

```python
class Dog:
    species = "Canis"  # атрибут класса — общий

    def __init__(self, name):
        self.name = name  # атрибут экземпляра — свой у каждой собаки

a = Dog("Бобик")
b = Dog("Шарик")
print(a.species)   # Canis — из класса
print(b.species)   # Canis
Dog.species = "Canis lupus"
print(a.species)   # Canis lupus — изменилось у всех

a.species = "домашняя"  # создали атрибут экземпляра у a
print(a.species)   # домашняя
print(b.species)   # Canis lupus — у b по-прежнему из класса
```

**Почему `self.items.append(x)` идёт в классовый список.** При обращении `self.items` Python ищет `items` — в экземпляре нет, идёт в класс. Находит `items = []` в классе. `append` вызывается для этого списка. Присваивания нет — мы не делаем `self.items = ...`, мы меняем содержимое списка. Поэтому все экземпляры модифицируют один и тот же классовый список.

**Частая ошибка:** использовать атрибут класса как «общий список» и модифицировать его в `__init__`:

```python
class Bad:
    items = []  # один список на ВСЕ экземпляры!

    def __init__(self, x):
        self.items.append(x)  # все добавляют в один список!

a = Bad(1)
b = Bad(2)
print(a.items)  # [1, 2] — неожиданно!
```

Правильно — создавать свой список в экземпляре: `self.items = []` в `__init__`.

**Что может быть непонятно (классовые vs экземплярные).**

- «Когда использовать атрибут класса?» Когда значение **общее** для всех экземпляров и его редко меняют: константы (`MAX_SIZE = 100`), справочники, счётчики (осторожно с мутабельными!). Для списков, словарей и прочего мутабельного, который должен быть свой у каждого — только в `__init__`.
- «Почему `a.species = "домашняя"` создаёт атрибут экземпляра?» При присваивании `obj.attr = value` Python **всегда** пишет в экземпляр. Даже если до этого `attr` брался из класса — теперь у объекта появится свой `attr`, и он «закроет» классовый.

#### 3.3.7. Другие частые ловушки: float и ==, мутабельные объекты как ключи dict

**Сравнение float через `==` — подводный камень.**

Числа с плавающей точкой в компьютере хранятся приближённо (двоичная дробь). Из-за округления `0.1 + 0.2` может быть не ровно `0.3`, а `0.30000000000000004`. Поэтому `0.1 + 0.2 == 0.3` может дать `False`.

```python
print(0.1 + 0.2)                    # 0.30000000000000004
print(0.1 + 0.2 == 0.3)             # False!

# Правильно — сравнивать с допуском (epsilon):
import math
def approx_equal(a, b, tol=1e-9):
    return abs(a - b) < tol
approx_equal(0.1 + 0.2, 0.3)        # True

# Или использовать math.isclose (Python 3.5+)
math.isclose(0.1 + 0.2, 0.3)        # True
```

**Правило:** для float не полагайтесь на `==`. Используйте `math.isclose(a, b)` или сравнение с допуском `abs(a - b) < epsilon`.

**Мутабельные объекты как ключи dict — TypeError.**

Ключи словаря должны быть **hashable** — неизменяемыми и с устойчивым хешем. Список, dict, set — мутабельные, их нельзя использовать как ключи.

```python
d = {}
d[[1, 2]] = 3   # TypeError: unhashable type: 'list'
d[{1, 2}] = 3   # TypeError: unhashable type: 'set'
d[{"a": 1}] = 3 # TypeError: unhashable type: 'dict'

# Правильно — кортеж (если элементы hashable)
d[(1, 2)] = 3   # OK
d[("a", "b")] = 3  # OK
d[(1, [2, 3])] = 3  # TypeError — кортеж содержит список!
```

**Почему нужна неизменяемость.** Хеш ключа используется для поиска в хеш-таблице. Если ключ изменится после вставки (например, `lst.append(4)`), хеш изменится — запись станет «потерянной», dict будет работать неправильно. Поэтому только иммутабельные типы (str, int, tuple, frozenset) допустимы как ключи.

**Что может быть непонятно.**

- «Почему tuple, а не list?» Кортеж иммутабелен — после создания его нельзя изменить. Список — мутабельный. Если бы список был ключом, вы могли бы сделать `key.append(1)` — ключ изменился, хеш изменился, dict «сломался».
- «Элементы кортежа тоже должны быть hashable?» Да. `(1, [2, 3])` — кортеж, но содержит список. Такой кортеж нельзя использовать как ключ — `hash((1, [2, 3]))` выдаст TypeError.
