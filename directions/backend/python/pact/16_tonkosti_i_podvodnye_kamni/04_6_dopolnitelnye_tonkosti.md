### 6.1. Bound vs unbound метод, truthiness, цепочка сравнений

#### 6.1.1. Bound и unbound метод (подробно)

**Простыми словами.**  
Метод класса — это функция с первым параметром `self`. Когда вы вызываете `obj.method()`, Python «подвязывает» `obj` к `self` — это **bound method**. Когда вы берёте `Class.method` — это просто функция, при вызове нужно самому передать экземпляр — это **unbound** (в Python 3 это обычная функция).

```python
class C:
    def f(self):
        return 42

c = C()

# Через экземпляр — bound method
print(c.f)    # <bound method C.f of <__main__.C object>>
c.f()         # Python сам подставляет c как self. Эквивалент C.f(c)

# Через класс — unbound (просто функция)
print(C.f)    # <function C.f at ...>
C.f(c)        # нужно явно передать экземпляр
```

**Когда это важно.**  
При передаче метода как callback: `button.on_click(obj.handle_click)` — передаётся bound method, при вызове `self` уже будет `obj`.

**Что может быть непонятно (bound vs unbound).**  
- «Почему `c.f` и `C.f` — разные вещи?» `c.f` — это «метод, уже привязанный к объекту c». При вызове `c.f()` Python не спрашивает, кто вызывающий — он уже знает: `c`. `C.f` — это «сырая» функция класса; при вызове `C.f(c)` вы сами передаёте экземпляр.  
- «Когда нужен unbound?» Редко. Иногда для явной передачи: `map(C.transform, list_of_c_objects)` — `C.transform` как функция, к которой по очереди подставят каждый объект.

#### 6.1.2. Truthiness — что считается True и False

В условиях `if x:` и в `bool(x)` Python определяет «истинность» объекта. Следующее считается **falsy** (ложь):

- `None`, `False`
- Числа: `0`, `0.0`, `0j`
- Пустые коллекции: `''`, `[]`, `()`, `{}`, `set()`, `range(0)`
- Для своих классов — результат `__bool__` или `__len__` (если `__bool__` нет, используется `len(x) != 0`)

Всё остальное — **truthy**.

**Что может быть непонятно (truthiness).**  
Для своих классов Python смотрит: есть ли `__bool__` — вызывается он. Нет — смотрит `__len__`, если есть: `len(x) == 0` → falsy, иначе truthy. Нет ни того, ни другого — всегда truthy. Пустой список `[]` falsy, потому что `len([]) == 0`.

```python
if []:           # falsy — блок не выполнится
    ...
if "hello":      # truthy — выполнится
    ...
if not items:    # «если список пустой» — пустой список falsy, not [] → True
    ...
```

#### 6.1.3. Цепочка сравнений

`a < b < c` в Python — это **не** то же самое, что `(a < b) < c`. Это эквивалентно `a < b and b < c`, причём `b` вычисляется **один раз**. Удобно для диапазонов:

```python
if 0 <= x < 10:   # x в диапазоне [0, 10)
    ...
```

**Почему «один раз» важно.** Если `b` — вызов функции с побочным эффектом, при `a < b < c` она вызовется один раз. При `(a < b) and (b < c)` — два раза. Пример: `if 0 < get_value() < 10` — `get_value()` вызовется один раз; `if 0 < get_value() and get_value() < 10` — два раза.

#### 6.1.4. Операторы и магические методы

За каждым оператором стоит вызов метода:

| Операция | Метод |
|----------|-------|
| `obj[key]` | `__getitem__(key)` |
| `obj(...)` | `__call__(...)` |
| `x in obj` | `__contains__(x)` или итерация |
| `with obj:` | `__enter__`, блок, `__exit__` |
| `for x in obj:` | `iter(obj)` получает итератор, затем вызывается `next()` до StopIteration |

**Что может быть непонятно (магические методы).**  
Это «под капотом» — когда вы пишете `obj[key]`, Python вызывает `obj.__getitem__(key)`. Вы можете определить эти методы в своих классах, чтобы объект вёл себя как список, вызываемый объект и т.д.

### 6.2. else в цикле и try, scope comprehensions

#### 6.2.1. else в цикле (часто непонятно)

У `for` и `while` может быть блок `else`. Он выполняется **только если цикл завершился нормально** — не по `break`. Если был `break` — `else` **не** выполняется.

**Ментальная модель.**  
Думайте так: «else = цикл сделал всё, что мог, и не нашёл повода выйти раньше». Если вышли по `break` — «повод нашёлся», else не нужен. Если прошли все итерации без break — «повода не было», выполняем else.

**Зачем это нужно.**  
Поиск элемента: если нашли — `break`. Если не нашли — цикл закончится сам, сработает `else`.

```python
for x in []:
    pass
else:
    print("Цикл закончился без break")  # выполнится! Цикл пустой — итераций 0, break не было
# Пустой цикл «завершился нормально» — ни разу не сработал break. Значит, else выполнится.

for x in [1, 2, 3]:
    if x == 2:
        break
else:
    print("Не выполнится")  # был break — else пропускается

# Пример использования: поиск
for item in items:
    if item == target:
        print("Найдено")
        break
else:
    print("Не найдено")  # выполнится, только если break не сработал
```

**Что может быть непонятно (else в цикле).**  
«Почему «else» — это логично?» Думайте так: цикл ищет «повод выйти раньше» (break). Если повод найден — выходим по break, else не выполняем. Если повода не было — прошли все итерации «честно», тогда else — «иначе (когда не нашли повод) делаем вот это». То есть else = «цикл завершился штатно, без break».

#### 6.2.2. else в try

Блок `else` у `try` выполняется **только если в try не было исключения**. Порядок: try → (если нет исключения) else → finally. Если было исключение: except → finally.

**Зачем нужен else.**  
Иногда вы хотите выполнить код **только при успехе**, но не в том же блоке try (чтобы не ловить исключения от этого кода). `else` идеален: код в else выполнится только если try прошёл без исключения, при этом исключения из else не перехватятся этим except.

```python
try:
    result = risky_operation()
except ValueError:
    handle_error()
else:
    # сюда попадём только если risky_operation() не выбросил
    # process(result) — если упадёт, except ValueError его НЕ поймает
    process(result)
finally:
    cleanup()  # всегда
```

**Без else** пришлось бы писать `process(result)` внутри try — тогда исключение из `process` поймал бы except, что может быть нежелательно.

**Конкретный пример.** Допустим, `risky_operation()` может выбросить `ValueError`. При успехе хотим вызвать `process(result)`. Главное: **исключения из else не ловятся этим except**. Если `process(result)` в else выбросит `KeyError` — он улетит наверх, `except ValueError` его не поймает. Если бы `process(result)` был в try — то же самое (ValueError ловит только ValueError). Зачем тогда else? Два плюса: (1) семантика — «else = только при успехе», код чище; (2) если в except ловите несколько типов (`except (ValueError, TypeError)`) или общий `except Exception`, то код в else не будет пойман — он выполняется «вне зоны перехвата» try/except.

#### 6.2.3. Scope comprehensions

В Python 3 переменная цикла в **comprehension** (`[x for x in ...]`, `{x for x in ...}` — list/set/dict comprehension) **не утекает** наружу — у comprehension своя область видимости. После `[x for x in range(5)]` снаружи `x` остаётся прежним (или будет NameError, если не было).

```python
x = 99
lst = [x for x in range(5)]
print(x)   # 99 — в Py3. В Py2 было бы 4 (утечка переменной цикла)
```

**Почему это важно.**  
В Python 2 переменная цикла «утекала» — после comprehension она имела последнее значение. Это приводило к багам. В Python 3 comprehension ведёт себя как вложенная функция — своя область видимости, переменная цикла снаружи не видна.

**Что может быть непонятно (scope comprehensions).**  
«Почему x остаётся 99?» `[x for x in range(5)]` создаёт временную «внутреннюю» переменную x для цикла. Она живёт только внутри comprehension. Снаружи переменная x (равная 99) не трогается. Comprehension как бы «изолирован» — его x не пересекается с внешним. «Comprehension» = компактная запись цикла: `[x*2 for x in lst]` то же, что `result = []; for x in lst: result.append(x*2)`.
