### 4.1. Производительность и профилирование (§89)

#### 4.1.1. Цель раздела

Научиться **измерять**, а не гадать: использовать cProfile, timeit, tracemalloc и понимать, когда переходить на PyPy/Cython/Numba. «Преждевременная оптимизация — корень зла»: сначала измеряем (cProfile, timeit), потом оптимизируем. Иначе можно тратить время на то, что на общее время почти не влияет.

#### 4.1.2. Термины

| Термин | Определение простыми словами |
|--------|------------------------------|
| **Профилирование** | Замер времени по функциям/строкам — где программа проводит больше всего времени. Нужно, чтобы найти «узкие места» (bottleneck) — те функции, на которые уходит основное время. |
| **Бенчмарк** | Замер времени одного фрагмента кода (например, «сколько времени занимает append в список»). |
| **Профилирование памяти** | Отслеживание, сколько памяти выделяется и где — помогает найти утечки. |

#### 4.1.3. cProfile — пошагово, как использовать

**Что делает cProfile.**  
Записывает каждый вызов функции: кто вызвал, сколько раз, сколько времени заняло. В результате видно, какие функции «съедают» время.

**Базовый пример с разбором вывода:**

```python
import cProfile
import pstats

def slow_func():
    total = 0
    for i in range(1000000):
        total += i
    return total

# Запускаем и сохраняем в файл
cProfile.run('slow_func()', filename='profile.stats')

# Читаем и сортируем по cumulative time (время вместе с вызванными функциями)
stats = pstats.Stats('profile.stats')
stats.sort_stats('cumulative')
stats.print_stats(10)  # топ 10 функций
```

**Как читать вывод cProfile (пример):**

```
   ncalls  tottime  percall  cumtime  percall filename:lineno(function)
        1    0.050    0.050    0.050    0.050 script.py:5(slow_func)
  1000001    0.030    0.000    0.030    0.000 {built-in method builtins.range}
```

- **ncalls** — сколько раз вызвана функция. Например, `range(1000000)` вызовет `range` один раз, но внутренний итератор может вызывать вспомогательные функции — поэтому иногда ncalls больше ожидаемого (1000001 = 1 вызов range + 1000000 вызовов next при итерации).
- **tottime** — время, проведённое **внутри** самой функции (без вызванных). Если функция вызывает других — их время сюда не входит.
- **cumtime** — время **вместе с вызванными** функциями. «Сколько всего ушло, пока я выполнялся, включая всех, кого я позвал». Это главный показатель для поиска «узких мест» — смотрите на функции с наибольшим cumtime.

**Что может быть непонятно (cumtime vs tottime).** Если `main()` имеет cumtime 10 сек и tottime 0.01 сек — значит почти всё время ушло в функции, которые вызвал `main`. Смотрите список callees (кого вызвал main) и ищите там bottleneck. Если tottime большой — сама функция долго выполняется (циклы, вычисления).
- **percall** — tottime/cumtime, делённое на ncalls. «Среднее время на один вызов».

**Из командной строки (без изменения кода):**

```bash
python -m cProfile -o output.prof script.py   # профилировать script.py
python -m pstats output.prof                 # анализировать
# В pstats (интерактивная оболочка) введите:
# sort cumulative   — сортировать по cumtime
# stats 20          — показать топ 20 функций
# help              — список команд
```

#### 4.1.4. timeit — микробенчмарки (подробно)

**Зачем.**  
Сравнить два варианта кода: «что быстрее — список или генератор?» Timeit запускает код много раз и усредняет время, чтобы уменьшить влияние случайных факторов (другие процессы, кеш и т.д.).

**Зачем нужен `setup`.**  
Setup — код, который выполняется **один раз** в начале (перед серией прогонов), чтобы подготовить окружение. Время на setup **не включается** в замер — меряется только выполнение `stmt`. Например, `setup='lst = []'` создаёт пустой список один раз; затем `stmt='lst.append(1)'` выполняется много раз (number=1000000). Список будет расти — для append это обычно приемлемо (амортизированная сложность O(1)). Но если важно мерять «один append в пустой список», можно включить создание списка в сам stmt: `stmt='lst=[]; lst.append(1)'`.

**Что может быть непонятно (timeit).**  
«Setup считается во время?» Нет. timeit замеряет только выполнение `stmt`. Setup нужен, чтобы переменные (lst и т.д.) существовали в области видимости при выполнении stmt.

```python
import timeit

# Строка кода, которую меряем
code = '"-".join(str(n) for n in range(100))'
# number=10000 — выполнить 10000 раз
t = timeit.timeit(code, number=10000)
print(t)

# С setup — код, который выполняется один раз в начале (подготовка окружения)
t = timeit.Timer('lst.append(1)', setup='lst = []')
print(t.timeit(number=1000000))  # сколько секунд на 1 млн append
```

**Из командной строки:**

```bash
python -m timeit "'-'.join(str(n) for n in range(100))"
```

#### 4.1.5. tracemalloc — профилирование памяти (пошагово)

**Когда нужно.**  
Программа «жрёт» память — непонятно где. tracemalloc показывает, какие строки кода выделяют больше всего памяти.

```python
import tracemalloc

tracemalloc.start()
# Ваш код
my_list = [i ** 2 for i in range(1000000)]

current, peak = tracemalloc.get_traced_memory()
print(f"Сейчас: {current / 1024 / 1024:.2f} MB")
print(f"Пик:    {peak / 1024 / 1024:.2f} MB")

# Топ выделений по файлам/строкам
snapshot = tracemalloc.take_snapshot()
top = snapshot.statistics('lineno')
for stat in top[:5]:
    print(stat)

tracemalloc.stop()
```

**Как читать вывод `statistics('lineno')`.**  
Каждая строка — файл, номер строки и размер выделенной памяти. Пример:

```
script.py:8: 45.2 MiB
```

Означает: в `script.py` на строке 8 выделено 45.2 MiB. Ищите строки с самыми большими числами — там основной расход памяти.

**Что может быть непонятно (tracemalloc).**  
- «current vs peak?» current — сколько памяти занято **сейчас**; peak — **максимум** за всё время трассировки. Если программа освободила память — current меньше peak. peak помогает понять «сколько максимум жрало».  
- «Когда вызывать take_snapshot()?» После того участка кода, который хотите проанализировать. snapshot фиксирует «моментальный снимок» выделений на этот момент.

#### 4.1.6. faulthandler — traceback при segfault

При падении C-расширения или segfault обычно видно только «Segmentation fault» без стека вызовов. faulthandler «подписывается» на сигналы и при падении выводит Python traceback в stderr — так можно увидеть, в каком месте Python-кода произошёл вызов в C, который упал.

**Когда использовать.** Подключайте faulthandler в начале скрипта, когда отлаживаете C-расширения, ctypes или NumPy — иначе при segfault вы не поймёте, откуда идёт вызов.

```python
import faulthandler
faulthandler.enable()
# или: python -X faulthandler script.py
```

#### 4.1.7. py-spy, memory_profiler, memray, scalene — подробно

**py-spy**  
Подключается к уже запущенному процессу по PID, **без изменения кода**. Удобно для production — процесс уже работает, вы «подглядываете» извне.

```bash
pip install py-spy
# Запустите свой скрипт в фоне (или он уже работает)
python script.py &
# Узнайте PID процесса (например, 12345)
py-spy top --pid 12345      # интерактивный «топ» по функциям
py-spy dump --pid 12345     # текущий Python traceback (стек вызовов)
py-spy record -o profile.svg --pid 12345  # записать профиль, сохранить в SVG
```

**Когда использовать:** процесс «висит» в production, нужно понять, где он проводит время, без перезапуска с профилированием.

---

**memory_profiler**  
Декоратор `@profile` — показывает расход памяти **по строкам** внутри функции.

```python
# pip install memory_profiler
from memory_profiler import profile

@profile
def load_data():
    data = [i ** 2 for i in range(1_000_000)]  # строка с большим выделением
    return sum(data)

load_data()
```

Запуск: `python -m memory_profiler script.py`. Вывод — каждая строка с MiB, прирост и т.д.

**Когда использовать:** функция «жрёт» память, нужно увидеть, какая именно строка выделяет больше всего.

---

**memray**  
Детальный анализатор памяти от Bloomberg: утечки, allocation timeline, flamegraph.

```bash
pip install memray
memray run script.py              # запустить с трассировкой
memray run -o output.bin script.py   # сохранить в файл для анализа
memray stats output.bin            # статистика
memray flamegraph output.bin       # создать flamegraph (HTML)
memray tree output.bin             # дерево выделений
```

**Когда использовать:** подозрение на утечку памяти; нужно понять, какие объекты не освобождаются и кто их держит. Графики помогают визуализировать «кто что создал».

---

**scalene**  
Профилировщик CPU + памяти + GPU по **строкам кода**. Показывает, какая строка сколько времени/памяти занимает — без ручной расстановки замеров.

```bash
pip install scalene
scalene script.py                 # CPU + память по строкам
scalene --cpu-only script.py      # только CPU
scalene --memory-only script.py   # только память
# Вывод — таблица: файл, строка, % CPU, % памяти
```

**Когда использовать:** нужно быстро увидеть «горячие» строки — и по времени, и по памяти. Удобнее, чем расставлять `@profile` или `tracemalloc` вручную. Хорош для первых прогонов: «где вообще тормозит?».

#### 4.1.8. Когда использовать PyPy, Cython, Numba

| Инструмент | Суть | Когда применять |
|------------|------|-----------------|
| **PyPy** | JIT-интерпретатор (JIT = Just-In-Time — компиляция «на лету» во время выполнения) | Чистый Python, много циклов; часть C-расширений может не работать |
| **Cython** | Python → C компиляция | Критичные циклы, интеграция с C |
| **Numba** | JIT для числовых функций | NumPy-подобные вычисления, декоратор `@njit` |

Сначала **профилируйте** — убедитесь, что оптимизируете то, что реально тормозит.

#### 4.1.9. Список vs кортеж, генераторы vs списки

- **Кортеж** — создаётся быстрее, меньше памяти; неизменяемый. Для константных данных (координаты, конфиг) — лучше. Нельзя изменить элемент.
- **Генератор** — не хранит все элементы в памяти, выдаёт по одному. Для потоковой обработки больших объёмов — экономия памяти. Нельзя обратиться по индексу (`gen[5]` — ошибка), проход только один раз. После прохода генератор «исчерпан».

```python
# Список — все 1_000_000 элементов в памяти
lst = [i ** 2 for i in range(1_000_000)]

# Генератор — круглые скобки () вместо []! По одному элементу, память константная
gen = (i ** 2 for i in range(1_000_000))
for x in gen:
    process(x)  # обрабатываем по одному
# gen исчерпан, повторно итерировать нельзя
```

**Что может быть непонятно (генератор).**  
- «Почему «исчерпан»?» Генератор выдаёт элементы по одному и «помнит» позицию. Когда все элементы выданы, повторный вызов `next(gen)` вызовет `StopIteration`. Генератор не «сбрасывается» — чтобы пройти снова, нужно создать новый: `gen2 = (x for x in range(10))`.  
- «Когда генератор лучше списка?» Когда элементов много и обрабатывать их нужно по одному (файл построчно, поток данных). Список загрузит всё в память; генератор держит только текущий элемент.

### 4.2. Отладка (§89a)

#### 4.2.1. pdb — пошаговая отладка (подробно)

**Как начать.**  
Вставьте в нужное место `breakpoint()` (Python 3.7+) или `import pdb; pdb.set_trace()`. При выполнении программа остановится, и вы окажетесь в интерактивной оболочке pdb. Вы увидите приглашение `(Pdb)` — это значит, что отладчик ждёт вашей команды. Вводите `n`, `s`, `p x` и т.д., нажимайте Enter.

```python
def buggy():
    x = 1
    y = 2
    breakpoint()   # <-- выполнение остановится здесь
    return x + y

buggy()
```

**Что делать в pdb — пошагово.**

1. **Понять, где вы:** `l` (list) — показать код вокруг текущей строки.
2. **Проверить переменные:** `p x`, `p x + y` — напечатать значение выражения.
3. **Шаг вперёд без захода в функции:** `n` (next) — выполнить текущую строку и перейти к следующей.
4. **Шаг с заходом в функцию:** `s` (step) — если на строке вызов функции, зайти внутрь.

**Разница `n` и `s`.** Строка `x = len(lst)`. Нажали `n` — выполнится `len(lst)`, вы не зайдёте внутрь `len`, перейдёте к следующей строке. Нажали `s` — зайдёте внутрь `len`, будете отлаживать его построчно. Для стандартных функций обычно `n` — иначе утонете в коде. `s` — когда баг в вашей функции, которую вызываете.

5. **Продолжить до следующего breakpoint:** `c` (continue).
6. **Понять стек вызовов:** `w` (where) — кто кого вызвал.
7. **Подняться/опуститься по стеку:** `u` (up), `d` (down).
8. **Выход:** `q` (quit).

**Полная шпаргалка команд:**

| Команда | Полное имя | Действие |
|---------|------------|----------|
| `n` | next | Следующая строка, не заходя в вызовы |
| `s` | step | Шаг с заходом в функцию |
| `c` | continue | До следующего breakpoint |
| `l` | list | Показать код вокруг текущей строки |
| `p x` | print | Вывести значение выражения |
| `pp x` | pretty print | Вывести с форматированием (для списков/словарей) |
| `w` | where | Стек вызовов |
| `u` | up | Подняться по стеку |
| `d` | down | Опуститься по стеку |
| `q` | quit | Выход из отладчика |

**Пример сессии:**

```
> script.py(5)buggy()
-> return x + y
(Pdb) p x
1
(Pdb) p y
2
(Pdb) n
--Return--
> script.py(5)buggy()->3
-> return x + y
(Pdb) c
```

**Типичный порядок действий при отладке.**

1. Найти место, где баг проявляется (по traceback или логике). **Как читать traceback:** сверху вниз — цепочка вызовов. Самая верхняя строка — где ошибка произошла (там указан файл, номер строки, функция). Строки ниже — «кто кого вызвал»: следующая функция вызвала ту, что выше. Читайте снизу вверх: «main вызвал foo, foo вызвала bar, bar вызвала qux — и в qux случилась ошибка». Последняя строка — тип ошибки (ValueError, KeyError и т.д.) и сообщение.
2. Вставить `breakpoint()` прямо перед этим местом.
3. Запустить скрипт. Он остановится на breakpoint.
4. `l` — посмотреть, какой код выполняется.
5. `p переменная` — проверить значения переменных. Не те, что ожидали?
6. `n` — шаг вперёд (не заходя в функции). Повторять, пока не найдёте строку, где всё ломается.
7. Если нужно зайти в функцию — `s` вместо `n`.
8. `w` — посмотреть стек вызовов (кто кого вызвал).
9. `c` — продолжить до следующего breakpoint или до конца. Или `q` — выйти.

#### 4.2.2. PYTHONBREAKPOINT — отключить breakpoint в production

Иногда в коде остаются `breakpoint()`, а скрипт запускают в production. Чтобы они не останавливали программу, задайте переменную окружения:

```bash
PYTHONBREAKPOINT=0 python script.py   # breakpoint ничего не делает — программа не остановится
PYTHONBREAKPOINT=ipdb.set_trace python script.py  # использовать ipdb вместо pdb (цветной, автодополнение)
```

#### 4.2.3. sys.settrace и sys.setprofile

Низкоуровневые хуки — вызываются при каждом вызове/возврате функции. `settrace` — при каждой строке кода; `setprofile` — при вызове/возврате функции. Используются для трассировки и кастомного профилирования. **frame** — объект, представляющий кадр выполнения (функция, локальные переменные). `frame.f_code.co_name` — имя текущей функции.

```python
import sys

def trace_calls(frame, event, arg):
    if event == 'call':
        print(f"Вызов: {frame.f_code.co_name}")
    return trace_calls

sys.settrace(trace_calls)
my_function()  # будет печатать все вызовы
sys.settrace(None)
```

#### 4.2.4. debugpy — отладка в VS Code

`pip install debugpy`. В VS Code — «Run and Debug», выбор конфигурации Python. Можно подключаться к уже запущенному процессу (attach), ставить точки останова мышкой в редакторе (клик слева от номера строки), смотреть переменные в боковой панели. Удобнее pdb — не нужно вводить команды вручную, всё визуально.

## 5. Безопасность (§90, §90a, ReDoS)
