### Цель раздела

Понять, что такое **метакласс**, как он участвует в создании классов и зачем он нужен. Увидеть, в каких случаях действительно оправдано использование метаклассов, и когда достаточно `__init_subclass__`.

### Базовые термины

- **Метакласс** — класс, экземплярами которого являются **другие классы**.
- `type` — стандартный метакласс для всех классов в Python.
- `type(name, bases, namespace)` — функция/конструктор для создания нового класса.

### Как Python создаёт класс

Когда интерпретатор видит:

```python
class MyClass(Base1, Base2):
    x = 42

    def method(self):
        ...
```

Он грубо делает следующее:

1. Выполняет тело класса **как обычный блок кода**, формируя словарь `namespace`.
2. Вызывает **метакласс**:

   ```python
   MyClass = metaclass("MyClass", (Base1, Base2), namespace)
   ```

   По умолчанию `metaclass` — это `type`.

### Пример собственного метакласса

```python
class UpperAttrMeta(type):
    def __new__(mcls, name, bases, namespace):
        new_namespace = {
            (attr.upper() if not attr.startswith("__") else attr): value
            for attr, value in namespace.items()
        }
        return super().__new__(mcls, name, bases, new_namespace)


class MyClass(metaclass=UpperAttrMeta):
    x = 10

    def hello(self):
        return "hi"
```

Теперь:

```python
print(hasattr(MyClass, "x"))       # False
print(hasattr(MyClass, "X"))       # True
print(MyClass.X)                   # 10
```

Метакласс перехватил создание класса и изменил его словарь атрибутов.

### Где это реально применяется

Метаклассы используются, когда нужно:

- **автоматически регистрировать** подклассы в каком‑то реестре;
- выполнять **валидацию** определённых атрибутов класса;
- генерировать методы/атрибуты на лету (например, ORM‑ы вроде Django);
- реализовывать продвинутые **DSL‑ы** (domain specific languages).

Важно понимать, **что именно мы перехватываем**:

- `__new__` метакласса — момент **создания класса** (можно модифицировать `namespace`, проверять атрибуты, добавлять методы);
- `__init__` метакласса — инициализация уже созданного класса;
- `__call__` метакласса — момент **создания экземпляра класса** (можно влиять на конструирование объектов).

Пример: реестр плагинов.

```python
class PluginMeta(type):
    registry: dict[str, type] = {}

    def __new__(mcls, name, bases, namespace):
        cls = super().__new__(mcls, name, bases, namespace)
        if not name.startswith("Base"):
            PluginMeta.registry[name] = cls
        return cls


class BasePlugin(metaclass=PluginMeta):
    ...


class JsonPlugin(BasePlugin):
    ...


class XmlPlugin(BasePlugin):
    ...


print(PluginMeta.registry)
# {'JsonPlugin': <class '...JsonPlugin'>, 'XmlPlugin': <class '...XmlPlugin'>}
```

### `__init_subclass__` как альтернатива

Во многих случаях достаточно **хука в базовом классе**:

```python
class BasePlugin:
    registry: dict[str, type] = {}

    def __init_subclass__(cls, **kwargs):
        super().__init_subclass__(**kwargs)
        if cls is not BasePlugin:
            BasePlugin.registry[cls.__name__] = cls
```

Теперь любой наследник `BasePlugin` автоматически регистрируется — **без метаклассов**.

Когда достаточно `__init_subclass__`:

- вы контролируете поведение **только для одной иерархии** (один базовый класс и его наследники);
- вам не нужно навешивать один и тот же метакласс на множество несвязанных иерархий;
- нужно просто отреагировать на факт появления подкласса (регистрация, проверка атрибутов, установка значений по умолчанию).

Когда действительно нужен метакласс:

- вы хотите **переопределить поведение создания классов в целом** (например, реализовать «singleton класс» через метакласс);
- требуется единый механизм для разных иерархий с общими правилами;
- вы пишете фреймворк/библиотеку, где пользователи будут определять свои классы, а вы хотите управлять их структурой.

### Запомните

- Метаклассы — мощный, но **редко действительно необходимый** инструмент.
- Часто вместо метакласса достаточно `__init_subclass__` или **простого фабричного метода**.
- Если вы пишете что‑то вроде ORM, фреймворка или сложной библиотеки — метаклассы могут быть уместны; в бизнес‑логике приложений почти всегда можно обойтись без них.

## 65a. Все вариации связей в ООП

Эта секция систематизирует **виды связей между классами и объектами**, о которых уже шла речь.

При чтении этого раздела полезно задавать себе вопросы:

- «Кто **владеет** кем?»
- «Кто может существовать **сам по себе**, а кто — только как часть другого?»
- «Как легко заменить одну сторону связи другой реализацией?»

### Композиция (Composition)

- «Содержит» — объект владеет другим; жизненный цикл **связан**.
- Внутренний объект обычно создаётся в конструкторе и не живёт без владельца.

```python
class Engine:
    def start(self):
        print("Engine start")


class Car:
    def __init__(self):
        self.engine = Engine()  # создаём внутри

    def drive(self):
        self.engine.start()
        print("Car moves")
```

Признаки композиции:

- жизненный цикл вложенного объекта **полностью контролируется** владельцем;
- при удалении `Car` движок больше «не нужен» (часто его тоже можно считать удалённым);
- владелец обычно создаёт вложенный объект **внутри себя** (в `__init__` или отдельном фабричном методе).

### Агрегация (Aggregation)

- «Имеет ссылку» — связь слабее; объект может существовать отдельно.

```python
class Student:
    def __init__(self, name: str):
        self.name = name


class Course:
    def __init__(self, name: str):
        self.name = name
        self.students: list[Student] = []

    def add_student(self, student: Student):
        self.students.append(student)
```

`Student` может существовать **без** `Course`, сменить курс и т.д.

Признаки агрегации:

- объекты могут существовать **независимо** друг от друга;
- один и тот же объект может входить в состав **нескольких агрегатов** (студент может учиться на нескольких курсах);
- агрегат обычно **не отвечает** за создание/уничтожение вложенных объектов, он только хранит ссылки.

### Ассоциация (Association)

- Общая связь между классами, часто двунаправленная.

```python
class Author:
    def __init__(self, name: str):
        self.name = name
        self.books: list["Book"] = []


class Book:
    def __init__(self, title: str, author: Author):
        self.title = title
        self.author = author
        author.books.append(self)
```

Ассоциация может быть:

- **однонаправленной** — только одна сторона знает о другой (например, `Order` хранит ссылку на `Customer`, а `Customer` не знает обо всех заказах);
- **двунаправленной** — обе стороны хранят ссылки друг на друга (как в примере с `Author` и `Book`).

### Наследование (Inheritance)

- Связь **is‑a** (см. §58).

```python
class Animal: ...

class Dog(Animal): ...
```

### Делегирование (Delegation)

- Один объект **передаёт вызов** другому объекту.

```python
class ListProxy:
    def __init__(self, inner: list):
        self._inner = inner

    def append(self, item):
        print("Logging append")
        self._inner.append(item)

    def __getattr__(self, name):
        # Делегируем остальное внутреннему объекту
        return getattr(self._inner, name)
```

### Миксин (Mixin)

- Класс для добавления **поведения**, обычно используется в **множественном наследовании**.

```python
class JsonMixin:
    def to_json(self) -> str:
        import json
        return json.dumps(self.__dict__)


class User(JsonMixin):
    def __init__(self, name: str):
        self.name = name
```

Рекомендации по миксинам:

- миксин **не должен** иметь самостоятельного смысла как отдельный тип («пользователь»/«заказ» и т.п.);
- обычно миксины:
  - не хранят сложное состояние;
  - не определяют `__init__` (или делают это очень аккуратно и кооперативно через `super()`);
  - содержат **поведение**, которое осмысленно «добавить» к разным классам (логирование, сериализация, кеширование).

### Трейт (Trait)

- В других языках (Rust, Scala) — отдельная сущность, похожая на интерфейс с реализацией.
- В Python аналогом чаще всего являются **миксины** или протоколы + миксины.

### Интерфейс (Interface)

- Набор методов, которые должен реализовать класс.
- В Python: `abc.ABC` + `@abstractmethod`, либо `typing.Protocol`.

### Абстрактный класс (Abstract Base Class)

- См. §61. Базовый класс, задающий **часть реализации** + **абстрактные методы**.

### Параметрический полиморфизм

- Обобщённые типы (`Generic[T]`, `TypeVar`).

```python
from typing import Generic, TypeVar

T = TypeVar("T")


class Box(Generic[T]):
    def __init__(self, value: T):
        self.value = value
```

### Ad‑hoc полиморфизм

- Перегрузка операторов (`__add__`, `__radd__`) и `functools.singledispatch`.

```python
from functools import singledispatch


@singledispatch
def dump(value):
    print("Default:", value)


@dump.register
def _(value: int):
    print("Int:", value)
```

### Dependency Injection

- Зависимости передаются **извне**, а не создаются внутри.

```python
class Repository(ABC):
    @abstractmethod
    def get_user(self, user_id: int) -> dict:
        ...


class UserService:
    def __init__(self, repo: Repository):
        self._repo = repo  # внедрение зависимости
```

### Запомните

- Связь между классами — это не только наследование.
- Чем более **слабая** связь (агрегация, композиция, делегирование), тем проще менять и тестировать код.

### Упражнения для самостоятельной работы

1. Нарисуйте диаграмму для небольшой части своего проекта и попытайтесь классифицировать связи:
   - где у вас композиция (жизненный цикл вложенных объектов жёстко связан с владельцем);
   - где агрегация (объекты могут жить независимо и «переходить» между владельцами);
   - где ассоциация (просто ссылки между сущностями).
2. Возьмите пример, где вы сейчас используете наследование, и попробуйте переформулировать его в терминах композиции/агрегации/делегирования:
   - оцените, стало ли проще объяснять этот кусок кода коллеге;
   - подумайте, как будет меняться система при добавлении новых требований.
3. Подумайте, где в вашем коде логичны миксины (поведение, которое хочется «примешать» к разным классам), и сформулируйте требования к этим миксинам:
   - какое минимальное состояние они допускают;
   - какие ожидания предъявляют к классам‑носителям (например, наличие `self.logger`).
