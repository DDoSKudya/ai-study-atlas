# Часть XII. ООП углублённо

> **Цель:** разобраться в продвинутых возможностях объектно‑ориентированного программирования в Python:
> 
> - понять, **как именно работает наследование и MRO** (Method Resolution Order);
> - научиться **осознанно выбирать** между наследованием и композицией;
> - использовать **абстрактные базовые классы и протоколы** для описания контрактов;
> - применять **SOLID и принципы ООП** на практике, а не только «знать расшифровку букв»;
> - грамотно использовать **инкапсуляцию и сокрытие**, понимать соглашения Python;
> - познакомиться с **метаклассами** и смежными приёмами (`__init_subclass__`);
> - видеть разные виды **связей между объектами** (композиция, агрегация, ассоциация, делегирование, миксины, трейты, интерфейсы, DI);
> - узнать дополнительные **ООП‑техники и паттерны** (Object Pool, Flyweight, Proxy, Decorator и др.).
> 
> Всё это — **простым языком**, но с сохранением корректной терминологии.

---

## Общая картина: зачем всё это

Базовое ООП (классы, объекты, методы, простое наследование) — это только первый слой. В реальных проектах быстро появляются вопросы:

- «Почему `super()` вызывает методы **во всех** базовых классах, а не только у ближайшего?»
- «Почему изменение порядка классов в списке наследования **ломает поведение**?»
- «Когда лучше **наследоваться**, а когда — просто **хранить** другой объект внутри (композиция)?»
- «Как задать **контракт**, чтобы не допускать неправильных реализаций?»
- «Можно ли описать интерфейс так, чтобы он **не требовал наследования**, а проверял только наличие методов?»
- «Зачем нужны метаклассы и почему ими пугают?»

Эта часть отвечает именно на такие вопросы.

## Краткая шпаргалка (суть Часть XII)

| Понятие | Суть |
|---------|------|
| **Наследование** | «Является» (is‑a). Один класс расширяет/специализирует другой. Может быть одиночным и множественным. |
| **MRO (Method Resolution Order)** | Порядок, в котором Python ищет атрибуты в иерархии классов. Рассчитывается алгоритмом **C3‑линеаризации**. |
| **Кооперативное наследование** | Стиль, при котором все классы иерархии вызывают `super()`, а не обращаются к конкретному родителю. |
| **Композиция** | «Содержит» (has‑a). Объект хранит другой объект как часть своего состояния. |
| **Агрегация** | Ослабленная композиция: объект **ссылается** на другой, но не владеет его жизненным циклом. |
| **Абстрактный класс (ABC)** | Класс, который задаёт **контракт** (абстрактные методы) и **не может** быть инстанциирован напрямую. |
| **Протокол (Protocol)** | Структурный тип: важна **форма** (набор методов), а не конкретная иерархия наследования. |
| **SOLID** | Набор практических принципов проектирования ООП‑систем (SRP, OCP, LSP, ISP, DIP). |
| **Инкапсуляция** | Сокрытие внутреннего состояния/реализации за стабильным интерфейсом. В Python — соглашения (`_attr`, `__attr`). |
| **Метакласс** | «Класс для классов». Управляет тем, **как создаются** и настраиваются классы. |
| **__init_subclass__** | Хук, который вызывается при наследовании. Альтернатива многим сценариям с метаклассами. |
| **Dependency Injection** | Передача зависимостей **извне**, а не создание их внутри класса. Улучшает тестируемость и гибкость. |

Дальше — по разделам §58–65b.

## Маршрут изучения

Рекомендуемый порядок:

1. **§58–59. Наследование и MRO.** Сначала чётко понять, как Python ищет методы и как работает `super()`.
2. **§60. Композиция vs наследование.** Научиться задавать себе вопрос: «Тут действительно is‑a или скорее has‑a?».
3. **§61–62. Абстрактные классы и протоколы.** Увидеть разницу между **номинальным** (наследование) и **структурным** (Protocol) подтипированием.
4. **§63–64. SOLID, инкапсуляция, сокрытие.** Связать ООП‑механику с архитектурными принципами.
5. **§65–65a–65b. Метаклассы и связи.** Понять, как устроено создание классов, и какие существуют типы связей и паттернов.
