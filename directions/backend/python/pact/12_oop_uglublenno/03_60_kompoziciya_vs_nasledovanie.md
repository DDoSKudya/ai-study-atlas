### Цель раздела

Научиться **осознанно решать**, когда использовать наследование, а когда — композицию/агрегацию. Избегать «бесконечных иерархий» и избыточных базовых классов.

### Базовые термины

- **Композиция (composition)** — объект **содержит** другие объекты как часть своего состояния.
- **Агрегация (aggregation)** — ослабленная композиция: объект **ссылается** на другие, но не владеет их жизненным циклом.
- **Принцип «Favor composition over inheritance»** — рекомендация чаще использовать композицию, а не наследование.

### Наследование: когда оно действительно нужно

Наследование оправдано, если:

1. Связь между типами — **is‑a**:
   - `Square` **является** частным случаем `Rectangle` (хотя и тут есть нюансы — см. LSP в §63).
   - `FileLogger` — частный случай абстрактного `Logger`.
2. От базового класса ожидают общие **инварианты и поведение**:
   - базовый класс гарантирует набор методов, семантику, контракт.
3. Есть необходимость **полиморфизма**:
   - код принимает аргумент типа `BaseClass`, а фактически передают экземпляры разных наследников.

Практический тест:

- Если заменить тип фразы «Х является Y» на «Х **имеет** Y», и фраза стала звучать естественнее — скорее всего, нужна **композиция**, а не наследование.
- Если вы **не планируете** полиморфизм (у вас всего один подкласс и не видно причин появляться другим) — наследование, вероятно, преждевременно.

### Композиция: пример

Вместо:

```python
class LoggedList(list):
    def append(self, item):
        print(f"Adding {item}")
        super().append(item)
```

Можно:

```python
class ListLogger:
    def __init__(self):
        self._items: list = []

    def append(self, item):
        print(f"Adding {item}")
        self._items.append(item)

    def __iter__(self):
        return iter(self._items)

    def __len__(self):
        return len(self._items)
```

Плюсы композиции:

- не тянутся **все особенности `list`** (например, неочевидные методы);
- легче контролировать инварианты;
- можно заменять внутренний контейнер (`list` → `deque`) без изменения внешнего интерфейса.

Частый паттерн: класс‑обёртка, который:

- хранит один или несколько объектов как атрибуты (`self._storage`, `self._client`);
- реализует **узкий публичный интерфейс** (пара методов), делегируя детали внутренним объектам;
- может в любой момент быть переписан на другую реализацию без изменения внешнего кода.

### Агрегация: пример

```python
class Employee:
    def __init__(self, name: str):
        self.name = name


class Department:
    def __init__(self, name: str, employees: list[Employee] | None = None):
        self.name = name
        self.employees = employees or []

    def add_employee(self, employee: Employee):
        self.employees.append(employee)
```

Здесь:

- `Department` **не владеет** жизненным циклом `Employee`;
- сотрудник может перейти в другой отдел;
- это **агрегация**, а не жёсткая композиция.

### Принцип «композиция вместо наследования»

Практически:

- Если вас интересует **поведение** и вы хотите собрать его из кусочков — чаще подходит **композиция** + делегирование.
- Если нужен **контракт типа** (is‑a) и полиморфизм — тогда **наследование** (часто от абстрактного базового класса или протокола).

### Запомните

- Наследование связывает типы **навсегда** (распилить иерархию потом сложно).
- Композиция даёт большую **гибкость** и упрощает тестирование.
- Если сомневаетесь — сначала попробуйте композицию; наследование добавляйте, когда у вас появился чёткий **контракт базового типа**.

### Упражнения для самостоятельной работы

1. Найдите в своём коде класс, который наследуется от стандартного контейнера (`list`, `dict`, `set`) только ради добавления логирования/валидации.\
   Попробуйте переписать его через композицию, как `ListLogger`, и оцените, какие операции стало легче/сложнее контролировать.
2. Смоделируйте два варианта дизайна:
   - `class CachedApiClient(ApiClient)` — логика кеша реализована через наследование;
   - `class CachedApiClient: def __init__(self, client: ApiClient, cache: Cache): ...` — кеш реализован через композицию.\
   Обсудите для себя плюсы/минусы: тестируемость, контролируемость зависимостей, гибкость замены реализации.
3. Придумайте пример, где наследование **оправдано** (чистый is‑a и полиморфизм), и пример, где наследование выглядит естественно, но на самом деле лучше композиция. Попробуйте формально сформулировать причину выбора.
