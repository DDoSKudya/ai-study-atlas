### Цель раздела

Понять разницу между **номинальным** и **структурным** подтипированием, научиться использовать `typing.Protocol` для описания интерфейсов без жёсткого наследования, а также увидеть, как протоколы помогают сделать код более гибким и типобезопасным.

### Базовые термины

- **Утиная типизация (duck typing)** — «если что‑то выглядит как утка, крякает как утка и ходит как утка, то это утка».\
  В Python это значит: важна **наличие методов/атрибутов и их поведение**, а не конкретный тип.
- **Номинальное подтипирование** — объект считается подтипом, если он **наследуется** от базового класса.
- **Структурное подтипирование** — объект считается подтипом, если он **соответствует структуре** (набору методов), даже без наследования.
- **Protocol** — механизм структурного подтипирования в модуле `typing`.

### Пример утинай типизации без типов

```python
class FileLike:
    def __init__(self, name: str):
        self.name = name

    def read(self) -> str:
        return "data"


def process_stream(stream):
    # Требуется только, чтобы у stream был метод read().
    data = stream.read()
    print(f"Got: {data}")
```

Функция `process_stream` не интересуется **конкретным типом** объекта, ей нужен только метод `read`.

Плюсы чистой утинай типизации:

- минимум ограничений в рантайме — код гибкий;
- удобно для быстрой разработки и прототипов.

Минусы без типизации:

- статический анализатор/IDE **не может заранее сказать**, что вы передали объект без метода `read`;
- сложнее документировать ожидания: программист должен «вычитать из кода», какие методы ожидаются.

### Протоколы в type hints

```python
from typing import Protocol


class Reader(Protocol):
    def read(self) -> str:
        ...


def process_stream(stream: Reader) -> None:
    data = stream.read()
    print(f"Got: {data}")
```

Теперь любой объект, у которого есть метод `read() -> str`, **совместим с типом** `Reader` **без наследования**:

```python
class FileReader:
    def read(self) -> str:
        return "file data"


class NetworkReader:
    def read(self) -> str:
        return "network data"


process_stream(FileReader())
process_stream(NetworkReader())
```

Статический анализатор (mypy, pyright и т.п.):

- увидит, что `FileReader` и `NetworkReader` **структурно соответствуют** `Reader`;
- будет ругаться, если вы передадите объект **без** метода `read()` или с несовместимой сигнатурой.

То есть мы формализовали «утиную типизацию» так, что её может проверять инструмент.

### `runtime_checkable` и isinstance

```python
from typing import Protocol, runtime_checkable


@runtime_checkable
class SupportsClose(Protocol):
    def close(self) -> None:
        ...


def safe_close(obj) -> None:
    if isinstance(obj, SupportsClose):
        obj.close()
```

С `@runtime_checkable` можно использовать `isinstance(obj, ProtocolClass)` — будет проверена **структура**, а не иерархия наследования.

Ограничения:

- `runtime_checkable` проверяет **только наличие атрибутов** (методов/свойств) по имени;
- не проверяется возвращаемый тип, количество параметров и т.п. — это остаётся на совести разработчика и статического анализатора.

### Протоколы в стандартной библиотеке

Многие «классические» ABC были продублированы протоколами:

- `typing.Iterable`, `Iterator`, `Sized`, `Container`, `Mapping`, `Sequence` и др.

Это позволяет:

- писать аннотации, не завязанные на конкретные классы;
- описывать интерфейсы для **утиной типизации** в статическом анализе.

Например, если функция ожидает `Iterable[str]`, то ей можно передать:

- список `list[str]`,
- кортеж `tuple[str, ...]`,
- генератор,
- любой объект, который реализует метод `__iter__` и возвращает итерируемый по строкам результат.

Ей **не нужно** знать, от чего наследуется этот объект — только то, что он соответствует протоколу `Iterable[str]`.

### Сравнение ABC и Protocol

- **ABC (номинальный подход)**:
  - вы явно наследуетесь от базового класса;
  - ABC может содержать **часть реализации** (методы по умолчанию, общие поля);
  - проверка в рантайме через `isinstance(obj, BaseClass)`.
- **Protocol (структурный подход)**:
  - наследование **не обязательно**;
  - протокол обычно не содержит «реальной» логики, только сигнатуры;
  - основная сила — в **статическом анализе** (mypy/pyright), а не в рантайме.

Практика:

- если вам нужен **контракт + общая реализация** → чаще всего ABC;
- если нужен **контракт по поведению**, но вы не хотите навязывать иерархию наследования (особенно для внешнего кода/библиотек) → Protocol.

### Протоколы и обобщённые типы

Протоколы можно сочетать с `TypeVar`/`Generic`, описывая **«коллекцию чего‑то»**, не привязанную к конкретному типу:

```python
from typing import Iterable, Protocol, TypeVar

T = TypeVar("T")


class Averagable(Protocol):
    def __float__(self) -> float:
        ...


U = TypeVar("U", bound=Averagable)


def average(values: Iterable[U]) -> float:
    total = 0.0
    count = 0
    for v in values:
        total += float(v)
        count += 1
    return total / count
```

Здесь:

- `Averagable` — протокол «из него можно получить `float`»;
- функция `average` работает с **любыми типами**, реализующими этот протокол: `Decimal`, `Fraction`, свои классы чисел и т.п.

### Типичные ошибки при работе с протоколами

- Пытаться использовать Protocol как **обычный базовый класс с реализацией**:
  - лучше вынести реализацию в отдельный миксин/ABC, а Protocol оставить интерфейсом.
- Ожидать, что `Protocol` «заставит» рантайм выкинуть ошибку, если объект не соответствует:
  - реальная проверка — это задача статического анализатора; в рантайме будет обычный `AttributeError`, когда вы вызовете несуществующий метод.
- Смешивать в одном месте **жёсткое наследование** и **структурные протоколы**, создавая дублирующие контракты:
  - лучше определиться: либо вы проектируете вокруг ABC, либо вокруг протоколов, а не одновременно вокруг двух иерархий.

### Практика: переписываем код с неявной утинай типизацией

Шаги:

1. Найти функции вида:

   ```python
   def do_something(obj):
       obj.connect()
       obj.send("data")
   ```

2. Выделить протокол:

   ```python
   from typing import Protocol


   class Sender(Protocol):
       def connect(self) -> None: ...
       def send(self, data: str) -> None: ...
   ```

3. Аннотировать функцию:

   ```python
   def do_something(obj: Sender) -> None:
       obj.connect()
       obj.send("data")
   ```

4. Прогнать статический анализатор — он покажет места, где вы передаёте объекты без нужных методов.

Так вы документируете контракт и получаете раннее обнаружение ошибок.

### Запомните

- `abc.ABC` описывает контракт через **наследование** (номинальный подход).
- `typing.Protocol` описывает контракт через **структуру** (утиная типизация).
- В Python в рантайме и так работает утиной тип; протоколы помогают **статической проверке** и документированию интерфейсов.

### Упражнения для самостоятельной работы

1. Найдите функции в своём коде, которые принимают «что‑то с методом `send`/`execute`/`read`», но не аннотированы типами.\
   Для каждой такой функции:
   - опишите протокол с нужными методами;
   - аннотируйте параметр этим протоколом;
   - прогоните статический анализатор и посмотрите, какие ошибки он нашёл.
2. Возьмите протокол `Iterable[T]` и реализуйте свой класс, который:
   - хранит данные во внутреннем буфере;
   - реализует только метод `__iter__`;
   - при этом корректно работает в `for`‑цикле и функциях `list()`, `sum()` и т.п.
3. Попробуйте написать протоколы для двух уровней абстракции:
   - низкоуровневый, описывающий, например, «соединение с БД»;
   - высокоуровневый, описывающий «репозиторий пользователей».\
   Оцените, насколько проще становится тестирование, если подставлять фейковые реализации этих протоколов.
