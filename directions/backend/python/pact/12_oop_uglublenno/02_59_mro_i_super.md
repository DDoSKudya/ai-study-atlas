### Цель раздела

Понять, **как Python ищет атрибуты и методы** в иерархии классов, что такое **C3‑линеаризация** и как правильно использовать `super()` во множественном наследовании.

### Базовые термины

- **MRO (Method Resolution Order)** — порядок, в котором Python ищет атрибут при обращении через экземпляр или класс.
- **C3‑линеаризация** — алгоритм, который вычисляет MRO при множественном наследовании.
- **Кооперативное наследование** — стиль, при котором все классы вызывают `super()` и уважают MRO.

### Как посмотреть MRO

```python
class A: ...

class B(A): ...

class C(A): ...

class D(B, C): ...

print(D.__mro__)
print(D.mro())
```

Оба варианта выводят кортеж классов MRO, например:

```python
(
    <class '__main__.D'>,
    <class '__main__.B'>,
    <class '__main__.C'>,
    <class '__main__.A'>,
    <class 'object'>
)
```

### Правила C3‑линеаризации (простыми словами)

Алгоритм C3 гарантирует:

1. **Локальный приоритет**: если в объявлении `class D(B, C)` сначала идёт `B`, а потом `C`, то в MRO `B` будет перед `C`, если это не противоречит другим ограничениям.
2. **Монотонность (monotonicity)**: порядок для каждого родителя **сохраняется** в потомках; нельзя так изменить MRO, чтобы он «ломал» порядок у родительских классов.
3. MRO — это **линейное упорядочивание** всех классов в иерархии, совместимое со всеми частичными порядками, заданными наследованием.

Если для заданной иерархии **невозможно** построить MRO, Python выдаст ошибку:

```python
TypeError: Cannot create a consistent method resolution order (MRO) for bases ...
```

### `super()` и кооперативное наследование

Ключевая идея: **`super()` следует MRO**, а не «родительскому классу по имени».

```python
class A:
    def __init__(self):
        print("A.__init__")
        super().__init__()


class B(A):
    def __init__(self):
        print("B.__init__")
        super().__init__()


class C(A):
    def __init__(self):
        print("C.__init__")
        super().__init__()


class D(B, C):
    def __init__(self):
        print("D.__init__")
        super().__init__()


D()
```

Вывод:

```text
D.__init__
B.__init__
C.__init__
A.__init__
```

Обратите внимание:

- **Каждый класс** вызывает `super().__init__()`.
- `super()` идёт **строго по MRO**: D → B → C → A → object.
- Ни один `__init__` не вызывается дважды.

Если бы в одном из классов мы вызвали `A.__init__(self)` напрямую, мы бы:

- **нарушили** кооперативность;
- могли бы вызвать `A.__init__` несколько раз;
- сломали бы работу других базовых классов.

### Почему `super()` без аргументов

В классах нового стиля (все классы в Python 3) можно писать:

```python
class MyClass(Base):
    def method(self):
        super().method()
```

Внутри метода Python подставляет текущий класс и экземпляр. Это:

- проще и короче;
- уменьшает вероятность ошибок;
- по‑прежнему следует MRO.

### Типичные ошибки с MRO

- **Жёсткая привязка к конкретному родителю**:
  - `BaseClass.__init__(self)` вместо `super().__init__()`.
- Переопределение метода и **отсутствие вызова `super()`**:
  - в результате базовый код вообще не выполняется.
- Слишком сложные иерархии с несколькими ромбами, когда **даже разработчик не может объяснить MRO**.

### Мини‑разбор: как C3 строит MRO на примере

Возьмём ту же иерархию:

```python
class A: ...
class B(A): ...
class C(A): ...
class D(B, C): ...
```

Упростим до списков:

- локальный порядок для `D`: `[B, C]`;
- MRO для `B`: `[B, A, object]`;
- MRO для `C`: `[C, A, object]`.

Алгоритм C3 на вход получает:

```text
L(D) = D + merge(L(B), L(C), [B, C])
     = D + merge([B, A, object],
                 [C, A, object],
                 [B, C])
```

Дальше он пошагово:

1. Берёт **первые элементы** всех списков: `B`, `C`, `B`.
2. Ищет такой элемент, который **нигде не встречается в хвостах** других списков:
   - `B` не встречается в хвостах `[A, object]`, `[A, object]`, `[C]` → берём `B`.
3. Убираем `B` из начала всех списков, где он был: получаем

   ```text
   merge([A, object],
         [C, A, object],
         [C])
   ```

4. Повторяем: первые элементы — `A`, `C`, `C`. `C` не встречается в хвостах `[object]`, `[A, object]` → берем `C`.
5. Убираем `C`:

   ```text
   merge([A, object],
         [A, object],
         [])
   ```

6. Остались только `[A, object]`, `[A, object]`. Аналогично берём `A`, затем `object`.

Итог:

```text
D.__mro__ == (D, B, C, A, object)
```

Если бы возникла ситуация, когда **ни один первый элемент списков** нельзя взять (он всегда встречается в хвостах других списков), C3 объявил бы, что «согласованный MRO построить невозможно» и Python выбросил бы `TypeError`.

### Запомните

- `super()` — это не «вызов родителя», а **переход к следующему классу по MRO**.
- При множественном наследовании **все** классы должны использовать `super()`, иначе кооперативность ломается.
- Если MRO стал трудно объясним — это сигнал, что архитектура нуждается в упрощении, часто — через **композицию**.

### Упражнения для самостоятельной работы

1. Реализуйте иерархию с ромбом, в которой:
   - `A.__init__` принимает параметр `value` и сохраняет его в `self.value`;
   - `B` и `C` дополняют поведение (например, логируют, модифицируют `value`);
   - `D` наследуется от `B` и `C`.
   Посмотрите, как меняется поведение при разных порядках наследования `class D(B, C)` vs `class D(C, B)`.
2. Возьмите существующий код, где используется **прямой вызов родителя** (`Base.__init__(self)`), и перепишите его на `super()`.\
   Затем добавьте ещё один класс в иерархию и убедитесь, что код продолжает работать корректно благодаря кооперативному наследованию.
3. В интерактивной консоли поэкспериментируйте с `__mro__` и `mro()` для нескольких искусственных иерархий, чтобы «на глаз» почувствовать, как C3‑линеаризация выстраивает порядок.
