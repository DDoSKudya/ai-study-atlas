### Цель раздела

Разобраться, **что такое наследование в Python на практике**, чем оно отличается от композиции, что такое **множественное наследование** и какие подводные камни оно создаёт.

### Базовые термины

- **Наследование (inheritance)** — механизм, при котором класс‑потомок **перенимает** атрибуты и методы базового класса и может их **расширять или переопределять**.
- **Суперкласс / базовый класс (base class, superclass)** — класс, от которого наследуются.
- **Подкласс / производный класс (subclass, derived class)** — класс, который наследуется от другого.
- **Одиночное наследование (single inheritance)** — у класса один базовый класс.
- **Множественное наследование (multiple inheritance)** — у класса несколько базовых классов.
- **Иерархия классов** — дерево (или общеграфовая структура при множественном наследовании) отношений «кто от кого наследовался».
- **Переопределение (override)** — подкласс предоставляет **свою реализацию** метода, заменяя базовую.
- **Расширение (extension)** — подкласс вызывает базовую реализацию (через `super()`) и **добавляет** дополнительное поведение.

### Как Python ищет атрибут при наследовании (базовый сценарий)

Пусть есть экземпляр `obj` класса `Child`, который наследуется от `Parent`. При обращении:

```python
obj.method(...)
```

Python делает, упрощённо, следующее:

1. Берёт класс `Child` и ищет атрибут `method` **в его словаре класса** (`Child.__dict__`).
2. Если не находит — идёт по **MRO** (`Child.__mro__`) и проверяет каждый следующий класс.
3. Как только атрибут найден (например, в `Parent.__dict__`), берётся **первая найденная реализация**.
4. Если это `function`, она превращается в **bound method** (ей привязывается `self = obj`).

Понимание этого механизма критично для корректного использования переопределения и `super()`.

### Простой пример одиночного наследования

```python
class Animal:
    def speak(self) -> str:
        return "???"


class Dog(Animal):
    def speak(self) -> str:
        return "Woof!"


dog = Dog()
print(dog.speak())      # Woof!
print(isinstance(dog, Dog))     # True
print(isinstance(dog, Animal))  # True
```

Здесь `Dog` **является** (`is-a`) `Animal` — любой `Dog` также допустим там, где ожидают `Animal`.

### Множественное наследование: когда и зачем

Множественное наследование позволяет одному классу **наследовать поведение сразу из нескольких источников**. Типичные сценарии:

- объединение нескольких независимых аспектов поведения (например, «умеет ходить» + «умеет говорить»);
- использование **миксинов** — небольших классов, добавляющих узкую функциональность;
- интеграция с фреймворками, где базовый класс задаёт жизненный цикл, а миксины добавляют фичи.

```python
class Walkable:
    def walk(self):
        print("I can walk")


class Talkable:
    def talk(self):
        print("I can talk")


class Human(Walkable, Talkable):
    pass


h = Human()
h.walk()  # I can walk
h.talk()  # I can talk
```

Такой стиль часто используется при создании **миксинов** (подробнее — в §65a).

### Diamond problem и почему нужен MRO

**Diamond problem** («ромб») — ситуация, когда класс наследуется от двух базовых, которые оба наследуются от общего предка. В графе наследования получается ромб:

```text
      A
     / \
    B   C
     \ /
      D
```

```python
class A:
    def who(self):
        print("A")


class B(A):
    def who(self):
        print("B")


class C(A):
    def who(self):
        print("C")


class D(B, C):
    pass


D().who()
```

Вопрос: **что выведет код?**\
Интуитивно могут ожидать любой вариант: `B`, `C` или ошибку. Python решает это с помощью **MRO** (см. §59) и выводит:

```text
B
```

Порядок поиска метода `who` будет: `D → B → C → A → object`. Этот порядок неслучаен — он подчиняется строгим правилам C3‑линеаризации.

### Когда следует избегать наследования

Наследование часто используют **слишком рано**:

- «У меня два похожих класса — значит надо общий базовый».
- «Нужно переиспользовать пару методов — давайте наследуемся».
- «Фреймворк предлагает базовый класс — давайте от него унаследуемся «на всякий случай»».

Это приводит к:

- **жёстким связям**: изменения базового класса ломают кучу потомков;
- хрупкой иерархии, которую сложно объяснить новым разработчикам;
- сложному MRO при множественном наследовании;
- появлению «божественных» (God Object) базовых классов, которые «умеют всё».

Ещё одна проблема — **лжe‑is‑a**. Примеры:

- `class User(Logger)`: «пользователь — это логгер?» — на самом деле нужна композиция: у пользователя **есть** логгер;
- `class Order(DatabaseConnection)`: заказ не является соединением с базой.

**Практическое правило:** если связь скорее «имеет» (has‑a), чем «является» (is‑a), лучше пользоваться **композицией** (см. §60).

### Запомните

- Наследование — мощный, но легко злоупотребляемый инструмент.
- Множественное наследование в Python **разрешено и активно используется** (например, для миксинов).
- При сложных иерархиях важно понимать **MRO** и **кооперативное наследование** через `super()`.
