### Цель раздела

Научиться использовать модуль `abc` для создания **абстрактных базовых классов (ABC)**, которые задают **обязательный интерфейс** для наследников.

### Базовые термины

- **Абстрактный базовый класс (ABC)** — класс, у которого есть хотя бы один **абстрактный метод**; такой класс **нельзя инстанциировать**.
- **Абстрактный метод** — метод, который **должен** быть реализован в подклассах.
- Декораторы `@abstractmethod`, `@abstractclassmethod`, `@abstractstaticmethod`, `@abstractproperty` — помечают методы как абстрактные.

### Простой пример ABC

```python
from abc import ABC, abstractmethod


class Storage(ABC):
    @abstractmethod
    def save(self, key: str, value: str) -> None:
        """Сохранить значение по ключу."""

    @abstractmethod
    def load(self, key: str) -> str:
        """Загрузить значение по ключу."""
```

Попытка создать экземпляр:

```python
Storage()  # TypeError: Can't instantiate abstract class Storage ...
```

Теперь реализуем конкретное хранилище:

```python
class MemoryStorage(Storage):
    def __init__(self):
        self._data: dict[str, str] = {}

    def save(self, key: str, value: str) -> None:
        self._data[key] = value

    def load(self, key: str) -> str:
        return self._data[key]
```

Здесь `MemoryStorage` обязан реализовать все абстрактные методы, иначе он **сам тоже будет абстрактным**.

Обратите внимание:

- наличие **хотя бы одного** абстрактного метода делает класс абстрактным;
- абстрактные методы могут иметь **реализацию по умолчанию** (тело не обязательно должно быть `...` или `raise NotImplementedError()`), но подкласс всё равно обязан их переопределить.

### Абстрактные методы разных видов

```python
from abc import ABC, abstractmethod, abstractclassmethod, abstractstaticmethod


class PluginBase(ABC):
    @abstractmethod
    def run(self, data: bytes) -> bytes:
        ...

    @abstractclassmethod
    def plugin_name(cls) -> str:
        ...

    @abstractstaticmethod
    def version() -> tuple[int, int]:
        ...
```

Потомок обязан реализовать их все.

### ABC в стандартной библиотеке

Многие интерфейсы описаны через ABC:

- `collections.abc.Iterable`, `Sequence`, `Mapping`, `MutableMapping` и др.;
- `io.IOBase`, `TextIOBase`, `BufferedIOBase` и т.п.;
- `numbers.Number`, `Integral`, `Real`.

Они позволяют писать код, который принимает «что‑то, что ведёт себя как последовательность/отображение», а не конкретные типы.

Например, если функция принимает `collections.abc.Mapping`, ей без разницы, передадут `dict`, `defaultdict`, `ChainMap` или кастомную реализацию словаря — главное, чтобы она соблюдала контракт `Mapping`.

### Виртуальные подклассы и `register()`

Многие ABC используют механизм **виртуального наследования** через метод `register`:

```python
from collections.abc import Mapping


class MyMappingLike:
    # реализуем минимальный набор методов Mapping
    def __getitem__(self, key): ...
    def __iter__(self): ...
    def __len__(self): ...


Mapping.register(MyMappingLike)

print(issubclass(MyMappingLike, Mapping))  # True
print(isinstance(MyMappingLike(), Mapping))  # True
```

Здесь класс **не наследуется** от `Mapping`, но объявляется «виртуальным» подклассом. Это даёт:

- совместимость с `isinstance`/`issubclass`;
- отсутствие жёсткой иерархии наследования.

### Когда использовать ABC

ABC полезны, когда:

- у вас есть **несколько реализаций** одного интерфейса (например, разные «хранилища», «транспорты», «плагины»);
- вы хотите **заставить** разработчиков реализовать нужные методы (ошибка на этапе определения класса, а не во время выполнения);
- вам важно иметь **единое место** с документацией и контрактом интерфейса.

Когда ABC не подходят:

- если вам нужен **чистый структурный контракт** без навязанной иерархии (тогда лучше `Protocol`);
- если реализация должна быть **композицией нескольких аспектов**, а не жёстким наследованием от одного предка;
- когда вы проектируете библиотеку, которой должны пользоваться из разных языков/стеков — иногда проще ограничиться протоколами поведения и документацией, а не «классической» иерархией.

### Запомните

- `abc.ABC` + `@abstractmethod` — стандартный способ задать **жёсткий контракт** по наследованию.
- Абстрактный класс **нельзя инстанциировать**, пока все абстрактные методы не реализованы.
- ABC хорошо сочетаются с **SOLID** (особенно с DIP — Dependency Inversion Principle).

### Упражнения для самостоятельной работы

1. Выберите в своём проекте место, где есть **несколько реализаций** одного интерфейса (например, разные способы отправки уведомлений или разные хранилища).\
   Попробуйте выделить для них общий ABC и переписать код так, чтобы потребители зависели от абстракции, а не от конкретных классов.
2. Возьмите стандартный ABC из `collections.abc` (например, `MutableMapping`) и реализуйте минимальный набор методов, чтобы ваш класс считался `MutableMapping`.\
   Проверьте, какие дополнительные методы «появятся сами» за счёт реализации по умолчанию в базовом ABC.
3. Поэкспериментируйте с `register()`:
   - создайте класс, который **не наследуется** от `Mapping`, но реализует его протокол;
   - зарегистрируйте его как виртуальный подкласс;
   - убедитесь, что `isinstance`/`issubclass` начинают считать его `Mapping`.
