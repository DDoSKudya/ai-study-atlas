### Цель раздела

Связать ООП‑механику (наследование, интерфейсы, композиция) с **архитектурными принципами SOLID** и понять, как они помогают уменьшать связность и упрощать код.

### Кратко о SOLID

- **S — Single Responsibility Principle (SRP)**\
  Один класс — одна причина для изменения.
- **O — Open/Closed Principle (OCP)**\
  Классы должны быть **открыты для расширения**, но **закрыты для изменения**.
- **L — Liskov Substitution Principle (LSP)**\
  Объекты подтипа должны быть **полностью взаимозаменяемы** с объектами базового типа.
- **I — Interface Segregation Principle (ISP)**\
  Лучше несколько **маленьких интерфейсов**, чем один «жирный», который все должны реализовывать.
- **D — Dependency Inversion Principle (DIP)**\
  Зависеть нужно от **абстракций**, а не от конкретных реализаций.

Важно: SOLID — это **не закон физики** и не догма, а набор «очков», через которые полезно смотреть на дизайн. Иногда принципы конфликтуют между собой, и приходится выбирать компромисс.

### S: Single Responsibility

Пример нарушения:

```python
class UserManager:
    def create_user(self, name: str):
        ...

    def send_welcome_email(self, email: str):
        ...
```

Класс отвечает и за **бизнес‑логику**, и за **отправку писем**. Лучше разделить:

```python
class UserService:
    def create_user(self, name: str):
        ...


class EmailService:
    def send_welcome_email(self, email: str):
        ...
```

Типичные «звоночки» нарушения SRP:

- класс знает **слишком много деталей** (работа с БД, бизнес‑логика, отправка писем, логирование — всё в одном месте);
- при изменении одной функциональности вы **постоянно трогаете один и тот же класс**;
- этот класс стал «бутылочным горлышком» для ревью и конфликтов при слиянии веток.

SRP не означает «один метод в классе» — иногда одна ответственость требует десятков методов. Важно, чтобы все они относились к **одной осмысленной задаче**.

### O: Open/Closed

Хотим добавить новый способ логирования, не меняя существующий код:

```python
from abc import ABC, abstractmethod


class Logger(ABC):
    @abstractmethod
    def log(self, message: str) -> None:
        ...


class ConsoleLogger(Logger):
    def log(self, message: str) -> None:
        print(message)
```

Код, который использует `Logger`, не меняется, если мы добавим `FileLogger`, `JsonLogger` и т.д.

Плохой пример (нарушение OCP):

```python
def log(message: str, target: str) -> None:
    if target == "console":
        print(message)
    elif target == "file":
        with open("log.txt", "a") as f:
            f.write(message + "\n")
    # дальше появляются новые ветки для "db", "syslog", "remote"...
```

Каждый раз при добавлении нового варианта логирования приходится **лезть в эту функцию и менять код**. Лучше инкапсулировать варианты логирования в отдельных классах и выбрать нужный экземпляр на уровне конфигурации/композиции.

### L: Liskov Substitution

Класс‑подтип **не должен нарушать ожидания**, установленные базовым классом.

Классический пример: `Square` как подтип `Rectangle`. Кажется логичным, но:

```python
class Rectangle:
    def __init__(self, width: int, height: int):
        self.width = width
        self.height = height

    def set_width(self, width: int) -> None:
        self.width = width

    def set_height(self, height: int) -> None:
        self.height = height


class Square(Rectangle):
    def set_width(self, width: int) -> None:
        self.width = self.height = width

    def set_height(self, height: int) -> None:
        self.width = self.height = height
```

Код, который ожидает, что `set_width` меняет **только ширину**, нарушается. Значит, с точки зрения LSP, `Square` **не является корректным подтипом** `Rectangle` при таком интерфейсе.

LSP можно переформулировать так:

> Если `S` — подтип `T`, то любой код, который корректно работает с объектами `T`, должен также корректно работать с объектами `S` **без изменений**.

Типичные нарушения LSP:

- Подкласс **ужесточает предусловия** (требует больше, чем базовый класс):
  - базовый метод принимает любое число, а подкласс — только положительные;
  - базовый метод допускает `None`, а подкласс — нет.
- Подкласс **ослабляет постусловия** (гарантирует меньше, чем базовый класс):
  - базовый метод гарантирует, что коллекция отсортирована, а подкласс — не всегда;
  - базовый метод обещает, что возвращаемое значение не `None`, а подкласс иногда возвращает `None`.
- Подкласс нарушает **инварианты состояния**, на которые рассчитывают пользователи базового класса.

Практика:

- полезно думать не только «является ли Х частным случаем Y», но и **«любой ли корректный сценарий использования Y будет корректен для Х»**;
- если ответ «нет», то, возможно, вместо наследования стоит использовать **композицию или отдельный интерфейс/протокол**.

### I: Interface Segregation

Плохой пример:

```python
from abc import ABC, abstractmethod


class Worker(ABC):
    @abstractmethod
    def work(self) -> None:
        ...

    @abstractmethod
    def eat(self) -> None:
        ...
```

Робот‑работник не ест, но при таком интерфейсе **вынужден** реализовывать `eat()`. Лучше разделить:

```python
class Workable(ABC):
    @abstractmethod
    def work(self) -> None:
        ...


class Eatable(ABC):
    @abstractmethod
    def eat(self) -> None:
        ...
```

### D: Dependency Inversion

Пусть сервис логирования зависит от **абстракции**, а не от конкретной реализации:

```python
class Service:
    def __init__(self, logger: Logger):  # Logger — абстракция
        self._logger = logger

    def do_something(self):
        self._logger.log("Something happened")
```

Теперь можно подставить любую реализацию `Logger`, в т.ч. фейковую для тестов.

Типичный анти‑пример:

```python
class Service:
    def __init__(self):
        self._logger = ConsoleLogger()  # прямое связывание с конкретным классом
```

- нельзя подменить логгер в тестах;
- сложнее перейти на другой механизм логирования (нужно менять код сервиса);
- нарушается DIP: «верхний» уровень зависит от деталей реализации, а не от абстракции.

Вместо этого зависимости лучше:

- передавать через конструктор (constructor injection);
- или через сеттер/метод конфигурации (setter injection);
- или через DI‑контейнер/фабрику, которая создаёт объект с нужными зависимостями.

### Запомните

- SOLID — это не «академическая расшифровка букв», а набор **очень практичных фильтров**, через которые стоит прогонять дизайн.
- ООП‑механизмы (наследование, интерфейсы, протоколы, композиция) — инструменты для воплощения SOLID на практике.

### Упражнения для самостоятельной работы

1. Возьмите один из своих сервисных классов и попробуйте:
   - выписать все причины, по которым он может измениться (правила бизнеса, формат хранения, формат внешнего API и т.д.);
   - разделить этот класс на несколько так, чтобы каждая причина осталась только в одном классе (SRP).
2. Попробуйте внедрить новый способ логирования/отправки писем/хранения данных так, чтобы:
   - **ни одного `if`/`elif` по «типу реализации»** не появилось в бизнес‑коде;
   - выбор реализации происходил только в конфигурации/композиции (OCP + DIP).
3. Найдите в коде пример, где подкласс фактически **ломает ожидания** базового класса (например, не поддерживает часть допустимых входных значений).\
   Попробуйте:
   - перепроектировать интерфейс так, чтобы LSP соблюдался;
   - либо заменить наследование на композицию/стратегию.
