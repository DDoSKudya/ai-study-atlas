## 4. Контейнеры, оркестрация и деплой (§131)

### 4.0. Зачем всё это: от «работает у меня» к «работает везде»

Проблема без контейнеров/оркестрации:

- на вашей машине всё собрано «из мусора»: чуть такой версии Python, чуть такой системной библиотеки;
- на сервере кто‑то поставил что‑то другое — и начинается «а у меня не воспроизводится».

**Контейнеризация** (Docker) и **оркестрация** (Kubernetes и др.) решают:

- упаковать приложение вместе с зависимостями;
- описать, как его запускать;
- тиражировать это описание на разные среды (dev/stage/prod).

#### 4.0a. Аналогия: чемодан против случайного пакета

Представьте, вы переезжаете:

- без Docker:
  - вы приносите вещи по одной: отдельно чашки, отдельно рубашки, отдельно документы;
  - на новой квартире нет нужных полок, часть документов потерялась, чашки разбились;
- с Docker:
  - вы аккуратно сложили всё нужное в **чемодан**;
  - куда бы вы его ни привезли — внутри одинаковый набор:
    - та же версия Python;
    - те же библиотеки;
    - тот же код и конфиги.

Kubernetes — это уже не чемодан, а:

- **склад**, где:
  - хранят много чемоданов;
  - есть правила, сколько чемоданов где лежит;
  - кто‑то следит, чтобы при поломке один чемодан заменился другим.

### 4.1. Docker: обернуть Python‑сервис в образ

Простейший `Dockerfile` для FastAPI‑приложения:

```dockerfile
FROM python:3.12-slim

WORKDIR /app

COPY pyproject.toml poetry.lock ./
RUN pip install --no-cache-dir poetry && poetry install --no-root --only main

COPY . .

CMD ["poetry", "run", "uvicorn", "myapp:app", "--host", "0.0.0.0", "--port", "8000"]
```

Или упрощённо без Poetry:

```dockerfile
FROM python:3.12-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY . .

CMD ["uvicorn", "myapp:app", "--host", "0.0.0.0", "--port", "8000"]
```

#### 4.1a. Что именно делает этот Dockerfile

Разберём упрощённый вариант:

```dockerfile
FROM python:3.12-slim
```

- **базовый образ**: внутри уже есть Linux + Python 3.12 + системные библиотеки;
- вы начинаете «строить» свой образ поверх этого слоя.

```dockerfile
WORKDIR /app
```

- задаёт рабочую директорию внутри контейнера (как `cd /app`);
- все последующие команды выполняются относительно неё.

```dockerfile
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
```

- `COPY` копирует файл `requirements.txt` с вашей машины **внутрь образа**;
- `RUN` запускает команду внутри образа:
  - устанавливает Python‑зависимости;
  - после её выполнения новый слой образа содержит уже установленный окружением Python.

```dockerfile
COPY . .
```

- копирует весь ваш проект в `/app` внутри образа (исключая то, что попадает под `.dockerignore`).

```dockerfile
CMD ["uvicorn", "myapp:app", "--host", "0.0.0.0", "--port", "8000"]
```

- указывает **команду по умолчанию**, которую запустит контейнер:
  - поднимает Uvicorn с вашим приложением `myapp:app`;
  - слушает порт 8000 на всех интерфейсах внутри контейнера.

Мини‑упражнение:

1. Подумайте, какие файлы не нужно копировать в образ (например, `.git`, кеши, локальные настройки) и как бы вы записали их в `.dockerignore`.
2. Как вы изменили бы `CMD`, чтобы запускать Flask‑приложение через Gunicorn?

Сборка и запуск:

```bash
docker build -t myapp:latest .
docker run -p 8000:8000 myapp:latest
```

Ключевые идеи:

- в образ попадает ровно то, что вы описали (Python‑версия, зависимости, код);
- запуск в любой среде (где есть Docker) даёт **одинаковое поведение**.

#### 4.1a. docker-compose: несколько сервисов вместе

Частая картина:

- веб‑сервис на Python;
- Redis;
- PostgreSQL.

`docker-compose.yml`:

```yaml
version: "3.9"

services:
  web:
    build: .
    ports:
      - "8000:8000"
    depends_on:
      - db
      - redis

  db:
    image: postgres:16
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: pass
      POSTGRES_DB: example

  redis:
    image: redis:7
```

Команда:

```bash
docker compose up
```

Поднимает сразу все три контейнера и настраивает сеть между ними.

### 4.2. Kubernetes: оркестрация контейнеров

Kubernetes («k8s») отвечает за:

- развёртывание контейнеров (pods, deployments);
- балансировку трафика (services, ingress);
- рестарты упавших контейнеров;
- скейлинг (ручной и автоматический).

Минимальный `Deployment` (идея, не полный продакшн‑пример):

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
        - name: myapp
          image: myapp:latest
          ports:
            - containerPort: 8000
```

Python‑клиент `kubernetes` позволяет:

- писать операторов/утилиты на Python;
- взаимодействовать с API кластера (создавать/обновлять ресурсы).

На начальном этапе достаточно:

- понимать, что ваш Docker‑образ будет жить в Pod’ах;
- знать, что Kubernetes «крутит» эти Pod’ы, перезапускает и скейлит.

#### 4.2a. Разбираем Deployment по частям

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
```

- `kind: Deployment` — объект, который говорит:  
  «запусти и поддерживай несколько копий (реплик) моего приложения»;
- `metadata.name` — имя этого объекта в кластере (как идентификатор).

```yaml
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
```

- `replicas: 3` — «хочу три копии (pod’а)»;
- `selector.matchLabels` — как найти pod’ы, которые относятся к этому Deployment’у:
  - те, у кого есть лейбл `app: myapp`.

```yaml
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
        - name: myapp
          image: myapp:latest
          ports:
            - containerPort: 8000
```

- `template` — шаблон pod’а:
  - метаданные (`labels`), которые потом используются селекторами/сервисами;
  - `containers` — список контейнеров внутри pod’а.
- `image: myapp:latest` — какой Docker‑образ запускать;
- `containerPort: 8000` — какой порт слушает контейнер (чтобы сервисы могли к нему подключаться).

Итого:

- Deployment говорит: «держи **N pod’ов**, каждый из которых — контейнер с моим образом»;
- если один pod упал, Kubernetes создаст новый, чтобы снова было N.

Мини‑упражнение:

1. Что будет, если поменять `replicas: 3` на `replicas: 1`?  
   Как это повлияет на отказоустойчивость и потребление ресурсов?
2. Почему важно, чтобы `selector.matchLabels` и `template.metadata.labels` совпадали?

### 4.3. AWS, Terraform, Ansible, serverless

Краткое описание:

- **AWS (boto3)**:
  - Python‑SDK для AWS: S3, EC2, RDS, Lambda, SQS и десятки других сервисов;
  - в Python‑коде вы можете создавать/удалять ресурсы, загружать файлы, вызывать Lambda.

- **Terraform**:
  - описывает **инфраструктуру как код** (HCL‑файлы);
  - пример: «нужна ВМ такого типа, S3‑бакет, база RDS»;
  - далее `terraform apply` — и Terraform создаёт/изменяет инфраструктуру.

- **Ansible**:
  - автоматизация конфигурации серверов;
  - playbooks (YAML) описывают, какие пакеты поставить, какие файлы скопировать, какие сервисы перезапустить.

- **Serverless (Lambda и аналоги)**:
  - вы пишете **функцию**, а не целый сервер;
  - платите только за фактическое время выполнения;
  - Python‑код загружается и вызывается по событию (HTTP‑запрос, очередь, крон и т.д.).

Пример простой AWS Lambda‑функции на Python (идея):

```python
def handler(event, context):
    name = event.get("name", "world")
    return {"statusCode": 200, "body": f"Hello, {name}!"}
```

Обёртки вроде **Zappa** и **Mangum** позволяют:

- взять существующее WSGI/ASGI‑приложение (Flask/FastAPI);
- упаковать его под формат Lambda.

#### 4.3a. Как мыслить про Terraform и Ansible

Аналогия:

- **Terraform** — это архитектор и строитель в одном:
  - вы пишете «чертёж» инфраструктуры (HCL‑файлы);
  - Terraform сравнивает «как есть» с «как должно быть» и приводит одно к другому.
- **Ansible** — это повар по рецепту:
  - вы пишете пошаговый «рецепт» (playbook), что нужно сделать на сервере;
  - Ansible по SSH заходит и выполняет шаги.

Практические отличия:

- Terraform:
  - лучше для создания/удаления облачных ресурсов (ВМ, базы, сети);
  - хранит состояние (state) и может аккуратно «докрутить» разницу.
- Ansible:
  - лучше для настройки уже существующих машин (пакеты, файлы, сервисы);
  - хорош для «конфигурации как кода».

Мини‑упражнение:

1. Представьте, что нужно:
   - поднять ВМ в AWS;
   - установить на неё Python и ваш сервис;
   - настроить systemd‑unit.  
   Какие шаги вы бы описали в Terraform, а какие — в Ansible?

#### 4.3b. Serverless: плюсы и ограничения

Плюсы:

- не нужно управлять серверами (обновления, патчи, масштабирование);
- платите только за время выполнения функции и количество вызовов;
- легко запускать мелкие задачи по расписанию или по событиям (крон, очереди, S3‑триггеры и т.д.).

Ограничения:

- жёсткие лимиты по времени выполнения, памяти, размеру пакета;
- холодный старт (первый вызов после простоя может быть медленнее);
- сложнее дебажить локально и симулировать полностью инфраструктуру.

Serverless хорошо подходит для:

- обработки событий (загрузка файла в S3, сообщение в очереди);
- небольших API/веб‑хуков;
- фоновых задач по расписанию.

---

### 4.4. Запомните по §131

- Docker решает задачу **упаковки** приложения, Kubernetes — задачу **масштабируемого запуска и управления**.
- AWS/Terraform/Ansible — про **определение и автоматизацию инфраструктуры**, а не только кода.
- Serverless‑подход удобен для «маленьких функций по запросу», но не всегда заменяет классический сервис.
