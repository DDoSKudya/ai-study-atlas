## 5. Мониторинг, логирование и трейсинг (§132)

### 5.0. Зачем наблюдаемость

Без мониторинга и логов продакшн выглядит так:

> «Пользователь говорит, что всё тормозит/падает, а вы глазами смотрите в код и гадаете».

**Observability** даёт:

- **логи** — «что произошло» (события, ошибки, важные шаги);
- **метрики** — численные показатели (QPS, время ответа, ошибки/сек, использование памяти/CPU);
- **трейсинг** — путь одного запроса через множество сервисов (где он «застрял»).

#### 5.0a. Очень короткая аналогия

Представьте машину:

- **логи** — это записи в бортовом журнале: «в 10:01 загорелась лампа», «в 10:05 заменили масло»;
- **метрики** — это приборы на панели: спидометр, тахометр, температура;
- **трейсинг** — это GPS‑трек: по какой дороге вы ехали и где застряли в пробке.

Без этого всего вы знаете только одно: «клиент говорит, что машина не едет».

### 5.1. Sentry: ошибки в проде

`sentry-sdk` интегрируется с веб‑фреймворками и автоматически ловит необработанные исключения.

Пример (FastAPI):

```python
import sentry_sdk
from fastapi import FastAPI

sentry_sdk.init(
    dsn="https://key@sentry.io/project",
    traces_sample_rate=0.1,
)

app = FastAPI()

@app.get("/boom")
def boom():
    raise RuntimeError("Something went wrong")
```

Что происходит:

- при старте приложения инициализируется Sentry SDK;
- любое необработанное исключение:
  - отправляется в Sentry;
  - в интерфейсе виден стек, окружение, частота ошибок.

Важно:

- Sentry не заменяет логи, а **дополняет** их;
- не нужно логировать чувствительные данные (пароли, токены) в контексте.

#### 5.1a. Зачем это нужно именно вам

Базовая боль без Sentry:

- пользователи иногда видят 500‑ошибку;
- кто‑то скринит и кидает вам в чат: «опять сломалось»;
- вы:
  - не знаете, **сколько** пользователей это видят;
  - не знаете, **какой именно запрос** упал;
  - не знаете, **какой стек вызовов** и переменные в момент ошибки.

С Sentry:

- каждая необработанная ошибка попадает:
  - в список инцидентов;
  - с полным стеком;
  - с частотой (сколько раз, за какой период);
  - зачастую с информацией о пользователе/запросе.

Практическое правило:

- даже в маленьком pet‑проекте полезно:
  - подключить Sentry (или аналог);
  - убедиться, что вы **видите каждую 500‑ку**, а не только те, о которых кто‑то пожаловался.

### 5.2. Prometheus и метрики

Prometheus — система мониторинга, которая:

- периодически «ходит» на HTTP‑эндпоинт (`/metrics`);
- собирает текстовые метрики;
- строит графики и алерты.

`prometheus_client` в Python:

```python
from prometheus_client import Counter, Histogram, start_http_server
import time

REQUESTS = Counter("myapp_requests_total", "Total requests")
LATENCY = Histogram("myapp_request_latency_seconds", "Request latency")

def handle_request():
    REQUESTS.inc()
    with LATENCY.time():
        time.sleep(0.1)  # работа

if __name__ == "__main__":
    start_http_server(8001)  # эндпоинт /metrics
    while True:
        handle_request()
```

Prometheus будет собирать:

- общее количество запросов;
- распределение латентности по `Histogram`.

#### 5.2a. Как читать пример с Prometheus‑клиентом

```python
REQUESTS = Counter("myapp_requests_total", "Total requests")
LATENCY = Histogram("myapp_request_latency_seconds", "Request latency")
```

- `Counter` — счётчик, который можно только увеличивать (`inc()`);
- `Histogram` — метрика, которая собирает распределение значений (например, времени выполнения).

```python
def handle_request():
    REQUESTS.inc()
    with LATENCY.time():
        time.sleep(0.1)  # работа
```

- при каждом вызове:
  - увеличиваем счётчик запросов;
  - контекст `LATENCY.time()`:
    - замеряет, сколько времени занял блок внутри;
    - записывает это время в гистограмму.

```python
start_http_server(8001)  # эндпоинт /metrics
```

- поднимает маленький HTTP‑сервер;
- по адресу `http://localhost:8001/metrics` приложение отдаёт **текст со всеми метриками**.

Дальше:

- Prometheus (отдельный сервис) регулярно ходит на `/metrics`;
- хранит значения и строит графики/алерты.

Мини‑упражнение:

1. Какие ещё метрики вам хотелось бы видеть в своём сервисе:
   - количество ошибок;
   - длина очередей;
   - размер кэша/количество активных пользователей.
2. Попробуйте мысленно придумать имена метрик:
   - `myapp_errors_total`, `myapp_jobs_in_queue`, и т.п.

### 5.3. OpenTelemetry и распределённый трейсинг

OpenTelemetry (OTel) — стандартный стек для:

- трейсинга (span’ы, trace id);
- метрик;
- логов.

Идея:

- каждый сервис помечает входящий запрос **trace id**;
- при вызовах других сервисов этот id прокидывается дальше;
- в трейсинг‑системе (Jaeger, Tempo, Datadog APM и т.п.) вы видите **сквозной путь** запроса.

На практике:

- вы подключаете OTel‑SDK/авто‑инструментацию;
- конфигурируете экспортер (куда отправлять трейсинг/метрики);
- часто этого достаточно, чтобы видеть «узкие места».

#### 5.3a. Мини‑пример в голове

Представьте запрос к вашему API:

1. Клиент вызывает `/api/pay` на **gateway‑сервисе**.
2. Gateway:
   - делает запрос в **сервис пользователей**;
   - делает запрос в **платёжный сервис**;
   - пишет событие в **сервис аудита**.

Без трейсинга:

- вы видите только логи по каждому сервису отдельно;
- понять, какие записи относятся к одному и тому же исходному запросу, трудно.

С OTel:

- у запроса есть `trace id` (общий для всей цепочки);
- в каждой точке создаются `span`’ы (отрезки времени):
  - `gateway_handle_request`,
  - `users_get_profile`,
  - `payments_charge`,
  - `audit_write_event`.
- в UI вы видите:
  - дерево этих span’ов;
  - где был самый большой вклад во время ответа.

### 5.4. structlog, ELK, Loki

**structlog**:

- позволяет писать **структурированные логи**:

```python
import structlog

structlog.configure(processors=[structlog.processors.JSONRenderer()])
log = structlog.get_logger()

log.info("user_logged_in", user_id=123, method="password")
```

Вывод — JSON, который:

- проще парсить;
- легче фильтровать и агрегировать.

**ELK‑стек** (Elasticsearch + Logstash + Kibana) и **Loki + Grafana**:

- собирают логи со многих сервисов (через агенты);
- дают UI для поиска и просмотра логов;
- часто поддерживают структурированный JSON как «родной формат».

#### 5.4a. Почему структурированные логи проще для жизни

Сравните:

- классический текстовый лог:

  ```text
  2026-02-11 12:00:00 INFO user 123 logged in from 1.2.3.4
  ```

- структурированный лог (JSON):

  ```json
  {"ts": "2026-02-11T12:00:00Z", "level": "info", "event": "user_logged_in", "user_id": 123, "ip": "1.2.3.4"}
  ```

Во втором случае:

- в ELK/Loki можно сказать:
  - «покажи все события `user_logged_in` за последний час»;
  - «покажи все события для `user_id=123` с ошибками».

Это намного проще, чем парсить текст регулярками.

---

### 5.5. Запомните по §132

- Логи, метрики и трейсинг — три опоры наблюдаемости; без них вы слепы в продакшене.
- Sentry — «глаз» на необработанные исключения, Prometheus/OpenTelemetry — на метрики и трейсинг.
- Структурированные логи (JSON + structlog) лучше подходят для централизованных систем логирования, чем «сырые строки».
