## 3. Интеграция с БД и внешними сервисами (§130)

### 3.0. Драйвер vs ORM vs high‑level клиент

При работе с внешними сервисами и БД обычно есть несколько уровней:

1. **Низкоуровневый драйвер**:
   - psycopg2, asyncpg, PyMySQL, pymongo, redis‑py, boto3, elasticsearch‑py;
   - близок к «сырому» протоколу (SQL, Redis commands, HTTP API).
2. **ORM/абстракции**:
   - SQLAlchemy, Django ORM;
   - маппинг таблиц → Python‑классы, миграции, связи.
3. **Высокоуровневые клиенты**:
   - обёртки вокруг API (SDK для облаков, клиентские библиотеки от производителей).

Важно понимать:

- ORM не отменяет необходимости **понимать, как работает драйвер** (соединения, пулы, транзакции);
- SDK не отменяет **ограничений самого сервиса** (лимиты, консистентность, задержки).

#### 3.0a. Аналогия: язык, переводчик и «секретарь»

Представьте:

- **БД/сервис** — это человек, говорящий только на SQL/HTTP/своём протоколе;
- **драйвер** (`psycopg2`, `asyncpg`, `redis-py`, `boto3`) — это тот, кто умеет говорить на языке сервиса:
  - вы даёте ему «команду» в питонической форме;
  - он переводит её в SQL/HTTP‑запрос и обратно.
- **ORM** (SQLAlchemy, Django ORM) — ещё один уровень:
  - вы общаетесь не SQL, а «создай пользователя с таким‑то именем»;
  - ORM под капотом сам пишет SQL.
- **SDK/высокоуровневый клиент** — как «секретарь»:
  - вы говорите: «создай бакет и загрузи туда файл»;
  - секретарь:
    - вызывает нужные HTTP‑методы;
    - обрабатывает авторизацию;
    - ретраи, пагинацию и т.д.

Чем ниже уровень (драйвер) — тем:

- больше контроля;
- но и больше деталей (строки подключения, курсоры, ручные транзакции).

Чем выше (ORM/SDK) — тем:

- проще писать бизнес‑логику;
- но нужно понимать, **что примерно происходит под капотом**, чтобы не удивляться поведению и нагрузке.

Мини‑упражнение:

1. Возьмите любую операцию с БД (например: «добавить пользователя»):
   - попробуйте описать её на уровне SQL;
   - затем на уровне ORM (Django/SQLAlchemy);
   - затем на уровне REST‑API (если бы это был микросервис).
2. Отметьте, какие детали «прячутся» на каждом уровне и где вы всё ещё должны о них помнить (например, транзакции, индексы, лимиты).

### 3.1. PostgreSQL: psycopg2 и asyncpg

**psycopg2** — классический синхронный драйвер PostgreSQL.

```python
import psycopg2

conn = psycopg2.connect(
    dbname="example",
    user="user",
    password="pass",
    host="localhost",
    port=5432,
)

try:
    with conn.cursor() as cur:
        cur.execute(
            "CREATE TABLE IF NOT EXISTS users (id SERIAL PRIMARY KEY, name TEXT)"
        )
        cur.execute("INSERT INTO users (name) VALUES (%s)", ("Alice",))
    conn.commit()
finally:
    conn.close()
```

Ключевые моменты:

- используйте **плейсхолдеры** (`%s`), а не конкатенацию строк — это защита от SQL‑инъекций;
- не забывайте про `commit()` и закрытие соединения;
- лучше использовать **пулы соединений** (через SQLAlchemy или пул psycopg2).

#### 3.1a. Разбор примера с psycopg2 «по шагам»

```python
conn = psycopg2.connect(
    dbname="example",
    user="user",
    password="pass",
    host="localhost",
    port=5432,
)
```

- `connect(...)`:
  - устанавливает TCP‑соединение с сервером PostgreSQL;
  - проходит аутентификацию (логин/пароль);
  - возвращает объект `conn` — «соединение с БД».
- `dbname`, `user`, `password`, `host`, `port` — это по сути та же информация, что в строке подключения `postgresql://user:pass@host:port/dbname`, только заданная по отдельности.

```python
try:
    with conn.cursor() as cur:
        cur.execute(
            "CREATE TABLE IF NOT EXISTS users (id SERIAL PRIMARY KEY, name TEXT)"
        )
        cur.execute("INSERT INTO users (name) VALUES (%s)", ("Alice",))
    conn.commit()
finally:
    conn.close()
```

Смысл:

1. `with conn.cursor() as cur:`:
   - создаём «курсор» — объект, через который:
     - отправляем SQL‑запросы;
     - читаем результаты;
   - контекстный менеджер (`with`) автоматически закрывает курсор по выходу из блока.
2. `cur.execute("CREATE TABLE ...")`:
   - отправляет SQL‑текст на сервер;
   - сервер создаёт таблицу (если её ещё не было).
3. `cur.execute("INSERT INTO users (name) VALUES (%s)", ("Alice",))`:
   - `%s` — плейсхолдер, **не формат строки Python!**;
   - реальные значения передаются вторым аргументом (`("Alice",)`), драйвер:
     - сам экранирует их;
     - не даёт вставить туда произвольный кусок SQL.
4. `conn.commit()`:
   - говорит: «всё, что мы сделали с момента открытия/последнего коммита — сохранить навсегда»;
   - без этого изменения могут откатиться (в зависимости от настроек автокоммита).
5. `finally: conn.close()`:
   - гарантирует закрытие соединения, даже если была ошибка;
   - важно, иначе можно «утечь» соединения и упереться в лимит БД.

Мини‑аналогия:

- `connect` — как открыть дверь в банк и получить персонального менеджера;
- `cursor` — как стол, за которым вы подписываете отдельные бумаги (запросы);
- `execute` — подписать конкретный документ (команду);
- `commit` — нажать «подтвердить сделку»;
- `close` — уйти из банка, освободив менеджера.

#### 3.1b. Типичные ошибки при работе с psycopg2

1. **Держат одно соединение «навсегда» на весь сервер**:
   - если у вас многопоточный/многопроцессный сервер, одно соединение может стать узким местом;
   - лучше использовать пул (например, через SQLAlchemy `create_engine` с пулом по умолчанию).

2. **Забывают вызывать `commit()`**:
   - в dev всё кажется рабочим (автокоммит может быть включён, или вы всегда всё делаете в одном скрипте);
   - в проде изменения могут не попадать в БД или откатываться.

3. **Собирают SQL «через f‑строку»**:

   ```python
   cur.execute(f"INSERT INTO users (name) VALUES ('{name}')")  # так делать нельзя
   ```

   - это открывает дверь для SQL‑инъекций (пользователь может передать `name="'; DROP TABLE users; --"` и т.п.);
   - всегда используйте плейсхолдеры и второй аргумент с данными.

Мини‑упражнение:

1. Попробуйте переписать кусок:

   ```python
   cur.execute(f"SELECT * FROM users WHERE name = '{name}'")
   ```

   на вариант с плейсхолдером.
2. Ответьте себе:
   - что произойдёт, если не вызвать `conn.close()` ни разу в долгоживущем процессе?


**asyncpg** — асинхронный драйвер PostgreSQL:

```python
import asyncpg
import asyncio

async def main():
    conn = await asyncpg.connect("postgresql://user:pass@localhost:5432/example")
    try:
        rows = await conn.fetch("SELECT id, name FROM users")
        for row in rows:
            print(row["id"], row["name"])
    finally:
        await conn.close()

asyncio.run(main())
```

Или пул:

```python
async def main():
    pool = await asyncpg.create_pool(dsn="postgresql://user:pass@localhost:5432/example")
    async with pool.acquire() as conn:
        await conn.execute("INSERT INTO logs(message) VALUES($1)", "hello")
```

Для высоконагруженных async‑сервисов пул **обязателен**, иначе вы будете тратить время на постоянное открытие/закрытие соединений.

#### 3.1c. Как читать пример с asyncpg (по пунктам)

```python
async def main():
    conn = await asyncpg.connect("postgresql://user:pass@localhost:5432/example")
    try:
        rows = await conn.fetch("SELECT id, name FROM users")
        for row in rows:
            print(row["id"], row["name"])
    finally:
        await conn.close()
```

1. `async def main():` — определяем корутину (асинхронную функцию).
2. `await asyncpg.connect(...)`:
   - устанавливает соединение **асинхронно**;
   - пока идёт сетевое рукопожатие, event loop может заниматься другими задачами.
3. `await conn.fetch("SELECT ...")`:
   - отправляет запрос и возвращает список строк (обычно как объекты похожие на словари).
4. `for row in rows:`:
   - обычный цикл по результатам;
5. `await conn.close()`:
   - закрывает соединение, опять же асинхронно.

В пуле (`create_pool`):

- `create_pool` создаёт **набор соединений** (например, от 1 до N штук);
- `async with pool.acquire() as conn:`:
  - берёт **одно свободное соединение** из пула;
  - внутри блока вы можете выполнить несколько команд;
  - после блока соединение возвращается в пул.

Типичные ошибки:

1. **Создавать/закрывать соединение в каждой маленькой функции**:
   - каждое открытие — дорого;
   - пулы решают эту проблему.
2. **Блокирующий код внутри async‑функций**:
   - использовать `time.sleep` или синхронный драйвер БД в `async def`;
   - это «морозит» event loop и убивает смысл async‑подхода.

Мини‑упражнение:

1. Представьте функцию `get_user(id)` на asyncpg:
   - как бы вы её написали с использованием пула?
2. Что произойдёт, если одновременно 100 корутин попытаются взять соединение из пула размером 5?

### 3.2. MongoDB: pymongo и motor

MongoDB — документо‑ориентированная БД, часто используется для логов, событий, прототипов.

Синхронный клиент (`pymongo`):

```python
from pymongo import MongoClient

client = MongoClient("mongodb://localhost:27017/")
db = client["example"]
users = db["users"]

users.insert_one({"name": "Alice", "age": 30})

for u in users.find({"age": {"$gte": 18}}):
    print(u)
```

Async‑клиент (`motor`):

```python
import motor.motor_asyncio
import asyncio

async def main():
    client = motor.motor_asyncio.AsyncIOMotorClient("mongodb://localhost:27017/")
    db = client["example"]
    users = db["users"]

    await users.insert_one({"name": "Bob", "age": 25})
    cursor = users.find({})
    async for doc in cursor:
        print(doc)

asyncio.run(main())
```

#### 3.2a. Когда MongoDB уместна, а когда нет

Mongo удобно использовать, когда:

- структура данных **меняется со временем** и строго фиксировать схему в БД не хочется;
- нужно хранить «гибкие» документы (JSON‑подобные объекты);
- много операций типа «логирование событий», «сохранение произвольных настроек».

Но важно помнить:

- отсутствие жёсткой схемы в БД **не отменяет** необходимости продумывать структуру данных:
  - если разные документы в коллекции сильно отличаются по полям, читать/поддерживать их становится трудно;
- транзакции и консистентность у Mongo отличаются от PostgreSQL:
  - долгие и сложные бизнес‑операции зачастую проще описывать в реляционной БД.

Типичные ошибки:

1. «Пихать всё подряд» в одну коллекцию без схемы → через год вы не понимаете, что где лежит.
2. Использовать Mongo там, где нужен сложный SQL и строгие ограничения (внешние ключи, транзакции) — проще взять PostgreSQL.

Мини‑упражнение:

1. Придумайте, как выглядела бы коллекция `logs` в MongoDB: какие поля, какие индексы могли бы понадобиться.
2. Сравните: что проще хранить в Mongo, а что — в PostgreSQL, если у вас:
   - таблица заказов интернет‑магазина;
   - сырые JSON‑логзаписи от микросервисов.

### 3.3. Redis: кеш, pub/sub, простые структуры

Синхронный клиент:

```python
import redis

r = redis.Redis(host="localhost", port=6379, db=0)

r.set("counter", 1)
r.incr("counter")
print(int(r.get("counter")))  # 2
```

Типичные сценарии:

- кеш (страниц, результатов запросов, токенов);
- счетчики (просмотры, лайки);
- pub/sub (простые уведомления);
- брокер для очередей (Celery/RQ/Huey/ARQ).

Асинхронный мир:

- исторически был `aioredis`, сейчас он слился в `redis-py` (поддержка async‑API внутри основной библиотеки).

#### 3.3a. Redis как блокнот на быстрой памяти

Представьте, что у вас на столе есть:

- **блокнот** — очень быстрый, но если его сдует ветром/зальёт водой, записи пропадут;
- **большая папка с документами в шкафу** — это ваша основная БД (PostgreSQL).

Redis — это этот самый быстрый блокнот:

- он хранит данные **в памяти**;
- идеально подходит для:
  - кэшей («если потеряется — воссоздадим из основной БД»);
  - счётчиков;
  - короткоживущих сессий.

Опасность:

- хранить в Redis **единственную копию критичных данных** (деньги, заказы) — рискованно;
- при перезапуске сервера/Redis без персистентности данные могут потеряться.

Типичные ошибки:

1. **Нет TTL у кэша**:
   - ключи никогда не протухают;
   - кэш раздувается, и Redis съедает всю память.
2. **Хранение сложных структур без явной сериализации**:
   - лучше явным образом сериализовать данные (JSON/pickle и т.п.) и понимать формат;
   - иначе трудно отлаживать и мигрировать данные.

Мини‑упражнение:

1. Придумайте ключи для кэша пользователей: как бы вы хранили пользовательские данные в Redis, если основная БД — PostgreSQL?
2. Где бы вы выставили TTL, а где нет (например, кэш профиля vs кэш временного токена)?

### 3.4. S3‑совместимое хранилище: boto3 / aioboto3

AWS S3 и совместимые сервисы (MinIO, Yandex Object Storage и т.п.) — это **объектные хранилища**:

- работают поверх HTTP;
- хранят «бакеты» и «объекты» (файлы).

Пример загрузки файла через `boto3`:

```python
import boto3

s3 = boto3.client(
    "s3",
    aws_access_key_id="...",
    aws_secret_access_key="...",
    region_name="us-east-1",
)

s3.upload_file("local.txt", "my-bucket", "folder/remote.txt")
```

Особенности:

- S3 — **не файловая система** (нет «папок» как сущности, только ключи с `/`);
- операции могут быть «дорогими» по времени → их часто выносят в фоновые задачи (Celery и др.);
- вместе с S3 обычно проектируют **стратегию версионирования и жизненного цикла** объектов (lifecycle).

Async‑варианты (`aioboto3`) строятся поверх `boto3`, оборачивая операции в async‑интерфейсы.

#### 3.4a. Как мыслить про S3 «чтобы не запутаться»

S3 — это похоже на:

- **ключ → файл**:
  - ключ `"avatars/user123.png"` — просто строка;
  - «папка» `avatars/` существует только логически (как префикс ключа).

Последствия:

- нельзя сделать мгновенный «переезд папки» — это будет переименование множества ключей;
- нет привычной блокировки файлов — вместо этого нужно проектировать уникальные имена и версии.

Практические советы:

1. Всегда продумывайте **схему имён**:
   - `project/environment/resource/id/filename.ext`;
   - так проще чистить старые объекты и искать нужные.
2. Не делайте много мелких запросов из синхронного кода:
   - каждое обращение к S3 — сетевой вызов с не маленькой латентностью;
   - операции над файлами часто выносят в фоновые задачи через очередь.

Мини‑упражнение:

1. Придумайте схему ключей для:
   - аватарок пользователей;
   - экспортов отчётов в PDF по датам.
2. Какие из этих операций вы бы выполняли из HTTP‑эндпоинтов, а какие — через Celery/RQ?

### 3.5. Elasticsearch, ClickHouse и аналитика

- **Elasticsearch**:
  - поисковый движок + аналитические запросы по логам/событиям;
  - клиент: `elasticsearch` / `elasticsearch-py`;
  - запросы в виде JSON‑DSL.

Пример (очень упрощённо):

```python
from elasticsearch import Elasticsearch

es = Elasticsearch("http://localhost:9200")

es.index(index="logs", document={"level": "info", "message": "hello"})

resp = es.search(index="logs", query={"match": {"message": "hello"}})
print(resp["hits"]["hits"])
```

- **ClickHouse**:
  - колоночная БД для аналитики;
  - драйвер: `clickhouse-driver`;
  - заточена под «большие селекты» и агрегации, а не под update/delete по одной записи.

Мини‑упражнение:

1. Посмотрите на структуру JSON‑ответа Elasticsearch (через запрос `/_search` в браузере или curl) и найдите, где именно лежат найденные документы.

### 3.6. SQLite и APSW

SQLite уже рассматривали ранее, но здесь важно:

- часто используется как **локальное хранилище** в маленьких сервисах/скриптах;
- `APSW` — более низкоуровневый доступ:
  - тонкий контроль над транзакциями;
  - доступ к расширенным возможностям SQLite.

В больших продакшн‑системах SQLite обычно уступает место PostgreSQL/MySQL, но как «встроенная БД» остаётся очень полезной.

### 3.7. Запомните по §130

- Всегда думайте о **жизненном цикле соединений**: открыли → использовали → закрыли/вернули в пул.
- Для async‑сервисов выбирайте async‑драйверы (asyncpg, motor, async‑Redis/aioboto3), иначе вы блокируете event loop.
- Разные БД и сервисы решают **разные задачи**: PostgreSQL — OLTP, ClickHouse — аналитика, Elasticsearch — поиск/логи, S3 — файлы.
