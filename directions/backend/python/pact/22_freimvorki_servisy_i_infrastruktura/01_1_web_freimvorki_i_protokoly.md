## 1. Web‑фреймворки и протоколы (§128)

### 1.0. Общая идея: приложение, протокол, сервер

Упрощённая картинка веб‑приложения на Python:

1. **Веб‑фреймворк** (Django, Flask, FastAPI и т.п.) — даёт удобный API:
   - объявить маршруты/эндпоинты;
   - разобрать запрос;
   - сформировать ответ (HTML/JSON);
   - (часто) работать с БД, сессиями, авторизацией.
2. **Протокол приложения** — WSGI или ASGI:
   - это **договор**, как сервер будет вызывать ваше приложение;
   - приложение — это, по сути, либо «функция» (WSGI), либо «async‑функция/объект» (ASGI).
3. **Сервер** (Gunicorn, Uvicorn, uWSGI, Hypercorn, Daphne…):
   - принимает **TCP‑соединения** от клиентов;
   - поднимает пул процессов/потоков;
   - для каждого запроса вызывает ваше приложение по протоколу.

Важно разделять:

- **код приложения** (то, что вы пишете на Python);
- **веб‑сервер** (что крутится «поверх» Python, обрабатывает подключения и балансирует воркеры);
- **обратный прокси / фронт** (Nginx/Traefik/Cloudflare — но это уже вне Python‑мира).

#### 1.0a. То же самое «на пальцах» (очень медленно)

Представьте обычное кафе:

- **покупатель** — браузер или мобильное приложение;
- **официант** — веб‑сервер (Gunicorn/Uvicorn);
- **кухня** — ваш Python‑код (Django/Flask/FastAPI);
- **правила общения официанта с кухней** — протокол WSGI или ASGI.

Как идёт процесс:

1. Покупатель делает заказ → **отправляет HTTP‑запрос**.
2. Официант (Gunicorn/Uvicorn):
   - принимает заказ (TCP‑соединение + HTTP‑запрос);
   - переводит его на «язык кухни» (формат WSGI/ASGI);
   - передаёт этот заказ на кухню (в ваш код).
3. Кухня (ваш код во фреймворке):
   - смотрит, какой это стол/маршрут (`/hello`, `/users`, `/orders/123`);
   - готовит блюдо (считает, идёт в БД, собирает JSON/HTML);
   - возвращает готовое блюдо обратно официанту (объект ответа).
4. Официант упаковывает блюдо (HTTP‑ответ, заголовки, статус) и несёт покупателю.

Почему это важно понимать:

- когда «тормозит» — нужно понять, кто виноват:
  - сам код (медленный запрос в БД, тяжёлый расчёт);
  - веб‑сервер (слишком мало воркеров, проблемы с конфигом);
  - или что-то ещё до/после (обратный прокси, сеть);
- многие настройки (количество воркеров, таймауты и т.п.) находятся **на уровне сервера**, а не фреймворка.

Мини‑упражнение (без кода, только подумать):

1. Представьте, что у вас один официант на 100 гостей — к чему это приведёт?  
   (Аналогия: один воркер Gunicorn для нагруженного сервиса.)
2. Представьте, что у вас 50 официантов, а кухня очень медленная — поможет ли это?  
   (Аналогия: много воркеров, но медленный код/БД.)
3. Попробуйте в голове найти аналогии:
   - «очередь задач» → заказы, которые кухня делает **не при госте**, а к определённому времени;
   - «брокер сообщений» → отдельная стойка, где лежат бумажки с заказами.

### 1.1. Django: «полный» фреймворк (MTV, ORM, админка)

**Когда думать о Django:**

- вам нужен **классический сайт/админка** + API;
- много **форм, валидации, сложной БД‑схемы**;
- хочется «из коробки»: ORM, миграции, админка, аутентификация, шаблоны.

Минимальный проект (очень упрощённо):

```bash
django-admin startproject mysite
cd mysite
python manage.py startapp blog
```

Простейшее представление (`views.py`):

```python
from django.http import HttpResponse

def hello(request):
    return HttpResponse("Hello, Django!")
```

Маршрут (`urls.py` корневой):

```python
from django.contrib import admin
from django.urls import path

from blog.views import hello

urlpatterns = [
    path("admin/", admin.site.urls),
    path("hello/", hello),
]
```

Запуск dev‑сервера:

```bash
python manage.py runserver
```

Ключевые идеи:

- **MTV (Model‑Template‑View)**:
  - Model — ORM‑модели (таблицы БД);
  - Template — HTML‑шаблоны (Jinja‑подобный язык шаблонов);
  - View — функция/класс, который принимает `request` и возвращает `response`.
- Миграции управляются через `python manage.py makemigrations` / `migrate`.
- Есть **админка**, которая умеет показывать и редактировать данные без отдельной панели.

#### 1.1b. Мини‑проект Django «по шагам, как для ребёнка»

Разберём последовательность чуть медленнее:

1. `django-admin startproject mysite`  
   Django создаёт папку `mysite/`:
   - `manage.py` — скрипт‑командный центр проекта;
   - пакет `mysite/`:
     - `settings.py` — все глобальные настройки;
     - `urls.py` — корневые маршруты;
     - `asgi.py` / `wsgi.py` — точки входа для ASGI/WSGI‑серверов.
2. `python manage.py startapp blog`  
   Появляется папка `blog/`:
   - `models.py` — описание таблиц БД;
   - `views.py` — функции/классы‑обработчики;
   - `admin.py` — регистрация моделей в админке;
   - `apps.py`, `migrations/` и другое служебное.
3. Добавляем `blog` в `INSTALLED_APPS` в `settings.py`:

   ```python
   INSTALLED_APPS = [
       # ...
       "blog",
   ]
   ```

   Иначе Django «не знает», что ваше приложение существует.

4. Пишем простое представление в `blog/views.py`:

   ```python
   from django.http import HttpResponse

   def hello(request):
       return HttpResponse("Hello, Django!")
   ```

   Здесь:
   - `request` — объект с данными запроса (метод, URL, заголовки, пользователь и т.п.);
   - `HttpResponse` — готовый HTTP‑ответ (тело, заголовки, код).

5. Подключаем маршрут в корневом `urls.py`:

   ```python
   from blog.views import hello

   urlpatterns = [
       path("admin/", admin.site.urls),
       path("hello/", hello),
   ]
   ```

   Это означает: «для URL `/hello/` вызвать функцию `hello`».

6. Запускаем dev‑сервер:

   ```bash
   python manage.py runserver
   ```

   Теперь `http://127.0.0.1:8000/hello/` отдаёт строку `"Hello, Django!"`.

Ещё раз в одном абзаце:

- `views.py` — **что делать** при конкретном URL;
- `urls.py` — **какой URL к чему ведёт**;
- `models.py` — **как выглядят данные в БД**;
- `settings.py` — **общие настройки проекта**.

#### 1.1c. Типичные ошибки новичка в Django

1. **Забыли добавить приложение в `INSTALLED_APPS`**:
   - миграции не создаются, модели не видны в админке;
   - решение: проверить `settings.py` и добавить `"blog"`.
2. **Подключили маршруты только в `blog/urls.py`, но забыли про корень**:
   - вы создали `blog/urls.py`, но не сделали в корне:

     ```python
     path("blog/", include("blog.urls"))
     ```

   - в результате URL просто не доступен.
3. **Меняете `models.py`, но не делаете миграции**:
   - код и БД «разъезжаются»;
   - правило: любое изменение моделей → `makemigrations` → `migrate`.
4. **Пихаете всю бизнес‑логику во view на 200 строк**:
   - тяжело тестировать, тяжело читать;
   - лучше выносить сложную логику в отдельные «сервисные» функции/классы.

#### 1.1d. Мини‑упражнения по Django

1. Сделайте view `/time/`, которое возвращает текущую дату/время как простой текст.
2. Создайте модель `Post(title: str, body: text)`, сделайте миграции и:
   - зарегистрируйте её в `admin.py`;
   - создайте одну запись через админку.
3. Напишите view `/posts/`, которое:
   - выбирает все объекты `Post`;
   - возвращает HTML‑список заголовков (можно без шаблона, через простую конкатенацию строк).

#### 1.1a. Django REST Framework и Django Ninja

Для построения **REST‑API** на Django:

- **Django REST Framework (DRF)**:
  - сериализаторы, viewsets, роутеры, аутентификация;
  - мощный, но иногда многословный;
- **Django Ninja**:
  - более «типизированный» и FastAPI‑подобный;
  - активно использует type hints и Pydantic.

Пример эндпоинта на Django Ninja (очень упрощённо):

```python
from ninja import NinjaAPI, Schema

api = NinjaAPI()

class Item(Schema):
    id: int
    name: str

@api.get("/items/{item_id}", response=Item)
def get_item(request, item_id: int):
    return Item(id=item_id, name="Example")
```

Здесь видно:

- указание типов параметров и схемы ответа;
- автоматическая генерация документации (OpenAPI/Swagger).

### 1.2. Flask: минималистичный микрофреймворк

**Идея Flask:** дать **минимальный каркас**, а всё остальное решается через расширения.

Простейшее приложение:

```python
from flask import Flask, jsonify

app = Flask(__name__)

@app.get("/hello")
def hello():
    return jsonify({"message": "Hello, Flask!"})

if __name__ == "__main__":
    app.run(debug=True)
```

Особенности:

- очень низкий «порог входа»;
- архитектуру вы придумываете **сами** (project layout, слои, сервисы);
- есть богатая экосистема расширений:
  - `Flask-SQLAlchemy`, `Flask-Login`, `Flask-Migrate`, `Flask-Admin` и т.д.

#### 1.2b. Разбор мини‑примера Flask по строкам

```python
from flask import Flask, jsonify
```

- `Flask` — класс приложения (каркас веб‑сервиса);
- `jsonify` — функция, которая превращает словари/списки в корректный JSON‑ответ (с заголовком `Content-Type: application/json`).

```python
app = Flask(__name__)
```

- создаём объект приложения;
- `__name__` говорит Flask, где «корень» проекта, чтобы искать шаблоны/статику.

```python
@app.get("/hello")
def hello():
    return jsonify({"message": "Hello, Flask!"})
```

- `@app.get("/hello")` — декоратор, который говорит:  
  «если пришёл GET‑запрос на `/hello`, вызови функцию `hello`»;
- функция `hello` ничего не знает о низкоуровневом HTTP:
  - она просто возвращает данные;
  - `jsonify` упакует их в объект ответа.

```python
if __name__ == "__main__":
    app.run(debug=True)
```

- этот блок запускается только при `python app.py`;
- `debug=True` включает:
  - авто‑перезапуск при изменении кода;
  - подробную страницу ошибок с трейсбеком.

#### 1.2c. Типичные ошибки во Flask

1. **Весь код в одном файле по 1000 строк**:
   - сначала удобно, потом невозможно ориентироваться;
   - лекарство: выносить маршруты в blueprints и модули.
2. **Смешивание бизнес‑логики и HTTP‑слоя**:
   - во view одновременно:
     - валидация входа;
     - работа с БД;
     - форматирование ответа;
   - лучше вынести «чистую логику» в отдельные функции/классы.
3. **Хранение секретов в коде**:
   - `app.config["SECRET_KEY"] = "секретик"` прямо в файле;
   - лучше читать секреты из переменных окружения или `.env`.

#### 1.2d. Мини‑упражнения по Flask

1. Добавьте эндпоинт `/ping`, который возвращает JSON `{"status": "ok"}`.
2. Добавьте эндпоинт `/echo/<name>`, который возвращает JSON `{"hello": name}`.
3. Попробуйте сделать эндпоинт `/sum?a=1&b=2`, который:
   - читает параметры `a` и `b` из query‑строки;
   - приводит их к `int`;
   - возвращает сумму в JSON.

#### 1.2a. Blueprints и структура проекта

Чтобы не держать всё в одном файле, используют **blueprints**:

```python
# blog/routes.py
from flask import Blueprint, jsonify

bp = Blueprint("blog", __name__, url_prefix="/blog")

@bp.get("/")
def list_posts():
    return jsonify({"posts": []})
```

```python
# app.py
from flask import Flask
from blog.routes import bp as blog_bp

def create_app():
    app = Flask(__name__)
    app.register_blueprint(blog_bp)
    return app

app = create_app()
```

Шаблон «фабрика приложения» (`create_app`) позволяет:

- создавать несколько экземпляров приложения с разной конфигурацией;
- удобнее писать тесты (каждый тест получает свой `app`).

### 1.3. FastAPI и Starlette: async, type hints и OpenAPI

**FastAPI** построен поверх **Starlette (ASGI‑toolkit)** и **Pydantic**.

Его сильные стороны:

- активное использование **type hints**:
  - автодокументация (OpenAPI/Swagger);
  - автоматическая валидация входа/выхода;
- асинхронность (ASGI, Uvicorn/Hypercorn);
- удобно строить **микросервисы** и API.

Пример:

```python
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()

class Item(BaseModel):
    id: int
    name: str

@app.get("/items/{item_id}", response_model=Item)
async def read_item(item_id: int):
    return Item(id=item_id, name="FastAPI example")
```

Запуск (локально):

```bash
uvicorn main:app --reload
```

Где:

- `main:app` — модуль `main.py` и переменная `app`;
- `--reload` — авто‑перезапуск при изменении кода.

#### 1.3b. Разбор примера FastAPI по шагам

```python
from fastapi import FastAPI
from pydantic import BaseModel
```

- `FastAPI` — класс приложения (ASGI‑приложение);
- `BaseModel` — базовый класс Pydantic‑моделей, описывающих структуру данных.

```python
app = FastAPI()
```

- создаём объект приложения, который потом передаётся Uvicorn’у.

```python
class Item(BaseModel):
    id: int
    name: str
```

- описываем форму данных:
  - `id` — целое число;
  - `name` — строка;
- FastAPI использует эту схему:
  - для валидации ответа;
  - для автогенерации документации.

```python
@app.get("/items/{item_id}", response_model=Item)
async def read_item(item_id: int):
    return Item(id=item_id, name="FastAPI example")
```

- `@app.get("/items/{item_id}")` — маршрут:
  - `{item_id}` — «дырка» в URL, попадает в аргумент функции `item_id`;
- `response_model=Item` говорит FastAPI:
  - «приведи результат функции к схеме `Item`»;
  - «если что‑то не сходится по типам — подними ошибку».
- `async def` — это корутина:
  - внутри неё можно вызывать `await` (например, async‑драйвера БД);
  - Uvicorn/ASGI‑сервер будет выполнять много таких корутин параллельно.

Мини‑упражнение:

1. Измените схему `Item`, добавив поле `tags: list[str] = []`, и верните в ответе несколько тегов.
2. Сделайте эндпоинт `/sum`, который принимает `a` и `b` как query‑параметры (`/sum?a=1&b=2`) и возвращает их сумму в виде Pydantic‑модели.

#### 1.3a. Отношения FastAPI, Starlette и ASGI‑сервера

- **FastAPI** — высокоуровневый фреймворк (маршруты, зависимости, схемы).
- **Starlette** — низкоуровневый ASGI‑toolkit (роутинг, middleware, background‑tasks, websockets).
- **Uvicorn** (или Hypercorn, Daphne) — ASGI‑сервер, который:
  - слушает порт;
  - принимает HTTP/WebSocket‑соединения;
  - вызывает ваше приложение (FastAPI/Starlette) как ASGI‑объект.

На пальцах:

> браузер → Uvicorn (ASGI‑сервер) → FastAPI/Starlette (ASGI‑приложение) → ваш код.

### 1.4. WSGI vs ASGI и роль серверов

**WSGI**:

- интерфейс «один запрос → один ответ»;
- по сути синхронная функция:

```python
def app(environ, start_response):
    ...
```

- пример серверов: **Gunicorn**, **uWSGI**.

**ASGI**:

- поддерживает **асинхронные** приложения;
- умеет несколько типов событий: HTTP, WebSocket, background;
- приложение — это **async‑функция/объект**, который принимает `scope`, `receive`, `send`.

Пример WSGI‑запуска Flask:

```bash
gunicorn "myapp:app" --workers 4
```

Пример ASGI‑запуска FastAPI:

```bash
uvicorn "myapp:app" --workers 4
```

Разница:

- Gunicorn по умолчанию ожидает **WSGI‑приложение** (Flask/Django‑WSGI);
- Uvicorn ожидает **ASGI‑приложение** (FastAPI/Starlette, современные async‑эндпоинты Django).

#### 1.4b. Ещё одна аналогия: телефонный звонок vs мессенджер

Представьте два способа общения:

- **Телефонный звонок (WSGI)**:
  - вы набрали номер и держите трубку;
  - пока вы говорите с одним человеком, вы не можете параллельно говорить ещё с десятью;
  - каждый звонок занимает «линию» целиком.
- **Мессенджер (ASGI)**:
  - вы отправили сообщение одному, пока ждёте ответ — пишете другому;
  - одно и то же соединение (event loop) может обслуживать множество диалогов.

В терминах Python:

- WSGI‑сервер (например, Gunicorn в синхронном режиме):
  - на каждый запрос выделяет поток/процесс и **ждёт**, пока view полностью отработает;
- ASGI‑сервер (Uvicorn/Hypercorn):
  - запускает корутину;
  - когда внутри неё `await` (ожидание сети/диска) — переключается на другие корутины.

Когда вам **точно хватит WSGI**:

- у вас классический сайт/админка;
- нет веб‑сокетов и тысяч висящих долгих соединений;
- основная нагрузка — обычные короткие запросы.

Когда **стоит смотреть в сторону ASGI**:

- много интеграций с внешними HTTP/гRPC‑сервисами;
- нужно держать множество открытых соединений (WebSocket, SSE);
- важно эффективно использовать ресурсы при большом числе одновременных запросов.

#### 1.4a. Когда выбирать что

- **Django**:
  - сложные бизнес‑приложения, админка, много форм;
  - «монолит» с мощным ORM и экосистемой.
- **Flask**:
  - маленькие сервисы, прототипы;
  - когда вы хотите **полный контроль** над архитектурой.
- **FastAPI / Litestar / Starlette**:
  - микросервисы и API с высокой нагрузкой;
  - нужен **async** (много внешних запросов, websockets, long‑polling);
  - нужны красивые схемы и автодокументация.

Мини‑упражнения:

1. Напишите на Flask эндпоинт `/ping`, который возвращает JSON `{"status": "ok"}`.
2. Напишите на FastAPI эндпоинт `/square/{x}`, который принимает целое `x` и возвращает JSON с полем `result = x*x`.

### 1.5. Запомните по §128

- Web‑фреймворк — это **слой над WSGI/ASGI**, а сервер (Gunicorn/Uvicorn) — это то, что фактически принимает соединения.
- Django даёт «всё и сразу», Flask — «каркас + расширения», FastAPI — «async + типы + OpenAPI».
- Выбор стека — это баланс между **простотой, гибкостью и требованиями к нагрузке/функционалу**.
