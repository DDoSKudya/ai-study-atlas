## 2. Очереди задач и брокеры сообщений (§129)

### 2.0. Зачем вообще очереди задач

Типичная проблема без очереди:

- пользователь делает запрос `/register`;
- вы:
  - создаёте запись в БД;
  - отправляете письмо;
  - генерируете PDF;
  - шлёте запросы в внешние API;
- запрос висит несколько секунд, а если внешний сервис «подвис», всё падает по таймауту.

С очередью задач:

1. HTTP‑эндпоинт делает **минимум**:
   - валидирует данные;
   - записывает в БД;
   - **кладёт задачу в очередь** (отправить письмо, подготовить PDF и т.п.).
2. **Воркеры** (отдельные процессы) читают задачи из очереди и выполняют их:
   - если задача упала — можно повторить;
   - нагрузка на внешние сервисы сглаживается очередью.

Преимущества:

- быстрый отклик для пользователя;
- повторяемость и устойчивость к временным проблемам;
- возможность горизонтального масштабирования воркеров.

#### 2.0a. Очереди задач «на бытовом примере»

Представьте парикмахерскую:

- вы приходите без очереди задач:
  - мастер одновременно:
    - встречает клиентов,
    - стрижёт,
    - отвечает по телефону,
    - делает запись в блокнот;
  - каждому приходится **ждать, пока мастер закончит всё сразу**.
- вы приходите с очередью:
  - администратор записывает вашу просьбу («стрижка», «окраска») в журнал (очередь);
  - мастер:
    - видит, какие задачи в очереди;
    - выполняет их одну за другой;
    - администратор может добавлять задачи **не мешая мастеру**.

В веб‑мире:

- HTTP‑обработчик — это «администратор»:
  - принимает запрос от пользователя;
  - валидирует данные;
  - кладёт задачу в очередь (в брокер сообщений);
  - быстро возвращает ответ «мы приняли ваш заказ».
- Воркер очереди — это «мастер»:
  - спокойно, в своём ритме, обрабатывает задачи;
  - может упасть и перезапуститься отдельно от веб‑части;
  - их можно поднять несколько штук для нагрузки.

Ключевая мысль:

> Очередь задач **отвязывает момент запроса пользователя**  
> от реального момента выполнения тяжёлой работы.

Мини‑упражнение:

1. Придумайте 3 действия в вашем будущем сервисе, которые не обязательно выполнять при пользователе (например, отправка отчёта на email, пересчёт статистики, обновление кэша).
2. Для каждого действия ответьте:
   - что именно пойдёт в очередь (какие аргументы функции);
   - что будет делать воркер с этими данными.

### 2.1. Celery: классический тяжёлый артиллерист

Celery — самая известная очередь задач в мире Python.

Архитектура:

- **producer** (ваше веб‑приложение) вызывает `.delay()` или `.apply_async()` на задаче;
- **broker** (Redis, RabbitMQ, др.) хранит сообщения (описания задач);
- **worker** (процесс Celery) слушает брокер и выполняет задачи;
- **result backend** (опционально) хранит результаты задач (Redis/БД и т.д.).

Минимальный пример (`celery_app.py`):

```python
from celery import Celery

app = Celery(
    "example",
    broker="redis://localhost:6379/0",
    backend="redis://localhost:6379/1",
)

@app.task
def add(x, y):
    return x + y
```

Вызов задачи из кода:

```python
from celery_app import add

result = add.delay(2, 3)   # задача уехала в очередь
print(result.id)           # id задачи
print(result.get(timeout=10))  # блокирующее ожидание результата
```

Запуск воркера:

```bash
celery -A celery_app worker --loglevel=info
```

Если разобрать этот пример «по косточкам»:

1. `app = Celery("example", broker=..., backend=...)`:
   - создаёт «объект очереди задач»;
   - `broker` — куда складывать задачи (Redis, RabbitMQ и т.д.);
   - `backend` — куда складывать результаты (можно не указывать, если они не нужны).
2. `@app.task` над `add`:
   - превращает обычную функцию `add` в **удалённую задачу**;
   - у этой функции появляется метод `.delay(...)`.
3. `add.delay(2, 3)`:
   - **не выполняет сразу** `add(2, 3)`;
   - создаёт сообщение «надо выполнить `add(2, 3)`» и отправляет его в брокер;
   - возвращает объект `AsyncResult`, у которого есть `id`, `get()`, `ready()` и т.п.
4. Воркер:
   - запускается отдельным процессом;
   - подключается к тому же брокеру;
   - читает задачи и вызывает `add(2, 3)` в своём процессе.

Очень важно:

- если воркер не запущен, `add.delay()` **не выполнит** задачу — она просто будет лежать в брокере;
- если воркер упадёт, веб‑часть всё равно продолжит принимать запросы (просто задачи будут копиться).

#### 2.1a. Celery + Django (очень кратко)

Типичный паттерн:

1. В `settings.py` описать параметры брокера/бэкенда.
2. В корне проекта создать `celery.py`, который инициализирует приложение Celery и автоподхватывает задачи из Django‑приложений.
3. В модулях приложений объявлять задачи с `@shared_task`.

Идея: задачи ощущаются как «часть Django‑проекта», в т.ч. могут использовать ORM, настройки и т.д.

#### 2.1b. Когда Celery — перебор

Celery мощный, но:

- конфигурация не самая простая;
- есть «свой мир» (beat для периодики, конфиги, мониторы).

Если вам нужно просто:

- раз в минуту дернуть пару функций;
- обрабатывать не миллион задач в минуту, а десяток —

иногда проще взять **RQ, Huey или Dramatiq**.

#### 2.1c. Типичные ошибки при использовании Celery

1. **Нет idempotency задач**:  
   задача «списать деньги» может выполниться **двежды** при ретрае → нужно проектировать задачу так, чтобы повторный запуск не ломал данные (например, хранить статусы операций).

2. **Слишком тяжёлые аргументы задач**:
   - в задачу передают огромные объекты (словарь на мегабайт) вместо айдишника;
   - решение: в задачу класть только минимальные идентификаторы и доставать данные из БД внутри воркера.

3. **Смешивание бизнес‑логики и Celery‑специфики**:
   - вся логика прячется внутри тела задачи;
   - сложнее переиспользовать/тестировать без Celery;
   - лучше иметь чистую функцию и тонкую Celery‑обёртку, которая её вызывает.

4. **Непродуманная ретрай‑политика**:
   - «ретраим всё бесконечно» → задачи висят, брокер забивается;
   - важно ограничивать количество повторов и типы ошибок, при которых есть смысл повторять.

Мини‑упражнение:

1. Возьмите функцию «отправить письмо» и попробуйте описать:
   - какие аргументы реально нужны (email, тема, тело);
   - какие из них можно/нужно логировать, а какие — нет (персональные данные).
2. Придумайте правила ретрая для этой задачи:
   - по каким типам ошибок повторять (сетевой таймаут? 5xx от сервиса?);
   - сколько раз и с каким интервалом.

### 2.2. RQ, Dramatiq, Huey, ARQ: более лёгкие очереди

Краткий обзор:

- **RQ (Redis Queue)**:
  - очень прост, завязан на Redis;
  - задачи — это просто вызовы Python‑функций;
  - есть веб‑панель для мониторинга.

- **Dramatiq**:
  - современнее Celery;
  - поддерживает Redis/RabbitMQ;
  - акцент на простоте и явности middleware (ретраи, логирование и т.п.).

- **Huey**:
  - лёгкая очередь;
  - может использовать Redis, SQLite, in‑memory;
  - удобен для небольших приложений/скриптов.

- **ARQ**:
  - async‑очередь на базе `asyncio` и Redis;
  - удобно, если у вас уже async‑стек (FastAPI, aioredis).

Пример на RQ:

```python
# tasks.py
import time

def long_task(x):
    time.sleep(5)
    return x * 2
```

```python
# producer.py
from redis import Redis
from rq import Queue
from tasks import long_task

redis_conn = Redis()
q = Queue("default", connection=redis_conn)

job = q.enqueue(long_task, 21)
print(job.id)
```

Запуск воркера:

```bash
rq worker default
```

Идея та же, что и у Celery, но конфигурация сильно проще.

#### 2.2a. Когда хватит RQ/Huey, а когда нужен Celery/Dramatiq

Условно:

- **RQ/Huey**:
  - несколько простых типов задач;
  - один брокер (Redis), одна среда;
  - не нужны сложные графики задач, отчёты, мониторинг на уровне большого кластера.
- **Celery/Dramatiq**:
  - десятки/сотни разных задач;
  - несколько брокеров/бэкендов, разные очереди (очередь «быстрых» задач, «медленных» и т.п.);
  - нужны планировщики периодических задач (beat), подробный мониторинг.

Если вы делаете **учебный проект или небольшой pet‑project**, почти всегда хватит:

- RQ + Redis;
- или Huey + Redis/SQLite.

А Celery лучше брать, когда вы **реально упрётесь** в ограничения простых очередей.

### 2.3. Брокеры: Redis, RabbitMQ, Kafka

Кратко о каждом:

- **Redis**:
  - in‑memory key‑value store;
  - поддерживает списки, pub/sub, стримы;
  - прост в установке и часто уже есть в проекте;
  - хороший брокер для Celery/RQ/Huey/ARQ.

- **RabbitMQ**:
  - полноценный AMQP‑брокер;
  - очереди, exchange, routing;
  - гибкие схемы маршрутизации (fanout, topic, direct).

- **Kafka**:
  - стриминговая платформа;
  - записи **не удаляются сразу**, а читаются потребителями в своём темпе;
  - используется для high‑throughput логов, событий, аналитики.

На уровне Python‑кода вы, как правило, не пишете «голый протокол», а:

- либо пользуетесь абстракцией очереди задач (Celery, Dramatiq и т.д.);
- либо используете специализированный клиент (`aiokafka`, `pika`, `confluent-kafka`).

#### 2.3a. Аналогии для брокеров

- **Redis как брокер** — это как:
  - быстрое табло с задачами «на сегодня» на стойке администратора;
  - записи туда/оттуда очень быстрые, но сам Redis обычно не хранит данные «навсегда».

- **RabbitMQ** — как:
  - централизованный офис, где конверты сортируют по разным папкам (очередям) и правилам маршрутизации;
  - можно сказать: «всё, что связано с биллингом, клади в эту очередь, а всё, что связано с email, — в ту».

- **Kafka** — как:
  - чёрный ящик самолёта: всё, что случилось, записывается в длинную ленту;
  - разные службы (аналитика, мониторинг, отчёты) могут читать эту ленту каждый в своём темпе, а запись при этом продолжается.

Важно: в обычных веб‑приложениях вы чаще всего начнёте с **Redis**; Kafka обычно приходит тогда, когда у вас уже много сервисов и нужно централизованно обрабатывать потоки событий.

---

### 2.4. Запомните по §129

- Очередь задач выносит тяжёлую и долгую работу **из HTTP‑запроса** в отдельный слой.
- Celery — мощный и «тяжёлый» инструмент, RQ/Huey/Dramatiq/ARQ — более лёгкие альтернативы.
- Redis/RabbitMQ/Kafka — это **инфраструктура**, в которую ходят ваши очереди: от выбора брокера зависит надёжность/масштабируемость.
