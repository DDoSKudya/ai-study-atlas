## §19. Оператор «морж» (:=) (Python 3.8+)

**Цель раздела:** использовать присваивание в выражении в условиях и comprehensions; знать, где нужны скобки; понимать область видимости переменной из `:=`.

| Термин | Определение |
|--------|-------------|
| **Walrus operator (морж)** | `:=` — присваивает и возвращает значение в одном выражении |
| **Область видимости** | Переменная из `:=` видна в теле условия/цикла и в той же области после |

**Суть:** `if (n := len(data)) > 0:` — сначала присваивание, затем сравнение; без скобок `n` получит булево значение. Морж в `for` внутри comprehension запрещён.

---

**Оператор присваивания в выражении (walrus operator, «морж»)** — **`:=`** — введён в **Python 3.8** (PEP 572 — Assignments in Expressions). Он **присваивает** значение переменной **и** возвращает это же значение как результат всего выражения. Таким образом, одно выражение одновременно и сохраняет результат (для последующего использования в теле условия или цикла), и участвует в вычислении условия — без повторного вызова функции или повторного вычисления тяжёлого выражения.

**Почему «морж»:** символ **:=** напоминает морду моржа на боку (два глаза и клыки); название закрепилось в сообществе.

**Почему это важно на практике:** типичная ситуация — «прочитать данные и проверить, не пусты ли они». Без моржа пишут **data = read_data(); if data: process(data)** — две строки и два упоминания **data**. С моржом: **if data := read_data(): process(data)** — один вызов **read_data()**, условие и тело используют одно и то же значение. В циклах **while (line := file.readline()):** избавляет от дублирования **readline** и от шаблонного **while True: line = file.readline(); if not line: break**. Это уменьшает ошибки (не забыть обновить переменную в двух местах) и делает намерение явным: «присвоить и проверить в одном месте».

### 19.1 Синтаксис и семантика

Общий вид: **`имя := выражение`**. Сначала вычисляется **выражение**, затем результат присваивается **имя**, и именно это значение возвращается как результат всего выражения **`имя := выражение`**.

```python
(n := 10)
# n равно 10; выражение (n := 10) тоже равно 10
```

**Пошагово: что происходит при if (data := read_data()): process(data).**

1. Вычисляется выражение в условии **if**.
2. Внутри него выполняется **data := read_data()**: вызывается **read_data()**, результат присваивается переменной **data**, и это же значение становится результатом выражения **:=**.
3. Условие **if** получает это значение (его truth value): пустая коллекция или **None** дадут **False**, непустые данные — **True**.
4. Если условие истинно, выполняется тело **if**, в котором **data** уже доступна — повторный вызов **read_data()** не нужен.

Итог: **read_data()** вызывается ровно один раз, и результат используется и в условии, и в теле.

**Приоритет операторов:** у **:=** низкий приоритет; он ниже, чем у сравнений (**<**, **>**, **==** и т.д.) и логических операторов. Поэтому **if n := len(data) > 0** парсится как **n := (len(data) > 0)** — в **n** попадёт булево значение, а не длина. Чтобы присвоить длину и сравнить с нулём, нужны скобки: **if (n := len(data)) > 0**.

#### Область видимости: переменная из моржа после цикла while

Переменная, которой присваивается значение через **:=** в условии **while**, принадлежит той же области видимости, что и сам цикл (функция, класс, модуль). После выхода из цикла эта переменная **остаётся** и содержит значение последней успешной итерации (последнее «истинное» значение) или значение первой «ложной» итерации, на которой цикл завершился.

**Пример:**

```python
def read_until_empty(f):
    lines = []
    while line := f.readline():
        lines.append(line)
    # line здесь — последняя прочитанная строка (пустая или последняя непустая до пустой)
    return lines
```

Если нужно использовать «последнее прочитанное значение» только внутри цикла, можно не полагаться на **line** после цикла; если нужно знать, на каком значении цикл остановился — переменная из **:=** доступна и после **while**.

**Граничный случай: морж и short-circuit (and/or).** В выражении **a and (b := f())** сначала вычисляется **a**; если **a** ложно, **f()** не вызывается и **b** не присваивается (short-circuit). Если **a** истинно, вычисляется **(b := f())** — вызывается **f()**, результат присваивается **b**, и это значение возвращается. Аналогично в **a or (b := g())**: если **a** истинно, **g()** не вызывается; если **a** ложно, выполняется **(b := g())**. Не полагайтесь на то, что переменная из **:=** в правой части **and**/ **or** всегда будет присвоена — она присваивается только если до неё дошли при вычислении.

### 19.2 Использование в условиях (if, while)

Типичный случай — вызов функции один раз, результат и проверить, и сохранить:

```python
# Без моржа: вызов дважды
data = read_data()
if data:
    process(data)

# С моржом: один вызов
if data := read_data():
    process(data)
```

В **while** удобно читать и проверять в одном заголовке:

```python
while chunk := file.read(1024):
    process(chunk)
```

### 19.3 Comprehensions (списковые, словарные и т.д.)

**`:=`** допустим внутри comprehensions (listcomp, dictcomp, setcomp, genexp), но с ограничениями: присваивание действует в области видимости, охватывающей comprehension (как в Python 3), и не создаёт «новую» переменную только внутри одной итерации. Использовать моржа в **for**-части цикла в comprehension нельзя (синтаксически запрещено).

```python
# Допустимо: в выражении-фильтре или в выражении значения
[y for x in items if (y := f(x)) is not None]

# В for-части нельзя: for (x := item) in ... — недопустимо
```

**Область видимости моржа в comprehension:** переменная, которой присваивается значение через **:=** внутри comprehension, принадлежит области видимости, **охватывающей** comprehension (функция, класс, модуль), а не «внутри» одной итерации. После завершения comprehension эта переменная сохраняет значение из **последней** итерации (или из той итерации, на которой comprehension завершился). Поэтому не полагайтесь на значение такой переменной после comprehension — оно может быть неочевидным. Используйте морж в comprehension только когда нужно отфильтровать по вычисленному значению или включить его в результат, не используя переменную после цикла.

**Пример:**

```python
# y остаётся в области видимости функции после listcomp
def demo():
    result = [y for x in range(5) if (y := x * 2) < 6]
    # result == [0, 2, 4]; после цикла y == 4 (последнее присваивание в условии)
    return result
```

Стиль: не перегружайте comprehension моржом; если логика усложняется, лучше вынести в обычный цикл.

### 19.4 Скобки в некоторых контекстах

В условиях **if** / **while** и в comprehensions выражение с **`:=`** часто нужно заключать в скобки, чтобы границы выражения были однозначны для парсера:

```python
if (n := len(data)) > 0:
    print(n)
```

Без скобок **`if n := len(data) > 0`** будет прочитано как **`n := (len(data) > 0)`** (присваивается булево значение), а не как «присвоить длину и сравнить с 0». Поэтому в таких местах скобки обязательны по смыслу.

### 19.5 Теоретическое обоснование

- **DRY:** один вызов вместо двух (получить значение и проверить/использовать).
- **Читаемость:** намерение «проверить и сохранить» выражается одной строкой.
- **PEP 572** — Assignments in Expressions: мотивация — уменьшение дублирования и более прямой код.

### 19.6 Граничные случаи

- В одном выражении можно использовать несколько моржей: `(a := 1, b := 2)` — допустимо; порядок вычисления слева направо.
- Область видимости: имя, которому присваивается значение через **`:=`**, принадлежит текущему блоку (функции, классу, модулю), а не только внутреннему блоку условия. В цикле **while** переменная из **`:=`** остаётся после выхода из цикла.

**Пошагово: что происходит при (a := 1, b := 2).** (1) Вычисляется кортеж: сначала первый элемент **a := 1** — выполняется присваивание **a = 1**, выражение **:=** возвращает **1**, поэтому первый элемент кортежа — **1**. (2) Затем второй элемент **b := 2** — **b = 2**, возвращается **2**. (3) Кортеж равен **(1, 2)**; переменные **a** и **b** доступны в текущей области видимости и равны **1** и **2**. Итог: несколько моржей в одном выражении допустимы; порядок вычисления слева направо гарантирует, что присваивания выполняются в предсказуемом порядке. Использовать такой стиль стоит только когда это действительно упрощает код (например, инициализация нескольких переменных в одной строке в тестах).

### 19.7 Типичные ошибки при использовании моржа

- **Забыли скобки в условии:** `if n := len(data) > 0` присваивает **n** результат сравнения (**True**/False), а не длину. Правильно: **`if (n := len(data)) > 0`**.
- **Использование в for-части comprehension:** **`[x for (x := item) in items]`** — синтаксическая ошибка; в **for**-части **:=** не допускается.
- **Чрезмерное увлечение моржом:** не стоит вставлять **:=** ради краткости, если это ухудшает читаемость; один вызов функции в условии — хороший кандидат, длинные выражения — лучше вынести в переменную отдельной строкой.

#### Плохо vs лучше (морж)

```python
# Плохо: вызов дважды — read_data() может быть тяжёлым или с побочным эффектом
data = read_data()
if data:
    process(data)

# Лучше: один вызов, результат и в условии, и в теле
if data := read_data():
    process(data)

# Плохо: бесконечный цикл с ручным break
while True:
    line = f.readline()
    if not line:
        break
    process(line)

# Лучше: условие и присваивание в заголовке
while line := f.readline():
    process(line)
```

#### Где морж недопустим (синтаксис)

- В **for**-части comprehension: **for (x := v) in seq** — **SyntaxError**.
- На верхнем уровне выражения без скобок в некоторых контекстах парсер может трактовать границы иначе — в условиях **if**/ **while** всегда используйте скобки вокруг **:=** при наличии сравнения/логического оператора справа.

**Запомните (§19):** морж присваивает и возвращает значение; в условии `if`/`while` при сравнении справа нужны скобки: `if (n := len(data)) > 0`; в comprehension морж допустим в выражении, но не в части `for`; переменная из моржа видна в той же области видимости, что и условие/цикл.

## §19a. Python 3.10+: Union | None, zip(strict)

**Цель:** писать Union через `X | Y`, использовать `isinstance(x, int | str)`; вызывать `zip(..., strict=True)` при необходимости проверки одинаковой длины.

**Суть:** `int | str` и `int | None` — без импорта typing; `zip(a, b, strict=True)` вызывает ValueError при разной длине.

### 19a.1 Синтаксис Union через | (PEP 604)

В **Python 3.10** в аннотациях типов можно писать **`X | Y`** вместо **`Union[X, Y]`**, и **`X | None`** вместо **`Optional[X]`**. Это распространяется и на **isinstance** / **issubclass**: второй аргумент может быть объединением типов через **`|`** (например, **isinstance(x, int | str)**).

```python
def concat(a: str | bytes, b: str | bytes) -> str | bytes:
    if isinstance(a, str) and isinstance(b, str):
        return a + b
    return bytes(a) + bytes(b)

# isinstance с объединением типов (3.10+)
if isinstance(x, int | str):
    ...

# Optional раньше и теперь
# Было: from typing import Optional; def f(x: Optional[int]) -> Optional[str]:
# Стало:
def f(x: int | None) -> str | None:
    return str(x) if x is not None else None
```

**Теоретическое обоснование:** один способ обозначить объединение типов без импорта **typing.Union** или **Optional**; аннотации становятся короче и единообразнее. В **isinstance**/ **issubclass** передаётся тот же синтаксис **X | Y** — тип проверки в runtime и аннотации совпадают по форме. Раньше для **isinstance(x, (int, str))** использовали кортеж типов; с 3.10 допустимо **isinstance(x, int | str)** (объединение типов в isinstance поддерживается с 3.10).

**Пошагово: как интерпретируется `isinstance(obj, int | str)`.** (1) Выражение **int | str** в Python 3.10+ формирует объект типа **types.UnionType** (результат операции **|** над типами). (2) **isinstance(obj, union_type)** проверяет, является ли **obj** экземпляром **любого** из типов, входящих в объединение. (3) Итог: **isinstance(5, int | str)** → **True**, **isinstance("x", int | str)** → **True**, **isinstance([], int | str)** → **False**. То же для **issubclass**: **issubclass(bool, int | str)** → **True** (bool — подкласс int).

**Граничные случаи Union в аннотациях:** можно писать цепочки: **int | str | None**, **list[int] | dict[str, int]** — интерпретируются как объединение всех перечисленных типов. **int | None** эквивалентно **Optional[int]**; **None** в объединении должно быть именно **None** (тип), а не строка **"None"**. В **isinstance** второй аргумент может быть только объединением типов (через **|**) или одним типом; нельзя передать, например, **list[int]** в старых версиях без **from **future** import annotations** — в runtime **list[int]** и **int | str** ведут себя по-разному (generic vs UnionType).

**Плохо vs лучше (Union в аннотациях):**

```python
# Плохо (работает, но длиннее): импорт и длинные имена
from typing import Optional, Union
def process(x: Optional[int], y: Union[str, bytes]) -> Optional[str]:
    ...

# Лучше (3.10+): без импорта Union/Optional, короче
def process(x: int | None, y: str | bytes) -> str | None:
    ...
```

### 19a.2 zip(..., strict=True) (Python 3.10)

**`zip(*iterables, strict=False)`** по умолчанию останавливается по кратчайшей итерабельной последовательности. При **`strict=True`** **zip** требует, чтобы все итерации имели **одинаковую длину**; иначе возбуждается **ValueError**. Это защищает от незаметных ошибок, когда одна из последовательностей короче.

```python
names = ["Alice", "Bob"]
ages = [30, 25]
list(zip(names, ages))                    # [('Alice', 30), ('Bob', 25)]
list(zip(names, ages, strict=True))      # то же

ages_long = [30, 25, 40]
list(zip(names, ages_long))              # [('Alice', 30), ('Bob', 25)] — тихо обрезано
list(zip(names, ages_long, strict=True)) # ValueError: zip() argument 2 is longer than argument 1
```

**Плохо vs лучше (zip):**

```python
# Плохо: без strict — при разной длине списков тихо обрежем по кратчайшему
names = ["Alice", "Bob"]
ages = [30, 25, 40]   # опечатка: лишний элемент
for name, age in zip(names, ages):
    process(name, age)   # третий возраст потерян без предупреждения

# Лучше: strict=True — при несовпадении длин сразу ValueError
for name, age in zip(names, ages, strict=True):
    process(name, age)
```

**Граничные случаи zip(strict=True):** пустые итерации — **zip([], [], strict=True)** даёт пустой итератор (успех). Одна непустая, остальные пустые — **ValueError** (разная длина). Если итерации имеют разную длину, **ValueError** возбуждается при **первом** исчерпании одной из последовательностей (при попытке взять следующий элемент из более короткой). **Один аргумент:** **zip([1, 2, 3], strict=True)** допустим; возвращает итератор кортежей из одного элемента: **list(zip([1, 2, 3]))** → **[(1,), (2,), (3,)]**. **strict=True** проверяет, что все переданные итерации имеют одинаковую длину; при одной итерации проверка тривиально выполняется.

**Пошагово: что происходит при zip(a, b, strict=True), когда len(a) < len(b).** (1) **zip** выдаёт пары **(a[0], b[0])**, **(a[1], b[1])**, … пока элементы есть в обоих. (2) Когда **a** исчерпан, **zip** запрашивает следующий элемент у **b**; если **b** ещё не исчерпан, **zip** обнаруживает несовпадение длин и возбуждает **ValueError: zip() argument 2 is longer than argument 1**. (3) Сообщение об ошибке указывает, какой по счёту аргумент длиннее или короче.

#### Миграция по версиям: когда какой синтаксис использовать

| Минимальная версия Python | Можно использовать                                                                                                               |
| ------------------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| **3.6**                   | f-строки **f"{x}"**                                                                                                              |
| **3.8**                   | **f"{x=}"**, морж **:=**                                                                                                         |
| **3.9**                   | **dict \|**, **dict \|=** (план §2.2)                                                                                            |
| **3.10**                  | **match/case**, **X \| Y** в типах и **isinstance**, **zip(..., strict=True)**                                                   |
| **3.11**                  | **ExceptionGroup**, **except\***, **Self**, **assert_never**, **TypeVarTuple**, **Required**/ **NotRequired**, **LiteralString** |
| **3.12**                  | **type** statement, **@override**, PEP 696 (TypeDefault), PEP 709 (comprehension inlining)                                       |
| **3.13**                  | Free-threaded (экспериментально), **@deprecated**, **-P** и др.                                                                  |

При поддержке старых версий избегайте синтаксиса и типов, появившихся в более новых версиях; при необходимости используйте **typing_extensions** для **Self**, **assert_never** и т.п. на Python 3.9–3.10.

**Запомните (§19a):** в 3.10+ `X | Y` и `X | None` заменяют `Union` и `Optional`; `isinstance(x, int | str)` допустим; `zip(..., strict=True)` требует одинаковую длину итераций, иначе ValueError.

## §19b. Python 3.11+: ExceptionGroup, except\*

**Цель:** понимать ExceptionGroup; обрабатывать часть исключений через `except*`; использовать `subgroup()`.

**Суть:** ExceptionGroup — контейнер нескольких исключений; `except* ValueError:` перехватывает только ValueError из группы, остальные пробрасываются в новой группе.

В **Python 3.11** (PEP 654) введены **ExceptionGroup** и синтаксис **`except*`** для обработки **нескольких исключений**, возникших параллельно (например, в **asyncio.TaskGroup**).

### 19b.1 ExceptionGroup

**`ExceptionGroup(msg, [exc1, exc2, ...])`** — контейнер исключений с сообщением. Содержимое доступно через атрибут **`.exceptions`** (список исключений). Метод **`.subgroup(тип)`** возвращает новую **ExceptionGroup** только с исключениями указанного типа (или **None**, если таких нет). Вложенные **ExceptionGroup** тоже могут быть среди **.exceptions**; **.subgroup(ValueError)** рекурсивно отбирает **ValueError** и из вложенных групп.

**Пример создания и разбора группы:**

```python
# Создание
eg = ExceptionGroup("несколько ошибок", [
    ValueError("bad value"),
    TypeError("bad type"),
    ValueError("another value"),
])
len(eg.exceptions)   # 3
eg.subgroup(ValueError)   # ExceptionGroup с двумя ValueError
eg.subgroup(KeyError)     # None
```

### 19b.2 except\*

**`except* TypeError:`** перехватывает **каждое** исключение типа **TypeError** внутри перехваченной **ExceptionGroup**; остальные исключения из группы формируют новую группу и пробрасываются дальше. Таким образом можно обработать часть исключений и не терять остальные.

```python
try:
    raise ExceptionGroup("ошибки", [ValueError(1), TypeError(2), ValueError(3)])
except* ValueError as eg:
    print("ValueError:", eg.exceptions)   # [ValueError(1), ValueError(3)]
except* TypeError as eg:
    print("TypeError:", eg.exceptions)   # [TypeError(2)]
```

**Пошагово: что происходит при except\* ValueError.** (1) Перехватывается **ExceptionGroup** с тремя исключениями. (2) **except\* ValueError** отбирает из группы только исключения типа **ValueError**; они попадают в **eg.exceptions**. (3) Оставшиеся исключения (здесь **TypeError(2)**) образуют новую **ExceptionGroup** и пробрасываются дальше. (4) Следующая ветка **except\* TypeError** перехватывает эту группу и обрабатывает **TypeError**. Итог: можно обработать часть типов и не «глотать» остальные.

**Метод subgroup:** **eg.subgroup(ValueError)** возвращает новую ExceptionGroup только с исключениями **ValueError** из **eg**, или **None**, если таких нет. Удобно для фильтрации группы по типу. Для нескольких типов: **eg.subgroup((ValueError, TypeError))** — вернёт группу из всех **ValueError** и **TypeError** в **eg**.

**except\* (A, B):** в одной ветке можно перехватывать несколько типов: **except\* (ValueError, TypeError):** — перехватываются все исключения, являющиеся **ValueError** или **TypeError**; остальные пробрасываются. Вложенные **ExceptionGroup** обрабатываются рекурсивно: **except\* ValueError** перехватит **ValueError** и внутри вложенных групп.

**Связь с asyncio:** при использовании **asyncio.TaskGroup** (Python 3.11+) несколько задач могут выбросить исключения «параллельно»; они собираются в **ExceptionGroup** и пробрасываются при выходе из контекста **async with task_group:**. Обработка через **except\*** позволяет обработать часть типов (например, логировать **ValueError**) и не терять остальные (пробросить **TypeError** выше).

#### Граничные случаи ExceptionGroup

- **Пустая группа:** **ExceptionGroup("msg", [])** допустим; **.exceptions** — пустой список.
- **Вложенные группы:** **ExceptionGroup** может содержать в **.exceptions** другие **ExceptionGroup**; **except\* ValueError** перехватывает **ValueError** и во вложенных группах (рекурсивно).
- **subgroup с несколькими типами:** **eg.subgroup((ValueError, TypeError))** возвращает новую группу только с исключениями этих типов; **eg.subgroup(ValueError).subgroup(TypeError)** даст группу только с **TypeError** из уже отфильтрованной подгруппы.

#### Пример: логирование части исключений и проброс остальных

```python
try:
    raise ExceptionGroup("ошибки", [
        ValueError("bad value"),
        TypeError("bad type"),
    ])
except* ValueError as eg:
    for e in eg.exceptions:
        log.error("ValueError: %s", e)
    # TypeError пробросится дальше
except* TypeError as eg:
    for e in eg.exceptions:
        log.error("TypeError: %s", e)
```

**Плохо vs лучше (ExceptionGroup):**

```python
# Плохо: перехватить всю группу и вручную разбирать по типам — легко забыть тип или потерять остальные
try:
    ...
except ExceptionGroup as eg:
    for e in eg.exceptions:
        if isinstance(e, ValueError):
            handle_value_error(e)
    # остальные исключения не проброшены — «проглочены»

# Лучше: except* по типам — обработали нужные, остальные автоматически пробрасываются
try:
    ...
except* ValueError as eg:
    for e in eg.exceptions:
        handle_value_error(e)
except* TypeError as eg:
    for e in eg.exceptions:
        handle_type_error(e)
```

**Терминология (ExceptionGroup):**

| Термин (EN)        | Пояснение                                                                                                                |
| ------------------ | ------------------------------------------------------------------------------------------------------------------------ |
| **ExceptionGroup** | Контейнер исключений с сообщением и списком **.exceptions**                                                              |
| **except\***       | Ветка перехвата: отбирает из группы только исключения указанного типа; остальные формируют новую группу и пробрасываются |
| **subgroup(тип)**  | Метод: возвращает новую ExceptionGroup только с исключениями данного типа из текущей группы, или **None**                |

**Запомните (§19b):** ExceptionGroup — контейнер нескольких исключений; `except* Тип:` перехватывает только этот тип из группы, остальные пробрасываются новой группой; `subgroup(тип)` — отфильтровать группу по типу.

## §19c. Python 3.11+: Self, Never, assert_never

**Цель:** аннотировать методы, возвращающие self, типом `Self`; использовать `Never` для недостижимого кода; применять `assert_never` для исчерпывающего match.

**Суть:** `-> Self` — тип-чекер понимает, что возвращается экземпляр подкласса; `assert_never(x)` — маркер «сюда не должны попасть»; `Never` — для функций, не возвращающих управление.

### 19c.1 typing.Self (PEP 673)

**`typing.Self`** обозначает тип экземпляра текущего класса. Удобно для методов, возвращающих **self** (например, цепочки вызовов или **copy**):

```python
from typing import Self

class Builder:
    def add(self, x: int) -> Self:
        ...
        return self

    def copy(self) -> Self:
        return type(self)(...)
```

### 19c.2 typing.Never (PEP 484)

**`typing.Never`** — тип для кода, который **никогда** не возвращает управление (бесконечный цикл, всегда исключение, выход из процесса). Полезен для аннотаций и для exhaustiveness в match. Тип-чекер считает, что после вызова функции с возвращаемым типом **Never** выполнение не продолжается — удобно для **sys.exit()**, **raise**, бесконечных циклов.

**Примеры использования Never:**

```python
from typing import Never

def never_returns() -> Never:
    raise RuntimeError("не должно вызываться")

def exit_app(code: int) -> Never:
    import sys
    sys.exit(code)
# После вызова exit_app() тип-чекер считает код недостижимым
```

**Граничные случаи Self и Never:**

- **Self в подклассе:** метод с аннотацией **-> Self** в подклассе должен возвращать экземпляр **этого** подкласса; **type(self)(...)** даёт правильный тип при копировании/фабрике. Тип-чекер проверяет совместимость возвращаемого значения с **Self**.
- **Never и циклы:** **while True:** с телом без **break**/ **return**/ **raise** в аннотированной функции можно пометить как **-> Never**, если функция действительно не возвращает управление; иначе тип-чекер может указать на недостижимый код после цикла.
- **assert_never без match:** **assert_never(x)** можно вызывать и вне **match** — например, после цепочки **if/elif**, когда все ветки должны обработать тип; тогда при появлении нового варианта типа статический анализатор и при необходимости runtime сообщат об ошибке.

### 19c.3 typing.assert_never (exhaustive matching)

**`typing.assert_never(x)`** принимает одно значение **x** и используется для проверки **исчерпывающего разбора** (exhaustive matching): статический анализатор считает, что сюда управление не должно попасть (все варианты типа уже обработаны выше). Если в runtime сюда всё же попали, **assert_never** возбуждает **TypeError**. В связке с **match/case** в последней ветке можно вызвать **assert_never(value)** — тогда при появлении нового непредусмотренного варианта тип-чекер и при необходимости runtime сообщат об ошибке.

**Пример: исчерпывающий match с assert_never**

```python
from typing import assert_never
from enum import Enum

class Color(Enum):
    RED = 1
    GREEN = 2
    BLUE = 3

def hex_code(c: Color) -> str:
    match c:
        case Color.RED:
            return "#ff0000"
        case Color.GREEN:
            return "#00ff00"
        case Color.BLUE:
            return "#0000ff"
        case _:
            assert_never(c)   # если добавили новый вариант в Enum и забыли обработать — ошибка
```

**Запомните (§19c):** `Self` — тип «экземпляр текущего класса» для возврата методов; `Never` — для функций, которые не возвращают управление; `assert_never(x)` в последней ветке match сигнализирует «сюда не должны попасть» и ловит непредусмотренные варианты.

## §19d. Python 3.11+: TypeVarTuple, Required, NotRequired

**Цель:** использовать TypeVarTuple для variadic-типов; помечать ключи TypedDict как Required/NotRequired; понимать LiteralString для безопасности SQL и т.п.

**Суть:** TypeVarTuple — переменное число типов; NotRequired[key] — необязательный ключ в TypedDict; LiteralString — строка только из литералов (защита от injection).

### 19d.1 TypeVarTuple и Unpack

**`TypeVarTuple`** (PEP 646) — переменное число типов (variadic type variable). Используется для обобщённых типов вроде **`tuple[str, *tuple[int, ...]]`** или сигнатур с переменным числом типов (например, для функций вроде **zip** или для массивов с переменной размерностью). **`Unpack[Ts]`** распаковывает TypeVarTuple в аннотации: в аннотации **tuple[*Ts]** звёздочка перед **Ts** означает «распаковать кортеж типов из TypeVarTuple».

**Полный пример: функция, возвращающая кортеж из переданных аргументов с сохранением типов**

```python
from typing import TypeVarTuple

Ts = TypeVarTuple("Ts")

def make_tuple(*args: *Ts) -> tuple[*Ts]:
    """Принимает произвольное число аргументов и возвращает кортеж тех же типов."""
    return args

# Тип-чекер выводит: make_tuple(1, "a", 3.0) -> tuple[int, str, float]
x: tuple[int, str, float] = make_tuple(1, "a", 3.0)

# Обёртка над zip с сохранением типов элементов (упрощённо)
def first_pair(*args: *Ts) -> tuple[*Ts]:
    return args
```

**Пошагово: как интерпретируется аннотация def make_tuple(*args: *Ts) -> tuple[*Ts].** (1) **Ts = TypeVarTuple("Ts")** создаёт переменную типов, способную принять **произвольное число** типов (variadic). (2) В **\*args: \*Ts** звёздочка перед **Ts** означает: типы позиционных аргументов собираются в кортеж типов **Ts** (каждый аргумент — свой тип в кортеже). (3) Возвращаемый тип **tuple[\*Ts]** — «распаковать» **Ts** в кортеж: если передали **int, str, float**, то **tuple[int, str, float]**. (4) Итог: тип-чекер знает точный тип возвращаемого кортежа по типам аргументов; без TypeVarTuple пришлось бы писать перегрузки (**@overload**) для разного числа аргументов.

**Граничный случай:** TypeVarTuple используется в аннотациях; в runtime **Ts** и **\*Ts** не создают отдельных объектов — это информация для статического анализатора. Для обычных функций с **\*args** без TypeVarTuple тип возврата часто задают как **tuple[Any, ...]**; TypeVarTuple даёт точную типизацию при переменном числе типов.

### 19d.2 TypedDict: Required и NotRequired (PEP 655)

В **TypedDict** все ключи по умолчанию считаются обязательными при создании словаря. **`Required[key]`** и **`NotRequired[key]`** явно помечают ключ как обязательный или необязательный. Это даёт точное описание словарей, у которых часть полей может отсутствовать (например, конфигурация с опциональными параметрами).

```python
from typing import TypedDict, Required, NotRequired

class Config(TypedDict):
    host: str                    # обязательный
    port: int                    # обязательный
    timeout: NotRequired[float]   # опциональный
    debug: NotRequired[bool]     # опциональный

c1: Config = {"host": "localhost", "port": 8080}  # OK
c2: Config = {"host": "x", "port": 80, "timeout": 5.0}  # OK
# Без host или port — ошибка типов
```

**Граничные случаи Required/NotRequired:** без **NotRequired** все ключи TypedDict обязательны при создании экземпляра; тип-чекер сообщит об ошибке, если ключа нет. **Required[key]** явно помечает ключ обязательным (по умолчанию все ключи и так обязательны; **Required** полезен при смеси с **NotRequired** для ясности). При наследовании TypedDict дочерний класс может добавлять **NotRequired** ключи; переопределить обязательный ключ на необязательный через **NotRequired** в подклассе допустимо.

### 19d.3 LiteralString (PEP 675)

**`LiteralString`** — тип для строк, составленных только из литералов (и конкатенаций таких строк). Используется для защиты от SQL injection: API может требовать **LiteralString**, чтобы в запрос не подставлялась произвольная строка из переменной. Тип-чекер (mypy, pyright) проверяет, что в функцию, принимающую **LiteralString**, передаются только строковые литералы или конкатенации литералов, а не результат **input()** или переменная с произвольной строкой.

```python
from typing import LiteralString

def execute_query(sql: LiteralString) -> None:
    ...

execute_query("SELECT * FROM users")   # OK
execute_query("SELECT * FROM " + "users")   # OK (конкатенация литералов)
user_input = input()
execute_query(user_input)   # Ошибка типов: user_input не LiteralString
```

**Запомните (§19d):** TypeVarTuple — переменное число типов, `tuple[*Ts]`; в TypedDict ключ необязателен с `NotRequired`; LiteralString — защита от подстановки произвольной строки (например, в SQL).

## §19e. Python 3.12+: type statement, @override

**Цель:** объявлять псевдонимы типов через `type`; помечать переопределения метода декоратором `@override`; знать про PEP 696 и PEP 709.

**Суть:** `type Point = tuple[float, float]` — псевдоним для аннотаций; `@override` — статический анализатор проверяет наличие метода в базовом классе.

### 19e.1 type statement (PEP 695)

В **Python 3.12** введён оператор **`type`** для объявления псевдонимов типов (type alias):

```python
type Point = tuple[float, float]
type IntOrStr = int | str
type StringMap = dict[str, str]
```

Так псевдонимы объявляются на уровне **оператора**, а не присваивания: имя **Point** в аннотациях типов обозначает **tuple[float, float]**, но в runtime **Point** не создаётся как переменная с этим значением (в отличие от **Point = tuple[float, float]**, где **Point** — обычная переменная, хранящая класс **tuple**). Это устраняет путаницу между «типом для аннотаций» и «переменной в runtime» и позволяет использовать одно и то же имя и в аннотациях, и при необходимости в коде (например, для isinstance), при этом семантика «это псевдоним типа» явная.

**Граничные случаи type statement:** псевдоним, объявленный через **type**, доступен в аннотациях и в том же модуле; для использования в runtime (например, **isinstance(x, Point)**) в Python 3.12 **Point** после **type Point = ...** ссылается на правую часть (тип), так что **isinstance((1.0, 2.0), Point)** эквивалентно **isinstance((1.0, 2.0), tuple[float, float])** (в 3.12+). Циклические псевдонимы типов (тип A ссылается на B, B на A) могут требовать **ForwardRef** или строковые аннотации в старых подходах; **type** statement обрабатывается анализатором типов по правилам PEP 695.

### 19e.2 @override (PEP 698)

Декоратор **`@override`** (из **typing**) помечает метод как переопределяющий метод базового класса. Статические анализаторы (mypy, pyright) могут проверять, что такой метод действительно переопределяет существующий метод предка; иначе — предупреждение. Это защищает от опечаток в имени метода и от «мнимого» переопределения, когда метод в базе переименовали, а в наследнике забыли обновить.

```python
from typing import override

class Base:
    def run(self) -> str:
        return "base"

class Derived(Base):
    @override
    def run(self) -> str:
        return "derived"
# Если бы написали def ran(self) с @override — анализатор сообщил бы, что ran не переопределяет метод базового класса
```

### 19e.3 PEP 696: TypeDefault (Python 3.12+)

**`TypeDefault`** (PEP 696) — способ задать **значение по умолчанию** для **TypeVar**: тип-переменная может быть привязана к типу по умолчанию, если тип не удаётся вывести из контекста. Используется в обобщённых классах и функциях, когда нужно указать «если тип не выведен — считать таким-то».

```python
from typing import TypeVar

# TypeVar с default (3.12+): если T не выводится, используется str
T = TypeVar("T", default=str)

def identity(x: T) -> T:
    return x

identity("hello")   # T = str, возврат str
identity(42)        # T = int, возврат int
# default=str используется, когда тип T не удаётся вывести из контекста (например, в некоторых обобщённых классах)
```

В аннотациях **default=...** задаёт тип по умолчанию для TypeVar; статические анализаторы (mypy, pyright) используют его при выводе типов. Подробности — PEP 696 и документация **typing**.

### 19e.4 PEP 709: comprehension inlining (3.12)

В Python 3.12 list/dict/set comprehensions оптимизированы (inlining), что уменьшает накладные расходы на создание фрейма итерации. Это внутренняя оптимизация: синтаксис не меняется, но выполнение быстрее.

#### type statement vs присваивание псевдонима (Плохо vs лучше)

```python
# Присваивание: Point в runtime — переменная, хранящая класс tuple
Point = tuple[float, float]
def distance(p: Point) -> float: ...

# type statement (3.12+): Point — псевдоним типа, не создаётся в runtime как переменная
type Point = tuple[float, float]
def distance(p: Point) -> float: ...
```

При присваивании **Point = tuple[float, float]** имя **Point** в runtime указывает на объект **tuple** (класс). При **type Point = ...** имя **Point** в аннотациях интерпретируется как псевдоним типа; семантика «это тип, а не значение» явная. Для совместимости с Python &lt; 3.12 используйте присваивание или **typing.TypeAlias**.

**Запомните (§19e):** в 3.12+ `type Имя = ...` — псевдоним типа; `@override` помечает переопределение метода для проверки анализатором; PEP 709 — внутренняя оптимизация comprehensions.

## §19f. Python 3.13+

**Цель:** ориентироваться в free-threaded Python, `@deprecated`, флаге `-P` и прочих нововведениях 3.13.

**Суть:** free-threaded — сборка без GIL (`-X gil=0`); `@deprecated("Используйте new_func()")` — пометка устаревшего кода; `-P` — не загружать site-packages в sys.path.

### 19f.1 Free-threaded Python (экспериментально, PEP 703)

Режим **без GIL** (global interpreter lock): интерпретатор можно собрать с опцией и запускать с **`python -X gil=0`**, чтобы несколько потоков выполняли Python-байтод одновременно. На 2025 год это экспериментальная возможность.

### 19f.2 @typing.deprecated (PEP 702)

Пометка устаревших сущностей (функций, классов и т.д.) для предупреждений в статических анализаторах и в runtime. При вызове помеченной функции или использовании класса в Python 3.13+ можно выдавать **DeprecationWarning** (или подавлять, в зависимости от настроек).

**Пример: помеченная устаревшая функция и класс**

```python
from typing import deprecated

@deprecated("Используйте new_helper() вместо helper()")
def helper(x: int) -> int:
    return x + 1

@deprecated("Класс OldConfig устарел, используйте Config")
class OldConfig:
    pass

helper(1)       # При вызове может быть выведено предупреждение об устаревании
c = OldConfig() # Аналогично для класса
```

Статические анализаторы (mypy, pyright) могут показывать предупреждение при обращении к помеченным сущностям; в runtime поведение зависит от версии Python и настроек предупреждений (**warnings.filterwarnings**).

### 19f.3 PEP 719: python -P

Флаг **`-P`** отключает автоматическую загрузку **site-packages** для большей изоляции окружения. Удобно для проверки, что скрипт не опирается на пакеты из пользовательского **site-packages** (например, в CI или при сборке).

**Пример запуска:**

```text
python -P -c "import sys; print(len(sys.path))"
```

С **-P** в **sys.path** не добавляется каталог **site-packages**; количество путей будет меньше, чем без **-P**. Скрипт, который при **-P** падает с **ModuleNotFoundError**, зависит от установленных в окружении пакетов — их нужно явно указать в зависимостях или установить в виртуальное окружение.

- **PEP 696 (TypeDefault)** — значение по умолчанию для TypeVar (Python 3.12+); см. §19e.3.
- Актуальный список изменений — раздел **What's New** в документации Python для каждой версии; для 3.13 и новее стоит сверяться с официальной документацией.
