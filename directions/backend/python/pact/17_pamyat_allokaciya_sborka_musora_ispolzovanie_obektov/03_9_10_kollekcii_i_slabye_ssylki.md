## 9. §95. Память коллекций: list, dict, set

### 9.1. Списки (list) — динамический массив

Список в CPython — это **динамический массив указателей**:

- внутри — **массив ссылок** на элементы;
- список хранит:
  - **текущую длину** (len),
  - **ёмкость** (capacity) — сколько элементов может поместиться без перевыделения.

При добавлении элементов:

- если ещё есть свободное место, элементы добавляются **без аллокации**;
- если места не хватает, список:
  - выделяет **больший** блок памяти,
  - копирует туда существующие ссылки,
  - добавляет новые,
  - старый блок освобождает.

При этом обычно используется стратегия **over-allocation**: при росте список увеличивает ёмкость **с запасом**, чтобы `append` был амортизированно O(1).

Проверить размер:

```python
import sys

lst = []
for i in range(10):
    lst.append(i)
    print(i, sys.getsizeof(lst))
```

Вы увидите, что размер `sys.getsizeof(lst)` увеличивается **не на каждом шаге** — именно за счёт предварительного выделения дополнительных слотов.

### 9.2. Словари (dict) — хеш-таблица

Словарь — одна из самых важных и оптимизированных структур в Python.

Ключевые моменты:

- dict — это **хеш-таблица**:
  - ключи хешируются (`__hash__`);
  - по хешу находится «корзина» (bucket) в внутреннем массиве;
  - при коллизиях используются дополнительные шаги поиска.
- Внутри хранится:
  - **массив ячеек** (слот, где указатель на запись),
  - **массив записей** с ключом, значением и хешем.

**Load factor**:

- словарь расширяет свою таблицу, когда он становится слишком заполненным;
- это тоже **over-allocation** — есть «свободные» слоты, чтобы операции поиска и вставки были быстрыми.

Поэтому:

- `len(d)` может быть гораздо меньше, чем количество доступных слотов внутри;
- `sys.getsizeof(d)` может быть заметно больше ожидаемого.

`d.clear()`:

- удаляет все элементы,
- но может **не уменьшать** внутреннюю ёмкость таблицы (слоты остаются выделенными);
- это быстрый способ очистить словарь, но не всегда освобождает память ОС.

Если нужна именно **перевыделенная** (меньшая) структура:

```python
d.clear()         # очистка, но ёмкость может остаться
d = {}            # старый словарь становится кандидатом на GC, новый — маленький
```

### 9.3. Множества (set)

Множества реализованы **аналогично словарям**, но:

- хранят только ключи (значение фактически не используется);
- многие особенности памяти такие же, как у dict:
  - хеш-таблица,
  - load factor,
  - over-allocation.

### 9.4. `dict.fromkeys` и общий объект значений

`dict.fromkeys(keys, value)` создаёт словарь, где:

- все ключи из `keys`,
- **одно и то же значение** `value` для всех них.

```python
d = dict.fromkeys(["a", "b", "c"], [])
print(d)  # {'a': [], 'b': [], 'c': []}

d["a"].append(1)
print(d)  # {'a': [1], 'b': [1], 'c': [1]}  ← ОДИН общий список!
```

Это **экономит память**, потому что:

- создаётся **один** объект `[]`,
- все значения в словаре — ссылки на этот один список.

Но:

- если значение **мутабельно**, изменения через один ключ будут видны через другие;
- для иммутабельных значений (числа, строки, кортежи) это безопасно и иногда удобно.

### 9.5. Запомните (память коллекций)

| Структура | Особенность памяти | Практический вывод |
|-----------|--------------------|---------------------|
| `list` | динамический массив с запасом ёмкости | `append` амортизированно O(1), `sys.getsizeof` может быть больше, чем «минимум» |
| `dict`/`set` | хеш-таблица с load factor и свободными слотами | `clear` не всегда уменьшает память, `len` не отражает количество слотов |
| `dict.fromkeys` | одно общее значение для всех ключей | удобно для иммутабельных значений, опасно для mutable |

## 10. §96. Слабые ссылки (weakref)

### 10.1. Зачем нужны слабые ссылки

Иногда нам нужно сделать вот что:

- «знать про объект», пока он **жив**,
- **но не мешать** сборщику мусора его удалить, если все остальные ссылки исчезли.

Примеры:

- **кеши**: хотим кэшировать дорогие в создании объекты, но не хотим, чтобы сам кеш удерживал их в памяти вечно;
- **обратные ссылки** в графах объектов (родитель ↔︎ дети), чтобы избежать циклов;
- **наблюдатели/callback’и** — не хотим, чтобы подписка мешала объекту быть собранным.

Решение — **слабые ссылки**: они **не увеличивают refcount**.

Ещё одна формулировка:

- обычная (сильная) ссылка говорит: «**пока я есть, объект должен жить**»;
- слабая ссылка говорит: «**если объект жив — я бы хотел к нему обратиться, но если его соберут, я не против**».

### 10.2. `weakref.ref` и `weakref.proxy`

Начнём с базового объекта слабой ссылки — `weakref.ref`.

```python
import weakref

class Data:
    def __init__(self, value):
        self.value = value

obj = Data(42)
ref = weakref.ref(obj)      # слабая ссылка

print(ref)          # <weakref at ...; to 'Data' at ...>
print(ref())        # <__main__.Data object at ...>  (пока объект жив)

del obj
print(ref())        # None  — объект собран GC, слабая ссылка обнулилась
```

Особенности:

- `weakref.ref(obj)` возвращает **объект слабой ссылки**;
- вызов `ref()`:
  - возвращает объект, если он ещё жив;
  - возвращает `None`, если объект уже собран.

`weakref.proxy(obj)`:

- создаёт **прокси-объект**, который «прозрачно» перенаправляет обращения к оригиналу;
- при попытке обратиться к прокси после уничтожения оригинала будет исключение.

#### 10.2.1. Аналогия «записка с адресом vs живой контакт»

Представь:

- сильная ссылка — это **живая связь**: человек, который всегда знает, где находится объект, и тем самым «заставляет» объект существовать;
- слабая ссылка — это **записка с адресом**, приклеенная к доске:
  - пока дом по этому адресу стоит — ты можешь прийти;
  - если дом снесли — по записке ты увидишь, что больше идти некуда (в терминах Python: `ref()` вернёт `None`).

Слабая ссылка **не мешает «снести дом»**, то есть не мешает GC собрать объект.

#### 10.2.2. Мини-эксперимент: посмотреть момент, когда слабая ссылка обнуляется

```python
import weakref
import gc

class Data:
    def __init__(self, value):
        self.value = value

def demo():
    obj = Data(42)
    ref = weakref.ref(obj)
    print("Внутри функции, ref():", ref())
    return ref  # возвращаем только слабую ссылку

ref_outside = demo()

gc.collect()  # просим GC убрать всё, что можно

print("Снаружи, ref_outside():", ref_outside())
```

Разбор:

1. Внутри `demo` мы создаём объект `Data(42)` и сильную ссылку `obj`.
2. Создаём слабую ссылку `ref`. Она **не увеличивает refcount**.
3. Возвращаем наружу **только слабую ссылку** `ref_outside`.
4. Когда `demo` заканчивается, локальная переменная `obj` уничтожается → **сильных ссылок на объект больше нет**.
5. После `gc.collect()` объект может быть собран.
6. Вызов `ref_outside()` вернёт `None` — объект исчез.

Ключ: слабая ссылка позволяет «узнать», что объект ещё жив (и получить его), но **не держит его в живых**.

### 10.3. Коллекции на слабых ссылках

Модуль `weakref` предоставляет структуры данных:

- `weakref.WeakKeyDictionary` — словарь, в котором **ключи хранятся как слабые ссылки**;
- `weakref.WeakValueDictionary` — словарь, в котором **значения — слабые ссылки**;
- `weakref.WeakSet` — множество слабых ссылок.

Пример использования `WeakValueDictionary` для кеша:

```python
import weakref

class User:
    def __init__(self, user_id):
        self.user_id = user_id

_cache = weakref.WeakValueDictionary()

def get_user(user_id):
    user = _cache.get(user_id)
    if user is None:
        user = User(user_id)
        _cache[user_id] = user
    return user
```

Здесь:

- пока на объект `User` есть обычные (сильные) ссылки — он жив;
- когда все обычные ссылки исчезают, GC может его собрать;
- соответствующая запись в `WeakValueDictionary` автоматически исчезнет.

То есть кеш **не удерживает** объекты насильно:

- кеш знает о них «пока они живут»;
- когда программа перестаёт ими пользоваться (сильных ссылок нет) — GC их собирает, и запись из кеша пропадает.

Это очень удобно для «ленивых кешей», которые **не растут бесконечно**.

### 10.4. Ограничения weakref

- Объект должен поддерживать слабые ссылки:
  - большинство встроенных типов (int, str, tuple) **не поддерживают** weakref;
  - пользовательские классы обычно поддерживают, но если у них есть `__slots__`, нужно явно добавить слот `__weakref__`.

```python
class A:
    __slots__ = ("x", "__weakref__")  # разрешаем слабые ссылки

    def __init__(self, x):
        self.x = x
```

Здесь:

- `__slots__` говорит интерпретатору: «у этого класса есть только атрибут `x` и служебное поле для weakref»;
- если убрать `__weakref__`, попытка создать слабую ссылку на объект `A` упадёт с ошибкой.

Про это важно помнить, если вы активно используете `__slots__` и планируете применять `weakref`.

### 10.5. Запомните (weakref)

| Использование | Инструмент |
|---------------|-----------|
| Кеши на объекты, которые можно собрать | `weakref.WeakValueDictionary` |
| Карта «объект → данные о нём», не блокирующая GC | `weakref.WeakKeyDictionary` |
| Безопасный финализатор вместо `__del__` | `weakref.finalize` |
| Проверить, жив ли объект | `ref() is not None` для `weakref.ref` |
