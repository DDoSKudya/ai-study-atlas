## 5. §91. Счётчик ссылок (reference counting)

### 5.1. Цель раздела

Понять:

- что такое **ссылка** на объект;
- как CPython считает количество ссылок (**refcount**);
- какие операции **увеличивают** и **уменьшают** счётчик ссылок;
- почему объект иногда освобождается **моментально** после выхода из области видимости.

### 5.2. Термины

| Термин | Определение простыми словами |
|--------|------------------------------|
| **Объект** | Любая сущность в Python: число, строка, список, функция, экземпляр класса и т.д. Всё — объекты. |
| **Ссылка** | «Стрелочка»/указатель на объект: имя переменной, элемент списка, значение в словаре, аргумент функции и т.п. |
| **Счётчик ссылок (refcount)** | Число в структуре объекта, показывающее, **сколько ссылок** сейчас ведут к этому объекту. |
| **Освобождение объекта** | Момент, когда счётчик ссылок падает до 0, и CPython освобождает память, вызывая деструктор и аллокатор. |

Внутри CPython большинство объектов начинаются с структуры примерно вида:

- `ob_refcnt` — счётчик ссылок;
- `ob_type` — указатель на объект типа (класс);
- плюс дополнительные поля.

### 5.3. Как работает счётчик ссылок (пошагово)

**Идея (в одном предложении):** каждый раз, когда в программе появляется **новая «стрелочка» к объекту** — счётчик ссылок этого объекта увеличивается; когда стрелочка исчезает — счётчик уменьшается. Как только счётчик становится 0, объект больше **нигде не достижим**, и его можно **немедленно выкинуть из памяти**.

#### 5.3.1. Аналогия со стикерами на коробке

Представьте, что у нас есть **коробка** — это объект в памяти (список, словарь, число — не важно).  
На коробку можно клеить **стикеры с именами**:

- на один и тот же объект можно наклеить стикеры `obj`, `a`, `b`, `lst[0]`, `d["key"]` и т.д.;
- каждый стикер — это **одна ссылка**;
- внутри коробки (в заголовке объекта) лежит счётчик: «сколько стикеров сейчас наклеено».

Пока на коробке **хотя бы один стикер** — её **нельзя выбрасывать**: кто-то ещё ей пользуется.  
Когда стикеры отклеили все — коробка никому не нужна → её можно выкинуть (освободить память).

#### 5.3.2. Мини-эксперимент с `sys.getrefcount`

Рассмотрим пример:

```python
import sys

obj = []  # создаём пустой список
print(sys.getrefcount(obj))  # например, 2
```

Почему 2, а не 1?

- Одна ссылка — это **переменная `obj`** (один стикер).
- Вторая — **временная ссылка**, которую создаёт сама функция `getrefcount`, когда передаёт `obj` внутрь (ещё один стикер на время вызова).

Как мысленно это представить:

1. Мы написали `obj = []`. Создалась коробка (список), на неё наклеили стикер `obj`. Счётчик внутри коробки = 1.
2. Мы вызываем `sys.getrefcount(obj)`. Перед вызовом Python говорит: «мне надо передать объект в функцию» и кладёт **ещё один стикер** для аргумента функции.
3. Внутри `getrefcount` считывается значение счётчика (1 от `obj` + 1 от аргумента функции) → 2.
4. После выхода из `getrefcount` временный стикер от аргумента убирается, счётчик снова становится 1.

**Важно:** `sys.getrefcount(x)` **всегда возвращает на 1 больше**, чем «реальное» количество ссылок в момент вызова, потому что **сама функция добавляет к объекту ещё одну временную ссылку**.

Если хочешь почувствовать это руками, можно сделать так:

```python
import sys

obj = []
print("Только obj:", sys.getrefcount(obj))  # допустим, 2

tmp = obj
print("obj и tmp:", sys.getrefcount(obj))   # стало 3

del tmp
print("Снова только obj:", sys.getrefcount(obj))  # вернулось к 2
```

Здесь мы явно видим: **каждая новая переменная**, указывающая на объект, увеличивает счётчик, а `del` уменьшает его.

Добавим ещё ссылок:

```python
import sys

obj = []
print("1:", sys.getrefcount(obj))  # допустим, 2

a = obj          # новая ссылка
print("2:", sys.getrefcount(obj))  # стало 3

lst = [obj, obj] # ещё две ссылки в списке
print("3:", sys.getrefcount(obj))  # стало 5 (obj, a, lst[0], lst[1], временная из getrefcount)
```

Разберём, что именно произошло, **по шагам**:

1. После `obj = []` у нас:
   - один стикер `obj` на коробке;
   - когда вызываем `getrefcount`, добавляется временный стикер аргумента.
   - внутри функции мы видим: 2.
2. После `a = obj` коробка всё та же, но:
   - стикеры: `obj`, `a`, плюс временный из `getrefcount`;
   - внутри функции мы видим: 3.
3. После `lst = [obj, obj]`:
   - список `lst` — это новая коробка, внутри которой лежат ещё **две стрелочки** на исходный объект;
   - теперь у исходного объекта есть ссылки: `obj`, `a`, `lst[0]`, `lst[1]`, плюс временная ссылка внутри `getrefcount` → всего 5.

Если тебе пока сложно «увидеть это в голове» — это нормально. Можно просто запомнить **правило**:

- «Каждый раз, когда я куда-то кладу объект (в переменную, в элемент списка, в значение словаря, в аргумент функции) — я добавляю к нему ссылку, а значит, увеличиваю счётчик».

#### 5.3.3. Когда счётчик уменьшается

Счётчик уменьшается, когда **стрелочка исчезает**:

- когда имя **перестаёт ссылаться** на объект (`del name`, присваивание другого объекта);
- когда объект **удаляется из контейнера** (`list.remove`, `dict.pop`, очистка списка/словаря);
- когда **функция заканчивает выполнение** — все её локальные переменные выходят из области видимости.

Пример:

```python
import sys

def demo():
    obj = []
    print("Внутри функции:", sys.getrefcount(obj))  # например, 2
    return obj

returned = demo()
# Здесь у нас есть только ссылка returned (и, возможно, ещё где-то)
print("Снаружи:", sys.getrefcount(returned))
```

Что здесь происходит:

1. Внутри `demo` создаётся новый список и переменная `obj`, которая на него указывает.
2. При вызове `sys.getrefcount(obj)`:
   - у списка есть ссылка `obj`,
   - плюс временная ссылка аргумента функции → видим 2.
3. Функция возвращает объект: за пределами функции появляется новая ссылка `returned`.
4. Функция заканчивает выполнение:
   - **локальная переменная `obj` уничтожается**;
   - её стрелочка отклеивается → счётчик уменьшается на 1.
5. Снаружи у нас осталась только ссылка `returned` (и временная ссылка внутри `getrefcount`).

Если мысленно представить: внутри функции коробка стоит на столе, и к ней приклеен стикер `obj`. Когда функция завершилась, **стол с функцией убрали**, стикер `obj` выбросили, но коробку перед этим успели передать наружу и наклеить на неё новый стикер `returned`.

### 5.4. Почему объект освобождается сразу

В CPython действует **синхронная модель уничтожения**: когда счётчик ссылок объекта падает до нуля, объект **освобождается сразу же**, а не «когда-нибудь потом».

По шагам:

1. Внутри объекта хранится число `ob_refcnt` — сколько ссылок на него сейчас есть.
2. Как только какая-то операция убирает **последнюю** ссылку (стрелочку), `ob_refcnt` становится равен 0.
3. В этот момент интерпретатор:
   - вызывает финализатор C-структуры (и, при необходимости, метод `__del__` у объекта);
   - отдаёт память этого объекта обратно аллокатору (`pymalloc` или системному).

То есть логика такая: **«ноль ссылок → этот объект больше никому не нужен → можно сразу уничтожать»**.

Пример:

```python
class Tracked:
    def __del__(self):
        print("Объект уничтожен")

def create_and_die():
    x = Tracked()
    print("Создали объект")

create_and_die()
print("Функция завершена")
```

Что тут происходит в терминах ссылок:

1. При создании `Tracked()` создаётся новый объект в памяти, на него клеят стикер `x`.
2. Внутри функции `create_and_die` есть одна ссылка на объект — `x`.
3. Функция печатает «Создали объект» и завершает выполнение.
4. При выходе из функции:
   - локальная переменная `x` уничтожается,
   - стикер `x` отклеивается от объекта,
   - счётчик ссылок падает до 0.
5. Интерпретатор видит refcount = 0 и сразу:
   - вызывает `__del__` → печатается «Объект уничтожен»;
   - освобождает память объекта.
6. После этого управление возвращается в глобальный код, который печатает «Функция завершена».

В большинстве случаев в CPython вы увидите именно этот порядок:

```text
Создали объект
Объект уничтожен
Функция завершена
```

То есть объект уничтожился **сразу после выхода из функции**, когда последняя ссылка на него была потеряна.

**Важно:** это поведение — **особенность именно CPython** (стандартная реализация).  
В других реализациях Python (PyPy, Jython и т.п.):

- может быть только **отложенный сборщик мусора** (mark-and-sweep, как в Java);
- объект может жить **дольше**, чем перестал быть нужным с точки зрения кода;
- `__del__` (или аналогичные механизмы) могут вызываться **не сразу**.

Поэтому нельзя писать код, который **жёстко полагается** на то, что `__del__` сработает прямо в этот момент — но для понимания устройства CPython полезно знать, что здесь уничтожение обычно происходит **немедленно при обнулении счётчика ссылок**.

### 5.5. Циклические ссылки и ограничения счётчика ссылок

Счётчик ссылок **не может** сам по себе освободить **цикл ссылок**:

```python
a = []
b = []

a.append(b)
b.append(a)

del a
del b
```

Здесь у нас два списка:

- `a` ссылается на `b`;
- `b` ссылается на `a`.

Даже когда мы делаем `del a` и `del b`, сами объекты списков всё ещё **ссылаются друг на друга**, их refcount **> 0**, и refcount **никогда не упадёт до нуля** — здесь нужен **циклический сборщик мусора** (см. §92).

#### 5.5.1. Разбор «на пальцах»: почему `del` не «удаляет объект»

Очень важная мысль:

- `del a` **не удаляет объект** в памяти.
- `del a` удаляет **имя `a`** (стикер/стрелочку), которая указывала на объект.

Если после `del a` на объект всё ещё указывает хоть что-то (например, `b[0]` внутри цикла),
то объект остаётся жив.

То есть:

- `del` = «убрать ссылку»,
- а не «уничтожить объект».

#### 5.5.2. Мини-эксперимент: создать цикл и посмотреть, что делает `gc.collect()`

```python
import gc

gc.collect()  # очистим «хвосты» на старте

a = []
b = []
a.append(b)
b.append(a)

# Убираем внешние ссылки-имена.
del a
del b

# Теперь снаружи у нас нет переменных a и b,
# но сами списки всё ещё ссылаются друг на друга.

collected = gc.collect()
print("GC collected:", collected)
```

Что тут важно почувствовать:

- без вызова `gc.collect()` цикл может висеть до следующего автоматического прохода GC;
- `gc.collect()` заставляет GC «прямо сейчас» поискать недостижимые циклы и убрать их.

### 5.6. Запомните (refcount)

| Ситуация | Что происходит |
|----------|----------------|
| `x = obj` | refcount увеличивается (появилась новая ссылка) |
| `lst.append(obj)` | refcount увеличивается (ссылка в списке) |
| `del x` | refcount уменьшается (исчезла ссылка-имя) |
| Удаление элемента из списка/словаря | refcount уменьшается (исчезла ссылка в контейнере) |
| Выход из функции | все локальные имена уничтожаются, refcount соответствующих объектов уменьшается |
| refcount = 0 | объект немедленно освобождается (в CPython) |
| Цикл ссылок | refcount не падает до 0, объект «застревает» до прохода GC |

## 6. §92. Циклический сборщик мусора (gc)

### 6.1. Зачем нужен циклический GC

Счётчик ссылок отлично справляется с большинством случаев, но **не умеет** работать с одной важной ситуацией:  
когда у нас есть **группа объектов, которые ссылаются друг на друга, но «снаружи» к ним больше никто не обращается**.

Такой случай называется **циклом ссылок**:

- объект `A` ссылается на `B`;
- `B` ссылается на `C`;
- `C` снова ссылается на `A`;
- снаружи (из глобальных переменных, стека функций и т.п.) на них больше **нет ни одной ссылки**.

Если смотреть только на refcount:

- у каждого из объектов счётчик **> 0**, потому что они держат ссылки друг на друга;
- с точки зрения простого счётчика ссылок «всё в порядке», хотя **добраться до этих объектов из программы уже нельзя**;
- на самом деле это **мусор, который занимает память впустую**.

Счётчик ссылок **не понимает концепции «достижимости» от корней программы** — он только считает количество стрелочек.  
Поэтому в CPython есть **дополнительный сборщик мусора** — модуль `gc`, который умеет:

- смотреть на объекты как на **граф** (узлы — объекты, рёбра — ссылки);
- находить **группы объектов, которые больше не достижимы от корней**;
- освобождать их, даже если их счётчики ссылок не равны нулю.

Основные идеи:

- GC периодически **сканирует граф объектов**, которые он отслеживает;
- находит объекты, которые **не достижимы** от корневых ссылок (стек, глобальные переменные, регистры и т.п.);
- освобождает их, даже если у них refcount > 0, но все ссылки — только из того же «мертвого» цикла.

### 6.2. Модуль `gc`: основные функции

Разберём функции модуля `gc` **не просто по списку, а по смыслу** — что и когда ими можно делать.

#### 6.2.1. Базовый «набор инструментов»

```python
import gc

gc.get_threshold()  # пороги для поколений (tuple из 3 чисел)
gc.get_count()      # текущие счётчики для поколений

gc.collect()        # принудительный запуск сборщика
gc.disable()        # временно отключить циклический GC
gc.enable()         # включить обратно

objs = gc.get_objects()  # список всех отслеживаемых объектов (опасно по памяти)
```

Смысл каждой функции:

- **`gc.get_threshold()`**  
  Показывает, **когда** GC решает, что пора запускать сборку автоматически.
  Возвращает кортеж из трёх чисел `(th0, th1, th2)` — пороги для поколений 0, 1, 2.

- **`gc.get_count()`**  
  Показывает, **сколько «событий» уже произошло** с момента последней сборки для каждого поколения.  
  Возвращает кортеж `(count0, count1, count2)`.

- **`gc.collect()`**  
  Сказать интерпретатору:  
  > «Сходи прямо сейчас, посмотри циклы и попробуй собрать мусор».  
  Возвращает число собранных объектов.

- **`gc.disable()` / `gc.enable()`**  
  Временно отключить/включить **циклический GC** (счётчик ссылок при этом продолжает работать).

- **`gc.get_objects()`**  
  Вернуть **список всех объектов**, которые GC отслеживает.  
  Осторожно: список может быть **очень большим**, легко съесть кучу памяти и времени.

#### 6.2.2. Когда полезно смотреть на все объекты (`gc.get_objects`)

`gc.get_objects()` — это как «рентген» памяти Python:

- вы получаете список всех объектов, про которые знает циклический GC;
- можете фильтровать по типу, чтобы увидеть, кого стало много.

Типичный сценарий:

1. Подозреваете, что **объектов какого-то типа становится всё больше и больше** (например, `MyBigObject`).
2. Перед подозрительным участком кода делаете снимок:

   ```python
   import gc

   before = [o for o in gc.get_objects() if isinstance(o, MyBigObject)]
   ```

3. Запускаете код, который «течёт».
4. После участка кода делаете ещё один снимок:

   ```python
   after = [o for o in gc.get_objects() if isinstance(o, MyBigObject)]
   print("До:", len(before), "После:", len(after))
   ```

Если `После` сильно больше, чем `До`, значит:

- где-то создаются объекты `MyBigObject`,  
- и **не освобождаются** (на них продолжают существовать ссылки).

Это ещё не говорит, *почему* так, но уже показывает **куда копать**.

#### 6.2.3. Кто держит ссылку? `gc.get_referrers` и `gc.get_referents`

Когда вы нашли объект, который «застрял», следующий вопрос:

> «**Кто именно** держит на него ссылку и не даёт ему освободиться?»

Для этого есть две функции:

- `gc.get_referrers(obj)` — **кто ссылается на obj** (родители);
- `gc.get_referents(obj)` — **на кого ссылается сам obj** (дети).

Простейший пример (игрушечный, но понятный):

```python
import gc

a = []
b = [a]      # b ссылается на a

refs_to_a = gc.get_referrers(a)
print("Насколько много referrers у a:", len(refs_to_a))

for r in refs_to_a:
    print("Тип referrer:", type(r))
    # здесь можно глазами посмотреть, кто это: список, словарь, фрейм функции и т.п.
```

В реальной отладке:

- фильтруют список `get_referrers` по типам (интересуют, например, списки или словари);
- иногда печатают не весь объект, а его «сводку» (id, тип, размер).

Важно понимать:

- `get_referrers` может возвращать и **служебные объекты** (фреймы, локальные словари, стеки);
- этим нельзя злоупотреблять в продакшене — это **диагностический инструмент**, а не нормальная часть логики программы.

#### 6.2.4. Мини-диагностика: посмотреть, кто держит наш объект

Пример чуть поближе к реальной ситуации:

```python
import gc

class MyBigObject:
    pass

leaks = []

def make_leak():
    obj = MyBigObject()
    leaks.append(obj)  # глобальный список "забывает" освобождать объекты

make_leak()
make_leak()

# Находим один экземпляр
all_objs = [o for o in gc.get_objects() if isinstance(o, MyBigObject)]
victim = all_objs[0]

print("Referrers of victim:")
for ref in gc.get_referrers(victim):
    print(type(ref), getattr(ref, "__name__", None))
```

С большой вероятностью вы увидите там:

- глобальный список `leaks`,
- возможно, какие-то служебные структуры.

Идея: вы можете **вычислить путь**, по которому объект «подвешен» в памяти, и понять, **что его держит**.

#### 6.2.5. Для чего в повседневной жизни достаточно знать

Чтобы **уверенно пользоваться GC**, достаточно:

- понимать роль `gc.collect()` — «собрать циклы прямо сейчас»;
- уметь заглянуть в `gc.get_count()` / `gc.get_threshold()` (для интереса/тонкой настройки);
- знать, что `gc.get_objects()` и `gc.get_referrers()` — это **мощные, но тяжёлые инструменты диагностики**, которые:
  - помогают искать утечки;
  - не предназначены для обычной логики приложения.

### 6.3. Поколения (generations)

#### 6.3.1. Главное, что надо понять

**Поколения** — это способ сделать сборку мусора быстрее.

Идея очень простая (и это реально можно запомнить):

- большинство объектов в программах **умирают быстро** (внутри функции создали список → что-то посчитали → вышли из функции → список больше не нужен);
- значит, выгодно чаще проверять «молодых» объектов и реже «старых».

Поэтому GC делит отслеживаемые объекты на «возрастные корзины» (поколения) и проверяет их с разной частотой.

#### 6.3.2. Какие поколения есть в CPython

CPython использует **поколенческую сборку**:

- **Поколение 0** — самые новые объекты (короткоживущие).
- **Поколение 1** — объекты, пережившие хотя бы одну сборку поколения 0.
- **Поколение 2** — самые старые, долго живущие объекты.

#### 6.3.3. Аналогия «три корзины на столе»

Представь, что у тебя на столе три корзины:

- **корзина 0**: «мусорка для свежих бумажек» (ты их постоянно создаёшь и тут же выбрасываешь);
- **корзина 1**: «то, что полежало и пережило одну уборку»;
- **корзина 2**: «старые важные документы».

Если ты будешь каждые 5 минут пересматривать **важные документы** (корзина 2), ты потратишь кучу времени на то, что почти всегда не меняется.

А если ты будешь часто пересматривать **свежие бумажки** (корзина 0), ты очень быстро найдёшь то, что уже не нужно.

Вот это и делает поколенческий GC.

#### 6.3.4. Почему так работает (интуиция)

- В типичной программе большинство объектов **живут недолго** (локальные переменные, временные списки и т.п.).
- Если часто проверять и собирать только «молодые» объекты, сборка будет **быстрой и эффективной**.
- Старые объекты реже меняются и реже становятся мусором, поэтому их реже проверяют.

#### 6.3.5. Что показывают `gc.get_count()` и `gc.get_threshold()`

Посмотреть состояние можно так:

```python
import gc

print(gc.get_count())      # (count0, count1, count2)
print(gc.get_threshold())  # (th0, th1, th2)
```

Как это читать «по-человечески»:

- `gc.get_count()` — три числа: сколько «событий/алокаций» накопилось с последней сборки для каждого поколения.
- `gc.get_threshold()` — три порога, после которых сборка запускается.

Упрощённая модель (для понимания):

1. Создаёшь много новых объектов → растёт `count0`.
2. Если `count0 > th0` → запускается сборка поколения 0.
3. Иногда при этом увеличиваются счётчики «глубже» (переходы между поколениями), и если `count1 > th1`, запускается сборка поколения 1; аналогично для поколения 2.

Это не «магия», это просто правило «когда пора делать уборку».

#### 6.3.6. Мини-эксперимент: увидеть, что счётчики меняются

Важно: это **не измерение памяти**, а просто способ почувствовать, что GC живёт своей жизнью и считает события.

```python
import gc

print("threshold:", gc.get_threshold())
print("count (start):", gc.get_count())

# Создадим много временных объектов
tmp = []
for _ in range(50_000):
    tmp.append({})  # создаём кучу маленьких dict

print("count (after alloc):", gc.get_count())

# Уберём сильную ссылку на список tmp и принудительно соберём мусор
tmp = None
collected = gc.collect()
print("collected:", collected)
print("count (after collect):", gc.get_count())
```

Что важно вынести:

- счётчики **двигаются**, когда вы создаёте объекты;
- `gc.collect()` принудительно запускает сборку и обычно меняет состояние.

### 6.4. Трёхцветная маркировка (tricolor marking)

#### 6.4.1. Зачем вообще нужны «цвета»

Алгоритм трёхцветной маркировки — это способ ответить на вопрос:

> **Какие объекты в памяти всё ещё достижимы из программы, а какие уже нет?**

Если объект **достижим**, значит до него можно добраться через цепочку ссылок от «корней программы» (глобальные переменные, стек вызовов, активные фреймы и т.п.).  
Если объект **недостижим**, значит он уже никому не нужен → это мусор.

Проблема в том, что объектов может быть очень много и они ссылаются друг на друга как огромная сеть.
«Цвета» — это просто удобные ярлыки статуса обработки, чтобы:

- не забыть, какие объекты мы уже просмотрели,
- какие ещё надо просмотреть,
- и какие точно «живые» (достижимые).

#### 6.4.2. Термины «по-простому»

- **Белый объект**: мы его ещё не трогали, и **пока считаем, что он может быть мусором**.
- **Серый объект**: мы нашли его как достижимый, но **ещё не посмотрели, на что он ссылается**.
- **Чёрный объект**: мы уже полностью обработали его ссылки (посмотрели «детей»), и он точно **живой**.

#### 6.4.3. Аналогия «обход комнат в здании»

Представь, что объекты — это **комнаты** в здании, а ссылки — это **двери** между комнатами.

«Корни программы» — это комнаты, в которые ты точно можешь попасть (например, вход в здание).

Твоя задача: понять, в какие комнаты можно попасть, а какие комнаты изолированы и никому не нужны.

- Белые комнаты — ты ещё туда не заходил.
- Серые комнаты — ты знаешь, что туда можно попасть, но ещё не обследовал двери из неё.
- Чёрные комнаты — ты уже зашёл, посмотрел все двери, отметил соседние комнаты, и полностью закончил с этой комнатой.

#### 6.4.4. Процесс маркировки (очень пошагово)

Упрощённая схема процесса:

1. **Все объекты считаем белыми.**
2. Берём «корни» (то, что прямо сейчас держит программа) и делаем их **серыми**.
3. Пока есть серые:
   1. Берём один серый объект.
   2. Смотрим на все объекты, на которые он ссылается (его «детей»).
   3. Если среди детей есть белые — делаем их **серыми** (потому что мы нашли путь к ним из корней).
   4. После того как мы обработали все его ссылки, этот объект красим в **чёрный**.
4. Когда серых не осталось:
   - все **чёрные** — достижимы (живые),
   - все **белые** — недостижимы → это мусор (кандидаты на освобождение).

#### 6.4.5. Маленькая ASCII-картинка (чтобы прямо «увидеть»)

Допустим, есть такие ссылки:

```
ROOT -> A -> B -> C
          ^    |
          |    v
          E <- D

X <-> Y   (отдельный цикл, но к нему нет пути из ROOT)
```

Здесь:

- `A, B, C, D, E` достижимы от `ROOT` → живые.
- `X` и `Y` образуют цикл, но **не достижимы** от `ROOT` → мусор.

Как это увидит трёхцветная маркировка:

1. ROOT серый.
2. Обрабатываем ROOT → красим A серым, ROOT чёрным.
3. Обрабатываем A → красим B серым, A чёрный.
4. Обрабатываем B → красим C и E серыми, B чёрный.
5. Обрабатываем C → красим D серым, C чёрный.
6. Обрабатываем E → (ссылка на B уже обработана), E чёрный.
7. Обрабатываем D → (ссылка на E), D чёрный.
8. Серых нет. Белыми остались X и Y → их можно освобождать.

#### 6.4.6. Важная мысль: GC ищет «недостижимость», а не «refcount=0»

Это ключевой момент:

- refcount отвечает на вопрос: «сколько стрелочек сейчас указывает на объект?»
- циклический GC отвечает на вопрос: «можно ли добраться до объекта из программы?»

Именно поэтому GC способен удалять циклы, даже когда refcount не падает до нуля.

CPython не раскрывает напрямую «цвет» объектов через API, но концептуально циклический GC опирается на эту идею.

### 6.5. Тонкости `__del__` и «неподдающиеся» объекты

`__del__` — это **финализатор** объекта, вызываемый перед уничтожением:

```python
class Resource:
    def __del__(self):
        print("Освобождаем ресурс")
```

#### 6.5.1. Почему `__del__` вообще опасен

На уровне «человеческой логики» хочется думать так:

> «Ну раз объект сейчас уничтожают, пусть он в `__del__` закроет файл/сокет/освободит ресурс».

Но у GC есть проблема: **когда мы уничтожаем группу объектов, связанных циклом, порядок уничтожения может быть неочевидным**.

Пример: два объекта ссылаются друг на друга и у обоих есть `__del__`.
Вопрос: кого уничтожать первым? А если `__del__` одного попытается обратиться к другому?
Это может привести к странному состоянию: часть объектов уже разрушена, часть ещё нет.

Поэтому CPython относится к `__del__` очень осторожно.

#### 6.5.2. Какие бывают проблемы

Проблемы:

- **Циклы с `__del__`**: если объект участвует в цикле ссылок и имеет `__del__`, сборщик не всегда может **безопасно** определить порядок вызова финализаторов. Раньше такие объекты попадали в список **uncollectable** (неподдающиеся сборке) и могли жить до конца процесса.
- **Resurrection (воскрешение)**: внутри `__del__` вы можете снова сохранить ссылку на объект (например, в глобальную переменную). Объект «оживает», хотя сборщик уже начал его уничтожение — это делает поведение сложным и потенциально опасным.

#### 6.5.3. Мини-демо: «воскрешение» (resurrection)

Это пример, который не нужно запоминать наизусть — важно понять сам факт: `__del__` может сделать поведение непредсказуемым.

```python
resurrected = None

class Zombie:
    def __del__(self):
        global resurrected
        resurrected = self   # сохраняем ссылку на объект → он «воскресает»
        print("Я воскрес!")

def make_zombie():
    z = Zombie()

make_zombie()

# В CPython __del__ может сработать сразу при выходе из функции.
# А затем resurrected будет ссылаться на объект, который мы вроде бы уже «удалили».
print("resurrected is None?", resurrected is None)
```

Идея: GC/интерпретатор начал уничтожать объект, но внутри `__del__` объект снова стал достижимым.
Это осложняет гарантию «когда именно и в каком порядке что будет уничтожено».

#### 6.5.4. Что делать вместо `__del__` (правильная практика)

Вместо `__del__` почти всегда лучше:

- **контекстный менеджер** (`with`) для ресурсов (файлы, сокеты, блокировки);
- `weakref.finalize` для callback при сборке объекта (если очень нужно).

Поэтому:

- в продакшен-коде **по возможности избегайте `__del__`**;
- вместо этого используйте:
  - **контекстные менеджеры** (`__enter__` / `__exit__`) и `with`;
  - `weakref.finalize(obj, callback, *args)` — безопасный финализатор.

Пример с `weakref.finalize`:

```python
import weakref

class Connection:
    def close(self):
        print("Закрываем соединение")

conn = Connection()
finalizer = weakref.finalize(conn, Connection.close, conn)

del conn  # когда объект станет недостижимым, finalizer вызовет close
```

- `weakref.finalize` **не требует** наличия `__del__`;
- вы можете явно отменить финализатор через `finalizer.detach()`;
- `finalizer.alive` показывает, будет ли callback ещё вызван.

#### 6.5.5. Практический пример: правильно освобождать ресурс через `with`

Допустим, у вас есть файл. Плохая идея — надеяться, что `__del__` когда-то его закроет.
Правильная идея — закрыть его **сразу**, когда вы закончите:

```python
with open("data.txt", "r", encoding="utf-8") as f:
    data = f.read()
# Здесь файл уже закрыт гарантированно, даже если внутри блока было исключение.
```

Это не про «память», это про **ресурсы** (дескрипторы файлов, сокеты). Но именно `__del__` часто пытаются использовать для ресурсов — и это обычно ошибка.

### 6.6. Когда отключать GC

Иногда циклический GC временно отключают:

- в **короткоживущих скриптах**, которые много работают с временными объектами и не создают циклов — так можно немного выиграть в производительности;
- в **критичных по latency участках**, где остановки на сборку нежелательны.

Пример:

```python
import gc

gc.disable()
try:
    # критичный по производительности участок кода
    ...
finally:
    gc.enable()
```

Но:

- если ваш код создаёт **циклы**, отключение GC приведёт к **утечкам**;
- обычно лучше сперва **измерить** (`tracemalloc`, профилировщики), чем слепо отключать GC.

#### 6.6.1. Мини-эксперимент: «что будет, если выключить GC и создать циклы»

Этот пример нужен, чтобы на практике увидеть простую вещь:

- refcount сам по себе циклы не освобождает;
- если выключить GC, циклы будут накапливаться.

```python
import gc

class Node:
    def __init__(self):
        self.other = None

def make_cycle():
    a = Node()
    b = Node()
    a.other = b
    b.other = a

# Выключаем циклический GC
gc.disable()

before = gc.get_count()
for _ in range(50_000):
    make_cycle()

after = gc.get_count()
print("count before:", before)
print("count after:", after)

# Включаем обратно и принудительно собираем
gc.enable()
collected = gc.collect()
print("collected:", collected)
```

Что важно понять:

- в `make_cycle()` после выхода из функции **нет внешних ссылок** на `a` и `b`,
  но у них есть ссылки друг на друга → refcount не падает до 0;
- поэтому без циклического GC эти объекты будут «висеть»;
- когда вы снова включите GC и вызовете `gc.collect()`, сборщик сможет их убрать.

Этот пример не идеален как «измеритель памяти» (он не показывает мегабайты),
но он отлично показывает **логическую причину**: циклы требуют GC.

### 6.7. Запомните (gc)

| Правило | Пояснение |
|---------|-----------|
| GC нужен для циклов | Refcount не умеет сам по себе разрывать циклы ссылок |
| Поколения экономят время | Молодые объекты проверяются чаще, старые — реже |
| `__del__` опасен | В циклах может блокировать сборку, возможны «воскрешения» |
| Предпочитайте `weakref.finalize` и `with` | Это более предсказуемые и безопасные механизмы освобождения ресурсов |
| Не отключайте GC без измерений | Сначала убедитесь, что он действительно проблема |
