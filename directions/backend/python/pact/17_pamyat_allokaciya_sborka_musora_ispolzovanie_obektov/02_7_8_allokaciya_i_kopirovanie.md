## 7. §93. Аллокация памяти при создании объектов

### 7.1. Как Python выделяет память

В CPython память под объекты выделяется **не напрямую через malloc** для каждого объекта:

- для **мелких объектов** (обычно до 512 байт) используется специальный аллокатор **pymalloc**;
- он работает поверх системного аллокатора (`malloc`/`free`) и управляет:
  - **аренами** (крупные блоки памяти, например по несколько мегабайт),
  - **пулами** (подобласти арен, каждая отвечает за блоки определённого размера),
  - **блоками** (конкретные кусочки памяти для объектов).

Идея:

- запрашивать у ОС память **большими кусками** (арены);
- нарезать их на блоки фиксированных размеров;
- переиспользовать блоки для объектов подходящего размера.

Это даёт:

- **быстрое выделение** (простое взятие свободного блока из пула);
- **меньшую фрагментацию** памяти;
- упрощённое управление памятью (внутри интерпретатора).

### 7.2. Структура объекта (PyObject) и накладные расходы

Каждый объект в CPython — это C-структура, которая начинается с заголовка:

- `ob_refcnt` — счётчик ссылок;
- `ob_type` — указатель на структуру типа (класс);
- иногда дополнительные поля (флаги, размер и т.п.).

Поэтому:

- **любой объект имеет накладные расходы** — даже простой `int` занимает больше, чем 8 байт;
- `sys.getsizeof(obj)` показывает **размер объекта с заголовком**, но **без вложенных объектов**.

Пример:

```python
import sys

print(sys.getsizeof(0))      # размер объекта int (например, 28 байт)
print(sys.getsizeof(10**100))  # тоже объект int, но может быть больше

print(sys.getsizeof([]))     # пустой список
print(sys.getsizeof([1]))    # список с одним элементом (содержит только ссылку)
```

Список хранит **массив ссылок** на элементы. `sys.getsizeof` считает только:

- заголовок списка;
- массив указателей (slots) для элементов.

Сами элементы считаются **отдельными объектами**.

### 7.3. `sys.getsizeof` и его ограничения

```python
import sys

lst = [1, 2, 3]
print(sys.getsizeof(lst))          # например, 88
print(sys.getsizeof(lst[0]))       # размер int
print(sys.getsizeof(lst) + sum(sys.getsizeof(x) for x in lst))
```

Ограничения:

- `sys.getsizeof` не знает о **пользовательских типах** за пределами CPython API;
- вложенные структуры (списки в списках, словари и т.п.) надо учитывать **рекурсивно**;
- циклы ссылок могут привести к **зацикливанию**, если считать размер вручную.

Для оценки общего потребления памяти сложных структур лучше использовать:

- `tracemalloc` (слои выделений);
- внешние инструменты (`pympler`, `guppy3`, `objgraph`).

### 7.4. Отладка аллокатора: `PYTHONMALLOC=debug`

Если нужно понять, **как именно** выделяется память, CPython позволяет включить режим отладки аллокатора:

```bash
PYTHONMALLOC=debug python script.py
```

В этом режиме:

- добавляются дополнительные проверки целостности памяти;
- при утечках или повреждении аллокатора Python может вывести подробные сообщения.

Это продвинутый инструмент, обычно нужен при разработке C-расширений или отладке низкоуровневых проблем.

## 8. §94. Память при вычислениях и копировании

### 8.1. Присваивание — это копирование ссылки, а не данных

Ключевая идея Python:

- `x = y` **не создаёт копию объекта**, а только делает ещё одну **ссылку** на тот же объект.

```python
lst1 = [1, 2, 3]
lst2 = lst1

lst2.append(4)
print(lst1)  # [1, 2, 3, 4]
print(lst1 is lst2)  # True
```

Обе переменные `lst1` и `lst2` указывают на один и тот же объект списка.

### 8.2. Поверхностная и глубокая копия (shallow/deep copy)

**Поверхностная копия**:

- создаётся **новый контейнер** (например, новый список),
- но его элементы — это **те же самые объекты**, что и в оригинале (копируются ссылки).

```python
import copy

lst = [[1], [2], [3]]
shallow = lst[:]           # или copy.copy(lst)

shallow[0].append(99)
print(lst)     # [[1, 99], [2], [3]]
print(shallow) # [[1, 99], [2], [3]]
```

**Глубокая копия**:

- создаётся новый контейнер,
- и **рекурсивно копируются все вложенные объекты** (насколько это возможно).

```python
import copy

lst = [[1], [2], [3]]
deep = copy.deepcopy(lst)

deep[0].append(99)
print(lst)   # [[1], [2], [3]]
print(deep)  # [[1, 99], [2], [3]]
```

Минусы глубокой копии:

- может быть **очень медленной** и **памятьёёмкой**;
- может не работать с объектами, которые не умеют корректно копироваться.

### 8.3. Конкатенация и изменение коллекций

#### 8.3.1. Строки

Строки в Python **иммутабельны**:

- любая операция, которая вроде бы «меняет» строку, на самом деле **создаёт новую строку**.

```python
s = "a"
s += "b"          # новая строка "ab"
```

Поэтому конкатенация в цикле:

```python
result = ""
for part in parts:
    result += part
```

часто приводит к **избыточным аллокациям**: каждый раз создаётся новая строка.

Правильный подход:

```python
result = "".join(parts)
```

Здесь:

- сначала может быть оценена общая длина;
- создаётся **одна большая строка**, и все части копируются в неё за один проход.

#### 8.3.2. Списки

Списки **мутабельны**, и `+=` для списков — это **in-place операция**:

```python
lst = [1, 2]
lst += [3, 4]  # тот же объект, расширенный новыми элементами
```

Если нужен новый список:

```python
new_lst = lst + [3, 4]  # создаётся новый список
```

### 8.4. Comprehensions и генераторы

**Comprehensions** (`[x for x in it]`, `{k: v for ...}`):

- создают **новый контейнер** в памяти;
- удобно и лаконично, но могут потреблять **много памяти**, если диапазон большой.

**Генераторы** (`(x for x in it)`):

- не хранят все значения в памяти;
- вычисляют элементы **по запросу** (лениво).

```python
squares_list = [x * x for x in range(10**6)]  # большой список в памяти
squares_gen = (x * x for x in range(10**6))   # почти не занимает памяти
```

Если вам не нужно беспорядочно прыгать по коллекции, а достаточно просто **один раз пройтись**, лучше использовать **генераторы** — это экономит память.

### 8.5. Запомните (копирование и вычисления)

| Паттерн | Пояснение |
|---------|-----------|
| `x = y` | только копия ссылки, объект не дублируется |
| `lst1 = lst2[:]` | новый список, но с теми же элементами (shallow copy) |
| `copy.deepcopy(x)` | полный дубль структуры, дорого по времени и памяти |
| `s += part` в цикле | создаёт новые строки → лучше `''.join(parts)` |
| Генераторы | ленивы, хранят только текущее состояние, экономят память |
