# Часть XVII. Память: аллокация, сборка мусора, использование объектов

> **Цель:** понять, **как Python управляет памятью**: счётчик ссылок, циклический сборщик мусора, аллокатор объектов, как именно объекты занимают память, почему возникают «утечки» и как их искать. Всё — **простым языком**, с чёткой терминологией и практическими примерами.
>
> После изучения этой части вы:
>
> - будете понимать, **почему объект «живёт» или «умирает»** в CPython;
> - разберётесь, **как работают refcount и gc**, что такое поколения и триколорная маркировка;
> - научитесь читать `sys.getsizeof`, использовать `gc`, `weakref`, `tracemalloc` и другие инструменты;
> - узнаете, **как устроены list/dict/set внутри**, почему они иногда «жрут» больше памяти, чем ожидается;
> - сможете **осознанно оптимизировать потребление памяти**, а не гадать «почему так много».

---

## 1. Общая картина: как Python управляет памятью

Чтобы уверенно писать и отлаживать код на Python, важно понимать несколько ключевых идей:

- **Python — язык с автоматическим управлением памятью.** Вам не нужно вызывать `free()` вручную, как в C. Но это **не значит**, что про память можно забыть.
- В **CPython** (стандартная реализация) память управляется в два слоя:
  - **Счётчик ссылок (reference counting):** каждый объект знает, **сколько на него сейчас есть ссылок**. Когда счётчик падает до нуля — объект освобождается сразу.
  - **Циклический сборщик мусора (gc):** отдельный механизм, который **находит циклы ссылок**, которые refcount не может освободить.
- Для эффективности CPython использует **собственный аллокатор (pymalloc)** поверх аллокатора ОС:
  - мелкие объекты (до ~512 байт) выделяются из специальных **пулов и арен**;
  - это уменьшает фрагментацию и ускоряет выделение/освобождение.
- Объекты Python — это **структуры с заголовком**:
  - в заголовке — **счётчик ссылок**, указатель на тип (класс) и служебные поля;
  - поэтому любой объект занимает **больше памяти**, чем «голые» данные.

Высокоуровнево:

- вы работаете с **ссылками на объекты**;
- интерпретатор следит, **сколько ссылок** есть на каждый объект;
- когда объект становится **недостижимым**, он рано или поздно **освобождается**: либо сразу (refcount), либо при проходе сборщика (`gc`).

## 2. Краткая шпаргалка по памяти в CPython

| Понятие | Кратко | API / инструменты |
|--------|--------|-------------------|
| **Счётчик ссылок** | Каждый объект хранит число ссылок на себя. При 0 — освобождается | `sys.getrefcount(obj)` (учитывайте +1 за аргумент), `id(obj)` — только для проверки идентичности |
| **Циклический GC** | Ищет **циклы ссылок**, которые refcount не может освободить | `import gc`; `gc.collect()`, `gc.get_objects()`, `gc.get_referrers()`, `gc.get_referents()` |
| **Поколения GC** | Объекты делятся на поколения по возрасту; старые реже проверяются | `gc.get_count()`, `gc.get_threshold()`, `gc.set_threshold()` |
| **Трёхцветная маркировка** | Алгоритм: белые/серые/чёрные объекты, обход графа ссылок | Детали — в теории; API напрямую не показывает цвета |
| **__del__ (финализатор)** | Метод, вызываемый при уничтожении объекта | Опасен в циклах; лучше `weakref.finalize` или context manager |
| **Аллокация объектов** | CPython использует `pymalloc` и собственные пулы для мелких объектов | Отладка: запуск Python c `PYTHONMALLOC=debug` |
| **Размер объекта** | `sys.getsizeof(obj)` показывает **размер самого объекта**, без вложенных | Учитывайте, что коллекции чаще всего хранят только ссылки на элементы |
| **Shallow vs deep copy** | Поверхностная копия копирует ссылки; глубокая — рекурсивно объекты | `copy.copy`, `copy.deepcopy` |
| **Память коллекций** | list — динамический массив; dict/set — хеш-таблицы с запасом | `.clear()`, `del d[:]`, `list(d)`, `dict.fromkeys()` (осторожно с mutable) |
| **Слабые ссылки** | Ссылки, не увеличивающие refcount; удобны для кешей | `weakref.ref`, `weakref.proxy`, `WeakKeyDictionary`, `WeakValueDictionary`, `WeakSet`, `weakref.finalize` |
| **Поиск утечек** | Анализ, что и почему не освобождается | `tracemalloc`, `gc.get_objects`, `objgraph`, `memory_profiler`, `guppy3` |
| **Оптимизация памяти** | Снижение накладных расходов и избыточных структур | `__slots__`, генераторы, `array.array`, `itertools`, `mmap`, специализированные структуры (NumPy) |

## 3. Маршрут изучения

Рекомендуемый порядок:

1. **§91 — Счётчик ссылок.** Понять, что такое refcount, какие операции его меняют и почему в CPython объекты обычно освобождаются **сразу**, как только становятся ненужными.
2. **§92 — Циклический сборщик мусора.** Разобраться, как `gc` находит циклы, что такое поколения, триколорная маркировка, чем опасен `__del__`.
3. **§93–95 — Аллокация и память коллекций.** Как объекты выделяются в памяти, как ведут себя list/dict/set, почему `len(d)` сильно меньше `sys.getsizeof(d)` и откуда берётся «запас» ёмкости.
4. **§96 — Слабые ссылки.** Понять, когда нужно **не увеличивать** счётчик ссылок: кеши, обратные ссылки, callback’и.
5. **§97–98 — Утечки и оптимизация.** Научиться находить утечки и осознанно уменьшать потребление памяти (и когда этим вообще стоит заниматься).
