## 11. §97. Утечки памяти и их поиск

### 11.1. Что такое «утечка» в Python

Строго говоря, в языке с автоматическим управлением памятью **утечка** — это ситуация, когда:

- объект **по логике программы больше не нужен** (вы не планируете к нему обращаться),
- но остаётся **достижимым** (на него всё ещё где-то есть ссылки),
- и поэтому **никогда не будет собран GC** (GC честно считает, что объект всё ещё нужен).

Очень важно: это **логическая утечка**, а не «сломался GC».

GC делает ровно то, что должен:

- видит, что на объект есть ссылки → считает его живым;
- не видит, что вы **по смыслу** перестали его использовать.

Аналогия: вы закончили читать книгу, но положили её не на полку «ненужные», а на стол «активных проектов».  
Уборщица не выбросит её, потому что формально она на столе → *значит, нужна*.

Типичные причины:

- **глобальные списки/словари/кеши**, в которые добавляют и ничего не удаляют;
- **замыкания** и **лямбды**, которые захватили большие объекты (например, данные) и долго живут;
- **циклы ссылок** с `__del__`, которые GC не может собрать;
- баги в **C-расширениях**, где забыли уменьшить refcount.

### 11.2. Пример утечки через глобальный кеш

```python
import time

cache = []

def process():
    data = [0] * 10_0000   # массив чисел
    cache.append(data)     # добавляем в глобальный список

while True:
    process()
    time.sleep(0.1)
```

Здесь:

- каждый вызов `process` создаёт новый большой список;
- он добавляется в глобальный `cache`, который **никогда не очищается**;
- GC не виноват — по логике программы на эти объекты **есть ссылки** → они не мусор.

Здесь важно увидеть:

- каждая итерация цикла создаёт **новый большой список**;
- глобальный `cache` всё это хранит, и нигде не написано «удалить старые данные»;
- программа как бы «забывает забывать» про старые объекты.

Утечка здесь — **ошибка нашего дизайна**, а не проблема сборщика мусора.

### 11.3. Инструменты поиска утечек

#### 11.3.1. `tracemalloc`

`tracemalloc` — это «фотоаппарат» для памяти Python:

- позволяет делать **снимки состояния выделений памяти**;
- сравнивать **два снимка**: «до» и «после» подозрительного участка кода;
- видеть, **какие строки/файлы** больше всего увеличили потребление памяти.

Пример (упрощённый, но рабочий):

```python
import tracemalloc

def work():
    # код, который подозревается в утечке
    data = [i for i in range(10**5)]
    return data

tracemalloc.start()

snapshot1 = tracemalloc.take_snapshot()
_ = [work() for _ in range(10)]  # подозрительный участок
snapshot2 = tracemalloc.take_snapshot()

stats = snapshot2.compare_to(snapshot1, 'filename')
for stat in stats[:10]:
    print(stat)
```

Что здесь происходит по шагам:

1. `tracemalloc.start()` включает «режим наблюдения» за выделениями памяти.
2. `snapshot1 = tracemalloc.take_snapshot()` — делаем первый снимок: *«как выглядит память сейчас»*.
3. Запускаем подозрительный код (`[work() for _ in range(10)]`).
4. `snapshot2 = tracemalloc.take_snapshot()` — второй снимок: *«как выглядит память после»*.
5. `compare_to(..., 'filename')` сравнивает два снимка и группирует разницу по имени файла.
6. В `stats[:10]` попадают файлы, которые **больше всего увеличили потребление памяти**.

Вывод покажет:

- какие файлы/строки кода больше всего увеличили потребление памяти;
- откуда «растут ноги» у вашей утечки.

Дальше можно:

- углубиться по конкретному файлу;
- посмотреть уже не только по файлам, но и по строкам (есть другие режимы группировки в `compare_to`).

#### 11.3.2. `gc.get_objects` и `objgraph`

`gc.get_objects()`:

- возвращает список всех объектов, отслеживаемых GC;
- можно фильтровать по типу и смотреть, сколько их:

```python
import gc

objs = [o for o in gc.get_objects() if isinstance(o, MyClass)]
print(len(objs))
```

`objgraph` (сторонняя библиотека):

- умеет строить **граф ссылок**;
- показывает, **кто держит ссылку** на объект и почему он не собирается.

### 11.4. Утечки в C-расширениях

Если модуль написан на C и неправильно управляет refcount:

- объект может оставаться с refcount > 0, даже когда на него нет «настоящих» ссылок из Python-кода;
- GC ничего с этим сделать не может — он считает, что на объект есть ссылки.

В таких случаях:

- используются низкоуровневые инструменты (отладка C-кода, `valgrind`, отладка аллокатора);
- для Python-разработчика это обычно «чёрный ящик»: важно просто **знать, что такое возможно**.

### 11.5. Запомните (утечки)

| Источник | Пример |
|----------|--------|
| Глобальные структуры | Список/словарь, куда добавляют объекты, но никогда не удаляют |
| Замыкания | Вложенные функции держат большие объекты через внешнюю область видимости |
| Циклы с `__del__` | GC не может безопасно вызвать финализаторы и не освобождает объекты |
| C-расширения | Ошибки в refcount на уровне C-кода, не видны из Python |

## 12. §98. Оптимизация потребления памяти

### 12.1. Когда вообще стоит думать о памяти

Не нужно **преждевременно оптимизировать**:

- пока программа работает стабильно и укладывается в разумные рамки потребления памяти, не стоит усложнять код;
- сначала **измерьте**: профилируйте время и память, выясните, где действительно проблема.

Когда стоит:

- сервис/скрипт работает на **ограниченных ресурсах** (маленькие контейнеры, встраиваемые устройства);
- обрабатываются **очень большие объёмы данных** (миллионы объектов);
- потребление памяти **растёт со временем** и это мешает (OOM, своп, падение производительности).

Просто правило:

- если у вас обычный скрипт/утилита, и она **не падает по OOM** и не ест гигабайты там, где ожидаете сотни мегабайт — живите спокойно;
- если вы видите, что **RSS процесса растёт со временем** без явной причины — самое время открыть эту главу и доставать `tracemalloc`/`gc`.

### 12.2. `__slots__` — экономия на атрибутах объектов

По умолчанию у экземпляров классов есть:

- словарь атрибутов `__dict__` — хеш-таблица;
- это гибко, но затратно по памяти при большом количестве экземпляров.

`__slots__`:

- позволяет **ограничить набор атрибутов**;
- хранить их в **фиксированном массиве** вместо словаря;
- уменьшает размер каждого экземпляра.

```python
class Point:
    __slots__ = ("x", "y")

    def __init__(self, x, y):
        self.x = x
        self.y = y
```

Особенности:

- нельзя динамически добавлять новые атрибуты (не указанные в `__slots__`);
- если нужен `__weakref__` (слабые ссылки), его нужно явно добавить в `__slots__`.

### 12.3. Генераторы вместо списков

Если вы просто **итерируете** по данным и не храните их все одновременно, используйте:

- генераторы (`(x for x in it)`);
- функции-генераторы (`yield`).

```python
def read_lines_lazy(path):
    with open(path, "r") as f:
        for line in f:
            yield line.rstrip("\n")
```

Так вы:

- не загружаете сразу весь файл в память;
- обрабатываете строки по одной.

### 12.4. Компактные структуры: `array`, `mmap`, специализированные библиотеки

Если вы храните **много чисел одного типа**, обычный список:

- хранит **Python-объекты int**;
- на каждый элемент приходится накладной заголовок → сильно раздувает память.

`array.array`:

- хранит **упаковку чисел в байтовый массив**;
- гораздо компактнее.

```python
from array import array

arr = array("I", range(10**6))  # миллион беззнаковых 32-битных целых
```

Для серьёзных числовых задач используют:

- **NumPy**: многомерные массивы, эффективные операции;
- **pandas**: табличные данные.

`mmap`:

- отображает файл в память;
- позволяет работать с очень большими файлами, как с массивом байт, **не загружая всё сразу**.

### 12.5. `itertools` и ленивые вычисления

Модуль `itertools`:

- предоставляет набор функций, работающих с **итераторами**;
- большинство из них не создают список, а возвращают **ленивый итератор**.

Примеры:

- `itertools.islice` — срез итератора;
- `itertools.chain` — последовательное объединение;
- `itertools.groupby` — группировка по ключу.

Используя их вместо явных списков, можно **существенно сократить** потребление памяти.

### 12.6. Запомните (оптимизация)

| Подход | Когда применять |
|--------|-----------------|
| `__slots__` | Много однотипных объектов с небольшим набором полей |
| Генераторы и `itertools` | Потоковая обработка данных, нет нужды хранить всё |
| `array`, NumPy | Большие числовые массивы |
| `mmap` | Огромные файлы, к которым нужен произвольный доступ |
| Переиспользование структур | Не создавать лишние временные списки/словари в циклах без необходимости |
