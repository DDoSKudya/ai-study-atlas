## §23. multiprocessing — мультипроцессинг

**Цель раздела §23:** создавать отдельные **процессы** через **`multiprocessing.Process`**, обмениваться данными через **Queue**, **Pipe**, **Value**, **Array**, **Manager** и **shared_memory**; использовать **Pool** для пула воркеров; понимать разницу **fork** и **spawn** и необходимость **`if __name__ == '__main__':`** при spawn.

### 23.1 Основы: Process, start, join, terminate

---

**Мультипроцессинг** — запуск нескольких **процессов**; у каждого своя память. GIL действует внутри каждого процесса, но между процессами его нет — реальный параллелизм на многоядерных системах. Подходит для **CPU-bound** задач. Минус — обмен данными между процессами требует сериализации (pickle) или разделяемой памяти.

#### fork vs spawn vs forkserver

- **fork** — копирование процесса (клонирование памяти). Быстрый старт; дочерний процесс наследует память родителя. На Windows недоступен.
- **spawn** — запуск нового интерпретатора Python; код модуля импортируется заново в дочернем процессе. Используется по умолчанию на Windows и macOS (3.8+). Без **`if __name__ == '__main__':`** код создания процессов выполнится при импорте в дочернем процессе и приведёт к бесконечному порождению процессов — поэтому точку входа **обязательно** оборачивают в `if __name__ == '__main__':`.
- **forkserver** — отдельный серверный процесс создаёт дочерние через fork; старт медленнее, но изоляция лучше. **Когда выбирать forkserver:** если после **fork** в основном процессе загружались потоки или сложное состояние (например, OpenMP, некоторые C-библиотеки), повторный fork может быть небезопасен; forkserver запускает «чистый» процесс-сервер без лишнего состояния и из него уже делает fork — меньше рисков. На Windows недоступен (только spawn).

**Создание процесса:**

```python
import multiprocessing

def worker(x):
    return x * 2

if __name__ == '__main__':
    p = multiprocessing.Process(target=worker, args=(5,))
    p.start()
    p.join()
    print(p.exitcode)
```

**`process.start()`** — запуск процесса; **`process.join(timeout=None)`** — ожидание завершения; **`process.terminate()`** — принудительное завершение; **`process.exitcode`** — код выхода после завершения. **`multiprocessing.set_start_method('spawn'|'fork'|'forkserver')`** — вызывать **до** создания любых процессов (обычно в main); иначе — **RuntimeError**.

### 23.2 Обмен данными между процессами: Queue, Pipe, Value, Array, Manager

**Цель подраздела:** обмениваться данными между процессами через **Queue** (pickle), **Pipe** (двунаправленный канал), **Value/Array** (разделяемая память с typecode), **Manager** (разделяемые dict/list); понимать ограничения pickle и когда использовать shared_memory.

- **`multiprocessing.Queue`** — очередь между процессами; объекты сериализуются через **pickle**. **`q.put(obj)`**, **`q.get()`**; при полной/пустой очереди блокируется (или таймаут). Очередь создаётся в родительском процессе и передаётся дочерним через аргументы.
- **`multiprocessing.Pipe(duplex=True)`** — возвращает **`(conn_parent, conn_child)`**; канал двунаправленный при **duplex=True**. **`conn.send(obj)`**, **`obj = conn.recv()`** — pickle; **`conn.send_bytes(buffer)`**, **`conn.recv_bytes()`** — сырые байты. Один конец передаётся дочернему процессу при создании.
- **`multiprocessing.Value(typecode, value, lock=True/False)`**, **`multiprocessing.Array(typecode, size_or_initializer, lock=True/False)`** — разделяемая память между процессами (типы C). **typecode:** `'i'` (int), `'d'` (double), `'f'` (float), `'c'` (char) и др. Доступ: **`value.value`** для Value; **`arr[i]`** для Array. При **lock=True** доступ атомарный (внутренний Lock).
- **`multiprocessing.Manager()`** — прокси к объектам (dict, list, Namespace и др.), живущим в отдельном процессе-менеджере. **`manager.dict()`**, **`manager.list()`** — разделяемые структуры; изменения видны во всех процессах. Удобно для сложных структур; медленнее, чем Value/Array.
- **`multiprocessing.shared_memory.SharedMemory(name=..., create=True, size=N)`** (3.8+) — именованная разделяемая память; **`shm.buf`** — **memoryview**; без pickle. Имя **`name`** передают в дочерние процессы для подключения к той же области (**`SharedMemory(name=name, create=False)`**).

#### Таблица типов для Value и Array (typecode)

| typecode | Тип C | Python |
|----------|-------|--------|
| `'b'` | signed char | int |
| `'B'` | unsigned char | int |
| `'h'` | short | int |
| `'i'` | int | int |
| `'l'` | long | int |
| `'f'` | float | float |
| `'d'` | double | float |
| `'c'` | char | str (длина 1) |

**Array** принимает **size_or_initializer**: целое (размер) или последовательность (инициализация). Например: **`Array('i', 10)`** — массив из 10 int; **`Array('d', [1.0, 2.0])`** — массив из двух double.

#### Пример: Pipe — обмен между процессами

```python
import multiprocessing

def child(conn):
    conn.send([42, 'hello'])
    x = conn.recv()
    print('Child received', x)
    conn.close()

if __name__ == '__main__':
    parent_conn, child_conn = multiprocessing.Pipe()
    p = multiprocessing.Process(target=child, args=(child_conn,))
    p.start()
    print(parent_conn.recv())  # [42, 'hello']
    parent_conn.send('ok')
    p.join()
```

#### Пример: Manager — разделяемый dict

```python
import multiprocessing

def worker(shared_dict, key, value):
    shared_dict[key] = value

if __name__ == '__main__':
    with multiprocessing.Manager() as manager:
        d = manager.dict()
        procs = [multiprocessing.Process(target=worker, args=(d, f'k{i}', i)) for i in range(5)]
        for p in procs:
            p.start()
        for p in procs:
            p.join()
        print(dict(d))  # {'k0': 0, 'k1': 1, ...}
```

### 23.3 Пулы процессов: Pool, map, apply_async, close, join

**Цель подраздела:** использовать **Pool** для пула воркеров: **map** (результаты в порядке аргументов), **apply_async** (неблокирующий вызов, результат через **get()**), **imap** / **imap_unordered** (ленивая итерация); корректно завершать пул (**close**, **join** или контекстный менеджер).

- **`multiprocessing.Pool(processes=N)`** — пул воркеров. **`pool.map(func, iterable)`** — применить функцию ко всем элементам, вернуть список результатов **в порядке iterable**; блокирует до завершения всех. **`pool.apply_async(func, args=(), callback=..., error_callback=...)`** — отправить одну задачу, вернуть **AsyncResult**; **result.get(timeout=None)** — дождаться результата. **`pool.close()`** — запретить новые задачи; **`pool.join()`** — ждать завершения всех принятых задач. **`pool.imap(func, iterable)`**, **`pool.imap_unordered(func, iterable)`** — ленивая итерация по результатам (по порядку или по мере готовности).
- **`Pool(..., initializer=func, initargs=())`** — при старте каждого воркера вызывается **`func(*initargs)`** (например, открыть соединение с БД). **`Pool(..., maxtasksperchild=M)`** — после M задач воркер завершается и создаётся новый — снижает утечки памяти в долгоживущих пулах.

#### Разница map и apply_async

| Метод | Поведение | Когда использовать |
|-------|-----------|--------------------|
| **map(func, iterable)** | Разбивает iterable на чанки, распределяет по воркерам, возвращает список результатов **в порядке iterable**. Блокирует до завершения всех. | Когда нужны все результаты в том же порядке, что и аргументы. |
| **apply_async(func, args)** | Отправляет одну задачу, возвращает **AsyncResult**; **result.get()** блокирует до результата. Можно отправить много задач и затем **get()** по одной или с **as_completed** (через конвертацию в list). | Когда нужны результаты по мере готовности или неблокирующая отправка. |
| **imap(func, iterable)** | Итератор по результатам **в порядке iterable**; следующий результат ждётся при следующей итерации. | Когда iterable большой и не нужно хранить все результаты в памяти. |
| **imap_unordered(func, iterable)** | Итератор по результатам **по мере готовности** (порядок не гарантирован). | Когда порядок не важен и нужна ленивая обработка. |
| **map_async(func, iterable)** | Неблокирующий аналог **map**: возвращает **AsyncResult** (как у **apply_async**); метод **get(timeout=None)** возвращает **список** результатов **в порядке iterable** после завершения всех задач. Удобно отправить все задачи «разом» и позже дождаться результатов одним вызовом **result.get(timeout)**. Поддерживает **callback** и **error_callback** — вызываются по завершении всех задач с итоговым списком (или исключением). | Когда нужны все результаты в порядке аргументов, но не хочется блокировать текущий поток до завершения — отправить задачи, заняться другим (или отправить ещё пулы), затем **get()**. |

**map_async** по смыслу — то же, что отправить много **apply_async** и собрать результаты в порядке; но API короче для типичного случая «применить функцию ко всем элементам iterable». Отличие от **map**: **map** блокирует вызывающий поток до завершения всех воркеров; **map_async** сразу возвращает **AsyncResult**, блокировка происходит только при вызове **result.get()**. Если нужно дождаться с таймаутом — **result.get(timeout=10)**; если задача ещё не готова, будет **multiprocessing.TimeoutError**.

### 23.4 Синхронизация процессов: Lock, Semaphore, Event в multiprocessing

В **multiprocessing** есть свои **`multiprocessing.Lock`**, **Semaphore**, **Event**, **Condition**, **Barrier** — они передаются между процессами (сериализуются) и используются для координации процессов. **`Value(..., lock=True)`** и **`Array(..., lock=True)`** дают атомарный доступ к значению/элементам через внутренний Lock.

### 23.5 Тонкости multiprocessing: spawn, fork, if __name__ == '__main__'

- При **spawn** (Windows, по умолчанию на macOS) код модуля импортируется заново в дочернем процессе; без **`if __name__ == '__main__':`** возможен бесконечный порождение процессов. Всегда оборачивайте точку входа в **`if __name__ == '__main__':`**.
- **fork** копирует память процесса (быстрый старт, но не на Windows); **spawn** запускает новый интерпретатор (чистый старт).

#### Zombie-процессы: что это и как избежать

**Zombie-процесс** (зомби) — дочерний процесс, который уже **завершился** (выполнил exit), но запись о нём ещё хранится в ядре ОС: ядро сохраняет код выхода и метаданные, пока родитель не прочитает их через **wait** (reap). В списке процессов (например, **ps** или **top**) такой процесс помечен состоянием **Z** (zombie); он не выполняет код и почти не потребляет CPU/память, но **занимает запись в таблице процессов**. Если таких записей много, можно упереться в лимит ОС на число процессов (ulimit, kernel limit). После того как родитель вызвал **wait** (в Python — **process.join()**), ядро освобождает запись и зомби исчезает. Если родитель **никогда** не вызывает **wait**, зомби остаются до завершения родителя; после смерти родителя их «подбирает» init (PID 1), и они исчезают.

**Почему появляются:** родитель создал процесс через **Process(...).start()**, но не вызвал **process.join()** (или не ждёт его через другой механизм). Пока родитель не дождётся завершения дочернего, ядро хранит код выхода дочернего — процесс остаётся в состоянии zombie. Типично: запустили много дочерних процессов и не вызываете **join()** для каждого; либо обрабатываете только часть процессов, остальные «забыты».

**Как избежать:** (1) **join()** — для каждого созданного процесса вызывать **process.join()** (или эквивалент), чтобы родитель дождался завершения и ядро освободило запись (reap). (2) **daemon** — установить **process.daemon = True** до **start()**: при завершении родителя дочерний принудительно завершается; родитель обычно не ждёт его через **join**, зомби не накапливаются (процесс либо ещё жив, либо уже подобран при выходе родителя). Минус: дочерний может быть оборван в любой момент. (3) **Pool** — использовать **pool.close()** и **pool.join()**, чтобы родитель дождался всех воркеров. (4) В особых случаях (например, обработчик SIGCHLD или множество короткоживущих детей) можно явно вызывать **os.waitpid(-1, os.WNOHANG)** в цикле, чтобы «подбирать» завершённых детей без блокировки; в обычном коде достаточно **join()** или **daemon**. Итог: всегда либо **join()** для не-daemon процессов, либо **daemon=True** с пониманием последствий.

### 23.6 multiprocessing.shared_memory (3.8+)

**`multiprocessing.shared_memory.SharedMemory(name=..., create=True, size=N)`** — создаёт или подключается к именованной области разделяемой памяти. **`shm.buf`** — **memoryview** для чтения/записи; **`shm.close()`** — закрыть доступ в текущем процессе; **`shm.unlink()`** — удалить область (обычно в процессе-создателе после завершения всех). Передача **`name`** в дочерние процессы позволяет им подключаться к той же области без pickle больших данных.

#### Manager().Value() / Manager() vs shared_memory — когда что использовать

| Критерий | **Manager().Value()** / **Manager().dict()** и т.п. | **shared_memory (SharedMemory)** |
|----------|-----------------------------------------------------|----------------------------------|
| **Тип данных** | Один скаляр (Value) или сложная структура (dict, list) через прокси. | Произвольный кусок байтов (буфер); типы задаёте сами (структуры C, массивы через memoryview). |
| **Сериализация** | Изменения через прокси сериализуются (pickle) и передаются процессу-менеджеру. | Нет pickle — прямая запись в общую память; быстрее для больших объёмов. |
| **Производительность** | Удобно для малого числа операций; при частом обмене накладные расходы на прокси и pickle заметны. | Минимальные накладные расходы; подходит для частого обмена большими данными (массивы, буферы). |
| **Когда использовать** | Несколько скаляров или разделяемые dict/list с нечастым доступом; проще API. | Большие массивы, буферы, бинарные данные; максимальная скорость; один и тот же layout памяти в разных процессах. |
| **Ограничения** | Manager — отдельный процесс-посредник; не все объекты сериализуемы. | Нужно самому задать layout (размер, типы); нет встроенной синхронизации — при необходимости использовать отдельно Lock. |

**Кратко:** для разделяемого **dict/list** или нескольких **скаляров** с простым API — **Manager**. Для **больших массивов или буферов** без pickle и с минимальными накладными расходами — **shared_memory**.

**Конкретные сценарии:** (1) Несколько процессов пишут в общий счётчик или флаг — **Manager().Value()** или **Value(..., lock=True)** из **multiprocessing** (проще, без отдельного процесса-менеджера). (2) Общий словарь/список (конфиг, очередь заданий) с нечастым доступом — **Manager().dict()** / **list()**. (3) Большая матрица или буфер (например, изображение, тензор), к которой часто обращаются несколько процессов — **shared_memory**: создаёте **SharedMemory(size=...)** в родителе, передаёте **name** дочерним, все работают с **shm.buf** (memoryview); типы и layout задаёте сами (например, **struct.pack**/unpack или numpy-массив поверх буфера). (4) Много мелких обновлений скаляров или структур — Manager даёт накладные расходы на каждый вызов; **Value/Array** с lock или **shared_memory** с отдельным **Lock** часто быстрее.

### 23.7 Pool: initializer и maxtasksperchild

**`Pool(processes=N, initializer=func, initargs=())`** — при **старте каждого воркера** вызывается **`func(*initargs)`** до выполнения первой задачи. Удобно для: открытия соединения с БД, загрузки тяжёлых данных в память, настройки окружения (например, переменные окружения). Инициализация выполняется один раз на процесс; все задачи в этом воркере видят результат (например, глобальное соединение с БД).

**`Pool(..., maxtasksperchild=M)`** — после выполнения **M задач** воркер завершается и создаётся новый процесс. Снижает накопление утечек памяти в долгоживущих пулах: если каждая задача понемногу течёт память, перезапуск воркера освобождает её. При **maxtasksperchild=None** (по умолчанию) воркеры живут до закрытия пула.

**Пример: initializer для соединения с БД (псевдокод)**

```python
_db_conn = None

def init_db():
    global _db_conn
    _db_conn = create_connection()

def worker(task_id):
    # _db_conn доступен в каждом воркере после init_db
    return _db_conn.execute("SELECT ...", task_id)

if __name__ == '__main__':
    with multiprocessing.Pool(4, initializer=init_db) as pool:
        results = pool.map(worker, range(100))
```

#### Граничные случаи multiprocessing (§23)

- **Забыть `if __name__ == '__main__':` при spawn:** при импорте модуля в дочернем процессе код создания процессов выполнится снова — бесконечное порождение процессов (на Windows/macOS с spawn). Всегда оборачивайте точку входа в **`if __name__ == '__main__':`**.
- **Передать в процесс объект, не сериализуемый через pickle:** lambda, вложенная функция, объект с несериализуемыми атрибутами — **PicklingError**. Функция должна быть доступна по имени в дочернем процессе (обычно функция верхнего уровня модуля); аргументы — pickle-совместимые.
- **Pool без close/join:** воркеры останутся висеть; при выходе из контекста **`with Pool() as pool`** вызываются **terminate** и **join**. Без контекстного менеджера нужно явно **pool.close()** и **pool.join()**.
- **Value/Array: типы не совпадают:** typecode должен соответствовать записываемым значениям (например, для int — `'i'` или `'l'`); иначе возможны некорректные данные или ошибки.
- **SharedMemory: не вызвать unlink():** именованная область остаётся в ОС после завершения процесса; при повторном запуске **create=True** создаст новую с другим именем. В процессе-создателе после завершения всех дочерних вызвать **shm.unlink()**.

#### Пошагово: что происходит при process.start() (spawn)

1. **Родитель:** создаётся объект **Process(target=func, args=(...))**; дочерний процесс ещё не существует.
2. **process.start()** — ОС создаёт новый процесс; в нём запускается новый интерпретатор Python.
3. **В дочернем процессе:** модуль с кодом **импортируется заново**; выполняется весь код верхнего уровня модуля. Если там есть вызовы **Process(...).start()** без защиты **`if __name__ == '__main__':`**, снова создадутся процессы — бесконечное размножение.
4. **В дочернем процессе:** выполняется **func(*args)**; аргументы и функция переданы через **pickle**.
5. **process.join()** в родителе — родитель блокируется до завершения дочернего процесса. **process.exitcode** — код выхода после завершения.

**Запомните:** при **spawn** код модуля выполняется в дочернем процессе при импорте; поэтому точка входа (создание процессов, main) **обязательно** под **`if __name__ == '__main__':`**.

#### Таблица методов и атрибутов Process

| Метод / атрибут | Назначение |
|-----------------|------------|
| `process.start()` | Запустить процесс (один раз). |
| `process.join(timeout=None)` | Блокировать до завершения процесса (или timeout). |
| `process.terminate()` | Принудительно завершить процесс (SIGTERM). |
| `process.kill()` | Немедленно завершить (SIGKILL). |
| `process.is_alive()` | Жив ли процесс в данный момент. |
| `process.exitcode` | Код выхода (None пока жив; 0 — успех; <0 — сигнал). |
| `process.pid` | Идентификатор процесса (или None до start). |

#### Плохо vs лучше (multiprocessing)

| Плохо | Лучше |
|-------|--------|
| Создавать процессы на верхнем уровне модуля (без `if __name__ == '__main__'`) при spawn | Весь код создания процессов и **main()** внутри **`if __name__ == '__main__':`**. |
| Передавать в процесс lambda или вложенную функцию | Функция верхнего уровня модуля; аргументы — pickle-совместимые. |
| Забыть **pool.close()** и **pool.join()** при ручном создании Pool | **`with Pool() as pool:`** или явно **close()** и **join()**. |
| Долгоживущий Pool без **maxtasksperchild** при утечках памяти | **Pool(..., maxtasksperchild=M)** — перезапуск воркеров после M задач. |
| SharedMemory: не вызвать **unlink()** после использования | В процессе-создателе после завершения всех дочерних вызвать **shm.unlink()**. |

#### Резюме по §23 (multiprocessing)

- **Процессы** — отдельная память, свой GIL; реальный параллелизм. Подходит для **CPU-bound**.
- **Создание:** **Process(target=..., args=...)**, **start()**, **join()**. При **spawn** — **`if __name__ == '__main__':`** обязательно.
- **Обмен данными:** **Queue**, **Pipe** (pickle); **Value**/ **Array** (разделяемая память, typecode); **Manager** (dict, list); **shared_memory** (3.8+, без pickle).
- **Пул:** **Pool(map, apply_async, close, join)**; **initializer** — при старте воркера; **maxtasksperchild** — против утечек.
- **fork** vs **spawn**: fork — копирование памяти (быстро, не на Windows); spawn — новый интерпретатор (чистый старт, нужен **if __name__**).

**Запомните:** для **CPU-bound** используйте **multiprocessing**; данные между процессами передавайте через **Queue**, **Pipe**, **Value/Array** или **shared_memory**. При spawn обязательно используйте **`if __name__ == '__main__':`**.
