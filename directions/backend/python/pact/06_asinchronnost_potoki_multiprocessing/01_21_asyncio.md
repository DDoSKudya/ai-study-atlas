## §21. asyncio — основы

**Цель раздела §21:** научиться писать асинхронный код на Python 3: понимать событийный цикл (event loop), объявлять корутины через `async def`, ожидать результат через `await`, запускать задачи через `create_task` и `gather`, использовать примитивы синхронизации (Lock, Queue, Event), вызывать блокирующий код через `run_in_executor` и `to_thread`, корректно обрабатывать отмену и исключения, использовать Streams и асинхронный subprocess. В итоге вы сможете строить эффективные I/O-bound приложения без блокировки одного потока на множество операций.

---

**Асинхронность** (asyncio) в Python — это модель **кооперативной многозадачности** в **одном потоке**: одна и та же нить выполнения переключается между задачами в моменты, когда задача явно «отдаёт управление» (например, при `await asyncio.sleep(0)` или при `await` на сетевой операции). Выполняется всегда только один кусок кода; переключение происходит без прерывания по таймеру ОС (в отличие от потоков). За счёт этого можно обслуживать тысячи соединений без создания тысяч потоков: пока одно соединение ждёт ответа по сети, цикл выполняет код другого.

**Термины:**

| Термин | Определение |
|--------|-------------|
| **Event loop** (событийный цикл) | Движок, который в одном потоке по очереди запускает готовые к выполнению корутины и реагирует на события I/O (сокеты, таймеры). |
| **Корутина** (coroutine) | Объект, порождённый вызовом функции, объявленной как `async def`. Выполняется только при передаче в event loop (через `await`, `create_task` или `asyncio.run()`). |
| **Корутинная функция** (coroutine function) | Функция, объявленная как `async def`. При вызове возвращает корутину и **не** выполняет тело функции до первого `await` или до планирования в loop. |
| **Awaitable** | Объект, который можно использовать в `await`: корутина, Task, Future и объекты с `__await__`. |
| **Task** | Планируемая в event loop «обёртка» вокруг корутины; позволяет запускать корутину «в фоне» и ждать её позже. |

**Запомните:** корутина **не выполняется** до тех пор, пока её не передадут в event loop (например, через `await` или `asyncio.create_task()`). Просто вызвать `async def f(): ...` и получить объект корутины — значит только создать объект; код внутри `f` начнёт выполняться, когда loop начнёт её исполнять.

#### Теоретическое обоснование: PEP 492 и PEP 3156

- **PEP 492** (Python 3.5) — «Coroutines with async and await»: введение синтаксиса `async def` и `await` как замену генераторной-based корутин и callback hell. Цель — читаемый асинхронный код без вложенных callback’ов.
- **PEP 3156** — «Asyncio — Asynchronous I/O, event loop, coroutines and tasks»: стандартная библиотека `asyncio` с единым event loop, примитивами (Task, Future, Lock, Queue) и транспортами/протоколами. Мотивация: единый способ писать асинхронный I/O в стандартной библиотеке.

Таким образом, asyncio — не «потоки без потоков», а **кооперативная многозадачность**: каждая корутина сама в точках `await` отдаёт управление циклу; цикл переключается на другую задачу или ждёт I/O. Блокирующий вызов без `await` (например, `time.sleep(1)`) останавливает весь loop — поэтому блокирующий код выносят в executor.

**Почему asyncio не заменяет threading:** asyncio эффективен, когда все операции I/O асинхронные (async-библиотеки). Если используется только синхронный код (requests, sync БД), каждый вызов блокирует loop — тогда проще использовать **threading** или **ThreadPoolExecutor**, либо оборачивать блокирующие вызовы в **asyncio.to_thread**. Кроме того, в одном процессе может быть только один активный event loop в одном потоке; для «фоновых» задач в другом потоке с своим loop нужна аккуратность.

**Почему multiprocessing «тяжёлый»:** каждый процесс — отдельный интерпретатор Python, отдельная память. Запуск процесса и обмен данными (pickle, очереди, pipe) дают накладные расходы. Поэтому multiprocessing выгоден для **долгих** CPU-bound задач; для коротких задач накладные расходы могут перевесить выигрыш от параллелизма.

### 21.1 Основы: event loop, async/await

**Событийный цикл** (event loop) — это однопоточный диспетчер: он хранит очередь готовых к выполнению задач (корутин и callback’ов), по одной выполняет их до точки «ожидания» (`await`), при ожидании регистрирует обратный вызов (например, «когда сокет станет читаемым — продолжить эту корутину») и переходит к следующей задаче. Когда I/O завершается или срабатывает таймер, соответствующая задача снова попадает в очередь. Таким образом, один поток переключается между множеством логических задач без создания потоков ОС.

#### Схема потока выполнения: как работает event loop

```
    asyncio.run(main())
           │
           ▼
    ┌──────────────────────┐
    │ Создать новый loop    │
    │ set_event_loop(loop)  │
    └──────────┬───────────┘
               ▼
    ┌──────────────────────┐
    │ loop.run_until_       │
    │   complete(main())    │
    └──────────┬───────────┘
               │
               ▼
    ┌──────────────────────┐     await?    ┌─────────────────────────┐
    │ Выполнять main()      │ ────────────► │ Приостановить корутину,  │
    │ до первого await      │               │ зарегистрировать callback│
    └──────────┬───────────┘               │ (таймер, сокет, ...)     │
               │                            └────────────┬────────────┘
               │                                         │
               │                            ┌────────────▼────────────┐
               │                            │ Взять следующую задачу   │
               │                            │ из очереди готовых       │
               │                            └────────────┬────────────┘
               │                                         │
               │         I/O готов / таймер сработал     │
               │◄────────────────────────────────────────┘
               │  (текущая корутина снова в очереди)
               ▼
    ┌──────────────────────┐
    │ main() завершилась    │
    │ вернуть результат    │
    └──────────┬───────────┘
               ▼
    ┌──────────────────────┐
    │ loop.close()          │
    │ Вернуть результат    │
    └──────────────────────┘
```

**Суть:** выполняется всегда только одна корутина в один момент. Переключение происходит **только** в точках `await`: корутина отдаёт управление циклу, цикл либо запускает другую готовую задачу, либо ждёт событий (I/O, таймер). Блокирующий вызов без `await` (например, `time.sleep(1)` или синхронный `requests.get`) **не** отдаёт управление — весь loop стоит.

#### Пошагово: что происходит при вызове `asyncio.run(main())`

1. **Создаётся новый event loop** для текущего потока (или используется политика по умолчанию).
2. **Устанавливается как текущий** для этого потока: `set_event_loop(loop)`.
3. **Планируется корутина** `main()`: она попадает в очередь готовых к выполнению задач.
4. **Цикл запускает** корутину `main()`: выполняется код до первого **`await`**.
5. При **`await expr`**: вычисляется `expr` (должен быть awaitable). Если это корутина или Task — она планируется/уже в очереди. Текущая корутина **приостанавливается**; цикл регистрирует callback (когда awaitable завершится — продолжить текущую корутину) и переходит к другой задаче или ждёт событий.
6. Когда **awaitable завершён** (например, `asyncio.sleep(1)` отработал): текущая корутина снова попадает в очередь готовых; цикл рано или поздно её подхватит и продолжит выполнение после `await`.
7. Когда **main() возвращает** (нормально или по исключению): `run_until_complete` возвращает результат (или пробрасывает исключение).
8. **Loop закрывается**: `loop.close()`. Повторно использовать этот объект loop нельзя.

**Запомните:** переключение между корутинами происходит **только** в точках `await`. Между двумя `await` выполняется атомарный кусок кода без переключения (но он не должен быть долгим, иначе блокируется весь loop).

#### Таблица: когда корутина приостанавливается и когда продолжается

| Ситуация | Поведение |
|----------|-----------|
| Корутина вызвана через `await coro()` или `asyncio.run(coro)` | Тело корутины выполняется до первого `await expr`. |
| Встречен `await expr` | Текущая корутина приостанавливается; управление в loop; когда `expr` завершён — корутина продолжается. |
| `expr` — уже завершённый Future/корутина | Корутина может продолжиться сразу (или на следующем «тике» цикла). |
| `expr` — Task или «долгая» корутина | Корутина приостанавливается на время выполнения другой задачи. |
| Корутина вызвана как `coro()` без await и без create_task | Тело **не** выполняется; создаётся только объект корутины. Предупреждение: «coroutine was never awaited». |
| Внутри корутины вызван блокирующий код (time.sleep, requests.get) | Весь loop блокируется; другие корутины не выполняются до возврата из блокирующего вызова. |

**Получение и запуск цикла:**

- **`asyncio.run(coro)`** (рекомендуемый способ в прикладном коде): создаёт новый event loop, запускает переданную корутину до завершения, закрывает loop и возвращает результат корутины. После вызова `asyncio.run()` этот loop закрыт — повторно использовать его нельзя.
- **Внутри кода:** `loop = asyncio.get_event_loop()` — получить текущий loop (в контексте потока); `asyncio.new_event_loop()` — создать новый; `asyncio.set_event_loop(loop)` — установить как текущий для потока. Обычно в скриптах достаточно `asyncio.run(main())`.

**Эквивалентность asyncio.run(main()) и ручного управления loop (псевдокод):**

```python
# asyncio.run(main()) по сути эквивалентно:

loop = asyncio.new_event_loop()
asyncio.set_event_loop(loop)
try:
    return loop.run_until_complete(main())
finally:
    try:
        loop.run_until_complete(loop.shutdown_asyncgens())
    finally:
        loop.close()
```

То есть создаётся новый loop, устанавливается как текущий, запускается корутина до завершения, затем вызывается shutdown для асинхронных генераторов и loop закрывается.

#### Когда создавать loop вручную: run_until_complete() vs asyncio.run()

**asyncio.run(coro)** — рекомендуемый способ для точки входа: создаёт loop, запускает корутину, по завершении вызывает **shutdown_asyncgens** и **close()**. Один вызов — один «прогон»; повторный вызов **asyncio.run()** создаёт **новый** loop. Для обычных скриптов этого достаточно.

**Ручное создание loop** (new_event_loop + set_event_loop + **loop.run_until_complete(coro)**) нужно, когда:

- **Несколько запусков в одном loop:** интерактивная среда (REPL, Jupyter), тесты (pytest с одним loop на модуль/сессию), скрипт, где вы по очереди вызываете несколько «шагов» — создаёте loop один раз, затем многократно **run_until_complete(step_coro)**. Если каждый раз вызывать **asyncio.run()**, loop каждый раз создаётся и закрывается — лишние накладные расходы и невозможность держать состояние в объектах, привязанных к loop (например, executor).
- **Доступ к loop до/после запуска:** регистрация callback’ов по таймеру, настройка **set_default_executor** или передача своего **Executor** в **run_in_executor**, использование **loop.add_reader** / **add_writer** (низкоуровневый API). Внутри **asyncio.run()** вы не получаете ссылку на созданный loop до начала выполнения корутины.
- **Встроенный/легаси код:** фреймворк или библиотека сами создают loop (**new_event_loop()**, **set_event_loop(loop)**) и запускают корутины через **run_until_complete**. Ваш код тогда получает уже существующий loop через **asyncio.get_event_loop()** (или **get_running_loop()** изнутри корутины) и не должен вызывать **asyncio.run()** — иначе «This event loop is already running».
- **Интеграция с GUI:** Qt, Gtk и т.п. предоставляют свой event loop; он часто «привязывается» к asyncio через адаптер. Запуск корутин идёт через их **run_until_complete**-аналог или через **create_task** в уже работающем loop.

**Как делать:** создать loop — **loop = asyncio.new_event_loop()**; установить как текущий для потока — **asyncio.set_event_loop(loop)** (иначе **get_event_loop()** в этом потоке может вернуть другой loop или выдать предупреждение в 3.10+). Запускать корутины — **result = loop.run_until_complete(coro)**. По завершении всей работы — **loop.run_until_complete(loop.shutdown_asyncgens())**, затем **loop.close()**. Если loop не закрывать (например, планируете ещё запуски), не вызывайте **close()** до конца. **Не** вызывать **asyncio.run()** изнутри уже работающего loop — будет **RuntimeError: This event loop is already running**.

```python
# Пример: ручной loop для нескольких запусков (псевдокод)
loop = asyncio.new_event_loop()
asyncio.set_event_loop(loop)
try:
    result1 = loop.run_until_complete(coro1())
    result2 = loop.run_until_complete(coro2())
finally:
    loop.run_until_complete(loop.shutdown_asyncgens())
    loop.close()
```

**Синтаксис:**

- **`async def name(...): ...`** — объявление корутинной функции. Вызов `name(...)` возвращает объект корутины (coroutine object), не запуская тело.
- **`await something`** — приостановить текущую корутину до завершения `something` (awaitable). Управление возвращается в event loop; когда `something` завершится, текущая корутина продолжит выполнение. Писать `await` можно только внутри `async def`.

**Пример: минимальный запуск**

```python
import asyncio

async def main():
    print("Старт")
    await asyncio.sleep(1)
    print("Через 1 с")

# Запуск: создаётся loop, выполняется main(), loop закрывается
asyncio.run(main())
```

**Различие: coroutine vs coroutine function vs Task**

- **Coroutine function** — это `async def f(): ...`. При вызове `f()` возвращается **coroutine** (объект типа `coroutine`).
- **Coroutine** — объект, который можно передать в `await` или в `asyncio.run()`. Выполняется только когда его «запускает» event loop.
- **Task** — объект, оборачивающий корутину и уже запланированный в event loop (например, через `asyncio.create_task(coro)`). Task сам является awaitable; при `await task` вы ждёте завершения этой задачи. Удобно для «запустить и ждать позже» или «запустить несколько задач параллельно».

**Пример: корутина не выполняется без передачи в loop**

```python
async def greet():
    print("Привет")

# Только создаётся объект корутины; "Привет" не печатается
c = greet()

# Выполнение начнётся здесь
asyncio.run(c)
```

#### Протокол Awaitable: что можно ставить после await

Объект можно использовать в **`await`**, если он реализует **awaitable protocol**: метод **`__await__`**, возвращающий итератор (часто сам генератор). В стандартной библиотеке awaitable являются:

- **Корутина** (результат вызова `async def`-функции).
- **asyncio.Task** — обёртка вокруг корутины, запланированная в loop.
- **asyncio.Future** — результат отложенного вычисления; loop устанавливает результат через `future.set_result()` / `future.set_exception()`.

Проверка «является ли объект awaitable» в коде: **`isinstance(obj, collections.abc.Coroutine)`** или **`inspect.iscoroutinefunction(func)`** для функции. Для произвольного awaitable — наличие **`__await__`**.

#### Граничные случаи и типичные ошибки (§21.1)

- **Вызвать корутинную функцию и не передать результат в loop:** `async def f(): ...; f()` — создаётся объект корутины, но тело не выполняется. При завершении программы в Python 3.7+ выдаётся предупреждение: «RuntimeWarning: coroutine 'f' was never awaited». Всегда либо **`await f()`**, либо **`asyncio.create_task(f())`**, либо **`asyncio.run(f())`**.
- **Вызвать `asyncio.run()` изнутри уже работающего loop:** нельзя. `asyncio.run()` создаёт новый loop и блокирует поток до завершения корутины. Изнутри async-кода используйте **`await`** для ожидания другой корутины, а не повторный `run()`. При необходимости вложенного запуска (не рекомендуется) используют библиотеки вроде **nest_asyncio**.
- **Использовать один и тот же объект loop после `loop.close()`:** запрещено. После `asyncio.run()` loop закрыт. Для повторного запуска снова вызовите **`asyncio.run(main())`** — будет создан новый loop.
- **Писать `await` вне async-функции:** синтаксическая ошибка. `await` допустим только внутри тела **`async def`**.

**Запомните:** для запуска «главной» корутины из синхронного кода используйте **`asyncio.run(main())`**. Внутри async-кода используйте **`await`** для ожидания другой корутины или **`create_task`** для параллельного запуска.

### 21.2 Примитивы: create_task, gather, sleep, wait_for, shield

**Цель подраздела:** запускать несколько корутин параллельно (**create_task**, **gather**), ограничивать время выполнения (**wait_for**), защищать задачу от отмены (**shield**), обрабатывать результаты по мере готовности (**wait**, **as_completed**); понимать разницу между **gather** и **wait** и когда что использовать.

**`asyncio.create_task(coro)`** — планирует корутину в текущем event loop и возвращает **Task**. Задача начинает выполняться «в фоне»; вызывающий код может продолжать работу и позже сделать `await task` для получения результата или отменить задачу через `task.cancel()`.

```python
async def main():
    task = asyncio.create_task(asyncio.sleep(2))
    print("Задача запущена")
    await task
    print("Задача завершена")

asyncio.run(main())
```

**`asyncio.gather(*aws, return_exceptions=False)`** — запускает несколько awaitable параллельно и ждёт все. Возвращает список результатов в порядке переданных аргументов. Если `return_exceptions=True`, исключения возвращаются как элементы списка вместо проброса.

```python
async def main():
    results = await asyncio.gather(
        asyncio.sleep(1),
        asyncio.sleep(2),
    )
    # results == [None, None]; оба sleep выполнены параллельно

asyncio.run(main())
```

**`asyncio.sleep(delay)`** — неблокирующая задержка: текущая корутина приостанавливается на `delay` секунд, управление отдаётся в loop. Это основной способ «подвесить» корутину без блокировки потока.

**`asyncio.wait_for(aw, timeout)`** — ждёт завершения awaitable не дольше `timeout` секунд. При превышении таймаута отменяет задачу и возбуждает **`asyncio.TimeoutError`**.

**`asyncio.shield(aw)`** — защищает awaitable от отмены: при отмене внешней задачи (например, при `wait_for` по таймауту) отмена не передаётся внутрь `shield`; внутренняя задача продолжит выполняться (но её результат во внешнем контексте уже не будет использован, если внешняя задача отменена).

**`asyncio.wait(aws, *, return_when=ALL_COMPLETED)`** — ждёт множество задач; `return_when` может быть `FIRST_COMPLETED`, `FIRST_EXCEPTION`, `ALL_COMPLETED`. Возвращает два множества: `(done, pending)`.

**`asyncio.as_completed(aws)`** — возвращает итератор по awaitable; при итерации выдаёт те из них, которые завершились (удобно обрабатывать результаты по мере готовности).

**`asyncio.TaskGroup`** (Python 3.11+) — контекстный менеджер для структурированной конкурентности: все задачи, созданные через `tg.create_task()`, отменяются при выходе из блока при первой ошибке или при отмене; `async with` ждёт завершения всех созданных задач.

```python
# Python 3.11+
async def main():
    async with asyncio.TaskGroup() as tg:
        tg.create_task(asyncio.sleep(1))
        tg.create_task(asyncio.sleep(2))
    # здесь все задачи уже завершены

asyncio.run(main())
```

#### Пример: TaskGroup — отмена всех при первой ошибке (Python 3.11+)

В **TaskGroup** при первом исключении в любой из созданных задач **остальные задачи отменяются**, группа дожидается их завершения, затем пробрасывается первое исключение. Удобно для «пачки» связанных задач: одна ошибка — вся группа отменяется, не остаётся висящих задач.

```python
# Python 3.11+
import asyncio

async def ok():
    await asyncio.sleep(0.5)
    return 1

async def fail():
    await asyncio.sleep(0.2)
    raise ValueError("ошибка в одной из задач")

async def main():
    try:
        async with asyncio.TaskGroup() as tg:
            tg.create_task(ok())
            tg.create_task(fail())
            tg.create_task(ok())
    except* ValueError as e:
        print("Поймано:", e)  # все задачи отменены, первое исключение проброшено

asyncio.run(main())
# Поймано: ExceptionGroup('', (ValueError('ошибка в одной из задач'),))
```

Без TaskGroup при **gather** без `return_exceptions=True` одна ошибка пробрасывается, но остальные задачи продолжают выполняться в фоне. С TaskGroup при первой ошибке **все** созданные в группе задачи отменяются — структурированная конкурентность.

#### Детали примитивов: когда что использовать

| Примитив | Назначение | Особенности |
|----------|------------|-------------|
| **create_task(coro)** | Запустить корутину «в фоне», не дожидаясь её завершения | Возвращает Task; можно позже `await task` или `task.cancel()`. Задача привязана к текущему loop. Вызывать только из контекста, где loop уже запущен (внутри другой корутины). |
| **gather(*aws)** | Запустить несколько awaitable параллельно и дождаться **всех** результатов в порядке аргументов | Результаты в списке в том же порядке, что и аргументы. Одна ошибка без `return_exceptions=True` — исключение пробрасывается, остальные задачи продолжают выполняться. |
| **wait(aws, return_when=...)** | Дождаться части или всех задач | Возвращает `(done, pending)` — множества. Можно `FIRST_COMPLETED`, `FIRST_EXCEPTION`, `ALL_COMPLETED`. Отменённые задачи нужно вручную отменять в `pending` при выходе. |
| **as_completed(aws)** | Итерировать по awaitable **по мере их завершения** | Удобно, когда порядок результатов не важен и нужно обрабатывать первый пришедший результат. |
| **wait_for(aw, timeout)** | Таймаут на одну операцию | При таймауте задача **отменяется** (ей уйдёт CancelledError). Чтобы внутренняя задача не отменялась — обернуть в **shield(aw)**. |
| **shield(aw)** | Защитить awaitable от отмены снаружи | При отмене внешней задачи (например, по wait_for) отмена не передаётся внутрь shield; внутренняя задача продолжит выполняться. Результат снаружи уже не получите (внешняя задача отменена). |
| **TaskGroup** (3.11+) | Структурированная конкурентность: одна ошибка — отмена всех, выход из блока — ожидание всех | При первом исключении в любой задаче остальные отменяются, затем исключение пробрасывается. Удобно для «пачки» связанных задач. |

#### Граничные случаи примитивов (§21.2)

- **gather() без аргументов:** возвращает пустой список `[]`.
- **gather с одной исключившейся корутиной и return_exceptions=False:** исключение пробрасывается в место вызова gather; остальные задачи продолжают выполняться в фоне (их результаты теряются при пробросе).
- **create_task вызывать вне async-контекста:** нужен уже запущенный loop. В точке входа (до любой корутины) loop ещё не запущен — используйте `asyncio.run(main())`, а внутри `main()` уже вызывайте `create_task()`.
- **wait_for(aw, timeout) и отмена:** при таймауте `aw` отменяется; если внутри `aw` перехватывают CancelledError и не пробрасывают — wait_for может не завершиться по таймауту (зависит от реализации). Рекомендация: в перехвате CancelledError всегда делать **raise** после очистки.
- **TaskGroup: выход из блока по исключению:** при выходе из `async with` все созданные задачи отменяются и группа ждёт их завершения, затем пробрасывается первое исключение.

#### Пример: gather с return_exceptions=True

Если одна из корутин в `gather` бросает исключение, по умолчанию это исключение пробрасывается в вызывающий код, а остальные задачи продолжают выполняться (но их результаты отбрасываются при пробросе). С **`return_exceptions=True`** исключения возвращаются в списке результатов вместо проброса — удобно для «собрать все результаты или ошибки».

```python
async def ok():
    return 1

async def fail():
    raise ValueError("ошибка")

async def main():
    results = await asyncio.gather(ok(), fail(), return_exceptions=True)
    # results == [1, ValueError("ошибка")]

asyncio.run(main())
```

#### Пример: wait_for и TimeoutError

```python
async def slow():
    await asyncio.sleep(10)
    return "готово"

async def main():
    try:
        result = await asyncio.wait_for(slow(), timeout=1.0)
    except asyncio.TimeoutError:
        print("Таймаут: задача отменена")
    # slow() получит CancelledError и завершится

asyncio.run(main())
```

#### Пример: as_completed — обработка по мере готовности

Когда порядок результатов не важен и нужно обрабатывать первый пришедший результат, удобен **asyncio.as_completed(aws)**. При итерации **async for** выдаётся очередной завершённый awaitable (корутина или Task).

```python
async def fetch(n):
    await asyncio.sleep(n)
    return n

async def main():
    tasks = [fetch(2), fetch(1), fetch(3)]
    for coro in asyncio.as_completed(tasks):
        result = await coro
        print("Готово:", result)
    # Порядок вывода: 1, 2, 3 (по мере завершения)

asyncio.run(main())
# Готово: 1
# Готово: 2
# Готово: 3
```

**Запомните:** для параллельного запуска нескольких корутин используйте **`asyncio.gather()`** или **`create_task`** + последующий `await`. Для таймаута — **`asyncio.wait_for()`**; при необходимости защитить внутреннюю задачу от отмены — **`asyncio.shield()`**. Для обработки по мере готовности — **as_completed()**.

### 21.3 Синхронизация: Lock, Semaphore, Queue, Event

**Цель подраздела:** защищать общие данные от гонки read–modify–write (**Lock**), ограничивать число одновременных операций (**Semaphore**), передавать данные между корутинами (**Queue**), сигнализировать о событии (**Event**); понимать, зачем в asyncio нужен Lock при одном потоке.

В asyncio один поток выполняет только одну корутину в момент времени, но переключение между корутинами происходит в точках **`await`**. Между двумя `await` выполняется атомарный кусок кода без переключения; но если одна корутина прочитала общую переменную, затем сделала `await` (и управление перешло к другой корутине), та может изменить ту же переменную, и после возврата управления первая корутина запишет устаревшее значение — классическая **гонка (read–modify–write)**. Поэтому для доступа к общему изменяемому состоянию используют примитивы синхронизации: критическую секцию защищают **Lock**, чтобы между «прочитал» и «записал» не было переключения на другую корутину.

**Почему Lock в asyncio нужен:** переключение возможно в каждой точке `await`. Если критические данные читаются и пишутся в разных корутинах, без Lock одна корутина может прочитать, затем сделать `await` (например, на I/O), другая в это время изменит данные, и первая по возврате запишет по старому значению — данные повредятся. Lock гарантирует: пока одна корутина держит lock, другая не войдёт в тот же `async with lock` — она приостановится на `await lock.acquire()` до освобождения.

**`asyncio.Lock`** — мьютекс: только одна корутина может держать блокировку. Использование: `async with lock: ...` или `await lock.acquire()` / `lock.release()`. **Не реентерабельный:** одна и та же корутина не может захватить один и тот же Lock дважды подряд без освобождения — будет deadlock (корутина ждёт сама себя).

```python
lock = asyncio.Lock()

async def critical():
    async with lock:
        # только одна корутина одновременно
        ...
```

**`asyncio.Semaphore(n)`** — ограничивает число одновременных «захватов» до `n`. Удобно ограничивать параллелизм (например, не более 10 одновременных запросов к API).

```python
sem = asyncio.Semaphore(2)

async def limited():
    async with sem:
        ...
```

**`asyncio.Queue`** — асинхронная очередь: **`await queue.put(item)`**, **`item = await queue.get()`**; при полной очереди **put** блокируется до освобождения места, при пустой **get** блокируется до появления элемента. **`queue.join()`** — ждать, пока для каждого **put** не вызван **`queue.task_done()`**. **`asyncio.Queue(maxsize=0)`** — при **maxsize=0** (по умолчанию) размер не ограничен; при **maxsize>0** **put** блокируется при полной очереди. **`put_nowait(item)`** / **`get_nowait()`** — не блокировать (при неудаче — **asyncio.QueueFull** / **QueueEmpty**). Потокобезопасна в рамках одного event loop.

**`asyncio.Event`** — событие: **`await event.wait()`** приостанавливает корутину до вызова **`event.set()`**; **`event.clear()`** сбрасывает флаг (после этого следующие `wait()` снова блокируются). Один раз `set()` — все ждущие в данный момент разблокируются; для повторного ожидания нужно сначала **clear()**, потом снова **wait()**.

```python
# Пример: одна корутина сигнализирует другой о готовности
async def waiter(ev: asyncio.Event):
    print("Ждём сигнал...")
    await ev.wait()
    print("Получили сигнал")

async def signaller(ev: asyncio.Event):
    await asyncio.sleep(1)
    ev.set()

async def main():
    ev = asyncio.Event()
    await asyncio.gather(waiter(ev), signaller(ev))
asyncio.run(main())
# Ждём сигнал... → (1 с) → Получили сигнал
```

**`asyncio.Condition`** — условная переменная: корутина захватывает связанный Lock и вызывает **`await condition.wait()`** — освобождает lock и ждёт, пока другая корутина вызовет **`condition.notify()`** или **`notify_all()`**. После пробуждения lock снова захватывается. Удобно для «жди, пока условие станет истинным» с уведомлением из другой корутины. Использование: **`async with condition:`** перед **wait()** / **notify()**; внутри блока **await condition.wait()** (освобождает lock, ждёт, затем снова захватывает) или **condition.notify(n=1)** / **notify_all()**.

**`asyncio.Barrier`** (Python 3.11+) — барьер на N участников: **`await barrier.wait()`** — корутина блокируется, пока N корутин не вызовут `wait()`; когда N-я вызовет — все N разблокируются одновременно. Используется для синхронизации фазы (например, «все воркеры закончили подготовку — старт»).

#### Пример: asyncio.Condition — ожидание условия с уведомлением

**Condition** отличается от **Event** тем, что ждём не просто «сигнала», а **истинности некоторого условия** над общими данными (например, «очередь не пуста»). Lock связывается с Condition; перед **wait()** и **notify()** нужно захватить lock (через **async with condition:**). Внутри блока: проверить условие в цикле **while not condition_met:** **await condition.wait()** — при вызове **wait()** lock освобождается (чтобы producer мог войти), корутина засыпает; после **notify()** она просыпается и **снова захватывает** lock, затем снова проверяет условие (на случай ложного пробуждения или конкуренции с другими consumer’ами). **notify(n=1)** будит одного ждущего, **notify_all()** — всех.

```python
import asyncio

items = []
condition = asyncio.Condition()

async def consumer():
    async with condition:
        while not items:
            await condition.wait()  # освобождает lock, ждёт; после пробуждения снова захватывает
        item = items.pop()
    print("Получено:", item)

async def producer(value):
    async with condition:
        items.append(value)
        condition.notify()  # разбудить одного ждущего

async def main():
    await asyncio.gather(
        consumer(),           # ждёт
        producer(42),          # кладёт и уведомляет
    )

asyncio.run(main())
# Получено: 42
```

**Когда использовать Condition, а не Event:** если нужно ждать «пока в очереди есть элемент» или «пока флаг + данные согласованы» — **Condition**. Если достаточно «кто-то сказал: готово» без проверки общего состояния — **Event**.

#### Пример: asyncio.Barrier — синхронизация N корутин (Python 3.11+)

**Barrier** нужен, когда N корутин должны **одновременно** перейти от одной фазы к другой (например, все загрузили конфиг — все стартуют). **await barrier.wait()** блокирует корутину, пока все N не вызовут **wait()**; когда N-я вызовет — все N разблокируются. Возвращаемое значение **wait()** — индекс участника (0..N-1), его можно использовать, чтобы назначить одну корутину «лидером» (например, только индекс 0 выполняет сброс общего состояния). После разблокировки барьер снова готов к следующему «раунду» (все N снова могут вызвать **wait()** для следующей фазы).

```python
import asyncio

async def worker(barrier, id):
    print(f"Корутина {id}: подготовка")
    index = await barrier.wait()  # ждём, пока все 3 корутины дойдут сюда; index — 0, 1 или 2
    print(f"Корутина {id}: старт (индекс в барьере: {index})")

async def main():
    barrier = asyncio.Barrier(3)
    await asyncio.gather(
        worker(barrier, 0),
        worker(barrier, 1),
        worker(barrier, 2),
    )

asyncio.run(main())
# Корутина 0: подготовка
# Корутина 1: подготовка
# Корутина 2: подготовка
# Корутина 0: старт (индекс в барьере: 0)
# Корутина 1: старт (индекс в барьере: 1)
# Корутина 2: старт (индекс в барьере: 2)
```

**Когда использовать Barrier:** синхронизация фаз (все воркеры закончили инициализацию — все начинают обработку), параллельные алгоритмы с «рандеву» (все дошли до точки — обменялись данными и пошли дальше).

**Взаимоблокировки в async:** если две корутины захватывают два лока в разном порядке (сначала A потом B у одной, сначала B потом A у другой), возможен deadlock: одна держит A и ждёт B, другая держит B и ждёт A. Решение: **единый порядок захвата** блокировок во всех корутинах (например, всегда сначала lock_a, потом lock_b) или использование таймаутов при acquire (если API поддерживает).

#### Пример: asyncio.Queue — производитель и потребитель

```python
import asyncio

async def producer(q: asyncio.Queue, n: int):
    for i in range(n):
        await q.put(i)
        await asyncio.sleep(0.1)
    await q.put(None)  # сигнал завершения

async def consumer(q: asyncio.Queue):
    while True:
        item = await q.get()
        if item is None:
            q.task_done()
            break
        print("Получено:", item)
        q.task_done()

async def main():
    q = asyncio.Queue()
    await asyncio.gather(
        producer(q, 5),
        consumer(q),
    )

asyncio.run(main())
# Получено: 0 ... Получено: 4
```

#### Граничные случаи asyncio.Queue (§21.3)

- **Queue(maxsize=0)** — по умолчанию размер не ограничен; **put** никогда не блокируется по «полной очереди». При **maxsize=2** третий **put** без **get** блокирует корутину до освобождения места.
- **put_nowait(item)** / **get_nowait()** — не блокируют; при полной очереди **put_nowait** возбуждает **asyncio.QueueFull**; при пустой **get_nowait** — **asyncio.QueueEmpty**. Удобно для «проверить и выйти» или с таймаутом через **wait_for**.
- **join()** — блокирует до тех пор, пока для каждого **put** не был вызван **task_done()**. Количество **task_done()** должно совпадать с количеством **put**; иначе **join** может зависнуть или **ValueError** (task_done больше раз, чем put).
- **Один потребитель, несколько производителей:** один и тот же **None** (или sentinel) можно положить по одному разу от каждого производителя; потребитель выходит после N sentinel'ов или по одному общему sentinel'у по договорённости.

**Запомните:** для эксклюзивного доступа к общему ресурсу в async-коде используйте **`asyncio.Lock`**; для ограничения параллелизма — **`asyncio.Semaphore`**; для передачи данных между корутинами — **`asyncio.Queue`**.

### 21.4 Совместимость с синхронным кодом: run_in_executor, to_thread

**Цель подраздела:** вызывать блокирующий код из async-функций без остановки event loop (**run_in_executor**, **to_thread**); выбирать ThreadPoolExecutor для I/O и ProcessPoolExecutor для CPU-bound; понимать ограничения pickle при ProcessPoolExecutor.

Блокирующий вызов (например, `time.sleep(1)` или синхронный HTTP-запрос) внутри корутины **останавливает весь event loop**: пока этот вызов не вернётся, остальные корутины не выполняются. Поэтому блокирующий код нужно выносить в пул потоков или процессов.

**`loop.run_in_executor(executor, func, *args)`** — выполняет `func(*args)` в указанном **Executor** (например, `ThreadPoolExecutor`) и возвращает awaitable (по сути Future); при `await` корутина приостанавливается до завершения функции. Если **`executor=None`**, используется пул потоков по умолчанию для данного loop (создаётся при первом вызове).

```python
import asyncio

def blocking_io():
    # имитация блокирующего I/O
    import time
    time.sleep(2)
    return "готово"

async def main():
    loop = asyncio.get_running_loop()
    result = await loop.run_in_executor(None, blocking_io)
    print(result)

asyncio.run(main())
```

**`asyncio.to_thread(func, *args, **kwargs)`** (Python 3.9+) — то же по смыслу: выполняет функцию в потоке пула по умолчанию и возвращает awaitable. Удобная обёртка без явного создания executor; поддерживает **kwargs**.

#### Когда использовать ThreadPoolExecutor, когда ProcessPoolExecutor

| Тип блокирующего кода | Рекомендация | Пояснение |
|------------------------|-------------|-----------|
| I/O (файлы, сеть, БД) или вызовы, отпускающие GIL | **ThreadPoolExecutor** или `executor=None` | Потоки разделяют память; GIL освобождается при I/O — другие потоки могут выполняться. |
| Тяжёлые вычисления (CPU-bound) | **ProcessPoolExecutor** | Обход GIL; реальный параллелизм. Минус: сериализация аргументов и результата через pickle; больше накладных расходов. |
| Смешанная нагрузка | I/O — в потоке, CPU — в процессе | В одной async-программе можно использовать и `run_in_executor(None, io_func)`, и `run_in_executor(ProcessPoolExecutor(), cpu_func)`. |

**Пример: run_in_executor с ProcessPoolExecutor для CPU-bound**

```python
import asyncio
from concurrent.futures import ProcessPoolExecutor

def cpu_bound(n):
    return sum(i * i for i in range(n))

async def main():
    loop = asyncio.get_running_loop()
    with ProcessPoolExecutor() as pool:
        result = await loop.run_in_executor(pool, cpu_bound, 1_000_000)
    print(result)

asyncio.run(main())
```

**Граничные случаи:** функция и её аргументы должны быть **сериализуемы через pickle** при использовании ProcessPoolExecutor (глобальные функции или функции, определённые на верхнем уровне модуля; аргументы — встроенные типы, совместимые с pickle). В потоке pickle не нужен.

**Запомните:** любой блокирующий вызов в async-функции блокирует весь event loop. Используйте **`run_in_executor`** или **`asyncio.to_thread()`** для выноса блокирующего кода в поток (или в процесс для CPU-bound).

### 21.5 Отмена и исключения: cancel, CancelledError

**Цель подраздела:** отменять задачи (**task.cancel()**), корректно обрабатывать **CancelledError** (очистка + re-raise), понимать поток отмены и проброс исключений из задачи при **await**; использовать **add_done_callback** при необходимости.

**`task.cancel()`** — запрашивает отмену задачи. Задача при следующей возможности получит **`asyncio.CancelledError`** (в точке ближайшего `await`); если не перехватывать его или перехватить и снова пробросить (**raise**), задача завершится как отменённая. Вызывающий код при `await task` получит **CancelledError**.

**Правило:** **`asyncio.CancelledError`** в корутине не следует ловить без последующего **re-raise** (или явного завершения отмены). Иначе вы «скрываете» отмену: задача считается успешно завершённой, а не отменённой; код, ожидающий отмены (например, wait_for по таймауту), может повести себя некорректно.

**Поток отмены:** при вызове `task.cancel()` задача не останавливается мгновенно. При следующем **await** внутри этой задачи туда «вбрасывается» CancelledError. Если корутина перехватывает CancelledError и не пробрасывает — отмена «поглощается» и задача продолжает выполняться. Если пробрасывает — задача завершается с CancelledError, и при `await task` вызывающий код получит это исключение.

**Проброс исключений:** при **`await task`** если задача завершилась с исключением (не отменой), это исключение пробрасывается в месте await. При **`gather(..., return_exceptions=True)`** исключения возвращаются в списке результатов вместо проброса; **CancelledError** при этом тоже попадает в список как элемент.

**`task.add_done_callback(callback)`** — вызвать **`callback(task)`** при завершении задачи (нормальном, по исключению или по отмене). Callback вызывается в контексте loop; не должен быть корутиной. Удобно для логирования или побочных действий при завершении.

**Пример: корректная обработка CancelledError с очисткой и re-raise**

```python
async def worker():
    resource = acquire()
    try:
        while True:
            await asyncio.sleep(1)
            do_work(resource)
    except asyncio.CancelledError:
        release(resource)  # очистка
        raise               # обязательно пробросить дальше
```

**Граничные случаи:** если в корутине нет `await` после вызова cancel (редко), CancelledError будет вброшен при первом же await ниже по коду. Если корутина перехватывает **BaseException** или **Exception** и не пробрасывает CancelledError — отмена скрывается; избегайте перехвата слишком широких типов в async-коде без re-raise CancelledError.

**Запомните:** при перехвате **`CancelledError`** обязательно снова поднимайте его (**`raise`**) после своей очистки, иначе отмена не будет корректно распространяться.

### 21.6 Тонкости asyncio

**Цель подраздела:** понимать модель «один loop на поток», поведение **asyncio.run()** (shutdown_asyncgens, close), отладочные средства (current_task, all_tasks, PYTHONASYNCIODEBUG); знать про uvloop и nest_asyncio и когда они уместны.

- **Один event loop на поток:** в одном потоке в один момент активен один loop; в другом потоке можно создать свой loop (например, через **`asyncio.new_event_loop()`** и **`asyncio.set_event_loop(loop)`**). Запуск одного и того же loop из двух потоков одновременно не поддерживается.
- **`asyncio.run()`** создаёт loop, запускает корутину, по завершении вызывает **`loop.shutdown_asyncgens()`** (закрытие асинхронных генераторов) и **`loop.close()`** — повторно использовать этот объект loop нельзя.
- **`asyncio.current_task(loop=None)`** — возвращает текущую **Task** в данном loop (или **`None`** вне async-контекста, например в callback или в run_in_executor). **`asyncio.all_tasks(loop=None)`** — множество всех активных задач (не завершённых) для указанного loop; если loop не передан — берётся текущий.
- **Вложенные loops:** по умолчанию в одном потоке нельзя вызывать **`run_until_complete`** или **`run`** изнутри уже работающего loop — будет ошибка «This event loop is already running». Библиотека **`nest_asyncio`** патчит loop и позволяет вложенный запуск (например, вызов синхронного кода, который внутри вызывает asyncio.run). Использовать с осторожностью: порядок выполнения и отмена могут вести себя иначе.
- **`uvloop`** — быстрая альтернативная реализация event loop на C (на базе libuv). Подмена: **`asyncio.set_event_loop_policy(uvloop.EventLoopPolicy())`** до первого **`asyncio.run()`** или создания loop. Совместим с большинством API asyncio; даёт существенный прирост производительности на I/O-bound нагрузках.
- **Отладка:** переменная окружения **`PYTHONASYNCIODEBUG=1`** включает режим отладки: предупреждения о корутинах, которые не были ожидаемы (never awaited), о долгих операциях (slow callback), и т.п. Полезно при разработке.
- **Запуск из Jupyter/IPython:** в среде с уже работающим loop (например, Jupyter) **`asyncio.run()`** может не подходить — loop уже запущен. Используйте **`await main()`** в ячейке или **`nest_asyncio.apply()`** если нужно вызывать run изнутри.
- **get_running_loop() vs get_event_loop():** внутри **async-кода** (в корутине) используйте **`asyncio.get_running_loop()`** — возвращает текущий запущенный loop; вызов из синхронного контекста без запущенного loop возбуждает **RuntimeError**. **`get_event_loop()`** в Python 3.10+ deprecated в пользу **get_running_loop()** внутри async и **new_event_loop()** / **set_event_loop()** в точке входа; в легаси-коде или при ручном создании loop **get_event_loop()** ещё можно использовать с осторожностью (в точке входа после **set_event_loop(loop)** вернёт установленный loop).

### 21.7 asyncio: Streams и subprocess

**Цель подраздела:** писать TCP-клиент и сервер через **open_connection** / **start_server** и **StreamReader/StreamWriter**; запускать подпроцессы через **create_subprocess_exec** и обмениваться данными через **communicate()**; не блокировать event loop на I/O сокетов и процессов.

**Сетевой I/O (Streams):**

- **`asyncio.open_connection(host, port)`** — TCP-клиент; возвращает **`(StreamReader, StreamWriter)`**. Reader и Writer привязаны к соединению; после использования нужно закрыть Writer.
- **`asyncio.start_server(callback, host, port)`** — TCP-сервер; возвращает **Server**; для каждого нового подключения вызывается **`callback(reader, writer)`** (callback может быть корутиной). Сервер закрывается через **`server.close()`** и **`await server.wait_closed()`**.
- **StreamReader:** **`await reader.read(n=-1)`** — до n байт или до EOF; **`await reader.readline()`** — до `\n`; **`await reader.readuntil(separator)`** — до указанного разделителя; **`reader.at_eof()`** — достигнут ли EOF.
- **StreamWriter:** **`writer.write(data)`** — записать байты (буферизовано); **`await writer.drain()`** — дождаться отправки буфера; **`writer.close()`**, **`await writer.wait_closed()`** — закрыть соединение.

#### Пример: StreamReader — read(n), readline и readuntil

У **StreamReader** три основных способа чтения: **read(n=-1)** — до **n** байт или до EOF (удобно для фиксированных блоков или буферов); **readline()** — до первого **b'\\n'** (текстовые протоколы по строкам); **readuntil(separator)** — до первого вхождения **separator** (например, **b'\\r\\n'** или **b'|'**). Чтение построчно или до разделителя типично для текстовых протоколов (HTTP заголовки, Redis, по одной строке).

```python
import asyncio

async def read_lines_client():
    reader, writer = await asyncio.open_connection('127.0.0.1', 8888)
    try:
        while not reader.at_eof():
            line = await reader.readline()  # до b'\n'
            if line:
                print("Строка:", line.decode().strip())
    finally:
        writer.close()
        await writer.wait_closed()
```

Чтение **до разделителя** (например, до **b'|'** или **b'\\r\\n'**):

```python
# Читаем до разделителя; если соединение закроют до появления separator — IncompleteReadError
try:
    chunk = await reader.readuntil(separator=b'|')
except asyncio.IncompleteReadError as e:
    chunk = e.partial  # прочитанные байты до закрытия
```

**read(n)** — когда нужен ровно **n** байт или «сколько есть» (например, **await reader.read(4096)** для чанков фиксированного размера). **readuntil(separator)** возбуждает **IncompleteReadError**, если соединение закрыто до появления разделителя; обрабатывайте исключение, если данные могут быть обрезаны.

**Пример: простой TCP-клиент (эхо)**

```python
import asyncio

async def tcp_client():
    reader, writer = await asyncio.open_connection('127.0.0.1', 8888)
    writer.write(b'Hello')
    await writer.drain()
    data = await reader.read(1024)
    print(data.decode())
    writer.close()
    await writer.wait_closed()

# asyncio.run(tcp_client())
```

**Пример: простой TCP-сервер (эхо)**

```python
async def handle(reader, writer):
    data = await reader.read(1024)
    writer.write(data)
    await writer.drain()
    writer.close()
    await writer.wait_closed()

async def main():
    server = await asyncio.start_server(handle, '0.0.0.0', 8888)
    async with server:
        await server.serve_forever()

# asyncio.run(main())
```

**Подпроцессы:**

- **`asyncio.create_subprocess_exec(cmd, *args, stdin=..., stdout=..., stderr=..., ...)`** — асинхронный запуск процесса (без shell). Возвращает **Process**; **`await process.communicate(input=None)`** — ожидание завершения процесса и обмен данными через stdin/stdout/stderr (возвращает `(stdout_data, stderr_data)`).
- **`asyncio.create_subprocess_shell(cmd)`** — запуск через shell (на Unix — `/bin/sh`, на Windows — `COMSPEC`). Команда передаётся как **одна строка** в shell; shell разбирает её сам (перенаправления, пайпы, подстановка переменных).

**Риски create_subprocess_shell и когда допустимо:** если в **cmd** попадают пользовательские или внешние данные (ввод пользователя, переменные из конфига, ответ API), возможна **инъекция команд**: shell интерпретирует `;`, `|`, `$()`, `` ` `` и т.д. Например, `cmd = f"echo {user_input}"` при `user_input = "; rm -rf /"` выполнит и echo, и rm. Даже «безопасные» символы в одном аргументе могут объединяться с другими частями строки и менять смысл команды. **Рекомендация:** по умолчанию **предпочитайте create_subprocess_exec(program, arg1, arg2, ...)** — аргументы передаются в процесс **напрямую**, без разбора shell’ом; инъекция невозможна, так как нет парсера командной строки. **create_subprocess_shell** допустим только когда (1) **cmd** полностью под вашим контролем (захардкоженная строка или строка без пользовательского ввода), либо (2) вам сознательно нужна семантика shell (перенаправления `>`, пайпы `|`, подстановка переменных `$VAR`). В последнем случае по возможности формируйте команду из доверенных частей и не подставляйте в одну строку непроверенный ввод.
- У процесса: **`process.pid`**, **`process.returncode`** (устанавливается после завершения), **`process.stdin`**, **`process.stdout`**, **`process.stderr`** (StreamWriter/StreamReader при использовании PIPE).

**Пример: асинхронный subprocess**

```python
async def run_cmd():
    proc = await asyncio.create_subprocess_exec(
        'echo', 'hello',
        stdout=asyncio.subprocess.PIPE,
        stderr=asyncio.subprocess.PIPE,
    )
    stdout, stderr = await proc.communicate()
    print(stdout.decode(), proc.returncode)

asyncio.run(run_cmd())
# hello 0
```

**Запомните:** для TCP в asyncio используйте **`open_connection`** / **`start_server`** и объекты **StreamReader/StreamWriter**; для запуска процессов — **`create_subprocess_exec`** и **`process.communicate()`**.

#### Плохо vs лучше (asyncio)

| Плохо | Лучше |
|-------|--------|
| `time.sleep(1)` внутри async-функции | `await asyncio.sleep(1)` — не блокирует loop |
| Синхронный HTTP в корутине без обёртки | `await asyncio.to_thread(requests.get, url)` или async-библиотека (aiohttp) |
| Ловить `CancelledError` и не пробрасывать дальше | В except: очистка, затем `raise` |
| Много `await coro1(); await coro2();` когда порядок не важен | `await asyncio.gather(coro1(), coro2())` — параллельно |
| Запуск корутины без передачи в loop (`coro();` без await) | `await coro()` или `asyncio.create_task(coro())` |

#### Таблица сигнатур основных примитивов asyncio (§21)

| Функция / метод | Сигнатура (кратко) | Возвращает |
|-----------------|--------------------|------------|
| `asyncio.run(coro)` | `run(coro, *, debug=None)` | Результат корутины |
| `asyncio.create_task(coro)` | `create_task(coro, *, name=None)` | Task |
| `asyncio.gather(*aws)` | `gather(*aws, return_exceptions=False)` | list результатов |
| `asyncio.sleep(delay)` | `sleep(delay, result=None)` | result после delay |
| `asyncio.wait_for(aw, timeout)` | `wait_for(aw, timeout)` | Результат aw |
| `asyncio.wait(aws)` | `wait(aws, *, timeout=None, return_when=ALL_COMPLETED)` | (done, pending) |
| `asyncio.shield(aw)` | `shield(aw)` | Awaitable (результат как у aw) |
| `loop.run_in_executor(executor, func)` | `run_in_executor(executor, func, *args)` | Future (awaitable) |
| `task.cancel()` | `cancel()` | True если отмена запланирована |

#### Резюме по §21 (asyncio)

- **Event loop** — однопоточный диспетчер; переключение только в точках **await**. Запуск: **asyncio.run(main())**.
- **Корутина** не выполняется до передачи в loop (**await**, **create_task**, **run**). **Task** — запланированная корутина; **create_task** только внутри уже работающего loop.
- **Параллельный запуск:** **gather** (все в порядке аргументов), **wait** (done/pending), **as_completed** (по мере готовности). **Таймаут:** **wait_for**; защита от отмены — **shield**. **TaskGroup** (3.11+) — структурированная конкурентность.
- **Синхронизация:** **Lock** (мьютекс), **Semaphore** (ограничение параллелизма), **Queue** (очередь), **Event** (сигнал). Lock в asyncio нужен из‑за переключения в **await** (гонка read–modify–write).
- **Блокирующий код** — **run_in_executor** или **asyncio.to_thread**; для CPU-bound — **ProcessPoolExecutor**.
- **Отмена:** **task.cancel()** → **CancelledError** при следующем await; при перехвате — очистка и **raise**.
- **Streams:** **open_connection** / **start_server**; **StreamReader** / **StreamWriter**. **Subprocess:** **create_subprocess_exec**, **communicate()**.
