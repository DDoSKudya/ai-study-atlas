## §22. threading — многопоточность

**Цель раздела §22:** понимать модель потоков в Python: один процесс, общая память, GIL (глобальная блокировка интерпретатора); создавать потоки через **`threading.Thread`**, синхронизировать доступ через Lock, RLock, Semaphore, Event, Condition, Barrier; использовать потокобезопасную очередь **`queue.Queue`**; понимать роль daemon-потоков и когда threading уместен (I/O-bound задачи).

### 22.1 Основы: Thread, start, join, is_alive

---

**Многопоточность** (threading) — несколько потоков выполнения в **одном процессе**. Потоки разделяют память процесса; переключение между ними управляет ОС (preemptive multitasking). В CPython в любой момент времени только **один** поток выполняет байткод Python (**GIL** — Global Interpreter Lock). GIL освобождается при операциях I/O (чтение/запись файлов, сеть), поэтому для I/O-bound задач потоки дают выигрыш: пока один поток ждёт ответа сети, другой может выполняться. Для чисто CPU-bound задач (много вычислений без I/O) потоки почти не дают прироста производительности — тогда используют **multiprocessing**.

#### GIL (Global Interpreter Lock) — подробнее

**GIL** — глобальная блокировка интерпретатора CPython: в один момент времени только один поток выполняет байткод Python. Это упрощает реализацию (управление памятью, счётчики ссылок) и защищает внутренние структуры от гонок. Следствия: для **чисто вычислительных** задач несколько потоков в одном процессе не ускоряют выполнение (даже замедляют из-за переключений); для **I/O** (файлы, сеть, сон) GIL освобождается на время ожидания — другие потоки могут выполняться. Поэтому threading полезен для I/O-bound, а для CPU-bound нужен **multiprocessing** (отдельные процессы — у каждого свой GIL). **Примечание:** в Python 3.13 доступна экспериментальная **free-threaded** сборка без GIL — потоки могут выполняться параллельно в одном процессе; пока не стандартная сборка.

**Когда GIL освобождается:** при операциях ввода-вывода (чтение/запись файлов, сокетов), при **`time.sleep()`**, при вызовах C-расширений, которые явно отпускают GIL (например, многие операции в NumPy при долгих вычислениях). Пока поток ждёт I/O, другой поток может выполнять Python-код.

**Когда GIL не освобождается:** при выполнении «чистого» Python-байткода (арифметика, циклы, вызовы функций на Python). Поэтому два потока, занятых только вычислениями на Python, по сути выполняются по очереди (с переключением по квантам времени или при I/O).

**Создание и запуск потока:**

```python
import threading

def worker():
    print("Поток выполняется")

t = threading.Thread(target=worker)
t.start()
t.join()
```

**`thread.start()`** — начать выполнение (вызывать один раз); **`thread.join(timeout=None)`** — блокирующее ожидание завершения потока; **`thread.is_alive()`** — выполняется ли поток в данный момент. Именованные потоки: **`threading.Thread(..., name="worker-1")`** — полезно при отладке.

### 22.2 Синхронизация потоков: Lock, RLock, Semaphore, Event, Condition, Barrier

**Цель подраздела:** защищать критические секции (**Lock**, **RLock**), ограничивать параллелизм (**Semaphore**), сигнализировать между потоками (**Event**, **Condition**), синхронизировать фазу (**Barrier**); хранить данные на поток (**threading.local()**).

- **`threading.Lock`** — мьютекс; `with lock:` или `lock.acquire()` / `lock.release()`.
- **`threading.RLock`** — реентерабельный лок: один и тот же поток может захватить его несколько раз (нужно столько же **release**). Удобно, когда одна и та же функция вызывается рекурсивно или из одного потока входит в несколько уровней кода, защищённых одним и тем же локом. Обычный **Lock** при повторном захвате тем же потоком приведёт к deadlock (поток ждёт сам себя).
- **`threading.Semaphore(n)`** — семафор: не более n потоков одновременно в «критической» зоне. **`sem.acquire()`**, **`sem.release()`** или **`with sem:`**.
- **`threading.Event`** — событие: **`event.wait(timeout=None)`** блокирует поток до вызова **`event.set()`**; **`event.clear()`** сбрасывает флаг; **`event.is_set()`** — проверка флага.
- **`threading.Condition`** — условная переменная: поток захватывает связанный Lock и вызывает **`condition.wait()`** — освобождает lock и ждёт; другой поток вызывает **`condition.notify()`** или **`notify_all()`** после изменения условия. После пробуждения lock снова захватывается.
- **`threading.Barrier(n)`** — барьер: n потоков вызывают **`barrier.wait()`**; когда n-й вызовет — все n разблокируются. **`barrier.abort()`** — сбросить барьер (ожидающие получат BrokenBarrierError).
- **`threading.local()`** — thread-local storage: у каждого потока свой экземпляр атрибутов. **`mydata = threading.local()`**; в потоке A: **`mydata.x = 1`**; в потоке B: **`mydata.x`** — другой объект (AttributeError, если не присваивали). Удобно для хранения «контекста» потока (request_id, соединение с БД и т.п.) без явной передачи по аргументам.

#### Пример: Condition — ожидание условия с уведомлением

Поток ждёт, пока другой поток не изменит общее условие и не вызовет **notify()**. Lock связывается с Condition; перед **wait()** и **notify()** lock должен быть захвачен.

```python
import threading

items = []
condition = threading.Condition()

def consumer():
    with condition:
        while not items:
            condition.wait()  # освобождает lock, ждёт; после пробуждения снова захватывает
        item = items.pop()
    print("Получено:", item)

def producer(value):
    with condition:
        items.append(value)
        condition.notify()  # разбудить одного ждущего

t1 = threading.Thread(target=consumer)
t2 = threading.Thread(target=lambda: (t1.start(), producer(42)))
t2.start()
t2.join()
t1.join()
# Получено: 42
```

#### Пример: Barrier — синхронизация N потоков

N потоков вызывают **barrier.wait()**; когда N-й вызовет — все N разблокируются и продолжают выполнение (удобно для «все готовы — старт»).

```python
import threading

def worker(barrier, id):
    print(f"Поток {id}: подготовка")
    barrier.wait()  # ждём, пока все N потоков дойдут сюда
    print(f"Поток {id}: старт")

barrier = threading.Barrier(3)
threads = [threading.Thread(target=worker, args=(barrier, i)) for i in range(3)]
for t in threads:
    t.start()
for t in threads:
    t.join()
# Поток 0: подготовка
# Поток 1: подготовка
# Поток 2: подготовка
# Поток 0: старт
# Поток 1: старт
# Поток 2: старт
```

### 22.3 Очереди: queue.Queue, LifoQueue, PriorityQueue

**Цель подраздела:** использовать потокобезопасную очередь **queue.Queue** (put/get, join/task_done), при необходимости **LifoQueue** или **PriorityQueue**; понимать блокирующее и неблокирующее поведение (timeout, block=False).

- **`queue.Queue`** — потокобезопасная FIFO очередь: **`queue.put(item, block=True, timeout=None)`** — при полной очереди блокируется (или таймаут); **`queue.get(block=True, timeout=None)`** — при пустой блокируется. **`queue.put_nowait(item)`** / **`queue.get_nowait()`** — не блокировать (эквивалент **block=False**); при неудаче — **queue.Full** / **queue.Empty**. **`queue.join()`** — блокировать до тех пор, пока для каждого **put** не был вызван **`queue.task_done()`**. **`queue.qsize()`** — приблизительный размер (не надёжен при высокой конкуренции). Есть **`queue.LifoQueue`**, **`queue.PriorityQueue`** (элементы — пары (приоритет, item) или объекты, сравниваемые по приоритету).

#### Таблица методов queue.Queue

| Метод | Поведение |
|-------|-----------|
| **put(item, block=True, timeout=None)** | Положить элемент; при полной очереди блокируется (или таймаут); при block=False — **queue.Full**. |
| **get(block=True, timeout=None)** | Забрать элемент; при пустой блокируется (или таймаут); при block=False — **queue.Empty**. |
| **put_nowait(item)** | Эквивалент put(..., block=False). |
| **get_nowait()** | Эквивалент get(block=False). |
| **empty()** | True если очередь пуста (приблизительно; не надёжно при конкуренции). |
| **full()** | True если очередь полна (при maxsize>0). |
| **qsize()** | Приблизительное число элементов. |
| **task_done()** | Вызвать после обработки каждого элемента (для join). |
| **join()** | Блокировать до тех пор, пока счётчик «необработанных» не станет 0 (каждый put увеличивает, каждый task_done уменьшает). |

### 22.4 Daemon-потоки

**`thread.daemon = True`** (установить до **`start()`**): такой поток автоматически завершается при выходе из программы главного потока; его не нужно явно **join**. Используются для фоновых задач, которые допустимо оборвать при выходе. Не ждать daemon при **join** — они могут быть прерваны в любой момент; для важной работы используйте обычные потоки и **join**.

### 22.5 Когда использовать threading

**Когда использовать:** I/O-bound задачи (сеть, диск, БД), параллельные HTTP-запросы, чтение файлов; GUI — основной поток для UI, фоновый для вычислений. Для чисто CPU-bound используйте **multiprocessing**.

#### Пример: потокобезопасный счётчик с Lock

```python
import threading

counter = 0
lock = threading.Lock()

def increment():
    global counter
    for _ in range(100_000):
        with lock:
            counter += 1

threads = [threading.Thread(target=increment) for _ in range(5)]
for t in threads:
    t.start()
for t in threads:
    t.join()
print(counter)  # 500_000
```

Без `lock` значение `counter` из-за гонок было бы меньше 500_000.

#### Пример: queue.Queue — воркеры

```python
import threading
import queue

def worker(q):
    while True:
        item = q.get()
        if item is None:
            q.task_done()
            break
        print("Обработка:", item)
        q.task_done()

q = queue.Queue()
threads = [threading.Thread(target=worker, args=(q,)) for _ in range(2)]
for t in threads:
    t.start()
for i in range(5):
    q.put(i)
for _ in threads:
    q.put(None)
q.join()
for t in threads:
    t.join()
```

#### Граничные случаи threading (§22)

- **Забыть join() для не-daemon потока:** главный поток может завершиться раньше воркера; в некоторых средах это допустимо, но для гарантированного завершения работы воркера нужен **join()** (или daemon=True, если обрыв допустим).
- **Захватить Lock и не освободить (исключение между acquire и release):** другой поток зависнет на acquire. Всегда использовать **`with lock:`** или try/finally с release.
- **RLock: один поток захватывает дважды без release:** с обычным Lock — deadlock (поток ждёт сам себя). С RLock — допустимо; нужно столько же release.
- **Queue.get() без таймаута при пустой очереди:** поток блокируется до появления элемента. Для «проверить и выйти» используйте **get(block=False)** или **get(timeout=...)**, обрабатывайте **queue.Empty**.
- **threading.local() в дочернем потоке:** атрибуты, установленные в главном потоке, не видны в дочернем; у каждого потока свой экземпляр атрибутов.

#### Пошагово: что происходит при thread.start() и thread.join()

1. **Thread(target=func, args=(...))** — создаётся объект потока; код **func** ещё не выполняется.
2. **thread.start()** — поток планируется ОС; в другой нити выполнения вызывается **func(*args)**. Главный поток продолжает выполнение без ожидания.
3. **thread.join(timeout=None)** — главный поток блокируется до завершения **func** (или до истечения **timeout**). После **join()** поток завершён, повторный **start()** вызвать нельзя.
4. **thread.is_alive()** — True, пока **func** выполняется; False после завершения.

**Запомните:** один вызов **start()** на один объект потока; повторный вызов — **RuntimeError**. Для повторного выполнения создайте новый **Thread**.

#### Таблица методов и атрибутов Thread

| Метод / атрибут | Назначение |
|-----------------|------------|
| `thread.start()` | Запустить поток (один раз). |
| `thread.join(timeout=None)` | Блокировать до завершения потока (или timeout). |
| `thread.is_alive()` | Выполняется ли поток в данный момент. |
| `thread.name` | Имя потока (для отладки); можно задать в конструкторе. |
| `thread.daemon` | True — поток завершится при выходе главного; задавать до **start()**. |
| `threading.current_thread()` | Текущий поток в данном контексте. |
| `threading.get_ident()` | Числовой идентификатор текущего потока. |

#### Плохо vs лучше (threading)

| Плохо | Лучше |
|-------|--------|
| Общая переменная без блокировки (инкремент из нескольких потоков) | **Lock** вокруг критической секции или **queue.Queue** для обмена. |
| Захватить Lock и не освободить при исключении | **`with lock:`** или try/finally с **lock.release()**. |
| Долгая работа в daemon без учёта обрыва | Либо non-daemon + **join()**, либо короткие задачи и допустимость обрыва. |
| Разный порядок захвата двух блокировок в разных потоках | Единый порядок (сначала A, потом B) во всех потоках. |
| Один поток захватывает один и тот же Lock дважды (без release) | **RLock** (реентерабельный лок) или пересмотреть логику. |

#### Резюме по §22 (threading)

- **Потоки** разделяют память процесса; в CPython в один момент выполняется один поток (GIL). GIL освобождается при I/O — threading полезен для I/O-bound.
- **Создание:** **Thread(target=..., args=...)**, **start()**, **join()**. **daemon=True** — поток не переживёт главный.
- **Синхронизация:** **Lock**, **RLock**, **Semaphore**, **Event**, **Condition**, **Barrier**; **threading.local()** — данные на поток.
- **Очередь:** **queue.Queue** (put/get, join/task_done); потокобезопасна. **list** и **dict** без блокировки — не потокобезопасны.
- **Когда использовать:** I/O-bound (сеть, диск, БД); GUI (основной поток — UI, фоновый — работа). CPU-bound — **multiprocessing**.
