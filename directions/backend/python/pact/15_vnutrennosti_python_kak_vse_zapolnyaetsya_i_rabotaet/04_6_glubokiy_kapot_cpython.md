### 6.1. AST, `compile`, `eval`/`exec` и полный путь «код → байткод → исполнение» (89a, 89a1)

#### 6.1.0. Общая картинка конвейера

Когда вы пишете:

```python
print(1 + 2)
```

Python не исполняет эту строку напрямую. Он проходит несколько этапов:

1. **Лексер**: разбивает текст на **токены**:
   - `NAME` (`print`), `NUMBER` (`1`, `2`), `OP` (`+`, `(`, `)`), переносы строк и т.д.
2. **Парсер**: из токенов строит **AST (Abstract Syntax Tree)** — абстрактное синтаксическое дерево.
3. **Компилятор**: по AST строит **байткод**:
   - последовательность инструкций VM (`LOAD_GLOBAL`, `LOAD_CONST`, `BINARY_ADD`, `CALL_FUNCTION`, ...).
4. **Виртуальная машина (eval loop)**:
   - читает байткод по одной инструкции;
   - работает со стеком значений;
   - вызывает нужные функции/операции.

Всё это делается **автоматически** при импортировании модуля или запуске скрипта.

Наша цель — уметь мысленно пройти этот конвейер и не пугаться слов «AST» и «байткод».

---

#### 6.1.1. От исходника к AST: модуль `ast`

Модуль `ast` позволяет увидеть тот самый шаг «текст → дерево».

```python
import ast

source = "x = 1 + 2"
tree = ast.parse(source)
print(ast.dump(tree, indent=2))
```

Примерный результат (упрощённо):

```text
Module(
  body=[
    Assign(
      targets=[Name(id='x', ctx=Store())],
      value=BinOp(
        left=Constant(value=1),
        op=Add(),
        right=Constant(value=2)
      )
    )
  ],
  type_ignores=[]
)
```

Разберём:

- `Module` — корень дерева, соответствует целому файлу/строке кода.
- `body` — список инструкций в модуле.
- `Assign` — узел присваивания (`x = ...`):
  - `targets` — список «левых частей» (у нас одна: `Name('x')`);
  - `value` — правая часть (`BinOp(...)`).
- `BinOp` — бинарная операция (`1 + 2`):
  - `left` — левый операнд (`Constant(1)`);
  - `op` — операция (`Add` — сложение);
  - `right` — правый операнд (`Constant(2)`).

То есть Python прежде, чем что‑то выполнять, **строит такое дерево**.

`ast.NodeVisitor` и `ast.NodeTransformer`:

- позволяют **обходить** это дерево (смотреть на каждую ноду);
- и **изменять** его (переписывать код на уровне структуры), а затем
  компилировать обратно в байткод.

На этом основаны:

- линтеры и форматтеры (`ruff`, плагины к `flake8`, `black`, `pyupgrade`);
- инструменты автоматического рефакторинга (`autoflake`, автогенераторы кода).

#### 6.1.2. `compile()` и режимы `eval`/`exec`/`single`

Функция `compile(source, filename, mode)` — это «ручной вход» в компилятор Python.

```python
code = compile("1 + 2", "<string>", mode="eval")
print(type(code))  # <class 'code'>
print(eval(code))  # 3
```

Параметры:

- `source` — строка с исходным кодом (или уже AST);
- `filename` — имя файла для сообщений об ошибках (можно фиктивное: `"<string>"`);
- `mode`:
  - `"eval"` — **одно выражение**, результат можно вычислить через `eval(code)`;
  - `"exec"` — **произвольный блок кода** (несколько строк, определения функций и т.п.);
  - `"single"` — режим интерактивной строки (как REPL).

Функции, работающие с результатом:

- `eval(expr, globals=None, locals=None)`:
  - принимает строку **или** объект кода;
  - вычисляет и **возвращает результат выражения**.
- `exec(code, globals=None, locals=None)`:
  - принимает строку **или** объект кода;
  - **выполняет инструкции**, но **ничего не возвращает** (возвращает `None`).

Иногда удобно разделить:

- `compile` — один раз, например, при загрузке конфига или шаблона;
- `eval`/`exec` — многократно, если нужно повторно исполнять один и тот же код.

---

#### 6.1.3. Почему `eval`/`exec` опасны и когда использовать `ast.literal_eval`

Ключевая мысль:

> `eval` и `exec` выполняют **произвольный Python‑код**, который вы им даёте.

То есть:

- если вы формируете строку для `eval` из **пользовательского ввода**:

  ```python
  user_input = input("Введите выражение: ")
  result = eval(user_input)  # ОЧЕНЬ опасно
  ```

  пользователь может ввести `__import__("os").system("rm -rf /")` и т.п.;

- это типичная уязвимость класса **RCE (Remote Code Execution)**.

В большинстве реальных задач нужно **только разобрать данные**, а не выполнять код.
Для этого есть `ast.literal_eval`:

- принимает строку;
- разбирает только **литералы Python**:
  - числа, строки;
  - списки, словари, кортежи, множества **из этих литералов**;
- выбрасывает ошибку, если видит что‑то «исполняемое» (вызовы функций, атрибуты и т.п.).

```python
import ast

safe = ast.literal_eval("[1, 2, 3]")
print(safe)  # [1, 2, 3]
```

Когда использовать `literal_eval`:

- вам приходит конфиг/данные вида `"[1, 2, 3]"`, `"{'a': 1}"`, `"('x', 'y')"`;
- вы хотите превратить строку в объекты Python **без выполнения произвольного кода**.

#### 6.1.4. Краткий конвейер компиляции в CPython (89a1)

Соберём всё в одну цепочку для одного модуля:

1. **Исходный текст** (`.py`) → **токены**:
   - лексер (`tokenize`) идёт по строкам и разбивает их на токены.
2. **Токены** → **AST**:
   - парсер строит синтаксическое дерево `ast.Module`, `ast.FunctionDef`, `ast.Assign`, ...
3. **AST** → **байткод**:
   - компилятор обходит дерево и генерирует для нод соответствующие опкоды.
4. **Байткод** → **исполнение**:
   - eval loop в VM шагает по байткоду;
   - в CPython 3.11+ используется **адаптивный интерпретатор**:
     - инструкции специализируются под часто встречающиеся типы (inline‑кеши).
5. **Байткод кешируется**:
   - может быть записан в `.pyc` в каталоге `__pycache__`;
   - при следующем запуске модуль загрузится быстрее (см. раздел 4.2.3).

Для разработчика важно:

- понимать, что часть «магии» (например, константные выражения, оптимизация `LOAD_ATTR`)
  происходит **на этапе компиляции**;
- некоторые странности (порядок вычисления списковых включений, захват переменных
  в замыканиях) связаны с тем, **как именно компилятор строит AST и байткод**.

### 6.2. Структура объектов в CPython: `PyObject`, `PyTypeObject`, `PyVarObject` (89a2)

Здесь мы на секунду представим себя не Python‑разработчиками, а разработчиками CPython на C.

Наша задача:

- понять, как интерпретатор хранит объекты в памяти;
- не выучить все поля, а уловить **общую идею**:
  - у любого объекта есть «шапка» (ссылки, тип);
  - у типов есть таблицы методов;
  - есть специальные структуры для объектов переменной длины.

---

#### 6.2.1. `PyObject`: заголовок каждого объекта

Почти каждый объект в CPython начинается с **C‑структуры `PyObject`**. В ней:

- `ob_refcnt` — **счётчик ссылок** (то, о чём мы говорили в 4.1.2);
- `ob_type` — указатель на **тип объекта** (`PyTypeObject`).

В терминах C это выглядит примерно так (упрощённо):

```c
typedef struct _object {
    Py_ssize_t ob_refcnt;   // число ссылок
    struct _typeobject *ob_type;  // указатель на тип
} PyObject;
```

Как это связано с Python‑миром:

- у каждого `int`, `list`, `dict`, вашего пользовательского объекта **есть такой заголовок**;
- `ob_type` указывает на объект типа (например, `&PyLong_Type` для целых, `&PyList_Type` для списков и т.п.);
- типы сами тоже являются объектами (имеют свой `ob_refcnt`, `ob_type`, и т.д.).

Типы (`int`, `list`, ваши классы) представлены структурой `PyTypeObject`:

- в ней есть:
  - `tp_name` — имя типа;
  - указатели на C‑функции для:
    - хеширования (`tp_hash`);
    - вызова (`tp_call`);
    - доступа к атрибутам (`tp_getattro`) и т.п.
- тип типов — это `type` (`PyType_Type`).

Практическая связь:

- `isinstance(x, MyClass)` в итоге сравнивает `ob_type` и ходит по MRO;
- выражение `x.__class__` в Python — это обёртка над `ob_type`;
- многие методы (`__hash__`, `__call__`, `__getattr__`) в конечном счёте переходят
  в вызовы функций из `PyTypeObject`.

---

#### 6.2.2. `PyVarObject`: объекты переменной длины

Некоторые объекты могут быть **разной длины**:

- списки (`list`) — разное количество элементов;
- кортежи (`tuple`);
- строки (`str`);
- словари (`dict`), множества (`set`) и т.д.

Для них используется расширение `PyVarObject`:

```c
typedef struct {
    PyObject ob_base;   // тот самый PyObject (refcnt + type)
    Py_ssize_t ob_size; // "логический размер" - количество элементов
} PyVarObject;
```

Идея:

- `PyVarObject` добавляет поле `ob_size`, чтобы один и тот же тип
  мог описывать объекты длиной 0, 1, 10, 100 и т.д.;
- сами данные (элементы списка, кортежа и т.п.) располагаются **после заголовка**
  в памяти.

Это позволяет:

- эффективно хранить массивы переменной длины;
- использовать один и тот же код работы с ними, зная только `ob_size`.

#### 6.2.3. Компактный dict, over‑allocation в list, интернирование строк, кеш малых чисел

Здесь чуть больше практических деталей, которые полезно знать «на уровне интуиции».

##### Компактный `dict`

`dict` в CPython реализован довольно хитро, но на верхнем уровне:

- используется **разделение на два массива**:
  - один (sparse) хранит информацию о **хешах и индексах**;
  - второй (dense) хранит сами **записи (key, value, hash)**.

Это даёт:

- экономию памяти по сравнению с «наивной» хеш‑таблицей;
- возможность **сохранять порядок вставки ключей** без дополнительных структур.

Порядок ключей (`dict` сохраняет порядок с 3.7+) достигается именно благодаря такому дизайну.

##### Over‑allocation в `list`

Списки (`list`) используют стратегию **перевыделения с запасом**:

- когда вы делаете много `append`, Python:
  - не увеличивает массив **по одному элементу**;
  - вместо этого иногда выделяет **новый, заметно больший блок памяти**,
    копирует туда старые элементы и продолжает работу.

Итог:

- отдельная операция `append` иногда стоит дорого (нужно копировать элементы) → O(n);
- но в среднем, если смотреть на **много `append` подряд**, стоимость размазывается:
  - получается **амортизированная O(1)** на вставку в конец.

Это тот самый механизм, который мы описывали в части об алгоритмах и структурах данных.

##### Интернирование строк

CPython может **интернировать** некоторые строки:

- автоматически:
  - идентификаторы (имена переменных, атрибутов);
  - некоторые короткие ASCII‑строки;
- вручную:

  ```python
  import sys

  s1 = sys.intern("hello world")
  s2 = sys.intern("hello world")
  print(s1 is s2)  # True
  ```

Зачем:

- экономия памяти при большом числе одинаковых строк;
- ускорение сравнений (`is` быстрее, чем посимвольное сравнение).

Важно:

- не стоит строить логику программы на сравнении строк через `is`;
- интернирование — **деталь оптимизации**, а не контракт API.

##### Кеш малых целых

Целые числа (`int`) в Python — объекты, и их создание стоит чего‑то.  
Чтобы ускорить часто используемые значения, CPython кеширует небольшой диапазон:

- обычно от `-5` до `256` (может меняться между версиями);
- при создании таких чисел они берутся из **готового кеша объектов**.

Следствие:

```python
>>> a = 10
>>> b = 10
>>> a is b
True  # часто (из-за кеша)
```

Но:

- это **не гарантируется** спецификацией языка;
- нельзя опираться на `is` для сравнения чисел, всегда используйте `==`.

### 6.3. Code objects и frame objects: как устроены функции под капотом (89a2a)

На уровне Python мы видим:

- функции (`def f(...): ...`);
- вызовы (`f(1, 2)`);
- стектрейсы при ошибках.

Под капотом у этого есть два ключевых участника:

- **объект кода (code object)** — «статическое описание» функции;
- **кадры (frame objects)** — «живые экземпляры выполнения» функции.

---

#### 6.3.1. Code object: «паспорт» функции

У любой функции в CPython есть атрибут `__code__`, который ссылается на объект типа `CodeType`:

```python
import inspect

def f(x: int, y: int) -> int:
    z = x + y
    return z

code = f.__code__
print(code.co_name)        # 'f'
print(code.co_varnames)    # ('x', 'y', 'z')
print(code.co_argcount)    # 2
```

`code` хранит:

- имя функции (`co_name`);
- **байткод** (`co_code`) — последовательность опкодов;
- кортеж констант (`co_consts`);
- имена локальных переменных (`co_varnames`);
- информацию о количестве аргументов (`co_argcount`, `co_kwonlyargcount`, ...);
- имена переменных из внешних областей (`co_freevars`) и т.д.

Можно думать так:

> Code object — это **не выполняемая программа**, а её «сжатое описание»:
> байткод + метаданные.

Один и тот же `code` используется сколько угодно раз:

- при каждом вызове функции создаётся **новый кадр (frame)**,
  но `f.__code__` остаётся тем же самым объектом.

---

#### 6.3.2. Frame object: «живой» вызов функции

Каждый раз, когда вы вызываете функцию, Python создаёт **кадр выполнения**:

- в нём есть:
  - ссылка на `f.__code__` (что именно выполняем);
  - локальные переменные (`frame.f_locals`);
  - глобальные переменные (`frame.f_globals`);
  - стек операндов VM;
  - ссылка на предыдущий кадр (`frame.f_back`).

Небольшой пример:

```python
import inspect

def debug_frame():
    frame = inspect.currentframe()
    print("Function:", frame.f_code.co_name)
    print("Locals:", frame.f_locals)

def wrapper():
    x = 10
    y = 20
    debug_frame()

wrapper()
```

Вывод покажет:

- `frame.f_code.co_name` — `"debug_frame"`;
- `frame.f_locals` — словарь локальных переменных в `debug_frame` (в нашем случае пустой или почти пустой);
- у этого кадра `f_back` — кадр для `wrapper`:
  - в нём `f_locals` будут содержать `x = 10`, `y = 20`.

**Отладчики и профилировщики** (pdb, `cProfile`, инструменты IDE) работают именно с:

- цепочкой `frame.f_back` (чтобы восстановить стек вызовов);
- `frame.f_code` (чтобы показать вам имя функции, файл и строку);
- `frame.f_locals`/`f_globals` (чтобы показать текущие переменные).

Понимание пары `code object` + `frame`:

- помогает читать и понимать стектрейсы;
- даёт основу для написания **своих** отладочных/трассировочных инструментов (если понадобится).

### 6.4. Эволюция GIL и многопоточность в CPython (89a3)

#### 6.4.0. Интуитивная картинка

Подумайте о GIL как о **единственном ключе от кухни** в ресторане:

- есть много поваров (потоки);
- но на кухню одновременно можно пустить **только одного**, потому что:
  - кухня маленькая;
  - там много общих ресурсов (ножи, плиты, продукты).

GIL (Global Interpreter Lock) — это:

- **глобальная блокировка интерпретатора CPython**;
- которая гарантирует, что **одновременно байткод выполняет только один поток**.

Это упрощает внутреннюю реализацию (не нужно сложное взаимодействие потоков внутри VM),  
но накладывает ограничения на то, **как мы можем использовать потоки в Python**.

---

#### 6.4.1. Классический GIL: один поток байткода за раз

В классической модели CPython (до недавних новшеств):

- только **один поток** в каждый момент времени:
  - держит GIL;
  - исполняет байткод Python.
- остальные потоки:
  - либо **ждут**, когда им отдадут GIL;
  - либо делают что‑то, что **не требует GIL** (обычно I/O в C‑библиотеках).

Последствия:

- для **CPU‑ограниченных задач** (много чистых вычислений в Python‑циклах):
  - добавление потоков **не ускоряет** программу (иногда даже замедляет);
  - потому что потоки всё равно исполняют байткод по очереди.
- для **I/O‑ограниченных задач** (сеть, файлы, ожидание ответа БД):
  - потоки **полезны**:
    - когда один поток ждёт сети/диска, GIL может перейти к другому потоку;
    - таким образом, процессор не простаивает.

CPython периодически «отдаёт» GIL:

- при операциях ввода‑вывода (вызовы системных функций, которые могут долго ждать);
- по внутреннему счётчику/таймеру (раньше — через N байткодов, сейчас — более сложный механизм).

Практический вывод:

- если нужно разогнать **численные вычисления**:
  - используем **процессы** (`multiprocessing`, `joblib`) или библиотеки на C/NumPy;
  - или выносим тяжёлые части в C‑расширения, которые сами могут освобождать GIL;
- если много **сетевых запросов, ожиданий, файловых операций**:
  - потоки/`asyncio` отлично помогают скрыть задержки I/O.

---

#### 6.4.2. Per‑interpreter GIL (PEP 684, Python 3.12)

Начиная с Python 3.12, в CPython появились **суб‑интерпретаторы** с отдельными GIL:

- можно создать несколько независимых интерпретаторов внутри одного процесса;
- у каждого свой:
  - набор модулей, глобальных переменных;
  - **свой GIL**.

Что это даёт:

- можно лучше нагружать многоядерные CPU:
  - разные суб‑интерпретаторы могут реально параллелить работу;
- улучшенная изоляция:
  - меньше шансов, что глобальные переменные «протекут» между изолированными задачами.

Ограничения:

- Python‑уровневый API для суб‑интерпретаторов пока довольно сырой и ограниченный;
- большинство библиотек всё ещё ориентировано на «один интерпретатор + классический GIL».

На практике сейчас это чаще интересует:

- авторов серверов/рантаймов (embed Python в приложения);
- разработчиков самих библиотек/инструментов.

---

#### 6.4.3. Free‑threaded Python (PEP 703, Python 3.13+ экспериментально)

PEP 703 предлагает **убрать GIL вообще** или сделать возможным сборку Python без GIL:

- вместо одной глобальной блокировки:
  - у каждого объекта/структуры будут свои **тонкие блокировки**;
  - C‑расширения должны научиться работать в такой среде.

На момент написания:

- есть **экспериментальный free‑threaded build** Python;
- экосистема (C‑расширения, библиотеки) постепенно адаптируется;
- поведение и производительность ещё активно исследуются.

Что это значит для вас:

- сейчас (Python 3.10–3.12) безопасно считать, что работает **классический GIL**:
  - потоки не ускоряют CPU‑тяжёлый Python‑код;
  - процессы и C/NumPy — основной путь для параллелизма в вычислениях;
- в перспективе нескольких версий:
  - появится возможность писать **по‑настоящему многопоточные** Python‑программы без ограничений GIL;
  - но нужно будет внимательно следить за совместимостью библиотек.

---

Если собрать блок 6 в одно предложение:

> CPython берёт ваш исходный код, превращает его в AST и байткод,
> хранит объекты как C‑структуры с заголовками `PyObject`/`PyVarObject`,
> исполняет байткод через пары `code object` + `frame`,
> а параллелизм контролирует через механизм GIL (который сейчас эволюционирует
> от одного глобального lock к более тонким и гибким моделям).
