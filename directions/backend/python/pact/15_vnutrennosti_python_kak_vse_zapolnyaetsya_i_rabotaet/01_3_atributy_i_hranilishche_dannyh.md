### 3.1. Как Python ищет атрибуты: от экземпляра до метакласса (§85)

#### 3.1.0. Интуитивная картинка

Представьте, что у вас есть **человек** (экземпляр) и **его профессия** (класс).

- У человека есть **личные вещи** — паспорт, телефон, ключи. Это как **атрибуты экземпляра**.
- У профессии есть **общие свойства** — «врач умеет лечить», «программист умеет писать код».
  Это как **атрибуты класса**.

Когда вы спрашиваете:

> «Что умеет вот этот конкретный программист Вася?»

вы:

1. Сначала смотрите на **личные особенности Васи** (знает Go, говорит по‑испански).
2. Если нужного навыка там нет — смотрите, что умеют **вообще программисты как класс**.

Ровно так же Python ищет `obj.attr`:

1. **Сначала** смотрит в **личный словарь объекта** (`obj.__dict__`).
2. Потом в **класс** объекта (`type(obj)`).
3. Потом в **родительские классы** по особому порядку (MRO).
4. Если всё ещё не нашёл — даёт шанс методу `__getattr__`.

Если коротко:

> Python всегда идёт примерно по цепочке:  
> **экземпляр → класс → базовые классы → `__getattr__` → ошибка**.

Теперь разберём это медленно и по шагам.

---

#### 3.1.1. Базовый сценарий: экземпляр → класс → базовые классы

Начнём с максимально простого примера, без магии:

```python
class User:
    role = "user"   # атрибут класса (общий для всех)

    def __init__(self, name):
        self.name = name  # атрибут экземпляра (личный)

u = User("Alice")
```

Что у нас есть:

- `User` — **класс**.
- `u` — **экземпляр** этого класса.
- `User.role` — **одна строка**, которая лежит внутри класса и **общая для всех пользователей**.
- `u.name` — **конкретное имя**, записанное только у этого экземпляра.

Посмотрим, что реально хранится:

```python
print(u.__dict__)      # {'name': 'Alice'}
print(User.__dict__['role'])  # 'user'
```

То есть:

- `u.__dict__` — словарь с **личными полями экземпляра**.
- `User.__dict__` — словарь с **тем, что объявлено в теле класса**.

Теперь самое важное: **как вычисляется `u.name`**.

1. Вы пишете `u.name`.
2. Python под капотом вызывает функцию:
   - `type(u).__getattribute__(u, "name")`.
   - В нашем случае `type(u) is User`, а реализация берётся из `object.__getattribute__`.
3. Эта базовая реализация делает примерно следующее (упрощённый псевдокод):

```python
def object__getattribute__(obj, name):
    # 1. Сначала ищем в личном словаре объекта
    if name in obj.__dict__:
        return obj.__dict__[name]

    # 2. Потом ищем в классе и по MRO
    cls = type(obj)
    # (упрощённо) поиск в cls и его базах
    value = find_in_class_and_bases(cls, name)
    if value is not found:
        # 3. Если ничего не нашли, пробуем __getattr__
        getattr_method = find_in_class_and_bases(cls, "__getattr__")
        if getattr_method is not found:
            raise AttributeError(name)
        return getattr_method(obj, name)

    return value
```

В реальности код сложнее (учитываются дескрипторы, `__getattribute__` у классов и т.д.),
но **идея остаётся такой же**:

> «Сначала проверяю личные данные объекта, потом «паспорт» класса, потом вызываю `__getattr__`».

Пример с `__getattr__` (чтобы прочувствовать шаг 3):

```python
class Dynamic:
    def __init__(self, data: dict):
        self._data = data

    def __getattr__(self, name: str):
        # Сюда мы попадаем ТОЛЬКО если обычный поиск не нашёл атрибут
        print(f"__getattr__ вызван для {name!r}")
        try:
            return self._data[name]
        except KeyError as e:
            raise AttributeError(name) from e

user = Dynamic({"name": "Alice"})

print(user.name)  # выведет: __getattr__ вызван для 'name'  → 'Alice'
print(user.age)   # __getattr__ вызван для 'age'  → AttributeError
```

Разберём по шагам `user.name`:

1. `user.__dict__` не содержит `name` (там только `_data`).
2. В классе `Dynamic` тоже нет атрибута `name`.
3. Но в `Dynamic` есть метод `__getattr__`, поэтому Python вызывает его.
4. `__getattr__` смотрит в `_data` и возвращает значение.

**Важно запомнить:**

- `__getattr__` — это **последний шанс** перед ошибкой, он *не* заменяет собой обычный поиск.
- Если вы хотите «подмешаться» **в любой доступ** к атрибуту — это уже про `__getattribute__`
  (см. ниже), а не про `__getattr__`.

#### 3.1.2. MRO: как выбирается метод при множественном наследовании

Когда у класса **несколько родителей**, нужно решить:

> «В каком порядке обходить базовые классы в поисках метода/атрибута?»

Этот порядок как раз и называется **MRO (Method Resolution Order)**.

Очень простой пример:

```python
class A:
    def who(self): return "A"

class B(A):
    def who(self): return "B"

class C(A):
    def who(self): return "C"

class D(B, C):
    pass

print(D.mro())
print(D().who())
```

Примерный вывод:

```python
[<class '__main__.D'>,
 <class '__main__.B'>,
 <class '__main__.C'>,
 <class '__main__.A'>,
 <class 'object'>]
B
```

Читаем:

- Python будет искать метод `who` в таком порядке:
  - `D` → `B` → `C` → `A` → `object`.
- В `D` метода `who` нет, в `B` есть → он и используется.

Если поменять местами родителей:

```python
class D2(C, B):
    pass

print(D2.mro())
print(D2().who())
```

порядок поиска и результат изменятся:

- MRO станет `D2, C, B, A, object`;
- первым будет найден метод из `C`.

**Практическая мысль:**

- при множественном наследовании **порядок базовых классов в объявлении очень важен**;
- если вы пишете класс вроде:

```python
class MyView(LoginRequiredMixin, SomeFrameworkBaseClass):
    ...
```

то порядок `LoginRequiredMixin` и `SomeFrameworkBaseClass` определяет,
**в каком порядке будут применяться методы инициализации, `dispatch` и т.п.**

Если коротко:

> MRO — это просто **список классов**, по которому Python идёт «сверху вниз»
> в поисках нужного атрибута.

#### 3.1.3. Дескрипторы: свойства, методы и «перехват» доступа

Давайте начнём с интуиции.

Хотим, чтобы:

- при чтении атрибута `user.age` он **автоматически проверялся** (например, неотрицательный);
- при записи `user.age = value` значение **автоматически нормализовалось** (например, переводилось в `int`);
- при доступе к `obj.method` нам возвращался **метод с привязанным `self`**, а не «голая функция».

Всё это делает **механизм дескрипторов**.

Официальное определение:

> **Дескриптор** — объект, у которого определён метод `__get__`, `__set__` или `__delete__`.

Примеры того, что *уже* является дескрипторами в стандартной библиотеке:

- `property` (через `@property`);
- функции, объявленные в теле класса (они превращаются в методы);
- `@classmethod`, `@staticmethod`;
- поля в `dataclasses`, поля моделей в Django ORM и т.д.

##### Как это работает в голове

Когда Python находит в **классе** атрибут, который является дескриптором, он **не возвращает
сам этот объект**, а вместо этого вызывает его методы:

- при чтении: `descriptor.__get__(instance, owner)`;
- при записи: `descriptor.__set__(instance, value)`.

Получается «крючок»:

- вы кладёте в класс **«умный объект‑дескриптор»**;
- Python при обращении к атрибуту этого имени **делегирует поведение этому объекту**.

##### Свой простой дескриптор

```python
class UpperCaseDescriptor:
    def __init__(self, name: str):
        # Имя, под которым будем хранить значение в __dict__ экземпляра
        self._name = name

    def __get__(self, instance, owner):
        # instance — это сам объект (например, Person)
        # owner   — это класс (например, Person)
        if instance is None:
            # Обращение через класс: Person.name → вернуть сам дескриптор
            return self
        value = instance.__dict__.get(self._name, "")
        return value.upper()

    def __set__(self, instance, value):
        # При записи p.name = value мы сами решаем, как хранить значение
        instance.__dict__[self._name] = str(value)


class Person:
    name = UpperCaseDescriptor("_name")

    def __init__(self, name):
        self.name = name

p = Person("alice")
print(p.name)  # 'ALICE'
```

Что здесь происходит пошагово:

1. При определении класса `Person` в его `__dict__` помещается:
   - ключ `'name'`;
   - значение — **экземпляр `UpperCaseDescriptor`**.
2. Когда вы пишете `p.name`, Python:
   1. находит в классе `Person` атрибут `'name'`;
   2. видит, что это объект с методом `__get__`;
   3. вместо простого возврата значения вызывает:
      - `UpperCaseDescriptor.__get__(p, Person)`.
3. Внутри `__get__` мы:
   - достаём «сырое» значение из `p.__dict__['_name']`;
   - преобразуем к строке верхнего регистра;
   - возвращаем.

Ключевая идея:

> Дескриптор позволяет **захватить момент чтения/записи атрибута**
> и выполнить в этот момент произвольный код.

#### 3.1.4. `__getattribute__` vs `__getattr__`: ещё раз медленно

Эти два метода легко перепутать, поэтому повторим их различие несколько раз.

- **`__getattribute__(self, name)`**
  - вызывается **при КАЖДОМ** обращении к атрибуту (`self.x`, `self.y`, даже `self.__dict__`);
  - это как «главный вход» в механизм доступа к атрибутам;
  - если вы его переопределяете, вы берёте на себя **всю ответственность** за поиск атрибутов.

- **`__getattr__(self, name)`**
  - вызывается **ТОЛЬКО если обычный поиск ничего не нашёл**;
  - это как «запасной выход»:
    - «если везде искали и не нашли, дай человеку последний шанс что‑то вернуть».

Классический пример с `__getattr__` (повторим):

```python
class Proxy:
    def __init__(self, target):
        self._target = target

    def __getattr__(self, name):
        # Сюда мы попадаем ТОЛЬКО если атрибут не найден в __dict__ и в классе
        return getattr(self._target, name)
```

Тут:

- `proxy.something` сначала ищется у `proxy` (экземпляр), потом в классе `Proxy`;
- если не нашли — вызывается `__getattr__`, который просто пробрасывает запрос на `_target`.

Теперь пример с `__getattribute__`:

```python
class Verbose:
    def __getattribute__(self, name):
        print("Accessing", name)
        # ВАЖНО: использовать реализацию базового класса, иначе рекурсия
        return object.__getattribute__(self, name)
```

Если внутри `__getattribute__` вы напишете `self.name`, то:

1. это опять вызовет `__getattribute__`;
2. который опять вызовет `self.name`;
3. и так бесконечно, пока не будет переполнен стек.

Поэтому внутри `__getattribute__` почти всегда используют именно:

```python
object.__getattribute__(self, name)
```

**Ещё раз в одно предложение:**

- `__getattribute__` — «ловит всё», им пользоваться надо очень аккуратно;
- `__getattr__` — вызывается «на крайний случай» и дружит с обычным поиском.

#### 3.1.5. Полный порядок поиска атрибутов (упрощённо, но по шагам)

Давайте соберём всё вместе для выражения:

```python
obj.attr
```

и представим **идеальный учебный порядок**:

1. Python вызывает: **`type(obj).__getattribute__(obj, 'attr')`**.
2. Базовая реализация внутри делает примерно следующее (с учётом дескрипторов):

   1. **Сначала ищем data‑дескрипторы в классе и его базах.**
      - Data‑дескриптор — объект, у которого есть `__get__` **и** `__set__` (или `__delete__`).
      - Примеры: свойства с setter’ом (`@property` + `.setter`), многие поля ORM.
      - Если нашли такой дескриптор — вызываем его `__get__` и **возвращаем результат**.

   2. **Если data‑дескриптор не найден — смотрим в `obj.__dict__`.**
      - Если там есть ключ `'attr'` — возвращаем его значение.

   3. **Если в `__dict__` ничего нет — ищем в классе и по MRO.**
      - Здесь могут встретиться:
        - обычные атрибуты (значения, функции, классы);
        - non‑data‑дескрипторы (объекты с одним `__get__`, но без `__set__`).

   4. **Если найдён non‑data‑дескриптор — вызываем его `__get__`.**

   5. **Если найден обычный атрибут — возвращаем как есть.**

3. Если в классе и по MRO ничего не нашли:

   - ищем метод `__getattr__` в классе и его базах;
   - если он есть — вызываем `__getattr__(obj, 'attr')`.

4. Если и `__getattr__` не нашёл/не вернул значение — бросаем `AttributeError`.

Если совсем коротко, для запоминания:

> **data‑дескрипторы → `obj.__dict__` → класс и MRO → `__getattr__` → ошибка**.

### 3.2. Где лежат данные: `__dict__` и `__slots__` (§86)

Теперь, когда мы понимаем **как ищутся атрибуты**, давайте посмотрим,
**где вообще хранятся данные**.

#### 3.2.1. `__dict__`: объект как обычный словарь атрибутов

По умолчанию любой ваш класс выглядит так:

```python
class User:
    pass

u = User()
u.name = "Alice"
u.age = 30

print(u.__dict__)  # {'name': 'Alice', 'age': 30}
```

То есть:

- у каждого экземпляра есть **словарь `__dict__`**;
- ключи — **строки‑имена атрибутов** (`"name"`, `"age"`);
- значения — сами объекты (`"Alice"`, `30`).

Можно думать так:

> Экземпляр класса — это **коробка**, в которой есть:
>
> - служебная информация (тип, счётчик ссылок и т.п.);
> - плюс отдельный «карман»‑словарь `__dict__`, где лежат все его поля.

У класса тоже есть свой `__dict__`:

```python
print(User.__dict__)         # mappingproxy({...})
print(dict(User.__dict__))   # превратить в обычный dict для удобного просмотра
```

Отличие:

- `User.__dict__` — это не настоящий dict, а **`mappingproxy`** — «чтение‑только» представление;
- это сделано, чтобы **нельзя было случайно сломать** внутреннюю структуру класса напрямую.

#### 3.2.2. Почему `__dict__` дорог по памяти

У каждого объекта с `__dict__`:

- хранятся **строки‑ключи** (имена атрибутов);
- сама структура словаря — это **хеш‑таблица** (ячейки, хеши, коллизии и т.д.);
- на каждый экземпляр приходится **дополнительный объект dict** плюс накладные расходы.

Если у вас:

- 10–100 объектов — это почти неважно;
- 100 000–1 000 000 объектов — накладные расходы становятся заметными.

Именно для таких случаев придумали `__slots__`.

---

#### 3.2.3. `__slots__`: фиксированные «ячейки» вместо `__dict__`

Идея:

> Вместо «мешка ключей и значений» (dict) мы заранее говорим:
>
> «В этом типе объектов будут только вот такие поля:
> `x`, `y`, `z`. Больше никаких».

Тогда Python может:

- не создавать `__dict__` у каждого экземпляра;
- вместо этого завести внутри объекта **фиксированный набор «слотов»** под эти имена.

Пример:

```python
class Point:
    __slots__ = ("x", "y")  # говорим: у Point будут только x и y

    def __init__(self, x, y):
        self.x = x
        self.y = y

p = Point(1, 2)
print(p.x, p.y)

print(hasattr(p, "__dict__"))  # False — словаря нет
```

Если попытаться добавить новый атрибут:

```python
p.z = 3  # AttributeError: 'Point' object has no attribute 'z'
```

**Что мы выиграли:**

- **меньше памяти**:
  - нет отдельного dict;
  - нет строк‑ключей на каждый экземпляр;
  - слоты — это просто несколько ячеек, заранее зарезервированных под значения.
- иногда **чуть быстрее доступ** к полям (меньше уровней косвенности).

**Что потеряли:**

- нельзя «на лету» дописывать новые атрибуты;
- наследование становится **чуть более хитрым** (сейчас разберём).

---

#### 3.2.4. Наследование и `__slots__`: аккуратные детали

Особенности:

- Слоты **не объединяются автоматически** между базовым и дочерним классом,
  если вы не объявите их явно.
- Если и родитель, и потомок объявляют `__slots__`, итоговый объект имеет
  **объединение всех слотов**.

Пример:

```python
class A:
    __slots__ = ("x",)

class B(A):
    __slots__ = ("y",)

b = B()
b.x = 1
b.y = 2
```

Здесь у объекта `B` есть:

- слот `x` (унаследован от `A`);
- слот `y` (объявлен в `B`).

Если вы хотите **и слоты, и `__dict__`** (для динамических атрибутов),
можно явно добавить `'__dict__'`:

```python
class C:
    __slots__ = ("x", "__dict__")

c = C()
c.x = 10          # в слоте
c.y = 20          # в c.__dict__
print(c.__dict__) # {'y': 20}
```

Такой гибрид даёт:

- экономию памяти на часто используемых полях (`x`, `y`, …);
- при этом возможность при необходимости добавлять «нестандартные» атрибуты.

---

#### 3.2.5. Когда стоит (и не стоит) использовать `__slots__`

**Имеет смысл:**

- вы написали небольшой тип данных, который создаётся **десятки/сотни тысяч раз**:
  - точки (`Point`);
  - вершины графа;
  - элементы AST‑дерева;
  - небольшие DTO‑объекты в бизнес‑логике;
- вы точно знаете, какие у него будут поля, и **они не меняются от экземпляра к экземпляру**;
- вы провели измерения (память/профилирование) и увидели, что объекты этого типа занимают
  значимую часть RAM.

**Лучше не использовать:**

- «везде, на всякий случай» — код станет тяжелее поддерживать, а выигрыш может быть нулевым;
- в больших, активно развивающихся иерархиях классов, где важно легко добавлять поля;
- в библиотеках, где пользователи могут ожидать, что у их объектов можно будет
  свободно заводить новые атрибуты.

Полезный подход:

1. Сначала пишем обычные классы **без `__slots__`**.
2. Запускаем приложение под профайлером памяти (`tracemalloc`, `memory_profiler` и т.п.).
3. Если видим «горячие» типы, которые плодятся сотнями тысяч, — **только тогда**
   думаем о `__slots__`.

Если коротко:

> `__slots__` — это **инструмент точечной оптимизации**, а не «обязательная практика».
