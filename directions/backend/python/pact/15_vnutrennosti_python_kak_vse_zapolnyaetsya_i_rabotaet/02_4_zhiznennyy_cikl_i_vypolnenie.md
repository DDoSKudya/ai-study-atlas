### 4.1. Жизненный цикл объекта (§87)

#### 4.1.0. Интуитивная картинка

Представьте, что объект — это **персонаж в игре**:

1. Сначала для него **выделяют «слот» в памяти** (как создать нового персонажа в базе).
2. Потом его **заполняют данными**:
   - имя;
   - уровень;
   - инвентарь.
3. Потом он **живет своей жизнью**:
   - вызываются его методы;
   - меняются поля.
4. В конце, когда на персонажа **никто больше не ссылается**, игра его удаляет:
   - память освобождается;
   - возможны какие‑то «финальные действия».

В Python эти этапы примерно соответствуют:

- `__new__` — выделить место и **создать «пустой» объект**;
- `__init__` — **инициализировать** этот объект (заполнить полями);
- счётчик ссылок + GC — контролируют, **когда объект можно удалить**;
- `__del__` — «финализация» (если вы её явно определили, но с оговорками).

---

#### 4.1.1. `__new__` и `__init__`: две разные задачи

Типичный класс, который вы пишете, выглядит так:

```python
class User:
    def __init__(self, name):
        self.name = name
```

Обратите внимание:

- вы **не писали** `__new__`;
- Python **сам** использует стандартную реализацию `object.__new__`.

Что на самом деле происходит, когда вы пишете:

```python
u = User("Alice")
```

Пошагово:

1. **Вызов конструктора**: `User("Alice")`.
2. Python ищет у класса `User` специальный метод `__new__`.
   - Вы его не определяли → берётся `object.__new__`.
3. Вызывается `User.__new__(User, "Alice")`:
   - на самом деле реализация `object.__new__` **игнорирует лишние аргументы**,
     ей нужен только класс;
   - она **выделяет память** под новый объект типа `User` и возвращает этот «сырой» объект.
4. Полученный объект (назовём его `self`) передаётся в `User.__init__`:

   ```python
   User.__init__(self, "Alice")
   ```

   Здесь уже выполняется ваш код:

   ```python
   self.name = "Alice"
   ```

5. Конструкция `User("Alice")` возвращает именно тот объект, который был создан в `__new__`
   и инициализирован в `__init__`.

**Коротко:**

- `__new__` — **создать объект** (низкоуровневое действие, обычно на C‑стороне).
- `__init__` — **настроить объект** (заполнить поля, проверить аргументы и т.п.).

##### Когда нужен `__new__`

В двух основных случаях:

1. **Неизменяемые типы** (`int`, `str`, `tuple`, `frozenset`, свои наследники от них).
   - После создания **их поля нельзя поменять**.
   - Поэтому всё, что связано с «формированием значения», нужно делать **на этапе `__new__`**.
2. Паттерны вроде **singleton**, кеширование экземпляров и прочий контроль «сколько объектов реально создаётся».

Пример с неизменяемым типом:

```python
class UpperStr(str):
    def __new__(cls, value: str):
        # Меняем строку ещё до создания объекта
        obj = super().__new__(cls, value.upper())
        return obj

s = UpperStr("hello")
print(s, type(s))  # 'HELLO', <class '__main__.UpperStr'>
```

Разбор:

- `UpperStr` наследуется от `str`, а `str` неизменяем;
- чтобы получить «верхний регистр», нужно:
  - **создать** строку уже в нужном виде;
  - поэтому мы переопределяем `__new__` и передаём в `super().__new__`
    уже модифицированное значение;
- `__init__` нам не нужен вообще, потому что «после создания» строку уже не изменить.

#### 4.1.2. Счётчик ссылок: когда Python понимает, что объект больше не нужен

CPython использует **счётчик ссылок (reference counting)**.

Идея очень простая:

- у каждого объекта есть число — **сколько сейчас на него ссылаются переменных/структур**;
- когда мы создаём новую ссылку на объект — счётчик **увеличивается**;
- когда ссылка исчезает — счётчик **уменьшается**;
- когда счётчик становится **0**, объект **можно немедленно удалить**.

На практике:

```python
def demo():
    x = [1, 2, 3]   # создаётся список, refcount = 1 (ссылка x)
    y = x           # ещё одна ссылка, refcount = 2
    z = x           # ещё одна, refcount = 3

    # внутри функции с x, y, z можно работать

# когда функция заканчивается, локальные переменные x, y, z исчезают:
# refcount последовательно уменьшается 3 → 2 → 1 → 0
# когда он становится 0, список может быть сразу удалён
```

Мы не видим refcount напрямую (есть служебная функция `sys.getrefcount`, но ею редко
пользуются в обычном коде). Важно **понимать идею**:

> Объект живёт, пока на него существует хотя бы одна ссылка.

#### 4.1.3. `__del__`: финализатор, на который нельзя полагаться

`__del__(self)` — специальный метод, который **может** быть вызван перед тем, как объект
будет окончательно удалён.

Простейший пример:

```python
class Trace:
    def __del__(self):
        print("Deleting", self)

def make():
    t = Trace()

make()  # после выхода из функции t обычно удаляется
```

Что происходит:

- внутри `make` создаётся объект `t` типа `Trace`;
- при выходе из функции `make` локальная переменная `t` исчезает → refcount объекта уменьшается;
- если **больше нигде** на объект нет ссылок, его счётчик становится 0;
- CPython решает, что объект можно удалить, и **вызывает `t.__del__()`**;
- потом память освобождается.

Ключевое слово — **«обычно»**:

- если есть **циклические ссылки** (когда объекты ссылаются друг на друга),
  refcount может никогда не стать ровно 0;
- объект попадает под действие **циклического сборщика мусора (gc)**;
- в сложных случаях `__del__` может:
  - не быть вызван **вообще**;
  - или быть вызван **позже**, чем вы ожидаете;
  - или не быть вызван при завершении интерпретатора.

Проблемы `__del__`:

- если в `__del__` выбросить исключение, оно **игнорируется**, но может мешать отладке;
- наличие `__del__` усложняет работу GC с циклами объектов;
- нельзя гарантировать порядок вызова `__del__` у связанных объектов.

Современный «правильный» подход:

- **не использовать `__del__` там, где можно обойтись контекстным менеджером**:

  ```python
  class Resource:
      def __enter__(self):
          print("open resource")
          return self

      def __exit__(self, exc_type, exc, tb):
          print("close resource")
          # здесь гарантированно можно освободить ресурс

  with Resource() as r:
      ...
  ```

- или использовать `weakref.finalize`, если нужен аккуратный финализатор,
  не мешающий GC.

Поэтому мы формулируем так:

> Объект создаётся через `__new__`, настраивается в `__init__`,
> живёт, пока на него есть ссылки, а в конце **может** вызвать `__del__`,  
> но на `__del__` **нельзя полагаться как на надёжный деструктор**.

За деталями по сборщику мусора и циклам мы пойдём в **Часть XVII**, здесь нам
важна только **интуитивная картина жизни объекта**.

### 4.2. Байткод и виртуальная машина CPython (§88)

#### 4.2.0. Интуитивная картинка

Python‑код **не выполняется как текст**.  
Происходит несколько шагов:

1. Текст кода превращается в **дерево разбора** (AST) — структура, отражающая синтаксис.
2. AST **компилируется в байткод** — последовательность простых команд для виртуальной машины.
3. Виртуальная машина (VM) **исполняет байткод**, двигаясь по инструкциям и
   используя **стек значений**.

Можно думать так:

- ваш исходник → «перевод» на **мини‑язык байткода**;
- байткод → **интерпретируется** как программа для маленького виртуального процессора.

---

#### 4.2.1. Как выглядит байткод простой функции

Возьмём совсем простую функцию:

```python
import dis

def add(a, b):
    return a + b

dis.dis(add)
```

Примерный вывод (он может немного отличаться между версиями Python, но суть та же):

```text
  4           0 LOAD_FAST                0 (a)
              2 LOAD_FAST                1 (b)
              4 BINARY_ADD
              6 RETURN_VALUE
```

Разбираем **строка за строкой**:

- `LOAD_FAST 0 (a)`
  - «загрузить локальную переменную под индексом 0 (это `a`) на стек».
  - Представьте себе стек как **стопку тарелок**: команда кладёт сверху `a`.
- `LOAD_FAST 1 (b)`
  - кладём поверх стека значение `b`.
- `BINARY_ADD`
  - снимает **два верхних значения** со стека (это `a` и `b`);
  - вычисляет `a + b`;
  - результат кладёт обратно на вершину стека.
- `RETURN_VALUE`
  - берёт верхнее значение со стека;
  - возвращает его из функции как итог.

То есть:

> Вся логика функции `add` в байткоде — это **4 маленькие инструкции** над стеком.

Отсюда несколько важных выводов:

- Python‑VM — **стековая машина**: почти всё крутится вокруг стека значений;
- каждая строка кода в итоге становится набором таких инструкций (порой довольно длинным).

---

#### 4.2.2. Локальные и глобальные переменные: `LOAD_FAST`, `LOAD_GLOBAL`, `LOAD_ATTR`

Почему нас вообще волнует тип инструкции?

Потому что разные инструкции имеют **разную стоимость**:

- `LOAD_FAST` — доступ к **локальной переменной**:
  - очень быстрый, почти прямой доступ по индексу;
- `LOAD_GLOBAL` — доступ к **глобальной переменной**:
  - нужно искать имя в глобальном словаре модуля, потом в `builtins`;
- `LOAD_ATTR` — доступ к **атрибуту объекта**:
  - запускает весь механизм, который мы описывали в разделе 3.1
    (дескрипторы, `__dict__`, `__getattr__` и т.д.).

Иерархия примерно такая:

> **локальные** (быстрее) → **глобальные** → **атрибуты/сложные цепочки** (дороже).

Пример оптимизации:

```python
import math

def compute(xs):
    res = 0.0
    sqrt = math.sqrt  # локальная ссылка
    for x in xs:
        res += sqrt(x)
    return res
```

Если бы мы писали в цикле `math.sqrt(x)`, что бы происходило:

1. `LOAD_GLOBAL math`
2. `LOAD_ATTR sqrt`
3. вызов результата.

А так:

- `math.sqrt` извлекается **один раз** до цикла;
- дальше в цикле мы делаем только `LOAD_FAST sqrt` и вызов:
  - это дешевле для интерпретатора.

В реальном коде это даёт:

- небольшой, но приятный выигрыш производительности в горячих циклах;
- особенно заметно, если вызывается тяжёлая функция в tight‑loop.

#### 4.2.3. .pyc файлы и кеширование байткода

Когда вы запускаете модуль:

```bash
python myscript.py
```

CPython делает примерно следующее:

1. Проверяет, есть ли в каталоге `__pycache__` соответствующий `.pyc`‑файл:
   - имя зависит от версии Python (`myscript.cpython-311.pyc` и т.п.).
2. Если файл есть и он:
   - соответствует текущей версии интерпретатора;
   - не старее, чем исходник `.py`;
   - или проходит проверку по hash (в зависимости от режима),
   то **байткод загружается прямо из него**.
3. Если файла нет или он устарел:
   - Python читает `.py`;
   - парсит текст → AST;
   - компилирует AST → байткод;
   - при необходимости сохраняет `.pyc` в `__pycache__`.

Зачем нам это знать:

- **первый запуск** модуля может быть чуть медленнее (из‑за компиляции);
- последующие запуски экономят время за счёт **повторного использования байткода**;
- руками трогать `.pyc` обычно не нужно, но:
  - если `.pyc` повреждены или конфликтуют, можно удалить `__pycache__` — Python пересоберёт всё заново.

---

Если всё это собрать в одну фразу:

> Python сначала **компилирует** ваш код в байткод (и может кешировать его в `.pyc`),
> а затем **интерпретирует** этот байткод в стековой виртуальной машине,
> где каждая инструкция — маленький шаг над стеком значений и локальными/глобальными переменными.
