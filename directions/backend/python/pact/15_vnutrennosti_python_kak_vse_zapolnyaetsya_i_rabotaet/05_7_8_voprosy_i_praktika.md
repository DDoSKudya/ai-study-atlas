### 7.1. Где помогает знание поиска атрибутов

- **Проектирование API классов.**
  - Понимание MRO и дескрипторов помогает создавать аккуратные mixin‑ы,
    добавляющие функциональность без конфликтов.
- **Отладка «магических» библиотек.**
  - Если фреймворк подменяет `__getattr__`/`__getattribute__`, вы будете понимать,
    почему IDE не видит атрибуты, но код работает.
- **Оптимизация горячих участков.**
  - Уменьшение количества обходов цепочки `экземпляр → класс → дескриптор`
    (например, кеширование часто используемых атрибутов в локальную переменную).

Анти‑паттерн:

- писать сложную логику в `__getattribute__`, не понимая, что она будет выполняться **при каждом доступе к любому атрибуту**.

### 7.2. Где помогают `__slots__` и знание структуры объектов

- Вы пишете высоконагруженный сервис, который создаёт миллионы однотипных объектов —
  например, ноды AST, объекты доменной модели, результаты парсинга логов.
- Измерения (через `tracemalloc`, `sys.getsizeof`, профайлеры) показывают,
  что значимая часть памяти уходит в `__dict__` объектов.

Тогда:

- ввод `__slots__` для этих объектов может сильно уменьшить потребление памяти;
- но всегда сначала **измеряйте**, а не оптимизируйте «на глаз».

### 7.3. Где помогают `dis` и `inspect`

- Понимание, как долго живут локальные переменные, помогает лучше писать
  **генераторы, comprehension и замыкания**.
- `dis` полезен для:
  - объяснения странного поведения (например, замыкания с late binding;
    это рассмотрено в Часть XVI);
  - исследования, как Python компилирует **синтаксический сахар** (`async`, `with`, `for`).
- `inspect` нужен:
  - при написании **декораторов «как у библиотек»** (с сохранением сигнатур);
  - для инструментов командной строки, которые автоматически строят
    парсер аргументов по сигнатуре функции.

### 9.1. Вопросы к разделу 3.1 (поиск атрибутов)

1. **Куда Python смотрит в первую очередь при обращении `obj.attr` у обычного экземпляра класса?**

<details>
<summary>Ответ</summary>
В первую очередь Python смотрит в **личный словарь объекта** — `obj.__dict__`.  
Если там есть ключ `'attr'`, значение возвращается оттуда.
</details>

2. **Что такое MRO и зачем он нужен при множественном наследовании?**

<details>
<summary>Ответ</summary>
**MRO (Method Resolution Order)** — это **упорядоченный список классов**, по которому Python идёт в поисках атрибутов и методов.  
При множественном наследовании MRO определяет, **из какого именно родительского класса** будет взят нужный метод/атрибут, если он определён в нескольких местах.
</details>

3. **Чем `__getattr__` отличается от `__getattribute__`?**

<details>
<summary>Ответ</summary>
- `__getattribute__(self, name)` вызывается **при каждом обращении** к атрибуту и перехватывает весь механизм доступа.  
- `__getattr__(self, name)` вызывается **только если обычный поиск не нашёл атрибут** (в `__dict__`, классе, MRO и дескрипторах).  
Поэтому `__getattribute__` используют очень аккуратно, а `__getattr__` — как «последний шанс».
</details>

### 9.2. Вопросы к разделу 3.2 (`__dict__` и `__slots__`)

1. **Зачем большинству пользовательских классов нужен `__dict__`?**

<details>
<summary>Ответ</summary>
`__dict__` — это **словарь атрибутов экземпляра**.  
Он позволяет:
- динамически добавлять новые атрибуты (`obj.new_attr = ...`);
- хранить данные объекта в виде пары «имя → значение».  
Без `__dict__` экземпляр не смог бы принимать новые атрибуты «на лету».
</details>

2. **Что даёт использование `__slots__` по сравнению с `__dict__`?**

<details>
<summary>Ответ</summary>
`__slots__`:
- **запрещает** создание произвольных атрибутов, оставляя только указанные имена;
- отключает автоматическое создание `__dict__` у экземпляра;
- уменьшает накладные расходы по памяти (нет словаря и строк‑ключей для каждого объекта);
- иногда даёт немного более быстрый доступ к полям.  
Цена — потеря гибкости: нельзя динамически добавлять новые атрибуты (если явно не добавить `'__dict__'` в слоты).
</details>

3. **Когда `__slots__` использовать не стоит, даже если вы о нём знаете?**

<details>
<summary>Ответ</summary>
Не стоит:
- «по привычке» во всех классах — код станет сложнее без заметной выгоды;
- в больших иерархиях, где важно свободно добавлять новые поля в потомках;
- в библиотеках, которые должны быть максимально гибкими для пользователей.  
Рекомендуемый подход: сначала писать обычные классы, а потом, **после профилирования памяти**, добавлять `__slots__` только к «горячим» типам с огромным числом экземпляров.
</details>

### 9.3. Вопросы к разделу 4.1 (жизненный цикл объекта)

1. **В чём принципиальная разница между `__new__` и `__init__`?**

<details>
<summary>Ответ</summary>
- `__new__(cls, *args, **kwargs)` **создаёт объект**: выделяет память и возвращает новый экземпляр (обычно на C‑уровне).  
- `__init__(self, *args, **kwargs)` **инициализирует уже созданный объект**: записывает поля, выполняет проверки и т.п.  
Сначала вызывается `__new__`, потом на возвращённом объекте вызывается `__init__`.
</details>

2. **Почему для неизменяемых типов (например, наследников `str`) настройку объекта делают в `__new__`, а не в `__init__`?**

<details>
<summary>Ответ</summary>
Неизменяемый объект **нельзя изменить после создания**.  
`__init__` работает уже с готовым объектом, поэтому для `str`/`tuple` и их наследников изменить значение в `__init__` невозможно.  
Поэтому надо сформировать нужное значение **ещё в `__new__`** и вернуть уже «готовый» неизменяемый объект.
</details>

3. **Какова базовая идея счётчика ссылок (reference counting) в CPython?**

<details>
<summary>Ответ</summary>
У каждого объекта есть число — **сколько ссылок на него существует**.  
Когда создаётся новая ссылка (переменная, элемент списка/словаря) — счётчик увеличивается.  
Когда ссылка исчезает — уменьшается.  
Когда счётчик становится **0**, объект больше никому не нужен, и его можно сразу удалить.
</details>

### 9.4. Вопросы к разделу 4.2 (байткод и виртуальная машина)

1. **Почему говорят, что CPython — стековая виртуальная машина?**

<details>
<summary>Ответ</summary>
Потому что почти все инструкции байткода работают с **одним стеком значений**:
- `LOAD_FAST` кладёт значение переменной на стек;
- арифметические операции (`BINARY_ADD` и т.п.) снимают операнды со стека и кладут результат обратно;
- `RETURN_VALUE` берёт результат со стека и возвращает его из функции.  
Таким образом, выполнение байткода — это последовательность операций «положить на стек / взять со стека».
</details>

2. **Чем отличается `LOAD_FAST` от `LOAD_GLOBAL`, и почему это важно для производительности?**

<details>
<summary>Ответ</summary>
- `LOAD_FAST` загружает **локальную переменную** по её индексу в таблице локалов — это очень быстрый доступ.  
- `LOAD_GLOBAL` ищет имя сначала в глобальном словаре модуля, потом в `builtins`, что дороже.  
Поэтому вынесение часто используемых функций/объектов в локальные переменные (например, `sqrt = math.sqrt`) может заметно ускорить горячие циклы.
</details>

3. **Зачем Python хранит скомпилированный байткод в `.pyc` файлах?**

<details>
<summary>Ответ</summary>
Чтобы **не компилировать один и тот же исходник заново при каждом запуске**.  
При первом запуске:
- исходник `.py` парсится в AST и компилируется в байткод;
- байткод сохраняется в `.pyc` в `__pycache__`.  
При последующих запусках, если `.pyc` актуален, байткод загружается прямо из него, что ускоряет старт программы.
</details>

### 9.5. Вопросы к разделу 5.1 (`inspect` и интроспекция)

1. **Что такое интроспекция на уровне идей?**

<details>
<summary>Ответ</summary>
Интроспекция — это способность программы **исследовать саму себя во время выполнения**.  
В Python это означает:
- узнать, какие атрибуты есть у объекта;
- посмотреть сигнатуру функции;
- получить исходный код, MRO класса, текущее состояние стека и т.п.
</details>

2. **Что возвращает `inspect.signature(func)` и какую информацию можно из этого извлечь?**

<details>
<summary>Ответ</summary>
`inspect.signature(func)` возвращает объект `Signature`, в котором есть:
- упорядоченный словарь параметров (`.parameters`);
- информация о типе каждого параметра (`Parameter.kind`: позиционный, по имени, `*args`, `**kwargs`);
- значения по умолчанию;
- аннотации типов.  
Это позволяет программно анализировать, какие аргументы принимает функция и как их нужно передавать.
</details>

3. **Зачем нужен `inspect.getmro(SomeClass)`?**

<details>
<summary>Ответ</summary>
`inspect.getmro(cls)` возвращает **кортеж классов в порядке MRO (Method Resolution Order)**.  
По нему видно, в каком порядке Python будет искать методы и атрибуты:
- сначала сам класс;
- потом его базовые классы в определённом порядке;
- в конце `object`.  
Это помогает разбираться в сложном множественном наследовании и отладке mixin‑ов/метаклассов.
</details>

### 9.6. Вопросы к разделу 5.2 (аннотации, декораторы, `__wrapped__`)

1. **В чём практический смысл директивы `from __future__ import annotations`?**

<details>
<summary>Ответ</summary>
Она делает так, что **все аннотации типов хранятся как строки**, а не живые объекты.  
Это:
- позволяет ссылаться на типы, объявленные ниже по коду;
- уменьшает накладные расходы при импорте (типы не создаются сразу).  
При необходимости строки‑аннотации потом можно «разрешить» в реальные типы через `typing.get_type_hints`.
</details>

2. **Что ломается, если писать декораторы без `functools.wraps`?**

<details>
<summary>Ответ</summary>
Без `@wraps`:
- у обёртки меняется `__name__` (становится `"wrapper"`);
- теряются `__doc__` (докстринг) и `__annotations__`;
- `inspect.signature` и `get_type_hints` начинают показывать сигнатуру обёртки, а не исходной функции.  
В результате фреймворки (FastAPI, pytest и др.) могут некорректно работать с такой функцией: неправильно строить схемы запросов, документацию, автотесты.
</details>

3. **Зачем `inspect.unwrap` использует атрибут `__wrapped__`?**

<details>
<summary>Ответ</summary>
`@wraps` ставит на обёрнутую функцию атрибут `__wrapped__`, указывающий на оригинальную.  
`inspect.unwrap` проходит по цепочке `__wrapped__ → __wrapped__ → ...` и возвращает **исходную функцию без декораторов**.  
Это нужно, чтобы инструменты могли работать именно с оригинальной сигнатурой и аннотациями, а не с обёрткой.
</details>

### 9.7. Вопросы к разделу 6.1 (AST, `compile`, `eval`/`exec`)

1. **Зачем Python строит AST перед тем, как компилировать код в байткод?**

<details>
<summary>Ответ</summary>
AST (абстрактное синтаксическое дерево) — это удобное внутреннее представление кода:  
- каждая конструкция (`if`, `for`, `x = 1 + 2`) представлена узлом дерева;  
- компилятору гораздо проще работать с деревом, чем с «сырым» текстом.  
AST позволяет:
- проверять корректность синтаксиса;
- применять высокоуровневые оптимизации;
- а уже потом генерировать байткод по этому дереву.
</details>

2. **Чем `mode="eval"` в `compile()` отличается от `mode="exec"`?**

<details>
<summary>Ответ</summary>
- `mode="eval"` ожидает **одно выражение** и возвращает объект кода, который можно вычислить через `eval()` и получить результат.  
- `mode="exec"` принимает **произвольный блок кода** (несколько строк, определения функций и т.д.) и возвращает объект кода, который нужно исполнять через `exec()`, не ожидая возвращаемого значения.
</details>

3. **Почему `ast.literal_eval` безопаснее, чем `eval`?**

<details>
<summary>Ответ</summary>
Потому что `ast.literal_eval`:
- умеет разбирать только **литералы Python** (числа, строки, коллекции из них);  
- не выполняет произвольный код и не разрешает обращения к функциям/атрибутам.  
`eval` же может выполнить всё, что содержится в строке, включая опасные вызовы системных функций.
</details>

### 9.8. Вопросы к разделу 6.2 (структура объектов: `PyObject`, `PyVarObject`)

1. **Какие две ключевые вещи хранит `PyObject` в заголовке любого объекта?**

<details>
<summary>Ответ</summary>
`PyObject` хранит:
- `ob_refcnt` — **счётчик ссылок** на объект;
- `ob_type` — указатель на **тип объекта** (`PyTypeObject`).  
Через `ob_type` интерпретатор знает, «какой это объект» и какие операции с ним разрешены.
</details>

2. **Зачем нужен `PyVarObject` поверх `PyObject`?**

<details>
<summary>Ответ</summary>
`PyVarObject` расширяет `PyObject`, добавляя поле `ob_size` — «логический размер».  
Это нужно для объектов **переменной длины** (списки, кортежи, строки и т.п.), чтобы один и тот же тип мог представлять экземпляры разной длины и хранить эту длину в заголовке.
</details>

3. **Почему `list.append` в среднем O(1), хотя иногда приводит к копированию массива?**

<details>
<summary>Ответ</summary>
Списки используют стратегию **over‑allocation**:
- иногда при `append` выделяется новый, более крупный блок памяти и все элементы копируются туда (операция O(n));  
- но такие «дорогие» расширения происходят не на каждом шаге, а лишь иногда.  
Если смотреть на длинную последовательность `append`, средняя стоимость одной операции получается **амортизированно O(1)**.
</details>

### 9.9. Вопросы к разделу 6.3 (code objects и frame objects)

1. **Что такое code object и где его можно увидеть из Python‑кода?**

<details>
<summary>Ответ</summary>
Code object — это объект типа `CodeType`, который хранит:
- байткод функции (`co_code`);
- имена локальных переменных (`co_varnames`);
- константы (`co_consts`) и другую статическую информацию.  
Его можно увидеть через атрибут функции `func.__code__`.
</details>

2. **В чём различие между code object и frame object?**

<details>
<summary>Ответ</summary>
- **Code object** — «паспорт» функции, статическое описание кода (общее для всех вызовов).  
- **Frame object** — конкретный **кадр выполнения** при одном вызове:
  - содержит ссылку на `f.__code__`;
  - хранит локальные переменные, стек и ссылку на предыдущий кадр.  
Одна и та же функция (один code object) при многократных вызовах создаёт множество разных фреймов.
</details>

3. **Как отладчики и профайлеры используют фреймы?**

<details>
<summary>Ответ</summary>
Они обходят цепочку `frame.f_back`:
- чтобы восстановить стек вызовов (какие функции вызывали друг друга);  
- читают `frame.f_code` для имени функции/файла/строки;  
- анализируют `frame.f_locals` и `frame.f_globals` для показа переменных.  
Так строятся трейсбеки и отчёты профилировщиков.
</details>

### 9.10. Вопросы к разделу 6.4 (GIL и многопоточность)

1. **Почему GIL мешает ускорять CPU‑тяжёлые задачи через потоки в CPython?**

<details>
<summary>Ответ</summary>
Потому что **в каждый момент времени байткод исполняет только один поток**, удерживающий GIL.  
Другие потоки ждут своей очереди.  
Если задача — чистые вычисления в Python‑цикле, потоки будут по очереди получать GIL, но реально параллельного выполнения на нескольких ядрах не будет.
</details>

2. **В каком случае потоки при наличии GIL всё-таки полезны?**

<details>
<summary>Ответ</summary>
При **I/O‑ограниченных задачах**:
- когда один поток ждёт сеть/диск/БД, он может отпустить GIL;  
- в это время другой поток может исполнять свой код;  
- таким образом, мы скрываем задержки I/O и лучше используем процессор.
</details>

3. **В чём идея free‑threaded Python (PEP 703)?**

<details>
<summary>Ответ</summary>
Идея — **убрать глобальный GIL** и вместо этого:
- использовать более тонкие блокировки на уровне отдельных объектов/структур;
- адаптировать C‑расширения к новой модели.  
Цель — позволить Python‑потокам **реально параллельно** выполнять байткод на нескольких ядрах, не блокируясь одним общим замком.
</details>
