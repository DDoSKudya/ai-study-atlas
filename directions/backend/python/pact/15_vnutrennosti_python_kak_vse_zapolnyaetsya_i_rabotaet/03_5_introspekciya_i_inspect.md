В этом блоке важно **не зазубрить функции модуля `inspect`**, а:

- понять, **что такое интроспекция** в принципе;
- увидеть, **как с помощью неё думают фреймворки** (FastAPI, Django, pytest и т.п.);
- научиться **спокойно читать сигнатуры функций, фреймы и аннотации**.

### 5.1. Интроспекция, рефлексия и модуль `inspect`

#### 5.1.0. Интуитивная картинка

Представьте, что программа — это **город**:

- у города есть **карта**:
  - какие есть улицы (модули);
  - какие здания (классы, функции);
  - кто в каком доме живёт (экземпляры, атрибуты).

**Интроспекция** — это способность **самого города достать свою карту**:

- узнать: «какие вообще тут есть дома и кто в них живёт?»;
- посмотреть: «как выглядит план конкретного здания?» (сигнатура функции);
- прочитать: «что написано в документации на этот дом?» (docstring).

**Рефлексия** идёт дальше:

- не только смотреть на карту, но и **менять город, пользуясь картой**:
  - строить новые дома по описанию;
  - менять поведение на лету;
  - поднимать REST‑эндпоинты на основе сигнатур функций и аннотаций.

В Python:

- за «достать карту» в основном отвечает модуль **`inspect`**;
- за «менять город по карте» — метаклассы, `type()`, AST, `exec` и т.п. (это уже раздел 6.x).

Если коротко:

> Интроспекция = «посмотреть на структуру программы изнутри».  
> `inspect` = «набор инструментов для этого».

#### 5.1.1. Сигнатуры функций: `inspect.signature` и `Parameter`

Сигнатура функции — это **формальное описание того, какие аргументы она принимает** и что возвращает:

- имена параметров;
- какие из них позиционные, какие только по ключу;
- есть ли `*args` / `**kwargs`;
- какие значения по умолчанию;
- какие аннотации типов.

Пример:

```python
import inspect

def example(a, b: int, *args, c: str = "x", **kwargs) -> bool:
    return True

sig = inspect.signature(example)
print(sig)  # (a, b: int, *args, c: str = 'x', **kwargs) -> bool

for name, param in sig.parameters.items():
    print(name, "|", param.kind, "|", param.default, "|", param.annotation)
```

Разберём, что тут происходит:

- `inspect.signature(example)` возвращает объект `Signature`:
  - в нём хранится **полная информация о параметрах** и возвращаемом типе.
- `sig.parameters` — упорядоченный словарь:
  - ключ — имя параметра (`"a"`, `"b"`, `"args"`, `"c"`, `"kwargs"`);
  - значение — объект `inspect.Parameter`.

У `Parameter` есть важное поле `kind`:

- `POSITIONAL_ONLY` — **только позиционный аргумент**:
  - его можно передать только как `func(10, ...)`, но не как `func(x=10, ...)`;
- `POSITIONAL_OR_KEYWORD` — **и позиционный, и по имени** (обычный параметр в Python);
- `VAR_POSITIONAL` — `*args`:
  - «собери все лишние позиционные аргументы в кортеж»;
- `KEYWORD_ONLY` — **только по имени**:
  - объявляются **после звёздочки** `*`;
- `VAR_KEYWORD` — `**kwargs`:
  - «собери все лишние аргументы по имени в словарь».

Мини‑пример для `KEYWORD_ONLY`:

```python
def f(a, *, b, c=10):
    ...
```

- `a` — `POSITIONAL_OR_KEYWORD`;
- `b` и `c` — `KEYWORD_ONLY` (их нельзя передать по позиции, только так: `f(1, b=2, c=3)`).

Зачем всё это:

- при написании **декораторов**, которые хотят «вести себя как исходная функция»:
  - нужно знать, какие аргументы ждать и как их проксировать;
- в фреймворках:
  - FastAPI читает сигнатуру функции‑обработчика, чтобы понять:
    - какие параметры брать из пути;
    - какие — из query;
    - какие — из тела запроса;
  - DI‑контейнеры (dependency injection) решают, какие зависимости нужно подставить.

#### 5.1.2. Получение исходника и MRO

Модуль `inspect` умеет не только смотреть на сигнатуры, но и:

- **доставать исходный код** объекта;
- **смотреть MRO класса** (то самое Method Resolution Order из раздела 3.1).

Пример:

```python
import inspect

def example(a, b):
    return a + b

class Base: ...
class Child(Base): ...

print(inspect.getsource(example))   # исходный код функции
print(inspect.getmro(Child))        # кортеж классов в порядке MRO
```

`inspect.getsource`:

- находит файл, где определена функция/класс/метод;
- читает оттуда нужный фрагмент текста;
- возвращает его как строку.

Это полезно:

- для генерации документации;
- для автогенерации конфигов (например, когда нужно сгенерировать «скелет» теста);
- при отладке:
  - когда вы получили **функцию как объект** (например, из декоратора),
    и хотите увидеть её «сырой» исходник.

`inspect.getmro(Child)`:

- возвращает кортеж классов в порядке, в котором Python **будет искать методы и атрибуты**;
- это тот же самый порядок, что и у `Child.mro()`;
- удобно, когда вы:
  - разбираетесь в сложной иерархии mixin‑ов;
  - пишете или отлаживаете метаклассы.

#### 5.1.3. Фреймы: `inspect.currentframe`, `inspect.stack`

Вернёмся к нашему «городу» и «карте».

**Frame (кадр выполнения)** — это как **снимок одного уровня стека вызовов**:

- какая функция сейчас выполняется;
- какие у неё локальные переменные;
- какие глобальные переменные видны;
- откуда она была вызвана.

Технически frame хранит:

- `frame.f_code` — объект кода (о нём говорили в разделе 6.3):
  - в нём есть имя функции, имена локальных переменных и т.п.;
- `frame.f_locals` — словарь с **локальными переменными**;
- `frame.f_globals` — словарь с **глобальными переменными модуля**;
- `frame.f_back` — ссылка на **предыдущий кадр** (кто нас вызвал).

Пример:

```python
import inspect

def inner():
    frame = inspect.currentframe()
    print("Сейчас в функции:", frame.f_code.co_name)           # 'inner'
    print("Нас вызвала функция:", frame.f_back.f_code.co_name) # 'outer'

def outer():
    inner()

outer()
```

Здесь:

- `inspect.currentframe()` возвращает **кадр для текущей функции** (`inner`);
- `frame.f_back` — это кадр для `outer`, которая вызвала `inner`.

`inspect.stack()` идёт дальше:

- возвращает **список** кадров (как трейсбек):
  - где каждый элемент описывает:
    - функцию;
    - файл и номер строки;
    - локальные переменные.

Почему это важно:

- отладчики, профилировщики, логгеры высокого уровня **строятся именно на этих объектах**;
- если вы понимаете, что такое frame, вам легче читать:
  - трейсбеки;
  - вывод профилировщика;
  - нестандартные ошибки, связанные с рекурсией и вызовами.

Но:

- `inspect.stack()` и `currentframe()` — **тяжёлые по производительности**:
  - собирают много информации;
  - могут мешать оптимизациям интерпретатора;
- поэтому в продакшн‑коде их стоит использовать:
  - только в отладочных и логирующих ветках;
  - не вызывать в tight‑loop’ах.

### 5.2. Аннотации, `typing.get_type_hints`, декораторы и `__wrapped__`

В этом разделе три темы, которые в реальном коде часто переплетаются:

- **аннотации типов** (`__annotations__`, `from __future__ import annotations`);
- **декораторы и метаданные** (`functools.wraps`, `inspect.unwrap`);
- **низкоуровневый доступ к фреймам** (`sys._getframe`).

---

#### 5.2.1. `__annotations__` и `from __future__ import annotations`

У любой функции/класса/метода может быть атрибут `__annotations__` —
**словарь с подсказками типов**:

```python
def f(x: "User", y: int) -> bool:
    return True

print(f.__annotations__)
# {'x': 'User', 'y': <class 'int'>, 'return': <class 'bool'>}
```

Особенности:

- ключи словаря — имена параметров и специальный ключ `'return'` для возвращаемого типа;
- значения — **объекты типов** или **строки** (зависит от версии/настроек).

Директива:

```python
from __future__ import annotations
```

делает так, что **все аннотации хранятся как строки**, а не живые объекты:

- `"User"` вместо настоящего класса `User`;
- `"list[int]"` вместо `list[int]` и т.п.

Зачем так делают:

- можно ссылаться на типы, которые будут объявлены **позже по коду**;
- при импорте модуля Python не обязан немедленно создавать кучу объектов типов:
  - это уменьшает накладные расходы;
  - ускоряет импорт, особенно в больших проектах.

Чтобы получить «настоящие» объекты типов, используют:

```python
from typing import get_type_hints

print(get_type_hints(f))
```

`get_type_hints`:

- берёт `__annotations__`;
- при необходимости **выполняет `eval` этих строк** в пространстве имён модуля;
- возвращает словарь, где значения — **реальные типы**, а не строки.

Из этого следуют важные выводы:

- вы **не должны** вызывать `get_type_hints` на **недоверенном коде**:
  - если там «плохие» строки, `eval` может выполнить произвольный код;
- в обычных проектах это безопасно, если вы контролируете свой код и зависимости.

#### 5.2.2. Декораторы, `functools.wraps` и `inspect.unwrap`

Почти все современные библиотеки используют **декораторы**:

- для логирования;
- для кеширования (например, `functools.lru_cache`);
- для проверки прав (в web‑фреймворках);
- для описания эндпоинтов (`@app.get("/path")` в FastAPI и т.п.).

Проблема в том, что наивный декоратор **ломает метаданные функции**:

```python
def deco(func):
    def wrapper(*args, **kwargs):
        return func(*args, **kwargs)
    return wrapper

@deco
def greet(name: str) -> None:
    """Поздороваться по имени."""
    print("Hi", name)

print(greet.__name__)         # 'wrapper', а не 'greet'
print(greet.__doc__)          # None
print(greet.__annotations__)  # {}
```

С точки зрения интроспекции:

- мы **потеряли всю информацию** об исходной функции;
- `inspect.signature(greet)` теперь покажет сигнатуру `wrapper`, а не `greet`;
- фреймворки вроде FastAPI/pytest могут начать работать неправильно.

Решение — использовать `functools.wraps`:

```python
from functools import wraps

def deco(func):
    @wraps(func)
    def wrapper(*args, **kwargs):
        return func(*args, **kwargs)
    return wrapper
```

Что делает `@wraps(func)`:

- копирует на `wrapper`:
  - `__name__`;
  - `__module__`;
  - `__doc__`;
  - `__annotations__`;
- ставит атрибут `__wrapped__`, который указывает на **оригинальную функцию**.

Тогда:

```python
@deco
def greet(name: str) -> None:
    """Поздороваться по имени."""
    print("Hi", name)

print(greet.__name__)         # 'greet'
print(greet.__doc__)          # 'Поздороваться по имени.'
print(greet.__annotations__)  # {'name': <class 'str'>, 'return': None}
```

А `inspect.unwrap` умеет «раскрутить» цепочку декораторов:

```python
import inspect

orig = inspect.unwrap(greet)
print(orig is greet)   # False, это исходная функция под обёртками
print(orig.__name__)   # 'greet'
```

Почему это критично:

- для **генерации документации**: нужно знать настоящий текст и сигнатуру функции;
- для таких библиотек, как FastAPI:
  - они смотрят на аннотации оригинальной функции, чтобы строить схемы запросов/ответов;
- для `typing.get_type_hints`:
  - он умеет использовать `__wrapped__`, чтобы анализировать **исходную** функцию.

Если коротко:

> Всегда, когда пишете декоратор, заверните обёртку в `@wraps(func)`.  
> Иначе вы ломаете интроспекцию и делаете жизнь себе и фреймворкам сложнее.

#### 5.2.3. `sys._getframe(depth)` — мощный, но опасный инструмент

Функция `sys._getframe(depth)` — это почти «голый доступ» к стеку вызовов:

- `depth = 0` — кадр текущей функции;
- `depth = 1` — кадр вызывающей функции;
- `depth = 2` — кадр «дедушки» и т.д.

Пример:

```python
import sys

def who_called_me():
    frame = sys._getframe(1)  # 1 = наш вызывающий
    print("Меня вызвала функция:", frame.f_code.co_name)

def outer():
    who_called_me()

outer()  # выведет: Меня вызвала функция: outer
```

Что даёт `_getframe`:

- доступ к:
  - `frame.f_locals` — локальным переменным вызывающей функции;
  - `frame.f_globals` — её глобальной области;
  - `frame.f_code` — объекту кода;
- возможность строить:
  - хитрые логгеры и трассировщики;
  - DSL‑ы (внутренние языки), которые зависят от контекста вызова.

Почему это **опасно**:

- `_getframe` — **неофициальный, низкоуровневый API**:
  - подчёркнуто подчёркивается `_` в имени;
  - реализация может отличаться в других интерпретаторах (PyPy, Jython и т.п.);
- частое использование `_getframe`:
  - мешает оптимизациям (интерпретатор не может так агрессивно менять представление стека);
  - замедляет код;
- код становится **сложно читаемым и хрупким**.

Практическая рекомендация:

- `_getframe` уместен:
  - в **инструментах**: профиляторы, отладчики, логгеры, тестовые раннеры;
  - в очень низкоуровневых библиотеках, где других вариантов нет.
- В обычном прикладном коде:
  - лучше использовать более высокоуровневые механизмы (`logging`, `inspect.traceback`,
    контекстные менеджеры и т.д.);
  - и избегать `_getframe`, если вы не пишете отладчик/профайлер.
