## Специальные атрибуты объекта

| Атрибут | Что хранит / зачем нужен |
|---------|--------------------------|
| `__dict__` | Словарь атрибутов экземпляра. Есть не у всех (нет при `__slots__` без `__dict__`). |
| `__class__` | Класс объекта. |
| `__slots__` | Описание фиксированного набора атрибутов; оптимизация памяти, запрет динамических атрибутов. |
| `__mro__` | Порядок разрешения методов (tuple классов). |
| `__bases__` | Базовые классы класса. |
| `__name__` | Имя объекта (у функций, классов, модулей). |
| `__qualname__` | Полное имя (включая вложенность). |
| `__module__` | Имя модуля, в котором объявлен объект. |
| `__doc__` | Докстрока. |
| `__annotations__` | Аннотации типов (у функций, классов, модулей). |
| `__weakref__` | Поддержка слабых ссылок (если включено). |

`__slots__` — важный механизм для **снижения потребления памяти** и иногда ускорения доступа, но:

- усложняет наследование;
- ломает динамическое добавление атрибутов без дополнительных настроек.

### Как работают `__dict__` и `__slots__`

По умолчанию каждый экземпляр класса имеет:

- скрытое поле «указатель на словарь атрибутов»;
- сам словарь `__dict__` (обычно хеш-таблица).

Это удобно, но даёт накладные расходы по памяти. Если вы объявляете:

```python
class Point:
    __slots__ = ("x", "y")
```

интерпретатор:

- вместо словаря заводит **фиксированный массив «слотов»** для атрибутов `x` и `y`;
- не добавляет в объект указатель на словарь (если вы явно не включили `"__dict__"` в `__slots__`).

Последствия:

- Экземпляры занимают меньше памяти (особенно при большом числе объектов).
- Нельзя динамически добавлять новые атрибуты: `p.z = 3` вызовет `AttributeError`.
- Множественное наследование с `__slots__` становится сложнее (каждый родительский класс со слотами добавляет свои слоты).

Если нужно и экономить память, и поддерживать дополнительные атрибуты, можно:

```python
class WithDictAndSlots:
    __slots__ = ("x", "y", "__dict__")
```

Здесь `x`/`y` будут в слотах, а остальные атрибуты — в `__dict__`.

### `__mro__` и поиск методов

`Class.__mro__` — это кортеж классов, определяющий **порядок поиска атрибутов** при обращении к классу или его экземплярам.  
Он вычисляется алгоритмом **C3 linearization** (подробнее — в части про ООП и MRO).

Пример:

```python
class A: ...
class B(A): ...
class C(A): ...
class D(B, C): ...

print(D.__mro__)
```

Знание `__mro__` важно при:

- множественном наследовании (понимание, чей метод будет вызван первым);
- использовании `super()` (он опирается на `__mro__`);
- отладке конфликтов имён и «странного» поведения при наследовании.

### `__qualname__` и вложенные определения

`__qualname__` (qualified name) — полное имя объекта, включающее **контекст**:

```python
def outer():
    class Inner:
        pass
    return Inner

InnerClass = outer()
print(InnerClass.__name__)     # Inner
print(InnerClass.__qualname__) # outer.<locals>.Inner
```

Это полезно:

- для человеко-читаемых сообщений об ошибках;
- при сериализации/логировании (легче понять, откуда взялся класс/функция);
- для introspection и инструментов (IDE, дебаггеры).

### `__module__` и «где определён объект»

`__module__` — имя модуля, в котором объект **был объявлен** (не обязательно где он сейчас импортирован). Это используется:

- при генерации repr/traceback’ов;
- в инструментах документирования;
- при сериализации (в связке `__module__` + `__qualname__` можно однозначно найти объект в большинстве случаев).
