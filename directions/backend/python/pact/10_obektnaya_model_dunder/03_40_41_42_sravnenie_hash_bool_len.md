### Базовые методы и операторы

| Оператор | Метод |
|----------|-------|
| `==` | `__eq__(self, other)` |
| `!=` | `__ne__(self, other)` (по умолчанию `not __eq__`) |
| `<` | `__lt__(self, other)` |
| `<=` | `__le__(self, other)` |
| `>` | `__gt__(self, other)` |
| `>=` | `__ge__(self, other)` |

**Важно:** в Python 3 **нет** универсального сравнения «всего со всем» (как было в Py2). Если типы не совместимы, корректно вернуть `NotImplemented`.

#### Отношения порядка и инварианты

Если вы реализуете **полный порядок** (total ordering), логика сравнения должна удовлетворять стандартным математическим свойствам:

- **Рефлексивность:** `a == a` (если не NaN-подобный случай);
- **Симметричность:** `a == b` ⇔ `b == a`;
- **Транзитивность:** `a < b` и `b < c` ⇒ `a < c`.

В реальной жизни:

- большинство ваших классов должны вести себя **как числа / строки** с точки зрения порядка;
- если порядок «частичный» (например, граф с неполной упорядоченностью), лучше не реализовывать все операторы — или чётко задокументировать поведение.

#### Как сравнение используется стандартными структурами данных

Понимание того, где именно используются `__eq__`/операторы порядка, помогает проектировать свои классы:

- **`list` / `tuple` / `dataclass` по умолчанию:**
  - сравниваются **лексикографически** по элементам/полям;
  - используют `__eq__`/`__lt__` их элементов (или встроенные правила для чисел/строк).
- **`set` / `frozenset`:**
  - равенство и операции включения (`<=`, `<`, `>=`, `>`) основаны на **элементах множества**;
  - порядок элементов в множестве **не определён** — их нельзя сортировать напрямую, если только вы не создадите список/кортеж и не отсортируете его отдельно.
- **`dict`:**
  - в Python 3.7+ порядок ключей при итерации — **порядок вставки**, а не результат сортировки;
  - сравнение словарей (`==`) использует равенство ключей и значений (без порядка);
  - операторов `<`, `>` и т.п. для `dict` нет (поднимут `TypeError`).

В пользовательском коде чаще всего:

- **`__eq__`** нужен для:
  - сравнения объектов (в тестах, при проверке изменений, в бизнес-логике);
  - работы `in` в списках/множествах/словарях (поиск по значению, ключу);
  - корректной работы `set`/`dict` с вашими объектами.
- Операторы порядка (`<`, `>`, …) — для:
  - сортировок (`sorted`, `.sort()`);
  - алгоритмов, опирающихся на сравнение (кучи, деревья, интервалы).

В современных Python-программах сортировки чаще используют **параметр `key=`**, чем сами операторы `<`/`>`. Поэтому иногда достаточно:

- реализовать только `__eq__`/`__hash__`;
- а порядок задавать через отдельную функцию/метод, передавая его через `key=`.

### Возврат `NotImplemented`

Если ваш метод сравнения **не знает**, как сравнить объект с `other`, он должен вернуть `NotImplemented`, а не поднимать исключение.

Тогда интерпретатор:

- попробует **обратный** метод на другом объекте (`other.__eq__(self)`, `other.__lt__(self)` и т.п.);
- если и он возвращает `NotImplemented`, поднимет `TypeError`.

```python
class Temperature:
    def __init__(self, celsius: float):
        self.celsius = celsius

    def __eq__(self, other):
        if isinstance(other, Temperature):
            return self.celsius == other.celsius
        return NotImplemented

    def __lt__(self, other):
        if isinstance(other, Temperature):
            return self.celsius < other.celsius
        return NotImplemented
```

#### Пример проблемы, если вернуть `False` вместо `NotImplemented`

```python
class Weird:
    def __eq__(self, other):
        # ПЛОХО: просто False, даже если другой знает, как сравнивать
        return False

class WeirdAware:
    def __eq__(self, other):
        if isinstance(other, WeirdAware):
            return True
        return NotImplemented

print(Weird() == WeirdAware())       # False (Weird "отменил" шанс WeirdAware сравнить)
print(WeirdAware() == Weird())       # True  (WeirdAware смог сравнить первым)
```

Такое несогласованное поведение делает код трудноотлаживаемым. Если ваш класс **не знает**, как сравниваться с другим типом, лучше вернуть `NotImplemented` и дать шанс второму операнду.

### `functools.total_ordering`

Часто достаточно реализовать `__eq__` и один из `<`, `<=`, `>` или `>=`. Декоратор `functools.total_ordering` сгенерирует остальные.

```python
from functools import total_ordering

@total_ordering
class Version:
    def __init__(self, major: int, minor: int):
        self.major = major
        self.minor = minor

    def __eq__(self, other):
        if not isinstance(other, Version):
            return NotImplemented
        return (self.major, self.minor) == (other.major, other.minor)

    def __lt__(self, other):
        if not isinstance(other, Version):
            return NotImplemented
        return (self.major, self.minor) < (other.major, other.minor)
```

**Плюсы `total_ordering`:**

- меньше повторяющегося кода;
- легче поддерживать инварианты (все операции порядка выразятся через одну).

**Минусы:**

- лёгкое замедление (каждая операция может вызывать несколько сравнений);
- если вы ошиблись в базовой реализации (`__lt__`/`__eq__`), ошибка **распространится** на все операторы.

Практический совет: используйте `total_ordering`, когда ваш порядок естественно выражается через **одну** основную операцию (`<` или `>`), и обязательно покрывайте сравнения **юнит-тестами**.

## §41. Хеширование

### Контракт `__hash__` и `__eq__`

**Требование:**

- если `a == b`, то **обязательно** `hash(a) == hash(b)`;
- если `a != b`, хеши **могут совпадать** (коллизии допускаются, но их нужно минимизировать).

Python применяет правило:

- если вы переопределили `__eq__` и **не** переопределили `__hash__`, то по умолчанию `__hash__ = None`, и объект становится **не hashable** (нельзя использовать как ключ в `dict` и элемент `set`).

Это **намеренное поведение**, введённое для того, чтобы:

- не допустить нарушений инварианта словаря/множества;
- заставить разработчика **осознанно** решить, должен ли объект быть hashable.

### Когда объект должен быть hashable

Объект должен быть hashable, если:

- его экземпляры используются как **ключи словарей** или элементы **множеств**;
- он логически **immutable**: его сравнение и хеш не должны меняться за время жизни.

#### Пример: hashable объект

```python
class Point2D:
    def __init__(self, x, y):
        self._x = x
        self._y = y

    @property
    def x(self):
        return self._x

    @property
    def y(self):
        return self._y

    def __eq__(self, other):
        if not isinstance(other, Point2D):
            return NotImplemented
        return (self.x, self.y) == (other.x, other.y)

    def __hash__(self):
        return hash((self.x, self.y))
```

**Практика:** обычно хешируют **кортеж полей**, участвующих в `__eq__`.

Преимущества такого подхода:

- кортеж уже умеет корректно считать хеш (с учётом порядка и типов);
- вы автоматически получаете согласованность с `__eq__`, если он основан на том же кортеже.

#### Делать объект заведомо не hashable

Если объект **изменяемый**, лучше явно указать:

```python
class MutablePoint:
    __hash__ = None
```

Тогда попытка использовать его как ключ словаря приведёт к `TypeError`, а не к тихим, трудноуловимым багам.

#### Типичные ошибки при хешировании

1. **Хеш зависит от изменяемого поля.**

   ```python
   class Bad:
       def __init__(self, items):
           self.items = list(items)

       def __hash__(self):
           return hash(tuple(self.items))
   ```

   Если `self.items` меняется после помещения объекта в `set`/`dict`, поведение коллекции становится неопределённым: объект «теряется» внутри структуры.

   **Решение:**  
   - либо сделать объект логически immutable (не изменять поля, участвующие в `__hash__`);  
   - либо запретить хеширование (`__hash__ = None`).

2. **Несогласованность `__eq__` и `__hash__`.**

   ```python
   class Bad2:
       def __init__(self, x):
           self.x = x

       def __eq__(self, other):
           return isinstance(other, Bad2) and self.x == other.x

       def __hash__(self):
           # ПЛОХО: хеш не зависит от x
           return 1
   ```

   Формально контракт не нарушен (`a == b` ⇒ `hash(a) == hash(b)`), но практически вы получите **лавину коллизий** и сильное замедление `dict`/`set`.

   **Решение:** хешировать именно те поля, которые участвуют в `__eq__`, стараясь не создавать искусственно огромных коллизий.

## §42. Булево значение и `__len__`

### `__bool__` и truthiness

Функция `bool(obj)` и условные конструкции (`if obj:`) используют:

1. Если определён `__bool__(self)`, берут его результат.
2. Иначе, если определён `__len__(self)`, объект считается `False`, если `len(self) == 0`, и `True` иначе.
3. Иначе объект считается `True`.

#### Пример: объект-результат валидации

```python
class ValidationResult:
    def __init__(self, errors):
        self.errors = errors

    def __bool__(self):
        return not self.errors

    def __repr__(self):
        return f"ValidationResult(errors={self.errors!r})"

result = ValidationResult(["too short"])
if not result:
    print("Валидация не прошла:", result)
```

### `__len__`

Метод `__len__(self)`:

- должен возвращать **неотрицательное целое**;
- используется `len(obj)` и как fallback для `bool(obj)`.

#### Частая ошибка

- возвращать из `__len__` отрицательное число или нецелое;
- вызывать `len()` с побочными эффектами (длительные вычисления / сетевые запросы).

**Правило:** `__len__` должен быть **быстрым** и **детерминированным**.

#### Truthiness стандартных типов (для интуиции)

| Тип | Что считается `False` | Что считается `True` |
|-----|------------------------|----------------------|
| `NoneType` | `None` | — |
| `bool` | `False` | `True` |
| `int`, `float`, `Decimal` и числа | `0`, `0.0`, `0j` | Любое ненулевое число |
| `str`, `bytes`, `bytearray` | Пустая строка/последовательность | Любая непустая |
| `list`, `tuple`, `dict`, `set`, `frozenset` | Пустая коллекция | Непустая |
| Пользовательские классы | Обычно: `__bool__` или `__len__` | По умолчанию — всегда `True` |

Понимание этой таблицы помогает проектировать свои классы так, чтобы они **вели себя как стандартные коллекции и значения**: пустой результат — `False`, непустой — `True`.
