## 7. Дата и время (§123)

### 7.1. `datetime` и «наивные» vs «осознанные» даты

```python
from datetime import datetime, timezone

now_utc = datetime.now(timezone.utc)
print(now_utc.isoformat())
```

**Наивный `datetime`** (без `tzinfo`) не содержит информации о таймзоне и часто приводит к ошибкам при сравнении/сохранении.

Правило:

- в продакшене работайте либо в **UTC**, либо в явных таймзонах;
- храните даты в БД как UTC + отдельное поле с нужной TZ, если это действительно важно.

Представьте:

- `datetime(2020, 1, 1, 12, 0)` — просто «2020‑01‑01 12:00». **Где?** В Москве? В Нью‑Йорке? В UTC?
- `datetime(2020, 1, 1, 12, 0, tzinfo=timezone.utc)` — чётко: «полдень по UTC».

Проблема «наивных» дат:

- пока вы работаете на одной машине → может казаться, что всё нормально;
- как только:
  - дата попала в БД;
  - её прочитал другой сервис/скрипт;
  - кто‑то сравнил две даты с разными «скрытыми» таймзонами, —
  ошибки становятся почти неизбежными.

Рабочая стратегия:

- **внутри системы** (в БД, логике) использовать UTC;
- **на границах** (UI, отчёты для пользователя) **переводить** в его локальную таймзону.

Мини‑упражнение:

- создайте два объекта: `naive = datetime(2020, 1, 1, 12, 0)` и `aware = datetime(2020, 1, 1, 12, 0, tzinfo=timezone.utc)`;  
  попробуйте их вычесть и посмотреть, какое исключение выдаст Python.

### 7.2. `zoneinfo`: таймзоны в stdlib

```python
from zoneinfo import ZoneInfo
from datetime import datetime

dt = datetime(2020, 5, 1, 10, 0, tzinfo=ZoneInfo("Europe/Moscow"))
```

`zoneinfo` использует базу IANA, учитывает переходы на летнее/зимнее время и исторические изменения.

Ключевая идея:

- не хранить таймзону как «+3 часа от UTC», а использовать **имя региона** (`Europe/Moscow`, `America/New_York`);
- потому что смещение может меняться (летнее/зимнее время, реформы времени).

Пример:

```python
from datetime import datetime
from zoneinfo import ZoneInfo

moscow = datetime(2020, 3, 29, 1, 0, tzinfo=ZoneInfo("Europe/Moscow"))
```

`zoneinfo` знает:

- были ли в этот момент переходы на/с летнего времени;
- какое фактическое смещение от UTC использовать.

Мини‑упражнение:

- создайте `datetime` для `Europe/Moscow` и `America/New_York` на один и тот же момент UTC;  
  переведите оба в `.isoformat()` и сравните, как выглядят смещения.

### 7.3. `python-dateutil` и удобный парсинг

```python
from dateutil.parser import isoparse

dt = isoparse("2020-01-01T12:34:56+03:00")
```

Пакет также умеет:

- относительные дельты (`relativedelta`);
- удобное добавление месяцев/лет с учётом календаря.

Почему это удобно:

- стандартный `datetime.strptime` требует жёсткого формата (`"%Y-%m-%dT%H:%M:%S%z"`);
- `isoparse` понимает «нормальные» ISO‑строки «из коробки»:
  - `2020-01-01T12:34:56Z`;
  - `2020-01-01 12:34:56+03:00`;
  - и т.п.

`relativedelta`:

- в отличие от обычного `timedelta`, учитывает длину месяцев и годы:
  - `+1 месяц` от января = февраль;
  - корректно ведёт себя при сложных комбинациях (год+месяц+день).

Мини‑упражнение:

- попробуйте с `relativedelta` добавлять 1 месяц к дате «31 января» и посмотрите, как библиотека решает неоднозначности.

### 7.4. `pendulum` / `arrow`: альтернативы с удобным API

`pendulum` и `arrow` предлагают более «человечный» API:

```python
import pendulum

dt = pendulum.now("Europe/Moscow")
print(dt.add(days=3).to_iso8601_string())
```

Основное преимущество:

- API, который «читается как английский текст»:
  - `now("Europe/Moscow")`;
  - `add(days=3)`;
  - `subtract(months=1)`;
  - `in_timezone("UTC")`.

Минус:

- это внешние зависимости, за которыми нужно следить (версии, поддержка);
- перед выбором библиотеки полезно оценить, достаточно ли вам стандартного `datetime` + `zoneinfo`.

---

### 7.5. Запомните по §123

- Никогда не смешивайте «наивные» и timezone‑aware даты без явного приведения.
- Для большинства приложений достаточно: `datetime` + `zoneinfo` + аккуратное хранение в UTC.
- При работе с пользовательским временем (календарь, расписание) учитывайте таймзону пользователя и переходы DST.
