## 10. Валидация и схемы (§126)

### 10.1. Ещё раз `pydantic`

Pydantic удобно использовать:

- на границе API (FastAPI и др.);
- для конфигов и настроек;
- для обмена данными между сервисами.

Сильная сторона — **статически описанная структура** и автоматическая валидация при создании объектов.

Если коротко сравнить инструменты из этого раздела:

- **Pydantic** — классы‑модели с аннотациями типов, очень удобен в Python‑коде;
- **Marshmallow** — декларативные схемы, хорошо дружит с веб‑фреймворками;
- **Cerberus** — схемы‑словари, просто и понятно для конфигов;
- **JSONSchema** — стандарт, который понимают разные языки и сервисы.

---

### 10.2. `marshmallow`

```python
from marshmallow import Schema, fields

class UserSchema(Schema):
    id = fields.Int(required=True)
    name = fields.Str(required=True)

schema = UserSchema()
data = schema.load({"id": "1", "name": "Alice"})  # приведёт к int при необходимости
dumped = schema.dump(data)
```

Marshmallow гибок и хорошо интегрируется с многими веб‑фреймворками.

---

### 10.3. `cerberus`

`cerberus` валидирует словари против словарей‑схем:

```python
from cerberus import Validator

schema = {"age": {"type": "integer", "min": 0}}
v = Validator(schema)
print(v.validate({"age": 10}))  # True
print(v.errors)
```

Подходит для простых случаев, когда не нужны классы‑модели.

---

### 10.4. `jsonschema`

`jsonschema` проверяет JSON‑документы на соответствие **JSON Schema**:

- удобно для контрактов API;
- стандарт понятен не только Python‑миру (есть реализации в других языках).

---

### 10.5. Запомните по §126

- Валидация и описание схемы должны быть **в одном месте**, а не размазаны по коду `if`‑ами.
- Выбор инструмента зависит от стиля проекта: Pydantic хорош для «питоничного» кода, JSONSchema — для контрактов между разными сервисами/языками.
- Не бойтесь вводить «слой схем» — это сильно упрощает сопровождение и отладку.
