## 2. Базы данных (§118)

### 2.0. Общая идея

Библиотеки для БД решают три большие задачи:

1. **Соединение и запросы** (драйвер: `sqlite3`, `asyncpg`, `psycopg2` и т.п.).
2. **Маппинг на объекты** (ORM / Core: `SQLAlchemy`).
3. **Эволюция схемы** (миграции: `alembic`).

Главные тонкости:

- не держите **соединения открытыми дольше нужного**;
- понимайте, где ORM делает **ленивые запросы** (lazy loading, N+1);
- миграции — не «бонус», а необходимый инструмент управления схемой.

Если объяснить бытовой аналогией:

- **драйвер** (sqlite3/asyncpg) — это как «чистый SQL‑терминал»: вы открываете дверь в БД и руками говорите ей `SELECT ...`;
- **ORM/SQLAlchemy** — это как «переводчик» между таблицами и Python‑классами: вы говорите «создай пользователя», а он сам пишет правильный SQL;
- **миграции Alembic** — это «история ремонта квартиры»:  
  было 2 комнаты → сделали перепланировку → добавили ещё одну стену.  
  В любой момент вы знаете, **как прийти к нужному состоянию** и можете применить те же шаги на другой БД.

### 2.1. `sqlite3`: встроенная БД

```python
import sqlite3

conn = sqlite3.connect("example.db")
try:
    cur = conn.cursor()
    cur.execute("CREATE TABLE IF NOT EXISTS users (id INTEGER PRIMARY KEY, name TEXT)")
    cur.executemany(
        "INSERT INTO users(name) VALUES (?)",
        [("Alice",), ("Bob",)],
    )
    conn.commit()
finally:
    conn.close()
```

**Тонкости:**

- SQLite рассчитан на **одновременного писателя** (одна запись за раз); чтений может быть много.
- При массе вставок используйте **`executemany`** — это гораздо быстрее, чем вызывать `execute` в цикле.
- Для более безопасного параллелизма включают **WAL‑режим**:

```python
conn.execute("PRAGMA journal_mode=WAL;")
```

Разберём этот пример «для мозга, который боится БД»:

1. `sqlite3.connect("example.db")`:
   - если файла `example.db` нет — SQLite сам создаёт его;
   - `conn` — это **соединение** с БД (как открытый файл).
2. `conn.cursor()`:
   - `cursor` — как «ручка», которой вы пишете команды в БД и читаете ответы;
   - через него выполняются SQL‑запросы.
3. `CREATE TABLE IF NOT EXISTS ...`:
   - создаёт таблицу `users`, **если её ещё нет**;
   - `id INTEGER PRIMARY KEY` — числовой идентификатор, уникальный для каждой строки;
   - `name TEXT` — обычная текстовая колонка.
4. `executemany("INSERT ...", [("Alice",), ("Bob",)])`:
   - запускает один и тот же запрос **несколько раз**, подставляя разные значения;
   - это эффективнее, чем в цикле `for name in ...: execute("INSERT ...", (name,))`.
5. `conn.commit()`:
   - говорит БД: «всё, что мы сделали внутри этого соединения, **подтверди и запиши навсегда**»;
   - без commit изменения могут оказаться не сохранены (или откатиться при ошибке).
6. Блок `try/finally`:
   - `finally: conn.close()` гарантирует, что соединение **закроется даже при ошибке**;
   - это важно: открытые соединения — это ресурсы (файловые дескрипторы, блокировки).

Мини‑упражнение:

- измените код так, чтобы прочитать все строки из `users` и вывести их;
- попробуйте убрать `conn.commit()` и посмотреть, будут ли данные доступны в новом соединении.

#### 2.1a. Что значит «один writer» в SQLite

SQLite хранит данные в **одном файле**. Это удобно, но создаёт ограничения:

- одновременно **много процессов могут читать** файл (как PDF);
- но **записывать** по‑настоящему может только один (иначе файл может повредиться).

Отсюда следствия:

- если вы часто пишете в SQLite из разных потоков/процессов, вы можете получать блокировки;
- для большей устойчивости включают WAL‑режим (`journal_mode=WAL`), который:
  - разделяет журнал изменений и основной файл;
  - делает одновременное чтение/запись безопаснее.

Для учебных и небольших утилит SQLite идеален. Для многопользовательских веб‑сервисов чаще берут полноценные СУБД (PostgreSQL, MySQL и т.п.).

### 2.2. `SQLAlchemy`: ORM и Core

`SQLAlchemy` даёт:

- **Core**: явное составление SQL через выражения;
- **ORM**: маппинг таблиц на классы.

Минимальный пример ORM:

```python
from sqlalchemy import create_engine, Column, Integer, String
from sqlalchemy.orm import declarative_base, sessionmaker

engine = create_engine("sqlite:///example.db", echo=False, future=True)
Base = declarative_base()

class User(Base):
    __tablename__ = "users"
    id = Column(Integer, primary_key=True)
    name = Column(String, nullable=False)

SessionLocal = sessionmaker(bind=engine, expire_on_commit=False)

def get_user(user_id: int) -> User | None:
    with SessionLocal() as session:
        return session.get(User, user_id)
```

**Что здесь происходит шаг за шагом:**

1. `engine = create_engine("sqlite:///example.db", ...)`:
   - создаёт объект «движка» — соединитель с БД;
   - строка `"sqlite:///example.db"` говорит: используй SQLite и файл `example.db`.
2. `Base = declarative_base()`:
   - `Base` — «родительский» класс для всех наших моделей (таблиц);
   - через него SQLAlchemy понимает, какие классы нужно привязать к БД.
3. Класс `User(Base)`:
   - `__tablename__ = "users"` — имя таблицы в БД;
   - `id = Column(Integer, primary_key=True)` — колонка `id` (целое, первичный ключ);
   - `name = Column(String, nullable=False)` — колонка `name` (строка, не может быть `NULL`).
4. `SessionLocal = sessionmaker(bind=engine, expire_on_commit=False)`:
   - `sessionmaker` — «фабрика» сессий;
   - `SessionLocal()` каждый раз создаёт **новую сессию** (как отдельное соединение/контекст работы с БД).
5. `with SessionLocal() as session:`:
   - открывает сессию;
   - в конце блока сессия **закроется автоматически** (и соединение вернётся в пул).
6. `session.get(User, user_id)`:
   - генерирует SQL вида `SELECT * FROM users WHERE id = :id`;
   - возвращает объект `User` или `None`.

**Тонкости:**

- **Жизненный цикл сессии**: каждая сессия — это «мелкий рабочий процесс»:
  - открыть → сделать несколько запросов → зафиксировать/откатить → закрыть;
  - не держите одну сессию «навсегда» (например, на весь срок жизни приложения).
- **Lazy loading**:
  - если у `User` есть связь `posts`, обращение к `user.posts` может вызвать **дополнительный запрос**;
  - если вы в цикле проходите по 100 пользователям и у каждого читаете `user.posts`, получите **N+1 запрос**;
  - лечится «жадной загрузкой» (`selectinload`, `joinedload`) и вниманием к профилировщику запросов.
- В SQLAlchemy 2.x рекомендован новый стиль (через `select()` и `session.execute(select(User)...)`), старый `session.query()` считают «устаревающим стилем», но он пока жив.

Мини‑упражнение:

- попробуйте добавить к `User` поле `email` и сгенерировать SQL для создания новой таблицы через `Base.metadata.create_all(engine)`;
- напишите функцию `create_user(name: str)`, которая создаёт пользователя, вызывает `session.add(user)`, `session.commit()` и возвращает созданный объект.

### 2.3. `asyncpg`: быстрый async PostgreSQL

```python
import asyncpg
import asyncio

async def main():
    conn = await asyncpg.connect("postgresql://user:pass@localhost:5432/dbname")
    try:
        rows = await conn.fetch("SELECT id, name FROM users")
        for r in rows:
            print(r["id"], r["name"])
    finally:
        await conn.close()

asyncio.run(main())
```

Чаще используют **пул соединений**:

```python
async def main():
    pool = await asyncpg.create_pool(dsn="postgresql://user:pass@localhost:5432/dbname")
    async with pool.acquire() as conn:
        await conn.execute("INSERT INTO logs(message) VALUES($1)", "hello")
```

**Тонкости:**

- `asyncpg` — **только async**, он не будет работать из обычных sync‑функций без event loop.
- Важно правильно настроить **размер пула** (слишком маленький — очередь, слишком большой — нагрузка на БД).

Чтобы не бояться, разберём пример с пулом:

1. `create_pool(...)`:
   - создаёт **набор соединений** к PostgreSQL (например, от 1 до 10 штук);
   - pool управляет тем, кому какое соединение выдать.
2. `async with pool.acquire() as conn:`:
   - берёт одно свободное соединение из пула;
   - внутри блока вы можете выполнять несколько запросов;
   - при выходе из блока соединение автоматически **возвращается** в пул.

Если пула нет и каждое соединение создаётся отдельно:

- на высоких нагрузках вы тратите много времени на «рукопожатие» с БД;
- рискуете открыть слишком много соединений, которые БД не потянет.

Мини‑упражнение:

- напишите функцию, которая в цикле делает 10 вставок в таблицу `logs` через пул, и измерьте общее время;
- затем попробуйте делать `asyncpg.connect`/`close` внутри каждой итерации и сравните (будет медленнее).

### 2.4. `alembic`: миграции для SQLAlchemy

`Alembic` отслеживает изменения моделей и генерирует миграции:

- `alembic revision --autogenerate -m "add users table"` → создаёт скрипт миграции;
- `alembic upgrade head` → применяет все незапущенные миграции.

**Тонкости:**

- Не полагайтесь слепо на `--autogenerate`: всегда просматривайте скрипт миграции.
- Не меняйте руками существующие миграции на проде — лучше создать новую «фиксирующую» миграцию.

Представьте, что у вас есть проект, который живёт несколько лет:

- в начале у вас была только таблица `users`;
- через год вы добавили таблицу `orders`;
- ещё через год — поле `phone` в `users`.

Если вы не ведёте миграции:

- на новом окружении (стейдж/прод) вам придётся **вручную вспоминать** все изменения и в каком порядке их применять;
- легко допустить ошибку и получить расхождение схем между окружениями.

Alembic делает по сути следующее:

1. Смотрит на ваши модели (классы SQLAlchemy) и сравнивает их с текущей схемой БД.
2. Генерирует скрипт, который «доводит» БД до нужного состояния.
3. Позволяет хранить эти скрипты в Git и повторно применять на других БД.

Мини‑упражнение (понятийно, без запуска):

- мысленно представьте, какие миграции понадобятся, если:
  - сначала есть только `User(id, name)`;
  - потом вы меняете `name` на `full_name`;
  - затем добавляете таблицу `Address` с внешним ключом на `User`.

### 2.5. Запомните по §118

- Поддерживайте **чистый цикл жизни соединений и сессий**: открыли → использовали → вернули в пул/закрыли.
- Для сложных приложений почти всегда лучше иметь **миграции** (Alembic), чем «создать таблицы руками в БД».
- Следите за **N+1 запросами** и lazy loading в ORM, используйте инструменты профилирования запросов.
