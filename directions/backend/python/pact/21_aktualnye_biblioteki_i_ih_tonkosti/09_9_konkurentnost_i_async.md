## 9. Конкурентность и async (§125)

### 9.1. `asyncio`: основы

```python
import asyncio

async def worker(i: int):
    await asyncio.sleep(1)
    print(f"Done {i}")

async def main():
    tasks = [asyncio.create_task(worker(i)) for i in range(5)]
    await asyncio.gather(*tasks)

asyncio.run(main())
```

**Важные моменты:**

- `async def` → корутина, которую нужно `await`‑ить;
- `asyncio.run` создаёт event loop и запускает «главную» корутину;
- `asyncio.gather` позволяет запускать несколько корутин параллельно (в одном потоке, но с неблокирующим I/O).

Разбор примера:

1. `async def worker(i: int):`:
   - определяет задачу, которая:
     - ждёт 1 секунду (`await asyncio.sleep(1)`);
     - печатает `Done i`.
2. В `main`:
   - создаём список задач: `asyncio.create_task(worker(i))` для `i` от 0 до 4;
   - `asyncio.gather(*tasks)` ждёт, пока все они завершатся.
3. `asyncio.run(main())`:
   - создаёт event loop;
   - запускает `main`;
   - корректно закрывает event loop после завершения.

Важное наблюдение:

- все 5 задач выполняются **примерно за 1 секунду**, а не за 5;
- потому что, пока одна корутина «спит», event loop выполняет другие.

Мини‑упражнение:

- замените `await asyncio.sleep(1)` на `time.sleep(1)` и посмотрите, как изменится время выполнения и поведение (подсказка: event loop «встанет колом»).

### 9.2. Async‑драйверы: `aiohttp`, `asyncpg`, `aioredis`

Общая идея:

- драйвер предоставляет async‑функции (`await client.get(...)`, `await conn.fetch(...)`);
- под капотом — неблокирующий I/O, интеграция с event loop.

Тонкость: многие драйверы используют **свои пулы соединений**, поэтому важно закрывать их корректно (через `async with` или явные методы `close`).

---

### 9.3. `anyio` и `trio`

`anyio` — абстракция поверх `asyncio`/`trio`, дающая единый API:

- задачи (`create_task_group`);
- тайм‑ауты;
- синхронизацию.

`trio` — альтернативный async‑фреймворк с акцентом на **structured concurrency** (все задачи группируются и корректно отменяются).

Иногда библиотеки объявляют поддержку `anyio`/`trio`, что делает код переносимым между разными event loop.

---

### 9.4. Не смешивайте sync/async без `run_in_executor`

Классическая ошибка:

- внутри `async def` вызывать **блокирующие** операции (например, `requests.get`, `open().read()` больших файлов);
- в результате event loop блокируется, преимущества async теряются.

Решение: оборачивать блокирующие операции в `run_in_executor` или использовать async‑аналоги.

---

### 9.5. Запомните по §125

- Async‑код полезен, когда вы много ждёте сети/диска, а не когда упираетесь в CPU.
- Любая блокирующая операция в async‑функции должна быть либо заменена на async‑аналог, либо вынесена в пул потоков.
- Не усложняйте: если приложение маленькое и не упирается в I/O, часто проще остаться на sync‑подходе.
