## 1. HTTP и сети (§117)

### 1.0. Общая идея

HTTP‑клиентские библиотеки в Python делятся на две большие группы:

- **синхронные** — блокируют поток до завершения запроса (`requests`, sync‑режим `httpx`);
- **асинхронные** — отдают управление event loop, пока «ждут сеть» (`httpx` (async), `aiohttp`).

В реальных проектах типичный путь такой:

1. Начинаете с **`requests`** — он проще, чем всё остальное.
2. Когда нужно **много параллельных запросов** (поллинг, микросервисы, большое количество внешних API) — переходите на **`httpx`/`aiohttp`** и async.

Если объяснить совсем на пальцах:

- **синхронный запрос** — как если бы вы позвонили по телефону и держали трубку, пока вам диктуют ответ. Весь этот момент вы не можете параллельно разговаривать с кем‑то ещё;
- **асинхронный запрос** — вы отправили запрос и пока ждёте ответ, успели сделать ещё 10 дел. Как мессенджер: сообщения уходят/приходят, а вы не «зависаете» на одном диалоге.

### 1.1. `requests`: базовый HTTP‑клиент (sync)

```python
import requests

resp = requests.get(
    "https://api.github.com/repos/org/repo",
    timeout=5,  # ВАЖНО: не забываем таймаут
)
resp.raise_for_status()  # выбросит HTTPError при 4xx/5xx
data = resp.json()
```

**Ключевые идеи:**

- **Всегда задавайте `timeout`** — без него запрос может висеть «вечно», блокируя поток.
- Используйте **`Session`** для повторных запросов к одному хосту: keep‑alive, куки, общие заголовки.

```python
session = requests.Session()
session.headers.update({"User-Agent": "my-app/1.0"})

resp = session.get("https://api.example.com/items", timeout=5)
```

**Тонкости и подводные камни:**

- Методы `get/post/...` на верхнем уровне (`requests.get`) создают отдельный сеанс под капотом → чаще используйте `Session`.
- `resp.json()` может бросить `ValueError`/`JSONDecodeError`, если ответ не JSON → для ненадёжных API проверяйте `Content-Type` и оборачивайте в `try`.
- `stream=True` позволяет читать тело по частям (для больших файлов), но не забывайте `resp.close()`/контекстный менеджер:

```python
with session.get(url, stream=True, timeout=30) as r:
    r.raise_for_status()
    for chunk in r.iter_content(chunk_size=8192):
        ...
```

Чуть подробнее разберём, **что именно происходит** в базовом примере:

1. `requests.get(...)`:
   - создаёт под капотом объект сессии (или использует общий пул, но без ваших настроек);
   - открывает TCP‑соединение к серверу (если его ещё нет);
   - отправляет HTTP‑запрос и ждёт ответ;
   - возвращает объект `Response`.
2. `timeout=5`:
   - говорит: «если в сумме ожидание ответа заняло больше 5 секунд — **упади с ошибкой**»;
   - без таймаута вы можете зависнуть, если сервер «умер», а соединение формально не закрыто.
3. `resp.raise_for_status()`:
   - если код ответа 200–299 — ничего не делает;
   - если 4xx/5xx — поднимает исключение `HTTPError`.  
   Это удобно, потому что вы **явно** обрабатываете неуспех (через `try/except`), а не надеетесь, что всё «как‑то пронесёт».
4. `resp.json()`:
   - пытается распарсить тело ответа как JSON;
   - если это невалидный JSON, вылетает `JSONDecodeError` (наследник `ValueError`).

Мини‑упражнение:

- поменяйте URL на заведомо несуществующий домен и посмотрите, какое исключение получаете;
- попробуйте получить страницу HTML и вызвать `resp.json()` — посмотрите, какую ошибку покажет Python и в какой строке.

#### 1.1a. Сессии в `requests` на пальцах

Почему вообще нужен `Session`, а не всегда просто `requests.get`?

- Каждый HTTP‑запрос — это:
  - поиск DNS‑имени;
  - установка TCP‑соединения;
  - (иногда) TLS‑рукопожатие.
- Если вы каждый раз делаете это заново, это **дорого по времени** и создаёт ненужную нагрузку.

`Session`:

- переиспользует уже открытое соединение;
- может хранить:
  - куки (для авторизации);
  - общие заголовки (User‑Agent, токен и т.п.).

Пример типичного использования:

```python
session = requests.Session()
session.headers.update({"Authorization": f"Bearer {token}"})

def get_user(user_id: int) -> dict:
    resp = session.get(f"https://api.example.com/users/{user_id}", timeout=5)
    resp.raise_for_status()
    return resp.json()
```

Так вы не копируете заголовки/URL‑базу по всему коду, а держите их в одном месте.

---

#### 1.1b. Типичные ошибки при работе с `requests`

1. **Отсутствие таймаута:**
   - код «работает на локалке», но в проде сервис иногда зависает, и один подвисший запрос может повесить воркер.
2. **Игнорирование кода ответа:**
   - просто берут `resp.json()`/`resp.text`, не проверяя статус;
   - в итоге 500‑ошибки сервера превращаются в «какие‑то странные словари».
3. **Логирование секретов:**
   - при отладке иногда логируют весь `resp.request.headers`/`resp.request.body` и случайно пишут в логи токены/пароли.  
   Следите, что вы печатаете в логи.

### 1.2. `httpx`: современный HTTP (sync + async)

**Зачем нужен, если есть `requests`:**

- поддерживает **и sync, и async** API;
- умеет **HTTP/2**, прокси, тестовые клиенты для FastAPI/Starlette;
- во многом совместим по идеям с `requests`, но API немного отличается.

Sync‑пример:

```python
import httpx

with httpx.Client(timeout=5) as client:
    resp = client.get("https://api.example.com/items")
    resp.raise_for_status()
    data = resp.json()
```

Async‑пример:

```python
import httpx
import asyncio

async def main():
    async with httpx.AsyncClient(timeout=5) as client:
        resp = await client.get("https://api.example.com/items")
        resp.raise_for_status()
        return resp.json()

asyncio.run(main())
```

**Тонкости:**

- В async‑варианте всегда используйте `async with` → это закрывает соединения корректно.
- В `httpx` есть полезные механизмы:
  - `follow_redirects=True` — автоматические редиректы;
  - `limits=httpx.Limits(max_connections=100, max_keepalive_connections=20)` — ограничения пула соединений;
  - `transport` — можно подменить транспорт (для тестов/моков).

Дополнительные моменты:

- `httpx` умеет **тестовые клиенты** для FastAPI/Starlette: можно вызывать эндпоинты без реального HTTP‑сервера (полезно в тестах).
- В async‑режиме он естественно встраивается в `asyncio`‑приложения и не блокирует event loop.

Мини‑упражнение:

- перепишите простой код на `requests` с GET‑запросом на `httpx.Client` (sync);
- затем напишите такой же запрос в async‑виде (`AsyncClient`) и сравните код.

### 1.3. `aiohttp`: async‑клиент и мини‑сервер

`aiohttp` — «старичок» мира async HTTP в Python:

- **клиент** (`aiohttp.ClientSession`) — отправка запросов;
- **сервер** (`aiohttp.web`) — лёгкий веб‑фреймворк на async.

Клиент:

```python
import aiohttp
import asyncio

async def fetch(session, url):
    async with session.get(url, timeout=5) as resp:
        resp.raise_for_status()
        return await resp.json()

async def main():
    async with aiohttp.ClientSession() as session:
        data = await fetch(session, "https://api.example.com/items")
        print(data)

asyncio.run(main())
```

**Тонкости:**

- как и в `httpx`, важно **делиться одним `ClientSession`** между запросами;
- `timeout` по умолчанию не бесконечный, но лучше задавать явно;
- нельзя вызывать `time.sleep` / блокирующий I/O внутри async‑функций — это заморозит event loop, используйте `asyncio.sleep` и async‑библиотеки.

---

#### 1.3a. Как читать пример с `aiohttp` по шагам

Разберём пример буквально построчно:

1. `async def fetch(session, url):`
   - объявляем **асинхронную функцию** (корутину);
   - она не выполняется сразу, а только создаёт объект «плана» выполнения.
2. `async with session.get(url, timeout=5) as resp:`
   - `session.get(...)` возвращает специальный объект‑контекстный менеджер, который **откроет HTTP‑соединение**;
   - `async with` говорит: «войти в контекст» асинхронно (с `await` внутри);
   - внутри блока у нас есть `resp` — объект ответа.
3. `resp.raise_for_status()` — как и в `requests`, проверяем код ответа и бросаем `HTTPError` при 4xx/5xx.
4. `return await resp.json()`:
   - `resp.json()` — **асинхронный метод** (поэтому `await`);
   - он читает тело ответа и парсит его как JSON.

`main()`:

1. `async with aiohttp.ClientSession() as session:` — создаём клиентскую сессию, которая:
   - переиспользует соединения;
   - обеспечивает корректное закрытие всех соединений при выходе из блока.
2. `data = await fetch(session, "...")` — вызываем нашу корутину и **обязательно** пишем `await`, иначе она не выполнится.
3. `asyncio.run(main())` — точка входа: создаёт event loop, запускает `main` и ждёт его завершения.

Мини‑упражнение:

- внутри `main` сделайте список `urls = ["https://httpbin.org/get", "https://api.github.com"]`  
  и запустите несколько `fetch` параллельно через `asyncio.gather`;
- добавьте искусственную задержку `await asyncio.sleep(1)` внутри `fetch` и посмотрите, как меняется общее время выполнения.

---

#### 1.3b. Когда выбирать `requests`, когда `httpx`, когда `aiohttp`

- **`requests`**:
  - подходит для простых скриптов, однократных запросов, CLI‑утилит;
  - проще в освоении, огромная база примеров в интернете;
  - плохо подходит, если нужно **очень много одновременных запросов**.
- **`httpx`**:
  - если вы уже знаете `requests`, но хотите перейти на **современный клиент** с поддержкой async/HTTP‑2;
  - удобно в проектах на FastAPI/Starlette;
  - один и тот же пакет закрывает и sync, и async‑нужды.
- **`aiohttp`**:
  - когда вам нужен не только клиент, но и **микросервис/мини‑сервер** на async;
  - когда вы работаете в уже существующей экосистеме `aiohttp`.

Общее правило: **начинайте с самого простого, что решает вашу задачу**.  
Если вы пишете маленький скрипт — не спешите в async‑мир. Если упёрлись в ограничения sync‑подхода (много одновременных запросов, нужен настоящий неблокирующий I/O) — переходите на `httpx`/`aiohttp`.

---

### 1.4. Краткое резюме по HTTP

- Любой HTTP‑запрос — это сеть, а сеть **ломается**: обязательно думайте про таймауты, повторы (`retry`) и обработку ошибок.
- Сессии (`Session`, `ClientSession`) экономят время и ресурсы, а также позволяют удобно хранить общее состояние (куки, заголовки).
- Async‑клиенты (`httpx`, `aiohttp`) дают выигрыш, когда запросов **много** и они часто ждут ответов; но они требуют понимания `async`/`await` и event loop.

---

### 1.4. `urllib3`: низкоуровневый HTTP

`urllib3` — низкоуровневая библиотека:

- ей пользуются сами `requests`;
- напрямую чаще используют, когда нужен более тонкий контроль над пулами соединений, сертификатами и т.п.

В простых приложениях хватает `requests`/`httpx`. Полезно просто знать, что `urllib3` — «кирпичик» под капотом.

---

### 1.5. Запомните по §117

- **Таймауты не опциональны.** В проде всегда явно задавайте `timeout` и продумывайте политику повторов (см. `tenacity`).
- Для множества запросов к одному хосту используйте **сессии** (`requests.Session`, `httpx.Client`, `aiohttp.ClientSession`).
- При высокой нагрузке и необходимости масштабирования — переходите на **async‑подход** (`httpx`/`aiohttp`), но не смешивайте sync и async вперемешку.
