## 5. Тестирование (§121)

### 5.1. `pytest`: основной инструмент

Базовый тест:

```python
# test_example.py

def inc(x: int) -> int:
    return x + 1

def test_inc():
    assert inc(1) == 2
```

Запуск: `pytest`.

Фикстура:

```python
import pytest

@pytest.fixture
def sample_user():
    return {"id": 1, "name": "Alice"}

def test_user(sample_user):
    assert sample_user["name"] == "Alice"
```

**Тонкости:**

- Общие фикстуры выносят в `conftest.py` — это «скрытый» модуль, где `pytest` ищет фикстуры по умолчанию.
- Не злоупотребляйте сложной иерархией фикстур — лучше несколько простых, чем одна «магическая» всё‑в‑одном.

Разберём базовый тест:

1. Функция `inc` — обычный код, который вы хотите проверить.
2. Функция `test_inc`:
   - название начинается с `test_` → `pytest` **автоматически** найдёт её и выполнит;
   - `assert inc(1) == 2` — проверка: если условие ложное, тест считается упавшим.

Отличия от «ручных проверок через `print`»:

- `pytest` сам:
  - запускает все тесты;
  - показывает, какие упали и почему;
  - даёт понятный diff значений в `assert`.

Фикстуры:

- `@pytest.fixture` помечает функцию как «подготовку окружения»;
- имя фикстуры (`sample_user`) потом просто указываете параметром в тесте;
- `pytest` сам поймёт, что перед запуском теста нужно вызвать фикстуру и подставить результат.

Мини‑упражнение:

- добавьте ещё один тест, который проверяет, что `inc(-1) == 0`;
- измените `inc` так, чтобы он содержал баг (например, `return x + 2`) и посмотрите, как `pytest` покажет падение теста.

### 5.2. Async‑тесты: `pytest-asyncio`

```python
import pytest

@pytest.mark.asyncio
async def test_async_function():
    result = await some_async_call()
    assert result == 42
```

**Важно:** следите, чтобы в async‑тестах вы не вызывали блокирующие функции (например, обычные HTTP‑клиенты без `run_in_executor`).

Что делает `@pytest.mark.asyncio`:

- говорит `pytest`: «эта функция — **асинхронный тест**, её нужно запускать через event loop»;
- под капотом `pytest-asyncio` создаёт event loop, выполняет ваш `async def` и ждёт его завершения.

Типичная ошибка:

- внутри async‑теста вызвать `requests.get` (sync) вместо async‑клиента;
- такой вызов **заблокирует** event loop и убьёт смысл async‑подхода.

### 5.3. Покрытие: `pytest-cov`

Установка плагина позволяет получить отчёт покрытия:

- `pytest --cov=package_name --cov-report=term-missing`.

Не стремитесь к «100% ради 100%», важнее **качество** тестов.

По шагам:

- `--cov=package_name` говорит: «отслеживай выполнение кода в этом пакете»;
- `--cov-report=term-missing` выводит в терминал файл/строки, которые:
  - ни разу не выполнялись при запуске тестов;
  - дают подсказку: «вот сюда тесты не доходят».

Важно понимать:

- высокая «покрытость» ≠ хорошие тесты;
- но **очень низкая** покрытость почти всегда значит, что большая часть кода вообще не проверяется.

Мини‑упражнение:

- запустите `pytest` с `--cov` и посмотрите, какие файлы почти не покрыты;
- выберите одну функцию и добавьте для неё простой тест — проверьте, как изменится отчёт.

### 5.4. `hypothesis`: property‑based testing

```python
from hypothesis import given, strategies as st

def reverse_twice(xs: list[int]) -> list[int]:
    return list(reversed(list(reversed(xs))))

@given(st.lists(st.integers()))
def test_reverse_twice(xs):
    assert reverse_twice(xs) == xs
```

`hypothesis` сам генерирует сотни вариантов входных данных, включая крайние кейсы (пустые списки, большие числа и т.п.).

Идея:

- вместо того, чтобы вручную придумывать 3–5 примеров (`xs = []`, `xs = [1, 2]`, `xs = [1, 2, 3, 4]`),
- вы описываете **свойство**, которое должно выполняться для всех допустимых входов:
  - пример: «если перевернуть список дважды, он не изменится»;
  - пример: «сортировка результата даёт отсортированный список».

`@given(st.lists(st.integers()))`:

- говорит: «подставляй в тест всевозможные списки целых чисел»;
- `hypothesis`:
  - сам придумывает простые случаи (пустой список, один элемент);
  - сложные случаи (повторы, отрицательные числа, длинные списки);
  - если находит контрпример — сохраняет **минимальный** неработающий вход.

Мини‑упражнение:

- напишите функцию `add_one(x: int) -> int` и свойство: `add_one(x) > x` для всех `x`;  
  попробуйте сломать реализацию и посмотреть, какой пример найдёт `hypothesis`.

### 5.5. Фейковые данные и моки HTTP/времени

- **`factory_boy` + `Faker`** — фабрики объектов для тестов (пользователи, заказы и т.п.).
- **`responses`** (для `requests`) и **`respx`** (для `httpx`) — моки HTTP:

```python
import requests
import responses

@responses.activate
def test_api_call():
    responses.add(
        responses.GET,
        "https://api.example.com/items",
        json={"items": []},
        status=200,
    )
    resp = requests.get("https://api.example.com/items")
    assert resp.json() == {"items": []}
```

- **`freezegun`** — фиксация времени:

```python
from freezegun import freeze_time
from datetime import datetime

@freeze_time("2020-01-01")
def test_now():
    assert datetime.now().year == 2020
```

Как этим пользоваться «на практике»:

- `factory_boy` + `Faker`:
  - позволяют создавать «реалистичные» объекты (пользователи, заказы) с правдоподобными данными;
  - вы не придумываете руками `user = {"name": "test", "email": "test@test.test"}`, а получаете разнообразные, но контролируемые значения.
- `responses` / `respx`:
  - перехватывают HTTP‑запросы в тестах и подставляют **заранее описанные ответы**;
  - ваши тесты не ходят в реальный интернет/службы;
  - вы контролируете, что вернётся (200, 500, таймаут и т.п.).
- `freezegun`:
  - фиксирует время «как будто сейчас 2020‑01‑01»;
  - избавляет от нестабильных тестов, которые зависят от текущей даты/времени.

Мини‑упражнение:

- напишите функцию, которая пишет в лог текущий год, и тест с `@freeze_time`, проверяющий, что строка содержит нужный год.

### 5.6. Запомните по §121

- Тесты должны быть **изолированными**: никакого общего глобального состояния, случайных запросов в сеть, «живых» БД без фикстур.
- Используйте плагины по делу: async‑тесты, покрытие, генерация данных, моки HTTP и времени.
- `pytest` мощен, но не превращайте его в «магический чёрный ящик» — лучше несколько явных шагов, чем непонятные глобальные фикстуры.
