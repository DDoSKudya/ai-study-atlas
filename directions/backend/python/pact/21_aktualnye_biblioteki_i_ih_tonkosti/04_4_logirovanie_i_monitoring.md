## 4. Логирование и мониторинг (§120)

### 4.1. `logging`: стандартное логирование

Минимальный пример:

```python
import logging

logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s %(levelname)s [%(name)s] %(message)s",
)

log = logging.getLogger(__name__)

log.info("Starting app", extra={"version": "1.0"})
```

**Ключевые идеи:**

- логгеры имеют **имена** (обычно `__name__`), через них строится иерархия;
- сообщения проходят через **handlers** (куда писать — файл, stdout, syslog и т.п.);
- форматирование можно настраивать через `Formatter` или `logging.config.dictConfig`.

Если совсем по‑простому:

- `logging` — это **не «принты на стероидах»**, а система, которая:
  - разделяет «что логируем» (logger) и «куда логируем» (handler);
  - добавляет к сообщению полезный контекст (время, уровень, имя модуля).

Разберём пример по шагам:

1. `logging.basicConfig(...)`:
   - настраивает **глобальную конфигурацию**:
     - `level=logging.INFO` — всё, что ниже INFO (DEBUG), по умолчанию не выводится;
     - `format="%(asctime)s %(levelname)s [%(name)s] %(message)s"` — шаблон строки:
       - `asctime` — время;
       - `levelname` — уровень (`INFO`, `ERROR` и т.п.);
       - `name` — имя логгера (обычно имя модуля);
       - `message` — текст, который вы передаёте в `log.info(...)`.
2. `log = logging.getLogger(__name__)`:
   - создаёт (или берёт существующий) логгер с именем **текущего модуля**;
   - это имя попадёт в `%(name)s` в формате.
3. `log.info("Starting app", extra={"version": "1.0"})`:
   - пишет сообщение уровня INFO;
   - `extra` добавляет в запись дополнительные поля (если форматтер умеет их подхватывать).

Мини‑упражнение:

- поменяйте уровень в `basicConfig` на `DEBUG` и добавьте `log.debug("Debug message")`;  
  посмотрите, когда эта строка появляется, а когда — нет.

### 4.2. `structlog`: структурированные логи

`structlog` помогает писать логи как **структуры данных**:

```python
import structlog

structlog.configure(
    processors=[
        structlog.processors.add_log_level,
        structlog.processors.TimeStamper(fmt="iso"),
        structlog.processors.JSONRenderer(),
    ],
)

log = structlog.get_logger()

log.info("user_logged_in", user_id=123, method="password")
```

Вывод — JSON, который удобно парсить централизованными системами логирования (ELK, Loki и т.п.).

Чем `structlog` отличается от «обычного» `logging`:

- в классическом `logging` вы чаще пишете **одну строку**:

  ```python
  log.info("User %s logged in from %s", user_id, ip)
  ```

- в `structlog` вы формируете **словарь** (событие + поля):

  ```python
  log.info("user_logged_in", user_id=user_id, ip=ip)
  ```

Дальше эти данные:

- превращаются в JSON;
- отправляются в систему логирования;
- легко фильтруются: «покажи все `user_logged_in` за сегодня от такого‑то IP».

Для чего это нужно:

- когда приложение маленькое — достаточно обычных строк;
- когда сервисов много и логи попадают в общую систему (ELK, Loki и т.п.), **структура** логов становится критичной.

### 4.3. `loguru`: удобный API логов

```python
from loguru import logger

logger.add("app.log", rotation="10 MB")

logger.info("Starting app")
logger.error("Something went wrong", exc_info=True)
```

Тонкость: при интеграции с существующим кодом на `logging` нужно аккуратно «перехватывать» стандартные логи, чтобы не дублировать сообщения.

`loguru` удобен тем, что:

- даёт **очень простой старт**:

  ```python
  from loguru import logger
  logger.add("app.log")
  logger.info("Hello")
  ```

- формат по умолчанию уже довольно информативный (время, уровень, модуль, строка кода).

Но в больших проектах:

- часто уже настроен `logging`;
- важно не дублировать одну и ту же запись в нескольких местах (и в `logging`, и в `loguru`).

Общее правило:

- либо вы строите систему вокруг `logging` + `structlog`;
- либо осознанно используете `loguru` как главную точку входа и аккуратно интегрируете его с `logging`.

### 4.4. `sentry-sdk`: ошибки в продакшене

```python
import sentry_sdk

sentry_sdk.init(
    dsn="https://key@sentry.io/project",
    traces_sample_rate=0.1,  # процент запросов с трассировкой
)
```

После инициализации необработанные исключения будут автоматически отправляться в Sentry, где вы увидите стек, переменные окружения, пользователя и т.п.

Sentry можно представить как:

- «чёрный ящик» для вашего приложения:
  - если в проде произошла ошибка, о которой никто не позаботился (`except` не поймал) — Sentry её «поднимет руку»;
  - вы увидите:
    - точный стек вызовов (по каким функциям ходил код);
    - значения некоторых переменных;
    - в веб‑приложениях — информацию о запросе/пользователе.

Важно:

- Sentry **не заменяет** логи, а дополняет их:
  - логи → хороши для понимания «как мы сюда пришли»;
  - Sentry → показывает конкретные исключения и их частоту.

Мини‑упражнение (мысленно или на тестовом проекте):

- инициализируйте Sentry в маленьком скрипте, намеренно вызовите исключение и посмотрите, как оно появится в интерфейсе Sentry (если у вас есть аккаунт).

### 4.5. Запомните по §120

- Логирование нужно не ради «галочки», а чтобы можно было **ответить на конкретные вопросы**: «что пошло не так?», «кто вызвал этот код?», «какой был вход?».
- Для продакшена всё чаще используют **структурированные логи** (JSON), а не произвольный текст.
- Инструменты вроде Sentry значительно ускоряют поиск и устранение ошибок, если их правильно настроить.
