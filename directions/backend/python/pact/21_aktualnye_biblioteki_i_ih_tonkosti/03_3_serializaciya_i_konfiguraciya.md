## 3. Сериализация и конфигурация (§119)

### 3.1. `json`: стандартная библиотека

```python
import json

data = {"id": 1, "name": "Alice"}
s = json.dumps(data, ensure_ascii=False, indent=2)
print(s)

loaded = json.loads(s)
```

**Ключевые параметры:**

- `ensure_ascii=False` — сохраняет не‑ASCII‑символы как есть (удобно для русского языка).
- `indent=2` — человекочитаемый формат.
- `default` — функция для сериализации нестандартных типов:

```python
from datetime import datetime

def default(o):
    if isinstance(o, datetime):
        return o.isoformat()
    raise TypeError

s = json.dumps({"ts": datetime.now()}, default=default)
```

Парсинг с обработкой ошибок:

```python
try:
    obj = json.loads(bad_json)
except json.JSONDecodeError as exc:
    print(exc.lineno, exc.colno, exc.msg)
```

Разберём `json.dumps`/`json.loads` «на пальцах»:

- `dumps` (dump **s**tring) — «запаковать Python‑объект в строку JSON»;
- `loads` (load **s**tring) — «распаковать JSON‑строку обратно в Python‑объект».

Пример:

- до `dumps`: у вас есть словарь `{"id": 1, "name": "Alice"}` — это **внутреннее** представление данных;
- после `dumps`: получаете строку `'{"id": 1, "name": "Alice"}'`, которую можно:
  - записать в файл;
  - отправить по сети;
  - сохранить где‑то ещё.

Параметры:

- `ensure_ascii=False`:
  - по умолчанию Python превращает русские буквы в `\u041f\u0440\u0438\u043c\u0435\u0440`;
  - с `False` вы увидите нормальный текст `"Пример"`;
  - для логов/конфигов на русском почти всегда хотите `False`.
- `indent=2`:
  - добавляет переносы строк и отступы;
  - удобно для человека, но немного увеличивает размер файла/ответа.

Про `default`:

- JSON «по умолчанию» не знает, что такое `datetime`, `Decimal` и другие сложные типы;
- `default` — это «план Б»:
  - если объект стандартный (dict/list/str/int/float/bool/None) — сериализуется сам;
  - если нет — вызывается `default(o)`, и вы должны вернуть что‑то сериализуемое (например, строку).

Парсинг с ошибками:

- `json.loads` ожидает **строку с JSON**;
- если там поломанный текст (запятая лишняя, кавычка не закрыта) — бросит `JSONDecodeError`;
- у исключения есть `lineno`, `colno`, `msg` — они помогают понять, где именно всё сломалось.

Мини‑упражнение:

- попробуйте сериализовать словарь с `datetime` **без** `default` и посмотреть, какую ошибку вы получите;
- затем добавьте `default` (как в примере) и убедитесь, что теперь всё работает.

### 3.2. YAML (PyYAML) и безопасность

```python
import yaml

with open("config.yml", "r", encoding="utf-8") as f:
    cfg = yaml.safe_load(f)
```

**Важно:** используйте **`safe_load`**, а не `load` — последний может выполнять произвольный код при загрузке специально подготовленных YAML‑файлов.

YAML удобен для конфигов, но:

- он сложнее JSON;
- может вести себя неожиданно (например, автоприведение типов).

Для важных систем разумно ограничить набор типов и придерживаться строгих схем.

Если сравнить JSON и YAML очень грубо:

- JSON:
  - строже и проще;
  - меньше «магии» (типы не угадываются автоматически);
  - хорош как формат обмена между сервисами.
- YAML:
  - более «человечный» на вид (комментарии, много вариантов записи);
  - но из‑за своей гибкости может вести себя неожиданно.

Почему `safe_load`:

- старый `yaml.load` мог, например, при определённых конструкциях создать объекты произвольных классов, вызвать их и т.п.;
- если вы грузите YAML, который вам прислал кто‑то из вне, это превращается в дыру безопасности;
- `safe_load` ограничивает набор типов до «безопасных» (dict, list, str, числа и т.д.).

Мини‑упражнение:

- создайте простой `config.yml` с полями `debug: true`, `database_url: "sqlite:///example.db"`;
- напишите маленький скрипт, который загружает YAML в dict и печатает его.

### 3.3. TOML (`tomllib`, `tomli`, `tomli-w`)

- В Python 3.11+ есть модуль `tomllib` (только **чтение** TOML).
- Для записи используют отдельные пакеты (`tomli-w`, `tomlkit` и т.п.).

```python
import tomllib

with open("pyproject.toml", "rb") as f:
    data = tomllib.load(f)
```

TOML часто используют для:

- `pyproject.toml` (метаданные проекта, зависимости);
- конфигов библиотек.

Почему TOML вообще появился:

- JSON неудобен для длинных конфигов (нет комментариев, всё в фигурных скобках);
- YAML слишком гибкий и иногда непредсказуемый;
- TOML старается быть:
  - строгим, как JSON;
  - но при этом удобным для человека (разделы `[tool.poetry]`, `[project]` и т.п.).

`tomllib` (stdlib, только чтение) и `tomli`/`tomli-w`:

- если вы на Python 3.11+ и хотите **только читать** TOML — используйте `tomllib`;
- если нужно писать TOML из Python‑кода — добавляйте сторонний пакет.

Мини‑упражнение:

- откройте свой `pyproject.toml` в любимом проекте и попробуйте через `tomllib` прочитать оттуда, например, имя проекта и список зависимостей.

### 3.4. `pydantic`: валидация и модели настроек

`pydantic` позволяет описать структуру данных через классы:

```python
from pydantic import BaseModel, ValidationError

class User(BaseModel):
    id: int
    name: str
    is_active: bool = True

try:
    user = User(id="1", name="Alice")
    # id будет приведён к int, если включены нужные настройки
except ValidationError as e:
    print(e.json())
```

Пример настроек из `.env`/переменных окружения:

```python
from pydantic import BaseSettings

class Settings(BaseSettings):
    db_url: str
    debug: bool = False

    class Config:
        env_file = ".env"

settings = Settings()
```

**Тонкости:**

- В новых версиях pydantic изменились детали API (особенно между v1 и v2) — всегда смотрите документацию под свою версию.
- Не забывайте включать строгие режимы, если вам не нужно неявное приведение типов (иначе `id="1"` станет `1`).

Разберём пример с `User`:

1. `class User(BaseModel):`:
   - вы описываете **форму данных**: какие поля есть и какие у них типы;
   - это **не** ORM‑модель, а именно схема валидации.
2. Поля:
   - `id: int` — поле `id` должно быть целым числом;
   - `name: str` — обязательная строка;
   - `is_active: bool = True` — булево поле с дефолтом `True`.
3. При создании `User(id="1", name="Alice")`:
   - pydantic смотрит на типы и **пытается привести строку `"1"` к `int`**;
   - если приведение не удаётся (например, `"abc"` → int), он кидает `ValidationError`.

`ValidationError`:

- содержит подробную информацию:
  - какое поле не прошло валидацию;
  - какой тип ожидался и что пришло;
  - можно вывести в JSON‑виде (`e.json()`).

`BaseSettings`:

- это специальный класс pydantic, который:
  - читает значения **из переменных окружения** (и `.env`, если задан `env_file`);
  - валидирует и приводит их к нужным типам.

Например:

- в `.env` написано `DEBUG=true`;
- в `Settings` поле `debug: bool = False`;
- при создании `Settings()` значение `true` будет прочитано из окружения и приведено к `True` (булеву).

Мини‑упражнение:

- добавьте в `Settings` поле `port: int = 8000` и задайте переменную окружения `PORT=9000`;  
  проверьте, что при создании объекта вы получаете `port=9000` и тип `int`.

### 3.5. `.env` и `python-dotenv`

`.env`‑файл — это текстовый файл с парами `KEY=VALUE`, который удобно использовать для локальной разработки.

```python
from dotenv import load_dotenv
import os

load_dotenv()  # по умолчанию ищет .env в текущей директории

db_url = os.getenv("DATABASE_URL")
```

**Правило:** не коммитьте реальные секреты в репозиторий, используйте `.env.example` c «пустыми» значениями в качестве шаблона.

### 3.6. Запомните по §119

- JSON — основной «язык обмена»; научитесь пользоваться `default`/`object_hook` и корректно обрабатывать `JSONDecodeError`.
- Для конфигов используйте отдельный слой (YAML/TOML + Pydantic), а не «магические константы» в коде.
- Не храните секреты в коде; используйте `.env`/переменные окружения + инструменты вроде `python-dotenv` и моделей Settings.
