## 6. Message serialization: msgpack, protobuf и другие (§153)

### 6.0. Почему не только JSON

JSON — отличный формат:

- человекочитаем;
- везде поддерживается;
- легко отлаживать.

Но:

- **объём**: JSON — текстовый, ключи и числа передаются как строки;
- **скорость**: разбор больших JSON‑ов может быть дорогим;
- **типизация**: нет строгой схемы, легко сделать несовместимые изменения.

Для высоконагруженных и распределённых систем часто нужны:

- более компактные бинарные форматы;
- возможность иметь **строгую схему** и версионирование.

Метафора:

- JSON — это как **вежливое письмо на бумаге**:
  - слова целиком, с пробелами;
  - удобно читать человеку;
  - но занимает много места и не очень быстро обрабатывается.
- Бинарные форматы — это как **аккуратно упакованный чемодан**:
  - вещи сложены плотно и продуманно;
  - человеку трудно понять содержимое «с одного взгляда»;
  - но **места занимает меньше**, и **таскать легче**.

Про строгую схему:

- в JSON вы можете сегодня отправлять `{ "name": "Alice" }`,
  а завтра `{ "full_name": "Alice" }`, и потребитель может растеряться;
- в protobuf/Avro вы явно говорите:
  - какие поля есть;
  - какие у них типы;
  - какие номера полей используются.

Это как **чертёж** vs «нарисовал от руки на салфетке»:

- с чертежом проще эволюционировать систему,
  когда много сервисов и команд.

### 6.1. Msgpack

Msgpack — бинарный формат, похожий на «бинарный JSON»:

- хранит знакомые типы: dict, list, строки, числа, bool, None;
- обычно **быстрее и компактнее, чем JSON**;
- хорошо подходит, когда:
  - обе стороны — ваши сервисы;
  - человекочитаемость не критична.

Установка:

```bash
pip install msgpack
```

Пример:

```python
import msgpack

data = {"id": 1, "name": "Alice", "tags": ["admin", "active"]}

packed = msgpack.packb(data, use_bin_type=True)
print("Размер JSON:", len(str(data).encode()))
print("Размер msgpack:", len(packed))

unpacked = msgpack.unpackb(packed, raw=False)
print(unpacked)
```

Как это читать:

- `packb` превращает обычный Python‑словарь в **байтовую строку**:
  - она может быть передана по сети, сохранена в файл или в Redis;
  - по размеру она обычно **меньше**, чем строка JSON.
- `unpackb` делает обратное:
  - из байтов снова получается привычный dict.

Типичный сценарий использования msgpack:

- если у вас есть **несколько своих сервисов**, которые общаются друг с другом;
- вам важно:
  - скорость;
  - компактность;
  - но формат не должен быть понятен внешним людям;
- тогда msgpack — удобный компромисс:  
  JSON‑подобная структура, но в бинарном виде.

### 6.2. Protocol Buffers (protobuf)

Protobuf от Google — формат с **явной схемой**:

- вы описываете структуры в `.proto` файле;
- компилятор `protoc` генерирует Python‑классы;
- данные сериализуются в компактный бинарный формат.

Пошагово:

1. Пишем `.proto` файл:
   - описываем сообщения (`message`), их поля и типы;
   - каждому полю даём **уникальный номер** (`= 1`, `= 2` и т.д.).
2. Прогоняем `protoc`:
   - он читает `.proto`;
   - генерирует для вас Python‑классы (и/или классы для других языков).
3. В Python:
   - создаём объект класса (`User(...)`);
   - заполняем поля;
   - вызываем `SerializeToString()` → получаем байты.
4. На принимающей стороне:
   - создаём пустой объект того же класса;
   - вызываем `ParseFromString(payload)`;
   - получаем заполненный объект с полями.

Пример `.proto`:

```proto
syntax = "proto3";

package users;

message User {
  int64 id = 1;
  string name = 2;
  repeated string roles = 3;
}
```

После генерации Python‑классов (через `protoc`):

```python
from users_pb2 import User

user = User(id=1, name="Alice", roles=["admin", "active"])
payload = user.SerializeToString()

new_user = User()
new_user.ParseFromString(payload)
print(new_user)
```

Плюсы:

- строгая схема;
- поддержка эволюции схемы (добавление полей, изменения с сохранением совместимости);
- компактный бинарный формат.

Минусы:

- нужна отдельная стадия генерации кода;
- человекочитаемость низкая (для отладки нужны инструменты).

Как думать о protobuf:

- он особенно полезен, когда:
  - у вас **много сервисов и команд**;
  - требуется **строгий контракт** между ними;
  - один и тот же формат нужно использовать на **разных языках** (Python, Go, Java…).
- JSON хорош для:
  - публичных API;
  - случаев, когда важно «чтоб было удобно дебажить в браузере».
- Protobuf хорош для:
  - внутренних высоконагруженных микросервисов;
  - коммуникации между языками;
  - долгоживущих систем, где важно аккуратное изменение схемы.

### 6.3. Avro, Cap'n Proto, FlatBuffers — обзорно

- **Avro**:
  - популярен в экосистеме Hadoop/Kafka;
  - схема может идти **рядом с данными**;
  - часто используется для событийных логов.
- **Cap'n Proto**:
  - оптимизирован под **zero‑copy** (минимум копирований в памяти);
  - быстрый доступ к полям без десериализации целого объекта.
- **FlatBuffers**:
  - также zero‑copy;
  - используется, например, в игровых движках и мобильной разработке.

Главная мысль:

- для **обычных веб‑сервисов** часто достаточно JSON/Msgpack;
- для **больших распределённых систем и RPC** полезно освоить protobuf/Avro.

Если очень упростить «дерево выбора формата»:

- **Начало пути / учебные проекты / простые REST‑API** → JSON.
- **Внутренние сервисы, важна скорость и размер** → JSON + msgpack там, где это даёт профит.
- **Микросервисы на разных языках, долгоживущие схемы, RPC** → protobuf или Avro.

Со временем вы сможете «спускаться глубже» (Cap'n Proto, FlatBuffers),
но для практики на Python чаще всего достаточно связки:

- JSON на внешних границах;
- msgpack/protobuf внутри системы, когда появятся реальные проблемы
  с объёмами и скоростью.
