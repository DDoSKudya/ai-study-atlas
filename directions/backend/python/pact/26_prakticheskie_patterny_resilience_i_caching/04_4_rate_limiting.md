## 4. Rate limiting (§151)

### 4.0. Зачем ограничивать частоту

Rate limiting нужен в двух направлениях:

1. **Вы — поставщик API**:
   - хотите защитить свои сервисы от злоупотреблений:
     - боты;
     - DDoS‑подобная активность;
     - баги в клиентских приложениях.
2. **Вы — потребитель внешнего API**:
   - провайдер говорит: «Не более 1000 запросов в минуту»;
   - нужно **уважать этот лимит**, иначе вас заблокируют.

Проще всего думать так:

- **rate limiting** — это вежливый способ сказать:
  - «столько запросов в секунду/минуту — ок»;
  - «всё, что сверху — либо медленнее, либо с ошибкой».

И это касается **обоих направлений**:

- вы защищаете **свои** сервисы от чужих;
- вы защищаете **чужие** сервисы от своих багов.

### 4.1. Базовые алгоритмы

#### 4.1.1. Token bucket (ведро токенов)

Идея:

- есть «ведро», в которое по фиксированной скорости **добавляются токены** (например, 10 в секунду);
- каждый запрос «забирает» токен:
  - если токен есть — запрос разрешён;
  - если токенов нет — запрос отклоняется или откладывается.

Свойства:

- можно копить «запас» токенов (burst) до размера ведра;
- удобен для **среднего лимита + небольших пиков**.

Метафора:

- у вас есть **кошелёк с жетонами на аттракционы**:
  - каждую минуту в кошелёк «падает» один новый жетон (фиксированная скорость пополнения);
  - на каждый аттракцион нужно потратить один жетон.
- Если вы долго ничего не делали:
  - жетонов накопилось много;
  - вы можете пару минут покататься **чаще обычного** (burst).
- Если вы катаетесь без остановки:
  - жетоны быстро заканчиваются;
  - пока новые не «накапают», вы временно не можете кататься.

#### 4.1.2. Leaky bucket (протекающее ведро)

Идея:

- запросы попадают в очередь (ведро);
- из очереди они **выходят с фиксированной скоростью**;
- если ведро переполнено — новые запросы отбрасываются.

Свойства:

- выход запросов **строго ограничен по скорости**;
- подходит для сглаживания входных пиков.

Метафора:

- у вас есть ведро с маленьким отверстием внизу:
  - вода (запросы) может литься в ведро **рывками**;
  - через отверстие вода вытекает **равномерно**.
- Если наливать воду не быстрее, чем она вытекает:
  - уровень в ведре остаётся примерно стабильным;
  - система обрабатывает запросы с ровной скоростью.
- Если наливать гораздо быстрее:
  - ведро переполняется;
  - лишняя вода переливается через край (запросы отбрасываются).

#### 4.1.3. Sliding window (скользящее окно)

Идея:

- мы храним **временные метки последних запросов**;
- в любой момент считаем, сколько запросов было за последний интервал (например, 1 минуту);
- если их больше порога — новые запросы отклоняем.

Свойства:

- интуитивно понятно: «не более N запросов за последние T секунд».

Пример «на пальцах»:

- правило: «не более 100 сообщений в чат за последние 60 секунд»;
- каждую секунду вы:
  - смотрите на все сообщения **за прошлую минуту**;
  - если их уже 100 — новое сообщение **отклоняете**;
  - если меньше — разрешаете и добавляете текущее.

### 4.2. Простая реализация в Python (in‑memory)

Для одного процесса можно реализовать простой sliding window:

```python
import time
from collections import deque
from typing import Deque


class RateLimiter:
    def __init__(self, max_calls: int, period: float) -> None:
        self.max_calls = max_calls
        self.period = period
        self.calls: Deque[float] = deque()

    def allow(self) -> bool:
        now = time.monotonic()
        # Удаляем все вызовы, которые старше окна
        while self.calls and self.calls[0] <= now - self.period:
            self.calls.popleft()

        if len(self.calls) < self.max_calls:
            self.calls.append(now)
            return True
        return False
```

Использование:

```python
limiter = RateLimiter(max_calls=5, period=1.0)  # не более 5 вызовов в секунду

if limiter.allow():
    # выполняем запрос
    ...
else:
    # отклоняем или ставим в очередь
    ...
```

Проблема:

- такой лимитер работает **только в рамках одного процесса**;
- при нескольких воркерах/серверов нужно хранилище, доступное всем (часто — Redis).

Как это читать:

- `self.calls` — это **очередь с временными метками** последних запросов;
- при каждом новом запросе:
  1. мы выкидываем из очереди всё, что **старше нашего окна** (`now - self.period`);
  2. смотрим, сколько меток осталось:
     - если меньше лимита — добавляем текущую метку и разрешаем запрос;
     - если уже достигли лимита — отклоняем.

Ограничения:

- такой подход хорошо работает:
  - в простых CLI‑утилитах;
  - в скриптах/сервисах **с одним процессом**.
- как только у вас:
  - несколько процессов (gunicorn/uvicorn с несколькими воркерами),
  - несколько машин,
  — **каждый из них будет считать лимит только для себя**.

Поэтому для прод‑сервисов rate limiting почти всегда завязан на:

- Redis;
- или на внешний уровень (API‑шлюз, балансировщик и т.п.).

### 4.3. Библиотека `limits`

`limits` — библиотека, реализующая различные алгоритмы rate limiting
и поддерживающая разные backend‑хранилища.

```bash
pip install limits
```

Пример (in‑memory):

```python
from limits import RateLimitItemPerSecond
from limits.storage import MemoryStorage

storage = MemoryStorage()
limit = RateLimitItemPerSecond(5, 1)  # 5 запросов в секунду

identifier = "user:42"

if storage.incr(limit, identifier):
    # запрос разрешён
    ...
else:
    # лимит превышен
    ...
```

С Redis‑backend:

```python
from limits.storage import RedisStorage

storage = RedisStorage("redis://localhost:6379/0")
```

Интерпретируем этот пример:

- `RateLimitItemPerSecond(5, 1)`:
  - правило: «не более 5 запросов в 1 секунду»;
  - есть аналогичные классы «в минуту», «в час» и т.д.
- `identifier = "user:42"`:
  - **по какому признаку** мы считаем лимит:
    - по пользователю;
    - по IP;
    - по API‑ключу.
- `storage.incr(limit, identifier)`:
  - пытаемся «увеличить счётчик» для этого лимита и идентификатора;
  - если ещё не вышли за предел — вернётся `True`;
  - если лимит уже исчерпан — вернётся `False`.

Смена backend‑а на Redis:

- практически **одна строка** меняется (инициализация `storage`);
- логика проверки лимита в коде остаётся той же.

### 4.4. Практические советы по rate limiting

- Делайте лимиты **по разным измерениям**:
  - на IP;
  - на пользователя;
  - на API‑ключ.
- Отдавайте **понятные ошибки**:
  - HTTP 429 Too Many Requests;
  - заголовки типа `Retry-After`.
- Логируйте факты «перелимитивания»:
  - это помогает увидеть аномальную активность или баги в клиентах.
