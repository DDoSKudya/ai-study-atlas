## 3. Кеширование (§150)

### 3.0. Зачем кеш

Типичный кейс:

- у вас есть функция, которая:
  - делает тяжёлые вычисления (ML‑модель, сложный SQL‑запрос),
  - или ходит во внешний API, который:
    - медленный,
    - стоит денег.
- входные данные часто **повторяются**:
  - одинаковые параметры запроса,
  - одни и те же пользователи/товары.

Без кеша:

- вы каждый раз проделываете ту же тяжёлую работу;
- время ответа и нагрузка на систему растут.

С кешем:

- результат **первого вызова** сохраняется в быстрый storage (память, Redis и т.п.);
- **повторные вызовы** с теми же аргументами получают результат **почти мгновенно**.

Метафора:

- Представьте, что вы живёте на 9 этаже без лифта.
- Каждый раз, когда кому‑то нужно узнать «какая погода на улице»,  
  вы спускаетесь вниз, выходите на улицу, смотрите и поднимаетесь обратно.
- Через пару дней вы догадываетесь:
  - достаточно **один раз спуститься**,
  - записать «+5°, ветер» на листочек,
  - и **повесить этот листочек на дверь**.

Теперь:

- если кто‑то спрашивает «какая погода?» через 5 минут,
  вы просто **смотрите на листочек** (кеш);
- если прошло несколько часов, и вы понимаете, что погода могла измениться,
  вы **обновляете листочек** (инвалидация кеша).

Кеш в программе — это тот самый листочек:

- вы меняете **дорогую операцию** (запрос к БД/сетевой вызов/тяжёлое вычисление)
  на **быстрый просмотр уже сохранённого результата**.

### 3.1. Встроенный кеш: `functools.lru_cache`

`lru_cache` — простой способ добавить кеш к чистой функции:

- кеширует результаты вызовов по аргументам;
- использует стратегию **Least Recently Used (LRU)**:
  - когда кеш переполнен, выбрасывает наименее недавно использованные записи.

По‑простому:

- у вас есть **маленький шкафчик** с ограниченным числом полок (`maxsize`);
- когда приходит новый объект (новый набор аргументов функции),  
  вы:
  - либо ставите его на свободную полку;
  - либо, если свободных полок нет, **выбрасываете то, чем давно не пользовались**.

Почему так:

- считается, что если чем‑то **давно не пользовались**,  
  то, скорее всего, ещё долго и не понадобится;
- а то, к чему **обращались недавно**, с большей вероятностью снова скоро понадобится.

```python
import time
from functools import lru_cache


@lru_cache(maxsize=128)
def slow_fibonacci(n: int) -> int:
    """Очень неэффективная реализация Фибоначчи, но с кешем."""
    time.sleep(0.1)  # имитация тяжёлого вычисления
    if n < 2:
        return n
    return slow_fibonacci(n - 1) + slow_fibonacci(n - 2)


if __name__ == "__main__":
    start = time.perf_counter()
    print(slow_fibonacci(30))
    print(f"Первый раз: {time.perf_counter() - start:.2f} c")

    start = time.perf_counter()
    print(slow_fibonacci(30))
    print(f"Второй раз (из кеша): {time.perf_counter() - start:.4f} c")
```

Особенности:

- `maxsize=None` — бесконечный кеш (осторожно с памятью);
- `typed=True` — различает аргументы по типу (`1` и `1.0` будут разными ключами);
- метод `slow_fibonacci.cache_clear()` — сбросить кеш.

Как читать этот пример:

- первый вызов `slow_fibonacci(30)`:
  - функция реально считается;
  - занимает заметное время (мы специально добавили `time.sleep(0.1)`).
- второй вызов с тем же аргументом:
  - Python даже не заходит внутрь функции;
  - он сразу возвращает **сохранённый ранее результат** из кеша.

Шаги, чтобы добавить `lru_cache` в свой код:

1. Найдите **функцию без побочных эффектов** (чистую функцию), у которой:
   - тот же вход всегда даёт тот же выход;
   - нет записей в БД/файлы/внешний мир;
   - внутри — тяжёлая логика или запрос.
2. Добавьте над ней `@lru_cache(maxsize=...)`.
3. Проверьте, что:
   - функция продолжает выдавать те же результаты;
   - время выполнения повторных вызовов заметно упало.

### 3.2. Более гибкий in‑memory кеш: `cachetools`

`cachetools` даёт разные типы кешей:

- `LRUCache` — как `lru_cache`, но в виде объекта;
- `TTLCache` — записи живут не дольше заданного TTL;
- `LFUCache` — вытесняет **наименее часто используемые** записи.

Интуитивно:

- `LRUCache`:
  - важно **«когда в последний раз пользовались»**;
  - если давно не трогали — можно выбросить.
- `LFUCache`:
  - важно **«сколько раз пользовались»**;
  - то, что почти никогда не запрашивают, — первый кандидат на вылет.

Установка:

```bash
pip install cachetools
```

Пример с `TTLCache`:

```python
import time
import random
from cachetools import TTLCache, cached

# Кеш на 100 элементов, каждый живёт не более 10 секунд
cache = TTLCache(maxsize=100, ttl=10)


@cached(cache)
def get_exchange_rate(base: str, target: str) -> float:
    """Эмуляция запроса к внешнему сервису курсов валют."""
    print("Запрашиваем внешний сервис...")
    time.sleep(1)  # имитация сети
    # В реальном коде здесь был бы HTTP-запрос
    return round(50 + random.random(), 2)


if __name__ == "__main__":
    for _ in range(3):
        print(get_exchange_rate("USD", "RUB"))
    print("Ждём, пока TTL истечёт...")
    time.sleep(11)
    print(get_exchange_rate("USD", "RUB"))  # снова обращение к внешнему сервису
```

Важные моменты:

- `TTLCache` отлично подходит для данных,
  которые **немного устаревшие — не проблема** (например, курсы валют, погода).
- вы можете управлять кешом как объектом:

```python
cache.clear()
cache.pop(("USD", "RUB"), None)
```

Когда стоит перейти от `lru_cache` к `cachetools`:

- нужно более тонко контролировать:
  - **размер** кеша (в штуках объектов, в байтах — можно считать размер вручную);
  - **время жизни** записей (через TTL);
  - **поведение при вытеснении** (LRU vs LFU).
- нужно иметь **несколько разных кешей** с разными настройками:
  - один — для курсов валют (TTL 10 секунд);
  - другой — для справочников (TTL 1 час и большой `maxsize`);
  - третий — для результатов вычислений (LFU).

### 3.3. Внешние кеши: Redis и Memcached

Когда один процесс Python — это уже не вся система:

- у вас несколько воркеров (gunicorn, celery и т.п.);
- у вас несколько машин (микросервисы, кластер);

встроенный in‑memory кеш **на процессе** уже не хватает:

- каждый процесс держит свой кеш;
- данные не разделяются;
- при рестарте воркера кеш полностью теряется.

Решение — использовать **внешний кеш**:

- Redis;
- Memcached;
- специализированные решения (Hazelcast и др., но за рамками Python‑фокуса).

Интуитивное отличие от `lru_cache`/`cachetools`:

- **встроенный кеш в процессе**:
  - живёт внутри **одного процесса Python**;
  - у каждого воркера — свой отдельный кеш;
  - при рестарте процесса всё забывается.
- **внешний кеш**:
  - это **отдельный сервис** (как отдельная БД);
  - к нему подключаются сразу **несколько приложений/воркеров/языков**;
  - данные переживают перезапуск приложений (пока живёт сам кеш‑сервер).

#### 3.3.1. Redis: базовый пример

Установка клиента:

```bash
pip install redis
```

Пример:

```python
import json
import time
from typing import Any

import redis

r = redis.Redis(host="localhost", port=6379, db=0)


def get_user_from_db(user_id: int) -> dict:
    """Заглушка вместо реального запроса к БД."""
    print(f"Запрос к БД для user_id={user_id}")
    time.sleep(1)
    return {"id": user_id, "name": f"User {user_id}"}


def get_user(user_id: int, ttl_seconds: int = 60) -> dict:
    cache_key = f"user:{user_id}"

    cached = r.get(cache_key)
    if cached is not None:
        return json.loads(cached)

    user = get_user_from_db(user_id)
    r.setex(cache_key, ttl_seconds, json.dumps(user))
    return user
```

Что здесь важно:

- мы явно формируем **ключ кеша** (`user:{id}`);
- используем `SETEX` (`setex`) — записать значение **с TTL**;
- сериализуем данные в JSON (в практических проектах часто используют `msgpack`).

Разберём по шагам, что делает `get_user`:

1. **Собираем ключ**:
   - `cache_key = f"user:{user_id}"` → для `user_id=5` ключ будет `"user:5"`;
   - ключ — это **строка‑идентификатор**, по которой мы будем искать данные в Redis.
2. **Пытаемся прочитать из кеша**:
   - `cached = r.get(cache_key)`:
     - если такого ключа нет — вернётся `None`;
     - если он есть — вернётся **байтовая строка** (то, что мы раньше туда положили).
3. **Если в кеше что‑то нашлось**:
   - `if cached is not None: return json.loads(cached)`:
     - превращаем байты обратно в Python‑словарь (`json.loads`);
     - сразу возвращаем результат, **не трогая БД**.
4. **Если в кеше ничего нет**:
   - зовём «настоящую» функцию `get_user_from_db`;
   - получаем словарь `user`;
   - кладём его в Redis с помощью `setex`:
     - `r.setex(cache_key, ttl_seconds, json.dumps(user))`;
     - `json.dumps(user)` превращает словарь в строку JSON;
     - `ttl_seconds` задаёт **время жизни записи**.
5. **Возвращаем свежие данные**:
   - первый вызов чуть медленнее (БД),  
   - следующие — быстрые (из Redis).

Метафора:

- Redis — это **общий «листочек на двери подъезда»**,  
  который видят **все жильцы** (все процессы/машины);
- `lru_cache` — это «листочек на **двери вашей квартиры**»:
  - соседи его не видят;
  - при переезде/ремонте вы его выбрасываете.

#### 3.3.2. Memcached: кратко

Memcached — простой **in‑memory key‑value store**:

- нет сложных структур данных как в Redis;
- очень быстрый;
- хорошо подходит для:
  - кеша HTML‑страниц;
  - результатов запросов;
  - небольших объектов.

Клиенты для Python:

- `pymemcache`;
- `python-memcached`.

Интерфейс похож: `get(key)`, `set(key, value, expire=...)`.

Упрощённо:

- если вам нужен **минималистичный, очень быстрый кеш** без сложных структур —
  **Memcached** может быть проще;
- если нужны:
  - списки, множества, сортированные множества, pub/sub и т.п.;
  - счётчики, блокировки, более «богатый» функционал,
  — тогда чаще выбирают **Redis**.

### 3.4. Инвалидация кеша

Классическая шутка:

> В программировании есть две сложные задачи:  
> — инвалидация кеша  
> — и назвать вещи правильными именами.

Типичные стратегии:

- **TTL (time‑based)**:
  - запись живёт, например, 5 минут;
  - после этого автоматически считается устаревшей;
  - просто, но данные могут быть немного «старые».
- **Manual / event‑based**:
  - при изменении данных **явно очищаем** связанный кеш;
  - требует дисциплины и хорошего понимания, какие ключи связаны.
- **Tag‑based / key‑prefix**:
  - группируем ключи по префиксам или «тегам»;
  - при изменении данных можем очищать **всю группу**.

Пример с «версией» кеша в префиксе:

```python
import json
import redis

r = redis.Redis()

CACHE_VERSION = 1  # увеличиваем при изменении формата или логики


def _user_cache_key(user_id: int) -> str:
    return f"v{CACHE_VERSION}:user:{user_id}"


def get_user_cached(user_id: int) -> dict:
    key = _user_cache_key(user_id)
    if (raw := r.get(key)) is not None:
        return json.loads(raw)
    # здесь реальный запрос к БД
    user = {"id": user_id, "name": "Alice"}
    r.setex(key, 300, json.dumps(user))
    return user
```

Если вы меняете формат данных или подход к кешированию:

- достаточно **увеличить `CACHE_VERSION`**;
- старые ключи автоматически перестанут находиться и будут вымываться по TTL.

Интуиция:

- раньше вы писали на листочке: «Погода: +5°»;
- теперь решили указывать ещё и влажность: «Погода: +5°, влажность 80%»;
- старые листочки **по формату уже не подходят**.

Вместо того, чтобы:

- ходить по всему дому и вручную снимать старые листочки,

вы:

- меняете префикс с `v1:` на `v2:`;
- старые записи с `v1:` просто **перестают находиться**;
- через какое‑то время (TTL) они исчезнут сами.

### 3.5. Когда кеш не нужен (или вреден)

- **очень изменчивые данные**:
  - каждый запрос возвращает новое значение (например, текущий баланс в трейдинге);
- **сложные правила инвалидации**:
  - легко сделать систему, где вы не уверены,
    какие данные на самом деле «актуальны»;
- **без мониторинга и метрик**:
  - вы не видите «cache hit ratio»;
  - не понимаете, действительно ли кеш помогает.

Правило:

- начинайте с **простых и очевидных кейсов**;
- добавляйте кеш **после измерений**, а не «на всякий случай».

Хороший признак, что кеш вам действительно нужен:

- вы **померили** (через профилирование/метрики) и видите:
  - одна и та же тяжёлая операция повторяется часто;
  - она занимает значимый процент времени/ресурсов;
  - результат этой операции приемлемо хранить какое‑то время.

Плохой признак:

- кто‑то в команде говорит: «давайте всё обкешируем, чтобы было быстрее»,
  без измерений и понимания, что именно является узким местом.
