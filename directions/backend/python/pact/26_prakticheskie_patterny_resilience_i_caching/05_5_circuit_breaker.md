## 5. Circuit breaker (§152)

### 5.0. Проблема каскадных сбоев

Сценарий:

- у вас есть сервис A, который вызывает сервис B (внешний API);
- по каким‑то причинам B начинает **массово возвращать ошибки** или зависать;
- A на каждый пользовательский запрос:
  - делает запрос в B;
  - долго ждёт;
  - получает ошибку.

Результат:

- все запросы к A **тормозят** или падают;
- нагрузка на B не падает, а наоборот, может расти из‑за ретраев;
- графики в мониторинге выглядят как «обвал домино».

Ещё один способ представить:

- У вас в квартире есть электроприбор (B), который начал **коротить**:
  - он постоянно даёт перегрузку сети;
  - если ничего не делать, может загореться проводка.
- В щитке у вас стоит **автоматический предохранитель** (circuit breaker):
  - при серии перегрузок он **отключает линию**;
  - вы **теряете часть комфорта** (нет света в одной комнате),
  - но **квартира не сгорает целиком**.

В микросервисах без circuit breaker картина такая же:

- один «сломаный» сервис постепенно тянет за собой весь контур,
  потому что остальные продолжают его дёргать и ждать ответа.

### 5.1. Идея circuit breaker

Circuit breaker («предохранитель», «автомат») — паттерн, который:

- отслеживает **успехи и неудачи** вызова зависимости;
- при серии неудач **«размыкает цепь»**:
  - перестаёт реально вызывать зависимость;
  - сразу возвращает ошибку или заранее заданный fallback;
- через некоторое время пробует **единичные «пробные» вызовы**;
- если они успешны — **закрывает цепь** и продолжает работу в обычном режиме.

Состояния:

- `closed` — всё хорошо, все вызовы проходят;
- `open` — «цепь разомкнута», вызовы не проходят;
- `half-open` — режим «пробных» вызовов:
  - пропускаем, например, каждый N‑й вызов;
  - если успешен — возвращаемся в `closed`;
  - если снова ошибки — обратно в `open`.

Интуитивная картина по состояниям:

- **`closed` (закрыт)**:
  - цепь «замкнута», ток (запросы) **свободно проходит**;
  - всё выглядит нормально, мы не вмешиваемся.
- **`open` (открыт)**:
  - цепь «разомкнута», мы **вообще не подаём ток** на проблемное устройство;
  - все запросы к зависимому сервису **даже не пытаются идти наружу**;
  - вызывающий сервис быстро возвращает ошибку или fallback.
- **`half-open` (полуоткрыт)**:
  - режим осторожной проверки: «а вдруг оно уже починилось?»;
  - мы **пускаем ограниченное число запросов**:
    - если они проходят успешно — считаем, что сервис ожил, и возвращаемся в `closed`;
    - если снова сыпятся ошибки — возвращаемся в `open` и ждём ещё.

### 5.2. Библиотека `pybreaker`

Установка:

```bash
pip install pybreaker
```

Пример:

```python
import requests
import pybreaker


breaker = pybreaker.CircuitBreaker(
    fail_max=5,               # после 5 неудач подряд - open
    reset_timeout=30,         # через 30 секунд попробовать снова (half-open)
)


@breaker
def call_external_api(url: str) -> dict:
    resp = requests.get(url, timeout=3)
    resp.raise_for_status()
    return resp.json()
```

Поведение:

- пока число подряд идущих ошибок < 5 — состояние `closed`;
- как только порог превышен:
  - breaker переходит в `open`;
  - дальнейшие вызовы `call_external_api` **не пойдут в сеть**,
    а сразу выбросят `CircuitBreakerError`;
- через 30 секунд breaker переходит в `half-open`:
  - пропускает ограниченное число «пробных» вызовов;
  - в зависимости от их результатов возвращается в `closed` или `open`.

Разбор параметров:

- `fail_max=5`:
  - если подряд случилось **5 неудачных вызовов** функции `call_external_api`,
  - считаем, что зависимость в проблемном состоянии, и **открываем** breaker.
- `reset_timeout=30`:
  - после перехода в состояние `open` мы **30 секунд не трогаем** внешний сервис;
  - по истечении этого времени breaker переходит в `half-open`
    и начинает пускать пробные запросы.

Таким образом:

- первая серия ошибок → **срабатывание предохранителя**;
- дальше некоторое время:
  - пользователи получают быстрый и понятный отказ;
  - внешний сервис **не закидывается** дополнительными запросами;
- затем аккуратная проверка: «а он уже жив?» → и возврат к нормальной работе.

### 5.3. Callback‑и и мониторинг

`pybreaker` поддерживает слушателей событий:

```python
class LoggingListener(pybreaker.CircuitBreakerListener):
    def state_change(self, cb, old_state, new_state):
        print(f"Circuit breaker {cb.name} изменил состояние: {old_state} -> {new_state}")


breaker = pybreaker.CircuitBreaker(
    fail_max=5,
    reset_timeout=30,
    listeners=[LoggingListener()],
    name="external-api",
)
```

Это удобное место, чтобы:

- логировать изменения состояния;
- отправлять метрики в Prometheus/StatsD;
- уведомлять команду, если зависимость «упала».

Практически:

- каждый переход `closed → open`:
  - повод завести инцидент или хотя бы тикет;
  - сигнал, что зависимость **превысила порог ошибок**, который вы считаете допустимым.
- каждое `open → half-open → closed`:
  - полезно для отчётов: сколько времени сервис был недоступен;
  - помогает оценивать стабильность внешних API во времени.

### 5.4. Практические советы по circuit breaker

- Не забывайте про **fallback**:
  - кэшированный ответ;
  - упрощённую версию функционала;
  - аккуратное сообщение пользователю.
- Настраивайте пороги исходя из:
  - SLA сервиса;
  - «нормального» уровня ошибок;
  - допустимой деградации.
- Комбинируйте circuit breaker с:
  - **retry/backoff** (но осторожно, чтобы не усиливать проблему);
  - **rate limiting**;
  - **кешированием**.

Можно мыслить так:

- **retry** — пытается **достучаться** до сервиса, если тот иногда «чихает»;
- **circuit breaker** — **защищает систему**, если сервис **реально лежит**;
- **rate limiting** — не даёт вам **заспамить** зависимость запросами;
- **кеширование** — уменьшает количество запросов к зависимости в принципе.

Один и тот же внешний сервис может быть окружён **всеми этими слоями сразу**:

- кеш → меньше запросов;
- rate limiting → ограничение пиковой нагрузки;
- retry → аккуратные повторы при временных сбоях;
- circuit breaker → быстрое «отключение» при длительных проблемах.
