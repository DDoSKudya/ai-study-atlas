## 2. Retry и backoff (§149)

### 2.0. Наивный retry и его проблемы

Частое желание: «Если запрос к API упал — давай просто повторим».

Наивная реализация:

```python
import time
import requests

def get_user(user_id: int, max_retries: int = 3) -> dict:
    for attempt in range(1, max_retries + 1):
        try:
            resp = requests.get(f"https://api.example.com/users/{user_id}", timeout=3)
            resp.raise_for_status()
            return resp.json()
        except Exception as exc:
            print(f"Попытка {attempt} не удалась: {exc}")
            if attempt == max_retries:
                raise
            time.sleep(1)  # всегда 1 секунда
```

Проблемы:

- **фиксированная задержка** между попытками:
  - если сервис перегружен, вы продолжаете его «долбить» с той же частотой;
- **нет jitter**:
  - множество клиентов могут синхронно отправлять повторы;
- **повторяем всё подряд**:
  - в том числе ошибки, которые *не починятся* от повтора (валидация, 4xx);
- **нет явного контроля, на что мы смотрим**:
  - какой код ответа, какие исключения.

Это похоже на ситуацию:

- дверь в магазин закрыта;
- вы дёргаете ручку каждые 5 секунд с одинаковой силой;
- не смотрите на табличку «обед до 13:00»;
- не проверяете, не сорвали ли вы замок.

С наивным retry вы:

- **можете усугубить проблему** (ещё сильнее перегрузить зависимость);
- тратите время пользователя на бесполезные попытки;
- усложняете анализ инцидентов: в логах куча одинаковых ошибок.

### 2.1. Идея exponential backoff

Exponential backoff: при каждом неуспехе **увеличиваем время ожидания**:

- первая ошибка — подождали 0.5 секунды;
- вторая — 1 секунду;
- третья — 2 секунды;
- четвёртая — 4 секунды;
- и так далее (пока не достигнем верхней границы).

Интуиция:

- если зависимость «пришла в себя» быстро — мы не ждём слишком долго;
- если зависимость **явно в проблемном состоянии** — мы не усиливаем её страдания
  постоянными запросами, а наоборот, даём ей время «отдышаться».

Удобно думать так:

- первая попытка — «просто попробовали»;
- вторая — «что‑то не так, давай дадим чуть больше времени»;
- третья и дальше — «каждый раз увеличиваем паузу, потому что, похоже, проблема серьёзная».

С точки зрения математики:

- задержка на попытке \(n\) примерно равна \(base \times 2^{(n-1)}\)  
  (например, `base=0.5`: 0.5, 1, 2, 4, 8…);
- мы почти всегда ограничиваем задержку сверху (`max_delay`),  
  чтобы не ждать, например, час на одной функции.

### 2.2. Jitter (избегаем «стада буйволов»)

Если у вас **много клиентов**, которые:

- одновременно получили ошибку;
- одновременно начали делать retry;
- одновременно ждут 1, 2, 4, 8 секунд…

…то они снова и снова будут **синхронно обрушиваться** на сервис.

Решение: добавить **случайную компоненту** (jitter):

- вместо «ровно 4 секунды» делаем «случайно от 3 до 5 секунд»;
- пики нагрузки **размазываются по времени**.

Метафора:

- представьте, что в метро все вагоны останавливаются и двери закрываются;
- по команде «поехали» **все пассажиры всех вагонов** одновременно штурмуют выход;
- если же задержка у каждого немного случайная — люди выходят **чуть‑чуть вразнобой**,
  и толчеи меньше.

В коде jitter выглядит как «слегка потрясти» детерминированное время:

```python
import random

base_delay = 4.0
jitter = random.uniform(-1.0, 1.0)  # случайное число от -1 до +1
actual_delay = max(0.0, base_delay + jitter)
```

В `tenacity` это уже реализовано в виде готовых стратегий ожидания.

### 2.3. Библиотека `tenacity`: базовый пример

`tenacity` — удобная библиотека для **декларативного** описания retry‑логики.

Что значит «декларативного» по‑человечески:

- вы **не пишете вручную цикл** `for attempt in range(...): try/except/...`;
- вы **одним декоратором** говорите:
  - «эту функцию, пожалуйста, повторяй не более N раз»;
  - «между попытками жди вот так»;
  - «ретрай только такие‑то ошибки»;
- `tenacity` берёт на себя всю «грязную работу»:
  - считает попытки;
  - спит нужное количество секунд;
  - решает, когда остановиться.

Ещё одно важное преимущество:

- все настройки retry **видно в одном месте, над функцией**;
- не нужно листать код, чтобы понять, как именно происходит повтор.

Установка:

```bash
pip install tenacity
```

Простейший пример:

```python
import random
from tenacity import retry, stop_after_attempt, wait_exponential


@retry(
    stop=stop_after_attempt(3),          # не более 3 попыток
    wait=wait_exponential(multiplier=1, min=1, max=8),  # 1, 2, 4, 8 сек (ограничено max)
)
def flaky_operation() -> int:
    value = random.random()
    if value < 0.7:
        print(f"Случайное число {value:.2f} < 0.7 — считаем, что произошла ошибка")
        raise RuntimeError("Временный сбой")
    print(f"Случайное число {value:.2f} ≥ 0.7 — успех")
    return 42


if __name__ == "__main__":
    result = flaky_operation()
    print("Результат:", result)
```

Что здесь происходит:

- `@retry(...)` оборачивает функцию так, что в случае исключения:
  - она будет **автоматически вызвана ещё раз**;
  - задержка между попытками будет расти по **экспоненте**;
  - количество попыток ограничено.

Разберём параметры по строчкам, чтобы не было «магии»:

- `@retry(...)` — надпись над функцией говорит: «эту функцию можно безопасно повторять».
- `stop=stop_after_attempt(3)`:
  - максимум 3 вызова функции: **1 оригинальный + 2 повтора**;
  - если все 3 раза закончатся ошибкой — ошибка «вылетит наверх».
- `wait=wait_exponential(multiplier=1, min=1, max=8)`:
  - стратегия ожидания: «ждём по экспоненте»;
  - `multiplier=1` — базовый множитель, от которого растут задержки;
  - `min=1` — **минимальная** задержка между попытками — 1 секунда;
  - `max=8` — **максимальная** задержка — не больше 8 секунд, даже если формула даёт больше.

Если вы не любите запоминать параметры, можно мысленно переписать:

> «Эта функция может упасть,  
> попробуй её максимум 3 раза,  
> между попытками жди 1 → 2 → 4 → 8 секунд,  
> но не больше 8 секунд».

### 2.4. Ограничение по времени и типам ошибок

Часто нас интересует **не только число попыток**, но и:

- **общее время** ретраев;
- **конкретные типы исключений**;
- иногда — **значение результата**, если он «подозрительный».

Пример:

```python
import requests
from tenacity import (
    retry,
    stop_after_attempt,
    stop_after_delay,
    wait_exponential_jitter,
    retry_if_exception_type,
)


class TemporaryHTTPError(Exception):
    """Наше обёрточное исключение для временных HTTP-проблем."""


def _is_temporary_status(status_code: int) -> bool:
    # Пример: считаем 500, 502, 503, 504 временными
    return status_code in {500, 502, 503, 504}


@retry(
    stop=(stop_after_attempt(5) | stop_after_delay(20)),  # не более 5 попыток И не более 20 секунд
    wait=wait_exponential_jitter(initial=1, max=10),      # экспонента + jitter
    retry=retry_if_exception_type(TemporaryHTTPError),
    reraise=True,
)
def fetch_user(user_id: int) -> dict:
    resp = requests.get(
        f"https://api.example.com/users/{user_id}",
        timeout=3,
    )
    if _is_temporary_status(resp.status_code):
        raise TemporaryHTTPError(f"Временная ошибка {resp.status_code}")
    resp.raise_for_status()  # остальные ошибки пробрасываем как есть
    return resp.json()
```

Ключевые моменты:

- **`stop_after_attempt` + `stop_after_delay`**:
  - ограничиваемся и по числу попыток, и по общему времени;
- **`wait_exponential_jitter`**:
  - экспоненциальный backoff + jitter «из коробки»;
- **`retry_if_exception_type`**:
  - ретраим **только** наши временные ошибки,
  - остальные исключения (`ValueError`, `TimeoutError` и т.п.) сразу пробрасываются.

Полезно запомнить такую «шпаргалку по параметрам `retry`»:

- **`stop=...`** — отвечает на вопрос: *«когда мы окончательно сдаёмся?»*
  - по числу попыток (`stop_after_attempt`);
  - по общему времени (`stop_after_delay`);
  - можно комбинировать (как в примере).
- **`wait=...`** — *«сколько ждём между попытками?»*
  - фиксированное время (`wait_fixed`);
  - экспонента (`wait_exponential`);
  - экспонента с jitter (`wait_exponential_jitter`).
- **`retry=...`** — *«что именно мы считаем поводом для повтора?»*
  - по типам исключений (`retry_if_exception_type`);
  - по результату (`retry_if_result`);
  - можно комбинировать условия.
- **`reraise=True`** — *«если все попытки провалились, пробрось последнее исключение наружу»*.

### 2.5. Retry по результату функции

Иногда операция не бросает исключений, но возвращает **«плохой» результат**,
который хотелось бы повторить:

- `status == "PENDING"` слишком долго;
- `data is None` и т.п.

`tenacity` умеет ретраить и по результату через `retry_if_result`.

```python
from tenacity import retry, stop_after_attempt, wait_fixed, retry_if_result


def is_bad_result(response: dict) -> bool:
    # Например, пока сервер не подготовил данные
    return response.get("status") == "PENDING"


@retry(
    stop=stop_after_attempt(5),
    wait=wait_fixed(2),
    retry=retry_if_result(is_bad_result),
    reraise=True,
)
def poll_job(job_id: str) -> dict:
    # Здесь могла бы быть реальная HTTP-логика
    # Для примера считаем, что она уже реализована
    ...
```

Типичный житейский пример:

- вы заказываете доставку и периодически обновляете статус:
  - сначала он «в обработке»;
  - потом «курьер забрал»;
  - потом «доставлено».
- Пока статус «в обработке», **сам запрос проходит успешно** (сервер не падает),
  но **результат нам не подходит** — мы ждём «доставлено».

`retry_if_result` как раз про такие ситуации:

- сама операция технически успешна;
- но бизнес‑результат временно нас не устраивает;
- мы хотим **подождать и повторить**, пока не станет «готово».

### 2.6. Практические советы по retry

- **Всегда ставьте таймауты** на операции, которые ретраите:
  - иначе каждая попытка может «висеть» слишком долго,  
    и суммарное время вылезет за всё разумное.
- **Ретрайте только «временные» ошибки**:
  - сетевые timeouts,
  - коды 5xx,
  - временную недоступность БД.
- **Не ретрайте ошибки валидации и бизнес‑ошибки**:
  - 4xx вроде 400, 401, 403, 404;
  - «пользователь не найден»;
  - «недостаточно средств».
- **Думайте об идемпотентности**:
  - безопасно ли повторно выполнять операцию?
  - если операция делает «списать деньги», ретраи могут привести к двойному списанию.
- Комбинируйте retry с **логированием и метриками**:
  - удобно видеть, сколько было ретраев, какие ошибки, с какими задержками.

Можно думать о добавлении retry как о **рецепте из 5 шагов**:

1. **Найдите место, где есть внешняя зависимость**:
   - HTTP‑запрос;
   - обращение к БД;
   - вызов очереди/брокера.
2. **Решите, какие ошибки считаются временными**:
   - timeouts;
   - коды 5xx;
   - конкретные исключения драйвера БД.
3. **Придумайте разумные ограничения**:
   - максимум попыток;
   - максимум общего времени;
   - разумные задержки (не миллисекунды и не минуты).
4. **Оберните функцию в `@retry`** с понятными параметрами:
   - сначала можно сделать очень простой вариант (например, 3 попытки, фиксированная задержка);
   - потом постепенно усложнять (экспонента, jitter, разные типы ошибок).
5. **Добавьте логирование и метрики**:
   - логируйте хотя бы факт ретрая: «попытка X из Y из‑за такой‑то ошибки»;
   - по метрикам проверяйте, что ретраи действительно помогают, а не ухудшают ситуацию.
