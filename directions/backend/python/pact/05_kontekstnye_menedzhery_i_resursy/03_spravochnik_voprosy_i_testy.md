## Частые сценарии и решения (Часть V)

| Задача | Решение |
|--------|---------|
| Гарантированно закрыть файл | `with open(path) as f: ...` |
| Временно подменить stdout/stderr | `with redirect_stdout(io.StringIO()) as buf: ...` |
| Игнорировать одно ожидаемое исключение в блоке | `with suppress(FileNotFoundError): os.remove(path)` (см. **Часть III, §14** — **suppress**) |
| Открыть несколько файлов из списка путей | `ExitStack` + цикл `stack.enter_context(open(p))` |
| Условный контекст (иногда с файлом, иногда без) | `ctx = open(path) if path else nullcontext(); with ctx: ...` (см. **Часть III, §14** — **nullcontext**) |
| Объект с методом close(), но без with | `with closing(obj): ...` |
| Временно сменить текущую директорию (3.11+) | `with chdir(path): ...` |
| Ограничить время async-операции (3.11+) | `async with asyncio.timeout(seconds): ...` |

## Практические рекомендации: когда что использовать (Часть V)

| Ситуация | Рекомендация | Обоснование |
|----------|--------------|-------------|
| Простой контекст «вход → использование → выход», одна функция | **@contextmanager** + yield | Меньше кода, намерение видно сразу; один yield, при необходимости try/finally. |
| Контекст с богатой логикой, переиспользование состояния, несколько методов | Класс с **__enter__**/ **__exit__** | Удобнее хранить состояние и обрабатывать исключения в одном объекте. |
| Число контекстов неизвестно заранее или задаётся списком/конфигом | **ExitStack** | Невозможно написать N вложенных with; enter_context в цикле. |
| Два-три фиксированных контекста | **with A, B, C:** | Короче и читаемее, чем вложенные with. |
| Объект с методом close(), но без протокола with | **closing(obj)** | Гарантированный вызов close() при выходе без написания своего менеджера. |
| Подавить одно ожидаемое исключение в блоке | **suppress(типы)** | Явнее и безопаснее, чем пустой except. |
| Условный контекст (иногда с ресурсом, иногда без) | **nullcontext()** | Единообразный код «with ctx:» при разных ветках. |

**Стиль кода:** предпочитайте **with** открытию/закрытию вручную; возвращайте из **__exit__** **False** (или ничего), если не обрабатываете исключение осознанно; в **@contextmanager** всегда используйте **try/finally** после yield для очистки; при исключении в **__enter__** откатывайте уже приобретённые ресурсы в except внутри **__enter__**.

## Краткое повторение терминологии

| Термин | Определение |
|--------|-------------|
| **Контекстный менеджер** | Объект с методами `__enter__` и `__exit__`, используемый в `with`. |
| **Протокол контекстного менеджера** | Соглашение: `__enter__()` при входе, `__exit__(exc_type, exc_val, exc_tb)` при выходе. |
| **Вход в контекст** | Вызов `__enter__`; обычно приобретение ресурса. |
| **Выход из контекста** | Вызов `__exit__`; обычно освобождение ресурса. |
| **Подавление исключения** | Возврат `True` из `__exit__` — исключение не пробрасывается. |
| **exc_type, exc_val, exc_tb** | Аргументы `__exit__`: тип исключения, экземпляр, traceback; при нормальном выходе все **None**. |
| **Генераторный контекстный менеджер** | Реализация через `@contextmanager` и функцию с одним `yield`. |
| **ExitStack** | Контекстный менеджер для динамического набора контекстов и callback; выход в порядке LIFO. |
| **enter_context(cm)** | Метод ExitStack: войти в контекст **cm** и зарегистрировать его выход при выходе из ExitStack. |
| **Асинхронный контекстный менеджер** | Объект с `__aenter__` и `__aexit__` (корутины); используется в `async with`. |
| **closing(obj)** | Контекстный менеджер из contextlib: при выходе вызывает **obj.close()**. |
| **nullcontext(enter_result)** | «Пустой» контекст: при входе возвращает **enter_result**, при выходе ничего не делает. |
| **LIFO** | Last In, First Out — последний вошёл, первый вышел; порядок выхода из вложенных контекстов и из ExitStack. |
| **Ресурс** | То, что нужно явно приобрести (открыть, захватить) и явно освободить (закрыть, снять). |
| **Утечка ресурса** | Ситуация, когда ресурс приобретён, но не освобождён (например, из-за исключения между open и close). |

## Примеры из практики (Часть V)

| Сценарий | Как реализовать | Замечание |
|----------|-----------------|-----------|
| **Тест: перехватить print()** | **with redirect_stdout(io.StringIO()) as buf: ...; buf.getvalue()** | В тестах проверяют вывод функции без реального вывода в консоль. |
| **Временный файл на время блока** | **with tempfile.NamedTemporaryFile(delete=False) as f:** или **@contextmanager** с **mkstemp** + **finally: os.remove(path)** | Файл удаляется при выходе из with. |
| **Критическая секция в многопоточности** | **with threading.Lock():** или **with lock:** | Блокировка захватывается при входе, освобождается при выходе; при исключении в секции лок освобождается. |
| **Чтение конфига из файла с fallback** | **ctx = open(path) if path else nullcontext(); with ctx as f: ...** | Если путь не задан, контекст «пустой» — в теле можно проверить **f is None** или не использовать f. |
| **Ограничить время HTTP-запроса (async)** | **async with asyncio.timeout(10): await session.get(url)** | При превышении 10 секунд задача отменяется, возбуждается TimeoutError. |
| **Открыть все файлы из списка (часть может не существовать)** | **with ExitStack() as stack:** в цикле **try: f = stack.enter_context(open(p)); except FileNotFoundError: continue** | Только существующие файлы открываются; при выходе все закрываются. |

## Типичные ошибки (Часть V)

- **Забыть `with` и писать только open/close:** при исключении между открытием и закрытием ресурс не освобождается. Всегда использовать `with open(...) as f:` (или аналог) для файлов и других ресурсов.
- **Возвращать `True` из `__exit__` «на всякий случай»:** исключения перестают пробрасываться; ошибки «исчезают» и их трудно отлаживать. Возвращать `True` только когда исключение осознанно обрабатывается.
- **Не использовать `finally` после `yield` в `@contextmanager`:** при исключении в теле блока код после `yield` не выполнится без `finally`; ресурс останется не освобождённым. Всегда оборачивать код очистки в `try: ... finally: ...` после `yield`.
- **Два и более `yield` в функции под `@contextmanager`:** допускается только один `yield`; иначе семантика входа/выхода ломается. Один контекст — один `yield`.
- **Путать порядок выхода при вложенных with:** выход идёт в порядке LIFO (последний вошёл — первый вышел). При отладке утечек ресурсов проверять, что все контексты действительно выходят (логировать в `__exit__`).
- **Использовать `async with` с обычным (синхронным) контекстным менеджером:** синтаксически не сработает — у объекта должны быть `__aenter__` и `__aexit__`. Для синхронного объекта в async-коде иногда используют `await loop.run_in_executor` или обёртку.
- **Исключение в коде до yield не откатывать:** если в `@contextmanager` вы приобрели ресурс до `yield`, а затем исключение произошло до первого `yield`, код после `yield` не выполнится. Нужно освобождать уже приобретённые ресурсы в **except** внутри функции до повторного **raise**.
- **Поднимать новое исключение в __exit__ без необходимости:** если в `__exit__` при очистке возникла ошибка и вы поднимаете новое исключение, исходное (из тела блока) будет скрыто в **__context__**. Очистку в `__exit__` оборачивайте в try/except с логированием и не поднимайте новое исключение, если можно.
- **Использовать `with obj:` когда obj не контекстный менеджер:** объект без **__enter__**/ **__exit__** вызовет **AttributeError** при входе в блок. Убедитесь, что выражение справа от **with** возвращает контекстный менеджер (например, **open(path)** возвращает файловый объект с протоколом, а не сам тип **open**).
- **Путать «with open(path) as f» — f это файл:** переменная **f** — это **файловый объект** (результат открытия), а не тип **open** и не «менеджер открытия». У **f** есть методы **read**, **write**, **close** и т.д.; протокол **with** реализован у файлового объекта.

## Отладка и диагностика (Часть V)

- **Ресурс не освобождается (файл остаётся открытым, блокировка не снимается):** убедиться, что используется именно `with obj:` и что выход из блока происходит (нет бесконечного цикла или исключения до входа в контекст). Проверить, что в `__exit__` или после `yield` действительно вызывается освобождение; при исключении в самом `__exit__` исходный ресурс может не освободиться — обернуть очистку в try/finally внутри `__exit__`.
- **Исключение «пропало» после with:** проверить возвращаемое значение `__exit__`. Если возвращается `True`, исключение подавляется; поменять на `False` или убрать return, если исключение должно пробрасываться.
- **Исключение в `__enter__`:** в этом случае `__exit__` не вызывается. Если ресурс был частично приобретён в `__enter__`, обрабатывать это внутри `__enter__` (try/except и откат) или документировать, что при ошибке входа объект остаётся в согласованном состоянии.
- **ExitStack: не все контексты закрылись:** убедиться, что каждый `enter_context(cm)` выполняется до того, как может произойти исключение, и что при исключении уже вошедшие контексты откатываются (ExitStack делает это сам). Проверить порядок: последний добавленный закрывается первым.
- **@contextmanager: генератор не доходит до кода после yield:** при исключении в теле `with` исключение передаётся в генератор; если в функции нет `try/except` или `finally`, код после `yield` может не выполниться. Всегда использовать `try: yield ... finally: cleanup()`.
- **Проверить порядок вызова __exit__ при вложенных with:** при отладке утечек или «странного» порядка закрытия добавьте логирование в `__exit__` (например, print с именем ресурса). Убедитесь, что выход идёт в порядке LIFO.
- **Проверить exc_type/exc_val в __exit__:** при отладке подавления исключений выведите в лог **exc_type**, **exc_val** в начале `__exit__`; убедитесь, что возвращаете **True** только для тех типов, которые действительно хотите подавить.
- **async with: объект не поддерживает асинхронный протокол:** ошибка вида «object has no attribute '__aenter__'» означает, что вы передали синхронный контекстный менеджер в **async with**. Используйте объект с **__aenter__** и **__aexit__** (например, **asyncio.Lock()**, **asyncio.timeout()**) или обёртку для синхронного ресурса.

## Вопросы и задания (Часть V)

### Вопросы по теме с краткими ответами (под спойлером)

<details>
<summary><strong>1.</strong> В каком порядке вызываются <code>__enter__</code> и <code>__exit__</code> при <code>with A(), B():</code>?</summary>

Вход: сначала **A.__enter__()**, затем **B.__enter__()** (слева направо). Выход: сначала **B.__exit__(...)**, затем **A.__exit__(...)** (LIFO — последний вошёл, первый вышел).
</details>

<details>
<summary><strong>2.</strong> Что будет, если при <code>with A(), B():</code> исключение произойдёт при вызове <code>B.__enter__()</code>?</summary>

Тело блока не выполняется. Вызывается только **A.__exit__(exc_type, exc_val, exc_tb)** — контекст A уже был открыт, его нужно закрыть. Исключение затем пробрасывается дальше (если A.__exit__ не вернула True).
</details>

<details>
<summary><strong>3.</strong> Что произойдёт, если в <code>__exit__</code> вернуть <code>True</code> при исключении в теле блока?</summary>

Исключение будет **подавлено**: выполнение продолжится со следующей строки после блока <code>with</code>, вызывающий код не увидит исключение. Уместно только при осознанной обработке (например, логирование и решение не пробрасывать).
</details>

<details>
<summary><strong>4.</strong> Зачем в <code>@contextmanager</code> оборачивать код после <code>yield</code> в <code>finally</code>?</summary>

Чтобы код очистки (освобождение ресурса) выполнился **при любом** выходе из блока — в том числе при исключении в теле <code>with</code>. Без <code>finally</code> при исключении генератор получит его через <code>throw()</code>, и код после <code>yield</code> мог бы не выполниться.
</details>

<details>
<summary><strong>5.</strong> Когда использовать <code>ExitStack</code>, а когда <code>with A, B, C:</code>?</summary>

**ExitStack** — когда число контекстов **неизвестно заранее** или задаётся данными (список путей, конфиг). **with A, B, C:** — когда контекстов **фиксированное** небольшое число и они известны в момент написания кода.
</details>

<details>
<summary><strong>6.</strong> Что делает <code>contextlib.closing(obj)</code>?</summary>

При выходе из блока вызывает **obj.close()**. Объект **obj** не обязан быть контекстным менеджером; ему достаточно иметь метод **close()**. Удобно для API (например, urllib), которые возвращают объект с <code>close()</code>, но без протокола <code>with</code>.
</details>

<details>
<summary><strong>7.</strong> Можно ли использовать <code>async with</code> с обычным объектом, у которого есть только <code>__enter__</code>/<code>__exit__</code>?</summary>

Нет. Для <code>async with</code> нужны методы **__aenter__** и **__aexit__** (корутины). Обычный контекстный менеджер блокировал бы event loop при входе/выходе. Для синхронного ресурса в async-коде используют, например, <code>await loop.run_in_executor()</code> или обёртку.
</details>

<details>
<summary><strong>8.</strong> Что произойдёт, если в <code>__exit__</code> при очистке ресурса возникнет исключение?</summary>

Новое исключение **заменит** исходное (из тела блока): вызывающий код увидит только исключение из <code>__exit__</code>, а исходное будет сохранено в атрибуте **__context__** нового исключения (implicit chaining). Поэтому очистку в <code>__exit__</code> обычно оборачивают в **try/except** с логированием и не поднимают новое исключение, либо в **try/finally**, чтобы гарантировать выполнение очистки.
</details>

<details>
<summary><strong>9.</strong> Зачем в <code>@contextmanager</code> оборачивать код после <code>yield</code> в <code>try/finally</code>?</summary>

Чтобы код очистки (освобождение ресурса) выполнился **при любом** выходе из блока — в том числе при исключении в теле <code>with</code>. Без <code>finally</code> при исключении декоратор «вбрасывает» исключение в генератор через <code>throw()</code>; если после <code>yield</code> нет <code>try/except</code> или <code>finally</code>, код очистки может не выполниться (генератор завершится с исключением до выполнения кода после yield). С <code>try: yield ... finally: cleanup()</code> блок **finally** выполняется всегда.
</details>

<details>
<summary><strong>10.</strong> Что попадёт в переменную после <code>as</code>, если в <code>@contextmanager</code> написать просто <code>yield</code> без значения?</summary>

В переменную попадёт **None**. Выражение **yield** без аргумента «возвращает» **None** наружу; декоратор передаёт это значение как результат **__enter__**. Удобно для контекстов, где в теле блока не нужен «результат входа» (например, смена директории, подмена stdout).
</details>

<details>
<summary><strong>11.</strong> Как проверить, поддерживает ли объект протокол контекстного менеджера?</summary>

Проверить наличие методов **__enter__** и **__exit__**: <code>hasattr(obj, '__enter__') and hasattr(obj, '__exit__')</code>. Объект, возвращаемый **open(path)** (файловый объект), поддерживает протокол; сам тип **open** при вызове **open(path)** возвращает такой объект.
</details>

## Ответы и подсказки к тестовым заданиям (Часть V)

1. **Timer:** в **__enter__** сохранить **time.monotonic()** (или **time.perf_counter()**) в атрибут; в **__exit__** вычислить разницу с текущим временем и напечатать. Возвращать **self** из **__enter__**, чтобы в **as t** можно было использовать объект при необходимости.
2. **no_raise(exception_type):** **@contextmanager** с **try: yield** и в **except** проверить **exc_type is exception_type** (или **issubclass(exc_type, exception_type)**); если да — **return** (или не пробрасывать); иначе **raise**. Либо класс с **__exit__**: при **exc_type is exception_type** вернуть **True**, иначе **False**.
3. **ExitStack:** цикл по путям; для каждого пути проверить **Path(p).exists()** (или **try: open(p)** и **enter_context** в try); **stack.enter_context(open(p))** для существующих; второй цикл — читать первую строку из каждого файла и выводить. При выходе из **with ExitStack()** все файлы закроются.
4. **@asynccontextmanager:** **async def**: до **yield** — **print("start")**; после **yield** в **finally** — **print("end")**. Вызвать с **async with** из корутины и **asyncio.run(main())**.

## Резюме по Части V

**Часть V** охватывает контекстные менеджеры и работу с ресурсами:

- **§20** — синхронные контекстные менеджеры: протокол `__enter__`/`__exit__` (включая схему потока выполнения, проверку протокола через hasattr, эквивалентность `with` и try/except/finally, break/return/continue внутри блока, что возвращать из `__enter__`, объект traceback), обработка исключений в `__exit__` (возврат `True` — подавить; очистка без потери исходного исключения; BaseException vs Exception), `@contextmanager` с `yield` (в т.ч. yield без значения, внутренняя механика, исключение до/после yield, вложенный @contextmanager), сводка по **suppress**, **redirect_stdout**, **nullcontext**, вложенные `with` (порядок вызовов, исключение при открытии второго контекста), **ExitStack** (enter_context, push, pop_all, partial), **closing** и **chdir** (3.11+, Path).
- **§20a** — асинхронные: протокол `__aenter__`/`__aexit__`, пошаговый поток `async with`, вложенный async with, `@asynccontextmanager`, **asyncio.timeout** (3.11+, в т.ч. delay=None, обработка TimeoutError).

Дополнительно: понятие **ресурса** и **утечки ресурса**, практические рекомендации (когда класс, когда @contextmanager, когда ExitStack), сводная таблица «когда что использовать», примеры из практики, связь с другими темами плана (исключения, генераторы, contextlib, asyncio, магические методы).

Использование `with` и `async with` обеспечивает гарантированное освобождение ресурсов и единообразный стиль кода; протокол и утилиты **contextlib** позволяют и реализовывать свои контексты, и комбинировать готовые.

**Связь с другими частями плана:** Часть III (§14) — модуль **contextlib** и его функции; Часть VI — **asyncio** и асинхронные примитивы; Часть VII / Часть X — магические методы **__enter__**/ **__exit__** в общем обзоре протоколов.
