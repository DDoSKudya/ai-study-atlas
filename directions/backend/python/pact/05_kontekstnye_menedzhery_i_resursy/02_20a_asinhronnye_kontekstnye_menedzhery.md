## §20a. Асинхронные контекстные менеджеры

**Цель раздела §20a:** научиться использовать асинхронные контекстные менеджеры в коде **async/await**: понимать протокол **__aenter__**/ **__aexit__**, синтаксис **async with**, декоратор **@asynccontextmanager** и утилиту **asyncio.timeout** (3.11+). В итоге вы сможете гарантированно освобождать асинхронные ресурсы (соединения, блокировки) и ограничивать время ожидания операций.

В асинхронном коде (**async/await**) ресурсы тоже нужно приобретать и освобождать. Для этого используется **асинхронный контекстный менеджер**: объект с методами **`__aenter__`** и **`__aexit__`**, которые являются **корутинами** (вызываются с **await**). Синтаксис: **`async with obj as x:`** — при входе вызывается **`await obj.__aenter__()`**, при выходе — **`await obj.__aexit__(exc_type, exc_val, exc_tb)`**. Семантика совпадает с синхронным протоколом: гарантированный выход при нормальном завершении, при исключении и при return/break; возврат **True** из **__aexit__** подавляет исключение.

**Почему отдельный протокол:** в синхронном коде **__enter__** и **__exit__** — обычные вызовы. В асинхронном коде приобретение и освобождение ресурса могут быть операциями ввода-вывода (установка соединения с БД, ожидание освобождения сокета). Поэтому протокол сделан асинхронным: **__aenter__** и **__aexit__** — корутины, их нужно **await**. Так event loop может переключаться на другие задачи во время ожидания; использование синхронного протокола в **async with** привело бы к блокировке цикла.

### 20a.1 Протокол `__aenter__`, `__aexit__`

**Цель подраздела:** понимать асинхронный протокол контекстного менеджера: методы **__aenter__** и **__aexit__** — корутины; порядок вызовов и аргументы **__aexit__** совпадают с синхронным протоколом; возврат **True** из **__aexit__** подавляет исключение.

| Метод | Сигнатура | Назначение |
|-------|-----------|------------|
| **`__aenter__(self)`** | Асинхронный метод (**async def**) | Вызывается при входе в **async with** как **await obj.__aenter__()**. Результат попадает в переменную после **`as`**. |
| **`__aexit__(self, exc_type, exc_val, exc_tb)`** | Асинхронный метод (**async def**) | Вызывается при выходе как **await obj.__aexit__(exc_type, exc_val, exc_tb)**. Аргументы — как у синхронного **__exit__**. Возврат **True** подавляет исключение. |

Семантика совпадает с синхронным протоколом: вход → тело блока → выход; при исключении в теле вызывается **__aexit__** с параметрами исключения; возврат **True** подавляет исключение. **Запомните:** объект с **__enter__**/ **__exit__** **не** подходит для **async with**; для **async with** нужны **__aenter__** и **__aexit__**. В стандартной библиотеке **asyncio.Lock**, **asyncio.Semaphore**, **asyncio.Queue** и др. поддерживают **async with**.

**Пошагово: что происходит при выполнении `async with obj as x: body`**

1. Вычисляется выражение **obj** (справа от **async with**).
2. Вызывается **await obj.__aenter__()**. Результат присваивается переменной **x** (если указано **as x**).
3. Выполняется **тело блока** **body** (асинхронно; в нём могут быть **await**).
4. При **нормальном** завершении тела вызывается **await obj.__aexit__(None, None, None)**.
5. При **исключении** в теле вызывается **await obj.__aexit__(exc_type, exc_val, exc_tb)**; возврат **True** подавляет исключение, **False** или отсутствие return — пробрасывает.

**Вложенный async with:** порядок входа и выхода такой же, как у синхронного: вход слева направо, выход LIFO. Пример: **async with lock1: async with lock2: ...** — при входе сначала **await lock1.__aenter__()**, затем **await lock2.__aenter__()**; при выходе сначала **await lock2.__aexit__(...)**, затем **await lock1.__aexit__(...)**.

**Пример: простой async lock (иллюстрация протокола)**

```python
class AsyncSimpleLock:
    def __init__(self):
        self.locked = False

    async def __aenter__(self):
        self.locked = True
        return self

    async def __aexit__(self, exc_type, exc_val, exc_tb):
        self.locked = False
        return False

async def main():
    async with AsyncSimpleLock() as lock:
        print(lock.locked)  # True
    print(lock.locked)      # False
```

### 20a.2 async with и @asynccontextmanager

**Цель подраздела:** использовать **async with** для асинхронных ресурсов и описывать асинхронный контекст через **@asynccontextmanager** (async-генератор с одним **yield**); понимать, что правила те же, что у синхронного **@contextmanager** (один yield, при необходимости try/finally после yield).

---

**`async with obj as x:`** — асинхронный аналог **`with`**. Выражение **obj** должно быть объектом с **`__aenter__`** и **`__aexit__`** (оба — корутины). При входе вызывается **await obj.__aenter__()**, результат попадает в **x**; при выходе вызывается **await obj.__aexit__(...)**. Использование то же: гарантированный вход и выход в async-потоке; при исключении в теле блока **__aexit__** вызывается с параметрами исключения.

**`contextlib.asynccontextmanager`** — декоратор для **async-генераторной функции** (функция с **`async def`** и **`yield`**). Код **до** **`yield`** выполняется при входе (как «setup»); код **после** **`yield`** — при выходе (как «teardown»). Значение **yield** попадает в переменную после **`as`**. Правила те же, что у синхронного **@contextmanager**: ровно один **yield**; при необходимости оборачивать очистку в **try/finally**, чтобы она выполнялась при любом исходе.

**Пример:**

```python
from contextlib import asynccontextmanager

@asynccontextmanager
async def async_resource():
    # вход
    resource = await acquire_async()
    try:
        yield resource
    finally:
        await release_async(resource)

async def main():
    async with async_resource() as r:
        await use(r)
```

Правила те же, что у синхронного `@contextmanager`: один `yield`, при необходимости обрамлять в try/finally, чтобы освобождение выполнялось при любом исходе.

### 20a.3 asyncio.timeout (Python 3.11+)

**Цель подраздела:** использовать **asyncio.timeout(delay)** (3.11+) для ограничения времени выполнения блока **async with**; понимать, что при превышении времени текущая задача отменяется и возбуждается **TimeoutError** (или подкласс); знать, что **delay=None** отменяет таймаут (удобно во вложенных timeout).

---

В **Python 3.11** в **asyncio** добавлен асинхронный контекстный менеджер **`asyncio.timeout(delay)`** (или **`asyncio.timeout(seconds=...)`**). Внутри блока **`async with asyncio.timeout(5):`** любая операция, занимающая больше **5** секунд, приведёт к **отмене** текущей задачи (как при **task.cancel()**) и к возбуждению **asyncio.TimeoutError** (или подкласса). Удобно ограничивать время ожидания одной операции или группы операций без ручного создания таймера и отмены задачи. Передача **`delay=None`** отменяет таймаут (удобно во вложенных **async with asyncio.timeout(...)** когда внутренний блок должен выполняться без ограничения по времени).

**Пример:**

```python
import asyncio

async def main():
    try:
        async with asyncio.timeout(2):
            await asyncio.sleep(5)  # через 2 секунды — TimeoutError
    except TimeoutError:
        print("Истекло время ожидания")

asyncio.run(main())
```

**Обработка asyncio.TimeoutError при использовании asyncio.timeout:** при превышении времени текущая задача отменяется (как при **task.cancel()**), и в блоке **async with** возбуждается **asyncio.TimeoutError** (или подкласс). Его можно поймать снаружи блока:

```python
import asyncio

async def main():
    try:
        async with asyncio.timeout(2):
            await asyncio.sleep(5)
    except asyncio.TimeoutError:
        print("Операция не успела завершиться за 2 секунды")

asyncio.run(main())
```

#### Теоретическое обоснование: зачем отдельный асинхронный протокол

В синхронном коде вызов `obj.__enter__()` и `obj.__exit__(...)` — обычные вызовы. В асинхронном коде **приобретение** и **освобождение** ресурса могут быть операциями ввода-вывода (например, установка соединения с БД, освобождение сокета). Поэтому протокол сделан асинхронным: **`__aenter__`** и **`__aexit__`** — корутины, их нужно **await**. Так event loop может переключаться на другие задачи во время ожидания. Использование синхронного протокола в `async with` привело бы к блокировке цикла на время входа/выхода.

**Запомните:** объект с `__enter__`/`__exit__` не подходит для `async with`; для него нужны `__aenter__`/`__aexit__`. В стандартной библиотеке: **asyncio.Lock**, **asyncio.Semaphore**, **asyncio.Queue** и др. поддерживают `async with`.

Подробнее об **asyncio** см. глобальный план **Часть VI (§21)**.
