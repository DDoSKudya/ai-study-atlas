### Термины

| Термин | Определение |
|--------|-------------|
| **Миграция (migration)** | Переход кода/зависимостей/окружения на новую платформу (Py2→Py3) с сохранением поведения. |
| **Рефакторинг (refactoring)** | Изменение структуры кода без изменения внешнего поведения. Автоконвертация — это «массовый рефакторинг». |
| **Fixer** | Правило преобразования исходного кода (например, заменить `print x` на `print(x)`). |
| **2to3** | Инструмент, который прогоняет набор fixers по исходникам Py2 и переписывает код ближе к Py3. |
| **lib2to3** | Библиотека, на которой основан 2to3 (парсер + fixers). |
| **Dual-support** | Период, когда один код должен работать и на Python 2.7, и на Python 3.x. |
| **futurize** | Инструмент из пакета `future`, генерирующий правки для dual-support (этапы stage1/stage2). |

### 35.1. `2to3`: что делает и как применять

#### Что делает 2to3 (приближённая модель)

`2to3` умеет **механически** преобразовывать некоторые распространённые конструкции Py2 в их Py3-эквиваленты:

- `print x` → `print(x)`
- `except E, e:` → `except E as e:`
- `xrange(...)` → `range(...)`
- `dict.iteritems()` → `dict.items()` (и похожие методы)
- импорты и некоторые API стандартной библиотеки (частично)

То есть `2to3` — это **не «магическая миграция»**, а **первый проход по синтаксису и тривиальным API**.

#### Как 2to3 работает «под капотом» (простая, но полезная модель)

Полезно один раз понять механику 2to3 — это сразу объясняет, почему он «не всесилен».

1) `2to3` **парсит** исходный код (строит дерево разбора).  
2) Применяет набор **fixers** (правил преобразования), которые сопоставляют шаблоны в дереве и заменяют их на новые узлы.  
3) Генерирует обновлённый исходник (pretty-print) и либо показывает diff, либо перезаписывает файлы.

Отсюда следуют ограничения:

- если код не парсится (нестандартные ходы, сломанный синтаксис, очень «динамический» кодогенератор) — инструмент не поможет;
- если преобразование зависит от смысла (например, где у вас «текст», а где «байты») — инструмент не может гарантировать корректность;
- многие исправления «возможны», но опасны, потому что меняют поведение — поэтому 2to3 их не делает или делает частично.

#### Практический минимум опций 2to3 (что реально нужно знать)

| Опция | Смысл | Практический совет |
|------|-------|--------------------|
| (без `-w`) | показать diff/вывод, не переписывая файлы | Всегда делайте первый прогон без `-w` |
| `-w` | переписать файлы на месте | Применяйте в отдельной ветке/коммите |
| `-n` | не создавать `.bak` | Если код под Git — почти всегда |
| `-o DIR` | писать результат в отдельный каталог | Удобно для сравнения «до/после» без порчи исходников |
| `-f FIX` | применить только выбранный fixer | Для «дозирования» правок и отладки проблемного fix |
| `-x FIX` | исключить fixer | Если конкретное правило ломает стиль/поведение в вашем коде |
| `-l` | показать список fixers | Чтобы понять, что именно может меняться |
| `-j N` | параллельная обработка | На больших репозиториях ускоряет прогон |

**Пример «сначала посмотреть, потом применить»:**

```bash
# Предпросмотр: никаких изменений в файлах
python -m lib2to3 path/to/your_package

# Применение: перезапись файлов, без .bak
python -m lib2to3 -w -n path/to/your_package
```

> Практическое правило: **никогда не начинайте с `-w`**, пока не увидели характер диффа и не поняли, что именно будет переписано.

#### Почему 2to3 не решает миграцию целиком

Потому что самые болезненные изменения Py2→Py3 — **семантические**:

- В Py3 `str` — это **Unicode-текст**, а `bytes` — **байты**. В Py2 `str` часто означал «байты», а `unicode` — «текст».
- В Py3 многие вещи стали **итераторами/ленивыми объектами** (например, `map`, `zip`, `dict.items()` возвращает view).
- Изменились некоторые правила сравнения, сортировки, поведения исключений, импортов.

Инструмент не может безопасно понять вашу бизнес-логику: где строка — это «текст», а где — «сырой протокол в байтах».

#### Мини-чеклист ревью после автоконвертации 2to3

Даже если проект «собрался», часто остаются поломки поведения. Быстрый список «куда смотреть в первую очередь»:

- **Границы I/O:** файлы, сеть, сокеты, HTTP, subprocess, работа с БД-драйвером.  
  Здесь чаще всего всплывают `str/bytes` и кодировки.
- **Итераторы:** `map/filter/zip`, `dict.items()/keys()/values()`, `range`.  
  Если код ожидал список, в Py3 он может получить «одноразовый» итератор/view.
- **Сортировки/сравнения:** Py2 мог «терпеть» смешанные типы, Py3 чаще бросает `TypeError`.  
  Проверяйте `sorted(...)` и места, где ключи/значения могли быть разных типов.
- **Логирование/печать:** если в Py2 вы печатали bytes как str, в Py3 это станет `b'...'` или вызовет ошибки при форматировании.

#### Практический workflow с 2to3 (безопасный)

1) **Заморозьте поведение:** прогоните тесты на Py2.7 (если они есть), зафиксируйте baseline.

2) **Сделайте ветку миграции** (или отдельную папку/коммит-границу) — автоправки должны быть отделены от ручных.

3) **Запустите 2to3 так, чтобы он не трогал файлы без необходимости**, и вы могли откатить изменения.

Типичная команда (классическая форма):

```bash
2to3 -w -n path/to/your_package
```

Где:
- `-w` — переписать файлы «на месте» (write)
- `-n` — не создавать backup-файлы `.bak`

Если `2to3` как отдельная команда не доступен, часто работает эквивалент:

```bash
python -m lib2to3 -w -n path/to/your_package
```

> Важно: в современных версиях Python `lib2to3` помечен как устаревший и может отсутствовать. Если у вас Python 3.13+ и `lib2to3` не найден — используйте отдельную среду с Python 3.10/3.11 только для запуска конвертера или переходите на альтернативы (см. ниже).

#### Минимальный пример «до/после»

**До (Py2):**

```python
# -*- coding: utf-8 -*-

def read_lines(path):
    f = open(path)
    try:
        return map(lambda s: s.strip(), f.readlines())
    finally:
        f.close()

try:
    print "OK"
except Exception, e:
    print e
```

**После 2to3 (приблизительно):**

```python
def read_lines(path):
    f = open(path)
    try:
        return [s.strip() for s in f.readlines()]
    finally:
        f.close()

try:
    print("OK")
except Exception as e:
    print(e)
```

Заметьте:
- `map(...)` мог стать list comprehension (или остаться `map`, зависит от fixer’ов и контекста).
- **Кодировки и `open()`** 2to3 обычно не исправляет «как надо» (про это — §37).

#### Ограничения и «подводные камни» 2to3

| Проблема | Почему возникает | Что делать |
|---------|------------------|-----------|
| `str/bytes` по-прежнему сломаны | 2to3 не знает, где у вас текст, а где байты | Ручная правка + тесты + явные `encode/decode` на границах |
| Неполное исправление stdlib | API менялось не только механически | Используйте `six.moves`/`future.standard_library` временно или переписывайте на Py3 API |
| Конвертация даёт «компилируется, но не работает» | Семантика изменилась (итераторы, сравнения) | Догонять тестами + ревью ключевых мест |
| Проблемы с современным синтаксисом | lib2to3 «не любит» новые конструкции Python | Запускайте конвертацию на исходном Py2-коде до внедрения Py3-only синтаксиса |

#### «Альтернативы» (коротко, если 2to3 недоступен)

В реальных проектах часто используют:

- **`python-modernize`** (на базе `six`) — помогает для dual-support, модернизирует код под `six`-швы.
- **`pyupgrade`/Ruff** — обновляет уже Py3-код (например, `format()` → f-строки и т.д.). Это не «Py2→Py3», а «Py3→более современный Py3».

Если ваша цель — именно миграция с Py2, то эти инструменты обычно идут **после** первичного перехода.

### 35.2. `futurize` (пакет `future`): миграция с периодом dual-support

#### Идея futurize простыми словами

Если вы не можете «разом» перейти на Python 3 (например, прод окружение или часть зависимостей ещё Py2), часто делают **переход в два шага**:

1) Сначала доводят код до «хорошего Py2» (без ловушек, с `from __future__ import ...`), чтобы он **уже выглядел как Py3**, но всё ещё работал на Py2.
2) Затем добавляют слой совместимости, чтобы тот же код работал и на Py3.

`futurize` автоматизирует обе стадии.

#### Два этапа futurize

- **Stage 1** — «безопасные» изменения, которые работают и в Py2, и в Py3:
  - добавление `from __future__ import print_function, division, absolute_import, unicode_literals` (в зависимости от режима)
  - исправления, не меняющие поведение слишком резко

- **Stage 2** — изменения, которые обычно требуют `future`-обёрток/импортов:
  - добавление импортов из `builtins` (пакет `future`)
  - подмена частей stdlib через `future.standard_library`

Пример команд:

```bash
# Сначала "подготовка" (stage1)
futurize --stage1 -w -n path/to/your_package

# Потом dual-support (stage2)
futurize --stage2 -w -n path/to/your_package
```

#### Что именно меняют `__future__`-импорты (и почему stage1 действительно важен)

В Python 2.7 директива `from __future__ import ...` включает поведение, которое в Python 3 стало поведением «по умолчанию». Поэтому stage1 — это не «косметика», а способ заранее устранить классы проблем, которые иначе всплывут позже и в более хаотичном виде.

| `__future__` import | Что меняется в Python 2.7 | Типовые последствия в коде | Зачем это для миграции |
|---------------------|---------------------------|-----------------------------|------------------------|
| `print_function` | `print` превращается в функцию | `print x` ломается → нужно `print(x)`; появляются `sep/end/file` | Выравнивает синтаксис и API с Py3 |
| `division` | `/` становится «настоящим» делением | `1/2` становится `0.5` (а не `0`); тесты ловят ошибки математики | Рано выявляет места, где логика зависела от целого деления |
| `absolute_import` | импорты становятся абсолютными по умолчанию | меньше сюрпризов с «локальными» модулями, которые затеняют stdlib | Уменьшает «магические импорты», делает проект структурно чище |
| `unicode_literals` | `"..."` становится `unicode`, а не `str` (bytes) | всплывают проблемы смешивания текста/байтов; начинают падать протоколы/I/O | Приближает модель строк к Py3, но требует дисциплины (см. ниже) |

**Практический вывод:** stage1 обычно помогает даже если вы планируете big-bang. Он превращает часть «скрытых» проблем в явные и проверяемые.

#### Самый частый «взрыв» — `unicode_literals`

`unicode_literals` — это главный источник боли в Py2-переходе, потому что:

- многие проекты на Py2 держали внутри системы «строки» как bytes (тип `str`);
- Python 3 требует держать текст как `str` (Unicode), а bytes отдельно.

Когда `unicode_literals` включён, строка `"abc"` в Py2 становится `unicode`. Дальше любые места, которые ожидают bytes, начинают падать.

**Классические точки, где ломается:**

- `hashlib.md5("text")` (в Py3 хэши принимают bytes; в Py2 с unicode могут происходить неявные преобразования)
- сокеты и низкоуровневый I/O: `.send()`/`.write()` ожидают bytes
- бинарные форматы/протоколы (gzip/zip, protobuf, собственные бинарные протоколы)
- конкатенация/форматирование: `"x" + b"y"` → ошибка

**Как вводить `unicode_literals` безопасно (рабочая стратегия):**

1) Не включайте его сразу «везде». Начните с 1–2 модулей, где много текстовой логики (парсинг, шаблоны, обработка сообщений).  
2) Введите правило: **внутри приложения — текст (`unicode`/`str`), на границах — bytes**.  
3) Добавьте две функции-конвертера (в идеале в одном месте: `compat/text.py`) и используйте их на границах.

Пример (dual-support, через `six`):

```python
import six

def to_text(x, encoding="utf-8", errors="strict"):
    """Возвращает text (unicode в Py2, str в Py3)."""
    if x is None:
        return None
    if isinstance(x, six.text_type):
        return x
    if isinstance(x, six.binary_type):
        return x.decode(encoding, errors)
    return six.text_type(x)

def to_bytes(x, encoding="utf-8", errors="strict"):
    """Возвращает bytes и никогда не возвращает text."""
    if x is None:
        return None
    if isinstance(x, six.binary_type):
        return x
    if isinstance(x, six.text_type):
        return x.encode(encoding, errors)
    return six.text_type(x).encode(encoding, errors)
```

4) Покройте тестами именно эти границы (файл/сеть/subprocess) — не пытайтесь тестировать «всё сразу».

**Антипаттерн:** «чинить» ошибки хаотичным `.encode("utf-8")` по месту, не понимая, где у вас текст, а где байты. Это приводит к двойному кодированию и нестабильным багам.

#### Пример: что может появиться в коде после futurize

```python
from __future__ import absolute_import, division, print_function, unicode_literals

from builtins import str
from future import standard_library
standard_library.install_aliases()

from urllib.request import urlopen
```

Такой код может быть неожиданным, если вы «не просили» его. Но смысл именно в том, чтобы:

- в Py2 получить максимально похожие на Py3 типы/импорты;
- минимизировать расхождения между ветками Py2 и Py3.

#### Когда futurize оправдан

- Вам **нужно поддерживать Py2 и Py3 одновременно** (dual-support период).
- У вас **большой код**, и вы хотите автоматизировать «тяжёлый» первичный проход.
- Вы готовы принять, что часть кода станет более «шумной» из-за импортов совместимости.

#### Когда лучше не использовать futurize

- Если Py2 уже можно отключить — чаще выгоднее **сделать миграцию на чистый Py3** без слоя `future`.
- Если проект маленький и понятный — ручной переход + тесты может быть быстрее и чище.

**Запомните:** `futurize` — инструмент для переходного периода. Он помогает «пережить» dual-support, но не должен стать вечным стилем кода.

### 35.3. Как встроить автоконвертацию в процесс разработки

Практически полезная схема, которая снижает хаос:

1) **Feature freeze** на время конвертации (или очень строгий контроль изменений).
2) **Один большой автокоммит** (только результат `2to3`/`futurize`), без ручных правок.
3) Потом **серия маленьких коммитов**: исправляем тесты, правим `str/bytes`, меняем stdlib API.
4) Параллельно держим **CI**: тесты на обеих версиях (если dual-support), либо хотя бы на целевой Py3.

Почему это важно:
- автоконвертация меняет тысячи строк — её невозможно нормально ревьюить вместе с ручными смысловыми правками;
- отделённый автокоммит позволяет проще делать `git blame` и понимать: «это инструмент сделал, или человек?».

#### Практика: разделяйте «автоправки» и «исправление поведения»

Самая частая организационная ошибка миграции — смешать в одном потоке:

- автоконвертацию (2to3/futurize) — механические правки;
- исправление ошибок поведения — смысловые изменения, часто требующие решения «как должно быть».

**Рекомендуемый паттерн коммитов:**

- **Коммит A:** только результат автоконвертации (максимально механический).  
  Цель: «получили новый исходник» — без попыток сделать его правильным.
- **Коммиты B..N:** правки поведения, *маленькими шагами* и *с тестами*.  
  Цель: «чиню конкретный класс проблем» (импорты / итераторы / строки / subprocess / сортировки).

Почему это так важно:

- автокоммит можно «принять как данность» и не пытаться ревьюить каждую строку;
- баги легче бисектить: вы сразу понимаете, на каком этапе «сломалось»;
- проще откатить/поменять стратегию (например, заменить futurize на modernize).

#### Как организовать миграцию как проект (а не как «чиню по месту»)

Миграция — это почти всегда работа уровня «мини‑проекта», особенно если кодовая база большая.

**Минимальный набор артефактов миграции:**

- **Документ “Definition of Done”**: что значит «мы перешли на Py3» (версии, окружения, CI, мониторинг).
- **Список границ I/O**: файлы, сеть, БД, subprocess, очереди, сериализация.
- **Чеклист классов проблем** (см. §37.3) и порядок фикса.
- **План dual-support** (если нужен): срок, критерии завершения, политика для нового кода.

#### Автоматические проверки на каждом «круге» правок

Идея: после каждого класса исправлений вы должны быстро получать ответ «стало лучше или хуже».

| Проверка | Что ловит | Почему полезно именно при миграции |
|---------|-----------|-------------------------------------|
| Компиляция модулей | синтаксис/ошибки импорта | быстро подтверждает, что «проект хотя бы парсится» |
| Unit-тесты | локальные регрессии | ловят изменения поведения после правок |
| Интеграционные smoke | ошибки на границах I/O | именно там всплывает `str/bytes` и stdlib переезды |

**Пример (как общий смысл):**

```bash
python -m compileall -q your_pkg
pytest -q
```

#### Порядок «пачек» исправлений (чтобы не утонуть)

Типичная ошибка — начать со строк и кодировок, когда проект ещё не импортируется. Правильнее чинить в таком порядке:

1) **Импорты и переезды stdlib** (проект должен стартовать)
2) **Итераторы vs списки** (`map/zip/dict.items/range`)
3) **`str/bytes` и кодировки на границах** (файлы/сеть/subprocess/логирование)
4) **Сортировки и сравнения** (особенно «разнотипные» коллекции)
5) **Сериализация и протоколы** (json/pickle/hashlib/base64/struct)

Так вы сохраняете возможность постоянно запускать код и тесты и не чините «вслепую».
