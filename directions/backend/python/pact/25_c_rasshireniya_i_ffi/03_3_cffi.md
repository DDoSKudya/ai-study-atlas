## 3. `cffi`: более удобный FFI (§146)

### 3.0. Зачем ещё одна библиотека, если есть `ctypes`

Проблемы `ctypes`:

- сигнатуры описываются **на языке `ctypes`**, а не на «настоящем C»;
- код быстро превращается в «суп» из `c_int`, `POINTER`, `byref`;
- поддержка PyPy и альтернативных реализаций Python ограничена.

Если сказать грубо, `ctypes` просит вас:

- думать не только как Python‑разработчик, но и как «полумануальный конвертер типов»;
- постоянно держать в голове таблицу соответствий `int`/`long`/`double` ↔ `c_int`/`c_long`/`c_double`;
- вручную строить указатели, структуры, массивы.

`cffi` решает это так:

- вы описываете интерфейс **на языке C**, почти как в заголовочном файле `.h`:
  - привычный синтаксис `int func(int a, double b);`;
  - меньше «ctypes‑магии»;
- можете работать в двух режимах:
  - **ABI mode** — без компиляции, как `ctypes`;
  - **API mode** — с компиляцией маленького C‑модуля (без ручного `setup.py`).

Мысленная аналогия:

- `ctypes` — как словарь «русский → псевдо‑английский для программистов»;
- `cffi` — как обычный англо‑русский словарь: вы работаете с теми же словами, что и в исходном C‑коде.

### 3.0.1. Как думать о `cffi` шагами

В обоих режимах (ABI и API) можно думать по одному и тому же шаблону:

1. **Что у меня за C‑функция / структура?**
   - найдите или напишите её объявление в стиле C (`.h`‑файл);
2. **Как описать её в `cffi` через `cdef`?**
   - почти всегда можно просто скопировать кусок из `.h`;
3. **Где берётся реализация?**
   - ABI: библиотека уже существует (`libc`, `libm`, ваша `libfoo.so`);
   - API: вы пишете реализацию внутри `set_source`;
4. **Какой объект нужно создать в Python, чтобы передать в функцию?**
   - `ffi.new("double[]", values)` или `ffi.from_buffer(buf)`, или простая строка/байты;
5. **Как безопасно проверить работу?**
   - сделайте несколько простых вызовов и сравните с чистым Python;
   - проверьте граничные случаи (`NULL`, пустые массивы, `n <= 0`).

### 3.1. ABI mode: быстро и без компиляции

ABI‑режим похож на `ctypes`: вы подключаетесь к уже существующей библиотеке.

Схема:

1. Найти библиотеку (`libc`, `libm`, ваша `libfoo.so`).
2. Скопировать нужные объявления функций из `.h` в `ffi.cdef(...)`.
3. Вызвать `ffi.dlopen(...)`, чтобы получить объект `lib`.
4. Вызывать `lib.func(...)` как обычные Python‑функции.

Пример: вызов `strlen` из стандартной библиотеки C.

```python
from cffi import FFI

ffi = FFI()

# Описываем только то, что нам нужно, как в .h
ffi.cdef("size_t strlen(const char *s);")

# dlopen("c") — особый случай: стандартная библиотека
libc = ffi.dlopen(None)  # на Linux можно явно указать "libc.so.6"

text = "hello"
data = text.encode("utf-8")

length = libc.strlen(data)
print(length)  # 5
```

Особенности (разбираем по шагам):

- `ffi = FFI()`:
  - создаём объект, через который будем всё описывать и вызывать;
- `ffi.cdef("size_t strlen(const char *s);")`:
  - буквально «копируем» объявление функции из заголовка `<string.h>`;
  - `cffi` теперь знает:
    - каково имя функции;
    - какие у неё аргументы и тип результата;
- `ffi.dlopen(None)`:
  - загружает стандартную C‑библиотеку;
  - на Linux можно явно указать `"libc.so.6"`;
  - на других ОС имена будут другими (это нужно смотреть в документации/поиском).

Плюсы ABI‑режима:

- код очень похож на обычное использование C‑заголовков;
- меньше `ctypes`‑специфичного синтаксиса;
- не нужно писать/собирать свой C‑код, если библиотека уже существует.

### 3.1.1. Чек‑лист перед использованием ABI‑режима

Перед тем как вызывать функцию через ABI‑режим, ответьте себе:

1. **Я точно знаю сигнатуру функции?**
   - посмотрели `.h` или документацию;
   - не придумываем `int`, где на самом деле `long`/`size_t`.
2. **Я правильно закодировал строки и массивы?**
   - строки → `bytes` с верной кодировкой (`utf-8` и т.д.);
   - массивы → `ffi.new("type[]", [...])` или `ffi.from_buffer(...)`.
3. **Я понимаю, кто владеет памятью?**
   - если функция возвращает `char *`, нужно посмотреть в документации:
     - кто освобождает память — библиотека или вызывающий код;
   - при сомнениях лучше использовать API‑режим и оборачивать более явно.

### 3.1.2. Типичные ошибки в ABI‑режиме

- **Неверная сигнатура в `cdef`.**
  - В `.h` написано `size_t`, а вы написали `int`.
  - Может всё «работать», пока строки короткие, а на длинных вы увидите странные числа.

- **Неправильное имя библиотеки в `dlopen`.**
  - Имя `libc.so.6` может отличаться между дистрибутивами и платформами.
  - Иногда проще использовать `None` (текущий процесс) или поискать реальное имя через `ldconfig -p` (Linux).

- **Игнорирование `errno` и кодов ошибок.**
  - Многие C‑функции в случае ошибки:
    - возвращают специальное значение;
    - или выставляют `errno`.
  - Если вы не проверяете эти коды, вы можете не заметить, что функция работала с ошибкой.

### 3.2. API mode: собственный мини‑модуль на C

API‑режим — более мощный: вы пишете небольшой кусок C‑кода,  
`cffi` компилирует его и даёт вам Python‑модуль.

Пример: модуль, который считает `average` по массиву double.

```python
# file: avg_build.py
from cffi import FFI

ffi = FFI()

ffi.cdef(
    """
    double average(const double *values, int n);
    """
)

ffi.set_source(
    "_avg_cffi",  # имя модуля
    r"""
    double average(const double *values, int n) {
        if (n <= 0) {
            return 0.0;
        }
        double sum = 0.0;
        for (int i = 0; i < n; i++) {
            sum += values[i];
        }
        return sum / n;
    }
    """,
)

if __name__ == "__main__":
    ffi.compile(verbose=True)
```

Сборка:

```bash
python avg_build.py
```

Появится файл `_avg_cffi.*.so` (или `.pyd` на Windows).

Использование:

```python
from _avg_cffi import ffi, lib

values = [1.0, 2.0, 3.0, 4.0]
n = len(values)

# Создаём C‑массив double[n]
c_arr = ffi.new("double[]", values)

result = lib.average(c_arr, n)
print(result)  # 2.5
```

Здесь:

- `ffi.new("double[]", values)`:
  - выделяет C‑массив;
  - инициализирует его значениями из Python‑списка;
- `lib.average` — C‑функция, доступная как метод объекта `lib`.

Чтобы не запутаться, можно проговаривать вслух:

1. «Я объявляю функцию **в `cdef`** — это как `.h`‑файл».
2. «Я даю `set_source` **кусок C‑кода** — это как `.c`‑файл».
3. «Я вызываю `ffi.compile()` — это как `gcc average.c -shared ...`».
4. «Я импортирую модуль и **вызываю C‑функцию через `lib`**, передавая ей C‑массив, созданный `ffi.new`».

### 3.2.1. Типичные ошибки в API‑режиме

- **Несовпадение между `cdef` и реализацией.**
  - В `cdef` вы написали `double average(const double *values, int n);`,
  - а в C‑коде — `float average(...)` или другой порядок аргументов.
  - Итог — странные числа или краши.  
  Решение: всегда сначала написать `cdef` по заголовку, потом на его основе реализацию.

- **Забыли добавить `#include` для стандартных функций.**
  - Например, используете `memcpy`, `strlen`, `toupper`, но не подключили `<string.h>` / `<ctype.h>`.
  - В лучшем случае компилятор ругнётся, в худшем — будет неявное объявление с неверным типом.

- **Слишком много логики в одном куске C‑кода.**
  - Для обучения и отладки лучше писать очень маленькие функции:
    - «посчитать сумму»;
    - «подсчитать вхождения символа»;
    - «сдвинуть координату».
  - Крупную логику проще сначала реализовать и протестировать как отдельную C‑программу, а уже потом переносить в `set_source`.

### 3.3. Работа с памятью: `ffi.new()`, `ffi.from_buffer()`

Ключевые функции:

- `ffi.new("type *")` / `ffi.new("type[]")` — выделяют память в стиле C;
- `ffi.from_buffer(py_buffer)` — создаёт C‑указатель к уже существующему буферу.

Пример: изменяем Python‑`bytearray` из C:

```python
# file: upper_build.py
from cffi import FFI

ffi = FFI()
ffi.cdef(
    """
    void to_upper(unsigned char *data, int n);
    """
)

ffi.set_source(
    "_upper_cffi",
    r"""
    #include <ctype.h>

    void to_upper(unsigned char *data, int n) {
        for (int i = 0; i < n; i++) {
            data[i] = (unsigned char) toupper(data[i]);
        }
    }
    """,
)

if __name__ == "__main__":
    ffi.compile(verbose=True)
```

Сборка:

```bash
python upper_build.py
```

Использование:

```python
from _upper_cffi import ffi, lib

buf = bytearray(b"Hello, world!")

c_buf = ffi.from_buffer(buf)
lib.to_upper(c_buf, len(buf))

print(buf)  # b'HELLO, WORLD!'
```

Здесь:

- `bytearray` хранится в памяти Python;
- `ffi.from_buffer` создаёт C‑представление того же буфера;
- C‑функция изменяет данные «на месте».

Важно:

- **не храните C‑указатели дольше, чем живёт Python‑объект**,  
  из которого они были получены.

### 3.4. PyPy‑совместимость и производительность

`cffi`:

- задумывался как FFI, который хорошо работает не только на CPython, но и на **PyPy**;
- часто показывает **лучшие результаты**, чем `ctypes`,  
  благодаря более прямому соответствию C‑типам.

Если вы планируете:

- разворачивать код на разных реализациях Python;
- использовать сложные структуры данных C;

— `cffi` обычно является **лучшим выбором**, чем `ctypes`.

### 3.5. Мини‑упражнение

1. Реализуйте через `cffi` C‑функцию `int count_char(const char *s, char ch)`,
   которая считает количество вхождений символа в строку.
2. В Python:
   - передайте ей обычную Python‑строку;
   - проверьте, что результат совпадает с `s.count(...)`.

### 3.6. Дополнительные заметки и подводные камни `cffi`

- **Смешивание `ffi.new` и `malloc/free`.**  
  Старайтесь не вызывать вручную `malloc`/`free` внутри C‑кода, который идёт через `cffi`, без крайней необходимости.  
  Если память выделена через `ffi.new`, то `cffi` сам корректно освободит её, когда Python‑объект больше не нужен.

- **Использование `ffi.string()`.**  
  Для превращения `char *` в Python‑байты используйте `ffi.string(ptr)`.  
  Это делает копию строки, обрезая по нулевому байту. Если вам нужен срез определённой длины, можно передать длину вторым аргументом.

- **Структуры и выравнивание.**  
  При работе со структурами (`struct`) всегда:
  - описывайте их в `cdef` **точно так же**, как в реальном `.h`‑файле;
  - не меняйте порядок полей;
  - не вставляйте «лишних» полей.  
  Иначе вы получите несоответствие выравнивания и странные значения в полях.

- **Отладка через временный C‑`main`.**  
  Если что‑то ведёт себя странно, очень полезно:
  - написать маленькую C‑программу с `main`, которая вызывает вашу функцию напрямую;
  - убедиться, что без Python всё работает, как нужно;  
  - только потом возвращаться к FFI‑обёртке.

- **Логирование и печать из C.**  
  На этапе отладки не бойтесь использовать `printf` внутри C‑кода, который вы компилируете в API‑режиме:
  - так вы увидите реальные значения аргументов и индексов;
  - главное — убрать лишние `printf` перед продакшеном.

- **Проверка границ.**  
  Всегда думайте: «а что будет, если сюда придёт `n <= 0`, `NULL`, пустая строка и т.п.?».  
  В C лучше явно проверять:

  ```c
  if (values == NULL || n <= 0) {
      return 0.0;  // или другой осмысленный результат / код ошибки
  }
  ```

  Это в разы уменьшает шанс словить «подлый» segfault.

### 3.7. Вопросы для самопроверки по разделу 3 (`cffi`)

1. **В чём принципиальная разница между ABI‑ и API‑режимами `cffi`?**

<details>
<summary>Ответ</summary>
В ABI‑режиме мы подключаемся к уже скомпилированной библиотеке и описываем только её интерфейс (как в `.h`), не компилируя свой C‑код; в API‑режиме мы вместе с Python‑частью компилируем небольшой C‑модуль, написанный в `set_source`, и получаем собственную библиотеку.
</details>

2. **Когда удобнее использовать `ffi.new("type[]", values)`, а когда `ffi.from_buffer(py_buffer)`?**

<details>
<summary>Ответ</summary>
`ffi.new("type[]", values)` удобен, когда нужен отдельный C‑массив, не связанный с существующим Python‑объектом, и `cffi` сам управляет временем его жизни; `ffi.from_buffer(py_buffer)` полезен, когда нужно предоставить C‑коду доступ к уже существующему буферу (например, `bytearray` или NumPy‑массиву) и изменять его «на месте».
</details>

3. **Почему нельзя хранить C‑указатель, полученный через `ffi.from_buffer`, дольше, чем живёт исходный Python‑объект?**

<details>
<summary>Ответ</summary>
Потому что этот указатель указывает на память, которой владеет Python‑объект; если объект будет собран сборщиком мусора или изменит свой внутренний буфер, указатель станет висячим и обращение к нему может привести к повреждению памяти или падению процесса.
</details>
