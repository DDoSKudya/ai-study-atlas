## 6. Практикум: что сделать руками

### 6.0. Мини‑проект 1: «обёртка над libm»

Цель:

- научиться через `ctypes`/`cffi` вызывать пару функций из стандартной библиотеки `libm`.

Шаги:

1. **Подготовка директории.**
   - Создайте новую папку, например `ffi_libm_demo/`.
   - Внутри неё создайте файл `demo_ctypes.py`.
2. **Найдите имя `libm` в вашей системе.**
   - На Linux часто это `libm.so.6` или `libm.so`.
   - Можно попробовать по очереди: `"libm.so.6"`, `"libm.so"`, а если не сработает — поискать через `ldconfig -p | grep libm`.
3. **Через `ctypes`:**
   - загрузите библиотеку;
   - опишите `double sin(double)`, `double cos(double)`;
   - напишите Python‑функции `sin_deg`, `cos_deg`, которые принимают градусы,
     переводят их в радианы и вызывают C‑функции;
   - сравните результат с `math.sin(math.radians(...))`.
4. **Через `cffi` (ABI‑режим):**
   - создайте файл `demo_cffi_abi.py`;
   - опишите `double sin(double x); double cos(double x);` в `cdef`;
   - загрузите `libm` через `ffi.dlopen`;
   - реализуйте те же `sin_deg`/`cos_deg` и сравните поведение и код с версией на `ctypes`.

### 6.0.1. Подсказка: пример структуры `demo_ctypes.py`

```python
import ctypes
import math

libm = ctypes.CDLL("libm.so.6")  # при необходимости поменяйте имя

libm.sin.argtypes = [ctypes.c_double]
libm.sin.restype = ctypes.c_double

libm.cos.argtypes = [ctypes.c_double]
libm.cos.restype = ctypes.c_double


def sin_deg(angle_deg: float) -> float:
    radians = math.radians(angle_deg)
    return libm.sin(radians)


def cos_deg(angle_deg: float) -> float:
    radians = math.radians(angle_deg)
    return libm.cos(radians)


if __name__ == "__main__":
    for deg in [0, 30, 45, 60, 90]:
        print(
            deg,
            sin_deg(deg),
            math.sin(math.radians(deg)),
        )
```

Попробуйте:

- изменить имя библиотеки на неправильное и посмотреть на ошибку;
- убрать или исказить `argtypes`/`restype` и посмотреть, как меняется поведение (это хороший способ «почувствовать», что даёт правильное описание типов).

### 6.1. Мини‑проект 2: «быстрый средний» (`average`)

Цель:

- реализовать C‑функцию `average` и вызвать её разными способами.

Варианты:

1. Чистый `ctypes`:
   - C‑файл `average.c` + `gcc -shared -fPIC -o libaverage.so average.c`;
   - Python‑биндинги на `ctypes`.
2. `cffi` в API‑режиме:
   - `avg_build.py` + `ffi.compile()`;
   - вызов через `_avg_cffi.lib.average`.
3. Cython:
   - `fast_avg.pyx` + `setup.py` или `pyproject.toml`;
   - `cpdef`‑функция, принимающая NumPy‑массив.

Сравните:

- удобство написания;
- читаемость кода;
- скорость (хотя бы грубо через `timeit`).

### 6.1.1. Рекомендуемый порядок экспериментов

1. **Сначала сделайте версию на `ctypes`.**
   - Это самый прямой и «шумный» путь, где хорошо видны типы и указатели;
   - дополнительно вы закрепите материал раздела 2.
2. **Потом повторите ту же задачу на `cffi` (API‑режим).**
   - Обратите внимание, как меняется «читаемость» кода;
   - посмотрите, где вам проще рассуждать про типы.
3. **Затем реализуйте вариант на Cython.**
   - Попробуйте сначала переписать Python‑реализацию почти без изменений;
   - затем постепенно добавить типы и посмотреть, как меняется производительность.

По итогам этого мини‑проекта вы получите **своё собственное чувство**, когда вам лично удобнее:

- просто оборачивать C через `ctypes`/`cffi`;
- писать на Cython;
- или даже идти в сторону `pybind11` для C++.

### 6.2. Мини‑проект 3: маленький модуль на `pybind11`

Цель:

- собрать своё первое C++‑расширение.

Идея:

1. Написать `bindings.cpp` с:
   - функцией `int clamp(int value, int low, int high)`,
     которая ограничивает значение интервалом `[low, high]`;
   - экспортом этой функции через `PYBIND11_MODULE`.
2. Собрать модуль через `setuptools` (см. раздел 5).
3. В Python:
   - протестировать граничные случаи;
   - использовать в небольшом примере.

### 6.2.1. Пошаговый план для `clamp` на `pybind11`

1. **Подготовка проекта.**
   - Создайте папку `pybind11_clamp_demo/`.
   - Внутри создайте файлы:
     - `bindings.cpp`;
     - `setup.py`.
2. **Напишите `bindings.cpp`.**

   ```cpp
   // file: bindings.cpp

   #include <algorithm>
   #include <pybind11/pybind11.h>

   namespace py = pybind11;

   int clamp(int value, int low, int high) {
       if (low > high) {
           std::swap(low, high);
       }
       if (value < low) {
           return low;
       }
       if (value > high) {
           return high;
       }
       return value;
   }

   PYBIND11_MODULE(clamp_mod, m) {
       m.doc() = "Пример простого модуля с функцией clamp";

       m.def(
           "clamp",
           &clamp,
           "Ограничить значение интервалом [low, high]",
           py::arg("value"),
           py::arg("low"),
           py::arg("high")
       );
   }
   ```

3. **Напишите `setup.py`.**

   ```python
   from setuptools import setup, Extension

   import pybind11

   ext = Extension(
       "clamp_mod",
       sources=["bindings.cpp"],
       include_dirs=[pybind11.get_include()],
       language="c++",
   )

   setup(
       name="clamp-mod-demo",
       version="0.1.0",
       ext_modules=[ext],
   )
   ```

4. **Соберите модуль.**

   В виртуальном окружении, где установлен `pybind11`:

   ```bash
   pip install pybind11
   python setup.py build_ext --inplace
   ```

   В каталоге появится файл `clamp_mod.*.so` / `.pyd`.

5. **Протестируйте модуль в Python.**

   Создайте файл `test_clamp.py`:

   ```python
   import clamp_mod


   def test():
       cases = [
           (5, 0, 10, 5),
           (-1, 0, 10, 0),
           (100, 0, 10, 10),
           (5, 10, 0, 5),  # проверка перестановки low/high
       ]

       for value, low, high, expected in cases:
           result = clamp_mod.clamp(value, low, high)
           print(value, low, high, "->", result, "(ожидали", expected, ")")


   if __name__ == "__main__":
       test()
   ```

   Обратите внимание:

   - мы специально добавили случай, где `low > high`, чтобы проверить корректность обработки;
   - в реальном проекте вы бы оформили это как тесты (`pytest`), но для начала достаточно простого скрипта.

## 7. Справочник, термины и самопроверка

### 7.1. Терминология

- **FFI** — интерфейс для вызова кода на C/С++ из Python.
- **ABI** — соглашение на уровне бинарного кода: как функции вызываются, как устроены типы.
- **API** — соглашение на уровне исходного кода: какие функции/типы доступны и как ими пользоваться.
- **shared library** — динамическая библиотека (`.so`, `.dll`, `.dylib`), загружаемая во время выполнения.
- **GIL** — глобальная блокировка интерпретатора CPython; в один момент байткод исполняет один поток.
- **освобождение GIL** — временное снятие GIL в C/С++‑коде для параллельной работы других потоков.
- **`ctypes`** — модуль stdlib для FFI с уже скомпилированными C‑библиотеками.
- **`cffi`** — библиотека для более удобного и переносимого FFI (C‑интерфейс в виде `cdef`, ABI/API‑режимы).
- **`pybind11`** — C++11‑библиотека для генерации Python‑биндингов к C++‑коду.
- **Cython** — надстройка над Python, позволяющая скомпилировать типизированный Python‑подобный код в C.

### 7.2. Типичные ошибки и как их избежать

- **Ошибка сигнатуры** (не те типы в `ctypes`/`cffi`):
  - следствие: неверные значения, segfault;
  - решение: всегда сверяйтесь с C‑заголовками (`.h`), не угадывайте типы.
- **Утечки и двойное освобождение памяти**:
  - следствие: рост памяти, нестабильность;
  - решение: используйте высокоуровневые обёртки (`cffi.new`, RAII в C++/`pybind11`);
    избегайте «голых» `malloc`/`free`, если можно.
- **Использование указателей на уже освобождённые объекты**:
  - решение: не храните C‑указатели дольше, чем живут соответствующие Python‑объекты;
    документируйте «кто владеет памятью».
- **Манипуляции Python‑объектами без GIL**:
  - решение: в участках кода, где вы освобождаете GIL,  
    не трогайте списки, словари, строки и т.п.
- **Преждевременная оптимизация**:
  - решение: сначала профилируйте Python‑код,  
    используйте NumPy/Cython, только потом — «чистый» C/С++.

### 7.3. Вопросы для самопроверки (с краткими ответами)

1. **Когда имеет смысл выносить часть логики в C‑расширение, а когда лучше остаться на чистом Python?**  
   _Ответ:_ когда есть узкий, хорошо изолируемый и многократно вызываемый участок,
   который после профилирования действительно является «горячей точкой»,  
   и его логика достаточно проста, чтобы переписать на C/С++ без чрезмерных рисков.

2. **Чем ABI‑режим `cffi` отличается от API‑режима?**  
   _Ответ:_ в ABI‑режиме вы подключаетесь к уже скомпилированной библиотеке без отдельной компиляции C‑кода,
   в API‑режиме `cffi` сам компилирует небольшой C‑модуль на основе вашего `set_source`.

3. **Какую роль играют `argtypes` и `restype` в `ctypes` и что будет, если их не указать?**  
   _Ответ:_ они описывают типы аргументов и результата C‑функции;  
   без них `ctypes` попытается угадать, что часто приводит к мусорным значениям или падениям.

4. **Почему `pybind11` удобнее «чистого» C‑API Python для C++‑кода?**  
   _Ответ:_ потому что он автоматически конвертирует типы Python ↔ C++,
   управляет жизнью объектов и GIL, и позволяет описывать биндинги в современном C++‑стиле
   (классы, шаблоны, STL), без ручного управления refcount и макросами C‑API.

5. **Какую проблему решает освобождение GIL в тяжёлых C/C++‑функциях?**  
   _Ответ:_ позволяет другим Python‑потокам продолжать работу,
   пока долго выполняется вычисление в C/С++, вместо того чтобы блокировать весь интерпретатор.

6. **В чём практический плюс использования Cython по сравнению с ручным написанием C?**  
   _Ответ:_ вы можете сохранять Python‑синтаксис и постепенно добавлять типы и оптимизации,
   не переписывая код полностью на C и не работая напрямую с C‑API Python.

### 7.4. Короткое резюме

- C‑расширения и FFI — это **инструмент для точечного ускорения** и доступа к существующим C/С++‑библиотекам,
  а не способ «переписать всё приложение на C».
- `ctypes` даёт быстрый, но низкоуровневый доступ к функциям C;  
  `cffi` делает это удобнее и переносимее;  
  `pybind11` и Cython помогают интегрировать серьёзный C/C++‑код.
- Главное — держать **границу между Python и C простой, хорошо задокументированной и протестированной**,  
  и всегда измерять производительность до и после оптимизаций.
