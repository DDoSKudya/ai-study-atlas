## 2. `ctypes`: быстрый доступ к C‑библиотекам (§145)

### 2.0. Идея простыми словами (аналогия с переводчиком)

Представьте, что:

- Python — это человек, который говорит только по‑русски;
- C‑библиотека — человек, который говорит только по‑английски;
- `ctypes` — это **переводчик между ними**.

Что делает `ctypes`:

1. **Находит и загружает** «англоязычную книгу» — файл `.so` / `.dll` / `.dylib`.
2. Вы говорите переводчику:
   - «в этой книге есть функция `cos`, она:
     - принимает одно число `double`;
     - возвращает одно число `double`».
3. Дальше вы:
   - даёте переводчику обычное Python‑число;
   - он:
     - превращает его во внутренний C‑тип;
     - вызывает C‑функцию;
     - переводит результат обратно в Python‑число.

Очень важно:

- `ctypes` **не проверяет за вас, что вы всё описали правильно**;
- если вы сказали:
  - «эта функция возвращает `int`»,
  - а на самом деле она возвращает `double`,
  - вы получите мусор или падение процесса.

Поэтому раздел про `ctypes` мы будем проходить **медленно и по шаблону**:

1. Что делает функция на C (словами).
2. Как её объявление выглядит на C.
3. Как описать её в `ctypes`.
4. Как проверить, что всё реально работает.

### 2.0.1. Мини‑шпаргалка по шагам

Каждый раз, когда вы хотите вызвать C‑функцию через `ctypes`, думайте так:

1. **Где взять библиотеку?**
   - системная (уже стоит в ОС) → `libc`, `libm`, и т.п.;
   - своя (вы её скомпилировали) → `./libsomething.so`.
2. **Как выглядит сигнатура функции на C?**
   - есть ли заголовочный файл `.h`?
   - можно ли его открыть и увидеть что‑то вроде  
     `int add(int a, int b);`
3. **Какой тип у каждого аргумента и результата?**
   - `int`, `double`, `char *`, `void *`, `struct ...`?
4. **Какой тип `ctypes` этому соответствует?**
   - `c_int`, `c_double`, `c_char_p`, `POINTER(...)` и т.п.
5. **Как безопасно проверить работу?**
   - написать 2–3 вызова с понятными входными данными;
   - сравнить результат с тем, что даёт чистый Python.

Если вы будете **каждый раз** проходить этот список,  
шанс совершить фатальную ошибку сильно снизится.

### 2.1. Загрузка библиотеки: `ctypes.CDLL`

Разберёмся на очень простом примере: вызов `cos` из стандартной математической библиотеки `libm` на Linux.

```python
import ctypes
import math
from ctypes import c_double

# В Linux математические функции обычно в libm.so
libm = ctypes.CDLL("libm.so.6")  # в некоторых дистрибутивах имя может отличаться

# Берём функцию cos
cos = libm.cos

# Описываем сигнатуру: double cos(double x);
cos.argtypes = [c_double]
cos.restype = c_double

print(cos(0.0))           # 1.0
print(cos(math.pi / 3))   # ~0.5
```

Разбор:

- `ctypes.CDLL("libm.so.6")` — загружает динамическую библиотеку;
- `cos = libm.cos` — берём из неё символ `cos`;
- `argtypes` — список типов аргументов;
- `restype` — тип результата.

Если **не указать** `argtypes`/`restype`, то:

- `ctypes` подумает, что:
  - все аргументы — это нечто вроде `c_int`,
  - результат — тоже `c_int`;
- а функция в реальности может ожидать и возвращать `double`;
- в результате:
  - часть битов числа будет «обрезана» или неправильно интерпретирована;
  - возможно, вы сразу этого не заметите, но получите неверную математику.

Поэтому **привыкайте**:

- как только вы взяли функцию из библиотеки (`cos = libm.cos`) —
- **сразу** задайте ей `argtypes` и `restype`.

### 2.2. Базовые типы: `c_int`, `c_char_p`, `POINTER` (очень медленно)

В C есть свои базовые типы:

- целые числа (`int`, `long`, `short`, `unsigned ...`);
- числа с плавающей точкой (`float`, `double`);
- символы и строки (`char`, `char *`);
- указатели на эти типы (`int *`, `double *`, `void *`).

В `ctypes` у каждого из этих типов есть **примерный аналог**:

- `c_int` — похож на C‑`int`;
- `c_long` — похож на C‑`long`;
- `c_double` — похож на C‑`double`;
- `c_char_p` — «указатель на строку `char *`»;
- `POINTER(T)` — «указатель на тип `T`», например `POINTER(c_int)` ~ `int *`.

Важно понимать:

- **слово «похож» здесь значит:** размер и представление совпадает на вашей платформе;
- именно поэтому иногда важна разница между `int` и `long`:
  - на разных ОС и архитектурах размеры могут отличаться;
  - всегда сверяйтесь с документацией библиотеки.

Пример: вызываем простую C‑функцию

```c
// file: example.c

int add(int a, int b) {
    return a + b;
}
```

Компиляция (Linux, gcc):

```bash
gcc -shared -fPIC -o libexample.so example.c
```

Python‑код:

```python
import ctypes
from ctypes import c_int

lib = ctypes.CDLL("./libexample.so")

add = lib.add
add.argtypes = [c_int, c_int]
add.restype = c_int

print(add(2, 3))  # 5
```

Здесь всё просто: только целые числа, без указателей и без строк.

### 2.2.1. Как «чувствовать» соответствие типов

Очень полезно руками написать себе такую табличку (можно даже на бумаге рядом с ноутбуком):

| На C | В `ctypes` (обычно) | Комментарий |
|------|---------------------|------------|
| `int` | `c_int` | «обычное» целое число |
| `double` | `c_double` | число с плавающей точкой двойной точности |
| `float` | `c_float` | число с плавающей точкой одинарной точности |
| `const char *` (строка только для чтения) | `c_char_p` | указатель на нуль‑терминированную строку |
| `void *` | `c_void_p` | «просто указатель», без типа |
| `int *` | `POINTER(c_int)` | указатель на целое |

Эта табличка не покрывает все случаи, но для **первых шагов** её обычно достаточно.

### 2.3. Указатели и `byref` / `pointer`

Указатель в C — это **адрес ячейки памяти**, где лежит значение.

Простейший пример‑аналогия:

- у вас есть лист бумаги (значение);
- есть стикер с надписью «лист лежит в ящике №3» (указатель);
- можно:
  - передать кому‑то сам лист бумаги;
  - а можно — только стикер «где он лежит».

Чаще всего в C встречается паттерн:

```c
void fill_array(int *out, int size);
```

или

```c
void increment(int *value);
```

В Python через `ctypes` нам нужно:

1. создать объект `c_int` или массив;
2. получить **указатель** на него;
3. передать этот указатель в C‑функцию.

Пример:

```c
// file: counter.c

void increment(int *value) {
    (*value)++;
}
```

Компиляция:

```bash
gcc -shared -fPIC -o libcounter.so counter.c
```

Python‑код:

```python
import ctypes
from ctypes import c_int, byref

lib = ctypes.CDLL("./libcounter.so")

increment = lib.increment
increment.argtypes = [ctypes.POINTER(c_int)]
increment.restype = None

value = c_int(10)
print("Before:", value.value)   # 10

increment(byref(value))

print("After:", value.value)    # 11
```

Обратите внимание:

- `c_int(10)` создаёт объект, который **живёт в памяти по правилам C**;
- `byref(value)` — это способ получить **адрес** этого объекта;
- C‑функция меняет значение по адресу;
- после вызова мы читаем новое значение через `value.value`.

Если переписать это на человеческий язык:

- Python: «Вот переменная `value`, в ней 10».
- C‑функция: «Дай мне листочек, на котором написано, **где лежит** это число».
- Python с помощью `byref` такой листочек выдаёт.

### 2.3.1. `byref` vs `pointer`

- `byref(obj)`:
  - не создаёт новый объект;
  - просто даёт «указатель на уже существующий»;
  - чаще всего **именно это вам и нужно**.
- `pointer(obj)`:
  - создаёт **новый объект‑указатель**, который сам живёт в Python;
  - используется реже, когда вы хотите хранить указатель как отдельную сущность.

Для первых шагов **считайте, что вам почти всегда нужен `byref`**.

### 2.4. Строки и буферы: `c_char_p`, массивы

Строки в C — это `char *` + нулевой байт в конце.  
В `ctypes`:

- `c_char_p` — указатель на нуль‑терминированную строку;
- Python‑строка должна быть **закодирована** в байты (обычно UTF‑8).

Пример: C‑функция считает длину строки (аналог `strlen`):

```c
// file: strlen_like.c

int my_strlen(const char *s) {
    int n = 0;
    while (*s) {
        n++;
        s++;
    }
    return n;
}
```

Компиляция:

```bash
gcc -shared -fPIC -o libstrlen_like.so strlen_like.c
```

Python‑код:

```python
import ctypes
from ctypes import c_char_p, c_int

lib = ctypes.CDLL("./libstrlen_like.so")

my_strlen = lib.my_strlen
my_strlen.argtypes = [c_char_p]
my_strlen.restype = c_int

text = "Привет"
encoded = text.encode("utf-8")  # bytes

length = my_strlen(encoded)
print(length)
```

Важный момент:

- если C‑функция ожидает **модифицируемый буфер** (`char *`),  
  лучше использовать **массив байтов**, а не `c_char_p`.

Пример создания буфера:

```python
from ctypes import create_string_buffer

buf = create_string_buffer(b"hello", 16)
# теперь в buf лежит массив из 16 байтов:
# первые 5 — "hello", дальше нули
```

Такой буфер можно безопасно передавать в функции, которые:

- **меняют содержимое** строки;
- ожидают, что там есть место «с запасом».

### 2.5. GIL и `ctypes`

`ctypes` может **освобождать GIL** на время вызова C‑функции, но:

- это поведение зависит от:
  - платформы;
  - настроек сборки CPython;
  - того, возвращает ли функция `py_object`.
- безопаснее считать:
  - если ваша C‑функция **долго считает** и не трогает Python‑объекты —
    лучше явно вынести её:
    - в отдельный процесс (через `multiprocessing`);
    - или написать полноценное C‑расширение с явным управлением GIL.

Для целей этой части:

- запомните только, что **долгие блокирующие операции в C могут тормозить весь Python‑процесс**,
  если GIL не освобождается;
- если вы пишете действительно долгие вычисления на C и активно используете потоки,
  лучше сразу смотреть в сторону:
  - полноценного C‑расширения;
  - или отдельных процессов.

### 2.6. Ограничения и типичные ошибки

Главные проблемы при работе с `ctypes`:

- **Несовпадение сигнатур:**
  - неверные `argtypes`/`restype`;
  - неправильно выбранные C‑типы (`int` vs `long`, `double` vs `float`).
- **Неправильные указатели:**
  - передали указатель на Python‑объект, который уже был освобождён;
  - использовали буфер меньше нужного размера.
- **Работа со строками:**
  - забыли кодировать/декодировать;
  - путаете `char *` (одна строка) и `char **` (массив строк).

Правило (очень важное):

> Если вы не уверены в типах и размерах —  
> сначала напишите и отладьте маленький C‑пример **без Python**,  
> а уже потом подключайте его через `ctypes`.

То есть:

1. Напишите маленький `.c`‑файл с `main`, который:
   - вызывает нужную функцию;
   - печатает результат;
   - проверяет граничные случаи.
2. Убедитесь, что **сам C‑код работает правильно**.
3. Только после этого:
   - выносите функцию в библиотеку (`.so` / `.dll`);
   - подключайте её через `ctypes`.

Так вы разделяете:

- ошибки в самой логике функции;
- и ошибки в FFI‑обёртке.

### 2.7. Мини‑упражнение (пошагово, с подсказками)

**Задача.** Написать и вызвать C‑функцию `double average(const double *values, int n)`,  
которая считает среднее значение массива.

Шаг 1. C‑файл `average.c`:

```c
// file: average.c

double average(const double *values, int n) {
    if (n <= 0) {
        return 0.0;
    }
    double sum = 0.0;
    for (int i = 0; i < n; i++) {
        sum += values[i];
    }
    return sum / n;
}
```

Шаг 2. Компиляция в библиотеку (Linux, gcc):

```bash
gcc -shared -fPIC -o libaverage.so average.c
```

- `-shared` — собрать разделяемую библиотеку (а не обычный exe);
- `-fPIC` — позиционно‑независимый код (на современных Linux обычно обязателен);
- `-o libaverage.so` — имя выходного файла.

Шаг 3. Вызов из Python через `ctypes`:

```python
import ctypes
from ctypes import c_double, c_int

lib = ctypes.CDLL("./libaverage.so")

average = lib.average
average.argtypes = [ctypes.POINTER(c_double), c_int]
average.restype = c_double

values = [1.0, 2.0, 3.0, 4.0]
n = len(values)

# создаём C-массив double[n]
ArrayType = c_double * n
c_array = ArrayType(*values)

result = average(c_array, n)
print("average =", result)  # должно быть 2.5
```

Шаг 4. Проверьте граничный случай:

- что будет, если передать `n = 0`?
- как поведёт себя ваша функция?

Попробуйте:

```python
empty = (c_double * 0)()
print("average(empty) =", average(empty, 0))
```

И объясните себе:

- почему вы именно так обрабатывали `n <= 0` в C‑коде;
- устраивает ли вас такое поведение в вашем проекте.

### 2.8. Вопросы для самопроверки по разделу 2 (`ctypes`)

1. **Зачем в `ctypes` нужно явно задавать `argtypes` и `restype` для каждой C‑функции?**

<details>
<summary>Ответ</summary>
Чтобы `ctypes` знал, как упаковывать Python‑аргументы в C‑типы и как интерпретировать результат; без этого он пытается угадать, что часто приводит к мусорным значениям или падению процесса (segfault) при несовпадении реальной сигнатуры и ожидаемой.
</details>

2. **Чем `byref(obj)` отличается от `POINTER(T)(obj)` и когда обычно нужен `byref`?**

<details>
<summary>Ответ</summary>
`byref(obj)` не создаёт новый объект, а даёт указатель на уже существующий C‑объект, тогда как `POINTER(T)(obj)` создаёт отдельный объект‑указатель. В типичных сценариях (передать адрес переменной в C‑функцию для модификации) почти всегда достаточно и предпочтителен `byref`.
</details>

3. **В каких случаях вместо `c_char_p` стоит использовать `create_string_buffer`?**

<details>
<summary>Ответ</summary>
Когда C‑функция ожидает модифицируемый буфер (`char *`) и может менять содержимое строки или писать в заранее выделенный буфер; `c_char_p` подходит для неизменяемых строк, а `create_string_buffer` создаёт массив байтов фиксированной длины, который безопасно передавать в такие функции.
</details>
