## 4. `pybind11`: биндинги для C++ (§147)

### 4.0. Для чего нужен `pybind11`

Когда у вас есть **существенный C++‑код**:

- шаблоны;
- классы с методами;
- использование STL (`std::vector`, `std::string`);

— писать биндинги вручную через C‑API Python (или даже `cffi`) становится:

- громоздко;
- хрупко;
- сложно поддерживать.

`pybind11`:

- позволяет писать **нормальный C++11‑код**;
- берёт на себя:
  - конвертацию типов Python ↔ C++;
  - жизнь объектов (refcount);
  - многие детали GIL.

### 4.1. Минимальный пример: функция `add`

Структура проекта:

- `bindings.cpp` — C++‑файл с биндингами;
- `setup.py` и/или `pyproject.toml` — сборка расширения.

`bindings.cpp`:

```cpp
// file: bindings.cpp

#include <pybind11/pybind11.h>

namespace py = pybind11;

int add(int a, int b) {
    return a + b;
}

PYBIND11_MODULE(example, m) {
    m.doc() = "Простой модуль на pybind11";

    m.def("add", &add, "Сложить два числа",
          py::arg("a"), py::arg("b"));
}
```

Разбор:

- `PYBIND11_MODULE(example, m)`:
  - `example` — имя модуля, которое вы будете импортировать в Python;
  - `m` — объект‑модуль, в который вы добавляете функции и классы.
- `m.def("add", &add, ...)`:
  - регистрирует функцию `add` как `example.add`.

Далее нужен `setup.py` (см. раздел 5), но суть биндингов уже видна.

### 4.1.1. Как читать пример `add` по шагам

Можно проговорить вслух:

1. «У меня есть **обычная C++‑функция** `int add(int a, int b)`».
2. «Я хочу, чтобы в Python можно было писать `example.add(2, 3)` и получать 5».
3. «Я создаю модуль через `PYBIND11_MODULE(example, m)` и говорю:  
   `m.def("add", &add, ...)` — свяжи имя `"add"` в Python с этой C++‑функцией».
4. «При сборке создаётся файл `example.*.so`, в котором уже зашит весь этот код».
5. «Когда я делаю `import example` в Python,  
   Python вызывает `PYBIND11_MODULE`, а я получаю готовый модуль с функцией `add`».

Важно:

- `pybind11` сам:
  - конвертирует Python‑`int` в C++‑`int`;
  - вызывает `add`;
  - конвертирует результат обратно в Python‑`int`.
- Ваша задача — **не забыть**:
  - правильно указать имя модуля (`example` в макросе и в `setup.py`);
  - экспортировать нужные функции/классы через `m.def` / `py::class_`.

### 4.2. Пример с классом

```cpp
// file: bindings_point.cpp

#include <pybind11/pybind11.h>

namespace py = pybind11;

class Point {
public:
    Point(double x, double y) : x_(x), y_(y) {}

    double x() const { return x_; }
    double y() const { return y_; }

    void move(double dx, double dy) {
        x_ += dx;
        y_ += dy;
    }

private:
    double x_;
    double y_;
};

PYBIND11_MODULE(point_mod, m) {
    py::class_<Point>(m, "Point")
        .def(py::init<double, double>(), py::arg("x"), py::arg("y"))
        .def("x", &Point::x)
        .def("y", &Point::y)
        .def("move", &Point::move);
}
```

В Python (после сборки):

```python
import point_mod

p = point_mod.Point(1.0, 2.0)
print(p.x(), p.y())  # 1.0 2.0

p.move(0.5, -1.0)
print(p.x(), p.y())  # 1.5 1.0
```

Особенности:

- `py::class_<Point>(m, "Point")`:
  - создаёт Python‑тип `Point`;
  - управляет созданием/удалением C++‑объектов;
- `py::init<double, double>()` — конструктор;
- методы оборачиваются через `.def(...)`.

### 4.2.1. Подводные камни при биндинге классов

- **Жизненный цикл объектов.**  
  `pybind11` по умолчанию хранит C++‑объект внутри Python‑обёртки и уничтожает его, когда Python‑объект умирает.  
  Обычно это то, что нужно, но:
  - не стоит вручную удалять объект (`delete`) изнутри ваших методов;
  - если вы храните «сырые» указатели (`Point *`) где‑то ещё, важно следить, чтобы они не переживали Python‑объект.

- **Копирование vs ссылки.**  
  Если ваши методы возвращают ссылки (`Point&`, `const Point&`),  
  убедитесь, что возвращаемые объекты живут достаточно долго. В простых случаях лучше возвращать по значению (`Point`), чтобы избежать висячих ссылок.

- **Имена в Python.**  
  Названия методов в `.def("move", &Point::move)` — это именно Python‑имена:
  - вы можете назвать их по‑другому, чем в C++;
  - главное — чтобы это было понятно пользователю модуля (например, `move_to`, `length`, `to_dict` и т.п.).

### 4.3. Освобождение GIL: `py::gil_scoped_release`

Если у вас есть:

- тяжёлая C++‑функция, которая **не трогает Python‑объекты**;
- и вы хотите, чтобы другие Python‑потоки работали параллельно;

— вы можете временно освободить GIL.

```cpp
#include <pybind11/pybind11.h>
#include <cmath>

namespace py = pybind11;

double heavy_compute(double x) {
    double result = 0.0;
    for (long i = 0; i < 100000000; ++i) {
        result += std::sin(x + i);
    }
    return result;
}

PYBIND11_MODULE(heavy_mod, m) {
    m.def(
        "heavy_compute",
        [](double x) {
            py::gil_scoped_release release;  // освободить GIL в этом блоке
            return heavy_compute(x);
        },
        "Тяжёлое вычисление без удержания GIL",
        py::arg("x")
    );
}
```

Здесь:

- лямбда‑обёртка сначала освобождает GIL, затем выполняет `heavy_compute`;
- по выходе из лямбда‑функции GIL автоматически захватывается обратно.

Важно:

- пока GIL освобождён, **нельзя трогать Python‑объекты** (списки, словари, строки и т.п.).

### 4.4. NumPy‑массивы: `py::array_t`

`pybind11` умеет **zero‑copy** работать с NumPy‑массивами.

Простейший пример (без полной сборки, только идея):

```cpp
#include <pybind11/pybind11.h>
#include <pybind11/numpy.h>

namespace py = pybind11;

double sum_array(py::array_t<double> arr) {
    auto buf = arr.request();  // информация о буфере
    double *ptr = static_cast<double *>(buf.ptr);
    size_t n = buf.size;

    double sum = 0.0;
    for (size_t i = 0; i < n; ++i) {
        sum += ptr[i];
    }
    return sum;
}

PYBIND11_MODULE(num_mod, m) {
    m.def("sum_array", &sum_array);
}
```

В Python:

```python
import numpy as np
import num_mod

arr = np.array([1.0, 2.0, 3.0])
print(num_mod.sum_array(arr))  # 6.0
```

Главная мысль:

- `py::array_t<double>` даёт вам прямой доступ к памяти NumPy‑массива,
  без лишних копий.

### 4.4.1. Подводные камни при работе с `py::array_t`

- **Предположения о типе и форме.**  
  Перед тем как брать `buf = arr.request()`, полезно:
  - проверить `arr.ndim` и `arr.shape`, чтобы убедиться, что массив той размерности, которую вы ожидали;
  - при необходимости явно потребовать `C`‑контигуозный массив (см. документацию `pybind11`).

- **Модификация данных.**  
  Если ваша функция меняет массив «на месте», это будет видно в Python:
  - это удобно, но может удивлять пользователя модуля;
  - документируйте такие функции как «in‑place» (изменяющие аргументы).

- **Время жизни массива.**  
  Не храните «сырые» указатели (`double *ptr` из `buf.ptr`) где‑то глобально:
  - они действительны только пока живёт соответствующий Python‑/NumPy‑объект;
  - если нужно сохранить данные — скопируйте их в свой буфер.

### 4.6. Вопросы для самопроверки по разделу 4 (`pybind11`)

1. **Что делает макрос `PYBIND11_MODULE(name, m)` и чем важно имя `name`?**

<details>
<summary>Ответ</summary>
Макрос разворачивается в функцию инициализации Python‑модуля: он создаёт модуль `name` и передаёт объект `m`, через который вы регистрируете функции и классы. Имя `name` должно совпадать с именем расширения, которое вы будете импортировать в Python (`import name`), и с именем, указанным при сборке (`Extension("name", ...)`).
</details>

2. **Какую роль играет `py::class_<T>(m, "T")` при биндинге классов?**

<details>
<summary>Ответ</summary>
Он создаёт новый Python‑тип с именем `"T"`, связанный с C++‑классом `T`, и позволяет через цепочку вызовов (`.def(py::init<...>(), ...)`, `.def("method", &T::method)`) описать конструкторы и методы. `pybind11` затем управляет жизненным циклом C++‑объектов, которые лежат «внутри» Python‑обёрток.
</details>

3. **Почему при работе с тяжёлыми C++‑вычислениями полезно использовать `py::gil_scoped_release`, и что при этом делать нельзя?**

<details>
<summary>Ответ</summary>
`py::gil_scoped_release` временно освобождает GIL, позволяя другим Python‑потокам работать параллельно, пока выполняется тяжёлая C++‑функция. Внутри участка кода с освобождённым GIL нельзя трогать Python‑объекты (создавать/удалять/изменять списки, словари, строки и т.п.), иначе можно повредить внутренние структуры интерпретатора.
</details>
