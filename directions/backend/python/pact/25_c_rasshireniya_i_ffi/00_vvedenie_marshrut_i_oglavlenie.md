# Часть XXV. C-расширения и FFI

> **Цель:** показать, **как «скрестить» Python и C/C++**:
> - вызывать функции из готовых C‑библиотек (быстрые алгоритмы, системные вызовы);
> - писать свои C/C++‑расширения под Python;
> - понимать, **где это действительно нужно**, а где «преждевременная оптимизация».
>
> Мы не делаем из вас системного программиста на C,  
> но даём **рабочие шаблоны и ментальные модели**, чтобы:
> - не бояться FFI (Foreign Function Interface);
> - не «убить» процесс из‑за ошибок с памятью;
> - осознанно выбирать между `ctypes`, `cffi`, `pybind11`, Cython.

После изучения этой части вы:

- поймёте, **зачем вообще нужны C‑расширения**, если Python «и так всё может»;
- научитесь через **`ctypes`** вызывать функции из `libc`, `libm` и своих `.so/.dll`;
- разберётесь с **двумя режимами `cffi` (ABI/API)** и базовой работой с памятью через `ffi.new()`/`ffi.from_buffer()`;
- увидите, как на практике выглядит небольшой модуль на **`pybind11`** для C++;
- сможете собрать простое C‑расширение через **`setuptools.Extension`** и `pyproject.toml`;
- будете понимать, **когда проще взять Cython**, а когда — «чистый» C/С++ с биндингами.

- **§148** — сборка C‑расширений: `setuptools.Extension`, `python setup.py build_ext --inplace`,
  `pyproject.toml`, Cython, `numpy.distutils`.

Важно:

- Здесь мы **не читаем полный курс по C/C++**.
- Мы даём **минимум C‑кода**, необходимый, чтобы:
  - понять, как связать Python и C;
  - не сломать память;
  - собрать и импортировать модуль.
- Любые фрагменты C/С++ можно воспринимать как «чёрный ящик», если ваш фокус — Python.  
  Главное — понимать, **что вы делаете на границе языков**.

## 0.05. Мини‑словарь (простыми словами)

- **FFI (Foreign Function Interface)** — механизм, позволяющий коду на одном языке (Python)
  вызывать функции и использовать типы другого языка (C/C++).
- **shared library / динамическая библиотека** — файл `.so` (Linux), `.dll` (Windows), `.dylib` (macOS),
  который можно загружать во время выполнения программы.
- **ABI (Application Binary Interface)** — «договор» на уровне бинарного кода:
  как передаются аргументы, как устроен стек, какие типы имеют какой размер.
- **API (Application Programming Interface)** — «договор» на уровне исходного кода:
  какие функции есть, какие у них сигнатуры, как ими пользоваться.
- **GIL (Global Interpreter Lock)** — глобальная блокировка интерпретатора CPython:
  в один момент байткод Python исполняет только один поток.
- **освобождение GIL** — временное снятие GIL при выполнении C‑функции, чтобы другие Python‑потоки могли работать.
- **refcount (счётчик ссылок)** — основной способ управления памятью в CPython:
  объект живёт, пока на него есть ссылки; когда счётчик становится 0 — объект освобождается.
- **segmentation fault (segfault)** — аварийное завершение процесса из‑за некорректного обращения к памяти.
  Частый гость при ошибках в C/С++‑расширениях.
- **binding / биндинги** — обёртка, которая делает функции C/С++ доступными из Python.

## 0.2. Как читать эту часть

Рекомендация:

1. Прочитайте разделы **1–2** целиком, даже если вы пока не любите C:
   - они задают **ментальную модель FFI**;
   - показывают, где граница ответственности Python и C.
2. Выберите **один инструмент для практики**:
   - хотите «минимум порога входа» → начните с `ctypes`;
   - хотите «чуть более современный и аккуратный» FFI → смотрите `cffi`;
   - хотите работать с C++ и шаблонами → `pybind11`.
3. Обязательно сделайте **микро‑проект**:
   - оберните одну‑две функции из `libm` (`sqrt`, `sin`, `cos`);
   - или напишите свою C‑функцию и вызовите её из Python.
4. Только после того, как вы **руками собрали хотя бы один модуль**,  
   читайте раздел про Cython и более сложную сборку.
