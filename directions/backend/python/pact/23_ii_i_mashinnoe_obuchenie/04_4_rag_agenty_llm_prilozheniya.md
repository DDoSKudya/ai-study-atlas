 ## 4. RAG, агенты, LLM‑приложения (§137)

 ### 4.0. Идея RAG

 Проблема «чистой» LLM:

 - модель обучена на данных до некоторой даты;
 - она не знает **ваших внутренних документов**;
 - она может **галлюцинировать** (уверенно придумывать).

 Идея **RAG**:

 1. Храним документы (PDF, wiki, код, базы) в виде **эмбеддингов** во **векторном хранилище**.
 2. При запросе пользователя:
    - превращаем вопрос в эмбеддинг;
    - ищем **похожие документы** по косинусному расстоянию;
    - берём топ‑N фрагментов.
 3. Собираем промпт: «Вот документы, вот вопрос, **ответь, опираясь на эти документы**».
 4. Отправляем промпт в LLM → получаем ответ, в котором модель использует «свежие» данные.

 Таким образом:

 - LLM — **генератор**;
 - векторная БД — **память/поисковик**.

#### 4.0a. Аналогия с «очень умным, но забывчивым экспертом»

Представьте эксперта:

- он прочитал тысячи книг и статей → это его **предобучение**;
- он умеет красиво и связно объяснять;
- но:
  - не знает свежих внутренних документов вашей компании;
  - иногда **уверенно придумывает** детали, если не уверен.

Если просто задавать ему вопросы «из головы», он:

- иногда отвечает блестяще;
- иногда ошибается или придумывает.

RAG добавляет эксперту:

- **личную библиотеку с вашими документами**;
- навык: «прежде чем отвечать, загляни в библиотеку, найди 3–5 самых подходящих документов и **опирайся на них**».

Тогда схема становится:

1. Вопрос → поиск по библиотеке (векторная БД);
2. Находим фрагменты, куда, скорее всего, смотреть;
3. Кладём их вместе с вопросом эксперту;
4. Эксперт читает **не только свой опыт, но и свежие документы** и даёт ответ.

Главная цель RAG — **снизить галлюцинации и добавить свежие/доменные знания**.

 ### 4.1. Векторные БД и эмбеддинги

 Инструменты:

 - **Chroma** — лёгкая встроенная векторная БД для Python;
 - **FAISS** — библиотека от Facebook для быстрого поиска по большим векторным коллекциям;
 - **Milvus, Pinecone** — более тяжёлые/облачные решения;
 - **sentence-transformers** — модели для генерации эмбеддингов предложений/документов.

 Мини‑пример (очень условный, упрощённый):

 ```python
 from sentence_transformers import SentenceTransformer
 import numpy as np

 model = SentenceTransformer("all-MiniLM-L6-v2")

 docs = [
     "Python is a programming language.",
     "Machine learning is about learning from data.",
     "The capital of France is Paris.",
 ]

 doc_embeddings = model.encode(docs)  # shape (3, dim)

 query = "What is ML?"
 query_emb = model.encode([query])[0]  # shape (dim,)

 # Косинусная похожесть
 def cosine_sim(a, b):
     return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

 sims = [cosine_sim(query_emb, d) for d in doc_embeddings]
 best_idx = int(np.argmax(sims))
 print("Best doc:", docs[best_idx])
 ```

 Далее вместо «списка векторов в памяти» используют специализированные хранилища (Chroma, FAISS и др.).

#### 4.1a. Как представить себе эмбеддинги и векторный поиск

Подумайте об **эмбеддинге** как о координатах точки на многомерной карте:

- каждое предложение/документ превращается в вектор, например, размерности 384 или 768;
- если два предложения **про одно и то же**, их точки на карте оказываются рядом;
- если они про разное — далеко.

Тогда:

- **векторная БД** — это большой «склад» таких точек;
- запрос пользователя тоже превращается в точку;
- задача поиска — найти те точки‑документы, которые **геометрически ближе всего** к точке‑запросу (по косинусному расстоянию или евклидову).

Почему нужны специализированные хранилища (Chroma, FAISS и др.):

- когда у вас тысячи документов — можно хранить векторы в памяти и просто перебирать;
- когда документов миллионы/десятки миллионов:
  - нужен более умный способ индексировать векторы;
  - нужны структуры данных, которые ускоряют поиск ближайших соседей (ANN — Approximate Nearest Neighbors);
  - нужна возможность хранить векторы на диске, в кластере и т.д.

**Мини‑упражнение (умственный эксперимент):**

1. Представьте, что вы разложили карточки с названиями статей по столу так, чтобы **похожие лежали рядом**:
   - все про Python рядом друг с другом;
   - статьи про футбол — в другом углу стола;
   - статьи про кулинарию — ещё где‑то.
2. Теперь вы берёте новую карточку «Как обучить модель на данных о клиентах»:
   - куда вы её положите на стол?
   - какие ближайшие карточки окажутся рядом?
3. Именно это и делает модель эмбеддингов + векторный поиск, только в многомерном пространстве и автоматически.

 ### 4.2. Агенты и цепочки

 **Chain** (цепочка) — последовательность шагов, которая:

 - принимает ввод пользователя;
 - может:
   - вызвать поиск (RAG);
   - вызвать внешний API;
   - выполнить код;
 - в итоге формирует итоговый ответ.

 **Agent (агент)** — более гибкая сущность, которая:

 - решает, **какой инструмент** использовать на каждом шаге;
 - может планировать несколько шагов вперёд;
 - иногда строится как «LLM, который сам себе пишет план».

 Инструменты:

 - **LangChain** — популярный фреймворк для chains/agents;
 - **CrewAI, AutoGPT, BabyAGI** — примеры мультиагентных или автономных систем.

 На уровне «первого знакомства» достаточно:

 - понимать, что агент = **LLM + набор инструментов + стратегия выбора, что делать дальше**;
 - осознавать, что это **всё ещё ML‑система с ограничениями**, а не магический «ИИ‑человек».

#### 4.2a. Пример, как мыслить цепочкой и агентом

Представьте задачу: «Дай мне краткое резюме последних 5 сообщений из этого Slack‑канала и создай тикет в трекере, если там про баг».

**Цепочка (chain) может выглядеть так:**

1. Получить последние 5 сообщений через API Slack.
2. Передать их в LLM с промптом: «Сделай краткое резюме».
3. Проверить, упоминается ли в резюме баг/ошибка.
4. Если да — сформировать текст тикета и вызвать API трекера задач.

Все шаги и порядок **заранее заданы вами**. LLM лишь помогает на текстовых этапах.

**Агент может вести себя иначе:**

1. Получает задачу в более общем виде: «Следи за каналом и помогай управлять задачами».
2. На каждом новом сообщении:
   - решает, нужно ли вообще что‑то делать;
   - если да — может:
     - сначала уточнить детали у пользователя;
     - потом сам решить, вызывать ли поиск по документации;
     - затем создать тикет или предложить патч.

То есть агент:

- принимает **решения о том, какие инструменты вызывать и в каком порядке**;
- использует LLM как мозг для этих решений, а LangChain/CrewAI и т.п. — как «скелет» для организации вызовов.

На вашем уровне сейчас важно:

- видеть, что «агент» — это **архитектурный паттерн**, а не какая‑то волшебная сущность;
- помнить, что агент по‑прежнему ограничен данными, промптами и качеством инструментов, которые вы ему дали.

#### 4.2b. CrewAI, AutoGPT, BabyAGI — что это за звери

Эти названия часто всплывают вместе, поэтому важно понимать их роли:

- **AutoGPT / BabyAGI** — ранние эксперименты с «автономными агентами»:
  - вы задаёте общую цель («исследуй тему и сделай отчёт»);
  - система с помощью LLM сама придумывает под‑задачи, ходит в интернет, сохраняет заметки и т.д.;
  - во многом это «исследовательские демо», но они популяризировали идею агентности.
- **CrewAI** — более «инженерный» фреймворк для мультиагентных систем:
  - можно задавать роли (developer, researcher, reviewer…);
  - описывать, какие инструменты есть у каждого;
  - строить взаимодействие нескольких агентов друг с другом.

На уровне вашего плана важно:

- видеть, что это **надстройки** над базовым понятием агента:
  - несколько агентов → несколько LLM‑экземпляров с разными ролями/инструментами;
  - плюс логика координации (кто с кем и когда разговаривает, кто что проверяет).

Практически:

- для старта достаточно понять идею из `4.2a` и уметь описать **одного** агента;
- когда вы почувствуете себя уверенно, можно смотреть на CrewAI и аналоги,  
  как на способ «склеить» несколько таких агентов в команду.
