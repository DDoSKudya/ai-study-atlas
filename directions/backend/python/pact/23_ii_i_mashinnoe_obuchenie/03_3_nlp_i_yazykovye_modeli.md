 ## 3. NLP и языковые модели (§136)

 ### 3.0. От «классического» NLP к трансформерам

 Исторически NLP выглядело так:

 1. Текст → токены (слова/символы).
 2. Токены → признаки (bag‑of‑words, TF‑IDF, n‑gram).
 3. Классические модели (логистическая регрессия, SVM, Naive Bayes и т.п.).

 Сейчас доминирует подход:

 1. Текст → токены (под‑слова, Byte‑Pair Encoding и т.п.).
 2. Токены → **эмбеддинги** (векторы фиксированной размерности).
 3. Модель‑трансформер (BERT, GPT…), обученная предсказывать маски/следующее слово.

 Идея трансформера:

 - использовать механизм **attention** (внимание), который позволяет модели «смотреть» на разные части входа,  
   оценивая, какие слова важны друг для друга.

#### 3.0a. Интуиция: как компьютер «смотрит» на текст

Человек читает предложение и **естественно удерживает контекст**:

- видит слово «банк» и по соседним словам понимает, это «берег реки» или «учреждение с деньгами»;
- может «перематывать» предложение туда‑сюда взглядом, связывая слова между собой.

Раньше модели работали гораздо примитивнее:

- считали, сколько раз каждое слово встречается в документе (bag‑of‑words);
- почти не учитывали порядок слов и контекст;
- от этого страдала точность на сложных задачах.

Трансформер с attention делает примерно то, что делает ваш мозг:

- для каждого слова он смотрит, **на какие другие слова ему стоит обратить внимание**;
- для слова «банк» он может дать большой вес словам «кредит», «депозит» или, наоборот, «река», «берег»;
- за счёт этого получаются **контекстные эмбеддинги**: одно и то же слово в разных фразах кодируется по‑разному.

Держите в голове упрощённую картинку:

> Текст → токены → векторы (эмбеддинги) → несколько слоёв внимания → выход (классификация, генерация и т.д.).

Ниже, в примерах с Hugging Face, мы фактически пользуемся уже **готовыми трансформерами**,  
не залезая в их внутренности — как будто берём библиотеку, где всё за нас уже реализовано.

 ### 3.1. Hugging Face Transformers: готовые модели

 **Hugging Face Transformers** — де‑факто стандарт для работы с предобученными языковыми моделями.

 Простой пример: **классификация текста** с использованием `pipeline`:

 ```python
 from transformers import pipeline

 classifier = pipeline("sentiment-analysis")  # скачает дефолтную модель

 texts = [
     "I love this product, it's amazing!",
     "This is the worst experience I've ever had.",
 ]

 for text in texts:
     result = classifier(text)[0]
     print(text, "->", result["label"], "(", result["score"], ")")
 ```

 Под капотом:

 - текст токенизируется в ID токенов (`tokenizer`);
 - модель считает эмбеддинги и логиты;
 - `pipeline` декодирует результат в удобный вид.

 Пример более «ручной» работы:

 ```python
 from transformers import AutoTokenizer, AutoModelForSequenceClassification
 import torch

 model_name = "distilbert-base-uncased-finetuned-sst-2-english"

 tokenizer = AutoTokenizer.from_pretrained(model_name)
 model = AutoModelForSequenceClassification.from_pretrained(model_name)

 text = "I really enjoyed this movie!"

 inputs = tokenizer(text, return_tensors="pt")
 with torch.no_grad():
     outputs = model(**inputs)

 logits = outputs.logits
 probs = torch.softmax(logits, dim=-1)
 print(probs)
 ```

#### 3.1a. Что делает `pipeline` «по шагам»

Когда вы пишете:

```python
classifier = pipeline("sentiment-analysis")
result = classifier("I love this product!")
```

в реальности происходит примерно следующее:

1. **Загрузка модели и токенизатора.**
   - Hugging Face по имени задачи (`"sentiment-analysis"`) подбирает **подходящую предобученную модель**;
   - скачивает её веса и токенизатор с model hub (если они ещё не закешированы).
2. **Токенизация.**
   - строка `"I love this product!"` разбивается на токены;
   - каждый токен превращается в целое число (ID) по словарю токенизатора;
   - формируется батч входных данных (тензоры `input_ids`, `attention_mask` и т.п.).
3. **Прогон через модель.**
   - модель считает для каждого примера в батче набор логитов (сырых оценок) по классам;
   - внутри она уже использует трансформерные слои, attention и т.д.
4. **Пост‑обработка.**
   - `pipeline` применяет `softmax`, чтобы получить вероятности;
   - выбирает класс с максимальной вероятностью;
   - возвращает удобную структуру: `{ "label": "POSITIVE", "score": 0.98 }`.

То есть `pipeline` — это **обёртка, которая прячет рутину**:

- вам не нужно руками думать про токенизацию, паддинг, тензоры и т.д.;
- вы концентрируетесь на задаче: «у меня есть текст → хочу метку/ответ».

**Мини‑упражнение:**

1. Попробуйте запустить `pipeline("sentiment-analysis")` на 3–5 фразах с разной окраской:
   - явно положительной;
   - явно отрицательной;
   - нейтральной/двусмысленной.
2. Посмотрите, уверена ли модель (`score`) и всегда ли вы согласны с её выводом.
3. Сформулируйте вслух: в каких случаях **модель может ошибаться концептуально** (например, на иронии).

#### 3.1b. Ручной путь: tokenizer + model

Во втором примере мы делаем почти то же, что и `pipeline`, но вручную:

1. `AutoTokenizer.from_pretrained(model_name)` — загружаем подходящий токенизатор;
2. `AutoModelForSequenceClassification.from_pretrained(model_name)` — загружаем модель;
3. `inputs = tokenizer(text, return_tensors="pt")`:
   - токенизация + формирование тензоров для PyTorch;
4. `outputs = model(**inputs)`:
   - прямой проход по сети, получаем `logits`;
5. `softmax` по логитам → вероятности по классам.

Зачем вам уметь делать это вручную:

- когда вам нужно **тонко контролировать**:
  - работу с батчами;
  - несколько текстов сразу;
  - использование GPU;
  - сохранение промежуточных эмбеддингов (а не только предсказаний).

На первых порах можно смело держаться за `pipeline`,  
а ручной путь включать, когда почувствуете себя увереннее.

 ### 3.2. Tokenizers, spaCy, NLTK

 - **tokenizers** (библиотека Hugging Face) — быстрые и эффективные токенизаторы:
   - поддерживают BPE, WordPiece и др.;
   - могут обучать токенизатор на своём корпусе.
 - **spaCy** — современный NLP‑фреймворк:
   - токенизация, POS‑теги, лемматизация, NER;
   - удобно использовать для **практических пайплайнов** (обработка текстов, извлечение сущностей).
 - **NLTK** — классическая академическая библиотека:
   - корпусы, токенизация, стемминг, POS‑теги, парсеры;
   - часто встречается в учебных материалах, но в проде реже, чем spaCy/Transformers.

#### 3.2a. Где что использовать на практике

Очень грубая ориентировка:

- если вам нужна **современная модель для смысла текста** → чаще всего Transformers (Hugging Face);
- если нужна **структурная разметка текста** (предложения, токены, части речи, именованные сущности):
  - spaCy → обычно удобнее и быстрее настроить для продакшена;
  - NLTK → полезен как «ящик с академическими инструментами», но реже в боевых системах;
- если вам нужно **глубоко контролировать токенизацию** под свои модели:
  - отдельная библиотека `tokenizers` или токенизаторы из `transformers`.

Можно думать так:

- Transformers → «мозг», который понимает смысл;
- spaCy/NLTK → «языковой разметчик», который аккуратно режет и подписывает текст;
- tokenizers → «металлическая мясорубка», превращающая строки в токены для моделей.

 ### 3.3. LLM, LangChain, LlamaIndex

 **LLM** (GPT‑подобные модели) сегодня часто используются как:

 - чат‑боты;
 - генераторы кода, текстов;
 - «умные» ассистенты поверх внутренних документов.

 **LangChain** и **LlamaIndex** помогают:

 - строить **цепочки** (chain) действий вокруг LLM:
   - взять вопрос пользователя;
   - найти документы;
   - сформировать промпт;
   - вызвать LLM;
   - пост‑обработать ответ;
 - подключать **инструменты** (tools), векторные БД, внешние API.

 Низкоуровневый пример с OpenAI API (концептуально, без конкретных ключей):

 ```python
 import os
 from openai import OpenAI

 client = OpenAI(api_key=os.environ["OPENAI_API_KEY"])

 response = client.chat.completions.create(
     model="gpt-4.1-mini",
     messages=[
         {"role": "system", "content": "You are a helpful assistant."},
         {"role": "user", "content": "Explain logistic regression in simple terms."},
     ],
 )

 print(response.choices[0].message.content)
 ```

 Важно:

 - LLM не даёт «волшебный интеллект», но даёт **мощный интерфейс к тексту и коду**;
 - вокруг LLM почти всегда есть **архитектура**: базы, векторный поиск, кеши, инструменты.

#### 3.3a. Как мыслить про LLM и цепочки

Полезно разделять:

- **сама LLM‑модель**:
  - умеет по истории сообщений выдавать следующий текст;
  - «знает» много фактов из обучения, но может ошибаться и придумывать;
- **окружение вокруг неё**:
  - векторные базы и RAG;
  - внешние инструменты (поиск, базы данных, ваши API);
  - логика «что вызывать и когда».

LangChain / LlamaIndex помогают вам писать не только:

- «User → Model → Ответ»,

а более сложные сценарии:

- «User → Понять задачу → Найти релевантные документы → Собрать промпт → Вызвать LLM → При необходимости сходить в ещё один сервис → Сформировать итоговый ответ».

Можно думать так:

- LLM — это **умный собеседник, который умеет читать и писать текст**;
- LangChain / LlamaIndex — это **режиссёр**, который говорит:
  - «сначала прочитай эти документы»;
  - «теперь спроси базу данных»;
  - «теперь сформулируй аккуратный ответ».

**Мини‑упражнение (мысленное):**

1. Представьте внутреннего ассистента в компании, который отвечает на вопросы по документации.
2. Разложите его работу на шаги:
   - где он хранит документы;
   - как ищет нужные;
   - как понимает сам вопрос сотрудника;
   - как формулирует ответ.
3. Попробуйте сопоставить эти шаги с блоками:
   - эмбеддинги → векторная БД → RAG → LLM → пост‑обработка.

#### 3.3b. Разные стеки LLM: OpenAI, Anthropic, Ollama, vLLM/TGI

Чтобы не путаться, разделим стек на три большие категории:

- **облачные API** (OpenAI, Anthropic и др.);
- **локальные рантаймы** (Ollama и подобные);
- **серверы быстрого инференса** (vLLM, TGI).

**Облачный API Anthropic (Claude) концептуально похож на OpenAI.**

Пример (упрощённый, без конкретных ключей и версий SDK):

```python
from anthropic import Anthropic
import os

client = Anthropic(api_key=os.environ["ANTHROPIC_API_KEY"])

response = client.messages.create(
    model="claude-3-opus-20240229",
    max_tokens=300,
    messages=[
        {"role": "user", "content": "Объясни, что такое градиентный спуск простым языком."}
    ],
)

print(response.content[0].text)
```

Логика:

- вы передаёте историю сообщений;
- указываете модель и ограничения (например, `max_tokens`);
- получаете сгенерированный текст в ответ.

**Ollama: локальные LLM как сервис.**

Идея Ollama:

- вы заранее скачиваете модель (`llama3`, `mistral`, и т.п.);
- запускаете локальный сервер Ollama;
- общаетесь с ним по HTTP или через их CLI/библиотеку.

Пример запроса через HTTP (очень схематично):

```python
import requests
import json

resp = requests.post(
    "http://localhost:11434/api/chat",
    json={
        "model": "llama3",
        "messages": [
            {"role": "user", "content": "Привет! Объясни разницу между списком и кортежем в Python."}
        ],
    },
)

for chunk in resp.iter_lines():
    if not chunk:
        continue
    data = json.loads(chunk.decode("utf-8"))
    if "message" in data:
        print(data["message"]["content"], end="", flush=True)
```

Смысл:

- вы получаете **похожий на облачный** интерфейс, но модель работает локально;
- удобно для:
  - приватных данных;
  - экспериментов без внешнего интернета.

**vLLM, TGI: быстрый сервер инференса.**

- **vLLM** и **Text Generation Inference (TGI)** — это специализированные серверы, которые:
  - эффективно используют GPU;
  - делают батчинг нескольких запросов;
  - могут держать в памяти большие модели и обслуживать много запросов параллельно.

Типичная схема:

- вы поднимаете vLLM/TGI как отдельный сервис (Docker, Kubernetes, bare metal…);
- он предоставляет HTTP/gRPC API;
- ваши приложения (или LangChain / LlamaIndex) ходят к нему, как к любому другому LLM‑ендпоинту.

На практике:

- OpenAI/Anthropic — удобны для быстрого старта и задач без жёстких ограничений по данным;
- Ollama/локальные модели — когда важна приватность и офлайн‑режим;
- vLLM/TGI — когда вы строите **свой LLM‑сервис**, который должен выдерживать серьёзную нагрузку.

#### 3.3c. Мини‑пример LangChain‑цепочки и агента

Чтобы связать всё вместе, посмотрим на очень упрощённые кусочки кода с LangChain.

**Простая цепочка: вопрос → поиск → ответ.**

```python
from langchain_openai import ChatOpenAI
from langchain.chains import LLMChain
from langchain.prompts import ChatPromptTemplate

llm = ChatOpenAI(model="gpt-4.1-mini")

prompt = ChatPromptTemplate.from_template(
    "Ты опытный разработчик на Python. Объясни простым языком: {question}"
)

chain = LLMChain(llm=llm, prompt=prompt)

result = chain.run({"question": "что такое кросс-валидация в ML?"})
print(result)
```

Идея:

- `ChatPromptTemplate` описывает шаблон сообщения;
- `LLMChain` знает, как подставить переменные и вызвать LLM;
- вы вызываете `.run`, передаёте параметры и получаете готовый текст.

**Простейший агент с инструментом (поиск по документации).**

```python
from langchain_openai import ChatOpenAI
from langchain.agents import initialize_agent, Tool

def search_docs(query: str) -> str:
    # Здесь может быть обращение к векторной БД или обычному поиску
    # Пока сделаем заглушку:
    return f"Результаты поиска по '{query}': ... (заглушка)"

tools = [
    Tool(
        name="docs_search",
        func=search_docs,
        description="Поиск по внутренней документации по строке запроса",
    )
]

llm = ChatOpenAI(model="gpt-4.1-mini")

agent = initialize_agent(
    tools=tools,
    llm=llm,
    agent="chat-zero-shot-react-description",
    verbose=True,
)

response = agent.run("Найди в документации, как настроить логирование в нашем сервисе.")
print(response)
```

Ключевые идеи:

- вы описываете **инструмент** как обычную Python‑функцию + описание;
- агент:
  - читает задачу пользователя;
  - решает, вызывать ли инструмент (и с каким текстом запроса);
  - комбинирует результаты инструмента со своим ответом.

Дальше вокруг этого можно строить гораздо более сложные сценарии (несколько инструментов, несколько агентов и т.п.), но базовый «скелет» именно такой.
