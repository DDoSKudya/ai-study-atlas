## 8. Практические кейсы: как алгоритмы «проживают» в реальном Python‑коде

В этом разделе — **“мост” от теории к практике**. Для нескольких типичных областей разработки посмотрим:

- какие **структуры данных и алгоритмы** там реально используются;
- какие есть **анти‑паттерны**;
- как «переписать» на более эффективный вариант.

### 8.1. Backend‑сервис: логирование, кеши и очереди задач

Типичная картина:

- у вас есть **веб‑приложение** (FastAPI, Django, Flask);
- оно:
  - пишет логи;
  - кеширует результаты тяжёлых запросов;
  - складывает задачи в очередь (email, SMS, отчёты).

#### 8.1.1. Логи: список vs очередь

Наивная реализация «буфера логов»:

```python
logs: list[str] = []

def add_log(message: str) -> None:
    logs.append(message)

def flush_logs() -> list[str]:
    """Забирает все логи и очищает буфер."""
    global logs
    result = logs
    logs = []  # создаём новый список
    return result
```

Почему это ок:

- `append` — амортизированная O(1);
- когда `flush_logs` вызывается редко, стоимость «создать новый список» мала.

Где может быть плохо:

- если `flush_logs` вызывается **крайне часто**, возможны:
  - лишние аллокации списков;
  - GC‑паузы (в очень больших приложениях).

Вариант с `deque`:

```python
from collections import deque

logs: deque[str] = deque()

def add_log(message: str) -> None:
    logs.append(message)

def flush_logs() -> list[str]:
    result = list(logs)   # один проход → O(n)
    logs.clear()          # O(n), но без смены объекта
    return result
```

Практический вывод:

- **важно не столько выбрать “идеальную” структуру, сколько понимать её сложность**;
- для логов обычно «бутылочное горлышко» — не структура, а запись на диск/в сеть.

#### 8.1.2. Кеш: словарь + политика вытеснения

Простой кеш по ключу:

```python
cache: dict[str, str] = {}

def get_from_cache(key: str) -> str | None:
    return cache.get(key)

def set_to_cache(key: str, value: str) -> None:
    cache[key] = value
```

Проблема:

- кеш может **расти бесконечно**, занимая всю память.

Решение:

- ограничить размер и использовать политику **LRU (Least Recently Used)**;
- в stdlib есть готовый инструмент: `functools.lru_cache`.

```python
from functools import lru_cache

@lru_cache(maxsize=1024)
def get_user(user_id: int) -> dict:
    # тяжёлый запрос в БД или внешний сервис
    ...
```

Под капотом:

- `lru_cache` использует комбинацию:
  - **словари** (хеш‑таблицы) для доступа по ключу;
  - **двусвязный список** для отслеживания порядка использования.

Это пример, где **в реальном коде** используются идеи:

- хеш‑таблица (`dict`);
- связный список;
- очередь с приоритетом «самый давно не использовавшийся».

#### 8.1.3. Очередь задач: FIFO + приоритет

На практике задачи обрабатываются:

- либо в порядке поступления (FIFO);
- либо по приоритету (более важные — раньше).

Структуры:

- `collections.deque` — очередь FIFO;
- `heapq` — приоритетная очередь.

Пример: простая очередь задач с приоритетом:

```python
import heapq
from dataclasses import dataclass, field

@dataclass(order=True)
class PriorityTask:
    priority: int
    task_id: int = field(compare=False)
    payload: str = field(compare=False)


class PriorityTaskQueue:
    def __init__(self) -> None:
        self._heap: list[PriorityTask] = []

    def put(self, task: PriorityTask) -> None:
        heapq.heappush(self._heap, task)

    def get(self) -> PriorityTask | None:
        if not self._heap:
            return None
        return heapq.heappop(self._heap)
```

Здесь:

- `heapq` обеспечивает **O(log n)** на вставку и извлечение;
- «самый маленький приоритет» = «самый высокий приоритет задачи» (или наоборот, по договору).

### 8.2. Аналитика и обработка данных: большие файлы, стриминг, агрегации

Типичная задача:

- есть **очень большой лог** (гигабайты);
- нужно:
  - посчитать агрегации (количество запросов по пользователям/ендпоинтам);
  - найти топ‑N самых частых значений.

#### 8.2.1. Словарь + однопроходный подсчёт

```python
from collections import Counter

def count_users_from_log(path: str, limit: int | None = None) -> Counter[str]:
    counts: Counter[str] = Counter()
    with open(path, "r", encoding="utf-8") as f:
        for i, line in enumerate(f):
            if limit is not None and i >= limit:
                break
            user_id = parse_user_id(line)  # ваша логика
            counts[user_id] += 1
    return counts
```

Идея:

- один проход по файлу → **O(n)** по времени, где n — количество строк;
- хранится только словарь: **O(u)** по памяти, где u — число уникальных пользователей.

Вместо `dict` + `+= 1` можно использовать `Counter`, который оптимизирован под такие сценарии.

#### 8.2.2. Топ‑N: сортировка vs куча

Найти топ‑N пользователей по количеству запросов:

```python
def top_n_users(counts: Counter[str], n: int) -> list[tuple[str, int]]:
    return counts.most_common(n)
```

`most_common` внутри:

- для больших словарей использует **кучу** (`heapq`);
- асимптотика близка к O(u log n), если n «маленькое» относительно u.

Наивный подход:

```python
sorted(counts.items(), key=lambda item: item[1], reverse=True)[:n]
```

- всегда сортирует **все** элементы → O(u log u);
- при большом u и малом n это заметно хуже кучи.

Вывод:

- понимать разницу между «отсортировать всё» и «вытащить топ‑N» с помощью кучи — **очень практично**.

### 8.3. ML/DS: подготовка данных, матрицы, графы особенностей

В ML/DS‑коде:

- часто используются массивы/матрицы (`numpy`, `pandas`);
- но концептуально это те же:
  - **массивы** и **векторы**;
  - **матрицы смежности** для графов (Graph Neural Networks, рекомендации).

Важно:

- понимать, что векторизация в `numpy` — это фактически использование **эффективных низкоуровневых алгоритмов** поверх массивов;
- алгоритмы работы с разреженными матрицами (sparse) опираются на идеи:
  - списков смежности;
  - экономии памяти за счёт хранения только ненулевых значений.

### 8.4. Dev‑tools: статический анализ, графы зависимостей

Инструменты:

- линтеры;
- форматтеры;
- анализаторы зависимостей (по импортам).

Внутри:

- строятся **графы модулей**:
  - вершины — модули/файлы;
  - рёбра — импорт «из модуля А в модуль Б».
- используются:
  - **DFS** для поиска циклических зависимостей;
  - **топологическая сортировка** для определения порядка сборки/загрузки.

Пример: упрощённая топологическая сортировка:

```python
def topo_sort(graph: dict[str, list[str]]) -> list[str]:
    visited: set[str] = set()
    temp_mark: set[str] = set()
    order: list[str] = []

    def visit(v: str) -> None:
        if v in temp_mark:
            raise ValueError("Цикл в графе зависимостей")
        if v in visited:
            return
        temp_mark.add(v)
        for u in graph.get(v, []):
            visit(u)
        temp_mark.remove(v)
        visited.add(v)
        order.append(v)

    for v in graph:
        if v not in visited:
            visit(v)

    order.reverse()
    return order
```

Здесь:

- мы используем **DFS + порядок выхода из рекурсии**;
- получаем упорядочивание без циклов (если их нет).

## 9. Типичные формулировки задач по темам

Этот раздел — **ориентир для подготовки к собеседованиям**.  
Для каждой темы указаны **типичные задачи** и **на что смотрят интервьюеры**.

### 9.1. Массивы, списки, слайды окон (sliding window)

Типичные задачи:

- найти подмассив с максимальной суммой (Kadane);
- найти длину подмассива, удовлетворяющего условию (сумма/число уникальных элементов);
- удалить дубликаты, сжать массив.

На что смотрят:

- умеете ли вы:
  - перейти от O(n²) к O(n) с помощью **скользящего окна**;
  - объяснить, почему окно «двигается только вперёд» (инвариант);
  - выбирать между `list`, `deque`, `set`, `dict`.

### 9.2. Хеш‑таблицы (`dict`, `set`)

Задачи:

- найти первый повторяющийся элемент;
- определить, есть ли пара элементов с заданной суммой;
- проверить, являются ли две строки анаграммами.

Сигналы:

- если вы **перебираете все пары** с вложенным циклом, хотя условие позволяет использовать `set`/`dict`, — это красный флаг;
- хотят увидеть:
  - решение через словарь/множество за O(n);
  - понимание компромиссов по памяти.

### 9.3. Строки

Задачи:

- разворот строки/слов;
- поиск первого уникального символа;
- сжатие строки (run‑length encoding);
- проверки палиндромов (с учётом/без учёта символов).

На что смотрят:

- аккуратность в обращении с индексами и срезами;
- умение писать **линейные решения** вместо «проверяем каждую подстроку»;
- понимание, когда достаточно встроенных методов (`in`, `find`, `split`, `join`), а когда нужно ручное решение.

### 9.4. Деревья и графы

Задачи:

- обход дерева разными способами (inorder, preorder, BFS);
- поиск пути между двумя вершинами графа;
- проверка, является ли дерево бинарным деревом поиска;
- поиск кратчайшего пути в «лабиринте» (решётка с препятствиями).

На что смотрят:

- различаете ли вы:
  - **дерево** и **общий граф**;
  - ориентированный и неориентированный;
  - взвешенный и невзвешенный;
- умеете ли:
  - уверенно применять BFS/DFS;
  - хранить **visited** и объяснить, зачем он нужен.

### 9.5. Динамическое программирование и жадные алгоритмы

Задачи:

- размен монет (минимум монет);
- рюкзак;
- лестница (`n` ступенек, можно шагать на 1 или 2, сколько способов?);
- задачи на максимальную сумму/количество с ограничениями.

На что смотрят:

- можете ли вы:
  - самим вывести **рекуррентное соотношение**;
  - объяснить, почему жадный подход не работает (если не работает);
  - предложить DP‑решение с оценкой O(n)/O(n²).

## 10. Подробные разборы задач «от формулировки до кода»

В этом разделе — **пошаговые разборы** нескольких典ичных задач.

### 10.1. Задача: максимальная сумма подмассива (Kadane)

> Дан массив целых чисел (могут быть отрицательные).  
> Найти максимальную возможную сумму непрерывного подмассива (как минимум один элемент).

#### Шаг 1. Наивная идея

Перебрать все пары индексов `i, j` и посчитать сумму `xs[i:j+1]`:

- двойной цикл по `i` и `j` → O(n²);
- каждый раз считать сумму с нуля → ещё O(n), всего O(n³);
- можно улучшить до O(n²), если использовать префиксные суммы, но это всё равно много.

#### Шаг 2. Наблюдение

Пусть мы идём слева направо и для каждой позиции `i` знаем:

- `best_ending_here` — **максимальная сумма подмассива, который заканчивается в i**.

Тогда:

- либо выгодно **расширить** предыдущий подмассив: `best_ending_here + xs[i]`;
- либо **начать новый** подмассив с `xs[i]`, если он лучше:
  - `best_ending_here = max(xs[i], best_ending_here + xs[i])`.

А глобальный ответ:

- максимум из всех `best_ending_here`.

#### Шаг 3. Код

```python
from typing import Iterable

def max_subarray_sum(xs: Iterable[int]) -> int:
    iterator = iter(xs)
    try:
        first = next(iterator)
    except StopIteration:
        raise ValueError("Массив не должен быть пустым")

    best_ending_here = first
    best_overall = first

    for x in iterator:
        best_ending_here = max(x, best_ending_here + x)
        if best_ending_here > best_overall:
            best_overall = best_ending_here

    return best_overall
```

Сложность:

- O(n) по времени, O(1) по памяти.

Идея:

- мы «переносим» максимум вперёд, решая **локальную подзадачу** для каждого суффикса.

### 10.2. Задача: количество путей в решётке (DP)

> Есть решётка размером \( m \times n \).  
> Можно двигаться только **вправо** или **вниз**.  
> Сколько существует различных путей из верхнего левого угла в правый нижний?

#### Шаг 1. Рекуррентное соотношение

Пусть `dp[i][j]` — количество путей в клетку `(i, j)`:

- в неё можно прийти:
  - сверху `(i-1, j)`;
  - слева `(i, j-1)`.
- значит:
  - `dp[i][j] = dp[i-1][j] + dp[i][j-1]`.

База:

- `dp[0][0] = 1` (один способ «уже стоять» в стартовой клетке);
- первая строка и первый столбец:
  - `dp[0][j] = 1` (можно только идти вправо);
  - `dp[i][0] = 1` (можно только идти вниз).

#### Шаг 2. Код

```python
def grid_paths(m: int, n: int) -> int:
    dp = [[0] * n for _ in range(m)]

    for i in range(m):
        dp[i][0] = 1
    for j in range(n):
        dp[0][j] = 1

    for i in range(1, m):
        for j in range(1, n):
            dp[i][j] = dp[i - 1][j] + dp[i][j - 1]

    return dp[m - 1][n - 1]
```

Сложность:

- O(m·n) по времени;
- O(m·n) по памяти, можно оптимизировать до O(n), если хранить только последнюю строку.

## 11. Как дальше прокачивать алгоритмическое мышление

1. **Минимум теории → максимум практики**:  
   на каждый новый алгоритм/структуру — **2–3 задачи**, написанные *руками*.
2. **Регулярность важнее рывков**:  
   лучше заниматься по 30–60 минут **каждый день**, чем один раз 6 часов.
3. **Микро‑дневник**:
   - записывать, какие задачи решены;
   - что было непонятно;
   - какие идеи нового алгоритма вы узнали.
4. **Связывать с реальной работой**:
   - замечать, где в кодовой базе используются:
     - словари/множества;
     - очереди/кучи;
     - графообразные структуры;
   - пытаться **осмысленно объяснить**, почему выбран именно этот инструмент.
5. **Разбирать чужие решения**:
   - смотреть решения других людей на платформах задач;
   - пытаться **объяснить каждый шаг** вслух или в конспекте.

Главная цель этой части — не «натаскать на задачки», а **научить видеть в любой задаче ядро структуры данных и алгоритма**, а затем **осознанно выбирать** подходящие инструменты в Python.

### 12.1. Таблица: “задача → структура данных”

| Ситуация / задача | Что хочется сделать | Подходящая структура | Почему |
|-------------------|---------------------|----------------------|--------|
| Нужна последовательность с быстрым доступом по индексу и редкими вставками в середину | Перебирать по индексам, хранить k‑й элемент | `list` | O(1) доступ по индексу, хорошая локальность данных |
| Часто добавляем/удаляем элементы в начале/конце | Реализовать очередь/дек | `collections.deque` | O(1) операции с обоих концов, оптимизировано под такие сценарии |
| Нужно много раз спрашивать «был ли уже этот элемент?» | Проверять наличие без порядка | `set` | Среднее O(1) на проверку вхождения |
| Нужна ассоциация ключ → значение | Хранить настройки, кэш, мапу id → объект | `dict` | Среднее O(1) на доступ по ключу, гибкость типов ключей |
| Нужно хранить отсортированную последовательность небольшого/среднего размера | Часто делаем бинарный поиск и вставку | `list` + `bisect` | Для десятков/сотен элементов комбинация вполне эффективна |
| Нужен быстро меняющийся минимум/максимум | Приоритеты задач, топ‑N по текущему потоку | `heapq` (минимальная куча) | O(log n) на вставку и извлечение минимума |
| Нужна история операций (undo/redo) | Откатывать состояния назад | Стек (`list` + `append`/`pop`) | LIFO, O(1) операции на вершине |
| Нужна модель графа (связи между объектами) | Роутинг, зависимости, граф вызовов | `dict` с `list`/`set` соседей (список смежности) | Гибко, разреженные графы, удобно для BFS/DFS |
| Нужен счётчик частот | Подсчитать, сколько раз встречается элемент | `collections.Counter` | Поверх `dict`, но с удобными методами (`most_common`) |
| Нужен LRU‑кеш | Кешировать результаты вызовов функции | `functools.lru_cache` / свой dict + двусвязный список | Реализует политику вытеснения, скрывая сложную логику |

### 12.2. Шаблон решения задач на “скользящее окно”

Задачи этого типа:

- есть массив/строка;
- ищем подмассив/подстроку с некоторым свойством;
- движем **левую** и **правую** границы окна.

**Шаблон:**

```python
def solve(xs: list[int]) -> int:
    left = 0
    best = 0
    # структура, в которой храним информацию об окне
    current_sum = 0

    for right, value in enumerate(xs):
        # 1. Расширяем окно вправо
        current_sum += value

        # 2. Пока нарушено условие — сдвигаем левую границу
        while not condition(current_sum):
            current_sum -= xs[left]
            left += 1

        # 3. Обновляем ответ по необходимости
        best = max(best, some_function_of_window(current_sum, left, right))

    return best
```

Ключевые моменты:

- **инвариант**: в любой момент `left <= right`, и внутреннее состояние соответствует окну `[left, right]`;
- **переход**:
  - `right` всегда идёт вперёд;
  - `left` также только увеличивается — поэтому суммарно алгоритм O(n), даже если внутри есть `while`.

### 12.3. Шаблон для задач “проверить, есть ли пара/тройка с определённой суммой”

Наивный подход:

- двойной/тройной вложенный цикл → O(n²)/O(n³).

Стандартные улучшения:

1. **Пара с суммой S** — использовать `set`:

```python
def has_pair_with_sum(xs: list[int], target: int) -> bool:
    seen: set[int] = set()
    for x in xs:
        need = target - x
        if need in seen:
            return True
        seen.add(x)
    return False
```

2. **Тройка с суммой S**:

- сортируем массив;
- фиксируем первый элемент;
- на оставшемся отрезке делаем **двухуказательный** поиск (`two pointers`).

```python
def has_triplet_with_sum(xs: list[int], target: int) -> bool:
    xs = sorted(xs)
    n = len(xs)
    for i in range(n):
        a = xs[i]
        left, right = i + 1, n - 1
        while left < right:
            s = a + xs[left] + xs[right]
            if s == target:
                return True
            if s < target:
                left += 1
            else:
                right -= 1
    return False
```

### 12.4. Шаблон для задач на графы (BFS/DFS)

**BFS** (поиск в ширину) — когда нужен:

- кратчайший путь по числу рёбер;
- минимальное количество шагов из состояния А в состояние B (игры, головоломки);
- нахождение всех вершин в “слое” от начальной.

Шаблон BFS:

```python
from collections import deque
from typing import Hashable

def bfs_layers(graph: dict[Hashable, list[Hashable]], start: Hashable) -> dict[Hashable, int]:
    dist: dict[Hashable, int] = {start: 0}
    queue = deque([start])

    while queue:
        v = queue.popleft()
        for u in graph.get(v, []):
            if u not in dist:
                dist[u] = dist[v] + 1
                queue.append(u)
    return dist
```

**DFS** — когда нужен:

- поиск компонент связности;
- проверка, есть ли цикл;
- топологическая сортировка.

Шаблон DFS (итеративный):

```python
def dfs_iterative(graph: dict[Hashable, list[Hashable]], start: Hashable) -> list[Hashable]:
    visited: set[Hashable] = set()
    order: list[Hashable] = []
    stack: list[Hashable] = [start]

    while stack:
        v = stack.pop()
        if v in visited:
            continue
        visited.add(v)
        order.append(v)
        for u in reversed(graph.get(v, [])):  # reversed, чтобы порядок был ближе к рекурсивному
            if u not in visited:
                stack.append(u)
    return order
```

### 12.5. Шаблон для DP “по строкам” (Левенштейн, LCS, подсчёт вариантов)

Типовые элементы:

- есть строки `s` и `t`;
- строится таблица `dp` размера `(len(s)+1) × (len(t)+1)`;
- переход опирается на 3–4 соседние клетки (слева, сверху, по диагонали).

Общий скелет:

```python
def solve_dp_on_strings(s: str, t: str) -> int:
    m, n = len(s), len(t)
    dp = [[0] * (n + 1) for _ in range(m + 1)]

    # 1. Инициализация базовых случаев
    for i in range(m + 1):
        dp[i][0] = base_for_s_prefix(i)
    for j in range(n + 1):
        dp[0][j] = base_for_t_prefix(j)

    # 2. Основной цикл
    for i in range(1, m + 1):
        for j in range(1, n + 1):
            # 3. Определение cost / совпадения
            cost = 0 if s[i - 1] == t[j - 1] else 1
            # 4. Переход
            dp[i][j] = combine(
                dp[i - 1][j],      # вариант 1
                dp[i][j - 1],      # вариант 2
                dp[i - 1][j - 1],  # вариант 3
                cost
            )

    # 5. Ответ
    return dp[m][n]
```

Под это можно подставить:

- расстояние Левенштейна;
- количество общих подпоследовательностей;
- задачи на выравнивание последовательностей.

### 12.6. Частые “подводные камни” в задачах на алгоритмы

- **Игнорирование крайних случаев**:
  - пустые массивы/строки;
  - один элемент;
  - все элементы одинаковые;
  - уже отсортированные или отсортированные в обратном порядке данные.
- **Мутация аргументов функции** без явного намерения:
  - сортировка списка «на месте» (`list.sort`) вместо `sorted`;

- **Неверное использование рекурсии**:
  - отсутствие базового случая;
  - рекурсия, которая не уменьшает задачу;
  - переполнение стека при глубокой рекурсии → там, где нужен явный стек.

## 14. Теоретические заметки и доказательства корректности

Этот раздел нужен, чтобы:

- **не просто “знать, что работает”**, а понимать **почему это работает всегда**;
- иметь возможность **самому выводить новые алгоритмы** или правильно адаптировать уже известные.

### 14.1. Почему BFS находит кратчайший путь (по числу рёбер)

Напоминание: BFS обходит граф слоями:

- сначала все вершины на расстоянии 1 ребро от старта;
- потом на расстоянии 2 рёбер;
- и так далее.

#### Формулировка утверждения

Пусть:

- граф невзвешенный (все рёбра имеют “стоимость” 1);
- стартовая вершина — `s`.

Тогда:

- после работы BFS расстояние `dist[v]` для любой вершины `v`, до которой можно добраться из `s`, равно **минимальному числу рёбер** в пути `s → v`.

#### Идея доказательства (по индукции по слоям)

1. **База**:
   - `dist[s] = 0` — очевидно, ноль рёбер.
2. **Индукционное предположение**:
   - считаем, что для всех вершин на расстоянии ≤ k от `s` BFS уже записал корректное расстояние.
3. **Переход**:
   - рассмотрим вершину `v`, находящуюся на расстоянии k+1 от `s` в некотором кратчайшем пути.
   - пусть `u` — её предшественник в этом пути (расстояние `dist[u] = k`).
   - когда BFS обрабатывает `u`, он смотрит на всех соседей, в том числе на `v`.  
     В этот момент:
     - `v` ещё не посещена → ей присвоится `dist[v] = dist[u] + 1 = k + 1`;
     - позже никакой другой путь не может дать меньшее расстояние, потому что:
       - любые другие вершины в очереди имеют расстояние ≥ k;
       - у BFS **строгий порядок по расстоянию**: сначала обрабатываются все вершины с расстоянием k, потом k+1 и т.д.

Таким образом, BFS **никогда не “перескакивает” через более короткий путь** и **никогда не даёт расстояние больше минимального**, что и требовалось.

Практическое следствие:

- как только вы достали вершину `t` из очереди BFS, **уже можно останавливать поиск** — кратчайший путь найден.

### 14.2. Почему алгоритм Дейкстры корректен (без отрицательных весов)

Напоминание:

- храним `dist[v]` — текущие оценки расстояний;
- на каждом шаге достаём из кучи вершину `v` с минимальным `dist[v]`;
- “фиксируем” её расстояние и расслабляем все рёбра `v → u`.

#### Инвариант

После того, как вершина `v` извлечена из кучи, **её `dist[v]` — окончательное минимальное расстояние** от старта до `v`.

#### Почему это верно

Предположим противное:

- есть путь `s → v` короче, чем `dist[v]` в момент извлечения из кучи;
- пусть `u` — **последняя вершина на этом кратчайшем пути**, которая ещё не была извлечена из кучи, а её предшественник `p` уже был извлечён раньше.

В момент извлечения `p`:

- мы релаксируем ребро `p → u`;
- получаем новую оценку `dist[u]` равную длине кратчайшего пути до `u`;
- при этом в куче **всегда хранится вершина с минимальным `dist`**.

Следовательно:

- `dist[u]` должно быть **не больше**, чем `dist[v]`;
- значит, `u` должна быть извлечена **раньше**, чем `v`, что противоречит выбору `u` как “последней необработанной” на кратчайшем пути.

Ключевая роль **неотрицательных весов**:

- если все веса неотрицательны, то как только путь до `v` найден, **любой новый путь через ещё необработанные вершины будет не короче**;
- с отрицательными весами это ломается: можно пойти “в обход” и удешевить путь после того, как вершина уже казалась оптимальной.

### 14.3. Почему жадный алгоритм для выбора интервалов оптимален

Задача напомнена в разделе 6.12:

- нужно выбрать максимальное количество непересекающихся интервалов.

Жадный алгоритм:

1. сортируем интервалы по возрастанию `end`;
2. проходим по ним слева направо;
3. берём интервал, если он не пересекается с уже взятым **последним**.

#### Скетч доказательства оптимальности (метод обменов)

Идея:

- берём **любое** оптимальное решение `OPT` (набор интервалов максимального размера);
- покажем, что можно **шаг за шагом “переделать” его в результат жадного алгоритма**, не уменьшая количество интервалов.

Шаг:

1. Пусть жадный алгоритм выбрал первый интервал `G1` (с минимальным `end`).
2. В оптимальном решении `OPT` первый интервал назовём `O1`.
3. Поскольку `G1` заканчивается не позже `O1`, мы можем:
   - **заменить** `O1` на `G1` в `OPT`,  
   - при этом **оставшиеся интервалы** в `OPT` по‑прежнему не пересекаются с `G1`, потому что они все начинаются не раньше конца `O1`, а `end(G1) ≤ end(O1)`.

Значит, существует оптимальное решение, где первый интервал — `G1`. Дальше повторяем рассуждение для хвоста. В итоге получаем, что решение жадного алгоритма **не хуже** любого оптимального, следовательно — **оптимально**.

### 14.4. Нижняя граница O(n log n) для сравнительных сортировок

Утверждение:

- Любой алгоритм сортировки, который опирается только на **сравнения элементов**, в худшем случае требует **O(n log n)** сравнений.

Идея доказательства:

1. **Модель решения**:
   - алгоритм сортировки можно представить в виде **дерева решений**;
   - в каждой вершине дерева делается сравнение `a[i] < a[j]`;
   - от результата (`True`/`False`) зависит, по какому ребру мы пойдём дальше;
   - листы дерева соответствуют **возможным перестановкам** входного массива.
2. **Количество листьев**:
   - всех возможных входных перестановок — `n!`;
   - значит, в дереве решений должно быть как минимум `n!` листьев (каждая перестановка должна привести к своему листу).
3. **Высота дерева решений**:
   - бинарное дерево высоты `h` имеет не более `2^h` листьев;
   - следовательно, `2^h ≥ n!` → `h ≥ log2(n!)`.
4. **Асимптотика `log(n!)`**:
   - по формуле Стирлинга `log(n!) ≈ n log n − n + O(log n)` → **O(n log n)**.

Следовательно:

- никакой сравнительный алгоритм сортировки не может иметь худший случай **лучше O(n log n)**.
- `Timsort`, `merge sort`, `heap sort` — асимптотически **оптимальны** в этой модели.

### 14.5. Почему DP‑решения не “теряют” оптимальность

DP часто описывают как “перебор с запоминанием”. Важно понимать, **почему мы можем ограничиться локальными переходами** (из меньших состояний в большие) и всё равно получить глобальный оптимум.

Ключевое свойство — **оптимальность подструктур**:

- если оптимальное решение задачи использует решения подзадач,
- то эти решения подзадач **тоже оптимальны**.

Примеры:

- в задаче о рюкзаке:
  - оптимальное заполнение рюкзака до веса `W` либо берёт предмет i, либо нет;
  - в обоих случаях используется **оптимальное заполнение** для меньшего веса/меньшего количества предметов.
- в расстоянии Левенштейна:
  - любой оптимальный путь преобразований между `s[:i]` и `t[:j]` использует оптимальные пути для меньших подстрок.

Формально:

1. предполагаем, что существует оптимальное решение, где подзадача решена **неоптимально**;
2. заменяем подрешение на **оптимальное** для этой подзадачи;
3. общая стоимость **улучшается или остаётся такой же**,  
   значит исходное решение не было оптимальным — противоречие.

Это свойство не всегда выполняется:

- если выбор в подзадаче влияет на **возможность** решений в других подзадачах, может понадобиться более сложная модель (например, графы состояний, дополнительные измерения в `dp`).

- начинать с **простой рекурсивной формулировки** (даже если она экспоненциальная);
- затем:
  - выделять повторяющиеся подзадачи;
  - переводить рекурсивную формулу в **таблицу dp**;
  - аккуратно продумывать порядок заполнения.

Со временем мозг начинает **распознавать типовые шаблоны**, и многие задачи превращаются в вариации 2–3 знакомых схем.

## 16. Глубокий разбор: динамическое программирование “на практике”

В этом разделе разберём DP **ещё более детально**, на нескольких типовых задачах:

- от **наивной рекурсии** → к **мемоизации** → к **табличному DP**;
- с **пошаговым анализом состояний** и переходов;
- с **разбором ошибок**, которые делают почти все.

### 16.1. Шаг за шагом: “лестница” (классический Fibonacci‑паттерн)

Задача:

> Есть лестница из `n` ступенек.  
> За один шаг можно подняться на 1 или 2 ступеньки.  
> Сколькими способами можно подняться на самый верх?

#### 16.1.1. Мыслительный процесс “с нуля”

1. **Не думаем сразу про DP.** Представим человека на ступеньке `n`.  
   Как он туда попал?
2. Ответ: последний шаг:
   - либо с `n-1` (шаг на 1),
   - либо с `n-2` (шаг на 2).
3. Значит, **все пути на `n`** состоят из:
   - всех путей на `n-1`, к которым добавили шаг на 1;
   - всех путей на `n-2`, к которым добавили шаг на 2.

Обозначим `f(n)` — количество способов попасть на ступеньку `n`. Тогда:

- `f(n) = f(n-1) + f(n-2)`.

База:

- `f(0) = 1` — “стоим внизу, уже на нулевой ступеньке”;
- `f(1) = 1` — можно сделать один шаг с 0 на 1.

#### 16.1.2. Наивная рекурсия и почему она медленная

```python
def ways_naive(n: int) -> int:
    if n == 0:
        return 1
    if n == 1:
        return 1
    return ways_naive(n - 1) + ways_naive(n - 2)
```

Что происходит:

- дерево вызовов разрастается экспоненциально;
- `ways_naive(5)` вызывает `ways_naive(4)` и `ways_naive(3)`;
- `ways_naive(4)` снова вызывает `ways_naive(3)` и `ways_naive(2)` и т.д.;
- одни и те же аргументы считаются десятки/сотни раз.

На рисунке (в голове/на бумаге) видно, что подзадачи **“перекрываются”**.

#### 16.1.3. Мемоизация: top‑down DP

Идея:

- если мы **уже считали `f(k)`**, второй раз считать не надо;
- сохраняем результат в кеш.

```python
from functools import lru_cache

@lru_cache(maxsize=None)
def ways_memo(n: int) -> int:
    if n == 0:
        return 1
    if n == 1:
        return 1
    return ways_memo(n - 1) + ways_memo(n - 2)
```

Что изменилось:

- логика задачи осталась **та же**;
- но теперь для каждого `n` рекурсивная функция считается **ровно один раз**;
- сложность по времени стала O(n), по памяти — O(n).

Это уже **динамическое программирование сверху вниз** (top‑down):

- двигаемся от большой задачи к маленьким;
- сохраняем решения подзадач.

#### 16.1.4. Табличная форма: bottom‑up DP

Теперь перепишем без рекурсии:

```python
def ways_dp(n: int) -> int:
    if n == 0:
        return 1
    if n == 1:
        return 1

    dp = [0] * (n + 1)
    dp[0] = 1
    dp[1] = 1

    for i in range(2, n + 1):
        dp[i] = dp[i - 1] + dp[i - 2]

    return dp[n]
```

Смысл:

- `dp[i]` играет роль `f(i)`;
- список `dp` — это **таблица значений** рекуррентной формулы;
- порядок заполнения (`i` от 2 до `n`) обеспечивает, что:
  - когда мы считаем `dp[i]`, значения `dp[i-1]` и `dp[i-2]` уже вычислены.

Оптимизация по памяти:

- нам никогда не нужны **все прошлые значения**, только два последних.

```python
def ways_dp_optimized(n: int) -> int:
    if n == 0:
        return 1
    if n == 1:
        return 1

    prev2 = 1  # f(0)
    prev1 = 1  # f(1)

    for _ in range(2, n + 1):
        cur = prev1 + prev2
        prev2, prev1 = prev1, cur

    return prev1
```

Вывод по “лестнице”:

- это **канонический шаблон DP**, который повторяется в массе задач:
  - количество путей/способов, зависящее от двух (или нескольких) предыдущих состояний.

### 16.2. Рюкзак 0/1: не только максимум, но и какие предметы

Эта задача уже была в главе, но здесь **разложим её очень медленно**.

Формулировка:

> Есть `n` предметов. У i‑го предмета:
> - вес `w[i]`,
> - ценность `v[i]`.
>
> Рюкзак выдерживает суммарный вес не больше `W`.  
> Каждый предмет можно взять **не более одного раза**.  
> Нужно:
> 1. узнать максимальную возможную суммарную ценность;
> 2. понять, **какие именно предметы** надо взять.

#### 16.2.1. Неформальное объяснение рекурсии

Представим себе, что мы стоим перед решением:

- рассмотреть **первые i предметов** и ограничение по весу `w`.

Есть два варианта:

1. **Не брать** i‑й предмет (индекс i‑1 в 0‑индексации):
   - тогда ценность = “лучшее, что можно сделать, используя первые i-1 предметов и вес `w`” = `dp[i-1][w]`.
2. **Взять** i‑й предмет (если он вообще помещается, то есть `w_i <= w`):
   - тогда мы “съели” `w_i` веса и добавили `v_i` ценности;
   - дальше можем тратить **оставшийся вес `w - w_i`** на первые i-1 предметов:
     - ценность = `v_i + dp[i-1][w - w_i]`.

Мы выбираем **лучший** из этих двух вариантов:

- `dp[i][w] = max(dp[i-1][w], dp[i-1][w - w_i] + v_i)` (если `w_i <= w`);
- если `w_i > w`, второй вариант невозможен → `dp[i][w] = dp[i-1][w]`.

#### 16.2.2. Что такое `dp[i][w]` на человеческом языке

- `i` — сколько первых предметов мы **разрешаем себе использовать**;
- `w` — какой **максимальный вес** мы можем потратить;
- `dp[i][w]` — **лучшая (максимальная) суммарная ценность**, которую можно получить, не:
  - превышая вес `w`;
  - и не используя больше, чем первые `i` предметов.

База:

- `dp[0][w] = 0` — если нам не разрешено использовать ни одного предмета, ценность 0 при любом весе;
- `dp[i][0] = 0` — если вес 0, мы ничего не можем взять, ценность 0.

#### 16.2.3. Код с восстановлением ответа

```python
from dataclasses import dataclass
from typing import List, Tuple

@dataclass
class Item:
    weight: int
    value: int


def knapsack_with_items(W: int, items: List[Item]) -> Tuple[int, List[int]]:
    n = len(items)
    # dp[i][w] — макс. ценность, используя первые i предметов при ёмкости w
    dp = [[0] * (W + 1) for _ in range(n + 1)]

    for i in range(1, n + 1):
        w_i = items[i - 1].weight
        v_i = items[i - 1].value
        for w in range(W + 1):
            best = dp[i - 1][w]  # вариант "не брать предмет i-1"
            if w_i <= w:
                cand = dp[i - 1][w - w_i] + v_i
                if cand > best:
                    best = cand
            dp[i][w] = best

    max_value = dp[n][W]

    # Восстановление: какие предметы мы реально взяли
    chosen: List[int] = []
    w = W
    for i in range(n, 0, -1):
        if dp[i][w] != dp[i - 1][w]:
            # Значит, предмет i-1 был взят
            chosen.append(i - 1)
            w -= items[i - 1].weight

    chosen.reverse()
    return max_value, chosen
```

Объяснение восстановления:

- мы идём **с конца** (от `i = n` к `1`);
- если значение `dp[i][w]` **совпадает** с `dp[i-1][w]`:
  - значит, i‑й предмет **не улучшал** решение → мы его не брали;
- если **не совпадает**:
  - значит, i‑й предмет **улучшил** решение → мы его брали;
  - убираем его вес `w_i` из оставшейся ёмкости и продолжаем.

### 16.3. Наибольшая возрастающая подпоследовательность (LIS): от простого к сложному

Задача:

> Дан массив чисел `a[0..n-1]`.  
> Нужно найти длину **наибольшей строго возрастающей подпоследовательности** (элементы идут по увеличению индекса, но не обязательно подряд).

#### 16.3.1. Почему “в лоб” — тяжело

- Если пытаться перебирать **все подпоследовательности**, их количество — 2ⁿ;
- это невозможно для сколь‑нибудь больших n.

Нужен **систематический** способ.

#### 16.3.2. Идея DP за O(n²)

Определим:

- `dp[i]` — длина наибольшей возрастающей подпоследовательности, **заканчивающейся ровно в позиции i**:
  - то есть последним элементом подпоследовательности является `a[i]`.

Как получить `dp[i]`?

- мы хотим взять `a[i]` **последним**;
- до него может быть любой `a[j]` (где `j < i`), но только если:
  - `a[j] < a[i]` (строго возрастающая подпоследовательность).

Тогда:

- если мы закончили подпоследовательность на `j`, её длина `dp[j]`;
- можем добавить `a[i]` и получить длину `dp[j] + 1`;
- нужно взять **лучший** такой `j`.

Формула:

```text
dp[i] = 1 + max(dp[j]) для всех j < i, где a[j] < a[i],
если таких j нет — dp[i] = 1.
```

Код:

```python
from typing import List

def lis_n2(a: List[int]) -> int:
    n = len(a)
    if n == 0:
        return 0

    dp = [1] * n

    for i in range(n):
        for j in range(i):
            if a[j] < a[i]:
                cand = dp[j] + 1
                if cand > dp[i]:
                    dp[i] = cand

    return max(dp)
```

Сложность:

- вложенный цикл по i и j → O(n²);
- память O(n).

Важно:

- этот алгоритм **прост и надёжен**;  
  для повседневной разработки и большинства задач его бывает достаточно.

#### 16.3.3. Интуиция O(n log n)‑решения (без жёстких формул)

Идея:

- поддерживаем массив `tails`, где:
  - `tails[k]` — **минимальный возможный хвост** возрастающей подпоследовательности длины `k+1`;
- когда смотрим на очередной элемент `x`:
  - находим (бинарным поиском) позицию `k`, **где `x` должен находиться** в `tails`:
    - если `x` больше всех — удлиняем `tails`;
    - иначе заменяем `tails[k]` на `x` (получаем более “выгодный” хвост).

Почему это работает:

- мы не знаем **конкретные элементы каждой LIS**, но знаем:
  - какие **минимальные возможные значения** могут быть хвостами LIS данной длины;
  - чем меньше хвост — тем больше шансов продолжить последовательность дальше.

Это продвинутая тема, важная для глубокой алгоритмической подготовки.  
Для прочной базы **достаточно хорошо понимать O(n²)‑версию** и уметь её написать.

### 16.4. DP по графу: считаем количество путей в DAG

Задача:

> Есть ориентированный ациклический граф (DAG).  
> Нужно посчитать, **сколько различных путей** существует из вершины `s` в вершину `t`.

#### 16.4.1. Переход “по здравому смыслу”

Пусть `dp[v]` — количество путей из `v` в `t`.

Тогда:

- если `v == t`, очевидно, что `dp[t] = 1` (есть один “путь длины 0” — стоять на месте);
- если `v` — любая другая вершина:
  - чтобы дойти из `v` до `t`, надо:
    - сначала пойти в одного из соседей `u`, потом из `u` дойти до `t`;
  - количество путей через конкретного `u` — `dp[u]`;
  - суммарно: `dp[v] = sum(dp[u] for u in соседях v)`.

Но есть проблема:

- если в графе будут **циклы**, можно ходить по ним бесконечно и “накручивать” пути;
- поэтому важно, что граф — **ациклический** (DAG).

#### 16.4.2. Почему нужен топологический порядок

Если мы попытаемся считать `dp[v]` в произвольном порядке, может оказаться:

- `dp[v]` зависит от `dp[u]`;
- `dp[u]` зависит от `dp[w]`;
- и так далее — сложно обеспечить, что зависимые значения уже посчитаны.

В DAG можно:

- построить **топологический порядок** вершин — такой, что все рёбра идут “вперёд”;
- тогда удобно идти **с конца** этого порядка:
  - для вершины `v` все `dp[u]` для её потомков уже известны.

```python
from collections import defaultdict
from typing import Hashable, Dict, List

def count_paths_dag(graph: Dict[Hashable, List[Hashable]], s: Hashable, t: Hashable) -> int:
    visited: set[Hashable] = set()
    order: List[Hashable] = []

    def dfs(v: Hashable) -> None:
        visited.add(v)
        for u in graph.get(v, []):
            if u not in visited:
                dfs(u)
        order.append(v)

    dfs(s)

    dp: Dict[Hashable, int] = defaultdict(int)
    dp[t] = 1

    for v in order:
        if v == t:
            continue
        total = 0
        for u in graph.get(v, []):
            total += dp[u]
        dp[v] = total

    return dp[s]
```

Эта задача объединяет:

- понимание графов (DAG, топологическая сортировка);
- и динамическое программирование (состояния и переходы).
