## 1. Картина целиком: где живут алгоритмы в Python‑разработке

Миф: «Алгоритмы и структуры данных нужны только олимпиадникам / собеседованиям».  
Реальность:

- **любой нетривиальный сервис** (backend, ETL, аналитика, highload, ML‑системы) постоянно упирается в:
  - время работы (сколько секунд/минут выполняется запрос/задача);
  - потребление памяти (можно ли обработать данные на одной машине);
  - пропускную способность (сколько запросов/сообщений в секунду выдержит система).
- **каждое решение “как хранить и обрабатывать данные” — это выбор структуры данных и алгоритма**:
  - лог запросов: список, очередь, куча по времени/приоритету;
  - кеш: словарь/OrderedDict/LRU‑структуры;
  - граф связей пользователей/сервисов: списки смежности + BFS/DFS;
  - поиск по тексту: индекс + строковые алгоритмы.

Python скрывает часть сложности (есть `list`, `dict`, `set`, `heapq`, `bisect` и т.д.), но:

- **под капотом — те же структуры и алгоритмы**, что и в учебниках;
- понимая их, вы:
  - перестаёте писать «скрыто медленный» код;
  - можете обосновать архитектурные решения;
  - легче проходите собеседования (где проверяют именно это).

## 3. Базовые структуры и алгоритмический инструментарий

### 3.1. Асимптотический анализ и O‑нотация (§70)

#### Что такое «сложность алгоритма»

**Идея:** вместо измерения времени «в секундах на моём ноутбуке» мы оцениваем **как время и память растут при увеличении размера входа `n`**.

- **Временная сложность** — сколько **элементарных операций** выполняет алгоритм в зависимости от \( n \).
- **Пространственная сложность** — сколько памяти он требует.

Нас интересует **порядок роста**, а не точные константы:

- алгоритм A: выполняет \( 3n + 10 \) операций → O(n);
- алгоритм B: выполняет \( 0.001n^2 \) операций → O(n²).

При маленьком `n` B может быть быстрее (меньше константа), но при большом `n` **квадратичный рост “переедет” линейный**.

#### Big‑O, Big‑Ω, Big‑Θ

- **Big‑O (O‑нотация)** — **верхняя оценка**.  
  «Алгоритм выполняется **не дольше**, чем C·f(n) для достаточно больших n».
- **Big‑Ω** — **нижняя оценка**.  
  «Алгоритм выполняется **не быстрее**, чем C·f(n)».
- **Big‑Θ** — **точная асимптотика**, когда верхняя и нижняя оценки совпадают.

На практике почти всегда говорят «сложность O(f(n))» и имеют в виду **порядок роста во “вменяемых” сценариях**.

#### Типичные классы сложности

- **O(1)** — константная.  
  Время **не зависит** от размера входа. Примеры:
  - доступ к элементу списка по индексу `lst[i]` (в среднем);
  - доступ к значению словаря по ключу `d[key]` (в среднем).

- **O(log n)** — логарифмическая.  
  В каждую итерацию вход **делится** на константный коэффициент (например, пополам):
  - бинарный поиск в отсортированном массиве.

- **O(n)** — линейная.  
  Количество операций пропорционально числу элементов:
  - один проход по списку, фильтрация, подсчёт суммы.

- **O(n log n)** — «почти линейно», часто встречается:
  - эффективные сортировки (merge sort, quicksort в среднем, Timsort);
  - многие алгоритмы на деревьях/графах.

- **O(n²)** — квадратичная.  
  Много вложенных циклов по всему массиву:
  - наивные сортировки (bubble, selection, insertion);
  - сравнение каждой пары элементов (двойной `for`).

- **O(2ⁿ)**, **O(n!)** — экспоненциальная и факториальная.  
  Обычно означают **полный перебор комбинаций/перестановок**:
  - наивное решение задачи о рюкзаке;
  - перебор всех перестановок элементов.

#### Пример на Python: считаем шаги

```python
def sum_linear(xs: list[int]) -> int:
    total = 0              # 1 операция
    for x in xs:           # n итераций
        total += x         # n операций
    return total           # 1 операция
```

Всего \(\approx 2n + 2\) операций → **O(n)**.

#### Амортизированный анализ

**Амортизированная сложность** — усреднённая по последовательности операций.

Классический пример — `list.append`:

- Внутри `list` — **динамический массив** с некоторым запасом ёмкости.
- Пока есть свободное место:
  - `append` добавляет элемент → **O(1)**.
- Когда место кончилось:
  - Python **выделяет новый, больший массив**, копирует туда все элементы и добавляет новый → эта конкретная операция стоит **O(n)**.

Если смотреть на **длинную последовательность `append`**, то:

- редкие дорогие расширения **размазываются** на множество дешёвых `append`;
- получается **амортизированная O(1)** на операцию.

### 3.2. Массивы и списки (§71)

#### Динамический массив в Python (`list`)

`list` в Python — это **динамический массив**:

- хранит элементы в **непрерывном блоке памяти**;
- знает **фактический размер** и **ёмкость** (capacity);
- при нехватке места **перевыделяет** массив большего размера и копирует элементы.

Ключевые свойства:

- **быстрый доступ по индексу**: `lst[i]` — O(1);
- **быстрый проход по всему списку** (локальность данных, хорошо кэшируется);
- **плохая вставка/удаление в начале и середине**: нужно сдвигать элементы.

#### Основные операции и их сложность

- `lst.append(x)` — **амортизированная O(1)**;
- `lst.pop()` (с конца) — O(1);
- `lst[i]`, `lst[i] = value` — O(1);
- `lst.insert(0, x)` — O(n) (сдвиг всех элементов вправо);
- `lst.pop(0)` — O(n) (сдвиг элементов влево);
- `lst[a:b]` — создание среза → O(k), где k — длина среза;
- конкатенация `lst1 + lst2` — O(n + m).

#### Срезы и копирование

Важный момент: **срез создаёт новый список**.

```python
xs = [1, 2, 3, 4, 5]
ys = xs[1:4]  # [2, 3, 4]

xs[1] = 200
print(xs)  # [1, 200, 3, 4, 5]
print(ys)  # [2, 3, 4] — не изменился
```

Если вы часто берёте **длинные срезы**, вы **копируете много данных**. Иногда выгоднее:

- хранить **индексы начала/конца**;
- использовать **итераторы** и `itertools` вместо явного копирования.

#### Типичная ошибка

```python
def bad_prefixes(xs: list[int]) -> list[list[int]]:
    prefixes = []
    for i in range(len(xs)):
        prefixes.append(xs[:i])  # каждый раз копируем O(i)
    return prefixes
```

- Внешний цикл — O(n).
- Внутри каждого шага — копирование среза длиной `i` → суммарно `1 + 2 + ... + (n-1) = O(n²)`.

### 3.3. Стеки и очереди (§72)

#### Стек (Stack) — LIFO

**Стек** — структура по принципу **LIFO (Last In, First Out)**:

- последним положили — первым достали;
- пример: история действий, отмена операций (undo), вызовы функций.

В Python стек удобно реализуется **через `list`**:

```python
stack: list[int] = []

stack.append(1)  # push
stack.append(2)
stack.append(3)

top = stack.pop()  # 3
top = stack.pop()  # 2
```

- `append` и `pop()` с конца — **O(1)**.

#### Очередь (Queue) — FIFO

**Очередь** — структура по принципу **FIFO (First In, First Out)**:

- первым пришёл — первым обслужен;
- пример: очередь запросов, задачи в очереди обработки.

**Неправильная реализация** через `list`:

```python
queue: list[int] = []

queue.append(1)   # enqueue
queue.append(2)
queue.append(3)

first = queue.pop(0)  # O(n) — сдвиг всех элементов
```

Для **частых pop(0)** это будет **медленно**.

#### Правильный инструмент: `collections.deque`

`deque` — двусторонняя очередь:

- **O(1)** `append`/`appendleft`/`pop`/`popleft`;
- оптимизирована для добавления/удаления с обоих концов.

```python
from collections import deque

queue = deque()

queue.append(1)
queue.append(2)
queue.append(3)

first = queue.popleft()  # O(1)
```

Используйте:

- `list` → как **стек** (append/pop);
- `deque` → как **очередь** или **дек**.

#### Очереди с приоритетами

Когда нужно доставать **элемент с минимальным/максимальным приоритетом**, подходит **очередь с приоритетами**:

- реализуется с помощью **кучи** (см. 3.5);
- в stdlib — `heapq` и `queue.PriorityQueue`.

### 3.4. Связные списки (§73)

В Python **нет стандартного связного списка** (`linked list`) в stdlib — потому что **в большинстве бытовых задач он не нужен**, достаточно `list`/`deque`.

Но важно понимать идею:

- **односвязный список** — структура из узлов:

```python
class Node:
    def __init__(self, value, next_=None):
        self.value = value
        self.next = next_
```

- каждый узел хранит **значение** и **ссылку на следующий узел**;
- голова списка (`head`) — первый узел.

Характеристики:

- **вставка/удаление после известного узла** — O(1);
- **поиск по индексу** — O(n), нужно пройти список.

Используется:

- в учебных задачах;
- внутри некоторых структур (очереди, хеш‑таблицы в других языках);
- там, где важно **постоянное время вставки/удаления** в середине.

Для большинства Python‑проектов достаточно знать, **что это такое и чем отличается от массива**, и уметь при необходимости реализовать простейший вариант.

### 3.5. Кучи (Heaps) (§74)

**Куча** — структура данных на основе дерева, которая быстро позволяет:

- получить **минимальный (или максимальный)** элемент;
- добавить новый элемент;
- удалить минимальный/максимальный.

Важное:

- в Python `heapq` реализует **минимальную бинарную кучу** на массиве (`list`);
- инвариант: **каждый родитель ≤ своих детей**;
- элемент с минимальным значением всегда находится в корне (индекс 0).

#### Базовые операции `heapq`

```python
import heapq

data = [5, 1, 8, 3]
heapq.heapify(data)     # превращаем список в кучу, O(n)

heapq.heappush(data, 2) # добавить элемент, O(log n)
min_elem = heapq.heappop(data)  # достать минимальный, O(log n)
```

Примеры использования:

- **очередь задач по приоритету** (сначала более важные/срочные);
- поиск **k наименьших/наибольших** элементов:

```python
import heapq

nums = [5, 1, 8, 3, 10, 2]
print(heapq.nsmallest(3, nums))  # [1, 2, 3]
print(heapq.nlargest(2, nums))   # [10, 8]
```

#### Сложность

- `heapify` — O(n);
- `heappush` / `heappop` — O(log n);
- `nsmallest(k, iterable)`:
  - эффективно для **больших n и малых k**;
  - часто лучше, чем `sorted(iterable)[:k]`, которая стоит O(n log n).

### 3.6. Деревья (§75)

**Дерево** — иерархическая структура:

- узел имеет **родителя** (кроме корня) и **потомков**;
- нет циклов (в отличие от графа);
- естественно представляет:
  - файловую систему;
  - DOM‑дерево HTML;
  - иерархию категорий/разделов.

#### Бинарное дерево и BST

- **бинарное дерево** — у каждого узла не более двух детей (левый/правый);
- **бинарное дерево поиска (BST)**:
  - для каждого узла: `left < root < right`;
  - поиск по значению — **O(h)**, где h — высота дерева;
  - при балансировке h = O(log n), в худшем случае (вырожденное дерево) — O(n).

Обходы:

- **inorder** (лево, корень, право) — для BST даёт отсортированную последовательность;
- **preorder**, **postorder** — для других задач (копирование, удаление).

В стандартной библиотеке Python **нет готового BST**, но есть сторонние библиотеки, например:

- `sortedcontainers.SortedDict` — словарь, реализованный на деревьях.

### 3.7. Хеш‑таблицы: `dict` и `set` (§76)

`dict` и `set` — **ключевые структуры Python**, реализованные через **хеш‑таблицы**.

Идея:

- есть **хеш‑функция** `hash(key)` → число;
- это число используется для выбора «корзины» (bucket) в массиве;
- в корзине хранятся пары «ключ–значение» (или просто ключи для множества).

Сложность:

- **в среднем**:
  - чтение `d[key]` — O(1);
  - вставка `d[key] = value` — O(1);
  - удаление `del d[key]` — O(1).
- **в худшем случае** (много коллизий) — O(n), но в хорошо спроектированной реализации это почти не встречается.

Практические советы:

- словари и множества — **главный инструмент** для:
  - подсчёта частот;
  - быстрых проверок «содержится ли элемент»;
  - кеширования результатов;
  - реализации графов (списки смежности через словарь).
- помните, что хеш‑таблица:
  - **не сохраняет порядок** во всех языках,  
    но **в современных версиях Python** порядок вставки ключей сохраняется (это деталь реализации, на которую уже можно опираться).

### 3.8. Графы (§77)

**Граф** описывает **объекты и связи между ними**:

- вершины (nodes, vertices) — сущности: города, пользователи, сервисы;
- рёбра (edges) — связи: дороги, подписки, вызовы, зависимости;
- могут быть:
  - **ориентированные** (направленные);
  - **неориентированные**;
  - **взвешенные** (стоимость, длина, время).

#### Представления графа

1. **Список смежности (adjacency list)**:

```python
graph = {
    "A": ["B", "C"],
    "B": ["C"],
    "C": ["A"],
}
```

- удобно для разреженных графов;
- хранится только существующие рёбра.

2. **Матрица смежности (adjacency matrix)**:

- квадратная матрица `n × n`, где `matrix[i][j] = 1`, если есть ребро;
- удобна для плотных графов и некоторых матричных алгоритмов.

В Python чаще всего используется **список смежности** через `dict` + `list`/`set`.

#### Поиск в ширину (BFS)

**BFS** (Breadth‑First Search) — поиск в ширину:

- обходит вершины **слоями** от начальной;
- находит **кратчайший путь по числу рёбер** в невзвешенном графе.

Использует **очередь (FIFO)**:

```python
from collections import deque

def bfs(graph: dict[str, list[str]], start: str) -> list[str]:
    visited: set[str] = set()
    order: list[str] = []
    queue = deque([start])

    visited.add(start)

    while queue:
        v = queue.popleft()
        order.append(v)
        for u in graph.get(v, []):
            if u not in visited:
                visited.add(u)
                queue.append(u)
    return order
```

Сложность: **O(V + E)**, где V — число вершин, E — число рёбер.

#### Поиск в глубину (DFS)

**DFS** (Depth‑First Search) — поиск в глубину:

- «ныряет» как можно глубже, затем откатывается;
- удобно для:
  - проверки связности;
  - поиска циклов;
  - топологической сортировки.

```python
def dfs(graph: dict[str, list[str]], start: str) -> list[str]:
    visited: set[str] = set()
    order: list[str] = []

    def _dfs(v: str) -> None:
        visited.add(v)
        order.append(v)
        for u in graph.get(v, []):
            if u not in visited:
                _dfs(u)

    _dfs(start)
    return order
```

### 3.9. Сортировки (§78)

Сортировка — один из самых важных классов алгоритмов.

#### Простые O(n²) сортировки

Используются в учебных примерах и **для очень маленьких массивов**.

1. **Bubble sort** (пузырьковая)
2. **Selection sort** (выбором)
3. **Insertion sort** (вставками)

Пример insertion sort:

```python
def insertion_sort(xs: list[int]) -> None:
    for i in range(1, len(xs)):
        key = xs[i]
        j = i - 1
        while j >= 0 and xs[j] > key:
            xs[j + 1] = xs[j]
            j -= 1
        xs[j + 1] = key
```

- В худшем случае — **O(n²)**;
- хорошо работает на **почти отсортированных** массивах.

#### Эффективные сортировки O(n log n)

- **Merge sort** (сортировка слиянием):
  - рекурсивно делит массив пополам;
  - сортирует каждую половину;
  - сливает два отсортированных массива;
  - **стабильная** (равные элементы сохраняют относительный порядок).

- **Quick sort**:
  - выбирает опорный элемент (pivot);
  - разделяет массив на элементы «меньше» и «больше» опорного;
  - рекурсивно сортирует части;
  - в среднем O(n log n), но в худшем случае — O(n²).

#### Timsort в Python

В Python `list.sort()` и `sorted()` используют **Timsort**:

- гибрид merge sort и insertion sort;
- оптимизирован под **реальные данные**:
  - хорошо обрабатывает уже частично отсортированные последовательности;
  - эффективно работает с большим количеством одинаковых элементов.

Поэтому в Python **почти всегда следует использовать встроенную сортировку**, а не писать свою.

### 3.10. Поиск (§79)

#### Линейный поиск

Самый простой способ найти элемент:

```python
def linear_search(xs: list[int], target: int) -> int | None:
    for i, x in enumerate(xs):
        if x == target:
            return i
    return None
```

- сложность — **O(n)**;
- не требует сортировки.

#### Бинарный поиск

Работает только на **отсортированном массиве**:

```python
def binary_search(xs: list[int], target: int) -> int | None:
    left, right = 0, len(xs) - 1
    while left <= right:
        mid = (left + right) // 2
        if xs[mid] == target:
            return mid
        elif xs[mid] < target:
            left = mid + 1
        else:
            right = mid - 1
    return None
```

- сложность — **O(log n)**;
- цена — нужно поддерживать **отсортированность**.

#### Модуль `bisect`

`bisect` — стандартный модуль для работы с **отсортированными списками**:

- `bisect_left(a, x)` — индекс, куда можно вставить `x`, сохранив порядок (слева);
- `bisect_right(a, x)` — аналогично, но справа;
- `insort_left`, `insort_right` — вставка с сохранением порядка.

```python
import bisect

xs = [1, 3, 4, 10]
idx = bisect.bisect_left(xs, 4)  # 2
idx2 = bisect.bisect_left(xs, 5) # 3
```

Полезно, когда:

- данные **редко модифицируются**, но часто выполняется поиск/вставка;
- нужен **упорядоченный список** небольшого/среднего размера.

### 3.11. Рекурсия и динамическое программирование (§80)

#### Рекурсия

**Рекурсия** — функция, которая вызывает **саму себя**.

Типичный шаблон:

```python
def f(n: int) -> int:
    # 1. Базовый случай
    if n == 0:
        return 1

    # 2. Рекурсивный вызов, который уменьшает задачу
    return n * f(n - 1)
```

Важно:

- всегда должен быть **базовый случай**, который **не вызывает** функцию дальше;
- каждый шаг должен **делать задачу проще** (уменьшать `n`, сокращать список и т.д.).

#### Мемоизация

Многие рекурсивные алгоритмы **повторно считают одни и те же значения**:

```python
def fib_slow(n: int) -> int:
    if n <= 1:
        return n
    return fib_slow(n - 1) + fib_slow(n - 2)  # экспоненциальное время
```

Решение — **мемоизация**: кеширование результатов.

```python
from functools import lru_cache

@lru_cache(maxsize=None)
def fib(n: int) -> int:
    if n <= 1:
        return n
    return fib(n - 1) + fib(n - 2)  # теперь O(n)
```

#### Динамическое программирование (DP)

**DP** — техника, когда мы:

1. Формулируем задачу через **рекуррентное соотношение**:
   - решение для `n` выражается через меньшие `k < n`.
2. Запоминаем промежуточные результаты:
   - либо **сверху вниз** (рекурсия + мемоизация),
   - либо **снизу вверх** (заполняем таблицу).

Пример: **Fibonacci снизу вверх**:

```python
def fib_dp(n: int) -> int:
    if n <= 1:
        return n
    dp = [0, 1] + [0] * (n - 1)
    for i in range(2, n + 1):
        dp[i] = dp[i - 1] + dp[i - 2]
    return dp[n]
```

Типичные задачи для DP:

- **рюкзак** (knapsack);
- **расстояние Левенштейна** (редактирование строк);
- **наибольшая общая подпоследовательность** (LCS).

### 3.12. Жадные алгоритмы (§81)

**Жадный алгоритм** (greedy) **на каждом шаге** выбирает «локально лучшее» решение:

- берёт **самый выгодный на данный момент** шаг;
- надеется, что из таких шагов сложится **глобально оптимальное** решение.

Примеры:

- **код Хаффмана** — построение оптимального префиксного кода;
- **выбор непересекающихся отрезков/активностей**:
  - сортируем по времени окончания;
  - всегда берём следующую активность, которая **начинается не раньше, чем заканчивается текущая**.

Важно:

- жадный алгоритм **работает не во всех задачах**;
- часто нужен **доказательный аргумент**, почему локальный выбор оптимален глобально.

---

### 3.13. Разделяй и властвуй (§82)

Шаблон **divide & conquer**:

1. **Разделить** задачу на несколько подзадач (обычно одинакового типа).
2. **Решить** каждую подзадачу (часто рекурсивно).
3. **Объединить** решения подзадач.

Примеры:

- **merge sort**:
  - делим массив пополам;
  - сортируем каждую половину;
  - сливаем;
- **binary search**:
  - сравниваем с элементом в середине;
  - решаем задачу для половины.

Преимущество:

- часто дает **O(n log n)** вместо O(n²);
- хорошо ложится на **рекурсивную реализацию**.

### 3.14. Строковые алгоритмы (§83)

Строки — одна из самых частых структур в приложениях:

- запросы;
- логи;
- текст;
- протоколы.

#### Поиск подстроки

1. **Наивный алгоритм** — сравниваем шаблон с каждой позицией строки → O(n·m).
2. Улучшенные:
   - **KMP (Knuth–Morris–Pratt)** — строит префикс‑функцию, позволяет «не откатываться назад» в исходной строке;
   - **Boyer–Moore** и варианты — начинают сравнение с конца шаблона, умеют «перепрыгивать» куски текста.

Для практики в Python обычно:

- использовать встроенные методы:
  - `"needle" in haystack`;
  - `haystack.find("needle")`;
- понимать, что под капотом применяются более быстрые алгоритмы, чем наивный.

#### Регулярные выражения

Модуль `re` реализует поиск по шаблонам:

- мощный инструмент, но:
  - не всегда эффективен;
  - некоторые шаблоны вызывают **катастрофический backtracking** (ReDoS).

#### Префиксные структуры: trie

**Trie** — дерево, где каждый уровень соответствует символу:

- удобно для:
  - автодополнения;
  - проверок по префиксам;
  - словарей с общими префиксами.

В Python редко реализуется вручную, но важно понимать идею.

### 3.15. Числовые алгоритмы (§84)

Числовые задачи часто попадаются в:

- конкурсах/собеседованиях;
- криптографии;
- обработке больших чисел.

Ключевые алгоритмы:

- **НОД (GCD) — алгоритм Евклида**:

```python
def gcd(a: int, b: int) -> int:
    while b != 0:
        a, b = b, a % b
    return a
```

- **быстрое возведение в степень** (бинарное):

```python
def power(base: int, exp: int) -> int:
    result = 1
    while exp > 0:
        if exp & 1:
            result *= base
        base *= base
        exp >>= 1
    return result
```

- **решето Эратосфена** — поиск всех простых до N;
- **комбинаторика**:
  - факториалы;
  - перестановки;
  - сочетания (`itertools.permutations`, `itertools.combinations`).

## 4. Мини‑шпаргалка по выбору структуры данных

- **Нужно просто хранить последовательность и быстро бегать по индексам?**  
  → `list`.

- **Нужно часто добавлять/удалять элементы в начале и в конце?**

- **Нужно быстро проверять наличие элемента / хранить уникальные элементы?**  
  → `set`.

- **Нужны пары ключ–значение с быстрым доступом по ключу?**

- **Нужно быстро получать минимум/максимум из изменяющегося множества?**  
  → `heapq` (`heapq.heappush` / `heapq.heappop`).

- **Нужен отсортированный список небольшого размера, по которому иногда ищем бинарным поиском?**  
  → `list` + `bisect`.

### 6.1. Углубление: асимптотика и O‑нотация (§70)

#### 6.1.1. Почему нас не интересуют константы

Когда мы говорим, что алгоритм «O(n)», мы:

- **отбрасываем константные множители** (2n, 100n → просто n);
- отбрасываем **низшие степени** (n² + 100n + 1000 → просто n²).

Почему это разумно:

- На практике нас интересует **поведение при росте n**:
  - для n = 1000 разница между 2n и 100n важна;
  - для n = 10⁶ разница между n и n² **критична**, а множители почти не играют роли.
- Константы **зависят от реализации и железа**:
  - компилятор;
  - интерпретатор (CPython / PyPy);
  - CPU, кеш, память и т.д.

Мы сознательно **жертвуем точностью ради универсальности**: утверждение «алгоритм O(n log n)» будет правдой и через 10 лет, даже если реализация и железо изменятся.

#### 6.1.2. Лучший, средний и худший случай

У многих алгоритмов есть:

- **лучший случай** — когда данные «удобные»;
- **средний случай** — типичный сценарий;
- **худший случай** — самый тяжёлый.

Пример: **линейный поиск** по списку:

- лучший случай: элемент — **первый** → O(1);
- средний случай: элемент где‑то посередине → O(n);
- худший случай: элемента нет, нужно проверить **всё** → O(n).

В алгоритмах обычно указывают:

- **худший случай**, если нужно гарантировать время работы (криптография, real‑time системы);
- **средний случай**, если худший возможен, но искусственен (quicksort при случайном выборе pivot).

#### 6.1.3. Практическая оценка сложности на Python

Для реального кода иногда полезно:

1. **Предсказать** порядок сложности «на бумаге».
2. **Проверить** гипотезу простым экспериментом (`timeit`).

```python
from timeit import timeit

def build_list(n: int) -> list[int]:
    return [i for i in range(n)]

for n in (10_000, 20_000, 40_000, 80_000):
    t = timeit(lambda: build_list(n), number=10)
    print(n, t)
```

Вы увидите, что время примерно **удваивается при удвоении n** → поведение **линейное**, O(n), даже если в точных числах есть разброс.

#### 6.1.4. Временная vs пространственная сложность

**Пространственная сложность** часто забывают, но она критична:

- держать в памяти «все данные» иногда **невозможно**;
- нужно уметь **стримить** (обрабатывать по частям).

Пример:

```python
# Плохо: читаем весь файл в память
with open("big.log", "r", encoding="utf-8") as f:
    lines = f.readlines()   # O(n) по памяти

# Лучше: читаем построчно (генератор)
with open("big.log", "r", encoding="utf-8") as f:
    for line in f:
        process(line)       # почти O(1) доп. памяти
```

Здесь алгоритмическая **временная сложность** может быть одинаковой (O(n)), но **пространственная** радикально различается.

#### 6.1.5. Типичные ошибки в оценке сложности

- **Игнорировать вложенные циклы**:
  - «У меня один цикл — значит O(n)»;
  - но внутри цикла вы делаете `in` по списку (O(n)) → получаете O(n²).

- **Путать `list` и `set`**:
  - `x in list_` — O(n);
  - `x in set_` — O(1) в среднем.

- **Не учитывать срезы**:
  - `xs[:i]` — **копирование** O(i), а не O(1).

Мини‑задача:

> В функции три вложенных цикла по n, но во внутреннем цикле `break` срабатывает почти сразу. Какую сложность вы назовёте на собеседовании?

Ответ: **формально O(n³)**, но можно добавить **пояснение**, что на реальных данных внутренний цикл часто завершается раньше, и фактическое время ближе к O(n²) или O(n² log n) — в зависимости от конкретной логики.

### 6.2. Углубление: массивы и списки (§71)

#### 6.2.1. Модель памяти динамического массива

Представьте `list` как:

- **массив ячеек фиксированного размера**, хранящих ссылки на объекты;
- две переменные:
  - `size` — сколько элементов реально лежит;
  - `capacity` — сколько ячеек выделено.

Когда `size < capacity`:

- `append` просто записывает ссылку в следующую ячейку → **O(1)**.

Когда `size == capacity`:

- Python:
  1. Выделяет **новый массив большего размера**.
  2. **Копирует** туда все ссылки.
  3. Освобождает старый массив.

Это дорого (O(n)), но происходит **редко** → амортизированная O(1).

#### 6.2.2. Паттерны эффективного использования списков

- **Не вставляйте в начало** часто:
  - `insert(0, x)`, `pop(0)` — O(n);
  - лучше `deque` или изменить логику (перебирать с конца).

- **Не создавайте огромное количество временных списков**:

```python
# Нехорошо
xs = transform_3(transform_2(transform_1(xs)))

# Лучше (если можно стейджить вычисления)
xs = transform_1(xs)
xs = transform_2(xs)
xs = transform_3(xs)
```

Ещё лучше — использовать **итераторы** и `itertools`, если возможна ленивая обработка.

#### 6.2.3. Списки vs генераторы

```python
def squares_list(n: int) -> list[int]:
    return [i * i for i in range(n)]  # создаёт список

def squares_gen(n: int):
    for i in range(n):
        yield i * i  # генератор, по одному значению
```

Список:

- мгновенно выделяет память под n элементов;
- удобен, если нужны **все значения сразу**.

Генератор:

- хранит только **состояние итерации**;
- подходит для:
  - обработки **потоков данных**;
  - ситуаций, где мы прерываемся рано (`break`).

### 6.3. Углубление: стеки и очереди (§72)

#### 6.3.1. Рекурсия vs явный стек

Любая рекурсивная функция может быть переписана через **явный стек**:

```python
def dfs_recursive(graph, start):
    visited = set()

    def _dfs(v):
        visited.add(v)
        for u in graph.get(v, []):
            if u not in visited:
                _dfs(u)

    _dfs(start)
    return visited


def dfs_iterative(graph, start):
    visited = set()
    stack = [start]

    while stack:
        v = stack.pop()
        if v in visited:
            continue
        visited.add(v)
        for u in graph.get(v, []):
            if u not in visited:
                stack.append(u)
    return visited
```

Здесь:

- рекурсивная версия использует **стек вызовов интерпретатора**;
- итеративная — **наш собственный список как стек**.

Это важно:

- рекурсия в Python имеет **ограничение глубины** (`sys.getrecursionlimit()`), явный стек — нет;
- в задачах с очень глубокими деревьями/графами лучше использовать **итеративный DFS**.

#### 6.3.2. Реализация простой очереди задач

```python
from collections import deque
from dataclasses import dataclass

@dataclass
class Task:
    id: int
    payload: str


class TaskQueue:
    def __init__(self) -> None:
        self._q: deque[Task] = deque()

    def put(self, task: Task) -> None:
        self._q.append(task)

    def get(self) -> Task | None:
        if not self._q:
            return None
        return self._q.popleft()

    def __len__(self) -> int:
        return len(self._q)
```

Это простой пример, но он подчёркивает:

- **интерфейс очереди** (`put`/`get`);
- **сложность операций** — O(1) для добавления и извлечения.

### 6.4. Углубление: связные списки (§73)

#### 6.4.1. Реализация и операции

```python
class Node:
    def __init__(self, value, next_=None):
        self.value = value
        self.next = next_


class SinglyLinkedList:
    def __init__(self) -> None:
        self.head: Node | None = None

    def push_front(self, value) -> None:
        self.head = Node(value, self.head)  # O(1)

    def append(self, value) -> None:
        if self.head is None:
            self.head = Node(value)
            return
        cur = self.head
        while cur.next is not None:   # O(n)
            cur = cur.next
        cur.next = Node(value)

    def find(self, value) -> Node | None:
        cur = self.head
        while cur is not None:        # O(n)
            if cur.value == value:
                return cur
            cur = cur.next
        return None
```

Здесь видно:

- вставка в голову — **O(1)**;
- поиск/вставка в конец — **O(n)**, потому что нужно пройти список.

#### 6.4.2. Почему в Python почти не используют linked list

- Каждый `Node` — **отдельный объект**, с накладными расходами по памяти;
- переход по `next` **плохо использует кеш** процессора;
- почти всегда проще:
  - `list` + индексы;
  - `deque`;
  - специализированные структуры из библиотек.

Связные списки важны **как концепт**:

- для понимания **устройств других структур** (очереди, хеш‑таблицы в C/Java);
- на собеседованиях.

### 6.5. Углубление: кучи и `heapq` (§74)

#### 6.5.1. Модель бинарной кучи

Бинарная куча (min‑heap):

- представляется массивом `a`;
- для индекса `i`:
  - левый ребёнок: `2*i + 1`;
  - правый: `2*i + 2`;
  - родитель: `(i - 1) // 2`.

Инвариант:

- `a[i] <= a[left_child(i)]` и `a[i] <= a[right_child(i)]` (если дети есть).

Из инварианта:

- минимальный элемент **всегда в `a[0]`**;
- при вставке/удалении мы:
  - вставляем/меняем элемент внизу;
  - «просеиваем» его вверх/вниз (`sift_up` / `sift_down`) за O(log n).

#### 6.5.2. Пример: слияние k отсортированных списков

Задача:

- есть k отсортированных списков;
- нужно получить **один общий отсортированный**.

Решение через кучу:

```python
import heapq

def merge_k_sorted(lists: list[list[int]]) -> list[int]:
    heap: list[tuple[int, int, int]] = []
    # (значение, индекс_списка, индекс_внутри_списка)

    for list_idx, xs in enumerate(lists):
        if xs:  # не пустой
            heapq.heappush(heap, (xs[0], list_idx, 0))

    result: list[int] = []

    while heap:
        value, list_idx, elem_idx = heapq.heappop(heap)
        result.append(value)

        next_idx = elem_idx + 1
        if next_idx < len(lists[list_idx]):
            next_value = lists[list_idx][next_idx]
            heapq.heappush(heap, (next_value, list_idx, next_idx))

    return result
```

Анализ:

- всего элементов `N` (сумма длин списков);
- каждая операция `heappush`/`heappop` — O(log k);
- всего O(N log k).

Наивное решение (склеить все списки и вызвать `sorted`) — O(N log N).  
Когда `k` меньше `N`, куча даёт **выигрыш**.

### 6.6. Углубление: деревья (§75)

#### 6.6.1. Реализация простого BST

```python
class BSTNode:
    def __init__(self, key, left=None, right=None):
        self.key = key
        self.left: BSTNode | None = left
        self.right: BSTNode | None = right

    def insert(self, key):
        if key < self.key:
            if self.left is None:
                self.left = BSTNode(key)
            else:
                self.left.insert(key)
        elif key > self.key:
            if self.right is None:
                self.right = BSTNode(key)
            else:
                self.right.insert(key)
        # равные ключи можно хранить отдельно или считать дубликатами

    def find(self, key) -> bool:
        if key == self.key:
            return True
        if key < self.key and self.left is not None:
            return self.left.find(key)
        if key > self.key and self.right is not None:
            return self.right.find(key)
        return False
```

Сложность:

- **в среднем** (случайные данные) — высота дерева h ≈ O(log n);
- операции `insert`/`find` — O(h) = O(log n);
- **в худшем** (отсортированные данные) дерево вырождается в список → O(n).

Поэтому на практике используют **самобалансирующиеся деревья** (AVL, Red‑Black и др.), которые гарантируют h = O(log n).

#### 6.6.2. Обходы дерева

```python
def inorder(node: BSTNode | None) -> list:
    if node is None:
        return []
    return inorder(node.left) + [node.key] + inorder(node.right)


def preorder(node: BSTNode | None) -> list:
    if node is None:
        return []
    return [node.key] + preorder(node.left) + preorder(node.right)


def postorder(node: BSTNode | None) -> list:
    if node is None:
        return []
    return postorder(node.left) + postorder(node.right) + [node.key]
```

Для BST:

- `inorder` даёт **отсортированный список ключей** → полезно для проверки корректности.

### 6.7. Углубление: хеш‑таблицы, `dict`, `set` (§76)

#### 6.7.1. Концепция коллизий

**Коллизия** — когда два разных ключа имеют **одинаковый хеш**.

Хеш‑функция:

- отображает множество ключей (строки, кортежи, числа) в большое, но конечное множество целых;
- по принципу «голубятни» (pigeonhole principle) коллизии **неизбежны**.

Стратегии борьбы:

- **открытая адресация** — при коллизии ищем другую свободную ячейку;
- **цепочки** — в каждой ячейке хранится список/связный список элементов.

В CPython используется **сложная гибридная схема** с открытой адресацией и эвристиками перераспределения.

#### 6.7.2. Зачем знать про хеш‑таблицы Python‑разработчику

- понять, почему:
  - `in` по словарю/множеству — O(1) в среднем;
  - перебор **всех** элементов всё равно O(n);
  - **порядок ключей** (начиная с Python 3.7) сохраняется **по порядку вставки**;
- понимать эффекты:
  - `dict` **расширяется** при большом числе элементов — как `list`;
  - при «почти полном» заполнении производительность падает.

#### 6.7.3. Сравнение: `list` vs `set` для проверок вхождения

```python
def contains_list(xs: list[int], target: int) -> bool:
    return target in xs  # O(n)


def contains_set(xs: set[int], target: int) -> bool:
    return target in xs  # O(1) в среднем
```

Если вы часто делаете проверки вида «элемент уже был/не был», почти всегда выгоднее:

- превратить список в множество один раз (O(n));
- далее выполнять проверки за O(1).

### 6.8. Углубление: графы (§77) — кратчайшие пути и остовные деревья

#### 6.8.1. Кратчайший путь: алгоритм Дейкстры

Условие:

- граф **ориентированный или нет**;
- рёбра имеют **неотрицательные веса**.

Цель:

- найти **кратчайшее расстояние** от одной вершины до всех остальных.

Идея Дейкстры:

1. Храним для каждой вершины **текущую оценку расстояния** `dist[v]` (инициализируем бесконечностью).
2. Берём вершину с **минимальной оценкой** (через кучу).
3. «Фиксируем» её расстояние (оно уже оптимально).
4. **Релаксируем** рёбра из этой вершины (пытаемся улучшить `dist[сосед]`).

```python
import heapq

INF = float("inf")

def dijkstra(graph: dict[str, list[tuple[str, int]]], start: str) -> dict[str, float]:
    dist: dict[str, float] = {v: INF for v in graph}
    dist[start] = 0.0

    heap: list[tuple[float, str]] = [(0.0, start)]

    while heap:
        cur_dist, v = heapq.heappop(heap)
        if cur_dist > dist[v]:
            continue  # устаревшая запись

        for u, w in graph.get(v, []):
            new_dist = cur_dist + w
            if new_dist < dist[u]:
                dist[u] = new_dist
                heapq.heappush(heap, (new_dist, u))

    return dist
```

Сложность:

- для V вершин и E рёбер: **O((V + E) log V)**.

Краткое обоснование корректности:

- когда вершина `v` извлекается из кучи, её `cur_dist` — **минимальная возможная длина пути** из `start` в `v`, иначе был бы путь короче, который привёл бы к более раннему извлечению.

#### 6.8.2. Bellman‑Ford

Когда в графе возможны **отрицательные веса рёбер**, Дейкстра не работает.  
Bellman‑Ford:

- проходит по всем рёбрам **V−1 раз** и релаксирует их;
- умеет обнаруживать **отрицательные циклы**, если на V‑й итерации расстояния всё ещё улучшаются;
- сложность — **O(V·E)**, медленнее, но более общий алгоритм.

#### 6.8.3. Минимальное остовное дерево: Prim и Kruskal

**Остовное дерево** (spanning tree) — подграф, который:

- содержит **все вершины**;
- является деревом (связный, без циклов).

**Минимальное остовное дерево (MST)** — остовное дерево **минимального суммарного веса рёбер**.

- Алгоритм **Prim** похож на Дейкстру:
  - растим дерево от одной вершины, всегда добавляя **самое дешёвое ребро**, соединяющее уже построенную часть с новой вершиной.
- Алгоритм **Kruskal**:
  - сортируем все рёбра по весу;
  - итерируемся по ним, добавляя ребро, если оно **не создаёт цикл**;
  - проверку «создаёт ли цикл» делаем через **Union‑Find (DSU)**.

#### 6.8.4. Union‑Find (Disjoint Set Union, DSU)

Структура данных для поддержки **разбиения множества** на непересекающиеся подмножества:

- операции:
  - `find(x)` — найти представителя множества, в котором лежит x;
  - `union(x, y)` — объединить множества, содержащие x и y.

Реализуется как **леса деревьев**:

- с **сжатием пути** (path compression);
- с **объединением по рангу/размеру**.

Эти оптимизации дают **почти константную амортизированную сложность** операций.

### 6.9. Углубление: сортировки (§78)

#### 6.9.1. Стабильность и её важность

Сортировка называется **стабильной**, если **равные элементы сохраняют относительный порядок**.

Пример:

```python
users = [
    {"name": "Alice", "age": 30},
    {"name": "Bob", "age": 25},
    {"name": "Carol", "age": 25},
]

sorted_by_age = sorted(users, key=lambda u: u["age"])
```

Если сортировка стабильная:

- для двух пользователей с возрастом 25 сначала будет Bob, потом Carol.

Почему это важно:

- можно делать **многоступенчатую сортировку**:

```python
users = sorted(users, key=lambda u: u["name"])  # 1: по имени
users = sorted(users, key=lambda u: u["age"])   # 2: по возрасту
```

После этого список будет отсортирован:

- сначала по возрасту;
- при одинаковом возрасте — по имени (как во втором критерии остаётся порядок из первого).

Потому что сортировка стабильна (`Timsort`).

#### 6.9.2. Использование `key` вместо `cmp`

В Python 3:

- вместо функции сравнения (`cmp`) используется функция **ключа** (`key`);
- это важно для производительности:
  - `key` вычисляется **один раз на элемент**;
  - сравниваются уже готовые ключи, а не вызывается тяжёлая функция сравнения многократно.

```python
words = ["10", "1", "20", "3"]

sorted_as_numbers = sorted(words, key=int)   # ['1', '3', '10', '20']
sorted_as_strings = sorted(words)            # ['1', '10', '20', '3']
```

#### 6.9.3. Анти‑паттерн: сортировать в цикле

```python
# Плохо
sorted_lists = []
for xs in list_of_lists:
    sorted_lists.append(sorted(xs))  # сортировка каждый раз
```

Если можно:

- либо отсортировать **один раз на входе**;
- либо пересмотреть алгоритм, чтобы не вызывать сортировку **во внутреннем цикле**, особенно если можно использовать кучу (`heapq`) или `bisect`.

### 6.10. Углубление: поиск (§79)

#### 6.10.1. Бинарный поиск и инвариант

Ключ к пониманию бинарного поиска — **инвариант отрезка** `[left, right]`:

- на каждом шаге **гарантируется**, что:
  - если элемент существует, он **обязательно** лежит в `[left, right]`.

```python
def binary_search(xs: list[int], target: int) -> int | None:
    left, right = 0, len(xs) - 1

    while left <= right:
        mid = (left + right) // 2
        if xs[mid] == target:
            return mid
        if xs[mid] < target:
            left = mid + 1   # теперь ищем только справа
        else:
            right = mid - 1  # ищем только слева

    return None
```

Инвариант:

- после обновления `left` или `right` мы **никогда не отбрасываем** отрезок, в котором потенциально может лежать `target`.

#### 6.10.2. Бинарный поиск на ответ

Шаблон:

1. Есть монотонная булева функция `f(x)`:
   - для `x` меньше некоторого порога `T` → False;
   - для `x >= T` → True.
2. Нужно найти минимальный `x`, для которого `f(x)` истина.

Мы можем:

- искать `x` бинарным поиском на отрезке `[low, high]`;
- в каждом шаге проверять `f(mid)` и сдвигать границы.

Пример задач:

- минимальный объём памяти/диска/серверов, при котором система выдержит нагрузку;
- минимальное время, за которое можно обработать N задач при k машинах.

### 6.11. Углубление: рекурсия и DP (§80)

#### 6.11.1. Шаги решения задач на DP

1. **Сформулировать состояние**:
   - что обозначает `dp[i]` или `dp[i][j]`?
2. **Определить переход**:
   - как выразить `dp[i]` через более «малые» состояния?
3. **Указать базовые случаи**:
   - какие значения известны сразу (обычно `i = 0`, `j = 0`).
4. **Выбрать порядок заполнения**:
   - в каком порядке состояние зависит от других?

Пример: **классический рюкзак 0/1**.

> Есть предметы с весом `w[i]` и ценностью `v[i]`.  
> Есть рюкзак максимального веса `W`.  
> Нужно максимизировать суммарную ценность, не превышая вес.

Состояние:

- `dp[i][w]` — **максимальная ценность**, которую можно набрать,
  используя **первые i предметов** и имея **допустимый вес w**.

Переход:

- не берём i‑й предмет:
  - `dp[i][w] = dp[i-1][w]`;
- берём i‑й предмет (если `w[i] <= w`):
  - `dp[i][w] = max(dp[i][w], dp[i-1][w - w[i]] + v[i])`.

Код:

```python
def knapSack(W: int, weights: list[int], values: list[int]) -> int:
    n = len(weights)
    dp = [[0] * (W + 1) for _ in range(n + 1)]

    for i in range(1, n + 1):
        w_i = weights[i - 1]
        v_i = values[i - 1]
        for w in range(W + 1):
            dp[i][w] = dp[i - 1][w]  # не берём предмет i
            if w_i <= w:
                cand = dp[i - 1][w - w_i] + v_i
                if cand > dp[i][w]:
                    dp[i][w] = cand

    return dp[n][W]
```

Сложность:

- по времени — O(n·W);
- по памяти — O(n·W), можно оптимизировать до O(W) при аккуратной одномерной DP.

#### 6.11.2. Расстояние Левенштейна

Состояние:

- `dp[i][j]` — минимум операций для превращения **первых i символов** строки `s` в **первые j символов** строки `t`.

Переход:

1. **Удаление**: `dp[i-1][j] + 1`;
2. **Вставка**: `dp[i][j-1] + 1`;
3. **Замена**: `dp[i-1][j-1] + cost`, где `cost = 0`, если `s[i-1] == t[j-1]`, иначе 1.

```python
def levenshtein(s: str, t: str) -> int:
    m, n = len(s), len(t)
    dp = [[0] * (n + 1) for _ in range(m + 1)]

    for i in range(m + 1):
        dp[i][0] = i
    for j in range(n + 1):
        dp[0][j] = j

    for i in range(1, m + 1):
        for j in range(1, n + 1):
            cost = 0 if s[i - 1] == t[j - 1] else 1
            dp[i][j] = min(
                dp[i - 1][j] + 1,        # удаление
                dp[i][j - 1] + 1,        # вставка
                dp[i - 1][j - 1] + cost  # замена/совпадение
            )

    return dp[m][n]
```

### 6.12. Углубление: жадные алгоритмы (§81)

#### 6.12.1. Пример, где жадный работает: выбор интервалов

Задача:

- есть множество интервалов `[start, end]`;
- нужно выбрать **максимальное число непересекающихся интервалов**.

Жадное решение:

1. Отсортировать интервалы по времени окончания `end`.
2. Идти по ним и брать интервал, который **не пересекается** с последним взятым.

```python
from dataclasses import dataclass

@dataclass
class Interval:
    start: int
    end: int


def select_intervals(intervals: list[Interval]) -> list[Interval]:
    intervals = sorted(intervals, key=lambda it: it.end)
    result: list[Interval] = []
    current_end = float("-inf")

    for it in intervals:
        if it.start >= current_end:
            result.append(it)
            current_end = it.end

    return result
```

Интуиция корректности:

- чем раньше «закончится» текущий интервал, тем **больше места остаётся** для последующих;
- выбирать самый ранний возможный конец — **глобально оптимальная стратегия**.

#### 6.12.2. Пример, где жадный ломается: размен монет

Система монет: 1, 3, 4.  
Нужно набрать сумму 6, минимизируя количество монет.

Жадный алгоритм (брать **самую крупную** монету, не превышающую остаток):

- 4 + 1 + 1 → 3 монеты.

Оптимальное решение:

- 3 + 3 → **2 монеты**.

Вывод:

- жадные алгоритмы нужно **строго обосновывать**;
- если нельзя доказать, что жадное решение оптимально, то обычно используют **DP**.

### 6.13. Углубление: разделяй и властвуй (§82)

#### 6.13.1. Рекуррентные соотношения и оценка

Многие divide & conquer‑алгоритмы описываются рекуррентным соотношением:

- \( T(n) = aT(\frac{n}{b}) + f(n) \),
  где:
  - `a` — сколько подзадач;
  - `n/b` — размер подзадачи;
  - `f(n)` — стоимость «склеивания».

Примеры:

- **merge sort**:
  - деление пополам: `a = 2`, `b = 2`;
  - слияние двух половин — O(n): `f(n) = n`;
  - итог: **O(n log n)**.

Интуиция:

- каждый уровень рекурсии «стоит» O(n) (слияние всех элементов на уровне);
- уровней **log n** (пока не разрежем на кусочки длиной 1);
- значит, общая сложность — **n · log n**.

### 6.14. Углубление: строковые алгоритмы (§83)

#### 6.14.1. KMP: идея префикс‑функции

KMP строит для шаблона `pattern` массив `pi`, где:

- `pi[i]` — длина **наибольшего собственного префикса**, который одновременно является суффиксом подстроки `pattern[:i+1]`.

Это позволяет:

- при несоответствии символа не «откатываться» в начало;
- а использовать уже известную информацию о совпавших префиксах/суффиксах.

В Python редко нужен вручную (есть быстрые встроенные операции), но **идея важна**:

- использование **дополнительной структуры данных (таблицы)**,
  чтобы избегать повторной работы.

#### 6.14.2. Реализация расстояния Левенштейна уже была в 6.11

Строковые алгоритмы часто сводятся к:

- подсчёту расстояний (Левенштейн, Дамерау–Левенштейн);
- построению префиксных деревьев/trie;
- построению суффиксных массивов/деревьев (более продвинутые темы).

### 6.15. Углубление: числовые алгоритмы (§84)

#### 6.15.1. Решето Эратосфена

```python
def sieve(n: int) -> list[int]:
    """Возвращает все простые числа <= n."""
    is_prime = [True] * (n + 1)
    is_prime[0] = is_prime[1] = False

    p = 2
    while p * p <= n:
        if is_prime[p]:
            for multiple in range(p * p, n + 1, p):
                is_prime[multiple] = False
        p += 1

    return [i for i, prime in enumerate(is_prime) if prime]
```

Сложность:

- по времени — O(n log log n);
- по памяти — O(n).

Идея:

- каждое составное число помечается как «непростое» кратно нескольким простым;
- количество операций получается существенно меньше, чем n².

#### 6.15.2. Быстрое возведение в степень и `pow`

Алгоритм двоичного возведения в степень (см. 3.15) позволяет:

- считать `a^b` за O(log b) умножений.

В Python ещё есть:

- встроенная функция `pow(a, b, mod)`:

```python
pow(a, b, m)  # возвращает (a ** b) % m эффективно
```

Она использует **быстрое возведение в степень по модулю**, не создавая огромные промежуточные числа.

#### 6.15.3. Большие целые числа

В Python `int` — **произвольной точности**:

- не переполняется, как в C/Java;
- но **операции становятся дороже** при росте числа:
  - сложение/вычитание — примерно O(k);
  - умножение — O(k²) или лучше (в зависимости от размера и алгоритма);
  - где k — число машинных слов в представлении числа.

Вывод:

- нельзя бездумно считать, что `int` всегда «почти бесплатно»;
- в криптографии и больших численных расчётах нужно учитывать цену операций с большими числами.
