## 5. Практика: как применять знания из PEP

### 5.1. Шаблон чтения PEP под конкретную задачу

Допустим, у вас есть задача:

- «мы хотим активнее использовать типизацию в проекте»  
  или  
- «нам нужно понять, что нас ждёт в Python 3.13 по части async».

Простой протокол:

1. Сформулируйте **основной вопрос**:  
   «типизация в библиотеке X», «миграция на `pyproject.toml`», «будущее GIL» и т.п.
2. На `pep.python.org`:
   - ищите по ключевым словам;
   - отфильтруйте по статусу (Final/Accepted/Active).
3. Для каждого кандидата:
   - прочитайте **Title + Abstract**;
   - если релевантно — переходите к **Motivation + Rationale**;
   - сохраните себе краткий конспект «проблема → решение → последствия».
4. По итогам:
   - составьте **маленькую таблицу** (номер PEP, тема, что меняется, влияние на вас);
   - при необходимости — вынесите это в обсуждение с командой.

#### 5.1.1. Пример: как найти PEP под свою реальную проблему

Вообразим ситуацию:

- у вас в проекте растёт объём кода;
- типизация используется кусочно;
- вы хотите «навести порядок» и понимаете, что без PEP не обойтись.

Шаги:

1. Формулируете вопрос:  
   «как правильно типизировать большой проект, чтобы не утонуть?».
2. На `pep.python.org` ищете `type hints`, `typing`, `py.typed`.
3. В списке PEP обращаете внимание на:
   - **PEP 484** (база для type hints);
   - **PEP 561** (типизированные пакеты);
   - PEP 585, 604, 544 и др. (уточнения и улучшения).
4. Для **PEP 484**:
   - читаете Abstract + Motivation;
   - понимаете цель: «опциональная типизация как дополнительный слой».
5. Для **PEP 561**:
   - узнаёте про `py.typed` и то, как ваши пакеты могут сигнализировать анализаторам:  
     «у нас типы — часть публичного API».

Результат:

- у вас не просто «есть типы»;
- вы понимаете:
  - какие PEP описывают базовые правила игры;
  - какие PEP расширяют их для ваших сценариев;
  - на какие номера можно потом ссылаться в ADR/документации.

### 5.2. Как использовать PEP при миграциях

Когда вы переходите, например, с 3.8 на 3.12:

- прочитайте «What's New» для промежуточных версий;
- возьмите оттуда **список PEP**, влияющих на:
  - синтаксис;
  - модули stdlib;
  - поведение рантайма;
- по каждому PEP:
  - посмотрите раздел **Backwards Compatibility**;
  - отметьте:
    - какие deprecated фичи мы используем;
    - какие warning'и могут появиться;
    - какие новые возможности стоит сразу внедрить.

Сделав так один раз, вы перестаёте воспринимать миграцию как «чёрный ящик»
и начинаете видеть её как **набор осмысленных изменений, каждое со своей историей**.

#### 5.2.1. Конкретный пример миграции: с Python 3.8 на 3.12

Упрощённый план:

1. Открываете `What's New in Python 3.9`, `3.10`, `3.11`, `3.12`.
2. Из каждого документа выписываете:
   - ссылки на PEP про:
     - синтаксис (`match/case`, `X | Y`, `:=`);
     - типизацию (новые фичи из `typing`);
     - изменения stdlib и рантайма.
3. Для каждого PEP:
   - смотрите разделы:
     - **Backwards Compatibility**;
     - **Deprecation** / **Removal**;
     - иногда — раздел «How to migrate».
4. Дальше превращаете это в список задач:
   - «заменить `asyncio.get_event_loop()` на новый API из PEP NNNN»;
   - «перейти с `typing.List` на `list[...]` там‑то и там‑то»;
   - «обновить обработку исключений с учётом `ExceptionGroup`, если используем `TaskGroup`».

Так миграция перестаёт быть «обновили интерпретатор и молимся»  
и превращается в серию осознанных, небольших шагов.

### 5.3. PEP и дизайн API

При проектировании API или внутренних соглашений:

- ищите:
  - PEP, описывающие **схожие абстракции** (контекстные менеджеры, протоколы, исключения);
  - PEP, задающие **широко принятые соглашения** (PEP 8, PEP 20, PEP 484 и др.).
- используйте формулировки из PEP:
  - в RFC/ADR внутри компании;
  - в docstring'ах и README:
    - «поведение хранит совместимость с PEP NNNN»;
    - «аннотации соответствуют рекомендациям PEP 484/544».

Это:

- уменьшает число «изобретений велосипеда»;
- облегчает вход новичкам, знакомым с Python, но не с вашим кодом;
- повышает предсказуемость взаимодействия между сервисами и библиотеками.

### 5.4. Личная база знаний по PEP

Практическая рекомендация:

- завести **отдельный раздел в личном или командном knowledge‑base**:
  - «PEP, важные для нашего проекта»;
  - для каждого:
    - номер, название, ссылка;
    - 3–5 строк: «что поменялось»;
    - 1–2 строки: «что это значит для нас».

Пример записи:

- **PEP 517/518 — pyproject.toml для сборки**  
  *Суть:* стандартизированный способ описания сборки проектов.  
  *Для нас:* новые проекты создаём сразу с `pyproject.toml`, старые постепенно мигрируем,
  чтобы избавиться от `setup.py`.

Со временем у вас сформируется **карта PEP, релевантных именно вашей работе**,
а не абстрактный реестр на сотни документов.

### 5.5. Мини‑упражнения

Чтобы закрепить материал, выполните хотя бы часть заданий:

1. Выберите **любой PEP из списка в глобальном плане**, который затрагивает ваши текущие задачи.
   - Прочитайте **Abstract, Motivation, Rationale**.
   - Напишите себе в 5–7 предложениях:
     - какую проблему он решает;
     - какие альтернативы обсуждались;
     - какие компромиссы выбраны.
2. Найдите **минимум один Superseded или Rejected PEP**:
   - разберитесь, почему идея была заменена или отклонена;
   - сформулируйте, чему это учит вас как дизайнера API/систем.
3. Создайте небольшой документ/заметку:
   - «PEP, которые мы учитываем в проекте N»;
   - внесите туда хотя бы 3–5 PEP с кратким описанием;
   - поделитесь с командой.

Если вы сделаете это один‑два раза, PEP перестанут казаться «чем‑то страшным с конференций core‑разработчиков» и превратятся в **удобный, понятный и практичный источник знаний** о том, *почему Python такой, какой он есть*.
