## 3. Ключевые PEP по категориям (с мотивацией)

Здесь мы пройдёмся по **основным кластерам PEP**, упомянутым в глобальном плане,
и каждый раз будем отвечать на один вопрос:

> **«Какую реальную боль они лечат и что для меня, как разработчика, из этого следует?»**

### 3.1. Синтаксис и семантика

#### PEP 8 — стиль кода как социальный контракт

- **Что это:** рекомендации по оформлению кода (отступы, длина строк, имена, импорты).
- **Мотивация:** Python создавался как язык, где **читаемость важнее «умности»**.  
  PEP 8 фиксирует:
  - единый стиль → меньше споров на ревью;
  - предсказуемость → проще читать чужой код.
- **Практический вывод:**  
  *PEP 8 — не «полицейский код‑стайл», а инструмент снижения трения в команде.*  
  Лучше один раз договориться «мы следуем PEP 8 (+ пару исключений)», чем обсуждать каждый пробел.

##### 3.1.1. PEP 8 «на пальцах»: зачем он вам

Представьте два файла:

- в первом отступы то 2 пробела, то 4;  
  имена переменных то `data`, то `d`, то `someVar`;  
  импорты разбросаны по всему файлу;
- во втором:
  - везде одинаковые отступы;
  - имена осмысленные и в одном стиле;
  - импорты вверху и упорядочены.

Во втором файле:

- вы быстрее понимаете чужой код;
- меньше тратите времени на «расшифровку» стиля;
- проще находите ошибки.

**PEP 8 = договорённость «пишем во втором стиле, а не в первом».**

Это важно по трём причинам:

- **Меньше конфликтов на ревью.**  
  Вместо:
  - «мне так красивее»;
  - «а мне так»  
  вы оба опираетесь на общий документ: «в PEP 8 написано вот так, давай ему следовать».

- **Легче заходить в новый код.**  
  Если вы видели стиль PEP 8 один раз, вам **легче** читать любой другой проект,
  который тоже ему следует.

- **Меньше мусора в голове.**  
  Вы тратите мозг **на логику**, а не на то, где тут пробел, а где нет.

Простой **микро‑шаблон**, как использовать PEP 8 в жизни:

- Лично для себя:
  - включить линтер/форматер (`flake8`, `ruff`, `black`, `isort`);
  - относиться к их замечаниям не как к «занудству», а как к «автоматическому редактору».
- В команде:
  - договориться: «мы в целом следуем PEP 8»;
  - список отступлений оформить отдельно («мы допускаем X и Y, хотя PEP 8 рекомендует Z»).

#### PEP 572 — оператор `:=` (walrus)

- **Проблема до PEP:** часто нужно:
  - вычислить значение;
  - проверить его;
  - и использовать.
  При этом приходилось либо:
  - дублировать выражение;
  - либо разрывать логику на несколько строк.

Пример до PEP 572:

```python
line = f.readline()
while line:
    process(line)
    line = f.readline()
```

С `:=`:

```python
while line := f.readline():
    process(line)
```

- **Мотивация:** уменьшить **дублирование и шум**, сделать типичные паттерны
  (чтение потоков, парсинг, поиск) компактнее при сохранении читаемости.
- **Компромисс:**  
  PEP обсуждает множество вариантов синтаксиса и приходит к выводу,
  что оператор должен:
  - быть **визуально заметным**;
  - не ломать уже существующий код;
  - не смешиваться по смыслу с `=` в выражениях.

##### 3.1.2. Как думать про `:=`, чтобы не запутаться

**Если совсем по‑простому:**

- `=` — обычное присваивание в **отдельной строке**;
- `:=` — присваивание **внутри выражения**, чаще всего:
  - в условии `if`;
  - в цикле `while`;
  - в генераторах и comprehensions.

Типичный полезный случай:

```python
# До: считаем длину два раза
if len(items) > 10:
    print("Too many:", len(items))

# После: один раз посчитали и переиспользовали
if (n := len(items)) > 10:
    print("Too many:", n)
```

Чем это хорошо:

- вы **не дублируете** тяжёлое или длинное выражение;
- при этом **видно**, что происходит присваивание (символ `:` бросается в глаза).

Где `:=` может навредить:

- когда вы пытаетесь «уместить полпрограммы в одну строку»:

```python
if (user := get_user()) and (order := get_order(user)) and (items := order.items):
    process(items)
```

Формально это ок, но читать сложно. Здесь лучше развернуть в несколько обычных строк.

**Практическое правило для себя:**

- Использовать `:=`, когда:
  - вы **убираете дублирование**;
  - и при этом строка остаётся читаемой;
  - и вы можете проговорить вслух: «здесь я *один раз вычисляю X и сразу его использую*».
- Не использовать, когда:
  - строка становится слишком длинной;
  - в ней появляется несколько `:=` подряд;
  - вы сами с трудом читаете этот код через день.

#### PEP 634–636 — структурное сопоставление с образцом (`match/case`)

- **Проблема:** в Python традиционно много кода вида:

```python
if kind == "point":
    ...
elif kind == "circle":
    ...
elif kind == "polygon" and len(points) == 3:
    ...
```

или вложенных `if`/`isinstance`/проверок структуры словарей.

- **Мотивация:** дать **декларативный способ описывать форму данных**:
  - pattern matching из функциональных языков;
  - возможность выразить «если объект такой структуры — делаем X».

Пример из PEP (упрощённо):

```python
match command:
    case ["load", filename]:
        load(filename)
    case ["save", filename]:
        save(filename)
    case _:
        raise ValueError("Unknown command")
```

- **Практический вывод:**  
  этот PEP не просто «синтаксический сахар», а приглашение:
  - думать **структурами данных**;
  - документировать возможные варианты явно;
  - писать более проверяемый и исчерпывающий код (exhaustive matching).

##### 3.1.3. `match/case` «на пальцах»: когда это реально помогает

Подумайте о типичном коде обработки команд/сообщений:

```python
if cmd["type"] == "load":
    load(cmd["filename"])
elif cmd["type"] == "save":
    save(cmd["filename"])
elif cmd["type"] == "delete":
    delete(cmd["filename"])
else:
    raise ValueError("Unknown command")
```

Проблемы такого подхода:

- в каждой ветке вы снова обращаетесь к `cmd["type"]`, `cmd["filename"]`;
- легко допустить ошибку в ключе словаря (`"filenmae"` вместо `"filename"`);
- с ростом числа веток код превращается в «лес `elif`».

С `match/case` та же логика записывается **как таблица вариантов**:

```python
match cmd:
    case {"type": "load", "filename": name}:
        load(name)
    case {"type": "save", "filename": name}:
        save(name)
    case {"type": "delete", "filename": name}:
        delete(name)
    case _:
        raise ValueError("Unknown command")
```

Что здесь важно:

- Вы **один раз** говорите, что работаете с объектом `cmd`.
- В каждом `case` явно описана **форма данных**:
  - есть ключ `"type"` с конкретным значением;
  - есть ключ `"filename"`, который сразу даёт вам переменную `name`.

**Если совсем по‑простому:**

- `if/elif` — это «руками проверяем кусочки условия»;
- `match/case` — это «описываем шаблоны данных, с которыми умеем работать».

Где это особенно полезно:

- когда у вас много вариантов структур:
  - разные формы словарей (разные ключи);
  - разные варианты кортежей/списков;
  - иерархии классов (pattern matching по типу и полям).

Где не стоит усложнять:

- если у вас **один‑два простых условия**, обычный `if` понятнее и привычнее;
- `match/case` раскрывает потенциал, когда:
  - вариантов действительно много;
  - вам важно, чтобы **все варианты были перечислены явно** (exhaustive).

#### PEP 479 — `StopIteration` в генераторах

- **Проблема:** ранее выброс `StopIteration` внутри генератора мог **тихо завершать** цикл,
  что приводило к неожиданным багам:
  - трудно понимать, где ошибка, а где нормальное завершение.
- **Мотивация:** сделать поведение генераторов **более предсказуемым и явным**:
  - превращать внутренний `StopIteration` в `RuntimeError`, если он не используется
    по назначению.
- **Вывод:** PEP подчёркивает философию «**явное лучше неявного**» и
  «ошибка должна быть очевидной».

##### 3.1.4. Зачем трогать `StopIteration`: пример скрытого бага

До PEP 479 можно было нечаянно написать генератор, который «ломал» цикл,  
и это было очень трудно отследить.

Представьте генератор:

```python
def gen():
    for x in range(5):
        if x == 3:
            raise StopIteration  # автор думал "просто закончить"
        yield x
```

Снаружи:

```python
for value in gen():
    print(value)
```

Что не так:

- `StopIteration` — это **внутренний сигнал протокола итерации**;
- выбрасывать его руками внутри генератора — всё равно что:
  - вручную дёргать «провода» внутри механизма вместо использования кнопки.

До PEP 479:

- генератор мог тихо завершиться **в неожиданный момент**;
- трудно понять, это:
  - нормальное завершение;
  - или логическая ошибка.

После PEP 479:

- такой внутренний `StopIteration` превращается в **`RuntimeError`**;
- вы сразу видите:
  - «здесь я использую `StopIteration` не по назначению»;
  - ошибку проще отладить.

**Вывод для практики:**

- внутри генераторов:
  - **не бросайте `StopIteration` руками**;
  - используйте обычное завершение функции (`return`) или условия в цикле;
- PEP защищает вас от тонких багов, делая ошибку **явной и громкой**,  
  а не «тихой и странной».

#### PEP 3132 — расширенная распаковка (`*rest` в середине)

- **Проблема до PEP:** обычная распаковка списков/кортежей была жёстко «по краям»:

```python
first, second = [1, 2]        # работает
first, rest = [1, 2, 3, 4]    # rest получит только один элемент → ValueError
```

- нельзя было удобно сказать:
  - «возьми первый элемент»;
  - «все середние в отдельный список»;
  - «последний — отдельно».

Приходилось писать:

```python
items = [1, 2, 3, 4]
first = items[0]
last = items[-1]
middle = items[1:-1]
```

PEP 3132 вводит **расширенную распаковку**:

```python
first, *middle, last = [1, 2, 3, 4]
# first -> 1
# middle -> [2, 3]
# last -> 4
```

- **Мотивация:** дать **короткий и читаемый** способ:
  - вычленять «голову» и «хвост» последовательности;
  - работать с «серединой» без ручного индексирования.

##### 3.1.6. Расширенная распаковка «на пальцах»

Типичные паттерны:

- **Игнорировать хвост:**

```python
head, *rest = items
```

- **Игнорировать голову:**

```python
*rest, last = items
```

- **Разделить на «первый», «последний» и «всё остальное»:**

```python
first, *middle, last = items
```

Польза:

- код ближе к тому, как вы **словами описываете задачу**:
  - «возьми первый элемент, все середние и последний»;
- меньше шансов ошибиться с индексами (`[1:-1]`, `[-1]` и т.п.);
- читателю проще понять вашу мысль.

**Если совсем по‑простому:**

- PEP 3132 — это «нормальный человеческий способ разложить последовательность на куски»,
  без ручной работы с индексами и с более понятным, декларативным синтаксисом.

#### PEP 604 — объединения типов через `X | Y`

- **Проблема:** типы вида `Union[int, str]` были:
  - шумными;
  - требовали импорта `typing.Union`.
- **Мотивация:** сделать аннотации **компактными и визуально очевидными**:
  - `int | str` вместо `Union[int, str]`;
  - меньше импортов, лучше читается.
- **Вывод:** язык постепенно уходит от «DSL в виде функций из `typing`» к
  **естественному синтаксису уровня языка**.

##### 3.1.5. `X | Y` вместо `Union[X, Y]`: почему это не просто «красивее»

До PEP 604:

```python
from typing import Union

def parse_id(value: str) -> Union[int, str]:
    ...
```

После:

```python
def parse_id(value: str) -> int | str:
    ...
```

Выгоды:

- **Меньше визуального шума:**
  - `int | str` читается как обычное логическое «или» в типах;
  - глаз не спотыкается о слово `Union` и лишние скобки.

- **Меньше импортов:**
  - не нужно каждый раз писать `from typing import Union`;
  - аннотации выглядят ближе к синтаксису самого языка.

**Если совсем по‑простому:**

- `Union[int, str]` — как «функция из мира `typing`»;
- `int | str` — как «встроенная операция языка»:
  - легче читать;
  - проще объяснять новичку: «функция может вернуть `int` **или** `str`».

Практически:

- в современном коде на Python 3.10+ имеет смысл **использовать `|`**,  
  а старый синтаксис `Union[...]` оставлять только:
  - в старых проектах;
  - или когда вас ограничивает совместимость с очень старыми версиями Python.

### 3.2. Типизация

Здесь самая длинная эволюционная линия.

#### PEP 484 — type hints

- **Проблема:** Python динамичен, и:
  - IDE и анализаторы плохо понимают типы;
  - документация часто расходится с реальностью;
  - большие проекты страдают от «скрытых» несоответствий типов.
- **Мотивация:** ввести **аннотации типов как опциональный слой**:
  - не ломать динамичность;
  - позволить инструментам (mypy, pyright, IDE) проверять больше ошибок до рантайма;
  - использовать аннотации как **живую документацию интерфейсов**.
- **Вывод:** PEP 484 — стартовая точка всей последующей истории типизации в Python.

##### 3.2.1. Зачем типы, если Python динамический: взгляд «с земли»

Типичный сценарий без типизации:

```python
def apply_discount(price, discount):
    return price - price * discount
```

Пока вы сами вызываете функцию правильно — всё хорошо.  
Но как только кто‑то из команды передаст:

```python
apply_discount("1000", 0.1)   # строка вместо числа
```

вы можете получить:

- странный `TypeError` далеко от места вызова;
- или неочевидное поведение, если Python попробует сделать не то, что вы ожидали.

С аннотациями:

```python
def apply_discount(price: float, discount: float) -> float:
    return price - price * discount
```

Что меняется **на практике**:

- IDE / mypy / pyright заранее скажут:
  - «сюда нельзя передавать строку»;
  - «здесь вы возвращаете не тот тип, что обещали»;
- документация становится **жёстко привязанной** к коду:
  - если вы поменяли тип, это отразится в аннотации;
  - меньше рассинхрона «в доке одно, в коде другое».

**Важно:** PEP 484 подчёркивает:

- типы — **опциональны**;
- интерпретатор их не навязывает;
- но они дают:
  - больше ошибок на этапе разработки;
  - лучшее автодополнение и навигацию в IDE;
  - более уверенную рефакторинг‑историю в больших кодовых базах.

#### PEP 585 — встроенные обобщения (`list[int]` и др.)

- **Проблема:** необходимость импортировать `List`, `Dict`, `Set` и т.п. из `typing`
  выглядела громоздко и неестественно.
- **Мотивация:** позволить использовать **встроенные коллекции как обобщённые типы**:
  - `list[int]`, `dict[str, float]`, `set[User]` и т.д.;
  - меньше магии, больше «нативного» синтаксиса.
- **Связь:** PEP 585 **частично заменяет** подходы из PEP 484, упрощая повседневное использование.

##### 3.2.2. `list[int]` вместо `List[int]`: меньше магии, больше естественности

До PEP 585:

```python
from typing import List, Dict

def load_users() -> List[Dict[str, str]]:
    ...
```

После:

```python
def load_users() -> list[dict[str, str]]:
    ...
```

Разница:

- вы **используете те же самые `list`, `dict`, `set`, что и в коде**;
- `typing.List`, `typing.Dict` превращаются в «наследие» для старых версий Python.

**Если совсем по‑простому:**

- раньше для типов коллекций существовал «второй параллельный мир» (`List`, `Dict`);
- теперь **тот же объект** (`list`, `dict`) используется и в коде, и в аннотациях:
  - легче объяснять новичкам;
  - меньше путаницы, что где импортировать.

#### PEP 563 — отложенная оценка аннотаций (`from __future__ import annotations`)

- **Проблема:** аннотации вычислялись при загрузке модуля:
  - это усложняло ссылки на ещё не объявленные типы («forward references»);
  - увеличивало накладные расходы.
- **Мотивация:** рассматривать аннотации как **строки**, которые можно разбирать позже:
  - делает код проще (нет нужды в кавычках вокруг имён классов);
  - позволяет инструментам более гибко интерпретировать типы.

##### 3.2.3. Отложенные аннотации: почему это вообще кому‑то нужно

Проблема без отложенной оценки:

```python
class Node:
    def __init__(self, next: Node | None = None):  # NameError в старых версиях
        self.next = next
```

На момент чтения аннотации `Node` ещё **не объявлен**, интерпретатор не знает этот тип.

Один из обходных путей был:

```python
class Node:
    def __init__(self, next: "Node | None" = None):
        self.next = next
```

То есть писать тип **как строку**. PEP 563 формализует эту идею:

- интерпретатор **считает все аннотации строками**;
- реальный разбор делает уже:
  - анализатор типов;
  - фреймворк;
  - ваша утилита.

Практически это:

- облегчает работу с **self‑referential типами** (типы, ссылающиеся сами на себя);
- убирает часть странностей вокруг «порядка объявления» классов.

#### PEP 593 — `Annotated`

- **Проблема:** иногда нужно добавлять **метаданные к типам**, не влияя на сам тип:
  - валидация;
  - ограничения;
  - документация для web‑фреймворков (FastAPI и др.).
- **Мотивация:** позволить писать конструкции вроде:

```python
from typing import Annotated

UserId = Annotated[int, "pk in users table"]
```

- **Вывод:** типы становятся **носителями дополнительной информации** для фреймворков,
  генераторов схем и т.п.

##### 3.2.4. `Annotated`: тип + подсказки для фреймворков

Иногда одного типа недостаточно. Например:

- `int` — это просто число;
- но вам нужно обозначить:
  - «это ID пользователя в БД»;
  - «это должно быть положительное число»;
  - «это поле приходит из заголовка HTTP».

Без `Annotated` это обычно живёт в комментариях или докстрингах.

С `Annotated`:

```python
from typing import Annotated

UserId = Annotated[int, "pk in users table"]
```

Фреймворки (FastAPI, Pydantic и др.) могут:

- читать эти метаданные;
- автоматически:
  - генерировать схемы;
  - делать валидацию;
  - строить документацию.

**Если совсем по‑простому:**

- базовый тип (`int`, `str`) — «форма контейнера»;
- `Annotated` позволяет приклеить к нему **«наклейки» с подсказками** для инструментов.

#### PEP 544 — Protocol (структурная типизация)

- **Проблема:** традиционная номинативная типизация (когда важен конкретный класс)
  плохо сочетается с «утиным» стилем Python:  
  *«если объект ведёт себя как файл, мне всё равно, как он называется»*.
- **Мотивация:** ввести **протоколы**, описывающие **набор методов/атрибутов**,
  а не конкретные классы:

```python
from typing import Protocol

class SupportsWrite(Protocol):
    def write(self, data: bytes) -> int: ...
```

- **Вывод:** статическая типизация в Python **не убивает «утиность»**, а наоборот,
  помогает её формализовать.

##### 3.2.5. Protocol: формализуем «ведёт себя как утка»

Классический питоновский подход:

```python
def write_all(f, data: bytes) -> None:
    f.write(data)  # нам всё равно, File это или Socket, лишь бы был .write
```

Нас не интересует **конкретный класс** объекта `f`.  
Важно только, что у него есть метод `write(self, bytes) -> int`.

Без протоколов типизировать это корректно сложно:

- если вы напишете тип `IO[bytes]` — вы привязываетесь к конкретному интерфейсу;
- если напишете `Any` — теряете пользу от типизации.

С `Protocol`:

```python
from typing import Protocol

class SupportsWrite(Protocol):
    def write(self, data: bytes) -> int: ...

def write_all(f: SupportsWrite, data: bytes) -> None:
    f.write(data)
```

Теперь:

- **любой объект**, у которого есть подходящий `write`, будет считаться совместимым;
- не нужно объявлять наследование от `SupportsWrite`;
- анализатор типов всё равно сможет проверить совместимость.

**Если совсем по‑простому:**

- `Protocol` — это не «класс‑родитель»;
- это **описание контракта по методам/атрибутам**:
  - «мне не важно, что ты за зверь, главное — умей вот это и вот это».

#### PEP 647, 673, 695 и др.

- **PEP 647 (TypeGuard)** — позволяет писать функции, которые **сужают тип** аргумента
  для анализатора, если возвращают `True`.
- **PEP 673 (Self)** — аккуратно решает проблему «этот метод возвращает экземпляр
  *конкретного* подкласса».
- **PEP 695 (Type statement)** — двигает язык к более аккуратному объявлению
  параметризованных типов.

Все они — **продолжение линий, начатых PEP 484**.

##### 3.2.6. «Продолжатели» PEP 484: делаем типы всё умнее, но не страшнее

Кратко, чем полезны эти PEP в живом коде:

- **PEP 647 (`TypeGuard`)**  
  Позволяет описать функции, которые **сузивают тип** аргумента:

  ```python
  from typing import TypeGuard

  def is_str_list(value: list[object]) -> TypeGuard[list[str]]:
      return all(isinstance(x, str) for x in value)
  ```

  После такого вызова анализатор понимает, что список стал `list[str]`, а не просто `list[object]`.

- **PEP 673 (`Self`)**  
  Решает проблему методов, которые возвращают **экземпляр текущего (под)класса**:

  ```python
  from typing import Self

  class Builder:
      def set_x(self, x: int) -> Self:
          self.x = x
          return self
  ```

  Это особенно полезно в паттернах вроде «builder», где методы возвращают `self` для чейнинга.

- **PEP 695 (`type` statement)**  
  Делает объявление параметризованных типов:
  - более явным;
  - менее завязанным на «фокусах» с `TypeVar`.

Все они вместе:

- не ломают базовую идею PEP 484;
- **надстраивают удобства** поверх уже привычной системы типов;
- позволяют инструментам давать **более точные подсказки**, не усложняя ваш повседневный код.

### 3.3. Асинхронность и конкурентность

#### PEP 3156 — `asyncio`

- **Проблема:** экосистема была разрозненной:
  - разные event loop'ы и библиотеки взаимно несовместимы;
  - нет стандартного способа писать async‑код.
- **Мотивация:** создать **единый стандартный event loop и набор абстракций**:
  - Tasks, Futures, Transport/Protocol;
  - базовый каркас для async‑фреймворков.

##### 3.3.1. `asyncio`: зачем вообще нужен единый event loop

До `asyncio` ситуация выглядела так:

- один фреймворк использует свой цикл событий (`twisted`);
- другой — патчинг стандартной библиотеки (`gevent`);
- третий — ещё что‑то своё.

В результате:

- библиотеки **плохо дружат друг с другом**;
- вы не можете легко:
  - взять HTTP‑клиент из одной экосистемы;
  - и использовать его внутри другой;
- каждый фреймворк по‑своему решает:
  - как создавать задачи;
  - как управлять временем;
  - как обрабатывать сеть.

`asyncio` вводит:

- **единый event loop** в стандартной библиотеке;
- базовые понятия:
  - `Task` — «запущенная корутина»;
  - `Future` — «обещание результата в будущем»;
  - примитивы для работы с сетью и временем.

**Если совсем по‑простому:**

- раньше каждый строил свою площадь и свой светофор;
- теперь язык даёт **общую дорогу и стандартизированный светофор**:
  - разные библиотеки могут ездить по одним и тем же правилам.

#### PEP 492 — `async`/`await`

- **Проблема:** использование генераторов для асинхронности выглядело как «магия»:
  - сложно читать;
  - трудно объяснить новичкам;
  - код выглядел непривычно.
- **Мотивация:** ввести **явный синтаксис** для корутин:

```python
async def fetch(url: str) -> bytes:
    async with session.get(url) as resp:
        return await resp.read()
```

- **Вывод:** Python делает ставку на **читаемость асинхронного кода**, даже ценой
  введения новых ключевых слов.

##### 3.3.2. `async/await`: разваливаем магию на понятные кусочки

До появления `async`/`await` асинхронный код часто писали на основе:

- генераторов;
- декораторов, которые превращали генераторы в «почти корутины».

Код выглядел **подозрительно магическим**:

```python
@asyncio.coroutine
def fetch(url):
    resp = yield from session.get(url)
    body = yield from resp.read()
    return body
```

С `async`/`await` тот же код:

```python
async def fetch(url):
    resp = await session.get(url)
    body = await resp.read()
    return body
```

Плюсы:

- по сигнатуре функции (`async def`) **сразу видно**, что это корутина;
- ключевое слово `await` показывает места, где:
  - корутина может «отдать управление» циклу событий;
  - происходят потенциально долгие операции (сеть, диск и т.п.).

**Практическое правило:**

- всё, что делает I/O (запросы, БД, файлы, sleep):
  - в async‑мире должно быть помечено `await`;
  - так вы сами себе помечаете «точки ожидания».

#### PEP 684 и 703 — путь к более «честному» multithreading

- **PEP 684 (Per‑Interpreter GIL)** и **PEP 703 (Free‑Threaded)**:
  - документируют эволюцию **ограничений GIL**;
  - объясняют, какие компромиссы ради многопоточности Python готов принять,
    а какие — нет.
- **Практический вывод:**  
  читая эти PEP, вы понимаете, чего **реально ждать от будущих версий Python**
  в плане параллелизма, а не опираетесь на слухи и заголовки статей.

##### 3.3.3. GIL, per‑interpreter GIL и free‑threaded: что это меняет для вас

GIL (Global Interpreter Lock):

- упрощает внутреннюю реализацию Python;
- но ограничивает **настоящий параллелизм потоков** для CPU‑задач.

Что предлагают PEP 684 и 703 в общих чертах:

- **PEP 684** — двигаться к модели, где:
  - каждый интерпретатор имеет **свой GIL**;
  - можно запускать несколько интерпретаторов в одном процессе;
  - потоки разных интерпретаторов меньше мешают друг другу.
- **PEP 703** — шаг к полностью **без‑GIL режиму** для Python:
  - намного больше параллелизма;
  - но и больше требований к библиотекам и расширениям.

Для вас это значит:

- не ждать «магического ускорения всего кода»;
- понимать, что:
  - CPU‑heavy задачи будут получать больше выгоды;
  - экосистема должна адаптироваться (расширения, C‑библиотеки, GIL‑чувствительный код).

Читая эти PEP, вы видите **реальный план** развития многопоточности, а не маркетинговые лозунги.

### 3.4. ООП и модель объектов

- **PEP 3135** — `super()` без аргументов:
  - упрощает правильное использование `super()` в иерархиях множественного наследования;
  - мотивация: меньше boilerplate, меньше ошибок с порядком аргументов.
- **PEP 422/487** — новые хуки создания классов:
  - дают способ влиять на создание подклассов **без тяжёлых метаклассов**;
  - мотивация: сделать «магические» API проще и прозрачнее.
- **PEP 562** — `__getattr__` на уровне модуля:
  - позволяет делать **ленивые импорты и совместимость имён** в модулях;
  - мотивация: гибкость API модулей при сохранении обратной совместимости.

#### 3.4.1. `super()` без аргументов: меньше шансов выстрелить себе в ногу

До PEP 3135:

```python
class Base:
    def __init__(self, x):
        self.x = x

class Child(Base):
    def __init__(self, x, y):
        Base.__init__(self, x)   # надо не забыть правильный класс
        self.y = y
```

Проблемы:

- нужно явно писать имя базового класса;
- при сложных иерархиях (множественное наследование) легко запутаться в MRO.

С `super()` без аргументов:

```python
class Child(Base):
    def __init__(self, x, y):
        super().__init__(x)
        self.y = y
```

Теперь:

- вам не нужно помнить, как именно вызвать конструктор родителя;
- `super()` сам учитывает MRO и вызывает правильный метод.

**Если совсем по‑простому:**

- `super()` без аргументов = «позвать родителя правильно, не вспоминая его имя и сигнатуру».

---

#### 3.4.2. PEP 422/487: хуки создания классов без тяжёлых метаклассов

Иногда вам нужно:

- каждый раз при создании подкласса:
  - регистрировать его где‑то;
  - проверять наличие нужных атрибутов;
  - автоматически добавлять поведение.

Раньше для этого часто использовали метаклассы, которые:

- сложны для понимания;
- тяжело читаются и отлаживаются.

PEP 422/487 добавляют более простые хуки вроде `__init_subclass__`:

```python
class PluginBase:
    registry: list[type["PluginBase"]] = []

    def __init_subclass__(cls, **kwargs):
        super().__init_subclass__(**kwargs)
        PluginBase.registry.append(cls)
```

Каждый новый подкласс `PluginBase` автоматически попадает в `registry`.  
Без явной регистрации, без метаклассов.

**Практически:**

- вы можете строить «плагино‑подобные» системы;
- при этом код остаётся значительно проще, чем с кастомными метаклассами.

---

#### 3.4.3. PEP 562: `__getattr__` на уровне модуля

Обычно `__getattr__` есть у **классов/объектов**.  
PEP 562 позволяет задать `__getattr__` и для **модуля**:

- можно:
  - лениво импортировать части модуля;
  - поддерживать старые имена, перенаправляя их на новые;
  - аккуратно делать deprecation.

Проще говоря:

- вы получаете дополнительный уровень гибкости для **публичного API модуля**;
- при этом не обязаны сразу ломать старые точки входа.

### 3.5. Исключения и контроль потока

- **PEP 3134** — exception chaining (`raise ... from ...`):
  - мотивация: сохранить **контекст исходной ошибки**;
  - чтобы в traceback было видно, «что сломалось изначально».
- **PEP 654** — `ExceptionGroup` и `except*`:
  - мотивация: асинхронный и параллельный код может выбросить **несколько ошибок одновременно**;
  - нужен механизм сообщать о них разом.
- **PEP 678** — `BaseException.add_note()`:
  - мотивация: добавлять **человеко‑понятный контекст** к исключениям;
  - пример: URL запроса, ID пользователя, конфигурацию.

#### 3.5.1. PEP 3134: `raise ... from ...` — ниточка от обёртки к исходной ошибке

Типичный код без chaining:

```python
try:
    user = load_user(user_id)
except DatabaseError:
    raise ServiceError("failed to load user")
```

В traceback вы видите только `ServiceError`,  
а первопричина (`DatabaseError`) теряется.

С `raise ... from ...`:

```python
try:
    user = load_user(user_id)
except DatabaseError as e:
    raise ServiceError("failed to load user") from e
```

Теперь:

- traceback показывает **обе ошибки**;
- явно видно:
  - какая ошибка была «внизу»;
  - какая — «обёртка» на более высоком уровне абстракции.

**Практически:**

- так проще отлаживать и логировать;
- вы не теряете важный технический контекст из‑за перехвата.

---

#### 3.5.2. PEP 654: несколько ошибок одновременно (`ExceptionGroup`, `except*`)

Для асинхронного/параллельного кода типична ситуация:

- вы запускаете **несколько задач**;
- сразу несколько из них падают с ошибками.

Раньше:

- приходилось выбирать, **какую одну ошибку показать**;
- или придумывать свои структуры для «набора ошибок».

`ExceptionGroup` позволяет:

- собрать **несколько исключений** в один объект;
- обработать их через `except*`.

Это особенно полезно:

- в библиотеках, работающих с `asyncio.TaskGroup`;
- в системах, где параллельно выполняются десятки независимых операций.

---

#### 3.5.3. PEP 678: `add_note` — человеческое объяснение к стеку

Часто в ошибке хочется дописать:

- какой URL вы вызывали;
- какой ID пользователя;
- какую конфигурацию использовали.

PEP 678 добавляет метод `add_note`:

```python
try:
    resp = client.get(url)
    resp.raise_for_status()
except Exception as e:
    e.add_note(f"While fetching {url}")
    raise
```

В traceback вы увидите дополнительную строку‑подсказку.  
Это:

- делает логи **более говорящими**;
- экономит время при разборе инцидентов.

### 3.6. Импорт, упаковка и окружения

- **PEP 420** — namespace packages:
  - мотивация: позволить нескольким дистрибутивам **делить одно и то же пространство имён**;
  - удобно для крупных компаний и экосистем плагинов.
- **PEP 517/518** — `pyproject.toml` для сборки:
  - мотивация: уйти от жёсткой привязки к `setuptools` и `setup.py`;
  - описать сборку в декларативной форме.
- **PEP 561** — `py.typed`:
  - мотивация: сообщать анализаторам, что пакет **поставляется с аннотациями**,
    которым можно доверять.
- **PEP 621** — метаданные проекта в `pyproject.toml`:
  - мотивация: стандартизировать хранение имени, версии, авторов и т.п.
- **PEP 668** — externally managed environments:
  - мотивация: защитить системный Python от **случайной порчи через `pip install`**;
  - заставить пользователей использовать виртуальные окружения.

#### 3.6.1. PEP 420: namespace packages — один пакет, много дистрибутивов

Представьте большую компанию:

- есть много команд;
- каждая делает подпакет:
  - `company.auth`;
  - `company.billing`;
  - `company.analytics`.

Раньше:

- всё это приходилось складывать в один физический пакет;
- или придумывать сложные схемы импорта.

Namespace packages позволяют:

- разным дистрибутивам «делить» одно имя пакета `company`;
- каждая команда может распространять **свой кусок** пространства имён.

---

#### 3.6.2. PEP 517/518 и 621: `pyproject.toml` как единая точка правды

До этих PEP:

- сборка и метаданные проекта были размазаны:
  - `setup.py`;
  - `setup.cfg`;
  - `MANIFEST.in`;
  - и ещё куча файлов.

`pyproject.toml`:

- становится **единым местом**, где описано:
  - как собирать проект (build backend);
  - какие зависимости нужны для сборки;
  - основные метаданные (имя, версия, автор и т.д.).

Плюсы:

- меньше «магии» в виде исполняемого `setup.py`;
- проще для инструментов (pip, build, poetry и т.д.) анализировать проект.

---

#### 3.6.3. PEP 561: `py.typed` — сигнал о типизированном пакете

Анализаторам типов нужно знать:

- можно ли доверять аннотациям внутри установленного пакета;
- или нужно искать `.pyi`‑стабы отдельно.

Файл `py.typed` в пакете:

- говорит: «здесь типы **часть публичного контракта**, можно на них опираться».

Практически:

- ставите пакет;
- анализатор видит `py.typed` и начинает использовать его аннотации в проверках.

---

#### 3.6.4. PEP 668: защитим системный Python от случайных `pip install`

Проблема:

- многие пользователи ставят пакеты прямо в системный Python;
- это может ломать:
  - системные утилиты;
  - пакетный менеджер ОС.

PEP 668 вводит идею «externally managed» окружений:

- системный Python помечен как управляемый системой;
- `pip`:
  - предупреждает или запрещает установку туда;
  - мягко подталкивает к использованию виртуальных окружений (`venv`, `conda` и т.п.).

**Для вас:**

- меньше шансов случайно «убить» системный Python;
- более аккуратное разделение:
  - «Python для системы»;
  - «Python для ваших проектов».

### 3.7. Производительность и интерпретатор

- **PEP 659** — специализированный адаптивный интерпретатор:
  - мотивация: ускорить Python **без изменения кода пользователя**;
  - интерпретатор подстраивается под часто выполняющиеся пути.
- **PEP 709** — инлайнинг comprehensions:
  - мотивация: уменьшить накладные расходы генерации временных объектов;
  - оптимизация, «прозрачная» для пользователя.

Главная мысль этого раздела:

> **почти любое «подозрительно умное» поведение Python можно объяснить через PEP, который это поведение вводил.**

#### 3.7.1. PEP 659 и 709: «ускоряем под капотом, не ломая внешний вид»

Оба этих PEP важны тем, что:

- **не требуют от вас менять код**;
- нацелены на:
  - анализ часто выполняемых участков;
  - оптимизацию генераторов списков, циклов и т.п.

PEP 659:

- интерпретатор запоминает, какие опкоды и ветки выполняются чаще всего;
- подстраивает исполнение под эти «горячие пути»;
- результат — более быстрый код на тех же исходниках.

PEP 709:

- убирает лишние временные объекты при comprehensions;
- делает их выполнение ближе к тому, как если бы вы писали руками оптимизированный цикл.

##### 3.7.2. Примеры для интуиции: что именно оптимизируется

Рассмотрим два фрагмента кода:

```python
data = [process(x) for x in items]
total = sum(process(x) for x in items)
```

Раньше:

- каждое такое comprehension приводило к созданию:
  - дополнительного генератора;
  - иногда временного списка;
- интерпретатор проходил через несколько уровней абстракций.

После оптимизаций из PEP 709:

- многие из этих промежуточных шагов **«сплющиваются» в более прямой код**;
- меньше накладных расходов:
  - меньше объектов;
  - меньше переходов между фреймами.

Вам не нужно менять стиль:

- вы **как писали comprehensions, так и пишете**;
- интерпретатор сам делает их выполнение ближе к рукописному циклу:

```python
data = []
for x in items:
    data.append(process(x))
```

То же самое с PEP 659:

- интерпретатор, «научившись» на типичных путях выполнения,  
  перестаёт каждый раз всё интерпретировать одинаково медленно;
- горячие места получают **более специализированный код**,  
  но ваш исходник при этом не меняется.

**Вывод:**

- современные версии Python становятся быстрее не за счёт того,  
  что вы пишете «особенный» код,
- а за счёт умных оптимизаций **внутри интерпретатора**, описанных и обоснованных в PEP.

##### Контрольные вопросы к разделу 3.7

<details>
<summary>Показать вопросы и ответы</summary>

1. **Вопрос:** В чём общий замысел PEP 659 и 709 с точки зрения разработчика прикладного кода?  
   **Ответ:** Оба PEP стремятся **ускорить выполнение существующего Python‑кода**,  
   не требуя переписывать его. Они оптимизируют интерпретатор (горячие участки, comprehensions),  
   чтобы ваш «обычный» код работал быстрее «из коробки».

2. **Вопрос:** Почему PEP 709 особенно важен для comprehensions?  
   **Ответ:** Потому что раньше comprehensions создавали дополнительные объекты  
   (генераторы, временные списки), что давало накладные расходы.  
   PEP 709 позволяет интерпретатору «склеивать» эти шаги в более прямой код,  
   приближая выполнение к вручную написанному циклу без лишних временных структур.

3. **Вопрос:** Какой вывод можно сделать для стиля написания кода после этих PEP?  
   **Ответ:** Вам не нужно избегать comprehensions или каких‑то конструкций из‑за страха  
   «они медленные по определению». Напротив, можно писать **читаемый, идиоматичный код**,  
   полагаясь на то, что интерпретатор станет всё лучше оптимизировать типичные паттерны.

</details>
