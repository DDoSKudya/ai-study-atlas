## 6.1 Основы Unicode

### 6.1.1 Unicode code points и представление в памяти

#### Что такое Unicode и зачем он нужен

**Unicode** — это международный стандарт кодирования текста, разрабатываемый консорциумом Unicode Consortium (unicode.org). Его цель: **один универсальный набор символов для всех письменностей мира** — латиницы, кириллицы, иероглифов, арабской вязи, деванагари, эмодзи и т.д. Стандарт включает не только символы, но и правила нормализации, свойства символов (категория, направление письма, регистр) и алгоритмы (бидирекциональный текст, перенос строк).

**Проблема до Unicode:** каждая кодировка (ASCII, KOI8-R, CP1251, ISO-8859-*, EBCDIC) покрывала лишь ограниченный набор символов. Байт `0xC0` в CP1251 — буква «А», в KOI8-R — «ю», в Latin-1 — «À». При обмене файлами между Windows и Unix, при отправке email или при чтении файла, созданного на другой системе, одни и те же байты интерпретировались по-разному — возникали «кракозябры», потеря данных или падение программ. Не было способа однозначно передать текст независимо от платформы.

**Решение Unicode:** присвоить каждому символу уникальное число — **code point** (кодовая позиция). Это абстрактный идентификатор: он не определяет, как символ будет представлен в байтах (это задача **кодировки** — UTF-8, UTF-16 и т.д.), а лишь однозначно идентифицирует символ в универсальном реестре. Одна и та же строка в виде последовательности code points будет одинаково интерпретироваться в любой системе; различие только в том, как эта последовательность кодируется в байты при записи в файл или передаче по сети.

**Теоретическое обоснование:** Разделение на (1) абстрактные символы (code points) и (2) их представление в байтах (encoding) — это **двухуровневая модель**. Уровень 1: «что за символ» — однозначно и независимо от платформы. Уровень 2: «как он хранится/передаётся» — зависит от выбранной кодировки. Такое разделение позволяет: (а) обмениваться текстом между системами без потерь; (б) выбирать кодировку под задачу (компактность UTF-8 vs фиксированность UTF-32); (в) добавлять новые символы в Unicode без изменения существующих кодировок — достаточно расширить диапазон code points.

#### Code point: определение и диапазон

**Code point** — это целое число в диапазоне **0–1 114 111** (шестнадцатерично: 0x0–0x10FFFF). Запись в формате **U+HHHH** или **U+HHHHHH** (шестнадцатерично, с ведущими нулями): например, **U+0041** — латинская буква A, **U+041F** — кириллическая П, **U+1F600** — эмодзи grinning face (😀).

**Почему именно 0x10FFFF?** Диапазон Unicode ограничен 21 битом (2^21 ≈ 1.1 млн кодовых позиций). Исторические причины:

1. **UTF-16 и surrogate pairs:** В UTF-16 символы BMP (U+0000–U+FFFF) кодируются одним 16-битным значением. Символы вне BMP (U+10000–U+10FFFF) кодируются парой 16-битных значений — surrogate pair — из диапазона U+D800–U+DFFF. Этот диапазон (surrogate code points) **зарезервирован** в Unicode и не назначается отдельным символам. Максимальный кодируемый code point в UTF-16: 0x10FFFF.

2. **Историческое соглашение:** Ранние версии Unicode использовали 16 бит (до U+FFFF); позже диапазон расширен до 21 бита. 0x10FFFF — максимальное значение, которое можно представить surrogate pair в UTF-16.

Фактически присвоенные символы занимают лишь часть диапазона (на 2024 год — порядка 150 тысяч); остальное зарезервировано для будущего расширения (новые иероглифы, эмодзи, исторические письменности и т.д.). Диапазон U+D800–U+DFFF (surrogate code points) **никогда не будет** присвоен отдельным символам — он зарезервирован исключительно для surrogate pairs в UTF-16.

**Примеры code points:**

| Символ | Code point (дес.) | Code point (hex) |
|--------|-------------------|------------------|
| A      | 65                | U+0041           |
| П      | 1055              | U+041F           |
| €      | 8364              | U+20AC           |
| 中     | 20013             | U+4E2D           |
| 😀     | 128512            | U+1F600          |
| NULL   | 0                 | U+0000           |

#### Плоскости Unicode (Planes)

Unicode организован в **17 плоскостей** по 65 536 code points каждая (0x10000). Первая плоскость — **BMP (Basic Multilingual Plane)**, U+0000–U+FFFF — содержит большинство современных письменностей, пунктуацию, символы валют, математические операторы и т.д. Остальные плоскости — **supplementary planes** (U+10000–U+10FFFF):

| Плоскость | Диапазон | Содержимое |
|-----------|----------|------------|
| **Plane 0 (BMP)** | U+0000–U+FFFF | Латиница, кириллица, греческий, иероглифы, арабский, иврит, эмодзи (часть) и др. |
| **Plane 1 (SMP)** | U+10000–U+1FFFF | Древние письменности (Linear B, готика), музыкальные символы, математические буквы, эмодзи |
| **Plane 2 (SIP)** | U+20000–U+2FFFF | Редкие иероглифы (CJK Extension B) |
| **Planes 3–13** | U+30000–U+DFFFF | Зарезервированы |
| **Plane 14 (SSP)** | U+E0000–U+EFFFF | Специальные (теги, вариационные селекторы) |
| **Planes 15–16** | U+F0000–U+10FFFF | Частное использование (PUA) |

**Практический вывод:** символы вне BMP (например, 😀 U+1F600) требуют в UTF-8 четырёх байт и в UTF-16 — surrogate pair. При итерации по строке `for c in s:` каждый элемент — один code point; для «😀» это одна итерация (один code point), хотя визуально и в байтах это «больше».

#### Как Python хранит строки внутри (PEP 393 — Flexible String Representation)

В **Python 3** тип **`str`** — это **последовательность Unicode code points**. Внутри интерпретатор CPython (Python 3.3+) может хранить строку в одном из нескольких представлений (PEP 393):

| Представление | Условие | Байт на символ | Пример строк |
|---------------|---------|----------------|--------------|
| **ASCII-only** | Все символы в 0–127 | 1 | `"Hello"`, `"ABC123"` |
| **Latin-1 (UCS-1)** | Все символы в 0–255 | 1 | `"café"`, `"naïve"` |
| **UCS-2** | Все символы в BMP (0–65535) | 2 | `"Привет"`, `"日本語"` |
| **UCS-4** | Есть символы вне BMP | 4 | `"😀"`, `"𝕳𝖊𝖑𝖑𝖔"` |

**Механика выбора:** интерпретатор при создании строки сканирует все символы и выбирает **минимально необходимое** представление. Например, строка `"Hello"` — все символы в ASCII → 1 байт на символ. Добавляем `"é"` (U+00E9) → переход в Latin-1, по-прежнему 1 байт. Добавляем `"П"` (U+041F) → переход в UCS-2, 2 байта. Добавляем `"😀"` (U+1F600) → переход в UCS-4, 4 байта. Строка при конкатенации может «расширяться» до более широкого представления; обратного перехода (из UCS-4 в UCS-2 при удалении символов) не происходит — представление остаётся максимальным для этой строки.

**Пользователю это не нужно знать в деталях:** строки всегда ведут себя как последовательность «символов» (code points). При итерации `for c in s:` вы получаете строки длиной 1. При индексации `s[0]` — тоже строка длиной 1. API не раскрывает внутреннее представление. Проверить представление можно через `sys.getsizeof(s)` (размер в байтах) или внутренние атрибуты (не документировано, зависит от версии CPython).

**Пример оценки памяти (упрощённо):** строка `"Hello"` (5 ASCII) — примерно 54 байта накладных расходов + 5×1 = 59 байт. Строка `"Привет"` (6 символов BMP) — накладные + 6×2 = 66+ байт. Строка `"😀"` (1 символ вне BMP) — накладные + 4 = 58+ байт. Точные цифры зависят от версии CPython и оптимизаций.

**Почему это важно:** экономия памяти. Строка `"Hello"` хранится компактнее, чем строка `"Привет"` или `"😀"`. Для приложений, работающих в основном с ASCII (логи, JSON, имена, ключи), это даёт существенную экономию — до 4 раз по сравнению с фиксированным UCS-4 для чисто ASCII-текста.

#### Терминология (подробно)

| Термин | Определение |
|--------|-------------|
| **Code point** | Целое число 0–1 114 111, однозначно идентифицирующее символ в Unicode |
| **Character** (символ) | В контексте str — один code point (может быть буква, цифра, иероглиф, эмодзи, пробел и т.д.) |
| **Grapheme cluster** | Видимая единица текста для пользователя; может состоять из нескольких code points (буква + combining accent, эмодзи с модификатором кожи и т.д.) |
| **Encoding** (кодировка) | Правило преобразования code points ↔ последовательность байт (UTF-8, UTF-16, Latin-1 и т.д.) |
| **Decoding** (декодирование) | Преобразование байт → строка (code points); метод `bytes.decode()` |
| **Encoding** (в смысле encode) | Преобразование строки (code points) → байты; метод `str.encode()` |
| **BMP** (Basic Multilingual Plane) | Диапазон U+0000–U+FFFF; большинство современных письменностей |
| **Surrogate pair** | Пара 16-битных значений в UTF-16 для кодов U+10000–U+10FFFF |
| **Byte order (endianness)** | Порядок байт: little-endian (младший первым), big-endian (старший первым) |
| **Code unit** | Минимальная единица: UTF-8 — 8 бит, UTF-16 — 16 бит |

**Важно различать:** один «визуальный символ» на экране может состоять из **нескольких code points**. Например, буква **é** может быть:

1. **Одним code point** — U+00E9 (LATIN SMALL LETTER E WITH ACUTE) — **precomposed** форма
2. **Двумя code points** — U+0065 (LATIN SMALL LETTER E) + U+0301 (COMBINING ACUTE ACCENT) — **decomposed** форма (базовая буква + combining character)

В первом случае `len("é")` равен 1, во втором — 2. Оба варианта выглядят одинаково на экране (эквивалентны с точки зрения Unicode Normalization). При сравнении строк `"é" == "e\u0301"` может быть `False` (разные представления); для канонического сравнения используется нормализация (unicodedata.normalize). Ещё примеры: эмодзи с модификатором кожи (👋 + Fitzpatrick modifier → несколько code points); буквы с несколькими диакритиками (e + acute + macron). Расширенная работа с grapheme clusters и нормализацией (NFC/NFD/NFKC/NFKD) — в §15d (unicodedata).

#### Unicode Normalization (NFC, NFD, NFKC, NFKD) — краткий обзор

**Проблема:** один и тот же «визуальный» символ может быть представлен разными последовательностями code points (precomposed vs decomposed). Сравнение строк напрямую даёт `False`, хотя на экране текст выглядит одинаково.

**Решение:** нормализация — приведение строк к канонической форме.

| Форма | Название | Описание | Пример для é |
|-------|----------|----------|--------------|
| **NFC** | Canonical Composition | Максимально «собранные» символы | U+00E9 (1 code point) |
| **NFD** | Canonical Decomposition | Разложены на базовую букву + combining marks | U+0065 + U+0301 (2 code points) |
| **NFKC** | Compatibility Composition | Композиция + compatibility mappings | Может объединить лигатуры и т.п. |
| **NFKD** | Compatibility Decomposition | Максимальное разложение | Разбивает лигатуры (ﬁ → f + i) |

**В Python (модуль unicodedata):**

```python
import unicodedata

# é как precomposed (NFC) и decomposed (NFD)
e_precomposed = "é"           # U+00E9
e_decomposed = "e\u0301"      # U+0065 + U+0301 (e + acute)
e_precomposed == e_decomposed  # False — разные представления!

unicodedata.normalize("NFC", e_decomposed)  # 'é' (precomposed)
unicodedata.normalize("NFD", e_precomposed) # 'e\u0301' (decomposed)
unicodedata.normalize("NFC", e_decomposed) == e_precomposed  # True
```

**Разница NFKC/NFKD:** Compatibility формы объединяют символы, которые выглядят одинаково, но имеют разные свойства (ширина, лигатуры). Пример: `"ﬁ"` (U+FB01, LATIN SMALL LIGATURE FI) в NFKD становится `"fi"` (два символа). При поиске «fi» в тексте с лигатурой без нормализации поиск не найдёт; с NFKD — найдёт.

**Когда использовать:** при сравнении пользовательского ввода, поиске, сортировке — приведение к одной форме устраняет «призрачные» различия. По умолчанию для обмена данными и хранения чаще используют **NFC**. NFKD полезен для полнотекстового поиска (приведение лигатур и вариантов написания к базовой форме).

#### Grapheme clusters — подробнее: когда len(s) ≠ числу «видимых» символов

**Grapheme cluster** — минимальная единица текста, которую пользователь воспринимает как один символ. Может состоять из нескольких code points.

**Примеры:**

| Строка | len(s) | Визуально | Пояснение |
|--------|--------|-----------|-----------|
| `"a"` | 1 | a | Один code point |
| `"é"` (U+00E9) | 1 | é | Precomposed — один code point |
| `"e\u0301"` | 2 | é | Decomposed — e + combining acute |
| `"👋"` | 1 | 👋 | Один code point (wave emoji) |
| `"👋\u200d"` | 2 | 👋 (с ZWJ) | Zero-width joiner может менять отображение |
| `"👋🏻"` | 2 | 👋 (светлая кожа) | Base emoji + Fitzpatrick modifier (U+1F3FB) |
| `"café"` | 4 | café | 4 code points; é может быть 1 или 2 |
| `"ﬁ"` (лигатура) | 1 | ﬁ | Один code point U+FB01; NFKD → f + i |

**Zero-Width Joiner (U+200D):** невидимый символ, «склеивающий» несколько эмодзи в один визуальный кластер. Например, семья: 👨 (мужчина) + ZWJ + 👩 (женщина) + ZWJ + 👧 (девочка) = 👨‍👩‍👧 (одна «картинка» семьи). `len("👨‍👩‍👧")` может быть 7 (семь code points), но визуально — один символ.

**Практический вывод:** не полагайтесь на `len(s)` для подсчёта «видимых» символов или позиции курсора. Для разбиения на grapheme clusters нужны библиотеки (например, `unicodedata` + ручной разбор или `regex` с границами grapheme). Простой случай: для многих европейских языков NFC-нормализация сокращает число code points и приближает len к «видимому» количеству символов.

#### Примеры: str как последовательность code points

```python
# str — последовательность code points
s = "ABC"
len(s)           # 3 — три code point
s[0]             # 'A' (строка длиной 1, тип str)
type(s[0])       # <class 'str'>
ord(s[0])        # 65 (code point латинской A)

# Кириллица
s = "Привет"
len(s)           # 6 (шесть code points — шесть букв)
ord(s[0])        # 1055 (U+041F — буква П)
ord(s[1])        # 1088 (U+0440 — буква р)

# Эмодзи — один code point (в BMP+ или в отдельной плоскости)
s = "😀"
len(s)           # 1
ord(s[0])        # 128512 (U+1F600)

# Комбинированные символы (е + combining accent)
s = "e\u0301"    # é как два code points
len(s)           # 2
ord(s[0])        # 101 (e)
ord(s[1])        # 769 (combining acute accent)

# Пустая строка
s = ""
len(s)           # 0
s[0]             # IndexError: string index out of range
```

#### Пошагово: что происходит при итерации по строке

При выполнении `for c in s:` интерпретатор:

1. Вызывает `iter(s)` — возвращает итератор строки
2. На каждой итерации вызывает `next(iterator)` — возвращает **следующий элемент** последовательности
3. Для `str` каждый элемент — это **строка длиной 1** (один code point)
4. Не происходит декодирования или кодирования — строка уже хранит code points; итерация просто отдаёт их по одному

Таким образом, `for c in "Привет":` даёт `c = 'П'`, затем `'р'`, `'и'`, `'в'`, `'е'`, `'т'` — шесть итераций. Каждый `c` — это объект типа `str` длиной 1.

#### Граничные случаи и тонкости

- **Пустая строка:** `""` — последовательность нулевой длины; `len("") == 0`; итерация не выполняется ни разу
- **Строка из одного символа:** `s = "A"`; `s[0] == s[-1] == "A"`
- **Строка с null:** `"\x00"` — допустима; null (U+0000) является валидным Unicode-символом; в C-строках он означает конец строки, в Python — обычный символ
- **Суррогатные коды (U+D800–U+DFFF):** зарезервированы для UTF-16 surrogate pairs; в Python они допустимы в str как отдельные «символы», но при encode в UTF-8 могут вызвать ошибку (в зависимости от errors)

#### История: Python 2 vs Python 3 и модель строк (PEP 393, PEP 358, PEP 3100)

В **Python 2** тип **`str`** хранил байты (в системной кодировке или в указанной); тип **`unicode`** — последовательность Unicode code points. Путаница возникала из-за неявного смешения: один и тот же литерал `'hello'` мог быть и «текстом», и «байтами»; склейка `str + unicode` вызывала неявный encode/decode и могла падать с `UnicodeDecodeError`.

**PEP 3100** — Unicode literals; **PEP 358** — bytes type; **PEP 393** — flexible string representation. В Python 3 **`str`** всегда Unicode; **`bytes`** — всегда байты; **`unicode`** удалён. Операции между str и bytes вызывают **TypeError** — программист вынужден явно вызвать encode/decode и указать кодировку.

**Сводная таблица Py2 vs Py3 для строк:**

| Аспект | Python 2.7 | Python 3 |
|--------|------------|----------|
| `'hello'` | str (байты) | str (Unicode) |
| `u'hello'` | unicode | str (u-префикс опционален) |
| `b'hello'` | str (синоним) | bytes |
| `str + unicode` | неявный encode (могла падать) | TypeError |
| `str.encode()` | кодирует в байты | str → bytes |
| `str.decode()` | декодирует в unicode | нет у str (только у bytes) |
| `len(u'Привет')` | 6 | 6 |
| `type(open().read())` | str (байты) | str (Unicode) |

#### Пошагово: декодирование UTF-8 — байты b'\xd0\x9f' → символ 'П'

При выполнении `b'\xd0\x9f'.decode('utf-8')`:

1. **Первый байт 0xD0:** старшие биты `110` → двухбайтовая последовательность
2. **Второй байт 0x9F:** должен быть continuation byte (`10xxxxxx`) — проверка пройдена
3. **Извлечение бит:** из 0xD0 берём 5 бит (xxxxx), из 0x9F — 6 бит (xxxxxx); собираем 11 бит: 10000011111 = 1055
4. **Code point:** 1055 = U+041F — буква 'П'
5. **Результат:** строка `"П"`

### 6.1.2 UTF-8, UTF-16, UTF-32, Latin-1

#### Зачем нужны кодировки: проблема представления

**Code points** существуют «в уме» — абстрактные числа (0–1 114 111). При записи в файл, передаче по сети, хранении в памяти как «сырые байты» или взаимодействии с внешними системами нужен **способ превратить code points в последовательность байт**. Этот способ и есть **кодировка (encoding)**. Обратный процесс — **декодирование (decoding)** — превращает байты обратно в code points.

**Почему одна кодировка не подходит для всего?** Разные кодировки делают разные компромиссы:

- **Размер:** UTF-8 компактен для ASCII и западных языков; UTF-32 фиксирует 4 байта на символ — предсказуемо, но расточительно
- **Совместимость:** UTF-8 совместим с ASCII (первые 128 символов совпадают); Latin-1 совместим с первыми 256
- **Сложность:** Latin-1 — простое 1:1 соответствие байт↔code point; UTF-8 — переменная длина, более сложные правила
- **Порядок байт:** UTF-16 и UTF-32 требуют выбора byte order (big/little endian); UTF-8 — последовательность байт однозначна

#### UTF-8 (Unicode Transformation Format, 8-bit) — подробно

**UTF-8** — кодировка переменной длины: 1–4 байта на символ. Разработана Кеном Томпсоном и Робом Пайком (создатели Unix и Go) для совместимости с ASCII и эффективности для западных языков.

**Диапазоны code points и длина в UTF-8:**

| Диапазон code points | Байт в UTF-8 | Примеры |
|----------------------|--------------|---------|
| U+0000–U+007F (0–127) | 1 байт | ASCII: A, 0, пробел |
| U+0080–U+07FF (128–2047) | 2 байта | Кириллица, греческий, латиница с диакритиками |
| U+0800–U+FFFF (2048–65535) | 3 байта | Иероглифы, большинство символов BMP |
| U+10000–U+10FFFF (65536–1114111) | 4 байта | Эмодзи, редкие символы вне BMP |

**Структура UTF-8 по битам (RFC 3629):**

- **1 байт:** `0xxxxxxx` — 7 бит данных; code point 0–127 (ASCII)
- **2 байта:** `110xxxxx 10xxxxxx` — 11 бит данных; code point 128–2047
- **3 байта:** `1110xxxx 10xxxxxx 10xxxxxx` — 16 бит данных; code point 2048–65535
- **4 байта:** `11110xxx 10xxxxxx 10xxxxxx 10xxxxxx` — 21 бит данных; code point 65536–1114111

Первый байт по старшим битам (`0`, `110`, `1110`, `11110`) определяет длину последовательности. Все **continuation bytes** (второй, третий, четвёртый байты в многобайтовой последовательности) начинаются с `10` — это позволяет при потере синхронизации найти границу следующего символа (self-synchronizing).

**Пошагово: как 'П' (U+041F) кодируется в UTF-8**

1. Code point: **1055** (десятично) = **0x041F** (hex) = **0000 0100 0001 1111** (16 бит)
2. Диапазон: 1055 < 2048 → попадает в U+0080–U+07FF → **2 байта**
3. Формат 2 байт: `110xxxxx 10xxxxxx`; нужно 11 бит данных
4. 1055 в двоичном: **100 0001 1111** (11 бит). Разбиваем: 5 бит в первый байт (после `110`), 6 бит во второй (после `10`)
5. Первый байт: `110` + `10000` = 11010000 = 0xD0
6. Второй байт: `10` + `011111` = 10011111 = 0x9F
7. Результат: **b'\xd0\x9f'**

**Пошагово: как '€' (U+20AC) кодируется в UTF-8 (3 байта)**

1. Code point: **8364** = **0x20AC** = **0010 0000 1010 1100** (16 бит)
2. Диапазон U+0800–U+FFFF → **3 байта**
3. Формат: `1110xxxx 10xxxxxx 10xxxxxx`; нужно 16 бит
4. 0x20AC в двоичном: **0010 0000 1010 1100**. Разбиваем: 4 бита → первый байт, 6+6 → второй и третий
5. Первый: `1110` + `0010` = 11100010 = 0xE2
6. Второй: `10` + `000010` = 10000010 = 0x82
7. Третий: `10` + `101100` = 10101100 = 0xAC
8. Результат: **b'\xe2\x82\xac'**

**Пошагово: как '😀' (U+1F600) кодируется в UTF-8 (4 байта)**

1. Code point: **128512** = **0x1F600** = **0001 1111 0110 0000 0000** (17 бит)
2. Диапазон U+10000–U+10FFFF → **4 байта**
3. Формат: `11110xxx 10xxxxxx 10xxxxxx 10xxxxxx`; нужно 21 бит
4. 0x1F600: **000 011111 011000 000000** — разбиваем по шаблону
5. Первый: `11110` + `000` = 11110000 = 0xF0
6. Второй: `10` + `011111` = 10011111 = 0x9F
7. Третий: `10` + `011000` = 10011000 = 0x98
8. Четвёртый: `10` + `000000` = 10000000 = 0x80
9. Результат: **b'\xf0\x9f\x98\x80'**

**Примеры в Python:**

```python
"A".encode("utf-8")      # b'A'           — 1 байт (0x41), совпадает с ASCII
"П".encode("utf-8")      # b'\xd0\x9f'    — 2 байта
"€".encode("utf-8")      # b'\xe2\x82\xac' — 3 байта (евро)
"中".encode("utf-8")     # b'\xe4\xb8\xad' — 3 байта (китайский)
"😀".encode("utf-8")     # b'\xf0\x9f\x98\x80' — 4 байта (эмодзи)

# Проверка размера
len("Hello".encode("utf-8"))    # 5 — по одному байту на ASCII-символ
len("Привет".encode("utf-8"))   # 12 — по 2 байта на кириллическую букву
len("😀".encode("utf-8"))       # 4 — один эмодзи
```

**Плюсы UTF-8:** (1) совместимость с ASCII — файлы в ASCII остаются валидным UTF-8; (2) self-synchronizing — можно найти начало следующего символа с любой позиции; (3) компактность для западных языков; (4) отсутствие BOM и проблем с порядком байт; (5) де-факто стандарт для веба, JSON, современных ОС.

**Минусы UTF-8:** для восточноазиатских текстов (много иероглифов) каждый символ занимает 3 байта; UTF-16 даёт 2 байта на большинство иероглифов в BMP — может быть компактнее для таких текстов.

**Таблица битовых масок UTF-8 (для справки):**

| Первый байт | Маска | Бит данных | Continuation |
|-------------|-------|------------|--------------|
| 0x00–0x7F | 0xxxxxxx | 7 | нет |
| 0xC0–0xDF | 110xxxxx | 5 + 6 = 11 | 1 байт 10xxxxxx |
| 0xE0–0xEF | 1110xxxx | 4 + 6 + 6 = 16 | 2 байта |
| 0xF0–0xF7 | 11110xxx | 3 + 6 + 6 + 6 = 21 | 3 байта |

#### Self-synchronizing: почему UTF-8 устойчив к потере синхронизации

В UTF-8 **первый байт** каждой последовательности однозначно определяет её длину: `0xxxxxxx` — 1 байт; `110xxxxx` — 2 байта; `1110xxxx` — 3 байта; `11110xxx` — 4 байта. Все **continuation bytes** начинаются с `10`. Если при чтении потока байт произошла ошибка (пропуск, вставка), декодер может найти начало следующего корректного символа, пропустив continuation bytes (они не могут быть первым байтом) и дойдя до валидного первого байта. В кодировках с фиксированной длиной (UTF-32) или без явной структуры (некоторые legacy) потеря одного байта ломает всё последующее декодирование.

#### Примеры невалидных последовательностей UTF-8

- **0xFF** — невалидный первый байт (в UTF-8 нет байтов, начинающихся с 11111)
- **0xC0 0x80** — overlong encoding для NULL (валиднее использовать 0x00)
- **0xE0 0x80** — обрезанная трёхбайтовая последовательность
- **0x80** — continuation byte без предшествующего первого байта

При `errors='strict'` каждая из них вызывает `UnicodeDecodeError`.

#### Overlong encoding — почему 0xC0 0x80 невалиден для NULL

**Overlong encoding** — кодирование code point с использованием большего числа байт, чем необходимо. В UTF-8 это **запрещено** по соображениям безопасности (раньше использовалось в атаках) и однозначности.

**NULL (U+0000)** в UTF-8 кодируется одним байтом: `0x00` (формат `0xxxxxxx`). Последовательность **0xC0 0x80** формально имеет структуру 2-байтовой последовательности: `11000000 10000000` → биты данных: 0000000 = 0. То есть декодер мог бы интерпретировать это как code point 0. Но спецификация UTF-8 требует **минимальной** длины: если символ можно закодировать в 1 байт, недопустимо использовать 2. Аналогично: 'A' (U+0041) должна быть `0x41`, а не `0xC1 0x81`.

**Другие overlong примеры:**
- `0xE0 0x80 0x80` — overlong для U+0000 (3 байта вместо 1)
- `0xF0 0x80 0x80 0x41` — overlong для 'A' U+0041 (4 байта вместо 1)

Python при decode с `errors='strict'` отклоняет overlong последовательности.

#### UTF-16 (Unicode Transformation Format, 16-bit)

**UTF-16** — 2 или 4 байта на символ. Используется в Windows API, Java, .NET, JavaScript (внутреннее представление строк).

- **BMP (U+0000–U+FFFF):** 2 байта (один 16-битный code unit). Code point напрямую кодируется как 16-битное число.
- **U+10000–U+10FFFF:** 4 байта (surrogate pair). Code point кодируется двумя 16-битными значениями:
  - **High surrogate:** 0xD800–0xDBFF
  - **Low surrogate:** 0xDC00–0xDFFF
  - Формула: для code point `U` (U > 0xFFFF): `U' = U - 0x10000`; high = `0xD800 + (U' >> 10)`; low = `0xDC00 + (U' & 0x3FF)`.

**Пример: '😀' (U+1F600) в UTF-16.** U' = 0x1F600 - 0x10000 = 0xF600. High = 0xD800 + (0xF600 >> 10) = 0xD800 + 0x3D = 0xD83D. Low = 0xDC00 + (0xF600 & 0x3FF) = 0xDC00 + 0x200 = 0xDE00. В little-endian: 0x3D 0xD8 0x00 0xDE → `b'\x3d\xd8\x00\xde'`.

**Пошагово: формула surrogate pair для U+1F600 (128512):**

1. Code point U = 0x1F600 (вне BMP)
2. Вычисляем U' = U - 0x10000 = 0x1F600 - 0x10000 = **0xF600** (число в диапазоне 0–0xFFFFF, 20 бит)
3. Разбиваем U' на старшие 10 бит и младшие 10 бит: U' = (high_10 << 10) | low_10
   - high_10 = U' >> 10 = 0xF600 >> 10 = **0x3D** (61)
   - low_10 = U' & 0x3FF = 0xF600 & 0x3FF = **0x200** (512)
4. High surrogate = 0xD800 + high_10 = 0xD800 + 0x3D = **0xD83D**
5. Low surrogate = 0xDC00 + low_10 = 0xDC00 + 0x200 = **0xDE00**
6. В UTF-16-LE каждый code unit кодируется 2 байтами (младший, старший): 0xD83D → 0x3D 0xD8; 0xDE00 → 0x00 0xDE
7. Итого: `b'\x3d\xd8\x00\xde'`

**Обратное преобразование (декодирование surrogate pair):** high=0xD83D, low=0xDE00 → U' = ((high - 0xD800) << 10) + (low - 0xDC00) = (0x3D << 10) + 0x200 = 0xF600; U = U' + 0x10000 = 0x1F600.

Порядок байт: **big-endian** (BOM `\xfe\xff`) или **little-endian** (BOM `\xff\xfe`). В Python при `encode('utf-16')` по умолчанию добавляется BOM; варианты `utf-16-le` (little-endian без BOM) и `utf-16-be` (big-endian без BOM).

**BOM (Byte Order Mark) — подробнее:** BOM — это символ U+FEFF (ZERO WIDTH NO-BREAK SPACE), записанный в начале файла. Его байтовое представление однозначно указывает порядок байт: `\xff\xfe` — little-endian (младший байт первым), `\xfe\xff` — big-endian. Декодер читает BOM и переключается в нужный режим. В UTF-8 BOM опционален: `\xef\xbb\xbf` — три байта, кодирующие U+FEFF. Некоторые программы (например, старые версии Excel) требуют BOM для корректного распознавания UTF-8.

```python
"A".encode("utf-16")      # b'\xff\xfeA\x00' — BOM + 2 байта (little-endian)
"П".encode("utf-16-le")   # b'\x1f\x04' — 0x041F в little-endian
"😀".encode("utf-16-le")  # b'\x3d\xd8\x00\xde' — surrogate pair (4 байта)
```

#### UTF-32 (Unicode Transformation Format, 32-bit)

**UTF-32** — фиксированные 4 байта на символ. Простая структура: каждый code point кодируется одним 32-битным значением. Неэффективна по размеру для текстов с много ASCII/латиницы (в 4 раза больше, чем UTF-8 для таких символов).

```python
"A".encode("utf-32")      # b'\xff\xfe\x00\x00A\x00\x00\x00' — BOM + 4 байта
"П".encode("utf-32-le")   # b'\x1f\x04\x00\x00' — 4 байта
```

#### Latin-1 (ISO-8859-1)

**Latin-1** — 1 байт на символ, диапазон 0–255. Соответствует первым 256 code points Unicode (U+0000–U+00FF). Включает латиницу, базовую пунктуацию и часть расширенной латиницы (á, é, ñ, ü и т.д.).

**Ключевая особенность:** **любой байт 0–255 можно однозначно превратить в символ Latin-1 и обратно** — round-trip без потерь. Формула: `chr(byte)` для decode (байт 0–255 → символ U+0000–U+00FF), `ord(char)` для encode (если code point 0–255). Используется, когда нужно «протащить» произвольные байты через текстовый интерфейс (имена файлов в POSIX, старые протоколы, surrogateescape).

**Теоретическое обоснование:** Latin-1 устанавливает **биекцию** между множеством байт {0..255} и множеством code points {U+0000..U+00FF}. Нет неоднозначности: один байт → один символ. Поэтому decode('latin-1') **никогда не выдаёт UnicodeDecodeError** — все 256 возможных байт валидны. Это делает Latin-1 «пасsthrough» для произвольных байт: если вы не знаете кодировку, но хотите сохранить данные в str без потерь, Latin-1 — единственная кодировка с такой гарантией (в сочетании с surrogateescape для UTF-8 получаем аналогичный round-trip для невалидного UTF-8).

```python
# Любой байт 0–255 — валидный символ в Latin-1
bytes([0x00, 0xff]).decode("latin-1")  # '\x00\xff' (два символа)
"\x00\xff".encode("latin-1")           # b'\x00\xff'

# Round-trip: байты → str → байты
raw = bytes(range(256))
s = raw.decode("latin-1")
back = s.encode("latin-1")
back == raw   # True
```

#### Кодировки для кириллицы: CP1251, CP866, KOI8-R

При работе с русским текстом в legacy-системах встречаются разные кодировки:

| Кодировка | Платформа | Диапазон | Примечание |
|-----------|-----------|----------|------------|
| **CP1251** (Windows-1251) | Windows (русская локаль) | 1 байт, 0x80–0xFF для кириллицы | Основная кодировка Windows для русского |
| **CP866** (DOS) | DOS, консоль Windows (старая) | 1 байт, другая таблица | Консоль до chcp 65001 |
| **KOI8-R** | Unix, электронная почта (исторически) | 1 байт, другая таблица | Порядок букв оптимизирован для сравнения |
| **UTF-8** | Все современные системы | 2 байта на кириллическую букву | Рекомендуется для новых проектов |

```python
# Чтение файла в CP1251 (Windows)
with open("legacy.txt", encoding="cp1251") as f:
    text = f.read()

# Запись в CP1251 для совместимости
with open("out.txt", "w", encoding="cp1251") as f:
    f.write("Привет")

# Конвертация CP1251 → UTF-8
text = open("cp1251_file.txt", encoding="cp1251").read()
open("utf8_file.txt", "w", encoding="utf-8").write(text)
```

**Практический вывод:** для новых проектов используйте UTF-8 везде; при чтении legacy-файлов явно указывайте encoding (CP1251, CP866 и т.д.).

#### Детали кодировок кириллицы: CP1251 vs CP866 vs KOI8-R

**CP1251 (Windows-1251):** русская буква «А» (U+0410) → байт 0xC0; «я» (U+044F) → 0xDF. Кириллица занимает диапазон 0xC0–0xFF; порядок букв соответствует Unicode (А–Я, а–я). Используется в Windows (русская локаль), офисных документах (.doc, .xls), многих веб-страницах из СНГ.

**CP866 (DOS):** та же «А» (U+0410) → байт 0x80; «п» (U+043F) → 0xAF. Кириллица в диапазоне 0x80–0xAF (псевдографика — в 0xB0–0xDF). Консоль DOS и старая консоль Windows (до chcp 65001) использовали CP866. При открытии файла CP1251 с encoding='cp866' (и наоборот) русские буквы отображаются как «кракозябры» — таблицы несовместимы.

**KOI8-R (Unix, почта):** буквы расположены в другом порядке: «р» (U+0440) → 0xD2, «ю» (U+044E) → 0xD3. Особенность: латинские буквы A–Z кодируются теми же кодами, что и русские А–Я (поразрядное сравнение даёт алфавитный порядок). Исторически — Unix, электронная почта в СНГ.

**Пример: одна и та же строка «АБВ» в разных кодировках:**

```python
"АБВ".encode("cp1251")   # b'\xc0\xc1\xc2'
"АБВ".encode("cp866")    # b'\x80\x81\x82'
"АБВ".encode("koi8-r")   # b'\xe1\xe2\xe3'
"АБВ".encode("utf-8")    # b'\xd0\x90\xd0\x91\xd0\x92'
```

**Определение кодировки:** однозначно определить кодировку по байтам невозможно (одни и те же байты в разных кодировках — разные символы). Эвристики: (1) попробовать декодировать — если нет ошибок и результат «выглядит осмысленно»; (2) использовать библиотеки вроде `chardet`; (3) указывать кодировку в метаданных (XML: `encoding="cp1251"`, HTTP: `Content-Type: charset=...`). (4) BOM в начале файла — надёжный признак UTF-8 (`\xef\xbb\xbf`), UTF-16 LE (`\xff\xfe`), UTF-16 BE (`\xfe\xff`).

#### Сравнительная таблица кодировок

| Кодировка | Байт на символ | BOM | Использование |
|-----------|----------------|-----|---------------|
| UTF-8     | 1–4            | нет (опционально \xef\xbb\xbf) | Веб, файлы, сеть, JSON, Python 3 по умолчанию |
| UTF-16    | 2 или 4        | да (по умолчанию в Python) | Windows API, Java, .NET, JavaScript |
| UTF-32    | 4              | да | Внутреннее представление, редко в I/O |
| Latin-1   | 1              | нет | Round-trip байт 0–255, старые протоколы |
| CP1251    | 1              | нет | Русская Windows |
| CP866     | 1              | нет | Русская DOS, старая консоль |

#### Плохо vs лучше: выбор кодировки

```python
# Плохо: полагаться на кодировку по умолчанию при обмене с внешним миром
with open("data.txt") as f:
    text = f.read()   # на одной машине UTF-8, на другой CP1251 — нестабильно

# Лучше: явно указывать кодировку
with open("data.txt", encoding="utf-8") as f:
    text = f.read()

# Плохо: использовать Latin-1 для кириллицы
"Привет".encode("latin-1")   # UnicodeEncodeError

# Лучше: UTF-8 для универсального текста
"Привет".encode("utf-8")     # b'\xd0\x9f...'

# Плохо: терять байты при decode
raw = b'\xff\xfe'
raw.decode("utf-8")          # UnicodeDecodeError

# Лучше: surrogateescape для round-trip
s = raw.decode("utf-8", errors="surrogateescape")
s.encode("utf-8", errors="surrogateescape") == raw   # True
```

### 6.1.3 ord() и chr() — код и символ

#### ord(c) — символ → code point

**Сигнатура:** `ord(c) -> int`

**Назначение:** возвращает **Unicode code point** символа `c` — целое число, однозначно идентифицирующее символ в Unicode.

**Параметр:** `c` — строка ровно из одного символа (одного code point).  
**Возвращает:** целое число в диапазоне 0–1 114 111.  
**Исключения:** `TypeError`, если `c` не строка; `TypeError`, если `len(c) != 1`.

```python
ord('A')       # 65
ord('П')       # 1055
ord('€')       # 8364
ord('😀')      # 128512
ord(' ')       # 32 (пробел)
ord('\n')      # 10 (LF)

# Ошибки
ord('')        # TypeError: ord() expected a character, but string of length 0
ord('AB')      # TypeError: ord() expected a character, but string of length 2
ord(65)        # TypeError: ord() expected str, got int

# Частая ошибка: b'A'[0] — это int (65), ord() ожидает str
ord(b'A'[0])   # TypeError: ord() expected str, got int
```

**Пошагово: что происходит при ord('П').** (1) Интерпретатор проверяет тип аргумента — должен быть `str`. (2) Проверяет длину строки — должна быть ровно 1 (один code point). (3) Извлекает code point первого (и единственного) символа — для 'П' это 1055 (U+041F). (4) Возвращает целое число 1055.

**Теоретическое обоснование:** `ord` и `chr` — взаимно обратные функции: `chr(ord(c)) == c` и `ord(chr(n)) == n` (для валидных code points). Они образуют мост между **абстрактным миром символов** (str) и **числовым представлением** (code point). Это нужно для: (1) работы с кодировками — encode/decode оперируют code points; (2) сортировки по кодам — сравнение `ord(a) < ord(b)`; (3) генерации символов по диапазонам — `chr(i) for i in range(start, end)`; (4) проверки принадлежности к диапазону — `ord('0') <= ord(c) <= ord('9')` для цифры.

**Инварианты:** (а) `ord(chr(i)) == i` для i в 0..1114111; (б) `chr(ord(c)) == c` для любого символа c (str длиной 1); (в) `ord` и `chr` не зависят от кодировки — они работают с абстрактными code points, а не с байтами.

#### chr(i) — code point → символ

**Сигнатура:** `chr(i) -> str`

**Назначение:** возвращает **строку из одного символа** с Unicode code point `i`.

**Параметр:** `i` — целое число в диапазоне 0–1 114 111 (0x0–0x10FFFF).  
**Возвращает:** строка длиной 1.  
**Исключения:** `ValueError`, если `i` вне допустимого диапазона.

```python
chr(65)        # 'A'
chr(1055)      # 'П'
chr(128512)    # '😀'
chr(0x41)      # 'A' (шестнадцатеричный литерал)
chr(1114111)   # '\U0010ffff' — максимальный валидный code point
chr(0)         # '\x00' — null (U+0000)

# Ошибки
chr(1114112)   # ValueError: chr() arg not in range(0x110000)
chr(-1)        # ValueError: chr() arg not in range(0x110000)
chr(1.5)       # TypeError: integer argument expected, got float
```

**Граничные случаи chr:**

| Аргумент | Результат | Примечание |
|----------|-----------|------------|
| 0        | `'\x00'`  | Null — валидный символ Unicode |
| 127      | `'\x7f'`  | DEL (последний ASCII) |
| 255      | `'\xff'`  | Последний Latin-1 |
| 1114111  | `'\U0010ffff'` | Максимальный валидный code point |
| -1, 1114112 | ValueError | Вне диапазона |

#### Практическое применение ord и chr

**1. Генерация символов по диапазону:**

```python
# Латинский алфавит в нижнем регистре
"".join(chr(i) for i in range(ord('a'), ord('z') + 1))
# 'abcdefghijklmnopqrstuvwxyz'

# Цифры 0–9
"".join(chr(i) for i in range(ord('0'), ord('9') + 1))
# '0123456789'

# Кириллица А–Я (неполный диапазон — зависит от блока)
"".join(chr(i) for i in range(0x0410, 0x0430))  # А–Я
```

**2. Проверка типа символа по коду:**

```python
def is_ascii_digit(c):
    return len(c) == 1 and ord('0') <= ord(c) <= ord('9')

def is_ascii_letter(c):
    return len(c) == 1 and (
        ord('a') <= ord(c) <= ord('z') or ord('A') <= ord(c) <= ord('Z')
    )
```

**3. Ручное «декодирование» одного байта в Latin-1:**

```python
byte_val = 65
char = chr(byte_val)  # 'A' — байт 65 = Latin-1 символ U+0041
```

**4. Преобразование числа в строку в произвольной системе счисления (цифры 0–9, A–Z):**

```python
def to_base_str(n, base=16):
    if n == 0:
        return "0"
    digits = []
    while n:
        n, r = divmod(n, base)
        digits.append(chr(ord('0') + r) if r < 10 else chr(ord('A') + r - 10))
    return "".join(reversed(digits))
to_base_str(255, 16)  # 'FF'
```

**5. Сравнение символов по коду (лексикографический порядок Unicode):**

Сортировка по `ord` даёт порядок по числовым значениям code points, что совпадает с лексикографическим порядком Unicode (по умолчанию `sorted(s)` использует тот же порядок).

```python
def sort_by_code(s):
    return "".join(sorted(s, key=ord))
sort_by_code("Привет")  # 'Пвеиирт' (по возрастанию code points)
```

**6. Фильтрация «печатаемых» ASCII-символов:**

```python
def ascii_printable_only(s: str) -> str:
    return "".join(c for c in s if 32 <= ord(c) <= 126)
# Удаляет управляющие символы, null, и символы вне ASCII
```

**7. Проверка, что строка — только ASCII (без вызова s.isascii()):**

```python
def is_ascii_string(s: str) -> bool:
    return all(ord(c) < 128 for c in s)
```

**8. Преобразование bytes в читаемый hex-дамп (по 16 байт в строке):**

```python
def hexdump(data: bytes, width: int = 16) -> str:
    lines = []
    for i in range(0, len(data), width):
        chunk = data[i:i+width]
        hex_part = ' '.join(f'{b:02x}' for b in chunk)
        ascii_part = ''.join(chr(b) if 32 <= b < 127 else '.' for b in chunk)
        lines.append(f"{i:04x}  {hex_part:<{width*3}}  {ascii_part}")
    return '\n'.join(lines)
```

#### Граничные случаи ord и chr

| Вызов | Результат | Исключение |
|-------|-----------|------------|
| `ord('')` | — | TypeError: expected a character, but string of length 0 |
| `ord('AB')` | — | TypeError: expected a character, but string of length 2 |
| `ord(65)` | — | TypeError: expected str, got int |
| `chr(-1)` | — | ValueError: arg not in range(0x110000) |
| `chr(1114112)` | — | ValueError: arg not in range(0x110000) |
| `chr(0)` | `'\x00'` | — |
| `chr(1114111)` | `'\U0010ffff'` | — |
| `ord(chr(1055))` | 1055 | — (round-trip) |

**Инвариант:** для любого символа `c` (str длиной 1) с валидным code point: `chr(ord(c)) == c`. Для любого int `i` в 0–1114111: `ord(chr(i)) == i`.

#### Плохо vs лучше: ord и chr

```python
# Плохо: ord() от int (результат bytes[i])
ord(b'A'[0])   # TypeError: ord() expected str, got int
# b'A'[0] уже 65 (int); ord не нужен

# Лучше: понимать, что bytes[i] — int
b'A'[0]        # 65
chr(b'A'[0])   # 'A'

# Плохо: chr() от bytes (ожидается int)
chr(b'A')      # TypeError: an integer is required

# Лучше: chr от code point
chr(65)        # 'A'

# Плохо: итерация по bytes с ord
for byte in b'AB':
    c = ord(byte)   # TypeError — byte уже int

# Лучше: byte уже int
for byte in b'AB':
    char = chr(byte)  # 'A', 'B'
```

### 6.1.4 Escape-последовательности: \N{NAME}, \uHHHH, \UHHHHHHHH

В строковых литералах Python поддерживает несколько способов задать символ по его Unicode-коду или имени. Это полезно, когда символ невозможно ввести с клавиатуры или когда нужна явная читаемость кода.

#### \uHHHH (4 hex-цифры) — символы BMP

**Формат:** `\uHHHH` — четыре шестнадцатеричные цифры (H — 0–9, A–F).  
**Диапазон:** code points U+0000–U+FFFF (BMP — Basic Multilingual Plane).  
**Результат:** один символ (строка длиной 1).

```python
'\u0041'      # 'A' (U+0041)
'\u041f'      # 'П' (U+041F)
'\u00e9'      # 'é' (U+00E9)
'\u2022'      # '•' (BULLET)
s = '\u041f\u0440\u0438\u0432\u0435\u0442'  # 'Привет'
```

**Граничные случаи:** `\u0000` — null; `\u000a` — перевод строки (то же, что `\n`); `\uffff` — последний символ BMP. Регистр hex-цифр (a-f, A-F) не важен. Должно быть ровно 4 hex-цифры; лидирующие нули допустимы: `\u0041` — 'A'.

**Ограничение \u:** только BMP (U+0000–U+FFFF). Для символов вне BMP (например, 😀 U+1F600) используйте `\U0001F600` (8 hex-цифр).

#### \UHHHHHHHH (8 hex-цифр) — любые code points

**Формат:** `\UHHHHHHHH` — восемь шестнадцатеричных цифр.  
**Диапазон:** code points U+000000–U+10FFFF (весь Unicode).  
**Результат:** один символ (строка длиной 1).

```python
'\U0001F600'   # '😀' (U+1F600 — GRINNING FACE)
'\U000041'     # 'A' (можно лидирующие нули)
'\U0010FFFF'   # '\U0010ffff' — максимальный валидный code point
```

**Ошибка:** `\U00110000` — вне допустимого диапазона; будет `SyntaxError` или `ValueError` при интерпретации.

#### \N{name} — символ по Unicode name

**Формат:** `\N{name}` — имя символа из Unicode Character Database (UCD).  
**Результат:** один символ, соответствующий указанному имени.

Имена **чувствительны к регистру** и должны точно соответствовать официальному Unicode name. Список имён можно посмотреть в модуле `unicodedata`: `unicodedata.name(ch)`.

```python
'\N{LATIN CAPITAL LETTER A}'     # 'A'
'\N{EM DASH}'                    # '—' (длинное тире, U+2014)
'\N{BULLET}'                     # '•' (маркер списка)
'\N{GRINNING FACE}'              # '😀'
'\N{CYRILLIC CAPITAL LETTER PE}' # 'П'
'\N{ZERO WIDTH SPACE}'           # невидимый символ U+200B
'\N{COMBINING ACUTE ACCENT}'     # U+0301 (combining accent)
```

**Практический вывод:** `\N{name}` удобен для редких символов, когда не хочется искать code point. Для путей Windows **осторожность:** `\N` в `C:\Users\name\file.txt` может интерпретироваться как escape. Используйте `r'...'` или прямые слэши.

#### Обычные escape-последовательности (полная таблица)

| Последовательность | Значение | Code point |
|--------------------|----------|------------|
| `\n` | Перевод строки (LF, newline) | U+000A |
| `\r` | Возврат каретки (CR) | U+000D |
| `\t` | Табуляция (horizontal tab) | U+0009 |
| `\v` | Вертикальная табуляция | U+000B |
| `\f` | Перевод страницы (form feed) | U+000C |
| `\\` | Обратный слэш | U+005C |
| `\'` | Одинарная кавычка | U+0027 |
| `\"` | Двойная кавычка | U+0022 |
| `\xHH` | Символ по байту (2 hex-цифры, 0–255) | U+0000–U+00FF |
| `\0` | Null | U+0000 |
| `\a` | Звонок (BEL) | U+0007 |
| `\b` | Backspace | U+0008 |

**`\xHH`** — особенно полезен для указания символов по байту (0–255). В Latin-1 байт N соответствует символу U+00NN: `\x41` = 'A', `\xff` = 'ÿ'. Регистр hex-цифр не важен: `\x41` и `\x41` эквивалентны.

**Граничные случаи `\xHH`:** `\x00` — null; `\x7f` — DEL; `\xff` — максимальный байт. В строке `"\x41\x42"` — два символа 'A' и 'B'. Одна пара hex даёт один байт/символ; `\x0` (без второй цифры) — невалидно, нужны ровно две hex-цифры.

#### bytes-литералы: ограничения на escape

В **bytes-литералах** (`b'...'`) допустимы только escape-последовательности, дающие **один байт** (0–255):

- `\xHH` — один байт
- `\n`, `\r`, `\t`, `\v`, `\f`, `\a`, `\b` — соответствующие байты
- `\\`, `\'`, `\"` — обратный слэш, кавычки

**`\u` и `\U` в bytes не поддерживаются** — bytes не знают о Unicode; они хранят сырые байты 0–255.

```python
b'\x41'       # b'A' — байт 65
b'\x00\xff'   # b'\x00\xff' — два байта
b'\u0041'     # SyntaxError в Python 3 — \u в bytes недопустим
b'\U0001F600' # SyntaxError — \U в bytes недопустим
```

**Почему:** bytes представляют последовательность байт, а не символов. Один символ Unicode может занимать 1–4 байта в UTF-8; bytes-литерал задаёт байты напрямую, без интерпретации Unicode.

**Ограничение `\xHH` в bytes:** каждая пара hex-цифр даёт один байт. `\x41` → байт 65; `\x9f` → байт 159. Максимум — `\xff` (255). Для задания UTF-8 последовательности кириллической «П» в bytes нужно писать `b'\xd0\x9f'` — два escape, два байта.

#### Сырые строки (r'...') и escape

В **сырых строках** (r-strings) обратный слэш **не экранирует** следующий символ — `r'\n'` это два символа `\` и `n`, а не перевод строки. Удобно для путей Windows (`r'C:\Users\file.txt'`), регулярных выражений и случаев, когда нужно передать `\` как есть.

**Важно:** в Python 3.12+ поведение `\u`, `\U`, `\N` в r-строках изменилось — они **не** интерпретируются в r-строках (PEP 701). В более ранних версиях они могли интерпретироваться, что вызывало путаницу с путями типа `C:\Users\name\file.txt` (буква `n` после `\U` могла дать неожиданный результат).

**Безопасный способ для путей Windows:**

```python
# Вариант 1: сырая строка
path = r'C:\Users\name\file.txt'

# Вариант 2: прямые слэши (работают на Windows)
path = 'C:/Users/name/file.txt'

# Вариант 3: удвоенные обратные слэши
path = 'C:\\Users\\name\\file.txt'
```

#### Пошагово: что происходит при интерпретации '\u041f'

1. Интерпретатор встречает escape-последовательность `\u041f` в строковом литерале.
2. `\u` — признак 4-значного hex-кода; следующие 4 символа `041f` интерпретируются как шестнадцатеричное число.
3. `0x041f` = 1055 (десятичное) — code point U+041F.
4. Символ U+041F — кириллическая буква **П**.
5. Результат: строка `"П"` (один символ).

**Важно:** escape обрабатывается **на этапе разбора литерала**; к моменту выполнения строка уже содержит символ 'П', а не последовательность `\`, `u`, `0`, `4`, `1`, `f`.

#### Плохо vs лучше: escape и bytes

```python
# Плохо: пытаться задать Unicode в bytes
b'\u041f'      # SyntaxError — \u в bytes недопустим

# Лучше: задать UTF-8 байты явно
b'\xd0\x9f'    # b'П' в UTF-8

# Плохо: смешивать типы при конкатенации
"Привет" + b'\xd0\x9f'   # TypeError

# Лучше: один тип
"Привет" + b'\xd0\x9f'.decode('utf-8')   # "ПриветП"
```

#### Граничные случаи escape-последовательностей

- **`\x41`** — один байт 0x41; в str это символ 'A' (U+0041)
- **`\x7f`** — DEL (U+007F); допустим в str
- **`\x80`** — в str это символ U+0080 (Control character)
- **`\xff`** — в str это символ U+00FF (ÿ)
- **`\u0000`** — null; допустим в str
- **`\U0010FFFF`** — максимальный валидный code point
- **`\U00110000`** — SyntaxError (вне диапазона)

#### Восьмеричные escape \OOO (устаревшие в Python 3)

В Python 3 восьмеричный escape `\OOO` (1–3 восьмеричные цифры) поддерживается только для значений 0–255. В строках `\0` — null; `\101` — 'A' (65 в восьмеричной). В bytes: `b'\101'` → байт 65. **Рекомендация:** предпочитать `\xHH` (hex) — нагляднее и однозначнее.

```python
'\101'   # 'A' (восьмерично 65)
'\0'     # '\x00' — null
```

#### PEP 701 (Python 3.12+): f-strings и Unicode escapes

В Python 3.12+ (PEP 701) f-строки стали более гибкими; escape-последовательности `\u`, `\U`, `\N` в f-строках интерпретируются на этапе разбора. В сырых f-строках `rf'...'` поведение согласовано с обычными r-строками.
