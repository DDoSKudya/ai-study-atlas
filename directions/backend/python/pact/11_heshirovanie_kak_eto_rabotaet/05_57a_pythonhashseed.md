### Цель раздела

Понять влияние `PYTHONHASHSEED` на хеши строк и bytes, зачем нужна рандомизация (защита от hash-flooding DoS), что такое SipHash, и когда нужна воспроизводимость (тесты, CI).

### Термины

| Термин | Определение |
|--------|-------------|
| **PYTHONHASHSEED** | Переменная окружения; задаёт seed для рандомизации хешей str/bytes. `random` (по умолчанию) или целое (0, 1, 2, ...). |
| **Hash randomization** | Введение случайного seed в хеш-функцию (для str, bytes, datetime и др.), чтобы злоумышленник не мог предсказать хеши. |
| **Hash-flooding DoS** | Атака: злоумышленник подбирает много ключей с одинаковым хешем → деградация dict/set до O(n²) → отказ в обслуживании. |
| **SipHash** | Криптографически стойкая хеш-функция (PRF); используется в Python для str/bytes (PEP 456). Устойчива к предсказанию при неизвестном секретном ключе. |

### PEP 456 и SipHash

**PEP 456** (Python 3.4) ввёл:
1. **SipHash-24** (или SipHash-13 в части сборок) для str, bytes, bytearray, datetime.
2. **PYTHONHASHSEED** — при `random` seed генерируется при старте интерпретатора; при целочисленном значении — детерминированный seed.

**Зачем SipHash:** старые хеш-функции (FNV, DJB2 и др.) были предсказуемы: по алгоритму можно было вычислить множество строк с одинаковым хешем. Злоумышленник отправлял такие строки в веб-приложение (например, POST с полями `k1`, `k2`, ..., `kn`), сервер парсил их в dict — все ключи коллидировали, dict вырождался в O(n²) на операцию → DoS. SipHash — PRF (pseudorandom function): при неизвестном 128-битном ключе (seed) предсказать хеш невозможно. Без seed атакующий не может сгенерировать коллидирующие ключи.

### PYTHONHASHSEED: значения и влияние

| Значение | Поведение | Когда использовать |
|----------|-----------|-------------------|
| Не задан или `random` | Случайный seed при каждом запуске | По умолчанию; защита от DoS |
| `0` | Детерминированный seed (0) | Воспроизводимость: тесты, CI |
| `1`, `2`, ... | Фиксированный seed | Отладка, воспроизведение конкретного поведения |

**Важно:** PYTHONHASHSEED должен быть установлен **до** запуска интерпретатора. Изменить его из кода нельзя.

```bash
# Воспроизводимые тесты
PYTHONHASHSEED=0 pytest

# Случайный хеш (по умолчанию)
python script.py
```

### Какие типы зависят от PYTHONHASHSEED

- **str** — да
- **bytes** — да
- **bytearray** — не hashable (`hash(bytearray(...))` → TypeError); SipHash для bytes применяется к `bytes(bytearray)` при внутренних операциях, если они есть. В практике хеширования bytearray не участвует.
- **datetime.datetime**, **datetime.date**, **datetime.time** — да
- **int**, **float**, **tuple** (и т.д.) — **не зависят** от PYTHONHASHSEED; их хеш детерминирован.

### Порядок итерации

| Структура | Порядок | С Python |
|-----------|---------|----------|
| **dict** | Порядок вставки | 3.7 (гарантия языка) |
| **set** | Не гарантирован | — |
| **frozenset** | Не гарантирован | — |
| **kwargs** в вызове функции | Порядок передачи; ключи сохраняются (dict) | 3.6 |

Порядок dict определяется структурой indices + entries (см. §54), а не хешами. Порядок set зависит от хешей и истории вставок; при разном PYTHONHASHSEED порядок может отличаться между запусками.

**kwargs:** с Python 3.6 порядок ключей в `**kwargs` сохраняется (dict хранит порядок вставки). Это влияет на `inspect.signature`, порядок параметров при `*args, **kwargs` и отображение именованных аргументов. Связь с хешированием косвенная: dict использует хеш для поиска, но итерация идёт по порядку entries.

### Примеры

```python
# Хеши str зависят от PYTHONHASHSEED
# Запуск 1: hash("hello") = -1234567890
# Запуск 2: hash("hello") = 9876543210
# При PYTHONHASHSEED=0: hash("hello") одинаков в любом запуске

# int не зависит от PYTHONHASHSEED
hash(42)  # Всегда 42

# Порядок dict стабилен (порядок вставки)
d = {"z": 1, "a": 2, "m": 3}
print(list(d))  # ['z', 'a', 'm']

# Порядок set не гарантирован
s = {"a", "b", "c"}
print(s)  # Может быть разным между запусками
```

### Hash-flooding DoS: суть атаки

1. Злоумышленник знает алгоритм хеширования (без рандомизации).
2. Он находит множество ключей k₁, k₂, ..., kₙ с одинаковым `hash(k) % size` для данной таблицы.
3. Он отправляет их в приложение (например, POST с полями `k1`, `k2`, ...).
4. Сервер парсит данные в dict; все ключи коллидируют → длинная цепочка проб.
5. Операции с dict становятся O(n²) → CPU перегружен → DoS.

**Защита:** SipHash + PYTHONHASHSEED=random. Без знания seed атакующий не может вычислить коллидирующие ключи.

### Когда устанавливать PYTHONHASHSEED в production

**Рекомендация:** оставлять `PYTHONHASHSEED=random` (или не задавать — по умолчанию random). Это защита от hash-flooding DoS. Устанавливать `PYTHONHASHSEED=0` в production **не рекомендуется**: атакующий может предсказать хеши и подобрать коллидирующие ключи (хотя SipHash всё равно затрудняет это, но зачем ослаблять защиту).

**Исключение:** если приложение не принимает пользовательский ввод в качестве ключей dict (например, внутренний сервис с доверенными данными), теоретический риск DoS ниже, но лучше всё равно не менять настройки без веской причины.

### sys.hash_info

```python
import sys
print(sys.hash_info)
# hash_info(width=64, modulus=..., inf=314159, nan=0, ...,
#           algorithm='siphash24', hash_bits=64, seed_bits=128, ...)
```

- `algorithm` — `'siphash24'` или аналогичный.
- `seed_bits` — разрядность seed (128 для SipHash).

### Запомните

> **PYTHONHASHSEED=random** — защита от hash-flooding DoS (SipHash + случайный seed). **PYTHONHASHSEED=0** — для воспроизводимых тестов и CI. Порядок dict — порядок вставки (3.7+); порядок set — не гарантирован. Хеши int, float, tuple не зависят от PYTHONHASHSEED.
