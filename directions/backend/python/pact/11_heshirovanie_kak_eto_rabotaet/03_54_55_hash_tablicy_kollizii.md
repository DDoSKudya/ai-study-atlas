### Цель раздела

Понять **внутреннее устройство** хеш-таблиц в CPython: как ключ превращается в индекс, что такое бакеты, load factor, open addressing, perturb-схема и почему dict сохраняет порядок вставки.

### Термины

| Термин | Определение |
|--------|-------------|
| **Хеш-таблица** (hash table) | Структура данных: массив слотов (бакетов); индекс вычисляется из хеша ключа; обеспечивает в среднем O(1) для get/set/delete. |
| **Bucket** (бакет, ячейка) | Один слот в массиве хеш-таблицы; хранит ключ и значение (в dict) или только ключ (в set). |
| **Load factor** (коэффициент загрузки) | Отношение числа занятых слотов к общему размеру таблицы. При слишком высоком load factor — много коллизий, нужен resize. |
| **Open addressing** (открытая адресация) | Стратегия разрешения коллизий: при занятом слоте ищем следующий свободный (probing). В CPython используется именно она. |
| **Chaining** (цепочки) | Альтернатива: в каждом слоте — связный список элементов с одинаковым хешем. |
| **Perturb** (perturbation) | В CPython: при коллизии используют старшие биты хеша для вычисления следующего индекса; уменьшает кластеризацию. |
| **Sparse / Dense** | В CPython 3.6+: dict использует комбинированную структуру: «разреженный» массив индексов + «плотный» массив пар (key, value) для сохранения порядка. |

### Внутренняя структура dict в CPython (упрощённо)

С Python 3.6 (и гарантированно с 3.7) dict хранит элементы в **порядке вставки**. Для этого используются две структуры:

1. **Indices** (индексы) — массив размером, равным степени двойки. В каждой ячейке — индекс в массиве entries или маркер «пусто».
2. **Entries** (записи) — массив пар `(hash, key, value)` в порядке вставки. При итерации обходят именно entries, поэтому порядок гарантирован.

**Поиск по ключу (упрощённо):**
1. Вычислить `i = hash(key) & (size - 1)` (размер — степень двойки; `& (size-1)` эквивалентно `% size`, но быстрее).
2. Взять `indices[i]` — это индекс в entries.
3. Если слот пуст (маркер) — ключ не найден.
4. Если слот занят — проверить `key == entries[idx].key`. Если да — вернуть значение. Если нет (коллизия) — perturb: сдвинуть h, пересчитать i, повторить.

**Схема (концептуально):**
```
indices:  [ ... ][ idx_5 ][ ... ][ idx_2 ][ ... ]   ← массив индексов
                ↑                    ↑
entries:  [ (hash, key, val) ][ (hash, key, val) ]  ← плотный массив в порядке вставки
           idx_2              idx_5
```

### Perturb-схема (упрощённое описание)

При коллизии CPython не просто делает `(i + 1) % size` (linear probing), а использует **perturbation**:

```python
# Псевдокод (по мотивам dictobject.c)
def lookup(key, table):
    h = hash(key)
    i = h & mask  # mask = len(table) - 1
    while table[i] is not EMPTY:
        if table[i].key == key:
            return table[i].value
        # Perturb: сдвигаем h вправо, добавляем в i
        h >>= PERTURB_SHIFT  # например, 5
        i = (i * 5 + 1 + h) & mask
    return NOT_FOUND
```

**Зачем perturb:** уменьшает **первичную кластеризацию** — когда коллизии образуют длинные цепочки подряд. Старшие биты хеша «разбрасывают» следующие пробы по таблице.

**Конкретные значения в CPython:** `PERTURB_SHIFT` обычно 5. На каждой итерации `h >>= 5` — старшие биты хеша «входят» в расчёт следующего индекса. Формула `i = (i * 5 + 1 + h) & mask` даёт псевдослучайное смещение, зависящее от хеша; при linear probing `i+1` следующий индекс зависел бы только от позиции, а не от самого ключа.

### Размер таблицы и load factor

- Начальный размер dict — **8** слотов (для пустого `{}` выделяется минимальная структура).
- Resize срабатывает, когда число элементов превышает порог (в CPython используется формула с учётом размера; ориентировочно load factor ~0.66–0.75).
- Новый размер — следующая степень двойки: 8 → 16 → 32 → 64 → 128 → ... — достаточная для размещения всех элементов без переполнения.
- При resize все пары (key, value) перехешируются и размещаются в новой таблице; старые «удалённые» слоты при этом очищаются.

### set — аналогично dict

`set` в CPython реализован поверх той же структуры, что и dict, но **без** значений — только ключи. Порядок итерации set **не гарантирован** (зависит от хешей и истории вставок/удалений), в отличие от dict.

### Примеры

```python
# dict и set — это хеш-таблицы
d = {"a": 1, "b": 2}
# Внутри: indices + entries; hash("a") → индекс в indices → запись в entries
# При d["a"]: hash("a") → индекс → проверка ключа (==) → значение

s = {1, 2, 3}
# Аналогично: только ключи, без значений

# Порядок в dict гарантирован с Python 3.7 (порядок вставки)
d = {}
d["first"] = 1
d["second"] = 2
print(list(d.keys()))  # ['first', 'second']

# Размер и выделение памяти (детали зависят от реализации)
import sys
d = {i: i for i in range(10)}
print(sys.getsizeof(d))  # размер объекта dict в байтах
```

### Производительность: O(1) vs O(n)

| Операция | Средний случай | Худший случай |
|----------|----------------|---------------|
| `d[key]` | O(1) | O(n) при многих коллизиях |
| `d[key] = v` | O(1) | O(n) |
| `del d[key]` | O(1) | O(n) |
| `key in d` | O(1) | O(n) |
| Итерация | O(n) | O(n) |

### Когда хеш может быть «медленным»

- **Длинные строки:** SipHash обрабатывает каждый байт; хеш строки длины n — O(n). Для очень длинных строк (мегабайты) это заметно.
- **Глубокие кортежи:** хеш tuple — комбинация хешей элементов; глубоко вложенные структуры требуют рекурсивного хеширования.
- **Частый resize:** при массовой вставке dict многократно увеличивается; каждый resize — O(n) на перехеширование. Создание dict из N элементов — O(n) в среднем, но с константным множителем из-за resize.

### Рекомендации по производительности

- **Ключи dict:** предпочитайте короткие строки, int, tuple малой длины. Длинные строки как ключи — приемлемо, но хеш O(n) по длине.
- **Избегайте вложенных структур как ключей:** `{(1, (2, (3, (4, ...))))}` — каждый уровень вызывает hash рекурсивно.
- **Предварительный resize:** если знаете размер, можно создать dict с нужным числом элементов сразу: `dict.fromkeys(range(n), None)` — один resize вместо нескольких.

### Теоретическое подтверждение

**Почему O(1) в среднем:**

- При хорошей хеш-функции (SipHash для str) и умеренном load factor (< 2/3) большинство ключей попадают в свой слот без коллизий.
- Perturb снижает кластеризацию; число проб при коллизии ограничено.
- При resize таблица остаётся «разряженной», коллизий немного.

**Почему худший случай O(n):**

- При намеренно подобранных ключах (hash-flooding DoS) можно создать много коллизий.
- Все элементы попадут в длинную цепочку проб — поиск выродится в линейный. SipHash и PYTHONHASHSEED защищают от этого.

### Запомните

> **dict** и **set** — хеш-таблицы с open addressing и perturb. dict с 3.7 хранит порядок вставки (indices + entries). Load factor ~2/3, при превышении — resize. В среднем O(1), в худшем O(n).

## 55. Коллизии и перехеширование

### Цель раздела

Понять, что такое коллизия, **почему она неизбежна**, как её разрешают (chaining vs open addressing, linear vs quadratic vs perturb), и что происходит при resize — с точки зрения алгоритма и асимптотики.

### Термины

| Термин | Определение |
|--------|-------------|
| **Коллизия** | Ситуация, когда два разных ключа дают один и тот же индекс в хеш-таблице (либо одинаковый хеш, либо разные хеши, но `hash % size` совпадает). |
| **Chaining** | В каждом слоте — связный список (или другая структура) элементов; коллизии разрешаются списком. Каждый слот может хранить много элементов. |
| **Open addressing** | Слот — один элемент; при коллизии ищем **следующий свободный** слот (probing). Все элементы хранятся в самой таблице. |
| **Linear probing** | Следующий слот = `(index + 1) % size`. Просто, но даёт первичную кластеризацию. |
| **Quadratic probing** | Следующий слот = `(index + i²) % size` для i = 1, 2, 3, ... Уменьшает кластеризацию. |
| **Perturbation** | Использование старших бит хеша при вычислении следующего индекса; CPython использует вариант этой идеи. |
| **Resize** (перехеширование) | Увеличение (или уменьшение) размера таблицы; все элементы заново размещаются; load factor снижается. |
| **Кластеризация** | Образование «сгустков» занятых слотов подряд; ухудшает производительность при linear probing. |

### Два подхода к разрешению коллизий

#### Chaining (цепочки)

- В каждом бакете — связный список (или дерево при большом числе коллизий).
- Плюсы: простота, возможность хранить сколько угодно коллизий в одном бакете.
- Минусы: дополнительные аллокации, хуже locality of reference (кэш).

#### Open addressing (открытая адресация)

- Все элементы в самой таблице; при коллизии ищем другой слот.
- Плюсы: нет дополнительных структур, лучше использование кэша.
- Минусы: load factor должен быть < 1; при удалениях нужна осторожность (маркеры «удалён»).

CPython использует **open addressing** с perturb-схемой.

### Почему коллизия неизбежна: принцип Дирихле

**Принцип ящиков ( pigeonhole principle):** если n+1 объектов разместить в n ящиках, минимум в одном ящике будет не меньше 2 объектов.

**Применительно к хеш-таблице:**
- Число слотов конечно (например, 8, 16, 32, ...).
- Число возможных ключей бесконечно (или очень велико): строки любой длины, числа, комбинации.
- Значит, по принципу Дирихле коллизии **гарантированы** при достаточном числе вставок.

**Вывод:** коллизия — не баг, а норма. Хеш-таблица **обязана** корректно её обрабатывать.

### Почему perturb, а не просто (i+1)

При **linear probing** (`i+1`, `i+2`, ...) коллизии образуют длинные цепочки подряд — **первичная кластеризация**. Следующий ключ с коллизией с высокой вероятностью попадёт в конец цепочки → цепочка растёт. Среднее число проб растёт быстрее, чем при равномерном распределении.

**Perturb** использует старшие биты хеша: при каждой следующей пробе хеш сдвигается, и новый индекс «разбрасывается» по таблице. Цепочки не слипаются в один длинный кластер. Это уменьшает вторичную кластеризацию и улучшает среднее время поиска.

### Математика: load factor и число проб

При open addressing с равномерным распределением:

- **Среднее число проб** при поиске (успешном и неуспешном) приближённо пропорционально `1 / (1 - α)`, где α — load factor.
- При α = 0.5: около 2 проб в среднем.
- При α = 2/3: около 3 проб.
- При α → 1: число проб стремится к бесконечности — производительность падает.

**Числовой пример:** таблица из 16 слотов, 10 элементов (α ≈ 0.625). Среднее число проб при поиске ≈ 1/(1-0.625) ≈ 2.7. При 14 элементах (α ≈ 0.875) уже ≈ 8 проб — поэтому CPython делает resize до того, как α станет слишком большим.

Поэтому CPython держит load factor ниже порога (например, 2/3) и делает **resize**, когда таблица заполняется.

### Resize: когда и как

1. **Триггер:** число элементов превышает порог (например, `2/3 * size` или похожее).
2. **Новый размер:** обычно следующая степень двойки, достаточная для размещения всех элементов (часто `new_size = old_size * 2` или больше).
3. **Перехеширование:** для каждой пары (key, value) вычисляется новый индекс в новой таблице; элементы размещаются заново.
4. **Уменьшение:** CPython **не** уменьшает размер dict при удалении элементов. После массового удаления таблица остаётся того же размера; память не освобождается. Другие реализации (PyPy и др.) могут вести себя иначе.

### Удаление при open addressing

При удалении элемента слот нельзя просто пометить «пусто» — иначе разорвётся цепочка проб, и последующие элементы станут «невидимыми». Поэтому используют маркер **«удалён» (dummy)**: слот считается занятым для целей пробинга, но пустым для вставки. При следующем resize такие слоты очищаются.

**Пример (концептуально):**
- Таблица: `[A, B, C, пусто, пусто]`. A, B, C коллидировали и заняли слоты 0, 1, 2.
- Удаляем B. Если пометить слот 1 «пусто», то при поиске C: начальный индекс C — 0, слот 0 = A (не C), слот 1 = пусто → поиск остановится, C не найден (хотя он в слоте 2).
- С маркером «удалён»: слот 1 = dummy. Поиск C: 0 → A, 1 → dummy (продолжаем), 2 → C. Найден.

### Примеры

```python
# Коллизии могут возникать даже у «хороших» ключей
# Разные строки обычно имеют разный хеш, но коллизии возможны
a, b = "abc", "abd"
print(hash(a), hash(b))  # Обычно разные

# Искусственная коллизия (зависит от размера таблицы и хеш-функции)
# hash(x) % 8 может совпасть для разных x
for i in range(20):
    print(i, hash(i) % 8)  # Некоторые остатки совпадут

# Resize при добавлении многих элементов
d = {}
for i in range(1000):
    d[i] = i
# Внутри произошло несколько resize: 8 → 16 → 32 → 64 → 128 → 256 → 512 → 1024
```

### Теоретическое подтверждение

**Принцип Дирихле:** при числе ключей > числа слотов коллизия гарантирована.

**Load factor:** при α = 2/3 среднее число проб ≈ 3; при α = 0.9 оно резко растёт. Resize поддерживает α в приемлемых пределах.

**Худший случай O(n):** при злонамеренно подобранных ключах (hash-flooding) можно добиться, что все элементы попадут в одну длинную цепочку проб. SipHash и PYTHONHASHSEED делают такую атаку практически нереализуемой.

### Запомните

> **Коллизия** — норма, не ошибка. В CPython — open addressing с perturb; при load factor > порога — resize. При удалении слот может помечаться «удалён». Порядок в dict (3.7+) сохраняется благодаря структуре indices + entries.

## 56. Слабые ссылки и хеши
