## §28. Виртуальные окружения

**Цель раздела:** создавать и активировать виртуальные окружения (venv), понимать **PEP 668** (externally managed environment) и зачем не устанавливать пакеты в системный Python, пользоваться pip (установка, freeze, requirements.txt), pip-tools для детерминированных зависимостей, ensurepip и zipapp.

---

### Термины

| Термин | Определение |
|--------|-------------|
| **Виртуальное окружение (virtual environment)** | Изолированная директория с собственным интерпретатором Python и каталогом установленных пакетов (site-packages); зависимости одного проекта не смешиваются с системными и с другими проектами. |
| **venv** | Модуль стандартной библиотеки Python 3 (с 3.3) для создания виртуальных окружений; создаёт папку с копией/симлинком интерпретатора и подкаталогом для пакетов. |
| **PEP 668** | «Marking Python base environments as “externally managed”» — помечает системный (дистрибутивный) Python как управляемый извне; `pip install` в такой среде блокируется, чтобы не ломать пакеты ОС; ожидается использование venv или conda. |
| **Externally managed environment** | Среда, которую управляет не пользователь (например, пакетный менеджер ОС); в таких средах pip отказывается устанавливать пакеты без явного обхода (--break-system-packages), чтобы избежать конфликтов. |
| **pip** | Менеджер пакетов Python; устанавливает пакеты из PyPI (и других индексов) в текущее окружение. |
| **pip-tools** | Набор утилит (pip-compile, pip-sync) для получения детерминированного списка зависимостей из верхнеуровневого перечня (например, requirements.in → requirements.txt) и синхронизации окружения с этим списком. |
| **zipapp** | Стандартный модуль для упаковки приложения в один файл .pyz (ZIP-архив с точкой входа); удобно распространять скрипт с зависимостями внутри архива. |

### Создание и активация venv

**Создание окружения:**

```bash
python3 -m venv .venv
```

Имя `.venv` — распространённая договорённость (часто добавляют в `.gitignore` саму папку, но не шаблон). Можно использовать любое имя, например `venv`, `env`. Команда **должна** вызываться с тем интерпретатором, версию которого вы хотите в окружении: `python3.11 -m venv .venv` создаёт окружение с Python 3.11.

**Что создаётся (пошагово):** в каталоге `.venv` появляется следующая структура.

| Путь (Unix) | Назначение |
|-------------|------------|
| `.venv/bin/python` | Ссылка или копия интерпретатора; при запуске `sys.prefix` и `sys.executable` указывают на окружение. |
| `.venv/bin/pip` | Скрипт запуска pip; при вызове использует `python` из этого же каталога, поэтому пакеты ставятся в site-packages окружения. |
| `.venv/bin/activate` | Скрипт активации (source .venv/bin/activate); модифицирует PATH и устанавливает переменную VIRTUAL_ENV. |
| `.venv/lib/python3.x/site-packages/` | Каталог установленных пакетов; сюда попадают пакеты при `pip install`. |
| `.venv/lib/python3.x/site.py` | Настройки путей для этого окружения (используется интерпретатором). |

На Windows вместо `bin/` используется `Scripts/`; там же лежат `python.exe`, `pip.exe`, `activate.bat`, `Activate.ps1`. Без активации вы можете вызывать явно `.venv/bin/python` (Unix) или `.venv\Scripts\python.exe` (Windows) — тогда этот интерпретатор будет использовать site-packages именно этого окружения.

**Активация** — подстановка путей окружения в PATH и переменная `VIRTUAL_ENV`:

- **Unix/macOS (bash/zsh):**
  ```bash
  source .venv/bin/activate
  ```
- **Windows (cmd):**
  ```cmd
  .venv\Scripts\activate.bat
  ```
- **Windows (PowerShell):**
  ```powershell
  .venv\Scripts\Activate.ps1
  ```
  **Важно:** если выполнение скриптов по умолчанию запрещено (ошибка «running scripts is disabled»), установите политику для текущего пользователя: `Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser` (в PowerShell от имени пользователя). Либо запускайте активацию через cmd: `cmd /c ".venv\Scripts\activate.bat && your_command"`.

После активации в терминале обычно появляется префикс `(.venv)`, а команды `python` и `pip` указывают на окружение.

**Проверка активации по шагам:**

1. **Unix:** `which python` — путь должен быть внутри `.venv/bin` (например, `/path/to/project/.venv/bin/python`). `echo $VIRTUAL_ENV` — путь к `.venv`.
2. **Windows (cmd):** `where python` — путь внутри `.venv\Scripts`. `echo %VIRTUAL_ENV%` — путь к `.venv`.
3. **Внутри Python:** `import sys; print(sys.prefix)` — должен вывести полный путь к `.venv`; `print(sys.executable)` — путь к интерпретатору внутри venv.

**Что именно делает скрипт активации (пошагово):**

1. Сохраняет текущий **PATH** в переменную (чтобы восстановить при deactivate).
2. Добавляет в **начало** PATH путь к каталогу с исполняемыми файлами окружения: `.venv/bin` (Unix) или `.venv\Scripts` (Windows). Поэтому при вводе `python` или `pip` первым в PATH оказывается именно исполняемый файл из venv.
3. Устанавливает переменную окружения **VIRTUAL_ENV** в путь к корню venv (например, `/path/to/project/.venv`). Инструменты и скрипты могут проверять её, чтобы понять, что выполнение идёт в виртуальном окружении.
4. На некоторых платформах отключает или модифицирует переменную **PYTHONHOME**, чтобы интерпретатор не искал библиотеки в системном prefix.
5. Определяет функцию или алиас **deactivate**, которая при вызове восстанавливает прежний PATH, сбрасывает VIRTUAL_ENV и удаляет себя.

Проверить, что окружение активно, можно так: `echo $VIRTUAL_ENV` (Unix) или `echo %VIRTUAL_ENV%` (Windows cmd) — должен вывести путь к .venv; `which python` (Unix) — должен показать путь внутри .venv.

**Деактивация:** выполнить команду `deactivate` (она появляется в окружении после активации). После деактивации PATH и VIRTUAL_ENV возвращаются к прежним значениям.

**Пошаговый сценарий: первый запуск проекта (с нуля):**

1. Перейти в каталог проекта: `cd /path/to/project`.
2. Создать окружение: `python3 -m venv .venv` (или `python3.11 -m venv .venv` для конкретной версии).
3. Активировать: `source .venv/bin/activate` (Unix) или `.venv\Scripts\Activate.ps1` (Windows PowerShell).
4. Обновить pip: `pip install --upgrade pip` (в новых venv часто старый pip).
5. Если есть `requirements.txt`: `pip install -r requirements.txt`. Если проект использует pip-tools: `pip install pip-tools`, затем `pip-sync requirements.txt`.
6. Если проект с pyproject.toml: `pip install -e ".[dev]"` для установки пакета в режиме разработки с dev-зависимостями.
7. Запустить тесты: `pytest` или `python -m pytest`.
8. При выходе из каталога в новом терминале — снова выполнить шаги 3–7 (активация нужна в каждом новом окне терминала).

**Дополнительные опции pip install (кратко):**

| Опция | Назначение |
|-------|------------|
| `--user` | Установить в пользовательский каталог (не в venv); в venv обычно не используют. |
| `--target dir` | Установить пакеты в указанный каталог (удобно для изолированного набора библиотек). |
| `--no-deps` | Не устанавливать зависимости (только указанный пакет). |
| `--dry-run` | Показать, что было бы установлено, без реальной установки. |
| `-v` / `-vv` | Подробный вывод (уровень детализации). |

#### Параметры venv (важные опции)

| Опция | Назначение |
|-------|------------|
| `--without-pip` | Не устанавливать pip в окружение; полезно, если pip будет ставиться отдельно или через ensurepip. |
| `--clear` | Если каталог окружения уже существует — очистить его и создать заново. |
| `--copies` | Копировать файлы интерпретатора вместо симлинков (по умолчанию на некоторых системах создаются симлинки). |
| `--symlinks` | Использовать симлинки на интерпретатор (если ОС поддерживает); экономит место. |

**Когда использовать --copies:** на некоторых системах (например, Windows без прав на создание симлинков или сетевые диски) симлинки могут не создаваться или работать некорректно; тогда явно указывают `--copies`. Копии занимают больше места, но окружение переносимо (можно скопировать .venv на другую машину при той же ОС и пути). **Когда использовать --symlinks:** на Unix при локальном диске симлинки экономят место и ускоряют создание venv; при обновлении системного Python симлинки в venv продолжают указывать на старый путь — может понадобиться пересоздать venv.

Примеры:

```bash
python3 -m venv .venv --without-pip
python3 -m venv .venv --clear
```

После `--without-pip` pip в окружении не будет; его можно добавить: `python -m ensurepip --upgrade` (вызывать из активированного окружения или явно: `.venv/bin/python -m ensurepip --upgrade`).

**Запомните:** без активации команда `python` в терминале — системная; чтобы использовать окружение, его нужно активировать в каждом новом терминале или вызывать явно: `.venv/bin/python` (Unix), `.venv\Scripts\python.exe` (Windows).

### PEP 668 и «externally managed» окружения

На многих дистрибутивах Linux (например, Fedora 37+, Ubuntu 23.04+, Debian 12+) системный Python помечен как **externally managed**. Дистрибутив создаёт в каталоге **sys.prefix** (то есть в корне установки этого Python) специальный файл — маркер. Имя и формат маркера описаны в PEP 668: обычно это файл **`EXTERNALLY-MANAGED`** в каталоге, который для системного Python совпадает с путём установки дистрибутивного Python (например, `/usr` или `/usr/local`). Внутри файла — метаданные в INI-подобном формате; сам факт наличия файла говорит pip: «эта среда управляется извне, не устанавливай сюда пакеты».

**Почему pip отказывается:** пакеты, установленные через apt/dnf/yum, рассчитаны на конкретные версии библиотек. Если пользователь установит через pip другую версию того же пакета (или пакет, конфликтующий с зависимостями ОС), это может сломать системные скрипты или обновления. Поэтому pip при обнаружении маркера **прерывает** установку и выводит сообщение об ошибке с рекомендацией использовать venv или conda.

**Где искать маркер:** в каталоге **sys.prefix** того Python, которым вызван pip. Для системного Python это часто `/usr` или `/usr/local`. Файл может называться `EXTERNALLY-MANAGED` (PEP 668). Проверить в Python: `import sys; print(sys.prefix)` — затем посмотреть наличие такого файла в этом каталоге.

**Типичное сообщение об ошибке:**

```
error: externally-managed-environment

× This environment is externally managed
╰─> To install Python packages system-wide, try ...
    ...
    and create an environment with the tool's recommended method.
```

**Что делать по шагам:**

1. **Не** использовать обход `pip install --break-system-packages` в повседневной работе — это отключает защиту и может привести к поломке зависимостей ОС.
2. Создать виртуальное окружение в каталоге проекта: `python3 -m venv .venv`.
3. Активировать его: `source .venv/bin/activate` (Unix) или `.venv\Scripts\activate` (Windows).
4. Устанавливать пакеты уже в активированном окружении: `pip install requests` и т.д.

Альтернатива — **conda** или другой менеджер окружений: они используют свой каталог (не системный prefix), поэтому маркер PEP 668 на них не распространяется.

**Запомните:** на современных Linux-дистрибутивах считайте правило «всегда работать в venv или conda» обязательным; PEP 668 формализует эту практику и защищает системный Python от конфликтов с pip.

### pip: установка, freeze, requirements.txt

**Установка пакетов** (в активированном venv):

```bash
pip install requests
pip install requests==2.28.0
pip install "requests>=2.28,<3"
```

**Сохранение текущего набора пакетов** — команда `pip freeze` выводит все установленные пакеты в формате `name==version`:

```bash
pip freeze > requirements.txt
```

**Восстановление окружения по файлу:**

```bash
pip install -r requirements.txt
```

**Практика:** в `requirements.txt` обычно кладут полный замороженный вывод `pip freeze` для **воспроизводимости** (одинаковые версии у всех и в CI). Верхнеуровневые зависимости (что реально импортирует проект) часто ведут в отдельном файле (например, `requirements.in`) и затем генерируют `requirements.txt` с помощью **pip-tools** (см. ниже).

#### Дополнительные команды и опции pip

| Команда / опция | Назначение |
|-----------------|------------|
| `pip list` | Список установленных пакетов (с опцией `--outdated` — только устаревшие). |
| `pip show <package>` | Подробная информация о пакете: версия, расположение, зависимости. |
| `pip uninstall <package>` | Удаление пакета (с `-y` — без подтверждения). |
| `pip install --upgrade pip` | Обновить сам pip; в новом venv часто нужен более свежий pip. |
| `pip install --no-cache-dir ...` | Не использовать кеш при загрузке; полезно в CI для воспроизводимости. |
| `pip install -e <path>` | Установить пакет в режиме editable из локального пути (каталог с setup.py или pyproject.toml). |
| `pip install --constraint constraints.txt ...` | Установить пакеты с учётом версий из файла ограничений (аналог частичного lock). |
| `pip download -r requirements.txt -d ./wheels` | Скачать все пакеты из requirements.txt в каталог (например, для офлайн-установки). |
| `pip install --no-index --find-links ./wheels -r requirements.txt` | Установить из локального каталога wheels без обращения к PyPI. |

Пример офлайн-установки (сначала на машине с интернетом, затем без):

```bash
pip download -r requirements.txt -d ./wheels
# перенос каталога wheels на целевую машину
pip install --no-index --find-links ./wheels -r requirements.txt
```

**Специфика requirements.txt (по шагам):**

- **Строка с версией:** `requests==2.31.0` — точная версия; `requests>=2.28,<3` — диапазон; `requests` — любая версия (не рекомендуется для воспроизводимости).
- **Директивы для pip:** в начале файла можно указать `--extra-index-url https://...`, `--no-index`, `--find-links ./wheels`; pip применит их ко всем последующим строкам.
- **Комментарии:** строки, начинающиеся с `#`, игнорируются; пустые строки — тоже.
- **Редактирование вручную:** допустимо, но при использовании pip-tools версии в requirements.txt не правят вручную — их пересобирают через `pip-compile requirements.in`.

**Кеш pip:** по умолчанию pip кеширует скачанные пакеты (на Unix: `~/.cache/pip`; на Windows: `%LOCALAPPDATA%\pip\Cache`). Команда `pip cache dir` показывает путь к кешу; `pip cache purge` — очистить кеш. В CI часто используют `pip install --no-cache-dir`, чтобы не зависеть от кеша и экономить место.

**Конфигурация pip (pip.conf / pip.ini):** настройки можно задать в файле, чтобы не указывать опции в каждой команде. **Расположение:** пользовательский — на Unix `~/.config/pip/pip.conf` или `~/.pip/pip.conf`, на Windows `%APPDATA%\pip\pip.ini`; проект — `pip.conf` в корне проекта; глобальный — в зависимости от ОС (например, `/etc/pip.conf`). **Пример содержимого:**

```ini
[global]
index-url = https://pypi.org/simple/
extra-index-url = https://my-private-index/simple/
trusted-host = pypi.org my-private-index
timeout = 60
no-cache-dir = false

[install]
no-deps = false
```

Переменные окружения переопределяют конфиг: **PIP_INDEX_URL**, **PIP_EXTRA_INDEX_URL**, **PIP_TRUSTED_HOST**, **PIP_NO_CACHE_DIR=1**. В CI часто задают **PIP_INDEX_URL** и **PIP_EXTRA_INDEX_URL** секретами.

**Запомните:** для воспроизводимости всегда фиксируйте версии (pip freeze или pip-tools); в CI часто используют `--no-cache-dir` и установку из lock-файла.

### pip-tools: детерминированные зависимости

**Задача:** в `requirements.txt` при повторной установке могут попадать разные версии транзитивных зависимостей; сборки становятся невоспроизводимыми.

**pip-tools** даёт:
- **pip-compile** — по файлу верхнеуровневых зависимостей (например, `requirements.in`) генерирует полный `requirements.txt` с зафиксированными версиями всех пакетов.
- **pip-sync** — ставит в окружении ровно те пакеты и версии, что в `requirements.txt`, и удаляет лишние.

**Установка:** `pip install pip-tools` (лучше в том же venv, где будете использовать).

**Пример минимального workflow.**

Файл `requirements.in` (только верхнеуровневые зависимости проекта):

```
requests>=2.28
pytest>=7.0
```

Команда:

```bash
pip-compile requirements.in
```

Создаётся или обновляется `requirements.txt` с **полным деревом**: все транзитивные зависимости (urllib3, certifi, ...) с **точными** версиями (например, `requests==2.31.0`, `urllib3==2.0.7`). В CI и у всех разработчиков затем:

```bash
pip-sync requirements.txt
```

`pip-sync` устанавливает ровно те пакеты и версии, что в `requirements.txt`, и **удаляет** из окружения всё, чего нет в файле. Итог: окружение полностью совпадает с файлом.

#### Опции pip-compile (важные)

| Опция | Назначение |
|-------|------------|
| `--generate-hashes` | Добавить в requirements.txt хеши (--hash=sha256:...) для проверки целостности; повышает безопасность в строгих средах. |
| `--allow-unsafe` | Включить в вывод «небезопасные» зависимости (например, setuptools), которые pip-compile по умолчанию не фиксирует. |
| `--upgrade` | Обновить все пакеты до последних версий, удовлетворяющих ограничениям из .in. |
| `--upgrade-package <pkg>` | Обновить только указанный пакет. |
| `-o file` / `--output-file file` | Записать вывод в указанный файл (по умолчанию имя .in с расширением .txt). |
| Несколько .in файлов | Можно компилировать несколько файлов: `pip-compile requirements.in dev-requirements.in` — зависимости объединяются в один вывод (или в отдельные .txt по желанию). |

**Порядок разрешения зависимостей:** pip-compile использует тот же резолвер, что и pip; при конфликте версий (два пакета требуют несовместимые версии третьего) pip-compile выдаст ошибку. Решение: ослабить ограничения в .in или зафиксировать версию конфликтующего пакета в .in.

Пример с хешами (для строгой воспроизводимости):

```bash
pip-compile requirements.in --generate-hashes
```

**Граничный случай:** если в `requirements.in` указать только `requests`, а потом добавить туда `pytest`, после повторного `pip-compile` в `requirements.txt` появятся новые строки для pytest и его зависимостей; старые строки для requests сохранятся (версии не меняются, если ограничения выполнены). Чтобы подтянуть новые версии всех пакетов, запускают `pip-compile --upgrade`.

**Несколько входных файлов (dev / prod):**

Часто разделяют зависимости на «продакшен» и «разработка»:

- `requirements.in` — только зависимости, нужные в рантайме (requests, django и т.д.).
- `requirements-dev.in` — включает `-r requirements.in` и добавляет pytest, mypy, ruff и т.д.

Пример `requirements-dev.in`:

```
-c requirements.txt
-r requirements.in
pytest>=7.0
mypy>=1.0
ruff>=0.1
```

Команда: `pip-compile requirements.in` (создаёт requirements.txt), затем `pip-compile requirements-dev.in` (создаёт requirements-dev.txt; опция `-c requirements.txt` подтягивает ограничения из requirements.txt, чтобы версии совпадали). В CI для тестов устанавливают: `pip-sync requirements-dev.txt`; для деплоя — `pip-sync requirements.txt`.

**Имена выходных файлов:** по умолчанию `pip-compile requirements.in` пишет в `requirements.txt`. Другой вывод: `pip-compile requirements.in -o requirements-prod.txt`. Для нескольких .in с разными выводами: `pip-compile base.in -o base.txt`, `pip-compile dev.in -o dev.txt`.

**Запомните:** pip-compile + pip-sync помогают получать одинаковое окружение везде; версии в requirements.txt не правят вручную, их пересобирают через pip-compile при изменении requirements.in.

### ensurepip — установка pip в venv

В некоторых сборках Python (например, минимальная сборка под Linux или встроенная в ОС) модуль **pip** может не быть установлен в venv по умолчанию. Модуль **ensurepip** (стандартная библиотека с Python 3.4) устанавливает pip и setuptools в текущее окружение, используя встроенные в интерпретатор копии wheel’ов pip/setuptools (они не требуют сети).

**Вызов:**

```bash
python -m ensurepip --upgrade
```

- Без аргументов — устанавливает pip (и setuptools) в текущее окружение (то есть в тот Python, которым вы вызвали команду). Поэтому команду нужно выполнять **из активированного venv** или явно: `.venv/bin/python -m ensurepip --upgrade`.
- **--upgrade** — обновить pip/setuptools до версий, встроенных в интерпретатор (если уже установлены старые).
- **--default-pip** — устаревший вариант; в современных версиях не нужен.

**Когда нужен ensurepip:** если вы создали venv с флагом **--without-pip** (например, для минимального окружения или для воспроизводимой установки pip отдельно), после создания окружения вызовите ensurepip или установите pip вручную через get-pip.py. При обычном `python -m venv .venv` (без --without-pip) pip в окружении уже есть — venv при создании сам вызывает логику, аналогичную ensurepip.

**Граничные случаи ensurepip:**

- **Ошибка «Could not find a version that satisfies the requirement pip»:** ensurepip не использует сеть; ошибка обычно связана с тем, что вызывается не тот Python (например, системный вместо .venv/bin/python). Решение: явно вызвать `.venv/bin/python -m ensurepip --upgrade` (Unix) или `.venv\Scripts\python.exe -m ensurepip --upgrade` (Windows).
- **Минимальная сборка Python без ensurepip:** в некоторых сборках (например, встроенных в ОС или собранных с отключённым ensurepip) модуль ensurepip отсутствует. Решение: скачать get-pip.py с https://bootstrap.pypa.io/get-pip.py и выполнить `python get-pip.py` в активированном venv.
- **Обновление pip после ensurepip:** после установки pip через ensurepip версия pip — та, что встроена в интерпретатор. Для более новой версии: `python -m pip install --upgrade pip` (уже требует работающий pip).

**Запомните:** ensurepip ставит pip без доступа к сети, из встроенных в Python wheel’ов; используйте его после создания venv с --without-pip.

### venv: программное создание и virtualenv

**Программное создание окружения** (без вызова команды в shell):

```python
import venv
builder = venv.EnvBuilder(system_site_packages=False, clear=False)
builder.create(".venv")
```

**EnvBuilder** — тот же механизм, что и `python -m venv`; параметры: `system_site_packages` (доступ к системным site-packages), `clear` (очистить каталог перед созданием), `symlinks` (Unix: использовать симлинки на интерпретатор) и др.

**virtualenv** — сторонний пакет, расширенная альтернатива venv:
- Установка: `pip install virtualenv` (в системный или в уже активированный venv).
- Создание: `virtualenv .venv`; опция `--system-site-packages` — подтянуть пакеты из системы.
- Даёт более «полное» изолированное окружение и поддерживает старые версии Python; в современной практике часто достаточно встроенного venv.

**Сравнение venv и virtualenv:**

| Аспект | venv | virtualenv |
|--------|------|------------|
| Входит в Python | Да (с 3.3) | Нет, отдельный пакет |
| Версии Python | Только 3.3+ | Поддержка старых версий (в т.ч. 2.7) |
| Скорость создания | Быстро | Сопоставимо |
| Опции | --without-pip, --clear, --copies, --symlinks | Те же + --system-site-packages, копирование скриптов активации под разные оболочки |
| Рекомендация | Использовать по умолчанию для Python 3 | Когда нужен Python 2.7 или расширенные опции |

**Индексы pip (--index-url, --extra-index-url):** по умолчанию pip обращается к PyPI (https://pypi.org/simple/). Для приватного репозитория или зеркала: `pip install --index-url https://my-index/simple/ mypackage` — все пакеты ищутся только в этом индексе. Чтобы дополнять PyPI другим индексом: `pip install --extra-index-url https://my-index/simple/ mypackage` — сначала поиск на PyPI, затем в дополнительном индексе. В requirements.txt можно указать: `--extra-index-url https://...` в первой строке (директива для pip).

**Конфигурация pip (pip.conf / PIP_*):** настройки pip можно задать в конфиге (пользовательский: `~/.config/pip/pip.conf` на Unix или `%APPDATA%\pip\pip.ini` на Windows; проект: `pip.conf` в корне проекта; глобальный: в зависимости от ОС). Секция `[global]`: `index-url`, `extra-index-url`, `trusted-host`, `timeout`, `no-cache-dir` и др. Переменные окружения переопределяют конфиг: **PIP_INDEX_URL**, **PIP_EXTRA_INDEX_URL**, **PIP_NO_CACHE_DIR=1**, **PIP_TRUSTED_HOST**. В CI часто задают **PIP_INDEX_URL** и **PIP_EXTRA_INDEX_URL** для приватного индекса.

### zipapp — приложение в одном .pyz файле

Модуль **zipapp** (стандартная библиотека с Python 3.5) упаковывает каталог с Python-кодом в один файл с расширением **.pyz**. По сути это **ZIP-архив** с особым соглашением: при запуске `python myapp.pyz` интерпретатор добавляет содержимое архива в **sys.path** и выполняет точку входа.

**Структура каталога перед упаковкой:**

В каталоге (например, `myapp/`) должны лежать модули приложения. Обязательна **точка входа**:

- Либо файл **`__main__.py`** в корне каталога — при запуске `python myapp.pyz` выполняется именно он.
- Либо при создании архива указывают **`-m`** с именем модуля: `python -m zipapp myapp -o myapp.pyz -m "pkg.main:run"` — тогда точкой входа будет функция `run` в модуле `pkg.main`.

**Создание архива:**

```bash
python -m zipapp myapp -o myapp.pyz
```

Если в `myapp/` есть только `__main__.py` (и при необходимости другие модули), архив создаётся; при запуске выполнится `myapp/__main__.py`.

**Запуск:**

```bash
python myapp.pyz
# или, если задан shebang (см. ниже):
./myapp.pyz
```

**Параметр `-p` (shebang):**

На Unix можно сделать .pyz исполняемым и запускать без явного `python`:

```bash
python -m zipapp myapp -o myapp.pyz -p "/usr/bin/env python3"
chmod +x myapp.pyz
./myapp.pyz
```

Строка `-p "/usr/bin/env python3"` записывается в начало архива как shebang; ОС при запуске `./myapp.pyz` вызовет указанный интерпретатор, который распознаёт .pyz и выполняет код из архива.

**Ограничения и практика:**

- В .pyz по умолчанию попадает **только ваш код** из указанного каталога; **сторонние пакеты** (requests, numpy и т.д.) в архив не включаются. При запуске интерпретатор ищет их в системном site-packages или в активированном venv.
- Чтобы распространять приложение **вместе с зависимостями** в одном файле, используют: **PyInstaller** или **cx_Freeze** (сборка в исполняемый файл или в каталог с зависимостями), либо ручную упаковку: положить зависимости в подкаталог внутри каталога приложения и настроить `sys.path` в `__main__.py`, затем упаковать zipapp’ом (сложнее и редко нужно).
- Типичное использование zipapp — **скрипт без внешних зависимостей** или с зависимостями, которые уже установлены в целевом окружении (venv + requirements.txt).

**Пример с точкой входа через `-m`:**

Каталог `myapp/` содержит пакет `myapp` с модулем `main.py` и функцией `run()`. Создание архива с точкой входа в эту функцию:

```bash
python -m zipapp myapp -o myapp.pyz -m "myapp.main:run"
```

При запуске `python myapp.pyz` интерпретатор импортирует `myapp.main` и вызывает `run()`.

**Дополнительные опции zipapp:**

| Опция | Назначение |
|-------|------------|
| `-o path/to/out.pyz` | Путь к выходному файлу (по умолчанию имя каталога + .pyz в текущей директории). |
| `-m module:callable` | Точка входа: импортировать модуль и вызвать callable (функцию или класс). |
| `-p interpreter` | Shebang для Unix (например, `-p "/usr/bin/env python3"`). |
| `-c` | Сжать архив (по умолчанию сжатие включено). |

**Запомните:** zipapp упаковывает только указанный каталог с кодом; точка входа — `__main__.py` или `-m package.module:callable`; зависимости в архив не входят, для «всё в одном» используют PyInstaller или аналог.

#### Теоретическое обоснование: зачем виртуальные окружения

Без изоляции все пакеты устанавливаются в **один** каталог (системный site-packages или пользовательский). Недостатки:

1. **Конфликт версий:** проект A требует `requests==2.28`, проект B — `requests==2.31`; в одном окружении может быть только одна версия.
2. **Зависимости ОС:** многие дистрибутивы Linux поставляют пакеты (в т.ч. Python-модули) через apt/dnf. Установка через pip в тот же prefix может перезаписать или конфликтовать с пакетами ОС; обновление системы или пакета может сломать другой.
3. **Воспроизводимость:** на другой машине или в CI набор установленных пакетов может отличаться; без фиксации версий (requirements.txt, lock) сборка и тесты ведут себя по-разному.

**Виртуальное окружение** даёт каждому проекту свой каталог site-packages и (при необходимости) свой интерпретатор. Зависимости проекта не смешиваются ни с системой, ни с другими проектами. PEP 668 формализует правило «не трогать системный Python»: дистрибутивы помечают его как externally managed, и pip отказывается устанавливать пакеты туда, подталкивая к использованию venv или conda.

#### Граничные случаи и отладка (§28)

| Ситуация | Что проверить | Решение |
|----------|----------------|---------|
| После активации `python` всё ещё системный | PATH: в начале должен быть путь к .venv/bin (Unix) или .venv\Scripts (Windows). | Проверить вывод `which python` (Unix) / `where python` (Windows); при необходимости активировать заново: `source .venv/bin/activate`. |
| pip ставит пакеты «не туда» | Какой pip вызывается: `which pip` (Unix). | Убедиться, что используется pip из активированного venv; при сомнении вызывать явно: `.venv/bin/pip install ...`. |
| В новом venv нет pip | Сборка Python без ensurepip или venv создан с --without-pip. | Выполнить `python -m ensurepip --upgrade` (из активированного окружения или .venv/bin/python). |
| pip-compile не обновляет версии | Ограничения в requirements.in могут допускать только старые версии. | Запустить `pip-compile --upgrade requirements.in` для обновления всех пакетов в рамках ограничений. |
| zipapp не запускается | В каталоге нет __main__.py или не указан -m. | Добавить __main__.py в корень каталога приложения или указать точку входа: `-m "pkg.main:run"`. |
| PowerShell: «running scripts is disabled» | Политика выполнения скриптов по умолчанию Restricted. | `Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser` или использовать cmd и activate.bat. |
| После `python -m venv .venv` нет pip | Минимальная сборка Python без ensurepip или venv создан с --without-pip. | `python -m ensurepip --upgrade` (из активированного venv или `.venv/bin/python -m ensurepip --upgrade`). |
| Несколько версий Python: какая попадёт в venv? | venv создаётся той версией, которой вызван `python` или `python3`. | Явно: `python3.11 -m venv .venv` или `py -3.11 -m venv .venv` (Windows Launcher). |
| pip install падает с «externally-managed-environment» | Системный Python помечен PEP 668 (Linux-дистрибутив). | Не использовать --break-system-packages; создать venv: `python3 -m venv .venv`, активировать, ставить пакеты в venv. |
| Окружение «разъехалось»: странные версии пакетов | Ручные правки requirements.txt, разные pip-compile в разное время. | Пересобрать: `pip-compile requirements.in --upgrade`, затем `pip-sync requirements.txt`; в CI всегда из одного lock-файла. |
| zipapp: ModuleNotFoundError при запуске .pyz | Зависимости не в архиве; в окружении нет пакета. | Установить зависимости в venv: `pip install -r requirements.txt`; либо использовать PyInstaller для «всё в одном». |

#### Частые вопросы (§28)

| Вопрос | Краткий ответ |
|--------|----------------|
| Нужно ли коммитить .venv в git? | Нет. Каталог .venv добавляют в .gitignore; окружение воссоздаётся по requirements.txt или pyproject.toml. |
| Можно ли переименовать .venv? | Да, но после переименования нужно заново активировать (путь в VIRTUAL_ENV и скриптах изменится). |
| Чем pip install -r requirements.txt отличается от pip-sync? | `pip install -r` добавляет/обновляет пакеты из файла, но не удаляет лишние. `pip-sync` приводит окружение в точное соответствие с файлом (удаляет всё, чего нет в файле). |
| Зачем requirements.in, если есть requirements.txt? | .in — только верхнеуровневые зависимости (удобно править); .txt генерируется pip-compile с полным деревом и точными версиями для воспроизводимости. |
| Когда использовать --break-system-packages? | Только в исключительных случаях (например, разовая установка инструмента в системный Python в изолированной среде). В обычной работе — никогда. |
| Как указать pip использовать другой индекс? | `pip install --index-url https://...` или в pip.conf / PIP_INDEX_URL; в requirements.txt — строка `--extra-index-url https://...` в начале. |

**Запомните:** venv изолирует окружение проекта; PEP 668 запрещает установку в системный Python на многих дистрибутивах; pip-tools даёт воспроизводимый набор зависимостей; zipapp упаковывает приложение в один .pyz, но без автоматического включения сторонних пакетов.

## §29. pyproject.toml
