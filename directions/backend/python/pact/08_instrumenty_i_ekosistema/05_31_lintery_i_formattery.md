**Цель раздела:** использовать **Ruff** (линтер и форматтер), при необходимости **Pylint**, **Black** (форматирование кода), **isort** (сортировка импортов); настраивать их через pyproject.toml или отдельные конфиги.

---

### Термины

| Термин | Определение |
|--------|-------------|
| **Линтер** | Инструмент, анализирующий код на ошибки, типичные баги и стилевые нарушения (неиспользуемые переменные, несоответствие PEP 8 и т.д.). |
| **Форматтер** | Инструмент, автоматически переписывающий код в едином стиле (отступы, переносы строк, кавычки); не исправляет логику. |
| **Ruff** | Быстрый линтер (на Rust) с правилами, совместимыми с Flake8, isort, частью Pylint и др.; также умеет форматировать (режим, совместимый с Black). |
| **Black** | «Бескомпромиссный» форматтер; минимум опций (например, длина строки), единый стиль в проекте. |
| **isort** | Сортирует и группирует импорты по правилам (stdlib, third-party, local; алфавит). |

---

### Ruff

**Ruff** — быстрый линтер и форматтер (реализован на Rust). Совместим с правилами Flake8 (E, F, W и др.), isort (I), частью Pylint (C, R и др.) и другими наборами; также умеет форматировать код в стиле Black. Один инструмент заменяет связку Flake8 + isort + Black (и частично Pylint) и работает значительно быстрее.

**Установка:** `pip install ruff`.

**Основные команды:**

| Команда | Назначение |
|---------|------------|
| `ruff check .` | Проверить код на нарушения правил (без исправлений). |
| `ruff check . --fix` | Проверить и применить автоисправления, где возможно. |
| `ruff check . --fix --unsafe-fixes` | Включить «небезопасные» исправления (могут менять поведение). |
| `ruff format .` | Отформатировать код (стиль, совместимый с Black). |
| `ruff check . --output-format=grouped` | Группировать вывод по файлам. |
| `ruff rule E501` | Показать описание правила E501. |

**Категории правил (кратко):**

| Код | Назначение |
|-----|------------|
| E, W | Ошибки и предупреждения из pycodestyle (PEP 8: отступы, пробелы, длина строки и т.д.). |
| F | Pyflakes: неиспользуемые импорты/переменные, неопределённые имена. |
| I | isort: порядок и группировка импортов. |
| N | pep8-naming: соглашения об именовании. |
| D | pydocstyle: стиль docstrings. |
| B | flake8-bugbear: типичные баги и сомнительные конструкции. |
| C, R | Pylint: сложность, стиль кода. |
| S | flake8-bandit: проверки безопасности (пароли, небезопасные функции). |

**Примеры кодов правил (для select/ignore):** E501 — длина строки; F401 — неиспользуемый импорт; F841 — неиспользуемая переменная; I001 — порядок импортов (isort); B008 — не вызывать функции в аргументах по умолчанию; S101 — использование assert (часто отключают в тестах). Полный список: `ruff rule --all` или документация Ruff.

**Примеры нарушений и исправлений (Ruff):**

| Код | Пример нарушения | Как исправить (или ignore) |
|-----|------------------|----------------------------|
| F401 | `import os` в начале файла, но `os` не используется | Удалить импорт или использовать; в `__init__.py` для re-export часто разрешают через per-file-ignores. |
| E501 | Строка длиннее 88 символов (при line-length=88) | Разбить строку или в [tool.ruff] задать ignore = ["E501"] и полагаться на ruff format. |
| B008 | `def f(x=default_factory())` — вызов в значении по умолчанию | Использовать None и присваивание в теле: `def f(x=None): x = x if x is not None else default_factory()`. |
| S101 | В тестах `assert x == 1` — assert может быть отключен при -O | В тестах обычно разрешают: per-file-ignores для tests/* с S101. |

**Конфигурация в pyproject.toml (подробно):**

```toml
[tool.ruff]
line-length = 88
target-version = "py311"
src = ["src", "tests"]
exclude = [".git", "__pycache__", "build"]

[tool.ruff.lint]
select = ["E", "F", "I", "N", "B"]
ignore = ["E501"]
fixable = ["ALL"]
unfixable = []

[tool.ruff.lint.per-file-ignores]
"tests/*" = ["S101"]
```

- **select** — включить наборы правил (E, F, I и т.д.); **ignore** — отключить конкретные правила (например, E501 — длина строки, если форматтер уже переносит).
- **per-file-ignores** — отключить правила для отдельных путей (например, в тестах разрешить assert без сообщения: S101).
- **fixable** — какие правила можно автоисправлять; по умолчанию многие включены.
- **extend** — расширить конфиг из другого файла (например, `extend = "pyproject.toml"` в ruff.toml или наследование от пресета). В pyproject.toml можно разнести конфиг по файлам через `tool.ruff.extend` (если поддерживается).

**Исключение и включение файлов:** **exclude** — глобальные исключения (например, `["build", ".git"]`); **include** — явный список включаемых путей (если задан, проверяются только они). **extend-exclude** дополняет exclude. Для одного правила исключить файл: **per-file-ignores** с путём и кодом правила.

**Запомните:** Ruff — один быстрый инструмент для линтинга и форматирования; конфигурацию удобно держать в [tool.ruff] в pyproject.toml; для совместимости с Black используйте line-length = 88 и ruff format.

### Black и isort

**Black** — «бескомпромиссный» форматтер: минимум опций, единый стиль (отступы, кавычки, переносы строк, пробелы вокруг операторов и т.д.). Не исправляет логику, только форматирование. Опции часто задают в pyproject.toml:

```toml
[tool.black]
line-length = 88
target-version = ['py311']
include = '\.pyi?$'
exclude = '''
/(
    \.git
  | __pycache__
  | build
)/
'''
```

Команды: **`black .`** — переформатировать файлы на месте. **`black --check .`** — только проверить: не менять файлы, выйти с ненулевым кодом, если код не соответствует стилю Black (удобно в CI, чтобы не коммитить автоматические правки). **`black --diff .`** — показать diff между текущим кодом и отформатированным, без записи в файлы (удобно посмотреть, что изменится).

**isort** — сортирует и группирует импорты (stdlib, third-party, local; алфавит). Профили выравнивают настройки с форматтером: `profile = "black"` — совместимость с Black (длина строки, кавычки). Ruff включает правила isort (I), поэтому отдельно isort можно не ставить, если используете Ruff.

### Pylint

**Pylint** — линтер с большим числом правил и проверками качества дизайна (дублирование кода, сложность, именование, docstrings и т.д.). Тяжелее и медленнее Ruff; часто используют выборочно (только для части модулей) или в CI без автофикса. Запуск: `pylint mypkg` или `pylint src/`. Конфигурация: `.pylintrc` или `[tool.pylint.*]` в pyproject.toml (если поддерживается). Ruff покрывает часть правил Pylint (C, R); для глубокого анализа оставляют Pylint.

### Flake8

**Flake8** — классический линтер, объединяющий **pycodestyle** (PEP 8: E, W), **pyflakes** (F: неиспользуемые импорты/переменные, неопределённые имена) и **mccabe** (C: цикломатическая сложность). Даёт только **стиль и базовые ошибки**, без автоисправления; быстрый и предсказуемый. В новых проектах часто заменяют на **Ruff** (те же правила, но быстрее и с --fix); Flake8 по-прежнему полезен в legacy-проектах, в CI без изменения кода или когда нужна минимальная зависимость.

**Когда использовать Flake8 сам по себе:**

- **Старые проекты:** уже настроен .flake8 или setup.cfg; миграция на Ruff не приоритетна.
- **Строго «только проверка»:** Flake8 не меняет файлы; удобно в CI, где правки в коде нежелательны.
- **Минимальный набор правил:** только E (pycodestyle), F (pyflakes), W (warnings), C (mccabe) — без сотен дополнительных правил Ruff/Pylint.

**Установка и запуск:**

```bash
pip install flake8
flake8 src/ tests/
```

**Основные команды:**

| Команда | Назначение |
|---------|------------|
| `flake8 .` | Проверить текущий каталог (и подкаталоги). |
| `flake8 src/ --max-line-length=88` | Проверить каталог; длина строки 88 (совместимость с Black). |
| `flake8 --select=E,F,W --ignore=E501 .` | Включить только E, F, W; отключить E501 (длина строки). |
| `flake8 --count --show-source` | Показать количество нарушений и фрагмент кода. |

**Коды ошибок (кратко):** **E/W** — pycodestyle (отступы, пробелы, длина строки E501, пустые строки и т.д.); **F** — pyflakes (F401 неиспользуемый импорт, F841 неиспользуемая переменная, F821 неопределённое имя); **C** — mccabe (C901 — слишком высокая сложность функции). Полный список: документация Flake8 и pycodestyle.

**Конфигурация:** в корне проекта файл **`.flake8`** или секция `[flake8]` в **setup.cfg** / **tox.ini**:

```ini
[flake8]
max-line-length = 88
extend-ignore = E203, W503
exclude = .git,__pycache__,build,.venv
per-file-ignores =
    __init__.py:F401
    tests/*:F401
```

- **extend-ignore** — добавить к игнорируемым кодам (E203, W503 часто отключают при использовании Black).
- **per-file-ignores** — для указанных путей игнорировать указанные коды (например, в `__init__.py` разрешить неиспользуемые импорты F401, если они нужны для re-export).

**Flake8 vs Ruff:** Ruff реализует совместимые с Flake8 правила (E, F, W и др.) и добавляет isort, часть Pylint, автоисправления; на больших репозиториях Ruff значительно быстрее. Если начинаете новый проект — предпочтительнее Ruff; если поддерживаете старый с Flake8 — можно оставить Flake8 или мигрировать постепенно (Ruff поддерживает те же коды).

#### Граничные случаи и отладка (§31)

| Ситуация | Что проверить | Решение |
|----------|----------------|---------|
| Ruff не находит конфиг | Путь к pyproject.toml или ruff.toml; запуск из подкаталога. | Запускать из корня проекта; Ruff ищет конфиг вверх по дереву; явно указать `--config pyproject.toml`. |
| Ruff исправляет код не так, как ожидается | Правила fixable, --fix --unsafe-fixes. | Отключить автофикс для правила: в ignore добавить код правила; для «небезопасных» исправлений — `--unsafe-fixes` (осторожно). |
| Black и Ruff конфликтуют по форматированию | line-length, стиль переносов. | Использовать line-length = 88 в обоих; Ruff format совместим с Black — можно оставить только Ruff format. |
| Flake8 и Ruff дают разные коды | Ruff реализует подмножество правил Flake8. | Коды совместимы (E, F, W); при миграции с Flake8 на Ruff те же коды можно оставить в ignore/select. |
| Линтер ругается на тестах (assert, фикстуры) | per-file-ignores для tests/. | В [tool.ruff.lint.per-file-ignores] добавить `"tests/*" = ["S101"]` (разрешить assert без сообщения). |

#### Частые вопросы (§31)

| Вопрос | Краткий ответ |
|--------|----------------|
| Нужны ли Black и isort, если есть Ruff? | Ruff включает правила isort (I) и форматтер в стиле Black; для многих проектов достаточно одного Ruff. |
| Как отключить одно правило Ruff? | В [tool.ruff.lint] добавить код правила в ignore: `ignore = ["E501"]`. |
| Чем Pylint отличается от Ruff? | Pylint — больше правил и проверок дизайна (сложность, дублирование); Ruff быстрее и покрывает базовые правила; можно использовать оба (Ruff для быстрой проверки, Pylint выборочно). |

**Запомните:** Ruff — один инструмент для линтинга и форматирования; Black и isort — классические форматтер и сортировка импортов; Flake8 — классический «только проверка» (стиль + ошибки); конфигурацию удобно держать в [tool.ruff] и [tool.black] в pyproject.toml или в отдельном .flake8.

## §31a. Конфигурация инструментов и pre-commit

**Цель подраздела:** настраивать линтеры и типы в pyproject.toml (или mypy.ini, pyrightconfig.json) и запускать проверки автоматически через **pre-commit** перед каждым коммитом.

---

### Конфигурация в pyproject.toml

Конфигурацию инструментов удобно хранить в **одном** файле pyproject.toml в секциях **[tool.\*]** — так не разбрасываются отдельные конфиги по проекту.

**Примеры секций:**

```toml
[tool.mypy]
python_version = "3.11"
strict = true
warn_return_any = true
warn_unused_configs = true
disallow_untyped_defs = true

[[tool.mypy.overrides]]
module = "legacy.*"
ignore_errors = true

[tool.ruff]
line-length = 88
target-version = "py311"

[tool.ruff.lint]
select = ["E", "F", "I"]

[tool.black]
line-length = 88
```

Для **pyright** часто используют отдельный **pyrightconfig.json** в корне проекта (или секцию [tool.pyright], если инструмент её читает):

```json
{
  "include": ["src"],
  "pythonVersion": "3.11",
  "typeCheckingMode": "strict"
}
```

**Запомните:** единый конфиг в pyproject.toml упрощает поддержку; mypy и Ruff читают [tool.mypy] и [tool.ruff] из pyproject.toml автоматически.

#### Альтернативные конфиги (mypy.ini, ruff.toml, pyrightconfig.json)

Инструменты могут читать настройки из **отдельных файлов** вместо pyproject.toml — удобно, если конфиг общий для нескольких языков, или в проектах без pyproject.toml. Приоритет: обычно сначала ищется свой файл (mypy.ini, ruff.toml, pyrightconfig.json), затем секция в pyproject.toml.

**mypy.ini** — INI-формат в корне проекта. Ищется автоматически при запуске `mypy` из каталога проекта.

```ini
[mypy]
python_version = 3.11
strict = true
warn_return_any = true
warn_unused_configs = true
disallow_untyped_defs = true

[mypy-legacy.*]
ignore_errors = true
```

**Основные секции и опции mypy.ini:**

| Секция/опция | Назначение |
|--------------|------------|
| `[mypy]` | Глобальные настройки. |
| `python_version` | Версия Python для проверки (например, 3.11). |
| `strict` | Включить строгий режим (много проверок). |
| `warn_unused_configs` | Предупреждать о неиспользуемых конфигах. |
| `disallow_untyped_defs` | Требовать аннотации у всех функций. |
| `[mypy-имя.модуля]` | Переопределения для модулей (например, `[mypy-tests.*]` — для tests.*). |
| `ignore_errors` | Игнорировать ошибки в этих модулях. |

Если указать и mypy.ini, и [tool.mypy] в pyproject.toml, приоритет зависит от порядка; обычно один источник выбирают и придерживаются его.

**ruff.toml** — TOML-файл в корне (или указать `--config ruff.toml`). Ruff ищет ruff.toml, затем pyproject.toml [tool.ruff].

```toml
line-length = 88
target-version = "py311"
src = ["src", "tests"]
exclude = [".git", "__pycache__"]

[lint]
select = ["E", "F", "I"]
ignore = ["E501"]

[lint.per-file-ignores]
"tests/*" = ["S101"]
```

**Ключевые поля ruff.toml:** на верхнем уровне — **line-length**, **target-version**, **src**, **exclude**; секция **[lint]** — **select**, **ignore**, **fixable**, **per-file-ignores** (аналогично [tool.ruff] и [tool.ruff.lint] в pyproject.toml). В pyproject.toml те же опции задаются под [tool.ruff]; ruff.toml удобен, когда не хотят смешивать конфиг линтера с остальным pyproject.toml или когда Ruff запускают из подкаталога (поиск конфига идёт вверх по дереву). **extend** — можно подтянуть настройки из другого файла: `extend = "pyproject.toml"` в ruff.toml.

**pyrightconfig.json** — JSON в корне; Pylance/pyright ищут его автоматически.

**Основные поля pyrightconfig.json:**

| Поле | Назначение |
|------|------------|
| **include** | Каталоги/файлы для анализа (например, `["src"]`). |
| **exclude** | Исключить из анализа (например, `["tests", "**/__pycache__"]`). |
| **pythonVersion** | Версия Python: `"3.11"`. |
| **typeCheckingMode** | Уровень строгости: `"off"`, `"basic"`, `"standard"`, `"strict"`. |
| **venvPath** / **venv** | Путь к виртуальному окружению (для Pylance). |
| **reportMissingImports** | Сообщать о недостающих импортах. |
| **reportUnusedVariable** | Сообщать о неиспользуемых переменных. |

Итог: **pyproject.toml [tool.*]** — один файл для всего; **mypy.ini**, **ruff.toml**, **pyrightconfig.json** — альтернативы с тем же набором опций, при необходимости (отдельный репозиторий конфигов, мультиязычный проект, предпочтения команды).

### pre-commit

**pre-commit** — фреймворк для **Git-хуков**: перед каждым коммитом выполняются заданные проверки (линтер, форматтер, типы, тесты и т.д.). Если хотя бы один хук завершается с ошибкой, коммит **не создаётся** до исправления кода (или до явного обхода `git commit --no-verify`, что не рекомендуется в обычной работе).

**Установка:** `pip install pre-commit` (или через pipx: `pipx install pre-commit`).

**Настройка в репозитории:** в корне создаётся файл **`.pre-commit-config.yaml`**. Пример с Ruff, mypy и **pytest**:

```yaml
default_language_version:
  python: python3.11

repos:
  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.4.0
    hooks:
      - id: ruff
        args: [--fix, --exit-non-zero-on-fix]
      - id: ruff-format

  - repo: https://github.com/pre-commit/mirrors-mypy
    rev: v1.8.0
    hooks:
      - id: mypy
        additional_dependencies: [types-requests]

  - repo: local
    hooks:
      - id: pytest
        name: pytest
        entry: pytest
        args: [--tb=short, -q]
        language: system
        pass_filenames: false
```

**pytest в pre-commit:** хук **pytest** запускает тесты при каждом коммите. Используется **repo: local** и **language: system** — выполняется установленный в окружении `pytest`. **pass_filenames: false** — pytest запускается без списка файлов (обычно прогон всего набора тестов). Плюсы: код не попадёт в репозиторий, если тесты падают. Минусы: при большом наборе тестов коммит может занимать десятки секунд или минуты, что замедляет работу. **Когда ставить pytest в pre-commit:** маленький/средний проект, быстрые тесты (< 10–30 с). **Когда не ставить:** долгие тесты, тяжёлая БД/сеть — лучше запускать pytest только в CI (GitHub Actions, GitLab CI и т.д.) и локально по желанию. Если хук pytest не нужен — блок `repo: local` с хуком pytest можно удалить; в конфиге выше достаточно Ruff и mypy для большинства проектов.

**Установка хуков в Git:** один раз в клонированном репозитории выполнить:

```bash
pre-commit install
```

После этого при каждом `git commit` будут запускаться хуки из .pre-commit-config.yaml. Хук устанавливается в `.git/hooks/pre-commit`.

**Основные команды pre-commit:**

| Команда | Назначение |
|---------|------------|
| `pre-commit install` | Установить хук pre-commit в репозиторий. |
| `pre-commit run --all-files` | Запустить все хуки по всем файлам (удобно в CI и для первичной проверки). |
| `pre-commit run ruff --all-files` | Запустить только хук с id ruff. |
| `pre-commit autoupdate` | Обновить ревизии репозиториев в .pre-commit-config.yaml до последних. |

**Параметры хуков (важные):**

| Параметр | Назначение |
|----------|------------|
| `rev` | Ревизия репозитория (тег или коммит); фиксирует версию инструмента. |
| `args` | Дополнительные аргументы для команды (например, `[--fix]` для ruff). |
| `files` | Регулярное выражение для фильтрации файлов (по умолчанию по типам из хука). |
| `stages` | На каком этапе запускать: **commit** (по умолчанию — при git commit), **push** (при git push), **manual** (только при явном вызове pre-commit run). Например, тяжёлые проверки (полный pytest) можно запускать только при push: `stages: [push]`. |
| `additional_dependencies` | Доп. пакеты для окружения хука (например, types-* для mypy). |

**Граничные случаи:** если хук меняет файлы (например, ruff --fix), после исправлений нужно снова добавить файлы в индекс (`git add .`) и повторить коммит. Для пропуска хуков в исключительных случаях используют `git commit --no-verify` (не делать по привычке).

#### Граничные случаи и отладка (§31a, pre-commit)

| Ситуация | Что проверить | Решение |
|----------|----------------|---------|
| pre-commit не запускается при git commit | Хук не установлен в .git/hooks/pre-commit. | Выполнить `pre-commit install` в корне репозитория. |
| Хук падает в CI, локально проходит | Версии инструментов (rev), additional_dependencies. | Зафиксировать rev в .pre-commit-config.yaml; в CI добавить те же additional_dependencies; запустить `pre-commit run --all-files` локально. |
| mypy в pre-commit долго выполняется | Много файлов, строгий режим. | Ограничить файлы хука (files: "\\.py$"); или убрать mypy из pre-commit и запускать только в CI. |
| pre-commit autoupdate сломал хуки | Новая версия репозитория несовместима. | Откатить rev в .pre-commit-config.yaml на предыдущий тег; обновлять по одному хуку и проверять. |

#### Частые вопросы (§31a)

| Вопрос | Краткий ответ |
|--------|----------------|
| Нужно ли коммитить .pre-commit-config.yaml? | Да. Конфиг хуков хранится в репозитории; все разработчики и CI используют одни и те же проверки. |
| Как обновить версии хуков? | Выполнить `pre-commit autoupdate`; проверить, что тесты и линтеры проходят после обновления. |
| Ставить ли pytest в pre-commit? | Только если тесты быстрые (< 10–30 с); иначе лучше запускать pytest только в CI. |

**Запомните:** pre-commit гарантирует, что в репозиторий не попадёт код, не проходящий линтер/типы (если хуки настроены); конфиг инструментов храните в pyproject.toml; ревизии в .pre-commit-config.yaml фиксируют версии инструментов для воспроизводимости.
