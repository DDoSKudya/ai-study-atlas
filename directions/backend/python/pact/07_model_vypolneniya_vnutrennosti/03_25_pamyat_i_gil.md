**Цель раздела §25:** понимать основы управления памятью в CPython: **reference counting** (счётчик ссылок) и **циклический сборщик мусора (gc)**. Понимать, что такое **GIL (Global Interpreter Lock)** и почему в одном процессе только один поток в данный момент выполняет Python-байткод; какие последствия это имеет для CPU-bound задач и когда GIL освобождается (I/O, C-расширения). В итоге вы будете осознанно выбирать многопоточность vs мультипроцессинг и понимать ограничения потоков для вычислений.

### 25.1 Reference counting (счётчик ссылок)

В CPython у каждого объекта в памяти хранится **счётчик ссылок (reference count)** — целое число, показывающее, сколько «ссылок» на этот объект сейчас существует. Ссылкой считается: (1) имя в каком-либо namespace (глобальном, локальном, атрибут объекта); (2) элемент контейнера (списка, словаря, кортежа и т.д.); (3) ссылка из стека вызовов (аргумент, локальная переменная); (4) ссылка из другого объекта (например, **__dict__** класса).

**Когда счётчик увеличивается:** при присваивании имени объекту (**a = obj**), при добавлении объекта в контейнер (**lst.append(obj)**), при передаче аргумента в функцию (**f(obj)** — параметр ссылается на объект), при создании ссылки в другом объекте (атрибут **obj.x = value**).

**Когда счётчик уменьшается:** при удалении имени (**del a**), при выходе имени из области видимости (конец функции, выход из **with**); при удалении элемента из контейнера или при уничтожении контейнера; при переназначении имени на другой объект (**a = 0** — старая ссылка теряется).

Когда счётчик становится **нулём**, объект больше недостижим. Интерпретатор вызывает деструктор **__del__** (если он определён), затем освобождает память. Поэтому в Python нет явного «free»: объекты удаляются автоматически, когда на них больше не ссылается никто.

**Ограничение:** циклические ссылки (A ссылается на B, B ссылается на A) не позволяют счётчику ни у A, ни у B стать нулём — оба объекта ссылаются друг на друга. Для обнаружения и удаления таких «островов» недостижимых объектов служит **циклический сборщик мусора (gc)**.

#### Пошагово: когда счётчик ссылок увеличивается и уменьшается (пример)

```python
def f(lst):
    lst.append(1)   # lst ссылается на переданный список; счётчик списка не меняется при append
    x = [1, 2]      # создаётся новый список; x ссылается на него — счётчик нового списка = 1
    return x        # возвращаем x; вызывающий код получит ссылку — счётчик станет 2 у вызывающего

outer = []
f(outer)            # при вызове f(outer): параметр lst ссылается на outer — счётчик outer +1 (внутри f)
                    # при выходе из f: lst исчезает — счётчик outer -1
                    # возвращённый список [1, 2] теперь ссылка только в результате вызова (если не сохранили — объект станет недостижим и будет собран)
```

Упрощённо: **присваивание имени объекту** увеличивает счётчик; **удаление имени** или **выход из scope** уменьшает; **добавление в контейнер** увеличивает; **удаление из контейнера** или **уничтожение контейнера** уменьшает.

**Таблица: когда reference count увеличивается и уменьшается (типичные случаи)**

| Действие | Эффект на счётчик |
|----------|-------------------|
| **a = obj** | obj: +1 (имя a ссылается на obj) |
| **del a** | obj: -1 (имя a удалено) |
| **a = other** (переназначение) | obj: -1, other: +1 |
| **lst.append(obj)** | obj: +1 (контейнер хранит ссылку) |
| **lst.pop()** | извлечённый объект: -1 (ссылка из контейнера исчезла) |
| **return obj** | obj: +1 у вызывающего (результат привязывается к имени) |
| Выход из функции (локальные имена) | все локальные объекты: -1 по каждому имени |
| **dict[key] = obj** | obj: +1 |
| **del dict[key]** | объект-значение: -1 |

### 25.2 Циклический сборщик мусора (gc)

Модуль **`gc`** предоставляет доступ к **циклическому сборщику мусора**: он находит группы объектов, ссылающихся друг на друга, но **недостижимых** из «корней» (корни — глобальные имена, стек вызовов, регистры, статические переменные C-расширений и т.д.). Такие группы (циклы) не могут быть освобождены только за счёт reference counting — счётчик у каждого объекта в цикле не меньше 1. Сборщик помечает такие объекты и при определённых условиях вызывает их деструкторы и освобождает память.

**Основные функции и атрибуты gc:**

- **gc.collect(generation=2)** — принудительный запуск сборки мусора; по умолчанию полный цикл (поколение 2). Возвращает число собранных объектов.
- **gc.get_count()** — кортеж счётчиков (например, (m, n, k)) — сколько раз с момента последней сборки «подрастали» поколения; при достижении порога сборка запускается автоматически.
- **gc.get_objects()** — возвращает список всех отслеживаемых объектов (осторожно: для отладки; не держать долго, не модифицировать).
- **gc.disable()** / **gc.enable()** — отключить/включить автоматическую сборку.

**Важно:** объекты с переопределённым **__del__**, участвующие в циклах, могут не собираться вовремя или в предсказуемом порядке (см. документацию **gc** и **__del__**). Для большинства кода достаточно понимать: «недостижимые циклы рано или поздно собираются»; при необходимости можно вызывать **gc.collect()** (например, после освобождения больших структур).

#### Когда запускается автоматическая сборка gc (упрощённо)

Сборщик поколения 0 вызывается автоматически, когда счётчик **get_count()[0]** достигает порога (зависит от версии CPython). При большом количестве аллокаций порог достигается чаще — сборка запускается без явного вызова **gc.collect()**. Сборка поколения 1 и 2 — реже. Явный вызов **gc.collect()** полезен после освобождения больших структур (например, удалили глобальный кеш), чтобы сразу вернуть память и не ждать следующего автоматического цикла.

#### Поколения объектов (generations) в gc — кратко

Циклический сборщик в CPython использует **три поколения** (0, 1, 2). Новые объекты попадают в поколение 0. При каждой сборке поколения 0 объекты, пережившие сборку, перемещаются в поколение 1; при сборке поколения 1 пережившие — в поколение 2. Сборка поколения 2 затрагивает все поколения. Частота сборки: поколение 0 проверяется чаще, поколение 2 — реже. Цель — не тратить время на «долгоживущие» объекты при каждой мелкой сборке. Вызов **gc.collect(generation)** принудительно запускает сборку до указанного поколения (0, 1 или 2); **gc.collect()** без аргументов — полная сборка (поколение 2).

### 25.3 GIL (Global Interpreter Lock)

**GIL** — глобальная блокировка интерпретатора: в любой момент времени только **один** поток выполняет байткод Python. Даже при нескольких потоках (модуль **threading**) они по очереди получают GIL; параллельного выполнения чисто Python-кода в одном процессе **нет**. Поэтому **CPU-bound** задачи в потоках не ускоряются (и даже замедляются из-за переключений). **I/O-bound** задачи выигрывают: пока один поток ждёт I/O, он освобождает GIL, и другой поток может выполняться.

**Когда GIL освобождается:** при ожидании I/O (чтение/запись файлов, сеть, **time.sleep** и т.д.); в C-расширениях — явно (Py_BEGIN_ALLOW_THREADS / Py_END_ALLOW_THREADS) или при вызове блокирующих системных вызовов.

#### Пошагово: когда поток получает и отпускает GIL (упрощённо)

В CPython каждый поток по очереди получает право выполнять байткод: (1) поток захватывает GIL (если он свободен); (2) выполняет некоторое количество байткод-инструкций (квант); (3) при определённых условиях (истечение кванта, вызов блокирующего I/O, явное освобождение в C-коде) поток отпускает GIL; (4) другой поток может захватить GIL и выполнять свой код. Таким образом, в один момент времени только один поток «держит» GIL; остальные ждут. При **time.sleep(1)** поток отпускает GIL на время сна — другие потоки могут выполняться. При чистом цикле **for i in range(10**7): pass** поток держит GIL до переключения по кванту (или до явного освобождения в C-расширении). Детали квантов и переключений зависят от версии CPython. Поэтому численные библиотеки (NumPy, криптография и т.д.), реализованные в C, могут отпускать GIL во время вычислений и давать реальный параллелизм по потокам.

**Следствия:**

- Для **CPU-bound** параллелизма в Python используют **multiprocessing** (несколько процессов — несколько GIL).
- Для **I/O-bound** многопоточность или **asyncio** эффективны: пока один поток/задача ждёт I/O, другие выполняются.

#### Когда GIL освобождается (практика)

- **Встроенный I/O:** чтение/запись файлов, сокеты, **time.sleep** — GIL отпускается на время ожидания.
- **C-расширения:** NumPy, OpenSSL, zlib и т.д. часто отпускают GIL во время вычислений (Py_BEGIN_ALLOW_THREADS), поэтому многопоточный код с такими библиотеками может давать реальный параллелизм.
- **Чистый Python-код в потоках:** GIL переключается между потоками по квантам времени (или при освобождении в I/O); параллельного выполнения двух строк Python-кода в одном процессе нет.

#### Пример: GIL и CPU-bound (иллюстрация)

```python
# Два потока, каждый считает в цикле — ускорения не будет (один GIL)
import threading
def cpu_work():
    n = 0
    for _ in range(10_000_000):
        n += 1

t1 = threading.Thread(target=cpu_work)
t2 = threading.Thread(target=cpu_work)
t1.start(); t2.start(); t1.join(); t2.join()
# Время ≈ 2 × время одного потока (или чуть больше из-за переключений)
# Для ускорения CPU-bound — multiprocessing, не threading
```

#### Схема: один GIL на процесс

```
    Процесс Python
    ┌─────────────────────────────────────────────────────────┐
    │  GIL (Global Interpreter Lock) — одна блокировка         │
    │  ┌─────────┐  ┌─────────┐  ┌─────────┐                 │
    │  │ Thread 1 │  │ Thread 2 │  │ Thread 3 │  ...           │
    │  └────┬─────┘  └────┬─────┘  └────┬─────┘                 │
    │       │             │             │                      │
    │       └─────────────┼─────────────┘                      │
    │                     ▼                                     │
    │              В каждый момент только ОДИН поток           │
    │              выполняет Python-байткод (держит GIL)       │
    │              Остальные ждут освобождения GIL              │
    └─────────────────────────────────────────────────────────┘
```

**Таблица: когда GIL освобождается (типично)**

| Ситуация | GIL освобождается? | Комментарий |
|----------|--------------------|-------------|
| Выполнение Python-байткода | Нет | Только один поток выполняет байткод |
| **time.sleep(n)** | Да | Поток отдаёт GIL на время сна |
| Чтение/запись файла (встроенный open) | Да | Блокирующий I/O — GIL отпускается |
| Сетевой I/O (socket.recv/send) | Да | Блокирующий I/O |
| **numpy** / C-расширения с Py_BEGIN_ALLOW_THREADS | Да | Расширение явно отпускает GIL |
| Чистый цикл на Python (без I/O) | Нет (или по квантам) | Переключение по квантам времени (interpreter switch) |

**Запомните:** GIL не отменяет многопоточность, но ограничивает параллельное выполнение **именно Python-байткода**. Используйте потоки для I/O и задач, отпускающих GIL; процессы — для тяжёлых вычислений на Python.

#### Версии Python и GIL (кратко)

| Версия | GIL | Комментарий |
|--------|-----|--------------|
| CPython 3.x (обычная сборка) | Один GIL на процесс | Один поток выполняет байткод в данный момент |
| Python 3.13+ (experimental) | **free-threaded** build | Сборка без GIL: потоки могут выполнять байткод параллельно; пока экспериментально, не по умолчанию |
| PyPy, Jython, IronPython | Своя модель | Могут иметь свой GIL или другую модель параллелизма |

В обычной сборке CPython для параллельного выполнения **чистого** Python-кода используйте **multiprocessing**. Free-threaded сборка (3.13+) в будущем может изменить рекомендации для многопоточности.
