### 27.1 pickle — основы

**pickle** превращает объекты Python в поток байтов и восстанавливает их обратно. Поддерживаются встроенные типы, функции и классы (по имени модуля и имени класса), экземпляры с сохранением атрибутов (через **__dict__** или **__getstate__**/__setstate__).

**Что нельзя сериализовать (типичные случаи):**

| Объект / тип | Причина |
|--------------|---------|
| Файловый объект (open), сокет | Содержит системные дескрипторы; при загрузке в другом процессе они невалидны |
| Поток (threading.Thread), блокировка | Привязаны к текущему процессу/памяти |
| Функция/класс, определённые в **__main__** | При загрузке модуль **__main__** может быть другим; имя не разрешится |
| Лямбда-функция, вложенная функция без имени | Нет стабильного «имени модуля.имя функции» для восстановления |
| Объекты с циклическими ссылками | Поддерживаются: pickle записывает ссылки и при загрузке восстанавливает граф объектов |
| Системные объекты (модули, классы) | Сериализуются по имени (модуль.имя); при загрузке импортируется модуль и берётся атрибут |

При попытке сериализовать неподдерживаемый объект возникает **PicklingError** (или **TypeError**). Для кастомных классов используют **__reduce__** или **__getstate__**/**__setstate__**, чтобы представить объект в виде поддерживаемых типов.

**Протоколы pickle — сводная таблица:**

| Протокол | Версия Python | Особенности |
|----------|----------------|-------------|
| 0 | Все | Текстовый формат (ASCII); обратная совместимость с Python 2 |
| 1 | Все | Бинарный формат; совместим с Python 2 |
| 2 | 2.3+ | Поддержка __reduce__, новые типы; совместим с Python 2 |
| 3 | 3.0+ | По умолчанию в 3.0–3.7; не совместим с Python 2 (bytes) |
| 4 | 3.4+ | Поддержка больших объектов, улучшенная эффективность |
| 5 | 3.8+ | **PickleBuffer** для zero-copy; out-of-band data; быстрее для больших буферов |

Чем выше протокол, тем компактнее и быстрее сериализация; **протокол 5** поддерживает **pickle.PickleBuffer** для zero-copy передачи больших буферов между процессами (удобно для multiprocessing и shared memory). **`pickle.HIGHEST_PROTOCOL`** — максимальный протокол для текущей версии интерпретатора. Старые версии Python **не могут** загружать данные, сохранённые более новым протоколом; для совместимости с Python 3.7 и ниже используйте протокол 4 или ниже.

**Таблица: выбор протокола pickle по задаче**

| Задача | Рекомендуемый протокол | Комментарий |
|--------|------------------------|-------------|
| Максимальная скорость и размер в текущей версии | **pickle.HIGHEST_PROTOCOL** | По умолчанию в 3.8+ это 5 |
| Совместимость с Python 3.4–3.7 | **4** | Протокол 5 не читается в 3.7 и ниже |
| Совместимость с Python 2 | **2** | Протоколы 3+ дают bytes, в Py2 могут быть отличия |
| Читаемый текст (отладка) | **0** | ASCII-формат; большой размер |
| Zero-copy буферы между процессами (3.8+) | **5** | **PickleBuffer**, out-of-band data |

**Безопасность:** **никогда** не загружайте (**pickle.load** / **pickle.loads**) данные из ненадёжных источников (сеть, пользовательский ввод, непроверенные файлы). При десериализации pickle **выполняет произвольный код**: восстанавливая объект, он вызывает указанные в потоке функции и конструкторы. Злоумышленник может подготовить поток данных так, чтобы при загрузке выполнилась вредоносная функция (например, удаление файлов или выполнение команд). Для обмена данными с ненадёжными источниками используйте безопасные форматы (JSON, XML, Protocol Buffers и т.д.).

#### Пошагово: что записывается при pickle.dumps (упрощённо)

При **pickle.dumps(obj)** сериализатор обходит граф объектов, начиная с **obj**: (1) для каждого объекта определяется тип (встроенный или пользовательский); (2) встроенные типы (dict, list, int, str и т.д.) записываются в компактном бинарном виде (протокол 3+); (3) пользовательские экземпляры записываются по имени класса (модуль.имя класса) и по состоянию — либо **__dict__**, либо результат **__getstate__()** или **__reduce__()**; (4) вложенные объекты (значения в dict, элементы списка) сериализуются рекурсивно; (5) при повторной встрече того же объекта (по id) записывается ссылка на уже сохранённый объект — граф сохраняется без дублирования. В итоге в буфер попадает последовательность байт, достаточная для восстановления графа объектов при **pickle.loads**.

#### Пошагово: что происходит при pickle.loads (восстановление графа)

При **pickle.loads(buf)** десериализатор читает байты из буфера: (1) определяется тип следующего объекта (встроенный или имя класса); (2) для встроенных типов данные восстанавливаются из бинарного формата; (3) для пользовательских экземпляров загружается класс по имени (импорт модуля и получение атрибута класса), затем создаётся экземпляр — либо через **__new__** + **__setstate__**, либо через **__reduce__** (вызов callable с аргументами); (4) вложенные объекты восстанавливаются рекурсивно; (5) при встрече ссылки на уже восстановленный объект подставляется этот объект — граф связей восстанавливается. Важно: в процессе загрузки вызываются конструкторы и **setstate**; если в потоке указан произвольный callable (как в атаке на pickle), он будет вызван — поэтому загрузка из ненадёжных данных недопустима.

#### Пример: pickle.dumps / loads

```python
import pickle
data = {"a": 1, "b": [2, 3]}
buf = pickle.dumps(data, protocol=pickle.HIGHEST_PROTOCOL)
restored = pickle.loads(buf)
print(restored)  # {'a': 1, 'b': [2, 3]}
```

### 27.2 __reduce__ и __reduce_ex__

Если объект нужно сериализовать нестандартно, класс может реализовать **`__reduce__(self)`** или **`__reduce_ex__(self, protocol)`**. При сериализации pickle вызывает **__reduce_ex__(protocol)** (если нет — **__reduce__()**). Возвращаемое значение — кортеж одной из форм:

- **(callable, args)** — при загрузке вызывается **callable(*args)**; результат и есть восстановленный объект.
- **(callable, args, state)** — **callable(*args)** создаёт объект; затем вызывается **obj.__setstate__(state)** (если есть) или state присваивается **obj.__dict__**.
- **(callable, args, state, list)** — list — итератор по объектам для поочерёдного добавления (для подпротокола).
- **(callable, args, state, list, dict)** — dict — пары (key, value) для присваивания атрибутов.

Чаще всего используют **(callable, args)** или **(callable, args, state)**. **callable** должен быть импортируемым по имени (модуль + имя функции/класса), чтобы при загрузке pickle мог его найти. При загрузке **нельзя** передавать произвольный объект — только то, что можно восстановить по имени (built-in или из импортируемых модулей).

#### Пример: __reduce__ для кастомной сериализации

```python
import pickle

class Point:
    def __init__(self, x, y):
        self.x, self.y = x, y
    def __reduce__(self):
        # При загрузке pickle вызовет Point(self.x, self.y)
        return (Point, (self.x, self.y))

p = Point(1, 2)
buf = pickle.dumps(p)
p2 = pickle.loads(buf)
print(p2.x, p2.y)  # 1 2
```

### 27.3 __getstate__ и __setstate__

**`__getstate__(self)`** — вызывается при сериализации вместо сохранения **__dict__**; должен вернуть то, что будет передано в **__setstate__**. **`__setstate__(self, state)`** — вызывается при десериализации с этим state. Позволяет исключить временные или несериализуемые поля и восстановить их при загрузке.

#### Пример: __getstate__ / __setstate__

```python
class C:
    def __init__(self, x, temp=None):
        self.x = x
        self.temp = temp  # не хотим сохранять
    def __getstate__(self):
        return {"x": self.x}
    def __setstate__(self, state):
        self.x = state["x"]
        self.temp = None
```

#### Pickle и __slots__: граничный случай

У классов с **__slots__** нет **__dict__**; по умолчанию pickle сериализует экземпляр через механизм, который учитывает слоты (состояние хранится в слотах). При кастомной сериализации с **__getstate__** нужно возвращать словарь с именами слотов и значениями; в **__setstate__** присваивать значения слотам (не в **__dict__**). Если класс с **__slots__** наследует от класса без слотов, в **__getstate__** может понадобиться объединить состояние слотов и **super().__getstate__()** при наличии.

#### Pickle и наследование

При сериализации подкласса pickle записывает имя класса (модуль.имя); при загрузке импортируется модуль и берётся класс по имени — создаётся экземпляр **именно этого подкласса**. Атрибуты, определённые в подклассе и в базовом классе, восстанавливаются (через **__dict__** или **__getstate__**). Если базовый класс определяет **__getstate__**/__setstate__**, подкласс может вызвать **super().__getstate__()** и дополнить состояние своими полями, затем в **__setstate__** вызвать **super().__setstate__(state)** и восстановить свои поля.

### 27.4 marshal и shelve

- **marshal** — низкоуровневая сериализация; используется **внутри интерпретатора** для записи и чтения скомпилированного байткода в **.pyc**. Поддерживает ограниченный набор типов (None, bool, int, float, str, bytes, tuple, list, dict, set, frozenset, code objects и т.д.). **Нестабилен** между версиями Python: формат может меняться; один и тот же объект может сериализоваться по-разному в разных сборках. **Не предназначен** для персистентного хранения данных или обмена между системами. Документация явно предупреждает: не используйте marshal для персистентных данных — используйте **pickle** (или другие форматы). **Безопасность:** как и pickle, при загрузке выполняется интерпретация данных — не загружайте marshal из ненадёжных источников.
- **shelve** — persistent-словарь: ключи — строки, значения — объекты, сериализуемые через **pickle**; хранится в файле (часто через **dbm**). **shelve.open(filename, flag='c', ...)** возвращает объект, ведущий себя как dict (чтение/запись по ключу). Методы **sync()** — сбросить буферы на диск; **close()** — закрыть и синхронизировать. Транзакционность и блокировки зависят от бэкенда (dbm); при одновременной записи из разных процессов возможны конфликты — для многопроцессного доступа нужна внешняя блокировка или другой хранилищный слой.

#### shelve: транзакционность и блокировки

У **shelve** нет полноценных транзакций (нет ACID): изменения пишутся в бэкенд (dbm) по мере операций; откат к «состоянию до начала серии записей» средствами shelve невозможен. Запись может быть отложена в буфер — до вызова **sync()** или **close()** данные не обязательно видны другому процессу, открывшему тот же файл. Поэтому после критичных записей имеет смысл вызывать **db.sync()**, чтобы сбросить буферы на диск и снизить риск потери при сбое.

Блокировка файла на уровне ОС зависит от бэкенда: типичные реализации dbm (ndbm, gdbm и т.д.) блокируют файл при записи (или весь файл при открытии), но единого стандарта нет; при одновременной записи из двух процессов возможны повреждение файла или неожиданные исключения. Для многопроцессного доступа рекомендуется внешняя блокировка (например, **fcntl.flock**, или файл-замок, или отдельный сервис) либо отказ от shelve в пользу хранилища с явной поддержкой конкурентного доступа.

**Практический вывод:** вызывайте **sync()** после серии важных записей; для совместной записи из нескольких процессов используйте внешнюю блокировку или другое хранилище.

#### Пример: shelve

```python
import shelve
with shelve.open("mydata") as db:
    db["key"] = [1, 2, 3]
with shelve.open("mydata") as db:
    print(db["key"])  # [1, 2, 3]
```

**Запомните:** для переносимой и безопасной сериализации данных предпочитайте **JSON** или другие форматы; **pickle** — для доверенных источников и Python-специфичных объектов. **marshal** — только для внутренних целей интерпретатора.
