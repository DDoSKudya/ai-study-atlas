**Цель раздела §24:** понимать, как Python находит и загружает модули: роль **`sys.path`** и **`PYTHONPATH`**, разница между **`import mod`** и **`from mod import name`**, устройство пакетов (`__init__.py`) и namespace packages (PEP 420), относительный импорт (`.` и `..`) и **`__all__`**. ---

**Импорт** в Python — это механизм загрузки кода из файлов модулей и пакетов и привязки имён к загруженным объектам. При первом обращении к модулю интерпретатор ищет его по списку путей (**`sys.path`**), загружает и помещает в **`sys.modules`**; повторный импорт того же модуля просто возвращает уже загруженный объект из `sys.modules`.

#### Теоретическое обоснование: PEP 302 и PEP 420

- **PEP 302** — «New Import Hooks»: введение **import hooks** через **sys.meta_path** (MetaPathFinder) и **sys.path_hooks** (PathEntryFinder). Импорт разделён на две фазы: **finding** (поиск модуля и создание ModuleSpec) и **loading** (выполнение кода и создание объекта модуля). Это позволило реализовать импорт из zip-архивов, виртуальных файловых систем и т.д.
- **PEP 420** — «Implicit Namespace Packages»: пакеты без **__init__.py** (namespace packages). Несколько каталогов с одним именем пакета в разных местах **sys.path** объединяются в один логический пакет; подмодули ищутся во всех этих каталогах. Упрощает распределённые пакеты и плагины.

**Термины:**

| Термин | Определение |
|--------|-------------|
| **Модуль** | Объект, представляющий загруженный код (обычно один .py файл или пакет); имеет атрибуты `__name__`, `__file__` и т.д. |
| **Пакет** | Модуль, содержащий другие модули (подкаталог с `__init__.py` или namespace package). |
| **sys.path** | Список строк (путей), по которым ищутся модули; формируется из текущей директории, PYTHONPATH, установленных пакетов и т.д. |
| **sys.modules** | Словарь «имя модуля → объект модуля»; кеш уже загруженных модулей. |

#### Пошагово: что происходит при «import mod»

Ниже — полная последовательность действий интерпретатора при первом **`import mod`** (без подмодулей). При повторном **`import mod`** выполняется только шаг 1.

1. **Проверка кеша:** интерпретатор смотрит в **sys.modules**: есть ли ключ **"mod"**. Если да — объект модуля уже загружен; он возвращается, и **ни один из следующих шагов не выполняется**. Имя **mod** в текущем пространстве имён привязывается к этому объекту. Код модуля **не выполняется повторно**.
2. **Поиск (finding):** если в **sys.modules** ключа **"mod"** нет, вызываются **finders** из **sys.meta_path** по очереди. У каждого finder’а вызывается метод **find_spec("mod", path=None)** (для подмодулей пакета передаётся **path** — список каталогов пакета). Первый finder, вернувший не **None** (объект **ModuleSpec**), используется; остальные не вызываются.
3. **Создание модуля и регистрация в sys.modules:** по найденному **ModuleSpec** создаётся новый объект модуля (тип **types.ModuleType**). Этот объект **сразу** помещается в **sys.modules["mod"]** — **до** выполнения кода модуля. Именно поэтому при циклическом импорте второй модуль при обращении к первому получает объект модуля, который ещё не до конца инициализирован (часть атрибутов может отсутствовать).
4. **Загрузка (loading):** у **loader** из spec вызывается **loader.exec_module(module)**. В этот метод передаётся тот же объект модуля, что лежит в **sys.modules["mod"]**. **exec_module** выполняет код модуля (файл .py или код из другого источника) в namespace этого модуля; так заполняются атрибуты модуля (функции, классы, переменные).
5. **Привязка имени:** в текущем пространстве имён (глобальном или локальном, в зависимости от того, где выполняется import) имя **mod** привязывается к объекту **sys.modules["mod"]**.

**Важно:** модуль попадает в **sys.modules** **до** завершения **exec_module**. Поэтому если код модуля **mod** в процессе выполнения импортирует другой модуль **other**, а **other** в свою очередь делает **import mod**, то на шаге 1 для **mod** в **sys.modules** уже будет объект модуля **mod** — но его код ещё не до конца выполнен (текущий вызов **exec_module** для **mod** не завершён). В **other** при обращении к **mod.attr** атрибут **attr** может ещё не существовать — получим **AttributeError**.

### 24.1 import и from ... import

- **`import mod`** — загружает модуль `mod` (если ещё не загружен) и привязывает имя **`mod`** к объекту модуля в текущем пространстве имён. Обращение к атрибутам: **`mod.attr`**.
- **`from mod import name`** — загружает модуль `mod` и привязывает имя **`name`** в текущем пространстве к `mod.name` (без префикса `mod.`).
- **`from mod import *`** — импортирует имена, перечисленные в **`__all__`** модуля `mod` (если `__all__` задан); иначе все публичные имена (не начинающиеся с `_`). Не рекомендуется: засоряет пространство имён и затрудняет понимание, откуда имя.

**Поиск модуля:** интерпретатор перебирает **finders** (в т.ч. из **`sys.meta_path`** и **`sys.path`**), находит **loader** и с его помощью создаёт объект модуля и выполняет код модуля. Детали — в §24a.

#### Пошагово: разница «import mod» и «from mod import name»

- **`import mod`:** (1) Загружается модуль **mod** (если ещё не в **sys.modules** — полный цикл finding → создание модуля → exec_module). (2) В текущем пространстве имён привязывается имя **mod** к объекту **sys.modules["mod"]**. В коде вы обращаетесь к атрибутам как **mod.attr**.
- **`from mod import name`:** (1) Загружается модуль **mod** (тот же процесс, что выше). (2) Вычисляется **mod.name** (атрибут модуля). (3) В текущем пространстве имён привязывается имя **name** к объекту **mod.name**. Имя **mod** в текущем пространстве **не** появляется (модуль есть в **sys.modules**, но не как локальное имя). В коде вы используете **name** напрямую.
- **`from mod import name as local_name`:** то же, что **from mod import name**, но в текущем пространстве привязывается **local_name**, а не **name**.

**Итог:** при **import mod** в namespace попадает только **mod**; при **from mod import name** в namespace попадает **name** (или **local_name**), но не **mod**. Модуль в обоих случаях загружается один раз и кешируется в **sys.modules**.

#### Трассировка импорта (отладка): как увидеть порядок загрузки

Чтобы наглядно увидеть порядок выполнения при импорте (в том числе при циклических зависимостях), в начало каждого модуля можно добавить **print** с **__name__** и моментом входа/выхода:

```python
# В начале каждого из a.py, b.py:
print(f"[ENTER] {__name__}")   # в самой первой строке модуля

# В конце модуля (после всех определений):
print(f"[EXIT]  {__name__}")
```

При **import a** (если a импортирует b, а b импортирует a) вывод будет примерно таким: **[ENTER] a** → **[ENTER] b** → **[ENTER] a** (повторно не печатаем, a уже в sys.modules) — то есть при входе в b модуль a уже «в процессе». Затем **[EXIT] b**, **[EXIT] a**. Так видно, что **a** завершает выполнение только после полного выхода из **b**. Если в **b** на верхнем уровне обратиться к **a.x**, а **x** в **a** определён после **import b**, в логе будет **[ENTER] a**, **[ENTER] b**, и затем **AttributeError** — потому что **[EXIT] a** ещё не было, **x** не создан.

#### Примеры: import и from

```python
import os
print(os.path.join("a", "b"))  # os — объект модуля

from os import path
print(path.join("a", "b"))    # path — имя в текущем пространстве

from os.path import join
print(join("a", "b"))         # join — функция в текущем пространстве
```

```python
# __all__ ограничивает «import *»
# В модуле mymod:
# __all__ = ["public_func"]
# def public_func(): pass
# def _private(): pass

# from mymod import *
# public_func доступна, _private — нет (если __all__ задан)
```

### 24.2 sys.path и PYTHONPATH

**`sys.path`** — список строк (путей к каталогам или к .zip архивам), по которым интерпретатор ищет модули. При **запуске** интерпретатора он формируется в следующем порядке (на примере CPython):

1. **Первый элемент — стартовая директория (script directory):**
   - Если программа запущена как **`python script.py`** — каталог, в котором лежит **script.py** (абсолютный путь), либо пустая строка **`''`**, которую интерпретатор трактует как «текущая рабочая директория» (cwd) в момент запуска.
   - Если программа запущена как **`python -m package.module`** — в начало **sys.path** помещается пустая строка **`''`** (текущая рабочая директория, т.е. каталог, из которого вызвана команда **python**), а не каталог файла **module.py**. Это важно: при **`-m`** импорты разрешаются относительно **cwd**, поэтому обычно запускают **python -m** из корня проекта.
2. **Переменная окружения PYTHONPATH:** если задана, её значение (каталоги через разделитель: **:** в Unix, **;** в Windows) добавляются в **sys.path** после стартовой директории.
3. **Сайт-пакеты и зависимости интерпретатора:** каталоги, зависящие от установки (например, **site-packages** виртуального окружения или системного Python), добавляются через механизм **site** (при импорте модуля **site** при старте).

**Таблица: откуда берётся sys.path**

| Источник | Когда добавляется | Типичное содержимое |
|----------|-------------------|----------------------|
| Стартовый скрипт / -m | При запуске | Каталог script.py или '' (cwd) |
| PYTHONPATH | При запуске | Каталоги из переменной окружения |
| site | При запуске (импорт site) | site-packages, стандартная библиотека |

Порядок важен: **finder** (например, **PathFinder**) перебирает каталоги в **sys.path** по порядку; первый каталог, в котором найден модуль с запрошенным именем, используется. Поэтому добавление пути в начало (**sys.path.insert(0, "/my/project")**) даёт ему приоритет над site-packages — удобно для разработки, но может «затенять» установленные пакеты. Для библиотек предпочтительнее установка в venv или корректный **PYTHONPATH**.

#### Пример: sys.path

```python
import sys
print(sys.path)
# Типичный вывод (сокращённо): ['', '/usr/lib/python3.12', ...]

# Добавить каталог в начало поиска
sys.path.insert(0, "/path/to/my/project")
import mypackage  # теперь ищется в /path/to/my/project
```

### 24.3 Модули и пакеты, __init__.py

- **Модуль** — один файл **`something.py`**; при **`import something`** интерпретатор ищет **something.py** в **sys.path**, создаёт объект модуля, помещает его в **sys.modules["something"]** и выполняет код файла в namespace этого модуля. Все глобальные имена (функции, классы, переменные), определённые в файле, становятся атрибутами объекта модуля.
- **Пакет** — каталог, который интерпретатор рассматривает как модуль. Классический пакет — каталог с файлом **`__init__.py`**. При **`import pkg`** выполняется **pkg/__init__.py**; объект модуля пакета получает атрибут **__path__** — список каталогов, в которых ищутся подмодули (обычно один элемент — путь к каталогу пакета). Подмодули — файлы **pkg/submod.py** (импорт: **import pkg.submod**) или подкаталоги **pkg/subpkg/** с **__init__.py** (импорт: **import pkg.subpkg**).

**Порядок выполнения при импорте пакета:** при **`import pkg.subpkg.deep`** интерпретатор по очереди загружает: **pkg** (выполняет **pkg/__init__.py**), затем **pkg.subpkg** (выполняет **pkg/subpkg/__init__.py**), затем **pkg.subpkg.deep** (выполняет **pkg/subpkg/deep.py**). Каждый из этих модулей один раз регистрируется в **sys.modules** и при повторном импорте берётся из кеша.

**Namespace packages (PEP 420):** каталог **без** **__init__.py** может выступать как пакет, если он встречается в **sys.path** или как подкаталог при поиске подмодуля. Несколько каталогов с **одним и тем же именем пакета** в разных местах **sys.path** объединяются в один «виртуальный» пакет (namespace package): **__path__** такого пакета содержит список всех этих каталогов; при импорте подмодуля поиск идёт по всем каталогам из **__path__**. Для обычных пакетов с одним местоположением по-прежнему принято использовать **__init__.py** — так явно видно, что каталог является пакетом, и можно выполнить код инициализации при первом импорте.

#### Пример структуры пакета

```
mypkg/
  __init__.py    # выполняется при import mypkg
  submod.py      # подмодуль: import mypkg.submod
  subpkg/
    __init__.py
    deep.py      # import mypkg.subpkg.deep
```

```python
# __init__.py может быть пустым или экспортировать имена
# mypkg/__init__.py:
# from . import submod
# __all__ = ["submod"]
```

#### Пример: относительный импорт внутри пакета

```python
# Файл mypkg/subpkg/deep.py (внутри пакета mypkg.subpkg):
# Импорт из текущего пакета (subpkg):
from . import sibling_module
from .sibling_module import foo

# Импорт из родительского пакета (mypkg):
from .. import submod
from ..submod import some_func

# Импорт из «соседнего» подпакета того же уровня:
from .. import other_subpkg
```

Относительный импорт возможен **только** в коде, который загружается как часть пакета (т.е. **__package__** модуля не **None**). В скрипте, запущенном как **python script.py**, **__package__** равно **None** — относительный импорт (**from . import ...**) использовать нельзя.

### 24.4 Относительный импорт: . и ..

Внутри пакета можно использовать **относительный импорт**:

- **`from . import name`** — импорт из текущего пакета (текущий каталог как пакет).
- **`from .submod import foo`** — из подмодуля текущего пакета.
- **`from .. import sibling`** — из пакета уровня выше (родительский каталог).
- **`from ..sibling import foo`** — из соседнего пакета того же уровня.

Точка обозначает «текущий пакет», две точки — «родительский пакет». Относительный импорт возможен только внутри пакета (в коде, который загружается как часть пакета); в скрипте, запущенном напрямую, пакет не определён — там нельзя писать `from . import ...`.

**`__package__`** — атрибут модуля, имя пакета, которому модуль принадлежит (для главного модуля может быть `None`). Используется механизмом импорта для разрешения относительных импортов.

### 24.5 __all__

**`__all__`** — необязательный атрибут модуля (список строк с именами). Определяет, **какие имена** экспортируются при **`from mod import *`**.

- **Если `__all__` задан:** при **`from mod import *`** в текущее пространство имён попадают **только** имена, перечисленные в **__all__**. Имена, не указанные в **__all__**, не импортируются (даже если они публичные).
- **Если `__all__` не задан:** при **`from mod import *`** импортируются все имена, определённые в модуле, **кроме** тех, что начинаются с символа подчёркивания **`_`**. То есть «приватные» по соглашению имена (_private) не экспортируются.

Задание **__all__** полезно: (1) явно документирует публичный API модуля; (2) ограничивает «шум» при **import ***; (3) помогает инструментам (IDE, документация) определять, что считается публичным. **import *** в целом не рекомендуется (PEP 8): лучше явно перечислять импортируемые имена.

### 24.6 importlib — динамический импорт

Модуль **`importlib`** позволяет загружать модули по строковому имени и перезагружать их.

- **`importlib.import_module(name)`** — загружает и возвращает модуль по имени (например, `"mypkg.submod"`). Эквивалент `import` в рантайме.
- **`importlib.reload(module)`** — перезагружает уже загруженный модуль (выполняет его код заново). Полезно при разработке без перезапуска интерпретатора. **Важно:** **reload** не меняет существующие ссылки на старые объекты модуля в других модулях. Например, если **mymod** импортирован в **main** как **import mymod**, после **reload(mymod)** имя **mymod** в **main** будет указывать на **обновлённый** объект модуля (потому что **mymod** — это имя, привязанное к **sys.modules["mymod"]**, а **reload** обновляет именно этот объект). Но если в **main** было **from mymod import foo**, то **foo** остаётся ссылкой на **старый** объект функции — он не обновится. Поэтому при использовании **reload** предпочтительно везде использовать **import mod** и обращаться как **mod.attr**, либо после **reload** заново выполнять **from mymod import foo**.

#### Пример: динамический импорт

```python
import importlib
mod = importlib.import_module("json")
print(mod.dumps({"a": 1}))  # '{"a": 1}'

# Перезагрузка (осторожно: остальные модули могут держать старую версию)
import mymod
importlib.reload(mymod)
```

## §24a. Импорт — углублённо

**Цель:** понимать цепочку **finders** и **loaders**: **`sys.meta_path`**, **`sys.path_hooks`**, **`sys.path_importer_cache`**, а также загрузку из файла и из zip-архивов. Уметь создавать модуль из spec (**`importlib.util.spec_from_file_location`**, **`module_from_spec`**) и запускать модуль как скрипт (**`runpy.run_module`**).

### sys.meta_path

**`sys.meta_path`** — список объектов **MetaPathFinder** (или совместимых с этим протоколом). При каждом **`import name`** (или **import pkg.submod**) интерпретатор по очереди вызывает у каждого finder’а метод **`find_spec(name, path=None, target=None)`**:

- **name** — полное имя импортируемого модуля (строка), например **"os"**, **"pkg.submod"**.
- **path** — для подмодулей пакета: список каталогов (атрибут **__path__** пакета), в которых искать подмодуль; для импорта модуля верхнего уровня — **None**.
- **target** — опционально объект модуля (при reload); обычно **None**.

Finder возвращает **ModuleSpec** или **None**. Первый finder, вернувший не **None**, «выигрывает» — его spec используется для создания модуля и загрузки. Остальные finder’ы для этого импорта не вызываются. Если все вернули **None**, возбуждается **ModuleNotFoundError**.

**Встроенные finder’ы (типичный порядок в sys.meta_path):** (1) **BuiltinImporter** — встроенные модули (C); (2) **FrozenImporter** — frozen модули; (3) **PathFinder** — модули из **sys.path** (файлы .py, пакеты, zip). Можно **вставить** свой finder в **sys.meta_path** (например, в начало — **sys.meta_path.insert(0, MyFinder())**), чтобы реализовать импорт из БД, сети или виртуальной файловой системы.

#### Каркас кастомного MetaPathFinder (импорт из словаря)

Чтобы реализовать импорт из нестандартного источника (словарь имён и кода, сеть, БД), реализуют класс с методом **find_spec** и **Loader** с **exec_module**, затем добавляют finder в **sys.meta_path**:

```python
import sys
import importlib.abc
import importlib.util

class DictFinder(importlib.abc.MetaPathFinder):
    def __init__(self, modules_dict):
        self.modules_dict = modules_dict
    def find_spec(self, fullname, path=None, target=None):
        if fullname not in self.modules_dict:
            return None
        loader = DictLoader(fullname, self.modules_dict[fullname])
        return importlib.util.spec_from_loader(fullname, loader, origin="<dict>")

class DictLoader(importlib.abc.Loader):
    def __init__(self, name, code):
        self.name, self.code = name, code
    def create_module(self, spec):
        return None
    def exec_module(self, module):
        exec(self.code, module.__dict__)

d = {"mymod": "x = 42\ndef f(): return x"}
sys.meta_path.insert(0, DictFinder(d))
import mymod
print(mymod.x, mymod.f())  # 42 42
```

**find_spec** возвращает **ModuleSpec** с кастомным **Loader**; **exec_module** выполняет код в **module.__dict__**. Для загрузки из файла или сети в **Loader** подставляют чтение из нужного источника.

### sys.path_hooks и sys.path_importer_cache

**`sys.path_hooks`** — список фабрик (callable). При поиске модуля по **sys.path** для **каждого** элемента пути (строки) интерпретатор по очереди вызывает каждый callable из **sys.path_hooks**, передавая этот путь. Callable либо возвращает **PathEntryFinder** (объект с методом **find_spec** для подмодулей на этом пути), либо выбрасывает **ImportError** (этот hook не подходит для данного пути). Первый успешный результат используется. Так реализуется поддержка нестандартных «путей» — например, **zipimport** регистрирует hook, который для пути к .zip архиву возвращает finder для импорта из архива.

**`sys.path_importer_cache`** — словарь: **путь (строка) → PathEntryFinder или None**. После того как для пути найден finder (или решено, что путь не поддерживается — **None**), результат кешируется в **path_importer_cache**. При следующем обращении к тому же пути path_hooks не вызываются снова — используется кеш. Это ускоряет повторные импорты и уменьшает количество вызовов.

### importlib.util: spec и загрузка из файла

- **`importlib.util.spec_from_file_location(name, location)`** — создаёт **ModuleSpec** для модуля, загружаемого из файла `location` (например, `.py` или `.pyc`).
- **`importlib.util.module_from_spec(spec)`** — создаёт объект модуля по spec (без выполнения кода); выполнение кода делает loader через **`spec.loader.exec_module(module)`**.
- **`importlib.util.find_spec(name)`** — находит ModuleSpec для имени (без загрузки). Полезно для проверки наличия модуля.

#### Пример: загрузка модуля из файла по пути (полный цикл)

```python
import importlib.util
import sys

# 1. Создать ModuleSpec для модуля, загружаемого из файла
spec = importlib.util.spec_from_file_location("mymod", "/path/to/mymod.py")
# spec.origin == "/path/to/mymod.py", spec.loader — FileLoader для этого пути

# 2. Создать объект модуля (пустой, без выполнения кода)
mod = importlib.util.module_from_spec(spec)
# У mod есть __name__, __loader__, __spec__; атрибутов из mymod.py ещё нет

# 3. (Опционально) Зарегистрировать в sys.modules, чтобы импорты внутри mymod.py работали
sys.modules["mymod"] = mod

# 4. Выполнить код модуля — заполняются атрибуты mod
spec.loader.exec_module(mod)
# Теперь mod.func, mod.Class и т.д. доступны

# 5. Использовать модуль
mod.some_function()
```

Если не добавить **sys.modules["mymod"] = mod** перед **exec_module**, то код внутри **mymod.py** при встрече **import mymod** или **from mymod import ...** не найдёт модуль в **sys.modules** и попытается загрузить его заново по **sys.path**, что может привести к повторной загрузке или к ошибке.

### importlib.reload и runpy

- **`importlib.reload(module)`** — перезагружает модуль (см. выше).
- **`runpy.run_module(name)`** — запускает модуль по имени как скрипт (аналогично **`python -m name`**): устанавливает **`__name__ == "__main__"`** и выполняет код. Удобно для вызова «главного» кода модуля из программы.

### pkgutil.iter_modules

**`pkgutil.iter_modules(path=None)`** — итератор по модулям, найденным по `path` (по умолчанию — текущий пакет по стеку вызовов). Возвращает кортежи `(module_loader, name, ispkg)`. Используется для обнаружения плагинов или списка подмодулей пакета.

### zipimport и .pyc

Модуль **`zipimport`** позволяет импортировать модули из **.zip** архивов, если путь к архиву добавлен в **`sys.path`**. Внутри zip могут лежать файлы **`.pyc`** (скомпилированный байткод); интерпретатор при наличии подходящего **.pyc** может загружать его вместо **.py** для ускорения. Формат **.pyc** и расположение (**`__pycache__`**) — детали реализации; между версиями Python они несовместимы.

**Как ищется .pyc внутри zip.** При импорте модуля **`pkg.mod`** загрузчик (ZipImporter) ищет в архиве один из двух вариантов путей:

- **Имя модуля как путь к .py:** например **`pkg/mod.py`** (для **`pkg.mod`**). Если в zip есть такой файл, он компилируется в байткод «на лету» и выполняется.
- **Имя в каталоге __pycache__:** путь вида **`pkg/__pycache__/mod.cpython-312.pyc`** (для Python 3.12). Имя файла в **`__pycache__`** формируется как **`<name>.cpython-<version>.pyc`** (например **`mod.cpython-312.pyc`**). Если такой файл есть, загружается готовый байткод без компиляции исходника.

То есть структура внутри zip повторяет структуру на диске: пакет — каталог с **`__pycache__`**, внутри — файлы **`<module>.cpython-<ver>.pyc`**. Поиск идёт по полному имени модуля (с точками), преобразованному в путь с **`/`** и суффиксом **`.py`** или в **`__pycache__/<name>.cpython-<ver>.pyc`**. Версия в имени **.pyc** должна совпадать с версией интерпретатора; иначе используется **.py** или выдаётся ошибка.

**Запомните:** импорт определяется **meta_path** и **path**; загрузчик (loader) привязан к **ModuleSpec**. Для кастомного импорта (из БД, сети) достаточно реализовать finder и loader и добавить finder в **`sys.meta_path`**.

## §24b. Импорт: крайние случаи (циклические импорты)

**Цель:** понимать, почему возникают **циклические импорты** (A импортирует B, B импортирует A), как интерпретатор ведёт себя при этом (модуль может оказаться в **`sys.modules`** до завершения выполнения своего кода — «частично загружен»), и какие приёмы помогают: отложенный импорт (внутри функции), реструктуризация кода, вынос общих зависимостей. Также знать роль **`__path__`**, **`__package__`** и **zipimport** в пограничных случаях.

---

### Циклические импорты: как это происходит (пошаговый сценарий)

Пусть **a.py** импортирует **b.py**, а **b.py** импортирует **a.py**. Ниже — пошаговая временная шкала при выполнении **import a**.

1. **Начало загрузки A:** выполняется **import a**. Finder находит **a.py**, создаётся объект модуля **A**, он **сразу** помещается в **sys.modules["a"]**. Начинается выполнение **a.py** (loader.exec_module).
2. **В a.py встречается «import b»:** управление переходит к загрузке модуля **b**. В **sys.modules** ключа **"b"** ещё нет.
3. **Загрузка B:** создаётся объект модуля **B**, он помещается в **sys.modules["b"]**. Начинается выполнение **b.py**.
4. **В b.py встречается «import a»:** интерпретатор смотрит в **sys.modules["a"]** — ключ **"a"** уже есть (объект A создан на шаге 1). Возвращается **этот же объект A** — но выполнение **a.py** ещё **не завершено** (оно приостановлено на шаге 2 в ожидании загрузки b). То есть объект A «частично загружен»: код **a.py** до строки **import b** уже выполнен, код после — ещё нет.
5. **В b.py обращение к a.attr:** если **attr** определён в **a.py** **ниже** строки **import b**, то атрибута **attr** у объекта A ещё нет — **AttributeError**. Если **attr** определён **выше** **import b**, обращение сработает.

**Итог:** при циклическом импорте порядок выполнения кода критичен. Всё, что в **a.py** идёт **после** **import b**, в момент выполнения **b.py** ещё не выполнено — соответствующих атрибутов у **a** нет.

**Граничный случай: атрибут определён до import b.** Если в **a.py** сначала идёт **x = 1**, затем **import b**, то к моменту выполнения **b.py** атрибут **a.x** уже существует (строка **x = 1** выполнена до перехода к загрузке b). В таком порядке обращение к **a.x** в **b.py** на верхнем уровне сработает. Проблема возникает только когда атрибут в **a** определён **ниже** строки **import b** — тогда при обращении из **b** к **a.attr** атрибута ещё нет. Поэтому при циклическом импорте либо определяют все нужные другим модулям атрибуты **до** любых **import** других модулей цикла, либо используют отложенный импорт в том модуле, который обращается к «недогруженному» атрибуту.

**Решения:**

- **Отложенный импорт:** выполнять **import a** или **from a import x** не на верхнем уровне модуля **b**, а **внутри функции**, которая вызывается уже после того, как оба модуля полностью загружены (например, при первом вызове функции из b, которая использует a). Тогда к моменту обращения A уже полностью инициализирован.
- **Реструктуризация:** вынести общую зависимость в третий модуль **C**; **A** и **B** импортируют только **C**, цикл A↔B разрывается.
- **Dependency Injection:** передавать нужный объект (или модуль) в функцию/класс извне, а не импортировать модуль с ним на верхнем уровне.

#### Пример: Dependency Injection (обход циклического импорта)

Вместо импорта модуля **a** на верхнем уровне **b.py** зависимость передаётся аргументом — цикл разрывается: **b** не импортирует **a** при загрузке.

```python
# a.py
import b
def fa(helper=None):
    helper = helper or b.get_default_helper()  # или передаём снаружи
    return helper.do_something()
x = 1

# b.py — без «import a»; зависимость от «a» передаётся извне
def get_default_helper():
    import a  # отложенный импорт, если нужен по умолчанию
    return a
def fb(amod=None):
    if amod is None:
        import a
        amod = a
    return amod.x

# Точка входа (main или тесты) собирает зависимости и передаёт их
# main.py
import a, b
result = b.fb(amod=a)  # передаём модуль a в b.fb — цикл при загрузке не возникает
```

Так **a** и **b** могут вызывать друг друга через объекты/модули, переданные параметрами, без взаимного импорта на верхнем уровне.

#### Пример: циклический импорт и отложенный импорт

**Проблемный вариант (AttributeError при определённом порядке):**

```python
# a.py
import b       # 1) Начали выполнять a.py; 2) переходим к загрузке b
def fa():
    return b.fb()
x = 1          # 3) Эта строка выполнится только после полной загрузки b

# b.py
import a       # 4) a уже в sys.modules — возвращается частично загруженный a
def fb():
    return a.x # 5) В момент выполнения b.py у a ещё нет атрибута x (a.x не выполнен!)
# При вызове b.fb() из a после загрузки — a.x уже есть, всё ок.
# Но если из b при инициализации обратиться к a.x — AttributeError: module 'a' has no attribute 'x'
```

**Исправление: отложенный импорт в b.py**

```python
# b.py — импорт a перенесён внутрь функции
def fb():
    import a   # Импорт в момент вызова fb() — к этому времени a.py полностью выполнен
    return a.x
```

**Альтернатива: реструктуризация — вынести общее в третий модуль**

```python
# common.py
x = 1

# a.py
import common
import b
def fa():
    return b.fb()

# b.py
import common
def fb():
    return common.x  # Нет цикла: a и b зависят только от common
```

### __path__ и __package__

- **`__path__`** — атрибут пакета, список каталогов (и при необходимости zip), в которых ищутся подмодули. Можно изменять для расширения поиска (например, для namespace packages или плагинов).
- **`__package__`** — имя пакета, к которому принадлежит модуль; используется для разрешения относительных импортов. У модуля, запущенного как скрипт (**`python script.py`**), **`__name__ == "__main__"`**, а **`__package__`** может быть **`None`**; при запуске как **`python -m pkg.mod`** интерпретатор устанавливает **`__package__`** в **`"pkg"`**, чтобы относительные импорты в **pkg.mod** работали.

### pkgutil.extend_path

**`pkgutil.extend_path(pkg_path, name)`** — расширяет **`__path__`** пакета другими каталогами, найденными по тому же имени пакета в **sys.path**. Используется для **namespace packages** в «старом» стиле (до PEP 420), когда пакет мог состоять из нескольких каталогов с одним именем.

**Запомните:** при **AttributeError** или **ImportError** в цепочке импортов проверяйте циклы A→B→A и используйте отложенный импорт или реструктуризацию.

#### Практический вывод: когда отложенный импорт, когда реструктуризация

- **Отложенный импорт (import внутри функции):** подходит, когда цикл простой (A↔B), и обращение к «другому» модулю нужно только внутри нескольких функций. Плюс: минимум изменений в структуре. Минус: импорт выполняется при каждом вызове функции (кеш **sys.modules** смягчает — модуль не перезагружается, но поиск в **sys.modules** и привязка имени происходят).
- **Реструктуризация (вынос общего в третий модуль C):** подходит, когда A и B действительно зависят от одной и той же сущности (константы, общие типы, утилиты). Плюс: явная зависимость, нет цикла. Минус: нужно выделить общий код в отдельный модуль и возможно перенести туда часть логики.
- **Определение атрибутов до импорта:** если в **a.py** все атрибуты, нужные **b**, определены **до** строки **import b**, то обращение к **a.attr** из **b** на верхнем уровне сработает. Подходит только когда таких атрибутов мало и они не зависят от **b**. При росте числа связей лучше отложенный импорт или реструктуризация.

## §24c. Модульная система: ModuleSpec и __main__

**Цель:** знать, что такое объект **ModuleSpec** (**`__spec__`**): атрибуты **origin**, **loader**, **submodule_search_locations**, **parent** — и как на его основе реализовать кастомный импорт (интерфейсы **importlib.abc**: **MetaPathFinder**, **PathEntryFinder**, **Loader**). Понимать разницу между запуском **`python script.py`** и **`python -m package.module`**: во втором случае корректно задаётся **`__package__`** и текущая директория для импортов.

---

### ModuleSpec (__spec__)

У каждого загруженного модуля есть атрибут **`__spec__`** — объект **ModuleSpec** (или **None** у модулей, созданных без spec, например старых расширений). В нём хранятся:

| Атрибут | Описание |
|---------|----------|
| **name** | Полное имя модуля (строка), например **"os"**, **"pkg.submod"** |
| **loader** | Объект загрузчика (Loader), у которого вызывается **exec_module(module)** |
| **origin** | Откуда загружен: путь к файлу .py, строка **"frozen"**, **"built-in"** и т.д. |
| **submodule_search_locations** | Для пакетов: список каталогов (или путей) для поиска подмодулей; аналог **__path__** |
| **parent** | Имя родительского пакета (строка или **None** для модуля верхнего уровня) |
| **cached** | Путь к скомпилированному .pyc (если применимо) |
| **has_location** | Есть ли физическое расположение (файл/архив) |

**importlib.abc** задаёт абстрактные классы **MetaPathFinder** (метод **find_spec**), **PathEntryFinder**, **Loader** (методы **create_module**, **exec_module**) — их можно реализовать для импорта из нестандартных источников (БД, сеть, виртуальная ФС).

### __main__ и флаг -m

- **`python script.py`** — скрипт выполняется как главный модуль; **`__name__ == "__main__"`**, **`__file__`** — путь к скрипту, **`__package__`** часто **None**; текущая директория для **sys.path** — каталог скрипта.
- **`python -m package.module`** — модуль выполняется как **__main__**; **`__name__ == "__main__"`**, **`__package__`** устанавливается в **"package"**; текущая директория обычно корень проекта (откуда запущена команда). Относительные импорты в **package.module** работают корректно.

**Запомните:** для пакетов с относительными импортами запускайте точку входа через **`python -m package.module`**, а не через **`python package/module.py`**.

#### Пример: разница python script.py и python -m

```python
# В модуле mypkg/main.py:
# print("__name__ =", __name__)
# print("__package__ =", __package__)
# from . import submod  # относительный импорт

# Запуск: python mypkg/main.py
# __name__ = __main__
# __package__ = None  → from . import submod вызовет ImportError (нет текущего пакета)

# Запуск: python -m mypkg.main (из каталога выше mypkg)
# __name__ = __main__
# __package__ = mypkg  → относительный импорт работает
```

#### runpy.run_module — запуск модуля как скрипта из кода

```python
import runpy
# Эквивалент команды: python -m mypkg.main
runpy.run_module("mypkg.main", run_name="__main__")
```

#### Паттерн if __name__ == "__main__": зачем и когда выполняется

Часто в конце модуля пишут:

```python
if __name__ == "__main__":
    main()  # или тесты, или CLI
```

**Когда это выполняется:** атрибут **__name__** модуля при **импорте** устанавливается в **имя модуля** (например, **"mypkg.main"**). При запуске файла **как скрипта** (**python mypkg/main.py** или **python -m mypkg.main**) интерпретатор устанавливает **__name__** в **"__main__"**. Поэтому блок **if __name__ == "__main__":** выполняется **только** когда файл запущен как точка входа (скрипт или **-m**), и **не** выполняется при **import mypkg.main** из другого модуля. Так в одном файле совмещают «модуль с функциями/классами» и «скрипт для запуска»: при импорте загружаются только определения, при запуске — дополнительно вызывается **main()** или тесты.

**Запомните:** используйте **if __name__ == "__main__":** для кода, который должен выполняться только при запуске файла, а не при импорте.
