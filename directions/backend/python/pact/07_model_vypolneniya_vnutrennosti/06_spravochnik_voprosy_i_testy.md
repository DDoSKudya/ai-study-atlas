### §23. Объектная модель

1. **В чём разница между тождеством и равенством? Как их проверять?**
   <details>
   <summary>Ответ</summary>
   **Тождество** — «один и тот же объект в памяти»: проверяют **`is`** или **`id(a) == id(b)`**. **Равенство** — «значения считаются равными по смыслу типа»: проверяют **`==`** (вызывается **`__eq__`**). Два объекта могут быть равны (**`==`** True), но разными (**`is`** False).
   </details>

2. **Почему для проверки на None рекомендуют `x is None`, а не `x == None`?**
   <details>
   <summary>Ответ</summary>
   **PEP 8** явно требует сравнения с синглтонами (None, True, False) через **`is`** / **`is not`**. **`is`** проверяет тождество — быстрее и не зависит от переопределения **`__eq__`** у сравниваемого объекта. None в языке один (синглтон), поэтому **`x is None`** однозначно и идиоматично.
   </details>

3. **Что такое hashable? Какой контракт должен соблюдаться между __hash__ и __eq__?**
   <details>
   <summary>Ответ</summary>
   **Hashable** — объект, который можно использовать как ключ dict и элемент set: у него есть **`__hash__`** и **`__eq__`**. Контракт: если **`a == b`**, то обязательно **`hash(a) == hash(b)`**. Хеш не должен меняться, пока объект живёт (неизменяемость по смыслу). Иначе поиск в dict/set станет некорректным.
   </details>

4. **Почему tuple (1, [2, 3]) нельзя использовать как ключ словаря?**
   <details>
   <summary>Ответ</summary>
   Кортеж hashable **только если все его элементы hashable**. Список **`[2, 3]`** — изменяемый и не hashable. Поэтому **`hash((1, [2, 3]))`** вызовет **TypeError: unhashable type: 'list'**; такой кортеж нельзя класть в set или использовать ключом dict.
   </details>

5. **Что такое кеш мелких целых в CPython? Почему на это нельзя полагаться в коде?**
   <details>
   <summary>Ответ</summary>
   В CPython целые от **-5 до 256** кешируются: при создании такого числа возвращается ссылка на уже существующий объект. Поэтому **`100 is 100`** может быть True. Это **деталь реализации**; в Jython, PyPy и т.д. кеша может не быть или диапазон другой. Для сравнения чисел всегда используйте **`==`**.
   </details>

6. **Когда использовать type(obj) is Type, когда isinstance(obj, Type)?**
   <details>
   <summary>Ответ</summary>
   **`isinstance(obj, Type)`** учитывает наследование (подклассы проходят) — для обычной проверки типа предпочтительно. **`type(obj) is Type`** — жёсткая проверка «ровно этот класс»; подклассы не проходят. Второй вариант нужен, когда надо явно отсечь подклассы (например, разное поведение для «ровно list» и для MyList).
   </details>

### §24. Импорт (базовый)

7. **В каком порядке формируется sys.path при запуске интерпретатора?**
   <details>
   <summary>Ответ</summary>
   (1) Стартовая директория: при **python script.py** — каталог скрипта (или ''); при **python -m pkg.mod** — '' (текущая рабочая директория). (2) **PYTHONPATH** (каталоги из переменной окружения). (3) Каталоги из механизма **site** (site-packages, стандартная библиотека).
   </details>

8. **Чем отличается `import mod` от `from mod import name` с точки зрения пространства имён?**
   <details>
   <summary>Ответ</summary>
   **`import mod`** загружает модуль и привязывает в текущем пространстве имя **mod** к объекту модуля. **`from mod import name`** загружает модуль, но в текущем пространстве привязывает только **name** (к **mod.name**); имя **mod** в текущем пространстве не появляется. Модуль в обоих случаях один раз в **sys.modules**.
   </details>

9. **Что такое __all__ и как он влияет на `from mod import *`?**
   <details>
   <summary>Ответ</summary>
   **`__all__`** — необязательный атрибут модуля (список строк-имён). При **`from mod import *`** если **__all__** задан, импортируются **только** перечисленные в нём имена; если не задан — все имена, не начинающиеся с **`_`**. Нужен для явного объявления публичного API и ограничения «звёздного» импорта.
   </details>

10. **Что такое пакет? Чем классический пакет отличается от namespace package?**
    <details>
    <summary>Ответ</summary>
    **Пакет** — модуль-каталог, содержащий подмодули. **Классический пакет** — каталог с **`__init__.py`**; при импорте выполняется **__init__.py**, есть **__path__**. **Namespace package** (PEP 420) — каталог **без** **__init__.py**; несколько каталогов с одним именем в разных местах **sys.path** объединяются в один пакет с общим **__path__** (список всех этих каталогов).
    </details>

11. **В каких случаях возможен относительный импорт (from . import ...)?**
    <details>
    <summary>Ответ</summary>
    Относительный импорт возможен **только** в коде, загружаемом как часть пакета, т.е. когда у модуля задан **`__package__`** (не None). В скрипте, запущенном как **python script.py**, **__package__** обычно **None** — относительный импорт использовать нельзя (ImportError).
    </details>

12. **Для чего нужен importlib.import_module(name)?**
    <details>
    <summary>Ответ</summary>
    **importlib.import_module(name)** загружает модуль по **строковому** имени (например, **"json"**, **"pkg.submod"**) и возвращает объект модуля. Эквивалент **import** в рантайме: полезно для плагинов, динамической загрузки по имени из конфига или БД.
    </details>

### §24a. Импорт углублённо (meta_path, zipimport, загрузка из файла)

13. **Какую роль играет sys.meta_path при импорте? Что возвращает finder?**
    <details>
    <summary>Ответ</summary>
    **sys.meta_path** — список **MetaPathFinder** (или совместимых). При **import name** интерпретатор по очереди вызывает у каждого finder **find_spec(name, path=None)**. Первый, вернувший не **None**, даёт **ModuleSpec**; по нему создаётся модуль и вызывается **loader.exec_module(module)**. Если все вернули None — **ModuleNotFoundError**.
    </details>

14. **Для чего нужны sys.path_hooks и sys.path_importer_cache?**
    <details>
    <summary>Ответ</summary>
    **sys.path_hooks** — список фабрик (callable): для каждого элемента **sys.path** (строка-путь) по очереди вызывается фабрика; та возвращает **PathEntryFinder** для этого пути (или выбрасывает ImportError). **path_importer_cache** — кеш «путь → finder или None»; при повторном обращении к тому же пути hook’и не вызываются снова.
    </details>

15. **Как загрузить модуль из файла по произвольному пути (не из sys.path)?**
    <details>
    <summary>Ответ</summary>
    **importlib.util.spec_from_file_location(name, path)** создаёт **ModuleSpec**; **importlib.util.module_from_spec(spec)** — объект модуля; перед **exec_module** нужно зарегистрировать его в **sys.modules[name]**; затем **spec.loader.exec_module(module)** выполняет код. Так модуль загружается из любого файла по пути.
    </details>

16. **Как внутри zip-архива ищется .pyc для модуля pkg.mod?**
    <details>
    <summary>Ответ</summary>
    ZipImporter ищет один из двух вариантов: (1) путь **pkg/mod.py** — компиляция «на лету»; (2) путь **pkg/__pycache__/mod.cpython-<ver>.pyc** (например, **mod.cpython-312.pyc**). Имя в **__pycache__** формируется как **&lt;name&gt;.cpython-&lt;version&gt;.pyc**. Версия в имени должна совпадать с версией интерпретатора.
    </details>

17. **Что делает importlib.reload(module)? Какая типичная ловушка с `from mod import foo`?**
    <details>
    <summary>Ответ</summary>
    **reload(module)** заново выполняет код модуля (перезагружает его). Объект модуля в **sys.modules** обновляется. Ловушка: если в другом модуле было **from mymod import foo**, то имя **foo** там по-прежнему указывает на **старый** объект функции; он не обновится. После **reload** нужно снова выполнить **from mymod import foo** или обращаться как **mymod.foo**.
    </details>

18. **Для чего используется pkgutil.iter_modules(path)?**
    <details>
    <summary>Ответ</summary>
    **pkgutil.iter_modules(path=None)** — итератор по модулям, найденным по **path** (по умолчанию — путь текущего пакета по стеку). Возвращает кортежи **(loader, name, ispkg)**. Используется для обнаружения плагинов или получения списка подмодулей пакета без явного импорта каждого.
    </details>

### §24b. Циклические импорты, __path__, __package__

19. **Почему при циклическом импорте (a→b→a) в b может не быть атрибута a.attr?**
    <details>
    <summary>Ответ</summary>
    Объект модуля **a** помещается в **sys.modules["a"]** **до** завершения выполнения **a.py**. Когда **b** делает **import a**, он получает этот же объект **a**, но выполнение **a.py** в этот момент приостановлено (на строке **import b**). Код **a.py** **ниже** **import b** ещё не выполнен — атрибут **attr**, определённый там, ещё не существует. Отсюда **AttributeError**.
    </details>

20. **Какие три основных способа обойти циклический импорт?**
    <details>
    <summary>Ответ</summary>
    (1) **Отложенный импорт** — выполнять **import a** не на верхнем уровне **b**, а внутри функции, которая вызывается после полной загрузки обоих модулей. (2) **Реструктуризация** — вынести общую зависимость в третий модуль **C**; **A** и **B** импортируют только **C**, цикл разрывается. (3) **Dependency Injection** — передавать нужный модуль/объект аргументом в функцию или конструктор вместо импорта на верхнем уровне.
    </details>

21. **Что такое __path__ и зачем его можно изменять?**
    <details>
    <summary>Ответ</summary>
    **__path__** — атрибут **пакета**, список каталогов (и при необходимости zip), в которых ищутся подмодули. Его можно изменять для расширения поиска: например, подключать дополнительные каталоги с плагинами или реализовывать namespace packages (несколько каталогов с одним именем пакета).
    </details>

22. **Когда у модуля __package__ бывает None? Почему это важно для относительного импорта?**
    <details>
    <summary>Ответ</summary>
    **__package__** бывает **None** у модуля, запущенного как скрипт (**python script.py**): интерпретатор не считает его частью пакета. Относительный импорт (**from . import ...**) разрешается по **__package__** — без него «текущий пакет» не определён, поэтому **from . import ...** в таком модуле вызывает **ImportError**.
    </details>

23. **Для чего нужен pkgutil.extend_path(pkg_path, name)?**
    <details>
    <summary>Ответ</summary>
    **pkgutil.extend_path(pkg_path, name)** расширяет **__path__** пакета другими каталогами, найденными по тому же имени пакета в **sys.path**. Используется для **namespace packages** в «старом» стиле (до PEP 420), когда пакет мог состоять из нескольких каталогов с одним именем в разных местах.
    </details>

### §24c. ModuleSpec, __main__, python -m

24. **Какие атрибуты есть у ModuleSpec (__spec__) и для чего они нужны?**
    <details>
    <summary>Ответ</summary>
    **name** — полное имя модуля; **loader** — загрузчик (вызывается **exec_module**); **origin** — откуда загружен (путь к файлу, "built-in", "frozen" и т.д.); **submodule_search_locations** — для пакетов список каталогов поиска подмодулей (**__path__**); **parent** — имя родительского пакета; **cached** — путь к .pyc; **has_location** — есть ли физическое расположение.
    </details>

25. **Чем запуск `python script.py` отличается от `python -m package.module` с точки зрения __package__ и sys.path?**
    <details>
    <summary>Ответ</summary>
    При **python script.py**: **__package__** часто **None**, первый элемент **sys.path** — каталог, в котором лежит скрипт. При **python -m package.module**: **__package__** устанавливается в **"package"**, первый элемент **sys.path** — текущая рабочая директория (откуда вызвана команда). Поэтому относительные импорты в пакете работают при **-m** и обычно не работают при прямом запуске файла пакета как скрипта.
    </details>

26. **Когда выполняется блок `if __name__ == "__main__":` и когда — нет?**
    <details>
    <summary>Ответ</summary>
    Выполняется **только** при запуске файла как точки входа: **python script.py** или **python -m pkg.mod** — тогда **__name__** устанавливается в **"__main__"**. При **import** этого модуля **__name__** равен имени модуля (например, **"pkg.mod"**) — блок не выполняется. Так совмещают «модуль для импорта» и «скрипт для запуска».
    </details>

27. **Как из кода программы запустить модуль так же, как `python -m pkg.module`?**
    <details>
    <summary>Ответ</summary>
    **runpy.run_module("pkg.module", run_name="__main__")** — устанавливает **__name__ == "__main__"** и выполняет код модуля. Эквивалент команды **python -m pkg.module** из кода.
    </details>

### §25. Память и GIL

28. **Когда в CPython объект удаляется (освобождается память)? Что такое reference counting?**
    <details>
    <summary>Ответ</summary>
    В CPython у каждого объекта есть **счётчик ссылок**. Он увеличивается при присваивании имени, добавлении в контейнер, передаче в функцию и т.д.; уменьшается при удалении имени, выходе из scope, удалении из контейнера. Когда счётчик становится **нулём**, объект недостижим — вызывается **__del__** (если есть), затем память освобождается. Явного «free» нет.
    </details>

29. **Зачем нужен циклический сборщик мусора (gc), если есть reference counting?**
    <details>
    <summary>Ответ</summary>
    **Reference counting** не освобождает **циклические** ссылки (A ссылается на B, B на A): счётчики у A и B не станут нулём. **gc** находит такие недостижимые из «корней» группы объектов и освобождает их. Для обычных объектов без циклов достаточно reference counting.
    </details>

30. **Что такое GIL и почему потоки не ускоряют CPU-bound задачу?**
    <details>
    <summary>Ответ</summary>
    **GIL (Global Interpreter Lock)** — глобальная блокировка: в один момент только **один** поток выполняет байткод Python. Потоки по очереди получают GIL; параллельного выполнения чисто Python-кода в одном процессе нет. Поэтому CPU-bound задачи в потоках не ускоряются (и могут замедляться из-за переключений). Для CPU-bound нужен **multiprocessing** (несколько процессов — несколько GIL).
    </details>

31. **В каких ситуациях поток освобождает GIL?**
    <details>
    <summary>Ответ</summary>
    При **ожидании I/O** (чтение/запись файлов, сеть, **time.sleep**); при вызове блокирующих системных вызовов; в C-расширениях — при явном освобождении (**Py_BEGIN_ALLOW_THREADS**). Во время выполнения чисто Python-байткода GIL переключается между потоками по квантам времени, но в один момент его держит только один поток.
    </details>

32. **Что делают gc.collect() и gc.get_count()?**
    <details>
    <summary>Ответ</summary>
    **gc.collect(generation=2)** — принудительный запуск сборки мусора (по умолчанию полная, по поколению 2). Возвращает число собранных объектов. **gc.get_count()** — кортеж счётчиков по поколениям (0, 1, 2); при достижении порогов сборка запускается автоматически.
    </details>

33. **Сколько поколений у объектов в gc и зачем они нужны?**
    <details>
    <summary>Ответ</summary>
    Три поколения (0, 1, 2). Новые объекты попадают в 0. Пережившие сборку переходят в следующее поколение. Поколение 0 проверяется чаще, 2 — реже. Цель — не тратить время на «долгоживущие» объекты при каждой мелкой сборке; молодые объекты часто быстро становятся мусором.
    </details>

### §26. Модуль copy

34. **Чем поверхностная копия (copy.copy) отличается от глубокой (copy.deepcopy) для списка с вложенным списком?**
    <details>
    <summary>Ответ</summary>
    **copy.copy** создаёт новый внешний список, но элементы — те же ссылки. Вложенный список **общий**: изменение **copy[1].append(5)** изменит и оригинал. **copy.deepcopy** рекурсивно копирует все достижимые объекты; вложенный список — новый объект, изменения в копии не затрагивают оригинал.
    </details>

35. **Что такое memo в __deepcopy__(self, memo) и зачем он нужен?**
    <details>
    <summary>Ответ</summary>
    **memo** — словарь **id(объект) → уже созданная копия**. Нужен при циклических ссылках (A→B→A): при повторной встрече того же объекта возвращается копия из memo, иначе рекурсия уйдёт в бесконечность. Пользовательский **__deepcopy__** должен проверять **id(self) in memo** и при копировании вложенных объектов передавать **memo** в **copy.deepcopy(attr, memo)**.
    </details>

36. **Когда для списка достаточно lst.copy() или list(lst), а когда нужен copy.deepcopy?**
    <details>
    <summary>Ответ</summary>
    **lst.copy()** или **list(lst)** — поверхностная копия: новый список, те же ссылки на элементы. Достаточно, когда вложенных изменяемых объектов нет или их общность с оригиналом допустима. **copy.deepcopy** нужен, когда есть вложенные списки/словари и их нельзя делить с оригиналом — иначе изменение в копии изменит и оригинал.
    </details>

37. **Зачем классу определять __copy__ и __deepcopy__?**
    <details>
    <summary>Ответ</summary>
    Чтобы задать **кастомное** поведение при **copy.copy(self)** и **copy.deepcopy(self)**: например, копировать не все атрибуты, или создавать объект другим способом (синглтон — вернуть **self**). **__deepcopy__(self, memo)** обязателен для корректной обработки циклических ссылок (использовать memo).
    </details>

### §27. Сериализация: pickle, __reduce__, __getstate__, marshal, shelve

38. **Почему нельзя загружать (unpickle) данные из ненадёжного источника?**
    <details>
    <summary>Ответ</summary>
    При десериализации pickle **выполняет код**: вызывает конструкторы и методы (например, **__setstate__**, callable из **__reduce__**), указанные в потоке данных. Злоумышленник может подставить в поток вызов произвольной функции — при загрузке выполнится вредоносный код (удаление файлов, RCE). Для ненадёжных данных используют JSON, XML и т.д.
    </details>

39. **Чем отличаются протоколы pickle 0, 3 и 5? Когда какой выбирать?**
    <details>
    <summary>Ответ</summary>
    **0** — текстовый (ASCII), обратная совместимость с Python 2, большой размер. **3** — бинарный, по умолчанию в 3.0–3.7, не совместим с Python 2. **5** (3.8+) — поддержка **PickleBuffer**, out-of-band data, эффективнее для больших буферов. Для максимальной скорости/размера в текущей версии — **pickle.HIGHEST_PROTOCOL**; для совместимости со старыми версиями — явно задать меньший протокол.
    </details>

40. **Что должен возвращать __reduce__ для корректной сериализации и восстановления объекта?**
    <details>
    <summary>Ответ</summary>
    Кортеж вида **(callable, args)** или **(callable, args, state)**. При загрузке вызывается **callable(*args)** — создаётся объект; при наличии **state** затем вызывается **__setstate__(state)** или state присваивается **__dict__**. **callable** должен быть доступен по имени (модуль.имя), чтобы pickle мог его найти при загрузке.
    </details>

41. **Когда использовать __getstate__/__setstate__ вместо __reduce__?**
    <details>
    <summary>Ответ</summary>
    **__getstate__**/**__setstate__** удобны, когда нужно **исключить** часть атрибутов из сериализации (временные, несериализуемые) или представить состояние иначе, чем **__dict__**. При загрузке **__setstate__(state)** восстанавливает объект. **__reduce__** нужен, когда объект создаётся не простым вызовом конструктора (например, синглтон, объект из кеша).
    </details>

42. **Почему marshal нельзя использовать для персистентного хранения данных?**
    <details>
    <summary>Ответ</summary>
    **marshal** — низкоуровневый формат для байткода в .pyc; **нестабилен** между версиями Python (и между сборками). Документация явно запрещает использовать его для персистентных данных и обмена между системами. Для хранения и обмена — **pickle** или другие форматы (JSON и т.д.).
    </details>

43. **Есть ли у shelve транзакции (ACID)? Когда вызывать sync() и зачем нужна внешняя блокировка?**
    <details>
    <summary>Ответ</summary>
    У **shelve** **нет** полноценных транзакций (нет ACID): откат к «состоянию до начала серии записей» невозможен. Запись может буферизоваться — до **sync()** или **close()** данные не обязательно видны другому процессу. **sync()** после критичных записей снижает риск потери при сбое. Блокировка файла зависит от бэкенда dbm; при записи из нескольких процессов возможны конфликты — нужна **внешняя** блокировка (fcntl.flock, файл-замок) или другое хранилище.
    </details>

44. **Как pickle ведёт себя с классами с __slots__?**
    <details>
    <summary>Ответ</summary>
    У классов с **__slots__** нет **__dict__**. По умолчанию pickle сериализует экземпляр через механизм, учитывающий слоты. При кастомной сериализации (**__getstate__**/**__setstate__**) нужно возвращать/присваивать значения по именам слотов (не через **__dict__**). При наследовании от класса без слотов может понадобиться комбинировать состояние с **super().__getstate__()**.
    </details>

45. **Как при pickle восстанавливается подкласс, а не только базовый класс?**
    <details>
    <summary>Ответ</summary>
    Pickle записывает **имя класса** (модуль.имя). При загрузке импортируется модуль и по имени берётся **именно этот класс** (подкласс); создаётся экземпляр подкласса. Атрибуты подкласса и базового восстанавливаются через **__dict__** или **__getstate__**/**__setstate__**. Подкласс может вызывать **super().__getstate__()** и **super().__setstate__(state)** для базовой части.
    </details>

### Сводные и граничные случаи

46. **Как проверить, загружен ли уже модуль? Как узнать, откуда он загружен?**
    <details>
    <summary>Ответ</summary>
    Загружен ли: **name in sys.modules**. Откуда загружен: **module.__spec__.origin** или **module.__file__** (у модулей из файла).
    </details>

47. **Почему после добавления объекта в set нельзя менять поля, участвующие в __hash__/__eq__?**
    <details>
    <summary>Ответ</summary>
    Поиск в set идёт по **hash(obj)** и **obj == other**. Если после **s.add(obj)** изменить такое поле, хеш или результат сравнения изменится — при **obj in s** поиск пойдёт в другое «ведро» или сравнение даст False, объект «потеряется» в множестве. Элементы set должны быть неизменяемыми по смыслу.
    </details>

48. **Что делать при AttributeError «module 'X' has no attribute 'Y'» в цепочке импортов?**
    <details>
    <summary>Ответ</summary>
    Часто это **циклический импорт**: модуль X ещё не до конца выполнен, атрибут Y определён ниже строки, где X импортирует другой модуль. Решения: отложенный импорт (import X внутри функции в том модуле, где нужен Y), реструктуризация (вынести общее в третий модуль), Dependency Injection или определение в X всех нужных другим атрибутов **до** любых импортов из цикла.
    </details>

## Типичные ошибки (Часть VII)

- **Путать `is` и `==` для чисел или строк:** для значений всегда используйте **==**; **is** — только для синглтонов (None, True, False) или когда нужна именно ссылка на один объект.
- **Использовать изменяемый тип как ключ dict или элемент set:** ключи и элементы множества должны быть hashable и неизменяемыми (например, не list/dict).
- **Запуск пакета как `python package/module.py`:** при относительных импортах в пакете используйте **python -m package.module**.
- **Импорт на верхнем уровне при циклической зависимости:** перенесите импорт внутрь функции (отложенный импорт) или реструктурируйте код.
- **Ожидание ускорения CPU-bound задач от threading:** в CPython потоки не дают параллелизма по байткоду из-за GIL; используйте **multiprocessing** или C-расширения, отпускающие GIL.
- **Путать shallow и deep copy:** изменение вложенного объекта после **copy.copy** меняет и оригинал; для полной копии — **copy.deepcopy**.
- **Загрузка pickle из ненадёжного источника:** риск выполнения произвольного кода; для внешних данных используйте JSON или другие безопасные форматы.

## Примеры из практики (Часть VII)

**Кеш с ключами-объектами:** если нужно кешировать по составному ключу (например, (user_id, resource_id)), используйте **tuple** из hashable элементов: **cache[(user_id, resource_id)] = value**. Не используйте список **[user_id, resource_id]** — list не hashable.

**Динамическая загрузка плагинов:** по имени модуля из конфига или БД: **plugin = importlib.import_module(plugin_name)**; затем **getattr(plugin, "run")()** — вызов функции **run** из плагина. Убедитесь, что путь к плагинам есть в **sys.path** или используйте **spec_from_file_location** для загрузки из произвольного каталога.

**Сохранение состояния приложения (доверенная среда):** **shelve.open("state.db")** — простой persistent dict; ключи — строки, значения — любые объекты, сериализуемые через pickle. При выходе вызывайте **db.sync()** и **db.close()**. Для ненадёжных данных используйте **json** или **sqlite3**.

**Копирование конфигурации перед изменением:** если конфиг — вложенный dict/list и вы хотите изменить копию без влияния на оригинал: **config_copy = copy.deepcopy(config)**; затем **config_copy["section"]["key"] = new_value**. **copy.copy(config)** даст общие вложенные объекты — изменение **config_copy["section"]** изменит и **config["section"]**.

**Запуск тестов или CLI из корня проекта:** из корня репозитория запускайте **python -m pytest tests/** или **python -m mypkg.cli**, чтобы **sys.path** содержал корень проекта и относительные импорты в пакетах работали без **sys.path.insert**.

## Проверка: как убедиться, что всё настроено (Часть VII)

- **Объектная модель:** в консоли выполнить **id(0), type(0), 0 is 0, [] is []** — убедиться, что **id** и **type** возвращают значения, **0 is 0** даёт **True**, **[] is []** — **False** (разные объекты). **hash(1), hash("a"), hash((1,2))** — без ошибок; **hash([1])** — **TypeError: unhashable type: 'list'**.
- **Импорт:** **import sys; print("sys.path" in dir(sys), "sys.modules" in dir(sys))** — **True**. **import importlib; importlib.import_module("json")** — без ошибок. Запуск **python -m json.tool** из консоли — без **ModuleNotFoundError**.
- **Память и GIL:** **import gc; gc.get_count(), gc.collect()** — возвращают кортеж и число. **import threading; t = threading.Thread(target=lambda: None); t.start(); t.join()** — поток запускается и завершается.
- **copy:** **import copy; copy.copy([1,2]), copy.deepcopy([1,[2]])** — без ошибок; проверить **copy.deepcopy([1,[2]])[1] is not [1,[2]][1]** — **True** (разные вложенные списки).
- **pickle:** **import pickle; pickle.loads(pickle.dumps({"a":1}))** — возвращает **{"a": 1}**. **pickle.HIGHEST_PROTOCOL** — число (например, 5 в Python 3.8+).

## Отладка и диагностика (Часть VII)

- **AttributeError при импорте:** если при импорте модуля **B** в коде **B** обращаются к атрибуту модуля **A**, а **A** импортируется в **B** — проверьте **циклический импорт**. Решение: отложенный импорт в **B** (внутри функции) или реструктуризация.
- **ModuleNotFoundError:** проверьте **sys.path** — есть ли в нём каталог с пакетом/модулем. Запуск через **python -m pkg.module** из корня проекта обычно решает проблему; либо **sys.path.insert(0, "/path/to/project")** (для скриптов).
- **TypeError: unhashable type:** объект (список, dict, set или класс с **__eq__** без **__hash__**) использован как ключ dict или элемент set. Используйте **tuple** из hashable элементов или реализуйте **__hash__** и **__eq__** с контрактом «равные — равный хеш».
- **Изменение «копии» меняет оригинал:** вы использовали **copy.copy** для контейнера с вложенными изменяемыми объектами. Используйте **copy.deepcopy** для полной независимой копии.
- **Потоки не ускоряют CPU-bound задачу:** из-за GIL только один поток выполняет байткод. Используйте **multiprocessing** или C-расширения, отпускающие GIL.
- **pickle.load падает или выполняет неожиданный код:** не загружайте pickle из ненадёжных источников. Для внешних данных используйте **json** или другие безопасные форматы.

## Резюме по Части VII

- **§23:** В Python всё — объект; **id**, **type**, **is** (тождество) и **==** (равенство); **hashable** — для ключей dict и элементов set; мелкие целые в CPython кешируются — не полагайтесь на **is** для чисел.
- **§24:** Импорт идёт по **sys.path** и **sys.meta_path**; модули кешируются в **sys.modules**; пакеты — каталоги с __init__.py или namespace packages; относительный импорт (.**, ..); **importlib** для динамической загрузки; циклические импорты обходятся отложенным импортом или реструктуризацией; **ModuleSpec** и **-m** для корректного __package__.
- **§25:** Память в CPython — reference counting + циклический gc; **GIL** ограничивает параллельное выполнение байткода одним потоком; для CPU-bound — multiprocessing; для I/O — потоки или asyncio.
- **§26:** **copy.copy** — поверхностная копия; **copy.deepcopy** — глубокая; **__copy__** / **__deepcopy__** для кастомного поведения.
- **§27:** **pickle** — сериализация с протоколами 0–5, **__reduce__** / **__getstate__** / **__setstate__** для кастомизации; не загружать из ненадёжных источников; **marshal** — внутренний формат; **shelve** — persistent dict на pickle.
