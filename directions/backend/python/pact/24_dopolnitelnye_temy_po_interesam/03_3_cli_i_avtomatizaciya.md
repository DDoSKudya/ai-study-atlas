## 3. CLI‑утилиты и автоматизация (§143)

### 3.0. Зачем превращать скрипты в CLI (очень медленно)

#### 3.0.1. Боль от «одноразовых» скриптов

Типичный сценарий:

1. Вам нужно **быстро что‑то почистить/переименовать/перегнать** → вы пишете скрипт `rename_files.py`.
2. Часть параметров вы **хардкодите прямо в код**:
   - путь к папке;
   - маску файлов;
   - префикс/суффикс.
3. Через месяц нужно сделать то же самое, но чуть иначе:
   - другая папка;
   - другая маска;
   - другое правило имени.
4. Вы:
   - копируете скрипт «на всякий случай»;
   - меняете захардкоженные значения;
   - через год у вас **несколько похожих файлов**, и вы путаетесь, какой из них «последний».

Это:

- неудобно;
- плохо читается через время;
- невозможно использовать другими людьми (им нужно лезть в код и что‑то править).

#### 3.0.2. Что даёт нормальный CLI

Лучший путь:

- выделить **чистую функцию**:
  - с входными параметрами (`pattern`, `prefix`, `path`…);
  - без захардкоженных путей и магических значений;
- поверх неё построить **CLI‑обёртку**:
  - аргументы командной строки;
  - help‑строка;
  - разумные значения по умолчанию.

Результат — **одна утилита**, которую удобно вызывать:

```bash
python -m tools.rename --pattern "*.txt" --prefix "old_"
```

Преимущества:

- вы **не трогаете код** для каждого запуска — меняете только аргументы;
- утилиту легко отдать коллеге: «запусти вот так…»;
- через полгода всё ещё понятно, как ей пользоваться (см. `--help`).

#### 3.0.3. Цель раздела

После этого раздела вы:

- будете понимать разницу между:
  - «скрипт для себя»;
  - «CLI‑утилита, которой можно делиться»;
- научитесь:
  - делать простейший CLI на `argparse` (встроенный модуль);
  - строить более удобные CLI на `click`/`typer`;
  - аккуратно вызывать внешние программы через `subprocess`;
  - использовать `rich`, чтобы утилиты были «человечнее».

### 3.1. Базовый `argparse` (стандартный инструмент)

#### 3.1.1. Идея простыми словами

`argparse` — это модуль из стандартной библиотеки, который:

- разбирает аргументы командной строки (`--pattern`, `--prefix`, флаги и т.п.);
- автоматически генерирует:
  - help‑строку (`-h/--help`);
  - сообщения об ошибках, если аргументы указаны неправильно.

То есть он превращает «сырой» список строк из терминала в **объект с параметрами**, удобный для кода.

#### 3.1.2. Пример CLI‑утилиты «переименовать файлы» (построчно)

```python
import argparse
from pathlib import Path


def rename_files(pattern: str, prefix: str) -> None:
    for path in Path(".").glob(pattern):
        new_name = path.with_name(prefix + path.name)
        print(f"{path} -> {new_name}")
        path.rename(new_name)


def main() -> None:
    parser = argparse.ArgumentParser(
        description="Переименовать файлы, добавив префикс."
    )
    parser.add_argument(
        "--pattern",
        default="*.txt",
        help="Глоб‑шаблон файлов (по умолчанию *.txt)",
    )
    parser.add_argument(
        "--prefix",
        required=True,
        help="Префикс к имени файла",
    )

    args = parser.parse_args()
    rename_files(pattern=args.pattern, prefix=args.prefix)


if __name__ == "__main__":
    main()
```

Разбор:

- `ArgumentParser(...)`:
  - создаёт объект, который знает, какие аргументы ожидаются;
  - `description` попадёт в `--help`.
- `add_argument("--pattern", ...)`:
  - описывает опцию `--pattern`;
  - `default="*.txt"` — значение по умолчанию;
  - `help="..."` — текст, который увидит пользователь в `--help`.
- `add_argument("--prefix", required=True, ...)`:
  - аргумент обязателен → если его не указать, `argparse` сам покажет ошибку.
- `args = parser.parse_args()`:
  - разбирает фактические аргументы из командной строки;
  - создаёт объект `args` c атрибутами `args.pattern`, `args.prefix`.
- `rename_files(...)`:
  - чистая функция, которая ничего не знает про `argparse`;
  - её можно использовать и из других частей кода, и в тестах.

#### 3.1.3. Мини‑упражнение

1. Скопируйте пример в файл `rename_cli.py`.
2. Запустите:
   - `python rename_cli.py --help` — посмотрите, какой help‑текст сгенерировался;
   - `python rename_cli.py --prefix test_` в папке с несколькими `.txt` файлами.
3. Попробуйте вызвать утилиту **без `--prefix`** и посмотрите на сообщение об ошибке.

#### 3.1.4. Плюсы и минусы `argparse`

**Плюсы:**

- входит в стандартную библиотеку → **никаких дополнительных зависимостей**;
- достаточно гибкий:
  - позиционные аргументы;
  - флаги (`--verbose`);
  - подкоманды (`git commit`, `git push` и т.д. можно делать через subparsers).

**Минусы:**

- код получается более «шумным», чем на `click`/`typer`;
- на сложных CLI с подкомандами конфигурация может разрастаться.

### 3.2. `click` и `typer`: декларативный CLI (через декораторы)

#### 3.2.1. Почему иногда лучше взять не `argparse`

Когда CLI становится:

- с подкомандами (`mytool convert`, `mytool sync`, `mytool status`);
- с кучей опций;
- с вложенной структурой,

код на `argparse` может становиться тяжёлым для чтения.

`click` и `typer` решают это так:

- вы описываете **обычные функции** с параметрами;
- добавляете сверху декораторы;
- библиотека сама превращает это в CLI.

#### 3.2.2. Click: CLI через декораторы

```python
import click
from pathlib import Path


@click.command()
@click.option("--pattern", default="*.txt", help="Глоб‑шаблон файлов.")
@click.option("--prefix", required=True, help="Префикс к имени файла.")
def rename(pattern: str, prefix: str) -> None:
    for path in Path(".").glob(pattern):
        new_name = path.with_name(prefix + path.name)
        click.echo(f"{path} -> {new_name}")
        path.rename(new_name)


if __name__ == "__main__":
    rename()
```

Разбор:

- `@click.command()` — помечает функцию как «команду CLI»;
- `@click.option(...)`:
  - описывает опции командной строки;
  - значения попадают в аргументы функции (`pattern`, `prefix`);
- внутри функции вы просто пишете бизнес‑логику.

Запуск:

```bash
python rename_click.py --pattern "*.log" --prefix "old_"
```

#### 3.2.3. Typer: те же идеи, но больше упора на type hints

```python
import typer
from pathlib import Path

app = typer.Typer()


@app.command()
def rename(
    pattern: str = "*.txt",
    prefix: str = typer.Option(..., help="Префикс к имени файла"),
) -> None:
    for path in Path(".").glob(pattern):
        new_name = path.with_name(prefix + path.name)
        typer.echo(f"{path} -> {new_name}")
        path.rename(new_name)


if __name__ == "__main__":
    app()
```

Особенности:

- `pattern: str = "*.txt"`:
  - строка по умолчанию;
  - тип подсказан прямо в сигнатуре;
- `prefix: str = typer.Option(...)`:
  - обязательная опция (многоточие `...` говорит, что значение нужно указать);
  - `help="..."` пойдёт в help.

Typer по аннотациям:

- понимает типы аргументов;
- строит красивую help‑строку;
- может генерировать автодокументацию.

#### 3.2.4. «Запомните» по `click`/`typer`

- `argparse` хорош, когда:
  - вам не хочется тащить зависимости;
  - CLI небольшой.
- `click` и `typer` удобнее, когда:
  - команд много;
  - хочется максимально «чистый» код.

Если вы уже уверенно пользуетесь type hints — **Typer** часто оказывается самым приятным вариантом.

### 3.3. `subprocess`, `shutil`, `pathlib`: обвязка вокруг других инструментов

#### 3.3.1. Зачем вызывать внешние программы

Иногда:

- задача уже отлично решается сторонним инструментом:
  - `ffmpeg` для видео/аудио;
  - `git` для работы с репозиториями;
  - `zip`/`tar` для архивации;
- вам нужно просто:
  - **удобно вызывать эти программы**;
  - обработать результат;
  - возможно, объединить несколько шагов в одном скрипте.

Здесь и нужны:

- `subprocess` — запуск внешних команд;
- `shutil` — копирование/перемещение файлов;
- `pathlib` — удобная работа с путями.

#### 3.3.2. Пример: заархивировать директорию

```python
import subprocess
from pathlib import Path


def compress_directory(path: Path) -> None:
    archive = path.with_suffix(".zip")
    subprocess.run(
        ["zip", "-r", str(archive), str(path)],
        check=True,
    )
    print(f"Создан архив: {archive}")
```

Обратите внимание:

- список `["zip", "-r", str(archive), str(path)]`:
  - каждая часть команды — отдельный элемент списка;
  - мы **не пишем** одну строку `"zip -r ..."` и не используем `shell=True` без нужды;
- `check=True`:
  - если команда вернёт ненулевой код выхода — будет исключение;
  - вы сразу увидите, что что‑то пошло не так, а не «тихо проигнорируете» ошибку.

#### 3.3.3. Пример: резервное копирование файла

```python
import shutil
from pathlib import Path


def backup_file(src: Path, dest_dir: Path) -> None:
    dest_dir.mkdir(parents=True, exist_ok=True)
    shutil.copy2(src, dest_dir / src.name)
```

Разбор:

- `dest_dir.mkdir(parents=True, exist_ok=True)`:
  - создаёт директорию (и родительские, если нужно);
  - не падает, если директория уже есть.
- `shutil.copy2`:
  - копирует файл;
  - старается сохранить метаданные (время изменения и т.п.).

Такие функции удобно вызывать **из CLI‑команд**, описанных в предыдущем подразделе.

#### 3.3.4. Типичные ошибки

- использовать `subprocess.run("rm -rf ...", shell=True)` без крайней необходимости;
- не проверять код возврата (нет `check=True`);
- подставлять пользовательский ввод в команду без проверки (риск инъекций).

### 3.4. `rich`: сделать терминал дружелюбным

#### 3.4.1. Зачем украшать вывод

Можно думать: «я же инженер, мне всё равно, красиво или нет».  
На практике:

- красивый и понятный вывод:
  - экономит время (видно прогресс, видно ошибки);
  - снижает раздражение пользователя;
- особенно важен в утилитах, которые:
  - работают долго;
  - делают несколько шагов;
  - используются не только вами.

#### 3.4.2. Пример с прогресс‑баром

```python
from time import sleep

from rich.console import Console
from rich.progress import track

console = Console()

for i in track(range(5), description="Обрабатываем элементы..."):
    sleep(0.5)

console.print("[green]Готово![/green]")
```

Что здесь происходит:

- `track(range(5), ...)`:
  - оборачивает iterable в прогресс‑бар;
  - на каждой итерации рисует обновление;
- `console.print("[green]Готово![/green]")`:
  - печатает текст зелёным цветом.

При запуске:

- вы видите движущийся прогресс‑бар;
- по окончании — зелёное «Готово!».

#### 3.4.3. Как комбинировать `rich` с `click`/`typer`

Обычно:

- в модуле создают один `Console()`;
- передают его в функции (или делают глобальным, если аккуратно);
- используют:
  - `console.print` вместо `print`;
  - прогресс‑бары для долгих операций;
  - таблицы (`rich.table.Table`) для аккуратного вывода результатов.

#### 3.4.4. «Запомните» по CLI

- отделяйте:
  - **чистую логику** (функции, которые не знают о терминале);
  - **слой CLI** (разбор аргументов, красивый вывод, ошибки);
- `argparse`/`click`/`typer` + `subprocess`/`shutil`/`pathlib` + `rich` дают вам:
  - мощный набор для превращения разовых скриптов в **инструменты, которыми приятно пользоваться**.
