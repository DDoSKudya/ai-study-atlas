## 4. Декораторы углублённо (§144)

### 4.0. Напоминание базовой идеи (по‑детски)

#### 4.0.1. Что такое декоратор на человеческом языке

Декоратор — это **функция, которая принимает функцию и возвращает новую функцию**.

Аналогия:

- у вас есть «чистая» функция `готовить_пиццу`;
- вы хотите:
  - перед каждым вызовом сказать «приняли заказ»;
  - после — «заказ готов».
- вместо того чтобы каждый раз писать эти две строки вокруг вызова,
  вы один раз пишете **обёртку**, которая:
  - вызывает дополнительные действия;
  - потом вызывает исходную функцию;
  - возвращает результат.

```python
def simple_decorator(func):
    def wrapper(*args, **kwargs):
        print("До вызова")
        result = func(*args, **kwargs)
        print("После вызова")
        return result

    return wrapper
```

Применение:

```python
@simple_decorator
def greet(name: str) -> None:
    print(f"Hello, {name}!")
```

Строчка `@simple_decorator` сверху **эквивалентна**:

```python
greet = simple_decorator(greet)
```

То есть мы берём исходную функцию, оборачиваем её, и дальше под именем `greet` живёт уже **обёртка**.

#### 4.0.2. Зачем нужны декораторы

- повторяющиеся шаблоны:
  - логирование;
  - замер времени;
  - проверка прав;
  - кеширование;
- хочется:
  - описать шаблон **один раз**;
  - применять его к разным функциям **одной строчкой сверху**.

Главное — не превращать декораторы в «магический туман», а использовать их **осознанно и прозрачно**.

### 4.1. Декоратор без параметров (самый простой случай)

#### 4.1.1. Минимальный пример с логированием

```python
from functools import wraps
from typing import Callable, TypeVar, Any

F = TypeVar("F", bound=Callable[..., Any])


def log_calls(func: F) -> F:
    @wraps(func)
    def wrapper(*args: Any, **kwargs: Any) -> Any:
        print(f"Вызов {func.__name__} с args={args}, kwargs={kwargs}")
        result = func(*args, **kwargs)
        print(f"{func.__name__} вернула {result!r}")
        return result

    return wrapper  # type: ignore[return-value]
```

Разбор:

- `log_calls` — **декоратор**:
  - принимает функцию `func`;
  - внутри создаёт `wrapper`, который:
    - печатает информацию о вызове;
    - вызывает исходную функцию;
    - печатает результат;
    - возвращает его наружу;
  - возвращает `wrapper`.
- `@wraps(func)`:
  - копирует имя, докстроку и аннотации типов из `func` в `wrapper`;
  - это важно для:
    - IDE;
    - `help()`;
    - типизации.

Применение:

```python
@log_calls
def add(a: int, b: int) -> int:
    return a + b


add(2, 3)
```

Вывод:

- «Вызов add с args=(2, 3), kwargs={}»
- «add вернула 5»

#### 4.1.2. Мини‑упражнение

1. Напишите функцию `square(x)`, которая возвращает `x * x`.
2. Оберните её декоратором `log_calls`.
3. Вызовите с разными значениями и посмотрите, как меняется вывод.

#### 4.1.3. «Запомните» про простой декоратор

- декоратор без параметров — это функция `decorator(func) -> wrapper`;
- всегда используйте `@wraps`, чтобы не ломать метаданные;
- начинайте с **простых, читаемых декораторов**, которые легко объяснить словами.

### 4.2. Декоратор с параметрами (фабрика декораторов)

#### 4.2.1. Почему одного уровня иногда мало

Иногда нам нужно **настраивать поведение** декоратора:

- сколько раз повторить вызов;
- какой уровень логирования использовать;
- какой лимит времени выставить.

Тогда одного уровня `decorator(func)` не хватает. Нужен ещё один — **фабрика**, которая принимает параметры и возвращает конкретный декоратор.

#### 4.2.2. Пример: повторять вызов несколько раз

```python
from functools import wraps
from typing import Callable, TypeVar, Any

F = TypeVar("F", bound=Callable[..., Any])


def repeat(times: int = 1) -> Callable[[F], F]:
    def decorator(func: F) -> F:
        @wraps(func)
        def wrapper(*args: Any, **kwargs: Any) -> Any:
            result = None
            for _ in range(times):
                result = func(*args, **kwargs)
            return result

        return wrapper  # type: ignore[return-value]

    return decorator
```

Использование:

```python
@repeat(times=3)
def say(name: str) -> None:
    print(f"Hello, {name}!")
```

Разбор по слоям:

- `repeat(times=3)`:
  - вызывается при импорте модуля;
  - возвращает конкретный `decorator`, в котором `times` уже зафиксирован как `3`;
- `decorator(say)`:
  - принимает функцию `say`;
  - создаёт `wrapper`, который трижды вызывает `say`;
  - возвращает `wrapper`;
- дальше под именем `say` живёт уже `wrapper`.

#### 4.2.3. Мини‑упражнение

1. Напишите декоратор `limit_calls(max_calls: int)`, который:
   - разрешает вызвать функцию не более `max_calls` раз;
   - после превышения печатает сообщение и **не вызывает** функцию.
2. Примените его к простой функции `ping()`, которая печатает `"pong"`.
3. Проверьте, что на 4‑м вызове при `max_calls=3` исходная функция не срабатывает.

#### 4.2.4. «Запомните» про фабрики декораторов

- декоратор с параметрами = **фабрика** + обычный декоратор:
  - внешний уровень принимает параметры;
  - средний принимает функцию;
  - внутренний (`wrapper`) выполняет логику;
- если вы видите `@something(...)` — почти наверняка это такой случай.

### 4.3. Сохранение сигнатуры и работа с `inspect`

#### 4.3.1. Почему без `wraps` всё ломается

Без `@wraps(func)`:

- имя функции становится `wrapper`;
- у неё пропадает нормальная сигнатура;
- IDE и инструменты типизации видят только `(*args, **kwargs)`.

Это:

- усложняет отладку (в трейсбеках вы видите кучу `wrapper`);
- мешает автодополнению;
- ухудшает документацию.

Поэтому **почти всегда**:

- пишем:

```python
@wraps(func)
def wrapper(...):
    ...
```

#### 4.3.2. Использование `inspect.signature` для отладки

Иногда полезно:

- явно посмотреть сигнатуру функции;
- залогировать, с какими аргументами её вызывают.

```python
import inspect
from functools import wraps
from typing import Callable, Any, TypeVar

F = TypeVar("F", bound=Callable[..., Any])


def debug_signature(func: F) -> F:
    sig = inspect.signature(func)
    print(f"Регистрируем функцию {func.__name__}{sig}")

    @wraps(func)
    def wrapper(*args: Any, **kwargs: Any) -> Any:
        bound = sig.bind_partial(*args, **kwargs)
        print(f"Вызов {func.__name__}{sig} с {bound.arguments}")
        return func(*args, **kwargs)

    return wrapper  # type: ignore[return-value]
```

Применение:

```python
@debug_signature
def multiply(a: int, b: int, scale: int = 1) -> int:
    return (a * b) * scale
```

При первом импорте вы увидите:

- «Регистрируем функцию multiply(a: int, b: int, scale: int = 1)»

При вызове:

- «Вызов multiply(a: int, b: int, scale: int = 1) с {'a': 2, 'b': 3}»

### 4.4. Декораторы классов

#### 4.4.1. Зачем вообще декорировать класс

Иногда нужно:

- автоматически **регистрировать классы** (плагины, обработчики форматов);
- добавить всем классам **общее поведение** (например, красивый `__repr__`);
- не трогать код внутри классов, а лишь **обернуть их снаружи**.

Для этого подходят декораторы классов.

#### 4.4.2. Пример: реестр обработчиков

```python
REGISTRY: dict[str, type] = {}


def register(name: str) -> Callable[[type], type]:
    def decorator(cls: type) -> type:
        REGISTRY[name] = cls
        return cls

    return decorator
```

Использование:

```python
@register("csv")
class CSVExporter:
    def export(self, data: list[dict]) -> str:
        # реализация экспорта в CSV (упрощённо)
        return "..."
```

Где‑то в коде:

```python
def get_exporter(kind: str) -> type:
    return REGISTRY[kind]
```

Идея:

- при определении класса он **автоматически добавляется** в реестр;
- вы можете выбирать реализацию по строковому ключу.

#### 4.4.3. Пример: добавить `__repr__` всем классам

```python
def add_repr(cls: type) -> type:
    def __repr__(self) -> str:  # type: ignore[override]
        attrs = ", ".join(f"{k}={v!r}" for k, v in self.__dict__.items())
        return f"{cls.__name__}({attrs})"

    cls.__repr__ = __repr__  # type: ignore[assignment]
    return cls
```

Использование:

```python
@add_repr
class Point:
    def __init__(self, x: int, y: int) -> None:
        self.x = x
        self.y = y
```

Теперь:

```python
print(Point(1, 2))  # Point(x=1, y=2)
```

#### 4.4.4. Мини‑упражнение

1. Напишите декоратор класса `add_id`, который:
   - добавляет атрибут `id`, автоинкрементируемый при создании нового объекта;
   - например, `Point()` → `id = 1`, следующий `Point()` → `id = 2` и т.д.
2. Примените его к 2–3 классам и проверьте, что счётчик работает независимо/совместно (как вы решите).

### 4.5. Типичные ошибки с декораторами

#### 4.5.1. Список граблей

- забыли `@wraps` → плохие трейсы, сломанные аннотации;
- ловите и проглатываете все исключения (`except Exception`) внутри декоратора;
- декоратор меняет семантику функции настолько сильно, что **по имени/сигнатуре нельзя понять, что он делает**;
- используете декораторы для «магии», которую сложно объяснить новичку в команде;
- пишете слишком много логики внутри декоратора вместо того, чтобы вынести её в обычные функции.

#### 4.5.2. Здоровое правило

> Если вы не можете объяснить поведение декоратора **в 2–3 предложения простым языком** —  
> вероятно, он слишком сложный и его стоит упростить/разбить.

#### 4.5.3. «Запомните» по декораторам

- декораторы — это **инструмент для повторно используемых обёрток**, а не способ спрятать логику;
- сначала пишите **явный код**, потом, когда шаблон повторяется 3+ раз, выносите его в декоратор;
- всегда заботьтесь о читаемости: `@wraps`, понятные имена, короткие докстроки.
