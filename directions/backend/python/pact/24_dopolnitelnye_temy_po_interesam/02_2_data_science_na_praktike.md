## 2. Data Science на практике (§142)

### 2.0. Общая идея: меньше циклов, больше «массовых операций»

#### 2.0.1. Зачем вообще нужен отдельный раздел

В Части XXIII глобального плана уже есть **огромный обзор** по NumPy, Pandas, matplotlib, Seaborn и т.п.  
Но часто после прочтения остаётся ощущение:

- «я видел кучу функций, но **не понял, как этим пользоваться в реальной задаче**»;
- «я всё равно пишу `for`‑циклы и `if`, потому что так привычнее».

Эта часть нужна, чтобы:

- **переключить мышление**: от «обрабатываю элементы по одному» к «обрабатываю массивы целиком»;
- показать на маленьких, очень простых примерах, **как выглядит настоящий DS‑код**;
- сформировать **несколько базовых рефлексов**:
  - «а могу ли я здесь обойтись векторной операцией?»;
  - «а могу ли я сделать выборку маской, а не циклом и `if`?».

#### 2.0.2. Как обычно пишут новички

Код в лоб:

```python
result = []
for x in values:
    result.append(x * 2)
```

Это:

- **понятно мозгу** (мы привыкли думать циклами);
- **непонятно компьютеру** (Python‑цикл делает много лишней работы, не использует оптимизации).

В мире NumPy/Pandas так делать **невыгодно**, потому что:

- циклы на Python‑уровне медленные;
- операции с массивами написаны на C и используют:
  - SIMD‑инструкции процессора;
  - оптимальную работу с кешем;
  - быстрые библиотеки линейной алгебры (BLAS).

Поэтому мы хотим **заменять «миллион маленьких действий» одним большим действием** над массивом.

#### 2.0.3. Что вы должны вынести из этого раздела

После прочтения и практики вы:

- будете **узнавать ситуации, где нужен broadcasting**, и не бояться его использовать;

- будете понимать, **когда Pandas тормозит** и как его «подтолкнуть»:
  - тип `category`,

  - чтение по кускам (chunking);
- сможете быстро сделать **аккуратный график**, который не стыдно вставить в отчёт/презентацию.

### 2.1. NumPy: broadcasting (по шагам)

#### 2.1.1. Идея простыми словами

Представьте, что у вас есть:

- список температур по дням: `[0, 10, 20, 30]` (каждый элемент — день);
- одно число `+273.15`, которое вы хотите **прибавить ко всем температурам**.

В «обычном» мышлении:

```python
result = []
for t in temperatures_c:
    result.append(t + 273.15)
```

В NumPy:

```python
import numpy as np

temperatures_c = np.array([0.0, 10.0, 20.0, 30.0])
offset = 273.15  # скаляр

temperatures_k = temperatures_c + offset
print(temperatures_k)
```

Вы не пишете цикл вообще — **цикл спрятан внутри NumPy**, и он написан на C, очень эффективно.

#### 2.1.2. Как NumPy «видит» формы (shape)

Каждый массив NumPy имеет **форму** — кортеж размеров по осям:

- вектор длины 4 → `shape == (4,)`;
- матрица 2×3 → `shape == (2, 3)`.

Посмотрим:

```python
import numpy as np

v = np.array([1, 2, 3, 4])
m = np.array([[1, 2, 3],
              [4, 5, 6]])

print(v.shape)  # (4,)
print(m.shape)  # (2, 3)
```

Broadcasting работает тогда, когда **формы массивов можно «совместить»** по простому правилу.

#### 2.1.3. Правила broadcasting (ещё медленнее)

Правило можно запомнить так:

1. Сравниваем размеры по осям **справа налево** (как бы выравниваем их по правому краю).
2. Ось **совместима**, если:
   - размеры **равны**, или
   - один из размеров равен `1` (его можно «растянуть»).
3. Если есть хотя бы одна пара осей, которая не совместима, — будет ошибка.

Пример:

- массив `matrix` формы `(2, 3)`;
- массив `col_offsets` формы `(3,)`.

```python
import numpy as np

matrix = np.array([[1, 2, 3],
                   [4, 5, 6]])

col_offsets = np.array([10, 20, 30])  # shape (3,)

result = matrix + col_offsets  # shape (2, 3)
print(result)
```

Как NumPy думает:

- форма `matrix`: `(2, 3)`;
- форма `col_offsets`: `(3,)` → для выравнивания это как `(1, 3)`;
- сравниваем справа налево:
  - ось 0: `2` и `1` → совместимы (1 можно растянуть до 2);
  - ось 1: `3` и `3` → равны.

Итог:

- `col_offsets` копируется по первой оси (строкам);
- на каждой строке прибавляются `10, 20, 30`.

#### 2.1.4. Типичная ошибка и как её прочитать

Иногда вы увидите ошибку:

> `ValueError: operands could not be broadcast together with shapes (2,3) (4,)`

Это значит:

- вы пытались сложить массив формы `(2, 3)` и `(4,)`;
- NumPy пытался представить `(4,)` как `(1, 4)` и совместить:
  - ось 0: `2` и `1` → ок;
  - ось 1: `3` и `4` → **не ок**, нет ни равенства, ни единицы.

Лекарство:

- либо изменить саму задачу (вам действительно нужны 4 столбца?);
- либо явно изменить форму одного массива (`reshape`, `[:, None]` и др.).

#### 2.1.5. Мини‑упражнение

Попробуйте (в REPL или ноутбуке):

1. Создать массив `a = np.array([[1, 2], [3, 4]])` формы `(2, 2)`.
2. Создать `b = np.array([10, 20])` формы `(2,)`.
3. Посчитать `a + b` и проверить, как меняется каждая колонка.
4. Изменить `b` на `np.array([[10], [20]])` формы `(2, 1)` и посчитать `a + b` ещё раз.

Сравните результаты и объясните себе «на пальцах», что произошло.

#### 2.1.6. «Запомните» по broadcasting

- broadcasting — это способ **одной операцией обработать много элементов**, а не писать циклы;
- формы сравниваются **справа налево**, `1` можно растягивать;
- если что‑то не сходится по форме — сначала нарисуйте формы на бумаге и выровняйте их по правому краю.

### 2.2. NumPy: advanced indexing и булевы маски (думать «масками», а не `if`)

#### 2.2.1. Идея простыми словами

Вместо того чтобы писать:

```python
result = []
for x in data:
    if условие(x):
        result.append(x)
```

мы хотим:

```python
mask = условие_над_массивом(data)
result = data[mask]
```

То есть:

- сначала строим **булев массив** такой же длины, как `data`;
- затем **одним шагом выбираем все элементы**, где маска `True`.

#### 2.2.2. Простая маска: чётные числа

```python
import numpy as np

data = np.array([1, 2, 3, 4, 5, 6])
mask = data % 2 == 0  # чётные

even = data[mask]
print(mask)  # [False  True False  True False  True]
print(even)  # [2 4 6]
```

Здесь:

- `data % 2 == 0` вернул массив `bool` той же длины;
- `data[mask]` выбрал только элементы, где `mask` равно `True`.

#### 2.2.3. Индексация списком индексов

Иногда мы заранее знаем, какие именно позиции нас интересуют:

```python
idx = np.array([0, 2, 5])
subset = data[idx]  # элементы с индексами 0, 2, 5
print(subset)       # [1 3 6]
```

Это называется **advanced indexing по индексам**.

#### 2.2.4. Комбинация условий (и/или)

```python
mask = (data > 2) & (data < 6)
print(data[mask])  # [3 4 5]
```

Важно:

- используем `&` (логическое И) и `|` (логическое ИЛИ), а не `and`/`or`;
- круглые скобки обязательны, чтобы приоритет операций был правильным.

#### 2.2.5. Типичные ошибки

- Пишут `mask = data > 2 and data < 6` → **ошибка**, потому что `and` не работает поэлементно.
- Забывают скобки: `data > 2 & data < 6` → сработает не так, как вы ожидаете.
- Перемешивают индексацию масками и обычными индексами без понимания форм.

#### 2.2.6. Мини‑упражнение

В NumPy:

1. Создайте массив `arr = np.arange(10)` → `[0, 1, 2, ..., 9]`.
2. Постройте маску для:
   - чисел, кратных 3;
   - чисел, которые **не** кратны 2.
3. Получите два подмассива:
   - `multiples_of_3 = arr[mask_3]`;
   - `not_even = arr[mask_not_even]`.

Попробуйте объяснить себе словами, что вы сделали. Если можете — вы уже «думаете масками».

#### 2.2.7. «Запомните» по маскам и индексации

- **все условия фильтрации** старайтесь формулировать как выражения над массивами;
- результат фильтрации — это либо **булева маска**, либо **массив индексов**;
- циклы с `if` внутри постепенно заменяйте на «построение маски → применение маски».

### 2.3. Pandas: как не убить производительность (оптимизация, `category`, chunking)

#### 2.3.1. Почему Pandas часто «тормозит» у новичков

Типичный путь:

1. Читаем CSV в `DataFrame`.
2. Пишем кучу `df.apply(lambda row: ...)` по строкам.
3. Удивляемся, что на миллионе строк это работает **очень медленно и жрёт память**.

Проблема:

- `apply` по строкам делает **Python‑циклы внутри себя**;
- каждая строка — это по сути маленький объект‑словарь;
- интерпретатор тратит много времени на служебные операции.

Наша цель:

- как можно häufiger использовать **векторные выражения** (`df["a"] + df["b"]`);
- правильно выбирать типы (`category` для повторяющихся строковых значений);
- при больших файлах **не грузить всё в память сразу**, а читать по кускам (chunking).

#### 2.3.2. Векторизация вместо `apply`

```python
import pandas as pd

df = pd.DataFrame(
    {"price": [100, 200, 300],
     "discount": [0.1, 0.2, 0.0]}
)

# Плохо: apply по строкам (медленно на больших данных)
df["final_price_slow"] = df.apply(
    lambda row: row["price"] * (1 - row["discount"]),
    axis=1,
)

# Лучше: векторное выражение
df["final_price"] = df["price"] * (1 - df["discount"])
```

Разница:

- `apply`:
  - на каждой строке создаёт `Series`;
  - вызывает Python‑функцию `lambda`;
  - медленно и тяжело для памяти.
- векторное выражение:
  - берёт два столбца как массивы;
  - умножает и вычитает их **одной операцией на C‑уровне**.

Правило:

> Если вы в `apply` по строкам просто комбинируете значения столбцов арифметикой  
> или простыми условиями — почти всегда есть способ написать то же самое в векторном виде.

#### 2.3.3. Тип `category`: экономим память и ускоряем группировки

Столбец с городами:

```python
import pandas as pd

df = pd.DataFrame({"city": ["Moscow", "Berlin", "Moscow", "London"]})

df["city_cat"] = df["city"].astype("category")
print(df["city_cat"].dtype)  # category
print(df["city_cat"].cat.categories)
```

Что произошло:

- строковые значения заменились на **коды** (0, 1, 2, …);
- в памяти хранятся:
  - массив кодов;
  - таблица соответствия «код → строка».

Плюсы:

- меньше памяти, особенно если значение повторяется много раз (страна, город, тип товара);
- быстрые группировки/агрегации, потому что сравниваются **коды, а не строки**.

Когда использовать:

- столбцы с **ограниченным числом разных значений**:
  - пол (`M/F`);
  - статус (`new`, `in_progress`, `done`);
  - город/страна;
  - категория товара.

#### 2.3.4. Chunking: как работать с очень большими файлами

Проблема:

- есть огромный `big.csv` на десятки гигабайт;
- машина не тянет загрузить его целиком в память.

Решение:

- читать файл **порциями (чанками)**;
- каждую порцию обрабатывать и записывать результат отдельно.

```python
import pandas as pd

total_rows = 0

for chunk in pd.read_csv("big.csv", chunksize=50_000):
    total_rows += len(chunk)
    # здесь можно фильтровать/агрегировать chunk и писать результат в отдельный файл

print("rows:", total_rows)
```

Идея:

- в памяти одновременно находится только один `DataFrame` размером 50k строк;
- вы можете:
  - отфильтровать строки (например, только по нужному городу);
  - посчитать агрегаты и копить их;
  - писать очищенные данные в другой файл.

#### 2.3.5. Мини‑упражнение

1. Возьмите небольшой CSV (хотя бы 1000 строк).
2. Напишите версию обработки с `apply` по строкам.
3. Напишите векторную версию.
4. Используйте `%%timeit` в Jupyter или `time.perf_counter()` в обычном скрипте, чтобы сравнить время.

Цель — **руками увидеть**, насколько векторный код быстрее.

#### 2.3.6. «Запомните» по Pandas

- `apply(axis=1)` — красный флаг; сначала подумайте, можно ли переписать выражение в векторном виде;
- тип `category` полезен для столбцов с повторяющимися строками;
- большие файлы читаем **по кускам**, не пытаясь засунуть всё в память сразу.

### 2.4. Matplotlib и Seaborn: «чтобы графики было не стыдно показать»

#### 2.4.1. Зачем вообще об этом говорить

Многие разработчики:

- либо вообще не рисуют графики;
- либо рисуют «как получится»:
  - без подписей осей;
  - с мелким шрифтом;
  - с обрезанными подписями.

В результате:

- график **ничего не объясняет**;
- его стыдно показывать менеджеру или заказчику.

Наша цель — дать **минимальный чек‑лист**, чтобы:

- любой ваш график был понятен человеку,  
- даже если он видит его в первый раз и не читал ваш код.

#### 2.4.2. Базовый рецепт аккуратного графика

```python
import matplotlib.pyplot as plt
import seaborn as sns

sns.set_theme(style="whitegrid")  # аккуратная тема по умолчанию

fig, ax = plt.subplots(figsize=(8, 4))  # размер в дюймах (ширина, высота)

ax.plot([1, 2, 3, 4], [10, 15, 13, 17], marker="o")
ax.set_title("Пример динамики показателя")
ax.set_xlabel("День")
ax.set_ylabel("Значение")

plt.tight_layout()  # чтобы ничего не обрезалось по краям
plt.show()
```

Разбор:

- `sns.set_theme(style="whitegrid")` — включает аккуратный стиль (сеточка, цвета);
- `figsize=(8, 4)` — задаёт понятный размер (8×4 дюйма);
- `set_title`, `set_xlabel`, `set_ylabel`:
  - **обязательно** подпишите, что по осям и в каких единицах;
- `tight_layout()`:
  - просит matplotlib **поджать отступы**, чтобы подписи не обрезались.

#### 2.4.3. Мини‑чек‑лист перед тем, как показать график

- **Подписаны ли оси?**
  - «ось X» и «ось Y» ничего не говорят зрителю;
  - лучше «День» / «Продажи, шт» / «Температура, °C».
- **Понятны ли единицы измерения?**
  - проценты, штуки, доллары, секунды — укажите это в подписях.
- **Читаем ли шрифт?**
  - слишком маленький шрифт (= по умолчанию) на слайде будет нечитабельным;
  - при необходимости увеличьте размер (`plt.rcParams["font.size"] = 12`).
- **Нет ли визуального мусора?**
  - лишние линии сетки;
  - слишком много серий на одном графике.

#### 2.4.4. Мини‑упражнение

1. Нарисуйте простой график без подписей осей и названия.
2. Посмотрите на него через 5 минут и честно ответьте:
   - «что по оси X?»
   - «что по оси Y?»
3. Добавьте подписи, заголовок, увеличьте размер шрифта.
4. Снова посмотрите и сравните ощущения.

Это кажется мелочью, но в реальной работе **график — это инструмент коммуникации**,  
и от его читаемости часто зависит, поймут ли вашу идею.

#### 2.4.5. «Запомните» по визуализации

- базовый график = данные + подписи + читаемый размер + отсутствие мусора;
- `seaborn` помогает сразу получить аккуратный стиль;
- относитесь к графику как к **мини‑слайду презентации**, а не как к «быстрому черновику для себя».
