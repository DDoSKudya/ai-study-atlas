## 1. Выбор веб‑фреймворка и протокола (§141)

### 1.0. Общая идея: не «лучший фреймворк», а «подходящий под контекст»

#### 1.0.1. Почему вообще сложно выбрать

Частая ошибка начинающих — искать **«лучший web‑фреймворк»** или спорить «Django vs FastAPI» в отрыве от задач.  
Кажется, что где‑то есть **один правильный ответ**, а если ты его не знаешь — «значит, ты тупой».

На самом деле:

- у каждого фреймворка есть **сильные и слабые стороны**;
- почти всегда можно сделать проект и на Django, и на Flask, и на FastAPI;
- разница в том, **где будет легче жить через полгода–год**.

Поэтому цель этого раздела — не «навесить ярлыки», а дать **очень медленную и честную модель выбора**:

> «Если у меня вот такой проект и вот такая команда,  
> то с наибольшей вероятностью нам будет легче с X, а не с Y».

#### 1.0.2. Что такое «контекст проекта» (по‑детски)

Контекст — это ответы на несколько простых вопросов:

- **Размер команды.**
  - 1–2 человека.
  - 3–7 человек.
  - 8+ человек, разные роли (backend, frontend, devops).
- **Опыт команды.**
  - Python новичок / джун.
  - делали пару pet‑проектов;
  - работали в проде на Django/Flask/FastAPI.
- **Тип проекта.**
  - «толстая админка» (много форм, таблиц, фильтров);
  - «чистый API» (всё съедает мобильное приложение/SPA/другие сервисы);
  - «почти realtime» (чат, стриминг, много websocket‑ов).
- **Нефункциональные требования.**
  - нужно «сделать к дедлайну»;
  - важно легко поддерживать 3–5 лет;
  - ожидается большая нагрузка (много запросов, много одновременных пользователей).

Если честно ответить хотя бы на эти вопросы — вы уже на 50% сделали выбор.

#### 1.0.3. Модель выбора (пошагово, максимально медленно)

**Шаг 1. Определяем тип проекта.**

- **Классическое CRUD‑приложение:**
  - есть пользователи, роли, авторизация;
  - много страниц: списки, детали, формы «создать/изменить»;
  - сложная реляционная БД (много таблиц, связи).
  - Пример: CRM, админка для продукта, личный кабинет.
- **Лёгкий API‑шлюз:**
  - почти нет HTML, только JSON;
  - большая часть логики — «принять запрос → сходить в другие сервисы → вернуть ответ»;
  - важны простота и скорость разработки.
  - Пример: BFF для мобильного приложения, тонкий слой над несколькими микросервисами.
- **Нагруженный/«болтающийся» сервис:**
  - много одновременных подключений;
  - websocket‑ы, стриминг, long‑polling;
  - много внешних HTTP/БД‑запросов.
  - Пример: чат, стриминговый сервис, публичное API с высоким трафиком.

**Шаг 2. Смотрим на команду.**

- Если **у команды уже есть опыт Django**:
  - для классического CRUD‑приложения → почти всегда лучше взять Django;
  - вы выиграете месяц‑два разработки просто за счёт привычных инструментов.
- Если команда писала на **Flask**:
  - для лёгких API → логично остаться на Flask или плавно перейти к более структурированному стеку;
  - нет смысла брать FastAPI только потому, что он «модный».
- Если команда **уверенно владеет async**:
  - и у вас реально много параллельных запросов или websocket → ASGI‑стек оправдан.

**Шаг 3. Горизонт жизни проекта.**

- **Прототип / внутренний тул:**
  - главное — быстро сделать и поэкспериментировать;
  - можно позволить себе больше «костылей» и упрощений.
- **Долгоживущий продукт:**
  - важны миграции, админка, консистентность архитектуры;
  - здесь ценится зрелый фреймворк и устоявшиеся практики.

> Главное: **выбор фреймворка — это не экзамен на «умность»**,  
> а просто взвешивание плюсов и минусов в вашем конкретном контексте.

В следующих подразделах мы разберём Django, Flask и FastAPI **очень подробно и по шаблону**:  
цель → когда подходит → когда не подходит → пример → типичные ошибки → что запомнить.

### 1.1. Django: когда это «ваш выбор по умолчанию»

#### 1.1.1. Цель раздела

Понять:

- **в каких ситуациях Django почти всегда «в точку»;**
- какие реальные плюсы вы получаете;
- какие ограничения за это платите.

#### 1.1.2. Идея простыми словами

Django — это как **готовый конструктор дома**:

- у вас уже есть:
  - фундамент (настройки, структура проекта),
  - стены (модели, ORM),
  - окна/двери (формы, админка),
  - проводка (роутинг, middleware).
- вам не нужно заново решать:
  - «как хранить пользователей»,
  - «как делать миграции»,
  - «как устроена админка» — всё уже придумано.

Цена:

- вы играете **по правилам Django**;
- если вам нужно что‑то сильно «нестандартное», иногда придётся бороться с фреймворком.

#### 1.1.3. Когда Django особенно уместен

Запоминающаяся формулировка:

> «**Много форм + много таблиц + админка для менеджеров**» → думайте о Django в первую очередь.

Примеры:

- корпоративный кабинет с ролями (менеджер, оператор, админ);
- CRM‑система;
- админ‑панель для продукта (много справочников, отчётов).

Признаки, что вам сюда:

- вы нарисовали схему БД, и там **много связей** (OneToMany, ManyToMany);
- хотите, чтобы **некоторые пользователи могли править данные через UI**, а не только через API;
- вам важны **миграции**: вы будете развивать схему БД годами.

#### 1.1.4. Мини‑пример (по шагам, очень медленно)

1. Устанавливаем Django:

```bash
pip install django
```

2. Создаём проект:

```bash
django-admin startproject mysite
cd mysite
```

3. Создаём приложение:

```bash
python manage.py startapp blog
```

4. Пишем простейшую вьюху:

```python
# blog/views.py
from django.http import HttpRequest, HttpResponse


def hello(request: HttpRequest) -> HttpResponse:
    return HttpResponse("Hello, Django!")
```

5. Регистрируем маршрут:

```python
# mysite/urls.py
from django.contrib import admin
from django.urls import path

from blog.views import hello


urlpatterns = [
    path("admin/", admin.site.urls),
    path("hello/", hello),
]
```

6. Запускаем dev‑сервер:

```bash
python manage.py runserver
```

7. Открываем в браузере `http://127.0.0.1:8000/hello/` — видим «Hello, Django!».

Это **минимальный скелет**, но:

- у вас уже есть админка по адресу `/admin/`;
- есть система пользователей;
- есть миграции.

#### 1.1.5. Плюсы и минусы (по пунктам)

**Плюсы Django:**

- **богатая экосистема:**
  - Django REST Framework (DRF) для API;
  - Django‑allauth для аутентификации;
  - куча пакетов для типовых задач (блог, CMS, платежи и т.п.).
- **Админка из коробки:**  
  можно очень быстро дать нетехническим людям интерфейс для работы с данными.
- **Цельная архитектура:**
  - есть «официальный» способ делать многие вещи;
  - новичкам проще «вписаться» в существующий проект.

**Минусы/ограничения:**

- **меньше свободы:**
  - часть решений «зашита» во фреймворк;
  - чтобы сделать что‑то очень нетипичное, иногда приходится идти против стандартных паттернов.
- **исторически sync‑ориентированный стек:**
  - async‑поддержка развивается, но далеко не все расширения к ней готовы;
  - если вам нужен «чистый async‑стек» — проще смотреть на FastAPI/Starlette.

#### 1.1.6. Когда Django НЕ лучший выбор

- вы строите **маленький микросервис**, для которого:
  - нет админки,
  - мало таблиц,
  - основная задача — принять/отдать JSON;
- вы хотите **максимально лёгкий слой** поверх уже существующей бизнес‑логики;
- вы заранее знаете, что архитектура будет **нестандартной**, и вам нужна максимальная гибкость.

В таких случаях Django может оказаться **избыточным**.

#### 1.1.7. «Запомните» по Django

- если вы делаете **первый серьёзный web‑проект с БД и админкой** — начинать с Django **нормально и разумно**;
- Django **не устарел** и не «только для старых монолитов» — он решает свой класс задач;
- главное — честно ответить себе: «у меня действительно много форм/таблиц/админки?».

### 1.2. Flask: когда нужен лёгкий скелет

#### 1.2.1. Цель раздела

Понять, когда Flask:

- даёт вам **нужную свободу и простоту**;
- а когда он, наоборот, приведёт к хаосу.

#### 1.2.2. Идея простыми словами

Flask — это как **минимальный каркас здания**:

- есть несущие колонны (роутинг, базовый объект приложения);
- дальше вы сами решаете:
  - какую БД использовать;
  - как именно строить слои приложения;
  - какие расширения подключать.

Это даёт много свободы — но и много ответственности.

#### 1.2.3. Мини‑пример (шаг за шагом)

```python
from flask import Flask

app = Flask(__name__)


@app.get("/hello")
def hello():
    return {"message": "Hello, Flask!"}


if __name__ == "__main__":
    app.run(debug=True)
```

Разбор:

- `Flask(__name__)` — создаёт приложение;
- `@app.get("/hello")` — регистрирует обработчик GET‑запросов по пути `/hello`;
- функция `hello` возвращает dict → Flask сам превратит его в JSON‑ответ;
- `app.run(debug=True)` — запускает встроенный dev‑сервер (для продакшена нужен отдельный сервер, например Gunicorn).

#### 1.2.4. Плюсы Flask

- **очень простой старт**:
  - один файл;
  - несколько строк кода;
  - уже работает.
- **гибкость:**
  - вы сами выбираете ORM, логирование, структуру проекта;
  - легко встроить Flask в уже существующую архитектуру.
- **хорош как учебный инструмент:**
  - помогает понять, что именно делает web‑фреймворк;
  - многие туториалы начинают именно с Flask.

#### 1.2.5. Минусы и подводные камни

- легко получить **«зоопарк расширений»**:
  - каждое расширение со своим стилем и понятиями;
  - без общей архитектуры проект превращается в «слоёный пирог» из случайных решений.
- больше архитектурных решений на вашей совести:
  - как организовать модули;
  - как выстраивать слои (view → service → repository);
  - как тестировать.

Признак проблемы:

- вы часто ловите себя на мысли «а тут, наверное, есть расширение»;
- в итоге у вас десяток плагинов, которые не очень хорошо дружат друг с другом.

#### 1.2.6. Когда Flask уместен

- небольшой API‑сервис без сложной админки;
- прототип / pet‑проект;
- вы хорошо понимаете, **как сами хотите строить архитектуру**.

#### 1.2.7. «Запомните» по Flask

- Flask хорош, когда вы **осознанно хотите конструктор** и понимаете, как его собирать;
- для полностью зелёной джун‑команды без опыта иногда лучше начать с Django, где больше «rails»;
- если проект растёт, имеет смысл ввести **чёткую структуру** (blueprints, отдельные модули, слои) как можно раньше.

### 1.3. FastAPI и другие ASGI‑фреймворки: когда нужен async

#### 1.3.1. Цель раздела

Разобраться:

- чем FastAPI отличается от Django/Flask по сути;
- когда async действительно даёт выигрыш;
- какие риски вы берёте, переходя на async‑стек.

#### 1.3.2. Идея простыми словами

FastAPI — это фреймворк:

- поверх **Starlette** (ASGI‑фреймворк);
- заточенный под **JSON‑API** и **асинхронный код**;
- максимально использующий **аннотации типов**.

Его часто выбирают, когда:

- нужно **много параллельных запросов**;
- сервис сильно зависит от внешних API/БД;
- хочется иметь **автоматическую документацию** и схемы.

#### 1.3.3. Мини‑пример (медленно, с пояснениями)

```python
from fastapi import FastAPI

app = FastAPI()


@app.get("/hello")
async def hello(name: str = "world"):
    return {"message": f"Hello, {name}!"}
```

Обратите внимание:

- функция помечена **`async def`** — это корутина;
- параметр `name: str = "world"`:
  - тип **`str`** пойдёт в схему OpenAPI;
  - значение `"world"` — это значение по умолчанию;
- FastAPI сам:
  - разберёт query‑параметры;
  - проверит типы;
  - сформирует JSON‑ответ и документацию.

#### 1.3.4. Плюсы FastAPI

- **современный async‑подход (ASGI):**
  - один процесс может держать очень много одновременных подключений;
  - удобно для API‑шлюзов и сервисов, которые много ходят наружу.
- **Сильная интеграция с типами и Pydantic:**
  - входные/выходные модели описываются классами;
  - документация и валидация получаются «почти бесплатно».
- **Автодока из коробки:**
  - Swagger UI / ReDoc без дополнительного кода.

#### 1.3.5. Минусы и риски

- **async требует дисциплины:**
  - нельзя вызывать блокирующие функции (`time.sleep`, sync‑драйверы БД) внутри `async def`;
  - нужно понимать, что такое event loop и как с ним жить.
- **экосистема меньше, чем у Django, в части «готовой админки»:**
  - если вам нужно много админских UI — придётся писать/подключать отдельно.

#### 1.3.6. Когда FastAPI действительно нужен

- у вас настоящий **API‑шлюз** или «прокладка» между кучей внешних сервисов;
- вы ожидаете **много одновременных подключений** (особенно при использовании long‑polling или websocket‑ов);
- команда:
  - понимает async;
  - готова следить, чтобы не проскакивали блокирующие вызовы.

Если этого нет, **обычный WSGI‑стек (Django/Flask)** почти всегда достаточен и проще для жизни.

#### 1.3.7. «Запомните» по FastAPI

- FastAPI — не «обновлённый Flask» и не «замена Django», а **инструмент для async‑JSON‑API**;
- он даёт выигрыш в **нагрузке с большим числом соединений** и удобстве документирования;
- но за это вы платите необходимостью **строго следить за блокирующими операциями**.

### 1.4. Async vs sync, WSGI vs ASGI «на пальцах»

#### 1.4.1. Аналогия с кафе (ещё медленнее)

Представьте кафе.

- **Синхронная модель (WSGI):**
  - у вас есть несколько официантов;
  - каждый официант:
    - принимает заказ,
    - ждёт, пока кухня приготовит,
    - относит заказ клиенту,
    - только потом берёт следующий.
  - пока официант ждёт, он **ничего другого не делает**.

- **Асинхронная модель (ASGI):**
  - у вас есть один очень организованный администратор;
  - он:
    - принял заказ → передал на кухню → **не ждёт**, а идёт к следующему клиенту;
    - как только кухня кричит «готово!» — он подхватывает блюдо и несёт.
  - администратор **постоянно переключается между задачами**, почти не простаивая.

В терминах Python:

- WSGI: один запрос/подключение жёстко держит поток/процесс;
- ASGI: множество запросов делят один event loop, который «переключается» между задачами.

#### 1.4.2. Когда WSGI (sync) почти всегда достаточно

- обычные корпоративные CRUD‑сервисы;
- внутренние API без огромной публичной нагрузки;
- проекты, где:
  - вы в основном ходите в одну БД;
  - нет websocket‑ов и длинных стриминговых ответов.

Плюсы sync‑подхода:

- проще дебажить (стек вызовов «прямой», без переключений);
- меньше ловушек с блокирующими вызовами;
- большинство библиотек написано под sync‑мир.

#### 1.4.3. Когда ASGI (async) оправдан

- чат‑приложения с большим числом одновременных подключений;
- публичные API с высокой RPS и долгими внешними запросами;
- сервисы, активно использующие:
  - websocket;
  - long‑polling;
  - стриминговые ответы.

Но:

- нужно убедиться, что **вся цепочка ввода‑вывода async**:
  - HTTP‑клиенты (`httpx`, `aiohttp`);
  - драйверы БД (`asyncpg`, async‑варианты ORM);
  - файловые операции (`aiofiles`) и т.п.
- всё, что остаётся блокирующим, нужно выносить в:
  - другие процессы (очереди задач);
  - или `run_in_executor`.

#### 1.4.4. Простой чек‑лист выбора

Задайте себе вопросы:

1. Есть ли у меня **websocket/long‑polling/streaming**?  
   - Нет → скорее всего WSGI достаточно.
2. Сколько **одновременных запросов** я реально ожидаю?
   - Десятки/сотни → sync нормально.
   - Тысячи/десятки тысяч → есть смысл смотреть на ASGI.
3. Моя команда **понимает async и готова его поддерживать**?
   - Нет → лучше не закапываться в async без необходимости.

#### 1.4.5. «Запомните» по async vs sync

- async — это **инструмент под конкретный класс задач**, а не «обязательная мода»;
- пока вы не упёрлись в реальные проблемы с масштабируемостью —  
  **простое sync‑решение часто выигрывает за счёт предсказуемости и простоты**;
- если же у вас много одновременных соединений и внешних I/O —  
  тогда ASGI и FastAPI/Starlette становятся естественным выбором.
