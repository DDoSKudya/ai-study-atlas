## Справочник по части 23

### Краткая сводка понятий

| Понятие | Кратко |
| --- | --- |
| SSR | Серверный рендеринг на каждый (или почти каждый) запрос, затем гидрация и SPA‑навигация. |
| SSG | Генерация HTML на этапе билда, раздача как статика, часто через CDN. |
| ISR | Инкрементальное обновление статически сгенерированных страниц по таймеру/запросу. |
| CSR | Рендеринг HTML в браузере JS‑кодом; сервер отдаёт данные. |
| Hydration | Подключение JS‑логики к уже существующему HTML, чтобы сделать его интерактивным. |
| CDN/Edge | Распределённая сеть точек присутствия, раздающая статику и кеширующий HTML ближе к пользователю. |
| Revalidation | Проверка и обновление закешированного HTML/данных; основа ISR и стратегий кеширования. |
| BFF | Прослойка между фронтом и сервисами, используемая в том числе для SSR/ISR и агрегации данных. |

### Инструменты и фреймворки (кратко)

| Инструмент | Базовая модель | Особенности для архитектуры |
| --- | --- | --- |
| **Next.js** | SSR + SSG + ISR + CSR | Богатая поддержка per‑route стратегий, встроенный ISR, хорошая интеграция с Vercel/edge. Хорош для гибридов «маркетинг + приложение». |
| **Nuxt** | SSR + SSG + CSR | Схож с Next.js в подходе, но для Vue; удобен для проектов, где команда уже на Vue. |
| **Remix** | SSR + data loaders + CSR | Делает акцент на **серверной загрузке данных** и прогрессивном улучшении; удобно думать про «маршруты как форму», хорошо стыкуется с BFF. |
| **SvelteKit** | SSR + SSG + CSR | Лёгкий рантайм, хороший кандидат, если важен небольшой размер JS; поддерживает гибридные схемы. |
| **Astro** | SSG + Islands (частично SSR) | Делает ставку на статику и островную архитектуру; хорошо подходит для контентных сайтов с локальной интерактивностью. |

Важно: выбор конкретного инструмента **не меняет базовую архитектурную ось** (где и когда рождается HTML), но влияет на:

- удобство настройки per‑route стратегий (SSR/SSG/ISR/CSR),  
- интеграцию с edge/serverless‑хостингами,  
- размер JS‑бандлов и сложность гидрации.

### Краткая сводка решений

- **Используй SSG**, если:
  - контент меняется редко,  
  - SEO важен,  
  - нет индивидуальной персонализации на странице.  
- **Используй ISR**, если:
  - контент меняется, но не каждую секунду,  
  - страниц очень много,  
  - полные билды занимают слишком много времени.  
- **Используй SSR**, если:
  - нужен свежий и/или персонализированный контент на первом экране,  
  - SEO важен,  
  - есть ресурсы на инфраструктуру и кеши.  
- **Используй SPA/CSR**, если:
  - SEO вторичен,  
  - приложение богато интерактивностью и живёт в кабинете/админке.

   - Отметь, где можно кешировать HTML, где — данные, и какие метрики нужно мониторить.  
   - Подумай, что произойдёт при росте трафика в 10 раз.

3. **Сделай маленький эксперимент с инструментом.**  
   - Выбери любой фреймворк из списка (Next/Nuxt/Remix/SvelteKit) и создай простое приложение с двумя маршрутами:  
     - `/` — SSG или ISR,  
     - `/dashboard` — SSR или CSR.  
   - Посмотри в DevTools Network, как отличаются ответы сервера и какие файлы загружаются при первом входе и при переходе между страницами.

4. **Напиши мини‑ADR про выбор рендеринга.**  
   - В формате ADR (см. часть 32) опиши для одного продукта:

     - решение (какие стратегии рендеринга где используются),  
     - последствия (плюсы, минусы, что нужно мониторить).  
   - Это хорошая тренировка именно архитектурного мышления.

## Частые сценарии

### Сценарий 1. Документация и блог продукта

- **Задача**: быстро грузится, хорошо индексируется, статьи обновляются иногда.  
- **Решение**: SSG для всех статей; при росте объёма и частоты обновлений — ISR.  
- **Комментарий**: SSR здесь редко нужен, потому что персонализации почти нет.

### Сценарий 2. Интернет‑магазин

- **Задача**: хорошее SEO, быстрые страницы каталога, актуальные цены и остатки.  
- **Решение**:
  - ISR для категорий и карточек (с разумным интервалом ревалидации),  
  - SSR/SPA для корзины и чекаута,  
  - отдельный BFF для агрегации данных под SSR/ISR.  
- **Риски**: слишком частая ревалидация может перегрузить БД; нужно аккуратно выбирать интервалы.

### Сценарий 3. SaaS‑кабинет и маркетинг

- **Задача**: маркетинговые страницы с хорошим SEO + богатый кабинет, SEO внутри которого не нужен.  
- **Решение**:
  - маркетинг → SSG/ISR (возможно, отдельное приложение),  
  - кабинет → SPA с опциональным SSR для стартовой страницы (ускорить first view, но без SEO‑фокуса).  
- **Комментарий**: разделение по поддоменам/путям (`www.` и `app.`) помогает разграничить архитектуры.

### Сквозной план внедрения “per‑route SSR/SSG/ISR” (кеши, безопасность, фоллбеки)

Чтобы гибрид не превратился в хаос, важно заранее зафиксировать **где какой рендер** и как он обслуживается в проде.

#### 1) Карта маршрутов (главный артефакт)

Составь таблицу:

- маршрут,
- стратегия (SSR/SSG/ISR/CSR),
- что кешируем (HTML / data),
- требования к свежести,
- есть ли персонализация (важно для кешей).

#### 2) Кеш‑стратегия по уровням

- CDN/edge: HTML можно кешировать только если он **не персональный** (иначе риск утечек);
- data‑кеш: чаще безопаснее кешировать **данные**, а не HTML (если есть персонализация);
- для ISR: ограничь ревалидацию, иначе “освежение” превратится в DDoS на БД.

#### 3) Фоллбеки при деградации бекенда

План “что покажем, если API лёг”:

- ISR/SSG: отдаём последнюю успешную страницу (stale) + баннер “данные могут быть устаревшими”;
- SSR: либо деградируем на кеш/статический шаблон, либо быстро отдаём ошибку (но осознанно, с метриками).

#### 4) Метрики, без которых гибрид опасен

- TTFB/LCP по маршрутам (per‑route),
- cache hit ratio (HTML и data),
- частота ревалидации ISR и её latency,
- error rate SSR‑рендера и timeouts к API.

#### 5) План отката стратегии

Откат часто проще сделать “поменять стратегию рендера”, чем “откатить код”:

- фича‑флаг per‑route: SSR → SSG/ISR (временно) или SSR → CSR (для кабинета),
- хранить предыдущую конфигурацию маршрутов как код (git) и уметь быстро вернуть.

## Вопросы для самопроверки по части 23

1. Сравни SSR, SSG и ISR по трём осям: **когда генерируется HTML, где он хранится, как обновляется**.  
2. Какие **риски и ограничения** несёт «полный SSR для всего приложения»?  
3. В каком случае выгоднее использовать ISR вместо чистого SSG?  
4. Как сочетание **SSR/SSG/ISR с CDN и BFF** влияет на LCP и нагрузку на бекенд?  
5. Придумай свою архитектуру фронтенда для продукта «обучающая платформа с курсами и личным кабинетом» и обоснуй выбор рендеринга для каждой зоны.

<details><summary>Ответ</summary>

1. SSR: HTML генерируется **при запросе**, хранится в памяти/кеше (иногда не хранится совсем), обновляется через новые запросы и (опционально) кеш‑инвалидацию. SSG: HTML генерируется **при билде**, хранится как файлы, обновляется при новом билде. ISR: HTML сначала генерируется при билде или первом запросе, хранится как файл/в кеше, обновляется **частично и по расписанию/триггерам**.  
2. Риски: высокая нагрузка на SSR‑ноды и бекенд, сложность и стоимость инфраструктуры (кластер SSR, кеши, мониторинг), сложность отладки ошибок рендера, риск деградации производительности при росте трафика или усложнении шаблонов.  
3. Когда страниц много и они **квази‑статические** (например, товары, статьи), но их периодически обновляют: пересобирать весь сайт (SSG) дорого, а ISR позволяет «освежать» только востребованные страницы по мере запросов.  
4. CDN снижает TTFB и разгружает бекенд за счёт кеширования HTML и статики; BFF позволяет агрегировать данные и уменьшать число запросов к микросервисам. SSR/SSG/ISR определяют, **когда именно** эти кеши наполняются и обновляются, а значит — как распределяется нагрузка во времени.  
5. Возможный ответ: маркетинговый сайт и каталог курсов → SSG/ISR; страницы уроков → SSG или SSR (если важен контроль доступа/персонализация); личный кабинет и прогресс → SPA/SSR; админка → SPA. Обоснование — баланс SEO, производительности и сложности.

</details>

## Типичные ошибки и антипаттерны

- **SSR «по умолчанию» для всего**:
  - без кешей и CDN приводит к высокому TTFB и росту затрат;  
  - тяжело отлаживать и масштабировать.  
- **SSG всего каталога без учёта объёма**:
  - многочасовые билды, сложность деплоя, задержка обновления контента.  
- **Использование ISR без ограничений**:
  - агрессивная ревалидация для миллионов страниц → массовые обращения к БД и API.  
- **Отсутствие стратегии ошибок SSR/ISR**:
  - при падении бекенда пользователи получают пустые/битые страницы, а система не имеет fallback‑поведения.  
- **Смешение персонализированного и публичного HTML в одном кеше**:
  - утечка данных между пользователями.
