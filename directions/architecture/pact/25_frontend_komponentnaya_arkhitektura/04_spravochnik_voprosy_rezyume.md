## Справочник по части 25

| Тема | Коротко |
| --- | --- |
| Иерархия компонентов | Atomic Design (атомы, молекулы, организмы, шаблоны, страницы) помогает мыслить уровнями абстракции и переиспользования. |
| FSD и структура проекта | Feature‑Sliced Design и folder‑by‑feature организуют код по фичам и слоям, а не только по типам файлов; хорошо стыкуются с иерархией компонентов. |
| Презентационные компоненты | Отвечают только за UI и пропсы, не знают об API/store/роутере, идеально подходят для Storybook и тестов. |
| Контейнеры | Подключают данные и состояние (API, store, роутер), передают их в презентационные компоненты, инкапсулируют бизнес‑логику. |
| Compound components | Составные компоненты (Tabs, Accordion и т.п.) с общим контекстом, позволяющие строить гибкие иерархии без пропс‑дриллинга. |
| Дизайн‑система | Библиотека компонентов + токены + процессы и документация; архитектурный слой, задающий единый визуальный и поведенческий язык. |
| Токены дизайна | Программируемые константы дизайна (цвета, отступы, шрифты), используемые по всему UI вместо «магических чисел». |
| Error boundaries | Границы сбоев в дереве компонентов, которые ловят ошибки и показывают fallback, не роняя всё приложение. |
| A11y | Доступность: семантика, ARIA, клавиатура, контраст; должна быть встроена в компоненты дизайн‑системы. |

## Частые сценарии

### Сценарий 1. Небольшой внутренний админ‑инструмент

- **Контекст**: 1–2 разработчика, несколько таблиц и форм, нет строгих требований по бренд‑стилю.  
- **Решение**:
  - простая структура: `components/`, `pages/`;  
  - минимальная иерархия компонентов (атомы/молекулы неявные);  
  - лёгкое разделение контейнеров и UI там, где это явно нужно.  
- **Когда масштабировать**:
  - при росте числа экранов и повторяющихся паттернов стоит вынести `ui/` и общие компоненты.

### Сценарий 2. Продукт с публичным фронтом и кабинетом

- **Контекст**: маркетинговый сайт, документация, личный кабинет.  
- **Решение**:
  - дизайн‑система (отдельный пакет `ui` или `design-system`);  
  - основное приложение на FSD/folder‑by‑feature;  
  - чёткое разделение `shared/ui`, `entities`, `features`, `pages`.  
- **Выигрыш**:
  - единообразный UI во всех зонах;  
  - разные команды (маркетинг, продукт, платформа) работают поверх общих компонентов.

### Сценарий 3. Миграция с legacy‑SPA

- **Контекст**: монолитный SPA с гигантскими компонентами, слабый контроль над дизайном.  
- **Решение**:
  - шаг 1: выявить повторяющиеся элементы и вынести их в `shared/ui` (кнопки, поля, модалки);  
  - шаг 2: начать разделять контейнеры и презентационные компоненты для ключевых экранов;  
  - шаг 3: постепенно перестраивать структуру в сторону FSD/folder‑by‑feature;  
  - шаг 4: внедрять дизайн‑систему по мере появления критической массы общих компонентов.  
- **Важно**:
  - не пытаться «переписать всё с нуля»;  
  - двигаться итеративно, фиктируя архитектурные решения в ADR.

### Сквозной план внедрения дизайн‑системы (чтобы это не стало “ещё одной библиотекой”)

Дизайн‑система — это не набор компонентов, а **контракт и процесс**, иначе она деградирует в папку `components/` с вечными спорами.

#### 1) Определи scope v1 (что входит, а что нет)

v1 обычно включает:

- токены (цвета/типографика/отступы),
- базовые формы (Button/Input/Select),
- паттерны ошибок/загрузки (Alert/Skeleton/Empty),
- a11y‑правила “по умолчанию”.

Не включай в v1 “бизнес‑виджеты” (таблица заказов/графики) — это быстрее убьёт стабильность API.

#### 2) Контракт компонентов: что стабильно

Зафиксируй:

- публичные пропсы и семантика,
- правила темизации,
- политика breaking changes (SemVer + deprecation период),
- принципы “UI не ходит в API/store”.

#### 3) Ownership и gatekeeping

- назначь владельца/команду DS,
- введи review‑правило: изменения DS не мержатся без DS‑review,
- веди changelog (чтобы потребители понимали, что произошло).

#### 4) Миграция без big‑bang

Путь “основательно, но реалистично”:

1) сначала заменить 2–3 самых повторяющихся элемента (Button/Input/Modal),
2) затем “обвязки” (FormField, Validation, Error states),
3) затем сложные паттерны (Table, DatePicker) — только когда стабилизировались базовые.

#### 5) Метрики здоровья дизайн‑системы

- доля экранов, использующих DS‑компоненты,
- количество “локальных копий” компонентов (дубликаты),
- скорость миграции (components/week),
- число регрессий a11y/визуальных багов после обновлений DS.

## Вопросы для самопроверки по части 25

1. Как бы ты описал(а) разницу между **Atomic Design**, **FSD** и **folder‑by‑type** в одном абзаце?  
2. Как разделение на **контейнеры и презентационные компоненты** помогает тестированию и рефакторингу?  
3. Какие требования ты бы записал(а) к **компонентам дизайн‑системы** (минимум 3–4 пункта)?  
4. Где в твоём текущем или воображаемом проекте имеет смысл разместить **error boundaries**, и какие критерии выбора этих мест?  
5. Придумай пример «божественного» компонента и опиши, на какие части (контейнеры, UI‑компоненты) ты бы его разделил(а).

<details><summary>Ответ</summary>

1. Atomic Design даёт **иерархию уровней UI** (атомы → молекулы → организмы → шаблоны → страницы), FSD — **архитектуру приложения по фичам и слоям** (`app`, `pages`, `widgets`, `features`, `entities`, `shared`), а folder‑by‑type — более простую структуру «по типам артефактов» (`components/`, `services/`, `store/`), удобную в небольших проектах.  
2. Презентационные компоненты легко тестировать и показывать в Storybook, потому что они не зависят от API или store; контейнеры изолируют сложную логику, так что мы можем менять источники данных без переписывания UI. При рефакторинге мы часто трогаем только контейнер или только UI, что снижает риск поломок.  
3. Примеры требований:  
   - не содержат бизнес‑логики и не ходят напрямую в API/store;  
   - используют токены дизайна вместо магических значений;  
   - соблюдают a11y (семантика, ARIA, клавиатура, фокус);  
   - имеют стабильный API и версионируются по semver;  
   - задокументированы (истории/примеры, гайды применения).  
4. Error boundaries стоит размещать:  
   - на уровне корня (чтобы приложение не падало целиком);  
   - на уровне крупных страниц и виджетов, где возможны ошибки рендера (сложные диаграммы, интеграции, динамические компоненты).  
   Критерии: высокая сложность и риск ошибок, бизнес‑критичность, необходимость ограничить зону падения.  
5. Пример «божественного» компонента: `Dashboard.tsx` на 1000 строк, который загружает метрики, рендерит графики, формы фильтров, таблицу инцидентов и модалки. Разделение:  
   - `DashboardPage` (роут);  
   - `DashboardContainer` (данные и состояние);  
   - `DashboardLayout` (композиция виджетов);  
   - `MetricsChart`, `FiltersPanel`, `IncidentsTable`, `SummaryCards` (презентационные компоненты);  
   - отдельные контейнеры для сложных виджетов, если нужно.

</details>

## Типичные ошибки и антипаттерны

- **Божественные компоненты**:
  - компонент делает всё сразу (данные + бизнес‑логика + UI + навигация);  
  - сложно тестировать, страшно менять, нельзя переиспользовать части.  
- **Отсутствие структуры**:
  - папка `components/` с сотнями файлов без иерархии и слоёв;  
  - повторы логики и UI, нет очевидных общих компонентов.  
- **Смешение слоёв**:
  - доменная логика и API в компонентах дизайн‑системы;  
  - компоненты фич импортируют друг друга хаотично, нарушая границы FSD.  
- **Пропс‑дриллинг и отсутствие композиции**:
  - пропсы проходят через множество слоёв без надобности;  
  - нет контекста, compound components, разумной композиции.  
- **Игнорирование a11y и error boundaries**:
  - интерактивные элементы без семантики и фокуса;  
  - ошибка в одном виджете валит всё приложение.

#### Мини‑кейс: “протекла граница” и теперь невозможно переиспользовать компонент

Ситуация: команда сделала компонент `UserSelect` и положила его в “общие компоненты”, но внутри он:

- сам ходит в API,
- сам тянет токен/авторизацию,
- сам знает про формат конкретного бекенда.

**Что происходит дальше**:

- компонент нельзя использовать в админке/мобиле/другом приложении без копипаста;
- при смене API ломаются десятки мест;
- тестирование превращается в интеграционное даже там, где нужен простой UI‑тест.

**Как чинить** (архитектурно, а не “переписать”):

- разделить на:
  - `UserSelectView` (презентационный, чистый),
  - `UserSelectContainer` (данные/загрузка/кеш/ошибки),
  - адаптер данных (hook/service), который можно заменить.

Это ровно тот же принцип границ, что в бекенде (части 4–7): **контракт наружу, детали внутрь**.

### Что будет, если… (сводные граничные случаи)

- …пытаться сразу построить «идеальную» дизайн‑систему и FSD в маленькой команде?  
  Есть риск потратить слишком много времени на абстракции и папки, не успев сделать продукт; лучше эволюционировать: сначала разумная структура, потом дизайн‑система и строгий FSD.
- …игнорировать композицию и контейнеры, строя только «толстые» компоненты?  
  Любой рост функциональности будет дорого обходиться: сложнее понимать код, выше вероятность регрессий, тяжелее внедрять новые архитектурные решения (состояние, рендеринг, дизайн‑систему).

## Резюме части 25

- В этой части мы рассмотрели **компонентную архитектуру фронтенда**: от иерархии компонентов (Atomic Design и аналоги) до структуры проекта (FSD, folder‑by‑feature), разделения контейнеров и презентационных компонентов, дизайн‑систем, error boundaries и a11y.  
- Ключевая идея: **компонент — это архитектурная единица** с чёткой ответственностью и границами, а не просто файл с JSX/шаблоном.  
- Хорошая компонентная архитектура:
  - упрощает рефакторинг и рост продукта,
  - делает UI‑код более предсказуемым и переиспользуемым,
  - создаёт фундамент для дизайн‑системы и разделения зон ответственности между командами.  
- Дальше, в частях 26–27, мы свяжем эту картину с управлением состоянием и роутингом/загрузкой, а в частях 29 и далее — с монорепо, сборкой и командными процессами вокруг фронтенд‑архитектуры.

## Практика после части 25 (60–120 минут)

Цель практики — закрепить, что «компоненты» — это не про папки ради папок, а про **границы ответственности**, переиспользование и управляемость изменений.

### Практика 1. Разрежь “божественный компонент” на границы

Возьми любой реальный или воображаемый компонент‑монстр (например `Dashboard`, `Checkout`, `UserProfile`) и выпиши:

1) какие обязанности он сейчас несёт (данные, состояние, UI, навигация, ошибки),  
2) на какие части его логично разрезать:
   - **контейнер(ы)** (данные/состояние/интеграции),
   - **презентационные** (чистый UI),
   - **хуки/сервисы** (извлечение логики),
   - (опционально) **виджеты** (крупные блоки UI).

<details><summary>Ответ (шаблон результата)</summary>

- `UserProfilePage` (роут)  
- `UserProfileContainer` (загрузка user/settings, обработка ошибок, сохранение)  
- `UserProfileView` (UI, принимает props)  
- `useUserProfileForm` (валидация/маппинг)  
- `ProfileHeader`, `SecuritySettings`, `NotificationsSettings` (виджеты/организмы)

</details>

### Практика 2. Скелет структуры проекта: FSD vs folder-by-feature

Для того же сценария набросай файловую структуру (10–30 файлов максимум), выбрав один вариант:

- FSD (слои `pages/widgets/features/entities/shared`)  
или
- folder‑by‑feature (папки по фичам, внутри — UI/logic/api адаптеры).

Добавь 3 правила импортов (что запрещено) и 1 правило “public API” (как экспортировать наружу).

<details><summary>Ответ (пример правил)</summary>

- `features/*` не импортируют `pages/*`  
- `shared/ui` не импортирует `entities/*` и не ходит в API/store  
- `entities/*` не знают о роутере  
- Public API: наружу экспортируем только из `index.ts` модуля/слоя, прямые импорты “внутрь” запрещены

</details>

### Практика 3. Мини‑контракт дизайн‑системы (если она есть/планируется)

Сформулируй «минимальный контракт» DS из 8–12 пунктов:

- **что входит** (кнопки, формы, токены, иконки, accessibility‑практики),
- **что не входит** (доменные компоненты),
- **как версионируем** (SemVer, deprecation),
- **как выкатываем** (changelog, правила breaking changes),
- **как гарантируем a11y** (минимум 2 механизма: lint/tests/storybook‑a11y).

<details><summary>Ответ (минимум)</summary>

DS обязана иметь owner, публичный API, SemVer, changelog, токены и базовые компоненты, а также встроенную a11y‑политику; DS не содержит доменных сущностей и не ходит в API/store.

</details>
