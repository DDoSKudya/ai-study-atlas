## 14.5. Где хранить события: EventStoreDB, Kafka, обычная БД

### Цель раздела

Разобрать критерии выбора хранилища событий и понять, почему «Kafka — это event store» иногда правда, а иногда — ловушка; а «обычная БД» иногда достаточно.

### В этом разделе главное

- Event store должен поддерживать **append**, **чтение по stream**, **версионность/конкуренцию**, **порядок** (в пределах stream).
- Kafka — отличный event log, но его модель ретеншна/партиций влияет на ES.
- Специализированные продукты (EventStoreDB) дают удобные механики (streams, expected version, subscriptions).
- «Обычная БД» возможна, но придётся самим собрать много деталей (конкуренция, подписки, ретраи).

### Термины

| Термин | Определение |
|---|---|
| **Retention** | Сколько времени события хранятся (политика хранения). |
| **Compaction** | Сжатие лога (в Kafka — хранение последнего значения по ключу, не всегда совместимо с ES‑историей). |
| **Subscription** | Механизм подписки на поток событий для проекторов/интеграции. |

### Теория и правила

#### Базовые требования к event store (минимальный чек‑лист)

1. **Append‑only запись** событий.  
2. **Чтение по stream** (по идентификатору агрегата) в порядке.  
3. **Optimistic concurrency** (expected version).  
4. **Подписки/доставка** событий к проекторам (pull или push).  
5. **Наблюдаемость**: метрики лагов подписок, ошибки, ретраи.  

#### Вариант 1: EventStoreDB (и похожие)

Сильные стороны (концептуально):

- поток (stream) — первичная сущность;
- expected version — встроено;
- подписки и «догонялки» — удобны;
- модель хорошо совпадает с ES.

Компромиссы:

- отдельная технология (операционная нагрузка);
- нужно обучить команду;
- нужно думать про бэкапы/восстановление.

#### Вариант 2: Kafka как event log

Когда Kafka подходит:

- у тебя уже есть Kafka как инфраструктурный стандарт;
- события должны жить долго (или бесконечно) и быть доступны для реплеев;
- ты умеешь обеспечить привязку ключей так, чтобы порядок в пределах агрегата соблюдался (обычно key = aggregateId).

Где подводные камни:

- ретеншн может удалить историю (если настроен ограниченно) → реплей агрегата станет невозможен;
- партиции: порядок гарантируется только внутри партиции; нужно аккуратно выбирать ключи;
- компактирование (compaction) может разрушить «историю», если применено не по делу;
- «expected version» нужно реализовывать на уровне приложения/доп. хранилища, если нужен строгий контроль.

#### Вариант 3: обычная БД (PostgreSQL и т.п.)

Плюсы:

- проще стартовать, особенно для учебных/небольших систем;
- транзакции, индексы, резервное копирование знакомы.

Минусы:

- подписки/стриминг/обработка «с какого оффсета продолжить» — нужно делать самому;
- expected version можно сделать, но надо проектировать схему и блокировки аккуратно;
- при большом объёме событий нужна стратегия партиций, архивации, TTL.

#### Инструменты и платформы (по плану): EventStore, Axon Framework, Kafka

Глобальный план просит не только «где хранить», но и **чем обычно пользуются на практике**. Ниже — короткая ориентация (без “рекламы”, только смысл).

##### EventStoreDB / EventStore (как класс продуктов)

Это специализированные event stores, где:

- streams и подписки — **первоклассные сущности**,
- optimistic concurrency и expected version — **часть модели**,
- удобнее строить подписки на «все события» или «на stream‑категорию».

**Когда выбирать**:

- вы реально строите ES как базовую модель (а не «один экспериментальный агрегат»);
- важны удобные реплеи/подписки/механика конкуренции “из коробки”;
- команда готова вести отдельную инфраструктуру.

##### Axon Framework (Java‑экосистема)

Axon — это фреймворк, который помогает строить **CQRS + Event Sourcing** в приложении:

- даёт каркас для команд, агрегатов, событий, обработчиков;
- помогает с event store интеграцией и подписками;
- часто используется, когда команда в Java и хочет “платформенный” подход.

**Когда выбирать**:

- вы в Java, и хотите ускорить внедрение CQRS/ES единым подходом;
- готовы принять рамки фреймворка (его модель и способы).

**Когда осторожнее**:

- если вам нужен максимально “ручной” и минималистичный контроль;
- если команда не готова к фреймворку как к платформе (обучение, стандарты, стиль).

##### Kafka как event log

Kafka — сильная база для событийных потоков, и **иногда** она становится и event log для ES.

**Когда выбирать**:

- Kafka уже стандарт в компании;
- вы готовы обеспечить долгий ретеншн/архив и правильные ключи/партиции;
- у вас сильная компетенция по эксплуатации Kafka.

**Когда не стоит “натягивать”**:

- если Kafka у вас настроена как “очередь” с коротким ретеншном;
- если вы не готовы обеспечить порядок в пределах агрегата;
- если вы ожидаете expected version “само появится” без дополнительной логики.

#### Рост event log и стратегии хранения (архив, TTL, снапшоты) — практическая часть

План прямо говорит: **рост лога событий** — обязательная тема. Чтобы она была не “в воздухе”, держи практическую рамку.

##### Откуда берётся рост (простая оценка)

Даже грубая оценка помогает не ошибиться:

- 1 событие ≈ 0.5–2 КБ (payload + metadata)  
- 100 событий на заказ × 1 млн заказов = 100 млн событий  
- 100 млн × 1 КБ ≈ 100 ГБ (плюс индексы/овер‑хед хранилища)

Это не “страшилка”, а напоминание: ES требует **плана по объёму**.

##### Базовые стратегии (что реально применяют)

1) **Снапшоты** — уменьшают стоимость реплея, но не уменьшают историю.  
2) **Партиционирование / сегментация** — чтобы хранить и читать быстрее (часть 18 даст больше деталей).  
3) **Архивирование старых событий**:
   - “горячий” слой (последние N месяцев) — быстро доступен;
   - “холодный” слой (старше N месяцев) — в архиве, но доступен для расследований/редких реплеев.
4) **TTL/retention** (только если бизнес допускает):
   - опасно применять “просто потому что место кончается”;
   - допустимо для “низкоценной” истории, но тогда ES как «источник истины на века» уже не получается.

##### Важный граничный случай: удаление персональных данных

Event Sourcing конфликтует с идеей «удалить данные навсегда», потому что история — истина. Практический компромисс обычно такой:

- **не класть лишние PII в события** (минимизация);
- использовать **токены/идентификаторы**, а детали хранить в отдельном месте с политикой удаления;
- для неизбежных PII — продумать **шифрование/ключи** и процесс “crypto‑erase” (удаление ключа) как способ сделать данные нечитабельными.

Эта тема пересекается с частью 18 (данные и регуляторика), но для ES важно запомнить: **события живут долго → данные в событиях выбираем крайне осторожно**.

### Простыми словами

Хранилище событий — это не «куда складывать JSON».  
Это система, которая должна:

- хранить историю надёжно,
- отдавать её по частям и в порядке,
- помогать подписчикам не терять события,
- давать способ бороться с конкуренцией.

### Картинка в голове

**Event store как библиотека с каталожными карточками**:

- ты можешь быстро найти все события для `Order-123`;
- можешь продолжить чтение «с того места, где остановился»;
- библиотека не «перемешивает страницы» (порядок).

### Как запомнить

Если у хранилища нет:

- чтения по stream,
- контроля версии,
- понятной модели подписки,

то это будет **дорогое “DIY event store”**, даже если данные лежат где-то надёжно.

### Примеры

#### Сравнение вариантов (простая таблица)

| Критерий | EventStoreDB | Kafka как event log | Обычная БД |
|---|---|---|---|
| Streams «из коробки» | да | нет (ключи/партиции) | да (если спроектировать) |
| Expected version | да | обычно нет | можно сделать |
| Подписки | да | да (consumer groups) | DIY |
| Ретеншн | управляемый | критично настроить | управляемый |
| Сложность внедрения | средняя | высокая (если нет Kafka) / средняя (если есть) | низкая/средняя |

### Практика / реальные сценарии

- **Небольшой продукт**: PostgreSQL как event store на старте, затем миграция.
- **Крупная компания с Kafka**: Kafka как стандартный event log, отдельные проекции и строгие правила ключей/ретеншна.
- **Сильный фокус на ES**: специализированный event store, где модель совпадает с доменом.

### Типичные ошибки

- Использовать Kafka с коротким ретеншном «как очередь» и при этом ожидать реплеев ES.
- Не обеспечивать порядок событий внутри агрегата (разные ключи, разные партиции).
- Выбрать event store без продакшн‑готовой истории бэкапов/восстановления.

### Что будет, если…

- …история удалится из-за ретеншна: агрегаты станет невозможно восстановить; проекции можно перестроить частично, но истина потеряна.
- …порядок нарушится: реплей даст неправильное состояние (например, «оплата до создания заказа»).

### Проверь себя

1. Какие минимальные способности должно иметь хранилище, чтобы быть event store?
2. Почему ретеншн в Kafka — критический параметр для ES?
3. В чём главный риск «обычной БД» как event store?

<details><summary>Ответ</summary>

1. Append‑only запись, чтение по stream в порядке, контроль конкуренции (expected version или эквивалент), механизм подписок/доставки, наблюдаемость.  
2. Потому что ES рассчитывает на возможность восстановить состояние из истории. Если история удаляется, реплей невозможен.  
3. Большая часть «продуктовых» возможностей event store (подписки, оффсеты, ретраи, контроль порядка/версий) станет задачей вашей команды — это операционная и инженерная нагрузка.

</details>

### Запомните

Выбор event store — это выбор «сколько инфраструктуры мы готовы вести сами». ES быстро наказывает за недоговорённости про порядок, ретеншн и конкуренцию.
