### Чек‑листы (быстрое принятие решения и внедрение)

#### Чек‑лист 1. Нужны ли вам микрофронтенды вообще? (Decision checklist)

Отметь “да/нет”. Если “да” меньше чем в половине пунктов — микрофронтенды чаще всего будут лишней сложностью.

| Вопрос | Да/нет | Почему это важно |
| --- | --- | --- |
| Есть 3+ команды, и они реально мешают друг другу релизами? |  | без организационной боли микрофронтенды не окупаются |
| Границы доменов/владения стабильны (не меняются каждую неделю)? |  | иначе границы придётся постоянно перекраивать |
| Нужна независимая поставка хотя бы части фич? |  | иначе проще модульный монолит/монорепо |
| Вы готовы владеть shared‑слоем (DS/i18n/SDK) как продуктом? |  | иначе shared станет источником инцидентов |
| Вы готовы к усложнению эксплуатации (кэш/версии/совместимость)? |  | цена микрофронтендов — операционная сложность |

#### Чек‑лист 2. Пилот микрофронтендов (чтобы не “внедрить и утонуть”)

Минимальный пилот, который даёт сигнал “работает/не работает”:

| Шаг | Готово? | Критерий |
| --- | --- | --- |
| Выбран 1 домен для пилота (самый автономный) |  | изменения редко пересекаются с другими |
| Определён shell и его “тонкий” набор обязанностей |  | URL/ошибки/загрузка remotes/shared политики |
| Зафиксирован протокол `v1` (lifecycle/nav/data/errors) |  | документ в репо + примеры |
| Выбран способ интеграции и объяснены trade‑off’ы |  | не “по моде”, а по ограничениям |
| Включены contract checks + smoke композиции в CI |  | ловит несовместимость до прод |
| Добавлен fallback + тайм‑аут загрузки remote |  | нет “вечного спиннера/белого экрана” |
| Есть стратегия canary/rollback |  | независимые релизы без отката опасны |
| Определён измеримый успех |  | скорость релиза / число конфликтов / LCP/INP |

#### Чек‑лист 3. Поддержка в продакшене (чтобы система не деградировала)

| Риск | Контрмера |
| --- | --- |
| “Старый remote + новый host” | content‑hash + manifest + negotiation + fallback |
| CSS конфликты | изоляция root‑контейнера + минимум глобального CSS + владелец стилей |
| Дрейф дизайн‑системы | владелец shared‑пакета + SemVer + deprecation |
| Инциденты “между MF” | контракт наблюдаемости (мета + trace_id + события lifecycle) |

### Частые сценарии (как рассуждать)

#### Сценарий A. Одна команда, один фронт, релизы раз в день

Рекомендация:

- **скорее всего микрофронтенды не нужны**;
- лучше: модульный монолит, хорошая компонентная архитектура (часть 25), продуманный state (часть 26), роутинг/сплиттинг (часть 27).

Почему:

- микрофронтенды добавят сложности, но не решат организационную боль (её нет).

#### Сценарий B. Несколько команд, конфликт релизов, разные темпы изменений

Рекомендация:

- микрофронтенды **могут быть оправданы**, но начни с пилота:
  - 1 shell + 1–2 микрофронтенда,
  - контракт v1,
  - минимум общих зависимостей,
  - измерение перф и стабильности.

##### Пилот “по-взрослому”: пошаговый план (контракт, версии, LKG, откат)

Чтобы пилот действительно снимал риск, он должен проверять **не только рендер**, а всю эксплуатационную цепочку: доставка, кэш, совместимость, наблюдаемость.

1) **Выбери домен пилота** (не самый критичный, но реальный)  
   Пример: “Каталог” или “Профиль”, а не “Чекаут”.

2) **Опиши контракт протокола v1** (как файл рядом с кодом)  
   Минимум:
   - lifecycle: `mount/unmount`,
   - навигация: кто владеет URL и как делаем deep link,
   - данные: какие входные параметры/контекст (tenant, locale, auth) передаются,
   - ошибки: как remote сообщает об ошибке и какой fallback обязан показать host.

3) **Введи negotiation совместимости**  
   - поле `protocolVersion` (major),
   - правило: host монтирует remote только если major совместим, иначе показывает fallback.

4) **Сделай LKG (last known good)**  
   - хранить “последнюю стабильную версию remote”,
   - уметь быстро переключить remote на LKG без релиза host (фича‑флаг/manifest).

5) **Кэш и доставка** (самая частая реальная боль)  
   - `remoteEntry`/manifest кэшировать коротко,
   - чанки с hash кэшировать долго,
   - таймаут загрузки remote + понятный fallback.

6) **Наблюдаемость “на границе”**  
   - `trace_id`/correlation для переходов host↔remote,
   - метрики: время загрузки remote, error rate монтирования, доля fallback.

7) **Критерии успеха пилота (измеримые)**  
   - снизилось число конфликтов релизов между командами,
   - время релиза домена пилота сократилось,
   - LCP/INP не ухудшились,
   - есть быстрый откат (LKG) и он реально проверен.

#### Сценарий C. Нужно встроить внешнюю систему (легаси/партнёр)

Рекомендация:

- iframe + `postMessage` протокол + строгие ограничения;
- затем (если нужно “нативнее”) — миграция к более тесной интеграции.

### Вопросы для самопроверки (сводно)

1. В чём **главная причина** микрофронтендов и какая главная “цена” за них?  
2. Почему “общий глобальный store на всё” часто убивает смысл микрофронтендов?  
3. В каком случае Module Federation действительно даёт выигрыш, а в каком — просто добавляет сложности?  
4. Какие 3 контракта ты считаешь минимальными для работающей системы (lifecycle/nav/data/errors)?  
5. Как бы ты объяснил(а) продакшен‑команде, почему **кэш remote‑артефактов** — это риск совместимости?

<details><summary>Ответ</summary>

1. Причина — независимость команд и релизов. Цена — операционная сложность (сборки/версии/контракты/отладка) и риск перф‑деградации.  
2. Потому что общий store делает систему снова связанной: любая правка состояния влияет на всех, возникают общие инварианты и “скрытые зависимости”, независимость релизов пропадает.  
3. Выигрыш — когда нужно runtime‑подключение независимых сборок и shared‑зависимости без дубляжа (много команд, независимые релизы). Сложность — если релизы всё равно совместные и нет потребности в независимости: тогда проще монолит/монорепо.  
4. Минимум: lifecycle (mount/unmount) + навигация (URL/deep link) + данные/контекст (events/URL/context). Ошибки/fallback — крайне желательны, иначе один фрагмент “роняет” всё.  
5. Потому что пользователь может держать старый remote в кэше, а host уже обновлён. Если контракт изменился, совместимость ломается. Нужны content‑hash, стратегия инвалидации и версионирование протокола.

</details>

### Практика после части 28 (обязательное закрепление)

Сделай маленькое упражнение “на бумаге” или в любом своём проекте. Цель — не написать код, а научиться **принимать архитектурное решение и фиксировать контракты**.

1) **Выбери одну область продукта** (например, “Каталог” или “Профиль”).  
2) Опиши **границы микрофронтенда**:
   - что внутри (страницы/виджеты/данные),
   - что остаётся в shell,
   - какие зависимости считаешь shared.
3) Выбери **способ интеграции** (iframe / JS‑bundle / Web Components / Federation) и выпиши 2–3 trade‑off’а, почему так.  
4) Назови:
   - **одно ограничение** (операционная сложность, перф, отладка, версии),
   - **одну типичную ошибку**, которую ты заранее предотвратишь.
5) Зафиксируй **минимальный контракт v1** (текстом):
   - lifecycle (`mount/unmount`),
   - навигация (кто владеет URL),
   - канал данных (URL/events/backend),
   - поведение при ошибке (fallback).

#### Проверь себя по практике

1. Можешь ли ты назвать **один конкретный контракт**, который ты бы положил в репозиторий рядом с кодом (и зачем)?  
2. Можешь ли ты честно сказать, **какую боль** микрофронтенды решают в твоём сценарии (и что будет, если боли нет)?  
3. Можешь ли ты назвать **один измеримый критерий успеха** (например, скорость релиза, количество конфликтов, перф‑метрика)?

<details><summary>Ответ</summary>

1. Например, `contracts/mf-protocol-v1.md` с описанием `mount/unmount`, структуры `ctx`, событий `mf:navigate/mf:ready` и версии протокола. Это снижает “магические договорённости” и делает интеграцию проверяемой.  
2. Если боль — релизы и независимость команд, микрофронтенды могут помочь. Если боли нет, они почти наверняка добавят сложности без выгоды (хуже перф/отладка/релизы).  
3. Пример: “время от merge до релиза фичи в домене X сократилось с 7 дней до 2”, “количество конфликтов/регрессий между командами снизилось”, “LCP на странице /checkout не ухудшился после внедрения”.

</details>

### Типичные ошибки (список “анти‑паттернов”)

- **Микрофронтенды без боли**: одна команда, один релиз — внедрили “потому что надо как в FAANG”.
- **Контракт на словах**: нет версионирования, нет тестов композиции, нет политики shared‑версий.
- **Дублирование тяжёлых зависимостей** в каждом микрофронтенде без необходимости.
- **Глобальные стили** как “общий мусор” — всё ломается от любой правки.
- **Shell‑монолит**: все важные решения и логика в shell, микрофронтенды превращены в “виджеты без власти”.

### Резюме части

Микрофронтенды — это архитектурный подход, который особенно полезен, когда продукт и организация выросли настолько, что **команды мешают друг другу выпускать изменения**. Но это подход с ценой: нужно сознательно проектировать **границы**, фиксировать **контракты**, управлять **версиями и стилями**, и держать под контролем **производительность**.

Дальше по плану логично идти в **часть 29**: сборка, монорепо и **Module Federation** как практическая инфраструктура доставки и совместного использования кода.
