## Справочник по части 4

### Краткий справочник: слои, ошибки, эволюция

| Тема | Ключевые пункты |
|------|-----------------|
| **Слоёная архитектура (N‑tier)** | Логические слои внутри приложения: презентация → домен → данные → инфраструктура; зависимости только вниз; вертикальные срезы реализуют бизнес‑сценарии. |
| **Презентационный слой** | Принимает запросы, валидирует на уровне протокола, мапит на внутренние команды, формирует ответы; не содержит сложной бизнес‑логики и SQL. |
| **Бизнес‑слой/домен** | Сердце приложения: сущности, правила, use cases; не знает о HTTP/JSON/SQL; опирается на абстракции репозиториев и инфраструктуры. |
| **Слой данных** | Репозитории, ORM, SQL/NoSQL; инкапсулирует детали хранилищ; реализует интерфейсы, ожидаемые доменом. |
| **Инфраструктура** | Очереди, email/SMS‑провайдеры, внешние API, логирование; домен общается через интерфейсы, чтобы не зависеть от конкретных SDK. |
| **Правило зависимостей** | Зависимости идут сверху вниз: презентация → домен → данные/инфраструктура; обратных зависимостей быть не должно. |
| **Типичные ошибки** | Толстые контроллеры; бизнес‑логика в данных; обход слоёв; общий `core/common` как свалка; циклы зависимостей. |
| **BFF и фронтенд** | BFF как дополнительный презентационный/оркестрационный слой; слоистое ядро облегчает его построение; разные фронтенд‑архитектуры → разные требования к API. |
| **Эволюция** | От просто слоистой архитектуры → к модульному монолиту (часть 5) → к гексагональной/Clean Architecture (части 6–7); слои — фундамент всех этих шагов. |

### Быстрый чек‑лист диагностики слоистости

| Вопрос | Если ответ «да» | Что делать |
|--------|-----------------|-----------|
| В контроллерах есть SQL/ORM‑запросы? | Признак толстых контроллеров и обхода слоя данных. | Вынести логику в use case/домен, доступ к БД — в репозитории. |
| Доменные сервисы знают о HTTP/JSON или конкретных HTTP‑клиентах? | Домен зависит от транспорта, сложнее переиспользовать и тестировать. | Ввести DTO/команды, передавать в домен только бизнес‑модели и абстракции. |
| Репозитории принимают/возвращают DTO из API? | Слой данных «протекает» в презентацию и домен. | Развести доменные модели и API‑DTO, добавить мапперы. |
| Есть «общий» `core/common`, зависящий от доменных модулей? | Риск циклов и «чёрной дыры» зависимостей. | Разделить `core` на инфраструктурные/утиль‑модули, запретить ссылки на домен. |
| Есть циклические зависимости между модулями/слоями? | Сильная связность, сложно выделять модули и сервисы. | Использовать инструменты анализа зависимостей, разорвать циклы, ввести интерфейсы и явные контракты. |

## Частые сценарии

### Сценарий 1. «У нас толстеют контроллеры, что делать?»

- **Симптомы**:
  - методы контроллеров по 100+ строк;
  - сложные бизнес‑условия внутри;
  - SQL/ORM‑запросы прямо в контроллерах;
  - интеграции с внешними API там же.
- **Ход решения**:
  1. Выделить один бизнес‑сценарий (например, «создать заказ»).
  2. Создать `CreateOrderUseCase`/сервис в бизнес‑слое.
  3. Перенести туда бизнес‑правила и работу с репозиториями/инфраструктурой.
  4. Оставить в контроллере только маппинг запрос/ответ.
  5. Повторить шаги для других сценариев.

### Сценарий 2. «Нужно добавить мобильный клиент без переписывания бекенда»

- **Исходные данные**:
  - есть SPA, работающая с REST‑API;
  - бекенд имеет (или не имеет) слои.
- **Ход решения**:
  - если слои есть:
    - добавить BFF, который обращается к доменному слою/внутреннему API;
    - адаптировать DTO под мобильный клиент;
    - домен и слой данных не трогать.
  - если слоистости нет:
    - сначала минимально навести порядок: выделить доменный слой для ключевых сценариев;
    - затем строить BFF, опираясь на этот слой.

### Сценарий 3. «Планируем перейти к модульному монолиту»

- **Исходные данные**:
  - есть слоистый монолит;
  - бизнес‑области начинают конфликтовать.
- **Ход решения**:
  1. Зафиксировать слои и правило зависимостей.
  2. Внутри бизнес‑слоя выделить доменные модули (`orders`, `users`, `billing` и т.п.).
  3. Запретить прямые зависимости между модулями, кроме явных контрактов.
  4. Использовать архитектурные тесты для проверки зависимостей.

### Сквозной пример: один бизнес‑сценарий, но 2 разных “протечки” границ

Возьмём простой сценарий **«оформить заказ»** и посмотрим, как он выглядит в слоистой архитектуре — и как обычно ломают границы.

**Вариант 1 (нормальный):**

- **Controller**: принимает HTTP, валидирует формат, мапит в команду `CreateOrderCommand`.
- **Use case / доменный сервис**: решает бизнес‑правила:
  - проверяет лимиты/остатки,
  - создаёт заказ,
  - инициирует оплату (через порт/интерфейс).
- **Repository**: сохраняет доменную сущность, детали ORM/SQL спрятаны внутри.
- **Инфраструктура**: конкретный клиент платёжного провайдера, логирование, отправка события.

**Вариант 2 (типичная ошибка):**

- в контроллере появляется:
  - SQL (или ORM‑запросы),
  - вызов платёжного SDK,
  - “кусочки” бизнес‑условий.

**Чем это плохо практически**:

- тесты становятся либо дорогими интеграционными, либо “мок‑адом”;
- при добавлении второго клиента (например, мобильного/BFF) приходится дублировать логику;
- любые изменения бизнес‑правил размазаны по слоям → растёт регрессия.

## Вопросы для самопроверки по части 4

1. В чём различие между **слоями** и **tier‑ами**, и почему важно не путать эти понятия при обсуждении архитектуры?  
   <details><summary>Ответ</summary>
   Слои — это логические уровни внутри приложения (презентация, домен, данные, инфраструктура), определяющие разделение ответственности и направление зависимостей.  
   Tier‑ы — физические уровни развёртывания (клиент, сервер, БД, очереди).  
   Путаница приводит к тому, что обсуждение архитектуры застревает на инфраструктурном уровне («у нас есть три сервера»), игнорируя структуру кода и зависимости, которые на самом деле важны для поддерживаемости и скорости изменений.
   </details>

2. Почему правило **«зависимости только вниз»** критично для поддерживаемости и тестируемости системы?  
   <details><summary>Ответ</summary>
   Потому что оно:
   - упрощает ментальную модель (понятно, кто кого может вызывать);
   - предотвращает циклы и запутанные графы зависимостей;
   - позволяет изолированно тестировать домен (он зависит только от абстракций);
   - облегчает замену деталей (БД, протоколы, инфраструктуру) без переписывания домена.  
   Нарушение правила ведёт к «слоёному шару из грязи», где любое изменение тянет за собой каскад правок.
   </details>

3. Как слоистая архитектура помогает тебе в будущем лучше понять и применять **модульный монолит, гексагональную и Clean Architecture**?  
   <details><summary>Ответ</summary>
   Слои формируют базовую вертикальную структуру: где живут правила, где данные, где взаимодействие с внешним миром.  
   Модульный монолит добавляет горизонтальное разбиение по доменам внутри этих слоёв.  
   Гексагональная и Clean Architecture переосмысляют слои как ядро и адаптеры с зависимостями внутрь, но не отменяют саму идею слоистости.  
   Если ты уверенно ориентируешься в слоях, переход к этим архитектурам становится не «революцией», а эволюцией уже знакомых принципов.
   </details>

## Типичные ошибки по части 4

- Путать «слои» с «tier‑ами» и считать, что наличие клиента, сервера и БД автоматически означает хорошую слоистую архитектуру.
- Думать, что достаточно назвать папки `controllers/`, `services/`, `repositories/`, чтобы архитектура стала слоистой.
- Пытаться «вылечить» хаотичный бекенд добавлением BFF, не наводя порядок в слоях.
- Игнорировать вертикальные срезы и думать только о слоях, забывая о реальных бизнес‑сценариях.

## Резюме части 4

В этой части ты:

- разобрался(лась), что такое **слоёная (N‑tier) архитектура** и чем логические слои отличаются от физических tier‑ов;
- научился(лась) **выделять основные слои** бекенда (презентация, домен, данные, инфраструктура) и применять правило **зависимостей только вниз**;
- увидел(а), как из аккуратной трёхслойки получается **«слоёный шар из грязи»** — через толстые контроллеры, обход слоёв, свалки в `common` и циклы;
- связал(а) слои с **фронтендом и BFF**, поняв, как они влияют на удобство API и эволюцию системы;
- подготовился(лась) к следующим шагам: **модульный монолит (часть 5)**, **гексагональная (часть 6)** и **Clean Architecture (часть 7)**, где идеи слоистости будут развиваться и усиливаться.

Эта часть — твой «архитектурный уровень 1» по структуре бекенда. Дальше мы будем строить на нём **доменные модули, порты и адаптеры, слои use cases и ядра**, но именно понимание слоёв даст тебе устойчивую картину для всех последующих архитектур.
