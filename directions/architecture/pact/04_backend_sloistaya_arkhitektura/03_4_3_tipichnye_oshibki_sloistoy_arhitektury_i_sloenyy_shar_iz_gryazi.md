## 4.3. Типичные ошибки слоистой архитектуры и «слоёный шар из грязи»

### Цель раздела

Научить тебя **распознавать архитектурные запахи в слоистых системах** и понимать, как из аккуратной трёхслойки получается «слоёный шар из грязи» — и что с этим можно сделать.

### В этом разделе главное

- Слоистая архитектура сама по себе **не гарантирует порядка** — важно, как слои используются.
- Типичные проблемы:
  - «толстые контроллеры»;
  - бизнес‑логика в слое данных;
  - обход слоёв (прямые вызовы в БД/инфраструктуру из верхних слоёв);
  - «общий утиль‑слой» как помойка;
  - циклические зависимости между модулями и слоями.
- Эти ошибки напрямую бьют по **поддерживаемости, тестируемости и скорости изменений**.
- Есть конкретные **тактики оздоровления**: вынос логики в домен, введение интерфейсов, явная проверка зависимостей.

### Термины

- **Толстый контроллер** — контроллер, который содержит существенную часть бизнес‑логики и работы с данными.
- **Антипаттерн «общий core/common»** — модуль/слой, куда складывают всё подряд, ломая границы.
- **Cyclic dependency (циклическая зависимость)** — ситуация, когда модуль/слой A зависит от B, а B — от A (прямо или через цепочку).
- **Bypass слоя** — прямой доступ к нижнему слою, минуя положенный промежуточный слой.

### Теория и правила

Рассмотрим ключевые типы ошибок.

1. **Толстые контроллеры**
   - Признаки:
     - десятки/сотни строк в методе контроллера;
     - SQL/ORM‑вызовы внутри контроллера;
     - бизнес‑правила, написанные прямо в обработчике;
     - обращения к внешним API напрямую.
   - Последствия:
     - сложно тестировать бизнес‑логику отдельно;
     - сложно переиспользовать логику в других протоколах;
     - контроллеры становятся «бутылочным горлышком» для изменений.

2. **Бизнес‑логика в слое данных**
   - Примеры:
     - сложные `stored procedures`, где в SQL живут бизнес‑правила;
     - репозитории, которые принимают HTTP‑DTO и внутри решают бизнес‑правила.
   - Последствия:
     - домен «худеет» до уровня «thin domain»;
     - логика рассредоточена по SQL‑процедурам и репозиториям;
     - перенос на другую БД становится дорогим.

3. **Обход слоёв (bypass)**
   - Примеры:
     - контроллеры напрямую вызывают репозитории, игнорируя домен;
     - фронтенд обходит BFF и ходит напрямую в бекенд‑ядро;
     - доменные сервисы напрямую используют HTTP‑клиенты вместо абстракций.
   - Последствия:
     - размывание ответственности;
     - рост связности;
     - сложности с тестированием и эволюцией.

4. **Общий `Core/Common` как свалка**
   - Поначалу разумный модуль для общих утилит постепенно превращается в:
     - набор всего, что «непонятно куда класть»;
     - модуль, от которого зависят все остальные;
     - источник циклических зависимостей.
   - Последствия:
     - любое изменение в `core` может сломать пол‑системы;
     - невозможно выделить модули/слои независимо.

5. **Циклические зависимости**
   - Возникают, когда:
     - доменный модуль зависит от утиль‑модуля, а тот — от доменного;
     - слой данных тянет доменные сервисы;
     - презентационный слой импортирует код из слоя данных напрямую (и наоборот).
   - Последствия:
     - сложности со сборкой/деплоем;
     - невозможность выделить части в отдельные модули;
     - запутанные потоки управления.

### Простыми словами

Можно представить слоистую архитектуру как **многослойный торт**:

- пока каждый слой аккуратно отделён кремом, торт:
  - красиво режется;
  - каждый кусочек держит форму;
  - можно добавить новый слой, не разваливая всё.

- если же:
  - кто‑то воткнул ложку и размазал слои;
  - часть крема ушла внутрь, часть — наружу;
  - сверху налили ещё что‑то;  
  торт превращается в **бесформенную массу**. Вкус может остаться, но **нарезать, подать и доработать** его намного сложнее.

### Картинка в голове

Диаграмма, показывающая **обход слоёв и свалку в `Common`**:

```mermaid
graph TB
    P["Контроллеры"]
    B["Бизнес-слой"]
    D["Слой данных"]
    C["Common/Core\n("всё подряд")"]

    P --> B
    B --> D

    %% Обход слоёв
    P -.-> D

    %% Общий core как свалка
    P --> C
    B --> C
    D --> C
    C --> P
    C --> B
    C --> D
```

Здесь видно:

- стрелка `P -.-> D` — обход слоёв;
- модуль `Common` связан **со всеми**, что создаёт высокий риск циклов.

### Как запомнить

Короткая формула:

> **«Слоёный шар из грязи» — это когда слои есть на схеме, но нет в зависимостях кода**.

Если:

- контроллеры знают о БД;
- домен зависит от HTTP/ORM;
- `common` зависит от всего и всё зависит от него —  
значит, слоистая архитектура есть **только в документации**, а не в реальном приложении.

### Примеры

**Пример 1. Толстый контроллер и бизнес‑логика в репозитории**

```ts
// Контроллер
async function applyDiscount(req, res) {
  const user = await userRepo.findById(req.userId);
  const cart = await cartRepo.findByUser(user.id);

  // "Немного" бизнес-логики
  if (user.isVip && cart.total > 1000) {
    cart.discount = 0.2;
  } else if (cart.total > 500) {
    cart.discount = 0.1;
  }

  await cartRepo.save(cart);
  res.json(cart);
}
```

Через пару месяцев в `applyDiscount`:

- появляются специальные условия для акций;
- добавляется логика работы с внешним купон‑сервисом;
- меняется схема БД — и всё это живёт в одном методе.

**Пример 2. Свалка в `core`**

```ts
// core/index.ts
export * from "../orders/OrderService";
export * from "../users/UserService";
export * from "../payments/PaymentGateway";
export * from "./utils/logger";
export * from "./utils/config";
```

Любой модуль, импортирующий `core`, получает:

- доступ ко всему;
- неявные зависимости;
- риск циклов (например, `orders` → `core` → `orders`).

### Практика / реальные сценарии

- **Рефакторинг толстого контроллера.**
  - шаг 1: выделить `UseCase`/сервис и перенести туда бизнес‑логику;
  - шаг 2: оставить в контроллере только маппинг запрос/ответ;
  - шаг 3: постепенно переносить SQL в репозитории.

- **Оздоравливание `common`.**
  - шаг 1: проанализировать, какие модули действительно «общие» и не зависят от домена;
  - шаг 2: разделить на несколько более конкретных модулей (`logging`, `config`, `shared-kernel`);
  - шаг 3: запретить зависеть от доменных модулей внутри этих общих модулей.

- **Детектирование циклических зависимостей.**
  - использовать инструменты вроде `ArchUnit`/`dependency-cruiser`;
  - завести **архитектурные тесты**, запрещающие циклы и обходы слоёв.

#### Проверь себя по сценариям ошибок

1. Как в примере с толстым контроллером можно по шагам превратить его в «тонкий» контроллер, не переписывая всю систему сразу?  
   <details><summary>Ответ</summary>
   Можно двигаться по вертикальным срезам:  
   - выбрать один сценарий (например, «применить скидку к корзине»);  
   - вынести бизнес‑логику из контроллера в отдельный use case/сервис;  
   - оставить в контроллере только маппинг запрос/ответ;  
   - постепенно перенести SQL в репозитории.  
   Повторяя это по одному сценарию, мы уменьшаем толщину контроллеров без «большого взрыва».
   </details>

2. В примере с `core/index.ts` почему подход «реэкспортировать всё из разных модулей» приводит к архитектурным проблемам?  
   <details><summary>Ответ</summary>
   Потому что такой `core` становится:  
   - модулем, от которого **зависят все остальные** (единая точка входа);  
   - местом, через которое легко образуются **циклы**: `orders` → `core` → `orders`;  
   - непрозрачным слоем, скрывающим реальные зависимости.  
   В результате любое изменение в одном из экспортируемых модулей потенциально затрагивает всех потребителей `core`, а разорвать зависимости становится сложно.
   </details>

3. Как практические шаги «оздоравливания `common`» и «детектирование циклов» помогают готовиться к модульному монолиту и сервисам?  
   <details><summary>Ответ</summary>
   Они:  
   - уменьшают количество скрытых и циклических зависимостей;  
   - делают границы модулей и слоёв более явными;  
   - позволяют вычленить модули, которые реально самодостаточны.  
   Это необходимо, чтобы в части 5 (модульный монолит) и далее можно было:  
   - фиксировать модульные границы;  
   - перемещать модули между проектами/сервисами;  
   - минимизировать количество связей при разрезе.
   </details>

### Пошагово: как провести быструю диагностику слоистой архитектуры

1. **Посмотри на контроллеры.**
   - Есть ли в них:
     - прямые SQL/ORM‑вызовы?
     - сложные `if/else` с бизнес‑правилами?
     - прямые вызовы внешних API?
   - Если да — это кандидаты на вынос в домен и слой данных/инфраструктуры.
2. **Проверь репозитории и слой данных.**
   - Используют ли они HTTP‑DTO или доменные сущности в качестве входа?
   - Содержатся ли там бизнес‑условия (скидки, статусы, лимиты)?
   - Если да — это правило: часть логики нужно поднять в домен.
3. **Проанализируй `core/common` и утиль‑модули.**
   - Что туда действительно должно входить (инфраструктурные вещи, чистые утилиты)?
   - Импортируют ли они доменные модули/сервисы?
   - Всё, что привело к циклам или неочевидным зависимостям, нужно разнести.
4. **Построй карту зависимостей.**
   - Инструментами (`ArchUnit`, `dependency-cruiser`, IDE) построй граф модулей.
   - Отметь:
     - циклы;
     - зависимости «сверху вниз и обратно»;
     - модули‑«хабы» с чрезмерным числом входящих/исходящих связей.
5. **Выбери один вертикальный срез и наведи порядок.**
   - Возьми конкретный сценарий (например, «создать заказ»):
     - выдели use case/сервис;
     - вынеси логику из контроллера и репозитория;
     - введи интерфейсы репозиториев/инфраструктуры;
     - добавь тесты на домен.
   - Повтори для следующих критичных сценариев.

### Типичные ошибки

- «Мы же использовали контроллеры/сервисы/репозитории, значит у нас уже хорошая слоистая архитектура».
- «Сложную бизнес‑логику удобнее писать в `stored procedures`, там же БД, пусть она всё считает».
- «Давайте сделаем один `core`, в который положим всё общее, а потом разберёмся».

### Что будет, если…

- **…оставить толстые контроллеры как есть?**  
  С ростом продукта:
  - контроллеры станут точками концентрации сложности;
  - тесты будет трудно писать и поддерживать;
  - любая правка API/логики станет рискованной.

- **…смириться с `common`‑свалкой и циклами?**  
  - будет невозможно выделить части системы в отдельные модули/сервисы;
  - архитектурный долг будет накапливаться экспоненциально;
  - команда начнёт бояться правок в этих «центральных» местах.

### Проверь себя

1. Какие признаки указывают на то, что контроллер стал **«толстым»**?  
   <details><summary>Ответ</summary>
   Признаки:
   - много строчек кода и сложная логика в одном методе;
   - SQL/ORM‑вызовы прямо в контроллере;
   - проверка бизнес‑правил (лимиты, статусы) внутри контроллера;
   - обращения к внешним API/очередям напрямую;
   - сложность выделить логику в переиспользуемый сервис.
   </details>

2. Почему «общий `core/common`‑модуль» часто превращается в архитектурный антипаттерн?  
   <details><summary>Ответ</summary>
   Потому что:
   - в него начинают складывать всё, что «сложно отнести к конкретному модулю»;
   - он становится модулем, от которого зависят все остальные;
   - ради удобства туда протаскивают доменные сущности и сервисы, создавая циклы;
   - любое изменение в `core` начинает потенциально ломать пол‑системы.  
   В итоге `core` становится «чёрной дырой» зависимостей.
   </details>

3. Какой первый шаг ты бы сделал(а) для оздоровления проекта с толстым контроллером и свалкой в `common`?  
   <details><summary>Ответ</summary>
   Возможный первый шаг:
   - выделить конкретный бизнес‑сценарий (например, «создать заказ»);
   - вынести его логику из контроллера в отдельный use case/сервис;
   - ограничить экспорт `common`, оставив только действительно общие утилиты и инфраструктурные обёртки;
   - добавить архитектурные тесты/линтинг для контроля зависимостей.  
   Важно начать с малого, но осознанно, не пытаясь переписать всё разом.
   </details>

### Запомните

- Наличие слоёв **по названиям** ещё не значит наличие здоровой слоёной архитектуры.
- Основные враги — толстые контроллеры, домен в данных, обход слоёв, `common`‑свалка и циклы.
- Оздоровление всегда идёт **по маленьким вертикальным срезам**, а не переписыванием всей системы сразу.
