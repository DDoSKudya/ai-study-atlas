## 5.2. Как выделять доменные модули и задавать границы

### Цель раздела

Научить тебя **проводить правильные границы модулей** внутри монолита: по предметной области, сценариям и владению данными, а не по слоям или случайным техническим деталям.

### В этом разделе главное

- Модули в модульном монолите должны быть **доменно‑ориентированными** (по поддоменам, bounded contexts), а не «техническими» (`controllers`, `services`, `repositories`).
- У каждого модуля есть:
  - **область ответственности**;
  - **внутренний код** (private);
  - **публичный API** (facade/интерфейсы/события).
- Границы модулей идут **вместе с границами данных**: кто владеет какими таблицами/коллекциями.
- Между модулями действуют **ограниченные способы взаимодействия**:
  - вызовы публичных фасадов;
  - доменные события;
  - в редких случаях — специализированные shared‑модули.
- Неправильные границы (анти‑границы) порождают тесную связанность и усложняют эволюцию.

### Термины

- **Bounded context** — логическая область предметной области, внутри которой термины и модели используются в одном и том же смысле.
- **Доменный модуль** — реализация bounded context (или значимого поддомена) внутри монолита.
- **Публичный API модуля** — набор классов/интерфейсов/фасадов/событий, которые разрешено использовать другим модулям.
- **Анти‑граница** — модуль, граница которого проведена так, что он «режет» естественный домен (например, модуль `services` вместо `orders`/`users`).

### Теория и правила

1. **От предметной области к модулям.**

Возьми простой домен — интернет‑магазин:

- **Поддомены**:
  - каталог товаров;
  - корзина и заказы;
  - пользователи и профили;
  - оплата;
  - доставка.

Естественные доменные модули:

- `catalog`;
- `orders`;
- `users`;
- `billing`;
- `shipping` (доставка).

2. **Границы по данным и ответственности.**

- Модуль `orders`:
  - отвечает за:
    - жизненный цикл заказа;
    - статусы;
    - бизнес‑правила скидок/лимитов;
  - **владеет** таблицами `orders`, `order_items`, `order_status_history`.
- Модуль `users`:
  - отвечает за:
    - учётные записи;
    - профили;
    - статусы пользователей;
  - владеет таблицами `users`, `user_profiles` и т.п.

Правило:

- другие модули **не трогают напрямую чужие таблицы**;
- они обращаются к чужим данным через **публичный API модуля‑владельца**.

3. **Внутренний и публичный код модуля.**

Для каждого модуля:

```text
module orders
  ├─ api/          # контроллеры/эндпоинты модуля (если они нужны)
  ├─ domain/       # сущности, value-объекты, сервисы домена
  ├─ app/          # use cases, фасады модуля
  ├─ infra/        # репозитории, интеграции, адаптеры
  └─ public/       # "витрина" модуля для других: интерфейсы, DTO, события
```

В `public/` могут жить:

- фасады `OrdersFacade` для вызова сценариев;
- абстракции событий `OrderCreated`, `OrderCancelled`;
- интерфейсы для запросов из других модулей.

4. **Анти‑границы: модули по слоям вместо доменов.**

Пример **плохого** разбиения:

```text
module controllers
module services
module repositories
module common
```

Здесь модули:

- не соответствуют предметной области;
- фактически отражают **слои**, а не домены;
- приводят к высокой связанности:
  - `services` нужно знать о `repositories` и `controllers`;
  - доменная логика размазывается.

5. **Связь с bounded contexts из DDD.**

Полное DDD выходит за рамки этой части, но идея простая:

- **bounded context** — это «место, где слова имеют одно значение»;
- доменный модуль часто ≈ bounded context (или явный поддомен);
- если в системе несколько контекстов (например, `Sales` и `Billing`), разумно делать разные модули:
  - `sales` (оформление заказов);
  - `billing` (финансовый учёт, инвойсы);
  - между ними — явные контракты и события.

6. **Модули и фронтенд/BFF.**

- Фронтенд часто «видит» домены как:
  - страницы/разделы: «каталог», «заказы», «профиль», «оплата»;
  - виджеты/фичи.  

Гармонично, когда:

- доменные модули в бекенде **созвучны** этим областям;
- BFF может опираться на модули:
  - `orders` → API/фасады по заказам;
  - `users` → API по профилям;
  - `billing` → API по платежам.

### Пошагово: как нарезать модули в существующем монолите

1. **Нарисуй карту домена.**
   - Выдели крупные предметные области:
     - заказы, пользователи, каталог, оплата, отчёты, интеграции и т.п.
   - Можно воспользоваться:
     - существующей архитектурной документацией;
     - списком бизнес‑функций;
     - списком команд (если уже есть доменные команды).

2. **Сопоставь области с кодом.**
   - Найди:
     - пакеты/неймспейсы, связанные с каждой областью;
     - таблицы/коллекции БД, относящиеся к ней;
     - основные API‑эндпоинты.

3. **Сгруппируй код в модули.**
   - Для каждого домена:
     - создай модуль (`orders`, `users`, `billing` и т.п.);
     - перенеси туда:
       - доменные сущности;
       - сервисы;
       - репозитории;
       - связанные контроллеры.

4. **Определи публичный API.**
   - Реши:
     - какие сценарии/операции модуль предоставляет другим модулям;
     - в каком виде (фасады, интерфейсы, события).
   - Спрячь всё остальное как `internal`/`package-private`/просто неэкспортируемый код.

5. **Запрети нежелательные зависимости.**
   - Введи правила:
     - модуль `orders` не может напрямую импортировать `billing.infra`;
     - доступ к чужим данным — только через интерфейсы/фасады.
   - Позже (в 5.3) мы добавим **архитектурные тесты**.

### Простыми словами

Можно думать так:

- **Модуль** — это **мини‑сервис внутри монолита**, у которого:
  - есть своя область ответственности и данных;
  - есть своя публичная витрина (API) для остальных;
  - есть «чердак» и «подвал», куда посторонних не пускают (внутренний код).

- Плохие модули — это как комнаты, разделённые **по цвету стен**, а не по назначению:
  - «все красные стены в одном модуле»;
  - «все синие — в другом».

Хочется, чтобы модули были:

- **по функциям** (кухня, спальня, ванная), а не:
- **по цвету краски или типу мебели**.

### Картинка в голове

Представь сетку:

- по вертикали — слои (из части 4);
- по горизонтали — модули по доменам.

```text
             Модуль Users       Модуль Orders       Модуль Billing
--------------------------------------------------------------------
Презентация  Users API          Orders API          Billing API
Домен        Users Domain       Orders Domain       Billing Domain
Данные       Users Repos        Orders Repos        Billing Repos
Инфраструкт. Users Integr.      Orders Integr.      Billing Integr.
```

Каждая клетка — **код, принадлежащий одному модулю и одному слою**.

### Примеры

**Пример 1. Хорошее разбиение по доменам**

- `module orders`:
  - API: создание/изменение заказов;
  - Domain: сущности `Order`, `OrderItem`, правила статусов, лимиты;
  - Data: таблицы `orders`, `order_items`;
  - Public API: `OrdersFacade`, события `OrderCreated`.
- `module users`:
  - API: регистрация/управление пользователями;
  - Domain: `User`, `UserProfile`, политики блокировок;
  - Data: таблицы `users`, `user_profiles`.

**Пример 2. Плохое разбиение по слоям**

- `module api`:
  - все контроллеры для всех доменов;
- `module domain`:
  - все сущности и сервисы;
- `module data`:
  - все репозитории и таблицы;
- `module common`:
  - часть домена, непонятно какая.  

В результате:

- любая фича тянет код из нескольких модулей;
- модули перетекают друг в друга;
- тестировать и эволюционировать очень тяжело.

### Практика / реальные сценарии

- **Рефакторинг существующего монолита.**
  - Начинают с небольшой части:
    - выделяют модуль `orders`;
    - переносят туда связанную логику и данные;
    - прописывают публичный фасад `OrdersFacade`;
    - переводят другие части кода на использование фасада.
  - Постепенно то же делают для `users`, `billing` и т.д.

- **Проектирование нового продукта.**
  - Сразу проектируют структуру:
    - по доменам;
    - внутри каждого — по слоям.
  - Это уменьшает риск «монолитного болота» уже на старте.

### Типичные ошибки

- Модуль = «папка с файлами по типу» (`controllers`, `services`, `dto`), а не по домену.
- Использовать `shared`/`common` как **второй монолит**:
  - туда складывают всё, что не понятно куда деть;
  - доменные вещи просачиваются внутрь.
- Думать, что доменный модуль — это **только слой домена**, а контроллеры/репозитории живут «отдельно», без принадлежности к модулю.

### Что будет, если…

- **…порезать модули по слоям, а не по доменам?**  
  Получится:
  - сильная связанность;
  - невозможность выделить конкретный домен в отдельное приложение;
  - рост затрат на понимание: бизнес‑логика размазана между модулями.

- **…вообще не думать о модулях и жить только слоями?**  
  С ростом продукта:
  - слои переполняются;
  - внутри слоя «всё про всё»;
  - любые правила зависят от кучи несвязанных друг с другом фич.

### Проверь себя

1. Почему модули в модульном монолите должны быть **доменно‑ориентированными**, а не техническими (`controllers`, `services`, `repositories`)?  
   <details><summary>Ответ</summary>
   Потому что цель модульного монолита — изолировать **предметные области** (заказы, пользователи, биллинг), а не технические слои. Модуль по домену может эволюционировать и, при необходимости, быть вынесен в сервис. Модуль по слою содержит кусочки всех доменов вперемешку и не соответствует реальным бизнес‑границам, поэтому плохо подходит и для понимания, и для эволюции.
   </details>

2. Как понять, какие таблицы/коллекции БД должны принадлежать какому модулю?  
   <details><summary>Ответ</summary>
   Нужно посмотреть на **семантику данных и доменные операции**: если таблица описывает сущности и факты, относящиеся к конкретному домену (например, `orders` и `order_items` — к заказам), то она должна **принадлежать** соответствующему модулю (`orders`). Другие модули не должны напрямую изменять эту таблицу, а только обращаться к модулю‑владельцу через его API или события.
   </details>

3. Приведи пример **анти‑границы** в реальном или воображаемом проекте.  
   <details><summary>Ответ</summary>
   Например, модуль `notifications`, куда поместили и отправку писем, и push‑уведомления, и саму бизнес‑логику «какие события считать важными» для разных доменов (заказы, пользователи, биллинг). В итоге модуль отвечает и за доменную логику разных областей, и за инфраструктуру. Правильнее было бы: оставить в `notifications` только инфраструктурную часть (отправка), а правила «когда и что отправлять» держать в доменных модулях (`orders`, `users` и т.д.).
   </details>

4. Как ты поймёшь, что граница модуля проведена удачно, если смотришь только на код и схемы БД, без бизнес‑документации?  
   <details><summary>Ответ</summary>
   Признаки: внутри модуля все сущности и операции «про одно и то же» (высокая связность), таблицы БД логично группируются вокруг одних и тех же бизнес‑объектов, а подавляющее большинство изменений по конкретной фиче затрагивает код и данные одного модуля. В графе зависимостей у модуля немного внешних зависимостей, и он не образует циклов с другими модулями. Это говорит о том, что граница совпала с естественной доменной областью.
   </details>

5. Какие риски появляются, если два доменных модуля начинают «делить» одну и ту же таблицу БД, и как это связано с понятием анти‑границы?  
   <details><summary>Ответ</summary>
   Совместное владение таблицей размывает границы: модули начинают полагаться на внутреннюю схему друг друга, любое изменение структуры данных требует согласований и массовых правок, растёт связанность и сложнее контролировать инварианты. Фактически граница модуля перестаёт совпадать с границей данных — это и есть анти‑граница: домены «режутся» так, что каждый из них владеет только частью своей логики и данных, а остальное оказывается «где‑то ещё».
   </details>

### Запомните

- Модуль = **доменная область + свои данные + свой API**, а не просто «группа файлов одного типа».
- Хорошие границы модулей опираются на:
  - **предметную область**;
  - **bounded contexts**;
  - **владение данными**.
- От качества границ модулей напрямую зависит:
  - поддерживаемость;
  - тестируемость;
  - возможность эволюции к сервисам.
