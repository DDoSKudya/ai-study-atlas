## Справочник по части 22

### Ключевые понятия и определения

| Тема | Кратко |
| --- | --- |
| SPA | Одностраничное приложение: один HTML, рендер и роутинг на клиенте, общение с сервером через API. |
| CSR | Client‑Side Rendering: рендер интерфейса в браузере JS‑кодом. |
| Клиентский роутер | Сопоставляет URL → компоненты/экраны в SPA, перехватывая навигацию. |
| Состояние в SPA | Данные в памяти JS (и частично в хранилищах браузера), влияющие на рендер. |
| Бандл | Скомпилированный JS/CSS‑код приложения. Может быть монолитным или разбитым на чанки. |
| Code splitting | Разделение кода на отдельные бандлы по маршрутам/фичам для лучшей производительности. |
| Lazy‑loading | Отложенная загрузка модулей/компонентов по мере использования. |
| TTI | Time To Interactive: момент, когда SPA действительно готова к работе пользователя. |
| Пререндер | Предварительный рендер HTML для ботов/первого захода, поверх SPA. |
| SSR/SSG/ISR | Режимы серверного рендеринга и статической генерации, часто поверх тех же SPA‑фреймворков. |
| Hybrid frontend | Сосуществование SPA и других архитектур (MPA/SSR/SSG/Islands) в одном продукте. |

## Частые сценарии

### Сценарий 1. Внутренняя админка

- **Задача**: удобный интерфейс для сотрудников, много экранов, фильтры, формы, графики.  
- **Решение**:
  - SPA для зоны `/admin` или `admin.example.com`;
  - code splitting по разделам (`/orders`, `/users`, `/reports`);
  - BFF для агрегирования данных из микросервисов.  
- **Почему так**:
  - SEO не нужен;
  - важна отзывчивость, быстрые переходы, сложное состояние.

### Сценарий 2. Публичный маркетинговый сайт + личный кабинет

- **Задача**: максимальный SEO и быстрый первый экран для лендингов + удобный интерфейс в кабинете.  
- **Решение**:
  - публичная часть — MPA/SSR/SSG;
  - кабинет — SPA на `/app`;  
  - единый дизайн, но разные архитектуры.  
- **Почему так**:
  - SEO и производительность важны только для части трафика;
  - кабинет гораздо больше выигрывает от SPA.

### Сценарий 3. Документация и раздел с интерактивными примерами

- **Задача**: документация должна хорошо индексироваться, но есть раздел с живой песочницей/редактором.  
- **Решение**:
  - документация — SSG/SSR;
  - интерактивная зона — отдельный SPA/mini‑SPA внутри страницы (islands).  
- **Почему так**:
  - SEO и простота стека для основного контента;
  - SPA‑подход локализован в зоне, где он даёт максимальную пользу.

### Сквозной план доведения SPA до “прод‑уровня” (TTI/бандлы/API/ошибки)

Цель: чтобы SPA была не просто “рендер в браузере”, а **управляемая система**: предсказуемая загрузка, понятные границы, контролируемые риски (особенно при деплоях и кэшах).

#### 1) Старт: минимальный “core bundle” и карта маршрутов

- выпиши маршруты и “первый экран” для каждого;
- выдели core (auth, layout, router, базовые виджеты) и всё остальное вынеси в lazy chunks по маршрутам/фичам.

Критерий: при входе в `/app` не должен грузиться код `/reports` и “редких” экранов.

#### 2) API под первый экран: BFF/агрегация вместо N запросов

SPA часто “медленная” не из‑за React, а из‑за модели данных:

- если на экран нужно 10 запросов в разные сервисы — делай BFF под экран/домены (часть 30);
- фиксируй контракты и ошибки единым форматом (часть 15/30).

#### 3) Измерение (без него оптимизация превращается в суету)

Минимум метрик:

- LCP/INP (Core Web Vitals),
- размер initial JS/CSS,
- время до “данные первого экрана готовы” (app‑specific),
- доля ошибок загрузки чанков / fallback‑экранов.

#### 4) Ошибки, о которых забывают: ChunkLoadError и “висим на старом HTML”

Типовой прод‑инцидент SPA: пользователь держит вкладку открытой, вы деплоите новую версию, HTML/manifest/чанки рассинхронизировались → `ChunkLoadError`.

План защиты:

- стратегия кэша для HTML (обычно `no-store` или короткий TTL) и длинный кэш для fingerprint‑ассетов;
- обработчик ошибки загрузки чанка:
  - показать “обновите страницу”,
  - или auto‑reload (с лимитом, чтобы не уйти в loop).

#### 5) План отката SPA‑релиза

- откат не только “вернуть версию”:
  - нужно, чтобы CDN/edge не раздавал смесь HTML от версии \(N\) и чанков от \(N+1\);
- практично:
  - хранить 1–2 предыдущих набора ассетов,
  - иметь быстрый rollback release‑тега.

#### 6) Когда SPA не выдерживает — и это нормально

Если:

- критичен SEO на большей части трафика,
- LCP/TTI невозможно удержать без серьёзной инфраструктуры,

то следующая ступень — SSR/SSG/ISR гибрид (часть 23), а не бесконечная “оптимизация бандла”.

## Практика после части 22

Эти мини‑задания помогают закрепить ключевые идеи (в духе требований глобального плана).

1. **Выбери сценарий и обоснуй SPA**  
   - Возьми один сценарий (например, админка, дашборд аналитики или личный кабинет).  
   - Ответь письменно:
     - почему здесь ты выбираешь **SPA, а не MPA/чистый SSR**;
     - какая **одна мера по SEO** тебе всё же нужна (например, отдельный SEO‑дружественный лендинг, пререндер пары маркетинговых страниц или документации);
     - какая **одна типичная ошибка SPA** здесь наиболее опасна (например, огромный бандл и долгий TTI, отсутствие границ между публичной и SPA‑зоной).  

2. **Нарисуй код‑сплиттинг для своей SPA**  
   - Возьми предполагаемые маршруты будущего приложения.  
   - Разбей их на **логические бандлы** (core + бандлы по разделам + тяжёлые lazy‑виджеты) и коротко подпиши, **какие данные от API нужны для первого экрана** каждого бандла.

3. **Продумай гибридную архитектуру**  
   - Для выбранного продукта нарисуй **две зоны**:
     - публичную (тип: MPA/SSR/SSG);
     - приложение (тип: SPA/Islands).  
   - Подпиши:
     - как пользователь попадает из первой зоны во вторую (URL/домен, кнопка «Войти»/«Перейти в кабинет»);
     - какие API используются в SPA;
     - нужен ли BFF и за что он отвечает.

Эти упражнения можно делать в свободной форме (текст + эскизы диаграмм). Главное — **потренироваться принимать архитектурные решения, а не только узнавать термины**.

## Вопросы для самопроверки по части 22

1. В чём ключевые отличия SPA от MPA и SSR с точки зрения:
   - места рендеринга HTML,
   - места, где живёт роутинг,
   - расположения основного состояния?  
2. Почему SPA часто оказывается **медленным при первом запуске**, и какие **архитектурные решения** помогают это исправить?  
3. В каких случаях SPA **совершенно не нужен**, и какие архитектурные альтернативы более уместны?  
4. Как можно **разделить фронтенд‑архитектуру** продукта на зоны, используя одновременно SPA и другие подходы?  
5. Какие три типичные ошибки команды совершают, выбирая SPA «по умолчанию»?

<details><summary>Ответ</summary>

1. Отличия:
   - MPA/SSR рендерят HTML на сервере для каждого запроса, SPA — в браузере;  
   - роутинг в MPA/SSR чаще всего серверный, в SPA — клиентский роутер;  
   - состояние MPA живёт в сессиях/БД, SPA — в памяти JS (частично в хранилищах браузера).  
2. Причины медленности:
   - большой бандл;
   - тяжёлый JS при старте;
   - последовательные запросы к API.  
   Решения:
   - код‑сплиттинг по маршрутам и фичам;
   - lazy‑loading тяжёлых модулей;
   - BFF и оптимизация API под первый экран;
   - skeleton‑экраны.  
3. SPA не нужен:
   - для простых лендингов и маркетинговых страниц;
   - для сайтов с фокусом на статическом контенте (блоги, документация, новости),
   - когда команда не готова к SPA‑стеку и нет веских причин идти в него.  
   Альтернативы: MPA, SSR, SSG, islands/Turbo/HTMX.  
4. Разделение:
   - публичная зона — MPA/SSR/SSG (SEO и скорость первого экрана);
   - зона приложения `/app` — SPA;
   - отдельные mini‑SPA/islands для особо интерактивных частей.  
5. Типичные ошибки:
   - выбирают SPA для всего без анализа сценариев и SEO;
   - не продумывают код‑сплиттинг и API → огромные бандлы и медленный старт;
   - не рисуют границы SPA‑зон и смешивают подходы хаотично.

## Типичные ошибки и антипаттерны

- **SPA «ради моды»**: архитектура выбирается потому, что «так делают все», без анализа сценариев, SEO и ограничений.  
- **Один монолитный бандл**: отсутствие код‑сплиттинга, lazy‑loading и продуманной структуры модулей.  
- **Отсутствие границ SPA**: SPA‑логика просачивается во все зоны сайта, смешиваясь с MPA/SSR.  
- **Игнорирование API‑архитектуры**: SPA вынуждена вызывать множество микросервисов напрямую, что ломает производительность и связность.  
- **SEO «потом»**: сначала делают чистый SPA, затем в панике пытаются прикрутить SSR/пререндер.  
- **Отсутствие стратегии ошибок**: нет обработки сбоев загрузки бандла/ресурсов, нет fallbacks без JS.

## Резюме части 22

- **SPA — это архитектура фронтенда**, в которой:
  - один HTML загружается один раз;
  - интерфейс и роутинг живут в браузере;
  - сервер отдаёт в основном данные через API.  
- SPA особенно силён для **богатых интерактивных приложений**: админок, дашбордов, инструментов и сложных кабинетов.  
- Главные архитектурные риски SPA:
  - **производительность первого запуска** (бандлы, TTI),
  - **SEO** (требует пререндера/SSR/SSG),
  - **сложность стека** (state management, роутинг, сборка).  
- Зрелая архитектура фронтенда почти всегда **гибридная**: SPA сосуществует с MPA/SSR/SSG/Islands, а архитектор умеет **рисовать границы** и обосновывать решения.
