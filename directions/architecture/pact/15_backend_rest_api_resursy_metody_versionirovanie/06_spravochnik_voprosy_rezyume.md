## Справочник по части 15

| Тема | Коротко (что важно помнить) |
|---|---|
| Ресурсы и URI | URL — существительные; коллекции и подресурсы; не RPC |
| Методы | GET safe; PUT/DELETE обычно идемпотентны; POST — осторожно |
| Пагинация | коллекции почти всегда: limit+cursor/page, sort, filter |
| Коды ответа | 2xx успех; 4xx ошибка запроса/прав; 5xx проблема сервера |
| Ошибки | единый формат (желательно RFC 7807), `traceId` обязательно |
| Версионирование | breaking change → новая версия; deprecation policy |
| Идемпотентность | Idempotency-Key для опасных мутаций и ретраев |
| Real-time | streaming/SSE/long-polling — компромиссы, помнить ограничения |

## Частые сценарии

### Сценарий 1. «Нужно создать ресурс и вернуть его ID»

- **Выбор**: `POST /collection` → `201 Created` + `Location`.
- **Если есть риск ретраев**: поддержать `Idempotency-Key`.

### Сценарий 2. «Нужно обновлять часть ресурса»

- **Выбор**: `PATCH /resource/{id}` с понятным телом (частичное обновление).
- **Риск**: «PATCH как команда» → уход в RPC. Следи, чтобы смысл был «изменить ресурс», а не «запустить сценарий».

### Сценарий 3. «Нужно вернуть ошибки валидации формы»

- **Выбор**: `400` или `422` (по вашей политике) + единый формат с `errors[]` по полям.
- **Важно**: стабильные коды ошибок (`code`) и `traceId`.

### Сценарий 4. «Нужны обновления статуса без WebSocket»

- **Выбор**: SSE (если достаточно server → client).
- **Fallback**: long-polling.

## Вопросы для самопроверки (с ответами)

1. Что такое ресурс в REST и почему он не обязан совпадать с таблицей БД?
2. Почему POST обычно не идемпотентен, и как это компенсируют в продакшне?
3. В чём разница 401 и 403?
4. Назови 3 breaking change для JSON API.
5. Когда уместен SSE, а когда лучше WebSocket?

<details><summary>Ответ</summary>

1. Ресурс — это объект контракта, «вещь» для клиента. Он должен отражать доменную модель и потребности клиентов, а не структуру хранения. БД может меняться, а контракт желательно сохранять стабильным.
2. POST «создай новый» при повторе создаёт ещё один новый. Компенсация: Idempotency-Key, детерминированные идентификаторы, сохранение результата первой попытки.
3. 401 — нет/невалидна аутентификация. 403 — аутентификация есть, но нет прав.
4. Удалить поле; изменить тип; изменить смысл/единицы; сделать поле обязательным; изменить контракт пагинации.
5. SSE — когда нужен push server → client и достаточно HTTP. WebSocket — когда нужен двусторонний интерактив/частые сообщения/низкая задержка в обе стороны.

</details>

## Типичные ошибки

1. **RPC через HTTP**: `/doX`, `/runY`, глаголы в URL.
2. **Отсутствие пагинации** на коллекциях.
3. **200 для ошибок**, «ошибка в JSON» без статуса.
4. **Нет единого формата ошибок**, нет `traceId`.
5. **Breaking change без версии** или без deprecation.
6. **Нет идемпотентности** на важных POST/операциях.
7. **Real-time через бесконтрольный polling** (нагрузка и шум).

## Резюме части

- REST API — это прежде всего **контракт ресурсов**, усиленный стандартной семантикой HTTP.
- Самые дорогие ошибки — не «не тот эндпоинт», а отсутствие дисциплины в **кодах, ошибках, версиях и идемпотентности**.
- Хороший REST помогает не только клиентам, но и эксплуатации: кэши, метрики, алерты, ретраи становятся предсказуемыми.
- Следующие логические шаги по плану: сравнить REST с **GraphQL** (часть 16) и с **gRPC/tRPC/BaaS** (часть 17), а затем — связать всё это в **BFF и контрактах** (часть 30).
