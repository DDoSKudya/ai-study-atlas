## Часть 15. Бекенд: REST API — ресурсы, методы, идемпотентность, ошибки и версионирование

> Цель этой части — научить тебя **проектировать и поддерживать REST API как контракт**: выбирать правильные **ресурсы и URI**, осознанно применять **HTTP-методы** (безопасность и идемпотентность), возвращать **корректные коды ответа**, оформлять **единый формат ошибок** (например, RFC 7807), аккуратно решать **версионирование**, и понимать, где REST «заканчивается» (real‑time, стриминг, SSE, long‑polling) и какие есть безопасные компромиссы.
>
> REST в продакшне — это не «поставить GET/POST и готово», а **множество мелких решений**, каждое из которых влияет на UX, надёжность (retry), безопасность, кэширование и способность системы эволюционировать без поломок клиентов.

Эта часть по глобальному плану идёт после архитектурных подходов бекенда (части 3–14) и готовит почву для:

- **части 30 (BFF, контракты, аутентификация)** — REST-контракты почти всегда проходят через слой BFF или API Gateway;
- **части 31 (безопасность, наблюдаемость)** — правильные ошибки, коды и идемпотентность сильно влияют на безопасность и диагностику;
- **части 16–17 (GraphQL, gRPC/tRPC/BaaS)** — чтобы сравнивать альтернативы, нужно уверенно владеть базовой «универсальной» опцией REST.

---

### После этой части ты сможешь

- проектировать REST API **от ресурсов**, а не от «действий» и RPC-стиля;
- уверенно применять термины **ресурс**, **коллекция**, **подресурс**, **representation**, **контракт**, **идемпотентность**, **безопасный метод**;
- выбирать и объяснять правильные **HTTP-методы** (GET/POST/PUT/PATCH/DELETE) и их свойства;
- понимать, что такое **HATEOAS** и как ссылки в ответах помогают эволюции и навигации по API;
- строить предсказуемую политику **кодов ответа** (2xx/3xx/4xx/5xx) без «200 для всего»;
- делать **единый формат ошибок** для всех эндпоинтов (и понимать, что показывать клиенту, а что — в логи);
- проектировать **версионирование** (когда нужно, как вводить, как не убить кэширование);
- применять **Idempotency-Key** для безопасных повторов мутаций (особенно для мобильных и при ретраях);
- понимать, где уместны **стриминг ответа**, **SSE** и **long-polling** в REST-ландшафте, и какие у них ограничения;
- видеть типичные ошибки дизайна REST и заранее ставить «ограждения» (чек-листы, линтеры контрактов, тесты).

#### Проверь себя по целям части

1. Почему «правильный REST» начинается с **ресурсов**, а не с выбора методов?
2. В чём практическая разница между «операция идемпотентна» и «клиент может безопасно повторить запрос»?
3. Почему «200 с `{ success: false }`» — почти всегда архитектурная ошибка, а не «вкус API»?

<details><summary>Ответ</summary>

1. Потому что REST — это архитектурный стиль про **ресурсы и их представления**. Методы — это уже «инструменты» для операций над ресурсами. Если начать с методов, легко уйти в RPC (`/doSomething`) и потерять унифицированные свойства HTTP (кэширование, семантику кодов, стандартные клиенты/прокси).
2. Идемпотентность — свойство операции: повтор применения даёт тот же конечный эффект. Но чтобы клиент мог безопасно повторить, нужна ещё **правильная реализация на сервере**: стабильные коды, одинаковые ответы для повторов, защита от дублей (например, Idempotency-Key) и ясная обработка таймаутов/unknown outcome.
3. Потому что это ломает семантику протокола: коды 4xx/5xx используются инфраструктурой, прокси, логированием, мониторингом, библиотеками ретраев. «200 для всего» делает ошибки невидимыми и превращает контракт в «угадай по полю», усложняя диагностику и безопасность.

</details>

Если чего-то из этого пока нет — нормально: в этой части будет много «простыми словами» и примеров запросов/ответов.

#### Проверь себя: готов ли ты к части 15

1. Отличаешь ли ты «**ресурс**» от «**метода/действия**»?
2. Понимаешь ли ты, что запрос может **не дойти**, но клиент об этом узнает только по таймауту?
3. Можешь ли ты объяснить, зачем в API вообще нужны разные **коды ответа**, если «и так можно JSON вернуть»?

<details><summary>Ответ</summary>

1. Ресурс — это «вещь», на которую мы ссылаемся (заказ, пользователь, файл). Действие — это то, что мы делаем с ресурсом. В REST мы стараемся выражать операции через **унифицированные методы** над ресурсами.
2. Да: из-за сети/таймаутов клиент часто попадает в состояние «не знаю, выполнилось ли». Это главная причина, почему идемпотентность и Idempotency-Key — не абстракция, а практика.
3. Потому что статус-код — это «быстрый, стандартный сигнал» для клиентов и инфраструктуры. Он влияет на ретраи, кэширование, алерты, метрики и логи. Если всё «внутри JSON», система теряет стандартизированную наблюдаемость и управляемость.

</details>

### Краткая шпаргалка по терминам

| Понятие | Суть простыми словами |
|---|---|
| **REST** | Стиль проектирования API, где мы работаем с **ресурсами** через стандартные возможности HTTP. |
| **Ресурс (resource)** | «Сущность/вещь», к которой обращаемся по URI: заказ, пользователь, комментарий, файл. |
| **Коллекция** | Набор ресурсов: `/orders` — коллекция заказов. |
| **Подресурс** | Ресурс, живущий «внутри» другого: `/orders/{id}/items`. |
| **Representation** | Представление ресурса (обычно JSON): какие поля возвращаем и в каком формате. |
| **Безопасный метод (safe)** | Метод, который **не должен менять** состояние (GET, HEAD, OPTIONS). |
| **Идемпотентный метод** | Повтор запроса даёт тот же итоговый эффект (GET, PUT, DELETE — обычно да; POST — обычно нет). |
| **Контракт API** | Согласованное описание запросов/ответов: методы, поля, коды, ошибки, версии. |
| **RFC 7807 Problem Details** | Стандартный JSON-формат ошибки: `type/title/status/detail/instance` и расширения. |
| **Версионирование** | Способ менять контракт, не ломая клиентов (v1/v2, совместимые изменения, deprecation). |
| **Idempotency-Key** | Ключ, который помогает серверу распознать повтор POST/операции и **не создать дубль**. |
| **SSE** | «Сервер шлёт события» поверх HTTP: односторонний push (без WebSocket). |
| **Long-polling** | Клиент держит запрос открытым, пока сервер не ответит событием/таймаутом; затем повторяет. |

#### Проверь себя по шпаргалке

1. Почему «идемпотентность» — это не про HTTP как транспорт, а про смысл операции?
2. Почему **POST** «обычно не идемпотентен», но мы всё равно можем сделать повтор безопасным?
3. Зачем нужен единый формат ошибок, если можно просто `{ message: "ошибка" }`?

<details><summary>Ответ</summary>

1. Потому что идемпотентность определяется **эффектом на состояние системы**: «если повторить, итог тот же». Один и тот же HTTP-запрос может быть идемпотентным или нет — зависит от реализации (например, POST «создать заказ» без защиты создаст два заказа).
2. Мы можем сделать повтор безопасным, если введём **серверное распознавание дублей**: Idempotency-Key, детерминированные идентификаторы, сохранение результата первого выполнения и возврат его при повторе.
3. Потому что формат ошибок — это часть контракта. Единый формат даёт предсказуемую обработку на клиентах, единые метрики/логирование и меньше «зооопарка» случаев.

</details>

### Маршрут изучения (простыми словами)

1. Сначала соберём **картинку REST**: как мыслить «ресурсами», как выбирать URI и не уйти в RPC.
2. Затем разберём **HTTP-методы**, безопасность и идемпотентность на практических сценариях.
3. После этого — **коды ответа и ошибки**: как сделать контракт предсказуемым и диагностируемым.
4. Дальше — **версионирование**: как менять API годами без постоянных «ломающих релизов».
5. Затем — **Idempotency-Key и ретраи**: как выживать в реальной сети и на мобильных.
6. И в конце — **стриминг/SSE/long-polling**: что можно сделать в REST-экосистеме для «почти real‑time».
