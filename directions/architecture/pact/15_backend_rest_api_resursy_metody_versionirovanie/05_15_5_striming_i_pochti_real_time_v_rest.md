## 15.5. Стриминг и «почти real-time» в REST

### Цель раздела

Понять, какие «реалтайм‑компромиссы» можно сделать в REST/HTTP: **streaming response**, **SSE** и **long-polling**, и где лучше сразу перейти на WebSocket/отдельный real-time канал.

### В этом разделе главное

- REST — это в основном request/response, но HTTP позволяет **стримить** данные.
- **SSE** — хороший вариант «сервер → клиент» (push) без WebSocket.
- **Long-polling** — простое, но менее эффективное решение (fallback).
- У каждого подхода есть ограничения по инфраструктуре (прокси, таймауты, лимиты соединений).

### Термины

| Термин | Определение |
|---|---|
| **Streaming response** | Сервер отправляет ответ частями (chunks) по мере готовности. |
| **SSE** | Долгое HTTP-соединение, по которому сервер шлёт события (`text/event-stream`). |
| **Long-polling** | Клиент делает запрос и держит его открытым до события/таймаута, затем повторяет. |

### Теория и правила

#### 1) Streaming response

Уместно, когда:

- нужно отдавать большой результат по частям (экспорт, генерация отчёта),
- хочется «первый байт быстро» (TTFB), даже если расчёт долгий,
- нужно показывать прогресс.

Риск: прокси/балансеры могут буферизовать ответ, и «стриминг» превратится в «получили всё в конце».

#### 2) SSE (Server-Sent Events)

SSE — это односторонний push:

- клиент открывает `GET /events`,
- сервер шлёт события,
- клиент переподключается при обрыве.

Пример:

```http
GET /orders/ord_123/events
Accept: text/event-stream
```

Ответ:

```http
200 OK
Content-Type: text/event-stream
Cache-Control: no-cache
Connection: keep-alive
```

Тело (идея):

```
event: status
data: {"orderId":"ord_123","status":"paid"}

event: status
data: {"orderId":"ord_123","status":"shipped"}
```

Где удобно: уведомления о статусах, прогресс, «живые» дашборды.

Ограничения:

- только сервер → клиент (клиент не шлёт данные в этом же канале);
- нужно следить за количеством одновременных соединений;
- учитывать прокси/таймауты, иногда нужны heartbeats.

#### 3) Long-polling

Работает почти везде, но:

- выше нагрузка (частые повторные запросы),
- хуже латентность (окно ожидания/таймауты),
- больше «шума» в логах.

### Пошагово: выбираем подход для «статуса заказа»

Требование: показать, как меняется статус доставки.

1) Если статусы меняются редко и «задержка ок» — хватит **обычного GET** + периодический опрос (polling) с разумным интервалом и ETag/If-None-Match.
2) Если нужен push «как только изменилось» и достаточно server → client — выбирай **SSE**.
3) Если инфраструктура не любит длинные соединения — **long-polling** как fallback.
4) Если нужен двусторонний интерактив (чат, игры) — чаще нужен **WebSocket** или отдельный real-time сервис.

### Простыми словами

- Streaming — это «я буду диктовать ответ по мере готовности».
- SSE — это «подпишись на обновления, я буду присылать новости».
- Long-polling — это «позвони мне и жди на линии, пока я не скажу новости; потом позвони снова».

### Картинка в голове

```mermaid
flowchart LR
  A["Client"] -->|GET /resource| B["REST read"]
  A -->|GET /events (SSE)| C["SSE stream"]
  A -->|GET /wait (long-poll)| D["Long-poll"]
  A <--> |WebSocket| E["Realtime channel"]
```

### Как запомнить

- SSE — «push без WebSocket».
- Long-polling — «fallback, но шумный».
- Стриминг ответа — «полезно для больших/долгих ответов».

### Примеры

#### Пример. Long-polling для уведомлений

```http
GET /notifications/long-poll?timeoutMs=25000
```

Если есть событие — 200 с событием. Если нет — 204 и клиент повторяет.

### Практика / реальные сценарии

- Экспорты и отчёты: стриминг экономит память и даёт быстрый «первый байт».
- Статусы заказов: SSE даёт «живость» без сложности WebSocket.
- Внутренние админки: long-polling часто достаточно и проще.

### Типичные ошибки

- Делать SSE, но не учитывать лимиты соединений и таймауты прокси.
- Делать long-polling без backoff — перегрузить API.
- Не разделить «контракт события» (schema) и «контракт REST ресурса».

### Что будет, если…

- …пытаться сделать настоящий real-time на long-polling без дисциплины?  
  Будет много лишних запросов, шум в логах и нестабильность под нагрузкой.

### Проверь себя

1. Почему SSE проще, чем WebSocket, но не заменяет его полностью?
2. В каких случаях стриминг ответа полезнее, чем «сначала посчитай всё, потом отдай»?
3. Почему long-polling может давать «логический DDoS», если сделать неаккуратно?

<details><summary>Ответ</summary>

1. SSE односторонний (server → client) и использует обычный HTTP, поэтому проще по инфраструктуре и клиентам. Но он не даёт двустороннего канала и имеет ограничения по соединениям/буферизации.
2. Когда результат большой/долгий: экспорт, генерация отчёта, поиск с постепенной выдачей, прогресс. Стриминг снижает TTFB и требования к памяти.
3. Потому что клиенты будут постоянно держать и пересоздавать соединения. Без ограничений, backoff и кэширования это превращается в высокую нагрузку и «шум» в системе.

</details>

### Запомните

- В REST-экосистеме есть компромиссы для «почти real-time», но у каждого есть цена.
