## 15.2. Коды ответа и контракт ошибок

### Цель раздела

Сделать API **диагностируемым и предсказуемым**: по одному статус-коду и телу ошибки клиент должен понимать, что произошло и что делать дальше (повторить? исправить запрос? показать сообщение?).

### В этом разделе главное

- 2xx — успех, 4xx — проблема в запросе/правах, 5xx — проблема на сервере.
- Не прячь ошибку в «200 + success=false».
- Договорись о **едином формате ошибок**.
- Разделяй: **сообщение для пользователя** vs **детали для логов**.
- RFC 7807 (Problem Details) — хороший базовый стандарт.

#### Пример: единый контракт ошибок (RFC 7807 + бизнес‑код + trace_id)

Пример ответа, который одновременно:

- понятен клиенту (что показать пользователю),
- диагностируем (что искать в логах),
- стабилен как контракт (код ошибки и поля).

```http
HTTP/1.1 409 Conflict
Content-Type: application/problem+json
```

```json
{
  "type": "https://example.com/problems/order-already-paid",
  "title": "Order conflict",
  "status": 409,
  "detail": "Заказ уже оплачен и не может быть оплачен повторно.",
  "instance": "trace:01J2ABCDEF1234567890",
  "code": "ORDER_ALREADY_PAID",
  "retryable": false
}
```

Мини‑правило: клиент должен уметь **ветвиться по `code`**, а человек — находить инцидент/запрос по `instance` (trace_id).

#### Пример: идемпотентный `POST` через `Idempotency-Key` (чтобы не было “двойной оплаты”)

Ситуация: пользователь нажал кнопку дважды, сеть/таймауты, повтор запроса.

```http
POST /payments
Idempotency-Key: 8d6f0c74-3e1f-4e6a-9b1c-7f2c2a5b2f01
Content-Type: application/json
```

```json
{ "orderId": "ord_123", "amount": 1290, "currency": "RUB" }
```

Правило сервера: если видим тот же `Idempotency-Key` для того же клиента/контекста — возвращаем **тот же результат**, не выполняя списание второй раз.

### Термины

| Термин | Определение |
|---|---|
| **Problem Details (RFC 7807)** | Стандартный JSON для ошибок. |
| **instance** | Идентификатор конкретного случая ошибки (например, URL или trace-id). |
| **type** | Категория ошибки (URI/строка), по которой клиент может понимать класс проблемы. |
| **application error code** | Короткий код бизнес-ошибки (например, `ORDER_ALREADY_PAID`). |

### Теория и правила

#### 1) Семантика кодов ответа (практический минимум)

- **200 OK** — успешно, тело есть.
- **201 Created** — создан новый ресурс (обычно с `Location`).
- **204 No Content** — успешно, тела нет (часто для DELETE).
- **400 Bad Request** — запрос неправильный по форме (валидаторы, JSON, параметры).
- **401 Unauthorized** — нет аутентификации (нет/невалидный токен).
- **403 Forbidden** — аутентификация есть, но прав недостаточно.
- **404 Not Found** — ресурса нет (или вы скрываете факт существования — отдельная политика).
- **409 Conflict** — конфликт состояния (например, версия ресурса устарела, или бизнес-конфликт).
- **422 Unprocessable Entity** — запрос синтаксически ок, но не проходит бизнес-валидацию (используется по-разному; главное — единообразие).
- **429 Too Many Requests** — rate limit.
- **500 Internal Server Error** — неожиданная ошибка сервера.
- **503 Service Unavailable** — сервис временно недоступен (перегруз/maintenance).

Важно: коды — это не «идеальная математика», а **сигналы для поведения клиента и инфраструктуры**.

#### 2) Почему нельзя «200 для ошибок»

Потому что:

- библиотеки HTTP и прокси часто ориентируются на статус-код;
- мониторинг и SLO строятся по 4xx/5xx;
- ретраи и circuit breaker принимают решение на основе кодов;
- безопасность и аудит проще, когда ошибки типизированы.

#### 3) Единый формат ошибок: RFC 7807 как база

Пример Problem Details:

```http
400 Bad Request
Content-Type: application/problem+json
```

```json
{
  "type": "https://example.com/problems/validation-error",
  "title": "Validation error",
  "status": 400,
  "detail": "Field 'email' must be a valid email address.",
  "instance": "/users",
  "traceId": "01J0...ABC",
  "errors": [
    { "field": "email", "code": "INVALID_FORMAT" }
  ]
}
```

Практические правила:

- **`title`** — коротко и стабильно (для логов/клиента).
- **`detail`** — человекочитаемо (может быть локализовано на клиенте).
- **`type`** — «класс ошибки» (можно сделать просто строкой-ключом).
- **`traceId`** (или `requestId`) — критично для поддержки.
- Поле **`code`** (приложения) — удобно для клиентской логики (например, подсветить конкретную форму).

#### 4) Кэширование GET и safe-методов

Одна из сильных сторон REST/HTTP — встроенное **кэширование чтений**. Если:

- ресурс меняется не слишком часто;
- ответ помечен корректными заголовками (`Cache-Control`, `ETag`, `Last-Modified`);

то браузеры, CDN и прокси могут **повторно использовать ответ**, снижая нагрузку на бекенд.

Минимальный практический набор:

- для `GET`:
  - отдавать `Cache-Control` (например, `max-age=60` или `no-cache` там, где кэш нежелателен);
  - отдавать `ETag` — хэш/версию представления;
- принимать условные запросы:

```http
GET /orders/ord_123
If-None-Match: "etag-123"
```

и отвечать:

```http
304 Not Modified
```

когда содержимое не поменялось.

Это особенно важно для:

- тяжёлых списков (дашборды, отчёты);
- публичных/полупубличных ресурсов, которые идут через CDN.

##### Проверь себя по кэшированию

1. Почему именно safe‑методы (в первую очередь GET) обычно кэшируются промежуточными узлами (браузер, CDN, прокси), а POST/PUT/PATCH/DELETE — нет?
2. Чем `304 Not Modified` выгоднее для сервера и клиента, чем повторный `200 OK` с полным телом?
3. В каком случае ты осознанно поставишь `Cache-Control: no-cache` даже для GET‑ресурса?

<details><summary>Ответ</summary>

1. Safe‑методы по определению не должны менять состояние системы, поэтому повторное обращение к тому же ресурсу с теми же заголовками можно обслуживать из кэша без риска нарушить инварианты. Мутации же меняют состояние, и их повторная «отдача из кэша» скорее всего нарушит ожидаемое поведение.
2. При `304` сервер может не пересылать тело — это уменьшает трафик и нагрузку на генерацию ответа, а клиент может мгновенно показать уже имеющиеся данные. Это особенно заметно для тяжёлых JSON‑структур или больших списков.
3. Когда данные слишком чувствительны к устареванию (например, баланс счёта в банке) или содержат персональную/секретную информацию, которую нельзя хранить в общих кэшах. В таких случаях лучше всегда ходить до origin‑сервера, даже ценой дополнительной нагрузки.

</details>

### Пошагово: как спроектировать контракт ошибок

1. Выбери базовый формат (RFC 7807 или свой совместимый).
2. Определи минимальный набор полей (status/type/title/traceId).
3. Определи правила для 400/401/403/404/409/422/429/5xx.
4. Определи формат детализации валидации (список ошибок по полям).
5. Запрети «сырой stack trace» в ответах (это безопасность).
6. Добавь документацию и тесты контрактов.

### Простыми словами

Статус-код — это «цвет светофора», а тело ошибки — это «объяснение инспектора».  
Если всегда горит зелёный (200), то у клиента нет шанса понять, что произошло, пока он не прочитает длинный текст.

### Картинка в голове

```
HTTP status (короткий сигнал)  +  error body (детали)  +  traceId (нить к логам)
```

Если нет `traceId`, поддержка выглядит как: «у пользователя не работает где-то».

### Как запомнить

- **4xx**: «ты (клиент) что-то сделал не так или не имеешь права».
- **5xx**: «мы (сервер) сломались или зависимость сломалась».
- Ошибка должна быть **стабильной** (код/тип), а текст — **человечным**.

### Примеры

#### Пример 1. 401 vs 403

Нет токена:

```http
401 Unauthorized
WWW-Authenticate: Bearer
Content-Type: application/problem+json
```

Есть токен, но прав нет:

```http
403 Forbidden
Content-Type: application/problem+json
```

#### Пример 2. 409 Conflict для бизнес-конфликта

Сценарий: заказ уже оплачен, а клиент повторно вызывает «оплатить».

```http
409 Conflict
Content-Type: application/problem+json
```

```json
{
  "type": "https://example.com/problems/order-already-paid",
  "title": "Order already paid",
  "status": 409,
  "detail": "Order ord_123 is already paid.",
  "instance": "/orders/ord_123/pay",
  "code": "ORDER_ALREADY_PAID",
  "traceId": "01J0...XYZ"
}
```

### Практика / реальные сценарии

- **Формы на фронтенде**: 422/400 с деталями по полям позволяют подсвечивать ошибку прямо в форме.
- **Партнёрские интеграции**: стабильные коды ошибок помогают партнёрам быстро чинить интеграции без переписки.
- **Инциденты**: traceId в ошибке позволяет за минуты найти нужные логи/трейсы.

### Типичные ошибки

- Всегда возвращать **200** и «ошибка внутри JSON».
- Не различать 401/403.
- Возвращать подробности внутренней инфраструктуры в 5xx (утечки).
- Делать «поле message» без кода/типа — клиенты начинают парсить текст.

### Что будет, если…

- …не договориться о формате ошибок?  
  Каждый эндпоинт начнёт возвращать своё. Клиенты будут содержать кучу `if/else`, поддержка станет дорогой.
- …использовать 200 для бизнес-ошибок?  
  Ошибки перестанут быть «видны» инфраструктуре, и вы будете ловить их только через жалобы пользователей.

### Проверь себя

1. Почему `409 Conflict` часто лучше, чем `400`, для конфликтов состояния?
2. Какие поля должны быть в ошибке, чтобы поддержка могла быстро расследовать проблему?
3. Почему опасно возвращать stack trace клиенту?

<details><summary>Ответ</summary>

1. Потому что 409 явно сигнализирует: «запрос сам по себе корректен, но конфликтует с текущим состоянием». Это помогает клиенту и логике ретраев, и делает поведение более предсказуемым.
2. Как минимум: `status`, стабильный `code/type`, `traceId/requestId`, и `detail` (кратко). Для валидации — список ошибок по полям.
3. Это раскрывает внутреннюю структуру системы и может помогать атакующему (пути, версии, зависимости). Кроме того, это делает контракт нестабильным.

</details>

### Запомните

- Статус-код — часть контракта и наблюдаемости.
- Ошибка должна быть стандартизирована и содержать `traceId`.
- Не прячь ошибки в 200.
