## 30.2 API‑контракты и контрактное тестирование

### Цель раздела

Сформировать “контрактное мышление” на границе: как описывать API так, чтобы оно было **версионируемым**, **обратно совместимым**, **генерируемым** (типы/моки) и **проверяемым** (контрактные тесты).

### В этом разделе главное

- **Контракт — это не документация “для людей”.** Это артефакт, который должен участвовать в CI и предотвращать несовместимость.
- CDC (Consumer‑Driven Contracts) помогает, когда потребителей несколько и совместимость важнее “один раз договорились”.
- Контрактные тесты не заменяют E2E, но отлично ловят класс ошибок: “бекенд изменил ответ — фронт упал”.
- Генерация типов/моков из контракта снижает “расхождения” между фронтом и бекендом.

### Термины

| Термин | Коротко |
| --- | --- |
| **OpenAPI** | Схема REST API (эндпоинты, параметры, модели), пригодная для генерации клиентов и валидации. |
| **GraphQL schema** | Контракт GraphQL: типы, поля, запросы/мутации, директивы. |
| **protobuf** | Контракт сообщений (часто gRPC), строгая схема с генерацией кода. |
| **Backward compatibility** | Новая версия API не ломает старых потребителей. |
| **Breaking change** | Изменение, которое ломает потребителя (удалили поле, поменяли тип, поменяли смысл). |
| **Pact** | Инструмент CDC: потребитель публикует «контракт ожиданий», провайдер верифицирует. |

### Теория и правила

#### 1) Контракт API: что именно мы фиксируем

Контракт — это “форма” и “поведение”, которое клиент ожидает:

- **форма**: какие поля есть, какие типы, какие обязательны, какие значения допустимы,
- **поведение**: какие статус‑коды и ошибки возможны, какие правила пагинации/сортировки, какие гарантии идемпотентности.

Важно: контракт — это не только “JSON‑структура”.

Пример “минимального” контракта для экранного эндпоинта в BFF:

- `GET /bff/order-screen?orderId=...`
- возвращает:
  - `order` (id, status, totals),
  - `items` (список),
  - `delivery` (адрес/тайм‑слот),
  - `recommendations` (может отсутствовать при деградации).

##### Проверь себя (30.2.1 — что фиксирует контракт)

1. Почему “контракт = только JSON‑структура” — опасное упрощение? Назови 2 элемента поведения, которые тоже должны быть частью договора.  
2. Как бы ты отличил(а) “контракт экрана” (screen API) от “доменных эндпоинтов” по форме ответа и стабильности?  
3. Придумай один пример контракта ошибки (форма + смысл), который важнее, чем “успешный ответ”.

<details><summary>Ответ</summary>

1. Потому что важно не только “поля”, но и поведение: статус‑коды/ошибки, правила пагинации/сортировки, гарантии идемпотентности, семантика полей.  
2. Screen API ориентирован на UI‑сценарий, часто агрегирует и может меняться вместе с UX; доменные эндпоинты ближе к устойчивым сущностям и инвариантам. Screen API должен быть контрактным, но его эволюция обычно быстрее — поэтому особенно важны deprecation и совместимость.  
3. Например: `403 FORBIDDEN` с кодом ошибки и причиной (нет прав на ресурс) — это критично для UX (“показать нет доступа”, а не “что‑то сломалось”).

</details>

##### 1.1) Мини‑пример контракта в OpenAPI (чтобы “увидеть форму”)

Ниже — не полный файл, а небольшой фрагмент, показывающий, как контракт фиксируется схемой.

```yaml
openapi: 3.0.3
info:
  title: BFF Screen API
  version: 1.0.0
paths:
  /api/screen/order:
    get:
      parameters:
        - name: orderId
          in: query
          required: true
          schema: { type: string }
      responses:
        "200":
          description: Aggregated order screen
          content:
            application/json:
              schema:
                $ref: "#/components/schemas/OrderScreen"
components:
  schemas:
    OrderScreen:
      type: object
      required: [order, items, delivery, recommendations]
      properties:
        order:
          type: object
          required: [id, status]
          properties:
            id: { type: string }
            status: { type: string, enum: [NEW, PAID, SHIPPED, CANCELLED] }
        items:
          type: array
          items:
            type: object
            required: [sku, qty]
            properties:
              sku: { type: string }
              qty: { type: integer, minimum: 1 }
        delivery:
          type: object
          required: [slot]
          properties:
            slot: { type: string, format: date-time }
        recommendations:
          type: object
          required: [items, degraded]
          properties:
            degraded: { type: boolean }
            items:
              type: array
              items:
                type: object
                required: [sku]
                properties:
                  sku: { type: string }
```

Что важно увидеть:

- контракт фиксирует **обязательность**, **типы**, **ограничения**;
- флаг `recommendations.degraded` делает деградацию **явной частью контракта** (а не “просто не пришло”).

##### Проверь себя (30.2.1.1 — OpenAPI как контракт)

1. Почему `required` в схеме — это не “формальность”, а источник breaking‑изменений при неосторожном редактировании?  
2. Какой риск у “слишком строгих” enum’ов в контрактах, и как обычно с ним живут?  
3. Зачем помечать деградацию как поле (`degraded`), а не “удалять блок” при ошибке?

<details><summary>Ответ</summary>

1. Потому что сделать поле required означает “клиент всегда получит это поле”. Если провайдер иногда не может его дать, контракт станет ложным. А изменение optional→required — breaking change для клиентов.  
2. Риск: добавление нового значения enum ломает строгие клиенты. Обычно используют default/UNKNOWN‑ветки, версионирование, CDC‑проверки и осторожную эволюцию.  
3. Чтобы отличать управляемую деградацию от поломанного контракта и чтобы UI/мониторинг могли реагировать предсказуемо.

</details>

##### 1.2) Мини‑пример контракта в GraphQL (чем он отличается по стилю)

В GraphQL контракт чаще всего эволюционирует через добавление полей и deprecation:

```graphql
type Query {
  orderScreen(orderId: ID!): OrderScreen!
}

type OrderScreen {
  order: Order!
  items: [OrderItem!]!
  delivery: Delivery!
  recommendations: Recommendations!
}

type Recommendations {
  degraded: Boolean!
  items: [Recommendation!]!
}
```

Ключевой момент: вместо “/v1, /v2” чаще используют **deprecated‑поля** и правила миграции.

##### Проверь себя (30.2.1.2 — GraphQL контракт)

1. Почему GraphQL чаще эволюционирует через deprecation, а не через версию в URL?  
2. Что произойдёт, если поменять тип поля с nullable на non‑nullable (или наоборот) для клиентов?  
3. Приведи пример, как GraphQL может “скрыть” breaking‑изменение, если команда не дисциплинирует deprecation.

<details><summary>Ответ</summary>

1. Потому что в GraphQL клиент выбирает поля, и добавление полей обычно совместимо. Версионирование чаще делается через постепенное добавление новых полей и deprecation старых.  
2. Nullable→non‑nullable обычно breaking: старые клиенты могут ожидать `null` и падать при отсутствии значения или схема станет несовместимой.  
3. Например, удалить поле без периода deprecation: клиенты, которые его запрашивают, начнут получать ошибки исполнения/валидации.

</details>

##### 1.3) Мини‑пример контракта в protobuf (важное правило совместимости)

protobuf‑контракты часто используют в gRPC, но правило совместимости полезно знать всем:

```proto
message OrderScreen {
  Order order = 1;
  repeated OrderItem items = 2;
  Delivery delivery = 3;
  Recommendations recommendations = 4;
}

message Recommendations {
  bool degraded = 1;
  repeated Recommendation items = 2;
}
```

Главное правило protobuf:

- **нельзя переиспользовать номера полей** (удалили поле — номер “сжигается”),
- иначе старые клиенты начнут читать “не то”, и баг будет крайне трудно диагностировать.

##### Проверь себя (30.2.1.3 — protobuf совместимость)

1. Почему переиспользование номера поля опаснее, чем “просто удалить поле”?  
2. Какая стратегия эволюции protobuf обычно считается безопасной (в 2–3 пунктах)?  
3. Почему protobuf‑контракты часто требуют отдельной проверки breaking‑изменений (по сравнению с “просто написать тест”)?

<details><summary>Ответ</summary>

1. Потому что старый клиент может интерпретировать новое поле как старое (или наоборот): данные “съедут” по смыслу, и это будет выглядеть как загадочная порча данных.  
2. Добавлять новые поля с новыми номерами, не переиспользовать номера, не менять смысл существующих полей; удалённые номера “резервировать/сжигать”.  
3. Потому что ошибки совместимости могут не проявиться сразу в функциональных тестах, но проявятся в смешанных версиях клиентов/сервисов. Автопроверка breaking изменений ловит это раньше.

</details>

#### 2) Версионирование и обратная совместимость: базовые правила

Самые практичные правила (которые реально применяют):

- **добавление новых полей** обычно обратно совместимо, если:
  - поля optional,
  - старые поля не меняют смысл,
  - клиент “терпим” к дополнительным полям.
- **удаление или переименование полей** — почти всегда breaking change.
- **изменение типа поля** (string → number) — breaking change.
- **изменение смысла поля** (“amount теперь с налогом”) — скрытый breaking change, самый опасный.

Практика “deprecated → removal”:

1) объявили поле deprecated, добавили альтернативу;  
2) дали время миграции;  
3) удалили в новой мажорной версии API/контракта.

##### Проверь себя (30.2.2 — обратная совместимость)

1. Почему “изменение смысла поля” опаснее, чем “удаление поля”, хотя формально поле может остаться?  
2. Приведи пример изменения, которое кажется совместимым (“мы просто добавили”), но может сломать клиента.  
3. Какой минимальный процесс делает breaking‑изменения управляемыми?

<details><summary>Ответ</summary>

1. Потому что ломается бизнес‑логика без явного сигнала: типы и поля на месте, но расчёты/интерпретации становятся неверными, баги трудно диагностировать.  
2. Добавили новое значение enum; сделали поле required; начали возвращать `null` там, где раньше было число; изменили контракт ошибок (404→200+error).  
3. `deprecated → migration window → removal` + версионирование/документация + автоматические проверки совместимости (CDC/consumer tests).

</details>

##### 2.1) Где именно “версия” в API: URL, заголовок, схема (и почему это важно решить заранее)

Распространённые стратегии:

- **версия в URL**: `/v1/orders/...`  
  - проще объяснять и дебажить,
  - но легко плодить версии и дубли кода;
- **версия в заголовке** (или content negotiation):  
  - гибко, но сложнее в инструментах и инфраструктуре;
- **версия как эволюция схемы без версий URL** (часто в GraphQL):  
  - добавляем поля, старые помечаем deprecated, затем удаляем.

Плохая практика: “версии нет нигде, но мы иногда ломаем”.

Архитектурное правило:

- версия должна быть **видимой** и **управляемой**, иначе совместимость превращается в инциденты.

##### Проверь себя (30.2.2.1 — где живёт версия)

1. Почему “версия нигде” хуже, чем “версия в URL”, даже если “мы маленькая команда”?  
2. В каком случае версионирование через deprecation в GraphQL может быть удобнее, чем /v1,/v2?  
3. Какой один критерий поможет выбрать стратегию (URL vs schema evolution) в вашем контексте?

<details><summary>Ответ</summary>

1. Потому что breaking изменения становятся невидимыми: клиенты ломаются внезапно, нет механизма миграции и диагностики. URL‑версия хотя бы делает разрыв явным.  
2. Когда клиенты могут выбирать поля и вы хотите эволюционировать API добавлением, постепенно выводя старые поля из использования.  
3. Например: сколько потребителей и насколько независимы их релизы. Чем больше независимых потребителей, тем важнее управляемая эволюция и “видимая версия”.

</details>

#### 3) CDC: почему “потребитель диктует контракт”

Интуиция:

В реальности провайдер часто “не знает”, какие поля реально критичны клиентам. Он может:

- убрать поле “кажется не нужным”,
- изменить формат,
- “почистить” структуру.

А клиент в этот момент падает в проде.

CDC говорит:

- потребитель формализует, что ему нужно,
- провайдер доказывает в CI, что это выдаёт,
- несовместимость ловится до релиза.

Картинка в голове: кто кому что должен

```text
Потребитель (frontend/BFF):
  "Я ожидаю, что ответ содержит поля A,B,C в таком виде"

Провайдер (backend):
  "Я проверяю в CI, что мои ответы удовлетворяют этим ожиданиям"

=> если не удовлетворяет — релиз блокируется
```

##### Проверь себя (30.2.3 — CDC)

1. Почему “потребитель диктует ожидания” не означает “потребитель командует провайдером”?  
2. Как CDC помогает в ситуации “несколько потребителей используют разные поля одного ответа”?  
3. Назови один риск CDC, если применять его без дисциплины (что может пойти не так?).

<details><summary>Ответ</summary>

1. Потому что потребитель фиксирует только то, что реально использует. Провайдер может развивать API дальше, но обязан не ломать эти ожидания без процесса миграции.  
2. Каждый потребитель публикует свои ожидания по “своему подмножеству” — провайдер проверяет совместимость для всех, уменьшая риск случайной ломки.  
3. Риск: контракты станут слишком “широкими” или неактуальными; без broker/версий/verification можно получить ложную уверенность или наоборот чрезмерную бюрократию.

</details>

#### 4) Где живут тесты на границе фронт–бекенд (по плану)

На практике полезно разделить уровни:

- **контрактные тесты (CDC)**:
  - *потребитель* описывает ожидания (контракт),
  - *провайдер* верифицирует контракт в своём CI;
- **моки и типы из контракта**:
  - фронт и BFF могут работать изолированно, не поднимая весь бекенд;
- **интеграционные тесты BFF**:
  - BFF проверяет взаимодействие с реальным бекендом (или “провайдер‑моком”);
- **E2E**:
  - проверяет пользовательский путь “клиент → BFF → сервисы” в тестовом окружении.

Диаграмма: “слои тестов” на границе

```mermaid
flowchart TB
  subgraph Consumer["Потребитель"]
    FE["Frontend"]
    BFF["BFF"]
    CT["Контракт (ожидания)\n+ генерация моков/типов"]
  end

  subgraph Provider["Провайдер"]
    BE["Backend API"]
    PV["Provider verification\n("в CI бекенда")"]
  end

  FE -->|использует типы/моки| CT
  BFF -->|использует типы/моки| CT
  CT -->|публикация контрактов| PV
  PV --> BE

  subgraph E2E["End-to-End"]
    E["E2E тесты"]
  end
  E --> FE
  E --> BFF
  E --> BE
```

##### Проверь себя (30.2.4 — где какие тесты)

1. Чем контрактный тест отличается от интеграционного теста BFF с реальным бекендом?  
2. Почему E2E не заменяет CDC, даже если “E2E всё проверяет”?  
3. Придумай минимальный “пакет” тестов, который даст быстрый выигрыш на старте (из 2–3 пунктов).

<details><summary>Ответ</summary>

1. Контрактный тест проверяет совместимость интерфейса (ожидания consumer ↔ provider) как артефакт; интеграционный тест BFF проверяет реальное взаимодействие BFF с бекендом в конкретном окружении/данных.  
2. Потому что E2E дорогие, медленные, трудно локализуют причину, и могут не покрыть все сочетания потребителей/версий. CDC ловит несовместимость раньше и точнее.  
3. Схема (OpenAPI/GraphQL) в репо + генерация типов для клиента + provider verification по ключевым контрактам (CDC) + 1–2 smoke E2E для критичных потоков.

</details>

#### 5) Pact / Spring Cloud Contract: что проверяется “по сути”

Технические инструменты разные, смысл один:

- контракт описывает конкретные взаимодействия (request → response),
- провайдер запускает тесты, которые подтверждают:
  - при таком запросе он отдаёт ответ, совместимый с контрактом.

Важно: CDC не означает, что потребитель диктует “всю форму API”. Он описывает то, что реально использует. Это снижает риск “случайной ломки”.

##### Проверь себя (30.2.5 — что именно проверяет Pact/контрактные тесты)

1. Почему важно проверять не “всё API”, а “то, что реально использует потребитель”?  
2. Какая категория багов ловится контрактными тестами лучше всего?  
3. Какой сигнал подсказывает, что вам нужны provider states?

<details><summary>Ответ</summary>

1. Иначе сопровождение становится слишком дорогим и контракты быстро устаревают. Цель — защитить реальные зависимости.  
2. “Сломали форму/статусы/ошибки ответа” после изменения провайдера — несовместимость интерфейса.  
3. Когда ответы зависят от состояния данных (“заказ существует/нет”, “нет прав”), и без подготовки состояния верификация становится флейковой.

</details>

##### 5.1) Жизненный цикл CDC/Pact: потребитель → публикация → верификация провайдера

```mermaid
flowchart LR
  C["Consumer tests\n("frontend/BFF")"] --> P["Pact file\n("ожидания")"]
  P --> B["Pact Broker\n("хранилище контрактов")"]
  B --> V["Provider verification\n("CI backend")"]
  V --> R{"Совместимо?"}
  R -->|да| OK["Deploy allowed"]
  R -->|нет| Fail["Deploy blocked"]
```

Ключевые детали, которые часто решают успех:

- **Broker** нужен, чтобы контракты были версионируемыми артефактами, а не “файлом у кого‑то в репо”.  
- Провайдер‑верификация должна быть частью CI **провайдера**, иначе он сможет выкатить несовместимый релиз.

Практический “архитектурный” вывод:

- совместимость должна становиться **гейтом релиза** (идея “can‑i‑deploy”): если текущие версии consumer/provider несовместимы по контрактам, деплой должен быть запрещён автоматически, а не “надеемся, что пронесёт”.

##### Проверь себя (30.2.5.1 — lifecycle и gate релиза)

1. Почему без гейта “совместимость в CI” может не спасти прод?  
2. Где должен жить этот гейт: в CI потребителя, провайдера или обоих? Почему?  
3. Что будет, если провайдер верифицирует контракты “иногда”, а не на каждый релиз?

<details><summary>Ответ</summary>

1. Потому что несовместимый релиз всё равно может быть выкачен, если проверка не блокирует деплой. Тогда совместимость превращается в “совет”, а не контракт.  
2. Минимум — у провайдера (он не должен выпускать несовместимое). Часто полезно и у потребителя (не деплоить consumer, который требует контрактов, ещё не поддержанных провайдером).  
3. Несовместимость попадёт в прод “между окнами проверки” и станет инцидентом, особенно при независимых релизах.

</details>

##### 5.2) Provider states: почему без них контрактные тесты часто “не взлетают”

Потребитель ожидает ответы в конкретных состояниях данных:

- “заказ существует”,
- “заказ не существует”,
- “у пользователя нет прав”.

В CDC это фиксируется как **provider states**: провайдер перед запуском проверки переводит систему в нужное состояние (фикстуры, сидирование, моки зависимостей).

Простыми словами: контрактный тест не должен зависеть от случайного состояния тестовой базы.

##### Проверь себя (30.2.5.2 — provider states)

1. Почему “случайная тестовая база” делает контрактные тесты ненадёжными?  
2. Приведи пример provider state для “нет прав” и что именно должно быть подготовлено.  
3. Чем provider states отличаются от “просто мокнуть всё”?

<details><summary>Ответ</summary>

1. Потому что результаты зависят от данных, которые могут меняться: тесты флейкуют, и команда перестаёт им доверять.  
2. Например: пользователь U не владеет заказом O; нужно подготовить заказ и пользователя, права/роли/tenant так, чтобы запрос гарантированно возвращал 403.  
3. Provider states готовят реальный провайдер к проверке ожидаемого поведения (частично реальная система), а “мокнуть всё” может проверить только мок, но не реальное поведение провайдера.

</details>

##### 5.3) Схема vs CDC: почему одно не заменяет другое

- Схема отвечает: “какая поверхность API существует вообще” (удобно для генерации типов/клиентов).  
- CDC отвечает: “какие конкретные ожидания реально используют потребители и не должны ломаться”.

Они дополняют друг друга: схема даёт “карту”, CDC даёт “страховку” от ломки реальных ожиданий.

##### Проверь себя (30.2.5.3 — схема vs CDC)

1. Почему можно иметь идеальную OpenAPI‑схему и всё равно ломать клиентов?  
2. Почему можно иметь CDC и всё равно иметь проблемы, если нет схемы?  
3. В каких двух случаях CDC особенно полезен?

<details><summary>Ответ</summary>

1. Если изменения схемы не сопровождаются процессом совместимости или если клиенты используют подмножества и провайдер не проверяет реальные ожидания — можно сделать breaking change “по смыслу”.  
2. Без схемы сложнее генерация типов/клиента и сложнее увидеть “поверхность API” целиком; CDC обычно покрывает только используемые взаимодействия, а не все варианты.  
3. Когда потребителей несколько и они релизятся независимо; когда риски несовместимости высоки (BFF ↔ сервисы, фронт ↔ бекенд).

</details>

#### 6) Генерация моков и типов из контракта: зачем это архитектурно

Проблема “на словах”:

- фронт делает свои типы и моки “как понял”,
- бекенд меняет поле,
- на интеграции всё ломается.

Если типы/моки генерируются из одной схемы:

- фронт и BFF получают “единую истину” о форме данных,
- изменения становятся заметными сразу (type errors, diff контракта).

Пример идеи (не привязка к конкретной библиотеке):

```text
openapi.yaml  -> generate types -> frontend uses types
openapi.yaml  -> generate mock server -> frontend dev uses mocks
openapi.yaml  -> contract tests -> backend verifies compatibility
```

Небольшая “приземляющая” подсказка по инструментам (примерно, не догма):

```bash
# OpenAPI: генерация типов/клиента (варианты: openapi-generator, swagger-codegen)
# npx openapi-typescript openapi.yaml -o src/api/types.ts

# GraphQL: генерация типов по схеме и операциям (graphql-codegen)
# npx graphql-codegen --config codegen.yml

# protobuf/gRPC: генерация кода и проверка совместимости (часто через buf)
# buf generate
# buf breaking --against ".git#branch=main"
```

##### Проверь себя (30.2.6 — генерация типов/моков)

1. Почему “типизация из контракта” снижает расхождения сильнее, чем “мы договорились писать интерфейсы одинаково”?  
2. В каком месте pipeline удобнее всего ловить breaking изменения схемы: на PR схемы, в CI провайдера или в E2E? Почему?  
3. Придумай пример, как генерация мок‑сервера помогает фронту и одновременно не ломает контрактность.

<details><summary>Ответ</summary>

1. Потому что типы становятся прямым производным артефактом схемы: любые изменения схемы проявляются как diff/ошибка типов, а не как “помнили/не помнили договорённость”.  
2. Раньше всего — на PR/CI (schema diff + breaking checks + provider verification), потому что E2E придёт поздно и хуже локализует причину.  
3. Фронт использует mock server, сгенерированный из схемы, для разработки без настоящего бекенда; при этом схема — источник истины, а провайдер в CI подтверждает, что соответствует этой схеме/контрактам.

</details>

#### 7) Практический чек‑лист обратной совместимости (очень полезно в ревью)

Перед тем как менять API/контракт, спроси:

- удаляем ли мы поле/эндпоинт? (почти точно breaking)
- меняем ли тип/формат? (breaking)
- меняем ли смысл? (самый опасный breaking)
- добавляем ли поле? (обычно ok, если optional)
- добавляем ли новый вариант enum? (может сломать строгих клиентов)
- меняем ли поведение ошибок/статусов? (может сломать ретраи/обработку)

##### Проверь себя (30.2.7 — чек‑лист совместимости)

1. Почему “добавили поле” может быть совместимым, но “добавили required поле” — почти всегда breaking?  
2. Приведи пример, когда изменение статуса/ошибки ломает ретраи или UX, хотя “данные те же”.  
3. Какая одна привычка в ревью помогает ловить “смену смысла поля”?

<details><summary>Ответ</summary>

1. Required означает обязательное присутствие; старые клиенты могут не ожидать/не обрабатывать отсутствие или по контракту провайдер не сможет гарантировать. Optional обычно не ломает старых клиентов.  
2. Например, вместо 404 стали отдавать 200 с `error` в теле — клиент может считать запрос успешным и вести себя неправильно; или вместо 409 стали отдавать 500 — ретраи начнут дублировать операции.  
3. Требовать описания семантики поля и примеров (контракт + “что значит”), а при изменении — явный changelog и обсуждение миграции.

</details>

##### 7.1) Граничные случаи совместимости (часто забывают)

- **nullable → non‑nullable** (в GraphQL это почти всегда breaking change)  
- **optional → required** (breaking)  
- **изменили единицы измерения/смысл** (“amount теперь с налогом”) — самый опасный breaking change  
- **изменили контракт ошибок** (раньше 404, теперь 200 + `error`) — ломает обработку  
- **добавили новое значение enum** — ломает строгих клиентов, поэтому:
  - держите `UNKNOWN`/default‑ветку,
  - и проверяйте потребителей.

##### Проверь себя (30.2.7.1 — граничные случаи)

1. Почему `nullable -> non-nullable` в GraphQL часто breaking даже без изменения названия поля?  
2. Как `UNKNOWN` ветка в enum помогает переживать эволюцию?  
3. Почему “изменили единицы измерения” — это breaking, даже если тип тот же?

<details><summary>Ответ</summary>

1. Потому что клиенты могут получать `null` и быть к этому готовы; запрет `null` меняет контракт выполнения и может ломать запросы/валидаторы.  
2. Клиент может корректно обработать новое значение как “неизвестное”, вместо падения; это снижает риск при расширении enum.  
3. Потому что меняется смысл и расчёты: 100 рублей vs 100 копеек дают разные бизнес‑результаты.

</details>

##### 7.2) Как диагностировать “контрактный” инцидент в проде (коротко)

Если после релиза фронт начал получать ошибки парсинга/рендера:

- проверь, не изменили ли форму ответа (схема/контракт diff),
- сравни версии consumer/provider (если вы их логируете),
- используй `trace_id` для связи клиентской ошибки и серверных логов.

##### Проверь себя (30.2.7.2 — диагностика)

1. Какие два артефакта ты сравнишь первым делом при подозрении на контрактный инцидент?  
2. Почему версии consumer/provider полезно логировать рядом с `trace_id`?  
3. В чём разница симптомов “сломали контракт формы ответа” и “зависимость упала”?

<details><summary>Ответ</summary>

1. Diff схемы/контракта (OpenAPI/GraphQL/protobuf) и результаты provider verification/CDC.  
2. Чтобы быстро понять, какая комбинация версий столкнулась у пользователя и воспроизвести именно её.  
3. Контракт: ошибки парсинга/рендера/валидации, неожиданные типы/поля. Зависимость: таймауты/502/504 и деградации, при этом форма ответа может быть корректной.

</details>

### Пошагово: как внедрить контрактность без “бюрократии ради бюрократии”

1) Выбери 1–2 критичных API (например, логин и “экран заказа”).  
2) Опиши контракт (OpenAPI/GraphQL/protobuf) и сделай его частью репозитория.  
3) Подключи генерацию типов/клиента для фронта (минимально).  
4) Добавь CDC:

- потребитель публикует ожидания,
- провайдер верифицирует в CI.

5) Введи правило “изменение контракта = видимое изменение”:

- PR‑diff схемы,
- review на совместимость,
- deprecation‑период для breaking changes.

#### Проверь себя (30.2 — пошаговое внедрение контрактности)

1. Почему разумно начинать с 1–2 критичных API, а не “сразу всё описать”?  
2. Что должно произойти в команде/процессе, чтобы пункт “PR‑diff схемы” действительно работал как защита, а не как формальность?  
3. Какой минимальный артефакт ты добавишь, чтобы deprecation‑период был проверяемым (а не “на словах”)?

<details><summary>Ответ</summary>

1. Потому что контрактность требует дисциплины и инструментов; если начать “со всего”, сопровождение станет дорогим и команда потеряет доверие/мотивацию. На критичных API быстрее видно пользу.  
2. Схема должна быть частью ревью, изменения — видимыми, и должна быть проверка совместимости (CDC/provider verification) в CI. Иначе это просто файл, который никто не соблюдает.  
3. Например: changelog/ADR по изменению контракта + метка deprecated в схеме/доках + контрактные тесты, которые подтверждают поддержку старого и нового вариантов в переходный период.

</details>

### Простыми словами

Контракт — это “правила игры” между фронтом и бекендом, записанные так, чтобы компьютер мог проверить. Без контракта вы в итоге всегда приходите к “сломалось после релиза и непонятно почему”.

#### Проверь себя (30.2 — простыми словами)

1. Почему “контракт как проверяемый артефакт” меняет культуру релизов сильнее, чем “документация API”?  
2. Назови один признак, что контракт в команде “живой”, и один — что “мёртвый”.  
3. Как контрактность помогает, когда фронт и бекенд релизятся независимо?

<details><summary>Ответ</summary>

1. Потому что он становится гейтом CI: несовместимость блокирует релиз, а не обсуждается постфактум в инциденте.  
2. Живой: схемы/CDC обновляются вместе с кодом и проверяются в CI. Мёртвый: файл есть, но CI не проверяет, и изменения делают “в обход”.  
3. Позволяет ловить несовместимость до выкатки и планировать deprecation/migration без “внезапных” падений.

</details>

### Типичные ошибки

- **«Контракт есть, но это только документ»** (не участвует в CI) → несовместимость всё равно прилетает в прод.  
- **Breaking changes без deprecation** → фронт ломается “внезапно”, миграции становятся инцидентами.  
- **Контракт покрывает всё подряд** → слишком тяжёлое сопровождение, контракт становится “мертвым”.  
- **CDC понимают как замену интеграции/E2E** → нет проверки реального стека и инфраструктуры.  

#### Проверь себя (30.2 — типичные ошибки)

1. Почему “контракт только документ” не защищает от инцидентов?  
2. Чем опасен “контракт покрывает всё подряд” для скорости разработки?  
3. Почему CDC не может заменить E2E полностью?

<details><summary>Ответ</summary>

1. Потому что нет автоматической проверки: несовместимость всё равно попадёт в прод.  
2. Сопровождение становится слишком дорогим, контракты устаревают, команда перестаёт им доверять.  
3. CDC проверяет совместимость интерфейса, а E2E — реальный стек/инфраструктуру/конфигурации и пользовательский поток.

</details>

### Что будет, если…

- …не версионировать контракт и менять “по-тихому”?  
  Совместимость будет ломаться непредсказуемо: часть клиентов обновилась, часть — нет, а вы это увидите в прод‑ошибках.
- …делать мок на фронте “вручную”?  
  Мок со временем начинает врать. Фронт “работает на моках”, а на реальном бекенде ломается.

#### Проверь себя (30.2 — что будет если)

1. Почему “по-тихому менять контракт” приводит к “непредсказуемым” поломкам именно при частичных обновлениях клиентов?  
2. Какой один процесс помогает сделать deprecation управляемым?  
3. Чем опасен ручной мок без генерации из контракта?

<details><summary>Ответ</summary>

1. Потому что в проде одновременно живут разные версии клиентов; без видимой версии/переходного периода часть клиентов начнёт падать, а часть — нет.  
2. Явная метка deprecated + окно миграции + запрет удаления до выполнения проверок/миграции (CI‑гейт).  
3. Он расходится с реальностью, и команда тестирует “несуществующий API”.

</details>

### Проверь себя

1. Почему добавление нового значения в enum может быть breaking change для некоторых клиентов?  
2. Чем CDC отличается от “мы написали интеграционный тест, который дергает бекенд”?  
3. Где в цепочке “контракт → генерация типов → тесты” ты бы поставил минимум усилий на старте, чтобы получить максимум пользы?

<details><summary>Ответ</summary>

1. Потому что клиент мог писать `switch` по enum без default и считать, что других значений не будет; новое значение приводит к падению или некорректной ветке.  
2. CDC проверяет именно совместимость ожиданий потребителя и реализации провайдера как отдельный артефакт, который может быть опубликован/версионирован и проверен независимо. Интеграционный тест часто локальный и не фиксирует “контракт как объект”, плюс может быть дорогим и редким.  
3. Минимум: держать схему (OpenAPI/GraphQL) рядом с кодом + генерировать типы для фронта + сделать простую провайдер‑верификацию по ключевым контрактам. Это быстро снижает класс ошибок “сломали форму ответа”.

</details>

### Запомните

- Контракт — это артефакт, который должен быть **проверяемым**, а не только читаемым.  
- CDC особенно ценен там, где **потребителей несколько** и совместимость критична.  
- Генерация типов/моков из контракта снижает рассинхрон между командами.
