## Справочник по части 16

### Краткое повторение терминов и идей

| Тема | Ключевые пункты |
| --- | --- |
| Интуиция GraphQL | Решает проблему over/under‑fetching и даёт типизированный контракт; клиент описывает форму ответа. |
| Схема и типы | `schema`, `Query`, `Mutation`, `Subscription`, объектные типы, скаляры, enum; аргументы у полей. |
| Запросы и мутации | `query` для чтения, `mutation` для изменения состояния; запросы декларативны, мутации часто возвращают payload. |
| Резолверы и N+1 | Каждый field → resolver; наивная реализация даёт N+1; нужны batch‑загрузчики и кэш. |
| DataLoader | Собирает множество `load(key)` в один батч‑запрос, распределяет результат и кэширует. |
| Авторизация | Проверки на уровне типов/полей и операций; директивы и middleware; контекст (`ctx.user`, `ctx.scopes`). |
| Ошибки | HTTP‑статус + `data`/`errors`; бизнес‑ошибки в payload; `extensions.code`, `traceId` для диагностики. |
| Кэширование | На клиенте (нормализация), на сервере (persisted queries, кэш доменных запросов); нет простого кэша по URL. |
| Подписки | Долгоживущие операции поверх WebSocket/SSE, питаются доменными событиями (Pub/Sub). |
| Federation | Объединение схем нескольких сервисов в единую через gateway; subgraphs, владение типами, key‑поля. |
| Инструменты | GraphiQL/Apollo Sandbox для изучения схемы; Apollo Studio/инспекторы для мониторинга операций; `graphql-codegen` и аналоги для генерации типов и клиентов. |

#### Проверь себя по справочнику

1. В чём принципиальная разница между **GraphQL‑схемой** и **моделью хранения данных (БД)**?  
2. Почему **N+1** считается архитектурной проблемой GraphQL‑слоя, а не просто «неудачным запросом к БД»?  
3. Какую роль играют **Operation Name и persisted queries** в наблюдаемости и безопасности GraphQL‑API?

<details><summary>Ответ</summary>

1. Схема описывает **доменные типы и операции**, доступные клиентам, и может агрегировать данные из нескольких источников; она не обязана совпадать с таблицами или коллекциями. Модель хранения — это внутренняя реализация; её можно менять, не нарушая схемы (если контракт не ломается).  
2. Потому что GraphQL по своей природе строит **дерево резолверов**, и наивный слой доступа к данным приводит к множеству мелких запросов. Это системный эффект взаимодействия GraphQL и бекенда, который нужно решать архитектурно (слой репозиториев, батчи, лимиты), а не просто переписыванием одного SQL.  
3. Operation Name позволяет видеть, какие именно логические операции вызываются (метрики, алерты, лимиты), а persisted queries позволяют ограничить набор допустимых запросов (whitelist по хэшу), упростить кэширование на прокси и защититься от произвольных/очень тяжёлых запросов.

</details>

## Частые сценарии и практикум

### Сценарий 1. CRM‑панель с богатой навигацией

**Задача.** Есть веб‑CRM:

- экран «Компания» показывает:
  - данные компании,
  - контакты,
  - сделки,
  - задачи.

REST‑вариант:

- `GET /companies/{id}`,
- `GET /companies/{id}/contacts`,
- `GET /companies/{id}/deals`,
- `GET /companies/{id}/tasks`.

GraphQL‑вариант:

```graphql
query CompanyPage($id: ID!) {
  company(id: $id) {
    id
    name
    contacts(limit: 5) {
      id
      name
      email
    }
    deals(limit: 5) {
      id
      amount
      stage
    }
    tasks(limit: 5) {
      id
      title
      dueDate
      completed
    }
  }
}
```

**Как рассуждать:**

- GraphQL даёт **один запрос на экран** вместо нескольких REST‑запросов.
- Но без DataLoader‑подхода появится N+1 (например, при загрузке задач/сделок для многих компаний).
- Архитектурное решение:
  - GraphQL‑слой как BFF,
  - batch‑загрузчики для связей `Company → Deals/Tasks`,
  - лимиты глубины и сложностей запросов.

### Сценарий 2. Мобильное приложение и медленная сеть

**Задача.** Есть мобильное приложение, которое:

- показывает дашборд (несколько виджетов),
- иногда показывает «тяжёлую» аналитику.

Решение:

- для дашборда: один GraphQL‑запрос с нужными полями, упор на:
  - минимизацию полей,
  - кэширование на клиенте,
  - Operation Name и persisted queries;
- для тяжёлой аналитики: возможно, отдельный REST‑/download‑эндпоинт с пред‑сгенерированными отчётами.

### Сценарий 3. Микросервисы и единый API

**Задача.** Есть несколько микросервисов:

- `users`, `orders`, `billing`, `inventory`.

Хочется для фронтенда **единый API**, но не «монолитный» gateway с бизнес‑логикой.

Решение:

- первый этап: GraphQL‑gateway (BFF), который ходит к сервисам по REST/gRPC;
- второй этап: Federation, когда:
  - каждый сервис владеет своей частью схемы,
  - gateway только сшивает.

### Практика после части 16 (самостоятельная)

Выбери один домен:

- интернет‑магазин,
- трекер задач,
- сервис подписок.

И сделай:

1. **Мини‑схему** (3–5 типов, 3–4 поля в `Query`, 2–3 мутации).  
2. Опиши **один GraphQL‑запрос** под конкретный экран.  
3. Определи, **где может появиться N+1**, и придумай, какой DataLoader понадобится.  
4. Ответь: зачем этому проекту GraphQL, и где лучше оставить REST.

#### Проверь себя по сценариям

1. В сценарии CRM‑панели какие связи в первую очередь требуют **батч‑загрузчиков** и почему?  
2. В мобильном сценарии с медленной сетью что важнее всего продумать в GraphQL‑API: структуру схемы, кэширование на клиенте или Federation — и почему?  
3. В сценарии с микросервисами и единым API почему разумно начинать с **GraphQL‑gateway без Federation**, а Federation вводить позже?

<details><summary>Ответ</summary>

1. В первую очередь связи вида `Company → Contacts/Deals/Tasks`, потому что один экран может запрашивать несколько компаний и для каждой — связанный список сущностей. Без батч‑загрузчиков это выльется в N+1‑запросы к БД/сервисам.  
2. Важнее всего продумать **форму запросов и кэширование на клиенте**: минимизировать поля, использовать нормализованный кэш, переиспользовать фрагменты и использовать Operation Name/persisted queries. Federation в мобильном клиенте почти не видна и нужна скорее для организации бекенда, чем для сети клиента.  
3. Потому что один gateway проще запустить, наблюдать и отлаживать; он уже даёт фронтенду единый API. Federation добавляет уровень сложности (subgraphs, ключи, маршрутизация), который оправдан только при росте числа сервисов и команд. Раннее её внедрение может привести к лишней операционной и архитектурной нагрузке.

</details>

## Вопросы для самопроверки по части 16

1. Объясни своими словами, **какую основную боль фронтенда** решает GraphQL по сравнению с REST.  
2. В чём разница между **GraphQL как схемой/контрактом** и просто «слоем над БД»?  
3. Опиши, как из GraphQL‑запроса рождается **последовательность обращений к БД/сервисам**.  
4. Что такое N+1 и как его распознать в метриках/логах?  
5. Когда GraphQL явно избыточен и REST проще?  
6. Что добавляет Federation в архитектуру, по сравнению с простым GraphQL‑gateway?

<details><summary>Ответ</summary>

1. Он решает проблему **over/under‑fetching**: вместо фиксированных REST‑эндпоинтов с заранее заданными формами ответов фронтенд может запросить ровно те поля и связи, которые нужны конкретному экрану, одним запросом.  
2. GraphQL‑схема описывает **доменные типы и операции**, а не таблицы. Она не обязана совпадать с моделью хранилища; под ней могут быть несколько БД/сервисов. GraphQL не даёт прямой произвольный доступ к данным, а лишь к тому, что описано в схеме.  
3. Запрос парсится и валидируется по схеме → строится дерево полей → для каждого поля вызывается резолвер → резолверы обращаются к доменным сервисам/репозиториям → данные собираются в структуру, совпадающую с формой запроса, и сериализуются в JSON.  
4. N+1 — когда один запрос верхнего уровня порождает «лесенку» обращений к БД (один запрос за коллекцией + N запросов за связями). В метриках это видно как большое количество похожих SQL‑запросов с разными ID и очень большое общее количество запросов к БД на один GraphQL‑запрос.  
5. Когда домен прост (CRUD), экранов немного и запросы предсказуемы; когда кэш по URL даёт огромную пользу; когда нет потребности в сложной агрегации данных из разных сервисов. В таких случаях REST дешевле в разработке и эксплуатации.  
6. Federation даёт возможность **разделить владение схемой между несколькими сервисами/командами**, сохранив единый GraphQL‑endpoint для клиентов. Вместо одного монолитного GraphQL‑сервер
а появляются subgraphs и gateway, что помогает масштабировать разработку, но добавляет сложность маршрутизации, наблюдаемости и управления схемой.

</details>

## Типичные ошибки и антипаттерны

- **«GraphQL вместо REST везде»** — переписывание всего API на GraphQL без реальных причин; рост сложности без выгоды.
- **Тонкая обёртка над ORM** в резолверах:
  - N+1,
  - отсутствие явных доменных сервисов,
  - трудная оптимизация.
- **Авторизация только на уровне endpoint**:
  - поля, которые не должны быть видны, случайно утекут через вложенные запросы.
- **Отсутствие лимитов глубины и сложности**:
  - любой клиент может составить запрос, тянущий огромный граф.
- **Отсутствие метрик по операциям**:
  - всё выглядит как `/graphql`, трудно понять, какие запросы «тяжёлые».
- **Единый GraphQL‑монолит без Federation или других мер** в большой системе:
  - одна команда становится «бутылочным горлышком» для всех изменений API.

#### Проверь себя по типичным ошибкам

1. Почему «GraphQL вместо REST везде» опасен именно для **архитектуры и эксплуатации**, а не только для скорости разработки?  
2. Какие последствия у отсутствия **лимитов глубины и сложности** запросов для безопасности и стабильности системы?  
3. Чем плоха «тонкая обёртка над ORM» в резолверах с точки зрения эволюции кода и производительности?

<details><summary>Ответ</summary>

1. Потому что GraphQL добавляет новый слой (схема, резолверы, N+1, авторизация на полях, кэширование), который нужно разрабатывать, тестировать и мониторить. Если домен прост и REST достаточно, это лишний контур сложности, который повышает стоимость поддержки, риск инцидентов и барьер входа для команды.  
2. Без лимитов злоумышленник или неосторожный клиент может отправить запрос с глубоко вложенными полями и большими списками, который потянет огромный объём данных и обойдёт все оптимизации. Это может привести к DoS‑подобным ситуациям: высокое потребление CPU/памяти, перегруз БД, рост латентности и отказ легитимным пользователям.  
3. Такая обёртка скрывает бизнес‑логику в произвольных местах, затрудняет тестирование (нет явных доменных сервисов), усиливает N+1 (каждый резолвер делает свой ORM‑запрос) и делает любую оптимизацию/рефакторинг более болезненной. Гораздо лучше иметь явный слой доменных сервисов/репозиториев и централизованно решать вопросы доступа к данным.

</details>

## Резюме части 16

- GraphQL — это **типизированный контракт и язык запросов** к API, который решает проблему гибких потребностей фронтенда, но добавляет задачи по производительности, авторизации и кэшированию.
- Ключевые элементы:
  - **схема** (`Query`, `Mutation`, `Subscription`, типы, связи),
  - **резолверы** и **слой доступа к данным** (N+1, DataLoader, лимиты),
  - **авторизация и ошибки** на уровне типов/полей,
  - **подписки и Federation** как инструменты для real‑time и микросервисной архитектуры.
- GraphQL хорошо работает:
  - в BFF‑слое,
  - в сложных UI с богатой навигацией по графу данных,
  - как единый API‑слой над несколькими сервисами.
- Он **не заменяет** REST, gRPC и EDA, а дополняет их. Зрелый архитектор выбирает GraphQL **там, где его преимущества перевешивают цену**, и строит вокруг него слой данных, авторизации и наблюдаемости, а не только красивую схему.

#### Проверь себя по резюме части

1. Сформулируй в одном абзаце, **в чём практическая ценность GraphQL** для архитектуры бекенда и фронтенда.  
2. Назови по одному примеру, где GraphQL:
   - (а) явно даёт пользу,
   - (б) добавляет только сложность.  
3. Какие три элемента ты бы проверил(а) в первую очередь, оценивая **зрелость уже существующего GraphQL‑слоя** в проекте?

<details><summary>Ответ</summary>

1. GraphQL даёт возможность фронтенду запрашивать ровно те данные, которые нужны конкретным экранам, одним запросом, и описывает это через типизированную схему, понятную и фронту, и бэку. Для архитектуры бекенда он даёт единый контракт и точку агрегации данных из разных сервисов, но требует дисциплины в резолверах, доступе к данным, авторизации и наблюдаемости.  
2. (а) CRM/дашборд с множеством связанных сущностей и разными представлениями для web/mobile/admin — GraphQL упрощает жизнь фронтенду и снижает связность по эндпоинтам. (б) Небольшой CRUD‑сервис с несколькими фиксированными экранами и высоким упором на кэш по URL — здесь REST проще, а GraphQL добавит слой сложности без ясной пользы.  
3. (1) Есть ли слой доменных сервисов/репозиториев и используются ли батч‑загрузчики для борьбы с N+1. (2) Есть ли метрики и логирование по Operation Name (какие запросы самые тяжёлые, где ошибки). (3) Как устроены авторизация и лимиты сложности/глубины: есть ли field‑level auth, коды ошибок, persisted queries/whitelisting для публичных API.

</details>
