## 16.4. Авторизация, ошибки и кэширование в GraphQL

### Цель раздела

Понять, как **проектировать авторизацию**, ошибки и кэширование в мире GraphQL, где клиент сам выбирает поля и связи, и где один запрос может сочетать «чужие» контексты (например, `me` и `adminPanel`), а также разобраться с **лимитами и сложными запросами**.

### В этом разделе главное

- Авторизация в GraphQL нужна **на уровне типов и полей**, а не только на входе.
- Ошибки бывают:
  - **транспортные** (HTTP‑уровень),
  - **GraphQL‑ошибки запроса** (схема, валидация),
  - **бизнес‑ошибки** внутри мутаций (в `errors`/payload).
- Кэширование сложнее, чем в REST: нет простого «URL → ответ».
- Важны:
  - Operation Name,
  - persisted queries,
  - нормализация и кэширование на клиенте (Apollo/Urql/Relay).
- Нужны **лимиты сложности запросов** и защита от «дорогих» запросов.

### Термины

- **Field‑level auth** — авторизация на уровне отдельных полей (например, `User.email` скрыт).
- **GraphQL error** — элемент в `errors[]` ответа GraphQL, с `message` и, возможно, `path`, `extensions`.
- **Complexity** — оценка «стоимости» запроса (количество полей, вложенность, ожидаемые записи).

### Теория и правила

#### 1) Модели авторизации

1. **Auth только на уровне запроса (`context.user`)**  
   - Проверка прав в корневых резолверах (`Query`, `Mutation`).
   - Недостаточно, если разные поля имеют разные правила доступа.

2. **Auth на уровне типов и полей**  
   - Проверки в резолверах отдельных полей (`User.email`, `Order.internalComment`).
   - Можно использовать декларативные директивы (`@requiresRole("ADMIN")`) или middleware.

Практическое правило:

> Всё, что в REST ты бы **закрыл в отдельный эндпоинт или scope**, в GraphQL должно иметь **явную проверку** в резолвере или на уровне схемы.

#### 2) Ошибки в GraphQL‑ответе

Ответ GraphQL обычно выглядит так:

```json
{
  "data": { ... или null ... },
  "errors": [
    {
      "message": "Access denied",
      "path": ["user", "email"],
      "extensions": {
        "code": "FORBIDDEN",
        "traceId": "01J0..."
      }
    }
  ]
}
```

Уровни:

- **HTTP‑статус**:
  - 200 для «успешного обработанного запроса GraphQL» (даже если есть частичные ошибки),
  - 400/500 для **транспортных/фатальных** проблем (невалидный JSON, ошибка сервера).
- **GraphQL‑ошибки**:
  - нарушение схемы и валидации,
  - ошибки резолверов (доступ, отсутствующий объект),
  - бизнес‑ошибки мутаций (часто выносятся в `payload.errors`).

#### 3) Кэширование: где и как

- **На клиенте**:
  - Apollo/Relay/Urql делают **нормализацию** данных по типу/ID;
  - кэшируются фрагменты и результаты запросов.
- **На сервере/прокси**:
  - кэш по **persisted query ID + variables**,
  - либо по Operation Name и стабилизированным параметрам.
- **На уровне доменных сервисов**:
  - кэш запросов к БД/внешним API (как обычно).

GraphQL плохо сочетается с **простым кэшем по URL**, потому что:

- все запросы идут на `/graphql`,
- тело запроса может быть разным.

#### 4) Наблюдаемость и инструменты

Чтобы GraphQL в продакшене был управляемым, важны **инструменты и метрики**:

- **Инспекция схемы и запросов**:
  - встроенные IDE (`GraphiQL`, `Apollo Sandbox/Studio`, `Altair` и др.),
  - автогенерация документации по схеме.
- **Кодогенерация типов**:
  - генерация типов/клиентов по схеме (`graphql-codegen`, Apollo Codegen, Relay codegen);
  - снижение рассинхронизации между фронтом и бекендом.
- **Мониторинг запросов**:
  - сбор статистики по Operation Name: частота, среднее/персентили латентности, количество возвращаемых записей;
  - отдельные алерты на «дорогие» операции (по времени/объёму/количеству подзапросов).
- **Безопасность**:
  - ограничения на глубину, сложность и размер тела запроса;
  - whitelisting/persisted queries для публичных API.

Без этих элементов GraphQL‑слой легко превращается в «чёрный ящик»: клиенты посылают произвольные запросы, а команда видит только общий рост нагрузки.

### Пошагово: авторизация и ошибки

1. Определи **модели прав**:
   - роли (`USER`, `ADMIN`, `MANAGER`),
   - скопы (`orders:read`, `orders:manage`).
2. Внеси это в `context` (например, `ctx.user`, `ctx.scopes`).  
3. Реализуй **middleware/директивы** для типичных проверок:
   - `@auth(role: "ADMIN")`,
   - `@scope("orders:read")`.
4. В резолверах **явно кидай ошибки** с кодами:

   - `FORBIDDEN`, `UNAUTHENTICATED`, `NOT_FOUND`, `VALIDATION_ERROR`, `INTERNAL_ERROR`.

5. Для мутаций возвращай payload с `errors[]`:

   - чтобы бизнес‑ошибки не путались с системными.

### Простыми словами

GraphQL делает контракты **гибче**, но из‑за этого:

- невозможно «схлопнуть авторизацию в один эндпоинт»;
- нужно думать, **кто имеет право видеть конкретное поле**;
- нужно осознанно решать, какие ошибки видит пользователь, а какие — только логи.

### Картинка в голове

```mermaid
graph TD
  Req["GraphQL request"] --> Auth["Auth middleware\n("проверка токена/ролей")"]
  Auth --> Res["Resolvers"]
  Res -->|field-level checks| SecuredFields["Защищённые поля\n("User.email, Order.internalComment")"]
  Res --> Errors["GraphQL errors + payload.errors"]
```

### Как запомнить

> В GraphQL **нет «бесплатной» авторизации по URL** — она вся в резолверах и схеме.  
> Ошибки должны быть **типизированы и отслеживаемы** (коды + traceId).

### Примеры

#### Пример 1. Запретить доступ к email для других пользователей

Идея: `User.email` доступен только самому пользователю и админам.

Псевдокод:

```pseudo
resolve User.email(user, args, ctx):
  if ctx.user.id == user.id or ctx.user.hasRole("ADMIN"):
    return user.email
  else:
    throw GraphQLError("Access denied", extensions = { code: "FORBIDDEN" })
```

#### Пример 2. Бизнес‑ошибка в мутации

```graphql
mutation PayOrder($id: ID!) {
  payOrder(id: $id) {
    order {
      id
      status
    }
    errors {
      code
      message
      field
    }
  }
}
```

Если заказ уже оплачен:

```json
{
  "data": {
    "payOrder": {
      "order": null,
      "errors": [
        { "code": "ORDER_ALREADY_PAID", "message": "Order is already paid.", "field": "id" }
      ]
    }
  }
}
```

### Практика / реальные сценарии

- **Комбинированные запросы** (dashboard): в одном запросе клиент может запросить и публичные данные, и «админские». Нужны чёткие проверки по каждому полю.
- **Мобильный клиент с плохой сетью**: хочется кэшировать ответы GraphQL, но нужно придумывать ключи и политику кэша.

### Типичные ошибки

- Делать авторизацию только в `Query`/`Mutation`, игнорируя поля.
- Возвращать сырые stack trace/SQL‑ошибки в `errors[]`.
- Не использовать коды ошибок (`extensions.code`), заставляя клиентов парсить текст.
- Игнорировать лимиты сложности и глубины.

### Что будет, если…

- …не ограничивать сложность запросов?  
  - Клиенты (или злоумышленники) смогут строить запросы, которые тянут колоссальные объёмы данных и грузят все зависимости, вплоть до отказа.

### Проверь себя

1. Почему авторизация на уровне только `Query`/`Mutation` может быть недостаточной?  
2. Как ты будешь различать **системную ошибку** и **бизнес‑ошибку** в ответе GraphQL?  
3. Приведи пример лимита на «сложность» запроса.

<details><summary>Ответ</summary>

1. Потому что разные поля одного и того же типа могут иметь разные правила доступа (например, `User.name` — публичен, `User.email` — приватен). Если проверять только на входе, можно случайно выдать лишние данные через вложенные поля.  
2. Системная ошибка попадает в `errors[]` уровня GraphQL с кодом вроде `INTERNAL_ERROR`, и при этом `data` может быть `null` или частично пустым. Бизнес‑ошибки лучше включать в `payload.errors` конкретной мутации (например, `OrderAlreadyPaid`) и не считать это «падением» на уровне транспорта.  
3. Например: дать каждому полю «вес» (1 для простого, 5 для списка, 10 для сложной связи) и ограничить сумму весов (например, 100) для одного запроса. Любой запрос с оценкой выше отклонять или требовать повышенный токен.

</details>

### Запомните

- GraphQL требует **сознательной авторизации и обработки ошибок**.
- Кэширование GraphQL ближе к **кэшу графа данных**, чем к кэшу URL.
