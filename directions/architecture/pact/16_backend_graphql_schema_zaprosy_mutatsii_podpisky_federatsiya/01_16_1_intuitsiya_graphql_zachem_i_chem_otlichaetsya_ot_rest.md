## 16.1. Интуиция GraphQL: зачем и чем отличается от REST

### Цель раздела

Сформировать **интуитивное понимание**, зачем появился GraphQL, какую реальную боль он решает для фронтенда и бекенда, и чем он **архитектурно отличается** от REST и RPC — так, чтобы дальше мы не «тащили» GraphQL туда, где он не нужен.

### В этом разделе главное

- GraphQL решает **проблему гибкого получения данных** (over/under‑fetching), а не «проблему REST».
- Архитектурно GraphQL — это **единый типизированный контракт (схема)** и язык запросов к нему.
- В отличие от REST, где сервер жёстко определяет формы ответов, в GraphQL **клиент формирует структуру ответа** (в рамках схемы).
- GraphQL **не отменяет** REST, gRPC и событийные API — он дополняет их и хорошо вписывается в BFF‑слой.
- Выбор GraphQL всегда связан с **ценой**: N+1, авторизация, кэширование, сложность разработки и наблюдаемости.

### Термины

- **Over‑fetching** — когда API возвращает **больше данных**, чем нужно клиенту (лишние поля/связи).
- **Under‑fetching** — когда одного запроса недостаточно, приходится делать **«лесенку» запросов** (chattiness).
- **BFF (Backend for Frontend)** — слой, адаптирующий API под конкретный тип клиента; часто GraphQL живёт именно здесь.

### Теория и правила

#### 1) Проблема over/under‑fetching в REST

При REST‑подходе типичная картина:

- для страницы «Профиль пользователя» нужно:
  - данные пользователя,
  - последние заказы,
  - уведомления,
  - настройки.

Варианты:

1. Один «жирный» REST‑эндпоинт `/profile` с кучей полей и связей → **over‑fetching** для простых экранов.
2. Много мелких эндпоинтов (`/user`, `/orders`, `/notifications`, `/settings`) → клиент делает **несколько запросов подряд** → under‑fetching (лесенка).

GraphQL предлагает:

> Один endpoint `/graphql`, но клиент **сам описывает нужную структуру данных**.

#### 2) GraphQL как типизированный контракт

Ключевая идея:

- **Сервер публикует схему** — типизированное описание всего, что можно запросить;
- **Клиент формирует запрос**, используя эту схему;
- сервер **валидирует запрос** и возвращает ровно то, что запрошено.

Это похоже на «SQL для API», но:

- GraphQL **ограничен схемой** (нельзя запросить поле, которого нет),
- GraphQL работает **с доменными типами**, а не с таблицами (это не прямая оболочка над БД).

#### 3) GraphQL vs REST: не «кто лучше», а «где чей профиль»

В очень грубом приближении:

- REST:
  - хорошо подходит для **ресурсных, предсказуемых API**;
  - кэшируется по URL;
  - прост для мониторинга и лимитов;
  - отлично работает для публичных API и микросервисов.
- GraphQL:
  - хорош для **богатых UI и сложных графов данных**;
  - упрощает жизнь фронтенду;
  - централизует контракт для нескольких клиентов;
  - усложняет кэш, мониторинг, лимиты и авторизацию.

#### Эксплуатационные риски GraphQL (о которых забывают на старте)

- **Сложнее кэширование**: нет простого «кэш по URL», нужно думать про persisted queries, кеширование по операции, нормализацию на клиенте.
- **Сложнее лимиты и защита от “дорогих запросов”**: нужны depth/complexity лимиты, rate limiting по операциям, защита от GraphQL‑DoS.
- **N+1 и латентность**: без батчинга/дедупликации (DataLoader) один запрос UI может превратиться в десятки походов к данным.
- **Наблюдаемость**: важно метить и логировать **имя операции** и ключевые поля, иначе «/graphql медленно» бесполезно для диагностики.
- **Авторизация “на полях”**: появляется необходимость контролировать доступ не только к эндпоинту, но и к отдельным полям/веткам запроса.

Это не “минусы GraphQL”, а цена, которую нужно принять осознанно — и заранее заложить в дизайн и эксплуатацию.

Практичный **чек‑лист выбора GraphQL vs REST/gRPC**:

- **GraphQL подходит лучше, если**:
  - у тебя **много разных клиентов и экранов** с разными наборами полей (web, mobile, back‑office);
  - домен естественно описывается как **граф связей** (`user → orders → items → merchant…`);
  - есть явная боль over/under‑fetching в REST;
  - есть возможность инвестировать в:
    - слой резолверов и DataLoader‑ы,
    - наблюдаемость (метрики по операциям),
    - авторизацию на уровне полей.
- **REST/gRPC проще и надёжнее, если**:
  - API относительно простое (CRUD, несколько предсказуемых сценариев);
  - кэширование по URL и простые лимиты важнее гибкости;
  - домен больше про «операции/команды» между сервисами, чем про богатый граф (для внутреннего RPC‑взаимодействия между сервисами часто лучше gRPC);
  - команда не готова сейчас к дополнительной сложности GraphQL‑слоя.

### Простыми словами

Представь, что REST — это **набор заранее собранных коробок**:

- «коробка профиля»,
- «коробка заказов»,
- «коробка уведомлений».

Ты можешь взять только целые коробки. Если тебе нужно чуть‑чуть из каждой — ты либо тащишь лишнее (over‑fetching), либо несколько раз ходишь в кладовку (under‑fetching).

GraphQL — это **прилавок с ингредиентами и список того, что разрешено**. Ты приходишь и говоришь:

> «Дай мне пользователя с такими‑то полями, плюс его последние 3 заказа и только статусы, плюс количество непрочитанных уведомлений».

Сервер проверяет, что такой рецепт разрешён схемой, и собирает для тебя ровно то, что ты просишь.

### Картинка в голове

```mermaid
graph LR
  subgraph REST
    C1["Client (UI)"] -->|GET /profile| P["/profile\nжирный ресурс"]
    C2["Client (Lite)"] -->|GET /user\nGET /orders| P2["несколько эндпоинтов"]
  end

  subgraph GraphQL
    C3["Client (UI)"] -->|POST /graphql\n{"query {..."} }| GQL["GraphQL Server\n("схема + резолверы")"]
    GQL --> B1["Backend services / DB"]
  end
```

### Как запомнить

> **REST** — «сервер решает форму ответов по URL».  
> **GraphQL** — «схема + язык запросов, клиент описывает нужную форму в рамках схемы».

GraphQL — не про «быстрее» или «моднее», а про **гибкость контракта между фронтендом и бекендом**.

### Примеры

#### Пример 1. REST vs GraphQL для профиля

REST:

- `GET /users/42` → `{ id, name, email, ... }`
- `GET /users/42/orders?limit=3` → `[ ... ]`

GraphQL:

```graphql
query UserProfile($id: ID!) {
  user(id: $id) {
    id
    name
    email
    lastOrders(limit: 3) {
      id
      status
      total
    }
    unreadNotificationsCount
  }
}
```

### Практика / реальные сценарии

- **Мобильные и веб‑клиенты с разными экранами**: один и тот же REST‑эндпоинт либо перегружен, либо недостаточен. GraphQL позволяет каждому экрану запросить свои поля.
- **Несколько фронтендов (web, mobile, back‑office)**: хочется один типизированный контракт, а не зоопарк REST‑API.

### Типичные ошибки

- «GraphQL — это **замена REST**» → попытки переписать все сервисы на GraphQL без реальной нужды.
- «GraphQL = база данных по HTTP» → попытки отдавать напрямую таблицы и SQL через GraphQL, игнорируя доменную модель.
- «GraphQL решит все проблемы фронтенда» → без внимания к архитектуре бекенда (N+1, авторизация, кэш) всё станет хуже.

### Что будет, если…

- …использовать GraphQL там, где обычный REST‑ресурс с парой эндпоинтов достаточно?  
  - Усложнится инфраструктура, появятся новые риски (N+1, сложные запросы), но реальной выгоды не будет.

### Проверь себя

1. Приведи пример over‑fetching и under‑fetching для знакомого экрана в твоём проекте.  
2. Почему GraphQL считают особенно полезным в BFF‑слое?  
3. Как ты одним предложением объяснишь продуктовым/менеджменту, **зачем** вам GraphQL (если он действительно нужен)?

<details><summary>Ответ</summary>

1. Over‑fetching: эндпоинт `/user` возвращает кучу данных (права, настройки, предпочтения), а экрану нужен только `name` и `avatar`. Under‑fetching: экран дашборда грузит `/user`, `/orders`, `/notifications` тремя запросами подряд.  
2. Потому что BFF как раз и занимается **агрегацией и адаптацией API под конкретный клиент**. GraphQL даёт BFF мощный инструмент: одну схему для нескольких клиентов, возможность собирать данные из нескольких сервисов и отдавать ровно то, что нужно.  
3. «GraphQL позволяет фронтенду запрашивать ровно те данные, которые нужны экрану, одним запросом, вместо множества жёстких REST‑эндпоинтов, и даёт нам типизированный, централизованный контракт между командами».

</details>

### Запомните

- GraphQL — это **про контракт и граф данных**, а не про «модную технологию».
- Он хорошо решает проблему **гибких запросов** и **агрегации данных для UI**, но добавляет серьёзные задачи по **производительности, авторизации и кэшированию**.
