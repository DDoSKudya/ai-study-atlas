## Часть 11. Бекенд: Actor model (акторная модель)

> Цель этой части — построить **устойчивую ментальную модель акторной модели**: что такое актор и mailbox, почему внутри актора нет гонок, как работают **асинхронные сообщения**, **супервизия** и **иерархия акторов**, какие бывают **гарантии доставки** (at-most-once / at-least-once), как акторы сочетаются с микросервисами (часто **акторы внутри сервиса**), и где скрыты типовые риски (потеря состояния при сбое, «слишком жирные акторы», неконтролируемые очереди сообщений).

Actor model — это не «модный фреймворк». Это способ **организовать конкурентность и состояние** так, чтобы:
  
- состояние было **локализовано** и защищено от гонок,
- взаимодействие происходило через **сообщения**, а не через общий доступ к памяти,
- сбои были **ожидаемыми** и «починяемыми» через супервизию.

Часть особенно полезна, если ты сталкиваешься с ситуациями:

- «у нас тысячи параллельных операций по одним и тем же сущностям»;
- «счётчики/балансы/статусы постоянно ловят гонки и дедлоки»;
- «нужно реактивно обрабатывать потоки событий»;
- «нужно удержать сложность параллельной обработки и сделать её предсказуемой».

---

### После этой части ты сможешь

- объяснить **акторную модель** не лозунгом, а через её базовые свойства: **актор + mailbox + обработка сообщений по одному**;
- понимать, почему акторная модель помогает избегать гонок без «магических блокировок»;
- различать:
  - **акторную модель** (состояние + сообщения + супервизия),
  - **очереди/топики** (как транспорт сообщений),
  - **микросервисы** (как архитектура системы по границам),
  - и видеть, как они **сочетаются**, а не «заменяют» друг друга;
- выбирать между подходами для «высокой конкуренции за состояние»:
  - актор,
  - очередь + воркеры,
  - транзакции/локи в БД,
  - in-memory синхронизация,
  и обосновывать выбор по ограничениям;
- понимать супервизию: **restart / stop / escalate**, зачем нужно дерево акторов и как оно ограничивает blast radius;
- знать ключевые идеи про персистентность и распределённость акторов:
  - **персистентные акторы**, снапшоты, event sourcing (на уровне идеи),
  - кластерные акторы и шардинг (на уровне модели);
- распознавать типичные ошибки внедрения акторов и предотвращать их (backpressure, «жирные акторы», отсутствие идемпотентности).

#### Проверь себя по целям части

1. Можешь ли ты внятно объяснить, почему «один актор обрабатывает сообщения последовательно» помогает избежать гонок без mutex?
2. Можешь ли ты описать разницу между «актор как модель конкурентности» и «очередь сообщений как транспорт»?
3. Для сценария «счётчик просмотров с тысячами параллельных инкрементов» можешь ли ты сравнить 3 решения: актор vs очередь+воркеры vs БД?

<details><summary>Ответ</summary>

1. Потому что состояние актора не разделяется между потоками: только один обработчик внутри актора в каждый момент времени меняет его состояние. Нет одновременной записи — значит нет гонки. Параллельность достигается не потоками по одному состоянию, а множеством акторов/сообщений.
2. Actor model определяет, **как** устроены вычисления и состояние (акторы, mailbox, обработка сообщений, супервизия). Очередь — это **как доставлять** сообщения между компонентами (очереди/топики, гарантии доставки, ретраи). Можно иметь акторы без внешнего брокера (локально), и можно иметь брокер без акторов (воркеры со shared state).
3. Актор: локализует состояние счётчика и обрабатывает инкременты последовательно; легко избежать гонок, но нужен контроль очереди и персистентность/репликация по требованиям. Очередь+воркеры: хорошо для масштабирования обработки, но если состояние «общее» — всё равно нужен механизм консистентности (actor/lock/DB). БД: просто и надёжно для многих сценариев, но при очень высокой конкуренции может стать узким местом (локи, contention) и потребовать шардирования/агрегации.

</details>

### Краткая шпаргалка по терминам

| Понятие | Суть простыми словами |
|---|---|
| **Actor model (акторная модель)** | Модель конкурентности: **состояние живёт внутри акторов**, а взаимодействие идёт через **асинхронные сообщения** |
| **Актор** | «Капсула» с **состоянием** и **обработчиком сообщений**; снаружи к состоянию напрямую не лезут |
| **Mailbox (почтовый ящик)** | Очередь сообщений, которые ждут обработки актором |
| **Message (сообщение)** | Данные/команда/событие, которое актор получает и обрабатывает |
| **Address / reference** | «Адрес» актора: чтобы отправлять ему сообщения, не зная, где он физически живёт |
| **Супервизия** | Подход «пусть падает и перезапускается»: родитель следит за детьми и решает, что делать при ошибке |
| **At-most-once** | «Не более одного раза»: сообщение может потеряться, но не дублируется (обычно быстрее/проще) |
| **At-least-once** | «Хотя бы один раз»: сообщение гарантированно доставят, но возможны дубликаты (нужна идемпотентность) |
| **Backpressure** | Механизм «не захлёбываться»: что делать, если сообщений приходит больше, чем успеваем обработать |
| **Persistent actor** | Актор, который восстанавливает состояние после падения из событий/снапшота |
| **Cluster / sharding** | Распределённое выполнение акторов на нескольких узлах + распределение акторов по узлам |
