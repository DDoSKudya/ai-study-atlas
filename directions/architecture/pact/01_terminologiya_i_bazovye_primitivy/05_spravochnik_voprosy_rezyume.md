## Справочник по части 1

### Краткий справочник понятий

| Понятие | Кратко |
|--------|--------|
| Компонент | Логически цельная часть системы с понятной ответственностью и границей. |
| Модуль | Единица поставки/сборки кода (библиотека, пакет, сервис) с публичным интерфейсом. |
| Слой | Горизонтальный уровень (UI, бизнес‑логика, данные и т.п.) с правилами зависимостей. |
| Сервис | Компонент с сетевым API, обычно отдельный процесс/контейнер. |
| Контракт | Формальное описание взаимодействия (API, события, очереди, шины). |
| Граница | Линия раздела между частями системы; пересекается только через контракт. |
| Связность (cohesion) | Насколько элементы внутри модуля относятся к одной задаче. |
| Связанность (coupling) | Насколько сильно модуль зависит от других. |
| API | Синхронный интерфейс запрос‑ответ. |
| Событие | Асинхронное сообщение о факте в прошлом. |
| Очередь/топик | Механизм доставки сообщений, буфер между производителями и потребителями. |
| Хранилище | Место долговременного хранения данных; часть архитектуры, а не просто «табличка». |
| Интерфейс | Обещание набора операций и форматов (контракт). |
| Реализация | Конкретный код и инфраструктура, которые выполняют контракт. |

### Частые сценарии

- **Сценарий 1. Монолит, в котором хочется навести порядок.**  
  Используй понятия компонентов, модулей и слоёв, чтобы:
  - выделить модули по доменам (заказы, пользователи, оплата);
  - навести порядок в зависимостях (уменьшить связанность, поднять связность);
  - подготовиться к возможному выделению сервисов.

- **Сценарий 2. Переход к микросервисам.**  
  Понимание:
  - что такое сервис (и чем он отличается от класса `Service`);
  - что такое контракт и граница;
  - что такое связность и связанность;  
  поможет выбирать разумные границы микросервисов и не строить «распределённый монолит».

- **Сценарий 3. Развитие фронтенд‑архитектуры.**  
  Чёткое разделение:
  - компонент vs модуль vs слой (UI, состояние, данные);
  - понимание, где нужна BFF/SPA/SSR;
  - использование API и событий;  
  позволит строить масштабируемые фронтенд‑архитектуры, интегрированные с бекендом.

### Практика после части 1

Попробуй выполнить упражнение:

1. Выбери знакомый проект (рабочий или учебный).
2. Определи:
   - 2–3 **компонента** в нём;
   - 1–2 **модуля** (по поставке/деплою);
   - какие **слои** можно выделить (UI, бизнес‑логика, данные).
3. Для одного из модулей:
   - опиши его **границу и контракт** (что он обещает наружу);
   - назови один пример **высокой связности** и один — **низкой/высокой связанности**.
4. Для одного взаимодействия между частями системы реши, что уместнее: **синхронный API** или **события/очередь**, и обоснуй выбор (что важно: латентность, количество потребителей, требования к надёжности).
5. Сохрани этот разбор рядом с конспектом — мы к нему вернёмся в частях 3–5 (монолит, слои, модульный монолит) и в частях 9, 12–15 (микросервисы, события, API).

### Вопросы для самопроверки по части 1

1. Сформулируй своими словами разницу между компонентом, модулем, слоем и сервисом.  
   <details><summary>Ответ</summary>
   Компонент — логическая часть системы с определённой ответственностью (например, «каталог товаров»). Модуль — единица поставки/сборки кода (библиотека, пакет, микросервис). Слой — горизонтальный уровень архитектуры (UI, бизнес‑логика, данные) с правилами зависимостей. Сервис — компонент, доступный по сети через API, обычно отдельный процесс/контейнер.
   </details>

2. Объясни, почему чёткие контракты и границы критичны при росте команды и системы.  
   <details><summary>Ответ</summary>
   Без них разные части системы начинают незаметно ломать друг другу предположения: модули и сервисы лезут напрямую в чужие БД, опираются на внутренние детали. Любое изменение приводит к каскаду багов. Чёткие контракты и границы позволяют развивать части системы независимо, тестировать по границам, разделять ответственность между командами и эволюционировать архитектуру без постоянных инцидентов.
   </details>

3. Приведи по одному примеру ситуации, где: (а) высокая связность полезна, (б) низкая связанность спасает от проблем.  
   <details><summary>Ответ</summary>
   (а) Высокая связность полезна, когда вся логика обработки заказов (модели, бизнес‑правила, адаптеры к БД) находится в одном модуле `orders` — изменения в заказах касаются только этого модуля. (б) Низкая связанность спасает, когда модуль `notifications` зависит только от абстрактного интерфейса `Queue` и модельных событий, а не от конкретной реализации БД или бизнес‑логики заказов — смена очереди или детализации домена не ломает уведомления.
   </details>

4. Как по артефактам проекта (структура репозитория, диаграммы, схемы API/событий) ты можешь проверить, что термины из этой части (компонент, модуль, слой, сервис, контракт, граница, API, событие, очередь, хранилище) используются последовательно, а не вперемешку?  
   <details><summary>Ответ</summary>
   Можно посмотреть, как называются и сгруппированы директории и пакеты (есть ли явные модули по доменам или всё в `common`), какие уровни выделены на диаграммах (UI/Domain/Data), как описаны сервисы и их API (есть ли формальные контракты), как разведены API и события (отдельные схемы, топики), как описаны хранилища и кто за них отвечает. Если архитектурные артефакты и кодовую структуру можно легко «прочитать» в терминах из части 1 и они не противоречат друг другу, термины используются последовательно; если же одни и те же вещи называются по‑разному и границы неясны — это сигнал к наведению порядка в языке и структуре.
   </details>

### Типичные ошибки по части 1

- Считать, что термины — «формальность из книжек», а не инструмент для уменьшения путаницы.
- Продолжать называть всё словом «сервис», не различая уровень абстракции.
- Игнорировать связность/связанность при проектировании модулей и сервисов.

### Резюме части 1

- Мы зафиксировали **единый словарь**: компонент, модуль, слой, сервис, контракт, граница, связность, связанность, API, события, очереди, хранилища, интерфейс и реализация.
- Эти понятия — **базовый язык**, на котором будут описаны все последующие части плана (монолит, слои, модульный монолит, микросервисы, SPA/SSR, BFF, EDA, CQRS, данные и масштабирование).
- Чем лучше ты чувствуешь эти термины, тем проще будет понимать более сложные архитектурные стили и обсуждать их с командой **без «каши в терминах»**.
