## Справочник по части 0

### Краткий справочник понятий

| Понятие | Кратко |
|--------|--------|
| Архитектура ПО | Набор ключевых решений о структуре системы и их последствия |
| Архитектура приложений | Структура бекенда, фронтенда, данных и взаимодействий |
| Компонент / модуль / сервис | Относительно цельная часть системы с понятной границей |
| Слой | Уровень, отвечающий за свой тип задач и зависящий только вниз |
| Контракт / API | Явное соглашение о взаимодействии частей системы |
| Trade‑off | Осознанный компромисс между несколькими важными свойствами системы |
| Архитектурный долг | Решения прошлого, которые мешают текущим целям и требуют пересмотра |
| Эволюция архитектуры | Последовательное изменение архитектуры под изменяющийся контекст |

### Частые сценарии

- **Сценарий 1. Ты старший разработчик в монолите и хочешь понять, нужен ли вам переход к микросервисам.**  
  Используй модели из 0.1 и 0.3: какие решения уже приняты, какие бегунки качества важны сейчас, что именно не устраивает в текущей архитектуре.

- **Сценарий 2. Ты фронтендер, который устал от «магического SPA» и хочет понимать, что такое SSR/Islands и когда они уместны.**  
  Опирайся на определения архитектуры, границы трека и модель компромиссов из 0.2 и 0.3 — так легче будет выбирать маршрут по фронтенд‑частям плана.

- **Сценарий 3. Ты тимлид или архитектор, который хочет выстроить дорожную карту развития архитектуры.**  
  Используй структуру глобального плана из 0.4 как чек‑лист: какие части вы уже «закрыли», какие ещё нет, и где вас поджимают архитектурные долги.

#### Мини‑кейсы из проектов: как архитектура “проявляется” в быту

Ниже — короткие ситуации, где архитектура внезапно становится очень конкретной (и обычно болезненной). Это хорошие «якоря», чтобы не воспринимать архитектуру как абстракцию.

**Кейс 1. “Белый экран” после релиза бекенда**

- **Симптомы**: фронт падает на конкретном экране; в логах часто `undefined`/`null`/ошибка парсинга; воспроизводится не у всех (кеши/версии).  
- **Что это на языке архитектуры**: сломалась **граница и контракт** (часть 30), а не “просто баг на фронте”.  
- **Что сделать первым шагом**: зафиксировать контракт (схема), включить deprecation‑правила, добавить контрактные проверки (CDC).

**Кейс 2. “У нас 7 запросов на один экран, всё медленно на мобилке”**

- **Симптомы**: высокий TTI, много сетевых походов, сложная клиентская “склейка”.  
- **Архитектурная суть**: проблема **формы границы** между клиентом и сервером (BFF/агрегация), а не “нужен новый фреймворк”.  
- **Первый шаг**: описать “экранный контракт” и решить, где агрегация живёт (BFF/GraphQL/серверные компоненты) и какой ценой.

**Кейс 3. “Релизить страшно: меняем маленькое — ломаем другое”**

- **Симптомы**: правка “в одном месте” ломает “в другом”, регрессии непредсказуемы.  
- **Архитектурная суть**: слабые **границы модулей/слоёв**, высокая связанность (части 4–7).  
- **Первый шаг**: нарисовать as‑is границы, ограничить зависимости, начать с модульности внутри монолита (часть 5).

**Кейс 4. “Инциденты ночью: непонятно, где именно сломалось”**

- **Симптомы**: “всё медленно”, “всё падает”, но нет ответа “что именно” и “почему”.  
- **Архитектурная суть**: отсутствие наблюдаемости как свойства архитектуры (часть 31), а не “не хватает логов”.  
- **Первый шаг**: единый `trace_id`, структурные логи, метрики по ключевым операциям, алерты по SLO.

### Практика после части 0

Попробуй выполнить небольшое практическое упражнение:

1. Выбери любой знакомый тебе проект (рабочий или учебный).  
2. Сформулируй **одно значимое архитектурное решение** в этом проекте, например:
   - «У нас один монолитный бекенд и SPA‑фронтенд»;
   - «Мы разделили систему на три доменных сервиса и используем BFF для фронтенда»;
   - «Фронт — это MPA, а бекенд — набор REST‑сервисов без общего API‑шлюза».
3. Для этого решения выпиши:
   - **плюсы** (что оно улучшает: скорость разработки, независимость команд, UX и т.д.);
   - **минусы** (что стало сложнее или дороже);
   - **стоимость смены** (что пришлось бы сделать, чтобы это решение заменить на другое).

Сохрани этот разбор рядом с конспектом: к нему полезно будет вернуться после изучения следующих частей плана.

### Вопросы для самопроверки по части 0

1. Сформулируй своими словами, что такое архитектура ПО и чем она отличается от «просто кода».  
   <details><summary>Ответ</summary>
   Архитектура ПО — это набор ключевых решений о том, из каких частей состоит система, как они связаны, какие данные где живут и на каких технологиях всё построено, плюс понимание последствий этих решений. Код — это конкретная реализация этих решений. Можно писать код без осознанной архитектуры, но последствия таких решений всё равно проявятся.
   </details>

2. Назови три архитектурных решения в любом знакомом тебе проекте и попробуй описать, какие trade‑offs они реализуют.  
   <details><summary>Ответ</summary>
   Например: «Монолит вместо микросервисов» (быстрый старт и простота разработки vs меньшая организационная масштабируемость), «SPA вместо MPA» (богатый UX и ощущение скорости vs больший bundle и сложность состояния), «единая БД для всего» (меньше накладных расходов на инфраструктуру vs более плотная связанность модулей и сложность выделения сервисов).
   </details>

3. Какой маршрут по глобальному плану ты бы выбрал на ближайшие 3–6 месяцев, и какие конкретные навыки хочешь получить на выходе?  
   <details><summary>Ответ</summary>
   Возможный ответ: «Я middle‑бекенд‑разработчик. В ближайшие месяцы хочу: (1) укрепить фундамент (части 0–2), (2) разобраться в монолите, слоистой архитектуре и модульном монолите (3–5), (3) получить базовое понимание микросервисов и EDA (8–12), (4) научиться лучше работать с контрактами и связкой фронт–бекенд (30). План: идти по частям в этом порядке, по каждой части делать мини‑конспект и пробовать хотя бы одно улучшение в текущем проекте».
   </details>

### Типичные ошибки по части 0

- Недооценивать значение фундаментальных понятий и пытаться сразу «прыгать в микросервисы».
- Перепутать архитектуру с инфраструктурой или процессами.
- Не связывать архитектурные решения с реальными последствиями для команды и бизнеса.

### Резюме части 0

- Архитектура ПО — это **осознанный набор решений** о структуре системы (бекенд + фронтенд), их последствия и эволюция во времени.
- Трек **«Архитектуры (Backend & Frontend)»** фокусируется на архитектуре приложений: стилях построения бекенда и фронтенда, связке между ними, данных и эволюции.
- Любая архитектура — это **компромисс**: нельзя получить всё сразу, важно понимать, какие качества системы для тебя сейчас приоритетны.
- Глобальный план (части 0–35) — это **карта развития инженера и архитектора**; эта часть 0 даёт тебе язык и ментальные модели, чтобы двигаться по этой карте осознанно.
