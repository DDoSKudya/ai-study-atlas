## Справочник по части 26

### Краткая сводка понятий

| Понятие | Кратко |
| --- | --- |
| Уровни состояния | Локальное, поднятое, контекст, глобальный стор, серверное состояние. |
| UI‑состояние | То, что описывает вид и поведение UI (модалки, вкладки, инпуты). |
| Серверное состояние | Данные, источник истины которых на сервере, клиент — только кэш и UI‑обвязка. |
| Контекст | Способ делиться данными в поддереве без пропс‑дриллинга; хорош для редко меняющихся «фоновых» значений. |
| Глобальный стор | Общее хранилище состояния для кросс‑страничных фич; не должен хранить всё подряд. |
| Кэш запросов | Слой, который управляет запросами, кэшированием, ревалидацией и инвалидацией серверных данных. |
| Оптимистичные обновления | Временное «опережающее» обновление UI с возможным откатом при ошибке. |
| Инвалидация | Пометка кэша устаревшим и инициирование перезапроса. |
| Prop‑drilling | «Сверление пропсами» — признак, что пора поднять состояние, использовать контекст или реорганизовать компоненты. |

### Краткая сводка решений

- **Хранить локально**, если:
  - состояние нужно одному компоненту,
  - оно не влияет на другие части приложения.  
- **Поднимать выше/использовать контекст**, если:
  - одно и то же состояние нужно нескольким соседним компонентам/поддереву.  
- **Использовать глобальный стор**, если:
  - состояние нужно на многих экранах/в шапке и глубоко в дереве,  
  - важна предсказуемость и отладка.  
- **Использовать серверное состояние/кэш**, если:
  - данные живут на сервере,
  - требуются кэш, ревалидация, оптимистичные апдейты.

## Частые сценарии

### Сценарий 1. Список сущностей + фильтры + деталка

- **Задача**: сделать страницу «список заказов + фильтры + панель деталей» так, чтобы её можно было масштабировать.  
- **Решение**:
  - заказы и деталка → серверное состояние (React Query/RTK Query);  
  - фильтры и выбранный ID → поднятое состояние на уровне страницы;  
  - флаги UI (открыта ли панель, какая вкладка активна) → локально/поднято.  
- **Комментарий**: при появлении новых экранов, использующих те же данные, мы переиспользуем серверное состояние, а не дублируем запросы.

### Сценарий 2. Корзина и шапка сайта

- **Задача**: корзина должна быть видна и в шапке, и на страницах товара/каталога, плюс переживать перезапуск вкладки.  
- **Решение**:
  - глобальный стор для корзины (Zustand/Redux/др.);  
  - синхронизация с сервером (серверное состояние для «официальной» корзины);  
  - локальное состояние для временных шагов в оформлении заказа.  
- **Комментарий**: корзина — хороший кандидат для глобального стора, но всё равно важно разделять UI‑этапы и серверную версию.

### Сценарий 3. Настройки темы и локали

- **Задача**: дать пользователю переключение темы и языка, чтобы это влияло на всё приложение.  
- **Решение**:
  - контекст или простой глобальный стор для `theme` и `locale`;  
  - возможная синхронизация с локальным хранилищем/сервером;  
  - UI‑состояние переключателей — локально в компонентах.  
- **Комментарий**: тема и локаль меняются редко, поэтому контекст — почти идеальный вариант.

### Сквозной план “стратегии состояния” (как документ и как миграция, если уже хаос)

Состояние ломает проекты не из‑за Redux/Zustand, а из‑за отсутствия **единой стратегии**: что где живёт, кто владеет, как синхронизируем.

#### 1) Напиши документ “State Strategy v1” на 1 страницу

В документе должно быть:

- список типов состояния (UI / server / session / URL),
- где хранится каждый тип (локально/контекст/стор/query cache),
- правила: “что запрещено класть в глобальный стор”,
- как мы делаем optimistic updates и откаты.

#### 2) Раздели server state и UI state как разные подсистемы

Практика:

- server state ведёт query‑cache (React Query/RTK Query/аналоги),
- глобальный стор хранит только то, что действительно cross‑page и не является “кэшем сервера”.

#### 3) Миграция, если “всё уже в глобальном сторе”

Пошагово:

1) выдели 1 домен данных (например, `orders`) и перенеси чтение/инвалидацию в query‑cache,  
2) оставь в сторе только UI‑обвязку (выбранный id/фильтры/вкладки),  
3) добавь правила/линт на новые поля “UI‑мелочи в сторе” (код‑ревью + соглашение).

#### 4) Метрики успеха

- уменьшается объём “шумного” глобального стора (полей/экшенов),
- меньше ручной синхронизации “после мутации обнови 5 мест”,
- меньше багов “устаревшие данные на экране” (за счёт ревалидации/инвалидации).

1. Сравни **UI‑состояние** и **серверное состояние** по трём осям: источник истины, жизненный цикл, требования к консистентности.  
2. Для экрана «список задач + фильтры + доска Kanban» опиши, какое состояние где бы ты хранил(а) и почему.  
3. Какие признаки того, что в проекте возник антипаттерн «всё в глобальном сторе»?  
4. Как бы ты объяснил(а, о) разработчику, почему «серверные данные только в Redux» без ревалидации — плохая идея?  
5. Сформулируй кратко стратегию состояния для продукта «обучающая платформа с курсами, прогрессом и личным кабинетом».

<details><summary>Ответ</summary>

1. UI‑состояние:  
   - источник истины — клиент,  
   - жизненный цикл привязан к компонентам и навигации,  
   - консистентность — локальная (важно, чтобы UI не противоречил сам себе).  
   Серверное состояние:  
   - источник истины — сервер/БД,  
   - жизненный цикл включает запросы, кэш, ревалидацию, инвалидацию,  
   - консистентность — глобальная (нельзя показывать уже «устаревшие» данные как будто они свежие).  
2. Примерный ответ:  
   - задачи и доски → серверное состояние (кэш запросов);  
   - фильтры → поднятое состояние на уровне страницы;  
   - положения карточек при drag‑and‑drop → локальное/поднятое состояние + оптимистичные апдейты;  
   - выбранная доска → глобальный стор/контекст, если это влияет на другие части приложения.  
3. Признаки:  
   - в сторе десятки/сотни полей, включая вещи вроде `isModalOpen`, `searchInputValue` и т.п.;  
   - сложно понять, к какой фиче относится конкретный кусок стейта;  
   - большинство action’ов и редьюсеров относятся к UI‑мелочам;  
   - частые ререндеры и трудности с отладкой.  
4. Потому что Redux сам по себе не даёт:  
   - кэширования и ревалидации,  
   - повторов запросов и экспоненциального бэк‑оффа,  
   - стратегий `stale-while-revalidate`,  
   - автоматического обновления всех мест, где используются одни и те же данные.  
   В итоге нужно либо поверх Redux строить свой «мини React Query», либо использовать готовое решение.  
5. Возможный ответ:  
   - курсы/модули/уроки → серверное состояние (кэш запросов);  
   - прогресс прохождения → серверное состояние + локальные оптимистичные апдейты;  
   - выбранный курс/урок, статус UI (открыта ли панель, какая вкладка выбрана) → локальное/поднятое состояние;  
   - авторизация, общие настройки пользователя (тема, предпочитаемый язык) → контекст/глобальный стор;  
   - всё это описано в документе стратегии состояния и поддерживается инструментами типа React Query + небольшой стор.

</details>

## Типичные ошибки и антипаттерны

- **Всё в глобальном сторе**:
  - UI‑мелочи (модалки, ховер, временные флаги) живут в Redux/Zustand/др.;  
  - любое изменение тянет каскад перерисовок и усложняет ментальную модель.  

- **Серверное состояние без кэша и ревалидации**:
  - данные читаются один раз и считаются «истиной» навсегда;  
  - нет обновления при возвращении на вкладку или изменении на сервере.  

- **Контекст как «суперглобальная переменная»**:
  - один контекст хранит слишком много разных значений;  
  - из‑за этого рендерится всё поддерево при каждом изменении.  

- **Пропс‑дриллинг вместо реорганизации**:
  - состояние передаётся через множество слоёв, хотя его можно было поднять в общий предок или вынести в контекст/фичу.  

- **Нет явной стратегии состояния**:
  - каждый разработчик решает по‑своему, где хранить данные;  
  - через год никто не может быстро ответить, «кто владеет чем».

### Что будет, если… (сводные граничные случаи)

- …пытаться сразу построить «идеальную» сложную модель состояния (с десятком библиотек и паттернов) в маленьком стартапе?  
  - Риск утонуть в инфраструктуре и не успеть сделать продукт; лучше начать проще и эволюционировать по мере роста.  

- …наоборот, игнорировать архитектурные вопросы состояния «до последнего», пока приложение не разрослось?  
  - Переписывать хаотичный state‑менеджмент в большом продукте **очень дорого**: придётся менять множество экранов одновременно, часто без достаточного покрытия тестами.

### Практика 2. Напиши “State Strategy v1” на 1 страницу

Сделай документ (в любом формате), где есть:

- 4 типа состояния: **UI / server / URL / session**
- 5 правил “что запрещено” (например: “не кладём серверные списки в глобальный стор без query cache”)

<details><summary>Ответ (минимальные правила)</summary>

- server state ведёт query cache; Redux/Zustand — не “кэш сервера”  
- URL хранит навигационный контракт (фильтры/ID), но не PII/токены  
- UI‑мелочи по умолчанию локально; глобально — только действительно cross‑page  
- после мутаций: либо optimistic + rollback, либо invalidate + refetch (должно быть явно)

</details>
