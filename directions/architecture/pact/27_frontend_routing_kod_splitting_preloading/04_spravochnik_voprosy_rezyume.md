## Сквозной процесс: как внедрять оптимизации роутинга/сплиттинга без “улучшили — стало хуже”

В этой теме часто делают одну ошибку: “оптимизировали” локально, а в проде:

- выросло количество запросов,
- ухудшился INP из‑за лишних прелоадов,
- появились chunk‑ошибки из‑за кэша,
- и никто не может доказать, стало ли лучше.

Ниже — процесс, который делает оптимизации **инженерными**, а не вкусовыми.

### 1) Зафиксируй целевую метрику и baseline

Выбери 1–2 целевые метрики на итерацию:

- старт: LCP + размер initial bundle,
- навигация: INP + “time-to-next-screen”,
- стабильность: доля chunk load ошибок.

Сними baseline “до” (реальные p75/p95, а не локальные ощущения).

### 2) Одна гипотеза → одно изменение

Пример гипотез:

- “route splitting уменьшит initial JS на 30%”,
- “prefetch на hover уменьшит время до деталей на 200ms”.

Не делай 5 оптимизаций сразу — ты не поймёшь, что сработало.

### 3) Встрои performance budget в CI

Минимум:

- лимит на initial JS/CSS,
- лимит на самый тяжёлый route chunk,
- запрет на рост дубликатов зависимостей.

Иначе через месяц оптимизация “утонет” в регрессиях.

### 4) Rollout как у бэкенда: canary и откат

Оптимизации загрузки — это архитектурные изменения доставки.

Практика:

- включать prefetch/preload по feature flag,
- выкатывать на часть трафика (если есть возможность),
- иметь быстрый откат (выключить флаг), если INP/ошибки растут.

### 5) Обязательный fallback для chunk‑ошибок

Даже “идеальная” стратегия кэша не исключит редких рассинхронов.

Минимум:

- route‑level error UI,
- retry 1 раз,
- затем “обновить приложение” (и не уходить в бесконечный цикл).

## Частые сценарии

### Сценарий 1. “Кабинет: список → детали → назад”

Цель: быстрые переходы и сохранение контекста.

- **Роутинг**: nested routes под layout “кабинет”.  
- **Загрузка**: детали — отдельный чанк; prefetch при hover/вьюпорте.  
- **Состояние**: выбранный `id` в URL, фильтры в query; данные — серверное состояние (часть 26).  
- **UX**: scroll restoration на back.

### Сценарий 2. “Защищённая страница без мигания”

Цель: не показывать контент без прав и не делать “flash of unauthorized content”.

- guard показывает “checking access…” до решения,
- после — либо рендер страницы, либо редирект на логин/403,
- сервер всё равно проверяет доступ на API.

### Сценарий 3. “Тяжёлый виджет редкий, но важный”

Цель: не тормозить старт, но не раздражать пользователя.

- lazy‑loading виджета,
- хороший skeleton,
- optional prefetch по явному сигналу (например, пользователь открыл меню “Редактор”).

### Сценарий 4. “После деплоя у части пользователей ломаются переходы (chunk load error)”

Симптом:

- приложение открылось,
- но при переходе на редкую страницу появляется ошибка загрузки чанка.

Как рассуждать:

1) **Понять, что это не “ошибка роута” и не “ошибка API”**, а рассинхрон статических ассетов.  
2) **Дать пользователю путь восстановления**:
   - понятное сообщение,
   - кнопка “Перезагрузить приложение”,
   - (опционально) автоматическая попытка перезагрузки после 1 retry.
3) **Починить причину архитектурно**:
   - entry/HTML не должен кэшироваться “навечно”,
   - ассеты должны иметь content hash,
   - при наличии Service Worker — управлять обновлениями, а не “надеяться”.

Мини‑чек‑лист:

- есть ли обработчик ошибок загрузки чанков на уровне роутера/Suspense boundary;
- настроены ли кэш‑заголовки для HTML и статики;
- не “залипает” ли старая версия из-за Service Worker.

## Вопросы для самопроверки по части 27

1. Назови три вещи, кроме “перехода по URL”, которые должен решать роутинг в реальном приложении.  
2. Как бы ты объяснил(а) junior’у, почему “глобальный preload всего” может ухудшить LCP?  
3. Что ты положишь в URL для экрана “каталог + фильтры”, а что точно не положишь?  
4. Если у тебя плохой INP, какие два архитектурных изменения ты попробуешь в первую очередь?  
5. Как отличить “404 маршрута” от “404 ресурса” и почему это важно для UX?

<details><summary>Ответ</summary>

1. Доступ (guards), загрузка данных (loaders), обработка ошибок/404, восстановление скролла/состояния, логирование навигации — любые три из этих.  
2. Потому что браузер будет тратить сеть/CPU на ресурсы, которые не нужны для текущего экрана, и критичные ресурсы (CSS/шрифт/главный JS) могут прийти позже → LCP ухудшится.  
3. В URL: фильтры/сортировка/пагинация/категория. Не в URL: токены, PII, большие структуры данных.  
4. Уменьшить стартовый JS (route‑splitting, убрать тяжёлые зависимости), вынести тяжёлые виджеты в lazy; затем — профилировать long tasks и оптимизировать критические обработчики.  
5. 404 маршрута — URL не соответствует ни одному экрану; 404 ресурса — экран есть, но конкретная сущность отсутствует/недоступна. Для UX это разные сообщения и разные “пути выхода” (например, “вернуться к списку”).

</details>

## Типичные ошибки и антипаттерны

- **Один огромный бандл** “потому что так проще”.  
- **Lazy‑loading без нужды** (лениво грузим то, что нужно сразу).  
- **Prefetch без лимитов** (нагрузка на сеть и API, ухудшение LCP).  
- **Роут без guard для защищённых страниц**: пользователь успевает увидеть “кусок приватного UI”, потом редирект/401; в лучшем случае — плохой UX, в худшем — утечки через неверно спроектированные границы.  
- **Guards только на клиенте** (ложное чувство безопасности).  
- **Отсутствие deep links** (невозможно делиться ссылками, сложнее поддержка).  
- **Оптимизация без измерений** (регрессии и “скорость по ощущениям”).  
- **Нет стратегии для “chunk load error”**: при новом деплое у пользователя в кеше старый app shell, а чанк уже новый → переход ломается без понятного recovery.  
- **Глобальный fallback на весь экран**: один spinner на всё приложение, который прячет контент даже когда грузится маленькая часть.

## Резюме части 27

- Роутинг — это **контракт навигации**: URL ↔ экраны ↔ доступ ↔ данные ↔ ошибки.  
- Code splitting и стратегия загрузки — это часть архитектуры доставки: “что грузим сейчас, что потом, как ускоряем вероятные шаги”.  
- Preload/prefetch дают выигрыш только при **правильном приоритете и ограничениях**.  
- Метрики (LCP/INP/CLS/TTI/TTFB) — ваш “язык правды”: оптимизируем через гипотезы и измерения.

Связь со следующими частями:

- **Часть 28** расширит тему на уровень “несколько фронтендов”: микрофронтенды и границы между ними (роутинг и загрузка там становятся ещё важнее).  
- **Часть 29** углубит “как именно собирается и доставляется код”: чанки, tree‑shaking, кэширование, Module Federation.

## Практика после части 27 (60–120 минут)

Цель практики — собрать роутинг/загрузку/метрики в один “мини‑дизайн”, который можно реально внедрять и измерять.

### Практика 1. Спроектируй роутинг и layout’ы для SPA на 10+ экранов

Выбери пример: “SaaS‑кабинет” или “интернет‑магазин”.

Сделай:

- карту маршрутов (10–15 URL),
- какие из них вложенные (nested routes),
- где стоят layout’ы (app layout, settings layout и т.п.),
- где ставишь error boundaries и почему.

<details><summary>Ответ (минимум)</summary>

Стабильный app layout (шапка/меню) + nested layout’ы для крупных разделов (настройки/кабинет), error boundary на уровне раздела (не ронять всё приложение из‑за одной страницы).

</details>

### Практика 2. Стратегия доставки: что в app shell, что — lazy

Для той же карты маршрутов выпиши:

- что входит в **app shell** (роутер, дизайн‑система, auth контекст, базовые страницы),
- какие страницы/виджеты грузим как route‑chunks,
- какие “тяжёлые штуки” выносим в отдельные lazy‑чанки (редактор, графики).

Добавь 2 правила preload/prefetch:

- один preload (для “нужно сейчас”),
- один prefetch (для “вероятный следующий шаг”),
- и 2 лимита (например, “не больше N prefetch одновременно”, “не на 2G/Save‑Data”).

<details><summary>Ответ (ориентир)</summary>

App shell должен быть минимальным, а тяжёлые виджеты — отдельными чанками. Prefetch ограничиваем по количеству и условиям сети; preload — только для реально критичного ресурса текущего экрана.

</details>

### Практика 3. Набор метрик и “план проверки гипотез”

Сформулируй одну гипотезу и как ты её проверишь:

- гипотеза (например: “route‑splitting уменьшит initial JS на 30% и улучшит INP”),
- метрики до/после (INP/LCP/TTI, размер initial JS, error rate),
- как катить (feature flag / canary),
- как откатывать (выключить флаг).

<details><summary>Ответ (пример)</summary>

Гипотеза → одно изменение → измерение в RUM/CI → rollout через флаг → быстрый откат. Без этого оптимизация превращается в “на глаз”.

</details>
