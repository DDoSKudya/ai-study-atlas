## Справочник и проверка по части 29

### Справочник по части (таблица)

| Тема | Ключевые пункты |
| --- | --- |
| Бандлеры | Роли dev/prod; Webpack/Vite/Rollup/esbuild — выбирать по сценарию, а не по моде |
| Tree‑shaking | Нужны ESM и контроль side effects; `sideEffects` в package.json; библиотека должна поддерживать |

### Частые сценарии (задача → как рассуждать)

#### Сценарий 1. “Бандл вырос на 40%, страница стала тормозить”

- **Шаг 1**: измерь (bundle size, INP/TTI, LCP).  
- **Шаг 2**: bundle analyzer — кто виноват (дубли, “импорт всей либы”, тяжёлый виджет).  
- **Шаг 3**: вынеси тяжёлое в lazy‑чанк или замени зависимость.  
- **Шаг 4**: проверь, что не ухудшил(а) количество запросов/переходы.  
- **Шаг 5**: добавь/обнови **performance budget** в CI, чтобы регрессия не вернулась через месяц.

#### Сценарий 2. “Белый экран после релиза у части пользователей”

- проверь кэш HTML vs чанков,
- проверь наличие content hash,
- добавь обработку ChunkLoadError + fallback,
- проверь CDN и пути до чанков.

#### Сценарий 3. “Две команды мешают друг другу, общий UI расползается”

- монорепо может помочь, если:
  - есть shared design system с владельцем и версиями,
  - есть правила импортов и public API,
  - CI проверяет совместимость.

#### Сценарий 4. “Нужны независимые релизы частей страницы”

- сначала границы и контракты (часть 28),
- затем оценка: Module Federation vs отдельные бандлы vs Web Components,
- обязательно: shared deps стратегия + fallback + мониторинг,
- отдельно: стратегия кэша `remoteEntry` (коротко) и LKG/фичефлаг, чтобы быстро отключать проблемный remote.
- если remotes много и URL часто меняются → введи **remote manifest** (таблица remote → url/version/channel), чтобы убрать хардкод и управлять доставкой.

#### Сценарий 5. “После релиза remote часть пользователей получила несовместимость (host v2 + remote v1)”

- проверь, есть ли **протокол совместимости** (например `protocolVersion`) и deprecation‑период,
- проверь стратегию **кэша `remoteEntry`** (часто должен обновляться быстрее, чем хеш‑чанки),
- включи/проверь **fallback по несовместимости версии** (не пытаться монтировать),
- используй LKG: откат remote или отключение фичефлагом без релиза host,
- зафиксируй инцидент как архитектурное решение: “как обновляем host/remotes” (в идеале — ADR, см. часть 32).

## Процесс обновления host/remotes (делаем совместимость “системной”)

Самый частый корень проблем Module Federation — не Webpack, а **процесс релиза и кэша**. Этот блок — про то, как сделать совместимость управляемой.

### 1) Протокол совместимости и deprecation‑окно

- вводим `protocolVersion` (major),
- host монтирует remote только при совместимом major,
- задаём deprecation‑окно: сколько времени держим старый major (иначе совместимость “на удаче”).

### 2) Remote manifest как единый источник истины доставки

Если remotes больше 1–2, убирай хардкод URL и версий.

Мини‑формат:

| remote | url | version | channel | protocolVersion | lkgUrl |
| --- | --- | --- | --- | --- | --- |
| `catalog` | `https://cdn/.../remoteEntry.js` | `2.3.1` | `prod` | `2` | `.../2.3.0/...` |

Это даёт быстрый откат “переключили manifest”, а не “срочно пересобрали host”.

### 3) Canary + LKG обязателен

- выкатывать remote через channel/manifest (малый % → 100%),
- держать LKG для каждого remote,
- LKG переключается фичефлагом/manifest без релиза host.

### 4) Наблюдаемость совместимости

Минимум метрик:

- доля клиентов по `protocolVersion`,
- частота fallback “incompatible version”,
- error rate монтирования remote,
- время загрузки remote (p95/p99).

### Вопросы для самопроверки (с ответами)

1. Почему “сборка — часть архитектуры доставки”, и какие 3 последствия это имеет в продакшене?  
2. Почему tree‑shaking зависит от ESM и side effects?  
3. Опиши безопасную стратегию кэширования: что кэшируется долго, а что — нет, и почему.  
4. В каких случаях монорепо даёт реальную выгоду, а в каких — превращается в “release train”?  
5. Когда Module Federation оправдан, и какие 2–3 “минимальных условия” должны быть выполнены для безопасного внедрения?

<details><summary>Ответ</summary>

1. Потому что сборка определяет форму артефактов, их кэш и обновления. Последствия: (а) производительность (размер/чанки/приоритеты), (б) устойчивость релизов (кэш и совместимость), (в) скорость разработки (HMR, инкрементальные сборки, CI).  
2. ESM даёт статическую структуру импортов/экспортов, которую можно анализировать. Side effects мешают выкидывать код безопасно: удалишь — изменишь поведение при импорте.  
3. HTML обновляется быстро (revalidate), чанки с content hash кэшируются долго (`immutable`). Это предотвращает ситуацию, когда HTML указывает на устаревшие или несовместимые файлы.  
4. Выгода: общий UI/утилиты/конфиги и атомарные изменения при 1–нескольких командах. Release train: десятки команд с независимыми релизами без чётких пакетов/ownership/правил зависимостей — все блокируют всех.  
5. Оправдан: независимая поставка модулей/микрофронтов. Минимальные условия: контракты границ (lifecycle/nav/data), стратегия shared deps (версии, singleton), fallback на сбой загрузки, и CI/наблюдаемость для диагностики совместимости.

</details>

### Типичные ошибки (с последствиями)

- **Долгий кэш HTML** → пользователи получают старые ссылки на чанки → белый экран.  
- **Секреты/чувствительные данные в клиентском бандле** → утечки (это не “почти секрет”, это публично).  
- **Shared‑пакеты без владельца** → непредсказуемые breaking changes, тормоза команд.  
- **UI‑пакет тянет `react` как dependency (а не peer dependency)** → риск “две React”, лишний вес и тонкие баги (особенно в MF).  
- **Библиотека публикуется без корректных `exports/types`** → у потребителей “не находятся модули/типы”, ломается сборка или tree‑shaking, появляются странные импорты “внутрь dist”.  
- **CSS‑side effects в библиотеке не обозначены** (`sideEffects` не настроен) → стили могут “пропасть” после оптимизаций или начать вести себя непредсказуемо.  
- **Module Federation без fallback** → сеть/CDN превращают UI в неработающий.  
- **Загрузка remote без таймаута** → вечные спиннеры и “зависшие” экраны; без наблюдаемости непонятно, где проблема (CDN/CSP/версии).  
- **`remoteEntry` кэшируется “как статика на год”** → смешанные версии host/remote у пользователей и случайные несовместимости.  
- **Захардкоженные URL remotes без окружений/конфига** → сложные деплои, “всё ломается при смене CDN/домена”.  
- **Отсутствие budgets/гейтов в CI** → бандл “распухает” незаметно, а деградация производительности копится месяцами.  
- **Нет детерминированных идентификаторов/стабильности чанков** → мелкие правки инвалидируют кэш больших файлов.  
- **Оптимизация без измерений** → “сделали лучше”, а стало хуже (например, много чанков ухудшили INP).

### Резюме части

- Сборка фронтенда — это **архитектура доставки**: чанки, кэш, ассеты, отладка и безопасность.  
- Tree‑shaking и splitting работают хорошо, когда выполнены условия (ESM, side effects, разумные границы чанков).  
- Монорепо — это **граф пакетов + правила + ownership**; оно ускоряет, если границы правильные, и замедляет, если превратилось в “всё зависит от всего”.  
- Module Federation даёт независимую поставку, но требует контрактов, стратегии shared‑версий и fallback’ов.  

Дальше по плану логично идти в **часть 30** (связка фронт–бекенд: BFF, контракты, аутентификация), потому что после “доставки кода” следующий крупный архитектурный вопрос — **как фронт договаривается с бекендом и как безопасно живёт на границе**.

## Практика после части 29 (90–180 минут)

Цель практики — перевести “сборка/монорепо/MF” из теории в **набор решений и чек‑листов**, которые реально предотвращают инциденты (ChunkLoadError, “две React”, release train).

### Практика 1. Инцидент‑разбор: “ChunkLoadError после релиза”

Составь для себя (и команды) короткий runbook из 10–15 пунктов:

- как распознать (по ошибкам/логам),
- что проверить первым (кэш HTML vs статики, content hash, publicPath/CDN),
- что сделать быстро (fallback “обновить приложение”, retry 1 раз),
- как исправить системно (заголовки кэша, стабильность чанков, SW‑политика).

<details><summary>Ответ (минимальный скелет)</summary>

1) HTML не кэшируем “навечно”; 2) ассеты только с content hash; 3) есть обработчик ChunkLoadError; 4) при наличии SW — управляем обновлением и версиями кэша; 5) проверяем CDN‑пути/headers/manifest.

</details>

### Практика 2. Монорепо‑дизайн: граф пакетов и ownership

Набросай граф из 6–12 пакетов для монорепо (даже если это “воображаемое”):

- `app-web`, `app-admin`,
- `ui-kit` (design system),
- `api-client` (контракты/SDK),
- `shared-config` (eslint/tsconfig),
- `domain-*` или `entities-*` (по доменам).

Для каждого пакета укажи:

- owner (команда/роль),
- public API (что экспортируем наружу),
- правила зависимостей (что кому можно импортировать),
- версионирование (нужно ли SemVer и deprecation).

<details><summary>Ответ (ключевое)</summary>

Самое важное в монорепо — не инструмент, а **границы и правила**: ownership + public API + проверки зависимостей в CI, иначе монорепо быстро превращается в “болото”.

</details>

### Практика 3. Module Federation: “минимальные условия безопасности”

Сделай чек‑лист внедрения MF (12–20 пунктов), где есть:

- кэш: `remoteEntry` обновляется быстрее, чем хеш‑чанки,
- shared deps: какие пакеты singleton и какие версии допустимы,
- протокол совместимости (`protocolVersion`) и deprecation‑окно,
- fallback: что показываем при недоступном/incompatible remote,
- наблюдаемость: метрики загрузки remote, доля incompatible, error rate mount.

<details><summary>Ответ (минимум)</summary>

MF без (а) стратегии shared deps/версий, (б) fallback/recovery, (в) политики кэша `remoteEntry`, (г) наблюдаемости — почти гарантированный источник “плавающих” прод‑инцидентов.

</details>
