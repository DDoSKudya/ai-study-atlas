## Справочник по части 7

### Краткий справочник: слои и понятия

| Тема | Ключевые пункты |
|------|------------------|
| Clean Architecture | Entities → Use Cases → Interface Adapters → Frameworks & Drivers; зависимости внутрь; домен независим от инфраструктуры. |
| Onion Architecture | Домен в центре; вокруг — приложение и инфраструктура; те же зависимости внутрь; акцент на «луковичных слоях». |
| Entities | Долгоживущие доменные сущности и правила; минимальная зависимость от технологий. |
| Use Cases | Сценарии использования: координация сущностей для выполнения конкретной задачи; определяют интерфейсы к инфраструктуре. |
| Interface Adapters | Контроллеры, presenters, репозитории, gateways; переводят данные и протоколы внешнего мира во внутренний формат. |
| Frameworks & Drivers | Веб‑фреймворки, ORM, драйверы, брокеры, UI‑фреймворки; самый внешний слой, который не должен влиять на домен. |
| Инверсия зависимостей | Внутренний слой определяет интерфейсы; внешние слои реализуют их через конкретные технологии. |
| Анемичный домен | Сущности без поведения; нарушает идею домена в центре; логика размазана по сервисам и контроллерам. |

---

### Частые сценарии (как рассуждать)

- **«Нужно добавить ещё один способ вызова той же логики» (новый API/очередь/CLI).**
  - Ищем соответствующий use case.
  - Добавляем новый адаптер, который вызывает use case.
  - Проверяем, не протаскиваем ли детали протокола внутрь.

- **«Меняем БД или ORM»**
  - Интерфейсы репозиториев остаются прежними.
  - Реализуем новые адаптеры поверх новой БД/ORM.
  - Доменные сущности и use cases остаются как есть.

- **«Хотим улучшить тестируемость домена»**
  - Проверяем, что домен не зависит от фреймворков.
  - Пишем unit‑тесты для сущностей и use cases с in‑memory репозиториями.

### Вопросы для самопроверки по части 7

1. В чём общая идея Clean Architecture и Onion Architecture и чем они помогают по сравнению с простой слоистой архитектурой?

<details><summary>Ответ</summary>

Общая идея — **домен в центре и зависимости внутрь**:  
бизнес‑логика не зависит от фреймворков, БД и протоколов,  
инфраструктура подключается через адаптеры, реализующие интерфейсы, определённые во внутренних слоях.  
По сравнению с простой слоистой архитектурой Clean/Onion более чётко фиксируют это правило и позволяют  
легче тестировать домен, менять технологии и добавлять новые интерфейсы без дублирования логики.

</details>

2. Как ты поймёшь по коду, что проект формально «выглядит как Clean Architecture», но по сути не следует её духу?

<details><summary>Ответ</summary>

Если:

- сущности не содержат поведения, а только данные;
- доменные правила размазаны по use cases, контроллерам и ORM‑хукам;
- внутренние слои зависят от фреймворков и инфраструктуры;  
- добавление нового интерфейса требует копировать бизнес‑логику,  
— значит, архитектура использует названия слоёв, но **не соблюдает ключевые принципы** (домен в центре, зависимости внутрь, инверсия зависимостей).

</details>

3. Опиши, как бы ты постепенно рефакторил небольшой сервис с логикой в контроллерах в сторону Clean/Onion.

<details><summary>Ответ</summary>

Примерный план:

1. Выбрать один важный сценарий (use case) и явно его назвать.  
2. Выделить доменные сущности и перенести в них правила (инварианты, валидацию, переходы состояний).  
3. Создать use case‑класс, который:
   - принимает входные данные в удобной форме (DTO);
   - использует сущности и репозитории;
   - не зависит от HTTP и ORM.  
4. Определить интерфейсы репозиториев и шлюзов во внутреннем слое.  
5. Реализовать адаптеры (репозитории, контроллеры), которые переводят внешние данные и вызывают use case.  
6. Написать unit‑тесты для домена и use case; оставить интеграционные тесты для adapter‑слоя.  
7. Повторять шаги для следующих сценариев по мере необходимости.

</details>

### Типичные ошибки по части 7 (сводка)

- Пытаться внедрить Clean/Onion **без чёткого понимания целей**.
- Использовать названия слоёв, но **оставлять домен завязанным на инфраструктуру**.
- Вводить слишком много слоёв с минимальной логикой, создавая **схему «просто делегируем дальше»**.
- Останавливаться на полпути: вынесли сущности, но продолжили писать логику в контроллерах.

#### Антипример: “Clean как бюрократия” (как это выглядит и как лечится)

**Как выглядит в коде/команде:**

- на один простой CRUD появляется 6–10 классов “на будущее”;
- use case ничего не делает, кроме “передать дальше”;
- сущности пустые (анемичный домен), а правила живут в контроллерах/ORM‑хуках;
- разработчики боятся менять структуру, потому что непонятно, где “правильно”.

**Почему это происходит:** стиль внедряют как “шаблон папок”, а не как инструмент под цели (тестируемость домена, независимость от инфраструктуры, добавление интерфейсов).

**Как лечить (практично):**

1. Выбрать 1–2 сценария, где реально важны правила/инварианты, и вынести *туда* доменные сущности и use case.  
2. Для простых CRUD оставить упрощённую структуру (не строить “собор абстракций”).  
3. Проверять принцип: “добавление нового интерфейса (например, очередь вместо HTTP) не требует копировать бизнес‑логику”.
