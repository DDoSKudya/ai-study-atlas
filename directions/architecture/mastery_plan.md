# Глобальный план освоения «Архитектуры (бекенд и фронтенд)»

> Цель: осознанно проектировать и эволюционировать архитектуру backend- и frontend-систем — от базовых слоёв и компонентов до микросервисов, событийных систем, микрофронтендов и современных гибридных схем; понимать trade-offs, антипаттерны и критерии выбора; уметь документировать и мигрировать архитектуру.
>
> **Структура плана:** каждая архитектура/способ из «Карты всех архитектур» выделена в **отдельную часть** (отдельную тему). Так план готов к последующей детальной проработке: каждая часть может быть подробно расписана в своих материалах (0–35 частей).

- [ ] **Границы плана:** в план входит архитектура приложений (бекенд и фронтенд): структура, границы, контракты, данные, развёртывание, эволюция. Не входит как отдельные курсы: глубокий DDD (тактическое моделирование), детали сетевых протоколов, управление проектом и людьми; эти темы упоминаются как смежные и «что изучать дальше».
- [ ] **Результат обучения:** по завершении плана вы сможете обоснованно выбирать и комбинировать архитектурные подходы под контекст, читать и рисовать диаграммы (слои, сервисы, потоки), выявлять типичные ошибки и антипаттерны, документировать решения (ADR) и планировать миграции.

---

## Предварительные знания / что желательно знать заранее

- [ ] Базовый опыт разработки: написал backend- или frontend-приложение (хотя бы учебное), понимаешь, что такое API, база данных, HTTP.
- [ ] Представление о клиент-серверной модели, запрос-ответ, асинхронности (callback/promise/async).
- [ ] Для бекенд-части: основы одного из языков (Java, C#, Go, Node.js, Python и т.д.), работа с БД (SQL или NoSQL).
- [ ] Для фронтенд-части: основы HTML/CSS/JS, один из фреймворков (React, Vue, Angular, Svelte) или ванильный JS.
- [ ] Для частей про масштабирование и инфраструктуру: общее представление о сетях, процессах, контейнерах — полезно, но не обязательно для старта.
- [ ] Для продвинутых частей (микросервисы, event sourcing, микрофронтенды): опыт поддержки хотя бы одного «живого» приложения.
- [ ] Представление о семантическом версионировании (semver) и обратной совместимости — полезно для частей 10, 20 (API, контракты).
- [ ] Базовое представление о сетях (DNS, HTTP, запрос-ответ) и о деплое (что такое сборка, окружение) — полезно для частей 12–13, не обязательно для старта.
- [ ] **Чего не требуется для старта:** глубокое знание DDD, Kubernetes или конкретного облака; план вводит нужные концепции по мере необходимости. При затруднениях можно вернуться к смежным темам (см. часть 35).
- [ ] **При отсутствии части предварительных знаний:** начинать с частей 0–2 и блока «Как читать этот план»; при необходимости параллельно восполнять базу (сети, HTTP, один язык/фреймворк); не блокировать старт по плану из-за идеального набора навыков.

## Как читать этот план

1. **Порядок:** иди по шагам 0 → 1 → … → N. Поздние разделы опираются на ранние; без базы (части 0–2, 3–4) сложно осмысленно изучать микросервисы или микрофронтенды.
2. **Роли:** новичок — шаги 0–7 (бекенд-ядро) или 0–2 + 21–27 (фронтенд-ядро); практик — до 20 (бекенд) и до 29 (фронтенд); архитектор/лид — весь план + акцент на 8–17, 28–29, 30–33.
3. **Маршруты по стеку:** если ты только бекенд — части 0–20, 30–33; только фронтенд — 0–2, 21–29, 30–33; fullstack — весь план по порядку (0–35).
4. **Практика:** после каждого крупного шага — мини-проект или рефакторинг: нарисовать диаграмму текущей/целевой архитектуры, применить один приём из части.
5. **Блоки в контенте:** «Простыми словами», «Картинка в голове», «Проверь себя», «Запомните», «Типичные ошибки», «Антипаттерны» — создаются в отдельных учебных файлах по каждой части; в плане указано, где они уместны.
6. **Детализация плана:** каждый пункт и подпункт с чекбоксом [- [ ] ] — тема для раскрытия в учебном контенте; по плану можно однозначно определить, что должно войти в материалы по каждой части. В каждой части явно указаны цель части, связь с другими частями, типичные ошибки/ограничения, при необходимости — «Практика после части N» и блок «Проверь себя» в контенте.
6a. **Результаты по частям:** раздел «Результаты по частям (чек-лист для самопроверки)» перечисляет, что вы должны уметь или знать после каждой части; использовать для самопроверки перед переходом дальше.
7. **Связи между частями:** в начале каждой части указано, на что она опирается и что готовит; используй для навигации и повторения.
8. **Карта всех архитектур:** раздел «Карта всех архитектур (полный перечень способов)» перечисляет все значимые архитектуры и способы организации бекенда и фронтенда; по нему можно проверить полноту обзора и найти, в какой части плана раскрыт каждый способ.
9. **Поперечные маршруты:** безопасность (части 15–17, 20, 30, 31), производительность (19, 27, 29), контракты и API (15–17, 30), эволюция (32, 33), данные и консистентность (9, 12–14, 18, 26).
10. **Где в архитектуре живут тесты:** по слоям и границам — части 4 (тесты по слоям), 5 (тесты по модулям и контрактам), 6–7 (unit ядра/use cases, интеграционные адаптеров), 9 (unit внутри сервиса, контрактные между сервисами, E2E), 30 (контрактные фронт–бекенд, Pact); инструменты ArchUnit, dependency-cruiser, Pact — в соответствующих частях; сводка в «Основные добавления».

## Карта структуры плана

- **База (части 0–2):** без этого нельзя осмысленно рассуждать об архитектуре; общий язык, цели, ограничения, trade-offs. В каждой части — подпункты по аспектам, типичным ошибкам и граничным случаям.
- **Бекенд-ядро (части 3–7):** монолит, 2/3/4-tier, слоёная (N-tier), модульный монолит, гексагональная и Clean Architecture — фундамент любого серверного дизайна. Детализация: транзакции, инструменты, вертикальные срезы, подготовка к разрезу, ограничения подходов.
- **Бекенд-расширение (части 8–20):** SOA и ESB; микросервисы (границы, database per service, service mesh, API Gateway); cell-based, actor model; EDA, сообщения, pipe-and-filter; CQRS, event sourcing; API (REST, GraphQL, Federation, gRPC, tRPC, BaaS); данные (репликация, кворумы, шардирование, space-based, миграции, регуляторика); масштабирование (health checks, jitter); развёртывание (стратегии деплоя, GitOps, rollback, graceful shutdown).
- **Фронтенд-ядро (части 21–27):** типы приложений (MPA, SPA, SSR, SSG, ISR, части 21–24); islands, гидрация, resumability, PWA, TWA; компоненты (Atomic Design, FSD, folder-by-feature, MVC/MVVM/MVI, часть 25); состояние (часть 26); роутинг (Suspense, prefetch, scroll restoration, часть 27).
- **Фронтенд-расширение (части 28–29):** микрофронтенды (навигация, передача данных, тестирование композиции); сборка и монорепо (анализ бандла, source maps, Module Federation).
- **Связка и качество (части 30–33):** BFF, контракты, аутентификация (часть 30); безопасность, наблюдаемость (логи, метрики, трейсы — три столпа; SLI/SLO, алертинг, часть 31); эволюция (as-is/to-be, shadowing, ADR, C4, часть 32); антипаттерны и критерии выбора (чек-лист, зрелость, часть 33).
- **Современность и экспертиза (части 34–35):** edge, serverless, server components, JAMstack, offline-first (часть 34); когда привлекать архитектора, сводка выбора, что изучать дальше (часть 35).

## Карта всех архитектур (полный перечень способов)

Ниже перечислены **все значимые архитектуры и способы организации** бекенда и фронтенда. **Каждая структура — отдельная тема (отдельная часть) в глобальном плане**, чтобы в будущем её можно было подробно расписать. Ссылки «→ часть N» указывают на единственную часть, где дана детализация этой структуры.

### Бекенд: архитектуры и способы организации

- [ ] **Монолит:** одно развёртываемое приложение, один процесс (или тесно связанные процессы). → часть 3.
- [ ] **2-tier (двухзвенка):** клиент + сервер (приложение и БД в одном или рядом); «толстый» клиент или «толстый» сервер. → часть 3.
- [ ] **3-tier (трёхзвенка):** клиент, сервер приложений, СУБД; классическое разделение. → часть 3.
- [ ] **4-tier и N-tier:** добавление слоя инфраструктуры (очереди, внешние API) между бизнес-логикой и данными. → часть 3.
- [ ] **Слоёная (layered) архитектура:** горизонтальные слои (презентация → бизнес-логика → данные); зависимости только вниз. → часть 4.
- [ ] **Shared-nothing:** узлы не разделяют память и диск; масштабирование за счёт независимых копий и партиционирования данных. → части 9, 18, 19.
- [ ] **Модульный монолит:** монолит с явными границами модулей и контрактами между ними; подготовка к разрезу на сервисы. → часть 5.
- [ ] **Гексагональная (порты и адаптеры):** ядро не зависит от фреймворков и БД; входящие и исходящие адаптеры. → часть 6.
- [ ] **Clean Architecture / Onion:** слои сущностей, use cases, адаптеров, инфраструктуры; зависимости внутрь. → часть 7.
- [ ] **Вертикальные срезы (vertical slices):** организация по фичам/сценариям (от API до БД), а не по слоям; каждая фича — срез. → части 5, 33.
- [ ] **Плагинная архитектура:** расширяемость через плагины и точки расширения; загрузка модулей динамически. → части 2 (extensibility), 34.
- [ ] **SOA (Service-Oriented Architecture):** сервисы как единицы; часто ESB, более крупные границы, чем микросервисы; отличия от микросервисной архитектуры. → часть 8.
- [ ] **Микросервисы:** независимо развёртываемые сервисы с собственными данными; границы по домену/команде; API Gateway, Service mesh, database per service. → часть 9.
- [ ] **Cell-based / клеточная архитектура:** изоляция по «ячейкам» (полные копии стека на группу пользователей/регион); масштабирование и изоляция сбоев. → часть 10.
- [ ] **Actor model (акторная модель):** акторы как единицы конкуренции и состояния (Akka, Orleans, Erlang); сообщения асинхронные; когда применять. → часть 11.
- [ ] **Синхронная коммуникация (запрос-ответ):** HTTP, gRPC; связь по доступности и латентности. → части 9, 15.
- [ ] **Асинхронная коммуникация (события, очереди):** шины событий, очереди сообщений; развязка по времени и доступности. → части 9, 12.
- [ ] **Оркестрация vs хореография:** центральный оркестратор vs сервисы реагируют на события; сага. → части 9, 12.
- [ ] **Event-Driven Architecture (EDA):** события как первый класс; производители и потребители развязаны; message-driven, Pipe-and-Filter. → часть 12.
- [ ] **Message-driven:** обмен сообщениями (запросы/команды через очередь), не только события; отличия от чистого EDA. → часть 12.
- [ ] **Pipe-and-Filter (конвейер):** данные проходят через цепочку обработчиков; потоковая обработка, ETL. → часть 12, 34.
- [ ] **CQRS:** разделение моделей и путей чтения и записи; проекции под запросы. → часть 13.
- [ ] **Event Sourcing:** хранение потока событий как источника истины; реплеи и снапшоты. → часть 14.
- [ ] **REST API:** ресурсы, HTTP-методы, идемпотентность; универсальность и кэширование. → часть 15.
- [ ] **GraphQL:** схема, запросы/мутации/подписки; гибкость запросов клиента; Federation. → часть 16.
- [ ] **GraphQL Federation:** несколько GraphQL-схем от разных сервисов, объединённые шлюзом. → часть 16.
- [ ] **gRPC:** контракты (protobuf), бинарный протокол, стриминг; внутренние сервисы. → часть 17.
- [ ] **Real-time и streaming API:** WebSockets, SSE (Server-Sent Events), потоковая отдача в REST, GraphQL subscriptions, gRPC streaming; когда какой способ; масштабирование и липкость. → части 12, 15, 16, 17.
- [ ] **tRPC и типобезопасные API:** end-to-end типы без кодогенерации; когда уместно. → часть 17.
- [ ] **BaaS (Backend as a Service):** Firebase, Supabase и аналоги; готовые auth, БД, хранилище; ограничения. → части 17, 30, 34.
- [ ] **API Gateway:** единая точка входа; маршрутизация, агрегация, аутентификация, rate limiting. → часть 9.
- [ ] **BFF (Backend for Frontend):** слой под тип клиента (веб, мобильный); агрегация и адаптация API. → часть 30.
- [ ] **Database per service / владение данными:** каждый сервис владеет своими данными; доступ других только через API/события. → часть 9.
- [ ] **Shared database (временный вариант):** общая БД при миграции; риски и план перехода к database per service. → часть 9.
- [ ] **Репликация (master-replica, кворумы):** масштабирование чтения и отказоустойчивость. → часть 18.
- [ ] **Шардирование и партиционирование:** разделение данных по ключу; решардинг. → часть 18.
- [ ] **Space-based (in-memory data grid):** данные в распределённой памяти (tuple space, кэш-сетка); снятие нагрузки с БД, сессии. → части 18, 19, 34.
- [ ] **Полиглот персистентность:** разные хранилища под разные задачи (OLTP, OLAP, кэш, поиск, события). → часть 18.
- [ ] **FaaS / Serverless:** функции по событию или HTTP; автомасштабирование до нуля; ограничения. → части 19, 34.
- [ ] **Контейнеры и оркестрация (Kubernetes и др.):** единица развёртывания; масштабирование, пробы, стратегии деплоя. → часть 20.
- [ ] **Service mesh:** управление трафиком между сервисами, mTLS, retry/circuit breaker на уровне сети. → часть 9.
- [ ] **Multi-tenancy (мультитенантность):** общая БД с tenant_id, схема на тенанта, БД на тенанта; изоляция и безопасность. → части 18, 31, 34.
- [ ] **Геораспределение:** активный-активный, активный-пассивный; репликация между регионами; консистентность и латентность. → часть 18, «что изучать дальше».

### Фронтенд: архитектуры и способы организации

- [ ] **MPA (Multi-Page Application):** каждая страница — отдельный запрос; полная или частичная перезагрузка (Turbo, HTMX). → часть 21.
- [ ] **SPA (Single-Page Application):** один HTML, роутинг на клиенте; богатая интерактивность. → часть 22.
- [ ] **SSR (Server-Side Rendering):** HTML на сервере при запросе; гидрация на клиенте. → часть 23.
- [ ] **SSG (Static Site Generation):** HTML на этапе сборки; раздача статики. → часть 23.
- [ ] **ISR (Incremental Static Regeneration):** статика с фоновым обновлением. → часть 23.
- [ ] **Островная архитектура (Islands):** в основном статика; интерактивные «островки» гидратируются выборочно (Astro и др.). → часть 24.
- [ ] **Гидрация:** full, partial, progressive, streaming; влияние на TTI и размер JS. → часть 24.
- [ ] **Resumability (возобновляемость):** отсутствие повторного выполнения на клиенте (Qwik и др.); отличия от классической гидрации. → часть 24, 34.
- [ ] **PWA (Progressive Web App):** сервис-воркер, офлайн, установка; архитектура кэширования и обновлений. → часть 24, 34.
- [ ] **TWA (Trusted Web Activity):** PWA в оболочке для публикации в сторах. → часть 24.
- [ ] **JAMstack:** JavaScript, API, Markup; статика + API + клиентская интерактивность. → часть 24, 34.
- [ ] **Компонентная архитектура:** атомы, молекулы, организмы; композиция. → часть 25.
- [ ] **Atomic Design:** иерархия атомы → молекулы → организмы → шаблоны → страницы; когда избыточен. → часть 25.
- [ ] **Контейнеры и презентационные компоненты:** разделение по ответственности (данные/логика vs отображение). → часть 25.
- [ ] **Feature-Sliced Design (FSD):** слои (app, processes, pages, widgets, features, entities, shared); границы и публичный API слоёв. → часть 25.
- [ ] **Вертикальные срезы / folder-by-feature:** структура папок по фичам, а не по типам; колокация всего, что относится к фиче. → часть 25.
- [ ] **MVC на фронтенде:** Model, View, Controller; классическое разделение; где встречается. → часть 25.
- [ ] **MVVM (Model-View-ViewModel):** ViewModel как посредник; биндинги; типично в Angular, Vue. → часть 25.
- [ ] **MVI (Model-View-Intent):** односторонний поток: Intent → Model → View; предсказуемость состояния. → часть 25.
- [ ] **Flux / односторонний поток данных:** действия → диспетчер → store → представление; Redux и аналоги. → части 25, 26.
- [ ] **Микрофронтенды:** независимые фрагменты от разных команд; shell/хост, способы интеграции (iframe, JS-bundle, Web Components, Module Federation). → часть 28.
- [ ] **Монорепо:** несколько приложений/пакетов в одном репозитории; общие зависимости и версии (Nx, Turborepo, Lerna). → часть 29.
- [ ] **Module Federation:** загрузка удалённых модулей в runtime; совместное использование и изоляция версий. → часть 29.
- [ ] **Монолит фронтенд vs микрофронтенды:** один репозиторий и один деплой vs несколько команд и независимый релиз фрагментов. → части 28, 29.
- [ ] **BFF под клиент:** один BFF на все клиенты vs отдельный BFF под веб/мобильный/десктоп. → часть 30.
- [ ] **Edge rendering / edge functions:** рендер или логика на границе сети (CDN); низкая латентность. → части 23, 30, 34.
- [ ] **Server Components (React и др.):** компоненты только на сервере; нулевой JS на клиенте для них. → часть 34.
- [ ] **Offline-first:** локальное хранилище, синхронизация при появлении сети; конфликты. → часть 34.

Использование карты: каждая структура имеет свою часть в плане; при создании контента каждая часть раскрывает одну структуру (или одну группу тесно связанных подструктур) подробно. При затруднениях — отнести к «углублению» или «что изучать дальше».

## Часть 0. Введение: что такое архитектура ПО

Материалы: [→ Навигация по части 0](pact/00_vvedenie_chto_takoe_arkhitektura_po/index.md)

> Зачем нужна явная архитектура, как она соотносится с кодом и инфраструктурой, какие ментальные модели использовать.

- [ ] **Цель части 0:** сформировать рабочее определение архитектуры ПО, границы темы и ментальные модели (решения с последствиями, эволюция, границы и контракты) для осмысленного изучения частей 1–35.
- [ ] **Связь с другими частями:** введение ко всему плану; без этой части контекст частей 1–35 теряется. Прямо готовит к части 1 (терминология) и части 2 (цели и ограничения).

### 0.1 Определение архитектуры ПО

- [ ] **Определения:** набор значимых решений о структуре системы, компонентах, способах взаимодействия и обосновании этих решений; «то, что дорого менять потом».
- [ ] Примеры значимых решений: выбор типа приложения (монолит/микросервисы), границы компонентов, способ взаимодействия (синхронный/асинхронный), владение данными, стратегия консистентности.
- [ ] Архитектура vs «про код»: архитектура затрагивает разрез по компонентам, границам, данным и их эволюцию; код реализует архитектуру.
- [ ] Кто принимает архитектурные решения: роль архитектора, лида, команды; коллегиальность и документирование.
- [ ] История и эволюция понятия: от «архитектура = структура кода» к явным границам, контрактам и операционным аспектам; влияние DDD, микросервисов, облака.
- [ ] Ограничения: что архитектура не гарантирует (корректность алгоритмов, отсутствие багов, скорость разработки без дисциплины).
- [ ] Связь с качеством доставки: как плохая архитектура проявляется в инцидентах, времени на фичи и страхе менять код; обратная связь «архитектура → метрики».
- [ ] *(В контенте: блок «Простыми словами» — одна суть для запоминания.)*
- [ ] *(В контенте: блок «Запомните» — 2–5 тезисов.)*

### 0.2 Зачем нужна явная архитектура

- [ ] Поддерживаемость: понятная структура упрощает изменения и онбординг; локализация изменений по границам; рефакторинг без «всего трогать»; в контенте раскрыть конкретные механизмы (слои, модули, контракты).
- [ ] Масштабируемость: возможность масштабировать части системы и данные независимо; горизонтальное vs вертикальное.
- [ ] Предсказуемость: последствия изменений локализованы; меньше сюрпризов при деплое; тестируемость по границам.
- [ ] Стоимость владения: плохая архитектура увеличивает время и риск изменений; кривая накопления долга.
- [ ] Связь с бизнес-целями: время вывода фич, стабильность, стоимость разработки; архитектурный runway (задел для будущих фич).
- [ ] Альтернатива «без явной архитектуры»: когда допустимо (прототип, одноразовый скрипт) и когда опасно (долгоживущий продукт, команда).
- [ ] Баланс «достаточно архитектуры»: слишком мало — долг и хаос; слишком много — over-engineering и замедление; признаки обоих крайностей; контекст как критерий.

### 0.3 Границы темы

- [ ] Архитектура vs код: архитектура — «что и как связано»; код — реализация.
- [ ] Архитектура vs инфраструктура: инфраструктура обеспечивает запуск и сеть; архитектура решает, что где запускать и как общаться.
- [ ] Архитектура vs процессы: Agile, DevOps влияют на то, как архитектура эволюционирует; не подменяют архитектурные решения.
- [ ] Пересечения: качество кода (чистый код, тесты) поддерживает архитектуру; безопасность и производительность — архитектурные ограничения.
- [ ] Смежные дисциплины: качество кода, тестирование, продукт-менеджмент, DevOps — что решается в архитектуре, что смежно; границы ответственности.

### 0.4 Ментальные модели

- [ ] **Архитектура как набор решений с последствиями:** каждое решение имеет плюсы, минусы и стоимость смены.
- [ ] **Эволюция вместо чертежа:** архитектура меняется; важна способность к изменению, а не «идеальная картинка раз».
- [ ] **Границы и контракты:** чёткие границы снижают связанность; контракты позволяют эволюционировать части независимо.
- [ ] Типичные заблуждения: «архитектура нужна только большим системам»; «сначала сделаем идеальную архитектуру» — разбор и контраргументы.
- [ ] *(В контенте: блок «Проверь себя (0)» — 1–3 вопроса.)*
- [ ] *(В контенте: блок «Картинка в голове» — аналогия: архитектура как план этажей с коммуникациями.)*
- [ ] **Практика после части 0:** сформулировать одно значимое архитектурное решение в знакомом или учебном проекте и перечислить его последствия (плюсы, минусы, стоимость смены).

## Часть 1. Терминология и базовые примитивы

Материалы: [→ Навигация по части 1](pact/01_terminologiya_i_bazovye_primitivy/index.md)

> Общий язык: компонент, модуль, слой, контракт, граница, связность, связанность — чтобы однозначно обсуждать архитектуру.

- [ ] **Цель части 1:** зафиксировать единые определения компонента, модуля, слоя, сервиса, контракта, границы, связности и связанности, а также базовые сущности (API, событие, очередь, хранилище) для однозначного обсуждения частей 3–35.
- [ ] **Связь с другими частями:** опирается на часть 0; даёт язык для частей 3–22. Без единой терминологии легко путать слои, модули и сервисы.

### 1.1 Компонент, модуль, слой, сервис

- [ ] **Компонент:** логически выделяемая часть системы с ответственностью и границей; может быть классом, пакетом, процессом, сервисом.
- [ ] **Модуль:** единица развёртывания или компиляции; пакет, библиотека, микросервис; имеет публичный контракт и скрытую реализацию.
- [ ] **Слой:** горизонтальный срез архитектуры (презентация, бизнес-логика, данные); правила: кто кого может вызывать.
- [ ] **Сервис:** компонент, предоставляющий возможности по сети (API); в микросервисной архитектуре — основная единица развёртывания.
- [ ] Различие по контексту: в монолите «модуль» часто = пакет/namespace; в микросервисах «сервис» = отдельный процесс/контейнер.
- [ ] Путаница терминов: «компонент» в UML (физическая единица развёртывания) vs в архитектуре ПО (логическая часть с границей); в контенте при необходимости пояснить, чтобы избежать разночтений в команде.

### 1.2 Контракт и граница

- [ ] **Контракт:** явное описание взаимодействия (формат запроса/ответа, события, гарантии); API, схема сообщения, IDL.
- [ ] Формальность контракта: от неформального («договорились в чате») до формального (OpenAPI, protobuf, схема событий); риски неформальных контрактов при росте команды и времени.
- [ ] **Граница:** граница компонента/модуля/сервиса; пересекать только через контракт; за границей — скрытая реализация.
- [ ] Зачем явные границы: независимая эволюция, замена реализации, тестирование через подмену.
- [ ] Типичные ошибки: неявные контракты (например, общая БД как «контракт»); утечка внутренней структуры через API.

### 1.3 Связность и связанность

- [ ] **Связность (cohesion):** насколько элементы внутри компонента относятся к одной задаче; высокая связность — хорошо.
- [ ] **Связанность (coupling):** насколько компоненты зависят друг от друга; низкая связанность — хорошо.
- [ ] Направление зависимостей: зависимость от абстракций, инверсия (внутрь к домену/ядру).
- [ ] Граф зависимостей: ацикличность; модули верхнего уровня не должны зависеть от деталей нижних.
- [ ] *(В контенте: блок «Картинка в голове» — граф узлов и стрелок; циклы = запах.)*
- [ ] *(В контенте: блок «Проверь себя (1)» — отличить компонент от слоя; назвать пример высокой связности и низкой связанности.)*

### 1.4 Базовые сущности архитектуры

- [ ] **API (запрос-ответ):** синхронное взаимодействие через контракт; REST, gRPC, GraphQL; идемпотентность и безопасность методов.
- [ ] **Событие:** асинхронное сообщение о факте; топик, очередь; доставка «at least once» / «exactly once» — последствия для потребителей.
- [ ] **Очередь:** буфер между производителем и потребителем; сглаживание нагрузки, развязка по времени; гарантии доставки.
- [ ] **Хранилище:** персистентность данных; выбор типа (реляционная, документная, кэш) — часть архитектуры; источник истины (source of truth) для доменных данных.
- [ ] **Интерфейс vs реализация:** контракт как интерфейс; смена реализации за границей без изменения потребителей.
- [ ] Инструменты для анализа графа зависимостей: линтеры модулей (ArchUnit, jqassistant, dependency-cruiser); визуализация циклов и нарушений правил.
- [ ] *(В контенте: блок «Запомните» — 3–5 тезисов по терминам.)*
- [ ] **Практика после части 1:** для одного модуля или сервиса назвать его границу, контракт и один пример высокой связности или низкой связанности.

## Часть 2. Цели и ограничения архитектуры

Материалы: [→ Навигация по части 2](pact/02_tseli_i_ogranicheniya_arkhitektury/index.md)

> Чего мы хотим достичь архитектурой и чем ограничены; trade-offs и отсутствие серебряной пули.

- [ ] **Цель части 2:** сформулировать цели архитектуры (масштабируемость, поддерживаемость, тестируемость, безопасность, стоимость, расширяемость), ограничения (команда, сроки, технологии, регуляторика) и критерии «хорошей» архитектуры; ввести trade-offs и признаки архитектурного долга для оценки решений в частях 3–35.
- [ ] **Связь с другими частями:** опирается на части 0–1; задаёт критерии для оценки решений в частях 3–35.

### 2.1 Цели архитектуры

- [ ] **Масштабируемость:** возможность увеличивать нагрузку (горизонтально/вертикально) без переписывания.
- [ ] **Поддерживаемость:** понятная структура, локализация изменений, тестируемость.
- [ ] **Тестируемость как цель:** тестируемость по границам (изоляция модулей/сервисов); подмена зависимостей (mock, stub); возможность интеграционных и контрактных тестов без «всей системы».
- [ ] **Надёжность и отказоустойчивость:** деградация вместо полного падения; повторные попытки, изоляция сбоев.
- [ ] **Безопасность:** границы доверия, минимизация поверхности атаки, защита данных в покое и при передаче.
- [ ] **Стоимость владения:** разработка, эксплуатация, инфраструктура; баланс «сделать правильно» и «сделать вовремя».
- [ ] **Расширяемость (extensibility):** возможность добавлять фичи и интеграции без переписывания ядра; точки расширения, плагины; связь с открытостью/закрытостью (OCP).
- [ ] Конфликты целей: например, низкая связанность vs простота; консистентность vs доступность (CAP); явность контрактов vs скорость разработки.

### 2.2 Ограничения

- [ ] Команда: размер (одна vs несколько команд — влияние на границы); навыки (доступность технологий и практик); распределённость (явные контракты и артефакты при распределённой команде); архитектура должна быть реализуема и поддерживаема командой.
- [ ] Сроки и бюджет: не всегда возможно «идеально»; приемлемый уровень архитектурного долга.
- [ ] Технологии: существующий стек, лицензии, экосистема; legacy ограничивает выбор; технический долг как ограничение на скорость и направление изменений.
- [ ] Регуляторика и compliance: хранение данных, аудит, географические ограничения.
- [ ] Организационные ограничения: матричная структура, аутсорс, распределённые команды — влияние на границы сервисов/модулей и выбор контрактов.
- [ ] *(В контенте: блок «Простыми словами» — цели и ограничения в одном абзаце.)*

### 2.3 Trade-offs и «нет серебряной пули»

- [ ] Любой архитектурный выбор — компромисс; важно явно фиксировать обоснование (ADR).
- [ ] Примеры trade-offs: монолит vs микросервисы (простота vs независимое масштабирование); сильная консистентность vs доступность (CAP); синхронный vs асинхронный API; тонкий vs толстый BFF; общая БД vs владение данными сервисом.
- [ ] Cost of delay: когда «сделать проще сейчас» оборачивается дороже потом; признаки и как оценивать.
- [ ] **Стоимость архитектуры (минимальный срез):** порядок стоимости внедрения и поддержки — монолит/модульный монолит (ниже на старте, один деплой, проще отладка); микросервисы (выше: операционная сложность, контракты, мониторинг, несколько деплоев); событийная архитектура (шина, мониторинг lag, идемпотентность); когда какая архитектура сколько стоит во внедрении и поддержке; учитывать в критериях выбора (часть 33) и в trade-offs.
- [ ] Критерии «хорошей» архитектуры: соответствует целям и ограничениям; допускает эволюцию; документирована; тестируема по границам.
- [ ] Измеримость целей: как проверить достижение целей (метрики латентности, доступности, времени релиза; ревью границ и контрактов); неформальные критерии vs метрики.
- [ ] Признаки архитектурного долга: циклы зависимостей, «всё знает обо всём», невозможность изолированного теста, частые каскадные изменения, страх менять части системы.
- [ ] Граничные случаи: «идеальная» архитектура недостижима; приемлемый уровень долга в зависимости от фазы продукта.
- [ ] *(В контенте: блок «Проверь себя (2)» — назвать один trade-off для монолита и один для микросервисов.)*
- [ ] **Типичные ошибки:** выбирать архитектуру по моде, а не по контексту; игнорировать ограничения команды и сроков; откладывать фиксацию решений (ADR) «на потом».
- [ ] **Практика после части 2:** для своего или учебного проекта записать одну цель архитектуры, одно ограничение и один trade-off; проверить измеримость цели (как узнаем, что достигнута).

## Часть 3. Бекенд: монолит

Материалы: [→ Навигация по части 3](pact/03_backend_monolit/index.md)

> Классический монолит; 2-tier, 3-tier, 4-tier; когда уместен, плюсы, минусы и ограничения.

- [ ] **Цель части 3:** дать определение монолита и вариантов 2/3/4-tier, плюсы и минусы, когда уместен, типичные ошибки и направление эволюции (к слоям и модульности) как базу для частей 4–7.
- [ ] **Связь с другими частями:** опирается на части 0–2; база для частей 4–7 (слои, модульность, гексагональная, Clean). Каждая архитектура рассматривается отдельным шагом — монолит как первый способ организации бекенда.

### 3.1 Определение и варианты (2-, 3-, 4-tier)

- [ ] **Определение:** одно развёртываемое приложение; все компоненты в одном процессе (или тесно связанных процессах); одна база данных (часто).
- [ ] **2-tier (двухзвенка):** клиент + сервер (БД и приложение на сервере или «толстый» клиент с прямым доступом к БД); когда уместна и ограничения.
- [ ] **3-tier (трёхзвенка):** клиент, сервер приложений, СУБД — классическое разделение; тонкий клиент и «толстый» сервер приложений.
- [ ] **4-tier:** добавление явного слоя между бизнес-логикой и данными (инфраструктура: очереди, внешние API); когда вводить.
- [ ] Плюсы: простота развёртывания, отладки, транзакций (ACID в рамках одного процесса); низкий порог входа; один артефакт деплоя.
- [ ] Минусы: масштабирование только «копией всего»; жёсткая связанность; один баг может положить всё; длительная сборка и тесты при росте.
- [ ] Когда монолит уместен: маленькая команда, один домен, умеренная нагрузка, быстрый старт продукта; отсутствие потребности в независимом масштабировании частей.
- [ ] Транзакции в монолите: единая БД — возможность локальных транзакций; границы транзакций и их влияние на слои.
- [ ] Ограничения: потолок масштабирования; сложность координации при росте команды; риск «большого шарика грязи» без дисциплины слоёв.
- [ ] Типичная эволюция монолита: от «всё в одном» к слоям (N-tier), затем к модульности; порядок шагов при рефакторинге.
- [ ] Типичные ошибки монолита: монолит без слоёв (всё в одном); размытые границы — первый шаг введение слоёв (см. часть 4).
- [ ] *(В контенте: блок «Картинка в голове» — один большой блок с подписью «приложение + БД».)*
- [ ] **Практика после части 3:** для одного знакомого приложения назвать вариант (2/3/4-tier или «без явных слоёв»); перечислить один плюс и одно ограничение монолита.

## Часть 4. Бекенд: слоёная (N-tier) архитектура

Материалы: [→ Навигация по части 4](pact/04_backend_sloistaya_arkhitektura/index.md)

> Разделение на слои; правило зависимостей «только вниз»; типичные ошибки слоёв.

- [ ] **Цель части 4:** зафиксировать слои (презентация, бизнес-логика, данные, при необходимости инфраструктура), правило зависимостей «только вниз» и типичные ошибки/антипаттерны (толстый контроллер, логика в слое данных, пропуск слоя, шарболл) как базу для частей 5–7.
- [ ] **Связь с другими частями:** опирается на часть 3 (монолит); база для частей 5–7 (модульность, гексагональная, Clean). Слоёная архитектура — отдельный шаг для детального разбора правил и ошибок.

### 4.1 Слои и правила

- [ ] **Слои:** презентационный (API, контроллеры), бизнес-логика (сервисы, домен), слой доступа к данным (репозитории, DAO); иногда отдельно — инфраструктура (внешние вызовы, очереди).
- [ ] Четыре слоя vs три: когда вводить отдельный слой инфраструктуры (внешние API, очереди, почта) между бизнес-логикой и данными; изоляция внешних вызовов от домена.
- [ ] **Правило:** зависимости только «вниз»; верхний слой вызывает нижний, не наоборот. Презентация → бизнес-логика → данные.
- [ ] Модули внутри слоёв: слой может состоять из нескольких модулей (по фичам или поддоменам); зависимости между модулями одного слоя — осторожно, без циклов.
- [ ] Зачем: изоляция бизнес-логики от способа вызова (HTTP, очередь) и от способа хранения (БД, файл); замена адаптеров без смены ядра.
- [ ] Трёхзвенка (3-tier): клиент, сервер приложений, БД; эволюция к тонкому клиенту и «толстому» серверу.
- [ ] Реализация по стекам: как слои воплощаются в Java (.NET, Node, Go и др.): пакеты, аннотации, конвенции; типичные фреймворки по слоям.
- [ ] *(В контенте: блок «Запомните» — направление зависимостей; не путать слой и модуль.)*

### 4.2 Типичные ошибки и антипаттерны (слои)

- [ ] **Толстый контроллер:** бизнес-логика в контроллере; решение — перенос в сервисный слой.
- [ ] **Бизнес-логика в слое данных:** сложная логика в SQL или в репозитории; решение — доменная логика в отдельном слое.
- [ ] **Циклические зависимости между слоями:** слой данных обращается к сервису; решение — строгие правила и линтеры зависимостей.
- [ ] **Пропуск слоя:** контроллер напрямую вызывает репозиторий; решение — явные сервисы.
- [ ] **Антипаттерн «шарболл» (big ball of mud):** границы слоёв размыты; всё связано со всем.
- [ ] **Когда использовать слои:** практически всегда в монолите как минимум; основа для дальнейшего разбиения на модули/порты.
- [ ] **Где в слоёной архитектуре живут тесты:** unit — бизнес-логика (сервисный слой), изоляция от контроллеров и репозиториев; интеграционные — на границах слоёв (контроллеры с подменёнными сервисами, репозитории с тестовой БД); направление зависимостей «вниз» позволяет подменять нижние слои при тестах верхних.
- [ ] *(В контенте: блок «Проверь себя (4)» — на схеме из трёх слоёв указать, где ошибка — стрелка снизу вверх.)*
- [ ] **Практика после части 4:** нарисовать диаграмму слоёв (3–4 слоя) для одного приложения; указать направление зависимостей и один пример нарушения (стрелка снизу вверх).

## Часть 5. Бекенд: модульный монолит

Материалы: [→ Навигация по части 5](pact/05_backend_modulnyy_monolit/index.md)

> Чёткие границы модулей внутри монолита; подготовка к возможному «разрезу» на сервисы.

- [ ] **Цель части 5:** ввести границы модулей, публичную поверхность, запрет циклов и организацию кода по модулям; подготовка к «разрезу» на сервисы и когда модульного монолита достаточно; типичные ошибки и практика (часть 9 — микросервисы).
- [ ] **Связь с другими частями:** опирается на части 3–4 (монолит, слои); готовит к части 8 (микросервисы). Модульный монолит — отдельная структура с собственными правилами границ.

### 5.1 Границы модулей

- [ ] **Модуль:** пакет/namespace/библиотека с явной публичной поверхностью (API); внутренняя реализация скрыта.
- [ ] Публичная поверхность модуля: минимальный API; что не должно «просачиваться» наружу (внутренние типы, детали реализации, зависимости от фреймворка); принцип сокрытия информации.
- [ ] Граница = контракт (интерфейсы, DTO, события); зависимости между модулями только через публичный контракт.
- [ ] Запрет циклических зависимостей: граф модулей должен быть ациклическим (DAG); как обнаруживать и устранять циклы.
- [ ] Общие типы и контракты: вынести в отдельный модуль «контракты» или «shared kernel»; осторожно с ростом общего ядра (shared kernel как антипаттерн при раздувании).
- [ ] Вертикальные срезы (фичи) vs горизонтальные (слои): модуль может быть вертикальным (фича от API до БД) или горизонтальным (слой домена); комбинации и когда что предпочесть.

### 5.2 Организация кода по модулям

- [ ] Разбиение по доменным областям (bounded context lite) или по фичам; единый стиль в рамках одного проекта.
- [ ] Именование модулей: по домену (order, payment) vs по типу (services, repositories); избегать анемичных названий, не отражающих ответственность.
- [ ] Зависимости: модуль верхнего уровня (оркестратор, API) зависит от модулей домена; домен не зависит от API и инфраструктуры.
- [ ] Общие библиотеки: утилиты, логирование — осторожно, чтобы не создавать «бога»-модуль, от которого все зависят.
- [ ] Инструменты: линтеры зависимостей (например, jqassistant, ArchUnit, dependency-cruiser) для запрета циклов и нарушения правил.
- [ ] **Где живут тесты в модульном монолите:** unit — внутри модуля (ядро модуля, логика); интеграционные — на контрактах между модулями (вызов публичного API модуля с подменой зависимостей); запрет циклов упрощает изоляцию и тестирование модулей по отдельности.

### 5.3 Подготовка к «разрезу» на сервисы

- [ ] Модульный монолит с явными границами и контрактами легче «разрезать» на микросервисы: модуль → сервис; контракт модуля становится API или событиями.
- [ ] Важно: не делать общую БД между будущими сервисами «по привычке»; уже в монолите думать о границах данных (кто владеет какими данными).
- [ ] Граничные случаи при разрезе: общие транзакции (как заменить сагой или событиями); общие типы (дублирование vs общая библиотека); порядок разреза (зависимости между модулями).
- [ ] Когда модульного монолита достаточно: одна команда, один домен, масштаб не требует независимого масштабирования частей; не усложнять раньше времени.
- [ ] *(В контенте: блок «Простыми словами» — модульный монолит как «микросервисы в одном процессе» с чёткими перегородками.)*
- [ ] **Типичные ошибки:** модули-«боги», циклы через «общий» модуль; отсутствие явного контракта между модулями; общая БД без границ владения данными.
- [ ] **Практика после части 5:** для одного модуля описать публичную поверхность (что можно вызывать снаружи); проверить отсутствие циклов в графе зависимостей (на бумаге или линтером).

## Часть 6. Бекенд: гексагональная архитектура (порты и адаптеры)

Материалы: [→ Навигация по части 6](pact/06_backend_geksagonalnaya_arkhitektura/index.md)

> Ядро приложения не зависит от фреймворков, БД и способов вызова; входящие и исходящие адаптеры.

- [ ] **Цель части 6:** ввести порты (входящие/исходящие) и адаптеры, правило зависимостей внутрь, тестируемость через подмену адаптеров; когда применять и типичные перегибы; связь с DDD (часть 7 — Clean Architecture).
- [ ] **Связь с другими частями:** опирается на части 3–5 (монолит, слои, модульность); развивает идею границ и инверсии зависимостей. Часть 7 (Clean Architecture) — более детализированная слоистая модель поверх той же идеи. Гексагональная архитектура — отдельный шаг для детального разбора.

### 6.1 Порты и адаптеры

- [ ] **Порт:** интерфейс (вход или выход); абстракция, которую определяет ядро.
- [ ] **Входящий адаптер:** реализует входящий порт; преобразует внешний вызов (HTTP, сообщение из очереди) в вызов приложения (use case). Примеры: REST-контроллер, обработчик сообщения из очереди, CLI-команда.
- [ ] **Исходящий адаптер:** реализует исходящий порт; преобразует вызов приложения (например, «сохранить») в обращение к БД, внешнему API, очереди. Примеры: репозиторий (PostgreSQL), HTTP-клиент к внешнему сервису, издатель в шину событий.
- [ ] Ядро (домен + прикладные сценарии) не зависит от адаптеров; адаптеры зависят от ядра. Зависимости направлены внутрь.
- [ ] *(В контенте: блок «Картинка в голове» — шестиугольник в центре, снаружи адаптеры: HTTP, Kafka, PostgreSQL и т.д.)*

### 6.2 Входящие и исходящие порты

- [ ] Входящие: «что приложение может сделать для внешнего мира» (обработать заказ, принять платёж); один use case — один порт (или группа операций).
- [ ] Исходящие: «что приложению нужно от внешнего мира» (сохранить заказ, отправить уведомление); репозитории, клиенты внешних сервисов, издатели событий.
- [ ] Тестирование: ядро тестируется с подменёнными (mock/fake) исходящими адаптерами; входящие адаптеры тестируются интеграционно.
- [ ] **Где живут тесты в гексагональной архитектуре:** unit — ядро (входящие и исходящие порты подменены mock/fake); интеграционные — по каждому адаптеру отдельно (входящий HTTP-контроллер, исходящий репозиторий к тестовой БД); сквозные — при необходимости от входящего адаптера до исходящего.
- [ ] Связь с DDD: домен в центре; порты — границы контекста; адаптеры — технические детали.

### 6.3 Когда применять и типичные перегибы

- [ ] Один порт — несколько адаптеров: один входящий порт может реализовываться HTTP-контроллером, обработчиком очереди, CLI; один исходящий — репозиторием в PostgreSQL и in-memory для тестов.
- [ ] Один внешний сервис — один или несколько адаптеров: один адаптер при едином формате вызова; несколько адаптеров при разных контекстах (разные форматы ответа, разные эндпоинты); когда дублировать адаптер, когда общий фасад.
- [ ] Тестирование: unit-тесты ядра с mock-адаптерами; интеграционные тесты входящих/исходящих адаптеров по отдельности; когда нужны сквозные тесты.
- [ ] Когда применять: нужна независимость от фреймворка и БД; высокая потребность в тестируемости; долгоживущий продукт с ожидаемой сменой технологий.
- [ ] Гексагональная в микросервисе: один сервис — один «шестиугольник»; границы между сервисами — контракты (API, события); внутренняя структура сервиса может следовать портам и адаптерам.
- [ ] Перегибы: слишком много мелких портов; дублирование DTO на каждой границе без необходимости; «гексагональная» только на бумаге, а в коде зависимости нарушены.
- [ ] Ограничения гексагональной: не задаёт внутреннюю структуру ядра (домен vs use case); для детализации — Clean Architecture или явное разделение внутри ядра.
- [ ] *(В контенте: блок «Проверь себя (5)» — нарисовать один входящий и один исходящий порт для сервиса «Заказы».)*
- [ ] **Запомните:** ядро не импортирует фреймворки и драйверы БД; все внешние вызовы идут через порты.
- [ ] **Практика после части 6:** для одного use case нарисовать один входящий порт и один исходящий; указать, какой адаптер реализует каждый (например, HTTP-контроллер, репозиторий PostgreSQL).

## Часть 7. Бекенд: Clean Architecture / Onion

Материалы: [→ Навигация по части 7](pact/07_backend_clean_architecture_i_onion/index.md)

> Слои: сущности, use cases, адаптеры, инфраструктура; правило зависимостей — зависимости направлены внутрь.

- [ ] **Цель части 7:** зафиксировать слои Clean Architecture (сущности, use cases, адаптеры, инфраструктура), роль use case как оркестратора, отличия от гексагональной; когда не применять и типичные перегибы; миграции схемы БД и антипаттерны.
- [ ] **Связь с другими частями:** опирается на части 5–6 (модульный монолит, гексагональная); более детальная модель, чем гексагональная. Используется в монолите и внутри микросервиса. Clean Architecture — отдельный шаг.

### 7.1 Слои Clean Architecture

- [ ] **Сущности (Entities):** доменные объекты и правила домена; самый внутренний слой; не зависят ни от чего; сущности vs value objects (иммутабельные значения) в домене.
- [ ] **Use cases (Application / Interactors):** сценарии применения; оркестрируют сущности и вызывают исходящие порты (репозитории, шлюзы).
- [ ] **Доменные сервисы (Domain Services):** бессостоятельная доменная логика, не привязанная к одной сущности; слой домена (между сущностями и use cases); когда выносить из сущностей.
- [ ] **Адаптеры (Interface Adapters):** преобразуют данные между внешним миром и use cases; контроллеры, презентеры, реализация репозиториев (вызовы к БД).
- [ ] **Инфраструктура (Frameworks & Drivers):** фреймворк, БД, HTTP-клиенты; реализация деталей, которые «снаружи».
- [ ] Правило: зависимости только внутрь. Внутренние слои не знают о внешних; внешние реализуют интерфейсы, объявленные внутренними (инверсия зависимостей).
- [ ] Тестируемость по слоям: unit-тесты сущностей и use cases с подменой портов (без БД и HTTP); интеграционные тесты адаптеров по отдельности; когда нужен сквозной тест.
- [ ] **Где в Clean Architecture живут тесты:** unit — сущности и use cases (исходящие порты подменены); интеграционные — входящие и исходящие адаптеры по отдельности (контроллер + мок use case, репозиторий + тестовая БД); контрактные на границе API — см. часть 30; сквозной тест — от входящего адаптера при необходимости.
- [ ] Когда не применять Clean: простой CRUD без сложного домена; прототип; команда без опыта — риск перегрузить абстракциями.

### 7.2 Use cases как оркестраторы

- [ ] Use case: один сценарий («оформить заказ», «отменить подписку»); получает входные данные, вызывает домен и исходящие порты, возвращает результат.
- [ ] Use case не содержит деталей HTTP или БД; только вызовы портов (интерфейсов). Реализация портов — в адаптерах.
- [ ] Один use case — одна транзакция (как правило): граница use case как граница консистентности; когда разбивать на несколько use cases (разные транзакции, разные контексты).
- [ ] Транзакции: граница транзакции обычно на адаптере (входящий или исходящий); use case остаётся «чистым» от транзакционной семантики при необходимости.

### 7.3 Отличия от гексагональной; когда применять

- [ ] Гексагональная не фиксирует внутренние слои (домен vs use case); Clean явно разделяет сущности и сценарии.
- [ ] Onion — та же идея с другим названием; акцент на «ядре» и кольцах снаружи.
- [ ] Когда применять: сложный домен; несколько способов вызова (API, очередь, CLI); ожидаемая замена БД или фреймворка.
- [ ] Типичные перегибы: избыточное количество слоёв для простого CRUD; копирование сущностей на каждой границе без выгоды; «Clean» только в одной части системы.
- [ ] *(В контенте: блок «Проверь себя (6)» — куда поместить новую операцию «экспорт в CSV»: слой и направление зависимостей.)*
- [ ] **Миграции схемы БД в архитектуре:** zero-downtime миграции (добавление колонки с default, backfill, переключение); расширяемая схема (новые поля без ломающих изменений); версионирование миграций и откат; разделённые БД в микросервисах — миграция только в рамках владельца данных.
- [ ] **Антипаттерны:** доменная логика в use case (должна быть в сущностях/доменных сервисах); use case, зависящий от конкретной БД.
- [ ] **Практика после части 7:** для новой операции (например, «экспорт в CSV») определить слой и направление зависимостей; проверить, не попала ли доменная логика в use case.

## Часть 8. Бекенд: SOA (Service-Oriented Architecture)

Материалы: [→ Навигация по части 8](pact/08_backend_soa_service_orientirovannaya_arkhitektura/index.md)

> Сервисы как единицы; ESB; более крупные границы, чем у микросервисов; когда выбирать SOA.

- [ ] **Связь с другими частями:** опирается на части 3–7 (монолит, слои, модульность); предшествует части 9 (микросервисы). SOA — отдельная тема для детального разбора: границы сервисов, ESB, отличия от микросервисной архитектуры.

### 8.1 Суть SOA

- [ ] **Сервисы как единицы развёртывания и владения:** крупные сервисы с чёткими контрактами; бизнес-функции как сервисы; размер сервиса в SOA vs микросервисы.
- [ ] **ESB (Enterprise Service Bus):** центральная шина для маршрутизации, трансформации, оркестрации; плюсы (единая точка интеграции, централизованный мониторинг) и минусы (узкое горло, связность с шиной, единая точка отказа).
- [ ] **Паттерны интеграции через ESB:** медиатор, маршрутизация по контенту, трансформация сообщений; когда какой; ограничения латентности.
- [ ] **Отличия от микросервисов:** SOA — более крупные границы, часто общая инфраструктура (ESB); микросервисы — мелкие сервисы, умная периферия, независимый деплой, database per service; таблица сравнения в контенте.
- [ ] **Когда выбирать SOA:** крупные организации, интеграция множества legacy-систем, централизованное управление контрактами; регуляторика и единый аудит; когда переходить к микросервисам.
- [ ] **Эволюция SOA → микросервисы:** пошаговый отказ от ESB; выделение сервисов из шины; контракты и обратная совместимость при переходе.
- [ ] **Инструменты и экосистема:** типичные ESB-продукты (Mule, WSO2, коммерческие); альтернатива «ESB-less» интеграция; когда ESB избыточен.
- [ ] **Граничные случаи:** один сервис в SOA — по сути монолит; «SOA» только по названию без шины — что это значит на практике.
- [ ] **Типичные ошибки:** использовать ESB для всей бизнес-логики; не закладывать эволюцию к более мелким сервисам при росте команд; централизовать в шине то, что лучше оставить в сервисах.
- [ ] *(В контенте: блок «Проверь себя (8)» — сравнить SOA и микросервисы для заданного сценария; когда выбрать ESB.)*
- [ ] **Практика после части 8:** для одного знакомого или учебного контекста сформулировать, когда уместна SOA (ESB) и когда — переход к микросервисам; назвать один риск централизации в шине.

## Часть 9. Бекенд: микросервисы

Материалы: [→ Навигация по части 9](pact/09_backend_mikroservisy/index.md)

> Границы сервисов, синхронная и асинхронная коммуникация, контракты и версионирование; API Gateway, Service mesh, database per service; когда микросервисы оправданы.

- [ ] **Цель части 9:** дать границы сервисов (домен, команда, данные), database per service, синхронную и асинхронную коммуникацию, контракты и версионирование; сага и 2PC; когда микросервисы оправданы и операционная сложность; API Gateway, service mesh; типичные ошибки и антипаттерн «распределённый монолит».
- [ ] **Связь с другими частями:** опирается на части 3–7, 8 (SOA); пересекается с частями 12–14 (события, CQRS, ES), 15–17 (API), 18 (данные), 19 (масштабирование). Микросервисы — отдельная тема для детального разбора границ и коммуникации.

### 9.1 Границы сервисов

- [ ] **Сервис:** независимо развёртываемая единица с собственной кодовой базой и, как правило, собственным хранилищем данных (или явно разделённым владением данными).
- [ ] Обнаружение сервисов (service discovery): как клиент находит инстансы сервиса; DNS, Consul, Kubernetes Service; влияние на архитектуру вызовов и отказоустойчивость.
- [ ] Database per service: каждый сервис владеет своими данными; доступ других — только через API или события; не разделять одну таблицу между сервисами.
- [ ] Границы по домену (bounded context): один сервис — одна зона ответственности в домене; границы данных совпадают с границами сервиса.
- [ ] Границы по команде (Conway's law): сервисы часто соответствуют командам; проектирование с учётом организации; Team Topologies (stream-aligned, platform, и др.) и связь с границами сервисов.
- [ ] Размер сервиса: антипаттерн «nanoservice» (слишком мелкие сервисы — операционные и сетевые издержки); разумный минимум — возможность независимого деплоя и владения данными.
- [ ] Shared database как временный этап: допустим при миграции от монолита; план перехода к database per service; риски застревания в общей БД.
- [ ] Распределённый монолит как антипаттерн: сервисы жёстко связаны по данным (общая БД без границ) или по вызовам (каскадные синхронные вызовы, общая модель); меняется одна фича — трогают много сервисов.
- [ ] *(В контенте: блок «Картинка в голове» — граф сервисов как узлов; толстые стрелки и общая БД = запах.)*

### 9.2 Синхронная и асинхронная коммуникация

- [ ] Синхронная: HTTP/gRPC запрос-ответ; простота, но связь по доступности и латентности; каскадные сбои при падении одного сервиса.
- [ ] Асинхронная: очереди, шины событий; развязка по времени и доступности; eventual consistency, идемпотентность потребителей.
- [ ] Выбор: операция «сейчас и ответ нужен» — синхронно; уведомление, фоновые задачи, репликация — асинхронно.
- [ ] Сага и распределённые транзакции: компенсирующие операции vs 2PC; когда что применять; идемпотентность шагов саги.
- [ ] 2PC (two-phase commit): когда допустим (одна инфраструктура, короткие блокировки); ограничения в распределённых системах и почему предпочитают саги.
- [ ] Хореография vs оркестрация саги: события и локальные решения (хореография) vs центральный оркестратор (оркестрация); плюсы и минусы каждого.
- [ ] Типичные ошибки при саге: неидемпотентные шаги; отсутствие компенсирующих операций; неопределённый порядок при асинхронности.

### 9.3 Контракты и версионирование

- [ ] Контракт API: схема запроса/ответа (OpenAPI, protobuf, GraphQL schema); обратная совместимость при эволюции.
- [ ] Версионирование: в URL, в заголовке, в контенте; политика поддержки старых версий и deprecated.
- [ ] Контракт событий: схема сообщения; версионирование полей (добавление optional); обратная совместимость для потребителей.
- [ ] Контрактное тестирование: Consumer-Driven Contract (Pact и аналоги); проверка совместимости при сборке/деплое.

- [ ] **Где живут тесты в микросервисной архитектуре:** unit — внутри сервиса (ядро, use cases, порты подменены); интеграционные — на границе сервиса (API-адаптер, адаптер БД, подписчик на события); контрактные — между сервисами и между фронтом и бекендом (Pact, Spring Cloud Contract); E2E — на границе системы (один или несколько сервисов); тестируемость как критерий при выборе границ сервиса.

### 9.4 Когда микросервисы оправданы; операционная сложность

- [ ] Оправданы: разные команды/домены, независимое масштабирование частей, разные стеки/частота релизов; зрелая DevOps и наблюдаемость.
- [ ] Не оправданы: маленькая команда, один домен, нет потребности в независимом масштабировании; высокая операционная нагрузка без выгоды.
- [ ] Операционная сложность: деплой многих сервисов, мониторинг, отладка распределённых сценариев, сеть как источник сбоев; необходимость распределённой трассировки и единого лога/метрик.
- [ ] *(В контенте: блок «Проверь себя (9)» — назвать два признака распределённого монолита и два способа уменьшить связанность сервисов.)*
- [ ] **Service mesh (Istio, Linkerd и др.):** управление трафиком между сервисами, mTLS, retry/circuit breaker на уровне сети; когда уместен (много сервисов, единые политики) и когда избыточен.
- [ ] **API Gateway:** единая точка входа для внешних клиентов; маршрутизация, аутентификация, rate limiting, преобразование протоколов; отличие от BFF (BFF — под тип клиента, gateway — общая точка входа).
- [ ] **Типичные ошибки:** резать по слоям (отдельный «сервис БД» или «сервис контроллеров»); общая БД без владения данными; отсутствие идемпотентности при асинхронной обработке.

## Часть 10. Бекенд: Cell-based (клеточная) архитектура

Материалы: [→ Навигация по части 10](pact/10_backend_cell_based_kletochnaya_arkhitektura/index.md)

> Изоляция по «ячейкам» — полная копия стека на группу пользователей/регион/тенант; масштабирование и ограничение blast radius.

- [ ] **Связь с другими частями:** опирается на части 9 (микросервисы); пересекается с частями 18–19 (данные, масштабирование). Cell-based — отдельная тема для детального разбора изоляции и масштабирования по ячейкам.

### 10.1 Суть клеточной архитектуры

- [ ] **Ячейка (cell):** полная копия стека (или подмножества сервисов) на группу пользователей, регион или тенанта; трафик и данные изолированы внутри ячейки; размер ячейки и критерии разбиения.
- [ ] **Маршрутизация в ячейки:** как трафик попадает в нужную ячейку (по user_id, tenant_id, региону); стабильность маршрутизации при миграции пользователей.
- [ ] **Зачем:** ограничение blast radius при сбоях; изоляция по тенантам/регионам; предсказуемое масштабирование (добавление ячеек); соответствие data residency.
- [ ] **Когда уместна:** крупные мультитенантные системы, геораспределение, регуляторика по регионам; высокие требования к изоляции; необходимость изолировать «шумных» тенантов.
- [ ] **Ограничения:** операционная сложность (много копий стека); решардинг и перенос пользователей между ячейками; обновление всех ячеек (стратегии деплоя).
- [ ] **Граничные случаи:** общие компоненты (аутентификация, биллинг) — как размещать; кросс-ячеечные запросы — когда допустимы и как ограничивать.
- [ ] **Инструменты и практики:** как организуют ячейки в крупных продуктах (Amazon, Netflix cell-based); инструменты маршрутизации и изоляции.
- [ ] **Типичные ошибки:** слишком мелкие ячейки (операционные издержки); общие компоненты без изоляции сбоев; маршрутизация без учёта консистентности данных.
- [ ] *(В контенте: блок «Проверь себя (10)» — для сценария «мультитенант, 3 региона» предложить размер ячейки и стратегию маршрутизации.)*
- [ ] **Практика после части 10:** описать для учебного сценария одну ячейку (что входит, по какому ключу маршрутизация); назвать одно ограничение клеточной архитектуры.

## Часть 11. Бекенд: Actor model (акторная модель)

Материалы: [→ Навигация по части 11](pact/11_backend_actor_model_aktornaya_model/index.md)

> Акторы как единицы состояния и обработки сообщений; асинхронная передача сообщений; когда применять (высокая конкуренция, изоляция состояния).

- [ ] **Связь с другими частями:** опирается на части 9 (микросервисы, асинхронность); пересекается с частями 12 (события). Actor model — отдельная тема для детального разбора (Akka, Orleans, Erlang).

### 11.1 Суть акторной модели

- [ ] **Актор:** единица состояния и обработки сообщений; изолированный mailbox; один поток обработки на актора (нет гонок внутри актора); адрес и доставка сообщений.
- [ ] **Асинхронная передача сообщений:** акторы обмениваются сообщениями; развязка по времени и месту; отказоустойчивость через супервизоров и перезапуск; стратегии доставки (at-most-once, at-least-once).
- [ ] **Супервизия и иерархия:** дерево акторов; перезапуск при сбое; изоляция сбоя; типичные стратегии (restart, stop, escalate).
- [ ] **Когда применять:** высокая конкуренция за общее состояние; изоляция состояния по сущностям (один актор на сущность или агрегат); потоковая обработка, реактивные системы; чат-системы, игровые движки, телеком.
- [ ] **Когда не применять:** простой CRUD; команда без опыта; недостаток экосистемы под выбранный стек.
- [ ] **Отличия от «классических» микросервисов по HTTP:** гранулярность (акторы мельче сервисов), модель сообщений, локализация состояния; гибриды (акторы внутри сервиса).
- [ ] **Персистентность и кластеризация:** персистентные акторы (snapshot, event sourcing); распределённые акторы (кластер); консистентность и разделение при партиционировании.
- [ ] **Инструменты:** Akka (JVM), Akka Cluster, Orleans (.NET), Erlang/Elixir, Cloudstate; сравнение по модели (shared memory vs распределённые); типичные ошибки при выборе и внедрении.
- [ ] **Типичные ошибки:** акторы как «золотой молот» для всего; игнорирование персистентности при сбоях; слишком крупные акторы (потеря преимуществ изоляции).
- [ ] *(В контенте: блок «Проверь себя (11)» — для сценария «высокая конкуренция за счётчик» сравнить актор vs очередь vs БД.)*
- [ ] **Практика после части 11:** для одного сценария (например, «обработка заказов по типам») решить, уместна ли акторная модель; назвать один инструмент и одно ограничение.

## Часть 12. Бекенд: Event-Driven Architecture (EDA), сообщения, Pipe-and-Filter

Материалы: [→ Навигация по части 12](pact/12_backend_event_driven_architecture_eda_i_pipe_and_filter/index.md)

> События как первый класс; шины и топики; event-driven vs message-driven; Pipe-and-Filter; Outbox, ordering, идемпотентность.

- [ ] **Связь с другими частями:** опирается на части 9 (микросервисы, асинхронность), 1 (событие как сущность). Пересекается с частями 13–14 (CQRS, Event Sourcing), 18 (данные, проекции). EDA — отдельная тема для детального разбора событий и шин.

### 12.1 События и шины

- [ ] **Событие:** неизменяемое сообщение о факте, произошедшем в прошлом; имя в прошедшем времени (OrderPlaced, PaymentReceived).
- [ ] **Event-driven vs message-driven:** event-driven — уведомления о фактах, несколько потребителей, слабая связь; message-driven — команды/запросы через очередь, адресат может быть один; гибриды (команды в очередь, события в топик).
- [ ] **Pipe-and-Filter (конвейер):** данные проходят через цепочку обработчиков; потоковая обработка, ETL, пайплайны данных; когда применять и отличия от событийной модели; ограничения: не подходит при необходимости двусторонней связи или сложной маршрутизации по типу сообщения.
- [ ] Типы событий: доменное (внутри контекста) vs интеграционное (между контекстами); версионирование и обратная совместимость схемы.
- [ ] Эволюция схемы событий между сервисами: обратная совместимость (добавление полей, optional); несовместимые изменения — новая версия топика или новый тип события; конвертеры и адаптеры для старых потребителей.
- [ ] Шина событий (Kafka, RabbitMQ, AWS EventBridge и др.): топики/очереди, доставка, персистентность; at-least-once vs exactly-once семантика и идемпотентность потребителей.
- [ ] Ordering и партиционирование: порядок сообщений в пределах партиции (Kafka); ключ партиционирования и консистентность порядка для агрегата.
- [ ] Публикация: кто публикует, когда (после сохранения в БД или транзакционно); Outbox pattern для консистентности публикации и персистентности; транзакционный outbox vs polling outbox — реализация и ограничения.
- [ ] Типичные ошибки при публикации событий: публиковать до коммита транзакции (риск потери или неконсистентности); не использовать outbox при необходимости атомарности записи и публикации; публиковать «сырые» данные без доменного смысла.
- [ ] Подписчики: несколько потребителей одного события; независимая обработка; повторная обработка (replay) при необходимости; consumer lag и мониторинг отставания.
- [ ] **Граничные случаи:** один производитель — один потребитель (достаточна очередь, топик может быть избыточен); broadcast без персистентности (pub/sub в памяти) vs персистентная шина; событие как «команда» (антипаттерн — смешение намерения и факта).
- [ ] **Альтернативы EDA для простых сценариев:** синхронные вызовы при одном потребителе и жёсткой латентности; polling при нечастых обновлениях; когда не вводить шину событий.
- [ ] **Инструменты:** Kafka, RabbitMQ, AWS EventBridge, Azure Event Grid, Google Pub/Sub; сравнение по гарантиям доставки, ordering, персистентности; выбор под объём и латентность.
- [ ] **Real-time и push-доставка клиенту:** WebSockets, SSE (Server-Sent Events) как способы доставки событий/данных от сервера клиенту; отличия от очередей и топиков (брокер vs прямое соединение); когда real-time в архитектуре (уведомления, дашборды, чаты); масштабирование и липкость (sticky sessions), ограничения при горизонтальном масштабировании.

### 12.2 Резюме и типичные ошибки (план)

- [ ] В контенте: блок «Типичные ошибки (12)» и «Проверь себя (12)» — выбор event-driven vs message-driven под сценарий; проверка outbox и ordering.

## Часть 13. Бекенд: CQRS (Command Query Responsibility Segregation)

Материалы: [→ Навигация по части 13](pact/13_backend_cqrs_razdelenie_chteniya_i_zapisi/index.md)

> Разделение моделей чтения и записи; проекции и материализованные представления; eventual consistency; когда вводить CQRS.

- [ ] **Связь с другими частями:** опирается на части 12 (события), 9 (микросервисы). Пересекается с частью 14 (Event Sourcing), 18 (данные). CQRS — отдельная тема для детального разбора.

### 13.1 CQRS: модели и проекции

- [ ] Разделение моделей чтения и записи: команды меняют состояние; запросы читают; модели могут быть разными (нормализованная запись vs денормализованные проекции).
- [ ] Зачем: оптимизация чтения (проекции под конкретные экраны); масштабирование чтения отдельно от записи; сложные отчёты без нагрузки на запись.
- [ ] Проекции и материализованные представления: обновление проекций по событиям; eventual consistency между записью и чтением.
- [ ] Задержка проекций (lag): как измерять и мониторить; когда lag критичен (real-time дашборды) и когда допустим (отчёты).
- [ ] Обработка сбоев при обновлении проекций: повторная обработка событий, идемпотентность; восстановление после падения потребителя.
- [ ] Ограничения CQRS: сложность двух моделей; когда не вводить (простой CRUD, одна модель достаточна).
- [ ] Типичные ошибки при CQRS: игнорирование задержки проекций в UX (показать «сохранено» до обновления проекции); одна модель для записи и чтения «потому что проще»; отсутствие мониторинга lag.
- [ ] Когда не нужен: простой CRUD, одна модель достаточна; не усложнять без причины.
- [ ] *(В контенте: блок «Проверь себя (13)» — для сценария «лента заказов + отчёт по сумме» предложить команду/запрос и одну проекцию.)*
- [ ] **Практика после части 13:** описать одну команду и одну проекцию для учебного сценария; указать, как учитывать lag в UX.

## Часть 14. Бекенд: Event Sourcing

Материалы: [→ Навигация по части 14](pact/14_backend_event_sourcing_potok_sobytiy_kak_istochnik_istiny/index.md)

> Хранение потока событий как источника истины; реплеи и снапшоты; когда применять и ограничения.

- [ ] **Связь с другими частями:** опирается на части 12 (EDA), 13 (CQRS). Пересекается с частью 18 (данные, хранилища). Event Sourcing — отдельная тема для детального разбора.

### 14.1 Event Sourcing: суть и практика

- [ ] Хранение потока событий как источника истины; состояние = применение событий к пустому состоянию (replay).
- [ ] Снапшоты: периодическое сохранение состояния для ускорения восстановления; реплеи от снапшота.
- [ ] Версионирование событий: эволюция схемы событий; upcast при реплее старых событий.
- [ ] Когда применять: аудит, необходимость «переиграть» историю, сложная доменная логика с временной осью; ограничения — сложность, рост хранилища событий, квалификация команды.
- [ ] Ограничения по хранению: рост лога событий; стратегии (архивация, TTL, снапшоты); выбор хранилища (Kafka, EventStore, БД).
- [ ] Инструменты и платформы: EventStore, Axon Framework, Kafka как event log; когда что выбирать.
- [ ] *(В контенте: блок «Простыми словами» — event sourcing как «лента всех изменений вместо одной картинки сейчас».)*
- [ ] **Запомните:** CQRS не обязан использовать event sourcing; event sourcing часто сочетается с CQRS (события как источник для проекций).
- [ ] **Типичные ошибки:** хранить в событиях только «сырые» данные без смысла; игнорировать необходимость идемпотентности при обработке событий; смешивать команды и события (команда — намерение, событие — факт).
- [ ] *(В контенте: блок «Проверь себя (14)» — когда применить event sourcing, когда достаточно CQRS без ES.)*

## Часть 15. Бекенд: REST API

Материалы: [→ Навигация по части 15](pact/15_backend_rest_api_resursy_metody_versionirovanie/index.md)

### 15.1 REST: ресурсы и операции

- [ ] Ресурсы и идентификаторы (URI); HTTP-методы (GET, POST, PUT, PATCH, DELETE) и идемпотентность/безопасность.

- [ ] Версионирование: в URL (/v1/), в заголовке (Accept), в теле; обратная совместимость: добавление полей, опциональные поля; не ломать существующих клиентов.
- [ ] Документация: OpenAPI (Swagger); генерация клиентов и моков; инструменты (Postman, Insomnia, кодогенерация из OpenAPI).
- [ ] API-first: проектирование и фиксация контракта (OpenAPI) до реализации; польза для параллельной разработки фронта и бекенда; итерация контракта по версиям.
- [ ] Политика deprecation: срок поддержки старых версий; коммуникация с потребителями; удаление устаревших версий по календарю.
- [ ] **Идемпотентность мутаций:** идемпотентные ключи (Idempotency-Key) для POST/PUT; повтор запроса с тем же ключом возвращает сохранённый результат; необходимость при retry и мобильных клиентах.
- [ ] **Стриминг и long-polling в REST:** потоковая отдача ответа (streaming response), SSE (Server-Sent Events) для push от сервера; long-polling как fallback; когда уместны в REST-архитектуре; ограничения и альтернативы (WebSockets, отдельный real-time канал).

## Часть 16. Бекенд: GraphQL

### 16.1 GraphQL: схема и практика

- [ ] Схема: типы, запросы (query), мутации (mutation), подписки (subscription); клиент запрашивает нужные поля (нет over/under-fetching).

- [ ] **Ограничения и когда не выбирать GraphQL:** REST достаточен при простых ресурсах и фиксированном наборе клиентов; GraphQL избыточен при одном типе клиента и простых запросах; учёт операционной сложности (мониторинг запросов, лимиты по глубине/сложности).

- [ ] Инструменты GraphQL: Apollo Studio, GraphiQL/Playground; кодогенерация типов и хуков из схемы; мониторинг запросов.
- [ ] **GraphQL Federation:** несколько GraphQL-схем от разных сервисов, объединённые шлюзом; когда применять.
- [ ] **Подписки (subscription) и real-time:** GraphQL subscriptions — транспорт (WebSockets, SSE), когда применять для обновлений в реальном времени; ограничения масштабирования (липкость, масштабирование подписок); сравнение с polling и с отдельным real-time каналом (WebSockets вне GraphQL).

## Часть 18. Бекенд: данные в архитектуре

Материалы: [→ Навигация по части 18](pact/18_backend_dannye_v_arkhitekture_replikatsiya_shardirovanie_kesh_poliglot_persistenciya/index.md)

> Репликация, шардирование, консистентность; выбор хранилищ; кэш как слой; space-based.

- [ ] **Цель части 18:** дать репликацию (мастер-реплика, кворумы, read-your-writes), шардирование и решардинг, полиглот персистентность и кэш; space-based; данные и регуляторика; антипаттерны и типичные ошибки при выборе хранилища.
- [ ] **Связь с другими частями:** опирается на части 3–9 (слои, модули, микросервисы). Критично для частей 13–14 (проекции, event store), 19 (масштабирование).

### 18.1 Репликация и консистентность

- [ ] Мастер-реплика (master-replica): запись в мастер, чтение из реплик; задержка репликации (replication lag); чтение после записи (read-after-write) — когда нужна консистентность сразу после записи.
- [ ] Read-your-writes (чтение своей записи): гарантия для пользователя видеть только что сделанные изменения; как обеспечить (чтение из мастера, сессионная консистентность, заголовок); влияние на архитектуру API и репликации.
- [ ] Синхронная vs асинхронная репликация: гарантии и влияние на латентность записи; мониторинг replication lag и алерты.
- [ ] Многомастер: конфликты и их разрешение; сценарии (геораспределение, офлайн).
- [ ] Уровни консистентности: strong, eventual; CAP и выбор при партиционировании; практические гарантии (linearizability, causal consistency).
- [ ] Кворумы (quorum) при записи и чтении: согласованность при отказе части реплик; типичные настройки (W+R>N); split-brain и его предотвращение при мастер-реплика.

### 18.2 Шардирование и партиционирование

- [ ] Шардирование: разделение данных по ключу (user_id, tenant_id, region); равномерность нагрузки и горячие ключи.
- [ ] Выбор ключа шардирования: избегать ключей, по которым делают диапазонные запросы без ключа; составной ключ для мультитенантности.
- [ ] Партиционирование внутри одной БД: партиции по времени, по диапазону; управление жизненным циклом партиций.
- [ ] Решардинг: изменение ключа или количества шардов при росте; double-write и миграция данных; инструменты (Vitess, Citus, ручной решардинг); риски и простои.
- [ ] Типичные ошибки: один «логический» шард на всё приложение; шардирование по ключу, по которому нужны джойны между шардами (приводят к распределённым джойнам).

### 18.3 Полиглот персистентность и кэш

- [ ] Выбор СУБД под сценарий: OLTP (транзакционная нагрузка, консистентность) vs OLAP (аналитика, тяжёлые запросы); колоночные, поисковые движки как часть архитектуры данных.
- [ ] Временные ряды и потоковые данные: сценарии (метрики, телеметрия, логи); выбор хранилищ (TimescaleDB, InfluxDB, потоковые БД) как часть архитектуры данных.
- [ ] Разные хранилища под разные задачи: реляционная для транзакций, документная для гибкой схемы, поисковая для full-text, кэш для горячих данных.
- [ ] Бэкапы и восстановление в архитектуре: что бэкапить (БД, event store, конфигурация); RPO/RTO; влияние на дизайн (например, событийный лог для восстановления).
- [ ] Миграции данных при шардировании: перераспределение при изменении ключа шардирования; double-write, обратное чтение; риски и план отката.
- [ ] Кэш как слой: стратегии (cache-aside, write-through, write-behind); инвалидация и TTL; консистентность кэша с источником истины.
- [ ] Кэш на уровне приложения vs распределённый (Redis, Memcached); когда что; опасность устаревших данных.
- [ ] **Space-based архитектура (in-memory data grid):** данные в распределённой памяти (tuple space, кэш-сетка); снятие нагрузки с БД, сессии, кэш; продукты (Hazelcast, Ignite, Coherence и аналоги); когда уместна — высокая пропускная способность чтения, сессионное состояние; ограничения — консистентность, сложность операций.
- [ ] **Данные и регуляторика:** хранение по юрисдикциям (data residency); GDPR и право на удаление — влияние на архитектуру (удаление каскадом, анонимизация); аудит доступа к персональным данным как требование к наблюдаемости.
- [ ] *(В контенте: блок «Запомните» (18) — владение данными в микросервисах: один сервис — владелец данных; доступ других только через API или события.)*
- [ ] **Антипаттерны:** общая БД как единственный способ обмена данными между сервисами; кэш без стратегии инвалидации; шардирование «на будущее» без реальной нагрузки.
- [ ] Типичные ошибки при выборе хранилища: выбор по популярности, а не по паттерну доступа (OLTP vs OLAP, поток чтения/записи); один тип хранилища для всех сценариев.
- [ ] *(В контенте: блок «Проверь себя (18)» — для сценария выбрать хранилище, стратегию репликации или шардирования и роль кэша.)*

## Часть 19. Бекенд: масштабирование и устойчивость

Материалы: [→ Навигация по части 19](pact/19_backend_masshtabirovanie_i_ustoychivost_balansirovka_kesh_ocheredi_resilience/index.md)

> Горизонтальное и вертикальное масштабирование; балансировка; кэш; очереди; circuit breaker, retry, rate limiting.

- [ ] **Связь с другими частями:** опирается на части 9, 18. Пересекается с частью 31 (resilience, наблюдаемость).

### 19.1 Масштабирование

- [ ] Вертикальное: больше CPU/RAM на одном узле; простота, но потолок и стоимость.
- [ ] Горизонтальное: больше узлов; stateless сервисы как предпосылка; сессии и липкость (sticky sessions) — когда допустимы и когда избегать.
- [ ] Балансировка нагрузки: L4 vs L7; алгоритмы (round-robin, least connections, weighted); health checks и исключение нездоровых инстансов.
- [ ] Типы health check: liveness (жив ли процесс) vs readiness (готов ли принимать трафик); влияние на балансировщик и деплой; медленный старт и readiness.
- [ ] Автомасштабирование: метрики (CPU, RPS, очередь, кастомные); пороги scale-up vs scale-down; масштабирование до пиков и экономия в провалах.
- [ ] Scale to zero vs минимальная мощность: когда держать минимум инстансов (cost vs холодный старт, латентность первого запроса); применимо к serverless и контейнерам.

### 19.2 Кэширование и очереди

- [ ] Кэш: снижение нагрузки на БД и внешние API; стратегии инвалидации; кэш на границе (CDN, edge).
- [ ] Очереди: сглаживание пиков; асинхронная обработка; гарантии доставки и повторные попытки; dead letter queue.
- [ ] Мониторинг очередей: глубина очереди, размер DLQ, возраст сообщений в DLQ; алерты; ручная обработка или реплей сообщений из DLQ.
- [ ] Backpressure: ограничение скорости входящих запросов при перегрузке; отказ с 503 vs очередь.

### 19.3 Устойчивость (resilience)

- [ ] Circuit breaker: при частых ошибках «размыкать» вызовы к зависимому сервису; быстрый отказ вместо таймаутов; полуоткрытое состояние для проверки восстановления.
- [ ] Retry: экспоненциальный backoff; jitter (случайная добавка) для избежания «грозы повторных попыток»; идемпотентность операции при повторе; ограничение числа попыток.
- [ ] Bulkhead: изоляция пулов ресурсов (потоки, соединения), чтобы сбой одного потребителя не забирал все ресурсы.
- [ ] Rate limiting: ограничение запросов по ключу (IP, user_id, API key); защита от перегрузки и злоупотреблений.
- [ ] Каскад таймаутов: таймауты на каждом уровне вызова, чтобы сбой одного сервиса не блокировал все; связь с circuit breaker и значениями таймаутов.
- [ ] Инструменты и библиотеки resilience: Polly (.NET), Resilience4j (Java), go-resilience и аналоги; конфигурируемые retry, circuit breaker, bulkhead.
- [ ] *(В контенте: блок «Проверь себя (19)» — когда применить circuit breaker, когда retry, когда очередь.)*
- [ ] **Типичные ошибки:** retry без backoff и лимита (усиление шторма); отсутствие идемпотентности при retry; circuit breaker без мониторинга состояния.
- [ ] **Практика после части 19:** для одного внешнего вызова (БД, другой сервис) выбрать и обосновать: только retry, только circuit breaker, или оба; указать значения (таймаут, порог срабатывания).

## Часть 20. Бекенд: развёртывание и инфраструктура

Материалы: [→ Навигация по части 20](pact/20_backend_razvertyvanie_i_infrastruktura/index.md)

> Контейнеры, оркестрация (Kubernetes и аналоги); CI/CD; конфигурация и секреты; инфраструктура как код.

- [ ] **Связь с другими частями:** опирается на части 9, 19. Нужна для надёжного запуска и обновления сервисов; пересекается с частью 31 (наблюдаемость в production).

### 20.1 Контейнеры и образы

- [ ] Образ: слои, базовый образ; мультистейдж-сборка для уменьшения размера; один процесс на контейнер.
- [ ] Best practices образов: минимальный базовый образ (alpine, distroless); non-root пользователь; неизменяемость слоёв.
- [ ] Инструменты сканирования образов: Trivy, Snyk, Clair — уязвимости и соответствие политикам; интеграция в CI.
- [ ] Идемпотентность и неизменяемость: образ не меняется после сборки; конфигурация снаружи (переменные, секреты, монтирование).
- [ ] Реестр образов: хранение, тегирование (семантические версии, теги по коммиту); сканирование уязвимостей.

### 20.2 Оркестрация (Kubernetes и аналоги)

- [ ] Pod, Deployment, Service; масштабирование (HPA); пробы (liveness, readiness); ресурсные лимиты (requests/limits); Resource Quotas и LimitRange.
- [ ] Зависимости и readiness: сервис зависит от БД или другого сервиса — readiness должен отражать доступность зависимостей; иначе трафик попадёт на неготовый инстанс; связь с circuit breaker и health check зависимостей.
- [ ] Сетевая модель: Service discovery, DNS; Ingress для входящего трафика; секреты и ConfigMap; сетевые политики (NetworkPolicy) для изоляции трафика.
- [ ] Стратегии деплоя: rolling update; blue-green (два полных окружения, переключение); canary (часть трафика на новую версию); критерии выбора и откат.
- [ ] Rollback: откат к предыдущей ревизии (kubectl rollout undo); проверка здоровья после деплоя; автоматический откат при падении проб.
- [ ] Альтернативы и упрощения: managed Kubernetes (EKS, GKE, AKS); серверные решения (Nomad, Docker Swarm); serverless-платформы без оркестрации.
- [ ] Когда оркестрация избыточна: один сервис, простая схема деплоя; не усложнять без необходимости.

### 20.3 CI/CD и конфигурация

- [ ] Пайплайн: сборка, тесты, артефакт, деплой по окружениям (dev, staging, prod); ручные gates vs автоматический деплой.
- [ ] GitOps (опционально): желаемое состояние окружения в репозитории (манифесты, Helm); оператор (Argo CD, Flux) синхронизирует кластер с репо; аудит изменений, откат через git revert.
- [ ] Артефакт деплоя: что продвигать (образ контейнера, пакет); один артефакт через все окружения vs пересборка под окружение; неизменяемость артефакта для воспроизводимости.
- [ ] Конфигурация: окружение (env), конфиг-файлы, централизованное хранилище (Vault, AWS Secrets Manager); не хранить секреты в коде и образах.
- [ ] Ротация секретов: смена секретов без полного перезапуска (интеграция с Vault, внешние хранилища); кто и когда обновляет; влияние на доступность.
- [ ] Инфраструктура как код (Terraform, Pulumi, CloudFormation): воспроизводимость окружений; ревью изменений инфраструктуры.
- [ ] **Graceful shutdown:** завершение приложения с дообработкой текущих запросов; снятие с балансировщика перед остановкой; таймаут на завершение; влияние на zero-downtime деплой.
- [ ] *(В контенте: блок «Запомните» (20) — контейнер должен быть stateless для горизонтального масштабирования; данные и секреты — снаружи.)*
- [ ] **Типичные ошибки:** жёстко зашитая конфигурация в образе; один образ с разными «режимами» через флаги вместо отдельной конфигурации; деплой без отката (rollback) и проверки здоровья; игнорирование graceful shutdown (обрыв соединений при деплое).
- [ ] **Практика после части 20:** описать для одного сервиса стратегию деплоя (rolling/blue-green/canary) и критерий отката; перечислить, что в конфигурации снаружи образа, что — внутри.

## Часть 21. Фронтенд: MPA (Multi-Page Application)

Материалы: [→ Навигация по части 21](pact/21_frontend_mpa_multi_page_application/index.md)

> Каждая страница — отдельный запрос; полная или частичная перезагрузка (Turbo, HTMX); когда выбирать MPA.

- [ ] **Связь с другими частями:** опирается на части 0–2; база для частей 22–24 (SPA, SSR/SSG, Islands). MPA — отдельная тема для детального разбора.

### 21.1 MPA: суть и практика

- [ ] **MPA:** каждая страница — отдельный запрос к серверу; полная перезагрузка или частичное обновление (Turbo, HTMX); SEO естественно; меньше клиентского JS; состояние на сервере (сессии) или в cookie.
- [ ] **Частичное обновление (Turbo Drive, HTMX):** замена фрагментов без полной перезагрузки; сохранение состояния вкладки; когда достаточно для «почти SPA» без тяжёлого фреймворка.
- [ ] **Критерии выбора MPA:** контент-ориентированные сайты; потребность в SEO без сложного SSR; простота стека; команда без глубокого фронтенд-опыта; низкие требования к интерактивности «на лету».
- [ ] **Ограничения MPA:** полная перезагрузка — потеря клиентского состояния при навигации; латентность при каждом переходе; сложнее богатая интерактивность без перехода к SPA/гибриду.
- [ ] **Инструменты:** классические фреймворки (Rails, Django, Laravel — server-rendered); Turbo (Hotwire), HTMX; когда добавить «островки» интерактивности (см. часть 24).
- [ ] **Граничные случаи:** MPA с тяжёлым JS на части страниц — риск дублирования логики; гибрид MPA + SPA для одной зоны (например, админка внутри MPA-сайта).
- [ ] **Типичные ошибки:** смешение MPA и SPA без чёткой границы; игнорирование кэширования страниц и частичных обновлений; полная перезагрузка там, где достаточно Turbo/HTMX.
- [ ] *(В контенте: блок «Проверь себя (21)» — для контент-сайта и для админки выбрать MPA vs SPA и обосновать.)*
- [ ] **Практика после части 21:** для одного знакомого или учебного сайта назвать тип (MPA/гибрид) и один критерий выбора; перечислить одно ограничение MPA.

## Часть 22. Фронтенд: SPA (Single-Page Application)

Материалы: [→ Навигация по части 22](pact/22_frontend_spa_single_page_application/index.md)

> Один HTML, роутинг на клиенте; богатая интерактивность; когда выбирать SPA; SEO и пререндер.

- [ ] **Связь с другими частями:** опирается на части 0–2; пересекается с частями 23–24 (SSR для SEO, гидрация). SPA — отдельная тема для детального разбора.

### 22.1 SPA: суть и практика

- [ ] **SPA:** один HTML, приложение на клиенте; роутинг на клиенте; богатая интерактивность; SEO требует SSR или пререндера; первая загрузка может быть тяжёлой; состояние в памяти (и при необходимости в localStorage/sessionStorage).
- [ ] **Критерии выбора SPA:** приложение-ориентированные сценарии (админки, дашборды, инструменты); высокая интерактивность; когда допустима задержка первого контента; внутренние приложения без жёстких требований к SEO.
- [ ] **Ограничения SPA:** первый загрузка (bundle size, TTI); SEO без доп. мер (пререндер, SSR); зависимость от JS; сложнее кэширование (не по URL страницы).
- [ ] **SPA и SEO:** пререндер для краулеров (Prerender.io, рендер на сервере по запросу бота); когда достаточно, когда нужен полноценный SSR (часть 23).
- [ ] **Инструменты:** React, Vue, Angular, Svelte — типичная связка SPA (роутер + состояние + сборка); сравнение по размеру бандла и порогу входа.
- [ ] **Граничные случаи:** SPA с серверными данными на каждый роут (data loaders) — приближение к SSR; гибрид SPA + MPA (разные зоны сайта).
- [ ] **Типичные ошибки:** выбор SPA «по умолчанию» без учёта SEO и TTI; один огромный бандл без код-сплиттинга; игнорирование доступности (a11y) и восстановления при отключённом JS.
- [ ] *(В контенте: блок «Проверь себя (22)» — когда SPA достаточен, когда нужен пререндер или SSR.)*
- [ ] **Практика после части 22:** для одного сценария (админка, лендинг) обосновать выбор SPA; назвать одну меру по SEO и одну типичную ошибку (bundle, TTI).

## Часть 23. Фронтенд: SSR, SSG, ISR

Материалы: [→ Навигация по части 23](pact/23_frontend_ssr_ssg_isr/index.md)

> Серверный рендер при запросе; статическая генерация на сборке; инкрементальная регенерация; гибриды и per-route стратегии.

- [ ] **Связь с другими частями:** опирается на части 21–22 (MPA, SPA); база для части 24 (гидрация, islands). SSR/SSG/ISR — отдельная тема для детального разбора.

### 23.1 SSR, SSG, ISR

- [ ] **SSR (Server-Side Rendering):** HTML генерируется на сервере при запросе; быстрый First Contentful Paint; гидрация на клиенте; нагрузка на сервер; поток данных (data fetching на сервере перед рендером).
- [ ] **SSG (Static Site Generation):** HTML на этапе сборки; раздача статики (CDN); контент, не меняющийся при каждом запросе; время сборки и инвалидация при изменении контента.
- [ ] **ISR (Incremental Static Regeneration):** статика с фоновым обновлением по таймеру или по запросу; revalidate; компромисс между SSG и свежестью; когда применять.
- [ ] **Гибриды:** per-route стратегии (Next.js, Nuxt, SvelteKit); одна страница — SSG, другая — SSR; выбор по типу контента и частоте обновления.
- [ ] **Ограничения SSR:** нагрузка на сервер при росте RPS; необходимость гидрации (размер JS, TTI); холодный старт в serverless-окружении.
- [ ] **Ограничения SSG:** длительная сборка при большом числе страниц; устаревание контента до следующей сборки; не подходит для персонального контента на каждую страницу.
- [ ] **Инструменты:** Next.js (SSR/SSG/ISR), Nuxt, Remix (SSR + data loaders), SvelteKit; сравнение по дефолтной модели и экосистеме.
- [ ] **Граничные случаи:** SSR с потоковой отдачей (streaming); SSG с онлайновой подстановкой персональных блоков; смешение стратегий в одном приложении.
- [ ] **Типичные ошибки:** SSG для часто меняющегося или персонального контента без ISR; SSR без учёта нагрузки и кэширования; игнорирование hydration mismatch.
- [ ] *(В контенте: блок «Картинка в голове» — ось «статика ↔ динамика» и ось «клиент ↔ сервер».)*
- [ ] *(В контенте: блок «Проверь себя (23)» — для трёх страниц назначить SSG/SSR/ISR и обосновать.)*
- [ ] **Практика после части 23:** для одного приложения выбрать per-route стратегию (SSR/SSG/ISR) для 2–3 типов страниц; назвать одно ограничение SSR и одно SSG.

## Часть 24. Фронтенд: Islands, гидрация, PWA, JAMstack

Материалы: [→ Навигация по части 24](pact/24_frontend_islands_gidratciya_pwa_jamstack/index.md)

> Островная архитектура; стратегии гидрации (full, partial, progressive, streaming); Resumability; PWA, TWA; JAMstack.

- [ ] **Связь с другими частями:** опирается на части 21–23 (MPA, SPA, SSR/SSG). Пересекается с частью 34 (современные направления). Islands и гидрация — отдельная тема для детального разбора.

### 24.1 Островная архитектура (Islands)

- [ ] **Islands:** страница в основном статическая; интерактивные «островки» гидратируются выборочно (виджет корзины, форма); только островки тянут фреймворк; остальное — статический HTML.
- [ ] **Когда уместна:** контент-сайты с локальной интерактивностью; снижение объёма гидрации и TTI; маркетинговые и контент-сайты с формами/виджетами.
- [ ] **Ограничения:** координация состояния между островками (общий контекст, события); не все фреймворки поддерживают нативно; не подходит для «целиком приложения» с богатым состоянием.
- [ ] **Инструменты:** Astro (островки для React, Vue, Svelte и др.); Next.js — островки через границы «use client» и ленивую загрузку; сравнение по объёму JS и порогу входа.
- [ ] **Граничные случаи:** много островков на странице — когда выгода теряется; островок с тяжёлой зависимостью — влияние на LCP островка.
- [ ] **Типичные ошибки:** использовать islands для приложения с глобальным состоянием без продуманной передачи; дублирование фреймворка в каждом островке при неправильной настройке.

### 24.2 Гидрация и Resumability

- [ ] **Гидрация (hydration):** после SSR клиент «оживляет» разметку (привязка событий, реактивность); hydration mismatch — источник багов (даты, random, условный рендер по «на клиенте»); стратегии: full, partial, progressive, streaming; влияние на TTI и объём JS.
- [ ] **Partial и progressive hydration:** гидратировать только видимые или приоритетные компоненты; отложенная гидрация по scroll/viewport; инструменты и ограничения.
- [ ] **Resumability (Qwik и др.):** состояние сериализуется в HTML; отсутствие повторного выполнения на клиенте; отличия от классической гидрации; когда рассматривать.
- [ ] **Типичные ошибки гидрации:** несовпадение серверного и клиентского дерева; тяжёлая гидрация всего дерева; игнорирование размера payload для гидрации.

### 24.3 PWA, TWA, JAMstack

- [ ] **PWA:** сервис-воркер, кэширование (стратегии: network-first, cache-first, stale-while-revalidate); офлайн; установка; архитектура слоёв кэша и обновление приложения (skipWaiting, обновление кэша); ограничения (квоты, совместимость).
- [ ] **TWA:** PWA в нативной оболочке для публикации в сторах; Trusted Web Activity; границы и ограничения по платформам.
- [ ] **JAMstack:** JavaScript, API, Markup; статика + API + клиентская интерактивность; SSG + CDN + headless CMS или API; когда подходит (контент, маркетинг) и когда нет (частые персональные данные, real-time).
- [ ] **Инструменты PWA:** Workbox, фреймворк-интеграции (Next.js PWA, Vite PWA); инструменты аудита (Lighthouse).
- [ ] *(В контенте: блок «Проверь себя (24)» — для блога, админки, маркетингового лендинга предложить тип приложения и стратегию рендеринга.)*
- [ ] **Типичные ошибки:** выбор SPA «по умолчанию» без учёта SEO и TTI; SSG для часто персонального контента; игнорирование гидрации и размера бандла; PWA без стратегии обновления кэша.
- [ ] **Практика после части 24:** для блога, админки и маркетингового лендинга предложить тип приложения и стратегию рендеринга (из плана); назвать одну стратегию гидрации и одно ограничение PWA.

## Часть 25. Фронтенд: компонентная архитектура

Материалы: [→ Навигация по части 25](pact/25_frontend_komponentnaya_arkhitektura/index.md)

> Компоненты: атомы, молекулы, организмы; контейнеры и презентационные компоненты; FSD, дизайн-системы.

- [ ] **Связь с другими частями:** опирается на части 21–24 (типы приложений); определяет организацию кода и переиспользование. Связано с частью 26 (состояние), 29 (сборка и общие компоненты в монорепо).

### 25.1 Иерархия компонентов (Atomic Design и аналоги)

- [ ] **Атомы:** кнопка, инпут, иконка, лейбл — минимальные переиспользуемые элементы без смысловой логики.
- [ ] **Feature-Sliced Design (FSD):** слои app, processes, pages, widgets, features, entities, shared; публичный API слоя и запрет импортов «снизу вверх»; организация по фичам и сущностям; когда уместна и сравнение с Atomic Design / folder-by-type; ограничения: кривая обучения, дисциплина импортов, избыточность для маленького проекта без чётких слоёв.
- [ ] **Вертикальные срезы / folder-by-feature:** структура папок по фичам (feature/order, feature/cart), а не по типам (components/, services/); колокация всего, что относится к фиче; плюсы для навигации и границ.
- [ ] **MVC, MVVM, MVI на фронтенде:** MVC — Model, View, Controller; MVVM — ViewModel и биндинги (Angular, Vue); MVI — односторонний поток Intent → Model → View; когда какой подход встречается и как соотносится с компонентной архитектурой.
- [ ] **Молекулы:** комбинация атомов (поисковая строка = инпут + кнопка); одна зона ответственности.
- [ ] **Организмы:** блоки интерфейса (шапка, карточка товара, форма заказа); могут содержать состояние и вызовы API.
- [ ] Шаблоны и страницы: композиция организмов; страница = шаблон + данные. Не обязательно строго следовать атомам/молекулам — важно разделение по ответственности и переиспользованию.
- [ ] Когда Atomic Design избыточен: маленькая команда, простой продукт; риск переусложнения иерархии; достаточно «компонент + контейнер» без атомов/молекул.
- [ ] *(В контенте: блок «Простыми словами» — компонент как «кирпичик с пропсами и своим куском UI/логики».)*

### 25.2 Контейнеры и презентационные компоненты

- [ ] **Презентационные:** только UI и пропсы; не знают об API и глобальном состоянии; легко тестировать и переиспользовать.
- [ ] **Контейнеры:** подключают данные (хуки, store, API); передают данные и колбэки в презентационные компоненты.
- [ ] Композиция компонентов: render props, slots (Vue), children; композиция vs пропс-дриллинг для передачи данных и слотов.
- [ ] Паттерн составных компонентов (compound components): группа связанных компонентов с общим контекстом (например, Tabs + TabList + Tab + TabPanel); когда применять для гибкого API без пропс-дриллинга.
- [ ] Разделение упрощает тесты (моки для контейнеров) и смену источника данных; не перегружать контейнеры логикой — выносить в хуки и сервисы.
- [ ] Утечка абстракций: презентационный компонент, который «знает» про конкретный API или формат бекенда — антипаттерн.
- [ ] Инструменты: Storybook и аналоги для изолированной разработки и документации компонентов; колокация стилей (CSS modules, styled-components, Tailwind) как часть границы компонента.

### 25.3 Дизайн-системы и границы компонентов

- [ ] Дизайн-система: библиотека компонентов + токены (цвета, отступы, типографика) + документация; единый визуальный язык и доступность.
- [ ] Версионирование дизайн-системы: обратная совместимость компонентов (breaking changes в пропсах); semver для библиотеки компонентов; стратегия обновления потребителей.
- [ ] Границы: компонент не должен знать о родительской странице и глобальной структуре приложения; общение через пропсы и события (или контекст при необходимости).
- [ ] **Обработка ошибок в архитектуре компонентов:** error boundaries (React и аналоги) — граница сбоя; где размещать (страница, виджет, корень); fallback UI и логирование; не ловить всё подряд, только ожидаемые границы.
- [ ] **Доступность (a11y) как архитектурное решение:** семантика разметки, фокус, клавиатурная навигация, ARIA; дизайн-система и общие компоненты как место гарантий a11y; влияние роутинга (фокус при смене страницы) и модальных окон.
- [ ] **Типичные ошибки:** «божественный» компонент (всё в одном); смешение уровней абстракции в одном компоненте; хардкод стилей и текстов вместо токенов и i18n; отсутствие error boundary для изоляции сбоев.
- [ ] *(В контенте: блок «Проверь себя (25)» — разнести один «толстый» компонент на контейнер + 2–3 презентационных.)*
- [ ] **Антипаттерны:** пропс-дриллинг через много уровней без контекста или композиции; компонент, который сам ходит в API и рисует — смешение ответственностей.

## Часть 26. Фронтенд: управление состоянием

Материалы: [→ Навигация по части 26](pact/26_frontend_upravlenie_sostoyaniem/index.md)

> Локальное, поднятое, глобальное, серверное состояние; выбор библиотек и стратегий; антипаттерны.

- [ ] **Связь с другими частями:** опирается на части 21–25; пересекается с частью 30 (данные с бекенда, кэш на клиенте).

### 26.1 Уровни состояния

- [ ] **Локальное (компонент):** useState, локальный стейт; форма, раскрытие списка; не поднимать без необходимости.
- [ ] Сохранение состояния при навигации: когда сохранять состояние страницы при уходе (keep-alive, кэш роута, scroll restoration); когда сбрасывать; влияние на память и предсказуемость.
- [ ] **Поднятое (lifting state up):** общее состояние у ближайшего общего предка; передача вниз и колбэки вверх.
- [ ] **Контекст (React Context и аналоги):** тема, локаль, аутентификация; не использовать для часто меняющихся данных (лишние ререндеры).
- [ ] **Глобальное хранилище (Redux, Zustand, Jotai и др.):** общее состояние приложения; когда много компонентов читают/пишут одни и те же данные; предсказуемые обновления (редьюсеры, подписки).
- [ ] Инструменты отладки состояния: Redux DevTools, React Query DevTools; трассировка обновлений и подписок.

### 26.2 Серверное состояние

- [ ] Данные с сервера: кэш, ревалидация, фоновая подстановка (stale-while-revalidate). Библиотеки: React Query (TanStack Query), SWR, Apollo (для GraphQL).
- [ ] Не дублировать серверное состояние в глобальном store без необходимости; сервер — источник истины, клиент — кэш и UI-состояние (оптимистичные обновления, pending).
- [ ] Оптимистичные обновления и откат: обновление UI до ответа сервера; откат при ошибке; конфликты и разрешение.
- [ ] Нормализация в store (если храним на клиенте): нормализованная структура vs дублирование; селекторы и мемоизация (reselect и аналоги) для производных данных.
- [ ] Кэш по ключу запроса; инвалидация после мутаций; prefetch для предсказуемой навигации.
- [ ] Стратегии ревалидации серверного состояния: on focus, on reconnect, по интервалу, при мутации; когда какую использовать; влияние на свежесть и нагрузку.

### 26.3 Выбор стратегии и антипаттерны

- [ ] Выбор: «где живут данные» — только в компоненте, в контексте, в store, на сервере с клиентским кэшем; минимизировать дублирование и лишние ререндеры.
- [ ] Сложные формы (многошаговые, большие): состояние формы (локально vs store); сброс при навигации; синхронизация с сервером и оптимистичные обновления.
- [ ] **Антипаттерны:** всё в глобальном store «на всякий случай»; серверные данные только в Redux без ревалидации; контекст для часто меняющихся данных; пропс-дриллинг вместо контекста или композиции.
- [ ] *(В контенте: блок «Запомните» — серверное состояние отделять от UI-состояния; глобальный стейт только для того, что реально разделяется между далёкими компонентами.)*
- [ ] *(В контенте: блок «Проверь себя (26)» — для сценария «список заказов + фильтры + деталь заказа» предложить, где хранить список, фильтры, выбранный ID.)*

## Часть 27. Фронтенд: роутинг и загрузка

Материалы: [→ Навигация по части 27](pact/27_frontend_routing_kod_splitting_preloading/index.md)

> Роутинг (file-based, config-based); код-сплиттинг; прелоадинг; метрики производительности.

- [ ] **Связь с другими частями:** опирается на части 21–26; влияет на TTI и LCP; пересекается с частью 29 (сборка и чанки).

### 27.1 Роутинг

- [ ] **File-based routing:** маршруты по структуре папок (Next.js, Nuxt, SvelteKit); удобно для статики и предсказуемых URL.
- [ ] **Config-based routing:** явная конфигурация маршрутов (React Router, Vue Router); вложенные маршруты, guards, lazy-loaded роуты.
- [ ] Вложенные маршруты и layout: общая оболочка (layout) для группы страниц; outlet/nested routes.
- [ ] Guards и middleware: проверка доступа, редиректы, загрузка данных перед входом на страницу (data loaders); защита от неавторизованного доступа.
- [ ] Обработка «страница не найдена» (404) и ошибок загрузки роута: отдельная страница 404 vs error boundary на уровне роута; когда показывать fallback, когда редирект.
- [ ] Data loaders и параллельная загрузка: загрузка данных для роута до или при монтировании; параллельные запросы; ошибки загрузки и error boundary на уровне роута/страницы.

### 27.2 Код-сплиттинг и ленивая загрузка

- [ ] Разбиение бандла по маршрутам (route-based code splitting): каждая страница — отдельный чанк; загрузка при переходе.
- [ ] Ленивая загрузка компонентов: dynamic import(); отложенная загрузка тяжёлых виджетов (модалки, редакторы); Suspense и границы загрузки (React); fallback UI.
- [ ] Прелоадинг: предзагрузка следующего вероятного маршрута (hover, prefetch, link rel=prefetch); предзагрузка данных (preload в React Query/SWR) для мгновенного отображения; стратегии (on hover, on viewport, после idle).
- [ ] Preload vs prefetch: preload — критичный ресурс для текущей страницы; prefetch — вероятный следующий маршрут/данные; приоритет и влияние на LCP/TTI.
- [ ] Scroll restoration: сохранение позиции скролла при навигации назад; влияние роутинга и SPA на поведение браузера.
- [ ] Deep linking и восстановление состояния: сохранение достаточного контекста в URL для повторного входа и шаринга ссылки; что хранить в URL vs в локальном state; ограничения: не класть в URL чувствительные данные (токены, персональные данные), только идентификаторы и параметры навигации.
- [ ] Баланс: не дробить слишком мелко (много маленьких запросов); не грузить всё сразу (большой начальный бандл).

### 27.3 Метрики и влияние архитектуры

- [ ] TTI (Time to Interactive), LCP (Largest Contentful Paint), CLS (Cumulative Layout Shift), FID/INP; Core Web Vitals.
- [ ] Влияние архитектуры: SSR/SSG улучшают LCP; код-сплиттинг и ленивая загрузка — TTI; стабильные размеры блоков и шрифты — CLS.
- [ ] *(В контенте: блок «Проверь себя (27)» — для SPA с 10 страницами предложить стратегию сплиттинга и одну метрику для приоритизации.)*
- [ ] **Типичные ошибки:** один огромный бандл; ленивая загрузка только «для галочки» без измерения; роут без guard для защищённых страниц.

## Часть 28. Фронтенд: микрофронтенды

Материалы: [→ Навигация по части 28](pact/28_frontend_mikrofrontendy_granicy_integraciya/index.md)

> Границы микрофронтендов; способы интеграции; стили, версии, общие зависимости; когда оправданы.

- [ ] **Связь с другими частями:** опирается на части 21–27; аналог микросервисов на фронтенде. Пересекается с частью 29 (модульная федерация, сборка).

### 28.1 Границы и владение

- [ ] Микрофронтенд: отдельная зона ответственности (домен/фича); отдельная кодовая база и деплой; владение своим фрагментом страницы.
- [ ] Границы по вертикали: один микрофронтенд может иметь свой BFF или общий API; границы по странице/виджету (страница собирается из фрагментов от разных команд).
- [ ] Shell/хост: приложение-оболочка, которое загружает и размещает микрофронтенды; роутинг на уровне shell или делегирование.
- [ ] Жизненный цикл микрофронтенда с хостом: загрузка, монтирование, обновление (обновление пропсов/контекста), размонтирование; контракт жизненного цикла (mount/unmount, обновление при смене роута) для предсказуемого поведения.
- [ ] Общая навигация и deep linking: единый URL и роутинг при нескольких микрофронтендах; передача параметров (query, state); браузерная история и кнопка «Назад».
- [ ] Передача данных между микрофронтендами: через URL, custom events, общий шинный слой; контракты и минимизация связанности.
- [ ] Тестирование композиции: как тестировать страницу из нескольких микрофронтендов; контракты интерфейса между хостом и микрофронтендами.

### 28.2 Способы интеграции

- [ ] **iframe:** изоляция стилей и скриптов; минусы: UX (высота, скролл), связь через postMessage; подходит для изолированных виджетов.
- [ ] **JS-bundle:** каждый микрофронтенд — бандл; хост загружает скрипты и монтирует в заданные контейнеры; риск конфликтов версий (React, глобальные переменные).
- [ ] **Web Components:** кастомные элементы как граница; изоляция стилей (Shadow DOM); совместимость фреймворков; ограничения по роутингу и данным.
- [ ] **Module Federation (Webpack):** общая рантайм-загрузка модулей; обмен компонентами и зависимостями между сборками; одна версия React при согласовании.
- [ ] Выбор: изоляция vs простота; согласование версий vs независимый деплой.

### 28.3 Стили, версии, общие зависимости

- [ ] Стили: изоляция (CSS-in-JS, Shadow DOM, префиксы классов) vs общая дизайн-система (общая библиотека компонентов); конфликты при глобальных стилях.
- [ ] Версии: согласованная версия React/Vue (shared runtime) vs дублирование; обновление shared — координация между командами.
- [ ] Общие зависимости: кто владеет shared-библиотекой; контракты версий (semver); тестирование совместимости при обновлении.
- [ ] Версионирование контракта между хостом и микрофронтендами: обратная совместимость пропсов/слотов/событий; deprecated и удаление; один контракт на границу.
- [ ] *(В контенте: блок «Простыми словами» — микрофронтенды как «несколько маленьких приложений на одной странице с согласованными правилами стыковки».)*
- [ ] **Когда оправданы:** несколько команд, независимый релиз фич, разные технологии в пределах одной страницы; иначе — избыточная сложность.
- [ ] **Ограничения:** операционная сложность (несколько сборок, окружений, координация релизов); отладка и трассировка запросов через несколько фрагментов; производительность (несколько фреймворков на странице).
- [ ] **Граничные случаи:** один микрофронтенд на страницу — когда избыточно; «гигантский» микрофронтенд на целый домен — граница с монолитным фронтом; смешение микрофронтендов и обычных компонентов в одном хосте.
- [ ] **Типичные ошибки:** микрофронтенды «по моде» при одной команде; отсутствие контракта версий shared; дублирование тяжёлых зависимостей в каждом микрофронтенде.
- [ ] *(В контенте: блок «Проверь себя (28)» — когда оправданы микрофронтенды; выбор способа интеграции под сценарий.)*
- [ ] **Практика после части 28:** описать границы одного «микрофронтенда» и способ интеграции с хостом (iframe/Module Federation/др.); назвать одно ограничение и одну типичную ошибку.

## Часть 29. Фронтенд: сборка и монорепо

Материалы: [→ Навигация по части 29](pact/29_frontend_sborka_monorepo_module_federation/index.md)

> Бандлеры; tree-shaking, chunk splitting; монорепо; Module Federation.

- [ ] **Связь с другими частями:** опирается на части 25, 27, 28; реализация доставки кода и общих библиотек.

### 29.1 Бандлеры и оптимизации

- [ ] Webpack, Vite, esbuild, Rollup: назначение (сборка приложения vs библиотеки); скорость сборки и HMR; когда что выбирать.
- [ ] Tree-shaking: удаление неиспользуемого кода; условия (ESM, side-effect-free модули); проверка размера бандла; sideEffects в package.json.
- [ ] Chunk splitting: по маршрутам, по вендорам (отдельный чанк для react, библиотек); долго кэшируемые вендор-чанки; имена чанков и стабильность хешей.
- [ ] Кэширование чанков в production: content hash в имени файла для долгого кэша; инвалидация при новом деплое; стратегии cache headers для статики.
- [ ] Оптимизация ассетов: изображения (форматы, размеры, lazy loading), шрифты (subset, font-display); минификация и сжатие (gzip, brotli).
- [ ] Анализ бандла: инструменты (webpack-bundle-analyzer, source-map-explorer); поиск дублирования и тяжёлых зависимостей.
- [ ] Source maps и отладка: dev vs production source maps; влияние на размер и безопасность (не выкладывать детальные карты в публичный prod).
- [ ] Сборки под окружения: environment-specific (dev/staging/prod); переменные окружения и подстановка на этапе сборки; не хранить секреты в клиентском бандле.

### 29.2 Монорепо

- [ ] Один репозиторий — несколько пакетов/приложений; общие зависимости и скрипты; инструменты: Nx, Turborepo, Lerna, pnpm workspaces.
- [ ] Когда предпочесть мультирепо монорепо: разные циклы релиза, разные организации/владельцы; границы репозитория как архитектурное решение; trade-offs координации vs независимости.
- [ ] Общие пакеты: дизайн-система, утилиты, конфиги; версионирование (internal packages, изменения в shared — влияние на потребителей).
- [ ] Зависимости между пакетами: граф сборки; кэширование сборки (Turborepo, Nx); публикация пакетов (npm private, внутренний registry).

### 29.3 Module Federation и удалённые модули

- [ ] Module Federation: экспорт и потребление модулей между независимыми сборками в рантайме; сценарии: микрофронтенды, загрузка плагинов.
- [ ] Remote и host; shared dependencies (одна версия React); версионирование и fallback при несовместимости.
- [ ] Типичные ловушки: рассинхрон версий shared; увеличение сложности отладки; не использовать без реальной потребности в распределённой разработке.
- [ ] *(В контенте: блок «Проверь себя (29)» — когда выбрать монорепо, когда мультирепо; когда Module Federation, когда отдельные бандлы.)*
- [ ] **Запомните:** сборка — часть архитектуры доставки; размер и количество чанков влияют на производительность и кэширование.
- [ ] **Практика после части 29:** для одного приложения описать стратегию chunk splitting и кэширования чанков; назвать одну типичную ловушку Module Federation или монорепо.

## Часть 30. Связка фронт–бекенд: BFF, контракты, аутентификация

Материалы: [→ Навигация по части 30](pact/30_svyazka_frontend_backend_bff_kontrakty_autentifikaciya/index.md)

> Backend for Frontend; API-контракты и контрактное тестирование; аутентификация и авторизация в архитектуре.

- [ ] **Связь с другими частями:** объединяет части 15–17 (API), 21–26 (фронт), 31 (безопасность). Критично для fullstack-понимания.

### 30.1 BFF (Backend for Frontend)

- [ ] **Зачем:** разный фронт (веб, мобильное приложение, десктоп) может нуждаться в разных формах данных и агрегации; один BFF на тип клиента или один универсальный.
- [ ] Тонкий BFF: прокси, маршрутизация, аутентификация; агрегация на клиенте или через несколько вызовов к бекенду.
- [ ] Толстый BFF: агрегация данных, преобразование форматов, оркестрация нескольких сервисов; меньше круглых поездок с клиента.
- [ ] Trade-off: тонкий — проще, меньше дублирования логики; толстый — меньше запросов, но дублирование между BFF при нескольких клиентах.
- [ ] Когда одного BFF достаточно: один тип клиента (только веб); простая агрегация; нет противоречивых требований по формату данных.
- [ ] Ограничения BFF: дополнительная латентность (один hop); единая точка отказа при одном BFF; масштабирование BFF вместе с числом клиентов.
- [ ] BFF и i18n/локализация: где хранить локаль (заголовок, cookie, URL); форматирование дат и чисел на BFF vs на клиенте; влияние на кэш и контракт.

### 30.2 API-контракты и контрактное тестирование

- [ ] Контракт: схема запросов/ответов (OpenAPI, GraphQL schema, protobuf); версионирование и обратная совместимость.
- [ ] Контрактное тестирование: потребитель (фронт или другой сервис) публикует ожидания; провайдер (бекенд) проверяет совместимость; Consumer-Driven Contracts (Pact, Spring Cloud Contract).
- [ ] **Где живут тесты на границе фронт–бекенд:** контрактные тесты (Pact и аналоги) — потребитель (фронт или BFF) и провайдер (бекенд) проверяют совместимость по контракту; моки и типы из контракта для изолированной разработки; интеграционные тесты BFF с реальным или мок-бекендом; E2E — на границе клиент/BFF/сервисы.
- [ ] Генерация моков и типов из контракта: клиентский код и тесты от одной схемы; меньше расхождений.
- [ ] Когда обновлять контракт: добавление полей (обратная совместимость); удаление и breaking changes — фаза deprecated и версия API.

### 30.3 Аутентификация и авторизация

- [ ] Аутентификация: установление личности (логин/пароль, OAuth2, OIDC, JWT); сессии vs токены; хранение токена (cookie httpOnly vs localStorage — риски XSS).
- [ ] Refresh token flow: короткоживущий access token, долгоживущий refresh token; где хранить refresh; ротация токенов и отзыв.
- [ ] Аутентификация для сервисов (machine-to-machine): сервис-сервис вызовы; client credentials, API keys, mTLS; не путать с пользовательским контекстом.
- [ ] CORS и архитектура: кросс-доменные запросы браузера; когда BFF снимает необходимость CORS для клиента; настройка заголовков и preflight.
- [ ] Авторизация: проверка прав (RBAC, ABAC, права на ресурс); на уровне API (middleware, граф резолверов в GraphQL) и на уровне BFF.
- [ ] Передача контекста: заголовки (Authorization), контекст в запросе; бекенд не доверяет клиенту в определении прав — проверка на каждом запросе.
- [ ] *(В контенте: блок «Проверь себя (30)» — для мобильного и веб-клиента предложить один или два BFF и способ передачи токена.)*
- [ ] **Типичные ошибки:** доверять клиенту в вопросе прав; хранить чувствительные токены в localStorage без необходимости; отсутствие контракта между фронтом и бекендом (договорённости «на словах»).
- [ ] **Практика после части 30:** описать поток аутентификации для одного клиента (токен, refresh, где хранить); перечислить, что не должно попадать в контракт и в логи.

## Часть 31. Безопасность, наблюдаемость, отказоустойчивость

Материалы: [→ Навигация по части 31](pact/31_bezopasnost_nablyudaemost_otkazoustoychivost/index.md)

> Безопасность в архитектуре; логи, метрики, трейсы; resilience на границах.

- [ ] **Связь с другими частями:** сквозные качества; опирается на части 15–17, 19, 20, 30. Применимо и к бекенду, и к фронтенду (безопасность клиента, мониторинг ошибок).

### 31.1 Безопасность в архитектуре

- [ ] Границы доверия: что считается «внутри» и «снаружи»; не доверять входящим данным; валидация на границе (вход в систему).
- [ ] Zero Trust как принцип: не считать внутреннюю сеть доверенной; проверка и авторизация на каждой границе; влияние на дизайн API и сервисов.
- [ ] Минимизация поверхности атаки: открывать только необходимые API; принцип наименьших привилегий для сервисов и БД.
- [ ] Защита данных: шифрование в покое и при передаче (TLS); чувствительные данные не логировать; секреты в хранилищах секретов, не в коде.
- [ ] Защита от типовых атак: инъекции (параметризованные запросы, валидация); CSRF, XSS (санитизация, CSP); rate limiting и защита от DDoS на уровне архитектуры.
- [ ] Защита от перебора (brute force): ограничение попыток входа (логин, сброс пароля); где размещать (API gateway, BFF, отдельный сервис); связь с rate limiting и блокировкой.
- [ ] Аудит безопасности в архитектуре: логирование доступа к чувствительным операциям; хранение аудит-логов; соответствие регуляторике (например, доступ к персональным данным).
- [ ] Типичные ошибки в безопасности: доверять внутренней сети без проверки (zero trust); логировать токены или PII; отсутствие валидации на границе; жёстко зашитые секреты.

### 31.2 Наблюдаемость (Observability)

- [ ] **Логи:** структурированные логи (JSON, единые поля: timestamp, level, message, trace_id, context); уровни; что не логировать (пароли, токены, PII); агрегация (ELK, Loki, CloudWatch).
- [ ] **Метрики:** бизнес и технические (RPS, латентность, ошибки); экспорт в Prometheus и аналоги; дашборды и алерты; SLI/SLO и целевые значения (например, p99 латентности).
- [ ] **Трейсы:** распределённая трассировка (trace ID, span ID); корреляция запросов через сервисы; OpenTelemetry, Jaeger, Zipkin; самплирование при высокой нагрузке.
- [ ] Корреляция: один идентификатор запроса от фронта до БД; логи + метрики + трейсы для диагностики.
- [ ] Распространение контекста трассировки: W3C Trace Context (traceparent, tracestate); передача trace_id между сервисами и от фронта к бекенду; инструменты и заголовки.
- [ ] «Три столпа» наблюдаемости: логи (что произошло), метрики (сколько, как долго), трейсы (где прошёл запрос); как они дополняют друг друга при расследовании инцидента.
- [ ] Алертинг: на что алертить (доступность, ошибки, латентность); избежание шума и усталости от алертов; runbooks и эскалация.
- [ ] *(В контенте: блок «Запомните» — без наблюдаемости нельзя уверенно эксплуатировать распределённую систему; закладывать с самого начала.)*

### 31.3 Отказоустойчивость на границах

- [ ] Таймауты на все внешние вызовы; повторные попытки с backoff; circuit breaker для зависимостей.
- [ ] Выбор значений таймаутов: связь с SLA зависимостей; таймаут меньше суммы ожидаемой латентности и retry; типичные значения по типам вызовов (БД, HTTP, очередь).
- [ ] Идемпотентность: повтор запроса не должен менять результат; идемпотентные ключи для критичных операций.
- [ ] Деградация: при недоступности зависимости возвращать кэш или упрощённый ответ вместо полного падения; явные сообщения пользователю; многоуровневая деградация (цепочка fallback: основной API → кэш → заглушка).
- [ ] **Типичные ошибки:** отсутствие таймаутов; логирование чувствительных данных; единственная точка отказа без резерва и мониторинга.
- [ ] *(В контенте: блок «Проверь себя (31)» — что логировать на границе, что не логировать; как распространять trace_id.)*
- [ ] **Практика после части 31:** для одного сервиса или BFF перечислить, что логировать на границе и что не логировать; описать передачу trace_id от клиента к бекенду.

## Часть 32. Эволюция архитектуры

Материалы: [→ Навигация по части 32](pact/32_evolyutsiya_arkhitektury_migratsii_adr_c4/index.md)

> Миграции; ADR; документация (C4, диаграммы); когда пересматривать решения.

- [ ] **Связь с другими частями:** опирается на все предыдущие; без эволюции архитектура «застывает» или хаотично дрейфует.

### 32.1 Стратегии миграции

- [ ] **Strangler Fig:** постепенная замена старой системы новыми модулями; прокси/маршрутизация переключает трафик на новое; старый код удаляется по частям.
- [ ] Порядок миграции: какие модули/сервисы резать первыми (по графу зависимостей, по риску, по ценности); зависимые модули после независимых.
- [ ] Параллельный запуск: старый и новый вариант работают параллельно; сравнение результатов; переключение трафика понемногу (canary, feature flags); shadowing (дублирование трафика на новую версию без влияния на ответ) для проверки.
- [ ] Документирование текущего и целевого состояния: as-is диаграмма (что есть сейчас); to-be (целевая архитектура); билеты/этапы миграции с привязкой к as-is → to-be.
- [ ] Обратная миграция (rollback миграции): когда и как откатывать шаг миграции; сохранение возможности отката данных и трафика.
- [ ] Риски миграции: данные, консистентность, откат; план отката и проверки на каждом шаге.
- [ ] *(В контенте: блок «Простыми словами» — миграция как «ремонт дороги по полосам, не перекрывая движение».)*

### 32.2 Документация решений и структуры

- [ ] **ADR (Architecture Decision Record):** контекст, решение, последствия; версионирование; когда и почему выбрали технологию или подход.
- [ ] **C4 model:** Context, Container, Component, Code — уровни детализации диаграмм; единый язык для обсуждения.
- [ ] Диаграммы потоков данных и вызовов; обновление при изменениях; хранить рядом с кодом (docs/, ADR folder).
- [ ] Ответственность за актуальность диаграмм: кто обновляет, триггеры (релиз, новый ADR); диаграммы как часть ревью изменений; «docs as code» — версионирование вместе с кодом.
- [ ] Инструменты для диаграмм: Structurizr, Mermaid, PlantUML, draw.io; согласование нотации в команде; генерация из кода (опционально).
- [ ] Шаблоны ADR: формат (контекст, решение, последствия); версионирование ADR; где хранить (docs/adr/, отдельный репозиторий).
- [ ] Ревизия архитектуры: периодический пересмотр (раз в квартал/полгода); признаки необходимости пересмотра — частые сбои, невозможность менять части системы, рост долга.
- [ ] **Feature flags в архитектуре:** включение/выключение фич и экспериментов без деплоя; хранение флагов (конфиг, отдельный сервис); влияние на тестирование и консистентность; снятие флагов после стабилизации (техдолг).

### 32.3 Когда пересматривать архитектуру

- [ ] Изменение требований: масштаб, мультитенантность, регуляторика; появление новых типов клиентов.
- [ ] Боль технического долга: невозможно добавлять фичи без поломок; команда не понимает части системы.
- [ ] Смена команды или стека: онбординг затруднён; устаревший стек мешает найму и развитию.
- [ ] *(В контенте: блок «Проверь себя (32)» — описать один шаг миграции монолита к микросервису с помощью Strangler Fig.)*
- [ ] **Запомните:** архитектура — живой набор решений; документировать и пересматривать так же важно, как и проектировать изначально.

## Часть 33. Антипаттерны и критерии выбора архитектуры

Материалы: [→ Навигация по части 33](pact/33_antipatterny_i_kriterii_vybora_arkhitektury/index.md)

> Распространённые антипаттерны; типичные ошибки; как выбирать архитектуру под контекст.

- [ ] **Связь с другими частями:** обобщает части 3–32; даёт критерии для принятия решений и самопроверки.

### 33.1 Антипаттерны

- [ ] **Big Ball of Mud:** отсутствие явной структуры; всё связано со всем; исправления — латки.
- [ ] **Распределённый монолит:** микросервисы по форме, но жёсткая связь по данным и вызовам; один деплой по сути.
- [ ] **Золотой молоток:** одно решение для всех задач (всё в микросервисах, всё в событиях); игнорирование контекста.
- [ ] **Преждевременная абстракция:** множество слоёв и интерфейсов «на будущее» без текущей потребности; усложнение без выгоды.
- [ ] **Игнорирование консистентности:** распределённая система без явной модели консистентности; «как-нибудь сойдётся» — баги и невоспроизводимые сбои.
- [ ] **Скрытые зависимости:** неявное использование общей БД, общего кэша, глобального состояния; при падении одного — падает другое.
- [ ] **Риски выбора технологий:** «золотой молоток» одной технологии; избыточная привязка к вендору (vendor lock-in) без абстракций; «not invented here» — отказ от готовых решений без обоснования; баланс между стандартностью и контролем.
- [ ] **«Телеграфный столб» (много мелких сервисов без смысловых границ):** разрез по техническим слоям вместо домена; операционные издержки без выгоды; связь с антипаттерном nanoservice.
- [ ] *(В контенте: блок «Картинка в голове» — антипаттерн как «красивая картинка снаружи, внутри — спагетти зависимостей».)*

### 33.2 Типичные ошибки

- [ ] Резать по слоям вместо домену (отдельный «сервис БД», «сервис API»).
- [ ] Микросервисы без готовности к операционной сложности (мониторинг, деплой, отладка).
- [ ] Отсутствие контрактов между фронтом и бекендом или между сервисами; «договорились в чате».
- [ ] Одна общая БД для всех микросервисов без владения данными.
- [ ] Игнорирование идемпотентности при асинхронной обработке и retry.
- [ ] Выбор архитектуры по тренду, а не по размеру команды, домену и ограничениям.
- [ ] Игнорирование обратной совместимости при изменении API или событий; отсутствие фазы deprecated.
- [ ] Отсутствие явной документации границ (кто за что отвечает, контракты); «знание в головах».
- [ ] Дублирование кода и логики между сервисами «ради независимого деплоя» без оценки стоимости поддержки и согласованности.

### 33.3 Критерии выбора

- [ ] **Размер и структура команды:** одна команда — монолит или модульный монолит часто достаточно; несколько команд с разными доменами — микросервисы или чёткие границы модулей; распределённая команда — явные контракты и независимые артефакты.
- [ ] **Домен:** сложные границы контекста (DDD) подталкивают к явным границам сервисов или модулей; простой CRUD — не усложнять; нестабильные границы домена — риск частого «разреза» сервисов.
- [ ] **Масштаб и латентность:** необходимость независимо масштабировать части; требования к задержкам (edge, real-time) влияют на размещение и форму API.
- [ ] **Сроки и зрелость:** зрелая DevOps и культура документирования — предпосылки для микросервисов; при жёстких сроках — минимально достаточная архитектура с возможностью эволюции.
- [ ] **Стоимость внедрения и владения:** монолит/модульный монолит — ниже на старте (один деплой, проще отладка); микросервисы — выше (деплой многих сервисов, контракты, наблюдаемость); событийная архитектура — стоимость шины, мониторинга lag, идемпотентности; BFF — дополнительный hop и поддержка; учитывать в чек-листе выбора и при обосновании решений.
- [ ] Чек-лист выбора: перечень вопросов (размер команды, домен, нагрузка, необходимость независимого релиза, готовность к операционной сложности); использование при принятии решений.
- [ ] Архитектурное ревью: что проверять (границы, контракты, долг, соответствие целям); как проводить и документировать результат; связь с ADR и критериями готовности.
- [ ] Модели зрелости (опционально): уровни зрелости архитектуры и организации (например, от «хаотичный монолит» к «модульный монолит» к «микросервисы»); не как жёсткая лестница, а как ориентир.
- [ ] *(В контенте: блок «Проверь себя (33)» — для сценария «стартап, 3 разработчика, один домен» предложить архитектуру и обосновать.)*
- [ ] **Запомните:** правильная архитектура — та, что соответствует контексту и допускает эволюцию; не «та, что в тренде».

## Часть 34. Современные направления

Материалы: [→ Навигация по части 34](pact/34_sovremennye_napravleniya/index.md)

> Edge, serverless, server components, JAMstack; гибридные схемы.

- [ ] **Связь с другими частями:** опирается на части 20, 23, 30; показывает эволюцию практик и куда двигаться после освоения ядра.

### 34.1 Edge и периферийные вычисления

- [ ] Вычисления на границе сети (CDN, edge workers): низкая латентность для пользователя; геораспределение без своих дата-центров.
- [ ] Сценарии: персонализация, A/B, редиректы, лёгкая логика; ограничения — малая среда выполнения, лимиты CPU/памяти, холодный старт на edge.
- [ ] BFF на edge: агрегация или адаптация ответов ближе к клиенту; когда уместно и ограничения (доступ к внутренним API, секреты).
- [ ] Инструменты: Cloudflare Workers, Vercel Edge, Deno Deploy, AWS Lambda@Edge; совместимость с Node/Web API; ограничения по провайдерам (runtime, лимиты).

### 34.2 Serverless и FaaS

- [ ] Функции как сервис: вызов по событию или HTTP; автомасштабирование до нуля; оплата за использование.
- [ ] Плюсы: отсутствие управления серверами; масштабирование; короткие задачи и события.
- [ ] Ограничения: холодный старт; лимиты времени выполнения и памяти; состояние только во внешних хранилищах; отладка и мониторинг распределённых вызовов.
- [ ] Локальная разработка и эмуляция: SAM, serverless-offline, эмуляторы провайдеров; ограничения эмуляции по сравнению с production (таймауты, лимиты, холодный старт).
- [ ] Когда уместен: события, очереди, редкие вызовы, всплески; не для долгих процессов и тяжёлой stateful-логики без доработок.
- [ ] Гибрид serverless + контейнеры: часть системы в FaaS (события, API-ручки), часть — в контейнерах (долгие процессы, специфичный runtime); критерии разнесения.

### 34.3 Server components и гибриды

- [ ] Server Components (React): компоненты, выполняющиеся только на сервере; нулевой JS на клиенте для них; передача данных и разметки без гидрации.
- [ ] Гибрид: часть дерева — server, часть — client; граница (use client) явная; снижение размера клиентского бандла и времени гидрации.
- [ ] JAMstack: JavaScript, API, Markup; статика + API + клиентская интерактивность; SSG + CDN + сервисы; когда подходит (контент, маркетинг) и когда нет (частые персональные данные).
- [ ] **Offline-first как архитектурный выбор:** локальное хранилище (IndexedDB, SQLite в PWA), синхронизация при появлении сети; конфликты и разрешение; когда уместно (мобильные полевые приложения, документы) и сложности (консистентность, разрешение конфликтов).
- [ ] **Плагинная архитектура (backend и frontend):** расширяемость через плагины, точки расширения, динамическая загрузка модулей; когда уместна (редакторы, платформы, продукт с дополнениями); контракты плагинов и изоляция.
- [ ] **Multi-tenancy как архитектурный выбор:** общая БД с tenant_id, отдельная схема на тенанта, отдельная БД/инстанс на тенанта; изоляция, безопасность, влияние на масштабирование и регуляторику.

### 34.4 Направления развития

- [ ] Управление состоянием на клиенте: движение к серверному источнику истины и тонкому клиенту (React Query, Server State).
- [ ] Полноценные fullstack-фреймворки: Next.js, Remix, SvelteKit — роутинг, данные, сервер и клиент в одном стеке; влияние на границу фронт–бекенд.
- [ ] *(В контенте: блок «Запомните» — современные направления не отменяют базовые принципы: границы, контракты, наблюдаемость, соответствие контексту.)*
- [ ] **Типичные ошибки:** применять edge/serverless/server components везде без оценки выгоды; игнорировать холодный старт и лимиты в production.
- [ ] Устаревающие и классические подходы: полностью серверный рендер без SPA (традиционные MVC-фреймворки); jQuery и «страницы с скриптами» без компонентной модели; где ещё встречаются и когда допустимы (внутренние инструменты, legacy).
- [ ] *(В контенте: блок «Проверь себя (34)» — когда edge, когда serverless, когда server components; ограничения каждого.)*
- [ ] **Практика после части 34:** для одного сценария (персонализация, A/B, API-ручка) выбрать edge vs serverless vs контейнер и обосновать; назвать одно ограничение (холодный старт, лимиты).

## Часть 35. Экспертиза: когда что выбирать; что изучать дальше

Материалы: [→ Навигация по части 35](pact/35_expertiza_kogda_chto_vybirat_chto_izuchat_dalshe/index.md)

> Сводка решений по контексту; баланс идеала и реальности; смежные области и ресурсы.

- [ ] **Связь с другими частями:** завершает план; даёт «шпаргалку» по выбору и направлениям углубления.

### 35.1 Сводка: когда какую архитектуру

- [ ] **Монолит (слоёный, модульный):** одна команда, один домен, быстрый старт, умеренная нагрузка; эволюция в сторону модульности и при необходимости «разреза».
- [ ] **2/3/4-tier, слоёная:** когда достаточно N-tier без микросервисов; shared-nothing при масштабировании монолита.
- [ ] **Микросервисы:** несколько команд/доменов, независимое масштабирование и релизы; готовность к операционной сложности и контрактам.
- [ ] **SOA vs микросервисы:** когда уместны более крупные сервисы и ESB; когда — мелкие сервисы и умная периферия.
- [ ] **События и CQRS:** асинхронные сценарии, несколько потребителей одних данных, отчётность и проекции; eventual consistency приемлема.
- [ ] **Actor model, pipe-and-filter, space-based:** нишевые варианты под высокую конкуренцию, потоковую обработку, in-memory data grid; когда рассматривать.
- [ ] **BFF:** несколько типов клиентов с разными потребностями в данных; агрегация и адаптация API под клиента.
- [ ] **Типы фронтенда (MPA/SPA/SSR/SSG/Islands):** контент vs приложение, SEO, TTI; гидрация и resumability.
- [ ] **Микрофронтенды:** несколько команд владеют частями одной страницы; независимый релиз; иначе — монорепо и общие компоненты.
- [ ] **Организация фронта:** Atomic Design, Feature-Sliced Design, folder-by-feature, MVC/MVVM/MVI — когда что выбирать.
- [ ] Гибридные варианты: модульный монолит с одним-двумя выделенными сервисами; BFF + общий бекенд; комбинация подходов под разные подсистемы; когда уместно смешивать.
- [ ] **Edge/Serverless/BaaS:** латентность, всплески, события; готовый бекенд (Firebase и др.) — когда достаточно и ограничения.
- [ ] Ссылка на «Карту всех архитектур»: полный перечень способов — в соответствующем разделе плана; при выборе проверять, что рассмотрены все релевантные варианты.
- [ ] Таблица/чек-лист «ситуация → архитектура»: в контенте раскрыть сводную матрицу (размер команды × домен × масштаб × латентность → рекомендуемый тип системы); использование при принятии решений.

### 35.2 Баланс идеала и реальности

- [ ] Идеальная архитектура недостижима; важна «достаточно хорошая» под контекст и способность к эволюции.
- [ ] Когда фиксировать решение в ADR: новое значимое архитектурное решение, смена подхода или технологии, отказ от рассмотренной опции; не на каждую мелкую деталь.
- [ ] Работа с ограничениями: сроки, legacy, навыки команды; поэтапное улучшение вместо «переписать с нуля».
- [ ] Признаки здоровой архитектуры: понятные границы, документированные решения, возможность менять части системы без каскадных поломок.

### 35.3 Смежные области и что изучать дальше

- [ ] **Domain-Driven Design (DDD):** bounded context, агрегаты, язык домена; углубление в границы и моделирование.
- [ ] **DevOps и SRE:** эксплуатация, надёжность, SLA/SLO; связь архитектуры с деплоем и мониторингом.
- [ ] **Тестирование архитектуры:** ArchUnit, dependency-cruiser; контрактное и интеграционное тестирование.
- [ ] Когда привлекать архитектора (или проводить архитектурный ревью): новый продукт или крупный модуль; миграция; повторяющиеся сбои или невозможность менять части системы; онбординг новых команд.
- [ ] Вопросы для самопроверки и интервью: перечень типовых вопросов по архитектуре (выбор монолит/микросервисы, контракты, консистентность, масштабирование); использование при ревью и найме.
- [ ] Литература и ресурсы: книги (Building Microservices, Clean Architecture, Designing Data-Intensive Applications); блоги и доклады; практика на реальных проектах и ревью чужой архитектуры.
- [ ] *(В контенте: блок «Проверь себя (35)» — без подсказок назвать три критерия выбора между монолитом и микросервисами.)*

## Контрольные вопросы для самодиагностики

1. **Понимание:** Чем архитектура отличается от «просто кода»? Что такое граница и контракт в ваших словах?
2. **Объяснение:** Почему в слоёной архитектуре зависимости должны идти только «вниз»? Что ломается при обратных зависимостях?
3. **Сравнение:** В чём разница между гексагональной архитектурой и Clean Architecture? Когда достаточно одной из них?
4. **Проектное мышление:** Команда из 5 человек, один домен, планируют рост до 50k RPS. Предложите архитектуру (монолит/микросервисы/гибрид) и обоснуйте.
5. **Диагностика:** Назовите три признака «распределённого монолита» и что делать, чтобы от него уйти.
6. **API и контракты:** Когда вы бы выбрали REST, когда GraphQL, когда gRPC? Приведите по одному сценарию для каждого.
7. **Фронтенд:** Для маркетингового лендинга и для админки с десятками экранов — какой тип приложения (MPA/SPA/SSR/SSG) и почему?
8. **Связка:** Зачем нужен BFF? Опишите один сценарий, когда один BFF на все клиенты достаточен, и один — когда нужны разные BFF.
9. **Эволюция:** Опишите один шаг миграции от монолита к микросервису по стратегии Strangler Fig. Что должно быть в as-is и to-be диаграмме?
10. **Критерии:** Назовите три фактора, при которых вы бы не рекомендовали микросервисы, и три — при которых они оправданы.
11. **Данные:** В чём разница между shared database для нескольких сервисов и database per service? Какие риски у первого подхода?
12. **Наблюдаемость:** Что такое liveness и readiness пробы и зачем разделять их? Как они связаны с деплоем и балансировкой?
13. **CQRS и события:** Когда вы бы ввели CQRS без event sourcing? Зачем мониторить consumer lag и lag проекций?
14. **Сага и BFF:** В чём разница между хореографией и оркестрацией в саге? Когда одного BFF на все клиенты достаточно и когда нужны отдельные BFF?
15. **Инструменты и практика:** Для чего в архитектуре используются линтеры зависимостей (ArchUnit, dependency-cruiser)? Зачем фиксировать решения в ADR и где их хранить?
16. **Наблюдаемость:** Зачем распространять trace_id от фронта к бекенду и между сервисами? Что даёт W3C Trace Context для расследования инцидентов?
17. **Контракты и миграции:** В чём риск неформального контракта («договорились в чате») при росте команды? Как определить порядок миграции модулей при переходе от монолита к сервисам?
18. **Цели архитектуры:** Приведите пример, когда расширяемость (extensibility) влияет на архитектурное решение. Что такое read-your-writes и зачем это учитывать в API?
19. **Полнота обзора:** Назовите по три архитектуры или способа организации для бекенда и для фронтенда (из плана), которые вы бы рассмотрели при выборе архитектуры нового продукта. Где в плане найти полный перечень?
20. **Практика и самопроверка:** После изучения частей 0–2 как вы бы проверили, что готовы перейти к части 3? Назовите один пункт из «Результаты по частям» и один из «Контрольные вопросы», релевантный для частей 3–5.

## Основные добавления / современные направления / что изучать после ядра

- **Domain-Driven Design (DDD):** bounded context, агрегаты, Ubiquitous Language, стратегическое и тактическое моделирование — углубление в границы и домен; связь с модульным монолитом и микросервисами.
- **Event-Driven Architecture в деталях:** паттерны (Event Notification, Event-Carried State Transfer, Event Sourcing); Kafka vs RabbitMQ; exactly-once семантика и идемпотентность.
- **Реактивные и потоковые системы:** backpressure, реактивные стеки (Project Reactor, Rx); когда и зачем.
- **Multi-tenancy и изоляция:** архитектурные варианты (общая БД с tenant_id, схема на тенанта, БД на тенанта); безопасность и производительность.
- **Геораспределение:** репликация между регионами; консистентность и латентность; выбор стратегии (активный-активный, активный-пассивный).
- **Тестирование архитектуры:** ArchUnit, jqassistant, dependency-cruiser; контрактное тестирование (Pact); хаос-инжиниринг и тесты отказоустойчивости.
- **Литература и курсы:** Building Microservices (O’Reilly), Clean Architecture (R. Martin), Designing Data-Intensive Applications (Kleppmann); курсы по Kubernetes, observability, безопасности.
- **Практика:** участие в ревью архитектуры; написание ADR в реальном проекте; эксперименты с одним новым подходом (например, event sourcing для одного поддомена) в учебном проекте.
- **Тестирование в контексте архитектуры:** где живут юнит-тесты (ядро, use cases); интеграционные тесты на границах (адаптеры, API); контрактные тесты между сервисами/фронт–бекенд; E2E и границы системы; тестируемость как критерий качества архитектуры. В основном потоке плана «где живут тесты» явно задано по частям 4, 5, 6, 7, 9, 30 (см. пункт 10 в «Как читать этот план»).
