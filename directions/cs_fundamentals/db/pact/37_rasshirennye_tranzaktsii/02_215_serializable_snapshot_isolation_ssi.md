## §215. Serializable snapshot isolation (SSI)

### Цель раздела

Понять, как СУБД может дать **сериализуемость** без глобальных блокировок чтения: через **SSI**, обнаружение **rw-конфликтов** и отмену одной из транзакций; научиться проектировать корректную **retry-логику** и писать транзакции так, чтобы повторы были безопасны.

### В этом разделе главное (если ты боишься не понять — начни отсюда)

1) **SERIALIZABLE в PostgreSQL = SSI.** Это “как SI”, но если параллельность могла дать неправильный результат, PostgreSQL **откатывает** одну транзакцию.  
2) Ты увидишь ошибку вида `could not serialize access...` и SQLSTATE **`40001`**. Это не “сломалось”, это “система защитила правильность”.  
3) Правильная реакция приложения — **повторить всю транзакцию** (retry) 2–5 раз с backoff+jitter.  
4) Чтобы retry был безопасным, транзакция должна быть **короткой** и **без внешних побочных эффектов** (или эти эффекты должны быть идемпотентными / вынесены в outbox).

### Термины

| Термин | Определение |
|---|---|
| **Serializable** | Самый строгий уровень изоляции: результат эквивалентен некоторому последовательному порядку транзакций. |
| **SSI** | “Сериализуемость поверх снимков”: техника, где чтения идут по снимку, но система отслеживает конфликтные зависимости и может отменить транзакцию. |
| **rw-conflict** | Зависимость “read → write” или “write → read” между транзакциями, которая может создать не-сериализуемый цикл. |
| **Dangerous structure** | Шаблон зависимостей, который может привести к циклу и не-сериализуемости (идея SSI). |
| **Abort / rollback** | Принудительная отмена транзакции системой для сохранения сериализуемости. |
| **Retry** | Повтор транзакции целиком (с новой попытки) после `serialization_failure`. |
| **Idempotency** | Свойство операции “повторить можно безопасно”: повтор не приводит к двойным эффектам. |

### Теория и правила

#### 0) Что значит “сериализуемость” на пальцах (без формул)

“Сериализуемо” означает: если бы мы остановили время и выполнили все транзакции **по очереди**, в каком-то порядке, мы смогли бы получить **точно такой же результат**, как при параллельном выполнении.

Если такого порядка не существует — значит, параллельность создала “невозможный результат” (логический баг), и SERIALIZABLE не должен его допустить.

#### 0.1) Почему SI иногда даёт “невозможный результат”

Потому что под SI две транзакции могут опираться на один и тот же снимок и принимать решения, которые вместе ломают инвариант (write skew). SI “не видит” логический конфликт, если нет конфликта “по одной строке”.

SSI решает это: он пытается сохранить скорость снимков, но не дать появиться “невозможному результату”.

#### 1) Почему SERIALIZABLE “стоит” дороже, но часто окупается

Под SI читаем без блокировок, но возможны write skew и прочие не-сериализуемые ситуации.  
Самый простой способ “исправить” это — поставить блокировки чтения как в классическом S2PL (two-phase locking). Но это снижает параллелизм.

SSI делает иначе:

- продолжает использовать снимки (чтения дешёвые);
- но **следит за rw-конфликтами**;
- если видит, что параллельность привела бы к не-сериализуемому результату, **отменяет** одну из транзакций.

То есть цена SERIALIZABLE в SSI — это не “всё стало медленно”, а:

- больше метаданных/проверок;
- иногда — **rollback и повтор** (retry).

#### 2) Как проявляется SSI в PostgreSQL (практически)

В PostgreSQL `SERIALIZABLE` реализован через SSI. Снаружи ты видишь:

- часть транзакций завершается ошибкой вида “could not serialize access due to read/write dependencies among transactions”;
- задача приложения — **повторить** транзакцию.

Важный вывод: **ошибка сериализации — не баг**, а нормальный “механизм сохранения правильности”.

Практически ты обычно увидишь:

- SQLSTATE **`40001`**
- текст ошибки, похожий на:
  - `could not serialize access due to read/write dependencies among transactions`
  - `DETAIL: Reason code: Canceled on identification as a pivot, during commit attempt.`
  - `HINT: The transaction might succeed if retried.`

Если ты видишь `HINT: ... succeed if retried` — это прямое указание “делай retry”.

#### 2.1) Мини‑пример: SI даёт write skew, а SSI не даст

Возьмём пример с врачами на смене из §214. Под SI обе транзакции могли бы закоммититься и сломать инвариант.

Под SSI (SERIALIZABLE) произойдёт так:

1. T1 и T2 читают снимок: “на смене 2 врача”.
2. T1 снимает врача A, T2 снимает врача B.
3. При попытке коммита система видит опасную структуру rw‑зависимостей (ниже объясню) и **отменяет** одну транзакцию.
4. Приложение повторяет отменённую транзакцию: теперь снимок уже другой (один врач уже снят), и вторая транзакция увидит “на смене 1” и не выполнит “снять” (или выполнит компенсационную логику).

То есть SSI превращает “тихий логический баг” в **явную, обрабатываемую ошибку** (retry).

#### 3) Почему нужен retry, а не “попробуем продолжить”

Если СУБД говорит, что сериализация невозможна, то она фактически говорит:

- “я не могу гарантировать корректность результата, если позволю вам обоим закоммититься”.

Поэтому она откатывает одну транзакцию. Единственный корректный путь — **повтор** целиком, уже на другом снимке.

Важно: retry — это не “попробуй ещё раз выполнить последний UPDATE”. Это “выполни всю бизнес‑операцию заново”.

#### 3.1) Почему retry не бесконечный

Если система перегружена или у тебя очень конфликтная операция (например, тысячи клиентов одновременно пытаются забрать последний товар), сериализация будет часто падать. Поэтому retry ограничивают:

- 3–5 попыток (типично),
- backoff с jitter, чтобы потоки не били в одну точку синхронно,
- после исчерпания попыток — “мягкая ошибка” пользователю (“попробуйте ещё раз”).

#### 4) Требования к коду: транзакция должна быть “повторяемой”

Чтобы retry был безопасным, внутри транзакции должны выполняться условия:

- **минимум внешних побочных эффектов** (отправка письма, HTTP-запрос, публикация в брокер) — нельзя делать “внутри” без защиты, иначе при retry будет дубль;
- если побочные эффекты нужны — используй **Outbox** (см. §219) или идемпотентные вызовы;
- транзакция должна быть **короткой**: чем дольше, тем выше шанс конфликта и `serialization_failure`.

#### 4.1) Что значит “короткая транзакция” на практике

Короткая — это когда внутри транзакции:

- нет сетевых вызовов (HTTP/gRPC);
- нет долгих вычислений;
- нет ожидания пользователя;
- нет чтения “миллионов строк ради проверки”, если можно проверить индексом/агрегатом/ограничением.

Плохой паттерн:

```text
BEGIN;
SELECT ...;
HTTP call to another service (500ms..2s);
UPDATE ...;
COMMIT;
```

Хороший паттерн:

```text
Сначала все внешние данные/решения вне транзакции (где можно)
BEGIN;
быстрые SELECT/UPDATE/INSERT
COMMIT;
публикация события через outbox (если нужно)
```

#### 4.2) Что значит “без внешних side‑effects”

Сайд‑эффект — это действие, которое нельзя “откатить” обычным ROLLBACK:

- отправка письма,
- списание денег в платёжке,
- HTTP‑запрос в другой сервис,
- публикация сообщения в брокер.

Если ты сделаешь это внутри транзакции, а потом получишь `serialization_failure`, ты повторишь транзакцию и **сайд‑эффект будет дважды**.

Выходы:

- делать сайд‑эффекты идемпотентными (идемпотентный ключ),
- или использовать outbox (см. §219),
- или переносить сайд‑эффект после коммита и принимать, что между коммитом и сайд‑эффектом будет окно (а это опять приводит к outbox как к “правильному” решению).

### Пошагово: правильная retry-логика (практический шаблон)

Шаблон мысли:

1) Выполняем транзакцию.  
2) Если получили `serialization_failure` (или эквивалент), **rollback** и **повтор**.  
3) Ограничиваем число попыток и добавляем jitter/backoff, чтобы не сделать “шторм повторов”.

Псевдокод:

```text
for attempt in 1..max_attempts:
  begin
    выполнить все SQL
    commit
    return успех
  catch serialization_failure:
    rollback
    sleep(backoff_with_jitter)
    continue
throw "слишком много конфликтов"
```

**Ключевой момент:** повторять нужно **всю транзакцию**, а не отдельный запрос.

### Retry “по-взрослому”: backoff + jitter + классификация ошибок

Чтобы retry не превращался в DDoS на БД, делай так:

1. Повторяй только ошибки, которые действительно требуют retry (для PostgreSQL SERIALIZABLE это обычно SQLSTATE `40001`).
2. Делай паузу, растущую с попытками (exponential backoff).
3. Добавляй jitter (случайность), чтобы 100 потоков не повторяли одновременно.

Пример схемы задержек:

- attempt 1: 0–10ms
- attempt 2: 10–30ms
- attempt 3: 30–80ms
- attempt 4: 80–200ms

Идея: чем больше конфликт, тем больше “разъехаться во времени”.

### Мини‑алгоритм SSI в голове: как появляются rw‑конфликты (очень упрощённо)

Здесь важно не выучить академическую теорию, а понять “почему БД откатывает”.

Если одна транзакция **читала** то, что другая потом **записала**, возникает зависимость.

Обозначения:

- `T1 →rw T2` означает: T1 читала данные, которые T2 записала (или наоборот — суть в “read/write связь”).

Если из таких связей получается “опасная структура” (упрощённо: потенциальный цикл), SSI решает: “лучше отменю одну транзакцию, чем дам неправильный итог”.

Это похоже на правило безопасности: “если я вижу, что два действия могут привести к логической аварии — я отменю одно”.

### SSI “внутри PostgreSQL”: predicate locks (SIREAD), pivot и dangerous structures

Это тот слой, который делает SSI “магией”, но его можно понять.

#### 1) Почему SSI не может полагаться только на “конфликт по строке”

Write skew — это когда две транзакции не пишут одну и ту же строку. Поэтому классическое правило “кто первый записал строку — тот победил” не срабатывает.

Нужно ловить конфликт на уровне:

- “я читал диапазон/условие”
- “кто-то потом записал в этот диапазон/условие”

Именно поэтому SSI использует **predicate locks**.

#### 2) Что такое predicate lock (SIREAD) простыми словами

Predicate lock — это “замок не на строку, а на условие чтения”.

Когда транзакция в SERIALIZABLE читает:

- конкретную строку
- диапазон по индексу
- результат условия (“все строки где status='active'”)

PostgreSQL может поставить **SIREAD** (Serializable Isolation READ) “пометку”:

> “Эта транзакция прочитала вот этот набор/диапазон. Если кто-то потом запишет сюда так, что нарушит сериализуемость — надо будет отменить одну транзакцию.”

Важно: SIREAD **не блокирует** как `FOR UPDATE`. Это не “ждать”, это “отслеживать”.

#### 3) Какие операции создают predicate locks

В общих чертах:

- чтения, которые могут участвовать в логическом конфликте:
  - index range scans (диапазон по индексу)
  - чтения “по условию”
  - иногда — чтение конкретной записи тоже

Чем “шире” чтение (например, seq scan всей таблицы под SERIALIZABLE), тем больше потенциальных зависимостей и тем выше шанс конфликта/abort.

Поэтому практическое правило из §215 (“не сканируй таблицу под SERIALIZABLE без необходимости”) имеет очень конкретную причину: ты ставишь слишком широкий predicate lock.

#### 4) rw‑конфликт в терминах SSI

rw‑конфликт появляется, когда:

- T1 сделала SIREAD (прочитала некоторый набор)
- T2 сделала запись так, что она “попадает” в прочитанный набор T1

Это создаёт зависимость, которую SSI учитывает при поиске опасных структур.

#### 5) Dangerous structure и “pivot”

SSI ловит ситуации, которые могут привести к циклу сериализации. Одна из ключевых эвристик — обнаружить “опасную структуру” зависимостей.

В сообщениях PostgreSQL ты иногда увидишь:

- `Canceled on identification as a pivot`

“Pivot” — это транзакция, которая оказывается “в центре” опасной структуры, и система выбирает её в качестве жертвы (или выбирает другую, но смысл: кого-то нужно abort).

Опять же: это не “случайная ошибка”, а механизм сохранения гарантии.

#### 6) Что делать, если SSI часто abort’ит: инженерные варианты

Если `40001` слишком много:

- укоротить транзакции
- сузить чтения (индексы, фильтры, не делать seq scan)
- перенести “горячий инвариант” на:
  - уникальность
  - строку‑замок
  - `FOR UPDATE`
  - advisory lock

#### 7) Мини‑лаборатория: predicate lock “по диапазону” (идея)

Это сложно показать без глубокого инструментария, но идея такая:

- транзакция SERIALIZABLE делает чтение диапазона по индексу
- параллельная транзакция вставляет строку, которая попадает в этот диапазон
- одна из транзакций при коммите может получить `40001`

Это и есть конфликт “по предикату”, а не по одной строке.

### Практика: как писать транзакции так, чтобы SSI срабатывал реже

- **Делай транзакции короче** (самое важное).
- **Используй индексы**, чтобы чтения были быстрыми (меньше времени на конфликт).
- **Не делай “скан таблицы” внутри SERIALIZABLE**, если можно проверять условие по индексу или ограничением.
- **Для горячих инвариантов** иногда проще “материализовать инвариант” (одна строка‑замок) или `SELECT ... FOR UPDATE`, чем часто ловить `40001`.

### Простыми словами

SSI — это как если бы два человека параллельно редактировали документ на своих копиях. Система смотрит на то, какие куски кто читал/менял, и если понимает, что итог нельзя сделать корректным “как будто кто-то был первым”, она говорит одному: “перечитай и сделай заново”.

### Картинка в голове

Представь дорожное движение на перекрёстке без светофора. Большинство машин проезжает без остановки, но иногда возникает опасная ситуация, и регулировщик говорит одной машине “сдай назад и поезжай заново”. Это дешевле, чем постоянно держать всех на красном.

### Как запомнить

- **Serializable (SSI)**: иногда будет **abort**, это нормально.
- Ошибка сериализации лечится **retry транзакции**, а не “костылём в SQL”.
- Транзакция для SERIALIZABLE должна быть: **короткой**, **идемпотентной** по эффектам, без внешних side-effects или с Outbox.

### Проверь себя

1) Почему в SSI иногда нужна отмена транзакции?  
<details><summary>Ответ</summary>
Потому что при конкретном параллельном расписании чтений/записей возникает не-сериализуемая зависимость (потенциальный цикл). Чтобы сохранить гарантию SERIALIZABLE, СУБД должна не дать всем конфликтующим транзакциям закоммититься.
</details>

2) Почему retry должен повторять всю транзакцию, а не один запрос?  
<details><summary>Ответ</summary>
Потому что конфликт определяется совокупностью действий транзакции и снимком. Если повторить только один запрос, остальные действия могут остаться неконсистентными относительно нового снимка, а логика приложения может сломаться.
</details>

3) Назови два правила для кода, чтобы retry не создавал дублей.  
<details><summary>Ответ</summary>
Например: (1) не делать внешних побочных эффектов внутри транзакции без идемпотентности; (2) использовать outbox для публикации событий; (3) использовать идемпотентные ключи/дедупликацию.
</details>

### Запомните

SSI — это “правильный SERIALIZABLE без постоянных read-lock’ов”, но цена — **иногда повторить**. Если ты умеешь делать retry и не допускаешь внешних side-effects в транзакции — это очень мощный инструмент.

### Типичные ошибки

- **Ошибка: “SERIALIZABLE — значит просто включил и забыл”.**  
  Нет: обязательно нужен retry на `40001`. Без него пользователи увидят “случайные ошибки”.

- **Ошибка: “Поймал `40001`, повторю только последний запрос”.**  
  Нельзя: повторять надо всю транзакцию, потому что снимок и логика решения меняются.

- **Ошибка: “Сделаю внутри транзакции отправку письма/вызов платёжки”.**  
  При retry получишь дубли. Правильнее: outbox или идемпотентный ключ.

### Что будет, если…

- **Что будет, если поставить max_attempts=100 и без задержек?**  
  Ты устроишь шторм: десятки потоков будут бесконечно конфликтовать и повторять синхронно, а БД станет ещё медленнее.

### Мини‑лаборатория §215: “поймай `40001` и увидь retry”

Цель: увидеть, что SERIALIZABLE иногда **специально** отменяет одну транзакцию.

Идея эксперимента: сделаем таблицу “счётчик” и две транзакции будут пытаться принять решение на одном снимке.

1) Подготовка:

```sql
CREATE TABLE ssi_demo (
  id int PRIMARY KEY,
  value int NOT NULL
);
INSERT INTO ssi_demo(id, value) VALUES (1, 0);
```

2) **Сеанс A**:

```sql
BEGIN TRANSACTION ISOLATION LEVEL SERIALIZABLE;
SELECT value FROM ssi_demo WHERE id = 1;
-- подожди пару секунд, чтобы сеанс B успел сделать своё
UPDATE ssi_demo SET value = value + 1 WHERE id = 1;
COMMIT;
```

3) **Сеанс B** (запусти почти одновременно):

```sql
BEGIN TRANSACTION ISOLATION LEVEL SERIALIZABLE;
SELECT value FROM ssi_demo WHERE id = 1;
UPDATE ssi_demo SET value = value + 1 WHERE id = 1;
COMMIT;
```

В зависимости от тайминга одна из транзакций может упасть с `40001`. Это нормально: БД говорит “повтори”.  
Если не поймал с первого раза — повтори эксперимент несколько раз (SSI не “обязан” падать всегда, это зависит от конкретного расписания).

### FAQ §215 (самые тупые вопросы, которые реально мешают понимать)

<details><summary>Почему PostgreSQL просто не “подождёт”, вместо того чтобы отдавать ошибку `40001`?</summary>
Если “просто ждать”, то мы по сути вернёмся к жёстким блокировкам чтения (как в S2PL) и убьём параллелизм. SSI выбрал другую стратегию: большинство транзакций проходит быстро, а редкие “опасные” откатываются и повторяются.
</details>

<details><summary>Почему нельзя сделать retry 100 раз, чтобы “точно получилось”?</summary>
Потому что при конфликте 100 потоков будут повторять синхронно и только усиливать конфликт, создавая шторм. Правильнее: 2–5 попыток + backoff+jitter.
</details>

<details><summary>Если транзакцию откатили, значит данные “потерялись”?</summary>
Нет. Откатили именно твою попытку, чтобы не допустить неправильного итогового состояния. Ты повторяешь, и новая попытка либо успешно выполнится, либо снова будет откат (если конфликт сохраняется).
</details>
