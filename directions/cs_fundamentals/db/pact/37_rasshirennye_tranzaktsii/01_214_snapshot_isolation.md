## §214. Snapshot isolation

### Цель раздела

Понять, что такое **снимок (snapshot)** и **Snapshot Isolation (SI)**, какие гарантии она даёт, почему она быстрая и почему при этом **не равна сериализуемости**, а также научиться узнавать и предотвращать ключевую аномалию SI — **write skew**.

### В этом разделе главное (если читать совсем тяжело — прочитай только это)

1) **SI = “мир заморожен” на начало транзакции.** Ты читаешь одну и ту же картину данных, даже если другие уже закоммитили новое.  
2) SI защищает от грязных чтений, но **не гарантирует “как будто транзакции шли по очереди”** (не гарантирует serializable).  
3) Самая опасная логическая ошибка под SI — **write skew**: две транзакции читают один снимок и пишут в разные строки так, что **инвариант ломается**.  
4) Лечится: либо **SERIALIZABLE (SSI) + retry**, либо **делать конфликт явным** (блокировки `FOR UPDATE`, уникальные ограничения, “одна строка-инвариант”, счётчики/статусы).

### Термины

| Термин | Определение |
|---|---|
| **MVCC** | Многоверсионность: обновления создают новые версии строк, а видимость определяется снимком. |
| **Snapshot** | Набор “видимых” транзакций/версий для данного момента. |
| **Consistent read** | Консистентное чтение: все SELECT внутри транзакции видят одно согласованное состояние. |
| **SI** | Изоляция на основе снимка: чтения не блокируют записи, записи не блокируют чтения (в общем случае). |
| **Write skew** | Две транзакции читают один и тот же снимок и пишут в разные строки так, что глобальный инвариант нарушается. |
| **First-committer-wins** | Правило “первый коммит победил”: два конкурирующих обновления одной и той же строки не могут оба успешно зафиксироваться. |

### Теория и правила

#### 0) Микро‑карта уровней изоляции (очень упрощённо)

Чтобы не путаться:

- **Read Committed (RC)**: каждый `SELECT` может видеть **новую** картину (новый снимок на каждый запрос).  
  Простыми словами: “в одной транзакции цифры могут прыгать между SELECT’ами”.

- **Snapshot Isolation / Repeatable Read (SI‑подобное)**: **один снимок на всю транзакцию**.  
  Простыми словами: “цифры внутри транзакции не прыгают”.

- **Serializable (SSI в PostgreSQL)**: как SI по чтению, **но** если параллельность могла бы дать неправильный (не‑сериализуемый) результат, СУБД **откатывает** одну транзакцию (будет `serialization_failure`) → нужен **retry**.

Эта часть про “средний слой”: SI (быстро и удобно) и почему иногда нужно подниматься до SERIALIZABLE.

#### 1) Что значит “снимок” на практике

В MVCC каждая строка имеет версии. Когда транзакция делает `SELECT`, она не “останавливает мир” и не ставит read lock на каждую строку. Вместо этого она получает **снимок видимости**: какие транзакции считаются “уже произошедшими”, какие — “ещё нет”.

Интуитивно:

- “видимые” версии — те, что были зафиксированы до момента снимка;
- “невидимые” — то, что закоммичено позже или ещё не закоммичено.

Главное: **внутри SI снимок фиксирован**, поэтому повторный `SELECT` (без дополнительных блокировок) будет видеть то же самое.

#### 2) SI и уровни изоляции в реальных СУБД

Важно различать:

- **Repeatable Read** в PostgreSQL — это по сути **Snapshot Isolation** (снимок на начало транзакции).
- **Serializable** в PostgreSQL — это **SSI** (см. §215), то есть SI + обнаружение опасных rw-конфликтов.
- В InnoDB (MySQL) **REPEATABLE READ** тоже основан на снимке (consistent read), но есть свои нюансы с “next-key locks” при определённых запросах.

В этой части мы говорим про идею SI как модели и про то, как она реализуется в популярных системах.

#### 3) Что SI гарантирует (и почему это быстро)

**Гарантии:**

- **Повторяемое чтение**: один и тот же `SELECT` в одной транзакции видит одну и ту же картину (если ты не используешь блокирующие чтения вроде `SELECT ... FOR UPDATE`).
- **Нет грязных чтений (dirty reads)**: ты не видишь незакоммиченные данные других транзакций.
- **Конфликт на одну и ту же строку**: если две транзакции пытаются записать одну и ту же строку, не получится “оба закоммитились” — обычно сработает “first-committer-wins” и одна получит ошибку/ожидание.

**Почему быстро:**

- Чтения не блокируют записи → меньше ожиданий.
- Чтения работают по версиям → хорошая параллельность.

#### 4) Чего SI НЕ гарантирует: SI ≠ serializable

Сериализуемость означает: результат параллельного выполнения эквивалентен некоторому последовательному порядку транзакций.

SI может нарушать сериализуемость без прямого “конфликта по строке”. Самая известная аномалия — **write skew**.

#### 4.1) Список аномалий: что точно не перепутать (простыми словами)

Ниже — “шпаргалка”, чтобы ты мог быстро понять, что именно может/не может случиться под SI.

- **Dirty read (грязное чтение)**: увидеть незакоммиченное.  
  Под SI **нет** (и в RC тоже обычно нет).

- **Non-repeatable read (неповторяемое чтение)**: один и тот же `SELECT` в транзакции возвращает разные данные.  
  Под SI **нет** (потому что один снимок).

- **Phantom (фантом)**: “вдруг появились строки, которые удовлетворяют условию”.  
  Под SI в классическом смысле “внутри транзакции” фантомов обычно **нет** для обычных SELECT (снимок фиксирован).  
  Но важно: если ты делаешь “проверку условия” и потом “вставку/обновление”, всё равно могут быть логические гонки (см. write skew и случаи с `INSERT`).

- **Lost update (потерянное обновление)**: две транзакции читают одно значение, обе пишут новое, и одна “затирает” другую.  
  Во многих MVCC‑СУБД это **предотвращается** конфликтом на одну строку (нужно понимать конкретную СУБД и запрос).  
  Но на уровне бизнес‑логики “потерянное обновление” часто всё равно возникает, если приложение делает read‑modify‑write без условий (см. ниже практику).

- **Write skew (перекос записи)**: главная боль SI.  
  Под SI **возможен** и именно он делает SI ≠ serializable.

Если ты запомнишь только одно: **под SI логические инварианты могут ломаться**, даже когда “никто никого не трогал в одну строку”.

#### 4.2) Практический пример “потерянного обновления” и как его избежать

Сценарий: у пользователя есть счётчик `balance`. Две транзакции одновременно делают “прибавить 10”.

Плохой способ (read‑modify‑write в приложении):

```sql
-- T1
BEGIN;
SELECT balance FROM accounts WHERE id = 1; -- допустим, 100
-- приложение вычисляет 110
UPDATE accounts SET balance = 110 WHERE id = 1;
COMMIT;

-- T2 (параллельно)
BEGIN;
SELECT balance FROM accounts WHERE id = 1; -- тоже 100 (по своему снимку)
UPDATE accounts SET balance = 110 WHERE id = 1;
COMMIT;
```

Итог: должно быть 120, стало 110.

Как правильно (атомарный update):

```sql
UPDATE accounts
SET balance = balance + 10
WHERE id = 1;
```

Или “оптимистично” через сравнение версии/значения:

```sql
UPDATE accounts
SET balance = :new_balance, version = version + 1
WHERE id = 1 AND version = :old_version;
-- если обновилось 0 строк → кто-то уже изменил, надо перечитать и повторить
```

Смысл: если ты делаешь изменение вида “старое значение → новое значение”, старайся делать это **в одном SQL**, а не “прочитал в приложении → посчитал → записал”.

#### 5) Как выглядит работа SI по шагам (на примере PostgreSQL)

Разберём прям по шагам, что происходит, когда клиент начинает транзакцию под уровнем `REPEATABLE READ` (SI‑подобное поведение в PostgreSQL):

1. Клиент посылает `BEGIN;`.
2. PostgreSQL:
   - регистрирует новую транзакцию с внутренним идентификатором (XID);
   - фиксирует для неё **снимок**: какие другие транзакции уже закоммичены, какие активны;
   - “замораживает” границу видимости `xmin` для этой транзакции.
3. Клиент выполняет `SELECT ...`:
   - планировщик строит план;
   - при чтении каждой строки сервер смотрит на её версии (MVCC‑метаданные);
   - выбирает **ту версию**, которая видима в рамках снимка (XID версии попадает в диапазон “видимых” и не помечен как удалённый для данного снимка).
4. Клиент выполняет ещё один `SELECT` по тем же данным:
   - PostgreSQL **не строит новый снимок** (при `REPEATABLE READ`);
   - используется тот же снимок, что и в п.2 → те же видимые версии, даже если другие транзакции уже что‑то успели закоммитить.
5. Клиент делает `UPDATE`:
   - на уровне строки создаётся **новая версия** (новый XID версии);
   - старая версия остаётся в таблице (пока кто‑то может её видеть), но для будущих снимков она постепенно перестаёт быть видимой.
6. Клиент делает `COMMIT;`:
   - транзакция помечается как зафиксированная;
   - новые версии становятся видимыми для будущих снимков;
   - VACUUM позже сможет удалить старые версии, когда ни одна активная транзакция их уже не видит.

Ключевая мысль: **снимок создаётся один раз** (на начало транзакции) и живёт до `COMMIT`, а видимость строк определяется правилами MVCC относительно этого снимка.

### MVCC “внутри PostgreSQL” на уровне строк (xmin/xmax/ctid) — максимально подробно

Это тот слой, который обычно “прячут”, но он очень помогает перестать путаться.

#### 1) `ctid`: физический “адрес” версии строки

В PostgreSQL каждая версия строки физически лежит в конкретном месте в heap‑файле (страница + смещение). Это место можно увидеть через системный столбец `ctid`.

- `ctid` выглядит как `(page, offset)` — условно “на какой странице и в каком слоте”.
- Когда строку обновляют, PostgreSQL **не переписывает её на месте**, а создаёт **новую версию** (часто в другом месте) → у новой версии **другой `ctid`**.

Практика:

```sql
SELECT ctid, *
FROM some_table
WHERE id = 1;
```

После `UPDATE` у этой строки `ctid` обычно изменится, потому что это уже **другая версия**.

Важно: `ctid` — не стабильный идентификатор (не используйте как PK). Это “координаты версии”.

#### 2) `xmin` и `xmax`: кто создал и кто “убил” версию

У каждой версии строки (tuple) в PostgreSQL есть MVCC‑метаданные:

- `xmin` — XID транзакции, которая **создала** эту версию.
- `xmax` — XID транзакции, которая **пометила версию как удалённую/заменённую** (DELETE или UPDATE).

Интуитивно:

- если `xmax` “нет” (или считается невалидным) → версия “жива” для будущих снимков;
- если `xmax` есть и транзакция `xmax` закоммичена → версия “мертва” для новых снимков, но может быть видима для старых.

Ты можешь посмотреть это напрямую (для обучения):

```sql
SELECT
  id,
  xmin, xmax,
  ctid,
  *
FROM some_table
WHERE id = 1;
```

#### 3) Как UPDATE выглядит физически: цепочка версий

Когда ты делаешь:

```sql
UPDATE some_table SET x = x + 1 WHERE id = 1;
```

Внутри происходит (упрощённо):

1) старая версия получает `xmax = XID_текущей_транзакции` (то есть “её заменили”)
2) создаётся новая версия с `xmin = XID_текущей_транзакции`
3) между версиями есть “ссылка” (цепочка обновлений), чтобы система могла находить актуальную версию

Именно поэтому под MVCC:

- SELECT может читать старую версию (если она видима по снимку)
- другая транзакция может читать новую версию (если её снимок “позже”)

#### 4) Snapshot внутри PostgreSQL: xmin/xmax/xip (снимок как “границы и список активных”)

Снимок (snapshot) в PostgreSQL (упрощённо) — это:

- `xmin` (нижняя граница): всё, что **младше** этой границы, считается “слишком новое/не видно”
- `xmax` (верхняя граница): всё, что **старше** этой границы, точно “попадает в прошлое”
- `xip[]` — список XID активных транзакций на момент снимка (их изменения не видны)

Модель: “я считаю видимым всё закоммиченное до снимка, кроме того, что принадлежит транзакциям, которые ещё активны”.

#### 5) Правила видимости версии строки “построчно” (самое важное)

Когда транзакция с некоторым snapshot читает версию строки, она примерно делает так:

**Шаг A: проверка `xmin` (создателя)**

- Если транзакция `xmin` не закоммичена → версия не видима.
- Если `xmin` закоммичена **после** твоего snapshot → версия не видима.
- Если `xmin` закоммичена **до** твоего snapshot → идём дальше.

**Шаг B: проверка `xmax` (удалителя/заменителя)**

- Если `xmax` отсутствует/невалиден → версия видима (она “живая”).
- Если `xmax` принадлежит транзакции, которая закоммичилась **до** твоего snapshot → версия не видима (она “уже удалена/заменена”).
- Если `xmax` закоммичилась **после** твоего snapshot или ещё не закоммичилась → версия видима (для тебя она ещё не удалена).

Эта логика и создаёт эффект:

- старые транзакции видят “старые версии”
- новые транзакции видят “новые версии”

#### 6) Как это связано с bloat, VACUUM и freeze

Теперь главный мост к §220:

- UPDATE/DELETE создаёт “мёртвые версии” (версии, которые не видны новым снимкам)
- VACUUM может физически убрать версию только когда уверен, что **ни один snapshot** больше её не увидит
- если есть long‑running транзакция (старый snapshot), она удерживает горизонт → VACUUM не может убрать старые версии → bloat растёт

**Freeze (заморозка):** у PostgreSQL есть отдельная задача — предотвращать “переполнение” XID (wraparound). Для этого старые `xmin` могут быть “заморожены” (обозначены как очень старые/всегда видимые), но это тоже завязано на то, чтобы не было транзакций, которые удерживают слишком старый горизонт.

#### 7) Мини‑лаборатория: наблюдаем xmin/xmax/ctid до/после UPDATE

```sql
DROP TABLE IF EXISTS mvcc_demo;
CREATE TABLE mvcc_demo(id int PRIMARY KEY, value int NOT NULL);
INSERT INTO mvcc_demo VALUES (1, 10);

SELECT id, value, xmin, xmax, ctid FROM mvcc_demo WHERE id = 1;

UPDATE mvcc_demo SET value = value + 1 WHERE id = 1;

SELECT id, value, xmin, xmax, ctid FROM mvcc_demo WHERE id = 1;
```

Что ты увидишь:

- `ctid` обычно изменится
- `xmin/xmax` поменяются (потому что это уже новая версия)

#### 6) Ещё один пример: отчёты “на вчера” и почему SI удобно для аналитики

Во многих системах отчёты строят так: “покажи состояние системы на конец вчерашнего дня”.

Под SI можно сделать:

```sql
BEGIN TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- Фиксируем "точку во времени" на уровне транзакции
SET LOCAL report_date = '2024-12-31'::date;

-- Все запросы ниже видят согласованную картину на момент начала транзакции
SELECT SUM(amount) FROM payments WHERE created_at < '2025-01-01';
SELECT COUNT(*) FROM users WHERE created_at < '2025-01-01';
SELECT status, COUNT(*) FROM orders WHERE created_at < '2025-01-01' GROUP BY status;

COMMIT;
```

Пока транзакция выполняется, в таблицы могут прилетать новые платежи и заказы, но **отчёт видит мир “замороженным” на момент начала**. Это удобно, потому что:

- не нужны глобальные блокировки;
- можно запускать тяжёлые аналитические запросы, не мешая онлайн‑операциям;
- все части отчёта согласованы между собой (нет “прыгающих” цифр).

#### 7) “Блокирующее чтение” как инструмент: `SELECT ... FOR UPDATE` / `FOR SHARE`

Иногда тебе нужно не просто “увидеть снимок”, а **временно запретить другим менять то, что ты сейчас проверяешь**.

Тогда вместо обычного `SELECT` используют “блокирующее чтение”:

- `SELECT ... FOR UPDATE` — блокирует строки так, что другой `UPDATE/DELETE` этих строк будет ждать.
- `SELECT ... FOR SHARE` (или аналоги) — “мягче”: блокирует от удаления/изменения (детали зависят от СУБД).

Зачем это надо под SI:

- SI даёт красивую картину, но логический инвариант может ломаться (write skew).
- Если инвариант завязан на набор строк, ты можешь **взять блокировку на этот набор** и сделать решение “в очереди”.

Важно: `FOR UPDATE` помогает только если ты **действительно блокируешь те строки**, которые определяют инвариант. Если инвариант “про сумму по таблице”, это сложнее — тогда лучше SERIALIZABLE (SSI) или “материализовать инвариант” (см. ниже).

#### 8) Практические способы “прибить инвариант”, чтобы write skew не происходил

Подходов много, вот самые понятные.

**Способ 1: SERIALIZABLE (SSI) + retry**  
Самый универсальный: СУБД сама ловит опасную параллельность. Цена: иногда будет retry.

**Способ 2: сделать конфликт физическим (чтобы он стал “по одной строке”)**  
Например:

- завести строку “инвариант” (одна запись), которую обновляют все конкурирующие транзакции;
- завести счётчик/агрегат в отдельной таблице и обновлять его (тоже один объект конфликтов).

**Способ 3: взять блокировку на “ресурс инварианта”**  
Например, если инвариант привязан к конкретной смене/складу/заказу, можно блокировать строку “смена/склад/заказ”:

```sql
BEGIN;
SELECT * FROM shift WHERE shift_id = :id FOR UPDATE; -- один “замок” на инвариант
-- дальше проверяем людей на смене и обновляем
COMMIT;
```

**Способ 4: использовать ограничения, которые превращают гонку в ошибку**  
Классика: уникальный индекс (`UNIQUE`) делает невозможным “две активные записи одного типа”.

Пример: “у пользователя может быть только один активный токен”.

```sql
CREATE UNIQUE INDEX ux_one_active_token
ON tokens(user_id)
WHERE is_active = true;
```

Теперь две параллельные транзакции не смогут обе вставить “активный токен” — одна упадёт с конфликтом уникальности.

Этот способ особенно хорош, потому что:

- прост;
- быстрый;
- ошибки легко обрабатывать (retry/показать пользователю).

#### 9) Большой блок “как проектировать инварианты под SI, чтобы не страдать”

Это один из самых важных практических навыков. Он звучит сложно, но идея простая:

> Если правило бизнеса (инвариант) важно — сделай так, чтобы БД могла **физически** поймать конфликт, а не оставляла это “на честное слово” двух параллельных транзакций.

Ниже — 6 самых частых шаблонов (pattern’ов).

##### Pattern 1: Инвариант = уникальность (самый простой и лучший)

Если твой инвариант можно выразить как “не бывает двух одинаковых активных X”, то это идеально.

Примеры:

- “у пользователя 1 активная сессия”
- “у заказа 1 активная доставка”
- “у комнаты 1 активная бронь в конкретный момент”

Решение: `UNIQUE` (иногда частичный индекс).

```sql
-- “у пользователя может быть только один активный токен”
CREATE UNIQUE INDEX ux_token_one_active
ON tokens(user_id)
WHERE is_active = true;
```

Почему это круто:

- конфликт превращается в явную ошибку (уникальность);
- нет скрытых логических багов;
- retry/обработка ошибки понятны.

##### Pattern 2: Инвариант = “нет пересечений во времени” (диапазоны / исключающие ограничения)

Пример инварианта: “в одной комнате нельзя иметь две брони, которые пересекаются по времени”.

В PostgreSQL это удобно делать через **exclusion constraint** (если ты используешь range‑типы):

```sql
-- Идея: у одной комнаты периоды не пересекаются
-- (потребуется расширение btree_gist для некоторых типов)
-- CREATE EXTENSION IF NOT EXISTS btree_gist;

-- booking(room_id, period tstzrange)
ALTER TABLE booking
ADD CONSTRAINT booking_no_overlap
EXCLUDE USING gist (
  room_id WITH =,
  period WITH &&
);
```

Смысл: оператор `&&` означает “пересекается”. БД запрещает пересечение автоматически.

##### Pattern 3: Инвариант “по сумме” → материализуй “замок” в одной строке

Если правило проверяется “по сумме” (COUNT/SUM), часто проще завести “строку‑инвариант”.

Пример: “в смене всегда >= 1 врач”.

Можно завести таблицу `shift(shift_id, on_call_count)` и обновлять `on_call_count` в тех же транзакциях:

- изменения “распределены” по двум врачам,
- но инвариант теперь проходит через **одну строку shift**, значит конфликт становится физическим.

##### Pattern 4: Явная сериализация через `SELECT ... FOR UPDATE` на ресурсе

Если есть “ресурс”, вокруг которого крутится инвариант (заказ, смена, кошелёк, склад), можно:

```sql
BEGIN;
SELECT * FROM resource WHERE id = :id FOR UPDATE;
-- теперь все операции на этот ресурс становятся “в очереди”
-- проверка инварианта
-- изменение доменных таблиц
COMMIT;
```

Это простой способ сделать “один за другим”, но помни:

- увеличиваются ожидания (это осознанная цена);
- держи транзакцию короткой.

##### Pattern 5: “Оптимистичная конкуренция” через версию (CAS‑подход)

Когда изменение выглядит как “взял старое → посчитал новое → записал”, делай это с версией:

```sql
UPDATE item
SET value = :new_value,
    version = version + 1
WHERE id = :id AND version = :old_version;
```

Если обновилось 0 строк — значит кто‑то уже успел обновить, делай retry с перечитыванием.

##### Pattern 6: Advisory locks (когда модель сложная, а ресурс логический)

Иногда инвариант привязан не к строке таблицы, а к логическому ключу (“пользователь‑день”, “склад‑товар”, “пара сущностей”).

В PostgreSQL можно использовать advisory lock:

```sql
-- Пример: “запираем” операции по user_id
SELECT pg_advisory_xact_lock(hashtext(:user_id::text));
```

Это делает сериализацию “по ключу” без явной строки‑замка. Но:

- это уже “инструмент инженера”, используй осознанно;
- всё равно держи транзакции короткими.

##### Главный вывод по инвариантам

Если ты видишь, что инвариант проверяется `SELECT COUNT(*) ...` и потом отдельными `UPDATE` по разным строкам — это красный флаг.  
Либо SERIALIZABLE (SSI) + retry, либо один из pattern’ов выше.

### Пошагово: write skew на примере (инвариант “всегда есть хотя бы один врач на смене”)

Схема:

- Таблица `on_call(doctor_id, is_on_call)`.
- Инвариант: **нельзя**, чтобы в итоге `COUNT(is_on_call=true) = 0`.

Начальное состояние:

- Доктор A на смене
- Доктор B на смене

Две транзакции одновременно решают “уйти со смены”, проверяя, что “кроме меня есть кто-то ещё”.

Транзакция T1:

```sql
BEGIN;
SELECT COUNT(*) FROM on_call WHERE is_on_call = true; -- видит 2
-- “окей, уйду, ведь кто-то останется”
UPDATE on_call SET is_on_call = false WHERE doctor_id = 'A';
COMMIT;
```

Транзакция T2 (параллельно):

```sql
BEGIN;
SELECT COUNT(*) FROM on_call WHERE is_on_call = true; -- тоже видит 2 (тот же снимок!)
UPDATE on_call SET is_on_call = false WHERE doctor_id = 'B';
COMMIT;
```

Если каждая обновляет **разные строки**, прямого конфликта “одна строка — две записи” нет, и под SI обе могут успешно закоммититься. Итог: на смене 0 врачей — инвариант нарушен.

Это и есть **write skew**: конфликт “логический”, но не “по одной и той же строке”.

Обрати внимание на важный нюанс: если бы обе транзакции изменяли **одну и ту же строку**, то под SI одна из них получила бы конфликт при коммите (first‑committer‑wins) и не смогла бы завершиться — инвариант бы сохранился. Проблема именно в том, что они трогают **разные строки**, а инвариант проверяется “в сумме”.

### Ещё один пример write skew (про лимит “не больше 1 активного админа”)

Инвариант: в системе должен быть **ровно один** активный админ (или “не больше одного”).

Если модель сделана так:

- таблица `users(id, is_admin, is_active)`
- инвариант проверяется запросом `COUNT(*) WHERE is_admin AND is_active`

Две транзакции могут одновременно:

- прочитать “активный админ есть”,
- деактивировать текущего,
- активировать нового,

И на короткое время “админов = 0” или “админов = 2”, в зависимости от порядка.

Правильные решения обычно такие:

- материализовать “текущего админа” в одной строке (таблица `system_settings` с `current_admin_id`) и блокировать/обновлять её;
- или сделать ограничение/уникальность на “роль админ активен” (если модель допускает);
- или SERIALIZABLE (SSI) + retry.

### Как предотвращать аномалии SI

Есть несколько подходов — выбирай по ситуации.

#### Подход A: поднять уровень изоляции до SERIALIZABLE (и делать retry)

В PostgreSQL `SERIALIZABLE` реализован через SSI и как раз ловит такие случаи (см. §215). Плюсы: просто для разработчика. Минусы: иногда будут `serialization_failure`, нужна retry-логика.

#### Подход B: “прибить инвариант” на уровне БД (constraints / locks / materialized invariant)

Инвариант “хотя бы один врач” нельзя выразить обычным `CHECK` на строке. Но можно сделать так, чтобы конфликт стал “по одной строке”, например:

- завести строку “счетчик активных” и обновлять её (это создаёт конфликт на одну строку);
- использовать явную блокировку: `SELECT ... FOR UPDATE` по набору строк, который отвечает за инвариант;
- использовать `SERIALIZABLE` именно для таких транзакций.

#### Подход C: изменить модель данных

Например, вместо флага `is_on_call` использовать сущность “смена” и отдельную запись “назначен ответственный” (одна строка) — снова переводим конфликт к одной строке.

### Простыми словами

Snapshot Isolation — это как читать книгу в библиотеке по копии, которая не меняется, пока ты читаешь. Другие могут приносить новые издания, но твоя копия та же.  
Проблема: два человека могут прочитать одинаковую копию и **оба принять решения**, которые вместе ломают “правило мира”, если это правило не сводится к конфликту по одной строке.

### Картинка в голове

Представь, что снимок — это “замороженный кадр” базы. Ты смотришь на кадр и принимаешь решение. Если два человека смотрят один и тот же кадр, они могут **согласованно ошибиться**, потому что не видят решений друг друга.

### Как запомнить

- **SI = один снимок на транзакцию** (обычно).
- **SI защищает от грязи**, но **не гарантирует сериализуемость**.
- Опасность SI: **write skew** (логический инвариант, нет конфликта по строке).

### Проверь себя

1) Почему under SI “чтения не блокируют записи”?  
<details><summary>Ответ</summary>
Потому что чтение работает не через удержание read-lock’ов на строках, а через MVCC-видимость: транзакция выбирает версию строки, которая видима в её снимке.
</details>

2) Почему write skew возможен, если “две транзакции не обновляют одну и ту же строку”?  
<details><summary>Ответ</summary>
Потому что конфликт не по физической строке, а по логическому инварианту. Каждая транзакция принимает решение на одном и том же снимке и обновляет разные строки, поэтому “first-committer-wins” не срабатывает.
</details>

3) Назови два способа бороться с write skew.  
<details><summary>Ответ</summary>
Например: (1) использовать `SERIALIZABLE` (SSI) и retry-логику; (2) делать блокирующие чтения/явные блокировки (`SELECT ... FOR UPDATE`) на наборе строк, отвечающих за инвариант; (3) перевести инвариант на одну конфликтующую строку (счётчик/“главная запись”).
</details>

### Запомните

Snapshot Isolation — отличный компромисс “скорость/удобство”, но под SI **логические инварианты** могут рушиться без явных конфликтов. Если инварианты важны — либо **делай конфликт явным**, либо используй **SSI/Serializable**.

### Типичные ошибки

- **Ошибка: “SI значит всё безопасно, ведь чтение консистентное”.**  
  Консистентное чтение ≠ сериализуемость. Можно получить красивый, но логически неправильный результат.

- **Ошибка: “Проверю инвариант SELECT’ом, а потом UPDATE’ом, и всё ок”.**  
  Под SI две транзакции могут проверить одно и то же и обе принять решение. Если инвариант не сводится к одной строке — будет write skew.

### Что будет, если…

- **Что будет, если в web‑приложении держать транзакцию открытой, пока идёт внешний HTTP‑вызов?**  
  Транзакция станет длинной, повысится шанс конфликтов/блокировок, начнёт копиться мусор версий, VACUUM будет страдать.

- **Что будет, если “обработать конфликт” просто повтором одного UPDATE?**  
  Можно получить неконсистентность: повторить нужно всю транзакцию (см. SSI/retry), иначе часть логики останется на старом снимке.

### Мини‑лаборатория §214 (если хочется почувствовать руками)

Если у тебя есть доступ к PostgreSQL, можно воспроизвести “замороженный снимок”.

**Сеанс A:**

```sql
BEGIN TRANSACTION ISOLATION LEVEL REPEATABLE READ;
SELECT COUNT(*) FROM orders;
-- НЕ КОММИТЬ
```

**Сеанс B (параллельно):**

```sql
INSERT INTO orders(id, ...) VALUES (...);
COMMIT;
```

**Сеанс A (снова):**

```sql
SELECT COUNT(*) FROM orders;  -- увидишь то же число, что и до вставки (внутри одного снимка)
COMMIT;
```

Идея: под SI (Repeatable Read) второй `SELECT` в транзакции A обычно не увидит вставку из B, потому что A живёт на своём снимке.

### Мини‑лаборатория §214.2: воспроизводим write skew “два врача ушли”

Цель: увидеть, что под SI/Repeatable Read можно сломать инвариант, если он “не по одной строке”.

1) Подготовка:

```sql
DROP TABLE IF EXISTS on_call;
CREATE TABLE on_call (
  doctor_id text PRIMARY KEY,
  is_on_call boolean NOT NULL
);
INSERT INTO on_call(doctor_id, is_on_call)
VALUES ('A', true), ('B', true);
```

2) **Сеанс A** (Repeatable Read):

```sql
BEGIN TRANSACTION ISOLATION LEVEL REPEATABLE READ;
SELECT COUNT(*) FROM on_call WHERE is_on_call = true; -- 2
UPDATE on_call SET is_on_call = false WHERE doctor_id = 'A';
-- НЕ КОММИТЬ, пока сеанс B не сделает UPDATE
```

3) **Сеанс B** (почти одновременно, тоже Repeatable Read):

```sql
BEGIN TRANSACTION ISOLATION LEVEL REPEATABLE READ;
SELECT COUNT(*) FROM on_call WHERE is_on_call = true; -- тоже 2 (свой снимок)
UPDATE on_call SET is_on_call = false WHERE doctor_id = 'B';
COMMIT;
```

4) **Сеанс A**:

```sql
COMMIT;
SELECT COUNT(*) FROM on_call WHERE is_on_call = true; -- получится 0
```

Итог: инвариант “хотя бы один врач” сломан.  
Теперь повтори то же самое, но с `SERIALIZABLE` — одна транзакция должна упасть с `40001` и потребует retry.

### PostgreSQL vs MySQL (InnoDB): где чаще путаются про “Repeatable Read”

Очень грубая, но полезная памятка:

- **PostgreSQL `REPEATABLE READ`**: это “чистый” снимок на транзакцию (SI‑подобное чтение). Логические инварианты могут ломаться (write skew), если не использовать SERIALIZABLE/locks/constraints.
- **MySQL/InnoDB `REPEATABLE READ`**: тоже есть consistent read по снимку, но при **locking reads** (`SELECT ... FOR UPDATE`, `LOCK IN SHARE MODE`) включаются **next-key locks** (gap locks), из‑за чего поведение вокруг “фантомов” и конкурентных вставок может отличаться.

Главная мысль: одинаковое название уровня ≠ одинаковое поведение во всех деталях. Если инвариант критичен — проектируй его явно (constraints/locks) или поднимайся к сериализуемости.
