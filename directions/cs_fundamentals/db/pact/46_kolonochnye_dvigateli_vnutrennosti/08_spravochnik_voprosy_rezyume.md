## Справочник по части (ключевые пункты)

### Механизмы кодирования и исполнения

| Тема                   | Ключевые мысли                                                                                                          |
| ---------------------- | ----------------------------------------------------------------------------------------------------------------------- |
| RLE                    | Отлично сжимает длинные последовательности одинаковых значений; плохо работает при частой смене значений.              |
| Dictionary encoding    | Эффективен при низкой кардинальности; хранит словарь + индексы; часто комбинируется с RLE/bit‑packing.                 |
| Delta / Frame‑of‑reference | Для числовых/упорядоченных колонок; жмут за счёт малых дельт; помогают кодекам и bit‑packing.                      |
| Bit‑packing            | Упаковывает числа в минимальное количество бит; хорошо работает при известном небольшом диапазоне значений.           |
| Vectorized execution   | Обрабатывает данные пакетами (векторами), снижает накладные расходы, улучшает кеш‑поведение.                           |
| SIMD                   | Выполняет одну инструкцию над несколькими значениями; идеально для плотных колонок и простых операций.               |
| Late materialization   | Откладывает сборку строк до конца; уменьшает объём данных в pipeline, улучшает производительность.                     |

### Layout и движки

| Тема              | Ключевые мысли                                                                                                    |
| ----------------- | ----------------------------------------------------------------------------------------------------------------- |
| Segments/row groups | Крупные блоки строк; единицы чтения/сжатия/параллелизма.                                                       |
| Zone maps         | Min/max и статистика по блокам; позволяют отбрасывать ненужные данные до чтения с диска.                         |
| Партиционирование | Логическое разделение таблицы (часто по дате); критично для pruning и управления жизненным циклом данных.       |
| ClickHouse        | MergeTree, parts, партиции, `ORDER BY`; материализованные представления; широкий набор кодеков.                 |
| Snowflake         | Micro‑partitions, clustering keys, automatic clustering; разделение storage/compute; богатые метаданные.        |
| Columnstore в SQL Server | Columnstore‑индексы поверх rowstore; batch mode; delta store для модификаций.                           |

### Оптимизация

| Тема               | Ключевые мысли                                                                                                      |
| ------------------ | ------------------------------------------------------------------------------------------------------------------- |
| Layout             | Первое, что нужно проверить: партиции, сортировка, clustering keys.                                                |
| Compression tuning | Баланс между размером и CPU; горячие данные — менее агрессивное сжатие, холодные — более агрессивное.             |
| Batch size         | Размер вектора влияет на эффективность векторизации и кеш‑поведение; обычно разумные дефолты хороши.             |
| Параллелизм        | Слишком много потоков = contention и thrashing; нужен баланс под железо и тип нагрузки.                           |
| Метрики            | Важно смотреть на scan bytes, CPU, latency (p50/p95/p99), стоимость запросов (в облаке).                          |

## Частые сценарии

### Сценарий 1. Медленные отчёты в ClickHouse по логам

- **Симптомы:** отчёты по логам за месяц по стране/типу события выполняются дольше ожидаемого, CPU загружен, но диски не “красные”.
- **Шаги:**
  1. Проверить layout таблицы:
     - `PARTITION BY` по месяцу?
     - `ORDER BY` по `(event_date, country, event_type)` или подобному ключу?
  2. Посмотреть, какие колонки реально используются в запросах:
     - убрать `SELECT *`, оставить только нужные;
  3. Проверить кодеки для горячих колонок (`event_type`, `country`, `amount`):
     - при необходимости заменить слишком тяжёлые на более быстрые;
  4. Сравнить scan bytes до и после изменений.

### Сценарий 2. Дорогие запросы в Snowflake

- **Симптомы:** счёт за Snowflake растёт, отдельные отчёты сканируют десятки гигабайт, хотя “логически” должны затрагивать только небольшой диапазон.
- **Шаги:**
  1. Посмотреть explain/PROFILE запросов:
     - сколько микропартиций сканируется;
  2. Проверить, по каким колонкам чаще всего фильтруют:
     - `event_date`, `region`, `product_id`?
  3. Настроить clustering key по этим колонкам;
  4. Дождаться automatic clustering и повторно измерить стоимость и время.

### Сценарий 3. Тяжёлые отчёты на SQL Server поверх OLTP‑таблиц

- **Симптомы:** отчёты по продажам/заказам на тех же таблицах, что использует OLTP‑система, выполняются медленно даже ночью.
- **Шаги:**
  1. Определить таблицы, которые чаще всего участвуют в отчётах;
  2. Для них рассмотреть:
     - `CLUSTERED COLUMNSTORE` (если таблица почти чисто аналитическая);
     - `NONCLUSTERED COLUMNSTORE` (если важен OLTP);
  3. Измерить эффект:
     - время выполнения отчётов;
     - нагрузка на CPU/диски;
  4. Следить за влиянием на write‑нагрузку (delta store).

### Проверь себя (по сценариям)

1. В сценарии с медленными отчётами в ClickHouse по логам почему первым шагом предлагается проверить `PARTITION BY` и `ORDER BY`, а не сразу менять кодеки?

<details><summary>Ответ</summary>
Потому что layout (`PARTITION BY`, `ORDER BY`) определяет, **какие данные вообще придётся читать** для запроса. Если партиционирование и сортировка “не дружат” с типичными фильтрами и группировками, движок будет сканировать слишком много сегментов независимо от кодеков. Сначала нужно убедиться, что таблица физически разложена под запросы, и только потом оптимизировать сжатие/кодеки; иначе можно улучшать второстепенный слой, игнорируя главный источник лишней работы.
</details>

2. В сценарии с дорогими запросами в Snowflake как clustering key помогает уменьшить стоимость, если железо и warehouse остаются теми же?

<details><summary>Ответ</summary>
Правильно выбранный clustering key улучшает **кластеризацию микропартиций** по полям, часто используемым в фильтрах (дата, регион, продукт и т.п.). Это позволяет Snowflake по статистике быстрее отсекать микропартиции, которые точно не содержат нужные данные, и вообще не читать их с облачного хранилища. В результате уменьшается объём сканируемых данных (scan bytes), а значит — сокращается и время, и стоимость запросов при том же самом размере warehouse’а.
</details>

3. Почему в сценарии с тяжёлыми отчётами на SQL Server поверх OLTP‑таблиц columnstore‑индекс рассматривается как первый шаг, а не немедленный перенос данных в отдельный колончатый DWH?

<details><summary>Ответ</summary>
Потому что columnstore‑индекс — это способ **добавить колончатые преимущества “на месте”**, без большой миграции и изменения архитектуры. Его можно внедрить точечно на несколько таблиц, измерить эффект и понять, насколько сильно отчёты страдали именно от отсутствия колончатого хранения и batch mode. Перенос в отдельный DWH дороже и сложнее организационно (ETL/CDC, отдельная инфраструктура, согласование с командами), поэтому columnstore‑индекс — разумный первый шаг, который даёт много выигрыша при сравнительно небольших изменениях.
</details>

## Вопросы для самопроверки по части

1. В чём принципиальное отличие того, как колончатые движки используют сжатие, от классических row‑ориентированных СУБД?

<details><summary>Ответ</summary>
Колончатые движки сжимают **каждую колонку отдельно**, используя кодировки, адаптированные под распределение значений в этой колонке (RLE, dictionary, delta и т.п.). В пределах сегмента значения одной колонки часто сильно коррелированы и однотипны, что даёт большой выигрыш по размеру и скорости. В row‑ориентированных СУБД сжатие обычно работает поверх строк или страниц, где разные типы данных перемешаны, что ограничивает эффективность специализированных кодировок.
</details>

2. Почему vectorized execution особенно важен для аналитических запросов, а не так критичен для типичных OLTP‑операций?

<details><summary>Ответ</summary>
Потому что аналитические запросы обрабатывают **большие объёмы данных** и выполняют над ними относительно простые, повторяющиеся операции (фильтры, агрегаты, join’ы). Векторизация позволяет резко уменьшить накладные расходы на обработку каждой строки и лучше использовать SIMD и кеш. В OLTP‑операциях запросы короткие, работают с небольшим числом строк и часто содержат сложную бизнес‑логику; выигрыш от векторизации там меньше, а накладные расходы на её реализацию могут не окупаться.
</details>

3. Как выбор `ORDER BY`/clustering key влияет на эффективность кодировок и zone maps в колончатых БД?

<details><summary>Ответ</summary>
Сортировка по ключам, которые часто участвуют в фильтрах и агрегациях, приводит к тому, что:
- значения внутри сегмента лежат в **узких диапазонах** (для чисел и дат) или образуют **длинные run’ы** (для статусов/категорий);
- min/max по сегментам становятся более информативными и позволяют агрессивнее отсекать блоки;
- кодировки (RLE, delta, Frame‑of‑reference) получают “идеальный материал” для сжатия.  
Небрежный выбор ORDER BY/кластера приводит к перемешиванию значений и ухудшению всех этих эффектов.
</details>

4. В каких случаях имеет смысл строить отдельный колончатый движок/хранилище (ClickHouse, Snowflake), а не ограничиваться columnstore‑индексами в существующей OLTP‑базе?

<details><summary>Ответ</summary>
Когда:
- объёмы данных велики (сотни гигабайт/терабайты);
- аналитическая нагрузка тяжёлая и постоянная (много отчётов, сложные срезы, интерактивный BI);
- нужно масштабироваться горизонтально или использовать облачное хранилище.  
Columnstore‑индексы хороши как “мост” и для умеренных объёмов, но полноценные колончатые движки дают:
- лучшую компрессию и производительность при масштабах DWH;
- гибкое управление кластером/облаком;
- богатый набор специализированных фич для аналитики.
</details>

5. Почему нельзя считать, что достаточно “перелить данные в колончатый движок”, и все проблемы с производительностью аналитики исчезнут?

<details><summary>Ответ</summary>
Потому что:
- плохой дизайн схемы, неудачные запросы (`SELECT *`, отсутствие нужных фильтров), отсутствие агрегатов и материализованных представлений никуда не денутся;
- колончатый движок всё ещё подчиняется тем же законам: layout, сжатие, параллелизм, ресурсы;
- без осмысленного планирования партиций, сортировки и кластеризации можно легко “убить” преимущества колончатой архитектуры.  
Колончатый движок — мощный инструмент, но он не заменяет инженерного проектирования.
</details>

6. Чем отличается подход к тюнингу колончатых БД в “железном” мире (on‑prem ClickHouse) и в облачном (Snowflake, BigQuery)?

<details><summary>Ответ</summary>
В on‑prem:
- ты напрямую управляешь железом: диски, CPU, память, сеть;
- тюнинг включает и уровень ОС/железа, и layout/кодеки.  
В облаке:
- “железо” скрыто, но ты платишь за **сканируемые данные и потреблённый compute**;
- фокус смещается на: layout (кластеризация, партиционирование), проектирование запросов, выбор размеров warehouse’ов/слотов.  
В обоих случаях важны принципы колончатых движков, но практические рычаги разные.
</details>

## Типичные ошибки по части

- **Считать колончатые БД “автоматически быстрыми”**, не думая о layout’е, сортировке и кластеризации.
- **Использовать `SELECT *` во всех аналитических запросах**, лишая движок возможности читать только нужные колонки.
- **Максимально сжимать все колонки**, не разделяя горячие и холодные данные.
- **Переоценивать auto‑magic в облачных DWH** (Snowflake/BigQuery) и забывать про clustering keys и статистику.
- **Вешать columnstore‑индексы на все большие таблицы в OLTP‑базе**, не измеряя влияния на write‑нагрузку.
- **Не измерять scan bytes и CPU**, делая выводы “на глаз” по времени одного запроса.

## Резюме части XLVI

В этой части мы заглянули **внутрь колончатых движков**: разобрали, как они кодируют данные (RLE, dictionary, delta, bit‑packing, Frame‑of‑reference), как выполняют запросы в векторном режиме с SIMD и late materialization, как организуют layout (segments, row groups, zone maps), и как эти идеи реализованы в конкретных системах — ClickHouse, Snowflake, columnstore‑индексах SQL Server.  
Мы увидели, что производительность колончатых БД — это результат **совместной работы кодировок, layout’а, vectorized execution и грамотного проектирования схем и запросов**, а не только “колоночного формата”.  
Освоив материал этой части, ты можешь **сознательно выбирать и настраивать колончатые движки под свои задачи**, понимать их ограничения и читать документацию/профилировку “между строк”, а не воспринимать их как чёрный ящик. В следующих частях (про большие данные и SQL поверх data lake) этот взгляд на колончатые системы станет основой для понимания Presto/Trino, Spark SQL, Hive/Impala, Parquet/ORC и архитектур на уровне всего data lake.
