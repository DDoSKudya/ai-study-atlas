## 281. Snowflake внутренности

### Цель раздела

Понять, **как Snowflake реализует колончатое хранение и исполнение в облаке**:  
что такое micro‑partitions, clustering keys, automatic clustering,  
как устроено разделение storage/compute,  
и как это влияет на проектирование схем и стоимости запросов.

### В этом разделе главное

1. Snowflake хранит данные в виде **micro‑partitions** — маленьких колончатых блоков с богатыми метаданными.
2. **Clustering keys** и automatic clustering управляют физическим расположением данных, даже если пользователь об этом не думает.
3. Storage и compute **жёстко разделены**: данные лежат в облачном хранилище, а compute‑кластеры поднимаются/опускаются отдельно.
4. Оптимизация в Snowflake — это часто **игра с кластеризацией, микропартициями и размерами warehouse’ов**, а не ручной тюнинг “железа”.
5. Понимание внутренних механизмов помогает **читать объяснения стоимости запросов** и избегать неожиданных счетов.

### Термины

- **Micro‑partition** — маленькая колончатая партиция (обычно десятки тысяч строк) с собственными статистиками.
- **Clustering key** — набор колонок, по которым Snowflake старается держать данные кластеризованными.
- **Automatic clustering** — фоновый процесс, который перераскладывает данные для поддержания кластеризации.
- **Virtual warehouse** — compute‑кластер для выполнения запросов.

### Теория и правила

#### 1) Micro‑partitions

В Snowflake:

- данные таблицы хранятся как множество **микропартиций**;
- каждая микропартиция содержит:
  - данные нескольких тысяч/десятков тысяч строк;
  - колончатое представление;
  - статистику по колонкам (min/max, distinct estimates, NULL‑count и т.п.).

При запросах Snowflake:

- ещё до чтения данных **анализирует метаданные микропартиций**;
- отбрасывает те, которые точно не содержат нужных значений (как zone maps, но богаче).

#### 2) Clustering keys и automatic clustering

По умолчанию:

- Snowflake сам решает, как разложить микропартиции;
- для многих сценариев этого достаточно.

Если нужно более точное управление:

- задаём **clustering key**:

```sql
ALTER TABLE events CLUSTER BY (event_date, user_id);
```

- Snowflake:
  - собирает статистику по существующим микропартициям;
  - в фоне **рекластеризует** данные;
  - поддерживает кластеризацию при последующих загрузках.

Это влияет на:

- эффективность pruning’а;
- использование кешей;
- стоимость запросов (меньше прочитанных микропартиций → меньше сканированного объёма).

#### 3) Storage vs compute

В Snowflake:

- **storage**:
  - хранится в облачном объектном хранилище (S3, GCS, Azure Blob);
  - относительно дешёвый, но с некото­рым latency.
- **compute**:
  - один или несколько виртуальных warehouse’ов;
  - масштабируются независимо (размер, количество).

Это значит:

- одна и та же таблица может обслуживаться разными кластерами (dev/prod, разные команды);
- оптимизация:
  - иногда проще **увеличить warehouse**, чем “копаться в железе”;
  - но в долгосрочной перспективе layout и кластеризация важнее.

### Пошагово

Упрощённый сценарий:

```sql
CREATE TABLE events (
    event_time TIMESTAMP,
    user_id    NUMBER,
    country    STRING,
    event_type STRING,
    amount     NUMBER
);
```

1. Загрузка данных (`COPY INTO events FROM ...`):
   - Snowflake разбивает входные файлы на микропартиции;
   - сохраняет статистику по колонкам.
2. Запрос:

```sql
SELECT user_id, sum(amount)
FROM events
WHERE event_time BETWEEN '2024-03-01' AND '2024-03-31'
  AND country = 'RU'
GROUP BY user_id;
```

3. Snowflake:
   - смотрит статистику микропартиций по `event_time` и `country`;
   - отбрасывает те, где:
     - интервал дат не пересекается с запросом;
     - `country` никогда не бывает `'RU'`;
   - оставшиеся микропартиции читаются, и на них уже включается vectorized execution.

### Простыми словами

Snowflake — это как **огромный облачный архив**, где:

- все документы разложены по маленьким коробкам (микропартициям);
- на каждой коробке написано **очень много подробностей** о том, что внутри (метаданные);
- за раскладку коробок отвечает **робот‑архивариус** (automatic clustering), который время от времени всё перекладывает, чтобы чаще нужные документы были рядом.

Ты платишь **за работу робота и за чтение коробок**, а не за сами полки.

### Картинка в голове

Представь:

- облачное хранилище — это **большой озеро коробок**;
- виртуальные warehouse’ы — это **команды работников**, которые плавают по озеру и достают коробки;
- микропартиции — это **маленькие коробки**, каждая с собственной аннотацией/карточкой;
- clustering keys — это “правила, по которым работники раскладывают коробки по озеру”.

### Как запомнить

- **Snowflake = “метаданные‑центричный колончатый DWH в облаке”**.
- **Micro‑partitions + clustering keys → основа для pruning’а и производительности**.
- **Storage/compute разделены, тюнинг — это скорее игра с layout и размерами warehouse’ов**.

### Примеры

#### Настройка кластеризации

```sql
ALTER TABLE events
CLUSTER BY (event_date, country, user_id);
```

- полезно, если типичные запросы:
  - фильтруют по дате и стране;
  - группируют по user_id.

#### Просмотр микропартиций (упрощённо, через системные представления)

Snowflake предоставляет представления, где можно посмотреть:

- диапазоны значений по колонкам;
- размеры микропартиций;
- степень кластеризации (как хорошо данные упорядочены по ключам).

### Практика / реальные сценарии

- **Data warehouse для продукта/маркетинга**:
  - крупные фактовые таблицы (события, клики, транзакции);
  - типичные запросы фильтруют по дате/региону/каналу;
  - грамотный clustering key значительно снижает объём сканируемых данных.
- **Многокомандное использование**:
  - один storage, несколько виртуальных warehouse’ов (prod, staging, ad‑hoc аналитика).

### Типичные ошибки

- Игнорировать clustering keys и надеяться, что “Snowflake сам всё сделает”:
  - для простых случаев это верно, но при росте объёма и сложности запросов **кластеризация становится важной**.
- Недооценивать стоимость лишнего сканирования:
  - “ну прочитали не 10 ГБ, а 60 ГБ” — в облаке это прямо деньги.

### Что будет, если…

- **Не задавать clustering keys и много писать/обновлять данные в случайном порядке:**
  - микропартиции будут содержать смесь разных дат/ключей;
  - pruning станет хуже;
  - стоимость и время запросов вырастут.
- **Слишком часто менять clustering keys или переусердствовать с automatic clustering:**
  - Snowflake будет тратить много ресурсов на перекладку данных;
  - счёт за compute может вырасти без видимого выигрыша.

### Проверь себя

1. Почему даже в полностью управляемом облачном DWH вроде Snowflake важно понимать, как устроено колончатое хранение?

<details><summary>Ответ</summary>
Потому что, несмотря на “магическую” автоматизацию, **основные физические принципы остаются теми же**: есть колончатые блоки, статистика, pruning, кластеризация. Понимание этих принципов позволяет:
- проектировать схемы и ключи, которые естественно хорошо работают;
- интерпретировать объяснения стоимости запросов;
- принимать осознанные решения о том, где тратить compute, а где изменить структуру данных.
</details>

2. Как clustering keys влияют на стоимость и время выполнения запросов?

<details><summary>Ответ</summary>
Кластеризация по ключам, которые часто используются в фильтрах/группировках, позволяет Snowflake **отбрасывать больше микропартиций по статистике**, не читая их физически. Это уменьшает объём сканируемых данных и, соответственно, время и стоимость запросов. Плохой clustering key (или его отсутствие) приводит к тому, что многие запросы вынуждены читать гораздо больше микропартиций.
</details>

3. Почему разделение storage/compute не отменяет необходимости думать о layout данных?

<details><summary>Ответ</summary>
Потому что разделение storage/compute лишь позволяет **независимо масштабировать** хранение и вычисления. Но каждая операция чтения/декодирования данных всё равно стоит **CPU и времени**, за которые ты платишь. Если layout неудачен, warehouse’у придётся сканировать и обрабатывать больше данных, чем нужно, что приведёт к росту времени и стоимости, независимо от того, насколько гибко масштабируются ресурсы.
</details>

### Запомните

- Snowflake скрывает от тебя многое, но **физика колончатых движков всё равно работает**.
- Micro‑partitions и clustering keys — главные рычаги влияния на производительность и стоимость.
- Даже “serverless”/“fully managed” системы выигрывают от грамотного моделирования данных.
