## 279. Storage layout колончатых БД

### Цель раздела

Понять, **как данные колончатых БД разложены на диске и в памяти**:  
что такое сегменты, stripe’ы, row group’ы, zone maps, min/max pruning,  
как сортировка и партиционирование влияют на I/O,  
и как layout реализован в ClickHouse, Parquet/ORC, Snowflake, SQL Server columnstore и др.

### В этом разделе главное

1. Колончатые БД группируют строки в **крупные блоки (segments/row groups/stripe’ы)**, внутри которых колонки хранятся отдельными массивами.
2. Каждый блок сопровождается **метаданными и статистикой** (min/max, количество NULL и т.п.), которые позволяют отсекать ненужные блоки.
3. **Сортировка и партиционирование** определяют, какие запросы будут “попадать” в компактные блоки, а какие — сканировать всё.
4. Layout влияет не только на скорость, но и на **сжатие, параллелизм и балансировку нагрузки**.
5. Понимание layout’а позволяет проектировать схемы и ключи, которые **естественно помогают движку**.

### Термины

- **Segment / row group / stripe** — крупная единица хранения строк в колончатой БД.
- **Column chunk** — данные одной колонки внутри сегмента.
- **Zone map / min‑max index** — min/max и другие статистики по сегменту/column chunk’у.
- **Partition** — логическое разделение таблицы по какому‑то ключу (часто по дате/диапазону).
- **Order key / sorting key** — порядок сортировки внутри партиции/сегмента.

### Теория и правила

#### 1) Логическая vs физическая структура

Логически:

- таблица — это набор строк с колонками `c1, c2, c3, …`.

Физически в колончатой БД:

- строки разбиваются на **блоки** (segments/row groups/stripe’ы) по N строк;
- для каждого блока:
  - данные каждой колонки хранятся в отдельном массиве (column chunk);
  - есть метаданные и статистика по колонкам.

Примеры терминов:

- Parquet — **row group**;
- ORC — **stripe**;
- ClickHouse — **part** (часть) + внутри `marks`/granules;
- SQL Server columnstore — **row group**.

#### 2) Зачем нужны блоки

Блоки позволяют:

- **работать единицами**, удобными для:
  - чтения с диска (большие последовательные чтения);
  - сжатия (внутри блока много похожих данных);
  - параллелизма (обрабатывать разные блоки на разных потоках/узлах);
- хранить **статистику** по блокам и колонкам:
  - min/max;
  - количество NULL;
  - гистограммы и т.п.

#### 3) Zone maps и min‑max pruning

Для каждого блока и колонки:

- движок хранит `min(value)`, `max(value)`, иногда ещё статистику (например, “все значения одинаковы”).

При запросе с условием `WHERE`:

- если диапазон запроса **не пересекается** с `[min, max]` блока:
  - блок можно **совсем не читать** с диска;
  - это называется **min‑max pruning**.

Пример:

- блок 1: `event_date` в `[2024‑01‑01, 2024‑01‑31]`;
- блок 2: `event_date` в `[2024‑02‑01, 2024‑02‑29]`;
- запрос `WHERE event_date >= '2024‑03‑01'`:
  - оба блока отбрасываются даже **до чтения данных**.

#### 4) Партиции и сортировка

Parquet/ORC, ClickHouse, Snowflake и др. активно используют:

- **партиционирование** (обычно по дате/диапазону);
- **сортировку/кластеризацию** внутри партиций.

Это даёт:

- компактные блоки, где:
  - значения по ключу (дата, user_id,…) сгруппированы;
  - min/max позволяют агрессивно отсекать блоки;
- хорошую базу для кодировок (см. §277) и векторизации (§278).

### Пошагово

Рассмотрим упрощённый Parquet‑layout для таблицы логов:

```text
logs/
  event_date=2024-01-01/   part-0001.parquet
  event_date=2024-01-02/   part-0002.parquet
  ...
```

Внутри `part-0001.parquet`:

1. Несколько **row groups** по N строк (например, 1 млн).
2. В каждом row group:
   - column chunks для `event_time`, `user_id`, `country`, `event_type`, `amount`, …;
   - статистика (min/max и т.п.) по каждому column chunk’у.

Запрос:

```sql
SELECT user_id, sum(amount)
FROM logs
WHERE event_date BETWEEN '2024-01-10' AND '2024-01-15'
  AND country = 'RU'
GROUP BY user_id;
```

Pipeline:

1. Сначала по **каталогу партиций** выбираются только нужные даты (`2024‑01‑10..15`).
2. По **метаданным row groups** выбрасываются группы, где:
   - `country` никогда не равно `'RU'` (по статистике).
3. По метаданным внутри файла система решает, какие column chunks вообще нужно читать.

Результат:

- данные считываются только там, где есть шанс найти `country='RU'` в нужном диапазоне дат.

### Простыми словами

Представь архив документов:

- все документы разложены по **годам/месяцам** (партиции);
- внутри каждого месяца — по **типу документа** или по алфавиту (сортировка);
- на коробках написано:
  - “январь 2024, от A до M, от N до Z”;
  - “все документы только по России/США/Европе” (статистика).

Когда тебя просят “найти все документы из России за 10–15 января”, ты:

- не заходишь во все комнаты архива;
- открываешь только коробки **нужных дат**;
- и даже среди них быстро отбрасываешь коробки, где нет России.

### Картинка в голове

Представь таблицу как **огромный прямоугольник**:

- по вертикали — строки;
- по горизонтали — колонки.

Колончатый layout режет этот прямоугольник:

- по вертикали — на **сегменты/row groups**;
- по горизонтали — на отдельные **полосы колонок**.

Для каждого “квадратика” (column chunk в row group) хранится “аннотация” с min/max и статистикой.

### Как запомнить

- **Row group/segment/stripe = блок строк**.
- **Column chunk = данные одной колонки в блоке**.
- **Zone maps = “аннотация на блоке: что примерно внутри”**.

### Примеры

#### ClickHouse

В ClickHouse:

- данные хранятся в **частях** (parts) движка MergeTree;
- каждая часть соответствует диапазону сортировочного ключа;
- внутри части:
  - данные колонок разбиваются на **granules**;
  - для granules хранятся метаданные (marks, min/max и т.п.);
- при запросах ClickHouse использует:
  - партиционирование по дате/диапазону;
  - отсечение по `ORDER BY`‑ключу и min/max.

#### SQL Server columnstore

В columnstore‑индексах:

- данные хранятся в **row groups** (до ~1 млн строк);
- для каждой колонки в row group:
  - отдельный column segment;
  - сохраняются min/max и статистика.

### Практика / реальные сценарии

- **Хранилище событий/логов**:
  - партиционирование по дате;
  - сортировка по (date, user_id) или (date, event_type);
  - даёт быстрые выборки по интервалам времени и/или по пользователям.
- **Отчёты по продажам**:
  - партиционирование по дате;
  - сортировка по (date, region, product_id);
  - min/max и dictionary помогают отбрасывать ненужные регионы/продукты.

### Типичные ошибки

- Партиционировать “по мелочи” (например, по часу) → слишком много мелких файлов/частей → накладные расходы на метаданные и управление.
- Не думать, **какие запросы будут самыми частыми**, и выбирать сортировку “наугад”.
- Переоценивать zone maps:
  - если данные “случайно перемешаны”, min/max мало что сокращают.

### Что будет, если…

- **Выбрать плохой layout (партиции/сортировку) под частые запросы:**
  - запросы будут читать слишком много блоков;
  - I/O и CPU будут расходоваться впустую;
  - выигрыш колончатого движка сильно снизится.
- **Переборщить с количеством партиций:**
  - можно получить много мелких файлов/частей;
  - накладные расходы на open/close, метаданные, планирование сильно вырастут.

### Проверь себя

1. Почему min‑max статистика по сегментам особенно полезна для колонок с “естественным” порядком (даты, числа), а не для произвольных строк?

<details><summary>Ответ</summary>
Потому что для отсечения по условию `WHERE` важно, чтобы значения в сегменте **лежали в более‑менее компактном диапазоне**. Для дат и чисел это естественно: в партиции “январь 2024” значения `event_date` будут лежать в узком интервале. Для произвольных строк min/max тоже можно считать, но диапазон может быть слишком широким и не давать полезного pruning’а.
</details>

2. Зачем колончатым БД разбивать данные на большие блоки (row groups/segments), а не хранить каждую строку отдельным объектом?

<details><summary>Ответ</summary>
Потому что большие блоки:
- позволяют **делать крупные последовательные чтения** с диска, что гораздо эффективнее случайных маленьких I/O;
- дают достаточно данных для **эффективного сжатия** и кодировок;
- упрощают **параллелизм** (разные блоки можно обрабатывать независимо).  
Хранение каждой строки отдельно увеличило бы накладные расходы и разрушило бы преимущества колончатого формата.
</details>

3. Почему при проектировании схемы в ClickHouse или Parquet‑хранилище важно думать о партиционировании и сортировке до загрузки больших объёмов данных?

<details><summary>Ответ</summary>
Потому что изменить партиционирование/сортировку **задним числом** — дорого: нужно пересоздавать таблицу, переливать и пересортировывать данные. Если сразу выбрать неудачный layout, ты будешь годами платить за лишний I/O и CPU на запросах. Поэтому лучше заранее понять, какие поля чаще всего участвуют в фильтрах/группировках, и спроектировать layout под них.
</details>

### Запомните

- Layout — это “география” таблицы: как данные разложены по файлам и блокам.
- Партиции и сортировка должны **соответствовать типовым запросам**, а не только “красиво выглядеть”.
- Хороший layout позволяет движку **читать минимально возможный объём данных** для ответа на запрос.
