## 282. Columnar индексы

### Цель раздела

Понять, **как колончатые индексы реализованы в классических СУБД** (например, SQL Server),  
чем они отличаются от “полных” колончатых движков,  
когда их стоит использовать для аналитики поверх OLTP‑схем,  
и какие компромиссы по ресурсам и сложности они привносят.

### В этом разделе главное

1. Columnstore‑индексы в SQL Server (и аналоги) — это **колончатое представление данных**, добавленное поверх rowstore‑таблиц.
2. Они дают преимущества колончатых движков для аналитики, но сохраняют возможность OLTP‑операций по rowstore.
3. Batch mode и векторизация в таких индексах **существенно ускоряют агрегаты и сканы**.
4. Вставки/обновления в presence columnstore требуют **дельта‑хранилищ и переработки**, что влияет на нагрузку.
5. Нужно тщательно выбирать, на какие таблицы и колонки вешать columnstore‑индексы, чтобы выигрыш перевесил накладные расходы.

### Термины

- **Clustered columnstore index** — основной (кластерный) колончатый индекс, заменяющий rowstore‑структуру таблицы.
- **Nonclustered columnstore index** — дополнительный колончатый индекс поверх существующей rowstore‑таблицы.
- **Row group** — блок строк в columnstore‑индексе.
- **Delta store** — буфер для новых/изменённых строк до включения их в columnstore.

### Теория и правила

#### 1) Columnstore в SQL Server

В SQL Server:

- **clustered columnstore index**:
  - хранит данные таблицы **только в колончатом виде**;
  - хорош для чисто аналитических таблиц.
- **nonclustered columnstore index**:
  - добавляется поверх обычной rowstore‑таблицы;
  - таблица остаётся пригодной для OLTP;
  - аналитика может использовать columnstore‑индекс.

Columnstore:

- хранит строки в **row groups**;
- в каждом row group данные колонок кодируются и сжимаются (см. §277);
- метаданные и статистика помогают pruning’у (см. §279).

#### 2) Batch mode

При доступе через columnstore SQL Server может использовать **batch mode**:

- операторы (join, aggregate, filter) работают с **пакетами строк**;
- включаются векторизация и SIMD;
- существенно уменьшается накладной overhead по сравнению с row‑by‑row.

#### 3) Delta store и модификации

Проблема:

- колончатая структура плохо подходит для **частых мелких обновлений/вставок**.

Решение:

- использовать **delta store**:
  - новые строки сначала попадают в rowstore‑буфер;
  - при накоплении до порога — преобразуются в columnstore‑row group;
  - периодически “подчищаются”/компактифицируются.

Это добавляет:

- фоновые операции;
- накладные расходы при интенсивной write‑нагрузке.

### Пошагово

Пример:

```sql
CREATE TABLE Sales
(
    SaleId      INT IDENTITY PRIMARY KEY,
    OrderDate   DATE,
    CustomerId  INT,
    ProductId   INT,
    Quantity    INT,
    Amount      MONEY
);
```

1. Таблица создаётся как обычная rowstore.
2. Для аналитики создаём nonclustered columnstore:

```sql
CREATE NONCLUSTERED COLUMNSTORE INDEX IX_Sales_Columnstore
ON Sales (OrderDate, CustomerId, ProductId, Quantity, Amount);
```

3. OLTP‑операции:
   - `INSERT/UPDATE/DELETE` продолжают работать по rowstore;
   - изменения отражаются в delta store columnstore‑индекса;
   - периодически данные “перетекают” в columnstore.
4. Аналитические запросы:

```sql
SELECT CustomerId, SUM(Amount)
FROM Sales
WHERE OrderDate >= '2024-01-01'
GROUP BY CustomerId;
```

- планировщик может использовать columnstore‑индекс и batch mode, значительно ускоряя запрос.

### Простыми словами

Columnstore‑индекс — это как **отдельная “аналитическая витрина”** внутри той же СУБД:

- основная таблица остаётся **строковой** для OLTP, с привычными индексами;
- колончатый индекс хранит данные **по столбцам**, специально для аналитики;
- запросы могут выбирать, откуда читать:
  - для точечных операций — rowstore;
  - для агрегатов и отчётов — columnstore.

### Картинка в голове

Представь:

- есть **книга продаж** (rowstore) — в ней все продажи пишутся по строкам;
- сверху ты делаешь **отдельный конспект**, где выписываешь по столбцам:
  - все даты,
  - всех клиентов,
  - все суммы.

Когда нужно посчитать суммарные продажи по клиентам — легче идти по конспекту (columnstore), чем перелистывать всю книгу.

### Как запомнить

- **Columnstore index = колончатый “слой” поверх строковой таблицы**.
- **Batch mode = векторизированное исполнение поверх columnstore**.
- Хорош для **аналитики поверх OLTP‑данных**, но требует осознанного управления модификациями.

### Примеры

#### Аналитический сценарий

```sql
SELECT
    ProductId,
    SUM(Amount) AS TotalSales
FROM Sales
WHERE OrderDate BETWEEN '2024-01-01' AND '2024-03-31'
GROUP BY ProductId;
```

- без columnstore — скан большого rowstore‑индекса/таблицы;
- с columnstore — чтение только нужных колонок (`OrderDate`, `ProductId`, `Amount`) в batch mode.

### Практика / реальные сценарии

- **Корпоративные OLTP‑системы на SQL Server**, где:
  - есть тяжёлые отчёты по тем же таблицам, что используют транзакции;
  - columnstore‑индексы позволяют ускорить отчёты без переноса данных в отдельный DWH.
- **Постепенная миграция в аналитику**:
  - сначала добавляем columnstore‑индекс;
  - потом, при необходимости, переносим аналитику в отдельные колончатые движки (Snowflake, ClickHouse).

### Типичные ошибки

- Вешать columnstore‑индекс на таблицы с **очень интенсивной write‑нагрузкой**, не измеряя накладные расходы delta store.
- Ожидать, что columnstore автоматически ускорит все запросы:
  - точечные look‑up’ы по PK могут вообще не выиграть.

### Что будет, если…

- **Добавить columnstore‑индекс на “горячую” OLTP‑таблицу без анализа:**
  - можно получить рост времени на insert/update;
  - фоновые процессы по обслуживанию delta store будут конкурировать за ресурсы.
- **Не использовать columnstore там, где есть тяжёлые отчёты по большим таблицам:**
  - ты будешь продолжать платить за полный скан rowstore;
  - возможность ускорить аналитику “малой кровью” будет упущена.

### Проверь себя

1. В чём главное отличие clustered и nonclustered columnstore‑индексов?

<details><summary>Ответ</summary>
Clustered columnstore index делает таблицу **полностью колончатой** — данные хранятся только в columnstore‑формате. Хорош для чисто аналитических таблиц.  
Nonclustered columnstore index добавляется поверх **обычной строковой таблицы**: rowstore остаётся для OLTP, columnstore используется для аналитики. Это компромисс между OLTP и OLAP в одной базе.
</details>

2. Почему при наличии columnstore‑индекса SQL Server может всё равно выбирать rowstore‑индексы для некоторых запросов?

<details><summary>Ответ</summary>
Потому что columnstore оптимален для **сканов и агрегатов по значительной части таблицы**, а не для точечных запросов по PK/уникальному ключу. Для выборок “по одной строке” или очень узких диапазонов rowstore‑индекс с бинарным поиском и небольшим количеством I/O может быть быстрее и дешевле. Планировщик выбирает структуру, которая по его оценке выполняется быстрее.
</details>

3. Какие дополнительные накладные расходы вносит columnstore‑индекс на таблицу с частыми обновлениями?

<details><summary>Ответ</summary>
Обновления и вставки попадают в **delta store**, который нужно:
- поддерживать в актуальном состоянии;
- периодически “сбрасывать” в основной columnstore (сжатие, перекодирование, статистика).  
Это подразумевает дополнительные фоновые операции, потребление CPU и I/O. При очень интенсивной write‑нагрузке эти накладные расходы могут быть значительными.
</details>

### Запомните

- Columnstore‑индексы — мост между миром OLTP и OLAP внутри классических СУБД.
- Они требуют **осознанного применения**: под аналитические запросы, а не “на всякий случай”.
- Batch mode и колончатое хранение дают серьёзный выигрыш в аналитике, если правильно спроектированы.
