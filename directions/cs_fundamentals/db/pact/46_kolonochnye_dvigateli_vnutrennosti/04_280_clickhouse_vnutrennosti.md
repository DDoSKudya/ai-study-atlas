## 280. ClickHouse внутренности

### Цель раздела

Разобраться, **как внутри устроен ClickHouse как колончатый движок**:  
что такое MergeTree и его семья, как работают primary key и `ORDER BY`,  
как реализованы партиции, материализованные представления, сжатие,  
и какие практические выводы из этого следуют для моделирования данных и запросов.

### В этом разделе главное

1. Основной движок ClickHouse — семейство **MergeTree**: данные пишутся частями и периодически сливаются (merge/compaction).
2. `ORDER BY` и primary key определяют **физический порядок данных внутри частей**, влияя на pruning и сжатие.
3. **Партиции** (обычно по дате/диапазону) управляют жизненным циклом данных и помогают ограничивать объём сканирования.
4. **Материализованные представления** позволяют строить предагрегаты и ускорять тяжёлые отчёты.
5. Сжатие и кодировки на уровне колонок тесно связаны с сортировкой и паттернами данных.

### Термины

- **MergeTree** — основной движок хранения ClickHouse (и его вариации).
- **Part (часть)** — логический блок данных внутри таблицы MergeTree.
- **Primary key / ORDER BY** — ключи сортировки и поиска в MergeTree.
- **Materialized view** — таблица, автоматически заполняемая по результатам запроса.
- **TTL** — механизм автоматического удаления/перемещения данных по времени.

### Теория и правила

#### 1) MergeTree: вставка и merge

Когда ты делаешь `INSERT` в таблицу MergeTree:

- данные не сразу попадают в “одну большую структуру”;
- они:
  1. собираются в память/микроблоки;
  2. сортируются по `ORDER BY`;
  3. записываются как отдельная **часть (part)** на диск.

Периодически:

- фоновые потоки **сливают** несколько мелких частей в одну крупную:
  - объединяют данные;
  - поддерживают сортировку по `ORDER BY`;
  - обновляют статистику и индексы.

Это похоже на LSM‑деревья (см. Часть XLIV), но заточено под колончатый формат и аналитику.

#### 2) ORDER BY и primary key

В ClickHouse:

- `ORDER BY` определяет, в каком порядке строки внутри части отсортированы;
- `PRIMARY KEY` по умолчанию совпадает с `ORDER BY` (может отличаться, но редко).

Важные эффекты:

- **запросы с фильтрами по ключам** могут сильно отсекать части и блоки;
- сортировка помогает:
  - с кодировками (RLE/delta);
  - с min‑max pruning (см. §277–§279).

Типовой паттерн:

```sql
ORDER BY (event_date, user_id)
```

- удобен для запросов:
  - “все события за период для пользователя”;
  - “агрегации по пользователю за период”.

#### 3) Партиции и TTL

Партиционирование:

- задаётся в `ENGINE`:

```sql
PARTITION BY toYYYYMM(event_date)
```

- определяет, как данные физически разбиваются на директории/части;
- удобно для:
  - удаления старых данных (drop whole partition);
  - ограничения по времени в запросах.

TTL:

- позволяет автоматически:
  - удалять старые данные;
  - перемещать их в другие диски (например, “холодное” хранилище).

### Пошагово

Пример таблицы:

```sql
CREATE TABLE analytics.events
(
    event_date Date,
    user_id    UInt64,
    event_type LowCardinality(String),
    amount     Float64
)
ENGINE = MergeTree
PARTITION BY toYYYYMM(event_date)
ORDER BY (event_date, user_id);
```

Вставка данных:

1. Клиент делает `INSERT` батчем.
2. Сервер:
   - разбирает данные;
   - сортирует по `(event_date, user_id)`;
   - пишет часть (part) на диск под соответствующей партицией (`202403`, `202404`,…).
3. Фоновый merge:
   - берёт несколько мелких частей за `202403`;
   - объединяет их в крупную, сохраняя сортировку;
   - обновляет индексы и статистику.

Запрос:

```sql
SELECT user_id, sum(amount)
FROM analytics.events
WHERE event_date BETWEEN '2024-03-01' AND '2024-03-31'
  AND event_type = 'purchase'
GROUP BY user_id;
```

Шаги:

1. Отбрасываются партиции, не попадающие в март `2024` (`PARTITION BY`).
2. Внутри подходящих частей:
   - по `ORDER BY`/primary key` отсеиваются блоки вне диапазона дат;
   - используется статистика и dictionary по `event_type` для дополнительного pruning’а.
3. Дальше включаются vectorized execution и кодировки (см. §§277–278).

### Простыми словами

ClickHouse — это как **склад, куда коробки с данными сначала ставят “как есть”, а потом периодически перекладывают на аккуратные стеллажи по порядку**.

- `INSERT` добавляет **коробки** (parts);
- фоновые процессы **перекладывают и складывают коробки** по дате/ключу;
- сортировка по ключу и статистика позволяют быстро понять, **какие коробки вообще нужно открывать** для запроса.

### Картинка в голове

Представь:

- каждая партиция — это **комната** с коробками (parts);
- сортировка по `(event_date, user_id)` — это порядок коробок на стеллажах;
- zone maps и индексы — это **наклейки на коробках**:
  - “даты от … до …”;
  - “user_id от … до …”.

При запросе ты смотришь на наклейки и открываешь только часть коробок, а не весь склад.

### Как запомнить

- **MergeTree = “LSM‑подобный колончатый движок”**.
- **ORDER BY/PRIMARY KEY задают физический порядок** → сильно влияют на pruning и сжатие.
- Партиционирование по дате — почти всегда хорошая идея для временных данных.

### Примеры

#### Материализованные представления

```sql
CREATE MATERIALIZED VIEW analytics.daily_revenue
ENGINE = SummingMergeTree
PARTITION BY toYYYYMM(day)
ORDER BY (day, country)
AS
SELECT
    toDate(event_time) AS day,
    country,
    sum(amount)        AS revenue
FROM analytics.events
GROUP BY day, country;
```

- новое событие в `analytics.events` автоматически попадает в `daily_revenue`;
- отчёты по дневной выручке работают **по предагрегированным данным**, а не по сырой таблице.

### Практика / реальные сценарии

- **Хранилище событий для веб‑приложения**:
  - `PARTITION BY toYYYYMM(event_date)`;
  - `ORDER BY (event_date, user_id)` или `(event_date, event_type, user_id)`;
  - материализованные представления для предагрегатов по дням/странам/кампаниям.
- **Финансовые отчёты**:
  - партиции по месяцу;
  - сортировка по (дата, счёт, продукт).

### Типичные ошибки

- Использовать **слишком много колонок в `ORDER BY`** → избыточная сортировка, уменьшение эффективности сжатия и pruning’а.
- Партиционировать по полю с очень высокой кардинальностью (например, `user_id`):
  - получится огромное количество партиций;
  - управление ими станет кошмаром.
- Слепо копировать примеры из документации без учёта своих запросов.

### Что будет, если…

- **Выбрать `ORDER BY`, не соответствующий частым запросам:**
  - ClickHouse будет вынужден читать больше данных;
  - min‑max/zone maps будут давать мало пользы;
  - нагрузка по I/O и CPU вырастет.
- **Забыть о merge и TTL:**
  - со временем накопится много мелких частей;
  - возрастут накладные расходы на планирование и чтение;
  - диск может раздуться из‑за “висящих” старых партиций.

### Проверь себя

1. Почему в ClickHouse так часто рекомендуют партиционировать по дате и сортировать по `(date, key)`?

<details><summary>Ответ</summary>
Потому что большинство аналитических запросов естественно ограничены по времени (“за день/неделю/месяц”) и часто группируются/фильтруются по каким‑то ключам (пользователь, регион, кампания). Партиционирование по дате позволяет быстро отсекать большие объёмы старых/ненужных данных, а сортировка по `(date, key)` даёт компактные блоки, которые эффективно сжимаются и легко отсекаются по min‑max.
</details>

2. В чём сходство MergeTree с LSM‑деревьями и в чём принципиальное отличие?

<details><summary>Ответ</summary>
Сходство:
- данные пишутся “кусками” и периодически **сливаются (merge/compaction)**;
- дизайн ориентирован на высокую скорость записи и чтение крупных блоков.  
Отличия:
- MergeTree — **колончатый** и заточен под аналитику, LSM чаще ассоциируется с key‑value/LSM‑таблицами;
- MergeTree активно использует **сортировку по ключу, колончатое сжатие и zone maps**, тогда как LSM‑слои могут иметь другие структуры индексов и форматы.
</details>

3. Почему большое количество мелких parts в ClickHouse — проблема?

<details><summary>Ответ</summary>
Потому что каждая часть:
- занимает файловые дескрипторы и метаданные;
- участвует в планировании запросов и merge‑операциях;  
Если частей слишком много, растут накладные расходы на управление ими (метаданные, планирование, open/close файлов), а фоновые merge’и могут постоянно догонять хвост, создавая нестабильную производительность.
</details>

### Запомните

- Понимание MergeTree (parts, paritions, ORDER BY, merge) — ключ к эффективному ClickHouse.
- В ClickHouse легко получить отличную производительность, если layout спроектирован под типовые запросы.
- Материализованные представления позволяют “перенести тяжёлую работу” в момент вставки, чтобы отчёты были быстрыми.
