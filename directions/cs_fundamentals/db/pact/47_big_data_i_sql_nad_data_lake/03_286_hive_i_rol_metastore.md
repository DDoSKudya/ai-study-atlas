## 286. Hive и роль Metastore

### Цель раздела

Понять, **как развивался Hive как SQL‑слой поверх Hadoop**,  
что такое Hive Metastore и почему он критичен для большинства SQL‑движков по data lake (Presto/Trino, Spark, Impala),  
какие execution‑движки (MapReduce, Tez, LLAP) использовались и используются.

### В этом разделе главное

1. Hive — исторически первый массовый **SQL поверх Hadoop**: изначально MapReduce, затем Tez и LLAP.
2. **Metastore** — сервис, где живут схемы таблиц, partition’ы, статистика; он стал **де‑факто стандартом метаданных** для Hadoop‑экосистемы.
3. Современные SQL‑движки по data lake почти всегда интегрируются с Hive Metastore или его аналогами.
4. Hive как engine используется реже, но как **слой метаданных** — по‑прежнему повсюду.

### Термины

- **Hive** — система, реализующая SQL‑подобный язык поверх Hadoop, исторически через MapReduce, затем Tez/LLAP.
- **HiveQL** — SQL‑подобный язык запросов Hive.
- **Hive Metastore** — сервис (обычно в виде отдельной БД + API), который хранит метаданные о таблицах/partition’ах.
- **Tez** — DAG‑движок, более гибкий и быстрый, чем MapReduce, для выполнения SQL‑запросов.
- **LLAP (Live Long and Process)** — механизм длительно живущих процессов/демонов для ускорения Hive‑запросов (кэширование, меньше cold‑start).

### Теория и правила

#### 1) Hive как “SQL к MapReduce”

Историческая идея:

- У нас есть **Hadoop + HDFS** и данные в файлах.
- Писать MapReduce‑программы на Java трудно и долго.
- Hive предлагает язык, похожий на SQL (HiveQL), который **компилируется в MapReduce‑job’ы**.

Последствия:

- ранние Hive‑запросы были **очень медленными** (каждый запрос → один или несколько MapReduce‑job’ов);
- латентность была измеряема минутами, а не секундами;
- Hive принёс SQL‑мышление в мир Hadoop, но execution‑движок был тяжёловесным.

#### 2) Эволюция: Tez и LLAP

Чтобы ускорить Hive:

- появился **Tez** — движок DAG‑задач, который позволял строить более гибкие графы работы, чем MapReduce;
- затем — **LLAP**:
  - длительно живущие процессы, которые не нужно запускать каждый раз заново;
  - кэширование данных в памяти;
  - уменьшение накладных расходов на инициализацию job’ов.

Hive с Tez/LLAP стал куда более пригоден для задач, где нужна **умеренная интерактивность**, но всё ещё уступает специализированным MPP‑движкам (Impala, Presto/Trino).

#### 3) Metastore как центр экосистемы

Hive Metastore хранит:

- описания **баз данных/таблиц**;
- список и структуру **partition’ов**;
- пути к файлам (директории в HDFS/S3);
- **статистику** (количество строк, min/max, размер и т.п.).

Другие движки:

- Presto/Trino → читают схемы и partition’ы из Metastore;
- Spark SQL → может использовать Metastore для `spark.sql.catalogImplementation=hive`;
- Impala → тоже интегрируется с Metastore.

В итоге Hive Metastore стал **“регистром таблиц”** для всего кластера, даже если сам Hive как engine почти не используется.

### Пошагово: создание таблицы в Hive и её использование из Presto

1. В Hive:

```sql
CREATE TABLE analytics.events (
  event_id    STRING,
  user_id     STRING,
  event_dt    DATE,
  payload     STRING
)
PARTITIONED BY (event_month STRING)
STORED AS PARQUET
LOCATION 's3://datalake/analytics/events/';
```

2. Hive заносит в Metastore:
   - схему таблицы;
   - partition‑ключ `event_month`;
   - базовый путь `s3://.../events/`.
3. ETL‑процесс пишет данные в каталоги вида `.../events/event_month=2025-02/part-00001.parquet` и добавляет partition’ы в Metastore.
4. Presto/Trino, используя Hive‑connector, видит в каталоге `hive.analytics.events` ту же таблицу, читает partition’ы и файлы.
5. Аналитики могут выполнять запросы к `analytics.events` как из Hive, так и из Presto/Trino.

### Простыми словами

Сегодня Hive часто — это **не столько движок, сколько “записная книжка о таблицах”**:

- Metastore знает, что есть за таблицы, где лежат их данные, какие у них столбцы и partition’ы.
- Остальные инструменты (Presto, Spark, Impala) используют эту записную книжку, чтобы не гадать, как устроены данные.

### Картинка в голове

Представь большой склад (HDFS/S3) с коробками (директориями и файлами).  
Hive Metastore — это **каталог склада**, где написано:

- в каком ряду и на какой полке чьи коробки,
- что внутри коробок (схема),
- какой общий объём и примерное содержание.

Без этого каталога каждый новый работник (SQL‑движок) будет бегать по складу вслепую.

### Как запомнить

- **Hive (ранний) = SQL → MapReduce. Hive (сейчас) = SQL → Tez/LLAP + Metastore.**
- Metastore пережил несколько поколений execution‑движков и стал стандартом для метаданных.

### Примеры

- Кластер, где старые отчёты до сих пор крутятся на Hive, но новые ad‑hoc‑запросы делают через Trino, используя тот же Metastore.
- Spark‑pipeline, который пишет Parquet‑файлы и регистрирует таблицы/partition’ы в Hive Metastore, чтобы downstream‑системы могли их читать.

### Практика / реальные сценарии

- **Миграция с Hive на Spark/Presto**: Metastore остаётся, меняются только движки выполнения.
- **Общий каталог таблиц для data lake**: все команды публикуют свои наборы данных как Hive‑таблицы, доступные разным инструментам.

### Типичные ошибки

- Считать Hive Metastore “второстепенным сервисом”: при его сбое весь SQL‑слой по lake становится слепым.
- Хранить в Metastore “мусорные” таблицы, не чистить устаревшие partition’ы — это усложняет эксплуатацию и увеличивает накладные расходы.
- Игнорировать настройки consistency между Metastore и фактическими файлами в хранилище (race‑conditions при записи/удалении).

### Что будет, если…

- … Metastore недоступен или работает медленно?  
  → Presto/Trino/Spark/Impala не смогут быстро получить схему/partition’ы; запросы будут падать или сильно замедляться.

- … таблицы создаются “в обход” Metastore (просто пишутся файлы без регистрации partition’ов)?  
  → SQL‑движки не увидят новые данные или увидят их с задержкой; возможен рассинхрон между тем, что реально лежит в lake, и тем, что видят инструменты.

### Проверь себя

1. Какую роль играет Hive Metastore в современных data lake‑архитектурах?

<details><summary>Ответ</summary>
Это **центральный реестр таблиц и partition’ов**, который описывает, какие таблицы существуют, какие у них схемы и где лежат их данные. Большинство SQL‑движков (Presto/Trino, Spark, Impala) используют Metastore как источник правды о структуре данных.
</details>

2. Почему Hive‑engine (MapReduce/Tez/LLAP) сегодня используется реже, но Metastore остаётся актуальным?

<details><summary>Ответ</summary>
Появились более быстрые и гибкие execution‑движки (Spark, Presto/Trino, Impala), которые лучше справляются с аналитикой и ETL.  
Однако **метаданные — общая проблема**, и простое, проверенное решение в виде Metastore продолжает устраивать большинство проектов. Заменять его нет необходимости, если execution можно обновить отдельно.
</details>

3. Что будет, если проект решит обойтись без централизованного Metastore и “довериться только структуре директорий и файлов”?

<details><summary>Ответ</summary>
Придётся самому изобретать способ описания и согласования схем, хранить знания о структурах в коде ETL/скриптах, а инструменты SQL будут видеть данные только через adhoc‑описания или не видеть вовсе. Это ведёт к **хаосу схем**, дублированию логики и сложности интеграции разных команд и инструментов.
</details>

### Запомните

- Hive как execution‑движок — это исторически важный, но не всегда актуальный сегодня слой; **Hive Metastore — живой и критичный компонент** современных data lake.
- Понимание Metastore и его роли помогает проектировать устойчивую архитектуру метаданных для SQL‑аналитики поверх файловых хранилищ.
