## 288. Формат Parquet

### Цель раздела

Разобраться, **как устроен формат Parquet внутри**: row group, column chunk, pages, статистика и сжатие,  
как эти механизмы помогают SQL‑движкам по data lake (Presto/Trino, Spark, Impala, Hive) быстро читать и фильтровать данные,  
и какие практические настройки важны в продакшене.

### В этом разделе главное

1. Parquet — это **колончатый формат файлов**, оптимизированный для аналитических сканов и агрегаций.
2. Файл Parquet делится на **row group’ы**, внутри которых данные разрезаны по колонкам (column chunk).
3. В каждом column chunk данные разбиваются на **pages**, к которым применяются кодеки сжатия и кодировки.
4. Файл содержит **статистику (min/max, count и др.)** по группам/страницам, что позволяет движкам не читать лишние данные (pruning).
5. Размеры row group’ов и стратегия записи сильно влияют на производительность и проблему small files.

### Термины

- **Row group** — блок строк в файле Parquet (обычно от десятков тысяч до миллионов строк); минимальная единица, которую многие движки читают целиком.
- **Column chunk** — данные одной колонки в пределах row group.
- **Page** — ещё более мелкая единица внутри column chunk (несколько тысяч значений), к которой применяется сжатие/кодирование.
- **Metadata / footer** — хвост файла, где хранится схема, структура row group’ов и статистика.

### Теория и правила

#### 1) Колоночный layout Parquet

В отличие от строчных форматов (CSV, Avro), Parquet хранит:

- значения **каждой колонки подряд** (по row group’ам);
- вместе со вспомогательной информацией (уровни повторения/дефиниции для массивов/struct).

Преимущества:

- чтение **только нужных колонок**;
- эффективное сжатие однородных значений;
- хорошая совместимость с vectorized execution.

#### 2) Row group, column chunk, page

Интуитивно:

- **row group** — “кубик” данных, содержащий набор строк по всем колонкам;
- внутри него по каждой колонке есть свой **column chunk**;
- column chunk делится на **pages**.

Многие движки:

- читают и обрабатывают данные **row group’ами**;
- могут пропускать целые row group’ы, если min/max‑статистика показывает, что данные не попадают под фильтр.

#### 3) Статистика и pruning

Parquet хранит статистику:

- `min`, `max` по колонкам (часто по каждой page и row group);
- count, null‑count и др.

SQL‑движки используют это для:

- **partition pruning** (через Metastore/путь к файлам);
- **row group/page pruning** (на основе min/max внутри файлов).

Пример:

- условие `WHERE event_dt BETWEEN '2025-02-01' AND '2025-02-28'` позволит:
  - не читать файлы, где partition по `event_dt` вне этого диапазона;
  - внутри файла не читать row group’ы, где min/max `event_dt` не пересекается с диапазоном.

#### 4) Сжатие и кодировки в Parquet

Parquet — это не только “данные по столбцам”, но и **набор кодеков и схем кодирования**, которые сильно влияют на размер и скорость:

- **Сжатие (compression codec)**:
  - `snappy` — быстрый, умеренно сжимает; часто выбор “по умолчанию” для аналитики;
  - `gzip`/`zstd` — сильнее жмут, но могут больше грузить CPU;
  - `lz4`/`brotli` — разные компромиссы между скоростью и степенью сжатия.
- **Кодировки (encoding)**:
  - **RLE (Run‑Length Encoding)** — хорош, когда подряд много одинаковых значений (например, статус, регион);
  - **Dictionary encoding** — полезен для колонок с небольшим числом уникальных значений (страны, типы событий);
  - **Delta / delta‑binary‑packed** — для монотонно растущих чисел (id, временные метки);
  - простые битовые схемы (bit‑packing) — “упаковывают” числа в минимально необходимое количество бит.

Важные мысли:

- Parquet обычно **подбирает кодировку автоматически по колонке**, но результат всё равно зависит от характера данных.
- Чем **лучше данные “подходят” кодировке**, тем:
  - меньше размер файлов;
  - меньше чтение с диска/сети;
  - быстрее работают фильтры и агрегации.
- Но слишком “тяжёлое” сжатие (например, максимальный уровень `gzip`) может **перегрузить CPU**, особенно если кластеры слабые по процессору.

Практически:

- для “горячих” таблиц, по которым бегают интерактивные запросы, часто выбирают **быстрые кодеки** (Snappy, LZ4);
- для архивных слоёв (`cold storage`) можно использовать **более сильное сжатие** (ZSTD с более высоким уровнем).

Если упростить:

- **Кодировки** помогают “подогнать” данные под CPU и память (меньше байтов, больше плотность);
- **Сжатие** уменьшает объём на диске и в сети;
- **Статистика** помогает вообще не читать ненужные куски.

### Пошагово: как Spark/Presto читают Parquet

1. Движок получает список файлов (обычно через Metastore и/или list‑операции).
2. Для каждого файла читает **footer**:
   - схему;
   - структуру row group’ов;
   - статистику.
3. На основании фильтров в запросе выбирает:
   - какие файлы/row group’ы нужно читать;
   - какие можно пропустить.
4. При чтении колонок:
   - читает только нужные column chunk’и;
   - внутри них — только нужные pages.

### Простыми словами

Parquet — это **“ZIP‑архив, в котором каждая колонка упакована отдельно и снабжена оглавлением”**:

- можно открыть только нужные “папки” (колонки);
- можно по оглавлению (статистике) понять, какие “папки” внутри не содержат того, что нам нужно.

### Картинка в голове

Представь книжный шкаф:

- **шкаф** — файл Parquet;
- **полки** — row group’ы;
- **коробки на полках** — column chunk’и;
- **маленькие пачки листов в коробке** — pages;
- **наклейки на каждой коробке** с min/max — статистика.

Движок, как умный библиотекарь, не листает каждую страницу, а **смотрит на наклейки и берет только нужные коробки**.

### Как запомнить

- **Row group — основной блок чтения**.  
  Делать их слишком маленькими — увеличивать overhead, слишком большими — ухудшать локальность и гибкость pruning.

- **Parquet любит широкие таблицы с аналитическими запросами**, где можно читать только несколько колонок.

### Примеры

- Таблица фактов `fact_sales` с десятками колонок: для отчёта нужны только `sale_dt`, `region`, `amount`. Parquet позволяет читать только эти 3 столбца, не трогая остальные.
- Лог‑таблица `events` с датой `event_dt` в имени директории и в колонке: сначала отбираем папки по `event_dt`, затем row group’ы внутри Parquet‑файлов.

### Практика / реальные сценарии

- Настройка размера row group (через конфиги Spark/Writer) для баланса между гибкостью pruning и эффективностью чтения.
- Анализ статистики Parquet‑файлов (через инструменты/утилиты) для понимания, насколько хорошо работает pruning.

### Типичные ошибки

- Создавать **очень маленькие файлы Parquet** (десятки килобайт/несколько десятков килобайт): теряется преимущество row group’ов, overhead метаданных растёт.
- Неправильно выбирать колонки для partitioning: партиционировать по высоко кардинальным полям (вроде user_id), создавая миллионы директорий/файлов.
- Не обновлять статистику/не анализировать layout: тяжёлые запросы гуляют по всем файлам и row group’ам.

### Что будет, если…

- … записывать Parquet‑файлы без контроля размера row group и числа файлов?  
  → Получим либо огромные файлы с плохо управляемым layout’ом, либо миллионы мелких файлов; в обоих случаях SQL‑движки будут работать хуже.

- … не использовать Parquet‑статистику для pruning?  
  → Придётся читать гораздо больше данных, чем нужно; выигрыш от формата сильно уменьшится.

### Проверь себя

1. Зачем Parquet делит данные на row group’ы и column chunk’и?

<details><summary>Ответ</summary>
Row group’ы позволяют **работать с крупными, но управляемыми блоками строк**; column chunk’и — хранить данные по колонкам отдельно внутри каждого блока. Это даёт:
- эффективное сжатие;
- возможность читать только нужные колонки;
- гибкий pruning по статистике row group’ов.
</details>

2. Как статистика (min/max) по row group’ам помогает уменьшить объём чтения?

<details><summary>Ответ</summary>
По min/max можно **быстро понять, пересекается ли диапазон значений в row group с фильтром запроса**. Если нет — row group можно вообще не читать, экономя I/O и CPU.
</details>

3. Почему проблема small files особенно критична для Parquet?

<details><summary>Ответ</summary>
Каждый Parquet‑файл несёт свой footer и структуру row group’ов. Множество маленьких файлов означают:
- множество footers для чтения;
- множество операций открытия/закрытия файлов;
- много мелких row group’ов, что снижает эффективность чтения.  
В итоге overhead на метаданные и I/O растёт быстрее, чем полезная работа по обработке данных.
</details>

### Запомните

- **Parquet — базовый колончатый формат data lake**: row group’ы, column chunk’и, pages и статистика — его основные элементы.
- Грамотный выбор размеров файлов и row group’ов, а также использование статистики для pruning — ключ к высокой производительности SQL‑движков поверх Parquet.
