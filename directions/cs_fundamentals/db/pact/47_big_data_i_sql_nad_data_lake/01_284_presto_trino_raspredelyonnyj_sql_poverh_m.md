## 284. Presto/Trino: распределённый SQL поверх множества источников

### Цель раздела

Понять, **как устроены Presto и Trino** как распределённые SQL‑движки поверх множества источников (S3, HDFS, Kafka, традиционные СУБД),  
что такое coordinator и worker, как работает execution model, и в каких сценариях Presto/Trino особенно сильны — а где лучше выбрать другой инструмент.

### В этом разделе главное

1. Presto/Trino — это **“SQL‑федератор”**: один язык SQL по многим источникам (каталогам и коннекторам).
2. Архитектура: **один coordinator** строит план запроса, **множество worker‑ов** выполняют его параллельно.
3. Execution‑модель заточена под **низкую латентность чтения и агрегаций**, а не под долгие batch‑job’ы или тяжёлые upsert.
4. Основные bottleneck’и: **сетевой shuffle, медиатор‑coordinator, metadata‑слой (Metastore/каталоги)** и small files.
5. Presto/Trino отлично подходят для **ad‑hoc‑аналитики и интерактивных запросов** по data lake и сторонним СУБД.

### Термины

- **Coordinator** — процесс, который принимает SQL‑запрос, парсит его, строит логический и физический план, координирует выполнение и собирает результаты.
- **Worker** — узел, который выполняет отдельные фрагменты плана (операторы скана, `JOIN`, агрегации), обмениваясь данными с другими worker‑ами.
- **Connector** — модуль, который позволяет Presto/Trino читать (а иногда писать) данные из конкретного источника (S3, Hive Metastore, PostgreSQL и т.д.).
- **Catalog / schema / table** — логическая иерархия: `catalog.schema.table`, где catalog привязан к коннектору и набору настроек.
- **Split** — единица работы по чтению данных (обычно часть файла или диапазон блоков в HDFS/объекте).

### Теория и правила

#### 1) Архитектура “один координатор – много воркеров”

Интуитивная картина:

- Coordinator:
  - принимает SQL‑текст;
  - парсит и строит логический план;
  - оптимизирует его (перестановка `JOIN`, pushdown фильтров и т.д.);
  - разбивает план на стадии (stages) и задачи (tasks);
  - распределяет задачи по worker‑ам;
  - собирает результаты и отдаёт клиенту.
- Worker:
  - получает задачу;
  - читает данные из источника через connector;
  - выполняет локальные операции (`FILTER`, `PROJECT`, локальная агрегация);
  - участвует в shuffle (пересылке промежуточных данных);
  - отправляет результат координатору или другим worker‑ам.

Важно:

- **координатор — точка координации**, и при больших нагрузках он может стать bottleneck’ом;
- worker‑ы **без shared‑state хранения** — данные живут в внешнем хранилище (S3/HDFS/БД), а worker‑ы — статeless‑вычислители.

#### 2) Connector‑архитектура

Presto/Trino умеют ходить в:

- Hive Metastore + файловое хранилище (S3/HDFS/облачный объектный storage);
- традиционные реляционные БД (PostgreSQL, MySQL, SQL Server, Oracle);
- NoSQL/KV‑системы;
- Kafka, Elasticsearch и др.

Идея:

- каждый connector реализует:
  - список таблиц и их схем (через каталог/Metastore/собственный API);
  - разбиение таблиц на splits (части для параллельного чтения);
  - чтение данных (Parquet/ORC/Avro/CSV или низкоуровневый протокол);
  - иногда — pushdown фильтров и агрегаций “в источник”.

Так появляется **“SQL над всем зоопарком”** — удобно для аналитиков, BI и быстрых исследований.

#### 3) Execution model

Ключевые свойства:

- **in‑memory‑ориентированное исполнение**: Presto/Trino стараются держать промежуточные данные в памяти (с контролем по лимитам/спиллам на диск);
- **pipeline‑исполнение**: операторы объединяются в pipeline’ы, данные течёт потоками (operators → drivers);
- **распределённость**: heavy‑lifting по чтению и агрегациям распределён по worker‑ам;
- **ориентация на низкую латентность**: запросы часто на секунды, а не на часы, хотя это зависит от объёма данных и сложности плана.

### Пошагово: что происходит при запросе к таблице в data lake

Пусть у нас есть таблица `sales` в Hive‑каталоге (S3+Metastore), и мы выполняем:

```sql
SELECT region, SUM(amount) AS total_amount
FROM hive.analytics.sales
WHERE sale_dt BETWEEN DATE '2025-01-01' AND DATE '2025-01-31'
GROUP BY region;
```

По шагам:

1. Клиент (CLI, JDBC, BI‑инструмент) отправляет запрос координатору.
2. Coordinator:
   - парсит SQL;
   - получает из Metastore схему таблицы и список файлов/partition’ов по `sale_dt`;
   - строит логический план (`SCAN` → `FILTER` → `AGGREGATE`).
3. Coordinator разбивает таблицу на splits (обычно — по файлам/частям файлов).
4. Splits распределяются по worker‑ам.
5. Worker‑ы:
   - читают соответствующие Parquet‑файлы;
   - применяют фильтр по `sale_dt` (частично может использоваться partition pruning и min/max статистика);
   - делают **локальную агрегацию** `SUM(amount)` по `region` внутри каждого worker’а.
6. Результаты локальных агрегаций передаются по сети (shuffle) на узлы, которые делают **финальную агрегацию** по `region`.
7. Итоговый набор строк возвращается координатору и далее клиенту.

### Простыми словами

Presto/Trino — это **“SQL‑мозг” без собственного хранилища**, который:

- умеет говорить на SQL,
- знает, как выглядели бы таблицы в разных источниках,
- и распределяет работу по кластеру worker‑ов, которые читают данные где‑то “там” (S3, HDFS, БД).

Он не пытается быть “вторым PostgreSQL” — его сила в том, чтобы **быстро читать и соединять большие данные** из разных мест.

### Картинка в голове

Представь:

- **Coordinator** — дирижёр оркестра.
- **Worker‑ы** — музыканты, каждый играет свою партию (читает свою часть данных, считает агрегаты).
- **Connectors** — разные залы/архивы с нотами и инструментами (S3, БД, Kafka).
- SQL‑запрос — партитура: дирижёр распределяет, кто когда играет, и собирает общую музыку.

### Как запомнить

- **Presto/Trino = “легковесный SQL‑движок по data lake и БД”**.
- Координатор думает, воркеры считают.
- Чем больше источников и форматов, тем ценнее connector‑архитектура.

### Примеры

- Ad‑hoc‑аналитика: аналитик запускает интерактивный запрос по сырым логам в S3 (Parquet) через Presto, чтобы проверить гипотезу.
- BI‑дашборд: ClickHouse/Redshift держат витрины, но часть отчётов крутится напрямую по data lake через Trino.
- Федерированный запрос: объединить данные из PostgreSQL (справочники) и S3 (факты) в одном SQL‑запросе.

### Практика / реальные сценарии

- **Интерактивные аналитические запросы по S3/HDFS** — Presto/Trino нередко используют в паре с Spark: Spark — для тяжёлых batch’ей, Presto — для исследовательских запросов.
- **Федеративные запросы** — когда нужно быстро собрать отчёт из нескольких систем без жёсткого ETL.
- **Data mesh / data product‑подходы** — отдельные домены публикуют свои “data products” в разных хранилищах, а Presto/Trino позволяет поверх них строить единый SQL‑слой.

### Типичные ошибки

- Считать, что Presto/Trino — это “общая замена” DWH/СУБД: пытаться делать через него тяжёлые upsert/merge/OLTP.
- Игнорировать **Metastore/каталог**: хаотичные схемы, отсутствие описаний таблиц, дублирование в разных каталогах.
- Не думать о **partitioning и small files**: каждый запрос читает тысячи мелких файлов, координатор тонет в метаданных.
- Недооценивать роль **сетевого shuffle**: слишком тяжёлые `JOIN` без продуманного распределения ключей.

### Что будет, если…

- … coordinатор перегружен большим количеством тяжёлых запросов?  
  → Он становится bottleneck’ом: растёт latency планирования и координации, возможны очереди запросов и timeouts.

- … коннектор к источнику настроен без pushdown’а фильтров?  
  → В Presto/Trino прилетает слишком много данных, которые можно было бы отфильтровать “на месте” — нагрузка на сеть и worker‑ы растёт.

### Проверь себя

1. В чём ключевое отличие архитектуры Presto/Trino от классической MPP‑СУБД, которая сама хранит данные?

<details><summary>Ответ</summary>
Классическая MPP‑СУБД (типа Vertica) **сама владеет данными**: хранит их в своих сегментах, управляет репликацией, индексами.  
Presto/Trino — **“stateless‑движок исполнения”**, который читает данные из внешних источников (S3/HDFS/БД) через коннекторы. Это даёт гибкость (много источников), но удерживает ответственность за хранение и транзакции на этих источниках.
</details>

2. Почему в архитектуре Presto/Trino важен отдельный слой Metastore (например, Hive Metastore)?

<details><summary>Ответ</summary>
Потому что Metastore хранит **схемы таблиц, пути к файлам/partition’ам, статистику**. Без него каждый запрос вынужден был бы сканировать файловую систему и гадать о структуре данных. Metastore позволяет Presto/Trino быстро находить нужные файлы и строить планы запросов, а также согласовывать схему между разными инструментами.
</details>

3. Чем опасна проблема small files для кластера Presto/Trino?

<details><summary>Ответ</summary>
Каждый мелкий файл — это отдельный split, который нужно учесть в плане, открыть, прочитать его метаданные и содержимое.  
Когда таких файлов сотни тысяч, координатор перегружается управлением задач, а worker‑ы тратят много времени на запуск мелких операций вместо обработки крупных блоков данных. Это резко снижает и throughput, и интерактивность.
</details>

### Запомните

- **Presto/Trino — это распределённый SQL‑движок для чтения по множеству источников**, особенно по data lake.
- Архитектура “один координатор — много воркеров” и connector‑подход дают гибкость, но требуют аккуратной работы с метаданными и partitioning.
- Основные враги производительности: small files, плохая схема partitioning, перегруженный координатор и сетевой shuffle без продуманного дизайна запросов.
