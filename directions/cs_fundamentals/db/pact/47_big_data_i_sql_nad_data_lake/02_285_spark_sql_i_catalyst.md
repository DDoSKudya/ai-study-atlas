## 285. Spark SQL и Catalyst

### Цель раздела

Разобраться, **как устроен SQL‑слой в Spark**: что такое Spark SQL, DataFrame API, Catalyst optimizer и физическое исполнение планов,  
как Spark читает data lake (Parquet/ORC/Avro), какие у него сильные и слабые стороны по сравнению с Presto/Trino.

### В этом разделе главное

1. Spark SQL — это **часть Spark**, а не отдельный движок: SQL‑запросы превращаются в планы Spark‑job’ов.
2. Catalyst optimizer строит и переписывает **логический и физический планы**, применяя rule‑based и cost‑based оптимизации.
3. DataFrame/Dataset API — **программируемый слой**, который живёт над теми же планами, что и SQL.
4. Spark заточен под **batch‑обработку больших объёмов** (в т.ч. тяжёлые pipeline’ы), но умеет и интерактивные запросы (с нюансами).
5. Интеграция с data lake и lakehouse (особенно Delta Lake) делает Spark SQL одним из основных инструментов для сложных ETL/ELT и витрин.

### Термины

- **Spark SQL** — модуль Spark, позволяющий выполнять SQL‑запросы поверх распределённых данных.
- **DataFrame / Dataset** — абстракция табличных данных в Spark (строки/объекты с типами), над которой строятся планы исполнения.
- **Catalyst optimizer** — оптимизатор запросов Spark SQL (правила переписывания, cost‑based оценка, выбор физического плана).
- **Logical plan / physical plan** — логический и физический планы исполнения запроса.
- **Stage / task** — единицы работы Spark: stage — набор task’ов без shuffle; task — конкретная работа на executor’е.

### Теория и правила

#### 1) Spark как универсальный вычислительный движок

Spark умеет:

- batch‑обработку (Spark Core, RDD, DataFrame);
- SQL (Spark SQL);
- streaming (Structured Streaming);
- ML (MLlib) и graph‑задачи.

SQL‑слой сидит поверх этого, а значит:

- SQL‑запросы **встраиваются в более сложные pipeline’ы** (чистка, обогащение, ML);
- один и тот же движок обрабатывает и SQL, и последующие шаги.

#### 2) Catalyst optimizer

Путь запроса:

1. SQL → парсер → **initial logical plan**.
2. Набор **rule‑based переписиваний**:
   - упрощения выражений;
   - pushdown фильтров;
   - перераспределение `JOIN`;
   - совместное выполнение `PROJECT` и `FILTER` и т.д.
3. Построение **physical plan**:
   - выбор конкретных операторов (sort‑merge join vs broadcast join и т.д.);
   - выбор стратегий чтения (Parquet, pushdown).
4. Оценка стоимости (cost‑based) на основе статистики, где возможно.

Важно:

- качество планов сильно зависит от **статистики** по данным и настроек кластера;
- оптимизации Catalyst “видны” как в SQL, так и в DataFrame/Dataset API.

#### 3) Execution model и связь с data lake

Spark:

- читает данные **батчами** (обычно крупными block’ами);  
- использует **RDD/Partition** как единицу распределения данных;
- разбивает запрос на **stages** (участки без shuffle) и **tasks** (конкретные работы на executor’ах).

При работе с data lake:

- читаются файлы Parquet/ORC/Avro;
- используются механизмы **partition pruning** и **predicate pushdown**;
- lakehouse‑слой (Delta/Iceberg/Hudi) может давать Spark дополнительную информацию о снапшотах и изменениях.

### Пошагово: SQL‑запрос поверх Parquet в Spark

Пусть есть таблица `events` в Delta/Parquet, и мы выполняем:

```sql
SELECT user_id, COUNT(*) AS cnt
FROM events
WHERE event_dt >= DATE '2025-02-01'
GROUP BY user_id;
```

1. Пользователь отправляет запрос Spark (через `spark-sql`, JDBC или API).
2. Spark SQL строит **logical plan**: `SCAN` → `FILTER` → `AGGREGATE`.
3. Catalyst применяет правила:
   - фильтр по `event_dt` “спускается” ближе к `SCAN`;
   - определяется, что можно использовать partition pruning по `event_dt`;
   - возможно, выбирается стратегия локальной/глобальной агрегации.
4. Строится **physical plan**: чтение файлов, распределение по partition’ам, локальные агрегаты, shuffle, финальная агрегация.
5. План разбивается на **stages**:
   - stage 1: чтение + локальный `GROUP BY` → `user_id, partial_cnt`;
   - stage 2: shuffle по `user_id`, финальный `GROUP BY`.
6. На кластере запускаются задачи (tasks) на executor’ах; данные читаются батчами, агрегации делаются в колонках (если Parquet/ORC).
7. Результат собирается и возвращается клиенту.

### Простыми словами

Spark SQL — это **“прикрученное к Spark умение говорить на SQL”**:

- всё, что можно выразить в SQL, превращается в **последовательность Spark‑операций**;
- ты можешь начинать с SQL, а продолжать обработку уже через DataFrame API/ML/streaming.

Если Presto/Trino — это “быстрый SQL‑фронтенд” по множеству источников, то Spark — это **“фабрика данных”**, где SQL — лишь один из рабочих инструментов.

### Картинка в голове

Представь Spark как **конвейер на заводе**:

- есть цехи: SQL‑цех, ML‑цех, streaming‑цех;
- конвейер возит “ящики с данными” между цехами;
- план (physical plan) — это маршрут, по которому конвейер повезёт данные.

Spark SQL — это **участок конвейера, где каждый шаг описан SQL‑операторами**, но физически это всё те же ленты и роботы Spark.

### Как запомнить

- **Spark SQL = SQL‑лицо универсального batch‑движка**.
- Catalyst optimizer делает с планом примерно то же, что и оптимизаторы PostgreSQL/Oracle, но в контексте распределённого кластера.

### Примеры

- Batch‑витрины: каждую ночь Spark SQL читает сырые логи из S3, чистит, агрегирует и пишет витрины в Delta‑таблицы.
- Ad‑hoc‑аналитика по “средним” объёмам: инженеры запускают Spark SQL‑job’ы с ограниченным временем выполнения для исследовательских задач.
- Интеграция с ML: SQL‑часть готовит фичи (feature store) в DataFrame, которые далее идут в ML‑обучение.

### Практика / реальные сценарии

- **ETL/ELT в lakehouse** — одна из типичных задач Spark SQL: чтение CDC‑логов, нормализация, джойны с измерениями, формирование фактов.
- **Миграция legacy Hive‑скриптов** — перевод существующих Hive‑SQL/QL в Spark SQL для ускорения и упрощения pipeline’ов.
- **Сложные pipeline’ы**: когда одних SQL‑запросов мало, и требуется логика на языке общего назначения (Scala/Python) поверх тех же данных.

### Типичные ошибки

- Использовать Spark SQL как “более тяжёлую версию Presto” для интерактивных запросов — без учёта затрат на инициализацию job’ов.
- Писать SQL так, будто это одна машина: игнорирование shuffle‑стоимости и skew (перекосов данных).
- Не следить за **partitioning и размером файлов**: Spark тоже страдает от small files и неправильных партиций.
- Надеяться на Catalyst, не думая о **качествах статистики** и конфигурации кластера.

### Что будет, если…

- … не настроить auto‑broadcast и join‑стратегии?  
  → Spark может выбрать suboptimal план: например, sort‑merge join вместо broadcast, что сильно увеличит shuffle и время выполнения.

- … использовать Spark SQL только для крошечных таблиц?  
  → overhead на запуск job’ов и планирование может оказаться больше, чем реальная работа; традиционная СУБД будет эффективнее.

### Проверь себя

1. В чём принципиальное отличие Spark SQL от Presto/Trino по роли в архитектуре?

<details><summary>Ответ</summary>
Presto/Trino ориентированы на **интерактивные запросы и федерацию источников**. У них нет собственного batch‑движка общего назначения.  
Spark SQL — это часть **универсального batch‑движка** Spark, где SQL встроен в более широкий контекст (ETL, ML, streaming). Spark чаще используют для длинных pipeline’ов и тяжёлых вычислений, Presto/Trino — для интерактивной аналитики по готовым данным.
</details>

2. Зачем Spark нужен Catalyst optimizer, если уже есть планировщик задач и RDD‑уровень?

<details><summary>Ответ</summary>
Catalyst работает **на SQL/логическом уровне**: он решает, какие именно операции над таблицами нужно сделать, в каком порядке, какие `JOIN` использовать и т.д.  
Планировщик задач и RDD‑уровень работают ниже: они исполняют уже конкретный физический план. Без Catalyst SQL‑слой был бы наивным и неэффективным; без планировщика RDD Spark не мог бы выполнить даже оптимальный план.
</details>

3. Почему Spark SQL и lakehouse‑форматы часто идут в паре?

<details><summary>Ответ</summary>
Потому что lakehouse‑форматы (Delta/Iceberg/Hudi) дают Spark **табличный слой с транзакциями и снапшотами**, а Spark даёт им **мощный execution‑движок** для ETL/ELT и аналитики. Вместе они превращают “набор файлов” в управляемое DWH‑подобное хранилище на дешёвом объектном storage.
</details>

### Запомните

- **Spark SQL — это SQL‑интерфейс к мощному batch‑движку Spark**, с Catalyst‑оптимизатором и глубокими возможностями интеграции с другими задачами.
- Он отлично подходит для тяжёлых ETL/ELT и построения витрин, но не всегда оптимален для чисто интерактивной аналитики по маленьким объёмам.
