## Часть XLVII. Большие данные и SQL поверх data lake

> Цель этой части — **понять, как выполнять SQL‑аналитику по “сырым” большим данным в data lake**,  
> чем отличаются движки Presto/Trino, Spark SQL, Hive, Impala, какие роли у форматов Parquet/ORC/Avro  
> и как lakehouse‑слой (Delta, Iceberg, Hudi) добавляет транзакционность и управление схемой поверх файловых хранилищ.

---

### После этой части ты сможешь

- **объяснять, что такое SQL‑движки поверх data lake**: Presto/Trino, Spark SQL, Hive, Impala — какую задачу решают, как архитектурно устроены и чем отличаются от “классической” СУБД;
- **понимать execution‑модели Presto/Trino и Spark SQL**: координация, worker‑ы, план выполнения, shuffle, влияние на latency и throughput;
- **разбираться в роли форматов Parquet, ORC и Avro**: чем они отличаются, когда и какой формат выбирать, как работает columnar layout и schema evolution;
- **объяснять, как устроен lakehouse‑слой (Delta Lake, Apache Iceberg, Apache Hudi)**: метаданные, снапшоты, time travel, ACID‑транзакции поверх объектного хранилища;
- **выбирать архитектуру для аналитики по большим данным**: когда достаточно “голого” data lake + Presto, когда нужен Spark, когда удобнее Impala/Hive;
- **осмысленно настраивать и использовать SQL‑движки поверх data lake в продакшене**: понимать типичные bottleneck’и (metadata, small files, shuffle), грабли и способы диагностики;
- **избегать распространённых ошибок**: превращение data lake в data swamp, миллионы мелких файлов, неправильный выбор формата/partitioning, смешивание “сырого” и очищенного слоя.

#### Проверь себя (что желательно знать заранее)

1. Почему для понимания Presto/Trino и Spark SQL важно уже различать OLTP и OLAP?

<details><summary>Ответ</summary>
Presto/Trino и Spark SQL изначально проектировались под **аналитические (OLAP) нагрузки**: большие сканы, тяжёлые `JOIN` и агрегации, работа по data lake.  
Если не различать OLTP и OLAP, легко ожидать от них “быстрых транзакций как в PostgreSQL” и разочароваться, не понимая, что архитектура оптимизирована под другие задачи — throughput по большим данным, а не мс‑латентность точечных операций.
</details>

2. Как знания о колоночном хранении и MPP помогают понять форматы Parquet/ORC и движки Impala/Presto?

<details><summary>Ответ</summary>
Parquet и ORC — **колоночные форматы файлов**, а Impala/Presto — **MPP‑движки**, которые читают эти файлы по сети.  
Понимание колоночного layout, сжатия, min/max‑статистики и MPP‑распределения работы по узлам помогает видеть, **какие части запроса реально упираются в диск/сеть/CPU**, и почему, например, большое количество мелких файлов или плохое partitioning так сильно бьёт по производительности.
</details>

3. Зачем перед этой частью иметь интуитивное представление о data lake и ETL/ELT?

<details><summary>Ответ</summary>
SQL‑движки поверх data lake — это **только вершина айсберга**: под ними лежат слои `raw`, `cleaned`, `mart`, процессы ETL/ELT, CDC, partitioning и т.д.  
Если понимать лишь сами движки без контекста data lake, легко решить, что “Presto всё починит” и начать стрелять по сырым, грязным данным, не разделяя слои и не думая о качестве и структуре данных. Интуиция по data lake даёт картину, **где место каждого инструмента**.
</details>

### Краткая шпаргалка (термины по части)

| Термин                         | Что это простыми словами                                                                                                  |
| ----------------------------- | --------------------------------------------------------------------------------------------------------------------------- |
| **Data lake**                 | “Озеро данных”: хранилище файлов (обычно Parquet/ORC/Avro) в объектном сторидже (S3‑подобном), поверх которого читают SQL. |
| **SQL‑движок поверх lake**   | Система, которая умеет выполнять SQL‑запросы, читая данные напрямую из файлов в data lake, а не из “классической” СУБД.    |
| **Presto / Trino**           | Лёгкий распределённый SQL‑движок “для чтения”: коннекторы к куче источников, быстрая интерактивная аналитика.             |
| **Spark SQL**                | SQL‑слой поверх Spark: Catalyst‑оптимизатор, DataFrame API, распределённое исполнение batch‑задач.                         |
| **Hive**                     | Старейший SQL‑слой поверх Hadoop: изначально MapReduce, затем Tez/LLAP; хранит метаданные (Metastore).                    |
| **Impala**                   | MPP‑движок для SQL по HDFS, заточен под низкую латентность запросов по Parquet/ORC.                                         |
| **Parquet**                  | Колончатый формат файлов: постолбцовое хранение, row group/column chunk/pages, статистика, сжатие.                        |
| **ORC**                      | Ещё один колончатый формат, с богатыми индексами и статистикой внутри файла; популярен в Hadoop‑экосистеме.              |
| **Avro**                     | Строчный формат с отдельной схемой и хорошей поддержкой schema evolution; часто для логов/стримов.                        |
| **Schema evolution**         | Возможность менять схему таблицы/файлов (добавлять/удалять поля) без полного переписывания всех данных.                   |
| **Lakehouse**                | Слой поверх data lake, который добавляет ACID‑транзакции, версии, таблицы и индексы (Delta, Iceberg, Hudi).               |
| **Delta Lake**               | Lakehouse‑формат (часто с Spark): транзакции, time travel, schema evolution поверх Parquet‑файлов.                        |
| **Apache Iceberg**           | Табличный формат для data lake: уровни метаданных, hidden partitioning, версионирование, поддержка разных движков.       |
| **Apache Hudi**              | Lakehouse‑формат с упором на инкрементальные upsert/insert и CDC‑сценарии.                                                 |
| **Coordinator / Worker**     | Роли в Presto/Trino/Impala: координатор строит план и управляет запросом, worker‑ы выполняют части плана.                 |
| **Small files problem**      | Проблема “миллиона мелких файлов” в data lake, когда overhead на открытие/чтение метаданных убивает производительность.   |

#### Проверь себя (по шпаргалке)

1. Чем “SQL‑движок поверх data lake” принципиально отличается от классической СУБД вроде PostgreSQL?

<details><summary>Ответ</summary>
Классическая СУБД **сама хранит данные** в своих файлах, управляет буферным кешем, журналом, индексами.  
SQL‑движок поверх data lake читает данные **изнаружи** — из файлов (Parquet/ORC/Avro) в объектном хранилище или HDFS, и опирается на внешний слой (lakehouse/Metastore) для метаданных и транзакций. Он часто не умеет/не хочет делать `UPDATE/DELETE` как классическая БД — основная цель: **чтение и аналитика**.
</details>

2. Зачем в data lake‑экосистеме понадобились именно lakehouse‑форматы (Delta/Iceberg/Hudi), если уже были Parquet и ORC?

<details><summary>Ответ</summary>
Parquet/ORC хорошо решают задачу **эффективного хранения и чтения** (колоночный layout, сжатие, статистика), но **не дают таблиц и транзакций**: сложно гарантировать целостность, делать upsert, вести историю изменений.  
Lakehouse‑форматы добавляют **метаданные, снапшоты, транзакционный журнал, правила schema evolution**, благодаря чему data lake **становится больше похож на СУБД**, сохраняя дешевое хранилище файлов.
</details>

3. Почему проблема small files так болезненна именно в контексте data lake и SQL‑движков?

<details><summary>Ответ</summary>
В объектном хранилище/распределённой ФС **каждое открытие/закрытие файла и чтение его метаданных стоит дорого** (сетевые round‑trip, RPC, метаданные NameNode/Metastore).  
Когда вместо крупных row group’ов есть десятки/сотни тысяч мелких файлов, SQL‑движок тратит большую часть времени не на обработку данных, а на **управление файлами и метаданными**, что убивает и throughput, и latency.
</details>

### Маршрут изучения (простыми словами)

1. Сначала разберёмся с **Presto/Trino** (§284): что такое “SQL‑федерация” по множеству источников, как устроены coordinator и worker‑ы, как выглядит execution model и где границы этого подхода.
2. Затем перейдём к **Spark SQL** (§285): Catalyst‑оптимизатор, физическое исполнение через RDD/задания, DataFrame API и интеграция с data lake.
3. После этого посмотрим на **Hive** (§286): как исторически строился SQL поверх Hadoop, как эволюционировали execution engine (MapReduce → Tez → LLAP), что такое Metastore и зачем он нужен даже для Presto/Trino.
4. Далее разберём **Impala** (§287) как пример MPP‑движка, заточенного под Parquet/ORC и низкую латентность запросов по HDFS.
5. На этой базе детально изучим **форматы Parquet, ORC и Avro** (§288–§289): внутреннюю структуру файлов, статистику, индексы, schema evolution и сравнительные плюсы/минусы.
6. Завершим **lakehouse‑слоем** (§290): Delta Lake, Apache Iceberg, Apache Hudi — как поверх файлов реализуются транзакции, снапшоты и upsert, и как это по‑разному делают разные проекты.
7. В конце используем **справочник, частые сценарии, вопросы и типичные ошибки** — как чек‑лист при проектировании/эксплуатации SQL поверх data lake.

#### Проверь себя (по маршруту)

1. Почему в маршруте сначала идут движки (Presto/Trino, Spark, Hive, Impala), а уже потом форматы и lakehouse?

<details><summary>Ответ</summary>
Потому что читателю проще “зацепиться” за **конкретные инструменты**, которыми обычно и пользуются (Presto, Spark), а уже потом углубиться в детали форматов и lakehouse‑слоя, которые под ними лежат.  
Форматы и lakehouse‑механизмы **общие для многих движков**, поэтому после знакомства с несколькими движками уже легче увидеть, почему все они так завязаны на Parquet/ORC/Avro и Delta/Iceberg/Hudi.
</details>

2. Почему Spark SQL идёт после Presto/Trino, а не до него?

<details><summary>Ответ</summary>
Presto/Trino концептуально проще: “SQL‑движок для чтения по многим источникам”, без тяжёлого batch‑движка и сложной интеграции с ML/streaming.  
Spark — это **целая экосистема** (batch, streaming, ML), и SQL там лишь один из слоёв. Поэтому логично сначала освоить простую картину “SQL → план → распределённое исполнение” на примере Presto/Trino, а уже потом разбираться с Catalyst, DataFrame API и прочими слоями Spark.
</details>

3. Почему lakehouse‑слой вынесен в отдельный завершающий раздел, а не “растворён” внутри рассказа о форматах?

<details><summary>Ответ</summary>
Потому что lakehouse‑форматы решают **отдельный класс задач**: транзакции, истории версий, upsert/mutate поверх файлов.  
Если их рассматривать как “ещё один формат рядом с Parquet/ORC”, теряется главное — они меняют **модель работы с данными** (таблицы, снапшоты, ACID), а не только layout байтов в файле. Отдельный раздел подчёркивает, что lakehouse — это шаг от “набора файлов” к “почти базе данных”.
</details>

2. А куда отнести вопрос: “Почему таблица в Iceberg лучше переносит schema evolution, чем «голый» набор Parquet‑файлов?”

### Как устроены разделы и как пользоваться оглавлением

Разделы (§284–§290) построены по единому шаблону (там, где это уместно):

- **Цель раздела** — что именно ты поймёшь/сможешь делать после чтения.
- **В этом разделе главное** — несколько опорных мыслей.
- **Термины** — мини‑словарик по разделу.
- **Теория и правила** — как это работает внутри движка/формата.
- **Пошагово** — типичный сценарий работы “по шагам” (запрос, чтение файлов, план, транзакция и т.п.).
- **Простыми словами** — объяснение без перегруза формальностями.
- **Картинка в голове** — образ, помогающий запомнить идею.
- **Как запомнить** — короткие формулы/сравнения.
- **Примеры** — SQL‑запросы, конфиги, сценарии.
- **Практика / реальные сценарии** — как тема проявляется “в бою”.
- **Типичные ошибки** — где чаще всего ломают производительность/надёжность.
- **Что будет, если…** — граничные случаи и последствия рискованных решений.
- **Проверь себя** — 2–3 вопроса с ответами под `<details>`.
- **Запомните** — краткая выжимка по разделу.

Оглавление сгруппировано по этапам:

- сначала **движки** (Presto/Trino, Spark, Hive, Impala),
- затем **форматы файлов** (Parquet/ORC/Avro),
- и напоследок **lakehouse и справочник** (Delta/Iceberg/Hudi, общие сценарии и ошибки).

Если тебе нужно:

- **быстро сравнить Presto и Spark для ad‑hoc‑аналитики** — смотри §284 и §285;
- **решить, какой формат выбрать для фактов** — смотри §288–§289;
- **понять, нужен ли тебе Delta/Iceberg/Hudi** — иди в §290 и заключительные разделы.

#### Проверь себя (по устройству разделов и оглавления)

1. Зачем в каждом разделе повторяется структура “цель → термины → теория → практика → ошибки → проверь себя”?

<details><summary>Ответ</summary>
Чтобы обучение было **предсказуемым и мозгу было проще**. Когда читатель знает, что в начале — цель и суть, потом — теория, а в конце — проверка и выжимка, ему не нужно “искать” нужный кусок. Это снижает когнитивную нагрузку и помогает сравнивать разные темы по одним и тем же “точкам опоры”.
</details>

2. Как практически пользоваться оглавлением, если нужно только освежить знания о Parquet?

<details><summary>Ответ</summary>
Найти в оглавлении этап **«Форматы файлов для data lake»**, перейти в §288 (Parquet), и в первую очередь прочитать блоки **«В этом разделе главное»**, **«Простыми словами»**, **«Картинка в голове»** и **«Запомните»**. При необходимости уже нырять в подробности теории/примеров. Это быстрее, чем читать всю часть подряд.
</details>

3. Почему lakehouse‑раздел идёт последним этапом, а не вставлен, скажем, после Parquet?

<details><summary>Ответ</summary>
Потому что lakehouse‑слой **опирается на понимание и движков, и форматов**. Сначала мы разбираем, как SQL‑движки “видят” файлы, затем — как именно эти файлы устроены. Только после этого логично говорить, как поверх этого всего организовать транзакции, версии и upsert — иначе будет ощущение “магии вокруг файлов”.
</details>
