## Справочник по части (SQL поверх data lake)

### Движки → сценарии и особенности

| Движок          | Сильные стороны                                             | Слабые стороны / ограничения                                                 | Типичные сценарии                                   |
|-----------------|------------------------------------------------------------|------------------------------------------------------------------------------|----------------------------------------------------|
| **Presto/Trino** | Интерактивный SQL по многим источникам, низкая latency; rich connectors | Нет собственного хранилища, ограниченная поддержка mutating‑операций       | Ad‑hoc‑аналитика, BI по lake и внешним БД          |
| **Spark SQL**   | Мощный batch‑движок, интеграция с ETL/ML/streaming, lakehouse | Overhead на job‑инициализацию, не всегда оптимален для мелких запросов     | ETL/ELT, витрины, сложные pipeline’ы               |
| **Hive**        | Исторически массовый SQL‑слой на Hadoop, Metastore         | MapReduce/Tez‑based (медленнее), сложная настройка для интерактивности      | Legacy‑отчёты, Metastore как единый каталог        |
| **Impala**      | Низкая latency SQL по Parquet/ORC на HDFS, MPP‑подход      | Привязан к Hadoop/Cloudera‑стеку, не универсален для всех источников       | DWH‑подобная аналитика на Hadoop                   |

### Форматы → плюсы/минусы

| Формат          | Тип хранения | Плюсы                                                | Минусы / ограничения                                  | Типичные использования                        |
|-----------------|-------------|------------------------------------------------------|-------------------------------------------------------|-----------------------------------------------|
| **Parquet**     | Колоночный  | Широкая поддержка, хорошее сжатие и статистика      | Требует продуманных размеров файлов и row group’ов    | Факты и витрины в lake, аналитика             |
| **ORC**         | Колоночный  | Богатые индексы, тесная интеграция с Hive/Impala    | Меньше кросс‑платформенной поддержки, завязан на Hadoop | Таблицы Hive/Impala                           |
| **Avro**        | Строчный    | Отличный schema evolution, интеграция с Kafka       | Не столь эффективен для аналитических сканов          | Логи, события, сырой слой (`raw`)             |

### Lakehouse → особенности

| Формат        | Фокус                         | Ключевые фичи                                        | Кому особенно подходит                            |
|--------------|-------------------------------|------------------------------------------------------|--------------------------------------------------|
| **Delta**    | Spark‑центричность, простота  | `_delta_log`, time travel, schema evolution, merge   | Команды на Spark, Databricks‑экосистема          |
| **Iceberg**  | Мульти‑движковость, метаданные| manifests/snapshots, hidden partitioning, multi‑engine | Организации с разными движками (Spark, Flink, Trino) |
| **Hudi**     | CDC и инкременты              | CoW/MoR, инкрементальные view, upsert/insert         | Потоки изменений, лог‑ориентированные данные     |

## Частые сценарии

### Сценарий 1. Ad‑hoc‑аналитика по сырым логам в S3

- **Контекст**: продуктовая команда пишет события в Kafka → Avro → S3 (raw слой). Аналитики хотят быстро проверять гипотезы.
- **Решение**:
  - Хранить сырые логи в Avro с явной схемой.
  - Поднять Presto/Trino с коннектором к Hive Metastore + S3.
  - Зарегистрировать таблицы в Metastore так, чтобы они указывали на Avro‑файлы.
- **SQL‑пример**:

```sql
SELECT country, COUNT(*) AS events_cnt
FROM raw.events
WHERE event_dt >= DATE '2025-02-01'
GROUP BY country
ORDER BY events_cnt DESC
LIMIT 20;
```

### Сценарий 2. Ежедневные витрины продаж в lakehouse

- **Контекст**: есть OLTP‑БД (PostgreSQL), из неё CDC‑поток попадает в data lake; нужно строить витрины продаж по дням/регионам.
- **Решение**:
  - CDC в Avro/JSON → Spark Structured Streaming → Delta/Iceberg таблицы `stg_orders`, `stg_customers`.
  - Spark SQL nightly job строит агрегаты `fct_sales_daily` в том же lakehouse‑формате.
- **SQL‑пример** (Spark SQL):

```sql
INSERT INTO fct_sales_daily
SELECT
  DATE(order_ts) AS sale_dt,
  region,
  SUM(amount)    AS total_amount
FROM stg_orders
GROUP BY DATE(order_ts), region;
```

### Сценарий 3. BI по lakehouse‑таблицам

- **Контекст**: BI‑инструмент (Tableau/Power BI) должен читать данные напрямую из lakehouse.
- **Решение**:
  - Подключить BI к SQL‑движку (Spark Thrift Server, Trino, Dremio и др.), который понимает lakehouse‑формат.
  - В BI использовать таблицы `fct_*` и `dim_*` как обычные DWH‑таблицы.

## Вопросы для самопроверки по части

1. Объясни на уровне архитектуры различия между Presto/Trino, Spark SQL, Hive и Impala.

<details><summary>Ответ</summary>
- **Presto/Trino** — stateless‑движок исполнения SQL по множеству источников, с одним координатором и множеством worker’ов; нет собственного хранилища.  
- **Spark SQL** — SQL‑слой поверх универсального batch‑движка Spark; запросы превращаются в Spark‑job’ы, Catalyst optimizer строит планы; хорош для ETL/ELT и сложных pipeline’ов.  
- **Hive** — исторический SQL на Hadoop; сегодня ценен в первую очередь Metastore’ом; execution (MapReduce/Tez/LLAP) используется меньше.  
- **Impala** — MPP‑движок на HDFS, с длительно живущими демонами и сильной оптимизацией под Parquet/ORC и низкую latency.
</details>

2. В чём сила и слабость форматов Parquet, ORC и Avro с точки зрения аналитики и эволюции схемы?

<details><summary>Ответ</summary>
- **Parquet/ORC** — сильны в аналитике: колоночное хранение, сжатие, статистика, pruning. Но менее удобны для частой смены схемы и мелких записей.  
- **Avro** — гибок для эволюции схемы и удобен для логов/стримов, но не даёт тех же плюсов для массовой аналитики. Обычно Avro → Parquet/ORC как следующий шаг.
</details>

3. Что даёт lakehouse‑слой (Delta/Iceberg/Hudi) сверх “голого” набора Parquet‑файлов?

<details><summary>Ответ</summary>
Он даёт:
- транзакции (ACID на уровне таблиц);
- snapshots и time travel;
- управляемую schema evolution;
- механизмы upsert/merge без ручной работы с файлами.  
Всё это делает data lake похожим на управляемое DWH‑подобное хранилище, сохраняя плюсы файлового storage.
</details>

4. Почему small files — системная проблема для SQL поверх data lake и как с ней бороться?

<details><summary>Ответ</summary>
Small files увеличивают overhead на:
- открытие/закрытие файлов (сетевые round‑trip);
- чтение footers/метаданных;
- планирование множества крошечных tasks/splits.  
Бороться можно:
- увеличивая целевой размер файлов (через настройки writers/ETL);
- делая периодический compaction (особенно в lakehouse‑таблицах);
- правильно выбирая partitioning и batch‑размеры записей.
</details>

## Типичные ошибки по части

- **“Одна магическая технология решит всё”**: попытка строить всю аналитику только на Presto/Trino или только на Spark, игнорируя сильные и слабые стороны каждого инструмента.
- **Отсутствие чёткого разграничения слоёв data lake** (`raw`/`cleaned`/`mart`), перемешивание Avro‑логов и Parquet‑витрин в одних и тех же каталогах/таблицах.
- **Игнорирование метаданных**: отсутствие централизованного Metastore/каталога, хаотичные схемы, неуправляемая эволюция колонок.
- **Small files everywhere**: отсутствие политики по размерам файлов, компакции и partitioning → резкое падение производительности всех SQL‑движков.
- **“Ручное управление файлами” поверх lakehouse‑таблиц**: удаление/перемещение Parquet‑файлов вне журналов, что ломает ACID‑гарантии и приводит к потерям/дублированию данных.
- **Выбор lakehouse‑формата “по моде”**: не учитываются реальные сценарии (CDC vs batch, необходимость мульти‑движковости, плотная интеграция со Spark и т.д.).

## Резюме части XLVII

- SQL поверх data lake — это не один инструмент, а **экосистема**: Presto/Trino, Spark SQL, Hive, Impala, форматы Parquet/ORC/Avro и lakehouse‑слой (Delta/Iceberg/Hudi).
- **SQL‑движки** дают разные execution‑модели: от интерактивных запросов (Presto/Trino, Impala) до тяжёлых batch‑pipeline’ов (Spark SQL); важно выбирать подходящий инструмент под задачу.
- **Форматы файлов** определяют, насколько эффективно движки могут читать, сжимать и фильтровать данные; Parquet/ORC — база аналитики, Avro — база логов и эволюции схемы.
- **Lakehouse‑форматы** превращают data lake из “наборов файлов” в управляемые таблицы с транзакциями, снапшотами и upsert, что критично для надёжной продакшн‑аналитики.
- Понимание этих слоёв и их взаимодействия позволяет строить **практичные, масштабируемые и понятные архитектуры больших данных**, а не “болота данных” с хаотичной аналитикой.
