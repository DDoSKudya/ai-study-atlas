## Вопросы для самопроверки по разделам

**§120. Docker и БД**

1. Почему без volume данные БД в контейнере теряются при пересоздании контейнера?  
2. Зачем в docker-compose указывать health check для сервиса БД?

<details>
<summary>Ответы §120</summary>
1. Файловая система контейнера эфемерна: при удалении контейнера все изменения в ней пропадают. Volume хранит данные на хосте (или в хранилище Docker), поэтому они не привязаны к жизненному циклу контейнера.  
2. Чтобы зависимые сервисы (приложение) могли стартовать только после готовности БД к приёму подключений (depends_on: condition: service_healthy), что уменьшает ошибки подключения при старте.
</details>

**§121. Kubernetes и БД**

1. Чем volumeClaimTemplates в StatefulSet отличаются от обычного volume в Pod?  
2. Зачем оператору БД CRD (Custom Resource Definition)?

<details>
<summary>Ответы §121</summary>
1. volumeClaimTemplates создают для каждого пода StatefulSet отдельный PVC с уникальным именем (например data-postgres-0, data-postgres-1); при пересоздании пода тот же PVC привязывается снова. Обычный volume в Pod — один общий том на все поды или статический PVC без привязки к идентичности пода.  
2. CRD позволяет описать желаемое состояние БД (число реплик, версия, размер тома) декларативно; оператор по этому ресурсу создаёт и обновляет объекты K8s (StatefulSet, Service, бэкапы и т.д.) — единая точка конфигурации вместо ручного редактирования множества манифестов.
</details>

**§122. Managed БД в облаке**

1. Что такое endpoint в managed БД и что в первую очередь проверить при таймауте подключения к RDS?  
2. Чем Aurora выгодно отличается от обычного RDS PostgreSQL при высокой нагрузке?  
3. Зачем перед миграцией на RDS проверять parameter group и расширения?

<details>
<summary>Ответы §122</summary>
1. Endpoint — hostname БД в сети; по нему приложение подключается. При таймауте в первую очередь проверить Security Group: порт 5432/3306 должен быть открыт для входящих с IP или CIDR приложения.  
2. Aurora даёт автоскейлинг хранилища, быстрее добавление read replicas и обычно быстрее failover; при росте нагрузки на чтение проще масштабироваться.  
3. В managed БД не все параметры СУБД и не все расширения доступны; parameter group задаёт только разрешённый набор. Если нужный параметр или расширение недоступны, после миграции окажется «нельзя так настроить» — лучше проверить до переезда.
</details>

**§123. Облачные NoSQL**

1. Зачем в DynamoDB нужен sort key в дополнение к partition key?  
2. Что такое GSI и чем он помогает при запросах по «чужому» полю?  
3. Что такое «горячая партиция» и как уменьшить риск её появления?

<details>
<summary>Ответы §123</summary>
1. Sort key задаёт порядок внутри партиции и позволяет эффективно делать диапазонные запросы (например «все заказы пользователя за март») без сканирования всей таблицы.  
2. GSI — дополнительный индекс со своим partition key и (опционально) sort key; даёт быстрые запросы по другим полям без полного scan таблицы (scan дорог и медленен).  
3. Горячая партиция — один partition key получает непропорционально много запросов; одна партиция перегружена. Уменьшить риск: выбрать partition key так, чтобы нагрузка распределялась (комбинация ключей, случайный суффикс); избегать ключей с очень высокой частотой обращений.
</details>

**§124. Облачные аналитические БД**

1. В чём разница в оплате между кластером Redshift и serverless BigQuery?  
2. Когда выгоднее serverless аналитика, а когда выделенный кластер?  
3. Почему в BigQuery «обязательно партиционирование и не SELECT *» для снижения счёта?

<details>
<summary>Ответы §124</summary>
1. Redshift: платишь за время работы кластера (часы узлов), даже при простое. BigQuery: платишь за объём данных, прочитанных запросом (и за хранилище); кластер не настраиваешь.  
2. Serverless выгоднее при нерегулярной или переменной нагрузке (редкие отчёты, пики). Кластер — при постоянном потоке отчётов и когда нужна предсказуемая стоимость и производительность.  
3. BigQuery тарифицирует по прочитанным байтам. Партиционирование позволяет в WHERE ограничивать сканирование (только нужные партиции). SELECT * читает все столбцы — максимум байт; нужно выбирать только нужные столбцы и фильтровать по партиции.
</details>

**§125. Backup и restore в облаке**

1. Зачем нужен PITR поверх ежедневных снимков?  
2. Что даёт копирование снимков в другой регион (cross-region)?  
3. Почему после восстановления через PITR приложение нужно переключить на другой endpoint?

<details>
<summary>Ответы §125</summary>
1. Ежедневный снимок восстанавливает состояние на момент снимка; если сбой днём — теряются все изменения после последнего снимка. PITR позволяет восстановиться до любой секунды в окне (до момента перед ошибкой) — минимальная потеря данных (RPO).  
2. При падении всего региона снимки в этом регионе недоступны. Cross-region позволяет поднять копию БД в рабочем регионе — основа disaster recovery.  
3. PITR создаёт новый инстанс с новым endpoint; старый (испорченный) инстанс остаётся. Приложение по-прежнему смотрит на старый endpoint. Чтобы видеть восстановленные данные, нужно обновить connection string на endpoint нового инстанса и перезапустить приложение.
</details>

**§126. Multi-cloud стратегии**

1. Назови три конкретных шага, снижающих vendor lock-in при использовании облачной БД.  
2. Чем «снижение lock-in в одном облаке» отличается от «multi-cloud»?  
3. Нужно ли с первого дня разворачивать приложение в нескольких облаках? Почему?

<details>
<summary>Ответы §126</summary>
1. Использовать стандартный SQL и стандартные драйверы; периодически экспортировать критические данные в переносимом формате (дамп в S3/Blob, Parquet); абстрагировать доступ к данным в коде (репозитории, конфигурируемый connection string).  
2. Снижение lock-in — стандарты, экспорт данных, абстракция; цель — возможность переехать без полной переделки. Multi-cloud — несколько облаков или облако + ЦОД; цель — отказоустойчивость и диверсификация; репликация между облаками, DR-планы.  
3. Не обязательно. Multi-cloud усложняет и удорожает; часто достаточно одного облака + бэкапы и cross-region в том же облаке; снижение lock-in через стандарты и экспорт. Второе облако — при явной необходимости (комплаенс, договор, DR между провайдерами).
</details>

## Резюме части XXIII

**Одним абзацем.** Часть посвящена **инфраструктуре и облакам** для баз данных: запуск БД в **Docker** (образы, docker-compose, volumes, окружения, миграции и health check), развёртывание в **Kubernetes** (StatefulSets, PV/PVC, операторы, ConfigMaps/Secrets, Headless Service и ограничения), выбор **managed БД** в облаке (AWS RDS, Aurora, Azure Database, Google Cloud SQL), **облачные NoSQL** (DynamoDB, DocumentDB, Cosmos DB, Firestore) и **облачную аналитику** (Redshift, Synapse, BigQuery, Snowflake), **бэкапы и restore в облаке** (автобэкапы, PITR, cross-region, снимки, экспорт/импорт) и **multi-cloud стратегии** (репликация между облаками, DR, снижение vendor lock-in, гибрид). Материал опирается на хранение и репликацию (части V, XI), операционку и безопасность (XII) и аналитику (XVI).

**Одна фраза — одно действие (по разделам):**

- **§120:** Используй официальные образы БД в Docker с volume для данных; оркестрируй через docker-compose с health check и при необходимости init-скриптами.
- **§121:** В K8s используй StatefulSet и PVC для БД; рассмотри оператора для автоматизации бэкапов и failover; учитывай ограничения по персистентности и I/O.
- **§122:** Для снижения операционной нагрузки выбирай managed БД (RDS, Aurora, Azure, Cloud SQL); учитывай ограничения параметров и vendor lock-in.
- **§123:** Облачные NoSQL (DynamoDB, Cosmos DB и др.) дают serverless и автоскейлинг; сравнивай с self-hosted по контролю и стоимости.
- **§124:** Облачная аналитика (Redshift, BigQuery, Snowflake) — выбор между кластером и serverless по паттернам нагрузки и оплате.
- **§125:** В облаке включай автобэкапы и PITR; настраивай cross-region для DR; при необходимости экспортируй в S3/Blob/GCS.
- **§126:** Multi-cloud и гибрид повышают отказоустойчивость и снижают lock-in; планируй DR и тестируй восстановление; учитывай сложность и стоимость.

**Итоговые вопросы по всей части (если хочешь проверить связное понимание)**

1. Опиши по шагам: новый разработчик клонировал репозиторий. Как он за минуту получает у себя работающую БД и приложение с одинаковой средой у всей команды? Что обязательно должно быть в репозитории и что он не должен забыть сделать?

<details>
<summary>Ответ</summary>
В репозитории есть docker-compose.yml с сервисами db (образ postgres:16 или mysql) и app. Для БД обязательно указан volume для каталога данных (например pgdata:/var/lib/postgresql/data), иначе данные не сохранятся. Разработчик выполняет docker-compose up (или up -d). Docker поднимает БД, при необходимости выполняет init-скрипты из initdb.d при первом запуске, health check ждёт готовности БД, затем стартует приложение с depends_on: condition: service_healthy. Приложение подключается к host=db. Разработчик не должен забыть: не подключаться к localhost из приложения; при первом запуске убедиться, что volume создан (данные сохранятся между перезапусками).
</details>

2. Почему для БД в Kubernetes используют StatefulSet, а не Deployment? Объясни, что произойдёт с данными при пересоздании пода в обоих случаях.

<details>
<summary>Ответ</summary>
Deployment создаёт поды без стабильных имён; том (PVC) не привязан к идентичности пода. При пересоздании пода новый под может получить другой PVC или пустой — данные БД теряются или смешиваются. StatefulSet даёт подам стабильные имена (postgres-0, postgres-1) и через volumeClaimTemplates создаёт для каждого пода свой PVC (data-postgres-0, data-postgres-1). При пересоздании postgres-0 новый под с тем же именем снова получит тот же PVC — данные на месте. Поэтому для stateful-приложений (БД) в K8s используют StatefulSet.
</details>

3. Команда выбирает между self-hosted PostgreSQL в K8s и AWS RDS. Назови по два аргумента «за» каждый вариант и один общий риск при выборе RDS.

<details>
<summary>Ответ</summary>
За RDS: меньше операционной нагрузки (патчи, бэкапы, мониторинг у провайдера); встроенные бэкапы и PITR, Multi-AZ для отказоустойчивости. За self-hosted в K8s: полный контроль над параметрами СУБД и конфигом; нет привязки к одному облаку (vendor lock-in). Общий риск при RDS: ограничения parameter group и расширений; при росте нагрузки стоимость может быть выше; vendor lock-in — перенос в другое облако потребует миграции.
</details>

4. Зачем включать PITR поверх ежедневных автоматических снимков? Что даёт копирование снимков в другой регион?

<details>
<summary>Ответ</summary>
Ежедневный снимок восстанавливает состояние на момент снимка (например ночь). Если сбой (ошибочное удаление, поломка) произошёл днём, все изменения после последнего снимка потеряны. PITR позволяет восстановиться до любой секунды в окне (например до момента перед ошибкой) — RPO (потеря данных) минимальна. Копирование снимков в другой регион — основа disaster recovery: при падении всего региона (катастрофа, сбой провайдера в регионе) инстансы в этом регионе недоступны; снимки в другом регионе позволяют поднять копию БД там и продолжить работу.
</details>

5. Три конкретных шага, которые снижают vendor lock-in при использовании облачной БД (одним абзацем).

<details>
<summary>Ответ</summary>
Использовать стандартный SQL и стандартные драйверы, не завязываться на эксклюзивные API провайдера. Периодически экспортировать критические данные в переносимом формате (логический дамп в S3/Blob/GCS, открытые форматы) для возможности переноса. В коде приложения абстрагировать доступ к данным (репозитории, конфигурируемый connection string), чтобы смена провайдера сводилась к смене конфига и тестам, а не к массовой переделке кода.
</details>

**Вся часть на одной странице (супер-краткий повтор за минуту).**  
Если нужно быстро освежить память — прочитай этот блок. Каждый пункт = суть раздела; если что-то стёрлось, вернись к соответствующему §.

- **§120 Docker:** Образ БД + **volume** для каталога данных (иначе данные теряются при пересоздании контейнера). docker-compose: сервисы видят друг друга по **имени** (host=db, не localhost). Init-скрипты — **только при первом** запуске (пустой том). Health check и depends_on: service_healthy — приложение ждёт готовности БД.
- **§121 K8s:** Для БД — **StatefulSet**: стабильные имена подов (postgres-0) и **свой PVC каждому** (volumeClaimTemplates). Пересоздали под — тот же PVC, те же данные. **Headless Service** — доступ к поду по имени. Без **StorageClass** PVC останутся Pending. Оператор — автоматизация кластера и бэкапов.
- **§122 Managed БД:** RDS, Aurora, Azure, Cloud SQL — провайдер ставит и ведёт БД; ты подключаешься по endpoint. Multi-AZ — failover в другую зону. Aurora — отдельный движок, автоскейлинг. Минусы: не все параметры, lock-in. Проверять parameter group до выбора.
- **§123 Облачные NoSQL:** DynamoDB (partition + sort key), DocumentDB (MongoDB), Cosmos DB (multi-model), Firestore (real-time). Managed = автоскейлинг, без серверов. Выбор по облаку и модели данных.
- **§124 Аналитика:** Кластер (Redshift) — платишь за время. Serverless (BigQuery) — за обработанные данные. Snowflake — storage и compute раздельно. В BigQuery: партиции, не SELECT *, чтобы не раздувать счёт.
- **§125 Бэкапы:** Автобэкапы + **PITR** (восстановление до секунды). **Cross-region** — снимки в другом регионе для DR. Реплика **не заменяет** бэкап. RPO — допустимая потеря данных, RTO — допустимое время простоя. Тестировать восстановление.
- **§126 Multi-cloud:** Снижение **lock-in**: стандартный SQL, экспорт данных, абстракция в коде. Multi-cloud и гибрид — отказоустойчивость; DR-планы и тесты. Не обязательно с дня один — достаточно одного облака + бэкапы.
