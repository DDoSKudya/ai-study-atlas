## 123. Облачные NoSQL

**Перед чтением (§123):** Раздел про «NoSQL как сервис»: DynamoDB (partition key + sort key), Cosmos DB (несколько API), Firestore (real-time). Важно: в DynamoDB запросы по ключам — дёшево; запросы по произвольным полям (scan) — дорого. **Горячая партиция** = один ключ получает слишком много трафика. Выбор по облаку и модели данных.

**Одной фразой:** Облачные NoSQL — это managed сервисы ключ–значение и документов: AWS DynamoDB, DocumentDB (MongoDB-совместимый), Azure Cosmos DB (multi-model), Google Firestore; автоскейлинг и оплата по использованию; сравнение с self-hosted упирается в контроль, стоимость и vendor lock-in.

### Цель раздела

Понять **какие облачные NoSQL-сервисы** есть (DynamoDB, DocumentDB, Cosmos DB, Firestore), чем они отличаются (модель данных, API, глобальное распределение), и когда выбирать managed вместо развёртывания своего MongoDB, Cassandra и т.д.

### В этом разделе главное (три строки)

1. **DynamoDB** (AWS) — managed key-value/document; таблицы с partition key и опционально sort key; автоматическое масштабирование; оплата за чтение/запись (provisioned или on-demand); нет серверов — полностью serverless.
2. **DocumentDB** (AWS) — совместим с MongoDB API; managed документная БД; подходит для миграции с MongoDB без смены драйвера. **Cosmos DB** (Azure) — multi-model (document, key-value, graph и др.); несколько API (SQL, MongoDB, Gremlin и т.д.); глобальное распределение с настраиваемой согласованностью.
3. **Firestore** (GCP) — документная БД; real-time подписки на изменения; тесная интеграция с Firebase; удобна для мобильных и веб-приложений с синхронизацией в реальном времени.

### Простыми словами

Как и с реляционными БД, провайдеры предлагают «NoSQL как сервис»: не нужно поднимать и администрировать MongoDB или Cassandra. DynamoDB — «таблицы» с гибкой схемой и ключами; Cosmos DB умеет работать в разных режимах (документы, граф, ключ–значение); Firestore заточена под приложения с подпиской на изменения. Плюсы — автоскейлинг и отсутствие серверов; минусы — привязка к API и провайдеру, не все сценарии self-hosted NoSQL покрыты.

### Если ты боишься не понять — самая простая версия (§123)

**Что такое облачный NoSQL буквально?**  
Ты не ставишь MongoDB или Redis на сервер. Ты создаёшь «таблицу» или «коллекцию» в консоли AWS/Azure/GCP — и сразу пишешь и читаешь данные по API. Провайдер сам масштабирует ресурсы (при росте трафика добавляет мощность), ты платишь за операции (чтение/запись) или за выделенную пропускную способность. Серверов «под капотом» для тебя не видно — serverless.

**Три картинки в голове:**

1. **DynamoDB** — как склад с ячейками по номеру. Ты говоришь: «положить в ячейку (partition key + sort key) такой item» или «достать из ячейки». Масштабирование и надёжность — забота AWS.
2. **Cosmos DB** — один склад с разными «входами»: заходишь как в MongoDB — видишь коллекции документов; заходишь как в граф — видишь узлы и рёбра. Один сервис, несколько API по выбору.
3. **Firestore** — склад с «подпиской на полку»: как только в полке что-то изменилось, тебе приходит уведомление (real-time). Удобно для приложений, где клиент должен сразу видеть изменения (чаты, совместное редактирование).

**Одна мысль — пять раз (облачные NoSQL):**

1. **DynamoDB** — key-value/document от AWS; partition key + sort key; оплата за RCU/WCU или on-demand; нет серверов.
2. **Cosmos DB** — multi-model в Azure; один сервис — API SQL, MongoDB, Cassandra, Gremlin; глобальное распределение.
3. **DocumentDB** — MongoDB-совместимый сервис AWS; миграция с MongoDB с минимальными изменениями кода.
4. **Firestore** — документная БД GCP; real-time подписки; тесная связка с Firebase.
5. **Managed vs self-hosted:** managed — быстрый старт, автоскейлинг; self-hosted — полный контроль, при больших объёмах может быть дешевле, но нужна экспертиза.

### Термины

- **DynamoDB** — managed NoSQL (key-value/document) от AWS; partition key + sort key; GSI/LSI; streams для изменений.
- **Cosmos DB** — multi-model БД Azure; несколько API (SQL, MongoDB, Cassandra, Gremlin); глобальное распределение; уровни согласованности (strong, eventual и др.).
- **Firestore** — документная БД GCP; коллекции и документы; real-time listeners; интеграция с Firebase.

### Теория и правила

- **DynamoDB:** таблицы, items с атрибутами. **Partition key** определяет, в какую физическую партицию попадёт запись (все данные с одним значением partition key в одной партиции). **Sort key** — порядок внутри партиции (диапазоны запросов по sort key эффективны). Пример: таблица заказов — partition key = user_id, sort key = order_date; запрос «все заказы пользователя X за март» — один раздел по user_id, диапазон по order_date. **GSI (Global Secondary Index)** — индекс с другими ключами (другой partition + sort); оплата за RCU/WCU или on-demand.

**Ещё раз: что такое GSI (DynamoDB) одной фразой.**  
**GSI** — это **дополнительный индекс** к таблице DynamoDB с **собственным** partition key и (опционально) sort key. Основная таблица может быть ключ user_id + order_date; GSI может быть по order_id (partition) + created_at (sort) — тогда быстрые запросы «найти заказ по order_id» без сканирования всей таблицы. GSI хранится отдельно и тарифицируется по потреблению (RCU/WCU); без GSI запрос по «чужому» полю = полный scan таблицы — дорого и медленно. **LSI (Local Secondary Index)** — индекс с **тем же** partition key, но другим sort key; запросы в пределах одной партиции по альтернативному порядку; создаётся только при создании таблицы (в отличие от GSI, который можно добавить потом).

**Почему partition key и sort key — так важно понять (разжёвано).**  
Partition key — это «адрес ячейки» в распределённой системе: DynamoDB раскладывает данные по партициям (кускам хранилища). Все записи с одним и тем же значением partition key попадают в **одну** партицию. Поэтому если ты выберешь partition key плохо (например один ключ на миллионы записей), одна партиция станет «горячей» — все запросы к ней, остальные простаивают. **Sort key** — порядок внутри этой партиции: ты можешь запрашивать «все заказы пользователя с датой от A до B» одним запросом (range query), без сканирования всей таблицы. Без sort key пришлось бы хранить все заказы пользователя в одном огромном item (массив) — обновления и чтения стали бы тяжёлыми. Итог: partition key = куда положить и как распределить нагрузку; sort key = как упорядочить и как эффективно запрашивать диапазоны.

**Проверь себя (partition key, sort key, GSI)**

1. Зачем в DynamoDB нужен sort key, если partition key уже определяет партицию?  
2. Что такое GSI и чем он отличается от запроса по основной таблице без индекса?  
3. Почему запрос по полю, не входящему в partition/sort key или GSI, в DynamoDB «дорогой»?

<details>
<summary>Ответы</summary>
1. Sort key задаёт порядок внутри партиции и позволяет эффективно делать диапазонные запросы (например «все заказы пользователя за март») без сканирования всей таблицы; без sort key пришлось бы хранить всё в одном item (массив) — тяжёлые обновления и чтения.  
2. GSI — дополнительный индекс со своим partition key и (опционально) sort key; даёт быстрые запросы по другим полям (например по order_id) без полного scan таблицы. Без GSI запрос по «чужому» полю = scan всей таблицы — дорого и медленно.  
3. Такой запрос требует полного сканирования таблицы (scan) — читаются все items; тарификация за все прочитанные данные и высокая задержка. Нужны ключи или GSI под реальные запросы.
</details>

- **Cosmos DB:** контейнеры (таблицы/коллекции); при создании выбираешь API (SQL, MongoDB, Cassandra, Gremlin); репликация в несколько регионов; **consistency levels** от strong (чтение всегда последнее записанное) до eventual (чтение может быть с задержкой).
- **DocumentDB:** совместимость с MongoDB 4.0/5.0; те же драйверы и запросы; миграция с MongoDB с минимальными изменениями кода; managed репликация и бэкапы.
- **Firestore:** иерархия коллекция → документ; офлайн-поддержка (кэш на устройстве); **real-time listeners** — подписка на документ/коллекцию, при изменении приложение получает обновление; правила безопасности задаются в консоли (кто что может читать/писать).

**Managed vs self-hosted:** managed — быстрый старт, автоскейлинг, нет администрирования; self-hosted — полный контроль, возможность вынести на свой ЦОД или другой облак; при больших объёмах и специфичных требованиях self-hosted может быть дешевле, но требует экспертизы.

#### Картинка в голове (облачные NoSQL)

Облачный NoSQL — как «склад с доставкой под ключ»: ты не покупаешь стеллажи и не нанимаешь кладовщиков (серверы и админов); ты говоришь «мне нужны документы с такими ключами» и платишь за операции или пропускную способность. DynamoDB — «склад по ключам» (partition + sort key); Cosmos DB — «склад с разными входами» (один контейнер — документы, граф или ключ–значение в зависимости от API); Firestore — «склад с подпиской на изменения» (real-time обновления в приложение). Self-hosted MongoDB/Cassandra — ты сам строишь и обслуживаешь склад; больше контроля, больше работы.

#### Сравнение облачных NoSQL (кратко)

| Сервис | Модель | API / совместимость | Сильная сторона |
|--------|--------|----------------------|-----------------|
| DynamoDB | Key-value / document | Собственный (AWS SDK) | Serverless, автоскейлинг, интеграция с AWS |
| DocumentDB | Document | MongoDB-совместимый | Миграция с MongoDB без смены кода |
| Cosmos DB | Multi-model | SQL, MongoDB, Cassandra, Gremlin и др. | Один сервис — несколько API; глобальное распределение |
| Firestore | Document | Собственный (Firebase SDK) | Real-time, офлайн, мобильные и веб-приложения |

#### Что может пойти не так (облачные NoSQL)

| Проблема | Причина | Что делать |
|----------|---------|------------|
| Высокая стоимость при росте трафика | Оплата за RCU/WCU или запросы; горячие ключи | Оптимизировать модель данных (партиции); использовать кэш перед БД; рассмотреть reserved capacity (DynamoDB). |
| Нужна миграция на другой облак | Lock-in в API и форматах | Заранее проектировать слой абстракции; экспортировать данные в переносимом формате (JSON, Parquet); тестировать совместимость с целевой БД. |
| Ограничения API (лимиты размера, операций) | Ограничения сервиса | Изучить лимиты (размер item, throughput); шардировать данные или переходить на другой сервис/self-hosted. |
| Запросы по полям без partition/sort key очень медленные и дорогие | DynamoDB оптимизирован под доступ по ключам; запрос по произвольным полям = **scan** всей таблицы | Спроектировать ключи (partition + sort) под реальные запросы; для отчётов по разным полям использовать отдельное хранилище (BigQuery, RDS) или GSI. |

**На примере: когда выбрать DynamoDB, а когда реляционную БД в RDS?**

**Сценарий:** Нужно хранить сессии пользователей (session_id → данные, TTL). Запросы: «получить сессию по id», «записать сессию», «удалить по TTL». Связи между таблицами не нужны, сложные JOIN не нужны. **DynamoDB** подходит: один ключ (partition key = session_id), быстрый доступ по ключу, встроенный TTL, serverless — платишь за операции. **RDS** тоже можно, но придётся управлять инстансом и масштабированием; для чисто ключевого доступа DynamoDB часто проще и дешевле при переменной нагрузке. Если же нужны сложные запросы (агрегации по многим полям, связи между сущностями), реляционная БД (RDS) удобнее.

**Не путай: когда DynamoDB, когда реляционная БД (RDS) — одной таблицей.**

| Нужно | Логичнее |
|-------|----------|
| Доступ по ключу (один id → один документ/запись), простые запросы, переменная нагрузка, TTL | **DynamoDB** (или аналог): serverless, платишь за операции. |
| Связи между таблицами (JOIN), сложные запросы, агрегации по разным полям, транзакции | **RDS** (PostgreSQL, MySQL): реляционная модель, SQL, индексы. |
| И то и другое в одном приложении | Часто используют **обе** (полиглот): RDS для основных данных и отчётов, DynamoDB для сессий, кэша, очередей по ключу. |

#### Мини-практикум (§123)

**Упражнение 1.** Зачем компании выбирают DocumentDB вместо развёртывания своего MongoDB?

<details>
<summary>Ответ</summary>
DocumentDB совместим с API MongoDB — приложение может использовать те же драйверы и запросы. Плюсы managed: не нужно администрировать кластер (репликация, бэкапы, обновления); автоскейлинг; интеграция с AWS (VPC, IAM, CloudWatch). Минусы: привязка к AWS, не все возможности MongoDB могут быть доступны; при очень больших объёмах self-hosted MongoDB может быть дешевле.
</details>

**Упражнение 2.** В DynamoDB таблица с partition key = user_id и sort key = order_date. Зачем нужен sort key, если можно хранить всё в одном item по user_id (например массив заказов)?

<details>
<summary>Ответ</summary>
С sort key можно эффективно запрашивать диапазоны (например «все заказы пользователя за март») и ограничивать объём одного partition (один partition key не должен «раздуваться» миллионами записей — иначе горячая партиция). Если хранить всё в одном item (массив заказов), при росте массива item станет огромным, обновления дорогими, а запрос «заказы за март» всё равно потребует чтения всего item и фильтрации в приложении. Sort key даёт порядок и возможность запроса по диапазону на стороне БД.
</details>

#### Частые вопросы по §123

**В: DynamoDB подходит для отчётов и сложных запросов с фильтрами по разным полям?**  
О: DynamoDB оптимизирован под доступ по ключам (partition + sort) и ограниченные запросы. Сложные фильтры по произвольным полям возможны через scan (сканирование всей таблицы) — дорого и медленно. Для аналитики и сложных отчётов лучше колоночное хранилище (Redshift, BigQuery) или реляционная БД с индексами.

**В: Что такое «горячая партиция» в DynamoDB?**  
О: Partition key определяет партицию. Если один ключ (например один user_id) получает непропорционально много запросов, одна партиция перегружена — это «горячая» партиция. Решение: распределить нагрузку (например добавить суффикс к ключу случайно), или выбрать другой ключ, чтобы трафик распределялся по партициям равномернее.

#### Памятка: 3 вещи (§123)

1. **DynamoDB** — key-value/document, partition key + sort key; serverless; оплата за операции или пропускную способность.
2. **Cosmos DB** — один сервис, несколько API (SQL, MongoDB, Gremlin и др.); глобальное распределение.
3. **Выбор:** по уже используемому облаку, по модели данных (документы, граф, ключ–значение) и по совместимости (MongoDB API и т.д.).

**Пошагово: первые шаги с облачным NoSQL (концептуально).**

1. **Выбор сервиса:** Если уже в AWS и нужен ключ–значение/документы по ключу — DynamoDB. Если нужна совместимость с MongoDB — DocumentDB (AWS) или Cosmos DB с API MongoDB (Azure). Если нужны real-time подписки и мобильное приложение — Firestore (GCP). Cosmos DB — если нужны несколько моделей (документы, граф) в одном сервисе или глобальное распределение.
2. **DynamoDB:** В консоли AWS создай таблицу. Задай имя; выбери **partition key** (обязательно) и при необходимости **sort key**. Продумай модель: partition key должен распределять нагрузку (не один ключ на весь трафик); sort key — для диапазонных запросов внутри партиции. Режим оплаты: provisioned (RCU/WCU заранее) или on-demand (платишь за фактические операции).
3. **Подключение:** Используй SDK провайдера (AWS SDK, Azure SDK, Firebase SDK). Connection string или конфиг региона/имени таблицы — в конфиге приложения. Для DynamoDB нет «подключения к серверу» как в SQL — обращаешься к API по имени таблицы и ключам.
4. **Ограничения:** Проверь лимиты (размер item, throughput). Для аналитики и сложных отчётов по произвольным полям DynamoDB не оптимален — лучше колоночное хранилище (BigQuery, Redshift) или реляционная БД.

**Чек-лист перед использованием облачного NoSQL (проверь да/нет)**

- [ ] Выбрал сервис по облаку и модели данных (key-value, документы, real-time, совместимость MongoDB и т.д.).
- [ ] Для DynamoDB: partition key и при необходимости sort key спроектированы так, что нагрузка распределяется по партициям (нет «горячей» партиции).
- [ ] Понимаю, что сложные запросы по произвольным полям (как в SQL) — через scan, дорого; для аналитики рассмотрел отдельное хранилище.
- [ ] Учтён vendor lock-in: данные и API привязаны к провайдеру; при необходимости экспорт в переносимом формате и абстракция доступа в коде.

### Проверь себя (§123)

1. Чем DynamoDB отличается от запуска своего Redis или MongoDB?  
2. Зачем в Azure выбирать Cosmos DB с API MongoDB вместо отдельного MongoDB?  
3. Что такое «горячая партиция» в DynamoDB и как уменьшить риск её появления?

<details>
<summary>Ответы</summary>
1. DynamoDB — полностью managed: нет серверов, автоматическое масштабирование, оплата за операции или пропускную способность; данные хранятся и реплицируются провайдером. Свой Redis/MongoDB — свои инстансы, настройка репликации, бэкапы, мониторинг.  
2. Cosmos DB с API MongoDB даёт глобальное распределение, встроенную отказоустойчивость и интеграцию с Azure без необходимости управлять кластером MongoDB; подходит при желании остаться в экосистеме Azure и использовать знакомый API.  
3. Горячая партиция — когда один partition key получает непропорционально много запросов (например один user_id с огромной активностью); одна партиция перегружена, остальные недогружены. Уменьшить риск: выбрать partition key так, чтобы нагрузка распределялась (например не один ключ на всех, а комбинация; или добавить случайный суффикс к ключу для равномерного распределения); избегать ключей с очень высокой частотой обращений.
</details>

### Запомните

Облачные NoSQL (DynamoDB, DocumentDB, Cosmos DB, Firestore) дают managed ключ–значение и документы с автоскейлингом и pay-per-use. Выбор зависит от уже используемого облака, модели данных (key-value, документы, граф) и требований к совместимости (MongoDB API, SQL API и т.д.) и к снижению vendor lock-in.

**Резюме §123 в одну строку:** Облачные NoSQL — DynamoDB (partition + sort key), DocumentDB (MongoDB), Cosmos DB (multi-model), Firestore (real-time); serverless и автоскейлинг; выбор по облаку и модели данных.

**Если забыл §123 — вернись сюда.** Одна фраза: Облачные NoSQL — это «NoSQL как сервис»: не поднимаешь MongoDB/Cassandra сам; создаёшь таблицу/коллекцию в консоли и работаешь по API; partition key и sort key в DynamoDB задают, куда попадают данные и как их эффективно запрашивать.
