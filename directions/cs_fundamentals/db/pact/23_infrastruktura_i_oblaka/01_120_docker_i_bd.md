## 120. Docker и БД

**Перед чтением (§120):** В этом разделе ты узнаешь, зачем поднимать БД в Docker, почему без volume данные пропадут, как подключаться из приложения по имени сервиса (не localhost) и почему init-скрипты выполняются только один раз. Главное запомнить: **volume для данных обязателен**, **host=db**, **health check**.

**Одной фразой:** Docker позволяет запускать БД в изолированном контейнере; официальные образы (postgres, mysql) и docker-compose упрощают разработку; named volumes сохраняют данные между перезапусками; окружения и миграции настраиваются через переменные и init-скрипты.

### Цель раздела

Понять **зачем запускать БД в Docker**, как устроены **официальные образы** PostgreSQL и MySQL, как использовать **docker-compose** для оркестрации контейнеров (БД + приложение), как обеспечить **персистентность** через volumes, как различать **окружения** (development, testing) и как запускать **миграции и health checks** в контейнерах.

### В этом разделе главное (три строки)

1. **Официальные образы** `postgres` и `mysql` на Docker Hub — стандартный способ поднять БД одной командой; теги задают версию (например `postgres:16`, `mysql:8.0`); образы рассчитаны на переменные окружения для пароля, имени БД и опционально конфига.
2. **docker-compose** описывает сервисы (БД, приложение, очереди) в одном YAML; общая сеть позволяет обращаться по имени сервиса; **volumes** (named или bind mount) привязывают каталог хоста или именованный том к данным БД — без volume при пересоздании контейнера данные теряются.
3. **Окружения и миграции**: разные compose-файлы или профили для dev/test; init-скрипты (каталог в образе или монтирование) выполняются при первом запуске контейнера; health check гарантирует, что БД готова принимать подключения, прежде чем стартуют зависимые сервисы.

### Простыми словами

Docker упаковывает БД (исполняемые файлы, конфиг, ОС) в образ. Ты запускаешь контейнер из образа — получаешь работающую БД без ручной установки на машину. Для разработки удобно поднять «всё в куче»: БД + бэкенд + фронт через docker-compose; данные не теряются, если использовать volume. Образы БД не хранят данные внутри образа — данные пишутся в примонтированный том (volume). Без тома каждый `docker run` даёт «чистую» БД; с томом — та же БД после перезапуска.

### Если ты боишься не понять — самая простая версия (§120)

**Зачем БД в Docker?**  
Чтобы любой разработчик мог за секунды получить такую же БД, как у всех: одна команда (`docker run` или `docker-compose up`), без «у меня не ставится PostgreSQL». Одинаковая версия, одинаковые расширения — меньше «у меня работает, у тебя нет».

**Что такое образ и контейнер?**  
**Образ** — шаблон: файловая система (ОС + бинарники PostgreSQL/MySQL + конфиги по умолчанию). **Контейнер** — запущенный экземпляр образа. Можно запустить несколько контейнеров из одного образа; у каждого своя изолированная среда (процессы, сеть), но без тома данные у каждого свои и при удалении контейнера пропадают.

**Почему нужен volume?**  
Всё, что записывает БД (таблицы, WAL, индексы), по умолчанию пишется внутрь контейнера. Контейнер «эфемерный»: `docker rm` — и данные исчезли. **Volume** — каталог вне контейнера (на диске хоста или именованный том Docker). Монтируешь его в каталог данных БД (например `/var/lib/postgresql/data`) — при перезапуске или пересоздании контейнера данные остаются в томе.

**Картинка в голове (первая):**  
Стопка контейнеров — как несколько ноутбуков с одинаковой программой. Volume — как общая папка в облаке: на каком «ноутбуке» ни открой — видишь одни и те же файлы. Перезапустил контейнер — «ноутбук» новый, но «папка» (volume) та же, БД «вспоминает» данные.

**Ещё две картинки (запомни хотя бы одну):**

1. **Контейнер — это коробка с программой.** Внутри коробки лежит PostgreSQL и пишет данные «на дно коробки». Выбросил коробку — данные пропали. **Volume** — это когда дно коробки прозрачное и под ним лежит настоящая папка на твоём компьютере. Выбросил коробку, взял новую такую же — подставил ту же папку, и все данные на месте.
2. **БД — это блокнот.** Контейнер — стол, на котором лежит блокнот. Перезапустил контейнер — стол заменили, блокнот сдуло. Если блокнот привязан к **volume** (настоящему ящику стола), то при замене стола ты ставишь тот же ящик — блокнот остаётся. Так «блокнот» (данные БД) переживает перезапуск «стола» (контейнера).

**Одна мысль — пять раз (чтобы точно запомнить про Docker и БД):**

1. **Образ** — шаблон (файлы + программа БД). **Контейнер** — запущенный экземпляр. Данные БД в образ не входят — они пишутся «внутрь» контейнера или в примонтированный том.
2. **Без volume** при удалении или пересоздании контейнера все данные БД теряются. Поэтому для БД **всегда** монтируем volume в каталог данных (`/var/lib/postgresql/data` или `/var/lib/mysql`).
3. **docker-compose** — один файл YAML, в котором перечислены сервисы (БД, приложение). Одна команда `docker-compose up` поднимает всё; сервисы видят друг друга по **имени** (например `host=db`), а не по localhost.
4. **Init-скрипты** выполняются **только один раз** — когда каталог данных пуст (первый запуск). Потом БД уже инициализирована, скрипты не запускаются. Для обновления схемы используй миграции (отдельно от init).
5. **Health check** в compose нужен, чтобы приложение не стартовало раньше БД: `depends_on: db: condition: service_healthy` — «поднимай приложение только когда БД уже принимает подключения».

**Пошагово: что происходит при `docker-compose up` (только цепочка, без кода):**

1. Docker читает `docker-compose.yml`, видит сервис `db` с образом `postgres:16` и томом `pgdata`.
2. Если тома `pgdata` ещё нет — создаётся пустой именованный том. Контейнер запускается, том монтируется в `/var/lib/postgresql/data`.
3. При **первом** запуске каталог данных пуст → PostgreSQL выполняет инициализацию кластера (создаёт системные каталоги). Затем выполняются все скрипты из `/docker-entrypoint-initdb.d/` (если смонтированы). БД готова.
4. При **следующих** запусках в томе уже лежат данные → PostgreSQL просто стартует, init-скрипты **не** выполняются.
5. Health check периодически запускает `pg_isready`; когда он успешен, контейнер помечается как healthy.
6. Сервис `app` с `depends_on: db: condition: service_healthy` стартует только после этого и подключается к `host=db`, порт 5432.

**Ещё раз самыми простыми словами: первый и второй запуск (чтобы точно не путать).**

- **Первый раз** `docker-compose up`: тома `pgdata` нет → Docker создаёт пустой том и монтирует его в контейнер. Внутри контейнера каталог `/var/lib/postgresql/data` пуст. PostgreSQL видит пустой каталог и думает: «это новая установка». Он создаёт все системные файлы БД (инициализация), потом запускает все скрипты из `initdb.d` (если ты их смонтировал). После этого БД готова, данные записаны **в том** `pgdata`. Контейнер можно остановить — данные остались в томе.
- **Второй раз** `docker-compose up` (после down или перезагрузки): том `pgdata` уже есть, и в нём уже лежат файлы БД (созданные при первом запуске). Docker монтирует **тот же** том в новый контейнер. PostgreSQL видит каталог с данными и думает: «это уже инициализированный кластер». Он **не** переинициализирует и **не** запускает init-скрипты — просто стартует поверх существующих данных. Поэтому твои таблицы и данные на месте. Запомни: init-скрипты срабатывают **только когда каталог данных пуст**, то есть только при самом первом создании тома.

**Разбор по «дням» (чтобы навсегда запомнить, когда что происходит):**

| Момент | Что делаешь | Что с томом pgdata | Что делает PostgreSQL | Init-скрипты? |
|--------|-------------|---------------------|------------------------|---------------|
| **День 1, первый up** | `docker-compose up` | Тома ещё не было → создаётся пустой, монтируется в контейнер | Видит пустой каталог → инициализирует кластер (создаёт системные файлы) | **Да** — выполняются все файлы из initdb.d по алфавиту |
| **День 1, после работы** | Создал таблицы, вставил данные | Данные записаны **в том** (файлы в pgdata) | Работает как обычно | — |
| **День 1, down** | `docker-compose down` | Том **остаётся** на хосте (контейнер удалён, том не удалён) | Контейнер удалён | — |
| **День 2, up** | `docker-compose up` | Тот же том pgdata монтируется в **новый** контейнер | Видит каталог с данными → **не** инициализирует заново, просто стартует | **Нет** — каталог не пустой, init не запускается |
| **День 2** | Подключаешься к БД | — | Таблицы и данные на месте | — |

Итог: init = только «День 1, первый up», когда тома не было. Всё остальное время — только старт поверх существующих данных.

1. Чем образ отличается от контейнера и где физически лежат данные БД при использовании volume?  
2. Почему init-скрипты не выполняются при втором запуске `docker-compose up`?  
3. Что произойдёт с данными, если сделать `docker-compose down -v`?

<details>
<summary>Ответы</summary>
1. Образ — неизменяемый шаблон (файловая система + приложение). Контейнер — запущенный экземпляр образа. При использовании volume данные БД лежат в томе (на хосте или в хранилище Docker), а не внутри контейнера; контейнер монтирует этот том в каталог данных.  
2. Init-скрипты выполняются только при инициализации кластера, т.е. когда каталог данных пуст (первый запуск). При втором up том уже содержит данные — PostgreSQL не переинициализирует кластер и не запускает initdb.d.  
3. Флаг `-v` удаляет именованные тома этого compose. Все данные в томах (в т.ч. данные БД) будут удалены. Для сохранения данных использовать `down` без `-v`.

**Почему путь именно `/var/lib/postgresql/data`?**  
Это стандартный каталог, в котором образ PostgreSQL (и MySQL — свой путь) ожидает найти данные кластера. Если смонтировать volume в другой путь (например `/data`), PostgreSQL всё равно будет писать в свой дефолтный путь внутри контейнера — том не будет использоваться для данных. Поэтому в compose всегда указываем том **именно** в `/var/lib/postgresql/data` (для PostgreSQL) или `/var/lib/mysql` (для MySQL). Так все файлы БД оказываются в томе, а не внутри контейнера.

**Типичная ошибка новичка:** Запустил БД в Docker без volume, поработал, налил данных. Сделал `docker-compose down` и потом `up` — контейнер новый, тома для данных не было, БД снова пустая. Решение: в compose всегда указывать volume для каталога данных БД; для уже потерянных данных — только восстанавливать из бэкапа или начать с нуля.

**Не путай: volume и «данные внутри образа»**

| | Данные «внутри» контейнера (без volume) | Данные в volume |
|--|------------------------------------------|------------------|
| **Где физически лежат файлы БД?** | В файловой системе контейнера (внутри слоёв контейнера) | В каталоге на хосте или в хранилище Docker (вне контейнера) |
| **Удалили контейнер (`docker rm`)** | Все данные БД пропали навсегда | Данные остались в томе; новый контейнер может примонтировать тот же том |
| **Когда использовать?** | Никогда для данных БД в разработке/продакшене | Всегда для каталога данных БД |
| **Как указать в compose?** | Не указывать volumes для каталога данных | `volumes: - pgdata:/var/lib/postgresql/data` (named) или `- ./data:/var/lib/postgresql/data` (bind) |

Итог: образ содержит только программу и конфиги; данные должны быть в volume, иначе они эфемерны.

**Ещё раз: зачем подключаться к БД по имени сервиса (host=db), а не по localhost.**  
Внутри контейнера приложения слово **localhost** означает «этот самый контейнер», а не твой компьютер и не контейнер с БД. У контейнера приложения нет PostgreSQL — он в другом контейнере. Поэтому строка подключения с host=localhost из приложения ведёт «в никуда» или к порту внутри своего контейнера, и ты получаешь «connection refused» или таймаут. В сети docker-compose у каждого сервиса есть **имя** (то, что ты написал в YAML под ключом сервиса: `db`, `app`). По этому имени Docker даёт DNS: когда приложение запрашивает хост `db`, DNS возвращает IP контейнера с БД. Итог: из контейнера приложения всегда подключайся к БД по **имени сервиса** (host=db), порт — внутренний порт контейнера БД (5432). С **хоста** (твоя машина) подключаться к БД можно по localhost и порту из секции `ports` (например 5432:5432), потому что Docker пробросил порт контейнера на хост.

**На примере: новый разработчик пришёл в команду**

**Без Docker:** Ему говорят: «Поставь PostgreSQL 16, создай БД myapp, пользователя app с паролем secret, включи расширение uuid-ossp». Он гуглит, ставит, у него может получиться другая версия или другой конфиг — «у меня не воспроизводится баг».  
**С Docker:** В репозитории лежит `docker-compose.yml` с образом `postgres:16`, переменными `POSTGRES_USER=app`, `POSTGRES_DB=myapp`, томом для данных. Он делает `docker-compose up -d` — через минуту у него та же БД, что у всех. Init-скрипт в `./init/01-extensions.sql` с `CREATE EXTENSION IF NOT EXISTS "uuid-ossp";` выполнится при первом запуске. Одинаковая среда у всех — меньше «работает у меня».

### Термины

- **Docker image** — неизменяемый шаблон: файловая система и метаданные (точка входа, переменные). Собирается из Dockerfile или берётся из реестра (Docker Hub).
- **Docker container** — запущенный экземпляр образа; изолированные процессы, сеть, файловая система (с возможностью монтирования томов).
- **Volume (Docker)** — механизм хранения данных вне контейнера. **Named volume** — том с именем, управляется Docker; **bind mount** — каталог хоста примонтирован в контейнер.
- **docker-compose** — инструмент описания мультиконтейнерного приложения в YAML; сервисы, сети, тома; команды `up`/`down` поднимают и останавливают всё описанное.
- **Init script** — скрипт, выполняемый при первом запуске контейнера (например создание БД, пользователя, расширений); в образах postgres/mysql — файлы в `/docker-entrypoint-initdb.d/`.

### Теория и правила

#### Официальные образы БД

**PostgreSQL:** образ `postgres` на Docker Hub (поддерживается сообществом и официально). Основные переменные окружения:

- `POSTGRES_USER` — суперпользователь (по умолчанию `postgres`);
- `POSTGRES_PASSWORD` — пароль (обязателен для продакшена);
- `POSTGRES_DB` — имя БД при первом запуске (по умолчанию совпадает с `POSTGRES_USER`).

Версии задаются тегом: `postgres:16`, `postgres:16-alpine` (меньший образ). При первом запуске контейнера выполняется инициализация кластера в каталоге данных; если в этот каталог примонтирован пустой volume — создаётся новая БД; если там уже есть данные — просто стартует существующий кластер.

**MySQL:** образ `mysql` (официальный). Переменные:

- `MYSQL_ROOT_PASSWORD` — пароль root (в новых версиях обязателен);
- `MYSQL_DATABASE` — БД, создаваемая при первом запуске;
- `MYSQL_USER`, `MYSQL_PASSWORD` — дополнительный пользователь и пароль.

Теги: `mysql:8.0`, `mysql:8.0-oracle` и т.д.

**Теоретическое обоснование персистентности:** данные СУБД должны переживать перезапуск процесса и машины. В контейнере по умолчанию файловая система контейнера эфемерна. Поэтому каталог данных БД **обязательно** должен быть томом (volume или bind mount). Официальные образы ожидают данные в стандартных путях (`/var/lib/postgresql/data`, `/var/lib/mysql`); при монтировании тома в этот путь всё сохраняется.

#### docker-compose для разработки

Типичная структура: один файл `docker-compose.yml` с сервисами `db` и `app` (и при необходимости `redis`, `kafka` и т.д.). Сервисы попадают в одну сеть по умолчанию и обращаются друг к другу по **имени сервиса** (DNS внутри Docker). Например приложение подключается к БД по строке `host=db` (не `localhost`).

**Пример минимального docker-compose с PostgreSQL и volume:**

```yaml
services:
  db:
    image: postgres:16
    environment:
      POSTGRES_USER: app
      POSTGRES_PASSWORD: secret
      POSTGRES_DB: myapp
    volumes:
      - pgdata:/var/lib/postgresql/data
    ports:
      - "5432:5432"
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U app -d myapp"]
      interval: 5s
      timeout: 5s
      retries: 5

volumes:
  pgdata:
```

Пояснения:

- **volumes: pgdata** — именованный том; при первом `up` создаётся, при `down` (без `-v`) сохраняется. Путь `/var/lib/postgresql/data` — стандартный каталог данных PostgreSQL в образе.
- **healthcheck** — контейнер считается «здоровым», когда `pg_isready` успешен; другие сервисы могут зависеть от `condition: service_healthy` и не стартовать, пока БД не готова.
- **ports** — публикация порта на хост; для разработки удобно подключаться с хоста к `localhost:5432`.

**Bind mount** вместо named volume — когда нужно дать контейнеру доступ к конкретной папке на хосте (например конфиг или init-скрипты):

```yaml
volumes:
  - ./init:/docker-entrypoint-initdb.d
```

Файлы из `./init` (например `01-init.sql`) выполняются при первом запуске в **алфавитном порядке**; после инициализации скрипты из `initdb.d` при следующих запусках не выполняются (данные уже есть).

**Ещё раз: в каком порядке выполняются init-скрипты и зачем нумеровать файлы.**  
Образ PostgreSQL (и MySQL) при первом запуске выполняет все файлы из каталога `/docker-entrypoint-initdb.d/` **по имени, в алфавитном порядке**. Поэтому если тебе важен порядок (сначала создать расширения, потом таблицы, потом данные), именуй файлы так, чтобы алфавитный порядок совпадал с нужным: например `01-extensions.sql`, `02-schema.sql`, `03-seed.sql`. Если назвать их `schema.sql`, `extensions.sql`, то сначала выполнится `extensions.sql` (буква e раньше s), потом `schema.sql` — и может быть ошибка, если схема ссылается на расширения. Нумерация в начале имени — простой способ задать порядок без запоминания алфавита.

#### Окружения: development, testing, production

- **Development:** один compose-файл, порты открыты, простые пароли, один инстанс БД; цель — быстро поднять и разрабатывать.
- **Testing:** часто те же образы, но отдельный compose или профиль: изолированная БД для тестов, можно поднимать через `docker-compose --profile test up` или отдельный файл `docker-compose.test.yml`.
- **Production:** БД в Docker в продакшене возможна, но обычно для серьёзных нагрузок выбирают либо Kubernetes с персистентными томами и операторами, либо managed БД в облаке. В продакшене не открывают порты БД наружу без необходимости; секреты (пароли) не держат в compose-файле, а в переменных окружения или секрет-менеджере.

#### Миграции в контейнерах

Варианты:

1. **Ручной запуск:** зайти в контейнер приложения или на хост и выполнить `migrate up` (Alembic, Flyway и т.д.), указывая connection string к контейнеру БД (`host=db`).
2. **Init-скрипт БД:** положить SQL-миграции в `/docker-entrypoint-initdb.d/` — выполнятся только при **первом** создании кластера (пустой volume). Для уже существующей БД не подходит.
3. **Отдельный сервис в compose:** сервис «migrator», который при старте выполняет миграции и завершается; от него зависит сервис приложения (`depends_on: migrator`). Важно: миграции должны быть идемпотентны или проверять «уже применено», иначе повторный запуск контейнера не должен ломать БД.
4. **CI/CD:** миграции выполняются в пайплайне перед деплоем приложения; подключаются к БД (например staging или production) по connection string из секретов.

**Health check** для БД в compose позволяет не стартовать приложение до готовности БД:

```yaml
app:
  depends_on:
    db:
      condition: service_healthy
```

Так уменьшается количество ошибок подключения при старте.

#### Что может пойти не так

| Проблема | Причина | Решение |
|----------|---------|--------|
| После перезапуска контейнера БД пустая | Данные писались внутрь контейнера, volume не использовался | Примонтировать volume в каталог данных БД и пересоздать контейнер (при первом запуске с томом БД инициализируется заново; старые данные в контейнере уже потеряны). |
| Init-скрипты не выполняются | Том уже был создан с данными; init выполняется только при пустом каталоге данных | Удалить volume (`docker volume rm ...`) и поднять заново (потеря данных!) или выполнять миграции вручную/через отдельный сервис. |
| Приложение не может подключиться к БД | Подключение к localhost вместо имени сервиса | Из контейнера приложения использовать хост `db` (имя сервиса), не `localhost`. На хосте — `localhost` и порт из `ports`. |
| Порт 5432 уже занят на хосте | Другая установка PostgreSQL или другой контейнер | Сменить маппинг, например `"5433:5432"`, и подключаться к `localhost:5433`. |
| После `docker-compose down -v` данные пропали | Флаг `-v` удаляет именованные тома вместе с контейнерами | Для сохранения данных не использовать `-v` при обычном down. Использовать `-v` только когда сознательно очищаешь всё (например пересоздаёшь среду с нуля). |

**Ещё раз: docker-compose down без -v и с -v — когда что.**  
**`docker-compose down`** — останавливает и удаляет контейнеры; именованные тома (volumes), объявленные в секции `volumes:` в конце файла, **остаются**. При следующем `up` контейнеры подхватят те же тома — данные БД на месте. **`docker-compose down -v`** — флаг **-v** означает «удалить и тома». Все именованные тома этого compose будут удалены, данные в них пропадут. Используй **down** в обычной работе (остановить на ночь, перезапустить); **down -v** — только когда хочешь полностью очистить среду и начать с пустой БД.

**Ещё раз: что именно писать в connection string (чтобы не гадать).**  
Если приложение работает **в контейнере** в том же docker-compose, что и БД, строка подключения должна указывать на **имя сервиса** БД, а не на localhost. Пример для PostgreSQL: `host=db port=5432 user=app password=secret dbname=myapp` (или в URL: `postgresql://app:secret@db:5432/myapp`). Важно: **host=db** — это имя сервиса из `docker-compose.yml` (ключ под `services:`). Порт 5432 — внутренний порт контейнера БД (тот, что слушает PostgreSQL внутри контейнера). Если бы ты написал `host=localhost`, приложение искало бы PostgreSQL на порту 5432 **внутри своего контейнера** — его там нет, получишь ошибку подключения. С **твоего компьютера** (вне Docker) подключаться к БД нужно так: `host=localhost port=5432` (порт тот, что слева в `ports: "5432:5432"` — порт на хосте).

**Ещё раз: зачем health check и что происходит по шагам.**  
Без health check Docker Compose поднимает контейнеры в порядке зависимостей, но **не ждёт**, пока БД реально начнёт принимать подключения. PostgreSQL может ещё инициализироваться 10–30 секунд; приложение стартует сразу и пытается подключиться — получает «connection refused» и падает или уходит в ретраи. **Health check** — это команда, которую Docker периодически запускает в контейнере БД (например `pg_isready -U app -d myapp`). Пока команда возвращает ошибку, контейнер считается «не готов». Когда `pg_isready` успешен, контейнер помечается как **healthy**. Если у приложения указано `depends_on: db: condition: service_healthy`, Compose **не запустит** приложение, пока БД не станет healthy. Итог: health check + condition: service_healthy = приложение стартует только когда БД уже принимает подключения; меньше падений при старте.

**Проверь себя (docker-compose, host=db, health check)**

1. Почему из контейнера приложения нельзя подключаться к БД по localhost? Какой хост указывать?  
2. Зачем нужен health check для сервиса БД и что даёт `depends_on: db: condition: service_healthy` для приложения?  
3. В каком порядке выполняются init-скрипты в `/docker-entrypoint-initdb.d/` и как задать нужный порядок?

<details>
<summary>Ответы</summary>
1. localhost внутри контейнера приложения — это сам этот контейнер, а не хост и не контейнер БД. Указывать нужно имя сервиса БД в compose (например host=db), по нему Docker даёт DNS в общей сети.  
2. Health check проверяет готовность БД к приёму подключений (например pg_isready). condition: service_healthy означает, что Compose не запустит приложение, пока БД не станет healthy — меньше ошибок подключения при старте.  
3. Скрипты выполняются по имени в алфавитном порядке. Чтобы задать порядок, именуют файлы с префиксом (01-extensions.sql, 02-schema.sql, 03-seed.sql).
</details>

#### Мини-практикум (§120)

**Упражнение 1.** Ты описал в compose сервис `db` с образом `postgres:16`, но не добавил секцию `volumes` для каталога данных. Запустил `up`, создал таблицы, сделал `down` и снова `up`. Что увидишь при подключении к БД и почему?

<details>
<summary>Ответ</summary>
БД снова пустая: таблиц нет. Без volume данные писались внутрь контейнера. При `down` контейнер удалился вместе со своей файловой системой. При новом `up` создался новый контейнер с пустым каталогом данных — PostgreSQL инициализировал кластер заново. Вывод: для данных БД всегда нужен volume.
</details>

**Упражнение 2.** В `docker-entrypoint-initdb.d` положили файл `02-seed.sql` с INSERT тестовых данных. После первого `docker-compose up` данные появились. Потом добавили ещё один файл `03-new-table.sql` и перезапустили `docker-compose up`. Новая таблица не создалась. Почему?

<details>
<summary>Ответ</summary>
Init-скрипты выполняются только при **первом** запуске, когда каталог данных БД пуст. После первого запуска в volume уже лежат данные — при следующих запусках PostgreSQL не переинициализирует кластер и не запускает скрипты из initdb.d. Чтобы применить 03-new-table.sql, нужно либо выполнить его вручную (подключиться и выполнить), либо оформить как миграцию и запускать мигратором при старте приложения.
</details>

**Упражнение 3.** Приложение в compose подключается к БД по строке `host=localhost, port=5432`. Из контейнера приложения подключение падает (connection refused или timeout). Что не так и как исправить?

<details>
<summary>Ответ</summary>
Из контейнера приложения `localhost` — это сам контейнер приложения, а не хост или контейнер БД. В Docker Compose сервисы в одной сети обращаются друг к другу по **имени сервиса**. Нужно использовать `host=db` (имя сервиса БД в compose), порт 5432 (внутренний порт контейнера). Тогда подключение пойдёт к контейнеру БД по внутренней сети Docker.
</details>

#### Частые вопросы по §120

**В: Можно ли хранить пароль БД прямо в docker-compose.yml?**  
О: Для локальной разработки часто так и делают (простой пароль в переменных окружения в compose). Для продакшена или общего репозитория — нельзя: файл может попасть в Git. Используй переменные окружения с подстановкой: `POSTGRES_PASSWORD: ${DB_PASSWORD}`, а значение задавай в `.env` (не коммитить!) или через секрет-менеджер при деплое.

**В: Чем named volume лучше bind mount для данных БД?**  
О: Named volume управляется Docker, обычно лежит в каталоге Docker и бэкапится стандартными средствами Docker. Bind mount — привязка к конкретной папке на хосте: удобно, если хочешь смотреть файлы с хоста или бэкапить эту папку своим скриптом. На Windows/macOS производительность bind mount иногда хуже named volume; на Linux разница меньше. Для «просто сохранить данные» чаще используют named volume.

**В: Почему приложение не видит только что созданные таблицы после миграции?**  
О: Если миграции запускаются в отдельном контейнере или вручную после старта приложения, возможен race: приложение стартует раньше и подключается к БД, где таблиц ещё нет. Решение: в compose указать `depends_on: db: condition: service_healthy` для мигратора и приложения, а мигратор пусть запускается первым и завершается (приложение — после него). Тогда порядок: БД готова → мигратор выполнил миграции → приложение стартует.

**В: Нужен ли Docker для БД в продакшене?**  
О: Можно, но часто для серьёзного продакшена выбирают либо Kubernetes (StatefulSet + оператор) для единой платформы, либо managed БД в облаке (RDS, Azure, Cloud SQL). Причины: в Docker на одном хосте при падении хоста падает и БД; нет встроенного автоматического failover; бэкапы и мониторинг нужно настраивать самим. Docker отлично подходит для разработки и тестов.

**В: Запустил БД без volume, уже налил данных. Потом узнал про volume и добавил его в compose. Сохранятся ли старые данные после down/up?**  
О: Нет. Данные, которые уже записаны, лежали **внутри** старых контейнеров. Когда ты добавляешь volume и делаешь down/up, создаётся **новый** контейнер и при первом запуске к нему примонтируется **пустой** том (или новый именованный том). PostgreSQL видит пустой каталог и инициализирует кластер заново — старых данных там нет. Старые данные остались только в удалённом контейнере, к которому уже нет доступа. Вывод: если данные нужны — до пересоздания с volume сделай дамп (pg_dump/mysqldump) и после первого up восстанови из дампа. Впредь всегда указывай volume с самого начала.

#### Памятка: 3 вещи, которые нельзя забыть (§120)

1. **Volume для каталога данных БД обязателен.** Без него при пересоздании контейнера все данные теряются. В compose: `volumes: - pgdata:/var/lib/postgresql/data` (PostgreSQL) или аналог для MySQL.
2. **Из контейнера приложения подключайся к БД по имени сервиса (`host=db`), а не по localhost.** localhost внутри контейнера — это сам контейнер.
3. **Init-скрипты в `/docker-entrypoint-initdb.d/` выполняются только при пустом каталоге данных (первый запуск).** Для обновления схемы после первого запуска используй миграции, а не добавление файлов в initdb.d.

#### Чек-лист перед использованием БД в Docker (проверь да/нет)

- [ ] В compose для сервиса БД указан volume для каталога данных (`/var/lib/postgresql/data` или `/var/lib/mysql`).
- [ ] Пароли не захардкожены в compose в открытом виде для продакшена/общего репо; используются переменные окружения или секреты.
- [ ] У зависимых сервисов (приложение, мигратор) указано `depends_on: db: condition: service_healthy`, чтобы не стартовать до готовности БД.
- [ ] Из приложения connection string использует хост = имя сервиса БД в compose (например `db`), а не localhost.
- [ ] Если нужны расширения или начальные данные при первом запуске — используешь init-скрипты в `initdb.d`; понимаешь, что они выполнятся только один раз.

#### Повтори вслух (§120) — одна фраза на каждое понятие

- **Образ и контейнер:** «Образ — шаблон, контейнер — запущенный экземпляр; данные БД в образ не входят».
- **Volume:** «Без volume данные БД теряются при удалении контейнера; volume — хранилище снаружи контейнера, переживает пересоздание».
- **docker-compose:** «Один YAML с сервисами; одна команда up поднимает БД и приложение; друг друга видят по имени сервиса».
- **Init-скрипты:** «Выполняются только при первом запуске, когда каталог данных пуст; потом не запускаются».
- **Health check:** «Нужен, чтобы приложение не стартовало раньше БД; depends_on с condition: service_healthy».

**Пошагово: первые шаги с Docker и БД (что сделать на практике).**

1. Установи Docker и docker-compose на свою машину (официальная документация по ОС).
2. Создай каталог проекта и файл `docker-compose.yml`: сервис `db` с образом `postgres:16` (или `mysql:8.0`), переменные POSTGRES_USER, POSTGRES_PASSWORD, POSTGRES_DB (или MYSQL_*), и **обязательно** секция volumes с именованным томом, примонтированным в `/var/lib/postgresql/data` (или `/var/lib/mysql`).
3. Добавь health check для сервиса db (test: pg_isready или аналог для MySQL).
4. Если у тебя есть сервис приложения — добавь его в compose и укажи depends_on: db: condition: service_healthy; в connection string приложения используй host=db (имя сервиса), не localhost.
5. Выполни `docker-compose up -d`; проверь, что контейнеры запущены (`docker-compose ps`); подключись к БД с хоста по localhost и порту из ports (например 5432).
6. Сделай `docker-compose down` и снова `up -d` — данные должны сохраниться (потому что есть volume). Если данных нет — проверь, что том указан именно для каталога данных БД.

### Проверь себя (§120)

1. Зачем при запуске PostgreSQL в Docker обязательно использовать volume для `/var/lib/postgresql/data`?  
2. В чём разница между named volume и bind mount в docker-compose?  
3. Почему init-скрипты в `/docker-entrypoint-initdb.d/` выполняются только один раз?  
4. Ты сделал первый `docker-compose up`, создал в БД таблицы, потом `docker-compose down`. Затем другой разработчик на той же машине сделал `docker-compose up`. Увидит ли он твои таблицы? Почему да или нет?  
5. Чем команда `docker-compose down` отличается от `docker-compose down -v` для сохранения данных БД?

<details>
<summary>Ответы</summary>
1. Без volume данные хранятся только в файловой системе контейнера. При удалении контейнера (docker rm) или пересоздании (docker-compose down && up) эти данные теряются. Volume хранит данные на хосте (или в хранилище Docker), поэтому они переживают перезапуск контейнера.  
2. Named volume — том с именем, создаётся и управляется Docker; данные лежат в каталоге Docker (часто /var/lib/docker/volumes/...). Bind mount — привязка конкретного каталога хоста к пути в контейнере; удобно для конфигов и скриптов, видимых и редактируемых на хосте.  
3. Скрипты в docker-entrypoint-initdb.d выполняются только при инициализации кластера, т.е. когда каталог данных пуст. При следующих запусках контейнера каталог уже содержит данные — инициализация не повторяется, скрипты не запускаются.  
4. Да, увидит — если в compose указан тот же именованный том (pgdata) для каталога данных БД. При down контейнер удалился, но том pgdata остался на хосте. При следующем up создаётся новый контейнер, но он монтирует **тот же** том pgdata, в котором уже лежат данные с таблицами. PostgreSQL при старте видит существующие данные и не переинициализирует кластер — таблицы на месте. Если бы volume не было, при новом up данные были бы пустые.  
5. `docker-compose down` останавливает и удаляет контейнеры, но **не удаляет** именованные тома (volumes). Данные в томах остаются; при следующем `up` контейнеры подхватят те же тома. `docker-compose down -v` (флаг `-v`) **удаляет** именованные тома, описанные в этом compose. После `down -v` все данные в этих томах теряются. Поэтому для сохранения данных БД не используй `-v` при обычном завершении работы; `-v` — только когда сознательно очищаешь всё.
</details>

### Запомните

Docker даёт переносимый способ запуска БД (образы postgres, mysql). Для сохранения данных обязательно используйте volume для каталога данных. docker-compose упрощает оркестрацию БД и приложения, общую сеть и health check. Init-скрипты выполняются только при первом создании БД (пустой volume). Для продакшена с высокими требованиями к надёжности чаще используют Kubernetes с StatefulSets/операторами или managed БД в облаке.

**Резюме §120 в одну строку:** БД в Docker — образ + volume для данных; docker-compose для нескольких сервисов и health check; init-скрипты только при пустом каталоге данных.

**Если забыл §120 — вернись сюда.** Одна фраза: БД в Docker нужен volume для каталога данных, иначе при пересоздании контейнера всё теряется; docker-compose поднимает БД и приложение одной командой, health check ждёт готовности БД перед стартом зависимых сервисов.
