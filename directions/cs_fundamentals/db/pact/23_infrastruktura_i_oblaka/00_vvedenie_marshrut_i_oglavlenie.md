## Часть XXIII. Инфраструктура и облака

> Цель этой части — **научиться запускать и эксплуатировать базы данных в контейнерах (Docker), в оркестраторах (Kubernetes) и в облаке (managed БД, облачные NoSQL и аналитика, бэкапы, multi-cloud)**.

После этой части ты сможешь:

- **использовать Docker для БД**: официальные образы (PostgreSQL, MySQL), docker-compose для разработки, volumes для персистентности, окружения и миграции в контейнерах;
- **понимать БД в Kubernetes**: StatefulSets, PersistentVolumes, операторы (PostgreSQL Operator, MySQL Operator), ConfigMaps/Secrets, Headless Services и ограничения;
- **выбирать managed БД**: AWS RDS (PostgreSQL, MySQL, Aurora), Azure Database, Google Cloud SQL — преимущества, ограничения и когда что использовать;
- **ориентироваться в облачных NoSQL**: DynamoDB, DocumentDB, Cosmos DB, Firestore — managed vs self-hosted, стоимость и vendor lock-in;
- **знать облачную аналитику**: Redshift, Azure Synapse, BigQuery, Snowflake, serverless-опции;
- **строить бэкапы и restore в облаке**: автоматические бэкапы, PITR, cross-region репликация, снимки и экспорт/импорт;
- **продумывать multi-cloud**: репликация между облаками, disaster recovery, снижение vendor lock-in, гибридные решения.

Материал опирается на:

- **часть V** (хранение, WAL, страницы);
- **часть VIII** (реляционные СУБД, подключение);
- **часть XI** (масштабирование, репликация, бэкапы);
- **часть XII** (операционка, безопасность, мониторинг);
- **часть XVI** (аналитика, хранилища данных).

В этой части мы смотрим на БД **как на компонент инфраструктуры**: как поднять её в контейнере для разработки, как разместить в кластере (Kubernetes), когда перейти на managed-сервис в облаке, как бэкапить и восстанавливать в облаке и как снижать риски привязки к одному провайдеру (multi-cloud, гибрид).

### Краткая шпаргалка по части (термины)

| Термин | Что это значит простыми словами |
|--------|----------------------------------|
| **Docker volume** | Именованное или примонтированное хранилище, привязанное к контейнеру; данные в volume сохраняются при удалении контейнера. |
| **docker-compose** | Файл (YAML), в котором описываются сервисы (контейнеры), сети и тома; одной командой поднимается несколько контейнеров (например БД + приложение). |
| **StatefulSet** | Ресурс Kubernetes для stateful-приложений (БД): поды получают стабильные имена (pod-0, pod-1) и каждый — свой PersistentVolume; порядок создания/удаления управляется. |
| **PersistentVolume (PV)** | Кусок хранилища в кластере K8s; привязывается к поду через PersistentVolumeClaim (PVC); данные переживают перезапуск пода. |
| **PersistentVolumeClaim (PVC)** | «Заявка» пода на хранилище: указываешь размер и StorageClass; K8s привязывает к поду подходящий PV (или создаёт динамически). |
| **StorageClass** | «Класс» хранилища в K8s; при создании PVC с указанным StorageClass provisioner автоматически создаёт PV (например облачный диск); без default StorageClass или явного имени PVC могут остаться в Pending. |
| **Headless Service** | Сервис в K8s без кластерного IP (clusterIP: None); DNS возвращает IP отдельных подов; нужен для доступа к конкретному поду StatefulSet по имени (postgres-0.postgres). |
| **Operator (K8s)** | Контроллер, который «знает», как разворачивать и управлять сложным приложением (например PostgreSQL): бэкапы, репликация, обновления по заданным правилам. |
| **Managed БД** | База данных как сервис: провайдер (AWS, Azure, GCP) управляет установкой, патчингом, бэкапами, мониторингом; ты настраиваешь параметры и подключаешься. |
| **Endpoint** | Адрес для подключения к БД в облаке (hostname или host:port), который выдаёт провайдер; после failover или смены инстанса endpoint может измениться — приложение подключается по нему. |
| **Multi-AZ** | Развёртывание инстанса БД в нескольких зонах доступности (AZ) внутри региона; при сбое одной AZ провайдер переключает на другую (failover), данные реплицируются. |
| **AZ (Availability Zone)** | Зона доступности — изолированный дата-центр внутри региона облака (свой источник питания и сети); Multi-AZ = несколько таких зон для отказоустойчивости. |
| **Failover** | Переключение нагрузки на резервный узел при отказе основного (например с мастера на реплику); в managed БД часто автоматический. |
| **Aurora** | Облачная СУБД от AWS; совместима с MySQL/PostgreSQL; масштабирование хранилища автоматическое; реплики для чтения; автоматический failover. |
| **Read replica** | Реплика БД только для чтения; разгружает primary от SELECT-запросов; в RDS/Aurora можно добавить read replica в другой AZ; для failover при падении primary используется отдельная настройка (Multi-AZ). |
| **Snapshot** | Моментальный снимок диска/данных БД на определённый момент времени; используется для бэкапа и восстановления; можно копировать в другой регион. |
| **PITR (Point-in-Time Recovery)** | Восстановление базы до произвольного момента в прошлом (в пределах окна хранения логов); опирается на полный бэкап + WAL/binlog. |
| **RPO (Recovery Point Objective)** | Допустимая потеря данных: до какого момента в прошлом мы согласны «откатиться» (например не более 1 часа данных); определяет частоту бэкапов и retention логов. |
| **RTO (Recovery Time Objective)** | Допустимое время простоя: за сколько времени нужно восстановить работу после сбоя; влияет на выбор DR-стратегии (реплика, холодный стенд и т.д.). |
| **Retention (бэкапов)** | Срок хранения бэкапов и логов (например 7 дней, 35 дней); дольше — больше возможностей PITR и восстановления, но выше стоимость хранения. |
| **Vendor lock-in** | Зависимость от одного провайдера (облака или продукта): перенос на другую платформу дорог или невозможен без переделки. |
| **DR (Disaster Recovery)** | План и средства восстановления после крупного сбоя (регион, ЦОД); задаётся целями RTO (время простоя) и RPO (потеря данных); тесты переключения обязательны. |
| **Гибридное облако** | Часть нагрузки в своём ЦОДе (on-premise), часть — в публичном облаке; сложнее управление и сеть, но гибкость и соответствие требованиям. |
| **RCU (Read Capacity Unit)** | В DynamoDB — единица пропускной способности чтения (1 RCU ≈ 1 строго консистентное чтение до 4 КБ в секунду); при provisioned-режиме задаёшь RCU/WCU заранее; при on-demand платишь за фактические операции. |
| **WCU (Write Capacity Unit)** | В DynamoDB — единица пропускной способности записи (1 WCU ≈ 1 запись до 1 КБ в секунду); вместе с RCU определяет throughput таблицы в provisioned-режиме. |
| **Provisioned / On-demand (DynamoDB)** | **Provisioned** — задаёшь RCU и WCU заранее, платишь за выделенную пропускную способность; предсказуемо при стабильной нагрузке. **On-demand** — платишь за фактические операции чтения/записи; автоскейлинг без настройки; удобно при переменной или непредсказуемой нагрузке. |
| **GSI (Global Secondary Index)** | В DynamoDB и аналогах — дополнительный индекс с собственным partition key (и опционально sort key); быстрые запросы по другим полям; можно добавить к существующей таблице; оплата по потреблению (RCU/WCU или on-demand). |
| **LSI (Local Secondary Index)** | В DynamoDB — индекс с **тем же** partition key, что и таблица, но другим sort key; запросы в пределах одной партиции по альтернативному порядку; создаётся только при создании таблицы. |
| **VPC (Virtual Private Cloud)** | Изолированная виртуальная сеть в облаке; БД и приложение размещают в одной VPC (или связанных) для доступа по приватному IP и правил Security Group. |
| **Security Group** | Виртуальный файрвол: правила «кто с каким портом может подключаться к инстансу» (по IP или CIDR); для БД обычно разрешают только приложение, не 0.0.0.0/0. |
| **CIDR** | Нотация диапазона IP-адресов (например 10.0.0.0/24 — одна подсеть); в Security Group указывают разрешённые CIDR для доступа к БД. |

Используй шпаргалку как **карту**: нашёл термин в первом столбце — во втором видишь суть и зачем это нужно.

**Проверь себя (шпаргалка терминов)**

1. Чем PVC отличается от PV в Kubernetes?  
2. Что такое PITR и чем он лучше только ежедневных снимков?  
3. Зачем нужен Headless Service при использовании StatefulSet для БД?

<details>
<summary>Ответы</summary>
1. PV (PersistentVolume) — это кусок хранилища в кластере (диск). PVC (PersistentVolumeClaim) — заявка пода на хранилище (размер, StorageClass); Kubernetes связывает PVC с подходящим PV, и под монтирует том по этой заявке.  
2. PITR (Point-in-Time Recovery) — восстановление до произвольной секунды в прошлом в пределах окна хранения логов. Ежедневный снимок даёт только состояние на момент снимка; если сбой днём, все изменения после снимка теряются. PITR позволяет «откатиться» до момента прямо перед ошибкой.  
3. Headless Service (clusterIP: None) даёт DNS-имена каждому поду StatefulSet (например postgres-0.postgres-headless). Так можно подключаться к конкретному поду (primary/replica) по имени, а не к одному балансируемому IP.
</details>

### Три предложения — суть всей части

1. **Docker и Kubernetes** позволяют поднять БД в контейнере (образы postgres/mysql, docker-compose, volumes) и в кластере (StatefulSets, PV, операторы); для продакшена с высокими требованиями к персистентности и SLA часто выбирают managed БД в облаке.
2. **Облачные БД** — managed SQL (RDS, Aurora, Azure, Cloud SQL), облачные NoSQL (DynamoDB, Cosmos DB, Firestore) и аналитика (Redshift, Synapse, BigQuery, Snowflake) дают управление «из коробки» и масштабирование, но ограничивают тонкую настройку и создают привязку к провайдеру.
3. **Бэкапы и multi-cloud**: в облаке — автоматические бэкапы, PITR, cross-region репликация и снимки; стратегии multi-cloud и гибрид снижают vendor lock-in и улучшают DR, но усложняют архитектуру и стоимость.

**Понятия по темам.**  
Кратко: о чём каждая группа разделов и зачем она нужна.

| Понятие | Суть |
|--------|------|
| **Контейнеры и оркестрация (§120–121)** | Docker: образы БД, docker-compose, volumes для данных, init-скрипты, health check; из контейнера подключаться по имени сервиса (host=db). Kubernetes: StatefulSet даёт подам стабильные имена и свой PVC каждому поду; Headless Service — доступ по имени; операторы автоматизируют кластер и бэкапы. |
| **Managed БД (§122)** | RDS, Aurora, Azure Database, Cloud SQL — провайдер ставит и ведёт БД; endpoint, Multi-AZ, parameter group; перед выбором проверить доступные параметры и Security Group при подключении. |
| **Облачные NoSQL и аналитика (§123–124)** | NoSQL: DynamoDB (partition key + sort key, GSI), Cosmos DB, Firestore — serverless, оплата за операции. Аналитика: Redshift/кластер (платишь за время) vs BigQuery/serverless (платишь за объём запроса); в BigQuery — партиции и не SELECT * для снижения счёта. |
| **Бэкапы и multi-cloud (§125–126)** | Автобэкапы, PITR (восстановление до секунды), cross-region для DR; реплика не заменяет бэкап. Multi-cloud и гибрид — отказоустойчивость и снижение lock-in; стандартный SQL, экспорт данных, абстракция в коде. |

### Одна цепочка: как связаны разделы части (от разработки до продакшена)

Чтобы не путаться, «где что стоит», представь путь от локальной разработки до отказоустойчивого продакшена.

1. **§120 Docker.** Разработчик поднимает БД и приложение локально через docker-compose; данные в volume, миграции через init или отдельный сервис. Одинаковая среда у всей команды.
2. **§121 Kubernetes.** Команда деплоит приложение в K8s; БД тоже в кластере — StatefulSet, PV, при необходимости оператор. Единая платформа, но персистентность и бэкапы нужно настраивать и тестировать.
3. **§122 Managed БД.** Вместо БД в K8s выбирают RDS/Aurora/Azure/Cloud SQL: меньше операционной работы, встроенные бэкапы и Multi-AZ; приложение подключается по выданному endpoint.
4. **§123–124.** Если нужны NoSQL или аналитика — подключают облачные сервисы (DynamoDB, Cosmos DB, BigQuery и т.д.) рядом с реляционной БД (полиглот персистентности).
5. **§125 Backup.** В managed-сервисе включают автобэкапы и PITR; настраивают cross-region для снимков или реплик; при необходимости экспортируют в S3/Blob для своих правил хранения.
6. **§126 Multi-cloud.** Для DR или снижения lock-in планируют репликацию в другое облако или гибрид; тестируют восстановление и документируют процедуры.

Итого: **§120** — старт в контейнерах; **§121** — БД в оркестраторе; **§122** — переход на managed для простоты; **§123–124** — облачные NoSQL и аналитика; **§125–126** — бэкапы и стратегии с несколькими облаками.

**Краткая таблица: главный выбор по разделам (чтобы не путать и не ошибиться)**

| Тема | Не делай так | Делай так |
|------|----------------|-----------|
| **§120 Docker** | Запускать БД в контейнере без volume для каталога данных | Всегда монтировать volume в /var/lib/postgresql/data (или аналог); использовать health check и depends_on: condition: service_healthy |
| **§120 подключение** | Из контейнера приложения подключаться к БД по localhost | Подключаться по имени сервиса (host=db) в той же сети compose |
| **§121 БД в K8s** | Использовать Deployment для БД с одним общим PVC или emptyDir | Использовать StatefulSet с volumeClaimTemplates; у каждого пода свой PVC |
| **§122 выбор БД** | Выбирать managed только по цене, не учитывая ограничения параметров и lock-in | Сравнивать контроль/параметры, бэкапы, регион, совместимость и стоимость; при особых требованиях — self-hosted или K8s |
| **§125 бэкапы** | Полагаться только на один регион и один тип снимков | Включить PITR и cross-region копирование снимков; тестировать восстановление |
| **§126 lock-in** | Хранить всё в эксклюзивных API одного облака без экспорта | Использовать стандартный SQL, периодический экспорт в переносимом формате, абстракцию доступа к данным |

Используй таблицу для быстрого напоминания: левая колонка — типичные ошибки, правая — правильный подход.

**Проверь себя (структура части)**

1. В каком порядке идут этапы изучения: какие разделы в Этап 1, 2 и 3?
2. Зачем из контейнера приложения подключаться к БД по имени сервиса (db), а не по localhost?
3. Почему для БД в Kubernetes используют StatefulSet, а не Deployment?  
4. Какой этап изучает managed БД и облачные NoSQL и в каком порядке идут разделы внутри этапа?  
5. Что в разделе «Краткая таблица: главный выбор» нельзя делать в §120 и в §125?

<details>
<summary>Ответы</summary>
1. Этап 1: §120–121 (Docker и Kubernetes). Этап 2: §122–124 (managed БД, облачные NoSQL, облачная аналитика). Этап 3: §125–126 (бэкапы в облаке и multi-cloud).  
2. localhost внутри контейнера — это сам контейнер приложения, а не хост и не контейнер БД. В сети Docker Compose сервисы видят друг друга по имени сервиса (DNS). Поэтому хост для подключения к БД = имя сервиса (db).  
3. Deployment даёт поды без стабильных имён и без привязки «один под — один том». При пересоздании под может получить другой том или пустой. StatefulSet даёт подам имена postgres-0, postgres-1 и по volumeClaimTemplates каждому поду свой PVC; при пересоздании тот же под получает тот же PVC — данные сохраняются.  
4. Этап 2. Порядок: §122 Managed БД → §123 Облачные NoSQL → §124 Облачная аналитика.  
5. В §120 нельзя запускать БД без volume для каталога данных; нельзя из контейнера приложения подключаться к БД по localhost. В §125 нельзя полагаться только на один регион и один тип снимков — нужно PITR и cross-region, тестировать восстановление.
</details>
