## Часть XX. Администрирование и тюнинг

> Цель этой части — **научиться осознанно настраивать сервер БД и операционную систему под нагрузку, измерять производительность, понимать, «почему всё тормозит», и крутить параметры не по «магическим спискам из блогов», а по фактам и метрикам**.

После этой части ты сможешь:

- **готовить Linux‑сервер под базу данных**: huge pages, `vm.swappiness`, I/O‑планировщик, файловая система, `ulimit`, NUMA;
- **собирать метрики запросов и профилировать производительность** с помощью `pg_stat_statements`, `pg_stat_monitor`, slow query log, `pt-query-digest`, CPU‑профилировщиков (`perf`) и flame graph’ов;
- **запускать бенчмарки** (`pgbench`, `sysbench`, TPC‑подобные сценарии) и правильно интерпретировать результаты (TPS, latency p50/p95/p99);
- **читать мониторинг производительности**: wait events, блокировки, temp‑файлы, cache hit ratio, replication lag, долгие запросы;
- **тюнить память** (shared buffers, work_mem, effective_cache_size, OOM‑риски) и **I/O** (random/seq_page_cost, checkpoints, read‑ahead) под конкретную нагрузку;
- **использовать параллелизм и JIT‑компиляцию** в PostgreSQL (и аналогичные механизмы в других СУБД), понимая их выгоды и ограничения.

Материал опирается на:

- **части V–VII** (хранение, индексы, оптимизация запросов);
- **часть XII** (операционная часть и безопасность, мониторинг, алерты);
- **часть XIV** (партиционирование и внутренности хранилища);
- **часть XVII–XVIII** (детали PostgreSQL и MySQL, конфигурация, расширения);
- **часть XI** (масштабирование и отказоустойчивость: репликация, лаг, бэкапы).

В этой части мы смотрим на БД **как на систему, которую можно измерять и настраивать**: не «какие ручки крутить», а **откуда брать факты** (топ запросов, wait events, метрики ОС) и **как связывать симптомы с причинами**, чтобы менять параметры осознанно.

### Краткая шпаргалка по части

**Термины (одним списком).**  
Ключевые слова этой части — каждое в одном предложении. Держи под рукой при чтении разделов.

| Слово | Что это значит простыми словами |
|-------|----------------------------------|
| **Swap** | Часть диска, куда ОС «выгружает» данные из RAM, когда памяти не хватает. Для БД это почти всегда плохо: читать из swap в сотни раз медленнее, чем из RAM. |
| **vm.swappiness** | Число от 0 до 100. Показывает, насколько охотно Linux будет выталкивать память в swap. Для БД ставят 1–10, чтобы swap почти не трогали. |
| **Huge pages** | Большие куски памяти (2 МБ вместо 4 КБ). Меньше возни с «картой памяти» внутри CPU → быстрее работа с большими объёмами данных. |
| **TLB** | Маленький кэш внутри процессора: «этот виртуальный адрес → вот эта физическая страница». При 4 КБ страницах он быстро переполняется; при 2 МБ — реже. |
| **I/O‑планировщик** | Алгоритм в ядре Linux: в каком порядке отдавать диску запросы на чтение/запись. Для SSD часто ставят простой (`mq-deadline` или `none`), для HDD — умный (`bfq` и т.п.). |
| **NUMA** | Когда у сервера несколько «островков» CPU+памяти: доступ к «своей» памяти быстрее, к «чужой» — медленнее. Для БД важно не раскидывать процесс по всем островам. |
| **ulimit / nofile** | Лимит: сколько файлов (и сокетов) может открыть один процесс. БД открывает много файлов; если лимит маленький, будет ошибка «too many open files». |
| **pg_stat_statements** | Расширение PostgreSQL: запоминает каждый тип запроса и считает, сколько раз он выполнялся и сколько в сумме занял времени. Нужно, чтобы найти «кто съедает ресурсы». |
| **total_time / total_exec_time** | Суммарное время всех выполнений одного и того же запроса (по форме). Показывает «общий ущерб» от запроса, а не скорость одного вызова. |
| **mean_time / mean_exec_time** | Среднее время одного выполнения запроса. Может быть маленьким, но если запрос вызывают миллион раз, в сумме он всё равно дорогой. |
| **Slow query log** | Файл, куда пишутся только те запросы, которые выполнились дольше заданного порога (например, 200 мс). Удобно искать «выбросы». |
| **Wait event** | Ответ на вопрос «чего сейчас ждёт этот запрос?» — диск, блокировку, сеть, внутренний замок и т.д. Помогает понять, не SQL тормозит, а что‑то другое. |
| **TPS / QPS** | Транзакций (или запросов) в секунду. Грубая мера «пропускной способности» системы. |
| **Latency p50 / p95 / p99** | Задержка: медиана (50% запросов быстрее), 95‑й перцентиль (5% медленнее), 99‑й. p95/p99 показывают «плохие» случаи, которые чувствует пользователь. |
| **shared_buffers** | Кэш PostgreSQL в памяти: сюда попадают страницы таблиц и индексов. Чем больше (в разумных пределах), тем реже читаем с диска. |
| **work_mem** | Память на одну операцию сортировки или хеш‑джойна внутри запроса. Выделяется на каждую такую операцию; несколько запросов умножают потребление. |
| **effective_cache_size** | Не выделяет память, а «подсказка» планировщику: «сколько данных влезет в кэш». Влияет на выбор плана (seq scan vs index scan). |
| **OOM** | Out Of Memory: память кончилась, ядро убило процесс (часто PostgreSQL). Обычно из‑за слишком большого `work_mem` или слишком многих соединений. |
| **random_page_cost / seq_page_cost** | Оценка «стоимости» для планировщика: насколько дорого прочитать одну страницу случайно и одну подряд. На SSD случайное чтение не такое страшное, как на HDD. |
| **Checkpoint** | Момент, когда PostgreSQL сбрасывает накопленные изменения с страниц из памяти на диск. Если чекпоинты слишком частые или резкие — бывают пики нагрузки и латентности. |
| **Gather / parallel worker** | Узлы плана запроса: «несколько воркеров делают работу параллельно, один процесс собирает результат». Признак того, что параллелизм включился. |
| **JIT** | Just‑In‑Time компиляция: части запроса превращаются в машинный код на лету. Выгодно для очень тяжёлых запросов; для простых — лишние накладные расходы. |
| **queryid** | Уникальный числовой идентификатор «формы» запроса в pg_stat_statements; не меняется между рестартами (при save=on). Нужен, чтобы сопоставлять один и тот же запрос в разных снимках и в логах. |
| **Deadlock** | Взаимная блокировка: два и более запроса ждут друг друга по замкам. PostgreSQL сам обнаруживает deadlock, откатывает одну из транзакций и пишет в лог; исправляют порядком взятия блокировок в приложении. |

Когда будешь читать разделы, возвращайся к этой таблице: «вот это слово я уже видел в словарике».

### Три предложения — суть всей части (прочитай сейчас и в конце)

1. **Сначала измеряй** (топ запросов, wait events, блокировки, метрики ОС), **потом меняй настройки** — иначе это гадание.
2. **ОС и лимиты** (swap, диск, nofile) — фундамент: если они «сломаны», тюнинг параметров БД почти не поможет.
3. **Память в БД — это бюджеты** (`shared_buffers`, `work_mem` и т.д.): нельзя «всем дать побольше», иначе OOM; увеличивай точечно и по симптомам.

Если запомнишь только это — ты уже будешь действовать разумнее, чем тот, кто крутит «магические списки» без измерений.

**Понятия по темам.**  
Кратко: о чём каждая группа разделов и зачем она нужна.

| Понятие | Суть |
|--------|------|
| **Настройка ОС для БД** | Подготовка Linux под «тяжёлую» БД: huge pages, низкий `vm.swappiness`, правильный I/O‑планировщик и файловая система, отключение лишних фоновых процессов. |
| **Профилирование** | Измерение, **где реально тратится время**: статистика запросов (`pg_stat_statements`, `pg_stat_monitor`), slow‑лог, CPU‑профилировщики, flame graph’ы. |
| **Бенчмаркинг** | Повторяемые тесты (`pgbench`, `sysbench`, TPC‑сценарии), которые дают цифры TPS и latency для сравнения конфигураций и железа. |
| **Мониторинг производительности** | Непрерывный сбор метрик: wait events, блокировки, temp‑файлы, cache hit ratio, репликация, долгие запросы. |
| **Тюнинг памяти** | Настройки `shared_buffers`, `work_mem`, `maintenance_work_mem`, `effective_cache_size` и их влияние на кэш, сортировки и риск OOM. |
| **Тюнинг I/O** | Стоимости страниц (`random_page_cost`, `seq_page_cost`), параллелизм диска (`effective_io_concurrency`), чекпоинты и read‑ahead. |
| **Тюнинг параллелизма** | Параметры `max_parallel_workers`, `max_parallel_workers_per_gather`, `parallel_*_cost`, JIT‑компиляция и когда она реально помогает. |

Используй шпаргалку как **карту**: нашёл термин или понятие в первом столбце — во втором видишь суть и зачем это нужно.

### Маршрут изучения в этой части

Рекомендуемый порядок:

1. **§99. Настройка ОС для БД** — сначала наводим порядок на уровне хоста: память, swap, диски, файловая система, лимиты.
2. **§100. Профилирование производительности** — учимся измерять, где тратится время (запросы, CPU, память).
3. **§101. Бенчмаркинг** — получаем базовый «термометр» системы: TPS, latency, сравнение конфигураций.
4. **§102. Мониторинг производительности** — превращаем разовые измерения в постоянное наблюдение.
5. **§103. Тюнинг памяти** — настраиваем память в СУБД по симптомам, а не «наобум».
6. **§104. Тюнинг I/O** — крутим параметры, связанные с диском и WAL.
7. **§105. Тюнинг параллелизма** — включаем параллельные планы и JIT там, где это выгодно.

Во всех разделах этой части используется **единый шаблон**, чтобы идеи повторялись в разных формах:

**Оглавление** сгруппировано **по этапам изучения**: сначала ОС и измерения (§99–101), затем мониторинг и тюнинг (§102–105). В таблицах под каждым этапом — ссылки на разделы, краткое содержание и ключевые понятия. В конце части — **справочник по части**, **вопросы для самопроверки** (по 2–3 на раздел, ответы под спойлером), **итоговая карта** (симптом → причина → что проверить) и **шпаргалка** (команды и запросы).

**Как пользоваться оглавлением.**

1. **Выбери этап** (например, «Этап 1. ОС и измерения») — это твой следующий шаг в изучении.
2. **Пройди по строкам таблицы** — в «Содержание» указано, о чём раздел; в «Ключевые понятия» — термины, которые нужно знать после изучения.
3. **Переходи по ссылке** в первом столбце — попадёшь в раздел с целью, терминами, теорией, блоком «Проверь себя» (2–3 вопроса с ответами под спойлером) и «Запомните».
4. **Справочник и проверка** в конце части — когда этапы пройдены, используй их для повторения и самопроверки.
