## 180. Стратегии партиционирования

### В этом разделе главное

- Стратегию партиционирования выбирают **из паттернов запросов, объёма данных и требований к retention**, а не по принципу «по дате всегда хорошо».
- Варианты: по времени, по диапазону, по списку значений, по хешу и их комбинации (composite).
- Ошибка на этом уровне приводит к тому, что партиционирование не ускоряет запросы, а **усложняет схему и операционку**.

---

### Простыми словами

Представь, что у тебя есть огромная таблица `events`, где хранятся:

- миллионы лог‑записей;
- события мониторинга;
- заказы.

В один момент:

- `DELETE WHERE created_at < now() - interval '90 days'` начинает длиться часами;
- индекс на `created_at` разрастается до десятков гигабайт;
- nightly‑аналитика «притормаживает» прод.

Партиционирование говорит:

> «Давай не будем складывать все события в один гигантский сундук,
>  а разобьём их на несколько поменьше и более управляемых».

Дальше ключевой вопрос: **по какому признаку резать**, чтобы:

- удобно было **удалять старое**;
- запросы **не лазили по всем партициям**;
- нагрузка **распределялась равномерно**.

---

### Теория и правила

#### Партиционирование по времени

Классический сценарий: логи, метрики, события.

```sql
CREATE TABLE events (
  id         bigserial,
  created_at timestamptz NOT NULL,
  payload    jsonb        NOT NULL
) PARTITION BY RANGE (created_at);
```

Партиции — чаще всего по **месяцам или дням**:

```sql
CREATE TABLE events_2025_01
  PARTITION OF events
  FOR VALUES FROM ('2025-01-01') TO ('2025-02-01');
```

Плюсы:

- retention реализуется как `DROP TABLE events_YYYY_MM`;
- partition pruning по `WHERE created_at BETWEEN ...` работает естественно.

Минусы:

- последняя партиция часто становится **самой «горячей»**;
- если запросы часто бегают по нескольким годам, выгода уменьшается.

#### Партиционирование по диапазону (RANGE по другому полю)

Та же идея, но ключ — не время:

- числовой ID (`user_id`, `order_id`);
- диапазон score/версии.

Диапазоны должны быть:

- **непересекающимися**;
- как правило, **смежными**, чтобы не оставлять «дыр».

Сценарий:

- распределяем ID по диапазонам, чтобы контролировать **размер партиций**;
- запросы `WHERE id BETWEEN ...` работают на небольшой подвыборке партиций.

#### Партиционирование по списку (LIST)

Используем, когда:

- есть **конечный набор категорий**: страны, тарифы, кластеры;
- запросы почти всегда содержат фильтр по этой категории.

```sql
CREATE TABLE customers (
  id      bigserial,
  country text NOT NULL,
  name    text NOT NULL
) PARTITION BY LIST (country);

CREATE TABLE customers_eu
  PARTITION OF customers
  FOR VALUES IN ('DE', 'FR', 'ES', 'IT');
```

Плюсы:

- `WHERE country = 'DE'` → попадаем ровно в одну партицию;
- можно настраивать индексы по‑разному для разных групп стран.

Минусы:

- большое количество категорий → много партиций, overhead;
- приходится часто **создавать новые партиции**, если набор стран/категорий расширяется.

#### Партиционирование по хешу (HASH)

Идея:

- берём некоторый ключ (часто `user_id`);
- считаем его хеш;
- по остаткам от деления хеша на `N` раскидываем строки по партициям.

```sql
CREATE TABLE user_events (
  user_id    bigint       NOT NULL,
  created_at timestamptz  NOT NULL,
  payload    jsonb        NOT NULL
) PARTITION BY HASH (user_id);
```

Плюсы:

- хорошая **равномерность распределения**;
- удобно для балансировки нагрузки по партициям/нодам.

Минусы:

- **диапазонные запросы по `user_id`** перестают быть эффективными;
- изменение количества партиций требует **перебалансировки** (resharding).

#### Composite‑партиционирование

Комбинируем:

- «удобное удаление» по времени,
- с «равномерной нагрузкой» по ключу.

Например:

```sql
CREATE TABLE events (
  user_id    bigint,
  created_at timestamptz,
  payload    jsonb
) PARTITION BY RANGE (created_at);
```

А внутри `events_2025_01`:

- можно ещё разрезать по hash(`user_id`).

Это:

- упрощает retention (удаляем партиции месяцами);
- и позволяет распределять нагрузку по пользователям внутри месяца.

---

### Практические правила выбора

1. **Если данные естественно стареют** → начинай с **партиционирования по времени**.
2. **Если запросы почти всегда по диапазонам** (`BETWEEN`, `>`, `<`) по какому‑то полю → **RANGE по этому полю**.
3. **Если запросы почти всегда по конкретной категории** (`WHERE country = 'US'`) → **LIST по этому полю**.
4. **Если ключ сильно влияет на баланс нагрузки** (много запросов по `user_id`) → рассматривай **HASH по этому ключу** (часто в комбинации с временем).
5. Старайся, чтобы **ключ партиционирования обязательно фигурировал в типичных запросах**, иначе partition pruning не поможет.

---

### Пошагово: как выбрать стратегию для своей таблицы

Представь, что к тебе приходит существующая таблица, и нужно решить, **как её партиционировать**.  
Можно идти по простому чек‑листу.

**Шаг 1. Сформулируй цель.**

Ответь на два вопроса (лучше записать прямо в комментарии к задаче):

- «Зачем мне вообще партиционирование **для этой таблицы**?»  
  Примеры ответов:
  - быстрее удалять старые данные;
  - ускорить выборки за последние N дней;
  - равномернее распределить нагрузку по пользователям;
  - упростить миграции/перенос между нодами.
- «Какие **запросы самые важные** к этой таблице?»  
  Напиши 3–5 реальных запросов (или псевдо‑SQL), которые:
  - часто выполняются;
  - самые тяжёлые;
  - критичны по SLA.

Если цель не получается сформулировать чётко — это тревожный знак: возможно, партиционирование вообще не нужно.

**Шаг 2. Посмотри, по каким полям фильтруют чаще всего.**

Возьми твои 3–5 ключевых запросов и для каждого:

- выпиши поля из `WHERE` и `JOIN`;
- отметь, какие из этих полей:
  - **есть почти во всех запросах**;
  - или **сильно ограничивают объём данных** (селективны).

Часто это:

- `created_at` / `date`;
- `user_id` / `customer_id`;
- `country` / `region`;
- «логический сегмент» (`tenant_id`, `project_id`).

**Шаг 3. Реши, есть ли естественная «ось времени».**

Спроси себя:

- «Эти данные **имеют срок годности**?»
- «Есть ли запросы вида `за последние N дней/месяцев`?»

Если да:

- почти всегда разумно **сделать первую ось партиционирования по времени**;
- дальше можно думать о второй оси (composite).

**Шаг 4. Сопоставь ось запросов и список стратегий.**

Теперь сравни:

- если запросы почти всегда по интервалам времени → **RANGE по времени**;
- если всегда по интервалам ID/score → **RANGE по этому полю**;
- если всегда по конкретной категории (`=`, `IN`) → **LIST**;
- если цель — равномерно размазать нагрузку, а естественной оси немного → **HASH**.

Важно: **не пытайся одним партиционированием удовлетворить все возможные запросы**.  
Фокусируйся на **ключевых**.

**Шаг 5. Прикинь размер и количество партиций.**

Для каждого варианта ответь:

- сколько строк/гигабайт будет в одной партиции **через год**;
- сколько всего будет партиций;
- удобно ли:
  - делать `DROP` таких партиций;
  - поддерживать индексы;
  - бегать по ним аналитикой.

Грубое правило:

- партиций **в десятки–сотни** ещё нормально;
- **сотни–тысячи** — это уже «красный флаг», если нет очень хорошей причины.

**Шаг 6. Задокументируй выбранную стратегию.**

Прямо в репозитории (в README/комментарии к миграции) запиши:

- какой ключ партиционирования выбран;
- почему (цель, ключевые запросы);
- как будет выглядеть retention/архивирование.

Это защитит от ситуации, когда через год никто не понимает, «почему тут именно так» и можно ли это менять.

---

### Картинка в голове

Представь большой склад с коробками (строки таблицы).

- **Range по времени** — это комнаты «Январь», «Февраль», «Март»: старую комнату можно просто закрыть и освободить место.
- **List** — это комнаты «EU», «US/CA», «APAC»: каждая коробка идёт в комнату по стране.
- **Hash** — это когда каждая коробка отправляется в один из `N` стеллажей по номеру (`hash(id) % N`), чтобы везде было примерно поровну.
- **Composite** — многоуровневая схема: сначала комната по месяцу, внутри — стеллажи по хешу.

---

### Типичные ошибки

- Партиционировать **«на будущее»**, без анализа реальных запросов.
- Делать **слишком мелкие партиции** (по дню/часу при небольшом объёме данных).
- Выбирать ключ партиционирования, который **не фигурирует в WHERE** типичных запросов.
- Ставить **hash‑партиционирование**, а потом удивляться, почему диапазонные запросы по этому ключу не ускоряются.
- Не закладывать заранее **retention и архивирование**, оставляя старые партиции висеть бесконечно.

---

### Проверь себя

<details>
<summary>1. Почему стратегия партиционирования должна исходить из паттернов запросов, а не только из структуры данных?</summary>

Потому что:

- partition pruning и partition‑wise операции работают только тогда, когда **условия запросов «совпадают» с ключом партиционирования**;
- если выбрать ключ, по которому почти нет фильтров, база всё равно будет ходить по большинству партиций, и выигрыш потеряется.

</details>

<details>
<summary>2. В чём главное отличие LIST‑ и RANGE‑партиционирования?</summary>

- RANGE — делит значения на **диапазоны** (`1–100`, `100–200`, месяцы по датам);
- LIST — делит по **конечному набору конкретных значений** (`'US'`, `'DE'`, `'FR'`).

</details>

<details>
<summary>3. Зачем может понадобиться composite‑подход (время + hash)?</summary>

Чтобы одновременно:

- упростить **удаление старых данных** (режем по времени),
- и **равномерно распределить нагрузку внутри интервала** (режем по hash ключа, например `user_id`).

</details>
