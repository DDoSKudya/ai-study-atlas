## 184. Уникальность и FK при партиционировании

### В этом разделе главное

- При партиционировании **уникальные индексы и первичные ключи** должны включать ключ партиционирования (или быть совместимыми с ним).
- Во многих СУБД трудно или невозможно сделать **глобальный уникальный индекс** поверх всех партиций.
- Внешние ключи (FK) на/между партиционированными таблицами становятся сложнее и иногда требуют **обходных решений** на уровне приложения.

---

### Простыми словами

В одной таблице всё просто:

- есть `PRIMARY KEY (id)` — и точка, уникальность по всему набору строк.

В партиционированной таблице:

- строки физически лежат **в разных партициях**;
- чтобы проверить уникальность, нужно **смотреть сразу во все партиции**;
- делать это быстро и дёшево технически непросто.

Поэтому:

- СУБД часто **ограничивают** то, какие уникальные ключи и FK допустимы;
- приходится:
  - включать ключ партиционирования в PK;
  - или реализовывать часть ограничений на уровне приложения.

При проектировании важно всегда задавать себе вопрос:

> «Где именно я хочу гарантировать уникальность/целостность:
>  в самой СУБД, в приложении или в комбинации?» —
>  и честно ответить, что именно является источником правды.

---

### Теория и правила

#### Уникальные индексы и первичный ключ

В PostgreSQL для декларативного партиционирования:

- `PRIMARY KEY` и `UNIQUE` должны включать **ключ партиционирования**;
- иначе нельзя гарантировать уникальность **без глобального индекса**.

Пример:

```sql
CREATE TABLE events (
  id         bigint,
  created_at timestamptz,
  payload    jsonb,
  PRIMARY KEY (id, created_at)
) PARTITION BY RANGE (created_at);
```

Здесь:

- уникальность `id` **внутри одной партиции не гарантируется**;
- гарантируется уникальность пары `(id, created_at)` по всей логической таблице.

Если тебе нужна уникальность только по `id`:

- придётся делать:
  - отдельную непартиционированную таблицу‑справочник,
  - или хранить id в другом месте с глобальным индексом,
  - или принять, что уникальность только внутри интервала (партиции).

#### Локальные и глобальные индексы

- **Локальные индексы** — живут на каждой партиции отдельно:
  - быстрее строятся и обслуживаются;
  - хорошо масштабируются.
- **Глобальные индексы** — один индекс на все партиции:
  - дают глобальную уникальность;
  - сложнее в реализации (особенно при split/merge партиций).

В PostgreSQL:

- для декларативных партиций есть только локальные индексы (на момент написания этой части);
- глобальных уникальных индексов нет → отсюда ограничение на PK/UNIQUE.

#### Внешние ключи (FK)

Сложности:

- `REFERENCES partitioned_table`:
  - проверка ссылочной целостности требует смотреть по всем партициям;
  - в некоторых версиях/СУБД это ограничено или не поддерживается.
- FK **между двумя партиционированными таблицами** с разными схемами партиционирования — ещё сложнее.

В практических системах часто:

- используют FK **только между непартиционированными справочниками** и партиционированными факт‑таблицами;
- или отказываются от строгих FK и:
  - контролируют целостность на уровне приложения,
  - периодически делают проверки через батч‑скрипты.

---

### Пошагово: как думать про PK/UNIQUE/FK в партиционированной схеме

**Шаг 1. Определи, где тебе нужна настоящая глобальная уникальность.**

Для каждой сущности (`user`, `order`, `invoice` и т.п.):

- ответь:
  - «Должен ли её идентификатор быть **уникален во всей системе**?»
  - или «Мне достаточно уникальности внутри диапазона/периода/тенанта?».

Примеры:

- `user_id`:
  - обычно хочется, чтобы был уникален глобально.
- `log_id`:
  - может быть уникален только **внутри партиции** (например, `log_id` + `created_at`), если снаружи ты всё равно работаешь с другими ключами.

**Шаг 2. Посмотри, как это согласуется с партиционированием.**

Если таблица партиционирована по `created_at`:

- глобальный `PRIMARY KEY (id)` без `created_at`:
  - сложно реализовать без глобального индекса;
  - PostgreSQL с декларативным партиционированием не даст так сделать.

А вот `PRIMARY KEY (id, created_at)`:

- уже работает:
  - внутри каждой партиции часть диапазона по `created_at`;
  - комбинация `(id, created_at)` уникальна в рамках всей логической таблицы.

**Шаг 3. Реши, где будет жить глобальный идентификатор (если он нужен).**

Если нужен настоящий **глобальный** идентификатор:

- вариант 1: отдельная непартиционированная таблица‑реестр:

  ```sql
  CREATE TABLE users_registry (
    id bigserial PRIMARY KEY,
    ... -- минимальный набор полей
  );
  ```

  - а уже партиционированные таблицы (`user_events` и т.п.) ссылаются на этот id (не обязательно FK, можно логически).

- вариант 2: принять, что идентификатор **сам по себе** не даёт смысла, и в качестве уникального ключа использовать:
  - `(id, partition_key)` или `(tenant_id, local_id)`.

**Шаг 4. С FK будь прагматичным.**

Спроси себя:

- «Насколько критично, чтобы СУБД **строго** проверяла FK **онлайн** при каждой вставке/обновлении?»
- «Могу ли я:
  - поймать большинство проблем в приложении;
  - а оставшиеся — периодическими проверками/отчётами?»

Иногда:

- строгий FK на большой партиционированной таблице:
  - слишком дорог по производительности;
  - усложняет миграции.

Тогда:

- имеет смысл:
  - оставить FK только там, где объёмы умеренные (справочники);
  - а для больших факт‑таблиц:
    - явно документировать контракт;
    - добавить рутинные проверки (batch jobs), которые ищут «висячие» ссылки и шлют алерт.

---

### Мини‑пример: пользователи и события

Представь:

- `users` — небольшая таблица (миллионы записей максимум);
- `user_events` — огромная таблица событий (сотни миллионов/миллиарды строк), партиционирована по времени.

Что можно сделать:

- `users`:
  - обычная непартиционированная таблица;
  - `PRIMARY KEY (id)` — глобально уникальный идентификатор.
- `user_events`:

  ```sql
  CREATE TABLE user_events (
    user_id    bigint NOT NULL,
    created_at timestamptz NOT NULL,
    ...,
    PRIMARY KEY (user_id, created_at, local_seq)
  ) PARTITION BY RANGE (created_at);
  ```

  - внутри `user_events`:
    - уникальность обеспечивается комбинацией `(user_id, created_at, local_seq)`;
  - связь с `users`:
    - может быть FK на `users(id)`, если объёмы/нагрузка позволяют;
    - или логический (без FK), если FK слишком дорог.

---

### Картинка в голове

В одной таблице с PK:

- у тебя один список «зарегистрированных паспортов»; легко проверить, что номер уникален.

В партиционированной:

- паспорта хранятся в **нескольких сейфах** (партициях);
- чтобы проверить уникальность номера по всей стране, нужно **заглянуть во все сейфы**;
- чтобы этого не делать, система часто требует:
  - добавить в ключ ещё «код региона/периода»,
  - и гарантировать уникальность уже пары `(номер, регион)`.

---

### Типичные ошибки

- Пытаться сделать `PRIMARY KEY (id)` на партиционированной таблице, не включая ключ партиционирования.
- Переносить схему «как есть» из непартиционированного мира, не учитывая требования к PK/UNIQUE.
- Слепо полагаться на FK между большими партиционированными таблицами, не оценивая их влияние на производительность и сложность.

---

### Проверь себя

<details>
<summary>1. Зачем в партиционированной таблице часто включают ключ партиционирования в PK?</summary>

Чтобы:

- обеспечить уникальность **комбинации (ключ, значение партиционирования)**;
- при этом не требовать глобального индекса по одному полю, который технически сложен и дорог.

</details>

<details>
<summary>2. Почему глобальные уникальные индексы сложны для реализции поверх партиций?</summary>

Потому что:

- при split/merge/добавлении партиций нужно постоянно:
  - поддерживать согласованность индекса,
  - перемещать и обновлять его части;
- проверка уникальности при вставке требует **потенциально обращаться ко всем партициям** или к сложному распределённому индексу.

</details>

<details>
<summary>3. Какие есть альтернативы строгим FK в сильно партиционированных схемах?</summary>

Часто:

- FK оставляют только для **небольших справочников**;
- для больших факт‑таблиц:
  - целостность контролируют на уровне приложения и ETL‑процессов,
  - периодически запускают проверки и отчёты о «висячих» ссылках.

</details>
