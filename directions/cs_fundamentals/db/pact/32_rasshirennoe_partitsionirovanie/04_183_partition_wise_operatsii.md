## 183. Partition‑wise операции

### В этом разделе главное

- Partition‑wise join/aggregate позволяют **разбивать тяжёлые операции по партициям** и выполнять их параллельно.
- Для этого таблицы должны быть **совместимо партиционированы**: по одному и тому же ключу и схеме.
- Правильно спроектированное партиционирование под join‑ы и агрегации может давать **очень ощутимый выигрыш** на аналитических нагрузках.

---

### Простыми словами

Без партиционирования:

- большой `JOIN` двух таблиц по миллионам строк — это одна огромная операция, которая:
  - долго занимает CPU и память,
  - плохо масштабируется.

С партиционированием:

- если обе таблицы порезаны **одинаково**,
- база может сделать **много маленьких join‑ов по партициям**:
  - `orders_2025_01 JOIN customers_2025_01`,
  - `orders_2025_02 JOIN customers_2025_02` и т.д.

Это:

- лучше укладывается в память;
- хорошо параллелится по ядрам и нодам.

---

### Теория и правила

#### Partition‑wise join

Условия:

- обе таблицы:
  - партиционированы **по одному и тому же ключу**;
  - имеют **согласованные границы партиций**;
- `JOIN` идёт по этому ключу (либо по ключу, функционально эквивалентному).

Тогда СУБД может:

- вместо одного большого join‑а:
  - строить план из **нескольких join‑ов по партициям**;
- часто с параллелизмом на уровне процессов/воркеров.

#### Partition‑wise aggregate

Агрегации (`GROUP BY`, `SUM`, `COUNT`, и т.п.) могут делаться:

- сначала **по каждой партиции отдельно**;
- затем результаты агрегируются ещё раз (финальный слой).

Это:

- уменьшает объём данных, которые приходится «таскать» между узлами/процессами;
- лучше использует кэш и память.

#### Ограничения

- Таблицы должны быть:
  - разбиты по **одинаковой схеме** (например, по месяцу по одному и тому же полю);
  - иметь совместимые типы и границы партиций.
- Не все комбинации join‑ов и агрегатов могут быть преобразованы в partition‑wise операции — всё зависит от СУБД и сложности запроса.

---

### Пошагово: как подготовить таблицы к partition‑wise join/aggregate

**Шаг 1. Определи критичные JOIN/AGG‑запросы.**

Сначала нужно понять, **за что вообще стоит бороться**:

- какие запросы:
  - самые тяжёлые по времени;
  - или чаще всего попадают в алерты;
  - или выполняются регулярно (ежедневные отчёты).

Для каждого такого запроса выпиши:

- какие таблицы в нём участвуют;
- по каким полям они join‑ятся;
- есть ли уже партиционирование у этих таблиц.

**Шаг 2. Проверь совместимость партиционирования.**

Для пары таблиц `A` и `B`:

- узнай:
  - есть ли у них партиционирование;
  - если да — по каким полям и какому типу (RANGE/LIST/HASH);
  - какие диапазоны/значения покрывают партиции.

Сравни:

- партиционированы ли они:
  - по **одному и тому же полю** (или эквивалентному);
  - с **одинаковыми границами** (например, оба по месяцам с одинаковыми интервалами).

Если нет:

- partition‑wise join вряд ли заработает — таблицы нужно «привести» к одной схеме.

**Шаг 3. Спроектируй совместное партиционирование.**

Если ты только начинаешь:

- выбери общий ключ и стратегию:
  - например, для `orders` и `order_items`:
    - партиционирование по `order_date` месяцами;
    - при этом обе таблицы имеют **одинаковый набор партиций**.

Если таблицы уже существуют:

- оцени, насколько реально:
  - изменить партиционирование одной из них;
  - или создать новую партиционированную «витрину» (materialized view / отдельную таблицу) специально под отчёты.

**Шаг 4. Проверь планы с `EXPLAIN`.**

Когда партиционирование согласовано:

- запусти `EXPLAIN (ANALYZE, BUFFERS)` для большого join‑запроса;
- смотри:
  - появились ли в плане шаги вида «join по партициям» (зависит от формата плана СУБД);
  - как распределена работа по воркерам/процессам;
  - насколько уменьшились:
    - время выполнения;
    - потребление памяти;
    - объём прочитанных страниц.

Если partition‑wise join не появился:

- проверь:
  - совпадают ли действительно границы партиций;
  - не ломают ли оптимизацию какие‑то функции/выражения в `JOIN`/`WHERE`.

**Шаг 5. Определи, какие агрегаты можно делать по партициям.**

Смотри на `GROUP BY`:

- часто встречаются агрегаты:
  - `GROUP BY date`, `GROUP BY month`, `GROUP BY customer_id` и т.д.

Для каждой таблицы:

- подумай, можно ли:
  - сначала агрегировать **внутри партиции** (например, по дням внутри месяца);
  - потом сделать ещё один слой агрегации по результатам.

Это можно реализовать:

- как:
  - двухшаговый SQL (CTE/подзапрос);
  - или материализованную витрину, которая хранит агрегаты по партициям.

---

### Мини‑пример: отчёты по заказам

Представь:

- таблица `orders` (заказы);
- таблица `order_items` (позиции заказа);
- тебе нужно:
  - каждый день считать выручку по дням за последний год;
  - делать выборки по месяцам/кварталам.

**Вариант без партиционирования:**

- join по миллионам строк;
- большая работа за раз;
- план тяжёлый, плохо параллелится.

**Вариант с согласованным партиционированием:**

- обе таблицы:

  ```sql
  PARTITION BY RANGE (order_date)
  ```

  по месяцам с одинаковыми границами;

- аналитический запрос:
  - для каждого месяца фактически становится **набором join‑ов по партициям**;
  - агрегации можно делать:
    - внутри партиций (дни внутри месяца);
    - потом собирать результаты.

Результат:

- нагрузка размазывается;
- проще контролировать время и ресурсы;
- легче делать инкрементальные перерасчёты (пересчитать только последние партиции).

---

### Картинка в голове

Представь два огромных мешка с данными, которые нужно «склеить» и посчитать по ним отчёт.

- Без партиционирования — ты вываливаешь всё на один стол и пытаешься сразу пересчитать.
- С partition‑wise подходом — у тебя **много маленьких столиков** (по месяцам), и на каждом столике ты считаешь только локальный кусок, а потом складываешь результаты.

---

### Типичные ошибки

- Партиционировать связанные таблицы **по разным ключам**, а потом удивляться, что join‑ы тяжелы.
- Ожидать partition‑wise join, когда границы партиций **не совпадают**.
- Не включать ключ партиционирования в join‑условие.

---

### Проверь себя

<details>
<summary>1. Почему для partition‑wise join важно, чтобы обе таблицы были партиционированы по одному и тому же ключу?</summary>

Потому что:

- тогда строки, которые могут встретиться в join‑е, попадают в **одни и те же партиции**;
- СУБД может делать join **независимо по каждой партиции**, не выполняя дорогостоящий «перемешивающий» шаг.

</details>

<details>
<summary>2. Чем partition‑wise aggregate отличается от обычной агрегации?</summary>

Обычная агрегация:

- считает `GROUP BY` по всей таблице сразу.

Partition‑wise aggregate:

- сначала агрегирует **внутри каждой партиции**,
- потом агрегирует уже сжатые результаты, что:
  - уменьшает объём данных,
  - лучше параллелится.

</details>

<details>
<summary>3. Можно ли ожидать partition‑wise join, если таблицы партиционированы по разным полям?</summary>

Практически — нет:

- ключи партиционирования не совпадают,
- строки, которые должны встретиться в join‑е, разложены по разным партициям,
- СУБД не может заранее «сопоставить» нужные пары партиций.

</details>
