## 249. Алгоритмы JOIN: nested loop, hash, merge

### Цель раздела

Понять, **как СУБД реально выполняет `JOIN` под капотом**: чем отличаются nested loop, hash join и merge join, как планировщик выбирает между ними, какие предпосылки у каждого алгоритма и какие грабли появляются, когда ожидания не совпадают с реальностью.

### В этом разделе главное

1. `JOIN` — это не магия, а **конкретный алгоритм сопоставления строк** двух (или более) таблиц.
2. **Nested loop** эффективен при небольших выборках и/или хорошем индексе, но плохо масштабируется на “широкие” JOIN’ы без индексов.
3. **Hash join** хорош для больших объёмов и равенств (`=`), но требует памяти под хеш‑таблицу (или начинает писать временные файлы).
4. **Merge join** выигрывает, когда обе стороны уже отсортированы по ключу (например, идут по индексам или после сортировки).
5. В реальных планах **часто комбинируются** несколько уровней JOIN’ов разного типа, и важно понимать, почему планировщик принял такое решение.

### Термины

- **Nested loop join** — для каждой строки из внешней (outer) таблицы ищем подходящие строки во внутренней (inner).
- **Hash join** — строим хеш‑таблицу по одной стороне (build), затем “зондируем” её строками из второй (probe).
- **Merge join** — сливаем два отсортированных потока по ключу, двигая указатели, как в merge‑фазе сортировки слиянием.
- **Outer/inner side** — внешняя и внутренняя сторона JOIN’а (важно для nested loop).
- **Spill** — “перелив” структуры данных на диск, когда не хватает памяти (hash table, сортировка и т.п.).

### Теория и правила

#### 1) Nested loop: “в лоб, но с индексом”

Давай разберём этот алгоритм **совсем медленно**, по шагам.

##### 1.1. Интуиция простыми словами

Представь, что у тебя есть:

- маленький список `A` (например, 100 активных пользователей),
- большой список `B` (например, все заказы за год).

Тебе нужно для каждого пользователя найти все его заказы.

Самый прямой способ:

1. Берёшь первого пользователя из `A`.
2. Берёшь список `B` и проходишь его весь, отмечая все заказы с нужным `user_id`.
3. Переходишь ко второму пользователю и снова проходишь **весь** `B`.
4. И так далее.

Это и есть **наивный nested loop**: внешний цикл по пользователям, внутренний — по заказам.

Если же у тебя есть “волшебный указатель” (индекс) по `user_id` в заказах, то вместо полного прохода по `B` ты делаешь:

1. “Скажи мне все заказы, где `user_id = 42`”.
2. Получаешь их **сразу** (индекс идёт прямо к нужным записям).

То есть **внешний цикл остаётся**, а вот внутренний превращается из “пройти весь список” в “быстро найти по индексу”.

##### 1.2. Формальное описание

Идея:

- берём строку из внешней таблицы,
- ищем подходящие строки во внутренней (по условию JOIN),
- повторяем для всех строк.

Если внутреннюю таблицу **каждый раз** сканировать полностью, сложность порядка \(O(N \cdot M)\), что очень плохо для больших N и M.  
Но если по условию JOIN есть **селективный индекс** (например, `users.id`), то внутренняя часть превращается в **быстрый поиск по индексу**.

**Когда nested loop хорош:**

- одна сторона маленькая (десятки/сотни строк);
- по ключу JOIN есть индекс на другой стороне;
- условие даёт сильную селективность.

**Когда nested loop плох:**

- обе стороны большие, индексов нет;
- условие JOIN не селективное;
- планировщик ошибся в оценках кардинальности и выбрал nested loop там, где нужен hash join.

##### 1.3. Разбор конкретного примера (пошагово)

Пусть:

- таблица `users` — 1 000 000 строк;
- таблица `orders` — 50 000 000 строк;
- на `orders(user_id)` есть B‑tree индекс;
- запрос фильтрует пользователей:

```sql
SELECT o.*
FROM users u
JOIN orders o ON o.user_id = u.id
WHERE u.is_active = true
  AND u.country = 'DE';
```

Предположим, что фильтр по `WHERE` оставляет **только 100 пользователей**. Что делает nested loop:

1. Выполняет фильтрацию по `users` (может использовать индекс или seq scan + фильтр).
2. Получает набор из 100 `id`.
3. Для каждого `id`:
   - идёт в индекс `orders(user_id)`,
   - быстро находит все строки с этим `user_id`,
   - отдаёт их на выход.

Внутренний цикл **не сканирует всю таблицу `orders`**, а использует индекс как “телепорт” к нужным строкам. Именно поэтому nested loop в таком сценарии работает хорошо.

Если же убрать фильтр и оставить полноценный JOIN “всех на всех”, внешний цикл пройдётся по 1 000 000 пользователей, а индексные поиски по 50 000 000 заказов могут стать очень тяжёлыми. Тогда hash join, который один раз построит хеш‑таблицу по `orders` и потом один раз пройдётся по `users`, будет выгоднее.

##### 1.4. Блочный nested loop (block nested loop)

В учебниках и планах часто отдельно упоминают **блочный nested loop**. Это та же идея, что и у обычного nested loop, но с одной важной оптимизацией:  
мы **берём не одну строку из внешней таблицы, а целый блок строк**, и затем работаем с ними пачкой.

Интуиция:

- вместо того чтобы ходить к внутренней таблице “по одному пользователю”, мы берём, скажем, **1000 пользователей сразу**;
- загружаем их идентификаторы/ключи в память;
- затем за один проход по внутренней таблице (или по индексу/страницам) находим все заказы, которые относятся **к любому** из этих 1000 пользователей.

Зачем это нужно:

- чтобы **лучше использовать кэш и дисковые страницы**:
  - если мы уже прочитали страницу с заказами, хотелось бы “выжать” из неё максимум, а не возвращаться к ней много раз;
- чтобы уменьшить количество отдельных обращений к диску/индексу.

Блочный nested loop по сути остаётся nested loop’ом (логика “внешняя сторона → внутренняя сторона” сохраняется),  
но он **снижает накладные расходы на I/O**, особенно когда:

- внешняя выборка средняя по размеру (тысячи/десятки тысяч строк),
- внутренняя таблица очень большая,
- есть выгода от группировки обращений к внутренней стороне.

Формально, это объясняется тем, что мы увеличиваем “размер единицы работы” и уменьшаем количество переключений между внешней и внутренней сторонами.  
В планах СУБД детали реализации могут быть скрыты, но понимание этой идеи помогает не пугаться слова “block” в описаниях nested loop join.

#### 2) Hash join: “строим хеш‑таблицу”

Идея:

1. Выбираем **build‑сторону** — обычно ту, что меньше по оценке кардинальности.
2. Читаем строки build‑стороны и кладём их в **хеш‑таблицу** по ключу JOIN.
3. Читаем строки probe‑стороны и для каждой по хешу ищем совпадения в build‑таблице.

Сложность **приблизительно линейная** по сумме размеров сторон (\(O(N + M)\)), если хеш хорошо распределён и нет сильных коллизий.

Особенности:

- отлично работает для `JOIN ... ON a.key = b.key`;
- плохо подходит для неравенств (`>`, `<`, `BETWEEN`) и сложных условий — там чаще пригоден nested loop или merge;
- чувствителен к памяти: если хеш‑таблица не помещается, начинается spilling (partitioned hash join, временные файлы).

#### 3) Merge join: “слияние отсортированных потоков”

Идея почти как в merge‑сортировке:

1. Обе стороны отсортированы по ключу JOIN (либо уже по индексу, либо предварительно отсортированы).
2. Держим по указателю на текущую строку в каждом потоке.
3. Сравниваем значения ключей:
   - если `left_key < right_key` — двигаем левый указатель;
   - если `left_key > right_key` — двигаем правый;
   - если равны — получаем совпадения (при необходимости — несколько).

Сложность примерно \(O(N + M)\) **при условии, что сортировка уже сделана**. Но:

- если нужно предварительно сортировать **большие объёмы**, сортировка может быть дороже, чем hash join;
- при равенстве может быть много совпадений, и merge join должен аккуратно их обрабатывать.

#### 4) Adaptive join (адаптивный JOIN в SQL Server)

В некоторых СУБД (в первую очередь в **SQL Server**) есть ещё одна интересная идея — **adaptive join** (адаптивный JOIN).

Интуиция:

- планировщик иногда **не уверен**, насколько большой получится входной поток;
- для небольшого числа строк выгоден nested loop,  
  а для больших — hash join;
- adaptive join позволяет **начать запрос как nested loop**, а если по ходу выясняется, что строк гораздо больше, чем ожидалось, **переключиться на hash join**.

Происходит это так (упрощённо):

1. Выполняется часть плана, которая готовит данные для JOIN (например, скан/фильтр).
2. СУБД считает, сколько строк уже пришло “на вход” оператора.
3. Если строк мало — продолжает работать в режиме nested loop (дешевле построить индексные обращения).
4. Если строк слишком много — запускает фазу построения хеш‑таблицы и переходит к hash join.

Важно:

- это **конкретная особенность реализации** (SQL Server, некоторые MPP‑движки);
- в PostgreSQL/MySQL/Oracle аналогичная идея обычно выражается не как один оператор, а как **перестроение плана** (re‑planning) или выбор другого плана при следующем выполнении;
- от тебя, как от инженера, здесь важно главное:  
  СУБД пытается **адаптироваться к реальному размеру данных**, а не только к статистике.

Adaptive join — хороший пример того, что на практике план — это не всегда “жёстко выбранный однажды алгоритм”, а иногда **семейство алгоритмов, между которыми движок может переключаться в рантайме**.

### Пошагово: как движок выбирает алгоритм (упрощённая модель)

1. **Собирает статистику**: оценка числа строк, распределения значений, селективности условий.
2. Строит возможные планы:
   - nested loop с индексом или без;
   - hash join;
   - merge join (если есть подходящие индексы/сортировки).
3. Оценивает **стоимость** каждого плана:
   - I/O (сколько страниц будет прочитано/записано);
   - CPU (сколько операций сравнения/хеширования);
   - память (размер хеш‑таблиц, буферов сортировки).
4. Выбирает план с **минимальной оценочной стоимостью**.

Если статистика неправильная (устарела или не отражает реального распределения), планировщик может значительно ошибиться и, например, выбрать nested loop вместо hash join.

### Простыми словами

Представь, что у тебя есть **два списка людей** и ты хочешь их “склеить” по ID:

- **Nested loop** — для каждого человека из левого списка ты пролистываешь правый список, пока не найдёшь совпадения. Если правый список отсортирован и у тебя есть быстрый “указатель” (индекс), то поиск быстрый. Если нет — получается квадратичный ад.
- **Hash join** — ты сначала строишь **словарь** по одному списку (ID → человек), а потом по второму списку быстро заглядываешь в словарь. Важно лишь, чтобы словарь поместился в памяти или хотя бы в разумные структуры.
- **Merge join** — если оба списка **уже отсортированы по ID**, ты просто идёшь по ним одновременно, как по двум отсортированным книжным полкам, и “сшиваешь” совпадающие книги.

### Картинка в голове

- Nested loop — **двойной цикл**: внешний и внутренний.
- Hash join — **два этапа**: “построить хеш‑таблицу” и “пройтись по ней”.
- Merge join — **две отсортированные линейки**, по которым ты двигаешься двумя пальцами.

Если эта картинка держится в голове, читать `EXPLAIN` становится гораздо понятнее: ты видишь не просто слова, а конкретные шаги.

### Как запомнить

Короткая формула:

> **Nested loop** = “перебираем и ищем” (хорошо при малых/индексированных выборках).  
> **Hash join** = “строим словарь и зондируем” (хорошо для больших равенств).  
> **Merge join** = “сливаем отсортированные потоки” (хорошо, когда сортировка уже есть или почти бесплатна).

### Примеры

#### PostgreSQL: nested loop vs hash join

```sql
-- Маленькая таблица фильтрованных пользователей и большая таблица заказов
EXPLAIN ANALYZE
SELECT o.*
FROM users u
JOIN orders o ON o.user_id = u.id
WHERE u.country = 'DE' AND u.is_active = true;
```

- Если `WHERE` сильно сужает `users` (например, до сотен строк), а на `orders.user_id` есть индекс, PostgreSQL почти наверняка выберет **nested loop** с индексным доступом.
- Если бы `users` не фильтровалась и обе таблицы содержали миллионы строк, СУБД, скорее всего, выбрала бы **hash join**.

#### MySQL / SQL Server

- В MySQL/SQL Server названия операторов могут отличаться, но логика та же: **nested loop** как базовый join, **hash join** (в более новых версиях/режимах), **merge join** (SQL Server “merge join”).
- В планах можно увидеть, какой именно физический алгоритм выбран, и сопоставить его с нашей ментальной моделью.

### Практика / реальные сценарии

- **“Всё было быстро, потом внезапно стало медленно”** — после роста данных и без обновления статистики планировщик стал оценивать одну из таблиц как “маленькую” и перешёл с hash join на nested loop без индекса.
- **Добавили индекс, но стало медленнее** — планировщик начал использовать nested loop через новый индекс, хотя более дешёвым был бы hash join на полный скан (особенно, если выборка не селективна).

### Типичные ошибки

- Ожидание, что `JOIN` “сам по себе” масштабируется на миллиарды строк без индексов.
- Слепая вера, что “hash join всегда быстрее nested loop”.
- Игнорирование того, что hash join может начать писать крупные временные файлы при нехватке памяти.
- Чтение планов без понимания, какая сторона JOIN’а build, а какая probe (важно для оценки нагрузки на память).

### Что будет, если…

- …по ключам JOIN нет индексов, а таблицы большие?
  - Планировщик может выбрать nested loop с полным сканом внутренней таблицы (катастрофа по времени) или hash join с огромными сканами и spill’ами.
- …сильно занизить `work_mem`/аналог и запустить большой hash join?
  - Хеш‑таблица не влезет в память, план перейдёт к **partitioned hash join** с активным выводом временных файлов на диск → резкий рост времени и I/O.

### Проверь себя

1. Приведи пример запроса и структуры данных, где **nested loop** будет значительно быстрее **hash join**.

<details><summary>Ответ</summary>
Например, когда у тебя есть небольшая выборка (100 активных пользователей из миллиона) и большой индексированный лог заказов:

```sql
SELECT o.*
FROM users u
JOIN orders o ON o.user_id = u.id
WHERE u.is_active = true AND u.country = 'DE';
```

При наличии индекса `orders(user_id)` чтение по nested loop с индексом для 100 пользователей будет очень быстрым, а hash join потребует сначала прочитать и захешировать значительную часть `orders`, что дороже.
</details>

2. Почему merge join часто требует предварительной сортировки, и когда это не так?

<details><summary>Ответ</summary>
Merge join работает по двум **отсортированным** потокам. Если данные ещё не отсортированы по ключу JOIN, их нужно отсортировать (что может быть дорого). Но если обе стороны уже идут из индексов, упорядоченных по нужному ключу (например, B‑tree по полю JOIN), дополнительная сортировка не нужна, и merge join может оказаться очень выгодным.
</details>

### Запомните

- JOIN — это конкретный алгоритм (или набор алгоритмов), а не абстрактный “оператор склейки”.
- Понимание nested loop / hash / merge join помогает **читать планы запросов глазами движка**, а не только глазами SQL‑разработчика.

#### Где можно запутаться (и как не запутаться)

Чтобы ещё раз разложить всё по полочкам, давай посмотрим на три алгоритма через один и тот же вопрос:  
**“Сколько раз мы смотрим на строки правой таблицы?”**

- При **nested loop** без индекса мы фактически делаем полный просмотр правой таблицы **для каждой строки слева**. Это и даёт худший случай \(O(N \cdot M)\). Если есть индекс, то вместо полного просмотра мы делаем **быстрый точечный поиск**, и становится намного лучше.
- При **hash join** мы сначала **один раз** проходим по build‑стороне, кладём её в хеш, а потом **один раз** проходим по probe‑стороне. То есть мы смотрим на каждую строку примерно по одному разу, и это даёт почти линейное время.
- При **merge join** мы так же **по одному разу** проходим по каждой из отсортированных сторон: двигаем два указателя и никогда не возвращаемся назад. Но за это мы платим либо дополнительной сортировкой, либо поддержанием индексов.

Если держать в голове эту картинку, выбор плана становится понятнее:  
**nested loop** — много проходов по правой таблице, если нет индекса;  
**hash join** — два прохода, но нужна память;  
**merge join** — два прохода по отсортированным данным, но нужна сортировка/индекс.

#### Мини‑шпаргалка по JOIN’ам (в одно чтение)

- Маленькая выборка слева + хороший индекс справа → почти всегда **nested loop**.
- Большие таблицы, равенства по ключу, нет удобного порядка → чаще всего **hash join**.
- Обе стороны уже отсортированы по ключу (или легко отсортировать) → кандидат на **merge join**.

Если сомневаешься, **смотри на план**: какая сторона build/probe, есть ли сортировки перед merge, есть ли индексы для nested loop. Попробуй объяснить себе план словами — это лучший способ закрепить понимание.
