## 253. Альтернативные индексы: fractal tree, Bε‑tree, learned, bitmap

### Цель раздела

Познакомиться с **альтернативными индексными структурами**, которые идут дальше классического B‑дерева и LSM: fractal tree/Bε‑tree, learned indexes, bitmap‑индексы; понять, какие задачи они решают, зачем они нужны в реальных продуктах (TokuDB, аналитические СУБД, исследовательские системы), какие плюсы и минусы несут.

### В этом разделе главное

1. B‑деревья — не единственный способ индексировать данные; в некоторых сценариях **альтернативные структуры** дают существенный выигрыш.
2. **Fractal tree / Bε‑tree** уменьшают количество случайных записей, используя буферы обновлений внутри дерева.
3. **Learned indexes** рассматривают индексацию как **задачу приближения функции** “ключ → позиция” с помощью моделей (machine learning).
4. **Bitmap‑индексы** отлично работают в аналитических/колоночных системах для колонок с небольшим/умеренным числом различных значений.
5. Эти структуры почти всегда — **компромисс**, завязанный на конкретные сценарии (OLAP, time‑series, спец‑нагрузки).

### Термины

- **Fractal tree / Bε‑tree** — дерево, где на внутренних узлах есть буферы, накапливающие обновления и периодически “спускающие” их вниз пакетами.
- **Learned index** — структура, в которой позиция ключа оценивается моделью (например, небольшой нейросетью) вместо классического дерева поиска.
- **Bitmap‑индекс** — для каждого возможного значения хранится битовый вектор, указывающий строки, где это значение встречается.

### Теория и правила

#### 1) Fractal tree / Bε‑tree

Идея:

- в классическом B‑дереве каждое обновление/вставка может трогать путь от корня к листу;
- в Bε‑дереве на каждом узле есть **буфер обновлений**, куда складываются операции;
- обновления “спускаются” вниз **пакетами**, уменьшая количество I/O.

Плюсы:

- существенно меньше случайных записей;
- хорош для write‑heavy workloads.

Минусы:

- сложнее реализация;
- может увеличивать latency отдельных чтений/записей;
- мало где доступен “из коробки” (TokuDB/TokuMX — примеры, но они нишевые).

#### 2) Learned indexes

Идея:

- рассматривать отсортированный массив ключей как **функцию** `f(key) ≈ позиция_ключа`;
- вместо B‑дерева обучить **модель**, которая по ключу сразу даёт приблизительную позицию;
- после этого локальные структуры (малые деревья/поиски) доуточняют позицию.

Плюсы:

- потенциально меньше памяти;
- теоретически меньше уровней поиска.

Минусы:

- сложность обучения и переобучения;
- чувствительность к распределению данных;
- пока в основном **исследовательская тема**, но идеи просачиваются в промышленные продукты (оптимизация индексов/кардинальностей).

#### 3) Bitmap‑индексы

Идея:

- допустим, у нас есть колонка `status` с возможными значениями `{NEW, PAID, CANCELLED}`.
- Для каждого значения строится **битовый вектор** длиной `N` (число строк):
  - `status=NEW` → 1 в позициях, где строка имеет статус NEW, иначе 0;
  - аналогично для остальных.
- Запрос `WHERE status IN ('NEW','PAID')` превращается в **побитовое OR** над соответствующими битмапами.

Плюсы:

- очень быстрая фильтрация по нескольким значениям (битовые операции);
- отличная компрессия (run‑length encoding и др.) для колонок с длинными “полосами” одинаковых значений;
- часто используются в DWH, OLAP, колоночных СУБД (Oracle, Vertica, ClickHouse, SQL Server columnstore).

Минусы:

- для высококардинальных колонок (почти уникальные значения) битмапы становятся громоздкими;
- обновления могут быть дорогими (нужно модифицировать битмапы).

### Пошагово: как bitmap‑индекс помогает запросам

Представь таблицу продаж:

- `status` (NEW, PAID, CANCELLED),
- `channel` (WEB, MOBILE, STORE),
- `country` (десятки значений).

Запрос:

```sql
SELECT COUNT(*)
FROM sales
WHERE status = 'PAID'
  AND channel IN ('WEB', 'MOBILE')
  AND country = 'DE';
```

Bitmap‑индексы по этим колонкам позволяют:

1. взять битмап `status=PAID`;
2. взять битмапы `channel=WEB` и `channel=MOBILE` и сделать OR;
3. взять битмап `country=DE`;
4. сделать AND всех полученных битмапов;
5. посчитать единицы в результате (bit count).

Это делается гораздо быстрее, чем многократное сканирование B‑дерева/таблицы.

### Простыми словами

- **Fractal/Bε‑деревья** — это как B‑деревья, но с **почтовыми ящиками на каждом этаже**, куда сначала ссыпают письма (обновления), и только потом разносят их по квартирам пачками.
- **Learned indexes** — как **навигатор**, который по адресу примерно показывает, где дом на карте, а дальше ты немного уточняешь координаты.
- **Bitmap‑индексы** — как **цветные наклейки** на папках: у каждого статуса/канала свой цвет, и ты быстро находишь нужные комбинации цветовых меток.

### Картинка в голове

- Fractal/Bε‑дерево: дерево с буферами на каждом узле, куда стекаются операции.
- Learned index: кривая на графике `key → position`, которую мы приблизили моделью.
- Bitmap‑индексы: несколько длинных полосок с нулями и единицами, которые мы “И” и “ИЛИ” между собой.

### Как запомнить

> **Fractal/Bε** = “кэш обновлений внутри дерева”.  
> **Learned** = “индекс как модель распределения ключей”.  
> **Bitmap** = “битовые маски для значений”.

### Примеры

- **TokuDB/TokuMX** — реализации Bε‑деревьев (сейчас мало актуальны, но хорошо иллюстрируют идею).
- **Oracle bitmap indexes** — классический пример bitmap‑индексов в промышленной СУБД.
- **Columnstore (SQL Server, Vertica, ClickHouse)** — активно используют битмапы и сжатие.

### Практика / реальные сценарии

- В DWH проектировщик часто выбирает:
  - B‑tree индексы для high‑cardinality ключей (ID, timestamp),
  - bitmap‑индексы для low‑cardinality атрибутов (статусы, флаги).
- В OLTP‑системах bitmap‑индексы почти не используются (дорогие обновления), там доминируют B‑trees и, иногда, LSM‑подобные структуры.

### Типичные ошибки

- Пытаться использовать bitmap‑индексы для колонок с миллионами уникальных значений.
- Считать learned indexes “магией ML”, которую можно включить одним флажком в проде.
- Ожидать, что фрактальные деревья волшебно исчезнут издержки I/O (на самом деле они их перераспределяют).

### Что будет, если…

- …добавить bitmap‑индекс на колонку `user_id` с сотнями миллионов уникальных значений?
  - Индекс будет огромным и малоэффективным — сжатие не спасёт, так как паттерны битов будут сложными, а выгоды от битовых операций почти нет.
- …переоценить точность learned index и не учитывать выбросы/смещение распределения?
  - Модель будет часто ошибаться, локальные структуры поиска станут крупными, и теоретические преимущества пропадут.

### Проверь себя

1. Почему bitmap‑индексы особенно эффективны в колоночных/аналитических СУБД?

<details><summary>Ответ</summary>
Потому что аналитические запросы часто фильтруют по нескольким low‑cardinality атрибутам (статусы, флаги, категории), а колоночный формат позволяет эффективно хранить и сжимать битмапы. Битовые операции (`AND`/`OR`) над сжатыми битмапами выполняются очень быстро и масштабируются на большие объёмы данных.
</details>

2. В чём ключевая идея learned index по сравнению с классическим B‑деревом?

<details><summary>Ответ</summary>
В том, чтобы рассматривать индекс как **функцию распределения ключей** и приблизить эту функцию моделью (линейной/ML), которая по ключу даёт приблизительную позицию. Вместо фиксированной структуры дерева мы используем знание о распределении данных, чтобы уменьшить глубину поиска и объём вспомогательных структур.
</details>

### Запомните

- Альтернативные индексы — это не “модные игрушки”, а конкретные ответы на специфические нагрузки (write‑heavy, аналитика, особые распределения данных).
- В проде по‑прежнему доминируют B‑деревья и LSM, но знания о bitmap и learned index помогают лучше понимать современные СУБД и их эволюцию.

#### Где можно запутаться

Легко увлечься идеей: “раз есть какая‑то новая структура (fractal tree, learned index), значит, она **всегда** лучше классических индексов”. На практике:

- каждая структура “заточена” под **свой профиль запросов и данных**;
- сложность реализации и эксплуатации может перевесить теоретическую выгоду;
- не всегда есть промышленная, хорошо поддерживаемая реализация.

Для bitmap‑индексов типичная ошибка — использовать их там, где **слишком много разных значений**, и тогда вместо компактных битовых масок мы получаем огромные структуры без выигрыша в скорости.

#### Мини‑шпаргалка по альтернативным индексам

- Если у тебя классический OLTP с привычными запросами — в 90% случаев хватит **B‑деревьев** и иногда **LSM**.
- Если у тебя DWH/аналитика с фильтрацией по статусам и категориям — подумай о **bitmap‑индексах** или колоночном хранении.
- Если у тебя крайне специфическая нагрузка и очень большие объёмы — имеет смысл **изучать исследовательские структуры** (fractal, learned), но с учётом зрелости их реализаций.
