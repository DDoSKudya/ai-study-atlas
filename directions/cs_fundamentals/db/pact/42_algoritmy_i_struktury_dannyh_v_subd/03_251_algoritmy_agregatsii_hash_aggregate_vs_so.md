## 251. Алгоритмы агрегации: hash aggregate vs sort aggregate

### Цель раздела

Понять, **как реализуется `GROUP BY` и агрегатные функции внутри СУБД**, чем отличаются hash aggregate и sort aggregate, когда каждый из подходов эффективен, как они ведут себя при нехватке памяти и какие ловушки возникают в реальных запросах.

### В этом разделе главное

1. Агрегация — это **группировка строк по ключу и вычисление агрегатов** (COUNT, SUM, AVG и др.) по каждой группе.
2. **Hash aggregate** держит хеш‑таблицу “ключ → состояние агрегатов” и хорошо работает, когда количество групп умеренное и помещается в память.
3. **Sort aggregate** сначала сортирует данные по ключу, а потом проходит по отсортированным строкам, объединяя соседние одинаковые ключи.
4. Выбор между hash и sort aggregate зависит от **кардинальности групп, объёма данных, памяти и доступных сортировок**.
5. При нехватке памяти hash aggregate может разбиваться на несколько проходов с временными файлами, а sort aggregate — скатываться в external sort.

### Термины

- **Hash aggregate** — агрегирование с использованием хеш‑таблицы: ключ → накопленные значения.
- **Sort aggregate** — агрегирование по отсортированным данным: все строки одной группы идут подряд.
- **Кардинальность групп** — количество уникальных значений ключа группировки.
- **Partial / final aggregate** — частичные и финальные агрегаты (важно для параллельной/распределённой агрегации).

### Теория и правила

#### 1) Hash aggregate

Снова пойдём от интуиции к формализму.

##### 1.1. Интуитивно: коробки по городам

Представь, что ты считаешь количество заказов по странам:

- у тебя есть большой список заказов, каждый с полем `country`;
- ты ставишь перед собой **коробки с надписями**: `DE`, `FR`, `US`, `...`;
- берёшь заказ, читаешь `country` и:
  - кладёшь его в коробку `DE`, если это Германия;
  - в коробку `FR`, если Франция;
  - и т.д.

В конце просто считаешь, сколько бумажек в каждой коробке.

Это и есть **hash aggregate**:

- ключ группировки (`country`) → “коробка”;
- в коробке мы храним **состояние агрегатов** (счётчик, сумму, минимум, максимум).

##### 1.2. Формально

Алгоритм:

1. Инициализируем пустую хеш‑таблицу.
2. Для каждой входной строки:
   - вычисляем ключ группировки (например, `(country, status)`),
   - ищем/создаём запись в хеше,
   - обновляем счётчики/суммы/минимумы/максимумы.
3. В конце проходим по хеш‑таблице и возвращаем результат.

Плюсы:

- одна проходка по данным;
- не нужно сортировать;
- хорошо при относительно **небольшом количестве групп** (размер хеша умеренный).

Минусы:

- требует памяти, зависящей от количества групп;
- при очень высокой кардинальности групп может начать активно spill’ить;
- порядок вывода групп не определён (если нужен определённый порядок — всё равно нужна сортировка).

#### 2) Sort aggregate

Алгоритм:

1. Сначала сортируем данные по ключу группировки;
2. Затем идём по отсортированному массиву, аккумулируя значения для текущей группы, пока ключ не изменится;
3. Как только ключ меняется — “закрываем” группу и начинаем следующую.

Плюсы:

- память используется **для одной группы/небольшого окна**, а не для всех групп сразу;
- хорошо работает при **очень высокой кардинальности групп**;
- если данные уже отсортированы (по индексу), дополнительная сортировка может быть дешёвой или не нужна.

Минусы:

- требует сортировки (см. предыдущий раздел);
- при больших объёмах может перейти в external sort;
- в целом дороже по CPU/I/O, если hash aggregate “влезает” в память.

### Пошагово: выбор алгоритма

Упрощённо движок делает так:

1. Оценивает **количество строк** и **количество групп** (по статистике).
2. Оценивает **доступную память** для агрегации.
3. Смотрит, есть ли **уже отсортированные данные** по ключу (индексы, предыдущие сортировки).
4. Строит варианты:
   - hash aggregate (в один проход, в памяти);
   - hash aggregate с возможным spill’ом;
   - sort aggregate (с учётом стоимости сортировки).
5. Выбирает план с минимальной оценкой.

### Простыми словами

Если упрощать до аналогии:

- **Hash aggregate** — это как **журнал по каждому городу**: у тебя есть полка с папками по городам (ключи), и ты сразу кладёшь каждую новую запись в нужную папку. Нужно достаточно полок/места, чтобы все города помещались.
- **Sort aggregate** — это как **сначала отсортировать все записи по городу**, а потом идти по списку и складывать значения для каждого города подряд. Места нужно меньше, но предварительная сортировка — дорогая операция.

### Картинка в голове

- Hash aggregate — **большой хеш‑словарь** в памяти, куда мы всё складываем.
- Sort aggregate — **отсортированный список**, по которому мы двигаемся сверху вниз, как по телефонному справочнику.

### Как запомнить

> **Hash aggregate** = “один проход, много памяти”.  
> **Sort aggregate** = “сортировка + один проход, меньше памяти”.

### Примеры

#### PostgreSQL

```sql
EXPLAIN ANALYZE
SELECT country, COUNT(*)
FROM users
GROUP BY country;
```

- Если стран немного и таблица не гигантская, обычно будет `HashAggregate`.
- Если групп очень много или есть already‑sorted порядок, PostgreSQL может выбрать `GroupAggregate` (sort aggregate).

#### MySQL / SQL Server

У них свои названия операторов, но идея та же: агрегация либо по хешу, либо по сортировке, иногда с использованием временных таблиц.

### Практика / реальные сценарии

- `GROUP BY` по столбцу с очень высокой кардинальностью (почти каждая строка — своя группа) часто ведёт к:
  - hash aggregate с огромным хеш‑столом (risk of spill),
  - или sort aggregate с большой сортировкой.
- Оптимизация может включать:
  - предварительную фильтрацию;
  - разбиение запроса на этапы;
  - агрегацию по сэмплу (для аналитики) вместо точного подсчёта.

### Типичные ошибки

- Считать, что `GROUP BY` — “дешёвая операция”.
- Игнорировать предупреждения о spill’ах хеша/сортировки.
- Ожидать, что `GROUP BY` по текстовому столбцу с миллионами уникальных значений будет “летать”.

### Что будет, если…

- …`GROUP BY` возвращает миллион групп, а памяти мало?
  - hash aggregate почти наверняка начнёт spill’ить на диск или разобьёт обработку на несколько проходов → сильное замедление.
- …искусственно заставить планировщик использовать sort aggregate при наличии дешёвого hash aggregate?
  - Ты заплатишь стоимость сортировки и не получишь выгод хеша, если только нет специфических причин (например, нужны данные в отсортированном виде для следующих операторов).

### Проверь себя

1. Почему hash aggregate обычно предпочтительнее sort aggregate при умеренном количестве групп?

<details><summary>Ответ</summary>
Потому что он делает **один проход по данным без сортировки**, используя хеш‑таблицу. Это дешевле по CPU и I/O, если хеш‑таблица помещается в памяти и количество групп не слишком велико. Sort aggregate требует дополнительного шага сортировки, что добавляет \(O(N \log N)\) по CPU и потенциально I/O.
</details>

2. Приведи пример, когда sort aggregate может оказаться выгоднее hash aggregate.

<details><summary>Ответ</summary>
Например, когда:

- количество групп очень велико (чуть ли не каждая строка — своя группа);
- данные уже почти отсортированы по ключу группировки (идут по индексу);
- памяти мало для большого хеша.

В этом случае сортировка может быть сравнительно недорогой, а sort aggregate будет аккуратно идти по отсортированному потоку без огромного хеша.
</details>

### Запомните

- `GROUP BY` — это тяжёлая операция, особенно при высокой кардинальности.
- Hash aggregate и sort aggregate — два разных способа реализовать одну идею, и планировщик выбирает между ними, исходя из кардинальности и памяти.

#### Где можно запутаться

Главная точка путаницы — **количество групп**:

- при hash aggregate память расходуется **по числу групп** (запись на каждую группу в хеше);
- при sort aggregate память расходуется скорее по **размеру текущего “окна”** при проходе по отсортированным данным.

Отсюда интуитивное правило:

- если групп **мало**, hash aggregate почти всегда выгоден (одна проходка, без сортировки);
- если групп **очень много** (почти каждая строка — своя группа), сортировка может оказаться стабильнее по памяти, даже если чуть дороже по CPU.

#### Мини‑шпаргалка по `GROUP BY`

- Мало групп, достаточно памяти → смотри в сторону **hash aggregate**.
- Много групп, данные уже отсортированы по ключу → хорош кандидат **sort aggregate**.
- Если в плане видишь hash aggregate со spill’ами — подумай либо об увеличении памяти под этот запрос, либо о переписывании запроса/разбиении агрегации на этапы.
