## 254. Inverted index и полнотекстовый поиск

### Цель раздела

Понять, **как устроен инвертированный индекс**, на котором держится полнотекстовый поиск (PostgreSQL GIN/GiST, Elasticsearch, Solr, OpenSearch, Lucene), что такое термы и документы, как хранятся posting list’ы, как выполняется поиск и ранжирование, и какие компромиссы возникают при обновлениях и больших объёмах текста.

### В этом разделе главное

1. Inverted index — это структура, которая вместо “документ → слова” хранит “слово → список документов/позиций”.
2. Поиск по нескольким словам сводится к операциям над списками: пересечению (`AND`), объединению (`OR`), разности (`NOT`) posting list’ов.
3. В реальных системах inverted index сильно **сжат** (varint, bitset, блоки), а также дополняется структурой сегментов/шардов.
4. Обновления (добавление/удаление документов) реализуются через **append‑only сегменты** и последующий merge (аналог compaction’а).
5. PostgreSQL использует GIN/GiST‑индексы для полнотекста, а внешние движки типа Elasticsearch строят полноценные inverted index‑кластеры.

### Термины

- **Документ** — единица полнотекстового поиска (строка в таблице, JSON‑объект, запись).
- **Терм (term)** — нормализованное слово/токен (после стемминга/лемматизации).
- **Posting list** — отсортированный список пар (doc_id, позиция/частота), где встречается терм.
- **Segment** — отдельный кусок inverted index (часто append‑only файл).
- **Merge** — объединение сегментов, удаление устаревших записей (аналог compaction).

### Теория и правила

#### 1) Как строится inverted index (упрощённо)

1. Берём документ (текст).
2. Прогоним его через **анализатор**:
   - токенизация (разбиение на слова),
   - нормализация (lowercase, стемминг/лемматизация),
   - фильтрация стоп‑слов (the, и, а и т.п.).
3. Для каждого терма записываем `(term, doc_id, позиция/частота)` в промежуточную структуру.
4. После обработки множества документов:
   - группируем по терму;
   - сортируем списки документов/позиций;
   - записываем как posting list’ы в сегменты.

#### 2) Как работает поиск

Запрос: `"database indexing" AND NOT "fulltext"`:

1. Анализируем запрос (так же, как документы): получаем термы `database`, `index`, `fulltext`.
2. Находим posting list’ы:
   - `P(database)`,
   - `P(index)`,
   - `P(fulltext)`.
3. Вычисляем:
   - `P(database) ∩ P(index)` — документы, где есть оба терма;
   - вычитаем `P(fulltext)` — исключаем документы с этим термом.
4. Получаем список doc_id, иногда с дополнительными метриками (частота, позиции) для ранжирования.

#### 3) Ранжирование (очень коротко)

Алгоритмы ранжирования (TF‑IDF, BM25 и др.) используют:

- **TF (term frequency)** — насколько часто терм встречается в документе;
- **IDF (inverse document frequency)** — насколько терм редкий среди всех документов;
- длину документа и другие факторы (boost по полям, свежесть и т.д.).

Результат — **оценка релевантности** для каждого документа.

### Пошагово: обновления и сегменты

Inverted index обычно **append‑only**:

1. Новые документы:
   - добавляются в новый сегмент (или mem‑структуру, которая потом сбрасывается);
   - не трогают старые сегменты.
2. Удаления:
   - помечаются как tombstone’ы/дельта‑структуры;
   - фактическое удаление происходит при merge.
3. Merge:
   - объединяет несколько сегментов в один;
   - удаляет помеченные к удалению документы;
   - пересобирает posting list’ы.

Это похоже на LSM‑подход, но с упором на **поиск по тексту**.

### Простыми словами

Inverted index — это как **огромный словарь, в котором к каждому слову прикреплён список страниц книг, где это слово встречается**:

- когда ты ищешь слово, ты смотришь в словарь и сразу получаешь список страниц;
- чтобы найти книги, где встречаются два слова, ты пересекаешь списки страниц;
- чтобы исключить книги, где есть третье слово, ты вычитаешь соответствующий список.

### Картинка в голове

- Много “строчек” вида: `слово → [список doc_id]`.
- Для каждого слова список отсортирован, чтобы быстро пересекать/объединять.

### Как запомнить

> **Обычный индекс (B‑tree)**: “ключ → позиция строки”.  
> **Inverted index**: “слово → список документов”.

### Примеры

- **PostgreSQL**:

```sql
CREATE INDEX idx_posts_fulltext
    ON posts
    USING GIN (to_tsvector('russian', title || ' ' || body));

SELECT *
FROM posts
WHERE to_tsvector('russian', title || ' ' || body)
      @@ plainto_tsquery('russian', 'алгоритмы индексы');
```

- **Elasticsearch**:
  - под капотом использует Lucene с inverted index;
  - данные распределены по шард‑сегментам с собственными posting list’ами.

### Практика / реальные сценарии

- Индекс по полю `title/body` в PostgreSQL с GIN:
  - даёт быстрый поиск по словам, но его обновление дорогое;
  - для очень больших объёмов текста часто выносят поиск в отдельный кластер Elasticsearch.
- В Elasticsearch:
  - быстрый поиск и агрегации по термам;
  - но стоимость хранения и обновления индекса растёт с количеством документов и уникальных термов.

### Типичные ошибки

- Попытка использовать полнотекстовый индекс как “замену” обычному B‑дереву для точных совпадений по идентификатору.
- Игнорирование стоимости обновления полнотекстового индекса при массовых апдейтах документов.
- Чрезмерно сложные запросы (много термов, сложные булевы формулы) без понимания их внутренней стоимости.

### Что будет, если…

- …часто обновлять большие текстовые поля в PostgreSQL с GIN‑индексом?
  - Индекс будет постоянно перестраиваться/обновляться, что может резко увеличить нагрузку на CPU и диск, а также замедлить все операции записи.
- …не контролировать рост числа термов (например, позволить хранить произвольные токены из логов)?
  - Inverted index раздуется: posting list’ы станут длинными и фрагментированными, сжатие будет хуже, время поиска вырастет.

### Проверь себя

1. В чём принципиальное отличие inverted index от обычного B‑tree по направлению “ключ → данные”?

<details><summary>Ответ</summary>
В B‑tree мы храним “ключ → позиция строки/страницы” и ищем по конкретному ключу или диапазону. В inverted index мы храним “терм (слово) → список документов”, и основная операция — работа с этими списками (пересечение, объединение, вычитание) при полнотекстовом поиске.
</details>

2. Почему операции `AND`/`OR`/`NOT` в полнотекстовом запросе хорошо ложатся на inverted index?

<details><summary>Ответ</summary>
Потому что они превращаются в **операции над отсортированными списками/битмапами** doc_id: `AND` — пересечение списков, `OR` — объединение, `NOT` — разность. Эти операции можно выполнять эффективно (линейно по сумме длин списков) и сильно оптимизировать за счёт сжатия и блоков.
</details>

### Запомните

- Inverted index — ключевая структура для полнотекста, и она живёт по своим правилам: сегменты, merge, сжатие, ранжирование.
- Важно понимать, когда имеет смысл оставить полнотекст в СУБД (PostgreSQL GIN) и когда — вынести в специализированный движок.

#### Где можно запутаться

Здесь часто путают **“поиск по тексту”** и **“поиск по точному ключу”**:

- для точного ключа (ID, e‑mail) инвертированный индекс **избыточен** — лучше обычный B‑tree;
- для гибкого текста с морфологией, опечатками, несколькими термами и ранжированием B‑tree **неподходит**, нужен inverted index.

Ещё одно место запутаться — думать, что полнотекст “ничего не стоит” и его можно безболезненно включить на любые поля. На самом деле:

- каждый новый документ добавляет записи во множество posting list’ов;
- большие обновления и удаления документов требуют merge/compaction;
- размер индекса растёт вместе с количеством уникальных термов.

Поэтому полнотекстовые индексы нужно проектировать так же аккуратно, как и любые другие: **по нужным полям, с понятным профилем запросов**.

#### Мини‑шпаргалка по полнотексту

- Если у тебя немного текста и простые условия (`LIKE '%foo%'` по небольшим строкам) — можно обойтись без тяжёлого полнотекста.
- Если нужна выдача “как в поисковике” (поиск по словам, с учётом форм, с сортировкой по релевантности) — нужен **inverted index** (GIN в PostgreSQL или внешний движок).
- Если полнотекстовых запросов очень много и данные большие — почти всегда лучше вынести поиск в отдельный кластер (Elasticsearch/аналог), а в СУБД хранить только данные и ссылки.
