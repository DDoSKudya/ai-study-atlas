## Справочник по части (ключевые пункты)

### Алгоритмы JOIN

| Алгоритм      | Когда хорош                                                    | Когда плох                                         | Ключевые моменты                                      |
| ------------- | -------------------------------------------------------------- | -------------------------------------------------- | ----------------------------------------------------- |
| Nested loop   | Маленькая внешняя таблица, есть индекс на внутренней стороне  | Большие обе стороны без индексов                   | Простой, чувствителен к индексам и кардинальности    |
| Hash join     | Большие объёмы, равенства (`=`), умеренная кардинальность      | Нехватка памяти, сложные предикаты                | Требует памяти, может spill’ить на диск              |
| Merge join    | Обе стороны уже отсортированы или легко сортируются           | Дорого сортировать большие объёмы                  | Слияние отсортированных потоков, линейная сложность  |

### Сортировка и агрегация

| Механизм           | Плюсы                                      | Минусы                                          | Где встречается                                     |
| ------------------ | ------------------------------------------- | ----------------------------------------------- | --------------------------------------------------- |
| In‑memory sort     | Быстро, без диска                          | Ограничен памятью                               | Малые/средние сортировки, индексация                |
| External sort      | Работает на очень больших объёмах          | Много I/O, временные файлы                      | Крупные `ORDER BY`, `DISTINCT`, `GROUP BY`          |
| Hash aggregate     | Один проход, без сортировки                | Требует памяти, плохо при огромном числе групп  | `GROUP BY`, агрегации, часто по числовым/категориям |
| Sort aggregate     | Меньше зависимость от памяти               | Требует сортировки                               | `GROUP BY` с очень высокой кардинальностью          |

### Структуры хранения

| Структура        | Сильные стороны                         | Слабые стороны                                    | Типичные системы                          |
| ---------------- | ---------------------------------------- | ------------------------------------------------- | ----------------------------------------- |
| B‑дерево         | Быстрые чтения/поиск, хорошие диапазоны | Случайные записи, плохое поведение при write‑heavy | PostgreSQL, MySQL InnoDB, Oracle, SQL Server |
| LSM‑дерево       | Быстрые записи, хорошая компрессия      | Read/write amplification, сложная compaction      | Cassandra, HBase, RocksDB, LevelDB, MyRocks |
| Bitmap‑индекс    | Фильтрация по low‑cardinality колонкам  | Плохо для high‑cardinality, дорогие апдейты       | DWH, OLAP, columnstore                    |
| Inverted index   | Быстрый полнотекстовый поиск            | Дорогие обновления, рост термов                  | PostgreSQL GIN, Elasticsearch, Lucene     |

### Буферный менеджер

| Параметр/идея                    | Суть                                                                                   |
| -------------------------------- | -------------------------------------------------------------------------------------- |
| Buffer pool / shared_buffers     | Сколько памяти отдаём под страницы данных/индексов                                     |
| Алгоритм замены (LRU/clock/др.)  | Как выбираем, какие страницы вытолкнуть                                                |
| Cache hit ratio                  | Доля обращений, отслуженных из кэша (но это не единственный показатель здоровья)      |
| Scan pollution                   | Ситуация, когда разовые сканы вытесняют горячие страницы                               |

## Частые сценарии

### Сценарий 1. Запрос с внезапным переходом на external sort

- Был отчёт `SELECT ... ORDER BY ...`, который работал секунды; после роста данных стал работать минуты.
- В планах появился `Sort Method: external merge  Disk: ...MB`.
- Решения:
  - увеличить `work_mem`/аналогально для конкретного запроса;
  - пересмотреть индексы (возможно, можно использовать индекс вместо явной сортировки);
  - оптимизировать запрос (фильтровать раньше, уменьшать ширину строк).

### Сценарий 2. JOIN без индексов на больших таблицах

- Запрос объединяет две крупные таблицы по полям без индекса.
- Планировщик выбирает nested loop с полным сканом или hash join с огромными сканами.
- Решения:
  - добавить подходящий индекс;
  - реструктурировать запрос (разбить на этапы, предварительные фильтры);
  - обновить статистику, чтобы планировщик лучше оценивал селективность.

### Сценарий 3. LSM‑кластер “захлёбывается” в compaction

- В Cassandra/HBase/RocksDB растёт latency записей и чтений.
- Мониторинг показывает высокую нагрузку от compaction, рост write amplification.
- Решения:
  - настроить стратегию compaction (границы уровней, параллелизм);
  - добавить диски/узлы;
  - уменьшить частоту/объём тяжелых операций (batch updates, bulk load).

### Сценарий 4. Полнотекстовый поиск “съедает” базу

- В PostgreSQL GIN‑индексы на больших текстовых полях сильно увеличивают размер базы и замедляют обновления.
- Решения:
  - вынести тяжёлый полнотекст в отдельный кластер (Elasticsearch);
  - индексировать только нужные поля и/или сжать словарь термов;
  - использовать партиционирование/архивирование старых документов.

### Сценарий 5. Неправильный размер буферного пула

- При слишком маленьком буферном пуле:
  - постоянные physical read’ы, высокая нагрузка на диск;
  - даже простые запросы тормозят.
- При слишком большом:
  - ОС вынуждена свопиться;
  - другие процессы (приложения) страдают от нехватки памяти.

## Вопросы для самопроверки по части

1. Как планировщик выбирает между nested loop, hash join и merge join? Какие основные факторы он учитывает?
2. Чем отличаются in‑memory sort и external sort, и как это отражается на поведении запросов?
3. В чём суть LSM‑деревьев и как compaction влияет на write/read amplification?
4. Когда bitmap‑индексы и inverted index особенно полезны, а когда лучше не использовать такие структуры?
5. Как буферный менеджер влияет на производительность запросов, даже если алгоритмы JOIN/индексов выбраны оптимально?

<details><summary>Ответы</summary>

1. Планировщик оценивает **размеры таблиц, селективность условий, наличие индексов, стоимость сортировки и доступную память**. Nested loop хорош при малой внешней таблице и индексах, hash join — при больших объёмах и равенствах, merge join — когда данные уже отсортированы или сортировка недорога. Решение принимается по модели стоимости I/O+CPU+память.
2. In‑memory sort выполняется полностью в оперативной памяти и ограничен её объёмом; external sort разбивает данные на отсортированные раны и сливает их на диске, что даёт возможность сортировать огромные объёмы, но ценой множества операций чтения/записи. В планах и метриках это видно как использование временных файлов/`tempdb` и резкий рост I/O.
3. LSM‑деревья оптимизируют запись, сначала помещая данные в память (memtable) и потом сбрасывая их в отсортированные SSTable’ы, а compaction периодически сливает и чистит эти файлы. Write amplification растёт, потому что одна логическая запись переписывается несколько раз при compaction, а read amplification — потому что чтение может требовать просмотра множества SSTable’ов и уровней.
4. Bitmap‑индексы полезны для low‑cardinality колонок в аналитике (DWH, OLAP), где запросы фильтруют по нескольким атрибутам; inverted index — для полнотекста. Они плохо подходят для high‑cardinality и write‑heavy OLTP‑нагрузок, где обновления и размер индексов становятся проблемой.
5. Буферный менеджер определяет, **будут ли страницы для JOIN/индексов/агрегаций находиться в памяти или читаться с диска**. Даже идеальный план с правильными алгоритмами будет медленным, если кэш слишком мал или загрязнён, и каждая операция чтения требует дискового I/O. Поэтому настройки кэша и понимание паттернов доступа критически важны.

</details>

## Типичные ошибки по части

- Рассматривать `JOIN`, `ORDER BY`, `GROUP BY` как “просто SQL‑операторы”, не думая об их алгоритмической реализации.
- Считать, что LSM‑движок автоматически “ускорит всё”, не понимая стоимости compaction и write amplification.
- Использовать bitmap‑ или полнотекстовые индексы без учёта их влияния на размер базы и стоимость обновлений.
- Игнорировать spilling сортировок/агрегаций и состояние временных хранилищ (`tempdb`, временные файлы).
- Настраивать кэш и память “по шаблонам из интернета”, не опираясь на реальные паттерны нагрузки и метрики.

## Резюме части XLII

- **Алгоритмы JOIN, сортировки и агрегации** — это сердце выполнения запросов: nested loop/hash/merge join, in‑memory и external sort, hash/sort aggregate. Понимание их работы позволяет читать планы не как “чёрный ящик”, а как набор осмысленных шагов.
- **Структуры данных внутри СУБД** — B‑деревья, LSM‑деревья, bitmap‑ и inverted‑индексы, альтернативные деревья — задают фундаментальные trade‑off’ы между скоростью записи, чтения, размером и сложностью эксплуатации.
- **Буферный менеджер и алгоритмы замены страниц** определяют, насколько эффективно всё это будет работать в памяти и сколько раз придётся ходить на диск.
- Освоив эту часть, ты можешь смотреть на любую СУБД не только как на “ящик, который выполняет SQL”, а как на **набор конкретных алгоритмов и структур данных**, с которыми можно работать осознанно: выбирать подходящие настройки, диагностировать проблемы и объяснять поведение системы простыми и точными терминами.
