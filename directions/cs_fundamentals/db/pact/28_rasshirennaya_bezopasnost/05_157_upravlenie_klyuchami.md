## 157. Управление ключами

#### В этом разделе главное

- Шифрование без грамотного управления ключами даёт **ложное чувство безопасности**.
- Ключи должны храниться и выдаваться **централизованно**: KMS/Vault, а не «.env в репозитории».
- Важны **ротация ключей**, разделение по доменам данных и аудит доступа.

#### Простыми словами

Представь, что шифрование — это прочный замок на двери, а ключ висит на гвоздике рядом с дверью. Формально замок есть, но реальной безопасности нет.  
То же самое, когда ключи лежат:

- в коде,
- в открытых конфигурациях,
- в общем секрет‑хранилище без разделения прав.

#### KMS и Vault

**KMS (Key Management Service)**:

- Облачные и on‑prem решения:
  - AWS KMS,
  - Azure Key Vault,
  - GCP KMS,
  - HashiCorp Vault (может выполнять роль KMS).
- Основные функции:
  - генерация ключей;
  - хранение мастер‑ключей;
  - операции `encrypt/decrypt/sign` по API;
  - логирование всех операций (аудит).

**HashiCorp Vault**:

- Гибкий инструмент для:
  - хранения секретов (пароли, токены, ключи);
  - выдачи временных учётных записей в БД;
  - шифрования данных «как сервиса» (Transit backend).

Типичный паттерн:

- Приложение **не хранит ключ локально**, а обращается к Vault/KMS:
  - получает зашифрованные данные из БД;
  - отправляет их в KMS/Vault с запросом `decrypt`;
  - получает расшифрованный результат.

#### Ротация ключей

- Ротация — регулярная смена ключей (раз в N дней/месяцев или при инциденте).
- Важные моменты:
  - **как перевыпускать ключ, не потеряв данные**:
    - хранить метаданные «каким ключом шифровали»;
    - поддерживать несколько ключей для расшифровки (current + previous).
  - **как не сломать старые бэкапы**:
    - либо хранить старые ключи для расшифровки бэкапов;
    - либо пересоздавать бэкапы после ротации.

Представь простую историю:

- сегодня ты шифруешь данные ключом `K1` и складываешь их в БД и бэкапы;
- через полгода ты решаешь «повысить безопасность» и заводишь новый ключ `K2`;
- если ты просто начнёшь шифровать всё новым ключом и **забудешь про `K1`**, то:
  - старые данные и бэкапы, зашифрованные `K1`, станут **нечитаемыми**.

Поэтому реальная ротация всегда выглядит так:

1. создаёшь новый ключ (`K2`) в KMS/Vault и помечаешь его как «текущий»;  
2. **перестаёшь шифровать новым старым ключом** (`K1`), но пока его не удаляешь;  
3. новые данные шифруются `K2`, старые **по‑прежнему расшифровываются `K1`**;  
4. постепенно (или пакетно) можно перевычитывать/перешифровывать старые данные с `K1` на `K2`;  
5. только после этого `K1` можно выводить из эксплуатации (с учётом бэкапов).  

Простой паттерн:

- `key_id` + `ciphertext`:

```sql
CREATE TABLE secret_data (
    id        bigserial PRIMARY KEY,
    key_id    text NOT NULL,
    payload   bytea NOT NULL
);
```

- При шифровании:
  - выбирается текущий `key_id`;
  - вызывается KMS/Vault для шифрования;
  - результат и `key_id` сохраняются в БД.
- При расшифровке:
  - из строки читается `key_id`;
  - KMS/Vault вызывает нужный ключ для расшифровки.

Такое разделение:

- позволяет одновременно поддерживать **несколько поколений ключей**;  
- даёт возможность по логам KMS увидеть, **какие именно данные шифруются/расшифровываются каким ключом**;  
- упрощает ротацию: можно по шагам переводить данные с одного `key_id` на другой, не ломая работу приложения.  

#### Разделение ключей

- Не стоит одним и тем же ключом шифровать:
  - пароли/токены,
  - PII,
  - финансовые данные,
  - служебные конфигурации.
- Лучше разделить:
  - `user_pii_key`,
  - `payment_data_key`,
  - `logs_key` и т.п.

Так компрометация одного ключа не раскрывает все виды данных.

Полезно думать о ключах как о **зонах доступа**:

- есть «зона ПИИ», «зона платежей», «зона логов»;  
- у каждого ключа — своя зона;  
- сервисы получают доступ **только к тем KMS‑ключам, которые им действительно нужны**:
  - платежный сервис — к `payment_data_key`;
  - CRM — к `user_pii_key`;
  - система логирования — к `logs_key`.

Тогда даже если один сервис будет скомпрометирован, атакующий не сможет «по цепочке» расшифровать все остальные данные.

#### Аудит доступа к ключам

- KMS/Vault обычно ведут подробные логи:
  - кто/что (ID сервиса) вызвал операцию;
  - какой ключ использовался;
  - время и результат операции.
- Эти логи нужно:
  - собирать в централизованную систему логов;
  - настраивать алерты на аномальную активность:
    - всплески `decrypt`,
    - доступы из необычных регионов/служб.

Примеры подозрительных ситуаций:

- сервис, который обычно делает 10–20 `decrypt` в минуту, внезапно начал делать сотни/тысячи — это может быть массовая выгрузка данных;  
- появляются операции `decrypt` с ключом, который давно должен использоваться только для расшифровки редких «старых» данных;  
- попытки доступа к ключам из неожиданных регионов, аккаунтов или сервисов.  

Такие вещи сложно заметить вручную, но их **можно отловить автоматическими алертами** в системе логирования/мониторинга.

#### Типичные ошибки

- «Давайте положим ключ в `.env` и забудем» — такие файлы часто оказываются в логах, дампах и бэкапах.
- «У нас один мастер‑ключ на всё приложение» — при его утечке теряется весь смысл шифрования.
- «Ротация ключа = переписать значение в конфиге» — без продуманной миграции старых данных это может сломать расшифровку.

Дополнительные ловушки:

- ключи жёстко «зашиты» в код (`const MASTER_KEY = "..."`), и для их смены нужно заново собирать и выкатывать приложение;  
- один и тот же ключ используется **и** для шифрования данных, и для подписи токенов — компрометация ломает сразу два механизма;  
- в тестовых/стейджинг‑окружениях используются **те же ключи**, что и в проде (так делать крайне нельзя).  

Хорошая практика: считать ключи **такими же критичными артефактами, как доступы к прод‑БД**, и относиться к ним не менее аккуратно.

#### Проверь себя

<details>
<summary>1. Зачем хранить вместе с шифротекстом идентификатор ключа (`key_id`)?</summary>

Потому что через год у тебя уже может быть несколько ключей, и без `key_id` БД не поймёт, каким именно ключом зашифрован конкретный кусок данных.  
`key_id` позволяет:
- выбрать правильный ключ для расшифровки;  
- постепенно переводить данные с одного ключа на другой без поломки старых записей.  
</details>

<details>
<summary>2. Какие преимущества даёт KMS/Vault по сравнению с хранением ключей в файле конфигурации?</summary>

KMS/Vault даёт централизованное хранение и управление ключами: доступ по политике, аудит всех операций, простую ротацию, возможность временных токенов/учёток.  
Ключи не лежат «просто файлом» на сервере приложения, поэтому их сложнее украсть при взломе одного хоста.  
К тому же можно легко отозвать доступ конкретному сервису или человеку, не переламывая все конфиги.
</details>

<details>
<summary>3. Как организовать ротацию ключей, чтобы не потерять доступ к старым данным и бэкапам?</summary>

Схема в общих чертах такая:
- заводим новый ключ (`K2`), помечаем его как текущий;  
- новые данные шифруем `K2`, но старый ключ `K1` **оставляем в KMS** для расшифровки старых данных и бэкапов;  
- храним в строках `key_id`, чтобы понимать, каким ключом шифровали;  
- по возможности постепенно перешифровываем старые данные с `K1` на `K2`;  
- только когда уверены, что старые данные и бэкапы не нужны или перенесены, выводим `K1` из эксплуатации.  
</details>

Если хочется закрепить:

1. Нарисуй на бумаге схему: «приложение → БД → KMS/Vault», подпиши, где хранятся ключи, где лежат шифротексты.  
2. Придумай два ключа (`K1`, `K2`) и мысленно проиграй сценарий ротации по шагам для таблицы `secret_data`.  
3. Попробуй объяснить вслух (как будто другу), **зачем нужны `key_id` и KMS, а не просто `.env`**.
