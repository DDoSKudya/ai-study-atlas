## 154. Защита от инъекций

#### В этом разделе главное

- **Основной инструмент защиты — подготовленные выражения и параметры.**
- Валидация и allowlist усиливают защиту, но **не заменяют параметры**.
- Ущерб от инъекции сильно зависит от того, **какие права у пользователя БД**, под которым работает приложение.

#### Простыми словами

Представь, что ты бронируешь билет: ты выбираешь значения из выпадающих списков (город, дата), а не вводишь произвольный текст в строку SQL.  
Параметризованный запрос — это такой же «формуляр для БД»: структура фиксирована, а значения подставляются в строго отведённые места.

#### Теория и правила

**Правило 1. Всегда используй параметры.**

- В большинстве драйверов есть стандартный синтаксис:
  - PostgreSQL: `$1`, `$2` или `%s` (в зависимости от драйвера),
  - MySQL: `?` или `%s`,
  - ORM: плейсхолдеры / биндинги.
- СУБД знает, что это **данные**, а не код, и корректно экранирует значения.

**Правило 2. Валидация ввода — доп. слой защиты.**

- Проверяй:
  - тип (число/строка/дата),
  - длину,
  - формат (регулярки, allowlist).
- Валидация полезна и для UX, и для производительности (отсеиваешь мусор до БД).

**Правило 3. Allowlist > blacklist.**

- **Allowlist**: список допустимых значений (`status IN ('NEW','PAID','CANCELLED')`).
- **Blacklist**: «запрещённые паттерны» (`"DROP"`, `"--"` и т.п.).
- Чёрные списки легко обойти; белые лучше соответствуют доменной логике.

**Правило 4. Минимальные привилегии.**

- Пользователь БД для продакшена:
  - **не должен уметь DDL** (`CREATE/DROP/ALTER TABLE`) без крайней необходимости,
  - **не должен уметь `COPY PROGRAM`**, `xp_cmdshell` и аналогичные команды,
  - **не должен иметь доступ ко всем схемам/таблицам**, если приложение использует только часть.
- Даже при успешной инъекции ущерб будет ограничен доступом этого аккаунта.

#### Примеры

Параметризованный запрос (PostgreSQL, Python/psycopg):

```python
cur.execute(
    "SELECT id, email FROM users WHERE login = %s AND password_hash = crypt(%s, password_hash)",
    (login, password),
)
```

Пример allowlist для сортировки:

```python
def list_users(db, order_by: str):
    allowed = {"created_at", "login", "last_login"}
    if order_by not in allowed:
        order_by = "created_at"

    # Порядок колонок выбираем из allowlist, но значения — только через параметры
    query = f"SELECT id, login FROM users ORDER BY {order_by} DESC LIMIT %s"
    return db.execute(query, (100,)).fetchall()
```

> Здесь единственный «кусок SQL из строки» — имя колонки для сортировки, но оно выбирается из жёсткого списка, а не из произвольного ввода.

#### Картинка в голове

Думай о защите так:

- **Параметры** — это «автоматический ремень безопасности» в каждом запросе.
- **Валидация** — это рамка в дверях: мусор до БД не доходит.
- **Права в БД** — это стенки трассы: даже если кто‑то вылетел, он не попадает на встречку.

#### Типичные ошибки

- «У нас маленький админ‑скрипт, там можно и конкатенацию» — ровно такие скрипты часто запускают с правами суперпользователя.
- «Это только для отчётов, ничего критичного» — отчёты могут содержать PII и финансовые данные.
- «Я проверяю, что строка — это число, и потом просто подставляю её в SQL» — лучше использовать параметры даже при такой проверке.

#### Как запомнить

- Если в коде есть `"... WHERE ... " + userInput`, почти всегда это можно заменить на параметры.
- В идеале в проекте есть **правило code review**: «любая строка SQL без параметров требует отдельного обсуждения».

Хороший тест для себя: **открой любой файл с SQL в проекте и попробуй найти хотя бы один запрос без параметров**. Если такой нашёлся — подумай, можно ли его переписать по правилам этого раздела.

#### Проверь себя

<details>
<summary>1. Почему «экранировать кавычки руками» хуже, чем использовать параметры?</summary>

Потому что ручное экранирование легко сделать неправильно: в разных СУБД свои правила, легко забыть про какой‑то символ или новый тип данных.  
Параметризованные запросы делегируют экранирование драйверу/СУБД: они **строго отделяют данные от кода**, и риск ошибки сильно меньше.  
К тому же ручное экранирование часто «лечит симптомы», а не саму проблему смешения кода и данных.
</details>

<details>
<summary>2. Зачем нужен принцип минимальных привилегий, если запросы и так параметризованы?</summary>

Параметры защищают от того, чтобы ввод не превратился в SQL‑код, но **не защищают**, если в приложении есть логические баги или другие уязвимости (RCE, уязвимые админ‑скрипты).  
Если у аккаунта БД есть только нужные права (например, нет `DROP TABLE`, нет доступа ко всем схемам), то даже при взломе ущерб сильно ограничен.  
Это «вторая линия обороны», если первая (параметры) где‑то дала сбой.
</details>

<details>
<summary>3. Почему allowlist предпочтительнее blacklist?</summary>

Blacklist пытается перечислить **все плохие варианты** (запрещённые слова/паттерны), но атакующий всегда может придумать что‑то новое или обойти фильтр.  
Allowlist, наоборот, явно перечисляет **только допустимые** значения (например, фиксированный набор статусов или имён колонок), и всё остальное автоматически запрещено.  
Поэтому allowlist лучше совпадает с бизнес‑логикой и намного надёжнее в долгую.
</details>

Если что‑то не до конца понятно — попробуй взять один реальный запрос из своего (или учебного) проекта и:

1. выписать его в виде «до» (как сейчас);  
2. переписать его «после» по правилам этого раздела;  
3. проговорить вслух, **какие именно риски ушли** после переписывания.
