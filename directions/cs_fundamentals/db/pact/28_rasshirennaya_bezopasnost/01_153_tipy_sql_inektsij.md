## 153. Типы SQL‑инъекций

#### В этом разделе главное

- SQL‑инъекция — это **ошибка в составе запроса**, а не «хитрый пользователь».
- Важна **паттерн‑знание**: научиться *узнавать* разные виды инъекций по коду и по логам.
- Первое практическое правило: **никогда не склеивай SQL‑строки с пользовательским вводом**.

#### Простыми словами

Представь, что у тебя есть шаблон письма: «Привет, `<имя>`!». Если вместо имени ты без проверки вставляешь текст от пользователя, он может подставить туда, что угодно, вплоть до оскорбления.  
С SQL‑инъекцией то же самое: ты ожидаешь «Иванов», а получаешь `"Иванов' OR 1=1 --"`, и эта строка превращает твой запрос из «найди пользователя Иванов» в «верни всех пользователей».

#### Теория и правила

Ключевая идея: **в SQL есть код и есть данные**. Инъекция происходит, когда данные становятся частью кода.

**Классическая SQL‑инъекция**:

- Приложение собирает строку:
  - `"... WHERE login = '" + userInput + "' AND password = '" + passInput + "'"`.
- Атакующий подставляет: `userInput = 'admin'`, `passInput = "' OR '1'='1"`.
- В итоге получается:  
  `WHERE login = 'admin' AND password = '' OR '1'='1'`.
- Условие `'1'='1'` всегда истинно — авторизация обходится.

**Union‑based инъекция**:

- Атакующий добавляет `UNION SELECT ...` к исходному запросу.
- Цель — получить данные из других таблиц (логины, хеши паролей и т.п.) в том же результате.

**Error‑based инъекция**:

- Атакующий вызывает в запросе выражения, которые при ошибке «вываливают» полезную информацию:
  - неверное приведение типов,
  - деление на ноль,
  - вызов функций с некорректными аргументами.
- В текст ошибки иногда попадает часть запроса, имена таблиц и другие детали.

**Blind SQL‑инъекция**:

- Приложение **не показывает ошибку и результат**, но ведёт себя чуть по‑разному:
  - возвращает 200/404,
  - меняет размер страницы,
  - ведёт себя по времени.
- Атакующий задаёт вопросы в виде SQL‑условий и по реакции постепенно извлекает данные.

**Time‑based blind**:

- Частный случай blind‑инъекции: в условии используется функция задержки (`SLEEP`, `pg_sleep`).
- Пример: если условие истинно — «спи 5 секунд», если ложно — не спи.
- По времени ответа можно восстановить биты пароля/хеша.

**Second‑order injection**:

- На первом шаге приложение **сохраняет опасные данные как безобидные** (например, в профиле пользователя).
- На втором шаге эти данные попадают в другой запрос **без параметров**, и тогда код срабатывает.
- Опасно, потому что может пройти очень много времени между записью и срабатыванием.

#### Примеры (уязвимый vs безопасный код)

Пример уязвимого кода (псевдо‑Python, но идея универсальна):

```python
def get_user(db, login, password):
    # ПЛОХО: конкатенация строк
    query = f"SELECT id FROM users WHERE login = '{login}' AND password = '{password}'"
    return db.execute(query).fetchone()
```

Безопасный вариант с параметризованным запросом:

```python
def get_user(db, login, password):
    # ХОРОШО: параметры передаются отдельно
    query = "SELECT id FROM users WHERE login = %s AND password = %s"
    return db.execute(query, (login, password)).fetchone()
```

Пример time‑based blind‑инъекции (для понимания, **как выглядит полезная нагрузка**):

- Оригинальный запрос:  
  `SELECT * FROM products WHERE id = <user_input>`
- Ввод атакующего (PostgreSQL):  
  `1; SELECT CASE WHEN (SELECT current_user = 'postgres') THEN pg_sleep(5) ELSE pg_sleep(0) END--`

Если запрос внезапно стал занимать ~5 секунд — вероятно, условие истинно.

> На практике firewall, драйверы и пул соединений могут не дать выполнить несколько операторов в одном запросе, но паттерн атомарного вопроса «ДА/НЕТ через задержку» остаётся тем же.

#### Пошаговый сценарий атаки (как думает злоумышленник)

Чтобы лучше понять, «за что ты борешься», полезно пройти весь путь глазами атакующего.

1. **Поиск точки входа.**  
   Атакующий ищет любые формы/эндпоинты, где его ввод попадает в запрос:
   - поля логина/пароля;
   - формы поиска (`search`, `q`);
   - фильтры в таблицах/отчётах (status, sort, filter).

2. **Проверка, «дышит ли» SQL.**  
   Он пробует ввести:
   - одинарную кавычку `'`;
   - `"` или `\`;
   - куски типа `"' OR 1=1 --"`.  
   Если приложение:
   - падает с ошибкой SQL,
   - начинает возвращать странные результаты,
   — значит, где‑то есть незащищённая конкатенация.

3. **Определение структуры БД.**  
   Через union‑based или error‑based инъекции атакующий пробует:
   - понять количество столбцов в запросе (`ORDER BY 1`, `ORDER BY 2`, …);
   - считать имена таблиц и колонок из системных представлений (`information_schema.*`, `pg_catalog.*`).

4. **Извлечение полезных данных.**  
   Как только структура понятна, цель — вытащить:
   - логины/хеши паролей;
   - токены сессий;
   - PII/платёжные данные.

5. **Закрепление в системе.**  
   Дальше атакующий может:
   - создавать новых пользователей с нужными правами;
   - менять данные (балансы, статусы заказов);
   - подготавливать почву для дальнейших атак (second‑order).

Твоя задача как разработчика — **сломать этот сценарий как можно раньше**: уже на шагах 1–2 сделать так, чтобы даже самые смелые вводы оставались просто данными, а не кодом.

#### Как увидеть SQL‑инъекцию в логах и поведении

Типичные признаки, по которым даже «тупой мониторинг» может подсказать проблему:

- в логах появляются запросы с кусками пользовательского ввода, в которых видны:
  - `OR 1=1`,
  - `UNION SELECT`,
  - `--` (SQL‑комментарий),
  - функции `pg_sleep`, `SLEEP`, `BENCHMARK`;
- в логах ошибок часто появляются сообщения:
  - «unterminated string literal»,
  - «syntax error near 'OR'» или `'UNION'`;
- по метрикам видно, что:
  - резко увеличилось число ошибок SQL;
  - резко выросло количество очень длинных запросов.

Если настроить поиск по логам на подобные паттерны, можно **поймать попытки инъекций задолго до успешного взлома**.

#### Картинка в голове

Думай о запросе как о конструкторе:

- **каркас** — фиксированный SQL‑шаблон (SELECT, FROM, WHERE, JOIN);
- **кирпичики** — значения, которые ты подставляешь (логин, цена, дата).

Инъекция — это когда пользователю дали возможность подменить кусок **каркаса**, а не просто **кирпичик**.

#### Типичные ошибки

- «У нас ORM, значит, мы защищены» — ORM можно обойти, если есть сырые запросы или `.where(f"price > {user_input}")`.
- «Мы экранируем кавычки вручную» — в разных СУБД разные правила экранирования; легко ошибиться.
- «Это внутренний сервис, там нет пользователей» — инъекция через логи, очереди, интеграции и внутренних API тоже реальна.

#### Как запомнить

- Любая строка вида `"... " + userInput + " ..."` вокруг SQL‑кода — **красный флаг**.
- Если ты видишь, что пользовательский ввод влияет на:
  - имена таблиц/колонок,
  - операторы (`AND`, `OR`, `UNION`),
  - функции (`pg_sleep`, `SLEEP`, `BENCHMARK`),
  — значит, есть риск инъекции.

#### Проверь себя

<details>
<summary>1. Чем blind SQL‑инъекция отличается от классической?</summary>

При классической инъекции результат виден сразу: в ответе или в тексте ошибки ты прямо видишь данные из БД или подробный SQL‑текст.  
При blind‑инъекции приложение **не показывает** результаты и подробные ошибки, поэтому атакующий восстанавливает данные по косвенным признакам (код ответа, размер страницы, время ответа).
</details>

<details>
<summary>2. Почему time‑based инъекция возможна, даже если приложение не показывает текст ошибок?</summary>

Потому что атакующий использует **время ответа** как канал передачи информации.  
Он вставляет в условие `SLEEP` / `pg_sleep`: если условие истинно — запрос «зависает» на несколько секунд, если ложно — отвечает сразу.  
Даже без текста ошибок и без вывода данных разница во времени позволяет по шагам «вычитывать» биты информации.
</details>

<details>
<summary>3. В чём особая опасность second‑order инъекций?</summary>

Вредоносные данные сначала **спокойно сохраняются** в БД как будто «просто строка», не ломая первый запрос.  
Проблема проявляется позже, когда эти данные попадают во **второй запрос**, который кто‑то написал с конкатенацией строк.  
Из‑за этого:
- трудно связать инъекцию с местом, где данные вводились;  
- между вводом и срабатыванием может пройти много времени;  
- обычные проверки «на входе» могут ничего не заметить, если они сделаны только в одном месте.
</details>
