## 204. Составные индексы

### Связь с §200: что такое составной индекс «внутри»

В §200 мы выяснили: в B-tree в листовой странице хранятся **ключ + TID**. Ключ может быть не одной колонкой, а **несколькими**: `(customer_id, status, created_at)`. Тогда ключ индекса — это как бы «составное значение»: сначала сортировка по первой колонке, при равенстве — по второй, при равенстве первых двух — по третьей. Один такой индекс = один B-tree, но порядок записей в нём задаётся всем набором колонок. Поэтому **порядок колонок в определении индекса** и определяет, для каких запросов индекс можно использовать (правило left-prefix). **INCLUDE** (§200, листовые страницы) добавляет в лист ещё колонки — не для порядка, а только чтобы не ходить в таблицу (index-only scan).

---

### В этом разделе главное

- Порядок колонок в составном индексе **критичен**: работает правило **left-prefix**.
- Составной индекс `(a, b, c)` может эффективно использоваться для условий по `(a)`, `(a, b)`, но не по `(b)` без `a`.
- **Covering‑индексы** позволяют выполнять `index-only scan`, если **все нужные колонки запроса есть в индексе**.
- В PostgreSQL есть `INCLUDE`‑колонки, которые добавляют поля в индекс **только для покрытия**, не влияя на сортировку/поиск.

---

### Простыми словами

Составной индекс — это как телефонный справочник, где записи отсортированы **сначала по фамилии, потом по имени, потом по отчеству**:

- можно быстро искать:
  - «все Петровы» (фамилия);
  - «все Петровы Ивановы» (фамилия + имя);
- но нельзя напрямую искать:
  - «все Иваны вне зависимости от фамилии» — для этого порядок сортировки не подходит.

---

### Простыми словами (второй проход — почему порядок колонок важен)

Индекс хранит записи в **одном фиксированном порядке**. Порядок задаётся списком колонок: сначала сортировка по первой колонке, при равенстве — по второй, при равенстве первых двух — по третьей. Поэтому «найти все строки, где вторая колонка = 5» без условия по первой колонке нельзя за один проход: такие строки **разбросаны** по всему индексу. А «найти все строки, где первая колонка = 1» — можно: они идут подряд. Отсюда правило: индекс `(a, b, c)` полезен для условий по **префиксу слева направо** — по `a`, по `(a, b)`, по `(a, b, c)`, но не по `b` или `c` без `a`.

---

### Теория и правила

#### Left-prefix rule — подробно на примерах

Индекс: `CREATE INDEX idx ON orders (customer_id, status, created_at);`  
Записи в индексе упорядочены: сначала по `customer_id`, внутри по `status`, внутри по `created_at`.

- `WHERE customer_id = 5` — подходит: читаем «кусок» индекса для этого клиента.
- `WHERE customer_id = 5 AND status = 'paid'` — отлично: сужаем тот же кусок.
- `WHERE customer_id = 5 AND status = 'paid' ORDER BY created_at` — идеально: внутри отфильтрованного набора данные по дате уже в порядке.
- `WHERE status = 'paid'` (без customer_id) — индекс **не может** эффективно использоваться: строки с status = 'paid' разбросаны по всем customer_id, они не идут подряд. Планировщик скорее выберет seq scan.

**Итог:** использование идёт только по **левому префиксу**. Начинать поиск с колонки, которая не первая в индексе, нельзя.

**Ещё раз по шагам: почему без первой колонки индекс не работает.** Представь индекс `(customer_id, status, created_at)`. Записи в индексе лежат **сгруппированно**: сначала все с customer_id=1 (внутри по status, потом по created_at), потом все с customer_id=2, и т.д. Поэтому «все строки с customer_id = 5» — это один **непрерывный кусок** индекса, его можно быстро прочитать. «Все строки с status = 'paid'» — это строки из **разных** customer_id (5, 7, 12, 100, …); в индексе они **не идут подряд**: сначала идут все с customer_id=1 (среди них есть и paid, и new), потом с customer_id=2, и т.д. То есть строки с status = 'paid' **размазаны** по всему индексу. Чтобы их собрать по индексу, пришлось бы прочитать почти весь индекс и отфильтровать — выгоды нет, планировщик обычно выбирает seq scan. Поэтому правило: **поиск по индексу возможен только с первой колонки**; условия по остальным колонкам сужают уже отобранный по первой (и по второй, и т.д.) набор.

#### Как выбирать порядок колонок

1. **Первой** ставь колонку, по которой чаще всего фильтруют равенством (например, `customer_id = ?`).
2. Дальше — колонки по равенству или диапазону, которые часто идут в запросах вместе с первой.
3. В конец — колонку для **ORDER BY** или дополнительного фильтра по диапазону.

Один индекс не может идеально обслужить все запросы; выбирают порядок под **самые частые и тяжёлые** запросы.

#### Таблица: какой запрос использует индекс (a, b, c)?

| Условие в запросе | Индекс (customer_id, status, created_at) используется? | Почему |
|-------------------|--------------------------------------------------------|--------|
| WHERE customer_id = 5 | Да | Есть префикс (a). |
| WHERE customer_id = 5 AND status = 'paid' | Да | Префикс (a, b). |
| WHERE customer_id = 5 AND status = 'paid' ORDER BY created_at | Да | Полный префикс (a, b, c), сортировка совпадает. |
| WHERE customer_id = 5 AND created_at > '2025-01-01' | Да по customer_id | Префикс (a); created_at может использоваться как фильтр при скане. |
| WHERE status = 'paid' | Обычно нет | Нет левого префикса (a); по индексу искать нельзя. |
| WHERE status = 'paid' AND created_at > ... | Обычно нет | То же: без customer_id порядок индекса не помогает. |
| ORDER BY created_at (без WHERE по customer_id/status) | Нет для фильтрации | Сортировка по одной c без a, b по этому индексу не даёт выгоды. |

Запомни: **без первой колонки индекса** составной индекс для поиска по остальным колонкам по сути не пригоден — данные по ним разбросаны по всему индексу.

#### Left-prefix (краткая формулировка)

Индекс `(a, b, c)` эффективно используется, если условие или сортировка включает **левый префикс**: только `a`; или `a` и `b`; или `a`, `b` и `c`. `WHERE b = ?` без `a` — индекс почти не помогает.

#### Covering index и index-only scan

**Covering‑индекс** — это индекс, в котором есть **все** колонки, нужные запросу: и для фильтра/сортировки, и для SELECT. Тогда планировщик может выполнить **index-only scan**: читать только страницы индекса, не обращаясь к таблице (при условии что visibility map позволяет не проверять мёртвые строки). Это сильно снижает I/O. В B-tree по умолчанию в листе хранятся только ключевые колонки + TID; чтобы «покрыть» ещё и другие колонки, их добавляют через **INCLUDE**.

#### INCLUDE — зачем и как

Колонки в **INCLUDE** попадают в листовые страницы индекса, но **не участвуют в порядке** и не используются для поиска по дереву. Они нужны только чтобы **удовлетворить SELECT** без похода в таблицу. Плюс: ключ индекса остаётся коротким, дерево компактнее, но в листе есть данные для index-only scan.

```sql
CREATE INDEX idx_orders_customer_status_include
    ON orders (customer_id, status)
    INCLUDE (created_at, total_amount);
```

Поиск и сортировка — по `(customer_id, status)`. В листе лежат ещё `created_at` и `total_amount`. Запрос `SELECT created_at, total_amount FROM orders WHERE customer_id = ? AND status = 'paid'` может выполниться как index-only scan без чтения таблицы.

**Связь с §200 (B-tree и TID):** в обычном B-tree в листе хранятся только ключ и TID — после поиска по индексу движок идёт в **таблицу** по TID и читает строку (heap fetch). Если в запросе нужны колонки, которых **нет в индексе**, без INCLUDE/covering так и будет. **INCLUDE** добавляет эти колонки **в лист индекса** (но не в ключ дерева). Тогда для запроса, которому нужны только ключевые + INCLUDE колонки, движок может взять всё из листа индекса и **не ходить в таблицу** — это и есть index-only scan. Меньше I/O, быстрее.

**Условие для index-only scan: visibility map.** Чтобы действительно не читать таблицу, движок должен быть уверен, что все нужные строки **видимы** текущей транзакции. Для этого используется **visibility map** (битовая карта по страницам таблицы): если страница помечена как «все кортежи видимы всем», по TID из индекса в таблицу можно не ходить — данные берутся только из индекса. Visibility map обновляется при VACUUM; на только что изменённых страницах бит может быть не установлен, и тогда планировщик всё равно будет делать heap fetch для проверки видимости. Итог: index-only scan даёт выигрыш, когда все нужные колонки в индексе **и** по visibility map страницы таблицы считаются «все видимы» (подробнее — в частях про VACUUM и хранение).

**Повторение в одну строку:** INCLUDE — колонки **только в листе**, для покрытия SELECT; они **не участвуют** в порядке индекса и не используются для WHERE/ORDER BY. Поиск и сортировка — только по ключевым колонкам; INCLUDE нужен, чтобы не ходить в таблицу за этими полями.

### Пошагово/Практика

#### Анализ типичного запроса

Запрос:

```sql
SELECT id, created_at, total_amount
FROM orders
WHERE customer_id = $1
  AND status = 'paid'
ORDER BY created_at DESC
LIMIT 20;
```

Под него можно спроектировать индекс:

```sql
CREATE INDEX idx_orders_customer_status_created_at
    ON orders (customer_id, status, created_at DESC)
    INCLUDE (total_amount);
```

Здесь:

- `customer_id`, `status`, `created_at`:
  - используются для фильтрации и сортировки;
- `total_amount`:
  - просто «подтягивается» из индекса.

План:

- `index scan` или `index-only scan` по составному индексу без лишних random I/O по таблице.

#### Пошагово: как планировщик решает, использовать ли составной индекс

1. Смотрит условие WHERE и ORDER BY запроса.
2. Проверяет: есть ли в условии **префикс** колонок индекса слева направо? Например, для индекса (customer_id, status, created_at) — есть ли в WHERE customer_id (обязательно первый), потом по желанию status, created_at?
3. Если да — оценивает стоимость: чтение индекса + доступ к таблице по TID (или index-only scan, если все колонки в индексе). Сравнивает с seq scan.
4. Если условие **без первой** колонки (например, только status) — по этому индексу нельзя быстро сузить выборку, данные по status разбросаны; такой индекс для данного запроса обычно не выбирается.

**Итог:** планировщик не «угадывает», а проверяет совпадение **порядка колонок** в запросе с порядком в индексе. Без левого префикса индекс не помогает.

---

### Картинка в голове

Составной индекс — это **многоуровневая сортировка**:

- сначала по `a`, внутри по `b`, внутри по `c`.

Это как папки:

- папка `a`;
- внутри папки `a` папки `b`;
- внутри них папки `c`.

Чтобы быстро искать:

- нужно **всегда начинать с верхнего уровня** — с `a`.

---

### Как запомнить

- **Left-prefix**: «индекс работает слева направо».
- Сначала **набросай типичные запросы**, а потом решай, какое `(a, b, c)` под них лучше.

---

### Проверь себя

<details>
<summary>Может ли индекс (a, b, c) эффективно использоваться для запроса по условию WHERE b = ? AND c = ? (без a)?</summary>

Как правило, **нет**:

- порядок индекса — `(a, b, c)`;
- без `a` нельзя быстро сузить область поиска;
- оптимизатор, скорее всего, выберет **seq scan** или другой индекс.

</details>

<details>
<summary>Зачем использовать INCLUDE‑колонки в индексе?</summary>

Чтобы:

- **сделать индекс покрывающим** для конкретных запросов;
- позволить оптимизатору использовать `index-only scan`;
- при этом **не увеличивать ключ сортировки** (что могло бы сделать индекс тяжёлым и менее эффективным).

</details>

<details>
<summary>Почему запрос с ORDER BY b, c без условия по a не может эффективно использовать индекс (a, b, c)?</summary>

Потому что в индексе данные упорядочены **сначала по a**, потом по b, потом по c. Строки с разными значениями a перемешаны по всему индексу. Чтобы получить глобальную сортировку по (b, c), пришлось бы прочитать весь индекс и отсортировать результат заново — выгоды от индекса нет. Индекс полезен для сортировки только когда мы уже ограничили выборку по префиксу (например, по a), и внутри этого префикса порядок по (b, c) уже совпадает с порядком в индексе.

</details>

<details>
<summary>Что такое правило left-prefix своими словами?</summary>

Составной индекс упорядочен **сначала по первой колонке, потом по второй, потом по третьей**. Использовать его эффективно можно только **с начала**: по первой колонке, по первой+второй, по первой+второй+третьей. «Слева направо» — без пропусков. Поиск только по второй или третьей колонке без первой неэффективен — такие строки разбросаны по всему индексу.

</details>

<details>
<summary>Когда возможен index-only scan и что должно быть в индексе?</summary>

Когда **все колонки**, нужные запросу (и для WHERE/ORDER BY, и для SELECT), **есть в индексе** — в ключе или в INCLUDE. Тогда движок может взять данные только из страниц индекса, не обращаясь к таблице. Дополнительно нужно, чтобы visibility map позволяла не проверять видимость в таблице (страницы «все видимы» после VACUUM).

</details>

<details>
<summary>В каком порядке ставить колонки в составном индексе под запрос с WHERE и ORDER BY?</summary>

**Первой** — колонку из WHERE по равенству (самую селективную). **Дальше** — остальные условия WHERE по равенству или диапазону. **В конец** — колонку из ORDER BY, чтобы внутри отфильтрованного набора данные уже были отсортированы. Колонки только для SELECT — в **INCLUDE**, чтобы не раздувать ключ, но дать index-only scan.

</details>

---

### Запомните

- Составные индексы — это инструмент **под запросы**, а не «просто по нескольким колонкам».
- **Порядок колонок** и использование `INCLUDE` — ключевые рычаги оптимизации.
- **Left-prefix:** без первой колонки индекса поиск по остальным по этому индексу неэффективен.

---

### Типичные ошибки

- Делать составные индексы в порядке «как колонки идут в таблице», а не в порядке, в котором реально фильтруют/сортируют.
- Создавать **несколько индексов**, которые могли бы быть одним грамотно спроектированным составным.
- Забывать про `INCLUDE` и вынуждать планы читать таблицу ради пары колонок, не участвующих в предикатах.

---

### Что часто ломают (составные индексы)

**Типичный баг:** создали индекс `(status, created_at)` под запрос «заказы по статусу, отсортированные по дате». Потом появился запрос «последние заказы по дате по всем статусам» — `ORDER BY created_at DESC LIMIT 20` без фильтра по status. Этот индекс **не помогает** для сортировки: в индексе данные упорядочены сначала по status, потом по created_at; глобальной сортировки по одной created_at нет. Решение: либо отдельный индекс по `created_at` для этого запроса, либо явно понимать, что один составной индекс не обслуживает «все запросы подряд». Всегда смотри в плане (EXPLAIN), действительно ли используется нужный индекс.

---

### Итог §204 в трёх пунктах

1. **Порядок колонок = порядок сортировки в индексе.** Индекс `(a, b, c)` полезен для условий по `a`, по `(a, b)`, по `(a, b, c)`; по `b` или `c` без `a` — нет.
2. **INCLUDE** добавляет колонки в лист индекса только для покрытия запроса; по ним не ищут и не сортируют. Нужен для **index-only scan** (не ходить в таблицу).
3. **Проектируй индекс под запрос:** первая колонка — самая селективная в WHERE; в конец — ORDER BY; при необходимости INCLUDE для SELECT.
