## 203. BRIN детали

### Связь с §200: чем BRIN не похож на B-tree

В B-tree для **каждой строки** в индексе хранится запись (ключ + TID). Чем больше строк — тем больше индекс. BRIN **не хранит строки по одной**: он разбивает таблицу на **диапазоны блоков** (например, по 128 страниц) и для каждого диапазона хранит только **summary** (min, max по колонке). Поэтому объём BRIN = число диапазонов × размер summary — в разы меньше B-tree. Оборотная сторона: по BRIN нельзя найти «одну строку по ключу», можно только **отсечь целые диапазоны блоков**, которые заведомо не подходят. Отсюда требование: данные должны быть **упорядочены** по индексируемой колонке на диске, иначе min/max по блоку бесполезны.

---

### В этом разделе главное

- BRIN — это индексы **по диапазонам физических блоков** таблицы (block range), а не по каждой строке.
- Каждый диапазон блоков имеет **summary (min/max и др.)**, по которому можно быстро отсеивать ненужные блоки.
- BRIN эффективен, когда данные **упорядочены по индексируемому столбцу** (или близко к упорядоченности).
- Размер BRIN в разы меньше B-tree, но точность/выгода зависят от **корреляции логического и физического порядка**.

---

### Простыми словами (первый проход)

BRIN можно представить как **содержание архива по коробкам**:

- вместо того чтобы записывать, **в какой коробке лежит каждый отдельный документ**, мы для каждой коробки знаем:
  - минимальную и максимальную дату документов;
  - минимальное и максимальное значение ID и т.п.

Когда мы ищем «документы за март»:

- смотрим **summary по коробкам**;
- отбрасываем все коробки, в которых **диапазон дат вообще не пересекается с мартом**;
- читаем только подходящие коробки.

---

### Простыми словами (второй проход — почему BRIN такой маленький)

B-tree хранит **запись на каждую строку** (ключ + TID). В таблице на 100 млн строк индекс по одной колонке — десятки гигабайт. BRIN **не хранит строки по одной**: он разбивает таблицу на **куски по N страниц** (например, по 128 страниц = один «block range») и для каждого куска хранит только **краткое описание** — например, минимум и максимум значения колонки в этом куске. Записей в BRIN столько, сколько **диапазонов блоков**, а не строк. Поэтому BRIN может быть в **сотни раз меньше** B-tree. Минус: по summary мы не знаем «в какой именно строке» значение — мы знаем только «в этом блоке есть значения от min до max». Поэтому BRIN не заменяет B-tree для точечного поиска «найти строку с id = 123»; он хорош для **диапазонных** запросов по столбцу, по которому данные **физически упорядочены**.

---

### Теория и правила

#### Block range — что это такое по шагам

Таблица в PostgreSQL хранится в **страницах (блоках)** по 8 КБ. У каждой страницы есть **физический порядковый номер** (block number). BRIN не работает с отдельными строками, а работает с **непрерывными диапазонами страниц**. Один такой диапазон называется **block range**. Размер диапазона задаётся параметром **`pages_per_range`** (по умолчанию 128). То есть: страницы 0–127 — первый range, 128–255 — второй, и т.д.

Для **каждого** block range в индексе BRIN хранится одна (или несколько) **summary‑записей** — компактное описание значений индексируемой колонки в этом диапазоне страниц. Классический вариант — **min-max summary**: минимальное и максимальное значение этой колонки среди всех строк в данном диапазоне блоков.

Итог: объём BRIN = (число block range’ов) × (размер одной summary). Число block range’ов = ceil(число страниц таблицы / pages_per_range). При миллиарде строк и 128 страницах на range записей в BRIN — миллионы, но каждая запись крошечная по сравнению с записью B-tree на строку. Поэтому BRIN и получается таким маленьким.

**Числовой пример для закрепления:** таблица 100 млн строк, в среднем 100 строк на страницу → примерно 1 млн страниц. При `pages_per_range = 128` получается около 1_000_000 / 128 ≈ **7800** block range’ов. В BRIN хранится 7800 записей (min, max по колонке для каждого range). Одна такая запись — десятки байт. Итого BRIN — доли мегабайта. B-tree на 100 млн строк по одной колонке — гигабайты. Вот почему BRIN «микроскопический» по сравнению с B-tree.

#### Как по min-max summary отсекают блоки

Запрос: `WHERE created_at BETWEEN '2025-01-01' AND '2025-01-31'`.

Для каждого block range у нас есть [min, max] по `created_at` в этом диапазоне блоков.

- Если **max < '2025-01-01'** или **min > '2025-01-31'** — в этом range **нет ни одной подходящей строки**. Все страницы этого range можно **не читать**.
- Если [min, max] **пересекается** с [2025-01-01, 2025-01-31] — тогда **хотя бы одна** строка в этом range может подойти (а может и не подойти, если значения внутри range «прыгают»). BRIN такой range **не отсекает**: мы должны прочитать все страницы этого range и уже в них отфильтровать по `created_at`.

Чем **уже** диапазон [min, max] в каждом range (т.е. чем сильнее данные в таблице упорядочены по `created_at`), тем больше range’ов BRIN сможет отсечь и тем меньше блоков придётся читать. Если в каждом range min и max почти совпадают с реальным разбросом дат в этих блоках — BRIN очень эффективен. Если в каждом range min = 2020 год, max = 2030 год (данные перемешаны), то почти ни один range отсечь нельзя — BRIN почти бесполезен.

**Числовой пример отсечения блоков (чтобы запомнить):** Таблица логов 1 млн страниц, BRIN с `pages_per_range = 128` → 7800 range’ов. Запрос: логи за январь 2025. Данные на диске упорядочены по `created_at`: в range’ах 1–7000 лежат старые логи (max в каждом < 2025-01-01), в range’ах 7001–7800 — конец 2024 и январь 2025. По BRIN для каждого range смотрим summary: у range’ов 1–7000 **max < 2025-01-01** → все 7000 range’ов **отсекаем**, не читаем. Читаем только range’ы 7001–7800 (800 range’ов × 128 страниц ≈ 100 тыс. страниц вместо 1 млн). Так BRIN сокращает чтение в разы. Если бы данные были перемешаны, в каждом range min был бы 2020 год, max — 2030; отсечь нельзя ни один range — пришлось бы читать всю таблицу. **Итог:** отсечение работает только когда min/max по range **узкие** (данные упорядочены).

#### Упорядоченность данных — почему это критично

Данные **физически упорядочены** по колонке, когда строки на диске лежат примерно в том же порядке, в каком значения этой колонки. Типичные случаи:

- **Логи, события**: вставки только append по времени; на диске сначала старые даты, потом новые. Для `created_at` min-max по каждому range будет узким (например, один день или несколько часов).
- **Партиционированная таблица по дате**: внутри партиции данные тоже часто по дате; BRIN по дате внутри партиции может хорошо работать.
- **Автоинкрементный id и append-only**: если строки вставляются по порядку id, то id коррелирует с номером блока; BRIN по id может отсекать блоки для запросов вида `id BETWEEN ...`.

Данные **перемешаны**, когда в одном и том же блоке лежат строки с самыми разными значениями колонки. Например:

- таблица часто обновляется, строки переезжают;
- или вставки идут в случайном порядке по этому столбцу (например, UUID, случайные id).
Тогда в одном block range окажутся и min = 1, и max = 1000000; summary не отсекает ничего. **Вывод:** BRIN имеет смысл только когда есть **хорошая корреляция** между порядком строк на диске и значением индексируемой колонки.

#### Autosummarize и ручное обновление summary

При вставке новых строк в таблицу **новые страницы** сначала могут не быть «описаны» в BRIN: для них ещё нет summary. Запросы тогда должны будут читать эти страницы «вслепую» или summary для них строится по требованию. Параметр **autosummarize** (включить можно через `ALTER INDEX ... SET (autosummarize = on)`) заставляет периодически пересчитывать summary для новых диапазонов блоков, чтобы BRIN оставался актуальным. Без этого после массовой загрузки данных BRIN может временно не покрывать новые блоки; можно вручную вызывать `brin_summarize_new_values()` для индекса или полагаться на autosummarize.

### Пошагово/Практика

#### BRIN по дате для логов — когда и зачем

Таблица логов: вставки только в конец (append), по `created_at` данные на диске идут по возрастанию. Идеальный сценарий для BRIN.

```sql
CREATE TABLE logs (
    id         bigserial PRIMARY KEY,
    created_at timestamptz NOT NULL,
    message    text
);

CREATE INDEX idx_logs_created_at_brin
    ON logs
    USING brin (created_at)
    WITH (pages_per_range = 128);
```

Типичный запрос — «логи за последние сутки»:

```sql
SELECT *
FROM logs
WHERE created_at >= now() - interval '1 day';
```

**Пошагово, что делает движок:** (1) Читает BRIN — для каждого block range смотрит summary (min, max по created_at). (2) Для каждого range проверяет: если **max < (now() - 1 day)**, весь этот range можно **пропустить** — в нём нет подходящих строк. (3) Для остальных range’ов (где max >= границы) читает **все страницы** этого range с диска и применяет фильтр по created_at к каждой строке. (4) Возвращает отфильтрованные строки. Итог: вместо чтения всей таблицы читаются только «подозрительные» диапазоны блоков; при хорошей упорядоченности по дате таких диапазонов мало. План: по BRIN отсекаются все block range’ы, у которых **max(created_at) < (now() - 1 day)**. Обычно это почти все старые диапазоны; читаются только последние несколько range’ов (где лежат недавние логи). Размер BRIN при этом — доли мегабайта даже на сотни гигабайт таблицы.

Параметр **pages_per_range**: чем он больше (например, 256), тем меньше записей в BRIN и меньше индекс, но каждый range покрывает больше страниц — отсекаем «грубее». Чем меньше (например, 64), тем точнее отсечение, но индекс чуть больше. Для логов по времени часто оставляют 128.

#### Включение autosummarize

Чтобы новые страницы таблицы после вставок быстрее попадали в summary BRIN:

```sql
ALTER INDEX idx_logs_created_at_brin
    SET (autosummarize = on);
```

Тогда фоновые процессы будут периодически строить summary для новых диапазонов блоков. Без этого после большой загрузки запрос по новым данным может временно не использовать BRIN так эффективно, пока summary не построится (при следующем VACUUM или вручную).

---

### Картинка в голове

BRIN — это **карта архива по коробкам**, где на каждой коробке написано:

- «минимальная и максимальная дата документов внутри»;
- «минимальный и максимальный ID» и т.п.

Чтобы найти нужное, ты:

- **не листаешь все документы**;
- сначала выбираешь коробки по надписям (summary), потом листаешь только их.

**Повторение:** BRIN не умеет сказать «строка с id = 123 лежит в блоке X» — он умеет сказать «в блоках 100–227 значения этой колонки от минимум A до максимум B». Поэтому точечный запрос «найти одну строку по значению» BRIN не обслуживает; он отсекает **целые диапазоны блоков** по условию min/max. Для точечных запросов нужен B-tree (или другой индекс по строкам).

---

### Как запомнить

- **BRIN** = **Block Range Index** — индекс по блокам.
- Хорошо работает, когда данные **упорядочены по колонке**, по которой строится BRIN.
- **Один range = много строк** → мало записей в индексе → маленький размер. Но точность только «по блокам», не по строке.

---

### Проверь себя

<details>
<summary>Почему BRIN так сильно выигрывает по размеру по сравнению с B-tree?</summary>

Потому что:

- BRIN хранит **одну summary‑запись на диапазон блоков** (например, на тысячи строк);
- B-tree хранит **запись на каждую строку** (или почти на каждую);
- поэтому BRIN может быть **на порядки меньше** при больших таблицах.

</details>

<details>
<summary>Когда BRIN почти не даёт выигрыша?</summary>

Когда:

- значения в индексируемой колонке **не коррелируют с физическим порядком строк**;
- summary по блокам становятся **слишком широкими** (min/max почти покрывают весь диапазон);
- приходится читать большинство блоков, и BRIN мало что отсекает.

</details>

<details>
<summary>Что такое pages_per_range и как он влияет на размер и точность BRIN?</summary>

**pages_per_range** — число страниц таблицы в одном block range (по умолчанию 128). Чем он **больше** — тем меньше записей в BRIN (меньше индекс), но каждый range покрывает больше страниц → отсекаем **грубее**, можем прочитать лишние блоки. Чем **меньше** — тем точнее отсечение, но индекс чуть больше. Для логов по дате часто оставляют 128.

</details>

<details>
<summary>При запросе WHERE created_at >= $1 как BRIN отсекает блоки?</summary>

Для каждого block range в BRIN хранится summary (min, max по created_at). Если **max** в range **меньше** границы $1 — в этом range **нет ни одной подходящей строки**, весь range пропускаем (не читаем его страницы). Если [min, max] пересекается с условием — range читаем и в нём фильтруем по created_at. Чем упорядоченнее данные по колонке, тем уже min/max в каждом range и тем больше range’ов отсекается.

</details>

<details>
<summary>Зачем включать autosummarize для BRIN-индекса?</summary>

При вставке **новых страниц** таблицы для них может ещё не быть summary в BRIN. **autosummarize = on** заставляет периодически строить summary для новых диапазонов блоков, чтобы индекс оставался актуальным. Без этого после массовой загрузки запросы по новым данным могут не использовать BRIN эффективно, пока summary не построится (вручную или при VACUUM).

</details>

---

### Запомните

- BRIN — это **индекс для очень больших таблиц**, где данные **приблизительно отсортированы** по ключу (время/ID).
- Он микроскопически мал, но требует **понимания структуры данных** и периодического обслуживания summary.

**Ещё раз одной фразой:** BRIN не индексирует строки по одной, а запоминает для каждого куска блоков таблицы минимум и максимум значения колонки — и по этим min/max отсекает целые куски при диапазонном запросе; выгоден только когда порядок строк на диске совпадает с порядком по этой колонке.

---

### Типичные ошибки

- Использовать BRIN на колонке, где значения **случайны (UUID v4)**, и ждать чудес — min/max по блоку будут почти одинаковые у всех блоков, отсечь ничего не получится.
- Не включать `autosummarize` и не запускать `brin_summarize_new_values`, в результате summary для новых блоков не строится и индекс мало помогает по свежим данным.
- Ожидать, что BRIN заменит B-tree для точечных запросов вида `WHERE id = 123` — BRIN отсекает только **диапазоны блоков**, а не находит одну строку по ключу.
- Ставить BRIN на таблицу, которая активно обновляется «вразброс» (UPDATE по разным строкам), — физический порядок быстро перестаёт соответствовать логическому, BRIN деградирует.

---

### Итог §203 в трёх пунктах

1. **BRIN = summary по блокам:** не запись на строку, а одна запись на диапазон страниц (min/max по колонке). Размер — в разы меньше B-tree.
2. **Отсечение:** по min/max решаем, читать блоки range’а или нет. Эффективно только если данные **упорядочены** по колонке на диске (append-only по времени/ID).
3. **Когда использовать:** огромные таблицы, диапазонные запросы, данные по порядку. Точечный поиск и перемешанный порядок — не для BRIN.

---

### BRIN и B-tree: когда что (сводка)

| Критерий | B-tree | BRIN |
|----------|--------|------|
| Размер индекса | Запись на каждую строку → большой | Одна запись на диапазон блоков → очень маленький |
| Точечный запрос (найти одну строку по значению) | Да, быстро | Нет, не умеет |
| Диапазонный запрос (WHERE col BETWEEN ...) | Да, если данные любые | Да, **если** данные упорядочены по col на диске |
| Данные в случайном порядке по колонке | Работает | Почти бесполезен (min/max по блоку слишком широкие) |
| Таблица только append по времени/ID | Работает | Идеально, малый размер и хорошее отсечение |
| Обновления «вразброс» | Работает, но bloat | Деградирует (порядок на диске ломается) |

**Если забыл:** «BRIN = маленький, но только для упорядоченных данных и диапазонов; точечный поиск — только B-tree».

---

### Реальный сценарий (BRIN): когда он спасёт, когда нет

**Удачный сценарий:** таблица логов приложений, 500 млн строк. Вставки только **append** по времени (`INSERT ...` с `created_at = now()`). Запросы: «логи за последний час», «за вчера». Данные на диске **физически упорядочены** по `created_at`. BRIN по `created_at` с `pages_per_range = 128`: индекс несколько мегабайт; запрос «за последний час» читает только последние десятки range’ов из сотен тысяч — **в разы быстрее** seq scan и без гигабайтного B-tree.

**Неудачный сценарий:** та же таблица, но часто делают **UPDATE** по разным строкам (например, проставляют флаг «обработано»). Физический порядок строк на диске **перемешивается** (старые и новые версии в разных блоках). Min/max по каждому range становятся «от начала до конца времени» — отсечь по summary почти ничего нельзя. BRIN перестаёт помогать; запросы снова сканируют почти всю таблицу. **Вывод:** BRIN любит **append-only** или очень редкие обновления; при активных UPDATE «вразброс» не полагайся на BRIN.
