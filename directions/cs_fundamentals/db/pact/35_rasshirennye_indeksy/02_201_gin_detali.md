## 201. GIN детали

### Связь с §200: что мы уже знаем и что меняется

В §200 мы разобрали B-tree: **один ключ** (значение колонки) → одна или несколько записей в листе (ключ + TID). Индекс упорядочен по ключу; поиск по ключу быстрый. В GIN **ключ** — это не «значение целой колонки», а **один элемент** (один тег из массива, один ключ из JSONB). А **значение** индекса — не один TID, а **целый список TID** (posting list). То есть структура другая: «элемент → список строк, где он есть». Поэтому GIN не заменяет B-tree, а решает другую задачу: поиск по **внутренним элементам** поля (массив, JSON).

---

### В этом разделе главное

- GIN‑индекс строит **отображение «значение → список строк (posting list)»**, а для больших списков — **posting tree**.
- **Fast update** (pending list) позволяет **быстро записывать изменения**, откладывая тяжёлую работу по перестроению индекса.
- GIN идеально подходит для **массивов** и **JSONB**: индексация элементов/ключей и операторов `@>`, `?`, `?&`, `?|`.
- Неправильные настройки или отсутствие обслуживания pending list могут приводить к **резким провалам производительности**.

---

### Простыми словами (первый проход)

GIN можно представить как **огромный обратный словарь**:

- для каждого «слова» (ключа, элемента массива, JSON‑пути) он хранит **список страниц/строк, где это слово встречается**.
- если список короткий — это **просто список**;
- если список длинный — он превращается в **отдельное дерево (posting tree)**.

Чтобы не тормозить на каждом INSERT:

- новые записи сначала складываются в **pending list** — что‑то вроде «входящего ящика»;
- позже, когда наберётся достаточно, они **сливаются** в основной индекс пачкой.

---

### Простыми словами (второй проход — чем GIN отличается от B-tree)

В B-tree один ключ — это **одно значение колонки** и одна (или несколько) ссылок на строки. В GIN **один ключ** — это одно «элементное» значение (один тег, один ключ JSON, одно слово), а **значение** индекса — это **целый список строк**, где этот элемент встречается. То есть:

- B-tree: «значение колонки → одна или несколько строк» (часто одна строка на ключ, если ключ уникален).
- GIN: «элемент (тег, ключ JSON и т.д.) → список всех строк, где этот элемент есть».

Поэтому GIN тяжёлый на запись (нужно обновлять списки по многим элементам), но очень удобный для запросов вида «найти все строки, где в массиве есть тег X» или «где в JSONB есть ключ Y». Такие запросы B-tree напрямую не поддерживает.

**Почему GIN тяжёлый на запись (пошагово на одном примере):** вставили строку с `tags = ['db', 'postgres', 'index']`. Для GIN это значит: (1) найти или создать ключ `'db'` и добавить TID в его posting list; (2) то же для `'postgres'`; (3) то же для `'index'`. Одна вставка строки = **три обновления** в индексе (три списка трогаются). Если в JSONB десять ключей — десять обновлений. Плюс при fastupdate каждая такая запись ещё попадает в pending list. Поэтому таблицы с большими массивами/JSONB и частыми INSERT/UPDATE без GIN бывают быстрее на записи, но тогда запросы «по элементу» будут без индекса. GIN — осознанный компромисс: дороже писать, зато быстро искать по элементам.

---

### Теория и правила

#### Posting list — что это и зачем

Для каждого **уникального элемента** (например, тег `'postgres'`, ключ JSON `'user_id'`, слово в полнотексте) GIN хранит список **TID** — идентификаторов строк, в которых этот элемент встречается. Этот список и называется **posting list**.

Пример:

- в таблице 1 млн статей, у каждой поле `tags` — массив тегов;
- тег `'db'` встречается в 50 000 статей;
- в GIN для ключа `'db'` хранится posting list из 50 000 TID’ов (в сжатом виде).

При запросе «все статьи, где есть тег `'db'`»:

1. По индексу находим ключ `'db'`.
2. Читаем его posting list — получаем 50 000 TID’ов.
3. По этим TID’ам обращаемся к таблице и возвращаем строки (или отфильтровываем по другим условиям).

Без GIN пришлось бы сканировать всю таблицу и для каждой строки проверять «есть ли в массиве `'db'`» — это очень дорого.

**Повторение:** в GIN **один ключ** (один элемент — тег, ключ JSON) связан не с одной строкой, а с **целым списком TID**. Поэтому запрос «все строки, где есть тег X» решается так: нашёл ключ X в GIN → прочитал его список TID → по ним пошёл в таблицу. Никакого полного сканирования таблицы.

**Числовой пример для закрепления (posting list и пересечение):** Пусть в таблице статей три строки. Строка 1 (TID = 100): `tags = ['db', 'postgres']`. Строка 2 (TID = 101): `tags = ['db', 'index']`. Строка 3 (TID = 102): `tags = ['postgres', 'index']`. В GIN после вставки получится: ключ `'db'` → posting list [100, 101]; ключ `'postgres'` → [100, 102]; ключ `'index'` → [101, 102]. Запрос `tags @> ARRAY['db', 'postgres']` («есть и db, и postgres»): берём список для `'db'` — [100, 101]; для `'postgres'` — [100, 102]. **Пересечение** = [100]. Значит, подходит только строка с TID 100 — первая статья. Так и работает «содержит оба тега»: пересечение posting list’ов. Запрос `tags && ARRAY['db']` («есть хотя бы db»): достаточно списка для `'db'` — [100, 101]; объединение не нужно, одна клавиша. Результат — строки 100 и 101. Запомни: **@> (содержит все)** → пересечение списков; **&& (есть хотя бы один)** → объединение списков (или один список, если один элемент).

#### Posting tree — когда списка недостаточно

Когда posting list становится **очень большим** (много TID’ов для одного и того же элемента), хранить его одним плоским списком неэффективно: и по размеру, и по скорости доступа. Тогда GIN переводит этот список в структуру **posting tree** — по сути маленькое B-tree‑подобное дерево над TID’ами. Так можно быстрее искать в списке и эффективнее хранить его.

Типичный случай: очень частый тег или очень частый ключ в JSONB (например, `"type": "view"` у миллионов строк). Для такого значения будет posting tree вместо одного длинного списка.

**Итог:** posting list — «просто список TID для элемента»; posting tree — «то же самое, но упаковано в дерево», когда список слишком велик.

#### Fast update и pending list — пошагово

По умолчанию у GIN включён **fastupdate** (в PostgreSQL — да). Смысл:

1. **При INSERT/UPDATE** строки, затрагивающей индексируемое поле (массив, JSONB):
   - новые «элементы» (теги, ключи, пары ключ‑значение) **не сразу** вставляются в основное дерево GIN;
   - они складываются в отдельную структуру — **pending list** (список отложенных записей). Запись в pending list дёшевая: не нужно сразу обновлять много страниц основного индекса и не нужно сразу делать много split’ов.

2. **Когда pending list сливается в основной индекс:**
   - при следующем **VACUUM** (или AUTOVACUUM) по этой таблице;
   - либо когда размер pending list превышает порог (зависит от реализации).
   - Тогда все накопленные записи из pending list **пачкой** вливаются в основное дерево GIN. Это одна тяжёлая операция за раз вместо тысяч мелких.

3. **При чтении** запроса, использующего GIN:
   - движок смотрит **и** в основной индекс, **и** в pending list;
   - объединяет результаты (TID из обоих источников), чтобы не потерять только что вставленные строки.

Поэтому:

- **Плюс:** вставки и обновления по таблице с GIN остаются относительно быстрыми, даже когда в строке много тегов или ключей JSON.
- **Минус:** если pending list долго не сливать (например, VACUUM редко запускается или таблица только пишется и почти не читается), pending list разрастается. Тогда каждый запрос по GIN должен просматривать и основное дерево, и большой pending list → запросы **резко замедляются**. Размер индекса на диске тоже растёт из-за неслитой pending list.

#### Когда pending list сливается (практически)

- **AUTOVACUUM** по таблице (настраивается как для обычного VACUUM: `autovacuum_vacuum_scale_factor`, `autovacuum_vacuum_threshold` и т.д.). Когда autovacuum доходит до таблицы, он может вызвать слияние pending list в GIN.
- **Ручной VACUUM** по таблице: тоже запускает слияние.
- В некоторых версиях/настройках — при достижении размера pending list определённого лимита.

Если у тебя таблица только пишется (например, лог событий) и VACUUM по ней редко срабатывает, GIN по JSONB/массиву может «раздуть» pending list и потом читаться очень медленно. Решение: периодически вызывать VACUUM вручную или настроить autovacuum агрессивнее для этой таблицы.

#### GIN для массивов — что именно индексируется

Для типа **массив** (`int[]`, `text[]`, и т.д.) GIN по умолчанию индексирует **каждый элемент** массива как отдельный ключ. То есть строка с `tags = ['db', 'postgres', 'index']` даёт три ключа в индексе: `'db'`, `'postgres'`, `'index'`, и в posting list каждого из них будет добавлен TID этой строки.

Поддерживаемые операторы (в зависимости от класса оператора, обычно `gin__array_ops` или `array_ops`):

- **`@>`** — левый массив **содержит** правый (все элементы правого есть в левом). Пример: `tags @> ARRAY['db','postgres']` — строки, у которых есть и `db`, и `postgres`.
- **`<@`** — левый массив **содержится** в правом.
- **`&&`** — массивы **пересекаются** (есть хотя бы один общий элемент). Пример: `tags && ARRAY['db']` — строки, где есть тег `db`.

Для этих операторов планировщик может использовать GIN‑индекс и не сканировать всю таблицу.

#### GIN для JSONB — что именно индексируется

Для **JSONB** GIN может индексировать по-разному в зависимости от класса оператора:

- **`jsonb_ops`** (по умолчанию): индексируются **каждый ключ и каждое значение** в JSON. То есть для `{"type": "signup", "user_id": 123}` в индекс попадут и ключи (`type`, `user_id`), и значения (`signup`, `123`). Подходит для запросов «есть ли ключ», «есть ли такое значение», «содержит ли документ этот поддокумент».
- **`jsonb_path_ops`**: индексируются только **пары (путь → значение)** в более компактном виде. Индекс меньше, но запросы должны выражаться через `@>` (содержит) по структуре; запрос «есть ли ключ» через `?` с `jsonb_path_ops` не поддерживается.

**Когда какой класс оператора брать:** **jsonb_ops** — по умолчанию; поддерживает и `@>` (содержит поддерево), и `?` (есть ключ), и `?&`, `?|`; индекс получается больше, зато гибкость запросов выше. **jsonb_path_ops** — индекс **компактнее**, быстрее строить и часто быстрее искать по `@>`; но запросы только через «содержит» по структуре (`payload @> '{"type":"signup"}'`). Если тебе нужен поиск «есть ли ключ» (`payload ? 'user_id'`) или «есть ли один из ключей» (`?|`), используй **jsonb_ops**. Если почти все запросы — «документ содержит такой-то поддокумент» и хочется меньший индекс — **jsonb_path_ops**. Указать класс можно при создании индекса: `USING gin (payload jsonb_path_ops)`.

Типовые операторы для JSONB с GIN:

- **`@>`** — левый JSONB **содержит** правый (поддерево). Например, `payload @> '{"type":"signup"}'`.
- **`?`** — ключ **существует**. Например, `payload ? 'user_id'`.
- **`?&`** — существуют **все** перечисленные ключи. Например, `payload ?& array['type','user_id']`.
- **`?|`** — существует **хотя бы один** из ключей. Например, `payload ?| array['type','event']`.

### Пошагово/Практика

#### GIN для массивов — полный пример с пояснениями

```sql
CREATE TABLE articles (
    id      bigserial PRIMARY KEY,
    title   text,
    tags    text[]
);

CREATE INDEX idx_articles_tags_gin
    ON articles
    USING gin (tags);
```

Что при этом происходит в индексе (упрощённо):

- при вставке строки `(1, 'Postgres tips', ARRAY['db', 'postgres'])` в GIN добавляются два «ключа»: `'db'` и `'postgres'`, и в posting list каждого ключа попадает TID этой строки;
- при вставке строки `(2, 'Index design', ARRAY['db', 'index'])` в posting list ключа `'db'` добавляется ещё один TID, для `'index'` создаётся новый ключ с одним TID.

Запрос:

```sql
-- Найти статьи, у которых есть теги 'db' И 'postgres'
SELECT *
FROM articles
WHERE tags @> ARRAY['db', 'postgres'];
```

План: по индексу ищутся ключи `'db'` и `'postgres'`, берутся их posting list’ы, находится **пересечение** TID’ов (строки, где есть оба тега), затем по этим TID читаются строки из таблицы. Без GIN пришлось бы полное сканирование таблицы и проверка массива в каждой строке.

Ещё пример — «хотя бы один из тегов»:

```sql
SELECT *
FROM articles
WHERE tags && ARRAY['db', 'mysql'];
```

Индекс по `&&` используется: ищутся ключи `'db'` и `'mysql'`, объединение (union) их posting list’ов даёт все TID строк, где встречается хотя бы один из этих тегов.

**Пошагово: что происходит внутри при запросе `tags @> ARRAY['db', 'postgres']`**

1. Планировщик выбирает доступ по GIN‑индексу по `tags`.
2. Движок ищет в GIN ключ `'db'` и получает его posting list (список TID).
3. Движок ищет в GIN ключ `'postgres'` и получает его posting list.
4. Вычисляется **пересечение** двух списков TID: остаются только те TID, которые есть в обоих списках. Это и есть строки, где есть и тег `db`, и тег `postgres`.
5. По полученным TID читаются строки из таблицы (и при необходимости проверяются другие условия, если были в WHERE).

Запомни: **`@>` (содержит)** для массивов = «все перечисленные элементы есть» → в GIN делаем **пересечение** posting list’ов. **`&&` (пересекаются)** = «хотя бы один общий элемент» → в GIN делаем **объединение** posting list’ов.

#### GIN для JSONB — примеры под разные операторы

```sql
CREATE TABLE events (
    id      bigserial PRIMARY KEY,
    payload jsonb
);

CREATE INDEX idx_events_payload_gin
    ON events
    USING gin (payload);
```

Запрос «документ содержит поддерево» (оператор `@>`):

```sql
SELECT *
FROM events
WHERE payload @> '{"type": "signup"}';
```

Индекс используется: в GIN ищется представление для `{"type": "signup"}`, по соответствующему списку TID возвращаются строки.

Запрос «есть ли ключ» (оператор `?`):

```sql
SELECT *
FROM events
WHERE payload ? 'user_id';
```

С классом оператора `jsonb_ops` индекс по ключам поддерживается; с `jsonb_path_ops` оператор `?` для «есть ключ» не поддерживается — смотри документацию по своему классу оператора.

Запрос «все перечисленные ключи есть» (оператор `?&`):

```sql
SELECT *
FROM events
WHERE payload ?& array['type', 'user_id'];
```

Индекс используется: по сути пересечение posting list’ов для ключей `type` и `user_id`.

#### Настройка fastupdate и когда его отключать

Включён по умолчанию. Явно включить:

```sql
ALTER INDEX idx_events_payload_gin
    SET (fastupdate = on);
```

Отключить (осторожно):

```sql
ALTER INDEX idx_events_payload_gin
    SET (fastupdate = off);
```

После отключения **новые** записи будут сразу попадать в основное дерево GIN, без pending list. Это увеличивает стоимость каждой вставки/обновления, но запросы по индексу не должны просматривать pending list. Имеет смысл только если у тебя: очень много запросов по GIN и мало записи, и ты уверен, что pending list стал узким местом. Обычно лучше оставить fastupdate и наладить регулярный VACUUM.

#### Как проверить, что запрос использует GIN

Выполни `EXPLAIN (ANALYZE)` по запросу:

```sql
EXPLAIN (ANALYZE)
SELECT * FROM events WHERE payload @> '{"type": "signup"}';
```

В плане должно быть что-то вроде `Bitmap Index Scan on idx_events_payload_gin` или `Index Scan using idx_events_payload_gin`. Если видишь `Seq Scan` по таблице при большом объёме данных — возможно, оптимизатор посчитал, что индекс не выгоден (например, почти все строки подходят) или условие не подходит под класс оператора.

### Картинка в голове

GIN — это **обратный индекс словаря**:

- у тебя есть миллион книг (строк), и ты хочешь знать:
  - «в каких книгах встречается слово `postgres`?»
- GIN хранит:
  - `postgres` → [книга 1, книга 7, книга 123, …];
  - `index` → [книга 3, книга 4, …] и т.д.

`pending list` — это **коробка у входа**, куда временно складывают новые книги, пока библиотекарь не разложит их по полкам.

---

### Как запомнить

- **GIN** = «обратный индекс, значение → список строк».
- **posting list** = «просто список строк для значения».
- **pending list** = «коробка с неразобранными изменениями».
- **@> для массивов** = «содержит все перечисленные» → в GIN **пересечение** списков. **&&** = «хотя бы один общий» → **объединение** списков.

### Проверь себя

<details>
<summary>Чем GIN принципиально отличается от B-tree в том, как он хранит информацию?</summary>

B-tree хранит:

- отсортированные по ключу записи «ключ → ссылка на строку».

GIN хранит:

- для каждого **значения** (элемента, ключа) — **список строк (posting list)**, где это значение встречается;
- для очень частых значений — **отдельное дерево (posting tree)**.

</details>

<details>
<summary>Зачем нужен fastupdate в GIN‑индексе?</summary>

Чтобы:

- **ускорить вставки и обновления**:
  - изменения сначала попадают в pending list;
  - тяжёлая работа по перестроению структуры откладывается и выполняется пачками.

</details>

<details>
<summary>Почему GIN особенно полезен для JSONB и массивов?</summary>

Потому что:

- эти типы данных содержат **много возможных значений внутри одного поля**;
- GIN может:
  - индексировать **каждый элемент массива** или **каждый ключ/пару ключ‑значение JSONB**;
  - эффективно отвечать на запросы с операторами `@>`, `?`, `?&`, `?|` и т.п.

</details>

<details>
<summary>Что такое posting tree и когда он используется вместо posting list?</summary>

**Posting tree** — это дерево (похожее на B-tree) поверх списка TID. Когда для одного элемента (тега, ключа JSON) список TID становится **очень большим**, хранить его одним плоским списком неэффективно. GIN переводит такой список в posting tree: так быстрее искать и компактнее хранить. Типично для очень частых значений (например, тег «db» у миллионов статей).

</details>

<details>
<summary>Для запроса tags @> ARRAY['a','b'] что делает движок с posting list'ами?</summary>

Находит в GIN ключ 'a' и читает его posting list (список TID). Находит ключ 'b' и читает его posting list. Вычисляет **пересечение** двух списков TID — остаются только те TID, которые есть в обоих. Это и есть строки, где есть и тег 'a', и тег 'b'. По полученным TID читает строки из таблицы.

</details>

<details>
<summary>Когда pending list сливается в основной индекс GIN?</summary>

При **VACUUM** (ручном или AUTOVACUUM) по таблице; в некоторых версиях — при достижении порога размера pending list. После слияния накопленные записи пачкой попадают в основное дерево индекса. Если долго не запускать VACUUM, pending list растёт и запросы по GIN замедляются (движок смотрит и основной индекс, и pending list).

</details>

---

### Запомните

- GIN — это **индекс для «многозначных» полей**: массивы, JSONB, полнотекстовый поиск и т.п.
- Ключ к производительности — **понимать posting list/tree и fastupdate** и следить за состоянием pending list.

**Ещё раз одной фразой:** GIN отвечает на вопрос «в каких строках встречается этот элемент?» — по одному элементу хранит список TID; для массивов и JSONB это даёт быстрый поиск по @>, ?, && без полного сканирования таблицы.

---

### Когда выбрать GIN, когда B-tree (шпаргалка)

| Ситуация | Что выбрать | Почему |
|----------|-------------|--------|
| Одно значение на строку (status, type, category) | **B-tree** | Меньше индекс, быстрее равенство и диапазоны. GIN здесь избыточен. |
| Массив в строке (теги, список id) | **GIN** | Нужен поиск «есть элемент», «содержит все», «пересекается» — B-tree так не умеет. |
| JSONB: поиск по ключам, «содержит», несколько ключей | **GIN** | Много ключей/значений в одном документе — обратный индекс по элементам. |
| JSONB: одно поле вытащили в колонку (например, payload->>'type') | **B-tree** по выражению или по колонке | Одно значение на строку — достаточно B-tree. |

**Если забыл:** «много значений в одной строке (массив, JSON) → GIN; одно значение на строку → B-tree».

---

### Типичные ошибки

- Создавать GIN‑индекс на **огромном JSONB без понимания паттернов запросов**, а потом удивляться размеру и скорости обновлений.
- Игнорировать размер/состояние pending list и не понимать, почему вдруг GIN‑запросы стали **резко медленнее**.
- Использовать GIN там, где **достаточно обычного B-tree по одному полю** (например, поиск по `status` или `type` без сложных условий).

---

### Что будет, если…

- … GIN‑индекс с fastupdate долго не обслуживать (редкий VACUUM, мало активности)?
  - pending list может вырасти;
  - запросы начнут тратить много времени на просмотр pending list;
  - индекс может заметно увеличиться в размере.
- … поставить GIN на колонку с **одним значением на строку** (например, `status` с значениями `'new'`, `'paid'`, `'shipped'`)?
  - GIN будет работать, но для таких «одно значение на строку» обычно **достаточно B-tree**: он меньше, проще и быстрее для равенства и диапазонов. GIN имеет смысл, когда в **одной строке много значений** (массив тегов, ключи JSON).
- … не указать в запросе оператор, поддерживаемый GIN (например, искать по JSONB через `payload->>'type' = 'signup'`)?
  - выражение `->>` даёт текст, и это уже не оператор GIN; индекс по `payload` может не использоваться. Нужно переписать, например, как `payload @> '{"type":"signup"}'`, чтобы задействовать GIN.

---

### Часто задают вопрос (GIN)

**Почему для обычной колонки (например, `status` с значениями 'new', 'paid', 'shipped') не ставят GIN?** Потому что в такой колонке **одно значение на строку**. GIN рассчитан на случай, когда в **одной строке много элементов** (теги в массиве, ключи в JSONB). Для «найти все строки, где status = 'paid'» достаточно B-tree: он меньше, проще и быстрее на запись; порядок по одному значению B-tree как раз и даёт. GIN имеет смысл, когда запрос звучит как «найти строки, где **в** поле (массиве/JSONB) **есть** такой-то элемент» — там один ключ GIN соответствует списку многих TID. **Кратко:** одно значение на строку → B-tree; много элементов внутри одного поля → GIN.

---

### Итог §201 в трёх пунктах

1. **GIN = «элемент → список TID»:** каждый элемент массива/JSONB — отдельный ключ; значение — posting list (или posting tree при большом списке).
2. **Fastupdate (pending list):** новые записи сначала в буфер → быстрая запись; слияние в основной индекс при VACUUM. Если не сливать — запросы тормозят.
3. **Когда использовать:** массивы, JSONB, полнотекст — запросы по элементам/ключам (@>, ?, &&). Одно значение на строку — лучше B-tree.
