## 205. Частичные индексы

### В этом разделе главное

- Частичный индекс индексирует **только те строки, которые удовлетворяют условию `WHERE`**.
- Это уменьшает размер и стоимость обслуживания индекса, если большая часть таблицы **не участвует в критичных запросах**.
- Можно создавать **уникальные частичные индексы**, обеспечивающие уникальность только среди отфильтрованных строк.
- Частичный индекс работает, только если **условие в запросе логически совпадает с условием в индексе** (или его включает).

---

### Простыми словами (первый проход)

Частичный индекс — это как **отдельный, маленький справочник только по активным клиентам**:

- в таблице есть и активные, и архивные записи;
- для повседневной работы нужны в основном **активные**;
- мы делаем индекс **только по активным**, экономя место и ускоряя работу.

---

### Простыми словами (второй проход — когда это выгодно)

Обычный индекс содержит запись **на каждую строку** таблицы. Если в таблице 100 млн строк, а реально в запросах участвуют только 1 млн «активных» — мы зря храним в индексе 99 млн записей: индекс большой, обновления дорогие, кеш забит. Частичный индекс хранит записи **только для строк, прошедших WHERE**. Размер индекса и стоимость INSERT/UPDATE/DELETE по нему падают. Планировщик будет использовать такой индекс **только если запрос сам ограничивает выборку тем же условием** (или более строгим), иначе результат был бы неполным.

---

### Теория и правила

#### WHERE‑условие в индексе

Синтаксис:

```sql
CREATE INDEX idx_name
    ON table_name (column)
    WHERE some_condition;
```

Индекс включает **только строки, где `some_condition` = true**. При вставке/обновлении запись в индекс добавляется только если строка удовлетворяет `some_condition`; иначе индекс не трогают.

Чтобы планировщик **использовал** этот индекс, запрос должен **логически подразумевать** то же условие. Например, индекс с `WHERE is_active` будет рассмотрен для запроса `WHERE is_active AND email = $1`, потому что результат запроса — подмножество строк с `is_active`. Для запроса `WHERE email = $1` без `is_active` планировщик не может использовать частичный индекс: в индексе нет неактивных строк, и мы могли бы пропустить нужную строку (например, искали email неактивного пользователя). Поэтому частичный индекс либо игнорируется, либо запрос переписывают, добавив условие из индекса.

**Когда частичный индекс точно не используется:** если в запросе нет условия, эквивалентного или более строгого, чем WHERE индекса. Пример: индекс `WHERE is_active`, запрос `SELECT * FROM users WHERE email = 'x'` — в результате могут понадобиться и неактивные пользователи, а их в индексе нет → планировщик не возьмёт этот индекс для такого запроса.

**Почему так (пошагово):** Частичный индекс **физически не содержит** строк с `is_active = false`. Если бы планировщик использовал этот индекс для запроса `WHERE email = 'x'`, он нашёл бы в индексе только активного пользователя с таким email (если он есть). Но неактивного пользователя с тем же email в индексе **нет** — мы его никогда не добавили. Значит, ответ запроса был бы **неполным**: мы могли бы вернуть 0 строк, хотя в таблице есть неактивная строка с email = 'x'. Поэтому планировщик для запроса без условия `is_active` этот индекс не использует и идёт в таблицу (seq scan или полный индекс по email).

**Числовой образ:** в таблице 100 млн строк, из них 1 млн активных (`is_active = true`). Полный индекс по email содержал бы 100 млн записей. Частичный индекс с `WHERE is_active` — только 1 млн записей. Размер индекса и стоимость обновлений падают примерно в 100 раз для этого индекса.

**До и после частичного индекса:** Без частичного индекса — полный индекс по email на все 100 млн строк: большой размер, каждое INSERT/UPDATE по email трогает этот индекс. С частичным индексом `WHERE is_active` — в индексе только 1 млн записей (активные): индекс в разы меньше, обновления дешевле (в индекс попадают только строки с is_active = true), поиск «активный пользователь по email» быстрее за счёт меньшего объёма. Условие: в запросе всегда должно быть условие `is_active` (или эквивалентное), иначе этот индекс не используется.

#### Уникальные частичные индексы

```sql
CREATE UNIQUE INDEX idx_users_email_active_unique
    ON users (email)
    WHERE is_active;
```

Гарантия:

- среди **активных** пользователей `email` уникален;
- среди неактивных может быть дубликат — это **разрешено**.

Полезно, когда:

- нужно сохранять историю/архив;
- но уникальность требуется только для «живого» множества.

---

### Пошагово/Практика

#### Индекс только по активным записям

```sql
CREATE TABLE users (
    id        bigserial PRIMARY KEY,
    email     text NOT NULL,
    is_active boolean NOT NULL DEFAULT true
);

CREATE INDEX idx_users_active_email
    ON users (email)
    WHERE is_active;

-- Типичный запрос
SELECT *
FROM users
WHERE is_active
  AND email = $1;
```

Индекс:

- маленький (содержит только активных);
- быстрый для типичных запросов.

#### Уникальность только среди «не удалённых»

```sql
CREATE TABLE products (
    id          bigserial PRIMARY KEY,
    sku         text NOT NULL,
    deleted_at  timestamptz
);

CREATE UNIQUE INDEX idx_products_sku_not_deleted
    ON products (sku)
    WHERE deleted_at IS NULL;
```

Гарантия:

- SKU уникален среди **актуальных товаров**;
- при архивировании (установка `deleted_at`) можно **создать новый товар с тем же SKU**.

#### Пошагово: когда планировщик выберет частичный индекс

1. В запросе есть условие в WHERE. Планировщик проверяет: **подразумевает ли запрос то же подмножество строк**, что и условие частичного индекса?
2. Например, индекс с `WHERE is_active`. Запрос `WHERE is_active AND email = $1` — да, мы явно ограничиваемся активными, результат будет только из этого подмножества → индекс пригоден.
3. Запрос `WHERE email = $1` без is_active — мы **не** ограничиваемся активными; в индексе нет неактивных строк, значит мы могли бы пропустить искомую строку (если она неактивна) → частичный индекс **нельзя** использовать для такого запроса.
4. Итог: планировщик выбирает частичный индекс только когда условие запроса **эквивалентно или строже** условия индекса (то есть выборка запроса целиком лежит в проиндексированном подмножестве).

**Если забыл:** «частичный индекс = индекс на часть строк; запрос должен выбирать ту же часть (тот же WHERE), иначе индекс не подходит».

---

### Картинка в голове

Частичный индекс — это **избранный список**:

- вместо полного списка всех клиентов;
- есть «список VIP» — только те, кто **активен и важен**;
- он маленький и быстрый, и его легче держать под рукой.

---

### Как запомнить

- Частичный индекс = «**индекс с WHERE**».
- Работает **только там**, где запросы **тоже используют этот WHERE (или более сильный)».

---

### Проверь себя

<details>
<summary>Почему частичный индекс может быть намного меньше полного?</summary>

Потому что:

- он индексирует **только подмножество строк** (например, `is_active = true`, `deleted_at IS NULL`);
- все остальные строки **вообще не попадают в индекс**;
- меньше строк → меньше страниц индекса → меньше bloat и быстрее обслуживание.

</details>

<details>
<summary>Будет ли использоваться индекс с WHERE is_active, если запрос не содержит условия is_active?</summary>

Скорее всего, **нет**:

- оптимизатор не может быть уверен, что ему нужны **только активные** строки;
- ему придётся читать всю таблицу или использовать другой индекс;
- частичный индекс полезен, когда запрос **явно ограничивает** выборку тем же условием.

</details>

<details>
<summary>Зачем делать уникальный частичный индекс (UNIQUE ... WHERE ...), а не обычный UNIQUE по колонке?</summary>

Чтобы разрешить **дубликаты среди отфильтрованных «неважных» строк**, но запретить их среди «важных». Пример: уникальность email только среди активных пользователей (`WHERE is_active`). Неактивные (удалённые) пользователи могут иметь тот же email — для истории. Обычный UNIQUE не даёт двух строк с одинаковым email вообще.

</details>

<details>
<summary>Какое условие в запросе должно быть, чтобы планировщик выбрал частичный индекс?</summary>

Условие в запросе должно **логически совпадать или быть строже**, чем WHERE индекса. То есть выборка запроса должна целиком лежать в проиндексированном подмножестве. Пример: индекс `WHERE is_active` — в запросе нужно `is_active` (или `is_active AND ...`). Иначе в индексе нет части строк и результат мог бы быть неполным, планировщик такой индекс не использует.

</details>

<details>
<summary>Можно ли иметь два частичных индекса на одну колонку с разными WHERE?</summary>

**Да.** Например: один с `WHERE is_active` для поиска активных по email, другой с `WHERE NOT is_active` для архивных. Планировщик выберет индекс, условие которого совпадает с запросом. Минус — два индекса поддерживать (размер, обновления при INSERT/UPDATE/DELETE).

</details>

<details>
<summary>Индекс WHERE is_active по email. Запрос WHERE email = $1 без is_active. Почему индекс не используется?</summary>

Потому что в индексе **нет неактивных** строк. Если бы планировщик использовал этот индекс, он нашёл бы только активного пользователя с таким email (если есть). Неактивный с тем же email в индексе отсутствует — ответ был бы **неполным** (могли бы вернуть 0 строк при наличии неактивной строки). Поэтому для запроса без is_active этот индекс не подходит, идёт seq scan или полный индекс по email.

</details>

---

### Запомните

- Частичные индексы — мощный способ **уменьшить размер и стоимость индексации**, сфокусировавшись на важном подмножестве данных.
- Они требуют **дисциплины в запросах**: условие в запросе должно совпадать (или включать) условие индекса, иначе индекс не используется.

---

### Типичные ошибки

- Добавлять условие в частичный индекс, но **забывать добавить его в запросы** → индекс почти не используется.
- Использовать сложные условия, которые оптимизатор не может легко сопоставить.
- Переусердствовать с частичностью и создавать десятки частичных индексов с похожими условиями.

---

### Итог §205 в трёх пунктах

1. **Частичный индекс = индекс только по подмножеству строк** (WHERE при создании). Меньше размер, быстрее обновление — но запрос должен «попадать» в это подмножество.
2. **Планировщик выбирает частичный индекс**, только если условие запроса **совместимо** с условием индекса (равно или уже). Иначе в индексе просто нет нужных строк.
3. **Уникальный частичный** — уникальность только среди отфильтрованных строк (например, email уникален среди активных пользователей; среди удалённых дубликаты допустимы).

---

### Часто путают (частичные индексы)

| Путают | Пояснение |
|--------|-----------|
| **Частичный индекс и фильтр в запросе** | Частичный индекс **физически** содержит только часть строк (те, что прошли WHERE индекса). Запрос с другим фильтром (без условия индекса) не может использовать этот индекс — в индексе нет остальных строк. Не «забыли добавить WHERE», а «индекс по определению не покрывает все строки». |
| **Уникальный частичный и обычный UNIQUE** | Обычный UNIQUE — уникальность по **всей** таблице. Уникальный частичный — уникальность только среди строк, удовлетворяющих WHERE индекса. В остальных строках дубликаты разрешены (например, один email у активного и у удалённого пользователя). |

**Часто задают вопрос:** можно ли создать несколько частичных индексов на одну и ту же колонку с разными WHERE? Да. Например: один индекс с `WHERE is_active` для быстрого поиска активных по email, другой с `WHERE NOT is_active` для отчётов по архивным. Планировщик выберет тот индекс, условие которого совпадает с запросом. Минус — больше индексов поддерживать (размер, обновления).
