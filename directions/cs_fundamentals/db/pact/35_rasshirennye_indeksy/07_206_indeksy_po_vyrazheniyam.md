## 206. Индексы по выражениям

### В этом разделе главное

- Индекс по выражению хранит **значения функции/выражения**, а не «сырую» колонку (`LOWER(email)`, `date(created_at)` и т.п.).
- Он нужен, когда запросы **всегда используют функцию над колонкой** — без него индекс по колонке не помогает.
- Функции в выражении должны быть **immutable** (или по крайней мере не зависящими от внешнего состояния/времени).
- Это мощный инструмент для оптимизации кейсов вроде «поиск email без учёта регистра», «группировка по дате без времени».

---

### Простыми словами (первый проход)

Индекс по выражению — это как **заранее подготовленный список «email в нижнем регистре»**:

- вместо того чтобы каждый раз в запросе считать `LOWER(email)` по всем строкам;
- мы заранее считаем `LOWER(email)` и кладём его в индекс.

Важно: индекс хранит **результат выражения**, а не саму колонку. Поэтому планировщик может использовать этот индекс **только когда в запросе стоит то же выражение**. Запрос `WHERE email = 'test@mail.ru'` не совпадает с индексом по `(LOWER(email))`: в индексе порядок по нижнему регистру, а в запросе сравнивается «сырой» email. Запрос `WHERE LOWER(email) = 'test@mail.ru'` (или `= LOWER($1)`) — совпадает: мы ищем по значению выражения, и в индексе как раз хранятся эти значения.

Тогда запрос `WHERE LOWER(email) = LOWER($1)` может **использовать этот индекс**, а не считать функцию по всей таблице.

---

### Простыми словами (второй проход — почему индекс по колонке не срабатывает)

Индекс по колонке `email` хранит **исходные** значения: `User@Mail.ru`, `admin@site.com` и т.д. Запрос же ищет по **выражению** `LOWER(email)`. Для планировщика это другое значение: он не может по индексу, упорядоченному по `email`, эффективно найти «все строки, где LOWER(email) = 'user@mail.ru'», потому что порядок по `email` и порядок по `LOWER(email)` не совпадают (регистр меняет сортировку). Поэтому без индекса по **тому же выражению**, что и в запросе, будет полное сканирование таблицы с вычислением LOWER для каждой строки. Индекс по выражению `(LOWER(email))` хранит уже нормализованные значения; условие в запросе должно **точно совпадать** с выражением в индексе (то же выражение, та же функция).

**До и после индекса по выражению (на примере LOWER(email)):** Без индекса: планировщик выбирает **Seq Scan** по таблице; для каждой строки вычисляется `LOWER(email)` и сравнивается с искомым значением — при миллионах строк это очень дорого. С индексом по `(LOWER(email))`: планировщик может выбрать **Index Scan** по этому индексу; поиск по ключу в B-tree по уже вычисленному значению — считаются только нужные страницы индекса и затем строки по TID. Разница в скорости на больших таблицах — в разы или на порядки. Условие: в запросе должно быть **то же выражение** `LOWER(email)`, иначе индекс не используется.

---

### Теория и правила

#### Синтаксис

```sql
CREATE INDEX idx_users_lower_email
    ON users ((LOWER(email)));
```

Обрати внимание: выражение в индексе берётся в **двойные скобки** `(( ... ))`, чтобы отличить выражение от имени колонки.

Важно:

- выражение в индексе и в запросе должны **логически совпадать** (то же выражение, чтобы планировщик мог сопоставить условие с индексам);
- функции в выражении должны быть **immutable**.

#### Почему нужна immutable

**Immutable** в PostgreSQL значит: для одних и тех же аргументов функция **всегда возвращает один и тот же результат**, независимо от времени, сессии, других таблиц. Примеры immutable: `LOWER()`, `UPPER()`, арифметика, `length()`, многие преобразования типов. Не immutable: `now()`, `random()`, `current_user`, чтение из другой таблицы.

Если в индексе использовать не-immutable функцию (например, `now()`), то при каждом вызове значение может меняться. Индекс строится один раз; потом в нём хранятся «старые» результаты функции. При следующем запросе функция может вернуть другое значение — индекс станет **несогласованным** с ожиданиями запроса, и результат поиска будет неверным. Поэтому PostgreSQL **требует**, чтобы выражения в индексе использовали только immutable-функции. Для своих функций можно пометить их как `IMMUTABLE` при создании, но делать это нужно только если ты уверен в семантике (иначе возможна порча индекса).

**Почему выражение в запросе должно точно совпадать с выражением в индексе:** планировщик смотрит на условие в WHERE и проверяет: «есть ли индекс по такому же выражению?» Если в индексе записано `(LOWER(email))`, а в запросе написано `UPPER(email) = UPPER($1)` — это **другое** выражение, индекс по LOWER не подходит. Если в запросе `LOWER(email) = 'user@mail.ru'` — выражение совпадает с индексом, индекс может быть использован. Поэтому при смене формулировки запроса (например, с LOWER на INITCAP) нужно либо добавить новый индекс по новому выражению, либо вернуть в запросе то выражение, под которое уже есть индекс.

**Что будет, если пометить функцию как IMMUTABLE, хотя она таковой не является (например, зависит от времени):** при пересчёте значений в индексе или при поиске возможны неверные результаты или падения. Индекс может содержать «устаревшие» значения функции. Не помечай функцию как IMMUTABLE «для удобства» — только если она действительно всегда возвращает один результат для одних и тех же аргументов.

#### Типичные паттерны

- Поиск по email без учёта регистра:

```sql
CREATE INDEX idx_users_lower_email
    ON users ((LOWER(email)));
```

- Группировка по дате (без времени):

```sql
CREATE INDEX idx_events_date
    ON events ((date(created_at)));
```

- Индекс по «нормализованному» номеру телефона:

```sql
CREATE INDEX idx_users_norm_phone
    ON users ((normalize_phone(phone)));
```

где `normalize_phone` — ваша immutable‑функция.

#### Выражение в запросе и в индексе: совпадение (таблица)

| Выражение в индексе | Выражение в запросе | Индекс используется? |
|---------------------|---------------------|----------------------|
| (LOWER(email)) | LOWER(email) = $1 | Да |
| (LOWER(email)) | LOWER(email) = LOWER($1) | Да |
| (LOWER(email)) | email = $1 | Нет (другое выражение) |
| (LOWER(email)) | UPPER(email) = $1 | Нет |
| (LOWER(email)) | TRIM(email) = $1 | Нет |
| (date(created_at)) | date(created_at) = $1 | Да |
| (date(created_at)) | created_at::date = $1 | Зависит от того, приведёт ли планировщик к тому же выражению (часто да для ::date) |

Правило: условие в WHERE должно **логически совпадать** с выражением в индексе (та же функция, те же аргументы). Иначе планировщик не сопоставит условие с индексом.

**Реальный сценарий:** в приложении логин по email без учёта регистра: `WHERE LOWER(email) = LOWER($1)`. Индекс по `email` не используется — в индексе порядок по исходному email, а не по LOWER(email). Создаёшь индекс `CREATE INDEX ... ON users ((LOWER(email)))`. Теперь в индексе хранятся уже нормализованные значения; запрос с тем же выражением LOWER(email) использует индекс. Важно: в коде **не менять** на UPPER или TRIM без добавления соответствующего индекса или обратной замены в запросе.

### Пошагово/Практика

#### Поиск email без учёта регистра

```sql
CREATE TABLE users (
    id    bigserial PRIMARY KEY,
    email text NOT NULL
);

CREATE INDEX idx_users_lower_email
    ON users ((LOWER(email)));

SELECT *
FROM users
WHERE LOWER(email) = LOWER($1);
```

Оптимизатор:

- видит выражение `LOWER(email)` в запросе;
- сопоставляет его с выражением в индексе;
- использует индекс вместо seq scan.

#### Фильтрация по дате без времени

```sql
CREATE TABLE events (
    id         bigserial PRIMARY KEY,
    created_at timestamptz NOT NULL
);

CREATE INDEX idx_events_date
    ON events ((date(created_at)));

SELECT *
FROM events
WHERE date(created_at) = date($1);
```

Здесь:

- выражение `date(created_at)` соответствует выражению в индексе;
- запросы по конкретной дате используют индекс.

### Картинка в голове

Индекс по выражению — это **предвычисленный столбец**, который мы не храним явно в таблице, но храним в индексе:

- как если бы мы завели поле `normalized_email`;
- но вместо явного поля используем выражение в индексе.

---

### Как запомнить

- Если в запросах **всегда есть функция над полем**, индекс по «сырой» колонке чаще всего **не сработает**.
- Решение — **индекс по тому же выражению**, которое стоит в запросе.

---

### Проверь себя

<details>
<summary>Почему индекс по email (без LOWER) не помогает запросу WHERE LOWER(email) = LOWER($1)?</summary>

Потому что:

- индекс хранит **сырые значения email**;
- запрос сравнивает **результат функции LOWER(email)**;
- нет гарантии, что порядок по `email` и по `LOWER(email)` совпадает и подходит для поиска;
- оптимизатор не может использовать индекс напрямую для такого условия.

</details>

<details>
<summary>Какие функции безопасно использовать в индексах по выражениям?</summary>

Функции, которые:

- **immutable** — всегда возвращают одно и то же для одних и тех же аргументов;
- не зависят:
  - от текущего времени;
  - от других таблиц;
  - от глобального состояния.

Примеры:

- `LOWER`, `UPPER`;
- простая арифметика;
- нормализация строк/телефонов без обращения к внешним данным.

</details>

<details>
<summary>Индекс по (LOWER(email)). Будет ли он использоваться для запроса WHERE TRIM(email) = $1?</summary>

**Нет.** В индексе выражение — `LOWER(email)`, в запросе — `TRIM(email)`. Это разные выражения; планировщик не сопоставит условие с этим индексом. Чтобы использовать индекс, в запросе должно быть то же выражение: `LOWER(email) = LOWER($1)`. Если нужен поиск по TRIM, нужен отдельный индекс по `((TRIM(email)))` и в запросе условие с TRIM.

</details>

<details>
<summary>Почему функция в индексе по выражению должна быть immutable?</summary>

Индекс строится один раз и пересчитывается при изменении строк. Если бы функция зависела от времени, сессии или настроек (например, `now()`), значения в индексе со временем перестали бы соответствовать результатам той же функции при запросе — индекс стал бы **несогласованным** и мог бы возвращать неверные или неполные результаты. Immutable гарантирует: одни и те же аргументы → один и тот же результат всегда.

</details>

<details>
<summary>Зачем в CREATE INDEX по выражению двойные скобки: ((LOWER(email)))?</summary>

Чтобы отличить **выражение** от **имени колонки**. Одна пара скобок — синтаксис списка колонок индекса. Вторая — указание, что это именно выражение (функция от колонки), а не просто колонка. Без двойных скобок `LOWER(email)` могло бы трактоваться иначе. В PostgreSQL для индекса по выражению требуется `((выражение))`.

</details>

<details>
<summary>Индекс по (date(created_at)). Для какого запроса он подойдёт?</summary>

Для запросов, где в условии стоит **то же выражение** `date(created_at)`, например: `WHERE date(created_at) = '2025-01-15'` или `WHERE date(created_at) = date($1)`. Тогда планировщик сопоставляет условие с индексом и может использовать Index Scan. Запрос только по `created_at` (без date()) для этого индекса не подходит — выражение другое.

</details>

### Запомните

- Индекс по выражению — способ **подружить индекс с реальными запросами**, когда они используют функции/преобразования.
- Ключевые слова: **immutable**, совпадение выражений в индексе и в запросах.
- Выражение в WHERE должно **буквально совпадать** с выражением в индексе (та же функция, те же аргументы).

---

### Типичные ошибки

- Строить индекс по колонке (`email`), когда запросы все вида `WHERE LOWER(email) = ...`.
- Использовать в индексах **не‑immutable** функции (`now()`, `random()`), получая несогласованные данные.
- Менять выражения в запросе, не обновляя соответствующие индексы (нарушается соответствие).

---

### Итог §206 в трёх пунктах

1. **Индекс по выражению** хранит не колонку, а результат функции (например, `LOWER(email)`). Запрос должен использовать **то же выражение**, иначе планировщик не сопоставит условие с индексом.
2. **Функция в индексе должна быть immutable** — один и тот же аргумент всегда даёт один результат. Иначе индекс может стать некорректным.
3. **Типичная ошибка:** изменить запрос с `LOWER(email) = $1` на `email = LOWER($1)` — индекс по `(LOWER(email))` перестаёт использоваться. Либо оставлять выражение в запросе, либо отдельная колонка + индекс по ней.

---

### Часто путают (индексы по выражениям)

| Путают | Пояснение |
|--------|-----------|
| **Индекс по колонке и по выражению от колонки** | Индекс по `email` хранит исходные значения; запрос `WHERE LOWER(email) = ...` обращается к **другому** значению (результат функции). Порядок в индексе по email не совпадает с порядком по LOWER(email), поэтому такой индекс для этого условия не подходит. Нужен отдельный индекс по `(LOWER(email))`. |
| **Immutable и stable** | В индексе по выражению допустимы только **immutable** функции (результат не зависит ни от чего). **Stable** (например, результат может зависеть от timezone сессии) в индексах по выражению в PostgreSQL не допускается — индекс должен быть предсказуемо согласован. |

**Если забыл:** «запрос с функцией от колонки → индекс по той же функции; функция обязательно immutable».

---

### Часто задают вопрос (индексы по выражениям)

**Можно ли сделать составной индекс, где одна из колонок — выражение?** Да. Например: `CREATE INDEX ... ON users (LOWER(email), created_at);`. Порядок и правило left-prefix работают так же: индекс полезен для запросов с условием по `LOWER(email)` (и при необходимости по `created_at`). Поиск только по `created_at` без условия по email по такому индексу неэффективен — первая «колонка» индекса это выражение, без него нет левого префикса. В запросе при этом должно быть **то же выражение**: `WHERE LOWER(email) = $1 AND created_at > ...`.

---

### Что часто ломают (индексы по выражениям)

**Типичный баг:** сделали индекс по `(LOWER(email))`, в коде логина написали `WHERE LOWER(email) = $1`. Потом кто-то «улучшил» запрос на `WHERE email = LOWER($1)` (чтобы не вызывать LOWER по колонке). Индекс перестал использоваться: в условии нет выражения LOWER(email), планировщик не сопоставляет с индексом по выражению. В итоге — seq scan по большой таблице. Решение: либо везде в запросе оставлять то же выражение, что в индексе (`LOWER(email) = LOWER($1)`), либо завести отдельную колонку `email_lower` и индексировать её (тогда без индекса по выражению). Не менять форму запроса без проверки плана.

## Частые сценарии (задача → решение)

| Задача | Что сделать |
|--------|-------------|
| Таблица сильно обновляется по середине диапазона ключа, индекс раздулся. | Понизить fillfactor при следующем REINDEX; регулярно делать REINDEX CONCURRENTLY. |
| Поиск по массиву тегов или по ключам/значениям JSONB. | GIN по колонке (массив или jsonb); в запросах использовать операторы @>, ?, ?&, ?\|. |
| Запросы «все диапазоны, пересекающиеся с заданным» или геопоиск. | GiST по колонке (range или geometry); операторы &&, @>, <@ и т.д. |
| Огромная таблица логов, вставки только в конец по дате. | BRIN по created_at; включить autosummarize. |
| Частый запрос по (customer_id, status) с ORDER BY created_at и SELECT ещё пары колонок. | Составной индекс (customer_id, status, created_at) с INCLUDE этих колонок для index-only scan. |
| Индексировать только «активные» или «не удалённые» строки. | Частичный индекс с WHERE is_active / WHERE deleted_at IS NULL; в запросах всегда добавлять это условие. |
| Поиск по email без учёта регистра (WHERE LOWER(email) = ...). | Индекс по выражению ((LOWER(email))); в запросе писать то же выражение. |
| Уникальность только среди активных записей. | UNIQUE индекс с WHERE (например, UNIQUE (email) WHERE is_active). |

### Как не путать типы индексов (одна фраза на тип)

| Тип | В одну фразу |
|-----|----------------|
| **B-tree** | Одно значение на строку; порядок по ключу; точечный и диапазонный поиск; ключ → TID. |
| **GIN** | В одной строке **много элементов** (массив, JSONB); индекс: элемент → список TID; запросы «содержит», «есть ключ». |
| **GiST** | Значение — **область** (диапазон, геометрия); дерево регионов; запросы «пересекается», «содержит»; consistent отсекает ветки. |
| **BRIN** | Не по строкам, а по **блокам** таблицы; храним min/max по блоку; выгоден, когда данные **упорядочены** по колонке на диске. |
| **Составной** | Несколько колонок в **одном** индексе; порядок важен (left-prefix); один индекс под комбинацию условий. |
| **Частичный** | Индекс только по **части** строк (WHERE); запрос должен задавать то же условие. |
| **По выражению** | Индекс по **функции** от колонки (LOWER, date); в запросе — то же выражение; функция immutable. |

Когда сомневаешься — вернись к этой таблице и к чек-листу «какой тип индекса выбрать» ниже.

## Как связаны темы этой части (одна картинка)

- **B-tree (§200)** — база: как устроены страницы, split, fillfactor, bloat, TID. Это нужно для понимания любого индекса в Postgres и для решений «когда REINDEX», «почему индекс раздулся».
- **GIN (§201)** — когда в **одной строке много значений** (массив, JSONB): обратный индекс «элемент → список строк»; posting list/tree, pending list. Используешь, когда запросы по элементам/ключам/содержанию.
- **GiST (§202)** — когда значения это **области** (диапазон, геометрия): дерево регионов, consistent/penalty/picksplit. Используешь для пересечений, «содержит», «внутри».
- **BRIN (§203)** — когда таблица **огромная и данные упорядочены** по колонке: индекс по блокам (min/max), очень маленький. Используешь для диапазонных запросов по времени/ID при append-only.
- **Составные (§204)** — как **складывать колонки в один индекс**: порядок (left-prefix), covering, INCLUDE. От этого зависит, будет ли запрос использовать индекс и index-only scan.
- **Частичные (§205)** и **по выражениям (§206)** — как **уменьшить индекс** и подогнать под запросы: индексировать только часть строк (WHERE) или значение функции (LOWER, date), а не «всё подряд».

Итог: сначала понимаешь **как устроен** B-tree и что такое bloat/REINDEX; потом — **когда какой тип** индекса (GIN/GiST/BRIN); потом — **как спроектировать** составной/частичный/по выражению под свои запросы.

---

### Чек-лист: какой тип индекса выбрать (одна таблица)

Задай себе вопросы по строке и выбери тип:

| Вопрос | Если да → тип индекса | Кратко почему |
|--------|------------------------|---------------|
| Одно значение на строку (число, дата, строка, enum)? Запросы по равенству или диапазону? | **B-tree** | Стандартный индекс; порядок ключей, точечный и диапазонный поиск. |
| В одной строке много значений (массив тегов, ключи/значения JSONB)? Запросы «содержит», «есть ключ», «пересекается»? | **GIN** | Обратный индекс: элемент → список строк; операторы @>, ?, && и т.д. |
| Значение — это область (диапазон дат, период, геометрия)? Запросы «пересекается», «содержит», «внутри»? | **GiST** | Дерево регионов; consistent отсекает ветки по пересечению. |
| Таблица очень большая, данные упорядочены по колонке (append по времени/ID), запросы по диапазону? | **BRIN** | Индекс по блокам (min/max); маленький, отсекает целые диапазоны блоков. |
| Нужно искать по нескольким колонкам в одном запросе? | **Составной B-tree** | Порядок колонок = left-prefix; можно добавить INCLUDE для index-only scan. |
| Критичны только часть строк (активные, не удалённые)? | **Частичный индекс** | WHERE в индексе; запрос должен подразумевать то же условие. |
| В запросе всегда функция от колонки (LOWER, date, нормализация)? | **Индекс по выражению** | Индекс по (функция(колонка)); функция immutable, выражение в запросе то же. |

Если подходит несколько вариантов (например, B-tree и частичный B-tree) — выбирай по объёму данных и по тому, что важнее: скорость записи или чтения, размер индекса.

### Чек-лист перед созданием индекса (вопросы себе)

Перед тем как добавить новый индекс, полезно ответить:

1. **Какой запрос (или запросы) он должен ускорять?** Запиши конкретный WHERE/ORDER BY. Индекс проектируется под запрос, а не «на всякий случай».
2. **Одна колонка или несколько?** Если несколько — в каком порядке (left-prefix). Какие колонки только для SELECT — кандидаты в INCLUDE.
3. **Тип данных и операторы:** одно значение на строку → B-tree. Массив/JSONB по элементам → GIN. Диапазон/геометрия → GiST. Огромная таблица, упорядоченная по колонке → BRIN.
4. **Индексировать все строки или часть?** Если запросы всегда с `is_active` / `deleted_at IS NULL` — частичный индекс уменьшит размер и ускорит.
5. **В запросе есть функция от колонки?** Если да — индекс по выражению с той же функцией (immutable).
6. **Как часто пишем в таблицу?** Каждый индекс замедляет INSERT/UPDATE/DELETE. Лишние индексы «на каждую колонку» часто вредят записи.
7. **Планируешь ли REINDEX (CONCURRENTLY) для тяжёлых таблиц?** Для активно обновляемых индексов bloat неизбежен; заложи в эксплуатацию перестроение.

Если ответы неочевидны — посмотри план запроса (EXPLAIN) без индекса и подумай, какой доступ нужен (точечный, диапазон, по элементу, по региону).
