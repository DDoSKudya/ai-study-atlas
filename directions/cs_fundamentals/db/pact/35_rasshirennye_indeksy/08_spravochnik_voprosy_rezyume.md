## Краткое повторение терминологии

| Термин | Напоминание |
|--------|-------------|
| **Page (страница)** | Блок 8 КБ; единица хранения индекса и таблицы. |
| **Page split** | Разделение переполненной страницы на две при вставке. |
| **Fillfactor** | Процент заполнения страниц при создании; остаток — запас под вставки. |
| **Bloat** | Раздувание индекса; лечится REINDEX. |
| **TID** | Адрес строки в таблице; в индексе: ключ + TID. |
| **Posting list / Pending list (GIN)** | Список TID для элемента; буфер неслитых записей при fastupdate. |
| **Penalty / Picksplit / Consistent (GiST)** | Штраф при вставке, разбиение узла, проверка ветки под запрос. |
| **Block range / Summary (BRIN)** | Диапазон страниц; min/max по диапазону. |
| **Left-prefix** | Индекс (a,b,c) используется по a, (a,b), (a,b,c). |
| **Covering / INCLUDE** | Все нужные колонки в индексе; INCLUDE — только в листе, не для порядка. |
| **Частичный индекс** | WHERE в индексе; запрос должен подразумевать то же условие. |
| **Индекс по выражению** | Индекс по функции от колонки; функция immutable. |

<summary>Образец развёрнутого ответа на вопрос 1: почему частые UPDATE в середине диапазона приводят к bloat и как диагностировать?</summary>

**Почему bloat:** В PostgreSQL UPDATE не перезаписывает строку на месте, а создаёт новую версию (tuple). В индексе при этом добавляется новая запись (новый ключ + новый TID); старая запись (старый TID) остаётся до очистки. Когда обновления идут «в середину» диапазона ключей (например, меняем status у разных заказов), в одни и те же листовые страницы индекса постоянно вставляются новые записи. Страницы заполняются → срабатывают page split’ы → появляются новые страницы и «дыры». Плюс старые записи индекса (на мёртвые версии строк) накапливаются. В итоге индекс занимает больше места, чем нужно для актуальных данных, и чтение замедляется.

**Как диагностировать:** (1) Сравнить размер индекса с размером таблицы — если индекс заметно больше или сопоставим с таблицей при одном индексе по одной колонке, возможен bloat. (2) Посмотреть план запроса (EXPLAIN): если раньше использовался индекс, а теперь планировщик выбирает Seq Scan — могла вырасти оценка стоимости индекса из‑за раздутия. (3) Использовать расширения вроде pgstattuple для оценки доли мёртвых записей в индексе. (4) Регулярно смотреть на размеры индексов в мониторинге и планировать REINDEX CONCURRENTLY для активно обновляемых таблиц.

</details>

<summary>Образец развёрнутого ответа на вопрос 5: как выбирать порядок колонок в составном индексе?</summary>

**Идея:** Составной индекс хранит записи в **одном фиксированном порядке**: сначала по первой колонке, при равенстве — по второй, и т.д. Использоваться эффективно может только **левый префикс**: по первой колонке, по первой+второй, по первой+второй+третьей. По второй или третьей без первой — строки с нужным значением разбросаны по всему индексу, поиск по индексу не даёт выгоды.

**Как выбирать порядок:** (1) Определи **самые частые и тяжёлые запросы**. (2) **Первой** поставь колонку, по которой чаще всего фильтруют **равенством** (например, `customer_id = ?`) — она даёт сильное сужение набора. (3) Дальше — колонки по равенству или диапазону, которые в запросах идут **вместе** с первой. (4) В **конец** — колонку для **ORDER BY** или для дополнительного фильтра по диапазону; тогда внутри уже отфильтрованного набора данные уже отсортированы и не нужна отдельная сортировка. (5) Если в SELECT нужны колонки, которых нет в WHERE/ORDER BY, добавь их через **INCLUDE**, чтобы возможен был index-only scan. Один индекс не может идеально обслужить все запросы — выбирай порядок под 1–2 самых важных. Для запросов с другим порядком условий иногда нужен отдельный индекс или смириться с seq scan / другим планом.

</details>

<details>
<summary>Образец развёрнутого ответа на вопрос 2: в чём принципиальное отличие GIN от B-tree и почему GIN так хорошо подходит для JSONB и массивов?</summary>

**Отличие:** В B-tree **один ключ** — это одно значение колонки (например, одна строка `status = 'paid'`), и с ним связана одна или несколько записей «ключ → TID» в листе. То есть структура: «значение колонки → адреса строк». В GIN **один ключ** — это один **элемент внутри** составного значения (один тег из массива, один ключ или пара ключ–значение из JSONB), а **значение** — это **целый список TID** (posting list) всех строк, где этот элемент встречается. Структура: «элемент → список адресов строк». Поэтому B-tree отвечает на вопрос «найти строки, где колонка = X»; GIN — «найти строки, где **в** колонке (массиве/JSONB) **есть** элемент X» или «содержится подструктура Y».

**Почему GIN подходит для JSONB и массивов:** В одной строке массива или JSONB может быть **много элементов** (теги, ключи, значения). Запросы часто звучат как «все строки, где есть тег X», «где JSON содержит ключ type со значением signup». B-tree по всей колонке хранит одно значение на строку — для «содержит элемент» пришлось бы сканировать таблицу и проверять каждый массив/документ. GIN заранее разбивает каждое значение на элементы и для каждого элемента хранит список строк; запрос «есть тег X» сводится к поиску ключа X в GIN и чтению его posting list. Плюс операторы `@>`, `?`, `&&` и т.д. спроектированы так, что планировщик использует именно GIN. Итог: GIN не «лучше B-tree вообще», а решает **другую задачу** — поиск по внутренним элементам поля с множеством значений.

</details>

<summary>Образец развёрнутого ответа на вопрос 4: в каких сценариях BRIN даёт радикальный выигрыш по размеру и скорости, а в каких почти не помогает?</summary>

**Радикальный выигрыш:** (1) **Очень большая таблица** (сотни миллионов строк и больше), по которой идут **диапазонные запросы** (WHERE created_at BETWEEN ..., WHERE id > ...). (2) Данные **физически упорядочены** по индексируемой колонке на диске: типично append-only по времени или по автоинкрементному id (вставки в конец). Тогда в каждом block range значения колонки лежат в узком интервале (min и max близки); по summary можно отсечь большинство диапазонов блоков и читать только те, где могут быть подходящие строки. Размер BRIN при этом — одна запись на диапазон блоков (например, на 128 страниц), а не на каждую строку, поэтому индекс в **сотни раз меньше** B-tree. Скорость выигрывает за счёт малого I/O: читаем крошечный BRIN, отсекаем почти все блоки, читаем только «подозрительные» диапазоны таблицы.

**Почти не помогает:** (1) Данные **перемешаны** по индексируемой колонке: частые UPDATE по разным строкам, вставки в случайном порядке (UUID, случайный id). Тогда в одном блоке оказываются строки с самыми разными значениями; min/max по блоку почти совпадают с общим min/max по таблице, отсечь по summary почти ничего нельзя — придётся читать почти все блоки. (2) **Точечные запросы** (WHERE id = 123): BRIN не хранит «какая строка в каком блоке» по ключу, он только отсекает целые диапазоны блоков; для одной строки по ключу нужен B-tree. (3) Небольшая таблица: выгода от BRIN не успевает окупиться, а накладные расходы на чтение summary и отсечение есть. **Итог:** BRIN — для **больших, упорядоченных, диапазонных** сценариев; для маленьких, перемешанных или точечных — B-tree или другой индекс.

</details>

<summary>Образец развёрнутого ответа на вопрос 9: зачем в GIN нужен pending list и что будет, если его долго не сливать?</summary>

**Зачем нужен:** GIN при каждой вставке/обновлении строки должен добавить TID во **все** posting list’ы элементов этой строки (каждый тег массива, каждый ключ JSONB и т.д.). Если делать это сразу в основное дерево индекса, одна вставка превращается в много обновлений страниц и возможных split’ов — запись сильно замедляется. **Pending list** — это буфер: новые записи сначала попадают в него (дешёвая операция), а **слияние** в основное дерево откладывается и выполняется **пачкой** (при VACUUM или при достижении порога). Так вставки остаются быстрыми, а тяжёлая работа по перестроению дерева делается реже и сразу по многим записям.

**Если долго не сливать:** (1) Pending list **растёт**. При каждом запросе по GIN движок должен смотреть и в основное дерево, и в pending list, и объединять результаты. Чем больше pending list, тем дольше каждый такой запрос. (2) Запросы по индексу **замедляются**, вплоть до того, что планировщик может предпочесть seq scan. (3) Размер индекса на диске тоже растёт из-за неслитых данных. Решение: обеспечить регулярный **VACUUM** по таблице (в т.ч. AUTOVACUUM), чтобы pending list периодически сливался в основной индекс. Для таблиц с только записью и редким VACUUM стоит либо настроить autovacuum агрессивнее для этой таблицы, либо периодически вызывать VACUUM вручную.

</details>

<summary>Образец развёрнутого ответа на вопрос 3: какие типы данных и запросы особенно хорошо ложатся на GiST и почему?</summary>

**Типы данных:** (1) **Диапазоны** (range): `int4range`, `tstzrange`, `daterange` и т.п. — значение это не точка, а **интервал** «от и до». (2) **Геометрия** (PostGIS): точки, линии, полигоны — значение это **область** на плоскости (или в пространстве). (3) В общем случае — любой тип, который можно представить как **регион** (минимальный прямоугольник MBR, интервал) и для которого можно определить «пересекается / не пересекается» с запросом.

**Запросы:** «Найти все диапазоны, **пересекающиеся** с заданным» (`WHERE period && $1`). «Все объекты, **содержащие** эту точку» или «все точки **внутри** этого полигона». «Все записи, чей период **содержится** в заданном» (`<@`, `@>`). То есть запросы не по одному значению «равно/больше/меньше», а по **отношению регионов**: пересечение, включение, соседство.

**Почему GiST:** B-tree умеет только **линейный порядок** (одно значение на ключ, сравнение «меньше/больше»). Для «пересекается с [5,10]» подходящие диапазоны могут иметь любое начало и конец ([1,6], [9,15], [3,7]) — они не идут подряд в одном порядке по одному числу. GiST хранит в узлах **обобщённые регионы** (union дочерних) и при поиске вызывает **consistent**: «может ли эта ветка содержать ответ?» Если регион ветки не пересекается с запросом — вся ветка отсекается. Так за один проход отсекаются целые поддеревья без перебора всех листьев. **Итог:** GiST хорошо ложится на всё, что является **областью** и по чему нужны запросы на пересечение/включение, а не точечное сравнение.

</details>

<summary>Образец развёрнутого ответа на вопрос 6: в каких случаях стоит предпочесть частичный индекс полному и какие требования это накладывает на запросы?</summary>

**Когда предпочесть частичный:** (1) **Большая таблица**, но критичные запросы касаются только **части строк**: например, только активные пользователи (`is_active`), только не удалённые (`deleted_at IS NULL`), только последние N дней. Полный индекс по email на 100 млн строк — огромный; частичный с `WHERE is_active` на 2 млн активных — в разы меньше. (2) **Меньше размер индекса** → меньше места на диске, быстрее обновление при INSERT/UPDATE/DELETE (в индекс попадают только строки, прошедшие WHERE). (3) **Уникальность только среди подмножества**: например, email уникален среди активных, а среди удалённых дубликаты допустимы — делаем UNIQUE (email) WHERE is_active.

**Требования к запросам:** Планировщик использует частичный индекс **только если** условие запроса **подразумевает то же подмножество строк**, что и WHERE индекса. То есть в запросе должно быть условие, эквивалентное или **более строгое**, чем условие индекса. Пример: индекс `WHERE is_active` — запрос должен содержать `is_active` (или что-то вроде `is_active AND ...`), иначе в индексе нет неактивных строк и результат мог бы быть неполным. **Практика:** всегда добавлять в запросы то же условие, что в частичном индексе (например, везде писать `WHERE is_active AND ...`), тогда планировщик сможет использовать индекс и не будет «забытых» строк. **Итог:** частичный индекс — когда индексируем **осознанно только нужное подмножество**; запросы должны это подмножество явно задавать.

</details>

<summary>Образец развёрнутого ответа на вопрос 7: когда имеет смысл вводить индексы по выражениям и какие ограничения накладывает требование immutable‑функций?</summary>

**Когда вводить:** Когда запросы **постоянно используют одну и ту же функцию** от колонки: поиск по email без учёта регистра (`LOWER(email) = LOWER($1)`), группировка или фильтр по дате без времени (`date(created_at) = '2025-01-01'`), нормализация пробелов/регистра для поиска. Обычный индекс по колонке в таких случаях **не помогает**: в индексе порядок по «сырому» значению, а в запросе сравнивается результат функции — планировщик не сопоставляет условие с индексом. Индекс по **выражению** хранит уже вычисленное значение (например, `LOWER(email)`), и запрос с тем же выражением может его использовать. Имеет смысл вводить, когда такой запрос частый и тяжёлый (большая таблица, много строк отфильтровывается по этому условию).

**Ограничение immutable:** Функция в выражении индекса должна быть **immutable** — при одних и тех же аргументах возвращать один и тот же результат, не зависящий от времени, сессии, настроек (timezone и т.д.). Причина: индекс строится один раз и пересчитывается при изменении строк; если бы функция давала разный результат в разное время (например, `now()`), значения в индексе перестали бы соответствовать реальным результатам выражения при запросе — индекс стал бы некорректным. **Stable**-функции (результат может зависеть от сессии) в определении индекса по выражению в PostgreSQL не допускаются. **Итог:** индекс по выражению — когда запросы «через функцию»; функция обязательно immutable; в запросе должно стоять **то же выражение**, что в индексе.

</details>

<summary>Образец развёрнутого ответа на вопрос 8: что такое TID и зачем он хранится в листовой странице B-tree?</summary>

**TID (Tuple Identifier)** — это **адрес одной версии строки** в таблице. В PostgreSQL он состоит из номера **блока** (страницы) таблицы, где лежит строка, и **номера строки внутри этого блока** (позиция в странице). По TID движок однозначно находит конкретную строку на диске.

**Зачем хранится в листе B-tree:** В листовой странице B-tree хранятся пары **«ключ → TID»**: ключ — это значение индексируемой колонки (например, `status = 'paid'`), TID — адрес строки в таблице, где это значение встречается. Индекс **не хранит копию всей строки** (все колонки) — только ключ и адрес. Так индекс остаётся **компактным**: не дублируем данные таблицы. При поиске по индексу мы (1) находим ключ в листе и читаем его TID, (2) по TID идём в **таблицу** и читаем полную строку из соответствующего блока. То есть TID — это **связка** между индексом и таблицей: индекс отвечает на вопрос «где лежат строки с таким ключом?» (даёт TID’ы), а сами данные берутся из таблицы по этим адресам. Если бы в индексе хранились все нужные колонки (covering index, INCLUDE), по TID в таблицу можно было бы не ходить (index-only scan).

</details>

<details>
<summary>Образец развёрнутого ответа на вопрос 10: почему для «все диапазоны, пересекающиеся с [5,10]» одного B-tree по началу или концу диапазона недостаточно?</summary>

**Суть запроса:** Нам нужны все диапазоны, у которых есть **общая часть** с [5,10]. Подходят, например, [1,6] (пересекается по [5,6]), [4,8], [9,15], [3,7]. Не подходят [1,2], [20,21] — не пересекаются.

**Почему B-tree по началу не спасает:** В B-tree по «началу диапазона» ключи упорядочены по одному числу (например, по левой границе). Подходящие диапазоны имеют **разное** начало: 1, 3, 4, 9, … Неподходящие тоже имеют разное начало: 1, 20, … То есть «все начала от X до Y» не совпадают с «все диапазоны, пересекающиеся с [5,10]»: диапазон с началом 1 может быть и [1,2] (не подходит), и [1,6] (подходит). Один непрерывный диапазон ключей в B-tree по началу даёт либо лишние строки (придётся фильтровать), либо пропуски. Чтобы не пропустить ничего, пришлось бы читать почти весь индекс и проверять пересечение для каждой записи — выгоды от индекса почти нет.

**Почему B-tree по концу аналогично:** То же самое: подходящие диапазоны имеют разный конец (6, 8, 15, 7, …); по одному числу «конец» нельзя задать один отрезок ключей, который ровно соответствует «пересекается с [5,10]».

**Почему GiST решает задачу:** GiST хранит в узлах **сам регион** (или его обобщение — union дочерних). Функция **consistent** для запроса «пересекается с [5,10]» по ключу узла «здесь диапазоны [1,20]» возвращает: да, пересекается → идём в ветку. По ключу «здесь диапазоны [100,200]» — нет, не пересекается → **отсекаем всю ветку**. Так мы не перебираем все листья, а отсекаем целые поддеревья. **Итог:** B-tree хорош для порядка по **одному** значению; «пересекается с интервалом» — это условие над **парой** (начало, конец) и областью, поэтому нужен GiST (или аналог), а не один B-tree по одному полю.

</details>

## Типичные ошибки по части и резюме

- Считать, что «индекс — это просто B-tree по колонке», и игнорировать **fillfactor, bloat и необходимость REINDEX**.
- Использовать GIN/GiST/BRIN, не понимая **их внутреннюю модель и подходящие сценарии**, из‑за чего они кажутся «неэффективными».
- Проектировать индексы **по колонкам, а не по запросам**: не учитывать порядок в составных индексах, не использовать INCLUDE.
- Идексировать **всю таблицу**, даже если запросы касаются только небольшого подмножества строк (там, где частичные индексы идеально подходят).
- Писать запросы с функциями над колонками (`LOWER(email)`, `date(created_at)`), но не создавать соответствующие индексы по выражениям.

Главная мысль: **индексы — это не магия, а набор конкретных структур и правил**, которые можно понять и использовать осознанно. Понимая B-tree, GIN, GiST, BRIN и техники проектирования индексов (составные, частичные, по выражениям), ты можешь **делать базу быстрее на уровне архитектуры запросов и схемы**, а не только «настраивая параметры».

## Резюме части в одном абзаце (для быстрого повторения)

**Структуры индексов (§200–203).** **B-tree** — дерево страниц с ключами и TID; при переполнении страницы делается split, откуда растёт bloat; fillfactor оставляет запас под вставки; bloat снимается REINDEX (в проде — CONCURRENTLY). **GIN** — «элемент → список строк» (posting list/tree), удобен для массивов и JSONB; fastupdate (pending list) ускоряет запись, но требует VACUUM для слияния. **GiST** — каркас для «регионов» (диапазоны, геометрия); consistent/penalty/picksplit решают, как искать и куда вставлять. **BRIN** — индекс по диапазонам блоков (min/max summary), очень маленький; выгоден при данных, упорядоченных по колонке на диске.

**Проектирование под запросы (§204–206).** **Составные индексы** — порядок колонок критичен (left-prefix: без первой колонки поиск по остальным неэффективен); INCLUDE и covering дают index-only scan. **Частичные индексы** — индексируем только подмножество строк по WHERE; запрос должен подразумевать то же условие, иначе индекс не используется. **Индексы по выражениям** — индекс по функции от колонки (LOWER(email), date(created_at)); функция должна быть immutable, выражение в запросе — в точности совпадать с выражением в индексе.

**Если забыл, с чего начать повторение:** (1) **§200** — как устроен B-tree (страницы, TID, split, fillfactor, bloat, REINDEX); это база для всего. (2) Таблица **«Как не путать типы индексов»** и **чек-лист «какой тип индекса выбрать»** — чтобы быстро вспомнить: B-tree / GIN / GiST / BRIN под какую задачу. (3) **§204–206** — как проектировать индексы под запросы: порядок колонок, left-prefix, INCLUDE; частичный индекс и условие в запросе; индекс по выражению и совпадение выражения.
