## 178. Qdrant детали

### В этом разделе главное

- Qdrant — **векторная БД с простым и понятным HTTP/GRPC‑API**, ориентированная на практическое использование.
- Модель: **collections → points (vector + payload)**, фильтры по payload и поддержка **dense и sparse векторов**.
- Поддерживается **шардирование и репликация**, удобно для горизонтального масштабирования.

---

### Простыми словами

Qdrant — это:

- «векторный аналог Elasticsearch, но легче в подъёме и с простыми JSON‑запросами».

Ты:

- создаёшь **коллекцию**;
- добавляешь туда **points**:
  - `id`,
  - один или несколько векторов,
  - `payload` (метаданные);
- делаешь запросы:
  - **по ближайшим векторам**;
  - с **фильтрами по payload**.

---

### Теория и правила

#### Collections

Коллекция задаёт:

- размерность векторов;
- метрику (COSINE, DOT, EUCLID);
- параметры индекса (HNSW и др.);
- настройки шардирования/репликации.

Пример (идея конфигурации):

```json
{
  "name": "documents",
  "vectors": {
    "size": 768,
    "distance": "Cosine"
  }
}
```

#### Points и payload

Point:

- `id` — уникальный идентификатор (число или строка);
- `vector` — массив чисел;
- `payload` — произвольный JSON:
  - `{"language": "ru", "category": "faq", "tags": ["backup", "postgres"]}`.

Запись (концептуально):

```json
{
  "points": [
    {
      "id": 1,
      "vector": [0.1, 0.2, ...],
      "payload": {
        "language": "ru",
        "category": "faq",
        "tags": ["backup", "postgres"]
      }
    }
  ]
}
```

#### Фильтры по payload

Фильтры описываются JSON‑структурами:

```json
{
  "must": [
    {
      "key": "language",
      "match": { "value": "ru" }
    },
    {
      "key": "category",
      "match": { "any": ["faq", "guide"] }
    }
  ],
  "must_not": [
    {
      "key": "archived",
      "match": { "value": true }
    }
  ]
}
```

Их можно комбинировать (`must`, `should`, `must_not`).

#### Sparse vectors

Qdrant поддерживает:

- **dense vectors** — обычные эмбеддинги;
- **sparse vectors** — разреженные представления (например, BM25‑подобные веса по токенам).

Сценарий:

- dense vector — семантика;
- sparse vector — точные слова/формы;
- вместе можно делать **гибридный поиск**, сильно похожий на «BM25+вектор».

#### Sharding и масштабирование

Qdrant умеет:

- делить коллекцию на **shards**;
- держать несколько **реплик**;
- балансировать нагрузку по нодам кластера.

Важно:

- при проектировании нужно понимать:
  - сколько шардов нужно;
  - как распределится нагрузка;
  - какие требования по отказоустойчивости.

---

### Примеры

#### Пример 1. Поиск с фильтром по payload

```json
POST /collections/documents/points/search
{
  "vector": [0.1, 0.2, ...],
  "top": 5,
  "filter": {
    "must": [
      { "key": "language", "match": { "value": "ru" } }
    ]
  },
  "with_payload": true
}
```

Qdrant вернёт:

- id,
- расстояние,
- payload для каждого совпадения.

#### Пример 2. Использование sparse vectors (концепция)

1. Считаешь dense‑эмбеддинг текста;
2. Считаешь sparse‑вектор (например, BM25‑подобный вес по токенам);
3. Отправляешь оба в Qdrant как разные векторы одного point'а;
4. При поиске задаёшь:
   - `query_vector` для dense части,
   - `query_sparse` для sparse части;
5. Qdrant комбинирует скоринги по заданным весам.

Это гибкий способ объединить:

- «понимание смысла»,
- и «уважение к точным словам».

---

### Картинка в голове

Qdrant — это:

- **«коллекции ящиков», где в каждом ящике лежит вектор + наклейка с описанием (payload)**;
- фильтры по payload — это «найди ящики с такими‑то наклейками»;
- векторный поиск — «среди этих ящиков найди те, чьи отпечатки похожи на мой».

Sparse vectors — это:

- **отдельные наборы «наклеек‑по‑словам» с весами**, которые учитывают текстовую часть запроса.

---

### Типичные ошибки

- Игнорировать payload и пытаться кодировать всю дополнительную информацию **вовнутрь вектора** — сложнее отлаживать, хуже фильтровать.
- Не продумывать схему шардирования и столкнуться с «горячими шардами», когда часть данных используется намного чаще.
- Переусердствовать с количеством/сложностью payload‑полей и фильтров, не измеряя, как это влияет на задержки.
- Не использовать sparse vectors там, где **ключевые слова критичны** (например, в коде, названиях методов, технических терминах).

---

### Проверь себя

<details>
<summary>1. Чем отличается dense vector от sparse vector в контексте Qdrant?</summary>

- dense vector — обычный плотный эмбеддинг фиксированной длины (все координаты присутствуют),
- sparse vector — структура, где явно перечислены только **ненулевые** координаты и их веса (часто используется для текстовых моделей типа BM25).

</details>

<details>
<summary>2. Зачем в Qdrant хранят payload, если уже есть вектор?</summary>

Payload:

- позволяет делать **структурные фильтры** (по языку, типу, тегам и т.п.),
- нужен для отображения результата (название, текст, идентификатор),
- служит основой для гибридных стратегий (отбор по полям + векторный скоринг).

</details>

<details>
<summary>3. Когда шардирование коллекции становится необходимостью?</summary>

Когда:

- объём данных и нагрузка становятся такими, что **одна нода не справляется** по памяти/CPU/диску;
- нужно:
  - распределить вектора по нескольким нодам,
  - обеспечить отказоустойчивость.

</details>
