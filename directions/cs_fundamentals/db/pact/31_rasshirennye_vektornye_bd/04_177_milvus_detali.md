## 177. Milvus детали

### В этом разделе главное

- Milvus — **специализированный движок векторного поиска**, рассчитанный на огромные объёмы данных.
- Основные сущности: **collections, partitions, segments** и **индексы** (FLAT, IVF, HNSW…).
- Для продуктивной работы важно понимать **как Milvus грузит коллекции в память** и как он масштабируется.

---

### Простыми словами

Milvus — это как «PostgreSQL для векторов», но:

- без SQL в классическом виде,
- с архитектурой, заточенной под:
  - миллионы/миллиарды векторов,
  - распределённое хранение,
  - добавление/удаление/поиск с низкими задержками.

Ты:

- создаёшь **collection** (размерность, метрика, доп. поля),
- наполняешь её векторами,
- создаёшь индекс (например, IVF/HNSW),
- при запросах:
  - Milvus **загружает нужные сегменты в память**,
  - выполняет поиск, возвращает топ‑K.

---

### Теория и правила

#### Collections и schema

Схема коллекции:

- `id` (primary key),
- `vector` поле (с указанной размерностью и метрикой),
- произвольные дополнительные поля (metadata).

Пример (концептуально):

```text
Collection: documents
Fields:
  - id: int64 (primary key)
  - embedding: float_vector(768), metric_type = COSINE
  - language: varchar
  - category: varchar
```

#### Partitions и segments

- **Partition** — логическое разделение коллекции:
  - например, по языку или по времени;
  - позволяет ограничить поиск подмножеством данных.
- **Segment** — физический кусок данных:
  - Milvus хранит коллекцию как набор сегментов;
  - сегменты могут быть «горячими» (в памяти) и «холодными» (на диске).

Идея:

- при поиске Milvus решает:
  - какие partitions/segments затронуть,
  - какие нужно **подгрузить в память**.

#### Индексы: FLAT, IVF, HNSW

- **FLAT**:
  - точный поиск (полный перебор с SIMD‑ускорением),
  - очень точен, но дорог на больших объёмах.
- **IVF_FLAT / IVF_SQ8**:
  - аналогично `ivfflat` в `pgvector`,
  - `IVF_SQ8` — с квантизацией (меньше памяти, чуть хуже точность).
- **HNSW**:
  - графовый индекс, как раньше,
  - хорошо подходит для высококачественного ANN‑поиска.

Выбор:

- небольшие объёмы/строгая точность → FLAT;
- крупные объёмы/баланс → IVF (с тюнингом параметров);
- высокие требования к качеству и скорости → HNSW.

#### Load / release

Milvus явно различает:

- **хранятся данные** (на диске/объектном хранилище),
- **загружены ли они в память**.

Команды (концептуально):

- `load_collection("documents")` — загрузить коллекцию в память;
- `release_collection("documents")` — выгрузить, освободить RAM.

Когда это важно:

- если коллекция очень большая, **целиком держать её в памяти нельзя**;
- тогда:
  - либо грузят только нужные partitions,
  - либо полагаются на автоматическую подгрузку сегментов.

### Примеры

Примеры будут концептуальными — синтаксис API смотри в документации Milvus.

#### Пример 1. Создание коллекции и индекса

```python
from pymilvus import connections, FieldSchema, CollectionSchema, DataType, Collection

connections.connect("default", host="localhost", port="19530")

fields = [
    FieldSchema(name="id", dtype=DataType.INT64, is_primary=True, auto_id=True),
    FieldSchema(name="embedding", dtype=DataType.FLOAT_VECTOR, dim=768),
    FieldSchema(name="language", dtype=DataType.VARCHAR, max_length=8),
]

schema = CollectionSchema(fields=fields, description="documents embeddings")
collection = Collection(name="documents", schema=schema)

index_params = {
    "metric_type": "COSINE",
    "index_type": "IVF_FLAT",
    "params": {"nlist": 1024}
}

collection.create_index(field_name="embedding", index_params=index_params)
collection.load()  # загрузить в память
```

#### Пример 2. Поиск с фильтром по полю

```python
q_vec = embed("как восстановить базу")

search_params = {
    "metric_type": "COSINE",
    "params": {"nprobe": 32}
}

results = collection.search(
    data=[q_vec],
    anns_field="embedding",
    param=search_params,
    limit=5,
    expr="language == 'ru'"
)
```

### Картинка в голове

Milvus — это:

- **«ферма серверов, специализирующихся на векторном поиске»**;
- каждая коллекция — поле с коробками (векторами);
- partitions — **ограждения** на поле;
- segments — **кучки коробок**:
  - часть из них лежит прямо «под рукой» (в памяти),
  - часть — в дальнем сарае (на диске).

Команды `load`/`release` — это:

- «принести нужные кучки поближе» или «убрать обратно в сарай».

---

### Типичные ошибки

- Пытаться использовать FLAT на **многих миллионах векторов**, ожидая «магического» быстродействия.
- Не думать про partitions и грузить всю коллекцию в память, хотя реально используются только данные за последний месяц/для одной страны.
- Неправильно выбирать параметры IVF (`nlist`, `nprobe`) или HNSW (`M`, `efConstruction`, `efSearch`) — крайности «быстро, но неточно» или «медленно, но чуть лучше».
- Не мониторить использование памяти и диска, пока система не начинает «задыхаться».

### Проверь себя

<details>
<summary>1. Зачем Milvus разделяет понятия <code>load</code> и просто «наличие коллекции»?</summary>

Чтобы можно было:

- хранить огромные коллекции на диске,
- но **загружать в память только нужные части** (коллекции/partitions),
- управлять потреблением RAM в зависимости от сценария.

</details>

<details>
<summary>2. В чём практическая разница между FLAT и IVF/HNSW?</summary>

- FLAT:
  - ищет «в лоб» по всем векторам,
  - даёт точный ответ,
  - плохо масштабируется при росте данных.
- IVF/HNSW:
  - делают **приближённый поиск**,
  - радикально ускоряют запросы,
  - требуют тюнинга параметров и чуть жертвуют точностью.

</details>

<details>
<summary>3. Когда partitions в Milvus особенно полезны?</summary>

Когда:

- есть естественное деление данных (по времени, региону, клиенту),
- запросы почти всегда ограничены одной/несколькими такими группами.

Тогда можно:

- загружать только нужные partitions,
- ускорять поиск и снижать расходы ресурсов.

</details>
