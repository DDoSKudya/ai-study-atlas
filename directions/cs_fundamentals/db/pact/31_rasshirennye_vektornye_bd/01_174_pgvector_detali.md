## 174. pgvector детали

### В этом разделе главное

- `pgvector` добавляет в PostgreSQL **тип `vector(n)` и операторы расстояний**, превращая обычную БД в простую векторную — ты можешь хранить эмбеддинги прямо рядом с обычными таблицами.
- Для производительности нужны **специализированные индексы** (`ivfflat`, `hnsw`), иначе поиск по сотням тысяч/миллионам векторов будет слишком медленным.
- Качество поиска определяется **выбором метрики и тюнингом параметров индекса** (lists/probes для IVF, m/ef для HNSW) и тем, **как ты комбинируешь векторный поиск с обычными фильтрами/SQL**.
- `pgvector` хорош как **первый шаг в мир векторных БД**: ты не поднимаешь отдельный сервис, а добавляешь один extension к знакомому PostgreSQL.

---

### Простыми словами

#### Зачем вообще это нужно

Представь, что у тебя есть таблица `documents`:

- столбец `title` — заголовок статьи;
- столбец `content` — полный текст;
- столбец `language` — язык;
- и т.д.

Классический поиск:

- по `LIKE '%backup%'`,
- по полнотекстовому индексу (tsvector, BM25),
- по фильтрам (`WHERE language = 'ru'`).

Проблема:

- пользователь пишет: «как сделать резервную копию базы»,
- а у тебя статья называется: «Настройка WAL‑архивации и бэкапов в PostgreSQL»,
- слова «сделать резервную копию» там могут вообще не встречаться.

**Векторный подход**:

- ты прогоняешь текст статьи через модель (эмбеддер) → получаешь **вектор** (набор чисел), который как бы «описывает смысл текста»;
- ты прогоняешь запрос пользователя через ту же модель → получаешь ещё один вектор;
- если два текста «по смыслу похожи», **их вектора будут находиться близко** в векторном пространстве.

Тогда задача «найти похожие статьи» сводится к:

- «найти вектора, которые ближе всего к вектору запроса».

#### Где тут `pgvector`

Обычно для этого поднимают **отдельную векторную БД** (Milvus, Qdrant и т.п.).  
Но часто хочется:

- не городить новый сервис,
- не тянуть ещё одну технологию,
- а просто использовать имеющийся PostgreSQL.

`pgvector` решает именно это:

- даёт **тип данных** для хранения векторов,
- даёт **операторы расстояния**,
- даёт **индексы**, чтобы поиск был быстрым.

И всё это — прямо внутри PostgreSQL, который ты уже знаешь.

#### На пальцах

Представь, что:

- каждый документ — это **точка** в многомерном пространстве;
- чем ближе две точки — тем **похожее** содержимое документов.

`pgvector` позволяет:

- **записать этот вектор прямо в PostgreSQL** в столбец нужного типа;
- **сравнивать вектора между собой** (насколько два документа похожи);
- **искать ближайшие соседи** (top‑K похожих документов).

Но:

- Если делать поиск «в лоб» без индекса — PostgreSQL будет сравнивать вектор запроса **со всеми строками**.
- При десятках/сотнях тысяч векторов это ещё терпимо, при миллионах — уже нет.
- Поэтому `pgvector` даёт **индексы, заточенные под векторные расстояния**, а не под обычные операции сравнения.

### Теория и правила

#### Тип `vector(n)`

##### Как включить расширение

Объявление расширения:

```sql
CREATE EXTENSION IF NOT EXISTS vector;
```

Эта команда:

- говорит PostgreSQL: «подключи расширение `vector`»;
- после этого в системе появляется **новый тип данных** `vector`.

Обычно её выполняют:

- один раз на базу (или на нужную схему),
- под суперпользователем или пользователем с правами создания расширений.

##### Объявление столбца с вектором

Предположим, у нас уже есть таблица документов. Покажем полный DDL:

```sql
CREATE TABLE documents (
  id         bigserial PRIMARY KEY,
  title      text NOT NULL,
  content    text NOT NULL,
  language   text NOT NULL DEFAULT 'en',
  created_at timestamptz NOT NULL DEFAULT now(),
  embedding  vector(768)  -- 768‑мерный эмбеддинг
);
```

Разбор по кусочкам:

- `vector(768)`:
  - в скобках указано **количество измерений**;
  - у тебя модель эмбеддинга должна выдавать **вектора именно этой длины** (768 чисел).
- PostgreSQL хранит этот столбец как **массив фиксированной длины**:
  - примерно как `float4[768]`, но с более компактной упаковкой.

Важно:

- если ты попытаешься записать в `vector(768)` вектор длины 512 или 1024 — PostgreSQL выбросит ошибку;
- если ты потом решишь сменить модель на другую размерность (например, 1024), придётся:
  - либо завести новый столбец `embedding_1024 vector(1024)`,
  - либо новую таблицу,
  - и мигрировать данные.

##### Как вставлять и читать вектора

Есть два основных варианта:

1. **Вставлять явно строкой**:

   ```sql
   INSERT INTO documents (title, content, language, embedding)
   VALUES (
     'Как сделать бэкап',
     'Подробная инструкция по резервному копированию…',
     'ru',
     '[0.12, 0.03, -0.98, ...]'::vector
   );
   ```

   Здесь `[...]::vector` — строка, которую PostgreSQL понимает как вектор.

2. **Передавать из приложения параметром**:

   - ты считаешь эмбеддинг в коде (Python, Go и т.п.);
   - передаёшь массив чисел как параметр запроса;
   - драйвер/библиотека обычно знает, как его упаковать (или ты указываешь тип `vector` явно).

Практический совет:

- **не пытайся считать вектора внутри PostgreSQL** (через UDF и т.п.), если у тебя нет очень весомой причины;
- проще и надёжнее:
  - взять SDK к модели (OpenAI/transformers и т.д.) в приложении;
  - посчитать вектор там;
  - отправить в PostgreSQL как обычное значение.

#### Операторы расстояний

В `pgvector` есть несколько операторов:

- `<->` — L2‑расстояние (евклидово);
- `<=>` — косинусное расстояние;
- `<#>` — отрицательное скалярное произведение (для dot‑product).

Для запросов ближайших соседей:

- **чем меньше значение расстояния — тем ближе объекты**;
- при косинусной метрике часто **нормализуют вектора до длины 1**, тогда косинусный similarity легко выражается через расстояние.

Немного интуиции:

- L2‑расстояние:
  - представь, что каждая точка — дом в городе;
  - L2 — это «прямая линия» между домами (если можно летать на вертолёте);
  - чем меньше расстояние — тем ближе дома.
- Косинусное расстояние:
  - смотришь не на расстояние как таковое, а на **угол между направлениями**;
  - два длинных вектора, идущих примерно в одну сторону, считаются похожими;
  - длина (насколько далеко они находятся от начала координат) менее важна.
- Dot product:
  - если оба вектора нормализованы, dot product почти совпадает с косинусом;
  - удобно, если модель обучена именно на dot‑product.

Примеры:

```sql
-- Топ‑10 ближайших по L2
SELECT id, title, embedding <-> '[...]'::vector AS distance
FROM documents
ORDER BY embedding <-> '[...]'::vector
LIMIT 10;

-- Топ‑10 ближайших по косинусному расстоянию
SELECT id, title, embedding <=> '[...]'::vector AS distance
FROM documents
ORDER BY embedding <=> '[...]'::vector
LIMIT 10;
```

#### Индексы `ivfflat`

##### В чём вообще проблема без индекса

Если индекса нет, типичный запрос:

```sql
SELECT id, title, embedding <=> '[...]'::vector AS distance
FROM documents
ORDER BY embedding <=> '[...]'::vector
LIMIT 10;
```

делает следующее:

1. Берёт твой запрос‑вектор.
2. Для **каждой строки таблицы**:
   - считает расстояние между запросом и `embedding`;
3. Сортирует все строки по этому расстоянию.

Если у тебя:

- 10 000 строк — это ещё терпимо;
- 10 000 000 строк — это уже очень тяжело:
  - и по CPU (очень много операций),
  - и по памяти (сортировать столько результатов).

Нужен способ:

- **не смотреть на все вектора**, а только на «подозрительно близкие».

##### Идея `ivfflat` по‑человечески

`ivfflat` (Inverted File Flat) — классический приближённый индекс:

1. **Разбивает пространство векторов на кластеры**:
   - выбирается некоторое количество «центров» (centroids);
   - каждый вектор относится к ближайшему центру.
2. При поиске:
   - мы сначала ищем, **к каким центрам ближе всего запрос**;
   - выбираем несколько таких центров;
   - ищем ближайших соседей **только среди векторов внутри этих центров**.

Аналогия:

- у тебя огромный склад с коробками,
- ты строишь «карты проходов» и разбиваешь склад на **зоны** (кластеры),
- когда ищешь что‑то:
  - ты сначала примерно прикидываешь, **в какой зоне** это может быть,
  - а потом обыскиваешь **только эту зону**, а не весь склад.

##### Параметр `lists` — сколько зон на складе

- `lists` = количество кластеров (зон).
- Если `lists` маленькое (например, 10):
  - каждая зона очень большая;
  - поиск быстрый (мало зон, быстро определить, куда смотреть);
  - но внутри зоны много лишних коробок → **низкая точность**.
- Если `lists` большое (например, 10 000):
  - зон много, они маленькие и «тонкие»;
  - нужно больше памяти и времени на построение индекса;
  - поиск более точный, но может стать чуть медленнее из‑за накладных расходов.

Грубо:

- `lists` ≈ **грубина разбиения пространства**.

##### Параметр `probes` — в скольких зонах реально ищем

- `probes` = сколько кластеров мы реально **просматриваем при запросе**.
- Если `probes = 1`:
  - смотрим только в одну зону;
  - очень быстро;
  - можешь пропустить хорошие результаты, если запрос оказался на границе зон.
- Если `probes` ближе к `lists`:
  - мы смотрим почти во все зоны;
  - это похоже на **почти полный перебор**, но с небольшим ускорением;
  - качество близко к идеальному, но скорость снижается.

Так что:

- `lists` — на этапе создания индекса задаёт «сетку» по всему пространству;
- `probes` — на этапе запроса говорит «сколько клеток сетки мы реально обойдем».

##### Как создать индекс `ivfflat`

```sql
-- косинусная метрика
CREATE INDEX documents_embedding_ivfflat_cosine
ON documents
USING ivfflat (embedding vector_cosine_ops)
WITH (lists = 100);
```

Перед использованием:

- индексу нужно «обучение» — при создании он читает данные и строит кластеры;
- **важно**, чтобы таблица уже содержала «достаточно» данных (иначе разбиение будет странным):
  - если данных мало и они распределены неравномерно,
  - некоторые кластеры могут оказаться почти пустыми, а другие переполненными.

Настройка `probes`:

```sql
SET LOCAL ivfflat.probes = 10;

SELECT ...
FROM documents
ORDER BY embedding <=> '[...]'::vector
LIMIT 10;
```

Правило:

- `lists` — **структура** индекса (меняется при пересоздании);
- `probes` — **ручка качества/скорости на уровне запроса**.

#### Индексы `hnsw`

##### Общая идея HNSW

`hnsw` (Hierarchical Navigable Small World) — графовый индекс:

- строится **многоуровневый граф** из всех векторов;
- на верхнем уровне:
  - мало вершин и мало связей,
  - что‑то вроде «карты между крупными районами города»;
- на нижнем уровне:
  - много вершин и связей,
  - что‑то вроде детальной карты кварталов.

Когда ты ищешь ближайших соседей:

1. Начинаешь с **верхнего уровня**:
   - берёшь некую стартовую точку;
   - идёшь по рёбрам туда, где расстояние до запроса уменьшается;
   - как только перестаёшь находить более близких соседей — спускаешься уровнем ниже.
2. На следующем уровне:
   - делаешь то же самое, но уже в более плотном графе;
3. На самом нижнем уровне:
   - обходит относительно небольшое окрестное подмножество векторов;
   - находит кандидатов.

Интуиция:

- ты не пытаешься «обойти весь город»;
- ты сначала грубо перемещаешься по магистралям между районами;
- потом, когда оказался примерно «в нужном районе», уже бродишь по его улицам.

Параметры:

- `m` — **сколько связей хранить на узел**:
  - больше `m` → больше памяти, но больше альтернативных путей → лучше качество;
  - меньше `m` → компактнее, но потенциально хуже находка ближайших.
- `ef_construction` — «усилие» при построении:
  - больше → медленнее построение, лучше структура графа;
  - меньше → быстрее строится, но хуже качество.
- `ef_search` — «глубина» поиска:
  - больше → просматриваем больше кандидатов → выше точность, медленнее;
  - меньше → быстрее, но риск пропустить хорошие ответы.

Создание индекса:

```sql
CREATE INDEX documents_embedding_hnsw_cosine
ON documents
USING hnsw (embedding vector_cosine_ops)
WITH (m = 16, ef_construction = 128);
```

На запросе:

```sql
SET LOCAL hnsw.ef_search = 64;

SELECT id, title, embedding <=> '[...]'::vector AS distance
FROM documents
ORDER BY embedding <=> '[...]'::vector
LIMIT 10;
```

Общая идея:

- `ivfflat` проще, хорошо для относительно «гладких» данных и больших коллекций;
- `hnsw` чаще даёт **лучше качество при сопоставимой скорости**, но:
  - сложнее по памяти (граф нужно хранить),
  - требует чуть более аккуратного тюнинга.

#### Фильтрация по метаданным и гибридный поиск

В PostgreSQL у тебя есть **обычные столбцы** (язык, тип документа, статус) и **вектор**.

Ты можешь:

- сначала **сузить набор по фильтрам**,
- потом **сделать векторный поиск внутри этого подмножества**.

Пример:

```sql
SELECT id, title, language,
       embedding <=> '[...]'::vector AS distance
FROM documents
WHERE language = 'ru'
  AND created_at >= NOW() - interval '30 days'
ORDER BY embedding <=> '[...]'::vector
LIMIT 10;
```

Если фильтры достаточно селективны, PostgreSQL:

- либо использует обычные индексы для отбора подмножества,
- либо комбинирует их с векторным индексом (зависит от версии/плана).

Гибридный поиск:

- можно посчитать:
  - **BM25‑score** (через текстовый индекс/ранжирование),
  - **vector‑score** (через `<=>`),
  - а затем объединить их в одну метрику в приложении.

Очень простой вариант гибридного скоринга:

- нормализуешь оба скоринга в диапазон от 0 до 1;
- считаешь:

  \[
  final\_score = \alpha \cdot vector\_score + (1 - \alpha) \cdot bm25\_score
  \]

- сортируешь по `final_score`.

Где:

- `alpha ≈ 0.5` — равный вклад;
- `alpha ближе к 1` — упор на семантическую близость;
- `alpha ближе к 0` — упор на точные совпадения по словам.

### Примеры

#### Пример 1. Простое хранение и поиск

```sql
CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE faq (
  id        bigserial PRIMARY KEY,
  question  text,
  answer    text,
  embedding vector(384)
);
```

В приложении (псевдо‑Python):

```python
question = "Как настроить резервное копирование?"
q_vec = embed(question)  # вектор длины 384

# Важно: q_vec должен быть массивом из 384 float'ов,
# и модель embed() должна быть той же, которой считали embedding в БД.

cur.execute(
    """
    SELECT id, question, answer,
           embedding <=> %s::vector AS distance
    FROM faq
    ORDER BY embedding <=> %s::vector
    LIMIT 5;
    """,
    [q_vec, q_vec],
)
rows = cur.fetchall()
```

#### Пример 2. Использование `ivfflat` с тюнингом `lists`/`probes`

```sql
CREATE INDEX faq_embedding_ivfflat_cosine
ON faq
USING ivfflat (embedding vector_cosine_ops)
WITH (lists = 200);
```

При поиске:

```sql
SET LOCAL ivfflat.probes = 5;   -- быстрый черновой поиск
-- или
SET LOCAL ivfflat.probes = 40;  -- медленнее, но точнее
```

Ты можешь:

- для **онлайн‑автодополнения** использовать маленькие `probes`;
- для **важных действий** (например, финальный поиск документа для оплаты) — вышее `probes`.

#### Пример 3. HNSW с повышенным качеством

```sql
CREATE INDEX faq_embedding_hnsw_cosine
ON faq
USING hnsw (embedding vector_cosine_ops)
WITH (m = 32, ef_construction = 256);
```

А для критичных запросов:

```sql
SET LOCAL hnsw.ef_search = 128;
```

Такой индекс будет:

- тяжелее по памяти,
- дольше строиться,
- но **очень хорошо попадать в настоящих ближайших соседей**.

### Мини‑курс по pgvector на 5–7 дней (для «совсем с нуля»)

#### День 1. Поставить расширение и поиграться руками

Цель: увидеть живой `pgvector` и понять, что это всего лишь **ещё один тип данных**.

План:

- Убедиться, что у тебя есть PostgreSQL (любой свежий, 14/15/16).
- Установить расширение `pgvector`:
  - либо через пакеты ОС,
  - либо через `CREATE EXTENSION vector;`, если уже установлено.
- В консоли `psql`:
  1. Создать тестовую БД `pgvector_demo`.
  2. Выполнить `CREATE EXTENSION IF NOT EXISTS vector;`.
  3. Создать простую таблицу:

     ```sql
     CREATE TABLE demo_vectors (
       id    serial PRIMARY KEY,
       name  text,
       vec   vector(3)
     );
     ```

  4. Вставить несколько строк:

     ```sql
     INSERT INTO demo_vectors (name, vec) VALUES
       ('A', '[1, 0, 0]'),
       ('B', '[0, 1, 0]'),
       ('C', '[0, 0, 1]');
     ```

  5. Попробовать посчитать расстояния:

     ```sql
     SELECT
       name,
       vec <-> '[1, 0, 0]'::vector AS l2_dist,
       vec <=> '[1, 0, 0]'::vector AS cos_dist
     FROM demo_vectors
     ORDER BY l2_dist;
     ```

Задача дня — **увидеть чиселки расстояния** и понять, что:

- точка A к себе имеет расстояние 0;
- B и C находятся «дальше» по L2;
- косинусное расстояние даёт схожую картину.

#### День 2. Маленькая таблица FAQ + поиск без индексов

Цель: научиться:

- хранить эмбеддинги в реально полезной таблице;
- делать самый простой поиск ближайших соседей.

План:

- Создай таблицу `faq`, как в примере выше.
- Напиши в каком‑нибудь языке (Python удобно) маленький скрипт:
  - берёт список вопросов и ответов (можно руками написать 10–20 штук),
  - вызывает эмбеддер (любой, хоть внешний API, хоть локальная модель),
  - складывает:
    - `question`,
    - `answer`,
    - `embedding`
    в таблицу.
- Напиши простой запрос:

  ```sql
  SELECT id, question, answer,
         embedding <=> :q_vec::vector AS distance
  FROM faq
  ORDER BY embedding <=> :q_vec::vector
  LIMIT 5;
  ```

- Поиграйся с несколькими запросами:
  - «как сделать бэкап»,
  - «как поменять пароль»,
  - «почему база медленно работает».

Смотри, какие ответы попадают в топ.  
Если что‑то кажется странным — это нормально, дальше будем улучшать.

#### День 3. Индекс `ivfflat`: первая версия

Цель: почувствовать, **как индекс ускоряет поиск** и как влияют `lists`/`probes`.

План:

- На той же таблице `faq` (можно увеличить до 1000–10000 записей):
  - измерь время запроса **без индекса** (через `EXPLAIN ANALYZE`).
- Создай индекс:

  ```sql
  CREATE INDEX faq_embedding_ivfflat_cosine
  ON faq
  USING ivfflat (embedding vector_cosine_ops)
  WITH (lists = 100);
  ```

- Снова повтори запрос с `EXPLAIN ANALYZE`, но:
  - сначала с `SET LOCAL ivfflat.probes = 1;`,
  - потом с `probes = 10;`,
  - потом с `probes = 50;`.

Сравни:

- как меняется время выполнения;
- как меняются топ‑5 результатов (на глаз).

Задача дня — **увидеть компромисс**:

- маленький `probes` → быстро, но иногда результаты «прыгают»;
- большой `probes` → медленнее, но стабильнее и точнее.

#### День 4. Индекс HNSW: качественный поиск

Цель: сравнить HNSW с `ivfflat`.

План:

- Создай HNSW‑индекс:

  ```sql
  CREATE INDEX faq_embedding_hnsw_cosine
  ON faq
  USING hnsw (embedding vector_cosine_ops)
  WITH (m = 16, ef_construction = 128);
  ```

- Сделай несколько запусков запроса с разными `hnsw.ef_search`:
  - 16, 32, 64, 128.
- Для каждого:
  - запиши время;
  - посмотри, совпадают ли результаты с **точным** вариантом без индексa (или `probes` с большим значением).

Идея:

- HNSW обычно даёт:
  - более стабильный recall при тех же задержках;
  - но платой за это может быть больший расход памяти.

#### День 5–7. Фильтры и гибридный поиск

Цель: научиться комбинировать:

- векторный поиск,
- обычные SQL‑фильтры,
- и текстовый поиск.

План:

- Добавь к `faq` поля:
  - `language`,
  - `tags` (можно хранить как массив/JSON),
  - `created_at`.
- Попробуй сценарии:
  1. «Найти похожие вопросы **только на русском** за последний месяц».
  2. «Найти похожие вопросы, но **повысить в ранге** те, где встречается слово `backup`».
- Для (1) сделай фильтры в `WHERE`.
- Для (2):
  - добавь текстовый индекс по `question`,
  - вычисли BM25 или хотя бы флаг «есть слово backup»,
  - в приложении смешай скоринги.

После этих шагов `pgvector` перестанет быть «страшным зверем» и станет:

- понятным набором инструментов:
  - тип `vector`,
  - операторы расстояний,
  - два типа индексов,
  - комбинация с обычным SQL.

---

### Картинка в голове

Думай о `pgvector` так:

- **`vector(n)` = «коробка с числовым отпечатком смысла»** — эмбеддинг, который живёт прямо в обычной таблице.
- **`ivfflat` = склад, разбитый на комнаты**:
  - сначала выбираешь пару комнат, где «примерно должно лежать нужное»,
  - потом перебираешь только коробки в этих комнатах.
- **`hnsw` = многоуровневый город с дорогами**:
  - на верхнем уровне — магистрали между крупными районами,
  - на нижнем — мелкие улочки между домами;
  - навигатор спускается от магистралей к нужному дому, не обходя весь город.

Если теряешься в параметрах:

- `lists` — «сколько комнат на складе»;
- `probes` — «в скольких комнатах мы реально ищем»;
- `m` — «сколько дорог от каждого дома к соседям»;
- `ef_*` — «насколько тщательно обходим окрестности».

---

### Типичные ошибки

- **Хранить вектора без индекса и удивляться медленному поиску.**
- Выбирать **не ту метрику** (например, L2 для векторов, которые предполагались для cosine similarity).
- Создавать `ivfflat` на таблице с **очень малым количеством данных** — кластеры будут «кривыми».
- Забывать настроить `probes`/`ef_search` и считать, что «индекс плохой», хотя просто стоит взять немного более тщательный поиск.
- Пытаться запихнуть в PostgreSQL **миллиарды векторов** и ожидать, что он будет вести себя как специализированный векторный движок — Postgres хорош до определённого масштаба.

Из практики:

- иногда проще **хранить id и метаданные в PostgreSQL**, а сами вектора — в специализированной БД (Milvus/Pinecone/Qdrant) и соединять их на уровне приложения.

### Проверь себя

<details>
<summary>1. Чем отличаются операторы <code>&lt;-&gt;</code> и <code>&lt;=&gt;</code> в <code>pgvector</code>?</summary>

- `<->` — L2‑расстояние (евклидово).
- `<=>` — косинусное расстояние.

Оба возвращают «чем меньше, тем лучше», но интуиция и допустимые преобразования (например, нормализация векторов) различаются.

</details>

<details>
<summary>2. Зачем в <code>ivfflat</code> нужны параметры <code>lists</code> и <code>probes</code>?</summary>

- `lists` определяет, **на сколько кластеров разбить пространство** — чем их больше, тем тоньше разбиение, но дороже поддержка.
- `probes` задаёт, **сколько кластеров реально сканировать при запросе** — это «ручка балансировки» между скоростью и качеством.

</details>

<details>
<summary>3. Почему нельзя просто взять <code>hnsw</code> «по умолчанию» и не думать о параметрах?</summary>

Можно запустить с дефолтами, но:

- при слишком маленьких `m` и `ef_*` качество будет хуже, чем могло бы;
- при слишком больших — индекс станет тяжёлым и медленным в построении.

Понимание смысла параметров позволяет адаптировать индекс под твой конкретный объём и требования по задержкам.

</details>
