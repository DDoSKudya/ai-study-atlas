## 176. Weaviate детали

### В этом разделе главное

- Weaviate — это **векторная БД с явной схемой (classes/properties), встроенной векторизацией и GraphQL‑API**.
- Модули (`text2vec-*`, `img2vec-*`) позволяют **автоматически получать вектора** из текста/картинок.
- Поддерживается **гибридный поиск**: векторный + BM25, настраиваемые веса.

---

### Простыми словами

Weaviate — это как:

- «реляционная БД + ORM + векторизация + поиск» в одном флаконе.

Ты:

- описываешь **схему**: классы (`Article`, `Product`, `User`) и свойства;
- отмечаешь, какие свойства нужно векторизовать;
- говоришь: «вот текст — положи мне объект в класс `Article`»;
- Weaviate сам:
  - прогоняет текст через выбранный эмбеддер,
  - сохраняет объект + вектор,
  - даёт тебе GraphQL‑запросы вида:
    - «найди похожие статьи на этот текст»,
    - «с такими‑то фильтрами по полям».

---

### Теория и правила

#### Схема: classes и properties

Пример упрощённой схемы:

```json
{
  "class": "Article",
  "vectorizer": "text2vec-openai",
  "properties": [
    {
      "name": "title",
      "dataType": ["text"]
    },
    {
      "name": "content",
      "dataType": ["text"]
    },
    {
      "name": "language",
      "dataType": ["string"]
    },
    {
      "name": "publishedAt",
      "dataType": ["date"]
    }
  ]
}
```

Ключевые моменты:

- `class` — аналог таблицы;
- `dataType` — тип поля (text, string, int, date и т.п.);
- `vectorizer` — модуль, который будет генерировать вектора.

#### Модули векторизации

Примеры модулей:

- `text2vec-openai` — векторизация текста через OpenAI;
- `text2vec-transformers` — локальные трансформеры;
- `img2vec-*` — векторизация изображений.

Weaviate может:

- **сам вызывать модель** при добавлении/обновлении объекта;
- или принимать **готовый вектор** от клиента (`vector`).

То есть он способен быть и:

- просто «vector store» (ты сам считаешь вектора),
- и «end‑to‑end семантической БД», где он сам векторизует текст/картинки.

#### GraphQL API

Запрос на поиск похожих объектов:

```graphql
{
  Get {
    Article(
      nearText: {
        concepts: ["как настроить резервное копирование"]
      }
      limit: 5
      where: {
        path: ["language"]
        operator: Equal
        valueString: "ru"
      }
    ) {
      title
      language
      _additional {
        distance
      }
    }
  }
}
```

Ключевые поля:

- `nearText` / `nearVector` — способ задать запрос:
  - по тексту (Weaviate сам векторизует),
  - по готовому вектору.
- `where` — фильтр по свойствам (как обычный GraphQL‑фильтр).
- `_additional.distance` — метрика расстояния.

#### Batch operations

Weaviate поддерживает **batch‑вставку**:

- полезно, когда нужно загрузить тысячи/миллионы объектов;
- экономит накладные расходы на запросы.

Важно:

- при большом батче нужно следить за:
  - ограничениями на размер запроса,
  - тайм‑аутами,
  - объёмом векторов (в памяти).

#### Гибридный поиск: вектора + BM25

Weaviate умеет:

- проводить **BM25‑поиск** по текстовым полям;
- совмещать его с **векторным**.

Параметры (может меняться от версии к версии, смотри документацию):

- `alpha` — баланс между векторным и BM25‑скорингом:
  - `alpha ≈ 0` → почти чистый текстовый поиск;
  - `alpha ≈ 1` → почти чистый векторный;
  - середина — смесь.

Интуитивно:

- если у тебя **много коротких и чётких запросов**, BM25 важен;
- если запросы более «семантические» («объясни как…»), векторный поиск важнее.

### Примеры

#### Пример 1. Поиск похожих статей по тексту

```graphql
{
  Get {
    Article(
      nearText: {
        concepts: ["как восстановить базу после сбоя"]
      }
      where: {
        path: ["language"]
        operator: Equal
        valueString: "ru"
      }
      limit: 3
    ) {
      title
      language
      publishedAt
      _additional {
        distance
      }
    }
  }
}
```

#### Пример 2. Гибридный запрос (концепция)

```graphql
{
  Get {
    Article(
      hybrid: {
        query: "backup postgres wal"
        alpha: 0.5
      }
      limit: 5
    ) {
      title
      _additional {
        score  # комбинированный скоринг
      }
    }
  }
}
```

### Картинка в голове

Weaviate — это:

- **«граф объектов», где у каждого есть векторный «отпечаток смысла»**;
- поверх этого графа ты пишешь **GraphQL‑запросы**:
  - «найди похожее на…»,
  - «отфильтруй по языку и дате».

Модули векторизации — это:

- **«встроенные переводчики текста/картинок в вектора»** — тебе не нужно отдельно поднимать модель.

---

### Типичные ошибки

- Игнорировать схему и всё хранить в одном огромном классе «Document» без продуманной структуры — потом тяжело фильтровать и эволюционировать модель.
- Переоценивать встроенную векторизацию и не думать, **какие поля и как именно нужно векторизовать**.
- Не использовать гибридный поиск там, где **ключевые слова всё ещё важны** (например, техническая документация с кодом).
- Неправильно выбирать `alpha` и удивляться «странным» результатам.

### Проверь себя

<details>
<summary>1. Зачем Weaviate вообще нужна схема (classes/properties), если это векторная БД?</summary>

Схема:

- позволяет явно описать структуру данных (какие поля есть у объекта),
- упрощает фильтрацию и эволюцию,
- даёт возможность комбинировать **семантику (вектора)** и **структурную информацию (поля)**.

Без схемы БД превратилась бы в «мешок векторов без смысла».

</details>

<details>
<summary>2. В чём преимущество встроенных модулей векторизации?</summary>

Они позволяют:

- не поднимать и не обслуживать отдельный сервис эмбеддингов,
- снизить количество движущихся частей,
- быстрее прототипировать.

Цена — привязка к конкретным модулям/моделям и ограничения по обновлению.

</details>

<details>
<summary>3. Когда гибридный поиск (векторный + BM25) особенно полезен?</summary>

Когда:

- запросы короткие и содержат **важные ключевые слова**,
- при этом ты хочешь учитывать **семантическую близость** (синонимы, переформулировки).

Например, документация, поиск по тикетам, вопросы/ответы в техподдержке.

</details>
