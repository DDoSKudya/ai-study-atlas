## 175. Pinecone детали

### В этом разделе главное

- Pinecone — **управляемая векторная БД как сервис**: кластеры, шардирование, хранение — забота провайдера.
- Основные сущности: **индексы, namespaces, вектора с метаданными** и **фильтры**.
- Гибридный поиск (векторный + ключевой/BM25) реализуется **на уровне сервиса**, тебе остаётся выбрать стратегию и параметры.

---

### Простыми словами

Pinecone — это как «облачный Elasticsearch, но для векторов».

Ты:

- создаёшь **индекс** нужной размерности (например, 768);
- отправляешь туда вектора (`upsert`);
- делаешь запросы (`query`), указывая:
  - вектор запроса,
  - параметр `topK`,
  - фильтры по метаданным (язык, тип, теги),
  - иногда — настройки гибридного поиска.

Тебе **не нужно**:

- думать про диски, RAM, replica‑set, шардирование;
- настраивать HNSW/IVF вручную (чаще всего) — Pinecone подбирает конфигурацию/варианты сам.

---

### Теория и правила

#### Индексы и namespaces

В Pinecone есть:

- **Index** — основной объект, задающий:
  - размерность векторов,
  - тип метрики (cosine/dot/L2),
  - конфигурацию производительности.
- **Namespace** — логический «под‑индекс» внутри индекса:
  - например, `namespace="ru"` и `namespace="en"`;
  - или `namespace="tenant_123"`.

Зачем namespaces:

- изоляция данных (по проектам/клиентам/языкам);
- логичная группировка в рамках одного физического индекса.

#### Модель данных: ids, values, metadata

Вектор в Pinecone — это:

- `id` — строковый идентификатор (например, `doc-123`);
- `values` — сам вектор (`[float, float, ...]`);
- `metadata` — словарь с полями:
  - `{"language": "ru", "type": "faq", "category": "billing", ...}`.

Операция `upsert`:

```json
{
  "vectors": [
    {
      "id": "doc-123",
      "values": [0.1, 0.2, ...],
      "metadata": {
        "language": "ru",
        "type": "faq",
        "category": "billing"
      }
    }
  ],
  "namespace": "default"
}
```

`metadata` потом используется для фильтрации.

#### Metadata filtering

Фильтры строятся в виде JSON‑условий:

```json
{
  "filter": {
    "language": "ru",
    "category": { "$in": ["billing", "payments"] },
    "created_at": { "$gte": 1700000000 }
  }
}
```

Внутри движка:

- фильтрация может выполняться:
  - **до** векторного поиска (pre‑filter),
  - **после** (post‑filter),
  - или гибридно — в зависимости от плотности/индексов по метаданным.

Тебе важно понимать:

- чем более селективен фильтр — тем меньше векторов надо просматривать;
- при слишком узких/сложных фильтрах качество может немного просесть, если движок применяет их до поиска.

#### Hybrid search и RRF

Hybrid search в Pinecone:

- обычно это **комбинация векторного скоринга и текстового** (BM25/keyword);
- ты можешь:
  - или дать сразу «гибридный» запрос,
  - или сам посчитать два списка и объединить их.

Распространённая стратегия — **RRF (Reciprocal Rank Fusion)**:

- для каждого документа считается:

  \[
  score = \sum\_q \frac{1}{k + rank\_q}
  \]

  где:
  - `q` — источник (векторный поиск, BM25),
  - `rank_q` — позиция документа в списке этого источника,
  - `k` — сглаживающий параметр (обычно около 60).

Интуитивно:

- документ, который везде в топе, получает высокий score;
- документ, который высоко только в одном источнике, но далеко в другом, получает средний score.

### Примеры

Ниже — концептуальные примеры (синтаксис может немного отличаться от актуального SDK, смотри документацию Pinecone).

#### Пример 1. Базовый upsert и поиск

```python
from pinecone import Pinecone

pc = Pinecone(api_key="...")
index = pc.Index("my-index")

vec = embed("Как сменить тариф?")

index.upsert(
    vectors=[
        {
            "id": "faq-1",
            "values": vec,
            "metadata": {
                "language": "ru",
                "type": "faq",
                "category": "billing",
            },
        }
    ]
)

q_vec = embed("Как оплатить подписку?")

result = index.query(
    vector=q_vec,
    top_k=5,
    include_metadata=True,
    filter={"language": "ru", "type": "faq"}
)
```

#### Пример 2. Разделение по namespaces

```python
index.upsert(
    vectors=[...],
    namespace="ru"
)

index.upsert(
    vectors=[...],
    namespace="en"
)

result_ru = index.query(vector=q_vec_ru, top_k=5, namespace="ru")
result_en = index.query(vector=q_vec_en, top_k=5, namespace="en")
```

Так ты можешь:

- разделить языки/клиентов/проекты по namespace,
- ограничивать запросы только нужным подмножеством.

#### Пример 3. Гибридный подход с RRF в приложении

1. Делаешь **векторный поиск** в Pinecone (по эмбеддингам).
2. Делаешь **текстовый поиск** (например, по BM25 в Elastic/Postgres).
3. В приложении сливаешь результаты RRF‑формулой.

Это даёт:

- устойчивость к шуму в эмбеддингах/тексте;
- контроль баланса «по смыслу» vs «по точным словам».

---

### Картинка в голове

Pinecone — это:

- **«огромный шкаф с векторами»**, за которым следит чужой SRE‑отдел;
- ты просто говоришь: «положи сюда эти коробки с метками» и «найди мне похожие коробки с такими‑то свойствами».

Namespaces — это:

- **под‑шкафы** внутри этого шкафа:
  - справа полица `ru`,
  - слева `en`,
  - внизу `staging`.

Фильтры по метаданным — это:

- «ищи только в этих ящиках/полицах», а не по всему складу.

---

### Типичные ошибки

- Относиться к Pinecone как к «магическому ящику», не понимая, **какую метрику** он использует и какие есть ограничения по размеру/задержкам.
- Хранить слишком подробные/случайные метаданные и делать по ним сложные фильтры, ожидая, что поиск будет таким же быстрым, как без фильтров.
- Не думать про **разделение по namespaces** и получать «свалку всего подряд» в одном индексе.
- Пытаться класть в Pinecone **все типы данных без разбора**, не выбирая, какие сущности действительно нуждаются в семантическом поиске.

### Проверь себя

<details>
<summary>1. Зачем нужны <code>namespaces</code> в Pinecone?</summary>

Чтобы логически разделять данные внутри одного индекса (по клиентам, языкам, проектам), не создавая множество физических индексов.  
Это удобно для мультитенантности, A/B‑экспериментов и раздельных окружений (prod/stage/test).

</details>

<details>
<summary>2. В чём интуитивный смысл гибридного поиска и RRF?</summary>

Гибридный поиск позволяет:

- учитывать **и смысл (вектора)**, и **точные слова/формулировки (BM25/keyword)**;
- RRF объединяет несколько ранжированных списков так, что документы, стабильно высокие в разных сигналах, оказываются выше всего.

</details>

<details>
<summary>3. Почему важно хранить в метаданных только полезную для фильтрации информацию?</summary>

Потому что:

- сложные/малоиспользуемые поля увеличивают размер индексов/метаданных;
- неудачные фильтры могут сильно замедлить поиск;
- «мусорные» поля затрудняют понимание и эволюцию схемы.

</details>
