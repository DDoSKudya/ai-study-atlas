## 35. Мониторинг использования индексов

### 35.1. Статистика использования

**Цель раздела.**  
Использовать **pg_stat_user_indexes** для просмотра **idx_scan** (сколько раз индекс использовался для сканирования), **idx_tup_read** (сколько записей индекса прочитано), **idx_tup_fetch** (сколько строк таблицы получено по индексу). Сброс счётчиков при перезапуске или при pg_stat_reset().

#### Термины

- **pg_stat_user_indexes** — представление в PostgreSQL: для каждого пользовательского индекса — relname (таблица), indexrelname (имя индекса), **idx_scan** (число сканирований индекса — index scan или bitmap index scan), **idx_tup_read** (число записей, прочитанных из индекса), **idx_tup_fetch** (число строк таблицы, полученных по индексу).
- **idx_scan** — главный показатель «используемости» индекса: сколько раз планировщик выбрал этот индекс для сканирования.

#### Теория и правила

- Запрос к pg_stat_user_indexes даёт по каждому индексу счётчики с момента последнего сброса (или с запуска кластера). Сортировка по idx_scan ASC выводит **наименее** используемые индексы первыми — кандидаты на анализ (удалить или оставить).
- **pg_stat_reset()** сбрасывает все счётчики статистики; для анализа «за период» можно сбрасывать в начале периода и смотреть через время.

#### Пример запроса

```sql
SELECT schemaname, relname AS table_name, indexrelname AS index_name,
       idx_scan, idx_tup_read, idx_tup_fetch,
       pg_size_pretty(pg_relation_size(indexrelid)) AS index_size
FROM pg_stat_user_indexes
ORDER BY idx_scan ASC, pg_relation_size(indexrelid) DESC;
```

#### Простыми словами

**pg_stat_user_indexes** — «сколько раз каждый индекс использовался» (idx_scan) и сколько записей через него прочитано. По нему видно, какие индексы «работают», а какие нет.

#### Картинка в голове

**pg_stat_user_indexes** — как **счётчик посещений у каждого указателя в книге**: «Указатель по фамилиям — открывали 10 000 раз. Указатель по датам — 500 раз. Указатель по цвету рубашки — 0 раз». По idx_scan видно, **какими указателями пользуются**, а какими нет. Нулевой счётчик за долгий период = «этим указателем никто не пользовался» — он только утяжеляет книгу и его переписывают при каждом изменении. Но перед тем как вырвать страницу с «нулевым» указателем, нужно убедиться: а не заходят ли по нему раз в год (редкий отчёт)? Сортировка по idx_scan по возрастанию выводит **самые неиспользуемые** индексы первыми — с них и начинают разбор.

#### Как запомнить

**pg_stat_user_indexes**: idx_scan = число использований индекса; idx_tup_read, idx_tup_fetch. Сортировка по idx_scan ASC — неиспользуемые первыми.

#### Проверь себя (35.1)

1. Что означает idx_scan в pg_stat_user_indexes и почему по нему судят об «используемости» индекса?  
<details><summary>Ответ</summary> **idx_scan** — число раз, когда планировщик **выбрал этот индекс** для сканирования (index scan или bitmap index scan). Если idx_scan = 0 за длительный период — индекс ни разу не использовался для чтения; он только замедляет запись и занимает место. Поэтому idx_scan — главный показатель «нужности» индекса для запросов.</details>

2. Счётчики в pg_stat_user_indexes сбрасываются при перезапуске кластера. Как оценить использование индексов «за последнюю неделю»?  
<details><summary>Ответ</summary> В начале периода выполнить **pg_stat_reset()** — сбросить все счётчики. Через неделю посмотреть pg_stat_user_indexes — значения будут за эту неделю. Либо не сбрасывать и смотреть разницу за два момента времени (сохранить снимок в таблицу, потом сравнить). Либо использовать расширение **pg_stat_statements** и анализировать, какие запросы выполнялись и по каким таблицам/индексам (косвенно).</details>

3. Что означают idx_tup_read и idx_tup_fetch и чем они отличаются от idx_scan?  
<details><summary>Ответ</summary> **idx_scan** — **сколько раз** планировщик выбрал этот индекс для сканирования (один запрос = один сканирование, даже если по индексу прочитали миллион записей). **idx_tup_read** — сколько **записей индекса** (пар ключ–TID) было прочитано за все эти сканирования. **idx_tup_fetch** — сколько **строк таблицы** (heap) было получено по TID из индекса. Разница: idx_tup_read может быть больше idx_tup_fetch (например, часть записей отфильтровалась по видимости или по условию). По idx_scan судят «используется ли индекс»; по idx_tup_read/idx_tup_fetch — объём данных, прошедших через индекс.</details>

**Три пункта, чтобы не забыть (pg_stat_user_indexes).** 1) **idx_scan** — главный показатель: **сколько раз** индекс был использован для сканирования; idx_scan = 0 долгое время → индекс не используется, кандидат на удаление. 2) **idx_tup_read** — сколько записей индекса прочитано; **idx_tup_fetch** — сколько строк таблицы получено по индексу. 3) Счётчики сбрасываются при **перезапуске** и при **pg_stat_reset()**; для выводов нужен период наблюдения (неделя, месяц).

**Ещё раз самыми простыми словами:** pg_stat_user_indexes показывает «сколько раз каждый индекс реально использовался» (idx_scan). Индексы с нулём или очень малым idx_scan — кандидаты на удаление; но перед удалением проверить редкие запросы.

#### Запомните

- **pg_stat_user_indexes** — idx_scan (число использований), idx_tup_read, idx_tup_fetch; по idx_scan оценивают «используемость» индекса.

### 35.2. Неиспользуемые индексы

**Цель раздела.**  
Находить индексы с **idx_scan = 0** (или очень малым) за период наблюдения — кандидаты на удаление для ускорения записи и освобождения места. **Осторожность:** редкие запросы (раз в месяц) тоже могут быть критичны — перед удалением проверить, нет ли запросов, которые должны использовать этот индекс. После раздела ты будешь знать, как искать неиспользуемые индексы и почему idx_scan = 0 ещё не значит «смело удаляй».

#### Термины (расшифровка)

- **Неиспользуемый индекс (unused index)** — индекс, у которого **idx_scan = 0** за длительный период (с последнего сброса статистики или с рестарта кластера). Это значит: планировщик **ни разу** не выбрал этот индекс для сканирования. Такой индекс **не ускоряет** ни один запрос (по нему не ищут), но при **каждой** INSERT/UPDATE/DELETE по таблице СУБД обновляет и его — затраты времени и WAL. Плюс индекс занимает место на диске. **Кандидат на удаление** — но только после проверки: не обслуживает ли он редкие, но важные запросы.
- **Редкие запросы** — запросы, которые выполняются редко (раз в неделю, раз в месяц, раз в квартал): отчёты, админские операции, выгрузки. Если индекс создан **под такой** запрос, idx_scan за месяц может быть 0 или 1–2. Удаление индекса **замедлит** этот редкий запрос; если он критичен для бизнеса, индекс лучше оставить. Перед удалением «неиспользуемого» индекса нужно убедиться: нет ли в приложении или в скриптах запросов с WHERE/ORDER BY по столбцам этого индекса.

#### Теория и правила (подробно)

- **Как искать:** запрос к **pg_stat_user_indexes**, отбор по idx_scan = 0 (или очень малому значению); сортировка по размеру индекса (pg_relation_size) — большие неиспользуемые индексы дают больший выигрыш при удалении. **Исключить** из списка на удаление: PRIMARY KEY и UNIQUE, если они нужны для целостности (ограничения уникальности, внешние ключи). Для остальных — проверить логику приложения, логи запросов, pg_stat_statements (какие запросы выполнялись и по каким столбцам).
- **Удаление:** **DROP INDEX** имя_индекса; на продакшене для таблицы с активной записью — **DROP INDEX CONCURRENTLY**, чтобы не блокировать запись. После удаления вставки/обновления станут быстрее (на один индекс меньше), место освободится.
- **Избыточность:** индекс **(a)** при наличии **(a, b)** часто неиспользуем «по статистике», потому что запросы по a обслуживаются индексом (a, b). Отдельный (a) тогда **избыточен** — его удаление не ломает запросы и даёт выигрыш по записи и месту.

> **Важно.** idx_scan = 0 за месяц **не равно** «индекс никому не нужен». Возможны редкие запросы (отчёт раз в квартал, админка раз в месяц), которые этот индекс используют; за месяц они могли не выполниться. Всегда перед DROP проверять: есть ли в коде или в отчётах запросы по столбцам этого индекса; спросить у разработчиков; при сомнении — оставить или отложить удаление до следующего периода наблюдения.

#### Пошагово: как проверить индекс перед удалением (практика)

Индекс idx_orders_region по столбцу region; idx_scan = 0 за три месяца.

1. **Проверить назначение:** для чего индекс создавался? (по имени или по комментариям — «для отчёта по регионам».)
2. **Поискать запросы:** в коде приложения, в хранимых процедурах, в скриптах отчётов — есть ли WHERE region = ? или ORDER BY region? Если есть и запрос выполняется редко — индекс может быть нужен; idx_scan = 0 из‑за того, что за три месяца отчёт не запускали.
3. **Проверить избыточность:** есть ли индекс (region, created_at) или (region, status)? Тогда запросы только по region могут идти по нему; отдельный (region) избыточен и его удаление безопасно.
4. **Решить:** если запросов по region нет и индекс не нужен для целостности — **DROP INDEX CONCURRENTLY** idx_orders_region. Если есть редкий критичный запрос — оставить.

#### Простыми словами

**Неиспользуемый индекс** — idx_scan = 0 долгое время. Кандидат на удаление — но перед удалением проверить, нет ли редких важных запросов по этому индексу.

#### Как запомнить

**idx_scan = 0** за период → кандидат на удаление. Перед DROP проверить редкие запросы и избыточность (подмножество другого индекса). **Не удалять сходу** — редкие отчёты могут использовать индекс.

#### Картинка в голове

**Неиспользуемый индекс** — как **указатель в книге по разделу «погода в июле 2020»**, которым никто не пользуется: книгу всё равно обновляют (вставки в таблицу), и указатель при каждом обновлении переписывают (обслуживание индекса), но по нему никто не ищет. Удалить такой указатель — книга станет легче и обновлять её будет быстрее. Но перед тем как вырвать страницу с указателем, нужно убедиться: а не заходил ли кто раз в год именно по этому указателю (редкий отчёт)? Если да — лучше не трогать.

#### Проверь себя (35.2)

1. Индекс имеет idx_scan = 0 за месяц. Можно ли его смело удалить?  
<details><summary>Ответ</summary> **Не обязательно.** Могут быть **редкие** запросы (раз в квартал отчёт, раз в месяц админка), которые этот индекс используют; за месяц они могли не выполниться или выполниться до сброса статистики. Перед удалением: проверить логи запросов, pg_stat_statements, спросить у разработчиков; убедиться, что нет критичных запросов по этому столбцу. Также проверить **избыточность**: если есть индекс (a, b), то отдельный (a) может быть неиспользуемым «по статистике», но создан под редкий запрос — тогда решение об удалении (a) принимают с учётом того, что (a, b) покрывает запросы по a.</details>

2. Как на практике проверить, что индекс с idx_scan = 0 не нужен редкому отчёту?  
<details><summary>Ответ</summary> Поискать в коде приложения, в скриптах и в отчётах (BI, выгрузки) все запросы к этой таблице, в которых есть **WHERE** или **ORDER BY** по столбцам этого индекса. Если такие запросы есть и они выполняются (хоть раз в квартал) — индекс может им помогать; за месяц idx_scan мог остаться 0. Дополнительно: использовать **pg_stat_statements** (если установлено) и смотреть, какие запросы к этой таблице выполнялись и по каким столбцам шли условия. При сомнении — продлить период наблюдения или оставить индекс.</details>

3. Почему при удалении неиспользуемого индекса на живой таблице лучше использовать DROP INDEX CONCURRENTLY?  
<details><summary>Ответ</summary> **DROP INDEX** без CONCURRENTLY берёт **эксклюзивную блокировку** на таблицу на время удаления индекса: вставки, обновления и удаления по таблице **ждут**. На большой таблице или при длительном DROP это может вызвать таймауты и очереди. **DROP INDEX CONCURRENTLY** выполняет удаление **без** такой блокировки — другие сессии продолжают писать в таблицу. На продакшене для неиспользуемых индексов предпочтительнее DROP INDEX CONCURRENTLY.</details>

**Три пункта, чтобы не забыть (неиспользуемые индексы).** 1) **Неиспользуемый** = idx_scan = 0 за длительный период; индекс не ускоряет запросы, но **замедляет каждую** вставку/обновление/удаление и занимает место — кандидат на удаление. 2) **Перед удалением** проверить: нет ли **редких** запросов (отчёты, админка), использующих этот индекс; idx_scan = 0 за месяц не значит «точно не нужен». 3) На живой таблице удалять через **DROP INDEX CONCURRENTLY**, чтобы не блокировать запись.

**Ещё раз самыми простыми словами:** неиспользуемый индекс — тот, по которому за период ни разу не искали (idx_scan = 0). Его можно удалить, чтобы не тратить место и не тормозить вставки, но только убедившись, что нет важных редких запросов, которые по нему должны идти.

#### Запомните

- **Неиспользуемые индексы** — idx_scan = 0 за период; кандидаты на удаление после проверки (редкие запросы, избыточность).

### 35.3. Избыточные и дублирующие индексы

**Цель раздела.**  
Находить **избыточные** индексы (индекс — подмножество другого, например (a) при (a, b)) и **дублирующие** (два индекса с одинаковым определением). Удаление освобождает место и ускоряет запись. После раздела ты будешь уверенно отличать избыточный индекс от нужного и знать, когда можно безопасно удалить один из двух «одинаковых».

#### Термины (расшифровка)

- **Избыточный индекс (redundant index)** — индекс, который является **левым префиксом** другого: например, индекс **(a)** при наличии **(a, b)**. В B-tree индекс (a, b) упорядочен сначала по a, потом по b; запросы с условием **только по a** (WHERE a = ?) могут использовать этот индекс — по правилу левого префикса. Поэтому отдельный индекс **(a)** не даёт **нового** покрытия: всё, что ищут по a, уже обслуживается (a, b). Отдельный (a) только **дублирует** «левую часть» и создаёт лишние затраты при каждой вставке/обновлении и занимает место. **Избыточен** — можно удалить (a), запросы по a продолжат использовать (a, b).
- **Дублирующий индекс (duplicate index)** — два индекса с **полностью одинаковым** определением: те же столбцы, тот же порядок (ASC/DESC), тот же тип (B-tree и т.д.). Они делают одно и то же; один из них **лишний** (часто создан по ошибке или после миграции). Удалить один — запросы не пострадают, место и нагрузка на запись уменьшатся.

#### Теория и правила (подробно)

- Проверка вручную или скриптами: сравнить определения индексов (pg_indexes, pg_index). (a) избыточен при (a, b) или (a) INCLUDE (...); (a, b) избыточен при (a, b, c) и т.д. Дубликаты — полное совпадение определения.
- Удаление избыточного/дублирующего: DROP INDEX (или CONCURRENTLY); после удаления запросы по-прежнему обслуживаются оставшимся индексом.

#### Картинка в голове

**Избыточный индекс** — как два указателя в книге: один «Фамилия → страница», другой «Фамилия + Имя → страница». По первому указателю ищут только по фамилии; но то же самое можно сделать по второму (открыл раздел по фамилии и там уже фамилия+имя). Второй указатель «включает» первый. Значит, первый указатель **лишний** — его выкинули, по фамилии всё равно ищем по второму. **Дублирующий** — два **совершенно одинаковых** указателя (скопировали по ошибке). Один удалили — ничего не изменилось, только книга легче.

#### Простыми словами

**Избыточный** — «короткий» индекс при наличии «длинного»: (a) при (a, b). **Дублирующий** — два одинаковых индекса. Удалить лишний — меньше записи и места.

#### Как запомнить

**Избыточный** = подмножество другого (a при (a,b)). **Дублирующий** = одинаковое определение. Удалить лишний.

#### Проверь себя (35.3)

1. Есть индексы (user_id) и (user_id, created_at). Почему (user_id) считают избыточным и можно ли его удалить?  
<details><summary>Ответ</summary> Запросы с условием **только по user_id** (WHERE user_id = ?) могут использовать индекс **(user_id, created_at)** — по правилу **левого префикса** B-tree индекс (user_id, created_at) подходит для поиска по user_id (первый столбец ключа). Поэтому отдельный индекс **(user_id)** не даёт нового покрытия запросов — он **избыточен**. Его можно удалить: запросы по user_id продолжат использовать (user_id, created_at). Выигрыш: меньше места, меньше обновлений при вставке/обновлении (один индекс вместо двух).</details>

2. Два индекса с одинаковым определением (одни и те же столбцы, порядок, тип). Как поступить?  
<details><summary>Ответ</summary> Это **дублирующие** индексы; один из них лишний. Оставить один (например, первый созданный или с осмысленным именем), второй **удалить** (DROP INDEX или DROP INDEX CONCURRENTLY). Запросы не пострадают — оставшийся индекс полностью заменяет удалённый. Освободится место и уменьшится нагрузка на запись (каждая вставка/обновление обновляет на один индекс меньше).</details>

3. Есть индексы (a), (a, b) и (a, b, c). Какие из них избыточны при наличии остальных?  
<details><summary>Ответ</summary> **Избыточны (a)** и **(a, b)** при наличии **(a, b, c)**. По правилу левого префикса индекс (a, b, c) подходит для запросов по a, по a и b, и по a, b и c. Поэтому отдельный индекс (a) не даёт нового покрытия — избыточен; отдельный (a, b) тоже избыточен. Оставить имеет смысл только **(a, b, c)** (или при особых причинах один из «коротких»). Удаление (a) и (a, b) освободит место и ускорит запись.</details>

**Три пункта, чтобы не забыть (избыточные и дублирующие).** 1) **Избыточный** индекс = **подмножество** другого: (a) при наличии (a, b); запросы по a уже обслуживаются (a, b) по правилу левого префикса — отдельный (a) можно удалить. 2) **Дублирующий** = два индекса с **одинаковым** определением (столбцы, порядок, тип); один лишний — удалить. 3) Удаление лишнего индекса **не ломает** запросы (оставшийся покрывает те же условия), зато **меньше места** и **быстрее запись**.

**Ещё раз самыми простыми словами:** избыточный индекс — когда «короткий» индекс (a) уже покрыт «длинным» (a, b): по (a) и так можно искать через (a, b). Дублирующий — два индекса с одинаковым набором столбцов. В обоих случаях лишний индекс можно удалить — запросы не пострадают, запись станет быстрее.

#### Запомните

- **Избыточный индекс** — подмножество другого (левый префикс); **дублирующий** — одинаковое определение. Удаление освобождает место и ускоряет запись.
