## Справочник по части VI

| Тема | Ключевые пункты |
|------|------------------|
| **Назначение индексов** | Ускорение WHERE, ORDER BY, GROUP BY, JOIN; замедление INSERT/UPDATE/DELETE; когда не индексировать (маленькие таблицы, низкая селективность, столбец не в условиях); баланс числа индексов; pg_stat_user_indexes.idx_scan. |
| **B-tree** | Сбалансированное дерево; листья на одном уровне; поиск O(log N); диапазоны и ORDER BY по ключу; составной индекс — правило левого префикса; INCLUDE — покрывающий индекс; UNIQUE; ASC/DESC. |
| **Hash, GiST, GIN, BRIN** | Hash — только равенство; GiST — диапазоны, геометрия, полнотекст; GIN — инвертированный (массивы, JSONB, полнотекст); BRIN — min/max по блокам, высокая корреляция. Пространственные (PostGIS): GiST vs SP-GiST (точки — SP-GiST, полигоны/линии — GiST). Полнотекст (tsvector). |
| **Частичные и по выражениям** | Частичный индекс — WHERE в определении; уникальный частичный; индекс по выражению (LOWER, выражение) — запрос с тем же выражением; immutable. |
| **Создание и обслуживание** | CREATE INDEX CONCURRENTLY — без блокировки записи; после CREATE INDEX — ANALYZE таблицы; REINDEX CONCURRENTLY — пошагово (новый индекс, подмена, при сбое — удалить лишний по _ccnew/_ccold); fillfactor индекса; bloat; CLUSTER; партиционирование — локальные индексы. |
| **Сканирование** | Bitmap Index Scan + Bitmap Heap Scan — много строк, чтение страниц по порядку; Index-only scan — покрывающий индекс + visibility map; корреляция — pg_stats; пример плана (EXPLAIN): Index Scan, Bitmap Scan, Index Only Scan. |
| **Мониторинг** | pg_stat_user_indexes: idx_scan, idx_tup_read, idx_tup_fetch; неиспользуемые (idx_scan = 0); избыточные (подмножество другого); дублирующие (одинаковое определение). |

### Частые сценарии: задача → решение

| Тема | Задача | Решение |
|------|--------|---------|
| **Поиск** | Ускорить SELECT по столбцу | Индекс по этому столбцу (B-tree по умолчанию); для равенства можно Hash (PG 10+). |
| **Диапазон** | Ускорить WHERE created_at BETWEEN ... AND ... | B-tree по created_at. |
| **Составное условие** | WHERE status = 'active' AND created_at > ... | Составной B-tree (status, created_at) или (created_at, status) в зависимости от селективности и ORDER BY. |
| **Сортировка** | Ускорить ORDER BY created_at DESC | B-tree по created_at; тот же индекс подходит для WHERE + ORDER BY. |
| **Без обращения к таблице** | Уменьшить I/O — все нужные столбцы в индексе | Покрывающий индекс: INCLUDE (col1, col2) в PostgreSQL; index-only scan. |
| **Часть строк** | Индекс только для активных записей | Частичный индекс: CREATE INDEX ... WHERE status = 'active'. |
| **Регистронезависимый поиск** | WHERE LOWER(email) = LOWER(?) | Индекс по выражению: CREATE INDEX ... ON t (LOWER(email)). |
| **Массивы/JSONB** | Поиск по элементам массива или ключам JSONB | GIN-индекс по столбцу. |
| **Полнотекст** | Поиск по словам в тексте | GIN (или GiST) по tsvector. |
| **Данные по времени** | Большая таблица, упорядоченная по времени | BRIN по столбцу времени; минимальный размер индекса. |
| **Без блокировки** | Создать индекс на живой таблице | CREATE INDEX CONCURRENTLY; дольше, но запись не блокируется. |
| **Индекс раздут** | После массовых удалений/обновлений индекс большой и медленный | REINDEX или REINDEX CONCURRENTLY; при необходимости CLUSTER по индексу. |
| **Не используется** | Проверить, нужен ли индекс | pg_stat_user_indexes.idx_scan = 0 за длительный период; осторожно с редкими запросами. |

## Вопросы для самопроверки (часть VI)

1. Зачем нужны индексы и что они замедляют?  
<details><summary>Ответ</summary> Индексы **ускоряют** поиск по условию (WHERE), сортировку (ORDER BY по ключу), группировку (GROUP BY по ключу) и JOIN по проиндексированному столбцу. **Замедляют** INSERT, UPDATE и DELETE — каждое изменение обновляет все индексы таблицы (кроме HOT-обновлений в PostgreSQL).</details>

2. Что такое правило левого префикса для составного индекса (a, b, c)?  
<details><summary>Ответ</summary> Индекс (a, b, c) может использоваться для условий по **a**; по **a AND b**; по **a AND b AND c**. Он **не** может эффективно использоваться для условия только по **b** или только по **c** (или b AND c без a), потому что в дереве записи упорядочены сначала по a, потом по b, потом по c — нет порядка «только по b».</details>

3. Чем покрывающий индекс отличается от обычного и зачем INCLUDE?  
<details><summary>Ответ</summary> **Покрывающий индекс** содержит **все** столбцы, нужные запросу — возможен **index-only scan** (чтение только из индекса, без heap). **INCLUDE** в PostgreSQL добавляет столбцы в **листья** индекса без включения в ключ B-tree — не раздувает ключ, но позволяет «покрыть» запрос и не читать таблицу.</details>

4. Когда использовать Hash, GiST, GIN, BRIN?  
<details><summary>Ответ</summary> **Hash** — только WHERE col = constant. **GiST** — диапазоны (range), геометрия (PostGIS), полнотекст (tsvector). **GIN** — массивы (@>, &&), JSONB (@>, ?), полнотекст (tsvector). **BRIN** — большие таблицы, упорядоченные по столбцу (например, время); минимальный размер индекса.</details>

5. Зачем CREATE INDEX CONCURRENTLY и что делать при сбое?  
<details><summary>Ответ</summary> **CONCURRENTLY** — создание индекса **без эксклюзивной блокировки** таблицы; запись не блокируется. При сбое может остаться **invalid** индекс (pg_index.indisvalid = false). Нужно **DROP INDEX** (или DROP INDEX CONCURRENTLY) и при необходимости пересоздать.</details>

6. Что такое bloat индекса и как его уменьшить?  
<details><summary>Ответ</summary> **Bloat** — раздувание индекса из-за мёртвых записей и фрагментации после массовых UPDATE/DELETE. **VACUUM не сжимает** страницы индекса. **Уменьшить** — **REINDEX** или **REINDEX CONCURRENTLY** (перестроение индекса).</details>

7. Когда возможен index-only scan и что такое Heap Fetches?  
<details><summary>Ответ</summary> **Index-only scan** возможен, когда все нужные столбцы в индексе (покрывающий индекс) и для затронутых страниц таблицы в **visibility map** помечено «все строки видимы». Иначе СУБД обращается к heap для проверки видимости — это **Heap Fetches**. Heap Fetches > 0 — часть данных прочитана из таблицы.</details>

8. Как найти неиспользуемые и избыточные индексы?  
<details><summary>Ответ</summary> **Неиспользуемые:** запрос к pg_stat_user_indexes, idx_scan = 0 (или мал) за период; перед удалением проверить редкие запросы. **Избыточные:** индекс (a) при наличии (a, b) — запросы по a обслуживаются (a, b); (a) избыточен. **Дублирующие:** два индекса с одинаковым определением — удалить один.</details>

## Резюме части VI

- **Назначение индексов:** ускорение WHERE, ORDER BY, GROUP BY, JOIN; замедление DML; баланс числа индексов; мониторинг idx_scan.
- **B-tree:** сбалансированное дерево; диапазоны и сортировка; составные ключи и правило левого префикса; INCLUDE (покрывающий индекс); UNIQUE; ASC/DESC.
- **Специализированные индексы:** Hash (равенство); GiST (диапазоны, геометрия, полнотекст); GIN (массивы, JSONB, полнотекст); BRIN (min/max по блокам, высокая корреляция). Пространственные (PostGIS): GiST vs SP-GiST (точки — SP-GiST). Полнотекст (tsvector).
- **Частичный индекс и по выражению:** WHERE в определении; UNIQUE + WHERE; индекс на выражение (LOWER и т.д.); запрос с тем же выражением; immutable.
- **Создание и обслуживание:** CREATE INDEX CONCURRENTLY; после CREATE INDEX — ANALYZE таблицы; REINDEX CONCURRENTLY (пошагово, при сбое — удалить лишний индекс _ccnew/_ccold); fillfactor индекса; bloat; CLUSTER; локальные индексы при партиционировании.
- **Сканирование:** Bitmap Index Scan + Bitmap Heap Scan; Index-only scan (покрывающий индекс + visibility map); корреляция; пример плана (EXPLAIN) — узлы Index Scan, Bitmap Scan, Index Only Scan.
- **Мониторинг:** pg_stat_user_indexes (idx_scan); неиспользуемые, избыточные и дублирующие индексы.

**Связь с частью VII (Оптимизация запросов):** понимание индексов (B-tree, составные, покрывающие, типы сканирования) нужно для чтения планов (EXPLAIN): Index Scan, Bitmap Index Scan, Index Only Scan, выбор планировщика между seq scan и index scan по стоимости и статистике.
