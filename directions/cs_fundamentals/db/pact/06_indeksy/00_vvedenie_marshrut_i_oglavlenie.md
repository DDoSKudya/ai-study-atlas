## Часть VI. Индексы

> Цель этой части — **понять, зачем нужны индексы, как они устроены и когда их использовать**: ускорение поиска, сортировки и группировки; замедление вставок и обновлений; B-tree как основной тип; Hash, GiST, GIN, BRIN для специальных задач; составные и покрывающие индексы; частичные индексы и индексы по выражениям; создание и перестроение без блокировок; bitmap scan и index-only scan; мониторинг использования и выявление лишних индексов.

После этой части ты сможешь:

- **понимать назначение индексов** — ускорение WHERE, ORDER BY, GROUP BY, JOIN; замедление INSERT/UPDATE/DELETE; когда индексировать и когда нет;
- **разбираться в B-tree** — сбалансированное дерево, листья, диапазоны, составные ключи, порядок столбцов, INCLUDE (покрывающий индекс);
- **знать специализированные индексы** — Hash (равенство), GiST (диапазоны, геометрия, полнотекст), GIN (массивы, JSONB, полнотекст), BRIN (минимальный размер, данные по порядку);
- **применять частичные индексы и индексы по выражениям** — WHERE в индексе, функциональные индексы (LOWER, выражения);
- **создавать и обслуживать индексы** — CREATE INDEX CONCURRENTLY, REINDEX CONCURRENTLY; bloat, CLUSTER;
- **понимать bitmap index scan и index-only scan** — когда планировщик их выбирает; корреляция физического и логического порядка;
- **мониторить использование индексов** — неиспользуемые и избыточные индексы, pg_stat_user_indexes.

Материал опирается на **Часть V (хранение, heap, TID, visibility map)** и готовит к **Части VII (оптимизация запросов, EXPLAIN)**.

Контент в этой части **устроен для пошагового изучения**: не план «что читать», а единый шаблон разделов (цель → термины → теория → пошагово → простыми словами → картинка в голове → «Проверь себя» → «Запомните»). Оглавление сгруппировано по этапам — выбирай следующий шаг и переходи по ссылке в раздел; в конце части — справочник, вопросы для самопроверки и резюме для закрепления.

**Как устроено повторение в разделах.** Термины расшифрованы в начале каждого раздела. **Одна и та же идея повторяется** в разных формулировках: в «Теория», «Пошагово» (часто с числами), «Простыми словами», «Картинка в голове», блоке «Важно» (где есть), «Как запомнить», «Проверь себя» и «Запомните». В конце многих разделов — блок **«Ещё раз самыми простыми словами»**. Если с первого раза не уложилось — дочитай до «Запомните»; к тому моменту мысль повторится 4–5 раз.

**Если кажется, что «не врубаешься».** Не останавливайся на первом абзаце. В каждом разделе одна и та же суть дана по-разному: строго (термины, теория), с числами (пошагово), без жаргона (простыми словами), образом (картинка в голове), в блоке «Три пункта, чтобы не забыть» (где есть) и в конце — одной фразой («ещё раз самыми простыми словами»). Дочитай раздел до конца; если после «картинки» и «ещё раз» всё ещё туманно — перечитай только блоки «Простыми словами», «Картинка в голове», «Три пункта» и «Ещё раз самыми простыми словами». **Можно возвращаться к одному и тому же разделу несколько раз** — каждый проход закрепляет; проверь себя вопросами «Проверь себя» и только потом переходи дальше.

### Краткая шпаргалка по части

| Понятие / конструкция | Суть (что это и когда нужно) |
|----------------------|-----------------------------|
| **Индекс** | Дополнительная структура по одному или нескольким столбцам для быстрого поиска; ускоряет WHERE, ORDER BY, GROUP BY; замедляет INSERT/UPDATE/DELETE |
| **B-tree** | Сбалансированное дерево поиска; все листья на одном уровне; поиск O(log N); поддержка диапазонов (BETWEEN, >, <) и сортировки |
| **Составной индекс (a, b)** | Индекс по нескольким столбцам; порядок столбцов важен; правило левого префикса: (a, b) полезен для WHERE a = ? и WHERE a = ? AND b = ?, но не для одного WHERE b = ? |
| **Покрывающий индекс (covering)** | Все столбцы запроса есть в индексе (в PostgreSQL — INCLUDE); index-only scan без обращения к таблице |
| **Hash-индекс** | Поиск только по равенству (WHERE key = constant); нет диапазонов и сортировки; в PostgreSQL с версии 10 — полноценный |
| **GiST** | Обобщённое дерево поиска; диапазоны, геометрия, полнотекст, разнообразные типы; один индекс — разные стратегии |
| **GIN** | Инвертированный индекс; полнотекст, массивы, JSONB (ключи/значения); posting list и posting tree; fast update (pending list) |
| **BRIN** | Block Range INdex; минимальный размер; хранит min/max по блокам таблицы; эффективен для данных, упорядоченных по столбцу (время) |
| **Частичный индекс** | Индекс только по строкам, удовлетворяющим WHERE; меньше размер, быстрее обновление; уникальный частичный — одна запись на условие |
| **Индекс по выражению** | Индекс на выражение (например, LOWER(email)); функциональный индекс; функция должна быть immutable |
| **CREATE INDEX CONCURRENTLY** | Создание индекса без эксклюзивной блокировки таблицы; запись разрешена; дольше и в два прохода |
| **REINDEX CONCURRENTLY** | Перестроение индекса без длительной блокировки; с PostgreSQL 12 |
| **Bloat индекса** | Раздувание: много пустого места из-за обновлений/удалений; VACUUM не сжимает индекс; решается REINDEX или REINDEX CONCURRENTLY |
| **CLUSTER** | Физическая пересортировка таблицы по индексу; улучшает корреляцию и index-only scan; блокирует таблицу |
| **Bitmap Index Scan** | По индексу собирается битовая карта TID, затем страницы читаются по порядку; выгодно при множестве строк |
| **Index-only scan** | Данные берутся только из индекса (без heap), если visibility map помечает страницы как «все видимы» |
| **pg_stat_user_indexes** | Статистика использования: idx_scan (сколько раз индекс использовался для сканирования), idx_tup_read, idx_tup_fetch |

Используй таблицу как карту: нашёл понятие в первом столбце → во втором видишь суть и типичное применение.

**Когда какой индекс (кратко).** Чтобы не путаться после прочтения части — сверяйся с этой строкой. Нужен **поиск по одному значению** (WHERE col = ?) или **диапазон** (BETWEEN, >, <) или **ORDER BY** по столбцу → **B-tree** (по умолчанию). Только **равенство**, без диапазонов и сортировки → можно **Hash**. **Диапазоны дат/чисел** (тип range), **геометрия** (PostGIS), **полнотекст** (слова в тексте) → **GiST** или **GIN** (полнотекст и массивы/JSONB — чаще GIN). **Огромная таблица, данные по порядку по времени** → **BRIN**. Индекс **только по части строк** (например, только активные) → **частичный индекс** (WHERE в определении). Поиск **без учёта регистра** (LOWER(email)) → **индекс по выражению** LOWER(email); в запросе тоже LOWER. **Создавать индекс на живой таблице без простоя** → **CREATE INDEX CONCURRENTLY**. Индекс **раздулся** после массовых обновлений → **REINDEX** (или REINDEX CONCURRENTLY). Подробности — в соответствующих разделах.

### Маршрут изучения в этой части

1. **Назначение индексов (§29)**  
   Зачем индексы: ускорение поиска, сортировки, группировки, JOIN. Замедление DML. Когда не индексировать. Баланс числа индексов.

2. **B-tree (§30)**  
   Структура сбалансированного дерева, листья, диапазонные запросы. Составные ключи и порядок столбцов. INCLUDE (покрывающий индекс). Уникальный индекс.

3. **Hash, GiST, GIN, BRIN (§31)**  
   Hash — только равенство. GiST — обобщённое дерево (диапазоны, геометрия, полнотекст). GIN — инвертированный (массивы, JSONB, полнотекст). BRIN — по блокам (min/max). Кратко: пространственные (R-tree), полнотекст.

4. **Частичный индекс и индекс по выражению (§32)**  
   WHERE в определении индекса. Уникальный частичный. Функциональный индекс (LOWER, выражения); immutable.

5. **Создание и перестроение (§33)**  
   CREATE INDEX, CREATE INDEX CONCURRENTLY. REINDEX, REINDEX CONCURRENTLY. Bloat индекса. CLUSTER по индексу.

6. **Bitmap scan и index-only scan (§34)**  
   Как планировщик выбирает bitmap index scan. Index-only scan и visibility map. Корреляция физического и логического порядка.

7. **Мониторинг (§35)**  
   pg_stat_user_indexes: idx_scan. Неиспользуемые индексы. Избыточные и дублирующие индексы.
