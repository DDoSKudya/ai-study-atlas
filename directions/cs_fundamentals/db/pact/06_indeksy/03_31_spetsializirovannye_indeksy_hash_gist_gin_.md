## 31. Специализированные индексы: Hash, GiST, GIN, BRIN

### 31.1. Hash-индекс

**Цель раздела.**  
Понять **Hash-индекс**: поиск **только по равенству** (WHERE key = constant); нет поддержки диапазонов (BETWEEN, >, <) и ORDER BY по ключу. В PostgreSQL с версии 10 — полноценный тип индекса. После раздела ты будешь чётко знать, когда Hash уместен и почему для «больше/меньше» и сортировки он не подходит.

#### Термины (расшифровка)

- **Hash-индекс** — индекс, в котором от значения ключа вычисляется **хеш** (hash function — число фиксированной длины). По этому числу определяется **корзина (bucket)** — ячейка, в которой хранятся записи (ключ + TID) для всех ключей с таким хешем. **Поиск:** вычислить хеш от искомого значения → перейти в корзину → в ней найти **точное** совпадение ключа (хеш может совпадать у разных ключей — коллизия; тогда в корзине несколько записей). Поддерживается **только равенство** (WHERE col = constant). **Диапазоны** (col > x, BETWEEN) и **ORDER BY col** невозможны — записи в корзинах не упорядочены по значению ключа, «следующего» и «предыдущего» нет.
- **Равенство (equality)** — условие вида col = value. Hash-индекс заточен именно под такой запрос: один расчёт хеша, переход в одну корзину, поиск по ключу внутри. Для чисто точечных запросов (без диапазонов и сортировки) может быть сравним с B-tree или чуть выгоднее по памяти/скорости.

#### Теория и правила (подробно)

- **Создание в PostgreSQL:** `CREATE INDEX idx_name ON table_name USING HASH (column_name);`. По умолчанию CREATE INDEX создаёт **B-tree**; для Hash нужно явно указывать **USING HASH**.
- **Когда уместен:** только запросы **WHERE col = constant**. Нет условий col > ..., col < ..., col BETWEEN ...; нет ORDER BY col. Для одного равенства Hash может быть сравним с B-tree; на практике **B-tree чаще используют**, потому что он **универсален**: и равенство, и диапазоны, и сортировка. Hash выигрывает только там, где гарантированно только «найти ровно одно значение» и больше ничего от индекса не нужно.
- **В MySQL/InnoDB:** отдельного типа «Hash index» для пользовательских таблиц по умолчанию нет. Есть внутренний **adaptive hash index** для ускорения доступа по первичному ключу в памяти — это не тот же тип, что Hash в PostgreSQL.

#### Пошагово: поиск по Hash-индексу (с числами)

Запрос: `SELECT * FROM users WHERE email = 'a@b.com'`. Индекс: **USING HASH (email)**. В индексе 1 млн записей, 10 000 корзин.

1. Вычисляем **хеш** от 'a@b.com' (например, 38472).
2. По номеру корзины **38472** переходим к этой корзине (одно чтение страницы индекса или несколько, если корзина большая).
3. В корзине ищем запись с **ключом ровно** 'a@b.com' (в корзине могут лежать и другие ключи с тем же хешем — коллизии). Находим (email, TID).
4. По TID читаем страницу таблицы и возвращаем строку.

**Итог:** очень мало чтений (корзина + страница таблицы). Но запрос «все email > 'a@b.com'» или «ORDER BY email» по Hash сделать **нельзя** — в корзинах нет порядка по значению email, нельзя «идти по следующему».

> **Важно.** В Hash **нет порядка** ключей. Корзины нумеруются по хешу; внутри корзины записи не обязаны быть отсортированы по ключу. Поэтому «все ключи от A до B» или «отсортировать по ключу» по Hash-индексу реализовать нельзя — нет понятия «следующий ключ». Для диапазонов и ORDER BY нужен **B-tree** (или другой упорядоченный индекс).

#### Что будет, если попытаться использовать Hash для WHERE col > 100

Планировщик **не выберет** Hash-индекс для такого условия: у Hash нет операции «больше» или «диапазон». Записи разложены по корзинам по хешу; хеш от 100, 101, 102 — разные, корзины разные; нет непрерывного «диапазона» значений в одном месте. Чтобы найти «все col > 100», пришлось бы обойти **все** корзины и в каждой отфильтровать по col > 100 — это не быстрее полного скана таблицы. Поэтому для таких запросов планировщик использует B-tree или sequential scan; Hash не применяется.

#### Простыми словами

**Hash-индекс** — как ящики с номерами: по значению ключа считаем номер ящика (хеш) и ищем **только в этом ящике**. Найти «ровно это значение» — быстро. «Все значения больше X» или «отсортировать по ключу» — нельзя: в ящиках нет общего порядка. Уместен только для **WHERE col = ?**.

#### Картинка в голове

**Hash** — как ряд **пронумерованных ящиков**. По значению ключа считаем номер ящика (хеш) и ищем **только в этом ящике**. Найти «ровно это» — быстро. Но ящики не стоят «по порядку» по смыслу: ящик 5 не «следующий после» ящика 4 по значению ключа. Поэтому «все ключи больше 100» или «отсортировать по ключу» по Hash сделать нельзя — нет понятия «следующий ящик по значению». B-tree — как этажерка с полками по порядку: есть «следующий» и «предыдущий»; Hash — только «ящик с таким номером».

#### Как запомнить

**Hash** = только **равенство** (WHERE key = constant). Нет диапазонов, нет ORDER BY по ключу. В PostgreSQL: CREATE INDEX ... **USING HASH** (col).

#### Проверь себя (31.1)

1. Для какого условия подходит Hash-индекс?  
<details><summary>Ответ</summary> Только для **равенства**: WHERE col = constant. Не подходит для col > ..., col BETWEEN ..., ORDER BY col.</details>

2. Почему Hash не поддерживает диапазонные запросы?  
<details><summary>Ответ</summary> Потому что записи разложены по **корзинам по хешу**; порядок значений ключа в индексе не сохраняется. Нет «следующего» и «предыдущего» значения — нельзя пройти диапазон по порядку. Для диапазонов нужен упорядоченный индекс (B-tree).</details>

3. В каких случаях имеет смысл выбирать Hash вместо B-tree?  
<details><summary>Ответ</summary> Только когда запросы **строго** по равенству (WHERE col = constant) и никогда не нужны диапазоны (col > x, BETWEEN) и не нужен ORDER BY col. Тогда Hash может быть чуть эффективнее по памяти и скорости. На практике чаще оставляют B-tree: он один покрывает и равенство, и диапазоны, и сортировку; отдельный Hash добавляют только при доказанной выгоде на точечных запросах.</details>

**Ещё раз самыми простыми словами:** Hash — это «по номеру ящика найти ровно это значение». Очень быстро для «найти одну запись по точному совпадению». Всё, что требует «все больше/меньше» или «отсортировать по этому полю» — не для Hash, для этого B-tree.

#### Запомните

- **Hash-индекс** — только поиск по **равенству** (WHERE col = constant). Нет диапазонов и ORDER BY по ключу. В PostgreSQL: USING HASH.

### 31.2. GiST

**Цель раздела.**  
Понять **GiST (Generalized Search Tree)** — обобщённое дерево поиска; один интерфейс, разные стратегии для **диапазонов**, **геометрии**, **полнотекста** и других типов данных. Поддерживает операции «пересекается», «содержится», «слева/справа» и т.п. После раздела ты будешь знать, когда выбирать GiST вместо B-tree и что он даёт для диапазонов и геометрии.

#### Термины (расшифровка)

- **GiST (Generalized Search Tree)** — индексная структура, в которой логика «сравнения» и «разделения» задаётся **операторами** для типа данных (не только «меньше/больше», но и «пересекается», «содержится»). Узлы хранят **предикаты** — сжатое описание поддерева: например, «диапазон [a, b]» для range, «bounding box» (минимальный прямоугольник) для геометрии. При поиске: если условие запроса **не может** выполняться для записей в поддереве (по предикату узла), поддерево **пропускается** целиком. Подходит для типов без единственного линейного порядка: диапазоны, полигоны, полнотекст, сетевые адреса.
- **Операторы** — функции сравнения/проверки для типа. Для **диапазонов:** && (пересекается), @> (содержит), <@ (содержится в); для **геометрии:** && (пересечение bbox), ST_Contains, ST_Intersects; для **полнотекста:** @@ (tsvector соответствует tsquery). Планировщик выбирает GiST, когда в запросе есть условие с таким оператором по проиндексированному столбцу.

#### Теория и правила (подробно)

- **Применение в PostgreSQL:** **диапазоны** (range types: daterange, int4range, tstzrange и т.д.), **tsvector** (полнотекст), **геометрия** (PostGIS: geometry, geography), **сетевые адреса** (inet, cidr). Создание: `CREATE INDEX ... USING GIST (col);` — класс операторов задаётся типом столбца (или указывается явно).
- **Плюсы:** один тип индекса для разных доменов; гибкость; естественная поддержка «пересекается», «содержится». **Минусы:** для чисто точечного поиска по одному значению часто менее эффективен, чем B-tree; размер индекса может быть больше. Для полнотекста при больших объёмах **GIN** часто быстрее; GiST компактнее при частых обновлениях.

#### Пошагово: поиск по GiST (диапазоны)

Таблица: `reservations (id, period daterange)`. Запрос: «все брони, пересекающиеся с 2025-02-01 — 2025-02-07». Индекс: **USING GIST (period)**.

1. В корне дерева хранятся предикаты дочерних узлов (например, «периоды в январе», «в феврале», «в марте»). Условие «пересекается с [02-01, 02-07]» не пересекается с «январь» и «март» — эти поддеревья **пропускаем**.
2. Заходим только в поддерево «февраль». Во внутренних узлах снова отсекаем по предикатам (пересечение с [02-01, 02-07]).
3. В листьях получаем список TID записей, у которых period пересекается с заданным диапазоном.
4. По TID читаем страницы таблицы и возвращаем строки.

**Итог:** обходим только те ветки дерева, где **возможно** пересечение; не сканируем все записи таблицы. B-tree по одному числу так не умеет — «пересечение диапазонов» не сводится к одному «меньше/больше».

> **Важно.** GiST не привязан к одному способу сравнения. Для чисел — «меньше/больше», для диапазонов — «пересекается/содержится», для геометрии — «внутри прямоугольника», «пересекает». Один интерфейс дерева — разные операции в зависимости от типа данных. Поэтому GiST называют **обобщённым** деревом поиска.

#### Что будет, если использовать B-tree по столбцу с диапазонами для запроса «все диапазоны, пересекающиеся с [a, b]»

B-tree упорядочивает по **одному** порядку (например, по нижней границе диапазона). Запрос «пересекается с [a, b]» не эквивалентен «нижняя граница между X и Y»: пересекаться могут и диапазоны с нижней границей левее a, и правее b. Поэтому по B-tree пришлось бы перебирать много записей и проверять пересечение вручную. **GiST** заточен под это: в узлах хранятся предикаты (объединение диапазонов поддерева), и целые поддеревья отсекаются, если их предикат не пересекается с [a, b].

#### Картинка в голове

**GiST** — как **дерево с разными «вопросами» на каждом этаже**. Обычный указатель (B-tree) спрашивает только: «это значение меньше или больше?» GiST в зависимости от типа данных спрашивает по-разному: для диапазонов — «этот отрезок **пересекается** с моим?», для карты — «эта фигура **внутри** моего прямоугольника?», для текста — «**есть ли** это слово?». На каждом уровне дерева целые ветки отсекаются, если ответ «точно нет». Один и тот же механизм дерева — разные правила для диапазонов, геометрии и текста. Поэтому GiST называют **обобщённым** деревом.

#### Простыми словами

**GiST** — «дерево поиска под любую логику»: не только «меньше/больше», но и «пересекается», «содержится», «внутри прямоугольника». Используется для диапазонов дат/чисел, геометрии на карте, полнотекста. Один тип индекса — много типов данных.

#### Как запомнить

**GiST** = обобщённое дерево; диапазоны (range), геометрия (PostGIS), полнотекст (tsvector). Операторы: &&, @>, <@, @@ и др.

#### Проверь себя (31.2)

1. Для каких типов данных в PostgreSQL часто используют GiST?  
<details><summary>Ответ</summary> **Диапазоны** (range types: daterange, int4range и т.д.), **геометрия** (PostGIS: geometry, geography), **полнотекст** (tsvector), **сетевые адреса** (inet, cidr). Для них определены классы операторов GiST (пересечение, включение и т.д.).</details>

2. Чем GiST отличается от B-tree применительно к диапазонам?  
<details><summary>Ответ</summary> B-tree упорядочивает записи по **одному значению** (или кортежу); для диапазонов «значение» — это интервал, и естественный порядок не единственный. GiST хранит в узлах **предикаты** (например, bounding box или объединение диапазонов) и при поиске отсекает поддеревья по условию «пересекается / содержится» и т.п. Для запросов «найти все диапазоны, пересекающиеся с [x, y]» GiST подходит; B-tree по диапазонам тоже возможен (сравнение по границам), но GiST заточен под такие операции.</details>

3. Когда для полнотекста (tsvector) разумнее выбрать GiST, а когда GIN?  
<details><summary>Ответ</summary> **GiST** — когда важны **частые обновления** и меньший размер индекса при изменении данных; поиск может быть медленнее на больших объёмах. **GIN** — когда важна **скорость поиска** при больших объёмах и обновления реже; GIN обычно быстрее для сложных запросов (много токенов, фразы). Часто для полнотекста выбирают GIN; GiST — если доминируют вставки/обновления и размер индекса критичен.</details>

**Ещё раз самыми простыми словами:** GiST — это дерево, в котором «правила игры» задаются типом данных: для диапазонов — «пересекается ли?», для геометрии — «внутри ли прямоугольника?», для текста — «есть ли слово?». Один тип индекса решает много разных задач, где обычного «меньше/больше» недостаточно.

#### Запомните

- **GiST** — обобщённое дерево поиска; диапазоны, геометрия (PostGIS), полнотекст (tsvector), inet/cidr. Операторы: &&, @>, <@, @@ и др.

### 31.3. GIN

**Цель раздела.**  
Понять **GIN (Generalized Inverted Index)** — **инвертированный индекс**: по **элементам** (токенам, ключам JSONB, элементам массива) хранятся списки TID (posting list). Подходит для **полнотекста**, **массивов**, **JSONB**. После раздела ты будешь понимать, почему GIN быстр для «содержит» и как устроен поиск по элементам.

#### Термины (расшифровка)

- **GIN (Generalized Inverted Index)** — индекс, в котором **ключ** — это **элемент** (токен слова, элемент массива, ключ или значение JSONB), а **значение** — **список TID** (posting list) строк, содержащих этот элемент. «Инвертированный»: не «строка → список элементов» (как в таблице), а **«элемент → список строк»**. Поиск «все строки, где есть элемент X» — один доступ по ключу X и чтение списка TID; не нужно сканировать всю таблицу.
- **Posting list** — список идентификаторов строк (TID), в которых встречается данный ключ. В GIN хранится как плоский список или как дерево (posting tree), если TID много. **Fast update** — новые записи сначала попадают в **pending list** (отдельный список вне основного дерева), затем периодически **сливаются** в индекс; так при частых вставках не перестраивается всё дерево.

#### Теория и правила (подробно)

- **Применение:** **полнотекст** (tsvector) — ключ = токен слова; **массивы** — ключ = элемент массива; **JSONB** — ключи/значения (часто с **jsonb_path_ops** для меньшего размера). Создание: `CREATE INDEX ... USING GIN (col);` или с опцией, например `USING GIN (col jsonb_path_ops)`.
- **Плюсы:** очень эффективен для «содержит элемент», «содержит любой из», «содержит все»; запрос по нескольким элементам — пересечение/объединение списков TID. **Минусы:** нет поддержки обычного равенства/диапазона по **целому** значению столбца (только по элементам внутри); обновление дороже из-за размера posting list и merge.

#### Пошагово: поиск по GIN (полнотекст «слово A и слово B»)

Таблица: `docs (id, body text)`, индекс: **USING GIN (to_tsvector('simple', body))**. Запрос: «документы, где есть слово "кошка" и "собака"».

1. По ключу **"кошка"** в GIN получаем **posting list** — например, TID: (1,1), (2,3), (5,2), … (номера страниц и позиций в таблице).
2. По ключу **"собака"** получаем второй список TID.
3. Вычисляем **пересечение** двух списков (TID, которые есть в обоих) — это кандидаты на результат.
4. По полученным TID читаем строки из таблицы (и при необходимости перепроверяем условие, если индекс мог вернуть ложные кандидаты).

**Итог:** не сканируем все строки таблицы; только два поиска по индексу + пересечение списков + чтение найденных строк. Чем реже слово, тем короче список — быстрее.

> **Важно.** GIN отвечает на вопрос «**в каких строках встречается этот элемент**». Он не умеет «все строки, где столбец = ровно одно значение» в смысле обычного B-tree (если столбец — не «контейнер» элементов). Он заточен под «содержит», «содержит любой», «содержит все» по элементам внутри значения (слова, элементы массива, ключи JSONB).

#### Что будет, если искать по GIN «все строки, где массив ровно равен [1,2,3]»

Технически можно выразить как «содержит 1 и содержит 2 и содержит 3» и взять пересечение posting list. Но «ровно [1,2,3]» ещё означает «ничего лишнего» — GIN по элементам этого не гарантирует: строка [1,2,3,4] тоже попадёт в пересечение по 1, 2, 3. Поэтому после GIN часто нужна **recheck** — перепроверка условия по самой строке. Для «ровно равен массиву» иногда выгоднее B-tree по представлению массива (если СУБД поддерживает) или отдельное условие в WHERE после GIN.

#### Картинка в голове

**GIN** — как **указатель в конце учебника**: не «у какой темы какие страницы» (тема → страницы), а **наоборот** — «в каком слове на каких страницах»: «кошка — стр. 3, 7, 15», «собака — стр. 2, 7, 20». Чтобы найти страницы, где есть и «кошка», и «собака», смотришь оба списка и берёшь **общие** номера (страница 7). Таблица — это «страницы» (строки), элементы (слова, ключи JSONB, элементы массива) — это «слова в указателе». GIN хранит «элемент → список строк (TID)»; поиск «все строки, где есть этот элемент» — один заход в указатель и чтение списка. Поэтому GIN называют **инвертированным** индексом.

#### Простыми словами

**GIN** — указатель «наоборот»: не «страница → какие слова на ней», а «слово → на каких страницах оно встречается». Идеально для «найти все строки, где есть это слово / этот элемент массива / этот ключ JSON». Полнотекст, массивы, JSONB — типичные применения.

#### Как запомнить

**GIN** = инвертированный индекс: **элемент → список TID**. Полнотекст (tsvector), массивы (@>, &&), JSONB (@>, ?). Posting list, fast update (pending list).

#### Проверь себя (31.3)

1. Для каких типов столбцов в PostgreSQL используют GIN?  
<details><summary>Ответ</summary> **tsvector** (полнотекст — токены слов), **массивы** (любой тип массива — элементы), **jsonb** (ключи и/или значения; часто с jsonb_path_ops). Для них определены классы операторов GIN (содержит, пересекается и т.д.).</details>

2. Что такое posting list в GIN?  
<details><summary>Ответ</summary> **Posting list** — список **TID** (идентификаторов строк таблицы), в которых встречается данный ключ индекса (токен, элемент массива, ключ JSONB). По ключу GIN возвращает этот список; СУБД затем читает соответствующие строки из таблицы (или объединяет списки при нескольких ключах — пересечение для «содержит все», объединение для «содержит любой»).</details>

3. Почему обновления (INSERT/UPDATE) при GIN часто дороже, чем при B-tree?  
<details><summary>Ответ</summary> В GIN одна строка может добавлять **несколько** ключей (несколько слов, элементов массива, ключей JSONB). Каждый ключ — запись в индексе с posting list; при вставке нужно обновить много записей. Плюс при **fast update** данные сначала идут в pending list, потом сливаются — merge может быть тяжёлым. B-tree для одной строки обычно обновляет один путь от корня до листа; в GIN — много ключей и возможно несколько списков.</details>

**Ещё раз самыми простыми словами:** GIN — это «по слову/элементу узнать номера строк». Книжный указатель в конце книги: «кошка — стр. 1, 5, 12» — это и есть идея GIN. Идеально для «найти все, где есть это слово/элемент»; для обычного «найти строку по одному значению столбца» чаще используют B-tree.

**Три пункта, чтобы не забыть (GIN).** 1) **Инвертировано:** ключ индекса — это **элемент** (слово, элемент массива, ключ JSONB), значение — **список TID** (в каких строках он встречается). 2) Подходит для «содержит», «содержит любой», «содержит все» — полнотекст, массивы, JSONB; для обычного «столбец = значение» без контейнера элементов — обычно B-tree. 3) Одна строка даёт **много ключей** в индексе → обновления (INSERT/UPDATE) дороже, чем у B-tree.

#### Запомните

- **GIN** — инвертированный индекс: элемент → список TID. Полнотекст (tsvector), массивы, JSONB. Posting list, fast update (pending list).

### 31.4. BRIN

**Цель раздела.**  
Понять **BRIN (Block Range INdex)** — индекс **минимального размера**: хранит по каждому **блоку (диапазону) страниц** таблицы только **сводку** (min/max по столбцу). Эффективен для данных, **упорядоченных по столбцу** (время вставки, монотонный ключ). После раздела ты будешь знать, когда BRIN выгоден и почему при «вперемешку» данных он почти бесполезен.

#### Термины (расшифровка)

- **BRIN (Block Range INdex)** — индекс, в котором таблица мысленно разбита на **последовательные блоки страниц** (block range; размер блока задаётся, например 128 страниц). Для **каждого блока** в индексе хранится одна **сводка** по выбранному столбцу — обычно **min** и **max** значение в этом блоке. **Поиск:** при условии col > x СУБД смотрит только записи индекса: если у блока max < x, весь блок **пропускаем** (ни одна строка в нём не подходит); блоки, где min ≤ x ≤ max или max ≥ x, нужно прочитать. Размер индекса — **одна запись на блок** таблицы, т.е. в тысячи раз меньше, чем B-tree (запись на строку).
- **Корреляция** — насколько **физический порядок** строк на диске совпадает с **логическим порядком** по столбцу (например, по дате). Высокая корреляция: «сначала старые даты, потом новые» — в одном блоке страниц даты лежат в узком интервале, min/max хорошо описывают блок. Низкая корреляция: даты вперемешку — в каждом блоке и старые, и новые; отсечь блок по min/max почти нельзя, придётся читать почти все блоки.

#### Теория и правила (подробно)

- **Применение:** большие таблицы, **упорядоченные по времени** (created_at, event_time) или по монотонному ключу; лог-таблицы, события, партиции по дате. Создание: `CREATE INDEX ... USING BRIN (col);` или с параметром `pages_per_range` (сколько страниц в одном блоке).
- **Плюсы:** очень маленький размер индекса; быстрое создание; при вставке «в конец» обновляется в основном последняя сводка. **Минусы:** эффективен **только при высокой корреляции**; при случайном порядке отсекается мало блоков — выгода мала. Для точечного поиска (одна строка по ключу) и произвольной сортировки BRIN не заменяет B-tree.

#### Пошагово: поиск по BRIN (дата > 2025-01-01)

Таблица: 10 млн строк, данные **по порядку** по `created_at` (сначала старые, потом новые). BRIN по `created_at`, блок = 128 страниц (~128×≈200 строк ≈ 25 600 строк на блок). Всего блоков ~400.

1. Читаем **только индекс BRIN** — 400 записей (min, max по каждому блоку). Очень быстро.
2. Блоки 1..200: max < 2025-01-01 → **пропускаем целиком** (не читаем страницы таблицы).
3. Блоки 201..400: min ≤ 2025-01-01 ≤ max или max ≥ 2025-01-01 → **читаем** эти страницы таблицы и отбираем строки с created_at > 2025-01-01.
4. Итог: прочитали ~200 блоков таблицы вместо 400; индекс — крошечный. Если бы данные были **вперемешку**, в каждом блоке min был бы старым, max новым — отсечь блоки было бы почти нельзя, пришлось бы читать почти всё.

> **Важно.** BRIN не хранит, **в какой строке** какое значение; он хранит только «в этом куске таблицы минимум такой, максимум такой». Поэтому точечный поиск «найти строку с created_at = ровно 2025-02-15» по одному BRIN неоптимален: придётся открыть все блоки, где 2025-02-15 попадает в [min, max], и искать внутри. Для точечного поиска лучше B-tree. BRIN хорош для **диапазонов** по упорядоченному столбцу и при большой таблице.

#### Что будет, если создать BRIN по столбцу, по которому данные записаны в случайном порядке

Сводка (min, max) в каждом блоке будет охватывать **почти весь** диапазон значений столбца (потому что в каждом блоке вперемешку и маленькие, и большие значения). При запросе «col > x» почти ни один блок нельзя отсечь: в почти каждом блоке max > x. В итоге придётся прочитать почти все блоки таблицы — выигрыша по сравнению с полным сканом почти нет, а индекс занимает место. **Вывод:** BRIN имеет смысл только при **хорошей корреляции** (данные по столбцу упорядочены или близки к упорядоченным).

#### Простыми словами

**BRIN** — не «указатель на каждую строку», а «указатель на кусок таблицы»: в каждом куске записаны только минимум и максимум по столбцу. Если ищем «дата > 1 января», куски, где максимум раньше 1 января, можно не открывать. Работает отлично, когда данные в таблице лежат по порядку (например, по времени); если вперемешку — пользы мало.

#### Картинка в голове

**BRIN** — как **краткое содержание книги по главам**: «Глава 1: даты с 1 по 5 января. Глава 2: 6–12 января. Глава 3: 13–20 января…» Чтобы найти «всё за 10 января», смотришь содержание: глава 2. Открываешь только главу 2, остальные не трогаешь. Если бы главы были **вперемешку** (в каждой и 1-е, и 15-е, и 30-е число), в содержании пришлось бы писать «Глава 1: с 1 по 31 января» — отсечь ничего нельзя. То же с BRIN: порядок данных по столбцу = «главы по порядку»; без порядка «краткое содержание» почти не помогает.

#### Как запомнить

**BRIN** = Block Range INdex; **min/max по блоку страниц**; малый размер. Эффективен при **высокой корреляции** (данные упорядочены по столбцу). Типично: большие таблицы по времени.

#### Проверь себя (31.4)

1. В чём главное отличие BRIN от B-tree по размеру и структуре?  
<details><summary>Ответ</summary> **BRIN** хранит одну запись (сводку: min, max) на **блок страниц** таблицы (например, 128 страниц). Размер индекса — число блоков × размер записи сводки, т.е. очень мал. **B-tree** хранит запись на **каждую строку** (или на каждое значение ключа) — размер пропорционален числу строк. BRIN — «сжатый» индекс по диапазонам блоков.</details>

2. При каком условии BRIN даёт большой выигрыш?  
<details><summary>Ответ</summary> При **высокой корреляции**: физический порядок строк на диске близок к порядку по индексированному столбцу (например, вставки идут по времени — новые строки в конец). Тогда в одном блоке страниц значения столбца лежат в узком диапазоне; min/max хорошо отсекают блоки, и многие блоки можно не читать. При низкой корреляции (случайный порядок) в каждом блоке разброс значений большой, отсечь мало блоков — выгода BRIN мала.</details>

3. Почему BRIN плохо подходит для точечного запроса «найти одну строку с id = 12345»?  
<details><summary>Ответ</summary> BRIN хранит только **сводку по блоку** (min, max), а не указатели на отдельные строки. Для «id = 12345» пришлось бы перебрать все блоки, у которых 12345 попадает в [min, max]; таких блоков может быть много (если id разбросаны), и внутри каждого блока — полное сканирование. B-tree по id сразу ведёт к одной записи (или к малому числу записей). BRIN предназначен для **диапазонных** запросов по упорядоченному столбцу, а не для точечного поиска.</details>

**Три пункта, чтобы не забыть (BRIN).** 1) **Один блок таблицы = одна запись в BRIN** (min и max по столбцу в этом блоке); индекс очень маленький. 2) **Порядок данных критичен:** если по столбцу данные упорядочены (например, вставки по времени) — блоки можно отсекать по min/max; если вперемешку — почти ни один блок не отсечь, пользы нет. 3) Для **точечного** поиска («найти одну строку по id») BRIN не подходит — нужен B-tree; BRIN для **диапазонов** по большому упорядоченному столбцу.

**Ещё раз самыми простыми словами:** BRIN — это «краткое содержание по главам»: не «на какой странице слово», а «в этой главе минимум и максимум по дате». Если главы идут по порядку времени, по «краткому содержанию» видно, какие главы открывать. Если в каждой главе даты вперемешку — краткое содержание почти не помогает. То же с BRIN: порядок данных по столбцу критичен.

#### Запомните

- **BRIN** — сводка (min/max) по блокам страниц таблицы; минимальный размер индекса. Эффективен при **высокой корреляции** (данные упорядочены по столбцу); типично большие таблицы по времени.

### 31.5. Пространственные и полнотекстовые индексы

**Цель раздела.**  
Понять **пространственные индексы** (PostGIS) — GiST или SP-GiST по geometry/geography для запросов «в этой области», «рядом с точкой»; и **полнотекстовые** — GIN (или GiST) по tsvector для поиска по словам. После раздела ты будешь знать, какой тип индекса и какой оператор использовать для геометрии и для текста. Детали GiST и GIN — в 31.2 и 31.3.

#### Термины (расшифровка)

- **Пространственный индекс (spatial index)** — индекс по столбцу типа **geometry** или **geography** (расширение PostGIS). В PostgreSQL для них используют **GiST** или **SP-GiST**: в узлах хранятся «ограничивающие прямоугольники» (bounding box) объектов; при запросе «все объекты в этом прямоугольнике» или «пересекающие эту область» отсекаются поддеревья, чей bbox не пересекается с областью запроса. Операторы: **&&** (пересечение bbox), **ST_Contains**, **ST_Within**, **ST_Intersects** и др. Без индекса пришлось бы перебирать все объекты и проверять условие; с индексом — быстрый отсев по bbox и проверка только кандидатов.
- **Полнотекстовый индекс (full-text index)** — индекс по столбцу типа **tsvector**. В tsvector текст уже разбит на **токены** (слова) с учётом языка и стоп-слов; индекс (GIN или GiST) хранит «токен → список строк». Запрос задаётся **tsquery** (например, 'word1 & word2' — оба слова должны быть). Оператор **@@** проверяет соответствие tsvector и tsquery. Создание: `CREATE INDEX ... ON t USING GIN (col_tsvector);` или USING GIST. **GIN** обычно быстрее на больших объёмах; **GiST** компактнее при частых обновлениях (см. 31.2, 31.3).

#### Теория и правила (подробно)

- **PostGIS (пространственные данные):** после установки расширения **CREATE EXTENSION postgis;** создаётся индекс: `CREATE INDEX idx_geom ON points USING GIST (geom);` или `USING SPGIST (geom);`. Запросы вида `WHERE geom && ST_MakeEnvelope(...)` или `WHERE ST_Contains(area, geom)` используют индекс. Для **geography** (координаты на сфере) — тот же GiST/SP-GiST; внутри могут быть преобразования.
- **Полнотекст:** столбец типа **tsvector** (или выражение `to_tsvector('russian', text_col)`). Индекс: `CREATE INDEX idx_fts ON docs USING GIN (body_tsvector);`. Запрос: `WHERE body_tsvector @@ to_tsquery('russian', 'кошка & собака');` — документы, где есть оба слова. Функции **ts_rank**, **ts_headline** — для ранжирования и подсветки фрагментов; индекс ускоряет сам поиск по @@.

#### Пошагово: зачем нужен пространственный индекс (с числами)

Таблица: 10 млн точек на карте (geom geometry). Запрос: «все точки внутри прямоугольника [x1,y1]–[x2,y2]».

1. **Без индекса:** последовательное сканирование 10 млн строк; для каждой вызывается ST_Within(точка, прямоугольник) — очень долго.
2. **С GiST по geom:** в узлах индекса хранятся bounding box объектов. СУБД отсекает поддеревья, чей bbox **не пересекается** с прямоугольником запроса; в листья попадают только кандидаты. По TID читаются только нужные строки; для них выполняется точная проверка ST_Within (на случай сложных фигур). Итог: прочитано намного меньше страниц, ответ быстрее в десятки–сотни раз.

**Полнотекст:** запрос «документы, где есть слова "кошка" и "собака"». Без индекса — полный скан и проверка каждого документа. С GIN по tsvector — по ключам «кошка» и «собака» получаем списки TID, пересечение даёт кандидатов; читаем только их. Подробнее — в 31.3.

> **Важно.** Пространственный индекс и полнотекстовый — это **применение** GiST и GIN к специальным типам (geometry, tsvector). Сама структура — та же GiST (обобщённое дерево с предикатами) или GIN (инвертированный индекс по токенам). Операторы (&&, @@, ST_Contains и т.д.) определяют, как индекс используется в запросе.

#### GiST vs SP-GiST для пространственных данных: когда какой выбирать

- **SP-GiST (Space-Partitioned GiST)** — разновидность обобщённого дерева поиска, в котором пространство ключей **разбивается на непересекающиеся области** (например, по квадрантам или по разбиению плоскости). Поддерживает типы **point**, **range**, **text** (префиксы) и в расширениях — другие типы. В PostGIS для **geometry** можно создать индекс **USING SPGIST (geom)**.
- **GiST** для geometry хранит в узлах **bounding box** (ограничивающий прямоугольник) и при запросе отсекает поддеревья по пересечению bbox с областью запроса. **SP-GiST** использует **другое разбиение пространства** (например, k-d дерево или радиальное разбиение для точек): нет одного «bbox на поддерево», а последовательное разбиение области на части.
- **Когда выбирать что:**
  - **Точки (point), много точечных объектов:** **SP-GiST** часто даёт более сбалансированное дерево и быстрее точечный поиск и запросы вида «все точки в этом прямоугольнике»; индекс может быть компактнее. Для таблиц с **миллионами точек** (геолокации, события на карте) SP-GiST по **geometry(Point)** нередко предпочтительнее GiST.
  - **Полигоны, линии, смесь геометрий:** **GiST** — основной выбор в PostGIS; поддержка сложных типов и операторов (ST_Contains, ST_Intersects и т.д.) отработана для GiST. SP-GiST для произвольной geometry тоже поддерживается (PostgreSQL 14+), но GiST чаще используется для смешанных и сложных геометрий.
  - **Практика:** для столбца **point** или «только точки» — попробовать **SPGIST**; для полигонов, линий и общего случая **geometry** — **GIST**. При сомнении можно создать оба индекса на тестовой копии и сравнить размер и время типичных запросов (EXPLAIN ANALYZE).

**Кратко:** точки (point) — часто SP-GiST выгоднее; полигоны/линии/смесь — обычно GiST. Оба поддерживают операторы &&, ST_Contains, ST_Within и др.

#### Проверь себя (31.5)

1. Для чего нужен пространственный индекс в PostGIS и какой тип индекса обычно используют?  
<details><summary>Ответ</summary> Для ускорения запросов **«все объекты в этой области»**, «пересекающие полигон», «в радиусе от точки» по столбцу **geometry** или **geography**. В PostgreSQL с PostGIS используют **GiST** или **SP-GiST** по этому столбцу. Операторы: &&, ST_Contains, ST_Within, ST_Intersects. Без индекса пришлось бы перебирать все строки; с индексом отсекаются целые поддеревья по bounding box.</details>

2. Чем отличается полнотекстовый поиск от обычного LIKE '%слово%' и какой индекс для него нужен?  
<details><summary>Ответ</summary> **LIKE '%слово%'** — поиск подстроки в сыром тексте; индекс по обычному text почти не помогает (нельзя эффективно искать по середине строки). **Полнотекстовый поиск** — текст приводится к **tsvector** (токены, стоп-слова, язык); запрос — **tsquery** (слова, операторы &, |, !). Индекс по **tsvector** (GIN или GiST) хранит «токен → список документов» — поиск по словам быстрый. Нужен столбец tsvector и индекс: CREATE INDEX ... USING GIN (col_tsvector); запрос через оператор @@.</details>

3. Для таблицы с миллионами точечных объектов (geometry Point) в PostGIS когда предпочтительнее SP-GiST, а когда GiST?  
<details><summary>Ответ</summary> **SP-GiST** для **точек (point)** часто даёт более сбалансированное разбиение пространства (k-d дерево и т.п.) и быстрее точечный поиск и запросы «все точки в прямоугольнике»; индекс может быть компактнее. Для **миллионов точек** (события, геолокации) обычно предпочтительнее **USING SPGIST (geom)**. **GiST** с bounding box хорошо подходит для **полигонов, линий и смеси** геометрий; для чисто точечных данных SP-GiST нередко выигрывает по скорости и размеру. Итог: точки — пробуем SP-GiST; полигоны/линии/общий geometry — чаще GiST.</details>

**Ещё раз самыми простыми словами:** пространственный индекс — чтобы быстро отвечать на вопрос «что лежит в этом прямоугольнике на карте» (PostGIS, GiST/SP-GiST). Полнотекстовый — чтобы быстро искать по **словам** в тексте (tsvector, GIN/GiST, оператор @@). Оба — частные случаи GiST и GIN для специальных типов данных.

#### Запомните

- **Пространственные:** PostGIS — GiST/SP-GiST по geometry/geography; операторы &&, ST_Contains, ST_Intersects.
- **Полнотекст:** tsvector, GIN (или GiST); @@, to_tsquery; rank, headline.
