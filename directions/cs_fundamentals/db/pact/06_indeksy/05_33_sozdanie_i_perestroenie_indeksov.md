## 33. Создание и перестроение индексов

### 33.1. CREATE INDEX и CREATE INDEX CONCURRENTLY

**Цель раздела.**  
Понять разницу между обычным **CREATE INDEX** (эксклюзивная блокировка таблицы на время построения) и **CREATE INDEX CONCURRENTLY** (построение без блокировки записи; дольше и в два прохода). После раздела ты будешь знать, когда рисковать блокировкой, а когда обязательно использовать CONCURRENTLY.

#### Термины (расшифровка)

- **CREATE INDEX** — построение индекса **в одной транзакции**. Таблица на время построения блокируется (эксклюзивная или тяжёлая блокировка): **вставки, обновления, удаления** по этой таблице **ждут** до завершения создания индекса. На таблице в десятки/сотни миллионов строк построение может занимать минуты или часы — всё это время запись в таблицу блокирована. Подходит для пустой или редко используемой таблицы; на продакшене с активной записью — риск простоя.
- **CREATE INDEX CONCURRENTLY** — построение индекса **без блокировки записи**. Другие сессии могут продолжать INSERT/UPDATE/DELETE. Реализация: **два прохода** по таблице (или более): первый проход строит индекс по снимку данных; затем индекс помечается как «не готов к использованию» и второй проход подхватывает изменения, произошедшие за это время. В итоге индекс получается полным и консистентным. **Минусы:** дольше обычного CREATE INDEX; больше нагрузка на I/O; **нельзя** выполнять внутри явной транзакции (BEGIN ... COMMIT). При сбое (обрыв связи, cancel) может остаться индекс в состоянии **invalid** — он не используется и занимает место; такой индекс нужно удалить (DROP INDEX CONCURRENTLY) и при необходимости пересоздать.

#### Теория и правила (подробно)

- **Когда использовать CONCURRENTLY:** на **продакшене** при создании индекса на **живой** таблице — чтобы не блокировать запись и не вызывать таймауты/очереди. В разработке, на копии БД или при разовой загрузке данных можно использовать обычный CREATE INDEX — он быстрее и проще.
- **Ограничения:** CONCURRENTLY **нельзя** выполнять внутри транзакционного блока (BEGIN ... COMMIT). При прерывании проверить в **pg_index** поле **indisvalid**: если false — индекс в состоянии invalid, его нужно удалить (DROP INDEX ... CONCURRENTLY) и при необходимости пересоздать.

#### Пошагово: что происходит при CREATE INDEX CONCURRENTLY

Таблица 50 млн строк; создаём индекс по столбцу `created_at`.

1. **Первый проход:** СУБД читает таблицу (например, последовательным сканом) и строит индекс. За это время другие транзакции могут вставлять/обновлять/удалять строки — индекс пока **не виден** запросам и помечен как строящийся.
2. **Второй проход:** СУБД фиксирует изменения, произошедшие с начала первого прохода (через WAL или снимок), и **дописывает** в индекс записи для этих изменённых строк. Так индекс становится полным и консистентным.
3. Индекс помечается как **valid** (indisvalid = true) и становится доступен планировщику. Весь период построения **запись в таблицу не блокировалась** — вставки и обновления выполнялись параллельно.

Если на шаге 2 произошёл сбой (обрыв, cancel), индекс может остаться в состоянии **invalid**: он не используется, но занимает место. В каталоге pg_index у такого индекса indisvalid = false. Решение: DROP INDEX CONCURRENTLY имя_индекса; затем при необходимости снова CREATE INDEX CONCURRENTLY.

> **Важно.** Обычный CREATE INDEX на большой живой таблице = «все пишущие запросы ждут, пока индекс построится». Это может быть десятки минут и больше; пользователи видят таймауты и ошибки. На продакшене для индексов по большим таблицам **всегда** используйте CREATE INDEX CONCURRENTLY, если таблица доступна для записи. Дольше — но без простоя.

> **ANALYZE после CREATE INDEX.** Сразу после создания индекса (обычного или CONCURRENTLY) планировщик **уже может** использовать новый индекс, но для точной оценки стоимости ему нужна **актуальная статистика** по таблице (распределение значений, число строк, корреляция). Команда **ANALYZE** по таблице обновляет статистику в **pg_stats** (в т.ч. по новым индексам). Поэтому после **CREATE INDEX** (и после REINDEX) рекомендуется выполнить **ANALYZE имя_таблицы;** — тогда планировщик начнёт корректно сравнивать стоимость сканирования по новому индексу с другими планами. Без ANALYZE индекс может «молчать» первые запросы или использоваться с неточными оценками.

#### Что будет, если выполнить CREATE INDEX CONCURRENTLY внутри BEGIN ... COMMIT

PostgreSQL **не разрешит**: команда CREATE INDEX CONCURRENTLY не может выполняться внутри транзакционного блока. Причина: CONCURRENTLY использует несколько внутренних коммитов; в одной транзакции это противоречит семантике. Нужно выполнять CONCURRENTLY **автономно** (одна команда в одной транзакции).

#### Картинка в голове

**Обычный CREATE INDEX** — как **ремонт единственной двери в офис**: пока дверь перевешивают, **никто не может войти и выйти** (запись в таблицу заблокирована). **CREATE INDEX CONCURRENTLY** — как постройка **второй двери** рядом: основная дверь открыта, все ходят как раньше; вторую дверь делают в фоне, потом переключают поток на неё. Дольше и сложнее (два прохода, проверки), но офис не останавливается. На продакшене «ремонт единственной двери» на большой таблице = все пишущие запросы ждут минуты или часы; CONCURRENTLY = никто не ждёт.

#### Простыми словами

**CREATE INDEX** — строим индекс «за раз», таблица на это время заблокирована для записи. **CREATE INDEX CONCURRENTLY** — строим индекс «в фоне», запись в таблицу не блокируется; дольше, но без простоя.

#### Как запомнить

**CONCURRENTLY** = создание индекса без блокировки записи; дольше, в два прохода; при сбое проверить indisvalid, при необходимости DROP и пересоздать.

#### Проверь себя (33.1)

1. Зачем использовать CREATE INDEX CONCURRENTLY на продакшене?  
<details><summary>Ответ</summary> Чтобы **не блокировать** запись в таблицу на время построения индекса. Обычный CREATE INDEX берёт эксклюзивную блокировку — вставки/обновления/удаления ждут. На живой таблице это может вызвать длительный простой. CONCURRENTLY строит индекс без такой блокировки — запись продолжается; построение дольше, но без простоя.</details>

2. После обрыва сессии во время CREATE INDEX CONCURRENTLY индекс появился в списке индексов, но запросы его не используют. Что проверить и что делать?  
<details><summary>Ответ</summary> Проверить в **pg_index** поле **indisvalid** для этого индекса. Если indisvalid = false, индекс в состоянии **invalid** (не завершён из-за сбоя). Такой индекс не используется планировщиком и только занимает место. **Действие:** удалить индекс командой **DROP INDEX CONCURRENTLY** имя_индекса; при необходимости пересоздать индекс снова через CREATE INDEX CONCURRENTLY. Обычный DROP INDEX (без CONCURRENTLY) на живую таблицу берёт блокировку; DROP INDEX CONCURRENTLY — без блокировки записи.</details>

3. Зачем после CREATE INDEX выполнять ANALYZE по таблице?  
<details><summary>Ответ</summary> Планировщик выбирает план по **статистике** (число строк, распределение значений, корреляция) из **pg_stats**. Статистика обновляется при **ANALYZE**. После создания нового индекса без ANALYZE планировщик может не иметь актуальных данных по таблице и по новому индексу — оценки cost будут неточными, индекс может не выбираться. **ANALYZE имя_таблицы** обновляет статистику; после этого планировщик корректно учитывает новый индекс при выборе плана.</details>

**Ещё раз самыми простыми словами:** обычное создание индекса = «все ждут, пока индекс построится». CONCURRENTLY = «индекс строится в фоне, писать в таблицу можно». На продакшене на больших таблицах — только CONCURRENTLY, и после сбоя не забыть проверить invalid-индексы и удалить их.

**Три пункта, чтобы не забыть (CREATE INDEX CONCURRENTLY).** 1) **CONCURRENTLY** = создание индекса **без блокировки записи** в таблицу; на продакшене на больших таблицах — только так. 2) Строится в **два прохода** → дольше обычного CREATE INDEX; при сбое индекс может остаться **invalid** (indisvalid = false) — такой нужно удалить (DROP INDEX CONCURRENTLY) и при необходимости пересоздать. 3) **Нельзя** выполнять внутри BEGIN ... COMMIT — только одной отдельной командой.

#### Запомните

- **CREATE INDEX CONCURRENTLY** — построение индекса без блокировки записи; дольше; при сбое проверить indisvalid.

### 33.2. REINDEX и REINDEX CONCURRENTLY

**Цель раздела.**  
Понять **REINDEX** — перестроение индекса (удаление старого и построение заново или построение нового с заменой). **REINDEX CONCURRENTLY** (PostgreSQL 12+) — перестроение без длительной эксклюзивной блокировки: пошагово как это устроено, что делать при сбое (остаются два индекса — какой удалить). После раздела ты будешь уверенно перестраивать индексы на продакшене и разбираться с последствиями обрыва.

#### Термины (расшифровка)

- **REINDEX** — перестроение индекса: старый индекс удаляется (или помечается к удалению) и строится заново. Используется при **раздувании (bloat)** индекса, повреждении или смене параметров (fillfactor и т.д.). Блокирует запись в таблицу на время перестроения (для данного индекса).
- **REINDEX CONCURRENTLY** — перестроение **без эксклюзивной блокировки** записи. Реализация: создаётся **второй** индекс с тем же определением и временным именем (в каталоге — суффикс, например `_ccnew`); он строится в фоне (как CREATE INDEX CONCURRENTLY). Когда новый индекс готов, старый и новый **меняются местами по имени**: старый индекс переименовывается (например, в `_ccold`) и отключается, новый получает имя исходного индекса и становится основным. Затем старый (`_ccold`) удаляется. Запись в таблицу во время построения нового индекса разрешена. **При сбое** (обрыв, cancel) может остаться **два** индекса: один «старый» (исходное имя или с суффиксом _ccold), один «новый» (с суффиксом _ccnew). Лишний нужно определить и удалить вручную (см. ниже).

#### Теория и правила (подробно)

- **Когда делать REINDEX:** индекс раздут (много пустого места после массовых удалений/обновлений); индекс повреждён; изменились параметры (например, fillfactor). VACUUM **не сжимает** страницы индекса — только помечает место как свободное; перестроение даёт компактный индекс.
- **REINDEX CONCURRENTLY** — на продакшене для больших индексов, чтобы не блокировать запись. Требует **PostgreSQL 12+**.

#### Пошагово: что происходит при REINDEX CONCURRENTLY (с числами)

Команда: `REINDEX INDEX CONCURRENTLY idx_orders_created_at;` (индекс по таблице orders, 50 млн строк).

1. **Создание нового индекса.** СУБД создаёт **второй** индекс с тем же определением (те же столбцы, тот же тип), но с **временным именем** (внутри может отображаться как `idx_orders_created_at_ccnew` или аналогично). Этот индекс строится так же, как при **CREATE INDEX CONCURRENTLY**: два прохода по таблице, запись в таблицу не блокируется. Старый индекс `idx_orders_created_at` всё это время **работает** — запросы используют его.
2. **Ожидание завершения транзакций.** Перед подменой СУБД ждёт, пока все транзакции, которые могли видеть старый индекс, завершатся (чтобы никто не ссылался на старый объект по имени).
3. **Подмена (swap).** Старый индекс переименовывается во временное имя (например, `idx_orders_created_at_ccold`), новый индекс переименовывается в **исходное** имя `idx_orders_created_at`. В каталоге `pg_index` новый объект помечается как valid; старый — как невалидный для использования (или помечается к удалению). С этого момента планировщик использует **новый** индекс (компактный, без bloat).
4. **Удаление старого индекса.** Индекс с именем `_ccold` (или аналог) удаляется. Остаётся один индекс с исходным именем — перестроение завершено.

**Итог:** запись в таблицу не блокировалась; старый раздутый индекс заменён новым компактным.

#### Что делать при сбое REINDEX CONCURRENTLY (остались два индекса)

При обрыве (cancel, обрыв связи, сбой процесса) после шага 1 может остаться **два** индекса на одну и ту же таблицу с одним и тем же определением: один под **исходным** именем (старый), один с суффиксом **`_ccnew`** (новый, возможно не до конца построенный). Нужно понять, **какой удалить**, чтобы не потерять рабочий индекс.

- **Как проверить:** запрос к системному каталогу:
  - `pg_index`: у каждого индекса есть **indisvalid** (используется ли планировщиком) и **indisready** (готов ли к приёму записей при вставке).
  - Имена индексов: `SELECT indexrelid::regclass, indisvalid, indisready FROM pg_index WHERE indrelid = 'orders'::regclass;` (подставь свою таблицу). Индекс с суффиксом `_ccnew` — это «новый»; индекс с исходным именем — «старый».
- **Правило:** **рабочий** индекс — тот, у которого **indisvalid = true** (его использует планировщик). Обычно после сбоя **старый** индекс (исходное имя) остаётся **valid**, а новый (`_ccnew`) — **invalid** (не достроился). Тогда **удалить нужно индекс _ccnew** (он не используется и только занимает место): `DROP INDEX CONCURRENTLY имя_таблицы_idx_ccnew;` (точное имя смотри в `pg_class` или `\di` в psql). Если бы подмена уже прошла и сбой случился на шаге 4, то valid был бы новый (уже под исходным именем), а _ccold — лишний; тогда удаляют _ccold.
- **Кратко:** посмотреть в `pg_index` **indisvalid** и имена индексов; удалить **CONCURRENTLY** тот индекс, который **не используется** (invalid или дубликат с суффиксом _ccnew/_ccold). После удаления лишнего при необходимости повторить **REINDEX INDEX CONCURRENTLY** для перестроения.

#### Fillfactor для индекса (кратко)

- **Fillfactor индекса** — параметр хранения (в PostgreSQL задаётся при создании индекса: `CREATE INDEX ... WITH (fillfactor = 90)`; по умолчанию 100 для индексов). Означает: **заполнять страницы индекса только на fillfactor процентов**, оставляя остальное место **пустым**.
- **Зачем:** при **вставке** новых ключей в B-tree страницы могут **разделяться** (split): если страница заполнена, она делится на две. Если при создании индекса оставить место (fillfactor &lt; 100), часть новых вставок попадёт в это свободное место **без немедленного split** — меньше фрагментации и чуть меньше bloat при последующих обновлениях по этому индексу. То есть fillfactor &lt; 100 **снижает темп раздувания** индекса при частых INSERT/UPDATE по индексируемым столбцам.
- **Минус:** индекс занимает больше места (страницы заполнены не полностью). Обычно для индексов оставляют 100 (по умолчанию); fillfactor 90–95 имеет смысл на **очень часто обновляемых** индексах, где bloat растёт быстро. После смены fillfactor нужен **REINDEX**, чтобы применить новый параметр.

#### Простыми словами

**REINDEX** — «переписать индекс с нуля»; убирает раздувание и повреждения. **REINDEX CONCURRENTLY** — то же самое, но сначала строится второй индекс «рядом», потом он подменяет старый и старый удаляется; запись не блокируется. При сбое могут остаться два индекса — по pg_index (indisvalid) и по имени (_ccnew / _ccold) определяешь, какой лишний, и удаляешь его CONCURRENTLY. **Fillfactor** индекса — оставлять в страницах место «про запас», чтобы реже дробить страницы при вставках и чуть сдерживать bloat.

#### Как запомнить

**REINDEX** = перестроение индекса; убирает bloat. **REINDEX CONCURRENTLY** = построить новый индекс, подменить им старый, удалить старый; при сбое — два индекса, удалить лишний по indisvalid и имени (_ccnew/_ccold). **Fillfactor** индекса &lt; 100 — меньше bloat при частых обновлениях, но больше размер.

#### Проверь себя (33.2)

1. Почему VACUUM не избавляет от раздувания индекса, а REINDEX — да?  
<details><summary>Ответ</summary> **VACUUM** по индексу только помечает **мёртвые** записи (удалённые/обновлённые ключи) и обновляет статистику; страницы индекса **не перезаписываются** и не сжимаются — пустое место остаётся. **REINDEX** перестраивает индекс **с нуля**: новый компактный индекс без мёртвых записей и фрагментации; старый удаляется. Поэтому уменьшить bloat индекса можно только перестроением (REINDEX или REINDEX CONCURRENTLY).</details>

2. После сбоя REINDEX CONCURRENTLY на таблице видно два индекса с похожими именами (один с суффиксом _ccnew). Как понять, какой удалить?  
<details><summary>Ответ</summary> Посмотреть в **pg_index** поля **indisvalid** и имена индексов (pg_class.relname). Индекс с **indisvalid = true** — тот, который **используется** планировщиком (рабочий). Индекс с суффиксом **_ccnew** — это «новый», который строился при REINDEX CONCURRENTLY; при сбое он обычно **invalid** (не достроился). Удалить нужно **лишний** — как правило, индекс _ccnew (DROP INDEX CONCURRENTLY …). Исходный индекс (без суффикса) оставить. Если подмена уже произошла, то лишним может быть индекс с суффиксом _ccold — тогда удалять его. В любом случае удалять тот, у которого indisvalid = false или который явно временный (_ccnew/_ccold).</details>

3. Зачем задавать fillfactor &lt; 100 для индекса и когда это имеет смысл?  
<details><summary>Ответ</summary> **Fillfactor &lt; 100** оставляет в страницах индекса **свободное место**. При последующих **вставках** ключей в индекс это место заполняется без немедленного **разделения страницы** (split) — меньше фрагментации и чуть меньше накопления bloat при частых UPDATE/INSERT по индексируемым столбцам. Имеет смысл на индексах, которые **очень часто** обновляются (много вставок/изменений ключей); для большинства индексов оставляют 100. Минус — индекс занимает больше места. Применить новый fillfactor можно только после **REINDEX**.</details>

**Ещё раз самыми простыми словами:** REINDEX — «переписать индекс заново»: старый раздутый или повреждённый удаляется, строится новый компактный. REINDEX CONCURRENTLY — то же, но через подмену (сначала новый индекс, потом подмена им старого); при сбое остаются два индекса — лишний удаляешь по имени (_ccnew/_ccold) и по indisvalid. На живом продакшене для больших индексов — REINDEX CONCURRENTLY. Fillfactor индекса &lt; 100 — чуть сдерживает bloat при частых обновлениях.

#### Запомните

- **REINDEX** — перестроение индекса; убирает bloat и повреждения. **REINDEX CONCURRENTLY** — построение нового индекса и подмена старого без блокировки записи; при сбое удалить лишний индекс (_ccnew или _ccold) по pg_index.indisvalid и имени.
- **Fillfactor индекса** &lt; 100 — оставить место в страницах; меньше split и чуть меньше bloat при частых обновлениях; применяется при создании/REINDEX.

### 33.3. Bloat индекса

**Цель раздела.**  
Понять **bloat (раздувание) индекса**: после массовых обновлений и удалений в страницах индекса остаётся много **мёртвых** записей и пустого места. VACUUM не сжимает индекс; уменьшить bloat можно **REINDEX** (или REINDEX CONCURRENTLY).

#### Термины

- **Bloat индекса** — увеличение размера индекса из-за **мёртвых** записей (удалённые или обновлённые ключи) и фрагментации страниц. Страницы индекса не переиспользуются так же, как в heap — VACUUM по индексу только помечает мёртвые записи и обновляет статистику, но **не освобождает** страницы и не сжимает индекс. Со временем индекс может занимать в разы больше места, чем «полезное» содержимое.
- **Причины:** массовые UPDATE по столбцам, входящим в индекс (каждый UPDATE = удаление старой записи в индексе + вставка новой); массовые DELETE. Частое автообновление одного и того же ключа тоже раздувает индекс.

#### Теория и правила

- **Обнаружение:** сравнение размера индекса (pg_relation_size) с оценкой «полезного» размера; расширения вроде pgstattuple дают долю мертвых записей. Рост idx_scan при росте размера индекса без роста данных — косвенный признак bloat.
- **Решение:** **REINDEX** или **REINDEX CONCURRENTLY** для перестроения индекса; после перестроения размер индекса уменьшается. Для таблицы — VACUUM FULL перестраивает таблицу и индексы, но блокирует таблицу; обычно предпочитают REINDEX по индексам.

#### Простыми словами

**Bloat индекса** — индекс «раздулся»: много пустого и мёртвого. VACUUM индекс не сжимает. Уменьшить bloat — перестроить индекс (REINDEX).

#### Типичная ошибка

**Думать, что VACUUM «почистит» раздутый индекс.** По таблице (heap) VACUUM помечает мёртвые строки и может переиспользовать место; по **индексу** в PostgreSQL VACUUM только обновляет статистику и помечает мёртвые записи в индексе, но **не сжимает** страницы индекса и не освобождает их. Раздутый индекс так и остаётся большим. Чтобы уменьшить размер индекса после массовых UPDATE/DELETE по индексируемым столбцам — нужен **REINDEX** (или REINDEX CONCURRENTLY), а не повторный VACUUM.

#### Картинка в голове

**Bloat индекса** — как **тетрадь, в которой много вычеркнутых строк**: ты зачёркиваешь старое и пишешь новое ниже, но **не вырываешь страницы** и не сжимаешь текст. Тетрадь толстая, но половина — зачёркнутое и пустое место. VACUUM по индексу только «отмечает», что строки мёртвые, но **не переписывает** тетрадь заново. Чтобы тетрадь снова стала компактной — **переписать** её (REINDEX): новая тетрадь только с актуальными записями, без «трупов» и дыр.

#### Пошагово: откуда берётся bloat и что с ним делать

Индекс по столбцу `status`; часто выполняются UPDATE, меняющие `status` с 'new' на 'done'.

1. В B-tree каждая запись индекса — пара (ключ, TID). При **UPDATE** строки по столбцу, входящему в индекс, **старая** запись в индексе помечается мёртвой (удаляется логически), **новая** запись вставляется. Страница индекса при этом **не сжимается** — мёртвая запись просто помечена; место остаётся занятым.
2. После миллионов таких обновлений многие страницы индекса заполнены мёртвыми записями и фрагментами — индекс занимает в 2–3 раза больше места, чем «полезное» содержимое. Это и есть **bloat**.
3. **VACUUM** по индексу только обновляет статистику и помечает мёртвые записи; страницы не перезаписываются. **Уменьшить** размер индекса можно только **REINDEX** (или REINDEX CONCURRENTLY): построить новый индекс с нуля и заменить старый.

> **Важно.** Bloat индекса не «рассасывается» со временем сам — в отличие от heap, где VACUUM может переиспользовать место. Единственный способ вернуть компактный индекс — перестроить его. На таблицах с частыми UPDATE по индексированным столбцам bloat накапливается; периодический REINDEX CONCURRENTLY — нормальная практика.

#### Проверь себя (33.3)

1. Почему массовые UPDATE по столбцу, входящему в индекс, раздувают индекс?  
<details><summary>Ответ</summary> Каждый UPDATE такой строки приводит к **удалению** старой записи в индексе (ключ + старый TID) и **вставке** новой (новый ключ и/или новый TID). Удаление в B-tree обычно логическое — запись помечается мёртвой; страница не сжимается. Вставка занимает новое место (или переиспользует место мёртвых в той же странице, но фрагментация растёт). Со временем накапливаются мёртвые записи и пустое место — индекс раздувается. VACUUM индекс не сжимает, поэтому bloat остаётся до REINDEX.</details>

2. Можно ли уменьшить bloat индекса, несколько раз выполнив VACUUM на таблице?  
<details><summary>Ответ</summary> **Нет.** VACUUM по таблице (heap) помечает мёртвые строки и может переиспользовать место; по **индексу** в PostgreSQL VACUUM только обновляет статистику и помечает мёртвые записи в индексе, но **не сжимает** страницы индекса и не освобождает их. Раздутый индекс остаётся большим. Уменьшить размер индекса можно только **REINDEX** (или REINDEX CONCURRENTLY) — перестроить индекс с нуля; тогда новый индекс будет содержать только актуальные записи, без «трупов» и дыр.</details>

**Три пункта, чтобы не забыть (bloat индекса).** 1) **Bloat** — раздувание индекса из-за мёртвых записей (после UPDATE/DELETE по индексируемым столбцам) и фрагментации; страницы индекса не сжимаются при «очистке». 2) **VACUUM** по индексу **не уменьшает** его размер — только обновляет статистику; ждать «рассасывания» bloat бесполезно. 3) **Решение:** перестроить индекс — **REINDEX** или **REINDEX CONCURRENTLY**; после перестроения размер индекса станет соответствовать актуальному объёму данных.

**Ещё раз самыми простыми словами:** bloat индекса — когда внутри индекса много «трупов» записей и дыр после обновлений и удалений. VACUUM по индексу их не убирает. Чтобы индекс снова стал компактным — перестроить его (REINDEX).

#### Запомните

- **Bloat индекса** — раздувание из-за мёртвых записей и фрагментации; VACUUM не сжимает индекс. **Решение:** REINDEX (или REINDEX CONCURRENTLY).

### 33.4. CLUSTER

**Цель раздела.**  
Понять **CLUSTER** — физическая **пересортировка таблицы** по индексу: строки на диске перезаписываются в порядке ключа индекса. Улучшает **корреляцию** (физический порядок совпадает с логическим) и эффективность **index-only scan** и последовательного чтения по диапазону. **Блокирует** таблицу на время выполнения.

#### Термины

- **CLUSTER** — команда **CLUSTER table_name USING index_name** (или CLUSTER table_name по индексу по умолчанию). Таблица **перезаписывается** на диске в порядке ключа указанного индекса. После CLUSTER физический порядок строк близок к порядку по индексу — высокая корреляция.
- **Корреляция** — статистика в pg_stats: насколько порядок значений в столбце на диске совпадает с порядком по возрастанию (1 — идеально по возрастанию, -1 — по убыванию, 0 — случайный). Высокая корреляция выгодна для index-only scan и для BRIN.

#### Теория и правила

- **Когда использовать:** после массовой загрузки или когда таблица сильно фрагментирована; для улучшения последующих index-only scan и диапазонных запросов. **Минус:** эксклюзивная блокировка таблицы на время CLUSTER; на больших таблицах — долго.
- **Одноразово:** после CLUSTER порядок сохраняется только до следующих вставок/обновлений; со временем корреляция падает. Повторять CLUSTER периодически или полагаться на естественный порядок (вставки в конец по времени).

#### Простыми словами

**CLUSTER** — «переложить страницы книги» в порядке указателя (индекса). После этого чтение по индексу идёт почти последовательно по диску — быстрее. Таблица на время перезаписи блокируется.

#### Картинка в голове

**CLUSTER** — как **переставить страницы книги так, чтобы они шли в том же порядке, что и в указателе**. Сейчас в книге: страница 3, потом 15, потом 1, потом 8… (вразнобой). В указателе порядок: 1, 2, 3, 4… CLUSTER переписывает книгу: сначала все «первые» по указателю, потом «вторые» и т.д. После этого, листая указатель подряд, ты открываешь страницы книги **тоже подряд** — меньше прыжков, быстрее. Минус: пока книгу переставляют, **её нельзя читать и дописывать** (таблица заблокирована). И со временем новые вставки снова разбросают порядок — эффект не вечный.

#### Как запомнить

**CLUSTER** = пересортировка таблицы по индексу; повышает корреляцию и эффективность index-only scan; **блокирует** таблицу.

#### Проверь себя (33.4)

1. После CLUSTER таблица упорядочена по индексу. Надолго ли это?  
<details><summary>Ответ</summary> **Нет.** Новые **вставки** и **обновления** снова разбрасывают строки по разным страницам; физический порядок постепенно перестаёт совпадать с порядком по индексу (корреляция падает). CLUSTER — **разовая** операция; эффект сохраняется до следующих активных изменений. Для таблиц с постоянной записью корреляция со временем снижается; CLUSTER можно повторять периодически (но он блокирует таблицу) или полагаться на естественный порядок (например, вставки по времени в конец).</details>

2. Зачем делать CLUSTER по индексу и какой главный минус?  
<details><summary>Ответ</summary> **Зачем:** чтобы **физический порядок** строк на диске совпадал с порядком по ключу индекса (высокая корреляция). Тогда при чтении по этому индексу (диапазон, index-only scan) страницы таблицы читаются **почти последовательно** — меньше случайного I/O, быстрее. **Минус:** на время выполнения CLUSTER таблица **блокируется** (эксклюзивная блокировка) — запись и чтение недоступны; на больших таблицах это может занимать часы. Поэтому CLUSTER часто делают в окно обслуживания или используют REINDEX/другие меры без полной перезаписи таблицы.</details>

3. Сохраняется ли порядок строк на диске после CLUSTER до следующих вставок?  
<details><summary>Ответ</summary> **Нет, не навсегда.** Новые **вставки** и **обновления** снова размещают строки по разным страницам (по FSM и наличию места); физический порядок постепенно перестаёт совпадать с порядком по индексу (корреляция падает). CLUSTER даёт **разовый** эффект; для таблиц с активной записью его периодически повторяют или мирятся с падением корреляции со временем.</details>

**Три пункта, чтобы не забыть (CLUSTER).** 1) **CLUSTER** = перезаписать таблицу на диске **в порядке ключа указанного индекса**; после этого порядок строк на диске близок к порядку по индексу (высокая корреляция). 2) **Плюс:** чтение по этому индексу идёт почти последовательно по диску — быстрее index-only scan и диапазонные запросы. 3) **Минус:** таблица **блокируется** на время выполнения; эффект со временем размывается новыми вставками/обновлениями — порядок снова фрагментируется.

**Ещё раз самыми простыми словами:** CLUSTER — «переложить все страницы таблицы» так, чтобы порядок на диске совпадал с порядком по индексу. После этого чтение по этому индексу идёт почти последовательно по диску — быстрее. Минус: таблица на время перезаписи полностью блокируется.

#### Запомните

- **CLUSTER** — физическая пересортировка таблицы по индексу; повышает корреляцию; блокирует таблицу на время выполнения.

### 33.5. Партиционирование и индексы

**Цель раздела.**  
Понять, что при **партиционировании** таблицы в PostgreSQL индексы создаются **локальными** — отдельный индекс на каждой партиции. Запрос с условием по ключу партиции сначала отсекает партиции (partition pruning), затем сканирует только индексы оставшихся партиций. После раздела ты будешь знать, почему «один» CREATE INDEX на партиционированную таблицу даёт несколько индексов и как это сочетается с отсечением партиций.

#### Термины (расшифровка)

- **Локальный индекс (local index)** — индекс, построенный **только на данных одной партиции**. При **CREATE INDEX** на **партиционированной** таблице в PostgreSQL автоматически создаётся **отдельный индекс на каждой партиции** — у каждой партиции свой B-tree (или другой тип) по указанным столбцам. Имена индексов партиций обычно генерируются (например, имя_индекса_1, _2, или по имени партиции). **Плюсы:** при удалении партиции удаляется и её индекс; при добавлении новой партиции для неё можно создать индекс отдельно; запросы с условием по ключу партиции обращаются только к нужным партициям и их индексам.
- **Глобальный индекс (global index)** — один индекс, содержащий ключи **по всей таблице** (все партиции). По такому индексу можно искать без знания того, в какой партиции лежит строка. В **стандартном PostgreSQL глобальных индексов нет** — при CREATE INDEX на партиционированной таблице всегда создаются локальные индексы. В других СУБД (например, Oracle) глобальные индексы поддерживаются; в PostgreSQL для «общего» поиска используют локальные индексы на каждой партиции и при необходимости объединяют результаты (или запрос ограничивают ключом партиции).

#### Теория и правила (подробно)

- **Partition pruning:** если в запросе есть условие по **ключу партиционирования** (например, по created_at при партициях по месяцам), планировщик **отсекает** партиции, в которых заведомо нет подходящих строк. Сканируются только оставшиеся партиции. На каждой из них используется **локальный** индекс (если условие по индексированному столбцу подходит). Итог: меньше партиций, меньше индексов в работе — быстрее.
- **Создание индекса:** `CREATE INDEX idx_orders_created ON orders (created_at);` на таблице orders, партиционированной по created_at (RANGE по месяцам), создаёт индекс **на каждой партиции** с тем же определением (created_at). Новые партиции (например, добавленные позже) по умолчанию **не получают** этот индекс автоматически — индекс нужно создавать для новой партиции отдельно или использовать опции, если СУБД это поддерживает (в PostgreSQL 11+ индексы на партиционированной таблице при добавлении новой партиции могут настраиваться через шаблон).

#### Пошагово: запрос к партиционированной таблице с локальными индексами

Таблица **orders** партиционирована по **created_at** по месяцам (партиции: 2024-01, 2024-02, 2024-03, …). На таблице создан индекс: **CREATE INDEX ON orders (user_id)** — на каждой партиции создаётся свой индекс по user_id. Запрос: `SELECT * FROM orders WHERE created_at BETWEEN '2024-02-01' AND '2024-02-28' AND user_id = 5;`

1. **Partition pruning:** по условию created_at отсекаются все партиции, кроме **2024-02**. Остаётся одна партиция (февраль 2024).
2. На этой партиции выполняется поиск по **локальному** индексу (user_id): в индексе партиции 2024-02 ищем user_id = 5.
3. По TID читаются строки только из партиции 2024-02. Партиции 2024-01, 2024-03 и остальные **не сканируются** и их индексы не открываются.

**Итог:** один запрос использует одну партицию и один локальный индекс. Без партиционирования был бы один большой индекс по всей таблице; с партиционированием — много маленьких индексов, и в каждом запросе участвуют только те, что относятся к отобранным партициям.

> **Важно.** В PostgreSQL на партиционированной таблице **нет выбора** «локальный или глобальный» — всегда **локальные** индексы. Поэтому запрос **без** условия по ключу партиции (например, только WHERE user_id = 5) вынужден обходить **все** партиции и все их локальные индексы (или делать seq scan по всем партициям). Чтобы максимально использовать партиционирование, в запросах по возможности задают условие по ключу партиции (created_at, id диапазон и т.д.).

#### Проверь себя (33.5)

1. Что создаётся в PostgreSQL при CREATE INDEX на партиционированной таблице — один индекс или несколько?  
<details><summary>Ответ</summary> **Несколько** — отдельный индекс **на каждой партиции**. Структура индекса (столбцы, тип B-tree и т.д.) одна и та же, но данные каждого индекса — только строки своей партиции. В каталоге будут отдельные объекты индексов (по одному на партицию). Глобального «одного индекса на всю таблицу» в PostgreSQL для партиционированных таблиц нет.</details>

2. Запрос к партиционированной по дате таблице: WHERE created_at >= '2024-06-01' AND status = 'active'. Есть локальный индекс по (status) на каждой партиции. Как планировщик может выполнить запрос?  
<details><summary>Ответ</summary> Сначала **partition pruning** по created_at: отсекаются партиции с датами раньше июня 2024; остаются партиции за июнь и позже. На каждой из оставшихся партиций выполняется поиск по **локальному** индексу (status) для status = 'active', затем по TID читаются строки. Партиции до июня не сканируются и их индексы не используются. Чем уже диапазон по created_at, тем меньше партиций и индексов задействовано.</details>

3. Есть ли в PostgreSQL глобальный индекс по партиционированной таблице (один индекс на все партиции)?  
<details><summary>Ответ</summary> **Нет.** В стандартном PostgreSQL при CREATE INDEX на партиционированной таблице создаются **только локальные** индексы — отдельный индекс на каждой партиции с тем же определением. **Глобального** индекса (один индекс по всем партициям) нет. Поэтому запрос **без** условия по ключу партиции (например, только WHERE user_id = 5) вынужден обращаться ко **всем** партициям и их локальным индексам (или к полному скану партиций).</details>

**Ещё раз самыми простыми словами:** на партиционированной таблице «один» индекс в определении — на самом деле индекс **на каждую партицию** (локальные индексы). Запрос с условием по ключу партиции сначала отбрасывает лишние партиции, потом ищет только по индексам оставшихся. В PostgreSQL глобального индекса по всем партициям нет — только локальные.

#### Запомните

- На партиционированной таблице в PostgreSQL индексы по умолчанию **локальные** (на каждой партиции). Partition pruning отсекает партиции; индексы используются по нужным партициям.
