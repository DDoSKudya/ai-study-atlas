## 29. Назначение индексов

### 29.1. Зачем нужны индексы

**Цель раздела.**  
Понять, **зачем** создают индексы: ускорение поиска по условию (WHERE), сортировки (ORDER BY), группировки (GROUP BY) и JOIN; и почему без индекса СУБД вынуждена делать полное сканирование таблицы. После раздела ты будешь чётко представлять разницу «с индексом» и «без индекса» в цифрах и по шагам.

#### Термины (расшифровка по порядку)

- **Индекс (index)** — дополнительная структура данных по одному или нескольким столбцам таблицы, позволяющая быстро находить строки по значению ключа (или по диапазону) без просмотра всей таблицы. Индекс хранит **ключевые значения** и **указатели на строки** (в PostgreSQL — TID: номер страницы и смещение в странице). Простая аналогия: алфавитный указатель в конце книги — по фамилии находишь номер страницы, а не листаешь всю книгу с начала до конца.
- **Полное сканирование таблицы (full table scan, sequential scan)** — способ выполнения запроса, при котором СУБД читает **все строки** таблицы подряд (все страницы heap) и для каждой проверяет условие WHERE. Без подходящего индекса это **единственный** способ найти строки по условию: другой «короткой дороги» нет. На больших таблицах полное сканирование очень медленно: много чтений с диска (I/O) и много работы процессора (CPU).
- **Селективность (selectivity)** — доля строк таблицы, удовлетворяющих условию. **Высокая селективность** — условие отбирает **мало** строк (например, «id = 1» на таблице в миллион строк — одна строка). **Низкая селективность** — отбирает **много** строк (например, «status = 'active'», если 80% строк активны — 800 000 из миллиона). Индекс особенно выгоден при высокой селективности: мы быстро переходим к немногим подходящим строкам и почти не читаем лишнего.

#### Теория и правила (подробно)

- **Без индекса** по столбцу, участвующему в WHERE, СУБД может только **просканировать всю таблицу**: прочитать каждую страницу heap, перебрать все строки и применить условие. Число прочитанных страниц = **размер таблицы в страницах**. Время растёт **линейно** с размером таблицы: в два раза больше таблица — в два раза дольше поиск. Это называют **O(N)** — «порядок N», где N — число строк (или страниц).
- **С индексом** СУБД переходит по структуре индекса (например, B-tree) к нужному ключу и получает указатели (TID) на строки. Число обращений к страницам индекса пропорционально **логарифму** числа строк (обычно 3–4 шага для миллиона записей), плюс чтение только тех страниц таблицы, где лежат **найденные** строки. Это **O(log N) + k**, где k — число отобранных строк. На больших таблицах выигрыш огромный: вместо сотен тысяч страниц — десятки.
- Индекс ускоряет не только **WHERE**, но и:
  - **ORDER BY по ключу индекса** — листья B-tree уже упорядочены по ключу; СУБД может читать их по порядку и отдавать строки уже отсортированными. Отдельная операция **Sort** (сортировка в памяти или на диске) не нужна.
  - **GROUP BY по ключу индекса** — при чтении в порядке ключа одинаковые значения идут подряд; группировку можно делать «на ходу», без предварительной сортировки или большого хеша.
  - **JOIN** — если соединение идёт по проиндексированному столбцу (например, user_id), для каждой строки одной таблицы вторая таблица **ищется по индексу** (быстро), а не сканируется целиком (медленно).

```mermaid
flowchart LR
  Q[Запрос: WHERE key = ...] --> C{Есть подходящий индекс?}
  C -->|Нет| Seq[Sequential Scan\nчитать все страницы heap\nпроверять условие для каждой строки]
  C -->|Да| Idx[Index Scan\nобход B-tree (log N)\nполучить TID]
  Idx --> Heap[Heap fetch\nпрочитать нужные страницы по TID]
  Seq --> R[Результат]
  Heap --> R
```

#### Пошагово: поиск без индекса и с индексом (с числами)

**Без индекса.** Запрос: `SELECT * FROM users WHERE email = 'a@b.com'`. Таблица users: **1 миллион строк**, размер таблицы **1 GB**, страница 8 KB.

1. СУБД выбирает план: **Sequential Scan** (последовательное сканирование) по таблице users.
2. Читает **первую страницу** (8 KB), перебирает все строки на ней (например, 100 строк), проверяет `email = 'a@b.com'`. Не нашли — идём дальше.
3. Читает **вторую страницу**, перебирает строки, проверяет условие. Снова не нашли.
4. … повторяет для **каждой** страницы таблицы. Всего страниц: 1 GB / 8 KB = **131 072 страницы**. Каждую нужно прочитать с диска (или из кэша), каждую проверить.
5. В конце концов находит строку с email = 'a@b.com' (допустим, на странице 50 000) и возвращает её.

**Итог без индекса:** прочитано **131 072 страницы** (вся таблица). Даже если нужная строка была на первой сотне страниц, СУБД без индекса **не знает**, где она, и вынуждена просмотреть всё. Время и I/O — как при чтении всего 1 GB.

**С индексом по email.** Тот же запрос, та же таблица.

1. СУБД выбирает план: **Index Scan** по индексу на email.
2. Открывает **корень** B-tree индекса (одна страница). В корне — диапазоны ключей; по ним понимает, в какую ветку идти. Сравнивает 'a@b.com' с ключами — переходит по указателю на следующий узел.
3. Открывает **внутренний узел** (вторая страница). Снова сравнение — переходит в следующий узел. Так повторяется **3–4 раза** (глубина B-tree для миллиона записей).
4. Попадает в **лист** индекса. В листе ключи и TID упорядочены; находит ключ 'a@b.com' и читает **TID** (например: страница таблицы 50000, смещение 120).
5. По TID читает **одну страницу** таблицы (страницу 50000), по смещению находит строку и возвращает её.

**Итог с индексом:** прочитано **3–4 страницы индекса** + **1 страница таблицы** = примерно **5 страниц** вместо 131 072. Ускорение в десятки тысяч раз по числу прочитанных страниц.

> **Важно.** Без индекса СУБД **не знает**, на какой странице лежит строка с email = 'a@b.com'. Данные в таблице (heap) хранятся в порядке вставки и размещения на страницах, а не в порядке email. Поэтому единственный способ — проверить **все** строки подряд. Индекс как раз и хранит «значение email → номер страницы и смещение», то есть даёт «адрес» строки.

#### Что будет, если не использовать индекс (и когда это заметно)

- На **маленькой** таблице (несколько сотен строк, 1–2 страницы) разница может быть почти незаметна: прочитать 2 страницы подряд или 2 страницы индекса + 1 страницу таблицы — доли миллисекунды.
- На **большой** таблице (миллионы строк, гигабайты) без индекса каждый такой запрос читает **всю** таблицу. Десятки запросов в секунду превращаются в постоянную нагрузку на диск и в медленный ответ. С индексом тот же запрос читает считаные страницы — ответ приходит в миллисекундах.

#### Простыми словами (ещё раз)

**Индекс** — это «указатель»: по значению столбца (или комбинации столбцов) мы быстро переходим к нужным строкам, не просматривая всю таблицу. Без индекса база вынуждена «листать всю книгу» (полное сканирование); с индексом она «смотрит в оглавление» и переходит сразу к нужной странице. Индекс особенно полезен, когда условие отбирает **мало** строк (высокая селективность): мы сразу попадаем к ним и почти не читаем лишнего.

**Одной фразой:** индекс = быстрый способ найти строки по значению столбца без просмотра всей таблицы.

#### Картинка в голове

Книга (таблица) на 1000 страниц. Нужно найти все места, где упоминается «Иванов». **Без указателя:** листаешь все 1000 страниц подряд — долго и скучно. **С алфавитным указателем в конце:** открываешь указатель, находишь «Иванов» — там номера страниц 45, 120, 333; открываешь только эти три страницы. Индекс в БД — такой же «указатель» по значениям столбца: значение → адрес строки (TID).

#### Как запомнить

- **Индекс = быстрый переход к строкам по значению ключа.**  
- **Без индекса** — полное сканирование (вся таблица), время O(N).  
- **С индексом** — обход дерева (B-tree) и точечное чтение страниц таблицы, время O(log N) + k.  
- **Селективность высокая** (мало строк подходят) → индекс даёт большой выигрыш.

#### Проверь себя (29.1)

1. Почему без индекса по столбцу в WHERE СУБД вынуждена сканировать всю таблицу?  
<details><summary>Ответ</summary> Потому что **нет структуры**, по которой можно перейти к нужным строкам. Данные в таблице (heap) хранятся в порядке вставки/размещения на страницах, а не в порядке значений столбца. Единственный способ найти строки по условию — прочитать все страницы и проверить условие для каждой строки. Индекс как раз и даёт такую структуру: по значению ключа можно быстро найти указатели (TID) на строки.</details>

2. Что такое селективность и почему индекс выгоднее при высокой селективности?  
<details><summary>Ответ</summary> **Селективность** — доля строк, удовлетворяющих условию. Высокая селективность = условие отбирает **мало** строк (например, id = 1 на большой таблице). При высокой селективности индекс позволяет **сразу** перейти к немногим подходящим строкам и прочитать минимум страниц таблицы; при низкой (условие отбирает почти все строки) выгоднее может оказаться полное сканирование — меньше накладных расходов на обход индекса и множество случайных чтений.</details>

3. Как индекс помогает ORDER BY и GROUP BY?  
<details><summary>Ответ</summary> **ORDER BY** по столбцу(ам), по которому построен индекс: листья B-tree уже **упорядочены** по ключу. СУБД может читать данные из индекса по порядку и отдавать их без отдельной операции сортировки (Sort). **GROUP BY** по ключу индекса: при чтении в порядке ключа одинаковые значения идут подряд — группировка делается «на ходу», без предварительной сортировки или большого хеша.</details>

4. Таблица 10 млн строк, 10 GB. Запрос WHERE id = 1 (id уникален). Сколько примерно страниц прочитается без индекса и с B-tree индексом по id?  
<details><summary>Ответ</summary> **Без индекса:** СУБД вынуждена сканировать всю таблицу. Страниц: 10 GB / 8 KB ≈ **1 310 720 страниц** — все они читаются (или из кэша, или с диска). **С индексом по id:** обход B-tree — глубина примерно 4 уровня (log от 10 млн) → **4 страницы индекса** + **1 страница таблицы** по TID = **5 страниц**. Выигрыш по числу прочитанных страниц — в сотни тысяч раз.</details>

**Три пункта, чтобы не забыть (зачем индексы).** 1) **Без индекса** по столбцу из WHERE база **не знает**, где лежат подходящие строки — данные в таблице в порядке вставки, а не по значению столбца; единственный способ = прочитать **все** страницы (полное сканирование). 2) **С индексом** база по значению ключа быстро находит **указатели (TID)** на строки и читает только нужные страницы таблицы; число чтений ~ O(log N) + число отобранных строк. 3) Индекс выгоднее всего при **высокой селективности** (условие отбирает мало строк) — тогда читаем минимум лишнего; плюс индекс даёт «бесплатную» сортировку (ORDER BY по ключу) и помогает GROUP BY и JOIN.

#### Запомните

- **Индекс** ускоряет поиск по условию (WHERE), сортировку (ORDER BY по ключу), группировку (GROUP BY по ключу) и JOIN по проиндексированному столбцу.
- **Без индекса** СУБД делает **полное сканирование** таблицы — читает все страницы; время O(N).
- **С индексом** переход к строкам по структуре индекса — обычно O(log N) плюс чтение только отобранных строк. **Селективность** (доля отобранных строк) влияет на выгоду индекса.

**Ещё раз самыми простыми словами:** индекс — это «список: значение столбца → где лежит строка». Без него база перебирает все строки; с ним — смотрит в список и сразу идёт по адресу. На больших таблицах это даёт ускорение в тысячи раз.

### 29.2. Замедление DML

**Цель раздела.**  
Понять, что **каждая операция изменения данных (INSERT, UPDATE, DELETE)** должна обновить все индексы таблицы: при вставке — добавить запись в каждый индекс, при удалении — пометить или удалить запись, при обновлении индексируемых столбцов — изменить запись в индексе. Это замедляет запись и увеличивает объём WAL. Исключение — HOT-обновление в PostgreSQL, когда изменённые столбцы не входят ни в один индекс.

#### Термины

- **DML (Data Manipulation Language)** — операции изменения данных: INSERT, UPDATE, DELETE. В отличие от SELECT (чтение), они **модифицируют** таблицу и все её индексы.
- **Обслуживание индекса (index maintenance)** — при каждой вставке в таблицу СУБД добавляет соответствующую запись в каждый индекс (ключ + TID); при удалении — удаляет или помечает запись в индексе; при обновлении столбца, входящего в индекс, — обновляет запись в индексе (часто «удалить старую + вставить новую»). Всё это делается в той же транзакции, что и изменение строки.
- **HOT (Heap Only Tuple)** — в PostgreSQL обновление строки, при котором **новая версия** строки помещается на **ту же страницу** heap и **ни один из изменённых столбцов не входит ни в один индекс**. Тогда индексы **не обновляются** — все по-прежнему указывают на старый TID; цепочка версий на странице ведёт к новой версии. HOT-обновление не трогает индексы и быстрее; для него полезен fillfactor (оставлять место на странице под новые версии).

```mermaid
flowchart TB
  subgraph "Обычный UPDATE (меняем индексируемый столбец)"
    U1[Новая версия tuple] --> I1[Обновить индекс:\nудалить старую запись\n+ вставить новую]
  end

  subgraph "HOT UPDATE (индексы не трогаем)"
    H1[Старая версия tuple (TID0)] --> H2[Новая версия tuple (TID1)\nна той же странице]
    H3[Индекс всё ещё указывает на TID0] --> H1
    H1 -->|цепочка HOT| H2
  end
```

#### Теория и правила

- **INSERT:** для каждой вставленной строки СУБД записывает её в heap и **добавляет запись в каждый индекс** таблицы (ключ + TID новой строки). Чем больше индексов, тем больше работы и тем больше записей в WAL.
- **DELETE:** строка в heap помечается удалённой (или физически удаляется в зависимости от реализации); в **каждом индексе** запись, указывающая на эту строку, должна быть удалена или помечена. Иначе индекс будет содержать «мёртвые» указатели (сканирование вернёт TID, по которому строка уже не видна).
- **UPDATE:** если изменяется столбец, **не входящий ни в один индекс**, и новая версия строки помещается на ту же страницу (HOT) — индексы не обновляются. Если изменяется столбец, входящий в индекс, или новая версия не помещается на страницу — для каждого такого индекса выполняется «удалить старую запись + вставить новую» (по новому ключу и новому TID). Много индексов → много обновлений индексов при каждом таком UPDATE.
- **Итог:** больше индексов → быстрее чтение (SELECT) по разным условиям, но **медленнее запись** (INSERT/UPDATE/DELETE) и больше места на диске и в WAL. Баланс зависит от соотношения чтений и записей.

#### Пошагово: что происходит при INSERT с тремя индексами

Таблица `orders (id, user_id, created_at)` с индексами: PRIMARY KEY (id), индекс на user_id, индекс на created_at.

1. Выполняется `INSERT INTO orders (id, user_id, created_at) VALUES (1, 100, NOW())`.
2. СУБД записывает **строку в heap** (находит страницу с свободным местом по FSM, записывает строку, обновляет visibility и т.д.).
3. В **индекс по id** (PK) добавляется запись: ключ 1, TID новой строки.
4. В **индекс по user_id** добавляется запись: ключ 100, TID новой строки.
5. В **индекс по created_at** добавляется запись: ключ текущее время, TID новой строки.
6. Все изменения (heap + три индекса) попадают в **WAL** для долговечности.
7. Транзакция коммитится.

Если бы индексов не было — шаги 3–5 отсутствовали бы; вставка была бы быстрее и с меньшим объёмом WAL.

#### Простыми словами

**Каждый индекс — это отдельная «копия» части информации о строках (ключ + указатель).** Когда мы вставляем, удаляем или обновляем строку, нужно привести в соответствие **все** такие копии. Чем больше индексов, тем больше работы при каждой записи и тем больше «лишнего» в журнале (WAL). Исключение — HOT-обновление: если мы меняем только столбец, которого нет ни в одном индексе, и новая версия строки помещается на ту же страницу, индексы не трогаем.

#### Картинка в голове

Таблица — главная книга; индексы — несколько алфавитных указателей в конце (по имени, по дате, по номеру). Добавили новую запись в книгу — нужно **добавить её во все указатели**. Удалили запись — нужно **вычеркнуть из всех указателей**. Обновили только «заметку» в записи, которая ни в один указатель не входит — указатели можно не трогать (HOT). Чем больше указателей, тем дольше каждое изменение.

#### Как запомнить

**DML обновляет таблицу и все индексы.** INSERT → добавить запись в каждый индекс. DELETE → удалить/пометить в каждом индексе. UPDATE по индексируемому столбцу → обновить запись в индексе (или удалить + вставить). **HOT** = обновление без изменения индексов (изменённые столбцы не в индексах, новая версия на той же странице).

#### Типичная ошибка

**Думать, что «индексы нужны только для скорости чтения».** Да, они ускоряют чтение, но **каждая запись** (INSERT/UPDATE/DELETE) должна обновить все индексы. На таблицах с очень частыми вставками (например, лог событий) слишком много индексов сильно замедлит запись. Нужен баланс: индексировать только то, по чему реально ищут и сортируют.

#### Проверь себя (29.2)

1. Почему INSERT на таблице с пятью индексами медленнее, чем на таблице с одним индексом?  
<details><summary>Ответ</summary> Потому что при каждой вставке СУБД должна **добавить запись в каждый из пяти индексов** (ключ + TID). Больше индексов — больше операций вставки в структуры индексов, больше страниц индексов затрагивается, больше записей в WAL. Один индекс — одна вставка в индекс; пять индексов — пять вставок.</details>

2. Что такое HOT-обновление и при каком условии индексы не обновляются при UPDATE?  
<details><summary>Ответ</summary> **HOT (Heap Only Tuple)** — обновление, при котором новая версия строки помещается **на ту же страницу** heap и **ни один из изменённых столбцов не входит ни в один индекс**. Тогда все индексы по-прежнему указывают на старый TID; в heap на странице есть цепочка версий, и видимая версия — новая. Индексы не обновляются, запись в индексе не трогается — ускорение и меньше WAL.</details>

3. При DELETE нужно ли обновлять индексы? Зачем?  
<details><summary>Ответ</summary> **Да.** В каждом индексе есть записи (ключ + TID), указывающие на удаляемую строку. Если их не удалить или не пометить как недействительные, при следующем сканировании индекса СУБД получит TID «мёртвой» строки и будет вынуждена проверять видимость (и не найдёт строку). Чтобы индекс оставался консистентным и не вёл к удалённым строкам, записи в индексе при DELETE обновляются (удаляются или помечаются).</details>

4. Таблица с тремя индексами: при одной вставке сколько раз СУБД «трогает» индексы (вставок в индекс)? А при одном UPDATE по столбцу, который входит в два из трёх индексов?  
<details><summary>Ответ</summary> **При одной вставке:** СУБД добавляет **одну запись в каждый из трёх индексов** — итого **3 вставки в индексы** (плюс одна запись в таблицу). **При одном UPDATE** по столбцу, входящему в два индекса: в этих **двух** индексах выполняется «удалить старую запись + вставить новую» (по новому ключу и новому TID). Третий индекс (в котором этот столбец не участвует) **не обновляется**. Итого: 2 индекса обновлены, 1 не тронут.</details>

**Три пункта, чтобы не забыть (замедление DML).** 1) **INSERT:** одна новая строка → одна новая запись в **каждый** индекс таблицы (ключ + TID); пять индексов = пять вставок в индексы при каждой вставке в таблицу. 2) **UPDATE** по столбцу, входящему в индекс: в этом индексе «удалить старую запись + вставить новую»; если меняем столбец, которого **нет** ни в одном индексе, и новая версия помещается на ту же страницу — **HOT**, индексы не трогаем. 3) **DELETE:** в каждом индексе нужно удалить или пометить запись, указывающую на эту строку, иначе индекс будет вести к «мёртвым» TID.

#### Запомните

- **Каждый INSERT/UPDATE/DELETE** обновляет **все индексы** таблицы (добавить/удалить/изменить запись в индексе). Больше индексов → медленнее запись и больше WAL.
- **HOT-обновление** (PostgreSQL): изменённые столбцы не в индексах, новая версия на той же странице — индексы **не обновляются**.

**Ещё раз самыми простыми словами:** каждый индекс — это отдельный «список» (ключ → TID). Добавили строку — вписали во все списки. Удалили — вычеркнули из всех. Обновили столбец из индекса — в этом индексе поменяли запись. Чем больше списков (индексов), тем больше работы при каждой записи. HOT — когда меняем только то, чего нет ни в одном списке, тогда списки не трогаем.

### 29.3. Когда не индексировать

**Цель раздела.**  
Узнать ситуации, когда индекс **не даёт выгоды** или даже вредит: маленькие таблицы; столбцы с очень низкой селективностью (почти все значения одинаковы); столбцы, которые редко или никогда не участвуют в WHERE, ORDER BY, GROUP BY или JOIN. После раздела ты будешь понимать, почему «индекс на всё подряд» — плохая идея и когда от индекса лучше отказаться.

#### Термины (расшифровка)

- **Маленькая таблица** — таблица, умещающаяся в очень небольшое число страниц (например, несколько десятков или сотен строк — 1–10 страниц по 8 KB). Полное сканирование такой таблицы занимает **одно или несколько** чтений страниц (очень быстро). Обход индекса (корень → внутренние узлы → лист) плюс чтение страниц таблицы по TID может оказаться **дороже**: несколько чтений индекса + одно чтение таблицы — накладные расходы и случайный I/O. Планировщик часто выбирает **sequential scan** для маленьких таблиц даже при наличии индекса, потому что полный скан дешевле по оценке стоимости.
- **Низкая селективность** — условие по столбцу отбирает **большую долю** строк (например, «status = 'active'», когда 90% строк активны — 900 000 из миллиона). Индекс по такому столбцу вернёт очень много TID; СУБД придётся прочитать **почти все страницы таблицы** по TID (вразнобой — случайные чтения). Один **последовательный** полный скан таблицы может быть быстрее и дешевле, чем обход индекса + множество случайных чтений. Планировщик может предпочесть seq scan.
- **Редко в условиях** — столбец почти никогда не используется в WHERE, ORDER BY, GROUP BY или в условии JOIN. Индекс по нему будет **обновляться при каждой** вставке/обновлении/удалении (затраты времени и WAL), но **использоваться** редко или никогда — лишние затраты на запись и место на диске без пользы для запросов.

#### Теория и правила (подробно)

- **Маленькие таблицы:** ориентировочно — таблицы, занимающие меньше нескольких страниц (например, меньше 8–32 KB в PostgreSQL, то есть 1–4 страницы). Для них стоимость полного скана **мала** (1–4 чтения страниц подряд). Стоимость index scan: обход дерева (2–3 страницы индекса) + чтение страницы таблицы по TID (случайное чтение). На очень маленькой таблице накладные расходы и модель стоимости могут дать **более высокую** оценку для index scan — планировщик выберет seq scan. **Правило:** не создавать индексы «на всякий случай» на очень маленьких таблицах; при росте таблицы индекс станет выгоден, и его можно добавить позже.
- **Низкая селективность:** если условие по столбцу отбирает, скажем, **больше 5–10%** строк (зависит от СУБД и настроек стоимости random vs sequential), планировщик может выбрать sequential scan. Индекс по такому столбцу может использоваться редко или никогда. Не значит, что индекс «запрещён» — но выгода сомнительна, а затраты на запись при каждой вставке/обновлении/удалении остаются.
- **Столбец не в условиях:** если столбец не участвует в фильтрации (WHERE), сортировке (ORDER BY), группировке (GROUP BY) или в условии JOIN, индекс по нему **не ускорит ни один запрос** — запросы по этому столбцу не идут. Зато при каждой INSERT/UPDATE/DELETE СУБД будет обновлять этот индекс. **Правило:** индексировать только столбцы, которые **реально** участвуют в запросах (WHERE, ORDER BY, GROUP BY, JOIN).

> **Важно.** Индекс создаётся **не «на всякий случай»**, а под **конкретные запросы**. Если по столбцу никто не ищет и не сортирует — индекс только съедает место и замедляет запись. Сначала смотри: какие запросы медленные и по каким столбцам идут условия и сортировка; потом добавляй индексы под эти запросы.

#### Что будет, если на маленькой таблице (100 строк) создать пять индексов

- **Чтение:** планировщик может вообще **не использовать** часть индексов — полный скан 100 строк (1–2 страницы) настолько дёшев, что index scan не выигрывает.
- **Запись:** каждая вставка будет обновлять **все пять** индексов — вставка станет в несколько раз медленнее, чем с одним индексом (PK). Пользы от четырёх «лишних» индексов на такой таблице почти нет, а вред при записи — есть.

#### Простыми словами

Не стоит индексировать **всё подряд**. На очень маленькой таблице «пролистать всё» быстрее, чем пользоваться указателем. На столбце, где почти все значения одинаковы (например, «пол = мужской» на таблице сотрудников с 95% мужчин), индекс почти не отсекает строк и может не использоваться. И не нужен индекс по столбцу, по которому никто не ищет и не сортирует — он только будет тормозить вставки и занимать место.

#### Картинка в голове

**Маленькая таблица** — одна страница с 20 стройками. Открыть одну страницу и пробежать глазами быстрее, чем идти в указатель, искать ключ и потом открывать ту же страницу по номеру. **Низкая селективность** — указатель «все фамилии на букву А» в книге, где половина фамилий на А: придётся открыть полкниги по номерам страниц — проще прочитать книгу подряд. **Редко в условиях** — указатель по «цвету обложки», по которому никто не ищет; он только занимает место и его нужно обновлять при каждом изменении.

#### Как запомнить

**Не индексировать:** 1) очень маленькие таблицы (полный скан дёшев); 2) столбцы с очень низкой селективностью (условие отбирает большую долю строк); 3) столбцы, которые не используются в WHERE, ORDER BY, GROUP BY, JOIN.

#### Проверь себя (29.3)

1. Почему на маленькой таблице планировщик может выбрать sequential scan даже при наличии индекса?  
<details><summary>Ответ</summary> Потому что **полное сканирование** маленькой таблицы — это одно или несколько последовательных чтений страниц (очень быстро). **Index scan** требует обхода дерева индекса (несколько чтений страниц индекса) и затем **случайных** чтений страниц таблицы по TID. Накладные расходы и случайный I/O могут сделать индексный доступ **дороже** в пересчёте на cost, чем один быстрый последовательный скан. Планировщик сравнивает стоимости и выбирает seq scan.</details>

2. Что такое низкая селективность и почему индекс по такому столбцу может не использоваться?  
<details><summary>Ответ</summary> **Низкая селективность** — условие отбирает **большую долю** строк (например, 50% или 80%). Индекс по такому столбцу вернёт очень много TID; СУБД придётся прочитать много страниц таблицы (часто вразнобой — случайный I/O). Стоимость такого доступа может быть **выше**, чем последовательное чтение всей таблицы одним проходом. Планировщик тогда выбирает sequential scan, и индекс «не используется».</details>

3. Зачем не создавать индекс по столбцу, который не участвует в WHERE, ORDER BY, GROUP BY и JOIN?  
<details><summary>Ответ</summary> Потому что такой индекс **не ускорит ни один запрос** — запросы по этому столбцу не идут. Зато при **каждой** INSERT/UPDATE/DELETE СУБД будет обновлять этот индекс (затраты времени и WAL) и индекс будет **занимать место** на диске. Итог: только затраты, никакой пользы. Индексы стоит создавать под **реальные** условия и порядки сортировки в запросах.</details>

4. Таблица 500 строк (примерно 2 страницы). Есть индекс по столбцу status. Запрос WHERE status = 'active' (подходит 200 строк). Почему планировщик может выбрать sequential scan, а не index scan?  
<details><summary>Ответ</summary> Потому что **полное сканирование** 2 страниц (последовательно) очень дёшево — два чтения подряд. **Index scan** требует обхода дерева индекса (несколько чтений страниц индекса) и затем **множества** чтений страниц таблицы по TID (200 строк могут лежать на разных страницах — случайные чтения). Накладные расходы и случайный I/O могут сделать индексный доступ **дороже** в пересчёте на cost, чем один быстрый последовательный скан двух страниц. Планировщик сравнивает стоимости и выбирает seq scan.</details>

**Три пункта, чтобы не забыть (когда не индексировать).** 1) **Маленькая таблица** (1–4 страницы): прочитать всё подряд быстрее и дешевле, чем обход индекса + чтения по TID; планировщик часто выбирает seq scan — индекс «не используется». 2) **Низкая селективность** (условие отбирает 20–80% строк): индекс вернёт очень много TID, чтение таблицы вразнобой может быть дороже одного последовательного скана — планировщик может предпочесть seq scan. 3) **Столбец не в WHERE/ORDER BY/GROUP BY/JOIN:** индекс по нему **никогда** не ускорит запросы, зато каждую вставку/обновление/удаление замедлит — создавай индексы только под реальные условия запросов.

#### Запомните

- **Маленькие таблицы:** полный скан дёшев; индекс может быть дороже. Не обязательно индексировать «на всякий случай».
- **Низкая селективность:** условие отбирает большую долю строк — индекс может не использоваться (планировщик выберет seq scan).
- **Столбец не в условиях:** индекс не ускорит запросы и будет только замедлять запись и занимать место. Индексировать столбцы, участвующие в WHERE, ORDER BY, GROUP BY, JOIN.

**Ещё раз самыми простыми словами:** не индексируй «всё подряд». На очень маленькой таблице «пролистать всё» быстрее, чем пользоваться указателем. На столбце, где почти все значения одинаковы, индекс почти не отсекает строк. И не нужен индекс по столбцу, по которому никто не ищет — он только тормозит вставки и занимает место.

### 29.4. Баланс числа индексов

**Цель раздела.**  
Понять, что число индексов на таблицу — это **компромисс**: больше индексов — быстрее разные виды запросов, но медленнее запись и больше место. Научиться смотреть **idx_scan** в pg_stat_user_indexes и решать, какие индексы оставить, а какие удалить. После раздела ты будешь понимать, почему «чем больше индексов, тем лучше» — миф, и как проверять использование по цифрам.

#### Термины (расшифровка)

- **Trade-off (компромисс)** — выигрыш в одном за счёт проигрыша в другом. Здесь: **выигрыш** — скорость чтения по разным условиям (каждый индекс ускоряет свои запросы); **проигрыш** — скорость записи (каждая вставка/обновление/удаление обновляет все индексы), место на диске и объём WAL. Нет «правильного» числа индексов — зависит от нагрузки: читаем чаще или пишем чаще, и по каким столбцам идут условия в запросах.
- **idx_scan** — в представлении **pg_stat_user_indexes** (PostgreSQL) счётчик: **сколько раз** этот индекс был использован для **сканирования** (index scan или bitmap index scan). Каждый раз, когда планировщик выбрал этот индекс для выполнения запроса, счётчик увеличивается на 1. По нему видно, «работает» ли индекс. Счётчики сбрасываются при перезапуске кластера или при вызове pg_stat_reset(); для выводов нужен **период наблюдения** (например, неделя работы).
- **Неиспользуемый индекс** — индекс, у которого idx_scan = 0 (или очень мал) за длительный период. Такой индекс не ускоряет запросы (по нему почти не ищут), но **замедляет каждую** вставку/обновление/удаление и **занимает место**. Может быть кандидатом на удаление. **Осторожность:** редкие запросы (раз в месяц, отчёт) тоже могут быть критичны — перед удалением убедиться, что нет важных запросов, использующих этот индекс.

#### Теория и правила (подробно)

- **Сколько индексов «нормально»:** единого правила нет. **OLTP-таблица** с частыми вставками (например, очередь заказов, логи) и редкими сложными запросами — лучше **меньше** индексов (1–3 типично: PK, по внешнему ключу, по самому частому условию в WHERE). **Читаемая** таблица (справочник, отчёты) с редкими вставками и множеством разных запросов — можно **больше** индексов (5–10 и более), если каждый реально используется и ускоряет свои запросы.
- **Мониторинг:** в PostgreSQL запрос к **pg_stat_user_indexes** даёт по каждому индексу: relname (имя таблицы), indexrelname (имя индекса), **idx_scan**, idx_tup_read, idx_tup_fetch. Индексы с **idx_scan = 0** за период наблюдения — первые кандидаты на анализ: есть ли запросы, которые *должны* были бы их использовать; если таких нет — индекс можно удалить (ускорение записи и освобождение места). Сортировка по idx_scan ASC выводит наименее используемые индексы первыми.
- **Избыточный индекс:** индекс, который является **подмножеством** другого. Пример: есть составной индекс **(a, b)**. Запросы только по `a` могут использовать этот индекс (правило левого префикса). Отдельный индекс только по **(a)** в таком случае **избыточен** — он дублирует «левую часть» (a, b) и только занимает место и замедляет запись. Удаление (a) не ломает запросы по a — они продолжат использовать (a, b). **Дублирующий индекс:** два индекса с **одинаковым** определением (те же столбцы, тот же порядок) — один из них явно лишний.

> **Важно.** «Чем больше индексов, тем быстрее база» — **неверно**. Больше индексов — быстрее **разные** запросы по разным столбцам, но **медленнее** каждая запись (INSERT/UPDATE/DELETE). На таблицах с частыми вставками лишние индексы сильно бьют по производительности записи. Добавляй индексы **под конкретные медленные запросы** и удаляй те, что не используются (idx_scan = 0) или избыточны.

#### Пошагово: как проверить баланс индексов (практика)

1. Выполни запрос к **pg_stat_user_indexes** (пример ниже), отсортируй по idx_scan ASC и по размеру индекса DESC.
2. Посмотри индексы с **idx_scan = 0** и **большим размером** — это первые кандидаты: они занимают много места и ни разу не использовались за период (или с рестарта). Не трогай без анализа PRIMARY KEY и UNIQUE — они могут быть нужны для целостности.
3. Для каждого такого индекса задай вопрос: есть ли в приложении или в отчётах запросы с WHERE/ORDER BY по столбцам этого индекса? Если **нет** — индекс можно удалить (DROP INDEX или DROP INDEX CONCURRENTLY). Если **да** (редкий отчёт раз в месяц) — решай по важности: оставить или удалить и принять более медленный редкий запрос.
4. Проверь **избыточность**: есть ли два индекса, где один — левый префикс другого (например (a) и (a, b))? Отдельный (a) при наличии (a, b) можно удалить — запросы по a продолжат использовать (a, b).

#### Что будет, если держать десять индексов на таблицу с миллионом вставок в день

Каждая вставка обновляет **все десять** индексов: десять вставок в B-tree (или другие структуры), десять порций записей в WAL. Время одной вставки растёт; нагрузка на диск и CPU растёт. При 1–3 индексах (PK + 1–2 по частым условиям) вставки остаются быстрыми; при 10 индексах «на всякий случай» запись может стать узким местом. **Итог:** индексы создавать под **реальные** запросы и периодически смотреть idx_scan — неиспользуемые удалять.

#### Пример запроса: индексы с нулевым использованием (PostgreSQL)

```sql
SELECT
  schemaname,
  relname AS table_name,
  indexrelname AS index_name,
  idx_scan,
  pg_size_pretty(pg_relation_size(indexrelid)) AS index_size
FROM pg_stat_user_indexes
WHERE idx_scan = 0
  AND indexrelname NOT LIKE '%_pkey'  -- не трогаем PK без анализа
ORDER BY pg_relation_size(indexrelid) DESC;
```

Интерпретация: большие индексы с idx_scan = 0 за период наблюдения — первые кандидаты на проверку (используются ли кем-то; можно ли удалить).

#### Простыми словами

**Больше индексов** — быстрее разные запросы, но **медленнее вставки/обновления/удаления** и больше место. Нужно смотреть, какие индексы **реально используются** (idx_scan), и не держать лишние. Неиспользуемые и избыточные индексы лучше удалять; дубликаты — удалить один.

#### Картинка в голове

Индексы — как указатели в конце книги. Один указатель по фамилиям — нужен. Второй по датам — нужен, если часто ищем по дате. Пятый по «цвету рубашки» — если по нему никто не ищет, он только утяжеляет книгу и его нужно обновлять при каждом изменении. **Баланс:** столько указателей, сколько реально используют; лишние — убрать.

#### Как запомнить

**Баланс индексов:** больше индексов → быстрее чтение по разным условиям, медленнее запись. Мониторинг: **pg_stat_user_indexes.idx_scan** — сколько раз индекс использовался. idx_scan = 0 долгое время → проверить, нужен ли индекс; избыточные и дублирующие — удалять.

#### Проверь себя (29.4)

1. Почему на таблице с очень частыми вставками не стоит создавать десять индексов «на все столбцы»?  
<details><summary>Ответ</summary> Потому что при **каждой** вставке СУБД будет обновлять **все десять** индексов: добавлять по одной записи в каждый. Это замедлит каждую вставку в несколько раз (по сравнению с одной-двумя индексами) и значительно увеличит объём WAL и место на диске. Баланс: индексировать только те столбцы, по которым реально идут запросы и которые дают выгоду.</details>

2. Что показывает idx_scan в pg_stat_user_indexes и как это использовать?  
<details><summary>Ответ</summary> **idx_scan** — счётчик того, сколько раз данный индекс был использован для **сканирования** (index scan или bitmap index scan). По нему видно, «работает» ли индекс. Если за длительный период idx_scan = 0, индекс ни разу не использовался — можно проверить, есть ли запросы, которые должны его использовать; если таких нет, индекс — кандидат на удаление для ускорения записи и освобождения места. Перед удалением нужно убедиться, что нет редких критичных запросов.</details>

3. Что такое избыточный индекс? Пример.  
<details><summary>Ответ</summary> **Избыточный индекс** — индекс, который является **подмножеством** другого. Пример: есть составной индекс (a, b). Запросы только по `a` могут использовать этот индекс (правило левого префикса). Отдельный индекс только по (a) в таком случае **избыточен** — он дублирует «левую часть» составного индекса и только занимает место и замедляет запись. Удаление избыточного (a) оставляет все запросы по `a` обслуживаемыми индексом (a, b).</details>

4. Зачем смотреть idx_scan по каждому индексу и что делать, если idx_scan = 0 полгода?  
<details><summary>Ответ</summary> **idx_scan** показывает, сколько раз индекс реально использовался для сканирования. Если **idx_scan = 0** полгода — индекс за полгода **ни разу** не был выбран планировщиком. Значит, либо по нему никто не ищет, либо запросы очень редкие. Такой индекс только замедляет запись и занимает место. **Действия:** проверить, нет ли критичных редких запросов (отчёты, админка), использующих этот индекс; если нет — **удалить** индекс (DROP INDEX или DROP INDEX CONCURRENTLY), чтобы ускорить вставки и освободить место. Перед удалением убедиться, что индекс не нужен для целостности (UNIQUE) или для важных редких операций.</details>

**Три пункта, чтобы не забыть (баланс индексов).** 1) **Компромисс:** больше индексов = быстрее разные запросы (по разным столбцам), но **медленнее каждая** вставка/обновление/удаление и больше место; «чем больше индексов, тем лучше» — миф. 2) **idx_scan** в pg_stat_user_indexes = сколько раз этот индекс использовался для сканирования; idx_scan = 0 долгое время → индекс не используется, кандидат на удаление (сначала проверить редкие запросы). 3) **Избыточный** индекс — подмножество другого (например, индекс (a) при наличии (a, b)); **дублирующий** — два индекса с одинаковым определением; лишние удалять.

#### Запомните

- Число индексов — **компромисс**: больше индексов → быстрее разные запросы, медленнее запись и больше место.
- **Мониторинг:** pg_stat_user_indexes.idx_scan — как часто индекс использовался. Неиспользуемые (idx_scan = 0) — кандидаты на удаление после проверки. Избыточные (подмножество другого) и дублирующие — удалять.

**Ещё раз самыми простыми словами:** не вешай на таблицу «все индексы подряд». Каждый индекс при записи нужно обновлять — чем их больше, тем медленнее вставки. Смотри по idx_scan: если индекс полгода не использовался (idx_scan = 0) и под него нет важных редких запросов — смело удаляй. И проверяй избыточность: если есть индекс (a, b), отдельный индекс (a) чаще всего не нужен.
