## 34. Bitmap scan и index-only scan

### 34.1. Bitmap Index Scan

**Цель раздела.**  
Понять **Bitmap Index Scan**: по индексу собирается **битовая карта TID** (какие страницы таблицы содержат подходящие строки), затем страницы таблицы читаются **по порядку** (Bitmap Heap Scan). Понять, почему при **много** строк это выгоднее, чем простой Index Scan по каждому TID. После раздела ты будешь различать в плане Index Scan и Bitmap Index Scan + Bitmap Heap Scan и понимать, зачем нужны оба варианта.

#### Термины (расшифровка)

- **Bitmap Index Scan** — узел плана выполнения: обход индекса и сбор **TID** (идентификаторов строк) всех подходящих строк в **битовую карту (bitmap)**. В карте отмечено, **какие страницы** таблицы нужно прочитать: по каждому TID известен номер страницы heap; в bitmap ставятся биты для этих страниц. Итог — «список страниц», которые содержат хотя бы одну подходящую строку.
- **Bitmap Heap Scan** — узел плана: чтение **страниц таблицы (heap)** по этой битовой карте. Страницы читаются **в порядке возрастания номера страницы** (последовательное чтение), а не в порядке появления TID в индексе. На каждой прочитанной странице отсекаются невидимые строки (MVCC). Меньше **случайных** переходов по таблице, чем если бы мы шли по TID подряд (как при простом Index Scan при большом числе строк).

#### Теория и правила (подробно)

- Планировщик выбирает **Bitmap Index Scan + Bitmap Heap Scan**, когда по индексу отбирается **много** строк (оценка выше порога, обычно тысячи и больше). Идея: сначала **собрать** все TID по индексу в bitmap, затем **отсортировать** их по номеру страницы таблицы и читать страницы таблицы **последовательно** (страница 5, 6, 7, 10, …), а не вразнобой (страница 1000, 3, 500, 2, …). При **малом** числе строк (например, 1–10) выгоднее **простой Index Scan**: сразу по каждому TID идём в таблицу, накладные на bitmap не окупаются.
- **Плюсы bitmap подхода:** при большом числе строк **меньше случайных** чтений страниц таблицы — читаем страницы по порядку номеров, диск (или кэш) работает эффективнее. **Минусы:** два прохода (сначала полный обход индекса по диапазону и построение bitmap, потом чтение heap по bitmap) и память на bitmap.

#### Пошагово: что происходит при Bitmap Index Scan + Bitmap Heap Scan (с числами)

Запрос: `SELECT * FROM orders WHERE status = 'active'`. По индексу на status отбирается **20 000** строк (TID разбросаны по 8 000 страницам таблицы).

1. **Bitmap Index Scan:** обходим индекс по ключу status = 'active', собираем все **TID** подходящих строк (20 000 штук) в **битовую карту**. В карте помечаются **номера страниц** таблицы, на которых лежат эти строки (например, страницы 1, 2, 5, 7, 10, 11, … — всего 8 000 страниц с хотя бы одной активной строкой).
2. Карта (или список страниц) **сортируется по номеру страницы**, чтобы читать таблицу **последовательно**: сначала страница 1, потом 2, 3, 5, 7, …
3. **Bitmap Heap Scan:** читаем страницы таблицы **в этом порядке**. Для каждой страницы загружаем её (из кэша или с диска), отбираем строки с status = 'active' и проверяем видимость (MVCC). Отдаём строки запросу.

**Итог:** таблица читается **почти последовательно** (по возрастанию номера страницы), а не 20 000 раз «прыжок на случайную страницу». На HDD и при большом числе строк это сильно быстрее, чем Index Scan по каждому TID подряд (который давал бы случайный порядок чтения страниц).

> **Важно.** Разница с **простым Index Scan**: при Index Scan мы по каждому TID из индекса **сразу** идём в таблицу и читаем страницу. Порядок чтения страниц таблицы — такой, в каком TID идут в индексе (по порядку ключа), а не по порядку номеров страниц. Если строк много и они разбросаны по таблице, страницы таблицы читаются **вразнобой** — много случайного I/O. Bitmap подход сначала собирает все TID, упорядочивает по номеру страницы и читает страницы **по порядку** — меньше случайного I/O.

#### Что будет, если при 100 000 строк использовать только Index Scan (без bitmap)

При **Index Scan** для каждой из 100 000 строк мы читаем страницу таблицы по TID. TID приходят из индекса в порядке ключа (например, по status и created_at), а не в порядке номеров страниц таблицы. Значит, страницы таблицы будут читаться **в случайном порядке** (страница 50000, 12, 3000, 88, …). Каждое такое чтение на HDD — позиционирование головки; 100 000 случайных чтений очень медленные. **Bitmap Heap Scan** те же 100 000 строк читает с страниц, но **страницы** загружаются в порядке их номеров — гораздо меньше «прыжков» по диску, быстрее.

#### Картинка в голове

**Index Scan при многих строках** — как **собирать яблоки с дерева вразнобой**: «первое яблоко — ветка 50, второе — ветка 3, третье — ветка 120…» Ты бегаешь вверх-вниз по дереву (страницы таблицы читаются в случайном порядке) — устаёшь и тратишь время. **Bitmap способ** — сначала записать на листок **номера веток**, где есть яблоки (1, 2, 3, 5, 7, 10…), **отсортировать** этот список и обходить ветки **по порядку**: сначала 1, потом 2, 3, 5, 7… Меньше беготни, быстрее. Таблица = дерево, страницы = ветки, TID = «где яблоко». Bitmap Heap Scan читает страницы таблицы **по возрастанию номера** — меньше «прыжков» по диску.

#### Простыми словами

**Bitmap Index Scan** — сначала по индексу собираем «список страниц», на которых лежат подходящие строки, затем открываем эти страницы **по порядку номеров** (сначала 1, потом 2, 3, 5, …), а не вразнобой. Удобно, когда подходящих строк **много** — меньше прыжков по таблице и по диску.

#### Как запомнить

**Bitmap Index Scan** = обход индекса → сбор TID в bitmap (какие страницы трогать) → **Bitmap Heap Scan** читает эти страницы таблицы **по порядку номеров**. Выгодно при **множестве** отобранных строк; при малом числе строк выгоднее обычный Index Scan.

#### Типичная ошибка

**Ожидать, что для одного и того же запроса план всегда будет одинаковым (всегда Index Scan или всегда Bitmap).** План зависит от **оценки** числа строк (статистика, селективность) и от настроек стоимости. После **ANALYZE**, при изменении данных или при смене параметров один и тот же запрос может получить **другой** план: например, при 100 строках — Index Scan, при 50 000 — Bitmap Index Scan + Bitmap Heap Scan. Это нормально. Если план «прыгает» неожиданно — проверить актуальность статистики (ANALYZE) и границы порога, после которого планировщик переключается на bitmap.

#### Проверь себя (34.1)

1. Когда планировщик предпочитает Bitmap Index Scan вместо простого Index Scan?  
<details><summary>Ответ</summary> Когда по индексу отбирается **много** строк (оценка выше порога). При малом числе строк проще сразу идти по каждому TID в таблицу (Index Scan). При большом числе TID случайные чтения страниц таблицы дороги; выгоднее собрать TID в bitmap, упорядочить по номеру страницы и прочитать страницы **последовательно** (Bitmap Heap Scan) — меньше случайного I/O.</details>

2. В чём главное отличие чтения таблицы при Bitmap Heap Scan и при простом Index Scan при 10 000 отобранных строк?  
<details><summary>Ответ</summary> При **Index Scan** страницы таблицы читаются **в порядке появления TID в индексе** (по порядку ключа). Строки с близкими ключами могут лежать на разных страницах таблицы — страницы читаются вразнобой (много случайного I/O). При **Bitmap Heap Scan** сначала собираются все TID, затем страницы таблицы читаются **в порядке возрастания номера страницы** (последовательно). Одна и та же страница не читается дважды; порядок чтения дружелюбен к кэшу и к диску. При 10 000 строк разница по числу «прыжков» по таблице и по скорости может быть существенной.</details>

3. Что означает Recheck Cond в плане Bitmap Heap Scan?  
<details><summary>Ответ</summary> **Recheck Cond** — повторная проверка **условия запроса** на страницах таблицы после чтения по bitmap. По индексу мы отобрали TID по условию (например, status = 'active'); при чтении страницы таблицы СУБД ещё раз проверяет условие для каждой строки (и видимость MVCC). Это нужно, потому что индекс мог вернуть «кандидатов», которых после проверки видимости или точного условия отсекают; Recheck Cond отражает эту проверку в плане.</details>

#### Запомните

- **Bitmap Index Scan** — сбор TID по индексу в битовую карту; **Bitmap Heap Scan** — чтение страниц таблицы **по порядку номеров страниц** по этой карте. Выгодно при **множестве** отобранных строк (меньше случайного I/O).
- При **малом** числе строк планировщик выбирает обычный **Index Scan** (сразу по TID в таблицу).

**Ещё раз самыми простыми словами:** когда по индексу подходит много строк, «бегать» по каждой строке в таблицу вразнобой — медленно. Bitmap способ: сначала собрать «какие страницы таблицы трогать», потом открыть эти страницы **по порядку** (1, 2, 3, 5, 7…) — меньше прыжков, быстрее.

### 34.2. Index-only scan

**Цель раздела.**  
Понять **Index-only scan** — чтение **только из индекса**, без обращения к страницам таблицы (heap). Условия: покрывающий индекс (все нужные столбцы в индексе) и **visibility map** (страницы таблицы помечены «все строки видимы»). Иначе СУБД вынуждена обращаться к heap (Heap Fetches). После раздела ты будешь понимать, зачем нужен INCLUDE и почему в плане иногда видно Heap Fetches > 0.

#### Термины (расшифровка)

- **Index-only scan** — узел плана: данные для запроса берутся **только из листьев индекса** (ключ + столбцы в INCLUDE); страницы таблицы (heap) **не читаются**. Это уменьшает I/O — не нужно загружать страницы таблицы, только страницы индекса. Возможен только если **все** столбцы, которые запрос выбирает и по которым фильтрует, есть в индексе (покрывающий индекс).
- **Visibility map** — битовая карта в PostgreSQL: для **каждой страницы** таблицы хранится бит «все строки на этой странице **видимы всем** активным транзакциям» (никто не удалял и не обновлял их с незавершёнными транзакциями). Если бит **установлен**, СУБД считает, что по TID с этой страницы можно **не** обращаться к heap для проверки xmin/xmax — строка видима. Если бит **не установлен**, для проверки видимости нужно прочитать страницу heap (**heap fetch**). После **VACUUM** страницы помечаются в visibility map. В плане **Heap Fetches: N** (N > 0) значит: N раз пришлось прочитать страницу таблицы для проверки видимости — «чистого» index-only scan не получилось для части строк.

#### Теория и правила (подробно)

- **Условия для index-only scan:** 1) **Покрывающий индекс** — все столбцы из SELECT и из WHERE есть в индексе (в ключе или в INCLUDE). 2) Для **всех** TID, которые мы читаем из индекса, соответствующие **страницы таблицы** помечены в visibility map как «все видимы». Тогда СУБД не обращается к heap вообще — данные и видимость берутся из индекса и из карты. Если хотя бы для одной страницы таблицы бит не установлен (например, недавно были обновления), для строк с этой страницы выполняется **heap fetch** — чтение страницы таблицы для проверки xmin/xmax.
- **Практика:** после массовых обновлений или удалений часть страниц не помечена в visibility map — index-only scan даёт **Heap Fetches > 0**. Регулярный **VACUUM** помечает страницы; **CLUSTER** перезаписывает таблицу и улучшает и корреляцию, и заполнение visibility map. На стабильных таблицах (мало обновлений) большинство страниц помечены — Heap Fetches малы или нуль.

#### Пошагово: когда получается «чистый» index-only scan (без Heap Fetches)

Запрос: `SELECT id, status, created_at FROM orders WHERE status = 'active'`. Индекс: **(status) INCLUDE (id, created_at)** — покрывающий.

1. Планировщик выбирает **Index Only Scan** по этому индексу: все нужные столбцы (id, status, created_at) есть в индексе.
2. Читаем листья индекса по ключу status = 'active'; из каждой записи листа берём id, status, created_at (и TID). Чтобы отдать строку клиенту, нужно быть уверенным, что она **видима** текущей транзакции (не удалена и не скрыта другой транзакцией). Видимость в PostgreSQL хранится в **heap** (xmin, xmax в заголовке строки).
3. Для каждого TID смотрим **visibility map**: страница таблицы, на которую указывает TID, помечена как «все видимы»? Если **да** — обращаться к heap **не нужно**, считаем строку видимой, отдаём данные из индекса. Если **нет** — делаем **heap fetch**: читаем страницу таблицы, проверяем xmin/xmax, и только потом отдаём строку.
4. Если **все** затронутые страницы помечены в visibility map — **Heap Fetches = 0**, читаем только индекс. Если часть страниц не помечена — Heap Fetches > 0.

> **Важно.** Даже при **покрывающем** индексе планировщик может выбрать Index Only Scan, но с **Heap Fetches > 0**, если visibility map не полная. Это всё равно выгоднее, чем читать из heap все строки подряд (мы читаем heap только для «сомнительных» страниц). Чтобы максимизировать число строк без heap fetch, нужен **VACUUM** (и при необходимости CLUSTER), чтобы больше страниц было помечено в visibility map.

#### Что будет, если не помечать страницы в visibility map

Без visibility map СУБД **всегда** должна была бы проверять видимость по heap (читать страницу таблицы по каждому TID). Тогда «index-only» scan не давал бы выигрыша по I/O — мы бы всё равно читали каждую страницу таблицы. Visibility map позволяет **не** читать heap для страниц, где ни одна строка не была недавно изменена или удалена — такие страницы помечены «все видимы», и по TID с них данные можно брать только из индекса.

#### Картинка в голове

**Обычный поиск по индексу** — по указателю находишь «страница 45», идёшь в книгу и читаешь строку на странице 45. **Index-only scan** — в указателе уже **написана сама строка** (все нужные столбцы в индексе), и ты **знаешь**, что страница 45 «проверена» (visibility map: на ней никто ничего не менял с незавершёнными делами). Тогда в книгу можно **не ходить** — ответ целиком в указателе. Если для страницы 45 в карте не помечено «всё видимо», пришлось бы открыть книгу и убедиться, что строка актуальна — это и есть **Heap Fetch**. Чем больше страниц помечено «всё видимо» (после VACUUM), тем реже нужно открывать книгу (таблицу).

#### Простыми словами

**Index-only scan** — ответ на запрос собирается **только из индекса**, таблицу не открываем. Возможно, если в индексе есть **все** нужные столбцы (покрывающий индекс) и для затронутых страниц таблицы в **visibility map** помечено «страница вся видима». Тогда проверять видимость по таблице не нужно. Если для части страниц не помечено — будут **Heap Fetches** (чтение таблицы для проверки видимости).

#### Типичная ошибка

**Ожидать Heap Fetches = 0 сразу после создания покрывающего индекса.** Visibility map заполняется не при создании индекса, а когда **VACUUM** обрабатывает таблицу и помечает страницы «все строки видимы». На только что созданном индексе по таблице с активными обновлениями многие страницы могут быть **не помечены** — планировщик выберет Index Only Scan, но в плане будет **Heap Fetches > 0**. Это нормально. Чтобы уменьшить Heap Fetches, нужно **запустить VACUUM** (и при необходимости ANALYZE); на стабильных таблицах после VACUUM доля помеченных страниц растёт и Heap Fetches падают.

#### Как запомнить

**Index-only scan** = чтение только из индекса; нужен **покрывающий индекс** и **visibility map** (страницы «все видимы»). **Heap Fetches > 0** — часть данных или проверок видимости прочитана из heap; чем больше VACUUM и стабильнее данные, тем меньше Heap Fetches.

**Три пункта, чтобы не забыть (index-only scan).** 1) В индексе должны быть **все** столбцы из SELECT и WHERE (покрывающий индекс, при необходимости INCLUDE). 2) Для страниц таблицы, на которые указывают TID, в **visibility map** должно быть «все видимы» — иначе СУБД ходит в heap (Heap Fetches). 3) **VACUUM** помечает страницы в visibility map; без VACUUM или при частых обновлениях Heap Fetches будут больше нуля — это не баг, а норма.

#### Проверь себя (34.2)

1. При каком условии возможен «чистый» index-only scan без Heap Fetches?  
<details><summary>Ответ</summary> Когда **все** страницы таблицы, на которые указывают TID из индекса, помечены в **visibility map** как «все строки на странице видимы всем». Тогда СУБД не обращается к heap для проверки xmin/xmax — данные берутся только из индекса. После VACUUM страницы помечаются в visibility map; при стабильных данных (мало обновлений) большинство страниц помечены — Heap Fetches = 0 или малы.</details>

2. Запрос SELECT id, name FROM users WHERE email = ?. Индекс (email) INCLUDE (id, name). В плане — Index Only Scan, но Heap Fetches: 5000. Что это значит и от чего зависит?  
<details><summary>Ответ</summary> **Index Only Scan** выбран, потому что все нужные столбцы (id, name, email) есть в индексе (покрывающий индекс). **Heap Fetches: 5000** значит: для **5000** строк пришлось **прочитать страницу таблицы** (heap), чтобы проверить видимость (xmin/xmax). Это происходит, когда страницы таблицы, на которых лежат эти строки, **не помечены** в visibility map как «все видимы» (например, на них недавно были обновления или удаления). Зависит от того, как часто делают VACUUM и насколько стабильны данные: после VACUUM больше страниц помечается, Heap Fetches уменьшаются.</details>

3. Почему для index-only scan нужны и покрывающий индекс, и visibility map? Недостаточно одного покрывающего индекса?  
<details><summary>Ответ</summary> **Покрывающий индекс** даёт все **данные** для ответа (столбцы из SELECT и WHERE) в листьях индекса. Но в PostgreSQL информация о **видимости** строки (удалена ли, скрыта ли другой транзакцией) хранится в **heap** (xmin, xmax). Чтобы не ходить в heap за каждой строкой, СУБД смотрит **visibility map**: если страница помечена «все строки видимы», по TID с этой страницы можно не читать heap. Без visibility map пришлось бы читать каждую страницу таблицы для проверки видимости — выигрыша index-only не было бы. Поэтому нужны **оба**: покрытие (данные в индексе) и visibility map (не ходить в heap за видимостью).</details>

#### Запомните

- **Index-only scan** — данные только из индекса; нужен **покрывающий индекс** и актуальная **visibility map**. Heap Fetches > 0 — чтение из heap для проверки видимости (страница не помечена «все видимы»).
- Регулярный **VACUUM** увеличивает долю страниц в visibility map и уменьшает Heap Fetches.

**Ещё раз самыми простыми словами:** index-only scan = «отвечаем на запрос, не открывая таблицу» — только листаем индекс. Так можно, если в индексе есть все нужные столбцы и СУБД «доверяет», что строки на этих страницах видимы (visibility map). Если не доверяет — идёт в таблицу и проверяет (это и есть Heap Fetches). Чтобы Heap Fetches было меньше — делать VACUUM и по возможности CLUSTER.

### 34.3. Корреляция физического и логического порядка

**Цель раздела.**  
Понять **корреляцию** — статистику в pg_stats: насколько **порядок значений** в столбце на диске (физический порядок строк) совпадает с **порядком по возрастанию** (логический порядок). Высокая корреляция выгодна для **index-only scan** и **BRIN**; низкая — много случайных чтений при обходе по индексу.

#### Термины (расшифровка)

- **Корреляция (correlation)** — в **pg_stats** для каждого столбца хранится число от **-1** до **1**. **1** — значения на диске идут **по возрастанию**; **-1** — по убыванию; **0** — **случайный** порядок. Планировщик использует корреляцию для **оценки стоимости** index scan: при высокой корреляции страницы таблицы по TID из индекса идут подряд; при низкой — вразнобой, много случайного I/O.
- **Влияние на index-only scan и BRIN:** при **высокой** корреляции страницы по TID из индекса идут подряд — лучше кэш и меньше I/O. При **низкой** страницы разбросаны. **BRIN** эффективен только при высокой корреляции (в одном блоке значения в узком диапазоне).

#### Теория и правила (подробно)

- **CLUSTER** повышает корреляцию: таблица перезаписывается в порядке индекса; после CLUSTER — **ANALYZE**; **pg_stats.correlation** обновляется при ANALYZE.
- **BRIN** эффективен при высокой корреляции: min/max по блоку хорошо отсекают блоки. При корреляции близкой к 0 BRIN почти не помогает.

#### Пошагово: как корреляция влияет на чтение

Индекс по **created_at**; запрос — диапазон дат (50 000 строк). **Корреляция высокая (0.95):** страницы таблицы по TID идут почти подряд (5, 6, 7, 8…) — чтение последовательное. **Корреляция низкая (0.1):** страницы вразнобой (100, 50000, 2000…) — много случайного I/O. Планировщик учитывает корреляцию в стоимости плана.

> **Важно.** Корреляция — статистика для планировщика: «при чтении по этому индексу страницы таблицы идут подряд или вразнобой». Для BRIN она критична: без высокой корреляции min/max по блоку не отсекают блоки.

#### Картинка в голове

**Корреляция** — как **ответ на вопрос: «Страницы в книге лежат в том же порядке, что и в указателе?»** **Высокая (близко к 1):** да — открываешь указатель, читаешь страницы 1, 2, 3, 4… подряд; быстро. **Низкая (близко к 0):** нет — по указателю тебе нужны страницы 5, 89, 2, 156, 11… вразнобой; много «прыжков» по книге, медленнее. BRIN при высокой корреляции — как «краткое содержание по главам», когда главы действительно по порядку; при низкой в каждой «главе» всё вперемешку, и краткое содержание почти не помогает. **CLUSTER** — переложить страницы книги в порядок указателя, то есть **повысить** корреляцию.

#### Проверь себя (34.3)

1. Где в PostgreSQL хранится корреляция по столбцу?  
<details><summary>Ответ</summary> В **pg_stats**: столбец **correlation** для (schemaname, tablename, attname). Обновляется при **ANALYZE**. Пример: SELECT tablename, attname, correlation FROM pg_stats WHERE schemaname = 'public' AND tablename = 'orders';</details>

2. Почему после массовых вставок «в конец» корреляция по created_at высокая, а после множества случайных UPDATE — падает?  
<details><summary>Ответ</summary> Вставки в конец → новые строки в новых страницах; порядок на диске совпадает с порядком по дате. Случайные UPDATE → новые версии строк в других местах; порядок на диске перестаёт совпадать с датой. **CLUSTER** перезаписывает таблицу в порядке индекса и снова повышает корреляцию.</details>

3. Зачем планировщику знать корреляцию по столбцу?  
<details><summary>Ответ</summary> Чтобы **оценить стоимость** index scan: при **высокой** корреляции страницы таблицы по TID из индекса идут **подряд** (меньше случайного I/O, быстрее); при **низкой** страницы вразнобой — много случайных чтений, дороже. Планировщик закладывает это в cost и может предпочесть другой план (например, bitmap scan вместо простого index scan) в зависимости от корреляции. Для **BRIN** корреляция критична: при высокой min/max по блоку хорошо отсекают блоки; при низкой BRIN почти не помогает.</details>

**Три пункта, чтобы не забыть (корреляция).** 1) **Корреляция** (pg_stats.correlation) — число от -1 до 1: насколько **порядок на диске** совпадает с порядком по столбцу; 1 = по возрастанию, 0 = вразнобой. 2) **Высокая** корреляция выгодна для **index-only scan** и **BRIN** (страницы по индексу идут подряд; для BRIN — в блоке узкий диапазон значений). 3) **Повысить** корреляцию можно **CLUSTER** по индексу; обновляется при **ANALYZE**.

**Ещё раз самыми простыми словами:** корреляция — «насколько порядок на диске совпадает с порядком по столбцу». Высокая — чтение по индексу идёт почти подряд по диску; низкая — вразнобой. Улучшить можно **CLUSTER** по нужному индексу.

#### Запомните

- **Корреляция** (pg_stats) — степень совпадения физического порядка строк с порядком по столбцу. Высокая выгодна для index-only scan и BRIN; CLUSTER повышает корреляцию.

### 34.4. Как это выглядит в плане (EXPLAIN)

**Цель подраздела.**  
Увидеть в выводе **EXPLAIN** (и EXPLAIN ANALYZE) типичные узлы плана, связанные с индексами: **Index Scan**, **Bitmap Index Scan** + **Bitmap Heap Scan**, **Index Only Scan**. После подраздела ты будешь узнавать эти узлы в плане и связывать их с тем, что изучено в §34.1 и §34.2. Подробный разбор планов — в **Части VII (оптимизация запросов)**; здесь — только «как выглядят» узлы индексов.

#### Что смотреть в плане

- **Seq Scan** — полное сканирование таблицы (индекс не используется).
- **Index Scan** по имени_индекса — доступ по индексу: обход B-tree и чтение страниц таблицы по TID (порядок чтения таблицы — по порядку ключа в индексе).
- **Bitmap Index Scan** по имени_индекса + **Bitmap Heap Scan** — по индексу собрали битовую карту TID, затем читаем страницы таблицы **по порядку номеров страниц** (см. §34.1).
- **Index Only Scan** по имени_индекса — чтение **только из индекса** (покрывающий индекс + visibility map). В плане может быть строка **Heap Fetches: N** — сколько раз пришлось обратиться к таблице для проверки видимости (N = 0 при полном index-only).

#### Пример фрагментов плана (как они выглядят)

Один и тот же запрос на разных данных может дать разный план. Ниже — **типичный вид** узлов (не полный вывод EXPLAIN, а фрагменты для узнавания).

**1) Index Scan** (мало строк, планировщик выбрал прямой доступ по индексу):

```
Index Scan using idx_orders_created_at on orders  (cost=0.42..8.44 rows=1 width=40)
  Index Cond: (created_at = '2024-01-15'::date)
```

Здесь: узел **Index Scan**, используется индекс **idx_orders_created_at**; условие по индексу — **Index Cond**.

**2) Bitmap Index Scan + Bitmap Heap Scan** (много строк, выгодно читать страницы таблицы по порядку):

```
Bitmap Heap Scan on orders  (cost=154.33..321.00 rows=5000 width=40)
  Recheck Cond: (status = 'active'::text)
  ->  Bitmap Index Scan on idx_orders_status  (cost=0.00..153.08 rows=5000)
        Index Cond: (status = 'active'::text)
```

Сначала **Bitmap Index Scan** по **idx_orders_status** (собирается битовая карта), затем **Bitmap Heap Scan** по таблице orders (чтение страниц по порядку). **Recheck Cond** — повторная проверка условия на страницах таблицы (для корректности и видимости).

**3) Index Only Scan** (покрывающий индекс, данные только из индекса):

```
Index Only Scan using idx_orders_status_covering on orders  (cost=0.42..180.00 rows=5000 width=20)
  Index Cond: (status = 'active'::text)
  Heap Fetches: 0
```

Узел **Index Only Scan**, индекс **idx_orders_status_covering**; **Heap Fetches: 0** — таблица не читалась для получения данных (или для проверки видимости). Если бы было **Heap Fetches: 1200** — часть строк потребовала чтения heap (см. §34.2).

#### Простыми словами

В плане запроса (EXPLAIN) **Index Scan** — «иду по индексу и по каждому TID в таблицу». **Bitmap Index Scan** + **Bitmap Heap Scan** — «собрал по индексу список страниц таблицы, потом читаю эти страницы подряд». **Index Only Scan** — «всё нужное взял из индекса, в таблицу не ходил» (при Heap Fetches: 0). Узнавай эти названия узлов в выводе EXPLAIN — детальный разбор cost и строк будет в части VII.

#### Проверь себя (34.4)

1. В плане запроса видно узел «Bitmap Heap Scan» и под ним «Bitmap Index Scan». В каком порядке выполняется доступ: сначала индекс или сначала таблица?  
<details><summary>Ответ</summary> Сначала **Bitmap Index Scan** — по индексу собирается битовая карта (какие страницы таблицы содержат подходящие строки). Затем **Bitmap Heap Scan** — по этой карте читаются **страницы таблицы** в порядке их номеров. То есть сначала обход индекса, потом чтение heap. В плане дочерний узел (Bitmap Index Scan) выполняется первым; родительский (Bitmap Heap Scan) использует результат.</details>

2. В плане — Index Only Scan по индексу idx_orders_status_covering, в строке плана указано «Heap Fetches: 0». Что это значит?  
<details><summary>Ответ</summary> **Heap Fetches: 0** значит, что для **ни одной** из прочитанных строк СУБД **не обращалась** к страницам таблицы (heap) для проверки видимости. Все затронутые страницы были помечены в **visibility map** как «все строки видимы» — данные и видимость взяты только из индекса. Это «чистый» index-only scan, максимальная выгода по I/O.</details>

3. По какому признаку в плане отличить Seq Scan от Index Scan, не читая имена индексов?  
<details><summary>Ответ</summary> **Seq Scan** — узел плана **последовательного сканирования таблицы**; в плане не указывается индекс, потому что таблица читается подряд (все страницы heap). **Index Scan** (или Bitmap Index Scan, Index Only Scan) — в названии узла есть слово **Index** и указывается **имя индекса** (using idx_...). То есть: нет слова «Index» и нет имени индекса → скорее всего Seq Scan; есть «Index» и имя индекса → один из типов доступа по индексу.</details>

#### Запомните

- **Index Scan** — обход индекса и чтение таблицы по TID. **Bitmap Index Scan** + **Bitmap Heap Scan** — битовая карта по индексу, затем чтение страниц таблицы по порядку. **Index Only Scan** — чтение только из индекса (смотри **Heap Fetches** в плане).
- Полный разбор планов (cost, rows, время) — в **Части VII**.
