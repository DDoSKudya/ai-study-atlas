## 32. Частичный индекс и индекс по выражению

### 32.1. Частичный индекс (WHERE)

**Цель раздела.**  
Понять **частичный индекс**: индекс строится **только по строкам**, удовлетворяющим условию **WHERE** в определении. Меньше размер и быстрее обновление; запросы должны содержать **то же условие** (или более узкое), чтобы планировщик использовал индекс. После раздела ты будешь знать, когда частичный индекс выгоден и почему «забытое» условие в запросе приводит к тому, что индекс не используется.

#### Термины (расшифровка)

- **Частичный индекс (partial index)** — индекс с условием **WHERE** в определении: `CREATE INDEX ... ON t (col) WHERE condition;`. В индекс попадают **только** строки, для которых **condition** истинно. Остальные строки таблицы **в индексе не представлены**. Размер индекса меньше (индексируется подмножество); при вставке/обновлении/удалении индекс обновляется **только** для строк, удовлетворяющих condition — строки с condition = false не трогают индекс.
- **Условие индекса** — планировщик может использовать частичный индекс **только** если **условие запроса подразумевает условие индекса**. То есть множество строк, отбираемых запросом, должно быть **подмножеством** множества строк, входящих в индекс. Пример: индекс WHERE status = 'active' подходит для WHERE status = 'active' AND created_at > ... (сужаем активные); не подходит для WHERE status = 'closed' или для запроса без фильтра по status — в индексе нет закрытых и нет «всех» строк.

#### Теория и правила (подробно)

- **Создание:** `CREATE INDEX idx_orders_active ON orders (created_at) WHERE status = 'active';` — в индексе только строки с status = 'active'. Запрос `SELECT ... FROM orders WHERE status = 'active' AND created_at > '2025-01-01'` может использовать этот индекс (условие запроса подразумевает status = 'active'). Запрос `SELECT ... FROM orders WHERE created_at > '2025-01-01'` **без** status = 'active' — планировщик **не** сможет использовать этот индекс, потому что в индексе нет строк с status != 'active', и результат запроса не обязан быть подмножеством индекса.
- **Плюсы:** меньше размер индекса; быстрее вставка/обновление для строк, не попадающих в условие; фокус на «горячем» подмножестве (например, только активные заказы). **Минусы:** запросы **обязательно** должны содержать условие, совместимое с WHERE индекса, иначе индекс не применяется.

#### Пошагово: когда частичный индекс используется и когда нет

Индекс: `CREATE INDEX idx ON orders (created_at) WHERE status = 'active'`. В таблице 1 млн заказов, из них 100 тыс. активных.

1. **Запрос:** `SELECT * FROM orders WHERE status = 'active' AND created_at > '2025-01-01'`. Условие запроса **включает** status = 'active' → множество результата — подмножество индекса. Планировщик может использовать **idx**: ищет в индексе по created_at среди активных; индекс содержит 100 тыс. записей вместо 1 млн — быстрее и меньше.
2. **Запрос:** `SELECT * FROM orders WHERE status = 'closed'`. В индексе **нет** закрытых заказов (индекс только по status = 'active'). Этот индекс **не подходит** — планировщик выберет sequential scan или другой индекс.
3. **Запрос:** `SELECT * FROM orders WHERE created_at > '2025-01-01'` (без status). Результат может содержать и active, и closed. Индекс покрывает только active; по нему нельзя получить «все заказы после даты». Индекс **не используется** для такого запроса.

> **Важно.** Частичный индекс — это **подмножество** строк. Запрос может использовать его только если он ищет **внутри** этого подмножества. Как только запросу нужны строки «снаружи» (другое значение status или без фильтра по status), частичный индекс не применим — в нём просто нет этих строк.

#### Что будет, если забыть условие в запросе

Индекс: `(created_at) WHERE status = 'active'`. Запрос написан как `WHERE created_at > ?` без `status = 'active'`. Планировщик **не использует** этот частичный индекс: в индексе нет записей с status != 'active', а запрос не гарантирует, что мы ищем только активные. В итоге выполняется sequential scan (или другой полный индекс). **Типичная ошибка:** создали частичный индекс для ускорения «активных», но в коде запросы иногда делают без status = 'active' — тогда индекс простаивает. В запросе **всегда** должно быть условие, подразумевающее WHERE индекса.

#### Простыми словами

**Частичный индекс** — указатель не на всю книгу, а только на раздел «активные заказы». Меньше указатель, быстрее обновлять. Искать по нему можно только «внутри раздела» — запрос должен ограничивать выборку тем же условием (status = 'active').

#### Картинка в голове

**Частичный индекс** — как **указатель только на одну главу** книги (например, «Глава 5. Активные заказы»). В указателе перечислены только страницы из этой главы; про остальные главы в нём **ничего нет**. Искать по такому указателю можно только то, что **в этой главе**: «покажи активные заказы за январь» — подходит. «Покажи закрытые заказы» или «покажи все заказы за январь» — не подходит: закрытых и «всех» в этом указателе нет, он про другое подмножество. Поэтому в запросе **обязательно** должно быть условие «только активные» — иначе база не сможет опереться на этот указатель. Меньше указатель = быстрее в нём искать и быстрее его обновлять при изменениях только в этой главе.

#### Как запомнить

**Частичный индекс** = индекс с **WHERE** в определении; только строки, удовлетворяющие условию. Запрос должен **подразумевать** это условие, чтобы индекс использовался.

#### Проверь себя (32.1)

1. Когда планировщик может использовать частичный индекс WHERE status = 'active'?  
<details><summary>Ответ</summary> Когда условие запроса **подразумевает** status = 'active': например, WHERE status = 'active', или WHERE status = 'active' AND created_at > .... Тогда результат запроса — подмножество строк, входящих в индекс. Для запроса WHERE status = 'closed' или без условия по status этот индекс не подходит — такие строки в индексе отсутствуют.</details>

2. В таблице 10 млн строк; 200 тыс. со status = 'active'. Есть частичный индекс по (created_at) WHERE status = 'active'. Почему запрос WHERE status = 'active' AND created_at > ? может быть быстрее, чем с полным индексом по (status, created_at)?  
<details><summary>Ответ</summary> Частичный индекс содержит **только 200 тыс. записей** — дерево меньше по размеру и по высоте; поиск по created_at выполняется в меньшем объёме данных. Полный индекс по (status, created_at) содержит **все 10 млн** записей; при поиске status = 'active' AND created_at > ? всё равно отсекается большая часть, но дерево изначально больше, страниц индекса больше. Плюс при вставке/обновлении строк с status != 'active' частичный индекс не обновляется — меньше нагрузка на запись. Итог: меньше размер, быстрее поиск по подмножеству, быстрее обновления для «холодных» строк.</details>

3. Индекс: CREATE INDEX idx ON orders (created_at) WHERE status = 'active'. Будет ли он использоваться для запроса SELECT * FROM orders WHERE status = 'active' ORDER BY created_at LIMIT 10?  
<details><summary>Ответ</summary> **Да.** Условие запроса (status = 'active') **подразумевает** условие индекса; результат — подмножество строк, входящих в индекс. В индексе записи уже упорядочены по created_at среди активных, поэтому планировщик может использовать этот индекс и для фильтрации, и для ORDER BY created_at без отдельной сортировки. LIMIT 10 только уменьшает объём чтения.</details>

**Три пункта, чтобы не забыть (частичный индекс).** 1) В индекс попадают **только** строки, для которых условие **WHERE** в определении индекса истинно; остальные строки таблицы в индексе **нет**. 2) Запрос может использовать этот индекс **только** если его условие **подразумевает** условие индекса (результат запроса — подмножество индекса); если в запросе нет того же фильтра — планировщик индекс не возьмёт. 3) **Плюсы:** меньше размер индекса и быстрее обновление (строки вне условия не трогают индекс); в запросе **всегда** указывай условие, совпадающее с WHERE индекса.

#### Запомните

- **Частичный индекс** — CREATE INDEX ... WHERE condition; только строки, удовлетворяющие condition. Запрос должен подразумевать это условие.

### 32.2. Уникальный частичный индекс

**Цель раздела.**  
Понять **уникальный частичный индекс**: UNIQUE + WHERE. Уникальность гарантируется **только среди строк, удовлетворяющих условию**. Классический пример: одна активная подписка на пользователя — UNIQUE (user_id) WHERE status = 'active'.

#### Термины

- **Уникальный частичный индекс** — `CREATE UNIQUE INDEX ... ON t (col) WHERE condition;`. В таблице не может быть двух строк с одинаковым значением col среди тех, для которых condition истинно. Строки, не удовлетворяющие condition, в индекс не входят — их col может повторяться.

#### Теория и правила

- **Пример:** таблица подписок (user_id, status, ...). Ограничение: у пользователя не более одной **активной** подписки. UNIQUE (user_id) WHERE status = 'active'. Несколько подписок со status = 'expired' с одним user_id допустимы; две с status = 'active' и одним user_id — нет.
- **Плюсы:** выразительная модель ограничений без триггеров; индекс одновременно ускоряет поиск по (user_id) при status = 'active'.

#### Простыми словами

**Уникальный частичный** — «уникальность только среди выбранных»: например, один активный заказ на пользователя. Остальные (неактивные) могут дублироваться.

#### Типичная ошибка

Сделали **UNIQUE (user_id) WHERE status = 'active'**, думая, что «у пользователя один user_id». На самом деле ограничение говорит: **среди строк с status = 'active'** не может быть двух строк с одинаковым user_id. Две строки с user_id = 1 и status = 'expired' — **разрешены** (они не входят в индекс). Ошибка — путать «уникальность по всей таблице» и «уникальность только среди строк, попавших в частичный индекс». Проверяй условие WHERE в определении индекса: уникальность действует **только** в этом подмножестве.

#### Картинка в голове

**Уникальный частичный индекс** — как **правило «в каждой группе только один староста»**, причём группа — не весь класс, а только «активные ученики». В списке «активных» не может быть двух старост от одного класса. Но если ученик перешёл в «неактивные», в этой другой группе у того же класса может быть свой староста — ограничение «один на класс» действует **только внутри «активных»**. В таблице: один user_id с status = 'active' — нельзя второй active с тем же user_id; два user_id = 1 с status = 'expired' — можно, потому что в частичный индекс входят только active.

#### Как запомнить

**UNIQUE + WHERE** = уникальность среди строк, удовлетворяющих условию. Пример: одна активная подписка на user_id.

#### Проверь себя (32.2)

1. В таблице подписок (user_id, status, ...) создан UNIQUE (user_id) WHERE status = 'active'. Можно ли вставить две строки с user_id = 1 и status = 'expired'?  
<details><summary>Ответ</summary> **Да.** Уникальность проверяется **только** среди строк, удовлетворяющих условию (status = 'active'). Строки с status = 'expired' в индекс не входят; для них ограничение уникальности по user_id не действует. Две (и больше) подписки с user_id = 1 и status = 'expired' допустимы. Две строки с user_id = 1 и status = 'active' — недопустимы (нарушение уникального частичного индекса).</details>

2. Чем уникальный частичный индекс UNIQUE (user_id) WHERE status = 'active' отличается от обычного UNIQUE (user_id) без WHERE?  
<details><summary>Ответ</summary> **UNIQUE (user_id)** без WHERE: в **всей таблице** не может быть двух строк с одинаковым user_id (никаких дубликатов user_id). **UNIQUE (user_id) WHERE status = 'active'**: дубликаты user_id **запрещены только** среди строк с status = 'active'. Строки с другими status (expired, cancelled и т.д.) в индекс не входят — у нескольких строк может быть один и тот же user_id, если у них status != 'active'. Обычный UNIQUE ограничивает всю таблицу; уникальный частичный — только подмножество по условию WHERE.</details>

3. Зачем совмещать UNIQUE и частичный индекс (WHERE) в одном индексе вместо двух отдельных?  
<details><summary>Ответ</summary> **Один** индекс **UNIQUE (col) WHERE condition** одновременно: 1) **ограничивает** данные — не может быть двух строк с одинаковым col среди строк, удовлетворяющих condition; 2) **ускоряет** поиск по col при этом condition (тот же индекс используется и для проверки уникальности, и для поиска). Два отдельных объекта (обычный UNIQUE + частичный индекс без UNIQUE) не дают «уникальность только среди выбранных»; триггеры сложнее и медленнее. Один уникальный частичный индекс — и целостность, и скорость.</details>

**Три пункта, чтобы не забыть (уникальный частичный).** 1) **UNIQUE + WHERE** = уникальность проверяется **только** среди строк, удовлетворяющих условию WHERE; строки вне условия в индекс не входят и на уникальность по ним ограничение не действует. 2) Классический пример: «один активный заказ/подписка на пользователя» — UNIQUE (user_id) WHERE status = 'active'; закрытые/истёкшие могут дублировать user_id. 3) Такой индекс и **ограничивает** (нельзя две активных с одним user_id), и **ускоряет** поиск по (user_id) при condition.

#### Запомните

- **Уникальный частичный индекс** — UNIQUE (col) WHERE condition; уникальность только среди строк с condition.

### 32.3. Индекс по выражению

**Цель раздела.**  
Понять **индекс по выражению (функциональный индекс)**: индекс строится не по столбцу «как есть», а по **выражению** (например, LOWER(email), (created_at::date)). Условие/сортировка в запросе должны использовать **то же выражение**, чтобы индекс применялся. Функции в выражении должны быть **immutable**.

#### Термины

- **Индекс по выражению (expression index, functional index)** — `CREATE INDEX ... ON t (expression);`. Например, `CREATE INDEX idx_users_lower_email ON users (LOWER(email));`. В индексе хранятся **результаты выражения** для каждой строки; поиск по индексу — по значению выражения.
- **Immutable** — функция, результат которой для одних и тех же аргументов не меняется со временем и не зависит от сессии (например, LOWER, UPPER, date_trunc с константой). Только такие функции допустимы в индексе по выражению; иначе индекс мог бы стать некорректным при изменении результата функции.

#### Теория и правила

- **Запрос должен использовать то же выражение:** индекс по LOWER(email) используется для WHERE LOWER(email) = LOWER(?), но не для WHERE email = ? (разные выражения). Поэтому в приложении запросы к такому столбцу пишут через LOWER(email).
- **Примеры:** LOWER(email) для регистронезависимого поиска; (created_at::date) для поиска по дате без времени; выражение с константой для частичных индексов по выражению.

#### Простыми словами

**Индекс по выражению** — индекс не по «email», а по «LOWER(email)». Удобно для поиска без учёта регистра. Важно: в запросе тоже писать LOWER(email), иначе индекс не подойдёт. Функции в выражении должны быть стабильными (immutable).

#### Типичная ошибка

Создали индекс по **LOWER(email)** для регистронезависимого поиска, но в коде запросы пишут **WHERE email = ?** или **WHERE email = $1**. Планировщик **не использует** индекс по выражению: в индексе хранятся значения **LOWER(email)**, а условие в запросе — по столбцу **email** без функции. Это разные выражения. **Исправление:** везде, где ищем по email, писать **WHERE LOWER(email) = LOWER(?)**, тогда индекс будет использоваться. То же для любого индекса по выражению: условие в запросе должно **совпадать** с выражением в индексе.

#### Картинка в голове

**Индекс по выражению** — как **указатель не по «как напечатано»**, а по **нормализованному** виду. Вместо двух записей «Иванов» и «ИВАНОВ» в указателе одна запись «иванов» → страницы 5 и 12 (LOWER привёл к одному виду). Ищешь тоже по «иванову» — попадаешь в указатель. Если искать по «Иванов» как есть, в указателе такой записи нет — там только «иванов». Поэтому в запросе **обязательно** та же «формула» (LOWER(email)), и формула не должна со временем меняться (immutable): иначе в указателе одно, а в книге уже другое.

#### Как запомнить

**Индекс по выражению** = CREATE INDEX ... ON t (expression); запрос должен использовать **то же выражение**. Функции — **immutable**.

#### Проверь себя (32.3)

1. Почему индекс по LOWER(email) не используется для WHERE email = 'a@b.com'?  
<details><summary>Ответ</summary> Потому что в индексе хранятся **значения LOWER(email)** (например, 'a@b.com'), а не исходный email. Планировщик может использовать индекс только если условие запроса выражено через **то же выражение** — WHERE LOWER(email) = LOWER(?) или WHERE LOWER(email) = 'a@b.com'. Условие email = 'a@b.com' — другое выражение (просто столбец), индекс по выражению не подходит.</details>

2. Почему в выражении индекса допустимы только immutable-функции?  
<details><summary>Ответ</summary> Индекс хранит **результат выражения** на момент вставки/обновления строки. Если функция не immutable (например, зависит от текущего времени или от настроек сессии), тот же столбец позже может дать другое значение выражения — индекс станет **неконсистентным**: в индексе записано одно значение, а при поиске по «текущему» результату функции получится другое. Поэтому в индексе по выражению разрешены только функции, результат которых для одних и тех же аргументов **не меняется** (LOWER, date_trunc с константой и т.п.).</details>

3. Есть индекс CREATE INDEX idx ON events ((created_at::date)). Для какого условия в WHERE индекс будет использоваться: (A) WHERE created_at::date = '2024-01-15', (B) WHERE created_at = '2024-01-15 12:00:00'?  
<details><summary>Ответ</summary> Индекс будет использоваться для **(A)** WHERE created_at::date = '2024-01-15' — выражение в запросе **совпадает** с выражением в индексе (created_at::date). Для **(B)** условие задаёт полную дату-время (timestamp), это другое выражение; планировщик не привяжет его к индексу по (created_at::date). В запросе нужно писать **то же выражение**, что и в индексе.</details>

**Ещё раз самыми простыми словами:** индекс по выражению — индекс не по «как в столбце», а по «результату формулы» (LOWER(email), дата без времени и т.д.). В запросе нужно писать **ту же формулу**, иначе планировщик не привяжет условие к индексу. И формула должна быть «стабильной» (immutable) — иначе индекс и таблица разъедутся.

#### Запомните

- **Индекс по выражению** — индекс на выражение (LOWER(col), (col::date) и т.д.); запрос должен использовать то же выражение. Функции — immutable.
