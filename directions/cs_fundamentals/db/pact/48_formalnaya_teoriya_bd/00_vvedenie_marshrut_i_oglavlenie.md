## Часть XLVIII. Формальная теория баз данных

> Цель этой части — **собрать воедино формальные основы, на которых стоят СУБД и оптимизаторы запросов**,  
> разобраться, что на самом деле означают реляционное исчисление и алгебра, как устроены формальные фреймворки оптимизации (Volcano/Cascades),  
> чем отличаются эвристическая и стоимостная оптимизация, откуда берётся оценка кардинальностей,  
> и как формально описываются типы, NULL, ACID и сериализуемость.

---

### После этой части ты сможешь

- **объяснять реляционное исчисление (tuple/domain calculus) и его связь с реляционной алгеброй и SQL**: что именно означает теорема Кодда и почему “SELECT‑запрос — это формула логики предикатов”;
- **понимать архитектуру оптимизаторов запросов в стиле Volcano/Cascades**: что такое search space, memo‑структура, transformation rules, cost model, чем эта теория отличается от “куска кода с `if` в планировщике”;
- **различать heuristic (rule-based) и cost-based оптимизацию**: когда достаточно набора правил, когда нужен расчёт стоимости и как современные оптимизаторы комбинируют подходы;
- **формально смотреть на типы и NULL в SQL**: three-valued logic, partial functions, CHECK/UNIQUE/FOREIGN KEY как логические формулы, чем тут отличаются PostgreSQL, MySQL, SQLite, Oracle, SQL Server;
- **воспринимать оптимизацию запросов как задачу поиска в огромном пространстве планов**: понимать, где используется динамическое программирование, где — жадные и эвристические алгоритмы;
- **разбираться в основах оценки кардинальностей**: какие статистики хранят СУБД, как строятся гистограммы и выборки, почему ошибки в оценке ломают планы и что с этим делают разные системы;
- **формально мыслить о транзакциях и сериализуемости**: что такое история транзакций, граф зависимостей, конфликтная/видовая сериализуемость, как это связано с реализацией в PostgreSQL, MySQL, Oracle, SQL Server и в некоторых NoSQL‑СУБД;
- **смотреть на БД как на строгую математическую модель, а не только как на “чёрный ящик с SQL”**, и использовать эту модель в архитектурных решениях и при разборе сложных багов.

#### Проверь себя (что желательно знать заранее)

1. Можешь ли ты объяснить, чем `JOIN` в SQL отличается от декартова произведения с фильтром в реляционной алгебре?

<details><summary>Ответ</summary>
`JOIN` в SQL концептуально эквивалентен **декартову произведению с последующей фильтрацией по условию соединения**. В реляционной алгебре это `σ_условие (R × S)`. На практике СУБД реализует соединения оптимально (hash join, nested loop, merge join), но формально это всё та же операция над отношениями.
</details>

2. Понимаешь ли ты разницу между **логическим планом** запроса и **физическим**?

<details><summary>Ответ</summary>
Логический план описывает *что* нужно сделать с отношениями (проекция, селекция, соединение, группировка).  
Физический план описывает *как* это делать: какие алгоритмы использовать (hash join vs nested loop), по каким индексам ходить, в каком порядке выполнять операции. Формальная теория в этой части в основном живёт на границе между логическим и физическим планом.
</details>

### Краткая шпаргалка (термины по части)

| Термин                               | Что это простыми словами                                                                                                  |
| ------------------------------------ | ------------------------------------------------------------------------------------------------------------------------- |
| **Реляционная алгебра**             | Набор операций над отношениями (таблицами) как над множествами: проекция, селекция, соединение, разность, объединение.   |
| **Реляционное исчисление**          | Описание запросов как **логических формул**: “кортежи, для которых выполняется такой‑то предикат”.                        |
| **Tuple calculus / domain calculus** | Два варианта исчисления: по кортежам и по отдельным значениям (доменам).                                                 |
| **Теорема Кодда**                   | Утверждение, что реляционная алгебра и реляционное исчисление **эквивалентны по выразительной мощности**.               |
| **Volcano / Cascades framework**    | Формальные фреймворки для построения оптимизаторов: search space, memo‑структура, transformation rules, cost model.     |
| **Search space (пространство поиска)** | Набор всех допустимых логических/физических планов для одного и того же запроса.                                         |
| **Cost model**                      | Функция/модель, которая примерно оценивает “цену” плана: число чтений страниц, операций ввода‑вывода, CPU и т.п.        |
| **Heuristic / rule-based optimizer** | Оптимизатор, который применяет набор правил переписывания без точного расчёта стоимости всех вариантов.                 |
| **Cost-based optimizer (CBO)**      | Оптимизатор, который использует статистику и пытается выбрать план с минимальной оценочной стоимостью.                  |
| **Three-valued logic**             | Логика с тремя значениями: TRUE, FALSE, UNKNOWN (важна для работы с `NULL` в SQL).                                       |
| **Partial function (частичная функция)** | Функция, которая не определена для всех возможных аргументов (например, `1 / 0`).                                        |
| **Constraint**                      | Ограничение целостности (CHECK, UNIQUE, FOREIGN KEY) как логическая формула, которая должна быть истинна для данных.    |
| **Cardinality estimation**          | Оценка количества строк, которые вернёт подзапрос/операция (кардинальности промежуточных отношений).                     |
| **Histogram / MCV / NDV**          | Типы статистики: гистограммы распределения, наиболее частые значения, количество различных значений.                     |
| **History (история транзакций)**   | Последовательность операций разных транзакций во времени; основа для формального анализа сериализуемости.               |
| **Conflict / dependency graph**    | Граф конфликтов между операциями разных транзакций (по записи/чтению одних и тех же данных).                             |
| **Serializability**                | Свойство, что результат параллельного выполнения транзакций эквивалентен некоторому их **последовательному** порядку.   |

#### Проверь себя (по шпаргалке)

1. Что означает “эквивалентность по выразительной мощности” реляционной алгебры и исчисления?

<details><summary>Ответ</summary>
Что **любой запрос**, который можно выразить в реляционной алгебре, можно выразить и в реляционном исчислении, и наоборот.  
То есть это не “две разные теории”, а **два разных языка описания одного и того же класса запросов**.
</details>

2. Зачем оптимизатору нужна оценка кардинальностей?

<details><summary>Ответ</summary>
От кардинальностей зависят:
- выбор порядка соединений (join order),
- решение, стоит ли использовать индекс или сканировать таблицу целиком,
- выбор типа соединения (nested loop, hash, merge).  
Без разумной оценки количества строк cost model становится слепой, и оптимизатор может выбирать заведомо плохие планы.
</details>

### Маршрут изучения (простыми словами)

1. Сначала разберёмся с **реляционным исчислением** (§291): что это за логический язык, как он связан с алгеброй и SQL, зачем вообще инженеру БД думать об этом, если “SQL и так работает”.
2. Затем перейдём к **формальным фреймворкам оптимизации** (§292): Volcano/Cascades, memo‑структуры, правила переписывания и cost model — это “скелет” современных оптимизаторов PostgreSQL, SQL Server, Oracle и многих других СУБД.
3. На этой базе обсудим **эвристическую против стоимостной оптимизации** (§293): почему в одних местах достаточно набора правил, а в других важна точная оценка стоимости, и как гибридные подходы используются в реальных СУБД.
4. Потом погрузимся в **теорию типов и логику SQL** (§294): NULL, three-valued logic, частичные функции и ограничения как логические формулы — чтобы понимать, почему запросы с `NULL` часто ведут себя “не так, как ожидалось”.
5. После этого посмотрим на **оптимизацию как поиск** (§295): как устроено пространство планов, почему полный перебор невозможен, как помогают динамическое программирование и жадные алгоритмы.
6. Следующий шаг — **оценка кардинальностей** (§296): какие статистики поддерживают PostgreSQL, MySQL, Oracle, SQL Server; где ломается независимость столбцов; почему NoSQL‑движки часто используют упрощённые модели.
7. Завершим **теоретическими основами модели БД и транзакций** (§297): формальное определение реляционной модели, ACID, истории транзакций и сериализуемости. Это связывает всё, что мы делали раньше, с более строгой математической картиной.
8. В конце используем **справочник, частые сценарии, вопросы и типичные ошибки** — как чек‑лист и инструмент повторения перед собеседованиями и архитектурными решениями.

#### Проверь себя (по маршруту)

1. Почему мы начинаем с реляционного исчисления, а не сразу с Volcano/Cascades?

<details><summary>Ответ</summary>
Потому что Volcano/Cascades и вообще оптимизаторы работают с **логическими представлениями запросов**.  
Если не понимать, что запрос — это в сущности **формула логики над отношениями**, сложнее почувствовать, какие переписывания допустимы и почему оптимизатор “имеет право” менять порядок соединений или группировок.
</details>

2. Зачем в маршруте выделен отдельный шаг про теорию типов и `NULL`, если она вроде бы “и так понятна на практике”?

<details><summary>Ответ</summary>
Потому что именно `NULL` и трёхзначная логика чаще всего становятся источником **неочевидных багов и неправильных оптимизаций**.  
Формальный взгляд (NULL как bottom type, partial functions, логические формулы CHECK/FOREIGN KEY) помогает понять, почему разные СУБД иногда ведут себя неинтуитивно и какие запросы лучше переписать для предсказуемости.
</details>

### Как устроены разделы и как пользоваться оглавлением

Разделы (§291–§297) построены по единому шаблону (там, где это уместно):

- **Цель раздела** — что именно ты поймёшь/сможешь делать после чтения.
- **В этом разделе главное** — 3–6 опорных мыслей.
- **Термины** — мини‑словарик по разделу.
- **Теория и правила** — формальные определения в связке с интуитивными объяснениями.
- **Пошагово** — типичные процессы: как работает оптимизатор, как строится пространство планов, как считается кардинальность.
- **Простыми словами** — “человеческое” объяснение без формальных символов.
- **Картинка в голове** — метафора, которая помогает запомнить идею.
- **Как запомнить** — короткие формулы/антипутаница.
- **Примеры** — SQL‑запросы, куски псевдокода оптимизатора, иллюстрации поведения разных СУБД.
- **Практика / реальные сценарии** — как раздел помогает в продакшене (разбор планов, диагностика странных багов, проектирование схем).
- **Типичные ошибки** — что чаще всего делают не так, когда формальной картины нет.
- **Что будет, если…** — граничные случаи и последствия игнорирования теории.
- **Проверь себя** — 2–3 вопроса с ответами в `<details>`.
- **Запомните** — короткая выжимка.

Оглавление сгруппировано по этапам:

- **Этап 1** — формальные модели запросов и базовая логика (§291, §294).
- **Этап 2** — фреймворки оптимизации и поиск планов (§292–§293, §295).
- **Этап 3** — оценка кардинальностей и формальные основы транзакций (§296–§297) плюс справочник и вопросы.

Если тебе нужно:

- **понять, что именно “означает” SQL‑запрос с точки зрения логики** — сосредоточься на §291 и §294;
- **разобраться, как устроен оптимизатор PostgreSQL/MySQL/Oracle/SQL Server на концептуальном уровне** — прочитай §292–§293 и §295;
- **выяснить, почему оптимизатор упорно выбирает плохой план** — загляни в §296 (кардинальности) и вспомни группу про поиск планов (§295);
- **углубить понимание транзакций и уровней изоляции** — используй §297.
