## Справочник по части (формальная теория баз данных)

### Формальные модели и SQL

| Тема                                | Ключевые мысли                                                                                           |
| ----------------------------------- | -------------------------------------------------------------------------------------------------------- |
| Реляционная алгебра                 | Набор операций над отношениями как над множествами; основа для логических и физических планов.          |
| Реляционное исчисление              | Логика первого порядка над отношениями; запросы как формулы.                                            |
| Теорема Кодда                       | Безопасное исчисление и алгебра эквивалентны по выразительной мощности.                                 |
| SQL как исчисление                  | SQL — удобный синтаксис для ограниченной формы исчисления + алгебра “под капотом”.                      |
| Безопасные запросы                  | Запросы, дающие конечное множество, зависящее только от содержимого БД.                                 |

### Оптимизатор, правила и стоимость

| Тема                                | Ключевые мысли                                                                                           |
| ----------------------------------- | -------------------------------------------------------------------------------------------------------- |
| Volcano/Cascades                    | Фреймворки оптимизации: memo, группы, transformation rules, cost model.                                 |
| Memo                                | Группирует эквивалентные выражения, позволяет переиспользовать оценки и подрешения.                    |
| Rule-based оптимизация             | Набор эвристик/правил переписывания без точной оценки стоимости всех вариантов.                         |
| Cost-based оптимизация              | Использует статистику и модель стоимости, чтобы выбрать лучший план.                                    |
| Поиск планов                        | Комбинация DP, жадных алгоритмов, pruning; пространство планов растёт комбинаторно.                     |

### Логика, `NULL` и типы

| Тема                                | Ключевые мысли                                                                                           |
| ----------------------------------- | -------------------------------------------------------------------------------------------------------- |
| Three-valued logic                  | TRUE/FALSE/UNKNOWN; сравнения с `NULL` дают UNKNOWN; `IS NULL`/`IS NOT NULL` — специальные операторы.   |
| `NULL` как bottom type              | Отсутствующее/неизвестное значение, принадлежащее любому типу, но не равное ни одному значению.        |
| Частичные функции                   | Множество функций в SQL не определены для всех аргументов; часто “нет результата” выражается через `NULL`. |
| Constraints как формулы             | CHECK/UNIQUE/FOREIGN KEY — логические ограничения, которые должны быть истинны для всех строк.         |

### Кардинальности и транзакции

| Тема                                | Ключевые мысли                                                                                           |
| ----------------------------------- | -------------------------------------------------------------------------------------------------------- |
| Cardinality estimation              | Оценка числа строк на выходе операторов; ключ к работе cost model.                                      |
| Статистика (MCV, NDV, гистограммы)  | Сводки о распределении значений; основы для вычисления селективности.                                  |
| Истории транзакций и конфликты      | Формальное описание параллельного выполнения; основа для сериализуемости.                              |
| Сериализуемость                     | Эквивалентность некоторому последовательному порядку; проверяется через граф конфликтов.               |
| ACID формально                      | Строгие определения атомарности, согласованности, изоляции и устойчивости.                             |

## Частые сценарии

### Сценарий 1. Два эквивалентных запроса ведут себя по‑разному

- **Симптом**: запрос с `EXISTS` работает намного быстрее, чем, казалось бы, эквивалентный запрос с `IN` или `JOIN`.
- **Анализ**:
  - формально оба запроса эквивалентны (исчисление/алгебра),
  - но оптимизатор в одном случае находит более удачный план (например, semi‑join вместо обычного `JOIN`).
- **Действия**:
  - выписать формулы исчисления/алгебры;
  - посмотреть `EXPLAIN`, чтобы понять, какое переписывание использует оптимизатор;
  - при необходимости **помочь** оптимизатору (переписать запрос ближе к форме, которую он лучше оптимизирует).

### Сценарий 2. Внезапно плохой план после роста данных

- **Симптом**: запросы внезапно стали медленными, план лишился индексного доступа или выбрал неверный порядок `JOIN`.
- **Анализ**:
  - вероятно, изменилась статистика и/или распределение значений;
  - cost-based оптимизатор на основе **устаревших кардинальностей** принял плохое решение.
- **Действия**:
  - сравнить оценочные и фактические `rows` в плане;
  - обновить статистику (`ANALYZE` / `UPDATE STATISTICS`);
  - при необходимости добавить многомерную статистику или изменить запрос.

### Сценарий 3. Странные результаты из‑за `NULL` и `NOT IN`

- **Симптом**: запрос `NOT IN` “иногда” возвращает пустой набор, хотя явно должны быть строки.
- **Анализ**:
  - в подзапрос попали строки с `NULL`;
  - трёхзначная логика делает сравнение `NOT IN (… NULL …)` неопределённым.
- **Действия**:
  - добавить `WHERE col IS NOT NULL` внутрь подзапроса;
  - или заменить на `NOT EXISTS`‑форму;
  - перепроверить логику на формальном уровне.

### Сценарий 4. Аномалии транзакций на “безопасных” уровнях

- **Симптом**: редкие кейсы, когда данные “теряются” или ведут себя неконсистентно при уровне `READ COMMITTED`/`REPEATABLE READ`.
- **Анализ**:
  - формальная теория показывает, что при этих уровнях допускаются некоторые аномалии;
  - конкретная комбинация операций привела к неповторяемым чтениям или фантомам.
- **Действия**:
  - описать ситуацию как историю операций;
  - проверить, к какому классу аномалий она относится;
  - при необходимости перейти на более строгий уровень изоляции или переписать логику.

## Вопросы для самопроверки по части

1. Объясни, как связаны между собой реляционная алгебра, реляционное исчисление и SQL.

<details><summary>Ответ</summary>
Реляционная алгебра и безопасное реляционное исчисление **эквивалентны по выразительной мощности** (теорема Кодда).  
SQL исторически создавался как удобный текстовый синтаксис, близкий к исчислению (формулы с кванторами и предикатами), а оптимизаторы переписывают SQL в **алгебраические выражения**, для которых легче строить планы. Таким образом, SQL → исчисление → алгебра → план.
</details>

2. Почему оптимизатору достаточно иметь набор transformation rules и cost model, чтобы находить хорошие планы (вообще говоря)?

<details><summary>Ответ</summary>
Правила описывают **пространство эквивалентных выражений** (логических планов), а cost model даёт способ **оценить качество каждого кандидата**.  
Если пространство поиска достаточно богато, а cost model хотя бы приближённо отражает реальные затраты, то оптимизатор, перебирая (часть) этого пространства и сравнивая стоимости, способен находить планы, близкие к оптимальным.
</details>

3. Как формальная трёхзначная логика помогает объяснить поведение `NOT IN` с `NULL`?

<details><summary>Ответ</summary>
Потому что сравнение `x = NULL` даёт **UNKNOWN**, а не FALSE, и логические операции с UNKNOWN ведут себя по таблице трёхзначной логики.  
В выражении `x NOT IN (… NULL …)` результат становится UNKNOWN, если в множестве есть `NULL`, и в контексте `WHERE` это означает, что строка отбрасывается. Формальная таблица истинности явно показывает, почему так происходит.
</details>

4. Что такое история транзакций и как её использовать для анализа уровней изоляции?

<details><summary>Ответ</summary>
История (schedule) — это упорядоченная последовательность операций чтения/записи/фиксации разных транзакций.  
Анализируя историю, можно:
- построить граф конфликтов,
- проверить, сериализуема ли она,
- отнести наблюдаемое поведение к определённому классу аномалий (потерянное обновление, фантомы и т.п.),  
и затем сопоставить это с тем, какие аномалии допускает или запрещает конкретный уровень изоляции.
</details>

5. Почему без обновлённой статистики даже формально корректная cost model даёт плохие планы?

<details><summary>Ответ</summary>
Потому что модель опирается на **входные данные — кардинальности и распределения**, а статистика — это именно эти входные данные.  
Если статистика устарела (например, таблица выросла в 10 раз или распределение значений сильно изменилось), оценка селективности и кардинальностей будет неверной, и оптимизатор будет сравнивать планы по “фейковым” числам, выбирая неправильные стратегии.
</details>

## Типичные ошибки по части

- **Считать SQL чисто “императивным языком”**, не видя за ним формальной логики и алгебры.
- **Игнорировать трёхзначную логику и `NULL`** при проектировании схемы и запросов.
- **Ожидать, что оптимизатор всегда найдёт лучший план**, не заботясь о статистике и пределах пространства поиска.
- **Переоценивать rule-based подход** (“мы напишем пару правил, и всё будет хорошо”) для сложных запросов и больших данных.
- **Не учитывать, что уровни изоляции в разных СУБД реализованы по‑разному**, и переносить интуицию “как в PostgreSQL” на MySQL/Oracle/SQL Server без проверки.
- **Не закладывать в код обработку serialization failures** при работе с `SERIALIZABLE`, считая, что это “редкая и ненормальная ситуация”.
