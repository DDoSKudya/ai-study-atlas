## 295. Оптимизация запросов как поиск

### Цель раздела

Понять, **как выглядит пространство планов для запроса**,  
почему полный перебор невозможен, какие используются алгоритмы поиска (динамическое программирование, жадные, эвристические),  
и как это связано с практическими настройками оптимизаторов в СУБД.

### В этом разделе главное

1. Пространство возможных планов для запроса растёт **комбинаторно** с числом таблиц.
2. Классическая работа System R предложила использовать **динамическое программирование** для поиска порядка соединений.
3. Современные оптимизаторы комбинируют DP, жадные алгоритмы и pruning (отсечение плохих веток).
4. Параметры СУБД (`join_collapse_limit` в PostgreSQL и аналоги в других СУБД) напрямую ограничивают пространство поиска.
5. Иногда **уменьшение** пространства поиска улучшает не только время планирования, но и качество планов (за счёт отсечения заведомо плохих вариантов).

### Термины

- **Search space (пространство поиска)** — множество всех возможных планов (логических/физических) для запроса.
- **Dynamic programming (DP)** — метод, при котором сначала оптимизируются **малые подмножества** таблиц, а затем результаты переиспользуются.
- **Greedy algorithm (жадный алгоритм)** — алгоритм, который на каждом шаге делает “локально лучший” выбор.
- **Pruning** — отсечение ветвей поискового дерева, которые заведомо не дадут лучшего результата.

### Теория и правила

#### 1) Размер пространства планов

Для запроса с `n` таблицами:

- количество разных **деревьев соединений** (join trees) растёт примерно как числа Каталана;
- если учесть разные типы `JOIN` и предикаты — рост ещё быстрее.

На практике уже при 10–12 таблицах:

- полный перебор всех join‑порядков становится нереалистичным.

#### 2) Dynamic programming (подход System R)

Идея (упрощённо):

1. Для каждого **одиночного отношения** `R` вычислить лучший план доступа (скан, индекс).
2. Для каждой пары отношений `R, S` вычислить лучший план `R ⋈ S`, используя результаты из шага 1.
3. Для троек `R, S, T` — использовать уже оптимальные планы для `R ⋈ S`, `R ⋈ T`, `S ⋈ T` и т.д.
4. Продолжать до подмножества из всех `n` таблиц.

Преимущества:

- переиспользование информации о подзапросах;
- возможность систематически наращивать подмножества.

Недостатки:

- всё ещё экспоненциальный рост в худшем случае;
- для очень больших `n` требуется ограничивать подмножества.

#### 3) Greedy и гибридные алгоритмы

Жадная стратегия:

- начиная с одной таблицы, **на каждом шаге добавлять следующую таблицу**, выбирая соединение, которое даёт минимальное увеличение стоимости.

Плюсы:

- быстро, не требует хранения всех подмножеств.

Минусы:

- может легко пропустить глобально лучший план (классическая проблема жадных алгоритмов).

Гибрид:

- использовать DP для **небольших подмножеств** (например, до 8–10 таблиц),
- для больших запросов включать жадные/эвристические стратегии и pruning.

### Пошагово

Пусть есть запрос:

```sql
SELECT *
FROM A
JOIN B ON ...
JOIN C ON ...
JOIN D ON ...;
```

**Dynamic programming (упрощённо)**:

1. Рассчитать лучшие планы для:
   - `{A}`, `{B}`, `{C}`, `{D}`.
2. Рассчитать лучшие планы для:
   - `{A,B}`, `{A,C}`, `{A,D}`, `{B,C}`, `{B,D}`, `{C,D}`.
3. Рассчитать лучшие планы для:
   - `{A,B,C}`, `{A,B,D}`, `{A,C,D}`, `{B,C,D}`.
4. Рассчитать лучший план для:
   - `{A,B,C,D}`.

**Жадный алгоритм**:

1. Выбрать таблицу с минимальной оценочной “стоимостью” скана (скажем, `A`).
2. Выбрать таблицу `X`, при присоединении которой к `A` получается минимальная стоимость (`A ⋈ B`).
3. К `{A,B}` добавить третью таблицу `Y`, дающую минимальное увеличение стоимости (`(A ⋈ B) ⋈ C`).
4. И так далее.

### Простыми словами

Оптимизация запросов — это как **поиск лучшего маршрута** в графе:

- узлы — **состояния** (“мы уже соединили такие‑то таблицы таким образом”),
- рёбра — **шаги соединения** (“добавить таблицу D”),
- вес — **оценочная стоимость**.

Алгоритмы:

- DP — это как “посчитать всевозможные под‑маршруты и потом их комбинировать”;
- жадный — это “на каждом шаге брать следующий поворот, который сейчас кажется лучшим”.

### Картинка в голове

Представь дерево, где:

- корень — “ещё не начали соединять таблицы”,
- листья — “все таблицы соединены”.

Полный перебор — это пройти **все пути от корня к листьям**.  
DP и жадные алгоритмы — способы **умно пропускать** часть путей, не теряя (или почти не теряя) качество.

### Как запомнить

- **DP → “строим снизу вверх, переиспользуем подрешения”.**
- **Greedy → “на каждом шаге берём локально лучший вариант”.**
- **Pruning → “не тратим время на ветки, которые уже точно хуже лучших найденных”.**

### Примеры

- PostgreSQL:
  - параметр `join_collapse_limit` определяет, до какого числа таблиц оптимизатор будет активно переставлять join‑порядок;
  - при очень больших запросах он использует более простые стратегии.
- Oracle/SQL Server:
  - имеют сложные механизмы поиска планов, комбинируя DP, эвристики и hints;
  - документация и whitepaper’ы от Microsoft/Oracle описывают эти подходы в терминах DP и поиска.

### Практика / реальные сценарии

- Большие запросы (20+ таблиц) почти всегда требуют:
  - либо **упрощения** запроса (разделение на несколько шагов, CTE с материализацией),
  - либо **подсказок** оптимизатору (hints, re‑writing), чтобы пространство поиска было управляемым.
- DBA и разработчики могут:
  - смотреть на `EXPLAIN` и понимать, **что оптимизатор даже не пытался рассмотреть** некоторые варианты из‑за ограничений поиска.

### Типичные ошибки

- Писать “монолитные” запросы на десятки таблиц, надеясь, что оптимизатор **сам всё разрулит**.
- Неправильно интерпретировать поведение оптимизатора, не учитывая ограничения search space.
- Считать, что “если мы включили все фичи оптимизации, хуже точно не станет” — в реальности рост search space может ухудшить и время планирования, и качество планов.

### Что будет, если…

- **Сильно увеличить лимиты пространства поиска**:
  - время планирования может вырасти на порядки;
  - некоторые запросы будут “думать” дольше, чем выполняться.
- **Слишком сильно ужать пространство поиска**:
  - оптимизатор перестанет находить хорошие планы для сложных запросов;
  - производительность упадёт без видимых причин.

### Проверь себя

1. Почему при 10+ таблицах полный перебор join‑порядков практически невозможен?

<details><summary>Ответ</summary>
Потому что количество разных деревьев соединений растёт **комбинаторно** (примерно как числа Каталана).  
Даже если рассматривать только бинарные `JOIN` и игнорировать разные типы соединений, число вариантов становится астрономическим, и просчитать их все нереально.
</details>

2. В чём ключевое преимущество динамического программирования по сравнению с наивным перебором?

<details><summary>Ответ</summary>
В том, что DP **переиспользует результаты**: один раз посчитав лучший план для подмножества таблиц `{A,B}`, можно использовать его во многих более крупных комбинациях (`{A,B,C}`, `{A,B,D}` и т.п.), а не пересчитывать каждый раз с нуля.
</details>

3. Почему жадный алгоритм не гарантирует нахождение глобально оптимального плана?

<details><summary>Ответ</summary>
Потому что он делает выбор, исходя только из **локальной информации** “здесь и сейчас”, не видя всех последствий на последующих шагах.  
Классический пример: чтобы прийти к действительно лучшему плану, иногда нужно на промежуточном шаге выбрать вариант, который кажется дороже, но открывает более дешёвые комбинации дальше.
</details>

### Запомните

- Оптимизация запросов — это **задача поиска**, а не “магическая формула”.
- DP, жадные алгоритмы и pruning — стандартные инструменты, которыми современные СУБД управляют взрывным ростом пространства планов.
