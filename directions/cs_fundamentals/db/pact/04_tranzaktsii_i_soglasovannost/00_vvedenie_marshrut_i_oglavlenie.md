## Часть IV. Транзакции и согласованность

> Цель этой части — **понять, как СУБД обеспечивает надёжность и предсказуемость при одновременной работе многих пользователей**: что такое транзакция и ACID, как работают уровни изоляции и аномалии чтения, зачем нужны блокировки и MVCC, как устроены двухфазный коммит и длинные транзакции, и как защищаться от lost update с помощью оптимистичной и пессимистичной блокировки и идемпотентности.

После этой части ты сможешь:

- **понимать ACID** — атомарность, согласованность, изоляцию и долговечность — и применять `BEGIN`, `COMMIT`, `ROLLBACK`, `SAVEPOINT` осознанно;
- **различать уровни изоляции** (READ UNCOMMITTED, READ COMMITTED, REPEATABLE READ, SERIALIZABLE) и **аномалии** (грязное, неповторяющееся, фантомное чтение);
- **использовать блокировки** — строковые и табличные, `SELECT FOR UPDATE` / `FOR SHARE`, `SKIP LOCKED`, advisory locks — и понимать, что такое deadlock и как его избегать;
- **понимать MVCC** — версионирование строк (xmin/xmax), видимость, snapshot isolation, SSI (serializable snapshot isolation);
- **знать основы двухфазного коммита и XA** для распределённых транзакций и **влияние длинных транзакций** на bloat и репликацию;
- **применять оптимистичную и пессимистичную блокировку**, защищаться от lost update и проектировать идемпотентные операции.

Материал опирается на **Часть II и III (SQL)** и готовит к темам хранения (WAL, VACUUM) и репликации. Если при чтении кажется, что «слишком много терминов» — в начале раздела они расшифрованы; блоки **«Простыми словами»** и **«Запомните»** повторяют ту же идею. Иди по оглавлению по этапам; по ссылке переходи в раздел; в конце части — справочник, вопросы для самопроверки и резюме для повторения. Примеры стоит пробовать в своей БД.

### Краткая шпаргалка по части

| Понятие / конструкция | Суть (что это и когда нужно) |
|----------------------|-----------------------------|
| **ACID** | Атомарность, согласованность, изоляция, долговечность — четыре свойства надёжной транзакции |
| **Атомарность** | Транзакция выполняется целиком или не выполняется; при ошибке — откат |
| **Согласованность** | Переход из одного корректного состояния в другое; ограничения и инварианты соблюдаются |
| **Изоляция** | Параллельные транзакции не «видят» незавершённые изменения друг друга (в зависимости от уровня) |
| **Долговечность** | После COMMIT данные сохранены на диске (через WAL) и не теряются при сбое |
| **BEGIN / START TRANSACTION** | Начало транзакции; последующие команды выполняются в её контексте |
| **COMMIT** | Завершение транзакции с сохранением всех изменений |
| **ROLLBACK** | Отмена транзакции; все изменения в рамках транзакции отменяются |
| **SAVEPOINT name** | Точка сохранения внутри транзакции; можно откатиться к ней через ROLLBACK TO |
| **READ UNCOMMITTED** | Уровень изоляции: возможно грязное чтение; редко используется |
| **READ COMMITTED** | Только закоммиченные данные; возможно неповторяющееся чтение |
| **REPEATABLE READ** | Снимок данных на начало транзакции; в некоторых СУБД возможно фантомное чтение |
| **SERIALIZABLE** | Полная изоляция; результат как при последовательном выполнении; блокировки или SSI |
| **Грязное чтение** | Чтение незакоммиченных данных другой транзакции |
| **Неповторяющееся чтение** | Повторное чтение той же строки даёт другое значение (другая транзакция изменила и закоммитила) |
| **Фантомное чтение** | Появление или исчезновение строк в диапазоне между двумя чтениями |
| **Блокировка строки** | При UPDATE/DELETE строка блокируется до COMMIT/ROLLBACK |
| **SELECT FOR UPDATE** | Явная блокировка строк для последующего обновления; другие не могут их изменить |
| **SELECT FOR UPDATE SKIP LOCKED** | Блокировать только незаблокированные строки; для очередей задач без ожидания |
| **LOCK TABLE** | Блокировка всей таблицы (SHARE / EXCLUSIVE) |
| **Deadlock** | Две (или более) транзакции ждут друг друга; СУБД откатывает одну из них |
| **MVCC** | Multiversion Concurrency Control — несколько версий строки; читатели не блокируют писателей |
| **xmin / xmax** | Идентификаторы транзакций в заголовке строки (PostgreSQL); видимость версии |
| **Snapshot isolation** | Транзакция видит снимок данных на момент своего начала |
| **SSI** | Serializable Snapshot Isolation — сериализуемость через обнаружение rw-конфликтов |
| **2PC** | Двухфазный коммит: PREPARE TRANSACTION → COMMIT PREPARED; для распределённых транзакций |
| **XA** | Стандарт распределённых транзакций с внешним координатором |
| **Пессимистичная блокировка** | SELECT FOR UPDATE — блокируем строку заранее, чтобы никто не изменил |
| **Оптимистичная блокировка** | Версионирование (version/timestamp); при UPDATE проверяем, что версия не изменилась |
| **Lost update** | Две транзакции читают и обновляют одну строку; вторая перезаписывает первую; решается FOR UPDATE или версией |
| **Идемпотентность** | Повторный вызов даёт тот же результат; идемпотентный ключ, ON CONFLICT DO NOTHING |

Используй таблицу как карту: нашёл понятие в первом столбце → во втором видишь суть и типичное применение. Для повторения после изучения раздела — сверяйся с этой шпаргалкой.

### Маршрут изучения в этой части

1. **ACID и управление транзакциями (раздел 16)**  
   Понять четыре свойства транзакции и как явно управлять ею: `BEGIN`, `COMMIT`, `ROLLBACK`, `SAVEPOINT`, `ROLLBACK TO SAVEPOINT`. Разобраться с автокоммитом и неявным началом транзакции.

2. **Уровни изоляции (раздел 17)**  
   Разобрать четыре уровня изоляции по стандарту SQL и три классические аномалии (грязное, неповторяющееся, фантомное чтение). Понять, как устанавливать уровень в сессии и чем отличаются реализации в PostgreSQL и MySQL.

3. **Блокировки (раздел 18)**  
   Научиться блокировать строки и таблицы: блокировки при UPDATE/DELETE, `SELECT FOR UPDATE` / `FOR SHARE`, `SKIP LOCKED`, `LOCK TABLE`, advisory locks. Понять deadlock и стратегии уменьшения блокировок.

4. **MVCC (раздел 19)**  
   Понять многоверсионность: версионирование строк (xmin/xmax), правила видимости, snapshot isolation, связь с REPEATABLE READ и SERIALIZABLE в PostgreSQL (SSI). Кратко — мёртвые строки, VACUUM, bloat, transaction ID wraparound.

5. **Двухфазный коммит и длинные транзакции (раздел 20)**  
   Разобрать 2PC (`PREPARE TRANSACTION`, `COMMIT PREPARED`) и XA для распределённых транзакций. Понять влияние длинных транзакций на bloat, репликацию и idle in transaction; рекомендации по коротким транзакциям.

6. **Оптимистичная и пессимистичная блокировка (раздел 21)**  
   Сравнить пессимистичную блокировку (`SELECT FOR UPDATE`) и оптимистичную (версия в столбце). Разобрать lost update и способы защиты. Понять идемпотентность и `SELECT FOR UPDATE SKIP LOCKED` для очередей.

### Частые сценарии: задача → решение

Таблица для быстрого поиска: по задаче находишь нужную конструкцию (терминология сохранена).

| Тема | Задача | Решение |
|------|--------|---------|
| **Транзакция** | Несколько операций «всё или ничего» (перевод, заказ с позициями) | `BEGIN` → команды → `COMMIT` или `ROLLBACK` |
| **Транзакция** | Откатить только часть изменений внутри транзакции | `SAVEPOINT name` перед риском; при ошибке `ROLLBACK TO SAVEPOINT name` |
| **Изоляция** | Увидеть только закоммиченные данные | Уровень `READ COMMITTED` (по умолчанию в PostgreSQL) |
| **Изоляция** | Повторное чтение той же строки даёт то же значение | Уровень `REPEATABLE READ` или выше |
| **Блокировки** | Защитить строку от изменения до своего UPDATE (защита от lost update) | `SELECT ... FOR UPDATE` перед изменением |
| **Блокировки** | Взять задачи из очереди без ожидания чужих блокировок | `SELECT ... FOR UPDATE SKIP LOCKED` |
| **Блокировки** | Заблокировать по прикладному ключу (не по строке таблицы) | Advisory lock: `pg_advisory_lock(key)` / `pg_advisory_unlock(key)` |
| **Блокировки** | Избежать взаимной блокировки (deadlock) | Одинаковый порядок блокировок во всех транзакциях (например, по возрастанию id) |
| **MVCC** | Понять, почему читатели не блокируют писателей | Несколько версий строки; читатель видит снимок на начало транзакции |
| **2PC** | Атомарно закоммитить в двух базах или база + очередь | `PREPARE TRANSACTION` у всех → `COMMIT PREPARED` или `ROLLBACK PREPARED` |
| **Длинные транзакции** | Не держать транзакцию открытой без активности | Короткие транзакции; `idle_in_transaction_session_timeout` в production |
| **Lost update** | Две транзакции не должны перезаписать друг друга | Пессимистично: `SELECT FOR UPDATE`; оптимистично: столбец `version`, проверка при UPDATE |
| **Идемпотентность** | Повторная отправка запроса не должна дублировать эффект (оплата, создание заказа) | Идемпотентный ключ от клиента; таблица ключей + результат; при повторе — вернуть сохранённый результат |

### Как связаны темы в этой части (карта идей)

Главные идеи части в одном месте — чтобы привязывать к ним новые примеры по ходу чтения:

- **Транзакция** — пакет изменений с BEGIN и COMMIT/ROLLBACK; ACID — атомарность, согласованность, изоляция, долговечность.
- **Уровни изоляции** — что видит транзакция от других; аномалии: грязное, неповторяющееся, фантомное чтение.
- **Блокировки** — защита критических участков: FOR UPDATE, SKIP LOCKED, advisory locks; deadlock — избегать одинаковым порядком блокировок.
- **MVCC** — несколько версий строки; читатели не блокируют писателей; снимок на начало транзакции; мёртвые строки → VACUUM, bloat.
- **2PC и длинные транзакции** — распределённый коммит (PREPARE → COMMIT PREPARED); длинная транзакция «держит» старые версии → bloat и lag репликации.
- **Lost update и идемпотентность** — пессимистичная блокировка (FOR UPDATE) vs оптимистичная (version); повторный вызов не дублирует эффект (идемпотентный ключ).

**Логика по шагам:** разделы 16–17 — **что такое транзакция и изоляция**; раздел 18 — **блокировки**; раздел 19 — **как СУБД даёт параллелизм** (MVCC); разделы 20–21 — **распределённые транзакции, длинные сессии, защита от потери обновлений и идемпотентность**.
