## §224. JSON и JSONB

### Цель раздела

Разобраться, когда JSON в реляционной БД — хороший инструмент, чем отличается JSON от JSONB, какие есть операторы и индексы (особенно GIN), как использовать `jsonpath`, и как избежать типичных проблем: медленные запросы, отсутствие ограничений, “динамическая схема без контроля”.

### В этом разделе главное

1) **JSON — это формат, JSONB — структура для запросов.** В PostgreSQL почти всегда выигрывает `jsonb`, если ты собираешься фильтровать/индексировать.  
2) JSON удобен для “полуструктурированных” данных, но **инварианты и ограничения в JSON сложнее** (по сравнению с колонками).  
3) Основные операторы Postgres: `->`, `->>`, `#>`, `#>>`, `@>`, `?` и др. Они задают разные типы результата и разную индексируемость.  
4) **GIN индекс даёт ускорение для операторов, которые “смотрят внутрь” JSONB** (например, `@>`), но не делает магии для любых выражений.  
5) Практический баланс: **часто используемые поля — в колонках**, “редкие/гибкие” — в JSONB. Это даёт и скорость, и эволюцию схемы.

### Термины

| Термин | Определение |
|---|---|
| **JSON** | Текстовое представление JSON (в PostgreSQL — хранится как текст). |
| **JSONB** | Бинарное/разобранное представление JSON (PostgreSQL), удобно для поиска и индексации. |
| **Containment (`@>`)** | Оператор “содержит”: левый JSONB содержит правый JSONB как подструктуру. |
| **Key existence (`?`)** | Проверка существования ключа на верхнем уровне (и есть варианты для нескольких ключей). |
| **GIN** | Индекс для “множества элементов” (ключи/значения), часто применяют к JSONB. |
| **jsonpath** | Язык пути/фильтрации для JSON (SQL/JSON Path, в Postgres — функции/операторы). |

### Теория и правила

#### 0) Сначала “когда вообще JSON уместен” (интуитивно, до деталей)

Очень частая ошибка новичка: “JSON — удобно, значит всё сложим в JSON”.

Чтобы не попасть в это, держи в голове простую шкалу:

- **Колонки** — когда поле важно, часто используется, по нему строятся индексы, ограничения, JOIN и отчёты.
- **JSONB** — когда поле “гибкое”: много редких атрибутов, которые меняются, и ты не хочешь каждую неделю делать миграции схемы.

Практический критерий (очень приземлённый):

- если по ключу внутри JSON появляется `WHERE`, `ORDER BY`, `GROUP BY`, `JOIN` **в каждом втором запросе** — это почти всегда сигнал: “пора вынести в колонку”.

##### Проверь себя (к подпункту “когда JSON уместен”)

<details>
<summary>Сравни два решения: “всё в JSONB” и “колонки + JSONB”. Что ты выигрываешь и что теряешь в каждом?</summary>

“Всё в JSONB” выигрывает гибкость и скорость изменений схемы, но теряет строгую типизацию, ограничения (NOT NULL/FK/CHECK), предсказуемую индексацию и простоту отчётов. “Колонки + JSONB” сохраняют строгие инварианты и быстрые запросы по горячим полям, а JSONB остаётся “карманом” для редких/расширяемых атрибутов.
</details>

<details>
<summary>Приведи пример “редкого/гибкого” атрибута, который разумно оставить в JSONB, и “горячего” атрибута, который почти всегда нужно вынести в колонку.</summary>

Редкий/гибкий: экспериментальные параметры события, набор пользовательских настроек, который часто расширяется. Горячий: `status`, `created_at`, `user_id`, `country`, `price` — то, что регулярно в `WHERE/ORDER/GROUP/JOIN` и требует индекса и строгого типа.
</details>

<details>
<summary>Почему критерий “ключ внутри JSON встречается в каждом втором запросе” — хороший сигнал, даже если “пока работает быстро”?</summary>

Потому что рост данных превращает “пока работает” в “внезапно стало дорого”: фильтрация внутри JSON часто хуже индексируется и дороже по CPU/I/O. Если ключ стал частью основной модели доступа, его выгоднее сделать частью схемы (колонка/индекс/ограничения).
</details>

#### 1) JSON vs JSONB (PostgreSQL) — смысловая разница

##### Сначала интуитивно (коробка vs полки — ещё раз, но точнее)

JSON (text) — это как “коробка с бумажной наклейкой”: чтобы найти “status”, нужно открыть коробку и прочитать бумагу.  
JSONB — это как “коробка, внутри которой уже есть отсортированный список вещей”: искать проще, и можно строить индекс.

**JSON (text):**

- хранится как текст;
- может сохранять форматирование (пробелы), порядок ключей (в некоторых реализациях) — но это редко полезно;
- обычно хуже для частых запросов по полям.

**JSONB:**

- хранится в бинарной форме (разобранное дерево);
- порядок ключей не важен (нормализуется);
- поддерживает индексацию и быстрые операторы.

Практическое правило для PostgreSQL:

- если JSON нужен только “как blob” и ты редко его читаешь — JSON или JSONB не принципиально;
- если ты фильтруешь/индексируешь/агрегируешь по содержимому — **почти всегда JSONB**.

##### Проверь себя (к подпункту “JSON vs JSONB”)

<details>
<summary>Почему JSONB обычно быстрее для запросов по содержимому, чем JSON (текст), даже если “данные те же”?</summary>

JSON как текст нужно каждый раз разбирать/интерпретировать при сложных операциях, а JSONB хранит разобранную структуру (дерево), что позволяет эффективнее выполнять операторы и строить индексы (GIN). Данные “те же”, но форма хранения разная.
</details>

<details>
<summary>Назови ситуацию, где JSON (текст) может быть допустим, и почему при этом ты сознательно принимаешь цену.</summary>

Когда JSON хранится как “blob” для редкого чтения/логирования/аудита, и по нему почти не фильтруют. Цена: сложные запросы и индексация по содержимому будут хуже, поэтому ты заранее выбираешь “храню, но не ищу”.
</details>

<details>
<summary>Почему “почти всегда JSONB” — это не догма, а правило с контекстом?</summary>

Потому что выбор зависит от паттернов доступа: если JSON нужен только как текст и не участвует в фильтрах/индексах, выигрыш JSONB может быть несущественным. Но как только появляются запросы по ключам/значениям, JSONB почти всегда практичнее.
</details>

#### 2) Главный риск JSON в реляционной БД: потеря строгих ограничений

Когда ты переносишь поля в JSON, ты обычно теряешь:

- `NOT NULL` по конкретным ключам;
- `FOREIGN KEY` по значениям внутри JSON;
- `CHECK` в удобном виде;
- типизацию на уровне колонок.

Выходы:

- держать критичные поля в колонках (и лишь “доп. атрибуты” в JSONB);
- использовать `generated columns` (где доступно) для “вытаскивания” ключей в индексируемую колонку;
- использовать `CHECK` на JSONB (в PostgreSQL можно проверять `jsonb_typeof`, наличие ключей и т.п.), но это сложнее.

##### Проверь себя (к подпункту “потеря ограничений”)

<details>
<summary>Какие 3 вида ограничений ты чаще всего “теряешь”, когда переносишь поля из колонок в JSON?</summary>

Обычно: `NOT NULL` по конкретному полю, `FOREIGN KEY` на связанный идентификатор, и простые `CHECK`/типизацию (число/дата/enum‑домен). В JSON это либо сложнее, либо переносится в приложение.
</details>

<details>
<summary>Почему generated column/вычисляемая колонка — универсальный паттерн для JSON почти во всех СУБД?</summary>

Потому что он превращает “ключ внутри документа” в обычное типизированное поле, по которому можно строить привычный B-tree индекс и ограничения. Это работает в PostgreSQL/MySQL/SQL Server/Oracle (с разным синтаксисом), потому что идея одинаковая.
</details>

<details>
<summary>Когда CHECK на JSONB оправдан, а когда лучше вынести поле в колонку?</summary>

CHECK на JSONB оправдан для простых требований (“ключ существует”, “тип такой-то”) и когда поле остаётся “второстепенным”. Если поле становится горячим для запросов/отчётов/джойнов — лучше вынести в колонку: проще, быстрее, легче поддерживать.
</details>

##### Мини‑пример: как данные “расползаются”, если не контролировать тип

Представь ключ `price` в JSON. Сегодня он число, завтра строка, послезавтра null:

```sql
-- PostgreSQL: три разных "вида" price в JSONB
SELECT
  '{"price": 100}'::jsonb AS a,
  '{"price": "100"}'::jsonb AS b,
  '{"price": null}'::jsonb AS c;
```

Выглядит “почти одинаково”, но для запросов это три разных мира:

- `100` — число;
- `"100"` — строка;
- `null` — отсутствие значения.

**Почему это плохо:** запросы с кастом `(payload->>'price')::numeric` начнут падать или давать странные результаты на части данных.  
**Вывод:** если ключ важный, фиксируй тип (колонкой или строгой валидацией).

##### Проверь себя (к мини‑примеру “расползание типов”)

<details>
<summary>Почему `"100"` и `100` в JSON “похожи глазами”, но ломают запросы и индексацию?</summary>

Потому что это разные типы: строка и число. `->>` вернёт текст, касты к числу могут падать на части данных, сравнения становятся лексикографическими, а индексировать “как число” без нормализации невозможно.
</details>

<details>
<summary>Какой практический инвариант ты бы зафиксировал, чтобы ключ price не “расползался” по типам?</summary>

Либо вынести `price` в колонку numeric и считать JSON “дополнением”, либо добавить валидацию на входе/`CHECK`, что `payload` содержит `price` нужного типа (например, `jsonb_typeof(payload->'price') = 'number'`) и запрещать записи, которые нарушают контракт.
</details>

<details>
<summary>Почему “лечить” расползание типов только на чтении (через CASE/COALESCE/regex) — плохая стратегия?</summary>

Потому что ты превращаешь каждый запрос в сложную обработку, теряешь индексацию и получаешь постоянный риск ошибок/падений. Правильнее исправлять на границе записи: контракт/колонка/валидация.
</details>

#### 3) Операторы PostgreSQL: что возвращают и почему это важно

Коротко:

- `->` возвращает **json/jsonb** (объект/массив/значение как JSON);
- `->>` возвращает **текст** (готово для сравнения как строка);
- `#>` / `#>>` — путь (массив ключей) для вложенных структур;
- `@>` — “содержит” (часто индексируется GIN);
- `?` — ключ существует (часто индексируется GIN на jsonb).

Почему важно:

- если ты используешь `->>` и сравниваешь как текст, ты можешь получить неожиданный порядок/сравнение (например, числа сравниваются как строки);
- индексация зависит от оператора: `@>` дружит с GIN, а произвольное выражение — нет.

##### Остановка: “->> всегда строка” и типичные ошибки сравнения

`->>` возвращает **TEXT**. Поэтому выражение:

```sql
-- ПЛОХО: сравнение чисел как строк
-- '9' > '100' будет true, потому что строки сравниваются лексикографически
SELECT '9' > '100' AS string_compare;
```

Если ты вытаскиваешь число как текст, а потом сравниваешь — ты можешь получить неправильную логику.

Правильный подход:

- либо приводить к типу (`::numeric`) и индексировать это значение (generated/expr index),
- либо держать поле в колонке нужного типа.

##### Проверь себя (к подпункту “операторы и тип результата”)

<details>
<summary>В чём практическая разница между -> и ->> для последующих условий WHERE/ORDER BY?</summary>

`->` возвращает JSON/JSONB, а `->>` возвращает TEXT. `->>` удобно для сравнения как строки, но это ловушка для чисел/дат. Если дальше нужен числовой диапазон или сортировка как по числу — нужно приводить к типу или хранить в колонке.
</details>

<details>
<summary>Почему выражение payload->>'price' > '100' логически опасно, даже если “на тестовых данных вроде работает”?</summary>

Потому что это сравнение строк, а не чисел: `'9' > '100'` истинно лексикографически. На тестовых данных могло “случайно совпасть”, но на проде даст неверные выборки и отчёты.
</details>

<details>
<summary>Какой самый надёжный способ сделать “price &gt; 100” быстрым и типобезопасным, если price живёт в JSON?</summary>

Материализовать `price` в типизированную колонку (generated column или обычная), поставить B-tree индекс и фильтровать по колонке. Тогда и скорость, и логика предсказуемы.
</details>

#### 4) Индексы GIN: что ускоряют и где ломаются ожидания

GIN на JSONB помогает, когда запрос использует:

- `@>` (containment),
- `?` (existence),
- некоторые другие операторы (зависит от operator class).

Но GIN **не** поможет автоматически для:

- `WHERE (data->>'price')::numeric > 100` (нужен индекс по выражению или generated column),
- “сложных” выражений, которые не соответствуют operator class.

Практика: если у тебя есть “горячее” поле в JSON (например, `status`), часто лучше:

- вынести его в отдельную колонку,
- или сделать generated column + индекс,
- или индекс по выражению (PostgreSQL).

##### Сначала интуитивно: почему GIN помогает для `@>` и не помогает для “любого выражения”

GIN по JSONB — это не “индекс на всё”, а индекс, который хранит “кусочки” документа (ключи/значения) в формате “элемент → список строк”.

Оператор `@>` (“содержит”) задаёт запрос, который можно разложить на “найди строки, где встречается такой набор ключей/значений” → это подходит под идею GIN.

А выражение `(payload->>'price')::numeric > 100` — это уже не “содержит кусочек”, а “вычисли число и сравни по диапазону” → это естественно для B-tree, а не для GIN.

**Короткий вывод:** GIN ускоряет те операторы, под которые он спроектирован; для диапазонов/сортировок по конкретному полю чаще нужен B-tree по этому полю (пусть даже вычисляемому).

##### Проверь себя (к подпункту “GIN и ожидания”)

<details>
<summary>Почему GIN по JSONB хорошо подходит под @>, но плохо подходит под “извлеки поле и сравни по диапазону”?</summary>

`@>` можно разложить на “в документе присутствует такой набор ключей/значений”, что соответствует модели “элемент → строки”. Диапазон по одному вычисленному значению (price>100) требует упорядочивания/диапазонного поиска — это естественно для B-tree по значению, а не для GIN по элементам документа.
</details>

<details>
<summary>Какая проверка (на уровне вопроса) помогает понять, поможет ли GIN в конкретном запросе?</summary>

Посмотреть, какой оператор используется: если запрос выражен через containment/existence (`@>`, `?` и родственные) — шанс высокий. Если через вычисления/касты/диапазоны по извлечённому тексту — чаще нужен индекс по выражению или материализованная колонка.
</details>

<details>
<summary>Что такое “recheck” в контексте индексов по составным/элементным структурам, и почему он важен?</summary>

Индекс часто даёт “кандидатов”, а не гарантирует истинность условия во всех сложных случаях. После индекса СУБД перепроверяет условие на строках‑кандидатах. Это объясняет, почему индекс может давать ускорение, но не “чудо”, и почему селективность важна.
</details>

#### 4.1) jsonpath: как думать о нём и что он даёт

##### Сначала интуитивно

Операторы `->`, `->>`, `@>` удобны для простых случаев (“достать ключ”, “проверить содержит кусочек”).  
`jsonpath` нужен, когда тебе надо выразить более сложное условие “внутри документа”: фильтры по вложенным массивам, проверки нескольких условий, “есть ли где-то элемент массива, удовлетворяющий…”.

Проще думать так:

- простое → операторы (`->>`, `@>`)
- сложное → `jsonpath` (но осторожно с производительностью)

##### Пример → почему → вывод (PostgreSQL)

Допустим, в `payload` есть массив `items`, и мы хотим найти события, где **есть хотя бы один** item с `sku = 'A1'` и `qty >= 2`.

```sql
-- PostgreSQL (идея): jsonpath-проверка существования элемента по условию
-- Конкретные функции/операторы зависят от версии Postgres (jsonb_path_exists / @? и т.п.)
SELECT *
FROM events
WHERE jsonb_path_exists(
  payload,
  '$.items[*] ? (@.sku == "A1" && @.qty >= 2)'
);
```

**Почему это работает:** `jsonpath` проходит по элементам массива `items` и проверяет предикат для каждого.  
**Вывод:** jsonpath — мощнее, но его нужно использовать там, где простые операторы становятся слишком громоздкими или невозможными.

##### Проверь себя (к подпункту “зачем jsonpath”)

<details>
<summary>В чём разница “операторы” vs “jsonpath” на уровне задачи, а не синтаксиса?</summary>

Операторы удобны для простых условий (“ключ равен”, “документ содержит кусочек”). jsonpath нужен для сложных предикатов внутри вложенных структур (особенно массивов) — “есть ли элемент, удовлетворяющий нескольким условиям”.
</details>

<details>
<summary>Почему jsonpath повышает выразительность, но не гарантирует индексируемость?</summary>

Потому что выразительный предикат часто требует “обойти и проверить” документ, особенно массивы и сравнения. Индексы работают для определённых классов запросов; jsonpath может выйти за их рамки.
</details>

<details>
<summary>Когда jsonpath — хороший выбор, даже если запрос будет дорогим?</summary>

В редких/аналитических запросах, где важнее корректность и выразимость, чем миллисекундная latency, и где можно ограничить объём (по датам, партициям, типу события) или запускать батчами.
</details>

##### Остановка про индексы: “jsonpath ≠ автоматически быстро”

Очень важно: наличие `jsonpath` не означает, что запрос станет индексируемым.  
Часто стратегия такая же, как раньше:

- если условие часто используется — вынести нужное в колонку/generated column и индексировать B-tree;
- `jsonpath` использовать для редких/глубоких фильтров или в аналитических запросах.

##### Проверь себя (к подпункту “jsonpath ≠ быстро”)

<details>
<summary>Какой “золотой” вопрос ты задашь, чтобы решить: материализовать поле или оставить jsonpath?</summary>

“Это условие — горячее (в проде в каждом втором запросе) или редкое/аналитическое?” Если горячее — материализовать/индексировать. Если редкое — можно оставить jsonpath и принять цену.
</details>

<details>
<summary>Почему “поставлю GIN на payload и всё полетит” — опасная стратегия?</summary>

Потому что GIN ускоряет не все выражения, а конкретные операторы/паттерны. Сложные вычисления/касты/диапазоны по извлечённым значениям могут игнорировать GIN и превращаться в Seq Scan.
</details>

<details>
<summary>Какая минимальная практика диагностики помогает отличить “индекс не используется” от “индекс используется, но мало помогает”?</summary>

Смотреть `EXPLAIN` (хотя бы на уровне Scan типа) и оценивать селективность: Seq Scan/фильтрация большого объёма → индекс не помогает; Index Scan с большим recheck/много возвращённых кандидатов → индекс есть, но запрос мало селективен или паттерн неудачный.
</details>

##### Ещё 3 сценария: “простые операторы” vs “jsonpath”

**Сценарий A (простое): проверить, что ключ равен значению.**  
Лучше `@>`:

```sql
-- PostgreSQL: быстро и понятно
SELECT * FROM events WHERE payload @> '{"status":"paid"}';
```

**Почему:** это ровно containment, под который хорошо ложится GIN.

**Сценарий B (простое): достать вложенный ключ и сравнить как строку.**  
Лучше `->>` (но помни про тип):

```sql
SELECT * FROM events WHERE payload->>'source' = 'mobile';
```

Если это частый фильтр — вынести в колонку/generated column и индексировать.

**Сценарий C (сложное): в массиве есть элемент, удовлетворяющий предикату.**  
Здесь jsonpath часто выразительнее:

```sql
SELECT *
FROM events
WHERE jsonb_path_exists(payload, '$.items[*] ? (@.sku == "A1" && @.qty >= 2)');
```

**Почему:** простыми операторами это либо громоздко, либо сложно поддерживать.

##### Типичные ошибки jsonpath (и как они проявляются)

1) **Использовать jsonpath для “простых” запросов** → запрос становится сложнее, а скорость не растёт.  
Симптом: “почему так медленно, я же написал умное условие”.

2) **Думать, что GIN индекс по payload ускорит любой jsonpath** → часто не ускорит.  
Симптом: `EXPLAIN` показывает Seq Scan или большой recheck, время растёт с объёмом.

3) **Писать условия, которые заставляют читать весь документ** (особенно большие payload) → нагрузка на CPU/I/O.  
Симптом: рост latency именно на чтении/фильтрации.

##### Мини‑пример: как jsonpath легко превращается в тяжёлый Seq Scan (симптом → причина → что делать)

**Симптом:** запрос “логически простой” (найти события с нужным условием внутри JSON), но на больших объёмах внезапно становится линейным по таблице.

```sql
-- PostgreSQL (идея): хотим найти события, где в payload есть item с qty >= 2
EXPLAIN
SELECT *
FROM events
WHERE jsonb_path_exists(payload, '$.items[*] ? (@.qty >= 2)');
```

Ты ожидаешь “у меня же есть индекс по payload”, но видишь что-то вроде **Seq Scan** (или план с огромным количеством строк на фильтрации).

**Почему так (типовая причина):**

- GIN по `payload` помогает для определённых операторов/классов (например, containment `@>` / existence `?`),  
  но **не обязан** уметь “ускорять произвольный предикат jsonpath”.
- jsonpath часто означает: “прочитай документ и проверь предикат”, особенно если условие затрагивает массивы/сравнения по диапазону.

**Что делать (практические варианты, от простого к надёжному):**

1) **Если условие можно выразить через `@>` (containment) — сделай так.**  
Это самый частый способ “попасть” в GIN:

```sql
-- Вместо jsonpath: если условие на равенство и структура известна
SELECT * FROM events WHERE payload @> '{"status":"paid"}';
```

2) **Если фильтр частый и важный — материализуй поле.**  
Вынеси нужный ключ в колонку/generated column и индексируй B-tree (самый надёжный прод‑путь):

```sql
-- Пример (рабочий и типовой): часто фильтруемся по source
ALTER TABLE events
  ADD COLUMN source TEXT GENERATED ALWAYS AS (payload->>'source') STORED;
CREATE INDEX events_source_idx ON events (source);

-- Теперь фильтр становится обычным и индексируемым:
SELECT * FROM events WHERE source = 'mobile';
```

3) **Если фильтр редкий/аналитический — оставь jsonpath, но будь честен с ценой.**  
То есть не пытайся “докрутить индекс магией”, а планируй нагрузку (батчи, ограничения по времени, выборки по партициям, предфильтр по другим колонкам).

**Короткий вывод:** `jsonpath` — это инструмент выразительности, а не гарантированная индексируемость. Если условие становится “горячим”, оно должно стать частью схемы (колонка/индекс), иначе ты обречён на чтение больших объёмов JSON.

##### Проверь себя (к мини‑примеру “Seq Scan на jsonpath”)

<details>
<summary>Почему запрос “по массиву внутри JSON” особенно часто приводит к Seq Scan?</summary>

Потому что условие требует проверить элементы массива, что обычно означает “прочитать документ и пройтись по элементам”. Это плохо раскладывается на простое “содержит ключ/значение” и не обязано быть индексируемым GIN.
</details>

<details>
<summary>Назови два способа “починить” горячий jsonpath‑фильтр, не переписывая всю модель данных.</summary>

1) Вынести конкретный ключ (например, `source/status/price`) в generated column и поставить B-tree индекс. 2) Переписать условие в `@>`/`?`, если это возможно по смыслу (равенство/наличие), чтобы использовать GIN.
</details>

<details>
<summary>Почему “оставить как есть и надеяться на железо” обычно проигрышная стратегия?</summary>

Потому что Seq Scan растёт линейно с данными и начинает съедать CPU/I/O, ухудшая не только один запрос, но и всю систему (кеш, конкуренция за диск, общий latency). Лучше сделать предсказуемый индексируемый путь.
</details>

##### Практическое решение “частый фильтр → generated column”

Если у тебя есть jsonpath/выражение, которое встречается постоянно (например “price > 100”), лучше материализовать поле:

```sql
-- PostgreSQL: вынести price в generated column и индексировать
ALTER TABLE events
  ADD COLUMN price numeric GENERATED ALWAYS AS ((payload->>'price')::numeric) STORED;
CREATE INDEX events_price_idx ON events (price);

-- Тогда запрос становится обычным и быстрым:
SELECT * FROM events WHERE price > 100;
```

**Почему это помогает:** индекс B-tree отлично работает по диапазону (`>`, `<`, `BETWEEN`), а jsonpath/JSON‑выражения не обязаны быть индексируемыми.

#### 5) Другие СУБД: общий обзор

- **MySQL**: есть тип `JSON`, поддержка JSON‑функций, индексация часто делается через **generated columns** (виртуальные/хранимые) и обычные индексы по ним.  
- **SQLite**: JSON чаще как TEXT + JSON1 функции; индексация — через computed/generated (в новых версиях) или отдельные колонки.  
- **SQL Server**: JSON хранится как NVARCHAR, есть функции `JSON_VALUE/JSON_QUERY`, индексация через computed columns.  
- **Oracle**: JSON имеет богатую поддержку, включая индексацию и SQL/JSON; но детали зависят от версии.

Вывод: подход “вынести ключи в вычисляемые колонки” — универсальный, потому что обычные B-tree индексы предсказуемы.

##### Проверь себя (к подпункту “другие СУБД”)

<details>
<summary>Почему во многих СУБД JSON “в итоге” индексируют через computed/generated columns?</summary>

Потому что “индексировать произвольный предикат внутри документа” сложно и сильно зависит от реализации. А вычисляемая колонка превращает частый ключ в обычное значение, для которого B-tree индексы работают предсказуемо и эффективно.
</details>

<details>
<summary>Что будет главным критерием “выносить в колонку vs оставить в JSON” в любой СУБД?</summary>

Частота и тип запросов: если ключ участвует в `WHERE/ORDER/GROUP/JOIN` регулярно — он становится частью схемы и должен быть колонкой/индексом/ограничением. Если ключ редкий и гибкий — можно оставить в JSON.
</details>

<details>
<summary>Почему хранение JSON в NVARCHAR (как в некоторых системах) не делает его “плохим”, но меняет подход к оптимизации?</summary>

Потому что функциональность смещается в сторону функций извлечения и computed columns. Сама идея JSON остаётся, но эффективный путь — материализовать нужные ключи и индексировать их, а не пытаться искать “внутри текста” каждый раз.
</details>

### Пошагово

#### Шаг 1. Признай: JSON — это “компромисс”

Задай два вопроса:

- какие поля будут в запросах **часто**?
- какие поля “редкие/произвольные/расширяемые”?

Частые — лучше в колонках. Редкие — можно в JSONB.

##### Проверь себя (к §224 шагу 1: “JSON — компромисс”)

<details>
<summary>Почему “признать компромисс” — это практическое действие, а не философия?</summary>

Потому что это заставляет заранее выбрать границу: какие поля будут индексироваться/ограничиваться/участвовать в JOIN и поэтому должны быть колонками, а какие можно оставить гибкими. Без этого граница появляется “сама” в виде медленных запросов и хаоса типов.
</details>

<details>
<summary>Какие два вопроса из этого шага самые важные для производительности на росте данных?</summary>

“Какие поля будут в частых запросах?” и “по каким полям будут фильтровать/сортировать/группировать?”. Эти поля должны стать колонками/генерируемыми колонками с индексами.
</details>

<details>
<summary>Почему “редкие поля” лучше оставить в JSONB, даже если их можно сделать колонками?</summary>

Потому что это уменьшает стоимость миграций и снижает сложность схемы, если поля действительно редко используются и не требуют строгих ограничений. Но границу нужно держать осознанно.
</details>

#### Шаг 2. Сделай схему “колонки + JSONB”

```sql
-- PostgreSQL: типовой паттерн
CREATE TABLE events (
  id BIGSERIAL PRIMARY KEY,
  event_type TEXT NOT NULL,
  created_at timestamptz NOT NULL DEFAULT now(),
  payload jsonb NOT NULL
);
```

##### Проверь себя (к §224 шагу 2: “колонки + JSONB”)

<details>
<summary>Почему created_at/event_type обычно должны быть колонками, а не ключами в JSON?</summary>

Потому что это системообразующие атрибуты: по ним часто фильтруют, сортируют, агрегируют и партиционируют. Колонки дают типизацию, индексы и предсказуемые планы.
</details>

<details>
<summary>Какой риск появляется, если payload сделать nullable, а “схему” хранить только в приложении?</summary>

Данные начинают “плыть”: часть строк без payload, часть с разными формами. Запросы и индексы становятся сложнее, появляется больше ветвлений и ошибок. `payload NOT NULL` фиксирует, что документ существует как часть модели.
</details>

<details>
<summary>Почему “типовой паттерн” важен методически?</summary>

Потому что он даёт повторяемую структуру: ты всегда знаешь, где искать “горячие” поля (колонки) и где “гибкую часть” (payload). Это упрощает развитие схемы и диагностику.
</details>

#### Шаг 3. Индексируй то, что реально фильтруется

Если фильтруешь по “подструктуре”:

```sql
-- PostgreSQL: GIN индекс для containment-запросов
CREATE INDEX events_payload_gin ON events USING gin (payload);
```

Если фильтруешь по конкретному ключу как строке/числу — часто лучше индекс по выражению:

```sql
-- PostgreSQL: индекс по выражению
CREATE INDEX events_payload_status_idx ON events ((payload->>'status'));
```

Или generated column (если удобно в модели данных):

```sql
-- PostgreSQL: generated + индекс (пример)
ALTER TABLE events
  ADD COLUMN status TEXT GENERATED ALWAYS AS (payload->>'status') STORED;
CREATE INDEX events_status_idx ON events (status);
```

##### Проверь себя (к §224 шагу 3: “индексируй то, что фильтруется”)

<details>
<summary>Почему GIN по payload и индекс по выражению/генерируемой колонке решают разные классы запросов?</summary>

GIN хорош для containment/existence (`@>`, `?`), где запрос можно разложить на “элементы документа”. Индекс по выражению/колонке хорош для равенства/диапазонов по конкретному ключу (status, price), где нужен B-tree по значению.
</details>

<details>
<summary>Как выбрать между “индекс по выражению” и “generated column + индекс”?</summary>

Индекс по выражению быстрее добавить без изменения модели данных, но generated column делает значение “первого класса”: его легче использовать в запросах, ограничениях, отладке и переносить между СУБД. Выбор — баланс удобства и строгости модели.
</details>

<details>
<summary>Почему “индексировать всё подряд” по JSON — плохая стратегия?</summary>

Потому что индексы стоят места и времени на обновление, а пользу дают только под реальные запросы. Индексация должна следовать за паттернами доступа, иначе получишь тяжёлую запись и мало выигрыша.
</details>

##### Дополнение: как это обычно выглядит в MySQL (похожая идея)

В MySQL часто делают так же, только через `generated column`, потому что JSON хранится как JSON, а индексировать “внутрь” удобнее через вычисляемое поле:

```sql
-- MySQL (концептуально): извлечь status в STORED generated column и индексировать
-- Синтаксис JSON_EXTRACT/-> зависит от версии; смысл в том, что статус становится обычной колонкой.
-- ALTER TABLE events ADD COLUMN status VARCHAR(32) GENERATED ALWAYS AS (JSON_UNQUOTE(JSON_EXTRACT(payload, '$.status'))) STORED;
-- CREATE INDEX events_status_idx ON events(status);
```

Смысл тот же: “горячее поле” становится индексируемым B-tree.

### Простыми словами

JSON — это “мешок” с данными разной формы. Удобно быстро начать и хранить “как пришло”.  
Но мешок плохо проверять и плохо индексировать, если ты хочешь делать быстрые запросы.  
Поэтому часто делают гибрид: важное — по полочкам (колонки), редкое — в мешок (JSONB).

### Картинка в голове

Представь склад:

- колонки — это стеллажи с подписанными ячейками (“товар A”, “товар B”), где легко найти и проверить;
- JSONB — это коробка “разное” с описью внутри: иногда удобно, но если ты каждый день ищешь один и тот же предмет в коробке — лучше положить его на стеллаж.

### Как запомнить

- **Часто фильтруемое → колонка или generated column + индекс.**
- **JSONB (Postgres) почти всегда лучше JSON, если есть запросы внутрь.**
- **GIN помогает для containment/existence, но не для любых выражений.**

### Примеры

#### Пример 1. Операторы `->` и `->>`

```sql
-- PostgreSQL
SELECT
  payload->'user' AS user_json,
  payload->>'user' AS user_text
FROM events
LIMIT 10;
```

#### Пример 2. Containment `@>` и индекс GIN

```sql
-- Найти события, где в payload есть {"status":"paid"}
SELECT *
FROM events
WHERE payload @> '{"status":"paid"}';
```

GIN индекс по `payload` обычно помогает именно здесь.

#### Пример 3. “Цена больше 100” внутри JSON

```sql
-- PostgreSQL: медленно без индекса по выражению/генерируемой колонки
SELECT *
FROM events
WHERE (payload->>'price')::numeric > 100;
```

Для ускорения обычно делают:

- generated column `price numeric` + индекс,
- или отдельную колонку `price`, если это ключевой атрибут.

### Практика / реальные сценарии

#### Сценарий: “мы храним профиль пользователя в JSON, но теперь нужен быстрый поиск”

Проблема: сначала JSON удобен (“профиль гибкий”), но потом появляются запросы:

- “все пользователи, у которых age > 30”
- “все пользователи из города X”
- “сортировать по last_login”

Решение:

- вынести нужные поля в колонки;
- оставить “нечастые” поля в JSONB;
- добавить индексы по колонкам.

Это нормальная эволюция: JSON помогает стартовать, реляционная схема помогает масштабироваться.

#### Сценарий: “в JSON нет ограничений, прилетели разные типы”

Сегодня `price` — число, завтра — строка `"100"`, послезавтра — `null`.  
Запросы начинают падать на кастах, отчёты расходятся.

Практика:

- ограничивать форму JSON на входе (валидация в приложении);
- для важных ключей — держать тип в колонке;
- добавлять `CHECK` на минимальные ожидания (например, ключ существует и имеет нужный тип) — если это реально нужно.

### Типичные ошибки

- **Использовать JSON как замену нормальной схемы**, когда поля стали “первого класса”.
- **Сравнивать числа как строки** (`payload->>'price' > '100'`) — даёт неправильный порядок.
- **Ставить GIN индекс и ждать, что “всё ускорится”.** Индекс ускоряет только операторы, которые умеют его использовать.
- **Хранить разные типы под одним ключом** без контроля — потом не проиндексировать и не запросить нормально.

### Что будет, если…

- …**оставить часто используемые поля внутри JSON без индексов**: запросы деградируют “по мере роста”, а потом резко становятся неприемлемыми.
- …**делать миграции формы JSON без контроля**: появятся “старые” и “новые” документы, запросы превратятся в набор `CASE` и костылей.

### Проверь себя

<details>
<summary>Почему в PostgreSQL чаще выбирают JSONB, а не JSON, если нужны запросы по полям?</summary>

Потому что JSONB хранится в разобранной бинарной форме, что позволяет эффективнее выполнять операции по ключам/значениям и использовать GIN‑индексы. JSON как текст хуже подходит для поиска и индексации.
</details>

<details>
<summary>В каких случаях “колонки + JSONB” лучше, чем “всё в JSONB”?</summary>

Когда есть набор “горячих” атрибутов, по которым часто фильтруют/сортируют/джойнят/строят отчёты. Эти атрибуты лучше вынести в колонки (типизация, ограничения, индексы), а редкие/гибкие — оставить в JSONB для расширяемости.
</details>

<details>
<summary>Почему GIN индекс по JSONB не ускоряет автоматически выражение (payload->>'price')::numeric > 100?</summary>

Потому что GIN индекс строится под конкретные операторы/классы (containment/existence и т.п.). Произвольное выражение с кастом к numeric — это другая операция; для неё нужен индекс по выражению или вычисляемая колонка, чтобы значение было индексируемо B-tree.
</details>

### Запомните

- JSONB — инструмент для гибкости, но **не бесплатный**: ограничения и быстрые запросы требуют осознанного проектирования.  
- Индексируй то, что фильтруется; часто это значит “вынести в колонку”.  
- Сравнения и типы внутри JSON нужно контролировать, иначе данные “расползутся”.
