## 271. NUMA и многопроцессорность в СУБД

### Цель раздела

Понять, **как NUMA‑архитектура и многопроцессорность влияют на производительность СУБД**,  
почему “больше ядер” не всегда равно “быстрее”, что такое локальная и удалённая память, pinning и распределение буферных пулов,  
и как практически подходить к настройке СУБД на NUMA‑серверах (PostgreSQL, MySQL, Oracle, SQL Server, NoSQL‑кластеры).

### В этом разделе главное

1. NUMA означает, что **время доступа к памяти зависит от того, на каком узле она физически лежит**.
2. Слепое масштабирование числа потоков может привести к тому, что запросы начнут постоянно ходить в **удалённую память**, теряя производительность.
3. Привязка процессов/потоков к NUMA‑узлам и “локализация” данных (буферных пулов, рабочих структур) **сильно улучшает предсказуемость**.
4. Разные СУБД по‑разному учитывают NUMA (PostgreSQL — в основном через настройки ОС и pinned‑процессы, SQL Server/Oracle — более NUMA‑aware).
5. Без анализа `numa_maps`, perf и метрик по локальности памяти любые “оптимизации” могут быть случайными.

### Термины

- **NUMA node** — набор CPU и памяти, где доступ к “своей” памяти быстрее, чем к памяти другого узла.
- **Local vs remote memory** — память, физически подключённая к тому же NUMA‑узлу / к другому узлу.
- **Affinity / pinning** — привязка процессов/потоков к ядрам/узлам.
- **Interconnect** — шина/сеть, по которой узлы взаимодействуют (QPI/UPI, Infinity Fabric и т.п.).

### Теория и правила

#### 1) Модель памяти в NUMA

В UMA (устаревшая простая модель):

- все ядра видят **одинаковое время доступа к памяти**;
- нет понятия “локальная/удалённая” память.

В NUMA:

- у каждого узла есть **свой блок памяти**;
- доступ к “своей” памяти быстрый, к чужой — медленнее;
- при большом числе потоков и неконтролируемом размещении данных можно получить:
  - **массу кросс‑узловых обращений**;
  - рост задержек и падение пропускной способности.

#### 2) СУБД и NUMA

Как СУБД живут на NUMA:

- **PostgreSQL**:
  - модель “один процесс на подключение”;
  - буферный пул общий;
  - NUMA‑осведомлённость в основном через ОС:
    - запускаем процессы, привязанные к узлам;
    - следим за тем, как распределяется память (transparent huge pages, `numactl`, `zone_reclaim_mode`).
- **MySQL (InnoDB)**:
  - многопоточный сервер;
  - буферный пул может быть разбит на несколько инстансов (buffer pool instances), что помогает уменьшить contention и иногда улучшает NUMA‑локальность.
- **SQL Server / Oracle**:
  - имеют более сложную NUMA‑осведомлённость:
    - планировщик задач, aware о NUMA;
    - распределение буферных пулов/worker‑threads по узлам;
    - иногда даже отдельные thread‑pools/allocators на узел.
- **NoSQL‑кластеры (Cassandra, MongoDB, Elasticsearch)**:
  - обычно используют **несколько JVM/процессов** или тюнят heap/allocators под NUMA, иногда “cthulhu‑подход” — по одному процессу на узел в рамках одной машины.

#### 3) Правила здравого смысла

- Если у тебя:
  - одна большая база;
  - один инстанс СУБД;
  - NUMA‑сервер на 2–4 узла;
  - **убедись**, что:
    - СУБД **не прыгает** по всем узлам (аффинити по CPU);
    - память выделяется **по возможности локально** (правильные настройки ОС);
    - нет странных режимов типа `zone_reclaim_mode=1`, которые заставляют ядро агрессивно вычищать локальную память.
- Если у тебя есть **несколько инстансов СУБД/шардов** на одной машине:
  - логично **привязать каждый инстанс к своему NUMA‑узлу**, чтобы минимизировать кросс‑узловой трафик;
  - выделить память/файлы так, чтобы I/O и память тоже были “локальны”.

### Пошагово

Пример базового аудита NUMA для PostgreSQL на Linux:

1. **Понять топологию NUMA:**

```bash
numactl --hardware
```

2. **Посмотреть, где живёт процесс Postgres:**

```bash
ps -o pid,psr,cmd -C postgres
numastat -p <pid_postmaster>
```

3. **Проверить `numa_maps`:**

```bash
grep -i numa_maps /proc/<pid_postmaster>/numa_maps | head
```

4. **Оценить, нет ли сильного перекоса в remote memory:**

- по `numastat` и `numa_maps` понять, какая доля памяти реально локальная;
- если remote memory много, рассмотреть:
  - запуск Postgres через `numactl --cpunodebind=0 --membind=0` (пример);
  - либо более тонкие настройки.

5. **Проверить настройки ядра:**

- `cat /proc/sys/vm/zone_reclaim_mode`
- `cat /sys/kernel/mm/transparent_hugepage/enabled`

6. **Наблюдать производительность**:

- до и после изменений — TPS, latency, CPU utilization по узлам.

Аналогичные шаги можно делать для MySQL, SQL Server (на Windows — через NUMA‑aware настройки, affinity mask и т.п.), Oracle.

### Простыми словами

NUMA — это как **квартира с несколькими комнатами**, в каждой комнате — свой шкаф с книгами и свой стол.

- Если ты сидишь за столом в одной комнате и читаешь книги **из шкафа этой комнаты**, всё хорошо.
- Если ты каждые 10 секунд бегаешь в соседнюю комнату за книгой — устаёшь и тратишь кучу времени.

СУБД на NUMA — это много людей (потоков), которые:

- либо аккуратно сидят в “своих комнатах” и читают “свои шкафы” (локальная память),
- либо хаотично бегают по всей квартире, забивая коридоры (interconnect) и создавая пробки.

### Картинка в голове

Представь, что NUMA‑узлы — это **острова**, соединённые мостами. На каждом острове:

- живёт группа людей (потоки);
- есть склад с ресурсами (локальная память).

Если все жители своего острова в основном берут ресурсы у себя — **мосты свободны**. Если все бегут по мостам туда‑сюда — возникает пробка, и добираться до ресурсов становится всё дольше.

### Как запомнить

- **Local good, remote bad** (с разумными оговорками).
- Если видишь сервер с 2+ NUMA‑узлами — **подумай, как СУБД использует их**, а не только “сколько там ядер”.

### Примеры

#### PostgreSQL

- Постгрес не имеет глубокой NUMA‑осведомлённости, но:
  - ты можешь запускать его с `numactl --interleave=all` или `--membind=X` в зависимости от сценария;
  - можешь использовать cgroups/affinity, чтобы рабочие процессы не прыгали по всем ядрам.

#### SQL Server

- SQL Server умеет:
  - создавать **soft NUMA nodes**;
  - распределять worker threads по NUMA‑узлам;
  - настраивать affinity mask;
  - при этом планировщик запросов может учитывать локальность.

#### Oracle

- Oracle RAC/Exadata — примеры систем, где NUMA‑ и topology‑осведомлённость встроена в глубину стека: процессы, буферы, I/O‑пути распределяются с учётом архитектуры.

### Практика / реальные сценарии

- **Сервер с 2×CPU, 2 NUMA‑узлами, PostgreSQL OLTP:**
  - без тюнинга часть воркеров уходит в “чужую” память;
  - latency растёт, профилировка показывает время в функциях управления памятью и interconnect.
- **Shard’ы MySQL/InnoDB на NUMA:**
  - по одному инстансу на NUMA‑узел;
  - pinning процессов к узлам;
  - локальные файлы данных на локальных контроллерах;
  - TPS растёт по мере добавления шардов предсказуемее.

### Типичные ошибки

- Игнорировать NUMA и просто “выкручивать” количество воркеров/соединений.
- Запускать **много разрозненных сервисов** на одной NUMA‑машине без учёта локальности.
- Включать агрессивные режимы `zone_reclaim_mode`, которые приводят к непредсказуемым паузам.

### Что будет, если…

- **Игнорировать NUMA:**
  - рост latency при большей нагрузке, хотя “CPU и диск вроде не загружены”;
  - непредсказуемые хвосты задержек.
- **Слишком жёстко “прибить” всё к одному узлу:**
  - недоиспользование ресурсов других узлов;
  - упирание в локальный лимит памяти/CPU раньше времени.

### Проверь себя

1. Почему на NUMA‑сервере увеличение числа соединений PostgreSQL может ухудшить latency даже при свободном диске и сети?

<details><summary>Ответ</summary>
Потому что больше процессов/соединений → больше миграций по ядрам → выше шанс, что рабочие процессы начнут активно использовать **удалённую память**. Это увеличивает задержки доступа к памяти и создаёт дополнительную нагрузку на interconnect, в результате средняя latency и хвосты растут, хотя диск/сеть могут быть далеко не пределом.
</details>

2. В каком случае разумно запускать **несколько инстансов СУБД** на одном NUMA‑сервере?

<details><summary>Ответ</summary>
Когда ты можешь **явно разделить нагрузку и данные** по NUMA‑узлам (например, разные шарды/базы) и привязать каждый инстанс к своему узлу (CPU+память+I/O). Тогда каждый инстанс в основном работает с локальными ресурсами, уменьшая кросс‑узловые обращения и повышая предсказуемость.
</details>

3. Почему “просто увеличить shared_buffers” или размер буферного пула без учёта NUMA может ухудшить ситуацию?

<details><summary>Ответ</summary>
Большой буферный пул сам по себе не гарантирует локальность: на NUMA‑системе память под него может быть распределена по всем узлам, а процессы — мигрировать между ядрами. Тогда большая часть обращений к буферам окажется **к удалённой памяти**, и каждый доступ будет дороже. В итоге ты получишь объёмный кеш, который плохо попадает в локальную память, и рост latency при нагрузке вместо ожидаемого ускорения.
</details>

### Запомните

- NUMA — это не “мелкая деталь”, а **часть архитектуры сервера**, влияющая на СУБД так же сильно, как выбор диска или сети.
- Любой серьёзный тюнинг на сервере с 2+ NUMA‑узлами должен учитывать **локальность памяти и потоков**.
