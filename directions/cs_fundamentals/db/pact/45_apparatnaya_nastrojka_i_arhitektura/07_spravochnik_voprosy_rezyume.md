## Справочник по части (ключевые пункты)

### Железо и архитектура

| Тема                           | Ключевые мысли                                                                                           |
| ------------------------------ | -------------------------------------------------------------------------------------------------------- |
| Cache‑friendly алгоритмы      | Идём по памяти последовательно, минимизируем случайные прыжки, выбираем layout под паттерн доступа.     |
| Row vs column layout          | Row — для OLTP и частых мелких апдейтов; column — для аналитики и сканирования по нескольким колонкам. |
| Cache line                    | Подгружается блоком (обычно 64 байта); false sharing — частая проблема при многопоточности.            |
| NUMA                          | Локальная память быстрее удалённой; pinning и локализация данных улучшают предсказуемость.             |
| Contention и блокировки       | Spinlock — для коротких секций, mutex — для длинных; hot spots нужно сегментировать или устранять.     |
| NVMe/SSD vs HDD               | NVMe — для журналов и latency‑чувствительных данных; HDD — для архивов и холодных данных.             |
| Durability                    | Настройки журналов и `fsync` определяют, какие данные переживут сбой; ослабление — всегда риск.       |
| RDMA и in‑memory              | Для экстремальных требований по latency; требуют сложной архитектуры и строгой дисциплины.            |
| Практический тюнинг           | Стековый взгляд, baseline, профилировка, поэтапные эксперименты.                                        |

### СУБД и их поведение

| СУБД         | Особенности по теме части                                                                                                           |
| ------------ | ---------------------------------------------------------------------------------------------------------------------------------- |
| PostgreSQL   | Rowstore, WAL, `synchronous_commit`, ограниченная NUMA‑осведомлённость, важны настройки ОС и дисков, богатый набор wait‑events.   |
| MySQL InnoDB | Rowstore, `innodb_flush_log_at_trx_commit`, `sync_binlog`, latch’и и mutex’ы InnoDB, adaptive hash index, буферный пул.           |
| SQLite       | Embedded, сильное влияние `PRAGMA synchronous` и режима журнала (`WAL` vs `DELETE`), часто работает на потребительском SSD.       |
| Oracle       | Глубокая интеграция с NUMA/кластерными решениями, redo log, развитые механизмы latch’ей и wait‑events.                            |
| SQL Server   | Rowstore + columnstore, In‑Memory OLTP (Hekaton), AlwaysOn, NUMA‑aware планировщик, гибкие настройки журналов и хранения.        |
| NoSQL        | Часто LSM‑движки (Cassandra, MongoDB/WiredTiger), репликация и durability на уровне кластера, особенности compaction/latency.     |

## Частые сценарии

### Сценарий 1. OLTP на PostgreSQL с ростом нагрузки

- **Проблема:** при росте числа подключений TPS почти не растёт, latency скачет.
- **Шаги:**
  - проверить планы запросов и индексы (исключить логические проблемы);
  - посмотреть wait‑events и профилировку CPU (contention vs I/O);
  - проверить NUMA‑настройки, pinning процессов;
  - рассмотреть перенос WAL на NVMe.

### Сценарий 2. Отчёты на SQL Server с columnstore

- **Проблема:** долгие аналитические запросы, высокий I/O и CPU.
- **Шаги:**
  - проанализировать, какие колонки реально используются;
  - убедиться, что columnstore‑индексы спроектированы под типичные запросы;
  - проверить storage layout (SSD vs HDD), compression, batch mode;
  - измерить профилировкой, не упирается ли всё в диск или в NUMA/contention.

### Сценарий 3. Cassandra кластер с массовыми DELETE

- **Проблема:** внезапный рост нагрузки на диск и latency после массовых удалений.
- **Шаги:**
  - вспомнить про tombstone‑ы и compaction (Часть XLIV);
  - проверить настройки compaction‑стратегии;
  - проанализировать I/O‑паттерны и расположение данных (SSD vs HDD),
  - скорректировать TTL, GC grace period, расписание heavy compaction.

## Вопросы для самопроверки по части

1. Как понять, что bottleneck находится **в кеше/памяти**, а не в диске или сети?

<details><summary>Ответ</summary>
Смотри:
- профилировку CPU — много времени в функциях работы с памятью/структурами данных, мало в I/O‑функциях;
- метрики I/O — низкая загрузка дисков при высоком CPU;
- поведение при переносе data/WAL на более быстрые диски — если почти не помогает, узкое место выше по стеку.  
Дополнительно можно использовать perf/eBPF для анализа cache miss’ов и NUMA‑локальности.
</details>

2. Почему иногда `Seq Scan` на PostgreSQL оказывается быстрее `Index Scan`, даже при наличии индекса?

<details><summary>Ответ</summary>
Потому что при высокой доле выборки `Seq Scan` читает страницы **последовательно**, хорошо используя кеш и предзагрузку, тогда как `Index Scan` приводит к множеству случайных чтений по индексам и data‑страницам. В итоге профиль latency и CPU может быть хуже у index‑варианта, несмотря на “теоретическую оптимальность”.
</details>

3. В каких случаях разумно использовать HDD для части данных, а не переводить всё на NVMe?

<details><summary>Ответ</summary>
Для **архивов, бэкапов, исторических и холодных данных**, к которым обращаются редко и без строгих SLA по latency. В таких случаях цена за гигабайт важнее скорости, а горячие данные (журналы, активные таблицы) можно держать на NVMe/SSD. Важно убедиться, что запросы, требующие низкой latency, не попадают на HDD‑разделы.
</details>

4. Чем железо‑ориентированный тюнинг отличается от “просто ускорения запросов с помощью индексов”?

<details><summary>Ответ</summary>
Индексы работают **на уровне логики запросов и структуры данных**, а железо‑ориентированный тюнинг учитывает:
- кеши CPU и layout;
- NUMA и локальность памяти;
- contention на внутренних блокировках;
- типы хранилищ и настройки durability;
- сетевые и in‑memory‑аспекты.  
Это другой уровень абстракции: индексы можно настроить идеально, но при неправильной конфигурации железа и ОС всё равно упираться в cache miss’ы, NUMA и блокировки.
</details>

5. Почему опасно переносить настройки, найденные в бенчмарке, в продакшн без адаптации?

<details><summary>Ответ</summary>
Потому что бенчмарки почти всегда имеют **упрощённый и более однородный профиль нагрузки**, чем реальный прод: другие типы запросов, размеры данных, паттерны пиков. Настройки, которые в бенчмарке уменьшили latency, в реальной системе могут:
- усилить contention на неожиданных структурах;
- привести к нехватке памяти/кеша под другие задачи;
- сместить баланс между durability и скоростью.  
Поэтому любые “успешные” настройки должны проходить цикл: гипотеза → ограниченный эксперимент в окружении, максимально похожем на прод → оценка хвостов latency и поведения под реальной нагрузкой.
</details>

6. Почему важно привлекать к обсуждению железо‑ориентированного тюнинга и разработчиков, и админов, и SRE?

<details><summary>Ответ</summary>
Потому что решения на уровне железа и СУБД **затрагивают интересы всех сторон**:
- разработчики определяют схемы данных, запросы и допустимые компромиссы (например, можно ли потерять часть логов);
- админы/инфра‑инженеры отвечают за конфигурацию ОС, дисков, сети, NUMA, резервирование;
- SRE/DBA следят за SLO, латентностью, стабильностью.  
Если тюнинг делает одна группа в изоляции (например, только DBA выключает `fsync` ради TPS), легко нарушить бизнес‑требования, ухудшить SLO или усложнить сопровождение. Совместное обсуждение помогает выбрать настройки, которые **и быстрые, и безопасные, и поддерживаемые**.
</details>

## Типичные ошибки по части

- **Игнорировать NUMA**, считая, что “ОС всё сама сделает”.
- **Отключать durability ради скорости** без формального анализа рисков и согласования с бизнесом.
- **Не использовать профилировку**, опираясь только на “ощущения” и отдельные метрики.
- Считать, что **NVMe автоматически решает все проблемы** — без учёта журналов, compaction, contention и планов запросов.
- Пытаться внедрять **in‑memory/RDMA‑решения** там, где bottleneck вообще не в железе, а в архитектуре приложения.

## Резюме части XLV

В этой части мы посмотрели на СУБД **с точки зрения железа**: от кешей CPU и layout данных до NUMA, contention, типов хранилищ, настроек durability, RDMA и in‑memory архитектур.  
Главная идея: **производительность и надёжность — это всегда результат взаимодействия кода, СУБД, ОС и железа**, а не только “быстрого диска” или “хорошего индекса”.  
Освоив материал этой части, ты можешь **осознанно разговаривать и с разработчиками, и с системными администраторами, и с инфраструктурной командой**, объясняя, почему именно такие настройки железа и СУБД нужны под конкретную нагрузку.  
В следующих продвинутых частях (про колоночные движки, большие данные, формальную теорию) этот взгляд “снизу вверх” поможет понимать, **почему те или иные архитектурные решения вообще стали возможны** и какие компромиссы за ними стоят.
