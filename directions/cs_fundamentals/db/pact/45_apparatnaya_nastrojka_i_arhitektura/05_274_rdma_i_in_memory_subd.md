## 274. RDMA и in‑memory СУБД

### Цель раздела

Разобраться, **как RDMA и in‑memory СУБД меняют традиционную картину “диск → память → CPU”**,  
какие преимущества и ограничения дают “базы в памяти” и “память по сети”,  
и когда такие архитектуры уместны по сравнению с классическими PostgreSQL/MySQL/Oracle/SQL Server и NoSQL‑движками.

### В этом разделе главное

1. RDMA позволяет **читать/писать память удалённого узла почти так же, как локальную**, обходя часть софтверных слоёв.
2. In‑memory СУБД убирают традиционный bottleneck диска, но взамен ограничивают **размером памяти и требованиями к отказоустойчивости**.
3. Комбинация RDMA + in‑memory позволяет строить **кластерные системы с очень низкими задержками**, но резко повышает требования к качеству кода и инфраструктуры.
4. Даже in‑memory базы всё равно нуждаются в **снапшотах, логах и механизмах восстановления**.
5. Такие решения оправданы для **критически latency‑чувствительных** систем (биржи, трейдинг, телеком, высокочастотный OLTP), но зачастую избыточны для обычных веб‑приложений.

### Термины

- **RDMA (Remote Direct Memory Access)** — технология, позволяющая узлу напрямую читать/писать память другого узла по сети.
- **In‑memory DBMS** — СУБД, в которой рабочие данные живут в оперативной памяти.
- **Snapshot** — моментальный снимок состояния базы (в память/на диск/в хранилище).
- **Log / journal** — журнал изменений, который используется для восстановления после сбоя.

### Теория и правила

#### 1) Что меняет RDMA

Обычная модель сети:

- приложение → системный вызов → стек TCP/IP → драйвер → сетевой адаптер → сеть → обратно;
- каждое обращение проходит много слоёв, создавая задержки.

С RDMA:

- можно настроить **прямой доступ к памяти** удалённого узла:
  - хост А может читать/писать участки памяти хоста Б с минимальными накладными расходами;
  - используются специальные протоколы (RoCE, iWARP, InfiniBand).

Последствия для СУБД:

- распределённые транзакции и буферные пулы можно проектировать так, как если бы “часть памяти была просто чуть дальше”;
- но:
  - алгоритмы становятся сложнее;
  - требования к надёжности сети и конфигурации растут.

#### 2) In‑memory СУБД

Примеры:

- **VoltDB** — in‑memory, ориентированная на большие скорости транзакций, снапшоты на диск.
- **MemSQL / SingleStore** — гибрид: in‑memory + диск, columnstore, аналитика.
- **SQL Server Hekaton (In‑Memory OLTP)** — in‑memory таблицы внутри SQL Server.

Особенности:

- данные живут в памяти, формат хранения оптимизирован под **минимальные накладные расходы** (без лишней сериализации).
- диск используется:
  - для снапшотов (checkpoint’ов);
  - для журналов изменений.
- при рестарте:
  - база загружается из снапшотов/журналов обратно в память.

Плюсы:

- очень низкая latency на запросы;
- высокая пропускная способность.

Минусы:

- ограничение **объёмом памяти**;
- усложнённая история с durability/репликацией;
- стоимость оборудования (большие объёмы RAM, RDMA‑сети).

### Пошагово

Пример упрощённого сценария in‑memory базы:

1. При старте:
   - читаем снапшот состояния из диска;
   - восстанавливаем структуры в памяти;
   - “догоняемся” по журналу изменений с момента снапшота.
2. При работе:
   - все изменения происходят в памяти;
   - журнал изменений пишется последовательно на диск/в лог (иногда с репликацией).
3. Периодически:
   - делаем новый снапшот;
   - очищаем старую часть журнала.
4. При сбое:
   - повторяем процесс старта (снапшот + журнал).

Аналогично классической СУБД, но **формат хранения заточен только под память**, а диск — вспомогательный.

### Простыми словами

In‑memory база — это как **огромная электронная таблица в оперативной памяти**, к которой:

- очень быстро обращаются пользователи;
- периодически делаются **фотографии (снапшоты)** и записываются **журнал изменений**, чтобы можно было восстановиться.

RDMA — это как **подключённый по кабелю второй компьютер**, у которого ты можешь брать данные **почти как из своей памяти**, минуя обычные протоколы общения.

### Картинка в голове

Представь:

- обычная БД — это дом с **подвалом (диск)** и **этажами (память)**:
  - чтобы что‑то сохранить навсегда, нужно спуститься в подвал.
- in‑memory БД — это дом, где **почти всё происходит на одном этаже**, а подвал — только для архивов/страховки.

RDMA — это **быстрая труба между этажами разных домов**, по которой можно передавать данные почти как по внутренней лестнице.

### Как запомнить

- **In‑memory = всё родное в RAM, диск — только страховой полис.**
- **RDMA = память по сети, но с дорогими требованиями к инфраструктуре.**

### Примеры

- **SQL Server Hekaton:**
  - in‑memory таблицы хранятся в особом формате;
  - для них используются специальные алгоритмы конкурентного доступа;
  - диск — для checkpoint’ов и логов.
- **VoltDB:**
  - partioning по ключам;
  - однопоточное выполнение на раздел таблицы (чтобы избежать сложных блокировок);
  - репликация и снапшоты для надёжности.

### Практика / реальные сценарии

- **Биржи, high‑frequency trading, телеком:**
  - in‑memory хранилища справочников и недавней истории;
  - строгие требования к latency (микросекунды/миллисекунды).
- **Обычные веб‑приложения/корпоративные системы:**
  - чаще всего достаточно хорошо настроенного PostgreSQL/MySQL/SQL Server с кэширующими слоями;
  - in‑memory базы используются точечно (кэши, очереди, специфичные задачи).

### Типичные ошибки

- Пытаться “перейти на in‑memory” без понимания, что **данные по‑прежнему нужно где‑то надёжно хранить**.
- Ожидать, что RDMA “сам по себе” решит все проблемы latency без серьёзной работы над архитектурой и кодом.
- Использовать in‑memory технологии там, где bottleneck — вовсе не диск (а, например, бизнес‑логика или сеть до клиента).

### Что будет, если…

- **Переоценить in‑memory / RDMA:**
  - можно получить очень сложную и дорогую систему, которая мало чем лучше “хорошо настроенной классической БД плюс кэши”;
  - команда может тратить много сил на поддержку инфраструктуры вместо фокуса на бизнес‑логике.
- **Недооценить их в latency‑критичных сценариях:**
  - при высоких требованиях к задержкам классические СУБД могут быть физически не способны уложиться в SLA при любых настройках.

### Проверь себя

1. Почему даже in‑memory БД нуждается в journaling/снапшотах?

<details><summary>Ответ</summary>
Потому что оперативная память **летучая**: при сбое питания/рестарте её содержимое теряется. Чтобы после перезапуска восстановить состояние БД, нужно иметь **устойчивый источник истины** — снапшот + журнал изменений. Иначе in‑memory БД была бы пригодна только как кэш, а не как долговременное хранилище.
</details>

2. В каких случаях выбор in‑memory/rdma‑решения оправдан по сравнению с традиционной СУБД + кэш?

<details><summary>Ответ</summary>
Когда **жёстко ограничена задержка** (микросекунды/единицы миллисекунд) и невозможно держать нужную семантику только с кэшем перед традиционной БД (например, нужны сложные транзакции, строгая согласованность, высокая частота обновлений). В таких сценариях in‑memory/rdma‑решение может быть единственным реалистичным способом уложиться в SLA, несмотря на сложность и стоимость.
</details>

3. Почему простое “положим Redis/RAM‑кэш перед БД” не всегда заменяет специализированную in‑memory СУБД?

<details><summary>Ответ</summary>
Потому что кэш, как правило, даёт **слабые или вероятностные гарантии**: данные могут устареть, потеряться при эвикции, не участвовать в транзакциях вместе с БД. Специализированная in‑memory СУБД строит **полную транзакционную модель в памяти**: она управляет конфликтами, журналированием, восстановлением, иногда шардированием и репликацией. Если бизнес‑логика требует жёстких гарантий (ACID, сложные связи), кэш + БД может быть недостаточен, а in‑memory движок — оправдан.
</details>

### Запомните

- In‑memory и RDMA — это **про крайние случаи по требованиям к производительности**, а не универсальное лекарство.
- Даже самые быстрые системы обязаны решать **те же вопросы durability и согласованности**, что и классические СУБД.
