## 270. Cache‑friendly алгоритмы и layout данных

### Цель раздела

Понять, **как расположение данных в памяти и шаблоны доступа к ним влияют на производительность запросов**,  
чем отличаются строковое и колоночное хранение, что такое cache line и cache‑friendly алгоритмы,  
и как это проявляется в реальных СУБД (PostgreSQL, MySQL, SQLite, Oracle, SQL Server, колоночные движки).

### В этом разделе главное

1. CPU работает с памятью **крупными блоками (cache line)**, а не байтами; неудачный доступ к памяти может убить производительность даже “простого” запроса.
2. **Строковый layout** удобен для OLTP, где часто читают и пишут целые строки; **колоночный** — для аналитики и агрегаций.
3. Cache‑friendly алгоритм **ходит по памяти “по порядку”**, минимизируя cache miss’ы и случайные прыжки.
4. На уровне СУБД это влияет на то, **как сканируются страницы и индексы**, как работают join’ы и агрегаты.
5. Без измерений (профилировка CPU, cache misses) оптимизации “на глаз” легко вводят в заблуждение.

### Термины

- **Cache line** — минимальная единица обмена между памятью и кешем CPU (обычно 64 байта).
- **Cache hit / miss** — попадание/промах по кэшу: данные уже в кеше или их нужно подтянуть из более медленной памяти.
- **Row‑oriented layout** — хранение данных по строкам.
- **Column‑oriented layout** — хранение данных по столбцам.
- **Cache‑friendly алгоритм** — алгоритм с хорошей локальностью ссылок (spatial/temporal locality).
- **Cache‑oblivious структура** — структура данных, которая хорошо работает с любыми размерами кеша.

### Теория и правила

#### 1) Как CPU реально работает с памятью

Интуитивно:

- у CPU есть **несколько уровней кеша** (L1, L2, L3) и основная память (RAM);
- доступ к L1 — **очень быстрый**, к RAM — **на порядки медленнее**;
- при обращении к адресу CPU подтягивает в кеш **целую cache line**, а не один байт.

Последствия:

- если код идёт **по массиву подряд**, CPU подгружает данные блоками, а предсказатель подкачки помогает заранее;
- если код “скачет” по случайным адресам (например, по указателям в связном списке) — почти каждый доступ к данным может быть cache miss.

В контексте СУБД:

- когда план запроса делает последовательное сканирование таблицы (sequential scan), он **по сути идёт по страницам и cache line‑ам подряд** — это выгодно;
- когда много мелких случайных чтений (random I/O по страницам или случайный доступ внутри страницы) — кеш процессора и память используются далеко не оптимально.

#### 2) Row‑ vs column‑oriented layout

Классические OLTP‑СУБД (PostgreSQL, MySQL InnoDB, Oracle, SQL Server в rowstore) хранят данные **по строкам**:

- вся строка (или почти вся) лежит в одном блоке/странице;
- чтение одной строки — **один‑два cache line** (если структура аккуратная);
- обновление части строки почти всегда приводит к чтению/записи всей страницы.

Колонночные движки (ClickHouse, columnstore в SQL Server, аналитические расширения PostgreSQL и т.д.) хранят данные **по столбцам**:

- значения одного поля из многих строк лежат подряд;
- агрегации `SUM(price)`, `AVG(duration)` читают только нужные колонки;
- кеш и CPU используются эффективнее для аналитических запросов.

**Правило:**

- **OLTP → row‑oriented**, потому что операции мелкие и точечные (по отдельным строкам);
- **OLAP/аналитика → column‑oriented**, потому что важно быстро сканировать большие массивы по нескольким колонкам.

#### 3) Cache‑friendly алгоритмы в планах запросов

Примеры:

- **Sequential scan** по таблице:
  - читает страницы по порядку;
  - часто хорошо ложится в кеш;
  - особенно выгоден при чтении большого процента таблицы.
- **Index range scan**:
  - идёт по B‑дереву и листьям в порядке ключей;
  - для узких диапазонов запросов тоже достаточно cache‑friendly.
- **Random index lookup**:
  - много мелких чтений разных страниц индекса и таблицы;
  - при больших объёмах данных может привести к множеству cache miss’ов, даже если всё “помещается в память”.

Внутри join’ов и агрегатов:

- hash join, который аккуратно раскладывает данные по buckets, может быть более cache‑friendly, чем nested loop с плохой локальностью;
- сортировка “кусочками” (runs) слиянием тоже проектируется с мыслью о кешах.

#### 4) Cache‑oblivious структуры

Идея cache‑oblivious:

- спроектировать структуру так, чтобы **для любых размеров кеша и страниц** большая часть работы приходилась на последовательный обход;
- классический пример — **cache‑oblivious B‑tree** или layout интервалов в массиве.

В практических СУБД:

- не всегда используются формальные cache‑oblivious структуры, но идеи из них влияют на layout страниц, индексов, зон (zone maps в колоночных движках) и т.д.

### Пошагово

Рассмотрим запрос:

```sql
SELECT user_id, SUM(amount)
FROM payments
WHERE created_at >= now() - interval '7 days'
GROUP BY user_id;
```

1. **Row‑based OLTP (PostgreSQL, MySQL InnoDB, SQLite):**
   - планировщик может выбрать:
     - либо `Index Range Scan` по `created_at` с последующим чтением строк;
     - либо `Seq Scan` по таблице, если почти все строки “свежие”.
   - при `Seq Scan`:
     - читаются страницы подряд;
     - каждая страница содержит и `user_id`, и `amount`, и другие поля;
     - CPU читает сразу много ненужных полей — но это всё равно достаточно cache‑friendly.
2. **Columnstore (ClickHouse, columnstore index в SQL Server):**
   - хранятся отдельно колонка `user_id`, отдельно `amount`, отдельно `created_at`;
   - фильтр по `created_at` может использовать zone maps/minmax для отсечения целых сегментов;
   - агрегация читает только `user_id`, `amount`, при этом данные идут “плотно” и отлично ложатся в кэш.
3. **Cache‑friendly доступ:**
   - в обоих случаях планы стараются идти по данным **последовательно**;
   - если запрос написан так, что требует случайного чтения по большому количеству первичных ключей, cache‑локальность ухудшается.

### Простыми словами

Представь, что память — это **книжный шкаф**, а кэш CPU — это **маленький стол**, на который ты можешь выложить только несколько книжек.

- Если ты читаешь книгу **по порядку**, ты кладёшь её на стол, перелистываешь страницы одну за другой — всё удобно.
- Если ты каждые 5 секунд **перескакиваешь на случайную страницу другой книги**, тебе приходится:
  - убирать одну книгу со стола;
  - доставать другую;
  - тратить время на переключения.

Cache‑friendly алгоритм — это когда код (и план запроса) **читает данные как можно более последовательно**, а не прыгает по случайным адресам.

Row‑layout — это когда “страница книги” — это одна строка со всеми полями. Column‑layout — когда у тебя **отдельные книги для каждого столбца**, и ты листаешь только нужные.

### Картинка в голове

Представь себе длинный коридор, по стенам которого висят папки:

- **row‑layout** — в каждой папке “строка целиком”: имя, возраст, город, баланс и т.д.;
- **column‑layout** — по одной стене только “имена”, по другой только “города”, по третьей только “балансы”.

Если задача — “прочитай всех, кто живёт в Москве и посчитай общую сумму балансов”, то:

- при row‑layout ты идёшь по всем папкам и каждый раз листаешь все поля;
- при column‑layout ты идёшь вдоль стены “город”, отмечаешь нужных, а потом берёшь только их балансы.

CPU‑кеш — это твоя “рабочая зона”, куда ты приносишь пачки папок. Чем более последовательно ты идёшь по коридору, тем меньше беготни.

### Как запомнить

- **Row → строки → OLTP. Column → столбцы → аналитика.**
- **Cache‑friendly = идти по памяти, как по списку, а не как по случайным ссылкам.**
- Если видишь в плане много случайных обращений к индексам при большом объёме данных — **подумай про последовательное сканирование и layout**.

### Примеры

#### PostgreSQL: seq scan vs index scan

```sql
EXPLAIN ANALYZE
SELECT *
FROM orders
WHERE created_at >= now() - interval '1 day';
```

- Если `orders` — 10 млн строк, а за день — 8 млн:
  - `Seq Scan` по таблице будет **кеш‑дружелюбен**: страницы читаются подряд.
  - `Index Scan` по `created_at` может прыгать по страницам таблицы и будет **хуже по кешу**.

#### ClickHouse: только нужные колонки

```sql
SELECT user_id, sum(amount)
FROM payments
WHERE created_at >= now() - INTERVAL 1 DAY
GROUP BY user_id;
```

- Читаются только столбцы `user_id`, `amount`, `created_at` (и, возможно, парочка служебных).
- Данные внутри каждого столбца лежат **плотно и отсортированно** — кэш используется максимально эффективно.

#### SQLite

SQLite — классическая row‑oriented СУБД, работающая часто в памяти/файле:

- для небольших таблиц всё помещается в page cache ОС/СУБД;
- но при росте объёма данных layout страниц и порядок доступа начинают играть всё большую роль.

### Практика / реальные сценарии

- **OLTP на PostgreSQL/MySQL:**
  - следи за тем, насколько запросы **отбирают узкие диапазоны** или бьют по всей таблице;
  - иногда `Seq Scan` быстрее, чем “красивый индекс”, именно из‑за кеша.
- **Аналитика на ClickHouse/columnstore:**
  - проектируй таблицы так, чтобы частые запросы использовали **минимальное количество столбцов**;
  - продумывай порядок сортировки/ключи, чтобы зоны/сегменты отсекались максимально.
- **Смешанные нагрузки:**
  - иногда выгодно разделить данные на **две таблицы**: одну под OLTP (row), вторую под аналитические выгрузки (column или отдельный движок).

### Типичные ошибки

- Думать, что “если всё в памяти, то кеш процессора не важен” — на самом деле разница между L1 и RAM огромна.
- Считать, что **индекс всегда быстрее, чем seq scan**, независимо от процентов выборки.
- Пытаться “оптимизировать” layout, **не видя плана запроса и профиля CPU**.
- Не обращать внимания на то, какие поля реально нужны запросу, и тащить `SELECT *` везде.

### Что будет, если…

- **Игнорировать cache‑дружелюбность:**
  - при росте данных и числа ядер ты внезапно получишь **огромный рост CPU time** без увеличения полезного TPS;
  - профилировщик покажет много времени в функциях чтения страниц/структур, а не в “полезной логике”.
- **Перепутать row vs column для нагрузки:**
  - OLTP на чистом columnstore — постоянные мелкие апдейты и вставки приведут к тяжёлым merge/compaction;
  - аналитика на чистом rowstore — придётся читать много лишних колонок, кеш будет забиваться “мусором”.

### Проверь себя

1. Почему запрос с `SELECT *` по широкой таблице может быть в разы медленнее запроса, который выбирает только несколько колонок, даже если есть подходящий индекс?

<details><summary>Ответ</summary>
Потому что `SELECT *` заставляет движок **читать и тянуть в кеш все колонки**, даже те, которые не нужны для логики. В rowstore это означает более толстые строки и больше cache line‑ов на строку, а в columnstore — чтение лишних колонок. Запрос с несколькими колонками лучше использует кеш и ширину памяти, особенно если движок умеет считывать только нужные столбцы.
</details>

2. В чём различие по cache‑поведению между `Seq Scan` и `Index Scan` по широкому диапазону значений?

<details><summary>Ответ</summary>
`Seq Scan` читает страницы таблицы **по порядку**, что удобно кешу и предзагрузке; `Index Scan` ходит по B‑дереву и затем по таблице, часто с большим количеством **случайных обращений** к страницам, особенно если диапазон широкий. В результате при высоком проценте выборки `Seq Scan` может быть быстрее, несмотря на “отсутствие индекса”, за счёт лучшего использования кеша и диска.
</details>

3. Почему даже при “полном попадании в память” (всё в page cache) запрос всё равно может быть медленным с точки зрения CPU?

<details><summary>Ответ</summary>
Потому что “всё в памяти” ещё не означает, что данные **хорошо попадают в кеш CPU**. Если структура хранения и алгоритмы обращаются к памяти случайно (много разрозненных указателей, прыжки по страницам), то почти каждый доступ становится cache miss’ом, и ядро тратит много циклов, ожидая данных из RAM. В итоге запрос может не ждать диска, но “ждать память” и тратить CPU впустую — это и есть плохая cache‑дружелюбность.
</details>

### Запомните

- **Кеш CPU так же важен, как и диск.** Плохо спроектированный доступ к памяти может убить производительность даже при “нулевом I/O”.
- **Row vs column** — не вопрос вкуса, а компромисс между OLTP и аналитикой.
- Прежде чем “оптимизировать структуру таблиц под кеш”, **посмотри план запроса и профилировку CPU**.
