## 152. Логи и диагностика

### Цель раздела

Научиться **использовать логи СУБД и диагностические запросы** для разбора инцидентов: понимать уровни логирования, коды ошибок, stack traces и уметь применять инструменты вроде pgBadger и pt‑query‑digest.

### В этом разделе главное

- Логи — это **история того, что реально происходило**: запросы, ошибки, deadlock'и, медленные операции.
- Без логов troubleshooting превращается в угадывание.
- Важно не только включить логи, но и **организовать их сбор, ротацию и анализ**.

### Простыми словами

Лог — это как **чёрный ящик в самолёте**: когда что‑то пошло не так, ты идёшь туда смотреть, что именно происходило по шагам.  
Хороший лог даёт ответы: какой запрос, кто его запустил, сколько он шёл, на чём упал, какой код ошибки вернула БД.

### Если вообще ничего не понятно (объяснение «на пальцах»)

Представь, что у тебя есть **ежедневник**, куда ты записываешь:

- кто к тебе приходил;
- о чём вы говорили;
- что делали;
- какие проблемы возникли.

Если **ничего не записывать**, а потом через неделю пытаться вспомнить, почему что‑то сломалось — это почти нереально.

В базе:

- логи — это такой же **ежедневник/журнал смены**:
  - какие запросы выполнялись;
  - кто их запускал;
  - сколько они шли;
  - с какими ошибками упали.

Без логов:

- при проблеме все говорят: «наверное, интернет лаганул»;
- никто не знает, **какой конкретно запрос, с какими данными, в какое время, в каком состоянии системы** сломался.

С логами:

1. Ты по времени и коду ошибки находишь **конкретный момент и конкретный запрос**.
2. Видишь, что именно случилось:
   - конфликт по уникальному ключу;
   - deadlock;
   - превышен таймаут;
   - не хватило прав;
   - что‑то ещё.
3. Уже на основе этого **принимаешь осознанное решение**, а не гадаешь.

Поэтому настройка логов — это не «для параноиков» и не «для дебага», а **обязательная часть нормальной эксплуатации БД**.

### Термины

- **Уровень логирования** — насколько подробно БД пишет сообщения (ERROR, WARNING, INFO, DEBUG и т.п.).
- **Код ошибки** — формализованный код (например, в PostgreSQL `23505` для уникального ключа), по которому можно понять тип проблемы.
- **Stack trace** — стек вызовов, показывающий, где в коде произошла ошибка.

### Теория и правила

1. **PostgreSQL: основные параметры логирования**:
   - `log_min_duration_statement` — порог для записи медленных запросов.
   - `log_statement` — какие запросы логировать (ALL, MOD, ddl и т.п.).
   - `log_line_prefix` — формат строки лога (кто, когда, откуда).

2. **Типичные виды записей в postgresql.log**:
   - Ошибки (`ERROR`), предупреждения (`WARNING`), замечания (`NOTICE`).
   - Deadlock'и (`deadlock detected`).
   - Медленные запросы (если включён `log_min_duration_statement`).

3. **MySQL: типы логов**:
   - `error log` — ошибки сервера.
   - `slow query log` — медленные запросы.
   - `general log` — все запросы (обычно включают временно).

4. **Инструменты анализа**:
   - **pgBadger** — разбирает `postgresql.log`, строит отчёты по медленным запросам, ошибкам, нагрузке.
   - **pt-query-digest** — анализирует MySQL slow query log.
   - Централизованные системы логирования (ELK, Loki, Cloud‑сервисы).

5. **Диагностические запросы**:
   - `pg_stat_activity` — что сейчас выполняется.
   - `pg_stat_database` — статистика по базам.
   - `pg_stat_all_tables`/`pg_stat_all_indexes` — активность по таблицам/индексам.

### Примеры

**Пример настройки PostgreSQL на логирование запросов дольше 500 мс:**

```sql
ALTER SYSTEM SET log_min_duration_statement = 500; -- миллисекунды
SELECT pg_reload_conf();
```

**Поиск ошибок уникального ключа по коду ошибки (PostgreSQL):**

```sql
-- В приложении или в логах ищи код SQLSTATE 23505
```

**Простой диагностический запрос по активности:**

```sql
SELECT
  datname,
  numbackends,
  xact_commit,
  xact_rollback,
  blks_read,
  blks_hit
FROM pg_stat_database;
```

### Картинка в голове

Представь, что сервер БД — это большая фабрика.  
Логи — это **журнал смены**, куда записывается, кто когда пришёл, что делал, что сломалось и как это починили.  
Если журнал пустой или хаотичный, разбирать инциденты — как восстанавливать события «по слухам».

### Как запомнить

- Настройка логов — это не «опция для отладки», а **обязательный элемент эксплуатации**.  
- Инструменты анализа (pgBadger, pt‑query‑digest) позволяют не листать сырые файлы, а видеть картину.

### Типичные ошибки

- Не включать slow query лог и потом не иметь истории медленных запросов.
- Писать слишком много (например, все запросы) в боевой лог без ротации → диск забивается.
- Не собирать логи централизованно и терять их при ротации или падении инстанса.

### Пошаговый алгоритм разбора инцидента по логам

1. **Сформулируй инцидент.**
   - Например: «в 14:32 пользователи видели ошибку 500 при сохранении заказа».

2. **Найди соответствующие записи в логах приложения.**
   - Там обычно есть:
     - время;
     - URL/операция;
     - текст ошибки БД (или хотя бы код SQLSTATE).

3. **По времени и коду ошибки найди записи в логах БД.**
   - Для PostgreSQL — `postgresql.log` с хорошим `log_line_prefix` (время, пользователь, база, PID).
   - Для MySQL — `error log` или `slow query log`.

4. **Разбери одну конкретную цепочку.**
   - Какой запрос выполнялся?
   - С какими параметрами?
   - Сколько он шёл?
   - На чём упал (код ошибки, текст)?

5. **Сопоставь с диагностическими представлениями.**
   - Если инцидент ещё продолжается:
     - посмотри `pg_stat_activity`, `pg_stat_database`, `pg_locks` и т.п.
   - Если уже прошёл:
     - сделай выводы по статистике и логам, нужно ли менять индексы, запросы или настройки.

### Мини‑пример: ошибка уникального ключа

В приложении видим ошибку:

> ERROR: duplicate key value violates unique constraint "users_email_key"  
> SQLSTATE: 23505

Шаги:

1. В логах БД ищем запись с этим кодом и временем.
2. Видим запрос, который пытался вставить пользователя с уже существующим email.
3. Дальше вопрос бизнес‑логики:
   - либо перехватывать эту ошибку и сообщать пользователю, что такой email уже существует;
   - либо менять схему данных/ограничения.

### Проверь себя

<details>
<summary>1. Зачем нужен `log_min_duration_statement` в PostgreSQL?</summary>

Чтобы автоматически логировать все запросы, которые выполняются дольше заданного порога. Это даёт историю медленных запросов без дополнительного профилирования.
</details>

<details>
<summary>2. Чем slow query log в MySQL отличается от general log?</summary>

Slow query log пишет только медленные запросы (дольше порога `long_query_time`).  
General log пишет **все** запросы и обычно включается временно, потому что может сильно нагружать систему и диск.
</details>

<details>
<summary>3. Зачем в логах приложения хранить код ошибки БД (SQLSTATE) и время?</summary>

Чтобы по 500‑ошибке в приложении можно было **быстро найти соответствующую запись в логе БД**:  
по времени и коду SQLSTATE ты находишь конкретный SQL‑запрос, видишь, на чём он упал (уникальный ключ, deadlock, таймаут и т.д.) и уже дальше разбираешься по сути проблемы.  
Без этого приходится «угадывать», какой именно запрос был источником ошибки.
</details>


### Запомните

**Логи и диагностические представления — это твой главный инструмент для понимания того, что реально происходило в БД.**  
Без организованного логирования troubleshooting превращается в угадывание и перекладывание вины на «случайные лаги».

### Сценарий 1 — страница отчёта открывается по 30 секунд (медленные запросы)

**Условие.**

- Пользователи жалуются: «страница «Отчёт по продажам за год» иногда открывается 20–30 секунд».
- Остальные страницы (список товаров, корзина) работают нормально.
- У тебя есть доступ к PostgreSQL с включённым `pg_stat_statements`.

**Вопросы к себе:**

1. Как поймать **конкретный запрос**, который отвечает за эту страницу?
2. Как понять, **почему именно он медленный**?
3. Какие **типовые решения** ты бы проверил в первую очередь?

<details>
<summary><strong>Разбор (один из правильных планов)</strong></summary>

1. **Сузить область поиска.**
   - В логах приложения посмотрю, какой endpoint/функция отвечает за отчёт.
   - Запущу страницу отчёта сам несколько раз, чтобы запрос «засветился» в статистике.

2. **Найти запрос в `pg_stat_statements`.**

   ```sql
   SELECT
     query,
     calls,
     total_time,
     mean_time
   FROM pg_stat_statements
   WHERE query ILIKE '%FROM sales%'  -- или другая табличка отчёта
   ORDER BY total_time DESC
   LIMIT 5;
   ```

   - Смотрю, какой запрос:
     - относится к нужной таблице;
     - имеет большой `total_time` или `mean_time`.

3. **Взять текст запроса и прогнать `EXPLAIN (ANALYZE, BUFFERS)`.**

   - Беру реальные параметры (`WHERE created_at >= now() - interval '1 year'` и т.п.).
   - Запускаю:

   ```sql
   EXPLAIN (ANALYZE, BUFFERS)
   ... проблемный запрос ...
   ```

4. **Найти узкое место в плане.**
   - Ищу:
     - `Seq Scan` по большой таблице;
     - `Nested Loop` с большим числом строк;
     - тяжёлый `Sort` или `HashAggregate`, который уходит в temp‑файлы.

5. **Сформулировать причину простым языком.**
   - Пример: «нет индекса по `created_at`, поэтому отчёт читает всю таблицу за 3 года».

6. **Применить типовые решения.**
   - Добавить/изменить индекс.
   - Переписать запрос так, чтобы использовать индекс (убрать функцию над колонкой в `WHERE`).
   - Если дело в сортировке — подумать о предагрегации, материализованном виде, индексах под `ORDER BY`.

7. **Проверить результат.**
   - Снова запускаю `EXPLAIN (ANALYZE, BUFFERS)`.
   - Через какое‑то время смотрю `pg_stat_statements` и убеждаюсь, что `mean_time` и `total_time` уменьшились.

**Главная мысль:** сначала нахожу *конкретного монстра* через статистику, потом **рентгеном плана** ищу орган, который болит, и только после этого выбираю лечение.

</details>

### Сценарий 2 — приложение «подвисает», а потом падает с ошибкой таймаута (блокировки)

**Условие.**

- В 11:15 пользователи начинают жаловаться: «при сохранении данных форма висит, потом ошибка «таймаут по ожиданию БД»».
- В логах приложения видно, что многие запросы падают по таймауту.
- В это время DevOps запускал миграцию, которая меняет структуру большой таблицы `orders`.

**Вопросы к себе:**

1. Как понять, **есть ли сейчас блокировки** и кто кого блокирует?
2. Как решить: **ждать миграцию или прерывать её**?
3. Что сделать, чтобы **в будущем** такие ситуации проходили мягче?

<details>
<summary><strong>Разбор</strong></summary>

1. **Смотрю текущую активность.**

   ```sql
   SELECT
     pid,
     now() - query_start AS duration,
     state,
     wait_event_type,
     wait_event,
     query
   FROM pg_stat_activity
   WHERE state <> 'idle'
   ORDER BY duration DESC;
   ```

   - Ищу процессы с большим `duration` и `wait_event_type = 'Lock'`.

2. **Нахожу, кто кого блокирует.**

   Быстрый путь:

   ```sql
   SELECT
     pid AS blocked_pid,
     pg_blocking_pids(pid) AS blocking_pids,
     query
   FROM pg_stat_activity
   WHERE cardinality(pg_blocking_pids(pid)) > 0;
   ```

   - Вижу, что множество запросов к `orders` блокируются одной сессией с миграцией.

3. **Решаю, что важнее: миграция или онлайн‑трафик.**
   - Если это плановая миграция и есть окно простоя — **можно подождать**, но тогда надо снизить нагрузку или перевести приложение в «режим обслуживания».
   - Если критично держать онлайн‑трафик:
     - сначала **обсуждаю с командой**;
     - потом делаю `pg_cancel_backend`/`pg_terminate_backend` для миграции.

4. **Профилактика на будущее.**
   - Миграции, которые могут держать большие блокировки:
     - запускать в заранее согласованное окно;
     - по возможности разбивать на несколько меньших изменений;
     - заранее оценивать, какие таблицы и как долго будут блокироваться.
   - В приложении:
     - настраивать разумные `lock_timeout` и `statement_timeout`;
     - не держать транзакции открытыми дольше, чем нужно.

**Главная мысль:** блокировки — это не «магия», а вполне конкретные PID и запросы. Их нужно **увидеть, понять, кто главный блокирующий, и уже потом решать, кого прерывать**.

</details>

### Сценарий 3 — отчёты с реплики возвращают старые данные (репликация)

**Условие.**

- У тебя есть мастер и одна реплика PostgreSQL.
- Приложение читает отчёты с реплики, чтобы не нагружать мастер.
- Пользователи жалуются: «мы внесли изменения, но в отчёте они появляются через несколько минут».

**Вопросы к себе:**

1. Как проверить, **какой сейчас lag** у реплики?
2. Как понять, **почему он растёт**?
3. Какие варианты **лечения** есть?

<details>
<summary><strong>Разбор</strong></summary>

1. **Смотрю состояние реплик.**

   ```sql
   SELECT
     application_name,
     client_addr,
     state,
     sync_state,
     pg_wal_lsn_diff(pg_current_wal_lsn(), replay_lsn) AS byte_lag,
     now() - replay_timestamp                         AS time_lag
   FROM pg_stat_replication;
   ```

   - Смотрю `time_lag` → если там минуты, это и есть причина «старых» данных.

2. **Проверяю, что происходит на реплике.**
   - CPU/память/диск/сеть.
   - Нет ли на реплике тяжёлых аналитических запросов, которые держат блокировки.

3. **Соотношу lag с событиями.**
   - Возможно, в это время:
     - запускаются тяжёлые batch‑операции на мастере;
     - крутится аналитика на реплике.

4. **Варианты решений.**
   - Если аналитика на реплике мешает применению WAL:
     - вынести часть запросов на отдельную реплику;
     - ограничить ресурсы аналитики;
     - запускать отчёты в менее «горячее» время.
   - Если реплика слабее мастера:
     - добавить ей ресурсов (CPU, диск);
     - или перераспределить поток запросов.

**Главная мысль:** lag — это измеримая величина, и его причины тоже конкретны: слабые ресурсы, тяжёлые запросы, большая нагрузка на мастер. Нужно **сначала измерить, потом смотреть, где bottleneck**.

</details>

### Сценарий 4 — процесс БД убивает OOM‑killer (память)

**Условие.**

- Ночью в логах ОС видишь сообщения вида `Out of memory: Kill process ... postgres`.
- После этого база перезапускается.
- В это же время по расписанию запускаются тяжёлые отчёты и batch‑обработки.

**Вопросы к себе:**

1. Как понять, **что именно съело память**?
2. Как отличить проблему **неудачных настроек** от **неудачного запроса**?
3. Какие шаги уменьшат вероятность повторного OOM?

<details>
<summary><strong>Разбор</strong></summary>

1. **Смотрю логи ОС и БД вокруг момента OOM.**
   - В логах ОС — кто был убит.
   - В логах БД — какие запросы выполнялись в это время.

2. **Проверяю настройки памяти.**

   ```sql
   SHOW shared_buffers;
   SHOW work_mem;
   SHOW maintenance_work_mem;
   SHOW max_connections;
   ```

   - Прикидываю: если `work_mem` высокий, а `max_connections` большой, **в худшем случае** все вместе могут превысить RAM.

3. **Смотрю temp‑файлы.**

   ```sql
   SELECT
     datname,
     temp_files,
     pg_size_pretty(temp_bytes) AS temp_bytes
   FROM pg_stat_database
   ORDER BY temp_bytes DESC;
   ```

   - Если значения большие — запросы активно выбрасывают временные данные на диск → им не хватало `work_mem`.

4. **Нахожу тяжёлые запросы.**
   - Смотрю `pg_stat_statements` или slow query логи.
   - Ищу запросы с большими сортировками/агрегациями (в плане).

5. **Решения.**
   - Для настроек:
     - уменьшить `work_mem` и/или `maintenance_work_mem` до разумного значения;
     - ограничить `max_connections`, если он сильно завышен.
   - Для запросов:
     - оптимизировать отчёты (предагрегация, разбиение, материализованные представления);
     - запускать тяжёлые batch‑операции в разные окна, а не все разом.

**Главная мысль:** OOM почти всегда — результат **сложения: настройки + паттерн запросов**. Нужно проверять **и то, и другое**, а не только «ставить памяти побольше».

</details>

### Сценарий 5 — диск забит, база не может писать WAL (I/O, слоты, bloat)

**Условие.**

- На мастере PostgreSQL резко кончилось место на диске.
- В логах — жалобы на невозможность записать WAL.
- При этом в `pg_replication_slots` видно несколько неактивных слотов с большим объёмом удерживаемого WAL.

**Вопросы к себе:**

1. Как аккуратно **освободить место**, не потеряв важных данных?
2. Как понять, **какой слот больше всего удерживает WAL**?
3. Как организовать мониторинг, чтобы **такое не повторялось**?

<details>
<summary><strong>Разбор</strong></summary>

1. **Смотрю слоты.**

   ```sql
   SELECT
     slot_name,
     slot_type,
     active,
     pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn) AS retained_wal
   FROM pg_replication_slots;
   ```

   - Ищу слоты с:
     - `active = false`;
     - большим `retained_wal`.

2. **Решаю судьбу каждого слота.**
   - Если за слотом **не стоит живая реплика/подписка**:
     - согласую с командой и **удаляю слот** (`SELECT pg_drop_replication_slot('имя');`).
   - Если слот нужен, но реплика отстаёт:
     - оценить, можно ли реплику «догнать» (ресурсы, сеть);
     - если нет — возможно, пересоздать реплику и слот.

3. **Параллельно оцениваю bloat и общий размер данных.**
   - Если диск забит не только WAL, но и раздутыми таблицами/индексами:
     - планирую `VACUUM`/`VACUUM FULL` в окне обслуживания;
     - рассматриваю партиционирование.

4. **Настраиваю мониторинг.**
   - Алерты на:
     - свободное место на диске;
     - объём WAL, удерживаемый слотами;
     - lag реплик.

**Главная мысль:** слоты репликации — очень полезный механизм, но без мониторинга они превращаются в «чёрную дыру» для диска. Нужно регулярно смотреть **retained_wal** и состояние слотов.

</details>

### Сценарий 6 — пользователи видят «случайные ошибки» при пиках нагрузки (логи и диагностика)

**Условие.**

- При пиковых нагрузках (распродажа, акция) пользователи периодически видят 500‑ошибки.
- В обычное время всё работает нормально.
- В логах приложения много однотипных сообщений «ошибка БД».

**Вопросы к себе:**

1. Как **привязать** конкретную 500‑ошибку к **конкретному запросу и коду ошибки БД**?
2. Как понять, **что именно ломается при пике** — блокировки, лимиты подключений, deadlock'и, что‑то ещё?
3. Какие данные о запросах и ошибках нужно **обязательно логировать**?

<details>
<summary><strong>Разбор</strong></summary>

1. **Смотрю логи приложения.**
   - Убеждаюсь, что туда пишутся:
     - время;
     - URL/endpoint;
     - идентификатор запроса/трейс‑id;
     - текст/код ошибки БД (SQLSTATE).

2. **По времени и коду ошибки иду в логи БД.**
   - Для PostgreSQL — `postgresql.log` с хорошим `log_line_prefix`.
   - По совместному:
     - времени;
     - PID;
     - коду ошибки

   нахожу конкретную запись, вижу:

   - текст SQL‑запроса;
   - код SQLSTATE;
   - иногда дополнительный контекст.

3. **Классифицирую тип ошибки.**
   - Уникальный ключ (`23505`) → конфликт данных.
   - Таймаут по блокировке → проблемы с блокировками.
   - Превышение лимита подключений → ресурсная проблема.
   - Deadlock detected → проблемы в порядке блокировок.

4. **Строю картину по инцидентам.**
   - Сколько было:
     - ошибок уникального ключа;
     - deadlock'ов;
     - таймаутов по блокировкам;
     - превышений лимитов.

5. **Решения.**
   - Для каждого типа ошибки — свои шаги (см. соответствующие разделы части XXVII).
   - В логах приложения:
     - записывать коды SQLSTATE;
     - логировать запросы/параметры хотя бы в обрезанном виде (без чувствительных данных).

**Главная мысль:** без нормальных логов troubleshooting превращается в гадание. Нужно всегда иметь возможность **из 500‑ошибки в приложении выйти на конкретный SQL, код ошибки и контекст в логах БД**.

</details>
