## 146. Диагностика медленных запросов

### Цель раздела

Научиться **системно находить и разбирать медленные запросы**: понимать, какие запросы реально «съедают» время и ресурсы, почему план выполнения оказался неудачным, и какие есть варианты исправления (индексы, переписывание, настройки, ресурсы).

### В этом разделе главное

- **Не оптимизируй «по ощущениям»** — сначала найди самые дорогие запросы по статистике.
- **Всегда смотри `EXPLAIN ANALYZE`** — именно он показывает реальный путь запроса и несоответствия оценок с фактом.
- **Типовые лекарства ограничены**: индексы, переписывание запроса, настройки планировщика и увеличение ресурсов.

### Простыми словами

Медленный запрос — это не просто тот, который «кажется медленным пользователю».  
Это запрос, который:

- либо **часто выполняется и в сумме съедает много времени**;
- либо **редко, но очень тяжёлый** и «забивает» базу, когда всё совпало неудачно.

Наша задача — **сначала найти таких «монстров»**, затем понять *почему* они медленные, и только после этого **подбирать лечение**.

### Если вообще ничего не понятно (объяснение «на пальцах»)

Представь, что база данных — это **столовая**, а запросы — это **люди, которые заказывают еду**.

- Одни берут **кофе навынос** — быстро.
- Другие заказывают **большой обед из пяти блюд** — долго.
- Если каждый день один и тот же человек заказывает огромный обед, именно он **дольше всего держит очередь и загружает кухню**.

В мире БД:

- быстрый запрос — это «кофе»;
- тяжёлый запрос — это «обед на 5 блюд»;
- твоя задача как администратора — **найти, какой именно «обед» тормозит кухню**.

Как ты это делаешь:

1. Сначала ты **не смотришь на весь день сразу**, а идёшь в отчёт (`pg_stat_statements` или slow query log), где видно:
   - какие «блюда» (запросы) заказывали;
   - сколько раз;
   - сколько всего времени кухня потратила на каждое.
2. Потом ты берёшь **конкретный тяжёлый заказ** (SQL‑запрос) и делаешь ему «рентген» (`EXPLAIN ANALYZE`), чтобы понять:
   - повар идёт короткой дорогой (использует индекс);
   - или бегает через весь склад (полный просмотр таблицы).
3. Когда понимаешь, **почему повар бегает так долго**, ты:
   - делаешь ему **короткий путь** (индекс);
   - или **облегчаешь заказ** (переписываешь запрос, убираешь лишние `JOIN`).

Важно: **не нужно запоминать сразу все команды и представления**.  
Запомни картинку:

- есть отчёт, где видно, **кто дольше всего ел время сервера**;
- есть рентген, который показывает, **где именно запрос тратит время**;
- есть простые действия: **индекс / переписать / подправить настройки**.

### Термины

- **Total time** — суммарное время, которое запрос потратил за период.
- **Mean time** — среднее время выполнения одного вызова.
- **Rows** в плане — ожидаемое/реальное количество строк, которое проходит через узел плана.
- **Cost** — абстрактная оценка стоимости узла, которую использует планировщик PostgreSQL.

### Теория и правила

1. **Где искать медленные запросы (PostgreSQL)**:
   - `pg_stat_statements` (нужно расширение): агрегирует запросы по нормализованному тексту.
   - `pg_stat_activity`: показывает текущие выполняющиеся запросы и их длительность.
   - Логи (`log_min_duration_statement`): можно настроить порог, начиная с которого запросы пишутся в лог.

2. **Где искать медленные запросы (MySQL)**:
   - **Slow query log**: включается настройками `slow_query_log = 1`, `long_query_time = N`.
   - `performance_schema` / `events_statements_summary_by_digest`: статистика по запросам.

3. **Основные метрики в `pg_stat_statements`**:
   - `total_time` — сколько *всего* времени этот тип запроса съел.
   - `mean_time` — среднее время.
   - `calls` — сколько раз вызывался.
   - Обычно **сортируют по `total_time`**, чтобы найти, *что в сумме* больше всего нагружает БД.

4. **EXPLAIN vs EXPLAIN ANALYZE**:
   - `EXPLAIN` — показывает только план (оценки, cost, ожидаемые строки).
   - `EXPLAIN ANALYZE` — **выполняет запрос** и показывает фактические времена и кол‑во строк.
   - В PostgreSQL желательно использовать `EXPLAIN (ANALYZE, BUFFERS, VERBOSE, FORMAT TEXT/JSON)` — так видно ещё и использование кэша.

5. **Как читать план** (очень упрощённо):
   - Дерево читается **снизу вверх** — от таблиц к верхнему узлу (например, `Aggregate` или `Sort`).
   - Смотри:
     - где `actual time` сильно больше ожидаемого;
     - где `rows` сильно отличаются от оценок;
     - где много повторных чтений (nested loop на большую таблицу).

6. **Типичные причины медленности**:
   - Нет нужных индексов → seq scan по большой таблице.
   - Неверные статистики → планировщик выбирает не тот план.
   - Избыточные `JOIN`/подзапросы → лишняя работа.
   - Функции над колонками в условиях (`WHERE lower(email) = ...`) → индекс не используется.
   - Нехватка памяти (`work_mem`) → тяжёлые сортировки и hash join'ы уходят в диск (temp‑файлы).

### Примеры

**Поиск самых тяжёлых запросов в PostgreSQL (`pg_stat_statements`):**

```sql
SELECT
  query,
  calls,
  total_time,
  mean_time,
  rows
FROM pg_stat_statements
ORDER BY total_time DESC
LIMIT 10;
```

**Поиск текущих «висящих» запросов:**

```sql
SELECT
  pid,
  now() - query_start AS duration,
  state,
  wait_event_type,
  wait_event,
  query
FROM pg_stat_activity
WHERE state != 'idle'
ORDER BY duration DESC
LIMIT 10;
```

**Пример `EXPLAIN ANALYZE` (PostgreSQL):**

```sql
EXPLAIN (ANALYZE, BUFFERS)
SELECT *
FROM orders
WHERE customer_id = 123
  AND created_at >= now() - interval '7 days';
```

Смотри, использует ли план `Index Scan` по индексу вида `(customer_id, created_at)` или делает `Seq Scan`.

**MySQL: включение slow query log (упрощённо):**

```sql
SET GLOBAL slow_query_log = 1;
SET GLOBAL long_query_time = 0.5; -- всё дольше 0.5 секунды идёт в лог
```

Потом логи можно разобрать, например, через `pt-query-digest`.

### Картинка в голове

Представь, что у тебя есть **список всех запросов за день** с их временем.  
Сортируя по `total_time`, ты как будто видишь, **кто украл больше всего времени у сервера**.  
А `EXPLAIN ANALYZE` — это **рентген** конкретного запроса: он показывает, какие «органы» работают нормально, а где узкое горлышко.

### Как запомнить

- Сначала **найди монстров** (`pg_stat_statements`, slow query log).  
- Потом **посвети рентгеном** (`EXPLAIN ANALYZE`).  
- И только потом **режь и лечи** (индексы, переписывание, настройки).

### Типичные ошибки

- Сразу переписывать запрос «на глаз», не посмотрев ни статистику, ни план.
- Оптимизировать редкие запросы, игнорируя те, которые в сумме съедают больше времени.
- Делать выводы по `EXPLAIN` без `ANALYZE`: реальное поведение может сильно отличаться от оценок.
- Запускать `EXPLAIN ANALYZE` на бою для супер‑тяжёлых запросов без понимания последствий (он их **реально выполнит**).

### Пошаговый алгоритм диагностики медленных запросов (PostgreSQL)

Предположим, пользователи жалуются: «страница отчёта открывается очень долго».  
Твой план действий может выглядеть так:

1. **Зафиксируй симптом простыми словами.**
   - Не «база тормозит», а: «запрос отчёта по продажам за год открывается 20–30 секунд».
   - Это важно, чтобы потом проверить: стало ли лучше именно это место.

2. **Найди запрос в статистике.**
   - Если включён `pg_stat_statements`:

     ```sql
     SELECT
       query,
       calls,
       total_time,
       mean_time
     FROM pg_stat_statements
     WHERE query ILIKE '%FROM sales%'
     ORDER BY total_time DESC
     LIMIT 5;
     ```

   - Смотри на комбинацию:
     - `total_time` — кто съел больше всего времени;
     - `mean_time` — отдельный вызов, может быть очень медленным.

3. **Возьми конкретный пример запроса.**
   - Скопируй текст запроса из статистики или логов.
   - Сократи всё лишнее (например, убери неиспользуемые поля в `SELECT`), но **не меняй логику фильтрации и `JOIN`**.

4. **Прогоняй `EXPLAIN ANALYZE` на «реальных» параметрах.**

   ```sql
   EXPLAIN (ANALYZE, BUFFERS)
   ... твой запрос ...
   ```

   - Важно запускать с теми значениями параметров (`WHERE`), которые действительно вызывают проблему (например, «последний год», а не «последняя неделя»).

5. **Ищи узкое место в плане.**
   - Найди узел, где:
     - `actual time` сильно больше соседей;
     - или `rows` фактические сильно отличаются от ожидаемых (`rows` vs `actual rows`);
     - или есть «подозрительный» `Seq Scan` по гигантской таблице.
   - Часто это `Nested Loop` по большой таблице или `Hash Join`/`Sort`, ушедший в диск (буферы, temp‑файлы).

6. **Сформулируй причину простыми словами.**
   - Примеры формулировок:
     - «Запрос делает полный проход по таблице `orders` за три года, потому что нет индекса по `created_at`.»
     - «Планировщик думает, что в таблице мало строк, и выбирает неправильный план (устаревшая статистика).»
     - «Большой `JOIN` приводит к сортировке на диске, потому что `work_mem` слишком мал и сортировка не влезает в память.»

7. **Подбери одно или несколько решений.**
   - Добавить/изменить индекс.
   - Переписать запрос (убрать функции над колонками, убрать лишние `JOIN`).
   - Обновить статистику (`ANALYZE`) или поменять настройки планировщика.
   - Изменить параметры (`work_mem`, лимитируемое кол-во строк и т.п.).

8. **Проверь результат.**
   - Повтори `EXPLAIN (ANALYZE, BUFFERS)` с новым индексом/запросом.
   - Снова посмотри в `pg_stat_statements` или slow query log, чтобы убедиться, что *среднее* и *общее* время упали.

### Практический пример: отчёт по заказам за год

**Исходная ситуация.**

Есть таблица:

```sql
CREATE TABLE orders (
  id           bigint PRIMARY KEY,
  customer_id  bigint NOT NULL,
  created_at   timestamp NOT NULL,
  amount       numeric(12,2) NOT NULL
);
```

Приложение выполняет такой запрос:

```sql
SELECT customer_id,
       SUM(amount) AS total_amount
FROM orders
WHERE created_at >= now() - interval '1 year'
GROUP BY customer_id
ORDER BY total_amount DESC
LIMIT 10;
```

Таблица уже выросла до десятков миллионов записей, и запрос стал выполняться по 20–30 секунд.

**Шаг 1. Смотрим план без индекса.**

План (упрощённо) показывает:

- `Seq Scan on orders` — полный проход по таблице;
- `Filter: (created_at >= now() - '1 year'::interval)` — фильтрация по дате;
- очень большое количество считанных страниц.

**Шаг 2. Формулируем причину.**

- «Запрос вынужден читать всю таблицу, потому что нет индекса по `created_at`, и он не может быстро найти последние записи.»

**Шаг 3. Добавляем индекс.**

```sql
CREATE INDEX CONCURRENTLY idx_orders_created_at
  ON orders (created_at);
```

Или комбинированный индекс, если часто фильтруем ещё и по `customer_id`:

```sql
CREATE INDEX CONCURRENTLY idx_orders_customer_created
  ON orders (customer_id, created_at);
```

**Шаг 4. Снова смотрим план.**

- Теперь план показывает `Index Scan` или `Bitmap Index Scan` по новому индексу.
- Считанных страниц в разы меньше, время выполнения падает, например, с 30 секунд до 1–2 секунд.

### Чек‑лист перед оптимизацией запроса

Перед тем, как что‑то менять, задай себе вопросы:

- **Я точно знаю, какой именно запрос тормозит?**
- **Я видел этот запрос в `pg_stat_statements` или slow query log?**
- **Я запускал `EXPLAIN (ANALYZE, BUFFERS)` и видел реальный план?**
- **Я понимаю, какая строка плана — узкое место?**
- **Я могу объяснить причину проблемы простым языком (без жаргона)?**
- **Я проверил результат после изменений (время, план, статистика)?**

### Проверь себя

<details>
<summary>1. В чём разница между `EXPLAIN` и `EXPLAIN ANALYZE`?</summary>

`EXPLAIN` показывает только план и оценки, не выполняя запрос.  
`EXPLAIN ANALYZE` выполняет запрос и показывает фактические времена и количество строк по каждому узлу.
</details>

<details>
<summary>2. Почему полезно сортировать `pg_stat_statements` по `total_time`, а не по `mean_time`?</summary>

Потому что нас интересует **общая нагрузка**: запрос, который выполняется быстро, но миллион раз, может быть хуже, чем медленный, но редкий. `total_time` показывает, кто в сумме больше всего нагружает систему.
</details>

### Запомните

**Оптимизация запросов начинается не с переписывания SQL, а с честного взгляда на статистику и планы выполнения.**  
Если ты не можешь показать, какой запрос и какой узел плана создаёт основную нагрузку, любая оптимизация — это угадывание.
