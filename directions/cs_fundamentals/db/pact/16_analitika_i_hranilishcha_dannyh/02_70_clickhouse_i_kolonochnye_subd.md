## §70. ClickHouse и колоночные СУБД

### 70.1. ClickHouse и движок MergeTree

**Цель раздела.**  
Разобраться, **что именно делает ClickHouse таким быстрым для аналитики**, как работает движок **MergeTree**, зачем мы при создании таблицы указываем `PARTITION BY` и `ORDER BY`, и какие грабли здесь самые частые.

---

### Сначала интуитивная картинка

Продолжим образ с таблицей продаж, но теперь представим, что все данные лежат:

- не просто в одной огромной таблице,
- а разбиты:
  - **по месяцам** (партиции),
  - и внутри каждого месяца ещё **отсортированы по определённому ключу** (например, по дате и user_id).

Образ:

- у тебя не один гигантский шкаф с документами,
- а **отдельный шкаф на каждый месяц** (партиция);
- внутри шкафа папки/стопки документов, **отсортированные по какому‑то правилу** (ключ сортировки);
- когда ты делаешь запрос «покажи мне данные за март по пользователям…», тебе:
  - не нужно лезть во все шкафы, достаточно посмотреть только шкаф «март»;
  - внутри шкафа ты быстро находишь нужные документы, потому что они **отсортированы**.

Вот это и есть идея **MergeTree**:

- таблица физически разбита на **партиции**;
- внутри партиций — на **части (parts)**, которые отсортированы по ключу `ORDER BY`;
- есть **индексы‑«пробники»**, позволяющие быстро понять, нужно ли вообще читать этот кусок.

**Схема структуры MergeTree (PARTITION BY месяц, ORDER BY date, user_id):**

```
Таблица events
│
├── Партиция 202501 (январь 2025)
│   ├── part_1 (отсортировано по date, user_id)
│   │   └── мин/макс по блокам → индекс-пробник
│   ├── part_2
│   └── ... (после INSERT появляются новые, затем merge)
│
├── Партиция 202502 (февраль 2025)
│   ├── part_1
│   └── ...
│
└── Партиция 202503 (март 2025)
    └── ...

Запрос: WHERE event_date BETWEEN '2025-03-01' AND '2025-03-31'
→ Смотрим ТОЛЬКО партицию 202503, остальные не открываем.
```

**Пример удаления старых данных:**

```sql
ALTER TABLE events DROP PARTITION '202301';  -- удалить все данные за январь 2023
```

---

### Термины (в ClickHouse‑контексте)

- **MergeTree** — базовое семейство движков хранения ClickHouse:
  - хранит данные **колоночно**;
  - делит таблицу на **партиции** и **части (parts)**;
  - внутри каждой части данные **отсортированы по ключу `ORDER BY`**;
  - периодически **сливает части** (merge), чтобы их не стало слишком много.

- **Партиция (PARTITION)** — логический/физический блок таблицы:
  - чаще всего соответствует диапазону времени (месяц/день/год);
  - операции типа «удалить/заархивировать старые данные» происходят **на уровне партиций**.

- **Часть (part)** — кусок данных внутри партиции:
  - появляется при вставке (`INSERT`);
  - внутри него строки **отсортированы** по ключу `ORDER BY`.

---

### Как живёт таблица MergeTree (по шагам)

Рассмотрим определение:

```sql
CREATE TABLE events
(
    event_date  Date,
    user_id     UInt64,
    event_type  String,
    revenue     Float64
)
ENGINE = MergeTree
PARTITION BY toYYYYMM(event_date)
ORDER BY (event_date, user_id)
SETTINGS index_granularity = 8192;
```

Разберём **каждую часть**:

1. `ENGINE = MergeTree`
   - говорим: «используй базовый движок MergeTree»;
   - на нём основаны и другие двигатели семейства (SummingMergeTree, AggregatingMergeTree и т.п.).

2. `PARTITION BY toYYYYMM(event_date)`
   - говорим: «разбивай таблицу по месяцу события»;
   - все события за январь 2025 идут в одну партицию `202501`, за февраль — в `202502` и т.д.;
   - это:
     - удобно для **управления данными**:
       - быстро удалить старые месяцы: `ALTER TABLE events DROP PARTITION 202301;`;
       - легко делать бэкапы/архивы по партициям;
     - помогает **ограничить объём чтения**:
       - если в запросе есть `WHERE event_date BETWEEN '2025-03-01' AND '2025-03-31'`,
       - можно не смотреть на партиции за другие месяцы.

3. `ORDER BY (event_date, user_id)`
   - говорим: «внутри каждой части сортируй строки по `event_date`, а при равенстве даты — по `user_id`»;
   - отсортированные данные:
     - позволяют строить **индиксы‑«пробники»**:
       - по сути, хранятся «минимум/максимум значений» по блокам;
       - если диапазон блока не попадает под `WHERE`, блок можно **не читать вообще**;
     - ускоряют запросы с фильтрами по `event_date`, `user_id` или их комбинации.

4. `index_granularity = 8192`
   - задаёт «шаг» индекса — сколько строк в одном блоке индекса;
   - компромисс:
     - меньший шаг → больше индекса, точнее отсев;
     - больший шаг → индекс компактнее, но отбрасывать блоки грубее.

---

### Что происходит при INSERT

1. При `INSERT` данные **сначала попадают в буфер** (в зависимости от настроек — память/диск).
2. Затем формируется **новая часть (part)**:
   - строки сортируются по `ORDER BY (event_date, user_id)`;
   - по каждой колонке создаются **сжатые файлы**;
   - создаются служебные файлы (индексы, метаданные).
3. В фоне запланированы задачи:
   - **слияния частей (merge)**:
     - если частей стало слишком много,
     - несколько маленьких частей объединяют в одну большую,
     - при этом:
       - сортировка по ключу сохраняется;
       - «схлопываются» дубликаты (для некоторых двигателей типа SummingMergeTree — суммируются значения).

Можно думать так:

- каждое `INSERT` создаёт «маленький отсортированный файл» внутри нужной партиции;
- фоновые процессы периодически склеивают маленькие файлы в крупные, чтобы:
  - улучшить чтение (меньше файлов → меньше накладных расходов),
  - выполнить специфическую для конкретного под‑движка логику (суммирование, агрегации и т.д.).

---

### Почему `PARTITION BY` и `ORDER BY` так важны

Эти два выражения **определяют физическую организацию данных**.

В идеале:

- `PARTITION BY`:
  - выбирается так, чтобы:
    - **соответствовать типичным диапазонным фильтрам** (например, по времени);
    - логически группировать данные (по проекту, региону и т.п.);
  - пример: `toYYYYMM(event_date)` или `toDate(event_time)`.

- `ORDER BY`:
  - выбирается так, чтобы:
    - отражать **типичные условия `WHERE`** и `GROUP BY`;
    - максимизировать эффективность индекса (отсев по ключу).
  - пример: `(event_date, user_id)` для запросов:
    - по диапазонам дат,
    - по пользователям в рамках диапазона дат.

**Неправильный выбор** часто означает:

- придётся читать намного больше данных, чем могло бы быть;
- тяжёлые запросы станут очень тяжёлыми;
- некоторые операции (удаление старых данных, управление хранением) будут болезненными.

### Проверь себя (PARTITION BY и ORDER BY в MergeTree)

1. Зачем в MergeTree выбирать `PARTITION BY` по полю даты (месяц/неделя)?  
   <details><summary>Ответ</summary>Чтобы типичные запросы с фильтром по дате могли отсекать целые партиции и не читать данные за другие периоды; это сильно сокращает объём IO и ускоряет запросы.</details>

2. Чем в ClickHouse `ORDER BY` отличается от primary key в PostgreSQL?  
   <details><summary>Ответ</summary>В ClickHouse ORDER BY задаёт ключ сортировки и используется для построения индекса (отсечение блоков по минимуму/максимуму), а не для гарантии уникальности строк. В PostgreSQL primary key — это ограничение уникальности и часто основная точка доступа.</details>

3. Что плохого в слишком мелких партициях (например, по дням при огромном потоке)?  
   <details><summary>Ответ</summary>Резко растёт количество партиций; операции слияния (merge), удаления старых данных и бэкапы становятся тяжелее и медленнее. Обычно разумнее партиционировать по месяцу или неделе.</details>

---

### Типичные ошибки при проектировании MergeTree‑таблиц

- **Ошибка 1. Выбирать `ORDER BY` «как primary key» из OLTP‑мира.**
  - В OLTP‑БД primary key часто делают чисто уникальным (`id`), не думая о типичных фильтрах.
  - В ClickHouse `ORDER BY` — это **ключ сортировки и индекс**, а не «ограничение уникальности».
  - Лучше ориентироваться на:
    - поля, по которым чаще всего фильтруют/группируют;
    - особенно временные и идентификаторы сущностей.

- **Ошибка 2. Слишком мелкие партиции.**
  - Например, `PARTITION BY toDate(event_time)` при огромном потоке событий:
    - создаётся огромное количество партиций (по одной на день),
    - операции обслуживания (merge, remove, backup) становятся тяжелее.
  - Чаще разумнее брать **месяц/неделю** как партицию.

- **Ошибка 3. Слишком крупные партиции.**
  - Например, по году при огромном трафике:
    - каждая партиция получается гигантской,
    - удалить старый месяц без трюков уже нельзя — придётся работать целым годом.

- **Ошибка 4. Игнорировать типичные запросы.**
  - Проектировать схему «в абстракции»,
  - не глядя, какие реально будут `WHERE`, `GROUP BY`, `ORDER BY`.

---

### Простыми словами / картинка в голове

MergeTree в ClickHouse — это:

- **шкафы‑партиции** (например, по месяцам),
- в каждом шкафу — **отсортированные папки** (parts) по ключу (например, дата + пользователь),
- на папках — стикеры‑«пробники» (минимум/максимум значений).

Когда ты спрашиваешь:

```sql
SELECT *
FROM events
WHERE event_date BETWEEN '2025-03-01' AND '2025-03-31'
  AND user_id = 123;
```

ClickHouse:

- сразу идёт в шкаф «март» (партиция за март),
- среди папок в этом шкафу быстро отбрасывает те, где:
  - даты вообще не попадают в нужный диапазон,
  - user_id гарантированно не встречается (по пробникам),
- читает только нужные файлы‑части и нужные колонки внутри них.

---

### Мини‑упражнения

1. У тебя таблица логов событий, большинство запросов:
   - фильтруют по дате диапазоном,
   - а внутри интересуются конкретным `user_id`.  
   Какой `PARTITION BY` и `ORDER BY` логично выбрать?

   <details><summary>Ответ</summary>Часто берут `PARTITION BY toYYYYMM(event_time)` (разбиение по месяцам) и `ORDER BY (event_time, user_id)`. Так диапазон по дате быстро отсеивает партиции и части, а внутри частей сортировка по дате+пользователю помогает отбрасывать ненужные блоки.</details>

2. Почему `PARTITION BY toDate(event_time)` может быть проблемой при очень большом потоке?  
   <details><summary>Ответ</summary>Потому что для каждого дня создаётся отдельная партиция. При большом потоке это приводит к огромному количеству партиций, усложняя обслуживание (merge, drop, backup и т.п.).</details>

3. Почему `ORDER BY id` (монотонно увеличивающийся счётчик) часто **не** лучший выбор в ClickHouse?  
   <details><summary>Ответ</summary>Потому что большинство аналитических запросов фильтруют/группируют не по искусственному `id`, а по дате, пользователю, региону и т.п. Сортировка по `id` не помогает отсекать ненужные блоки по типичным условиям `WHERE`.</details>

---

### «Проверь себя»

1. За что на самом деле отвечает `ORDER BY` в MergeTree?  
<details><summary>Ответ</summary>За физический порядок строк внутри частей и построение индекса-пробника по этому ключу. Это не ограничение уникальности, а ключ сортировки и средство ускорения фильтрации.</details>

2. Зачем вообще нужны партиции (`PARTITION BY`) в ClickHouse?  
<details><summary>Ответ</summary>Чтобы логически и физически разделить данные на крупные куски (часто по времени): легче удалять/архивировать старые данные, уменьшать объём чтения при запросах по диапазонам, управлять хранением и бэкапами.</details>

---

### Запомните

- В ClickHouse **MergeTree** — фундаментальный движок хранения для аналитики.  
- **`PARTITION BY`** и **`ORDER BY`** задают, **как физически лежат данные**, и от них во многом зависит производительность.  
- При проектировании таблиц нужно исходить из **типичных запросов**, а не только из логики предметной области.

---

### Если прочитал и не понял

Запомни так:  
**«MergeTree — это много отсортированных файлов, разложенных по папкам‑месяцам. `PARTITION BY` отвечает за папку (какой шкаф), `ORDER BY` — за порядок внутри файлов. От этого напрямую зависит, насколько быстро ClickHouse найдёт нужные строки».**

### 70.2. Primary key в ClickHouse и материализованные представления

**Цель раздела.**  
Усвоить две вещи:

1. В ClickHouse **primary key** — это **ключ сортировки и индекс**, а не «уникальный идентификатор» как в OLTP‑БД.  
2. **Материализованные представления** позволяют **считать агрегаты «на лету» при вставке**, чтобы потом отчёты летали.

---

### Primary key в ClickHouse ≠ primary key в PostgreSQL/MySQL

В классических OLTP‑БД:

- `PRIMARY KEY` означает:
  - **уникальность** (нельзя вставить две строки с одним и тем же значением PK),
  - **индекс** (по этому полю(полям) строится B‑tree‑индекс),
  - часто **кластеризацию** (физический порядок строк может соответствовать PK, но это деталь реализации).

В ClickHouse:

- под «primary key» обычно понимают:
  - **ключ сортировки** (`ORDER BY`) и **разреженный индекс** по нему;
  - **никакой гарантии уникальности нет**;
  - можно вставить сколько угодно строк с одинаковым набором значений primary key.

То есть в ClickHouse «primary key» отвечает за **физический порядок и индекс**, а не за **целостность данных**.

Если нужна реальная уникальность:

- её приходится обеспечивать:
  - на уровне **источника данных**,
  - или через **дополнительные таблицы/механизмы**,
  - или через периодические процедуры дедупликации (для некоторых сценариев).

---

### Материализованные представления: посчитать один раз и потом читать быстро

Идея:

- есть **сырые события** (`raw_events`);
- нам регулярно нужны **агрегаты** (например, дневная выручка);
- мы можем:
  - каждый раз при отчёте считать `SUM(revenue)` по `raw_events` (тяжело),
  - или **подсчитывать агрегаты заранее**, по мере поступления данных.

Материализованное представление в ClickHouse:

- «подписывается» на вставки в исходную таблицу;
- для каждой вставляемой порции данных:
  - прогоняет их через `SELECT`‑запрос;
  - результат пишет в **другую таблицу** (обычно аггрегирующую).

---

### Пример, но очень подробно

**Исходная таблица событий:**

```sql
CREATE TABLE raw_events
(
    event_time DateTime,
    user_id    UInt64,
    event_type String,
    revenue    Float64
)
ENGINE = MergeTree
PARTITION BY toYYYYMM(event_time)
ORDER BY (event_time, user_id);
```

Здесь мы просто храним **сырые события**:

- время события,
- пользователя,
- тип события,
- выручку.

**Таблица для дневной выручки:**

```sql
CREATE TABLE daily_revenue
(
    day     Date,
    revenue Float64
)
ENGINE = SummingMergeTree
ORDER BY day;
```

Эта таблица:

- хранит **уже агрегированные данные**:
  - одна строка на день (`day`),
  - сумма выручки за этот день (`revenue`);
- использует `SummingMergeTree`:
  - это под‑движок MergeTree, который:
    - при слиянии частей **складывает** значения по одинаковым ключам (`day`),
    - удобно для кумулятивных сумм.

**Материализованное представление:**

```sql
CREATE MATERIALIZED VIEW mv_daily_revenue
TO daily_revenue
AS
SELECT
    toDate(event_time) AS day,
    SUM(revenue)       AS revenue
FROM raw_events
GROUP BY day;
```

Что здесь происходит:

- Мы говорим: «создать материализованное представление `mv_daily_revenue`».
- `TO daily_revenue` — результат записывай **в таблицу `daily_revenue`**.
- Тело запроса:
  - берёт новые строки из `raw_events`,
  - превращает `event_time` в дату (`toDate`),
  - группирует по дню и считает сумму `revenue`.

При `INSERT` в `raw_events`:

1. Строки попадают в `raw_events` (сырые данные).
2. Тут же те же строки «прокатываются» через материализованное представление:

**Схема потока данных (материализованное представление):**

```
INSERT INTO raw_events (event_time, user_id, event_type, revenue)
VALUES ('2025-03-15 10:00', 1, 'purchase', 100),
       ('2025-03-15 11:30', 2, 'purchase', 250),
       ('2025-03-15 14:00', 1, 'purchase', 50);

                    ┌─────────────────┐
                    │   raw_events    │
                    │  (сырые данные) │
                    └────────┬────────┘
                             │
              ┌──────────────┼──────────────┐
              │              │              │
              ▼              ▼              ▼
        Строка 1       Строка 2       Строка 3
              │              │              │
              └──────────────┼──────────────┘
                             │
                             ▼
                    ┌─────────────────┐
                    │  mv_daily_      │
                    │  revenue        │  GROUP BY day, SUM(revenue)
                    │  (MV)           │
                    └────────┬────────┘
                             │
                             ▼
                    ┌─────────────────┐
                    │  daily_revenue  │
                    │  day='2025-03-15', revenue=400
                    └─────────────────┘
```

   - считаются суммы по дням;
   - результат записывается в `daily_revenue`.

**Итог:**

- `daily_revenue` всегда содержит **актуальные (с учётом задержки вставки) дневные суммы**;
- чтобы получить выручку по дням, достаточно:

```sql
SELECT *
FROM daily_revenue
ORDER BY day;
```

— это очень быстрый запрос, потому что:

- строк в `daily_revenue` **гораздо меньше**, чем в `raw_events`;
- `SummingMergeTree` упаковывает и сливает данные эффективно.

---

### Почему это так сильно ускоряет аналитику

Если бы мы каждый раз считали отчёт по `raw_events`:

- приходилось бы:
  - сканировать миллионы/миллиарды строк;
  - снова делать `GROUP BY`;
  - снова считать `SUM(revenue)`.

С материализованным представлением:

- тяжёлая работа делается:
  - **частями, по мере поступления данных**;
  - а не одним большим запросом;
- итог храним в более компактном виде в `daily_revenue`.

Отчёты превращаются в:

- «почитать одну относительно маленькую таблицу»,
- а не «пересчитать всё с нуля».

---

### В чём подвох и о чём нужно помнить

- **Ошибка 1. Считать, что MV «волшебно исправит» прошлые данные.**
  - Если ты **задним числом исправил строки** в `raw_events`,  
    агрегаты в `daily_revenue` **сами по себе не пересчитаются**.
  - Нужно:
    - либо специально перерасчёт делать,
    - либо аккуратно проектировать поток данных (чтобы исправления шли отдельным этапом).

- **Ошибка 2. Изменить структуру исходной таблицы и забыть про MV.**
  - Если ты изменил схему `raw_events` (новые столбцы, типы),
  - нужно проверить, **не сломало ли это** определение MV.

- **Ошибка 3. Пытаться в MV ставить слишком сложную бизнес‑логику.**
  - MV хороши для:
    - простых агрегатов (суммы, счётчики, максимум/минимум);
    - относительно простой логики.
  - Сложные кейсы лучше оставлять ETL/ELT‑конвейеру.

### Проверь себя (материализованные представления)

1. Почему при исправлении старых строк в исходной таблице агрегаты в MV «сами не обновятся»?  
   <details><summary>Ответ</summary>MV считает агрегаты только при вставке новых данных в исходную таблицу. Изменения или удаления уже обработанных строк не триггерят пересчёт; для обновления нужен отдельный процесс перерасчёта или повторная загрузка.</details>

2. Зачем в MV использовать простую логику (суммы, счётчики), а сложную оставлять ETL?  
   <details><summary>Ответ</summary>MV выполняются при каждой вставке; сложная логика увеличивает задержку и нагрузку. Простые агрегаты дают предсказуемую производительность; сложные преобразования и джойны лучше делать в отдельном конвейере (ETL/ELT).</details>

3. В чём выигрыш по производительности: читать MV вместо сырой таблицы?  
   <details><summary>Ответ</summary>В MV уже посчитаны агрегаты и записано меньше строк; отчёт читает компактную таблицу вместо полного скана сырых данных и не пересчитывает GROUP BY при каждом запросе.</details>

---

### Простыми словами / образ

Представь:

- у тебя есть **коробка с мельчайшими монетами** (сырые события);
- тебя интересует, **сколько денег в каждой дневной копилке**.

Подход без MV:

- каждый раз, когда спрашивают «сколько денег в копилке за день X»,  
  ты пересчитываешь все монетки за этот день в большой общей коробке.

Подход с MV:

- ты завёл **отдельные баночки по дням**;
- как только тебе приносят монетки за какой‑то день, ты **сразу раскладываешь их по баночкам**;
- когда тебя спрашивают про день X, ты просто **смотришь на банку «день X»**.

MV — это про «раскладывать монеты по баночкам сразу, а не пересчитывать всё каждый раз».

---

### Мини‑упражнения

1. Почему primary key в ClickHouse не гарантирует уникальность?  
   <details><summary>Ответ</summary>Потому что он используется как ключ сортировки и индекс для отсечения блоков, а не как ограничение целостности. ClickHouse не проверяет, есть ли уже строка с таким же набором значений PK.</details>

2. В чём главная идея материализованного представления с точки зрения производительности?  
   <details><summary>Ответ</summary>Посчитать тяжёлые агрегаты один раз (или небольшими порциями при вставке) и сохранить результат в отдельной таблице, чтобы потом отчёты читали уже предагрегированные данные, а не пересчитывали всё с нуля.</details>

3. Что произойдёт с агрегатами, если ты задним числом перепишешь старые строки в `raw_events`?  
   <details><summary>Ответ</summary>MV само по себе не пересчитает старые агрегаты. Данные в `daily_revenue` останутся как были, пока ты не организуешь перерасчёт вручную (например, пересоздание части данных или повторную загрузку).</details>

---

### «Проверь себя»

1. В чём главное отличие primary key в ClickHouse от primary key в типичной OLTP‑БД?  
<details><summary>Ответ</summary>В ClickHouse primary key — это ключ сортировки и индекс для ускорения чтения; он не навязывает уникальность и не является ограничением целостности.</details>

2. Зачем использовать материализованное представление для агрегаций?  
<details><summary>Ответ</summary>Чтобы «посчитать один раз при вставке» и хранить предагрегированные данные в отдельной таблице, уменьшив объём работы при каждом аналитическом запросе.</details>

### 70.3. Обзор Snowflake, BigQuery, Redshift, DuckDB

**Цель раздела.**  
Не выучить наизусть все фичи облачных платформ, а **построить в голове простую карту**, чем вообще отличаются типовые аналитические решения: Snowflake, BigQuery, Redshift, DuckDB — и увидеть, что у всех них внутри похожие идеи.

---

### Общие идеи, которые повторяются почти везде

Практически во всех современных аналитических платформах ты увидишь:

- **Колоночное хранение** — данные по столбцам, а не по строкам.
- **MPP/распределённое исполнение** — запрос разбивается на подзадачи по узлам.
- **Разделение хранения и вычислений** (особенно в облаке):
  - данные живут в «дешёвом» и надёжном хранилище;
  - вычислительные кластеры можно:
    - поднимать/гасить,
    - масштабировать вверх/вниз.
- **Тесную интеграцию с файловыми форматами и lake**:
  - Parquet, ORC и т.п.;
  - чтение данных из облачного стораджа (S3, GCS и т.д.).

Разница — в деталях:

- как именно масштабируются кластеры;
- как считаются деньги (за хранение, за запросы, за время работы кластера);
- какие есть дополнительные фичи (time travel, безопасность, интеграции).

---

### Snowflake — «склады» поверх общего хранилища

Мысленная модель:

- есть **единое хранилище данных** (data storage);
- вокруг него:
  - множество **виртуальных складов (warehouses)** — пулы вычислительных ресурсов.

Характерные особенности:

- **Разделение хранения и вычислений.**
  - Данные лежат в одном месте.
  - Для каждой команды/нагрузки можно поднять **свой склад**:
    - склад для прод‑аналитики,
    - склад для дата‑сайентистов,
    - склад для отчётов топ‑менеджмента.
  - Склады можно:
    - независимо масштабировать (S/M/L по мощности),
    - останавливать (экономить деньги, когда не используются).

- **Оплата**:
  - обычно:
    - отдельно за хранение данных,
    - отдельно за «кредиты» вычислений (время работы складов).

Образ:

- у тебя есть **единый мегасклад данных**;
- к нему могут подъезжать разные **погрузчики разной мощности** (warehouses):
  - маленький погрузчик для лёгких задач;
  - тяжёлый для массивных отчётов;
  - каждый платится по времени работы.

---

### BigQuery — «серверless»‑аналитика

BigQuery в голове можно представить так:

- ты **просто пишешь SQL**,
- а платформа:
  - сама решает, сколько ресурсов выделить;
  - сама масштабируется;
  - сама управляет кластерами.

Ты в меньшей степени управляешь:

- конкретным размером кластера;
- количеством нод;
- явно не поднимаешь/не гасишь «склады».

Характерная фишка:

- **оплата за объём сканированных данных** (в классической модели):
  - чем больше байт/строк таблиц было прочитано при запросе,
  - тем дороже запрос.

Отсюда важное практическое следствие:

- нужно писать запросы так, чтобы:
  - читать как можно **меньше данных**;
  - явно выбирать только нужные столбцы;
  - по возможности фильтровать и партиционировать.

Образ:

- ты приходишь в библиотеку и говоришь: «Дай мне ответ на такой‑то вопрос»,  
  библиотекарь сам разбирается, сколько сотрудников и полок ему поднять;  
  ты платишь не за количество сотрудников, а, условно, за количество просмотренных страниц.

---

### Redshift — «кластер» от AWS + интеграция с S3

Redshift — это:

- **кластерная колоночная СУБД** в AWS;
- сильно напоминает классический MPP‑кластер:
  - ты создаёшь кластер с определённым количеством/типом нод;
  - сам думаешь об их размере и масштабировании.

Особенность:

- **Redshift Spectrum**:
  - позволяет выполнять запросы к данным, лежащим прямо в **S3** (облачное файловое хранилище);
  - то есть можно:
    - часть данных держать **в самом Redshift** (часто используемое, горячее);
    - часть — **в lake на S3** (более сырое/архивное);
  - и в одном запросе обращаться и туда, и туда.

Образ:

- у тебя есть:
  - собственный «мини‑склад» (кластер Redshift),
  - и огромное внешнее хранилище (S3);
  - иногда тебе нужно что‑то взять сразу из обоих.

---

### Snowflake vs Redshift — в чём разница (очень коротко)

| Характеристика | Snowflake | Redshift |
|----------------|-----------|----------|
| **Модель ресурсов** | Виртуальные склады (warehouses): пулы вычислений, которые можно независимо поднимать/гасить под разные команды и задачи | Фиксированный кластер из нод: размер и тип нод задаёшь заранее, масштабируешь кластер целиком |
| **Оплата** | Отдельно за хранение и за «кредиты» вычислений (время работы складов); можно останавливать склады, чтобы не платить за простаивание | Обычно платишь за работающий кластер (тип и количество нод), даже если нагрузка мала; Spectrum — отдельные тарифы за запросы к S3 |
| **Управление масштабированием** | «Чёрный ящик» + удобный интерфейс: можно быстро увеличить/уменьшить размер склада, добавить независимые склады | Больше «ручного» администрирования кластера: нужно продумывать тип нод, распределение данных, масштабирование |
| **Хранение данных** | Данные лежат в общем хранилище Snowflake; склады лишь «подъезжают» считать | Данные живут в самом кластере Redshift; через Spectrum можно читать из S3 как внешний слой |
| **Типичный сценарий** | Много разных команд/нагрузок, нужно гибко поднимать/гасить ресурсы под каждую, минимизируя простои | Команда готова управлять кластером, плотно интегрирована с AWS и хочет MPP‑кластер с доступом к S3 через Spectrum |

Очень грубо: **Snowflake** больше ощущается как «аналитический сервис с гибкими складами», а **Redshift** — как «кластерная БД в AWS», которой ты управляешь как обычным MPP‑кластером.

---

### DuckDB — «SQLite, но для аналитики»

DuckDB удобно понимать так:

- это «**локальная аналитическая БД**»:
  - работает как библиотека (встраивается в приложение),
  - может запускаться прямо на ноутбуке;
  - отлично дружит с файлами (Parquet, CSV).

Чем она полезна:

- исследование данных **локально**, без разворачивания больших кластеров;
- прототипирование аналитики;
- интеграция с Python/R/Jupyter‑ноутбуками:
  - `SELECT * FROM 'file.parquet' WHERE ...` прямо из ноутбука.

Образ:

- если ClickHouse/Snowflake/BigQuery — это **большие складские комплексы**,  
  то DuckDB — это **очень умный карманный калькулятор и мини‑склад** у тебя на ноутбуке.

---

### Что общего у всех этих платформ

У всех четырёх (и многих других) ты почти наверняка увидишь:

- колоночное хранение;
- MPP/распределённое исполнение (кроме DuckDB, где всё локально, но всё равно векторное исполнение и колоночность);
- работу с файловыми форматами (Parquet, ORC и др.);
- сильную завязку на облачную инфраструктуру (кроме DuckDB, который наоборот хорош локально).

Разница в том:

- **насколько ты управляешь кластерами** (Snowflake/BigQuery — больше «сервис», Redshift — больше «кластер», DuckDB — «локальная библиотека»);
- **как платишь** (время работы складов, объём сканированных данных, размер кластера и т.п.);
- **какие дополнительные инструменты** есть вокруг (оркестрация, интеграции, BI‑слой, безопасность).

---

### Мини‑упражнения

1. Тебе нужно дать аналитикам возможность крутить тяжёлые запросы, но не хочешь, чтобы они думали про размер кластера. Что больше похоже на твою задачу — BigQuery или Redshift?  
   <details><summary>Ответ</summary>Ближе BigQuery. Там «серверless»‑подход: аналитики просто пишут SQL, а платформа сама масштабируется, а биллинг чаще всего зависит от объёма сканированных данных, а не от размера кластера.</details>

2. У тебя есть 2–3 больших Parquet‑файла на локальном диске, и ты хочешь быстро поэкспериментировать с запросами, не поднимая кластеры. Что логичнее — DuckDB или Snowflake?  
   <details><summary>Ответ</summary>DuckDB. Его можно запустить локально, подключить файлы Parquet и писать SQL прямо в ноутбуке без облачной инфраструктуры.</details>

3. Какое общее свойство по хранению данных объединяет Snowflake, BigQuery и Redshift?  
   <details><summary>Ответ</summary>Все они используют колоночное хранение (column‑store), оптимизированное под аналитические запросы и агрегации по большим объёмам данных.</details>

### Проверь себя (облачные аналитические платформы)

1. В чём основная идея разделения хранения и вычислений в Snowflake?  
   <details><summary>Ответ</summary>Данные лежат в общем хранилище; «склады» (warehouses) — это пулы вычислений, которые поднимают под конкретные задачи. Можно масштабировать и останавливать склады независимо от данных, не переливая их.</details>

2. Чем BigQuery отличается от Redshift с точки зрения управления ресурсами?  
   <details><summary>Ответ</summary>В BigQuery пользователь почти не управляет кластерами — платформа сама масштабируется, оплата часто за объём сканированных данных. В Redshift ты явно задаёшь размер и тип кластера и платишь за его работу.</details>

3. Когда логичнее выбрать DuckDB, а не Snowflake/BigQuery?  
   <details><summary>Ответ</summary>Когда нужна локальная аналитика: работа с файлами Parquet/CSV на ноутбуке, прототипирование, интеграция с Python/R без облачной инфраструктуры и без развёртывания кластеров.</details>

---

### «Проверь себя»

1. В чём основная идея разделения хранения и вычислений в Snowflake‑подобных системах?  
<details><summary>Ответ</summary>Данные хранятся централизованно и относительно независимо от вычислительных ресурсов. Для разных задач/команд поднимаются отдельные «склады» (пулы вычислений), которые можно независимо масштабировать и останавливать, не трогая сами данные.</details>

2. Чем BigQuery отличается от классического «кластерного» подхода вроде Redshift с точки зрения пользователя?  
<details><summary>Ответ</summary>В BigQuery пользователь в основном пишет SQL и почти не управляет кластерами; платформа автоматически масштабирует ресурсы, а оплата часто привязана к объёму сканированных данных. В Redshift ты явнее управляешь кластером (размер, тип нод) и платишь за его работу.</details>

---

### Запомните

- Snowflake, BigQuery, Redshift, DuckDB — разные по форме решения, но внутри у всех **колоночность, MPP (кроме локального DuckDB) и работа с файлами/lake**.  
- Выбор между ними — это часто вопрос:
  - **инфраструктуры** (GCP, AWS, on‑prem, локальный ноутбук),
  - **модели оплаты**,
  - **удобства для команды** (нужна ли «полная магия» или хочется ручного контроля).

### Если прочитал и не понял

Запомни кратко:  
**«Snowflake/BigQuery/Redshift — это большие облачные аналитические склады, у каждого свой стиль управления ресурсами и биллинг, но все они колоночные и MPP. DuckDB — это их «младший брат» для локальной аналитики прямо из ноутбука».**
