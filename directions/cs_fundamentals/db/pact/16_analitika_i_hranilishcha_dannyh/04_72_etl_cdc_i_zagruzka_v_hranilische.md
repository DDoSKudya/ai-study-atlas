## §72. ETL, CDC и загрузка в хранилище

### 72.1. ETL и ELT

**Цель раздела.**  
Понять разницу между подходами **ETL** и **ELT**, их плюсы и минусы, и уметь выбрать подход под задачу.

**Одна фраза — вся суть §72.**  
Данные из OLTP надо **доставить** в хранилище: ETL = вытащил → почистил/преобразовал снаружи → загрузил готовое; ELT = вытащил и загрузил сырое → чистишь и преобразуешь уже в хранилище. CDC = ловишь только изменения (INSERT/UPDATE/DELETE) и подливаешь их — без перелива целых таблиц. Data lake = файлы (Parquet и т.д.); Delta/Iceberg добавляют к файлам транзакции и версии.

---

**Термины**

- **ETL (Extract‑Transform‑Load)** — извлекаем данные из источников, **преобразуем** их во внешнем движке (скрипты, Spark и т.п.) и загружаем уже очищенные и агрегированные данные в хранилище.
- **ELT (Extract‑Load‑Transform)** — сначала загружаем **сырые данные** в хранилище/lake, а уже там преобразуем с помощью SQL/движка (Snowflake, BigQuery, Spark и т.д.).

**Схема потока: ETL vs ELT**

```
ETL (Extract → Transform → Load):
──────────────────────────────────────────────────────────────────
  OLTP‑БД          Внешний движок              Хранилище
  (PostgreSQL)     (Spark, Python, Airflow)    (ClickHouse, Snowflake)
       │                     │                        │
       │  Extract            │                        │
       │  (выгрузка)         │                        │
       ├────────────────────▶                        │
       │                     │  Transform             │
       │                     │  (очистка, джойны,     │
       │                     │   агрегаты)            │
       │                     │                        │
       │                     │  Load                  │
       │                     ├───────────────────────▶│
       │                     │  (готовые таблицы)     │
       ▼                     ▼                        ▼


ELT (Extract → Load → Transform):
──────────────────────────────────────────────────────────────────
  OLTP‑БД          Хранилище / Lake           Хранилище
  (PostgreSQL)     (сырые данные)             (витрины)
       │                     │                        │
       │  Extract + Load     │                        │
       │  (выгрузка «как есть»)                       │
       ├────────────────────▶                        │
       │                     │  Transform             │
       │                     │  (SQL внутри           │
       │                     │   хранилища)           │
       │                     ├───────────────────────▶│
       │                     │  raw → cleaned → mart  │
       ▼                     ▼                        ▼
```

**Сравнение (в упрощении)**:

- ETL:
  - Плюсы: хранилище получает уже «красивые» данные; можно разгрузить БД от тяжёлых преобразований.
  - Минусы: ETL‑конвейер становится сложным, обновление логики требует деплоя; хуже поддержка ad‑hoc кейсов по «сырым» данным.
- ELT:
  - Плюсы: проще сначала всё сложить в lake/хранилище и потом разными SQL‑представлениями строить разные витрины.
  - Минусы: больше требований к самому хранилищу (оно должно тянуть тяжёлые преобразования); нужно следить, чтобы сырые данные не превратились в «болото».

---

### Образ: кухня и заготовочный цех

Представь ресторан.

- **ETL**:
  - у тебя есть **отдельный заготовочный цех**:
    - там чистят овощи, варят бульоны, подготавливают полуфабрикаты;
  - в основной кухне (хранилище) уже:
    - только **собирают блюда** из подготовленных ингредиентов.

- **ELT**:
  - ты складываешь **все продукты прямо на кухню** (сырые данные в хранилище/lake);
  - повара (движок БД) сами:
    - чистят, режут, жарят и т.д., когда это нужно для конкретного блюда (витрины).

Где что удобно:

- если кухня слабая, а заготовочный цех мощный → **ETL**;
- если кухня очень мощная (Snowflake/BigQuery и т.п.), а заготовочного цеха нет → часто проще **ELT**.

---

### Простой пример ETL‑конвейера

1. **Extract (E)**:
   - каждый час выгружаем новые/изменённые заказы из OLTP‑БД в CSV/Parquet или сразу в промежуточную БД;
2. **Transform (T)**:
   - в отдельном скрипте/джобе:
     - чистим данные (убираем дубликаты, приводим типы, нормализуем значения);
     - агрегируем (например, считаем промежуточные суммы);
3. **Load (L)**:
   - загружаем уже подготовленные таблицы/файлы в хранилище:
     - `fact_orders_clean`,
     - `dim_customer_clean` и т.п.

Хранилище видит **уже более‑менее аккуратные данные**.

---

### Простой пример ELT‑конвейера

1. **Extract + Load (EL)**:
   - «как есть» вываливаем сырые данные в:
     - `raw_orders`,
     - `raw_customers`,
     - `raw_events` (часто прямо в lake/хранилище).
2. **Transform (T)**:
   - уже в самом хранилище (Snowflake/BigQuery/ClickHouse и т.п.):
     - пишем SQL:
       - `INSERT INTO fact_orders SELECT ... FROM raw_orders JOIN raw_customers ...`,
       - `CREATE VIEW clean_orders AS SELECT ... FROM raw_orders WHERE ...;`
   - строим слои:
     - `raw` (сырые),
     - `cleaned` (очищенные),
     - `mart` (витрины).

Плюс:

- всегда есть доступ к **сырым данным** для новых идей аналитиков;
- можно быстро строить новые витрины, не трогая исходные потоки.

Минус:

- нужно дисциплинированно следить, чтобы:
  - сырые таблицы не превратились в «свалку»;
  - был внятный слой `cleaned`/`core`, на который опираются витрины.

---

### Мини‑упражнения

1. У тебя мощный кластер Snowflake, а внешние скрипты на Python работают на небольших машинах. К какому подходу ты интуитивно будешь склоняться — ETL или ELT?  
   <details><summary>Ответ</summary>К ELT: проще загрузить сырые данные в Snowflake и делать тяжёлые преобразования там, используя мощь кластера, чем пытаться тянуть всё на внешних слабых машинах.</details>

2. Почему при ELT важно чётко разделять слои `raw` и `cleaned`?  
   <details><summary>Ответ</summary>Чтобы сырые данные не смешались с очищенными, и все отчёты строились поверх согласованного «чистого» слоя. Иначе легко запутаться, какие таблицы использовать, и получить противоречивую аналитику.</details>

3. В каком случае ETL предпочтительнее ELT, даже если хранилище мощное?  
   <details><summary>Ответ</summary>Когда преобразования требуют логики, которую сложно или неудобно реализовать в SQL хранилища (сложная валидация, вызов внешних API, обогащение из не‑SQL источников), или когда нужно не загружать в хранилище «грязные» данные по соображениям безопасности или порядка.</details>

### Проверь себя (ETL и ELT)

1. Чем принципиально отличается порядок операций в ETL и в ELT?  
   <details><summary>Ответ</summary>В ETL сначала извлекаем данные, затем преобразуем их во внешнем движке и только потом загружаем готовый результат в хранилище. В ELT сначала извлекаем и загружаем сырые данные в хранилище/lake, а преобразования делаем уже внутри хранилища.</details>

2. Когда удобнее выбирать ELT?  
   <details><summary>Ответ</summary>Когда хранилище достаточно мощное (Snowflake, BigQuery и т.п.), хочется иметь доступ к сырым данным для ad‑hoc аналитики и быстрого создания новых витрин без изменения внешних конвейеров.</details>

3. Какой минус у подхода «всё сырое в хранилище» без дисциплины слоёв?  
   <details><summary>Ответ</summary>Сырые таблицы могут превратиться в «свалку»; непонятно, какие данные актуальны и на что опираться. Нужен внятный слой cleaned/core и правила, чтобы витрины строились только поверх него.</details>

### 72.2. CDC и инкрементальная загрузка

**Цель раздела.**  
Понять, как **CDC (Change Data Capture)** помогает делать инкрементальные загрузки в хранилище и не переливать таблицы целиком.

**В этом разделе главное.**  
1) CDC позволяет получать **поток изменений** из источников (INSERT/UPDATE/DELETE).  
2) Эти изменения можно транслировать в хранилище или lake почти в реальном времени.  
3) Инкрементальная загрузка снижает объём передаваемых данных и ускоряет обновление витрин.

#### Термины

- **CDC** — набор техник (логическая репликация, чтение журнала транзакций, триггеры), позволяющих экспортировать изменения таблиц.
- **Debezium, Kafka Connect** — популярные инструменты для CDC поверх Kafka.

#### Теория

- Вместо ежедневного `TRUNCATE + INSERT SELECT ...` по всей таблице, мы:
  - Подписываемся на **лог журнала** (binlog, WAL) или используем триггеры.
  - Получаем события вида: `INSERT row`, `UPDATE old_row → new_row`, `DELETE row`.
  - Применяем их на стороне хранилища/витрин.
- Для PostgreSQL:
  - Используется **логическое декодирование WAL**.
  - Можно строить поток изменений в Kafka и далее грузить в хранилище (например, ClickHouse, Snowflake, BigQuery).
  - Концептуально это так: PostgreSQL пишет все изменения в журнал WAL; «логический декодер» (pgoutput, wal2json и др.) читает этот журнал и превращает записи в события `INSERT/UPDATE/DELETE`, которые можно отдавать наружу (Kafka, коннекторы и т.п.).
  - Важно запомнить именно идею: **«PostgreSQL умеет превращать WAL в поток бизнес‑изменений»**, а конкретные плагины и настройки можно разбирать уже отдельно по документации.

#### Почему здесь внезапно важны метаданные

CDC-пайплайн живёт не только на самих строках данных, но и на **системных метаданных**:

- какая версия схемы у источника и у приёмника;
- какие колонки появились, исчезли или сменили тип;
- до какого момента по времени или offset'у данные уже догружены;
- насколько свежи витрины и нет ли schema drift между OLTP и warehouse.

Именно поэтому рядом с ETL/CDC почти всегда живут служебные таблицы, catalog-views, freshness-checks и schema-registry-подобные механики. Эта линия глубже продолжается в `part-29-system-views-and-metadata.md`.

#### Инкрементальная загрузка

- В более простых случаях можно использовать **метку времени** или **инкрементальный ключ**:
  - `WHERE updated_at > last_successful_load_at`.
  - Но это требует корректного ведения `updated_at` в источнике и поддержки «маркера».

---

### Полная перезагрузка vs инкрементальная дельта

**Полная перезагрузка (full reload).**

- Шаблон: `TRUNCATE target_table; INSERT INTO target_table SELECT ... FROM source;`
- Плюсы:
  - просто реализовать;
  - легко рассуждать о целостности (все данные пересчитаны).
- Минусы:
  - каждый раз гоняем **всю таблицу**;
  - высокая нагрузка на источники и сеть;
  - долгие окна обновления; сложно делать часто.

**Инкрементальная дельта (incremental load).**

- Шаблон: «загружаем только новые/изменённые строки»:
  - по `updated_at`/`id`,
  - или по CDC‑потоку (binlog/WAL/триггеры).
- Плюсы:
  - передаём **только изменения**, а не весь объём;
  - можно обновлять витрины гораздо чаще;
  - меньше нагрузка на прод‑системы.
- Минусы:
  - нужна аккуратная логика маркеров (`last_successful_load_at`) или надёжный CDC;
  - сложнее отлаживать, чем «выгрузить всё заново».

Запомни в одном предложении: **полная перезагрузка — просто, но тяжело и редко; инкрементальная дельта — сложнее в настройке, но сильно экономит ресурсы и позволяет обновлять данные часто.**

---

### Образ: переписывать всю книгу vs дописывать только новые страницы

Без CDC и инкрементальной загрузки часто делают так:

- каждый день:
  - **полностью выгружают таблицу** (например, `orders`);
  - заливают её заново в хранилище.

Образ:

- у тебя есть большая книга, и каждый день ты:
  - вырываешь все страницы из копии;
  - переписываешь всю книгу заново.

С CDC:

- ты каждый день:
  - дописываешь **только новые страницы**;
  - помечаешь, какие строки изменились или удалились.

---

**Схема CDC‑пайплайна (от источника до хранилища):**

```
  PostgreSQL              Debezium /              Kafka                Коннектор /
  (OLTP)                  Kafka Connect           (топики)             Джоба
       │                        │                      │                    │
       │  INSERT/UPDATE/DELETE  │                      │                    │
       │  (транзакции)         │                      │                    │
       │                        │                      │                    │
       │  WAL (журнал)          │                      │                    │
       ├───────────────────────▶                      │                    │
       │  Логический декодер   │  События              │                    │
       │  (pgoutput, wal2json) │  {op, table, after}   │                    │
       │                        ├──────────────────────▶                    │
       │                        │                      │  Читаем топик      │
       │                        │                      ├───────────────────▶│
       │                        │                      │                    │
       │                        │                      │  Применяем к       │
       │                        │                      │  таблицам          │
       │                        │                      │  хранилища         │
       │                        │                      │                    ├───▶ ClickHouse
       ▼                        ▼                      ▼                    │    / Snowflake
                                                                             │    / BigQuery
                                                                             ▼
```

**Пример события CDC (JSON):**

```json
{"op": "u", "ts_ms": 1709123456789, "source": {"db": "shop", "table": "orders"},
 "before": {"id": 1, "status": "new", "updated_at": "2025-03-01 10:00:00"},
 "after":  {"id": 1, "status": "paid", "updated_at": "2025-03-01 10:05:00"}}
```

### Типичный сценарий CDC + хранилище

1. В OLTP‑БД (например, PostgreSQL):
   - включаем логическую репликацию или используем Debezium/Kafka Connect;
   - получаем поток событий:
     - `{"op": "c", "table": "orders", "after": {...}}` — create (INSERT),
     - `{"op": "u", "table": "orders", "before": {...}, "after": {...}}` — update,
     - `{"op": "d", "table": "orders", "before": {...}}` — delete.
2. Эти события кладём:
   - либо в Kafka (топики по таблицам),
   - либо в очереди/стримы облака.
3. Коннектор/джоба читает события и:
   - применяет их к таблицам в хранилище:
     - вставляет новые строки,
     - обновляет изменённые,
     - помечает удалённые.

Итог:

- хранилище получает **приближённо реальное время** по данным из OLTP;
- нет необходимости переливать всю таблицу каждый раз.

---

### Мини‑упражнения

1. В чём основное преимущество CDC перед полным ежедневным `TRUNCATE + INSERT SELECT ...`?  
   <details><summary>Ответ</summary>CDC передаёт только изменения (INSERT/UPDATE/DELETE), а не всю таблицу. Это снижает объём данных, уменьшает нагрузку на источники и ускоряет обновление хранилища.</details>

2. Почему простая инкрементальная загрузка по `updated_at` не всегда надёжна?  
   <details><summary>Ответ</summary>Потому что `updated_at` может обновляться не всегда корректно (ручные правки, пропущенные обновления), бывают задержки и часовые пояса. Плюс при смене логики легко забыть обновить метку. CDC учитывает реальные изменения на уровне журнала транзакций.</details>

3. Зачем в CDC‑пайплайне часто ставят Kafka между источником и хранилищем?  
   <details><summary>Ответ</summary>Kafka даёт буфер и надёжную доставку: события из источника не теряются при временной недоступности хранилища, можно масштабировать потребителей и переигрывать поток при сбоях.</details>

### Проверь себя (CDC и инкрементальная загрузка)

1. В чём основное преимущество CDC перед полной перезагрузкой таблицы?  
   <details><summary>Ответ</summary>Передаётся только поток изменений (INSERT/UPDATE/DELETE), а не вся таблица; снижается нагрузка на источник и сеть, обновление хранилища можно делать чаще и ближе к реальному времени.</details>

2. Что означает поле `op` в CDC‑событии и какие значения оно обычно принимает?  
   <details><summary>Ответ</summary>op — тип операции: «c» (create/INSERT), «u» (update), «d» (delete). По нему коннектор понимает, вставить новую строку, обновить существующую или пометить удаление.</details>

3. Когда инкрементальная загрузка по `updated_at` всё же приемлема?  
   <details><summary>Ответ</summary>Когда в источнике строго ведётся updated_at при каждом изменении, нет ручных правок без обновления метки и допустима небольшая задержка или пропуск части изменений; при этом не нужна near real-time синхронизация.</details>

### 72.3. Data lake, Delta и Iceberg

**Цель раздела.**  
Понять, что такое **data lake**, чем он отличается от классического хранилища и как форматы **Delta Lake** и **Apache Iceberg** добавляют **ACID и версионность**.

**Термины**

- **Data lake** — хранилище файлов (обычно в объектном сторидже вроде S3) с сырыми и обработанными данными.
- **Parquet** — колоночный формат файлов, часто используемый в lake.
- **Delta Lake, Apache Iceberg** — табличные форматы поверх файлового хранилища, которые добавляют:
  - транзакции (ACID),
  - версионность таблиц,
  - удобное управление партициями и эволюцией схемы.

**Идея.**  
Вместо того чтобы хранить всё в «чистой» РСУБД, мы храним данные как файлы (Parquet) в lake, а Delta/Iceberg обеспечивают:

- атомарные операции `INSERT/UPDATE/DELETE/MERGE`,
- возможность откатиться к предыдущей версии таблицы (`time travel`),
- контроль над партиционированием и чисткой старых файлов.

**Схема Data Lake и слоёв Delta/Iceberg:**

```
  ┌──────────────────────────────────────────────────────────────────────────┐
  │  DATA LAKE (S3, GCS, Azure Blob)                                         │
  │                                                                          │
  │  ┌─────────────────┐  ┌─────────────────┐  ┌─────────────────────────┐ │
  │  │  RAW (сырые)    │  │  PROCESSED       │  │  Delta / Iceberg         │ │
  │  │  *.parquet      │  │  (очищенные)     │  │  (таблица = метаданные  │ │
  │  │  *.json         │  │  *.parquet       │  │   + файлы .parquet)     │ │
  │  │  *.csv          │  │                 │  │                         │ │
  │  └────────┬────────┘  └────────┬────────┘  │  _delta_log/ или        │ │
  │           │                    │           │  metadata/               │ │
  │           │  ETL/ELT           │           │  ├─ версии               │ │
  │           └──────────────────▶│           │  ├─ ACID-транзакции     │ │
  │                                └───────────┤  └─ time travel          │ │
  │                                            └─────────────────────────┘ │
  └──────────────────────────────────────────────────────────────────────────┘

  Delta Lake / Iceberg — «табличный слой» поверх файлов:
  ─────────────────────────────────────────────────────
  Вместо «свалки файлов» — таблица с журналом изменений.
  INSERT/UPDATE/DELETE записывают новые файлы + обновляют метаданные.
  Читатель всегда видит согласованное состояние (версию) таблицы.
```

---

### Образ: озеро данных vs аккуратный бассейн

- **Классическое хранилище** — как **аккуратный бассейн**:
  - вода очищена и отфильтрована (данные уже очищены и выровнены);
  - есть чёткие правила (схема таблиц, ограничения);
  - не всё туда попадает — только то, что прошло обработку.

- **Data lake** — как **озеро**:
  - в него стекаются разные ручьи и реки (сырые данные из множества источников);
  - вода бывает разного качества;
  - задача — не потерять данные и иметь возможность потом «достать» нужное.

Проблема простого озера:

- без порядка оно превращается в **data swamp** («болото данных»):
  - сложно понять, какие файлы актуальны;
  - нет версионности и транзакций;
  - схемы постоянно меняются.

Delta/Iceberg добавляют к озеру:

- «берега» и «правила купания»:
  - транзакции (ACID),
  - версионность,
  - управление партиционированием и чисткой.

---

### Мини‑упражнения

1. Зачем в табличных форматах lake (Delta/Iceberg) нужны транзакции (ACID)?  
   <details><summary>Ответ</summary>Чтобы гарантировать целостность данных при одновременных записях/обновлениях и не оставлять «наполовину записанные» таблицы, особенно когда с lake работают многие процессы параллельно.</details>

2. Как «time travel» в Delta/Iceberg помогает при ошибках загрузки?  
   <details><summary>Ответ</summary>Можно откатиться к предыдущей версии таблицы (состоянию до ошибочной загрузки) или сравнить данные «до и после», не восстанавливая бэкапы вручную.</details>

3. Чем «data swamp» отличается от управляемого data lake?  
   <details><summary>Ответ</summary>В data swamp нет ясной структуры, версионности и транзакций; непонятно, какие файлы актуальны и как эволюционировала схема. В управляемом lake есть слои (raw/processed), табличные форматы (Delta/Iceberg) с метаданными и правилами.</details>

### Проверь себя (Data lake, Delta и Iceberg)

1. Что добавляют Delta Lake и Iceberg поверх «просто файлов» в S3?  
   <details><summary>Ответ</summary>Транзакционность (ACID), версионность таблиц (time travel), управление партициями и эволюцией схемы — то есть поведение, похожее на таблицу в СУБД, поверх набора файлов Parquet.</details>

2. Зачем в lake разделять слои raw и processed?  
   <details><summary>Ответ</summary>Raw — неизменённые данные из источников, для воспроизводимости и отладки. Processed — очищенные и приведённые к единому формату данные, на которых строятся витрины; так проще поддерживать порядок и не путать сырое с готовым.</details>

3. Когда достаточно «просто Parquet в S3» без Delta/Iceberg?  
   <details><summary>Ответ</summary>Когда запись в основном добавлением (append-only), нет частых обновлений и удалений, не нужны транзакции и откат к предыдущим версиям — например, архивы логов или разовые выгрузки для аналитики.</details>

### 72.4. Общая картинка: из OLTP в хранилище

Чтобы закрепить, сведём всё в единый поток:

**Схема полного потока данных (от приложения до отчётов):**

```
  ┌─────────────────────────────────────────────────────────────────────────┐
  │  ИСТОЧНИКИ (OLTP)                                                        │
  │  PostgreSQL, MySQL, CRM, биллинг, лог-файлы, API                         │
  └───────────────────────────────┬─────────────────────────────────────────┘
                                  │
                    ETL / ELT / CDC (выгрузка)
                                  │
                                  ▼
  ┌─────────────────────────────────────────────────────────────────────────┐
  │  DATA LAKE / СТЕЙДЖИНГ                                                    │
  │  raw_orders.parquet, raw_customers.json, Kafka‑топики                    │
  └───────────────────────────────┬─────────────────────────────────────────┘
                                  │
                    Transform (очистка, маппинг, обогащение)
                                  │
                                  ▼
  ┌─────────────────────────────────────────────────────────────────────────┐
  │  ХРАНИЛИЩЕ ДАННЫХ (модель)                                               │
  │  fact_sales, fact_orders | dim_product, dim_customer, dim_date            │
  │  (звёздная схема)                                                        │
  └───────────────────────────────┬─────────────────────────────────────────┘
                                  │
                    Витрины (агрегаты под конкретные отчёты)
                                  │
                                  ▼
  ┌─────────────────────────────────────────────────────────────────────────┐
  │  ВИТРИНЫ / BI                                                            │
  │  sales_by_region, revenue_by_month, dashboard_metrics                    │
  └───────────────────────────────┬─────────────────────────────────────────┘
                                  │
                                  ▼
  ┌─────────────────────────────────────────────────────────────────────────┐
  │  ПОТРЕБИТЕЛИ                                                             │
  │  Tableau, Metabase, Excel, аналитики, дашборды                           │
  └─────────────────────────────────────────────────────────────────────────┘
```

1. В операционных системах (OLTP) происходят события: заказы, клики, транзакции.
2. CDC/ETL‑процессы **извлекают** эти данные:
   - либо батчами (каждый час выгрузка),
   - либо в потоке (CDC → Kafka → коннектор).
3. Данные попадают в **data lake** (сырые Parquet/JSON) и/или **стейджинг‑слой** хранилища.
4. Поверх lake/стейджинга строятся:
   - **чистые слои** (очищенные и выровненные данные),
   - **модель хранилища** (факты и измерения в звёздной схеме),
   - **витрины** для BI и отчётов.
5. BI‑инструменты и аналитики используют эти витрины для построения отчётов и дашбордов.

### Проверь себя (общая картинка потока данных)

1. В каком порядке данные проходят этапы от OLTP до дашборда?  
   <details><summary>Ответ</summary>Источники (OLTP) → извлечение (ETL/ELT/CDC) → data lake/стейджинг → преобразование и слой хранилища (факты и измерения) → витрины → BI и потребители (дашборды, отчёты).</details>

2. Зачем между «хранилищем данных» и «потребителями» делают слой витрин?  
   <details><summary>Ответ</summary>Витрины — это предрассчитанные агрегаты и разрезы под конкретные отчёты и дашборды. Так потребители получают быстрые ответы без тяжёлых запросов к большим таблицам фактов каждый раз.</details>

3. Чем потоковая загрузка через CDC отличается от батчевой выгрузки по расписанию?  
   <details><summary>Ответ</summary>CDC передаёт изменения почти в реальном времени (события из WAL/binlog), хранилище обновляется с малой задержкой. Батчевая выгрузка по расписанию (например, раз в час) даёт данные с задержкой на интервал батча.</details>
