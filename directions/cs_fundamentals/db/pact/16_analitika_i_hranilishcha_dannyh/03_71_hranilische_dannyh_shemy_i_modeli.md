## §71. Хранилище данных: схемы и модели

### 71.1. Факты и измерения, звёздная схема и снежинка

**Цель раздела.**  
Научиться мыслить в терминах **фактов** и **измерений**, понимать разницу между звёздной схемой и снежинкой и уметь объяснить, когда какая модель уместна.

**В этом разделе главное.**  
1) **Факт** — это таблица событий/метрик (продажа, просмотр, транзакция), где есть числовые поля для агрегаций.  
2) **Измерение** — справочник атрибутов (продукт, клиент, дата, регион), по которым мы «разрезаем» метрики.  
3) **Звёздная схема** — факт в центре, измерения вокруг; **снежинка** — измерения ещё и нормализованы между собой.

**Одна фраза — вся суть §71.**  
Хранилище строится вокруг **фактов** (таблицы событий с суммами, количествами) и **измерений** (справочники: продукт, клиент, дата, магазин). Звезда = факт в центре, измерения рядом; снежинка = измерения ещё разбиты на подтаблицы. SCD решает, как хранить историю изменений в измерениях; таблица дат — готовый календарь для разрезов по времени.

#### Термины

- **Факт (fact table)** — таблица с большим количеством строк, содержащая ключи на измерения и числовые поля (меры), которые агрегируют.
- **Измерение (dimension table)** — относительно небольшая таблица‑справочник с атрибутами (имена, категории, иерархии).
- **Звёздная схема (star schema)** — схема, где факт напрямую связан с измерениями; измерения денормализованы «в ширину».
- **Снежинка (snowflake)** — схема, где измерения нормализованы, есть доп. справочники для иерархий (страна → регион → город и т.п.).

#### Пример звёздной схемы

- Факт: `fact_sales`
  - Ключи: `date_id`, `product_id`, `customer_id`, `store_id`
  - Меры: `quantity`, `amount`
- Измерения:
  - `dim_date(date_id, date, day_of_week, month, quarter, year, ...)`
  - `dim_product(product_id, name, category, brand, ...)`
  - `dim_customer(customer_id, name, segment, region, ...)`
  - `dim_store(store_id, name, region, city, ...)`

**Схема звёздной модели (визуально):**

```
                    dim_date
                        │
                        │ date_id
                        │
    dim_product ────────┼──────── dim_customer
    product_id          │         customer_id
          \             │             /
           \            │            /
            \    ┌──────┴──────┐    /
             \   │ fact_sales │   /
              ──▶│ (ЦЕНТР)    │◀──
            /   │ date_id    │   \
           /    │ product_id │    \
          /     │ customer_id│     \
    store_id   │ store_id   │   amount, quantity
          \    └──────┬──────┘    /
           \          │         /
    dim_store ────────┘──────── dim_...
    store_id

Один факт в центре, все измерения присоединяются напрямую. JOIN идёт только «факт ↔ измерение».
```

**Схема снежинки (измерения разбиты на подтаблицы):**

```
    dim_country     dim_region      dim_city
    country_id ◀─── region_id ◀─── city_id
         │              │               │
         └──────────────┼───────────────┘
                       │
                 dim_customer ──────┐
                 (customer_id)      │
                       │            │
                       └─────┬──────┘
                             │ customer_id
                       ┌─────┴─────┐
                       │ fact_    │
                       │ sales    │
                       └──────────┘

Иерархия: страна → регион → город вынесена в отдельные таблицы.
В снежинке больше JOIN'ов, но меньше дублирования.
```

Запрос:

```sql
SELECT
    d.year,
    p.category,
    SUM(f.amount) AS total_amount
FROM fact_sales AS f
JOIN dim_date    AS d ON f.date_id = d.date_id
JOIN dim_product AS p ON f.product_id = p.product_id
GROUP BY d.year, p.category;
```

#### Звезда vs снежинка

- **Звезда**
  - Плюсы: проще писать SQL (меньше JOIN), быстрее в аналитике.
  - Минусы: дублирование атрибутов (например, название страны и региона может повторяться во множестве строк).
- **Снежинка**
  - Плюсы: меньше дублирования, проще сопровождать и обновлять справочники.
  - Минусы: более сложные запросы, больше `JOIN`, потенциально медленнее.

#### Простыми словами

Факт — это **журнал событий**, измерения — это **справочники** того, что участвует в событиях.  
Звезда — когда справочники подключены прямо к факту. Снежинка — когда справочники ещё и «поделены» на подтемы (страна отдельно, регион отдельно и т.д.).

### Расширенный пример: интернет‑магазин

Представь, что ты строишь хранилище для интернет‑магазина.

**Какие могут быть факты:**

- `fact_orders` — заказы:
  - `order_id`, `date_id`, `customer_id`, `store_id`, `total_amount`, `discount_amount`, ...
- `fact_order_items` — позиции заказа:
  - `order_id`, `product_id`, `quantity`, `amount`, ...
- `fact_site_visits` — посещения сайта:
  - `visit_id`, `date_id`, `customer_id`, `traffic_source_id`, `views`, `duration_seconds`, ...

Здесь:

- каждая строка в факте — **одно событие** или **один измеримый факт**;
- в факте есть:
  - **ключи на измерения** (`date_id`, `customer_id`, `product_id`, `store_id`, ...),
  - **меры** — числовые поля, которые мы агрегируем (`amount`, `quantity`, `views`, ...).

**Какие могут быть измерения:**

- `dim_customer(customer_id, name, email, segment, region, ...)`
- `dim_product(product_id, name, category, subcategory, brand, price, ...)`
- `dim_store(store_id, name, city, region, format, ...)`
- `dim_traffic_source(traffic_source_id, channel, campaign, ...)`
- `dim_date(date_id, date, day_of_week, month, year, is_holiday, ...)`

Каждое измерение:

- относительно небольшое по числу строк (по сравнению с фактами);
- хранит **атрибуты**, по которым мы хотим «резать» отчёты:
  - по региону, по сегменту, по категории товара, по месяцу и т.д.

**Типичный аналитический запрос:**

> «Покажи выручку по категориям товаров и регионам клиентов за год, по месяцам».

SQL:

```sql
SELECT
    d.year,
    d.month,
    c.region,
    p.category,
    SUM(f.amount) AS total_amount
FROM fact_orders AS f
JOIN dim_date    AS d ON f.date_id     = d.date_id
JOIN dim_customer AS c ON f.customer_id = c.customer_id
JOIN dim_product AS p ON f.product_id  = p.product_id
WHERE d.year = 2025
GROUP BY d.year, d.month, c.region, p.category;
```

Здесь хорошо видно:

- `fact_orders` даёт **сами суммы** (`amount`);
- `dim_date` даёт **календарные признаки** (год, месяц);
- `dim_customer` — **регион клиента**;
- `dim_product` — **категорию товара**.

### Как отличить факт от измерения (чек‑лист)

Когда смотришь на таблицу и не уверен, факт это или измерение, задай себе несколько вопросов:

1. **Сколько в ней строк?**
   - миллионы/миллиарды → скорее факт;
   - тысячи/десятки тысяч → скорее измерение.
2. **Что в ней хранят: события или справочную информацию?**
   - «Заказ №123, клиент X, сумма Y» → факт;
   - «Клиент X живёт в городе Y, сегмент Z» → измерение.
3. **Есть ли там числовые поля, которые логично суммировать/считать?**
   - суммы, количества, длительности → признаки факта;
   - названия, статусы, категории → признаки измерения.
4. **Используется ли таблица как «центр» звезды или как «лепесток» вокруг?**
   - если вокруг неё подключаются многие справочники → это факт;
   - если наоборот, сама таблица подключается к фактам → это измерение.

### Проверь себя (факты и измерения, звезда и снежинка)

1. Таблица `page_views` с полями `session_id`, `page_id`, `timestamp`, `duration_sec`. Это факт или измерение и почему?  
   <details><summary>Ответ</summary>Факт. Каждая строка — событие (просмотр страницы), есть числовая мера `duration_sec`, по которой логично считать агрегаты (среднее время, количество просмотров по страницам и т.д.).</details>

2. Чем снежинка отличается от звезды с точки зрения запросов?  
   <details><summary>Ответ</summary>В снежинке измерения разбиты на несколько связанных таблиц (нормализованы), поэтому для отчёта нужно делать больше JOIN по иерархии измерений. В звезде измерения денормализованы и подключаются к факту напрямую.</details>

3. Когда выгоднее использовать снежинку вместо звезды?  
   <details><summary>Ответ</summary>Когда важно сэкономить место за счёт устранения дублирования в больших справочниках с иерархиями (например, категория → подкатегория → товар) и готовы усложнить запросы дополнительными JOIN.</details>

### Мини‑упражнения

1. Таблица `transactions` со столбцами: `id`, `account_id`, `date`, `amount`, `currency`. Что это больше похоже: факт или измерение?  
   <details><summary>Ответ</summary>Факт. Здесь каждая строка — операция (событие), есть числовое поле `amount`, по которому логично делать агрегаты (сумма по счёту, по дате и т.д.).</details>

2. Таблица `products` со столбцами: `product_id`, `name`, `category`, `brand`, `price`. Факт или измерение?  
   <details><summary>Ответ</summary>Измерение. Это справочник товаров, который будет подключаться к фактам продаж по `product_id`.</details>

3. Почему в хранилище выгодно иметь **несколько фактов** (отдельно заказы, позиции заказов, посещения сайта), а не один «мегараздутый» факт?  
   <details><summary>Ответ</summary>Потому что разные факты описывают разные типы событий и позволяют строить более точные и понятные модели: отдельный факт для заказов, отдельный для позиций даёт гибкость в анализе (например, среднее количество позиций в заказе, связка выручки с данными о кликах и т.п.).</details>

### 71.2. Slowly Changing Dimensions (SCD) и суррогатные ключи

**Цель раздела.**  
Научиться работать с изменяющимися измерениями (например, когда клиент меняет сегмент или адрес) и понимать, зачем использовать **суррогатные ключи**.

**В этом разделе главное.**  
1) Измерения со временем меняются, и нужно решить, **как отражать историю** (SCD).  
2) **SCD Type 1** — перезаписываем значение; **Type 2** — храним историю версий; **Type 3** — храним ещё и «предыдущее значение».  
3) **Суррогатные ключи** (искусственные id) помогают отвязать хранилище от ключей источников и удобно хранить версии.

#### Термины

- **SCD (Slowly Changing Dimension)** — паттерны работы с измерениями, которые «медленно меняются».
- **Surrogate key** — искусственный ключ (обычно integer/UUID), не имеющий бизнес‑смысла, используемый как первичный ключ измерения.

#### Типы SCD (кратко)

- **Type 1 (перезапись)**  
  Обновляем запись измерения: старое значение теряется.
  - Подходит, когда история изменений не важна (например, поправили опечатку в названии продукта).
- **Type 2 (история версий)**  
  Создаём новую строку при изменении, старую помечаем как неактивную (или датами `valid_from`/`valid_to`).
  - Подходит, когда важна «историческая» аналитика: хотим знать, в каком сегменте был клиент на момент покупки.
- **Type 3 (предыдущее значение)**  
  Храним и текущее, и предыдущее значение в одной строке.
  - Подходит для редких кейсов, когда важно лишь последнее изменение.

#### Роль суррогатных ключей

- Источники могут использовать разные ключи (ID в CRM, ID в биллинге и т.п.).
- Суррогатный ключ в измерении:
  - Делает модель **устойчивой** к смене источников.
  - Удобен для SCD Type 2: новая версия измерения получает **новый surrogate key**, а факт всегда ссылается на версию измерения, актуальную на момент события.

---

### Примеры SCD на простом кейсе

Представь клиента:

| Номер | Имя   | Сегмент       |
|-------|-------|---------------|
| 1     | Анна  | Standard      |

Через какое‑то время Анна становится **VIP‑клиентом**.

#### Type 1 (перезапись)

Мы просто обновляем строку:

| Номер | Имя   | Сегмент       |
|-------|-------|---------------|
| 1     | Анна  | VIP           |

История:

- раньше она была Standard — **теряется**;
- аналитик, который смотрит на старые заказы Анны, увидит её как VIP и в прошлом, и сейчас.

Подходит, если:

- изменение неважно для ретроспективного анализа (исправление опечатки, смена формата телефона и т.п.).

#### Type 2 (история версий)

Мы храним **две строки**:

| surrogate_key | Business ID (client_id) | Имя  | Сегмент  | valid_from | valid_to   | is_current |
|---------------|-------------------------|------|----------|-----------|-----------|-----------|
| 101           | 1                       | Анна | Standard | 2024‑01‑01 | 2024‑06‑30 | 0         |
| 102           | 1                       | Анна | VIP      | 2024‑07‑01 | 9999‑12‑31 | 1         |

**Временная линия (как факты ссылаются на версии):**

```
Время ───────────────────────────────────────────────────────────▶

2024-01   2024-03        2024-06   2024-07        2024-08
   │         │               │         │               │
   │  Покупка │               │  Покупка │               │
   │  Анны    │               │  Анны    │               │
   ▼         ▼               ▼         ▼               ▼
surrogate_key=101 (Standard)    surrogate_key=102 (VIP)
   │         │               │         │               │
   └─────────┴───────────────┘         └───────────────┘
        эти покупки ссылаются               эти покупки ссылаются
        на версию 101 (Standard)           на версию 102 (VIP)
```

Идея:

- **business ID** (`client_id = 1`) остаётся тем же — это «реальный» клиент;
- **surrogate_key** (101, 102) — разные версии этого клиента;
- факты (покупки) ссылаются на версию, которая была **актуальна на момент события**.

Например:

- покупка от 2024‑03‑10 ссылается на surrogate_key = 101 (Standard);
- покупка от 2024‑08‑05 — на surrogate_key = 102 (VIP).

Так мы можем ответить на вопросы:

- «Сколько денег приносили клиенты **до того, как стали VIP**?»;
- «Как менялась активность клиента **после смены сегмента**?».

#### Type 3 (предыдущее значение)

Пример таблицы:

| client_id | Имя  | current_segment | previous_segment |
|-----------|------|-----------------|------------------|
| 1         | Анна | VIP             | Standard         |

Мы:

- храним только **текущее** и **предыдущее** значение;
- истории дальше одного шага назад нет.

Подходит, когда:

- важна только последняя смена;
- не требуется следить за всеми версиями.

---

### Типичные ошибки с SCD

- **Смешивать Type 1 и Type 2 в одной таблице без чёткого правила.**
  - В результате часть полей ведёт историю, часть — нет,
  - аналитики путаются, какие поля можно использовать для исторического анализа.

- **Не использовать surrogate key при Type 2.**
  - Если делать primary key по business ID (`client_id`), а не по surrogate key,
  - будет трудно хранить несколько версий одной и той же сущности.

- **Забывать про даты действия (`valid_from`, `valid_to`) или флаг `is_current`.**
  - Без этого сложно:
    - выбрать актуальную версию на текущий момент,
    - понять, какая версия была актуальна в прошлое время.

### Проверь себя (SCD и суррогатные ключи)

1. Почему при SCD Type 2 нельзя использовать business key (например, `client_id`) как первичный ключ таблицы измерения?  
   <details><summary>Ответ</summary>Потому что при Type 2 у одной сущности (одного client_id) несколько строк — по одной на каждую версию. Первичный ключ должен быть уникальным; им делают surrogate key, а business key повторяется в разных версиях.</details>

2. В каком случае достаточно SCD Type 1 вместо Type 2?  
   <details><summary>Ответ</summary>Когда история изменений не нужна: исправление опечаток, обновление контактных данных, которые не используются для ретроспективного анализа «как было на момент события».</details>

3. Чем Type 3 ограничен по сравнению с Type 2?  
   <details><summary>Ответ</summary>Type 3 хранит только текущее и предыдущее значение в одной строке; полной истории версий нет. Type 2 хранит все версии в отдельных строках с датами действия.</details>

---

### Простыми словами / образ

Представь:

- **Type 1** — ты просто **перечёркиваешь старую запись в тетради** и пишешь новую поверх. То, что было раньше, тебя больше не интересует.
- **Type 2** — ты **заводишь новую строку в тетради** на того же человека, а старую помечаешь «действовала до такой‑то даты». Вся история сохраняется.
- **Type 3** — в одной строке ты хранишь «текущий статус» и маленькую пометку «а раньше был такой‑то статус».

---

### Мини‑упражнения

1. В CRM у клиента исправили опечатку в имени (`Ана` → `Анна`). Какой тип SCD чаще всего уместен и почему?  
   <details><summary>Ответ</summary>Type 1 (перезапись). Исторически нас не интересует, что имя было с опечаткой — это не бизнес‑событие, а исправление ошибки.</details>

2. Клиент переехал из одного региона в другой, и мы хотим уметь считать продажи по регионам **на момент покупки**. Какой SCD нужен?  
   <details><summary>Ответ</summary>Type 2. Нам нужна история версий клиента по регионам, чтобы понимать, в каком регионе он был на момент каждой транзакции.</details>

3. Почему surrogate key удобен при Type 2?  
   <details><summary>Ответ</summary>Потому что каждая версия измерения получает свой surrogate key, и факты могут ссылаться на нужную версию без конфликтов. Business ID (client_id) может повторяться в нескольких версиях, но surrogate key остаётся уникальным.</details>

### 71.3. Таблица дат (date dimension)

**Цель раздела.**  
Понять, зачем нужна отдельная **таблица дат** и какие атрибуты в неё обычно включают.

Основная идея: вместо того чтобы каждый раз считать «номер недели», «квартал», «рабочий ли день» и т.п., лучше иметь готовый справочник дат с предрасчитанными атрибутами и связывать его с фактами по `date_id`.

Пример структуры:

```sql
CREATE TABLE dim_date (
    date_id        Int32 PRIMARY KEY, -- суррогатный ключ, например 20260223
    date           Date,
    day_of_week    UInt8,  -- 1-7
    day_name       String, -- 'Понедельник', ...
    is_workday     UInt8,  -- 1/0
    week_of_year   UInt8,
    month          UInt8,
    month_name     String,
    quarter        UInt8,
    year           Int32
);
```

Преимущества:

- Упрощает запросы: не нужно дублировать логику вычислений дат.
- Позволяет использовать бизнес‑календарь (своё понимание рабочих/выходных и праздников).

---

### Почему «таблица дат» лучше, чем постоянные функции в SQL

Без `dim_date` разработчики часто пишут в запросах:

- `EXTRACT('year' FROM date)`,
- `EXTRACT('month' FROM date)`,
- `CASE WHEN day_of_week IN (6,7) THEN 'weekend' ELSE 'workday' END`,
- сложные `CASE` для своих бизнес‑праздников.

Проблемы:

- этот код **дублируется** во множестве отчётов;
- где‑то можно ошибиться (разные интерпретации выходных/праздников);
- изменить календарь (добавить новый праздник) сложно: нужно чинить все запросы.

С `dim_date`:

- логика дат сосредоточена **в одном месте** — в таблице;
- изменение календаря:
  - меняем несколько строк в `dim_date`,
  - все отчёты автоматически начинают использовать новый календарь;
- SQL‑запросы становятся проще и короче.

### Мини‑упражнения

1. Зачем может понадобиться поле `is_holiday` в `dim_date`?  
   <details><summary>Ответ</summary>Чтобы быстро строить отчёты по праздникам (выручка в праздники vs будни, влияние акций, поведение пользователей и т.д.), не прописывая руками каждый праздник в запросах.</details>

2. Как с помощью `dim_date` упростить отчёт «выручка по кварталам»?  
   <details><summary>Ответ</summary>Вместо вычислений квартала в каждом запросе, мы просто делаем `JOIN` на `dim_date` и группируем по полю `quarter`, уже предвычисленному в таблице дат.</details>

3. Почему изменение календаря (новый праздник, смена правил выходных) проще делать через `dim_date`, чем через функции в SQL?  
   <details><summary>Ответ</summary>Логика дат сосредоточена в одном месте — в таблице dim_date. Достаточно обновить несколько строк (например, пометить новые праздники), и все отчёты автоматически начнут использовать обновлённый календарь без правки множества запросов.</details>

### Проверь себя (таблица дат)

1. Зачем в фактах хранить ссылку на `date_id` из `dim_date`, а не просто колонку `date`?  
   <details><summary>Ответ</summary>Чтобы единообразно присоединять предвычисленные атрибуты (квартал, неделя, рабочий день, праздник) и не дублировать логику дат в каждом запросе; плюс целочисленный ключ часто эффективнее для JOIN и индексов.</details>

2. Какие типичные поля есть в `dim_date` и зачем они нужны отчётам?  
   <details><summary>Ответ</summary>date_id, date, day_of_week, week_of_year, month, quarter, year, is_workday, is_holiday и т.п. Они нужны для группировок (по кварталу, по неделе), фильтров (только рабочие дни) и анализа по праздникам без повторения логики в SQL.</details>
