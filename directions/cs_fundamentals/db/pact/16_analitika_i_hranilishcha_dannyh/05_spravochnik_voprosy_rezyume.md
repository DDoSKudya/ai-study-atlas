## Вопросы для самопроверки по части XVI

1. В чём главное отличие OLAP‑нагрузки от OLTP‑нагрузки?  
<details><summary>Ответ</summary>OLAP‑нагрузка — это большие сканы и агрегации по огромным объёмам данных, где важна пропускная способность; OLTP‑нагрузка — множество коротких транзакций с точечными чтениями/записями и строгими требованиями к задержке.</details>

2. Почему колоночное хранение эффективно для аналитических запросов?  
<details><summary>Ответ</summary>Оно позволяет читать только нужные столбцы и сильно их сжимать, уменьшая IO; вместе с векторным выполнением это ускоряет агрегации по большим объёмам данных.</details>

3. Что такое звёздная схема и чем она отличается от снежинки?  
<details><summary>Ответ</summary>Звёздная схема — факт в центре и денормализованные измерения вокруг; снежинка — когда измерения дополнительно нормализованы и разбиты на несколько связанных таблиц, что уменьшает дублирование, но усложняет запросы.</details>

4. Зачем использовать CDC при загрузке данных в хранилище?  
<details><summary>Ответ</summary>Чтобы получать поток изменений из источников и делать инкрементальные загрузки без перелива всей таблицы, тем самым ускоряя обновление данных и уменьшая нагрузку.</details>

5. Что добавляют Delta Lake и Apache Iceberg поверх обычного data lake?  
<details><summary>Ответ</summary>Они добавляют транзакционность (ACID), версионность таблиц, удобное управление партиционированием и эволюцией схемы, превращая набор файлов в более «табличный» и управляемый слой.</details>

## Справочник по части XVI

| Тема | Ключевые пункты |
|------|-----------------|
| OLTP vs OLAP | OLTP про короткие транзакции и низкую задержку; OLAP про большие сканы, агрегаты и пропускную способность |
| Колоночное хранение | Читаются только нужные столбцы, данные хорошо сжимаются, а CPU работает пачками значений |
| MPP и HTAP | MPP ускоряет аналитику за счёт параллелизма, а HTAP пытается совместить миры OLTP и OLAP с компромиссами |
| ClickHouse и облачные аналитические БД | MergeTree, ключ сортировки, партиции и материализованные представления помогают строить быстрые аналитические контуры |
| Хранилище данных | Факты, измерения, звезда, снежинка, SCD и surrogate key нужны не “для красоты”, а для управляемой аналитики |
| ETL / ELT / CDC | Хранилище живо только пока в него регулярно и предсказуемо поступают корректные данные |
| Data lake | Файловое озеро требует правил, слоёв и табличных форматов, иначе превращается в data swamp |

## Типичные ошибки по части XVI

- Пытаться держать тяжёлую аналитику прямо на OLTP-базе и удивляться, почему страдают боевые запросы.
- Копировать данные в warehouse без явной модели фактов, измерений и grain, из-за чего витрины начинают противоречить друг другу.
- Считать, что `updated_at` всегда достаточно для инкрементальной загрузки, и не проверять пропуски, late-arriving data и schema drift.
- Делать data lake без правил слоёв, метаданных и freshness-контроля, превращая его в “склад файлов без хозяина”.
- Выбирать аналитическую платформу по моде, а не по паттерну нагрузки, объёму данных и зрелости команды.

## Резюме части XVI

- **OLTP и OLAP решают разные задачи** и требуют разных архитектур и моделей данных.  
- **Колоночные и MPP‑СУБД** оптимизированы под аналитику: большие сканы, агрегации, параллелизм.  
- **Хранилище данных** строится вокруг фактов и измерений, чаще в виде звёздной схемы.  
- **ETL/ELT и CDC** — жизненная артерия хранилища: без регулярной, надёжной загрузки данные быстро устаревают.  
- **Data lake + табличные форматы** позволяют сочетать гибкость файлового хранилища с удобством табличной модели и транзакций.

### Шпаргалка «одна строка — вся тема» (перед экзаменом или собеседованием)

- **OLTP** — много маленьких операций «здесь и сейчас», миллисекунды, точечный доступ.
- **OLAP** — тяжёлые отчёты по большим данным, секунды/минуты, агрегации и сканы.
- **Колоночное хранение** — данные по столбцам → читаем только нужное, сильно сжимаем → быстрая аналитика.
- **MPP** — запрос делим по узлам, считаем параллельно; shuffle и data skew — главные враги.
- **MergeTree** — партиции (часто по месяцам) + сортировка по ключу (ORDER BY) + отсев блоков по индексу.
- **Primary key в ClickHouse** — это ключ сортировки и индекс, не гарантия уникальности.
- **Мат. представление** — при INSERT считаем агрегаты и пишем в отдельную таблицу; отчёты читают её.
- **Факт** — таблица событий с мерами (суммы, количества); **измерение** — справочник (атрибуты для разрезов).
- **Звезда** — факт в центре, измерения вокруг, денормализованы; **снежинка** — измерения ещё разбиты.
- **SCD 1** — перезаписать; **SCD 2** — новая строка (история); **SCD 3** — текущее + предыдущее.
- **ETL** — извлёк → преобразовал снаружи → загрузил готовое; **ELT** — загрузил сырое → преобразуешь в хранилище.
- **CDC** — поток изменений из источника (INSERT/UPDATE/DELETE) → инкрементальная загрузка без полного перелива.
- **Data lake** — файлы (Parquet и др.) в S3/аналоге; **Delta/Iceberg** — транзакции и версии поверх файлов.
