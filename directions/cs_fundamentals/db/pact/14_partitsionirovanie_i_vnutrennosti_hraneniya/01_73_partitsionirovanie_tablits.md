## 73. Партиционирование таблиц

### 73.1. Declarative partitioning

**Цель раздела.**  
Понять, что такое **декларативное партиционирование** (declarative partitioning) в PostgreSQL 10+: одна логическая таблица разбивается на несколько физических партиций по **ключу партиционирования**. Родительская таблица сама данных не хранит — данные лежат только в партициях. После раздела ты будешь знать, когда использовать RANGE, LIST и HASH и как объявить партиционированную таблицу.

**В этом разделе главное (три строки).** 1) **Партиционирование** — это когда одна большая таблица «разрезана» на несколько маленьких (партиций) по правилу: каждая строка попадает ровно в одну партицию в зависимости от значения столбца (или столбцов) — **ключа партиционирования**. 2) **Родительская таблица** — это только «обложка»: в ней объявлены столбцы и способ разбиения (`PARTITION BY RANGE/LIST/HASH`), но **самих данных в ней нет** — все строки лежат в партициях. 3) **Первичный ключ и уникальные индексы** обязательно должны включать ключ партиционирования — иначе СУБД не сможет гарантировать уникальность по всей таблице (две разные партиции могли бы хранить строки с одинаковым id).

**Термины.**

- **Партиционирование (partitioning)** — разбиение одной логической таблицы на несколько физических таблиц (партиций) по значению одного или нескольких столбцов (ключ партиционирования). Запросы пишутся к одной таблице; планировщик и движок направляют операции в нужные партиции.
- **Ключ партиционирования (partition key)** — столбец или набор столбцов, по которым определяется, в какую партицию попадёт строка. Должен входить в первичный ключ и уникальные индексы.
- **RANGE** — партиции по диапазону значений (например, по дате: до 2023-01-01, 2023-01-01–2024-01-01, 2024-01-01 и позже). Удобно для временных рядов и архивов по периодам.
- **LIST** — партиции по списку значений (например, по региону: eu, us, asia). Удобно, когда набор значений ограничен и известен.
- **HASH** — партиции по остатку от деления хеша ключа на число партиций. Равномерное распределение; порядок значений не важен.
- **Родительская таблица (partitioned table)** — таблица, объявленная с `PARTITION BY`; сама не хранит строки, только метаданные и список партиций.

**Теория и правила.**

- В PostgreSQL 10+ партиционирование **декларативное**: ты объявляешь способ разбиения (`PARTITION BY RANGE/LIST/HASH (столбец)`) и создаёшь партиции с границами. Раньше использовали наследование (inheritance) — функционально похоже, но синтаксис и поддержка планировщика хуже.
- **Родительская таблица** создаётся с теми же столбцами, что и партиции; в ней **нет** своего хранения — все `INSERT` идут в одну из партиций по ключу. Если подходящей партиции нет (и нет default) — ошибка.
- **Выбор типа:** RANGE — когда данные естественно разбиваются по периодам или числовым диапазонам; LIST — когда разбиение по категориям (регион, тип); HASH — когда нужна равномерная нагрузка по партициям без привязки к смыслу значения.
- Ограничение: **первичный ключ и уникальные индексы** должны включать ключ партиционирования (иначе нельзя гарантировать уникальность по всей таблице).

**Пошагово: как это устроено изнутри (с числами).**

1. **Шаг 1. Ты создаёшь родительскую таблицу** с `PARTITION BY RANGE (created_at)`. В каталоге данных для этой таблицы **не создаётся** отдельный файл с данными (heap) — только метаданные в системных каталогах: имя таблицы, список столбцов, тип партиционирования и ключ. То есть «таблица» есть в списке таблиц, но как хранилище строк она пустая и всегда будет пустой.

2. **Шаг 2. Ты создаёшь партиции** — например, `events_2023` и `events_2024`. Каждая партиция — это **отдельная физическая таблица**: у неё свой файл (или файлы) на диске, свои страницы, свои строки. В метаданных партиции записано: «я партиция таблицы events» и «ко мне попадают строки, у которых created_at от 2023-01-01 до 2024-01-01».

3. **Шаг 3. При INSERT в родительскую таблицу** (например, `INSERT INTO events (created_at, payload) VALUES ('2023-06-15', '{}')`) движок смотрит на значение ключа партиционирования — `created_at = 2023-06-15`. По границам партиций он определяет: эта дата попадает в интервал [2023-01-01, 2024-01-01), значит строка идёт в партицию `events_2023`. Запись физически пишется **только** в файл партиции `events_2023`; в родительскую таблицу ничего не пишется.

4. **Шаг 4. При SELECT от родительской таблицы** планировщик строит план: «прочитать данные из партиций». Если в WHERE есть условие по ключу (например, `created_at >= '2024-01-01'`), он может **отсечь** партиции, где таких данных заведомо нет (partition pruning), и сканировать только нужные — например, только `events_2024`. Если условия по ключу нет — сканируются все партиции по очереди и результаты объединяются (Append).

Итог: родитель — только «маршрутизатор» и «оглавление»; все данные живут в партициях. Поэтому **без партиций вставлять некуда** — если ты создал родителя с PARTITION BY, но не создал ни одной партиции, INSERT завершится ошибкой «no partition of relation "events" found for row».

**Пример: объявление партиционированной таблицы.**

```sql
-- По диапазону дат (типично для логов, событий)
CREATE TABLE events (
    id         bigint GENERATED ALWAYS AS IDENTITY,
    created_at timestamptz NOT NULL DEFAULT now(),
    payload    jsonb,
    PRIMARY KEY (id, created_at)
) PARTITION BY RANGE (created_at);

-- По списку регионов
CREATE TABLE orders_by_region (
    id        bigint PRIMARY KEY,
    region    text NOT NULL,
    amount    numeric(12,2)
) PARTITION BY LIST (region);

-- По хешу (равномерное распределение)
CREATE TABLE users_hash (
    id       bigint PRIMARY KEY,
    login    text NOT NULL,
    data     jsonb
) PARTITION BY HASH (id);
```

**Что будет, если сделать PRIMARY KEY только по id (без ключа партиционирования)?**  
PostgreSQL выдаст ошибку при создании таблицы: что-то вроде «primary key must include all partitioning columns». Причина: если бы разрешили PK только (id), то партиция events_2023 могла бы хранить строку с id=1, и партиция events_2024 — тоже строку с id=1. Проверка уникальности идёт в рамках одной партиции; чтобы гарантировать уникальность id по **всей** таблице, нужно, чтобы id однозначно определял партицию — то есть ключ партиционирования должен входить в PK. Обычно делают составной PK (id, created_at).

**Типичная ошибка.**  
Создают родительскую таблицу с PARTITION BY и забывают создать партиции. При первом же INSERT получают ошибку «no partition found for row». Решение: перед вставкой данных создать хотя бы одну партицию (или default для RANGE/LIST) для каждого возможного значения ключа.

**Часто путают.**  
**«Родительская таблица хранит общие данные»** — нет. Родительская таблица **вообще не хранит** строки; общих данных нет. Все строки лежат только в партициях. И **«партиции — это как отдельные таблицы, к которым надо обращаться по имени»** — для приложения и для запросов ты обращаешься к **одному** имени (родителю); к партициям по имени обращаются только при администрировании (например, DETACH, VACUUM конкретной партиции).

**Простыми словами.**  
Партиционирование — это «разложить одну большую таблицу по полочкам»: каждая полка — партиция с данными за свой период, регион или по остатку от хеша. Запросы обращаются к «одной таблице», а СУБД сама знает, в какие полки смотреть. Родительская таблица — только «оглавление», сами данные лежат в партициях.

**Картинка в голове.**  
Одна витрина (логическая таблица) с ящиками (партиции). На ящиках подписи: «до 2023», «2023–2024», «2024 и позже» (RANGE) или «EU», «US», «ASIA» (LIST), или «остаток 0», «остаток 1», … (HASH). Кладёшь запись — по ключу выбирается ящик; ищешь — открываешь только нужные ящики. Родительская таблица — это сама витрина с табличкой «события»: внутри неё нет товара, товар лежит в ящиках.

**Как запомнить.**  
Три якоря: 1) **Родитель без данных** — данные только в партициях. 2) **PK/UNIQUE включают ключ партиционирования** — иначе уникальность по всей таблице не гарантируется. 3) **RANGE — периоды, LIST — списки значений, HASH — остаток от деления** — когда что выбирать.

**Проверь себя.**

1. Куда физически попадает строка при `INSERT INTO events (...)` — в родительскую таблицу или в одну из партиций?
<details>
<summary>Ответ</summary>
Строка попадает **только в одну из партиций**. В родительскую таблицу ничего не пишется — у неё нет своего хранилища. Движок по значению ключа партиционирования (например, created_at) выбирает партицию и пишет в неё.
</details>

2. Почему нельзя сделать PRIMARY KEY (id) для таблицы, партиционированной по created_at?
<details>
<summary>Ответ</summary>
Потому что тогда две разные партиции могли бы хранить строки с одинаковым id (например, id=1 в events_2023 и id=1 в events_2024). Проверка уникальности в PostgreSQL при партиционировании выполняется в рамках одной партиции; чтобы гарантировать уникальность по всей таблице, ключ уникальности должен однозначно определять партицию — то есть включать ключ партиционирования. Поэтому PK делают составным: (id, created_at).
</details>

3. В чём разница между RANGE, LIST и HASH одной фразой?
<details>
<summary>Ответ</summary>
RANGE — партиции по диапазону значений (например, по датам: январь, февраль, …). LIST — по списку значений (например, регион: eu, us, asia). HASH — по остатку от деления хеша ключа на число партиций; значения «размазываются» равномерно, порядок не важен.
</details>

**Запомните.**  
Declarative partitioning: родитель с `PARTITION BY RANGE/LIST/HASH (столбец)`; данные только в партициях. PK и UNIQUE должны включать ключ партиционирования. Без созданных партиций INSERT невозможен.

### 73.2. Создание партиций

**Цель раздела.**  
Научиться создавать партиции для родительской таблицы: `CREATE TABLE ... PARTITION OF родитель`. Для RANGE задаётся диапазон `FOR VALUES FROM ... TO ...`, для LIST — `FOR VALUES IN (...)`, для HASH — `FOR VALUES WITH (MODULUS n, REMAINDER r)`.

**В этом разделе главное (три строки).** 1) Партиция создаётся командой **CREATE TABLE имя_партиции PARTITION OF родитель FOR VALUES ...** — столбцы не перечисляются, они наследуются от родителя. 2) Для **RANGE** границы задаются как **FROM значение TO значение**: левая граница **включительно** [ , правая **исключительно** ) — как в математике полуинтервал. 3) Для **HASH** нужно создать **ровно столько партиций**, сколько остатков: при MODULUS 4 — четыре партиции с REMAINDER 0, 1, 2, 3; иначе часть значений окажется «без ящика» и INSERT даст ошибку.

**Термины.**

- **Партиция (partition)** — физическая таблица, наследующая структуру родителя; хранит подмножество строк по границам.
- **Границы партиции (partition bounds)** — условие, при котором строка попадает в эту партицию. Для RANGE — интервал; для LIST — множество значений; для HASH — остаток от деления.

**Теория и правила.**

- Партиция создаётся командой `CREATE TABLE имя_партиции PARTITION OF родитель FOR VALUES ...`. Структура столбцов наследуется от родителя; отдельно можно задать tablespace, fillfactor и т.д.
- **RANGE:** `FOR VALUES FROM ('2023-01-01') TO ('2024-01-01')` — строки, у которых ключ ≥ 2023-01-01 и < 2024-01-01. Границы левые включительно, правые исключительно (как в математике [a, b)).
- **LIST:** `FOR VALUES IN ('eu', 'eu_west')` — строки, у которых ключ в этом списке.
- **HASH:** `FOR VALUES WITH (MODULUS 4, REMAINDER 0)` — строки, у которых `hash(ключ) % 4 = 0`. Создаётся столько партиций, сколько пар (MODULUS, REMAINDER) с одним MODULUS (например, REMAINDER 0, 1, 2, 3 при MODULUS 4).

**Пошагово: как создавать партиции (с числами и границами).**

1. **RANGE.** Допустим, ключ — `created_at` (дата). Партиция «за 2023 год» должна принять все даты с 2023-01-01 00:00:00 включительно по 2024-01-01 00:00:00 **исключительно**. Пишешь: `FOR VALUES FROM ('2023-01-01') TO ('2024-01-01')`. Строка с created_at = '2023-12-31 23:59:59' попадёт сюда; строка с created_at = '2024-01-01 00:00:00' — уже в следующую партицию (FROM '2024-01-01' TO '2025-01-01'). **Не делай дыр и перекрытий:** следующая партиция должна начинаться ровно с той же границы, которой закончилась предыдущая (2023 → 2024 → 2025).

2. **LIST.** Ключ — `region` (текст). Партиция «Европа» принимает значения 'eu', 'eu_west', 'eu_east'. Пишешь: `FOR VALUES IN ('eu', 'eu_west', 'eu_east')`. Одно и то же значение не должно входить в две разные партиции. Если вставишь строку с region = 'asia', а партиции для 'asia' нет и default нет — ошибка «no partition found for row».

3. **HASH.** Ключ — `id` (число). Делим на 4 партиции: остатки 0, 1, 2, 3. Создаёшь четыре таблицы: REMAINDER 0, REMAINDER 1, REMAINDER 2, REMAINDER 3. Строка с id = 10 попадёт в партицию с номером hash(10) % 4. **Обязательно создай все остатки:** если создашь только 0 и 1, значения с остатком 2 и 3 некуда класть — INSERT даст ошибку.

**Что будет, если для RANGE указать TO равное FROM следующей партиции?**  
Так и нужно делать: границы стыкуются. Первая партиция: FROM '2023-01-01' TO '2024-01-01'; вторая: FROM '2024-01-01' TO '2025-01-01'. Значение 2024-01-01 попадает во вторую партицию (левая граница включительно). Если оставить «дыру» — например, первая TO '2024-01-01', вторая FROM '2024-06-01' — то даты с января по май 2024 года не попадут ни в одну партицию (если нет default).

**Типичная ошибка.**  
Для HASH создают три партиции с REMAINDER 0, 1, 2 и забывают про REMAINDER 3. Около четверти значений (остаток 3) некуда класть — INSERT для таких строк завершится ошибкой. Решение: при MODULUS n создавать ровно n партиций с REMAINDER 0, 1, …, n-1.

**Часто путают.**  
**«В RANGE правая граница включена»** — нет, правая граница **исключена** (полуинтервал [FROM, TO)). И **«в CREATE TABLE партиции надо перечислять все столбцы»** — нет, столбцы наследуются от родителя; указываются только имя партиции, PARTITION OF родитель и FOR VALUES ... .

**Примеры.**

```sql
-- Партиции для events (RANGE по created_at)
CREATE TABLE events_2022 PARTITION OF events
    FOR VALUES FROM ('2022-01-01') TO ('2023-01-01');

CREATE TABLE events_2023 PARTITION OF events
    FOR VALUES FROM ('2023-01-01') TO ('2024-01-01');

CREATE TABLE events_2024 PARTITION OF events
    FOR VALUES FROM ('2024-01-01') TO ('2025-01-01');

-- Партиции для orders_by_region (LIST)
CREATE TABLE orders_eu PARTITION OF orders_by_region
    FOR VALUES IN ('eu', 'eu_west', 'eu_east');

CREATE TABLE orders_us PARTITION OF orders_by_region
    FOR VALUES IN ('us', 'us_east', 'us_west');

-- Партиции для users_hash (HASH, 4 партиции)
CREATE TABLE users_hash_0 PARTITION OF users_hash
    FOR VALUES WITH (MODULUS 4, REMAINDER 0);
CREATE TABLE users_hash_1 PARTITION OF users_hash
    FOR VALUES WITH (MODULUS 4, REMAINDER 1);
CREATE TABLE users_hash_2 PARTITION OF users_hash
    FOR VALUES WITH (MODULUS 4, REMAINDER 2);
CREATE TABLE users_hash_3 PARTITION OF users_hash
    FOR VALUES WITH (MODULUS 4, REMAINDER 3);
```

**Простыми словами.**  
Каждая партиция — отдельная таблица с правилом «сюда попадают только такие значения». Для дат — интервалы (от и до); для регионов — списки (eu, us, …); для хеша — остатки от деления (0, 1, 2, 3 при четырёх партициях).

**Картинка в голове.**  
RANGE — линейка с делениями: «от этой метки до той»; левая метка входит в отрезок, правая — уже следующий отрезок. LIST — набор корзин с надписями: «в эту корзину — eu, eu_west». HASH — четыре корзины с номерами 0, 1, 2, 3: число (ключ) делишь с остатком на 4 и кладёшь в корзину с этим остатком.

**Как запомнить.**  
Формула: **CREATE TABLE имя PARTITION OF родитель FOR VALUES** + для RANGE **FROM x TO y** (y не входит), для LIST **IN (a, b, c)**, для HASH **WITH (MODULUS n, REMAINDER r)** — и партиций с разными r должно быть ровно n.

**Проверь себя.**

1. Строка с created_at = '2024-01-01 00:00:00' — в какую партицию попадёт: в ту, у которой TO ('2024-01-01'), или в ту, у которой FROM ('2024-01-01')?
<details>
<summary>Ответ</summary>
В партицию с **FROM ('2024-01-01') TO ('2025-01-01')**. Правая граница в RANGE исключена: интервал [FROM, TO). Поэтому значение 2024-01-01 00:00:00 входит в следующий полуинтервал [2024-01-01, 2025-01-01), а не в предыдущий [2023-01-01, 2024-01-01).
</details>

2. Для HASH с MODULUS 4 ты создал только партиции с REMAINDER 0 и 1. Что будет при INSERT?
<details>
<summary>Ответ</summary>
Строки, у которых hash(ключ) % 4 даёт 2 или 3, некуда класть — для них нет партиции. INSERT завершится ошибкой «no partition of relation … found for row». Нужно создать все четыре партиции с REMAINDER 0, 1, 2, 3.
</details>

3. Нужно ли в CREATE TABLE партиции перечислять столбцы (id, created_at, payload)?
<details>
<summary>Ответ</summary>
Нет. Структура столбцов **наследуется** от родительской таблицы. Указываются только имя партиции, PARTITION OF родитель и FOR VALUES ... (границы). При необходимости можно задать, например, TABLESPACE или fillfactor для этой партиции.
</details>

**Запомните.**  
`CREATE TABLE партиция PARTITION OF родитель FOR VALUES ...`; RANGE — FROM/TO (правая граница исключена); LIST — IN (...); HASH — MODULUS/REMAINDER, партиций ровно столько, сколько остатков.

### 73.3. Default partition

**Цель раздела.**  
Понять партицию **по умолчанию** (default partition): в неё попадают все строки, которые не подошли ни под одну явную партицию. Поддерживается только для RANGE и LIST (для HASH любое значение всегда попадает в одну из партиций по остатку).

**В этом разделе главное (три строки).** 1) **Default partition** — это партиция «для всего остального»: в неё попадают строки, чей ключ не попал ни в одну явную партицию (дата вне всех диапазонов, регион не из списка). 2) Создаётся так: **CREATE TABLE имя PARTITION OF родитель FOR VALUES DEFAULT;** — только для RANGE и LIST. 3) Для **HASH default партиции нет**: любое значение даёт остаток от 0 до MODULUS-1, значит всегда есть «своя» партиция; «лишних» значений не бывает.

**Термины.**

- **Default partition** — партиция без явных границ; принимает все строки, не попавшие в другие партиции. Объявляется как `FOR VALUES DEFAULT`.

**Теория и правила.**

- Если вставить строку с ключом, для которого нет подходящей партиции (например, дата вне всех диапазонов или регион не из списка), без default партиции будет ошибка. Default партиция устраняет эту ошибку: «всё остальное» кладётся в неё.
- Удобно для «запасного» хранилища или когда границы партиций меняются и часть данных временно держат в default. Минус: при запросах с условием по ключу планировщик может не отсечь default (partition pruning сложнее), поэтому лучше явно создавать партиции под ожидаемые значения.
- Создание: `CREATE TABLE имя PARTITION OF родитель FOR VALUES DEFAULT;`

**Пошагово: когда default реально нужен.**

1. **RANGE.** Ты создал партиции за 2022, 2023, 2024 год. Кто-то вставляет строку с created_at = '2025-06-01'. Явной партиции за 2025 год ещё нет. Без default — ошибка «no partition found for row». С default — строка попадёт в default-партицию, и ты потом можешь создать партицию за 2025 год и при желании перенести данные (или оставить в default). Default страхует от «забыли создать партицию на новый период».

2. **LIST.** Партиции для регионов 'eu', 'us'. Появился новый регион 'asia'. Без default INSERT с region = 'asia' даст ошибку. С default — строка попадёт в default; позже можно создать партицию FOR VALUES IN ('asia') и перенести данные или оставить в default.

3. **Минус default:** при запросах с условием по ключу планировщик не всегда может отсечь default (он же «всё остальное»), поэтому может сканировать и её — partition pruning менее эффективен. Лучше по возможности создавать явные партиции под ожидаемые значения.

**Пример.**

```sql
-- Default для orders_by_region: все регионы, не попавшие в eu/us
CREATE TABLE orders_default PARTITION OF orders_by_region
    FOR VALUES DEFAULT;
```

**Что будет, если вставить в RANGE-таблицу дату из будущего, а default партиции нет?**  
Ошибка: «no partition of relation "events" found for row». Нужна либо партиция с границами, включающими эту дату, либо default партиция.

**Типичная ошибка.**  
Пытаются создать default для таблицы, партиционированной по HASH. В PostgreSQL default партиция поддерживается только для RANGE и LIST; для HASH синтаксис FOR VALUES DEFAULT не допускается (и не нужен — все значения уже распределены по остаткам).

**Простыми словами.**  
Default партиция — «корзина для всего остального»: что не подошло ни под один явный ящик, попадает сюда. Для HASH такой корзины нет — каждый ключ всегда попадает ровно в одну партицию по остатку от деления.

**Проверь себя.**

1. Почему для HASH не бывает default партиции?
<details>
<summary>Ответ</summary>
Потому что при HASH любое значение ключа даёт остаток от 0 до (MODULUS - 1). То есть каждое значение уже «приписано» к одной из партиций по формуле hash(ключ) % MODULUS. «Лишних» значений, которым не нашлось партиции, не бывает — default не нужен.
</details>

2. Зачем тогда default для RANGE полезен?
<details>
<summary>Ответ</summary>
Чтобы не падать с ошибкой, когда приходит строка с датой (или числом) вне заранее созданных диапазонов — например, данные за новый год, партицию для которого ещё не создали. Default принимает такие строки; потом можно создать нормальную партицию и при необходимости перенести данные.
</details>

**Запомните.**  
Default partition — `FOR VALUES DEFAULT`; только RANGE и LIST; принимает все значения, не попавшие в другие партиции. Для HASH нет и не нужна.

### 73.4. ATTACH и DETACH

**Цель раздела.**  
Научиться **присоединять** существующую обычную таблицу как партицию (`ATTACH PARTITION`) и **отсоединять** партицию от родителя (`DETACH PARTITION`). Отсоединённая партиция становится обычной таблицей — её можно архивировать или перенести.

**В этом разделе главное (три строки).** 1) **DETACH PARTITION** — «отцепить» партицию от родителя: она становится **обычной самостоятельной таблицей**, данные не удаляются и не копируются, меняются только метаданные. 2) **ATTACH PARTITION** — «прицепить» уже существующую таблицу к партиционированной в качестве партиции: структура и границы должны совпадать, данные остаются на месте. 3) Типичный сценарий: старую партицию (например, за прошлый год) **DETACH** → выгрузить в архив или удалить → освободить место; либо перенести на другой диск и при необходимости снова **ATTACH** (если структура и границы совместимы).

**Термины.**

- **ATTACH PARTITION** — присоединение существующей таблицы к партиционированной в качестве партиции. Таблица должна иметь ту же структуру и совместимые границы; данные остаются на месте.
- **DETACH PARTITION** — отсоединение партиции от родителя. Партиция становится обычной таблицей; данные не удаляются. Часто используют для архивирования старых периодов: DETACH → экспорт/архив → при необходимости можно снова ATTACH или хранить отдельно.

**Теория и правила.**

- **ATTACH:** `ALTER TABLE родитель ATTACH PARTITION существующая_таблица FOR VALUES ...` (RANGE/LIST) или `FOR VALUES WITH (MODULUS n, REMAINDER r)` (HASH). Таблица должна быть пустой или её данные должны попадать в указанные границы; структура (столбцы, типы) должна совпадать с родителем. После ATTACH таблица ведёт себя как партиция.
- **DETACH:** `ALTER TABLE родитель DETACH PARTITION партиция [CONCURRENTLY];` Партиция перестаёт быть партицией; доступ к ней только по её имени. `CONCURRENTLY` (PostgreSQL 14+) уменьшает блокировки при отсоединении.
- Типичный сценарий архивирования: раз в месяц/год создаёшь новую партицию на будущий период; самую старую партицию DETACH, сжимаешь/переносишь в холодное хранилище или удаляешь после экспорта.

**Пошагово: архивирование старой партиции.**

1. Допустим, у тебя партиция `events_2022` за 2022 год, данные больше не нужны в «горячей» таблице. Выполняешь: `ALTER TABLE events DETACH PARTITION events_2022;` Партиция перестаёт быть частью events; таблица `events_2022` остаётся в базе со всеми данными, но теперь это обычная таблица — запросы к `events` её не затрагивают.

2. Дальше по выбору: сделать `pg_dump` только таблицы events_2022 и сохранить в архив; или удалить: `DROP TABLE events_2022;` — место на диске освободится. Либо перенести таблицу в другое табличное пространство (tablespace) и оставить для редких запросов.

3. Если нужно **вернуть** ту же таблицу обратно как партицию (например, после переноса на другой диск): структура таблицы должна совпадать с родителем (те же столбцы и типы), и все строки в ней должны попадать в границы, которые ты укажешь. Команда: `ALTER TABLE events ATTACH PARTITION events_2022 FOR VALUES FROM ('2022-01-01') TO ('2023-01-01');` Если в таблице окажутся строки вне этих границ — ATTACH не выполнится (проверка целостности).

**Пример.**

```sql
-- Отсоединить партицию за 2022 год (она станет обычной таблицей events_2022)
ALTER TABLE events DETACH PARTITION events_2022;

-- Теперь events_2022 — отдельная таблица; можно pg_dump, сжать, перенести
-- Присоединить обратно (если структура и границы совместимы):
ALTER TABLE events ATTACH PARTITION events_2022
    FOR VALUES FROM ('2022-01-01') TO ('2023-01-01');
```

**Что будет, если после DETACH удалить таблицу?**  
Данные этой партиции будут удалены с диска. Для архива лучше сначала сделать дамп (pg_dump) или экспорт, потом уже DROP TABLE, если не нужна копия в базе.

**Типичная ошибка.**  
Делают ATTACH таблицы, в которой есть строки вне указанных границ (например, в таблице лежат данные за 2021–2023, а границы указали только 2022–2023). PostgreSQL при ATTACH проверяет соответствие данных границам (в новых версиях) и может отказать. Решение: перед ATTACH убедиться, что все строки попадают в объявленные границы, или загружать пустую таблицу и потом переносить данные.

**Простыми словами.**  
ATTACH — «прицепить готовый ящик к витрине»; DETACH — «отцепить ящик и унести» (архив, перенос). Данные при этом не копируются — меняется только метаданные: кто родитель, кто партиция.

**Проверь себя.**

1. После DETACH PARTITION куда деваются данные партиции?
<details>
<summary>Ответ</summary>
Никуда не деваются. Партиция становится **обычной таблицей** с тем же именем и теми же данными. Она просто перестаёт быть частью партиционированной таблицы; запросы к родителю её не затрагивают. Данные остаются на диске в той же таблице.
</details>

2. Зачем нужен ATTACH, если можно создать новую партицию через CREATE TABLE ... PARTITION OF?
<details>
<summary>Ответ</summary>
ATTACH нужен, когда у тебя **уже есть** таблица с данными (например, перенесённая с другого сервера, восстановленная из архива, или отсоединённая ранее через DETACH), и ты хочешь «включить» её в партиционированную таблицу без копирования данных. CREATE TABLE ... PARTITION OF создаёт новую пустую партицию; ATTACH подключает существующую таблицу как партицию.
</details>

3. Можно ли после DETACH PARTITION снова присоединить ту же таблицу как партицию?
<details>
<summary>Ответ</summary>
**Да.** Команда **ATTACH PARTITION** позволяет присоединить существующую таблицу к партиционированной. Структура таблицы (столбцы и типы) должна совпадать с родителем; все строки должны попадать в объявляемые границы. Если данные не нарушают границы — `ALTER TABLE родитель ATTACH PARTITION таблица FOR VALUES FROM (...) TO (...);` (или IN / WITH для LIST/HASH). Удобно для переноса партиции на другой диск или временного отсоединения.
</details>

**Запомните.**  
ATTACH PARTITION — присоединить существующую таблицу как партицию (структура и границы должны подходить). DETACH PARTITION — отсоединить партицию в обычную таблицу; данные не удаляются.

### 73.5. Уникальность и FK

**Цель раздела.**  
Усвоить ограничения: **первичный ключ и уникальные индексы** партиционированной таблицы должны включать ключ партиционирования. Внешние ключи на партиционированные таблицы и с партиционированных — особенности.

**В этом разделе главное (три строки).** 1) **PK и UNIQUE** на партиционированной таблице **обязательно** должны включать **ключ партиционирования** — иначе две разные партиции могли бы хранить строки с одинаковым значением уникального столбца, и СУБД не сможет это запретить (проверка уникальности идёт в рамках одной партиции). 2) **Ссылаться на партиционированную таблицу** (FK из другой таблицы на родителя) можно — проверка идёт по всем партициям. 3) Партиционированная таблица может ссылаться на другую таблицу (FK на обычную или партиционированную) — ограничений нет; важно только чтобы PK/UNIQUE на стороне партиционированной таблицы включали ключ партиционирования.

**Теория и правила.**

- **Уникальность:** уникальность по всей таблице можно гарантировать только если ключ уникальности содержит ключ партиционирования. Иначе две партиции могли бы хранить строки с одинаковым значением уникального столбца. Поэтому `PRIMARY KEY (id)` при `PARTITION BY RANGE (created_at)` недопустим; нужно `PRIMARY KEY (id, created_at)` или хотя бы включить `created_at` в уникальный индекс.
- **Внешние ключи (FK):**  
  - Ссылаться **на** партиционированную таблицу можно: FK указывает на родителя, проверка идёт по всем партициям.  
  - Партиционированная таблица может ссылаться на другую таблицу (обычную или партиционированную) — ограничений нет.  
  - Уникальные индексы/PK на стороне партиционированной таблицы по-прежнему должны включать ключ партиционирования.

**Пример.**

```sql
-- Правильно: PK включает ключ партиционирования created_at
CREATE TABLE events (
    id         bigint GENERATED ALWAYS AS IDENTITY,
    created_at timestamptz NOT NULL DEFAULT now(),
    payload    jsonb,
    PRIMARY KEY (id, created_at)
) PARTITION BY RANGE (created_at);

-- Неправильно (ошибка): id не включает created_at
-- CREATE TABLE events (id bigint PRIMARY KEY, created_at timestamptz, ...) PARTITION BY RANGE (created_at);
```

**Простыми словами.**  
Чтобы гарантировать «нет двух одинаковых» по всей таблице, уникальный ключ должен содержать то же поле, по которому мы разбрасываем данные по партициям. Иначе две разные партиции могли бы иметь по строке с одинаковым id.

**Что будет, если попытаться создать UNIQUE (login) на партиционированной по created_at таблице?**  
Ошибка: уникальный индекс должен включать ключ партиционирования (created_at). Иначе в партиции events_2023 могла бы быть строка с login='ivan', и в events_2024 — тоже login='ivan'; каждая партиция проверяет уникальность только у себя, и конфликта бы не было, но глобально уникальность была бы нарушена.

**Проверь себя.**

1. Почему PK не может быть только (id), если таблица партиционирована по created_at?
<details>
<summary>Ответ</summary>
Потому что проверка уникальности выполняется **внутри каждой партиции**. Партиция events_2023 не знает, что в events_2024 уже есть id=1. Чтобы гарантировать уникальность id по **всей** таблице, нужно, чтобы по значению id (и других полей уникального ключа) можно было однозначно определить партицию — то есть ключ партиционирования должен входить в PK. Тогда в одной партиции не может быть двух строк с одинаковым (id, created_at), и в разных партициях created_at разный, значит и (id, created_at) не совпадёт.
</details>

2. Может ли внешний ключ из таблицы orders ссылаться на партиционированную таблицу users (родителя)?
<details>
<summary>Ответ</summary>
Да. FK указывает на родительскую таблицу (users). PostgreSQL при проверке FK учитывает данные во **всех** партициях users. То есть ссылаться на партиционированную таблицу можно так же, как на обычную — по имени родителя.
</details>

3. Может ли партиционированная таблица иметь внешний ключ на обычную таблицу?
<details>
<summary>Ответ</summary>
**Да.** Партиционированная таблица может ссылаться на другую таблицу (обычную или партиционированную) — ограничений нет. Ограничение только на **уникальность** на стороне партиционированной таблицы: PK и UNIQUE должны включать ключ партиционирования. FK от партиционированной таблицы на другую таблицу создаётся как обычно (по имени родителя партиционированной таблицы при ссылке на неё; при ссылке с партиционированной таблицы — на целевую таблицу).
</details>

**Запомните.**  
PK и UNIQUE на партиционированной таблице должны включать ключ партиционирования. FK на партиционированную таблицу разрешены (ссылка на родителя); партиционированная таблица может иметь FK на другие таблицы.

### 73.6. Индексы

**Цель раздела.**  
Понять, что индекс, созданный **на родительской таблице**, автоматически создаётся на **каждой партиции** (локальные индексы). Отдельный глобальный индекс на всю партиционированную таблицу в PostgreSQL нет — только локальные по партициям.

**В этом разделе главное (три строки).** 1) Когда ты делаешь **CREATE INDEX ON родитель (столбец)**, PostgreSQL создаёт **такой же индекс на каждой существующей партиции** и будет создавать его на всех новых партициях — то есть индексы **локальные** (по одной копии на партицию). 2) **Глобального индекса** (одного дерева на всю таблицу) в PostgreSQL **нет** — поиск по индексу идёт так: сначала по условию по ключу партиционирования отсекаются партиции (partition pruning), затем в каждой нужной партиции используется свой локальный индекс. 3) Уникальный индекс на родителе возможен только если он включает ключ партиционирования — по тем же причинам, что и PK.

**Термины.**

- **Локальный индекс (local index)** — индекс, существующий в рамках одной партиции. В партиционированной таблице индекс на родителе разворачивается в набор таких индексов (по одному на партицию).
- **Глобальный индекс (global index)** — индекс по всей таблице, не привязанный к партициям. В PostgreSQL **нет** нативной поддержки глобальных индексов на партиционированных таблицах; уникальность и поиск обеспечиваются за счёт того, что ключ партиционирования входит в PK/UNIQUE и планировщик делает partition pruning.

**Теория и правила.**

- `CREATE INDEX ON родитель (столбец)` создаёт индекс на родителе; движок создаёт такой же индекс на каждой существующей партиции и будет создавать на новых партициях по умолчанию (если не указано `(default)` в определении партиции). Каждая партиция имеет свой B-tree (или другой тип) индекса — то есть индексы локальные.
- Запрос с условием по ключу партиционирования сначала сужает набор партиций (partition pruning), затем в каждой задействованной партиции используется свой индекс. Глобального «одного индекса по всей таблице» нет, но за счёт pruning часто читается одна или несколько партиций, и локальных индексов достаточно.
- Уникальный индекс на родителе возможен только если он включает ключ партиционирования (как и PK).

**Пример.**

```sql
CREATE INDEX ON events (created_at, id);       -- по партициям создастся локальный индекс
CREATE INDEX ON events USING gin (payload);    -- GIN по jsonb на каждой партиции
```

**Простыми словами.**  
Индекс на «витрине» на самом деле размножается по всем ящикам: в каждом ящике свой экземпляр индекса. Общего индекса по всем ящикам PostgreSQL не строит. Зато при запросе «по ключу партиционирования» открываются только нужные ящики, и в каждом свой индекс — этого достаточно для быстрого поиска.

**Что будет, если создать индекс на родителе после того, как уже есть 10 партиций?**  
PostgreSQL создаст один и тот же индекс на **всех десяти** партициях (и на каждой это может занять время и блокировки). Индекс создаётся на каждой партиции по очереди. Для больших таблиц лучше создавать индекс CONCURRENTLY, чтобы уменьшить блокировки.

**Проверь себя.**

1. Где физически хранятся данные индекса партиционированной таблицы — в одном дереве или в нескольких?
<details>
<summary>Ответ</summary>
В **нескольких**: по одному индексу (своему дереву) на каждую партицию. То есть локальные индексы. Одного общего «глобального» индекса по всей таблице в PostgreSQL нет.
</details>

2. Как тогда выполняется поиск по индексу, если партиций много?
<details>
<summary>Ответ</summary>
Сначала планировщик по условию WHERE по ключу партиционирования отсекает партиции (partition pruning) — определяет, в каких партициях могут быть нужные строки. Затем для каждой оставшейся партиции выполняется поиск по **локальному** индексу этой партиции. Результаты объединяются. То есть глобального индекса не нужно: мы и так обращаемся только к нужным партициям.
</details>

3. Нужно ли создавать индекс отдельно на каждой партиции вручную?
<details>
<summary>Ответ</summary>
**Нет.** Достаточно создать индекс **на родительской таблице**: `CREATE INDEX ON родитель (столбец)`. PostgreSQL автоматически создаёт такой же индекс на **каждой существующей** партиции и будет создавать его на **всех новых** партициях (при добавлении партиции через CREATE TABLE ... PARTITION OF). То есть индексы **локальные**, но создаются одной командой на родителе.
</details>

**Запомните.**  
Индекс на родителе = такой же индекс на каждой партиции (локальные индексы). Глобального индекса нет; поиск = partition pruning + локальный индекс по каждой задействованной партиции.
