## Частые вопросы (краткие ответы для повторения)

| Вопрос | Короткий ответ |
|--------|----------------|
| Где хранятся данные в партиционированной таблице? | Только в **партициях**; родитель — пустая «обёртка», без своих данных. |
| Почему PK должен включать ключ партиционирования? | Иначе две разные партиции могли бы хранить строки с одинаковым значением уникального столбца. |
| Что такое partition pruning? | Отсечение партиций по WHERE: не сканируются партиции, где данных заведомо нет. |
| Статическое и динамическое отсечение — в чём разница? | Статическое — при планировании (константы в запросе); динамическое — при выполнении (параметры $1, $2). |
| Зачем default partition? | Принимает строки, не попавшие ни в одну явную партицию (только RANGE и LIST). |
| Что делает обычный VACUUM? | Удаляет мёртвые строки, обновляет visibility map и FSM; **не** уменьшает размер файла на диске. |
| Когда нужен VACUUM FULL? | Когда нужно **уменьшить размер файла** после массового DELETE; таблица блокируется на время перезаписи. |
| Зачем VACUUM FREEZE и что такое wraparound? | Заморозка старых версий строк; защита от переполнения счётчика транзакций (XID); иначе база запретит запись. |
| Можно ли отключить autovacuum? | Не рекомендуется: bloat, устаревшая статистика, риск wraparound. Настраивают пороги, но не отключают. |
| Синхронная и асинхронная репликация — в чём разница? | Синхронная — primary ждёт подтверждения реплики (данные не теряются, запись медленнее); асинхронная — не ждёт (быстрее, но при сбое возможна потеря последних записей). |
| Чем synchronous_standby_names отличается от synchronous_commit? | synchronous_standby_names — **какие** реплики ждать (список имён). synchronous_commit — **ждать ли** сброс WAL и подтверждение реплик (on — ждать, off — быстрее, возможна потеря). |
| Зачем replication slot? | Primary не удаляет WAL, пока потребитель не подтвердит получение; реплика сможет догнать после отключения. Риск: при «мёртвой» реплике WAL копится на primary. |
| Репликация и бэкап — одно и то же? | Нет. Реплика — «живая копия», ошибки реплицируются; откатиться с реплики нельзя. Бэкап — снимок в другом месте, восстановление до момента до ошибки. |
| Что такое PITR? | Восстановление до произвольного момента: полный бэкап + архив WAL + restore_command + recovery_target_time (или _lsn). |
| pg_basebackup и pg_dump — когда что? | basebackup — полная копия **файлов** кластера (реплика, PITR); pg_dump — **логический** дамп схемы и данных (одна база, выборочные объекты). |
| Что в PostgreSQL играет роль «инкрементального» бэкапа? | **Архив WAL** между полными бэкапами: полный бэкап реже + непрерывный archive_command; восстановление = последний полный бэкап + применение WAL до нужного момента (PITR). Нативно «только изменённые файлы» нет. |

Используй эту таблицу для быстрого повторения перед экзаменом или после перерыва: вопрос → короткий ответ.

## Вопросы для самопроверки (часть XIV)

1. **Партиционирование.** Почему первичный ключ партиционированной таблицы должен включать ключ партиционирования? Что такое default partition и для каких типов партиционирования она поддерживается?

<details>
<summary>Ответ</summary>
Уникальность по всей таблице можно гарантировать только если ключ уникальности содержит ключ партиционирования — иначе две разные партиции могли бы хранить строки с одинаковым значением уникального столбца. Default partition — партиция «по умолчанию» для значений, не попавших ни в одну явную партицию; поддерживается только для RANGE и LIST (для HASH любое значение всегда попадает в одну из партиций по остатку).
</details>

2. **Partition pruning.** В чём разница между статическим и динамическим отсечением партиций? Почему при подготовленных запросах план может содержать все партиции?

<details>
<summary>Ответ</summary>
Статическое отсечение выполняется при планировании, когда условие содержит константы — планировщик сразу знает, какие партиции сканировать. Динамическое — при выполнении, когда условие содержит параметры (prepared statement): при первом вызове планировщик может не знать значения и построить generic plan со всеми партициями; при выполнении движок отсекает партиции по фактическим значениям (runtime pruning).
</details>

3. **WAL и PITR.** Что должно быть настроено для восстановления до произвольного момента времени (PITR)? Зачем нужны archive_command и restore_command?

<details>
<summary>Ответ</summary>
Для PITR нужны: 1) полная копия данных (pg_basebackup или файловый бэкап) на момент T0; 2) непрерывная цепочка сегментов WAL от T0 до момента T1, до которого восстанавливаем; 3) restore_command (достать сегмент из архива при восстановлении) и recovery_target_time (или recovery_target_lsn). archive_command копирует заполненные сегменты WAL в архив при работе primary; restore_command при восстановлении достаёт сегменты из архива по запросу сервера.
</details>

4. **VACUUM.** Чем обычный VACUUM отличается от VACUUM FULL? Зачем нужен VACUUM FREEZE и что такое transaction ID wraparound?

<details>
<summary>Ответ</summary>
Обычный VACUUM удаляет мёртвые строки, обновляет visibility map и FSM, но не перезаписывает таблицу и не возвращает место в ОС (только внутри таблицы для новых строк). VACUUM FULL перезаписывает таблицу с эксклюзивной блокировкой и возвращает место в ОС. VACUUM FREEZE замораживает старые версии строк (помечает их как «проверенные»), чтобы защититься от transaction ID wraparound: идентификаторы транзакций конечны; при приближении возраста самой старой незамороженной строки к пределу база принудительно требует заморозку, иначе запрещает запись.
</details>

5. **Репликация.** В чём разница между синхронной и асинхронной streaming-репликацией? Какой риск создаёт репликационный слот при отстающей реплике?

<details>
<summary>Ответ</summary>
Синхронная — primary ждёт подтверждения от реплики (synchronous_standby_names), что WAL записан на реплику, прежде чем считать транзакцию зафиксированной; при падении primary данные не теряются, но запись задерживается. Асинхронная — primary не ждёт; запись быстрее, но при падении primary последние транзакции могут не успеть попасть на реплику. Репликационный слот сохраняет WAL на primary до подтверждения потребителя; при отключённой или отстающей реплике слот не подтверждает получение — WAL накапливается на primary и может заполнить диск.
</details>

6. **Бэкап.** Почему реплика не заменяет резервное копирование? Когда использовать pg_basebackup, а когда pg_dump?

<details>
<summary>Ответ</summary>
Реплика копирует все изменения в реальном времени — случайное удаление данных или ошибка приложения реплицируется на все узлы; откатиться «на вчера» только с реплики нельзя. Бэкап — снимок/архив в отдельном хранилище; позволяет восстановить состояние на момент бэкапа или до произвольного момента (PITR). pg_basebackup — полная копия файлов кластера; для создания standby или полного бэкапа с последующим применением WAL (PITR). pg_dump — логический дамп (схема и данные в SQL или бинарном формате); для переноса схемы и данных между базами, выборочного восстановления объектов, бэкапа одной базы без WAL.
</details>

## Резюме части XIV

**Партиционирование** — разбиение одной логической таблицы на физические партиции по ключу (RANGE, LIST, HASH). Родительская таблица объявляется с `PARTITION BY`; данные хранятся только в партициях. PK и UNIQUE должны включать ключ партиционирования. Индексы на родителе создаются на каждой партиции (локальные индексы). Default partition принимает «лишние» значения (только RANGE/LIST). ATTACH/DETACH — присоединение и отсоединение партиций (архивирование).

**Partition pruning** — отсечение партиций по условию WHERE: статическое при планировании (константы), динамическое при выполнении (параметры). Partition-wise join и aggregate — соединение и агрегация по партициям при совместимом партиционировании. Субпартиционирование — партиция внутри партиции.

**WAL и PITR** — сегменты WAL по 16 MB; каждая запись WAL — заголовок (LSN, размер, resource manager) + тело; full page write — первая модификация страницы после checkpoint пишется в WAL целиком (защита от частичной записи); детали формата — Часть V. LSN — позиция в WAL. archive_command копирует сегменты в архив; restore_command при восстановлении достаёт их из архива. recovery_target_time/LSN задают точку остановки восстановления. standby.signal и primary_conninfo — режим standby; promote — переключение в primary. PITR = полный бэкап + архив WAL + restore_command + recovery_target.

**VACUUM** — удаление мёртвых строк, обновление visibility map и FSM; обычный VACUUM не возвращает место в ОС. VACUUM FULL — перезапись таблицы с эксклюзивной блокировкой. VACUUM FREEZE — заморозка старых версий; защита от transaction ID wraparound. Autovacuum запускает VACUUM по порогу (threshold + scale_factor * n_live_tup) и ANALYZE по порогу (autovacuum_analyze_threshold + autovacuum_analyze_scale_factor * n_live_tup). Мониторинг — pg_stat_user_tables.

**Репликация** — streaming (поток WAL от primary к standby; синхронная/асинхронная); synchronous_standby_names (какие реплики ждать), synchronous_commit (ждать ли сброс WAL и подтверждение реплик); репликационные слоты (сохранение WAL на primary до подтверждения потребителя; риск накопления WAL). Логическая репликация — публикация/подписка, репликация на уровне строк; replica identity (DEFAULT/FULL/INDEX) для UPDATE/DELETE; конфликты и разрешение (error — остановить подписку, skip — пропустить изменение, apply — перезаписать на подписчике, PG15+). Logical decoding — pgoutput, test_decoding, wal2json; CDC.

**Бэкап** — pg_basebackup (полная копия кластера; для реплики или полного бэкапа); инкрементальный бэкап в PostgreSQL — полный бэкап реже + непрерывный архив WAL между полными бэкапами (PITR до любого момента; нативно «только изменённые файлы» нет). pg_dump/pg_restore (логический дамп; форматы plain, custom, directory; параллельный режим -j; выбор объектов: -t таблица, -n схема, --exclude-*). RTO и RPO; регулярная проверка восстановления. Реплика не заменяет бэкап — случайное удаление реплицируется; бэкап — отдельная защита.

Материал части XIV опирается на **Часть V (хранение, WAL, страницы)** и **Часть IV (транзакции, MVCC)** и готовит к **Части XI (масштабирование, шардирование)** и операционным темам (мониторинг, DR, бэкапы).
