## 74. Partition pruning и partition-wise операции

### 74.1. Partition pruning

**Цель раздела.**  
Понять **partition pruning** (отсечение партиций): планировщик по условию WHERE определяет, какие партиции содержат подходящие данные, и **не сканирует** остальные. Различают **статическое** отсечение (при планировании, когда значения известны) и **динамическое** (при выполнении, когда используются параметры).

**В этом разделе главное (три строки).** 1) **Partition pruning** — это когда планировщик **не открывает лишние партиции**: по условию WHERE по ключу партиционирования он понимает, в каких партициях могут быть строки, и в плане остаются только они; остальные партиции вообще не читаются. 2) **Статическое** отсечение — условие в запросе задано константами (например, `WHERE created_at >= '2024-01-01'`), планировщик при построении плана сразу знает набор партиций. 3) **Динамическое** отсечение — условие содержит параметры (prepared statement); при планировании значения неизвестны, отсечение происходит **при выполнении** по фактическим значениям параметров (runtime pruning).

**Термины.**

- **Partition pruning** — оптимизация: исключение партиций, которые заведомо не содержат строк, удовлетворяющих условию запроса. В плане видно только задействованные партиции (узлы Append с дочерними Scan по партициям).
- **Статическое отсечение (static pruning)** — отсечение на этапе **планирования**, когда условие содержит константы (например, `WHERE created_at >= '2024-01-01' AND created_at < '2024-02-01'`). Планировщик сразу знает, какие партиции сканировать.
- **Динамическое отсечение (dynamic / runtime pruning)** — отсечение на этапе **выполнения**, когда условие содержит параметры (prepared statement, подзапрос). При первом выполнении планировщик может не знать значения параметров и заложить все партиции; при выполнении движок отсекает партиции по фактическим значениям.

**Теория и правила.**

- Если в WHERE есть условие по ключу партиционирования (равенство, диапазон, IN), планировщик сопоставляет его с границами партиций и оставляет только те партиции, в которых могут быть подходящие строки. В плане (EXPLAIN) видно: вместо сканирования всех партиций — только нужные (Append → Seq Scan / Index Scan по партициям).
- **Подготовленные запросы (prepared statements):** при первом вызове планировщик может не знать значения параметров и построить «универсальный» план с доступом ко всем партициям. При выполнении движок делает runtime pruning по фактическим значениям. В некоторых версиях при первом наборе параметров план может быть неоптимальным (все партиции в плане); при последующих вызовах с другими параметрами план может кэшироваться — тогда важен порядок первых вызовов.
- Для наилучшего pruning условие по ключу партиционирования должно быть в явном виде (например, `created_at BETWEEN ... AND ...` или `region = 'eu'`). Сложные выражения или функции от ключа могут помешать отсечению.

**Пошагово: что происходит при запросе с partition pruning.**

1. Запрос: `SELECT * FROM events WHERE created_at >= '2024-01-01' AND created_at < '2024-02-01';` Планировщик видит условие по ключу партиционирования (created_at) и границы партиций: events_2022 [2022–2023), events_2023 [2023–2024), events_2024 [2024–2025). Интервал запроса [2024-01-01, 2024-02-01) пересекается только с партицией events_2024. Партиции events_2022 и events_2023 **вообще не попадают в план** — для них не создаётся узел Scan.

2. В плане (EXPLAIN) ты увидишь что-то вроде: Append → Seq Scan on events_2024, Filter: ... То есть один Scan по одной партиции вместо трёх (или N) по всем. Меньше чтения с диска, меньше работы — запрос быстрее.

3. Если бы условия по created_at не было (`SELECT * FROM events WHERE payload @> '{"x":1}'`), планировщик не смог бы отсечь партиции — в плане были бы Scan по **всем** партициям и объединение результатов (Append). Поэтому для партиционированных таблиц важно, чтобы типичные запросы содержали условие по ключу партиционирования.

**Пример плана с partition pruning.**

```sql
EXPLAIN (COSTS OFF) SELECT * FROM events
WHERE created_at >= '2024-01-01' AND created_at < '2024-02-01';

-- Ожидаемо: только партиция events_2024 (или одна из партиций по диапазону)
-- Append
--   -> Seq Scan on events_2024
--        Filter: (created_at >= '2024-01-01' AND created_at < '2024-02-01')
```

**Что будет, если написать WHERE date_trunc('month', created_at) = '2024-01-01'?**  
Планировщик может не суметь отсечь партиции: условие выражено через функцию от ключа, а не через прямое сравнение created_at с константой. Лучше писать явно: `created_at >= '2024-01-01' AND created_at < '2024-02-01'` — тогда pruning сработает надёжно.

**Типичная ошибка.**  
Партиционируют таблицу по дате, но в приложении запросы часто без фильтра по дате (например, поиск по id или по другому полю). В итоге сканируются все партиции, и выигрыша от партиционирования почти нет. Решение: по возможности всегда передавать условие по ключу партиционирования в запросах (хотя бы диапазон дат).

**Простыми словами.**  
Partition pruning — «не открывать лишние ящики»: по условию запроса СУБД понимает, в каких ящиках могут быть нужные данные, и смотрит только в них. Статическое — когда условие известно уже при планировании; динамическое — когда значения подставляются при выполнении.

**Проверь себя.**

1. В запросе нет условия по ключу партиционирования (created_at). Сколько партиций будет просканировано?
<details>
<summary>Ответ</summary>
**Все** партиции. Без условия по ключу планировщик не может отсечь ни одну партицию — неизвестно, в каких могут быть подходящие строки. В плане будет Append с Seq Scan (или Index Scan) по каждой партиции. Поэтому для выгоды от партиционирования важно, чтобы запросы часто содержали фильтр по ключу.
</details>

2. В чём разница между статическим и динамическим pruning одной фразой?
<details>
<summary>Ответ</summary>
Статическое — отсечение партиций происходит **при планировании** запроса, когда в условии стоят константы (например, дата в тексте запроса). Динамическое — отсечение происходит **при выполнении**, когда условие содержит параметры (prepared statement): планировщик при построении плана не знает значения, отсечение делается движком по факту при выполнении.
</details>

**Как запомнить.**  
**Partition pruning** = «не открывать ящики, где данных точно нет»: по WHERE отсекаются партиции. **Статическое** — при планировании (константы в запросе); **динамическое** — при выполнении (параметры $1, $2). Условие по **ключу партиционирования в явном виде** (без функций) даёт лучшее отсечение. В плане — меньше партиций в Append.

**Запомните.**  
Partition pruning — отсечение партиций по WHERE; не сканируются партиции, где данных заведомо нет. Статическое — при планировании (константы); динамическое — при выполнении (параметры). Условие по ключу в явном виде (без функций) улучшает отсечение.

### 74.2. Подготовленные запросы

**Цель раздела.**  
Учитывать особенность **подготовленных запросов** (prepared statements) при партиционировании: план может кэшироваться с первым набором параметров; при первом вызове с «широким» диапазоном в плане могут оказаться все партиции, и последующие вызовы с узким диапазоном будут использовать тот же план (но runtime pruning при выполнении всё равно отсечёт лишние партиции).

**В этом разделе главное (три строки).** 1) **Подготовленный запрос** — это когда приложение один раз отправляет запрос с параметрами (`$1`, `$2`) и СУБД **запоминает план**; при следующих вызовах подставляются другие значения параметров, но план может оставаться тем же (generic plan). 2) При партиционировании **при первом вызове** планировщик не знает значений параметров — он может построить план, в котором фигурируют **все партиции** (на всякий случай). При **выполнении** движок подставляет фактические значения и делает **runtime pruning** — отсекает партиции по факту, так что читаются только нужные. 3) **Итог:** производительность при выполнении обычно нормальная (читаются только нужные партиции), но если снять план через EXPLAIN без подстановки конкретных значений, в плане могут быть все партиции — не пугайся, при реальном запросе отсечение сработает.

**Теория и правила.**

- В prepared statement условие содержит параметры (`$1`, `$2`). При **первом** выполнении планировщик может выбрать generic plan (универсальный план) или custom plan (под конкретные значения). Для партиционированных таблиц generic plan часто предполагает сканирование всех партиций; при выполнении движок отсекает партиции по значениям параметров (runtime pruning).
- Если приложение сначала выполняет запрос с параметрами, охватывающими все партиции (например, весь год), а потом — с узким диапазоном (один месяц), кэшированный план может по-прежнему перечислять все партиции в плане; фактическое чтение при runtime pruning будет только из нужных партиций. То есть производительность при выполнении обычно нормальная, но в EXPLAIN без выполнения может казаться, что план «тяжёлый».
- Рекомендация: по возможности передавать условия по ключу партиционирования в явном виде; при критичной производительности проверять планы с типичными наборами параметров (EXPLAIN с подставленными константами и EXPLAIN для prepared statement).

**Пошагово: что происходит при подготовленном запросе к партиционированной таблице.**

1. Приложение отправляет: `PREPARE plan1 AS SELECT * FROM events WHERE created_at >= $1 AND created_at < $2;` Затем выполняет: `EXECUTE plan1 ('2024-01-01', '2024-02-01');` При **первом** EXECUTE планировщик может не «зашивать» константы в план (generic plan) — в плане тогда будут все партиции (Append по всем).

2. При **выполнении** движок подставляет `$1 = '2024-01-01'`, `$2 = '2024-02-01'` и по этим значениям определяет: нужны только партиции, пересекающиеся с [2024-01-01, 2024-02-01). Читаются только эти партиции; остальные не открываются (runtime pruning). То есть **фактическая работа** — только по нужным партициям.

3. При **втором** EXECUTE с другими параметрами (`'2023-06-01'`, `'2023-07-01'`) используется тот же закэшированный план (с перечислением всех партиций), но при выполнении снова сработает runtime pruning — будут прочитаны только партиции за июнь 2023. Поэтому **не нужно** переподготавливать запрос для каждого диапазона — отсечение происходит при каждом выполнении.

**Простыми словами.**  
Подготовленный запрос «запоминает» план. Если первый раз подставили параметры «на весь год», план может учитывать все партиции; при следующих вызовах с «одним месяцем» движок при выполнении всё равно отсечёт лишние партиции, но в закэшированном плане они могут по-прежнему фигурировать.

**Проверь себя.**

1. При подготовленном запросе к партиционированной таблице в EXPLAIN видно все партиции. Значит ли это, что при каждом EXECUTE читаются все партиции?
<details>
<summary>Ответ</summary>
**Нет.** При **выполнении** (EXECUTE) движок подставляет фактические значения параметров и делает **runtime pruning** — отсекает партиции, в которых данных заведомо нет. Читаются только нужные партиции. В плане (EXPLAIN без выполнения) могут быть перечислены все партиции, потому что план строился без знания параметров — но при реальном EXECUTE отсечение срабатывает.
</details>

2. Как при подготовленном запросе оценить, какие партиции реально будут сканироваться?
<details>
<summary>Ответ</summary>
Сделать **EXPLAIN с подставленными константами** вместо параметров: выполнить тот же запрос с типичными значениями (например, `WHERE created_at >= '2024-01-01' AND created_at < '2024-02-01'`) и посмотреть план — тогда планировщик сделает статическое отсечение и в плане будут только нужные партиции. Либо использовать EXPLAIN (ANALYZE) для EXECUTE с нужными параметрами — в actual rows будет видно, по каким партициям шло чтение.
</details>

**Как запомнить.**  
**Prepared statement** = план кэшируется, параметры ($1, $2) при первом вызове могут быть неизвестны — в плане тогда **все партиции** (generic plan). При **выполнении** движок подставляет значения и делает **runtime pruning** — читаются только нужные партиции. В EXPLAIN без выполнения план может «пугать» списком партиций — при реальном EXECUTE отсечение сработает. Для оценки плана — EXPLAIN с типичными константами.

**Запомните.**  
При prepared statements план может строиться без знания параметров; возможен generic plan со всеми партициями. Runtime pruning при выполнении отсекает лишние партиции; для оценки плана смотри EXPLAIN с типичными значениями.

### 74.3. Partition-wise join

**Цель раздела.**  
Понять **partition-wise join**: когда две партиционированные таблицы соединены по ключу, совместимому с партиционированием, планировщик может выполнять соединение **по партициям** — каждая пара партиций соединяется отдельно. Это уменьшает объём данных в памяти и может улучшить параллелизм.

**В этом разделе главное (три строки).** 1) **Partition-wise join** — это когда две таблицы партиционированы **одинаково** (например, обе по дате с одними и теми же границами) и соединение идёт по ключу партиционирования (или условие гарантирует соответствие партиций). Тогда планировщик может **не смешивать** данные из разных партиций: партиция 1 таблицы A соединяется только с партицией 1 таблицы B, партиция 2 A — только с партицией 2 B и т.д. 2) **Выгода:** каждое соединение меньше по объёму — меньше памяти на хеш-таблицу или сортировку, лучше кэш. Можно параллельно соединять пары партиций разными воркерами. 3) **Условие:** границы партиций должны совпадать (или одна таблица — подмножество партиций другой), иначе «пары» партиций не определены.

**Термины.**

- **Partition-wise join** — соединение двух партиционированных таблиц так, что соединение выполняется между соответствующими партициями (партиция 1 таблицы A с партицией 1 таблицы B и т.д.). Результат — объединение результатов по парам партиций.
- Условие: ключи партиционирования и границы партиций должны быть **совместимы** (одинаковая стратегия и совпадающие границы, или одна таблица — подмножество партиций другой). Тогда планировщик может сопоставить партиции одна к одной.

**Теория и правила.**

- Если обе таблицы партиционированы одинаково (например, по дате с одинаковыми границами) и JOIN идёт по ключу партиционирования или по условию, сохраняющему соответствие партиций, планировщик может выбрать partition-wise join: вместо одного большого Hash Join или Merge Join — множество соединений по парам партиций. Каждое соединение меньше по объёму данных — меньше памяти и лучше кэш.
- В плане видно: несколько пар (Join между партицией A и партицией B), объединённых через Append. Требуется совместимое партиционирование (одинаковый тип и границы или их подмножество).

**Пример (концептуально).**

```sql
-- events и events_archive партиционированы по created_at одинаково
SELECT * FROM events e
JOIN events_archive a ON e.id = a.id AND e.created_at = a.created_at
WHERE e.created_at >= '2024-01-01';

-- Планировщик может сделать: для каждой партиции (2024-01, 2024-02, ...)
-- отдельное соединение events_2024_01 с events_archive_2024_01 и т.д.
```

**Пошагово: как это выглядит в плане.**

1. Без partition-wise join был бы один большой узел: Hash Join между «всей» таблицей events и «всей» events_archive (или Merge Join с сортировкой обеих сторон). В памяти держалась бы одна из таблиц целиком (или большие куски).

2. С partition-wise join планировщик разбивает работу: Join(events_2024_01, events_archive_2024_01), Join(events_2024_02, events_archive_2024_02), … Каждая пара партиций соединяется **независимо**; результаты объединяются через Append. В каждом Join участвуют только данные одной партиции с каждой стороны — объём меньше, хеш-таблица или сортировка меньше.

3. Для этого партиции должны **соответствовать**: у events партиция за январь 2024, и у events_archive тоже партиция за январь 2024 с теми же границами. Тогда строки из events_2024_01 могут соединяться только со строками из events_archive_2024_01 (по смыслу ключа партиционирования). Если границы разные или партиции не совпадают, планировщик не сможет сделать partition-wise join.

**Простыми словами.**  
Partition-wise join — «складывать ящики парами»: первый ящик первой таблицы с первым ящиком второй, второй с вторым и т.д. Так не нужно держать в памяти всю таблицу — только пару партиций.

**Проверь себя.**

1. Когда планировщик может выбрать partition-wise join?
<details>
<summary>Ответ</summary>
Когда обе таблицы **партиционированы совместимо**: одинаковый тип (RANGE/LIST/HASH) и **совпадающие границы партиций** (или одна таблица — подмножество партиций другой). И соединение должно идти по ключу партиционирования или по условию, сохраняющему соответствие партиций (например, e.created_at = a.created_at и оба — ключи партиционирования). Тогда планировщик может сопоставить партиции одна к одной и соединять пары отдельно.
</details>

2. В чём выгода partition-wise join по сравнению с одним большим Join?
<details>
<summary>Ответ</summary>
**Меньше памяти:** каждая пара партиций меньше по размеру — хеш-таблица или сортировка для одной пары меньше, чем для целых таблиц. **Лучше кэш:** данные одной партиции помещаются в кэш. **Параллелизм:** разные пары партиций можно соединять разными воркерами параллельно.
</details>

**Как запомнить.**  
Partition-wise join = **одинаковые границы партиций** у двух таблиц + JOIN по ключу партиционирования (или условие, сохраняющее пары). В плане — не один большой Join, а много маленьких Join по парам партиций (партиция A1 с B1, A2 с B2, …) и Append. Выгода: меньше памяти, лучше кэш, можно параллелить.

**Запомните.**  
Partition-wise join — соединение по партициям при совместимом партиционировании; меньше памяти, часто лучше параллелизм. В плане — несколько Join по парам партиций и Append.

### 74.4. Partition-wise aggregate

**Цель раздела.**  
Понять **partition-wise aggregate**: агрегация (GROUP BY, SUM, COUNT и т.д.) по партиционированной таблице может выполняться **по каждой партиции отдельно**, с последующим объединением результатов (например, суммированием сумм, сложением счётчиков). Планировщик может выбирать такой план, когда это выгодно.

**В этом разделе главное (три строки).** 1) **Partition-wise aggregate** — это когда вместо одного большого узла Aggregate по всей таблице планировщик разбивает работу: **по каждой партиции** выполняется своя агрегация (Partial Aggregate), затем частичные результаты **объединяются** (Finalize Aggregate: для SUM — суммируются суммы по группам, для COUNT — складываются счётчики). 2) **Выгода:** меньше данных в одном агрегационном узле — меньше памяти; можно параллельно считать по партициям разными воркерами. 3) Не все агрегатные функции «складываются» по частям: SUM, COUNT — да; AVG — через сумму и счётчик; MIN/MAX — в определённых условиях.

**Термины.**

- **Partition-wise aggregate** — выполнение агрегации по партициям: каждая партиция обрабатывается отдельно (своя группировка и агрегаты), затем частичные результаты объединяются (например, для SUM — суммируются суммы по группам, для COUNT — складываются счётчики).

**Теория и правила.**

- Для агрегатных функций, которые можно «складывать» по частям (SUM, COUNT, MIN, MAX в определённых условиях), планировщик может разбить работу по партициям: Partial Aggregate по каждой партиции, затем Finalize Aggregate поверх объединённых частичных результатов. Это уменьшает объём данных в одном агрегационном узле и может сочетаться с параллельным выполнением (каждая партиция — свой worker).
- В плане можно увидеть: Append над партициями → Partial Aggregate по каждой ветке → затем Finalize Aggregate (или объединение через GroupAggregate по уже сгруппированным ключам).

**Пошагово: как это работает на примере SUM.**

1. Запрос: `SELECT region, SUM(amount) FROM orders_by_region GROUP BY region;` Таблица orders_by_region партиционирована по region (LIST: eu, us, asia). Без partition-wise aggregate был бы один узел: прочитать все партиции, сгруппировать по region, посчитать SUM(amount).

2. С partition-wise aggregate: по партиции orders_eu — своя группировка и SUM (частичный результат по eu); по orders_us — своя (по us); по orders_asia — своя (по asia). Затем Finalize: для каждой группы (region) складываются частичные суммы. Для region = 'eu' в партиции orders_eu одна сумма, в других партициях по eu нет — итог по eu один. Для region, который есть в нескольких партициях (если бы так было), частичные суммы суммируются.

3. В плане можно увидеть: Append → для каждой партиции Partial HashAggregate → затем HashAggregate или GroupAggregate для финализации по ключам групп. Память: каждый Partial Aggregate обрабатывает только одну партицию — объём меньше.

**Простыми словами.**  
Вместо одного большого «посчитать по всей таблице» СУБД считает по каждому ящику отдельно, а потом складывает итоги (суммы, количества и т.д.).

**Проверь себя.**

1. Для каких агрегатных функций естественно «складывать» частичные результаты?
<details>
<summary>Ответ</summary>
**SUM:** сумма по партициям складывается. **COUNT:** счётчики по партициям складываются. **AVG:** хранятся сумма и счётчик по партициям, затем итоговый AVG = сумма сумм / сумму счётчиков. **MIN/MAX:** при определённых условиях (например, по ключу партиционирования) можно брать min/max по партициям и затем общий min/max.
</details>

**Как запомнить.**  
Partition-wise aggregate = **считаем по каждому ящику (партиции) отдельно**, потом **складываем** частичные результаты. SUM и COUNT складываются напрямую; AVG — через сумму и счётчик. В плане: Partial Aggregate по партициям → Finalize Aggregate. Меньше данных в одном узле — меньше памяти.

**Запомните.**  
Partition-wise aggregate — агрегация по партициям с объединением частичных результатов; выгодно по памяти и параллелизму при совместимых агрегатных функциях.

### 74.5. Субпартиционирование

**Цель раздела.**  
Понять **субпартиционирование** (subpartitioning): партиция сама может быть партиционированной таблицей (партиция внутри партиции). Например, сначала RANGE по году, внутри каждой годовой партиции — LIST по региону.

**В этом разделе главное (три строки).** 1) **Субпартиционирование** — это когда одна из партиций родителя **сама** объявлена как партиционированная таблица: у неё есть свои **подпартиции** (партиции второго уровня). Пример: партиция за 2024 год (events_2024) разбита на подпартиции по региону — events_2024_eu, events_2024_us и т.д. 2) **Создание:** сначала создаёшь партицию с PARTITION OF и **сразу** указываешь PARTITION BY для неё; затем создаёшь подпартиции через CREATE TABLE ... PARTITION OF events_2024 FOR VALUES ... 3) **Pruning** может работать на **обоих** уровнях: запрос с условием по году отсечёт лишние годовые партиции; с условием по году и региону отсечёт и год, и регион — останутся только нужные подпартиции.

**Термины.**

- **Субпартиционирование** — партиционированная таблица, у которой одна или несколько партиций сами объявлены как партиционированные (имеют свои подпартиции). Уровней может быть несколько (партиция → подпартиция → …).

**Теория и правила.**

- В PostgreSQL партиция создаётся как таблица с `PARTITION OF` родителя. Эта таблица, в свою очередь, может быть объявлена как партиционированная: создаётся как родитель с `PARTITION BY ...`, а затем создаются её партиции (подпартиции). В итоге получается иерархия: родитель → партиции → подпартиции.
- Пример: таблица по RANGE (год); партиция `events_2024` — сама партиционирована по LIST (регион). Тогда создаётся `events_2024` PARTITION OF events FOR VALUES FROM (...) TO (...), затем `CREATE TABLE events_2024_eu PARTITION OF events_2024 FOR VALUES IN ('eu');` и т.д.
- Планировщик может применять partition pruning на каждом уровне: сначала отсечь по году, затем внутри года по региону.

**Пример (структура).**

```sql
-- Родитель по RANGE (год)
CREATE TABLE events (...)
PARTITION BY RANGE (created_at);

CREATE TABLE events_2024 PARTITION OF events
    FOR VALUES FROM ('2024-01-01') TO ('2025-01-01')
PARTITION BY LIST (region);

-- Подпартиции events_2024
CREATE TABLE events_2024_eu PARTITION OF events_2024 FOR VALUES IN ('eu');
CREATE TABLE events_2024_us PARTITION OF events_2024 FOR VALUES IN ('us');
```

**Пошагово: как создаётся субпартиционирование.**

1. Родитель: `CREATE TABLE events (...) PARTITION BY RANGE (created_at);` Партиция за год: `CREATE TABLE events_2024 PARTITION OF events FOR VALUES FROM ('2024-01-01') TO ('2025-01-01') PARTITION BY LIST (region);` Обрати внимание: у events_2024 сразу указано **PARTITION BY LIST (region)** — значит, это не «листовая» партиция, а ещё один уровень; данные в events_2024 напрямую не хранятся, они будут в подпартициях.

2. Подпартиции: `CREATE TABLE events_2024_eu PARTITION OF events_2024 FOR VALUES IN ('eu');` `CREATE TABLE events_2024_us PARTITION OF events_2024 FOR VALUES IN ('us');` Строка с created_at в 2024 году и region = 'eu' попадёт в **events_2024_eu**; с region = 'us' — в **events_2024_us**. В таблицу events_2024 напрямую вставлять нельзя — у неё нет своего хранилища, только подпартиции.

3. Запрос `SELECT * FROM events WHERE created_at >= '2024-06-01' AND region = 'eu'` — планировщик отсечёт по году (останется events_2024) и по региону (останется events_2024_eu). В плане будет сканироваться только **events_2024_eu**.

**Простыми словами.**  
Субпартиционирование — «ящики внутри ящика»: сначала разбили по годам, внутри года разложили по регионам.

**Проверь себя.**

1. Где физически хранятся строки при субпартиционировании — в партиции второго уровня (events_2024) или в подпартициях (events_2024_eu)?
<details>
<summary>Ответ</summary>
В **подпартициях** (events_2024_eu, events_2024_us и т.д.). Партиция events_2024 — это «родитель» второго уровня: у неё нет своего хранилища, только список подпартиций. Данные лежат только в «листовых» партициях — в данном случае в events_2024_eu, events_2024_us.
</details>

2. Как partition pruning работает при двух уровнях партиционирования?
<details>
<summary>Ответ</summary>
На **каждом уровне** отдельно. Сначала по условию по первому ключу (например, created_at) отсекаются партиции первого уровня — остаётся, например, events_2024. Затем по условию по второму ключу (region) отсекаются подпартиции — остаётся, например, только events_2024_eu. В итоге сканируется только нужная подпартиция.
</details>

**Как запомнить.**  
Субпартиционирование = **партиция сама с PARTITION BY** (у неё свои подпартиции). Данные лежат только в **листовых** таблицах (самый нижний уровень). Pruning — сначала по первому ключу, потом по второму; в итоге сканируется одна подпартиция.

**Запомните.**  
Субпартиционирование — партиция объявлена как партиционированная таблица и имеет свои подпартиции. Данные только в «листовых» подпартициях. Pruning может работать на каждом уровне.
