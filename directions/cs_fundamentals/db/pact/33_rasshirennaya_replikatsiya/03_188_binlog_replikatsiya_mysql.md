## 188. Binlog‑репликация (MySQL)

### В этом разделе главное

- В MySQL основой репликации служит **binlog** — бинарный журнал всех изменений.
- Есть три формата записи в binlog: **statement‑based**, **row‑based** и **mixed** — каждый с компромиссами между точностью и объёмом.
- **GTID‑репликация** упрощает управление репликами, позволяя идентифицировать каждую транзакцию глобальным ID.
- Реплика получает события из binlog мастера, хранит их в **relay log** и применяет у себя.
- Основной показатель задержки — `Seconds_Behind_Master` (и другие показатели lag).

---

### Простыми словами

MySQL тоже ведёт **журнал изменений** — binlog.  
Репликация работает так:

- master пишет каждую транзакцию в binlog;
- slave:
  - читает этот журнал по сети;
  - записывает его во **временный relay log**;
  - «проигрывает» события у себя.

У binlog есть **форматы**:

- **statement‑based**: записываем SQL;
- **row‑based**: записываем сами изменения строк;
- **mixed**: MySQL выбирает, что ему выгоднее.

---

### Теория и правила

#### Форматы binlog

1. **Statement‑based (SBR)**:
   - в binlog попадает **текст SQL‑операции**:
     - `UPDATE accounts SET balance = balance - 100 WHERE id = 42;`
   - плюс контекст (например, значение `NOW()`).
   - плюсы:
     - компактно;
     - меньше трафика и диска.
   - минусы:
     - не все запросы детерминированы (функции `RAND()`, `NOW()`, триггеры);
     - возможно расхождение между master и slave при сложных конструкциях.

2. **Row‑based (RBR)**:
   - записываются **конкретные изменения строк**:
     - «была строка с такими полями, стала с такими».
   - плюсы:
     - детерминированно;
     - подходит для сложных операций, триггеров, нестандартного SQL.
   - минусы:
     - объёмнее;
     - особенно для массовых изменений.

3. **Mixed**:
   - MySQL автоматически переключается:
     - простые запросы — statement;
     - потенциально недетерминированные — row.

#### GTID (Global Transaction ID)

- GTID — это **уникальный идентификатор транзакции** в кластере:
  - обычно включает ID сервера и номер транзакции.
- Идея:
  - каждый мастер/реплика может сказать:
    - «я уже применил транзакции с такими GTID»;
  - при подключении новой реплики:
    - не нужно гадать, с какого места binlog начинать;
    - система сама определяет набор недостающих транзакций.

GTID сильно упрощает:

- переключение ролей master/slave;
- добавление новых реплик;
- восстановление после сбоев.

---

### Разбор на реальных сценариях binlog‑репликации

#### Сценарий 1. «Обычный master‑slave для разгрузки чтений»

У тебя:

- один **главный сервер MySQL** (master);
- к нему подключены приложения, которые:
  - много пишут (INSERT/UPDATE/DELETE);
  - много читают (SELECT).

Ты хочешь:

- разгрузить master от части SELECT;
- иметь **копию данных**, с которой можно:
  - запускать отчёты;
  - делать тяжёлые выборки.

Что обычно делают:

1. Включают binlog на мастере:
   - `log_bin = ON`;
   - выбирают формат `ROW` или `MIXED` для надёжности.
2. Делают **слепок данных** мастера:
   - через `mysqldump` или физический бэкап;
   - записывают **номер binlog‑файла и позицию**, с которой нужно продолжать.
3. Поднимают slave:
   - разворачивают бэкап;
   - говорят ему `CHANGE MASTER TO ... MASTER_LOG_FILE=..., MASTER_LOG_POS=...`;
   - запускают `START SLAVE;`.
4. Slave:
   - догоняет изменения из binlog мастера;
   - становится **почти актуальной копией**.
5. Приложения:
   - продолжают писать только на master;
   - часть SELECT перенаправляют на slave (обычно read‑only).

На что смотреть:

- `SHOW SLAVE STATUS\G` → `Seconds_Behind_Master`:
  - если там десятки секунд/минут — читаешь **устаревшие данные**;
  - если ≈ 0 — реплика догнала master.

Типичный компромисс:

- для «тяжёлых» отчётов, где допустимы данные с лагом в минуты, читать со slave нормально;
- для критичных запросов «покажи только что сделанный заказ» — нужно читать с master.

#### Сценарий 2. «Сделать тестовый стенд, максимально похожий на прод»

Ты хочешь:

- иметь тестовую базу, в которой:
  - много реальных данных;
  - регулярно обновляются последние изменения;
- но:
  - не хочешь давать тестам запись прямо в прод;
  - хочешь иногда «замораживать» состояние, чтобы проводить эксперименты.

Решение:

1. Делаешь отдельный slave:
   - он получает binlog от мастера как обычно;
   - на нём **запрещаешь запись** (только чтение).
2. Периодически:
   - делаешь снапшоты этого slave (dump/backup);
   - разворачиваешь их как **отдельные тестовые базы**.

Плюс binlog‑репликации:

- ты получаешь максимально «живые» данные;
- твой тестовый стенд не отстаёт от продакшена на месяцы.

#### Сценарий 3. «Переезд на новый мастер с минимальными потерями»

Если старый мастер:

- перегружен;
- старый «железяка»;
- или нужно сменить версию MySQL,

можно:

1. Поднять **новый сервер** и сделать его **репликой** старого мастера.
2. Подождать, пока `Seconds_Behind_Master` станет ≈ 0.
3. На короткое время остановить запись на старый мастер:
   - остановить приложение, которое пишет;
   - дождаться, пока новый сервер полностью дойдёт по binlog.
4. Переключить приложение:
   - новый сервер становится **новым мастером**;
   - старый можно оставить как реплику или выключить.

Здесь binlog даёт:

- непрерывную «линию изменений»;
- возможность аккуратно **пересадить роль мастера** на другую машину.

---

### Мини‑практика: почувствовать разницу форматов binlog

Идея:

1. Подними MySQL в контейнере с включённым binlog.
2. Сделай небольшую таблицу, например:

   ```sql
   CREATE TABLE demo (
     id INT PRIMARY KEY AUTO_INCREMENT,
     x  INT,
     ts TIMESTAMP
   );
   ```

3. Поиграй с форматами:
   - установи `binlog_format = STATEMENT`;
   - сделай пару запросов, использующих `NOW()`, `RAND()`;
   - вызови `SHOW BINLOG EVENTS\G` (или используй `mysqlbinlog`), чтобы посмотреть, что именно записалось.
4. Потом:
   - установи `binlog_format = ROW`;
   - сделай те же операции;
   - посмотри, как выглядит binlog в row‑формате.

Задача:

- увидеть, что:
  - statement‑формат хранит **текст SQL**;
  - row‑формат хранит **конкретные изменения строк**.

---

### Как диагностировать проблемы binlog‑репликации

Простейший чек‑лист:

1. **`SHOW SLAVE STATUS\G`**:
   - `Slave_IO_Running` и `Slave_SQL_Running` должны быть `Yes`;
   - `Seconds_Behind_Master` не должен расти бесконечно.
2. **Если `Slave_IO_Running = No`**:
   - проблема с подключением к мастеру:
     - сеть, права, пароль, адрес;
   - смотри поля `Last_IO_Error`.
3. **Если `Slave_SQL_Running = No`**:
   - проблема с применением binlog на реплике:
     - конфликт ключей;
     - отсутствующая таблица/колонка;
     - несовместимая схема;
   - смотри `Last_SQL_Error`.
4. **Если lag растёт**:
   - посмотри:
     - загруженность мастера (сильно ли он пишет в binlog);
     - загруженность реплики:
       - CPU/диск;
       - хватает ли ей ресурсов применять события.
   - возможно, реплика просто **не успевает**:
     - нужно усилить железо;
     - разделить нагрузку.
5. **Проблемы с binlog‑файлами**:
   - если binlog ротуются слишком агрессивно и удаляются:
     - реплика, которая долго была оффлайн, может «потерять» часть истории;
     - придётся делать новый полный бэкап.

---

### Как запомнить (мнемоника)

- **Binlog = хроника MySQL**:
  - это лента событий, которую можно сначала проиграть на реплике, а потом использовать для восстановления.
- **Statement = сценарий, Row = фильм**:
  - statement‑формат хранит сценарий («сыграем UPDATE…»),
  - row‑формат хранит уже отснятый материал (какие строки стали какими).
- **GTID = номер серии**:
  - по GTID любая реплика знает, какие серии (транзакции) она уже «посмотрела».
- **Seconds_Behind_Master = отставание по времени**:
  - чем оно больше, тем более «вчерашнюю» картинку ты видишь на реплике.

---


#### Как slave использует binlog

1. На master:
   - транзакции выполняются;
   - записываются в binlog.
2. На slave:
   - процесс IO‑thread:
     - читает binlog мастера;
     - пишет в `relay log`.
   - SQL‑thread:
     - читает `relay log`;
     - применяет события (SQL/row‑события) к своим данным.

При остановке:

- slave может позже догнать, пока binlog доступен (с диска или из архива).

#### Мониторинг lag: SHOW SLAVE STATUS

Исторически (и до переименования в source/replica) используется:

```sql
SHOW SLAVE STATUS\G
```

Из него часто берут:

- `Seconds_Behind_Master`:
  - оценка, **на сколько секунд отстаёт** реплика;
- состояние IO/SQL‑потоков (Running/Not running);
- позиции в master log / relay log.

Важно понимать:

- `Seconds_Behind_Master` — **оценка**, а не идеальный показатель;
- но он даёт интуицию:
  - реплика «догнала» master или сильно отстала.

---

### Картинка в голове

Мастер MySQL — это **редактор газеты**, который ведёт ежедневный дневник событий (binlog).

- Каждая реплика — «типография», которая:
  - забирает свежие выпуски дневника;
  - печатает их у себя (relay log → данные).

Форматы:

- statement‑based — как хранить **текст статьи**;
- row‑based — как хранить уже **готовую вёрстку страниц**.

GTID:

- это **номер выпуска и страницы**, по которым любая типография понимает:
  - «какие выпуски у меня уже есть»;
  - «какие нужно ещё забрать».

---

### Типичные ошибки

- Использовать statement‑based формат для сложных/недетерминированных запросов и потом удивляться расхождению данных.
- Игнорировать `Seconds_Behind_Master` и другие сигналы lag‑а.
- Не понимать, что **асинхронная** binlog‑репликация может потерять последние транзакции при аварии master.
- Смешивать конфигурации с/без GTID без чёткого плана миграции.

---

### Проверь себя

<details>
<summary>1. В чём основное отличие statement‑based и row‑based форматов binlog?</summary>

- **Statement‑based**:
  - пишет текст SQL‑запросов + контекст;
  - компактнее, но может быть недетерминированным.

- **Row‑based**:
  - пишет конкретные изменения строк (до/после);
  - надёжнее при репликации, но объёмнее.

</details>

<details>
<summary>2. Зачем MySQL ввели GTID?</summary>

Чтобы:

- упростить управление репликами:
  - легче подключать новые;
  - проще переключать роли master/slave;
  - проще понимать, **какие транзакции уже применены**;
- исключить ручную работу с позициями в binlog (номер файла, offset).

</details>

<details>
<summary>3. Что показывает Seconds_Behind_Master?</summary>

Это оценка того, **на сколько секунд отстаёт реплика**:

- по времени применения последней транзакции;
- при `0` — обычно означает, что реплика догнала master (с учётом погрешности).

</details>
