## 191. Репликация Redis

### В этом разделе главное

- Базовый режим Redis — **master‑replica** с асинхронной репликацией.
- **Redis Sentinel** мониторит мастера и реплики, и при падении мастера **переключает одну из реплик в master**.
- **Redis Cluster** добавляет к репликации ещё и **шардирование** по hash‑слотам.
- Репликация Redis, как правило, асинхронна, поэтому есть риск **потерять последние записи** при аварии мастера.

---

### Простыми словами

Redis:

- по умолчанию:
  - один узел (master) принимает все записи;
  - другие могут выступать как **реплики**, получая данные асинхронно.

Если master падает:

- без дополнительной логики:
  - тебе нужно вручную назначать нового master.

Sentinel:

- это отдельный компонент, который:
  - **следит** за мастер‑нодой;
  - при падении:
    - **выбирает реплику**;
    - делает её master;
    - уведомляет клиентов.

Redis Cluster:

- делит key‑пространство на **16384 hash‑слота**;
- каждый слот «принадлежит» определённой ноде (master);
- у каждого master есть реплики для отказоустойчивости.

---

### Теория и правила

#### Master‑replica

- Настраивается директивой `replicaof`/`slaveof`:
  - реплика подключается к мастеру;
  - делает **полную синхронизацию**:
    - получает snapshot (RDB);
    - затем поток изменений (AOF/команды).
- Репликация **асинхронная**:
  - мастер не ждёт подтверждения реплики;
  - возможен lag и потеря последних команд.

#### Redis Sentinel

- Набор процессов (Sentinel), которые:
  - опрашивают мастер и реплики;
  - обмениваются информацией друг с другом;
  - приходят к **согласию (кворум)** о том, что мастер действительно «упал».
- При признании мастера недоступным:
  - выбирается новая master‑нода из реплик;
  - на репликах переписываются конфигурации;
  - Sentinels уведомляют клиентов (через известные механизмы) о новом мастере.

#### Redis Cluster

- Ключи распределяются по **hash‑слотам** с помощью хеша от ключа:
  - всего 16384 слота;
  - каждый слот закреплён за одной master‑нодой;
  - у каждого мастер‑слота есть одна или более реплик.
- Репликация между master и replica:
  - похожа на обычный Redis master‑replica;
  - тоже асинхронная.
- При падении master:
  - реплика слота может быть **повышена** до master;
  - кластер перенастраивает маршрутизацию.

---

### Разбор на реальных сценариях Redis‑репликации

#### Сценарий 1. «Простой master‑replica + Sentinel для отказоустойчивости»

Исходные условия:

- Redis используется:
  - как кэш;
  - или как простое хранилище сессий;
- падение Redis:
  - не должно ломать всё приложение;
  - но возможная **потеря последних ключей** (сессий/кэша) допустима.

Что делают:

1. Запускают:
   - один master Redis;
   - одну‑две реплики (`replicaof master_host master_port`).
2. Настраивают Redis Sentinel:
   - несколько процессов Sentinel следят за мастер‑нодой;
   - если мастер «исчезает»:
     - они голосуют, действительно ли он недоступен;
     - выбирают одну из реплик как нового master;
     - перенастраивают другие реплики и уведомляют клиентов.
3. Приложение:
   - вместо того, чтобы ходить на `master:6379` напрямую;
   - общается с Redis через адрес/порт, который Sentinel/клиентская библиотека умеет переключать на новый master.

Компромисс:

- при падении мастера:
  - может потеряться часть последних команд, не успевших доехать до реплики;
  - но кластер поднимется **сам**, без ручного вмешательства.

#### Сценарий 2. «Redis Cluster для большого объёма ключей»

Когда:

- один инстанс Redis по памяти/нагрузке **уже не тянет**;
- нужно распределить ключи по нескольким нодам.

Решение:

1. Поднять Redis Cluster:
   - несколько master‑нод;
   - у каждой — реплики;
   - ключи распределяются по 16384 hash‑слотам.
2. Клиент:
   - по ключу вычисляет slot;
   - узнаёт, какая нода хранит этот slot;
   - отправляет команды туда.

Репликация:

- каждая master‑нода:
  - имеет свои реплики;
  - при падении master для конкретных слотов:
    - соответствующая реплика поднимается в master;
    - слоты «переезжают» на неё.

Важно:

- Redis Cluster — **не магическое распределённое хранилище с сильной консистентностью**;
- по‑прежнему:
  - репликация асинхронна;
  - возможна потеря последних записей при резком падении мастера.

#### Сценарий 3. «Почему Redis не всегда подходит как «единственная база»»

Распространённая ошибка:

- использовать Redis:
  - как **основное хранилище** критичных данных (заказы, деньги);
  - при этом полагаться только на его асинхронную репликацию + Sentinel.

Проблема:

- при failover:
  - часть последних записей может не попасть на реплику;
  - новый master не будет их содержать.

Пример:

1. Клиент записывает:
   - `SET balance:42 1000` на master;
   - мастер подтверждает запись клиенту;
   - реплика ещё не успела получить эту команду.
2. Мастер умирает.
3. Sentinel поднимает реплику как нового master:
   - но на ней **старое значение** или вообще нет ключа.

Если Redis используется как кэш:

- это терпимо (ключи потом восстановятся из основной БД).

Если Redis — **единственное хранилище**:

- такие потери могут быть фатальны.

---

### Мини‑практика: мысленная модель Redis + Sentinel

Представь:

1. Один кассовый аппарат (Redis master) и один резервный (replica).
2. Администраторы (Sentinels):
   - периодически проверяют, работает ли основной аппарат.
3. Если аппарат ломается:
   - администраторы собираются и голосуют:
     - «да, он точно сломался»;
   - объявляют резервный аппаратом «новым главным»;
   - рассылают всем продавцам объявление:
     - «отныне ходим к кассе №2».

Если в момент поломки:

- касса №1 пробила чек, но ещё не передала информацию кассе №2;
- то:
  - в журнале новой кассы (реплики) **этот чек не будет записан**.

Это и есть «окно потери данных» при асинхронной репликации.

---

### Как диагностировать проблемы Redis‑репликации

Основные моменты:

1. **Lag реплик**:
   - можно оценивать по метрикам:
     - задержка синхронизации;
     - разница в ключах/командах;
   - если lag большой:
     - эта реплика не годится как новый master без риска потерь.
2. **Состояние Sentinel**:
   - достаточно ли Sentinels (обычно 3+);
   - сходятся ли они в оценке состояния master;
   - нет ли «флаппинга», когда мастер часто объявляют то мёртвым, то живым.
3. **Сетевые проблемы**:
   - потеря связи между master и репликами;
   - потеря связи между Sentinels;
   - могут приводить к:
     - ложным failover;
     - ситуациям, когда несколько нод на короткое время считают себя master.

---

### Как запомнить (мнемоника)

- **Redis master‑replica = главный и зеркало**:
  - главный кассовый аппарат;
  - зеркало, которое с небольшой задержкой повторяет все операции.
- **Sentinel = дежурный админ**:
  - следит, жив ли главный;
  - если нет — назначает зеркало новым главным и сообщает всем.
- **Redis Cluster = много касс + распределение товаров**:
  - каждый товарный диапазон (hash‑слот) закреплён за своей кассой;
  - у каждой кассы есть резервная.
- **Асинхронность = риск потери последних чеков**:
  - если касса ломается до того, как зеркало успело скопировать последний чек, он пропадает.

---

### Картинка в голове

Redis master‑replica — это:

- один «главный кассовый аппарат» (master);
- несколько «зеркал» (реплики), которые:
  - получают поток чеков и операций;
  - отображают состояние кассы с небольшой задержкой.

Sentinel:

- это «дежурный администратор», который:
  - следит за кассами;
  - если главный аппарат сломался:
    - объявляет один из резервных «новым главным»;
    - рассказывает всем кассирам и продавцам, куда теперь ходить.

Redis Cluster:

- это уже **несколько касс**, каждая обслуживает **свой диапазон товаров** (hash‑слоты);
- у каждой кассы есть резервная;
- при поломке одной:
  - резервная становится основной;
  - схема обслуживания корректируется.

---

### Типичные ошибки

- Думать, что Redis‑репликация гарантирует **надёжное хранение критичных данных** без потерь.
- Не понимать, что failover через Sentinel может привести к **потере последних записей**, которые не успели доехать до реплик.
- Использовать Redis Cluster, как обычную «одну большую базу», не учитывая распределение по hash‑слотам и возможные ошибки MOVED/ASK.
- Не тестировать сценарии failover и неожиданно получать «дырки» в данных или конфигурации.

---

### Проверь себя

<details>
<summary>1. Почему при failover Redis с Sentinel возможна потеря данных?</summary>

Потому что:

- репликация асинхронна;
- master может подтвердить запись клиенту;
- но реплика ещё не успела получить/применить эту команду;
- если master падает до передачи, новая master‑нода **не будет содержать эти изменения**.

</details>

<details>
<summary>2. Что такое hash‑слоты в Redis Cluster?</summary>

Это:

- разбитое на 16384 «коробочек» пространство ключей;
- каждый hash‑слот закреплён за конкретным master‑узлом;
- ключ по результату хеш‑функции попадает в один слот, а значит, на одну ноду.

</details>

<details>
<summary>3. Зачем нужен Redis Sentinel, если уже есть репликация?</summary>

Репликация:

- просто создаёт копии данных на репликах.

Sentinel:

- следит за здоровьем нод;
- **обнаруживает падение master**;
- автоматически повышает одну из реплик до master;
- оповещает клиентов.

Без Sentinel failover приходится делать вручную.

</details>
