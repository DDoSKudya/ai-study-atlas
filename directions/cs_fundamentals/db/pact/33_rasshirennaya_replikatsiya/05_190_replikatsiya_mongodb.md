## 190. Репликация MongoDB

### В этом разделе главное

- Основная единица отказоустойчивости MongoDB — **replica set** (primary + несколько secondary).
- Все записи идут на **primary**, а secondary **асинхронно** применяют операции из **oplog**.
- При падении primary запускается **выбор нового primary** (election).
- Параметры **read preference** и **write concern** позволяют управлять компромиссом между консистентностью, задержкой и доступностью.

---

### Простыми словами

MongoDB replica set — это:

- один узел, который **принимает все записи** (primary);
- несколько копий (secondary), которые:
  - читают из журнала операций (oplog);
  - воспроизводят те же операции у себя.

Если primary падает:

- оставшиеся ноды собираются и решают:
  - «кто из нас станет новым primary»;
- клиенты, если правильно настроены:
  - автоматически переключаются.

При чтении и записи можно выбирать:

- насколько быстро и **из каких нод** читать;
- сколько нод должны **подтвердить запись**.

---

### Теория и правила

#### Replica set: роли нод

- **Primary**:
  - принимает все записи;
  - записывает операции в **oplog**.
- **Secondary**:
  - периодически забирают данные из oplog primary;
  - применяют их локально.
- **Arbiter** (опционально):
  - не хранит данные;
  - участвует в голосовании при выборе primary;
  - нужен, чтобы обеспечить нечётное число голосов.

#### Oplog

- Oplog (operations log) — это:
  - **специальная коллекция**, куда записываются операции:
    - «insert документ такой‑то»;
    - «update поля в документе №...»;
    - «delete документ».
- Secondary:
  - читают oplog primary;
  - применяют те же операции;
  - так «догоняют» primary.

Размер oplog:

- если он слишком мал:
  - при долгой остановке secondary:
    - нужные записи могут быть уже удалены;
  - тогда такую secondary нужно **пересинхронизировать полностью**.

#### Election: выбор primary

Когда:

- primary недоступен;
- secondary видят, что давно нет heartbeats;

они запускают **выборы**:

- ноды голосуют;
- учитываются:
  - приоритеты (priority);
  - актуальность данных (насколько далеко нода продвинулась по oplog);
- побеждает нода с:
  - достаточным уровнем приоритета;
  - самыми свежими данными;
  - поддержкой большинства голосов.

Клиентские драйверы:

- обычно настроены так, чтобы:
  - обнаруживать нового primary;
  - автоматически переключать записи на него.

### Разбор на реальных сценариях репликации MongoDB

#### Сценарий 1. «Классический replica set для боевой БД»

Исходные условия:

- у тебя одна Mongo‑нода;
- если она падает, всё приложение «умирает»;
- тебе нужно:
  - не терять данные;
  - минимизировать простой;
  - разгрузить чтения.

Что обычно делают:

1. Создают **replica set** из минимум трёх участников:
   - `primary`;
   - один‑два `secondary`;
   - иногда `arbiter` вместо одного secondary (если мало диска).
2. Приложение пишет на primary:
   - по умолчанию `writeConcern: { w: 1 }` (подтверждение только от primary);
   - или `w: "majority"`, если нужны более сильные гарантии.
3. Чтение:
   - по умолчанию `readPreference: primary` (строго консистентно);
   - для не столь критичных запросов:
     - `secondary` / `secondaryPreferred`, чтобы разгрузить primary.

Когда primary падает:

- оставшиеся ноды проводят **выборы**;
- новый primary выбирается автоматически;
- драйверы (если правильно настроены) переключаются на него.

Компромисс:

- больше нод → выше надёжность, но:
  - сложнее конфигурация;
  - дороже железо;
  - потенциально больше задержки при `writeConcern: "majority"`.

#### Сценарий 2. «Хотим быстрые чтения поближе к пользователям»

Представь:

- primary стоит в одном регионе (например, Европа);
- часть пользователей живёт в другом (например, США);
- если они читают строго с primary:
  - каждый запрос летит через океан;
  - задержка чувствуется.

Что делают:

1. Разворачивают replica set:
   - primary в Европе;
   - secondary в США.
2. Для пользователей из США:
   - настраивают драйвер/клиент:
     - `readPreference: nearest` или `secondaryPreferred`.
3. Они читают:
   - с ближайшей ноды (secondary в США);
   - иногда данные могут быть немного устаревшими (из‑за lag).

Если нужна более строгая консистентность:

- для критичных запросов (например, сразу после записи) можно:
  - делать `readPreference: primary`;
  - или поднимать write concern до `"majority"`.

#### Сценарий 3. «Как потерять данные, неправильно выбрав write concern»

Представь:

- у тебя replica set из 3 нод;
- все записи делаются с `writeConcern: { w: 1 }`.

Происходит:

1. Клиент пишет документ:
   - primary записывает его у себя;
   - отправляет подтверждение (`w:1`).
2. Но:
   - в этот момент secondary ещё не успели применить запись из oplog;
   - через секунду primary умирает (диск сломался).
3. Replica set выбирает нового primary:
   - одной из secondary;
   - на ней **этой записи может ещё не быть**.

Результат:

- запись, которую клиент считал успешно сохранённой, **теряется**.

Как уменьшить риск:

- использовать `writeConcern: { w: "majority" }`:
  - запись считается успешной только, если:
    - её подтвердило большинство нод replica set;
  - тогда падение одной ноды не уничтожит уже подтверждённые данные.

---

### Мини‑практика: мысленно покрутить readPreference и writeConcern

Сделай такую табличку (в уме или на бумаге):

| Сценарий | readPreference | writeConcern | Что получаем |
|----------|----------------|-------------|--------------|
| Критичные записи, читаем только «правду» | `primary` | `"majority"` | Максимально надёжно, но медленнее. |
| Обычные записи, допускаем редкую потерю при аварии | `primary` | `1` | Быстрее записи, риск потери последних операций. |
| Быстрый дешёвый отчёт, можно видеть вчерашнее | `secondaryPreferred` | `1` | Читаем с secondary, возможен lag, но разгружаем primary. |

Потренируйся:

- придумывать свои сценарии (например, метрики, логи, заказы);
- ставить туда комбинации readPreference + writeConcern;
- думать:
  - какие риски (устаревшие данные / потеря при сбое);
  - какие плюсы (скорость, разгрузка).

---

### Как диагностировать типичные проблемы replica set

Высокоуровневый чек‑лист:

1. **Lag secondary**:
   - смотри состояние вторичных нод (через `rs.status()` и мониторинг);
   - если одна secondary сильно отстаёт:
     - она не годится для чтений, где нужны свежие данные;
     - возможно, не хватает ресурсов (CPU/диск/сеть).
2. **Маленький oplog**:
   - если secondary была оффлайн долго:
     - часть нужных записей уже могла выбыть из oplog;
   - тогда ей придётся делать **полную ресинхронизацию**;
   - смотри размер oplog и историю простоев.
3. **Чтения с secondary без понимания lag**:
   - приложения жалуются на «странные» данные:
     - только что записали документ, но при чтении с secondary его не видят;
   - это не баг MongoDB — это следствие eventual consistency при чтении с secondary.
4. **Неправильный выбор writeConcern**:
   - `w: 1` для критичных данных → риск потери при падении primary;
   - `w: "majority"` в очень нагруженной системе → возможны задержки и таймауты.

---

### Как запомнить (мнемоника)

- **Replica set = семья кассиров**:
  - один главный кассир (primary) принимает деньги и пишет в журнал (oplog);
  - помощники (secondary) переписывают журнал к себе с небольшой задержкой.
- **readPreference = к какой кассе подойти за выпиской**:
  - только к главной (`primary`);
  - или к ближайшей/свободной (`secondary`, `nearest`), но там могут быть старые данные.
- **writeConcern = сколько кассиров должны расписаться под операцией**:
  - `w:1` — только главный;
  - `w:"majority"` — большинство из них; бумага почти точно не потеряется.
- **Oplog = тетрадь операций**:
  - если тетрадь слишком тонкая (маленький oplog), старые записи стираются быстрее, и отставшим придётся «учить всё заново» (полная ресинхронизация).

---


#### Read preference

Определяет, с каких нод можно **читать**:

- `primary`:
  - всегда читать только с primary;
  - максимум консистентности, но меньше масштабируемости.
- `primaryPreferred`:
  - обычно primary;
  - если он недоступен — можно secondary.
- `secondary`, `secondaryPreferred`, `nearest`:
  - читаем также с secondary и/или ближайшей ноды;
  - **меньше нагрузка на primary**, но есть лаг по данным.

#### Write concern

Определяет, сколько нод должны **подтвердить запись**:

- `w: 1`:
  - достаточно подтверждения от primary;
- `w: "majority"`:
  - нужно подтверждение от большинства реплик;
  - повышает надёжность, но увеличивает задержку;
- опционально:
  - `j: true` требует журналирования (fsync).

---

### Картинка в голове

MongoDB replica set — это:

- один «главный кассир» (primary), который:
  - принимает все платежи и пишет в свой журнал (oplog);
- несколько «помощников‑бухгалтеров» (secondary), которые:
  - регулярно копируют записи журнала и переписывают к себе.

Если главный кассир ушёл:

- помощники собираются и **выбирают нового главного** по принципу:
  - у кого свежие записи;
  - у кого выше приоритет;
  - кого поддерживает большинство.

Параметры read preference / write concern — это:

- **как быстро и из каких касс читать** (данные);
- **сколько кассиров должны расписаться** под одной операцией.

---

### Типичные ошибки

- Ставить слишком маленький oplog — secondary не успевают догонять и вынуждены пересинхронизироваться полностью.
- Читать с secondary данные, которые должны быть **строго консистентными**, не учитывая lag.
- Использовать `w: 1` при критичных записях, где потеря данных недопустима.
- Не понимать, как драйверы обрабатывают выборы primary, и получать неожиданные ошибки при failover.

### Проверь себя

<details>
<summary>1. Какую роль играет oplog в MongoDB?</summary>

Это:

- журнал операций (insert/update/delete), который:
  - записывается primary;
  - читается secondary;
- именно за счёт oplog secondary **повторяют операции** primary и становятся его копией.

</details>

<details>
<summary>2. Что такое read preference и почему его нужно настраивать осознанно?</summary>

Read preference:

- определяет, **с каких нод читать данные** (primary/secondary/nearest);
- влияет на:
  - консистентность (secondary могут отставать);
  - нагрузку (можно разгрузить primary);
  - задержку (чтение с ближайшей ноды может быть быстрее).

его нужно выбирать в зависимости от того, **насколько свежие данные нужны** и где находится клиент.

</details>

<details>
<summary>3. Что даёт write concern "majority"?</summary>

Он требует, чтобы:

- запись считалась успешной только после подтверждения **большинством реплик**;
- это:
  - уменьшает риск потери уже подтверждённых данных при сбое primary;
  - но увеличивает задержку записи.

</details>
