## 324. Паттерны кэширования

### 324.1. Зачем вообще кэшировать

**Цель раздела.**  
Понять, **зачем кэш вообще нужен**, где он даёт максимальный выигрыш по производительности и где может навредить из‑за рассинхронизации и сложности.

#### Термины

- **Источник истины (source of truth)** — место, где хранятся «настоящие» данные, которым нужно доверять (обычно основная БД).
- **Кэш** — быстрый, но потенциально «неточный» слой с копиями данных из источника истины.
- **Латентность** — задержка ответа (например, сколько миллисекунд занимает запрос).
- **Пропускная способность (throughput)** — сколько запросов/операций в единицу времени система может обработать.

#### Теория и правила

- Кэш уменьшает **среднее и p95/p99‑время ответа**, потому что **основная часть запросов читается из быстрой памяти**, а не из диска или распределённой сети.
- Кэш уменьшает **нагрузку на БД**, позволяя обслуживать больше пользователей тем же кластером.
- За это мы платим:
  - **сложностью инвалидации**: нужно понять, **когда считать кэш устаревшим**;
  - **возможной несогласованностью**: какое‑то время кэш может содержать старые данные;
  - **дополнительной инфраструктурой** (Redis‑кластеры, пулы соединений, мониторинг).
- Важно помнить: **кэш почти никогда не является источником истины**. Истина — в БД или другом хранилище; кэш — это **оптимизация**, которую **можно (и нужно) временно выключить**, если она ломается.

#### Простыми словами

Кэш — это как **записка с часто используемыми телефонами** рядом с рабочим местом. Можно каждый раз искать номер в общем справочнике (БД), но это дольше. Поэтому часто используемые номера мы выписываем на листок (кэш).  
Плюсы: **быстрее достать**, **меньше нагрузки** на «главный справочник». Минусы: если кто‑то поменял номер в главном справочнике, **листок не обновится сам**. Кто‑то должен **или регулярно переписывать листок**, или **стирать устаревший номер**.

Обрати внимание на один важный момент, который часто упускают.  
Мы не просто «хотим кэш, потому что он модный». На самом деле мы:

- **обмениваем часть точности и простоты** (всё читаем прямо из БД — просто, но иногда медленно),
- на **скорость и разгрузку БД** (иногда данные чуть старые, но зато система выдерживает нагрузку).

То есть, как только ты вводишь кэш, у тебя появляется **новый класс задач**:

- как понять, когда данные в кэше уже **слишком старые**;
- что делать, если БД и кэш **расходятся**;
- как вести себя, если **кэш недоступен** (работать без него, падать, включить деградацию?).

#### Картинка в голове

Представь трёхуровневый шкаф:

- нижняя полка — **диски с архивами** (самый медленный, но надёжный источник);
+- средняя полка — **БД на сервере** (быстрее, но всё ещё нужно дотянуться);
+- верхняя полка рядом с тобой — **кэш в памяти** (очень близко и быстро, но там лежат только некоторые книги и не всегда самые новые издания).

Чем **выше уровень**, тем **быстрее доступ**, но **меньше гарантий свежести** и **меньше объём**.

#### Как запомнить

- **Кэш = скорость в обмен на сложность**.
- **Источник истины всегда один**. Кэш — не истина, а **копия**.

#### Проверь себя

1. Почему кэш не должен быть «источником истины» в архитектуре?
2. Какие две основные метрики производительности улучшает кэш?

<details>
<summary>Ответ</summary>

1. Потому что кэш — это **копия** данных, которая может быть устаревшей или частично заполненной; если опираться на неё как на единственный источник, при сбоях или инвалидации можно потерять данные или принять неправильные решения. Источник истины должен быть **устойчивым и полным хранилищем** (обычно БД), а кэш — только оптимизация.  
2. Кэш уменьшает **латентность** (время ответа) и увеличивает **пропускную способность** (через него можно обработать больше запросов, не нагружая БД).

</details>

#### Запомните

- Кэш нужен, чтобы **снизить задержки** и **разгрузить БД**, но он **не заменяет** саму БД.
- Любое кэширование добавляет **сложность согласованности** — об этом нужно помнить с первого дня.

### 324.2. Паттерн cache‑aside

**Цель раздела.**  
Разобраться в классическом паттерне **cache‑aside**, понять, как он работает шаг за шагом, где он удобен и какие у него ограничения.

#### Термины

- **Cache‑aside (кэш сбоку)** — паттерн, при котором приложение само управляет кэшем: сначала смотрит туда, при промахе берёт данные из БД и кладёт их в кэш.
- **Cache hit (попадание)** — нужное значение найдено в кэше.
- **Cache miss (промах)** — значения в кэше нет.

#### Теория и правила

Алгоритм cache‑aside:

1. Приложение получает запрос «дай сущность X по ключу K».
2. Приложение **сначала смотрит в кэш** (Redis/Memcached) по ключу `entity:K`.
3. Если **hit** — возвращает данные из кэша.
4. Если **miss**:
   - делает запрос к БД (`SELECT ... WHERE id = K`);
   - получает результат и **кладёт его в кэш** с TTL;
   - возвращает результат клиенту.

При **обновлении данных** обычно:

- приложение сначала **обновляет БД** (`UPDATE ...`),
- затем **инвалидирует** ключи в кэше (`DEL entity:K`) или обновляет значение в кэше.

Cache‑aside:

- **прост в реализации** — логика в приложении, не нужен «умный» кэш;
- **гибкий** — можно по‑разному выбирать ключи и TTL;
- **позволяет легко отключать кэш** (обойтись без него на время).

Минусы:

- нужно **явно думать об инвалидации**;
- возможны **окна рассинхронизации**, когда БД обновлена, а кэш ещё нет.

#### Пошагово (пример)

Пусть у нас есть таблица пользователей:

```sql
CREATE TABLE users (
  id        bigint PRIMARY KEY,
  username  text NOT NULL,
  full_name text NOT NULL,
  updated_at timestamptz NOT NULL DEFAULT now()
);
```

И Redis в качестве кэша. Псевдокод на любом языке:

```pseudo
function getUser(id):
  key = "user:" + id
  cached = redis.get(key)
  if cached != null:
      return deserialize(cached)      -- cache hit

  -- cache miss: читаем из БД
  user = db.query("SELECT id, username, full_name FROM users WHERE id = $1", [id])
  if user == null:
      return null

  redis.setex(key, ttl=300s, value=serialize(user))  -- 5 минут
  return user
```

При обновлении:

```pseudo
function updateUserName(id, newName):
  db.execute("UPDATE users SET full_name = $1, updated_at = now() WHERE id = $2", [newName, id])
  key = "user:" + id
  redis.del(key)  -- инвалидация; при следующем чтении кэш заполнится заново
```

#### Простыми словами

Cache‑aside — это когда **приложение само ведёт блокнот с записями**. Сначала оно смотрит в блокнот (кэш), а если записи там нет, идёт в большую книгу (БД), находит нужную строку и **переписывает её в блокнот**, чтобы в следующий раз не листать книгу.

#### Картинка в голове

Представь библиотеку, где:

- библиотекарь — это приложение;
- большой архив — БД;
- маленькая полка возле стола библиотекаря — кэш.

Библиотекарь всегда **сначала смотрит на маленькую полку**. Если книги там нет — идёт в архив, приносит её и **кладёт на маленькую полку** для будущих запросов.

#### Как запомнить

- Cache‑aside = **«сначала в кэш, потом в БД»**.
- Инвалидация = **«выкинуть книгу с полки»** при изменении в архиве.

#### Типичные ошибки

- Забыть **инвалидировать кэш при изменениях** → пользователи видят старые данные.
- Не учитывать, что Redis может быть **недоступен** → приложение должно уметь работать «без кэша» (degrade gracefully).

#### Что будет, если…

- Если **кэш упадёт**, а приложение не готово работать без него → массовые промахи, все идут в БД, возможен перегруз.
- Если **TTL слишком большой**, а инвалидация не делается → пользователь долго видит устаревшие данные.

#### Проверь себя

1. Какой порядок действий при cache‑aside при промахе в кэше?
2. Почему при обновлении данных важно сначала обновить БД, а потом кэш?

<details>
<summary>Ответ</summary>

1. (1) Приложение смотрит в кэш → (2) при miss читает из БД → (3) записывает результат в кэш → (4) возвращает его клиенту.  
2. Если сначала обновить кэш, а потом БД, при сбое между операциями можно получить ситуацию, когда кэш содержит новое значение, а БД — старое. При восстановлении кэш может быть очищен, и в систему вернутся старые данные. Поэтому **источник истины (БД) обновляют первым**, а кэш — только как производную.

</details>

#### Запомните

- Cache‑aside — **дефолтный паттерн** кэширования для большинства веб‑приложений.
- Главный риск — **инвалидация и согласованность**, о них нужно думать заранее.

### 324.3. Read‑through и write‑through

**Цель раздела.**  
Понять, как работают паттерны **read‑through** и **write‑through**, чем они отличаются от cache‑aside и где применяются.

#### Термины

- **Read‑through** — приложение всегда читает только из кэша, а кэш при промахе сам ходит в БД.
- **Write‑through** — приложение пишет через кэш: запись в кэш синхронно «проталкивается» в БД.

#### Теория и правила

В **read‑through**:

- Приложение знает только кэш. Оно делает `get(key)`, и:
  - если значение есть, кэш возвращает его;
  - если нет, кэш **сам выполняет запрос к БД**, кладёт результат и возвращает его.
- Логика «как ходить в БД» зашита в кэширующем слое (например, в библиотеке или прокси).

В **write‑through**:

- Приложение делает `set(key, value)` в кэш;
- кэш **синхронно записывает** данные и в себя, и в БД;
- запись считается успешной только после успешной записи в БД (или по выбранной политике).

Комбинация read‑through + write‑through делает кэш **самодостаточным слоем**, где приложение «не знает» о существовании БД напрямую.

Плюсы:

- меньше дублирования логики в приложениях;
- кэш всегда **содержит актуальные данные** (при успешной записи);
- удобно при использовании **CDN / API Gateway** с вшитой логикой кэширования.

Минусы:

- кэш становится **критически важным компонентом**; его сбой = недоступность данных;
- сложнее отлаживать (часть логики в «чёрном ящике»);
- write‑through может быть **медленнее** (запись через кэш требует подтверждения БД).

#### Простыми словами

Read‑through — это когда библиотекарь **никогда сам не ходит в архив**. Он всегда обращается к помощнику (кэшу), а помощник сам решает, идти ли в архив.  
Write‑through — это когда библиотекарь отдаёт новую книгу помощнику, и тот **одновременно кладёт её и на маленькую полку, и в архив**.

#### Примеры (PostgreSQL, Redis, ORM)

В некоторых ORM есть встроенные кэши второго уровня с паттерном read‑through/write‑through: разработчик обращается к ORM, а тот уже решает, читать/писать из кэша или БД.

В микросервисах:

- сервис обращается к **API Gateway**, который **кэширует ответы** (read‑through) и при необходимости выполняет запросы к бекендам.

#### Типичные ошибки

- Рассчитывать, что read‑through/write‑through «магически» решит проблему инвалидации — **источник истины и кэш всё равно могут рассинхронизироваться** при сбоях.
- Не предусмотреть сценарий, когда кэш недоступен или медленный — тогда все запросы «залипают» на попытках дойти до БД через сломанный кэш.

#### Проверь себя

1. В чём главное отличие read‑through от cache‑aside?
2. Почему write‑through может быть медленнее прямой записи в БД?

<details>
<summary>Ответ</summary>

1. При cache‑aside **приложение само** решает, что при промахе нужно пойти в БД и положить результат в кэш. При read‑through приложение **знает только кэш**, а логика похода в БД и записи в кэш спрятана внутри кэширующего слоя.  
2. При write‑through запись идёт **через кэш**, который синхронно пишет и себя, и БД; это дополнительный слой сетевых и логических операций, поэтому latency обычно выше, чем при прямой записи в БД (особенно при сложной сериализации и проверках).

</details>

#### Запомните

- Read‑through и write‑through удобны там, где кэш — **официальный фронт** к данным.
- Они не отменяют необходимости думать о **согласованности и сбоях**.

### 324.4. Write‑behind (write‑back)

**Цель раздела.**  
Понять, как работает **write‑behind / write‑back**, где он даёт сильное ускорение и какие риски несёт.

#### Термины

- **Write‑behind / write‑back** — запись сначала в кэш, а затем **асинхронное** применение изменений к БД.
- **Durability (надёжность хранения)** — гарантия, что после подтверждённой записи данные не будут потеряны.

#### Теория и правила

В паттерне write‑behind:

- Приложение пишет `set(key, value)` в кэш.
- Кэш **считает операцию успешной**, хотя изменения ещё **не записаны** в БД.
- Отдельный процесс/поток **фоновой синхронизации** периодически выгружает накопленные изменения в БД (batch‑вставки/обновления).

Плюсы:

- **очень быстрое ощущаемое время записи** для клиента;
- возможность **пакетировать** множество мелких обновлений в крупные батчи к БД, уменьшая нагрузку.

Минусы:

- при падении кэша или процесса синхронизации **изменения могут потеряться**;
- БД и кэш могут **долго расходиться** по данным;
- сложно отлаживать и объяснять, почему «в отчёте одно, а в кэше другое».

Обычно write‑behind применяют:

- для **некритичных данных** (счётчики просмотров, лайки, метрики);
- для **логов и аналитики**, когда временная потеря части данных не критична.

#### Простыми словами

Write‑behind — это как **корзина с заявками на изменения**, которую кладут рядом со столом, а в «официальный журнал» эти изменения заносят раз в час. Клиенты счастливы: «заявка принята мгновенно», но если корзину унесли ветром, то под угрозой все необработанные заявки.

#### Пример (счётчик просмотров)

Типичный сценарий: счётчик просмотров статьи.

- Пользователь открывает страницу статьи → инкремент счётчика.
- Запись каждого инкремента в БД:
  - создаёт высокую нагрузку на запись;
  - может блокировать строки.

Write‑behind‑подход:

1. При каждом просмотре:
   - `INCR article:123:views` в Redis.
2. Раз в N секунд/минут:
   - процесс читает новые значения из Redis;
   - выполняет `UPDATE articles SET views = views + delta WHERE id = 123;`.

Если процесс упадёт, будут потеряны просмотры за последнее «окно». Для аналитики это приемлемо, для финансов — нет.

#### Типичные ошибки

- Применять write‑behind к **критичным данным** (балансы, заказы, транзакции).
- Не иметь **надёжного журнала** изменений (например, Kafka), который переживёт падение кэша.

#### Проверь себя

1. Почему write‑behind обычно не подходит для финансовых транзакций?
2. В каком типичном сценарии write‑behind оправдан и даёт большой выигрыш?

<details>
<summary>Ответ</summary>

1. Потому что при write‑behind подтверждение записи клиенту может прийти **до того, как изменения реально сохранены в БД**. При падении кэша или процесса синхронизации часть изменений может быть безвозвратно потеряна, что недопустимо для денег, заказов и других критичных операций.  
2. Для счётчиков просмотров, лайков, аналитики кликов и других **не критичных к потере пары процентов** данных сценариев: можно сильно уменьшить нагрузку на БД, копируя инкременты пакетами.

</details>

#### Запомните

- Write‑behind — это **трейдофф скорость ↔ надёжность**; применяй его только там, где **потеря части данных допустима**.

### 324.5. TTL, инвалидация и cache stampede

**Цель раздела.**  
Понять, как **управлять временем жизни данных в кэше**, как **инвалидировать** их и как **предотвращать кеш‑шторм**.

#### Термины

- **TTL (time‑to‑live)** — время жизни записи в кэше.
- **Инвалидация** — явное удаление/обновление кэшированной записи при изменении данных.
- **Cache stampede (кеш‑шторм)** — массовые промахи по одному и тому же ключу, ведущие к штурму БД.

#### Теория и правила

TTL нужен, чтобы:

- ограничивать время, пока устаревшие данные могут «жить» в кэше (через какое время мы готовы сказать: «это уже точно слишком старое значение, его надо пересчитать»);
- автоматически очищать неиспользуемые ключи (если ключ давно не запрашивали — он сам исчезнет).

Можно думать о TTL так:

- **короткий TTL** → данные точнее, но:
  - больше промахов в кэше;
  - чаще идём в БД;
  - больше нагрузка на БД;
- **длинный TTL** → нагрузка на БД меньше, но:
  - пользователи дольше могут видеть старые данные;
  - ошибки и баги в коде, положившие неправильное значение в кэш, будут «жить» дольше.

Поэтому TTL почти всегда — это **компромисс**:

- «насколько важна свежесть» vs «насколько нам дорого пересчитывать».

Инвалидация:

- **по записи** (write‑through, write‑around) — обновили БД → обновили/удалили кэш;
- **по TTL** — кэш автоматически «забывает» ключи через N секунд/минут;
- **по событиям** — приложение публикует события изменений, а слушатели инвалидируют кэш.

Cache stampede:

- Происходит, когда **популярный ключ истёк или был вычищен**, а сотни/тысячи клиентов почти одновременно запрашивают его.
- Все видят промах и идут в БД → резкий всплеск нагрузки, возможная деградация.

Стратегии борьбы:

- **Размазанный TTL (jitter)** — вместо ровно 300 секунд применять `300 ± random(0..60)`, чтобы ключи истекали не одновременно.
- **Mutex around recomputation** — только один поток/процесс «пересчитывает» значение, остальные ждут/получают старое значение.
- **Background refresh / refresh‑ahead** — обновлять популярные ключи заранее, до их истечения.

#### Простыми словами

TTL — это **дата окончания срока годности** записи в кэше. Когда срок вышел, продукт нужно выбросить или обновить. Если все банки йогурта в магазине имеют **одну и ту же дату истечения**, в день X на полках будет пусто (cache stampede) — все клиенты одновременно не найдут свой продукт и пойдут «в склад» (БД). Размазанный TTL и фоновые пополнения полок помогают избежать «штормов».

#### Пример (Redis, jitter)

```pseudo
function cacheSetWithJitter(key, value, baseTtlSeconds):
  -- добавляем случайный сдвиг до 20% от TTL
  jitter = random(0, baseTtlSeconds * 0.2)
  ttl = baseTtlSeconds + jitter
  redis.setex(key, ttl, serialize(value))
```

Mutex‑подход (очень упрощённо):

```pseudo
function getWithMutex(key):
  cached = redis.get(key)
  if cached not expired:
     return cached

  if acquire_lock("lock:"+key, ttl=5s) == false:
     -- другой процесс уже пересчитывает; подождать или вернуть старое значение
     sleep(100ms)
     return redis.get(key)  -- надеемся, что уже обновили

  value = recompute_from_db()
  redis.setex(key, 300s, value)
  release_lock("lock:"+key)
  return value
```

#### Типичные ошибки

- Ставить **одинаковый TTL на все ключи** и не добавлять jitter → синхронные истечения и шторма.
- При «шторме» пытаться «лечить» только БД (добавлять реплики), не исправляя паттерн кэширования.

#### Проверь себя

1. Что такое cache stampede и почему он опасен?
2. Назови два способа уменьшить риск cache stampede.

<details>
<summary>Ответ</summary>

1. Cache stampede — это ситуация, когда **много клиентов одновременно промахиваются по одному и тому же популярному ключу** (обычно после истечения TTL) и все идут за ним в БД. Это создаёт резкий всплеск нагрузки, может перегрузить БД и привести к деградации или падению сервиса.  
2. (1) Использовать **размазанный TTL (jitter)**, чтобы популярные ключи истекали не одновременно. (2) Применять **mutex/lock вокруг пересчёта** или **фоновые refresh‑ahead‑обновления**, чтобы в каждый момент только один процесс пересчитывал значение.

</details>

#### Запомните

- Ключ к безопасному кэшированию — **не только TTL, но и стратегия инвалидации и защиты от штормов**.
