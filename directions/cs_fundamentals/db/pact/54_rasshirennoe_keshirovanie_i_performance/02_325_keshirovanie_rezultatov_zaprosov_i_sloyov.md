## 325. Кэширование результатов запросов и слоёв

### 325.1. Redis/Memcached как кэш запросов

**Цель раздела.**  
Научиться использовать Redis/Memcached как **кэш результатов тяжёлых запросов**, выбирать ключи, TTL и понимать, где это помогает.

#### Термины

- **Query result cache** — кэш результатов запросов или их частей.
- **Key design** — проектирование ключей кэша: формат, составляющие, уровни.

#### Теория и правила

- Типичный паттерн: «дорогой SELECT» → **вытащить результат один раз**, сложить в Redis, дальше обслуживать большинство запросов из кэша.
- Ключ кэша обычно зависит от:
  - параметров запроса (id, фильтры, страница);
  - версии схемы/логики (для инвалидации при деплое);
  - иногда от **версии данных** (например, `updated_at`).

Давайте разложим это чуть медленнее.

1. Сначала мы **находим «дорогой» запрос**. Это не любой SELECT, а тот, который:
   - делает много JOIN’ов;
   - агрегирует большие объёмы;
   - или часто вызывается под нагрузкой (например, домашняя страница магазина в «чёрную пятницу»).
2. Потом спрашиваем себя: «Если бы мы могли **считать этот результат реже** и **раздавать копию** — помогло бы это?» Если да, значит, он кандидат на кэширование.
3. Затем нам нужно понять, при каких параметрах запрос считается «тем же самым». Например:
   - отчёт по магазину за **месяц** (store_id, месяц, год);
   - профиль пользователя (user_id);
   - список товаров в категории с конкретной сортировкой и страницей (category_id, sort, page).
4. Из этих параметров мы собираем **ключ кэша**, чтобы для одинаковых запросов ключ был одинаковым, а для разных — разным.

Пример ключа:

`user:123:profile:v2`  
`report:sales:2024-01:store-42:v1`

#### Пример (SQL + Redis)

```sql
-- тяжёлый агрегат по продажам за месяц
SELECT store_id,
       date_trunc('day', created_at) AS day,
       sum(amount) AS total
FROM sales
WHERE created_at >= '2024-01-01'
  AND created_at < '2024-02-01'
GROUP BY store_id, date_trunc('day', created_at);
```

Псевдокод кэширования:

```pseudo
function getMonthlySales(storeId, year, month):
  key = format("sales:%d:%04d-%02d", storeId, year, month)
  cached = redis.get(key)
  if cached != null:
     return deserialize(cached)

  rows = db.query("""
    SELECT date_trunc('day', created_at) AS day,
           sum(amount) AS total
    FROM sales
    WHERE store_id = $1
      AND created_at >= $2
      AND created_at < $3
    GROUP BY day
  """, [storeId, startOfMonth(year,month), endOfMonth(year,month)])

  redis.setex(key, ttl=3600s, value=serialize(rows))  -- час
  return rows
```

Для **PostgreSQL, MySQL, SQL Server** такой подход универсален: БД ничего «не знает» о кэше.

#### Простыми словами

Вместо того, чтобы каждый раз строить сложный отчёт «с нуля», мы делаем это **редко**, складываем результат в быстрый «ящик» (Redis) и **раздаём копию** всем желающим, пока она актуальна.

Можно провести ещё одну аналогию. Представь, что ты печатаешь **толстый отчёт на принтере**:

- печатать полный отчёт каждый раз, когда его кто‑то просит, — долго;
- гораздо выгоднее **распечатать один раз**, положить стопку копий на стойку и раздавать тем, кто просит;
- иногда отчёт нужно **обновить** (например, раз в час) — тогда ты печатаешь новую версию и убираешь старую стопку.

Redis в этом примере — это **стойка с распечатанными копиями**, а БД — это **система, где считаются реальные цифры**.

#### Типичные ошибки

- Кэшировать слишком **мелкие** результаты (каждую строку → overhead на сериализацию и сеть съедает выгоду).
- Не кэшировать **достаточно долго** — TTL в 5 секунд для тяжёлого отчёта часто бесполезен.

Ещё одна частая ошибка — пытаться кэшировать **слишком «живые» данные**:

- например, баланс кошелька, который меняется каждую секунду;
- или статус заказа, который в течение короткого времени проходит несколько состояний.

В таких случаях:

- выгода от кэширования маленькая (значение успевает устареть, пока его достают из кэша);
- риск показать пользователю **неактуальное** состояние выше.

Часто лучше кэшировать:

- не **сырые данные** (баланс по каждому действию),
- а **агрегаты среднего уровня** (например, суммарный оборот за день по магазину) или **редко меняющиеся части страницы** (описание товара, характеристики).

#### Проверь себя

1. От каких параметров должен зависеть ключ кэша для отчёта?
2. Почему бессмысленно кэшировать простой `SELECT * FROM users WHERE id = $1` с очень низким TTL, если у тебя уже есть индексы?

<details>
<summary>Ответ</summary>

1. От параметров запроса (например, `store_id`, месяц/год), а также, по возможности, от версии схемы/логики (`v1`, `v2`), чтобы при изменении запроса можно было легко инвалировать старый кэш.  
2. Индексированный `SELECT` по первичному ключу и так очень быстрый; накладные расходы на сериализацию/десериализацию и обращение к Redis могут быть сопоставимы или больше, чем просто запрос в БД, особенно если TTL мал и промахи часты. Кэшировать стоит **действительно тяжёлые** запросы.

</details>

#### Запомните

- Redis/Memcached как кэш запросов — мощный инструмент, но выигрывает прежде всего на **дорогих и часто повторяющихся** запросах.

### 325.2. Кэширование на уровне API Gateway и GraphQL

**Цель раздела.**  
Понять, как кэширование на уровне API Gateway/GraphQL дополняет кэш вокруг БД и когда его выгодно применять.

#### Термины

- **API Gateway cache / CDN cache** — кэширование HTTP‑ответов (по URL, заголовкам).
- **Persisted queries (GraphQL)** — заранее зарегистрированные запросы с короткими идентификаторами.

#### Теория и правила

- Многие API Gateway (Kong, AWS API Gateway, NGINX, Envoy) умеют кэшировать ответы:
  - по `Cache-Control`, `ETag`, `Last-Modified`;
  - по пути, query‑строке, заголовкам аутентификации.
- GraphQL поверх HTTP часто кэшируют через:
  - **persisted queries** — вместо длинного текста запроса клиент отправляет его ID;
  - HTTP‑кэш по `GET` запросам с этим ID.

Связь с БД:

- Кэш на уровне API **уменьшает число обращений к backend‑сервису**, а значит и число обращений к БД.
- Он хорошо подходит для **публичных/полустатичных данных** (каталог товаров, страницы контента).

#### Простыми словами

API Gateway и CDN — это как **витрина магазина**. Вместо того, чтобы каждый раз бежать на склад (бекенд + БД), они держат **готовые витрины** для популярных запросов. Если кто‑то хочет «всё то же самое, что и остальные» (например, домашняя страница каталога), он получает данные ещё быстрее, чем через Redis у приложения.

Чтобы почувствовать разницу между этим кэшем и кэшем в приложении, подумай так:

- **Кэш в приложении/Redis** работает **после** того, как запрос уже дошёл до бэкенда;
- **кэш на уровне API/CDN** может «поймать» запрос **раньше** — иногда вообще на другом континенте, не давая ему дойти ни до приложения, ни до БД.

#### Пример (HTTP‑кэш)

Ответ сервера:

```http
HTTP/1.1 200 OK
Cache-Control: public, max-age=60
ETag: "catalog-v42"
Content-Type: application/json

{ ... данные каталога ... }
```

CDN/API Gateway может кэшировать этот ответ на 60 секунд; за это время тысячи клиентов не дойдут до приложения и БД.

Теперь давай посмотрим, что происходит **пошагово**, когда приходит запрос на эту страницу:

1. Пользователь открывает `/catalog` в браузере.
2. Запрос попадает в **CDN/API‑шлюз**.
3. CDN смотрит: «У меня уже есть ответ для `/catalog` с валидным сроком `max-age=60` и тем же ETag?»  
   - Если **да** — он **сразу отдаёт сохранённый ответ** (даже не трогая твой backend).  
   - Если **нет** — запрос идёт в backend‑сервис → тот обращается к БД → формирует ответ → отдаёт его через CDN, и **CDN его запоминает**.
4. Для следующих пользователей в течение этих 60 секунд CDN будет просто **отдавать закэшированный ответ**, не нагружая приложение и БД.

Если данные меняются редко (например, список категорий товаров или баннеры), такая схема даёт огромный выигрыш по нагрузке и задержкам.

#### Проверь себя

1. Чем HTTP‑кэш на уровне API Gateway отличается от Redis‑кэша в приложении?
2. Почему для динамично изменяющихся персонализированных данных кэш API‑Gateway часто бесполезен?

<details>
<summary>Ответ</summary>

1. HTTP‑кэш работает на уровне **готовых HTTP‑ответов**, опирается на заголовки (`Cache-Control`, `ETag`) и часто расположен ближе к клиенту (CDN). Redis‑кэш в приложении работает на уровне **внутренних структур** (объекты, результаты SQL), и контролируется кодом приложения.  
2. Персонализированные ответы зависят от пользователя, токена, состояния сессии; их нельзя безопасно раздавать всем. Кэш API‑Gateway, работающий по URL и общим заголовкам, не сможет эффективно переиспользовать такие ответы между пользователями.

</details>

#### Запомните

- Кэш на уровне API/GraphQL особенно полезен для **публичных и полустатичных данных** и хорошо дополняет кэш вокруг БД.

### 325.3. Материализованные представления как persistent‑кэш

**Цель раздела.**  
Понять, как **материализованные представления** (матвью) работают как устойчивый кэш внутри БД и когда лучше их использовать вместо внешнего кэша.

#### Термины

- **Материализованное представление** (materialized view) — сохранённый результат запроса в виде таблицы, управляемой СУБД.
- **REFRESH MATERIALIZED VIEW** — команда обновления содержимого матвью в PostgreSQL.

#### Теория и правила

Матвью:

- содержит **результат сложного запроса** (агрегаты, джойны);
- хранится как **обычная таблица**, по которой можно строить индексы;
- обновляется:
  - **полным refresh**: пересчитать всё;
  - в некоторых СУБД — **инкрементально** (Oracle, SQL Server, некоторые расширения PostgreSQL).

Плюсы:

- нет отдельного внешнего кэша, всё внутри БД;
- СУБД обеспечивает **транзакционность** при refresh;
- можно сочетать с индексацией, планировщиком запросов.

Минусы:

- refresh может быть тяжёлым и занимать время;
- нужна дисциплина: **когда и как обновлять**.

#### Пример (PostgreSQL)

```sql
CREATE MATERIALIZED VIEW sales_daily AS
SELECT store_id,
       date_trunc('day', created_at) AS day,
       sum(amount) AS total
FROM sales
GROUP BY store_id, date_trunc('day', created_at);

-- индекс для быстрых выборок
CREATE INDEX ON sales_daily (store_id, day);

-- периодическое обновление (например, по cron)
REFRESH MATERIALIZED VIEW CONCURRENTLY sales_daily;
```

Теперь запросы:

```sql
SELECT * FROM sales_daily WHERE store_id = 42
  AND day BETWEEN '2024-01-01' AND '2024-01-31';
```

будут работать гораздо быстрее, чем каждый раз агрегировать `sales`.

#### Сравнение PostgreSQL / Oracle / SQL Server

- **PostgreSQL** — поддерживает материализованные представления с полным refresh; инкрементальные обновления требуют доп. логики или расширений.
- **Oracle / SQL Server** — имеют более продвинутые механизмы refresh (fast refresh, indexed views), но требуют аккуратной настройки.
- **MySQL** — нет нативных матвью; обычно используют обычные таблицы, которые заполняются/обновляются по расписанию или триггерами.

#### Проверь себя

1. Чем материализованное представление отличается от обычного VIEW?
2. В каких случаях матвью может быть лучше Redis‑кэша?

<details>
<summary>Ответ</summary>

1. Обычное VIEW — это **просто сохранённый SQL‑запрос**, который выполняется при каждом обращении; данные не хранятся отдельно. Материализованное представление — это **физически хранимая таблица с результатом запроса**, которую нужно периодически обновлять (REFRESH).  
2. Когда нужно кэшировать **сложные агрегаты и джойны**, при этом важна **транзакционная целостность и возможность использовать планировщик/индексы внутри БД**. В таких случаях матвью даёт кэш «внутри СУБД», управляемый теми же средствами, что и остальные таблицы.

</details>

#### Запомните

- Матвью — это **кэш, встроенный в БД**, особенно полезный для отчётности и аналитики.

### 325.4. Где кэшировать: приложение vs БД

**Цель раздела.**  
Научиться выбирать **уровень кэширования** для конкретной задачи: приложение, внешний кэш, БД, API‑уровень.

#### Теория и правила

Уровни кэширования:

1. **Внутри БД** (буферный пул, матвью, план‑кэш):
   - прозрачно для приложений;
   - ограничено возможностями конкретной СУБД.
2. **Рядом с приложением** (in‑memory кэш, Redis/Memcached):
   - гибко, управляется кодом;
   - требует отдельной инфраструктуры.
3. **На уровне API / CDN**:
   - снижает нагрузку на весь backend‑стек;
   - ограничен HTTP‑семантикой.

**Правило:** кэшировать **как можно ближе к потребителю**, но **не забывая об источнике истины**.

#### Простыми словами

Можно положить «записку с телефонами»:

- в **саму книгу** (БД) — закладки, указатели;
- на **стол библиотекаря** (приложение/Redis);
- на входе в библиотеку, на **доске объявлений** (API Gateway).

Выбор зависит от того, **кто и как часто читает** эти телефоны.

Посмотрим на тот же запрос разными глазами:

- **Глаза БД**: «Мне всё равно, кто меня спрашивает: приложение, Redis или Gateway. Я просто отвечаю на SQL».  
  Кэш на уровне БД (буферный пул, матвью) помогает самой БД делать это быстрее.
- **Глаза приложения**: «Я отвечаю бизнес‑логикой. Если смогу не ходить в БД ради каждого запроса — замечательно».  
  Redis/ Memcached — главный инструмент здесь.
- **Глаза пользователя/браузера**: «Мне просто нужен JSON/HTML как можно быстрее».  
  Кэш на уровне API/CDN важен именно с этой точки зрения.

Часто в реальных системах используются **два–три уровня кэширования сразу**:

- БД кэширует страницы в буфере;
- приложение кэширует результаты тяжёлых запросов в Redis;
- API/CDN кэширует самые популярные публичные ответы.

Важно не «накрутить кэшей ради кэшей», а понимать, **зачем каждый уровень** и **как он обновляется**.

#### Проверь себя

1. Почему не стоит полагаться только на кэш в памяти приложения (in‑process), если у тебя несколько экземпляров сервиса?
2. В каком случае кэш на уровне API Gateway может оказаться эффективнее Redis‑кэша в приложении?

<details>
<summary>Ответ</summary>

1. Потому что каждый экземпляр сервиса будет иметь **свой собственный кэш**, не разделяемый с другими; попадания в кэш будут локальными (не глобальными), а инвалидацию придётся делать во всех экземплярах. При автоскейлинге это усложняет управление. Общий Redis‑кэш или кэш на уровне БД решают эту проблему.  
2. Для **одинаковых публичных запросов**, особенно в веб‑приложениях с глобальной аудиторией: CDN/API‑кэш можно располагать ближе к пользователю, и он может перехватывать огромный процент трафика ещё до того, как запрос дойдёт до приложения и БД.

</details>

#### Запомните

- Нет «единственно правильного» уровня кэширования — есть **набор слоёв**, каждый со своими сильными и слабыми сторонами.
- Лучшие архитектуры комбинируют уровни так, чтобы: (1) БД делала как можно меньше лишней работы; (2) приложение не дёргало БД по пустякам; (3) пользователи получали быстрый ответ даже «с другого конца планеты».
