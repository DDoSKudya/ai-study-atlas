## 143. Преобразование типов

### Цель раздела

Научиться **осознанно приводить значения к нужным типам** (`CAST`, `::`, `CONVERT`, `TO_CHAR`/`TO_NUMBER`/`TO_DATE`), понимать, где возможна потеря точности, и избегать неожиданных неявных преобразований.

### В этом разделе главное

- Явные преобразования делают запросы **читабельнее и надёжнее**, чем полагаться на неявные.
- При переводе строк в числа и даты важно **контролировать формат** и быть готовым к ошибкам при некорректных данных.
- Преобразования с плавающей точкой (`float/double`) могут быть **неточными** — для денег и критичных измерений лучше `numeric/decimal`.

### Термины

- **Явное приведение** — когда ты сам пишешь `CAST(...)` или `::`.
- **Неявное приведение** — когда БД сама подбирает подходящий тип (например, при сравнении `integer` и `numeric`).
- **Форматная строка** — шаблон вида `'YYYY-MM-DD'`, по которому строка разбирается в дату/время.

### Теория и правила

- **CAST**:
  - Стандартная форма: `CAST(expr AS type)`.  
  - Пример: `CAST('123' AS INTEGER)`, `CAST(123 AS TEXT)`.

- **Сокращённая форма в PostgreSQL (`::`)**:

```sql
'123'::integer
now()::date
```

- **CONVERT (SQL Server)**:
  - `CONVERT(type, expr, style)` — стиль определяет формат (особенно для дат).

- **TO_CHAR / TO_NUMBER / TO_DATE / TO_TIMESTAMP**:
  - `TO_CHAR(value, format)` — значение → строка.  
  - `TO_NUMBER(str, format)` — строка → число.  
  - `TO_DATE(str, format)` — строка → дата.  
  - `TO_TIMESTAMP(str, format)` — строка → `timestamp`.

- **Опасные преобразования**:
  - Строка → число, если в строке «мусор» (`'123abc'`). В одних СУБД это ошибка, в других — усечение.  
  - `float` → `integer` — потеря дробной части; большие числа могут потерять точность.  
  - Неявное преобразование при сравнении разных типов (например, `WHERE int_col = '123'`).

### Разбор преобразований типов по одной

#### CAST и `::` — «сказать БД, каким типом считать»

- **`CAST(expr AS type)`** — стандарт SQL.
- **`expr::type`** — сокращённая форма (PostgreSQL и др.).

Примеры:

```sql
SELECT CAST('123' AS integer);   -- 123
SELECT '123'::integer;           -- 123 (PostgreSQL)

SELECT CAST(123 AS text);        -- '123'
SELECT 123::text;                -- '123' (PostgreSQL)
```

**Где полезно:**

- в условиях, когда параметры приходят как строки:

  ```sql
  WHERE id = CAST(:id_param AS integer)
  ```

- при явной конвертации временных значений:

  ```sql
  NOW()::date   -- только дата из timestamp
  ```

#### CONVERT (SQL Server и др.) — «специальный CAST»

- Синтаксис:

```sql
CONVERT(target_type, expression [, style])
```

- Параметр `style` особенно важен при конвертации дат/времени — он определяет формат.

Пример (SQL Server, формат `YYYY-MM-DD HH:MI:SS`):

```sql
SELECT CONVERT(varchar(19), GETDATE(), 120);
```

#### TO_CHAR / TO_NUMBER / TO_DATE / TO_TIMESTAMP

Эти функции часто встречаются в PostgreSQL и совместимых диалектах.

- **`TO_CHAR(value, format)`** — значение → строка.
- **`TO_NUMBER(str, format)`** — строка → число.
- **`TO_DATE(str, format)`** — строка → дата.
- **`TO_TIMESTAMP(str, format)`** — строка → timestamp.

Примеры:

```sql
SELECT TO_CHAR(TIMESTAMP '2024-05-10 15:20:00',
               'DD.MM.YYYY HH24:MI') AS human_time;
-- '10.05.2024 15:20'

SELECT TO_DATE('10.05.2024', 'DD.MM.YYYY');   -- 2024-05-10
```

**Важный момент:** форматная строка должна **точно соответствовать** тому, как устроена исходная строка. Ошибка в формате = сломанная дата.

#### Безопасные и опасные преобразования

- **Безопасно:**
  - `integer` → `numeric` (расширение точности).  
  - `date` → `timestamp` (добавляется время 00:00:00).  
  - `timestamp with time zone` → `timestamp without time zone` при осознанном понимании потери зоны.

- **Опасно:**
  - `double precision` → `numeric` с меньшим количеством знаков (возможна ошибка округления/усечения).  
  - `text` → `integer`, если строка может содержать нецифровые символы.  
  - автоматическое приведение типов в выражениях, когда одно значение `text`, другое — `integer`.

Пример опасного места:

```sql
WHERE int_column = '123'
```

БД **вынуждена выбрать**, что к чему приводить: либо `'123'` к `integer`, либо `int_column` к `text`. Это может повлиять и на производительность, и на точность. Лучше писать явно:

```sql
WHERE int_column = CAST('123' AS integer)
```

### Примеры

**1. Привести текстовые идентификаторы к числам:**

```sql
SELECT
  raw_id,
  CAST(raw_id AS INTEGER) AS id
FROM uploads;
```

**2. Преобразовать дату продажи в строку нужного формата:**

```sql
SELECT
  sale_id,
  TO_CHAR(sale_date, 'YYYY-MM-DD') AS sale_date_str
FROM sales;
```

**3. Разобрать строку суммы с фиксированным форматом:**

```sql
SELECT
  raw_amount,
  TO_NUMBER(raw_amount, '9999990D00') AS amount
FROM uploads;
```

**4. Избежать сюрпризов с типами в условиях:**

```sql
SELECT *
FROM orders
WHERE customer_id = CAST(:customer_id_str AS integer);
```

### Простыми словами

Преобразование типов — это **переливание воды из одного стакана в другой**: форма (тип) меняется, но ты должен быть уверен, что вода (значение) поместится и не «прольётся» в виде потери точности или ошибки.

### Картинка в голове

Представь набор **адаптеров для розеток**: разные страны — разные стандарты, но с правильным адаптером твой прибор заработает. `CAST` и `TO_...` — это адаптеры между типами в БД.

### Как запомнить

**`CAST/::` — «сказать БД, каким типом считать это сейчас», `TO_CHAR/TO_DATE/TO_NUMBER` — «перекодировать значение в конкретный строковый формат и обратно».**

### Типичные ошибки

- Опираться на неявные преобразования (`WHERE int_col = '123'`) и получать неожиданные планы/ошибки.
- Преобразовывать денежные значения к `float` и потом удивляться «странным» копейкам.
- Неправильно задавать формат в `TO_DATE`/`TO_TIMESTAMP` и тихо путать день/месяц.

### Проверь себя

1. Чем `CAST('2024-01-01' AS date)` отличается от `TO_DATE('2024-01-01', 'YYYY-MM-DD')`?  
2. Почему не стоит хранить деньги в `double precision`?  
3. Что произойдёт, если попытаться привести строку `'abc'` к `integer`?

<details>
<summary>Ответы</summary>

1. `CAST` полагается на **стандартный формат** строки для текущей СУБД, `TO_DATE` — на явно заданный формат; второй вариант надёжнее и переносимее.  
2. `double precision` — тип с плавающей точкой и двоичным представлением дробей, он не может точно представить многие десятичные числа (например, 0.1), что приводит к накоплению ошибок; для денег используют `numeric/decimal` с фиксированной точностью.  
3. В большинстве СУБД это приведёт к **ошибке преобразования** (`invalid input syntax for type integer`) — запрос упадёт.
</details>

### Запомните

Типы в БД — это **строгий контракт**. Если ты приводишь значения явно и осознанно, БД будет честно помогать тебе ловить ошибки и не даст тихо «съесть» некорректные данные.
