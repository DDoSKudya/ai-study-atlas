## Часть XXII. Интеграции и инструменты

> Цель этой части — **научиться связывать базы данных с внешним миром: поток изменений (CDC), ETL-пайплайны, драйверы и коннекторы, Kafka и очереди сообщений, GraphQL и REST API поверх БД**.

После этой части ты сможешь:

- **понимать и настраивать CDC**: Debezium, Maxwell, логирование изменений из БД в Kafka/очереди для интеграций, кэширования и аналитики;
- **выбирать и использовать ETL-инструменты**: Airflow (оркестрация DAG), dbt (трансформации в SQL), Pentaho/Talend (GUI), NiFi (потоковая обработка);
- **работать с коннекторами и драйверами**: JDBC, ODBC, специфичные драйверы (psycopg2, asyncpg), connection string, failover;
- **интегрировать БД с Kafka**: Kafka как persistent log, Kafka Connect (source/sink), ksqlDB, outbox-паттерн, exactly-once;
- **связывать БД с очередями сообщений**: RabbitMQ, Pulsar, транзакционный outbox для гарантии доставки, Saga через очереди;
- **строить GraphQL и REST API поверх БД**: резолверы и N+1, DataLoader, Prisma/Hasura; пагинация, фильтрация, сортировка, rate limiting, версионирование API.

Материал опирается на:

- **часть IV** (транзакции, ACID);
- **часть VIII** (подключение, ORM, миграции);
- **часть XI** (масштабирование, CDC, outbox, CQRS, event sourcing);
- **часть XII** (безопасность, SQL-инъекции);
- **часть XXI** (паттерны, идемпотентность, N+1).

В этой части мы смотрим на БД **как на узел в экосистеме**: как выносить изменения наружу (CDC), как оркестрировать загрузку и трансформации (ETL), как подключаться из приложений (коннекторы), как стыковать БД с очередями и потоковой обработкой (Kafka, RabbitMQ) и как экспонировать данные через API (GraphQL, REST).

### Краткая шпаргалка по части (термины)

| Термин | Что это значит простыми словами |
|--------|----------------------------------|
| **CDC (Change Data Capture)** | Механизм «подхватывания» изменений в БД (INSERT/UPDATE/DELETE) и передачи их наружу (в Kafka, очередь, другое хранилище) в реальном времени. |
| **Debezium** | Открытый CDC-коннектор для Kafka Connect: читает WAL (PostgreSQL), binlog (MySQL) и др. и публикует события в Kafka. |
| **ETL** | Extract — извлечь данные из источника; Transform — преобразовать; Load — загрузить в приёмник. Пайплайн загрузки и трансформации данных. |
| **DAG (Directed Acyclic Graph)** | Направленный ациклический граф задач: задачи связаны зависимостями, выполняются в порядке, без циклов. Основа оркестрации в Airflow. |
| **dbt** | Инструмент «трансформации в SQL»: модели (SELECT), тесты, документация; версионируется в Git; выполняется поверх уже загруженных данных (часто в DWH). |
| **JDBC** | Java Database Connectivity — стандартный Java-API для доступа к реляционным БД через драйверы (PostgreSQL, MySQL и т.д.). |
| **ODBC** | Open Database Connectivity — кроссплатформенный стандарт доступа к БД из разных языков и приложений. |
| **Connection string** | Строка подключения к БД: хост, порт, имя БД, пользователь, пароль, параметры (sslmode, timeout, application_name). |
| **Kafka Connect** | Фреймворк для потоковой интеграции: коннекторы «источник» (source) и «приёмник» (sink) связывают БД, файлы, очереди с Kafka. |
| **Outbox pattern** | Запись бизнес-данных и «сообщения для публикации» в одну транзакцию в БД; отдельный процесс (CDC/поллинг) читает outbox и публикует в очередь. Гарантия «записали — обязательно опубликуем». |
| **DataLoader** | Паттерн (и библиотеки) для батчинга и кэширования запросов к БД в рамках одного запроса (например GraphQL): N обращений к одному типу сущности → один батч-запрос. Решает N+1 в API-слое. |
| **Cursor-based pagination** | Пагинация по курсору (например, id последнего полученного элемента): следующий запрос — «где id > cursor». Стабильна при появлении новых данных в отличие от offset. |
| **Schema Registry** | Хранилище схем сообщений Kafka (например Avro); коннекторы и консьюмеры по id схемы сериализуют/десериализуют. Нужен при эволюции схем без поломки старых консьюмеров. |
| **SMT (Single Message Transforms)** | В Kafka Connect — встроенные преобразования сообщения между коннектором и топиком: переименование полей, маскирование, маршрутизация, заголовки. |
| **DLQ (Dead Letter Queue)** | Очередь для сообщений, которые не удалось обработать после нескольких попыток; не блокирует основную очередь; дальше — разбор, ретраи или переобработка. |
| **WAL** | Журнал записей PostgreSQL; все изменения пишутся подряд; CDC читает из WAL. |
| **Offset** | Позиция чтения в топике Kafka (или в логе); «с какого места консьюмер читает дальше». |
| **Резолвер** | В GraphQL — функция, которая возвращает значение поля; часто там запрос к БД. |
| **Белый список** | Заранее разрешённый набор значений (например полей для сортировки); то, чего нет в списке, запрещено — защита от инъекций. |

Используй шпаргалку как **карту**: нашёл термин в первом столбце — во втором видишь суть и зачем это нужно.

### Три предложения — суть всей части

1. **CDC и ETL** позволяют выносить изменения и данные из БД в потоковые системы (Kafka) и хранилища (DWH): Debezium/Maxwell для CDC, Airflow/dbt для оркестрации и трансформаций.
2. **Коннекторы и очереди** — как приложение подключается к БД (JDBC, нативные драйверы, connection string, failover) и как гарантировать доставку «БД → очередь» через транзакционный outbox и Kafka Connect.
3. **GraphQL и REST** поверх БД требуют аккуратной работы с N+1 (DataLoader), пагинацией (cursor предпочтительнее offset), фильтрацией без инъекций и версионирования API.

**Понятия по темам.**  
Кратко: о чём каждая группа разделов и зачем она нужна.

| Понятие | Суть |
|--------|------|
| **CDC** | Чтение изменений из лога БД (WAL, binlog) и публикация в Kafka/очередь; приложение только пишет в БД — один источник правды. |
| **ETL** | Оркестрация (Airflow: DAG, расписание) и трансформации в SQL (dbt: модели, тесты); визуальные ETL (Pentaho, Talend) и потоковая обработка (NiFi). |
| **Коннекторы и драйверы** | JDBC/ODBC и нативные драйверы (psycopg2, asyncpg); connection string, пул соединений, multi-host для failover; prepared statements — обязательно. |
| **Kafka и БД** | Kafka как persistent log; Kafka Connect (source/sink); outbox в одной транзакции с доменом, CDC публикует — гарантия доставки; ksqlDB — SQL поверх потоков. |
| **Message queues и БД** | RabbitMQ, Pulsar; та же идея outbox (домен + outbox в одной транзакции, воркер/CDC публикует); Saga через очереди, идемпотентность обработки. |
| **GraphQL поверх БД** | Резолверы ходят в БД; N+1 решается DataLoader (батч + кэш на запрос); Prisma (клиент + свои резолверы), Hasura (автогенерация API из схемы). |
| **REST поверх БД** | Ресурсы, пагинация (лучше cursor), фильтры и сортировка только через белый список и плейсхолдеры; rate limiting и версионирование API. |

Используй эту таблицу как **карту**: нашёл понятие — видишь суть и связь с другими темами.

### Одна цепочка: как связаны разделы части (от действия пользователя до результата)

Чтобы не путаться, «где что стоит», представь один сценарий от начала до конца. **Пользователь нажал «Оформить заказ».**

1. **§115 Коннекторы.** Приложение получает запрос и подключается к БД (драйвер, пул, connection string). Выполняет в одной транзакции INSERT в `orders` и INSERT в таблицу `outbox` (§116). COMMIT.
2. **§116 Kafka и outbox.** Запись в домен и в outbox сделана в одной транзакции — «намерение отправить событие» лежит в БД. Приложение больше ничего не шлёт в Kafka само.
3. **§113 CDC.** Debezium (CDC) читает WAL БД, видит новую строку в `outbox`, формирует сообщение и публикует его в топик Kafka. То есть CDC из §113 используется здесь как «доставщик» из outbox (§116).
4. Консьюмеры Kafka (сервис уведомлений, склад, аналитика) получают сообщение и обрабатывают его. Один из них может обновлять кэш или поисковый индекс — тогда при следующем чтении пользователь увидит актуальные данные.
5. **§118 GraphQL / §119 REST.** Когда пользователь открывает «Мои заказы», клиент шлёт запрос к API (GraphQL или REST). API подключается к БД (§115 снова: пул, connection string), выполняет запрос (в GraphQL — через резолверы, с DataLoader чтобы не было N+1; в REST — с пагинацией и фильтрами через белый список). Ответ отдаётся пользователю.

Итого: **§115** — как приложение говорит с БД. **§113** — как изменения из БД попадают в Kafka (логически; физически часто через ту же БД и таблицу outbox). **§116** — как гарантировать, что «записали в БД» и «событие в Kafka» не разъедутся (outbox). **§117** — то же для RabbitMQ/Pulsar. **§114** — как по расписанию гонять загрузку и трансформации (ETL), не «ручная» реакция на заказ. **§118–119** — как данные из БД отдать наружу через API. Все разделы про «БД в экосистеме»: подключение, вынос событий, доставка в очереди, отдача по API.

**Краткая таблица: главный выбор по разделам (чтобы не путать и не ошибиться)**

| Тема | Не делай так | Делай так |
|------|----------------|-----------|
| **§113 CDC** | Приложение само пишет в БД и шлёт событие в Kafka | Приложение только пишет в БД; CDC читает лог БД и публикует в Kafka |
| **§114 ETL** | Один гигантский скрипт без оркестратора и без тестов данных | Airflow — порядок и расписание; dbt — трансформации в SQL и тесты |
| **§115 Подключение** | Новое соединение на каждый запрос, пароль в коде | Пул соединений; connection string с sslmode, таймаутами; секреты из окружения |
| **§116 Kafka** | После коммита в коде вызвать producer.send() | Запись в домен и в outbox в одной транзакции; CDC читает outbox и шлёт в Kafka |
| **§117 Очереди** | После коммита в коде отправить в RabbitMQ | Та же схема: домен + outbox в одной транзакции; воркер публикует из outbox |
| **§118 GraphQL** | В резолвере вложенного поля делать запрос к БД на каждый вызов | DataLoader (батч + кэш на запрос) или один запрос с include/JOIN |
| **§119 REST** | Подставлять sort/фильтр от клиента в SQL строкой; только offset | Белый список полей и плейсхолдеры; cursor для длинных списков; rate limiting и версионирование |

Используй таблицу для быстрого напоминания: левая колонка — типичные ошибки, правая — правильный подход.

**Проверь себя (структура части)**

1. В каком порядке идут этапы изучения части (какие разделы в Этап 1, 2, 3)?
2. Зачем после «записали в БД» не вызывать сразу producer.send() в коде, а использовать outbox?
3. Какой инструмент оркестрирует порядок и расписание задач ETL, а какой делает трансформации в SQL?

<details>
<summary>Ответы</summary>
1. Этап 1: §113–114 (CDC и ETL). Этап 2: §115–117 (коннекторы и очереди). Этап 3: §118–119 (GraphQL и REST API).  
2. Между коммитом и send приложение может упасть — событие потеряно; или отправили до коммита — при откате в очереди ложное событие. Outbox: одна транзакция (домен + outbox), публикацию делает CDC/воркер — разрыва нет.  
3. Оркестрация — Airflow (DAG, расписание). Трансформации в SQL — dbt (модели, тесты).
</details>
