## Итоговая карта: симптом → причина → что проверить

| Симптом | Вероятная причина | Что проверить первым |
|--------|--------------------|----------------------|
| Подписчики не получают часть изменений из БД | Публикация из приложения после коммита, сбои между коммитом и отправкой | Ввести CDC (Debezium) или транзакционный outbox; не публиковать в очередь отдельно от транзакции |
| Дубликаты событий в консьюмере | Повторная доставка сообщения, неидемпотентная обработка | Идемпотентный ключ в сообщении; дедупликация по ключу при записи в БД; транзакционный outbox с пометкой «отправлено» |
| GraphQL-запрос с списком и вложенными полями очень медленный | N+1: один запрос за список + N за связи | Включить DataLoader для полей-связей; проверить логи запросов к БД (число round-trip) |
| REST API: страницы «прыгают», дубли/пропуски при листании | Offset-пагинация при частых вставках | Перейти на cursor-based пагинацию (after=id или составной курсор) |
| Ошибки «too many connections» при нагрузке | Нет пула или пул больше max_connections БД | Настроить пул соединений (HikariCP и т.п.); размер пула ≤ доступный лимит БД с учётом других приложений |
| При падении primary приложение не переключается на реплику | Один хост в connection string, нет multi-host | Добавить в connection string несколько хостов; проверить target_session_attrs (read-write / read-only) |
| ETL-пайплайн падает, непонятно с какого шага перезапускать | Нет оркестратора, один монолитный скрипт | Ввести оркестратор (Airflow): задачи по шагам, повтор при сбое, мониторинг |
| REST API: медленный ответ при запросе списка или клиент запрашивает «всё» | Нет ограничения max limit или очень большой offset | Задать максимальный limit (например 100); для длинных списков — cursor-based пагинация вместо большого offset |

## Одна мысль по всей части — повтори пять раз (чтобы точно запомнить)

1. **Вынос изменений из БД** — через CDC (Debezium и др.), а не через «записать в БД и вызвать producer.send()» в коде. Один источник правды — БД; CDC читает лог и публикует в Kafka/очередь.
2. **Оркестрация и трансформации** — Airflow решает «когда и в каком порядке» задачи; dbt — «как» превратить сырые данные в витрины (SQL + тесты). Визуальные ETL (Pentaho, Talend) — для тех, кто предпочитает рисовать пайплайны.
3. **Подключение к БД** — пул соединений, правильный connection string (sslmode, таймауты, при необходимости multi-host для failover), везде prepared statements.
4. **Гарантия «записали в БД ⇒ доставили в очередь»** — транзакционный outbox: запись в домен и в таблицу outbox в одной транзакции; CDC или поллинг публикует из outbox. То же для Kafka и для RabbitMQ/Pulsar. Sink/консьюмер при записи в БД — идемпотентно.
5. **API поверх БД** — GraphQL: N+1 решать DataLoader (батч + кэш). REST: пагинация (лучше cursor), фильтры и сортировка только через белый список и prepared statements, rate limiting, версионирование.

## Вопросы для самопроверки по разделам

**§113. CDC**

1. Что даёт использование CDC вместо «приложение пишет в БД и само публикует в Kafka»?  
2. Что такое snapshot mode в Debezium и когда он нужен?  
3. Почему консьюмер кэша должен обрабатывать не только op=c (create), но и op=u и op=d?

<details>
<summary>Ответы §113</summary>
1. Одна запись — только в БД; CDC читает закоммиченные изменения из WAL/binlog, порядок сохраняется, нет риска «записали в БД, не дошли до Kafka».  
2. Snapshot — начальная полная загрузка таблицы в виде потока INSERT-событий. Нужен при первом подключении коннектора или новом подписчике, чтобы построить состояние с нуля.  
3. Иначе при UPDATE и DELETE в БД кэш будет содержать устаревшие или «призрачные» (уже удалённые) записи; кэш разойдётся с БД.
</details>

**§114. ETL**

1. Какую роль играет dbt в пайплайне по сравнению с Airflow?  
2. Зачем в dbt использовать ref('имя_модели') вместо прямого имени таблицы?  
3. Когда уместна материализация incremental в dbt?

<details>
<summary>Ответы §114</summary>
1. Airflow — когда и в каком порядке запускать задачи. dbt — как трансформировать данные (SQL-модели, тесты); часто запускается из Airflow после загрузки.  
2. ref() задаёт зависимость между моделями; dbt по нему строит граф и порядок выполнения, а также подставляет корректное имя таблицы (схема, префикс).  
3. Для больших таблиц-фактов, когда полный пересчёт при каждом запуске слишком долгий; загружаем только новые/изменённые строки по ключу или фильтру.
</details>

**§115. Коннекторы и драйверы**

1. Зачем в connection string указывать несколько хостов?  
2. Почему запросы с параметрами нужно выполнять только через prepared statements?  
3. Что может произойти, если не использовать пул соединений при высокой нагрузке?

<details>
<summary>Ответы §115</summary>
1. Для failover: при недоступности первого хоста драйвер подключается к следующему (например реплика).  
2. Защита от SQL-инъекций (значение не становится частью текста SQL) и возможность кэширования плана на стороне БД.  
3. Каждый запрос открывает новое соединение; при пике — исчерпание max_connections БД, отказы или лавина открытий/закрытий, деградация.
</details>

**§116. Kafka и БД**

1. Как outbox обеспечивает гарантию доставки события из БД в Kafka?  
2. Зачем sink-коннектору (Kafka → БД) идемпотентная запись?  
3. В чём разница между source- и sink-коннектором в Kafka Connect применительно к БД?

<details>
<summary>Ответы §116</summary>
1. Запись в домен и в outbox в одной транзакции; CDC читает outbox и публикует в Kafka. Коммит = событие в outbox = будет доставлено.  
2. Сообщение может прийти консьюмеру дважды (повтор после таймаута, rebalance); без идемпотентности повторная запись создаст дубликат в БД.  
3. Source — из БД в Kafka (например Debezium читает WAL и пишет в топики). Sink — из топиков Kafka в БД (консьюмер пишет в таблицу). Направление потока противоположное.
</details>

**§117. Message queues и БД**

1. Почему публикацию в очередь нельзя делать «после коммита в БД» без outbox?  
2. Зачем в Saga обработка сообщений должна быть идемпотентной?  
3. Когда разумнее выбрать choreography, а когда orchestration в Saga?

<details>
<summary>Ответы §117</summary>
1. После коммита отправка может упасть — событие потеряно; до коммита — при откате в очередь уйдёт ложное событие. Outbox устраняет разрыв: одна транзакция (домен + outbox), публикация отдельным процессом.  
2. Очередь может доставить одно сообщение дважды; без идемпотентности повтор приведёт к двойному списанию или двойному резерву.  
3. Choreography — для простых цепочек и быстрой эволюции (добавление сервисов). Orchestration — когда сценарий сложный, компенсации многошаговые и нужна явная трассировка и мониторинг.
</details>

**§118. GraphQL и БД**

1. Как DataLoader устраняет N+1 в GraphQL?  
2. Зачем создавать новый экземпляр DataLoader на каждый запрос, а не один глобальный?  
3. В чём ограничение Hasura по сравнению с «своим» GraphQL-сервером на Prisma?

<details>
<summary>Ответы §118</summary>
1. Резолверы отдают ключи (id) в DataLoader; DataLoader батчит их и делает один запрос WHERE id IN (...), раздаёт результат по ключам и кэширует на запрос.  
2. Кэш DataLoader привязан к экземпляру; глобальный кэш жил бы между запросами и пользователями — риск утечки данных и устаревших данных.  
3. Сложная бизнес-логика не в БД: в Hasura её выносят в Actions или в триггеры/RLS; свой сервер на Prisma даёт полный контроль в резолверах (любой код, внешние сервисы).
</details>

**§119. REST API и БД**

1. Чем cursor-based пагинация лучше offset-based при частых вставках?  
2. Как защититься от SQL-инъекции при динамической сортировке (sort=...)?  
3. Зачем ограничивать максимальный limit в API (например max 100)?

<details>
<summary>Ответы §119</summary>
1. Cursor стабилен при вставках (нет сдвига страниц); запрос по id/курсору эффективнее большого OFFSET для БД (не нужно «пробегать» offset строк).  
2. Белый список полей для sort: в ORDER BY подставлять только разрешённые варианты из маппинга в коде; значение от клиента не вставлять в SQL как строку.  
3. Иначе клиент может передать limit=1000000 — один запрос вытащит огромный объём, нагрузит БД и приложение, возможны таймаут или OOM; всегда cap: limit = min(запрос, MAX_PAGE_SIZE).
</details>

## Итоговые вопросы по всей части (если хочешь проверить связное понимание)

1. Цепочка «приложение пишет заказ в БД → другое приложение должно отправить уведомление». Опиши два подхода: без outbox и с транзакционным outbox. В чём риск первого и как второй его снимает?

<details>
<summary>Ответ</summary>
Без outbox: приложение коммитит заказ в БД, потом вызывает отправку в очередь/Kafka. Если между коммитом и отправкой процесс упал — уведомление не уйдёт. Или отправляют до коммита — при откате транзакции в очереди уже ложное событие. С outbox: в одной транзакции пишем заказ в таблицу заказов и запись в таблицу outbox (тип события, payload). Коммит один — либо оба факта в БД, либо ни одного. Отдельный процесс (CDC или поллинг) читает outbox и отправляет в очередь. Риск потери или ложного события снимается: одна атомарная запись в БД, доставку делает надёжный процесс.
</details>

2. Зачем в одном пайплайне и Airflow, и dbt? Что было бы плохо, если бы использовали только dbt или только скрипты без оркестратора?

<details>
<summary>Ответ</summary>
Airflow решает «когда» и «в каком порядке»: сначала загрузить данные из источников (extract), потом запустить dbt (transform/load). При падении какой-то задачи — retry, алерты, история. Только dbt: нет расписания и порядка «сначала загрузка, потом dbt»; dbt не тянет данные из API/файлов. Только скрипты без оркестратора: при падении непонятно, с какого шага перезапускать; нет единого мониторинга и retry. Airflow даёт DAG задач, расписание и надёжность запусков.
</details>

3. В GraphQL запросе список заказов с полем «клиент» у каждого заказа. Объясни, откуда N+1 и как DataLoader уменьшает число запросов к БД.

<details>
<summary>Ответ</summary>
Резолвер заказов один раз возвращает N заказов. Для каждого заказа вызывается резолвер поля «клиент» с customer_id. Наивная реализация: в резолвере клиента выполняется SELECT * FROM customers WHERE id = ? — N раз. Итого 1 + N запросов (N+1). DataLoader: резолвер клиента вызывает customerLoader.load(customer_id), не ходит в БД сам. DataLoader накапливает все id за «тик», потом один раз выполняет SELECT * FROM customers WHERE id IN (...), раздаёт результат по id и кэширует на запрос. Итого 1 (заказы) + 1 (батч клиентов) запросов, при повторных id — из кэша.
</details>

4. REST API отдаёт список заказов с фильтром и сортировкой. Почему нельзя собирать SQL так: ORDER BY " + req.query.sort + " " + req.query.order? Как делать правильно?

<details>
<summary>Ответ</summary>
Потому что в sort/order может быть передано произвольное значение (инъекция). Например sort=created_at; DROP TABLE orders;-- превратит запрос в выполнение двух операторов. Правильно: белый список допустимых полей (created_at, total, status) и направлений (asc, desc). Значение от клиента только выбирает один из разрешённых вариантов (например через маппинг safe_key → "created_at DESC"). В сам SQL строка от клиента не подставляется; для значений фильтров — только плейсхолдеры (prepared statement).
</details>

5. Зачем при интеграции БД с Kafka/очередью делать запись в outbox в той же транзакции, что и бизнес-данные? Объясни одной фразой и чуть подробнее.

<details>
<summary>Ответ</summary>
Одной фразой: чтобы «намерение отправить событие» и «факт сохранения данных» были в одной атомарной единице — либо оба закоммичены, либо ни одного. Подробнее: транзакция в БД атомарна. Если писать в домен и в outbox в одной транзакции, после COMMIT в БД гарантированно есть и доменные данные, и строка в outbox. Отдельный процесс (CDC или поллинг) детерминированно прочитает outbox и доставит сообщение в Kafka/очередь. Разрыва «записали, но не доставили» или «доставили, а запись откатилась» не возникает.
</details>

6. Опиши цепочку от нажатия пользователем «Оформить заказ» до того, как событие попало в Kafka и как этот же пользователь потом видит заказ в API (одним абзацем).

<details>
<summary>Ответ</summary>
Приложение по connection string подключается к БД (пул, §115), в одной транзакции пишет заказ в таблицу orders и запись в таблицу outbox (§116), делает COMMIT. Debezium (CDC, §113) читает WAL, видит строку в outbox, формирует сообщение и публикует его в топик Kafka. Консьюмеры обрабатывают событие (уведомление, склад, аналитика). Когда пользователь открывает «Мои заказы», клиент шлёт запрос к API (GraphQL или REST). API подключается к БД (§115), выполняет запрос за заказы — в GraphQL с DataLoader чтобы не было N+1 (§118), в REST с пагинацией (лучше cursor) и фильтрами по белому списку (§119). Ответ с заказами возвращается пользователю.
</details>

## Резюме части XXII

**Одним абзацем.** Часть посвящена **интеграциям и инструментам** вокруг БД: вынос изменений в поток (CDC — Debezium, Maxwell), оркестрация и трансформации (ETL — Airflow, dbt, Pentaho, Talend), подключение приложений (коннекторы JDBC/ODBC, нативные драйверы, connection string, failover), интеграция с очередями (Kafka, Kafka Connect, outbox, ksqlDB; RabbitMQ/Pulsar, транзакционный outbox, Saga), и экспозиция данных через API (GraphQL с DataLoader, Prisma, Hasura; REST с пагинацией, фильтрацией, rate limiting и версионированием). Эти темы опираются на транзакции (часть IV), подключение и ORM (часть VIII), масштабирование и CDC (часть XI), безопасность (часть XII) и паттерны разработки (часть XXI).

**Одна фраза — одно действие (по разделам):**

- **§113:** Используй CDC (Debezium) для потока изменений из БД в Kafka; не пиши в Kafka из приложения вручную для тех же данных.
- **§114:** Оркестрируй ETL через Airflow (DAG); трансформации в SQL — через dbt; при необходимости визуального дизайна — Pentaho/Talend.
- **§115:** Подключайся к БД через пул и правильный connection string; для failover укажи несколько хостов; используй prepared statements.
- **§116:** Связывай БД с Kafka через Kafka Connect; для гарантии «записали ⇒ опубликовали» — транзакционный outbox + CDC.
- **§117:** Публикацию в RabbitMQ/Pulsar делай через outbox в одной транзакции с доменными данными; в Saga обеспечивай идемпотентность.
- **§118:** В GraphQL используй DataLoader для полей-связей, чтобы не было N+1; рассмотри Prisma или Hasura под задачу.
- **§119:** В REST API делай пагинацию (лучше cursor), фильтрацию и сортировку через параметры и белый список; rate limiting и версионирование обязательны для продакшена.

## Если забыл всю часть — прочитай эти 7 строк (якоря)

1. **§113 CDC:** Приложение пишет только в БД; изменения в Kafka доставляет CDC, читая лог БД (WAL/binlog). Snapshot = начальная полная выгрузка таблицы.
2. **§114 ETL:** Airflow = когда и в каком порядке запускать задачи (DAG). dbt = как из сырых таблиц сделать витрины (SQL + ref + тесты).
3. **§115 Коннекторы:** Пул соединений + правильный connection string (sslmode, таймауты, при необходимости multi-host). Запросы с параметрами — только prepared statements.
4. **§116 Kafka и outbox:** Домен и outbox пишем в одной транзакции; CDC читает outbox и шлёт в Kafka. Sink в БД пишет идемпотентно.
5. **§117 Очереди:** Та же идея: домен + outbox в одной транзакции; воркер публикует из outbox в RabbitMQ/Pulsar. Saga = компенсации при сбое; обработка идемпотентна.
6. **§118 GraphQL:** Вложенные поля по списку → не запрос в цикле, а DataLoader (батч + кэш) или JOIN/include. DataLoader — новый на каждый запрос.
7. **§119 REST:** Пагинация лучше cursor. Сортировка и фильтры — только белый список и плейсхолдеры. Rate limiting и версионирование API.

**Повтори каждую из 7 строк вслух** — одна фраза на раздел. Если какая-то строка «не откликается» — открой тот раздел и перечитай блок **«Если забыл §NNN — вернись сюда»** в конце раздела; там та же мысль в одном предложении.

**Одна цепочка в одно предложение (для повторения вслух):**  
«Приложение пишет в БД (и в outbox); CDC читает лог БД и шлёт события в Kafka; подписчики обрабатывают; данные наружу отдаём через API (GraphQL с DataLoader, REST с пагинацией и белым списком) — всё через пул соединений и без инъекций».
