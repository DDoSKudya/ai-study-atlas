## 115. Коннекторы и драйверы

**Одной фразой:** Приложения подключаются к БД через драйверы: стандартные (JDBC для Java, ODBC кроссплатформенно) и нативные (psycopg2, asyncpg для PostgreSQL); важно правильно формировать connection string и использовать multi-host для failover.

### Цель раздела

Понять **JDBC, ODBC и ADO.NET** как стандартные интерфейсы, **нативные драйверы** (psycopg2, asyncpg, pgx), **параметры connection string** (sslmode, timeout, application_name) и **настройку failover** (несколько хостов в строке подключения).

### В этом разделе главное (три строки)

1. **JDBC** — стандарт доступа к БД из Java; один API, разные драйверы (PostgreSQL, MySQL, …); пулы соединений (HikariCP, C3P0) обязательны в проде.
2. **ODBC** — кроссплатформенный слой; приложения (Excel, BI-инструменты, разные языки) подключаются через ODBC-драйвер к любой поддерживаемой БД.
3. **Connection string** задаёт хост, порт, БД, пользователя, пароль и опции (sslmode, connect_timeout, application_name); для отказоустойчивости в строке указывают несколько хостов (multi-host), драйвер при падении primary переключается на следующий.

### Простыми словами

Драйвер — это «переводчик» между приложением и БД: приложение вызывает стандартные методы (подключиться, выполнить запрос, прочитать результат), драйвер превращает это в протокол конкретной СУБД (например PostgreSQL wire protocol). Connection string — адрес и параметры: куда подключаться, как (SSL, таймауты). Multi-host в строке — список серверов; при обрыве соединения драйвер пробует следующий хост (failover).

### Если ты боишься не понять — самая простая версия (§115)

**Что такое драйвер и зачем он?**  
Приложение пишет на «общем» языке: «подключись к БД», «выполни этот запрос», «верни результат». У каждой СУБД свой протокол по сети (как «свой язык»). **Драйвер** — переводчик: принимает общие команды от приложения и переводит их в протокол конкретной БД (PostgreSQL, MySQL и т.д.). Меняешь драйвер (и connection string) — то же приложение может говорить с другой БД.

**Что такое connection string?**  
Одна строка (или набор пар ключ–значение), в которой написано: **куда** подключаться (хост, порт, имя базы), **как представиться** (пользователь, пароль), **как вести себя** (SSL включён или нет, таймаут в секундах, имя приложения для логов). Пример: `host=localhost port=5432 dbname=mydb user=app password=secret sslmode=require`. Без правильной строки приложение не найдёт БД или подключится небезопасно.

**Зачем пул соединений?**  
Каждое новое соединение к БД — это установка TCP, аутентификация, выделение памяти на стороне БД. Если на каждый HTTP-запрос открывать новое соединение и закрывать после ответа, при тысяче запросов в минуту получится лавина открытий/закрытий и риск упереться в лимит БД (max_connections). **Пул** — это фиксированный набор уже открытых соединений; запрос берёт соединение из пула, использует, возвращает в пул. Число одновременных соединений к БД ограничено и предсказуемо.

**Зачем несколько хостов в connection string (multi-host)?**  
Чтобы при падении одного сервера (например primary) приложение могло подключиться к другому (replica или другой primary). В строке перечисляют несколько хостов; драйвер при подключении перебирает их по порядку. При обрыве соединения (если драйвер это поддерживает) можно снова перебирать список — получится простой failover без отдельного прокси.

**Одна мысль — пять раз (коннекторы и драйверы):**

1. Драйвер переводит «общий» API приложения в протокол конкретной БД. JDBC/ODBC — стандарты; psycopg2/asyncpg — «родные» для PostgreSQL, быстрее и с большими возможностями.
2. Connection string = адрес (host, port, dbname) + учётные данные (user, password) + опции (sslmode, timeout, application_name). Ошибка в строке = нет подключения или небезопасное подключение.
3. Пул соединений обязателен в проде: ограничивает число соединений и переиспользует их. Без пула при пике — «too many connections» или деградация.
4. Multi-host в строке даёт базовый failover: драйвер пробует хосты по очереди. Для записи указывают target_session_attrs=read-write (подключиться только к primary).
5. Всегда используй подготовленные выражения (prepared statements) для запросов с параметрами — защита от инъекций и возможность кэширования плана на стороне БД.

**Prepared statement в одну фразу (зачем обязательно).**  
Запрос с параметрами нужно выполнять так: «SELECT * FROM users WHERE id = ?», а значение подставлять отдельно (через setLong(1, userId) или плейсхолдер `$1`). Тогда **значение от пользователя никогда не становится частью текста SQL** — драйвер передаёт его как данные. Если склеивать строку: `"SELECT * FROM users WHERE id = " + userId`, то злоумышленник может подставить `userId = "1; DROP TABLE users--"` — в БД уйдёт два оператора (инъекция). Плюс prepared statement позволяет БД закэшировать план выполнения для одного и того же шаблона запроса. Итог: с параметрами — только плейсхолдеры, никогда не склеивать SQL и значения строкой.

**Проверь себя (пул и prepared statements)**

1. Почему на каждый HTTP-запрос не стоит открывать новое соединение к БД и закрывать после ответа?
2. В чём риск подстановки значения от пользователя в SQL строкой (без плейсхолдера)?
3. Что даёт указание в connection string нескольких хостов (multi-host)?

<details>
<summary>Ответы</summary>
1. При высокой нагрузке получится лавина открытий/закрытий и риск упереться в max_connections БД. Пул ограничивает число соединений и переиспользует их.  
2. SQL-инъекция: пользователь может подставить часть SQL (например `1; DROP TABLE users--`), и выполнится произвольный запрос.  
3. Failover: при недоступности первого хоста драйвер подключается к следующему; при обрыве соединения можно заново перебрать список хостов.
</details>

**Пошагово: что происходит при запросе с пулом (в голове):**

1. Приложение получает HTTP-запрос «дать пользователя id=5».
2. Оно просит у пула: «дай мне одно соединение». Пул либо отдаёт свободное соединение из своего набора, либо ждёт (если все заняты), либо создаёт новое, если пул ещё не заполнен до максимума.
3. Приложение выполняет по этому соединению запрос: `SELECT * FROM users WHERE id = 5` (лучше через prepared statement с плейсхолдером).
4. Приложение читает результат, отдаёт ответ клиенту и возвращает соединение в пул (не закрывает его). Соединение снова доступно для следующего запроса.

**Типичная ошибка новичка:** Открывать соединение на каждый запрос через DriverManager.getConnection(url) и закрывать после ответа. Под нагрузкой число соединений взлетает, БД отдаёт «too many connections». Всегда использовать пул (HikariCP в Java, встроенный пул в asyncpg/pgx) и ограничивать размер пула под лимит БД.

### Термины

- **JDBC (Java Database Connectivity)** — стандартный Java API (интерфейсы `Connection`, `Statement`, `ResultSet`); реализации — драйверы для каждой СУБД (JDBC URL, класс драйвера).
- **ODBC (Open Database Connectivity)** — стандарт C/C++ API и слой абстракции; используется на Windows и Unix; приложение связывается с ODBC Driver Manager, тот — с драйвером конкретной БД.
- **ADO.NET** — стандартный интерфейс доступа к БД в .NET (C#, F#, VB); провайдеры для PostgreSQL (Npgsql), SQL Server, MySQL и др.; аналог JDBC для экосистемы Microsoft. Connection string и пулы (например DbConnection) задаются через конфигурацию или код.
- **Connection string (строка подключения)** — один строковый параметр с хостом, портом, именем БД, пользователем, паролем и опциями (например `host=localhost port=5432 dbname=mydb user=app password=secret sslmode=require`).
- **Failover (переключение при отказе)** — при недоступности текущего хоста клиент подключается к другому (например реплике); в connection string задаётся несколько хостов, драйвер перебирает их при подключении и при разрыве (если поддерживается).

### Теория и правила

#### JDBC

- **URL:** `jdbc:postgresql://host:5432/dbname?user=u&password=p&sslmode=require`.
- **Загрузка драйвера:** в современных версиях Java драйвер регистрируется автоматически (ServiceLoader). Явно: `Class.forName("org.postgresql.Driver")`.
- **Получение соединения:** `DriverManager.getConnection(url)` или через DataSource (рекомендуется для пулов). Пулы: HikariCP (быстрый, рекомендуемый), C3P0, Tomcat JDBC Pool.
- **Пример (минимальный):**

```java
String url = "jdbc:postgresql://localhost:5432/mydb?user=app&password=secret&ssl=true";
try (Connection conn = DriverManager.getConnection(url);
     PreparedStatement ps = conn.prepareStatement("SELECT id, name FROM users WHERE id = ?")) {
    ps.setLong(1, userId);
    ResultSet rs = ps.executeQuery();
    while (rs.next()) { ... }
}
```

#### ODBC

- **Использование:** установить ODBC-драйвер для нужной БД (например PostgreSQL ODBC); в приложении указать DSN (Data Source Name) или полную connection string. Типичные потребители: Excel, Tableau, Power BI, скрипты на Python (pyodbc), R, C#.
- **Connection string (пример):** `Driver={PostgreSQL Unicode};Server=localhost;Port=5432;Database=mydb;Uid=app;Pwd=secret;`

#### Нативные драйверы (PostgreSQL)

- **psycopg2** (Python): синхронный, широко используется; параметры через `%s`, возврат словарей/кортежей. Поддержка COPY, серверных курсоров.
- **asyncpg** (Python): асинхронный; быстрее и экономичнее соединений при большом числе одновременных запросов. API с async/await.
- **pgx** (Go): нативный драйвер для Go; пул встроен; подготовленные запросы, batch.

#### Параметры connection string (PostgreSQL, типичные)

| Параметр | Назначение |
|----------|------------|
| host, port, dbname, user, password | Подключение |
| sslmode | disable / allow / prefer / require / verify-ca / verify-full |
| connect_timeout | Таймаут установки соединения (секунды) |
| statement_timeout | Таймаут выполнения запроса (мс или сек) |
| application_name | Имя приложения (видно в pg_stat_activity) |
| options | Доп. параметры сессии (например -c search_path=myschema) |

**Multi-host (пример для PostgreSQL):**  
`host=host1,host2,host3 port=5432 dbname=mydb user=u password=p target_session_attrs=read-write`  
При подключении драйвер пробует host1, при неудаче — host2, затем host3. `target_session_attrs=read-write` — подключаться только к primary (для реплик — `read-only`).

**Пошагово: что происходит при failover (multi-host) в голове:**  
(1) Приложение при старте или при первом запросе подключается к БД. Драйвер перебирает host1, host2, host3. host1 отвечает, проверка read-write прошла — соединение установлено с host1. (2) Запросы идут через это соединение. (3) host1 падает (сеть или сервер). Текущее соединение рвётся. (4) Следующий запрос приложения запрашивает соединение из пула; пул обнаруживает разрыв (или не может выдать старое соединение). (5) Драйвер заново перебирает хосты: host1 недоступен, host2 доступен и read-write — новое соединение к host2. (6) Запросы продолжают идти уже на host2. Приложение не меняло код — только connection string с несколькими хостами.

#### Картинка в голове (коннекторы)

**JDBC/ODBC** — как единый «разъём»: приложение говорит «мне нужно соединение с БД», а драйвер — «я знаю протокол PostgreSQL/MySQL». Меняешь драйвер и connection string — приложение подключается к другой БД без смены кода. **Connection string** — адрес и паспорт: куда стучаться (host, port), как представиться (user, password), как вести себя (SSL, таймауты). **Multi-host** — список «запасных дверей»: первая закрыта (primary упал) — пробуем следующую (реплика).

#### Пример connection string (PostgreSQL) с основными параметрами

```
postgresql://app:secret@host1:5432,host2:5432/mydb?sslmode=require&connect_timeout=5&application_name=myapp&options=-c%20statement_timeout%3D30000
```

В виде ключ-значение (используется во многих клиентах):

```
host=host1,host2
port=5432
dbname=mydb
user=app
password=secret
sslmode=require
connect_timeout=5
application_name=myapp
statement_timeout=30000
```

#### Мини-практикум (§115)

**Упражнение 1.** В connection string забыли указать `sslmode=require`. Приложение подключается к БД в облаке. Что может пойти не так?

<details>
<summary>Ответ</summary>
Соединение может установиться без шифрования (если сервер разрешает). Пароль и данные передаются по сети в открытом виде — перехват трафика даст доступ к учётным данным и данным. В проде всегда использовать sslmode=require (или verify-full для проверки сертификата).
</details>

**Упражнение 2.** Пул настроен на 100 соединений, БД имеет max_connections=200. На одной машине работают три приложения с такими же пулами. Что произойдёт?

<details>
<summary>Ответ</summary>
В пике 300 соединений (3 × 100) при полной загрузке пулов. БД примет только 200; остальные получат отказ или будут ждать. Нужно либо уменьшить размер пулов (например по 60–65 на приложение), либо увеличить max_connections на БД с учётом запаса для админских подключений.
</details>

**Упражнение 3.** Чем asyncpg лучше psycopg2 при большом числе одновременных запросов в одном процессе?

<details>
<summary>Ответ</summary>
asyncpg не блокирует поток на время ожидания ответа БД: пока один запрос ждёт, другой может выполняться. Меньше потоков/процессов при той же пропускной способности; меньше накладных расходов на переключение контекста. psycopg2 синхронный — один запрос на соединение блокирует до ответа.
</details>

#### Частые вопросы по §115

**В: Как хранить пароль от БД в connection string в проде?**  
О: Не держать пароль в коде и не в открытом виде в конфиге. Использовать переменные окружения, секретное хранилище (Vault, AWS Secrets Manager) или connection string из защищённого источника. В коде подставлять значение при инициализации клиента/пула.

**В: target_session_attrs=read-write и read-only — когда что?**  
О: read-write — когда нужна запись (основное приложение, пишущее в БД); драйвер подключается только к primary. read-only — когда только чтение (отчёты, аналитика); можно подключаться к реплике, разгружая primary.

**В: ODBC нужен только на Windows?**  
О: Нет. ODBC есть и на Linux/macOS (unixODBC). Часто используется для BI-инструментов (Tableau, Power BI), Excel, скриптов на R/Python (pyodbc), когда нужен единый интерфейс к разным БД.

#### Памятка: 3 вещи, которые нельзя забыть (§115)

1. **Пул соединений обязателен в проде.** Размер пула — с учётом max_connections БД и других приложений.
2. **Connection string: хост, порт, БД, пользователь, пароль, sslmode (require в проде), таймауты, при необходимости — multi-host для failover.**
3. **Запросы с параметрами — только через prepared statements (плейсхолдеры), не склеивать SQL строкой.**

#### Чек-лист перед продакшеном (§115)

- [ ] Используется пул соединений; размер пула не превышает доступный лимит БД (max_connections минус запас на админку и другие приложения).
- [ ] В connection string указаны sslmode=require (или строже), connect_timeout и при необходимости statement_timeout.
- [ ] Для отказоустойчивости в строке несколько хостов (multi-host); target_session_attrs задан в зависимости от роли (read-write / read-only).
- [ ] Пароль и секреты не в коде и не в открытом виде в репозитории; подставляются из окружения или секрет-хранилища.

#### Повтори вслух (§115) — одна фраза на каждое понятие

- **Драйвер:** «Переводчик между общим API приложения и протоколом конкретной БД».
- **Connection string:** «Адрес (хост, порт, БД), учётные данные, опции (SSL, таймауты); без пула и правильной строки — риски и перегрузка».
- **Пул:** «Фиксированный набор соединений; запрос берёт из пула и возвращает; без пула при пике — too many connections».
- **Multi-host:** «Несколько хостов в строке; при недоступности одного драйвер пробует следующий — простой failover».

**Что делать на практике: первые шаги для подключения к БД (§115)**

1. **Выбрать драйвер:** для Java — JDBC (например PostgreSQL JDBC) + пул (HikariCP). Для Python — psycopg2 (синхронно) или asyncpg (асинхронно). Для Go — pgx. Не открывать соединение на каждый запрос — только через пул.
2. **Собрать connection string:** host, port, dbname, user, password. В проде обязательно: `sslmode=require`, `connect_timeout=5`, при необходимости `statement_timeout=30000`. Имя приложения (`application_name`) — чтобы в `pg_stat_activity` было видно, кто подключён.
3. **Размер пула:** не больше (max_connections БД − запас на админку и другие приложения). Например БД max_connections=100, два приложения — пул по 40–45 на каждое.
4. **Failover:** если есть несколько хостов (primary + replica), указать в connection string через запятую; задать `target_session_attrs=read-write` для записи или `read-only` для чтения.
5. **Секреты:** пароль не в коде. Брать из переменной окружения или из секрет-хранилища при старте приложения.
6. **Все запросы с параметрами** — только через prepared statements (плейсхолдеры). Никогда не склеивать SQL и значения строкой.

**Проверь себя (§115)**

1. Зачем использовать пул соединений (HikariCP) вместо DriverManager.getConnection на каждый запрос?  
2. Что даёт multi-host в connection string?  
3. В чём разница между psycopg2 и asyncpg?

<details>
<summary>Ответы</summary>
1. Ограничение числа соединений к БД и переиспользование; без пула при пиковой нагрузке можно исчерпать max_connections и получить отказы или лавину открытий/закрытий.  
2. Несколько хостов (например primary + standby); при недоступности текущего драйвер переключается на следующий — базовый failover без отдельного прокси.  
3. psycopg2 — синхронный; asyncpg — асинхронный (async/await), меньше блокировок и меньше соединений при высокой конкуренции.
</details>

### Запомните

JDBC — стандарт для Java; ODBC — кроссплатформенный слой. Нативные драйверы (psycopg2, asyncpg, pgx) дают лучшую производительность и контроль. Connection string задаёт хост, порт, БД, пользователя, опции (sslmode, timeout, application_name); multi-host обеспечивает переключение при отказе.

**Резюме §115 в одну строку:** Приложение подключается к БД через драйвер (JDBC/ODBC или нативный); обязательно пул соединений и корректный connection string (sslmode, таймауты, при необходимости multi-host для failover); запросы с параметрами — только prepared statements.

**Не путай: пул и «одно соединение на запрос».** Без пула каждый HTTP-запрос открывает новое соединение и закрывает после ответа — при 1000 RPS это тысячи открытий/закрытий в секунду и риск упереться в max_connections. Пул держит ограниченное число соединений (например 20) и отдаёт их запросам по очереди; запрос вернул соединение в пул — им может воспользоваться следующий. Один раз настроил размер пула — число соединений к БД предсказуемо.

#### Что может пойти не так (коннекторы и подключение) — и что делать

| Что случилось | Почему плохо | Что проверить / что делать |
|---------------|--------------|----------------------------|
| Нет пула, на каждый запрос новое соединение | При пике — «too many connections» от БД; лавина открытий/закрытий, деградация. | Использовать пул (HikariCP, встроенный в asyncpg/pgx); размер пула = (max_connections − запас) / число инстансов приложения. |
| В проде sslmode=disable или не указан | Трафик к БД не шифруется; пароль и данные можно перехватить. | В connection string всегда sslmode=require (или verify-full при проверке сертификата). |
| Соединение взяли из пула и не вернули (забыли close или исключение) | Утечка соединений; пул опустеет, новые запросы будут ждать или падать. | Всегда использовать try/finally или «using» и возвращать соединение в пул; при исключении тоже возвращать. |
| Сумма размеров пулов всех приложений > max_connections БД | Часть подключений отклоняется; ошибки при пиковой нагрузке. | Посчитать: N приложений × размер пула на инстанс × число инстансов. Должно быть меньше max_connections с запасом на админку и мониторинг. |

**Если забыл §115 — вернись сюда.** Драйвер = переводчик приложение ↔ БД. Connection string = куда и как подключаться. Пул обязателен. Multi-host = список хостов для failover. Параметры в запрос — только через prepared statements.
