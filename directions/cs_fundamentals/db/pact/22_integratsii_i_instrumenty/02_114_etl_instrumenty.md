## 114. ETL инструменты

**Одной фразой:** ETL — извлечение данных из источников, трансформация и загрузка в приёмники; оркестрируют пайплайны Airflow (DAG), трансформации в SQL делают dbt, визуальные ETL — Pentaho/Talend, потоковые — NiFi.

### Цель раздела

Понять **что такое ETL**, как устроены **Apache Airflow** (DAG, задачи, сенсоры), **dbt** (модели, тесты, документация), **Pentaho и Talend** (визуальное проектирование), **Apache NiFi** (потоковая обработка), и **когда что выбирать**.

### В этом разделе главное (три строки)

1. **Airflow** — оркестратор задач: DAG (направленный ациклический граф) задаёт порядок выполнения; задачи — Python-операторы (Bash, Python, запрос к БД, вызов API); расписание (cron); мониторинг и повтор при сбоях.
2. **dbt** — не оркестратор, а слой трансформаций: модели описываются SQL (SELECT); зависимости между моделями задаются через ref(); тесты (уникальность, not null, произвольные SQL); выполняется по расписанию или из Airflow.
3. **Pentaho, Talend** — enterprise ETL с GUI: визуальные пайплайны (источник → преобразования → приёмник); тяжёлые решения для больших команд и сложных интеграций; код часто генерируется из визуальной модели.

### Простыми словами

ETL — это «откуда взять данные, как их преобразовать и куда положить». Оркестратор (Airflow) решает **когда** и **в каком порядке** запускать шаги (загрузить из API, загрузить в DWH, запустить dbt). dbt решает **как** превратить сырые таблицы в агрегаты и витрины (чистый SQL + тесты). Визуальные инструменты (Pentaho, Talend) позволяют рисовать пайплайн и генерировать код; удобно тем, кто не пишет код, но сложнее версионировать и ревьюить.

### Если ты боишься не понять — самая простая версия (§114)

**Что такое ETL буквально?**  
Три буквы: **E**xtract (извлечь), **T**ransform (преобразовать), **L**oad (загрузить). То есть: взять данные откуда-то (база, API, файлы), как-то их обработать (почистить, посчитать суммы, соединить таблицы), и положить туда, где они нужны (другая база, хранилище данных, озеро).

**Зачем два слоя: Airflow и dbt?**  
Представь, что пайплайн — это рецепт ужина. **Airflow** — таймер и порядок: «в 18:00 достань продукты из холодильника (extract), в 18:05 запусти разделку и готовку (transform), в 19:00 разложи по тарелкам (load)». Airflow не знает, как именно резать лук; он только говорит «сейчас шаг „готовка“, запусти его». **dbt** — сам рецепт готовки: конкретные SQL-запросы — «возьми таблицу сырых заказов, сгруппируй по дням и клиентам, посчитай суммы». dbt не решает, в какой день и час запускаться — это делает Airflow. Вместе: Airflow в 2:00 ночи запускает «загрузить данные из продакшен-БД в DWH», потом запускает «dbt run» — и витрины пересчитываются.

**DAG — что это простыми словами?**  
DAG = направленный ациклический граф. По-русски: **граф** — это точки (задачи) и стрелки («сначала A, потом B»). **Направленный** — стрелки в одну сторону (A → B, не B → A одновременно). **Ациклический** — нет циклов (нельзя, чтобы A ждал B, B ждал C, C ждал A). В Airflow ты описываешь: задача «скачать файл» → задача «загрузить в БД» → задача «запустить dbt». Планировщик смотрит на стрелки и запускает задачи в порядке: сначала те, у кого нет входящих стрелок, потом следующие по стрелкам.

**Почему в DAG не бывает циклов?**  
Если бы был цикл (A → B → C → A), планировщик не смог бы решить, что запускать первым: A ждёт C, C ждёт B, B ждёт A — все ждут друг друга. Топологический порядок выполнения определён только для ациклического графа. Поэтому в Airflow зависимости задают так, чтобы циклов не было.

**Проверь себя (DAG и оркестрация)**

1. Почему граф задач в Airflow должен быть именно ациклическим?
2. В DAG три задачи: A → B, A → C, B → D, C → D. В каком порядке могут выполняться A, B, C, D?
3. Что даёт разбиение пайплайна на отдельные задачи в оркестраторе вместо одного скрипта?

<details>
<summary>Ответы</summary>
1. При цикле невозможно определить, какую задачу запускать первой — все ждут друг друга; топологический порядок не определён.  
2. Сначала A (нет входящих); затем параллельно B и C (обе зависят только от A); когда обе завершатся — D.  
3. Перезапуск только упавшей задачи и следующих; история запусков; retry и алерты; явные зависимости и мониторинг.
</details>

**Одна мысль — пять раз (ETL и оркестрация):**

1. ETL = извлечь → преобразовать → загрузить. Оркестратор (Airflow) решает **когда** и **в каком порядке**; трансформации (dbt) — **как** именно преобразовать данные.
2. Airflow не выполняет бизнес-логику данных — он запускает задачи (скрипты, SQL, dbt). Задача упала — он может перезапустить; расписание — cron или interval.
3. dbt — это SQL-модели (каждый файл = один SELECT → таблица/представление) и тесты. Зависимости между моделями — через ref('имя_модели'). Запуск — `dbt run` и `dbt test`; в проде часто вызывают из Airflow.
4. Визуальные ETL (Pentaho, Talend) — рисуешь блоки «источник → преобразование → приёмник», код генерируется сам. Плюс: понятно нетехническим специалистам. Минус: сложнее хранить в Git и ревьюить построчно.
5. Выбор: нужна оркестрация по расписанию и повтор при сбоях → Airflow. Нужны трансформации в SQL с тестами и версионированием → dbt. Нужен визуальный дизайн без кода → Pentaho/Talend.

**Пошагово: типичный ночной пайплайн (в голове):**

1. 02:00 — Airflow по расписанию запускает DAG «nightly_load».
2. Задача «extract_orders»: скрипт или оператор копирует данные из операционной БД в staging-таблицы DWH (или в S3/озеро).
3. Задача «extract_customers»: то же для справочника клиентов.
4. Обе задачи завершились → следующая задача «dbt_run»: выполняется `dbt run`. dbt по графу ref() сначала строит модели, от которых зависят другие (сырые → очищенные), потом агрегаты и витрины.
5. Задача «dbt_test»: `dbt test` — проверки (уникальность, not null, ссылочная целостность). Если тест упал — DAG помечается failed, приходит алерт.
6. Утром аналитики видят обновлённые витрины в DWH.

**Пошагово: как dbt определяет порядок выполнения моделей.**  
В каждой модели в SQL встречаются вызовы `ref('имя_модели')` — это зависимость «эта модель нуждается в результате той модели». dbt собирает все модели и строит граф: узел = модель, ребро A → B означает «B использует ref(A)». Порядок выполнения — топологическая сортировка этого графа: сначала выполняются модели, от которых никто не зависит (например `stg_orders`, `stg_customers`), потом модели, которые зависят только от них (например `fct_orders`), потом следующие. Так при `dbt run` сначала создаются таблицы-источники, потом те, что от них зависят, без ручного указания порядка — порядок выводится из ref().

**Конкретный пример: как ref() превращается в имя таблицы.**  
В модели `fct_daily_sales.sql` написано: `FROM {{ ref('stg_orders') }}`. При выполнении dbt подставляет вместо `ref('stg_orders')` **реальное имя таблицы** модели `stg_orders` в DWH (например `dwh.stg_orders` или с суффиксом схемы). Ты не пишешь имя таблицы вручную — dbt знает его по имени модели. Плюс по этим ref() строится граф: `fct_daily_sales` зависит от `stg_orders`, поэтому при `dbt run` сначала выполнится `stg_orders`, потом `fct_daily_sales`.

**Типичная ошибка новичка:** Писать один гигантский скрипт «скачать всё → преобразовать всё → загрузить всё». При падении на шаге 15 непонятно, с какого места перезапускать; нет истории «когда какой шаг выполнялся». Оркестратор разбивает на задачи с явными зависимостями и хранит историю запусков; при падении можно перезапустить только упавшую задачу и следующие.

### Термины

- **ETL (Extract, Transform, Load)** — извлечение данных из источника(ов), преобразование (очистка, агрегация, соединение), загрузка в целевую систему (БД, DWH, озеро).
- **DAG (Directed Acyclic Graph)** — граф задач без циклов; ребро A → B означает «B после A»; оркестратор выполняет задачи в топологическом порядке.
- **Airflow DAG** — Python-файл, в котором объявляются задачи (операторы) и зависимости (task1 >> task2); планировщик по расписанию запускает DAG и следит за выполнением.
- **dbt model** — один SQL-файл (SELECT), результат которого материализуется как таблица или представление; зависимости — через ref('other_model').
- **dbt test** — проверка данных: уникальность ключа, not null, допустимые значения или произвольный SQL (например, ссылочная целостность).

### Теория и правила

#### Apache Airflow

- **Задача:** оркестрация — когда запускать какую задачу, повтор при падении, алерты.
- **Единица работы:** DAG. Внутри DAG — задачи (Task); зависимости задаются операторами `>>` и `<<` (например `extract >> transform >> load`).
- **Операторы:** BashOperator, PythonOperator, PostgresOperator, MySqlOperator, HttpOperator и множество коннекторов (Kafka, S3, Snowflake и т.д.). **Сенсоры (Sensor)** ждут выполнения условия перед запуском следующей задачи: например FileSensor — ждёт появления файла по пути; SqlSensor — ждёт появления строки или выполнения условия в БД; внешний HTTP — ждёт ответа. Так можно строить DAG вида «запусти загрузку только когда в S3 появился файл-маркер» или «запусти dbt только когда все партиции за вчера загружены».
- **Расписание:** cron-выражение или interval; каждый запуск — отдельный DAG Run.
- **Пример фрагмента DAG (псевдокод):**

```python
from airflow import DAG
from airflow.operators.python import PythonOperator
from airflow.providers.postgres.operators.postgres import PostgresOperator
from datetime import datetime

with DAG("load_and_transform", start_date=datetime(2025, 1, 1), schedule="0 2 * * *") as dag:
    extract = PostgresOperator(task_id="extract", sql="INSERT INTO staging.events SELECT * FROM source.events WHERE ...")
    run_dbt = BashOperator(task_id="run_dbt", bash_command="cd /dbt && dbt run")
    extract >> run_dbt
```

#### dbt (data build tool)

- **Предпосылка:** данные уже загружены в хранилище (DWH). dbt не тянет из API/файлов — он трансформирует то, что уже в БД.
- **Модели:** папка `models/`, файлы `*.sql`. Каждый файл — один SELECT. Имя файла → имя таблицы/представления (настраивается). В SQL используется `ref('other_model')` — dbt подставит имя таблицы и построит граф зависимостей.
- **Материализация:** table, view, incremental. **Incremental** — только новые/изменённые строки (по ключу или фильтру): подходит для больших таблиц-фактов, когда полный пересчёт при каждом запуске слишком долгий; задаётся стратегия слияния (append, merge по ключу) и условие инкремента (например `where created_at > (select max(created_at) from this_model)`).
- **Тесты:** в `schema.yml` описываются тесты на столбцы (unique, not_null, accepted_values, relationships) или custom SQL. `dbt test` запускает проверки.
- **Запуск:** `dbt run` (выполнить модели), `dbt test` (запустить тесты). В проде часто вызывают из Airflow после загрузки данных.

#### Pentaho (PDI), Talend

- **Идея:** визуальный дизайн пайплайна: перетаскивание блоков «источник» (БД, файл, API), «преобразование» (фильтр, join, агрегация), «приёмник». Код генерируется автоматически.
- **Плюсы:** понятно аналитикам и интеграторам без глубокого программирования; много готовых коннекторов. **Минусы:** сложнее версионировать в Git (большие XML/проекты), код ревьюить неудобно; тяжёлая установка и лицензии (Talend коммерческий вариант).

#### Apache NiFi

- **Идея:** потоковая обработка данных с визуальным интерфейсом: процессоры (GetFile, InvokeHTTP, PutDatabaseRecord и т.д.) связаны очередями; данные идут потоком (FlowFile). Подходит для потоковой загрузки и маршрутизации, меньше для тяжёлых batch-трансформаций в DWH.

#### Картинка в голове (ETL)

**Airflow** — диспетчер: «в 2:00 запусти задачу A, когда A закончит — запусти B и C параллельно, потом D». Он не знает, что именно делают A и B (скачать файл, выполнить SQL, вызвать API); он только следит за порядком и за тем, чтобы упавшая задача была перезапущена. **dbt** — «повар трансформаций»: у тебя уже есть сырые таблицы в хранилище; dbt — это рецепты (SQL), которые из них готовят витрины и агрегаты; плюс проверки (тесты), что данные не сломались. **Визуальные ETL** — как конструктор: перетащил «источник БД», «фильтр», «приёмник»; программа сама генерирует код; удобно тем, кто мыслит потоками данных, а не строками кода.

#### Что будет, если не разделять оркестрацию и трансформации

Если всё писать одним скриптом «скачать → преобразовать → загрузить» без оркестратора: при падении на шаге 3 непонятно, с какого места перезапускать; нет единого расписания и мониторинга для десятков пайплайнов. Оркестратор (Airflow) даёт единую точку расписания, повтор при сбое, алерты и историю запусков. Если трансформации размазаны по скриптам без моделей и тестов (dbt): при изменении схемы источника трудно понять, какие витрины затронуты; регрессии в данных обнаруживаются поздно. dbt даёт граф зависимостей и тесты на каждом шаге.

#### Выбор инструмента

| Критерий | Airflow | dbt | Pentaho/Talend | NiFi |
|----------|---------|-----|----------------|------|
| Оркестрация по расписанию | Да | Нет (запуск извне) | Да | Да (поток) |
| Трансформации в SQL | Через операторы | Основная задача | Через блоки | Ограниченно |
| Визуальный дизайн | Нет (код DAG) | Нет | Да | Да |
| Версионирование (Git) | Удобно | Удобно | Сложнее | Сложнее |
| Команда «код vs визуал» | Разработчики | Аналитики/инженеры | Смешанная | Интеграторы |

**Когда что выбирать — в одну фразу.**  
Нужен **порядок выполнения и расписание** (когда запускать загрузку, когда dbt) → **Airflow**. Нужны **трансформации в SQL и тесты данных** (витрины, граф моделей) → **dbt** (часто вместе с Airflow). Нужен **визуальный дизайн пайплайна без кода** → **Pentaho или Talend**. Нужна **потоковая загрузка по событиям** (файлы, HTTP, поток записей) → **NiFi**. Для типичного ночного DWH: Airflow оркестрирует, dbt преобразует.

**Проверь себя (выбор ETL-инструмента)**

1. Кто тянет сырые данные из API или файлов в DWH — Airflow или dbt?
2. Когда разумно выбрать визуальный ETL (Pentaho/Talend) вместо Airflow + dbt?
3. Зачем в пайплайне запускать dbt test после dbt run?

<details>
<summary>Ответы</summary>
1. Airflow (или отдельные задачи/скрипты). dbt не тянет из API/файлов — он работает только с данными, уже загруженными в хранилище.  
2. Когда команда предпочитает рисовать пайплайны, мало пишет код, нужны готовые коннекторы и не критично версионировать каждый шаг в Git построчно.  
3. Чтобы проверить качество данных после трансформаций (уникальность, not null, ссылочная целостность); при падении теста пайплайн помечается failed, приходит алерт — ошибки в данных не уходят «тихо».
</details>

#### Мини-практикум (§114)

**Упражнение 1.** В dbt есть модель `stg_orders` (сырые заказы) и модель `fct_daily_sales`, в которой есть `ref('stg_orders')`. В каком порядке dbt выполнит эти модели при `dbt run`?

<details>
<summary>Ответ</summary>
Сначала `stg_orders` (от неё зависит `fct_daily_sales`), потом `fct_daily_sales`. dbt строит граф по ref() и выполняет модели в топологическом порядке.
</details>

**Упражнение 2.** Задача «load_csv» в Airflow упала с ошибкой через 10 минут. Что даёт оркестратор, чего не даёт «просто скрипт»?

<details>
<summary>Ответ</summary>
Историю запусков (когда упало), возможность перезапустить только эту задачу (и следующие по зависимостям), алерты, логи в одном месте. Со скриптом пришлось бы смотреть логи и вручную решать, с какой строки перезапускать.
</details>

**Упражнение 3.** Когда разумно выбрать визуальный ETL (Talend/Pentaho), а когда Airflow + dbt?

<details>
<summary>Ответ</summary>
Визуальный ETL — когда команда предпочитает рисовать пайплайны, мало пишет код, нужны готовые коннекторы «из коробки» и не критично версионировать каждый шаг в Git. Airflow + dbt — когда команда разработки/аналитики готова писать код (Python DAG, SQL), нужны тесты данных и явный граф зависимостей моделей, и важен код-ревью и Git.
</details>

#### Частые вопросы по §114

**В: dbt сам тянет данные из API или файлов?**  
О: Нет. dbt предполагает, что данные уже в хранилище (DWH, озеро). Extract и первичная загрузка — задача других задач в Airflow (или другого инструмента). dbt только Transform (и по сути Load в таблицы/представления внутри той же БД).

**В: Что если задача в Airflow зависла (не падает, но и не заканчивается)?**  
О: Настраивают таймаут на задачу (execution_timeout). По истечении таймаута задача помечается failed, можно перезапустить. Плюс смотреть логи и ресурсы (память, блокировки БД).

**В: Можно ли использовать только dbt без Airflow?**  
О: Да. Запуск по cron: `0 2 * * * cd /project && dbt run`. Для одного-двух пайплайнов этого достаточно. Airflow нужен, когда много разных источников, сложный порядок (сначала загрузка из трёх систем, потом dbt), retry, алерты и единый мониторинг.

**В: Что значит catchup=False в DAG?**  
О: По умолчанию (catchup=True) при первом включении DAG Airflow создаёт запуски за все пропущенные интервалы с start_date — «догоняет» прошлое. catchup=False — запускать только следующий по расписанию запуск; прошлые даты не догонять. Для ночного пайплайна обычно ставят catchup=False, чтобы не поднимать десятки исторических прогонов при деплое.

#### Памятка: 3 вещи, которые нельзя забыть (§114)

1. **Airflow = когда и в каком порядке.** dbt = как преобразовать данные (SQL + тесты). Часто вместе: Airflow запускает dbt после загрузки.
2. **DAG = задачи + стрелки зависимостей, без циклов.** Задачи выполняются в порядке «сначала без входящих стрелок, потом следующие».
3. **ref() в dbt связывает модели.** dbt по ref() строит граф и порядок выполнения; имя модели в ref() — это имя таблицы/представления другой модели.

#### Чек-лист перед использованием ETL в проде (§114)

- [ ] Задачи в DAG имеют таймауты и retry; при падении приходит алерт (email, Slack, PagerDuty).
- [ ] dbt-модели покрыты тестами (хотя бы unique, not_null на ключевых полях); `dbt test` запускается после `dbt run` в пайплайне.
- [ ] Секреты (пароли БД, API-ключи) не в коде DAG, а в Variables/Connections Airflow или в секретном хранилище.
- [ ] При использовании визуального ETL проект и конфигурация хранятся в Git или в системе версионирования; есть возможность отката.

#### Повтори вслух (§114) — одна фраза на каждое понятие

- **ETL:** «Извлечь данные → преобразовать → загрузить в приёмник».
- **Airflow:** «Оркестратор: когда и в каком порядке запускать задачи (DAG); не знает, как именно делаются трансформации».
- **dbt:** «Трансформации в SQL (модели + ref() + тесты); запускается по расписанию или из Airflow после загрузки».
- **DAG:** «Граф задач без циклов; стрелки задают порядок; оркестратор выполняет в топологическом порядке».

**Проверь себя (§114)**

1. Чем Airflow отличается от dbt по роли в пайплайне?  
2. Что такое dbt model и ref()?  
3. Когда разумно выбрать визуальный ETL (Pentaho/Talend) вместо Airflow + dbt?

<details>
<summary>Ответы</summary>
1. Airflow — оркестратор: когда и в каком порядке запускать задачи. dbt — слой трансформаций: как из сырых таблиц получить витрины (SQL + тесты). Часто Airflow запускает dbt после загрузки данных.  
2. Model — один SQL-файл (SELECT), результат — таблица/представление. ref('other_model') — ссылка на другую модель; dbt строит граф и порядок выполнения.  
3. Когда команда предпочитает визуальное проектирование и не хочет поддерживать код DAG/SQL в Git; при жёстких требованиях enterprise к поддержке и лицензиям.
</details>

### Запомните

ETL = Extract, Transform, Load. Airflow оркестрирует DAG задач; dbt делает трансформации в SQL с тестами и ref(); Pentaho/Talend — визуальные ETL; NiFi — потоковая обработка. Выбор зависит от команды, версионирования и того, нужна ли оркестрация или только трансформации.

#### Пример целиком: маленький DAG в Airflow и одна модель dbt

**DAG (файл `dags/nightly_dwh.py`):**

```python
from airflow import DAG
from airflow.operators.bash import BashOperator
from airflow.providers.postgres.operators.postgres import PostgresOperator
from datetime import datetime

with DAG(
    dag_id="nightly_dwh",
    start_date=datetime(2025, 1, 1),
    schedule="0 2 * * *",  # каждый день в 02:00
    catchup=False,  # не запускать «догоняющие» прогоны за прошлые даты с start_date
    tags=["dwh"],
) as dag:
    extract = PostgresOperator(
        task_id="extract_events",
        sql="INSERT INTO dwh.staging_events SELECT * FROM app.events WHERE created_at >= CURRENT_DATE - 1",
    )
    run_dbt = BashOperator(
        task_id="run_dbt",
        bash_command="cd /opt/dbt && dbt run --target prod",
    )
    test_dbt = BashOperator(
        task_id="test_dbt",
        bash_command="cd /opt/dbt && dbt test --target prod",
    )
    extract >> run_dbt >> test_dbt
```

**Одна модель dbt (файл `models/fct_daily_orders.sql`):**

```sql
-- Зависит от stg_orders (dbt подставит имя таблицы через ref)
SELECT
    date_trunc('day', order_date) AS day,
    customer_id,
    count(*) AS order_count,
    sum(total) AS total_amount
FROM {{ ref('stg_orders') }}
GROUP BY 1, 2
```

**Тест в `schema.yml`:** уникальность по (day, customer_id), not null по полям. Так ты видишь и порядок выполнения (extract → dbt run → dbt test), и как ref() связывает модели.

**Резюме §114 в одну строку:** Airflow решает «когда и в каком порядке» задачи (DAG); dbt решает «как» превратить сырые данные в витрины (SQL + ref() + тесты); вместе они дают ночной пайплайн загрузки и трансформаций.

#### Что может пойти не так (ETL) — и что делать

| Что случилось | Почему плохо | Что проверить / что делать |
|---------------|--------------|----------------------------|
| Задача «extract» упала после 30 минут, retry исчерпан | Следующие задачи (dbt run) не запустятся; витрины не обновятся; данные за вчера неполные. | Смотреть логи задачи; исправить причину (сеть, лимит БД, таймаут). Перезапустить только упавшую задачу (Airflow «Clear» или «Retry»). Настроить алерт при failed. |
| dbt test падает на unique | В витрине появились дубли по ключу — ошибка в логике модели или в сырых данных. | Разобрать модель (GROUP BY, ключи); проверить источник данных. Не отключать тест — он как раз ловит такие ошибки. |
| Пароль от БД в коде DAG или в репозитории | Утечка секретов; при смене пароля нужно менять код. | Хранить в Airflow Connections или в переменных окружения; в коде только ссылка на connection_id. |
| Задача «висит» (не падает, но и не завершается) | Долго держит ресурсы; следующие задачи не стартуют. | Задать execution_timeout на задачу; смотреть логи и нагрузку на БД (блокировки, долгие запросы). |

**Не путай: Airflow и dbt.** Airflow **не** подставляет SQL и **не** знает зависимости между таблицами — он только запускает задачи (например «выполни dbt run») по расписанию и по графу зависимостей *задач*. dbt **не** решает, в какой день и час запускаться и что делать до трансформаций (например загрузить данные из API) — он только по `ref()` строит граф *моделей* и выполняет SQL. В пайплайне сначала идут задачи загрузки (в Airflow), потом одна задача «dbt run» — внутри dbt свой порядок выполнения моделей.

**Если забыл §114 — вернись сюда.** Оркестратор (Airflow) = когда и в каком порядке. Трансформации (dbt) = как из сырых таблиц сделать витрины (SQL + ref + тесты). DAG = граф задач без циклов.

#### Что делать на практике: первые шаги для ETL (§114)

1. **Определить источники и приёмник:** откуда данные (операционная БД, API, файлы), куда (DWH, озеро, витрины). Какое расписание (ночью раз в день, каждый час).
2. **Airflow:** установить (или использовать managed, например Astronomer). Создать первый DAG: Python-файл с задачами и зависимостями (`task_a >> task_b`). Задачи — BashOperator или PostgresOperator для простых шагов. Задать schedule (cron) и start_date. Запустить вручную, проверить логи.
3. **Загрузка в staging:** задача «extract» — копирование из источника в промежуточные таблицы DWH (или в S3). Использовать операторы БД или скрипт; секреты — в Connections Airflow, не в коде.
4. **dbt:** в каталоге проекта выполнить `dbt init`. Описать модели (SQL-файлы с SELECT); зависимости через `ref('stg_orders')`. Запустить `dbt run` локально, проверить граф: `dbt docs generate` и посмотреть граф зависимостей. Добавить тесты в schema.yml (unique, not_null).
5. **Связать Airflow и dbt:** в DAG после задачи загрузки добавить задачу `BashOperator` с командой `dbt run` (и при необходимости `dbt test`). Убедиться, что путь к проекту dbt и переменные окружения (подключение к DWH) заданы.
6. **Граф зависимостей моделей:** выполнить `dbt docs generate` и открыть сгенерированную документацию — там визуальный граф «какая модель от какой зависит»; полезно при онбординге и при изменении источников.
7. **Мониторинг:** при падении задачи — retry и алерт (email/Slack). Не хранить пароли в коде; использовать таймауты на задачи.
