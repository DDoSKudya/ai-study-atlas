## 118. GraphQL и БД

**Одной фразой:** В GraphQL резолверы получают данные для полей; часто каждый резолвер ходит в БД — возникает N+1. Решение — батчинг и кэш в рамках одного запроса (паттерн DataLoader). Prisma и Hasura дают типизированный доступ и автогенерацию GraphQL API поверх схемы БД.

### Цель раздела

Понять **как резолверы связаны с БД**, **проблему N+1** в GraphQL (один запрос за корень, N за связанные сущности), **DataLoader** (батчинг и кэш per-request), и возможности **Prisma и Hasura** (схема из БД, подписки).

### В этом разделе главное (три строки)

1. **Резолвер** — функция, возвращающая значение поля; для полей типа «пользователь», «список заказов» резолвер часто делает запрос к БД. Если у каждого элемента списка есть вложенное поле (например автор), и резолвер вызывается по одному разу на элемент — получается N+1 запросов.
2. **DataLoader** — собирает ключи (например id пользователей) в рамках одного «тика» (frame) выполнения и делает один батч-запрос к БД (WHERE id IN (...)); результат раздаёт по ключам. Плюс кэш в рамках одного запроса — один и тот же id не запрашивается дважды.
3. **Prisma** — ORM с генерацией клиента и типов; можно вызывать из резолверов; батчинг через Prisma DataLoader или вручную. **Hasura** — автоматическая GraphQL API поверх PostgreSQL (и др.): схема из метаданных БД, подписки (real-time), права на уровне строк (RLS).

### Простыми словами

GraphQL разрешает клиенту запросить «заказы с именами клиентами». Сервер для каждого заказа может вызвать резолвер «клиент» — и сделать 1 запрос за заказы + N за клиентов (N+1). DataLoader — «собиратель»: резолверы отдают только id клиента; DataLoader накапливает id и один раз спрашивает БД «все клиенты с id из списка», потом раздаёт результат по id. Prisma и Hasura упрощают: Prisma — запросы к БД из кода с типами; Hasura — вообще не пишешь резолверы, настраиваешь схему и права, API генерируется.

### Если ты боишься не понять — самая простая версия (§118)

**Почему в GraphQL легко получить N+1?**  
В GraphQL клиент запрашивает поля, которые хочет. Например: «дай 50 заказов, у каждого заказа покажи поле customer (имя, email)». Сервер выполняет запрос по частям: сначала резолвер корня возвращает 50 заказов (один SQL: SELECT * FROM orders LIMIT 50). Потом для **каждого** из 50 заказов GraphQL вызывает резолвер поля `customer`, чтобы подставить имя и email. Наивная реализация: резолвер `customer` получает order.customer_id и делает SELECT * FROM customers WHERE id = ?. Так вызывается 50 раз — по разу на заказ. Итого **1 + 50 запросов** к БД. Это и есть N+1: один запрос за список + N запросов за связанные данные.

**Как DataLoader это исправляет?**  
Резолвер поля `customer` **не ходит в БД сам**. Он вызывает `customerLoader.load(order.customer_id)`. DataLoader не выполняет запрос сразу — он **накапливает** все переданные id (в рамках одного «тика» выполнения). Когда тик заканчивается, DataLoader один раз выполняет батч: SELECT * FROM customers WHERE id IN (1, 5, 7, ...) — один запрос на все 50 id. Результат раздаётся каждому вызову load(id) по ключу. Плюс **кэш в рамках запроса**: если у 10 заказов один и тот же customer_id, load() для этого id вызовется 10 раз, но запрос к БД будет один — остальные ответы берутся из кэша. Итого вместо 1+50 запросов получается **1+1** (или 1 + несколько батчей, если id очень много).

**Три картинки в голове:**

1. **N+1 без DataLoader.** Ты заказал 50 пицц и к каждой справку «кто заказчик». Без DataLoader ты 50 раз звонишь в справочную: «кто заказчик заказа 1?», «кто заказчик заказа 2?»… DataLoader — один звонок: «дайте справки по заказчикам заказов 1–50»; тебе привозят пачку, ты раздаёшь по номерам заказов.
2. **Кэш DataLoader.** Два заказа от одного и того же клиента (customer_id=7). Резолвер дважды вызовет customerLoader.load(7). DataLoader первый раз пойдёт в БД (или в батч), второй раз отдаст результат из кэша — в БД не идёт.
3. **Prisma vs Hasura.** Prisma — ты сам пишешь резолверы, внутри вызываешь prisma.order.findMany({ include: { customer: true } }) — один запрос с JOIN. Hasura — резолверов нет, ты настраиваешь схему БД и права; Hasura сама генерирует запросы и оптимизирует их (в том числе избегает N+1 на своей стороне).

**Одна мысль — пять раз (GraphQL и БД):**

1. Резолвер = функция, которая возвращает значение поля; часто там запрос к БД. Для списка с вложенным полем резолвер может вызываться N раз — отсюда N+1.
2. DataLoader батчит вызовы load(key) в один запрос (WHERE id IN (...)) и кэширует результат на запрос. N обращений к одному типу сущности превращаются в 1 (или несколько батчей).
3. Один DataLoader на один тип сущности (например customerLoader, orderLoader); создаётся на каждый запрос (или на каждый запрос GraphQL), чтобы кэш не тянулся между разными клиентами.
4. Prisma — ORM в твоём коде; в резолверах вызываешь Prisma, можно combine с DataLoader для сложных вложений. Hasura — без своего кода резолверов; схема и права в метаданных, API и подписки генерируются автоматически.
5. Ограничивай глубину запроса (max depth) и сложность (query complexity), иначе один тяжёлый запрос может положить БД.

**Пошагово: как работает DataLoader (в голове):**

1. Клиент шлёт запрос: orders(limit: 50) { id total customer { name } }.
2. Резолвер orders выполняется один раз → SELECT * FROM orders LIMIT 50 → возвращает 50 заказов.
3. GraphQL для каждого заказа вызывает резолвер поля customer с родителем order. Резолвер делает customerLoader.load(order.customer_id). 50 вызовов load() с разными id (возможно с повторами).
4. DataLoader собирает уникальные id (например 50 штук или меньше из-за повторов). После «тика» вызывается batch-функция с массивом id.
5. Batch-функция выполняет один SQL: SELECT * FROM customers WHERE id = ANY($1), [ids]. Результат раскладывается по id (Map), возвращается массив в том же порядке, что и ids. Каждый load(id) получает свой результат. Данные кэшируются — повторный load(7) в том же запросе не вызовет новый запрос.

**Пошагово: кэш DataLoader на примере двух заказов с одним customer_id:**  
Заказы 1 и 2 оба от customer_id=7. (1) GraphQL вызывает резолвер customer для заказа 1 → customerLoader.load(7). DataLoader ставит 7 в очередь, запроса к БД пока нет. (2) Резолвер customer для заказа 2 → customerLoader.load(7). DataLoader добавляет ещё один 7 (или просто помнит, что 7 нужен). (3) Тик завершается; batch-функция вызывается с массивом [7] (уникальные id). Один запрос: SELECT * FROM customers WHERE id = 7. Результат: один объект. DataLoader кладёт его в кэш по ключу 7. (4) Оба вызова load(7) получают один и тот же объект из кэша. В БД ушёл один запрос вместо двух. Так кэш сокращает и запросы при повторах id.

**Типичная ошибка новичка:** Написать резолвер customer так: return db.query('SELECT * FROM customers WHERE id = ?', [parent.customer_id]). Тогда при 50 заказах выполнится 50 запросов. Решение: вызов DataLoader (load(parent.customer_id)) или один запрос с JOIN/eager load на уровне резолвера orders (вернуть заказы уже с подгруженными клиентами).

### Термины

- **Resolver (резолвер)** — функция в GraphQL, которая возвращает значение для поля; для полей типа объект или список резолвер может обращаться к БД или другому источнику.
- **N+1 в GraphQL** — один запрос за родительский список (например заказы), затем для каждого элемента запрос за вложенные данные (клиент заказа) — итого 1 + N запросов к БД.
- **DataLoader** — паттерн и библиотеки (например graphql/dataloader): при запросе значения по ключу (id) запрос не выполняется сразу, а ключ добавляется в очередь; в конце «тика» выполняется один батч-запрос по всем накопленным ключам; результат кэшируется на время запроса.
- **Prisma** — ORM и генератор клиента для Node.js/TypeScript; схема в prisma.schema; миграции; вызов из резолверов с типизацией.
- **Hasura** — движок GraphQL поверх БД (PostgreSQL и др.); метаданные (таблицы, связи, права) задаются в консоли или YAML; API и подписки генерируются автоматически; RLS для доступа по строкам.

### Теория и правила

#### Откуда N+1 в GraphQL

Запрос клиента:

```graphql
query {
  orders(limit: 50) {
    id
    total
    customer { name }
  }
}
```

Наивная реализация: резолвер `orders` выполняет `SELECT * FROM orders LIMIT 50`; резолвер для поля `customer` у каждого заказа вызывается 50 раз с `order.customer_id` и каждый раз делает `SELECT * FROM customers WHERE id = ?`. Итого 1 + 50 запросов.

**Числовой пример (чтобы запомнить масштаб N+1):**  
Запрос: 50 заказов, у каждого поле `customer { name }`. **Без DataLoader:** 1 запрос (заказы) + 50 запросов (по одному на клиента) = **51 запрос** к БД. При латентности 2 мс на round-trip это уже 51 × 2 = 102 мс только на сеть; при 500 заказах — 501 запрос и ~1 секунда только на round-trip. **С DataLoader:** 1 запрос (заказы) + 1 батч-запрос (все клиенты по id из списка) = **2 запроса**; время сети ~4 мс. Разница в числе запросов и во времени — в десятки раз. Плюс без DataLoader при 100 одновременных таких запросах — 5100 запросов к БД в пике; с DataLoader — 200. Поэтому N+1 в GraphQL нельзя оставлять «на потом».

#### DataLoader (концепция)

1. Резолвер поля `customer` не ходит в БД сам, а вызывает `customerLoader.load(order.customer_id)`.
2. DataLoader собирает все переданные в текущем «тике» id (50 штук).
3. После завершения тика вызывается batch-функция с массивом id: один запрос `SELECT * FROM customers WHERE id IN (1,2,...,50)`.
4. Результат раздаётся по id каждому вызову `load(id)`. В рамках одного запроса один и тот же id не запрашивается дважды (кэш). При очень большом числе уникальных id (например 10 000) один батч может быть тяжёлым для БД; в таких случаях в batch-функции ограничивают размер батча (разбивать массив id на чанки по 500–1000 и выполнять несколько запросов) или настраивают лимит в самой библиотеке DataLoader, если она это поддерживает.

Пример (псевдокод, Node.js):

```javascript
const DataLoader = require('dataloader');

const customerLoader = new DataLoader(async (ids) => {
  const customers = await db.query('SELECT * FROM customers WHERE id = ANY($1)', [ids]);
  const byId = new Map(customers.map(c => [c.id, c]));
  return ids.map(id => byId.get(id) || null);
});

// В резолвере customer:
customer: (order) => customerLoader.load(order.customer_id)
```

**Почему return ids.map(id => byId.get(id)) — порядок важен.**  
DataLoader передаёт в batch-функцию массив **ids в том порядке, в каком вызывали load()**. Он ждёт массив **результатов в том же порядке**: первый элемент ответа — для первого id, второй — для второго id и т.д. Если вернуть просто `customers` (массив из БД), порядок может не совпасть с ids (БД не гарантирует порядок в WHERE id IN (...)). Поэтому собираем Map по id и затем **ids.map(id => byId.get(id))** — так каждый load(id) получит свой результат по позиции в массиве.

**Проверь себя (DataLoader и N+1)**

1. Запрос: 50 заказов с полем customer { name }. Сколько запросов к БД без DataLoader и сколько с DataLoader (в идеале)?
2. Зачем создавать новый DataLoader на каждый GraphQL-запрос, а не один глобальный?
3. Что должно возвращать batch-функция DataLoader по отношению к порядку переданных ids?

<details>
<summary>Ответы</summary>
1. Без DataLoader: 1 (заказы) + 50 (по одному на клиента) = 51. С DataLoader: 1 (заказы) + 1 (батч клиентов) = 2.  
2. Кэш DataLoader привязан к экземпляру; глобальный кэш жил бы между запросами и пользователями — утечка данных и устаревшие данные.  
3. Массив результатов **в том же порядке**, что и переданный массив ids: первый элемент — результат для первого id, и т.д. Иначе load(id) получит чужой результат.
</details>

#### Prisma

- Схема в `prisma.schema` (модели = таблицы); `prisma generate` создаёт клиент с типами. В резолверах вызываешь `prisma.order.findMany({ include: { customer: true } })` — один запрос с JOIN. Для сложных запросов с вложенными списками всё равно можно использовать DataLoader поверх Prisma (батч по id).

#### Hasura

- Подключаешь БД, задаёшь отношения между таблицами и права (roles, RLS). Hasura генерирует запросы, мутации и подписки. N+1 решается на стороне Hasura (оптимизация запросов). Ограничение — логика только в БД (триггеры, RLS) или через Actions/Remote schemas.
- **Подписки (subscriptions)** в GraphQL — real-time обновления: клиент подписывается на изменение данных (например «список заказов»), сервер пушит изменения при мутациях. Hasura даёт подписки из коробки (PostgreSQL LISTEN/NOTIFY или polling). В своём сервере (Prisma и др.) подписки реализуют через WebSocket и при изменении данных в БД отправляют обновление (триггер → очередь или поллинг изменений).

#### Ограничение глубины запросов

Глубоко вложенные запросы (order → customer → company → …) могут порождать тяжёлые JOIN или много round-trip. Ограничивают глубину (max depth) или сложность (query complexity) на уровне сервера.

#### Картинка в голове (GraphQL и N+1, DataLoader)

Клиент просит: «50 заказов, у каждого покажи клиента». Наивный сервер: один запрос — «дай 50 заказов»; потом для заказа 1 — «кто клиент?», для заказа 2 — «кто клиент?» … 50 раз. Как 50 человек в очереди, каждый спрашивает «а какой у меня клиент?» по одному. **DataLoader** — «подождите, я соберу все 50 вопросов и один раз спрошу базу: дай клиентов с такими-то id». Один запрос вместо 50; плюс если у нескольких заказов один и тот же клиент, DataLoader не спрашивает дважды (кэш в рамках запроса).

#### Что будет, если не использовать DataLoader в GraphQL

При росте числа элементов в списке (например 200 заказов с полем customer) число запросов к БД становится 1 + 200; латентность растёт, БД перегружается. В проде при сотнях одновременных таких запросов соединения и CPU исчерпываются. DataLoader сводит к 1 + 1 (или 1 + несколько батчей при очень большом N с лимитом размера батча) и убирает дубли по одному и тому же id.

#### Мини-практикум (§118)

**Упражнение 1.** Запрос: `users { id name posts { title } }`. Резолвер users возвращает 100 пользователей; резолвер posts для каждого пользователя делает SELECT * FROM posts WHERE user_id = ?. Сколько запросов к БД без DataLoader и как исправить?

<details>
<summary>Ответ</summary>
1 + 100 = 101 запрос. Исправить: использовать DataLoader для posts — load(user_id) накапливает user_id, один запрос SELECT * FROM posts WHERE user_id IN (...) или отдельный запрос по списку user_id с группировкой в коде по user_id.
</details>

**Упражнение 2.** Зачем создавать новый экземпляр DataLoader на каждый GraphQL-запрос, а не один глобальный?

<details>
<summary>Ответ</summary>
Кэш DataLoader привязан к экземпляру. Если один DataLoader на все запросы, кэш будет накапливать данные между разными клиентами и запросами: (1) утечка данных между пользователями; (2) устаревшие данные (кэш от предыдущего запроса). На запрос — свой DataLoader, после ответа кэш выбрасывается.
</details>

**Упражнение 3.** В чём ограничение Hasura по сравнению с «своим» GraphQL-сервером на Prisma?

<details>
<summary>Ответ</summary>
Сложная бизнес-логика не в БД: в Hasura её нужно выносить в Actions (вызов внешнего API) или в триггеры/RLS в БД. Свой сервер на Prisma даёт полный контроль в резолверах (любой код, вызовы внешних сервисов, кастомная валидация). Hasura сильна быстрым прототипом и real-time подписками из коробки.
</details>

#### Частые вопросы по §118

**В: DataLoader нужен только в GraphQL?**  
О: Нет. Паттерн батчинга и кэша по ключу применим везде, где есть «много обращений к одному типу сущности по ключам» в рамках одной операции (например один HTTP-запрос). В REST при формировании ответа с вложенными объектами тоже можно использовать такой же приём.

**В: Можно ли смешивать Prisma и DataLoader?**  
О: Да. Prisma даёт удобные запросы с include; для простых случаев одного include достаточно. Для вложенных списков (у пользователя список постов, у поста список комментариев) часто делают DataLoader поверх Prisma: batch-функция вызывает prisma.post.findMany({ where: { userId: { in: ids } } }) и группирует по userId.

**В: DataLoader и «обычный» кэш приложения (Redis, in-memory) — одно и то же?**  
О: Нет. Кэш DataLoader живёт **только в рамках одного запроса** (одного HTTP-запроса или одного выполнения GraphQL-запроса). После ответа клиенту кэш отбрасывается. Он нужен, чтобы в пределах одного запроса не запрашивать один и тот же id дважды (например два заказа с одним customer_id). Обычный кэш приложения (Redis, Memcached) живёт между запросами и между пользователями — для ускорения повторных обращений. DataLoader не заменяет такой кэш; он решает задачу N+1 внутри одного запроса (батчинг + кэш на запрос).

**В: Как ограничить глубину запроса в GraphQL?**  
О: В middleware или при разборе запроса считать глубину вложенности полей (например рекурсивно по дереву запроса). Если глубина > N (например 7) — вернуть ошибку. Аналогично query complexity: присваивать «стоимость» полям и ограничивать суммарную сложность запроса.

#### Памятка: 3 вещи, которые нельзя забыть (§118)

1. **Вложенные поля по списку → риск N+1.** Резолвер вызывается на каждый элемент списка; не делать там отдельный запрос к БД на каждый вызов — использовать DataLoader или один запрос с JOIN/eager load.
2. **DataLoader = батч + кэш на запрос.** Создавать новый экземпляр на каждый GraphQL-запрос.
3. **Ограничивать глубину и сложность запроса,** иначе один тяжёлый запрос может создать тысячи запросов к БД или один огромный JOIN.

#### Чек-лист перед продакшеном (§118)

- [ ] Для полей «связь один ко многим» или «родитель → вложенный объект» по списку используется DataLoader (или один запрос с include/JOIN), а не запрос в цикле.
- [ ] DataLoader создаётся на каждый запрос (не глобально); кэш не тянется между пользователями.
- [ ] Включено ограничение глубины запроса и при необходимости query complexity; логирование тяжёлых запросов.

#### Повтори вслух (§118) — одна фраза на каждое понятие

- **N+1 в GraphQL:** «Один запрос за список + N запросов за вложенное поле (по одному на элемент) — резолвер вызывается на каждый элемент».
- **DataLoader:** «Накапливаю id, один раз спрашиваю БД WHERE id IN (...), раздаю результат по ключам и кэширую на запрос».
- **Зачем DataLoader на каждый запрос:** «Чтобы кэш не жил между разными клиентами и запросами — иначе утечка данных или устаревшие данные».
- **Prisma vs Hasura:** «Prisma — вызываю в своих резолверах; Hasura — резолверов нет, API генерируется по схеме БД и правам».

**Проверь себя (§118)**

1. Почему в GraphQL легко получить N+1?  
2. Как DataLoader уменьшает число запросов к БД?  
3. В чём разница между Prisma и Hasura по роли в GraphQL?  
4. Запрос возвращает 100 заказов с полем `customer { name }`. Сколько запросов к БД без DataLoader и сколько с DataLoader (в идеале)? Почему разница критична под нагрузкой?

<details>
<summary>Ответы</summary>
1. Резолвер вызывается для каждого поля; для поля «customer» у каждого из N заказов резолвер может сделать отдельный запрос к БД — итого 1 + N.  
2. Резолвер отдаёт ключ (id) в DataLoader; DataLoader накапливает ключи и один раз выполняет батч-запрос (WHERE id IN (...)), затем раздаёт результат по ключам и кэширует на запрос.  
3. Prisma — ORM, ты сам пишешь резолверы и вызываешь Prisma в них. Hasura — автогенерация GraphQL API по схеме БД; резолверы не пишутся, настраиваются метаданные и права.  
4. Без DataLoader: 1 (заказы) + 100 (по одному на клиента) = 101 запрос. С DataLoader: 1 (заказы) + 1 (батч клиентов) = 2 запроса. Под нагрузкой: 1000 таких запросов в минуту без DataLoader — 101 000 запросов к БД; с DataLoader — 2000. Латентность и нагрузка на БД отличаются на порядки; без DataLoader БД и приложение могут не выдержать.
</details>

### Запомните

Резолверы в GraphQL часто ходят в БД; при вложенных полях по списку возникает N+1. DataLoader решает это батчингом и кэшем в рамках запроса. Prisma — ORM с типами для использования в резолверах; Hasura — автогенерация API и подписок поверх БД. Ограничивай глубину и сложность запросов.

**Резюме §118 в одну строку:** В GraphQL для полей-связей по списку не делай запрос в цикле — используй DataLoader (батч + кэш на запрос) или один запрос с include/JOIN; DataLoader создавай на каждый запрос.

**Prisma и Hasura в одну фразу.**  
**Prisma** — ты сам пишешь резолверы и внутри вызываешь Prisma (findMany, include и т.д.); полный контроль над логикой и запросами. **Hasura** — резолверов не пишешь; подключаешь БД, настраиваешь таблицы и права; GraphQL API и подписки генерируются автоматически; N+1 решается на стороне Hasura. Выбор: нужен кастомный код и гибкость → Prisma; нужен быстрый API и real-time из коробки без кода резолверов → Hasura.

**Проверь себя (Prisma и Hasura)**

1. Кто генерирует резолверы в GraphQL — Prisma или Hasura?
2. Когда разумнее выбрать Hasura, а когда Prisma?
3. Где решается N+1 при использовании Hasura — в твоём коде или на стороне Hasura?

<details>
<summary>Ответы</summary>
1. Hasura генерирует запросы, мутации и подписки автоматически из схемы БД. Prisma генерирует только клиент для доступа к БД; резолверы пишешь сам и вызываешь в них Prisma.  
2. Hasura — когда нужен быстрый GraphQL API и real-time подписки без написания резолверов; Prisma — когда нужна кастомная логика, полный контроль над запросами и гибкость в коде.  
3. На стороне Hasura: он оптимизирует запросы при генерации API. При Prisma N+1 решаешь сам (DataLoader, include/JOIN).
</details>

**Не путай: N+1 и «просто много запросов».** N+1 — это **один** запрос за список (например 50 заказов) и потом **ровно N** запросов за связь (50 раз «дай клиента по id»). Один и тот же шаблон запроса с разными id. «Просто много запросов» — например на странице 5 разных блоков, каждый со своим запросом (заказы, реклама, рекомендации) — это не N+1, а разная логика. Признак N+1: в логах один запрос за коллекцию и следом N одинаковых по форме запросов с разными id. Решение N+1 — батч (DataLoader или один запрос WHERE id IN (...)) или JOIN/include.

#### Что может пойти не так (GraphQL и БД) — и что делать

| Что случилось | Почему плохо | Что проверить / что делать |
|---------------|--------------|----------------------------|
| Резолвер вложенного поля делает запрос к БД на каждый вызов | N+1: при 100 заказах с полем customer — 101 запрос; под нагрузкой БД и приложение падают. | Использовать DataLoader (load(id) → батч WHERE id IN (...)) или один запрос с include/JOIN на уровне родительского резолвера. |
| Один DataLoader на всё приложение (глобальный) | Кэш живёт между запросами: утечка данных между пользователями; устаревшие данные. | Создавать новый экземпляр DataLoader на каждый GraphQL-запрос (или на каждый HTTP-запрос); после ответа кэш отбрасывается. |
| Не ограничена глубина или сложность запроса | Клиент может запросить 10 уровней вложенности или огромный список — один запрос породит тысячи обращений к БД или один тяжёлый JOIN. | Включить max depth (например 7) и при необходимости query complexity; при превышении возвращать ошибку. |
| В логах сотни одинаковых запросов с разными id после одного запроса за список | Классический N+1. | Найти резолвер поля-связи по списку; заменить прямой запрос на DataLoader.load(id) или вернуть данные уже с JOIN/include из родительского резолвера. |

**Если забыл §118 — вернись сюда.** Список с вложенным полем → резолвер вызывается N раз → не делай там запрос к БД на каждый вызов. DataLoader: накапливаю id, один запрос, раздаю по ключам, кэш на запрос. DataLoader — новый на каждый запрос.
