## 117. Message queues и БД

**Одной фразой:** Очереди сообщений (RabbitMQ, Apache Pulsar) доставляют сообщения подписчикам; чтобы «запись в БД и публикация в очередь» не разъезжались, используют транзакционный outbox (одна транзакция: данные + строка в outbox; отдельный процесс публикует из outbox). Очереди используются и для координации Saga (компенсирующие транзакции).

### Цель раздела

Понять **RabbitMQ и Pulsar** в контексте интеграции с БД, **паттерн outbox** для гарантии доставки, **транзакционный outbox** (реализация через таблицу и CDC/поллинг), и связь с **Saga** (распределённые сценарии через сообщения).

### В этом разделе главное (три строки)

1. **RabbitMQ** — брокер сообщений (очереди, обмены, маршрутизация); сообщения могут быть персистентны на диск; подписчики получают по очереди. **Pulsar** — распределённая очередь с персистентностью, мультитенантностью, совместимостью с Kafka API.
2. **Транзакционный outbox:** приложение в одной транзакции пишет в доменные таблицы и в таблицу outbox (тип события, payload); фоновый процесс (поллинг или CDC) читает outbox и публикует в RabbitMQ/Kafka; после успешной публикации помечает или удаляет строку. Так «записали в БД» и «опубликовали в очередь» не расходятся.
3. **Saga:** длинная бизнес-операция из нескольких шагов в разных сервисах/БД; каждый шаг публикует событие; при сбое выполняются компенсирующие транзакции. Очереди доставляют события между шагами; идемпотентность обработки обязательна.

### Простыми словами

Очередь — «почтовый ящик»: отправитель кладёт сообщение, получатель забирает. Чтобы не было ситуации «деньги списали в БД, но сообщение в очередь не ушло» (или наоборот), пишем в БД и «письмо для очереди» в одну транзакцию в таблицу outbox; отдельный «почтальон» забирает из outbox и реально шлёт в очередь. Saga — сценарий из нескольких действий в разных местах; если что-то пошло не так, откатываем через компенсирующие действия; сообщения в очередях координируют шаги.

### Если ты боишься не понять — самая простая версия (§117)

**Что такое очередь сообщений?**  
Отправитель кладёт сообщение в «ящик» (очередь); получатель забирает сообщения из ящика. Сообщения могут храниться на диске (персистентность) — при перезапуске брокера они не теряются. RabbitMQ и Pulsar — такие брокеры: много очередей/топиков, маршрутизация, подписчики.

**В чём та же проблема, что и с Kafka?**  
Приложение должно и «записать в БД», и «положить сообщение в очередь». Два шага: после коммита в БД отправка в очередь может упасть — сообщение потеряно. Или отправили в очередь до коммита — транзакция откатилась, в очереди ложное событие. **Транзакционный outbox** решает так же, как с Kafka: запись в доменные таблицы и в таблицу outbox в **одной транзакции**; отдельный процесс (поллинг по outbox или CDC) забирает записи из outbox и отправляет в RabbitMQ/Pulsar. Одна атомарная запись в БД — гарантия, что «письмо для очереди» зафиксировано и будет доставлено.

**Что такое Saga простыми словами?**  
Длинная операция из нескольких шагов в **разных сервисах** (каждый со своей БД). Например: «заказ» — списать деньги (сервис A), зарезервировать товар (сервис B), отправить уведомление (сервис C). Нет одной распределённой транзакции «всё или ничего»; каждый шаг — своя локальная транзакция. Если шаг B упал, нужно «откатить» A (вернуть деньги). **Saga** — это как раз такая последовательность шагов с **компенсациями**: при сбое выполняются обратные действия (компенсирующие транзакции). События «шаг выполнен» / «нужна компенсация» передаются через очереди. Обработка сообщений должна быть **идемпотентной**: повтор доставки не должен дважды списать деньги или дважды зарезервировать товар.

**Одна мысль — пять раз (очереди и outbox):**

1. Очередь (RabbitMQ, Pulsar) доставляет сообщения от отправителя к получателю; персистентные сообщения переживают перезапуск брокера.
2. «Записать в БД» и «отправить в очередь» в два шага — риск потери или ложного события. Транзакционный outbox: одна транзакция (домен + outbox); публикация из outbox — отдельным процессом (поллинг или CDC).
3. Процесс публикации: SELECT из outbox WHERE sent_at IS NULL; для каждой строки отправить в очередь; после успеха UPDATE sent_at или DELETE строки.
4. Saga = несколько локальных транзакций в разных сервисах + компенсации при сбое; координация через сообщения в очередях. Идемпотентность обработки обязательна.
5. RabbitMQ — классический брокер (очереди, exchange, routing). Pulsar — распределённый, с персистентностью, мультитенантностью; есть совместимость с Kafka API.

**Пошагово: outbox с RabbitMQ (в голове):**

1. Таблица outbox: id, topic/routing_key, payload, created_at, sent_at (NULL = ещё не отправлено).
2. Приложение в одной транзакции: INSERT в orders; INSERT в outbox (topic='order.created', payload=...). COMMIT.
3. Фоновый воркер раз в N секунд: SELECT * FROM outbox WHERE sent_at IS NULL LIMIT 100. Для каждой строки: отправить payload в RabbitMQ (в нужную очередь/exchange). После успешной отправки: UPDATE outbox SET sent_at = NOW() WHERE id = ?.
4. При сбое воркера неотправленные строки остаются в outbox; при следующем запуске они снова попадут в выборку. Доставка at-least-once; консьюмер должен быть идемпотентным.

**Сценарий от начала до конца: от нажатия кнопки до обработки в воркере (§117):**

- **Шаг 1.** Пользователь нажал «Отправить заказ». API получил запрос.
- **Шаг 2.** Приложение открыло транзакцию: INSERT в `orders`, INSERT в `outbox` (topic='order.created', payload={orderId, items, total}). COMMIT. В БД теперь заказ и одна строка в outbox с sent_at = NULL.
- **Шаг 3.** Ответ пользователю: «Заказ принят». Приложение может больше ничего не делать с очередью.
- **Шаг 4.** Воркер публикации (отдельный процесс) по таймеру выполняет: SELECT id, topic, payload FROM outbox WHERE sent_at IS NULL ORDER BY id LIMIT 100. Находит нашу строку.
- **Шаг 5.** Воркер отправляет payload в RabbitMQ (exchange + routing key, например очередь `order.created`). После успешного ack от брокера: UPDATE outbox SET sent_at = NOW() WHERE id = ?.
- **Шаг 6.** Консьюмер очереди `order.created` (например сервис уведомлений) получает сообщение. Обрабатывает идемпотентно (по order_id проверяет «уже обработано» или выполняет операцию по ключу). Отправляет письмо или обновляет склад.

Итог: один коммит в БД (заказ + outbox) → воркер гарантированно подхватит запись из outbox → сообщение попадёт в очередь → консьюмер обработает. Нет шага «приложение отправило в RabbitMQ» в том же процессе, что и коммит — только запись в БД; доставку делает воркер.

**Типичная ошибка новичка:** Делать отправку в очередь сразу после коммита в коде приложения и считать «мы же оба делаем». При падении между коммитом и send() событие теряется. Outbox убирает этот разрыв: коммит один (данные + outbox), доставку выполняет отдельный надёжный процесс.

**Не путай: outbox с Kafka (§116) и outbox с RabbitMQ (§117).**  
Идея **одна и та же**: приложение в одной транзакции пишет в доменные таблицы и в таблицу outbox; отдельный процесс (CDC или воркер с поллингом) читает outbox и публикует сообщения. Разница только **куда** публикуем: в §116 — в топики Kafka (часто через Debezium, который читает WAL и шлёт в Kafka); в §117 — в RabbitMQ или Pulsar (часто через воркер, который по таймеру делает SELECT из outbox и отправляет в очередь). Схема «одна транзакция, один источник правды в БД, доставку делает отдельный процесс» — общая; меняется только приёмник (Kafka vs RabbitMQ/Pulsar).

### Термины

- **RabbitMQ** — брокер сообщений на AMQP; очереди привязаны к обменам (exchange); маршрутизация по ключу, топику и т.д.; персистентные очереди и сообщения переживают перезапуск.
- **Apache Pulsar** — распределённая система обмена сообщениями; хранение в BookKeeper; темы (topics), подписки (subscriptions); совместимость с Kafka API (Kafka-on-Pulsar).
- **Транзакционный outbox** — таблица в той же БД, что и доменные данные; запись в домен + INSERT в outbox в одной транзакции; публикация из outbox во внешнюю очередь выполняется асинхронно (поллинг или CDC), с пометкой «отправлено» или удалением строки.
- **Saga** — паттерн распределённой транзакции: последовательность локальных транзакций в разных сервисах; каждое действие публикует событие; при сбое выполняются компенсирующие действия (обратные операции). Координация — через обмен сообщениями (choreography) или центральный оркестратор (orchestration).
- **Dead Letter Queue (DLQ)** — очередь для сообщений, которые не удалось обработать после нескольких попыток; позволяет не блокировать основную очередь и не терять события; дальше — разбор, ретраи или переобработка после исправления.

### Теория и правила

#### Зачем outbox с очередями

Без outbox типичный код: открыть транзакцию → записать в БД → закоммитить → отправить в очередь. Если после коммита отправка упадёт, событие потеряно; если отправить до коммита и потом коммит упадёт — в очереди «ложное» событие. Транзакционный outbox устраняет этот разрыв: единственная запись — в БД (домен + outbox); публикация в очередь — детерминированная функция от содержимого outbox, выполняемая отдельным процессом. Либо событие в outbox и будет доставлено, либо транзакция не закоммичена и события нет.

#### Реализация outbox с RabbitMQ

1. Таблица `outbox (id, topic, key, payload, created_at, sent_at)`.
2. Приложение: в одной транзакции INSERT в домен + INSERT в `outbox`.
3. Фоновый процесс: SELECT неотправленные строки (sent_at IS NULL), для каждой публикация в RabbitMQ; после успеха UPDATE sent_at = NOW() или DELETE. **Частота поллинга:** чем чаще опрос (например каждую секунду), тем меньше задержка доставки, но выше нагрузка на БД от SELECT; реже (раз в 5–30 сек) — меньше нагрузка, больше задержка. Выбирать по требованиям к латентности и по метрике «число неотправленных строк». Поллинг с интервалом или использование CDC (Debezium на outbox → Kafka → консьюмер в RabbitMQ или напрямую консьюмер из БД через поллинг).
4. Идемпотентность консьюмера: один и тот же outbox-ид может быть обработан не более одного раза (или консьюмер по ключу сообщения дедуплицирует).

#### Saga и очереди

- События «шаг выполнен» или «компенсация» передаются через очереди. Каждый сервис подписан на нужные темы, выполняет свою локальную транзакцию и публикует результат. При ошибке публикуется «компенсирующее» событие; затронутые сервисы выполняют откат. Идемпотентность обработки обязательна (повтор доставки не должен дублировать эффект).

**Choreography vs orchestration (когда что выбирать).** **Choreography** — каждый сервис подписан на события других и сам решает следующий шаг; нет единого «дирижёра»; проще добавлять новые сервисы, сложнее проследить весь сценарий и отлаживать. **Orchestration** — центральный оркестратор вызывает сервисы по шагам и решает, когда запускать компенсации; один поток управления, проще мониторинг и откат; оркестратор становится точкой отказа и узким местом. Выбор: choreography для простых цепочек и быстрой эволюции; orchestration когда сценарий сложный, компенсации многошаговые и нужна явная трассировка.

**Dead Letter Queue (DLQ).** При повторных неудачах обработки сообщение можно отправлять в отдельную очередь (DLQ), чтобы не блокировать основную и не терять событие. Дальше — разбор вручную, ретраи по расписанию или автоматическая переобработка после исправления кода. Идемпотентность основного консьюмера по-прежнему нужна; DLQ не заменяет её.

**Saga в одну фразу.**  
Несколько шагов в разных сервисах (у каждого своя БД); нет одной распределённой транзакции; при сбое откатываем уже сделанное через **компенсирующие** действия (вернуть деньги, отменить резерв). Очереди доставляют события между шагами; каждый обработчик должен быть идемпотентным (повтор сообщения не должен дважды списать или дважды зарезервировать).

**Проверь себя (Saga и очереди)**

1. Почему в Saga нельзя сделать одну распределённую транзакцию «всё или ничего»?
2. Зачем в Saga обработка каждого сообщения должна быть идемпотентной?
3. В чём разница между choreography и orchestration в контексте Saga?

<details>
<summary>Ответы</summary>
1. Сервисы и БД разные; классическая 2PC по всем участникам сложна, блокирует ресурсы и плохо масштабируется. Saga — последовательность локальных транзакций с компенсациями при сбое.  
2. Очередь может доставить одно сообщение дважды (повтор при таймауте, перезапуск). Без идемпотентности повтор приведёт к двойному списанию или двойному резерву.  
3. Choreography: сервисы подписаны на события друг друга, нет единого дирижёра. Orchestration: центральный оркестратор вызывает сервисы и решает порядок и компенсации.
</details>

#### Картинка в голове (outbox и очереди)

**Без outbox:** ты заплатил в кассе (закоммитил в БД) и пошёл к столу «сообщить о заказе» (очередь). По пути тебя отвлекли — сообщение не отправилось. Кухня не знает про заказ. Или наоборот: ты сначала крикнул «заказ!», а потом не смог заплатить — откатился; кухня уже начала готовить. **С outbox:** ты в одной операции и платишь, и кладёшь в ящик «для кухни» записку. Касса (БД) фиксирует и платёж, и записку разом. Потом курьер (CDC/поллинг) забирает записки из ящика и относит на кухню. Касса — единственное место правды; потерять или продублировать можно только до или после одной атомарной операции.

#### Мини-практикум (§117)

**Упражнение 1.** Почему при Saga обработка сообщений должна быть идемпотентной?

<details>
<summary>Ответ</summary>
Очередь может доставить одно и то же сообщение дважды (повтор при таймауте, перезапуск консьюмера). Если обработчик при каждом получении списывает деньги или резервирует товар, повтор приведёт к двойному списанию или двойному резерву. Идемпотентность (проверка «уже обработано» по ключу, или операция по ключу сама по себе идемпотентна) гарантирует, что повтор не изменит результат.
</details>

**Упражнение 2.** Outbox публикует в RabbitMQ по поллингу (раз в 5 секунд SELECT неотправленных). Что будет, если воркер публикации упадёт на 10 минут?

<details>
<summary>Ответ</summary>
Строки в outbox с sent_at = NULL накопятся. После перезапуска воркер снова сделает SELECT и отправит все накопившиеся сообщения. События не потеряются, будет задержка до 10 минут плюс время на отправку пачки.
</details>

**Упражнение 3.** В чём разница между choreography и orchestration в контексте Saga?

<details>
<summary>Ответ</summary>
Choreography: каждый сервис подписан на события других и сам решает, что делать (и публикует свои события). Нет единого «дирижёра». Orchestration: центральный оркестратор вызывает сервисы и решает, какой шаг следующий и когда запускать компенсации. Choreography проще масштабировать и добавлять сервисы; orchestration проще следить за общим сценарием и отладка централизована.
</details>

#### Памятка: 3 вещи, которые нельзя забыть (§117)

1. **Транзакционный outbox с очередями так же, как с Kafka:** одна транзакция (домен + outbox); публикация — поллинг или CDC; не «коммит → send» в коде приложения.
2. **Saga = несколько шагов в разных сервисах + компенсации.** Очереди доставляют события между шагами; обработка должна быть идемпотентной.
3. **После успешной отправки из outbox помечать строку (sent_at) или удалять,** чтобы не слать одно и то же сообщение бесконечно.

#### Чек-лист перед продакшеном (§117)

- [ ] Публикация в RabbitMQ/Pulsar идёт через outbox (одна транзакция с доменными данными), а не отдельным вызовом после коммита.
- [ ] Воркер публикации из outbox обрабатывает сбои (retry, не падать на одной неуспешной отправке); помечает sent_at после успеха.
- [ ] Консьюмеры обрабатывают сообщения идемпотентно (ключ сообщения, дедупликация или идемпотентная операция по ключу). В Saga компенсирующие шаги тоже идемпотентны.

#### Повтори вслух (§117) — одна фраза на каждое понятие

- **Outbox с очередью:** «Та же идея, что с Kafka: одна транзакция (домен + outbox), отдельный процесс публикует из outbox в RabbitMQ/Pulsar».
- **Почему не „коммит → send“:** «Между коммитом и отправкой приложение может упасть — событие потеряно; outbox убирает этот разрыв».
- **Saga:** «Несколько шагов в разных сервисах; при сбое — компенсации; события через очереди; обработка должна быть идемпотентной».

**Проверь себя (§117)**

1. Почему «записать в БД, потом отправить в очередь» в двух шагах рискованно?  
2. Как транзакционный outbox решает эту проблему?  
3. Зачем в Saga идемпотентность обработки сообщений?

<details>
<summary>Ответы</summary>
1. После коммита БД отправка может упасть — событие потеряно; подписчики не узнают об изменении. Если отправить до коммита — при откате транзакции в очередь уже ушло ложное событие.  
2. Запись в БД и в outbox в одной транзакции; публикация в очередь — отдельный процесс, читающий outbox. Либо транзакция не закоммичена (ни данных, ни события), либо событие есть в outbox и будет доставлено.  
3. Очередь может доставить одно сообщение дважды; без идемпотентности повторная обработка приведёт к двойному списанию или двойному действию. Компенсирующие шаги тоже должны быть идемпотентны.
</details>

### Запомните

RabbitMQ и Pulsar — брокеры сообщений с персистентностью. Транзакционный outbox: одна транзакция (домен + outbox), публикация из outbox отдельным процессом — гарантия «записали ⇒ доставим». Saga координирует распределённые шаги через сообщения; идемпотентность обработки обязательна.

#### Что может пойти не так (очереди и outbox) — и что делать

| Что случилось | Почему плохо | Что проверить / что делать |
|---------------|--------------|----------------------------|
| Воркер публикации из outbox упал на час | В outbox накопились неотправленные строки; подписчики не получают события. | После перезапуска воркер снова SELECT и отправит пачку. События не теряются. Следить за метрикой «число строк с sent_at IS NULL». |
| Воркер отправил сообщение в RabbitMQ, но не успел сделать UPDATE sent_at и упал | При следующем цикле та же строка снова попадёт в SELECT и отправится второй раз — дубликат в очереди. | Консьюмер обязан обрабатывать идемпотентно (по id события или order_id). Помечать sent_at только после успешной отправки; при сбое до UPDATE — допустим повтор, консьюмер дедуплицирует. |
| В Saga один шаг выполнился дважды из-за повторной доставки | Двойное списание, двойной резерв. | Идемпотентность обработчика: по ключу сообщения или бизнес-ключу проверять «уже обработано» или выполнять операцию, которая от повторного вызова не меняет результат (например SET status = 'reserved' вместо «добавь 1 к счётчику»). |

**Пример воркера публикации из outbox (псевдокод, поллинг раз в 5 сек):**

```python
while True:
    rows = db.execute("""
        SELECT id, topic, payload FROM outbox
        WHERE sent_at IS NULL ORDER BY id LIMIT 100
    """).fetchall()
    for row in rows:
        try:
            rabbitmq.publish(row.topic, row.payload)
            db.execute("UPDATE outbox SET sent_at = NOW() WHERE id = %s", [row.id])
        except Exception as e:
            log_error(e)  # retry позже, не помечаем sent_at
    sleep(5)
```

**Резюме §117 в одну строку:** С очередями (RabbitMQ, Pulsar) та же идея, что с Kafka: одна транзакция (домен + outbox), отдельный процесс публикует из outbox; в Saga обработка сообщений должна быть идемпотентной.

**Если забыл §117 — вернись сюда.** Outbox с очередью = так же как с Kafka: одна транзакция (домен + outbox), воркер поллит outbox и шлёт в RabbitMQ/Pulsar. Saga = несколько шагов в разных сервисах, при сбое — компенсации; обработка сообщений идемпотентна.
