## 119. REST API и БД

**Одной фразой:** REST API поверх БД проектируют как ресурсы и HTTP-методы; пагинация — offset или cursor (cursor стабильнее при появлении новых данных); фильтрация и сортировка через query-параметры с валидацией (защита от SQL-инъекций); rate limiting и версионирование API обеспечивают устойчивость и эволюцию.

### Цель раздела

Понять **принципы проектирования REST API** для доступа к данным из БД: ресурсы, методы, **пагинация** (offset vs cursor), **фильтрация и сортировка** (безопасно, без инъекций), **rate limiting** и **версионирование API** (URL, заголовок).

### В этом разделе главное (три строки)

1. **Ресурсы и методы:** сущности — ресурсы (например `/users`, `/orders`); GET — чтение, POST — создание, PUT/PATCH — обновление, DELETE — удаление. Идемпотентность GET, PUT, DELETE; POST — создание, не идемпотентен без идемпотентного ключа.
2. **Пагинация:** offset-based (`?limit=20&offset=40`) — просто, но при появлении новых строк «прыгают» страницы; cursor-based (`?limit=20&after=id_123`) — стабильнее, запрос «следующая страница после id_123».
3. **Фильтрация и сортировка:** параметры вида `?status=active&sort=created_at&order=desc`; значения **никогда** не подставляются в SQL строкой — только через подготовленные выражения (плейсхолдеры); белый список полей для sort и фильтров — защита от инъекций и произвольных полей.

### Простыми словами

REST API — способ открыть данные БД по HTTP: URL = ресурс, метод = действие. Чтобы не отдавать миллионы строк, делаем пагинацию: либо «страница 3 по 20» (offset), либо «дай 20 после такого-то id» (cursor). Фильтры и сортировка — через параметры запроса; на сервере строго: только разрешённые поля и операторы, значения в запрос только через параметры (prepared statement). Rate limiting — «не более N запросов в минуту с одного клиента», чтобы БД не легла. Версионирование — явно указывать версию API (/v1/ или заголовок), чтобы менять контракт без поломки старых клиентов.

### Если ты боишься не понять — самая простая версия (§119)

**Что такое REST API по сути?**  
Клиент шлёт HTTP-запрос по URL; сервер возвращает ответ (часто JSON). **Ресурс** — то, с чем мы работаем (пользователи, заказы), обычно отображается в путь: `/users`, `/orders`. **Метод** — что делаем: GET (прочитать), POST (создать), PUT/PATCH (обновить), DELETE (удалить). Данные в БД — сервер по запросу выполняет SQL и отдаёт результат. Чтобы не отдавать сразу миллион строк, делаем **пагинацию**: клиент запрашивает «порциями» (например по 20).

**Offset vs cursor — почему cursor лучше при частых вставках?**  
**Offset:** «дай 3-ю страницу» = limit=20, offset=40. БД делает LIMIT 20 OFFSET 40 — но внутренне она всё равно «пробегает» первые 40 строк (их не отдаёт, но читает). При большом offset запрос тормозит. Плюс: пока клиент листает, в таблицу могут вставлять новые строки в начало; тогда «страница 2» при следующем запросе уже сдвинется — пользователь увидит дубли или пропуски. **Cursor:** «дай 20 записей после id=12345». Запрос: WHERE id > 12345 ORDER BY id LIMIT 20. БД использует индекс по id, не пробегает лишние строки. Вставки в начало списка не сдвигают «следующую страницу», потому что курсор привязан к id. Стабильнее и быстрее.

**Почему нельзя подставлять sort в SQL строкой?**  
Клиент передаёт `?sort=created_at&order=desc`. Если на сервере сделать SQL так: `ORDER BY " + sort + " " + order`, то злоумышленник может передать sort=`created_at; DELETE FROM users --`. В запрос попадёт произвольный SQL (инъекция). Поэтому **только белый список**: в коде задан набор разрешённых полей (created_at, total, status); значение от клиента только выбирает один из них. Или маппинг «безопасный ключ → выражение ORDER BY». В сам SQL строка от клиента **никогда** не вклеивается; для значений фильтров — только плейсхолдеры (prepared statement).

**Одна мысль — пять раз (REST и БД):**

1. Ресурсы = URL, методы = GET/POST/PUT/PATCH/DELETE. Пагинация обязательна для списков; cursor-based стабильнее и быстрее при больших смещениях, чем offset.
2. Фильтрация и сортировка — через query-параметры. Имена полей — только из белого списка; значения — только через параметры запроса (prepared statement), никогда не склеивать с SQL строкой.
3. Rate limiting — ограничение запросов с одного клиента (по IP или токену) в единицу времени; при превышении 429 Too Many Requests. Защита БД от перегрузки и злоупотреблений.
4. Версионирование API (/v1/users или заголовок) — чтобы менять формат и поля без поломки старых клиентов; старую версию поддерживают до deprecation.
5. Идемпотентность: GET, PUT, DELETE идемпотентны; POST создаёт новый ресурс — не идемпотентен, если нет идемпотентного ключа (см. часть XXI).

**Пошагово: как безопасно собрать запрос с сортировкой (в голове):**

1. Клиент шлёт GET /orders?status=shipped&sort=created_at&order=desc&limit=20&offset=0.
2. Сервер парсит параметры. sort может быть только из списка: ALLOWED_SORT = ['created_at', 'total', 'status']. Если пришло что-то другое — подставляем значение по умолчанию (например created_at).
3. order только 'asc' или 'desc'; иначе — по умолчанию.
4. Собираем SQL: ORDER BY создаём из заранее заданного маппинга, например order_clause = {'created_at_asc': 'created_at ASC', 'created_at_desc': 'created_at DESC', ...}[safe_key]. В сам текст SQL **ни sort, ни order от клиента не подставляются как строки**.
5. WHERE status = $1, LIMIT $2, OFFSET $3 — все значения через плейсхолдеры. Выполняем запрос, возвращаем JSON.

**Типичная ошибка новичка:** Собрать ORDER BY как `ORDER BY ${sort} ${order}` из параметров запроса. Достаточно передать sort=`(SELECT 1); DROP TABLE orders;--` или тяжёлый подзапрос — инъекция или DoS. Всегда белый список полей и направлений.

### Термины

- **REST (Representational State Transfer)** — стиль API: ресурсы идентифицируются URL, операции — HTTP-методами (GET, POST, PUT, PATCH, DELETE); без состояния на сервере между запросами.
- **Offset-based pagination** — пагинация по смещению: `limit` и `offset`; запрос «следующая страница» — увеличить offset. При вставке новых строк порядок может меняться.
- **Cursor-based pagination** — пагинация по курсору (например id последнего элемента): «следующая страница» — `?after=last_id&limit=20`. Стабильнее при частых вставках.
- **Rate limiting** — ограничение числа запросов от клиента (по IP, по токену) в единицу времени; защита от перегрузки и злоупотреблений.
- **API versioning** — явное указание версии контракта: в URL (`/v1/users`) или в заголовке (`Accept: application/vnd.api+v1`); позволяет менять формат ответа и поля без поломки старых клиентов.

### Теория и правила

#### Ресурсы и методы

- GET `/users` — список (с пагинацией); GET `/users/:id` — один. POST `/users` — создание; PUT/PATCH `/users/:id` — обновление; DELETE `/users/:id` — удаление.
- Коды ответов: 200 OK (тело есть), 201 Created (создан ресурс, в теле или Location ссылка), 204 No Content (успех без тела — типично для DELETE или обновления без возврата объекта), 400 Bad Request, 404 Not Found, 429 Too Many Requests (rate limit). **204 vs 200 для DELETE:** оба допустимы; 204 явно говорит «успех, тела нет», экономя трафик; 200 с телом — если возвращаешь удалённый объект или метаданные.

#### Пагинация

- **Offset:** `GET /orders?limit=20&offset=40`. В БД: `LIMIT 20 OFFSET 40`. Минус: при большом offset запрос замедляется (БД всё равно «пробегает» offset строк); при вставке новых записей в начало списка страницы «сдвигаются».
- **Cursor:** `GET /orders?limit=20&after=12345` (id последнего выданного заказа). Запрос: `WHERE id > 12345 ORDER BY id LIMIT 21` (21 чтобы узнать «есть ли ещё»). Плюс: стабильный порядок, быстрый запрос при индексе по id. Курсор может быть составным (например created_at + id для уникальности).
- **Ограничение максимального limit:** Всегда задавать **верхнюю границу** размера страницы (например max 100). Иначе клиент может передать `limit=1000000` — один запрос вытащит огромный объём данных, нагрузит БД и приложение и может привести к таймауту или OOM. В коде: `limit = min(requested_limit, MAX_PAGE_SIZE)` перед подстановкой в запрос.

**Числовой пример (offset vs cursor, чтобы запомнить разницу):**  
В таблице 100 000 заказов. Клиент запрашивает «страницу по 20». **Offset: третья страница** — limit=20, offset=40. БД выполняет что-то вроде: прочитать строки в порядке ORDER BY, пропустить первые 40, вернуть следующие 20. На 50-й странице: offset=980 — БД всё равно «пробегает» 980 строк (по индексу или по таблице), чтобы их отбросить. **Offset 50 000:** limit=20, offset=50000 — БД должна обработать 50 020 строк и вернуть 20; время растёт с offset. **Cursor:** «20 записей после id=99980» — запрос WHERE id > 99980 ORDER BY id LIMIT 20. По индексу по id БД сразу попадает в нужное место и читает только 20 строк. Не важно, 50-я это «страница» или 5000-я — стоимость запроса примерно одинаковая. Поэтому для длинных списков и частых «глубоких» переходов cursor предпочтительнее.

**Проверь себя (пагинация REST API)**

1. Почему запрос с offset=100000 и limit=20 может быть медленным?
2. При cursor-пагинации клиент передаёт after=12345. Какой SQL разумно выполнить для «следующей страницы»?
3. Сортировка по полю created_at не уникальна. Как сделать курсор стабильным?

<details>
<summary>Ответы</summary>
1. БД должна «пройти» первые 100000 строк, чтобы отбросить их и вернуть 20; это много чтения. Cursor (WHERE id > last_id) использует индекс и читает только нужные строки.  
2. WHERE id > 12345 ORDER BY id ASC LIMIT 21 (21 чтобы понять, есть ли ещё страница; если вернулось 21 — next_cursor = id 20-го).  
3. Составной курсор: (created_at, id). Запрос «следующая страница»: WHERE (created_at, id) < (last_created_at, last_id) ORDER BY created_at DESC, id DESC LIMIT 21.
</details>

#### Фильтрация и сортировка

- Пример: `GET /orders?status=shipped&sort=created_at&order=desc`.
- **Безопасность:** не подставлять значения в SQL строкой. Разрешённые поля для sort — белый список (`created_at`, `total`); порядок — только `asc`/`desc`. Значения фильтров — только через плейсхолдеры. Так исключаются SQL-инъекции и произвольные поля.

**Правильный способ (безопасно):** не подставлять в SQL строку, пришедшую от клиента. Использовать **маппинг «безопасный ключ → фрагмент SQL»**; клиент может выбрать только один из ключей.

```python
# Белый список: только эти комбинации разрешены. В SQL попадает только значение из словаря.
SORT_CLAUSE = {
    'created_at_asc': 'created_at ASC',
    'created_at_desc': 'created_at DESC',
    'total_asc': 'total ASC',
    'total_desc': 'total DESC',
}
safe_key = params.get('sort', 'created_at_desc')
if safe_key not in SORT_CLAUSE:
    safe_key = 'created_at_desc'
order_clause = SORT_CLAUSE[safe_key]   # это строка из нашего кода, не от клиента
sql = f"SELECT * FROM orders WHERE status = %s ORDER BY {order_clause} LIMIT %s OFFSET %s"
cursor.execute(sql, [params.get('status'), limit, offset])
```

**Почему нельзя так (опасно):**  
Если сделать `sort_field = params.get('sort')` и потом `ORDER BY {sort_field}` в SQL, то злоумышленник передаст `sort=created_at; DROP TABLE orders;--` — в запрос попадёт произвольный SQL. Даже «проверка» `if sort_field in ALLOWED_SORT` не всегда спасает: если где-то по ошибке подставить значение не из белого списка, риск остаётся. Поэтому надёжно: **в ORDER BY подставлять только строки из заранее заданного словаря в коде**, а параметр от клиента только выбирает ключ этого словаря (и если ключа нет — подставляем ключ по умолчанию).

1. Почему параметр sort от клиента нельзя подставлять в SQL как строку?
2. Что такое «белый список» полей для сортировки и зачем он нужен?
3. Значение фильтра (например status=active) — как его безопасно передать в запрос к БД?

<details>
<summary>Ответы</summary>
1. В sort можно подставить произвольный SQL (инъекция): например sort=created_at; DROP TABLE orders;-- или тяжёлый подзапрос — потеря данных или DoS.  
2. Заранее заданный в коде набор разрешённых полей и направлений (asc/desc). Клиент может выбрать только один из них; в SQL попадает только строка из маппинга по ключу, а не произвольный ввод.  
3. Только через плейсхолдер (prepared statement): параметр запроса передаётся отдельно от текста SQL, значение не вклеивается в строку — защита от инъекций.
</details>

#### Картинка в голове (REST, пагинация, фильтры)

**Ресурс** — «объект» по URL: `/users` — коллекция, `/users/5` — один. Метод — действие: GET прочитал, POST создал, PUT обновил. **Offset-пагинация** — «дай страницу 3»: как в книге «открой с 41-й страницы». Если кто-то вставил листы в начало, нумерация съехала. **Cursor-пагинация** — «дай 20 записей после вот этой» (по id): следующая страница не зависит от вставок в начало. **Фильтры** — «покажи только активных»: параметр `?status=active`; на сервере значение подставляется только через параметр запроса (prepared statement), а не вклеивается в SQL — иначе злоумышленник может подставить `'; DROP TABLE users; --` и т.п.

#### Что будет, если подставлять параметры сортировки в SQL строкой

Если взять `sort=created_at` и собрать SQL как `ORDER BY " + sort + " DESC"`, а клиент передаст `sort=created_at; DELETE FROM users --`, в запрос попадёт произвольный SQL. Даже «безобидный» `sort=(SELECT COUNT(*) FROM payments)` может привести к тяжёлым подзапросам и DoS. Поэтому только белый список: допустимые имена полей и направление (asc/desc) задаются в коде; значение от клиента только выбирает один из разрешённых вариантов.

#### Что может пойти не так (REST API поверх БД) — и что делать

| Что случилось | Почему плохо | Что проверить / что делать |
|---------------|--------------|----------------------------|
| Сортировка собрана из параметра запроса строкой | SQL-инъекция: в sort подставят `; DROP TABLE ...` или тяжёлый подзапрос — потеря данных или DoS. | Только белый список полей и направлений; в SQL подставлять только заранее заданные фразы из маппинга по безопасному ключу. |
| Нет rate limiting | Один клиент или бот может слать тысячи запросов в секунду — БД и приложение падают, остальные пользователи не обслуживаются. | Включить rate limit (по IP или по токену); при превышении возвращать 429 Too Many Requests и Retry-After. Счётчики в Redis или в памяти. |
| Пагинация только offset, пользователи листают «глубоко» | Запросы с offset=100000 медленные (БД пробегает 100k строк); при вставке новых записей страницы «прыгают» — дубли или пропуски. | Добавить cursor-based пагинацию (after=id или составной курсор); для больших offset ограничить максимум или перейти на cursor. |
| API без версионирования, меняем формат ответа | Старые клиенты ломаются при изменении полей или структуры. | Версионировать API: /v1/ в URL или заголовок Accept; обратная совместимость или явный deprecation и срок поддержки старой версии. |
| Нет ограничения максимального limit в пагинации | Клиент может передать limit=1000000 — один запрос вытащит огромный объём, таймаут БД/приложения или OOM. | Всегда ограничивать сверху: limit = min(запрос_клиента, MAX_PAGE_SIZE), например 100; в документации API указать максимальный допустимый limit. |

#### Rate limiting

- Ограничение по IP или по API-ключу: не более N запросов в минуту/час. При превышении — 429 и заголовок Retry-After. Реализация: счётчики в Redis или в памяти; скользящее окно или фиксированное окно. Защищает БД от перегрузки и от злоупотреблений.

**Почему возвращаем именно 429 (и что это даёт).**  
Код **429 Too Many Requests** — стандартный HTTP-код для ситуации «клиент сделал слишком много запросов за период». Клиент (браузер, мобильное приложение, скрипт) может **обработать** этот код: например показать пользователю «подождите» или повторить запрос позже, используя заголовок **Retry-After** (через сколько секунд можно повторить). Если вместо 429 «просто ждать» и не отвечать, клиент будет держать соединение открытым — при массовом превышении лимита исчерпаются соединения и потоки на сервере. 429 явно говорит: «запрос не выполнен из-за лимита; прекратите слать запросы сейчас; повторите позже» — и сервер сразу освобождает ресурсы.

**Проверь себя (rate limiting и версионирование)**

1. Зачем возвращать 429 при превышении rate limit вместо того, чтобы просто дольше обрабатывать запрос?
2. Где обычно хранят счётчики для rate limiting при нескольких инстансах приложения?
3. Зачем версионировать API (/v1/ или заголовок)? Что будет, если менять формат ответа без версии?

<details>
<summary>Ответы</summary>
1. Если «просто ждать», клиент держит соединение открытым; при массовом превышении сервер исчерпает соединения и потоки. 429 освобождает ресурсы и явно говорит «повторите позже» (Retry-After).  
2. В Redis (или другом общем хранилище), чтобы лимит был общий для всех инстансов. В памяти — только для одного инстанса.  
3. Чтобы менять контракт (поля, структуру) без поломки старых клиентов: старая версия остаётся доступной до deprecation. Без версии изменение формата сломает существующие клиенты.
</details>

#### Версионирование API

- **В URL:** `/v1/users`, `/v2/users` — явно, кэширование и маршрутизация простые. При смене контракта добавляется v2; v1 поддерживается до deprecation.
- **В заголовке:** `Accept: application/vnd.myapi.v1+json` — URL не меняется, версия в контракте. Обратная совместимость: новые поля добавлять можно; удаление или переименование — новая версия.

#### Мини-практикум (§119)

**Упражнение 1.** Клиент запрашивает GET /orders?sort=created_at; DROP TABLE orders;--&order=desc. Сервер подставляет sort в SQL: ORDER BY created_at; DROP TABLE orders;-- DESC. Что произойдёт?

<details>
<summary>Ответ</summary>
Выполнится два оператора: ORDER BY created_at (часть запроса) и DROP TABLE orders; (инъекция). Таблица orders может быть удалена. Поэтому sort должен браться только из белого списка; значение от клиента не вставлять в SQL как строку.
</details>

**Упражнение 2.** Почему при offset=100000 запрос «SELECT * FROM orders ORDER BY created_at LIMIT 20 OFFSET 100000» может быть медленным?

<details>
<summary>Ответ</summary>
БД должна «пройти» первые 100000 строк (по индексу или таблице), чтобы отбросить их и вернуть следующие 20. Это много чтения и работы. Cursor (WHERE id > last_id ORDER BY id LIMIT 20) использует индекс по id и читает только нужные строки — быстрее.
</details>

**Упражнение 3.** Зачем возвращать 429 Too Many Requests вместо «просто подождать» при превышении rate limit?

<details>
<summary>Ответ</summary>
Если «просто ждать», клиент (или бот) будет держать соединение и поток занятым; при массовом превышении лимита сервер исчерпает ресурсы. 429 говорит «прекрати слать запросы сейчас»; клиент может повторить позже (Retry-After). Защита от перегрузки и от злоупотреблений.
</details>

#### Частые вопросы по §119

**В: Как реализовать cursor, если сортировка по полю не уникальному (например created_at)?**  
О: Использовать составной курсор: (created_at, id). Запрос «следующая страница»: WHERE (created_at, id) < (last_created_at, last_id) ORDER BY created_at DESC, id DESC LIMIT 21. Так однозначно определяем позицию; дубли по created_at не путают порядок.

**Составной курсор в одну фразу.**  
Если сортировка по одному полю (например created_at) не уникальна, курсор делают из **двух полей**: (created_at, id). В запрос «следующая страница» подставляем последние значения обоих полей; в ORDER BY указываем оба: ORDER BY created_at DESC, id DESC. Тогда позиция однозначна и страницы не «прыгают» при одинаковых датах.

**В: Где хранить счётчики для rate limiting?**  
О: В памяти (просто, но при нескольких инстансах приложения лимит не общий) или в Redis (общий лимит для всех инстансов, быстрый доступ). Алгоритм: скользящее окно или фиксированное окно; ключ — IP или API-ключ; значение — счётчик или список меток времени запросов.

**В: Версионирование в URL vs в заголовке — что выбрать?**  
О: В URL (/v1/...) проще кэширование и маршрутизация; версия явно видна. В заголовке URL не меняется; версия в контракте содержимого. Оба варианта допустимы; часто в URL проще для публичных API и прокси-кэша.

**В: Нужны ли в REST ссылки (HATEOAS) и ETag для кэширования?**  
О: **HATEOAS** (ссылки на связанные ресурсы в теле ответа) — опционально; многие API обходятся без него, отдавая только данные. Добавлять имеет смысл, если клиенты должны «ходить по ссылкам» без жёстко заданных URL. **ETag и условные запросы** (If-None-Match): при повторном GET с тем же ETag сервер может вернуть 304 Not Modified без тела — экономия трафика и нагрузки на БД; полезно для редко меняющихся ресурсов. Для списков с пагинацией и частыми изменениями ETag используют реже; приоритет в этой части — пагинация, белый список и rate limiting.

#### Памятка: 3 вещи, которые нельзя забыть (§119)

1. **Пагинация для списков:** offset прост, но при больших смещениях медленный и нестабильный при вставках; cursor (after=id или составной) стабильнее и быстрее.
2. **Фильтры и сортировка:** только белый список полей/операторов; значения в SQL только через плейсхолдеры (prepared statement). Никакой подстановки строк от клиента в SQL.
3. **Rate limiting и версионирование** — обязательны для продакшен-API: защита БД и возможность эволюции контракта без поломки клиентов.

#### Чек-лист перед продакшеном (§119)

- [ ] Списки отдаются с пагинацией (limit; offset или cursor). Есть максимальный limit (например не больше 100).
- [ ] Поля для sort и фильтров заданы белым списком; в SQL подставляются только через маппинг/белый список; значения — через prepared statement.
- [ ] Включён rate limiting (по IP или токену); при превышении возвращается 429 и при необходимости Retry-After.
- [ ] API версионируется (URL или заголовок); обратная совместимость соблюдается или объявлен deprecation старой версии.

#### Повтори вслух (§119) — одна фраза на каждое понятие

- **Пагинация cursor vs offset:** «Cursor = после какого id продолжать; стабильнее при вставках и быстрее на больших смещениях. Offset = с какой строки начать; при вставках страницы сдвигаются».
- **Сортировка безопасно:** «Имена полей и направление — только из белого списка в коде; значение от клиента никогда не вклеивать в SQL строкой».
- **Rate limiting:** «Ограничение числа запросов с клиента в единицу времени; при превышении 429; защита БД от перегрузки».
- **Версионирование API:** «Явно указывать версию (/v1/ или заголовок), чтобы менять контракт без поломки старых клиентов».

**Проверь себя (§119)**

1. Почему cursor-based пагинация предпочтительнее offset при частых вставках?  
2. Как защититься от SQL-инъекции при динамической сортировке (sort=...)?  
3. Зачем rate limiting для API поверх БД?

<details>
<summary>Ответы</summary>
1. При offset вставка новых строк «сдвигает» страницы — пользователь может увидеть дубли или пропуски. Cursor привязан к id (или паре полей); следующий набор стабилен. Плюс большие offset дороги для БД (проход по многим строкам).  
2. Белый список полей для sort: только разрешённые имена полей подставляются в ORDER BY; значение от клиента не вставляется в SQL как строка. Или маппинг «безопасный ключ → выражение ORDER BY».  
3. Ограничение числа запросов защищает БД от перегрузки (один клиент или бот не может исчерпать ресурсы) и от злоупотреблений; при превышении возвращается 429.
</details>

### Запомните

REST: ресурсы и HTTP-методы. Пагинация — offset (просто) или cursor (стабильнее и быстрее на больших смещениях). Фильтрация и сортировка — только через параметры и белый список полей, запросы через prepared statements. Rate limiting — защита БД; версионирование API — обратная совместимость и эволюция контракта.

#### Пример: безопасная сортировка и cursor-пагинация (псевдокод)

**Белый список для sort (никогда не подставлять значение от клиента в SQL как строку):**

```python
ALLOWED_SORT = {"created_at", "total", "status"}
SORT_SQL = {
    "created_at": "created_at ASC",
    "created_at_desc": "created_at DESC",
    "total": "total ASC",
    "total_desc": "total DESC",
}
sort_key = req.args.get("sort", "created_at_desc")
if sort_key not in SORT_SQL:
    sort_key = "created_at_desc"
order_clause = SORT_SQL[sort_key]
# В запрос подставляем только order_clause из словаря, не req.args.get("sort")
sql = f"SELECT * FROM orders WHERE status = %s ORDER BY {order_clause} LIMIT %s OFFSET %s"
# или для cursor:
# WHERE (created_at, id) < (%s, %s) ORDER BY created_at DESC, id DESC LIMIT %s
```

**Cursor-пагинация (следующая страница после last_id):**

```sql
-- Клиент передал ?limit=20&after=12345 (id последнего выданного заказа)
SELECT * FROM orders
WHERE id > 12345
ORDER BY id ASC
LIMIT 21;
-- 21 чтобы понять «есть ли ещё»; если вернулось 21 — next_cursor = id последнего из 20
```

**Резюме §119 в одну строку:** REST поверх БД: пагинация (лучше cursor), фильтры и сортировка только через белый список и плейсхолдеры, rate limiting и версионирование API обязательны.

**Сравнение: offset и cursor пагинация**

| | Offset | Cursor |
|--|--------|--------|
| **Запрос** | LIMIT 20 OFFSET 40 | WHERE id > last_id ORDER BY id LIMIT 20 |
| **При вставке новых строк «в начало»** | Страницы сдвигаются, возможны дубли/пропуски | Следующая страница стабильна (привязана к id) |
| **При большом смещении (100 000)** | БД по сути читает и отбрасывает 100k строк — медленно | Индекс по id, читаем только нужные строки — быстрее |
| **Сложность** | Проще (два числа: limit, offset) | Нужно передавать курсор (id или составной ключ) |

**Почему сортировку нельзя подставлять из запроса строкой.** Потому что тогда значение от клиента становится частью SQL. Достаточно передать `sort=created_at; DROP TABLE orders;--` — и в запрос попадёт два оператора. Даже без злого умысла опечатка или лишний символ могут сломать запрос. Белый список: в коде задан набор допустимых полей и направлений; из запроса берётся только «какой из разрешённых вариантов выбрать», а не произвольная строка для вставки в SQL.

**Если забыл §119 — вернись сюда.** Пагинация: cursor стабильнее и быстрее при больших смещениях. Сортировка и фильтры: только белый список + плейсхолдеры для значений. Rate limiting и версионирование API — обязательно.
