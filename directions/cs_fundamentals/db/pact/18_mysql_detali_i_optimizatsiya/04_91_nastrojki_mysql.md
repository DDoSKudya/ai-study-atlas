## 91. Настройки MySQL

### Цель раздела

Разобраться с **ключевыми параметрами конфигурации MySQL** (`my.cnf` / `my.ini`), которые влияют на память, I/O, InnoDB, логирование и подключения, и понять **подход** к их настройке.

### В этом разделе главное (три строки)

1. Конфигурация MySQL задаётся в файле `my.cnf` (Linux) или `my.ini` (Windows) и разделена на секции (`[mysqld]`, `[client]` и т.д.).
2. **Пара десятков параметров** (buffer pool, log file size, flush‑режимы, max_connections, slow query log) сильнее всего влияют на поведение сервера.
3. Настройка — это **реакция на симптомы и измерения**, а не копирование чужих «оптимальных конфигов».

### Термины

- **`my.cnf` / `my.ini`** — основной конфигурационный файл MySQL.
- **`innodb_buffer_pool_size`** — размер buffer pool InnoDB.
- **`innodb_log_file_size`** — размер одного файла redo‑лога InnoDB.
- **`innodb_flush_log_at_trx_commit`** — режим синхронизации redo‑лога при коммите.
- **`max_connections`** — максимальное количество одновременных подключений к серверу.
- **`slow_query_log`, `long_query_time`** — включение и порог логирования медленных запросов.
- **`innodb_io_capacity`** — оценка способности дисковой подсистемы обрабатывать I/O.

### Теория и правила

**1. Структура `my.cnf`.**

```ini
[mysqld]
innodb_buffer_pool_size = 8G
innodb_log_file_size    = 1G
innodb_flush_log_at_trx_commit = 1
max_connections         = 300
slow_query_log          = ON
long_query_time         = 0.5

[client]
default-character-set   = utf8mb4
```

- Секция `[mysqld]` — настройки сервера;
- `[client]` — настройки клиента по умолчанию;
- возможны и другие секции (`[mysqld_safe]`, `[mysql]` и т.п.).

**2. Ключевые группы параметров.**

- **Память:**
  - `innodb_buffer_pool_size` — 60–80% RAM на выделенном сервере БД (но не бездумно);
  - `innodb_buffer_pool_instances` — разбивка buffer pool на несколько инстансов при больших размерах;
  - `tmp_table_size`, `max_heap_table_size` — лимиты для внутренних/Memory‑таблиц.

- **InnoDB:**
  - `innodb_log_file_size` / `innodb_log_files_in_group` — суммарный размер redo‑лога;
  - `innodb_flush_log_at_trx_commit`:
    - `1` — максимальная надёжность: запись и flush на диск при каждом коммите;
    - `2` — запись в OS‑кэш при каждом коммите, flush раз в секунду;
    - `0` — запись и flush примерно раз в секунду (быстрее, но риск потери до 1 секунды данных при сбое).

- **Подключения:**
  - `max_connections` — лимит одновременных соединений;
  - необходимость пула соединений на стороне приложения.

- **Логирование:**
  - `slow_query_log` = ON;
  - `long_query_time` — порог, чаще ставят 0.2–1.0 секунды на бою;
  - `log_output` = FILE/TABLE.

- **I/O:**
  - `innodb_io_capacity`, `innodb_io_capacity_max` — ориентиры для фоновых операций (flush, purge);
  - на SSD можно ставить выше, чем на HDD.

- **Поведение SQL и строки:**
  - `sql_mode` — набор «флажков», управляющих строгим режимом:
    - `STRICT_TRANS_TABLES`, `STRICT_ALL_TABLES` — строгая обработка невалидных значений (ошибка вместо тихого усечения);
    - `ONLY_FULL_GROUP_BY` — запрет неоднозначных `GROUP BY`;
    - `NO_ZERO_DATE`, `ERROR_FOR_DIVISION_BY_ZERO` и т.п.;
  - `character_set_server`, `collation_server` — кодировка и сортировка по умолчанию (для новых баз/таблиц).

  Обычно для серьёзных проектов включают **строгий режим**:

  ```ini
  sql_mode = STRICT_TRANS_TABLES,NO_ENGINE_SUBSTITUTION
  ```

  или более жёсткий набор. Идея: **лучше уронить запрос с ошибкой, чем молча записать невалидные данные.**

- **Устаревший Query Cache (для старых версий MySQL < 8.0):**
  - `query_cache_size`, `query_cache_type` — старый механизм кеширования результатов `SELECT`;
  - казался привлекательным («сделаем кеш в самой MySQL»), но:
    - требовал глобальных блокировок при изменении таблиц;
    - плохо масштабировался на многопроцессорных системах;
    - приводил к непредсказуемым задержкам.
  - В MySQL 8.0 **полностью удалён**.  
    На старых версиях, как правило, рекомендуется:
    ```ini
    query_cache_type = 0
    query_cache_size = 0
    ```
    и кешировать на уровне приложения/прокси.

- **Потоки I/O InnoDB (для старых версий):**
  - `innodb_read_io_threads`, `innodb_write_io_threads` — количество потоков чтения/записи;
  - на современных версиях и при использовании `innodb_io_capacity` обычно трогать **не нужно**:
    - значения по умолчанию подобраны адекватно;
    - «закручивание» этих параметров без понимания редко даёт выигрыш.
  - Важно: если ты **не профилировал I/O и не читал документацию по своей версии**, лучше оставить эти параметры в покое.

#### Симптом → какие параметры смотреть в первую очередь

Чтобы было проще «привыкнуть» к конфигу, держи в голове несколько типовых ситуаций:

- **«Запросы с сортировкой и JOIN’ами безумно создают временные файлы на диске»**
  - смотри:
    - `tmp_table_size`, `max_heap_table_size` — мало памяти на временные таблицы;
    - `workload`: много `ORDER BY`/`GROUP BY` без индексов;
  - решение:
    - добавить индексы под типичные сортировки/группировки;
    - аккуратно увеличить лимиты для tmp‑таблиц.

- **«CPU свободен, а диск постоянно на 100%, всё тормозит»**
  - смотри:
    - `innodb_buffer_pool_size` — возможно, слишком мал, и всё летает через диск;
    - `innodb_io_capacity` — может быть занижен относительно возможностей диска;
    - частоту чекпоинтов и flush (InnoDB + binlog).

- **«Много соединений, но все как будто стоят в очереди»**
  - смотри:
    - `max_connections` — может быть слишком велико по сравнению с ресурсами;
    - используешь ли пул соединений в приложении;
    - нет ли долгих транзакций, держащих блокировки.

Идея простая: **никогда не трогать конфиг в вакууме** — сначала зафиксировать симптом, потом подумать, какие группы настроек на это влияют.

### Пошагово: минимальный набор действий при первом разворачивании MySQL

1. **Оценить доступную RAM и назначение сервера.**
   - Если сервер **выделен под MySQL**, можно дать `innodb_buffer_pool_size ~ 60–70%` RAM.
   - Если MySQL соседствует с другими сервисами — скромнее и с учётом их потребностей.

2. **Настроить buffer pool и redo‑лог.**

```ini
[mysqld]
innodb_buffer_pool_size = 8G
innodb_log_file_size    = 512M
innodb_log_files_in_group = 2
innodb_flush_log_at_trx_commit = 1
```

3. **Включить slow query log с разумным порогом.**

```ini
slow_query_log = ON
long_query_time = 0.5
```

4. **Поставить адекватный `max_connections` и использовать пул соединений.**

```ini
max_connections = 300
```

5. **После перезапуска — наблюдать метрики.**
   - нагрузка CPU;
   - I/O;
   - количество temp tables, full table scan, медленных запросов.

### Простыми словами

`my.cnf` — это **панель настроек сервера MySQL**. Пара ручек (buffer pool, режим flush, размер логов, slow log) определяют, будет ли сервер:

- экономить диск, но тормозить;
- беречь данные, но чуть медленнее писать;
- или наоборот, работать максимально быстро, но с риском потерь при краше.

### Картинка в голове / Как запомнить

Представь MySQL как **сервер в стойке**, а `my.cnf` — как **ряд крутилок спереди**: память, диск, журналирование, подключение клиентов. Если покрутить сразу все крутилки на максимум, сервер заорёт и уйдёт в своп/крешь. Настраивать нужно **по симптомам**.

### Проверь себя

1. **Чем отличается `innodb_flush_log_at_trx_commit=1` от `2` и когда можно использовать `2`?**
   <details><summary>Ответ</summary>
   `1` — запись и физический flush лога при каждом коммите (максимальная надёжность).  
   `2` — запись при каждом коммите, но flush раз в секунду (чуть быстрее, но можно потерять до 1 секунды коммитов при креше ОС).  
   `2` можно использовать там, где **допускается потеря небольшой части последних транзакций** ради производительности.
   </details>

2. **Почему установка `max_connections` на очень большое значение может быть опасна?**
   <details><summary>Ответ</summary>
   Потому что каждое соединение потребляет память и ресурсы; при большом числе активных коннектов сервер может упереться в RAM/CPU и упасть.  
   Обычно используют **пул соединений**, а не бесконечное число клиентов.
   </details>

### Запомните

- Важно знать **не все параметры**, а **свои ключевые 10–20**, которые вы осознанно крутите.
- Любое изменение нужно **сопровождать измерениями до/после**.
- Slow query log — один из лучших друзей при тюнинге MySQL.
