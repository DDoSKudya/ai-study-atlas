## 92. Оптимизация MySQL

### Цель раздела

Научиться **читать планы выполнения MySQL (`EXPLAIN`, `EXPLAIN FORMAT=JSON`)**, проектировать индексы (включая covering и prefix), использовать slow query log и инструменты анализа (`pt-query-digest`) для практической оптимизации запросов.

### В этом разделе главное (три строки)

1. `EXPLAIN` показывает **как именно MySQL планирует выполнять запрос**: какие таблицы, в каком порядке, какие индексы и оценки строк.
2. **Хорошие индексы (covering, продуманная селективность, разумный prefix)** — основа производительности, особенно при JOIN и сортировках.
3. Slow query log и инструменты анализа дают **список реальных «самых дорогих» запросов**, на которых и нужно фокусироваться.

### Термины

- **`EXPLAIN`** — команда для просмотра плана выполнения запроса.
- **`EXPLAIN FORMAT=JSON`** — расширенный JSON‑план с дополнительными полями (стоимость, фильтры, возможные индексы).
- **Covering Index** — индекс, содержащий все столбцы, используемые в запросе (в `SELECT`, `WHERE`, `ORDER BY`), позволяющий обойтись без чтения самой таблицы.
- **Prefix Index** — индекс по первым N символам строкового поля.
- **Slow Query Log** — лог медленных запросов.
- **`pt-query-digest`** — утилита из Percona Toolkit для агрегации и анализа запросов по slow log/binlog.

### Теория и правила

**1. Базовый `EXPLAIN`.**

```sql
EXPLAIN
SELECT o.id, o.total_amount, u.email
FROM orders o
JOIN users u ON u.id = o.user_id
WHERE o.created_at >= '2026-01-01'
  AND u.status = 'active'
ORDER BY o.created_at DESC
LIMIT 50;
```

Выдаёт таблицу с колонками `id`, `select_type`, `table`, `type`, `possible_keys`, `key`, `rows`, `Extra` и др.

- **`type`**:
  - `ALL` — полный скан таблицы (плохо на больших таблицах);
  - `range` — использование индекса по диапазону;
  - `ref`, `eq_ref` — хорошие типы join;
  - `const` — очень селективное условие.
- **`rows`** — оценка числа строк, которые придётся просмотреть.

**2. `EXPLAIN FORMAT=JSON`.**

```sql
EXPLAIN FORMAT=JSON
SELECT ...
```

Даёт JSON с полями:

- `query_block` / `nested_loop` / `table` — структура плана (вложенные блоки, join’ы, таблицы);
- `cost_info` — оценка стоимости;
- `rows_examined_per_scan`, `rows_produced_per_join` — сколько строк планируется читать/производить;
- `attached_condition` — реальные фильтры;
- `used_key_parts` — какие именно части индекса используются;
- `ordering_operation` / `grouping_operation` — где именно происходят сортировки и группировки.

На практике удобно:

- смотреть **какие именно таблицы и в каком порядке** участвуют (по `nested_loop` / `table`);
- отслеживать, какие фильтры реально применяются (`attached_condition`);
- понимать, **где тратятся строки** (`rows_examined_per_scan`).

**3. `EXPLAIN FORMAT=TREE` (MySQL 8.0+).**

```sql
EXPLAIN FORMAT=TREE
SELECT ...
```

Это «деревянное» представление плана, ближе по виду к explain’ам других СУБД.  
Пример (упрощённый, формат может отличаться):

```text
-> Limit: 50 row(s)
    -> Sort: o.created_at desc
        -> Nested loop inner join
            -> Filter: (o.created_at >= '2026-01-01')
                -> Index range scan on orders using idx_orders_user_created_at
            -> Filter: (u.status = 'active')
                -> Index lookup on users using PRIMARY (id=o.user_id)
```

Как это читать:

- каждая строка — **узел плана** (Limit, Sort, Nested loop, Filter, Index range scan, Index lookup и т.д.);
- отступы показывают **вложенность**: «что сверху» потребляет результаты того, что ниже;
- удобно визуально видеть:
  - где сортировка (`Sort`);
  - где фильтрация (`Filter`);
  - где какие индексы (`Index range scan`, `Index lookup`);
  - какие join’ы используются (`Nested loop`, `Hash join` — в будущем).

Практический подход:

- если только начинаешь — начинай с классического `EXPLAIN`‑таблички;
- когда чувствуешь себя увереннее, **сравнивай тот же запрос в `FORMAT=JSON` и `FORMAT=TREE`** — это помогает «срастить» внутреннюю модель плана;
- `FORMAT=TREE` часто удобнее, когда много JOIN’ов и подзапросов, потому что дерево сразу показывает, «что над чем».

**4. Индексы и паттерны.**

- **Covering index**:

```sql
CREATE INDEX idx_orders_user_created
  ON orders (user_id, created_at, total_amount);
```

Если запрос делает:

```sql
SELECT user_id, created_at, total_amount
FROM orders
WHERE user_id = ? AND created_at >= ?;
```

то MySQL может ответить **только по индексу**, без обращения к основной таблице.

- **Prefix index**:

```sql
CREATE INDEX idx_users_email_prefix
  ON users (email(20));
```

Используется, когда поле большое (`VARCHAR(255)`), но для селективности достаточно первых N символов. Экономит место, но может дать хуже селективность.

**4. Slow Query Log.**

- Включаем:

```ini
slow_query_log = ON
long_query_time = 0.5
log_queries_not_using_indexes = OFF   # обычно не включают в проде постоянно
```

- Анализируем:

```bash
pt-query-digest /var/log/mysql/mysql-slow.log > slow-report.txt
```

Отчёт покажет:

- какие запросы больше всего времени/ресурса съели;
- пример нормализованного текста запроса;
- распределение по времени.

#### Чуть подробнее про поля `EXPLAIN` (классический формат)

Когда смотришь `EXPLAIN`, обрати внимание на:

- **`table`** — с какой таблицей работает эта строка плана;
- **`type`** — тип доступа:
  - `ALL` — полный скан таблицы (обычно плохо на больших данных);
  - `index` — полный проход по индексу (лучше, чем `ALL`, но тоже дорогой);
  - `range` — диапазон по индексу (часто нормально);
  - `ref` — поиск по ссылке (FK, равенство по индексу);
  - `eq_ref` — по фактически уникальному совпадению (очень хорошо);
  - `const` / `system` — когда строка находится по PK/UNIQUE и таблицу можно считать константой.
- **`possible_keys`** — какие индексы теоретически можно было бы использовать;
- **`key`** — какой индекс реально выбран;
- **`rows`** — оценка, сколько строк придётся прочитать;
- **`Extra`** — дополнительные подсказки:
  - `Using where` — фильтрация идёт по условию;
  - `Using index` — запрос полностью обслуживается индексом (index‑only scan / covering index);
  - `Using temporary`, `Using filesort` — временные таблицы и доп. сортировка (зачастую сигналы к оптимизации).

Полезная привычка: **смотреть на `type`, `rows`, `key`, `Extra` вместе** и задаваться вопросом:  
«Мог ли я спроектировать такой индекс, чтобы `type` стал лучше, `rows` — меньше, а `Using filesort`/`Using temporary` исчезли?».

#### Разбор на мини‑примерах: `Using filesort` и `Using temporary` (почему это не «ошибка», но часто сигнал)

MySQL иногда делает два «дорогих» действия:

- **`Using filesort`** — дополнительная сортировка результатов (не обязательно «файлы на диске», но отдельный алгоритм сортировки).
- **`Using temporary`** — создаёт временную таблицу (в памяти или на диске), чтобы посчитать `GROUP BY`, `DISTINCT` или сложную агрегацию.

Это не всегда плохо. Плохо, когда:

- таблица большая,
- сортировка/группировка идёт по миллионам строк,
- а запрос выполняется часто.

##### Пример 1: сортировка без индекса

```sql
EXPLAIN
SELECT *
FROM orders
WHERE user_id = 42
ORDER BY created_at DESC
LIMIT 50;
```

Если индекс только по `user_id`, то сервер найдёт все строки user_id=42, а потом будет сортировать их по `created_at` → часто `Using filesort`.

Решение: индекс под запрос (с учётом левого префикса):

```sql
CREATE INDEX idx_orders_user_created_at
  ON orders (user_id, created_at);
```

Теперь MySQL может выдать строки уже в нужном порядке, без отдельной сортировки.

##### Пример 2: GROUP BY, который вынуждает временную таблицу

```sql
EXPLAIN
SELECT user_id, COUNT(*) AS cnt
FROM orders
WHERE created_at >= '2026-01-01'
GROUP BY user_id;
```

Если нет индекса `(created_at, user_id)` или `(user_id, created_at)` (в зависимости от фильтра/группировки), MySQL может:

- отфильтровать по created_at,
- затем собирать группы через временную таблицу → `Using temporary`.

Иногда это нормально. Иногда помогает индекс:

```sql
CREATE INDEX idx_orders_created_user ON orders (created_at, user_id);
```

Но важно: индекс — это не магия. Иногда `Using temporary` останется, и это ок, если данных мало или запрос редкий.

#### Индексы «по-человечески»: как новичку не промахнуться

Вот простой чек‑лист, который почти всегда работает:

1. Посмотри на запрос и выпиши:
   - **фильтры** из `WHERE` (особенно равенства `=` и диапазоны `>=`, `BETWEEN`);
   - **соединения** из `JOIN ON`;
   - **сортировку** из `ORDER BY`;
   - **группировку** из `GROUP BY`.
2. Если запрос частый и таблица большая — почти всегда нужен индекс, который начинается с:
   - столбцов из `WHERE` с `=` (самые селективные обычно ставят раньше);
   - затем столбцы для диапазона/сортировки.
3. Если запрос постоянно выбирает 2–4 столбца — подумай о **covering index**, чтобы не ходить в таблицу.
4. Не плодить индексы «на всё подряд»: каждый индекс
   - замедляет `INSERT/UPDATE/DELETE`;
   - занимает место;
   - усложняет оптимизатору выбор.

И очень практическая мысль: **сначала оптимизируй 5–10 самых дорогих запросов**, а не «всю систему сразу».

#### Оптимизационные хинты: `STRAIGHT_JOIN`, `USE INDEX`, `FORCE INDEX` (когда и как применять с умом)

Иногда MySQL‑оптимизатор выбирает **не самый удачный план**:

- в сложном запросе с несколькими JOIN он может начать «не с той таблицы»;
- может игнорировать полезный индекс по статистике, которая устарела или не отражает реальную селективность.

На такие случаи есть **хинты** — подсказки планировщику. Важно: это **не замена нормальным индексам и нормальному SQL**, а «последний уровень тонкой настройки».

##### `STRAIGHT_JOIN`: зафиксировать порядок JOIN

Без `STRAIGHT_JOIN` MySQL сам может менять порядок таблиц в `FROM`/`JOIN`, если считает какой‑то порядок выгоднее.

```sql
SELECT *
FROM big_table b
JOIN small_table s ON s.id = b.small_id
WHERE s.status = 'active';
```

Иногда выгоднее начать с `small_table`, но оптимизатор решает иначе.  
Хинт:

```sql
SELECT STRAIGHT_JOIN *
FROM big_table b
JOIN small_table s ON s.id = b.small_id
WHERE s.status = 'active';
```

говорит: «выполняй JOIN **в том порядке, как написано в запросе**».  
Использовать это стоит, только если ты **точно замерил**, что такой порядок реально лучше.

##### `USE INDEX` / `FORCE INDEX` / `IGNORE INDEX`

Эти хинты управляют **выбором конкретного индекса**.

Пример:

```sql
SELECT *
FROM users
USE INDEX (idx_users_email)
WHERE email = 'a@b.com';
```

- `USE INDEX (idx)` — «предпочитай этот индекс, но можешь выбрать другой, если он явно лучше».
- `FORCE INDEX (idx)` — «считай, что других индексов как будто нет; используй этот».
- `IGNORE INDEX (idx)` — «делай вид, что этого индекса не существует».

Опасности:

- если данные и распределения меняются, план, который сегодня хороший с `FORCE INDEX`, через год может стать **хуже, чем без хинта**;
- хинты лежат в коде, и о них легко забыть при изменении схемы/индексов.

Практический подход:

1. Сначала **чинить схему и индексы**, а не бросаться к хинтам.
2. Если оптимизатор упорно выбирает «не то», и ты это доказал экспериментами:
   - документируй хинт в коде (комментарием «почему он нужен»);
   - периодически **перепроверяй** такие места при росте данных и обновлении MySQL.

### Пошагово: типичный цикл оптимизации запроса

1. **Найти проблемный запрос.**
   - из slow query log + `pt-query-digest`;
   - или из мониторинга (PMM, графики latency).

2. **Посмотреть план через `EXPLAIN` и `EXPLAIN FORMAT=JSON`.**
   - понять, как идут JOIN’ы;
   - какие индексы используются;
   - где полный скан.

3. **Сопоставить план с индексами и условиями.**
   - добавить/изменить индекс (или переписать запрос) так, чтобы:
     - селективные условия были в начале составного индекса;
     - ORDER BY/ GROUP BY могли использовать индекс.

4. **Пере‑запустить `EXPLAIN` и замерить время.**
   - убедиться, что тип доступа и количество строк улучшились.

5. **Зафиксировать изменения и добавить проверку в мониторинг.**

### Простыми словами

Оптимизация MySQL — это **не поиск «магических параметров в my.cnf»**, а работа с реальными запросами:

- найти **что тормозит** (slow log);
- понять **как именно сервер это выполняет** (`EXPLAIN`);
- **дать ему правильные индексы и/или другой запрос**, который лучше вписывается в модель.

### Картинка в голове / Как запомнить

Представь, что MySQL — это курьерская служба, а `EXPLAIN` — это **карта маршрута доставок**.  
Плохой план — это когда курьер ездит кругами по всему городу (full scan). Хороший план — когда он ездит по оптимальному маршруту с заранее сгруппированными адресами (индексы, правильный порядок JOIN’ов).

### Проверь себя

1. **Что значит, если в `EXPLAIN` по большой таблице тип доступа `ALL` и `rows` очень велико?**
   <details><summary>Ответ</summary>
   Это значит, что MySQL планирует полный скан таблицы (full table scan), просматривая много строк.  
   На больших таблицах это почти всегда признак отсутствия подходящего индекса или слишком не‑селективного условия.
   </details>

2. **Чем полезен covering index по сравнению с «обычным»?**
   <details><summary>Ответ</summary>
   Covering index позволяет отвечать на запрос, **не обращаясь к основной таблице**, читая только индекс.  
   Это уменьшает число I/O операций и ускоряет запросы.
   </details>

### Запомните

- `EXPLAIN` — главный инструмент понимания поведения запросов в MySQL.
- Хорошие индексы проектируются **под типичные запросы**, а не «в общем случае».
- Slow query log + `pt-query-digest` дают **конкретный список приоритетов** для оптимизации.
