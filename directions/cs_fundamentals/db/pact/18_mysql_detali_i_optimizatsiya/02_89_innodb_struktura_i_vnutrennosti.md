## 89. InnoDB: структура и внутренности

### Цель раздела

Понять, как **устроен InnoDB изнутри**: tablespace, сегменты, экстенты и страницы, buffer pool, undo/redo, doublewrite buffer и как всё это влияет на производительность, надёжность и настройки.

### В этом разделе главное (три строки)

1. InnoDB хранит данные в **pages по 16 КБ**, сгруппированных в extent’ы и tablespace’ы.
2. **Buffer pool** — главный потребитель памяти InnoDB; **redo лог** и **undo пространства** обеспечивают crash recovery и MVCC.
3. Понимание doublewrite buffer, file‑per‑table и размеров логов помогает **объяснить поведение под нагрузкой и при сбоях**.

### Термины

- **Tablespace** — логическое (и файловое) пространство хранения таблиц и индексов InnoDB.
- **Extent** — группа из нескольких страниц (обычно 64 страницы по 16 КБ = 1 МБ).
- **Page** — минимальная единица чтения/записи InnoDB (обычно 16 КБ).
- **Buffer Pool** — кэш страниц InnoDB в памяти.
- **Redo Log** — журнал изменений страниц для восстановления после сбоя.
- **Undo Tablespace** — область для хранения старых версий строк (MVCC, откат транзакций).
- **Doublewrite Buffer** — область, куда страницы пишутся дважды для защиты от частичных записей.
- **`innodb_file_per_table`** — режим, при котором каждая таблица хранится в отдельном .ibd‑файле.

### Теория и правила

- Структура хранения:
  - **Tablespace** → содержит **segments** (таблицы, индексы) → содержит **extents** → содержит **pages (16 КБ)**.
  - В старых версиях был общий системный tablespace `ibdata1` для всего; с `innodb_file_per_table=ON` каждая таблица получает свой `.ibd`.

- **Buffer Pool**:
  - хранит кэш страниц данных и индексов;
  - размер настраивается через `innodb_buffer_pool_size` (часто 60–80% RAM на выделенном MySQL‑сервере);
  - страницы вытесняются по LRU‑подобному алгоритму, с отдельной «горячей/холодной» частью.

- **Redo Log**:
  - файлы `ib_logfile0`, `ib_logfile1` (и далее);
  - пишет **логические изменения страниц**, которые потом «накатываются» при recovery;
  - размер и поведение зависят от `innodb_log_file_size`, `innodb_log_files_in_group`, `innodb_flush_log_at_trx_commit`.

- **Undo**:
  - хранит предыдущие версии строк, чтобы:
    - откатывать транзакции;
    - давать читателям «снимки» (MVCC), не блокируя записи.

- **Doublewrite Buffer**:
  - защищает от ситуации, когда **страница 16 КБ пишется наполовину** при креше;
  - InnoDB сначала пишет страницу в специальную область doublewrite, затем в основное место;
  - при recovery сравнивает и при необходимости восстанавливает из doublewrite.

- Режим `innodb_file_per_table`:
  - **ON** (рекомендуется):
    - каждая таблица в отдельном `.ibd` файле, легче управлять размером, отдельные бэкапы, переносы, уменьшение bloat.
  - **OFF** (старый стиль):
    - всё в одном `ibdata1` — трудно уменьшить, растёт безвозвратно.

#### Как живёт одна страница: от диска до запроса

Попробуем «пройти путь» одной строки данных:

1. Ты делаешь запрос:
   ```sql
   SELECT * FROM orders WHERE id = 123;
   ```
2. MySQL (через InnoDB) по индексу понимает, **в какой странице** лежит нужная строка.
3. InnoDB смотрит в **buffer pool**:
   - если страница **уже там**, то просто читает её из памяти;
   - если нет — считывает **16 КБ с диска** (целую страницу) и кладёт её в buffer pool.
4. Если ты делаешь `UPDATE`:
   - InnoDB меняет копию страницы **в buffer pool**;
   - записывает в **redo лог**, что именно изменилось;
   - помечает страницу как **dirty** («грязная», то есть отличается от версии на диске).
5. Фоновые потоки InnoDB периодически:
   - берут dirty‑страницы из buffer pool;
   - пишут их в **doublewrite buffer**, затем в основной tablespace;
   - после успешной записи помечают страницу как чистую.

В результате большинство запросов работают **по памяти**, а диск подключается только при вытеснении/подгрузке страниц и при сбросе грязных страниц.

#### MVCC на пальцах: две параллельные транзакции

Пусть есть таблица `accounts(id, balance)` и строка `id=1, balance=100`.

1. Транзакция А:
   ```sql
   BEGIN;
   SELECT balance FROM accounts WHERE id = 1; -- видит 100
   ```
2. Транзакция B:
   ```sql
   BEGIN;
   UPDATE accounts SET balance = 150 WHERE id = 1;
   COMMIT;
   ```
3. Транзакция А **после того, как B закоммитила**:
   ```sql
   SELECT balance FROM accounts WHERE id = 1; -- при стандартном уровне REPEATABLE READ всё ещё видит 100
   COMMIT;
   ```

Как так? InnoDB хранит **старую версию строки** (balance=100) в **undo пространстве** и использует её для «снимка» транзакции А.  
Когда А началась, её снимок «запомнил», что строка выглядела как 100, и до конца транзакции он это не меняет.

Пока не исчезнут все транзакции, которым нужна старая версия, InnoDB **держит undo‑запись**.  
Фоновый процесс потом очищает ненужные undo‑версии (это и есть часть работы **purge**).

#### Уровни изоляции InnoDB простыми словами (READ COMMITTED vs REPEATABLE READ)

InnoDB по умолчанию работает на уровне **REPEATABLE READ**. Это звучит страшно, но идея простая:

- **READ COMMITTED** — «каждый запрос видит только уже закоммиченные изменения»:
  - первый `SELECT` видит одни данные;
  - если параллельная транзакция что‑то закоммитила, второй `SELECT` в той же транзакции уже увидит новые данные.
- **REPEATABLE READ** — «все `SELECT` внутри одной транзакции видят **один и тот же снимок**»:
  - первый `SELECT` «замораживает» картинку;
  - последующие `SELECT` по тем же строкам видят **ту же картинку**, даже если кто‑то параллельно закоммитил изменения.

Пример различия (упрощённо):

- Уровень **READ COMMITTED**:

  1. Транзакция А:
     ```sql
     BEGIN;
     SELECT balance FROM accounts WHERE id = 1; -- видит 100
     ```
  2. Транзакция B:
     ```sql
     BEGIN;
     UPDATE accounts SET balance = 150 WHERE id = 1;
     COMMIT;
     ```
  3. Транзакция А:
     ```sql
     SELECT balance FROM accounts WHERE id = 1; -- уже видит 150
     COMMIT;
     ```

- Уровень **REPEATABLE READ**:

  1. Транзакция А:
     ```sql
     BEGIN;
     SELECT balance FROM accounts WHERE id = 1; -- видит 100
     ```
  2. Транзакция B:
     ```sql
     BEGIN;
     UPDATE accounts SET balance = 150 WHERE id = 1;
     COMMIT;
     ```
  3. Транзакция А:
     ```sql
     SELECT balance FROM accounts WHERE id = 1; -- всё ещё 100 (тот же снимок)
     COMMIT;
     ```

Что важно помнить новичку:

- **READ COMMITTED**:
  - проще понять интуитивно («вижу только закоммиченное, но снимок может обновляться»);
  - снижает риск долгих «висящих» версий в undo (меньше bloat’а).
- **REPEATABLE READ (дефолт InnoDB)**:
  - удобен для типичных батч‑операций (ты видишь стабильную картину за время транзакции);
  - можно получить «фантомные» строки (другие транзакции добавляют новые строки, которых ты не видел изначально) — но InnoDB использует **next‑key locks** и часто защищает от них при `SELECT ... FOR UPDATE`.

Для начала достаточно запомнить:

- если ты не делаешь суперсложную логику, **дефолт REPEATABLE READ подходит**;
- если нужна совместимость с другой СУБД/ORM или более простой ментальный образ — можно рассматривать **READ COMMITTED**, но это отдельное архитектурное решение.

#### Самое важное про индексы InnoDB (очень простым языком)

Если запомнить **только одну вещь** про InnoDB‑индексы, то вот её формула:

> **В InnoDB таблица = B+tree по PRIMARY KEY (кластерный индекс).**  
> Все вторичные индексы (secondary indexes) хранят «ключ индекса → PRIMARY KEY строки».

Это объясняет кучу «странностей» поведения.

##### 1) Кластерный индекс (clustered index)

- В InnoDB **данные таблицы лежат внутри B+tree**, отсортированного по `PRIMARY KEY`.
- Поэтому:
  - `PRIMARY KEY` влияет не только на «уникальность», но и на **физическую организацию данных**;
  - вставки «в середину» ключевого пространства могут давать **page split** (разбиение страницы) и лишний I/O;
  - последовательный PK (`BIGINT AUTO_INCREMENT`) обычно даёт простые и быстрые вставки.

##### 2) Вторичный индекс (secondary index) и «допрыгивание» до таблицы

Когда ты делаешь запрос по вторичному индексу, происходит два шага:

1. MySQL ищет в secondary index нужные записи.
2. Из каждой записи secondary index берёт **PRIMARY KEY** и по нему идёт в clustered index за остальными столбцами.

Это называется «back to table lookup». Поэтому:

- если запросу нужны столбцы, которых нет в индексе, будет доп. чтение clustered index;
- если индекс покрывает запрос (covering index), back lookup не нужен — и запрос ускоряется.

Пример:

```sql
-- secondary index по (user_id, created_at)
CREATE INDEX idx_orders_user_created_at ON orders (user_id, created_at);

-- запрос просит ещё total_amount (нет в индексе)
SELECT user_id, created_at, total_amount
FROM orders
WHERE user_id = 42
ORDER BY created_at DESC
LIMIT 50;
```

В этом случае сервер:

- найдёт в индексе подходящие записи;
- а потом для каждой строки будет ходить в clustered index за `total_amount`.

А вот covering‑вариант:

```sql
CREATE INDEX idx_orders_user_created_cover
  ON orders (user_id, created_at, total_amount);
```

Теперь запрос можно обслужить индексом.

##### 3) «Левый префикс» составного индекса (правило, которое ломает мозг новичкам)

Если индекс `(a, b, c)`, то он помогает, когда условия в запросе начинаются **с a**:

- хорошо:
  - `WHERE a = ?`
  - `WHERE a = ? AND b = ?`
  - `WHERE a = ? AND b >= ?`
- часто не помогает (или помогает частично):
  - `WHERE b = ?` (без `a`)
  - `WHERE c = ?` (без `a` и `b`)

Запомнить можно так: **индекс — это сортировка**. Если ты сортировал сначала по `a`, а потом по `b`, то искать только по `b` без `a` сложно.

##### 4) Page split (почему «случайный PK» может быть болью)

Если `PRIMARY KEY` «скачет» (например, UUID v4), новые строки вставляются в случайные места B+tree.  
InnoDB вынужден:

- часто разрезать страницы (page split),
- перетасовывать данные,
- делать больше случайных I/O.

Это не «всегда плохо», но **под большой записью** может стать узким местом.

#### Блокировки InnoDB простыми словами: что именно «держится» во время транзакции

Здесь цель — убрать страх. Будет два правила и один пример.

**Правило 1.** InnoDB блокирует не «таблицу целиком», а **конкретные строки и/или диапазоны в индексе**.  
**Правило 2.** Важно не только «какая команда», но и **какой индекс используется**.

Пример:

```sql
BEGIN;
UPDATE accounts SET balance = balance - 10 WHERE id = 1;
-- держим блокировку строки PK=1, пока не сделаем COMMIT/ROLLBACK
COMMIT;
```

Пока транзакция не завершена, другая транзакция, пытающаяся обновить ту же строку, будет ждать.

А вот опасный случай:

```sql
BEGIN;
UPDATE accounts SET balance = balance - 10 WHERE email = 'a@b.com';
-- если email не индексирован, возможен большой скан и много блокировок
COMMIT;
```

Без индекса по `email` сервер может прочитать много строк и держать блокировки дольше.  
Отсюда практическая мораль: **индексы — это не только скорость SELECT, но и предсказуемые блокировки при UPDATE/DELETE.**

### Пошагово: что настроить и где смотреть

1. **Проверить режим file‑per‑table.**

```sql
SHOW VARIABLES LIKE 'innodb_file_per_table';
```

2. **Посмотреть размер buffer pool и redo логов.**

```sql
SHOW VARIABLES LIKE 'innodb_buffer_pool_size';
SHOW VARIABLES LIKE 'innodb_log_file_size';
SHOW VARIABLES LIKE 'innodb_log_files_in_group';
```

3. **Оценить использование buffer pool (через INFORMATION_SCHEMA и performance_schema).**

Например, в MySQL 8:

```sql
SELECT
  pool_id,
  pool_size,
  database_pages,
  free_pages,
  dirty_pages
FROM information_schema.innodb_buffer_pool_stats;
```

4. **Проверить состояние InnoDB.**

```sql
SHOW ENGINE INNODB STATUS\G
```

Там будет информация о количестве страниц, грязных страниц, очередях I/O и т.п.

### Простыми словами

InnoDB — это **машина, которая читает и пишет данные блоками по 16 КБ**.  
Эти блоки крутятся в большом «ведре» памяти (buffer pool). Все изменения сначала записываются в журналы (redo, undo), чтобы при креше можно было всё восстановить. Doublewrite — это «страховка» от того, что блок запишется только наполовину.

### Картинка в голове / Как запомнить

Представь большую библиотеку:

- **Страницы (pages)** — это реальные бумажные страницы книг.
- **Extents** — стопки страниц.
- **Buffer pool** — стол, на который библиотекарь выкладывает часто читаемые книги.
- **Redo лог** — блокнот, куда он записывает все изменения (что переписали/исправили).
- **Undo** — папка с копиями до изменений.
- **Doublewrite** — сначала черновик, потом чистовик.

### Проверь себя

1. **Зачем нужен doublewrite buffer, если у нас уже есть redo лог?**
   <details><summary>Ответ</summary>
   Redo лог знает, какие изменения нужно применить к странице, но если страница на диске «порезана» напополам (partial page write), то записывать туда изменения опасно.  
   Doublewrite гарантирует, что на диске всегда есть либо старая, либо полностью новая версия страницы, а не «половина‑половина».
   </details>

2. **Почему слишком маленький buffer pool — плохо, а слишком большой — тоже может быть проблемой?**
   <details><summary>Ответ</summary>
   Маленький buffer pool ведёт к частым чтениям с диска и падению производительности.  
   Слишком большой может конфликтовать с кэшем ОС и другими процессами, приводя к свопу и OOM. Нужно искать баланс относительно доступной RAM и нагрузки.
   </details>

### Запомните

- InnoDB думает **страницами 16 КБ** и крутит их в buffer pool.
- Undo/redo и doublewrite — это **три кита надёжности**: откат, восстановление, защита от частичных записей.
- `innodb_file_per_table=ON` и разумный `innodb_buffer_pool_size` — базис для здорового инстанса.
