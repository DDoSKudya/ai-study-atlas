## 88. Движки хранения MySQL

### Цель раздела

Разобраться, **что такое storage engine (движок хранения) в MySQL**, какие основные движки существуют (InnoDB, MyISAM, Memory, Aria, ColumnStore и др.), чем они отличаются и **почему в современных системах почти всегда выбирают InnoDB**.

### В этом разделе главное (три строки)

1. MySQL поддерживает **несколько движков хранения**, и каждая таблица может использовать свой движок.
2. **InnoDB — де‑факто стандарт**: транзакции, внешние ключи, row‑level locking, crash safety; MyISAM и др. — нишевые решения.
3. Выбор движка — это **компромисс между скоростью, надёжностью и функциональностью**; большинство проектов выигрывают от простого правила: «если не знаешь — бери InnoDB».

### Термины

- **Storage Engine (движок хранения)** — подсистема в MySQL, отвечающая за хранение таблиц/индексов и реализацию транзакций, блокировок, восстановления.
- **InnoDB** — транзакционный движок по умолчанию с поддержкой ACID, внешних ключей и блокировок на уровне строк.
- **MyISAM** — старый движок без транзакций и внешних ключей, с табличными блокировками; сейчас считается устаревшим (deprecated).
- **Memory (HEAP)** — in‑memory движок для временных/быстрых таблиц; данные теряются при перезапуске.
- **Aria** — движок MariaDB, позиционируемый как crash‑safe замена MyISAM.
- **ColumnStore (MariaDB)** — колоночный движок для аналитических нагрузок (OLAP).

### Теория и правила

- В MySQL **таблица = комбинация метаданных + реализация движком хранения**:
  - DDL (`CREATE TABLE ... ENGINE=InnoDB`) говорит MySQL, какой движок использовать;
  - дальше все детали (формат файлов, блокировки, транзакции) зависят от выбранного движка.

- Основные движки:
  - **InnoDB**:
    - транзакции, ACID, row‑level locking;
    - внешние ключи;
    - crash recovery (redo‑лог, doublewrite buffer);
    - хороший баланс скорости чтения/записи и надёжности.
  - **MyISAM**:
    - нет транзакций и внешних ключей;
    - tab‑level locking (табличные блокировки) — при записи блокируется вся таблица;
    - быстрый простейший read‑heavy сценарий в прошлом, но сейчас **не рекомендуется** для новых проектов.
  - **Memory**:
    - данные хранятся только в RAM;
    - очень быстрый доступ, но **потеря при перезапуске** сервера;
    - полезен для временных таблиц, кэшей, сессий (если можно жить с потерей).
  - **Aria, ColumnStore и др.**:
    - **Aria (MariaDB)** — crash‑safe, используется самим сервером для временных таблиц.
    - **ColumnStore (MariaDB)** — колоночное хранение для аналитики (подобие columnstore в других СУБД).

- Выбор движка:
  - для **90% OLTP‑систем** (бизнес‑приложения, веб‑проекты) — **InnoDB**;
  - MyISAM — только если есть **очень специфические требования** и полное понимание рисков;
  - Memory — для **временных** данных, которые можно пересоздать;
  - аналитика на MySQL — в реальной жизни лучше уходит в отдельные системы (ClickHouse, ColumnStore, аналитические БД).

#### Чуть глубже: что реально отличает движки

Чтобы не путаться, полезно держать в голове **три оси сравнения**:

1. **Транзакции и надёжность.**
   - InnoDB:
     - поддерживает **ACID**:
       - *Atomicity (атомарность)* — либо вся транзакция применена, либо ничего;
       - *Consistency (согласованность)* — данные не попадают в «полусломанное» состояние;
       - *Isolation (изолированность)* — параллельные транзакции не мешают друг другу;
       - *Durability (надёжность)* — после коммита данные не теряются при падении сервера (при правильных настройках flush).
     - есть **crash recovery**: после отключения света InnoDB сам дочитает журналы и откатит/докрутит транзакции.
   - MyISAM:
     - **нет транзакций**: каждый `INSERT/UPDATE/DELETE` живёт сам по себе;
     - при падении сервера таблица может остаться в полусломанном состоянии, нужна ручная проверка/починка (`REPAIR TABLE`).
   - Memory:
     - данные хранятся **только в оперативке** — при рестарте теряются все строки;
     - для временных вещей это нормально, для бизнес‑данных — катастрофа.

2. **Блокировки и параллелизм.**
   - InnoDB:
     - **row‑level locking** — блокируются только конкретные строки, с которыми работает транзакция;
     - десятки и сотни одновременных пользователей могут обновлять разные строки в одной таблице без сильных конфликтов.
   - MyISAM:
     - **табличные блокировки** — при записи блокируется вся таблица;
     - хорошо работает, пока почти нет записей; как только записей становится много — появляются очереди и «затыки».

3. **Функциональность.**
   - InnoDB:
     - **внешние ключи**, каскадное удаление/обновление;
     - `FOREIGN KEY` реально работают и помогают поддерживать целостность.
   - MyISAM:
     - внешние ключи **логически описать можно**, но движок их **не применяет** — целостность нужно поддерживать в приложении.
   - Memory:
     - ограничения и индексы есть, но всё в памяти; за пределами узкоспециализированных задач — рискованный выбор.

#### Мини‑сценарии: когда какой движок

- **Интернет‑магазин, заказы, пользователи** → InnoDB.  
  Транзакции, целостность, конкурентная запись — всё про него.

- **Короткоживущая аналитическая временная таблица** (промежуточные агрегаты для отчёта, живущие 5 минут) → Memory или InnoDB, но с явной пометкой, что потеря не критична.

- **Логи, которые потом выгружаются в отдельное хранилище** → скорее тоже InnoDB (проще сопровождать), а тяжёлую аналитику — в ClickHouse/ColumnStore.

Важно: **даже если где‑то MyISAM покажет +10–20% по скорости чтения**, цена потери транзакций и блокировок почти всегда этого не стоит.

#### Примеры DDL

```sql
CREATE TABLE orders (
  id           BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  user_id      BIGINT UNSIGNED NOT NULL,
  total_amount DECIMAL(10,2)   NOT NULL,
  created_at   DATETIME        NOT NULL,
  PRIMARY KEY (id),
  KEY idx_orders_user_id (user_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

```sql
CREATE TABLE tmp_session_data (
  session_id VARCHAR(64) NOT NULL,
  data       BLOB        NOT NULL,
  updated_at DATETIME    NOT NULL,
  PRIMARY KEY (session_id)
) ENGINE=Memory DEFAULT CHARSET=utf8mb4;
```

### Пошагово: как понять, какой движок используется и сменить его

1. **Посмотреть движок у конкретной таблицы.**

```sql
SHOW TABLE STATUS LIKE 'orders'\G
-- Поле Engine покажет, какой движок у таблицы
```

2. **Посмотреть список поддерживаемых движков.**

```sql
SHOW ENGINES;
```

3. **Создать таблицу с нужным движком.**

```sql
CREATE TABLE logs (
  id        BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  message   TEXT            NOT NULL,
  created_at DATETIME       NOT NULL,
  PRIMARY KEY (id)
) ENGINE=InnoDB;
```

4. **Изменить движок существующей таблицы (осторожно!).**

```sql
ALTER TABLE logs ENGINE=InnoDB;
```

Это операция с полным перезаписыванием таблицы; на проде её нужно делать осознанно (часто с помощью `pt-online-schema-change`).

### Простыми словами

Движок хранения — это как **разный тип файловой системы под одной и той же ОС**: интерфейс (SQL) один, но внутри всё работает по‑разному.  
InnoDB — это «современная журналируемая FS с транзакциями», MyISAM — «старый быстрый, но небезопасный формат», Memory — «таблица, живущая только в RAM».

### Картинка в голове / Как запомнить

Представь, что MySQL — это **розетка стандартного формата**, а storage engine — это **разные виды «переходников» к диску**.  
Ты включаешь туда свои таблицы, а то, как именно они пишутся на диск, зависит от выбранного переходника. В реальной жизни почти все переходят на **InnoDB‑переходник** и не возвращаются к старым.

### Проверь себя

1. **Почему в новых проектах MyISAM практически не используется, хотя может быть быстрее на чтение?**
   <details><summary>Ответ</summary>
   Потому что у MyISAM нет транзакций, внешних ключей и crash‑safe восстановления, а блокировки идут на уровне таблицы.  
   Для реальных приложений это означает риск потери/порчи данных и сильные блокировки при записи. InnoDB даёт более сбалансированный и безопасный набор свойств.
   </details>

2. **Для чего может пригодиться движок Memory?**
   <details><summary>Ответ</summary>
   Для временных таблиц и кэшей, где потеря данных при перезапуске не критична, а важна скорость доступа из RAM. Например, для промежуточных агрегатов или сессий, которые можно пересобрать.
   </details>

### Запомните

- MySQL поддерживает несколько движков, но **InnoDB — базовый стандарт**.
- MyISAM и прочие — это **исключения для специальных случаев**, а не выбор по умолчанию.
- При проектировании схемы всегда **осознанно указывай `ENGINE=InnoDB`**, чтобы не получить неожиданный MyISAM «из прошлого».
