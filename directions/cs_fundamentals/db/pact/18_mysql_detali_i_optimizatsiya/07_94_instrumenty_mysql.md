## 94. Инструменты MySQL

### Цель раздела

Познакомиться с основными **CLI и GUI‑инструментами** вокруг MySQL (`mysqldump`, `mysqlbinlog`, Percona Toolkit, MySQL Workbench, phpMyAdmin, PMM) и понять, **как вписать их в рабочий процесс** разработчика и администратора.

### В этом разделе главное (три строки)

1. `mysqldump` и `mysqlbinlog` — базовые инструменты для бэкапов и восстановления на уровне логики.
2. **Percona Toolkit** (особенно `pt-query-digest`, `pt-online-schema-change`) — стандарт де‑факто для анализа и безопасных изменений в MySQL.
3. MySQL Workbench, phpMyAdmin и Percona Monitoring and Management (PMM) закрывают потребности в GUI и мониторинге.

### Термины

- **`mysqldump`** — утилита логического бэкапа MySQL (SQL‑дамп).
- **`mysqlbinlog`** — утилита чтения и воспроизведения binlog.
- **Percona Toolkit** — набор CLI‑утилит для MySQL.
- **`pt-query-digest`** — анализатор slow log/binlog.
- **`pt-online-schema-change`** — инструмент для изменения схемы без длительных блокировок.
- **MySQL Workbench** — официальный GUI для администрирования и моделирования схем.
- **PMM (Percona Monitoring and Management)** — система мониторинга MySQL/PostgreSQL и др.

### Теория и правила

**1. `mysqldump`.**

- Используется для:
  - логического бэкапа баз/таблиц;
  - переноса схемы и данных между серверами;
  - начальной загрузки для репликации.

Пример:

```bash
mysqldump \
  --single-transaction \
  --routines --triggers \
  --databases mydb \
  > mydb-$(date +%F).sql
```

- `--single-transaction` — делает снимок на уровне транзакции (для InnoDB) без блокировок.
- `--master-data` — может записать в дамп информацию о позиции binlog для настройки репликации.

**2. `mysqlbinlog`.**

```bash
mysqlbinlog --start-datetime="2026-02-20 00:00:00" \
            --stop-datetime="2026-02-20 23:59:59" \
            binlog.000123 | mysql -u root -p
```

Используется для:

- point‑in‑time восстановления (применить binlog поверх дампа до нужного момента);
- анализа, какие операции выполнялись в конкретный период.

**3. Percona Toolkit.**

- `pt-query-digest`:

```bash
pt-query-digest /var/log/mysql/mysql-slow.log > slow-report.txt
```

- `pt-online-schema-change`:

```bash
pt-online-schema-change \
  --alter "ADD COLUMN extra_data JSON NULL" \
  D=mydb,t=orders \
  --execute
```

Выполняет «теневой» `ALTER TABLE` с минимизацией блокировок.

**4. GUI и мониторинг.**

- **MySQL Workbench** — удобен для:
  - визуального просмотра/редактирования схемы;
  - выполнения запросов;
  - простых задач администрирования.
- **phpMyAdmin / Adminer** — лёгкие веб‑GUI; хороши для простых админ‑задач.
- **PMM (Percona Monitoring and Management)**:
  - собирает метрики MySQL (планы, slow queries, статистику InnoDB);
  - даёт дашборды для анализа производительности и тюнинга.

#### Когда и что именно использовать в реальной жизни

- **`mysqldump`**:
  - хорошо:
    - для небольших и средних баз (десятки гигабайт — уже на границе);
    - для начальной загрузки данных на новый стенд/в тест;
    - для частичного бэкапа (отдельных схем/таблиц).
  - плохо:
    - как единственный способ резервирования очень большой боевой базы;
    - если нужны очень быстрые восстановление и минимальный downtime.

- **`mysqlbinlog`**:
  - незаменим, когда:
    - нужно «домотать» состояние между дампом и текущим моментом;
    - нужно откатиться «на 10 минут назад», воспроизведя binlog только до нужного времени;
    - нужно понять, «что именно делали» в определённое окно времени.

- **Percona Toolkit**:
  - `pt-query-digest` — регулярный инструмент:
    - раз в день/неделю прогонять по slow log и смотреть, какие запросы стали хуже;
  - `pt-online-schema-change`:
    - когда нужно **менять схему большой таблицы** в проде;
    - когда простой на время `ALTER TABLE` неприемлем.

- **Workbench / phpMyAdmin / PMM**:
  - Workbench:
    - удобно для локальной разработки, моделирования схем, быстрой инспекции данных;
  - phpMyAdmin/Adminer:
    - быстро подключиться к базе на хостинге без SSH;
  - PMM:
    - «постоянные глаза» на прод: дашборды по нагрузке, медленным запросам, состоянию InnoDB/репликации.

### Пошагово: простой пайплайн бэкапов и анализа

1. **Ежедневный логический бэкап.**

```bash
mysqldump --single-transaction --routines --triggers mydb \
  | gzip > /backups/mydb-$(date +%F).sql.gz
```

2. **Включён slow query log и периодический анализ.**

```bash
pt-query-digest /var/log/mysql/mysql-slow.log \
  > /reports/mysql-slow-$(date +%F).txt
```

3. **Мониторинг через PMM.**
   - нужно развернуть сервер PMM и агента на MySQL‑хосте;
   - подключить MySQL к PMM и использовать дашборды для анализа нагрузки.

#### Практический сценарий: как сделать point‑in‑time recovery (PITR) «по-человечески»

Идея PITR: уметь не просто «восстановить базу на вчера», а **на конкретный момент времени** — например, «за минуту до того, как баговый скрипт всё испортил».

Грубо упрощая, нужен такой набор:

1. **Регулярные полные бэкапы** (например, раз в ночь) — через `mysqldump` или физический бэкап.
2. **Непрерывный binlog** между бэкапами.

Пусть:

- в 02:00 ты делаешь ночной дамп:

```bash
mysqldump --single-transaction --routines --triggers mydb \
  | gzip > /backups/mydb-2026-02-23.sql.gz
```

- в 10:37 кто‑то запускает баговый скрипт и портит данные;
- ты хочешь вернуться в состояние на 10:36:30.

Шаги PITR (упрощённо):

1. **Поднять отдельный экземпляр MySQL для восстановления** (чтобы не портить текущий прод).
2. **Восстановить дамп**:

```bash
gunzip < /backups/mydb-2026-02-23.sql.gz | mysql -u root -p mydb
```

3. **Найти нужный интервал в binlog** (от конца дампа до 10:36:30):

```bash
mysqlbinlog --start-datetime="2026-02-23 02:00:00" \
            --stop-datetime="2026-02-23 10:36:30" \
            /var/lib/mysql/binlog.00* \
  | mysql -u root -p mydb
```

4. **Проверить состояние восстановленной базы** (данные на момент перед багом).
5. Уже дальше решать:
   - заменить боевую базу этой копией;
   - или частично вытянуть нужные данные (через `INSERT ... SELECT`, `mysqldump` отдельных таблиц и т.п.).

Ключевая идея: **дамп даёт базовую точку, binlog — «дорожку вперёд во времени»**.  
PITR — это «перемотать по этой дорожке ровно до нужной секунды и остановиться».

#### Мини‑чек‑лист по эксплуатации MySQL (на основе инструментов)

Чтобы закрепить, можно свести всё к короткому «боевому» списку:

- **Бэкапы:**
  - делай **ежедневный логический бэкап** (`mysqldump` или физический эквивалент);
  - регулярно **тестируй восстановление** на отдельном инстансе;
  - храни бэкапы в отдельном хранилище (не только на том же сервере).

- **Binlog и PITR:**
  - включи binlog (`log_bin=ON`, `binlog_format=ROW`);
  - храни binlog минимум столько, сколько нужно для твоего RPO;
  - знай, как запустить `mysqlbinlog` для восстановления до нужного момента.

- **Производительность:**
  - включи `slow_query_log` с разумным `long_query_time`;
  - хотя бы раз в неделю гоняй `pt-query-digest` и смотри ТОП‑медленных запросов;
  - держи под рукой PMM/аналог для графиков по CPU, I/O, lag реплик.

- **Схема и миграции:**
  - не делай `ALTER TABLE` на большой таблице «в лоб» на проде — используй `pt-online-schema-change` или аналогичный механизм;
  - документируй сложные миграции и тестируй их на стенде.

- **Репликация и DR:**
  - если есть реплики — мониторь lag и состояние (`SHOW REPLICA STATUS`);
  - имей план, как действовать при падении мастера (краткий сценарий failover).

### Простыми словами

Инструменты вокруг MySQL — это **набор отвёрток и сканеров**, без которых эксплуатация превращается в «ручное шаманство».  
`mysqldump` и `mysqlbinlog` отвечают за бэкапы и восстановление, Percona Toolkit — за анализ и безопасные изменения, Workbench/PMM — за удобный интерфейс и мониторинг.

### Картинка в голове / Как запомнить

Представь, что MySQL — это двигатель, а `mysqldump`, Percona Toolkit и PMM — это **набор инструментов в гараже**.  
Можно, конечно, пытаться чинить двигатель камнем и палкой, но нормальная работа требует нормальных инструментов.

### Проверь себя

1. **Почему для больших продакшн‑баз логический бэкап `mysqldump` может быть недостаточен как единственный способ резервирования?**
   <details><summary>Ответ</summary>
   Потому что логический бэкап:
   - долго создаётся и долго восстанавливается;
   - может быть очень большим по размеру;
   - не всегда удобен для восстановления отдельных таблиц или всего инстанса.  
   Часто его дополняют физическими бэкапами (xtrabackup) и репликацией.
   </details>

2. **Для чего использовать `pt-online-schema-change` вместо прямого `ALTER TABLE` на большой таблице?**
   <details><summary>Ответ</summary>
   Прямой `ALTER TABLE` на большой таблице может занять много времени и блокировать чтение/запись.  
   `pt-online-schema-change` создаёт новую таблицу с нужной схемой, постепенно копирует данные и переключает их с минимальным простоем.
   </details>

### Запомните

- Нормальная эксплуатация MySQL опирается на **CLI‑утилиты и внешние инструменты**, а не только на `mysql`‑консоль.
- Percona Toolkit — must‑have для продвинутой работы с MySQL.
- Мониторинг и регулярный анализ slow log — основа предсказуемой производительности.
