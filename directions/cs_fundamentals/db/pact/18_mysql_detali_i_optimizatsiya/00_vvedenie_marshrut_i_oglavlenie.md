## Часть XVIII. MySQL: детали и оптимизация

> Цель этой части — **понять, как устроен MySQL «под капотом» (InnoDB, binlog, репликация, ключевые настройки) и как осмысленно оптимизировать его под реальные нагрузки**, а не «крутить параметры наугад».

После этой части ты сможешь:

- **осознанно выбирать движок хранения** (InnoDB vs MyISAM vs Memory и др.) под конкретные задачи;
- **понимать внутреннее устройство InnoDB**: tablespace, страницы, buffer pool, undo/redo, MVCC, doublewrite buffer;
- **настраивать binlog и репликацию MySQL** (формат, GTID, semi-sync, group replication) и понимать, как это связано с надёжностью и производительностью;
- **подбирать ключевые настройки MySQL** в `my.cnf` (память, InnoDB, логирование, I/O) под свою нагрузку;
- **читать и интерпретировать `EXPLAIN` / `EXPLAIN FORMAT=JSON`**, строить и подбирать индексы (covering, prefix), понимать поведение оптимизатора;
- **проектировать и сопровождать репликацию MySQL** (master–slave, master–master, group replication), организовывать чтение с реплик и планировать failover;
- **использовать инструменты MySQL/Percona** (`mysqldump`, `mysqlbinlog`, Percona Toolkit, PMM, Workbench) в повседневной работе администратора/разработчика.

Материал опирается на:

- **Часть II–IV** (основы SQL, транзакции, индексы),
- **Часть V–VII** (хранение, планы запросов, оптимизация),
- **Часть XI–XII** (операции, безопасность),
- **Часть XVI** (аналитика, хранилища, CDC),
- а также общие принципы, разобранные на PostgreSQL, которые мы теперь перенесём на MySQL и увидим отличия.

В этой части мы смотрим на MySQL **как на реальную боевую систему**: какие решения за ним стоят (движки, формат журналов, репликация), какие настройки действительно важны, как читать планы и что считать «нормальным поведением», а что — симптомами проблем.

### Краткая шпаргалка по части

| Понятие / конструкция | Суть (что это и когда нужно) |
|-----------------------|------------------------------|
| **Storage Engine (движок хранения)** | Подсистема, которая отвечает за физическое хранение таблицы и индексов (InnoDB, MyISAM, Memory и др.). |
| **InnoDB** | Транзакционный движок по умолчанию (ACID, row-level locking, внешние ключи). Практически всегда правильный выбор. |
| **Buffer Pool** | Основной кэш страниц InnoDB в памяти; параметр `innodb_buffer_pool_size` определяет, сколько данных реально живёт в RAM. |
| **Undo Tablespace** | Хранит старые версии строк для отката транзакций и MVCC (снимки для чтения). |
| **Redo Log** | Журнал изменений InnoDB (`ib_logfile*`), из которого происходит восстановление после краша; зависит от `innodb_log_file_size`, `innodb_flush_log_at_trx_commit`. |
| **Doublewrite Buffer** | Механизм защиты от частичной записи страницы; InnoDB сперва пишет страницу в doublewrite‑область, затем в основной tablespace. |
| **Binlog (Binary Log)** | Журнал логических изменений MySQL (statement/row/mixed), основа для репликации и CDC. |
| **GTID** | Global Transaction ID — уникальный идентификатор транзакции, упрощающий конфигурацию и управление репликацией. |
| **Slow Query Log** | Лог медленных запросов (`slow_query_log`, `long_query_time`), основной источник кандидатов на оптимизацию. |
| **EXPLAIN / EXPLAIN FORMAT=JSON** | Инструмент просмотра плана выполнения запроса в MySQL; JSON‑формат даёт больше деталей (стоимости, фильтры, индексы). |
| **Covering Index** | Индекс, содержащий все необходимые столбцы запроса, позволяющий читать только индекс без обращения к таблице. |
| **Prefix Index** | Индекс по первым N символам строки для экономии места; компромисс между точностью и размером. |
| **Master–Slave / Master–Master** | Классические схемы репликации MySQL: один master и несколько read‑replica, либо двунаправленная репликация (с риском конфликтов). |
| **Group Replication** | Встроенная в MySQL multi‑master группа на основе консенсуса, с автоматическим failover и защитой от split‑brain. |
| **Percona Toolkit** | Набор утилит (pt-query-digest, pt-online-schema-change и др.) для анализа и безболезненных изменений в MySQL. |

Используй шпаргалку как **карту понятий**: увидел термин — сверился, чтобы понимать, зачем он тебе.

### Маршрут изучения в этой части

1. **§88. Движки хранения MySQL**  
   Что такое storage engine, чем InnoDB отличается от MyISAM и других движков, как выбрать правильный движок в реальных проектах.

2. **§89. InnoDB: структура и внутренности**  
   Как устроены tablespace, страницы, buffer pool, redo/undo, doublewrite; как это связано с производительностью и надёжностью.

3. **§90. Binlog и репликация MySQL**  
   Что такое binlog, какие бывают форматы (statement/row/mixed), как binlog используется для репликации и восстановления.

4. **§91. Настройки MySQL**  
   Конфигурационный файл `my.cnf`, ключевые параметры памяти, InnoDB, логирования и I/O, подход к их настройке.

5. **§92. Оптимизация MySQL**  
   Чтение планов `EXPLAIN`, индексы (covering, prefix), оптимизация JOIN’ов, slow query log и pt-query-digest.

6. **§93. Репликация MySQL (детали)**  
   Разные топологии репликации (master–slave, master–master, group replication), GTID, чтение с реплик, мониторинг лагов и failover.

7. **§94. Инструменты MySQL**  
   Повседневные утилиты (`mysqldump`, `mysqlbinlog`, Percona Toolkit, Workbench, PMM) и как они вписываются в рабочий процесс.
