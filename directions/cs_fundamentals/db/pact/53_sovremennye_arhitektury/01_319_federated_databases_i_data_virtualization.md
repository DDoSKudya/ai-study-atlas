## 319. Federated databases и data virtualization

### Цель раздела

Понять, **как устроены федеративные запросы и виртуализация данных**, в чём их сила и слабость, какие реальные продукты это поддерживают (PostgreSQL, MySQL, Oracle, SQL Server, внешние движки типа Trino/Presto), и когда **федерация — хорошее решение, а когда это “удобный, но опасный шорткат”**.

### В этом разделе главное

- Federated queries дают **удобный единый SQL‑слой**, но скрывают задержки, ошибки и несовместимость типов за красивым интерфейсом.
- Data virtualization — это **прослойка**, которая притворяется единой БД; чем “умнее” она, тем важнее понимать, **что именно она делает с запросами** (pushdown, кеширование, агрегации).
- Федерация полезна для **ад‑hoc анализа, миграций, постепенного перехода**; для тяжёлой и регулярной аналитики чаще лучше **выделенное хранилище (DWH)**.

### Термины

- **Federated query** — запрос, который объединяет данные из **нескольких источников**, часто разных СУБД или БД в разных регионах.
- **Foreign table / external table** — объект в СУБД, который представляет данных во внешнем источнике (другая БД, файл, объектное хранилище).
- **FDW (Foreign Data Wrapper)** в PostgreSQL — расширение, которое позволяет подключать внешние источники как таблицы (`postgres_fdw`, `mysql_fdw`, `file_fdw` и др.).
- **Pushdown** — способность федеративного движка **“продавливать” фильтры и агрегации** в источник, чтобы не тянуть лишние данные.
- **Data virtualization** — логический слой, объединяющий множество источников в **единое представление** (виртуальные схемы, представления).

### Теория и правила

1. **Базовая идея федерации**
   - Вместо того чтобы писать код интеграции в приложении (несколько подключений, join в памяти), мы даём СУБД/движку **знание о внешних источниках**.
   - Пользователь пишет:
     ```sql
     SELECT u.id, u.email, o.total_amount
     FROM users u
     JOIN orders_remote o ON o.user_id = u.id
     WHERE u.created_at >= now() - interval '30 days';
     ```
     где `orders_remote` — таблица, которая физически живёт в другой БД/регионе/облаке.

2. **Как это реализовано в разных СУБД**
   - PostgreSQL — через `postgres_fdw`, `mysql_fdw`, `oracle_fdw` и др.; планировщик умеет делать pushdown фильтров и join‑ов (частично).
   - MySQL — исторически через **FEDERATED storage engine** (довольно ограниченный), плюс внешние движки/коннекторы.
   - SQL Server — через **Linked Servers** и `OPENQUERY`/`OPENROWSET`.
   - Oracle — через **Database Links**.

3. **Data virtualization платформы**
   - Trino/Presto, Denodo, Dremio и др. — отдельные движки, которые подключаются к множеству источников (PostgreSQL, MySQL, S3, Hive, Elasticsearch) и дают **единый SQL‑интерфейс**.
   - Они обычно умеют **кэшировать и переписывать запросы**, делать pushdown, объединять и агрегировать результаты.

4. **Основные ограничения**
   - **Сетевая задержка**: каждый join к удалённой таблице — это сеть. Если запрос тянет много строк, это может “убить” производительность.
   - **Консистентность**: разные источники могут иметь **разные модели изоляции и задержки репликации**, нет гарантии, что данные согласованы.
   - **Совместимость типов**: маппинг типов (особенно дат, временных зон, JSON, массивов) может быть неточным.
   - **Сложность отладки**: трассировать, **где именно “тормозит” или ломается запрос**, сложнее — нужно смотреть и на федеративный слой, и на каждый источник (планы запросов, логи, сеть).

5. **Когда федерация хороша**
   - Быстрые **ad‑hoc запросы** для аналитика/инженера, который хочет что‑то проверить, не поднимая отдельное хранилище.
   - **Переезд/миграция**: временно смотрим “старую” и “новую” БД вместе.
   - Интеграция с “медленными” источниками, где **обновление раз в сутки** достаточно, но хочется гибкости запросов.

6. **Когда федерация опасна**
   - Регулярная тяжёлая аналитика по прод‑БД (особенно если это OLTP): federated‑запросы могут **создавать непредсказуемую нагрузку** и блокировки.
   - Критичные пути приложения: любая проблема в удалённой БД превращается в **проблему вашего сервиса**.
   - Попытки построить “универсальное DWH” только на федерации, без явной загрузки в аналитическое хранилище.

### Пошагово: как включать федерацию в PostgreSQL

Пример: есть основная БД `core_db` (PostgreSQL) и внешняя БД `billing_db` (PostgreSQL), хотим читать таблицу `payments` как обычную таблицу.

1. **Установить расширение** в `core_db`:
   ```sql
   CREATE EXTENSION IF NOT EXISTS postgres_fdw;
   ```
2. **Создать сервер** (описание внешнего подключения):
   ```sql
   CREATE SERVER billing_server
     FOREIGN DATA WRAPPER postgres_fdw
     OPTIONS (host 'billing-db.internal', port '5432', dbname 'billing');
   ```
3. **Создать пользователя‑маппинг**:
   ```sql
   CREATE USER MAPPING FOR app_user
     SERVER billing_server
     OPTIONS (user 'billing_reader', password '...');
   ```
4. **Определить внешнюю таблицу**:
   ```sql
   CREATE FOREIGN TABLE payments_remote (
     id          bigint,
     user_id     bigint,
     amount_cents bigint,
     status      text,
     created_at  timestamptz
   )
   SERVER billing_server
   OPTIONS (schema_name 'public', table_name 'payments');
   ```
5. **Использовать во вьюхе/запросе**:
   ```sql
   CREATE VIEW latest_successful_payments AS
   SELECT p.*
   FROM payments_remote p
   WHERE p.status = 'success'
     AND p.created_at >= now() - interval '7 days';
   ```

**Важный момент:** перед тем как дать это разработчикам/аналитикам, нужно:

- поставить лимиты (например, `statement_timeout`) для запросов к `payments_remote`;
- продумать, **какие запросы допустимы**, а какие нужно запретить (например, full‑scan по огромной удалённой таблице).

### Простыми словами

Федерация — это как **универсальный переходник**: вы говорите “у меня всё как одна БД”, а внутри этот переходник:

- открывает нужные соединения,
- тянет куски данных,
- пытается часть работы “спихнуть” на источники,
- а всё это рисует вам как один SQL.

Это удобно, как **удлинитель с тройником**: втыкаете одно устройство, а он распределяет нагрузку по розеткам. Но если в одной розетке пропало питание или она даёт мало тока, страдает всё.

### Картинка в голове

Представь, что перед тобой **большая карта метро**:

- Каждая линия — отдельная БД/система (PostgreSQL, MySQL, S3).
- Federated слой — это **карта с линиями и пересадками**, по которой ты планируешь маршрут (запрос).
- Тебе не нужно думать о том, в каком городе/стране каждая станция — тебя интересуют только переходы.

Но время поездки и пересадок **реальное**: если одна линия перегружена или закрыта, “красивый маршрут” перестаёт работать. Федерация не отменяет физику сети и нагрузки.

### Как запомнить

- **Federation = “делаем вид, что всё в одном месте”**, но физически нет.
- **Virtualization = “слой‑иллюзия”**, который рисует одну схему поверх многих.
- Чем больше “магии” во враппере, тем важнее **понимать его границы**: что он умеет pushdown‑ить, где кэширует, какие есть лимиты.

### Примеры

1. **PostgreSQL + MySQL для отчётов**
   - Prod‑БД PostgreSQL, но часть данных по платежам живёт в MySQL (старый биллинг).
   - Поднимаем `mysql_fdw`, создаём `payments_mysql`, объединяем с `users` в PostgreSQL.
   - Для быстрой аналитики это удобно, но для тяжёлой отчётности постепенно выносим данные в ClickHouse/BigQuery.

2. **SQL Server Linked Server**
   - BI‑команда пишет запросы, которые джойнят таблицы из `CRM_DB` и `ERP_DB` через Linked Server.
   - Проблема: “простые” запросы начинают делать full‑scan огромных таблиц в ERP. Приходится вводить явные ограничения и материализованные представления.

### Практика / реальные сценарии

- **Миграция с Oracle на PostgreSQL**: часть таблиц уже перенесли, часть ещё в Oracle. В PostgreSQL через FDW подключают “старые” таблицы для совместных отчётов и проверки целостности перед финальным cut‑over.
- **Обогащение данных для аналитики**: Trino подключает PostgreSQL (OLTP), S3 (сырые логи), ClickHouse (агрегаты) и даёт аналитикам один SQL‑интерфейс для сложных витрин.

### Типичные ошибки

- Давать федеративный доступ к прод‑БД **без лимитов и мониторинга**.
- Думать, что federated‑join в запросе эквивалентен join‑у в одной БД по производительности.
- Использовать федерацию как **единственный** способ интеграции (никаких ETL/CDC).

### Что будет, если…

- **…сделать federated‑join между двумя огромными таблицами по неиндексированному полю?**  
  Скорее всего, движок будет вынужден тянуть очень много строк по сети, делать join в памяти или на стороне одного из источников. Это приводит к долгим запросам, росту нагрузки и иногда — к падению узлов.

- **…считать федерацию “source of truth” и строить на ней критичный отчёт для финансов?**  
  Риск получить **неконсистентные данные** из‑за задержек репликации, временных ошибок источников или несовместимости типов. Для финансов (и вообще критичных отчётов) обычно лучше строить DWH с явной загрузкой и валидацией.

### Проверь себя

1. В чём принципиальная разница между `SELECT ... JOIN payments_remote` и ETL‑процессом, который раз в час заливает `payments` в локальную таблицу?

<details><summary>Ответ</summary>
В первом случае каждый запрос **напрямую зависит от удалённой БД**: если она медлит или недоступна, ваш запрос тоже.  
Во втором — вы работаете с **локальной копией**; ETL может упасть или задержаться, но уже загруженные данные доступны быстро и предсказуемо. Цена — задержка обновления (staleness).
</details>

2. Почему важно знать, какие операции федеративный движок умеет **pushdown‑ить** в источник?

<details><summary>Ответ</summary>
Потому что pushdown определяет, **где выполняется тяжёлая работа**: фильтры, агрегации, join‑ы.  
Если что‑то не pushdown‑ится, то:
- в сеть может полететь огромный объём сырых данных;
- часть вычислений будет делаться на “центральном” узле, который к этому не рассчитан.  
Это напрямую влияет на производительность и стоимость.
</details>

3. Почему для федерации между PostgreSQL и MySQL особенно внимание нужно уделить типам дат и времени?

<details><summary>Ответ</summary>
Потому что разные СУБД по‑разному хранят и трактуют типы `TIMESTAMP`, таймзоны, `DATETIME` без таймзоны, округления/диапазоны.  
Некорректный маппинг может дать **тихие ошибки**: смещения на час/день, некорректные фильтры по диапазону, странные результаты при агрегации по дате.
</details>

### Запомните

- Федерация — это мощный **инструмент для чтения**, но плохой кандидат на единственный способ интеграции.
- Всегда проверяйте: **что именно делает движок**, какие есть лимиты, где возможны узкие места.
- Для повторяющихся тяжёлых задач почти всегда лучше **выделить отдельное хранилище** и загрузку (ETL/CDC).

#### Дополнительные вопросы по разделу

1. Почему federated‑запросы особенно опасны, если источники данных живут в **разных регионах или облаках**?

<details><summary>Ответ</summary>
Потому что каждый federated‑join превращается в **сетевой обмен между регионами/облаками**:
- возрастает задержка выполнения (RTT между регионами),
- растёт стоимость трафика,
- риск частичных сбоев (один регион/облако “подвисло”) становится выше.  
В результате запрос, который выглядит “как обычный JOIN”, может внезапно оказаться и очень медленным, и очень дорогим.
</details>

2. В каких случаях data virtualization лучше строить **поверх аналитического хранилища (DWH)**, а не поверх прод‑БД?

<details><summary>Ответ</summary>
Когда:
- основная цель — **аналитика и отчётность**, а не оперативные транзакции;
- допустима некоторая задержка обновления (данные не обязаны быть “на секунду точными”);
- есть риск, что аналитические запросы будут **тяжёлыми и частыми**.  
В этом случае дешевле и безопаснее стянуть данные в DWH (через ETL/CDC) и делать виртуализацию уже поверх него, чем нагружать прод‑БД федеративными запросами.
</details>

3. Как понять, что federation используется “по назначению”, а не как “универсальный костыль”?

<details><summary>Ответ</summary>
Признаки “по назначению”:
- есть **чёткий список допустимых сценариев** (ad‑hoc анализ, миграция, редкие отчёты);
- есть **лимиты и мониторинг**: statement_timeout, ограничение прав, наблюдаемость по нагрузке и трафику;
- для регулярной тяжёлой отчётности предусмотрено **отдельное хранилище/витрины**.  
Если же federated‑таблицы начинают использовать “везде, где лень настроить ETL/CDC”, это сигнал, что federation стала “универсальным костылём”, а не осознанным инструментом.
</details>
