## 330. Логи и анализ логов БД

### Цель раздела

Научиться **настраивать и использовать логи БД** так, чтобы они:

- помогали диагностировать медленные запросы, ошибки и аномалии,
- были **структурированными и связаны с запросами приложений** через корреляционные ID,
- удобно собирались в централизованную систему (ELK, Loki, Cloud Logging и т.п.),
- не «убивали» производительность и диск.

#### Проверь себя (цель §330)

1. Почему «структурированные логи и связь с запросами приложений» указаны в цели раздела наравне с диагностикой медленных запросов и ошибок?

<details><summary>Ответ</summary>
Потому что **неструктурированные логи** и логи без корреляционных ID почти бесполезны при расследовании: нельзя быстро отфильтровать по полям, нельзя связать запрос к БД с конкретным HTTP‑запросом или трейсом. Структура и корреляция превращают «поток текста» в **пригодный для поиска и анализа сигнал**.
</details>

2. Что может «убить» производительность и диск при логировании БД и как этого избежать?

<details><summary>Ответ</summary>
Логирование **всех** запросов (general log, log_statement = 'all') создаёт огромный объём записей и нагрузку на I/O; хранение без ротации заполняет диск. Избегать: логировать только нужное (slow query выше порога, DDL, ошибки), настраивать ротацию и ретенцию, при необходимости выносить сбор в отдельный шipper и не писать на тот же диск, где активные данные БД.
</details>

### В этом разделе главное

- Логи — это **летопись того, что делает БД**: без них любая диагностика превращается в гадание.
- Структурированные логи (JSON) и корреляционные ID — ключ к тому, чтобы **свести воедино** логи приложений и БД.
- Нельзя «включить всё по максимуму и навсегда»: логирование требует **баланса между детализацией и стоимостью**.

### Термины

- **Query log / general log** — лог всех запросов (или выборки запросов), приходящих к БД.
- **Slow query log** — лог только медленных запросов (дольше порога).
- **Error log** — лог ошибок (внутренних ошибок БД, фатальных сбоев, предупреждений).
- **Audit log** — журнал действий пользователей/ролей (кто что делал).
- **Structured log** — лог в структурированном формате (JSON, key=value), а не просто текст.
- **Correlation ID / Request ID / Trace ID** — идентификатор запроса/трейса для связывания логов.
- **Log aggregation** — сбор логов в централизованное хранилище.
- **Retention** — срок хранения логов.

#### Проверь себя (термины §330)

1. Чем **slow query log** принципиально отличается от **error log** и когда нужен каждый?

<details><summary>Ответ</summary>
**Slow query log** фиксирует запросы, которые выполнялись **дольше заданного порога** (по времени); он нужен для поиска узких мест по производительности. **Error log** фиксирует **ошибки** (внутренние сбои БД, фатальные ошибки, предупреждения). Для полноты картины нужны оба: по slow query ищем медленные запросы, по error log — сбои и причины падений.
</details>

2. Зачем вводить отдельный термин **log aggregation** и что он даёт по сравнению с «просто смотреть логи на сервере БД»?

<details><summary>Ответ</summary>
**Log aggregation** — это сбор логов с **множества хостов/сервисов** в одно хранилище с поиском и индексацией. На одном сервере можно смотреть только его логи; при падении ноды логи могут быть потеряны. Централизованный сбор даёт единую точку поиска, сохранность логов при сбоях узла и возможность связывать события между приложением и несколькими БД/репликами.
</details>

### Теория и правила

#### 1. Типы логов в разных СУБД

- **PostgreSQL**
  - `postgresql.log` — общий лог (подключения, ошибки, запросы, чекпоинты и т.д.).
  - `log_min_duration_statement` — порог для логирования медленных запросов.
  - Можно включать `log_statement = 'all'`, но это дорого; обычно ограничиваются `ddl` или `mod`.
  - Есть расширения (`pgaudit`) для аудита.
- **MySQL**
  - **General log** — все запросы (опять же, дорого).
  - **Slow query log** — запросы дольше порога.
  - Отдельный error log.
- **SQLite**
  - Встроенного «сервера» нет, логи зависят от приложения.
- **Oracle/SQL Server**
  - Собственные журналы, event log’и, audit.
- **NoSQL**
  - MongoDB: профайлер запросов, логи операций, oplog (репликация).
  - Redis: логи команд, ошибок, репликации.

#### 2. Структурированные логи и корреляция

Если БД умеет писать JSON‑логи (PostgreSQL с `log_line_prefix` в формате key=value, многие облачные managed‑БД), это сильно упрощает:

- поиск по полям (duration, db, user, application_name),
- построение дашбордов по логам (гистограммы latency, число ошибок),
- связывание логов БД и приложений по Trace ID.

Важная идея: **приложение должно передавать в БД свой идентификатор запроса**.  
Например:

- В PostgreSQL: устанавливать `application_name` или передавать `SET LOCAL my.trace_id = '...'`.
- В MySQL: часто делают это на уровне драйвера/connection attributes.

#### Проверь себя (теория §330: типы логов и корреляция)

1. Почему в проде для PostgreSQL не рекомендуют держать `log_statement = 'all'`, а для MySQL — general log выключенным?

<details><summary>Ответ</summary>
Потому что логирование **каждого** запроса создаёт огромный объём записей и нагрузку на диск и I/O; в проде это часто приводит к переполнению диска и деградации производительности. Обычно включают только slow query log (запросы выше порога), DDL и ошибки; general log включают лишь кратковременно при отладке.
</details>

2. Что даёт передача Trace ID из приложения в БД (через application_name, SET LOCAL или connection attributes)?

<details><summary>Ответ</summary>
Это позволяет **связать строку в логе БД** с конкретным запросом приложения и с трейсом в APM. По одному Trace ID можно найти: какой HTTP‑запрос пришёл, какие запросы к БД выполнялись в его рамках, сколько они заняли и какие ошибки вернули. Без этого логи БД и приложения — два разрозненных потока, и восстановить цепочку при инциденте очень сложно.
</details>

### Пошагово: как навести порядок в логах БД

1. **Определи цели логирования**
   - Что хочешь видеть: медленные запросы, ошибки, попытки опасных операций, изменения схемы?
2. **Включи минимально нужные логи**
   - Для PostgreSQL:
     - `log_min_duration_statement` =, например, `100ms` (или выше, чтобы не утонуть сразу).
     - `log_statement = 'ddl'` (логировать DDL).
   - Для MySQL:
     - включить slow query log с порогом;
     - оставить general log выключенным в проде (или включать кратковременно).
3. **Сделай формат структурированным**
   - PostgreSQL: `log_line_prefix` с ключами `time`, `db`, `user`, `application_name`, `pid`, `vxid`, `txid`, `client`, `session_id`, `trace_id` (если передаёшь).
   - Managed‑БД в облаках: включи JSON‑формат для логов.
4. **Передавай корреляционный ID из приложений**
   - В APM/трейсе есть `Trace ID` → прошивай его в connection/session, чтобы он попадал в логи БД.
5. **Настрой ротацию и ретенцию**
   - Ограничения по размеру файлов и времени хранения.
   - Сбор логов в централизованный стек (ELK, Loki, Cloud Logging, CloudWatch).
6. **Проверь сценарии поиска**
   - «Найти все медленные запросы к таблице X за последний час».
   - «Найти все ошибки constraint violation по приложению Y».
   - «Найти по Trace ID, какие запросы к БД были в рамках инцидента».

#### Проверь себя (пошаговый порядок в логах §330)

1. Почему в пошаговом плане сначала идёт «определи цели логирования», а не «включи slow query log»?

<details><summary>Ответ</summary>
Потому что от **целей** зависит, что именно включать и в каком формате. Если цель — только искать медленные запросы, достаточно slow query log с порогом; если нужен аудит DDL или связь с приложением по Trace ID, нужны дополнительные настройки и структура. Без целей легко включить лишнее (перегрузка) или не включить нужное (при инциденте данных не окажется).
</details>

2. Зачем на последнем шаге «проверь сценарии поиска» и что может пойти не так, если этот шаг пропустить?

<details><summary>Ответ</summary>
Чтобы убедиться, что **реально можно ответить** на типичные вопросы (медленные запросы по таблице, ошибки по приложению, поиск по Trace ID). Если не проверить, может оказаться, что парсер не распознаёт формат, нужных полей нет в логе или в агрегаторе не настроены поля для фильтра — и в момент инцидента поиск не сработает.
</details>

### Простыми словами

Логи БД — это как **чёрный ящик самолёта**:

- пока всё хорошо, никто на него не смотрит;
- когда что‑то пошло не так, всем **очень важно**, что там записано;
- записывать нужно **достаточно подробную, но не избыточную** информацию.

Если логи:

- неструктурированы,
- не содержат времени, идентификаторов запросов, контекста приложения,
- хранятся только локально на сервере БД,

то при инциденте ты оказываешься без «чёрного ящика» — или он есть, но прочитать его почти невозможно.

### Картинка в голове

Представь, что у тебя есть **чат‑лог** команды во время инцидента:

- если там есть только сообщения «сломалось!», «чиню», без контекста — потом невозможно восстановить, **что именно происходило**;
- если каждое сообщение подписано временем, автором, ссылкой на тикет, результатом команды — можно восстановить **timeline** и понять, где ошиблись.

Логи БД — это такой же чат, только пишет его сама БД. От того, **какие поля ты включил** и как их структурировал, зависит, сможешь ли ты потом «прочитать историю».

### Как запомнить

- **Лог без контекста почти бесполезен.**
- **Структура важнее объёма.**
- **Сначала задумайся, какие вопросы ты хочешь уметь задавать к логам.**

### Примеры

#### PostgreSQL: пример `log_line_prefix`

```conf
log_line_prefix = 'time=%m db=%d user=%u app=%a pid=%p session=%c client=%h trace=%v '
log_min_duration_statement = 100ms
log_statement = 'ddl'
```

(здесь `trace=%v` условно; вместо него можно использовать собственный GUC‑параметр или `application_name`).

Пример строки лога (упрощённо):

```text
time=2026-03-12 10:15:32.123+00 db=orders user=app app=orders-api pid=12345 session=... client=10.0.0.1 trace=abcd1234 duration=35.4 ms statement: SELECT ...
```

#### MySQL: включение slow query log

```sql
SET GLOBAL slow_query_log = 'ON';
SET GLOBAL long_query_time = 0.1; -- 100ms
SET GLOBAL log_output = 'FILE';
```

#### ELK / Loki / Cloud Logging

- Настраиваем shipper (Filebeat/Fluent Bit/Vector) на чтение логов БД.
- Пишем парсер для `log_line_prefix` или JSON.
- Строим дашборды: количество ошибок, число slow queries, распределение latency.

#### Проверь себя (примеры и образы §330)

1. В примере `log_line_prefix` для PostgreSQL указаны поля time, db, user, app, pid, client, trace. Какие из них минимально необходимы для расследования «какой запрос от какого приложения и как долго выполнялся»?

<details><summary>Ответ</summary>
Минимум: **time** (когда), **db** (в какой базе), **user** или **app** (кто/какое приложение), **duration** (как долго; обычно выводится отдельно для slow query), **trace** или аналог (связь с запросом приложения). Без времени нельзя построить timeline; без app/user непонятно, кто вызвал; без duration не оценить тяжесть; без trace нельзя связать с трейсом и логами приложения.
</details>

2. Почему «лог без контекста почти бесполезен» и какой контекст имеется в виду?

<details><summary>Ответ</summary>
Имеется в виду контекст **запроса и окружения**: время, пользователь/приложение, база, длительность, идентификатор запроса (Trace ID), при необходимости client IP. Без этого строка лога — просто «кто‑то что‑то выполнил»; нельзя ни отфильтровать по приложению, ни связать с конкретным пользовательским запросом, ни построить хронологию по одному трейсу.
</details>

### Практика / реальные сценарии

- «Логировать всё» **коротким всплеском** при инциденте, а в обычном режиме держать адекватный порог slow query.
- Включать подробное логирование DDL, чтобы понимать, **когда в проде меняют схему**.
- Использовать централизованное логирование, чтобы при падении ноды **не потерять логи**.

### Типичные ошибки

- Включить `log_statement = 'all'` в проде навсегда → перегрузка диска и I/O.
- Писать логи в нестандартной текстовой форме без чётких разделителей → сложно парсить.
- Не передавать Trace ID / Request ID → невозможно связать логи БД и приложений.
- Не настраивать ротацию → диск неожиданно забивается логами.

#### Проверь себя (типичные ошибки §330)

1. Почему «писать логи в нестандартной текстовой форме без чётких разделителей» — ошибка с точки зрения последующего анализа?

<details><summary>Ответ</summary>
Потому что **парсить** такой лог сложно и ненадёжно: регулярные выражения ломаются при изменении формата, нельзя надёжно извлечь поля (duration, db, trace_id) для фильтрации и агрегации. Структурированный формат (JSON или key=value с разделителями) позволяет однозначно извлекать поля и строить запросы/дашборды без хрупкого парсинга.
</details>

### Что будет, если…

- **Если логи будут слишком подробными**, БД может тратить существенную часть времени на запись логов, а диск переполнится.
- **Если логи будут слишком скудными**, при инциденте ты не сможешь понять, какие запросы выполнялись и что их спровоцировало.
- **Если логов нет в централизованной системе**, половина расследований будет упираться в «у нас нет логов с того сервера / они уже ротацией удалены».

#### Проверь себя (что будет, если §330)

1. Почему «логи только локально на сервере БД» рискованны при инциденте?

<details><summary>Ответ</summary>
Потому что при падении ноды или при замене инстанса **локальные логи теряются**; даже при ротации старые файлы удаляются и до них нельзя добраться. В централизованной системе логи сохраняются независимо от судьбы конкретного сервера, их можно искать по времени, приложению, Trace ID по всем хостам сразу — что критично при расследовании распределённого инцидента.
</details>

### Проверь себя

1. Зачем нужны **slow query logs**, если у нас есть APM с разбором медленных запросов?

<details><summary>Ответ</summary>
APM видит запросы со стороны приложения. Slow query logs показывают **точно, что БД считала медленными, с её точки зрения**, включая запросы от утилит, админских скриптов, других сервисов. Кроме того, slow query logs присутствуют даже там, где APM не установлен или выключен, и иногда дают более низкоуровневую информацию.
</details>

2. Почему важно, чтобы логи были **структурированными**?

<details><summary>Ответ</summary>
Потому что структурированные логи позволяют **фильтровать и агрегировать** по полям (`duration`, `db`, `user`, `application`), строить дашборды, связывать события по Trace ID. В противном случае приходится парсить текст регулярками, что медленно, ненадёжно и часто ломается при изменении формата.
</details>

3. Какой минимум информации должен содержать лог запроса к БД, чтобы он был полезен при расследовании?

<details><summary>Ответ</summary>
Минимум: timestamp, база, пользователь/роль, приложение, client IP, duration, текст запроса (или его digest), код ошибки (если была), идентификатор запроса/Trace ID. Этого достаточно, чтобы понять, **кто** и **что** делал и **как долго** это длилось.
</details>

### Запомните

- Логи БД — основной источник правды при расследовании **конкретных запросов**.
- Лучше иметь **умеренный, но качественно структурированный лог**, чем «всё подряд» без структуры.
- Без корреляционных ID связать логи БД и приложений почти невозможно.
