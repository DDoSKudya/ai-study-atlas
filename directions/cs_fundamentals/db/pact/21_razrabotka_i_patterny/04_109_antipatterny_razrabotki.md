## 109. Антипаттерны разработки

**Связь с предыдущим.** В §106–§108 разобрали, как правильно организовывать доступ к данным и транзакции. Антипаттерны — это то, что получается, когда эти правила нарушают: много лишних запросов (N+1), отсутствие транзакций, лишние данные (SELECT *), долгие транзакции. Знание паттернов помогает сразу видеть, «что не так».

**Одной фразой:** N+1 запросы, избыточные запросы, игнорирование транзакций, отсутствие индексов, SELECT * и длинные транзакции — главные антипаттерны, которые убивают производительность и предсказуемость.

### Цель раздела

Научиться **распознавать и устранять типичные антипаттерны** при работе с БД: N+1, избыточные запросы, отсутствие транзакций там, где нужна атомарность, отсутствие индексов, выбор лишних столбцов (SELECT *) и длинные транзакции.

### В этом разделе главное (три строки)

1. **N+1** — один запрос за списком сущностей и N запросов в цикле за связанными данными. Решение: один запрос с JOIN или загрузка связей батчем (eager load, batch load).
2. **Игнорирование транзакций** — несколько изменений без общей транзакции; при сбое посередине получаем частичное состояние. Решение: явная граница транзакции на операцию.
3. **Отсутствие индексов**, **SELECT *** и **длинные транзакции** — приводят к полным сканам, лишним данным и блокировкам/bloat. Решение: индексы по условиям и ключам, явный список столбцов, короткие транзакции.

### Самая простая версия (почему антипаттерны “убивают” систему)

Антипаттерн — это «вроде работает», но:

- сначала нормально на 100 строках,
- потом плохо на 10 000,
- потом прод падает на 1 000 000.

У антипаттернов почти всегда один общий эффект: **ты делаешь больше работы, чем нужно** (больше запросов, больше данных, больше блокировок).

**Избыточные запросы (не только N+1):** Кроме N+1 к избыточным относят: запросы, которые можно объединить в один (несколько SELECT по одной таблице в одном use case — иногда один с IN или OR); запросы без кэша там, где данные редко меняются (справочники, настройки); повторный запрос тех же данных в рамках одного запроса (прочитали пользователя, потом снова запрашиваем его же). Идея: минимизировать число обращений к БД — только то, что нужно для операции.

#### Почему называется именно «N+1» (чтобы не путать)

**N+1** — это формула числа запросов: **1** запрос, чтобы получить список (например, 50 заказов), и потом **N** запросов — по одному на каждый элемент списка (50 раз «дай клиента по id»). Итого: **1 + N = 51** запрос. Отсюда и название: «один плюс эн». Если бы список был из 100 строк — было бы 1 + 100 = 101 запрос. Чем больше N, тем хуже латентность и нагрузка на БД. Исправление: один запрос с JOIN или один batch (WHERE id IN (...)) — тогда запросов 1 или 2, а не 1+N.

**Как N+1 выглядит в логах (узнать с первого взгляда):** Один и тот же запрос повторяется много раз с разными параметрами. Пример: одна строка `SELECT * FROM orders LIMIT 50`, а следом 50 строк вида `SELECT * FROM customers WHERE id = 1`, `... WHERE id = 2`, …, `... WHERE id = 50`. Или в логе «50 запросов к customers по id». Как только видишь «много одинаковых запросов, отличаются только id» — это почти наверняка N+1.

**Разбор по шагам: что именно происходит при N+1 (по шагам в коде).** Шаг 1: приложение выполняет один запрос — «дай все заказы» (например, 50 строк). Шаг 2: в коде цикл по этим 50 заказам. Шаг 3: на каждой итерации выполняется ещё один запрос — «дай клиента по id» для customer_id этого заказа. Итого: 1 запрос + 50 запросов = 51 запрос; каждый запрос — отдельный round-trip к БД (сеть, парсинг, выполнение). Шаг 4: при 100 заказах будет 1 + 100 = 101 запрос — время и нагрузка растут с N. Исправление: до цикла получить все нужные клиенты одним запросом (JOIN или WHERE id IN (...)) и в цикле только подставлять данные из памяти.

### Пошагово: как найти N+1 (простая диагностика)

Если ты боишься не понять — держи вот такой алгоритм (он прям рабочий):

1. **Открой логи запросов** или включи профилирование (в dev — логирование SQL, в prod — агрегированная статистика).
2. Найди типичную операцию (например, открыть страницу «Список заказов»).
3. Посмотри, сколько SQL‑запросов выполняется при одном действии:
   - если 1–5 — часто нормально;
   - если десятки/сотни — почти наверняка N+1.
4. Проверь, есть ли повторяющиеся запросы одинаковой формы, отличающиеся только параметром (например, `WHERE id = ?` в цикле).
5. Исправление:
   - **JOIN** (если надо «присоединить» связанные данные);
   - **batch load** (второй запрос `WHERE id IN (...)` по списку);
   - в ORM: eager loading (`prefetch_related`, `select_related`, `include`, `joinedload`).

### Пошагово: как понять, что “нет индекса”

1. Есть медленный запрос с WHERE/JOIN/ORDER BY.
2. Смотри план (EXPLAIN): видишь Seq Scan по огромной таблице → подозрение на отсутствие индекса.
3. Смотри, по каким полям фильтруешь/джойнишь/сортируешь.
4. Добавляешь индекс (часто B‑tree) по этим полям и проверяешь план снова.

**Важно:** индекс — это ускорение чтения ценой замедления записи и места на диске. Индексируем то, что реально нужно запросам.

#### Пошагово: как исправить N+1 (конкретно в коде)

Если нашёл «1 запрос + N в цикле» — делай так.

1. **Пойми, какие данные нужны.** Например: список заказов + для каждого заказа имя клиента. Значит, нужны заказы и клиенты по списку customer_id из этих заказов.
2. **Вариант A — один запрос с JOIN.** Замени цикл на один SELECT с JOIN: `SELECT orders.*, customers.name AS customer_name FROM orders JOIN customers ON customers.id = orders.customer_id WHERE ...`. В коде больше нет цикла с запросами.
3. **Вариант B — два запроса (batch).** Первый: получить заказы. Второй: получить всех клиентов по списку id: `SELECT * FROM customers WHERE id IN (?, ?, ...)` (список id из заказов). В памяти сопоставь заказы и клиентов по customer_id. Цикл с запросами исчез.
4. **В ORM** используй eager loading: Django — `select_related('customer')` или `prefetch_related('customer')`; Rails — `includes(:customer)`; Laravel — `with('customer')`. Смысл тот же: ORM сам сделает один JOIN или один batch вместо N запросов в цикле.
5. **Проверь:** снова открой ту же страницу/эндпоинт и посмотри число запросов в логах. Должно быть 1–2 вместо 1+N.

### Что будет, если оставить N+1 в проде (конкретные цифры)

Допустим, страница «Мои заказы» показывает 50 заказов и для каждого — имя клиента. Без исправления: 1 запрос за 50 заказов + 50 запросов за клиентов = **51 запрос** на один показ страницы. Каждый запрос — минимум один round-trip к БД (сеть + парсинг + выполнение). При латентности 2 мс к БД это уже **102 мс только на сеть**. Плюс нагрузка на БД: 51 запрос вместо 1–2. При 1000 пользователей, открывающих страницу в минуту, БД получает десятки тысяч лишних запросов. Страница начинает грузиться 1–3 секунды, БД упирается в CPU или соединения. Исправление одним JOIN или batch load снижает запросы до 1–2 и время до 2–5 мс — разница на порядки.

#### Что будет, если не оборачивать «перевод денег» в транзакцию

Операция: списать 100 руб. со счёта A, зачислить на счёт B. Два отдельных UPDATE без общей транзакции. Сбой произошёл после первого UPDATE и до второго. В БД: с A списали 100, на B не зачислили. Деньги «исчезли». Либо наоборот: первый UPDATE не прошёл (таймаут), второй прошёл — на B зачислили, с A не списали. Деньги «напечатались из воздуха». Итого: без одной транзакции на атомарную операцию возможна порча данных и финансовые потери. Это не теоретический случай — такие баги регулярно находят в проде.

### Термины

- **N+1 запросы** — паттерн, при котором сначала выполняется один запрос (например, список заказов), затем для каждой из N записей выполняется ещё один запрос (например, данные клиента). Итого 1 + N запросов; при большом N латентность и нагрузка на БД резко растут.
- **Антипаттерн** — типичный способ решения задачи, который на практике приводит к плохим последствиям (производительность, поддерживаемость, надёжность).

### Теория и правила

#### N+1

Пример: получить все заказы и для каждого заказа — имя клиента.

Плохо:

```python
orders = db.query("SELECT * FROM orders")
for o in orders:
    o.customer = db.query("SELECT name FROM customers WHERE id = %s", [o.customer_id])
```

Хорошо: один запрос с JOIN или два запроса — все заказы и все нужные клиенты по списку id, затем сопоставление в памяти.

```sql
SELECT o.*, c.name
FROM orders o
JOIN customers c ON c.id = o.customer_id
```

Или в ORM: использовать eager loading (например, `include` в ActiveRecord, `select_related`/`prefetch_related` в Django).

**Почему N+1 так вреден (числовой пример):** Допустим, 100 заказов и для каждого один запрос за клиента. Итого 1 + 100 = 101 запрос. Каждый запрос — минимум один round-trip к БД (сетевой обход + парсинг + выполнение). При латентности 1 мс к БД это уже 101 мс только на сеть; при 1000 заказов — более 1000 мс. Один запрос с JOIN или два батч-запроса дают 1–2 round-trip’а и время порядка 1–2 мс. Разница на порядки.

#### Игнорирование транзакций

Операция «списать со счёта A, зачислить на B» должна быть одной транзакцией. Если делать два отдельных UPDATE без общей транзакции, при сбое между ними получится потеря или дублирование денег. Решение: обернуть в одну транзакцию (транзакционный скрипт или граница на сервисе).

#### Отсутствие индексов

Запросы с WHERE, JOIN, ORDER BY по полям без индексов приводят к полному скану таблицы. На больших таблицах это недопустимо. Нужно добавлять индексы по ключам и часто используемым условиям и следить за планами (EXPLAIN). Не индексировать «всё подряд» — индексы замедляют запись и занимают место.

#### SELECT *

Выбор всех столбцов увеличивает объём данных, мешает использованию index-only scan (если не все столбцы в индексе) и связывает код с текущей структурой таблицы. Лучше явно перечислять нужные столбцы.

#### Длинные транзакции

Транзакция, открытая надолго (например, на время всего HTTP-запроса с долгими вычислениями или внешними вызовами), держит блокировки и в PostgreSQL способствует bloat. Нужно сужать транзакцию до минимума: прочитали → решили → записали → закоммитили.

1. Как по логам запросов понять, что у вас N+1?  
2. В каких операциях обязательно нужна транзакция (несколько изменений атомарно)?  
3. Чем плох SELECT * при росте таблицы и появлении тяжёлых столбцов?

<details>
<summary>Ответы</summary>
1. Один и тот же запрос (по шаблону) повторяется много раз подряд — 1 + N раз.  
2. Когда несколько изменений должны быть «всё или ничего»: перевод между счетами, создание заказа с несколькими строками, обновление нескольких связанных таблиц.  
3. Тянет лишние данные, увеличивает сеть и десериализацию; при появлении больших полей (blob, json) запрос становится тяжёлым; мешает index-only scan.
</details>

### Простыми словами

N+1 — «сначала список, потом в цикле по одному»: много лишних запросов. Транзакции нельзя пропускать там, где несколько шагов должны быть атомарными. Индексы нужны там, где мы ищем и сортируем; без них БД перебирает всю таблицу. SELECT * тянет лишнее и мешает оптимизациям. Длинная транзакция — как долго держать дверь закрытой: остальные ждут и страдают.

### Картинка в голове (антипаттерны)

- **N+1:** Ты зашёл в магазин и спросил «дайте список заказов». Тебе выдали список. Потом ты по одному спрашиваешь: «а кто клиент у заказа 1?», «а кто клиент у заказа 2?»… 50 раз. Вместо этого один раз: «дайте заказы с именами клиентов» — один запрос.
- **Игнорирование транзакций:** Два шага «списать с A» и «зачислить на B» без общей транзакции — как переложить деньги из одного кармана в другой в два приёма: если упал посередине, деньги «потерялись» или «удвоились».
- **Нет индекса:** БД ищет по email, перебирая всю таблицу — как искать человека в городе, обходя каждый дом, вместо того чтобы открыть справочник по фамилии.
- **SELECT *:** Ты просишь «дай пользователя» и получаешь всю папку с аватаром, настройками и историей — вместо одной страницы с именем и email.
- **Длинная транзакция:** Ты взял книгу в библиотеке (заблокировал) и пошёл пить кофе на час. Остальные ждут эту книгу. Короткая транзакция: взял → прочитал нужную страницу → положил обратно.

### Проверь себя

1. Почему N+1 сильно бьёт по производительности?  
2. В каких случаях обязательно нужна транзакция?  
3. Чем плоха длинная транзакция в PostgreSQL?

<details>
<summary>Ответы</summary>

1. Вместо одного (или двух) запросов выполняется 1 + N; латентность умножается на число строк, нагрузка на БД растёт; при большом N время ответа становится неприемлемым.  
2. Когда несколько изменений должны быть атомарными (всё или ничего) — например, перевод между счетами, создание заказа с несколькими строками. Без транзакции при сбое получится частичное состояние.  
3. Держит блокировки до коммита; накапливаются мёртвые строки (VACUUM не может их убрать), растёт bloat; выше риск deadlock и таймаутов.
</details>

### Мини‑практикум (антипаттерны глазами)

#### Упражнение 1. Это N+1 или нет?

Сценарий: ты открываешь страницу «Список заказов», и в логах видишь:

- 1 запрос: `SELECT * FROM orders LIMIT 50;`
- 50 запросов: `SELECT * FROM customers WHERE id = ?;` (разные id)

Это N+1?

<details>
<summary>Ответ</summary>
Да. 1 запрос за список + N запросов за связанные данные. Решение: JOIN или batch load `WHERE id IN (...)`, или eager loading в ORM.
</details>

#### Упражнение 2. Что плохого в SELECT * (практически)?

Сервис делает `SELECT * FROM users WHERE id = ?;`, а таблица `users` со временем выросла: появились столбцы `avatar_blob`, `big_json_profile`, `settings_json` и т.д.

Что произойдёт?

<details>
<summary>Ответ</summary>
Запрос начнёт тянуть лишние мегабайты данных, увеличится сеть/десериализация, может ухудшиться кэш, и часть оптимизаций (например, «всё есть в индексе») станет невозможной. Лучше выбирать только нужные столбцы.
</details>

#### Упражнение 3. «Нет индекса» — как заподозрить без EXPLAIN?

Под нагрузкой запрос «по email» начинает резко тормозить при росте таблицы.

<details>
<summary>Ответ</summary>
Если запрос раньше был быстрый, а с ростом таблицы стал медленным, это похоже на полное сканирование (Seq Scan). Почти всегда причина — нет индекса по полю из WHERE/JOIN/ORDER BY. Дальше подтверждаем EXPLAIN и добавляем индекс.
</details>

#### Одна мысль — пять раз (антипаттерны, чтобы точно запомнить)

Повторим главное разными словами. Если забыл — вернись сюда.

1. **N+1** = один запрос за список + в цикле по одному за связью. Плохо: латентность и нагрузка растут с числом строк. Хорошо: один JOIN или один batch (WHERE id IN (...)).
2. **Игнорирование транзакций** = несколько изменений без общей транзакции. Плохо: при сбое — частичное состояние (деньги потеряны или продублированы). Хорошо: одна транзакция на одну атомарную операцию.
3. **Отсутствие индексов** = ищем/сортируем по полю без индекса. Плохо: полный скан таблицы, на больших данных — тормоза. Хорошо: индекс по полям из WHERE, JOIN, ORDER BY.
4. **SELECT *** = тянем все столбцы. Плохо: лишние данные, нет index-only scan, при росте таблицы — ещё хуже. Хорошо: явно перечислять нужные столбцы.
5. **Длинная транзакция** = держим транзакцию открытой долго. Плохо: блокировки, bloat, deadlock. Хорошо: прочитали → решили → записали → закоммитили; не держать во время HTTP или долгих расчётов.

**Типичная ошибка новичка (N+1):** «У нас всего 100 заказов на странице, 101 запрос — не страшно». На локальной машине и при одном пользователе так и есть. Но в проде 1000 пользователей открывают страницу в минуту — это уже 101 000 запросов в минуту только на один экран; плюс латентность 2 мс × 101 = 202 мс на один запрос только на сеть к БД. При росте до 500 заказов на страницу получается 501 запрос на одного пользователя и падение под нагрузкой. N+1 надо устранять не «когда станет плохо», а сразу: один JOIN или batch load.

**Не путай: N+1 и «просто много запросов».** N+1 — это именно **один** запрос за список и потом **по одному** запросу на каждый элемент списка (одинаковая форма запроса, меняется только id). «Просто много запросов» может быть нормально: например, на странице 5 разных блоков, каждый делает свой запрос — итого 5 запросов, но не 1+N. Признак N+1: в логах видишь один запрос за коллекцию и затем N одинаковых запросов с разными id (часто `WHERE id = ?`). Исправление N+1 — объединить в один JOIN или один batch; «просто много разных запросов» оптимизируют по-другому (кэш, объединение логики и т.д.).

#### Частые вопросы по разделу (§109)

**В: Как быстро понять, что у нас N+1, без логов?**  
О: Включи в dev-режиме логирование всех SQL-запросов (или используй профилирование БД). Открой типичную страницу (например, список заказов). Если видишь десятки одинаковых запросов с разными id — почти наверняка N+1.

**В: Индекс замедляет запись — насколько?**  
О: Каждый индекс при INSERT/UPDATE/DELETE нужно обновлять. На таблицах с частой записью много индексов (десятки) действительно бьют по производительности. Обычно 3–10 индексов на таблицу — норма; индексируем то, по чему реально ищем и сортируем, а не «на всякий случай».

**В: «Длинная транзакция» — это сколько по времени?**  
О: Жёсткой границы нет. Правило большого пальца: если транзакция живёт дольше 1–2 секунд — уже подозрительно. Если внутри есть ожидание сети (HTTP, очередь) или тяжёлые вычисления — точно выносить за пределы транзакции.

#### Памятка: 3 вещи, которые нельзя забыть (§109)

1. **N+1 убивает под нагрузкой.** Один запрос за список + N в цикле = латентность и нагрузка растут с N. Всегда: JOIN или batch (WHERE id IN) или eager load в ORM.
2. **Несколько связанных изменений = одна транзакция.** Перевод денег, создание заказа с строками — без транзакции при сбое получится частичное состояние. Оберни в begin/commit.
3. **Медленный запрос с WHERE/ORDER BY — смотри план.** Seq Scan по большой таблице = скорее всего нет индекса. Добавь индекс по полям из условия и сортировки; не индексируй «всё подряд».

#### Чек-лист перед релизом: нет ли антипаттернов? (проверь да/нет)

- [ ] На страницах/эндпоинтах с списками и связями нет «1 запрос + N в цикле» (логи: число запросов на действие не десятки).
- [ ] Операции из нескольких изменений (перевод, заказ с строками) обёрнуты в одну транзакцию.
- [ ] Медленные запросы с WHERE/ORDER BY проверены через EXPLAIN; по нужным полям есть индексы.
- [ ] В коде не тянем SELECT * без необходимости; указаны нужные столбцы.
- [ ] Транзакции короткие: нет HTTP-вызовов и долгих расчётов внутри транзакции.

### Запомните

Антипаттерны: N+1 (решать через JOIN или batch load), игнорирование транзакций (одна транзакция на атомарную операцию), отсутствие индексов (индексировать по условиям и ключам), SELECT * (выбирать нужные столбцы), длинные транзакции (держать транзакции короткими). Их устранение — базовый шаг к предсказуемой производительности и надёжности.

**Резюме §109 в одну строку:** Не делай 1 запрос + N в цикле (делай JOIN или batch); несколько связанных изменений — в одну транзакцию; медленный запрос с WHERE — смотри план и индексы; не SELECT *; не держи транзакцию долго.
