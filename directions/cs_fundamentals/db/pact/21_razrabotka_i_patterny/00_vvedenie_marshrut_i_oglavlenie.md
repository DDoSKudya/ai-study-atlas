## Часть XXI. Разработка и паттерны работы с БД

> Цель этой части — **научиться проектировать доступ к данным и работу с БД так, чтобы код был поддерживаемым, тестируемым и масштабируемым: паттерны доступа, транзакции, масштабирование, антипаттерны, best practices, версионирование и идемпотентность**.

После этой части ты сможешь:

- **выбирать и применять паттерны доступа к данным**: Repository, Unit of Work, Active Record, Data Mapper — понимая их плюсы и границы;
- **управлять транзакциями в приложении**: транзакционный скрипт vs доменная модель, явные и неявные границы транзакций;
- **использовать паттерны масштабирования**: read replicas, write-through/write-behind cache, CQRS, event sourcing — когда что уместно;
- **избегать антипаттернов**: N+1, избыточные запросы, игнорирование транзакций, отсутствие индексов, длинные транзакции;
- **применять best practices**: подготовленные выражения, пулы соединений, таймауты, retry, circuit breaker;
- **версионировать данные и блокировки**: оптимистичная и пессимистичная блокировка, soft delete, аудит;
- **обеспечивать идемпотентность**: идемпотентные ключи, повторная обработка, idempotency key в API, обработка дублей.

Материал опирается на:

- **часть IV** (транзакции, ACID, уровни изоляции, блокировки);
- **часть VII** (оптимизация запросов, антипаттерны);
- **часть VIII** (ORM, миграции, пулы соединений);
- **часть XI** (масштабирование, репликация, CQRS, event sourcing);
- **часть XII** (безопасность, подготовленные выражения).

В этой части мы смотрим на БД **как на часть приложения**: не только «как писать SQL», а **как организовать код, транзакции и масштабирование**, чтобы система оставалась предсказуемой и устойчивой.

### Краткая шпаргалка по части

**Термины (одним списком).**  
Ключевые слова этой части — каждое в одном предложении. Держи под рукой при чтении разделов; при затруднении возвращайся к этой таблице.

| Слово | Что это значит простыми словами |
|-------|----------------------------------|
| **Repository** | Слой, который «прячет» БД: приложение работает с объектами, репозиторий знает, как их загрузить/сохранить. Удобно тестировать и менять источник данных. |
| **Unit of Work** | Объект, который запоминает все изменения за «логическую операцию» и одним коммитом отправляет их в БД. Гарантирует атомарность без размазывания BEGIN/COMMIT по коду. |
| **Active Record** | Объект = одна строка таблицы; у объекта есть методы save/delete. Просто, но объект «знает» про БД — тестировать и менять источник сложнее. |
| **Data Mapper** | Отдельный слой преобразует объекты домена в строки БД и обратно. Домен не знает про БД; гибко, но больше кода. |
| **Транзакционный скрипт** | Один метод/скрипт = одна бизнес-операция = одна транзакция. Всё в одном месте: открыл транзакцию, сделал шаги, закоммитил. |
| **Доменная модель** | Бизнес-логика в объектах; транзакции на границах (сервис, use case). Сложнее, но масштабируется на сложную логику. |
| **Read replica** | Копия БД только для чтения. Запросы на чтение идут на реплику, запись — на primary. Разгружает master; возможна задержка (lag). |
| **Write-through cache** | Запись идёт и в кэш, и в БД одновременно. Кэш и БД согласованы; запись чуть медленнее. |
| **Write-behind cache** | Запись сначала в кэш, в БД — асинхронно. Быстро, но при падении можно потерять данные. |
| **CQRS** | Command Query Responsibility Segregation: отдельные модели (и часто хранилища) для «записи» и «чтения». Удобно масштабировать чтение и менять представления. |
| **Event sourcing** | Храним не текущее состояние, а лог событий; состояние восстанавливаем, «проиграв» события. Полная история и гибкость представлений; сложнее реализация. |
| **N+1** | Один запрос за списком, потом N запросов в цикле за связанными данными. Резко растёт число запросов и латентность. Решение: JOIN или batch load. |
| **Prepared statement** | Запрос с плейсхолдерами (? или $1); параметры передаются отдельно. Защита от SQL-инъекций и возможность кэширования плана на стороне БД. |
| **Circuit breaker** | «Выключатель»: после серии ошибок перестаём вызывать сломанный сервис (БД) и сразу возвращаем ошибку; периодически пробуем снова. Защита от каскадных падений. |
| **Оптимистичная блокировка** | Не блокируем строку; при обновлении проверяем, что версия (или значение) не изменилась. Конфликт — если кто-то успел изменить. |
| **Пессимистичная блокировка** | Явно блокируем строку (SELECT FOR UPDATE) до изменения. Другие ждут; риск deadlock при неправильном порядке блокировок. |
| **Soft delete** | Вместо DELETE ставим флаг (deleted_at) или статус. Строка остаётся; выборки фильтруют по флагу. Восстановление и аудит проще. |
| **Идемпотентность** | Повторное выполнение операции даёт тот же результат и не создаёт «двойного» эффекта. Ключевое свойство для retry и повторной доставки сообщений. |
| **Idempotency key** | Уникальный ключ запроса от клиента; сервер по ключу понимает «это повтор» и возвращает сохранённый результат, не выполняя операцию снова. |

### Три предложения — суть всей части

1. **Абстракция над данными и явные границы транзакций** делают код предсказуемым и тестируемым; длинные транзакции и «размазанный» доступ к БД — источник багов и тормозов.
2. **Масштабирование** (replicas, кэш, CQRS, event sourcing) и **антипаттерны** (N+1, отсутствие индексов, игнорирование транзакций) — две стороны одной медали: правильный паттерн решает проблему, антипаттерн её усугубляет.
3. **Best practices** (prepared statements, пулы, таймауты, retry, circuit breaker), **версионирование** (оптимистичная/пессимистичная блокировка, soft delete) и **идемпотентность** — базовый «гигиенический» слой, без которого продовая система хрупкая.

**Понятия по темам.**  
Кратко: о чём каждая группа разделов и зачем она нужна.

| Понятие | Суть |
|--------|------|
| **Паттерны доступа к данным** | Repository (слой «дай/сохрани»), Unit of Work (одна транзакция на набор изменений), Active Record (объект сам save/find), Data Mapper (домен отдельно от БД). Выбор по сложности домена и тестируемости. |
| **Транзакции в приложении** | Транзакционный скрипт (один метод = одна транзакция) и доменная модель (транзакция на границе сервиса). Граница транзакции явная; один запрос — одна транзакция, короткая, без HTTP внутри. |
| **Паттерны масштабирования** | Read replicas (чтение с копий, lag), write-through/write-behind кэш, CQRS (разделение команд и запросов), event sourcing (лог событий вместо текущего состояния). |
| **Антипаттерны** | N+1 (один запрос + N в цикле), игнорирование транзакций, отсутствие индексов, SELECT *, длинные транзакции. Устранять: JOIN/batch, одна транзакция на атомарную операцию, индексы по условиям, короткие транзакции. |
| **Best practices** | Prepared statements (плейсхолдеры), пулы соединений, таймауты, retry с backoff (только для идемпотентного), circuit breaker при падении БД. |
| **Версионирование** | Оптимистичная блокировка (version при обновлении), пессимистичная (SELECT FOR UPDATE), soft delete (deleted_at вместо DELETE). |
| **Идемпотентность** | Повтор запроса не меняет результат. Idempotency key от клиента; результат по ключу хранить в БД; при повторе возвращать сохранённый ответ. |

Используй эту таблицу как **карту**: нашёл понятие в первом столбце — во втором видишь суть и зачем это нужно.

Во всех разделах этой части используется **единый шаблон**, чтобы идеи повторялись в разных формах:

**Оглавление** сгруппировано **по этапам изучения**: Этап 1 (§106–107), Этап 2 (§108–109), Этап 3 (§110–112), затем блок «Справочник и проверка». **Как пользоваться оглавлением.**

1. **Выбери этап** (например, «Этап 1. Доступ к данным и транзакции») — это твой следующий шаг в изучении.
2. **Пройди по строкам таблицы** — в «Содержание» указано, о чём раздел; в «Ключевые понятия» — термины, которые нужно знать после изучения.
3. **Переходи по ссылке** в первом столбце — попадёшь в раздел с целью, терминами, теорией, блоком «Проверь себя» (вопросы с ответами под спойлером) и «Запомните».
4. **Справочник и проверка** в конце части — когда этапы пройдены, используй их для повторения и самопроверки.
