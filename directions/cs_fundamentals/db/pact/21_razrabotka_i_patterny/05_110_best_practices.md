## 110. Best practices

**Связь с предыдущим.** Антипаттерны (§109) — чего избегать. Best practices — что делать положительно: как безопасно формировать запросы (prepared statements), как не исчерпать ресурсы (пулы, таймауты), как вести себя при сбоях (retry, circuit breaker). Это следующий шаг после «не делать плохо» к «делать хорошо и устойчиво».

**Одной фразой:** Подготовленные выражения, пулы соединений, таймауты, retry с backoff и circuit breaker — обязательный «гигиенический» слой при работе с БД в продакшене.

### Цель раздела

Закрепить **практики**, без которых работа с БД в продакшене небезопасна и нестабильна: защита от инъекций (prepared statements), ограничение и переиспользование соединений (пулы), таймауты, повтор при временных сбоях (retry), отключение при каскаде ошибок (circuit breaker) и аккуратное логирование.

### В этом разделе главное (три строки)

1. **Подготовленные выражения (prepared statements)** — запрос с плейсхолдерами; параметры передаются отдельно. Защита от SQL-инъекций и возможность кэширования плана на стороне БД. Всегда использовать для пользовательского ввода.
2. **Пулы соединений** — ограничение числа одновременных соединений к БД и их переиспользование. Без пула при пиковой нагрузке можно исчерпать лимит соединений БД и «положить» сервер.
3. **Таймауты** (connection, query), **retry** с экспоненциальной задержкой и **circuit breaker** (при серии ошибок перестаём слать запросы, потом пробуем снова) защищают от зависаний и каскадных падений.

### Самая простая версия (что нужно сделать “по умолчанию”)

Если ты запомнишь только это — уже будешь писать «по‑взрослому»:

1. **Никогда не склеивай SQL строками.** Только плейсхолдеры + параметры.
2. **Всегда используй пул соединений.** Не создавай соединение на каждый запрос.
3. **Поставь таймауты.** И на соединение, и на запрос.
4. **Retry делай только там, где безопасно.** И только с backoff.
5. **Если БД “умирает” — не добивай её.** Circuit breaker спасает.

### Примеры prepared statements (чтобы не было “а как именно?”)

**Python (psycopg/psycopg2‑подобно):**

```python
cur.execute("SELECT id, email FROM users WHERE email = %s", [email])
```

**Node.js (pg):**

```js
await client.query("SELECT id, email FROM users WHERE email = $1", [email])
```

**Java (JDBC):**

```java
PreparedStatement ps = conn.prepareStatement(
  "SELECT id, email FROM users WHERE email = ?"
);
ps.setString(1, email);
ResultSet rs = ps.executeQuery();
```

Во всех случаях идея одна: **значение передаётся отдельно от текста SQL**, поэтому оно не может стать «кусочком синтаксиса SQL».

#### Пошагово: как перейти на prepared statements в старом коде (без паники)

Если в коде уже есть склейка строк типа `"SELECT ... WHERE id = " + id` или `f"SELECT ... WHERE email = '{email}'"`:

1. **Найди все места**, где в SQL подставляются переменные (поиск по `execute`, `query`, конкатенации строк с кавычками).
2. **Замени на плейсхолдеры:** оставь в строке SQL знаки `?` или `%s` или `$1, $2` (как принято в твоём языке/драйвере), а значения передай вторым аргументом (массив/список параметров).
3. **Проверь:** тот же запрос должен выполняться и возвращать тот же результат; изменилось только то, что значение больше не «вклеено» в текст, а передано отдельно. После этого инъекция через переменную невозможна, и БД может кэшировать план.

Пример замены: было `cursor.execute("SELECT * FROM users WHERE id = " + str(id))` → стало `cursor.execute("SELECT * FROM users WHERE id = %s", [id])`.

### Пулы соединений: почему без них всё ломается (очень понятная причина)

Открыть соединение к БД — это не «дешёвая операция». Если каждый HTTP‑запрос открывает новое соединение:

- у БД резко растёт число соединений;
- БД тратит память на каждое соединение;
- ты упираешься в `max_connections`;
- начинаются ошибки «too many connections» и лавинообразный отказ.

**Практическое правило:** у приложения должен быть пул (например, 10–50 соединений, зависит от нагрузки и БД), а не «соединение на каждый запрос».

### Таймауты: не жди вечность

Минимальный набор:

- **connection timeout** — сколько ждать установления соединения;
- **query/statement timeout** — сколько максимум может выполняться запрос.

В PostgreSQL часто используют:

- `statement_timeout` (ограничить время запроса),
- `lock_timeout` (не ждать блокировку слишком долго),
- `idle_in_transaction_session_timeout` (убить соединение, которое «зависло» в транзакции).

Идея: лучше быстро вернуть ошибку и деградировать аккуратно, чем «повесить» сервис.

**Логирование при работе с БД:** Логировать ошибки соединения и запросов (без паролей и чувствительных данных в тексте); при необходимости — медленные запросы (выше порога, например 1 с) и их план. В проде не логировать каждый успешный запрос с полным телом — объём и риск утечки. Достаточно: ошибки, таймауты, метрики (число запросов, латентность). Это помогает при расследовании инцидентов и тюнинге.

### Retry: что можно повторять, а что нельзя (ключевой момент)

**Можно повторять** (обычно):

- чтения (SELECT) — почти всегда безопасно;
- операции с идемпотентным ключом (см. §112);
- ошибки сериализации / deadlock (PostgreSQL может вернуть serialization failure) — при правильной логике повтор допустим.

**Нельзя повторять просто так**:

- платежи, создание заказа, списания — если нет идемпотентности, повтор может создать «двойной эффект».

**Типичная ошибка новичка (retry):** Настроить retry на все ошибки или на таймаут для всех запросов к БД. При таймауте на создание заказа клиент повторяет запрос, сервер снова создаёт заказ — два заказа. Retry без идемпотентности безопасен только для чтения (SELECT) и для операций с idempotency key. Для POST создания заказа, платежа — либо не делать автоматический retry, либо ввести idempotency key и хранить результат по ключу.

### Circuit breaker: 3 состояния (на пальцах)

- **closed**: всё нормально, запросы идут в БД;
- **open**: БД часто отвечает ошибкой → мы временно перестаём её дергать, сразу отвечаем ошибкой;
- **half‑open**: прошло время, пробуем 1–N запросов; если успех — закрываем, если ошибка — снова открываем.

Это не «излишество», а защита от ситуации, когда умирающая БД получает лавину повторов и окончательно ложится.

**В двух словах: зачем circuit breaker.** Без него при падении БД каждый запрос ждёт таймаут (например 5 сек) — тысячи запросов = тысячи потоков висят 5 секунд = сервис «мёртв». С circuit breaker после N ошибок мы перестаём слать запросы и сразу возвращаем ошибку — система падает предсказуемо, а не зависает; через минуту пробуем снова.

#### Что будет, если не использовать prepared statements (даже «без пользователя»)

Классическая SQL-инъекция: в форму ввели `' OR '1'='1` — и условие WHERE превратилось в «всегда истина», отдаётся весь список пользователей. Но даже если «ввод внутренний» (id из другого сервиса, лог, очередь), строка может содержать кавычки или спецсимволы; склейка SQL строками даёт ошибки парсинга или неожиданное поведение. Плюс без параметров СУБД не может стабильно кэшировать план запроса: каждый раз разный текст → каждый раз новый разбор и план. Итого: prepared statements — это защита от инъекций **и** от ошибок **и** возможность кэширования плана. Всегда использовать.

#### Что будет, если не ставить таймауты

Запрос к БД «завис» (deadlock, тяжёлая блокировка, сеть). Без таймаута приложение ждёт бесконечно. Поток/воркер занят, соединение из пула не возвращается. Под нагрузкой все воркеры и все соединения забиваются «висящими» запросами. Новые запросы не обрабатываются — сервис «умер» для пользователей, хотя процесс жив. С таймаутом через N секунд запрос обрывается, соединение возвращается в пул, пользователь получает ошибку, но сервис продолжает обслуживать остальных.

#### Что будет, если не использовать пул соединений

Каждый HTTP-запрос открывает новое соединение к БД и закрывает после ответа. При 100 RPS это 100 новых соединений в секунду. Открытие соединения — не бесплатная операция (TCP, аутентификация, выделение памяти на стороне БД). БД имеет лимит `max_connections` (часто 100–200). При пике в 500 одновременных запросов 500 соединений пытаются открыться — лимит исчерпывается, новые запросы получают «too many connections» или зависают в ожидании. С пулом, например, 20 соединений, эти 20 переиспользуются; 500 запросов в секунду обслуживаются очередью к пулу, но лимит БД не превышается.

### Термины

- **Prepared statement (подготовленное выражение)** — SQL-запрос с плейсхолдерами (? или $1, $2, …); значения подставляются отдельно. СУБД может один раз построить план и переиспользовать его; пользовательский ввод не попадает в текст запроса, что исключает классическую SQL-инъекцию.
- **Connection pool (пул соединений)** — набор открытых соединений к БД, которые переиспользуются запросами. Приложение берёт соединение из пула, выполняет запрос, возвращает соединение в пул. Ограничивает максимальное число соединений и снижает накладные расходы на открытие/закрытие.
- **Circuit breaker (предохранитель)** — после N подряд ошибок при обращении к сервису (БД) «выключатель» переходит в состояние «открыто» и не вызывает сервис, сразу возвращая ошибку; через некоторое время делается попытка «полуоткрытого» состояния (пробуем снова). Снижает нагрузку на сломанный сервис и каскадные отказы.

### Теория и правила

#### Подготовленные выражения

Никогда не склеивать SQL со строкой от пользователя. Всегда: запрос с плейсхолдерами + передача параметров отдельно. Пример (псевдокод):

```python
cursor.execute("SELECT * FROM users WHERE email = %s", [user_input])
```

Так `user_input` трактуется как данные, а не как часть SQL. В ORM запросы обычно параметризуются автоматически; при raw SQL — только через плейсхолдеры.

**Теоретическое обоснование:** SQL-инъекция возможна, когда пользовательский ввод попадает в текст запроса как часть синтаксиса (например, подстановка строки в WHERE email = '" + user_input + "'"). Плейсхолдеры гарантируют, что значение передаётся как данные (bound parameter), а не как часть SQL; СУБД сама экранирует и типизирует значение. Стандарт SQL (SQL:1999 и далее) описывает подготовленные выражения (PREPARE, EXECUTE); все современные драйверы и ORM поддерживают параметризованные запросы.

#### Пулы соединений

Настройка: минимальное и максимальное число соединений в пуле, таймаут ожидания соединения. Размер пула не должен превышать лимит БД (max_connections) с учётом других приложений и реплик. Слишком большой пул — лишняя память на БД; слишком маленький — очереди и таймауты под нагрузкой.

#### Таймауты

- **Connection timeout** — сколько ждать установления соединения.  
- **Query (statement) timeout** — максимальное время выполнения одного запроса. В PostgreSQL: `statement_timeout`. Предотвращает «висящие» запросы.

#### Retry

При временных ошибках (потеря соединения, deadlock, перегрузка) повторять запрос ограниченное число раз с задержкой (лучше exponential backoff). Важно: повторять только идемпотентные операции или те, для которых повтор безопасен (иначе двойное списание и т.п.).

**Exponential backoff (экспоненциальная задержка):** после первой ошибки ждём, например, 100 мс; после второй — 200 мс; после третьей — 400 мс и т.д. (с верхним пределом и случайным джиттером, чтобы не синхронизировать повторы с других клиентов). Так снижается нагрузка на БД при массовом сбое. **Сколько раз повторять:** Обычно ограничивают число попыток (например, 3–5). После исчерпания — возвращать ошибку клиенту или логировать и прерывать. Бесконечный retry без лимита при «мёртвой» БД приведёт к бесконечному циклу; лимит обязателен.

**Почему именно exponential backoff, а не «подождать 1 секунду и повторить»:** При массовом сбое (БД перегружена или упала) тысячи клиентов одновременно делают retry. Если все ждут одну и ту же паузу (например, 1 сек), через секунду все снова бьют в БД — новый пик, БД снова падает. С экспоненциальной задержкой (100 мс, 200 мс, 400 мс…) и джиттером повторы размазываются во времени, нагрузка не собирается в один «удар». Плюс даём БД время восстановиться, прежде чем слать много запросов снова.

#### Circuit breaker

При множественных ошибках подряд не слать новые запросы к БД какое-то время; возвращать ошибку клиенту сразу. Периодически пробовать один запрос; если успех — «закрыть» выключатель. Так падающая БД не получает лавину запросов, а приложение не зависает в ожидании таймаутов.

**Проверь себя (подпункты §110: prepared statements, пулы, таймауты, retry, circuit breaker)**

1. Зачем prepared statements, если параметры не от пользователя?  
2. В каком случае retry после ошибки БД опасен?  
3. Чем circuit breaker лучше «просто таймаута» при падении БД?

<details>
<summary>Ответы</summary>
1. Защита от инъекций (внутренние данные тоже могут содержать кавычки); кэш планов выполнения в СУБД.  
2. Для неидемпотентных операций — повтор даст двойной эффект (два списания, два заказа). Retry только для идемпотентных или с идемпотентным ключом.  
3. Таймаут обрывает один запрос; при падении БД каждый новый запрос снова ждёт таймаут — все потоки висят. Circuit breaker после N ошибок перестаёт слать запросы и сразу возвращает ошибку, разгружая систему.
</details>

### Простыми словами

Prepared statements — «подставляй параметры отдельно, не вклеивай в текст запроса». Пул — «ограниченный набор соединений, которые переиспользуются». Таймауты — «не жди вечно ни соединение, ни ответ на запрос». Retry — «при временной ошибке попробуй ещё раз с паузой». Circuit breaker — «если всё падает, перестань слать запросы и дай БД отдохнуть, потом попробуй снова».

### Картинка в голове (best practices)

- **Prepared statements:** Ты не вклеиваешь текст от пользователя в предложение, а передаёшь его «отдельным пакетом». Как заполнить бланк: в графе «имя» пишешь имя, а не вставляешь целый абзац в середину инструкции.
- **Пул соединений:** Вместо того чтобы каждый раз строить новый мост к БД, у тебя есть 20 мостов (соединений). Запрос пришёл — взял мост, перешёл, вернул мост. Следующий запрос — тот же мост. Не 1000 мостов при 1000 запросах.
- **Таймауты:** «Жду ответ от БД не больше 5 секунд. Не ответила — обрываю, возвращаю ошибку, освобождаю соединение». Иначе «висящий» запрос навсегда занял бы воркер и соединение.
- **Retry:** «Упало по таймауту или deadlock — подожду 100 мс и попробую ещё раз (до N раз)». Только для операций, которые безопасно повторить (чтение, или операция с идемпотентным ключом).
- **Circuit breaker:** БД начала падать. Вместо того чтобы тысяча запросов ждала таймаут по 5 секунд каждый, мы «выключили» обращение к БД: сразу возвращаем ошибку. Через минуту пробуем один запрос; если ок — «включаем» снова.

### Проверь себя

1. Зачем использовать prepared statements даже когда «нет пользовательского ввода»?  
2. Что может произойти без пула соединений при пиковой нагрузке?  
3. В каком случае retry после ошибки опасен?

<details>
<summary>Ответы</summary>

1. Защита от инъекций — главная причина; кроме того, СУБД может кэшировать план выполнения для параметризованного запроса. Даже «внутренний» ввод (другой сервис, лог) может содержать опасные символы.  
2. Каждый запрос открывает новое соединение; при большом числе одновременных запросов можно исчерпать max_connections на БД, новые запросы начнут падать или зависать, возможен каскадный отказ.  
3. Retry опасен для неидемпотентных операций: повтор может привести к двойному списанию, двойной записи и т.д. Повторять имеет смысл только идемпотентные операции или с идемпотентными ключами.
</details>

### Мини‑практикум (best practices в реальных ситуациях)

#### Упражнение 1. Найди SQL‑инъекцию (самый частый баг новичка)

```python
sql = "SELECT * FROM users WHERE email = '" + email + "'"
row = db.execute(sql).fetchone()
```

Почему это опасно?

<details>
<summary>Ответ</summary>
Потому что `email` становится частью SQL‑синтаксиса. Если туда попадёт `x' OR '1'='1`, запрос изменится. Решение: плейсхолдеры/параметры.
</details>

#### Упражнение 2. Где ставить таймауты?

Что лучше: «таймаутов нет» или «таймауты есть, но иногда запрос падает по timeout»?

<details>
<summary>Ответ</summary>
Лучше таймауты есть. Без таймаутов сервис может повиснуть навсегда (и утащить за собой очередь, воркеры, память). С таймаутами ты контролируешь деградацию: да, иногда запрос упадёт, но система останется живой.
</details>

#### Упражнение 3. Retry: безопасно или нет?

Можно ли автоматически повторять `POST /pay` (оплата)?

<details>
<summary>Ответ</summary>
Только если операция идемпотентна (есть idempotency key / уникальный ключ операции) и повтор не создаёт второй платёж. Иначе повтор опасен.
</details>

#### Одна мысль — пять раз (best practices, чтобы точно запомнить)

1. **Prepared statements** — никогда не склеивать SQL и данные строкой. Всегда плейсхолдеры + параметры. Защита от инъекций и кэш плана.
2. **Пул соединений** — не открывать соединение на каждый запрос. Ограниченный пул (10–50 и т.д.), переиспользование. Иначе при пике — исчерпание max_connections.
3. **Таймауты** — на соединение и на запрос. Иначе «висящий» запрос занимает воркер и соединение навсегда, сервис «умирает».
4. **Retry** — только для идемпотентных операций или с идемпотентным ключом. С backoff (100 мс, 200 мс, 400 мс…). Иначе двойное списание, двойной заказ.
5. **Circuit breaker** — после серии ошибок не слать запросы к БД какое-то время; потом попробовать снова. Защита от добивания умирающей БД.

#### Частые вопросы по разделу (§110)

**В: Какой размер пула соединений ставить?**  
О: Зависит от нагрузки и лимита БД. Правило: сумма пулов всех приложений не должна превышать max_connections (с запасом на админские соединения). Часто 10–20 на инстанс приложения достаточно; при росте нагрузки смотрят на метрики (очередь к пулу, использование соединений).

**В: На какие ошибки делать retry?**  
О: Обычно retry делают на временные ошибки: потеря соединения, таймаут, deadlock, serialization failure (в PostgreSQL). На «дубликат ключа» или «нарушение ограничения» retry без изменения данных бессмысленен — нужно обработать ошибку по-другому.

**В: Нужен ли circuit breaker, если есть таймауты?**  
О: Таймауты обрывают один запрос; но если БД лежит, каждый новый запрос будет ждать таймаут (например, 5 секунд). Тысячи запросов = тысячи потоков ждут 5 секунд = сервис не отвечает. Circuit breaker после N ошибок перестаёт слать запросы и сразу возвращает ошибку — система деградирует предсказуемо, а не «висит».

#### Памятка: 3 вещи, которые нельзя забыть (§110)

1. **Всегда prepared statements.** Никогда не склеивать SQL и данные. Плейсхолдеры + параметры. Защита от инъекций и кэш плана.
2. **Пул соединений + таймауты.** Без пула — исчерпание max_connections при пике. Без таймаутов — «висящие» запросы забивают воркеры и соединения навсегда.
3. **Retry только для идемпотентного.** Повторять можно чтение или операции с idempotency key. Платёж, создание заказа без ключа — не повторять автоматически, иначе двойной эффект.

#### Чек-лист перед продом: работа с БД (проверь да/нет)

- [ ] Все запросы с данными — через prepared statements (плейсхолдеры + параметры), нигде нет склейки SQL и строк.
- [ ] Есть пул соединений с ограничением; размер пула не превышает лимит БД с учётом других приложений.
- [ ] Заданы таймауты: на установку соединения и на выполнение запроса (statement_timeout или аналог).
- [ ] Retry настроен только для идемпотентных операций или операций с idempotency key; для платежей/заказов без ключа — не автоматический retry.
- [ ] При серии ошибок к БД есть circuit breaker или аналог (не добиваем падающую БД лавиной запросов).

### Запомните

Best practices при работе с БД: подготовленные выражения для всех запросов с данными; пулы соединений с ограничением и таймаутами; таймауты на соединение и запрос; retry с backoff только для идемпотентных или безопасных повторов; circuit breaker при каскаде ошибок. Это базовый слой устойчивости и безопасности.

**Резюме §110 в одну строку:** Всегда плейсхолдеры (не склеивай SQL); пул соединений и таймауты; retry только для идемпотентного; при падении БД — circuit breaker.
