## 112. Идемпотентность

**Связь с предыдущим.** Best practices (§110) включают retry — повтор запроса при сбое. Но повтор платёжной или заказной операции без защиты создаст дубликат. Идемпотентность и idempotency key — как сделать так, чтобы повтор запроса не вёл к двойному списанию или двум заказам. Версионирование (§111) про конфликты при одновременном изменении; идемпотентность — про повтор одной и той же операции во времени.

**Одной фразой:** Операция идемпотентна, если повторное выполнение не меняет результат и не создаёт «двойного» эффекта; идемпотентные ключи и idempotency key в API позволяют безопасно повторять запросы и обрабатывать дубли.

### Цель раздела

Понять **идемпотентность** в контексте БД и API: зачем она нужна, как достигать с помощью идемпотентных ключей и idempotency key, как обрабатывать дубли и как сочетать с транзакциями и retry.

### В этом разделе главное (три строки)

1. **Идемпотентность** — свойство операции: выполненная дважды (с теми же входными данными) даёт тот же результат и не приводит к «двойному» эффекту (например, двойному списанию). Критично для retry и повторной доставки сообщений.
2. **Идемпотентный ключ** — уникальный идентификатор операции (например, id платежа, ключ сообщения). По нему можно понять, выполняли ли мы уже эту операцию, и при повторе вернуть сохранённый результат без повторного выполнения.
3. **Idempotency key в API** — клиент передаёт ключ с запросом; сервер при первом запросе выполняет операцию и сохраняет результат по ключу; при повторном запросе с тем же ключом возвращает сохранённый результат без повторного выполнения. Обработка дублей на уровне приложения и БД (UNIQUE constraint) дополняет это.

### Самая простая версия (главная мысль без теории)

**Жизнь несправедлива:** запросы и сообщения *иногда повторяются*. Не «возможно», а «точно будет»:

- клиент нажал кнопку два раза;
- сеть оборвалась и клиент сделал retry;
- очередь доставила сообщение повторно (at‑least‑once).

**Повторная обработка (в т.ч. из очереди):** Очереди сообщений часто гарантируют доставку «at least once» — сообщение может прийти дважды. Консьюмер (обработчик сообщения), который создаёт заказ или списывает деньги, обязан быть идемпотентным: по ключу сообщения (message_id, idempotency key) проверять «уже обрабатывали?» и при повторе не выполнять операцию снова, а возвращать сохранённый результат или игнорировать дубликат. Иначе одно и то же сообщение создаст два заказа или два списания.

Поэтому у тебя есть два выбора:

1) либо система иногда создаёт дубли (два заказа, два платежа) → боль, деньги, инциденты;  
2) либо ты заранее делаешь операции **идемпотентными** → повтор безопасен.

**Идемпотентность = “повтор не должен вредить”.**

### Пошагово: как реализовать idempotency key в API (так, чтобы было надёжно)

Ниже — самый понятный “скелет”, без микросервисной магии.

#### Что происходит по шагам: первый запрос и повтор с тем же ключом

**Первый запрос с ключом K:** сервер не находит ключ K в хранилище (БД). Выполняет операцию (создаёт заказ, списывает деньги). Сохраняет результат (и HTTP-код, и тело ответа) в хранилище по ключу K. Возвращает этот ответ клиенту. **Повторный запрос с тем же ключом K:** сервер находит ключ K в хранилище. Операцию не выполняет. Достаёт сохранённый ранее результат и возвращает его клиенту (тот же код, то же тело). Для клиента выглядит так, будто повторно получил ответ на тот же запрос — дубликата заказа/платежа нет. Итого: один ключ = один результат; повтор = отдача сохранённого результата, а не повторное выполнение.

**Сценарий из жизни (почему идемпотентность не «теория»):** Пользователь нажал «Оплатить заказ». Сеть подвисла, ответ не пришёл за 10 секунд. Пользователь подумал «не сработало» и нажал «Оплатить» ещё раз. Без идемпотентности сервер обработает оба запроса — два списания, два платежа, хаос и возвраты. С idempotency key второй запрос приходит с тем же ключом; сервер видит «этот ключ уже обработан» и не списывает повторно, а возвращает сохранённый результат первого запроса. Пользователь видит один платёж — всё корректно. То же с retry после таймаута: клиент повторяет запрос с тем же ключом; сервер не создаёт второй заказ. Поэтому идемпотентность — это про реальные двойные клики и обрывы сети, а не про «когда-нибудь».

**Не путай: идемпотентность и транзакция.** Транзакция — про атомарность (несколько изменений вместе или ничего). Идемпотентность — про повтор: если операцию выполнили дважды, не должно быть двойного эффекта. Транзакция не защищает от повтора: два запроса дадут две транзакции и два заказа. Идемпотентность: при повторе с тем же ключом не выполняем снова, возвращаем сохранённый результат. Нужны оба: транзакция внутри одной операции; идемпотентность при повторах операции.

**Цепочка «если — то» (idempotency key):** Если клиент прислал ключ K и в хранилище (БД) такого ключа ещё нет → считаем операцию новой → выполняем её → сохраняем результат по ключу K → возвращаем результат клиенту. Если клиент прислал ключ K и ключ K уже есть в хранилище → считаем запрос повторным → операцию не выполняем → достаём сохранённый результат по ключу K → возвращаем его клиенту (тот же код и тело, что и в первый раз). Один ключ всегда даёт один результат; повтор запроса не создаёт второй заказ или второе списание.

#### Шаг 0. Клиент обязан прислать ключ

Например: заголовок `Idempotency-Key: <uuid>`.

Ключ должен быть:

- уникальным для одной операции (создание заказа/платёж);
- одинаковым при повторах той же операции.

#### Шаг 1. Храни ключи в БД (а не только в памяти)

Память процесса ненадёжна: перезапуск — и ты забудешь, что ключ уже был.

#### Шаг 2. Делай проверку ключа и запись результата атомарно

То есть в одной транзакции: «если ключ новый → выполняем операцию → сохраняем ответ», иначе «читаем сохранённый ответ».

**Концептуальный шаблон (PostgreSQL):**

```sql
-- 1) Пытаемся зарегистрировать ключ.
-- Если ключ уже есть, вставка не пройдёт.
INSERT INTO idempotency_keys(key, response)
VALUES ('K', '{}'::jsonb)
ON CONFLICT (key) DO NOTHING;

-- 2) Смотрим, вставили ли мы ключ “сейчас”.
-- Если не вставили → это повтор → читаем сохранённый response и возвращаем.
```

Дальше есть 2 распространённые техники:

- **Техника A (проще):** если ключ новый — выполняем операцию и потом обновляем response в idempotency_keys.  
- **Техника B (строже):** вводим поле status (IN_PROGRESS / DONE), чтобы параллельные запросы с одним ключом понимали, что «первый уже выполняет», и ждали/возвращали 409/202.

### Обработка дублей через UNIQUE (самый дешёвый вид идемпотентности)

Иногда idempotency key не нужен, если у операции уже есть **естественный уникальный ключ**.

Пример: `payments` имеют `payment_id` от платёжной системы. Тогда:

- `payments(payment_id)` делаем UNIQUE,
- повторная вставка превращается в no-op (или в ошибку, которую ты ловишь и трактуешь как «уже обработано»).

### Термины

- **Идемпотентность (idempotency)** — свойство операции: многократное выполнение с одними и теми же входными данными эквивалентно однократному выполнению. Результат и конечное состояние не меняются после первого выполнения.
- **Idempotency key** — уникальный ключ, который клиент генерирует и передаёт с запросом; сервер использует его, чтобы распознать повтор и не выполнять операцию дважды, а вернуть сохранённый ответ.
- **Обработка дублей** — механизмы, позволяющие не создавать дубликаты при повторной доставке или повторном запросе: уникальные ограничения в БД, проверка «уже обработано» по ключу, идемпотентные ключи в API.

### Теория и правила

#### Зачем нужна идемпотентность

При сбоях и повторах (retry после таймаута, повторная доставка сообщения из очереди, двойной клик пользователя) одна и та же операция может быть вызвана несколько раз. Без идемпотентности: двойное списание, два одинаковых заказа, два раза начисленные бонусы. С идемпотентностью: повтор не меняет результат; можно безопасно retry и принимать повторные запросы.

#### Идемпотентные ключи

Операция имеет уникальный идентификатор (например, payment_id, idempotency_key). Перед выполнением проверяем: «уже выполняли с этим ключом?» — если да, возвращаем сохранённый результат. Реализация: таблица обработанных ключей (key, result, created_at) с UNIQUE(key); при вставке дубликата ключа получаем ошибку и трактуем как повтор — возвращаем сохранённый результат.

**Пример таблицы для хранения результатов по idempotency key (PostgreSQL):**

```sql
CREATE TABLE idempotency_keys (
  key        VARCHAR(255) PRIMARY KEY,
  response   JSONB NOT NULL,   -- сохранённый ответ (status, body)
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- При первом запросе: вставляем ключ и сохраняем ответ; при повторном — SELECT по ключу и возврат сохранённого response.
-- Срок хранения ограничивают (например, удалять ключи старше 24 часов) чтобы не копить бесконечно.
```

#### Idempotency key в API

Клиент отправляет заголовок, например `Idempotency-Key: <uuid>`. Сервер:

1. Проверяет, есть ли уже ответ для этого ключа (кэш или БД).
2. Если есть — возвращает сохранённый ответ (и тот же HTTP-код).
3. Если нет — выполняет операцию, сохраняет ответ по ключу, возвращает ответ.

Срок хранения ключей ограничивают (например, 24 часа), чтобы не копить бесконечно. Для GET и других безопасных методов ключ не обязателен.

#### Транзакции и идемпотентность

Откат транзакции не «отменяет» идемпотентность операции: после ROLLBACK повтор того же запроса с теми же данными не должен привести к другому результату. Поэтому идемпотентность проектируют так: либо операция по ключу «уже выполнена» (тогда повтор — возврат результата), либо «ещё не выполнена» (тогда выполняем и сохраняем результат по ключу в той же транзакции с изменением данных).

3. Что должен вернуть API при повторном запросе с тем же idempotency key?

### Простыми словами

Идемпотентность — «сделать два раза = как один раз». Идемпотентный ключ — «по этому ключу мы уже знаем, что делали; при повторе не делаем снова, а отдаём старый ответ». Idempotency key в API — то же самое: клиент даёт ключ, сервер при повторе не выполняет операцию заново, а возвращает сохранённый результат.

### Картинка в голове (идемпотентность)

- **Идемпотентность:** Ты нажал «Оплатить» два раза (сеть подвисла, кажется, не сработало). Без идемпотентности — два списания. С идемпотентностью — второй запрос распознаётся как повтор, сервер говорит: «уже оплатили», и не списывает второй раз.
- **Idempotency key:** Клиент при первом запросе прикладывает «билет» (уникальный ключ). Сервер выполняет операцию и кладёт результат «в кармашек с этим билетом». При втором запросе с тем же билетом сервер не выполняет операцию снова, а достаёт результат из кармашка и отдаёт его. Один билет = один результат.
- **Обработка дублей в БД:** Таблица «обработанные ключи» с UNIQUE(key). Первая вставка ключа — ок, операция выполнена. Вторая вставка того же ключа — ошибка дубликата; трактуем как «уже обработано», возвращаем сохранённый результат.

2. Как с помощью UNIQUE в БД помочь обработке дублей?  
3. Что должен вернуть сервер при повторном запросе с тем же idempotency key?

<details>
<summary>Ответы</summary>

1. При retry или повторной доставке одна и та же операция может выполниться дважды. Без идемпотентности это даёт двойной эффект (два списания, два заказа). С идемпотентностью повтор не меняет результат.  
2. Можно хранить идемпотентный ключ в таблице с UNIQUE constraint; при повторной вставке получим ошибку дубликата и интерпретируем её как «операция уже выполнена», вернув сохранённый результат без повторного выполнения.  
3. Тот же результат (и тот же HTTP-код), что и при первом запросе с этим ключом — без повторного выполнения операции.
</details>

### Мини‑практикум (идемпотентность в голове)

#### Упражнение 1. Дубль запроса: что именно “дублируется”?

Клиент сделал `POST /orders` и не получил ответ из-за таймаута, но на сервере запрос успел выполниться. Клиент делает retry.

Что может случиться без идемпотентности?

<details>
<summary>Ответ</summary>
Сервер создаст второй заказ. Клиент увидит «два одинаковых заказа», деньги/резервы могут списаться дважды, начнётся хаос.
</details>

#### Упражнение 2. Как сделать создание заказа идемпотентным?

Подсказка: `Idempotency-Key`.

<details>
<summary>Ответ (идея)</summary>
Клиент генерирует ключ (UUID) и отправляет его. Сервер сохраняет результат операции по ключу. При повторе с тем же ключом — возвращает сохранённый ответ и не создаёт второй заказ.
</details>

#### Упражнение 3. “exactly once” и реальность

Можно ли в распределённой системе обещать «точно один раз» без дополнительных механизмов?

<details>
<summary>Ответ</summary>
Обычно нет. Чаще реальность — **at least once** (может быть доставлено повторно) или **at most once** (может потеряться). Поэтому и нужна идемпотентность: делаем так, чтобы at least once было безопасным.
</details>

### Запомните

Идемпотентность — повторное выполнение не меняет результат. Достигается идемпотентными ключами (уникальный идентификатор операции), хранением результата по ключу и возвратом сохранённого результата при повторе. Idempotency key в API позволяет клиенту безопасно повторять запросы; обработка дублей в БД (UNIQUE, проверка «уже обработано») дополняет защиту. В сочетании с retry и транзакциями это основа устойчивости к сбоям и повторам.

**Резюме §112 в одну строку:** Повтор запроса не должен создавать дубликат: idempotency key от клиента, хранить результат по ключу в БД; при повторе возвращать сохранённый ответ; то же для консьюмеров очередей (message_id / ключ сообщения).

#### Частые вопросы по разделу (§112)

**В: Нужен ли idempotency key для GET-запросов?**  
О: Обычно нет. GET по определению идемпотентен (чтение не меняет состояние). Ключ нужен для операций, меняющих данные: POST (создание), PUT/PATCH, DELETE, а также для «опасных» операций вроде списания денег.

**В: Сколько хранить idempotency key?**  
О: Зависит от бизнеса. Часто 24 часа или до нескольких дней. Дольше — лишний объём данных; слишком мало — повторный запрос через день может создать дубликат. Обычно ключи периодически удаляют (cron, TTL).

**В: Что возвращать клиенту при повторном запросе с тем же ключом?**  
О: Тот же HTTP-код и тот же тело ответа, что и при первом успешном запросе. Не выполнять операцию снова — просто прочитать сохранённый результат по ключу и отдать его. Так клиент не видит разницы между первым и повторным запросом.

**Чек-лист: наша операция идемпотентна?** (если нет — дубли при retry/повторе опасны)  
- [ ] Операция меняет данные (создание заказа, платёж, списание)? Если да — нужна защита.  
- [ ] Есть ли уникальный ключ операции (idempotency key от клиента или payment_id и т.п.)?  
- [ ] Результат по ключу храним в БД (не в памяти)? При повторе возвращаем сохранённый результат?  
- [ ] GET и «безопасные» чтения можно не помечать ключом; для POST/PUT/DELETE и платёжных операций ключ обязателен.

#### Памятка: 3 вещи, которые нельзя забыть (§112)

1. **Идемпотентность = повтор не вредит.** Два одинаковых запроса дают тот же результат и не создают двойного эффекта (два списания, два заказа). Нужна для retry и повторной доставки сообщений.
2. **Idempotency key от клиента.** Клиент присылает уникальный ключ (заголовок). Сервер хранит результат по ключу; при повторе возвращает сохранённый результат, не выполняя операцию снова.
3. **Хранить ключи в БД, не в памяти.** Иначе после перезапуска «забудем» обработанные ключи и при повторе создадим дубликат. UNIQUE(key) в таблице или отдельная таблица idempotency_keys.

#### Одна мысль — пять раз (§112: идемпотентность, чтобы точно запомнить)

1. **Идемпотентность** = выполнить дважды с теми же данными = тот же результат, без двойного эффекта (без двух списаний, двух заказов). Нужна при retry и повторной доставке сообщений.
2. **Idempotency key** — клиент присылает уникальный ключ (заголовок). Сервер при первом запросе выполняет операцию и сохраняет результат по ключу; при повторе не выполняет снова, а возвращает сохранённый результат.
3. **Ключи хранить в БД** (таблица idempotency_keys с UNIQUE(key)), не в памяти процесса. Иначе перезапуск = «забыли» ключи = при повторе дубликат.
4. **При повторе с тем же ключом** — возвращать тот же HTTP-код и то же тело ответа, что и при первом успешном запросе. Клиент не должен видеть разницы.
5. **Естественный уникальный ключ** (например, payment_id от платёжной системы) — можно использовать вместо отдельного idempotency key: UNIQUE(payment_id), при дубле вставки ловить ошибку и трактовать как «уже обработано».

## Практикум: один и тот же кейс — разными подходами (очень подробно)

Этот практикум — чтобы у тебя в голове появилось ощущение: «Ага, я понимаю, как это выглядит в коде».

Мы возьмём одну задачу и посмотрим, как она выглядит:

- без паттернов (плохой старт),
- с Repository,
- с Repository + Unit of Work,
- с Active Record,
- с Data Mapper (концептуально).

### Кейс: «Сменить email пользователю»

#### Вариант 0 (плохой): SQL прямо в бизнес‑операции

Проблема: смешивание слоёв, тесты требуют БД, изменения схемы ломают сервис.

```python
def change_email(user_id: int, new_email: str):
    row = db.execute("SELECT id, email FROM users WHERE id=%s", [user_id]).fetchone()
    if not row:
        raise NotFound()
    if "@" not in new_email:
        raise ValueError("bad email")
    db.execute("UPDATE users SET email=%s WHERE id=%s", [new_email, user_id])
```

#### Вариант 1: Repository (минимально «по‑взрослому»)

Идея: сервис говорит «что» сделать, репозиторий решает «как» сходить в БД.

```python
def change_email(user_id: int, new_email: str, user_repo: UserRepository):
    user = user_repo.get_by_id(user_id)
    if not user:
        raise NotFound()
    user.change_email(new_email)
    user_repo.save(user)
```

Плюс: тестируем без БД (FakeUserRepo). Минус: без дисциплины репозитории разрастаются.

#### Вариант 2: Active Record (быстро, но привязка к БД)

```python
user = User.find(user_id)
user.change_email(new_email)
user.save()
```

Плюс: минимум кода. Минус: домен «знает» про БД.

#### Вариант 3: Data Mapper (домен «чистый»)

```python
user = user_repo.get_by_id(user_id)   # репозиторий/mapper собрал доменный объект
user.change_email(new_email)          # доменная логика без SQL
user_repo.save(user)                  # репозиторий/mapper сохранил
```

Снаружи похоже на Repository, но внутри доменные объекты не тянут зависимости на ORM/SQL.

#### Что будет, если не делать операции идемпотентными (реальные последствия)

Клиент нажал «Оплатить» один раз, но из‑за медленной сети отправил запрос дважды (или нажал дважды). Сервер обработал оба запроса. Без идемпотентности: **два списания**, два платежа, два заказа. Деньги списались дважды, пользователь в шоке, поддержка и возвраты. С идемпотентным ключом (Idempotency-Key): второй запрос распознаётся как повтор, сервер не выполняет платёж снова, а возвращает сохранённый результат первого запроса. Один платёж, один заказ — всё корректно. То же с очередями сообщений: доставка «at least once» означает, что одно и то же сообщение может прийти дважды; если обработка не идемпотентна, дубли создают двойные заказы, двойные списания и т.д.

### Кейс: «Создать заказ» (транзакция + конкуренция)

Тут специально много шагов — это типичный «живой» use case.

**Условия:**

- нужно создать order + order_items;
- уменьшить stock товаров;
- всё должно быть атомарно;
- возможны параллельные заказы одного и того же товара.

#### Вариант A: транзакционный скрипт (понятнее всего новичку)

```python
def place_order(user_id, items):
    with db.transaction():
        order_id = db.execute(
            "INSERT INTO orders(user_id, status) VALUES (%s, 'NEW') RETURNING id",
            [user_id]
        ).fetchone()["id"]

        for item in items:
            # 1) уменьшаем остаток (вариант без блокировки — условный UPDATE)
            updated = db.execute(
                "UPDATE products SET stock = stock - %s "
                "WHERE id = %s AND stock >= %s",
                [item.qty, item.product_id, item.qty]
            ).rowcount
            if updated == 0:
                raise OutOfStock()

            # 2) добавляем строку заказа
            db.execute(
                "INSERT INTO order_items(order_id, product_id, qty) VALUES (%s, %s, %s)",
                [order_id, item.product_id, item.qty]
            )

        return order_id
```

**Почему это хорошо:** граница транзакции ясна, атомарность есть.  
**Почему это может стать плохо:** если логики станет очень много, скрипт раздуется.

#### Вариант B: Repository + Unit of Work (когда логики много)

Идея: в use case остаётся «сценарий», а детали доступа к данным уходят в репозитории/мапперы, а коммит централизован.

```python
def place_order(user_id, items, uow):
    with uow:
        order = Order.create(user_id, items)
        uow.orders.add(order)

        for item in items:
            product = uow.products.get_by_id_for_update(item.product_id)  # пессимистично
            product.decrease_stock(item.qty)
            uow.products.track(product)

        uow.commit()
        return order.id
```

Здесь важна мысль: **uow.commit() = одна транзакция**, а use case остаётся читаемым.

**Проверь себя (Практикум: кейсы «Сменить email» и «Создать заказ»)**

1. В кейсе «Сменить email»: чем Вариант 1 (Repository) лучше Варианта 0 (SQL в сервисе) для тестов?  
2. В кейсе «Создать заказ»: зачем в Варианте B вызывать get_by_id_for_update для товара, а не обычный get_by_id?  
3. Почему в place_order транзакция охватывает и создание заказа, и уменьшение stock, а не два отдельных коммита?

<details>
<summary>Ответы</summary>
1. Сервис получает репозиторий извне; в тестах подставляем FakeUserRepo с предзаполненными данными — БД не нужна. При Варианте 0 тесты требуют реальную БД.  
2. get_by_id_for_update блокирует строку (SELECT FOR UPDATE), чтобы при параллельных заказах не продать больше остатка; обычный get_by_id не блокирует — два запроса могут прочитать один stock и оба уменьшить, получится отрицательный остаток.  
3. Всё должно быть атомарно: либо заказ создан и остатки уменьшены, либо ничего. Два коммита при сбое между ними дадут заказ без уменьшения stock или уменьшение без заказа.
</details>

### Мини‑чеклист “я правильно делаю?” (запомни на будущее)

- Есть ли у операции **одна граница транзакции**?
- Нет ли внешних HTTP‑вызовов внутри транзакции?
- Есть ли защита от дублей (идемпотентность) там, где есть retry?
- Нет ли N+1 на экранах/эндпойнтах?
- Есть ли индексы под WHERE/JOIN/ORDER BY?

**Почему каждый пункт важен:** Одна граница транзакции — иначе при сбое частичное состояние. Нет HTTP внутри транзакции — иначе транзакция живёт секунды, bloat. Идемпотентность при retry — иначе повтор даст два списания. Нет N+1 — иначе сотни запросов на экран. Индексы под условиями — иначе полный скан и тормоза.

**Что проверить перед код-ревью (работа с БД):** (1) SQL с данными — только через плейсхолдеры, нигде конкатенация строк. (2) Есть ли транзакция там, где несколько связанных изменений (перевод, заказ с строками). (3) Нет ли внутри транзакции HTTP-вызовов или долгих вычислений. (4) Списки с связями — не цикл с запросом на элемент (N+1), а JOIN или batch. (5) Медленные запросы с WHERE/ORDER BY — есть ли индексы, проверен ли план. (6) Операции с деньгами/заказами при retry — есть ли idempotency key или идемпотентность. (7) Одновременное изменение одной записи — есть ли version (оптимистичная) или блокировка (пессимистичная), если нужно. (8) Удаление — если нужна возможность восстановления, soft delete (deleted_at), а не DELETE.

**Проверь себя (Мини-чеклист и код-ревью)**

1. Назови три пункта из мини-чеклиста «я правильно делаю?» и почему каждый важен.  
2. Почему «нет HTTP внутри транзакции» входит в список того, что проверять перед код-ревью?  
3. Что проверить в коде, если операция — создание заказа или платёж и у клиента есть retry?

<details>
<summary>Ответы</summary>
1. Примеры: одна граница транзакции — иначе при сбое частичное состояние; нет HTTP внутри транзакции — иначе транзакция живёт секунды, bloat; нет N+1 — иначе сотни запросов на экран; индексы под WHERE — иначе полный скан; идемпотентность при retry — иначе повтор даст два списания.  
2. HTTP внутри транзакции держит её открытой на время сетевого вызова (секунды), увеличивает блокировки и bloat; при падении после HTTP состояние неочевидно.  
3. Есть ли idempotency key (от клиента) и сохранение результата по ключу в БД; при повторе возвращается сохранённый результат, а не повторное выполнение операции.
</details>
