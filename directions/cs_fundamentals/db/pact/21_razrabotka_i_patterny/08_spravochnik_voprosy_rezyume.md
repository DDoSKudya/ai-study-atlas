## Вопросы для самопроверки по разделам

**§106. Паттерны доступа к данным**

1. В чём главное отличие Repository от прямых вызовов БД в сервисе?  
2. Зачем Unit of Work, если можно явно открыть транзакцию и вызвать несколько repository.save()?  
3. Почему Active Record удобнее для простого CRUD и хуже для сложной доменной модели?  
4. Чем Data Mapper отличается от Repository? Когда выбирают Data Mapper?  
5. В каком случае Unit of Work не нужен?

<details>
<summary>Ответы</summary>
1. Repository инкапсулирует способ получения и сохранения данных; сервис не знает про таблицы и SQL. Это даёт единую точку изменения, тестируемость (подмена репозитория) и возможность сменить хранилище.  
2. Unit of Work централизует транзакцию: все изменения регистрируются в одном месте, commit один. Удобно, когда много сущностей меняется в одной операции и не хочется таскать соединение/транзакцию по всем методам.  
3. Active Record минимизирует код и быстро даёт CRUD; но при сложной логике объект совмещает данные, поведение и персистентность — тестировать и менять сложнее. Data Mapper отделяет домен от БД.  
4. Repository — слой доступа (методы get/save); Data Mapper — способ преобразования объект↔строка БД внутри этого слоя. Домен при Data Mapper не знает про БД. Выбирают Data Mapper при сложном домене и строгой тестируемости без БД.  
5. Когда одна операция меняет только одну сущность — достаточно одной транзакции и одного repository.save(); Unit of Work нужен при нескольких сущностях и одном атомарном коммите.
</details>

**§107. Паттерны работы с транзакциями**

1. Что такое граница транзакции и почему её не стоит растягивать на несколько HTTP-запросов?  
2. Чем транзакционный скрипт отличается от доменной модели в плане транзакций?  
3. Где в приложении открывать транзакцию — в контроллере или в сервисе? Почему?  
4. В каком порядке должны идти шаги внутри одной транзакции? Можно ли внутри транзакции вызывать внешний HTTP?

<details>
<summary>Ответы</summary>
1. Граница транзакции — начало (BEGIN) и конец (COMMIT/ROLLBACK). Если растянуть на несколько запросов, транзакция долго живёт, держит блокировки и увеличивает bloat; при падении приложения возможны «висящие» транзакции.  
2. В транзакционном скрипте одна операция = один метод с явной транзакцией внутри. В доменной модели транзакция открывается на границе (сервис), внутри вызываются доменные объекты; граница одна, логика разнесена по объектам.  
3. Обычно в сервисе (слой use case): один метод сервиса = одна бизнес-операция = одна транзакция. Контроллер не должен знать про транзакции — так граница одна и предсказуемая.  
4. Порядок: открыть транзакцию → прочитать минимум из БД → принять решение (без сети) → записать изменения → закоммитить. Внешний HTTP внутри транзакции вызывать нельзя — иначе транзакция живёт секунды, блокировки и bloat.
</details>

**§108. Паттерны масштабирования**

1. Как read replicas влияют на консистентность чтения?  
2. В чём риск write-behind кэша?  
3. Что такое cache stampede (dogpile) и как его избежать?  
4. Для каких данных допустим write-behind, для каких — только write-through?

<details>
<summary>Ответы</summary>
1. Реплика отстаёт от primary (lag); чтение сразу после записи может не увидеть новые данные. Для строгой консистентности чтение после записи делают с primary.  
2. Данные пишутся в БД асинхронно; при падении до сброса в БД они теряются. Подходит только для данных, потерю которых можно принять.  
3. Cache stampede — кэш на популярный ключ истёк, много запросов одновременно пошли в БД и создали пик нагрузки. Избегать: один запрос обновляет кэш (singleflight/mutex), остальные ждут или получают stale; джиттер к TTL.  
4. Write-behind допустим для счётчиков, метрик, «онлайн сейчас». Для денег, заказов, профилей — только write-through или без кэша при записи.
</details>

**§109. Антипаттерны разработки**

1. Почему N+1 сильно бьёт по производительности?  
2. В каких случаях обязательно нужна транзакция?  
3. Чем плох SELECT * на практике при росте таблицы?  
4. Что такое «избыточные запросы» кроме N+1? Приведи пример.

<details>
<summary>Ответы</summary>
1. Вместо одного (или двух) запросов выполняется 1 + N; латентность умножается на число строк, нагрузка на БД растёт; при большом N время ответа становится неприемлемым.  
2. Когда несколько изменений должны быть атомарными (всё или ничего) — например, перевод между счетами, создание заказа с несколькими строками. Без транзакции при сбое получится частичное состояние.  
3. Тянет лишние данные (в т.ч. тяжёлые столбцы), увеличивает сеть и десериализацию, мешает index-only scan; при появлении больших полей (blob, json) запрос может стать очень тяжёлым.  
4. Запросы, которые можно объединить в один; запросы без кэша там, где данные редко меняются; повторное чтение тех же данных в одном use case. Пример: два отдельных SELECT по одной таблице с разными WHERE вместо одного с IN или OR.
</details>

**§110. Best practices**

1. Зачем использовать prepared statements даже когда «нет пользовательского ввода»?  
2. В каком случае retry после ошибки опасен?  
3. Зачем ограничивать число попыток retry (например, 3–5)?  
4. Чем circuit breaker отличается от таймаута? Почему одного таймаута недостаточно при падении БД?

<details>
<summary>Ответы</summary>
1. Защита от инъекций — главная причина; кроме того, СУБД может кэшировать план выполнения для параметризованного запроса. Даже «внутренний» ввод может содержать опасные символы.  
2. Retry опасен для неидемпотентных операций: повтор может привести к двойному списанию, двойной записи. Повторять имеет смысл только идемпотентные операции или с идемпотентными ключами.  
3. Бесконечный retry при «мёртвой» БД даёт бесконечный цикл; лимит (3–5) даёт вернуть ошибку клиенту и не перегружать систему.  
4. Таймаут обрывает один запрос; при падении БД каждый новый запрос снова ждёт таймаут — тысячи потоков висят. Circuit breaker после N ошибок перестаёт слать запросы и сразу возвращает ошибку, разгружая БД и давая предсказуемую деградацию.
</details>

**§111. Версионирование данных**

1. Когда предпочтительна оптимистичная, а когда пессимистичная блокировка?  
2. Как в запросах учесть soft delete?  
3. Как при soft delete сохранить уникальность (например, email), чтобы удалённая запись не мешала создать новую с тем же email?  
4. Как по коду понять, что произошёл конфликт при оптимистичной блокировке? Что делать при deadlock при пессимистичной?

<details>
<summary>Ответы</summary>
1. Оптимистичная — при невысокой конкуренции за одну запись, когда не хочется блокировок. Пессимистичная — при высокой конкуренции, когда важно гарантировать последовательность изменений и готовы ждать.  
2. Во всех выборках добавлять условие deleted_at IS NULL (или is_deleted = false). В индексах при необходимости использовать частичный индекс по этому условию.  
3. Частичный уникальный индекс: CREATE UNIQUE INDEX ... ON users (email) WHERE deleted_at IS NULL. Тогда среди активных записей email уникален, а удалённая запись с тем же email не мешает.  
4. При оптимистичной: после UPDATE с WHERE id=? AND version=? смотреть affected rows; если 0 — конфликт, перечитать и повторить или показать ошибку. При пессимистичной deadlock — БД убивает одну из транзакций; обработать ошибку и повторить (идемпотентно) или сообщить пользователю; избегать deadlock — блокировать строки в одном порядке (например, по id ASC).
</details>

**§112. Идемпотентность**

1. Почему идемпотентность важна при retry и повторной доставке сообщений?  
2. Что должен вернуть сервер при повторном запросе с тем же idempotency key?  
3. Где хранить idempotency key и результат операции — в памяти процесса или в БД? Почему?  
4. Зачем идемпотентность консьюмеру очереди сообщений (at-least-once)?

<details>
<summary>Ответы</summary>
1. При retry или повторной доставке одна и та же операция может выполниться дважды. Без идемпотентности это даёт двойной эффект (два списания, два заказа). С идемпотентностью повтор не меняет результат.  
2. Тот же результат (и тот же HTTP-код), что и при первом запросе с этим ключом — без повторного выполнения операции.  
3. В БД (таблица с UNIQUE по ключу). В памяти при перезапуске приложения ключи теряются — при повторе запроса после рестарта создастся дубликат.  
4. Очередь может доставить одно и то же сообщение дважды; консьюмер должен по ключу сообщения проверять «уже обрабатывали» и не выполнять операцию повторно, иначе два заказа или два списания.
</details>

**Вопросы по мелким пунктам (Repository, Unit of Work, граница транзакции, read replica, кэш, N+1, prepared/retry, блокировки, soft delete, idempotency)**

- **Repository:** 1) Что возвращает метод getById в типичном Repository — сущность или DTO? 2) Кто решает, один запрос или два (заголовок + строки) — репозиторий или сервис?
- **Unit of Work:** 1) Кто вызывает commit — сервис или Unit of Work? 2) Нужен ли Unit of Work, если у вас только один repository.save() в операции?
- **Граница транзакции:** 1) Где правильнее открывать транзакцию — в контроллере или в сервисе? 2) Почему «чтение → HTTP-вызов → запись» внутри одной транзакции — плохо?
- **Read replica и кэш:** 1) После записи в primary когда реплика «догонит»? 2) Write-through и write-behind — где данные теряются при сбое?
- **N+1 и индексы:** 1) Как быстро проверить по логам, что у вас N+1? 2) Почему отсутствие индекса по полю в WHERE может дать «внезапное» торможение при росте таблицы?
- **Prepared, retry, circuit breaker:** 1) Зачем prepared statement, если параметры приходят из своей таблицы, а не от пользователя? 2) После скольких ошибок подряд разумно «открыть» circuit breaker и не слать запросы в БД?
- **Блокировки (оптимистичная/пессимистичная):** 1) Что делать, если UPDATE ... WHERE id=? AND version=? вернул 0 строк? 2) В каком порядке блокировать строки, чтобы снизить риск deadlock?
- **Soft delete:** 1) Как в уникальном индексе по email учесть soft delete, чтобы удалённый пользователь не мешал создать нового с тем же email? 2) Нужно ли менять все существующие SELECT при введении soft delete?
- **Idempotency key:** 1) Где хранить пару (idempotency_key, результат) — в памяти приложения или в БД? 2) Что вернуть клиенту при повторном POST с тем же idempotency key?

<details>
<summary>Ответы по мелким пунктам</summary>
**Repository:** 1) Обычно сущность домена (или опционально DTO — зависит от архитектуры). 2) Репозиторий инкапсулирует способ доступа; сервис вызывает, например, orderRepo.getWithLines(id), а репозиторий решает — один запрос с JOIN или два.  
**Unit of Work:** 1) Обычно сервис (или слой оркестрации) вызывает uow.commit() после всех изменений. 2) Не нужен — достаточно одной транзакции и одного save().  
**Граница транзакции:** 1) В сервисе: один метод сервиса = одна операция = одна транзакция. 2) Транзакция живёт секунды, держит блокировки, при падении после HTTP — неясное состояние; HTTP вне транзакции.  
**Read replica и кэш:** 1) Не гарантировано сразу; есть lag (секунды и больше). 2) При write-behind — данные в очереди до сброса в БД могут потеряться; при write-through пишем в БД сразу, потери только при сбое БД.  
**N+1 и индексы:** 1) Один и тот же запрос (по шаблону) много раз подряд в логах/профиле. 2) На малой таблице full scan незаметен; при росте строк время растёт, запрос «внезапно» тормозит.  
**Prepared, retry, circuit breaker:** 1) Защита от инъекций и кэш планов; «свои» данные тоже могут содержать кавычки/символы. 2) Обычно 5–10 подряд — потом открыть и не слать какое-то время (полминуты и т.п.).  
**Блокировки:** 1) Кто-то обновил запись (version изменился) — конфликт; перечитать, показать пользователю или повторить. 2) В одном порядке везде (например, по id ASC), чтобы не было циклического ожидания.  
**Soft delete:** 1) Частичный уникальный индекс: UNIQUE (email) WHERE deleted_at IS NULL. 2) Да — во всех выборках добавить условие deleted_at IS NULL (или аналог).  
**Idempotency key:** 1) В БД (с UNIQUE по ключу), иначе при рестарте приложения повторы дадут дубликаты. 2) Тот же результат и код ответа, что при первом запросе с этим ключом, без повторного выполнения.
</details>

**Справочник по части, итоговая карта, шпаргалка, практикум, чек-лист, резюме**

1. По справочнику: какие четыре паттерна доступа к данным перечислены в части и в чём кратко суть каждого?  
2. По итоговой карте: какой первый шаг диагностики, если «много запросов к БД на один экран»? Если «двойной заказ при повторе»?  
3. По практикуму: в чём главная идея исправления кода pay_order (вынести HTTP из транзакции)? В кейсе place_order зачем get_by_id_for_update?  
4. По чек-листу и код-ревью: назови три вещи из «что проверить перед код-ревью» по работе с БД.  
5. По резюме: одной фразой — что делать с транзакциями? С N+1? С идемпотентностью для платежей?

<details>
<summary>Ответы</summary>
1. Repository (слой «дай/сохрани», внутри SQL), Unit of Work (корзина изменений, один commit), Active Record (объект сам save/find), Data Mapper (объект не знает про БД, маппер переводит).  
2. Много запросов — проверить логи/профиль, заменить N+1 на JOIN или batch. Двойной заказ — ввести idempotency key, хранить результат по ключу в БД, при повторе возвращать сохранённый ответ.  
3. HTTP (платёжный шлюз) не должен быть внутри транзакции — иначе она живёт секунды. get_by_id_for_update — пессимистичная блокировка строки товара, чтобы не продать больше остатка при параллельных заказах.  
4. Примеры: SQL только через плейсхолдеры; транзакция для нескольких связанных изменений; нет HTTP внутри транзакции; нет N+1; индексы под WHERE/ORDER BY; идемпотентность при retry для денег/заказов; version или блокировка при одновременном изменении; soft delete при необходимости восстановления.  
5. Транзакции: одна операция = одна транзакция, короткая, без HTTP внутри. N+1: один JOIN или batch, не N запросов в цикле. Идемпотентность для платежей: idempotency key от клиента, хранить результат в БД, при повторе отдавать сохранённый.
</details>

## Итоговая карта: симптом → причина → что проверить

| Симптом | Вероятная причина | Что проверить первым |
|--------|--------------------|----------------------|
| Много запросов к БД на один экран | N+1 (один запрос + N в цикле) | Логи/профиль запросов; заменить на JOIN или batch load |
| Частичное состояние после сбоя | Нет транзакции на атомарную операцию | Граница транзакции: одна операция = одна транзакция |
| Медленные запросы по условию | Нет индекса по полю в WHERE/ORDER BY | EXPLAIN; добавить индекс по условиям и ключам |
| Двойное списание / двойной заказ при повторе | Операция не идемпотентна | Идемпотентный ключ; idempotency key в API; хранение результата по ключу |
| Конфликт при одновременном изменении записи | Нет версионирования или блокировки | Оптимистичная блокировка (version) или пессимистичная (SELECT FOR UPDATE) |
| «Висящие» запросы к БД | Нет таймаутов или circuit breaker | statement_timeout; connection timeout; circuit breaker при серии ошибок |
| Исчерпание соединений под нагрузкой | Нет пула или слишком большой пул | Пулы соединений; размер пула ≤ лимит БД с учётом других приложений |
| Долгие блокировки и bloat | Длинные транзакции | Граница транзакции: только чтение + решение + запись; вынести долгие вычисления и внешние вызовы за пределы транзакции |

**Проверь себя (Итоговая карта: симптом → причина → что проверить)**

1. Какой первый шаг диагностики, если на один экран уходит «много запросов к БД»?  
2. Какой симптом указывает на отсутствие идемпотентности? Что проверить первым?  
3. «Висящие» запросы к БД — что проверить по настройкам и коду?

<details>
<summary>Ответы</summary>
1. Логи/профиль запросов; заменить N+1 на JOIN или batch load.  
2. Двойное списание / двойной заказ при повторе. Проверить: идемпотентный ключ; idempotency key в API; хранение результата по ключу.  
3. statement_timeout; connection timeout; circuit breaker при серии ошибок.
</details>

1. Какие четыре паттерна доступа к данным в §106 и в чём суть каждого одной фразой?  
2. Что из §110 обязательно при работе с БД (подготовленные выражения, пулы, таймауты и т.д.)?  
3. Как в §111 добиться «не перезаписать чужое» и «не потерять историю»?

<details>
<summary>Ответы</summary>
1. Repository — абстракция над БД; Unit of Work — одна транзакция на набор изменений; Active Record — объект = строка + save; Data Mapper — домен отдельно от БД.  
2. Prepared statements; пулы соединений; таймауты; retry с backoff; circuit breaker.  
3. Оптимистичная блокировка (version) или пессимистичная (SELECT FOR UPDATE); soft delete (deleted_at) для возможности восстановления.
</details>

## Что положить в шпаргалку (один файл или закладки)

**Оптимистичная блокировка (PostgreSQL):**

```sql
-- Таблица с version
ALTER TABLE orders ADD COLUMN version INTEGER NOT NULL DEFAULT 0;

-- Обновление с проверкой версии
UPDATE orders SET total = 100, version = version + 1, updated_at = NOW()
WHERE id = 5 AND version = 3;
-- Проверить: affected rows = 0 → конфликт, перечитать и повторить
```

**Пессимистичная блокировка (PostgreSQL):**

```sql
BEGIN;
SELECT * FROM orders WHERE id = 5 FOR UPDATE;
-- ... логика в приложении ...
UPDATE orders SET total = 100 WHERE id = 5;
COMMIT;
```

**Soft delete (PostgreSQL):**

```sql
-- Удаление
UPDATE users SET deleted_at = NOW() WHERE id = 5;

-- Выборки
SELECT * FROM users WHERE deleted_at IS NULL;

-- Уникальность только для активных
CREATE UNIQUE INDEX users_email_key ON users (email) WHERE deleted_at IS NULL;
```

**Idempotency key (таблица для хранения результата):**

```sql
CREATE TABLE idempotency_keys (
  key        VARCHAR(255) PRIMARY KEY,
  response   JSONB NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

**Правила:** один запрос — одна транзакция; короткие транзакции; prepared statements для всех запросов с данными; пулы соединений и таймауты; retry только для идемпотентных операций; N+1 устранять JOIN или batch load.

**Проверь себя (Что положить в шпаргалку)**

1. В шпаргалке по части есть UPDATE с version. Для чего нужна проверка «affected rows = 0» после такого UPDATE?  
2. Как в шпаргалке задаётся уникальность email только для неудалённых пользователей (soft delete)?  
3. Где в шпаргалке хранить результат по idempotency key — в какой структуре и зачем не в памяти приложения?

<details>
<summary>Ответы</summary>
1. affected rows = 0 означает, что запись успел изменить кто-то другой (version изменился) — конфликт оптимистичной блокировки; перечитать и повторить или показать ошибку.  
2. Частичный уникальный индекс: CREATE UNIQUE INDEX ... ON users (email) WHERE deleted_at IS NULL.  
3. В таблице (например idempotency_keys: key, response, created_at). В памяти при рестарте приложения ключи теряются — при повторе после рестарта создастся дубликат.
</details>

## Резюме части XXI (если забыл всё — прочитай это)

**Одним абзацем.** В этой части мы учимся организовывать код и работу с БД так, чтобы система была предсказуемой и устойчивой: не размазывать SQL по сервисам (Repository, Unit of Work, Active Record, Data Mapper), держать транзакции короткими и с явными границами, масштабировать чтение (replicas, кэш, CQRS, event sourcing), избегать антипаттернов (N+1, отсутствие транзакций и индексов, длинные транзакции), применять базовые практики (prepared statements, пулы, таймауты, retry, circuit breaker), версионировать данные (оптимистичная/пессимистичная блокировка, soft delete) и обеспечивать идемпотентность (идемпотентные ключи, idempotency key в API). Каждая тема в разделе повторяется несколько раз: «главное» → «простыми словами» → «картинка в голове» → «что будет, если не делать» → «проверь себя» → «запомни». Если что-то забыл — вернись к «Одной фразой по разделам» и к шпаргалке в начале части.

**Проверь себя (Резюме части XXI)**

1. Одной фразой: что делать с транзакциями в приложении? С N+1?  
2. Что положить в шпаргалку по идемпотентности для API (ключ, хранение, повтор)?  
3. С чем из других частей курса связана эта часть (транзакции, оптимизация, ORM, масштабирование, безопасность)?

<details>
<summary>Ответы</summary>
1. Транзакции: одну бизнес-операцию делать в одной транзакции, держать её короткой, без HTTP и долгих расчётов внутри. N+1: списки с связями — один JOIN или batch, не N запросов в цикле.  
2. Idempotency key от клиента; хранить результат по ключу в БД (не в памяти); при повторе возвращать сохранённый ответ.  
3. Транзакции (часть IV), оптимизация запросов (часть VII), ORM и пулы (часть VIII), масштабирование (часть XI), безопасность (часть XII).
</details>

Часть посвящена **разработке и паттернам работы с БД**: как организовать доступ к данным (Repository, Unit of Work, Active Record, Data Mapper), как управлять транзакциями (транзакционный скрипт, доменная модель, короткие границы), как масштабировать (read replicas, кэш, CQRS, event sourcing), какие антипаттерны избегать (N+1, отсутствие транзакций и индексов, длинные транзакции), какие практики применять (prepared statements, пулы, таймауты, retry, circuit breaker), как версионировать данные (оптимистичная/пессимистичная блокировка, soft delete) и как обеспечивать идемпотентность (идемпотентные ключи, idempotency key в API). Эти темы опираются на транзакции (часть IV), оптимизацию запросов (часть VII), ORM и пулы (часть VIII), масштабирование (часть XI) и безопасность (часть XII).

#### Как всё из части складывается в одну систему (одна картинка)

Чтобы не теряться в «куче тем», держи в голове одну цепочку. **Как мы храним и достаём данные** (§106) — репозиторий, Unit of Work, Active Record, Data Mapper. **В каких границах меняем данные** (§107) — одна операция = одна транзакция, короткая, без HTTP внутри. **Как разгружаем и масштабируем** (§108) — реплики, кэш, CQRS, event sourcing; помним про lag и потери при write-behind. **Чего не делаем** (§109) — N+1, отсутствие транзакций и индексов, SELECT *, длинные транзакции. **Что делаем обязательно** (§110) — prepared statements, пулы, таймауты, retry только для идемпотентного, circuit breaker. **Как не перезаписать чужое и не потерять историю** (§111) — оптимистичная/пессимистичная блокировка, soft delete. **Как не создать дубликат при повторе запроса** (§112) — идемпотентность, idempotency key, хранение результата по ключу в БД. Вместе это даёт слой «как приложение разговаривает с БД»: безопасно, предсказуемо и устойчиво к сбоям и повторам.

**Одна фраза — одно действие (что делать по разделам, если забыл всё):**

- **§106:** Не пиши SQL в сервисах — вынеси в репозиторий (или Active Record); если одна операция трогает много сущностей — используй Unit of Work и один commit.
- **§107:** Одну бизнес-операцию делай в одной транзакции; транзакцию держи короткой, без HTTP и долгих расчётов внутри.
- **§108:** Чтение с реплики — помни про lag; после записи «прочитать своё» — с primary; write-behind только для некритичных данных.
- **§109:** Списки с связями — один JOIN или batch, не N запросов в цикле; несколько связанных изменений — одна транзакция; медленные запросы — смотри план и индексы.
- **§110:** Запросы — только prepared statements; соединения — через пул; поставь таймауты; retry только для идемпотентного; при падении БД — circuit breaker.
- **§111:** Одновременное изменение одной записи — version (оптимистичная) или SELECT FOR UPDATE (пессимистичная); удаление «без потери» — soft delete (deleted_at).
- **§112:** Операции, меняющие данные (заказ, платёж), делай идемпотентными: idempotency key от клиента, храни результат по ключу в БД, при повторе возвращай сохранённый ответ.

**Повтори одной фразой (после прочтения части — что сказать себе вслух):** §106 — доступ к данным через один слой (репозиторий или Active Record), много сущностей в одной операции — Unit of Work и один коммит. §107 — одна операция = одна транзакция, короткая, без HTTP внутри. §108 — реплика отстаёт; после записи «своё» читать с primary; write-behind только для некритичного. §109 — не N+1 (JOIN или batch), не без транзакции, не без индексов, не SELECT *, не длинная транзакция. §110 — prepared statements, пул, таймауты, retry только для идемпотентного, circuit breaker. §111 — при одновременном изменении — version или SELECT FOR UPDATE; удаление — soft delete. §112 — повтор запроса не должен дублировать эффект: idempotency key, хранить результат в БД, при повторе отдавать сохранённый.
