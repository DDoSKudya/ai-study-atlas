## Справочник по части II

**Как пользоваться справочником.** Ниже собраны сводные таблицы и короткие синтаксические шпаргалки по типам данных, DDL, DML, SELECT и агрегации. Используй их для быстрого повторения после изучения разделов: нашёл тему по заголовку, вспомнил синтаксис или выбор типа. Для глубокого понимания опирайся на основные разделы с примерами и блоки «Запомните».

**Когда какой тип (кратко):** целое для счётчиков/ID — INTEGER или BIGINT; деньги — NUMERIC(p,s); текст — TEXT или VARCHAR(n); момент времени с поясом — TIMESTAMPTZ; дата без времени — DATE; да/нет — BOOLEAN; гибкая структура (атрибуты, настройки) — JSONB; список значений одного типа (теги и т.п.) — массив (TEXT[]); фиксированный набор строк (статус) — ENUM или TEXT + CHECK.

### Шпаргалка DDL

```sql
-- Создать таблицу
CREATE TABLE IF NOT EXISTS tablename (
    col  type  [NOT NULL] [DEFAULT val] [constraints],
    ...,
    CONSTRAINT name PRIMARY KEY (col),
    CONSTRAINT name UNIQUE (col1, col2),
    CONSTRAINT name CHECK (expr),
    CONSTRAINT name FOREIGN KEY (col) REFERENCES other(col)
        ON DELETE CASCADE ON UPDATE CASCADE
);

-- Изменить таблицу
ALTER TABLE t ADD COLUMN col type;
ALTER TABLE t DROP COLUMN col;
ALTER TABLE t ALTER COLUMN col TYPE newtype USING expr;
ALTER TABLE t ALTER COLUMN col SET NOT NULL;
ALTER TABLE t ALTER COLUMN col DROP NOT NULL;
ALTER TABLE t ADD CONSTRAINT name CHECK (expr);
ALTER TABLE t DROP CONSTRAINT name;
ALTER TABLE t RENAME TO newname;
ALTER TABLE t RENAME COLUMN old TO new;

-- Удалить / очистить
DROP TABLE IF EXISTS t [CASCADE];
TRUNCATE TABLE t [RESTART IDENTITY] [CASCADE];
```

### Шпаргалка DML

```sql
-- Вставка
INSERT INTO t (col1, col2) VALUES (v1, v2);
INSERT INTO t (col1, col2) VALUES (v1, v2), (v3, v4);
INSERT INTO t (col1) SELECT col1 FROM other;
INSERT INTO t (col1, col2) VALUES (v1, v2)
    ON CONFLICT (col1) DO UPDATE SET col2 = EXCLUDED.col2;
INSERT INTO t (col1) VALUES (v1) RETURNING id;

-- Обновление
UPDATE t SET col = val WHERE cond;
UPDATE t SET col = val FROM other WHERE t.id = other.tid RETURNING *;

-- Удаление
DELETE FROM t WHERE cond;
DELETE FROM t USING other WHERE t.id = other.tid RETURNING *;

-- COPY
COPY t (col1, col2) FROM '/path/file.csv' CSV HEADER;
COPY t TO '/path/out.csv' CSV HEADER;
```

### Шпаргалка SELECT

```sql
SELECT [DISTINCT | DISTINCT ON (expr)]
    col [AS alias],
    expr [AS alias],
    agg_func(col) [FILTER (WHERE cond)] [AS alias],
    ...
FROM   table [AS alias]
WHERE  condition
GROUP BY col | expr
HAVING agg_condition
ORDER BY col [ASC|DESC] [NULLS FIRST|LAST], ...
LIMIT  n
OFFSET m;
```

## Типичные ошибки

Ниже — самые частые ошибки на уровне основ SQL. Перед переходом к Части III полезно пройтись по таблице и убедиться, что ты можешь **своими словами** объяснить, почему каждая строка «плохо» и как сделать правильно. Если какую-то строку не понимаешь — вернись к соответствующему разделу (номера разделов можно найти по ключевым словам в оглавлении).

**Самые опасные (могут привести к потере данных или денег):** UPDATE/DELETE без WHERE (можно затереть всю таблицу); FLOAT для денег (накопление ошибок округления); TRUNCATE вместо DELETE, когда нужна фильтрация (TRUNCATE очищает всё). Привыкай всегда проверять WHERE через SELECT перед UPDATE/DELETE и смотреть, сколько строк вернулось. После выполнения UPDATE/DELETE СУБД обычно сообщает «UPDATE 3» или «DELETE 15» — если ожидал одну строку, а обновилось 1000, сразу откатывай транзакцию (ROLLBACK) и разбирайся.

| Ошибка | Почему плохо | Как правильно |
|--------|--------------|---------------|
| `SELECT *` в продакшне | Возвращает все столбцы, может включать секретные данные, ломается при изменении схемы (добавили колонку — приложение получило лишнее поле и может упасть) | Явно перечисляй нужные столбцы: `SELECT id, name, email FROM users`. В коде обращайся только к ним |
| `UPDATE/DELETE` без `WHERE` | Изменяет/удаляет все строки | Всегда добавляй `WHERE`; проверяй через `SELECT` с тем же условием |
| `FLOAT`/`DOUBLE` для денег | Потеря точности при округлении | Используй `NUMERIC(p, s)` |
| `TIMESTAMP` вместо `TIMESTAMPTZ` | Проблемы при смене TZ сервера | Используй `TIMESTAMPTZ` |
| Нет индекса на FK-столбце | При DELETE из родителя — Seq Scan по всей дочерней таблице | Создавай индекс на FK-столбцах |
| `CHAR(n)` для строк | Дополняется пробелами, неожиданные сравнения | Используй `TEXT` или `VARCHAR(n)` |
| Условие по агрегату в `WHERE` | Синтаксическая ошибка | Перенеси в `HAVING` |
| Псевдоним из SELECT в WHERE | Псевдоним ещё не определён | Повтори выражение или используй подзапрос |
| `WHERE col = NULL` | В SQL `NULL = NULL` даёт не TRUE, а NULL; условие никогда не выполняется | Используй `WHERE col IS NULL` или `WHERE col IS NOT NULL` |
| `ENUM` при частых изменениях | Нельзя удалить значение без пересоздания типа | Используй `TEXT` + `CHECK` ограничение |
| `OFFSET 100000` для пагинации | СУБД читает 100010 строк, выбрасывает первые 100000 | Используй keyset-пагинацию по ID |
| Нет `IF NOT EXISTS` в DDL-скриптах | Скрипт падает при повторном запуске | Добавляй `IF NOT EXISTS` |
| Разрыв в ID как сигнал об ошибке | Sequence нетранзакционна — разрывы нормальны | Не строй логику на непрерывности ID |

## Вопросы для самопроверки

Ответь на вопросы **своими словами** (можно вслух или письменно), затем открой ответ под спойлером и сверься. Если не смог сформулировать — перечитай соответствующий раздел и блок «Запомните» в конце темы. Вопросы сгруппированы по разделам; в скобках указан раздел и тема.

---

### 5.1. Типы данных

**1. Какой тип данных использовать для хранения суммы транзакции в рублях с точностью до копейки?** *(Раздел 5.1 — числовые типы.)*

<details>
<summary>Ответ</summary>

`NUMERIC(15, 2)` — точный тип с фиксированной десятичной частью. Два знака после запятой для копеек. Никогда не используй `REAL` или `DOUBLE PRECISION` для денег — будет накапливаться ошибка округления.

</details>

**2. Чем `TIMESTAMPTZ` отличается от `TIMESTAMP`?** *(Раздел 5.1 — типы даты и времени.)*

<details>
<summary>Ответ</summary>

`TIMESTAMPTZ` хранит момент времени в UTC и при чтении показывает его в часовом поясе сессии. `TIMESTAMP` хранит дату/время без привязки к поясу — при смене TZ сервера значения «сдвигаются». Для моментов времени (создано, событие) используй `TIMESTAMPTZ`.

</details>

**3. Зачем нужен домен (CREATE DOMAIN) и чем он отличается от просто типа с CHECK в таблице?** *(Раздел 5.1 — домен.)*

<details>
<summary>Ответ</summary>

Домен — именованный тип с встроенным ограничением (например, «сумма ≥ 0»). Его можно использовать в нескольких таблицах; правило задаётся в одном месте. Отдельный CHECK в каждой таблице пришлось бы дублировать и менять в нескольких местах.

</details>

---

### 5.2. CREATE TABLE

**4. В чём разница между inline-ограничением и table-level ограничением?** *(Раздел 5.2 — CREATE TABLE.)*

<details>
<summary>Ответ</summary>

Inline — пишется рядом с колонкой: `col TYPE CONSTRAINT name ...`. Table-level — отдельной строкой в теле таблицы: `CONSTRAINT name PRIMARY KEY (col)` или `CONSTRAINT name CHECK (...)`. Составные ключи (несколько колонок) задаются только table-level.

</details>

**5. Можно ли в одной таблице объявить два PRIMARY KEY?** *(Раздел 5.2 — ограничения.)*

<details>
<summary>Ответ</summary>

Нет. В таблице может быть только один первичный ключ. Если нужна уникальность по другому набору столбцов — используй ограничение UNIQUE.

</details>

**6. Зачем давать ограничениям явные имена (CONSTRAINT name)?** *(Раздел 5.2 — именование ограничений.)*

<details>
<summary>Ответ</summary>

Чтобы при ошибке СУБД выводила понятное имя (например, `chk_positive_price`), а не сгенерированное. Так проще искать ограничение в коде и в сообщениях об ошибках; удобно при ALTER TABLE DROP CONSTRAINT.

</details>

---

### 5.3. ALTER TABLE, DROP, TRUNCATE

**7. В чём разница между `DELETE FROM t` и `TRUNCATE t`?** *(Раздел 5.3 — TRUNCATE vs DELETE.)*

<details>
<summary>Ответ</summary>

`DELETE FROM t` — DML: удаляет строки по одной, пишет в WAL каждую, поддерживает WHERE, запускает ROW-триггеры. `TRUNCATE t` — DDL: очищает таблицу «оптом», без per-row WAL, без триггеров, быстрее. TRUNCATE не позволяет фильтровать — удаляет всё.

</details>

**8. Что делать, если нужно добавить колонку NOT NULL в таблицу, где уже есть строки?** *(Раздел 5.3 — ALTER TABLE с данными.)*

<details>
<summary>Ответ</summary>

Сначала добавить колонку как nullable или с DEFAULT; затем заполнить пропуски (UPDATE ... SET col = value WHERE col IS NULL); затем ALTER COLUMN ... SET NOT NULL. Иначе СУБД не знает, какое значение подставить существующим строкам, и выдаст ошибку.

</details>

**9. Когда использовать ALTER COLUMN TYPE и зачем там USING?** *(Раздел 5.3 — изменение типа.)*

<details>
<summary>Ответ</summary>

ALTER COLUMN ... TYPE new_type меняет тип колонки. Если неявное приведение невозможно (например, TEXT → INTEGER), нужно указать USING expression — как вычислить новое значение из старого (например, col::INTEGER или length(col)).

</details>

---

### 5.4. Временные таблицы

**10. Чем временная таблица (TEMP) отличается от обычной?** *(Раздел 5.4 — временные таблицы.)*

<details>
<summary>Ответ</summary>

Временная таблица видна только в текущей сессии; после отключения она исчезает (или в конце транзакции при ON COMMIT DROP). Данные не видны другим сессиям. Обычная таблица общая для всех и хранится постоянно.

</details>

**11. Что делает ON COMMIT DROP ROWS у временной таблицы?** *(Раздел 5.4 — ON COMMIT.)*

<details>
<summary>Ответ</summary>

В конце каждой транзакции (COMMIT) все строки из временной таблицы удаляются; сама таблица остаётся. Удобно для промежуточных данных внутри одной транзакции без ручного DELETE.

</details>

### 6.1. PRIMARY KEY и UNIQUE

**12. Может ли UNIQUE-ограничение содержать несколько NULL?** *(Раздел 6.1 — NULL в UNIQUE.)*

<details>
<summary>Ответ</summary>

Да. В SQL NULL не равен NULL, поэтому несколько строк с NULL в уникальном столбце не считаются дубликатами. Чтобы запретить NULL, добавь отдельно NOT NULL.

</details>

**13. Чем составной PRIMARY KEY (a, b) отличается от двух отдельных UNIQUE на a и b?** *(Раздел 6.1 — составной ключ.)*

<details>
<summary>Ответ</summary>

Составной PK (a, b) требует уникальности **пары** (a, b): могут быть строки с одинаковым a при разном b и наоборот. Два отдельных UNIQUE требуют уникальности a по всей таблице и b по всей таблице — то есть и a, и b по отдельности не повторяются.

</details>

**14. Создаёт ли PRIMARY KEY индекс?** *(Раздел 6.1 — PK и индексы.)*

<details>
<summary>Ответ</summary>

Да. В PostgreSQL (и в большинстве СУБД) при создании PRIMARY KEY и UNIQUE автоматически создаётся уникальный B-tree индекс по этим столбцам для быстрого поиска и проверки уникальности.

</details>

---

### 6.2. NOT NULL и CHECK

**15. Проходит ли строка CHECK (price >= 0), если в price записан NULL?** *(Раздел 6.2 — CHECK и NULL.)*

<details>
<summary>Ответ</summary>

Да. В SQL сравнение с NULL даёт NULL (не TRUE и не FALSE). CHECK отсекает только строки, где выражение равно FALSE. NULL не FALSE — строка проходит. Чтобы запретить и NULL, нужен отдельно NOT NULL.

</details>

**16. Зачем именовать ограничение CHECK (CONSTRAINT name CHECK (...))?** *(Раздел 6.2 — именованные ограничения.)*

<details>
<summary>Ответ</summary>

Чтобы в сообщении об ошибке видеть понятное имя (например, chk_price_positive), а не автосгенерированное. Так проще искать в коде и при отладке; удобно для DROP CONSTRAINT и при миграциях.

</details>

**17. Можно ли в одном CHECK обратиться к другой колонке той же строки?** *(Раздел 6.2 — CHECK.)*

<details>
<summary>Ответ</summary>

Да. CHECK может содержать выражение с несколькими столбцами таблицы, например CHECK (date_end > date_start) или CHECK (min_val <= max_val). Проверка выполняется для каждой строки при вставке и обновлении.

</details>

---

### 6.3. FOREIGN KEY и каскады

**18. Кто «родитель», кто «дочерняя» таблица во внешнем ключе?** *(Раздел 6.3 — родитель и дочерняя.)*

<details>
<summary>Ответ</summary>

Таблица, на которую ссылаются (REFERENCES) — **родитель**. Таблица, в которой объявлен столбец с FOREIGN KEY — **дочерняя**. Дочерняя хранит значения, которые должны существовать в родителе (например, order.user_id → users.id).

</details>

**19. Что делают ON DELETE CASCADE и ON DELETE SET NULL?** *(Раздел 6.3 — каскады.)*

<details>
<summary>Ответ</summary>

ON DELETE CASCADE: при удалении строки в родителе СУБД автоматически удаляет все дочерние строки, ссылающиеся на неё. ON DELETE SET NULL: при удалении родителя значение FK в дочерних строках устанавливается в NULL (колонка должна допускать NULL).

</details>

**20. Зачем создавать индекс на столбце с FOREIGN KEY?** *(Раздел 6.3 — индекс на FK.)*

<details>
<summary>Ответ</summary>

При удалении или обновлении строки в родителе СУБД должна проверить, есть ли дочерние строки. Без индекса по FK-столбцу будет полный перебор дочерней таблицы (Seq Scan). С индексом проверка выполняется быстро.

</details>

---

### 6.4. DEFERRABLE

**21. Когда нужен DEFERRABLE INITIALLY DEFERRED для FOREIGN KEY?** *(Раздел 6.4 — DEFERRABLE.)*

<details>
<summary>Ответ</summary>

Когда вставляешь данные с циклическими или иерархическими ссылками (например, parent_id в той же таблице): без отложенной проверки нельзя вставить дочернюю запись до родительской. DEFERRED откладывает проверку до COMMIT.

</details>

**22. Как временно отложить проверку ограничения внутри транзакции?** *(Раздел 6.4 — SET CONSTRAINTS.)*

<details>
<summary>Ответ</summary>

В начале транзакции выполнить SET CONSTRAINTS имя_ограничения DEFERRED (или SET CONSTRAINTS ALL DEFERRED). Тогда проверка этого ограничения произойдёт при COMMIT, а не после каждого оператора.

</details>

---

### 6.5. EXCLUDE

**23. Для чего используется ограничение EXCLUDE в PostgreSQL?** *(Раздел 6.5 — EXCLUDE.)*

<details>
<summary>Ответ</summary>

Чтобы запретить две строки, для которых выполняется заданное условие (часто — перекрытие диапазонов). Например: нельзя иметь две брони одной комнаты с пересекающимися периодами (оператор && для диапазонов).

</details>

**24. Какой тип индекса нужен для EXCLUDE по диапазону и обычному столбцу (например, room_id + period)?** *(Раздел 6.5 — EXCLUDE и GiST.)*

<details>
<summary>Ответ</summary>

Используется индекс GiST (часто с расширением btree_gist для смешивания обычных столбцов и диапазонов). В определении EXCLUDE указывается USING gist (столбец_диапазон op, столбец_обычный op).

</details>

### 7.1. INSERT

**25. Зачем указывать список столбцов в INSERT (col1, col2) VALUES (...)?** *(Раздел 7.1 — INSERT.)*

<details>
<summary>Ответ</summary>

Чтобы не зависеть от порядка колонок в таблице и явно показать, какие поля заполняются. Если добавят новую колонку с DEFAULT, запрос не сломается. Для колонок с DEFAULT можно не указывать их в списке — подставится значение по умолчанию.

</details>

**26. Что возвращает RETURNING после INSERT?** *(Раздел 7.1 — RETURNING.)*

<details>
<summary>Ответ</summary>

RETURNING возвращает строки (и указанные столбцы), которые были вставлены — например, сгенерированные id, значения по умолчанию, выражения. Удобно получить id только что вставленной строки без отдельного запроса.

</details>

**27. Когда использовать INSERT ... SELECT вместо нескольких INSERT ... VALUES?** *(Раздел 7.1 — INSERT SELECT.)*

<details>
<summary>Ответ</summary>

Когда нужно вставить данные из другой таблицы или из результата запроса (копирование, перенос, выборка с преобразованием). Один запрос вместо цикла в приложении; меньше обращений к БД и меньше записей в WAL при массовой вставке.

</details>

### 7.2. UPDATE и DELETE

**28. Почему перед UPDATE/DELETE с WHERE полезно выполнить SELECT с тем же WHERE?** *(Раздел 7.2 — безопасность.)*

<details>
<summary>Ответ</summary>

Чтобы убедиться, что условие выбрало именно те строки, которые ты хочешь изменить или удалить. Иначе можно затронуть лишние строки или не те данные. После UPDATE/DELETE смотри на число затронутых строк — если не совпадает с ожиданием, делай ROLLBACK.

</details>

**29. Что делает UPDATE ... FROM другая_таблица WHERE t.id = другая_таблица.tid?** *(Раздел 7.2 — UPDATE FROM.)*

<details>
<summary>Ответ</summary>

Обновляет строки таблицы t на основе данных из другой таблицы: условие WHERE связывает строки (как JOIN). Удобно обновлять значения из справочника или из подзапроса без цикла в приложении.

</details>

**30. Можно ли в UPDATE или DELETE использовать псевдоним таблицы?** *(Раздел 7.2 — синтаксис.)*

<details>
<summary>Ответ</summary>

Да. В PostgreSQL для UPDATE и DELETE можно писать UPDATE table AS alias или DELETE FROM table AS alias и использовать alias в SET, WHERE и т.д. Это просто другое имя той же таблицы.

</details>

---

### 7.3. UPSERT, MERGE и RETURNING

**31. Что такое EXCLUDED в ON CONFLICT DO UPDATE SET col = EXCLUDED.col?** *(Раздел 7.3 — UPSERT.)*

<details>
<summary>Ответ</summary>

EXCLUDED — псевдотаблица с данными строки, которую не удалось вставить из-за конфликта по уникальному ключу. EXCLUDED.col — значение из «отклонённой» вставки. Так в SET подставляются новые значения при обновлении существующей строки.

</details>

**32. В чём разница между ON CONFLICT DO NOTHING и ON CONFLICT DO UPDATE?** *(Раздел 7.3 — UPSERT.)*

<details>
<summary>Ответ</summary>

DO NOTHING: при конфликте по уникальному ключу вставка для этой строки просто пропускается, ничего не меняется. DO UPDATE: при конфликте существующая строка обновляется заданными полями (часто из EXCLUDED). Первое — «вставь, если нет»; второе — полноценный «вставь или обнови».

</details>

**33. Когда предпочтительнее MERGE, а когда ON CONFLICT?** *(Раздел 7.3 — MERGE vs ON CONFLICT.)*

<details>
<summary>Ответ</summary>

ON CONFLICT проще и покрывает типичный случай «вставь или обнови по одному ключу». MERGE удобен, когда источник — подзапрос или набор строк, и нужны разные действия при MATCHED и NOT MATCHED (например, при MATCHED обновить только если изменилось). В PostgreSQL 15+ доступен MERGE.

</details>

---

### 7.4. Последовательности, SERIAL, IDENTITY и COPY

**34. Чем GENERATED ALWAYS AS IDENTITY отличается от SERIAL?** *(Раздел 7.4 — IDENTITY.)*

<details>
<summary>Ответ</summary>

SERIAL — синтаксический сахар: создаётся sequence и DEFAULT nextval(...). В столбец можно вставить значение вручную. IDENTITY — стандарт SQL: значение всегда генерируется СУБД; вставить вручную можно только через OVERRIDING SYSTEM VALUE. IDENTITY явнее и безопаснее.

</details>

**35. Почему в таблице с IDENTITY/SERIAL могут быть «дыры» в нумерации (1, 2, 5, 6...)?** *(Раздел 7.4 — последовательности.)*

<details>
<summary>Ответ</summary>

Sequence не привязана к транзакциям: nextval выдаёт значение и «съедает» его даже если транзакция откатилась. При откате или при параллельных вставках номера не переиспользуются. Это нормальное поведение; не строй логику на непрерывности id.

</details>

**36. В чём разница между COPY и \COPY в psql?** *(Раздел 7.4 — COPY.)*

<details>
<summary>Ответ</summary>

COPY выполняется сервером: файл читается/пишется на машине, где работает PostgreSQL (нужны права доступа сервера к файлу). \COPY — команда клиента psql: файл на твоей машине, psql передаёт данные серверу. Для загрузки с локального компьютера используй \COPY.

</details>

### 8.1. Базовый SELECT

**37. В каком порядке логически выполняются части запроса: FROM, WHERE, GROUP BY, HAVING, SELECT, ORDER BY, LIMIT?** *(Раздел 8.1 — порядок выполнения.)*

<details>
<summary>Ответ</summary>

FROM (источник строк) → WHERE (фильтр) → GROUP BY (группировка) → HAVING (фильтр по группам) → SELECT (вычисление столбцов и псевдонимов) → DISTINCT → ORDER BY (сортировка) → LIMIT/OFFSET. Поэтому в WHERE нельзя использовать псевдонимы из SELECT и агрегаты — они ещё не вычислены.

</details>

**38. Почему для «полного года» дат лучше писать >= '2024-01-01' AND < '2025-01-01', а не BETWEEN '2024-01-01' AND '2024-12-31'?** *(Раздел 8.1 — WHERE и даты.)*

<details>
<summary>Ответ</summary>

BETWEEN '2024-01-01' AND '2024-12-31' для типа TIMESTAMP включает только 2024-12-31 00:00:00, но не моменты в течение последнего дня. Условие с < '2025-01-01' включает весь день 31 декабря до полуночи.

</details>

**39. Что такое keyset-пагинация и чем она лучше OFFSET для больших сдвигов?** *(Раздел 8.1 — пагинация.)*

<details>
<summary>Ответ</summary>

Keyset-пагинация: вместо «пропусти N строк» используешь «дай строки, где id > последнего_увиденного_id, ORDER BY id LIMIT 10». СУБД может использовать индекс и не читать/отбрасывать тысячи строк. OFFSET 100000 заставляет прочитать и отбросить 100000 строк — медленно.

</details>

---

### 8.2. DISTINCT и DISTINCT ON

**40. Что делает DISTINCT и чем он отличается от DISTINCT ON (col)?** *(Раздел 8.2 — DISTINCT.)*

<details>
<summary>Ответ</summary>

DISTINCT убирает полностью одинаковые строки результата (все столбцы совпадают); остаётся по одной строке на каждый уникальный набор значений. DISTINCT ON (col) оставляет по одной строке на каждое уникальное значение col (какую именно — зависит от порядка; задаётся ORDER BY, причём первым должен быть тот же col).

</details>

**41. Почему при использовании DISTINCT ON (a) в запросе должен быть ORDER BY, начинающийся с a?** *(Раздел 8.2 — DISTINCT ON.)*

<details>
<summary>Ответ</summary>

Потому что из каждой группы строк с одинаковым a нужно выбрать одну — «первую». Какая считается первой, определяет ORDER BY. Если ORDER BY не начинается с a, порядок внутри группы не определён и результат непредсказуем; в PostgreSQL такой запрос выдаст ошибку.

</details>

---

### 8.3. Псевдонимы и выражения

**42. Почему нельзя использовать псевдоним из SELECT в условии WHERE?** *(Раздел 8.3 — псевдонимы.)*

<details>
<summary>Ответ</summary>

Логически WHERE выполняется до этапа SELECT. На этапе WHERE псевдонимы из SELECT ещё не определены — они вычисляются позже. Поэтому в WHERE нужно писать исходное выражение или имя столбца из таблицы.

</details>

**43. В чём разница между простым CASE (CASE x WHEN a THEN ...) и поисковым CASE (CASE WHEN условие THEN ...)?** *(Раздел 8.3 — CASE.)*

<details>
<summary>Ответ</summary>

Простой CASE сравнивает одно выражение с несколькими значениями: CASE col WHEN 'a' THEN 1 WHEN 'b' THEN 2. Поисковый CASE проверяет несколько независимых условий: CASE WHEN col > 0 THEN 1 WHEN col < 0 THEN -1. Для сложных условий (диапазоны, несколько столбцов) нужен поисковый CASE.

</details>

**44. Можно ли в SELECT использовать агрегатную функцию без GROUP BY?** *(Раздел 8.3 / 9.1 — агрегаты.)*

<details>
<summary>Ответ</summary>

Да. Если в запросе есть только агрегатные функции и нет обычных столбцов (или они все в агрегатах), GROUP BY не нужен: вся таблица (после WHERE) считается одной группой. Например, SELECT COUNT(*) FROM t даёт одно число — общее количество строк.

</details>

---

### 8.4. Сортировка с NULL

**45. Где по умолчанию оказываются NULL при ORDER BY col ASC и DESC в PostgreSQL?** *(Раздел 8.4 — NULLS FIRST/LAST.)*

<details>
<summary>Ответ</summary>

При ORDER BY col ASC по умолчанию NULL считаются «больше» любого значения — идут в конце (NULLS LAST). При ORDER BY col DESC по умолчанию NULL идут в начале (NULLS FIRST). Явно задать можно: NULLS FIRST или NULLS LAST после ASC/DESC.

</details>

**46. Зачем явно указывать NULLS FIRST или NULLS LAST?** *(Раздел 8.4 — сортировка.)*

<details>
<summary>Ответ</summary>

Чтобы не зависеть от реализации по умолчанию и сделать результат предсказуемым (например, «сначала неизвестные, потом по возрастанию» или «в конце списка»). Важно при пагинации и при единообразии между разными СУБД.

</details>

### 9.1. GROUP BY и агрегатные функции

**47. Что вернёт COUNT(col), если в col есть NULL?** *(Раздел 9.1 — COUNT.)*

<details>
<summary>Ответ</summary>

COUNT(col) считает только строки, где col IS NOT NULL. Строки с NULL в col не учитываются. Чтобы посчитать все строки группы, используй COUNT(*) или COUNT(1).

</details>

**48. Какое правило ограничивает список столбцов в SELECT при наличии GROUP BY?** *(Раздел 9.1 — GROUP BY.)*

<details>
<summary>Ответ</summary>

В SELECT можно выводить только: (1) столбцы, перечисленные в GROUP BY (или их части, если группировка по выражению), и (2) агрегатные функции от любых столбцов. Любой другой столбец без агрегата приведёт к ошибке — для него не определено единственное значение по группе.

</details>

**49. Чем GROUP BY a, b отличается от GROUP BY a и GROUP BY b по отдельности в двух запросах?** *(Раздел 9.1 — группировка.)*

<details>
<summary>Ответ</summary>

GROUP BY a, b даёт одну строку на каждую уникальную **пару** (a, b). Два отдельных запроса с GROUP BY a и GROUP BY b дадут два набора строк: в одном — по одному значению на каждое a, в другом — на каждое b. Это разные уровни детализации; объединить их в один результат можно через GROUPING SETS.

</details>

---

### 9.2. HAVING

**50. Почему условие по агрегату (например, COUNT(*) > 1) нельзя писать в WHERE?** *(Раздел 9.2 — HAVING.)*

<details>
<summary>Ответ</summary>

WHERE выполняется до группировки: на этом этапе агрегаты ещё не вычислены, строки не сгруппированы. Условие по результату агрегации нужно ставить в HAVING, который выполняется после GROUP BY и «видит» агрегатные значения по каждой группе.

</details>

**51. Может ли в одном запросе быть и WHERE, и HAVING?** *(Раздел 9.2 — WHERE и HAVING.)*

<details>
<summary>Ответ</summary>

Да. WHERE отфильтровывает строки до группировки (по «сырым» данным). HAVING отфильтровывает группы после группировки (по агрегатам и столбцам из GROUP BY). Сначала применяется WHERE, потом GROUP BY, потом HAVING.

</details>

---

### 9.3. FILTER в агрегатах

**52. Что делает COUNT(*) FILTER (WHERE status = 'paid')?** *(Раздел 9.3 — FILTER.)*

<details>
<summary>Ответ</summary>

Считает только те строки группы, для которых status = 'paid'. В одну группу можно посчитать несколько условных агрегатов: например, COUNT(*) FILTER (WHERE paid), COUNT(*) FILTER (WHERE cancelled) — без подзапросов и без CASE внутри COUNT.

</details>

**53. Чем FILTER (WHERE ...) в агрегате отличается от HAVING?** *(Раздел 9.3 — FILTER vs HAVING.)*

<details>
<summary>Ответ</summary>

FILTER отбирает, **какие строки внутри группы** попадают в конкретный агрегат (например, считать только «оплаченные»). HAVING отбирает, **какие группы** попадают в результат (например, оставить только группы с COUNT(*) > 5). FILTER — внутри одной группы; HAVING — между группами.

</details>

---

### 9.4. GROUPING SETS, ROLLUP, CUBE

**54. В чём разница между ROLLUP(a, b, c) и CUBE(a, b, c)?** *(Раздел 9.4 — ROLLUP и CUBE.)*

<details>
<summary>Ответ</summary>

ROLLUP даёт иерархические группировки: (a,b,c), (a,b), (a), () — как «сумма по году, по кварталу, по месяцу» и общий итог. CUBE даёт все комбинации: (a,b,c), (a,b), (a,c), (b,c), (a), (b), (c), () — полный многомерный анализ. CUBE даёт 2^N комбинаций, ROLLUP — N+1.

</details>

**55. Что такое суперитог в контексте ROLLUP/CUBE?** *(Раздел 9.4 — суперитог.)*

<details>
<summary>Ответ</summary>

Строка результата, где часть столбцов группировки «свёрнута» — в них NULL (или специальный маркер), а агрегаты посчитаны по большему множеству строк. Например, при ROLLUP(год, месяц) строка с годом и NULL в месяце — итог по всему году; строка с NULL, NULL — общий итог по всем данным.

</details>

**56. Для чего нужна функция GROUPING(col)?** *(Раздел 9.4 — GROUPING.)*

<details>
<summary>Ответ</summary>

Чтобы отличить «настоящий» NULL в данных от NULL в столбце, означающего суперитог (свёртку). GROUPING(col) возвращает 1, если этот столбец был свёрнут в данной строке (часть группировки), и 0 иначе. Удобно в отчётах и при форматировании итоговых строк.

</details>

---

### 9.5. string_agg и другие строковые агрегаты

**57. Чем string_agg(col, sep) отличается от array_agg(col)?** *(Раздел 9.5 — string_agg и array_agg.)*

<details>
<summary>Ответ</summary>

string_agg склеивает значения колонки в **одну строку** с заданным разделителем (например, «a, b, c»). array_agg собирает значения в **массив** типа PostgreSQL (например, {a,b,c}). Результат string_agg — текст; array_agg — массив, с которым можно дальше работать операторами массивов.

</details>

**58. Как задать порядок элементов внутри string_agg?** *(Раздел 9.5 — порядок в агрегате.)*

<details>
<summary>Ответ</summary>

Через ORDER BY внутри вызова: string_agg(col, sep ORDER BY col2). Порядок ORDER BY определяет, в какой последовательности значения колонки col будут склеены в результирующую строку.

</details>

**59. В какой СУБД используется listagg и чем он похож на string_agg?** *(Раздел 9.5 — listagg.)*

<details>
<summary>Ответ</summary>

listagg — в Oracle и в стандарте SQL; делает то же, что string_agg в PostgreSQL: собирает значения группы в одну строку с разделителем. Синтаксис другой: LISTAGG(col, sep) WITHIN GROUP (ORDER BY ...). В PostgreSQL аналог — string_agg.

</details>

## Резюме части II

В этой части мы прошли **фундаментальный SQL** — язык, который используется ежедневно в любом проекте с реляционной базой данных.

### Краткая сводка: что ты теперь умеешь (одними фразами)

- Выбрать правильный тип для денег (NUMERIC), дат (TIMESTAMPTZ), строк (TEXT) и объяснить, почему не FLOAT и не CHAR.
- Создать таблицу с колонками и ограничениями (PK, UNIQUE, NOT NULL, CHECK, FK) и прочитать чужой CREATE TABLE по шагам.
- Добавить/удалить колонку, переименовать таблицу; объяснить разницу между DELETE, TRUNCATE и DROP.
- Вставить одну и несколько строк, обновить и удалить по условию; всегда проверять WHERE через SELECT перед UPDATE/DELETE.
- Сделать «вставь или обнови» (ON CONFLICT) и вернуть вставленные данные (RETURNING).
- Написать SELECT с фильтром (WHERE), сортировкой (ORDER BY), ограничением (LIMIT/OFFSET) и понимать порядок выполнения.
- Убрать дубли (DISTINCT) и взять «первую строку из группы» (DISTINCT ON).
- Сгруппировать строки (GROUP BY), посчитать и суммировать по группам, отфильтровать группы (HAVING), использовать FILTER в агрегатах.

Если какую-то фразу не можешь ни объяснить, ни сделать — вернись к соответствующему разделу и к блоку «Запомните».

### Что освоено

**DDL — объявление структуры:**
- Типы данных: числа (`INTEGER`, `NUMERIC`), строки (`TEXT`, `VARCHAR`), дата/время (`TIMESTAMPTZ`), `BOOLEAN`, `UUID`, `JSONB`, массивы, ENUM, диапазоны.
- `CREATE TABLE` с именованными ограничениями; `ALTER TABLE` для изменения схемы; `DROP TABLE` и `TRUNCATE` с пониманием разницы.
- Временные таблицы для промежуточных вычислений.

**Ограничения целостности:**
- `PRIMARY KEY` = NOT NULL + UNIQUE + индекс; только один на таблицу.
- `UNIQUE` допускает несколько NULL; автоматически создаёт индекс.
- `NOT NULL` и `CHECK` — защита значений; CHECK не ловит NULL.
- `FOREIGN KEY` + каскады (`CASCADE`, `SET NULL`, `RESTRICT`) — ссылочная целостность.
- `DEFERRABLE` — для вставки взаимно зависимых данных.
- `EXCLUDE USING gist` — PostgreSQL-инструмент для диапазонов.

**DML — работа с данными:**
- `INSERT` с явным списком столбцов; multi-row INSERT; `RETURNING`.
- `UPDATE ... FROM` — обновление на основе другой таблицы.
- `DELETE ... USING` — удаление с JOIN.
- `ON CONFLICT DO UPDATE` — UPSERT; `MERGE` — стандартное слияние (PG 15+).
- `GENERATED AS IDENTITY` — современная автонумерация; `COPY` — массовая загрузка.

**SELECT — чтение данных:**
- Логический порядок выполнения: FROM → WHERE → GROUP BY → HAVING → SELECT → DISTINCT → ORDER BY → LIMIT.
- `DISTINCT ON (expr)` — первая строка из каждой группы.
- Псевдонимы, выражения, `CASE WHEN`, функции.
- `NULLS FIRST/LAST` — управление позицией NULL при сортировке.

**Агрегация:**
- `GROUP BY` + `COUNT`, `SUM`, `AVG`, `MIN`, `MAX`; правило: в SELECT только столбцы из GROUP BY или агрегаты.
- `HAVING` — фильтр по агрегату (не путать с WHERE).
- `FILTER (WHERE ...)` — условный агрегат без CASE.
- `GROUPING SETS`, `ROLLUP`, `CUBE` — несколько уровней группировки за один запрос.
- `string_agg`, `array_agg` — строковая и массивная агрегация.

### Что дальше

**Часть III. SQL — средний уровень** (§10–15):
- `JOIN` во всех видах: INNER, LEFT/RIGHT/FULL, CROSS, LATERAL.
- Подзапросы: скалярные, коррелированные, IN/EXISTS.
- Оконные функции: `ROW_NUMBER`, `RANK`, `LAG/LEAD`, `FIRST_VALUE`.
- CTE (`WITH`) — обычные и рекурсивные.
- Операции над множествами: `UNION`, `INTERSECT`, `EXCEPT`.
- NULL и трёхзначная логика: `COALESCE`, `NULLIF`, `IS DISTINCT FROM`.

Часть II — это «словарный запас» SQL. Часть III — это «грамматика»: те же идеи (типы, ограничения, SELECT) развиваются — появятся соединения таблиц (JOIN), подзапросы, оконные функции и более глубокая работа с NULL.
