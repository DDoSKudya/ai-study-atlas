## 84. Расширенные типы PostgreSQL

### Цель раздела

Познакомиться с **расширенными типами PostgreSQL** (`ltree`, `hstore`, `citext`, `pg_trgm`, `uuid-ossp`, `intarray`) и понять, какие задачи они решают, с какими индексами работают и какие паттерны помогут сделать схему проще и быстрее.

### В этом разделе главное (три строки)

1. PostgreSQL — это не только `int` и `text`: есть **специализированные типы**, которые делают модель данных ближе к задаче (иерархии, ключ–значение, нечёткий поиск).
2. Многие типы «приходят» вместе с расширениями (`ltree`, `pg_trgm`, `hstore`, `uuid-ossp`) и требуют **понимания подходящих индексов** (GiST, GIN).
3. Грамотное использование расширенных типов часто позволяет **сильно упростить запросы и ускорить их**, но важно не превращать схему в «чёрный ящик».

### Термины

- **`ltree`** — тип для представления иерархических путей (`'root.child.leaf'`).
- **`hstore`** — тип «ключ–значение» в одном столбце, с операциями `->`, `?`, `-` и т.п.
- **`citext`** — case-insensitive текст; сравнение без учёта регистра.
- **`pg_trgm`** — расширение триграмм: тип `gist_trgm_ops`/`gin_trgm_ops` для индексов, функции `similarity()`, `word_similarity()` и оператор `%`.
- **`uuid-ossp`** — расширение для генерации UUID (`uuid_generate_v4()` и др.).
- **`intarray`** — расширение и функции для массивов целых чисел.

### Теория и правила

**1. `ltree`.**

- Хранит путь в виде строкового значения с разделителем точкой: `root.topics.postgresql`.
- Поддерживает операторы:
  - `@>` — «является ли левый путь предком правого» (`'root.topics'::ltree @> 'root.topics.postgresql'::ltree`).
  - `<@` — обратное: «является ли левый путь потомком правого».
  - `~` — поиск по шаблонам.
- Легко строить **деревья и поддеревья** без рекурсивных запросов по ID родителя.
- Для быстрого поиска используются индексы **GiST**.

**2. `hstore`.**

- Хранит в одной колонке набор пар ключ–значение: `'a=>1, b=>2'::hstore`.
- Операторы:
  - `->` / `->>` — получить значение по ключу.
  - `?` — проверить наличие ключа.
  - `-` — удалить ключ.
- Индексируется через **GIN**, что делает возможным быстрый поиск по ключам/значениям.
- Полезен, когда есть **редко используемые и сильно разреженные поля**, но не хочется плодить «широкую» таблицу.

**3. `citext`.**

- Тип, который ведёт себя как `text`, но сравнения и сортировки выполняет **без учёта регистра**.
- Позволяет писать `WHERE email = 'USER@example.COM'`, не заботясь о `LOWER(email)`.
- Индексируется обычным B‑tree и остаётся эффективным.

**4. `pg_trgm`.**

- Разбивает строки на **триграммы** (последовательности из трёх символов).
- Позволяет делать:
  - поиск по подстроке (`LIKE '%foo%'`) с использованием индекса;
  - поиск по приблизительному совпадению (`similarity()`).
- Индексы GIN/GiST с опциями `gin_trgm_ops`/`gist_trgm_ops`.

**5. `uuid-ossp`.**

- Генераторы UUID разных версий: `uuid_generate_v4()` (случайный), `uuid_generate_v1()` (зависит от MAC/времени) и др.
- Удобно использовать как первичный ключ, если нужен **глобально уникальный идентификатор** без координации.

**6. `intarray`.**

- Функции для массивов целых: пересечение, объединение, сортировка, уникализация.
- Полезно, если ты хранишь множества ID в массиве (но помни о нормализации!).

#### Дополнительно: какой тип когда выбирать (на бытовых примерах)

- **`ltree`** — когда у тебя **дерево**:
  - пример: структура категорий (электроника → телефоны → смартфоны), оргструктура компании (компания → отдел → команда).
  - запросы: «дай всё поддерево вот от этой ветки», «есть ли узел внутри этой ветки».
- **`hstore`** — когда ты **не знаешь заранее набор полей**:
  - пример: пользователи могут добавлять произвольные настройки/атрибуты, и у каждого набор свой.
  - вместо 1000 nullable‑колонок с флагами можно завести одно поле `props hstore`.
- **`citext`** — когда **регистрозависимость не важна**, а писать `LOWER()` везде лень:
  - пример: логины/емейлы пользователей (`User@example.com` и `user@example.com` должны считаться одним и тем же).
- **`pg_trgm`** — когда тебе нужен **поиск по тексту с опечатками и подстроками**:
  - пример: поиск по фамилии, названию товара, адресу, когда пользователи часто ошибаются в написании.
- **`uuid-ossp`** — когда тебе нужны **глобально уникальные ID**, и ты не хочешь централизованный генератор:
  - пример: распределённая система с несколькими сервисами, каждый генерирует свои ID и не координируется с остальными.
- **`intarray`** — когда у тебя в одной строке лежит **множество int‑ID**, и ты хочешь над ними делать операции пересечения/объединения:
  - пример: «у пользователя есть набор прав в виде ID», и ты иногда хочешь посчитать пересечение с набором «прав группы».

Главное — **не перебарщивать**: если можно обойтись обычной нормализованной схемой — сначала делай так, а к спец‑типам переходи, когда видишь конкретную выгоду.

### Пошагово: типичный сценарий — нечёткий поиск по имени с `pg_trgm`

1. **Включить расширение `pg_trgm`.**
   ```sql
   CREATE EXTENSION IF NOT EXISTS pg_trgm;
   ```

2. **Создать индекс по триграммам.**
   ```sql
   CREATE INDEX idx_users_name_trgm
   ON users
   USING GIN (name gin_trgm_ops);
   ```

3. **Искать по подстроке или с опечатками.**
   ```sql
   SELECT *
   FROM users
   WHERE name ILIKE '%ivanov%';

   SELECT *
   FROM users
   WHERE similarity(name, 'ivanov') > 0.4
   ORDER BY similarity(name, 'ivanov') DESC
   LIMIT 10;
   ```

### Простыми словами

Расширенные типы — это **готовые модели для типичных сложных структур**: деревьев (`ltree`), непредсказуемых полей (`hstore`), текстов с ошибками (`pg_trgm`), идентификаторов (`uuid-ossp`).  
Вместо того чтобы «изобретать» свои схемы и функции, ты можешь **опереться на готовые решения**, которые уже оптимизированы и интегрированы в планировщик PostgreSQL.

### Картинка в голове / Как запомнить

Представь, что стандартные типы (`int`, `text`) — это **молоток и отвёртка**, а расширенные типы — это **дрель, уровень и перфоратор**.  
Ты можешь всё делать молотком, но намного проще и аккуратнее использовать **правильный инструмент под задачу**.

### Проверь себя

1. **Чем `hstore` отличается от JSONB?**
   <details><summary>Ответ</summary>
   `hstore` — более простой тип «ключ–значение» без вложенных структур (по сути, словарь строк → строк).  
   JSONB поддерживает вложенные объекты, массивы и богатый набор операторов, но часто тяжелее.  
   Для простых плоских наборов ключей `hstore` может быть легче и быстрее.
   </details>

2. **Какой тип и индекс использовать для хранения и быстрого поиска иерархий (категорий, меню)?**
   <details><summary>Ответ</summary>
   Тип `ltree` + индекс GiST по колонке типа `ltree`. Это позволяет быстро искать предков/потомков по операторам `@>` и `<@`.
   </details>

3. **Как ускорить поиск по `LIKE '%подстрока%'` по большому текстовому столбцу?**
   <details><summary>Ответ</summary>
   Включить расширение `pg_trgm` и создать GIN‑индекс с `gin_trgm_ops`. Тогда планировщик сможет использовать индекс для `LIKE`/`ILIKE` и `similarity()`.
   </details>

### Запомните

- Расширенные типы — это **часть силы PostgreSQL**; они позволяют моделировать данные ближе к предметной области.
- Многие из них приходят через расширения (`ltree`, `pg_trgm`, `hstore`, `uuid-ossp`).
- Выбор типа всегда связан с **выбором индекса и типичных запросов**.
