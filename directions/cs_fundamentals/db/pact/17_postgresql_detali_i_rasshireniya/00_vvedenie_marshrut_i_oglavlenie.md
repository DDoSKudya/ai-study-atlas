## Часть XVII. PostgreSQL: детали и расширения

> Цель этой части — **понять, как устроены детали PostgreSQL «под капотом» и как использовать мощные расширения и продвинутые возможности**: системные каталоги, конфигурацию, расширенные типы данных и функции, а также экосистему расширений для мониторинга, партиционирования, репликации, геоданных и векторного поиска.

После этой части ты сможешь:

- **уверенно устанавливать и настраивать расширения PostgreSQL** через `CREATE EXTENSION`, видеть список доступных и управлять версиями;
- **читать и использовать системные каталоги** (`pg_class`, `pg_attribute`, `pg_index`, `pg_constraint`, `pg_depend`) и представления `pg_stat_*` для диагностики и анализа;
- **осмысленно настраивать PostgreSQL** через `postgresql.conf` и `pg_hba.conf`: память, WAL, autovacuum, параллелизм, логирование;
- **применять расширенные типы и функции** (`ltree`, `hstore`, `citext`, `pg_trgm`, `uuid-ossp`, массивы, JSONB, `generate_series`, `unnest`, агрегаты и оконные функции) для компактных и быстрых решений;
- **использовать расширения для партиционирования, репликации и мониторинга** (`pg_partman`, `pg_pathman`, `pglogical`, `bucardo`, `pg_stat_monitor`, `pg_wait_sampling`) в реальных операционных сценариях;
- **ориентироваться в экосистеме расширений для данных** (PostGIS, `pgvector`, `hstore`) и понимать, когда база превращается в платформу данных.

Материал опирается на:

- **Часть V** (хранение, страницы, WAL, буферный кеш),
- **Часть VI–VII** (индексы, оптимизация запросов),
- **Часть XIV** (партиционирование и внутренности хранилища),
- **Часть XVI** (аналитика, хранилища, CDC).

В этой части мы выходим за рамки «как просто завести PostgreSQL» и учимся **думать в терминах внутреннего устройства и расширяемости**: не только «какая настройка», но и **почему именно такая**, не только «какой тип», но и **какое поведение и какие индексы за этим стоят**.

### Краткая шпаргалка по части

| Понятие / конструкция | Суть (что это и когда нужно) |
|----------------------|-----------------------------|
| **Расширение (extension)** | Пакет функциональности, который «подключается» к PostgreSQL через `CREATE EXTENSION` (типы, функции, операторы, индексы и т.п.). |
| **`CREATE EXTENSION`** | Команда установки расширения в текущую БД; фактически регистрирует ранее установленные на уровне кластера файлы расширения. |
| **`pg_available_extensions`** | Системная вьюха со списком доступных расширений, их версиями и описаниями. |
| **`pg_stat_statements`** | Расширение для сбора статистики по запросам (частота, время, отклонения); база для оптимизации. |
| **`pg_trgm`** | Расширение триграммного индекса и функции приблизительного поиска по строкам (поиск по подстроке, опечаткам). |
| **`pgcrypto`** | Набор криптографических функций: хэширование паролей, шифрование/расшифровка, генерация случайных значений. |
| **PostGIS** | Расширение для геопространственных данных (точки, полигоны, расстояния, пересечения, индексы GiST/SP-GiST). |
| **`pgvector`** | Тип `vector` и функции для векторного поиска (similarity search, ANN), полезен для рекомендаций и AI. |
| **`pg_partman` / `pg_pathman`** | Расширения для автоматического управления партиционированием (создание/удаление партиций, оптимизация). |
| **`pglogical` / `bucardo`** | Расширения для логической и multi-master репликации. |
| **`pg_stat_monitor`, `pg_wait_sampling`** | Расширения для продвинутого мониторинга запросов и ожиданий (wait events). |
| **`pg_class`** | Системный каталог с объектами (таблицы, индексы, последовательности); основа «карты» кластера. |
| **`pg_attribute`** | Столбцы таблиц: имена, типы, nullable-флаг, порядок. |
| **`pg_index` / `pg_constraint`** | Информация о индексах и ограничениях (PK, UNIQUE, FK, CHECK). |
| **`postgresql.conf`** | Главный конфигурационный файл кластера PostgreSQL. |
| **`pg_hba.conf`** | Конфигурация доступа: кто и откуда может подключиться, каким методом аутентификации. |
| **`ltree`** | Тип для иерархий в виде путей `root.child.leaf`; быстрый поиск предков/потомков. |
| **`hstore`** | Ключ–значение в одном столбце, ближе к JSON, но проще и легче. |
| **`citext`** | Case-insensitive текст: сравнения без `LOWER()`, индексируясь как обычный `text`. |
| **`generate_series`** | Функция, создающая набор чисел/дат; полезна для временных рядов и тестовых данных. |
| **`unnest`** | Преобразование массива в строки, чтобы соединять массивы с таблицами. |
| **`array_agg` / `jsonb_agg`** | Агрегаты, собирающие строки в массивы/JSON; удобны для выдачи API из одной таблицы. |

Используй шпаргалку как карту: нашёл слово в первом столбце — во втором видишь, **зачем оно нужно**.

### Маршрут изучения в этой части

1. **§81. Расширения PostgreSQL**  
   Что такое расширения, как их устанавливать (`CREATE EXTENSION`), какие популярные расширения есть (для статистики, поиска, криптографии, партиционирования, репликации, мониторинга и данных), как ими управлять.

2. **§82. Системные каталоги PostgreSQL**  
   Знакомство с `pg_class`, `pg_attribute`, `pg_index`, `pg_constraint`, `pg_depend`, `pg_stat_*` и удобными представлениями `pg_tables`, `pg_indexes`, `pg_stat_user_tables` — как читать структуру и состояние БД «изнутри».

3. **§83. Конфигурация PostgreSQL**  
   `postgresql.conf` и `pg_hba.conf`, основные группы параметров (память, WAL, autovacuum, параллелизм, логирование), `ALTER SYSTEM` и `pg_reload_conf()`, подход к настройке «от здравого смысла», а не «по списку тюнинга из блога».

4. **§84. Расширенные типы PostgreSQL**  
   Типы `ltree`, `hstore`, `citext`, `uuid-ossp`, `pg_trgm`, `intarray`: зачем они нужны, какие индексы к ним подходят (GiST, GIN), какие паттерны задач решают.

5. **§85. Расширенные функции PostgreSQL**  
   `generate_series`, `unnest`, `array_agg`, `jsonb_agg`, оконные функции и пользовательские агрегаты — как писать компактные, но читаемые запросы, и где не перегибать палку.

6. **§86. Расширения для партиционирования**  
   Как `pg_partman` и `pg_pathman` помогают автоматизировать партиционирование: создание/удаление партиций, стратегии (время, диапазон, хеш), cron/job-скрипты.

7. **§87. Расширения для репликации и мониторинга**  
   `pglogical`, `bucardo`, `pg_stat_monitor`, `pg_wait_sampling`: когда нужны, как вписываются в общую архитектуру и чего от них ожидать (и чего — нет).
