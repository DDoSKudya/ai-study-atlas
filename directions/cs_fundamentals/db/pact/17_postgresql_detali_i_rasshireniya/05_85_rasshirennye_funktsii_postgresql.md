## 85. Расширенные функции PostgreSQL

### Цель раздела

Научиться использовать **расширенные функции PostgreSQL** (`generate_series`, `unnest`, `array_agg`, `jsonb_agg`, оконные функции, пользовательские агрегаты) для написания выразительных и эффективных запросов.

### В этом разделе главное (три строки)

1. PostgreSQL имеет богатую библиотеку функций для **генерации данных** (`generate_series`), работы с массивами (`unnest`, `array_agg`) и JSONB (`jsonb_agg`, `jsonb_build_object`).
2. **Оконные функции** позволяют считать агрегаты поверх «окон» строк, не ломая результат в одну строку на группу.
3. **Пользовательские агрегаты** (`CREATE AGGREGATE`) дают возможность инкапсулировать сложную логику агрегации в переиспользуемый объект.

### Термины

- **`generate_series`** — функция, которая генерирует последовательность чисел или дат.
- **`unnest`** — разворачивает массив в набор строк.
- **`array_agg`** — агрегирует значения в массив.
- **`jsonb_agg`** — агрегирует значения в JSON‑массив.
- **Оконная функция** — функция, которая вычисляется **поверх «окна» строк**, определённого через `OVER (PARTITION BY ... ORDER BY ...)`.
- **Пользовательский агрегат** — агрегат, определяемый пользователем через `CREATE AGGREGATE` (указывается тип состояния, функции перехода и финализации).

### Теория и правила

**1. `generate_series`.**

- Генерирует числа или даты:
  ```sql
  SELECT generate_series(1, 5);         -- 1,2,3,4,5
  SELECT generate_series('2024-01-01'::date,
                         '2024-01-10'::date,
                         '1 day');     -- даты по дням
  ```
- Очень полезно для:
  - генерации **временных рядов**;
  - заполнения дыр в датах;
  - тестовых данных.

**2. `unnest` и массивы.**

- `unnest(array)` превращает массив в строки:
  ```sql
  SELECT unnest(ARRAY[1,2,3]); -- 1,2,3
  ```
- Можно с `LATERAL`, чтобы работать с массивами в таблицах:
  ```sql
  SELECT u.id, tag
  FROM users u
  CROSS JOIN LATERAL unnest(u.tags) AS tag;
  ```

**3. `array_agg` и `jsonb_agg`.**

- `array_agg(col)` собирает значения в массив:
  ```sql
  SELECT user_id, array_agg(order_id ORDER BY created_at) AS orders
  FROM orders
  GROUP BY user_id;
  ```
- `jsonb_agg` и `jsonb_build_object` используются для формирования структурированных JSON‑ответов прямо в SQL.

**4. Оконные функции.**

- Примеры:
  ```sql
  SELECT
    user_id,
    created_at,
    amount,
    SUM(amount) OVER (PARTITION BY user_id
                      ORDER BY created_at
                      ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS balance
  FROM payments;
  ```
- Позволяют считать:
  - скользящие суммы, средние;
  - ранги, процентили (`percentile_cont`);
  - массивы значений по окну (`array_agg(...) OVER (...)`).

**5. Пользовательские агрегаты.**

- `CREATE AGGREGATE` позволяет описать:
  - **тип состояния** (state);
  - **функцию перехода** (что происходит при добавлении строки);
  - **финальную функцию** (как из состояния получить результат).
- Это способ **инкапсулировать нетривиальные паттерны** агрегации (например, собственная медиана с особой логикой).

#### Дополнительно: как мыслить «оконными» категориями

Представь, что у тебя есть таблица транзакций по пользователям.  
Обычный `GROUP BY user_id` даст тебе **одну строку на пользователя** (например, общую сумму трат).  
Но часто тебе нужно:

- видеть **каждую транзакцию**, но при этом знать:
  - сколько пользователь потратил **до этого момента** (накопленная сумма);
  - какой это по счёту платёж (номер строки внутри пользователя);
  - какой процентиль по сумме транзакции он занимает.

Вот тут и вступают в игру оконные функции:

- `SUM(amount) OVER (PARTITION BY user_id ORDER BY created_at)` — накопленная сумма;
- `ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY created_at)` — номер транзакции;
- `percentile_cont(0.9) WITHIN GROUP (ORDER BY amount) OVER ()` — 90‑й процентиль по всем транзакциям и т.п.

Запомни образ: **оконная функция = «дополнительная колонка с агрегатом», не склеивающая строки в одну.**

### Пошагово: генерация временного ряда с пропусками

Задача: посчитать количество заказов по дням, показывая **даже те дни, когда заказов не было**.

1. **Сгенерировать диапазон дат.**
   ```sql
   WITH days AS (
     SELECT generate_series(
              date_trunc('day', MIN(created_at)),
              date_trunc('day', MAX(created_at)),
              '1 day'::interval
            )::date AS day
     FROM orders
   )
   ```

2. **Посчитать количество заказов по дням.**
   ```sql
   , orders_per_day AS (
     SELECT
       date_trunc('day', created_at)::date AS day,
       COUNT(*) AS cnt
     FROM orders
     GROUP BY 1
   )
   SELECT
     d.day,
     COALESCE(o.cnt, 0) AS cnt
   FROM days d
   LEFT JOIN orders_per_day o USING (day)
   ORDER BY d.day;
   ```

### Простыми словами

Расширенные функции — это **инструменты для «жонглирования» строками и наборами**: генерировать строки, собирать их в массивы и JSON, считать скользящие суммы, ранги и процентили.  
Много того, что в других СУБД пришлось бы писать на процедурном языке или в коде приложения, в PostgreSQL можно выразить **компактным и читаемым SQL‑запросом**.

### Картинка в голове / Как запомнить

Представь, что обычный SQL — это просто **фильтрация и объединение таблиц**, а расширенные функции — это **режиссёр, который умеет создавать, группировать и переигрывать сцены**.  
Ты можешь не только прочитать таблицу, но и **сгенерировать недостающие кадры** (`generate_series`), собрать сцены в один блок (`array_agg`, `jsonb_agg`) и посмотреть на них по‑разному (оконные функции).

### Проверь себя

1. **Чем отличается `GROUP BY` + агрегат от оконной функции?**
   <details><summary>Ответ</summary>
   `GROUP BY` с агрегатом сжимает строки в одну на группу. Оконная функция считает агрегат по «окну» строк, но **не меняет количество строк**: каждая строка получает значение агрегата по своему окну.
   </details>

2. **Как сгенерировать 100 случайных дат в пределах одного месяца с помощью PostgreSQL (без внешних скриптов)?**
   <details><summary>Ответ</summary>
   Сочетать `generate_series` и `random()`, например:  
   ```sql
   SELECT (DATE '2024-01-01'
           + (random() * 30)::int) AS rnd_date
   FROM generate_series(1, 100);
   ```
   </details>

3. **Когда имеет смысл заводить пользовательский агрегат?**
   <details><summary>Ответ</summary>
   Когда одна и та же нетривиальная логика агрегации (своё состояние, свои правила комбинирования) **повторяется во многих запросах**, и хочется вынести её в один переиспользуемый объект вместо копирования кода.
   </details>

### Запомните

- `generate_series`, `unnest`, `array_agg`, `jsonb_agg` и оконные функции — это **ключевые кирпичи выразительного SQL** в PostgreSQL.
- Пользовательские агрегаты позволяют **расширять язык** под свои задачи.
- Не злоупотребляй «магией»: запросы должны оставаться **читаемыми**.
