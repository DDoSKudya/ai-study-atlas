## 261. Реализация изоляции в СУБД

### Цель раздела

Понять, как **конкретные СУБД (PostgreSQL, MySQL InnoDB, Oracle, SQL Server)** реализуют уровни изоляции, какие механизмы они используют (MVCC, undo, gap/next‑key locks, row versioning), какие аномалии реально возможны и как правильно выбирать настройки в продакшене.

### В этом разделе главное

1. Все популярные СУБД используют вариации **MVCC и/или блокировок**, но детали сильно различаются.
2. В PostgreSQL основной режим — **snapshot isolation**, а уровень `SERIALIZABLE` реализован поверх MVCC через SSI.
3. В MySQL InnoDB `REPEATABLE READ` реализован через **gap/next‑key locks**, что даёт поведение, отличающееся от PostgreSQL.
4. Oracle использует **undo‑based MVCC**, давая очень сильное read‑consistency, но с собственными нюансами.
5. SQL Server сочетает **row versioning** (snapshot isolation/read committed snapshot) с традиционными блокировками.

### Термины

- **MVCC** — Multi‑Version Concurrency Control, многоверсионный контроль параллелизма.
- **Undo segment / undo log** — область, где хранятся старые версии строк для отката и чтения в прошлом.
- **Gap lock / next‑key lock** — блокировки “промежутков” и записей в индексах (InnoDB).
- **Row versioning** — хранение нескольких версий строки (SQL Server).
- **SSI (Serializable Snapshot Isolation)** — техника поверх SI, которая предотвращает опасные паттерны (dangerous structures).

### Теория и правила

#### 1) PostgreSQL

Ключевые моменты:

- PostgreSQL использует **MVCC**:
  - каждая строка имеет скрытые поля `xmin`/`xmax` (кто создал/удалил);
  - каждая транзакция имеет свой `txid`;
  - видимость строки определяется сравнением `xmin/xmax` с “снимком” активных транзакций.
- Уровни изоляции:
  - `READ COMMITTED`:
    - каждый запрос видит **снимок на момент начала запроса**;
    - разные запросы в одной транзакции могут видеть изменения, закоммиченные между ними.
  - `REPEATABLE READ`:
    - фактически **snapshot isolation**: весь транзакционный “разговор” видит один снимок на начало транзакции;
    - возможен write skew.
  - `SERIALIZABLE`:
    - реализован через **SSI**: система отслеживает конфликты чтения/записи и может откатывать транзакции, чтобы избежать не‑serializable историй.

Практика:

- большинство приложений используют `READ COMMITTED` или `REPEATABLE READ`;
- для критичных инвариантов можно точечно использовать `SERIALIZABLE` или явные блокировки (`SELECT ... FOR UPDATE`).

#### 2) MySQL InnoDB

Ключевые идеи:

- InnoDB использует **MVCC + блокировки**, но немного иначе, чем PostgreSQL.
- Уровни изоляции:
  - по умолчанию `REPEATABLE READ`;
  - `READ COMMITTED` доступен и даёт поведение ближе к Oracle.
- В InnoDB важны:
  - **gap locks** — блокировки “промежутков” в индексах;
  - **next‑key locks** — блокировки “запись + gap”.

Это позволяет:

- предотвращать **phantom reads** в `REPEATABLE READ`, но ценой большего числа блокировок;
- обеспечивать сериализуемость (часто близкую к serializable) через комбинацию индексов и этих блокировок.

Однако:

- реализация отличается от PostgreSQL snapshot isolation;
- некоторые аномалии могут проявляться по‑другому (особенно при отсутствии подходящих индексов).

#### 3) Oracle

Oracle традиционно использует **undo‑based MVCC**:

- при изменении строки её старая версия сохраняется в **undo‑сегментах**;
- чтения могут “спуститься назад во времени”, читая старые версии, чтобы обеспечить **read consistency**.

Особенности:

- уровень `READ COMMITTED` обеспечивает, что:
  - каждое чтение видит **коммитнутые на момент чтения** данные;
  - старые версии берутся из undo.
- уровень `SERIALIZABLE`:
  - ближе к snapshot isolation с дополнительными ограничениями;
  - при конфликте транзакция может получить ошибку “could not serialize access” и быть откатана.

Oracle активно использует undo и redolog для того, чтобы совместить:

- высокую конкурентность чтений,
- и строгие гарантии read consistency.

#### 4) SQL Server

SQL Server исторически ориентирован на **lock‑based** подход, но с добавлением **row versioning**:

- режим по умолчанию (`READ COMMITTED`) — классический locking, но можно включить `READ COMMITTED SNAPSHOT`, при котором чтения используют версии.
- есть отдельный уровень **SNAPSHOT**:
  - транзакция читает снимок на момент начала;
  - write‑write конфликты приводят к ошибкам.

Row versioning:

- версии строк хранятся в tempdb;
- чтения при snapshot‑режимах берут версии оттуда, минуя блокировки писателей.

SQL Server также предоставляет уровень `SERIALIZABLE` через строгие диапазонные блокировки.

### Пошагово: ментальная модель MVCC в PostgreSQL

1. Каждой транзакции присваивается идентификатор `txid`.
2. При вставке/обновлении строки:
   - создаётся новая версия строки с `xmin = txid` текущей транзакции;
   - старая версия остаётся в таблице (помечается через `xmax`, когда будет удалена/заменена).
3. Когда другой запрос читает таблицу, он:
   - смотрит на свой **snapshot** (множество активных `txid` + границы видимости);
   - считает строку видимой, если:
     - `xmin` коммитнута и не в будущем;
     - `xmax` либо пуст, либо транзакция‑удалитель ещё не закоммитила/откатилась.

Таким образом:

- чтения не блокируют записи и наоборот;
- разные транзакции могут видеть **разные версии** одной и той же строки.

### Простыми словами

- В PostgreSQL и Oracle много “магии” делает **MVCC/undo**: чтения как будто “перемещаются по времени” и видят консистентный снимок.
- В MySQL InnoDB и SQL Server, помимо версий, активно используются **блокировки на уровне индексов/диапазонов**, чтобы предотвратить определённые аномалии.
- Поэтому одинаковые уровни изоляции по имени могут вести себя **по‑разному** в разных СУБД.

### Картинка в голове

Вообрази таблицу как **стопку слоёв**:

- каждая новая транзакция добавляет новый слой (версию строки);
- разные транзакции видят **разные комбинации слоёв**, в зависимости от своего snapshot’а;
- блокировки/гамма‑локи при этом действуют как “заглушки” на некоторых слоях или промежутках.

### Как запомнить

- **PostgreSQL** = “чистый MVCC + SSI для serializable”.
- **MySQL InnoDB** = “MVCC + gap/next‑key locks”.
- **Oracle** = “undo‑based MVCC с сильным read consistency”.
- **SQL Server** = “locks по умолчанию, но можно включить row versioning для snapshot‑режимов”.

### Примеры

- **PostgreSQL**:

```sql
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
BEGIN;
-- критичный инвариант, считаем суммы, делаем принятие решения
...
COMMIT; -- возможна ошибка сериализации, нужно перезапускать транзакцию
```

- **MySQL InnoDB**:

```sql
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;
START TRANSACTION;
SELECT * FROM orders WHERE amount > 1000 FOR UPDATE;
-- диапазонная блокировка индекса, предотвращающая фантомы
COMMIT;
```

- **SQL Server** (включение snapshot‑режима для БД):

```sql
ALTER DATABASE MyDb SET ALLOW_SNAPSHOT_ISOLATION ON;
ALTER DATABASE MyDb SET READ_COMMITTED_SNAPSHOT ON;
```

### Практика / реальные сценарии

- При миграции приложения с PostgreSQL на MySQL:
  - код, опирающийся на snapshot isolation в PostgreSQL, может вести себя иначе в InnoDB `REPEATABLE READ` из‑за gap‑локов и другой семантики.
- При включении `READ_COMMITTED_SNAPSHOT` в SQL Server:
  - резко уменьшается количество блокировок чтения, но начинает расти объём tempdb и сложность анализа версий.

### Типичные ошибки

- Полагать, что `SERIALIZABLE` в разных СУБД означает одно и то же поведение.
- Включать snapshot‑режимы в SQL Server или read‑consistency‑режимы в Oracle без оценки нагрузки на storage/undo/tempdb.
- Проектировать схему и индексы, не учитывая, как именно движок реализует диапазонные блокировки (InnoDB, SQL Server serializable).

### Что будет, если…

- …перенести приложение с PostgreSQL `REPEATABLE READ` на MySQL InnoDB `REPEATABLE READ`, не меняя код?
  - Можно получить **дополнительные блокировки и дедлоки** (из‑за gap/next‑key locks), а также отличия в поведении некоторых аномалий; приложение может неожиданно “тормозить” или падать по дедлокам.
- …игнорировать ошибки сериализации в PostgreSQL `SERIALIZABLE` и не перезапускать транзакции?
  - Приложение будет **периодически падать** на таких операциях, а логика — не завершаться; это корректное поведение движка, но код должен быть готов к ретраю.

### Проверь себя

1. Почему PostgreSQL `REPEATABLE READ` фактически реализует snapshot isolation, а не классический repeatable read?

<details><summary>Ответ</summary>
Потому что в PostgreSQL при `REPEATABLE READ` транзакция получает **снимок на момент своего начала** и все запросы внутри неё видят именно этот снимок, независимо от последующих коммитов других транзакций. Это поведение соответствует snapshot isolation. Классический repeatable read (по стандарту) описывается через аномалии и не обязывает к snapshot‑модели; реализация может быть другой (например, через диапазонные блокировки).
</details>

2. Как gap/next‑key locks в MySQL InnoDB помогают бороться с фантомами?

<details><summary>Ответ</summary>
Они блокируют **не только существующие строки, но и промежутки в индексах**, где потенциально могли бы появиться новые строки, подходящие под условие запроса. Благодаря этому другая транзакция не может “вставить фантом” (строку, попадающую в диапазон), пока первая транзакция не завершит работу, что эффективно предотвращает phantom read в соответствующем уровне изоляции.
</details>

3. В чём практическая разница между `READ COMMITTED` и `READ COMMITTED SNAPSHOT` в SQL Server?

<details><summary>Ответ</summary>
В классическом `READ COMMITTED` чтения используют **блокировки** (shared locks) и могут ждать освобождения ресурсов писателей. В `READ COMMITTED SNAPSHOT` чтения используют **версии строк** (row versioning) и, как правило, не блокируют записи и не блокируются ими; это снижает количество блокировок, но увеличивает нагрузку на tempdb и требует учитывать, что чтения могут видеть слегка “отставшее” состояние.
</details>

### Запомните

- Одинаковые по названию уровни изоляции могут быть **по‑разному реализованы** в PostgreSQL, MySQL, Oracle и SQL Server.
- MVCC, undo, gap‑локи и row versioning — разные механизмы, и важно понимать их интуитивно, чтобы правильно настраивать СУБД и интерпретировать поведение.
- При смене СУБД/движка всегда нужно **перепроверять** инварианты и аномалии под реальными уровнями изоляции.
