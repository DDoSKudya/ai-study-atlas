## Справочник по части (ключевые пункты)

### Уровни изоляции → аномалии

| Уровень                 | Dirty read | Non‑repeatable read | Phantom | Write skew | Lost update (без явной защиты) |
| ----------------------- | ---------: | -------------------:| -------:| ---------: | ------------------------------: |
| Read uncommitted        |     допускает |          допускает | допускает |    допускает |             допускает           |
| Read committed          |   запрещает |          допускает | допускает |    допускает |             допускает           |
| Repeatable read (класс.) |   запрещает |         запрещает  | допускает |    допускает |             допускает           |
| Snapshot isolation (SI) |   запрещает |         запрещает  | частично допускает |   допускает |             запрещает (для одной строки) |
| Serializable            |   запрещает |         запрещает  | запрещает |   запрещает |             запрещает           |

> На практике поведение может отличаться в деталях между PostgreSQL, MySQL, Oracle и SQL Server.

### Модели согласованности → гарантии

| Модель                   | Гарантия                                                                                             | Типичные системы                                      |
| ------------------------ | ---------------------------------------------------------------------------------------------------- | ----------------------------------------------------- |
| Linearizability          | Один глобальный порядок операций, согласованный с реальным временем                                 | etcd, ZooKeeper, Raft‑кластеры                        |
| Sequential consistency   | Один глобальный порядок, совместимый с локальными порядками, но не обязательно по реальному времени | некоторая память/модели многопроцессорных систем     |
| Causal consistency       | Сохраняет причинно‑следственные зависимости, независимые операции могут быть упорядочены по‑разному | соцсети, чаты, некоторые NoSQL‑системы               |
| PRAM                     | Каждый процесс видит свои записи по порядку, нет глобального порядка                                | базовые модели распределённой памяти                  |
| Eventual consistency     | В отсутствие новых обновлений все реплики сходятся к одному состоянию                               | Dynamo‑подобные KV, Cassandra (при слабых настройках) |
| Strong eventual consistency | Eventual + одинаковый итог при одинаковом наборе обновлений                                        | CRDT‑системы, offline‑редакторы                       |

### СУБД → реализация изоляции (очень кратко)

| СУБД       | Основной механизм                 | Особенности уровней изоляции                                                   |
| ---------- | --------------------------------- | ------------------------------------------------------------------------------ |
| PostgreSQL | MVCC + SSI                        | `REPEATABLE READ` ≈ SI, `SERIALIZABLE` через SSI, сильный snapshot‑подход      |
| MySQL InnoDB | MVCC + gap/next‑key locks       | `REPEATABLE READ` с диапазонными блокировками, `READ COMMITTED` ближе к Oracle |
| Oracle     | Undo‑based MVCC                   | Сильный read consistency, snapshot‑подобные режимы                             |
| SQL Server | Locks + row versioning (snapshot) | `READ COMMITTED` + опция `SNAPSHOT`, `READ_COMMITTED_SNAPSHOT`, tempdb‑версии  |

## Частые сценарии

### Сценарий 1. Критичный финансовый инвариант

- **Задача**: сохранять инвариант “сумма балансов не уменьшается/не уходит в минус” в условиях конкуренции.
- Подход:
  - использовать либо `SERIALIZABLE` / сильную изоляцию (PostgreSQL SSI, строгие блокировки в других СУБД) для операций, изменяющих баланс;
  - либо комбинировать **snapshot isolation + жёсткие констрейнты** (CHECK, UNIQUE) и/или явные блокировки (`FOR UPDATE`);
  - избегать write‑skew‑уязвимых паттернов (“решение на основе чтения двух строк → запись в одну”).

### Сценарий 2. Высоконагруженный OLTP‑сервис

- **Задача**: максимальный throughput при разумных гарантиях.
- Подход:
  - использовать **MVCC‑режимы (read committed / snapshot)**;
  - минимизировать длительность транзакций;
  - для особо конфликтных участков — локальные блокировки или строгие уровни изоляции;
  - внимательно относиться к дедлокам и откатам, добавляя ретраи.

### Сценарий 3. Геораспределённый KV‑кластер

- **Задача**: низкая задержка для чтения/записи в разных регионах.
- Подход:
  - выбирать **AP/CA/CP‑настройки** осознанно (tunable consistency, quorum);
  - для не критичных данных (счётчики, ленты) использовать **eventual/SEC + CRDT**;
  - для критичных ключей (конфигурация, лидеры) — CP‑сервисы (ZooKeeper/etcd).

### Сценарий 4. Offline‑редактирование и синхронизация

- **Задача**: позволить пользователю работать без сети, а потом корректно синхронизироваться.
- Подход:
  - использовать **CRDT** для документов/строк/коллекций;
  - применять логические часы/векторные версии для отслеживания причинных зависимостей;
  - в случае критичных конфликтов применять гибридный подход (CRDT + ручное разрешение).

## Вопросы для самопроверки по части

1. Чем conflict serializability отличается от view serializability и почему на практике почти всегда используют первую?
2. В каких случаях модели causal/eventual consistency оказываются более подходящими, чем linearizability?
3. Как Two‑Phase Locking и timestamp ordering по‑разному решают задачу обеспечения serializability и какие компромиссы при этом делают?
4. Что именно утверждает CAP‑теорема и как PACELC дополняет её в повседневном режиме работы систем?
5. Как различия в реализации уровней изоляции в PostgreSQL, MySQL, Oracle и SQL Server могут повлиять на миграцию приложения?
6. Как CRDT и логические часы помогают построить системы со слабой согласованностью, но предсказуемым поведением?

<details><summary>Ответы</summary>

1. Conflict serializability сравнивает истории по **конфликтующим операциям** (read/write/write на один и тот же объект) и даёт удобный **графовый критерий** (отсутствие циклов в графе конфликтов). View serializability сравнивает истории по “виду” данных (кто какие значения прочитал и какие стали финальными) и допускает больше историй, но проверка её в общем случае **сложнее (NP‑полна)**. На практике системы проектируются так, чтобы быть хотя бы conflict‑serializable, потому что это **проверяемо и достаточно** для подавляющего большинства задач.

2. Там, где критична **доступность и низкая задержка**, а временные расхождения между репликами допустимы: соцсети, чаты, ленты новостей, аналитические/метрические системы, offline‑приложения. В таких сценариях явно линераризуемая система была бы либо слишком дорогой по latency/ресурсам, либо часто недоступной при partition, тогда как causal/eventual дают достаточно предсказуемости при меньших издержках.

3. 2PL обеспечивает serializability через **блокировки и ожидания**, что приводит к дедлокам и необходимости их предотвращать/обнаруживать, но уменьшает количество откатов (если не считать откаты из‑за дедлоков). Timestamp ordering и его варианты обеспечивают serializability через **упорядочение по меткам времени и откаты нарушителей**, снижая зависимость от блокировок, но увеличивая риск частых откатов и starvation. Первый подход платит ожиданием и блокировками, второй — откатами и возможной неравномерностью судьбы транзакций.

4. CAP утверждает, что в распределённой системе, подверженной **partition**, нельзя одновременно гарантировать и linearizable‑consistency (C), и строгую availability (A) для всех запросов к живым узлам: при partition придётся выбирать, чем жертвовать. PACELC добавляет, что **даже без partition** система всегда балансирует между **latency (L)** и **consistency (C)**: синхронная координация повышает задержку, но усиливает согласованность, и наоборот.

5. Потому что одинаковые названия уровней изоляции (например, `REPEATABLE READ`, `SERIALIZABLE`) могут соответствовать **разным внутренним механизмам**: PostgreSQL использует MVCC + SSI, InnoDB — MVCC + gap/next‑key locks, Oracle — undo‑based MVCC, SQL Server — locks + row versioning. Это означает, что одни и те же шаблоны запросов и транзакций могут вести себя по‑разному: больше/меньше блокировок, другие аномалии, другие требования к обработке ошибок (дедлоки, serialization failures).

6. CRDT задаёт **формальные правила для операций и слияния состояний**, благодаря которым любые две реплики с одинаковым набором обновлений сходятся к **одному и тому же состоянию**, независимо от порядка и задержек доставки (SEC). Логические часы (Lamport/vector clocks) позволяют отслеживать **причинно‑следственные зависимости** между событиями без глобального времени, что помогает реализовать causal/eventual consistency и корректно обрабатывать конфликты и слияния. Вместе они дают основу для построения надёжных реплицированных структур данных в слабоконсистентных системах.

</details>

## Типичные ошибки по части

- Путать уровни изоляции и модели согласованности, смешивая внутреннюю изоляцию транзакций с межрепликовой консистентностью.
- Считать, что snapshot isolation “решает всё” и эквивалентен serializability.
- Игнорировать различия в реализации изоляции между PostgreSQL, MySQL, Oracle и SQL Server при проектировании/миграции.
- Интерпретировать CAP как запрет иметь и Consistency, и Availability вообще, а не только в условиях partition.
- Использовать Eventually‑consistent хранилища там, где бизнес ожидает строгую финансовую или юридическую консистентность, без доп. механизмов.
- Применять CRDT “на глазок” без формального понимания операций и слияния, что приводит к скрытым расхождениям.

## Резюме части XLIII

- **Формальные модели изоляции (serializability, snapshot isolation, уровни SQL)** дают точный язык для описания корректности транзакций и аномалий.
- **Модели согласованности распределённых систем (linearizability, sequential, causal, eventual, SEC)** позволяют понимать, какие гарантии реально дают современные БД и хранилища.
- **Протоколы 2PL, TO/MVTO, MVCC, SSI** — это разные инженерные пути к достижению этих моделей, каждый со своими плюсами, минусами и областью применения.
- **CAP и PACELC** напоминают, что в распределённых системах всегда есть компромиссы между согласованностью, доступностью и задержкой, особенно при сетевых проблемах.
- **Реальные СУБД (PostgreSQL, MySQL, Oracle, SQL Server)** по‑разному реализуют изоляцию, и эти различия критично важно понимать при проектировании и миграции.
- **Слабая согласованность, CRDT и логические часы** дают инструменты для построения надёжных, геораспределённых и offline‑устойчивых систем, если применять их осознанно.
