## 267. Event sourcing и commit log

### Цель раздела

Понять, как **append‑only журналы (commit log, event log)** связаны с LSM‑подходом,  
что такое **event sourcing** и **event store**, чем отличается Kafka‑подобный лог от WAL/commit log в СУБД,  
и как использовать эти идеи при проектировании архитектуры (и где обычно путаются).

### В этом разделе главное

1. **Append‑only лог** — это фундаментальный строительный блок: мы **никогда не переписываем прошлое, а только добавляем новые записи**.
2. **WAL/commit log в СУБД** — это внутренний журнал для durability и репликации, а не “обычная таблица”.
3. **Event store** и **event sourcing** используют лог **как источник истины**: состояние восстанавливается из последовательности событий.
4. Kafka‑подобные логи и LSM‑деревья опираются на схожую идею: **append‑only + фоновая “уборка” (compaction, retention)**, но применяются по‑разному.

### Термины

- **Append‑only лог** — структура, куда новые записи **только дописываются в конец**, без изменения уже записанных.
- **Commit log / WAL** — журнал в СУБД (PostgreSQL WAL, MySQL binlog, SQL Server transaction log, Oracle redo log), куда пишутся изменения перед их применением.
- **Event store** — хранилище, где **каждое изменение системы — это событие**, а не “новое состояние строки”.
- **Event sourcing** — архитектурный паттерн: состояние рассчитывается как **свертка** (fold) по потоку событий.
- **Kafka log** — распределённый persistent лог сообщений с репликацией и retention‑политикой.

### Теория и правила

#### 1) Интуиция: записывать историю, а не только текущее состояние

Когда мы храним просто “текущее состояние” (например, таблица `accounts` с балансами), мы:

- в любой момент видим **только сейчас**;
- прошлое “размазывается”: мы не знаем, **как** мы пришли к этому состоянию.

Append‑only лог и event sourcing говорят:

> “Давай сначала **запишем все шаги**, которые привели систему к состоянию,  
> а уже потом, если нужно, посчитаем текущее состояние из этих шагов”.

По шагам:

1. Мы фиксируем **каждое изменение** как событие: “пользователь зарегистрировался”, “счёт пополнен на 100”, “заказ оплачен”.
2. Все эти события складываются в **упорядоченный лог** (по времени/offset’у).
3. Чтобы узнать текущее состояние:
   - мы **считываем лог** и применяем события по очереди;
   - либо периодически сохраняем **снапшоты**, чтобы не пересчитывать с нуля.

#### 2) Commit log / WAL в СУБД vs “business‑лог”

Важно не путать два уровня:

1. **Системный commit log / WAL**:
   - PostgreSQL WAL, MySQL binlog, Oracle redo, SQL Server transaction log;
   - пишется **всегда**, прозрачно для приложения;
   - его цель:
     - durability (восстановление после крэша);
     - репликация (передача изменений на реплики);
     - иногда — CDC (change data capture).
2. **Прикладной event log / event store**:
   - таблица или хранилище, где мы осознанно записываем **бизнес‑события**;
   - его цель:
     - модель event sourcing;
     - аудит;
     - повторное проигрывание событий для аналитики или пересборки состояния.

Можно думать так:

- WAL — это “чёрный ящик” для самой БД: она с его помощью **защищает себя**.
- Event store — это “журнал истории” для твоего **бизнес‑домена**.

#### 3) Kafka log и compaction vs LSM

Kafka log:

- это **длинная лента сообщений**, поделённая на партиции;
- каждое сообщение имеет **offset** (позицию);
- retention:
  - по времени (хранить N дней);
  - по размеру (не больше X ГБ);
  - compaction по ключу (оставить последние значения по ключу).

LSM‑дерево:

- внутренне тоже опирается на идею “append‑only + compaction”, но:
  - поверх этого строится **индекс по ключам**;
  - organize уровни и SSTable;
  - чтения — по ключам и диапазонам.

Связь:

- и Kafka‑log, и LSM:
  - пишут **последовательно**;
  - убирают старые версии/ события фоном (retention/compaction);
- но Kafka — про **стрим сообщений**,  
  а LSM — про **индексированное key‑value‑хранилище**.

#### 4) Event sourcing: как это работает шаг за шагом

Рассмотрим упрощённый пример для оплаты заказов.

1. **Определяем события домена**:
   - `OrderCreated(orderId, userId, amount, createdAt)`
   - `OrderPaid(orderId, paidAt)`
   - `OrderCancelled(orderId, cancelledAt, reason)`
2. **Записываем только события**:
   - в таблицу `order_events` или в Kafka‑topic:

```sql
CREATE TABLE order_events (
  event_id      BIGSERIAL PRIMARY KEY,
  order_id      BIGINT      NOT NULL,
  event_type    TEXT        NOT NULL,
  payload       JSONB       NOT NULL,
  occurred_at   TIMESTAMPTZ NOT NULL
);
```

3. **Состояние заказа** рассчитывается как функция от событий**:
   - если последним событием по `order_id` был `OrderPaid` → заказ оплачен;
   - если `OrderCancelled` → отменён;
   - если только `OrderCreated` → ожидает оплаты.
4. **Снапшоты**:
   - чтобы не пересчитывать всё с нуля,  
     мы периодически сохраняем “срез” (`order_state`), рассчитанный из событий:

```sql
CREATE TABLE order_state (
  order_id    BIGINT PRIMARY KEY,
  user_id     BIGINT,
  amount      NUMERIC,
  status      TEXT,
  updated_at  TIMESTAMPTZ
);
```

5. **При восстановлении**:
   - считываем снапшоты;
   - догоняем недостающими событиями из `order_events`.

### Простыми словами

- **Commit log / WAL** — это **внутренний дневник БД**, в который она записывает “что я делала с данными”, чтобы в случае аварии/репликации всё восстановить.
- **Event store** — это **дневник бизнеса**: “что происходило с заказом/счётом/пользователем”.
- **Event sourcing** — это подход “мы *никогда* не теряем историю, а текущее состояние — это просто результат применения всех событий”.
- **Kafka log** — это “очень надёжная очередная тетрадка”, в которой события хранятся в порядке времени и могут дочитываться многими потребителями.

### Картинка в голове

Представь:

- Снизу — **LSM‑хранилище** (или обычная БД) с WAL: это фундамент, который надёжно пишет всё на диск.
- Над ним — **таблица event‑store** или Kafka — “лента” событий.
- Ещё выше — **сервисы**, которые:
  - читают события;
  - поддерживают свои проекции (срезы состояния);
  - реагируют на изменения.

WAL защищает **физические данные**, event‑store описывает **бизнес‑историю**, а LSM/compaction/retention помогают всё это хранить эффективно.

### Как запомнить

> **WAL/commit log = тех. журнал БД** (durability, репликация).  
> **Event log/event store = журнал бизнес‑событий** (источник истины).  
> **Event sourcing = “состояние = свертка(события)”**.

И ещё:

> **Kafka log и LSM похожи по идее (append‑only + compaction),  
> но решают разные задачи: стриминг vs индексированное key‑value‑хранение.**

### Примеры

- **Kafka + RocksDB в stream processing**:
  - Kafka хранит поток событий (лог);
  - Flink/Kafka Streams используют RocksDB как **state store**:
    - ключ — идентификатор агрегата/ключа;
    - значение — текущее состояние, рассчитанное из событий.
- **PostgreSQL + event store в таблице**:
  - приложение пишет бизнес‑события в таблицу `events` (append‑only);
  - сам PostgreSQL пишет WAL для durability;
  - аналитика и другие сервисы читают `events` для построения витрин.

### Практика / реальные сценарии

- **Финтех/банкинг**:
  - события переводов, начислений, комиссий хранятся как лог операций;
  - состояние счёта — агрегат по событиям;
  - LSM‑подход (внутри TiKV/Cassandra/ClickHouse) помогает выдерживать поток событий.
- **Логирование и аудит**:
  - аудиторские события пишутся в append‑only‑таблицу или Kafka;
  - позднее можно “отмотать” историю и понять, что происходило, даже если состояние давно изменилось.

### Типичные ошибки

- Путать WAL/commit log с “обычной таблицей” и пытаться читать его как API.
- Строить event‑store на “случайных UPDATE/DELETE” вместо строгого append‑only.
- Пытаться использовать Kafka как “обычную БД” (без проектирования state store/проекций).
- Думать, что event sourcing автоматически “решит все проблемы” с историей, не учитывая стоимость хранения и сложность миграций схемы событий.

### Что будет, если…

- …воспринимать event‑store как обычную таблицу состояния и делать в нём UPDATE/DELETE?
  - Ты потеряешь историчность и возможность корректно пересчитать состояние или восстановить прошлое; event sourcing перестанет работать как модель.
- …пытаться напрямую использовать WAL/commit log БД как основной интерфейс для бизнес‑логики?
  - WAL не стабилен как публичный API (версия/формат/оптимизации могут меняться), и это сильно усложнит архитектуру и миграции; его правильная роль — внутренняя механика БД и канал для репликации/CDC, а не бизнес‑хранилище.

### Проверь себя

1. В чём ключевое отличие commit log (WAL) в СУБД от event‑store в бизнес‑архитектуре?

<details><summary>Ответ</summary>
Commit log/WAL — это внутренний технический журнал СУБД для durability и репликации: он описывает низкоуровневые изменения страниц/строк и не предназначен как публичное API. Event‑store — это осознанно спроектированное хранилище бизнес‑событий, с чёткой схемой и семантикой домена, из которого строятся проекции и восстанавливается состояние.
</details>

2. Почему Kafka log и LSM‑дерево можно считать “родственниками по идее”, но не по назначению?

<details><summary>Ответ</summary>
Потому что оба опираются на append‑only запись и фоновую “уборку” (compaction/retention), но Kafka хранит последовательность сообщений для стриминга и подписчиков, а LSM строит поверх append‑only структуры **индексированное key‑value‑хранилище** с быстрыми point‑запросами и диапазонами.
</details>

3. Как кратко сформулировать суть event sourcing?

<details><summary>Ответ</summary>
Состояние системы **не хранится как единственная “текущая строчка”**, а выводится как результат применения **полной последовательности событий**; события — источник истины, а таблицы/проекции — производные.
</details>

### Запомните

- Append‑only логи (WAL, Kafka, event‑store) и LSM‑деревья строятся вокруг одной идеи — **“не изменять прошлое, а добавлять новые записи и убирать старое фоном”**.
- Разница в том, **для чего** они используются:  
  WAL — для надёжности и реплики самой БД, Kafka — для стриминга, event‑store — для доменной истории, LSM — для индексированного хранения ключ→значение.  
- Понимание этих ролей помогает проектировать архитектуру, в которой каждый инструмент делает “свою работу”, а не пытается заменить остальные.
