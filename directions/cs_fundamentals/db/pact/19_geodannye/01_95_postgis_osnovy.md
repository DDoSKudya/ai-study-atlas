## 95. PostGIS: основы

### Цель раздела

Понять, **что такое PostGIS**, какие типы и функции он добавляет к PostgreSQL, что такое `geometry` и `geography`, как устроены SRID и EPSG‑коды и как перепроецировать геоданные, чтобы корректно считать расстояния и площади.

### В этом разделе главное (три строки)

1. **PostGIS — это расширение PostgreSQL**, которое превращает базу в GIS‑движок: новые типы (`geometry`, `geography`), функции `ST_*`, индексы.
2. **SRID и системы координат критически важны**: без них база не понимает, «где» находится точка, и расстояния могут считаться неправильно.
3. **Создание геометрий стандартно и воспроизводимо**: через WKT (`ST_GeomFromText`) или функции (`ST_MakePoint`), с явным указанием SRID.

### Термины

- **PostGIS** — расширение PostgreSQL для геопространственных данных.
- **`geometry`** — тип для геометрий на плоскости; хранит координаты в заданной системе координат (SRID).
- **`geography`** — тип для геометрий на сфере/эллипсоиде; функции учитывают кривизну Земли.
- **SRID (Spatial Reference System Identifier)** — целое число, идентификатор системы координат (например, 4326, 3857).
- **EPSG код** — стандартный набор SRID, определённый организацией EPSG (например, EPSG:4326).
- **WGS84 (EPSG:4326)** — глобальная система координат в градусах широты/долготы; используется в GPS.
- **Web Mercator (EPSG:3857)** — проекция, в которой работают Google Maps, OpenStreetMap и др.; координаты в метрах.
- **`ST_Transform`** — функция преобразования геометрии из одного SRID в другой.
- **WKT (Well-Known Text)** — текстовый формат описания геометрий, например `POINT(30 50)` или `POLYGON((...))`.
- **WKB (Well-Known Binary)** — бинарный формат той же геометрии; компактный, удобен для обмена между системами; создание через `ST_GeomFromWKB`/`ST_GeomFromEWKB`, вывод через `ST_AsBinary`/`ST_AsEWKB`.

### Теория и правила

- **Установка PostGIS.**
  - На уровне кластера PostgreSQL (через пакетный менеджер) ставятся файлы расширения.
  - В конкретной базе данных ты активируешь PostGIS командой:
    ```sql
    CREATE EXTENSION IF NOT EXISTS postgis;
    ```
  - После этого появляются новые типы (`geometry`, `geography`) и множество функций `ST_*`.
  - **Версии PostGIS (2.x, 3.x).**
    - PostGIS развивается отдельно от PostgreSQL: бывают ветки 2.x (стабильная для старых PG) и 3.x (актуальная, с улучшениями производительности и новыми возможностями).
    - Узнать установленную версию:
      ```sql
      SELECT postgis_version();
      -- или
      SELECT extversion FROM pg_extension WHERE extname = 'postgis';
      ```
    - При установке через пакетный менеджер ОС версия обычно привязана к версии PostgreSQL (например, `postgresql-15-postgis-3`). Для новых проектов разумно брать PostGIS 3.x.
    - Разница 2.x и 3.x для изучения: основные типы и функции те же; в 3.x улучшена работа с индексами и часть операций быстрее. Документация и примеры из туториалов подходят к обеим веткам.

- **Типы `geometry` и `geography`.**
  - `geometry`:
    - работает в произвольной системе координат (любой SRID);
    - координаты — абстрактные числа (часто метры, но не всегда);
    - операции быстрые и гибкие, но длины/площади считаются в единицах системы координат.
  - `geography`:
    - всегда интерпретируется как координаты на поверхности Земли;
    - удобно, когда есть широта/долгота (WGS84) и нужны расстояния в метрах;
    - вычисления тяжелее (учёт кривизны), часть функций ограничена.

- **SRID и системы координат.**
  - SRID говорит PostGIS: **в какой системе координат задана геометрия**.
  - Например:
    - 4326 — WGS84 (широта/долгота в градусах),
    - 3857 — Web Mercator (x/y в метрах).
  - Если у тебя есть координаты в градусах, но ты хочешь измерять расстояния в метрах и строить буферы, удобно:
    - хранить как `geometry` в «метровой» проекции (например, 3857);
    - или хранить как `geography` и использовать специальные функции.

- **Преобразование координат: `ST_Transform`.**
  - Если геометрия записана в одном SRID, а тебе нужен другой:
    ```sql
    SELECT ST_Transform(geom, 3857) AS geom_mercator
    FROM my_table;
    ```
  - Важно:
    - сначала **убедиться, что SRID у геометрии задан верно** (например, `ST_SRID(geom)` вернёт 4326);
    - затем трансформировать в нужный SRID.

- **Создание геометрий.**
  - Из WKT:
    ```sql
    SELECT ST_GeomFromText('POINT(30 50)', 4326);
    ```
  - Через функции:
    ```sql
    SELECT ST_SetSRID(ST_MakePoint(30, 50), 4326);
    ```
  - Разница:
    - WKT удобно хранить/логировать/передавать;
    - `ST_MakePoint` и друзья удобны в запросах и генерации.
  - **Формат WKB (Well-Known Binary).**
    - WKB — это **бинарное** представление той же геометрии (последовательность байт). Человеку читать неудобно, зато компактно и быстро обрабатывать программами.
    - Создать геометрию из WKB:
      ```sql
      SELECT ST_GeomFromEWKB(bytea_column);
      -- или с явным SRID:
      SELECT ST_GeomFromWKB(decode('0101000000...', 'hex'), 4326);
      ```
    - Получить WKB из геометрии (например, для передачи в другой сервис или в файл):
      ```sql
      SELECT ST_AsBinary(geom) AS wkb_bytes,
             ST_AsEWKB(geom) AS ewkb_with_srid
      FROM places
      LIMIT 1;
      ```
    - Когда использовать WKB: обмен с приложениями (драйверы часто умеют передавать геометрию как WKB), бэкапы, когда важна компактность. Для логов и отладки удобнее WKT (`ST_AsText`).

- **Хранение геоданных.**
  - Обычно в таблице есть колонка `geom geometry(Point, 4326)` или подобная:
    ```sql
    CREATE TABLE places (
      id   serial PRIMARY KEY,
      name text NOT NULL,
      geom geometry(Point, 4326) -- точка в WGS84
    );
    ```
  - Тип уточняется: `geometry(<подтип>, <SRID>)`, что помогает ловить ошибки (нельзя случайно записать линию в колонку для точки, или SRID 3857 в колонку 4326).

### Пошагово: включаем PostGIS и создаём первую точку

1. **Подключаемся к базе PostgreSQL.**
   ```sql
   \c mydb
   ```

2. **Проверяем, доступен ли PostGIS.**
   ```sql
   SELECT name, default_version, installed_version
   FROM pg_available_extensions
   WHERE name = 'postgis';
   ```

3. **Устанавливаем расширение в текущую БД.**
   ```sql
   CREATE EXTENSION IF NOT EXISTS postgis;
   ```

4. **Создаём таблицу с геометрией точки.**
   ```sql
   CREATE TABLE places (
     id   serial PRIMARY KEY,
     name text NOT NULL,
     geom geometry(Point, 4326)
   );
   ```

5. **Вставляем точку (допустим, координаты какого‑то объекта).**
   ```sql
   INSERT INTO places (name, geom)
   VALUES (
     'Магазин у метро',
     ST_SetSRID(ST_MakePoint(37.620393, 55.753960), 4326) -- (долгота, широта)
   );
   ```

6. **Проверяем, что всё работает.**
   ```sql
   SELECT id, name, ST_AsText(geom), ST_SRID(geom)
   FROM places;
   ```
   Ты увидишь WKT‑представление (`POINT(37.620393 55.75396)`) и SRID 4326.

### Простыми словами

PostGIS — это как **«мозги карт» внутри базы**.  
Без него PostgreSQL видит только числа и строки; с ним он понимает, что «эти числа — точка на Земле», может посчитать расстояние между точками, понять, лежит ли объект внутри района, и т.д.  
SRID — это как **подпись на карте**: «эта карта в градусах» или «эта в метрах». Если подпись неправильная, всё дальше ломается: расстояния будут неверными, буферы «кривыми».

### Картинка в голове / Как запомнить

Представь себе **слой карты в GIS‑программе**: у него есть проекция, и на нём нарисованы точки/линии/полигоны.  
PostGIS делает то же самое, только слой — это таблица, объекты — строки, а геометрия — колонка `geometry(...)` с правильным SRID.  
`ST_Transform` — это как «переключить проекцию» слоя, чтобы мерить расстояния/площади удобно.

### Проверь себя

1. **Чем отличается `geometry` от `geography` в PostGIS?**
   <details><summary>Ответ</summary>
   `geometry` работает в произвольной системе координат и считает расстояния в единицах этой системы (часто в метрах, но зависит от SRID).  
   `geography` всегда интерпретируется как координаты на поверхности Земли, функции сразу дают расстояния/площади «по шару» в метрических единицах, но вычисления дороже и набор функций ограниченнее.
   </details>

2. **Зачем указывать SRID при создании геометрии?**
   <details><summary>Ответ</summary>
   Чтобы база знала, в какой системе координат заданы точки/линии/полигоны.  
   Без SRID PostGIS не может корректно трансформировать геометрию в другие проекции и может неправильно считать расстояния и площади.
   </details>

3. **Чем удобен формат WKT (`ST_GeomFromText`) по сравнению с `ST_MakePoint`?**
   <details><summary>Ответ</summary>
   WKT удобен для хранения и обмена (это человекочитаемая строка, которую можно логировать, хранить во внешних системах, копировать).  
   Внутри БД обычно используют функции `ST_MakePoint` и т.п., но WKT часто полезен для дебага и интеграций.
   </details>

#### Проверь себя: ST_Transform и SRID

4. **Перед вызовом `ST_Transform(geom, 3857)` что нужно проверить и почему?**
   <details><summary>Ответ</summary>
   Нужно убедиться, что у `geom` задан правильный исходный SRID (например, через `ST_SRID(geom)`). Если SRID не задан или задан неверно, перепроецирование даст неверный результат: «мусор на входе — мусор на выходе».
   </details>

5. **В какой системе координат будут координаты после `ST_Transform(geom, 3857)`?**
   <details><summary>Ответ</summary>
   В Web Mercator (EPSG:3857): координаты в метрах, пригодны для расчёта расстояний и буферов в метрах в пределах отображаемой области. Искажения растут к полюсам, поэтому для глобальных задач часто предпочитают `geography`.
   </details>

#### Проверь себя: установка и версии PostGIS

6. **Как узнать, какая версия PostGIS установлена в базе?**
   <details><summary>Ответ</summary>
   Выполнить запрос: `SELECT postgis_version();` или `SELECT extversion FROM pg_extension WHERE extname = 'postgis';`.  
   Версия покажет ветку (2.x или 3.x) и номер.
   </details>

7. **Почему расширение нужно включать отдельно в каждой базе данных, а не один раз на кластер?**
   <details><summary>Ответ</summary>
   Файлы расширения лежат на уровне кластера, но объекты (типы, функции) создаются **в конкретной базе** при выполнении `CREATE EXTENSION`. Так каждая БД может независимо решать, нужен ли ей PostGIS, и в одной БД может быть PostGIS, в другой — нет.
   </details>

8. **Чем PostGIS 3.x отличается от 2.x для того, кто только начинает?**
   <details><summary>Ответ</summary>
   Основные типы и функции те же; документация и примеры применимы к обеим веткам. В 3.x улучшена работа с индексами и производительность части операций. Для новых проектов разумно выбирать PostGIS 3.x.
   </details>

#### Проверь себя: WKT и WKB

9. **В каких случаях лучше использовать WKB вместо WKT?**
   <details><summary>Ответ</summary>
   WKB удобен, когда важна компактность (бинарные данные меньше текста), обмен с приложениями или драйверами (многие передают геометрию в WKB), бэкапы. Для логов, отладки и человекочитаемого обмена лучше WKT (`ST_AsText`).
   </details>

10. **Как из геометрии получить бинарное представление с учётом SRID?**
    <details><summary>Ответ</summary>
    Использовать `ST_AsEWKB(geom)` — Extended WKB включает SRID. Обычный `ST_AsBinary(geom)` возвращает WKB без SRID.
    </details>

#### Проверь себя: хранение и типы колонок

11. **Зачем указывать подтип и SRID в определении колонки: `geometry(Point, 4326)`?**
   <details><summary>Ответ</summary>
   Чтобы СУБД проверяла данные: в колонку для точки нельзя будет записать линию или полигон, а геометрия с другим SRID (например, 3857) не попадёт в колонку 4326. Это снижает риск ошибок при вставке и обновлении.
   </details>

12. **Что вернёт `ST_SRID(geom)`, если при создании геометрии SRID не был указан?**
    <details><summary>Ответ</summary>
    Обычно вернётся 0 или неопределённое значение. Таким данным нельзя доверять для перепроецирования и расчёта расстояний — SRID нужно проставлять явно при вставке.
    </details>

### Запомните

- PostGIS — это **расширение PostgreSQL**, без него `geometry`/`geography` и `ST_*` недоступны.
- SRID — это **ключ к правильной геометрии**: без него или с неправильным значением нельзя доверять расстояниям и площадям.
- Геометрии удобно создавать через `ST_MakePoint`/`ST_GeomFromText` **с явным указанием SRID**.

#### Если мозг «закипел» — пересказ ещё проще

- **PostGIS** — это как плагин к PostgreSQL, который говорит: «Эти две колонки — не просто числа, это точка на планете».  
  До установки PostGIS база не умеет с этим работать как с картой.
- **SRID** — это как подпись под картинкой:  
  «Эта карта нарисована в градусах широты/долготы» или «в метрах в проекции X».  
  Без подписи невозможно честно мерить линейкой.
- **`geometry`** — ты сам решаешь, в какой системе координат работать. Быстрее, но ответственность на тебе.  
- **`geography`** — PostGIS «знает», что это точки на Земле, и сам считает расстояния в метрах, но чуть медленнее.
- **`ST_Transform`** — это «переключатель» между картами: та же точка, но на другой карте, где удобно мерить расстояния.

Если совсем коротко:  
**Включи PostGIS → храни координаты в колонке `geometry(...)`/`geography(...)` с правильным SRID → тогда все остальные функции начнут делать «правильную геомагию».**

#### Частые вопросы про SRID и координаты

1. **Почему вообще нужны разные SRID, нельзя ли один «универсальный»?**  
   В реальном мире карты рисуют по‑разному:
   - иногда важно, чтобы **углы и направления** были менее искажены (навигация);
   - иногда важно, чтобы **площади** считались честно (статистика по территориям);
   - иногда нужно, чтобы **всё влезло в квадратный прямоугольник экрана**.  
   Разные SRID — это разные ответы на вопрос: «Как именно мы сплющили Землю на плоскость и что хотим сохранить более точным?».

2. **Я получил из API широту/долготу. Что мне с ними делать?**
   - Сначала сохраняешь как **точку в 4326**:
     ```sql
     ST_SetSRID(ST_MakePoint(lon, lat), 4326)
     ```
   - Дальше варианты:
     - либо хранишь так же (и при вычислениях используешь `geography`/`ST_DistanceSphere`);
     - либо делаешь **рабочую колонку в метровой проекции** (например, 3857), куда пишешь `ST_Transform(..., 3857)`, и дальше все «метровые» вычисления делаешь там.

3. **Что будет, если я перепутаю местами широту и долготу?**  
   База не «увидит» ошибку: это просто два числа.  
   Но точка окажется в другом месте планеты (часто вообще в океане).  
   Поэтому очень полезно:
   - завести **маленький скрипт‑проверку**, который вытаскивает несколько точек и смотрит их на карте (например, через GeoJSON и любой онлайн‑просмотрщик);
   - в «боевых» таблицах иметь простую проверку диапазонов: `lat BETWEEN -90 AND 90`, `lon BETWEEN -180 AND 180`.

4. **Как понять, что у геометрии правильный SRID?**
   - Команда:
     ```sql
     SELECT ST_SRID(geom) FROM places LIMIT 10;
     ```
   - Если там `0` или странное значение — значит, при загрузке геоданных **кто‑то забыл проставить SRID**, и этим данным доверять нельзя до исправления.

#### Мини‑эксперимент: 15 минут практики

Если хочешь не просто прочитать, а **почувствовать разницу**, сделай мини‑практику в любой тестовой БД:

1. Включи PostGIS (если ещё нет):
   ```sql
   CREATE EXTENSION IF NOT EXISTS postgis;
   ```
2. Создай таблицу с двумя колонками — «сырые числа» и «геометрия»:
   ```sql
   CREATE TABLE geo_test (
     id        serial PRIMARY KEY,
     lat       double precision,
     lon       double precision,
     geom_geo  geography(Point, 4326),
     geom_geom geometry(Point, 4326)
   );
   ```
3. Вставь одну и ту же точку во все три вида:
   ```sql
   INSERT INTO geo_test (lat, lon, geom_geo, geom_geom)
   VALUES (
     55.753960,
     37.620393,
     ST_SetSRID(ST_MakePoint(37.620393, 55.753960), 4326)::geography,
     ST_SetSRID(ST_MakePoint(37.620393, 55.753960), 4326)
   );
   ```
4. Скопируй строку ещё пару раз, слегка изменив координаты (можно руками или через `INSERT ... SELECT ...`).
5. Попробуй посчитать расстояние между двумя строками:
   ```sql
   -- через geography (ожидаешь метры)
   SELECT ST_Distance(a.geom_geo, b.geom_geo) AS meters
   FROM geo_test a, geo_test b
   WHERE a.id = 1 AND b.id = 2;

   -- через geometry (в градусах)
   SELECT ST_Distance(a.geom_geom, b.geom_geom) AS degrees
   FROM geo_test a, geo_test b
   WHERE a.id = 1 AND b.id = 2;
   ```
6. Сравни цифры и **осознанно проговори себе**:
   - «Вот здесь у меня градусы, а здесь — метры».

После такого мини‑эксперимента SRID и разница `geometry`/`geography` обычно «приклеиваются» к голове гораздо лучше, чем от одной теории.

### Типичная ошибка / Что будет, если…

- **Ошибка:** хранить координаты как два `numeric` (`lat`, `lon`) без PostGIS и SRID, а потом пытаться считать расстояния «по формуле» в каждом запросе.  
  **Последствия:** сложные и медленные запросы, куча дублирующей логики, сложно использовать индексы и проверять корректность данных.  
  **Лучше:** использовать тип `geometry`/`geography` и функции PostGIS.
