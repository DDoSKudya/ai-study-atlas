## 97. Пространственные запросы

### Цель раздела

Научиться **задавать вопросы к геоданным на языке SQL**, используя пространственные функции: находить объекты в радиусе, проверять попадание в полигон, пересечение маршрутов, строить буферы и объединять/пересекать геометрии.

### В этом разделе главное (три строки)

1. Пространственные функции `ST_*` позволяют задавать **геологические вопросы**: «рядом», «внутри», «пересекается», «на расстоянии не более X».
2. Для **расстояний** нужно понимать, в каких единицах живут координаты (градусы или метры) и при необходимости использовать `ST_DistanceSphere`/`geography` или `ST_Transform`.
3. Функции вроде `ST_Buffer`, `ST_Union`, `ST_Intersection`, `ST_Envelope` помогают **строить зоны и агрегировать геоданные**.

### Термины

- **`ST_Distance`** — расстояние между двумя геометриями в единицах их системы координат.
- **`ST_DistanceSphere`** — приближённое расстояние по сфере (в метрах) между точками с широтой/долготой.
- **`ST_Within`** — возвращает `true`, если одна геометрия полностью содержится внутри другой.
- **`ST_Contains`** — обратное к `ST_Within`: содержит ли одна геометрия другую.
- **`ST_Intersects`** — есть ли любая общая точка между геометриями (пересечение).
- **`ST_Overlaps`** — геометрии пересекаются, но ни одна не содержится полностью в другой.
- **`ST_Buffer`** — строит зону (буфер) вокруг геометрии на заданное расстояние.
- **`ST_Union`** — объединяет геометрии в одну (совмещает области).
- **`ST_Intersection`** — общая часть двух геометрий.
- **`ST_Envelope`** — минимальный прямоугольник, содержащий геометрию (bounding box).
- **`ST_Area`** — площадь полигона (в единицах координат или в м² для geography/метровой проекции).
- **`ST_Length`** — длина линии (аналогично: градусы или метры).

### Теория и правила

- **Расстояния: `ST_Distance` и `ST_DistanceSphere`.**
  - `ST_Distance(geom1, geom2)`:
    - возвращает расстояние в единицах координат (метры, если SRID «метровый»; градусы, если WGS84 без трансформации).
  - `ST_DistanceSphere`:
    - принимает точки с широтой/долготой (обычно SRID 4326) и считает примерно «по шару» в метрах:
      ```sql
      SELECT
        ST_DistanceSphere(
          ST_MakePoint(lon1, lat1),
          ST_MakePoint(lon2, lat2)
        ) AS meters;
      ```

- **Отношения «внутри / содержит / пересекается».**
  - `ST_Within(a, b)` — `a` полностью внутри `b`.
  - `ST_Contains(b, a)` — `b` содержит `a` (обратное к `ST_Within`).
  - `ST_Intersects(a, b)` — есть любое пересечение (включая граничное соприкосновение).
  - `ST_Overlaps(a, b)` — области частично пересекаются, но ни одна не содержит другую.

- **Буферы и зоны.**
  - `ST_Buffer(geom, r)` строит геометрию, представляющую все точки на расстоянии ≤ r от исходной.
  - Применение:
    - зона радиуса N метров вокруг точки/линии (если SRID «метровый» или `geography`);
    - поиск объектов «в радиусе X».

- **Объединения и пересечения.**
  - `ST_Union`:
    - объединяет геометрии (например, несколько полигонов районов в одну зону):
      ```sql
      SELECT ST_Union(geom)
      FROM districts
      WHERE city_id = 1;
      ```
  - `ST_Intersection(a, b)`:
    - возвращает общую часть геометрий (например, пересечение зоны доставки и административного района).

- **Bounding box: `ST_Envelope`.**
  - Возвращает минимальный прямоугольник (ось‑ориентированный) вокруг геометрии.
  - Часто используется для быстрой предварительной фильтрации:
    - сначала грубо отсеять объекты по пересечению прямоугольников;
    - потом точнее проверять отношениями `ST_Intersects` и т.п.

- **Площадь и длина: `ST_Area`, `ST_Length`.**
  - **`ST_Area(geom)`** — площадь полигона (или мультиполигона). Единицы измерения зависят от системы координат:
    - для `geometry` в градусах (SRID 4326) результат будет в **квадратных градусах** — для реальных площадей в метрах не подходит;
    - для `geometry` в метровой проекции (например, 3857) или для **`geography`** — площадь в **квадратных метрах** (для geography — по эллипсоиду).
    - Пример (площадь в метрах через geography):
      ```sql
      SELECT ST_Area(geom::geography) AS area_m2
      FROM districts
      WHERE id = 1;
      ```
  - **`ST_Length(geom)`** — длина линии. Аналогично:
    - для `geometry` в градусах — длина в градусах (не метры);
    - для `geometry` в метровой проекции или для `geography` — длина в **метрах**.
    - Пример (длина маршрута в метрах):
      ```sql
      SELECT ST_Length(route_geom::geography) AS length_m
      FROM routes
      WHERE id = 1;
      ```
  - Итог: для площадей и длин в «нормальных» единицах либо используй `geography`, либо предварительно переводи геометрию в метровую проекцию (`ST_Transform(geom, 3857)` или подходящую для твоего региона).

### Пошагово: «объекты рядом» и «объекты внутри зоны»

Предположим, у нас есть таблицы:

```sql
CREATE TABLE shops (
  id   serial PRIMARY KEY,
  name text NOT NULL,
  geom geometry(Point, 4326)
);

CREATE TABLE districts (
  id   serial PRIMARY KEY,
  name text NOT NULL,
  geom geometry(Polygon, 4326)
);
```

1. **Найти все магазины в радиусе 1000 м от точки.**
   - Вариант через `geography`:
     ```sql
     WITH center AS (
       SELECT ST_SetSRID(ST_MakePoint(37.62, 55.75), 4326)::geography AS g
     )
     SELECT s.id, s.name
     FROM shops s, center c
     WHERE ST_Distance(s.geom::geography, c.g) <= 1000;
     ```

2. **Найти все магазины внутри заданного района.**
   ```sql
   SELECT s.id, s.name
   FROM shops s
   JOIN districts d ON d.name = 'Центральный район'
   WHERE ST_Within(s.geom, d.geom);
   ```

3. **Построить буфер вокруг дороги и найти объекты возле дороги.**
   Допустим, есть таблица `roads` с `LINESTRING` в метровой проекции `EPSG:3857`:
   ```sql
   -- буфер 50 м вокруг дороги с id = 1
   WITH road_buffer AS (
     SELECT ST_Buffer(geom, 50) AS geom
     FROM roads
     WHERE id = 1
   )
   SELECT s.id, s.name
   FROM shops s
   JOIN road_buffer b
     ON ST_Intersects(s.geom, b.geom);
   ```

4. **Найти площадь пересечения зоны доставки и района.**
   ```sql
   SELECT
     d.name AS district,
     ST_Area(ST_Intersection(d.geom, z.geom)::geography) AS area_m2
   FROM districts d
   CROSS JOIN (SELECT geom FROM delivery_zones WHERE id = 1) z;
   ```

### Простыми словами

Пространственные запросы — это как **вопросы к навигатору**:

- «Покажи все заправки в радиусе 5 км от меня» — это `ST_Distance` или `ST_Buffer` + `ST_Intersects`.
- «Покажи все адреса внутри этого района» — это `ST_Within`.
- «Покажи, где зона доставки пересекается с границами города» — это `ST_Intersection`.

Всё это формулируется как обычные SQL‑запросы, только вместо `=` и `>` ты используешь функции `ST_*`.

### Картинка в голове / Как запомнить

Представь, что у тебя есть **прозрачная плёнка с объектами** (магазины, дороги, районы).  
Функции:

- `ST_Within` — положить плёнку с точками поверх плёнки с полигонами и посмотреть, какие точки оказались внутри полигона;
- `ST_Buffer` — «распухание» объекта (точки/линии) до «колбасы» вокруг него;
- `ST_Intersection` — то, что осталось, когда наложил два слоя и вырезал общую часть.

### Проверь себя

1. **Почему нельзя вслепую использовать `ST_Distance` на WGS84 (EPSG:4326) и считать, что вернулись метры?**
   <details><summary>Ответ</summary>
   Потому что в WGS84 координаты в градусах, а `ST_Distance` возвращает расстояние в единицах координат.  
   Для метров нужно либо трансформировать геометрию в «метровую» проекцию (например, 3857) и считать там, либо использовать `geography`/`ST_DistanceSphere`.
   </details>

2. **В чём разница между `ST_Within` и `ST_Contains`?**
   <details><summary>Ответ</summary>
   `ST_Within(a, b)` — `a` полностью внутри `b`.  
   `ST_Contains(b, a)` — `b` содержит `a` (обратное отношение).  
   Часто эти функции дают одинаковый ответ, но важно помнить, кто «внутри», а кто «снаружи».
   </details>

3. **Зачем `ST_Envelope`, если есть `ST_Intersects`?**
   <details><summary>Ответ</summary>
   `ST_Envelope` даёт простой прямоугольник вокруг объекта, по которому проще и быстрее отбирать кандидатов (особенно вместе с индексом).  
   Часто сначала делают грубый отбор по пересечению прямоугольников, а потом точный — по `ST_Intersects`/`ST_Within`.
   </details>

#### Проверь себя: расстояния и единицы измерения

4. **В каких единицах вернёт расстояние `ST_Distance` для двух точек в SRID 4326 (WGS84)?**
   <details><summary>Ответ</summary>
   В **градусах** (единицах координатной системы 4326), а не в метрах. Для расстояния в метрах нужно использовать `geography` или `ST_DistanceSphere`, либо перевести геометрию в метровую проекцию и считать там.
   </details>

5. **Когда результат `ST_Area(geom)` будет в квадратных метрах?**
   <details><summary>Ответ</summary>
   Когда геометрия задана в метровой проекции (например, SRID 3857) или приведена к типу `geography`. Для `geometry` в WGS84 (4326) результат будет в квадратных градусах.
   </details>

#### Проверь себя: отношения и буферы

6. **Чем `ST_Intersects` отличается от `ST_Overlaps`?**
   <details><summary>Ответ</summary>
   `ST_Intersects` — есть любая общая точка (включая касание границ и полное вложение одного в другое). `ST_Overlaps` — геометрии пересекаются по площади, но ни одна не содержится полностью в другой (типичный случай — два пересекающихся полигона).
   </details>

7. **В каких единицах задаётся радиус в `ST_Buffer(geom, r)`?**
   <details><summary>Ответ</summary>
   В тех же единицах, что и система координат геометрии: для `geometry` в 4326 — в градусах (что неудобно); для `geography` или метровой проекции — в метрах. Для радиуса в метрах используй `geography` или предварительно `ST_Transform` в метровую проекцию.
   </details>

8. **Что вернёт `ST_Intersection(a, b)`, если геометрии не пересекаются?**
   <details><summary>Ответ</summary>
   Пустую геометрию (тип можно проверить через `ST_GeometryType` или `ST_IsEmpty`). Не NULL и не ошибка — именно пустая геометрия.
   </details>

### Запомните

- Пространственные функции `ST_*` делают SQL **языком геозапросов**.
- Для расстояний важно понимать единицы координат: градусы ≠ метры.
- `ST_Buffer`, `ST_Union`, `ST_Intersection`, `ST_Envelope` — это базовые строительные блоки сложных геозапросов.

#### Разложим пример «в радиусе 1 км» по шагам

Допустим, ты хочешь найти все кафе в радиусе 1000 метров от текущего местоположения пользователя.

1. **Исходные данные.**
   - У тебя есть точка пользователя: `user_lon`, `user_lat` (обычно из GPS).
   - Таблица `cafes(id, name, geom geometry(Point, 4326))`.

2. **Превращаем точку пользователя в геометрию.**
   ```sql
   SELECT ST_SetSRID(ST_MakePoint(:user_lon, :user_lat), 4326) AS user_point;
   ```

3. **Делаем запрос с приведением к `geography`, чтобы сразу работать в метрах.**
   ```sql
   WITH center AS (
     SELECT ST_SetSRID(ST_MakePoint(:user_lon, :user_lat), 4326)::geography AS g
   )
   SELECT c.id, c.name
   FROM cafes c, center cent
   WHERE ST_Distance(c.geom::geography, cent.g) <= 1000;
   ```

4. **Что здесь важно понять:**
   - `ST_MakePoint` + `ST_SetSRID` — мы говорим: «Это точка в системе координат 4326».
   - `::geography` — мы говорим PostGIS: «Считай это точкой на Земле, а не просто парой чисел».
   - `ST_Distance(..., ...) <= 1000` — спрашиваем: «Какие точки не дальше 1000 метров?».

Если этот пример расписывать словами, получается вопрос:  
**«PostGIS, вот точка пользователя и вот точки кафе. Дай мне все кафе, до которых не дальше 1000 метров, если считать расстояние по поверхности Земли».**

#### Частые вопросы про пространственные запросы

1. **`ST_Within` и `ST_Contains` — это одно и то же? Какой использовать?**  
   Логика одна и та же, но «с разных сторон»:
   - `ST_Within(a, b)` — «геометрия `a` целиком лежит внутри `b`»;
   - `ST_Contains(b, a)` — «геометрия `b` содержит `a`».  
   На практике выбирай ту формулировку, которая **понятнее тебе по смыслу** в конкретном запросе. Важно не перепутать, кто внутри, а кто снаружи.

2. **Когда лучше пользоваться `ST_DistanceSphere`, а когда `geography`?**  
   - Если ты **пока не хранишь данные как `geography`**, но хочешь посчитать дистанцию «по шару» для пары точек — удобно взять `ST_DistanceSphere`.  
   - Если у тебя **вся таблица уже хранит геоданные как `geography`**, проще сразу использовать `ST_Distance` по `geography` — меньше приведения типов и риск ошибок.

3. **Зачем вообще `ST_Buffer`, если можно просто мерить расстояние?**  
   Иногда удобнее мыслить **«зоной»**, а не «радиусом».  
   Например:
   - построить зону 500 м вокруг дороги (`ST_Buffer(road, 500)`) и посмотреть, какие дома туда попадают;
   - построить «шапку покрытия» мобильной вышки.  
   Внутри базы запросы вида «объекты пересекаются с этой зоной?» иногда пишутся и читаются проще, чем отдельные расстояния до каждого объекта.

4. **Как отлаживать сложный пространственный запрос, если ничего не понятно?**
   - Разбей запрос на части:
     1. Сначала посмотри **отдельно входные геометрии** через `ST_AsText` или в GIS‑клиенте.
     2. Потом посмотри **результат промежуточных операций** (`ST_Buffer`, `ST_Intersection`) тоже как WKT.
   - Для отладки часто помогают такие «диагностические» запросы:
     ```sql
     SELECT id, ST_AsText(geom), ST_IsValid(geom)
     FROM some_table
     LIMIT 5;
     ```
   - Если видишь, что геометрия «странная» или невалидна — сначала чини данные, потом логику запроса.

#### Мини‑проект: геофильтр для списка объектов

Представь, что у тебя есть простое веб‑приложение: список объектов (кафе, магазины, пункты выдачи), и пользователь хочет:

- фильтровать их «рядом со мной»;
- фильтровать «внутри выбранного района».

Пошаговый план:

1. **Схема БД.**
   ```sql
   CREATE TABLE poi (
     id   serial PRIMARY KEY,
     name text NOT NULL,
     geom geometry(Point, 4326)
   );

   CREATE TABLE regions (
     id   serial PRIMARY KEY,
     name text NOT NULL,
     geom geometry(Polygon, 4326)
   );
   ```

2. **Загрузка данных.**
   - При загрузке точек:
     ```sql
     INSERT INTO poi (name, geom)
     VALUES (
       :name,
       ST_SetSRID(ST_MakePoint(:lon, :lat), 4326)
     );
     ```
   - При загрузке регионов — через WKT/GeoJSON, но в итоге тоже получаются `geometry(Polygon, 4326)`.

3. **Фильтр «рядом со мной».**
   - На API приходят `lat`, `lon`, `radius_m`.
   - В SQL:
     ```sql
     WITH center AS (
       SELECT ST_SetSRID(ST_MakePoint(:lon, :lat), 4326)::geography AS g
     )
     SELECT p.id, p.name
     FROM poi p, center c
     WHERE ST_Distance(p.geom::geography, c.g) <= :radius_m;
     ```

4. **Фильтр «внутри региона».**
   - Пользователь выбирает регион по имени или ID.
   - В SQL:
     ```sql
     SELECT p.id, p.name
     FROM poi p
     JOIN regions r ON r.id = :region_id
     WHERE ST_Within(p.geom, r.geom);
     ```

5. **Добавляем индексы и проверяем планы (это уже мостик к разделу §98).**
   ```sql
   CREATE INDEX idx_poi_geom_gist
   ON poi
   USING GIST (geom);

   CREATE INDEX idx_regions_geom_gist
   ON regions
   USING GIST (geom);
   ```

   После этого `EXPLAIN ANALYZE` по запросам покажет `Index Scan`/`Bitmap Index Scan`, а время выполнения на больших таблицах резко сократится.

### Типичная ошибка / Что будет, если…

- **Ошибка:** искать «объекты в радиусе 5 км», сравнивая `ST_Distance` по WGS84 прямо с 5000.  
  **Последствия:** результат будет случайным: в разных широтах «5 градусов» — разные реальные расстояния, а функция вернула градусы, а не метры.  
  **Лучше:** использовать `geography`/`ST_DistanceSphere` или трансформировать в метровую проекцию.
