## 98. Пространственные индексы

### Цель раздела

Понять, **как ускорять пространственные запросы** с помощью индексов GiST и SP-GiST, как они используются планировщиком PostgreSQL, как правильно писать `WHERE`‑условия, чтобы индекс сработал, и какие есть приёмы оптимизации.

### В этом разделе главное (три строки)

1. Для геоданных используются **GiST/SP-GiST индексы**, которые хранят не только сами координаты, но и «оболочки» (bounding box) геометрий.
2. Чтобы индекс сработал, **нужно писать запросы в форме, понятной планировщику**: использовать `ST_Intersects`, `&&` (bbox‑оператор) и не заворачивать колонку в произвольные функции.
3. Часто оптимизация — это **двухэтапный фильтр**: сначала отбор по bounding box (быстро по индексу), затем точная проверка отношениями `ST_*`.

### Термины

- **GiST (Generalized Search Tree)** — обобщённое древовидное индексное дерево, поддерживающее разные типы данных, в т.ч. геометрии.
- **SP-GiST (Space-Partitioned GiST)** — семейство индексов, которые разрезают пространство на части (подходит некоторым типам геоданных).
- **Bounding box (envelope)** — минимальный прямоугольник, который полностью содержит геометрию.
- **Оператор `&&`** — проверка пересечения bounding box двух геометрий.
- **`ST_Collect`** — агрегирует несколько геометрий в одну коллекцию.
- **`ST_Union` (агрегат)** — объединяет геометрии при агрегации (например, `ST_Union(geom)` по группе).

### Теория и правила

- **Зачем нужны пространственные индексы.**
  - Без индекса запрос `WHERE ST_Intersects(geom, ...)` вынужден проверять **каждую строку** — полный перебор.
  - На десятках тысяч строк это ещё терпимо, на миллионах — уже нет.
  - Индекс по `geometry`/`geography` позволяет быстро отобрать кандидатов по их bounding box.

- **Создание GiST индекса.**
  ```sql
  CREATE INDEX idx_places_geom_gist
  ON places
  USING GIST (geom);
  ```

- **Как индекс используется.**
  - В первом приближении:
    - индекс хранит bounding box геометрий;
    - при запросе `WHERE geom && :filter_geom` он быстро находит те строки, чьи bbox пересекаются с bbox фильтра.
  - Дальше (после отбора по индексу) применяется точная проверка геометрий (`ST_Intersects`, `ST_Within` и т.п.).

- **Оператор `&&` и комбинированный фильтр.**
  - Частый паттерн:
    ```sql
    WHERE geom && :filter_geom
      AND ST_Intersects(geom, :filter_geom)
    ```
  - `&&` хорошо ложится на индекс (bbox‑сравнение),
  - `ST_Intersects` отсекает ложные срабатывания, когда прямоугольники пересеклись, а сами геометрии — нет.

- **SP-GiST.**
  - Альтернатива GiST для некоторых пространственных иерархий (например, quad-tree).
  - Применяется реже, и в большинстве случаев **GiST достаточно**.

- **Пространственные агрегаты.**
  - `ST_Collect(geom)`:
    - собирает геометрии в одну коллекцию (например, все точки района).
  - `ST_Union(geom)` (агрегат):
    - объединяет геометрии в одну область (например, суммарная зона присутствия компании).

- **Быстрое построение прямоугольной области: `ST_MakeEnvelope`.**
  - Частая задача: задать **прямоугольник по координатам двух углов** (левая‑нижняя и правая‑верхняя точки).
  - Вместо того чтобы писать длинный WKT от руки, можно использовать:
    ```sql
    SELECT ST_MakeEnvelope(
      30.1, 60.1,   -- x_min, y_min (левая‑нижняя точка)
      30.9, 60.9,   -- x_max, y_max (правая‑верхняя точка)
      4326          -- SRID
    ) AS bbox;
    ```
  - Это вернёт **полигон‑прямоугольник**, который удобно использовать как фильтр:
    ```sql
    WITH bbox AS (
      SELECT ST_MakeEnvelope(30.1, 60.1, 30.9, 60.9, 4326) AS geom
    )
    SELECT p.*
    FROM big_poi p, bbox b
    WHERE p.geom && b.geom
      AND ST_Within(p.geom, b.geom);
    ```
  - Так ты получаешь:
    - компактное задание прямоугольника;
    - удобный объект для индекса (`&&` и `ST_Within` прекрасно с ним работают).

### Пошагово: индекс по геоданным и план запроса

1. **Создадим индекс по колонке `geom` в таблице `shops`.**
   ```sql
   CREATE INDEX idx_shops_geom_gist
   ON shops
   USING GIST (geom);
   ```

2. **Напишем запрос «магазины в заданном районе» и посмотрим план.**
   ```sql
   EXPLAIN ANALYZE
   SELECT s.id, s.name
   FROM shops s
   JOIN districts d ON d.name = 'Центральный район'
   WHERE s.geom && d.geom
     AND ST_Within(s.geom, d.geom);
   ```
   - В хорошем сценарии ты увидишь в плане использование `Index Scan` или `Bitmap Index Scan` по `idx_shops_geom_gist`.

3. **Использование индекса с `ST_MakeEnvelope`: прямоугольная область «на лету».**
   - Часто фильтрующая область задаётся не таблицей, а **параметрами** (например, видимая область карты: min_lon, min_lat, max_lon, max_lat). Тогда прямоугольник удобно строить через `ST_MakeEnvelope` прямо в `WHERE` — планировщик **автоматически** использует GiST-индекс по колонке `geom`:
   ```sql
   SELECT id, name
   FROM shops
   WHERE geom && ST_MakeEnvelope(37.5, 55.7, 37.8, 56.0, 4326)
     AND ST_Intersects(geom, ST_MakeEnvelope(37.5, 55.7, 37.8, 56.0, 4326));
   ```
   - Здесь: первый аргумент `ST_MakeEnvelope(x_min, y_min, x_max, y_max, srid)` — левый нижний угол, второй — правый верхний. Оператор `&&` по такому прямоугольнику отбирает кандидатов по индексу, `ST_Intersects` уточняет результат (для прямоугольника можно было бы обойтись и только `ST_Within`, но для единообразия с другими фильтрами часто пишут `ST_Intersects`). Важно: **не** оборачивай в функцию саму колонку `geom`, тогда индекс используется.

4. **Сравним с вариантом без индекса.**
   ```sql
   EXPLAIN ANALYZE
   SELECT s.id, s.name
   FROM shops s
   JOIN districts d ON d.name = 'Центральный район'
   WHERE ST_Within(s.geom, d.geom);
   ```
   - На больших данных Postgres может выбрать `Seq Scan`, и время выполнения будет заметно выше.

5. **Агрегируем геоданные: строим общую зону доставки.**
   ```sql
   SELECT ST_Union(geom) AS unified_zone
   FROM delivery_zones
   WHERE company_id = 1;
   ```

6. **Оптимизация сложных запросов.**
   - Использовать:
     - `&&` для bbox‑фильтра;
     - `ST_Envelope` и/или «ограничивающий прямоугольник» входной геометрии.
   - Избегать:
     - оборачивания `geom` в функции, которые ломают использование индекса (например, `ST_Transform(geom, ...)` в `WHERE` без отдельной «проекции» колонки).

### Простыми словами

Пространственный индекс — это **«карта мини‑прямоугольников»** вокруг объектов.  
Когда ты спрашиваешь: «какие объекты пересекаются вот с этой областью?», индекс сначала смотрит на прямоугольники: «кто хотя бы теоретически может попасть?», а потом уже база проверяет точную форму геометрий.  
Без индекса ей пришлось бы сравнивать «каждого с каждым».

### Картинка в голове / Как запомнить

Представь склад с коробками разного размера.  
GiST‑индекс — это **стеллаж с надписями**, какой коробки какого размера в каком ряду искать.  
Сначала ты по надписям находишь «ряд коробок нужного диапазона размеров» (bbox‑фильтр), а потом уже смотришь внутри, подходит ли конкретная коробка (точная проверка геометрии).

### Проверь себя

1. **Почему иногда полезно писать и `geom && :filter_geom`, и `ST_Intersects(geom, :filter_geom)` в одном `WHERE`?**
   <details><summary>Ответ</summary>
   Потому что `&&` хорошо оптимизируется индексом (bbox сравнение), быстро отбрасывая большинство нерелевантных объектов.  
   Но bbox даёт только приближённый ответ, поэтому потом нужен `ST_Intersects` для точной геометрической проверки.
   </details>

2. **Что произойдёт, если в `WHERE` написать `ST_Intersects(ST_Transform(geom, 3857), :filter)` при индексе по `geom` в SRID 4326?**
   <details><summary>Ответ</summary>
   Планировщик, скорее всего, **не сможет использовать индекс по `geom`**, потому что колонка обёрнута в функцию `ST_Transform`.  
   Запрос превратится в более дорогой последовательный проход по таблице (или будет использовать индекс неэффективно).
   </details>

3. **Когда стоит задуматься о SP-GiST, а не GiST?**
   <details><summary>Ответ</summary>
   Когда тип данных и характер запросов хорошо ложатся на структуру «разрезания пространства» (например, quad-tree) и есть профиль нагрузки, при котором SP-GiST показывает более высокую производительность.  
   Для большинства «обычных» геозадач GiST достаточно, и к SP-GiST имеет смысл переходить осознанно, после измерений.
   </details>

#### Проверь себя: создание и использование индекса

4. **Какой синтаксис используется для создания пространственного индекса по колонке `geom`?**
   <details><summary>Ответ</summary>
   `CREATE INDEX idx_name ON table_name USING GIST (geom);` — для GiST. Для SP-GiST: `USING SPGIST (geom)`.
   </details>

5. **Зачем в одном запросе писать и `geom && :bbox`, и `ST_Within(geom, :poly)`?**
   <details><summary>Ответ</summary>
   `&&` (пересечение bounding box) быстро отсекает заведомо неподходящие строки и хорошо использует индекс. Затем `ST_Within` даёт точную проверку по геометрии и отсекает ложные срабатывания, когда прямоугольники пересекаются, а сами объекты — нет.
   </details>

6. **Как с помощью `ST_MakeEnvelope` задать прямоугольник для фильтрации по видимой области карты (min_lon, min_lat, max_lon, max_lat)?**
   <details><summary>Ответ</summary>
   `ST_MakeEnvelope(min_lon, min_lat, max_lon, max_lat, 4326)` — первые четыре аргумента: x_min, y_min, x_max, y_max (левая нижняя и правая верхняя точки), пятый — SRID. Использовать в условии: `WHERE geom && ST_MakeEnvelope(...)` и при необходимости добавить точную проверку `ST_Within` или `ST_Intersects`.
   </details>

#### Проверь себя: план запроса и оптимизация

7. **Что в плане запроса указывает на использование пространственного индекса?**
   <details><summary>Ответ</summary>
   Строки вида `Index Scan on ... using idx_name` или `Bitmap Index Scan on idx_name` с условием по `geom` (например, `Index Cond: (geom && ...)`). Отсутствие таких шагов при большом объёме данных обычно означает полный перебор (Seq Scan).
   </details>

8. **Почему условие `WHERE ST_Transform(geom, 3857) && ...` может не использовать индекс по `geom`?**
   <details><summary>Ответ</summary>
   Потому что в условии используется не колонка `geom` сама по себе, а выражение (результат функции). Планировщик не может применить индекс по `geom` к такому выражению. Нужно либо хранить геометрию уже в 3857 в отдельной колонке с индексом, либо подставлять в фильтр геометрию без оборачивания колонки в функцию.
   </details>

9. **Чем агрегат `ST_Collect(geom)` отличается от `ST_Union(geom)`?**
   <details><summary>Ответ</summary>
   `ST_Collect` собирает геометрии в одну коллекцию (MULTI* или GEOMETRYCOLLECTION) без слияния границ. `ST_Union` объединяет геометрии в одну (или несколько) слитых областей, устраняя перекрытия. Для «набора объектов» — Collect; для «одной зоны покрытия» — Union.
   </details>

### Запомните

- Пространственные индексы (GiST/SP-GiST) — **обязательны для серьёзных геозадач**: без них всё упирается в полный перебор.
- Для использования индекса важно писать `WHERE`‑условия «по канону» (использовать `&&`, `ST_Intersects` и не оборачивать индексируемую колонку в произвольные функции).
- Часто оптимизация геозапроса — это **разделение на грубый bbox‑фильтр и точную проверку геометрии**.

#### Как читать план запроса с пространственным индексом (по шагам)

Возьми упрощённый план (идея, а не точный вывод PostgreSQL):

```text
Bitmap Heap Scan on shops
  Recheck Cond: (geom && d.geom)
  Filter: ST_Within(geom, d.geom)
  ->  Bitmap Index Scan on idx_shops_geom_gist
        Index Cond: (geom && d.geom)
```

Разберём его, как если бы это был простой русский текст:

- **`Bitmap Index Scan on idx_shops_geom_gist`**  
  PostgreSQL идёт в индекс `idx_shops_geom_gist` и спрашивает:  
  «Какие строки *могут* подойти по условию `geom && d.geom` (пересечение прямоугольников)?»  
  Ответ — это набор «флажков» (битовая карта), где отмечены потенциально подходящие строки.

- **`Bitmap Heap Scan on shops`**  
  Затем PostgreSQL идёт в саму таблицу `shops`, но **не по всем строкам**, а только по тем, что отмечены «флажками» из индекса.  
  На этом шаге он:
  - ещё раз перепроверяет условие `geom && d.geom` (`Recheck Cond`);
  - применяет точный фильтр `ST_Within(geom, d.geom)` (геометрия, а не просто прямоугольник).

Если представить это образно:

1. Индекс — это **черновой список подозреваемых** (по форме прямоугольника).
2. Таблица — это **допрос каждого подозреваемого** с уточняющими вопросами (точная геометрия).

#### Простое правило для написания запросов с индексом

Чтобы не ломать использование пространственного индекса, держи в голове простой чек‑лист:

- **Да:**
  - `WHERE geom && :filter_geom`
  - `WHERE geom && :filter_geom AND ST_Intersects(geom, :filter_geom)`
  - `WHERE geom && ST_Envelope(:filter_geom)`

- **Осторожно или «нет»:**
  - `WHERE ST_Intersects(ST_Transform(geom, ...), :filter_geom)` — колонка внутри трансформации;
  - `WHERE ST_Buffer(geom, 100) && :filter_geom` — индекс уже трудно использовать эффективно;
  - любые сложные выражения, где `geom` спрятан глубоко внутри функций.

Если сомневаешься, включи `EXPLAIN (ANALYZE, BUFFERS)` и **посмотри, есть ли там `Index Scan` / `Bitmap Index Scan` по твоему пространственному индексу**. Нет индекса в плане — значит, нужно переписать запрос или индексацию.

#### Мини‑эксперимент: до и после индекса

Чтобы руками почувствовать, **что именно даёт пространственный индекс**, сделай мини‑практику:

1. **Создай таблицу с большим числом точек.**
   ```sql
   CREATE TABLE big_poi AS
   SELECT
     g AS id,
     ('POI ' || g)::text AS name,
     ST_SetSRID(
       ST_MakePoint(
         30 + random(), -- долготa в каком‑то диапазоне
         60 + random()  -- широта
       ),
       4326
     ) AS geom
   FROM generate_series(1, 200000) AS g;
   ```

2. **Сделай тестовый полигон (условный «район»).**
   ```sql
   SELECT ST_GeomFromText(
     'POLYGON((
        30.1 60.1,
        30.9 60.1,
        30.9 60.9,
        30.1 60.9,
        30.1 60.1
      ))',
     4326
   ) AS region;
   ```

3. **Запусти запрос БЕЗ индекса и измерь время.**
   ```sql
   EXPLAIN ANALYZE
   WITH region AS (
     SELECT ST_GeomFromText(
       'POLYGON((
          30.1 60.1,
          30.9 60.1,
          30.9 60.9,
          30.1 60.9,
          30.1 60.1
        ))',
       4326
     ) AS geom
   )
   SELECT COUNT(*)
   FROM big_poi p, region r
   WHERE ST_Within(p.geom, r.geom);
   ```
   Обрати внимание на:
   - тип скана (`Seq Scan` или что‑то похожее);
   - общее время выполнения.

4. **Создай GiST‑индекс.**
   ```sql
   CREATE INDEX idx_big_poi_geom_gist
   ON big_poi
   USING GIST (geom);
   ```

5. **Перезапусти тот же запрос и сравни план.**
   - Теперь ты почти наверняка увидишь `Bitmap Index Scan` или `Index Scan` по `idx_big_poi_geom_gist`.
   - Время выполнения должно заметно уменьшиться.

6. **Сделай выводы:**
   - что именно изменилось в плане (какой шаг появился/исчез);
   - как поменялось время;
   - сколько строк обрабатывается на каждом шаге.

После такого эксперимента пространственные индексы перестают быть «магией» и становятся **конкретным инструментом, который экономит секунды/минуты выполнения запросов**.

#### Частые вопросы про пространственные индексы

1. **Почему индекс по `geom` не используется, хотя я его создал?**
   - Проверь:
     - не оборачиваешь ли ты `geom` в функции (`ST_Transform(geom, ...)`, `ST_Buffer(geom, ...)`) прямо в `WHERE`;
     - совпадает ли SRID у `geom` и фильтрующей геометрии;
     - не слишком ли мала таблица (на крошечных таблицах планировщик иногда честно выбирает `Seq Scan`).

2. **Нужен ли индекс и на полигонах (`regions`), или достаточно только на точках (`poi`)?**
   - Зависит от запроса, но в реальных системах **часто индексируют обе стороны**:
     - точки — чтобы быстро находить кандидатов;
     - полигоны — если фильтрация идёт ещё и по ним (например, много разных регионов, а не один).

3. **GiST или SP-GiST — что выбрать по умолчанию?**
   - Если нет специфических требований и особого профиля данных, **GiST — безопасный и стандартный выбор** для PostGIS.
   - SP-GiST начинай рассматривать только тогда, когда:
     - у тебя есть чёткий профиль нагрузки;
     - ты уже упёрся в ограничения GiST;
     - ты готов измерять и сравнивать варианты.

### Типичная ошибка / Что будет, если…

- **Ошибка:** создать GiST‑индекс по `geom`, но в фильтрах всегда использовать `ST_Intersects(ST_Transform(geom, ...), ...)`.  
  **Последствия:** индекс почти не используется, запросы медленные, разработчики винят PostGIS.  
  **Лучше:** хранить геометрию в нужной проекции или в отдельной колонке, по которой и строится индекс; не оборачивать индексируемую колонку в тяжёлые функции в `WHERE`.
