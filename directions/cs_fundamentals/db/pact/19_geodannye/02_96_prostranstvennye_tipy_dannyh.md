## 96. Пространственные типы данных

### Цель раздела

Разобраться в **основных типах геометрий** в PostGIS (`POINT`, `LINESTRING`, `POLYGON`, `MULTI*`, `GEOMETRYCOLLECTION`), понять, чем они отличаются, какие задачи решают и какие ограничения есть на «валидность» геометрий.

### В этом разделе главное (три строки)

1. Базовые типы геометрий — это **точка, линия и полигон**; всё остальное — их комбинации (MULTI* и коллекции).
2. Геометрия должна быть **корректной (валидной)**: полигоны замкнуты, кольца не самопересекаются, точка имеет две координаты и т.п.
3. Выбор правильного типа — это **модель предметной области**: адрес чаще — точка, улица — линия, район — полигон, страна — multipolygon.

### Термины

- **`POINT`** — геометрия точки (X, Y), например, долгота/широта.
- **`LINESTRING`** — ломаная линия: последовательность точек, соединённых отрезками.
- **`POLYGON`** — многоугольник (замкнутая область), определённый внешним и, возможно, внутренними кольцами (дырами).
- **`MULTIPOINT`** — коллекция точек как один объект.
- **`MULTILINESTRING`** — коллекция линий.
- **`MULTIPOLYGON`** — коллекция полигонов (например, страна с несколькими островами).
- **`GEOMETRYCOLLECTION`** — коллекция геометрий разных типов.
- **Валидность геометрии** — соответствие набору правил (нет самопересечений, полигон замкнут, «дырки» внутри и т.п.).
- **`ST_IsValid`** — функция проверки валидности геометрии.
- **`ST_MakeLine`** — построение линии (LINESTRING) из массива точек или из набора строк (агрегат с ORDER BY).

### Теория и правила

- **Тип `POINT`.**
  - Представляет одну точку с координатами (X, Y).
  - В WKT: `POINT(37.62 55.75)`.
  - В таблице:
    ```sql
    geom geometry(Point, 4326)
    ```
  - Применение: адреса, маркеры на карте, объекты с чёткой точкой расположения.

- **Тип `LINESTRING`.**
  - Последовательность точек: `LINESTRING(x1 y1, x2 y2, x3 y3, ...)`.
  - Применение: улицы, маршруты, реки, линии связи.
  - В таблице:
    ```sql
    geom geometry(LineString, 4326)
    ```

- **Создание линии: `ST_MakeLine`.**
  - Линию можно собрать **из набора точек** без ручного WKT — удобно, когда точки приходят из подзапроса или массива.
  - Вариант 1 — массив геометрий (точки в нужном порядке):
    ```sql
    SELECT ST_SetSRID(
      ST_MakeLine(ARRAY[
        ST_MakePoint(37.60, 55.75),
        ST_MakePoint(37.61, 55.76),
        ST_MakePoint(37.62, 55.77)
      ]),
      4326
    ) AS route;
    ```
  - Вариант 2 — агрегация точек из таблицы (например, трек по времени):
    ```sql
    SELECT ST_MakeLine(geom ORDER BY recorded_at) AS track
    FROM gps_points
    WHERE device_id = 1 AND recorded_at::date = '2024-01-15';
    ```
  - Важно: точки должны идти в правильном порядке (например, хронологическом для маршрута). Для полигона потом можно замкнуть линию, добавив в конец первую точку, и передать в `ST_MakePolygon`.

- **Тип `POLYGON`.**
  - Замкнутая область, описанная кольцами:
    - внешний контур;
    - опционально один или несколько внутренних контуров (дырок).
  - Пример WKT:
    ```text
    POLYGON((
      0 0,
      10 0,
      10 10,
      0 10,
      0 0
    ))
    ```
  - Применение: границы районов, земельные участки, зоны доставки.

- **Получение координат точки: `ST_X()` и `ST_Y()`.**
  - Если у тебя есть колонка с точкой (`geometry(Point, 4326)` или похожая), ты можешь **вытащить из неё отдельно долготу и широту**:
    ```sql
    SELECT
      ST_X(geom) AS lon,
      ST_Y(geom) AS lat
    FROM demo_geoms
    WHERE name = 'Точка А';
    ```
  - Здесь:
    - `ST_X(geom)` — это «X‑координата» точки (обычно **долгота**);
    - `ST_Y(geom)` — это «Y‑координата» (обычно **широта**).
  - Это полезно, когда:
    - нужно отдать координаты наружу (в API, лог, CSV);
    - нужно проверить, корректно ли хранятся значения (смотрим, что `lat` в диапазоне `[-90, 90]`, `lon` — `[-180, 180]`).

- **MULTI‑типы.**
  - `MULTIPOINT`, `MULTILINESTRING`, `MULTIPOLYGON` хранят несколько однотипных геометрий в одном объекте.
  - Пример:
    ```text
    MULTIPOINT(0 0, 1 1, 2 2)
    ```
  - Применение: страна с островами (`MULTIPOLYGON`), набор точек измерений (`MULTIPOINT`).

- **`GEOMETRYCOLLECTION`.**
  - Содержит геометрии разных типов:
    ```text
    GEOMETRYCOLLECTION(
      POINT(0 0),
      LINESTRING(0 0, 1 1, 2 2)
    )
    ```
  - Применяется редко; чаще удобнее хранить объекты в разных строках/таблицах.

#### Проверь себя: MULTI* и GEOMETRYCOLLECTION

10. **Когда выбрать MULTIPOLYGON, а когда несколько строк с POLYGON в таблице?**
    <details><summary>Ответ</summary>
    MULTIPOLYGON — когда несколько полигонов образуют **один логический объект** (страна с островами, район из несвязных кусков). Несколько строк с POLYGON — когда это независимые объекты (разные районы, разные зоны доставки), у каждого свой id и атрибуты.
    </details>

11. **В чём минус хранения «разнородного» набора (точка + линия + полигон) в одном поле как GEOMETRYCOLLECTION?**
    <details><summary>Ответ</summary>
    Сложнее писать запросы (нужно разбирать коллекцию), сложнее индексировать и проверять типы. Обычно проще хранить разные типы в разных колонках или таблицах и связывать по id.
    </details>

- **Валидность геометрий.**
  - Многие функции PostGIS предполагают, что геометрия **валидна**.
  - Проверка:
    ```sql
    SELECT ST_IsValid(geom) FROM my_table;
    ```
  - Для полигонов:
    - контуры должны быть замкнуты (первая и последняя точка совпадают);
    - внутренние кольца должны находиться внутри внешнего;
    - контуры не должны самопересекаться.

### Пошагово: создаём разные геометрии

1. **Создадим таблицу для демонстрации.**
   ```sql
   CREATE TABLE demo_geoms (
     id   serial PRIMARY KEY,
     name text,
     geom geometry
   );
   ```

2. **Вставим точку (`POINT`).**
   ```sql
   INSERT INTO demo_geoms (name, geom)
   VALUES (
     'Точка А',
     ST_SetSRID(ST_MakePoint(37.62, 55.75), 4326)
   );
   ```

3. **Вставим линию (`LINESTRING`).**
   ```sql
   INSERT INTO demo_geoms (name, geom)
   VALUES (
     'Линия маршрута',
     ST_GeomFromText(
       'LINESTRING(37.60 55.75, 37.61 55.76, 37.62 55.77)',
       4326
     )
   );
   ```

4. **Вставим полигон (`POLYGON`).**
   ```sql
   INSERT INTO demo_geoms (name, geom)
   VALUES (
     'Зона доставки',
     ST_GeomFromText(
       'POLYGON((
          37.60 55.75,
          37.65 55.75,
          37.65 55.78,
          37.60 55.78,
          37.60 55.75
        ))',
       4326
     )
   );
  ```

5. **Альтернатива: построим полигон через `ST_MakePolygon`.**

Иногда удобнее сначала задать **линию‑контур**, а потом уже превратить её в полигон:

  ```sql
  -- внешний контур зоны доставки как линия
  WITH outer_ring AS (
    SELECT ST_MakeLine(ARRAY[
      ST_MakePoint(37.60, 55.75),
      ST_MakePoint(37.65, 55.75),
      ST_MakePoint(37.65, 55.78),
      ST_MakePoint(37.60, 55.78),
      ST_MakePoint(37.60, 55.75) -- замыкаем кольцо
    ]) AS line
  )
  INSERT INTO demo_geoms (name, geom)
  SELECT
    'Зона доставки (makepolygon)',
    ST_SetSRID(ST_MakePolygon(line), 4326)
  FROM outer_ring;
   ```

6. **Посмотрим, что получилось.**
   ```sql
   SELECT id, name, GeometryType(geom) AS type, ST_AsText(geom) AS wkt
   FROM demo_geoms;
   ```

7. **Проверим валидность полигонов.**
   ```sql
   SELECT name, ST_IsValid(geom)
   FROM demo_geoms
   WHERE GeometryType(geom) = 'POLYGON';
   ```

### Простыми словами

Типы геометрий — это просто **формы на карте**:

- точка — один маркер;
- линия — нарисованный маршрут;
- полигон — закрашенная область;
- multi‑варианты — набор таких форм в одном «объекте»;
- коллекция — «мешок» из разных форм.

База строго следит за тем, чтобы эти формы были **осмысленными**: многоугольник замкнут, не самопересекается, дырки внутри и т.д. Это нужно, чтобы операции (попадание внутрь, площади) работали корректно.

### Картинка в голове / Как запомнить

Представь бумажную карту и набор маркеров:

- точка — один наклеенный стикер;
- линия — линия, проведённая маркером по дороге;
- полигон — вырезанный из бумаги кусок района, который ты накладываешь на карту;
- multi‑типы — несколько таких стикеров/линий/полигонов, склеенных в один «набор».

PostGIS просто хранит это в виде чисел, но смысл остаётся тем же.

### Проверь себя

1. **В каких случаях лучше использовать `MULTIPOLYGON`, а не несколько строк с `POLYGON`?**
   <details><summary>Ответ</summary>
   Когда логически это **один объект**, состоящий из нескольких частей (например, страна с островами, район с разорванной территорией).  
   Тогда имеет смысл хранить его как один `MULTIPOLYGON`, а не как несколько несвязанных `POLYGON` в разных строках.
   </details>

2. **Что такое «валидный полигон» в терминах PostGIS?**
   <details><summary>Ответ</summary>
   Это полигон, у которого:
   - контуры замкнуты (первая и последняя точка совпадают);
   - внутренние кольца находятся внутри внешнего;
   - нет самопересечений и странных «петель».  
   Проверяется через `ST_IsValid`.
   </details>

3. **Как узнать тип геометрии в колонке?**
   <details><summary>Ответ</summary>
   Использовать функцию `GeometryType(geom)` или `ST_GeometryType(geom)`.  
   Она вернёт строку вроде `'ST_Point'`, `'ST_LineString'`, `'ST_Polygon'` и т.п.
   </details>

#### Проверь себя: POINT, LINESTRING, ST_MakeLine

4. **В каком порядке передавать координаты в `ST_MakePoint` для широты и долготы (WGS84)?**
   <details><summary>Ответ</summary>
   В PostGIS для точки принят порядок **(X, Y)**, то есть сначала долгота (longitude), потом широта (latitude): `ST_MakePoint(lon, lat)`. Частая ошибка — передать (lat, lon) и получить точку в неверном месте.
   </details>

5. **Чем `ST_MakeLine(ARRAY[...])` отличается от вставки LINESTRING через WKT?**
   <details><summary>Ответ</summary>
   Логически результат тот же — линия из точек. `ST_MakeLine` удобен, когда точки уже есть в виде геометрий (подзапрос, массив), и не хочется вручную собирать строку WKT. Для одной статичной линии WKT может быть проще.
   </details>

6. **Можно ли собрать линию из строк таблицы? Если да, как задать порядок точек?**
   <details><summary>Ответ</summary>
   Да: использовать агрегат `ST_MakeLine(geom ORDER BY выражение)`, например `ORDER BY recorded_at`. Порядок в `ORDER BY` задаёт последовательность точек в линии.
   </details>

#### Проверь себя: POLYGON, ST_X/ST_Y, валидность

7. **Что вернут `ST_X(geom)` и `ST_Y(geom)` для точки в WGS84?**
   <details><summary>Ответ</summary>
   `ST_X` — долгота (X в системе координат), `ST_Y` — широта (Y). Для точки в SRID 4326 это будут градусы (например, 37.62 и 55.75).
   </details>

8. **Почему полигон в WKT должен замыкаться: первая и последняя точка совпадают?**
   <details><summary>Ответ</summary>
   Полигон — замкнутая область; незамкнутый контур не определяет площадь и считается невалидным. PostGIS ожидает замкнутые кольца; иначе возможны ошибки или некорректные результаты при расчёте площади и при проверках (ST_Within и т.д.).
   </details>

9. **Как исправить геометрию, если `ST_IsValid(geom)` вернул false?**
   <details><summary>Ответ</summary>
   Использовать `ST_MakeValid(geom)` — функция пытается построить валидную геометрию (например, устранить самопересечения). Результат стоит проверить визуально или повторно вызвать `ST_IsValid`.
   </details>

### Запомните

- Основные кирпичики геоданных — **точки, линии и полигоны**, все остальные типы — их комбинации.
- Корректность геометрий важна: многие функции предполагают, что геометрия валидна.
- Тип геометрии — это часть модели домена: не ленись выбирать подходящий тип, а не хранить всё «как попало».

#### Ещё один взгляд: как выбрать тип в задачах

Подумай про **конкретные сценарии** и соотнеси их с типами:

- **Адрес или объект (дом, магазин, светофор)**  
  - Обычно это **одна точка** → `POINT`.  
  - Если объект размазанный (большой завод), иногда делают и `POINT` (условный центр), и `POLYGON` (точная граница).

- **Маршрут курьера, река, линия метро**  
  - Это **линия**, возможно с изгибами → `LINESTRING`.  
  - Если линия состоит из нескольких несвязанных кусочков (например, две ветки маршрута) — можно использовать `MULTILINESTRING`.

- **Район города, страна, зона доставки**  
  - Это **область** → `POLYGON`.  
  - Страна с островами или район из нескольких кусков → `MULTIPOLYGON`.

- **Набор отдельных точек как один объект**  
  - Например, все измерения датчика за день как один объект → `MULTIPOINT`.

Если сомневаешься — **начни с самого простого**:

1. Спроси себя: «Это точка, линия или область?»
2. Если ответ «несколько точек/линий/областей, но это всё один логический объект» — добавь приставку `MULTI`.
3. Если всё равно не понимаешь — значит, лучше сначала перерисовать задачу на бумаге и только потом думать о типе.

#### Мини‑упражнение: классифицируем объекты

Возьми лист бумаги и выпиши 10–15 реальных объектов:

- парковка у дома;
- маршрут от дома до работы;
- парк;
- река;
- граница города;
- список точек, где ты вчера был, и т.д.

И для каждого ответь:

1. Это **точка**, **линия** или **область**?
2. Это **один объект** или **логически набор объектов** (тогда возможно `MULTI`)?

Потом попробуй подобрать для каждого тип: `POINT`, `LINESTRING`, `POLYGON`, `MULTIPOINT`, `MULTILINESTRING`, `MULTIPOLYGON`.  
Так ты натренируешься автоматически видеть за словами «тип геометрии» просто форму на карте.

### Типичная ошибка / Что будет, если…

- **Ошибка:** хранить сложные объекты (например, страну с островами) как набор разрозненных полигонов без явной связи.  
  **Последствия:** сложнее агрегировать, считать площади/периметры «целого» объекта, обеспечивать целостность.  
  **Лучше:** использовать `MULTIPOLYGON` или ввести явный идентификатор объекта и хранить части связно.
