## Часть XII. Операционная часть и безопасность

> Цель этой части — **научиться эксплуатировать базу данных в продакшене безопасно и предсказуемо**: ставить мониторинг, проводить миграции без даунтайма, выстраивать аутентификацию и авторизацию, защищаться от SQL‑инъекций и выполнять требования по аудиту и комплаенсу. Всё — **простым языком, но с корректной терминологией и привязкой к PostgreSQL (как к самому типичному примеру)**.

После этой части ты сможешь:

- **понимать, какие метрики БД и ОС ключевые**, как построить дашборды и алерты и чем смотреть «что болит»;
- **планировать и проводить миграции схемы** (в том числе zero‑downtime или с минимальным простоем);
- **грамотно настраивать безопасность**: `pg_hba.conf`, роли, привилегии, RLS, шифрование, работу с секретами;
- **распознавать и предотвращать SQL‑инъекции** с помощью подготовленных выражений и модели прав;
- **строить аудит и маскирование данных** для соответствия требованиям (GDPR, PCI DSS и т.п.) и уменьшения рисков утечек;
- **узнавать операционные антипаттерны** и не повторять их.

Материал опирается на:

- части про SQL и моделирование (запросы, индексы, транзакции);
- часть про масштабирование и отказоустойчивость (репликация, бэкапы);
- базовое понимание Linux/инфраструктуры (метрики CPU/RAM/диска, логи).

### Краткая шпаргалка по части

| Понятие | Суть |
|--------|------|
| **Мониторинг БД** | Метрики запросов (QPS, latency p50/p95/p99, ошибки), соединения, блокировки, репликация, ресурсы ОС; экспортёры + дашборды (Grafana/PMM и т.п.). |
| **pg_stat\_statements** | Расширение PostgreSQL для статистики запросов (частота, суммарное/среднее время, план) — база для оптимизации. |
| **Версионирование схемы (миграции)** | Упорядоченный набор DDL‑шагов (мigrations), фиксирующий историю изменений; применяются автоматически через CI/CD. |
| **Zero‑downtime миграции** | Паттерны изменения схемы без (или почти без) простоя: совместимая схема, backfill, `CREATE INDEX CONCURRENTLY`. |
| **`pg_hba.conf`** | Таблица правил: кто и как может подключаться к PostgreSQL (тип подключения, база, пользователь, адрес, метод аутентификации). |
| **Роли и привилегии** | Роли (пользователи/группы) и права (GRANT/REVOKE) на базы, схемы, таблицы и столбцы; реализуют принцип минимальных привилегий. |
| **RLS (Row Level Security)** | Политики доступа на уровне строк: «какие строки этой таблицы видит/может менять данная роль». |
| **SQL‑инъекция** | Подмена структуры SQL‑запроса через неосторожную конкатенацию строк с пользовательским вводом. |
| **Подготовленные выражения** | Разделяют SQL и данные: запрос компилируется один раз, параметры передаются отдельно → защита от инъекций. |
| **Аудит и маскирование** | Логирование действий (кто, что, когда делал) и скрытие/обезличивание чувствительных данных (в логах, дампах, тестовых базах). |

### Как устроены разделы и оглавление

Во всех разделах этой части используется **единый шаблон**, чтобы тебе было проще ориентироваться и повторять:

| Блок | Для чего он |
|------|-------------|
| **Цель раздела** | Понять, что именно ты освоишь после чтения. |
| **«В этом разделе главное»** (если есть) | Несколько строк — суть раздела до деталей. |
| **Термины (расшифровка)** | Ключевые слова и их определения в одном месте. |
| **Теория и правила** | Как устроено явление, какие правила и trade‑off’ы, простые примеры. |
| **Пошагово** (где уместно) | Пошаговое описание процесса (как применить идею на практике). |
| **Простыми словами / Картинка в голове / Как запомнить** | Объяснение без жаргона, образ для запоминания, краткая формула. |
| **Типичные ошибки / Что будет, если…** (где уместно) | Граничные случаи и типичные промахи. |
| **«Проверь себя»** | **2–3 вопроса** с ответами под спойлером (`<details><summary>Ответ</summary>...</details>`). |
| **Запомните** | Краткий вывод: ключевые факты и формулировки. |

Как пользоваться оглавлением и структурой:

- смотри на таблицу **«Структура материала»** выше — это твой **маршрут по разделам** (мониторинг → миграции → безопасность → аудит и резюме);  
- внутри части двигайся **по разделам в указанном порядке** (`§73` → `§74` → … → `§78`), а внутри раздела — по блокам в том же порядке (цель → главное → термины → теория → пошагово → простыми словами → «Проверь себя» → «Запомните»);  
- когда пройдёшь все разделы, используй блоки **«Проверь себя»** и резюме как **mini‑экзамен** и шпаргалку для повторения.
