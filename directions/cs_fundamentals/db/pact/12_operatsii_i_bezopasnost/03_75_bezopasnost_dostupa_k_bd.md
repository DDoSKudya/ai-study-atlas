## §75. Безопасность доступа к БД

### Цель раздела

Понять, **как устроены аутентификация (`pg_hba.conf`) и авторизация (роли и привилегии) в БД**, зачем нужен принцип минимальных привилегий, как использовать RLS и базовые механизмы шифрования и как правильно хранить секреты (пароли, ключи).

### Главное в трёх строках

1. **Безопасность БД начинается с `pg_hba.conf`: кто может подключаться, откуда и каким методом.**  
2. **Роли и привилегии реализуют принцип минимальных прав: приложению не нужен `SUPERUSER`.**  
3. **RLS, шифрование и аккуратная работа с секретами защищают данные даже при утечке приложения или бэкапа.**

### `pg_hba.conf` в двух словах

Строка в `pg_hba.conf`:

```text
host    mydb    app_user    10.0.0.0/24    scram-sha-256
```

Значит:

- тип — TCP (`host`);
- база — `mydb`;
- пользователь — `app_user`;
- сеть — `10.0.0.0/24`;
- метод — SCRAM‑пароль.

Правила:

- строки проверяются **сверху вниз**, срабатывает первое подходящее;
- на проде не использовать `trust` и слишком широкие маски (`0.0.0.0/0`) без необходимости;
- по возможности включать TLS (`hostssl`, `ssl = on`, `sslmode=verify-full`).

### Роли и привилегии

Создаём:

```sql
CREATE ROLE app_user LOGIN PASSWORD '...';
CREATE ROLE app_readonly;
GRANT app_readonly TO some_analyst;
```

Выдаём права:

```sql
GRANT CONNECT ON DATABASE mydb TO app_user;
GRANT USAGE ON SCHEMA public TO app_user;
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO app_user;
```

Важно:

- приложение **не должно быть суперпользователем**;
- аналитикам обычно нужен только SELECT;
- админам — отдельные учётки, а не «тот же пользователь, что и у приложения».

### RLS (Row Level Security)

Для multi‑tenant‑таблиц:

```sql
ALTER TABLE invoices ENABLE ROW LEVEL SECURITY;

CREATE POLICY tenant_isolation ON invoices
USING (tenant_id = current_setting('app.tenant_id')::uuid);
```

Приложение выставляет `SET app.tenant_id = '...';`, и БД сама отфильтровывает строки по арендаторам.

### Шифрование и секреты

- **At rest**: шифрование диска (LUKS и аналоги), TDE (если есть); защищает при краже/утере носителя.  
- **In transit**: TLS между клиентом и сервером (сертификаты, `sslmode=verify-full`); защищает от перехвата трафика.  
- **Секреты**:
  - не храним пароли в репозитории;
  - используем переменные окружения/секрет‑менеджеры (Vault, AWS Secrets Manager, K8s Secrets и т.п.);
  - ограничиваем доступ к этим секретам по принципу минимальных прав.

### Проверь себя

1. **Какой порядок проверки правил в `pg_hba.conf` и почему это важно?**  
2. **Что значит принцип минимальных привилегий применительно к учётке приложения в БД?**

<details>
<summary>Ответ</summary>

1. Строки проверяются сверху вниз, и срабатывает первое подходящее правило. Если наверху есть слишком широкое правило (например, `host all all 0.0.0.0/0 trust`), оно «перекроет» все нижние, делая их бессмысленными и потенциально открывая БД всем подряд.  
2. Это значит, что учётке приложения выдают только те права, которые нужны для его работы (обычно SELECT/INSERT/UPDATE/DELETE на конкретные таблицы), и ничего лишнего: ни SUPERUSER, ни возможности создавать/удалять базы, ни доступа к чужим схемам.

</details>

### Запомните

- **`pg_hba.conf` и роли — первые линии защиты.**  
- **Даже идеальное приложение небезопасно, если в БД у него «бог‑режим».**

### Ещё глубже: безопасность БД «как офис с охраной»

Продолжим картинку с офисным зданием:

1. **Вход в здание (`pg_hba.conf`)**
   - На входе сидит охранник с журналом.  
   - В журнале есть строки вида:  
     «Пускать сотрудников отдела разработки по пропускам с 9 до 21 через главный вход».  
   - В терминах PostgreSQL это: тип подключения, база, пользователь, сеть, метод аутентификации.
   - Охранник идёт по журналу **сверху вниз** и смотрит, какая строчка подходит под человека у турникета.

   Если первой стоит строка:  
   «Пускать всех всегда без проверки документов»,  
   — журнал дальше можно даже не читать: здание почти не защищено.

2. **Ключи от комнат (роли и GRANT’ы)**
   - Допустим, человек прошёл через турникет (pg_hba.conf сказал «ОК»).  
   - Дальше вопрос: **в какие комнаты он может зайти?**
   - Кто‑то может зайти только в open space (SELECT).  
   - Кто‑то может заходить ещё в переговорки и подсобку (INSERT/UPDATE/DELETE).  
   - У кого‑то есть ключи от серверной (DDL, админские права).

   Выдавая человеку «ключи», ты, по сути, делаешь `GRANT` определённых прав на определённые объекты.

3. **Стеклянные перегородки (RLS)**
   - Представь open space, где сидят сотрудники разных компаний (multi‑tenant).  
   - Вроде как все в одном зале, но **стекло матовое**, и каждая компания видит только свои столы.  
   - Это RLS: одна таблица, но каждый видит только «свои строки» по `tenant_id`.

4. **Сейфы и шифрование**
   - В офисе есть сейфы, где лежат самые важные документы (номера карт, пароли).  
   - Даже если кто‑то ночуёт в офисе, без ключа от сейфа он не увидит содержимое.  
   - Это шифрование at rest.  
   - А сигнализация и зашифрованный канал до головного офиса — это шифрование in transit.

5. **Где лежат пароли (секреты)**
   - Плохой вариант: пароли записаны на стикерах на мониторе → аналог паролей в репозитории.  
   - Нормальный вариант: пароли лежат в сейфе, к которому доступ есть только у админов и по журналу.  
   - В IT это секрет‑менеджеры и переменные окружения.

Если будешь думать о БД как об офисе:

- `pg_hba.conf` = правила охраны на входе;  
- роли/GRANT’ы = набор ключей и прав доступа к комнатам;  
- RLS = стеклянные перегородки, скрывающие чужие столы;  
- шифрование и секреты = сейфы и пароли не на стикерах.

Так проще принять как норму:

- у приложения отдельный «рабочий пропуск» с ограниченными правами;  
- у админов — свои пропуска и отдельные логины;  
- никто не ходит по офису с «универсальным отмычком» (SUPERUSER для всех).

### Аутентификация подробнее: SCRAM, сертификаты, LDAP, Kerberos

1. **SCRAM‑SHA‑256 (современная парольная аутентификация)**  
   - Пароль никогда не хранится в БД «как есть» — хранится **хеш с солью** (salted hash);  
   - SCRAM‑SHA‑256 устойчив к ряду атак, лучше старого `md5`;  
   - в `pg_hba.conf` это `scram-sha-256`.

   Практически:
   - в `postgresql.conf` включаешь `password_encryption = 'scram-sha-256'`;  
   - создаёшь пользователей через `CREATE ROLE ... PASSWORD '...'`;  
   - в `pg_hba.conf` вместо `md5` используешь `scram-sha-256`.

2. **Сертификаты (clientcert)**

   Идея:
   - у сервера есть свой сертификат и закрытый ключ (TLS);  
   - у клиента тоже есть **клиентский сертификат**, подписанный доверенным CA;  
   - PostgreSQL может требовать, чтобы клиент предъявлял сертификат, и проверять его.

   Пример правила:

   ```text
   hostssl  mydb  app_user  10.0.0.0/24  cert clientcert=verify-full
   ```

   Это полезно для:
   - сервис‑to‑сервис взаимодействия в доверенной сети;  
   - когда не хочется хранить простые пароли, а использовать инфраструктуру сертификатов.

3. **LDAP**

   - PostgreSQL делегирует проверку паролей LDAP‑серверу (например, Active Directory);  
   - `pg_hba.conf` может содержать строку с методом `ldap`, плюс параметры LDAP (в `postgresql.conf`).

   Сценарий:
   - пользователи уже есть в корпоративном каталоге;  
   - БД не хранит свои пароли, а «спрашивает» LDAP: «этот логин/пароль корректен?»;  
   - права при этом всё равно настраиваются через роли/GRANT’ы в самой БД.

4. **Kerberos / GSSAPI**

   - Используется в больших доменах (Windows AD, Kerberos‑инфраструктура);  
   - пользователь проходит аутентификацию один раз (SSO), получает Kerberos‑тикет;  
   - PostgreSQL доверяет этому тикету и не просит пароль повторно.

   Это даёт:
   - бесшовный логин для пользователей домена;  
   - централизованное управление учётками.

Вывод:

- в простых проектах чаще всего достаточно SCRAM‑паролей + TLS;  
- в корпоративных/энтерпрайз‑сценариях добавляют сертификаты, LDAP или Kerberos, чтобы вписаться в общую инфраструктуру.

### RLS: USING и WITH CHECK, принудительная RLS

RLS‑политика в PostgreSQL может управлять:

- **какие строки можно читать** (`USING`);  
- **какие строки можно вставлять/обновлять** (`WITH CHECK`).

Пример более аккуратной политики:

```sql
ALTER TABLE invoices ENABLE ROW LEVEL SECURITY;

CREATE POLICY tenant_isolation_read ON invoices
  FOR SELECT
  USING (tenant_id = current_setting('app.tenant_id')::uuid);

CREATE POLICY tenant_isolation_write ON invoices
  FOR INSERT, UPDATE
  WITH CHECK (tenant_id = current_setting('app.tenant_id')::uuid);
```

Здесь:

- `USING` говорит: «при SELECT показывай только строки с `tenant_id` из сессионной переменной»;  
- `WITH CHECK` говорит: «при INSERT/UPDATE разрешай только такие изменения, чтобы `tenant_id` совпадал с тем же значением».

**Принудительная RLS (force RLS)**:

```sql
ALTER TABLE invoices FORCE ROW LEVEL SECURITY;
```

Это заставляет применять RLS‑политики даже к ролям с широкими правами (кроме суперпользователя).  
Полезно, чтобы кто‑то с «почти админскими» правами **случайно не обошёл** RLS.

### SECURITY DEFINER и подводные камни

Функция с `SECURITY DEFINER` выполняется с правами владельца:

```sql
CREATE FUNCTION do_sensitive_thing(...)
RETURNS void
LANGUAGE plpgsql
SECURITY DEFINER
AS $$
BEGIN
  -- некий код
END;
$$;
```

Идея:

- обычной роли нельзя, например, писать в таблицу `payments`;  
- но она может вызвать функцию, которая делает нужные операции под правами владельца (обычно более привилегированного пользователя).

Основные опасности:

1. **search_path**
   - если в функции не зафиксировать `search_path`, злоумышленник может создать таблицу/функцию с тем же именем в своей схеме;  
   - тогда SECURITY DEFINER‑функция может **обратиться не к тем объектам**, к которым ты думал.

   Правильнее:

   ```sql
   ALTER FUNCTION do_sensitive_thing(...) SET search_path = pg_catalog, public;
   ```

   или жёстко использовать **полные имена** (`schema.table`).

2. **Динамический SQL на основе пользовательского ввода**

   - Внутри SECURITY DEFINER‑функции писать что‑то вида:

   ```plpgsql
   EXECUTE 'DELETE FROM ' || user_input || ' WHERE ...';
   ```

   — почти всегда плохая идея:  
   - это даёт возможность через SQL‑инъекцию внутри функции сделать **любой SQL** с правами владельца функции.

Рекомендации:

- использовать SECURITY DEFINER **только там, где без него никак**;  
- сильно ограничивать, что именно функция делает и какие параметры принимает;  
- явно фиксировать `search_path`;  
- избегать динамического SQL с пользовательским вводом.
