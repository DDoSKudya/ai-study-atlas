## Часть XLI. Расширенные темы Oracle и SQL Server

> Цель этой части — **собрать “в одном месте” ключевые продвинутые особенности Oracle и Microsoft SQL Server**, которые чаще всего всплывают в реальных проектах:  
> кластеры **Oracle RAC**, хранилище **ASM**, язык **PL/SQL**, материализованные представления, партиционирование и индексы в Oracle;  
> **Always On Availability Groups**, **Columnstore индексы**, **In‑Memory OLTP**, специфику **T‑SQL**, партиционирование и индексы в SQL Server;  
> а также **сравнение основных СУБД**, **миграцию между ними** и **managed‑версии** (AWS RDS, Azure SQL и др.).

---

### После этой части ты сможешь

- **объяснять архитектуру Oracle RAC** простыми словами и понимать, чем она отличается от привычной репликации Postgres/MySQL;
- **понимать роль Oracle ASM** и отличать его от “просто файловой системы/RAID”;
- **читать и писать базовый PL/SQL‑код**, понимать, где он уместен, а где лучше бизнес‑логику держать в приложении;
- **использовать материализованные представления Oracle и SQL Server** для ускорения тяжёлых запросов и понимать их ограничения;
- **спроектировать партиционированную таблицу в Oracle и SQL Server**, различать локальные и глобальные индексы, понимать влияние на планы запросов;
- **ориентироваться в ключевых возможностях SQL Server**: Always On Availability Groups, Columnstore индексы, In‑Memory OLTP, особенности T‑SQL;
- **сравнивать PostgreSQL, MySQL, Oracle и SQL Server** с точки зрения лицензий, экосистемы и типичных сценариев использования;
- **набросать план миграции между СУБД** (например, Oracle → PostgreSQL или SQL Server → PostgreSQL/MySQL), учитывая типы данных, диалекты SQL и тестирование;
- **понимать специфику managed‑версий** Oracle/SQL Server (AWS RDS, Azure SQL и др.): какие плюсы даёт облако и какие ограничения остаются у тебя.

### Краткая шпаргалка (термины по части)

| Термин                                   | Что это простыми словами                                                                                                                                                               |
| ---------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Oracle RAC (Real Application Clusters)** | Кластер Oracle, где **несколько инстансов** одной БД работают с **общим хранилищем данных**, видя одни и те же файлы. Высокая доступность и масштабирование с жёсткой синхронизацией. |
| **ASM (Automatic Storage Management)**   | Подсистема Oracle, которая **сама управляет дисками** (stripe, mirror, балансировка) и отдаёт БД “табличные пространства” без привычной файловой системы.                            |
| **PL/SQL**                              | Процедурное расширение SQL в Oracle: **процедуры, функции, пакеты**, управляющие конструкции, курсоры.                                                                               |
| **Материализованное представление**     | Представление, чьи данные **физически хранятся** и периодически обновляются. Быстрые чтения, но есть стоимость обновления и консистентности.                                        |
| **Partitioned table (Oracle/SQL Server)** | Таблица, разбитая на **партиции** по ключу (диапазон, список, хеш и др.). Ускоряет запросы по диапазонам, упрощает архивирование и обслуживание.                                    |
| **Глобальный / локальный индекс**       | В Oracle: **глобальный индекс** строится на всю таблицу; **локальный** — отдельно на каждую партицию. Влияет на операции `DROP/EXCHANGE PARTITION` и планы запросов.               |
| **Always On Availability Groups**       | Механизм высокой доступности в SQL Server: **несколько реплик БД**, автоматический/ручной failover, чтение с вторичных.                                                             |
| **Columnstore index (SQL Server)**      | **Колоночный индекс** для аналитических запросов: данные хранятся по столбцам, сильно сжимаются и быстро сканируются.                                                               |
| **In‑Memory OLTP (Hekaton)**            | Функция SQL Server: **таблицы полностью в памяти** с специальным форматом и оптимизацией под очень высокую нагрузку.                                                                |
| **T‑SQL**                               | Диалект SQL Server: SQL + процедурные расширения (циклы, условия), специфичные типы и функции.                                                                                       |
| **Licensing (Oracle/SQL Server)**       | Лицензионные модели (per‑core, per‑socket, CAL и др.), сильно влияющие на стоимость владения.                                                                                        |
| **Managed Oracle / SQL Server**         | Облачные сервисы (AWS RDS, Azure SQL и др.), которые **управляют инстансами за тебя**: бэкапы, патчи, масштабирование, но с ограничениями.                                          |

#### Проверь себя (по шпаргалке)

1. В чём принципиальное отличие Oracle RAC от “обычной” репликации PostgreSQL/MySQL?

<details><summary>Ответ</summary>
В Oracle RAC **несколько инстансов** работают с **общим хранилищем данных**, то есть видят одни и те же файлы и согласуют доступ к ним через кластерный уровень (GCS/GES). В PostgreSQL/MySQL традиционная репликация — это **копии данных на разных узлах**, каждый со своим хранилищем; изменения разносятся через лог (WAL/binlog), а не через общий диск.
</details>

2. Зачем использовать локальные индексы на партиционированных таблицах в Oracle?

<details><summary>Ответ</summary>
Локальные индексы “привязаны” к партициям: при операциях с партициями (`DROP/EXCHANGE/TRUNCATE PARTITION`) не нужно перестраивать **весь** индекс, достаточно затронуть только соответствующую партицию. Это упрощает обслуживание и ускоряет bulk‑операции на уровне партиций.
</details>

### Маршрут изучения (простыми словами)

1. Начни с **основ Oracle‑мира**: что такое **RAC**, зачем нужен **ASM**, чем PL/SQL отличается от обычного SQL и как работают материализованные представления (**§242**).
2. Затем углубись в **партиционирование и индексы Oracle**: типы партиционирования, глобальные и локальные индексы, как это отражается на планах и обслуживании (**§243**).
3. Переключись на **SQL Server**: разберись, как устроены **Always On**, Columnstore индексы, In‑Memory OLTP и чем T‑SQL отличается от “обычного” SQL (**§244**).
4. После этого посмотри на **партиционирование и индексы в SQL Server**, роль статистики и query hints: как это всё влияет на оптимизацию и когда вообще стоит трогать hints (**§245**).
5. Когда картинка по Oracle и SQL Server более‑менее сложилась, **сравни их с PostgreSQL и MySQL**: лицензии, экосистема, сценарии использования (**§246**).
6. Затем разберись с **миграциями между СУБД**: различия в типах данных, диалектах SQL, инструментах и тестировании миграций (**§247**).
7. Заверши обзором **managed‑версий Oracle и SQL Server** (AWS RDS, Azure SQL и др.): плюсы, минусы, ограничения и типичные сценарии (**§248**).

#### Проверь себя (по маршруту)

1. Почему важно сначала разобрать “нативные” механизмы Oracle/SQL Server (RAC, партиционирование, Columnstore), а уже потом переходить к сравнению и миграциям?

<details><summary>Ответ</summary>
Потому что без понимания **внутренних сильных и слабых сторон** конкретной СУБД сравнение и миграции становятся поверхностными: легко принять неверное решение (“перейдём на PostgreSQL, там то же самое”) и недооценить, какие фичи ты теряешь или должен воспроизвести другими средствами. Сначала — “как это устроено внутри продукта”, потом — “как это соотносится с другими”.
</details>

2. Почему тема managed‑версий вынесена в конец, а не в начало части?

<details><summary>Ответ</summary>
Потому что managed‑сервисы в облаках **скрывают детали реализации**, но не отменяют их. Чтобы правильно использовать RDS/Azure SQL и понимать их ограничения, нужно сначала разобраться, **как работают сами Oracle/SQL Server**: партиционирование, индексы, HA‑механизмы и т.д.
</details>

#### Проверь себя (по структуре)

1. Почему темы сравнения и миграций вынесены в отдельную группу, а не “растворены” внутри разделов про Oracle и SQL Server?

<details><summary>Ответ</summary>
Потому что сравнение и миграции — это **отдельный уровень мышления**: не “как работает конкретная фича”, а “как все эти фичи складываются в картину для выбора архитектуры и переезда”. Если смешать их с техническими деталями, легко потерять целостное видение и фокус на принятии решений.
</details>

2. К какой группе ты бы отнёс вопрос: “Мы сидим на Oracle RAC, стоит ли переезжать на PostgreSQL + Patroni/pgBouncer?” и почему?

<details><summary>Ответ</summary>
К группе **“Сравнение и выбор СУБД”** и частично к **“Миграции и managed‑версии”**. Вопрос не только про технологию, но и про архитектуру, стоимость владения, экосистему, навыки команды — всё это обсуждается именно там.
</details>

### Как устроены разделы и как пользоваться оглавлением

Разделы (§242–§248) построены по единому шаблону (там, где это уместно):

- **Цель раздела** — чего ты конкретно научишься после чтения этого раздела.
- **В этом разделе главное** — несколько ключевых идей, которые надо удержать.
- **Термины** — мини‑словарик по разделу.
- **Теория и правила** — как это устроено внутри СУБД и какие есть варианты.
- **Пошагово** — практический алгоритм: как включить, настроить, использовать.
- **Простыми словами** — объяснение без перегруза жаргоном.
- **Картинка в голове** — образ или аналогия, помогающая запомнить.
- **Как запомнить** — короткие формулы или сравнения.
- **Примеры** — SQL, конфиги и реальные сценарии.
- **Практика / реальные сценарии** — как это выглядит “в бою” и какие компромиссы.
- **Типичные ошибки** — частые грабли и как их обойти.
- **Что будет, если…** — граничные случаи и последствия.
- **Проверь себя** — 2–3 вопроса с ответами под `<details>`.
- **Запомните** — краткая выжимка ключевых тезисов.

Оглавление сгруппировано по этапам: сначала **Oracle**, потом **SQL Server**, затем **сравнение и миграции**.  
Если ты:

- работаешь в компании с сильной Oracle‑историей — начни с §242–§243 и используй §246–§248 как “мостик” к миру PostgreSQL/MySQL/облаков;
- живёшь в мире Microsoft — углубись в §244–§245 и затем смотри на §246–§248;
- выбираешь СУБД для нового проекта — сначала почитай §246–§248, а потом возвращайся к нужным техническим разделам по мере необходимости.
