## 244. SQL Server: Always On, Columnstore, In‑Memory OLTP и T‑SQL

### Цель раздела

Понять, какие **ключевые продвинутые возможности** даёт Microsoft SQL Server:  
механизм высокой доступности **Always On Availability Groups**, **колоночные Columnstore индексы**, **In‑Memory OLTP (Hekaton)** и особенности **T‑SQL** как диалекта и процедурного языка.

### В этом разделе главное

1. **Always On Availability Groups** — основной современный механизм HA/DR в SQL Server, основанный на репликации transaction log.
2. **Columnstore индексы** превращают SQL Server в очень серьёзный инструмент для аналитических нагрузок (OLAP).
3. **In‑Memory OLTP** даёт возможность строить крайне быстрые OLTP‑сценарии, но с серьёзными ограничениями.
4. **T‑SQL** — полноценный процедурный язык, но его, как и PL/SQL, важно использовать с архитектурным пониманием.

### Термины

- **Availability Group (AG)** — группа баз, реплицируемая между несколькими инстансами SQL Server.
- **Primary / Secondary replica** — основной и вторичные узлы в AG.
- **Synchronous / Asynchronous commit** — режимы репликации transaction log в AG.
- **Readable secondary** — вторичная реплика, с которой можно выполнять только чтение.
- **Clustered Columnstore Index** — основной колоночный индекс, хранящий строки в столбцовом формате.
- **Nonclustered Columnstore Index** — колоночный индекс поверх строки‑ориентированной таблицы.
- **Memory‑optimized table** — таблица, полностью хранящаяся в памяти (In‑Memory OLTP).

### Теория и правила

Перед тем как идти в детали по каждому механизму, полезно **сложить картинку целиком**, но очень простыми словами:

1. Нам нужна **высокая доступность** (если один сервер с базой падает — должен быть запасной). Это решает Always On.
2. Нам нужна **быстрая аналитика** по большим таблицам. Это решает Columnstore.
3. Нам нужна **очень быстрая запись/чтение** для узкого круга критичных таблиц. Это решает In‑Memory OLTP.
4. Нам нужен **язык**, на котором можно описать процедуры и миграции, не выходя из БД. Это T‑SQL.

Дальше мы просто идём по этим пунктам и смотрим, **как именно SQL Server это реализует** и в чём подводные камни.

#### 1) Always On Availability Groups

Идея:

- мы выбираем **набор баз**, которые хотим реплицировать между серверами;
- для них создаётся **Availability Group**;
- изменения transaction log с primary реплицируются на secondary по сети.

Режимы:

- **Synchronous commit**:
  - запись считается закоммиченной **после подтверждения** от вторичной реплики;
  - даёт сильно более высокую гарантию (данные уже на двух узлах);
  - увеличивает latency записи.
- **Asynchronous commit**:
  - primary не ждёт подтверждения secondary;
  - меньше задержки;
  - возможна потеря части данных при падении primary (RPO > 0).

Возможности:

- автоматический/ручной **failover**;
- чтение с `readable secondary` (разгрузка primary по чтению);
- интеграция с Windows/WSFC или Linux‑кластером.

#### 2) Columnstore индексы

Columnstore:

- хранят данные **по столбцам**, а не по строкам:
  - сильно лучше сжатие;
  - сильно эффективнее сканы по нескольким агрегируемым колонкам.
- идеально подходят для:
  - отчётов;
  - дэшбордов;
  - аналитических запросов по большим таблицам (факты).

Варианты:

- **Clustered Columnstore Index**:
  - основное хранение таблицы — в колоночном формате;
  - хорош для “чистой” аналитики.
- **Nonclustered Columnstore Index**:
  - колоночный индекс поверх строковой таблицы;
  - компромисс для смешанных нагрузок.

Компромисс:

- запись и обновление дороже (внутренняя буферизация, deltastore);
- для чистого OLTP Columnstore не всегда уместен;
- но для аналитики даёт **огромный прирост**.

#### 3) In‑Memory OLTP (Hekaton)

Идея:

- таблицы (и часть индексов) живут **полностью в памяти**;
- используются специальные lock‑free структуры;
- транзакции компилируются в нативный код.

Плюсы:

- огромный throughput при правильной схеме;
- снижение contention (меньше блокировок).

Минусы и ограничения:

- ограничения на типы данных, операции, триггеры и т.д.;
- сложность отладки;
- риски при падении памяти (но есть логирование/чекпоинты).

#### 4) T‑SQL как процедурный язык

T‑SQL:

- расширяет SQL конструкциями:
  - `DECLARE`/`SET` переменных;
  - `IF`/`WHILE`/`BEGIN...END`;
  - курсоры;
  - процедуры/функции/триггеры.
- даёт возможность:
  - писать бизнес‑логику внутри БД;
  - делать сложные миграции и процедуры.

Те же компромиссы, что у PL/SQL:

- удобно для “рядом с данными” задач;
- опасно для больших монолитов бизнес‑логики.

### Пошагово: когда что использовать

1. **Always On AG**:
   - нужна высокая доступность/DR;
   - есть несколько инстансов SQL Server;
   - требуются readable secondary для аналитики/отчётов.
2. **Columnstore**:
   - есть большие фактовые таблицы;
   - большинство запросов — агрегации/аналитика, а не point lookups;
   - допустимо усложнение записи ради ускорения чтения.
3. **In‑Memory OLTP**:
   - экстремальные OLTP‑нагрузки (финансовые транзакции, телеком и т.п.);
   - чётко ограниченный набор таблиц и сценариев;
   - готовность инвестировать в сложную конфигурацию.
4. **T‑SQL**:
   - миграции схемы/данных;
   - обвязка для ETL внутри БД;
   - точечные процедуры для тяжёлых операций.

### Простыми словами

- **Always On** — это как система **горячих и тёплых копий**: есть главный сервер и пару подстраховочных, которые почти синхронно получают изменения.
- **Columnstore** — это как хранить данные **по столбцам в столбчатых полках**, а не “пакетами строк”: легко достать все значения одного столбца и сильно сжать их.
- **In‑Memory OLTP** — это как держать **всю таблицу в кэше** и писать в супероптимизированные структуры, но с массой правил и ограничений.

### Картинка в голове

Нарисуй:

- основной сервер SQL Server (primary);
- несколько secondary, к которым “течёт” поток log records (AG);
- рядом — большая колонка‑ориентированная таблица (Columnstore) для аналитики;
- ещё рядом — маленький, но сверхбыстрый in‑memory сегмент.

### Как запомнить

> **Always On** = высокодоступная репликация transaction log  
> **Columnstore** = OLAP‑движок внутри SQL Server  
> **In‑Memory OLTP** = сверхбыстрый, но ограниченный “режим турбо” для отдельных таблиц

### Примеры

#### Простая процедура на T‑SQL

```sql
CREATE PROCEDURE ApplyDiscounts
AS
BEGIN
    SET NOCOUNT ON;

    UPDATE o
       SET Discount = CASE
                          WHEN c.TotalAmount > 10000 THEN 0.10
                          WHEN c.TotalAmount > 5000  THEN 0.05
                          ELSE 0
                      END
      FROM Orders o
      JOIN CustomerStats c ON c.CustomerId = o.CustomerId
     WHERE o.Status = 'New';
END;
GO
```

#### Создание clustered Columnstore индекса

```sql
CREATE CLUSTERED COLUMNSTORE INDEX CCI_Sales
    ON dbo.SalesFact;
```

### Практика / реальные сценарии

- **Корпоративный DWH на SQL Server**:
  - фактовые таблицы с десятками/сотнями миллионов строк;
  - Columnstore индексы дают 5–20× ускорение отчётов;
  - AG используется для высокой доступности и разгрузки чтения.
- **Высоконагруженный биллинг**:
  - небольшое число горячих таблиц в In‑Memory OLTP;
  - фоновые процессы периодически “сливают” данные в обычные таблицы.

### Типичные ошибки

- Включать Columnstore “везде”, включая чисто OLTP‑таблицы с частыми точечными обновлениями.
- Рассматривать Always On как “волшебный” способ масштабировать запись (это в первую очередь **HA/DR**, а не шардинг).
- Переносить всю бизнес‑логику в T‑SQL, игнорируя практики модульного приложения и тестирования.

### Что будет, если…

- …использовать synchronous commit для AG между дата‑центрами с большой задержкой?
  - Запись начнёт “затыкаться” на сетевой латентности, RPS резко упадёт, вы начнёте видеть таймауты и блокировки.
- …создать Columnstore индекс на таблицу, где большинство запросов — точечные `SELECT ... WHERE Id = ...`?
  - Планы могут стать хуже: движок будет вынужден “доставать” отдельные строки из колоночного формата, теряя выигрыш от сканов и сжатия.

### Проверь себя

1. Для какого типа нагрузки Columnstore индекс даёт максимальный выигрыш?

<details><summary>Ответ</summary>
Для **аналитических запросов**, которые читают большие объёмы данных и агрегируют их по нескольким колонкам: отчёты, витрины, BI‑дашборды, когда одной операцией сканируется значительная часть таблицы.
</details>

2. Почему Always On AG не решает задачу шардинга?

<details><summary>Ответ</summary>
Потому что AG реплицирует **одни и те же базы** между несколькими инстансами — все реплики содержат одинаковые данные и предназначены для HA/DR и чтения. Шардинг — это горизонтальное разделение данных между кластерами по ключам (user_id, регион и т.п.), чего AG “из коробки” не делает.
</details>

3. В каких случаях In‑Memory OLTP чаще всего оказывается избыточным или неоправданно сложным решением?

<details><summary>Ответ</summary>
Когда нагрузка не требует экстремальных показателей по задержке и throughput, а узким местом являются, например, плохие индексы, неоптимальные запросы или медленная внешняя система. В таких случаях гораздо проще и дешевле сначала оптимизировать схемы и запросы, добавить обычное кэширование и масштабирование, чем внедрять In‑Memory OLTP с его ограничениями по типам, схеме и сложностью эксплуатации.
</details>

### Запомните

- SQL Server — это не только “OLTP‑база под .NET‑приложения”, но и серьёзный **аналитический и HA‑инструмент** (Columnstore, AG, In‑Memory).
- Как и в случае Oracle, эти фичи нужно использовать **осмысленно**, а не как галочки “чтобы было”.
