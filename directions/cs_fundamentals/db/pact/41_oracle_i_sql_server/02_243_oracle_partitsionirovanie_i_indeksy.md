## 243. Oracle: партиционирование и индексы

### Цель раздела

Разобраться, как в Oracle устроены **партиционированные таблицы и индексы**, какие есть типы партиционирования (range, list, hash, composite, interval, reference), чем отличаются **локальные и глобальные индексы**, и как всё это влияет на производительность и сопровождение.

### В этом разделе главное

1. Oracle поддерживает **множество типов партиционирования**, позволяя очень гибко разбивать большие таблицы.
2. Индексы могут быть **локальными** (по партиции) и **глобальными** (по всей таблице); это сильно влияет на операции обслуживания.
3. **Bitmap‑индексы** и **function‑based** индексы — мощные инструменты, особенно для аналитики и сложных условий.
4. Грамотное партиционирование + индексация делает огромные таблицы **управляемыми**: проще архивировать, обслуживать и ускорять запросы.

### Термины

- **Range partitioning** — разбиение по диапазонам (например, по дате).
- **List partitioning** — разбиение по спискам значений (страна, регион и т.п.).
- **Hash partitioning** — разбиение по хешу ключа (равномерное распределение нагрузки).
- **Composite partitioning** — сочетание схем (например, range‑hash).
- **Interval partitioning** — автоматическое создание новых партиций по мере роста диапазона (например, новые месяцы).
- **Reference partitioning** — партиционирование дочерней таблицы по партициям родительской (по FK).
- **Local index** — индекс на уровне партиции (каждая партиция имеет свой фрагмент индекса).
- **Global index** — единый индекс на всю таблицу, не совпадающий с границами партиций.
- **Bitmap index** — индекс, где значения представлены битовыми картами; эффективен для колонок с малым числом разных значений.
- **Function‑based index** — индекс не на колонку напрямую, а на результат выражения/функции (`UPPER(col)`, `TRUNC(date)` и т.п.).

### Теория и правила

Перед тем как разбирать конкретные типы партиционирования, зафиксируем **очень простую интуицию**, чтобы не потеряться в названиях:

- у нас есть **большая таблица**, которая “не влезает в голову” как единое целое — с ней сложно и работать, и обслуживать её;
- мы хотим **разрезать** эту таблицу на “куски” так, чтобы:
  - запросам было легче найти нужные строки;
  - операции по удалению/архивированию работали по кускам, а не по строкам;
  - при этом для приложений таблица выглядела **как одна**.

Все типы партиционирования ниже — это разные ответы на вопрос:  
**“По какому принципу мы режем таблицу на куски, чтобы было удобно жить?”**

#### 1) Зачем нужно партиционирование в Oracle

Цели:

- разделить огромную таблицу (миллионы/миллиарды строк) на **управляемые куски**;
- ускорить **диапазонные запросы** (partition pruning);
- упростить **архивирование, удаление и загрузку данных**:
  - `EXCHANGE PARTITION` для загрузки данных “в сторону”;
  - `DROP/TRUNCATE PARTITION` вместо массовых `DELETE`.

#### 2) Типы партиционирования

- **Range**:
  - классика: по дате (`order_date`), по числовому диапазону;
  - удобно для “скользящих окон”, архивирования старых периодов.
- **List**:
  - по списку значений (`country IN ('US', 'CA')` → одна партиция и т.п.);
  - полезно для разбиения по регионам/бизнес‑линиям.
- **Hash**:
  - равномерное распределение по хешу;
  - хорошо, когда нет естественных диапазонов, но нужно распределить нагрузку.
- **Composite** (например, range‑hash):
  - сначала range по дате, внутри — hash по customer_id;
  - сочетает удобство диапазонов и равномерное распределение.
- **Interval**:
  - автоматическое создание новых партиций по мере появления значений за новыми границами (например, новый месяц);
  - снижает операционные накладные расходы.
- **Reference**:
  - дочерняя таблица партиционируется **синхронно с родительской**;
  - удобно для фактов/деталей (факт + детали заказов).

#### 3) Локальные и глобальные индексы

- **Локальный индекс**:
  - каждая партиция таблицы имеет “свой кусок” индекса;
  - операции с партициями **не требуют перестройки всего индекса**:
    - `DROP PARTITION` → удаляется только фрагмент индекса;
    - `EXCHANGE PARTITION` → обновляется только соответствующий фрагмент.
- **Глобальный индекс**:
  - индексное дерево строится на всю таблицу;
  - может не совпадать с границами партиций (range по другим полям, комбинированные ключи);
  - операции с партициями могут:
    - сделать индекс “UNUSABLE”;
    - потребовать его перестройки.

Компромисс:

- локальные индексы → проще сопровождение, немного сложнее планы в некоторых случаях;
- глобальные индексы → гибче для запросов, но дороже в обслуживании данных по партициям.

#### 4) Bitmap и function‑based индексы

- **Bitmap‑индексы**:
  - эффективны для колонок с **низкой кардинальностью** (пол, статус, регион и т.п.);
  - отлично работают в аналитических запросах с множеством фильтров;
  - не подходят для интенсивно изменяемых OLTP‑таблиц (дорогие операции обновления).
- **Function‑based индексы**:
  - храним в индексе, например, `UPPER(email)` или `TRUNC(order_date)`;
  - оптимизатор может использовать индекс, даже если в запросе есть функция;
  - важно, чтобы функция была **детерминированной** и правильно зарегистрированной.

### Пошагово: проектируем партиционированную таблицу в Oracle

1. Определи **основной сценарий запросов**:
   - по дате (`WHERE order_date BETWEEN ...`);
   - по региону (`WHERE country = ...`);
   - по обоим.
2. Выбери тип партиционирования:
   - дата → range/interval;
   - регион → list;
   - нет явного ключа → hash или composite.
3. Реши, нужны ли **локальные или глобальные индексы**:
   - если планируется активная работа с партициями (архивирование, bulk‑загрузки) → локальные индексы по ключам запросов;
   - если важны сложные запросы с другими ключами → один или несколько глобальных индексов.
4. Подумай о bitmap/function‑based индексах для аналитики:
   - статус, категория, флаги → bitmap;
   - функции в фильтрах → function‑based.
5. Задокументируй:
   - схему партиционирования;
   - правила обслуживания партиций;
   - ограничения на типы запросов (например, избегать функций на колонках в `WHERE` без соответствующих function‑based индексов).

### Простыми словами

- Партиционирование — это как **разбить огромный архив документов по коробкам** (месяц, регион и т.п.), чтобы:
  - не перебирать все бумаги при поиске;
  - выбрасывать/архивировать целые коробки, а не каждый лист.
- Локальные индексы — это **карточки‑указатели внутри каждой коробки**.
- Глобальные индексы — это **одна большая картотека на весь архив**.

### Картинка в голове

Нарисуй:

- длинную ленту таблицы, нарезанную на блоки (партиции) по дате;
- внизу — локальные индексы, каждый под своим блоком;
- сверху — глобальный индекс, который “смотрит” на всю ленту.

### Как запомнить

> **Партиция = коробка с бумагами**  
> **Локальный индекс = указатель внутри коробки**  
> **Глобальный индекс = единая картотека на весь склад**

### Примеры

#### Range‑partitioned таблица по дате

```sql
CREATE TABLE orders
(
  order_id    NUMBER,
  customer_id NUMBER,
  order_date  DATE,
  status      VARCHAR2(20),
  amount      NUMBER
)
PARTITION BY RANGE (order_date)
(
  PARTITION p_2025q1 VALUES LESS THAN (DATE '2025-04-01'),
  PARTITION p_2025q2 VALUES LESS THAN (DATE '2025-07-01'),
  PARTITION p_2025q3 VALUES LESS THAN (DATE '2025-10-01'),
  PARTITION p_2025q4 VALUES LESS THAN (DATE '2026-01-01'),
  PARTITION p_max    VALUES LESS THAN (MAXVALUE)
);
```

#### Локальный индекс по `order_date`

```sql
CREATE INDEX idx_orders_date_local
  ON orders (order_date)
  LOCAL;
```

#### Bitmap‑индекс по статусу

```sql
CREATE BITMAP INDEX idx_orders_status_bm
  ON orders (status)
  LOCAL;
```

### Практика / реальные сценарии

- **Хранилище заказов за много лет**:
  - таблица `orders` партиционирована по дате;
  - старые кварталы регулярно `DROP PARTITION` и архивируются на холодное хранилище;
  - отчёты используют bitmap‑индексы по статусам и категориям.
- **Система биллинга**:
  - composite‑партиции: `range` по дате + `hash` по account_id;
  - локальные индексы позволяют прозрачно балансировать нагрузку.

### Типичные ошибки

- Партиционирование “на всякий случай” без чёткого понимания ключей запросов.
- Создание только глобальных индексов и последующие проблемы с обслуживанием партиций.
- Использование bitmap‑индексов на **часто изменяемых** колонках в OLTP‑таблицах → тормоза при каждом `UPDATE/DELETE`.
- Отсутствие документации, какие партиции и когда можно удалять/архивировать.

### Что будет, если…

- …удалить партицию таблицы с глобальными индексами, не перестроив их?
  - Индексы могут стать `UNUSABLE`, запросы начнут падать или перестанут использовать эти индексы; восстановление индекса может быть затратно.
- …использовать interval‑партиционирование без мониторинга?
  - Можно неожиданно получить **сотни мелких партиций**, если ключ партиционирования выбирается плохо (например, по миллисекундам), что усложнит планы и обслуживание.

### Проверь себя

1. В каких случаях локальные индексы предпочтительнее глобальных?

<details><summary>Ответ</summary>
Когда активно используются операции на уровне партиций (архивирование, `EXCHANGE/TRUNCATE/DROP PARTITION`), а запросы в основном “попадают” внутрь отдельной партиции (например, работа по одному периоду/региону). Локальные индексы позволяют не перестраивать всё дерево при работе с одной партицией и проще в сопровождении.
</details>

2. Назови пример колонки, для которой bitmap‑индекс подходит хорошо, и пример, где он станет проблемой.

<details><summary>Ответ</summary>
Хорошо: колонка `status` с небольшим фиксированным набором значений (`'NEW'`, `'PAID'`, `'CANCELLED'`) в аналитической таблице с редкими изменениями. Плохо: колонка `last_updated_at` или `amount` в сильно изменяемой транзакционной таблице — там bitmap‑индекс будет постоянно перестраиваться и тормозить операции записи.
</details>

3. В чём практическая разница между range‑ и list‑партиционированием при проектировании схемы?

<details><summary>Ответ</summary>
При range‑партиционировании мы режем таблицу по **непрерывным диапазонам** значений (чаще всего по дате или числовому интервалу): это удобно для временных данных и “скользящих окон”, когда мы регулярно добавляем новые периоды и удаляем старые. При list‑партиционировании мы заранее задаём **конечные списки значений** (страны, регионы, типы клиентов), и каждая партиция привязана к своему набору. Это удобно, когда разбиение по категориям важнее времени, а сами категории относительно стабильны.
</details>

### Запомните

- Правильно спроектированное партиционирование + индексы в Oracle — это **главный инструмент масштабирования и управляемости больших таблиц**.
- Локальные и глобальные индексы, bitmap и function‑based — это **разные рычаги**, которые нужно применять осознанно под конкретные запросы и операции.
