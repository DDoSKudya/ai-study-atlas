## 245. SQL Server: партиционирование, индексы, статистика и query hints

### Цель раздела

Разобраться, как SQL Server реализует **партиционирование таблиц и индексов**, какие бывают **clustered/non‑clustered/filtered** индексы, как работает **статистика** и когда вообще стоит использовать **query hints** (и когда не стоит).

### В этом разделе главное

1. Партиционирование в SQL Server похоже по идее на Oracle, но реализовано через **partition function** и **partition scheme**.
2. **Clustered индекс** определяет физический порядок строк в таблице; **non‑clustered** строится поверх него.
3. **Filtered индексы** позволяют индексировать только подмножество строк.
4. **Статистика** критична для планов запросов; hints — крайняя мера, когда штатный оптимизатор даёт плохие планы.

### Термины

- **Partition function** — функция, определяющая границы партиций.
- **Partition scheme** — сопоставляет партиции файловым группам.
- **Clustered index** — индекс, по которому физически упорядочены строки таблицы.
- **Nonclustered index** — отдельное индексное дерево, указывающее на строки таблицы (через clustering key).
- **Filtered index** — индекс, построенный только на части строк (`WHERE` в определении индекса).
- **Statistics** — данные о распределении значений (гистограммы, селективность), которые использует оптимизатор.

### Теория и правила

Чтобы не запутаться в терминах, полезно сначала честно ответить себе на вопрос:

> “Что именно я хочу от схемы индексов и партиционирования в SQL Server?”

Обычно ответ такой:

- “Хочу, чтобы **частые запросы** выполнялись быстро”.
- “Хочу, чтобы **манипуляции с данными во времени** (архив, разделение по годам/месяцам) были удобными”.
- “Хочу, чтобы оптимизатор **понимал распределение данных** и строил адекватные планы”.

Тогда несложно принять идею, что:

- партиционирование отвечает за **разрезание таблицы на логические фрагменты**;
- clustered/non‑clustered/filtered индексы — за **то, как мы находим строки**;
- статистика — за то, **как оптимизатор оценивает стоимость разных путей**;
- hints — это уже “ручное управление” в редких случаях, когда автоматике не хватает информации или она ошибается.

#### 1) Партиционирование в SQL Server

Состоит из двух шагов:

1. **Partition function**:

```sql
CREATE PARTITION FUNCTION pfOrdersByDate (DATE)
AS RANGE RIGHT FOR VALUES
(
    '2025-04-01',
    '2025-07-01',
    '2025-10-01',
    '2026-01-01'
);
```

2. **Partition scheme**:

```sql
CREATE PARTITION SCHEME psOrders
AS PARTITION pfOrdersByDate
TO (FG_2025Q1, FG_2025Q2, FG_2025Q3, FG_2025Q4, FG_2026P);
```

Далее можно создавать таблицу на основе схемы:

```sql
CREATE TABLE Orders
(
    OrderId     BIGINT NOT NULL,
    CustomerId  BIGINT NOT NULL,
    OrderDate   DATE   NOT NULL,
    Status      VARCHAR(20),
    Amount      DECIMAL(18,2),
    CONSTRAINT PK_Orders PRIMARY KEY CLUSTERED (OrderDate, OrderId)
) ON psOrders (OrderDate);
```

Идея похожа на Oracle: разбиение по дате, возможность удалять/архивировать отдельные партиции, улучшение планов за счёт partition elimination.

#### 2) Clustered vs non‑clustered индексы

- **Clustered**:
  - определяет физический порядок строк (B‑tree по ключу);
  - таблица без clustered индекса называется **heap**;
  - обычно один на таблицу (редкие исключения).
- **Non‑clustered**:
  - содержит ключи и указатель на строку (clustering key или RID для heap);
  - может быть много;
  - подходит для ускорения выборок по другим полям.

Правило:

- ключ clustered индекса должен:
  - быть **стабильным** (не часто меняться);
  - хорошо распределяться;
  - часто использоваться в `JOIN`/`WHERE`.

#### 3) Filtered индексы

Позволяют индексировать только часть строк, например:

```sql
CREATE NONCLUSTERED INDEX IX_Orders_Open
    ON Orders (CustomerId)
WHERE Status = 'OPEN';
```

Плюсы:

- меньше размер индекса;
- лучше селективность;
- оптимизатор может строить очень эффективные планы для типичных запросов (`WHERE Status = 'OPEN'`).

#### 4) Статистика и query hints

- Статистика (AUTO_CREATE_STATISTICS, AUTO_UPDATE_STATISTICS) даёт оптимизатору информацию о распределении значений.
- Если статистика устарела или плохая — планы становятся неадекватными (over/under‑estimation).
- Query hints (`OPTION (RECOMPILE)`, `FORCESEEK`, `MAXDOP` и др.) — способ **переопределить** решения оптимизатора.

Золотое правило:

- сначала **улучшай статистику и схему** (индексы, партиционирование),  
  и только потом в крайнем случае используй hints.

### Пошагово: проектируем схему индексов в SQL Server

1. Выбери **clustered key**:
   - часто `Id` (identity), но в DWH это может быть `Date + Id`.
2. Определи основные запросы:
   - по клиенту, по дате, по статусу и т.д.
3. Создай non‑clustered индексы под эти запросы:
   - по столбцам в `WHERE`/`JOIN`;
   - возможно с `INCLUDE` для covered‑запросов.
4. Там, где много редко изменяемых строк с определённым состоянием, подумай про filtered index.
5. Настрой авто‑статистику и периодически проверяй её состояние.

### Простыми словами

- В SQL Server ты **очень явно управляешь физикой**:
  - какой индекс “ведёт” таблицу (clustered),
  - как разбить таблицу по партициям,
  - какие подмножества строк индексировать отдельно.
- Статистика — это “глаза” оптимизатора; hints — очки, которые ему насильно надевают.

### Картинка в голове

Представь таблицу как **упорядоченный журнал**:

- clustered индекс — порядок страниц в журнале;
- non‑clustered индексы — закладки по разным темам;
- filtered индекс — закладки только по определённому разделу;
- партиционирование — разрезы журнала на тома по годам.

### Как запомнить

> **Clustered** = “физический порядок”  
> **Non‑clustered** = “указатель сверху”  
> **Filtered** = “указатель только на часть строк”  
> **Статистика** = “карта местности для оптимизатора”

### Примеры

#### Filtered индекс под частый запрос

```sql
CREATE NONCLUSTERED INDEX IX_Orders_Pending
    ON Orders (CustomerId, OrderDate)
WHERE Status = 'PENDING';
```

Запрос:

```sql
SELECT *
  FROM Orders
 WHERE Status = 'PENDING'
   AND CustomerId = @cid
   AND OrderDate >= @fromDate;
```

С высокой вероятностью будет использовать именно этот индекс.

### Практика / реальные сценарии

- **Интернет‑магазин на SQL Server**:
  - таблица `Orders` партиционирована по дате;
  - clustered индекс по `(OrderDate, OrderId)`;
  - множество non‑clustered и несколько filtered индексов под статусы и бизнес‑сценарии;
  - регулярное обслуживание статистики и индексов.

### Типичные ошибки

- Создавать clustered индекс по часто изменяемому полю (например, статусу), что ведёт к постоянным “перестановкам” строк.
- Пренебрегать статистикой и потом “лечить” всё query hints.
- Создавать слишком много перекрывающихся индексов без понимания нагрузки на запись.

### Что будет, если…

- …отключить автообновление статистики на активно изменяемой таблице?
  - Планы запросов начнут деградировать: оптимизатор будет думать, что селективность одна, а на деле — другая; время выполнения запросов может вырасти на порядки.
- …попробовать “починить” плохой план постоянным `OPTION (RECOMPILE)`?
  - Снизится кэширование планов, возрастёт нагрузка на CPU и оптимизатор; проблема с данными/статистикой никуда не исчезнет.

### Проверь себя

1. В каких случаях filtered индекс особенно полезен?

<details><summary>Ответ</summary>
Когда есть **часто используемые запросы к небольшому подмножеству строк** (например, только активные/ожидающие заказы), а остальная часть таблицы большая и редко затрагивается этими запросами. Filtered индекс даёт компактный и высокоселективный индекс именно для этих строк.
</details>

2. Почему clustered индекс желательно делать на “стабильном” поле?

<details><summary>Ответ</summary>
Потому что clustered индекс определяет **физический порядок строк**; если ключ часто меняется, это приведёт к частым перемещениям строк, фрагментации и лишней нагрузке на запись. Стабильный ключ минимизирует эти накладные расходы.
</details>

3. Почему перед использованием query hints имеет смысл сначала проверить состояние статистики?

<details><summary>Ответ</summary>
Потому что оптимизатор строит планы, опираясь именно на статистику (гистограммы, селективность, оценки кардинальности). Если статистика устарела или неполная, он будет делать неверные предположения и выбирать неоптимальные планы. Во многих случаях достаточно обновить статистику и, при необходимости, откорректировать индексы/партиционирование, чтобы планы стали адекватными **без ручных hints**. Использовать hints стоит только когда мы уверены, что статистика хорошая, а оптимизатор всё равно по каким‑то причинам выдаёт неудовлетворительный план.
</details>

### Запомните

- В SQL Server сочетание **партиционирования, грамотных clustered/non‑clustered/filtered индексов и хорошо настроенной статистики** — основа производительности.
- Query hints — это **скальпель**, а не молоток: применять осторожно и осознанно.
