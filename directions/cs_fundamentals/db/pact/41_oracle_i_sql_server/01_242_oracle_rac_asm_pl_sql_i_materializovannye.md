## 242. Oracle: RAC, ASM, PL/SQL и материализованные представления

### Цель раздела

Понять, **как выглядят “фирменные” технологии Oracle**, из‑за которых эту СУБД часто выбирают для критичных корпоративных систем:  
RAC (кластер), ASM (управление хранилищем), PL/SQL (процедурный язык) и материализованные представления.  
Научиться видеть не только маркетинговые названия, но и **внутреннюю механику, плюсы, минусы и компромиссы**.

### В этом разделе главное

1. **RAC** — это не “просто репликация”, а **общий кластер с общим хранилищем** и сложной синхронизацией.
2. **ASM** берёт на себя роль “файловой системы/RAID для БД”, оптимизируя работу с дисками под Oracle.
3. **PL/SQL** — мощный процедурный язык, но его нужно использовать **осознанно**, не превращая БД в “монолит‑приложение”.
4. **Материализованные представления** дают сильный выигрыш в скорости чтения, но требуют аккуратного проектирования обновлений.

### Термины

- **RAC (Real Application Clusters)** — кластер Oracle, где несколько инстансов обращаются к общему хранилищу.
- **GCS/GES** — подсистемы Global Cache/Enqueue Services, которые синхронизируют доступ к блокам данных между инстансами.
- **ASM (Automatic Storage Management)** — слой, который управляет дисками/томами для Oracle, похож на специализированный volume manager.
- **PL/SQL** — процедурный язык Oracle для написания хранимых процедур, функций, пакетов, триггеров.
- **Материализованное представление (materialized view)** — физически хранящееся представление, периодически (или по триггеру) обновляемое.

### Теория и правила

Прежде чем разбирать “взрослые” детали, давай очень медленно разложим, **чего вообще люди хотят от таких технологий**, как RAC и ASM, и почему им мало “обычной” базы на одном сервере.

Интуитивно потребности выглядят так:

1. “Если один сервер упадёт — сервис должен выжить”.
2. “Нам может не хватать мощности одного сервера — нужно несколько”.
3. “Хочется больше надёжности и удобства с дисками, чем просто один том/одна файловая система”.

Single‑instance PostgreSQL/MySQL с репликацией закрывают это **частично** (отказоустойчивость, чтение с реплик), а Oracle идёт дальше и говорит:  
“Давайте сделаем так, чтобы **несколько экземпляров** могли работать **над одними и теми же файлами данных**, и при этом не сойти с ума”.  
Отсюда и вырастают RAC (как кластер поверх общих файлов) и ASM (как “умный слой” под этими файлами).

#### 1) Oracle RAC: общие принципы

Интуитивно:

- у тебя есть **одна база данных** (набор файлов данных) в общем хранилище;
- к ней одновременно подключены **несколько инстансов Oracle**, каждый на своём сервере;
- клиенты могут ходить к любому инстансу, а Oracle внутри кластера **согласует доступ к блокам данных**.

Ключевые моменты:

- **Общее хранилище** (часто SAN или кластерная файловая система) обязательно: все инстансы видят одни и те же файлы.
- Для согласования доступа используется **глобальный кэш**: блоки данных “переезжают” между инстансами, блокируются, инвалидация кэша распространяется по кластеру.
- Это даёт:
  - высокую доступность (узел можно потерять, остальные продолжат работать);
  - масштабирование по чтению/отчасти по записи (но не линейное).

Компромисс:

- сильно растёт сложность:
  - нужен общий storage;
  - координация блоков данных по сети;
  - увеличивается чувствительность к latency между узлами.

#### 2) ASM: Oracle‑специфичное управление хранилищем

ASM решает задачу:

- “как разложить файлы БД по множеству дисков/томов так, чтобы **БД видела удобные табличные пространства**, а не возилась с файловой системой и RAID”.

Принципы:

- диски объединяются в **disk group**;
- ASM распределяет блоки данных по дискам (striping), реализует зеркалирование (mirroring);
- БД видит **tablespace** и файлы данных на уровне ASM, не заботясь о конкретных физических дисках.

Практические эффекты:

- проще управлять хранилищем для больших Oracle‑инсталляций;
- можно гибко настраивать уровни отказоустойчивости и производительность.

#### 3) PL/SQL: зачем он нужен

PL/SQL добавляет к SQL:

- **переменные, циклы, условия**;
- процедуры, функции, пакеты;
- курсоры и обработку ошибок.

Зачем:

- можно реализовать часть бизнес‑логики **ближе к данным**:
  - пакетная обработка;
  - сложные валидации;
  - “тяжёлые” отчёты;
  - интеграция с внешними системами через процедуры.

Компромисс:

- код в PL/SQL сложнее тестировать и версионировать;
- риск получить “толстую БД”, где большая часть логики спрятана в процедурах.

#### 4) Материализованные представления в Oracle

Материализованное представление:

- хранит **результат запроса** (например, агрегацию по огромной фактовой таблице);
- может обновляться:
  - по расписанию (`REFRESH FAST/COMPLETE ON SCHEDULE`),
  - по событию (`ON COMMIT`);
- даёт очень быстрые чтения, но:
  - требует места;
  - усложняет обновление (особенно fast refresh).

Важная идея:

- `FAST REFRESH` возможен только если система знает, **как отслеживать изменения** в базовых таблицах:
  - журнальные таблицы;
  - специальные ограничения;
  - определённые типы запросов.

### Пошагово: как “прочувствовать” Oracle‑мир

1. Представь, что у тебя есть **крупная монолитная база** (банк, биллинг, ERP).
2. С ростом нагрузки одной машины становится мало:
   - ставишь RAC: несколько инстансов, общее хранилище → высокая доступность и распределение нагрузки.
3. Чтобы управлять десятками дисков/томов, включаешь ASM:
   - БД говорит “дай мне tablespace такой‑то”, ASM решает, на какие диски писать.
4. Часть критичной логики выносится в PL/SQL:
   - триггеры, процедуры, которые гарантированно отрабатывают **рядом с данными**.
5. Тяжёлые отчёты и витрины можно вынести в материализованные представления:
   - пользователи читают уже агрегированные данные, а не считают их каждый раз.

### Простыми словами

- **RAC** — это как если бы несколько поваров работали с **одной и той же кастрюлей** супа, а не каждый со своей копией — нужен строгий порядок, кто и когда мешает.
- **ASM** — это как “умная кухня”, которая сама решает, на какую полку холодильника поставить продукты и как распределить их между морозилками.
- **PL/SQL** — это возможность **писать мини‑приложения прямо внутри БД**, которые можно вызывать из SQL.
- **Материализованное представление** — это как **предварительно приготовленный полуфабрикат**: еда (результат запроса) уже готова, нужно только разогреть; но кто‑то должен периодически обновлять этот полуфабрикат.

### Картинка в голове

Представь Oracle как **многоэтажное здание**:

- Подвал — **ASM**, где лежат все трубы/провода (диски) и распределяются по этажам.
- Этажи — отдельные инстансы в **RAC**, подключённые к одним и тем же коммуникациям.
- Комнаты — схемы/таблицы/индексы.
- Встроенные “сценарии автоматизации” (PL/SQL) управляют процессами внутри здания.
- Отдельные “быстрые буфеты” — это материализованные представления, где самые популярные блюда уже разложены по тарелкам.

### Как запомнить

Короткая формула:

> **RAC = несколько инстансов + общее хранилище**  
> **ASM = Oracle‑RAID/volume manager**  
> **PL/SQL = процедурный SQL внутри БД**  
> **MView = кэшированный результат запроса**

### Примеры

#### PL/SQL: простая функция и процедура

```sql
-- Функция для расчёта скидки
CREATE OR REPLACE FUNCTION calc_discount(p_customer_id IN NUMBER)
  RETURN NUMBER
IS
  v_discount NUMBER := 0;
BEGIN
  SELECT CASE
           WHEN total_amount > 10000 THEN 0.1
           WHEN total_amount > 5000  THEN 0.05
           ELSE 0
         END
    INTO v_discount
    FROM customer_stats
   WHERE customer_id = p_customer_id;

  RETURN v_discount;
EXCEPTION
  WHEN NO_DATA_FOUND THEN
    RETURN 0;
END;
/

-- Процедура, использующая функцию
CREATE OR REPLACE PROCEDURE apply_discounts IS
BEGIN
  UPDATE orders o
     SET discount = calc_discount(o.customer_id)
   WHERE o.status = 'NEW';
END;
/
```

#### Материализованное представление

```sql
CREATE MATERIALIZED VIEW sales_daily_mv
BUILD IMMEDIATE
REFRESH FAST ON COMMIT
AS
SELECT trunc(order_date) AS order_day,
       product_id,
       SUM(amount)      AS total_amount,
       COUNT(*)         AS order_count
  FROM orders
 GROUP BY trunc(order_date), product_id;
```

Идея: при каждом коммите в `orders` Oracle будет поддерживать агрегаты в `sales_daily_mv`. Запросы отчётов читают уже агрегированные данные.

### Практика / реальные сценарии

- **Большой банк на Oracle**:
  - использует RAC для высокой доступности в одном дата‑центре;
  - ASM управляет десятками томов SAN;
  - бизнес‑критичные проверки реализованы в PL/SQL‑пакетах, вызываемых из приложений;
  - отчётность строится поверх материализованных представлений.
- **Старый монолит**:
  - большая часть логики переехала в PL/SQL за годы;
  - миграция на другую СУБД усложняется именно этим (нужно переносить логику из PL/SQL в код приложений).

### Типичные ошибки

- Считать, что RAC — это “просто репликация” и недооценивать требования к **storage и сети**.
- Использовать PL/SQL “без тормозов”, превращая БД в монолит с сотнями тысяч строк кода.
- Строить материализованные представления с `FAST REFRESH`, не понимая условий его применимости, и получать ошибки/полные пересчёты.
- Игнорировать необходимость **наблюдаемости и мониторинга** за RAC/ASM — в инцидентах это очень больно.

### Что будет, если…

- …пытаться использовать RAC в географически распределённом кластере с большой задержкой между узлами?
  - Внутренние протоколы синхронизации начнут страдать от latency, производительность резко упадёт; дизайн RAC предполагает **низкую задержку** между узлами (обычно один дата‑центр).
- …всё бизнес‑правило “запихнуть” в PL/SQL, не оставив логики в приложении?
  - Получится сложно поддерживаемый монолит “внутри БД”: сложнее тестировать, версионировать, подключать новые языки/сервисы, повышается зависимость от конкретной СУБД.

### Проверь себя

1. В чём архитектурное отличие Oracle RAC от PostgreSQL с репликацией и Patroni?

<details><summary>Ответ</summary>
В RAC все инстансы работают с **одним набором файлов данных** в общем хранилище, согласуя доступ к блокам через кластерный уровень (global cache/locks). В PostgreSQL каждый узел имеет **свою копию данных**, которую он получает через WAL‑репликацию; нет общей файловой системы и глобального cache‑координатора.
</details>

2. Назови два сценария, где PL/SQL особенно уместен, и один сценарий, где его лучше избегать.

<details><summary>Ответ</summary>
Уместен: (1) тяжёлая пакетная обработка данных (batch‑процедуры, ночные расчёты), где важно минимизировать сетевой трафик и работать “рядом с данными”; (2) критичные валидации/констрейнты, которые должны выполняться для любых клиентов, независимо от приложений. Избегать: сложные бизнес‑workflow с большим количеством внешних интеграций, требующие тесной работы с приложенческим кодом и современными практиками разработки — такие процессы лучше оставить в приложении.
</details>

3. Какую задачу решают материализованные представления по сравнению с “обычными” представлениями?

<details><summary>Ответ</summary>
Обычное представление — это по сути “сохранённый SQL‑запрос”: при каждом обращении БД снова выполняет запрос по базовым таблицам. Материализованное представление **физически хранит результат** этого запроса и обновляет его по правилам (`ON COMMIT`, по расписанию и т.п.). Поэтому чтение из MView гораздо быстрее (читаем уже предвычисленные данные), но нужно платить за обновление и хранение, а также следить за условиями консистентности (особенно для `FAST REFRESH`).
</details>

4. Почему в RAC критична низкая сетeвая задержка между узлами?

<details><summary>Ответ</summary>
Потому что инстансы в RAC постоянно обмениваются служебными сообщениями о блоках данных (global cache/locks), передают друг другу права на изменение и инвалидируют кэши. Если задержка велика, время на эти “разговоры” становится сопоставимым или даже больше времени самой работы с данными: транзакции начинают ждать друг друга, производительность резко падает, а преимущества RAC превращаются в недостатки.
</details>

### Запомните

- Oracle RAC и ASM — это **про большие, дорогие, критичные инсталляции**, где платят за высокую доступность и предсказуемость.
- PL/SQL и материализованные представления — мощные инструменты, но их нужно использовать **как часть архитектуры**, а не “по умолчанию везде”.
