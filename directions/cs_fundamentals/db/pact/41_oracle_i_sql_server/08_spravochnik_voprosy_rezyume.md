## Справочник по части (ключевые пункты)

### Oracle vs SQL Server: ключевые фичи

| Тема                    | Oracle                                                | SQL Server                                                  |
| ----------------------- | ----------------------------------------------------- | ----------------------------------------------------------- |
| Кластеры/HA            | RAC (shared storage)                                  | Always On AG (log‑репликация)                              |
| Управление хранилищем  | ASM (Automatic Storage Management)                    | Files/Filegroups, интеграция с SAN/Storage Spaces          |
| Процедурный язык       | PL/SQL                                                | T‑SQL                                                       |
| Партиционирование      | range/list/hash/composite/interval/reference          | partition function + partition scheme (range/list‑подобно) |
| Индексы                | B‑tree, bitmap, function‑based, локальные/глобальные | clustered/non‑clustered, filtered, Columnstore             |
| Материализованные вью  | MViews (с `FAST/COMPLETE REFRESH`)                    | Indexed views, materialized‑подобные механизмы             |
| In‑memory              | In‑Memory Column Store (опция), другие фичи           | In‑Memory OLTP (Hekaton), Columnstore                      |

## Частые сценарии

### Сценарий 1. Крупный банк на Oracle рассматривает частичный переход на PostgreSQL

- **Задача**: снизить стоимость владения, сохранить надёжность и производительность.
- Подход:
  - оставить критичное ядро (биллинг, бухгалтерия) на Oracle (RAC/ASM);
  - новые микросервисы и часть аналитики строить на PostgreSQL;
  - постепенно выносить часть логики из PL/SQL в сервисы.

### Сценарий 2. Корпоративная .NET‑система на SQL Server переезжает в Azure

- **Задача**: уменьшить заботы об инфраструктуре, улучшить DR.
- Подход:
  - миграция на Azure SQL Managed Instance (максимальная совместимость);
  - использование встроенного HA/DR и бэкапов;
  - перенос части тяжёлых отчётов на Columnstore и/или отдельный DWH.

### Сценарий 3. Миграция с Oracle на PostgreSQL

- **Задача**: уйти от дорогих лицензий.
- Подход:
  - детальный аудит PL/SQL‑кода (что можно вынести в сервисы, что переписать на PL/pgSQL);
  - маппинг типов и схем;
  - репетиции миграции на копиях продданных;
  - поэтапный cutover (по подсистемам).

## Вопросы для самопроверки по части

1. Чем архитектурно отличается Oracle RAC от Always On Availability Groups в SQL Server?
2. Какие типы партиционирования есть в Oracle и как они помогают управлять большими таблицами?
3. В каких сценариях Columnstore индексы в SQL Server особенно полезны, а где они могут навредить?
4. Какие основные шаги нужно учесть при миграции с Oracle/SQL Server на PostgreSQL?
5. Какие плюсы и минусы дают managed‑версии Oracle/SQL Server (RDS, Azure SQL) по сравнению с self‑managed установками?

<details><summary>Ответы</summary>

1. В Oracle RAC несколько инстансов работают с **общим хранилищем данных**, согласуя доступ к блокам через кластерные сервисы (global cache/locks). В SQL Server Always On AG каждая реплика имеет **собственную копию данных**, синхронизируемую через transaction log; это репликация на уровне логов, а не общий диск.

2. В Oracle есть range, list, hash, composite, interval и reference‑партиционирование. Они позволяют разбивать большие таблицы по датам, регионам, хешу и связям между таблицами, тем самым ускоряя диапазонные запросы (`partition pruning`) и упрощая операции архивации/удаления (`DROP/TRUNCATE/EXCHANGE PARTITION`).

3. Columnstore индексы особенно полезны в **аналитических нагрузках** (DWH, отчётность), где типичны большие сканы и агрегации по нескольким колонкам; они дают сильное сжатие и ускорение чтения. Могут навредить в **чисто OLTP‑сценариях** с частыми точечными обновлениями/вставками и небольшими выборками по ключу — накладные расходы на поддержание Columnstore перевесят выгоду.

4. Основные шаги: (1) инвентаризация схем, данных и кода в БД; (2) mapping типов данных и диалектов SQL (PL/SQL/T‑SQL → PL/pgSQL/код приложений); (3) проектирование целевой схемы и партиционирования; (4) выбор стратегии переноса данных (дампы, репликация, ETL); (5) полноценное тестирование (функциональное, производительность, сравнение результатов); (6) план cutover и отката.

5. Плюсы: меньше операционной нагрузки (бэкапы, патчи, HA/DR “из коробки”), быстрый старт, интеграция с облачным мониторингом. Минусы: ограничения по фичам/расширениям/конфигам, более высокая стоимость за единицу ресурса при больших объёмах, риск vendor lock‑in и необходимость всё равно понимать внутреннюю механику СУБД для проектирования схем/запросов.

</details>

## Типичные ошибки по части

- Считать, что RAC/Always On “магически” решают все проблемы масштабирования и отказоустойчивости без понимания ограничений.
- Игнорировать разницу в типах данных и поведение дат/таймзон при миграциях.
- Переносить всю бизнес‑логику в PL/SQL/T‑SQL, не оставляя её в приложениях.
- Полагаться только на managed‑сервисы без своих бэкапов и плана DR.
- Выбирать СУБД из соображений “мода/религия”, а не из инженерного анализа требований и стоимости.

## Резюме части XLI

- Oracle и SQL Server — **богатые на Enterprise‑фичи СУБД**: RAC, ASM, Always On, Columnstore, In‑Memory и мощные процедурные языки PL/SQL и T‑SQL.
- Их сильные стороны раскрываются при **грамотном проектировании партиционирования, индексов, бэкапов и HA‑архитектуры**.
- Миграции между СУБД требуют **аккуратной работы с типами, диалектами SQL, кодом в БД и поведением транзакций** — “дампа” недостаточно.
- Managed‑версии (RDS, Azure SQL и др.) упрощают эксплуатацию, но не снимают ответственности за **архитектуру, данные и стратегию выхода из vendor lock‑in**.
