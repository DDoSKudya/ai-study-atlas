## 248. Управляемые версии Oracle и SQL Server

### Цель раздела

Понять, как устроены **managed‑версии Oracle/SQL Server** (AWS RDS, Azure SQL и др.):  
какие задачи они снимают (бэкапы, патчи, failover), какие ограничения вводят, как они сочетаются с традиционными установками и как влияют на архитектуру.

### В этом разделе главное

1. Managed‑БД снимают часть операционных задач, но **не отменяют необходимости понимать саму СУБД**.
2. RDS для Oracle/SQL Server и Azure SQL имеют **жёсткие ограничения** (расширения, доступ к файловой системе, кастомные конфиги).
3. Важные вопросы: **retention бэкапов, HA/DR, кросс‑регион, масштабирование, стоимость**.
4. Managed‑БД часто становятся отличным выбором для старта/средних нагрузок, но требуют продуманной стратегии для масштабирования и миграций.

### Термины

- **RDS for Oracle/SQL Server** — managed‑сервисы AWS для соответствующих СУБД.
- **Azure SQL Database / Managed Instance** — managed‑варианты SQL Server в Azure.
- **Vendor lock‑in** — зависимость от конкретного облачного поставщика и его API/функций.

### Теория и правила

#### 1) Что даёт managed‑БД “из коробки”

- **Автоматические бэкапы**:
  - окна бэкапов;
  - PITR в рамках retention.
- **Патчи и обновления**:
  - регулярные security‑патчи;
  - иногда мажорные апгрейды (по выбору).
- **Мониторинг и алертинг**:
  - базовые метрики (CPU, I/O, connections);
  - интеграция с облачными системами мониторинга.
- **Высокая доступность**:
  - Multi‑AZ (RDS);
  - зоны доступности/region‑replication (Azure SQL).

#### 2) Ограничения

- Ограниченный доступ:
  - нельзя ставить произвольные расширения/плагины;
  - нет прямого доступа к файловой системе.
- Лимиты:
  - по размерам БД;
  - по типам инстансов/дисков;
  - по версиям СУБД.
- Цена:
  - “плата за комфорт”: выше, чем “голый” IaaS;
  - особенно при больших объёмах и высокой нагрузке.

#### 3) Managed Oracle/SQL Server: особенности

- **RDS for Oracle**:
  - ограниченный набор опций Oracle;
  - часть enterprise‑функций может быть недоступна;
  - обновления и патчи контролируются AWS.
- **RDS for SQL Server**:
  - не всё, что доступно в on‑prem SQL Server, доступно в RDS (ограничения по Agent, CLR, linked servers и т.д.);
  - важно читать список поддерживаемых фич.
- **Azure SQL Database/Managed Instance**:
  - Database — “более облачный” вариант (multi‑tenant, ряд ограничений);
  - Managed Instance — ближе к полному SQL Server с большим набором совместимых фич.

### Пошагово: когда рассматривать managed‑варианты

1. Если:
   - у тебя нет сильной DBA‑команды;
   - ты хочешь быстро стартовать и не заниматься инфраструктурой;
   - нагрузки пока умеренные;
   → managed‑БД — отличный выбор.
2. Если:
   - у тебя критичный энтерпрайз с нестандартными требованиями;
   - нужны специфичные фичи/расширения;
   - большая часть логики “зашита” в БД;
   → возможно, on‑prem/самостоятельно управляемые БД более уместны.

### Простыми словами

Managed‑БД — это как **арендовать квартиру с обслуживанием**:

- тебе не нужно самому чинить крышу/лифт/коммуникации (бэкапы, патчи, failover);
- но ты не можешь свободно сносить стены и проводить любые эксперименты (ограничения на фичи и конфиги);
- и аренда дороже, чем “жить в своём доме”, особенно в долгосрочной перспективе.

### Картинка в голове

Нарисуй:

- on‑prem Oracle/SQL Server — “свой дом” с полным контролем и ответственностью;
- RDS/Azure SQL — “дом с управляющей компанией”, где большинство технических задач с тебя снято, но и свободы меньше.

### Как запомнить

> **Managed‑БД = меньше операционки, больше ограничений и выше цена за единицу ресурса.**

### Примеры

- **RDS for SQL Server** для средней CRM‑системы:
  - автоматические бэкапы и Multi‑AZ;
  - минимум DBA‑работ;
  - позже часть аналитики выносится в отдельный DWH.
- **Azure SQL Database** под небольшое SaaS‑приложение:
  - простой старт;
  - масштабирование по потреблению;
  - при росте потребностей возможен переезд на Managed Instance.

### Практика / реальные сценарии

- Компания запускает новый продукт и выбирает Azure SQL Database:
  - не нанимает отдельного DBA;
  - пользуется встроенными бэкапами и мониторингом;
  - через пару лет при росте данных и нагрузки переезжает на Managed Instance и/или отдельный DWH.

### Типичные ошибки

- Думать, что managed‑БД “решают все проблемы”, и не разбираться в транзакциях, индексах, бэкапах.
- Не читать список ограничений и обнаруживать их уже в проде.
- Не иметь стратегии выхода из vendor lock‑in (резервные дампы, кросс‑регион/кросс‑аккаунт).

### Что будет, если…

- …полагаться только на встроенные бэкапы managed‑БД без собственных дампов и плана DR?
  - При серьёзных инцидентах (проблемы с аккаунтом, регионом, ошибочные операции администратора) может оказаться сложно или невозможно восстановить данные вне конкретного провайдера/региона; риск vendor lock‑in возрастает.

### Проверь себя

1. Какие три вопроса ты задашь себе перед выбором managed‑версии Oracle/SQL Server?

<details><summary>Ответ</summary>
1) Какие фичи СУБД мне нужны и доступны ли они в managed‑варианте? 2) Как устроены бэкапы/PITR/HA/DR (retention, cross‑region, RPO/RTO)? 3) Какова стоимость (включая рост данных и нагрузки) и есть ли план на случай выхода из облака/смены провайдера?
</details>

2. Почему даже при использовании RDS/Azure SQL стоит понимать, как работает партиционирование, индексы и транзакции?

<details><summary>Ответ</summary>
Потому что managed‑сервис берёт на себя **операционку** (бэкапы, патчи, failover), но **не проектирование схемы и запросов**. Неправильные индексы, партиционирование и транзакции приведут к тем же проблемам, что и на self‑managed СУБД: медленным запросам, блокировкам, инцидентам.
</details>

3. Как managed‑БД влияют на стратегию выхода из vendor lock‑in?

<details><summary>Ответ</summary>
Managed‑БД завязывают бэкапы, HA/DR и часть инструментов на конкретного провайдера и его API. Если не продумать заранее независимые копии данных (логические дампы, реплики в другой среде, кросс‑аккаунт/кросс‑регион), то при необходимости сменить провайдера, регион или схему безопасности можно оказаться “запертым” внутри текущей платформы. Поэтому стратегия выхода должна включать: регулярные независимые бэкапы, документацию по восстановлению вне облака и понимание, какие фичи придётся реализовывать иначе при переезде.
</details>

### Запомните

- Managed‑БД — отличный инструмент, если понимать их **границы и ограничения**.
- Даже в облаке ответственность за **данные, схемы и архитектуру** остаётся на тебе.
