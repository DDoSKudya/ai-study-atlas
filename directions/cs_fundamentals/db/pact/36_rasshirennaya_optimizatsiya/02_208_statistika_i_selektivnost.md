## 208. Статистика и селективность

### В этом разделе главное

- Планировщик оценивает «сколько строк пройдёт фильтр» (**селективность**) на основе **статистики**: число строк в таблице, число уникальных значений, гистограммы и MCV по столбцам.
- **Гистограмма** разбивает диапазон значений столбца на «корзины» (buckets) и хранит частоты; по ней оцениваются условия вида `column < value`, `column BETWEEN a AND b`.
- **MCV (Most Common Values)** — список самых частых значений и их доля; даёт точную оценку для `column = constant`, когда constant входит в MCV.
- **Correlation** — коэффициент корреляции между логическим порядком значений и физическим на диске; используется для оценки стоимости index scan при range-условиях (высокая корреляция — последовательное чтение дешевле).
- Неверная или устаревшая статистика приводит к неверным оценкам rows и к плохим планам; **ANALYZE** обновляет статистику; для «сложных» столбцов можно увеличить объём статистики (**target**).
- **Расширенная статистика (CREATE STATISTICS)** — для коррелированных столбцов; улучшает оценки условий по нескольким столбцам.
- **Autovacuum / auto-analyze** — фоновый процесс автоматически запускает ANALYZE; после больших загрузок лучше явный ANALYZE.

---

### Простыми словами

Чтобы решить «использовать индекс или полное сканирование» и «в каком порядке соединять таблицы», планировщику нужно знать: **сколько примерно строк** отфильтруется каждым условием. Эту оценку он строит по **статистике**, которую собирает команда **ANALYZE** (или автоматический autovacuum/analyze). Статистика хранится в системных каталогах (в PostgreSQL — в pg_statistic и в виде человекочитаемых оценок в pg_stats).

**Гистограмма** — это упрощённая картина распределения: «сколько значений попало в первый диапазон, сколько во второй…». По ней планировщик оценивает условия вроде «где цена < 100»: какая доля строк попадает в этот диапазон. **MCV** — «топ» самых частых значений (например, статусы 'new', 'paid'); для точного равенства `status = 'paid'` планировщик смотрит долю 'paid' в MCV и получает хорошую оценку. **Корреляция** говорит: насколько порядок значений на диске совпадает с порядком по величине. Если корреляция высокая (например, даты вставки идут по возрастанию на диске), то range-скан по индексу будет читать блоки почти подряд — это дешевле; если низкая — чтение «вразброс», дороже.

Когда статистика устарела (много вставок/удалений после последнего ANALYZE) или столбец имеет «неудобное» распределение (например, много редких значений), оценки rows становятся неточными и план может быть плохим. Решение: регулярный ANALYZE и при необходимости увеличение **n_distinct** или **target** для гистограммы/MCV по конкретным столбцам.

**Когда статистика влияет сильнее всего.** (1) **Выбор index scan vs seq scan** — планировщик смотрит на оценку числа строк после фильтра; если из-за плохой статистики он завышает или занижает эту оценку, он может выбрать полное сканирование там, где выгоден индекс, или наоборот. (2) **Порядок и тип JOIN** — от оценок rows по каждой таблице зависит, какую таблицу считать «внешней» в Nested Loop и строить ли хеш по меньшей в Hash Join; неверные rows ведут к неоптимальному порядку соединений. (3) **Коррелированные подзапросы** — планировщик решает, разворачивать ли подзапрос в JOIN; неточная статистика может помешать ему выбрать развёртывание. Поэтому после больших изменений данных ANALYZE не просто «рекомендуется», а часто критичен для нормальных планов.

### Числовой пример: зачем нужна селективность

В таблице **1 000 000** строк. Запрос: `SELECT * FROM orders WHERE status = 'paid'`. Планировщику нужно решить: выгоднее прочитать по индексу по `status` или сделать полное сканирование таблицы? Для этого он должен **оценить**, сколько строк выйдет после фильтра. Если он думает, что строк с `status = 'paid'` будет **100** — скорее выберет индекс. Если думает, что таких строк **400 000** — может предпочесть полное сканирование. Откуда он берёт число? Из **статистики**: в столбце `status` хранятся, например, значения 'new', 'paid', 'shipped', 'cancelled'. Статистика может хранить: «значение 'paid' встречается в 30% строк» (MCV). Тогда оценка = 1 000 000 × 0,30 = **300 000** строк. Или если по равенству используется n_distinct: «уникальных значений 4», тогда грубая оценка 1/4 = 25% строк, т.е. 250 000. Эта **оценка числа строк (rows)** и есть то, что ты видишь в плане. Чем точнее статистика (актуальный ANALYZE, при необходимости больше MCV и корзин гистограммы), тем точнее rows и тем лучше выбор плана.

**Пример: как одна и та же оценка меняет план.** Таблица 1 000 000 строк, запрос `WHERE status = 'paid'`. Если статистика говорит «paid — 0,1%» → оценка 1000 строк → планировщик скорее выберет **Index Scan** по status (мало строк — индекс выгоден). Если статистика говорит «paid — 40%» → оценка 400 000 строк → планировщик скорее выберет **Seq Scan** (много строк — полное сканирование часто быстрее). Один и тот же запрос, **разная статистика** → **разный план**. Поэтому устаревшая статистика (например, после массовой загрузки, когда paid стало 30%, а в MCV ещё старая доля) приводит к неверному выбору: планировщик может выбрать Seq Scan, когда индекс был бы быстрее, или наоборот.

### Теория и правила

#### Селективность

**Селективность** условия — это доля строк таблицы, для которых условие истинно. Оценка числа строк = (число строк таблицы) × (селективность). Например, если селективность 0.01, а в таблице 1 000 000 строк, оценка выхода фильтра — 10 000 строк. Планировщик использует формулы и статистику:

- для `column = constant`: часто 1 / n_distinct или доля из MCV, если constant в MCV;
- для `column < constant` или `column BETWEEN a AND b`: по гистограмме оценивается доля значений в интервале;
- для `column IS NULL`: отдельно хранимая доля NULL;
- для комбинаций условий (AND/OR) — комбинирование селективностей по правилам вероятности (независимость обычно предполагается).

Чем точнее статистика (больше корзин гистограммы, больше MCV, актуальные n_distinct и null fraction), тем точнее оценки и планы.

#### Гистограммы

Гистограмма разбивает диапазон значений столбца на **равночастотные** или **равноширинные** корзины (в PostgreSQL — чаще равночастотные). В каждой корзине хранится граница (или границы) и информация о частоте. Для условия `column < X` планировщик определяет, в какие корзины попадает X, и суммирует доли — получает оценку селективности. Точность зависит от числа корзин (в PostgreSQL задаётся через **statistics target** для столбца; по умолчанию 100).

**Простой образ гистограммы.** Представь столбец `price` с значениями от 1 до 1000. Гистограмма с 10 корзинами может хранить: «в корзине 1 — 10% строк (цены примерно 1–100), в корзине 2 — 12% (100–200), …». Для условия `price < 250` планировщик складывает доли корзин 1 и 2 и часть корзины 3 и получает, например, «около 35% строк» — селективность 0,35. Если в таблице 100 000 строк, оценка выхода фильтра = 100 000 × 0,35 = 35 000 строк. Больше корзин (больше statistics target) — точнее границы и доли, точнее оценка для «странных» диапазонов.

**Равночастотная гистограмма — что это простыми словами.** В PostgreSQL гистограмма чаще всего **равночастотная**: корзины подбираются так, чтобы в **каждую корзину попадало примерно одинаковое число строк** (а не одинаковый диапазон значений). Например, если в столбце цены от 1 до 1 000 000 и много дешёвых товаров, первая корзина может охватывать цены 1–50 (много строк), а последняя — 500 000–1 000 000 (мало строк). Так планировщик лучше оценивает «сколько строк с ценой < 100»: он смотрит, в каких корзинах лежит 100, и суммирует их доли. **Равноширинная** (одинаковый диапазон в каждой корзине) в таком случае была бы менее точной, потому что в одних корзинах оказалось бы очень много строк, в других — мало.

#### MCV (Most Common Values)

Список из N самых частых значений и их доля в таблице (и при необходимости доля NULL). Для условия `column = 'value'` планировщик смотрит: если 'value' есть в MCV — берёт сохранённую долю (точная оценка); если нет — отступает к оценке через n_distinct (1/n_distinct или по остатку). Увеличение числа MCV (через увеличение statistics target по столбцу) улучшает оценки для столбцов с «пиками» по нескольким значениям.

**n_distinct — что это простыми словами.** **n_distinct** — это оценка **числа уникальных значений** в столбце. Например, в столбце `status` могут быть значения 'new', 'paid', 'shipped', 'cancelled' — n_distinct ≈ 4. Для условия `column = constant` при равномерном распределении планировщик часто оценивает селективность как **1 / n_distinct** (каждое значение встречается примерно в 1/4 строк). Если n_distinct = 1000, то грубая оценка для равенства — 1/1000 = 0,1% строк. n_distinct собирается при ANALYZE и хранится в pg_statistic. Если в столбце много уникальных значений (например, id или email), n_distinct большое — оценка для равенства получается маленькой (мало строк), и планировщик скорее выберет индекс.

#### Correlation

**Correlation** — коэффициент между порядком значений в столбце (логически) и порядком строк на диске (физически). Близок к 1 или -1 — значения в столбце сильно упорядочены на диске (например, автоинкрементный id или дата вставки); близок к 0 — порядок случайный. Используется при оценке **стоимости** index scan: при высокой корреляции индекс по range читает страницы таблицы почти последовательно (меньше random I/O), cost ниже; при низкой — много случайных обращений, cost выше. Это влияет на выбор между index scan и seq scan для range-условий.

#### Влияние на планы

Неверная статистика → неверные rows → неверный выбор плана (например, выбор seq scan вместо index scan или наоборот; неоптимальный порядок JOIN или тип JOIN). Типичные признаки в плане после EXPLAIN ANALYZE: сильное расхождение между **rows** и **actual rows** на узлах сканов или JOIN. Решение: **ANALYZE** таблицы (или всей БД); для проблемных столбцов — увеличить **statistics target**, затем снова ANALYZE.

#### Настройка сбора статистики (PostgreSQL)

- Глобально: `default_statistics_target` (по умолчанию 100) — число корзин гистограммы и размер списка MCV по умолчанию.
- По столбцу: `ALTER TABLE t ALTER COLUMN c SET STATISTICS 500;` — для столбца c собирать больше статистики; затем `ANALYZE t;`.

Увеличение target улучшает точность оценок, но увеличивает время ANALYZE и размер системных каталогов. Имеет смысл для столбцов, участвующих в важных фильтрах и JOIN и по которым распределение «сложное».

**Когда имеет смысл увеличивать statistics target по столбцу.** (1) В плане после EXPLAIN ANALYZE **rows** по этому столбцу (или по узлу, который от него зависит) сильно расходится с **actual rows** — значит, текущей статистики недостаточно. (2) Столбец участвует в частых условиях **равенства** (`column = constant`), а значений много и несколько из них «частые» — тогда увеличение target расширит список MCV и улучшит оценку. (3) Столбец участвует в условиях **диапазона** (`column < X`, BETWEEN), а распределение неравномерное (например, много дат в последнем месяце) — больше корзин гистограммы дадут точнее оценку. (4) После ANALYZE планы по-прежнему неоптимальны и ты уже проверил, что статистика свежая — попробуй увеличить target для ключевых столбцов (например, до 300–500) и снова ANALYZE.

**Что именно делает ANALYZE (упрощённо).** Команда **ANALYZE** читает **выборку** строк из таблицы (не всю таблицу целиком — это ускоряет сбор). По этой выборке считаются: число строк в таблице (экстраполяция), число уникальных значений по каждому столбцу (n_distinct), доли NULL, гистограмма по каждому столбцу (границы корзин и доли), список MCV. Результаты записываются в системные каталоги (в PostgreSQL — в `pg_statistic`; человеку удобнее смотреть представление `pg_stats`, где те же данные в читаемом виде). Планировщик при построении плана обращается к этой статистике. Поэтому после массовых вставок или удалений старые цифры перестают соответствовать данным — нужно снова запустить ANALYZE, чтобы обновить выборку и пересчитать статистику.

#### Расширенная статистика (Extended Statistics)

Обычная статистика собирается **по каждому столбцу отдельно**. Планировщик предполагает **независимость** столбцов: селективность `a = 1 AND b = 2` оценивается как (селективность a) × (селективность b). Но если столбцы **коррелированы** (например, `city` и `region` — у каждого города один регион), реальная селективность может быть **выше**: строк с city='Moscow' AND region='Central' меньше, чем «произведение долей». **Расширенная статистика** хранит информацию о **сочетаниях столбцов**.

**CREATE STATISTICS** — команда создания расширенной статистики. Примеры:
- `CREATE STATISTICS st1 ON (city, region) FROM orders;` — статистика по паре столбцов; планировщик лучше оценит условия `WHERE city = 'X' AND region = 'Y'`.
- `CREATE STATISTICS st2 (ndistinct) ON (customer_id, status) FROM orders;` — число уникальных комбинаций (customer_id, status); полезно для GROUP BY или JOIN по этим столбцам.
- `CREATE STATISTICS st3 (dependencies) ON (city, region) FROM orders;` — зависимости между столбцами (city → region).

После создания статистики выполни **ANALYZE** по таблице. **Когда использовать:** (1) В плане **rows** для условия по нескольким столбцам сильно расходится с **actual rows** — возможно, столбцы коррелированы. (2) JOIN по нескольким столбцам даёт неверную оценку. (3) GROUP BY по нескольким столбцам — расширенная ndistinct улучшает оценку. **Ограничение:** увеличивает время ANALYZE и размер каталогов; создавай только для «проблемных» комбинаций столбцов.

#### Autovacuum и auto-analyze

**Autovacuum** — фоновый процесс PostgreSQL, который периодически запускает **VACUUM** и **ANALYZE** по таблицам. **Auto-analyze** — часть autovacuum: когда в таблице накапливается достаточно изменений (вставок, обновлений, удалений), autovacuum запускает **ANALYZE** автоматически. Пороги задаются настройками:
- **autovacuum_analyze_threshold** (по умолчанию 50) — минимальное число изменённых строк, после которого рассматривается ANALYZE;
- **autovacuum_analyze_scale_factor** (по умолчанию 0.1) — доля от размера таблицы; ANALYZE запускается, когда число изменений превышает threshold + scale_factor × число строк.

**Пример:** таблица 1 000 000 строк, scale_factor=0.1 → порог 50 + 100 000 = 100 050 изменений. Пока не накопилось ~100 000 изменений, auto-analyze не запустится. **Проблема:** после массовой загрузки 500 000 строк autovacuum может не успеть (или ещё не сработал) — статистика устарела. **Решение:** явно запускать **ANALYZE** после больших загрузок; не полагаться только на autovacuum.

**Проверить, когда последний раз запускался analyze:** `SELECT relname, last_analyze, last_autoanalyze FROM pg_stat_user_tables WHERE relname = 'orders';` — **last_analyze** (ручной или авто), **last_autoanalyze** (только авто). Если оба NULL или давно — статистика может быть устаревшей.

### Пошагово: обновление и настройка статистики

1. **Обновить статистику по таблице:**  
   `ANALYZE orders;`  
   Или по всей БД: `ANALYZE;`

2. **Проверить текущий target по столбцу:**  
   В PostgreSQL: `SELECT attname, attstattarget FROM pg_attribute WHERE attrelid = 'orders'::regclass AND attnum > 0 AND NOT attisdropped;`

3. **Увеличить target для столбца (например status):**  
   `ALTER TABLE orders ALTER COLUMN status SET STATISTICS 500;`  
   Затем: `ANALYZE orders;`

4. **После изменения данных (массовые вставки/удаления):** запускать ANALYZE по расписанию или после пакетных загрузок, чтобы статистика не отставала.

**Как посмотреть статистику в PostgreSQL (pg_stats).** Чтобы увидеть, какую статистику планировщик использует по столбцу, выполни: `SELECT * FROM pg_stats WHERE tablename = 'orders' AND attname = 'status';` — увидишь `n_distinct`, `most_common_vals` (MCV), `histogram_bounds` (границы корзин гистограммы) и др. Это помогает понять, почему планировщик даёт ту или иную оценку rows. Если после ANALYZE статистика по столбцу пустая или странная — возможно, нужно увеличить `statistics target` для этого столбца.

**Пошагово: что делать при расхождении rows и actual rows.**  
Шаг 1: Сними план с `EXPLAIN (ANALYZE)` и найди узел, где **rows** сильно отличается от **actual rows** (например, rows=100, actual rows=50000).  
Шаг 2: Определи **таблицу и столбцы**, от которых зависит этот узел (какие таблицы сканируются, по каким столбцам фильтр).  
Шаг 3: Выполни `ANALYZE table_name;` по каждой таблице.  
Шаг 4: Снова сними план. Если rows и actual rows сблизились — статистика обновлена. Если расхождение осталось — перейди к шагу 5.  
Шаг 5: Увеличь статистику по столбцу: `ALTER TABLE table_name ALTER COLUMN column_name SET STATISTICS 500;` затем `ANALYZE table_name;`  
Шаг 6: Снова сними план. Если план стал лучше (другой тип JOIN, индекс вместо Seq Scan — по actual time) — проблема была в статистике.  
Шаг 7: Если после всего планы не улучшились — возможно, причина в другом (селективность условия неочевидна, нужен другой индекс или переписывание запроса).

**Как проверить, что статистика устарела.** (1) Сравни **rows** и **actual rows** в плане после EXPLAIN ANALYZE: сильное расхождение (в разы) — признак устаревшей или неточной статистики. (2) Посмотри дату последнего ANALYZE: `SELECT schemaname, relname, last_analyze FROM pg_stat_user_tables WHERE relname = 'orders';` — если last_analyze давно (до массовой загрузки), статистика устарела. (3) После массовых INSERT/UPDATE/DELETE — статистика почти всегда устаревает; делай ANALYZE по умолчанию.

**Что будет, если долго не запускать ANALYZE.** Статистика перестанет соответствовать реальным данным: число строк в таблице выросло или упало, распределение значений в столбцах изменилось. Планировщик будет опираться на **устаревшие** оценки. В результате он может выбрать seq scan там, где уже выгоден индекс (или наоборот), неоптимальный тип или порядок JOIN. В плане после EXPLAIN ANALYZE это проявится как сильное расхождение **rows** и **actual rows**. Поэтому после больших изменений данных всегда имеет смысл запускать ANALYZE.

**Ещё проще: образ гистограммы и MCV.** Представь столбец «цена» с значениями от 1 до 1000. **Гистограмма** — это разбиение на отрезки: «от 1 до 100 — 20% строк, от 100 до 200 — 15%, …». Планировщик по ней отвечает на вопрос: «какая доля строк с ценой < 250?» — складывает доли нужных отрезков. **MCV** — это «топ значений»: «цена=99 встречается в 5% строк, цена=199 — в 3%, …». Для условия «цена = 99» планировщик смотрит в MCV и сразу берёт 5%. Итого: **гистограмма** — для «меньше/больше/между»; **MCV** — для «равно» по частым значениям. Запомни так: диапазон → гистограмма; точное значение → MCV.

---

### Картинка в голове

Статистика — это **краткая сводка** таблицы: «сколько строк, сколько уникальных значений, как значения распределены по диапазону (гистограмма) и какие значения самые частые (MCV)». Планировщик по этой сводке прикидывает: «если применить этот фильтр, останется примерно столько строк». Чем сводка точнее и свежее, тем лучше прикидка и план.

---

### Как запомнить (§208)

- **Селективность** = доля строк, проходящих условие. Оценка строк = (число строк таблицы) × (селективность).
- **Гистограмма** — для условий по **диапазону** (<, >, BETWEEN): корзины и доли. **MCV** — для **равенства** (=) по частым значениям: точная доля.
- **Correlation** не меняет оценку строк (rows), а меняет **стоимость** index scan: высокая корреляция — дешевле читать по индексу при range.
- Плохие планы из-за статистики → **ANALYZE**; для «сложных» столбцов → **ALTER TABLE ... SET STATISTICS N** и снова ANALYZE.

---

### Типичные ошибки (§208)

- **Забывать запускать ANALYZE** после массовых вставок/удалений или импорта — статистика устаревает, rows становятся неверными, план портится.
- **Считать, что гистограммы хватает для равенства.** Для `column = constant` точнее MCV; если значения «пиковые», увеличивай statistics target по этому столбцу.
- **Не учитывать correlation.** На таблицах, упорядоченных по дате/id, correlation высокая — index range scan часто выгоден; после массовых обновлений порядок на диске может нарушиться, correlation падает — планировщик может переключиться на seq scan.

**Часто путают (§208).**

| Путают | В чём разница |
|--------|----------------|
| **Селективность и correlation** | **Селективность** — доля строк, проходящих условие; влияет на **оценку rows**. **Correlation** — связь порядка на диске с порядком по величине; не меняет rows, влияет на **cost** index scan при range (высокая корреляция — дешевле читать по индексу). |
| **Гистограмма и MCV** | **Гистограмма** — для условий по **диапазону** (<, >, BETWEEN). **MCV** — для **равенства** (=), точная доля частых значений. Для «цена < 100» используют гистограмму; для «status = 'paid'» — MCV. |

### Проверь себя

<details>
<summary>Зачем планировщику MCV, если есть гистограмма?</summary>

Гистограмма хорошо описывает **распределение по диапазону** (для условий <, >, BETWEEN). Но для условия **равенства** (`column = constant`) точнее знать долю именно этого значения. В MCV хранятся самые частые значения и их доли — для них оценка получается точной. Для значений вне MCV планировщик использует n_distinct или остаточную долю. То есть MCV дополняет гистограмму для точных оценок по равенству на «частых» значениях.
</details>

<details>
<summary>Как correlation влияет на выбор между index scan и seq scan для range-условия?</summary>

При **высокой** корреляции (значения в столбце упорядочены на диске) index range scan читает страницы таблицы почти **последовательно** — стоимость random I/O низкая, index scan часто выигрывает. При **низкой** корреляции index range scan даёт много **случайных** обращений к страницам таблицы — стоимость высокая, планировщик может предпочесть seq scan. То есть correlation не меняет оценку числа строк (rows), но меняет **стоимость** index scan и тем самым влияет на выбор плана.
</details>

<details>
<summary>Что делает команда ANALYZE и где хранится результат?</summary>

ANALYZE читает **выборку** строк из таблицы и по ней считает статистику: число строк (оценка), n_distinct по столбцам, гистограммы, MCV, доли NULL. Результат записывается в системные каталоги (в PostgreSQL — в pg_statistic; человеку удобнее смотреть pg_stats). Планировщик при построении плана использует эту статистику для оценки rows и cost. Без свежего ANALYZE оценки устаревают и планы становятся неоптимальными.
</details>

<details>
<summary>Чем равночастотная гистограмма отличается от равноширинной? Зачем нужна равночастотная?</summary>

**Равноширинная:** корзины делят **диапазон значений** на равные отрезки (например, 1–100, 100–200, …). В каждой корзине может быть разное число строк. **Равночастотная:** корзины подобраны так, чтобы в **каждую корзину попадало примерно одинаковое число строк**; диапазоны значений в корзинах при этом могут быть разной ширины. Равночастотная нужна, чтобы оценка селективности была точнее при «неровном» распределении: например, много дешёвых товаров и мало дорогих — тогда узкий диапазон цен попадёт в одну корзину с большим числом строк, и условие «цена < 100» даст более точную долю.
</details>

<details>
<summary>Зачем нужна расширенная статистика (CREATE STATISTICS)? Когда её создавать?</summary>

Обычная статистика — по каждому столбцу отдельно; планировщик предполагает **независимость** столбцов. Если столбцы **коррелированы** (например, city и region), реальная селективность `city = 'X' AND region = 'Y'` может быть выше, чем произведение долей. **CREATE STATISTICS** хранит информацию о сочетаниях столбцов и улучшает оценки. **Когда создавать:** в плане rows для условия по нескольким столбцам сильно расходится с actual rows; или JOIN/GROUP BY по нескольким столбцам даёт неверную оценку. После создания — **ANALYZE** по таблице.
</details>

<details>
<summary>Что такое autovacuum и auto-analyze? Почему после массовой загрузки лучше явный ANALYZE?</summary>

**Autovacuum** — фоновый процесс PostgreSQL, запускающий VACUUM и ANALYZE. **Auto-analyze** срабатывает, когда накопилось достаточно изменений (порог: autovacuum_analyze_threshold + scale_factor × число строк). **Проблема:** после массовой загрузки 500 000 строк autovacuum может ещё не сработать — статистика устарела, планы плохие. **Решение:** явно запускать **ANALYZE** после больших загрузок; не полагаться только на autovacuum.
</details>

<details>
<summary>Как по плану понять, что статистика устарела, и какие первые два шага по исправлению?</summary>

В плане после EXPLAIN ANALYZE смотри узлы, где **rows** сильно отличается от **actual rows** (например, rows=100, actual rows=20000). Это признак устаревшей или неточной статистики. **Первые шаги:** (1) Выполнить **ANALYZE** по затронутым таблицам. (2) Снова снять план; если расхождение осталось — увеличить **statistics target** по столбцам (`ALTER TABLE ... SET STATISTICS N`) и снова ANALYZE.
</details>

<details>
<summary>Для чего нужен statistics target и где его задать в PostgreSQL?</summary>

**statistics target** задаёт число корзин гистограммы и размер MCV (сколько самых частых значений хранить). Больше target — точнее оценки по «сложным» столбцам (неравномерное распределение, корреляция). В PostgreSQL задаётся по столбцу: `ALTER TABLE table_name ALTER COLUMN column_name SET STATISTICS 500;` (по умолчанию 100). После изменения нужен **ANALYZE** по таблице.
</details>

---

### Запомните

- **Селективность** оценивается по статистике: гистограммы (диапазоны), MCV (частые значения), n_distinct, correlation.
- Неверная или устаревшая статистика → неверные rows → плохой план; регулярный **ANALYZE** и при необходимости увеличение **statistics target** по столбцам улучшают планы.
- **Correlation** влияет на стоимость index scan при range-условиях (высокая корреляция — дешевле последовательное чтение по индексу).

**Повтор §208 в трёх пунктах.** (1) Планировщик оценивает «сколько строк пройдёт фильтр» по **статистике** (гистограммы — для условий по диапазону, MCV — для равенства на частых значениях). (2) Если статистика **устарела** или неточна, оценки rows неверны → планировщик выбирает плохой план (не тот индекс, не тот JOIN). (3) Решение: **ANALYZE** по таблицам после изменений данных; для «сложных» столбцов — **ALTER TABLE ... SET STATISTICS N** и снова ANALYZE.

**Сводка §208 в одну фразу.** Планировщик оценивает «сколько строк пройдёт фильтр» по статистике (гистограммы для диапазонов, MCV для равенства, correlation для стоимости index scan); неверная или устаревшая статистика даёт плохой план — обновляй ANALYZE и при необходимости увеличивай statistics target по столбцам.
