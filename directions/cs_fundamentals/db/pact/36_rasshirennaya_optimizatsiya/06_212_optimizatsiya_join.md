## 212. Оптимизация JOIN

### В этом разделе главное

- **Порядок таблиц** в JOIN влияет на план только в той мере, в какой планировщик перебирает варианты: он оценивает разные порядки и выбирает самый дешёвый по cost; явно задать порядок в SQL (какая таблица «внешняя», какая «внутренняя») в стандартном виде нельзя — это решает оптимизатор.
- **Nested Loop Join**: для каждой строки внешней таблицы поиск совпадений во внутренней; выгоден при **малой** внешней таблице и быстром доступе во внутренней (например, по индексу). Может быть очень плох при больших объёмах без индекса.
- **Hash Join**: строится хеш-таблица по одной таблице (обычно меньшей), вторая таблица прогоняется через хеш; выгоден при **больших** таблицах без естественного порядка; требует памяти под хеш.
- **Merge Join**: соединение двух **уже отсортированных** потоков «двумя указателями»; выгоден когда данные уже отсортированы или сортировка дешёвая (например, по индексу). Не требует хеш-памяти, но требует порядка.
- **Выбор оптимизатором** зависит от оценок rows, стоимости доступа, наличия индексов и порядка; на выбор можно влиять статистикой и отключением/включением типов join (enable_nestloop, enable_hashjoin, enable_mergejoin в PostgreSQL).

---

### Простыми словами

**JOIN** — это способ объединить строки двух (или более) таблиц по условию. Как именно объединять — решает планировщик. Он может выбрать один из трёх основных алгоритмов:

1. **Nested Loop**: «для каждой строки первой таблицы найди подходящие строки во второй». Если первая таблица маленькая, а по второй есть индекс по ключу соединения — это быстро. Если обе большие и индекс слабый — очень медленно.

2. **Hash Join**: «построй по первой таблице хеш-таблицу по ключу соединения, потом для каждой строки второй таблицы посмотри в хеш». Хорош когда таблицы большие и помещаются в память (или частично); порядок данных не важен.

3. **Merge Join**: «обе таблицы отсортируй по ключу соединения и пройдись двумя указателями параллельно». Хорош когда данные уже отсортированы (индекс даёт порядок) или сортировка недорогая.

Порядок таблиц в твоём запросе (`A JOIN B` vs `B JOIN A`) в реляционной модели не определяет порядок выполнения — оптимизатор сам перебирает варианты (A потом B, B потом A и т.д.) и выбирает план с минимальной стоимостью. Ты можешь повлиять на выбор: обновить статистику, добавить индекс по ключу соединения или по столбцам фильтра, или (в PostgreSQL) временно отключить нежелательный тип join через enable_*.

**Ещё проще: три типа JOIN в одной картинке.** (1) **Nested Loop** = «внешний цикл по строкам одной таблицы, внутренний — поиск по второй»; как два вложенных цикла в программе. (2) **Hash Join** = «построить по одной таблице хеш-таблицу в памяти, потом пройтись по второй и для каждой строки смотреть в хеш»; как словарь: один раз составили, потом быстро ищем. (3) **Merge Join** = «два отсортированных списка, два указателя идут параллельно, сравниваем и выдаём совпадения»; как слияние двух списков при сортировке слиянием. Запомни: цикл → Nested Loop; словарь в памяти → Hash; два указателя по отсортированным спискам → Merge.

**Краткая шпаргалка: когда какой JOIN выгоден (для запоминания).**

| Тип JOIN        | Когда выгоден | Когда плох |
|-----------------|---------------|------------|
| **Nested Loop** | Мало строк с одной стороны (например, 10–100) и быстрый поиск по второй (индекс по ключу JOIN). | Много строк с обеих сторон и нет индекса по внутренней таблице — получается «внешняя × внутренняя» обращений. |
| **Hash Join**   | Большие таблицы, порядок не важен; хеш-таблица помещается в память (work_mem). | Хеш не помещается в work_mem — часть на диск, план тормозит. |
| **Merge Join**  | Оба потока уже отсортированы по ключу соединения (индекс даёт порядок) или сортировка недорогая. | Данные вразброс и сортировка дорогая — планировщик предпочтёт Hash. |

**Когда в плане обращать внимание на тип JOIN.** (1) **Nested Loop** с **Seq Scan** по внутренней таблице и большим **loops** (тысячи и больше) — типичное узкое место: на каждую строку внешней таблицы читается вся внутренняя; рецепт: добавить индекс по ключу соединения по внутренней таблице или добиться выбора Hash Join (обновить статистику). (2) **Hash Join** и в выводе плана есть **Disk** (например, «Buckets: …, Batches: …, Disk Usage: …») — хеш не поместился в work_mem, часть на диск, план тормозит; рецепт: увеличить work_mem для сессии или оптимизировать запрос (уменьшить объём соединения). (3) Запрос с несколькими JOIN сильно тормозит — посмотри **actual time** по каждому узлу JOIN: какой узел даёт наибольшее время, тот и кандидат (часто первый «плохой» Nested Loop или Hash с Disk). Порядок таблиц в твоём SQL менять не обязательно — планировщик сам перебирает; важнее статистика и индексы.

### Теория и правила

#### Порядок таблиц

В SQL порядок таблиц в FROM (A JOIN B JOIN C) не гарантирует порядок выполнения. Планировщик рассматривает перестановки и выбирает порядок с наименьшей оценкой стоимости. Поэтому «поменять порядок таблиц в запросе» редко даёт стабильный эффект — оптимизатор может снова выбрать тот же план. Надёжнее влиять через статистику, индексы и при необходимости через hints (MySQL) или pg_hint_plan / enable_* (PostgreSQL).

#### Nested Loop Join

- **Условие выгодности**: маленькая внешняя таблица (или мало строк после фильтра), быстрый поиск во внутренней (индекс по ключу соединения).
- **Риск**: большая внешняя таблица × полное сканирование внутренней = очень много обращений и медленный план.
- В плане: **Nested Loop** с **Index Scan** (или Seq Scan) по внутренней таблице.

**Кто такая «внешняя» и «внутренняя» таблица.** В Nested Loop одна таблица обрабатывается **внешним** циклом (её строки перебираются по одной), вторая — **внутренним** (для каждой строки внешней ищутся совпадения во второй). «Внешняя» = та, по которой идёт цикл; «внутренняя» = та, по которой на каждой итерации делается поиск. В плане обычно сначала читается внешняя таблица (или её часть), затем для каждой её строки выполняется доступ к внутренней. Для Nested Loop выгодно, когда **внешняя** маленькая (мало итераций), а по **внутренней** есть быстрый поиск по ключу соединения (индекс), а не полное сканирование.

**Числовой пример, когда Nested Loop плох.** Внешняя таблица даёт 10 000 строк, внутренняя — 100 000 строк, по внутренней **нет индекса** по ключу соединения, поэтому на каждой из 10 000 итераций выполняется полное сканирование внутренней таблицы (100 000 строк). Итого порядка 10 000 × 100 000 = миллиард обращений к строкам — очень медленно. В таком случае Hash Join (построить хеш по одной таблице, один раз прочитать вторую) был бы намного быстрее. Поэтому в плане с Nested Loop всегда смотри: по внутренней таблице — Index Scan или Seq Scan? Если Seq Scan по большой таблице — типичная причина медленного плана.

**Как в плане отличить внешнюю и внутреннюю таблицу в Nested Loop.** В выводе EXPLAIN дочерние узлы Nested Loop обычно показываются с отступом. **Внешняя** таблица — тот скан (или подплан), который выполняется **один раз** и даёт поток строк для цикла; в плане у него часто **loops=1** (или он верхний из двух дочерних узлов по отображению). **Внутренняя** — тот скан, который выполняется **для каждой строки** внешней; у него в плане **loops** равно числу строк внешней таблицы (или видно, что узел вызывается многократно). В текстовом формате плана PostgreSQL порядок дочерних узлов: сначала обычно перечисляется «внешняя» сторона, затем «внутренняя». Смотри на **loops** в (actual time ... loops=...): у внутреннего узла loops будет большим (равным числу строк внешней таблицы).

#### Hash Join

- Одна из таблиц (часто меньшая) читается и строится **в памяти хеш-таблица** по ключу соединения; вторая таблица сканируется, для каждой строки делается поиск в хеше.
- **Условие выгодности**: большие таблицы, ключ соединения подходит для хеширования; важно наличие памяти (work_mem в PostgreSQL).
- **Ограничение**: если хеш не помещается в work_mem, часть уходит на диск (файлы в temp) — план замедляется.

**Почему хеш обычно строят по меньшей таблице.** Хеш-таблица должна **поместиться в память** (work_mem), иначе часть уходит на диск и план сильно замедляется. Поэтому планировщик старается строить хеш по **меньшей** из двух таблиц (по числу строк после фильтров), а **большую** прогонять по хешу: так хеш получается компактнее и с большей вероятностью помещается в work_mem. Если бы строили хеш по большой таблице, шанс «переполнения» work_mem и выгрузки на диск был бы выше. Итог: в плане Hash Join «внутренняя» (по которой строится хеш) — обычно меньшая таблица.

**Ещё проще: зачем work_mem в Hash Join и почему «Disk» — плохо.** Hash Join строит хеш-таблицу **в памяти** (лимит — work_mem, по умолчанию 4 МБ). Если хеш не влезает — часть уходит во **временные файлы на диске**. Поиск в хеше в памяти — очень быстро (O(1)); поиск на диске — медленно (чтение с диска в тысячи раз медленнее памяти). Поэтому в плане, когда видишь **Disk Usage** или **Batches** больше 1 — часть хеша на диске, план тормозит. Решение: увеличить **work_mem** для сессии, чтобы хеш поместился в память целиком.

**Sort и work_mem (аналогично Hash Join).** Узел **Sort** (ORDER BY, GROUP BY с сортировкой, Merge Join с предварительной сортировкой) тоже использует **work_mem**. Если данные для сортировки не помещаются в work_mem, Sort переключается на **внешнюю сортировку** (external merge): часть данных во временных файлах на диске, затем слияние. В плане после EXPLAIN (ANALYZE) это видно как **Sort Method: external merge** и **Disk: ...** (или аналогичные пометки). **Проблема:** сортировка на диске в разы медленнее, чем в памяти. **Решение:** увеличить **work_mem** для сессии или запроса. **Когда смотреть:** запрос с ORDER BY или GROUP BY тормозит — сними план, ищи узел **Sort**; если в выводе **external merge** или **Disk** — work_mem переполнена, увеличивай. Итог: **work_mem** ограничивает и **Hash Join**, и **Sort**; оба при переполнении уходят на диск и тормозят.

#### Merge Join

- Оба входа должны быть **отсортированы по ключу соединения**. Два указателя идут по обоим потокам, совпадения выдаются по порядку.
- **Условие выгодности**: данные уже отсортированы (индекс по ключу соединения) или сортировка недорогая; нет необходимости в хеш-памяти.
- В плане часто видно **Sort** перед Merge Join, если порядок не даёт индекс.

**Merge Join «два указателя» — простыми словами.** Представь два отсортированных списка по одному и тому же ключу (например, id): список A = [1, 3, 5, 7], список B = [2, 3, 5, 6]. Указатель по A стоит на первом элементе, указатель по B — на первом. Сравниваем: 1 < 2 — сдвигаем указатель по A. 3 > 2 — сдвигаем по B. 3 = 3 — совпадение, выдаём пару, сдвигаем оба. 5 = 5 — совпадение, выдаём, сдвигаем. И так далее, пока не дойдём до конца одного из списков. Каждый список проходится **один раз**, лишней памяти под хеш не нужно — нужен только порядок. Поэтому Merge Join выгоден, когда оба потока уже отсортированы (индекс даёт порядок) или когда сортировка не слишком дорогая.

#### Выбор оптимизатором

Планировщик оценивает cost для разных комбинаций: тип join (Nested Loop, Hash, Merge), порядок таблиц, способ доступа к каждой таблице (index vs seq scan). Выбирается план с минимальным total cost. Точность оценок rows и стоимости доступа (зависит от статистики и correlation) напрямую влияет на этот выбор.

### Примеры планов (понимание вывода)

После `EXPLAIN (ANALYZE)` смотри на тип узла соединения:

- **Nested Loop** — ожидай малую внешнюю сторону и индекс по внутренней; если inner — Seq Scan на большой таблице, часто плохо.
- **Hash Join** — смотри на размер хеш-таблицы и work_mem; если «Disk: ...» в плане — часть хеша на диске, можно увеличить work_mem.
- **Merge Join** — смотри, есть ли Sort перед ним; если оба входа уже отсортированы по индексу — план эффективен.

### Проверь себя

<details>
<summary>В каких случаях Nested Loop Join обычно выгоден, а в каких — нет?</summary>

**Выгоден**: когда **внешняя** таблица маленькая (мало строк после фильтра) и по **внутренней** есть эффективный доступ по ключу соединения (индекс). Тогда для каждой из немногих внешних строк делается быстрый поиск во внутренней. **Не выгоден**: когда внешняя таблица большая, а внутренняя сканируется полностью (Seq Scan) — получается «внешняя × внутренняя» обращений к данным, что очень дорого.
</details>

<details>
<summary>Чем Hash Join отличается от Merge Join по использованию памяти и порядка данных?</summary>

**Hash Join** строит хеш-таблицу в памяти (до work_mem) по одной из таблиц; вторая таблица может приходить в любом порядке. Память нужна под хеш. **Merge Join** не строит хеш; ему нужны **два потока, отсортированных по ключу соединения**. Память в основном под буферы сортировки (если сортировка делается). Merge выгоден, когда порядок уже есть (индексы); Hash — когда порядок не важен и данные большие.
</details>

<details>
<summary>Опиши алгоритм Merge Join «двумя указателями» в двух словах.</summary>

Оба входа **отсортированы** по ключу соединения. Два указателя: один на текущую строку первого потока, другой — второго. Сравниваем ключи: если равны — выдаём пару строк и сдвигаем оба указателя; если ключ в первом меньше — сдвигаем указатель по первому; если во втором меньше — по второму. Так проходим оба потока по одному разу, без хеш-таблицы. Нужен только порядок по ключу.
</details>

<details>
<summary>В плане Hash Join, запрос тормозит. На что смотреть и что делать?</summary>

Смотри, есть ли в выводе плана **Disk** (Disk Usage, Batches > 1) — это значит, часть хеш-таблицы не поместилась в **work_mem** и ушла на диск. Поиск на диске в тысячи раз медленнее, чем в памяти. **Что делать:** увеличить work_mem для сессии: `SET work_mem = '256MB';` (или другое значение, в зависимости от размера данных). Выполни запрос снова и проверь план: если Disk исчез — хеш поместился в память, запрос должен ускориться. Альтернатива — уменьшить объём соединения (более селективные фильтры, чтобы меньше строк участвовало в JOIN).
</details>

<details>
<summary>Запрос с ORDER BY или GROUP BY тормозит. В плане узел Sort с «external merge» или «Disk». Что это и что делать?</summary>

**Sort** тоже использует **work_mem**. Если данные для сортировки не помещаются — Sort переключается на **внешнюю сортировку** (external merge): часть данных во временных файлах на диске. В плане видно **Sort Method: external merge** или **Disk: ...**. **Что делать:** увеличить **work_mem** для сессии (аналогично Hash Join). Итог: work_mem ограничивает и Hash Join, и Sort; оба при переполнении уходят на диск.
</details>

<details>
<summary>Почему порядок таблиц в SQL (A JOIN B vs B JOIN A) не фиксирует порядок выполнения? Как тогда влиять на план?</summary>

Оптимизатор **сам перебирает** варианты порядка соединения и выбирает план с наименьшим cost. Порядок в запросе — только подсказка для читаемости. **Влиять:** обновить статистику (ANALYZE, statistics target), добавить или изменить индексы по ключу JOIN; в MySQL при необходимости — STRAIGHT_JOIN; в PostgreSQL — enable_* в сессии или pg_hint_plan.
</details>

<details>
<summary>Когда планировщик обычно выбирает Nested Loop, а когда Hash Join? От чего это зависит?</summary>

Выбор зависит от **оценок rows** и **cost**. **Nested Loop** выбирается, когда внешняя таблица даёт мало строк и по внутренней есть эффективный доступ (индекс по ключу JOIN) — cost вложенного цикла с индексным поиском оказывается ниже. **Hash Join** выбирается при больших объёмах с обеих сторон и когда порядок не важен — построение хеша и прогон по нему дешевле, чем Nested Loop с большим числом обращений. Точные оценки строятся по статистике (rows); неверная статистика может привести к неоптимальному выбору.
</details>

### Запомните

- **Nested Loop** — малая внешняя таблица + быстрый доступ во внутренней (индекс). **Hash Join** — большие таблицы, хеш в памяти. **Merge Join** — отсортированные по ключу соединения потоки. **Sort** — тоже использует work_mem; при переполнении — на диск.
- Порядок таблиц в запросе не задаёт порядок выполнения — его выбирает оптимизатор; влияем через статистику, индексы и при необходимости enable_* / hints.

### Как запомнить (§212)

- **Nested Loop** — «для каждой строки одной таблицы ищем во второй»; выгоден при **малой** внешней и **индексе** по внутренней.
- **Hash Join** — строим хеш по одной таблице, прогоняем вторую; выгоден при **больших** таблицах и достаточной **work_mem**.
- **Merge Join** — два **отсортированных** потока, проход «двумя указателями»; выгоден, когда порядок уже есть (индекс) или сортировка дёшево.
- Порядок таблиц в SQL (**A JOIN B** vs **B JOIN A**) **не фиксирует** порядок выполнения — оптимизатор перебирает варианты сам.

---

### Типичные ошибки (§212)

- **Думать, что «поменял порядок таблиц в запросе — план изменился».** Часто оптимизатор выберет тот же план; надёжнее влиять статистикой и индексами.
- **Игнорировать «Disk» в плане Hash Join.** Если в выводе EXPLAIN видно, что часть хеша ушла на диск — увеличить **work_mem** для этого сеанса или запроса.

**Часто путают (§212): когда какой JOIN.**

| Ситуация | Что часто путают | Как правильно |
|----------|------------------|----------------|
| Малая таблица + индекс по ключу JOIN | Думают, что «большая таблица должна быть первой» | Для **Nested Loop** выгодна **малая внешняя** и **индекс по внутренней**; кто «первый» в SQL — решает оптимизатор. |
| Большие таблицы, порядок не важен | Думают, что нужен Merge Join | **Hash Join** типичен для больших таблиц без порядка; Merge — когда оба потока уже отсортированы по ключу. |
| В плане Hash Join, запрос тормозит | Не смотрят на «Disk» | Если часть хеша на диске — увеличить **work_mem**; иначе Hash Join может быть медленным. |

**Повтор §212 в трёх пунктах.** (1) **Nested Loop**: для каждой строки одной таблицы ищем совпадения во второй; хорошо, когда первая таблица маленькая и по второй есть индекс по ключу JOIN; плохо, когда обе большие и по второй нет индекса (получается «внешняя × внутренняя» обращений). (2) **Hash Join**: строим хеш по одной таблице в памяти, прогоняем вторую через хеш; хорошо при больших таблицах; если в плане есть **Disk** — хеш не влез в work_mem, увеличивай work_mem. (3) **Merge Join**: оба потока отсортированы по ключу, идём «двумя указателями»; порядок таблиц в SQL решает оптимизатор — влияй статистикой и индексами.

**Сводка §212 в одну фразу.** Nested Loop выгоден при малой внешней таблице и индексе по внутренней; Hash Join — при больших таблицах и достаточной work_mem; Merge Join — при отсортированных по ключу соединения потоках; порядок таблиц в SQL выбирает оптимизатор, влияй статистикой и индексами.
