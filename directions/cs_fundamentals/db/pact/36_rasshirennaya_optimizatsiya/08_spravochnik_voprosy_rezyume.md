## Типичные ошибки по части и резюме

- **Не смотреть план** при медленном запросе — гадать по коду вместо EXPLAIN (ANALYZE) и сравнения rows с actual rows.
- **Игнорировать статистику**: не запускать ANALYZE после массовых изменений или не увеличивать statistics target для «сложных» столбцов — планы становятся неоптимальными.
- **Путать cost и actual time**: cost — для сравнения планов, actual time — реальное время; оптимизировать по факту (actual time и actual rows).
- **Считать, что порядок таблиц в JOIN задаёт порядок выполнения** — порядок выбирает оптимизатор; влиять через индексы и статистику.
- **Оставлять коррелированные подзапросы «как есть»** при плохом плане — не переписывать в JOIN и удивляться медленному Subplan.
- **Злоупотреблять hints (MySQL) или enable_* (PostgreSQL)** вместо исправления статистики и запросов — маскировка причины и хрупкость при обновлениях.

Главная мысль: **оптимизация опирается на чтение планов, точную статистику и осознанный выбор алгоритмов (JOIN, подзапросы)**. Понимание cost/rows/actual time, статистики, параллелизма, JIT и типов JOIN позволяет целенаправленно ускорять запросы и исправлять плохие планы.

### Проверь себя (типичные ошибки по части)

<details>
<summary>Почему «не смотреть план» и «путать cost и actual time» — типичные ошибки?</summary>

Без плана оптимизация превращается в гадание: непонятно, где узкое место (статистика, подзапрос, тип JOIN). **Cost** — условная единица для сравнения планов до выполнения; **actual time** — реальное время в мс после EXPLAIN ANALYZE. Оптимизировать нужно по факту (actual time и actual rows), а не по cost; путаница ведёт к неверным выводам.
</details>

<details>
<summary>Чем плохо «злоупотреблять hints вместо исправления статистики и запросов»?</summary>

Hints маскируют причину плохого плана: статистика остаётся устаревшей, запрос — неоптимальным. При изменении данных или обновлении СУБД подсказка может стать неверной и план — ещё хуже. Долгосрочно надёжнее: ANALYZE, statistics target, переписать коррелированный подзапрос в JOIN, добавить индекс.
</details>

<details>
<summary>Что общего у ошибок «оставить коррелированный подзапрос» и «забыть ANALYZE после загрузки»?</summary>

Обе ведут к **неоптимальному плану**: в первом случае — SubPlan с большим loops (подзапрос в цикле), во втором — неверные rows из-за устаревшей статистики. Исправление целенаправленное: переписать в JOIN и выполнить ANALYZE соответственно. Без чтения плана (actual time, rows vs actual rows) эти ошибки легко не заметить.
</details>

## Резюме части в одном абзаце

**Планы (§207).** Cost — условная единица для сравнения планов; rows — оценка числа строк; actual time/actual rows появляются после EXPLAIN ANALYZE. План читать снизу вверх; Initplan выполняется один раз, Subplan — многократно (часто узкое место). EXPLAIN (BUFFERS) — статистика I/O (Shared Hit/Read). Index Only Scan — скан только по индексу при covering index и visibility map. **Статистика (§208).** Гистограммы и MCV задают селективность; correlation влияет на стоимость index scan. Extended Statistics (CREATE STATISTICS) — для коррелированных столбцов. Autovacuum/auto-analyze — автоматический ANALYZE; после больших загрузок лучше явный ANALYZE. **Параллелизм (§209).** Gather собирает результаты workers; parallel seq scan/hash join/aggregate ускоряют тяжёлые запросы; есть ограничения и overhead. **JIT (§210).** Компиляция выражений и агрегатов по порогу jit_above_cost; выгоден на тяжёлых запросах, есть overhead компиляции. **Hints (§211).** В MySQL — USE INDEX/FORCE INDEX, STRAIGHT_JOIN (порядок JOIN); в PostgreSQL встроенных нет — статистика, enable_*, pg_hint_plan, переписывание запроса. **JOIN (§212).** Nested Loop для малых объёмов и индекса по внутренней; Hash для больших без порядка; Merge для отсортированных потоков; порядок таблиц выбирает оптимизатор. Sort, как и Hash Join, использует work_mem; при переполнении — на диск. **Подзапросы (§213).** Развёртывание в JOIN, материализация (когда выгодна/вредна); CTE (WITH) в PostgreSQL — оптимизационный барьер, материализуется; при тормозах попробовать без WITH. Коррелированные подзапросы переписывать в JOIN; EXISTS vs IN — смотреть план и при NOT IN остерегаться NULL.

### Проверь себя (резюме части)

<details>
<summary>Что такое Initplan и Subplan и почему в резюме сказано «Subplan — многократно (часто узкое место)»?</summary>

**Initplan** выполняется один раз, результат кэшируется. **Subplan** может выполняться многократно (например, для каждой строки внешнего запроса при коррелированном подзапросе). Узкое место: при 50 000 строк внешней таблицы подзапрос выполнится 50 000 раз — огромные накладные расходы; поэтому переписывают в JOIN.
</details>

<details>
<summary>В резюме упомянуты work_mem и CTE. С чем они связаны и что делать при тормозах?</summary>

**work_mem** — лимит памяти для Hash Join и Sort. При переполнении часть хеша или сортировки уходит на диск (в плане Disk) — запрос сильно тормозит. **Что делать:** увеличить work_mem для сессии. **CTE (WITH)** в PostgreSQL материализуется (оптимизационный барьер). Если запрос с WITH тормозит — попробовать переписать без WITH (подзапрос в FROM) и сравнить план.
</details>

## Частые сценарии (задача → что делать)

| Задача | Что делать |
|--------|------------|
| Запрос медленный, не знаю с чего начать | Снять **EXPLAIN (ANALYZE)**; читать план **снизу вверх**; смотреть **actual time** по узлам — какой узел съедает больше всего времени; сравнить **rows** и **actual rows** — большое расхождение говорит о проблеме со статистикой. |
| В плане rows сильно отличается от actual rows | Обновить статистику: **ANALYZE** по таблице (или по БД); для «сложных» столбцов увеличить **statistics target** и снова ANALYZE. |
| В плане вижу SubPlan с большим числом loops | Скорее всего **коррелированный подзапрос**; переписать в **JOIN** по ключу корреляции (столбец, по которому подзапрос ссылается на внешнюю таблицу). |
| Хочу заставить использовать индекс (PostgreSQL) | Встроенных hints нет. Варианты: обновить **ANALYZE** и при необходимости **statistics target**; временно **SET enable_seqscan = off** в сессии; переписать запрос; использовать расширение **pg_hint_plan**. |
| Хочу заставить использовать индекс (MySQL) | В запросе указать **USE INDEX (имя_индекса)** или **FORCE INDEX (имя_индекса)** после имени таблицы в FROM/JOIN. |
| План выбрал Seq Scan, а я думаю индекс был бы лучше | Часто причина — **неверная статистика** (планировщик завысил оценку строк по индексу). Сделать ANALYZE; проверить correlation по столбцу; при необходимости увеличить statistics target по столбцу индекса. |
| Hash Join тормозит, в плане «Disk» | Часть хеш-таблицы ушла на диск. Увеличить **work_mem** для сессии или для конкретного запроса (SET work_mem = '256MB';). |
| **Sort** тормозит, в плане «external merge» или «Disk» | Sort тоже использует work_mem; при переполнении — на диск. Увеличить **work_mem**. |
| Запрос с **WITH (CTE)** тормозит | CTE в PostgreSQL материализуется; попробуй переписать без WITH (подзапрос в FROM) и сравнить план. |
| Нужно понять, выгоден ли параллелизм / JIT | Снять план с EXPLAIN (ANALYZE); искать узел **Gather** (параллелизм) или пометки JIT в выводе. Для сравнения выполнить тот же запрос с отключённым параллелизмом (max_parallel_workers_per_gather = 0) или JIT (jit = off) и сравнить Execution time. |

### Проверь себя (частые сценарии)

<details>
<summary>Задача: «В плане rows сильно отличается от actual rows». Что делать и почему это важно?</summary>

Обновить статистику: **ANALYZE** по таблице (или по БД). Для «сложных» столбцов можно увеличить **statistics target** (`ALTER TABLE ... SET STATISTICS N`) и снова ANALYZE. Важно: планировщик выбирает план по оценкам rows; при неверной статистике план оказывается плохим (не тот индекс, не тот JOIN). После ANALYZE оценки сближаются с реальностью.
</details>

<details>
<summary>Задача: «Запрос с WITH (CTE) тормозит». Какой сценарий из таблицы применить и в чём причина?</summary>

Попробовать переписать **без WITH** (подзапрос в FROM) и сравнить план. Причина: в PostgreSQL CTE по умолчанию материализуется (оптимизационный барьер). Если CTE лёгкая и используется один раз, материализация добавляет overhead; без WITH планировщик может встроить подзапрос и выбрать лучший план.
</details>

<details>
<summary>В плане Hash Join, запрос тормозит. Какой сценарий из таблицы «Частые сценарии» применить?</summary>

Искать в плане **Disk** (Disk Usage, Batches > 1). Если есть — часть хеш-таблицы не поместилась в work_mem и ушла на диск. **Что делать:** увеличить **work_mem** для сессии (или для запроса). После этого снова снять план; если Disk исчез — хеш в памяти, запрос обычно ускоряется.
</details>

## Частые вопросы «Почему?» (для понимания, а не только запоминания)

| Вопрос | Короткий ответ |
|--------|----------------|
| **Почему cost — не секунды?** | Планировщик выбирает план **до** выполнения; он не может замерить реальное время. Cost — условные единицы для **сравнения** планов между собой. На разных дисках одно и то же число cost даст разное время. |
| **Почему порядок таблиц в SQL (A JOIN B vs B JOIN A) не фиксирует план?** | Оптимизатор **сам перебирает** варианты (A потом B, B потом A и т.д.) и выбирает порядок с наименьшим cost. Твой порядок в запросе — только подсказка для чтения; оптимизатор может выполнить в любом порядке. |
| **Почему SubPlan с большим loops — плохо?** | Подзапрос выполняется **столько раз, сколько строк** во внешней таблице. 100 000 строк → 100 000 выполнений подзапроса. Каждое — обращение к БД. Один JOIN читает каждую таблицу **один раз** — в разы быстрее. |
| **Почему NOT IN с NULL даёт пустой результат?** | `id NOT IN (1, 2, NULL)` по правилам SQL = «id ≠ 1 AND id ≠ 2 AND id ≠ NULL». Сравнение с NULL даёт **unknown**, не true. Вся цепочка AND с unknown не становится true → строка не попадает в результат. |
| **Почему на маленькой таблице параллелизм может быть медленнее?** | Overhead: создание workers, передача данных, координация. Если сама работа быстрая (10 мс), а overhead — 20 мс, итого 30 мс — медленнее, чем один процесс (10 мс). |
| **Почему хеш строят по меньшей таблице?** | Хеш должен **поместиться в память** (work_mem). Меньшая таблица → меньший хеш → больше шанс, что влезет. Если строить по большой — часть уйдёт на диск, план сильно замедлится. |
| **Почему после массовой загрузки нужен ANALYZE?** | Статистика собрана по **старой** выборке строк. После загрузки число строк и распределение значений изменились. Планировщик опирается на устаревшие цифры → неверные rows → плохой план. |

### Проверь себя (частые вопросы «Почему?»)

<details>
<summary>Почему cost — не секунды и зачем тогда планировщику cost?</summary>

Планировщик выбирает план **до** выполнения запроса и не может замерить реальное время. Cost — условные единицы для **сравнения** альтернативных планов на одной системе (один план дешевле другого). На разных дисках одно и то же cost даст разное время; важна только относительная величина.
</details>

<details>
<summary>Почему хеш в Hash Join строят по меньшей таблице?</summary>

Хеш-таблица должна **поместиться в память** (work_mem). Меньшая таблица → меньший хеш → больше шанс, что влезет в work_mem. Если строить по большой таблице, часть хеша уйдёт на диск (в плане Disk), и план сильно замедлится.
</details>

### Проверь себя (вопросы для самопроверки по части)

<details>
<summary>Что такое Initplan и Subplan? Почему Subplan в цикле по многим строкам часто медленный?</summary>

**Initplan** — подзапрос выполняется один раз, результат кэшируется. **Subplan** — подзапрос может выполняться многократно (для каждой строки внешнего запроса при корреляции). Subplan в цикле медленный: число выполнений = число строк внешней таблицы (например, 80 000 раз); каждое выполнение — обращение к БД; один JOIN читает таблицы по разу — в разы быстрее.
</details>

<details>
<summary>В каких случаях выгодны Nested Loop, Hash Join и Merge Join? Как порядок таблиц в запросе связан с порядком выполнения?</summary>

**Nested Loop** — малая внешняя таблица и индекс по ключу JOIN по внутренней. **Hash Join** — большие таблицы, порядок не важен, хеш в памяти (work_mem). **Merge Join** — оба потока уже отсортированы по ключу соединения. Порядок таблиц в SQL (A JOIN B vs B JOIN A) **не задаёт** порядок выполнения — его выбирает оптимизатор; влияем статистикой и индексами.
</details>

<details>
<summary>Когда JIT ускоряет запрос, а когда может замедлить?</summary>

**Ускоряет:** тяжёлый запрос (cost выше jit_above_cost) с большим числом строк и сложными выражениями или агрегатами — компиляция окупается. **Замедляет:** лёгкий запрос (выполняется быстрее, чем длится компиляция) или однократное выполнение, где overhead компиляции не окупается.
</details>

## Если забыл всё — супер-краткий чек-лист (6 пунктов)

1. **Запрос медленный** → `EXPLAIN (ANALYZE) ...`; читай план **снизу вверх**; смотри **actual time** — какой узел даёт самое большое время.
2. **rows и actual rows сильно расходятся** → `ANALYZE` по таблицам; при необходимости `ALTER TABLE ... SET STATISTICS 500` и снова ANALYZE.
3. **SubPlan с большим loops** → подзапрос в цикле; переписывай в **JOIN** по ключу связи (столбец, на который ссылается подзапрос).
4. **Nested Loop + Seq Scan по большой внутренней** → добавить индекс по ключу JOIN по внутренней таблице или обновить статистику (ANALYZE), чтобы планировщик выбрал Hash Join.
5. **Hash Join или Sort и в плане Disk** → увеличить **work_mem** для сессии.
6. **Запрос с WITH (CTE) тормозит** → попробуй переписать без WITH (подзапрос в FROM) и сравнить план.

Если всё ещё непонятно — возвращайся к «Что делать по шагам, когда запрос медленный» в начале части и проходи по шагам 1–7.

### Проверь себя (супер-краткий чек-лист)

<details>
<summary>По чек-листу: что делать, если «rows и actual rows сильно расходятся»? И если «SubPlan с большим loops»?</summary>

**rows и actual rows расходятся** → выполнить **ANALYZE** по таблицам; при необходимости `ALTER TABLE ... SET STATISTICS 500` и снова ANALYZE. **SubPlan с большим loops** → подзапрос выполняется в цикле; переписать в **JOIN** по ключу связи (столбец, на который ссылается подзапрос).
</details>

<details>
<summary>По чек-листу: в каких двух ситуациях помогает увеличение work_mem и что в плане на это указывает?</summary>

(1) **Hash Join** тормозит — в плане есть **Disk** (часть хеша на диске). (2) **Sort** тормозит — в плане «external merge» или **Disk**. В обоих случаях увеличить **work_mem** для сессии. work_mem ограничивает память для хеш-таблицы и для сортировки; при переполнении данные уходят на диск и план замедляется.
</details>

<details>
<summary>Первый пункт чек-листа: с чего начать при медленном запросе?</summary>

Выполнить **EXPLAIN (ANALYZE)** для этого запроса. Читать план **снизу вверх**. Смотреть **actual time** по узлам — какой узел даёт самое большое время. Это локализует узкое место; дальше действовать по остальным пунктам чек-листа (статистика, подзапрос в JOIN, индексы, work_mem, WITH).
</details>

## Краткий словарь (повторение терминов в одну строку)

| Термин | Определение в одну строку |
|--------|---------------------------|
| **cost** | Условная «стоимость» операции для сравнения планов; не секунды. |
| **rows** | Оценка планировщика «сколько строк выйдет»; строится по статистике. |
| **actual time** | Реальное время узла в мс (после EXPLAIN ANALYZE). |
| **actual rows** | Реальное число строк на выходе узла (после EXPLAIN ANALYZE). |
| **loops** | Сколько раз узел выполнился; loops=1 — один раз; большой loops — узел в цикле (часто внутренняя сторона Nested Loop). |
| **Execution Time** | Общее время выполнения запроса в мс (внизу плана после EXPLAIN ANALYZE); с ним сравнивают «до» и «после» оптимизации. |
| **Initplan** | Подзапрос, выполнившийся один раз; результат кэширован. |
| **Subplan** | Подзапрос, выполняющийся много раз (в цикле); часто узкое место. |
| **селективность** | Доля строк, проходящих условие; оценка строк = (число строк) × селективность. |
| **гистограмма** | Распределение значений по «корзинам»; для условий по диапазону (<, >, BETWEEN). |
| **MCV** | Список самых частых значений и их доля; для условия равенства (=). |
| **n_distinct** | Оценка числа уникальных значений в столбце; для равенства часто селективность ≈ 1/n_distinct. |
| **correlation** | Связь порядка значений на диске с порядком по величине; влияет на cost index scan. |
| **ANALYZE** | Команда обновления статистики по таблице; планировщик использует её для оценок. |
| **statistics target** | Число корзин гистограммы и размер MCV; больше — точнее оценки. |
| **Gather** | Узел плана, собирающий результаты от parallel workers в один поток. |
| **worker** | Отдельный процесс, выполняющий часть плана параллельно. |
| **JIT** | Компиляция выражений и агрегатов в машинный код во время выполнения. |
| **jit_above_cost** | Порог cost, выше которого включается JIT. |
| **USE INDEX / FORCE INDEX** | Подсказки в MySQL для указания индекса в запросе. |
| **enable_seqscan, enable_*** | Параметры PostgreSQL для включения/отключения типов узлов плана. |
| **Nested Loop** | JOIN «вложенным циклом»: для каждой строки одной таблицы поиск во второй. |
| **Hash Join** | Построение хеш-таблицы по одной таблице, прогон второй через хеш. |
| **Merge Join** | Соединение двух отсортированных потоков «двумя указателями». |
| **work_mem** | Лимит памяти для хеш-таблицы и сортировки; при переполнении — диск. |
| **коррелированный подзапрос** | Подзапрос со ссылкой на внешнюю таблицу; выполняется «для каждой строки». |
| **Semi Join** | «Есть совпадение» без размножения строк; EXISTS, IN. |
| **Anti Join** | «Нет совпадения»; NOT EXISTS (не NOT IN при NULL). |
| **LATERAL** | Подзапрос в FROM, который может ссылаться на столбцы «слева». |
| **BUFFERS** | Опция EXPLAIN: показывает Shared Hit, Shared Read, Temp — для анализа I/O. |
| **Index Only Scan** | Скан только по индексу, без обращения к таблице; нужен covering index и visibility map. |
| **Extended Statistics** | CREATE STATISTICS для коррелированных столбцов; улучшает оценки для условий по нескольким столбцам. |
| **autovacuum / auto-analyze** | Фоновый процесс, автоматически запускающий ANALYZE; после больших загрузок лучше явный ANALYZE. |
| **STRAIGHT_JOIN** | Hint в MySQL: зафиксировать порядок JOIN (слева направо). |
| **CTE (WITH)** | Подзапрос с именем; в PostgreSQL по умолчанию материализуется (оптимизационный барьер). |

### Проверь себя (краткий словарь)

<details>
<summary>Чем отличаются cost и actual time? Что такое rows и actual rows?</summary>

**cost** — условная «стоимость» операции для сравнения планов; не секунды. **actual time** — реальное время узла в мс (после EXPLAIN ANALYZE). **rows** — оценка планировщика «сколько строк выйдет» по статистике. **actual rows** — реальное число строк на выходе узла (после EXPLAIN ANALYZE). Сравнение rows с actual rows показывает точность статистики.
</details>

<details>
<summary>Что такое Initplan и Subplan в одну строку? Чем Semi Join отличается от Anti Join?</summary>

**Initplan** — подзапрос выполнился один раз, результат кэширован. **Subplan** — выполняется много раз (в цикле); часто узкое место. **Semi Join** — «есть совпадение» без размножения строк (EXISTS, IN). **Anti Join** — «нет совпадения» (NOT EXISTS; не NOT IN при NULL).
</details>

<details>
<summary>Что такое work_mem и при каких узлах плана он важен? Что такое correlation?</summary>

**work_mem** — лимит памяти для хеш-таблицы (Hash Join) и сортировки (Sort); при переполнении — диск. **correlation** — связь порядка значений на диске с порядком по величине; влияет на cost index scan при range-условиях (высокая корреляция — последовательное чтение дешевле).
</details>
