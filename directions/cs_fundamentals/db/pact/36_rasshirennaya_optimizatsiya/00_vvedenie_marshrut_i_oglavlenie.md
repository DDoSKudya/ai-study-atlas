## Часть XXXVI. Расширенные темы оптимизации

> Цель этой части — **перестать воспринимать «медленный запрос» как загадку** и начать **читать планы выполнения, понимать, откуда берутся оценки и почему оптимизатор выбирает тот или иной план**, а также **осознанно влиять на производительность** через статистику, параллелизм, JIT, оптимизацию JOIN и подзапросов.

---

После этой части ты сможешь:

- **читать и интерпретировать планы выполнения (EXPLAIN/ANALYZE)**: cost, rows, actual time, типы узлов, поддеревья, Initplan vs Subplan;
- **понимать связь статистики и селективности** с качеством планов: гистограммы, MCV, correlation и когда обновлять статистику;
- **разбираться в параллельном выполнении**: Gather, workers, parallel seq scan/hash join/aggregate и их ограничения;
- **понимать, когда и как JIT-компиляция ускоряет запросы** и как её настраивать;
- **знать про hints в MySQL (USE INDEX, FORCE INDEX)** и альтернативы в PostgreSQL;
- **осознанно подходить к оптимизации JOIN**: порядок таблиц, Nested Loop vs Hash vs Merge join и выбор оптимизатора;
- **оптимизировать подзапросы**: развёртывание в JOIN, материализация, коррелированные подзапросы, EXISTS vs IN.

Материал опирается на:

- части по **основам индексов и планам выполнения**;
- части по **устройству хранения и типам данных**;
- части по **настройке СУБД (память, I/O)**.

### Краткая шпаргалка (термины по части)

| Термин | Что это простыми словами |
|--------|---------------------------|
| **Cost (стоимость)** | Условная оценка «дороговизны» шага плана в единицах, придуманных планировщиком (не секунды); нужна для сравнения планов. |
| **Rows (оценка строк)** | Сколько строк планировщик ожидает на выходе узла; от этого сильно зависит выбор плана. |
| **Actual time** | Реальное время выполнения узла в миллисекундах (после EXPLAIN ANALYZE). |
| **Actual rows** | Реальное число строк на выходе узла (после EXPLAIN ANALYZE); сравнивай с rows. |
| **loops** | Сколько раз узел выполнился; loops=1 — один раз; большой loops — узел в цикле (часто узкое место). |
| **Execution Time** | Общее время запроса в мс (внизу плана); с ним сравнивают «до» и «после» оптимизации. |
| **Initplan** | Подзапрос, который выполняется **один раз** до основного запроса; результат кэшируется. |
| **Subplan** | Подзапрос, который может выполняться **много раз** (например, для каждой строки внешнего запроса). |
| **Гистограмма** | Распределение значений в столбце по «корзинам»; планировщик использует её для оценки селективности условий. |
| **MCV (Most Common Values)** | Список наиболее частых значений и их доля; улучшает оценку для равенства (=) на таких значениях. |
| **Correlation** | Связь между логическим порядком значений и физическим порядком на диске; влияет на оценку для range-сканов. |
| **Gather** | Узел плана, который собирает результаты от нескольких worker-процессов в один поток. |
| **Parallel worker** | Отдельный процесс/поток, выполняющий часть работы (например, кусок Seq Scan) параллельно с другими. |
| **JIT (Just-In-Time)** | Компиляция частей запроса в машинный код во время выполнения для ускорения выражений и агрегатов. |
| **USE INDEX / FORCE INDEX** | Подсказки в MySQL, заставляющие оптимизатор использовать (или принудительно использовать) указанный индекс. |
| **Nested Loop Join** | Соединение «вложенным циклом»: для каждой строки внешней таблицы ищем совпадения во внутренней; выгодно при малых объёмах. |
| **Hash Join** | Построение хеш-таблицы по одной таблице и прогон второй через неё; выгодно при больших объёмах без порядка. |
| **Merge Join** | Соединение двух уже отсортированных потоков «двумя указателями»; выгодно при отсортированных данных. |
| **Коррелированный подзапрос** | Подзапрос, который ссылается на столбцы внешнего запроса и может выполняться заново для каждой строки. |

**Где что искать:** cost, rows, actual time, узлы плана, Initplan/Subplan, BUFFERS, Index Only Scan → **§207**. Гистограммы, MCV, correlation, Extended Statistics, autovacuum → **§208**. Gather, workers, parallel scan/join/aggregate → **§209**. JIT, jit_above_cost → **§210**. Hints MySQL (включая STRAIGHT_JOIN), альтернативы в PostgreSQL → **§211**. Nested Loop, Hash, Merge join, Sort + work_mem → **§212**. Подзапросы, EXISTS vs IN, CTE, материализация → **§213**.

### Что делать по шагам, когда запрос медленный (общий алгоритм)

Чтобы не теряться, действуй по шагам:

1. **Снять план с фактом:** выполнить `EXPLAIN (ANALYZE) ...` для этого запроса. Запрос выполнится, в плане появятся actual time и actual rows.
2. **Найти узкое место:** читать план **снизу вверх**; посмотреть **actual time** у каждого узла — какой узел занимает большую долю общего времени (или смотреть **Execution time** внизу и мысленно приписывать его основным узлам). Это кандидат в узкие места.
3. **Проверить оценку:** для этого узла сравнить **rows** и **actual rows**. Если разница в разы (например, rows=100, actual rows=20000) — вероятна проблема со **статистикой** → перейти к шагу 4. Если видишь **SubPlan** с большим **loops** — подзапрос выполняется много раз → перейти к шагу 5.
4. **Обновить статистику:** выполнить `ANALYZE` по затронутым таблицам; при необходимости увеличить **statistics target** по столбцам (`ALTER TABLE ... SET STATISTICS N`) и снова ANALYZE. Затем снова снять план и проверить.
5. **Переписать подзапрос:** если узкое место — коррелированный подзапрос (SubPlan в цикле), переписать его в **JOIN** по ключу корреляции. Снять план и проверить.
6. **Проверить тип JOIN и индексы:** если узкое место — Nested Loop с Seq Scan по большой внутренней таблице, подумать об индексе по ключу соединения или о том, что планировщик мог бы выбрать Hash Join (часто помогает обновление статистики). Если Hash Join и в плане «Disk» — увеличить **work_mem**.
7. **При необходимости** использовать временные меры: в MySQL — USE INDEX / FORCE INDEX; в PostgreSQL — enable_* в сессии или pg_hint_plan. Долгосрочно лучше исправить статистику и запрос.

Этот список можно держать под рукой и проходить по пунктам, пока не станет ясно, где проблема и что менять.

**Схема принятия решений (если запрос тормозит — иди по стрелкам).**

```
Запрос медленный
    ↓
Сделал EXPLAIN (ANALYZE)?  →  НЕТ  →  Сделай. Без плана дальше идти нельзя.
    ↓ ДА
Читай план снизу вверх. Есть узел с rows сильно ≠ actual rows?
    ↓ ДА                              ↓ НЕТ
ANALYZE по таблицам.              Есть SubPlan с loops = тысячи?
Снова сними план.                     ↓ ДА              ↓ НЕТ
                                         ↓                ↓
                              Переписать подзапрос    Есть Nested Loop + Seq Scan
                              в JOIN по ключу.        по большой внутренней?
                                                          ↓ ДА        ↓ НЕТ
                                               Индекс по ключу JOIN   Hash Join + Disk
                                               или ANALYZE.           в плане?
                                                                         ↓ ДА
                                                                   Увеличить work_mem.
```

Если ни один пункт не подошёл — смотри **actual time** по узлам: какой узел даёт самое большое время? Это твоё узкое место. Действуй по контексту (индекс, фильтр, другой тип JOIN, разбить запрос).

**Пример полного разбора: от медленного запроса к быстрому (мини-кейс).**

**Шаг 1. Исходная ситуация.** Запрос: «Найти клиентов, у которых есть оплаченный заказ». Выполняется 12 секунд.  
**Шаг 2. Снимаем план.** `EXPLAIN (ANALYZE) SELECT * FROM customers c WHERE EXISTS (SELECT 1 FROM orders o WHERE o.customer_id = c.id AND o.status = 'paid');`  
**Шаг 3. Что видим в плане.** Nested Loop, внутри — SubPlan с **loops=50000**. Значит: 50 000 клиентов, подзапрос выполнился 50 000 раз. **Враг 2** — подзапрос в цикле.  
**Шаг 4. Переписываем в JOIN.** `SELECT DISTINCT c.* FROM customers c JOIN orders o ON o.customer_id = c.id AND o.status = 'paid';`  
**Шаг 5. Снова снимаем план.** SubPlan исчез. Один Hash Join, две таблицы читаются по разу. **Шаг 6. Результат.** Execution Time упал с 12 сек до 0,3 сек.  
**Вывод:** Увидел SubPlan с большим loops → переписал в JOIN → время упало в десятки раз. Так работает оптимизация по плану.

**Типичный сценарий из жизни (чтобы было понятнее).** Запрос отчёта по заказам раньше выполнялся за 200 мс, а после загрузки 500 000 новых строк стал занимать 15 секунд. Что делать? (1) Выполняешь `EXPLAIN (ANALYZE) SELECT ...` — смотришь план. (2) Читаешь снизу вверх, ищешь узел с наибольшим **actual time** — допустим, это Nested Loop с Seq Scan по таблице orders, loops=10000. (3) Сравниваешь **rows** и **actual rows** у этого узла: rows=50, actual rows=120000 — сильное расхождение. (4) Запускаешь `ANALYZE orders; ANALYZE customers;` — обновляешь статистику. (5) Снова снимаешь план: возможно, планировщик теперь выберет Hash Join вместо Nested Loop, и время упадёт. Если нет — смотришь, есть ли SubPlan с большим loops (подзапрос в цикле) или Hash Join с Disk (мало work_mem). Так по шагам находишь причину и исправляешь.

---

**Три главных врага быстрых запросов (запомни навсегда).**  
**Враг 1: Устаревшая статистика** — планировщик думает, что строк будет 100, а на самом деле 50 000. В плане: **rows** сильно отличается от **actual rows**. **Лечение:** `ANALYZE` по таблицам.  
**Враг 2: Подзапрос в цикле** — для каждой из 100 000 строк выполняется подзапрос. В плане: **SubPlan** с **loops** = тысячи или десятки тысяч. **Лечение:** переписать в **JOIN** по ключу связи.  
**Враг 3: Плохой тип JOIN** — либо Nested Loop с Seq Scan по большой внутренней таблице (на каждую строку читаем всю таблицу), либо Hash Join, у которого хеш не влез в память и ушёл на диск. В плане: Nested Loop + Seq Scan по большой таблице, или Hash Join + **Disk**. **Лечение:** индекс по ключу JOIN, или ANALYZE (чтобы планировщик выбрал Hash), или увеличить **work_mem**.  
Если запрос тормозит — ищи одного из этих трёх врагов в плане. В 90% случаев виноват кто-то из них.

**Разбор типичной ошибки: «оставил коррелированный подзапрос как есть».**

**Ошибка (как НЕ надо):** Задача — «вывести клиентов, у которых есть хотя бы один оплаченный заказ». Пишут:  
`SELECT * FROM customers c WHERE EXISTS (SELECT 1 FROM orders o WHERE o.customer_id = c.id AND o.status = 'paid');`  
Запрос тормозит 15 секунд. В плане: **Nested Loop** с **SubPlan**, **loops=80000**. Значит: 80 000 клиентов, подзапрос выполнился **80 000 раз**. Каждый раз — обращение к таблице orders. Итого 80 000 обращений вместо одного прохода.

**Почему так вышло:** Подзапрос **коррелированный** — внутри него `o.customer_id = c.id`, а `c` — таблица из **внешнего** запроса. Для каждой строки customers свой вопрос: «есть ли у этого клиента оплаченный заказ?» Планировщик не смог развернуть это в JOIN и оставил SubPlan в цикле.

**Правильно (как надо):** Переписать в **JOIN** по ключу связи (customer_id):  
`SELECT DISTINCT c.* FROM customers c JOIN orders o ON o.customer_id = c.id AND o.status = 'paid';`  
В плане: **Hash Join**, обе таблицы читаются **по одному разу**, SubPlan исчез. Время падает до 0,2–0,5 сек.

**Вывод:** Увидел в плане **SubPlan** с **loops** = тысячи → ищи в запросе **коррелированный подзапрос** (внутри подзапроса ссылка на внешнюю таблицу) → переписывай в **JOIN** по этому ключу. Это самая частая и самая «выгодная» оптимизация.

**Разбор типичной ошибки: «забыл ANALYZE после массовой загрузки».**

**Ошибка (как НЕ надо):** Загрузили 500 000 новых заказов в таблицу orders. Запрос `SELECT * FROM orders WHERE status = 'paid'` раньше выполнялся 50 мс, теперь — 8 секунд. В плане: **Seq Scan** по orders, **rows=500**, **actual rows=180000**. Планировщик думал, что строк с status='paid' будет ~500, а вышло 180 000. Он выбрал Seq Scan, хотя при такой селективности индекс по status мог бы быть выгоднее — или выбрал бы другой план.

**Почему так вышло:** Статистика (гистограммы, MCV по status) собрана по **старой** выборке. После массовой загрузки распределение значений изменилось: 'paid' теперь 36% строк, а в MCV ещё старая доля. Планировщик опирается на устаревшие цифры → неверная оценка rows → плохой план.

**Правильно (как надо):** Сразу после загрузки (или в конце пакетной вставки) выполнить:  
`ANALYZE orders;`  
Снять план снова: rows и actual rows сблизятся, планировщик может выбрать Index Scan или другой оптимальный план. Время часто падает в разы.

**Вывод:** После **массовых INSERT/UPDATE/DELETE** или **импорта** — всегда **ANALYZE** по затронутым таблицам. Иначе статистика устаревает, rows неверны, план портится.

**Разбор типичной ошибки: «Hash Join тормозит, не смотрю на Disk».**

**Ошибка (как НЕ надо):** Запрос с большим JOIN выполняется 20 секунд. В плане: **Hash Join**. Думают: «Hash Join — нормально, значит проблема в чём-то другом» — начинают искать в приложении, в сети. В выводе плана есть строка **Disk Usage: 1024 kB** или **Batches: 4** — не замечают.

**Почему так вышло:** Hash Join строит хеш-таблицу **в памяти** (лимит — work_mem, по умолчанию 4 МБ). Если хеш не влезает — часть уходит на **диск**. Поиск в памяти — микросекунды; поиск на диске — миллисекунды. Разница в тысячи раз. **Disk** в плане = часть хеша на диске = план сильно тормозит.

**Правильно (как надо):** Увидел Hash Join и запрос тормозит — ищи в плане **Disk**, **Batches > 1**, **Disk Usage**. Если есть — увеличить work_mem для сессии:  
`SET work_mem = '256MB';`  
(или другое значение под размер данных)  
Выполнить запрос снова. Если Disk исчез — хеш поместился в память, время обычно падает в разы.

**Вывод:** Hash Join + **Disk** в плане = хеш не влез в work_mem. Первое действие — увеличить **work_mem**, а не искать причину в коде приложения.

### Проверь себя (вводная часть)

<details>
<summary>Назови три главных «врага» быстрых запросов и что делать в каждом случае.</summary>

**Враг 1 — устаревшая статистика:** в плане rows сильно отличается от actual rows. **Лечение:** ANALYZE по таблицам. **Враг 2 — подзапрос в цикле:** в плане SubPlan с loops = тысячи. **Лечение:** переписать в JOIN по ключу связи. **Враг 3 — плохой тип JOIN:** Nested Loop + Seq Scan по большой внутренней или Hash Join + Disk. **Лечение:** индекс по ключу JOIN, ANALYZE или увеличить work_mem.
</details>

<details>
<summary>С чего начать, если запрос тормозит и ты ещё не смотрел план?</summary>

Выполнить **EXPLAIN (ANALYZE)** для этого запроса. Читать план **снизу вверх**, смотреть **actual time** по узлам — какой узел съедает больше всего времени. Сравнить **rows** и **actual rows**: большое расхождение → проблема со статистикой (ANALYZE). SubPlan с большим loops → переписать подзапрос в JOIN.
</details>

<details>
<summary>В чём ошибка: «забыл ANALYZE после массовой загрузки» и «Hash Join тормозит, не смотрю на Disk»?</summary>

После массовой загрузки статистика устаревает — планировщик ошибается в rows, план портится. Нужно явно выполнить **ANALYZE** по затронутым таблицам. При тормозах Hash Join нужно искать в плане **Disk** (часть хеша на диске из-за нехватки work_mem) и увеличить **work_mem** для сессии.
</details>
