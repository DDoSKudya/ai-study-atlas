## 170. Elasticsearch детали

### В этом разделе главное

- Elasticsearch — это **поисковый движок и аналитическая система**, а не транзакционная БД.
- Главные идеи: **mapping** (какие поля и как хранятся), **analyzers** (как обрабатывается текст) и **Query DSL** (как формулируются запросы).
- Агрегации позволяют строить **аналитику и отчёты** поверх логов и событий без отдельного хранилища.

### Простыми словами

Представь, что у тебя есть **огромная библиотека текстов**:

- Elasticsearch строит **инвертированный индекс** — вместо «документ → слова» он хранит «слово → список документов».
- Перед этим текст **анализируется**: разбивается на токены (слова), приводится к базовой форме, выкидываются стоп‑слова.
- Затем по этому индексу можно очень быстро искать по словам, фразам, фильтрам и строить агрегаты.

---

### Mapping

Mapping описывает:

- **поля документа** (string, number, date, boolean, объект, nested и т.п.),
- **как именно обрабатывать строки** (анализировать или хранить как keyword),
- **нужны ли индексы** по полю.

Пример упрощённого mapping:

```json
{
  "mappings": {
    "properties": {
      "title":    { "type": "text", "analyzer": "russian" },
      "title_raw":{ "type": "keyword" },
      "tags":     { "type": "keyword" },
      "created_at": { "type": "date" },
      "price":    { "type": "float" }
    }
  }
}
```

Идея:

- `text` — для полнотекстового поиска (анализируется),
- `keyword` — для точного совпадения и агрегаций,
- числа/даты — для фильтров и сортировок.

Можно думать так:

- `text` — **как книга, которую читают и ищут по словам**;
- `keyword` — **как штрих‑код на коробке**: нужен для точного совпадения, а не для «поиска по смыслу».

---

### Analyzers

Analyzer = tokenizer + фильтры:

- разбивает строку на токены,
- приводит к нижнему регистру,
- убирает знаки препинания,
- может делать стемминг (приведение к основе слова).

Пример: `"Привет, мир!"` → `["привет", "мир"]`.

От настройки analyzers сильно зависит:

- будут ли находиться нужные формы слова,
- как работает поиск по фразам,
- будут ли учтены особенности языка (русский, английский и т.д.).

Если проанализировать одно и то же поле разными analyzers, получится разный набор токенов — отсюда разница в результатах поиска.  
Полезное упражнение: взять одну строку текста и через Dev Tools в Kibana посмотреть, **как её видят разные analyzers**.

---

### Query DSL

Query DSL — это **JSON‑описание запроса**. База:

- `match` — полнотекстовый поиск по полю,
- `term` — точное совпадение,
- `range` — диапазон (числа, даты),
- `bool` — комбинация условий (`must`, `should`, `must_not`),
- `nested` — запросы по вложенным объектам.

Пример:

```json
{
  "query": {
    "bool": {
      "must": [
        { "match": { "title": "поисковая оптимизация" } }
      ],
      "filter": [
        { "term": { "tags": "seo" } },
        { "range": { "created_at": { "gte": "2025-01-01" } } }
      ]
    }
  }
}
```

Простыми словами: «найти документы, где в названии говорится о поисковой оптимизации, с тегом `seo`, созданные после 1 января 2025 года».

Паттерн, который стоит запомнить:

- **`must`** — «обязательно должно совпасть» (аналог AND),
- **`should`** — «желательно, но не обязательно» (вес для релевантности),
- **`filter`** — «жёсткий фильтр без влияния на релевантность» (часто для дат, тегов, статусных флагов).

---

### Релевантность и BM25

Когда ты делаешь полнотекстовый запрос (`match`), Elasticsearch должен решить, **насколько каждый документ «подходит» запросу**.  
Этот «уровень подходящести» — это **score** (балл релевантности), который считается по алгоритму **BM25** (по умолчанию).

Интуитивно BM25 учитывает:

- **частоту термина в документе** (чем чаще встречается слово из запроса — тем лучше);
- **редкость термина в коллекции** (слова вроде «и», «в», «на» мало помогают отличать документы);
- **длину документа** (слишком длинные документы слегка «штрафуются», чтобы не выигрывать только за счёт длины).

Что это даёт на практике:

- по запросу `поисковая оптимизация`:
  - документ, где эта фраза встречается в заголовке и несколько раз в тексте, получит **более высокий score**;
  - документ, где слово встречается один раз в большом тексте, — более низкий.

Можно дополнительно управлять релевантностью:

- **boost'ы**:
  - повышать вклад одних полей (`title`) по сравнению с другими (`body`);
  - усиливать важность определённых условий (например, свежих дат).
- **`function_score`**:
  - модифицировать score формулами (например, учитывать популярность, новизну, рейтинг).

Главная мысль:

- Elasticsearch — это не просто «нашёл / не нашёл»;
- он **сортирует результаты по степени «подходящести»**, и от того, как настроены mapping, analyzers и boost'ы, сильно зависит, какие документы окажутся в топ‑результатах.

---

### Агрегации

Агрегации — это **аналитические запросы**:

- **metric**: `avg`, `sum`, `min`, `max`, `cardinality` и др.
- **bucket**: группировки (`terms`, `date_histogram`, `range` и др.).
- **pipeline**: производные от других агрегатов (derivative, moving_avg и др.).

Пример: распределение заказов по дням и суммарная выручка:

```json
{
  "size": 0,
  "aggs": {
    "by_day": {
      "date_histogram": {
        "field": "created_at",
        "calendar_interval": "day"
      },
      "aggs": {
        "total_amount": {
          "sum": { "field": "amount" }
        }
      }
    }
  }
}
```

В ответе будет:

- список «вёдер» по дням,
- в каждом — сумма `amount`.

Так можно строить:

- дашборды выручки,
- графики количества ошибок,
- распределения по статусам и регионам —
без отдельного хранилища, используя тот же Elasticsearch, который и хранит логи/события.

---

### ILM (Index Lifecycle Management)

Идея:

- старые данные можно **перемещать в более дешёвое хранилище** (warm/cold),
- потом удалять (delete),
- всё это автоматически по политике.

Типичный сценарий:

- логи:
  - первые N дней — на быстрых дисках (hot),
  - следующие M дней — на дешёвых (warm),
  - ещё позже — только агрегаты или оффлайн‑архив.

Идея та же, что и с бэкапами/retention в других частях курса: **старые данные нужны реже и в более грубом виде**, поэтому нет смысла держать их на самых быстрых и дорогих дисках.

---

### Типичные ошибки

- Использовать Elasticsearch как **основную транзакционную БД**:
  - нет сильной транзакционности,
  - возможна задержка между записью и доступностью для поиска (`refresh_interval`).
- Лить в один индекс всё подряд без продуманного mapping.
- Индексировать поля как `text`, а потом удивляться, почему точные фильтры не работают (нужно `keyword`).
- Пытаться делать очень сложные финансовые отчёты там, где лучше подходит нормальный SQL.

Дополнительно:

- забывать про ограничения на размер индекса/кластера и воспринимать Elasticsearch как «бездонную коробку»;
- включать слишком частый `refresh` ради «почти моментальной видимости» документов и убивать производительность;
- писать громоздкие, плохо читаемые запросы в Query DSL без разбиения на шаги и без проверки в Dev Tools.

### Проверь себя

<details>
<summary>1. Зачем часто делают пару полей <code>text</code> + <code>keyword</code> (например, <code>title</code> и <code>title_raw</code>)?</summary>

Пара `text` + `keyword` позволяет:

- по `text` — делать полнотекстовый поиск (анализ, стемминг, релевантность),
- по `keyword` — делать точные фильтры и агрегации (`terms`, `terms aggregation`).

Так одно и то же логическое поле обслуживает два разных сценария.

</details>

<details>
<summary>2. Почему нельзя ожидать, что документ будет мгновенно виден в поиске сразу после индексации?</summary>

Потому что Elasticsearch **индексирует данные пакетами**:

- есть параметр `refresh_interval` (по умолчанию ~1 секунда),
- до следующего refresh новый документ может быть не виден в поиске.

Для некоторых сценариев это нормально (логи, аналитика), для других — нужно либо настраивать refresh, либо проектировать логику с учётом задержки.

</details>

<details>
<summary>3. Почему важно понимать, как именно считается score (BM25), даже если ты не меняешь настройки по умолчанию?</summary>

Потому что score определяет, **какие документы пользователь увидит первыми**, а какие — внизу или вообще не заметит.  
Понимание общих принципов BM25 (частота термина, редкость слова, длина документа) помогает объяснять, почему «странный» документ попал в топ, и как это можно поправить (mapping, analyzers, boosting).  
Без этого Elasticsearch может казаться «магией», а настройка релевантности превратится в угадайку.

</details>

#### Если прочитал и не понял

Сделай себе минимальный план:

1. Возьми один индекс с простым mapping (`title`, `tags`, `created_at`, `amount`).
2. Напиши **один простейший запрос `match`** по `title` и посмотри результат.
3. Добавь `filter` по тегу и дате — увидь, как он сужает выборку.
4. Открой раздел про агрегации и напиши агрегацию по дням, как в примере — посмотри, как выглядит ответ.

Главное — не пытаться «обнять» весь Query DSL сразу. Каждая новая конструкция — это ещё один кирпичик в понимании, а не новая вселенная.
