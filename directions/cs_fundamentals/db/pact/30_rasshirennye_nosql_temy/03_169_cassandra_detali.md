## 169. Cassandra детали

### В этом разделе главное

- Cassandra — это **широкостолбцовая распределённая БД**, заточенная под масштаб и скорость записи.
- В Cassandra ты сначала **проектируешь запросы**, а потом под них — таблицы и ключи (partition key, clustering key).
- Secondary indexes и materialized views — **инструменты на крайний случай**, основной путь — продуманная модель данных и уровни консистентности.

### Простыми словами

Представь, что Cassandra — это **огромный набор разделов (партиций)**:

- у каждой партиции есть **partition key** — по нему данные живут на конкретном наборе узлов;
- внутри партиции строки отсортированы по **clustering key**;
- запросы должны быть спроектированы так, чтобы **читать/писать по ключу**, а не «гулять по всей таблице».

Если тебе нужно «гибкое произвольное фильтрование» — это сигнал, что Cassandra может быть **не тем инструментом**.

---

### CQL: знакомо, но не SQL

Синтаксис CQL похож на SQL, но:

- нет JOIN (между таблицами),
- нет сложных подзапросов, как в PostgreSQL,
- многие операции запрещены, если они не могут эффективно отработать по ключам.

Пример таблицы:

```sql
CREATE TABLE user_events (
    user_id      uuid,
    event_date   date,
    event_time   timestamp,
    event_type   text,
    payload      text,
    PRIMARY KEY ((user_id, event_date), event_time)
) WITH CLUSTERING ORDER BY (event_time DESC);
```

Здесь:

- **partition key** = `(user_id, event_date)` — определяет, на каких узлах живёт партиция;
- **clustering key** = `event_time` — сортировка внутри партиции;
- запросы должны использовать `user_id` и `event_date`, чтобы быть эффективными.

Можно думать так:

- сначала ты решаешь: **какие запросы мне нужны?** (по пользователю, по дате, по типу события и т.п.);
- затем каждый запрос превращаешь в «формулу» вида:
  - «я знаю user_id и дату → хочу получить все события по времени»;
  - это и есть кандидат на таблицу/ключ;
- только после этого создаёшь таблицу в CQL — **таблица = оптимизированный ответ под конкретный запрос**.

---

### Materialized views

Materialized view — это **альтернативное представление таблицы** с другим ключом:

```sql
CREATE MATERIALIZED VIEW user_events_by_type AS
  SELECT user_id, event_date, event_time, event_type, payload
  FROM user_events
  WHERE event_type IS NOT NULL
    AND user_id IS NOT NULL
    AND event_date IS NOT NULL
    AND event_time IS NOT NULL
  PRIMARY KEY (event_type, event_date, event_time, user_id);
```

Идея:

- основная таблица остаётся «истинным источником»,
- view автоматически поддерживается Cassandra при изменениях,
- по view можно делать запросы по `event_type` как по partition key.

Но:

- это дополнительная нагрузка на запись,
- есть тонкости и баги (в разных версиях),
- использовать стоит осторожно и только там, где это реально упрощает жизнь.

---

### Secondary indexes

Secondary index в Cassandra — это **локальный индекс по каждому узлу**, а не глобальный, как в реляционных БД.

- Хорошо работает, если:
  - у тебя **много партиций, но мало строк внутри каждой**,
  - запросы по индексу выбирают **немного строк**.
- Плохо, если:
  - у тебя мало партиций, но каждая огромная,
  - по индексу выбирается много данных (индекс становится «почти вся таблица»).

Правило:

- **сначала** пробуй спроектировать модель без secondary indexes,
- **только потом**, при явной необходимости, аккуратно добавляй индекс и тестируй поведение.

---

### Batch‑операции

Batch в Cassandra — это способ **отправить несколько операций как один логический блок**.

Два основных вида:

- **logged batch**:
  - операции записываются в специальный лог;
  - Cassandra гарантирует, что либо **все операции будут применены**, либо никакая не «потеряется» (при сбоях они дозаписываются);
  - надёжнее, но **дороже по ресурсам**.
- **unlogged batch**:
  - нет дополнительного лога;
  - операции просто отправляются вместе;
  - быстрее, но без дополнительных гарантий целостности.

Правила здорового использования:

- batch **не для ускорения большого количества однотипных операций** по разным partition key;
- он полезен, когда:
  - несколько операций касаются **одного и того же partition key** — уменьшает накладные расходы;
  - нужно **согласованно обновить несколько строк**, связанных логически.

Анти‑паттерн:

- использовать один огромный batch, чтобы «разом загрузить много данных» по тысячам разных partition key;
- это может:
  - сильно перегрузить кластер,
  - дать противоположный эффект по производительности.

Удобно запомнить: **batch — это про удобство и согласованность, а не «турбо‑ускоритель записи всего сразу»**.

---

### Lightweight transactions (LWT)

LWT — это **сравнить‑и‑установить** (compare‑and‑set) операции:

```sql
INSERT INTO accounts (id, balance)
VALUES (123, 100)
IF NOT EXISTS;

UPDATE accounts
SET balance = balance - 10
WHERE id = 123
IF balance >= 10;
```

Идея:

- Cassandra выполняет операцию **только если условие истинно**,
- под капотом — консенсус между репликами (дорого!).

Применение:

- уникальные ограничения (создать запись, только если её не было),
- простые инварианты (не давать уйти балансу в минус).

Но:

- LWT **существенно медленнее обычных операций**,
- их нельзя использовать повсюду, иначе смысл Cassandra как бытеррой системы теряется.

Практический ориентир:

- если тебе хочется написать `IF` почти в каждой операции — это красный флаг, что ты пытаешься использовать Cassandra как обычную реляционную БД с жёсткими инвариантами;
- если LWT появляются только в **1–2 местах по всей системе** (создание уникальной записи, проверка остатка перед списанием) — это нормальный, осознанный компромисс.

---

### Consistency levels

Уровни консистентности определяют, **сколько реплик** участвуют в операции:

- `ONE`, `TWO`, `THREE` — запись/чтение с 1/2/3 реплик;
- `QUORUM` — большинство реплик в датацентре;
- `LOCAL_QUORUM` — большинство реплик в локальном датацентре;
- `ALL` — все реплики должны подтвердить.

Классический паттерн:

- `write: QUORUM`, `read: QUORUM` → сильная консистентность (если большинство реплик «живы»);
- `write: ONE`, `read: QUORUM` → быстрая запись, более надёжное чтение;
- `write: QUORUM`, `read: ONE` → более надёжная запись, быстрое чтение.

Смысл: можно **тонко настраивать баланс между скоростью, доступностью и консистентностью**.

Удобно запомнить простую формулу:

- хочешь **максимальной скорости** → `ONE`,
- хочешь **баланса** → `QUORUM`,
- хочешь **максимальной надёжности** (и готов платить временем) → `ALL`.

---

### Типичные ошибки

- Проектировать схему «как в реляционной БД», а потом пытаться писать произвольные фильтры — Cassandra так не работает.
- Переиспользовать одну и ту же таблицу для множества разных запросов.
- Обильно использовать secondary indexes «чтобы было как в SQL».
- Игнорировать уровни консистентности и LWT — либо всё писать с `ALL`, либо везде `ONE`, не понимая последствий.

Часто встречающийся анти‑паттерн:

- команда берёт «старую» схему из PostgreSQL,
- почти без изменений переносит её в Cassandra,
- удивляется, что:
  - половина запросов запрещена,
  - другая половина тормозит,
  - а кластер всё равно дорогой.

Правильный путь: **начинать с паттернов доступа**, а не с таблиц.

### Проверь себя

<details>
<summary>1. Почему в Cassandra нельзя просто сделать «таблицу пользователей» и потом свободно фильтровать по любому полю?</summary>

Потому что Cassandra распределяет данные по узлам на основе **partition key**.  
Если запрос не использует partition key (и подходящий clustering key), БД не знает, на каких узлах искать данные, и должна фактически опрашивать весь кластер.  
Это ломает масштабируемость и производительность, поэтому такие запросы либо запрещаются, либо работают очень плохо.

</details>

<details>
<summary>2. Чем опасно злоупотребление Lightweight transactions?</summary>

LWT требуют **координации между репликами** (консенсус), что:

- сильно увеличивает задержку операций,
- увеличивает нагрузку на кластер.

Если ими «обернуть» каждую запись, Cassandra превращается в медленную транзакционную БД, теряя своё главное преимущество — высокую скорость и масштабируемость.

</details>

<details>
<summary>3. Зачем в Cassandra вообще нужны materialized views, если можно просто завести ещё одну таблицу?</summary>

Materialized view позволяет **автоматически поддерживать второе представление тех же данных** с другим ключом.  
Это удобно, когда ты хочешь читать те же данные по альтернативному ключу (например, по `event_type`), но не хочешь руками дублировать логику обновления везде в коде.  
При этом нужно помнить, что это увеличивает нагрузку на запись и усложняет поведение, поэтому view используют только там, где это действительно упрощает жизнь по сравнению с двумя независимыми таблицами.

</details>

#### Если прочитал и не понял

Сфокусируйся на двух мысленных картинках:

1. **Карта кластера**: данные не хранятся в одном месте, а разбросаны по узлам по partition key. Любой запрос, который не знает partition key, автоматически подозрителен.
2. **Таблица как ответ**: каждая таблица — это как заранее подготовленный отчёт для одного‑двух типов запросов, а не универсальная «таблица обо всём».

Чтобы закрепить:

- придумай 2–3 запроса (например, «все события пользователя за день», «все события конкретного типа за день»),
- нарисуй, как могла бы выглядеть таблица под каждый запрос (какой partition key, какой clustering key),
- только потом смотри на пример DDL выше и ищи сходства.
