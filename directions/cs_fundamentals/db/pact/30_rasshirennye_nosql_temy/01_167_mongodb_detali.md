## 167. MongoDB детали

### В этом разделе главное

- MongoDB — **документная БД**, где главные инструменты — **коллекции, индексы и aggregation pipeline**.
- `Aggregation pipeline` — это **конвейер**, в котором каждый шаг фильтрует, группирует, сортирует или как‑то преобразует документы.
- Транзакции и тонкие настройки `read/write concern` нужны для случаев, где **важнее целостность**, чем «максимальная скорость любой ценой».

### Простыми словами

Представь, что у тебя есть **коробка с документами** (JSON‑объекты): каждый документ — это целый «профиль пользователя» или «заказ целиком».  
`Aggregation pipeline` — это как если бы ты:

1. Сначала **отобрал нужные документы** (`$match`),
2. Потом **сгруппировал их по какому‑то полю** (`$group`),
3. Затем **выбрал только нужные поля и переименовал их** (`$project`),
4. И в конце **отсортировал и обрезал список** (`$sort`, `$limit`).

Всё это происходит **внутри MongoDB**, без промежуточного вытаскивания данных в приложение — поэтому быстро и удобно.

---

### Теория и правила

#### Базовые блоки aggregation pipeline

Чаще всего ты будешь использовать несколько стандартных стадий:

- **`$match`** — фильтрация документов по условиям (аналог `WHERE` в SQL).
- **`$project`** — выбор и преобразование полей (аналог `SELECT` с вычисляемыми столбцами).
- **`$group`** — группировка и агрегаты (аналог `GROUP BY`).
- **`$sort`** — сортировка.
- **`$limit` / `$skip`** — пагинация.
- **`$lookup`** — «JOIN» по другой коллекции.
- **`$unwind`** — «разворачивание» массива: один документ с массивом превращается в несколько документов — по одному элементу.
- **`$facet`** — выполнение нескольких независимых пайплайнов над одним и тем же входным набором (например, и пагинация, и подсчёт статистики за один проход).

Главная идея: **все стадии идут строго последовательно**, каждый этап работает с результатом предыдущего.

#### Пример общей структуры

Типичный pipeline:

```javascript
db.orders.aggregate([
  { $match: { status: "PAID", createdAt: { $gte: ISODate("2025-01-01") } } },
  { $lookup: {
      from: "users",
      localField: "userId",
      foreignField: "_id",
      as: "user"
  }},
  { $unwind: "$user" },
  { $group: {
      _id: "$user.country",
      totalAmount: { $sum: "$amount" },
      ordersCount: { $sum: 1 }
  }},
  { $sort: { totalAmount: -1 } },
  { $limit: 10 }
])
```

Простыми словами: «найти оплаченные заказы, подтащить пользователей, посчитать оборот по странам и вывести топ‑10».

#### Индексы и производительность

Для MongoDB критично:

- **Индекс поддерживает быстрое `$match` и `$sort`**:
  - старайся, чтобы первые стадии пайплайна (`$match`, часть `$sort`) могли использовать индекс;
  - порядок полей в составном индексе имеет значение (как и в реляционных БД).
- **Индекс по полю из `$lookup`**:
  - в коллекции `from` (`users` в примере выше) обязательно должен быть индекс по `foreignField`, иначе `lookup` превратится в полный скан.

Если игнорировать индексы, MongoDB очень быстро превращается в медленную БД «с JSONчиками».

#### Change streams

`Change streams` позволяют **подписаться на изменения** в коллекции/БД/кластере:

- Под капотом используется **oplog реплика‑сета** — журнал операций.
- Можно слушать события `insert`, `update`, `replace`, `delete`.
- Частые сценарии:
  - поддержка **кэша** (обновлять Redis/ин‑мемори при изменениях);
  - **интеграции** (отправлять события в Kafka / другой сервис);
  - **аудит** (записывать важные изменения в отдельное хранилище).

Работает только на **реплика‑сетах и шардированных кластерах**, не на одиночном `standalone`‑инстансе.

#### Транзакции

MongoDB долго была «single‑document только», но теперь есть **мультидокументные транзакции**:

- Они нужны, когда **несколько коллекций/документов должны меняться атомарно** (например, «списать деньги и создать запись о платеже»).
- Но:
  - транзакции **дороже** по ресурсам, чем одиночные операции;
  - в документации MongoDB прямо советуют *по возможности проектировать модели так, чтобы хватало одной операции обновления документа*.

Практически:

- **Сначала** думай, можно ли решить задачу в рамках одного документа или через «event‑sourcing»‑подход.
- Только если никак — смотри в сторону транзакций.

#### Read concern и write concern

Эти параметры управляют тем, **какие гарантии ты получаешь**:

- `writeConcern`:
  - `w: 1` — запись считается успешной, когда принята *одной* репликой (обычно первичной);
  - `w: "majority"` — когда большинство реплик подтвердили запись;
  - дополнительный флаг `j: true` — требовать записи в журнал (journal) на диске.
- `readConcern`:
  - `local` — читаем, что есть на конкретной реплике (может отставать);
  - `majority` — читаем только данные, подтверждённые большинством;
  - есть и другие режимы (например, `linearizable`), но они дороже и реже нужны.

Идея та же, что и в Cassandra/Dynamo: **можно купить себе более сильные гарантии ценой задержки и нагрузки.**

### Примеры

#### Пример 1. Конвертация «SQL отчёта» в pipeline

SQL‑вариант:

```sql
SELECT  u.country,
        COUNT(*)           AS orders_count,
        SUM(o.amount)      AS total_amount
FROM    orders o
JOIN    users u ON u.id = o.user_id
WHERE   o.status = 'PAID'
  AND   o.created_at >= DATE '2025-01-01'
GROUP BY u.country
ORDER BY total_amount DESC
LIMIT 10;
```

В MongoDB:

```javascript
db.orders.aggregate([
  { $match: {
      status: "PAID",
      createdAt: { $gte: ISODate("2025-01-01") }
  }},
  { $lookup: {
      from: "users",
      localField: "userId",
      foreignField: "_id",
      as: "user"
  }},
  { $unwind: "$user" },
  { $group: {
      _id: "$user.country",
      ordersCount: { $sum: 1 },
      totalAmount: { $sum: "$amount" }
  }},
  { $sort: { totalAmount: -1 } },
  { $limit: 10 }
])
```

Если ты уже умеешь писать такие запросы в SQL, задача — **привыкнуть к «ступенчатому» синтаксису pipeline**.

#### Пример 2. `$facet`: статистика и страница данных за один запрос

Частый шаблон: **вернуть страницу данных и общую статистику**:

```javascript
db.orders.aggregate([
  { $match: { status: "PAID" } },
  { $facet: {
      page: [
        { $sort: { createdAt: -1 } },
        { $skip: 0 },
        { $limit: 20 }
      ],
      stats: [
        { $group: {
            _id: null,
            totalAmount: { $sum: "$amount" },
            count: { $sum: 1 }
        }}
      ]
  }}
])
```

В ответе:

- в `page` — 20 последних заказов,
- в `stats[0]` — агрегаты по всем совпавшим заказам.

Обе задачи решаются за **один проход по данным**.

### Картинка в голове

Думай о MongoDB так:

- **Документ = строка + связанные подтаблицы** (вложенные объекты и массивы).
- **Pipeline = лента конвейера**: каждый шаг что‑то делает с «текущей» пачкой документов.
- **Индекс = карта склада**: без карты рабочие ходят хаотично, с картой — быстро находят нужные коробки.

Ещё один полезный образ:

- коллекция — это **«ящик с папками»**;
- каждая папка (документ) содержит:
  - базовые поля (id, даты),
  - вложенные «поддокументы» (адреса, товары, настройки);
- pipeline — это **маршрут по офису**, где ты:
  - сначала проходишь по ряду шкафов и выбираешь только нужные (фильтрация),
  - потом на столе раскладываешь папки по стопкам (группировка),
  - потом с каждой стопки переписываешь в отчёт только нужные цифры (проекция),
  - потом сортируешь стопки по сумме и берёшь верхние (сортировка и лимит).

Если чувствуешь, что «тонешь» в стадиях — нарисуй этот маршрут на бумаге: каждая стрелка — новая стадия pipeline.

### Типичные ошибки

- «Мы используем MongoDB, поэтому нам не нужны индексы» — в итоге всё работает через полный скан и тормозит.
- «Можно всё держать в одной гигантской коллекции без продуманных схем» — потом тяжело мигрировать и оптимизировать.
- «Транзакции решат любые проблемы целостности» — они есть, но **дороги** и не отменяют необходимости хорошо проектировать документы.
- Игнорирование `read/write concern`: по умолчанию всё может быть настроено слишком «расслабленно» для критичных данных.

Дополнительные «подлянки» из практики:

- хранить **очень большие документы** (сотни килобайт/мегабайты) и постоянно их целиком перезаписывать — сильно бьёт по диску и сетевому трафику;
- делать `$lookup` между огромными коллекциями без индексов по полям соединения — получаешь гигантские временные структуры в памяти;
- использовать **динамическую схему без правил**: вчера поле называлось `price`, сегодня `amount`, завтра `sum` — через год уже никто не понимает, что происходит;
- не проверять результаты `explain()` и верить, что «раз работает, значит всё ок» — а на проде под нагрузкой всё рушится.

### Проверь себя

<details>
<summary>1. Зачем в MongoDB нужен <code>$unwind</code> и в каких случаях он полезен?</summary>

`$unwind` «разворачивает» массив внутри документа в несколько документов — по одному элементу массива на документ.  
Это полезно, когда нужно:

- агрегировать по элементам массива (например, по товарам в заказе),
- фильтровать по свойствам элементов массива,
- фактически сделать неявный «подзапрос» по вложенной коллекции.

</details>

<details>
<summary>2. Почему нельзя просто всегда ставить <code>writeConcern: { w: "majority", j: true }</code> и забыть?</summary>

Потому что это **дорогий режим**:

- каждая запись должна быть подтверждена большинством реплик и записана в журнал,
- это увеличивает задержки и нагрузку на кластер.

Для критичных данных это оправдано, но для второстепенных (логи, вторичные представления, кеши) можно выбрать более лёгкие режимы, чтобы не тратить ресурсы зря.

</details>

<details>
<summary>3. Почему MongoDB так сильно «завязана» на правильные индексы?</summary>

Потому что без индекса база вынуждена **просматривать все документы в коллекции** при каждом запросе (`collection scan`).  
На маленьких данных это почти незаметно, но при реальных объёмах (миллионы документов) такой подход делает даже простые запросы непригодными для продакшена.  
Индекс позволяет быстро найти только нужные документы и избежать лишнего чтения с диска и из памяти.

</details>

#### Если прочитал и не понял

Попробуй пройти путь «с нуля» по шагам:

1. Возьми **маленький пример**: коллекция `users` с полями `name`, `age`, `country`.  
   Напиши самый простой pipeline из одной стадии `$match`, который выбирает только пользователей старше 18 лет из конкретной страны.
2. Добавь вторую стадию `$project`: оставь только `name` и `age`, а поле `age` переименуй в `years`.  
   Почувствуй, что каждая стадия — это **ещё один фильтр/трансформатор** над потоком документов.
3. Добавь `$group` по стране и посчитай средний возраст.  
   Сравни с тем, как ты писал бы `GROUP BY` в SQL — логика та же, только запись другая.
4. Открой документацию MongoDB по `aggregation pipeline` и найди знакомые стадии из этого раздела.  
   Отметь галочкой: «это я уже видел, не так страшно».

Если мозг всё ещё сопротивляется — возвращайся к аналогии с конвейером: каждый блок не делает ничего «магического», он просто берёт пачку документов и превращает её в другую пачку.

### Мини‑курс по MongoDB на 5 дней (для «совсем с нуля»)

Если хочешь не просто прочитать, а **реально почувствовать MongoDB руками**, можно двигаться так.

#### День 1. Установить и «пощупать» документы

**Цель:** увидеть живую MongoDB и перестать бояться консоли.

- Поставь MongoDB (или запусти контейнер) — ориентируйся на актуальную документацию под свою ОС.
- Зайди в `mongosh` (интерактивная консоль).
- Создай базу и коллекцию:
  - вставь несколько простых документов в коллекцию `users`:
    - `{ name: "Alice", age: 25, country: "US" }`,
    - `{ name: "Bob", age: 30, country: "DE" }` и т.п.
- Сделай несколько простых `find`:
  - «все пользователи»;
  - пользователи из конкретной страны;
  - пользователи старше определённого возраста.

Главное на этом этапе — **увидеть, что документация и консоль не кусаются**: ты создаёшь документы, они возвращаются, всё предсказуемо.

#### День 2. Вложенные структуры и индексы

**Цель:** понять, что документ — это «строка + связанные подтаблицы».

- Измени схему `users`, добавив вложенные объекты:
  - `address` (город, улица),
  - массив `orders` (несколько заказов внутри пользователя).
- Вставь пару таких «богатых» документов.
- Создай **индекс** по полю `country` и посмотри разницу в плане выполнения (`db.users.find(...).explain()`).
- Поговори с собой:
  - какие данные логично хранить **внутри документа** (вложенные объекты/массивы),
  - а какие — в отдельной коллекции (где ожидается очень много записей, пересечения между пользователями и т.п.).

#### День 3. Первый aggregation pipeline

**Цель:** перестать бояться слова `aggregate`.

- Возьми коллекцию с пользователями и заказами (можно упростить: отдельная коллекция `orders`).
- Напиши **самый простой pipeline** из одной стадии `$match`.
- Добавь по очереди:
  - `$project` (оставить только нужные поля, переименовать пару полей),
  - `$group` (посчитать количество заказов по пользователю или по стране),
  - `$sort` и `$limit`.
- Для каждого шага:
  - записывай, **что было на входе и что стало на выходе**;
  - сравнивай с аналогичным SQL‑запросом.

Пока можно не трогать `$lookup` и `$facet` — важнее прочувствовать простую «ленту конвейера».

#### День 4. `$lookup`, `$unwind` и небольшие отчёты

**Цель:** потренироваться делать что‑то вроде JOIN с помощью MongoDB.

- Сделай две коллекции:
  - `users` (основные данные о пользователях),
  - `orders` (заказы с полем `userId`).
- Сконструируй pipeline из примера в этом разделе:
  - `$match` по статусу,
  - `$lookup` по пользователю,
  - `$unwind` пользователя,
  - `$group` по стране,
  - `$sort` и `$limit`.
- Поиграйся:
  - измени поле группировки (например, по статусу или продукту),
  - добавь ещё один агрегат (например, `avg` или `max`).

Важно не заучить синтаксис наизусть, а **увидеть, что pipeline — это тот же SQL‑отчёт, только записанный ступенчато**.

#### День 5. Индексы, `explain` и настройки надёжности

**Цель:** почувствовать, как MongoDB ведёт себя под нагрузкой и что дают индексы/concern.

- Возьми одну из коллекций и:
  - **сделай несколько тысяч/десятков тысяч вставок** (можно сгенерировать тестовые данные),
  - измерь время простых запросов до индекса и после.
- Поэкспериментируй с `explain()`:
  - посмотри, какие запросы используют индекс,
  - какие делают полный скан.
- Попробуй поменять `writeConcern` и `readConcern` на уровне отдельного запроса (в документации есть примеры) и подумать:
  - где в твоём реальном/будущем проекте нужна «супер‑надёжность»,
  - а где достаточно более «лёгкого» режима.

После этих 5 дней MongoDB перестанет быть «страшным зверем» и станет **понятным инструментом**, к которому в этой части ты можешь просто возвращаться как к более подробной шпаргалке.
