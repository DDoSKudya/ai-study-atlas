## 173. DynamoDB детали

### В этом разделе главное

- DynamoDB — управляемая key‑value/документная БД AWS с **почти бесконечным масштабом**, если правильно выбрать ключи и паттерны доступа.
- Модель «таблица → items → attributes», где основа — **partition key** и необязательный **sort key**.
- GSI/LSI, streams и TTL позволяют строить гибкие и производительные системы без ручного администрирования.

### Простыми словами

Представь большую таблицу:

- каждая строка — `item`,
- у каждого item есть **обязательный primary key**, состоящий из:
  - `partition key` — определяет, на каком «шарде» живёт запись,
  - `sort key` (опционален) — порядок внутри раздела.

По умолчанию DynamoDB заточена на:

- **очень быстрое чтение/запись по ключу**,
- автоматическое масштабирование под нагрузку,
- отказоустойчивость «как сервис» (AWS берёт администрирование на себя).

---

### Partition key и sort key

Примеры:

- только `partition key`:

  ```text
  PK = user_id
  ```

  Таблица «профили пользователей».

- составной ключ:

  ```text
  PK = user_id
  SK = order_id
  ```

  Таблица «заказы пользователя».

Составные ключи позволяют:

- хранить в одной таблице **разные типы сущностей** (pattern «single table design»),
- выполнять запросы по диапазонам sort key (`BETWEEN`, `begins_with` и т.п.).

Хороший практический приём:

- выписать все типы запросов, которые будут в системе (по пользователю, по заказу, по статусу и т.п.);
- для каждого спросить: **«какой ключ я знаю на момент запроса?»**;
- выбирать `partition key` так, чтобы:
  - он был известен в момент основного запроса,
  - его значения **равномерно распределялись** по пространству (избегать одного «супер‑популярного» значения).

---

### GSI и LSI

- **Global Secondary Index (GSI)**:
  - может иметь **свои PK и SK**, отличные от основной таблицы,
  - живёт на **отдельных ресурсах** (можно задавать свои лимиты),
  - подходит для запросов по альтернативным ключам (например, по email вместо user_id).
- **Local Secondary Index (LSI)**:
  - разделяет **тот же partition key**, что и основная таблица,
  - имеет другой sort key,
  - полезен для других способов сортировки/фильтрации в рамках одного partition key.

Оба типа индексов:

- автоматически поддерживаются DynamoDB,
- стоят **дополнительных ресурсов** (запись/место).

---

### Streams

DynamoDB Streams — это **лог изменений таблицы**:

- каждое изменение порождает запись в стриме,
- можно настроить Lambda‑функцию, которая будет:
  - обновлять кэши,
  - синхронизировать данные в другие системы,
  - считать дополнительные агрегаты.

Это удобный способ строить **event‑driven архитектуру** вокруг таблицы без ручного поллинга.

---

### TTL

TTL (Time To Live):

- можно задать атрибут (например, `expires_at`) как TTL‑поле,
- DynamoDB автоматически **удалит записи** после указанного времени,
- удобно для:
  - сессий,
  - одноразовых токенов,
  - временных данных.

Важно: удаление может происходить **с задержкой**, это не «строгий cron на секунду».

То есть TTL — это не про «жёсткий дедлайн до секунды», а про «эти данные можно удалить после такой‑то даты, как только у системы дойдут руки».

---

### On-demand vs provisioned

- **Provisioned**:
  - ты заранее указываешь, сколько `read capacity units` (RCU) и `write capacity units` (WCU) нужно,
  - дешево при **предсказуемой нагрузке**,
  - требует тюнинга (auto‑scaling).
- **On-demand**:
  - платишь за фактические запросы,
  - удобно для **непредсказуемой или маленькой** нагрузки,
  - при очень большой стабильной нагрузке может быть дороже.

Обычно:

- начинают с on-demand (быстро и просто),
- при росте и устоявшейся нагрузке переходят на provisioned с авто‑масштабированием.

---

### Типичные ошибки

- Проектировать таблицы как в реляционных БД (по сущностям), а не по паттернам доступа.
- Использовать `partition key` с очень малым числом значений (горячие ключи) → один раздел зашивается, остальные простаивают.
- Создавать слишком много GSI без учёта нагрузки и стоимости.
- Ожидать, что TTL удалит записи «точно в указанную секунду».

Дополнительно:

- не считать стоимость запросов и удивляться большому счёту за on‑demand при бурном росте продукта;
- «размазывать» данные по слишком большому количеству таблиц вместо осознанного **single table design** под паттерны доступа.

### Проверь себя

<details>
<summary>1. Почему важно выбирать <code>partition key</code> с хорошим распределением значений?</summary>

Потому что partition key определяет, на какие физические разделы/шарды попадут данные.  
Если у тебя мало уникальных значений (или одно очень популярное), нагрузка концентрируется на одном разделе, он становится «горячим», а остальная инфраструктура простаивает.  
Хороший partition key даёт **равномерное распределение** запросов и записей по разделам.

</details>

<details>
<summary>2. Чем GSI отличается от LSI с практической точки зрения?</summary>

- GSI:
  - может иметь **полностью свой PK и SK**,
  - масштабируется независимо от основной таблицы,
  - используется для запросов из других «проекций» (например, `email → user`).
- LSI:
  - разделяет **тот же partition key**, что и основная таблица,
  - отличается только sort key и набором проецируемых атрибутов,
  - подходит, когда нужно по‑разному сортировать/фильтровать данные в рамках одного partition key.

</details>

<details>
<summary>3. Чем опасно делать таблицы в DynamoDB «как в обычном SQL» — по сущностям, а не по паттернам доступа?</summary>

Потому что тогда многие реальные запросы **не смогут эффективно использовать partition/sort key** и будут требовать сканирования больших частей таблицы.  
В результате ты либо упрёшься в лимиты производительности и стоимости, либо начнёшь хаотично добавлять GSI/LSI, усложняя схему и рост расходов.  
Правильный путь — сначала сформулировать, **как именно приложение будет читать и писать данные**, и уже под эти паттерны проектировать ключи и структуру таблиц.

</details>
