## 168. Redis детали

### В этом разделе главное

- Redis — это **сервер структур данных в памяти**, а не просто key‑value.
- Главное — **подобрать правильный тип** (строка, список, множество, хэш, sorted set, stream, bitmap, HyperLogLog) под конкретный сценарий.
- Lua‑скрипты, транзакции (`MULTI/EXEC`) и pipeline позволяют **делать сложные операции минимальным числом сетевых «туда‑обратно»** и с нужной степенью атомарности.

### Простыми словами

Представь Redis как **огромный шкаф с ящиками**, где:

- в одном ящике лежит **список** (очередь задач),
- в другом — **множество** (уникальные пользователи за день),
- в третьем — **словарь** (профили пользователей),
- в четвёртом — **табло лидеров** (sorted set с баллами),
- а в пятом — **лентa событий** (stream).

Все ящики находятся **в памяти**, поэтому доступны очень быстро, а на диск пишутся снапшоты/журналы для восстановления.

---

### Обзор структур данных Redis

#### 1. Строки (`string`)

Базовый тип: под одним ключом хранится строка (байты).

- Применения:
  - кэш JSON‑ответов (`SET key value EX 60`),
  - счётчики (`INCR`, `INCRBY`),
  - флаги/lock'и (`SETNX` / `SET key value NX PX 10000`).

#### 2. Списки (`list`)

Упорядоченный список значений, можно пушить/попать с головы и хвоста.

- Команды: `LPUSH`, `RPUSH`, `LPOP`, `RPOP`, `BRPOP` (блокирующий pop).
- Применения:
  - простые очереди задач,
  - «последние N записей».

Недостаток: плохо масштабируются для **очень больших** очередей, и нет потребительских групп (для этого есть streams).

#### 3. Множества (`set`)

Набор **уникальных элементов** без порядка.

- Команды: `SADD`, `SREM`, `SISMEMBER`, `SMEMBERS`, `SUNION`, `SINTER`.
- Применения:
  - уникальные IP за день,
  - набор ролей/прав,
  - пересечения аудиторий (например, кто пользовался и продуктом A, и продуктом B).

#### 4. Хэши (`hash`)

Словарь полей внутри одного ключа.

- Команды: `HSET`, `HGET`, `HGETALL`, `HINCRBY`.
- Применения:
  - профиль пользователя (`user:123` с полями `name`, `email`, `balance`),
  - настройки конфигурации.

#### 5. Отсортированные множества (`sorted set`, `zset`)

Множество элементов с **числовым score**, упорядоченных по нему.

- Команды: `ZADD`, `ZRANGE`, `ZREVRANGE`, `ZRANGEBYSCORE`, `ZREMRANGEBYSCORE`.
- Применения:
  - рейтинги/лидерборды (score = очки/баллы),
  - очереди с приоритетом (score = приоритет или время),
  - хранение временных меток (score = timestamp).

#### 6. Streams

Структура для **упорядоченных по времени сообщений**:

- Команды: `XADD`, `XREAD`, `XGROUP`, `XREADGROUP`, `XACK`.
- Поддерживает **consumer groups**:
  - несколько воркеров читают одну и ту же ленту,
  - сообщения распределяются между ними,
  - есть подтверждения обработки.
- Применения:
  - простые очереди событий,
  - лог бизнес‑событий,
  - реализация «легковесного Kafka‑аналогa» для небольших систем.

#### 7. Bitmaps и HyperLogLog

- **Bitmaps**:
  - позволяют хранить флаги «1/0» в виде битового массива,
  - полезно для очень плотного хранения (например, «пользователь с ID N заходил сегодня или нет»).
- **HyperLogLog**:
  - приближённый подсчёт числа уникальных элементов,
  - очень компактно по памяти,
  - полезно для аналитики (уникальные пользователи в день/месяц).

Для закрепления можно связать каждую структуру с бытовым образом:

- `string` — **наклейка на коробке** («в этой коробке лежит JSON с профилем»);
- `list` — **очередь людей** в кассу, где новые становятся в начало или в конец;
- `set` — **множество бейджиков**: у каждого человека не может быть двух одинаковых бейджей;
- `hash` — **карточка в картотеке**, где внутри много полей;
- `sorted set` — **табло рейтинга** с очками;
- `stream` — **лента новостей**, куда новые события добавляются только в конец.

### Lua scripting

Lua‑скрипты позволяют:

- выполнять **несколько команд как одну атомарную операцию**,
- избежать лишних сетевых round‑trip'ов (логика выполняется на сервере Redis),
- гарантировать, что между шагами никто не «вклинится».

Простой пример — поп из одной очереди и push в другую (реализация надёжной очереди):

```lua
-- Пример идеи (псевдо-Lua для Redis):
-- 1. Взять элемент из main_queue
-- 2. Переложить его в processing_queue
-- 3. Вернуть элемент клиенту
```

В реальности ты пишешь Lua‑скрипт и вызываешь его через `EVAL`/`EVALSHA`.

Важно:

- скрипт должен быть **детерминированным** и достаточно быстрым — Redis однопоточен, долгий скрипт блокирует всё.

Полезный способ думать о Lua‑скриптах:

- без скрипта ты делаешь много мелких шагов:
  - клиент: «дай элемент» → Redis даёт → клиент: «переложи сюда» → Redis кладёт → клиент: «поставь ещё один ключ» → …
- с Lua‑скриптом ты говоришь Redis: **«сделай *вот этот сценарий* целиком сам и верни результат»**.

Если представить Redis как кассира в магазине:

- без Lua ты подходишь по десять раз, каждый раз с маленькой просьбой;
- с Lua ты один раз подходишь и говоришь: «пожалуйста, возьми эти товары, посчитай сумму, примени скидку и скажи финальный чек».

### Pub/Sub

**Pub/Sub (publish/subscribe)** — это модель, в которой:

- один или несколько отправителей **публикуют сообщения** в канал (`PUBLISH channel payload`);
- подписчики заранее **подписываются на канал** (`SUBSCRIBE channel`) и получают все сообщения, которые туда прилетают.

Ключевые особенности:

- **нет хранения сообщений** — если подписчик был отключён в момент публикации, он не получит это сообщение позже;
- всё работает в **реальном времени**: пока клиент подписан, он видит поток событий;
- один PUBLISH доставляет сообщение **всем** подписчикам канала.

Типичные сценарии:

- простые уведомления между сервисами (сигналы, что «что‑то произошло»);
- уведомления для WebSocket‑соединений (чат, онлайн‑события);
- отслеживание изменений без тяжёлых очередей.

Важно не ждать от Pub/Sub поведения «очереди с гарантированной доставкой» — для этого лучше подходят streams или внешние брокеры (Kafka, RabbitMQ и т.п.).

### Redis Modules

Redis Modules — это **расширения**, которые добавляют в Redis новые типы данных и команды.

Примеры популярных модулей:

- **RediSearch** — полнотекстовый поиск и индексация по полям;
- **RedisGraph** — графовая БД поверх Redis;
- **RedisTimeSeries** — удобные операции над временными рядами;
- другие модули для JSON, Bloom‑фильтров и т.д.

Как о них думать:

- базовый Redis даёт тебе **универсальные структуры данных** (строки, списки, множества и т.д.);
- модули добавляют **готовые специализированные решения**:
  - поиск по тексту,
  - графовые запросы,
  - работу с временными рядами.

Использование модулей обычно требует:

- загрузить модуль при запуске Redis (или через команду `MODULE LOAD`);
- изучить **дополнительный набор команд** (они описаны в документации каждого модуля).

Практический подход:

- сначала реши, можно ли обойтись базовыми структурами Redis;
- если ты понимаешь, что тебе нужен именно **поиск / граф / time‑series** и не хочется поднимать отдельную БД — посмотри, не решает ли задачу подходящий модуль.

### Транзакции и pipeline

- `MULTI/EXEC`:
  - группирует несколько команд в один блок,
  - гарантирует, что либо все выполнятся, либо ни одна (с некоторыми нюансами).
- `WATCH`:
  - позволяет реализовать **оптимистичную блокировку**: следить за ключами и отменять транзакцию, если кто‑то их изменил.
- Pipeline:
  - просто шлёшь **несколько команд подряд** без ожидания ответа,
  - Redis отвечает все результаты одной пачкой,
  - нет атомарности, но есть экономия на сетевых задержках.

Если путаешься:

- **pipeline** — про **скорость общения по сети** (меньше «туда‑обратно»),
- **MULTI/EXEC** — про **атомарность набора команд**,
- **Lua** — про **атомарную логику**, которую удобно выразить скриптом, а не набором простых команд.

### Типичные сценарии использования Redis

- **Кэш**:
  - ключ = hash запроса/параметров,
  - значение = сериализованный ответ (JSON),
  - `EX`/`TTL` задаёт время жизни.
- **Хранение сессий**:
  - ключ = `session:<token>`,
  - значение = JSON/хэш с данными сессии,
  - TTL равен сроку жизни сессии.
- **Счётчики и rate limiting**:
  - `INCR`/`INCRBY` по ключам вида `rate:<userId>:<minute>`,
  - TTL = 1 минута,
  - если счётчик превышает порог — блокируем.
- **Лидеры/рейтинги**:
  - `ZADD leaderboard score userId`,
  - `ZREVRANGE leaderboard 0 9 WITHSCORES` — топ‑10.

Полезное упражнение:

1. Возьми один реальный сценарий (например, «ограничить пользователя 100 запросами в минуту»).
2. Попробуй придумать решение **только на строках** (`INCR`, TTL).
3. Потом спроси себя: «а если мне ещё нужно топ‑N пользователей с наибольшим количеством запросов?» — здесь уже напрашивается `sorted set`.
4. Сравни оба решения и запиши, какие команды и типы данных ты выбрал и почему.

---

### Типичные ошибки

- Хранить **огромные JSON‑объекты в строках** и каждый раз перезаписывать их целиком — неэффективно, лучше использовать хэши/структуры.
- Использовать Redis как **единственное хранилище критичных данных**, без бэкапов и репликации.
- Писать тяжёлые Lua‑скрипты, которые блокируют сервер.
- Игнорировать TTL, и в итоге ключи никогда не протухают, память растёт бесконтрольно.

Ещё из практики:

- использовать один и тот же Redis и для **критичного кэша**, и для «игрушечных» экспериментов — один неосторожный `FLUSHALL` может убить весь прод;
- хранить в Redis **огромные ключи/значения** (сотни килобайт и больше), превращая быструю in‑memory БД в медленный файловый сервер;
- не настраивать мониторинг объёма памяти и просто «надеяться, что поместится».

### Проверь себя

<details>
<summary>1. Когда лучше использовать <code>sorted set</code>, а не обычный <code>set</code>?</summary>

Когда тебе важен **порядок по числовому признаку**:

- рейтинги (score = очки),
- сортировка по времени (score = timestamp),
- очереди с приоритетом (score = приоритет).

Обычный `set` хранит только «есть/нет» без порядка и score.

</details>

<details>
<summary>2. Зачем вообще нужны Lua‑скрипты, если есть <code>MULTI/EXEC</code> и pipeline?</summary>

Lua‑скрипты позволяют:

- выполнить сложную последовательность действий **целиком на стороне Redis**,
- сделать это **атомарно** (как один «кусок логики»),
- сильно сократить количество сетевых запросов.

`MULTI/EXEC` группирует команды, но не умеет в условные конструкции и циклы так гибко, как Lua; pipeline вообще не даёт атомарности, только экономит на сетевых задержках.

</details>

<details>
<summary>3. Почему Pub/Sub в Redis не заменяет очереди и streams?</summary>

Потому что Pub/Sub **не хранит сообщения**: если подписчик был офлайн в момент публикации, он никогда не получит эти данные.  
Streams и внешние очереди (Kafka, RabbitMQ) как раз решают задачу **гарантированной доставки и повторного чтения** сообщений.  
Поэтому Pub/Sub хорош для онлайн‑событий и уведомлений, но не подходит, когда важна надёжная обработка каждого сообщения.

</details>

#### Если прочитал и не понял

Не пытайся сразу запомнить все типы и команды. Сделай так:

1. Выбери **один тип данных** (например, `list`) и придумай маленький проект: очередь задач, список последних логов и т.п.
2. Реализуй это **только с этим типом**: потрогай команды руками (`LPUSH`, `RPOP`, `LRANGE`).
3. Затем добавь второй тип (`set` или `sorted set`) для новой задачи (лидерборд, уникальные пользователи).
4. Каждый раз, когда видишь новую команду, спрашивай себя: «а в каком житейском примере она пригодится?».

Через пару таких мини‑проектов Redis перестаёт быть «набором страшных слов» и превращается в набор знакомых «ящиков» с понятной логикой.
