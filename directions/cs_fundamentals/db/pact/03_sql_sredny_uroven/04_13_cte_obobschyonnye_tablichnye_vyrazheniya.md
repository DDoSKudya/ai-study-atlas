## 13. CTE (обобщённые табличные выражения)

**Цель раздела.**  
Научиться писать запросы с WITH (CTE): именованные подзапросы в начале запроса, несколько CTE подряд, рекурсивные CTE для иерархий (деревья, графы), ограничение глубины и обнаружение циклов. Понимать в PostgreSQL разницу между MATERIALIZED и NOT MATERIALIZED.

**Простыми словами: о чём этот раздел.**  
Здесь мы учимся давать **имя подзапросу** в начале запроса (WITH name AS (SELECT ...)) и обращаться к нему по имени как к таблице — так сложный запрос легче читать и не копировать. Несколько CTE подряд: первый считает одно, второй может использовать первый по имени (цепочка). Рекурсивный CTE — «шаг за шагом» строить результат: база (с чего начать) + шаг (как из уже полученного получить ещё), пока новые строки не перестанут появляться; удобно для дерева (сотрудник → руководитель → все подчинённые). В конце — когда в PostgreSQL указывать MATERIALIZED (посчитать один раз) или NOT MATERIALIZED (разрешить встраивание в план).

**Теоретическая основа.**  
CTE — это способ дать имя подзапросу и использовать его в основном запросе (и в других CTE) как таблицу. С точки зрения реляционной алгебры это переименование и композиция выражений. Рекурсивный CTE задаёт фиксированную точку: начальное множество (базовый член) плюс правило порождения новых строк из уже полученных (рекурсивный член), что соответствует обходу графа или дерева.

**Простыми словами (CTE).**  
CTE — это «временное имя» для результата запроса. Ты пишешь в начале: «пусть order_totals — это результат такого-то SELECT», а дальше в основном запросе обращаешься к order_totals как к таблице. Так сложный подзапрос не нужно копировать и легче читать.

**Простыми словами (рекурсивный CTE).**  
Рекурсивный CTE — это «шаг за шагом» строить результат: сначала базовый запрос даёт несколько строк (например, корень дерева), потом рекурсивный запрос добавляет новые строки, ссылаясь на уже полученные (например, детей по parent_id), и так повторяется, пока новые строки не перестанут появляться. В итоге получается обход дерева или графа.

### 13.1. WITH — обычный CTE

**Термины**

- **CTE (Common Table Expression, обобщённое табличное выражение)** — подзапрос, которому в начале запроса дано имя; далее по имени можно обращаться как к таблице.
- **WITH name AS (SELECT ...)** — объявление CTE.

**Синтаксис**

```sql
WITH cte_name AS (
    SELECT ...
)
SELECT * FROM cte_name [JOIN ...];
```

**Пример**

```sql
WITH order_totals AS (
    SELECT user_id, SUM(total) AS total, COUNT(*) AS cnt
    FROM orders
    GROUP BY user_id
)
SELECT u.name, ot.total, ot.cnt
FROM users u
JOIN order_totals ot ON ot.user_id = u.id
WHERE ot.total > 1000;
```

**Разбор этого запроса по шагам (что за что отвечает).**  
- **CTE (WITH order_totals AS ...):** сначала вычисляется подзапрос — по каждому user_id считаются сумма заказов (SUM(total)) и количество заказов (COUNT(*)). Результат — временная «таблица» с столбцами user_id, total, cnt (одна строка на пользователя). Этой «таблице» дано имя order_totals.  
- **Основной запрос (SELECT ... FROM users u JOIN order_totals ot ...):** берём таблицу users, соединяем с order_totals по user_id (чтобы к каждому пользователю подставить его сумму и количество заказов), оставляем только тех, у кого сумма больше 1000 (WHERE ot.total > 1000), выводим имя, сумму и количество.  
Итого: CTE считает «сумма и количество заказов по пользователям»; основной запрос берёт пользователей, подставляет эти данные и фильтрует по сумме. Без CTE пришлось бы писать подзапрос в FROM или повторять его; с CTE логика разбита на два понятных шага.

**Запомните**

- CTE улучшает читаемость: сложный подзапрос выносится в начало с именем.
- CTE можно использовать в основном запросе несколько раз (в разных JOIN); в части СУБД план может вычислять его один раз или повторно — зависит от оптимизатора и MATERIALIZED.

**Вопросы для самопроверки (13.1)**  
1. Где в запросе объявляется CTE и как к нему обращаться?  
   <details><summary>Ответ</summary> CTE объявляется **в начале запроса** после **WITH** (WITH name AS (SELECT ...)). В основном запросе к нему обращаются **по имени** как к таблице (FROM name, JOIN name ON ...). </details>  
2. Чем CTE отличается от подзапроса в FROM по записи?  
   <details><summary>Ответ</summary> CTE выносится **в начало** запроса с именем; основной запрос короче и читается сверху вниз. Подзапрос в FROM пишется **на месте** использования в скобках с алиасом. По смыслу оба задают временную «таблицу». </details>  
3. Можно ли использовать один CTE несколько раз в одном запросе?  
   <details><summary>Ответ</summary> **Да.** По имени CTE можно обращаться в нескольких местах (несколько JOIN, подзапросы). Будет ли он вычисляться один раз или несколько — зависит от СУБД и MATERIALIZED/NOT MATERIALIZED. </details>

**Проверь себя перед переходом к 13.2.**  
Ответь вслух: (1) Где в запросе объявляется CTE — в начале, в середине или в конце? (2) Зачем давать подзапросу имя (AS order_totals)? Если ответил «в начале, после WITH» и «чтобы обращаться к результату по имени как к таблице» — можно идти дальше.

### 13.2. Несколько CTE

Несколько CTE перечисляются через запятую; следующие могут ссылаться на предыдущие:

```sql
WITH
    order_totals AS (SELECT user_id, SUM(total) AS total FROM orders GROUP BY user_id),
    top_users AS (SELECT user_id FROM order_totals ORDER BY total DESC LIMIT 10)
SELECT u.* FROM users u
JOIN top_users t ON t.user_id = u.id;
```

**Простыми словами.**  
Сначала объявляешь первый CTE (order_totals) — «сумма заказов по пользователям». Второй CTE (top_users) может **использовать первый по имени**: «возьми из order_totals десять user_id с наибольшей суммой». Основной запрос использует уже top_users (и при необходимости order_totals). Порядок объявления важен: top_users не может стоять перед order_totals, потому что он от него зависит. Цепочка: order_totals → top_users → основной SELECT.

**Разбор по шагам (что за что отвечает).**  
- **order_totals:** подзапрос считает по каждому user_id сумму заказов (SUM(total)), одна строка на пользователя. Результат — «таблица» с полями user_id, total.  
- **top_users:** подзапрос берёт **из order_totals** (не из orders!) строки, сортирует по total по убыванию и оставляет первые 10. Результат — 10 user_id (топ-10 по сумме заказов).  
- **Основной запрос:** соединяем users с top_users по user_id и выводим все столбцы пользователей — в результате только те 10 пользователей, которые вошли в top_users.  
Итого: первый CTE считает агрегаты, второй отбирает по ним топ, основной запрос подставляет полные данные пользователей. Без цепочки CTE пришлось бы писать вложенные подзапросы в FROM — читать сложнее.

**Запомните**

- WITH a AS (...), b AS (...) — порядок важен: b может использовать a.

**Вопросы для самопроверки (13.2)**  
1. Может ли второй CTE в цепочке ссылаться на первый? В каком порядке их объявлять?  
   <details><summary>Ответ</summary> **Да.** Второй CTE может использовать первый по имени (например, SELECT ... FROM first_cte). Объявлять нужно так: **сначала первый**, потом второй — порядок объявления важен, зависимый CTE должен идти после того, от кого зависит. </details>  
2. Зачем писать несколько CTE подряд вместо одного запроса с вложенными подзапросами?  
   <details><summary>Ответ</summary> **Читаемость**: логика разбита на шаги с понятными именами (сначала агрегаты, потом отбор, потом основной запрос). Легче отлаживать и менять по частям. </details>  
3. Можно ли в основном запросе использовать и первый, и второй CTE?  
   <details><summary>Ответ</summary> **Да.** Основной запрос может обращаться к любому из объявленных CTE по имени (JOIN с любым из них, WHERE по их столбцам и т.д.). </details>

### 13.3. Рекурсивный CTE

**Синтаксис**

```sql
WITH RECURSIVE name AS (
    base_query
    UNION ALL
    recursive_query
)
SELECT * FROM name;
```

- **Базовый член (anchor)** — первый SELECT: начальное множество строк.
- **Рекурсивный член** — второй SELECT: он может ссылаться на имя CTE и порождать новые строки из уже полученных. СУБД повторяет рекурсивный член, пока он возвращает строки.

**Пример: числа от 1 до 5**

```sql
WITH RECURSIVE nums(n) AS (
    SELECT 1
    UNION ALL
    SELECT n + 1 FROM nums WHERE n < 5
)
SELECT * FROM nums;
```

**Разбор по шагам (чтобы точно понять, как работает рекурсия).**  
СУБД выполняет рекурсивный CTE так:  
- **Шаг 0 (база):** первый SELECT даёт одну строку: n = 1. В nums пока лежит только (1).  
- **Шаг 1:** рекурсивный запрос `SELECT n + 1 FROM nums WHERE n < 5` видит в nums строку (1). Условие 1 < 5 истинно. Он возвращает 1 + 1 = 2. В nums теперь (1) и (2).  
- **Шаг 2:** рекурсивный запрос снова выполняется над текущим содержимым nums. Он возвращает 2 + 1 = 3 (для строки n=2). В nums теперь (1), (2), (3).  
- **Шаг 3:** возвращается 3 + 1 = 4. В nums: (1), (2), (3), (4).  
- **Шаг 4:** возвращается 4 + 1 = 5. В nums: (1), (2), (3), (4), (5).  
- **Шаг 5:** рекурсивный запрос для n=5 даёт 5 + 1 = 6, но условие WHERE n < 5 отсекает строку с n=5 — для неё n < 5 ложно. Для остальных строк мы уже всё добавили. В итоге рекурсивный запрос не возвращает новых строк (или возвращает пусто). СУБД останавливается.  
Итоговый результат: 1, 2, 3, 4, 5. Условие `n < 5` в рекурсивном члене — это **условие выхода**: без него мы бы бесконечно добавляли 6, 7, 8, …

**Как запомнить рекурсивный CTE.**  
Два запроса в WITH RECURSIVE: **первый** — «с чего начать» (anchor, база): например, одна строка — корень. **Второй** — «как из уже полученного получить ещё» (рекурсивный шаг): он обращается к имени CTE и к основной таблице (например, «взять сотрудников, у которых manager_id в уже полученных»). СУБД повторяет второй запрос, пока он возвращает строки. Обязательно условие выхода в рекурсивном члене (WHERE), иначе цикл не остановится.

**Запомните**

- WITH RECURSIVE: первый запрос — база, второй — шаг рекурсии с обращением к имени CTE.
- Обязательно условие выхода (WHERE), иначе бесконечный цикл (или ограничение глубины в СУБД).
- Шпаргалка: «база + шаг из уже полученного; не забудь условие выхода».

**Вопросы для самопроверки (13.3)**  
1. Что даёт первый SELECT в WITH RECURSIVE и что — второй?  
   <details><summary>Ответ</summary> **Первый (базовый)** — **начальное множество** строк (anchor): «с чего начать». **Второй (рекурсивный)** — порождает **новые строки** из уже полученных, обращаясь к имени CTE; СУБД повторяет его, пока он возвращает строки. </details>  
2. Зачем в рекурсивном члене обязательно условие в WHERE (например, n < 5)?  
   <details><summary>Ответ</summary> Это **условие выхода**: без него рекурсивный запрос мог бы бесконечно добавлять новые строки (6, 7, 8, …). Условие ограничивает, когда перестать выполнять шаг рекурсии. </details>  
3. Как СУБД выполняет рекурсивный CTE по шагам?  
   <details><summary>Ответ</summary> Сначала выполняется **базовый** запрос — результат попадает в CTE. Затем **рекурсивный** запрос выполняется над текущим содержимым CTE; его результат добавляется. Шаг повторяется, пока рекурсивный запрос **не вернёт пусто**. Итог — объединение всех шагов. </details>

**Проверь себя перед переходом к 13.4.**  
Ответь вслух: (1) В WITH RECURSIVE два запроса: что даёт первый (базовый) и что делает второй (рекурсивный)? (2) Почему в рекурсивном члене обязательно условие в WHERE (например, n < 5)? Если ответил «первый — с чего начать, второй — как из уже полученного получить ещё» и «чтобы рекурсия остановилась (условие выхода), иначе бесконечный цикл» — можно идти к иерархиям (дерево).

### 13.4. Иерархии (дерево)

Типичный пример: таблица сотрудников с manager_id. Вывести всех подчинённых заданного руководителя (поддерево):

```sql
WITH RECURSIVE tree AS (
    SELECT id, name, manager_id, 1 AS level
    FROM employees
    WHERE id = 5
    UNION ALL
    SELECT e.id, e.name, e.manager_id, t.level + 1
    FROM employees e
    JOIN tree t ON e.manager_id = t.id
)
SELECT * FROM tree ORDER BY level, id;
```

**Разбор по шагам (дерево сотрудников).**  
Допустим employees: (id=5, name='Директор', manager_id=NULL), (id=10, name='Менеджер А', manager_id=5), (id=11, name='Менеджер Б', manager_id=5), (id=20, name='Сотрудник', manager_id=10). Базовый запрос: WHERE id = 5 → в tree попадает одна строка: (5, Директор, NULL, level=1). Рекурсивный запрос: «взять из employees тех, у кого manager_id совпадает с id из tree».  
- **Итерация 1:** в tree только id=5. Из employees выбираем строки с manager_id=5 → (10, Менеджер А, 5, level=2) и (11, Менеджер Б, 5, level=2). Добавляем в tree.  
- **Итерация 2:** в tree теперь id 5, 10, 11. Ищем employees с manager_id IN (10, 11) → (20, Сотрудник, 10, level=3). Добавляем.  
- **Итерация 3:** ищем с manager_id=20 — таких нет. Рекурсивный запрос возвращает пусто. Останавливаемся.  
Итог: все подчинённые узла 5 (включая косвенных) с уровнями 1, 2, 3.

**Ограничение глубины и циклы — простыми словами.**  
- **Глубина:** рекурсия добавляет уровень за уровнем (level 1, 2, 3, …). Если дерево очень глубокое, можно ограничить: в рекурсивном члене добавить `AND t.level < 10` — не брать сотрудников глубже 10-го уровня. Иначе запрос может «уйти» слишком далеко по иерархии.  
- **Циклы:** в таблице может быть ошибка данных: A → B → C → A (менеджер в итоге «подчинённый» своего подчинённого). Без проверки рекурсия будет бесконечно добавлять одних и тех же. Решение: в рекурсивном члене хранить путь (массив уже пройденных id) и отбрасывать строки, у которых id уже в пути — так повторно не заходим в один и тот же узел.

**Пример: ограничение глубины.**  
Добавляем в рекурсивный член условие «не глубже 5 уровней»:

```sql
WITH RECURSIVE tree AS (
    SELECT id, name, manager_id, 1 AS level
    FROM employees WHERE id = 5
    UNION ALL
    SELECT e.id, e.name, e.manager_id, t.level + 1
    FROM employees e
    JOIN tree t ON e.manager_id = t.id
    WHERE t.level < 5
)
SELECT * FROM tree ORDER BY level, id;
```

Условие `t.level < 5` останавливает рекурсию: сотрудников с level 5 и выше не добавляем. Итог — подчинённые не глубже 4 уровней ниже корня.

**Пример: обнаружение циклов (PostgreSQL).**  
Накапливаем путь (массив id) и отбрасываем строки, чей id уже в пути:

```sql
WITH RECURSIVE tree AS (
    SELECT id, name, manager_id, 1 AS level, ARRAY[id] AS path
    FROM employees WHERE id = 5
    UNION ALL
    SELECT e.id, e.name, e.manager_id, t.level + 1, t.path || e.id
    FROM employees e
    JOIN tree t ON e.manager_id = t.id
    WHERE NOT (e.id = ANY(t.path))
)
SELECT * FROM tree ORDER BY level, id;
```

В базовом члене `path` — массив из одного id (корень). В рекурсивном: `t.path || e.id` — дописываем id текущего сотрудника в путь. Условие `NOT (e.id = ANY(t.path))` отбрасывает строки, у которых id уже встречался в пути — так мы не заходим в узел повторно и рекурсия не зацикливается при ошибках в данных (A → B → C → A).

**Запомните**

- Рекурсивный CTE — базовый запрос (корень) + рекурсивный (дети по связи parent_id).
- Нужны условие выхода и при необходимости проверка на циклы.

**Вопросы для самопроверки (13.4)**  
1. В рекурсивном CTE для дерева сотрудников: что выбирает базовый запрос — одну строку или много?  
   <details><summary>Ответ</summary> Обычно **одну строку** (или несколько) — **корень** поддерева (например, WHERE id = 5 — руководитель, с которого начинаем обход). Дальше рекурсивный шаг добавляет «детей» по manager_id. </details>  
2. Что делает рекурсивный член в обходе дерева по manager_id?  
   <details><summary>Ответ</summary> Выбирает строки из таблицы сотрудников, у которых **manager_id** совпадает с **id** из уже полученных в CTE (т.е. **дети** уже добавленных узлов). Так уровень за уровнем добавляются все подчинённые. </details>  
3. Как ограничить глубину обхода (например, не глубже 5 уровней)?  
   <details><summary>Ответ</summary> В рекурсивном члене добавить условие по **уровню** (level): например, **WHERE t.level < 5**. Тогда строки с level >= 5 не добавляются, рекурсия останавливается. </details>

**Проверь себя перед переходом к 13.5.**  
Ответь вслух: (1) В рекурсивном CTE для дерева сотрудников: что даёт базовый запрос — одну строку или много? (2) Что делает рекурсивный член — добавляет «детей» по manager_id или «родителей»? Если ответил «одну строку (корень)» и «добавляет детей (тех, у кого manager_id в уже полученных)» — можно идти к MATERIALIZED.

### 13.5. MATERIALIZED / NOT MATERIALIZED

В PostgreSQL 12+ можно указать для CTE:
- **NOT MATERIALIZED** (по умолчанию в PG 12+) — оптимизатор может «встроить» CTE в основной запрос (как подзапрос).
- **MATERIALIZED** — CTE вычисляется один раз и результат материализуется (как временная таблица).

**Простыми словами: когда MATERIALIZED, когда NOT MATERIALIZED.**  
По умолчанию (NOT MATERIALIZED) СУБД может «развернуть» CTE в план основного запроса — как будто подзапрос встроен туда, где используется имя CTE; тогда CTE может выполняться несколько раз, если он используется в нескольких местах, или один раз — как решит оптимизатор. **MATERIALIZED** значит: «сначала посчитай этот CTE целиком, сохрани результат во временную таблицу, потом используй его». Если CTE тяжёлый и используется **несколько раз** в запросе — MATERIALIZED часто выгоднее (посчитали один раз, много раз читаем). Если CTE лёгкий или оптимизатор умеет хорошо встраивать — NOT MATERIALIZED может дать быстрее план. В сомнении можно проверить EXPLAIN и сравнить.

**Запомните**

- MATERIALIZED — посчитать CTE один раз; NOT MATERIALIZED — разрешить встраивание в план.
- В других СУБД поведение может отличаться (например, всегда материализация).

**Вопросы для самопроверки (13.5)**  
1. Чем MATERIALIZED отличается от NOT MATERIALIZED для CTE в PostgreSQL?  
   <details><summary>Ответ</summary> **MATERIALIZED** — CTE вычисляется **один раз**, результат сохраняется (как временная таблица), затем используется. **NOT MATERIALIZED** — оптимизатор может **встроить** CTE в план (как подзапрос); он может выполняться несколько раз или один — как решит оптимизатор. </details>  
2. Когда выгоднее указать MATERIALIZED?  
   <details><summary>Ответ</summary> Когда CTE **тяжёлый** и используется **несколько раз** в запросе: посчитать один раз и многократно читать выгоднее, чем повторять вычисление. При лёгком CTE или однократном использовании NOT MATERIALIZED может дать быстрее план. </details>  
3. Гарантирует ли NOT MATERIALIZED, что CTE выполнится один раз?  
   <details><summary>Ответ</summary> **Нет.** NOT MATERIALIZED разрешает оптимизатору встраивать CTE в план; он может выполнить его один раз или несколько — по усмотрению оптимизатора. Для гарантии однократного вычисления в PostgreSQL используют MATERIALIZED. </details>

**Резюме раздела 13 в одном абзаце.**  
CTE (WITH name AS (SELECT ...)) — именованный подзапрос в начале запроса; по имени к нему обращаются как к таблице. Несколько CTE перечисляются через запятую; следующие могут ссылаться на предыдущие. Рекурсивный CTE (WITH RECURSIVE): первый запрос — база (начальные строки), второй — рекурсивный шаг (ссылается на имя CTE и порождает новые строки); СУБД повторяет шаг, пока есть новые строки. Обязательно условие выхода в рекурсивном члене. Удобно для иерархий (дерево по parent_id). В PostgreSQL: MATERIALIZED — посчитать CTE один раз; NOT MATERIALIZED — разрешить встраивание в план.
