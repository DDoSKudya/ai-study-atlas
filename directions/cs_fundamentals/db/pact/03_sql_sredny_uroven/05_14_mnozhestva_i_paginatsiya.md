## 14. Множества и пагинация

**Цель раздела.**  
Научиться объединять результаты двух запросов (UNION, UNION ALL), находить пересечение (INTERSECT) и разность (EXCEPT), соблюдать требования к столбцам и порядку. Разобраться с пагинацией: недостатки OFFSET и преимущества keyset-пагинации.

**Простыми словами: о чём этот раздел.**  
Здесь мы учимся **склеивать два запроса в один**: UNION — объединить списки и убрать дубликаты; UNION ALL — объединить без удаления дублей (обычно быстрее). INTERSECT — оставить только то, что есть в **обоих** результатах; EXCEPT — оставить только то, что есть в **первом** и чего **нет во втором**. Оба запроса должны возвращать одинаковое число столбцов с совместимыми типами. Пагинация: OFFSET при больших номерах страниц медленный (СУБД считает и отбрасывает много строк впустую); keyset-пагинация (WHERE id > last_id LIMIT n) даёт стабильную скорость — передаём «последний увиденный ключ», а не номер страницы.

**Теоретическая основа.**  
В реляционной алгебре операции над множествами: объединение (∪), пересечение (∩), разность (−). В SQL UNION соответствует объединению (с опцией убрать дубликаты), INTERSECT — пересечению, EXCEPT — разности. Порядок строк в результате не определён, если не задан ORDER BY в конце всего запроса.

---

### 14.1. UNION и UNION ALL

**UNION** — объединить результаты двух запросов. По умолчанию дубликаты строк убираются (как в множестве). **UNION ALL** — объединить без удаления дубликатов; обычно быстрее, если дубликаты допустимы.

Требования: оба запроса должны возвращать **одинаковое число столбцов**, и типы столбцов должны быть **совместимы** (по позиции).

**Простыми словами.**  
UNION = «склеить два списка в один; одинаковые строки оставить один раз». UNION ALL = «склеить два списка в один; дубликаты не убирать». INTERSECT = «оставить только то, что есть в обоих списках». EXCEPT = «оставить только то, что есть в первом списке и чего нет во втором».

**Как запомнить.**  
UNION — как объединение множеств: все элементы из A и из B, без повторов. UNION ALL — все элементы подряд, с повторами. INTERSECT — пересечение: только общие. EXCEPT — разность: только из первого минус то, что во втором.

**Мини-пример «до и после» (UNION vs UNION ALL).**  
Запрос A возвращает: (1, 'Аня'), (2, 'Боря'). Запрос B возвращает: (2, 'Боря'), (3, 'Вася').  
- **UNION** (объединить и убрать дубликаты): результат — (1, 'Аня'), (2, 'Боря'), (3, 'Вася') — три строки; (2, 'Боря') встретился в обоих запросах, но в результате только один раз.  
- **UNION ALL** (объединить без удаления дублей): результат — (1, 'Аня'), (2, 'Боря'), (2, 'Боря'), (3, 'Вася') — четыре строки; дубликат (2, 'Боря') сохранён.

**Пример**

```sql
SELECT id, name FROM users WHERE role = 'admin'
UNION
SELECT id, name FROM users WHERE last_login > '2024-01-01';
```

Если нужны все строки без удаления дублей:

```sql
SELECT id, name FROM users WHERE role = 'admin'
UNION ALL
SELECT id, name FROM users WHERE last_login > '2024-01-01';
```

**Типичная ошибка.**  
Написать разное число столбцов в двух запросах под UNION — СУБД выдаст ошибку («each SELECT must have the same number of columns»). Столбцы сравниваются **по позиции** (первый с первым, второй со вторым), а не по имени; типы должны быть совместимы (например, INTEGER и BIGINT — ок, INTEGER и TEXT — ошибка).

**Запомните**

- UNION убирает дубликаты; UNION ALL — нет; UNION ALL обычно быстрее.
- Одинаковое число и совместимые типы столбцов по позиции.

**Вопросы для самопроверки (14.1)**  
1. Чем UNION отличается от UNION ALL по дубликатам и по скорости?  
   <details><summary>Ответ</summary> **UNION** убирает **дубликаты** строк в объединённом результате (как множество). **UNION ALL** дубликаты **не убирает** — все строки подряд; обычно **быстрее**, т.к. не нужна проверка на повторы. </details>  
2. Сколько столбцов должно быть в первом и во втором запросе под UNION? Как сравниваются столбцы?  
   <details><summary>Ответ</summary> **Одинаковое число** столбцов. Столбцы сопоставляются **по позиции** (первый с первым, второй со вторым); типы должны быть **совместимы** (например, INTEGER и BIGINT — ок). Имена столбцов берутся из первого запроса. </details>  
3. Что будет, если в первом запросе 2 столбца, во втором — 3?  
   <details><summary>Ответ</summary> СУБД выдаст **ошибку** (например, «each SELECT must have the same number of columns»). Нужно привести к одному числу (добавить NULL AS col3 во второй запрос или убрать лишний столбец). </details>

**Проверь себя перед переходом к 14.2.**  
Ответь вслух: (1) Чем UNION отличается от UNION ALL по дубликатам? (2) Сколько столбцов должно быть в первом и во втором запросе под UNION? Если ответил «UNION убирает дубликаты, UNION ALL нет» и «одинаковое число столбцов» — можно идти дальше.

### 14.2. INTERSECT и EXCEPT

**INTERSECT** — строки, которые есть в **обоих** результатах (пересечение). **EXCEPT** — строки первого запроса, которых **нет** во втором (разность).

**Мини-пример «до и после» (INTERSECT и EXCEPT).**  
Первый запрос (админы): id = 1, 2, 5. Второй запрос (заходили после даты): user_id = 2, 5, 7.  
- **INTERSECT:** в обоих списках есть 2 и 5. Результат: 2, 5 (только общие).  
- **EXCEPT** (первый минус второй): из первого (1, 2, 5) убираем то, что есть во втором (2, 5). Остаётся: 1 (только в первом, нет во втором).

**Пример**

```sql
-- Пользователи, которые и админы, и заходили после даты
SELECT id FROM users WHERE role = 'admin'
INTERSECT
SELECT user_id FROM logins WHERE login_at > '2024-01-01';

-- Пользователи-админы, у которых не было захода после даты
SELECT id FROM users WHERE role = 'admin'
EXCEPT
SELECT user_id FROM logins WHERE login_at > '2024-01-01';
```

**Сводка: разница между UNION, UNION ALL, INTERSECT, EXCEPT (чтобы не путать).**

| Операция | Что делаем | Дубликаты в результате |
|----------|------------|------------------------|
| **UNION** | Объединяем два запроса в один | Убираем (как в множестве) |
| **UNION ALL** | Объединяем два запроса в один | Не убираем (быстрее) |
| **INTERSECT** | Оставляем только то, что есть в **обоих** | Убираем |
| **EXCEPT** | Оставляем только то, что есть в **первом** и чего **нет во втором** | Убираем |

Памятка: UNION = «всё из A и из B»; INTERSECT = «только общее»; EXCEPT = «только из первого минус второе».

**Как запомнить (INTERSECT и EXCEPT).**  
INTERSECT = «пересечение» = только то, что есть **в обоих** списках. EXCEPT = «вычесть второе из первого» = только то, что есть в первом запросе и **чего нет** во втором. Фразы: «INTERSECT — общие»; «EXCEPT — первый минус второй». Оба убирают дубликаты в результате (как UNION без ALL).

**Запомните**

- INTERSECT — общие строки; EXCEPT — только в первом, не во втором.
- Дубликаты в результате убираются (как в UNION без ALL).

**Вопросы для самопроверки (14.2)**  
1. Что оставляет в результате INTERSECT — «всё из обоих запросов» или «только общие строки»?  
   <details><summary>Ответ</summary> **Только общие строки**: те, которые есть **и в первом, и во втором** результате (пересечение множеств). Дубликаты в результате убираются. </details>  
2. Что делает EXCEPT: «первый минус второй» или «общие строки»?  
   <details><summary>Ответ</summary> **Первый минус второй**: в результате только строки, которые есть в **первом** запросе и **отсутствуют** во втором. Это разность множеств. </details>  
3. Убираются ли дубликаты в результате INTERSECT и EXCEPT?  
   <details><summary>Ответ</summary> **Да.** Как и в UNION (без ALL), результат INTERSECT и EXCEPT не содержит дубликатов — каждая строка в результате один раз. </details>

**Проверь себя перед переходом к 14.3.**  
Ответь вслух: (1) INTERSECT оставляет только то, что есть в обоих запросах или «всё из первого и второго»? (2) EXCEPT — это «первый минус второй» или «общие строки»? Если ответил «только то, что в обоих» и «первый минус второй» — можно идти к порядку и типам столбцов.

### 14.3. Порядок и типы столбцов

Имена столбцов берутся из первого запроса. ORDER BY применяется ко всему результату и пишется **один раз в конце**; можно использовать позицию или алиас из первого запроса.

**Простыми словами.**  
Оба запроса под UNION должны возвращать **одинаковое число столбцов**, и типы по позиции должны совпадать (первый столбец с первым, второй со вторым). Имена столбцов в результате — те, что в **первом** запросе (второй может иметь другие имена — они игнорируются). Сортировать объединённый результат можно только **одним** ORDER BY в самом конце запроса — он относится ко всему результату UNION, а не к одному из запросов.

```sql
SELECT id, name FROM a
UNION ALL
SELECT id, name FROM b
ORDER BY 2;
```

**Что будет, если… (UNION: столбцы и порядок).**  
- **Разное число столбцов:** первый запрос вернул 2 столбца, второй — 3. СУБД выдаст ошибку: «each SELECT must have the same number of columns». Нужно привести к одному числу (например, во втором запросе добавить NULL AS third_column или убрать лишний столбец).  
- **Несовместимые типы по позиции:** первый столбец первого запроса — INTEGER, первого столбец второго — TEXT. СУБД может выдать ошибку или неявное приведение типов; лучше явно приводить (CAST).  
- **ORDER BY внутри первого или второго запроса:** в стандарте SQL ORDER BY внутри подзапроса под UNION может игнорироваться для определения порядка итогового результата. Чтобы отсортировать **весь** объединённый результат, пиши один ORDER BY в конце (после последнего запроса).

**Типичная ошибка: ORDER BY в неправильном месте.**  
Написать ORDER BY внутри первого или второго запроса под UNION — в большинстве СУБД это не отсортирует весь объединённый результат: ORDER BY внутри подзапроса может игнорироваться или применяться только к этой части. Нужно писать **один** ORDER BY в самом конце, после второго запроса: он сортирует уже объединённый результат. Пример правильного варианта: `(SELECT id, name FROM a UNION ALL SELECT id, name FROM b) ORDER BY name`.

**Запомните**

- ORDER BY — в конце всего выражения; один раз для объединённого результата.

**Вопросы для самопроверки (14.3)**  
1. Откуда берутся имена столбцов в результате UNION — из первого или из второго запроса?  
   <details><summary>Ответ</summary> Из **первого** запроса. Второй запрос может иметь другие имена столбцов — они **игнорируются** для итогового результата. </details>  
2. Где писать ORDER BY, чтобы отсортировать весь объединённый результат UNION?  
   <details><summary>Ответ</summary> **Один раз в самом конце** всего выражения — после последнего запроса. ORDER BY внутри первого или второго запроса может игнорироваться для итогового порядка; сортировка применяется к **уже объединённому** результату. </details>  
3. Что будет, если типы первого столбца в двух запросах несовместимы (например, INTEGER и TEXT)?  
   <details><summary>Ответ</summary> СУБД может выдать **ошибку** или выполнить неявное приведение типов; поведение зависит от СУБД. Надёжнее явно приводить типы (CAST). </details>

### 14.4. Пагинация: OFFSET и keyset

**OFFSET n** — пропустить n строк. Вместе с LIMIT часто используют для постраничного вывода. Недостаток: при большом OFFSET СУБД всё равно должна вычислить и отбросить первые n строк (полное сканирование/сортировка до offset), поэтому производительность падает при росте номера страницы.

**Keyset-пагинация (cursor-based)** — вместо «страница 100» передаётся «последний увиденный ключ» (например, id последней строки предыдущей страницы). Следующий запрос: `WHERE id > last_seen_id ORDER BY id LIMIT 20`. Так производительность стабильна: используется индекс по id, независимо от «номера» страницы. Подходит для бесконечной ленты и API.

**Почему OFFSET медленный при больших страницах?**  
OFFSET 1000 LIMIT 20 означает: «пропустить первые 1000 строк и вернуть следующие 20». СУБД всё равно должна **вычислить и отсортировать** первые 1000 строк (или прочитать их с диска), а потом отбросить — только затем вернуть 20. Чем больше OFFSET, тем больше работы «впустую». При keyset-пагинации мы не считаем «пропустить N строк», а говорим: «дать строки с id больше такого-то» — и индекс по id сразу находит нужное место; объём работы не растёт с номером страницы.

**Мини-пример: что делает СУБД при OFFSET vs keyset (по шагам).**  
Пусть в таблице 10 000 строк, страница по 20 строк.  
- **Страница 1:** OFFSET 0 LIMIT 20 — СУБД находит первые 20 строк (по индексу или сортировке) и возвращает. Работа небольшая.  
- **Страница 100:** OFFSET 1980 LIMIT 20 — СУБД должна **сформировать** первые 2000 строк (отсортировать/прочитать), **отбросить** их, и только потом вернуть строки 2001–2020. То есть работа «на 2000 строк» ради 20 в ответе.  
- **Keyset, «следующая страница»:** последний id на предыдущей странице был 2020. Запрос: WHERE id > 2020 ORDER BY id LIMIT 20. СУБД по индексу на id **сразу переходит** к первому id > 2020 и читает 20 строк. Объём работы не зависит от того, была ли это «страница 2» или «страница 100» — всегда «найти ключ и прочитать 20 строк».  
Итого: OFFSET растёт с номером страницы; keyset — постоянная работа на одну страницу.

**Как запомнить.**  
OFFSET = «пропустить N строк» — при большом N СУБД много работает впустую. Keyset = «дать строки после последнего увиденного ключа» — стабильная скорость. Фраза: «для больших страниц — keyset (WHERE id > last_id), не OFFSET».

**Пример keyset**

```sql
-- Первая страница
SELECT id, title, created_at FROM posts ORDER BY id LIMIT 20;

-- Следующая: передать last_id с предыдущей страницы
SELECT id, title, created_at FROM posts WHERE id > :last_id ORDER BY id LIMIT 20;
```

**Запомните**

- OFFSET при больших значениях медленный; keyset (WHERE key > last_key) даёт стабильную скорость.
- Для постраничного API предпочтительнее передавать курсор (последний ключ), а не номер страницы.

**Вопросы для самопроверки (14.4)**  
1. Почему OFFSET 10000 LIMIT 20 медленнее, чем OFFSET 0 LIMIT 20?  
   <details><summary>Ответ</summary> СУБД должна **вычислить и отсортировать** (или прочитать) первые 10020 строк, затем **отбросить** первые 10000 и вернуть только 20. Работа «на 10020 строк» ради 20 в ответе; чем больше OFFSET, тем больше работы «впустую». </details>  
2. Как устроена keyset-пагинация и почему она быстрее на больших страницах?  
   <details><summary>Ответ</summary> Вместо номера страницы передаётся **последний увиденный ключ** (например, id последней строки). Запрос: **WHERE id > :last_id ORDER BY id LIMIT 20**. СУБД по индексу на id **сразу переходит** к нужному месту и читает 20 строк; объём работы **не зависит** от номера страницы. </details>  
3. Что передавать клиенту API для «следующей страницы» при keyset-пагинации?  
   <details><summary>Ответ</summary> **Курсор** — значение ключа последней строки на текущей странице (например, last_id). Клиент отправляет его в следующем запросе (WHERE id > last_id). Номер страницы не нужен. </details>

**Резюме раздела 14 в одном абзаце.**  
UNION объединяет результаты двух запросов и по умолчанию убирает дубликаты; UNION ALL не убирает — обычно быстрее. INTERSECT — только строки, которые есть в обоих результатах; EXCEPT — только в первом, не во втором. Оба запроса должны возвращать одинаковое число столбцов с совместимыми типами по позиции. ORDER BY пишется один раз в конце всего выражения. Пагинация: OFFSET при больших значениях медленная; keyset-пагинация (WHERE id > last_id ORDER BY id LIMIT n) даёт стабильную скорость.
