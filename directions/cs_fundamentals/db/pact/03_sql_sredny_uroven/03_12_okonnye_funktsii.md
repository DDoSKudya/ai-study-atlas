## 12. Оконные функции

**Цель раздела.**  
Научиться использовать оконные функции: задавать окно (OVER, PARTITION BY, ORDER BY, рамку), применять ROW_NUMBER, RANK, DENSE_RANK, LAG/LEAD, агрегаты как окно (нарастающий итог и т.д.), FIRST_VALUE/LAST_VALUE/NTH_VALUE. Понимать разницу между группировкой (GROUP BY) и окном: окно не свёртывает строки.

**Простыми словами: о чём этот раздел.**  
Здесь мы учимся считать **по «окну»** (по группе строк) **без схлопывания**: строк в результате столько же, но в каждой появляется **новый столбец** — ранг в группе, нарастающая сумма, значение из предыдущей или следующей строки, первое/последнее значение в группе и т.д. GROUP BY «схлопывает» строки (одна строка на группу); оконная функция **не схлопывает** — просто дописывает в каждую строку число, посчитанное по её группе (партиции). Удобно для «топ-1 по группе», «разница с предыдущим», «нарастающий итог по пользователю» и подобных задач.

**Теоретическая основа.**  
Оконная функция вычисляется над **множеством строк**, связанных с текущей строкой (партиция + порядок + опционально рамка), но **результат приписывается каждой строке отдельно**, без свёртки в одну строку на группу. Это отличает окно от GROUP BY: при группировке мы получаем одну строку на группу; при оконной функции число строк остаётся таким же (или почти таким же), а в каждой строке появляется дополнительный столбец с результатом функции для «её» окна. В стандарте SQL (SQL:2003 и позже) оконные функции введены именно для таких «аналитических» расчётов без потери детализации: ранг, нарастающий итог, доступ к соседним строкам — всё это естественно выражается через окно, а не через самоприсоединение или коррелированные подзапросы, которые были бы гораздо тяжелее.

### 12.1. OVER() и PARTITION BY

**Термины**

- **Оконная функция (window function)** — функция, вызываемая с предложением OVER(), которое задаёт окно: набор строк, над которыми вычисляется функция. Результат записывается в текущую строку (строки не группируются).
- **OVER()** — пустое окно: все строки результата запроса (после WHERE и т.д.) образуют одно окно. Например, COUNT(*) OVER() даёт общее число строк в каждой строке.
- **PARTITION BY expr** — разбить строки на партиции по значению expr; окно ограничивается текущей партицией. В каждой партиции функция считается отдельно.

**Синтаксис**

```sql
function_name(...) OVER (
    [PARTITION BY expr [, ...]]
    [ORDER BY expr [ASC|DESC] [, ...]]
    [frame_clause]
)
```

**Простыми словами.** Группировка (GROUP BY) «схлопывает» строки: на каждую группу одна строка результата. Оконная функция не схлопывает: строк столько же (или почти столько же), но в каждой строке появляется ещё один столбец — результат функции, посчитанный по «её» окну (по всей таблице или по партиции). Например, «в каждой строке заказа написать общее число заказов пользователя» — это COUNT(*) OVER (PARTITION BY user_id): число строк не меняется, в каждой строке просто добавляется число.

**Картинка: GROUP BY vs оконная функция.**  
Допустим заказы: (id=1, user_id=10), (id=2, user_id=10), (id=3, user_id=20).  
- **GROUP BY user_id** даёт **две строки**: одна для user_id=10 (с агрегатом, например COUNT(*)=2), одна для user_id=20 (COUNT(*)=1). Строки «схлопнулись» в группы.  
- **COUNT(*) OVER (PARTITION BY user_id)** даёт **три строки**: у каждой строки заказа добавляется столбец «сколько заказов у этого пользователя» — у строк 1 и 2 будет 2, у строки 3 будет 1. Строки не схлопываются — просто в каждой появляется число, посчитанное по её партиции (по user_id).

**Как запомнить.**  
OVER() = «окно» = набор строк, по которому считается функция. PARTITION BY = разбить на группы **без свёртки**: каждая строка остаётся, но функция считается только по «своей» группе. Фраза: «окно — не схлопываем строки, а в каждой строке пишем результат по её группе».

**Примеры**

Общее количество заказов в каждой строке (окно — вся таблица):

```sql
SELECT id, user_id, total,
       COUNT(*) OVER () AS total_orders
FROM orders;
```

Количество заказов **по пользователю** (окно — партиция по user_id):

```sql
SELECT id, user_id, total,
       COUNT(*) OVER (PARTITION BY user_id) AS orders_per_user
FROM orders
ORDER BY user_id, id;
```

**Разбор этого запроса по частям (что за что отвечает).**  
- `SELECT id, user_id, total, ...` — выводим столбцы заказа плюс новый столбец.  
- `COUNT(*) OVER (PARTITION BY user_id)` — в **каждой** строке пишем число: «сколько всего заказов у этого user_id». Партиция по user_id значит: разбить строки на группы по user_id; в каждой группе посчитать COUNT(*); это число записать в каждую строку этой группы. Строки не схлопываются — их столько же, но в каждой появился столбец orders_per_user.  
- `FROM orders` — исходные данные из таблицы заказов.  
- `ORDER BY user_id, id` — порядок вывода (на работу оконной функции не влияет; порядок внутри партиции для COUNT без ORDER BY не важен).  
Если бы мы написали `GROUP BY user_id`, получили бы **одну строку на пользователя** с COUNT(*). С `OVER (PARTITION BY user_id)` получаем **все строки заказов**, и в каждой — число заказов этого пользователя.

**Запомните**

- OVER() — окно по всем строкам; PARTITION BY — разбиение на группы без свёртки.
- Оконная функция не уменьшает число строк: в каждой строке появляется новое значение.

**Вопросы для самопроверки (12.1)**  
1. Чем OVER() отличается от PARTITION BY в оконной функции?  
   <details><summary>Ответ</summary> **OVER()** — окно = **все строки** результата запроса (одно окно). **PARTITION BY expr** — разбить строки на группы по expr; в каждой партиции функция считается отдельно; строки не схлопываются. </details>  
2. Уменьшает ли оконная функция число строк в результате по сравнению с исходной таблицей?  
   <details><summary>Ответ</summary> **Нет.** Строк столько же (или почти столько же); в каждой строке появляется **дополнительный столбец** — значение функции по её окну. </details>  
3. Что даёт COUNT(*) OVER (PARTITION BY user_id) по смыслу в каждой строке?  
   <details><summary>Ответ</summary> В каждой строке — **число строк в её партиции** (по user_id): «сколько заказов у этого пользователя». Строки не группируются — число просто дублируется во всех строках партиции. </details>

### 12.2. ORDER BY в окне и рамка

**ORDER BY** в OVER задаёт порядок строк внутри партиции. Он влияет на:
- ранжирующие функции (ROW_NUMBER, RANK, DENSE_RANK) — порядок нумерации;
- функции доступа к соседним строкам (LAG, LEAD) — что считать «предыдущей»/«следующей»;
- агрегаты в окне — по умолчанию рамка «от начала партиции до текущей строки», т.е. нарастающий итог при ORDER BY.

**Рамка окна (frame)** — подмножество строк партиции, участвующих в расчёте. Задаётся через ROWS или RANGE:

- `ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW` — от первой строки партиции до текущей (по порядку ORDER BY). По умолчанию для агрегатов при наличии ORDER BY в OVER используется именно эта рамка.
- `RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW` — по значению: «все строки с значением ORDER BY expr не больше текущего».
- `ROWS BETWEEN 2 PRECEDING AND 2 FOLLOWING` — скользящее окно: 2 строки до и 2 после текущей.

**ROWS vs RANGE:** ROWS считает физические строки (по позиции), RANGE — по значению выражения ORDER BY; при одинаковых значениях в RANGE в окно попадут все строки с этим значением.

**Мини-пример: в чём разница ROWS и RANGE.**  
Партиция из четырёх строк, ORDER BY date: даты 01.01, 01.01, 02.01, 03.01 (две строки с одной датой). Рамка «до текущей строки».  
- **ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW:** для 3-й строки (02.01) в окно попадают **строго первые три строки** по порядку (две с 01.01 и одна с 02.01). Считаем по **позиции** строки.  
- **RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW:** для строки с датой 02.01 в окно попадают все строки, у которых date ≤ 02.01 — то есть все с 01.01 и 02.01. Если бы у нас было две строки с 02.01, в RANGE **обе** вошли бы в окно для «текущей» по значению 02.01; в ROWS — только по одной позиции. Итого: ROWS = по номеру строки, RANGE = по значению ORDER BY (при равенстве значений все такие строки в одной «группе» по рамке).

**Мини-пример рамки (нарастающий итог).**  
Партиция по user_id, ORDER BY created_at. Строки в партиции: заказ A (100 руб.), заказ B (200 руб.), заказ C (150 руб.). Рамка по умолчанию при ORDER BY — «от начала партиции до текущей строки».  
- Для строки A: в окне только A → SUM = 100.  
- Для строки B: в окне A и B → SUM = 300.  
- Для строки C: в окне A, B и C → SUM = 450.  
Так получается **нарастающий итог** по порядку created_at.

**Как запомнить.**  
ORDER BY в OVER задаёт порядок строк внутри партиции (кто «первый», кто «предыдущий», кто «следующий»). Рамка ROWS/RANGE ограничивает, **какие строки партиции** участвуют в расчёте агрегата: по умолчанию при ORDER BY — «от начала до текущей строки», поэтому агрегат даёт нарастающий итог.

**Запомните**

- ORDER BY в OVER задаёт порядок внутри партиции для рангов и LAG/LEAD.
- Рамка (ROWS/RANGE BETWEEN) ограничивает, какие строки партиции участвуют в расчёте агрегата.

**Вопросы для самопроверки (12.2)**  
1. На что влияет ORDER BY внутри OVER?  
   <details><summary>Ответ</summary> Задаёт **порядок строк внутри партиции** для ранжирующих функций (ROW_NUMBER, RANK, DENSE_RANK), для LAG/LEAD («предыдущая»/«следующая» строка) и для агрегатов (по умолчанию рамка «от начала партиции до текущей строки» — нарастающий итог). </details>  
2. Чем ROWS BETWEEN отличается от RANGE BETWEEN в рамке окна?  
   <details><summary>Ответ</summary> **ROWS** считает **физические строки** (по позиции: «2 строки до», «текущая»). **RANGE** — по **значению** выражения ORDER BY (все строки с тем же значением входят в одну «группу» по рамке). При одинаковых значениях ORDER BY RANGE включает все такие строки. </details>  
3. Почему при ORDER BY в OVER агрегат по умолчанию даёт нарастающий итог?  
   <details><summary>Ответ</summary> По умолчанию рамка — **от начала партиции до текущей строки** (UNBOUNDED PRECEDING AND CURRENT ROW). В этой рамке SUM даёт сумму «от первой строки до текущей» — то есть нарастающий итог. </details>

### 12.3. ROW_NUMBER, RANK, DENSE_RANK

- **ROW_NUMBER()** — уникальный порядковый номер строки в партиции (1, 2, 3, …). При одинаковых значениях ORDER BY порядок не гарантирован — номера всё равно разные.
- **RANK()** — ранг: одинаковые значения получают один и тот же ранг, следующий ранг «прыгает» (1, 2, 2, 4).
- **DENSE_RANK()** — плотный ранг: одинаковые — один ранг, следующий ранг идёт подряд (1, 2, 2, 3).

**Пример**

```sql
SELECT id, user_id, total,
       ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY total DESC) AS rn,
       RANK()       OVER (PARTITION BY user_id ORDER BY total DESC) AS rk,
       DENSE_RANK() OVER (PARTITION BY user_id ORDER BY total DESC) AS dr
FROM orders;
```

Типичное использование: «топ-1 по группе» — фильтр `WHERE rn = 1` после оборачивания в подзапрос или использование DISTINCT ON (в PostgreSQL).

**Как получить «одну строку на группу» (топ-1 по группе) пошагово.**  
Задача: «для каждого пользователя оставить только один заказ — например, с максимальной суммой». Шаги: (1) Считаем ROW_NUMBER() (или RANK()) в партиции по user_id, ORDER BY total DESC — у каждого заказа появится номер в «рейтинге» по пользователю. (2) Оборачиваем запрос в подзапрос: `SELECT * FROM (SELECT ..., ROW_NUMBER() OVER (...) AS rn FROM orders) sub`. (3) Фильтруем снаружи: `WHERE sub.rn = 1` — остаётся только первая строка в каждой партиции (заказ с максимальной суммой по пользователю). В WHERE нельзя писать rn = 1 напрямую в одном запросе — оконная функция считается в SELECT, а WHERE выполняется до SELECT по логике запроса; поэтому нужен подзапрос или CTE.

**Мини-пример: ROW_NUMBER vs RANK vs DENSE_RANK.**  
Допустим в партиции по user_id, ORDER BY total DESC, строки с total: 500, 300, 300, 100.  
- **ROW_NUMBER()**: 1, 2, 3, 4 — уникальный номер, порядок при равенстве не гарантирован.  
- **RANK()**: 1, 2, 2, 4 — одинаковые значения (300) получают один ранг (2), следующий ранг «прыгает» на 4.  
- **DENSE_RANK()**: 1, 2, 2, 3 — одинаковые получают один ранг, следующий идёт подряд (без пропуска).

**Как запомнить.**  
ROW_NUMBER = «просто номер по порядку», всегда разный. RANK = «место в рейтинге»: при равенстве одно место, следующее место с пропуском (как в спорте: два вторых — следующее место четвёртое). DENSE_RANK = «место без пропусков»: два вторых — следующее место третье.

**Запомните**

- ROW_NUMBER — уникальный номер; RANK — с пропусками при равенстве; DENSE_RANK — без пропусков.
- ORDER BY в OVER обязателен для этих функций (иначе порядок не определён).
- Шпаргалка: RANK — «прыгает», DENSE_RANK — «подряд».

**Вопросы для самопроверки (12.3)**  
1. При одинаковых значениях ORDER BY что даёт RANK, а что DENSE_RANK для следующего места?  
   <details><summary>Ответ</summary> **RANK** — следующий ранг **прыгает** (два вторых → следующее место четвёртое: 1, 2, 2, 4). **DENSE_RANK** — следующий ранг **подряд** (два вторых → следующее третье: 1, 2, 2, 3). </details>  
2. Как получить «одну строку на группу» (топ-1 по группе) с помощью ROW_NUMBER?  
   <details><summary>Ответ</summary> Обернуть запрос с ROW_NUMBER() OVER (PARTITION BY ... ORDER BY ...) в **подзапрос или CTE** и снаружи отфильтровать **WHERE rn = 1**. В одном SELECT нельзя фильтровать по результату оконной функции в WHERE — нужен подзапрос. </details>  
3. Чем ROW_NUMBER отличается от RANK при равенстве значений ORDER BY?  
   <details><summary>Ответ</summary> **ROW_NUMBER** присваивает **разные** номера (1, 2, 3, 4); порядок среди равных не гарантирован. **RANK** присваивает **одинаковый** ранг равным, следующий ранг с пропуском (1, 2, 2, 4). </details>

**Проверь себя перед переходом к 12.4.**  
Ответь вслух: (1) При PARTITION BY строки схлопываются в одну на группу или остаются все? (2) При одинаковых значениях ORDER BY что даёт RANK — один номер или разные? Если ответил «остаются все» и «один ранг (но следующий с пропуском)» — можно идти дальше.

### 12.4. LAG и LEAD

**LAG(column [, offset [, default]])** — значение столбца из строки, отстоящей на offset строк **назад** по порядку ORDER BY в окне. По умолчанию offset = 1. default — что вернуть, если такой строки нет (например, для первой строки партиции).

**LEAD(column [, offset [, default]])** — значение из строки на offset строк **вперёд**.

**Картинка в голове.**  
Строки в партиции упорядочены (ORDER BY). LAG — «посмотри на предыдущую строку в этом порядке и возьми из неё значение столбца». LEAD — «посмотри на следующую строку и возьми значение». Для первой строки «предыдущей» нет — LAG вернёт NULL (или default). Для последней строки «следующей» нет — LEAD вернёт NULL (или default).

**Как запомнить.**  
LAG = «отставание» (lag behind) = значение из **предыдущей** строки. LEAD = «опережение» (lead ahead) = значение из **следующей** строки. Третий аргумент — что подставить, если такой строки нет (иначе NULL).

**Пример: разница с предыдущим заказом по пользователю**

```sql
SELECT id, user_id, total, created_at,
       LAG(total) OVER (PARTITION BY user_id ORDER BY created_at) AS prev_total,
       total - LAG(total) OVER (PARTITION BY user_id ORDER BY created_at) AS diff_from_prev
FROM orders;
```

**Мини-пример: зачем третий аргумент (default) в LAG/LEAD.**  
Без третьего аргумента: для **первой** строки партиции у LAG нет «предыдущей» строки — функция вернёт **NULL**. Тогда diff_from_prev = total - NULL = NULL (а не число). Если нужно, чтобы для первого заказа «предыдущая сумма» считалась нулём (и разница была просто total), пиши **LAG(total, 1, 0)** — третий аргумент 0 подставится, когда предыдущей строки нет. Аналогично **LEAD(total, 1, 0)** для последней строки партиции вернёт 0 вместо NULL. Запомни: третий аргумент — «что подставить, если такой строки нет (иначе NULL)».

**Запомните**

- LAG — «предыдущая» строка в порядке окна; LEAD — «следующая».
- Третий аргумент — значение по умолчанию при отсутствии строки (иначе NULL).

**Вопросы для самопроверки (12.4)**  
1. Откуда LAG(total) берёт значение — из текущей строки или из другой?  
   <details><summary>Ответ</summary> Из **предыдущей** строки в порядке ORDER BY в окне (по умолчанию offset=1). LEAD(total) — из **следующей** строки. </details>  
2. Что вернёт LAG(total) для первой строки партиции, если третий аргумент не указан?  
   <details><summary>Ответ</summary> **NULL**: у первой строки нет «предыдущей» строки. Третий аргумент (default) как раз задаёт значение в таких случаях (например, LAG(total, 1, 0) даст 0 для первой строки). </details>  
3. Зачем нужен ORDER BY в OVER для LAG/LEAD?  
   <details><summary>Ответ</summary> Чтобы однозначно определить **«предыдущую»** и **«следующую»** строку. Без ORDER BY порядок в партиции не определён — LAG/LEAD давали бы непредсказуемый результат. </details>

**Проверь себя перед переходом к 12.5.**  
Ответь вслух: (1) LAG(total) — откуда берётся значение: из текущей строки или из предыдущей по порядку окна? (2) Для первой строки партиции LAG вернёт NULL или значение? Если ответил «из предыдущей строки» и «NULL (или default, если указан третий аргумент)» — можно идти к агрегатам как окно.

### 12.5. Агрегаты как окно

Агрегатные функции (SUM, AVG, COUNT, MIN, MAX) можно вызывать с OVER(): тогда они считаются по окну, а не по группе GROUP BY. Часто используют для **нарастающего итога**.

**Простыми словами.**  
Обычный SUM в запросе с GROUP BY «схлопывает» строки: одна строка на группу с суммой по группе. SUM(...) OVER (PARTITION BY ... ORDER BY ...) **не схлопывает**: строк столько же, но в **каждой** строке в новом столбце пишется сумма — причём по умолчанию «от первой строки партиции до текущей» (нарастающий итог). То есть в первой строке партиции — сумма одной строки, во второй — сумма двух, в третьей — трёх и т.д. Удобно для «сколько накопилось на текущий момент» по порядку (дата, id).

```sql
SELECT id, user_id, total, created_at,
       SUM(total) OVER (PARTITION BY user_id ORDER BY created_at
                        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS running_total
FROM orders;
```

Без указания рамки при наличии ORDER BY в OVER по умолчанию используется рамка от начала партиции до текущей строки, поэтому нарастающий итог можно писать короче:

```sql
SUM(total) OVER (PARTITION BY user_id ORDER BY created_at) AS running_total
```

**Мини-пример: нарастающий итог по шагам (одна партиция).**  
Партиция по user_id=1, заказы по created_at: A (100 руб.), B (200 руб.), C (150 руб.). Рамка по умолчанию — «от начала партиции до текущей строки».  
- Строка A: в окне только A → running_total = 100.  
- Строка B: в окне A и B → running_total = 300.  
- Строка C: в окне A, B и C → running_total = 450.  
В каждой строке — «сумма с начала партиции до этой строки включительно». Строки не исчезают — их по-прежнему три, просто в каждой появился столбец running_total.

**Запомните**

- SUM/AVG/COUNT/MIN/MAX с OVER — расчёт по окну; строки не свёртываются.
- При ORDER BY в OVER агрегат по умолчанию — нарастающий итог в партиции.

**Вопросы для самопроверки (12.5)**  
1. Чем SUM(...) OVER (PARTITION BY ... ORDER BY ...) отличается от SUM(...) с GROUP BY?  
   <details><summary>Ответ</summary> При **GROUP BY** строки **свёртываются** — одна строка на группу с суммой по группе. При **OVER** строки **не свёртываются** — в каждой строке появляется столбец с суммой; по умолчанию при ORDER BY это **нарастающий итог** (от начала партиции до текущей строки). </details>  
2. Как получить нарастающий итог по пользователю (по дате заказа)?  
   <details><summary>Ответ</summary> **SUM(total) OVER (PARTITION BY user_id ORDER BY created_at)**. Рамка по умолчанию «от начала партиции до текущей строки» даёт нарастающий итог. </details>  
3. Уменьшается ли число строк при использовании агрегата с OVER?  
   <details><summary>Ответ</summary> **Нет.** Агрегат с OVER считается по окну, но результат записывается в **каждую** строку; строки не группируются и не исчезают. </details>

### 12.6. FIRST_VALUE, LAST_VALUE, NTH_VALUE

- **FIRST_VALUE(expr)** — значение expr в первой строке окна (по ORDER BY).
- **LAST_VALUE(expr)** — в последней строке окна. Важно: при рамке по умолчанию «до текущей строки» последняя строка окна — текущая, поэтому LAST_VALUE часто совпадает с текущим значением. Чтобы получить значение в последней строке **всей партиции**, задают рамку: `ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING`.
- **NTH_VALUE(expr, N)** — значение expr в N-й строке окна (1-based).

**Простыми словами (почему LAST_VALUE «странный»).**  
По умолчанию рамка окна — «от начала партиции до **текущей** строки». Для каждой строки «последняя строка в окне» — это она сама. Поэтому LAST_VALUE(total) без явной рамки в каждой строке вернёт total этой же строки — не «последний заказ в партиции», а «текущий». Чтобы взять значение из **действительно последней** строки партиции (по ORDER BY), нужно расширить рамку до конца: ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING — «от первой строки партиции до последней».

**Пример**

```sql
SELECT id, user_id, total,
       FIRST_VALUE(total) OVER (PARTITION BY user_id ORDER BY created_at) AS first_order_total,
       LAST_VALUE(total) OVER (PARTITION BY user_id ORDER BY created_at
                                ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING) AS last_order_total
FROM orders;
```

**Мини-пример «до и после» (FIRST_VALUE и LAST_VALUE).**  
Партиция по user_id, ORDER BY created_at. Заказы по порядку: total = 100, 200, 150.  
- **FIRST_VALUE(total):** в каждой строке окна «первая строка» — это заказ с 100. Поэтому во **всех трёх** строках first_order_total = 100.  
- **LAST_VALUE(total) без рамки (по умолчанию «до текущей строки»):** для каждой строки «последняя строка в окне» — это она сама. В первой строке last = 100, во второй = 200, в третьей = 150. То есть LAST_VALUE без рамки просто повторяет текущее значение — это редко то, что нужно.  
- **LAST_VALUE(total) с рамкой UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING:** окно — вся партиция. «Последняя строка» партиции — заказ с 150. Во **всех трёх** строках last_order_total = 150.  
Итого: FIRST_VALUE — всегда значение из первой строки партиции; LAST_VALUE по всей партиции — только с явной рамкой до конца (UNBOUNDED FOLLOWING).

**NTH_VALUE(expr, N)** — значение expr в **N-й** строке окна (нумерация с 1). Удобно, когда нужно «второй по величине заказ», «третий по дате» и т.п. в каждой партиции.

**Мини-пример: NTH_VALUE.**  
Партиция по user_id, ORDER BY created_at. Заказы по порядку: total = 100, 200, 150 (три строки).  
- **NTH_VALUE(total, 1)** — значение в **первой** строке окна (по ORDER BY). Первая строка — заказ с 100. Во всех трёх строках результат = 100 (аналогично FIRST_VALUE).  
- **NTH_VALUE(total, 2)** — значение во **второй** строке окна. Вторая строка — заказ с 200. Во всех трёх строках результат = 200.  
- **NTH_VALUE(total, 3)** — третья строка, 150. Во всех строках = 150.  
Если в партиции меньше строк, чем N (например, NTH_VALUE(total, 5) при четырёх строках), функция вернёт NULL. Запомни: NTH_VALUE(expr, N) — «значение из N-й строки партиции по порядку ORDER BY»; нумерация с 1.

**Запомните**

- FIRST_VALUE / LAST_VALUE / NTH_VALUE — значение в первой/последней/N-й строке окна.
- Для LAST_VALUE по всей партиции нужна рамка до UNBOUNDED FOLLOWING.
- NTH_VALUE(expr, N) — N-я строка (1-based); если строк меньше N — NULL.

**Вопросы для самопроверки (12.6)**  
1. Почему LAST_VALUE(total) без явной рамки в каждой строке даёт значение текущей строки, а не последней в партиции?  
   <details><summary>Ответ</summary> По умолчанию рамка — **от начала партиции до текущей строки**. Для каждой строки «последняя строка в окне» — это **она сама**. Поэтому LAST_VALUE без рамки возвращает значение текущей строки. Для «последней в партиции» нужна рамка **UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING**. </details>  
2. Как получить значение из последней строки партиции (по ORDER BY) во всех строках?  
   <details><summary>Ответ</summary> **LAST_VALUE(expr) OVER (... ORDER BY ... ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING)** — рамка «вся партиция», тогда «последняя строка» одна и та же для всех. </details>  
3. Что вернёт NTH_VALUE(total, 5), если в партиции только 3 строки?  
   <details><summary>Ответ</summary> **NULL**: N-я строка (5-я) отсутствует; нумерация с 1. </details>

### 12.7. Типичные ошибки и производительность окон

- Забыть ORDER BY в OVER для RANK/ROW_NUMBER/LAG/LEAD — порядок не определён.
- Путать рамку по умолчанию при ORDER BY (до текущей строки) и «вся партиция» для LAST_VALUE.
- Оконные функции не допускаются в WHERE и в GROUP BY — только в SELECT и в ORDER BY. Для фильтрации по результату оконной функции использовать подзапрос или CTE.

**Что будет, если… (оконные функции).**  
- **Забыть ORDER BY в OVER для LAG/LEAD:** порядок строк в партиции не определён — «предыдущая» и «следующая» строка непонятны. LAG/LEAD могут вернуть случайные значения. Всегда указывай ORDER BY в OVER для LAG, LEAD, ROW_NUMBER, RANK, DENSE_RANK.  
- **Забыть PARTITION BY и написать только OVER():** окно будет **вся таблица** (или весь результат запроса). Например, ROW_NUMBER() OVER() даст сквозную нумерацию по всем строкам; ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY created_at) даст нумерацию **внутри каждого пользователя**. Для «топ-1 по группе» нужен именно PARTITION BY.  
- **Путать ORDER BY в запросе и ORDER BY в OVER:** ORDER BY в конце запроса (ORDER BY id) задаёт **порядок вывода** строк. ORDER BY внутри OVER задаёт **порядок строк внутри партиции** для расчёта функции (кто «первый», «предыдущий», «следующий»). Это разные вещи: можно выводить строки в одном порядке, а окно считать по другому.

**Типичная ошибка: фильтровать по результату оконной функции в WHERE.**  
Написать что-то вроде `SELECT ..., ROW_NUMBER() OVER (...) AS rn FROM t WHERE rn = 1` — **нельзя**: в WHERE нельзя использовать столбцы из SELECT, а оконная функция как раз считается на этапе SELECT. СУБД выдаст ошибку (rn неизвестен в WHERE). Правильно: обернуть запрос в подзапрос или CTE и фильтровать снаружи: `SELECT * FROM (SELECT ..., ROW_NUMBER() OVER (...) AS rn FROM t) sub WHERE rn = 1`.

**Вопросы для самопроверки (12.7)**  
1. Почему нельзя написать WHERE rn = 1 в том же запросе, где в SELECT считается ROW_NUMBER() AS rn?  
   <details><summary>Ответ</summary> **WHERE** выполняется **до** вычисления выражений в SELECT (по логике запроса). Столбец rn ещё не существует в момент WHERE; СУБД выдаст ошибку «столбец rn не найден». Нужен подзапрос или CTE: сначала вычислить rn, потом снаружи фильтровать. </details>  
2. Чем ORDER BY в конце запроса отличается от ORDER BY внутри OVER?  
   <details><summary>Ответ</summary> **ORDER BY в конце запроса** задаёт **порядок вывода** строк результата. **ORDER BY внутри OVER** задаёт **порядок строк внутри партиции** для расчёта функции (кто «первый», «предыдущий», «следующий»). Это разные вещи. </details>  
3. Что будет, если забыть PARTITION BY в ROW_NUMBER() OVER (ORDER BY total DESC)?  
   <details><summary>Ответ</summary> Окно будет **вся таблица** (одна партиция). ROW_NUMBER даст **сквозную** нумерацию по всем строкам (1, 2, 3, …), а не «номер внутри каждой группы». Для «топ-1 по группе» нужен именно PARTITION BY. </details>

**Резюме раздела 12 в одном абзаце.**  
Оконная функция считается по «окну» (набору строк), но строки не свёртываются — в каждой строке появляется новое значение. OVER() задаёт окно; PARTITION BY разбивает на группы без свёртки; ORDER BY задаёт порядок внутри партиции. Рамка (ROWS/RANGE BETWEEN) ограничивает, какие строки партиции участвуют в агрегате; по умолчанию при ORDER BY — «от начала до текущей строки» (нарастающий итог). ROW_NUMBER, RANK, DENSE_RANK — нумерация и ранги; RANK с пропусками, DENSE_RANK без. LAG/LEAD — значение из предыдущей/следующей строки. Агрегаты с OVER — расчёт по окну без GROUP BY. Фильтровать по результату оконной функции нельзя в WHERE — только через подзапрос или CTE.

**Вопросы для самопроверки (раздел 12)**

1. Чем оконная функция с PARTITION BY отличается от GROUP BY?  
   <details><summary>Ответ</summary>  
   При **GROUP BY** строки **свёртываются**: на каждую группу (например, по user_id) в результате получается **одна строка** с агрегатами (COUNT, SUM и т.д.). Исходное число строк уменьшается. При **PARTITION BY** в оконной функции строки **не свёртываются**: число строк результата остаётся таким же (или почти таким же), но в **каждой** строке появляется дополнительный столбец — значение функции, посчитанное по «её» партиции (по её группе). Итого: GROUP BY — «одна строка на группу»; PARTITION BY — «все строки на месте, в каждой — значение по её группе».  
   </details>

2. Как получить нарастающий итог по партиции?  
   <details><summary>Ответ</summary>  
   Используй агрегат с OVER и ORDER BY: `SUM(col) OVER (PARTITION BY ... ORDER BY ...)`. При наличии ORDER BY в OVER по умолчанию используется рамка «от начала партиции до текущей строки» (ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW). В этой рамке SUM даёт сумму «от первой строки партиции до текущей» — то есть нарастающий итог. Партиция задаётся PARTITION BY (например, по user_id), порядок — ORDER BY (например, по created_at).  
   </details>

3. В чём разница между RANK и DENSE_RANK?  
   <details><summary>Ответ</summary>  
   При **одинаковых** значениях в ORDER BY обе функции присваивают **один и тот же ранг** этим строкам. Разница в том, как нумеруется следующее место. **RANK**: следующий ранг «прыгает» — после двух вторых мест идёт четвёртое место (1, 2, 2, 4). **DENSE_RANK**: следующий ранг идёт подряд — после двух вторых идёт третье место (1, 2, 2, 3). Шпаргалка: RANK — «как в спорте, с пропусками»; DENSE_RANK — «места подряд, без пропусков».  
   </details>
