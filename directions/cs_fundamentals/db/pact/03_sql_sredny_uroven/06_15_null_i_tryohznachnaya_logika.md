## 15. NULL и трёхзначная логика

**Цель раздела.**  
Понять, что NULL означает «неизвестное/отсутствующее значение», что сравнение с NULL даёт UNKNOWN (не TRUE и не FALSE), как правильно проверять на NULL (IS NULL, IS NOT NULL), использовать COALESCE и NULLIF, сравнивать с учётом NULL (IS DISTINCT FROM). Знать поведение NULL в сортировке, группировке и агрегатах.

**Простыми словами: о чём этот раздел.**  
NULL — это «значение неизвестно или не указано». В SQL сравнение с NULL даёт не TRUE и не FALSE, а **UNKNOWN**; в WHERE остаются только строки, где условие **TRUE**, поэтому UNKNOWN отбрасывается — и **col = NULL в условии бесполезно** (никогда не найдёт строки). Проверять на NULL нужно **только** через IS NULL / IS NOT NULL. COALESCE — подставить первое не-NULL значение (например, «не указан» вместо пустого телефона); NULLIF — обнулить «нежелательное» значение (пустую строку → NULL). IS DISTINCT FROM — сравнение двух значений с учётом NULL (два NULL считаются «не разными»). В сортировке и группировке у NULL свои правила: NULLS FIRST/LAST в ORDER BY; в GROUP BY все NULL в одном столбце образуют одну группу. В агрегатах: COUNT(*) считает все строки, COUNT(col) — только не-NULL; SUM/AVG/MIN/MAX пропускают NULL. Всё это нужно, чтобы не «сломать» запросы и условия из-за пустых или неизвестных значений.

**Теоретическая основа.**  
В стандарте SQL принята **трёхзначная логика (three-valued logic, 3VL)**: кроме TRUE и FALSE есть **UNKNOWN** (результат сравнения с NULL). Логические операторы AND, OR, NOT определены так, что UNKNOWN «заразен» в большинстве случаев: TRUE AND UNKNOWN → UNKNOWN; FALSE OR UNKNOWN → UNKNOWN; NOT UNKNOWN → UNKNOWN. Предикат WHERE оставляет только строки, для которых условие равно **TRUE**; FALSE и UNKNOWN отбрасываются. Поэтому `WHERE col = NULL` никогда не вернёт строк — нужно `WHERE col IS NULL`. Такая логика соответствует интерпретации NULL как «значение неизвестно»: мы не можем сказать, что «неизвестное равно 5», поэтому результат сравнения — не истина и не ложь, а неизвестность (UNKNOWN). В реляционной теории и в логике предикатов NULL иногда моделируют как отсутствие значения (partial information); трёхзначная логика — компромисс между строгой двузначностью и полным отказом от сравнений с NULL.

### 15.1. NULL и трёхзначная логика

**NULL** — специальное значение, означающее «значение неизвестно или отсутствует». Оно не равно ни одному другому значению, **в том числе другому NULL**: выражение `NULL = NULL` в SQL даёт не TRUE, а **NULL** (интерпретируемый как UNKNOWN).

**Трёхзначная логика:** TRUE, FALSE, UNKNOWN. Примеры:
- `NULL = 5` → UNKNOWN  
- `NULL AND TRUE` → UNKNOWN  
- `NULL OR TRUE` → TRUE  
- `WHERE condition`: остаются только строки, где condition = TRUE; FALSE и UNKNOWN отбрасываются.

Краткая таблица истинности для AND и OR (U = UNKNOWN):

| AND  | T | F | U |
|------|---|---|---|
| T    | T | F | U |
| F    | F | F | F |
| U    | U | F | U |

| OR   | T | F | U |
|------|---|---|---|
| T    | T | T | T |
| F    | T | F | U |
| U    | T | U | U |

NOT: NOT TRUE → FALSE, NOT FALSE → TRUE, NOT UNKNOWN → UNKNOWN.

Поэтому **проверка на NULL только через IS NULL / IS NOT NULL**. Выражение `col = NULL` в условии бесполезно (всегда UNKNOWN).

**Мини-пример: почему WHERE col = NULL не находит строк.**  
Допустим в таблице есть строка с col = NULL. Условие WHERE col = NULL для этой строки: NULL = NULL в SQL даёт **UNKNOWN** (не TRUE). В WHERE остаются только строки, где условие **TRUE**; UNKNOWN отбрасывается. Поэтому эта строка **не попадёт** в результат. Итог: ни одна строка не будет выбрана, даже строки с NULL в col. Чтобы выбрать строки с NULL, нужно писать **WHERE col IS NULL** — это отдельный предикат, он возвращает TRUE именно когда значение NULL.

**Почему NULL = NULL даёт не TRUE?**  
В SQL NULL означает «значение неизвестно». Вопрос «неизвестное равно неизвестному?» — мы не знаем: может, оба значения на самом деле одинаковые, а может, разные. Поэтому результат сравнения — не TRUE и не FALSE, а **UNKNOWN**. А в WHERE остаются только строки, где условие **TRUE**; UNKNOWN отбрасывается. Поэтому `WHERE col = NULL` никогда не вернёт ни одной строки — нужно писать `WHERE col IS NULL` (проверка «значение отсутствует» — это отдельный предикат, не сравнение).

**Картинка в голове: как WHERE отбирает строки.**  
Для **каждой** строки СУБД вычисляет условие в WHERE. Результат — TRUE, FALSE или UNKNOWN. В результат запроса попадают **только** строки, для которых условие дало **TRUE**. Строки с FALSE отбрасываются. Строки с UNKNOWN тоже отбрасываются (не TRUE!). Поэтому: `WHERE col = 5` — остаются только строки, где col равно 5 (TRUE). `WHERE col = NULL` — для любой строки col = NULL даёт UNKNOWN → ни одна строка не проходит. `WHERE col IS NULL` — для строк с NULL в col предикат IS NULL даёт TRUE → эти строки остаются. Запомни: **WHERE оставляет только TRUE; FALSE и UNKNOWN — отбрасываются**.

**Как запомнить.**  
NULL = «не знаем значения». Сравнивать «неизвестное» с чем угодно — результат «неизвестно» (UNKNOWN). В WHERE проходят только TRUE; UNKNOWN и FALSE отбрасываются. Поэтому проверять на NULL можно **только** через IS NULL / IS NOT NULL. Фраза: «NULL не равен ничему, даже NULL; для проверки — только IS NULL».

**Запомните**

- NULL не равен ничему, в том числе NULL.
- В WHERE условие должно быть TRUE; сравнение с NULL даёт UNKNOWN — строки не проходят.
- Проверка на NULL: только IS NULL / IS NOT NULL.
- Шпаргалка: «col = NULL бесполезно; пиши col IS NULL».

**Вопросы для самопроверки (15.1)**  
1. Что даёт в SQL сравнение NULL = 5 — TRUE, FALSE или UNKNOWN? Почему WHERE col = NULL не находит строк с NULL?  
   <details><summary>Ответ</summary> **UNKNOWN.** В WHERE остаются только строки, где условие **TRUE**; UNKNOWN и FALSE отбрасываются. Для строки с col = NULL выражение col = NULL даёт UNKNOWN → строка не проходит. Нужно **col IS NULL**. </details>  
2. Какие три результата возможны в трёхзначной логике SQL?  
   <details><summary>Ответ</summary> **TRUE**, **FALSE**, **UNKNOWN**. Сравнение с NULL и логические операции с NULL дают UNKNOWN. WHERE оставляет только TRUE. </details>  
3. Почему NULL = NULL в SQL не TRUE?  
   <details><summary>Ответ</summary> NULL означает «значение неизвестно». «Неизвестное равно неизвестному?» — ответ не определён (может да, может нет). Поэтому результат сравнения — **UNKNOWN**, а не TRUE. </details>

**Сводка: когда что использовать с NULL.**  
- Проверить «значение отсутствует» или «присутствует» → **IS NULL** / **IS NOT NULL** (никогда col = NULL).  
- Подставить значение по умолчанию, если NULL (например, «не указан» вместо пустого телефона) → **COALESCE(col, 'значение по умолчанию')**.  
- Обнулить «нежелательное» значение (например, пустую строку превратить в NULL) → **NULLIF(col, '')**.  
- Сравнить два значения с учётом NULL (включая NULL = NULL как «равны») → **IS NOT DISTINCT FROM**; «различаются» → **IS DISTINCT FROM**.

**Самые частые ошибки с NULL (запомни раз и навсегда).**  
(1) Писать `WHERE col = NULL` вместо `WHERE col IS NULL` — условие никогда не выполнится. (2) Полагаться на `NOT IN (subquery)`, если в подзапросе может быть NULL — результат может быть пустым; лучше NOT EXISTS. (3) Забывать, что в агрегатах SUM/AVG/MIN/MAX пропускают NULL, а COUNT(col) не считает строки с NULL в col; COUNT(*) считает все строки. (4) В CHECK-ограничении думать, что `CHECK (col > 0)` запретит NULL — нет, NULL даёт UNKNOWN, и строка пройдёт; для запрета NULL нужен отдельно NOT NULL.

### 15.2. IS NULL, COALESCE, NULLIF

**IS NULL** — истина, если значение NULL. **IS NOT NULL** — истина, если значение не NULL.

**COALESCE(x, y, ...)** — возвращает первое из аргументов, которое не NULL. Удобно для подстановки значения по умолчанию: `COALESCE(phone, 'не указан')`.

**NULLIF(a, b)** — возвращает NULL, если a = b, иначе a. Нужен для приведения «нежелательного» значения к NULL: `NULLIF(col, '')` — пустую строку превратить в NULL.

**Как запомнить.**  
COALESCE = «объединить / выбрать первое непустое»: перебираешь аргументы слева направо и возвращаешь первый не NULL. NULLIF(a, b) = «сделать NULL, если a равно b»: если a и b равны — вернуть NULL, иначе вернуть a. Фраза: «COALESCE — первое не-NULL; NULLIF(a,b) — обнулить a, если a = b».

**Мини-пример: COALESCE и NULLIF по шагам.**  
- **COALESCE(phone, email, 'no contact')**: перебираем аргументы слева направо. Если phone не NULL — возвращаем phone. Если phone NULL, смотрим email; если email не NULL — возвращаем email. Если оба NULL — возвращаем 'no contact'. Пример: (phone='+7900', email='a@b.ru') → '+7900'; (phone=NULL, email='a@b.ru') → 'a@b.ru'; (phone=NULL, email=NULL) → 'no contact'.  
- **NULLIF(comment, '')**: если comment равно '' (пустая строка) — возвращаем NULL; иначе возвращаем comment. Нужно, чтобы пустые строки считались «отсутствием значения» и не мешали в сравнениях и агрегатах.

**Примеры**

```sql
SELECT id, COALESCE(phone, email, 'no contact') AS contact FROM users;
SELECT id, NULLIF(trim(comment), '') AS comment FROM feedback;
```

**Запомните**

- COALESCE — первое не-NULL; NULLIF(a, b) — NULL при a = b, иначе a.

**Вопросы для самопроверки (15.2)**  
1. Как подставить значение по умолчанию, если в столбце NULL (например, «не указан» для телефона)?  
   <details><summary>Ответ</summary> **COALESCE(col, 'значение по умолчанию')** — возвращает первый аргумент, который не NULL. Можно несколько аргументов: COALESCE(phone, email, 'no contact'). </details>  
2. Что делает NULLIF(a, b) и когда он полезен?  
   <details><summary>Ответ</summary> Возвращает **NULL**, если a = b, иначе возвращает **a**. Полезен, чтобы «обнулить» нежелательное значение: NULLIF(comment, '') превращает пустую строку в NULL, чтобы пустые не мешали в сравнениях и агрегатах. </details>  
3. В каком порядке COALESCE перебирает аргументы?  
   <details><summary>Ответ</summary> **Слева направо.** Возвращается **первый** аргумент, который не NULL. Если все NULL — результат NULL. </details>

**Проверь себя перед переходом к 15.3.**  
Ответь вслух: (1) Как подставить «значение по умолчанию», если в столбце NULL? (2) Как превратить пустую строку в NULL? Если ответил «COALESCE(col, значение)» и «NULLIF(col, '')» — можно идти к IS DISTINCT FROM.

### 15.3. IS DISTINCT FROM

**IS DISTINCT FROM** — сравнение с учётом NULL. Результат TRUE, если значения различаются **или** один NULL, а другой нет. **NULL IS DISTINCT FROM NULL** даёт FALSE (считаем, что «два NULL» не различаются). В PostgreSQL: `a IS DISTINCT FROM b`; отрицание — `a IS NOT DISTINCT FROM b`.

**Простыми словами: когда нужен IS DISTINCT FROM.**  
Обычное сравнение `a = b` при NULL даёт NULL (не TRUE): поэтому «найти строки, где статус изменился» через `o1.status <> o2.status` не сработает, если один из статусов NULL — выражение станет NULL и строка отфильтруется. **IS DISTINCT FROM** трактует NULL так: «значения различаются», только если один NULL а другой не NULL, или оба не NULL и не равны; «два NULL» считаются одинаковыми (не различаются) → FALSE. То есть это «сравнение как для человека»: NULL и NULL — не «разные», NULL и 'active' — разные. Используй IS DISTINCT FROM, когда нужно сравнивать два столбца (или версии одной записи) и корректно обрабатывать переходы в NULL и из NULL (например, история изменений, дифф версий).

**Мини-пример: обычное сравнение vs IS DISTINCT FROM.**

| a     | b     | a = b  | a <> b | a IS DISTINCT FROM b |
|-------|-------|--------|--------|------------------------|
| 1     | 1     | TRUE   | FALSE  | FALSE                 |
| 1     | 2     | FALSE  | TRUE   | TRUE                  |
| 1     | NULL  | NULL   | NULL   | TRUE                  |
| NULL  | NULL  | NULL   | NULL   | **FALSE**             |

В WHERE остаётся только TRUE. Поэтому `WHERE a <> b` не найдёт строку, где оба NULL (a <> b даёт NULL). `WHERE a IS DISTINCT FROM b` для (NULL, NULL) даёт FALSE — строка не попадёт; для (1, NULL) даёт TRUE — попадёт. Так можно находить «реально изменившиеся» пары с учётом NULL.

**Пример**

Нужно найти строки, где статус реально изменился (включая переход в NULL или из NULL):

```sql
SELECT * FROM orders o1
JOIN orders o2 ON o2.id = o1.id AND o2.version = o1.version + 1
WHERE o1.status IS DISTINCT FROM o2.status;
```

**Запомните**

- IS DISTINCT FROM — сравнение, при котором NULL не «заразен»: NULL IS DISTINCT FROM NULL → FALSE.

**Вопросы для самопроверки (15.3)**  
1. Что даёт NULL IS DISTINCT FROM NULL — TRUE или FALSE? Зачем это нужно?  
   <details><summary>Ответ</summary> **FALSE** — «два NULL» считаются **не различающимися**. Нужно, когда сравниваем две версии записи или два столбца и хотим считать «оба отсутствуют» как «одинаково» (без перехода в/из NULL). </details>  
2. Почему WHERE a <> b не находит строку, где оба значения NULL?  
   <details><summary>Ответ</summary> При a = NULL и b = NULL выражение a <> b даёт **NULL** (не TRUE). В WHERE остаются только TRUE; строка отбрасывается. Для учёта NULL при сравнении используй **a IS DISTINCT FROM b**. </details>  
3. Когда удобно использовать IS DISTINCT FROM?  
   <details><summary>Ответ</summary> Когда нужно найти **реально изменившиеся** пары (история версий, дифф), в том числе переход в NULL или из NULL. Обычное <> при NULL даёт UNKNOWN и отфильтрует строку. </details>

### 15.4. NULL в сортировке и группировке

**Сортировка:** по умолчанию в SQL порядок NULL относительно не-NULL значений не унифицирован (в PostgreSQL при ASC NULL идут последними, при DESC — первыми). Явно задать: **NULLS FIRST** или **NULLS LAST** в ORDER BY.

```sql
SELECT id, name, score FROM users ORDER BY score DESC NULLS LAST;
```

**Мини-пример: NULLS FIRST и NULLS LAST.**  
Столбец score: значения 90, NULL, 70, NULL, 80. При **ORDER BY score ASC** в PostgreSQL NULL по умолчанию в конце: 70, 80, 90, NULL, NULL. При **ORDER BY score ASC NULLS FIRST** NULL в начале: NULL, NULL, 70, 80, 90. При **ORDER BY score DESC NULLS LAST** сначала большие числа, NULL в конце: 90, 80, 70, NULL, NULL. Зачем это нужно: когда «нет оценки» не должен смешиваться с нулём как числом, или когда нужно явно вывести «без оценки» первыми/последними — задаёшь NULLS FIRST или NULLS LAST и не полагаешься на поведение СУБД по умолчанию.

**GROUP BY:** все строки с NULL в группируемом столбце попадают в **одну** группу (NULL считается одним значением для группировки). В UNIQUE-ограничении в PostgreSQL несколько NULL допустимы (NULL не равен NULL, поэтому дубликатом не считается).

**Простыми словами: почему в GROUP BY все NULL в одной группе.**  
При группировке СУБД смотрит на значение столбца и собирает строки с **одинаковым** значением в одну группу. Для NULL специальное правило: все NULL считаются «одинаковыми» в смысле группировки — они все попадают в одну группу. Поэтому GROUP BY col при наличии NULL даёт одну группа «с col = NULL», и агрегаты (COUNT, SUM и т.д.) считаются по всем этим строкам вместе.

**Запомните**

- ORDER BY: NULLS FIRST / NULLS LAST задают позицию NULL.
- В GROUP BY все NULL в одном столбце образуют одну группу.

**Вопросы для самопроверки (15.4)**  
1. Как явно задать, чтобы NULL в сортировке шли первыми или последними?  
   <details><summary>Ответ</summary> В **ORDER BY** указать **NULLS FIRST** или **NULLS LAST** (например, ORDER BY score DESC NULLS LAST). Поведение по умолчанию в разных СУБД разное; явно задать — предсказуемо. </details>  
2. В GROUP BY col: сколько групп дадут строки с NULL в col — одна или по одной на каждую строку с NULL?  
   <details><summary>Ответ</summary> **Одна группа**: все строки с NULL в col попадают в **одну** группу (в смысле группировки NULL считаются одинаковым значением). Агрегаты считаются по всем этим строкам вместе. </details>  
3. Зачем нужны NULLS FIRST / NULLS LAST в ORDER BY?  
   <details><summary>Ответ</summary> Чтобы **явно** задать позицию NULL (в начале или в конце результата), не полагаясь на поведение СУБД по умолчанию. Важно, когда «нет значения» не должно смешиваться с нулём или когда нужен стабильный порядок. </details>

### 15.5. NULL в агрегатах

- **COUNT(\*)** — считает все строки, включая те, где все столбцы NULL. **COUNT(col)** — считает только строки, где col не NULL.
- **SUM, AVG, MIN, MAX** — игнорируют NULL. Если все значения в группе NULL — результат агрегата NULL.

**Пример**

```sql
SELECT COUNT(*) AS total_rows, COUNT(phone) AS with_phone FROM users;
-- total_rows — все строки; with_phone — только с непустым phone.
```

**Мини-пример: COUNT(*) vs COUNT(col) по шагам.**  
Допустим в таблице users 5 строк: в трёх из них phone заполнен ('+79001', '+79002', '+79003'), в двух — phone = NULL.  
- **COUNT(*)** считает **все строки**: 5.  
- **COUNT(phone)** считает только строки, где phone **не NULL**: 3. Строки с phone = NULL пропускаются.  
Для SUM(total): если в группе есть 100, 200 и NULL, SUM даст 300 (NULL игнорируется). Если все значения в группе NULL, SUM даст NULL.

**Запомните**

- COUNT(*) — число строк; COUNT(col) — число не-NULL в col.
- SUM/AVG/MIN/MAX пропускают NULL.

**Вопросы для самопроверки (15.5)**  
1. Чем COUNT(*) отличается от COUNT(col) при наличии NULL в col?  
   <details><summary>Ответ</summary> **COUNT(*)** считает **все строки** (включая те, где col = NULL). **COUNT(col)** считает только строки, где **col не NULL**; строки с NULL в col пропускаются. </details>  
2. Учитывает ли SUM(col) строки с NULL в col? Что даёт SUM, если в группе все значения NULL?  
   <details><summary>Ответ</summary> **Нет** — SUM **пропускает** NULL. Суммируются только не-NULL значения. Если в группе **все** значения NULL — SUM даёт **NULL** (не 0). </details>  
3. Считает ли AVG(col) строки с NULL в col при расчёте среднего?  
   <details><summary>Ответ</summary> **Нет** — NULL **игнорируются**. Среднее считается по **не-NULL** значениям: и числитель (сумма), и знаменатель (количество не-NULL). Например, AVG(10, 20, NULL) = 15 (30/2). </details>

**Что будет, если… (NULL в агрегатах).**  
- **COUNT(*) при всех NULL в таблице:** COUNT(*) считает **строки**, а не значения. Даже если во всех столбцах NULL, строки есть — COUNT(*) вернёт их количество (например, 10).  
- **COUNT(col) при всех NULL в col:** COUNT(col) считает только не-NULL значения. Если в col везде NULL — COUNT(col) = 0.  
- **SUM(col) при NULL в группе:** NULL пропускается. Например, в группе значения 10, 20, NULL — SUM даст 30. Если в группе **все** NULL — SUM даст NULL (не 0).  
- **AVG(col) при NULL:** считаются только не-NULL значения. AVG(10, 20, NULL) = 15 (сумма 30, делённая на 2 строки).

**Проверь себя после раздела 15.**  
Ответь вслух: (1) Что считает COUNT(*) — все строки или только с не-NULL? (2) Что считает COUNT(col)? (3) Игнорирует ли SUM NULL в группе? Если ответил «все строки», «только где col не NULL», «да, игнорирует» — раздел 15 усвоен.

**Резюме раздела 15 в одном абзаце.**  
NULL — «значение неизвестно или отсутствует». В SQL трёхзначная логика: TRUE, FALSE, UNKNOWN (сравнение с NULL даёт UNKNOWN). В WHERE остаются только строки, где условие TRUE; UNKNOWN отбрасывается. Поэтому проверять на NULL можно только через IS NULL / IS NOT NULL; col = NULL бесполезно. COALESCE — первое не-NULL; NULLIF(a,b) — NULL, если a = b. IS DISTINCT FROM — сравнение с учётом NULL (NULL IS DISTINCT FROM NULL → FALSE). В ORDER BY: NULLS FIRST / NULLS LAST. В GROUP BY все NULL в одном столбце образуют одну группу. В агрегатах: COUNT(*) считает все строки, COUNT(col) — только не-NULL; SUM/AVG/MIN/MAX пропускают NULL.
