## 11. Подзапросы

**Цель раздела.**  
Научиться использовать подзапросы в разных местах: в SELECT (скалярные), в FROM (производные таблицы), в WHERE/HAVING (IN, EXISTS, ANY, ALL), понимать коррелированные подзапросы и уметь заменять подзапросы на JOIN там, где это улучшает читаемость и производительность.

**Простыми словами: о чём этот раздел.**  
Здесь мы учимся вкладывать **один запрос внутрь другого**: «одно значение» в выражении (скалярный подзапрос — например, средняя сумма заказов в каждой строке); «список» для проверки «входит/не входит» (IN); «есть ли хотя бы одна строка» (EXISTS); сравнение со множеством («больше хотя бы одного» — ANY, «больше всех» — ALL). Подзапрос в FROM даёт временную «таблицу» с именем (алиас). Коррелированный подзапрос — тот, что «привязан» к текущей строке внешнего запроса (выполняется заново для каждой строки). Многое из этого можно переписать через JOIN — когда что удобнее, разберём в разделе.

**Теоретическая основа.**  
В реляционной алгебре подзапрос в роли «множества значений» соответствует вложенному выражению. Скалярный подзапрос — выражение, возвращающее одно значение (одна строка, один столбец). Предикаты IN и EXISTS соответствуют проверке принадлежности кортежа множеству и непустоте множества. Коррелированный подзапрос формально зависит от текущей строки внешнего запроса и выполняется для каждой такой строки (или оптимизируется эквивалентным соединением).

### 11.1. Скалярный подзапрос

**Термины**

- **Подзапрос (subquery)** — запрос SELECT, вложенный в другой запрос (в SELECT, FROM, WHERE, HAVING и т.д.).
- **Скалярный подзапрос** — подзапрос, возвращающий ровно **одну строку и один столбец**. Его результат используется как одно значение (в выражении, в сравнении).

**Простыми словами:** Скалярный подзапрос — «запрос, который возвращает одно число или одну строку текста», и это значение подставляется в выражение. Например: «показать заказы и разницу суммы заказа со средней суммой по всем заказам» — средняя сумма считается скалярным подзапросом.

**Картинка в голове.**  
Представь, что в ячейке таблицы ты не пишешь число, а «вызываешь маленький запрос», который возвращает ровно одно значение (как функция). Это значение подставляется на место подзапроса. Если подзапрос вернёт 0 строк — подставится NULL. Если вернёт 2 строки — СУБД выдаст ошибку: «здесь ожидается одно значение, а пришло два».

**Как запомнить.**  
Скалярный = «одно значение» = одна строка и один столбец. Слово «скаляр» в математике значит «одно число». В SQL скалярный подзапрос — это подзапрос, который ведёт себя как одно значение: его можно писать в выражениях (SELECT, WHERE, HAVING), сравнивать с ним, складывать и т.д. Фраза: «скалярный — одна строка, один столбец; иначе ошибка или NULL».

**Правила**

- Подзапрос в месте, где ожидается одно значение, должен возвращать не более одной строки и одного столбца. Если строк больше одной — СУБД выдаст ошибку (в PostgreSQL: "more than one row returned by a subquery used as an expression").
- Если подзапрос возвращает 0 строк, результат считается NULL.

**Примеры**

В SELECT:

```sql
SELECT
    id,
    total,
    (SELECT AVG(total) FROM orders) AS avg_total,
    total - (SELECT AVG(total) FROM orders) AS diff_from_avg
FROM orders;
```

В WHERE:

```sql
SELECT id, name, salary
FROM employees
WHERE salary > (SELECT AVG(salary) FROM employees);
```

В HAVING:

```sql
SELECT department_id, COUNT(*) AS cnt
FROM employees
GROUP BY department_id
HAVING COUNT(*) > (SELECT AVG(cnt) FROM (SELECT COUNT(*) AS cnt FROM employees GROUP BY department_id) t);
```

**Граничные случаи**

- Если подзапрос возвращает несколько строк — ошибка. Ограничь результат (например, агрегатом или LIMIT 1 с ORDER BY), чтобы получить одно значение.
- Пустой результат (0 строк) даёт NULL; сравнения с NULL дают UNKNOWN, и строка может не попасть в вывод (WHERE отбрасывает UNKNOWN).

**Что будет, если… (скалярный подзапрос).**  
- **Подзапрос вернёт 0 строк:** в месте, где ожидается одно значение, подставится **NULL**. Например, (SELECT name FROM users WHERE id = 999) — если пользователя с id=999 нет, подзапрос вернёт пусто, и в ячейке будет NULL. Сравнение с NULL даёт UNKNOWN, и в WHERE такая строка может не попасть в результат.  
- **Подзапрос вернёт 2 или больше строк:** СУБД выдаст **ошибку** (в PostgreSQL: «more than one row returned by a subquery used as an expression»). В одном месте выражения может стоять только одно значение. Чтобы исправить: ограничь результат (агрегат AVG/MAX/MIN или подзапрос с LIMIT 1 и ORDER BY), чтобы гарантированно получать одну строку.

**Типичная ошибка.**  
Написать подзапрос в SELECT или WHERE, который возвращает **несколько строк** — СУБД выдаст ошибку («more than one row returned»). Всегда убеждайся, что подзапрос в месте «одного значения» возвращает не более одной строки: используй агрегат (AVG, MAX и т.д.) или LIMIT 1 с ORDER BY.

**Запомните**

- Скалярный подзапрос — одна строка, один столбец; иначе ошибка или NULL.
- Удобен в SELECT для «одного значения из другой таблицы» и в WHERE/HAVING для сравнения с агрегатом.

**Вопросы для самопроверки (11.1)**  
1. Сколько строк и столбцов должен возвращать скалярный подзапрос? Что будет, если вернётся 0 строк?  
   <details><summary>Ответ</summary> **Одна строка и один столбец.** Если 0 строк — в место подзапроса подставится **NULL**; сравнения с NULL дают UNKNOWN. </details>  
2. Почему подзапрос в SELECT, возвращающий несколько строк, вызывает ошибку?  
   <details><summary>Ответ</summary> В одном месте выражения может стоять **только одно значение**. Несколько строк — неоднозначность; СУБД выдаёт ошибку (например, «more than one row returned»). </details>  
3. Как гарантировать одну строку в подзапросе?  
   <details><summary>Ответ</summary> Использовать **агрегат** (AVG, MAX, MIN, COUNT и т.д.) или подзапрос с **LIMIT 1** и **ORDER BY**, чтобы однозначно выбрать одну строку. </details>

**Проверь себя перед переходом к 11.2.**  
Ответь вслух: (1) Сколько строк и столбцов должен вернуть скалярный подзапрос? (2) Что будет, если он вернёт 0 строк? Если ответил «одна строка, один столбец» и «подставится NULL» — можно идти дальше.

### 11.2. Подзапрос в FROM

Подзапрос в FROM задаёт **производную таблицу (derived table)**. Результат подзапроса ведёт себя как таблица: у него есть столбцы и строки, ему **обязательно** нужно дать алиас (в PostgreSQL и большинстве СУБД).

**Простыми словами.**  
Подзапрос в FROM — это «вместо имени таблицы подставляем целый запрос». Результат этого запроса ведёт себя как временная таблица: у него есть столбцы и строки. Дальше к нему можно присоединять другие таблицы (JOIN), фильтровать (WHERE) и т.д. Алиас (например, AS order_totals) **обязателен** — без него СУБД не знает, как к этому результату обращаться. Зачем это нужно: чтобы сначала посчитать что-то (например, сумму заказов по пользователям), а потом использовать этот результат в основном запросе (например, соединить с users и отфильтровать топ-10).

**Синтаксис**

```sql
SELECT ...
FROM (SELECT user_id, SUM(total) AS total FROM orders GROUP BY user_id) AS order_totals
JOIN users u ON u.id = order_totals.user_id;
```

**Пример: рейтинг пользователей по сумме заказов**

```sql
SELECT order_totals.user_id, u.name, order_totals.total,
       ROW_NUMBER() OVER (ORDER BY order_totals.total DESC) AS rank
FROM (SELECT user_id, SUM(total) AS total FROM orders GROUP BY user_id) AS order_totals
JOIN users u ON u.id = order_totals.user_id
ORDER BY rank;
```

**Разбор этого примера по шагам (что за что отвечает).**  
1. **Подзапрос в FROM:** `(SELECT user_id, SUM(total) AS total FROM orders GROUP BY user_id) AS order_totals` — сначала СУБД выполняет этот запрос. Результат: по каждому user_id одна строка с полями user_id и total (сумма заказов). Этому результату дано имя order_totals — дальше к нему обращаемся как к таблице.  
2. **JOIN users u ON u.id = order_totals.user_id:** к «таблице» order_totals присоединяем таблицу users по равенству id и user_id. В каждой строке теперь есть и данные из order_totals (user_id, total), и данные из users (u.name и др.).  
3. **SELECT ... ROW_NUMBER() OVER (...):** выводим user_id, имя, сумму и ранг. Ранг считается по убыванию total — у кого сумма больше, тот выше в рейтинге.  
4. **ORDER BY rank:** сортируем результат по рангу.  
Итого: подзапрос в FROM сначала «готовит» таблицу с агрегатами (сумма по пользователям), потом мы её соединяем с users и добавляем ранжирование. Без подзапроса в FROM пришлось бы писать агрегат и JOIN в одном уровне — сложнее читать.

**Типичная ошибка: забыть алиас у подзапроса в FROM.**  
Написать `FROM (SELECT user_id, SUM(total) AS total FROM orders GROUP BY user_id)` без `AS order_totals` — в PostgreSQL и многих СУБД будет ошибка: производная таблица должна иметь имя (алиас), чтобы к ней можно было обращаться (ORDER BY rank, ссылки на order_totals.total и т.д.). Всегда пиши `AS имя` после закрывающей скобки подзапроса в FROM.

**Запомните**

- Подзапрос в FROM обязан иметь алиас (например, AS t).
- Такой подзапрос задаёт временное множество строк для соединения и дальнейшей обработки.

**Вопросы для самопроверки (11.2)**  
1. Обязателен ли алиас у подзапроса в FROM? Зачем он нужен?  
   <details><summary>Ответ</summary> **Да**, в PostgreSQL и большинстве СУБД алиас **обязателен**. Он нужен, чтобы обращаться к результату подзапроса по имени (в JOIN, WHERE, ORDER BY и т.д.). </details>  
2. Чем подзапрос в FROM отличается от скалярного подзапроса по результату?  
   <details><summary>Ответ</summary> Подзапрос в FROM возвращает **таблицу** (много строк и столбцов) и используется как источник данных для JOIN/WHERE. Скалярный — **одно значение** (одна строка, один столбец) в выражении. </details>  
3. Когда удобно использовать подзапрос в FROM вместо одного запроса с GROUP BY и JOIN?  
   <details><summary>Ответ</summary> Когда логику удобнее разбить на шаги: сначала «посчитать агрегаты по группе» (подзапрос), потом соединить с другими таблицами или фильтровать по агрегату. Так запрос легче читать. </details>

### 11.3. IN и EXISTS

**IN (subquery)** — предикат «значение входит в множество результатов подзапроса». Подзапрос должен возвращать один столбец (или несколько — тогда сравнивается кортеж). Если подзапрос возвращает хотя бы одну строку с таким же значением — условие истинно.

**EXISTS (subquery)** — предикат «подзапрос возвращает хотя бы одну строку». Содержимое строк не важно; обычно пишут `EXISTS (SELECT 1 FROM ... WHERE ...)`. Как только найдена одна подходящая строка, результат TRUE и дальнейший поиск может не выполняться.

**Примеры**

Пользователи, у которых есть заказ больше 1000:

```sql
-- Через IN
SELECT * FROM users
WHERE id IN (SELECT user_id FROM orders WHERE total > 1000);

-- Через EXISTS
SELECT * FROM users u
WHERE EXISTS (SELECT 1 FROM orders o WHERE o.user_id = u.id AND o.total > 1000);
```

**Мини-пример: IN и EXISTS дают один и тот же результат, но работают по-разному.**  
Задача: «пользователи, у которых есть заказ больше 1000».  
- **IN:** подзапрос возвращает список user_id (например, 1, 3, 5). Для каждой строки users проверяется: id пользователя входит в этот список? Если да — строка попадает в результат. Список вычисляется **один раз**.  
- **EXISTS:** для каждой строки users выполняется подзапрос «есть ли в orders заказ с user_id = этот пользователь и total > 1000?». Если подзапрос вернул хотя бы одну строку — TRUE, строка попадает в результат. Подзапрос выполняется **для каждой строки** внешнего запроса (корреляция). В обоих случаях результат — одни и те же пользователи. EXISTS часто удобнее, когда условие сложное или когда нужна проверка «есть/нет» без явного списка; при NULL в подзапросе NOT EXISTS ведёт себя предсказуемо, в отличие от NOT IN.

**IN и NULL — почему NOT IN может «сломаться».**  
Если подзапрос в IN возвращает NULL, сравнение value = NULL в SQL даёт не TRUE и не FALSE, а **NULL** (неизвестно). Поэтому `x NOT IN (subquery)` при наличии NULL в подзапросе может дать **пустой результат**: для каждой строки подзапроса проверяется x = значение; если значение NULL, то x = NULL даёт NULL; NOT NULL тоже NULL; в WHERE остаются только строки, где условие TRUE — а у нас NULL, значит строка отбрасывается. В итоге может отброситься всё. Для «нет в списке» безопаснее **NOT EXISTS** или **LEFT JOIN ... WHERE right.key IS NULL** — они не зависят от NULL в списке так критично.

**Как запомнить.**  
IN = «значение входит в список». EXISTS = «есть хотя бы одна строка» (содержимое строк не важно, важен факт «есть/нет»). NOT IN с NULL в подзапросе — опасен; предпочтительнее NOT EXISTS.

**Запомните**

- IN (subquery) — «значение входит в множество»; подзапрос — один столбец (или кортеж).
- EXISTS (subquery) — «есть хотя бы одна строка»; часто с корреляцией по внешней таблице.
- При NULL в подзапросе NOT IN может вести себя неочевидно; предпочтительнее NOT EXISTS.

**Вопросы для самопроверки (11.3)**  
1. Что проверяет IN (подзапрос): «значение входит в список» или «есть хотя бы одна строка»?  
   <details><summary>Ответ</summary> **«Значение входит в список»**: подзапрос возвращает один столбец (или кортеж); проверяется, совпадает ли значение с хотя бы одной строкой результата. </details>  
2. Почему для «пользователей без заказов» лучше NOT EXISTS, а не NOT IN (подзапрос)?  
   <details><summary>Ответ</summary> При **NULL** в подзапросе сравнение value = NULL даёт UNKNOWN; NOT IN может дать **пустой результат** или неверный. NOT EXISTS ведёт себя предсказуемо и не зависит от NULL в подзапросе так критично. </details>  
3. Сколько раз выполняется подзапрос при IN и при EXISTS с корреляцией?  
   <details><summary>Ответ</summary> При **IN** подзапрос без корреляции выполняется **один раз**; при **EXISTS** с корреляцией (ссылка на внешнюю таблицу) — **для каждой строки** внешнего запроса. </details>

**Проверь себя перед переходом к 11.4.**  
Ответь вслух: (1) IN (подзапрос) — что проверяем: «значение входит в список» или «есть хотя бы одна строка»? (2) Для «пользователей, у которых нет заказов» что безопаснее — NOT IN (подзапрос) или NOT EXISTS? Если ответил «значение входит в список» и «NOT EXISTS» — можно идти к ANY/ALL.

### 11.4. ANY и ALL

**ANY (или SOME)** — сравнение значения с **каждым** из результатов подзапроса; достаточно одного совпадения. Синтаксис: `value operator ANY (subquery)`.

**ALL** — сравнение с **всеми** результатами подзапроса; условие должно быть истинно для каждой строки подзапроса.

**Простыми словами.**  
ANY = «хотя бы для одного». Условие `salary > ANY (подзапрос)` значит: «зарплата больше хотя бы одного значения из подзапроса» — то есть достаточно одного совпадения. ALL = «для всех». Условие `salary > ALL (подзапрос)` значит: «зарплата больше каждого значения из подзапроса» — должно быть истинно для всех строк подзапроса. `= ANY (подзапрос)` по смыслу то же, что IN (подзапрос).

**Мини-пример «до и после» (ANY vs ALL).**  
Подзапрос возвращает зарплаты в отделе 5: 50, 60, 70 (три сотрудника). Проверяем сотрудника с зарплатой 65.  
- **salary > ANY (подзапрос)** — «65 больше хотя бы одного из (50, 60, 70)?» Да: 65 > 50, 65 > 60. Условие **истинно** — сотрудник попадёт в результат.  
- **salary > ALL (подзапрос)** — «65 больше **каждого** из (50, 60, 70)?» Нет: 65 не больше 70. Условие **ложно** — сотрудник не попадёт.  
Если бы зарплата была 80: > ANY — истинно (80 > 50, 60, 70); > ALL — истинно (80 больше всех). Итого: ANY = «достаточно одного совпадения»; ALL = «должно выполняться для всех значений из подзапроса». Пустой подзапрос: > ALL (пусто) в большинстве СУБД даёт TRUE (нет ни одного значения, для которого условие ложно); > ANY (пусто) даёт FALSE (нет ни одного совпадения).

**Примеры**

```sql
-- Сотрудники с зарплатой выше хотя бы одного в отделе 5
SELECT * FROM employees
WHERE salary > ANY (SELECT salary FROM employees WHERE department_id = 5);

-- Сотрудники с зарплатой выше всех в отделе 5
SELECT * FROM employees
WHERE salary > ALL (SELECT salary FROM employees WHERE department_id = 5);
```

Эквивалентности:
- `= ANY (subquery)` совпадает с `IN (subquery)`.
- `<> ALL (subquery)` при непустом подзапросе совпадает с `NOT IN (subquery)` (с теми же оговорками по NULL).

**Запомните**

- ANY — «хотя бы для одного»; ALL — «для всех».
- `= ANY` эквивалентен IN; с ALL и NOT IN осторожнее с пустым подзапросом и NULL.

**Вопросы для самопроверки (11.4)**  
1. Чем условие salary > ANY (подзапрос) отличается от salary > ALL (подзапрос)?  
   <details><summary>Ответ</summary> **ANY** — «больше **хотя бы одного**» значения из подзапроса (достаточно одного совпадения). **ALL** — «больше **каждого**» значения (должно выполняться для всех строк подзапроса). </details>  
2. Какой предикат эквивалентен IN (подзапрос)?  
   <details><summary>Ответ</summary> **= ANY (подзапрос)** — по смыслу то же, что IN (подзапрос): значение равно хотя бы одному из результатов. </details>  
3. Что даёт > ALL (пустой подзапрос) в типичной СУБД?  
   <details><summary>Ответ</summary> В большинстве СУБД **TRUE**: «больше каждого из пустого множества» — нет ни одного значения, для которого условие ложно. С пустым подзапросом и NULL осторожнее. </details>

### 11.5. Коррелированные подзапросы

**Коррелированный подзапрос** — подзапрос, в котором есть ссылка на столбец (или выражение) внешнего запроса. Он логически выполняется для каждой строки внешнего запроса, поэтому при больших объёмах данных может быть медленным. Часто такой подзапрос можно переписать через JOIN или оконную функцию.

**Картинка в голове.**  
Представь цикл по строкам внешнего запроса: для каждой строки (например, каждого сотрудника) ты «вызываешь» подзапрос, который использует значение из этой строки (например, department_id). Подзапрос возвращает одно значение (например, среднюю зарплату по этому отделу), и ты сравниваешь: зарплата сотрудника больше этой средней? Так подзапрос «привязан» к текущей строке — это и есть корреляция. Без этой привязки подзапрос был бы один на весь запрос; с корреляцией он выполняется много раз (по разу на строку).

**Пример**

Сотрудники, чья зарплата выше средней по их отделу:

```sql
SELECT e.id, e.name, e.department_id, e.salary
FROM employees e
WHERE e.salary > (SELECT AVG(salary) FROM employees i WHERE i.department_id = e.department_id);
```

Внутренний подзапрос зависит от `e.department_id` — это корреляция.

**Мини-пример: как работает корреляция по шагам.**  
Сотрудники: (id=1, dept=5, salary=80), (id=2, dept=5, salary=50), (id=3, dept=7, salary=90). Отделы: в 5 средняя зарплата 65, в 7 — 90. Для **строки 1** (dept=5) подзапрос считает AVG по department_id=5 → 65; 80 > 65 → TRUE, строка 1 попадает в результат. Для **строки 2** (dept=5) подзапрос снова считает AVG по отделу 5 → 65; 50 > 65 → FALSE, строка 2 не попадает. Для **строки 3** (dept=7) подзапрос считает AVG по отделу 7 → 90; 90 > 90 → FALSE, не попадает. Итого в результате одна строка (сотрудник 1). Подзапрос выполнился **три раза** — по разу для каждой строки employees; каждый раз с «текущим» e.department_id. Это и есть корреляция: подзапрос «привязан» к текущей строке внешнего запроса.

Эквивалент через JOIN с агрегатом:

```sql
SELECT e.id, e.name, e.department_id, e.salary
FROM employees e
JOIN (SELECT department_id, AVG(salary) AS avg_sal FROM employees GROUP BY department_id) a
  ON a.department_id = e.department_id
WHERE e.salary > a.avg_sal;
```

**Запомните**

- Коррелированный подзапрос ссылается на столбцы внешнего запроса и выполняется в контексте каждой строки.
- При возможности заменяй на JOIN или оконную функцию для лучшей производительности.

**Вопросы для самопроверки (11.5)**  
1. Что такое коррелированный подзапрос и чем он отличается от обычного?  
   <details><summary>Ответ</summary> Коррелированный подзапрос **ссылается на столбцы внешнего запроса** (например, e.department_id). Он выполняется **для каждой строки** внешнего запроса. Обычный подзапрос не зависит от внешних столбцов и выполняется один раз. </details>  
2. Почему коррелированный подзапрос может быть медленным?  
   <details><summary>Ответ</summary> Он выполняется **много раз** (по разу на каждую строку внешнего запроса). При больших таблицах это даёт много обращений к данным; JOIN или оконная функция часто дают один проход. </details>  
3. Как переписать «зарплата выше средней по отделу» без коррелированного подзапроса?  
   <details><summary>Ответ</summary> **JOIN с подзапросом-агрегатом в FROM**: сначала посчитать среднюю зарплату по отделам (GROUP BY department_id), соединить с employees по department_id и сравнить salary с этой средней. Или использовать **оконную функцию** AVG() OVER (PARTITION BY department_id). </details>

### 11.6. Замена подзапросов на JOIN

Многие подзапросы можно выразить через JOIN:

- «Строки из A, для которых есть пара в B» → INNER JOIN с B или EXISTS.
- «Строки из A, для которых нет пары в B» → LEFT JOIN B ... WHERE B.key IS NULL или NOT EXISTS.
- Скалярный подзапрос в SELECT, возвращающий одно значение из связанной таблицы → LEFT JOIN и взять столбец из правой таблицы.

Замена не только улучшает читаемость, но и даёт оптимизатору больше вариантов (один общий план вместо «подзапрос на каждую строку»). Не всегда JOIN быстрее (зависит от индексов и распределения данных), но часто.

**Мини-пример «до и после»: скалярный подзапрос в SELECT → LEFT JOIN.**  
Задача: вывести заказы с именем клиента. **Вариант 1 (скалярный подзапрос):** `SELECT o.id, o.total, (SELECT name FROM users u WHERE u.id = o.user_id) AS user_name FROM orders o` — для каждой строки заказа подзапрос ищет имя по user_id; подзапрос выполняется для каждой строки. **Вариант 2 (LEFT JOIN):** `SELECT o.id, o.total, u.name AS user_name FROM orders o LEFT JOIN users u ON u.id = o.user_id` — один раз соединяем заказы с пользователями по user_id и берём u.name. Результат тот же (заказы с именами клиентов), но запрос читается проще и часто выполняется эффективнее (один проход соединения вместо подзапроса на каждую строку). Запомни: «одно значение из связанной таблицы» в SELECT удобнее через LEFT JOIN и столбец справа, чем через скалярный подзапрос.

**Сводка: подзапрос vs JOIN (когда что удобнее).**  
- «Есть ли пара в другой таблице» (пользователи, у которых есть заказ > 1000) → **IN** или **EXISTS**; то же можно через **INNER JOIN** с условием и DISTINCT, но IN/EXISTS читаются проще.  
- «Нет пары в другой таблице» (пользователи без заказов) → **LEFT JOIN + WHERE right.key IS NULL** или **NOT EXISTS**; NOT IN опасен при NULL в подзапросе.  
- «Одно значение из связанной таблицы» в SELECT (имя клиента к заказу) → **LEFT JOIN** и взять столбец справа проще, чем скалярный подзапрос в SELECT.  
- «Сравнение с агрегатом по группе» (зарплата выше средней по отделу) → коррелированный подзапрос возможен, но **JOIN с подзапросом-агрегатом в FROM** или **оконная функция** часто понятнее и быстрее.

**Запомните**

- IN/EXISTS и «есть/нет пары» часто переписываются в JOIN.
- Скалярный подзапрос в SELECT — в LEFT JOIN и выбор столбца.

**Вопросы для самопроверки (11.6)**  
1. Как переписать «пользователи, у которых есть заказ > 1000» с IN на JOIN?  
   <details><summary>Ответ</summary> **INNER JOIN** orders по user_id с условием total > 1000 и при необходимости **DISTINCT** по пользователю (если у пользователя несколько таких заказов и нужна одна строка на пользователя). Или EXISTS без изменения логики. </details>  
2. Вместо скалярного подзапроса в SELECT «имя клиента к заказу» что лучше использовать?  
   <details><summary>Ответ</summary> **LEFT JOIN users ON user_id = users.id** и в SELECT брать столбец имени из users. Один проход соединения вместо подзапроса на каждую строку заказа. </details>  
3. Зачем заменять подзапрос на JOIN кроме читаемости?  
   <details><summary>Ответ</summary> Оптимизатор получает **один общий план** (один проход по данным) вместо «подзапрос на каждую строку». Часто это быстрее; не всегда — зависит от индексов и данных. </details>

**Резюме раздела 11 в одном абзаце.**  
Подзапрос — это SELECT внутри другого запроса. Скалярный подзапрос возвращает одно значение (одна строка, один столбец) и используется в выражениях (SELECT, WHERE, HAVING). Подзапрос в FROM задаёт временную таблицу и обязан иметь алиас. IN (подзапрос) — «значение входит в список»; EXISTS — «есть хотя бы одна строка». ANY/ALL — сравнение со множеством («хотя бы для одного» / «для всех»). Коррелированный подзапрос ссылается на внешний запрос и выполняется для каждой строки; часто его можно заменить на JOIN или оконную функцию. NOT IN с NULL в подзапросе опасен — лучше NOT EXISTS.

**Проверь себя перед разделом 12.**  
Ответь вслух: (1) Что делает GROUP BY со строками — оставляет все строки или «схлопывает» в одну на группу? (2) Если нужен «рейтинг по группе» (номер строки внутри каждого пользователя), что логичнее — GROUP BY или что-то другое? Если ответил «схлопывает в одну на группу» и «что-то другое, потому что при GROUP BY строк станет меньше» — ты готов к разделу про оконные функции: они как раз дают «результат по группе» **без** схлопывания строк.
