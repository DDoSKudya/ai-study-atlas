## §45. SQLite

### 45.1. Модель и ограничения

**Цель раздела.**  
Понять **модель SQLite**: встраиваемая СУБД, **один файл** (или in-memory), **один писатель** в момент времени; типы (affinity); когда SQLite уместен (приложения, тесты, прототипы, встроенные устройства). После раздела ты будешь знать ограничения и сценарии применения.

**В этом разделе главное (три строки).** 1) **SQLite** — не сетевой сервер: **библиотека** в процессе приложения; база — один файл (или `:memory:`). 2) **Один писатель** — в любой момент только одна транзакция может изменять базу; читателей может быть много (в WAL mode читатели не блокируются писателем в большинстве случаев). 3) **Когда использовать** — локальные приложения, тесты, прототипы, встроенные устройства, кэш; **не** для множества конкурентных писателей и доступа по сети с многих узлов.

#### Термины (расшифровка)

- **SQLite** — **встраиваемая** реляционная СУБД: это **библиотека**, а не отдельный серверный процесс. Приложение подключает библиотеку и открывает базу — обычно **один файл** на диске (например, app.db) или **in-memory** (`:memory:` — данные только в RAM, при закрытии соединения исчезают). **Нет сетевого протокола** «из коробки» — доступ к базе только из того процесса, который открыл файл. Поддерживает большую часть SQL-92, транзакции (ACID), индексы, внешние ключи (включаются отдельно: PRAGMA foreign_keys=ON). Идеально для приложений на одном устройстве (десктоп, мобильное, встроенное), для тестов и прототипов.
- **Один писатель** — в любой момент времени только **одна** транзакция может **изменять** базу (INSERT/UPDATE/DELETE). Если вторая сессия пытается писать, она ждёт завершения первой (или получает ошибку/таймаут в зависимости от настроек). **Читателей** может быть много одновременно — они не блокируют друг друга. В **WAL mode** читатели обычно не блокируются писателем: пока один пишет в WAL, другие могут читать из основной БД и WAL. **Итог:** SQLite подходит для нагрузки «много чтений, мало записей» или «один писатель»; для множества конкурентных писателей лучше серверная СУБД (PostgreSQL, MySQL).
- **Type affinity** — в SQLite столбцу задаётся **рекомендация** по типу (INTEGER, TEXT, BLOB, REAL, NUMERIC), но фактически можно сохранить значение **любого** типа — SQLite приведёт по правилам (например, число в TEXT сохранится как строка). Это **гибкая типизация**: удобно для прототипов, но для строгой схемы и переноса в другие СУБД нужно следить за типами в приложении.

#### Теория и правила (подробно)

- **Ограничения.** **Нет сетевого доступа** — база открывается как файл; доступ только из процесса приложения. Для доступа с нескольких серверов по сети нужен промежуточный слой (приложение-сервер открывает SQLite и отдаёт данные по API). **Один писатель** — при высокой конкуренции на запись запросы выстраиваются в очередь. **ALTER TABLE** ограничен (добавление столбца в конец, переименование — есть; удаление столбца — в новых версиях). **Нет пользователей/ролей** как в PostgreSQL/MySQL — права доступа к файлу БД задаются на уровне ОС (файловая система). **Вывод:** SQLite — для одного приложения на одном хосте или для тестов/кэша.
- **Когда уместен.** **Локальные приложения** (десктоп, мобильные) — один файл БД, не нужен отдельный сервер. **Тесты** — быстрый старт, один файл или :memory:, изоляция тестов. **Прототипы** — быстро поднять схему и данные без установки PostgreSQL/MySQL. **Встроенные устройства** — малый размер библиотеки, один файл. **Кэш в приложении** — локальное хранилище. **Не подходит:** множество конкурентных писателей (очередь записи станет узким местом); доступ с многих серверов по сети (нет встроенного сервера); очень большие объёмы с высокой нагрузкой без шардирования.

#### Пошагово: как открыть базу SQLite (концептуально)

1. Приложение подключает библиотеку SQLite и вызывает «открыть файл» (например, sqlite3_open("app.db", &db)). Создаётся один файл app.db на диске (или используется существующий).
2. Все запросы (SELECT, INSERT, …) идут через это соединение в том же процессе. Другой процесс может открыть тот же файл (несколько читателей; при записи один писатель).
3. При закрытии соединения файл остаётся на диске (данные сохранены). Для in-memory открывают с путём ":memory:" — данные только в RAM.

#### Что будет, если открыть одну и ту же базу SQLite из многих процессов с активной записью

Несколько процессов могут открыть один и тот же файл БД. **Читать** могут все. **Писать** в любой момент может только один; остальные при попытке записи получат **SQLITE_BUSY** (или ждут по таймауту). При высокой конкуренции на запись производительность падает — все писатели ждут друг друга. **Вывод:** для многописательной нагрузки с нескольких процессов лучше серверная СУБД (PostgreSQL, MySQL) с пулом соединений.

#### Типичная ошибка

Пытаются использовать SQLite как общую БД для веб-приложения с множеством воркеров/процессов, каждый из которых пишет в один файл. Получают SQLITE_BUSY, блокировки и низкую пропускную способность записи. **Правильно:** для веб-приложения с множеством одновременных записей использовать серверную СУБД (PostgreSQL, MySQL); SQLite — для одного процесса или для тестов/кэша.

#### Часто путают

**SQLite** и **серверные СУБД** (PostgreSQL, MySQL). В серверных СУБД есть **отдельный процесс-сервер** и **сетевой протокол** — клиенты подключаются по сети (TCP). В SQLite **нет сервера** — библиотека работает внутри процесса приложения и обращается к файлу на диске. «Подключиться к SQLite по сети» из коробки нельзя — только открыть файл из приложения.

#### Простыми словами

SQLite — «БД в одном файле» внутри твоего приложения: нет отдельного сервера, нет сети. Писать может только один в момент времени; читателей — много. Типы гибкие (affinity). Идеален для приложения на одном компьютере, для тестов и прототипов; для тяжёлой многописательной нагрузки и сетевого доступа с многих узлов лучше серверная СУБД.

#### Картинка в голове

**SQLite** — **блокнот в твоём кармане**: один файл (или :memory: — в голове). Нет «сервера-секретаря» — ты сам открываешь блокнот из своего приложения. **Один писатель** — ручка одна; когда ты пишешь, другой процесс не может писать в тот же блокнот (ждёт или получает SQLITE_BUSY). Читать могут многие — смотреть в блокнот можно одновременно. Для «много людей пишут в один блокнот по сети» такой блокнот не подходит — нужен общий офис (серверная СУБД).

#### Как запомнить

**SQLite** = встраиваемая; один файл или :memory:; **один писатель**, много читателей; нет сетевого сервера. **Сценарии** — приложения, тесты, прототипы, встроенные устройства; не для множества конкурентных писателей и доступа по сети.

#### Проверь себя

1. Почему SQLite не подходит для «много серверов пишут в одну БД по сети»?  
<details><summary>Ответ</summary> SQLite — **встраиваемая** СУБД: нет отдельного серверного процесса и **сетевого протокола**. Доступ к базе только из процесса, который открыл файл. Чтобы «писать по сети», нужен промежуточный слой (приложение-сервер с API), который один открывает SQLite; тогда всё равно будет **один писатель** (этот сервер). Плюс один писатель — узкое место при высокой нагрузке записи.</details>

2. Что значит «один писатель» в SQLite?  
<details><summary>Ответ</summary> В любой момент времени только **одна** транзакция может выполнять **изменения** (INSERT/UPDATE/DELETE). Остальные писатели ждут (или получают SQLITE_BUSY). **Читателей** может быть много одновременно. Поэтому SQLite подходит для нагрузки с доминированием чтения или с одним источником записи.</details>

**Три пункта, чтобы не забыть (SQLite).** 1) **Встраиваемая** — библиотека в процессе приложения; один файл или :memory:. 2) **Один писатель** — запись последовательна; много читателей. 3) **Когда использовать** — приложения, тесты, прототипы; не для множества конкурентных писателей и доступа по сети.

#### Запомните

- **SQLite** — встраиваемая; один файл или :memory:; один писатель, много читателей.
- **Типы** — affinity (INTEGER, TEXT, BLOB, REAL, NUMERIC); хранение гибкое.
- **Сценарии** — приложения, тесты, прототипы, встроенные устройства; не для высоконагруженного многописательного доступа по сети.

**Ещё раз самыми простыми словами.** SQLite — БД в одном файле, без отдельного сервера. Писать может только один в момент; читателей — много. Удобен для приложения на одном компьютере и для тестов; для множества писателей и доступа по сети — серверная СУБД.

**Если прочитал и всё равно не понял.** Запомни одну фразу: **SQLite = один файл с базой внутри твоего приложения, без сетевого сервера; писать может только один в момент.** Подходит для программ на одном компьютере и для тестов. Не подходит, когда много процессов/серверов одновременно пишут в одну БД по сети.

### 45.2. WAL и PRAGMA

**Цель раздела.**  
Использовать **WAL mode** в SQLite (улучшение параллелизма чтение/запись) и основные **PRAGMA** (journal_mode, synchronous, cache_size). После раздела ты будешь знать, как включить WAL и какие настройки задать через PRAGMA.

**В этом разделе главное (три строки).** 1) **WAL (Write-Ahead Logging)** — режим журнала: изменения сначала пишутся в отдельный файл WAL, затем периодически переносятся в основную БД (checkpoint). Читатели видят консистентный снимок без блокировки писателя; запись часто быстрее. 2) **PRAGMA** — директивы настройки SQLite: задаются в сессии или при открытии БД. 3) Важные PRAGMA: **journal_mode=WAL**, **synchronous** (NORMAL/FULL), **cache_size**, **foreign_keys**.

#### Термины (расшифровка)

- **WAL mode** — режим журнала в SQLite; запись идёт в файл database-wal; читатели продолжают читать основную БД + WAL; checkpoint переносит изменения из WAL в основный файл. Улучшает параллелизм и часто производительность записи.
- **PRAGMA** — команда настройки SQLite. Примеры: PRAGMA journal_mode=WAL; PRAGMA synchronous=NORMAL; PRAGMA cache_size=-64000; (64 MB кэша). Некоторые PRAGMA действуют только на текущее соединение.
- **synchronous** — NORMAL (быстрее, при сбое возможна потеря последних операций), FULL (каждая фиксация сбрасывается на диск — безопаснее, медленнее).

#### Теория и правила (подробно)

- **Включение WAL.** PRAGMA journal_mode=WAL; — переключает в WAL. Обратно в delete/truncate mode: PRAGMA journal_mode=DELETE; (или TRUNCATE).
- **Практика.** Для большинства приложений WAL даёт лучший баланс скорости и безопасности. synchronous=NORMAL в WAL обычно достаточно (целостность сохраняется; при сбое питания возможна потеря последних миллисекунд). Для максимальной долговечности — FULL.

#### Примеры

```sql
-- Включить WAL
PRAGMA journal_mode=WAL;

-- Настройки (типичные для локального приложения)
PRAGMA synchronous=NORMAL;
PRAGMA cache_size=-64000;   -- 64 MB (отрицательное = KB)
PRAGMA foreign_keys=ON;     -- включить проверку FK

-- Узнать текущие значения
PRAGMA journal_mode;
PRAGMA synchronous;
```

#### Что будет, если оставить journal_mode по умолчанию (DELETE) при смешанной нагрузке чтение/запись

В режиме **DELETE** (классический rollback journal) при записи SQLite блокирует **всю базу** на время транзакции — читатели могут ждать писателя (или наоборот, в зависимости от настроек). В **WAL mode** запись идёт в отдельный файл WAL; читатели продолжают читать основную БД и WAL — параллелизм чтение/запись лучше. **Вывод:** для смешанной нагрузки включай **PRAGMA journal_mode=WAL;** — меньше блокировок и часто выше производительность записи.

#### Типичная ошибка

Забывают включить WAL и удивляются блокировкам при записи. **Правильно:** после открытия базы выполнить **PRAGMA journal_mode=WAL;** (один раз; сохраняется в файле). Для проверки FK не забывать **PRAGMA foreign_keys=ON;** (каждое соединение должно включить его при открытии).

#### Часто путают

**WAL в SQLite** и **WAL в PostgreSQL** — разные реализации. В SQLite WAL — это один файл database-wal, куда пишутся изменения, и checkpoint переносит их в основной файл; в PostgreSQL WAL — поток секций (сегментов) журнала, используемый для восстановления и репликации. Не переноси настройки и ожидания из одной СУБД в другую. **PRAGMA** — только в SQLite; в PostgreSQL настройки задаются через ALTER SYSTEM, postgresql.conf или SET.

#### Картинка в голове (WAL и PRAGMA)

**WAL** — **черновик отдельно от тетради**: ты пишешь в черновик (файл database-wal); потом периодически переносишь в тетрадь (основная БД — checkpoint). Пока ты пишешь в черновик, другие могут **читать** из тетради и из черновика — не нужно ждать. В старом режиме (DELETE) при записи тетрадь была «занята» — читатели ждали. **PRAGMA** — **ручки настройки** SQLite: одна ручка — «режим журнала» (WAL или DELETE), другая — «как часто сбрасывать на диск» (synchronous), третья — размер кэша, четвёртая — включить проверку внешних ключей.

#### Как запомнить

**WAL** — PRAGMA journal_mode=WAL; запись в отдельный WAL-файл; checkpoint переносит в основную БД; лучше параллелизм чтение/запись. **PRAGMA** — journal_mode, synchronous (NORMAL/FULL), cache_size, foreign_keys.

#### Проверь себя

1. Зачем включать WAL mode в SQLite?  
<details><summary>Ответ</summary> **WAL mode** улучшает **параллелизм** чтение/запись: запись идёт в отдельный WAL-файл, читатели продолжают читать основную БД и WAL без блокировки на время записи. В классическом режиме (DELETE) при записи база блокируется на время транзакции.</details>

2. Что делает PRAGMA synchronous?  
<details><summary>Ответ</summary> **synchronous** задаёт, как часто SQLite **сбрасывает** данные на диск после фиксации транзакции. **NORMAL** — быстрее, при сбое питания возможна потеря последних операций. **FULL** — каждая фиксация сбрасывается на диск — безопаснее, медленнее. В WAL mode обычно достаточно NORMAL.</details>

**Три пункта, чтобы не забыть (WAL и PRAGMA).** 1) **WAL** — PRAGMA journal_mode=WAL; лучше параллелизм; checkpoint переносит WAL в основную БД. 2) **PRAGMA** — journal_mode, synchronous, cache_size, foreign_keys. 3) synchronous=NORMAL в WAL — компромисс скорость/безопасность.

#### Запомните

- **WAL mode** — PRAGMA journal_mode=WAL; лучше параллелизм чтение/запись; checkpoint переносит WAL в основную БД.
- **PRAGMA** — journal_mode, synchronous (NORMAL/FULL), cache_size, foreign_keys.
- synchronous=NORMAL в WAL — компромисс скорость/безопасность; FULL — максимальная долговечность.

**Ещё раз самыми простыми словами.** WAL — «сначала пишем в отдельный лог, потом переносим в основную базу»; читатели при этом не мешают писателю. PRAGMA — настройки SQLite: режим журнала (WAL), как сбрасывать на диск (synchronous), размер кэша, внешние ключи.

**Если прочитал и всё равно не понял.** Главное: **PRAGMA journal_mode=WAL;** — включить WAL; после этого запись и чтение лучше сосуществуют. **PRAGMA synchronous=NORMAL;** — быстрее, при сбое питания возможна потеря последних миллисекунд. **PRAGMA foreign_keys=ON;** — включить проверку внешних ключей (нужно при каждом открытии соединения).
