## §46. Другие реляционные СУБД

### 46.1. Oracle и SQL Server

**Цель раздела.**  
Получить краткий обзор **Oracle** (RAC, PL/SQL, лицензии) и **SQL Server** (T-SQL, Always On, экосистема Microsoft). После раздела ты будешь знать основные отличия и типичные сценарии использования.

**В этом разделе главное (три строки).** 1) **Oracle** — коммерческая СУБД; **RAC** — кластер экземпляров с общей БД; **PL/SQL** — процедурный язык; мощная аналитика, партиционирование; лицензии дорогие. 2) **SQL Server** — СУБД Microsoft; **T-SQL** — диалект SQL; интеграция с Windows/Azure/.NET; **Always On** — группы доступности для HA и репликации. 3) Обе — зрелые корпоративные СУБД; выбор часто по уже имеющейся инфраструктуре и лицензиям.

#### Термины (расшифровка)

- **Oracle RAC (Real Application Clusters)** — технология Oracle, при которой **несколько экземпляров** Oracle на **разных серверах** работают с **одной общей базой данных** (данные на shared storage или кластерном хранилище). При отказе одного узла остальные продолжают обслуживать запросы — отказоустойчивость без простоя. Сложность и стоимость высокие; типично для критичных корпоративных систем.
- **PL/SQL** — **процедурный язык** Oracle: расширение SQL циклами, условиями, переменными, процедурами, функциями, пакетами, триггерами. Богатая стандартная библиотека; много бизнес-логики в корпорациях написано на PL/SQL. Для переноса приложений с Oracle на PostgreSQL аналог — PL/pgSQL.
- **SQL Server Always On** — механизм **высокой доступности и репликации** в SQL Server: создаётся **группа доступности (Availability Group)** — первичная реплика принимает запись, одна или несколько вторичных реплик получают изменения (синхронно или асинхронно). При отказе primary можно переключиться (failover) на вторичную реплику. Аналог комбинации streaming replication + автоматический failover в PostgreSQL.
- **T-SQL (Transact-SQL)** — **диалект SQL** в SQL Server: стандартный SQL плюс расширения Microsoft для процедур, переменных, управления потоком (IF, WHILE, TRY/CATCH и т.д.). Скрипты и хранимые процедуры пишутся на T-SQL. При переносе с SQL Server на другую СУБД часть T-SQL нужно переписать (есть отличия от стандарта и от PostgreSQL/MySQL).

#### Теория и правила (подробно)

- **Когда выбирают Oracle.** Уже есть лицензии и экспертиза; требования к масштабу, аналитике и надёжности (RAC, партиционирование, сжатие, богатый PL/SQL). Часто в связке с Java-экосистемой и корпоративными стандартами. Минусы — высокая стоимость лицензий и поддержки.
- **Когда выбирают SQL Server.** Инфраструктура на Windows; экосистема Microsoft (Azure, .NET, Visual Studio, SSIS, SSRS); нужна интеграция с Active Directory, Excel, другими продуктами Microsoft. Always On даёт отказоустойчивость и репликацию. Лицензии тоже платные (есть бесплатные редакции с ограничениями — Express, Developer).

#### Пошагово: как сориентироваться при выборе (кратко)

1. Уже есть **лицензии и команда** под Oracle или SQL Server? — часто остаются на том же.  
2. Стек **Windows + .NET**? — SQL Server естественно вписывается.  
3. Нужна **максимальная совместимость со стандартом SQL** и открытый код? — PostgreSQL.  
4. Нужен **кластер без shared storage** (RAC-подобный)? — Oracle RAC или решения поверх PostgreSQL (Patroni, Citus), или NewSQL (CockroachDB, TiDB).

#### Что будет, если переносить приложение с Oracle на PostgreSQL «как есть»

Синтаксис SQL и типы данных отличаются (например, Oracle NUMBER vs PostgreSQL NUMERIC; Oracle DATE включает время; кавычки для идентификаторов, последовательности, псевдонимы). PL/SQL не совместим с PL/pgSQL — процедуры и триггеры нужно переписывать. **Вывод:** перенос возможен, но требует анализа отличий и адаптации кода и схемы; есть инструменты миграции (Oracle FDW, ora2pg и др.).

#### Типичная ошибка

Считать Oracle и SQL Server «одинаковыми» реляционными СУБД — у них разные диалекты SQL, разные механизмы репликации и HA, разная стоимость и экосистема. Выбор «потому что все так делают» без учёта стека и лицензий может привести к лишним затратам или к неудобной интеграции.

#### Часто путают

**Oracle RAC** и **SQL Server Always On** — оба дают отказоустойчивость, но по-разному. **RAC** — несколько экземпляров Oracle с **одной общей БД** на shared storage; при отказе узла остальные продолжают работать с той же БД. **Always On** — один primary + реплики; репликация потока изменений; при failover реплика становится primary. **PL/SQL** и **T-SQL** — разные языки; код с Oracle не переносится в SQL Server «как есть» и наоборот. **Oracle** и **PostgreSQL** — разная лицензия (коммерческая vs открытая), разный диалект SQL и типы данных; миграция требует адаптации.

#### Простыми словами

Oracle — «тяжёлая» корпоративная СУБД с кластеризацией (RAC) и своим языком (PL/SQL); лицензии дорогие. SQL Server — «своя» СУБД Microsoft; удобна в экосистеме Windows/Azure; Always On даёт отказоустойчивость и репликацию. Выбор часто из-за уже купленных лицензий и стека (Java/Oracle vs .NET/SQL Server).

#### Картинка в голове

**Oracle** — **большой банк** с несколькими офисами (RAC): один «сейф» (общая БД), но обслуживать клиентов могут несколько офисов; если один упал, остальные работают. Внутри — свой язык инструкций (PL/SQL). **SQL Server** — **банк Microsoft**: свой диалект (T-SQL), тесно с Windows и Azure; Always On — «запасной офис» (реплика), куда переключаются при сбое основного. Выбор часто «у нас уже контракт с этим банком» (лицензии, команда).

#### Как запомнить

**Oracle** — RAC (кластер экземпляров), PL/SQL; дорогие лицензии; масштаб, аналитика. **SQL Server** — T-SQL, Always On; интеграция с Windows/Azure/.NET. Выбор часто по инфраструктуре и лицензиям.

#### Проверь себя

1. Что такое Oracle RAC?  
<details><summary>Ответ</summary> **RAC (Real Application Clusters)** — несколько **экземпляров** Oracle на разных серверах работают с **одной общей базой данных** (данные на shared storage). При отказе одного узла остальные продолжают обслуживать запросы — отказоустойчивость без единой точки отказа на уровне приложения.</details>

2. Чем SQL Server Always On похож на репликацию в PostgreSQL?  
<details><summary>Ответ</summary> **Always On** — первичная реплика принимает запись, вторичные получают поток изменений (синхронно или асинхронно); при отказе primary выполняется **failover** на вторичную реплику. Аналогично **streaming replication** в PostgreSQL + механизм автоматического failover (например, Patroni): один primary, одна или несколько replica, переключение при сбое.</details>

**Три пункта, чтобы не забыть (Oracle и SQL Server).** 1) **Oracle** — RAC, PL/SQL; дорогие лицензии; корпоративная экосистема. 2) **SQL Server** — T-SQL, Always On; интеграция с Windows/Azure/.NET. 3) Выбор часто по инфраструктуре, стеку и лицензиям.

#### Запомните

- **Oracle**: RAC (кластер экземпляров), PL/SQL; дорогие лицензии; сильные стороны — масштаб, аналитика, экосистема.
- **SQL Server**: T-SQL, Always On (группы доступности); интеграция с Windows/Azure/.NET.
- Выбор часто по инфраструктуре и лицензиям.

**Ещё раз самыми простыми словами.** Oracle — корпоративная СУБД с кластером (RAC) и языком PL/SQL; лицензии дорогие. SQL Server — СУБД Microsoft с T-SQL и Always On; удобна в мире Windows и Azure. Что выбирать — часто решают уже имеющаяся инфраструктура и бюджет.

**Если прочитал и всё равно не понял.** Запомни: **Oracle** и **SQL Server** — две другие «большие» реляционные СУБД кроме PostgreSQL и MySQL. Oracle — часто в корпорациях с Java и дорогими лицензиями; SQL Server — в мире Microsoft (Windows, .NET, Azure). Для нового проекта с открытым кодом и стандартным SQL чаще берут PostgreSQL; Oracle и SQL Server — когда уже есть контракт, команда или жёсткие требования к продукту.

### 46.2. NewSQL: CockroachDB, TiDB, YugabyteDB

**Цель раздела.**  
Понять, что такое **распределённые SQL** СУБД (NewSQL): **CockroachDB**, **TiDB**, **YugabyteDB** — глобальное распределение, совместимость с PostgreSQL или MySQL, когда что выбирать. После раздела ты будешь знать общие черты и отличия от «одноузловых» PostgreSQL/MySQL.

**В этом разделе главное (три строки).** 1) **NewSQL** — реляционная модель + SQL + распределённость: горизонтальное масштабирование, отказоустойчивость. 2) **CockroachDB** — PostgreSQL-совместимый, распределённый; **TiDB** — MySQL-совместимый, HTAP; **YugabyteDB** — PostgreSQL-совместимый API (YSQL). 3) Выбирают, когда одного узла PostgreSQL/MySQL недостаточно по масштабу или по отказоустойчивости.

#### Термины (расшифровка)

- **NewSQL** — класс СУБД, сочетающих **реляционную модель** и **SQL** с **распределённой архитектурой**: данные лежат на многих узлах, можно добавлять узлы (горизонтальное масштабирование), при отказе части узлов кластер продолжает работать; часто гарантируется **strong consistency**. Отличие от ручного шардирования MySQL — встроенное шардирование, репликация и согласованность «из коробки». Примеры: CockroachDB, TiDB, YugabyteDB.
- **CockroachDB** — **распределённая** СУБД; **протокол** и **диалект SQL** совместимы с **PostgreSQL** — многие приложения и драйверы работают без изменений. Данные автоматически **шардируются** по диапазонам ключей и **реплицируются** (консенсус **Raft**); поддерживается **глобальное распределение** (несколько регионов). Open source. Подходит, когда нужен масштаб или отказоустойчивость выше, чем у одного узла PostgreSQL.
- **TiDB** — **распределённая** СУБД; **совместимость с MySQL** по протоколу и диалекту. Архитектура: **TiDB** (сервер запросов, без состояния) + **TiKV** (распределённое key-value хранилище на RocksDB). **HTAP** — узлы **TiFlash** для аналитики (колоночное хранение), OLTP и аналитика в одном кластере. Open source. Выбор при необходимости масштабировать MySQL-совместимую нагрузку и/или совмещать OLTP и аналитику.
- **YugabyteDB** — **распределённая** СУБД; API **YSQL** совместим с **PostgreSQL** (драйверы libpq, часть SQL и кода). Данные в распределённом хранилище с репликацией; горизонтальное масштабирование. Open source. Подходит, когда нужна распределённость и хочется сохранить навыки и код под PostgreSQL.

#### Теория и правила (подробно)

- **Когда смотреть в сторону NewSQL.** Один узел PostgreSQL/MySQL не справляется по **нагрузке** или по **объёму данных**; нужна **отказоустойчивость** без ручного настраивания репликации и failover; нужна **горизонтальная масштабируемость** (добавление узлов без переделки приложения). NewSQL даёт это ценой большей сложности развёртывания и отличий в тонкостях (задержки распределённых транзакций, ограничения по некоторым конструкциям SQL).
- **Чем жертвуют.** Не все конструкции PostgreSQL/MySQL поддерживаются один в один; распределённые транзакции могут быть медленнее локальных; эксплуатация сложнее (кластер вместо одного процесса). NewSQL выбирают осознанно, когда ограничения одноузловой СУБД уже мешают.

#### Пошагово: как сориентироваться (кратко)

1. Нужна **совместимость с PostgreSQL** и распределённость? — CockroachDB или YugabyteDB.  
2. Нужна **совместимость с MySQL** и/или **HTAP** (OLTP + аналитика)? — TiDB.  
3. Достаточно ли **одного узла PostgreSQL** с репликой (streaming replication)? — часто да; NewSQL имеет смысл при реальной нехватке масштаба или требовании к глобальному распределению.

#### Что будет, если переносить приложение с PostgreSQL на CockroachDB «как есть»

Большая часть SQL и протокол совместимы — приложение часто работает. Но есть отличия: некоторые типы данных, функции, блокировки и изоляция могут вести себя иначе; распределённые транзакции могут быть медленнее. **Вывод:** тестировать под нагрузкой; изучить документацию по совместимости и ограничениям.

#### Типичная ошибка

Выбирать NewSQL «на будущее» без реальной потребности в масштабе или отказоустойчивости — усложняется эксплуатация без выигрыша. **Правильно:** начинать с PostgreSQL/MySQL; переходить на NewSQL, когда появятся конкретные ограничения (нагрузка, объём, требования к HA/распределению).

#### Часто путают

**NewSQL** и **ручное шардирование** (например, несколько инстансов MySQL с разными данными по ключу). При ручном шардировании приложение само решает, в какую БД писать; репликация и согласованность — зона ответственности приложения. NewSQL (CockroachDB, TiDB, YugabyteDB) даёт **встроенное** шардирование и репликацию; приложение видит одну логическую БД. **CockroachDB**, **TiDB** и **YugabyteDB** между собой — разная совместимость (PostgreSQL vs MySQL), разная архитектура хранения и HTAP; не взаимозаменяемы «один в один».

#### Простыми словами

NewSQL — это «реляционные БД, но распределённые»: SQL, транзакции, плюс горизонтальное масштабирование и отказоустойчивость. CockroachDB — как PostgreSQL, но «распределённый»; TiDB — как MySQL, но распределённый и с аналитикой; YugabyteDB — PostgreSQL-совместимый API поверх распределённого хранилища. Выбирают, когда одного узла недостаточно.

#### Картинка в голове

**NewSQL** — **одна большая таблица, но разложенная по многим серверам**: добавляешь сервер — таблица «растёт» на него; один сервер упал — остальные продолжают. **CockroachDB** — «как PostgreSQL», но данные автоматически шардируются и реплицируются по узлам. **TiDB** — «как MySQL» плюс аналитика в одном кластере. **YugabyteDB** — тот же интерфейс, что PostgreSQL (YSQL), но под капотом распределённое хранилище. Один узел PostgreSQL/MySQL — один сервер; NewSQL — «полка» растянута на несколько серверов.

#### Как запомнить

**NewSQL** — реляционные СУБД с распределённостью и горизонтальным масштабированием. **CockroachDB** — PostgreSQL-совместимый; **TiDB** — MySQL-совместимый + HTAP; **YugabyteDB** — PostgreSQL-совместимый API (YSQL). Выбор — когда одного узла мало.

#### Проверь себя

1. В чём главное отличие NewSQL от «обычного» PostgreSQL на одном сервере?  
<details><summary>Ответ</summary> **NewSQL** — данные **распределены** по нескольким узлам: встроенное **шардирование**, **репликация**, **горизонтальное масштабирование** (добавление узлов) и отказоустойчивость. Один узел PostgreSQL — одна инстанция; масштабирование в основном вертикальное (больше RAM/CPU) и репликация настраивается отдельно (streaming replication).</details>

2. Когда логично рассматривать TiDB вместо MySQL?  
<details><summary>Ответ</summary> Когда нужна **распределённость** и **горизонтальное масштабирование** при сохранении совместимости с MySQL (протокол, диалект), или когда нужен **HTAP** — OLTP и аналитика в одном кластере (TiFlash для аналитических запросов). Один сервер MySQL достаточен — TiDB не обязателен.</details>

**Три пункта, чтобы не забыть (NewSQL).** 1) **NewSQL** — реляционные СУБД с распределённостью и горизонтальным масштабированием. 2) **CockroachDB** — PostgreSQL-совместимый; **TiDB** — MySQL-совместимый + HTAP; **YugabyteDB** — PostgreSQL-совместимый API (YSQL). 3) Выбирают, когда одного узла PostgreSQL/MySQL недостаточно.

#### Запомните

- **NewSQL** — реляционные СУБД с распределённостью и горизонтальным масштабированием.
- **CockroachDB** — PostgreSQL-совместимый, распределённый; Raft, шардирование.
- **TiDB** — MySQL-совместимый; TiKV + TiDB; HTAP (TiFlash).
- **YugabyteDB** — YSQL (PostgreSQL) и YCQL (Cassandra); распределённое хранение.

**Ещё раз самыми простыми словами.** NewSQL — это реляционные БД, которые «размазаны» по многим узлам: добавляешь узлы — растёт мощность и отказоустойчивость. CockroachDB — для тех, кто с PostgreSQL; TiDB — для тех, кто с MySQL; YugabyteDB — тоже PostgreSQL-мир (YSQL). Нужны такие системы, когда одного сервера уже мало.

**Если прочитал и всё равно не понял.** Запомни: **NewSQL** — это когда реляционная БД (с SQL и таблицами) умеет **расти по узлам** и не падать при отказе части серверов. **CockroachDB** — «распределённый PostgreSQL»; **TiDB** — «распределённый MySQL» плюс аналитика; **YugabyteDB** — «распределённый» с PostgreSQL-совместимым интерфейсом (YSQL). Рассматривай их, когда один PostgreSQL или один MySQL уже не тянет по нагрузке или по надёжности.
