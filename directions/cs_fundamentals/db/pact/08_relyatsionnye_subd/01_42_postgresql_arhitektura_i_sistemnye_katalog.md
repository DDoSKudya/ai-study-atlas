## §42. PostgreSQL: архитектура и системные каталоги

### 42.1. Процессы PostgreSQL

**Цель раздела.**  
Понять, из каких процессов состоит работающий экземпляр PostgreSQL и за что отвечает каждый: **postmaster** (приём подключений и запуск backend), **backend-процессы** (один на соединение), **background workers** (автовакуум, запись WAL и др.). После раздела ты будешь знать, как «устроен» сервер PostgreSQL на уровне ОС и почему количество соединений так важно.

**В этом разделе главное (три строки).** 1) **postmaster** — единственный процесс, который слушает порт и принимает подключения; для каждого нового соединения он создаёт **backend-процесс** (postgres), который обрабатывает все запросы этого клиента. 2) **Background workers** — фоновые процессы без привязки к клиенту: autovacuum launcher, WAL writer, checkpointer, stats collector и др.; они выполняют служебные задачи (очистка, запись WAL, сбор статистики). 3) Модель **«один процесс на соединение»** — каждое TCP-соединение обслуживается своим процессом ОС; поэтому большое число соединений = много процессов = много памяти и контекстных переключений — отсюда важность пулов соединений (PgBouncer).

#### Термины (расшифровка)

- **postmaster** — главный (родительский) процесс всего экземпляра PostgreSQL. Он **один** на весь кластер. Запускается первым при старте кластера (команда `pg_ctl start` или напрямую `postgres -D /path/to/data`). Его задачи: слушать сетевой порт (по умолчанию 5432), принимать входящие TCP-подключения от клиентов и для **каждого** нового подключения создавать дочерний процесс — backend. Если postmaster завершить (убить процесс, остановить сервис), **все** дочерние backend-процессы тоже завершатся — кластер перестанет принимать подключения и обрабатывать запросы. То есть postmaster — «корень» дерева процессов; без него остальные процессы не работают.
- **backend-процесс (postgres)** — процесс, который обслуживает **ровно одно** клиентское соединение. В системе его обычно видно как процесс с именем `postgres` (то же имя, что и у сервера). Когда клиент (приложение, psql, любая программа) подключается к базе, postmaster создаёт **новый** такой процесс и «передаёт» ему это соединение. Всё дальнейшее общение клиента с БД идёт через этот процесс: он принимает SQL-запросы, парсит их, строит план, выполняет, возвращает результат. Работает с общей памятью (shared_buffers и др.). Когда клиент отключается, этот backend-процесс **завершается** — его память освобождается операционной системой. Если между приложением и PostgreSQL стоит PgBouncer (пул соединений), то с точки зрения PostgreSQL «клиентов» меньше — PgBouncer держит ограниченное число соединений к серверу и раздаёт их многим своим клиентам.
- **background worker** — фоновый процесс, **не привязанный** ни к какому клиенту. Он не обрабатывает запросы пользователей. Его задача — служебная работа «для всей базы». Примеры: **autovacuum launcher** — периодически запускает воркеры autovacuum, которые очищают мёртвые строки и обновляют статистику; **WAL writer** — сбрасывает буферы WAL на диск; **checkpointer** — периодически делает checkpoint (сбрасывает изменённые страницы данных на диск); **stats collector** — собирает статистику для представлений pg_stat_*; **logical replication launcher** — запускает воркеры логической репликации. Количество и поведение части из них настраиваются (например, autovacuum_max_workers).

#### Теория и правила (подробно)

- **Почему именно процессы, а не потоки.** PostgreSQL использует **многопроцессную** архитектуру: на каждое соединение — отдельный **процесс ОС**, а не поток внутри одного процесса. Плюсы: если один процесс «упадёт» (краш, ошибка), остальные продолжают работать; отладка проще (можно посмотреть процесс в отладчике, привязаться к PID). Минусы: каждый процесс занимает свою память (стек, служебные структуры) — обычно несколько мегабайт на процесс; при 1000 соединений это уже гигабайты только на процессы. Кроме того, ОС должна переключать контекст между множеством процессов — при очень большом числе соединений это создаёт нагрузку. Поэтому в реальных системах ограничивают **max_connections** (например, 100–200) и ставят перед PostgreSQL **пул соединений** (PgBouncer или пул в приложении), чтобы тысячи «логических» клиентов использовали меньше реальных соединений к серверу.
- **Порядок запуска кластера.** При старте (например, `pg_ctl start`) запускается **только** postmaster. Он читает конфигурацию, создаёт **shared memory** (общую память для всех процессов), открывает порт и запускает фоновые воркеры (checkpointer, background writer, WALwriter, autovacuum launcher и т.д.). Backend-процессов в этот момент **ещё нет** — они создаются **по мере** того, как клиенты подключаются. То есть: один postmaster + несколько background workers всегда есть; количество процессов postgres (backend) = количество текущих подключений.
- **Как посмотреть, что происходит.** В операционной системе: команда `ps aux | grep postgres` (или `ps -ef | grep postgres`). Увидишь один процесс postmaster и по одному процессу с именем postgres на каждое активное соединение. Внутри базы: запрос `SELECT pid, usename, application_name, state FROM pg_stat_activity;` — каждая строка это одно соединение; поле `pid` — это PID процесса ОС (тот же, что в `ps`). Так можно сопоставить «кто из приложения» с «каким процессом ОС».

#### Пошагово: что происходит при подключении клиента (с деталями)

Представь: твоё приложение открывает соединение к PostgreSQL (например, по адресу localhost:5432).

1. **Клиент** устанавливает TCP-соединение с сервером: пакет приходит на порт 5432 (или другой, если настроен в `listen_addresses`).
2. **postmaster** принимает это соединение (он единственный, кто слушает порт). Он не обрабатывает запросы сам — он создаёт **новый процесс**. На Unix-системах это обычно делается через **fork()**: создаётся копия процесса postmaster, и в дочернем процессе выполняется код, который становится backend'ом. В итоге появляется новый процесс с именем `postgres`.
3. **Новый backend** «получает» это TCP-соединение (передаётся от postmaster). Он выполняет **аутентификацию**: читает конфиг pg_hba.conf, проверяет, с какого адреса пришло подключение, запрашивает пароль (или проверяет сертификат, в зависимости от настроек). Если аутентификация не прошла — соединение закрывается, backend завершается.
4. После успешной аутентификации клиент может слать запросы. **Каждый** запрос обрабатывается этим же backend'ом: парсинг SQL, построение плана, выполнение, отправка результата обратно клиенту. Всё это время backend «занят» этим соединением.
5. Когда клиент закрывает соединение (закрыл программу, таймаут, разрыв сети), backend-процесс **завершает работу**. Память, которую он занимал, освобождается ОС. Соединения больше нет — процесса тоже нет.

**Итог:** одно подключение = один процесс на всё время подключения. Новый клиент = новый процесс. Отключился клиент = процесс завершился.

#### Что будет, если открыть 10 000 соединений к PostgreSQL без пула

Если приложение (или много экземпляров приложения) откроет 10 000 одновременных соединений к одному серверу PostgreSQL, и при этом max_connections разрешает столько (или больше), то PostgreSQL создаст **10 000 backend-процессов**. Каждый процесс — это минимум несколько мегабайт RAM (стек, буферы для запросов и т.д.). 10 000 × 5 MB ≈ 50 GB только на процессы — на многих серверах такой памяти просто нет, начнутся своп и OOM (Out Of Memory). Кроме того, планировщик ОС будет переключаться между 10 000 процессами — контекстные переключения съедят процессорное время. Запросы начнут выполняться медленно, возможны таймауты и обрывы. **Вывод:** так делать не нужно. Нужно ограничить число соединений к PostgreSQL (max_connections разумное, например 200) и использовать **пул соединений** (PgBouncer или в приложении): 10 000 «логических» клиентов работают через, скажем, 100 соединений к БД — и процессов будет 100, а не 10 000.

#### Типичная ошибка

Думают, что «чем больше соединений разрешить, тем лучше — все клиенты смогут подключиться». В результате поднимают max_connections до 1000–2000 без пула. При пиковой нагрузке все соединения открываются, создаётся 1000–2000 процессов, память заканчивается, база «падает» или начинает убивать процессы (OOM killer). **Правильно:** оставить max_connections в разумных пределах (например, 100–300), считать, сколько RAM можно выделить на процессы (например, 100 соединений × 10 MB ≈ 1 GB), и перед PostgreSQL поставить PgBouncer (или другой пул), чтобы сотни/тысячи клиентов приложения использовали это ограниченное число соединений.

#### Часто путают

**postmaster** и **backend (postgres)**. Оба могут называться в списке процессов «postgres». postmaster — **один**, он родительский; backend'ов — **много**, по одному на соединение. В `ps` обычно видно: один процесс с пометкой, что он слушает порт (или по родительскому PID можно понять: у postmaster дочерние — все postgres). В pg_stat_activity каждая строка — это backend (одно соединение); postmaster там **не** отображается, он не обрабатывает запросы.

#### Простыми словами

postmaster — это «диспетчер на входе»: сидит на порту и ни с кем сам не разговаривает. Как только приходит новый клиент — диспетчер «нанимает» одного работника (backend) и поручает ему этого клиента. Этот работник ведёт весь диалог с клиентом: принимает запросы, выполняет, отдаёт ответы. Когда клиент ушёл — работник увольняется (процесс завершается). Параллельно в здании работают «служебные» сотрудники (background workers): они не общаются с клиентами, а делают общую работу — пишут логи (WAL), убирают мусор (vacuum), считают статистику. Главное: **один клиент = один работник (процесс)**. Поэтому если клиентов очень много — работников будет очень много, и помещение (память сервера) может не вместить. Поэтому перед входом ставят «пул» (PgBouncer): много клиентов общаются с пулом, а пул держит ограниченное число реальных работников (соединений к PostgreSQL).

#### Картинка в голове

Ресторан. **postmaster** — администратор у входа: один. Он не обслуживает столики. Как только пришёл гость — администратор вызывает официанта (backend) и прикрепляет его к этому гостю. Один официант — один стол (одно соединение). Пока гость сидит — официант только с ним и работает. Гость ушёл — официант свободен (процесс завершился). Плюс на кухне и в подсобках работают свои люди (background workers): повара, уборщики, бухгалтерия — они не обслуживают гостей, но без них ресторан не работает. Если придёт 10 000 гостей и на каждого выделить официанта — 10 000 официантов; помещение и бюджет такого не выдержат. Поэтому делают «пул официантов»: гостей много, а официантов, например, 50 — они по очереди обслуживают столики (PgBouncer).

#### Как запомнить

**Один postmaster** — слушает порт, создаёт backend на каждое подключение. **Один backend на одно соединение** — обрабатывает все запросы этого клиента, потом завершается. **Background workers** — без клиентов, общая служба (vacuum, WAL, checkpoint). **Много соединений без пула = много процессов = много памяти и тормоза** — нужен пул (PgBouncer).

#### Проверь себя

1. За что отвечает postmaster и почему его остановка останавливает весь кластер?  
<details><summary>Ответ</summary> postmaster — главный процесс: он слушает порт, принимает подключения и создаёт backend-процессы для каждого клиента. Он не обрабатывает запросы сам. Если postmaster завершить, все дочерние backend-процессы тоже завершаются (они его дети). Кластер перестаёт принимать новые подключения и обрабатывать запросы — по сути, база «выключена».</details>

2. Почему большое число соединений к PostgreSQL может быть проблемой?  
<details><summary>Ответ</summary> Каждое соединение обслуживается **отдельным процессом ОС**. Каждый процесс потребляет память (стек, буферы — обычно несколько мегабайт). При тысячах соединений суммарное потребление RAM становится огромным, возможны OOM и своп. Плюс ОС переключается между тысячами процессов (контекстные переключения) — тратится CPU. Поэтому ограничивают max_connections и используют пул соединений (PgBouncer и т.п.), чтобы «логических» клиентов было много, а реальных соединений к PostgreSQL — мало.</details>

3. Чем backend отличается от background worker?  
<details><summary>Ответ</summary> **Backend** привязан к **одному клиентскому соединению**: обрабатывает запросы только этого клиента; создаётся при подключении, завершается при отключении. **Background worker** не привязан ни к какому клиенту: выполняет общие задачи (autovacuum, запись WAL, checkpoint, сбор статистики и т.д.); запускается при старте кластера и работает постоянно (или по расписанию).</details>

4. Как по числу процессов postgres в ОС понять, сколько сейчас активных соединений к базе?  
<details><summary>Ответ</summary> В PostgreSQL модель «один процесс на соединение»: каждое активное соединение = один backend-процесс с именем postgres. Поэтому **число процессов postgres** (не считая postmaster) = **число активных соединений**. Команда `ps aux | grep postgres | grep -v grep | wc -l` даст общее число процессов (включая postmaster); минус 1 = число соединений. Точнее — смотреть в базе: `SELECT count(*) FROM pg_stat_activity;`.</details>

**Три пункта, чтобы не забыть (процессы).** 1) **postmaster** — один, слушает порт, на каждое подключение создаёт **backend**. 2) **backend** — один на одно соединение, обрабатывает запросы этого клиента, при отключении клиента завершается. 3) **Много соединений = много процессов = много памяти** — нужен пул соединений (PgBouncer) и разумный max_connections.

#### Запомните

- **postmaster** — один процесс, слушает порт, создаёт **backend** на каждое соединение; при его остановке останавливается весь кластер.
- **backend** — один процесс на одно клиентское соединение; обрабатывает все запросы этого клиента; при отключении клиента процесс завершается.
- **background workers** — фоновые процессы (autovacuum, WAL writer, checkpointer и др.), не привязаны к клиентам.
- Модель «один процесс на соединение» → важно ограничивать число соединений (max_connections) и использовать пулы (PgBouncer).

**Ещё раз самыми простыми словами:** postmaster — «приёмная», один. Каждый подключившийся клиент получает своего «работника» (backend). Работников столько, сколько соединений. Плюс есть «службы» (background workers) без привязки к клиентам. Много клиентов без пула = много работников = не хватает памяти и всё тормозит — поэтому перед базой ставят пул (PgBouncer).

**Если прочитал и всё равно не понял.** Запомни одну цепочку: **подключился клиент → postmaster создал для него один процесс (backend) → этот процесс обрабатывает все запросы этого клиента → клиент отключился → процесс завершился.** Всё. Один клиент = один процесс. Поэтому 1000 клиентов без пула = 1000 процессов = очень много памяти. Чтобы не создавать столько процессов, между приложением и PostgreSQL ставят PgBouncer: приложение открывает много соединений к PgBouncer, а PgBouncer держит, например, 50 соединений к PostgreSQL — и процессов postgres будет 50, а не 1000.

### 42.2. Shared memory

**Цель раздела.**  
Понять, что такое **shared memory** (общая память) в PostgreSQL и какие структуры в ней хранятся: **shared_buffers** (кэш страниц данных), буферы WAL, таблица блокировок и т.д. После раздела ты будешь знать, зачем настраивать shared_buffers и почему эта память общая для всех процессов.

**В этом разделе главное (три строки).** 1) **Shared memory** — область RAM, доступная **всем** backend-процессам и фоновым воркерам; создаётся при старте postmaster. 2) **shared_buffers** — основная часть: кэш страниц данных и индексов; чем больше (в разумных пределах), тем больше данных держится в RAM и тем меньше обращений к диску. 3) Кроме shared_buffers в shared memory лежат: WAL buffers, структуры блокировок (lock table), информация о транзакциях и процессах (ProcArray, CLOG) и др. — всё, к чему нужен одновременный доступ из разных процессов.

#### Термины (расшифровка)

- **Shared memory (разделяемая память)** — сегмент оперативной памяти (RAM), который создаётся **postmaster** при старте кластера. К этой области имеют доступ **все** дочерние процессы: и backend'ы, и background workers. То есть это «общая» память: несколько процессов читают и пишут в неё. В ней хранится то, что должно быть **одним и тем же** для всех: кэш страниц таблиц и индексов, метаданные блокировок (кто какую строку/таблицу заблокировал), состояние транзакций (какая транзакция закоммичена, какая активна) и т.д. Без shared memory каждый процесс держал бы свою копию — тогда при изменении страницы одним процессом другие не видели бы изменения без чтения с диска, и согласованность данных нарушилась бы.
- **shared_buffers** — параметр конфигурации PostgreSQL (в postgresql.conf); задаёт **размер буферного кэша** в байтах (или в блоках по 8 KB). В этом кэше хранятся **страницы** (блоки по 8 KB) таблиц и индексов — те самые страницы, которые лежат на диске в файлах данных. Когда процесс читает строку из таблицы, он сначала смотрит, есть ли нужная страница в shared_buffers; если есть — читает из памяти (быстро); если нет — читает страницу с диска и кладёт копию в shared_buffers. Когда процесс изменяет строку, он изменяет страницу в shared_buffers; на диск страница сбрасывается позже (при checkpoint или при вытеснении из кэша). Чем больше shared_buffers, тем больше страниц «влезает» в RAM — тем реже приходится ходить на диск. Рекомендуемые значения для сервера, выделенного под БД: порядка **25–40%** от всей RAM (например, при 16 GB RAM — 4–6 GB; при 32 GB — 8–12 GB). Слишком маленький shared_buffers — много обращений к диску, медленнее. Слишком большой — не остаётся памяти для work_mem, кэша ОС, соединений и т.д.; возможны OOM и своп.
- **WAL buffers** — буфер в shared memory, в который сначала пишутся записи WAL (Write-Ahead Log) перед сбросом на диск. Процесс WAL writer периодически сбрасывает этот буфер в файлы WAL. Размер обычно невелик (несколько мегабайт по умолчанию); его редко нужно сильно увеличивать.

#### Теория и правила (подробно)

- **Зачем общая память.** Несколько backend-процессов одновременно работают с **одними и теми же** таблицами и индексами. Например: один процесс обновил строку в таблице orders — изменилась страница в памяти; другой процесс читает ту же таблицу — он должен увидеть либо старую версию (если его транзакция началась раньше), либо новую (если позже). Для этого все должны видеть **одну и ту же** копию страницы — та, что в shared_buffers. Если бы у каждого процесса был свой кэш, при изменении страницы процессом A процесс B не знал бы об этом, пока не прочитал с диска — и логика MVCC (многоверсионность) сломалась бы. Поэтому кэш страниц **общий** — shared_buffers в shared memory. То же с блокировками: если процесс A заблокировал строку, процесс B должен это «увидеть» (lock table в shared memory). И с состоянием транзакций: все процессы смотрят в ProcArray и CLOG, чтобы понять, видна ли строка данной транзакции. Всё это возможно только при общей памяти.
- **shared_buffers: как выбирать размер.** Увеличение shared_buffers уменьшает долю чтений с диска: часто используемые страницы остаются в RAM. Но память ограничена. Память нужна ещё для: каждого соединения (процесса) — стек, буферы; work_mem на каждую операцию (сортировка, хеш); maintenance_work_mem для VACUUM, CREATE INDEX; кэша ОС (файловая система кэширует дисковые чтения). Если отдать под shared_buffers 80% RAM, для всего остального может не хватить — начнутся своп и OOM. Практика: на сервере, который **только** под PostgreSQL, ставят 25–40% RAM в shared_buffers; на сервере, где ещё что-то крутится — меньше. После изменения shared_buffers нужен **перезапуск** кластера (не reload).
- **Другие структуры в shared memory.** Помимо shared_buffers и WAL buffers там лежат: **lock table** — кто какие блокировки держит (на строки, страницы, таблицы); **ProcArray** — список активных транзакций (для MVCC: какая транзакция видит какие строки); **CLOG** (commit log) — статус коммита транзакций; буферы для других служебных целей. Их размер обычно не задаётся вручную — он зависит от max_connections, количества объектов и т.д.

#### Пошагово: что происходит при чтении страницы (зачем нужен shared_buffers)

Запрос: `SELECT * FROM orders WHERE id = 100`. Таблица orders; строка с id=100 лежит на странице № 50 (условно).

1. Backend-процесс обрабатывает запрос. Ему нужно прочитать страницу 50 таблицы orders.
2. Он смотрит в **shared_buffers**: есть ли там страница (orders, блок 50)? Если **да** — читает данные из этой страницы в памяти, находит строку, возвращает клиенту. **Диск не трогается** — быстро.
3. Если **нет** — процесс читает страницу 50 **с диска** (файл данных таблицы orders), кладёт копию страницы в **shared_buffers** (вытесняя какую-то старую страницу, если кэш полон), затем читает из памяти. Диск использован один раз; при следующем запросе к этой же странице другой процесс (или тот же) найдёт её уже в shared_buffers.
4. Если shared_buffers маленький — страницы часто вытесняются; при повторном обращении к той же странице снова придётся читать с диска. Если shared_buffers большой — «горячие» страницы дольше остаются в RAM — меньше обращений к диску, быстрее запросы.

**Итог:** shared_buffers — это «общая полка» с копиями страниц БД. Все процессы смотрят на эту полку; если страница там есть — читают из памяти; если нет — тащат с диска и кладут на полку. Чем полка больше — тем больше страниц помещается, тем реже ходят на диск.

#### Что будет, если поставить shared_buffers очень большим (например, 90% RAM)

Если на сервере 16 GB RAM и поставить shared_buffers = 14 GB, то для всего остального останется 2 GB. В эти 2 GB должны поместиться: все backend-процессы (каждый — несколько мегабайт; при 100 соединениях — сотни мегабайт), work_mem для каждой активной операции (сортировки, хеши), maintenance_work_mem при VACUUM/CREATE INDEX, кэш ОС для файлов и т.д. Скорее всего, памяти не хватит — ОС начнёт использовать **своп** (диск как расширение RAM) — всё сильно замедлится; или сработает **OOM killer** и убьёт процесс (часто postmaster или какой-то backend). Плюс на Linux для shared_buffers больше определённого размера могут потребоваться настройки ядра (shmmax, shmall). **Вывод:** не ставить shared_buffers «на всё»; оставлять запас для соединений, work_mem и ОС. Ориентир — 25–40% RAM на выделенном сервере под БД.

#### Типичная ошибка

Оставляют shared_buffers по умолчанию (128 MB или 256 MB) на сервере с 32 GB RAM. В результате большая часть данных читается с диска — запросы медленные. **Правильно:** на сервере, выделенном под PostgreSQL, поднять shared_buffers до разумной доли RAM (например, 8 GB при 32 GB RAM) и перезапустить кластер.

#### Часто путают

**shared_buffers** и **кэш ОС (файловый кэш)**. shared_buffers — это кэш **внутри** PostgreSQL: сервер сам решает, какие страницы держать, по своей логике вытеснения. ОС тоже кэширует прочитанные с диска файлы в своей памяти — при повторном чтении того же файла ОС может отдать данные из своего кэша. Но PostgreSQL **не полагается** на кэш ОС для данных — он дублирует страницы в shared_buffers. То есть данные лежат и в shared_buffers, и (после чтения) могут лежать в кэше ОС. effective_cache_size в PostgreSQL — это подсказка планировщику: «считай, что столько памяти доступно для кэша» (суммарно shared_buffers + кэш ОС), чтобы планировщик лучше выбирал между index scan и seq scan.

#### Простыми словами

Shared memory — это «общая комната» для всех процессов PostgreSQL: в ней лежит одна копия кэша страниц (shared_buffers), одна «доска» с блокировками (lock table) и одна «картотека» транзакций (ProcArray, CLOG). Все работники (backend'ы) заходят в эту комнату и смотрят одни и те же данные — так все видят актуальные страницы и знают, кто что заблокировал. Настраивать вручную обычно нужно в основном **shared_buffers** — размер «полки» с копиями страниц. Чем полка больше — тем больше данных в RAM, тем меньше обращений к диску; но нельзя занять всю память сервера — нужно оставить место для соединений и других нужд.

#### Картинка в голове

Один общий шкаф (shared memory). В нём полки с копиями страниц БД (shared_buffers) — все берут данные отсюда, а не тащат с диска каждый раз. На дверце шкафа — журнал «кто что держит» (lock table) и «кто какую транзакцию сделал» (CLOG, ProcArray). Все работники (backend) подходят к одному шкафу — не нужно каждому иметь свой шкаф, иначе данные разъехались бы.

#### Как запомнить

**Shared memory** = общая память для всех процессов; создаётся при старте postmaster. **shared_buffers** = кэш страниц таблиц и индексов в этой памяти; основной «ручной» параметр; 25–40% RAM на выделенном сервере. Кэш общий, чтобы все видели одни и те же страницы и согласованность данных не ломалась.

#### Проверь себя

1. Что такое shared_buffers и как его размер влияет на производительность?  
<details><summary>Ответ</summary> **shared_buffers** — кэш страниц данных и индексов в общей памяти (shared memory). Увеличение размера позволяет держать в RAM больше страниц — при повторных чтениях обращений к диску меньше, запросы быстрее. Слишком маленький — много чтений с диска. Слишком большой — не остаётся памяти для соединений, work_mem, ОС; возможны своп и OOM. Типично 25–40% RAM на выделенном сервере под БД.</details>

2. Почему кэш страниц находится именно в shared memory, а не в каждом процессе?  
<details><summary>Ответ</summary> Несколько backend-процессов работают с **одними и теми же** страницами (одна таблица, один индекс). Если бы кэш был у каждого процесса, при изменении страницы одним процессом другие не видели бы актуальные данные без повторного чтения с диска — и логика MVCC (кто какую версию строки видит) сломалась бы. Общий кэш в shared memory даёт **единое** представление данных для всех процессов: все смотрят в одну и ту же копию страницы.</details>

3. Можно ли изменить shared_buffers без перезапуска PostgreSQL?  
<details><summary>Ответ</summary> **Нет.** Размер shared_buffers задаётся при создании shared memory при старте postmaster. Чтобы изменить значение, нужно отредактировать postgresql.conf (или ALTER SYSTEM) и **перезапустить** кластер (pg_ctl restart). Команда pg_reload_conf() (или SIGHUP) перезагружает только часть параметров и **не** меняет shared_buffers.</details>

**Три пункта, чтобы не забыть (shared memory).** 1) **Shared memory** — общая RAM для всех процессов; в ней shared_buffers, lock table, ProcArray, CLOG и др. 2) **shared_buffers** — кэш страниц; чем больше (в разумных пределах), тем меньше обращений к диску. 3) Ориентир по размеру — 25–40% RAM на выделенном сервере; не забирать всю память — оставить на соединения и work_mem.

#### Запомните

- **Shared memory** — общая область RAM для всех процессов кластера; создаётся при старте postmaster.
- **shared_buffers** — кэш страниц таблиц и индексов; основной настраиваемый параметр; ориентир — 25–40% RAM при выделенном сервере; изменение требует перезапуска.
- В shared memory также: WAL buffers, lock table, ProcArray, CLOG и др. — всё, к чему нужен совместный доступ из разных процессов.

**Ещё раз самыми простыми словами:** Shared memory — «общая комната» с одной копией кэша страниц и одной «доской» блокировок и транзакций. shared_buffers — размер этой «полки» с данными. Больше полка — больше данных в памяти, быстрее; но не занимать всю память сервера — нужен запас.

**Если прочитал и всё равно не понял.** Главное: у PostgreSQL есть **общая память** (shared memory), в которой лежит **кэш страниц** (shared_buffers). Все процессы читают и пишут страницы **из этой общей области** — так все видят одни и те же данные. Размер кэша настраивается параметром shared_buffers. Чем он больше — тем больше данных помещается в RAM и тем реже идём на диск. Ставь примерно четверть–треть RAM сервера (если сервер только под БД), не больше — остальное нужно для соединений и других нужд.

### 42.3. Расширения

**Цель раздела.**  
Научиться устанавливать и использовать **расширения** (extensions) PostgreSQL: что такое расширение, как его ставить (**CREATE EXTENSION**), какие расширения часто нужны (pgvector, PostGIS, pg_cron, pg_stat_statements и др.). После раздела ты будешь знать, как добавить в кластер новые типы данных, функции и операторы без пересборки сервера.

**В этом разделе главное (три строки).** 1) **Расширение** — упакованный набор объектов БД (типы, функции, операторы, индексы и т.д.), который устанавливается **одной** командой **CREATE EXTENSION** в нужную базу. 2) Расширения ставятся **в конкретную базу**; для использования в другой базе нужно выполнить CREATE EXTENSION там отдельно. 3) Популярные: **pgvector** (векторный поиск), **PostGIS** (геоданные), **pg_cron** (планировщик заданий), **pg_stat_statements** (статистика по запросам), **pg_trgm** (триграммы, нечёткий поиск), **uuid-ossp** (генерация UUID).

#### Термины (расшифровка)

- **Extension (расширение)** — именованный модуль, который добавляет в PostgreSQL новые объекты: типы данных, функции, операторы, индексы, таблицы и т.д. Файлы расширения лежат на сервере в каталоге `.../share/postgresql/extension/`: контрольный файл с именем `имя.control` (метаданные, зависимости) и SQL-файлы (создание объектов). Когда ты выполняешь **CREATE EXTENSION имя**, PostgreSQL читает эти файлы и выполняет SQL — в текущей базе появляются новые типы, функции и т.д. Список **доступных** расширений (файлы есть на сервере) — в представлении **pg_available_extensions**. Список **уже установленных** в текущей базе — в **pg_extension**. То есть «доступно» = можно поставить; «установлено» = уже стоит в этой базе.
- **CREATE EXTENSION** — команда SQL для установки расширения **в текущую базу данных**. Синтаксис: `CREATE EXTENSION [ IF NOT EXISTS ] имя_расширения;`. Пример: `CREATE EXTENSION IF NOT EXISTS "uuid-ossp";` — в текущей базе появятся функции `uuid_generate_v4()`, `uuid_generate_v1()` и др. Обычно нужны права суперпользователя или владельца базы (зависит от расширения). После установки расширение «привязано» к этой базе — в другой базе его нет, пока не выполнишь CREATE EXTENSION там же.
- **pgvector** — расширение для хранения **векторов** (массивов чисел) и поиска **ближайших соседей** (по косинусному расстоянию, L2 и т.д.). Добавляет тип `vector(n)`, операторы и индексы (ivfflat, hnsw). Используется для эмбеддингов (нейросети, RAG). Установка: пакет на сервере (например, `apt install postgresql-16-pgvector`) + в базе `CREATE EXTENSION vector;`.
- **PostGIS** — расширение для **геопространственных данных**: точки, линии, полигоны на карте. Добавляет типы (geometry, geography), сотни функций (ST_Distance, ST_Within, ST_Intersects и др.) и пространственные индексы (GiST). Установка: пакет postgresql-16-postgis-x.x + в базе `CREATE EXTENSION postgis;`.
- **pg_stat_statements** — расширение для **сбора статистики по запросам**: сколько раз выполнялся каждый запрос, суммарное время, среднее время, число строк и т.д. Нужно для поиска «тяжёлых» запросов. **Особенность:** это расширение должно быть загружено при **старте** кластера — в postgresql.conf добавляют `shared_preload_libraries = 'pg_stat_statements'` и **перезапускают** PostgreSQL. После перезапуска в нужной базе: `CREATE EXTENSION pg_stat_statements;`. Без shared_preload_libraries расширение не заработает (или будет ограничено).

#### Теория и правила (подробно)

- **Установка по шагам.** Сначала расширение должно быть **установлено на сервере** — пакетом (apt, yum, brew) или из исходников. Файлы попадают в каталог расширений PostgreSQL. После этого в **каждой базе**, где нужно использовать расширение, выполняется **CREATE EXTENSION имя**. Одна команда — и в этой базе появляются все объекты расширения. В другой базе тех же объектов нет — нужно отдельно CREATE EXTENSION в той базе.
- **Расширения, требующие перезапуска.** Некоторые расширения используют **shared library** (библиотеку), которая должна быть загружена при старте postmaster и разделять память со всеми процессами. Такие расширения перечисляются в **shared_preload_libraries** в postgresql.conf. После изменения shared_preload_libraries **обязателен перезапуск** кластера (pg_ctl restart). Примеры: pg_stat_statements, auto_explain (если как shared library). Без перезапуска CREATE EXTENSION может выполниться, но функциональность будет неполной или неработающей.
- **Версии и обновление.** У расширения может быть несколько версий. Текущая установленная версия видна в pg_extension. Обновление: `ALTER EXTENSION имя UPDATE;` — PostgreSQL выполнит скрипты обновления (если автор расширения их предусмотрел). Перед обновлением смотри документацию расширения.
- **Удаление.** `DROP EXTENSION имя;` — удаляет объекты, созданные этим расширением. Не сработает, если есть зависимости (например, таблица с типом из расширения). Сначала удаляют зависимые объекты или используют DROP EXTENSION ... CASCADE (осторожно — удалит и зависимые объекты).

#### Пошагово: установка расширения (пример — uuid-ossp, с нуля)

Ты хочешь генерировать UUID в таблице. В «голом» PostgreSQL нет функции uuid_generate_v4() — она поставляется расширением uuid-ossp.

1. **Проверь, доступно ли расширение.** Подключись к базе и выполни:  
   `SELECT name, default_version FROM pg_available_extensions WHERE name = 'uuid-ossp';`  
   Если вернулась строка — расширение установлено на сервере (файлы есть). Если пусто — нужно установить пакет (например, в Debian/Ubuntu пакет обычно уже входит в postgresql-contrib или основной пакет PostgreSQL).
2. **Установи расширение в базу.** В той базе, где нужны UUID:  
   `CREATE EXTENSION IF NOT EXISTS "uuid-ossp";`  
   Кавычки нужны из-за дефиса в имени. Сообщение вроде «CREATE EXTENSION» — успех.
3. **Проверь.** Выполни: `SELECT uuid_generate_v4();` — должна вернуться одна строка с UUID (например, `a0eebc99-9c0b-4ef8-bb6d-6bb9bd380a11`). Теперь можно использовать эту функцию в DEFAULT для столбца: `id uuid DEFAULT uuid_generate_v4() PRIMARY KEY`.

**Пошагово: pg_stat_statements (статистика по запросам)**

Ты хочешь видеть, какие запросы выполняются чаще всего и сколько времени съедают.

1. **Добавь в postgresql.conf:**  
   `shared_preload_libraries = 'pg_stat_statements'`  
   (Если там уже есть другие библиотеки, перечисли через запятую: `'pg_stat_statements', 'другая'`.)
2. **Перезапусти кластер.** Изменение shared_preload_libraries действует только после перезапуска: `pg_ctl restart` или перезапуск сервиса.
3. **В базе выполни:**  
   `CREATE EXTENSION pg_stat_statements;`
4. **Используй.** Запросы к представлению pg_stat_statements:  
   `SELECT query, calls, total_exec_time::numeric(12,2), mean_exec_time::numeric(12,2) FROM pg_stat_statements ORDER BY total_exec_time DESC LIMIT 10;`  
   — топ-10 запросов по суммарному времени. Поле `query` — текст запроса (может быть обрезан), `calls` — сколько раз выполнялся, `total_exec_time` — суммарное время в миллисекундах, `mean_exec_time` — среднее время одного вызова.

#### Что будет, если создать расширение в одной базе и ожидать, что оно будет в другой

Расширения устанавливаются **в конкретную базу**. Если ты выполнил CREATE EXTENSION uuid_ossp в базе `app_db`, то функции uuid_generate_v4() есть **только** в `app_db`. В базе `test_db` их нет — при запросе `SELECT uuid_generate_v4();` в test_db получишь ошибку «function uuid_generate_v4() does not exist». **Решение:** подключиться к test_db и выполнить там `CREATE EXTENSION "uuid-ossp";` — тогда объекты расширения появятся и в test_db.

#### Типичная ошибка

Пытаются использовать pg_stat_statements сразу после CREATE EXTENSION, не добавив его в shared_preload_libraries и не перезапустив кластер. В результате расширение создаётся, но представление pg_stat_statements пустое или не работает как надо. **Правильно:** сначала shared_preload_libraries, перезапуск, потом CREATE EXTENSION.

#### Часто путают

**pg_available_extensions** и **pg_extension**. Первое — **каталог** того, что **можно** установить (файлы расширений есть на диске сервера). Второе — что **уже установлено** в **текущей** базе. В pg_available_extensions могут быть десятки расширений; в pg_extension — только те, которые ты установил в эту базу через CREATE EXTENSION.

#### Простыми словами

Расширение — это «дополнение» к PostgreSQL: новые типы данных, функции, операторы. Вместо того чтобы писать всё самому или пересобирать PostgreSQL, ты ставишь расширение одной командой CREATE EXTENSION в нужную базу — и в ней появляются новые возможности. Например: uuid-ossp — генерация UUID; pgvector — векторы и поиск похожих; PostGIS — работа с картами; pg_cron — запуск SQL по расписанию; pg_stat_statements — статистика «кто какой запрос сколько раз выполнял и сколько времени съел». Каждая база подключает только то, что ей нужно — в другой базе нужно снова CREATE EXTENSION.

#### Картинка в голове

Расширение — как **плагин** в браузере или игре: один «клик» (CREATE EXTENSION) — и в базе появляются новые «способности» (типы, функции). Доступные расширения — это «магазин плагинов» (pg_available_extensions); установленные — то, что ты уже «поставил» в эту базу (pg_extension).

#### Как запомнить

**CREATE EXTENSION** = одна команда, в текущей базе появляются объекты расширения. Расширения **по базе** — в другой базе нет, пока не выполнишь CREATE EXTENSION там. Некоторые расширения (pg_stat_statements) требуют **shared_preload_libraries** и **перезапуска**.

#### Проверь себя

1. В чём разница между pg_available_extensions и pg_extension?  
<details><summary>Ответ</summary> **pg_available_extensions** — расширения, **доступные** для установки: файлы есть на сервере, их можно поставить через CREATE EXTENSION. **pg_extension** — расширения, **уже установленные** в **текущей** базе данных. В одной базе может быть установлено подмножество доступных расширений.</details>

2. Почему pg_stat_statements нужно указывать в shared_preload_libraries?  
<details><summary>Ответ</summary> pg_stat_statements собирает статистику по **всем** запросам на уровне кластера; для этого его библиотека должна быть загружена при **старте** postmaster и разделять память со всеми backend-процессами. **shared_preload_libraries** как раз загружает указанные библиотеки при старте — до того, как создаются backend'ы. Без этого расширение не может корректно учитывать запросы от всех процессов.</details>

3. Установил расширение в базе A. Будут ли его функции доступны в базе B?  
<details><summary>Ответ</summary> **Нет.** Расширения устанавливаются **в конкретную базу**. Объекты (типы, функции) создаются в той базе, где выполнен CREATE EXTENSION. В другой базе (B) этих объектов нет — нужно выполнить CREATE EXTENSION в базе B отдельно.</details>

**Три пункта, чтобы не забыть (расширения).** 1) **CREATE EXTENSION** ставит расширение в **текущую** базу; одна команда — набор типов/функций. 2) Расширения **по базе** — в каждой базе ставишь отдельно. 3) pg_stat_statements (и подобные) — сначала shared_preload_libraries, перезапуск, потом CREATE EXTENSION.

#### Запомните

- **CREATE EXTENSION** — установка расширения в текущую базу; расширение = набор типов, функций, операторов и т.д.
- Расширения устанавливаются **по базе**; в другой базе нужно выполнить CREATE EXTENSION отдельно.
- Часто используемые: pgvector (векторы), PostGIS (гео), pg_cron (задачи по расписанию), pg_stat_statements (статистика запросов; требует shared_preload_libraries + перезапуск), uuid-ossp (UUID), pg_trgm (триграммы).

**Ещё раз самыми простыми словами:** Расширение — «дополнение» к базе: новые типы и функции. Ставится одной командой CREATE EXTENSION в нужную базу. В другой базе — ставишь заново. Некоторые расширения (pg_stat_statements) перед установкой нужно прописать в shared_preload_libraries и перезапустить PostgreSQL.

**Если прочитал и всё равно не понял.** Представь: PostgreSQL «из коробки» не умеет генерировать UUID. Кто-то написал «дополнение» (расширение uuid-ossp) — набор функций. Ты в своей базе выполняешь одну команду: CREATE EXTENSION "uuid-ossp". После этого в **этой** базе появляются функции uuid_generate_v4() и др. В другой базе их по-прежнему нет — там нужно так же выполнить CREATE EXTENSION. Список «что можно поставить» — pg_available_extensions; «что уже стоит в этой базе» — pg_extension.

### 42.4. Системные каталоги

**Цель раздела.**  
Понять назначение основных **системных каталогов** PostgreSQL: **pg_class** (таблицы, индексы, представления и т.д.), **pg_attribute** (столбцы отношений), **pg_index** (описание индексов). Научиться использовать полезные **представления** (pg_tables, pg_indexes, pg_stat_user_tables, pg_size_*). После раздела ты будешь знать, где в системных таблицах искать метаданные объектов и размеры.

**В этом разделе главное (три строки).** 1) **pg_class** — одна строка на каждый «отношение»: таблицу (обычную, партиционированную), индекс, последовательность, представление (relkind: 'r' — таблица, 'i' — индекс, 'v' — представление, 'S' — последовательность). Ключевые поля: oid, relname, relnamespace (схема), relkind, reltuples (оценка числа строк), relpages. 2) **pg_attribute** — по одной строке на атрибут (столбец) каждого отношения; attrelid ссылается на pg_class.oid; attnum — номер столбца; attname — имя. 3) Для удобства используют **представления**: pg_tables, pg_indexes, information_schema.tables; для статистики — pg_stat_user_tables, pg_stat_user_indexes; для размеров — pg_total_relation_size(), pg_table_size(), pg_indexes_size().

#### Термины (расшифровка)

- **pg_class** — системный каталог: строки описывают «классы» отношений — таблицы, индексы, представления, последовательности, составные типы и т.д. **relkind**: 'r' = обычная таблица, 'p' = партиционированная таблица, 'i' = индекс, 'v' = представление, 'S' = последовательность, 'c' = составной тип и др. **reltuples** — оценка числа строк (обновляется ANALYZE); **relpages** — число страниц.
- **pg_attribute** — атрибуты (столбцы) отношений. **attrelid** — OID отношения в pg_class; **attnum** — номер столбца (отрицательные — системные столбцы, например xmin, xmax); **attname** — имя столбца; **atttypid** — OID типа в pg_type.
- **pg_index** — описание индексов. **indexrelid** — OID индекса в pg_class; **indrelid** — OID таблицы; **indkey** — номера столбцов индекса; **indisunique**, **indisprimary** — уникальность и первичный ключ.

#### Теория и правила (подробно)

- **Метаданные таблицы.** Список таблиц в схеме: `SELECT * FROM pg_tables WHERE schemaname = 'public';` или из pg_class: `SELECT relname FROM pg_class WHERE relkind = 'r' AND relnamespace = (SELECT oid FROM pg_namespace WHERE nspname = 'public');`. Столбцы таблицы: `SELECT attname, atttypid FROM pg_attribute WHERE attrelid = 'public.mytable'::regclass AND attnum > 0 AND NOT attisdropped;`.
- **Размеры.** `pg_total_relation_size('schema.tablename')` — таблица + индексы + TOAST; `pg_table_size(...)` — таблица + TOAST без индексов; `pg_indexes_size(...)` — размер индексов таблицы. Функция `pg_size_pretty()` переводит байты в читаемый вид (например, "152 MB").
- **Статистика использования.** **pg_stat_user_tables**: seq_scan, idx_scan, n_tup_ins/upd/del, n_live_tup, n_dead_tup (для оценки необходимости vacuum/analyze). **pg_stat_user_indexes**: idx_scan, idx_tup_read, idx_tup_fetch — как часто используются индексы.

#### Примеры запросов

```sql
-- Список таблиц и оценка числа строк
SELECT relname, reltuples::bigint
FROM pg_class
WHERE relkind = 'r'
  AND relnamespace = (SELECT oid FROM pg_namespace WHERE nspname = 'public')
ORDER BY reltuples DESC;

-- Размер таблиц (топ по размеру)
SELECT relname,
       pg_size_pretty(pg_total_relation_size(oid)) AS total_size
FROM pg_class
WHERE relkind = 'r'
  AND relnamespace = (SELECT oid FROM pg_namespace WHERE nspname = 'public')
ORDER BY pg_total_relation_size(oid) DESC
LIMIT 10;

-- Столбцы таблицы orders
SELECT a.attname, pg_catalog.format_type(a.atttypid, a.atttypmod) AS type
FROM pg_attribute a
WHERE a.attrelid = 'public.orders'::regclass
  AND a.attnum > 0
  AND NOT a.attisdropped
ORDER BY a.attnum;
```

#### Простыми словами

pg_class — «справочник всех таблиц и индексов»: имя, тип объекта, оценка числа строк и страниц. pg_attribute — «справочник столбцов»: у какой таблицы какой столбец и какого типа. pg_index — как устроены индексы (какие столбцы, уникальный ли). Чтобы не писать сложные JOIN по каталогам, используют готовые представления (pg_tables, pg_indexes) и функции размеров (pg_total_relation_size, pg_size_pretty).

#### Картинка в голове

pg_class — оглавление книги (все «обложки» объектов); pg_attribute — содержание каждой главы (столбцы); pg_index — указатель (какие индексы по каким столбцам). Представления и функции размера — готовые «краткие справки» поверх этих каталогов.

#### Что будет, если забыть фильтр attnum > 0 в pg_attribute

В pg_attribute хранятся не только пользовательские столбцы (id, name...), но и **системные** (xmin, xmax, ctid и др. — у них attnum отрицательный). Если не отфильтровать attnum > 0 AND NOT attisdropped, в списке «столбцов таблицы» окажутся и служебные поля. **Итог:** всегда добавляй attnum > 0 AND NOT attisdropped, когда нужны только пользовательские столбцы.

#### Типичная ошибка

Пытаются вывести размер таблицы через pg_table_size и удивляются, что «размер маленький», хотя индексы большие. pg_table_size не включает индексы — только саму таблицу (heap) и TOAST. Для полного размера таблицы со всеми индексами нужна **pg_total_relation_size**.

#### Часто путают

**pg_class** и **information_schema.tables**. Оба содержат информацию о таблицах. pg_class — внутренний каталог PostgreSQL; там есть и индексы, и представления (relkind разный). information_schema.tables — стандартный SQL-каталог; там только таблицы и view. Для «просто список таблиц» часто используют pg_tables; для «таблица + оценка строк + размер» удобнее запрос к pg_class.

#### Как запомнить

**pg_class** = одна строка на таблицу/индекс/view; **relkind** говорит тип (r/i/v/S). **pg_attribute** = столбцы; attrelid → какая таблица; для пользовательских столбцов: attnum > 0 AND NOT attisdropped. Размеры — **pg_total_relation_size()** и **pg_size_pretty()**.

#### Проверь себя

1. Как по pg_class отличить таблицу от индекса?  
<details><summary>Ответ</summary> По полю **relkind**: 'r' — обычная таблица, 'p' — партиционированная таблица, 'i' — индекс, 'v' — представление, 'S' — последовательность.</details>

2. Как узнать размер таблицы вместе с индексами?  
<details><summary>Ответ</summary> **pg_total_relation_size(oid или имя)** — размер таблицы + индексы + TOAST. Удобно выводить через **pg_size_pretty(...)** для читаемого формата (например, "256 MB").</details>

3. Что такое reltuples в pg_class и насколько это точно?  
<details><summary>Ответ</summary> **reltuples** — **оценка** числа строк, которую собирает ANALYZE. Это не точное число, а приближение. Используется планировщиком. Для точного числа — SELECT count(*) FROM таблица (дорого на больших таблицах).</details>

**Три пункта, чтобы не забыть (системные каталоги).** 1) **pg_class** — одна строка на таблицу/индекс/view; **relkind** — тип (r, i, v, S). 2) **pg_attribute** — столбцы; attrelid → pg_class; фильтруй attnum > 0 AND NOT attisdropped. 3) Размеры — **pg_total_relation_size()**, читаемый вид — **pg_size_pretty()**.

#### Запомните

- **pg_class** — одна строка на таблицу, индекс, представление, последовательность (relkind: r, i, v, S и др.); relname, reltuples, relpages.
- **pg_attribute** — столбцы отношений; attrelid → pg_class; attname, attnum, atttypid; для пользовательских столбцов: attnum > 0 AND NOT attisdropped.
- **pg_index** — описание индексов; indexrelid, indrelid, indkey.
- Полезные представления и функции: pg_tables, pg_indexes; pg_stat_user_tables, pg_stat_user_indexes; pg_total_relation_size(), pg_table_size(), pg_indexes_size(), pg_size_pretty().

**Ещё раз самыми простыми словами:** pg_class — «список всех таблиц и индексов»; pg_attribute — «список столбцов каждой таблицы». Используй pg_tables для списка таблиц и pg_total_relation_size + pg_size_pretty для размеров.

**Если прочитал и всё равно не понял.** В PostgreSQL есть внутренние таблицы (pg_class, pg_attribute), где записано, какие таблицы есть и какие у них столбцы. **pg_class** — по одной строке на каждую таблицу и индекс (поле relkind: r = таблица, i = индекс). **pg_attribute** — по строке на каждый столбец. Чтобы не копаться в них: **pg_tables** — список таблиц, **pg_total_relation_size('схема.таблица')** — размер с индексами, **pg_size_pretty(...)** — байты в "MB"/"GB".

### 42.5. Конфигурация PostgreSQL (postgresql.conf, pg_hba.conf)

**Цель раздела.**  
Понять, где и как настраивается PostgreSQL: **postgresql.conf** (основные параметры: память, соединения, WAL, логирование) и **pg_hba.conf** (кто с каких адресов может подключаться и как аутентифицироваться). После раздела ты будешь знать, как изменить ключевые настройки и как открыть доступ клиентам.

**В этом разделе главное (три строки).** 1) **postgresql.conf** — главный файл конфигурации: параметры памяти (shared_buffers, work_mem), соединений (max_connections), WAL, логирования и др.; изменения вступают в силу после перезапуска или после RELOAD (зависит от параметра). 2) **pg_hba.conf** — правила доступа: для каждого типа соединения (local, host, hostssl) задаётся адрес (или диапазон), база, пользователь и метод аутентификации (trust, md5, scram-sha-256, peer и др.). 3) Расположение файлов — обычно в каталоге данных кластера ($PGDATA); показать путь: `SHOW config_file;` и `SHOW hba_file;`.

#### Термины (расшифровка)

- **postgresql.conf** — текстовый файл конфигурации экземпляра PostgreSQL. В нём задаются **параметры сервера**: размер shared_buffers, max_connections, listen_addresses (с каких адресов принимать подключения), параметры WAL (wal_level, checkpoint_timeout), логирования (log_destination, log_statement, log_duration), таймауты (statement_timeout, idle_in_transaction_session_timeout) и сотни других. Строки вида `параметр = значение`; символ `#` — комментарий. **Когда вступает в силу:** часть параметров — только после **перезапуска** кластера (например, shared_buffers, max_connections); часть — после **RELOAD** (например, log_statement) без перезапуска. В логах и документации указано, что требует restart.
- **pg_hba.conf** (Host-Based Authentication) — файл правил **доступа к кластеру**: с каких машин и под какими учётными записями разрешено подключаться и каким методом проверять пароль. Каждая строка — правило: **тип** (local — Unix-сокет, host — TCP без SSL, hostssl — TCP с SSL), **база** (имя БД или all), **пользователь** (имя или all), **адрес** (для host/hostssl — IP или маска, например 127.0.0.1/32, 192.168.0.0/24), **метод аутентификации** (trust — без пароля, md5/scram-sha-256 — по паролю, peer — под именем ОС для local). Первое совпавшее правило применяется; порядок строк важен. После изменения pg_hba.conf нужен **RELOAD** (или перезапуск), чтобы правила перечитались.
- **RELOAD** — сигнал процессу postmaster перечитать конфигурационные файлы (postgresql.conf и pg_hba.conf) без перезапуска кластера. Команда: `pg_ctl reload -D $PGDATA` или из SQL: `SELECT pg_reload_conf();` (от суперпользователя). Параметры, требующие restart, не изменятся до перезапуска.

#### Теория и правила (подробно)

- **Где лежат файлы.** Путь к каталогу данных задаётся при инициализации кластера (initdb). Внутри — postgresql.conf и pg_hba.conf. Узнать из сессии: `SHOW config_file;` — полный путь к postgresql.conf; `SHOW hba_file;` — к pg_hba.conf. Часто используется **include** или **include_dir** в postgresql.conf, чтобы подключать дополнительные файлы (удобно для разнесения настроек по окружениям).
- **Ключевые параметры postgresql.conf.** **shared_buffers** — размер буферного кэша (уже разбирали в §42.2); **work_mem** — память на операции сортировки/хеш в одной операции (увеличивать осторожно при многих одновременных запросах); **maintenance_work_mem** — для VACUUM, CREATE INDEX; **max_connections** — максимум одновременных соединений; **listen_addresses** — с каких IP слушать ('' — только localhost, '*' — все интерфейсы); **log_statement** — логировать ли все запросы (none, ddl, mod, all); **statement_timeout**, **idle_in_transaction_session_timeout** — обрывать долгие запросы и «висящие» транзакции.
- **Правила pg_hba.conf.** Типично: **local** для пользователя postgres — trust (подключение с той же машины по Unix-сокету без пароля для администрирования); **host 127.0.0.1/32** — с localhost по TCP, например scram-sha-256; **host 192.168.0.0/24** — из внутренней сети с паролем. Не ставить **trust** для host с 0.0.0.0/0 (доступ из интернета без пароля) — небезопасно.

#### Пошагово: как изменить параметр и применить (без перезапуска, если возможно)

1. Узнать путь к конфигу: подключиться к БД и выполнить `SHOW config_file;`.
2. Открыть postgresql.conf, найти параметр (или добавить строку), задать значение, например `work_mem = '64MB'`.
3. Сохранить файл. Проверить, требует ли параметр перезапуска: в документации или `SELECT * FROM pg_settings WHERE name = 'work_mem';` — смотреть контекст (postmaster — перезапуск, user/superuser — RELOAD или только сессия).
4. Если достаточно RELOAD: выполнить `SELECT pg_reload_conf();` от суперпользователя или `pg_ctl reload -D $PGDATA`. Проверить: `SHOW work_mem;`.

#### Пошагово: как разрешить подключение с другого хоста (pg_hba.conf)

1. Узнать путь: `SHOW hba_file;`.
2. Открыть pg_hba.conf. Добавить строку, например: `host all all 192.168.1.0/24 scram-sha-256` — разрешить всем пользователям (all) ко всем базам (all) с подсети 192.168.1.0/24 по паролю (scram-sha-256).
3. Убедиться, что PostgreSQL слушает на нужном интерфейсе: в postgresql.conf `listen_addresses = '*'` или конкретный IP.
4. Сохранить pg_hba.conf. Выполнить `SELECT pg_reload_conf();`. Проверить подключение с клиентской машины.

#### Что будет, если поставить trust для host 0.0.0.0/0 в pg_hba.conf

Любой хост в интернете сможет подключиться к PostgreSQL **без пароля** под любым существующим пользователем. Это критическая уязвимость: данные могут быть прочитаны, изменены или удалены. **Вывод:** для сетевых подключений (host, hostssl) использовать аутентификацию по паролю (scram-sha-256 или md5) или сертификатам; не использовать trust для широких диапазонов, кроме local.

#### Типичная ошибка

Меняют postgresql.conf или pg_hba.conf и ждут эффекта без RELOAD или перезапуска. Часть параметров применяется только после перезапуска; pg_hba.conf — после RELOAD. **Правильно:** после правки конфига выполнить `pg_ctl reload` или `SELECT pg_reload_conf();`; для параметров с context = postmaster — перезапуск кластера.

#### Часто путают

**postgresql.conf** и **pg_hba.conf**. Первый — **что** делает сервер (размеры памяти, логи, таймауты). Второй — **кто** может подключаться и **как** проверять (trust, пароль, сертификат). Оба лежат в $PGDATA; оба перечитываются по RELOAD (но параметры с context postmaster в postgresql.conf не изменятся без restart). **RELOAD** и **restart**: RELOAD не перезапускает процессы — только перечитывает конфиги; часть параметров (shared_buffers, max_connections) требует полного перезапуска кластера.

#### Простыми словами

postgresql.conf — «настройки двигателя»: сколько памяти, сколько соединений, что писать в лог, какие таймауты. pg_hba.conf — «список гостей и пропусков»: с какого адреса кому разрешено подключаться и по паролю или без. После правки файлов нужно дать серверу команду «перечитать настройки» (RELOAD) или перезапустить его (restart), иначе изменения не применятся.

#### Картинка в голове

**postgresql.conf** — **панель управления** сервером: ручки «память», «соединения», «логи», «таймауты». Крутишь ручку — меняется поведение (после RELOAD или restart). **pg_hba.conf** — **охранник у входа**: список «такой-то адрес, такой-то пользователь — по паролю (или trust)». Первая подходящая строка решает: пустить или нет. Добавил правило — сказал охраннику «перечитать список» (RELOAD).

#### Как запомнить

**postgresql.conf** — параметры сервера (память, соединения, WAL, логи); путь — SHOW config_file; часть параметров — только после restart. **pg_hba.conf** — кто и как подключается (адрес, пользователь, метод аутентификации); путь — SHOW hba_file; после правки — RELOAD. **RELOAD** — pg_reload_conf() или pg_ctl reload; перечитывает конфиги без перезапуска процессов.

#### Проверь себя

1. Чем postgresql.conf отличается от pg_hba.conf?  
<details><summary>Ответ</summary> **postgresql.conf** задаёт **параметры работы сервера**: память (shared_buffers, work_mem), число соединений (max_connections), WAL, логирование, таймауты и т.д. **pg_hba.conf** задаёт **правила доступа**: с каких адресов каким пользователям разрешено подключаться и каким методом аутентификации (trust, md5, scram-sha-256 и др.).</details>

2. Почему после изменения postgresql.conf настройка может не примениться?  
<details><summary>Ответ</summary> Часть параметров вступает в силу только после **перезапуска** кластера (context = postmaster), например shared_buffers, max_connections. Другая часть — после **RELOAD** (pg_reload_conf() или pg_ctl reload). Если не выполнить RELOAD или не перезапустить (для параметров, требующих restart), изменение в файле не подхватится.</details>

3. Как безопасно разрешить подключение с другой машины в локальной сети?  
<details><summary>Ответ</summary> В **pg_hba.conf** добавить правило с подсетью этой машины (например, 192.168.0.0/24) и методом аутентификации по паролю (**scram-sha-256** или md5), а не trust. В **postgresql.conf** убедиться, что **listen_addresses** включает нужный интерфейс (например, '*' или конкретный IP). После правки выполнить **pg_reload_conf()**.</details>

**Три пункта, чтобы не забыть (конфигурация).** 1) **postgresql.conf** — параметры сервера (память, соединения, логи); **pg_hba.conf** — правила доступа (кто, откуда, как аутентифицироваться). 2) Пути — SHOW config_file; SHOW hba_file; после правки — RELOAD (или restart для части параметров). 3) Не использовать trust для сетевых подключений из интернета — только пароль или сертификаты.

#### Запомните

- **postgresql.conf** — главный файл конфигурации; параметры памяти, соединений, WAL, логирования; часть — только после restart.
- **pg_hba.conf** — правила доступа: тип (local/host/hostssl), база, пользователь, адрес, метод аутентификации; после правки — RELOAD.
- **RELOAD** — pg_reload_conf() или pg_ctl reload; перечитывает конфиги без перезапуска; параметры с context postmaster не меняются без restart.

**Ещё раз самыми простыми словами.** postgresql.conf — «как работает сервер» (память, соединения, логи). pg_hba.conf — «кто может подключаться и по паролю ли». После правки — RELOAD; часть настроек — только после перезапуска.

**Если прочитал и всё равно не понял.** Запомни: **postgresql.conf** — файл с настройками «сколько памяти, сколько соединений, что логировать»; **pg_hba.conf** — файл с правилами «с какого IP кому разрешено подключаться и по паролю или нет». Оба в каталоге данных; после изменения нужно выполнить RELOAD (или перезапуск для части параметров). Не ставить trust для доступа из интернета — только пароль.
