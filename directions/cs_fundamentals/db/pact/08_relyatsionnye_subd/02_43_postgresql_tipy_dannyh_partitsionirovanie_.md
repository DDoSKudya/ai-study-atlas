## §43. PostgreSQL: типы данных, партиционирование, схемы

### 43.1. Типы данных

**Цель раздела.**  
Использовать расширенные типы данных PostgreSQL: **JSONB**, **массивы**, **диапазоны** (int4range, tstzrange), **UUID**, **сетевые** (inet, cidr). После раздела ты будешь знать, когда какой тип применять и как с ним работать в запросах.

**В этом разделе главное (три строки).** 1) **JSONB** — бинарный JSON; поддерживает индексы GIN; операторы: @> (содержит), ? (ключ есть), -> ->> (доступ по ключу). 2) **Массивы** — тип `[]` (text[], integer[] и др.); операторы && (пересечение), @> (содержит), unnest() для развёртки; индексы GIN. 3) **Диапазоны** (int4range, tstzrange), **UUID**, **inet/cidr** — типы для интервалов, уникальных идентификаторов и IP/подсетей; у каждого свои операторы и сценарии.

#### Термины (расшифровка)

- **JSONB** — тип данных в PostgreSQL для хранения **JSON в бинарном виде** (уже разобранного, не как строка). В отличие от типа **JSON** (который хранит текст «как есть»), JSONB нормализует пробелы и порядок ключей, поэтому сравнение и поиск быстрее. **Зачем нужен:** гибкая структура данных без фиксированного набора столбцов — логи событий, конфиги, атрибуты товара. **Операторы:** @> — «левый JSON содержит правый» (проверка вложенности); ? — «есть ли ключ»; -> — получить значение как JSON; ->> — получить значение как текст. **Индексы:** GIN по JSONB ускоряет поиск по ключам и значениям (WHERE data @> '{"status": "active"}').
- **Массив (array)** — тип «список значений» одного типа: text[], integer[], uuid[] и т.д. Один столбец может хранить несколько значений (например, теги статьи: {'sql', 'postgres', 'db'}). **Операторы:** && — пересечение (есть ли общий элемент); @> — левый массив содержит правый; <@ — левый содержится в правом. **unnest(массив)** — развернуть массив в набор строк (по одной строке на элемент). **Индексы:** GIN по массиву ускоряет запросы с && и @>.
- **Диапазон (range)** — тип для **интервала** значений без двух отдельных столбцов «начало» и «конец». **int4range** — диапазон целых; **tstzrange** — диапазон времени с таймзоной; **daterange** — диапазон дат. Удобно для периодов бронирований, смен, подписок. **Операторы:** && — пересекаются ли диапазоны; @> — левый содержит правый; <@ — левый содержится в правом. **Индексы:** GiST для быстрого поиска по диапазонам.
- **UUID** — тип для **уникального идентификатора** (128 бит). Не зависит от автоинкремента и сервера — можно генерировать на клиенте или в разных узлах без коллизий. **Генерация:** расширение uuid-ossp даёт uuid_generate_v4() (случайный); в современных PostgreSQL есть gen_random_uuid() без расширения.
- **inet** — тип для **одного IP-адреса** (IPv4 или IPv6). **cidr** — тип для **подсети** с маской (например, 192.168.0.0/24). Удобно для фильтрации по IP, ACL, гео по диапазонам. **Операторы:** << — «адрес входит в подсеть»; >> — «подсеть содержит адрес»; && — пересечение подсетей.

#### Теория и правила (подробно)

- **Когда выбирать JSONB.** Когда структура данных **не фиксирована** или часто меняется (разные ключи у разных записей), или нужен поиск по вложенным полям без жёсткой схемы. Не заменяет нормализованные таблицы, когда связи и целостность важны — JSONB не проверяет внешние ключи и формат внутри.
- **Когда выбирать массивы.** Когда у сущности есть **список однотипных значений** (теги, роли, категории) и нужен поиск «записи, у которых есть такой элемент». Один столбец-массив проще, чем отдельная таблица связей, если список небольшой и не нужны сложные запросы по связям.
- **Когда выбирать диапазоны.** Когда есть **период** (начало–конец) и нужны запросы «все бронирования, пересекающиеся с данным периодом» или «содержащие данную дату». Один столбец типа tstzrange заменяет два столбца start_time и end_time и даёт готовые операторы (&&, @>) и индексы GiST.
- **UUID vs serial.** Serial (автоинкремент) — простой, компактный, но предсказуемый и при репликации/объединении данных возможны конфликты. UUID — глобально уникальный, можно генерировать до вставки; минус — больше размер (16 байт) и индекс менее плотный.

#### Пошагово: как найти все события с action = 'login' (JSONB)

Таблица: `events (id bigint, data jsonb)`.

1. Вставка: `INSERT INTO events (data) VALUES ('{"user": "alice", "action": "login"}');` — данные хранятся в бинарном JSONB.
2. Поиск по ключу и значению: `SELECT * FROM events WHERE data @> '{"action": "login"}';` — оператор @> проверяет, что в data есть ключ "action" со значением "login". Используй **GIN-индекс** по data: `CREATE INDEX ON events USING GIN (data);` — тогда такой запрос будет быстрым.
3. Достать одно поле как текст: `SELECT data->>'user' AS username FROM events;` — ->> возвращает текст; -> вернул бы JSON (например, вложенный объект).

#### Что будет, если хранить JSON как обычный текст (тип text) вместо JSONB

Если хранить JSON в столбце типа **text**, то каждый поиск по полю внутри JSON потребует **парсинга** строки и полного перебора — индексы по содержимому (как GIN по JSONB) недоступны, запросы с WHERE data @> '...' работать не будут (такой оператор для text нет). Сравнение и сортировка по полям внутри JSON тоже неудобны. **Вывод:** для поиска и фильтрации по полям JSON используй тип **JSONB** и при необходимости GIN-индекс.

#### Типичная ошибка

Путают операторы -> и ->>. **->** возвращает значение **как JSON** (тип jsonb) — например, вложенный объект или массив; с ним можно продолжать цепочку: data->'address'->>'city'. **->>** возвращает значение **как текст** (тип text) — для вывода и сравнения со строками. Если нужна строка для WHERE или вывода в колонку — чаще нужен ->>.

#### Часто путают

**JSON** и **JSONB** в PostgreSQL. **JSON** хранит строку «как ввели» (с пробелами, порядком ключей); при каждом чтении парсится. **JSONB** хранит разобранный бинарный формат; порядок ключей не сохраняется, пробелы нормализуются; поддерживает индексы GIN и операторы @>, ? и т.д. Для новых проектов обычно берут **JSONB** — быстрее и гибче для запросов.

#### Простыми словами

JSONB — «гибкая папка» с полями без жёсткой схемы: логи, настройки, атрибуты. Массивы — «список в одной ячейке»: теги, роли. Диапазоны — «период в одной ячейке»: с какого по какое время/число. UUID — уникальный id без автоинкремента. inet/cidr — IP и подсети для фильтрации. У каждого типа свои операторы — их нужно смотреть в справке, но главное: JSONB — @> и ->>; массивы — && и unnest(); диапазоны — && и @>.

#### Картинка в голове

JSONB — **папка с вкладками** (ключи и значения); ищешь по названию вкладки (@>). Массив — **список в одной ячейке**; unnest — «высыпать» список в отдельные строки. Диапазон — **отрезок на прямой** (от и до); && — «отрезки пересекаются?». UUID — **уникальный штрих-код** для записи. inet/cidr — **адрес и сеть**; << — «адрес в этой сети?».

#### Как запомнить

**JSONB:** @> (содержит), ? (ключ есть), -> (JSON), ->> (текст); GIN для поиска. **Массивы:** && (пересечение), @>, unnest(); GIN. **Диапазоны:** int4range, tstzrange; && (пересекается), @>; GiST. **UUID:** uuid_generate_v4() или gen_random_uuid(). **inet/cidr:** << (входит в подсеть).

#### Проверь себя

1. Чем оператор -> отличается от ->> в JSONB?  
<details><summary>Ответ</summary> **->** возвращает значение как **JSON** (тип jsonb) — можно продолжать цепочку (data->'addr'->>'city'). **->>** возвращает значение как **текст** (тип text) — для вывода и сравнения со строками.</details>

2. Как найти все строки, у которых в массиве tags есть элемент 'postgres'?  
<details><summary>Ответ</summary> Условие **массив && ARRAY['postgres']** (пересечение): `SELECT * FROM table WHERE tags && ARRAY['postgres'];` Либо оператор @>: `WHERE tags @> ARRAY['postgres']` (массив содержит указанный элемент).</details>

3. Зачем использовать тип диапазона (tstzrange) вместо двух столбцов start_time и end_time?  
<details><summary>Ответ</summary> Один столбец диапазона даёт **готовые операторы** (&& — пересекаются, @> — содержит) и **индексы GiST** для быстрого поиска «все периоды, пересекающиеся с данным». Запросы типа «бронирования, пересекающиеся с периодом» пишутся короче и эффективнее.</details>

**Три пункта, чтобы не забыть (типы).** 1) **JSONB** — гибкая структура; @>, ?, ->, ->>; GIN для поиска. 2) **Массивы** — список в одном столбце; &&, @>, unnest(); GIN. 3) **Диапазоны** — интервал в одном столбце; &&, @>; GiST. UUID и inet/cidr — под свои задачи (id, IP/сети).

#### Примеры

```sql
-- JSONB: создание и запрос
CREATE TABLE events (id bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY, data jsonb);
INSERT INTO events (data) VALUES ('{"user": "alice", "action": "login", "ts": "2024-01-15T10:00:00Z"}');
SELECT * FROM events WHERE data @> '{"action": "login"}';
SELECT data->>'user' AS username FROM events;

-- Массивы
CREATE TABLE tags (id serial PRIMARY KEY, names text[]);
INSERT INTO tags (names) VALUES (ARRAY['sql', 'postgres', 'db']);
SELECT * FROM tags WHERE names && ARRAY['postgres'];
SELECT unnest(names) AS tag FROM tags;

-- Диапазоны (период бронирования)
CREATE TABLE bookings (
  id serial PRIMARY KEY,
  room_id int,
  period tstzrange
);
INSERT INTO bookings (room_id, period)
VALUES (1, tstzrange('2024-02-01 14:00', '2024-02-01 18:00'));
SELECT * FROM bookings WHERE period && tstzrange('2024-02-01 12:00', '2024-02-01 16:00');

-- UUID
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";
CREATE TABLE users (id uuid PRIMARY KEY DEFAULT uuid_generate_v4(), name text);

-- inet / cidr
SELECT '192.168.1.5'::inet << '192.168.0.0/16'::cidr;  -- true (адрес в подсети)
```

#### Запомните

- **JSONB**: @>, ?, ->, ->>; индексы GIN для поиска по содержимому.
- **Массивы**: ARRAY[], &&, @>, unnest(); GIN для поиска по элементам.
- **Диапазоны**: int4range, tstzrange; &&, @>, <@; GiST.
- **UUID**: тип uuid; uuid_generate_v4() или gen_random_uuid().
- **inet/cidr**: IP и подсети; <<, >>, &&.

**Ещё раз самыми простыми словами.** JSONB — «папка с полями» без жёсткой схемы; ищешь по ключу (@>). Массивы — «список в одной ячейке»; проверка «есть ли элемент» — && или @>. Диапазоны — «от и до» в одном столбце; пересечение — &&. UUID — уникальный id; inet/cidr — IP и сеть.

**Если прочитал и всё равно не понял.** Запомни одно: **JSONB** — когда нужны «гибкие» поля (разная структура у разных строк) и поиск по ним — используй тип jsonb и оператор @>. **Массивы** — когда в одной ячейке несколько значений (теги) — тип text[] и оператор &&. **Диапазоны** — когда есть период (начало–конец) — тип tstzrange и оператор &&. Остальное подтянешь по примерам из раздела.

### 43.2. Партиционирование

**Цель раздела.**  
Создавать **партиционированные таблицы** в PostgreSQL: по диапазону (RANGE), по списку (LIST), по хешу (HASH); создание партиций, default partition. После раздела ты будешь знать, когда применять партиционирование и как объявлять родительскую таблицу и партиции.

**В этом разделе главное (три строки).** 1) **Партиционирование** — одна логическая таблица разбивается на несколько физических (партиций) по ключу; запросы пишутся к родителю; планировщик отсекает ненужные партиции по WHERE (partition pruning). 2) **PARTITION BY RANGE** — по диапазону (дата, число); **LIST** — по списку значений (регион); **HASH** — по хешу ключа (равномерное распределение). 3) Родитель создаётся с **PARTITION BY**; партиции — **CREATE TABLE ... PARTITION OF ... FOR VALUES ...** или **DEFAULT**; первичный ключ должен включать ключ партиционирования.

#### Термины (расшифровка)

- **Партиционирование (partitioning)** — разбиение **одной логической таблицы** на несколько **физических** (партиций) по значению столбца(ов) — ключа партиционирования. Для приложения это по-прежнему одна таблица: INSERT и SELECT идут к имени родителя. PostgreSQL сам решает, в какую партицию записать строку и из каких партиций читать. При чтении по условию на ключ партиционирования планировщик **отсекает** партиции, где данных заведомо нет (partition pruning) — меньше I/O, быстрее запрос.
- **PARTITION BY RANGE (key)** — партиции задаются **диапазонами** значений ключа. Пример: ключ created_at (дата) — партиция за январь 2024 (FROM '2024-01-01' TO '2024-02-01'), за февраль (FROM '2024-02-01' TO '2024-03-01') и т.д. Строка попадает в партицию, в диапазон которой попадает значение ключа. Удобно для логов, событий по дате.
- **PARTITION BY LIST (key)** — партиции задаются **списком значений**. Пример: ключ region — партиция eu (VALUES IN ('eu')), us (VALUES IN ('us')), asia (VALUES IN ('asia')). Строка попадает в партицию, чей список содержит значение ключа. Удобно для разделения по региону, типу, статусу.
- **PARTITION BY HASH (key)** — число партиций задаётся явно; строка попадает в партицию по **остатку от деления хеша ключа** на число партиций. Границы диапазонов не задаются — распределение автоматически равномерное. Удобно, когда нужна равномерная нагрузка по партициям без привязки к смыслу значения.
- **Default partition** — партиция «на все остальные значения». Только для RANGE и LIST. Строки, не попавшие ни в одну явную партицию (дата вне диапазонов или значение не из списка), пишутся в DEFAULT. Без default партиции такая вставка даст ошибку.

#### Теория и правила (подробно)

- **Ограничения.** **Первичный ключ и уникальные ограничения** должны включать **ключ партиционирования**. Причина: уникальность проверяется внутри каждой партиции; чтобы гарантировать уникальность по всей таблице, ключ партиционирования должен быть частью уникального ключа. Внешние ключи на партиционированную таблицу и с неё — поддержка ограничена (зависит от версии PostgreSQL).
- **Создание: сначала родитель, потом партиции.** Родительская таблица объявляется с **PARTITION BY RANGE/LIST/HASH (столбец)**; в ней **нет своих данных** — это «шаблон». Партиции создаются командами **CREATE TABLE имя_партиции PARTITION OF родитель FOR VALUES FROM (…) TO (…)** (для RANGE), **FOR VALUES IN (…)** (для LIST) или **FOR VALUES WITH (MODULUS n, REMAINDER r)** (для HASH). При вставке в родителя PostgreSQL смотрит значение ключа и направляет строку в нужную партицию.
- **Default partition (RANGE/LIST).** Создаётся как **CREATE TABLE log_default PARTITION OF log DEFAULT;** — сюда попадут все строки, не подошедшие под явные партиции. Удобно как «временную» партию, но лучше явно создавать партиции и не полагаться на DEFAULT для больших объёмов — иначе одна партиция раздуется.
- **Когда применять.** Большие таблицы (сотни гигабайт, миллиарды строк), особенно по времени (логи, события) — партиционирование по дате упрощает **архивирование** (DETACH старую партицию и вынести в архив) и ускоряет запросы за счёт partition pruning. По списку — когда логически данные делятся по региону/типу. По хешу — когда нужна равномерная нагрузка без явных диапазонов.

#### Пошагово: как создать партиционированную таблицу по дате

Ты хочешь таблицу events, разбитую по месяцу (created_at).

1. **Создай родительскую таблицу** с **PARTITION BY RANGE (created_at)**. Важно: ключ партиционирования (created_at) должен входить в **PRIMARY KEY**, иначе будет ошибка:  
   `CREATE TABLE events (id bigint GENERATED ALWAYS AS IDENTITY, created_at timestamptz NOT NULL DEFAULT now(), payload jsonb, PRIMARY KEY (id, created_at)) PARTITION BY RANGE (created_at);`  
   Данных в родителе нет — это только «шаблон».
2. **Создай партиции** — по одной на каждый диапазон. Январь 2024:  
   `CREATE TABLE events_2024_01 PARTITION OF events FOR VALUES FROM ('2024-01-01') TO ('2024-02-01');`  
   Февраль:  
   `CREATE TABLE events_2024_02 PARTITION OF events FOR VALUES FROM ('2024-02-01') TO ('2024-03-01');`  
   Границы **FROM … TO** — левая включена, правая исключена; соседние партиции не должны пересекаться и не должны иметь «дыр».
3. **(Опционально) Default-партиция** для «всех остальных» дат:  
   `CREATE TABLE events_default PARTITION OF events DEFAULT;`  
   Сюда попадут строки с датой вне объявленных диапазонов.
4. **Вставка и запросы** — как к обычной таблице: `INSERT INTO events (payload) VALUES ('{}');` — PostgreSQL по значению created_at (now()) определит партицию и запишет туда. `SELECT * FROM events WHERE created_at >= '2024-02-01' AND created_at < '2024-03-01';` — планировщик отсечет остальные партиции (partition pruning) и прочитает только events_2024_02.

#### Что будет, если забыть включить ключ партиционирования в PRIMARY KEY

При создании партиционированной таблицы **PRIMARY KEY и UNIQUE** должны включать **все столбцы ключа партиционирования**. Если написать только `PRIMARY KEY (id)` при `PARTITION BY RANGE (created_at)`, PostgreSQL выдаст ошибку: уникальность проверяется **внутри каждой партиции**; чтобы гарантировать уникальность по всей таблице, ключ партиционирования должен входить в уникальный ключ (иначе в разных партициях могли бы быть две строки с одинаковым id). **Итог:** всегда включай ключ партиционирования в PK/UNIQUE, например PRIMARY KEY (id, created_at).

#### Типичная ошибка

Создают родительскую таблицу и одну партицию, забывают создавать новые партиции по мере поступления данных (например, новый месяц). Вставка со значением ключа вне объявленных диапазонов даёт ошибку «no partition found» (если нет DEFAULT партиции). **Правильно:** либо заранее создавать партиции (скриптом/cron), либо использовать DEFAULT партицию и периодически «разбирать» её или создавать явные партиции и переносить данные.

#### Часто путают

**Родительская таблица** и **партиции**. Родитель — это «логическая» таблица: в ней **нет своих данных**, только определение (PARTITION BY …). Все данные лежат **в партициях**. Запросы и вставки обращаются к **имени родителя**; PostgreSQL сам выбирает партицию по значению ключа. В системном каталоге родитель и партиции — отдельные объекты (pg_class); у партиций relispartition = true.

#### Простыми словами

Партиционирование — разбить большую таблицу на «куски» по правилу (по дате, по региону или по хешу). Запросы пишутся к одной таблице; сервер сам решает, в какую партицию писать и из каких читать. Это упрощает архивирование (отсоединить старую партицию), ускоряет запросы с условием по ключу (pruning) и может улучшить параллелизм.

#### Картинка в голове

**Родительская таблица** — это **полка с ящиками**: на полке надпись «events», а данные лежат в ящиках (партициях). Каждый ящик подписан диапазоном: «январь 2024», «февраль 2024» (RANGE) или «eu», «us» (LIST). Когда ты говоришь «положи в events» — сервер смотрит на значение ключа (дата или регион) и кладёт в нужный ящик. Когда спрашиваешь «дай события за февраль» — сервер открывает только ящик «февраль», остальные не трогает (partition pruning). Ты всегда обращаешься к **полке** (родителю), а не к каждому ящику по отдельности.

#### Как запомнить

**PARTITION BY RANGE** — по диапазону (дата, число); **LIST** — по списку (регион, тип); **HASH** — по хешу (равномерно). Родитель — с PARTITION BY; партиции — CREATE TABLE … PARTITION OF … FOR VALUES … или DEFAULT. **PK/UNIQUE включают ключ партиционирования.**

#### Проверь себя

1. Почему первичный ключ партиционированной таблицы должен включать ключ партиционирования?  
<details><summary>Ответ</summary> Уникальность проверяется **внутри каждой партиции**. Чтобы гарантировать уникальность по **всей** таблице, ключ партиционирования должен входить в уникальный ключ — иначе одна и та же уникальная величина могла бы оказаться в разных партициях.</details>

2. Что такое partition pruning?  
<details><summary>Ответ</summary> **Partition pruning** — отсечение партиций по условию WHERE: планировщик определяет, какие партиции **не содержат** подходящих строк (по ключу партиционирования), и **не сканирует** их. Меньше партиций в плане — меньше I/O, быстрее запрос.</details>

3. В какую партицию попадёт строка при PARTITION BY RANGE (created_at), если created_at = '2024-02-15'?  
<details><summary>Ответ</summary> В партицию, у которой диапазон **FROM … TO** содержит эту дату: например, FOR VALUES FROM ('2024-02-01') TO ('2024-03-01') — левая граница включена, правая исключена. 2024-02-15 попадает в этот диапазон.</details>

**Три пункта, чтобы не забыть (партиционирование).** 1) Родитель — PARTITION BY RANGE/LIST/HASH; партиции — CREATE TABLE … PARTITION OF … FOR VALUES … или DEFAULT. 2) PK/UNIQUE должны включать ключ партиционирования. 3) Partition pruning — отсечение партиций по WHERE; запросы пишутся к родителю.

#### Запомните

- **PARTITION BY RANGE** — по диапазону (дата, число); **LIST** — по списку значений; **HASH** — по хешу (равномерное распределение).
- Родитель создаётся с PARTITION BY; партиции — CREATE TABLE ... PARTITION OF ... FOR VALUES ... или DEFAULT.
- PK/UNIQUE должны включать ключ партиционирования; default partition только для RANGE/LIST.

**Ещё раз самыми простыми словами.** Одна большая таблица разбита на «куски» (партиции) по правилу — по дате, по списку значений или по хешу. Ты работаешь с именем родителя; сервер сам пишет и читает из нужных партиций. Отсечение партиций по условию (pruning) ускоряет запросы.

**Если прочитал и всё равно не понял.** Запомни цепочку: **родитель** (CREATE TABLE … PARTITION BY RANGE (столбец)) — данных в нём нет; **партиции** (CREATE TABLE имя PARTITION OF родитель FOR VALUES FROM (…) TO (…)) — в них лежат данные. Вставка и SELECT идут к **родителю**; по значению столбца партиционирования строка попадает в нужную партицию. Ключ партиционирования должен входить в PRIMARY KEY.

### 43.3. Схемы и логическая репликация

**Цель раздела.**  
Понять **схемы** (namespace объектов) и основы **логической репликации**: публикация (publication), подписка (subscription), слоты репликации. После раздела ты будешь знать, как группировать объекты по схемам и как настроить репликацию на уровне таблиц (логическая репликация).

**В этом разделе главное (три строки).** 1) **Схема (schema)** — пространство имён внутри базы; полное имя объекта — `schema.table`; по умолчанию используется схема `public`; можно создавать свои схемы для разделения приложений. 2) **Логическая репликация** — репликация на уровне **строк/транзакций**, а не байтов WAL; на primary — **публикация** (набор таблиц), на replica — **подписка**; реплика получает INSERT/UPDATE/DELETE и применяет их. 3) **Replication slot** — «закладка» на primary: WAL не удаляется, пока подписчик не прочитал; при отстающей реплике WAL накапливается — риск заполнения диска.

#### Термины (расшифровка)

- **Schema (схема)** — **пространство имён** внутри одной базы данных. Полное имя таблицы (и других объектов) — `schema_name.object_name`, например `public.orders`, `app1.users`. В одной базе могут быть несколько схем; в разных схемах могут быть объекты с одинаковым именем (app1.orders и app2.orders). По умолчанию используется схема **public**. Поиск объектов идёт по **search_path** (список схем по порядку); если имя объекта указано без схемы, PostgreSQL ищет его в схемах из search_path. **Зачем:** разделить объекты по приложениям или окружениям (схема dev, test, отчётность) без создания отдельных баз.
- **Publication (публикация)** — объект на стороне **primary**: список таблиц (или все таблицы в схеме), изменения которых нужно реплицировать. Создаётся командой **CREATE PUBLICATION имя FOR TABLE t1, t2;** или **FOR ALL TABLES IN SCHEMA schema_name**. Логическая репликация передаёт изменения (INSERT, UPDATE, DELETE) только по таблицам, входящим в публикацию.
- **Subscription (подписка)** — объект на стороне **replica**: подписка на публикацию **другой** базы (другой кластер PostgreSQL). Создаётся **CREATE SUBSCRIPTION имя CONNECTION 'host=... dbname=... user=...' PUBLICATION имя_публикации**. Replica подключается к primary, читает поток изменений по публикации и применяет их к своим копиям таблиц. При создании подписки на primary автоматически создаётся **replication slot** (если не указано create_slot = false).
- **Replication slot** — именованный слот на primary; хранит **позицию в WAL** до которой потребитель (реплика или логический декодер) прочитал данные. PostgreSQL **не удаляет** сегменты WAL, пока они не прочитаны всеми слотами. **Плюс:** реплика может отстать и потом догнать — данные не потеряются. **Минус:** если подписчик «отвалился» или сильно отстаёт, WAL на primary накапливается — каталог pg_wal растёт, возможен **заполнение диска**. Нужно следить за слотами (pg_replication_slots) и при необходимости удалять слот (DROP SUBSCRIPTION на replica или удаление слота на primary).

#### Теория и правила (подробно)

- **Схемы: создание и использование.** Создать схему: **CREATE SCHEMA my_schema;** (или CREATE SCHEMA my_schema AUTHORIZATION user). Создать таблицу в схеме: **CREATE TABLE my_schema.my_table (...);** Обращаться к таблице: **SELECT * FROM my_schema.my_table;** или добавить my_schema в search_path и писать **SELECT * FROM my_table**. По умолчанию search_path обычно включает public; при необходимости задаётся: **SET search_path TO my_schema, public;**
- **Логическая репликация vs streaming (физическая).** **Streaming replication** копирует **байты WAL** — реплика получает те же страницы, что и primary; реплика — побайтовая копия. **Логическая репликация** передаёт **логические изменения** (какие строки вставлены/обновлены/удалены); реплика может содержать **подмножество** таблиц, другую схему, другую версию PostgreSQL (в пределах совместимости). Используется для: выборочной репликации таблиц, мультимастера, загрузки в хранилище (CDC), обновления мажорной версии через логическую реплику.
- **Риск слотов.** Слот «держит» WAL на primary. Мониторинг: **SELECT * FROM pg_replication_slots;** — смотреть lag (отставание), активен ли слот. При проблемах с подписчиком слот можно удалить на primary (осторожно — подписка перестанет получать данные до пересоздания).

#### Пошагово: как создать таблицу в своей схеме

1. Создать схему: `CREATE SCHEMA app1;`
2. Создать таблицу в схеме: `CREATE TABLE app1.users (id serial PRIMARY KEY, name text);`
3. Обращаться: `SELECT * FROM app1.users;` или установить search_path: `SET search_path TO app1, public;` и затем `SELECT * FROM users;`

#### Что будет, если подписчик логической репликации долго отключён

Пока подписка отключена (реплика не подключается к primary), **replication slot** на primary сохраняет позицию этого подписчика. WAL не удаляется — сегменты накапливаются в каталоге pg_wal. При длительном отключении или сбое реплики диск на primary может **заполниться** — тогда запись в БД остановится. **Вывод:** следить за слотами (pg_replication_slots) и отстающими репликами; при необходимости удалить подписку (и слот) или починить реплику.

#### Типичная ошибка

Забывают следить за размером pg_wal при использовании логической репликации. После сбоя реплики слот остаётся, WAL копится, диск заполняется. **Правильно:** настроить мониторинг слотов и размера pg_wal; при длительном отключении подписчика — удалить подписку (DROP SUBSCRIPTION на replica) или слот на primary, затем при необходимости создать подписку заново (с начальной синхронизацией данных).

#### Простыми словами

Схема — как «папка» для таблиц внутри одной базы; полное имя — schema.table. Логическая репликация — «копировать не байты WAL, а сами изменения по таблицам»: на primary объявляешь публикацию (какие таблицы), на replica — подписку; реплика получает INSERT/UPDATE/DELETE и применяет их. Слот — «закладка» на primary: WAL не удаляется, пока реплика не прочитала; если реплика отстала, WAL копится — следить за диском.

#### Картинка в голове

**Схема** — **папки в одном шкафу** (база): в папке `app1` лежат таблицы app1.users, app1.orders; в папке `public` — public.orders. Один шкаф, разные папки. **Логическая репликация** — primary рассылает **листочки с записями** «вставь эту строку», «обнови эту», «удали эту»; replica собирает листочки и вносит изменения у себя. Не копирует «страницы блокнота» (WAL), а именно записи. **Replication slot** — закладка в блокноте primary: «до этого места replica уже прочитала»; страницы до закладки нельзя выбросить. Если replica заснула — закладка не двигается, блокнот (pg_wal) растёт и может заполнить стол (диск).

#### Как запомнить

**Схема** — namespace в базе; search_path задаёт порядок поиска. **Публикация** — набор таблиц на primary; **подписка** — на replica подключается к публикации. **Replication slot** — держит WAL для подписчика; при отстающей реплике WAL растёт — мониторить pg_wal и слоты.

#### Проверь себя

1. Чем логическая репликация отличается от streaming replication?  
<details><summary>Ответ</summary> **Streaming** копирует **байты WAL** — реплика побайтовая копия primary. **Логическая** передаёт **логические изменения** (INSERT/UPDATE/DELETE по таблицам); можно реплицировать выборочно таблицы, в другую схему; реплика не обязана быть полной копией.</details>

2. Почему при отстающей логической реплике растёт диск на primary?  
<details><summary>Ответ</summary> Для подписки создаётся **replication slot**. Слот хранит позицию в WAL до которой подписчик прочитал; WAL **не удаляется**, пока не прочитан слотом. Если реплика отстаёт или отключена, слот не двигается — WAL накапливается в pg_wal, диск заполняется.</details>

**Три пункта, чтобы не забыть (схемы и логическая репликация).** 1) **Схема** — namespace в базе; полное имя schema.table. 2) **Публикация** (primary) + **подписка** (replica) — логическая репликация по таблицам. 3) **Replication slot** — WAL не удаляется до прочтения; при отстающей реплике следить за pg_wal.

#### Запомните

- **Схема** — namespace в базе; search_path задаёт порядок поиска объектов.
- **Публикация** (publication) — набор таблиц на primary для логической репликации; **подписка** (subscription) — на replica подключается к публикации.
- **Replication slot** — сохраняет WAL для потребителя; при отстающей реплике следить за размером WAL и слотами (pg_replication_slots).

**Ещё раз самыми простыми словами.** Схема — «папка» для таблиц. Логическая репликация — на primary «публикуешь» таблицы, на replica «подписываешься»; реплика получает изменения по строкам. Слот — закладка: WAL не удаляется, пока не прочитан; отстающая реплика = растущий WAL на primary.

**Если прочитал и всё равно не понял.** Схема = имя «папки» перед именем таблицы (app.users). Публикация = «эти таблицы реплицировать»; подписка = «подключиться к той базе и получать изменения по этим таблицам». Слот = на primary «не стирать WAL, пока реплика не прочитала» — поэтому если реплика не читает, WAL копится и диск может заполниться.
