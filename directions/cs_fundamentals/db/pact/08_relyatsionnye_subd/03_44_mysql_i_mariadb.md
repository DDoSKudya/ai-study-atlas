## §44. MySQL и MariaDB

### 44.1. Движки хранения

**Цель раздела.**  
Понять разницу между движками хранения MySQL: **InnoDB** (транзакционный, по умолчанию) и **MyISAM** (табличные блокировки, без транзакций). После раздела ты будешь знать, когда какой движок используется и почему для новых проектов выбирают InnoDB.

**В этом разделе главное (три строки).** 1) **InnoDB** — движок по умолчанию в MySQL; поддерживает **транзакции** (ACID), **внешние ключи**, **MVCC**; блокировки по **строкам**; redo log и crash recovery. 2) **MyISAM** — движок **без транзакций**; при записи блокируется **вся таблица** (один пишущий в момент); после сбоя нет автоматического восстановления через redo. 3) Для новых приложений используют **InnoDB**; MyISAM — только для legacy или таблиц только для чтения.

#### Термины (расшифровка)

- **InnoDB** — **транзакционный** движок хранения в MySQL; по умолчанию с MySQL 5.5+. Поддерживает **ACID** (атомарность, согласованность, изоляция, долговечность): BEGIN/COMMIT/ROLLBACK работают; при сбое питания или краше данные восстанавливаются по **redo log** (журнал изменений). Поддерживает **внешние ключи** (проверка ссылочной целостности). **MVCC** (многоверсионность) — читающие не блокируют писателей, писатели блокируют только изменяемые **строки**, а не всю таблицу. Данные и индексы хранятся в **tablespace** (файлы .ibd или общий системный tablespace). Запись идёт в redo log и в **binlog** (для репликации и point-in-time recovery). **Итог:** «полноценная» таблица для приложений с транзакциями и целостностью.
- **MyISAM** — движок **без транзакций**. Нет redo log — при сбое (краш, отключение питания) возможны повреждения; восстановление через утилиту **myisamchk** (проверка и починка файлов). **Блокировки на уровне всей таблицы:** при INSERT/UPDATE/DELETE блокируется **вся** таблица — в момент записи только один пишущий; читающие могут ждать (или писатель ждёт читающих — зависит от настроек). Файлы на диске: .MYD (данные), .MYI (индексы). **Плюс:** простой, быстрый для **только чтения** (нет накладных транзакций и блокировок строк). **Минус:** при смешанной нагрузке запись «запирает» таблицу — для новых проектов не рекомендуется.
- **ENGINE=InnoDB / ENGINE=MyISAM** — при **CREATE TABLE** можно указать движок: `CREATE TABLE t (...) ENGINE=InnoDB;` Если не указать, используется **default_storage_engine** (в современных MySQL обычно InnoDB). Для существующей таблицы движок не меняется «на лету» — нужно создавать новую таблицу с другим движком и переносить данные.

#### Теория и правила (подробно)

- **Почему InnoDB по умолчанию.** Большинству приложений нужны **транзакции** (несколько операций как одно целое) и **внешние ключи** (целостность связей). InnoDB даёт **строковые блокировки** — при UPDATE одной строки остальные строки таблицы доступны для чтения и записи; при MyISAM блокируется вся таблица. **Crash recovery** через redo log — после сбоя MySQL при старте применяет redo и восстанавливает согласованное состояние; MyISAM так не умеет — возможны битые таблицы. **Вывод:** для новых проектов всегда InnoDB, если нет особой причины использовать другой движок.
- **Когда ещё встречается MyISAM.** Старые базы (legacy); таблицы **только для чтения** (кэш, справочники), где запись редка и табличная блокировка не мешает; в старых версиях MySQL — полнотекстовый поиск (в новых версиях InnoDB тоже поддерживает полнотекст). В **MariaDB** есть движок **Aria** — замена MyISAM с поддержкой crash recovery; для новых таблиц в MariaDB тоже часто используют InnoDB.

#### Пошагово: как понять, какой движок у таблицы

В MySQL: `SHOW TABLE STATUS LIKE 'имя_таблицы';` — в столбце **Engine** будет InnoDB или MyISAM. Либо: `SELECT ENGINE FROM information_schema.TABLES WHERE TABLE_SCHEMA = 'имя_базы' AND TABLE_NAME = 'имя_таблицы';`

#### Что будет, если использовать MyISAM для таблицы с частыми обновлениями

При каждом UPDATE/DELETE/INSERT MyISAM блокирует **всю таблицу**. Пока одна сессия пишет, другие сессии, которые хотят писать или (при определённых настройках) читать эту таблицу, **ждут**. При высокой конкуренции запросы выстраиваются в очередь — задержки растут. Плюс при сбое питания или краше возможны повреждения данных — нет redo log. **Вывод:** для таблиц с записью используй InnoDB.

#### Типичная ошибка

Оставляют движок по умолчанию из старых настроек (MyISAM) или создают таблицу без указания ENGINE и получают MyISAM. Потом добавляют внешние ключи или ожидают транзакционное поведение — и сталкиваются с ограничениями (MyISAM не поддерживает FK и не даёт настоящих транзакций). **Правильно:** явно указывать ENGINE=InnoDB при создании таблиц (или убедиться, что default_storage_engine = InnoDB).

#### Часто путают

**InnoDB** и **MyISAM** по файлам на диске. InnoDB хранит данные в **tablespace** — либо один общий файл (ibdata1), либо отдельный файл .ibd на таблицу (при включённом file-per-table). MyISAM — всегда отдельные файлы: .frm (структура), .MYD (данные), .MYI (индексы). По расширениям файлов в каталоге базы можно понять движок: .ibd — InnoDB; .MYD/.MYI — MyISAM.

#### Простыми словами

InnoDB — «полноценная» таблица: транзакции, внешние ключи, блокировки по строкам, восстановление после сбоя по redo log. MyISAM — «простая» таблица: без транзакций, при записи блокируется вся таблица, при сбое нет автоматического восстановления. Для новых проектов — InnoDB; MyISAM — только если есть веская причина (legacy, только чтение).

#### Картинка в голове

**InnoDB** — **банковская ячейка**: можно положить несколько вещей (транзакция), закрыть и потом либо всё подтвердить (COMMIT), либо отменить (ROLLBACK); при сбое банк по журналу восстановит содержимое. Блокировка только той ячейки, с которой работаешь — остальные доступны. **MyISAM** — **общий стол**: когда кто-то пишет на листе, весь стол занят — никто другой не может ни писать, ни (при настройках) читать этот лист. Нет журнала: если свет выключили, что не успели записать — потеряно или испорчено.

#### Как запомнить

**InnoDB** = по умолчанию; транзакции, FK, MVCC, строковые блокировки; redo + crash recovery. **MyISAM** = без транзакций; блокировка всей таблицы при записи; для новых проектов не использовать.

#### Проверь себя

1. Почему при записи в MyISAM блокируется вся таблица?  
<details><summary>Ответ</summary> MyISAM не поддерживает **строковые блокировки** — механизм блокировок у него только на уровне **таблицы**. При любом INSERT/UPDATE/DELETE движок ставит блокировку на всю таблицу, поэтому в момент записи только одна сессия может менять эту таблицу.</details>

2. Зачем InnoDB пишет в redo log?  
<details><summary>Ответ</summary> **Redo log** — журнал изменений; запись в него идёт до сброса изменённых страниц на диск. При **сбое** (краш, питание) по redo log можно **восстановить** незаписанные изменения и привести данные в согласованное состояние при следующем старте MySQL. Без redo (как у MyISAM) после сбоя возможны повреждения.</details>

**Три пункта, чтобы не забыть (движки MySQL).** 1) **InnoDB** — по умолчанию; транзакции, FK, MVCC, строковые блокировки; redo и crash recovery. 2) **MyISAM** — без транзакций; блокировка всей таблицы при записи; не для новых проектов. 3) Движок задаётся при CREATE TABLE: ENGINE=InnoDB.

#### Запомните

- **InnoDB** — по умолчанию; транзакции, FK, MVCC, строковые блокировки; redo log и crash recovery.
- **MyISAM** — без транзакций; табличные блокировки; для новых проектов не рекомендуется.
- Указание движка: CREATE TABLE ... ENGINE=InnoDB;

**Ещё раз самыми простыми словами.** InnoDB — «нормальная» таблица с транзакциями и блокировками по строкам; при сбое восстанавливается по redo. MyISAM — без транзакций, при записи блокируется вся таблица; при сбое может повредиться. Новые таблицы делай на InnoDB.

**Если прочитал и всё равно не понял.** Запомни одно: в MySQL по умолчанию таблицы должны быть **InnoDB** — у них есть транзакции и блокировки по строкам. **MyISAM** — старый движок: при записи блокируется вся таблица и нет восстановления после сбоя. Если не уверен — создавай таблицы с **ENGINE=InnoDB**.

### 44.2. Binlog и репликация

**Цель раздела.**  
Понять роль **бинарного лога (binlog)** в MySQL и основы **репликации**: мастер–реплика, **GTID**, при необходимости — group replication. После раздела ты будешь знать, как устроена репликация на основе binlog и зачем нужен GTID.

**В этом разделе главное (три строки).** 1) **Binlog** — последовательность событий изменений на master (DML, DDL); replica читает binlog и применяет события у себя — получается копия данных. 2) **Репликация мастер–реплика** — один master, одна или несколько replica; replica подключается к master, запрашивает binlog с позиции (или с GTID) и применяет события. 3) **GTID** — уникальный идентификатор транзакции в кластере; репликация по GTID не привязана к позиции в файле — удобно для failover и смены master.

#### Термины (расшифровка)

- **Binlog (binary log)** — **лог изменений** на сервере MySQL: в него записываются все события, меняющие данные (INSERT, UPDATE, DELETE) и структуру (DDL: CREATE, ALTER, DROP). Форматы: **ROW** — записываются **изменённые строки** (до/после); надёжно для репликации (одинаковый результат на replica). **STATEMENT** — записывается **текст SQL**; меньше объём, но недетерминированные запросы (RAND(), NOW() без фиксации) могут дать расхождение на replica. **MIXED** — комбинация: по умолчанию STATEMENT, для недетерминированных — ROW. Binlog нужен для **репликации** (replica читает и применяет события) и для **point-in-time recovery (PITR)** — восстановление до произвольного момента по бэкапу + применение binlog.
- **GTID (Global Transaction Identifier)** — **глобальный идентификатор транзакции** в рамках кластера: пара (source_id, transaction_id). source_id — обычно server_uuid master’а; transaction_id — порядковый номер транзакции на этом сервере. **Одна и та же транзакция** на master и на replica имеет **один и тот же GTID**. Репликация «по GTID» означает: replica запрашивает у master «все транзакции с GTID больше, чем у меня» — не нужно хранить **позицию в файле** (имя файла binlog + смещение). При смене master (failover) новая replica просто подключается к новому master и продолжает по GTID — не нужно вручную искать позицию.
- **Replica (slave)** — сервер MySQL, который **читает binlog** с master и **применяет** события к своим данным (воспроизводит те же INSERT/UPDATE/DELETE). В итоге данные на replica **синхронизируются** с master (с задержкой — replication lag). Настраивается через **CHANGE REPLICATION SOURCE TO** (в MySQL 8.0+) или старый **CHANGE MASTER TO**: указывается хост master’а, пользователь, пароль, и позиция в binlog (или «начать с GTID»). После **START REPLICA** (или START SLAVE) replica подключается к master и начинает применять события.

#### Теория и правила (подробно)

- **Асинхронная репликация.** Master записывает транзакцию в binlog и **не ждёт**, пока replica получит и применит события — сразу возвращает успех клиенту. **Плюс:** минимальная задержка записи. **Минус:** при падении master в момент сбоя последние секунды данных могли ещё не попасть на replica — возможна **потеря данных**. **Полусинхронная (semi-sync)** — master ждёт подтверждения от **хотя бы одной** replica (что событие получено), до таймаута — снижает риск потери при сбое master, но увеличивает задержку записи.
- **Group replication (MySQL)** — несколько узлов MySQL в **группе**; репликация идёт через **протокол группы** (консенсус, как в распределённых системах). Можно настроить **single-primary** (один узел принимает запись, остальные — реплики для чтения и failover) или **multi-primary** (несколько узлов принимают запись). При отказе primary группа автоматически выбирает нового primary. Сложнее в настройке и эксплуатации, но даёт отказоустойчивость без ручного переключения.
- **Зачем GTID.** Без GTID replica хранит **позицию** в binlog (имя файла + смещение). При смене master нужно вручную определить «с какой позиции на новом master продолжать» — легко ошибиться. С GTID replica просто говорит «дай мне все транзакции, которых у меня нет» — по номерам GTID; не привязаны к конкретному файлу и смещению. Удобно для автоматического failover и для топологий с несколькими replica и сменой master.

#### Пошагово: что происходит при репликации (упрощённо)

1. На **master** клиент выполняет INSERT/UPDATE/DELETE; транзакция коммитится и записывается в **binlog** (и в redo log InnoDB).
2. **Replica** подключается к master и запрашивает события из binlog (с последней позиции или с последнего GTID).
3. Master отправляет события из binlog replica. Replica записывает их в свой **relay log** и применяет к своим данным (выполняет те же изменения в InnoDB).
4. Replica обновляет свою позицию (или набор применённых GTID) и снова запрашивает новые события. Цикл повторяется. Задержка между записью на master и применением на replica — **replication lag**.

#### Что будет, если использовать формат binlog STATEMENT для недетерминированных запросов

В формате **STATEMENT** в binlog пишется **текст SQL**. Запрос вроде `INSERT INTO t SELECT * FROM s ORDER BY RAND() LIMIT 1` на master выберет одну случайную строку; на replica **RAND()** даст другое значение — может быть выбрана **другая** строка. Итог — **расхождение** данных между master и replica. **Вывод:** для надёжной репликации используют формат **ROW** (в binlog пишутся сами изменённые строки) или **MIXED** (для недетерминированных автоматически ROW).

#### Типичная ошибка

Настраивают репликацию без GTID и при failover вручную ищут позицию на новом master — ошибаются в позиции или файле, replica начинает с неправильного места, данные расходятся. **Правильно:** включать **GTID** (gtid_mode=ON) и настраивать репликацию по GTID — при смене master replica подключается к новому master и продолжает по GTID без ручного подбора позиции.

#### Часто путают

**Binlog** и **redo log (InnoDB)**. **Redo log** — внутренний журнал **InnoDB** для восстановления после сбоя (crash recovery); не используется для репликации. **Binlog** — лог на уровне **сервера MySQL**; в него попадают все изменения (не только InnoDB); используется для **репликации** и для **PITR**. Оба пишутся при записи, но назначение разное: redo — восстановление самого сервера; binlog — репликация и восстановление до момента.

#### Простыми словами

Binlog — «дневник изменений» master’а; replica переписывает этот дневник у себя и повторяет те же изменения. GTID — номер каждой «операции» в дневнике; по нему легко понять, что уже применено и что ещё нет, и не привязываться к конкретному файлу и позиции. Репликация по GTID упрощает переключение на нового master (failover).

#### Картинка в голове (binlog и репликация)

**Master** ведёт **дневник** (binlog): каждая операция — запись «вставь эту строку», «обнови эту». **Replica** — копиист: подключается к master, переписывает дневник к себе и **выполняет** те же операции у себя — так данные совпадают. **GTID** — номер каждой записи в дневнике (например, «транзакция № 100»). При смене master replica говорит новому master: «дай мне всё после номера 100» — не нужно искать «на какой странице и в какой строке» остановились (позиция в файле). Удобно для автоматического переключения (failover).

#### Как запомнить

**Binlog** — лог изменений master’а; форматы ROW, STATEMENT, MIXED; для репликации и PITR. **Репликация** — replica читает binlog и применяет события; настройка CHANGE REPLICATION SOURCE TO. **GTID** — глобальный ID транзакции; репликация по GTID не привязана к позиции в файле; удобно для failover.

#### Проверь себя

1. Зачем нужен GTID при репликации MySQL?  
<details><summary>Ответ</summary> **GTID** — уникальный идентификатор каждой транзакции в кластере. Репликация по GTID не привязана к **позиции в файле** binlog (имя файла + смещение). При **failover** (смена master) replica подключается к новому master и запрашивает «все транзакции с GTID больше, чем у меня» — не нужно вручную искать позицию на новом master. Упрощает автоматический failover и снижает риск ошибки.</details>

2. Чем формат binlog ROW отличается от STATEMENT?  
<details><summary>Ответ</summary> **ROW** — в binlog записываются **изменённые строки** (значения до/после); на replica применяются те же строки — результат гарантированно совпадает. **STATEMENT** — записывается **текст SQL**; на replica выполняется тот же SQL — для недетерминированных запросов (RAND(), NOW() без фиксации) результат может отличаться. Для надёжной репликации предпочтителен ROW или MIXED.</details>

**Три пункта, чтобы не забыть (binlog и репликация).** 1) **Binlog** — лог изменений master’а; форматы ROW, STATEMENT, MIXED; для репликации и PITR. 2) **Репликация** — replica читает binlog и применяет события; настройка CHANGE REPLICATION SOURCE TO (или CHANGE MASTER TO). 3) **GTID** — глобальный ID транзакции; репликация по GTID удобна для failover.

#### Запомните

- **Binlog** — лог изменений master’а; форматы ROW, STATEMENT, MIXED; используется для репликации и PITR.
- **Репликация** — replica читает binlog с master и применяет события; настройка через CHANGE REPLICATION SOURCE TO (или CHANGE MASTER TO).
- **GTID** — глобальный ID транзакции; репликация по GTID не привязана к позиции в файле; удобно для failover.

**Ещё раз самыми простыми словами.** Binlog — дневник изменений на master; replica читает его и повторяет те же изменения у себя. GTID — номер каждой операции; по нему видно, что уже применено; при смене master не нужно искать позицию в файле — удобно для failover.

**Если прочитал и всё равно не понял.** Запомни цепочку: **master** пишет все изменения в **binlog**. **Replica** подключается к master, читает binlog и **применяет** те же изменения к своим таблицам — так данные копируются. **GTID** — это «номер» каждой транзакции; при смене master replica говорит «дай мне все транзакции с номерами больше N» — не нужно помнить «в каком файле и на каком месте» остановились.

### 44.3. Настройки и MariaDB

**Цель раздела.**  
Знать ключевые **настройки** MySQL (innodb_buffer_pool_size, изоляция по умолчанию) и кратко — отличия **MariaDB** (движки Aria, ColumnStore, совместимость с MySQL). После раздела ты будешь знать, что в первую очередь настраивать и чем MariaDB отличается от MySQL.

**В этом разделе главное (три строки).** 1) **innodb_buffer_pool_size** — главный параметр памяти: кэш данных и индексов InnoDB; обычно 50–80% RAM на выделенном сервере. 2) Изоляция по умолчанию в MySQL — **REPEATABLE READ** (в InnoDB с gap locks — защита от фантомов в пределах реализации). 3) **MariaDB** — форк MySQL; совместимость по протоколу и SQL; свои движки (Aria, ColumnStore); свои расширения.

#### Термины (расшифровка)

- **innodb_buffer_pool_size** — параметр конфигурации MySQL (в my.cnf или SET GLOBAL): **размер буферного пула InnoDB** в байтах. В этом пуле кэшируются **страницы** таблиц и индексов InnoDB — те же страницы, что лежат на диске в tablespace. При чтении страница попадает в buffer pool; при записи изменённая страница сначала в пуле, затем сбрасывается на диск. **Чем больше** пул — тем больше данных помещается в RAM, тем меньше обращений к диску. Основной потребитель RAM на сервере БД; типично **50–80%** доступной RAM при выделенном сервере под MySQL. Слишком большой пул (почти вся RAM) может привести к нехватке памяти для соединений и других нужд.
- **transaction_isolation** — уровень изоляции транзакций: READ UNCOMMITTED, READ COMMITTED, REPEATABLE READ, SERIALIZABLE. В MySQL (InnoDB) по умолчанию — **REPEATABLE READ**. В InnoDB при REPEATABLE READ используются **gap locks** (блокировки «промежутков») — это даёт защиту от фантомного чтения в большинстве случаев (в пределах реализации). Для смены уровня: SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED; (или глобально в конфиге).
- **MariaDB** — **форк** MySQL: исходный код MySQL был взят за основу, после чего MariaDB развивается **отдельно**. Совместим по **протоколу** и по **большинству SQL** — клиенты и приложения, работающие с MySQL, обычно работают и с MariaDB (подключение по тому же порту, те же драйверы). Отличия: свои движки хранения (**Aria** — замена MyISAM с поддержкой crash recovery; **ColumnStore** — колоночное хранение для аналитики; **MyRocks** — на базе RocksDB); свои расширения и оптимизации; свой цикл релизов.

#### Теория и правила (подробно)

- **Почему buffer pool так важен.** InnoDB не полагается только на кэш ОС — у него свой буферный пул. Чтение и запись идут через этот пул. При нехватке места в пуле «старые» страницы вытесняются; при повторном обращении к ним — чтение с диска. Чем больше пул — тем реже вытеснение и тем меньше дискового I/O. После изменения innodb_buffer_pool_size нужен **перезапуск** MySQL (или в некоторых версиях — динамическое изменение с ограничениями).
- **MariaDB: когда выбирать.** Если нужна совместимость с MySQL и те же драйверы/клиенты — MariaDB подходит. Если нужны специфичные движки MariaDB (Aria, ColumnStore) или их расширения — выбор в пользу MariaDB. Если экосистема и документация Oracle MySQL важнее — остаются на MySQL. В целом для нового проекта можно выбирать любую; миграция между ними возможна (дамп/восстановление, с учётом отличий в диалекте и возможностях).

#### Пошагово: как посмотреть и изменить buffer pool

Текущее значение: `SHOW VARIABLES LIKE 'innodb_buffer_pool_size';` — в байтах. Изменить на лету (если версия поддерживает): `SET GLOBAL innodb_buffer_pool_size = 8589934592;` (8 GB в байтах). Постоянно — в конфиге (my.cnf): `innodb_buffer_pool_size = 8G` и перезапуск MySQL.

#### Что будет, если поставить innodb_buffer_pool_size на 90% RAM

Для всего остального (соединения, буферы запросов, кэш таблиц, ОС) останется 10% RAM. При многих соединениях и тяжёлых запросах возможна нехватка памяти — своп, падение производительности или OOM. **Вывод:** оставлять запас; ориентир 50–80% RAM на выделенном сервере под БД.

#### Типичная ошибка

Оставляют innodb_buffer_pool_size по умолчанию (128 MB или 256 MB) на сервере с 16–32 GB RAM. В результате большая часть данных читается с диска — запросы медленные. **Правильно:** на сервере, выделенном под MySQL, поднять innodb_buffer_pool_size до 50–80% RAM (например, 8–12 GB при 16 GB RAM) и перезапустить MySQL.

#### Часто путают

**innodb_buffer_pool_size** — это именно **кэш InnoDB** (страницы таблиц и индексов), а не «вся память MySQL» и не кэш ОС. Другие буферы (connection buffer, sort buffer, join buffer и т.д.) настраиваются отдельно. **MariaDB** и **MySQL** — не одно и то же: это разные продукты с общим прошлым; совместимость высокая, но не стопроцентная (свои движки, свои настройки, свои версии). При миграции с MySQL на MariaDB (или наоборот) проверяй отличия в документации.

#### Простыми словами

Главная «ручка» памяти в MySQL под InnoDB — buffer pool: чем больше, тем больше данных в RAM и меньше обращений к диску. Уровень изоляции по умолчанию — REPEATABLE READ. MariaDB — «родственная» СУБД: можно подключаться теми же клиентами, но есть свои движки и возможности.

#### Картинка в голове

**Buffer pool** — **полка с папками** (страницы таблиц и индексов): чем больше полка (innodb_buffer_pool_size), тем больше папок помещается; не нужно каждый раз ходить в архив (диск). Ставь полку побольше (50–80% RAM), но не занимай всю комнату — остаётся место для стола (соединения) и для людей (ОС). **MariaDB** — **родственная фирма** с тем же входом (протокол) и теми же клиентами; внутри свои «цеха» (движки Aria, ColumnStore), но снаружи выглядит как MySQL.

#### Как запомнить

**innodb_buffer_pool_size** — кэш InnoDB; 50–80% RAM на выделенном сервере. **Изоляция** по умолчанию — REPEATABLE READ. **MariaDB** — форк MySQL; совместимость + движки Aria, ColumnStore и др.
