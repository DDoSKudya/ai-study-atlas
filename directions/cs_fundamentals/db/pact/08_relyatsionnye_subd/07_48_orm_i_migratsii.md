## §48. ORM и миграции

### 48.1. ORM vs raw SQL

**Цель раздела.**  
Понимать плюсы и минусы **ORM** (Object-Relational Mapping) и **raw SQL**; когда что применять. После раздела ты будешь осознанно выбирать между объектной работой с таблицами и прямыми запросами.

**В этом разделе главное (три строки).** 1) **ORM** — слой между приложением и БД: объекты языка (классы, экземпляры) отображаются на таблицы и строки; запросы строятся через API (например, User.query.filter_by(name='x').first()). Плюсы: меньше ручного SQL, типобезопасность в коде, защита от инъекций при использовании параметров, переносимость между СУБД (в пределах возможного). Минусы: сложные запросы (много JOIN, оконные функции, специфика СУБД) могут быть неудобны или невыразимы; риск N+1; необходимо понимать, какой SQL генерируется. 2) **Raw SQL** — прямые запросы к БД (строками или построителями). Полный контроль над планом и диалектом; для сложной аналитики и тонкой оптимизации часто нужен именно raw SQL. 3) На практике часто комбинируют: ORM для CRUD и простых сценариев; raw SQL (или «сырые» запросы через ORM) для тяжёлых отчётов и специфичных конструкций.

#### Термины (расшифровка)

- **ORM (Object-Relational Mapping)** — библиотека, которая отображает классы приложения на таблицы, экземпляры — на строки; типично предоставляет API для выборки (query), вставки, обновления, удаления и связей (relationship). Примеры: SQLAlchemy (Python), Hibernate (Java), TypeORM (Node.js), Entity Framework (.NET).
- **Raw SQL** — запросы, написанные вручную (строками или через query builder без полной абстракции объектов). Выполняются через драйвер или метод ORM (например, session.execute(text("SELECT ..."))).

#### Теория и правила (подробно)

- **Когда предпочтительнее ORM.** Типичный CRUD (создать/прочитать/обновить/удалить запись), работа со связями «один ко многим» (пользователь и его заказы), быстрая разработка и смена схемы через модели, типобезопасность в коде (IDE подсказывает поля). ORM сокращает объём кода и снижает риск опечаток и инъекций при использовании параметров через API.
- **Когда предпочтительнее raw SQL.** Сложные отчёты (много JOIN, подзапросы, оконные функции, агрегаты по группам), тонкая оптимизация плана (подсказки, специфика СУБД), конструкции, которые ORM не выражает или выражает нечитаемо. Raw SQL даёт полный контроль над текстом запроса и планом.
- **Комбинация.** Основной код приложения — ORM для сущностей и связей; тяжёлые отчёты и админ-запросы — raw SQL через session.execute(text("...")) или аналог. Следить за N+1 в ORM (eager loading для связей).

#### Пошагово: как выбрать — ORM или raw SQL (кратко)

1. Запрос простой: «получить запись по id», «сохранить объект», «список с фильтром по одному полю» — **ORM** (меньше кода, типобезопасность).
2. Запрос со связями: «пользователь и его заказы» — **ORM с eager loading** (один запрос с JOIN), не N запросов в цикле.
3. Запрос сложный: несколько JOIN, оконные функции, агрегаты, специфика СУБД — **raw SQL** (session.execute(text("SELECT ... JOIN ..."))); проще читать и оптимизировать.
4. Меняешь схему часто — ORM и миграции помогают держать модели и БД в sync; тяжёлые запросы по-прежнему можно выносить в raw SQL.

#### Что будет, если писать только raw SQL без ORM для всего приложения

Полный контроль над запросами и планами, но **больше кода**: каждый CRUD — ручной INSERT/UPDATE/SELECT с перечислением столбцов; при изменении схемы нужно править все места, где используется таблица; выше риск опечаток и SQL-инъекций при ручной подстановке. Для простых CRUD и связей (один пользователь — много заказов) ORM сокращает код и снижает ошибки. **Вывод:** комбинировать: ORM для типичного CRUD и связей; raw SQL для сложных отчётов и оптимизированных запросов.

#### Типичная ошибка

Пытаются «выжать» из ORM сложный запрос (много JOIN, оконные функции, специфика СУБД) — код становится нечитаемым или ORM генерирует неоптимальный SQL. **Правильно:** для таких запросов писать **raw SQL** (или query builder без полной объектной модели) и выполнять через session.execute(text("...")) или аналог — быстрее и яснее.

#### Часто путают

**ORM** и **query builder**. ORM отображает **объекты** на таблицы (класс User → таблица users); запросы строятся через объектный API (User.query.filter_by(...)). **Query builder** — построение SQL через цепочку методов без привязки к объектам (select().where().join()); это не полноценный ORM, но удобнее raw SQL-строк для динамических запросов. И ORM, и query builder могут генерировать SQL; разница в уровне абстракции (объекты vs запросы).

#### Простыми словами

ORM — «работать с таблицами как с объектами»: удобно для простых операций и быстрого развития; сложные запросы и тонкая настройка — часто проще и яснее на raw SQL. Лучшая практика — знать оба подхода и использовать ORM там, где он упрощает жизнь, и raw SQL там, где нужен контроль и производительность.

#### Картинка в голове

**ORM** — **переводчик объект ↔ таблица**: ты работаешь с объектом User (user.name, user.save()); ORM сам переводит это в INSERT/UPDATE и обратно. Удобно, когда много простых операций «взять запись — изменить — сохранить». **Raw SQL** — ты сам пишешь запрос: «SELECT … JOIN … WHERE …» — полный контроль, но больше кода и нужно знать диалект СУБД. Как **автомат** (ORM) vs **ручная коробка** (raw SQL): автомат проще в пробках, ручная — когда нужен точный контроль.

#### Как запомнить

**ORM** — плюсы: удобство CRUD, типобезопасность, переносимость; минусы: сложные запросы, N+1, нужно понимать генерируемый SQL. **Raw SQL** — полный контроль; для сложной аналитики и оптимизации. **Комбинация** — ORM для основного кода; raw SQL для отчётов и тяжёлых запросов.

#### Проверь себя

1. Когда предпочтительнее raw SQL, а когда ORM?  
<details><summary>Ответ</summary> **ORM** — для типичного CRUD, связей между сущностями, быстрого развития; когда запросы простые или средней сложности. **Raw SQL** — для сложных отчётов (много JOIN, оконные функции, агрегаты), тонкой оптимизации и специфичных конструкций СУБД; когда нужен полный контроль над планом и текстом запроса.</details>

2. Что такое антипаттерн N+1 в контексте ORM?  
<details><summary>Ответ</summary> **N+1** — один запрос за списком записей (например, 100 пользователей) и **N запросов** в цикле по каждой записи за связанными данными (заказы каждого пользователя). Итого 1 + N запросов к БД вместо одного запроса с JOIN (или batch load). В ORM решается **eager loading** (подгрузка связей одним запросом).</details>

**Три пункта, чтобы не забыть (ORM vs raw SQL).** 1) **ORM** — удобство CRUD и связей; минусы — сложные запросы, N+1. 2) **Raw SQL** — полный контроль; для отчётов и оптимизации. 3) **Комбинация** — ORM для основного кода; raw SQL для тяжёлых запросов.

#### Запомните

- **ORM** — плюсы: удобство CRUD, типобезопасность, переносимость; минусы: сложные запросы, N+1, нужно понимать генерируемый SQL.
- **Raw SQL** — полный контроль; для сложной аналитики и оптимизации.
- Комбинация: ORM для основного кода приложения; raw SQL для отчётов и тяжёлых запросов.

**Ещё раз самыми простыми словами.** ORM — работать с таблицами как с объектами; удобно для простых операций. Сложные запросы — часто проще и быстрее писать raw SQL. Используй ORM для основного кода и raw SQL для отчётов и оптимизации.

**Если прочитал и всё равно не понял.** Запомни: **ORM** = библиотека, которая «переводит» объекты приложения в таблицы и запросы; удобно для CRUD. **Raw SQL** = ты сам пишешь запрос строкой или построителем. Для сложных запросов (много JOIN, оконные функции) обычно проще raw SQL; для простого «сохранить/загрузить по id» — ORM удобнее.

### 48.2. Миграции схемы

**Цель раздела.**  
Понять, зачем нужны **миграции схемы** (версионирование DDL) и как работают инструменты **Alembic** (Python), **Flyway**, **Liquibase** (Java и универсальные); connection string и **failover** в драйвере. После раздела ты будешь знать, как вести историю изменений схемы и применять их в разных окружениях.

**В этом разделе главное.** 1) **Миграция** — версионированное изменение схемы БД (создание/изменение таблиц, индексов, ограничений и т.д.); хранится в коде (файлы миграций) и в БД (таблица версий). При деплое применяются только **новые** миграции — схема в разных окружениях и у разных разработчиков совпадает. 2) **Alembic** — миграции для SQLAlchemy (Python); команды: upgrade (применить), downgrade (откатить); история в таблице alembic_version. **Flyway** — миграции на чистом SQL (или Java); порядок по именам файлов (V1__description.sql); таблица flyway_schema_history. **Liquibase** — миграции в XML/YAML/SQL; изменения описываются декларативно. 3) **Connection string** — строка подключения к БД (хост, порт, база, пользователь, пароль, параметры: sslmode, connect_timeout и т.д.). **Failover в драйвере** — указание нескольких хостов (primary + standby) и политики при обрыве соединения (переподключиться к другому хосту); зависит от драйвера (например, pgjdbc с targetServerType=preferSecondary для чтения с реплики).

#### Термины

- **Миграция схемы** — версионированное изменение структуры БД (DDL); файл миграции содержит «вперёд» (up) и при необходимости «назад» (down). Применяются по порядку; в БД хранится текущая версия (номер или имя последней применённой миграции).
- **Alembic** — инструмент миграций для Python/SQLAlchemy; миграции пишутся на Python или генерируются по изменениям моделей; команды alembic upgrade head, alembic downgrade -1.
- **Flyway** — миграции на SQL (или Java); файлы V{version}__{description}.sql; при запуске приложения или отдельной команды применяются неприменённые миграции; таблица flyway_schema_history.
- **Connection string** — строка подключения, например: postgresql://user:pass@host:5432/dbname?sslmode=require&connect_timeout=5. Параметры: host, port, dbname, user, password, sslmode, connect_timeout, statement_timeout и др.
- **Failover** — переключение на другой узел при недоступности основного. В драйвере: несколько хостов в connection string; при разрыве соединения драйвер может переподключиться к следующему хосту (зависит от драйвера и настроек).

#### Теория и правила (подробно)

- **Зачем миграции в коде.** Схема БД (таблицы, столбцы, индексы) меняется со временем. Если каждый разработчик и каждое окружение (dev, test, prod) меняют схему **вручную** или в разном порядке, схема **расходится** — приложение падает с «column does not exist» или «table already exists». Миграции — это **версионированные файлы** с DDL (CREATE TABLE, ALTER TABLE и т.д.); при деплое применяются только **неприменённые** миграции (по таблице версий в БД). Так схема везде одинаковая.
- **Инструменты.** **Alembic** — для Python/SQLAlchemy; миграции на Python или автогенерация по моделям; команды alembic upgrade head, alembic downgrade -1. **Flyway** — на SQL (или Java); файлы V{version}__{description}.sql; применяются при старте приложения или отдельной командой. **Liquibase** — изменения в XML/YAML/SQL; кросс-платформенно. Выбор по языку стека и предпочтениям команды.
- **Connection string и failover.** В connection string указывают хост, порт, базу, пользователя, пароль и параметры (sslmode, connect_timeout, statement_timeout). Для отказоустойчивости некоторые драйверы поддерживают **несколько хостов** (primary + standby); при обрыве соединения драйвер пробует следующий хост (нужно смотреть документацию драйвера, например pgjdbc с failover-параметрами).

#### Пошагово: как применить миграции при деплое (общая схема)

1. **В коде** хранятся файлы миграций с версией (например, V001__create_users.sql, V002__add_email.sql) и таблица версий в БД (например, flyway_schema_history или alembic_version) записывает, какие миграции уже применены.
2. **При деплое** (или при старте приложения, если настроено так) запускается инструмент миграций (Alembic, Flyway, Liquibase). Он подключается к БД и смотрит таблицу версий.
3. **Сравнение:** какие файлы миграций есть в коде, но **нет** в таблице версий — те нужно применить.
4. **Применение:** миграции выполняются **по порядку** (по номеру версии). Каждая выполненная миграция записывается в таблицу версий.
5. **Итог:** в dev, test и prod после деплоя применены одни и те же миграции — схема совпадает. Новый разработчик, склонировав репозиторий и запустив миграции, получает актуальную схему.

#### Что будет, если менять схему вручную без миграций

Разработчики и окружения (dev, test, prod) применяют изменения **в разное время** и **в разном порядке** — схема в разных местах **расходится**. Кто-то добавил столбец вручную, кто-то забыл — при деплое приложения возможны ошибки «column does not exist» или «table already exists». **Вывод:** все изменения схемы вести как **миграции** (файлы в коде с версией); при деплое применять только неприменённые миграции — схема везде одинаковая.

#### Типичная ошибка

Меняют схему напрямую в prod (ALTER TABLE вручную) и не добавляют соответствующую миграцию в код. В dev/test схема отстаёт или отличается; при следующем деплое миграции могут попытаться применить уже сделанное изменение и упасть (или наоборот — в prod нет изменений из миграции). **Правильно:** любое изменение схемы — только через миграцию; в prod миграции применяются при деплое (или отдельным шагом), не вручную ALTER.

#### Часто путают

**Миграция** и **бэкап данных**. Миграция — это **изменение структуры** (DDL: CREATE/ALTER/DROP таблиц, индексов, ограничений); версионируется в коде и в таблице версий в БД. **Бэкап** — копия **данных** (и при необходимости схемы) для восстановления; не заменяет миграции. Миграции обеспечивают **одинаковую схему** в разных окружениях; бэкапы — восстановление после сбоя или потери данных.

#### Простыми словами

Миграции — «история изменений схемы в коде»: каждое изменение (новая таблица, новый столбец) — отдельный файл с версией; при деплое применяются только те, которых ещё нет в БД. Так схема в dev, test и prod остаётся согласованной. Alembic — для Python, Flyway/Liquibase — для Java и универсально. Connection string — «адрес и параметры БД»; failover в драйвере — «при падении primary подключаться к другому хосту» (если драйвер это поддерживает).

#### Картинка в голове

**Миграции** — **дневник ремонтов**: «версия 1 — добавили столбец email», «версия 2 — создали таблицу orders». При деплое на сервер смотришь: «какие версии уже применены?» — применяешь только новые. Так на всех окружениях схема одинаковая. **Connection string** — **адрес офиса**: хост, порт, имя базы, пользователь, пароль — «где и как подключаться». **Failover** — «запасной адрес»: если основной хост не ответил, драйвер пробует следующий из списка (реплика, другой узел).

#### Как запомнить

**Миграции** — версионирование DDL; применение по порядку; таблица версий в БД. **Alembic** (Python), **Flyway**, **Liquibase** — инструменты; выбор по языку. **Connection string** — хост, порт, база, пользователь, пароль, параметры. **Failover** в драйвере — несколько хостов при обрыве соединения.

#### Проверь себя

1. Зачем хранить миграции в коде и применять их при деплое?  
<details><summary>Ответ</summary> Чтобы **схема во всех окружениях** (dev, test, prod) и у всех разработчиков была **одинаковой**. Миграции — это версионированная история изменений DDL; при деплое применяются только **неприменённые** миграции (по таблице версий в БД). Так не бывает ситуации «у меня в dev добавлен столбец, а в prod нет».</details>

2. Что такое connection string и какие параметры в нём важны?  
<details><summary>Ответ</summary> **Connection string** — строка подключения к БД: хост, порт, имя базы, пользователь, пароль и параметры (sslmode, connect_timeout, statement_timeout и др.). Пример: postgresql://user:pass@host:5432/dbname?sslmode=require&connect_timeout=5. Важно: **sslmode** для шифрования; **connect_timeout** — сколько ждать соединение; **statement_timeout** — максимум времени выполнения запроса.</details>

**Три пункта, чтобы не забыть (миграции).** 1) **Миграции** — версионирование DDL; применение по порядку; таблица версий в БД. 2) **Alembic** (Python), **Flyway**, **Liquibase** — инструменты; выбор по языку. 3) **Connection string** — хост, порт, база, пользователь, пароль, параметры; **failover** — несколько хостов в драйвере.

#### Запомните

- **Миграции** — версионирование DDL; применение по порядку; таблица версий в БД.
- **Alembic** (Python), **Flyway**, **Liquibase** — инструменты миграций; выбор по языку и предпочтениям.
- **Connection string** — хост, порт, база, пользователь, пароль, параметры (sslmode, timeout). **Failover** в драйвере — несколько хостов и политика переподключения.

**Ещё раз самыми простыми словами.** Миграции — «история изменений схемы в коде»; при деплое применяются только новые — схема везде одинаковая. Connection string — адрес и параметры БД; failover — при падении primary подключаться к другому хосту (если драйвер поддерживает).

**Если прочитал и всё равно не понял.** Главное: **миграция** — это файл с изменением схемы (новая таблица, новый столбец и т.д.) и **версией**. При деплое приложение (или отдельная команда) смотрит таблицу версий в БД и применяет **только те миграции, которых ещё нет**. Так схема в dev, test и prod не расходится. **Connection string** — «адрес БД»: хост, порт, база, пользователь, пароль.
