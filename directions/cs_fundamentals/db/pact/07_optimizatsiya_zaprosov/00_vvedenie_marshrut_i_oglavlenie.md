## Часть VII. Оптимизация запросов

> Цель этой части — **понять, как СУБД выполняет запросы и как их ускорять**: чтение планов (EXPLAIN/EXPLAIN ANALYZE), роль статистики и cost-based оптимизатора, параллельное выполнение и JIT, типичные антипаттерны и настройки (work_mem, effective_cache_size, pg_stat_statements, auto_explain).

После этой части ты сможешь:

- **читать планы выполнения** — EXPLAIN и EXPLAIN ANALYZE; узлы Seq Scan, Index Scan, Bitmap, Nested Loop, Hash/Merge Join, Sort, Aggregate; cost, rows, actual time; BUFFERS;
- **понимать роль статистики** — ANALYZE, гистограммы, pg_stats, селективность, correlation; устаревшая статистика и ALTER TABLE ... SET STATISTICS;
- **понимать cost-based оптимизатор** — оценка стоимости (seq_page_cost, random_page_cost, cpu_tuple_cost), порядок JOIN, выбор плана;
- **разбираться в параллельном выполнении** — Gather, workers; JIT; подплан (subplan), initplan;
- **избегать антипаттернов** — N+1, отсутствие индексов, SELECT *, большие IN, несоответствие типов;
- **настраивать окружение** — work_mem, effective_cache_size, parallel workers; pg_stat_statements, auto_explain.

Материал опирается на **Часть VI (индексы, типы сканирования)** и готовит к **Части VIII (реляционные СУБД, продукты)**.

Материал сгруппирован по **этапам изучения**: планы выполнения (EXPLAIN, узлы плана, cost, BUFFERS) → статистика (ANALYZE, pg_stats, селективность) → cost-based оптимизатор (модель стоимости, порядок JOIN, подзапросы, партиционирование) → параллелизм и JIT → антипаттерны → настройка. В каждом разделе один и тот же шаблон: **цель**, **термины**, **теория и правила**, **пошагово** (с числами, где уместно), **простыми словами**, **картинка в голове**, **«Проверь себя»** (2–3 вопроса с ответами под спойлером) и **«Запомните»**. Иди по оглавлению по этапам — так удобнее планировать изучение и возвращаться к нужному месту. В конце части — справочник, вопросы для самопроверки и резюме для закрепления.

### Краткая шпаргалка по части

| Понятие / конструкция | Суть (что это и когда нужно) |
|----------------------|-----------------------------|
| **EXPLAIN** | Показать план запроса **без выполнения**; cost и rows — оценки планировщика |
| **EXPLAIN ANALYZE** | Выполнить запрос и показать план с **фактическими** временами и числом строк (actual time, actual rows) |
| **Seq Scan** | Полное последовательное сканирование таблицы; читаются все страницы heap подряд |
| **Index Scan** | Поиск по индексу (B-tree и т.д.); спуск по дереву + чтение страниц таблицы по TID |
| **Bitmap Index/Heap Scan** | По индексу собирается битовая карта TID, затем страницы таблицы читаются по порядку; выгодно при множестве строк |
| **Nested Loop Join** | Для каждой строки внешней таблицы — поиск подходящих строк во внутренней (индекс или скан); выгодно при малой внешней таблице |
| **Hash Join** | Хеш-таблица по одной таблице, проход по второй с пробами в хеше; выгодно при больших наборах без порядка |
| **Merge Join** | Обе стороны отсортированы по ключу соединения; слияние двух потоков; выгодно при уже отсортированных данных или индексах |
| **Sort** | Сортировка результата в памяти (work_mem) или на диске; при ORDER BY по ключу индекса может отсутствовать |
| **Aggregate** | Группировка (GROUP BY) и/или агрегатные функции (COUNT, SUM и т.д.) |
| **ANALYZE** | Сбор статистики по таблицам (распределение значений, гистограммы); нужен для точных оценок планировщика |
| **pg_stats** | Системный каталог со статистикой по столбцам: n_distinct, histogram_bounds, correlation, most_common_vals |
| **Селективность** | Доля строк, удовлетворяющих условию; планировщик оценивает по гистограмме и MCV |
| **Cost-based optimizer** | Выбор плана по **оценке стоимости** (cost): сумма затрат на I/O и CPU в условных единицах |
| **seq_page_cost / random_page_cost** | Стоимость одной страницы: последовательное и случайное чтение; для SSD random_page_cost часто уменьшают |
| **Gather** | Узел параллельного плана: объединение результатов нескольких worker-процессов |
| **JIT** | Just-In-Time компиляция выражений и частей запроса; для тяжёлых запросов снижает CPU |
| **N+1** | Антипаттерн: один запрос + N запросов в цикле вместо одного запроса с JOIN или IN |
| **work_mem** | Память на операцию (сортировка, хеш) для одной операции; слишком мало — сброс на диск (медленно) |
| **effective_cache_size** | Оценка кэша ОС+БД для планировщика; влияет на выбор index vs seq scan |
| **pg_stat_statements** | Расширение: накопленная статистика по запросам (время, вызовы, строки); для поиска тяжёлых запросов |
| **auto_explain** | Автоматическое логирование планов медленных запросов в лог |

Используй таблицу как карту: нашёл понятие в первом столбце → во втором видишь суть и типичное применение.

### Маршрут изучения в этой части

1. **EXPLAIN и планы выполнения (§36)**  
   EXPLAIN / EXPLAIN ANALYZE; узлы плана (Seq Scan, Index Scan, Bitmap, Join, Sort, Aggregate); cost, rows, actual time; BUFFERS.

2. **Статистика (§37)**  
   ANALYZE, гистограммы, pg_stats; селективность, correlation; устаревшая статистика; SET STATISTICS.

3. **Cost-based оптимизатор (§38)**  
   Оценка стоимости (seq_page_cost, random_page_cost, cpu_tuple_cost); порядок JOIN; выбор плана.

4. **Параллельное выполнение и JIT (§39)**  
   Gather, workers; параллельные Seq Scan, Hash Join, Aggregate; JIT; subplan, initplan.

5. **Антипаттерны запросов (§40)**  
   N+1, отсутствие индексов, SELECT *, большие IN, несоответствие типов; избыточные подзапросы, длинные транзакции.

6. **Настройка (§41)**  
   work_mem, effective_cache_size, parallel workers; pg_stat_statements, auto_explain; логирование медленных запросов.
