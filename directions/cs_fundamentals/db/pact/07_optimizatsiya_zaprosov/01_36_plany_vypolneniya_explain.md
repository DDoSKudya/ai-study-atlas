## 36. Планы выполнения (EXPLAIN)

### 36.1. EXPLAIN и EXPLAIN ANALYZE

**Цель раздела.**  
Понять разницу между **EXPLAIN** (план без выполнения) и **EXPLAIN ANALYZE** (план с реальным выполнением и фактическими временами и числом строк). После раздела ты будешь знать, когда использовать каждый вариант и как читать поля cost, rows, actual time, actual rows.

**В этом разделе главное (три строки).** 1) **EXPLAIN** — показывает **план** запроса **без выполнения**; в выводе только **оценки** (cost, rows). **EXPLAIN ANALYZE** — **выполняет** запрос и добавляет **фактические** показатели: **actual time** (миллисекунды), **actual rows** (сколько строк прошло на самом деле). 2) Для **поиска узкого места** в медленном запросе нужен именно **EXPLAIN ANALYZE** — по actual time видно, какой узел съел время; по cost и rows без выполнения можно ошибиться. 3) Для запросов с **UPDATE/INSERT/DELETE** при использовании ANALYZE **обязательно** оборачивай в транзакцию с **ROLLBACK**, иначе изменения сохранятся в базе.

```mermaid
flowchart TB
  A[Нужно понять план?] --> B{Нужны фактические времена/строки?}
  B -->|Нет| E[EXPLAIN\nплан + cost/rows (оценки)]
  B -->|Да| EA[EXPLAIN (ANALYZE)\nплан + actual time/rows]
  EA --> Buf{Нужно понять I/O и кэш?}
  Buf -->|Да| EAB[EXPLAIN (ANALYZE, BUFFERS)\nshared hit/read, temp]
  Buf -->|Нет| Done[Достаточно ANALYZE]
  EAB --> Mut{Запрос меняет данные?}
  Mut -->|Да| Tx[BEGIN → EXPLAIN ANALYZE → ROLLBACK]
  Mut -->|Нет| Ok[Можно без ROLLBACK]
```

#### Термины (расшифровка)

- **EXPLAIN** — команда SQL, которая **не выполняет** запрос, а только строит **план выполнения** и выводит его в виде дерева узлов. В выводе отображаются **оценки** планировщика: **cost** (стоимость в условных единицах), **rows** (оценка числа строк). Реального выполнения запроса нет — данные не возвращаются клиенту, изменения не фиксируются. Используется, чтобы **предсказать**, как СУБД будет выполнять запрос, и сравнить альтернативы (например, с разными индексами).
- **EXPLAIN ANALYZE** — то же, что EXPLAIN, но запрос **реально выполняется**. В вывод добавляются **фактические** показатели: **actual time** (реальное время в миллисекундах на шаг и суммарно), **actual rows** (реальное число строк), а при использовании **BUFFERS** — сколько страниц прочитано из кэша (shared hit) и с диска (shared read). Используется для **диагностики** уже выполняющегося запроса: где ушло время, совпадают ли оценки с фактом. **Внимание:** при EXPLAIN ANALYZE запрос выполняется полностью — INSERT/UPDATE/DELETE будут применены; в транзакции можно сделать ROLLBACK после анализа.
- **cost (стоимость)** — абстрактная величина в **условных единицах**. Планировщик считает cost для каждого узла плана (например, 0.00..10.50 — startup cost и total cost). Одна единица примерно соответствует чтению одной страницы последовательно. Cost нужен для **сравнения** планов между собой, а не для перевода в секунды; фактические миллисекунды даёт только EXPLAIN ANALYZE (actual time).
- **rows (оценка строк)** — число строк, которое планировщик **ожидает** на выходе узла. Основано на статистике (ANALYZE, гистограммы). Если оценка сильно расходится с фактом (actual rows), статистика может быть устаревшей или не подходить под предикат — стоит обновить ANALYZE или увеличить STATISTICS по столбцу.

#### Теория и правила (подробно)

- **Когда использовать EXPLAIN:** когда нужно **только посмотреть план** без выполнения: проверить, какой индекс выбран, какой порядок JOIN, есть ли Sort. Удобно для тяжёлых запросов (минуты выполнения) — план получаешь мгновенно. Оценки cost и rows могут отличаться от реальности — они зависят от статистики.
- **Когда использовать EXPLAIN ANALYZE:** когда нужно **измерить** реальное время и объём данных: где узкое место, сколько строк реально прошло через каждый узел, сколько страниц с диска vs из кэша. Запрос выполняется до конца — для изменяющих запросы используй в транзакции с ROLLBACK, если не хочешь сохранять изменения.
- **Формат вывода:** дерево узлов. Каждый узел — одна операция (Seq Scan, Index Scan, Nested Loop, Hash Join, Sort, Aggregate и т.д.). **Отступы** показывают вложенность: дочерние узлы — источники данных для родительского. В конце строки узла — **(cost=... rows=...)** для EXPLAIN и **(actual time=... rows=...)** для EXPLAIN ANALYZE. Суммарное **Execution Time** внизу — общее время выполнения (только при ANALYZE).

#### Другие опции EXPLAIN

Кроме **ANALYZE** и **BUFFERS** у EXPLAIN есть опции, которые полезны для разбора плана и для скриптов:

| Опция | Суть |
|-------|------|
| **FORMAT** | Формат вывода: **TEXT** (по умолчанию, дерево в консоли), **JSON**, **YAML**, **XML**. Например, `EXPLAIN (FORMAT JSON) SELECT ...` — план в JSON для парсинга программами. |
| **VERBOSE** | Добавляет детали: вывод столбцов по узлам, триггеры и т.д. |
| **COSTS** | Показывать cost (по умолчанию on). `COSTS false` — убрать cost/rows из вывода (короче план). |
| **SETTINGS** | Включить в вывод изменённые настройки сессии, влияющие на план (work_mem и т.д.). |
| **TIMING** | При ANALYZE показывать actual time по узлам (по умолчанию on при ANALYZE). `TIMING false` — только rows, без времени. |
| **SUMMARY** | В конце выводить итог (Execution Time, планирование); по умолчанию on при ANALYZE. |

**Пример:** `EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT) SELECT ...` — то же, что обычно. `EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON) SELECT ...` — план в JSON с actual time и BUFFERS для автоматической обработки. Для ручной диагностики медленных запросов чаще всего достаточно **ANALYZE** и **BUFFERS**.

#### Пошагово: как снять план (практика, с числами)

**Шаг 1. Только план, без выполнения.**  
Набери в клиенте: `EXPLAIN SELECT * FROM orders WHERE user_id = 5;` и выполни. Запрос **не выполнится** — СУБД только построит план и выведет его. В выводе увидишь дерево узлов; у каждого узла в конце строки будет что-то вроде **(cost=0.00..15.50 rows=100 width=40)**. Это **оценки** планировщика: cost — условная стоимость, rows — ожидаемое число строк, width — средний размер строки в байтах. Реального времени выполнения и реального числа строк **нет** — их даёт только EXPLAIN ANALYZE.

**Шаг 2. План с реальным выполнением.**  
Набери: `EXPLAIN (ANALYZE, BUFFERS) SELECT * FROM orders WHERE user_id = 5;` и выполни. На этот раз запрос **выполнится до конца**. В выводе у каждого узла появятся **actual time=0.05..1.20 rows=100 loops=1** — это **фактические** миллисекунды (actual time), **фактическое** число строк (rows) и сколько раз узел выполнялся (loops). Внизу плана будет **Execution Time: 1.25 ms** — общее время выполнения. Если указан BUFFERS, увидишь **Buffers: shared hit=5 read=2** — сколько страниц прочитано из кэша (hit) и с диска (read). Для запросов с изменением данных (UPDATE, INSERT, DELETE) **изменения сохранятся**. Чтобы не сохранять, оберни в транзакцию: `BEGIN; EXPLAIN ANALYZE UPDATE orders SET status = 'done' WHERE id = 1; ROLLBACK;` — после ROLLBACK изменения откатятся.

**Шаг 3. Интерпретация: где узкое место.**  
Прокрути вывод EXPLAIN ANALYZE сверху вниз. У каждого узла смотри на **actual time** — это время в миллисекундах (первое число — до первой строки, второе — полное время узла). Узел с **наибольшим** actual time — чаще всего узкое место. Например: Seq Scan имеет actual time=0.05..850.00, а Nested Loop ниже — actual time=0.10..5.00. Значит, основное время ушло на Seq Scan (850 ms). Дополнительно сравни **rows** (оценка) и **actual rows** (факт). Если rows=10, а actual rows=50000 — оценка сильно занижена; планировщик строил план в расчёте на 10 строк, а получил 50 000. Это признак **устаревшей статистики** — выполни `ANALYZE orders;` (или нужная таблица) и повтори запрос.

#### Что будет, если использовать только EXPLAIN для диагностики медленного запроса

Если запрос выполняется 10 секунд и ты делаешь только **EXPLAIN** (без ANALYZE), ты увидишь **план** и **оценки** (cost, rows), но **не узнаешь**, где реально ушло время. Оценки могут быть неверными (устаревшая статистика): планировщик думал, что будет 100 строк, а на деле 100 000 — и узкое место окажется совсем в другом узле, чем кажется по cost. **Итог:** для диагностики медленного запроса **всегда** используй **EXPLAIN ANALYZE** (и при необходимости BUFFERS), чтобы видеть **actual time** и **actual rows** — где время ушло по факту и совпадают ли оценки с реальностью.

#### Типичная ошибка

Делают **EXPLAIN ANALYZE** для запроса с **UPDATE** или **DELETE**, забывают обернуть в транзакцию с ROLLBACK — и **изменения сохраняются** в базе. На продакшене это может привести к случайному обновлению или удалению данных. **Правило:** для любого изменяющего запроса (INSERT, UPDATE, DELETE) при анализе плана всегда пиши: `BEGIN; EXPLAIN ANALYZE UPDATE ... ; ROLLBACK;` — так ты увидишь план и фактические времена, но изменения откатятся и в базе ничего не изменится.

#### Часто путают

**cost в плане** и **реальное время выполнения**. cost — это **условные единицы** («баллы»), которые планировщик использует только для **сравнения** вариантов плана между собой; в секунды или миллисекунды их **переводить нельзя**. Реальное время (сколько запрос действительно выполнялся) показывают только **actual time** и **Execution Time** в **EXPLAIN ANALYZE** — там миллисекунды. Запомни: cost ≠ время; время смотри в actual time.

#### Простыми словами

**EXPLAIN** — «покажи, как ты будешь делать запрос, но не делай его». Получаешь план и оценки (cost, rows). **EXPLAIN ANALYZE** — «сделай запрос и покажи, как делал и сколько реально заняло». Нужен для поиска узких мест по факту: actual time покажет, какой узел съел время, actual rows — сколько строк прошло на самом деле. **cost** — условная «цена» операции для планировщика (не секунды); **rows** — ожидаемое число строк; **actual time** и **actual rows** — что получилось на самом деле после выполнения.

#### Картинка в голове

EXPLAIN — как **чертёж маршрута** без поездки: видишь, какие дороги выбраны и примерную длину (cost, rows). EXPLAIN ANALYZE — как **поездка с секундомером**: проехал, замерил время на каждом участке (actual time) и реальное расстояние (actual rows). cost/rows — цифры на чертеже; actual time/rows — что показал секундомер и счётчик после поездки. Чтобы понять, где реально «застряло», нужна поездка с секундомером (ANALYZE), а не только чертёж.

#### Как запомнить

**EXPLAIN** = план без выполнения, только оценки (cost, rows). **EXPLAIN ANALYZE** = выполнить запрос и показать факт (actual time, actual rows). Для диагностики медленных запросов — всегда ANALYZE. Для изменяющих запросов — в транзакции с ROLLBACK.

#### Проверь себя (36.1)

1. Чем EXPLAIN отличается от EXPLAIN ANALYZE?  
<details><summary>Ответ</summary> **EXPLAIN** только строит план и **не выполняет** запрос; в выводе — cost и rows (оценки). **EXPLAIN ANALYZE** **выполняет** запрос и добавляет в вывод **actual time** и **actual rows** (фактические время и число строк). Для диагностики медленных запросов нужен EXPLAIN ANALYZE; для быстрого просмотра плана без выполнения — EXPLAIN.</details>

2. Что такое cost в плане и можно ли по нему перевести время в секунды?  
<details><summary>Ответ</summary> **cost** — это **условные единицы стоимости** (примерно 1 = одно последовательное чтение страницы). Планировщик использует cost для **сравнения** вариантов плана между собой. В секунды напрямую переводить **нельзя** — это не время, а абстрактная мера. Реальное время даёт **actual time** в EXPLAIN ANALYZE.</details>

3. Запрос с UPDATE внутри EXPLAIN ANALYZE — будут ли изменения сохранены?  
<details><summary>Ответ</summary> **Да.** EXPLAIN ANALYZE **выполняет** запрос до конца, включая UPDATE/INSERT/DELETE. Изменения будут зафиксированы (если не в явной транзакции с ROLLBACK). Чтобы проанализировать план изменяющего запроса без сохранения изменений, нужно: `BEGIN; EXPLAIN ANALYZE UPDATE ...; ROLLBACK;`</details>

4. Запрос выполняется 5 секунд. Достаточно ли сделать только EXPLAIN, чтобы найти узкое место?  
<details><summary>Ответ</summary> **Нет.** Только **EXPLAIN** покажет план и **оценки** (cost, rows), но не **фактическое** время по узлам (actual time) и не фактическое число строк (actual rows). Оценки могут быть неверными — узкое место по cost может не совпадать с узким местом по факту. Чтобы **найти**, где реально ушло время, нужен **EXPLAIN ANALYZE** — он выполнит запрос и выведет actual time и actual rows по каждому узлу. По ним видно, какой узел съел больше всего времени.</details>

**Три пункта, чтобы не забыть (EXPLAIN и EXPLAIN ANALYZE).** 1) **EXPLAIN** — план **без выполнения**; в выводе только cost и rows (оценки). **EXPLAIN ANALYZE** — запрос **выполняется**; в выводе добавляются actual time, actual rows (факт). Для диагностики медленных запросов нужен ANALYZE. 2) **cost** — условные единицы, **не секунды**; реальное время — **actual time** в EXPLAIN ANALYZE. 3) Для запросов с UPDATE/INSERT/DELETE при ANALYZE **обязательно** оборачивай в транзакцию с ROLLBACK, иначе изменения сохранятся в базе.

#### Запомните

- **EXPLAIN** — план без выполнения; cost и rows — оценки. **EXPLAIN ANALYZE** — выполнение + фактические actual time, actual rows.
- **cost** — условные единицы для сравнения планов, не секунды. Реальное время — **actual time** в EXPLAIN ANALYZE.
- Для изменяющих запросов с ANALYZE используй транзакцию с ROLLBACK, если не хочешь сохранять изменения.

**Ещё раз самыми простыми словами:** EXPLAIN — «покажи план, но не выполняй»; получаешь только оценки. EXPLAIN ANALYZE — «выполни и покажи, сколько по факту заняло»; по нему ищешь узкое место. cost — это не секунды, а «баллы» для сравнения планов; секунды смотри в actual time. И никогда не забывай ROLLBACK после EXPLAIN ANALYZE для UPDATE/DELETE/INSERT.

**Одна фраза (если забыл всё):** EXPLAIN — план без выполнения; EXPLAIN ANALYZE — выполнил и показал, где время (actual time). Для диагностики медленных запросов — всегда ANALYZE; для UPDATE/DELETE/INSERT — в транзакции с ROLLBACK.

**Если прочитал и всё равно не понял.** Главное: **EXPLAIN** — ты просишь базу «расскажи, как будешь делать запрос», но **не делай его**; в ответе только оценки (cost, rows). **EXPLAIN ANALYZE** — ты просишь «сделай запрос и расскажи, как делал и сколько по факту заняло»; в ответе появятся **actual time** и **actual rows** — по ним видно, **где** ушло время. Для медленного запроса нужен именно **ANALYZE**, иначе не узнаешь, какой узел тормозит. **cost** в плане — это не секунды, а «баллы» для сравнения планов; секунды смотри в **actual time**. Если запрос меняет данные (UPDATE/INSERT/DELETE), при ANALYZE они **сохранятся** — оборачивай в **BEGIN; ... ROLLBACK;** чтобы откатить. ### 36.2. Узлы сканирования

**Цель раздела.**  
Научиться различать в плане узлы **Seq Scan**, **Index Scan**, **Index Only Scan** и **Bitmap Index Scan / Bitmap Heap Scan**. Понимать, что каждый из них означает и когда планировщик их выбирает. После раздела ты будешь по названию узла понимать, как читаются данные таблицы.

**В этом разделе главное (три строки).** 1) **Seq Scan** — читаем **всю таблицу** подряд (все страницы); **Index Scan** — по индексу находим адреса строк (TID) и читаем **только нужные страницы таблицы**; **Index Only Scan** — читаем **только индекс**, к таблице не обращаемся (если все нужные столбцы в индексе). 2) **Bitmap Scan** — по индексу строим **карту** «какие страницы таблицы нужны», потом читаем эти страницы **по порядку** (выгодно при большом числе строк по индексу). 3) По **названию узла** в плане сразу видно способ доступа; Seq Scan по большой таблице с условием в WHERE — повод проверить, есть ли индекс и почему он не выбран.

```mermaid
flowchart LR
  subgraph S[Seq Scan]
    S1[Читаем все страницы heap подряд] --> S2[Фильтр WHERE на каждой строке]
  end

  subgraph I[Index Scan]
    I1[Индекс: key -> TID] --> I2[Heap fetch по TID (случайные чтения)]
  end

  subgraph IO[Index Only Scan]
    IO1[Читаем только индекс] --> IO2[Heap fetch = 0\n(если visibility map позволяет)]
  end

  subgraph Bm[Bitmap Scan]
    B1[Индекс: собрать bitmap страниц/TID] --> B2[Heap scan страниц по порядку]
  end
```

#### Термины (расшифровка)

- **Seq Scan (Sequential Scan)** — **полное последовательное сканирование** таблицы. СУБД читает **все страницы** heap подряд (блок за блоком), перебирает каждую строку и применяет условие WHERE. Не использует индексы. Выбирается, когда оценка стоимости полного скана **ниже**, чем индексный доступ (маленькая таблица, низкая селективность условия или нет подходящего индекса).
- **Index Scan** — доступ по **индексу**: спуск по структуре индекса (например, B-tree) по условию, получение TID (указателей на строки), затем чтение **страниц таблицы** по этим TID. Два источника I/O: страницы индекса и страницы таблицы. Выбирается, когда условие по ключу индекса и селективность достаточно высокая — читать по индексу дешевле, чем всю таблицу.
- **Index Only Scan** — чтение **только из индекса**, без обращения к таблице (heap). Возможно, если **все** столбцы запроса присутствуют в индексе (покрывающий индекс, в т.ч. INCLUDE) и для затронутых страниц таблицы в **visibility map** помечено, что все строки видимы. Иначе СУБД всё равно обращается к heap для проверки видимости — в плане будет **Heap Fetches**.
- **Bitmap Index Scan** — по индексу собирается **битовая карта** TID (какие страницы/строки подходят), без немедленного чтения таблицы. Затем узел **Bitmap Heap Scan** читает **страницы таблицы по порядку** по этой карте (снижая случайный I/O). Выгодно, когда по индексу отбирается **много строк** (не одна и не вся таблица) — порядок чтения страниц таблицы оптимизируется.

#### Теория и правила (подробно)

- **Seq Scan:** в плане — `Seq Scan on таблица`. Часто на больших таблицах без подходящего индекса или когда планировщик считает, что прочитать всё дешевле. При росте таблицы cost Seq Scan растёт **линейно** с числом страниц: в два раза больше таблица — в два раза выше cost. На таблице в 1 млн строк (например, 40 000 страниц по 8 KB) Seq Scan прочитает все 40 000 страниц подряд.
- **Index Scan:** в плане — `Index Scan using индекс on таблица` (или Index Scan using ... with Index Cond). Подразумевает условие по ключу индекса. Подходит для точечных и диапазонных запросов с хорошей селективностью. Читает **страницы индекса** (3–4 для B-tree на миллион записей) и затем **страницы таблицы** по TID — по одной странице на каждую найденную строку (или меньше, если несколько строк на одной странице).
- **Index Only Scan:** в плане — `Index Only Scan using индекс on таблица`. Если есть **Heap Fetches: N** — часть данных всё же читалась из таблицы (проверка видимости). Читает **только страницы индекса** — к таблице (heap) не обращается, если все нужные столбцы в индексе и visibility map помечает страницы как «все видимы».
- **Bitmap:** в плане обычно два узла: **Bitmap Index Scan** (по индексу строим карту TID) и **Bitmap Heap Scan** (читаем страницы таблицы **по порядку** по этой карте). Между ними может быть **Recheck Cond** — условие перепроверяется по строкам, т.к. в битовой карте помечаются **страницы/блоки**, а не отдельные строки. Выгодно, когда по индексу отбирается **много** строк (сотни, тысячи) — тогда чтение страниц таблицы по порядку дешевле, чем вразнобой по TID (Index Scan давал бы много случайных чтений).

#### Пошагово: что происходит при каждом типе сканирования (с числами)

**Seq Scan.** Запрос: `SELECT * FROM orders WHERE user_id = 5`. Таблица orders: 100 000 строк, 4 000 страниц по 8 KB.

1. Планировщик выбирает **Seq Scan** по orders.
2. СУБД читает **страницу 1** таблицы (8 KB), перебирает все строки на ней (~25 строк), для каждой проверяет `user_id = 5`. Подходящие добавляет в результат.
3. Читает **страницу 2**, перебирает строки, проверяет условие. Так до **последней** страницы (4 000).
4. **Итог:** прочитано **4 000 страниц** (вся таблица). Даже если нужные строки (user_id = 5) были на первых 100 страницах, СУБД без индекса не знает, где они — просматривает всё.

**Index Scan.** Тот же запрос, есть индекс по user_id.

1. СУБД спускается по B-tree индекса: **корень** (1 страница) → **внутренний узел** (1 страница) → **лист** (1 страница). В листе находит ключ user_id = 5 и читает **TID** (указатели на строки). Допустим, 50 строк с user_id = 5 лежат на 20 разных страницах таблицы.
2. По каждому TID читает **страницу таблицы** (или группирует TID по страницам и читает каждую страницу один раз). Прочитано **3 страницы индекса** + **20 страниц таблицы** = 23 страницы вместо 4 000.
3. **Итог:** в десятки раз меньше прочитанных страниц — запрос быстрее.

**Index Only Scan.** Запрос: `SELECT user_id, order_date FROM orders WHERE user_id = 5`. Есть **покрывающий** индекс: (user_id) INCLUDE (order_date). Visibility map для затронутых страниц таблицы помечает «все видимы».

1. СУБД читает только **страницы индекса** (3–4 страницы), в листах индекса уже есть и user_id, и order_date (INCLUDE). К **таблице не обращается** — Heap Fetches: 0.
2. **Итог:** минимум I/O — только индексы, без чтения heap.

**Bitmap Index Scan + Bitmap Heap Scan.** Запрос: `SELECT * FROM orders WHERE status = 'active'`. По индексу на status отбирается 30 000 строк (30% таблицы).

1. **Bitmap Index Scan:** по индексу обходим листья, собираем **битовую карту** — какие **страницы** таблицы содержат подходящие строки. Карта в памяти.
2. **Bitmap Heap Scan:** читаем **страницы таблицы по возрастанию номера** (1, 5, 7, 12, …), а не вразнобой по TID. Так чтение ближе к **последовательному** — меньше случайный I/O, быстрее на диске.
3. **Итог:** при большом числе строк (тысячи) Bitmap даёт выигрыш по сравнению с Index Scan (много случайных чтений по TID) и по сравнению с Seq Scan (читаем не всю таблицу, а только отмеченные страницы).

#### Что будет, если перепутать узлы в плане

Если в плане написано **Index Scan**, а ты думаешь, что это «полный скан» — будешь искать причину не там. **Index Scan** уже означает, что индекс **используется**; узкое место может быть в другом узле (например, в Nested Loop выше). Если видишь **Seq Scan** по большой таблице — первая мысль: есть ли подходящий индекс и почему планировщик его не выбрал (малая селективность? устаревшая статистика? неверный тип в условии?). По названию узла ты сразу понимаешь **способ доступа** — не путай их при разборе плана.

#### Типичная ошибка

Путать **Seq Scan** и **Index Scan** при разборе: смотрят на «медленный узел» и не замечают, что это уже Index Scan (индекс используется). Узкое место тогда может быть выше (например, Nested Loop с огромной внешней таблицей). И наоборот: видишь Seq Scan по большой таблице — не спеши «добавить индекс»; проверь, есть ли индекс и почему он не выбран (селективность, статистика, тип в условии).

#### Пример фрагмента плана (PostgreSQL)

```
Seq Scan on orders  (cost=0.00..100.00 rows=500 width=40)
  Filter: (user_id = 5)
```

Полное сканирование таблицы orders; фильтр по user_id; оценка 500 строк.

```
Index Scan using idx_orders_user_id on orders  (cost=0.28..8.30 rows=1 width=40)
  Index Cond: (user_id = 5)
```

Доступ по индексу idx_orders_user_id по условию user_id = 5; оценка 1 строка.

```
Index Only Scan using idx_orders_covering on orders  (cost=0.28..4.29 rows=1 width=20)
  Index Cond: (user_id = 5)
  Heap Fetches: 0
```

Чтение только из индекса; Heap Fetches: 0 — к таблице не обращались.

#### Простыми словами

**Seq Scan** — «прочитали таблицу от начала до конца». **Index Scan** — «по индексу нашли нужные места и прочитали только их в таблице». **Index Only Scan** — «всё нужное взяли из индекса, к таблице не ходили». **Bitmap Scan** — «по индексу собрали список страниц, потом прочитали эти страницы таблицы по порядку». По названию узла в плане ты сразу видишь способ доступа к данным.

#### Картинка в голове

**Seq Scan** — листаешь **всю книгу подряд** с первой до последней страницы; ищешь все упоминания темы — другого способа нет. **Index Scan** — открываешь **оглавление** (индекс), находишь номера страниц по теме и открываешь **только эти страницы** книги (таблицы); книгу целиком не листаешь. **Index Only Scan** — в **оглавлении** уже есть нужный тебе текст (покрывающий индекс: все столбцы запроса в индексе); **книгу вообще не открываешь** — всё взял из оглавления. **Bitmap Scan** — по оглавлению **выписал список номеров страниц**, отсортировал их (1, 5, 7, 12…) и открываешь страницы **по порядку номера**, а не вразнобой; так быстрее, чем прыгать 50 раз в разные места книги.

#### Как запомнить

**Seq Scan** = вся таблица подряд. **Index Scan** = индекс + чтение таблицы по TID. **Index Only Scan** = только индекс, таблицу не читаем (покрывающий индекс + visibility map). **Bitmap Scan** = индекс → карта страниц → чтение страниц таблицы по порядку (выгодно при многих строках).

#### Проверь себя (36.2)

1. В плане узел «Seq Scan on orders». Что это значит?  
<details><summary>Ответ</summary> Это **полное последовательное сканирование** таблицы orders: СУБД читает все страницы таблицы подряд и для каждой строки проверяет условие WHERE. Индекс не используется. Обычно так бывает при отсутствии подходящего индекса или когда планировщик посчитал полный скан дешевле индексного доступа (маленькая таблица или низкая селективность).</details>

2. Чем Index Only Scan отличается от Index Scan?  
<details><summary>Ответ</summary> **Index Scan** читает данные из индекса (ключи + TID), а затем по TID обращается к **таблице (heap)** за остальными столбцами строки. **Index Only Scan** читает **только индекс** — все нужные для запроса столбцы есть в индексе (покрывающий индекс), и для затронутых страниц таблицы в visibility map помечено «все видимы», поэтому к таблице не обращаются. Index Only Scan даёт меньше I/O.</details>

3. Зачем нужен Bitmap Heap Scan после Bitmap Index Scan?  
<details><summary>Ответ</summary> **Bitmap Index Scan** только строит **битовую карту** TID (какие строки/страницы подходят по индексу), но не читает таблицу. **Bitmap Heap Scan** по этой карте **читает страницы таблицы**, причём обычно в порядке возрастания номера страницы — так уменьшается случайный I/O. То есть сначала собираем «список адресов», потом читаем таблицу упорядоченно, а не вразнобой по TID.</details>

4. В плане «Index Only Scan», но «Heap Fetches: 500». Что это значит?  
<details><summary>Ответ</summary> **Heap Fetches** — число обращений к **таблице (heap)** для проверки видимости строк. Index Only Scan возможен, когда все нужные столбцы в индексе и для затронутых страниц в **visibility map** помечено «все строки видимы». Если для части страниц это не так, СУБД всё равно идёт в heap и проверяет, видима ли строка текущей транзакции. **Heap Fetches: 500** значит, что 500 раз пришлось прочитать строку из таблицы — полного «только индекса» не получилось. При 0 — к таблице не обращались вообще.</details>

**Три пункта, чтобы не забыть (узлы сканирования).** 1) **Seq Scan** — читаем **всю таблицу** подряд; **Index Scan** — по индексу находим TID и читаем **только нужные страницы таблицы**; **Index Only Scan** — читаем **только индекс**, к таблице не ходим (покрывающий индекс + visibility map). 2) **Bitmap Index Scan** строит карту TID по индексу; **Bitmap Heap Scan** читает страницы таблицы **по порядку** по этой карте — выгодно при **многом** числе строк. 3) По названию узла в плане сразу видно способ доступа; самый дешёвый по I/O — Index Only Scan (ноль обращений к heap).

#### Запомните

- **Seq Scan** — полное сканирование таблицы подряд; **Index Scan** — доступ по индексу + чтение таблицы по TID; **Index Only Scan** — только индекс (покрывающий + visibility map); **Bitmap Index/Heap Scan** — карта TID по индексу, затем чтение страниц таблицы по порядку.
- По названию узла в плане видно способ доступа к данным; самый дешёвый по I/O при хорошей селективности — Index Only Scan (без обращений к heap).

**Ещё раз самыми простыми словами:** Seq Scan — «прочитали всю таблицу от начала до конца». Index Scan — «по индексу нашли адреса строк и прочитали только эти страницы таблицы». Index Only Scan — «всё нужное было в индексе, к таблице не ходили». Bitmap Scan — «по индексу собрали список страниц и прочитали их по порядку». Запомни по одному образу на каждый узел — в плане сразу будет ясно, как читаются данные.

**Одна фраза (если забыл всё):** Seq Scan = вся таблица подряд; Index Scan = по индексу + чтение таблицы по TID; Index Only Scan = только индекс, к таблице не ходим; Bitmap = карта TID по индексу, потом читаем страницы таблицы по порядку. По названию узла в плане видно способ доступа.

**Если прочитал и всё равно не понял.** Главное — по **названию узла** в плане ты сразу видишь, **как** читаются данные. **Seq Scan** — читаем **всю таблицу** подряд, страница за страницей; индекс не используется. **Index Scan** — по индексу нашли адреса нужных строк (TID) и прочитали **только эти страницы** таблицы; таблицу целиком не читаем. **Index Only Scan** — всё нужное уже в индексе, к таблице **вообще не обращаемся** (если visibility позволяет). **Bitmap** — по индексу собрали «список страниц», потом читаем эти страницы **по порядку номера**, а не вразнобой. Если в плане по большой таблице с условием в WHERE стоит **Seq Scan** — первый вопрос: есть ли индекс по этому столбцу и почему он не выбран (статистика, тип в условии). Перечитай **«Пошагово: что происходит при каждом типе сканирования»** (с числами страниц) и **«Картинка в голове»** (книга и оглавление).

### 36.3. Узлы соединения

**Цель раздела.**  
Научиться различать в плане **Nested Loop Join**, **Hash Join** и **Merge Join** и понимать, когда планировщик выбирает каждый из них. После раздела ты будешь по названию узла JOIN понимать, как соединяются две таблицы и где может быть узкое место.

**В этом разделе главное (три строки).** 1) **Nested Loop** — для **каждой** строки первой (внешней) таблицы ищем подходящие во второй (внутренней); выгодно, когда внешняя **маленькая** и во второй ищем **по индексу**; при большой внешней и полном скане внутренней — очень медленно. 2) **Hash Join** — одну таблицу (обычно меньшую) строим в **хеш в памяти** (work_mem), вторую проходим и смотрим в хеш; выгодно при **больших** наборах без порядка; при нехватке памяти — сброс на диск («Disk»). 3) **Merge Join** — обе стороны **отсортированы** по ключу соединения; идём двумя указателями и совпадающее объединяем; выгодно, когда порядок уже есть (индексы) или нужен для ORDER BY.

#### Термины (расшифровка)

- **Nested Loop Join** — соединение «вложенным циклом»: для **каждой** строки **внешней** (левой) таблицы выполняется поиск подходящих строк во **внутренней** (правой) таблице. Внутренняя таблица может читаться по индексу (Index Scan) или сканироваться (Seq Scan). Эффективно, когда **внешняя** таблица **мала** (мало строк) и для каждой строки внутренняя таблица читается по индексу — тогда общее число обращений пропорционально размеру внешней таблицы.
- **Hash Join** — соединение по **хеш-таблице**: одна таблица (обычно меньшая) полностью читается и строится **в памяти** хеш по ключу соединения; затем вторая таблица сканируется и для каждой строки выполняется **проба** в хеше по ключу. Эффективно для **больших** наборов без естественного порядка по ключу; требует памяти (work_mem). Если памяти не хватает, часть хеша сбрасывается на диск — план может содержать «Disk: …» и время растёт.
- **Merge Join** — соединение **слиянием**: обе стороны **отсортированы** по ключу соединения (или приходят в порядке по индексу); два указателя идут по обоим потокам и совпадающие ключи объединяются. Эффективно, когда данные уже отсортированы (индекс по ключу JOIN) или когда нужна сортировка одной или обеих сторон — тогда одна сортировка может обслуживать и JOIN, и ORDER BY.

#### Теория и правила (подробно)

- **Nested Loop:** cost примерно = cost(внешняя) + N × cost(внутренняя на одну строку), где N — число строк внешней таблицы. Если N велико и внутренняя таблица каждый раз сканируется (Seq Scan), cost очень большой — планировщик предпочитает Hash или Merge. Nested Loop выгоден при **малой внешней** таблице и **индексном** доступе по внутренней.
- **Hash Join:** cost включает построение хеша (однократно по одной таблице) и проход по второй таблице с пробами. Память — work_mem; при нехватке — spill на диск. Обычно выбирается, когда нет порядка по ключу и наборы средние или большие.
- **Merge Join:** требует **сортировки** одной или обеих сторон (узлы Sort), если порядок не даёт индекс. Выгоден, когда порядок уже есть (индексы по ключу JOIN) или когда результат нужен отсортированным по тому же ключу.

#### Пошагово: как работает каждый тип JOIN (с числами)

**Nested Loop Join.** Запрос: `SELECT * FROM users u JOIN orders o ON o.user_id = u.id WHERE u.region = 'EU'`. Внешняя таблица (users после фильтра region = 'EU'): **100 строк**. Внутренняя (orders): 1 млн строк, есть индекс по user_id.

1. Читаем **внешнюю** таблицу (users с фильтром region = 'EU') — получаем 100 строк.
2. Для **каждой** из 100 строк (например, user_id = 5) ищем во **внутренней** таблице (orders) по индексу: `WHERE user_id = 5`. Индекс возвращает, допустим, 50 TID — читаем 50 строк из orders (несколько страниц).
3. Повторяем для всех 100 пользователей: 100 × (поиск по индексу + чтение страниц orders). **Итог:** 100 обращений к индексу orders и чтение только тех страниц orders, где лежат заказы этих 100 пользователей. Быстро, потому что внешняя таблица **мала** (100 строк) и внутренняя читается **по индексу**.

Если бы внешняя таблица была **100 000 строк** и внутренняя каждый раз сканировалась **полностью** (Seq Scan), было бы 100 000 × (полный скан 1 млн строк) — катастрофически медленно. Поэтому планировщик выбирает Nested Loop только при **малой внешней** и **индексном** доступе по внутренней.

**Hash Join.** Запрос: `SELECT * FROM orders o JOIN products p ON o.product_id = p.id`. Обе таблицы большие: orders — 500 000 строк, products — 10 000 строк. Нет порядка по product_id.

1. Читаем **меньшую** таблицу (products — 10 000 строк) **целиком** и строим в памяти **хеш-таблицу** по ключу соединения (product_id). Каждый product_id → указатель на строку (или список строк). Память — work_mem (например, 64 MB).
2. Читаем **вторую** таблицу (orders — 500 000 строк) **подряд** (Seq Scan). Для каждой строки orders берём product_id, делаем **проба** в хеше: есть ли такой product_id? Если да — объединяем строки и выдаём в результат.
3. **Итог:** один проход по products (построение хеша) + один проход по orders (проба в хеше). Память нужна на хеш (10 000 строк products). Если products не помещается в work_mem, часть хеша сбрасывается на диск — в плане появится «Disk: …» и запрос замедлится.

**Merge Join.** Запрос: `SELECT * FROM orders o JOIN orders o2 ON o.customer_id = o2.customer_id ORDER BY o.customer_id`. Обе стороны уже **отсортированы** по customer_id (или есть индекс по customer_id, отдающий строки в порядке).

1. Читаем **левую** сторону (orders) в порядке customer_id; читаем **правую** сторону (orders) в порядке customer_id. Два «курсора» идут по обоим потокам **параллельно**.
2. Сравниваем текущие ключи: если customer_id слева = customer_id справа — объединяем строки и выдаём; сдвигаем тот курсор, у которого ключ меньше. Если ключи равны — выдаём все совпадения (может быть несколько строк с одним customer_id с каждой стороны).
3. **Итог:** один проход по обоим потокам, без построения хеша и без N обращений к внутренней таблице. Выгодно, когда данные **уже отсортированы** по ключу JOIN (индексы) или когда сортировка всё равно нужна для ORDER BY.

#### Когда какой (кратко)

| Условия | Типичный выбор |
|---------|-----------------|
| Малая внешняя таблица, индекс по ключу JOIN на внутренней | Nested Loop |
| Большие наборы, нет порядка по ключу | Hash Join |
| Данные уже отсортированы по ключу JOIN (индексы) или нужен ORDER BY по ключу | Merge Join |

```mermaid
flowchart TB
  Q[JOIN двух таблиц] --> NL{Внешняя маленькая? \n+ внутри есть индекс?}
  NL -->|Да| N1[Nested Loop\nN раз искать по индексу]
  NL -->|Нет| Hm{Есть порядок по ключу\nили нужен ORDER BY?}
  Hm -->|Да| M1[Merge Join\nслияние двух отсортированных потоков]
  Hm -->|Нет| H1[Hash Join\nпостроить hash (work_mem)\nи пробить вторую сторону]
```

#### Что будет, если выбрать не тот тип JOIN

Если планировщик **ошибочно** выбрал Nested Loop при **большой** внешней таблице (из-за устаревшей статистики: оценил 100 строк, а получилось 100 000) — будет 100 000 обращений к внутренней таблице. Если внутренняя каждый раз сканируется (Seq Scan), запрос может выполняться минуты. **Решение:** обновить статистику (ANALYZE); проверить, есть ли индекс по ключу JOIN на внутренней таблице. Если планировщик выбрал Hash Join, но work_mem мал — хеш не помещается в память, сброс на диск («Disk: …») — запрос замедляется. **Решение:** увеличить work_mem для этого запроса или сессии.

#### Простыми словами

**Nested Loop** — «для каждой строки первой таблицы ищем подходящие во второй» (часто по индексу). Выгодно, когда первая таблица **маленькая** и во второй ищем по индексу. **Hash Join** — «строим справочник по одной таблице в памяти (хеш), потом проходим по второй и смотрим в справочник». Выгодно при **больших** наборах без порядка. **Merge Join** — «обе таблицы отсортированы по ключу; идём двумя указателями и совпадающее объединяем». Выгодно, когда порядок уже есть (индексы) или нужен для ORDER BY. По названию узла в плане видно стратегию соединения.

#### Картинка в голове

**Nested Loop** — для каждого человека из списка А ищешь его в списке Б по номеру (быстро, если список Б с индексом). **Hash Join** — выписал всех из списка Б в блокнот по номерам (хеш), потом проходишь список А и смотришь в блокнот. **Merge Join** — оба списка по номерам подряд; идёшь по обоим и совпадающие выписываешь.

#### Проверь себя (36.3)

1. Когда планировщик обычно выбирает Nested Loop Join?  
<details><summary>Ответ</summary> Когда **внешняя** (первая) таблица **мала** по числу строк и для каждой её строки внутренняя таблица читается **по индексу** по ключу соединения. Тогда общая стоимость = стоимость чтения внешней таблицы + (число строк внешней) × (стоимость одного поиска по индексу во внутренней). При малой внешней таблице это дёшево. При большой внешней и полном скане внутренней Nested Loop становится очень дорогим — планировщик выберет Hash или Merge.</details>

2. Что такое Hash Join и зачем ему work_mem?  
<details><summary>Ответ</summary> **Hash Join** — одна таблица (обычно меньшая) полностью читается и по ключу соединения строится **хеш-таблица в памяти**; вторая таблица сканируется и для каждой строки выполняется **проба** в хеше. **work_mem** ограничивает память на эту хеш-таблицу. Если данных больше, чем помещается в work_mem, часть хеша сбрасывается на диск (spill) — производительность падает. Увеличение work_mem может ускорить Hash Join, если видно «Disk: …» в плане.</details>

3. Когда выгоден Merge Join?  
<details><summary>Ответ</summary> Когда обе стороны **уже отсортированы** по ключу соединения (например, по индексу) или когда сортировка всё равно нужна для ORDER BY по тому же ключу. Тогда Merge Join соединяет два отсортированных потока за один проход без дополнительной сортировки или с одной сортировкой вместо двух. При больших наборах и наличии порядка по ключу Merge Join часто дешевле Nested Loop и может быть сопоставим с Hash Join.</details>

4. В плане Nested Loop, внешняя таблица — 50 000 строк, внутренняя каждый раз Seq Scan. Почему так могло получиться и что делать?  
<details><summary>Ответ</summary> Планировщик **оценил** число строк внешней таблицы **ниже**, чем на самом деле (например, 100 строк из-за устаревшей статистики), и решил, что Nested Loop дёшев. На деле внешняя таблица отдала **50 000 строк** — и для каждой выполняется полный скан внутренней таблицы (50 000 × cost(Seq Scan внутренней)) — запрос очень медленный. **Причина:** устаревшая статистика по внешней таблице. **Что делать:** выполнить **ANALYZE** по внешней таблице (и по внутренней); после обновления статистики планировщик может выбрать Hash Join или Merge Join вместо Nested Loop.</details>

**Три пункта, чтобы не забыть (узлы JOIN).** 1) **Nested Loop** — для **каждой** строки внешней таблицы поиск во внутренней; выгоден при **малой внешней** и **индексном** доступе по внутренней. При большой внешней и полном скане внутренней — очень медленно. 2) **Hash Join** — хеш по одной таблице в **памяти** (work_mem), проход по второй с пробами; выгоден при **больших** наборах без порядка; при нехватке work_mem — сброс на диск («Disk»). 3) **Merge Join** — слияние двух потоков, **отсортированных** по ключу JOIN; выгоден при уже отсортированных данных (индексы) или при нужном ORDER BY по ключу.

#### Как запомнить

**Nested Loop** = «для каждой строки первой таблицы — поиск во второй»; малая внешняя + индекс по внутренней. **Hash Join** = «хеш по одной таблице в памяти, проход по второй с пробами»; большие наборы, work_mem. **Merge Join** = «два отсортированных потока, слияние»; порядок по ключу уже есть или нужен.

#### Запомните

- **Nested Loop** — для каждой строки внешней таблицы поиск во внутренней; выгоден при малой внешней и индексном доступе по внутренней.
- **Hash Join** — хеш по одной таблице в памяти, проход по второй с пробами; выгоден при больших наборах без порядка; зависит от work_mem.
- **Merge Join** — слияние двух потоков, отсортированных по ключу JOIN; выгоден при уже отсортированных данных или при нужном ORDER BY по ключу.

**Ещё раз самыми простыми словами:** Nested Loop — «для каждого из списка А ищем в списке Б» (быстро, если список А маленький и в Б ищем по индексу). Hash Join — «выписали список Б в блокнот по номерам, проходим список А и смотрим в блокнот». Merge Join — «оба списка по номерам подряд, идём двумя пальцами и совпадающее выписываем». По названию узла в плане видно, как соединяются таблицы.

**Типичная ошибка:** видеть в плане Nested Loop с большой внешней таблицей (десятки тысяч строк) и думать «так и надо». При большой внешней и полном скане внутренней Nested Loop — очень медленный; скорее всего устаревшая статистика (планировщик недооценил rows). Сделай ANALYZE по затронутым таблицам и проверь план снова.

**Одна фраза (если забыл всё):** Nested Loop = для каждой строки первой таблицы ищем во второй (выгодно при малой первой и индексе по второй). Hash Join = хеш по одной таблице в памяти, проход по второй с пробами. Merge Join = два отсортированных потока, слияние. По названию узла видно стратегию соединения.

**Если прочитал и всё равно не понял.** Суть в одном: **как** соединяют две таблицы — видно по названию узла. **Nested Loop** — «для каждой строки из первой таблицы ищем подходящие во второй»; быстро, если первая таблица **маленькая** и во второй ищем **по индексу**; катастрофа, если первая большая и вторую каждый раз сканируем целиком. **Hash Join** — одну таблицу (обычно меньшую) «выписываем в блокнот» (строим хеш в памяти), потом проходим по второй и смотрим в блокнот; выгодно при **больших** объёмах без порядка; если блокнот не влезает в work_mem — пишем на диск, медленно. **Merge Join** — обе стороны уже **по порядку** по ключу соединения; идём двумя пальцами и совпадающее объединяем. Если в плане **Nested Loop** и внешняя таблица **огромная** (десятки тысяч строк) — скорее всего планировщик ошибся из‑за устаревшей статистики; сделай **ANALYZE** по затронутым таблицам. Перечитай **«Пошагово: как работает каждый тип JOIN»** (числа 100 строк vs 100 000) и **«Картинка в голове»** (списки и блокнот).

### 36.4. Sort, Aggregate, Limit

**Цель раздела.**  
Понять узлы **Sort**, **Aggregate** и **Limit** в плане: что они делают, откуда берутся данные и как они связаны с памятью (work_mem) и индексами. После раздела ты будешь по плану видеть, где происходит сортировка и группировка и когда они могут быть «бесплатными» за счёт индекса.

**В этом разделе главное (три строки).** 1) **Sort** — сортировка по ORDER BY; если порядок уже даёт **индекс** (запрос с ORDER BY по ключу индекса), узла Sort в плане **может не быть** — данные приходят из индекса уже отсортированными. Если данных больше work_mem — сброс на диск («Disk») и замедление. 2) **Aggregate** — группировка (GROUP BY) и подсчёт (COUNT, SUM и т.д.); может быть через хеш в памяти (Hash Aggregate) или «на ходу» (Group Aggregate), если данные уже по ключу группы. 3) **Limit** — берём только N строк; при Index Scan по ORDER BY выполнение может **остановиться** после N строк (не читаем всю таблицу).

#### Термины (расшифровка)

- **Sort** — узел плана, выполняющий **сортировку** набора строк по заданным столбцам (ORDER BY). Сортировка может выполняться **в памяти** (если данные помещаются в work_mem) или с **выгрузкой на диск** (external merge), если не помещаются — тогда в плане может быть «Sort Method: external merge» и «Disk: …». Если порядок строк уже обеспечивается **индексом** (Index Scan по ключу с нужным ORDER BY), узел Sort может **отсутствовать** — данные приходят из индекса уже отсортированными.
- **Aggregate** — узел плана, выполняющий **агрегацию**: GROUP BY (группировка по столбцам) и/или агрегатные функции (COUNT, SUM, AVG, MIN, MAX и т.д.). Может быть **Hash Aggregate** (строится хеш по ключам группы в памяти) или **Group Aggregate** (вход уже отсортирован по ключам группы — группировка «на ходу»). Hash Aggregate тоже ограничен work_mem — при нехватке памяти возможен spill на диск.
- **Limit** — узел плана, ограничивающий **число возвращаемых строк** (LIMIT n). Планировщик может «остановить» выполнение дочернего узла, как только получено n строк (например, при Limit поверх Index Scan по ORDER BY — читаем только нужное количество строк из индекса и останавливаемся).

#### Теория и правила (подробно)

- **Sort:** если в плане есть Sort с большим числом rows и «external merge» или «Disk», увеличение **work_mem** может убрать выгрузку на диск и ускорить запрос. Если запрос с ORDER BY по столбцу индекса и план использует Index Scan по этому индексу, Sort часто **нет** — данные приходят в порядке.
- **Aggregate:** Hash Aggregate строит хеш по ключам GROUP BY; при большом числе групп нужна память. Group Aggregate предполагает вход, отсортированный по ключам группы (часто за счёт индекса или предшествующего Sort). Если видно «Disk: …» у Hash Aggregate — можно увеличить work_mem.
- **Limit:** при Limit поверх узла, который может остановиться досрочно (например, Index Scan с ORDER BY), планировщик учитывает это в cost — стоимость ограничивается первыми N строками, а не полным сканом.

#### Пошагово: Sort, Aggregate, Limit (с числами)

**Sort.** Запрос: `SELECT * FROM orders ORDER BY created_at DESC LIMIT 10`. Таблица orders: 1 млн строк. **Без индекса** по created_at: планировщик выбирает Seq Scan + **Sort**. Сортировка 1 млн строк: если данные **помещаются в work_mem** (например, 64 MB), сортировка идёт **в памяти** — в плане «Sort Method: quicksort Memory: 50MB». Если **не помещаются** — часть данных сбрасывается на **диск** (external merge) — в плане «Sort Method: external merge Disk: 200MB». Чтение/запись на диск **медленная** — запрос может выполняться секунды. **С индексом** по created_at DESC: планировщик выбирает **Index Scan** по индексу — строки приходят **уже в порядке** created_at DESC. Узел **Sort в плане отсутствует** — данные приходят отсортированными из индекса. Читаем только **первые 10 строк** из индекса и останавливаемся (Limit). **Итог:** индекс по столбцу ORDER BY убирает Sort и сильно ускоряет запрос; при отсутствии индекса Sort жрёт work_mem или диск.

**Aggregate.** Запрос: `SELECT user_id, COUNT(*) FROM orders GROUP BY user_id`. **Hash Aggregate:** строим в памяти хеш по user_id; для каждой входящей строки находим или создаём группу в хеше и увеличиваем счётчик. Память — work_mem. При **большом числе групп** (100 000 разных user_id) хеш может не поместиться — в плане «Disk: …». **Group Aggregate:** если вход **уже отсортирован** по user_id (индекс по user_id или предшествующий Sort), группы идут подряд — считаем «на ходу», без хеша. Память на все группы не нужна.

**Limit.** Запрос: `SELECT * FROM orders ORDER BY created_at DESC LIMIT 10`. При **Index Scan** по индексу (created_at DESC) планировщик понимает: достаточно **10 строк**. Читаем из индекса **только первые 10** записей (10 TID), читаем 10 строк из таблицы — и **останавливаемся**. Остальные 999 990 строк не читаем. **Итог:** Limit в сочетании с Index Scan по ORDER BY даёт «early stop» — выполнение прекращается, как только получено N строк.

#### Что будет, если не хватает work_mem для Sort или Hash Aggregate

Если данных для **сортировки** или для **хеш-агрегации** **больше**, чем помещается в **work_mem**, СУБД сбрасывает часть данных на **диск** (временные файлы). В плане появится «Sort Method: external merge Disk: …» или «Disk: …» у Hash Aggregate. Чтение и запись на диск **в разы медленнее** памяти — запрос может выполняться 10–30 секунд вместо 1–2. **Решение:** увеличить **work_mem** для этой сессии (SET work_mem = '256MB';) или для этого запроса (SET LOCAL work_mem = '256MB'; в транзакции перед запросом). После увеличения повторить запрос — «Disk» может исчезнуть и запрос ускориться.

#### Типичная ошибка

Думать, что «Sort в плане всегда плохо». Sort **нужен**, когда ORDER BY по столбцу без индекса; плохо, когда Sort идёт с **Disk** (данные не влезли в work_mem) — тогда запрос тормозит. Решение при «Disk» — увеличить work_mem. Если же ORDER BY по ключу индекса — Sort в плане может **отсутствовать** (данные приходят отсортированными); это хорошо.

#### Простыми словами

**Sort** — «отсортировали строки»; если порядок уже даёт индекс (ORDER BY по ключу индекса), Sort в плане **нет** — данные приходят из индекса уже в порядке. **Aggregate** — «сгруппировали и посчитали» (GROUP BY и COUNT/SUM/…); может быть через хеш в памяти (Hash Aggregate) или «на ходу» (Group Aggregate), если данные уже по ключу группы. **Limit** — «взяли только N строк»; при удачном плане (Index Scan по ORDER BY) выполнение **останавливается** после N строк — остальное не читаем. Сортировка и хеш-агрегация жрут work_mem; если в плане «Disk» — стоит поднять work_mem для этого запроса или глобально.

#### Картинка в голове

**Sort** — раскладываешь **карточки по порядку** (по дате, по имени); если карточки уже пришли **по порядку из индекса**, раскладывать не надо — сразу отдаёшь. **Aggregate** — раскладываешь по **кучкам** (группы: по user_id, по региону) и в каждой куче **считаешь** (сколько штук, сумма); кучки могут быть в памяти (хеш) или «на ходу», если карточки уже приходят сгруппированными. **Limit** — «достаточно **первых 10**» — перестаёшь брать дальше; не перебираешь всю колоду, а только первые 10 карт.

#### Как запомнить

**Sort** — сортировка по ORDER BY; при порядке по индексу Sort **нет**; при «Disk» — увеличить work_mem. **Aggregate** — группировка и COUNT/SUM/…; Hash (хеш в памяти) или Group (вход отсортирован по ключу группы). **Limit** — только N строк; при Index Scan по ORDER BY — early stop.

#### Проверь себя (36.4)

1. Почему в плане запроса с ORDER BY по столбцу индекса может не быть узла Sort?  
<details><summary>Ответ</summary> Потому что при **Index Scan** по индексу, ключ которого совпадает с ORDER BY, строки приходят из индекса **уже в нужном порядке** (листья B-tree упорядочены по ключу). Отдельная операция сортировки не нужна — планировщик не добавляет узел Sort, и cost плана не включает стоимость сортировки.</details>

2. Что значит «Sort Method: external merge» и «Disk» в плане Sort?  
<details><summary>Ответ</summary> Это значит, что данных для сортировки **больше**, чем помещается в **work_mem**. СУБД разбивает данные на части, сортирует каждую часть в памяти, записывает на **диск** и затем **сливает** отсортированные части (external merge). Чтение/запись на диск замедляет запрос. **Решение:** увеличить work_mem для сессии или для этого запроса (SET LOCAL work_mem), чтобы сортировка умещалась в памяти.</details>

3. Чем Hash Aggregate отличается от Group Aggregate?  
<details><summary>Ответ</summary> **Hash Aggregate** строит в памяти **хеш-таблицу** по ключам GROUP BY; для каждой входящей строки находит или создаёт группу в хеше и обновляет агрегаты. **Group Aggregate** предполагает, что вход **уже отсортирован** по ключам группы (например, по индексу или после Sort); тогда группы идут подряд и агрегация делается «на ходу» без хеша. Group Aggregate не требует хранения всех групп в памяти сразу.</details>

4. В плане Sort с «Disk: 1000». Что это значит и что сделать?  
<details><summary>Ответ</summary> **Disk: 1000** значит, что данных для сортировки **больше**, чем помещается в **work_mem**. СУБД записала **1000 блоков** (страниц) во **временный файл** на диске и затем читает их при слиянии (external merge). Дисковый I/O **медленный** — запрос замедляется. **Что сделать:** увеличить **work_mem** для этой сессии или для этого запроса (SET work_mem = '128MB'; или SET LOCAL work_mem = '128MB';). После увеличения повторить запрос — сортировка может уместиться в памяти и «Disk» исчезнет, запрос ускорится.</details>

**Три пункта, чтобы не забыть (Sort, Aggregate, Limit).** 1) **Sort** — сортировка по ORDER BY; если порядок даёт **индекс** (Index Scan по ключу с нужным ORDER BY), узел Sort **отсутствует** — данные приходят из индекса уже в порядке. При «external merge» / «Disk» — увеличить work_mem. 2) **Aggregate** — Hash (хеш по ключам группы в памяти) или Group (вход уже отсортирован по ключу группы — группировка «на ходу»). При «Disk» у Hash Aggregate — увеличить work_mem. 3) **Limit** — при Index Scan по ORDER BY выполнение может **остановиться** после N строк (early stop) — остальное не читаем.

#### Запомните

- **Sort** — сортировка по ORDER BY; при порядке по индексу Sort может отсутствовать. При «external merge» / «Disk» — увеличить work_mem.
- **Aggregate** — группировка и агрегатные функции; Hash Aggregate (хеш в памяти) или Group Aggregate (вход отсортирован по ключу группы).
- **Limit** — ограничение числа строк; при удачном плане выполнение может остановиться досрочно.

**Ещё раз самыми простыми словами:** Sort — «отсортировали»; если данные уже приходят по порядку из индекса, Sort не нужен. Aggregate — «сгруппировали и посчитали»; либо хеш в памяти, либо «на ходу», если данные уже по группам. Limit — «взяли только N»; при удачном плане дальше не читаем. Если в плане «Disk» у Sort или Hash Aggregate — поднять work_mem.

**Одна фраза (если забыл всё):** Sort — сортировка по ORDER BY; при индексе по этому столбцу Sort может не быть. Aggregate — группировка и COUNT/SUM; Hash в памяти или Group «на ходу». Limit — только N строк. «Disk» у Sort/Aggregate — увеличить work_mem.

**Если прочитал и всё равно не понял.** Главное: **Sort** — «отсортировали строки»; если в запросе ORDER BY по столбцу, по которому есть индекс, данные уже приходят **в порядке** из индекса — узла Sort в плане **нет**, это хорошо. Если Sort есть и в плане написано **«Disk»** — данных больше, чем влезает в work_mem, идёт сброс на диск и запрос тормозит; **решение:** увеличить work_mem. **Aggregate** — «сгруппировали и посчитали» (GROUP BY, COUNT, SUM); может быть через хеш в памяти (Hash Aggregate) или «на ходу» (Group Aggregate), если данные уже по группам. **Limit** — «взяли только N строк»; при удачном плане (Index Scan по ORDER BY) база **останавливается** после N строк и остальное не читает. Перечитай **«Пошагово: Sort, Aggregate, Limit»** (с числами по work_mem и индексу) и **«Картинка в голове»** (карточки, кучки, первые 10).

### 36.5. Cost и сравнение планов

**Цель раздела.**  
Понять, что такое **cost** (стоимость) в плане: **startup cost** и **total cost**, как они складываются по дереву и как планировщик **сравнивает** альтернативные планы по суммарному cost. После раздела ты будешь понимать, почему выбран именно такой план и как параметры (seq_page_cost, random_page_cost, cpu_tuple_cost) на него влияют.

**В этом разделе главное (три строки).** 1) **cost** — это **условные единицы** («баллы»), **не секунды**; планировщик считает cost для каждого узла и выбирает план с **минимальным total cost** корня. 2) **startup cost** — стоимость до первой строки результата; **total cost** — полная стоимость узла; по дереву cost складывается. 3) На cost влияют **параметры** (seq_page_cost, random_page_cost, cpu_tuple_cost) и **оценка rows** из статистики; **устаревшая статистика** → неверный cost → плохой план → после массовых изменений делай ANALYZE.

#### Термины (расшифровка)

- **cost (стоимость)** — числовая оценка «цены» операции в **условных единицах**. Не время в миллисекундах: 1 единица примерно соответствует одному **последовательному** чтению страницы с диска (при настройках по умолчанию). Планировщик считает cost для каждого узла плана и **суммирует** по дереву; план с **меньшим total cost** выбирается как оптимальный.
- **startup cost** — стоимость «до первой строки результата». Например, у Sort — это стоимость полной сортировки (до выдачи первой строки нужно отсортировать всё); у Seq Scan — близка к нулю (первая строка сразу). В выводе плана часто пишут **cost=startup..total** (например, 0.00..10.50).
- **total cost** — полная стоимость узла (включая выдачу всех строк). У родительского узла total cost складывается из total cost дочерних и собственной работы узла (JOIN, Sort и т.д.). **Корневой** узел плана имеет total cost = оценка полной стоимости всего запроса; планировщик минимизирует именно её при выборе плана.

#### Теория и правила (подробно)

- Планировщик строит **несколько вариантов** плана (разные порядки JOIN, разные способы доступа — seq scan vs index scan) и для каждого считает **total cost**. Выбирается план с **минимальным** total cost. Поэтому изменение статистики (ANALYZE) или параметров стоимости (random_page_cost, work_mem и т.д.) может **переключить** план на другой вариант.
- **Параметры стоимости** (см. §38): **seq_page_cost** — стоимость одной последовательно прочитанной страницы; **random_page_cost** — одной случайно прочитанной страницы; **cpu_tuple_cost** — обработка одной строки. Уменьшение random_page_cost (например, для SSD) делает индексный доступ (много случайных чтений) «дешевле» относительно seq scan — планировщик чаще выбирает индекс.
- **Оценка строк (rows)** влияет на cost дочерних узлов и на cost родителя (например, cost Nested Loop зависит от числа строк внешней таблицы). Неверная оценка rows (устаревшая или неточная статистика) ведёт к неверному cost и к выбору плохого плана.

#### Пошагово: как планировщик сравнивает планы (с числами)

Допустим, запрос: `SELECT * FROM orders WHERE user_id = 5`. Таблица orders: 100 000 строк, 4 000 страниц по 8 KB. Есть индекс по user_id.

**Вариант 1 — Seq Scan.** Планировщик считает: прочитать 4 000 страниц **последовательно** — cost = 4 000 × seq_page_cost (по умолчанию 1.0) = 4 000. Плюс обработка 100 000 строк (cpu_tuple_cost). Допустим, total cost ≈ 4 500.

**Вариант 2 — Index Scan.** Планировщик считает: спуск по индексу — 3–4 страницы индекса (random или sequential). По индексу ожидается, допустим, 50 строк с user_id = 5 — 50 TID. Чтение 50 страниц таблицы **по TID** — это **случайные** чтения: 50 × random_page_cost (по умолчанию 4.0) = 200. Плюс 4 страницы индекса, плюс обработка 50 строк. Допустим, total cost ≈ 250.

Планировщик **сравнивает** 4 500 (Seq Scan) и 250 (Index Scan) — выбирает **Index Scan** (меньший cost). Если **статистика устарела** и планировщик оценил rows не 50, а 50 000 (думает, что user_id = 5 у половины строк), cost Index Scan станет огромным (50 000 × random_page_cost) — он выберет Seq Scan. На деле строк с user_id = 5 может быть 50 — тогда Seq Scan прочитает всю таблицу зря, а Index Scan был бы быстрее. **Итог:** неверная оценка rows (устаревшая статистика) ведёт к неверному cost и к выбору плохого плана.

#### Что будет, если не обновлять статистику после массовых изменений

После массового INSERT (загрузили 1 млн строк) или массового DELETE (удалили половину таблицы) **статистика** (число строк, распределение значений) **устаревает**. Планировщик оценивает **rows** по старой статистике: например, думает, что в таблице 10 000 строк, а их 1 000 000. Тогда cost Seq Scan он посчитает в 100 раз **ниже**, чем на самом деле — или наоборот, cost Index Scan посчитает неверно. В итоге выберет **план**, который при реальных данных оказывается **медленным** (например, Nested Loop при большой внешней таблице из-за заниженной оценки rows). **Решение:** после массовых изменений всегда выполнять **ANALYZE таблица;** — обновить статистику, чтобы оценки rows и cost были ближе к реальности.

#### Простыми словами

**cost** — это «баллы», которые планировщик начисляет за каждую операцию (чтение страниц, обработку строк, сортировку). Он перебирает варианты плана и выбирает тот, у кого **сумма баллов (total cost) наименьшая**. Эти баллы **не секунды** — они нужны только для **сравнения** вариантов между собой. Параметры вроде seq_page_cost и random_page_cost задают «цену» операций; их изменение меняет баланс между планами (например, seq scan vs index scan). Неверная оценка rows (устаревшая статистика) даёт неверный cost — планировщик может выбрать плохой план.

#### Картинка в голове

Планировщик — как **маршрутизатор**: считает «цену» каждого маршрута (cost) и выбирает **самый дешёвый**. Цена складывается из участков (узлов): столько-то за чтение страниц, столько-то за обработку строк. **Изменил тарифы** (seq_page_cost, random_page_cost) — другой маршрут может стать «дешевле» и будет выбран. **Ошибся в оценке** (устаревшая статистика — неверное число строк) — цена маршрута посчитана неверно, выбран неоптимальный маршрут.

#### Как запомнить

**cost** — условные единицы; планировщик выбирает план с **минимальным total cost** корня. **startup cost** — до первой строки; **total cost** — полная стоимость узла. Параметры (seq_page_cost, random_page_cost, cpu_tuple_cost) и **статистика (rows)** влияют на cost; устаревшая статистика → неверный cost → плохой план.

#### Проверь себя (36.5)

1. Что такое total cost в корневом узле плана?  
<details><summary>Ответ</summary> **Total cost** корневого узла — это **оценка полной стоимости выполнения всего запроса** в условных единицах. Планировщик строит несколько вариантов плана и выбирает тот, у которого этот total cost **минимальный**. Это не время в секундах — для реального времени смотри actual time в EXPLAIN ANALYZE.</details>

2. Как изменение random_page_cost влияет на выбор плана?  
<details><summary>Ответ</summary> **random_page_cost** — стоимость одного **случайного** чтения страницы. Индексный доступ обычно даёт много случайных чтений (страницы таблицы по TID). **Seq scan** — последовательные чтения (seq_page_cost). Если **уменьшить** random_page_cost (например, для SSD случайное чтение почти так же быстро, как последовательное), стоимость индексного доступа **снижается** относительно seq scan — планировщик **чаще** выбирает индекс (Index Scan) вместо Seq Scan.</details>

3. Почему при устаревшей статистике план может быть плохим?  
<details><summary>Ответ</summary> Планировщик оценивает **rows** (число строк) по **статистике** (гистограммы, n_distinct и т.д.). По rows считает **cost** каждого узла. Если статистика **устарела** (после массовых INSERT/UPDATE/DELETE не сделан ANALYZE), оценки rows неверны — тогда и **cost** считаются неверно. Планировщик может выбрать план, который при реальных данных оказывается медленным (например, Nested Loop при большой внешней таблице из-за заниженной оценки rows). Решение — обновить статистику: ANALYZE таблица;</details>

4. Что такое startup cost и зачем он нужен планировщику?  
<details><summary>Ответ</summary> **Startup cost** — стоимость «до первой строки результата» узла. Например, у **Sort** до первой строки нужно **отсортировать всё** — startup cost Sort равен полной стоимости сортировки. У **Seq Scan** первая строка приходит почти сразу — startup cost близок к нулю. Планировщику startup cost нужен для планов с **Limit** или для **вложенных** узлов: если родительский узел (например, Limit) может остановиться после первых N строк, он учитывает startup cost дочерних узлов — «сколько стоит добраться до первой строки». Так cost плана с Limit может быть ниже полного cost дочернего узла.</details>

**Три пункта, чтобы не забыть (cost).** 1) **cost** — **условные единицы**, не секунды; планировщик выбирает план с **минимальным total cost** корня. 2) **startup cost** — до первой строки; **total cost** — полная стоимость узла; по дереву cost складывается. 3) **Параметры** (seq_page_cost, random_page_cost, cpu_tuple_cost) и **статистика (rows)** влияют на cost; **устаревшая статистика** → неверные rows → неверный cost → плохой план. После массовых изменений — ANALYZE.

#### Запомните

- **cost** — условные единицы стоимости; планировщик выбирает план с **минимальным total cost** корня.
- **startup cost** — до первой строки; **total cost** — полная стоимость узла; по дереву cost складывается.
- Параметры (seq_page_cost, random_page_cost, cpu_tuple_cost) и статистика (rows) влияют на cost и на выбор плана.

**Ещё раз самыми простыми словами:** cost — это «баллы» за операцию; планировщик выбирает план с наименьшей суммой баллов. Баллы не секунды — только для сравнения. Неверная оценка строк (устаревшая статистика) даёт неверные баллы — планировщик может выбрать медленный план. Поэтому после больших изменений данных всегда делай ANALYZE.

**Одна фраза (если забыл всё):** cost — условные «баллы»; планировщик выбирает план с **минимальным total cost**. Не секунды. Устаревшая статистика → неверный cost → плохой план → после массовых изменений делай ANALYZE.

**Если прочитал и всё равно не понял.** Главное: **cost** в плане — это **не время в секундах**. Это условные «баллы», которые планировщик считает по правилам (сколько стоит прочитать страницу, обработать строку и т.д.). Он перебирает варианты плана и выбирает тот, у кого **сумма баллов (total cost) наименьшая**. Зачем так: реальное время выполнения заранее неизвестно (зависит от кэша, диска, нагрузки), поэтому сравнивают по единой «валюте» — cost. **Реальное время** ты видишь только после выполнения — в **EXPLAIN ANALYZE** в полях **actual time** и **Execution Time** (там миллисекунды). Если план получился плохой (медленный по факту) — часто виновата **устаревшая статистика**: оценки rows неверные → cost посчитан неверно → выбран не тот план. **Что делать:** после больших изменений данных выполнять **ANALYZE** по таблицам. Перечитай **«Часто путают»** (cost vs время) и **«Пошагово: как планировщик сравнивает планы»** (числа для Seq Scan и Index Scan).

### 36.6. BUFFERS

**Цель раздела.**  
Научиться использовать **EXPLAIN (ANALYZE, BUFFERS)** и читать в плане информацию о буферах: **shared hit**, **shared read**, **temp read/write**. Понимать, что значит «чтение из кэша» vs «чтение с диска» и как это связано с производительностью. После раздела ты будешь по плану видеть, сколько I/O было с диска и сколько из кэша.

**В этом разделе главное (три строки).** 1) **EXPLAIN (ANALYZE, BUFFERS)** добавляет к плану: **shared hit** — страницы, прочитанные **из кэша** (память), **shared read** — страницы, прочитанные **с диска**; много read — узкое место в дисковом I/O. 2) **temp read / temp written** — страницы **временных файлов** (сортировка или хеш не поместились в work_mem); много temp — увеличить work_mem. 3) Для **диагностики** медленных запросов используй всегда **ANALYZE + BUFFERS** — по ним видно, где время и откуда читали данные (кэш или диск).

#### Термины (расшифровка)

- **BUFFERS** — опция EXPLAIN (ANALYZE, BUFFERS), добавляющая в вывод плана **статистику по буферам** (страницам) для каждого узла: сколько страниц прочитано из **кэша** (shared buffers) и сколько с **диска**. Помогает понять, было ли узкое место в дисковом I/O.
- **shared hit** — число страниц, прочитанных из **кэша** PostgreSQL (shared_buffers). Чтение из памяти — очень быстрое. Большой shared hit при малом shared read — данные уже были в кэше, дисковый I/O не узкое место.
- **shared read** — число страниц, прочитанных **с диска** (не найдены в shared_buffers). Дисковое чтение — медленное; большой shared read указывает на то, что запрос «тащил» данные с диска. При повторном выполнении те же страницы могут оказаться в кэше (shared hit растёт, shared read падает).
- **temp read / temp written** — страницы, прочитанные из **временных файлов** или записанные в них (сортировка или хеш, не поместившиеся в work_mem). Большие temp read/written — признак нехватки work_mem и выгрузки на диск.

#### Теория и правила (подробно)

- **Формат в плане:** например, `Buffers: shared hit=100 read=50` — 100 страниц из кэша, 50 с диска. На уровне узла показываются буферы, использованные **этим узлом** (включая дочерние, в зависимости от СУБД/версии). В PostgreSQL 10+ для каждого узла выводятся свои Buffers.
- **Интерпретация:** если у узла **shared read** велико — этот узел много читал с диска; при повторном запуске после прогрева кэша shared read может уменьшиться. Если **temp read/write** велики — операция (Sort или Hash) не поместилась в work_mem; увеличение work_mem может убрать временные файлы и ускорить запрос.
- **Сброс кэша:** для «холодного» теста можно сбросить кэш (например, restart сервера или специальные расширения); в обычной работе смотреть на фактический вывод после типичной нагрузки.

#### Пример фрагмента плана (PostgreSQL)

```
Seq Scan on orders  (actual time=0.05..1.20 rows=1000 loops=1)
  Filter: (user_id = 5)
  Rows Removed by Filter: 9000
  Buffers: shared hit=80 read=20
```

80 страниц из кэша, 20 с диска. При повторном выполнении те же 20 могли бы быть hit.

#### Пошагово: как читать BUFFERS в плане (с числами)

Выполняешь: `EXPLAIN (ANALYZE, BUFFERS) SELECT * FROM orders WHERE user_id = 5;`. В выводе у узла Seq Scan (или Index Scan) видишь, например:

```
Seq Scan on orders  (actual time=0.05..850.00 rows=50000 loops=1)
  Filter: (user_id = 5)
  Rows Removed by Filter: 50000
  Buffers: shared hit=120 read=3880
```

**Расшифровка.** **shared hit=120** — **120 страниц** прочитано из **кэша** PostgreSQL (shared_buffers), то есть из **памяти**. Чтение из памяти — очень быстрое (доли миллисекунды на страницу). **shared read=3880** — **3880 страниц** прочитано **с диска** (не найдены в кэше). Чтение с диска — медленное (миллисекунды на страницу). **Итог:** запрос прочитал 120 страниц из памяти и 3880 с диска — всего 4000 страниц (вся таблица при Seq Scan). Основное время ушло на **дисковое чтение** (3880 страниц с диска) — узкое место в I/O. Если бы те же страницы были в кэше (при повторном выполнении после «прогрева»), было бы shared hit=4000, read=0 — запрос выполнился бы в разы быстрее.

У узла Sort может быть: **Buffers: temp read=500 written=500** — **500 блоков** записано во **временный файл** на диске и 500 прочитано обратно (сортировка не поместилась в work_mem). **Итог:** увеличение work_mem может убрать temp read/write и ускорить запрос.

#### Что будет, если игнорировать BUFFERS при диагностике

Если смотришь только **actual time** и не смотришь **BUFFERS**, можешь не понять **причину** медленного запроса. Узел с большим actual time может быть медленным из-за **дискового I/O** (shared read велико) — тогда решение: больше RAM, лучше кэширование, или уменьшить объём читаемых данных (индекс, фильтр). Или из-за **временных файлов** (temp read/write велики) — тогда решение: увеличить work_mem. **BUFFERS** даёт ответ: «где ушло время — в чтение с диска или в сброс на диск из-за нехватки памяти».

#### Простыми словами

**BUFFERS** в EXPLAIN (ANALYZE, BUFFERS) показывает: сколько страниц взято **из памяти** (shared hit) и сколько **с диска** (shared read). **Много read** — запрос активно читал с диска (узкое место в I/O). **Много hit** — данные уже в кэше (дисковый I/O не узкое место). **temp read/written** — страницы временных файлов (сортировка или хеш не поместились в work_mem — сброс на диск). По BUFFERS видно, было ли узкое место в диске или в нехватке памяти.

#### Картинка в голове

**shared hit** — книгу взял **с полки** (память, быстро). **shared read** — **пошёл в библиотеку** (диск, медленно). **temp read/write** — конспект **не влез в блокнот**, писал в **отдельную тетрадь** на столе (временные файлы на диске из-за нехватки work_mem). По числам в Buffers видно, где «ходили за данными» — в память (hit) или на диск (read, temp).

#### Как запомнить

**BUFFERS** — shared **hit** (из кэша), shared **read** (с диска), **temp** read/write (временные файлы при нехватке work_mem). Большой read — узкое место в дисковом I/O; большой temp — увеличить work_mem.

#### Проверь себя (36.6)

1. Что означают shared hit и shared read в выводе BUFFERS?  
<details><summary>Ответ</summary> **shared hit** — число страниц (буферов), прочитанных из **кэша** PostgreSQL (shared_buffers), то есть из памяти. **shared read** — число страниц, прочитанных **с диска**, потому что их не было в кэше. Большой shared read означает много дискового I/O для этого узла; при повторном выполнении после прогрева кэша часть read может превратиться в hit.</details>

2. В плане у узла Sort: temp written=1000, temp read=1000. Что это значит и что делать?  
<details><summary>Ответ</summary> Это значит, что данных для сортировки **больше**, чем помещается в **work_mem**. СУБД записала 1000 страниц во **временный файл** на диске (temp written) и затем прочитала их обратно (temp read) для слияния. Дисковый I/O замедляет запрос. **Что делать:** увеличить **work_mem** для этой сессии или для этого запроса (SET LOCAL work_mem), чтобы сортировка умещалась в памяти и временные файлы не создавались.</details>

3. Зачем использовать EXPLAIN (ANALYZE, BUFFERS) вместо одного ANALYZE?  
<details><summary>Ответ</summary> **BUFFERS** добавляет к фактическим временам и строкам **информацию об I/O**: сколько страниц прочитано из кэша (shared hit) и с диска (shared read), а также о временных файлах (temp read/write). По этому видно, было ли узкое место в дисковом чтении или в нехватке памяти (temp). Для диагностики медленных запросов комбинация ANALYZE + BUFFERS даёт полную картину: где время, сколько строк и откуда читали данные.</details>

4. В плане у узла Sort: temp written=1000, temp read=1000. Что это значит в пересчёте на страницы и что делать?  
<details><summary>Ответ</summary> **temp written=1000** и **temp read=1000** означают, что **1000 блоков** (страниц по 8 KB — итого около 8 MB данных) было **записано** во временный файл на диске и затем **прочитано** обратно при слиянии (external merge). Это значит, что данных для сортировки **больше**, чем помещается в **work_mem**. **Что делать:** увеличить **work_mem** для этой сессии или для этого запроса (SET work_mem = '128MB'; или SET LOCAL work_mem = '128MB';), чтобы сортировка умещалась в памяти и временные файлы не создавались. После увеличения повторить запрос — temp read/write должны исчезнуть и запрос ускориться.</details>

**Три пункта, чтобы не забыть (BUFFERS).** 1) **EXPLAIN (ANALYZE, BUFFERS)** выводит по узлам: **shared hit** (страницы из кэша), **shared read** (страницы с диска), **temp read/written** (временные файлы при нехватке work_mem). 2) **Большой shared read** — много чтений с диска, узкое место в I/O. **Большой temp read/write** — сортировка или хеш не поместились в work_mem — увеличить work_mem. 3) Для диагностики медленных запросов **всегда** используй ANALYZE + BUFFERS — по ним видно, где время и откуда читали данные.

#### Запомните

- **EXPLAIN (ANALYZE, BUFFERS)** выводит по узлам: **shared hit** (из кэша), **shared read** (с диска), **temp read/written** (временные файлы при нехватке work_mem).
- Большой shared read — много чтений с диска; большой temp read/write — сортировка/хеш не поместились в work_mem.
- Для диагностики медленных запросов используй ANALYZE + BUFFERS.

**Ещё раз самыми простыми словами:** BUFFERS показывает, откуда читали данные — из памяти (hit) или с диска (read). Много read — запрос «тащил» с диска, это медленно. temp read/write — данные не влезли в память (work_mem), писали на диск — поднять work_mem. Для разбора медленных запросов всегда снимай план с ANALYZE и BUFFERS.

**Одна фраза (если забыл всё):** shared **hit** = из кэша, shared **read** = с диска; много read — узкое место I/O. temp read/write — не хватило work_mem, сброс на диск — увеличить work_mem. Для диагностики — всегда EXPLAIN (ANALYZE, BUFFERS).

**Если прочитал и всё равно не понял.** Суть в одном: **BUFFERS** в EXPLAIN (ANALYZE, BUFFERS) показывает, **откуда** читались данные. **shared hit** — страницы взяты **из памяти** (кэш) — быстро. **shared read** — страницы прочитаны **с диска** — медленно; если **read** большое, узкое место в дисковом I/O (можно подумать об индексах, кэше, объёме данных). **temp read / temp written** — данные **не влезли** в work_mem (сортировка или хеш), часть сбросили на диск во временные файлы — запрос тормозит; **решение:** увеличить work_mem для этого запроса или сессии. Для разбора медленного запроса всегда снимай план с **ANALYZE и BUFFERS**: по actual time видно, где время, по BUFFERS — читали из кэша или с диска, было ли temp. Перечитай **«Пошагово: как читать BUFFERS»** (числа shared hit/read, temp) и **«Картинка в голове»** (полка vs библиотека, блокнот vs тетрадь).
