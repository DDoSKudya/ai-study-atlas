## 38. Cost-based оптимизатор

### 38.1. Модель стоимости

**Цель раздела.**  
Понять **модель стоимости** планировщика: какие параметры задают «цену» операций (**seq_page_cost**, **random_page_cost**, **cpu_tuple_cost** и др.) и как из них складывается cost узла. После раздела ты будешь понимать, как настройка этих параметров влияет на выбор плана (например, seq scan vs index scan).

**В этом разделе главное (три строки).** 1) Планировщик считает **cost** (стоимость) по «тарифам»: **seq_page_cost** — одна страница последовательно, **random_page_cost** — одна страница случайно, **cpu_tuple_cost** — обработка одной строки; из них и оценок (страницы, строки) складывается cost узла и всего плана. 2) План с **минимальным total cost** выбирается для выполнения. 3) На **SSD** случайное чтение почти как последовательное — **уменьшай random_page_cost** (например, до 1.1–1.2), иначе планировщик реже выбирает индекс; **effective_cache_size** — подсказка о размере кэша, влияет на выбор между seq scan и index scan.

#### Термины (расшифровка)

- **Cost-based optimizer** — оптимизатор, который выбирает план по **оценке стоимости (cost)** каждого варианта. Стоимость считается в **условных единицах**; 1 единица по умолчанию соответствует одному последовательному чтению страницы с диска. План с **минимальным total cost** выбирается для выполнения.
- **seq_page_cost** — стоимость **одного последовательного** чтения страницы (по умолчанию 1.0). Seq Scan читает страницы подряд — его cost пропорционален числу страниц таблицы × seq_page_cost.
- **random_page_cost** — стоимость **одного случайного** чтения страницы (по умолчанию в PostgreSQL 4.0). Index Scan читает страницы таблицы по TID «вразнобой» — каждое такое чтение считается как random_page_cost. На **SSD** случайное чтение почти так же быстро, как последовательное — random_page_cost часто уменьшают (например, до 1.1 или 1.2), чтобы планировщик чаще выбирал индекс.
- **cpu_tuple_cost** — стоимость обработки **одной строки** процессором (по умолчанию 0.01). Умножается на оценку числа строк, проходящих через узел. В сумме с I/O cost даёт total cost узла.

#### Теория и правила (подробно)

- **Формула (упрощённо):** cost узла ≈ (число страниц I/O) × (seq_page_cost или random_page_cost) + (число строк) × cpu_tuple_cost + узкоспецифичные затраты (например, сортировка). Планировщик суммирует cost по дереву; корневой total cost — критерий выбора плана.
- **Влияние random_page_cost:** при значении 4.0 индексный доступ «дорог» (много случайных чтений). При 1.1 на SSD индексный доступ дешевеет — планировщик чаще предпочитает Index Scan вместо Seq Scan там, где раньше выбирал seq scan.
- **effective_cache_size** (не единица cost, а подсказка планировщику): оценка объёма кэша (память + диск кэш ОС). Планировщик использует её, чтобы оценить вероятность попадания страницы в кэш при повторном чтении; влияет на выбор между seq scan и index scan (при большом кэше seq scan «дешевле», т.к. страницы могут быть в кэше).

#### Пошагово: как складывается cost узла (с числами)

**Seq Scan по таблице orders: 10 000 страниц по 8 KB.**  
Планировщик считает: прочитать **10 000 страниц последовательно** — cost I/O = 10 000 × **seq_page_cost** (по умолчанию 1.0) = **10 000**. Обработать, допустим, **1 000 000 строк** (оценка по статистике) — cost CPU = 1 000 000 × **cpu_tuple_cost** (по умолчанию 0.01) = **10 000**. **Total cost** узла Seq Scan ≈ 10 000 + 10 000 = **20 000** (упрощённо).

**Index Scan по индексу на user_id: ожидается 50 строк.**  
Планировщик считает: спуск по индексу — 3–4 страницы индекса (sequential или random). Чтение **50 страниц таблицы** по TID — это **50 случайных** чтений: 50 × **random_page_cost** (по умолчанию 4.0) = **200**. Плюс 4 страницы индекса, плюс обработка 50 строк (50 × 0.01 = 0.5). **Total cost** Index Scan ≈ 4 + 200 + 0.5 ≈ **205**. Планировщик **сравнивает** 20 000 (Seq Scan) и 205 (Index Scan) — выбирает **Index Scan** (меньший cost). **Итог:** из «тарифов» (seq_page_cost, random_page_cost, cpu_tuple_cost) и оценок (число страниц, число строк) складывается cost; план с **минимальным total cost** выбирается.

#### Часто путают

**seq_page_cost** и **random_page_cost** — за что что отвечает. **seq_page_cost** — стоимость **одного последовательного** чтения страницы (страницы читаются подряд, как при Seq Scan). **random_page_cost** — стоимость **одного случайного** чтения страницы (переход к другой странице «вразнобой», как при чтении таблицы по TID из индекса). На HDD случайное чтение сильно медленнее последовательного — поэтому по умолчанию random_page_cost = 4.0, а seq_page_cost = 1.0. На SSD разница маленькая — random_page_cost имеет смысл снижать (1.1–1.2), чтобы планировщик не переоценивал «цену» индексного доступа.

#### Что будет, если не уменьшить random_page_cost на SSD

На **HDD** случайное чтение страницы **значительно медленнее** последовательного (головка диска прыгает по дорожкам) — random_page_cost = 4.0 разумен. На **SSD** случайное и последовательное чтение **почти одинаковы по скорости** (нет механической головки). Если оставить random_page_cost = 4.0 на SSD, планировщик будет считать **индексный доступ** (много случайных чтений по TID) **в 4 раза дороже**, чем на самом деле. Он чаще выберет **Seq Scan** там, где **Index Scan** на SSD был бы быстрее. **Итог:** на сервере с SSD **уменьшай** random_page_cost (например, до 1.1–1.2) — планировщик будет чаще выбирать индекс там, где он реально выгоден.

#### Простыми словами

Планировщик считает «цену» плана по **тарифам**: сколько стоит одно последовательное чтение страницы (seq_page_cost), одно случайное (random_page_cost), обработка одной строки (cpu_tuple_cost). Из этих тарифов и оценок (сколько страниц, сколько строк) складывается **cost** каждого узла и всего плана. Меняешь тарифы (например, снижаешь random_page_cost для SSD) — меняется баланс между планами (чаще индекс вместо полного скана). **effective_cache_size** — подсказка планировщику о размере кэша; влияет на оценку «попадания в кэш» и на выбор между seq и index scan.

#### Картинка в голове

Тарифы — как **цены на дороги**: бесплатная дорога (seq_page_cost), платная (random_page_cost), плюс «время водителя» за каждую обработанную строку (cpu_tuple_cost). Планировщик считает стоимость маршрута: столько-то км по бесплатной, столько-то по платной, плюс время. **Снизил** цену платной дороги (random_page_cost на SSD) — маршрут через платную дорогу (индекс) может стать выгоднее объезда (seq scan). **effective_cache_size** — «сколько топлива уже в баке»: при большом кэше бесплатная дорога (seq) «дешевле», потому что данные уже в памяти.

#### Как запомнить

**Cost-based оптимизатор** выбирает план с **минимальным total cost**. **seq_page_cost** — цена одной последовательной страницы; **random_page_cost** — одной случайной (на SSD уменьшать до 1.1–1.2). **cpu_tuple_cost** — цена обработки одной строки. **effective_cache_size** — подсказка о кэше; влияет на выбор seq vs index scan.

#### Проверь себя (38.1)

1. Зачем уменьшать random_page_cost на сервере с SSD?  
<details><summary>Ответ</summary> На **SSD** случайное чтение страницы почти так же быстро, как последовательное (в отличие от HDD). По умолчанию **random_page_cost = 4.0** — планировщик считает случайное чтение в 4 раза дороже последовательного и часто выбирает **Seq Scan** вместо **Index Scan**, чтобы «не платить» за случайный I/O. На SSD это консервативно: индексный доступ на самом деле не так дорог. **Уменьшение** random_page_cost (например, до 1.1–1.2) делает cost индексного доступа **ниже** — планировщик **чаще** выбирает Index Scan там, где он реально быстрее.</details>

2. Что такое cpu_tuple_cost и на что он влияет?  
<details><summary>Ответ</summary> **cpu_tuple_cost** — стоимость **обработки одной строки** процессором (по умолчанию 0.01). Умножается на **оценку числа строк**, проходящих через узел. Влияет на общий cost узлов (Filter, Join, Aggregate и т.д.): при большом числе строк вклад CPU растёт. Относительно I/O (seq_page_cost, random_page_cost) он обычно мал; основное влияние на выбор плана оказывают именно затраты на чтение страниц.</details>

3. Что такое effective_cache_size и как он влияет на план?  
<details><summary>Ответ</summary> **effective_cache_size** — **оценка** объёма кэша, доступного для чтения с диска (память ОС + shared_buffers PostgreSQL). Планировщик **не резервирует** эту память — использует как **подсказку**: при **большом** effective_cache_size он считает, что часть страниц при **последовательном** чтении (Seq Scan) может **уже быть в кэше** — тогда cost Seq Scan снижается относительно Index Scan. Планировщик **чаще** выбирает Seq Scan для больших таблиц, если effective_cache_size большой (данные «помещаются» в кэш). Обычно ставят 50–75% от RAM сервера.</details>

4. Почему планировщик выбирает план с «минимальным total cost», а не «минимальным временем»?  
<details><summary>Ответ</summary> **Cost** — это **условные единицы** (не секунды): планировщик не знает заранее реальное время выполнения (оно зависит от кэша, нагрузки диска, конкуренции за CPU). Он может только **оценить** «цену» операций по тарифам (seq_page_cost, random_page_cost, cpu_tuple_cost) и по статистике (число страниц, строк). Поэтому он **сравнивает** варианты плана по **cost** и выбирает тот, у которого суммарный cost **минимальный** — при правильных тарифах и актуальной статистике этот план обычно оказывается и самым быстрым по времени. Реальное время видно только после выполнения (EXPLAIN ANALYZE, actual time).</details>

**Три пункта, чтобы не забыть (модель стоимости).** 1) **Cost-based оптимизатор** выбирает план с **минимальным total cost**; cost считается по **тарифам**: seq_page_cost, random_page_cost, cpu_tuple_cost и оценкам (страницы, строки). 2) **random_page_cost** на **SSD** стоит **уменьшать** (1.1–1.2) — на SSD случайное чтение почти так же быстро, как последовательное; иначе планировщик реже выбирает индекс. 3) **effective_cache_size** — подсказка о размере кэша; при большом кэше seq scan «дешевле» в модели — влияет на выбор между seq и index scan.

#### Запомните

- **Cost-based оптимизатор** выбирает план с **минимальным total cost**; cost считается по параметрам: **seq_page_cost**, **random_page_cost**, **cpu_tuple_cost** и др.
- **random_page_cost** на SSD стоит уменьшать (например, 1.1–1.2), чтобы планировщик чаще выбирал индексный доступ.
- **effective_cache_size** — подсказка планировщику о размере кэша; влияет на оценку «попадания в кэш» и на выбор между seq и index scan.

**Ещё раз самыми простыми словами:** Планировщик считает «цену» плана по тарифам: чтение страницы последовательно и случайно, обработка строки. Выбирает самый «дешёвый» план. На SSD случайное чтение почти как последовательное — снизь тариф за случайное чтение (random_page_cost), иначе планировщик будет реже выбирать индекс. effective_cache_size — подсказка «сколько данных уже в кэше»; влияет на выбор между полным сканом и индексом.

**Одна фраза (если забыл всё):** cost считают по seq_page_cost, random_page_cost, cpu_tuple_cost. План с минимальным total cost. На SSD уменьшай random_page_cost (1.1–1.2); effective_cache_size — подсказка о кэше.

**Если прочитал и всё равно не понял.** Суть в одном: планировщик не угадывает «сколько секунд» займёт запрос — он считает **условную цену** (cost) по **тарифам**: сколько стоит прочитать страницу последовательно (seq_page_cost), сколько — случайно (random_page_cost), сколько — обработать строку (cpu_tuple_cost). Из этого и оценок (сколько страниц, строк) получается cost плана. Он выбирает план с **наименьшей ценой**. На SSD случайное чтение почти как последовательное — поэтому тариф за случайное чтение (random_page_cost) стоит снизить, иначе планировщик будет «переплачивать» за индекс и реже его выбирать. **effective_cache_size** — просто подсказка «сколько данных уже в кэше»; память под неё не выделяется. Перечитай **«Пошагово: как складывается cost узла»** (числа для Seq Scan и Index Scan) и **«Картинка в голове»** (тарифы как цены на дороги).

### 38.2. Порядок JOIN

**Цель раздела.**  
Понять, как планировщик выбирает **порядок соединения таблиц** (join order): перебор вариантов, динамическое программирование, почему порядок важен для cost (Nested Loop vs Hash/Merge). После раздела ты будешь понимать, почему в плане таблицы соединены именно в таком порядке и как оценка rows влияет на этот выбор.

**В этом разделе главное (три строки).** 1) **Порядок JOIN** — в каком порядке «присоединять» таблицы к результату; от порядка зависит **cost** (при Nested Loop выгодно, чтобы **внешняя** таблица была **малой** — иначе огромное число обращений к внутренней). 2) Планировщик **перебирает** варианты порядка (не все подряд, а по алгоритму динамического программирования) и выбирает порядок с **минимальным cost**. 3) Если порядок получился «плохой» (запрос тормозит, в Nested Loop внешняя таблица огромная) — часто виновата **устаревшая статистика** (оценка rows занижена); решение — ANALYZE по затронутым таблицам.

#### Термины (расшифровка)

- **Порядок JOIN (join order)** — порядок, в котором к текущему результату присоединяются таблицы. Для A JOIN B JOIN C возможны порядки (A,B,C), (A,C,B), (B,A,C) и т.д. **Стоимость** зависит от порядка: например, при Nested Loop выгодно, чтобы **внешняя** (первая) таблица была **малой**, а внутренняя читалась по индексу; при Hash Join порядок может влиять на то, какая таблица идёт в хеш (меньшую выгоднее строить в памяти).
- **Динамическое программирование** — метод перебора: планировщик не перебирает все n! порядков для n таблиц (нереально много), а строит оптимальные подпланы для подмножеств таблиц и комбинирует их. Так число вариантов ограничивается разумным.
- **Оценка rows и выбор порядка:** для каждого варианта порядка планировщик оценивает cost (например, Nested Loop: cost внешней + rows(внешней) × cost доступа во внутреннюю). Порядок с **меньшей оценкой cost** выбирается. Если оценка rows одной таблицы занижена (устаревшая статистика), планировщик может поставить её «внешней» в Nested Loop, думая, что строк мало — на деле их много, и план оказывается медленным.

#### Теория и правила (подробно)

- **Почему порядок важен:** при Nested Loop cost ≈ cost(внешняя) + rows(внешняя) × cost(одна строка внутренней). Если внешняя таблица большая по rows, второй член огромен. При Hash Join одна из сторон строится в хеш — лучше строить хеш по **меньшей** стороне (меньше work_mem и быстрее). Планировщик перебирает порядки и выбирает тот, у которого суммарный cost минимален.
- **Ограничения:** не все порядки допустимы при внешних JOIN (LEFT/RIGHT/FULL) — внутренняя таблица не может быть «внешней» в смысле порядка до того, как присоединена таблица, от которой она зависит. Для сложных запросов планировщик может использовать эвристики или ограниченный перебор.
- **Подсказки (hints):** в PostgreSQL явных подсказок порядка JOIN в ядре нет (есть расширения). В MySQL можно влиять через STRAIGHT_JOIN и др. Обычно исправление статистики (ANALYZE) и параметров стоимости даёт нужный порядок без подсказок.

```mermaid
flowchart TB
  T[Таблицы: A, B, C, ...] --> S1[Посчитать лучшие планы для пар (A⋈B), (A⋈C), ...]
  S1 --> S2[Расширять подпланы:\n(лучший для A⋈B) ⋈ C,\n(лучший для A⋈C) ⋈ B, ...]
  S2 --> Best[Выбрать порядок с минимальным cost]
  Note1[Идея DP: не перебирать n! порядков\nв лоб, а строить оптимум по подмножествам] --- S1
```

#### Пошагово: как порядок JOIN влияет на cost (с числами)

**Шаг 1. Пример: три таблицы.**  
Запрос: `SELECT * FROM small_table s JOIN medium_table m ON s.id = m.sid JOIN big_table b ON m.id = b.mid;` Таблицы: small — 100 строк, medium — 10 000, big — 1 000 000. Варианты порядка: (small, medium, big), (small, big, medium), (medium, small, big) и т.д. Планировщик **не** перебирает все 6 перестановок «в лоб» — их слишком много при большом числе таблиц. Он использует **динамическое программирование**: считает лучший способ соединить пары (small+medium, small+big, medium+big), потом лучший способ добавить третью таблицу к каждой паре и выбирает порядок с **минимальным total cost**.

**Шаг 2. Почему «малая снаружи» выгодна для Nested Loop.**  
Допустим, выбран порядок (small, medium): small — внешняя, medium — внутренняя. Cost Nested Loop ≈ cost(сканировать small) + **100** × cost(одна строка medium по индексу). Если бы порядок был (medium, small): cost ≈ cost(medium) + **10 000** × cost(одна строка small). Второй вариант в **100 раз** дороже по второму слагаемому — потому что для **каждой** строки внешней таблицы выполняется поиск во внутренней. **Итог:** при Nested Loop планировщик старается поставить таблицу с **меньшей оценкой rows** «внешней».

**Шаг 3. Откуда берётся оценка rows.**  
Оценка rows для каждой таблицы и для промежуточных соединений берётся из **статистики** (ANALYZE, pg_stats, селективность условий). Если для medium по старой статистике было **100 строк** вместо 10 000, планировщик мог бы решить: «medium малая» и поставить её внешней в Nested Loop. Тогда реально выполнится 10 000 обращений к внутренней таблице — план будет **очень** медленным. Поэтому **актуальная статистика** критична для правильного порядка JOIN.

#### Что будет, если порядок JOIN выбран из-за устаревшей статистики

Планировщик поставит «малую» (по оценке) таблицу внешней в Nested Loop — а по факту строк в ней **много**. Число итераций внутреннего цикла станет огромным (десятки или сотни тысяч), время выполнения запроса — секунды или минуты вместо миллисекунд. При Hash Join неверная оценка может привести к тому, что в хеш пойдёт **большая** таблица — не хватит work_mem, сброс на диск, замедление. **Итог:** неверный порядок JOIN из-за устаревшей статистики — одна из главных причин «запрос вдруг стал медленным»; решение — **ANALYZE** по затронутым таблицам.

#### Картинка в голове

**Порядок JOIN** — как порядок звонков по делам: сначала позвонить в одно место (мало времени), получить список, потом обзвонить по этому списку. Если список **огромный** (внешняя таблица большая), обзвон займёт вечность. Планировщик старается «сначала малое, потом по малому списку — большое»; для этого ему нужна **правильная** оценка, кто «малый», а кто «большой» — отсюда важность статистики.

#### Простыми словами

**Порядок JOIN** — в каком порядке «присоединять» таблицы. Планировщик перебирает варианты (не все, а по умному алгоритму) и считает cost для каждого. Выбирает порядок с наименьшим cost. Если статистика по одной таблице неверна (мало rows), он может поставить её первой в Nested Loop — «думая», что строк мало; на деле их много — план плохой. Поэтому актуальная статистика важна для выбора порядка.

#### Проверь себя (38.2)

1. Почему при Nested Loop выгодно, чтобы внешняя таблица была малой?  
<details><summary>Ответ</summary> Потому что cost Nested Loop ≈ cost(внешняя) + **rows(внешняя)** × cost(одна строка внутренней). То есть для **каждой** строки внешней таблицы выполняется поиск во внутренней. Если **внешняя** таблица **большая** (много rows), второе слагаемое огромно — общий cost высокий. Если внешняя **малая**, число обращений к внутренней таблице небольшое — cost приемлемый. Поэтому планировщик старается поставить малую таблицу «внешней» при выборе Nested Loop.</details>

2. Как устаревшая статистика может испортить порядок JOIN?  
<details><summary>Ответ</summary> Планировщик оценивает **rows** для каждой таблицы по статистике. Если для одной таблицы rows **занижена** (реально 100 000, оценка 100), он может решить, что эта таблица «малая» и поставить её **внешней** в Nested Loop. Тогда будет 100 000 обращений к внутренней таблице вместо ожидаемых 100 — план окажется очень медленным. **Решение:** обновить статистику (ANALYZE), чтобы оценки rows были ближе к факту и порядок JOIN выбирался корректно.</details>

3. В плане Nested Loop: внешняя таблица — 50 000 rows, внутренняя — Index Scan. Почему запрос может быть медленным и что проверить?  
<details><summary>Ответ</summary> Cost Nested Loop ≈ cost(внешняя) + **50 000** × cost(одна строка внутренней). 50 000 обращений к индексу и таблице — могут дать секунды. **Проверить:** не занижена ли оценка rows внешней таблицы из-за устаревшей статистики (сравнить rows в плане с actual rows после EXPLAIN ANALYZE). Если оценка была сильно меньше (например, планировщик думал 100 строк), причина — устаревшая статистика; выполнить **ANALYZE** по внешней таблице. После обновления статистики планировщик может выбрать Hash Join или другой порядок JOIN с меньшим cost.</details>

**Как запомнить.** Порядок JOIN = порядок «присоединения» таблиц; cost зависит от порядка (Nested Loop: малая снаружи; Hash: меньшая в хеш). Планировщик перебирает варианты и выбирает минимальный cost; неверная оценка rows портит порядок — нужна актуальная статистика (ANALYZE).

**Три пункта, чтобы не забыть (порядок JOIN).** 1) **Порядок JOIN** сильно влияет на cost: при Nested Loop выгодно, чтобы **внешняя** таблица была **малой** (иначе огромное число обращений к внутренней). 2) Планировщик перебирает варианты порядка (динамическое программирование) и выбирает порядок с **минимальным cost**. 3) Если порядок «странный» или запрос резко замедлился — проверить **статистику** (ANALYZE); неверная оценка rows ведёт к плохому порядку JOIN.

#### Запомните

- **Порядок JOIN** влияет на cost (Nested Loop: малая таблица снаружи; Hash Join: меньшая в хеш). Планировщик перебирает варианты и выбирает порядок с минимальным cost.
- Неверная оценка rows (устаревшая статистика) может привести к плохому порядку JOIN; актуальная статистика критична.

**Ещё раз самыми простыми словами:** В каком порядке соединять таблицы — не безразлично: «малую» таблицу выгодно обрабатывать первой (меньше итераций по большой). Планировщик сам выбирает порядок по cost; если он ошибся — часто виновата старая статистика. Сделай ANALYZE по затронутым таблицам.

**Одна фраза (если забыл всё):** Порядок JOIN влияет на cost (Nested Loop: малая снаружи). Планировщик перебирает варианты и выбирает минимальный cost. Плохой порядок часто из-за устаревшей статистики → ANALYZE.

**Если прочитал и всё равно не понял.** Главное одно: **порядок**, в котором присоединяются таблицы (сначала A, потом B, потом C или в другом порядке), **влияет на скорость** — при Nested Loop выгодно, чтобы «малая» таблица была снаружи (первой). Планировщик сам перебирает варианты и выбирает порядок с наименьшим cost. Если он выбрал плохой порядок (запрос тормозит) — чаще всего виновата **устаревшая статистика**: он думал, что в таблице мало строк, а их много. **Что делать:** выполнить **ANALYZE** по затронутым таблицам и повторить запрос. Перечитай блоки **«Пошагово: как порядок JOIN влияет на cost»** (с числами small/medium/big) и **«Картинка в голове»** — там та же мысль.

### 38.3. Подзапросы и материализация

**Цель раздела.**  
Понять, как планировщик обрабатывает **подзапросы**: развёртывание в JOIN (inlining), **материализация** (выполнить один раз и использовать как таблицу), когда что выбирается. После раздела ты будешь по плану различать InitPlan (один раз) и SubPlan (на каждую строку) и понимать, почему коррелированный подзапрос может быть медленным.

**В этом разделе главное (три строки).** 1) **InitPlan** — подзапрос выполняется **один раз** до основного запроса, результат подставляется (типично некоррелированный подзапрос). **SubPlan** — подзапрос выполняется **на каждую строку** внешнего запроса (коррелированный подзапрос) — при большом числе строк это **узкое место**. 2) **Materialize** — подзапрос выполнили один раз, результат сохранили и читают как таблицу; в плане видно узел Materialize. 3) Если в плане **SubPlan с большим loops** — переписать запрос в **JOIN** или **IN** с некоррелированным подзапросом, чтобы была одна общая операция вместо тысяч выполнений.

#### Термины (расшифровка)

- **InitPlan (Initial Plan)** — подплан, который выполняется **один раз** до основного запроса (например, подзапрос в WHERE, не зависящий от внешней таблицы: `WHERE col = (SELECT ...)` с некоррелированным подзапросом). Результат кэшируется и подставляется в основной план. В плане отображается как **InitPlan** с номером.
- **SubPlan (Subquery Plan)** — подплан, который выполняется **для каждой строки** внешнего запроса (или для каждой группы). Типично для **коррелированного** подзапроса (подзапрос ссылается на столбец внешней таблицы). В плане — **SubPlan**; при большом числе строк внешней таблицы такой подплан выполняется много раз — возможное узкое место.
- **Материализация (Materialize)** — узел плана, который выполняет подзапрос **один раз**, сохраняет результат во временную структуру (например, в память или на диск) и затем основной план читает из неё как из таблицы. Выгодно, когда подзапрос тяжёлый и результат используется многократно; без материализации подзапрос мог бы выполняться для каждой строки (SubPlan).

#### Теория и правила (подробно)

- **Некоррелированный подзапрос:** `WHERE user_id = (SELECT id FROM ... LIMIT 1)` — подзапрос не ссылается на строку внешней таблицы. Обычно выполняется **один раз** (InitPlan), результат подставляется. В плане — InitPlan.
- **Коррелированный подзапрос:** `WHERE EXISTS (SELECT 1 FROM orders o WHERE o.user_id = users.id)` — подзапрос зависит от users.id. Для **каждой** строки users выполняется проверка подзапроса — в плане SubPlan. При большом числе строк users — много выполнений; часто выгоднее переписать в **JOIN** или **полу join** (WHERE user_id IN (SELECT ...)), чтобы планировщик мог применить Hash Join или другой способ один раз.
- **Развёртывание в JOIN:** планировщик может **преобразовать** подзапрос в соединение (например, IN (subquery) → semi-join / join с удалением дубликатов). Тогда в плане будет обычный Join, а не SubPlan — как правило, эффективнее.

#### Пошагово: InitPlan vs SubPlan vs Materialize (с числами)

**Шаг 1. Некоррелированный подзапрос — один раз.**  
Запрос: `SELECT * FROM orders WHERE user_id = (SELECT id FROM users WHERE email = 'a@b.com' LIMIT 1);` Подзапрос **не ссылается** на строку из orders — он возвращает одно значение (id пользователя). Планировщик выполняет этот подзапрос **один раз** до основного запроса, кэширует результат (например, 42) и подставляет в условие: `WHERE user_id = 42`. В плане — **InitPlan 1**. Сколько бы строк ни было в orders — подзапрос выполнился **1 раз**. Cost почти не зависит от размера orders.

**Шаг 2. Коррелированный подзапрос — на каждую строку.**  
Запрос: `SELECT * FROM users u WHERE EXISTS (SELECT 1 FROM orders o WHERE o.user_id = u.id);` Здесь подзапрос **зависит** от u.id — для каждой строки users нужно проверить, есть ли заказы с таким user_id. В плане — **SubPlan 1** внутри Nested Loop. Если users содержит **10 000 строк**, подзапрос выполнится **10 000 раз** — по разу на каждую строку. Cost ≈ 10 000 × cost(одно выполнение подзапроса). Это типичное узкое место.

**Шаг 3. Материализация — один раз, потом как таблица.**  
Запрос с подзапросом в FROM или с IN (некоррелированный подзапрос): планировщик может решить **выполнить** подзапрос один раз и сохранить результат во временную структуру (узел **Materialize**). Дальше основной план читает из неё как из таблицы. В плане видно: **Materialize** с дочерним подзапросом. Выгодно, когда подзапрос тяжёлый и результат используется многократно в плане.

#### Что будет, если оставить коррелированный подзапрос (SubPlan) без переписывания

При 100 000 строк во внешней таблице подзапрос выполнится **100 000 раз**. Даже если одно выполнение занимает 0,1 ms, суммарно — **10 секунд** только на подзапрос. На практике одно выполнение может быть тяжелее (сканирование, индекс) — десятки секунд или минуты. **Итог:** коррелированный подзапрос в WHERE/EXISTS при большом числе строк — одна из главных причин «запрос тормозит»; **обязательно** переписать в JOIN или в IN с некоррелированным подзапросом, чтобы планировщик сделал один проход (Hash Join, semi-join) вместо N выполнений.

#### Картинка в голове

**InitPlan** — один раз спросили «какой id у a@b.com?», записали ответ на бумажку и для каждой строки заказов смотришь на бумажку: совпадает — ок. **SubPlan** — для **каждого** человека из списка бежишь в другую комнату и проверяешь «есть у него заказы?» — 10 000 человек = 10 000 походов. **Materialize** — один раз сходил, принёс целую табличку с результатами и положил на стол; дальше только смотришь в эту табличку. Переписывание SubPlan в JOIN — вместо 10 000 походов делаешь **один** общий проход по двум таблицам.

#### Простыми словами

**InitPlan** — подзапрос выполнился **один раз**, результат запомнили и подставили. **SubPlan** — подзапрос выполняется **на каждую строку** внешнего запроса (коррелированный) — может быть очень медленно. **Материализация** — подзапрос выполнили один раз, результат «положили на стол» и дальше читают как таблицу. Если в плане видишь SubPlan с большим числом строк снаружи — стоит переписать запрос в JOIN или IN с некоррелированным подзапросом.

#### Проверь себя (38.3)

1. Чем InitPlan отличается от SubPlan?  
<details><summary>Ответ</summary> **InitPlan** выполняется **один раз** до основного запроса; результат кэшируется и подставляется (например, константа из подзапроса). **SubPlan** выполняется **для каждой строки** (или группы) внешнего запроса — типично для **коррелированного** подзапроса. SubPlan при большом числе строк снаружи может стать узким местом; InitPlan выполняется один раз и на стоимость почти не влияет при большом объёме данных.</details>

2. Почему коррелированный подзапрос в WHERE часто медленный и как улучшить?  
<details><summary>Ответ</summary> Коррелированный подзапрос **зависит от строки** внешней таблицы — для каждой такой строки подзапрос выполняется заново (SubPlan). При 10 000 строк снаружи подзапрос выполнится 10 000 раз — очень медленно. **Улучшение:** переписать в **JOIN** (например, JOIN orders o ON o.user_id = users.id) или в **EXISTS/IN** с таким подзапросом, который планировщик сможет развернуть в один проход (semi-join, hash join). Тогда вместо 10 000 выполнений подзапроса будет одно соединение.</details>

3. В плане узел Materialize с дочерним подзапросом. Что это значит?  
<details><summary>Ответ</summary> **Materialize** — подзапрос выполняется **один раз**, результат сохраняется во временную структуру (память или диск), и основной план дальше читает из неё как из таблицы. Так планировщик избегает многократного выполнения одного и того же подзапроса (например, в JOIN или при повторном использовании в плане). Это нормально и часто выгодно — не нужно «исправлять», если запрос быстрый.</details>

**Как запомнить.** InitPlan = один раз; SubPlan = на каждую строку (коррелированный — подзапрос ссылается на внешнюю таблицу). SubPlan при большом числе строк — узкое место; переписать в JOIN или IN с некоррелированным подзапросом. Materialize — выполнить подзапрос один раз и читать как таблицу.

**Три пункта, чтобы не забыть (подзапросы).** 1) **InitPlan** выполняется **один раз** (некоррелированный подзапрос); **SubPlan** — **на каждую строку** (коррелированный). 2) В плане с **SubPlan** и большим числом строк снаружи — запрос может быть очень медленным (подзапрос выполняется тысячи раз). 3) **Решение:** переписать коррелированный подзапрос в **JOIN** или в **IN (некоррелированный подзапрос)**; тогда планировщик сделает один проход вместо N выполнений.

#### Запомните

- **InitPlan** — подзапрос один раз; **SubPlan** — на каждую строку (коррелированный); **Materialize** — выполнить подзапрос один раз и читать результат как таблицу.
- Коррелированный подзапрос (SubPlan) при большом числе строк — типичное узкое место; переписать в JOIN или некоррелированный IN/EXISTS.

**Ещё раз самыми простыми словами:** Подзапрос «один раз» (InitPlan) — норма. Подзапрос «на каждую строку» (SubPlan) — когда он завязан на внешнюю таблицу — при большом списке строк очень медленно. Не пиши так: переделай в один запрос с JOIN или IN по списку из подзапроса, чтобы база сделала одну общую операцию вместо тысяч маленьких.

**Одна фраза (если забыл всё):** InitPlan — один раз; SubPlan — на каждую строку (коррелированный). SubPlan с большим числом строк — узкое место; переписать в JOIN или IN с некоррелированным подзапросом.

### 38.4. Планы с параметрами

**Цель раздела.**  
Понять особенность **подготовленных запросов (prepared statements)** и **generic plan**: план может кэшироваться и переиспользоваться с **разными** значениями параметров; при первом наборе параметров выбранный план может быть оптимальным, при других — неоптимальным. После раздела ты будешь знать, когда подозревать «parameter sniffing» и что делать (переподготовка, отключение generic plan, локальная настройка).

**В этом разделе главное (три строки).** 1) При **подготовленном запросе** (с параметрами $1, $2) СУБД может использовать **один и тот же план** (generic plan) для **всех** значений параметров — план строится «в среднем», без привязки к конкретным значениям. 2) Для **части** значений такой план может быть **неоптимальным** — один и тот же запрос то выполняется за миллисекунды, то за секунды в зависимости от параметра. 3) **Решения:** не использовать prepared statement для этого запроса; в PostgreSQL 12+ **plan_cache_mode = 'force_custom_plan'** (каждый раз планировать с учётом текущих параметров); или переписать запрос так, чтобы один план подходил всем случаям.

#### Термины (расшифровка)

- **Prepared statement (подготовленное выражение)** — запрос с **параметрами** (плейсхолдеры $1, $2 или ?), который компилируется (парсинг + планирование) **один раз**, а выполняется многократно с разными значениями параметров. СУБД может **кэшировать план** и не перепланировать при каждом выполнении — экономия CPU.
- **Generic plan** — в PostgreSQL: план, построенный **без привязки** к конкретным значениям параметров (значения считаются «неизвестными»). Планировщик использует средние оценки селективности (например, 1/n_distinct для равенства). Такой план **один** на все выполнения с разными параметрами. При первом выполнении с конкретными параметрами может выбираться generic plan или custom plan (см. ниже).
- **Custom plan** — план, построенный для **конкретных** значений параметров (при первом выполнении или когда планировщик решает перепланировать). Учитывает селективность именно этих значений (например, частоту из MCV). Может быть оптимальнее generic для данных параметров, но при каждом новом наборе параметров план пересчитывается (или кэшируется по значениям, в зависимости от СУБД и настроек).
- **Parameter sniffing (в других СУБД)** — ситуация, когда план строится под **первые** переданные значения параметров и кэшируется; при других значениях план оказывается неоптимальным (например, первый раз передали редкое значение — выбран индекс; потом передают частое — индекс невыгоден, но план тот же). В PostgreSQL с generic plan «sniffing» нет — план строится без значений; но generic план может быть средним и плохим для крайних значений.

#### Теория и правила (подробно)

- **PostgreSQL:** при подготовленном запросе первые несколько выполнений (по умолчанию 5) строят **custom plan** для конкретных параметров; сравнивается cost custom и **generic** плана. Если generic стабильно дешевле или не сильно дороже, дальше используется **generic plan** для всех выполнений. Поэтому при «плохих» параметрах (например, частое значение — полный скан выгоден) первые выполнения могли выбрать custom plan с полным сканном; потом переключились на generic с индексом — или наоборот. Поведение можно менять (plan_cache_mode в PostgreSQL 12+).
- **Когда страдает производительность:** запрос с одним параметром (например, user_id). Для user_id=1 (миллион заказов) выгоден seq scan; для user_id=2 (один заказ) выгоден index scan. Generic plan «средний» — может выбрать один вариант для всех; при части выполнений план будет неоптимальным. Решение: переписать запрос без подготовки для «критичных» путей; или использовать plan_cache_mode = 'force_custom_plan' для этого запроса (PostgreSQL 12+), чтобы всегда планировать с учётом текущих параметров (без кэша generic).
- **Партиционирование:** при подготовленном запросе с условием по ключу партиционирования generic plan может не знать, в какую партицию пойдёт запрос — все партиции в плане или выбор откладывается до выполнения (runtime pruning). Тогда план может быть консервативным.

#### Пошагово: как generic plan приводит к «то быстро, то медленно» (с числами)

**Шаг 1. Подготовленный запрос с одним параметром.**  
Приложение готовит запрос: `SELECT * FROM orders WHERE user_id = $1;` и выполняет его много раз с разными user_id. В PostgreSQL первые **5 выполнений** (по умолчанию) строят **custom plan** для конкретных значений и сравнивают его cost с **generic plan** (план без привязки к значению — параметр «неизвестный»). Если generic plan выигрывает или близок, с 6-го выполнения **всегда** используется generic plan для всех вызовов — перепланирования нет.

**Шаг 2. Почему generic plan может быть плох для части значений.**  
Для user_id = 1 в таблице **1 млн заказов** — выгоден **Seq Scan** (полный скан дешевле индекса при такой селективности). Для user_id = 99999 — **3 заказа** — выгоден **Index Scan**. Generic plan строится при «неизвестном» параметре: планировщик использует **усреднённую** селективность (например, 1/n_distinct). Он может выбрать один вариант — допустим, Index Scan. Тогда для user_id = 1 запрос будет **медленным** (индекс по 1 млн строк вместо полного скана); для user_id = 99999 — быстрым. **Итог:** один и тот же запрос с разными параметрами то 1 ms, то 5 s.

**Шаг 3. Что делать.**  
Вариант 1: **не использовать** prepared statement для этого запроса — каждый раз план строится с учётом текущего значения (если приложение передаёт значение как литерал или переподготавливает запрос). Вариант 2: в PostgreSQL 12+ **plan_cache_mode = 'force_custom_plan'** для сессии или для этого запроса — планировщик всегда строит план с учётом текущих параметров (кэш generic не используется). Вариант 3: переписать запрос так, чтобы один план был приемлем для всех (например, ограничить выборку LIMIT и всегда использовать индекс).

#### Что будет, если игнорировать «то быстро, то медленно» при подготовленных запросах

Часть пользователей (или часть вызовов API) будут получать ответ за миллисекунды, другая часть — за секунды. Нагрузка на БД будет «пиковая» при тех параметрах, для которых generic plan плох (например, частые user_id с большим числом заказов — каждый раз тяжёлый Index Scan). **Итог:** при использовании prepared statements с параметрами, сильно влияющими на селективность (user_id, tenant_id, category_id), стоит проверить планы для «крайних» значений и при необходимости включить force_custom_plan или отказаться от кэша плана для этого запроса.

#### Картинка в голове

**Generic plan** — как один ключ «под среднюю дверь»: открывает много дверей, но для очень маленькой или очень большой двери подходит плохо. **Custom plan** — ключ под конкретную дверь (текущее значение параметра). Если приложение всегда ходит то в маленькую, то в большую дверь, а ключ один «средний» — часть походов будет мучительной. Решение: либо каждый раз делать ключ под текущую дверь (force_custom_plan), либо не использовать один ключ для всех (не prepared statement для этого запроса).

#### Простыми словами

При **подготовленном запросе** план может кэшироваться и использоваться для **всех** значений параметров (generic plan). Такой план «усреднённый» — для одних значений он хорош, для других плох. Если видишь, что один и тот же запрос то быстрый, то медленный в зависимости от параметров — возможно, кэшированный план не подходит под часть значений. Решения: не использовать подготовленный запрос для этого конкретного запроса; принудительно перепланировать (force_custom_plan в PG 12+); или переписать запрос так, чтобы один план был хорош для всех случаев.

#### Проверь себя (38.4)

1. Что такое generic plan в контексте подготовленных запросов?  
<details><summary>Ответ</summary> **Generic plan** — план, построенный **без привязки к конкретным значениям** параметров (параметры считаются «неизвестными»). Минус: для некоторых значений параметров план может быть неоптимальным (например, для частого значения выгоден seq scan, а generic выбрал index scan).</details>

Для одних user_id этот план хорош (мало строк — index scan), для других плох (много строк — но план всё равно с index scan или наоборот). **Решения:** принудительно перепланировать для каждого набора параметров (plan_cache_mode = 'force_custom_plan' в PostgreSQL 12+ для этого запроса); или не использовать prepared statement для этого запроса; или переписать запрос так, чтобы один план был приемлем для всех (например, всегда использовать индекс и ограничивать выборку).</details>

В **PostgreSQL** с **generic plan** «sniffing» в таком виде нет: generic plan строится **без** конкретных значений (параметр «неизвестный»). Но результат похожий: один усреднённый план для всех значений — для части параметров он может быть плох. **Как запомнить.** Подготовленный запрос может кэшировать **generic plan** — один план для всех значений параметров; он «усреднён» и для части значений неоптимален. Запрос «то быстро, то медленно» при разных параметрах — типичный признак; решение: force_custom_plan, отказ от prepared statement для этого запроса или переписывание запроса.

**Три пункта, чтобы не забыть (планы с параметрами).** 1) **Generic plan** — план без привязки к значениям параметров; один план на все выполнения подготовленного запроса; может быть неоптимален для части значений. 2) Если один и тот же запрос с **разными параметрами** то быстрый, то очень медленный — причина часто в кэшированном (generic) плане. 3) **Решения:** в PG 12+ **plan_cache_mode = 'force_custom_plan'**; или не использовать prepared statement для этого запроса; или переписать запрос так, чтобы один план подходил для всех случаев.

#### Запомните

- **Подготовленный запрос** может использовать **generic plan** (один план для всех значений параметров); план «усреднён» и может быть неоптимален для части значений.

**Ещё раз самыми простыми словами:** Один и тот же запрос (с параметром) выполняется то мгновенно, то минуту — часто виноват «усреднённый» кэшированный план: он хорош для одних значений и плох для других. Либо заставь базу каждый раз строить план под текущее значение (force_custom_plan), либо не кэшировать план для этого запроса, либо переписать запрос так, чтобы один план подходил всем.

**Одна фраза (если забыл всё):** Generic plan — один план для всех значений параметров; для части значений может быть плох. Запрос «то быстро, то медленно» → force_custom_plan или не использовать prepared statement для этого запроса.

### 38.5. Партиционирование и планы

**Цель раздела.**  
Понять, как планировщик работает с **партиционированными таблицами**: отсечение партиций (partition pruning), когда оно возможно, как это отображается в плане и почему при подготовленных запросах с параметрами по ключу партиционирования план может быть консервативным. После раздела ты будешь по плану видеть обращение к одной или нескольким партициям и понимать, когда ожидать pruning.

**В этом разделе главное (три строки).** 1) **Partition pruning** — планировщик отбрасывает партиции, которые **заведомо не содержат** строк по условию (например, WHERE date_col = '2024-01-15' при партиционировании по месяцам — читаем только партицию за январь 2024). 2) В плане видно **Append** (или **Merge Append**) над дочерними узлами — по одному на партицию; при pruning дочерних узлов **меньше** — обращаемся не ко всей таблице, а только к отобранным партициям. 3) При **подготовленном запросе** с параметром по ключу партиционирования (WHERE partition_key = $1) generic plan может **не знать** значение $1 — тогда в плане могут фигурировать **все** партиции или выбор откладывается на выполнение (**runtime pruning**); при выполнении лишние партиции не сканируются, но план выглядит тяжёлым.

#### Термины (расшифровка)

- **Партиционированная таблица** — логическая таблица разбита на **партиции** (дочерние таблицы) по диапазону или списку значений ключевого столбца (partition key). Запрос к логической таблице планировщик превращает в объединение (Append) планов по партициям.
- **Partition pruning (отсечение партиций)** — исключение из плана партиций, которые **не могут** содержать строк по условию WHERE. Например, таблица партиционирована по created_at по месяцам; условие WHERE created_at BETWEEN '2024-01-01' AND '2024-01-31' позволяет оставить в плане только партицию за январь 2024. Меньше партиций — меньше I/O и быстрее запрос.
- **Static pruning** — отсечение на этапе **планирования**: значение в условии известно (константа или литерал). **Runtime pruning** — отсечение на этапе **выполнения**, когда значение приходит параметром ($1); для каждого выполнения отсекаются партиции по фактическому значению параметра.
- **Append / Merge Append** — узел плана, объединяющий результаты от нескольких дочерних узлов (по одному на партицию). При pruning дочерних узлов меньше — в плане видно не все партиции.

#### Теория и правила (подробно)

- **Когда возможен pruning:** условие по **ключу партиционирования** (или по его части для составного ключа) позволяет планировщику или исполняющей среде определить, какие партиции не содержат подходящих строк. Для **RANGE** — сравнения =, <, <=, >, >=, BETWEEN (например, WHERE created_at BETWEEN '2024-01-01' AND '2024-01-31' при партициях по месяцам). Для **LIST** — равенство или **IN** (например, WHERE region IN ('EU', 'US') при партициях по списку регионов). Составной ключ: условие может затрагивать только часть столбцов — тогда отсекаются партиции по тем столбцам, по которым условие задано.
- **В плане:** узел **Append** (или **Merge Append**, если порядок важен) с дочерними узлами **Seq Scan** / **Index Scan** по каждой партиции. При успешном pruning дочерних узлов **меньше** (только релевантные партиции). В PostgreSQL 14+ в EXPLAIN (ANALYZE) для партиционированных таблиц может выводиться **Partitions removed: N** — сколько партиций отсечено (static или runtime).
- **Подготовленный запрос:** при запросе с параметром по ключу партиционирования (WHERE partition_key = $1) на этапе планирования значение $1 неизвестно — **static pruning** невозможен. Планировщик может заложить в план **все** партиции; при **выполнении** (runtime pruning) обращение идёт только к нужной партиции. Поэтому план по тексту может выглядеть «тяжёлым», но фактическое выполнение — по одной партиции.

#### Пошагово: как выглядит партиционирование в плане (с числами)

**Шаг 1. Запрос с константой по ключу партиционирования.**  
Таблица orders партиционирована по **created_at** по месяцам (партиции orders_2024_01, orders_2024_02, …). Запрос: `SELECT * FROM orders WHERE created_at BETWEEN '2024-01-01' AND '2024-01-31';` Планировщик на этапе планирования знает диапазон — выполняет **static pruning** и оставляет в плане только партицию за январь 2024. В плане: **Append** → дочерний узел **Seq Scan on orders_2024_01** (или Index Scan по этой партиции). Остальные партиции в плане **не фигурируют** — не читаем их, запрос быстрее.

**Шаг 2. Запрос с параметром (prepared statement).**  
Запрос: `SELECT * FROM orders WHERE created_at = $1;` При **generic plan** значение $1 неизвестно — static pruning не может отбросить партиции. В плане может быть **Append** с дочерними узлами по **всем** партициям (много узлов). При **выполнении** с конкретным значением (например, '2024-01-15') исполняющая среда делает **runtime pruning** — фактически сканируется только партиция orders_2024_01. В EXPLAIN ANALYZE с подставленным значением видно, что строки вернула одна партиция; в плане без выполнения (EXPLAIN без ANALYZE) или при выводе generic плана — все партиции. Это не ошибка: для корректной работы с разными $1 план должен учитывать все партиции; отсечение происходит в runtime.

**Шаг 3. Зачем это знать.**  
Если в плане по партиционированной таблице видно много дочерних узлов Append — проверь, есть ли условие по ключу партиционирования и известно ли оно на этапе планирования. При константе в запросе ожидай **меньше** партиций в плане (static pruning). При параметре — все партиции в плане при планировании нормальны; смотри **actual rows** по узлам в EXPLAIN ANALYZE — по факту активна обычно одна партиция.

#### Часто путают

**«В плане с параметром по ключу партиционирования всегда сканируются все партиции»** — неверно. В **тексте плана** (EXPLAIN без ANALYZE или generic plan) могут фигурировать **все** партиции, потому что значение параметра неизвестно на этапе планирования. Но при **выполнении** (runtime pruning) СУБД отсекает лишние партиции по фактическому значению параметра — **сканируется только нужная партиция**. В EXPLAIN ANALYZE с подставленным значением по **actual rows** видно, что строки вернула одна партиция. Путать «все партиции в плане» с «все партиции читаются» не стоит — при runtime pruning читаются только релевантные.

#### Проверь себя (38.5)

1. Таблица партиционирована по дате по месяцам. Запрос: WHERE created_at = '2024-06-15'. Сколько партиций попадёт в план при static pruning?  
<details><summary>Ответ</summary> При **static pruning** значение известно на этапе планирования — в план попадёт **только одна** партиция (за июнь 2024). Остальные партиции отсекаются: условие created_at = '2024-06-15' однозначно указывает на партицию за этот месяц.</details>

2. Запрос с параметром: WHERE partition_key = $1. В плане видно Append с дочерними узлами по всем партициям. Значит ли это, что при выполнении читаются все партиции?  
<details><summary>Ответ</summary> **Нет.** При параметре по ключу партиционирования на этапе планирования значение $1 неизвестно — в плане могут быть **все** партиции. При **выполнении** срабатывает **runtime pruning**: по фактическому значению $1 отсекаются лишние партиции, и **сканируется только одна** (или те, куда попадает значение). В EXPLAIN ANALYZE по actual rows видно, что строки вернула одна партиция.</details>

3. Для каких условий по ключу партиционирования возможен pruning при RANGE и при LIST?  
<details><summary>Ответ</summary> Для **RANGE**: сравнения =, <, <=, >, >=, **BETWEEN** — планировщик или исполняющая среда определяет, какие диапазоны партиций не содержат подходящих строк. Для **LIST**: **равенство** (=) и **IN** (список значений) — отсекаются партиции, в чей список значений не входит условие.</details>

#### Простыми словами

**Партиционирование** — таблица разбита на части (партиции) по ключу. **Partition pruning** — «не трогаем партиции, в которых по условию точно ничего нет». В плане видно **Append** и список дочерних сканов по партициям; при хорошем pruning дочерних узлов мало. При запросе с **параметром** по ключу партиционирования план может содержать все партиции (generic plan), а отсечение происходит при выполнении (runtime pruning).

#### Как запомнить

**Partition pruning** — отбрасываем партиции, не подходящие по условию; в плане меньше дочерних узлов Append. **Static pruning** — при известном значении в WHERE на этапе планирования. **Runtime pruning** — при параметре ($1) отсечение при выполнении. По плану партиционированной таблицы смотри: сколько партиций в Append и есть ли условие по ключу партиционирования.

#### Запомните

- Партиционированная таблица в плане — узел **Append** (или Merge Append) с дочерними узлами по партициям. **Partition pruning** уменьшает число сканируемых партиций.
- При константе в условии по ключу партиционирования — **static pruning**, в плане только нужные партиции. При параметре — возможен **runtime pruning**; в generic плане могут фигурировать все партиции.

**Одна фраза (если забыл всё):** Партиционирование в плане — Append по партициям; partition pruning отбрасывает лишние партиции. При параметре по ключу — runtime pruning, в плане могут быть все партиции.
