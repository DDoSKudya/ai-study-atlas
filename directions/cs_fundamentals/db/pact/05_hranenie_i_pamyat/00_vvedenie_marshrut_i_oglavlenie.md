## Часть V. Хранение и память

> Цель этой части — **понять, как СУБД физически хранит данные на диске и как использует память**: что такое страница и блок, как устроены файлы данных и табличные пространства, чем row-oriented отличается от columnar; как работает буферный пул (кэш страниц), checkpoint и read-ahead; зачем нужен WAL и как по нему восстанавливаться; как устроен формат строки (heap, заголовок, NULL bitmap, TOAST, HOT, fillfactor); что такое undo log, visibility map и FSM; как оценивать размер таблиц и индексов и настраивать память (shared_buffers, work_mem); как устроен дисковый I/O и зачем уменьшать random_page_cost на SSD.

После этой части ты сможешь:

- **понимать физическое хранение** — страница (page/block), файлы данных, табличные пространства (tablespaces), row-oriented vs columnar;
- **разбираться в буферном пуле** — кэш страниц в RAM, dirty pages, замена (LRU/clock-sweep), checkpoint, read-ahead, настройка размера;
- **понимать WAL/redo** — write-ahead logging, LSN, full page write, восстановление после сбоя, doublewrite в InnoDB;
- **знать формат хранения строк** — heap, заголовок строки (xmin/xmax, NULL bitmap), TOAST, HOT, fillfactor;
- **понимать служебные структуры** — undo log (InnoDB), visibility map, FSM (free space map), сжатие страниц;
- **оценивать и настраивать память** — размер таблицы/индекса, shared_buffers, work_mem, maintenance_work_mem, риск OOM;
- **понимать дисковый I/O** — последовательное vs случайное чтение, random_page_cost, SSD, очереди I/O.

Материал опирается на **Часть IV (транзакции, MVCC)** и готовит к темам индексов и оптимизации. Термины расшифрованы в начале разделов; одна и та же идея повторяется в «Теория», «Пошагово», «Простыми словами», «Картинка в голове», «Запомните» и «Проверь себя» — так удобнее усвоить. Иди по оглавлению по этапам; по ссылке переходи в раздел; в конце части — справочник, вопросы для самопроверки и резюме.

### Краткая шпаргалка по части

| Понятие / конструкция | Суть (что это и когда нужно) |
|----------------------|-----------------------------|
| **Страница (page, block)** | Единица хранения на диске и в памяти; обычно 8 KB (PostgreSQL), 16 KB (InnoDB) |
| **Табличное пространство (tablespace)** | Логическая группа файлов данных; размещение на разных дисках; CREATE TABLESPACE |
| **Row-oriented** | Строки хранятся подряд (вся строка в одной странице или продолжение); типично для OLTP |
| **Columnar** | Значения одного столбца хранятся вместе; сжатие, аналитика (OLAP) |
| **Буферный пул (buffer pool)** | Кэш страниц в RAM; buffer pool (InnoDB), shared_buffers (PostgreSQL) |
| **Грязная страница (dirty page)** | Страница в буфере, изменённая, но ещё не сброшенная на диск |
| **Checkpoint** | Момент сброса грязных страниц на диск; уменьшает объём WAL при восстановлении |
| **Read-ahead** | Предзагрузка следующих страниц при последовательном чтении |
| **WAL (Write-Ahead Log)** | Журнал: запись изменений в WAL до сброса страниц на диск; основа долговечности |
| **LSN (Log Sequence Number)** | Позиция в WAL; порядок записей; используется при восстановлении |
| **Full page write** | Первая модификация страницы после checkpoint — в WAL пишется полная страница (защита от partial write) |
| **Doublewrite (InnoDB)** | Буфер/файл для записи страниц перед основной записью; защита от partial page write |
| **Heap** | Хранилище строк таблицы; указатели из индексов (TID) ведут на строку в heap |
| **TID (Tuple ID)** | Идентификатор строки: номер страницы + смещение в странице |
| **TOAST** | Хранение больших значений (>2KB) вне основной строки; PLAIN/EXTENDED/EXTERNAL/MAIN |
| **HOT (Heap Only Tuple)** | Обновление без изменения индексов, если меняемые столбцы не в индексах |
| **Fillfactor** | Доля страницы, оставляемая под обновления; остальное — свободное место для HOT |
| **Visibility map** | Битовая карта: страницы, где все строки видимы всем; ускоряет VACUUM и index-only scan |
| **FSM (Free Space Map)** | Карта свободного места в страницах; куда вставить новую строку |
| **shared_buffers** | Размер кэша страниц в PostgreSQL (основной потребитель RAM) |
| **work_mem** | Память на операцию (сортировка, хеш) на узел плана в PostgreSQL |
| **maintenance_work_mem** | Память для VACUUM, CREATE INDEX и т.п. в PostgreSQL |
| **random_page_cost** | Стоимость случайного чтения страницы относительно последовательного; для SSD уменьшают |
| **Последовательное vs случайное I/O** | Sequential: подряд блоки; random: вразнобой; на HDD random медленнее |

Используй таблицу как карту: нашёл понятие в первом столбце → во втором видишь суть и типичное применение. Для повторения после изучения раздела — сверяйся с этой шпаргалкой.

### Маршрут изучения в этой части

1. **Физическое хранение (раздел 22)**  
   Страница как единица хранения, файлы данных, табличные пространства, row-oriented и columnar. Влияние размера страницы на I/O и размер строки.

2. **Буферный пул (раздел 23)**  
   Кэш страниц в памяти, чтение/запись через буфер, грязные страницы, замена (LRU, clock-sweep), checkpoint, read-ahead, настройка размера.

3. **WAL / redo (раздел 24)**  
   Write-ahead: запись в журнал до сброса страниц. LSN, full page write, сегменты WAL, восстановление. Doublewrite в InnoDB, undo log отдельно.

4. **Формат хранения строк (раздел 25)**  
   Heap, заголовок строки (метаданные, xmin/xmax, NULL bitmap), TOAST для больших значений, HOT и fillfactor, ограничения размера строки.

5. **Undo log, visibility map, FSM (раздел 26)**  
   Undo log в InnoDB для отката и MVCC; visibility map и FSM в PostgreSQL; сжатие страниц.

6. **Память: оценка и настройка (раздел 27)**  
   Оценка размера таблицы и индекса; shared_buffers, work_mem, maintenance_work_mem; InnoDB buffer_pool_size; риск OOM.

7. **Дисковый I/O (раздел 28)**  
   Последовательное vs случайное чтение, влияние на производительность, random_page_cost для SSD, очереди I/O, мониторинг.

### Частые сценарии: задача → решение

| Тема | Задача | Решение |
|------|--------|---------|
| **Хранение** | Разместить таблицу на другом диске | CREATE TABLESPACE … LOCATION '…'; CREATE TABLE … TABLESPACE … |
| **Хранение** | Оценить размер таблицы до загрузки | Число строк × средний размер строки + выравнивание и заголовки страниц |
| **Буфер** | Уменьшить чтение с диска | Увеличить shared_buffers (PostgreSQL) или buffer_pool_size (InnoDB) в разумных пределах |
| **Буфер** | Частые checkpoint и лаги | Увеличить checkpoint_timeout; настроить checkpoint_completion_target |
| **WAL** | Быстрее запись, готовность к риску потери | synchronous_commit = off (осторожно: риск потери последних секунд при сбое) |
| **Строки** | Много обновлений без изменения индексов | Fillfactor < 100; столбцы, по которым не ищем, не включать в индексы (HOT) |
| **Строки** | Очень большие значения в столбце | TOAST: по умолчанию EXTENDED; сжатие и вынос в отдельное хранилище |
| **Память** | Сортировка/хеш съедают память | work_mem; при превышении — сброс на диск (temp files), риск OOM при большой работе |
| **I/O** | После перехода на SSD запросы всё ещё «дорогие» | Уменьшить random_page_cost (например, 1.1) для SSD |
