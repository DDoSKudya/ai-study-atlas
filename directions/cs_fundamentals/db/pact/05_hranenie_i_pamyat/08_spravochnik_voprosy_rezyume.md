## Вопросы для самопроверки (часть V)

1. Что такое страница (page) и почему чтение/запись идут целыми страницами?  
<details><summary>Ответ</summary> Страница — минимальная единица хранения в СУБД (обычно 8 KB в PostgreSQL, 16 KB в InnoDB). Чтение и запись идут целыми страницами, потому что диск и ОС эффективно работают блоками, а не отдельными байтами; так уменьшается число обращений к диску и упрощается управление буфером.</details>

2. Зачем нужен табличное пространство (tablespace)?  
<details><summary>Ответ</summary> Чтобы размещать таблицы и индексы на разных дисках или разделах: разнести I/O, вынести «горячие» объекты на SSD, а редко используемые оставить на HDD.</details>

3. В чём разница между row-oriented и columnar хранением?  
<details><summary>Ответ</summary> Row-oriented: строки хранятся подряд (вся строка целиком). Columnar: значения одного столбца хранятся вместе. Row-oriented удобен для OLTP (доступ по ключу, обновление строки); columnar — для OLAP (агрегаты по столбцам, сжатие).</details>

4. Что такое грязная страница и когда она сбрасывается на диск?  
<details><summary>Ответ</summary> Грязная страница — страница в буфере, изменённая после загрузки с диска и ещё не записанная обратно. Сбрасывается при checkpoint или при вытеснении из буфера по политике замены (LRU/clock-sweep); до сброса изменения должны быть записаны в WAL.</details>

5. Сформулируйте правило Write-Ahead. Зачем оно нужно?  
<details><summary>Ответ</summary> Правило Write-Ahead: запись в WAL (описание изменений) должна быть выполнена и сброшена на диск до записи самой изменённой страницы на диск. Нужно для долговечности: при сбое мы можем восстановить страницу, воспроизведя WAL; без этого при сбое между записью страницы и WAL данные могли бы быть потеряны или несогласованны.</details>

6. Что такое full page write и от какой проблемы он защищает?  
<details><summary>Ответ</summary> Full page write — при первой модификации страницы после checkpoint в WAL записывается полная копия страницы. Защищает от partial page write: если диск/ОС записали только часть страницы (например, из-за сбоя питания), страница на диске испорчена; по полной копии из WAL её можно восстановить и применить к ней последующие инкрементальные записи.</details>

7. Что такое heap и TID? Как они связаны с индексами?  
<details><summary>Ответ</summary> Heap — хранилище строк таблицы (куча). TID — идентификатор строки (номер страницы + смещение в странице). Индекс хранит ключ и TID; по TID СУБД находит строку в heap и читает её. При index-only scan, если страница помечена в visibility map как «все видимы», обращение к heap можно не делать.</details>

8. Как хранятся NULL в строке?  
<details><summary>Ответ</summary> NULL не хранится как значение в теле строки; в заголовке строки (или в NULL bitmap) для каждого nullable столбца есть бит: 1 = NULL, 0 = значение присутствует. Так экономится место.</details>

9. Что такое TOAST и когда он используется?  
<details><summary>Ответ</summary> TOAST — вынос больших значений (в PostgreSQL примерно > 2 KB) в отдельную TOAST-таблицу; в основной строке остаётся указатель. Используется для типов вроде text, bytea, jsonb при превышении порога размера. Стратегии: EXTENDED (сжатие + вынос), EXTERNAL (вынос без сжатия), MAIN, PLAIN.</details>

10. При каком условии обновление в PostgreSQL будет HOT?  
<details><summary>Ответ</summary> Новая версия строки помещается на ту же страницу (есть свободное место, в том числе за счёт fillfactor), и ни один из изменённых столбцов не входит ни в один индекс. Тогда индексы не обновляются — все по-прежнему указывают на старый TID, цепочка версий на странице ведёт к новой версии.</details>

11. Для чего нужны visibility map и FSM?  
<details><summary>Ответ</summary> Visibility map: бит на страницу «все строки на странице видимы всем»; ускоряет VACUUM (можно пропускать такие страницы) и даёт возможность index-only scan без обращения к heap. FSM: карта свободного места по страницам; при вставке по FSM выбирается страница с достаточным свободным местом.</details>

12. Что такое shared_buffers и work_mem? Чем опасен слишком большой work_mem при многих соединениях?  
<details><summary>Ответ</summary> shared_buffers — кэш страниц в PostgreSQL. work_mem — максимальная память на одну операцию (сортировка, хеш) в узле плана. Один запрос может иметь несколько таких узлов, и каждый узел может использовать до work_mem; при многих одновременных запросах суммарное потребление work_mem может привести к OOM.</details>

13. Почему на SSD рекомендуется уменьшать random_page_cost?  
<details><summary>Ответ</summary> На HDD случайное чтение значительно медленнее последовательного (позиционирование головки); по умолчанию random_page_cost = 4. На SSD разница между random и sequential мала; если оставить 4, планировщик переоценивает стоимость index scan и чаще выбирает sequential scan там, где index scan был бы быстрее. Уменьшение до 1.1–1.2 даёт более точную модель для SSD.</details>

## Резюме части V

- **Физическое хранение:** страница (page/block) — единица хранения; файлы данных; табличные пространства для размещения на разных дисках; row-oriented (OLTP) vs columnar (OLAP).
- **Буферный пул:** кэш страниц в RAM; чтение из буфера, при промахе — с диска; грязные страницы, checkpoint, замена (LRU/clock-sweep), read-ahead; настройка shared_buffers / buffer_pool_size.
- **WAL/redo:** Write-Ahead — запись в журнал до сброса страниц; LSN, сегменты WAL; full page write; восстановление = checkpoint + применение WAL; doublewrite в InnoDB; undo log отдельно для отката и MVCC.
- **Формат строк:** heap, TID; заголовок строки (xmin, xmax, NULL bitmap); TOAST для больших значений; HOT и fillfactor.
- **Служебные структуры:** undo log (InnoDB); visibility map и FSM в PostgreSQL; сжатие страниц.
- **Память:** оценка размера таблицы и индекса; shared_buffers, work_mem, maintenance_work_mem; риск OOM, temp files.
- **Дисковый I/O:** последовательное vs случайное чтение; random_page_cost для SSD (1.1–1.2); effective_io_concurrency; мониторинг wait events и статистики I/O.

**Связь с частью VI (Индексы):** понимание heap и TID, visibility map и index-only scan, размера индекса и влияния на память и I/O понадобится при изучении B-tree, GIN, BRIN и обслуживания индексов.
