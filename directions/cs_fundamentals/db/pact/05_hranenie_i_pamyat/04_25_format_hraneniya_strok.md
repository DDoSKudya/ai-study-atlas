## 25. Формат хранения строк

### 25.1. Heap и TID

**Цель раздела.**  
Понять, что данные таблицы хранятся в **heap** (куче строк), а индексы содержат **указатели на строки** — TID (Tuple ID).

#### Термины

- **Heap** — хранилище строк таблицы без гарантии физического порядка. Строки лежат на страницах; порядок вставки и порядок по значению ключа могут не совпадать. В PostgreSQL каждая обычная таблица имеет один heap.
- **TID (Tuple ID)** — идентификатор строки в heap: номер страницы (block number) + смещение внутри страницы (offset). Индекс хранит ключ и TID; по TID СУБД находит строку на странице и читает её.

#### Теория и правила

- При доступе по индексу: по ключу находится запись в индексе → из записи берётся TID → по TID читается страница heap и строка по смещению. Это **index scan**. Если все нужные столбцы есть в индексе и страница помечена в visibility map как «все видимы», возможен **index-only scan** без обращения к heap.
- Heap позволяет вставлять строки в любое свободное место (по FSM); обновление часто реализуется как вставка новой версии строки (в PostgreSQL при MVCC), старая версия остаётся до VACUUM.

#### Пошагово: как по индексу найти строку (heap + TID)

1. Запрос ищет строку по ключу (например, id = 100). Есть индекс по id.
2. СУБД ищет в **индексе** запись с ключом 100. В индексе хранится не вся строка, а **ключ (100) + TID** — указатель на строку в heap. TID = (номер страницы heap, смещение в странице). Например, TID = (страница 5, смещение 120).
3. По TID СУБД определяет: строка лежит на **странице 5** heap этой таблицы, по смещению **120** байт от начала страницы.
4. СУБД загружает **страницу 5** (если её ещё нет в буфере) и по смещению 120 читает **заголовок строки и данные** — полную строку (все столбцы).
5. Из полной строки отдаются запросу нужные столбцы (или вся строка).

Итог: **индекс = ключ + TID.** Heap = место, где лежат полные строки. Чтобы получить строку по индексу, мы **всегда** идём по TID в heap (если только не срабатывает index-only scan — см. раздел 26, visibility map).

#### Простыми словами

**Heap** — «куча» строк таблицы: каждая строка лежит где-то на страницах. **TID** — «адрес» строки: «страница № такой-то, запись № такая-то». Индекс не хранит все столбцы — он хранит ключ и этот адрес (TID). Чтобы получить полную строку, СУБД идёт по адресу в heap и читает строку там.

#### Что будет, если запросу нужен столбец, которого нет в индексе?

Индекс даёт только **ключ + TID**. Столбцов name, phone и т.д. в индексе нет. СУБД **всегда** идёт по TID в **heap** и читает полную строку — оттуда берутся остальные столбцы. Поэтому «index-only scan» возможен только когда **все нужные столбцы есть в индексе** и страница помечена в visibility map (см. 26.2) — иначе без heap не обойтись.

#### Картинка в голове

Heap — это **склад с полками (страницами)**. На каждой полке лежат **коробки (строки)**. TID — это «полка 5, коробка 3» (страница + смещение). Индекс — это **справочник**: «ключ 100 → полка 5, коробка 3». Чтобы взять содержимое, мы идём на склад по адресу (полка 5, коробка 3) и открываем коробку (читаем строку из heap).

#### Как запомнить

**Heap** = где лежат **полные строки**. **TID** = адрес строки (страница + смещение). **Индекс** = ключ + TID, не все столбцы. Чтобы получить строку по индексу: **индекс → TID → heap** (загрузить страницу heap, прочитать строку по смещению). Запомни цепочку: **ключ → индекс → TID → heap → строка**.

#### Проверь себя (25.1)

1. Индекс по столбцу email хранит пары (email, TID). Запрос: SELECT name, phone FROM users WHERE email = 'a@b.com'. Где будет взято значение name и phone — из индекса или из heap?  
<details><summary>Ответ</summary> **Из heap.** В индексе хранятся только email (ключ) и TID. Значений name и phone в индексе нет. По TID СУБД загружает страницу heap и читает полную строку по смещению; из неё берутся name и phone.</details>

2. Зачем в индексе хранить **TID** (указатель на строку в heap), а не саму строку целиком?  
<details><summary>Ответ</summary> Потому что **индекс** нужен для **поиска** по ключу (email, id и т.д.); он должен быть **компактным** и содержать только ключ + «адрес» строки (TID). Если бы в индексе хранилась **вся строка** (все столбцы), индекс был бы **огромным** (дублирование всех данных) и при каждом изменении любого столбца строки пришлось бы обновлять запись в индексе. TID — это 6 байт (номер страницы + смещение); по нему мы один раз идём в heap и читаем полную строку. Так индекс остаётся маленьким, а данные хранятся в одном месте (heap).</details>

3. Что такое **index-only scan** и при каком условии он возможен (кратко)?  
<details><summary>Ответ</summary> **Index-only scan** — план, при котором запрос читает **только индекс**, не обращаясь к heap за полной строкой. Возможен, когда **все столбцы**, нужные запросу (SELECT и WHERE), **есть в индексе** (ключ или INCLUDE), и страница heap помечена в **visibility map** как «все строки видимы». Тогда видимость можно не проверять по heap, и данные берутся из индекса. Если хотя бы одного столбца нет в индексе или страница не в visibility map — нужен переход по TID в heap (heap fetch).</details>

#### Типичные ошибки

- **Ожидать, что индекс «содержит все столбцы»**: индекс хранит только **ключ + TID**. Чтобы получить столбцы, которых нет в ключе индекса, СУБД **всегда** идёт по TID в heap. «Index-only scan» возможен только когда все нужные столбцы **включены в индекс** (например, через INCLUDE или составной ключ) и страница помечена в visibility map — иначе будет обращение к heap (index scan + heap fetch).
- **Создавать индекс «на все столбцы» в надежде ускорить любой SELECT**: индекс тогда огромный и при каждом UPDATE/INSERT все эти столбцы нужно писать в индекс. Обычно индексируют **условия WHERE и JOIN**, а нужные для SELECT столбцы добавляют через INCLUDE или просто читают из heap по TID.

#### Запомните

- **Heap** — хранилище строк таблицы (страницы с полными строками). **Индексы** хранят ключ и **TID** (указатель: номер страницы + смещение в странице).
- По TID читается **строка в heap**; по одному TID получаем одну строку со всеми столбцами.

### 25.2. Заголовок строки и NULL bitmap

**Цель раздела.**  
Понять структуру **заголовка строки (tuple header)** — «паспорт» каждой строки — и как хранятся **NULL**: не как значение в теле строки, а как **бит** в битовой карте (экономия места и однозначная проверка).

#### Термины

- **Заголовок строки (tuple header)** — **метаданные** в начале каждой строки: информация о видимости (**xmin**, **xmax** для MVCC), количество столбцов, **битовая карта NULL**, размер строки и т.д. Размер заголовка фиксирован или вычисляется (например, в PostgreSQL с учётом количества столбцов). Сразу за заголовком идут данные столбцов.
- **xmin, xmax** — идентификаторы транзакций в заголовке: **xmin** — транзакция, **создавшая** строку; **xmax** — транзакция, **удалившая** или **обновившая** её (при обновлении в PostgreSQL создаётся новая версия строки, у старой выставляется xmax). По ним и по снимку транзакции определяется **видимость** строки (видна ли она текущей транзакции). Подробнее — в Части IV (MVCC).
- **NULL bitmap** — биты в заголовке (или сразу после него): **один бит на столбец**. 1 = этот столбец **NULL**, 0 = значение **присутствует** в теле строки. **NULL не хранится как значение** в теле строки — только бит; так экономим место (не выделяем байты под «пустое» значение).

#### Теория и правила

- В PostgreSQL заголовок включает: размер строки, служебные биты (например, has nulls, has var-width), **xmin**, **xmax**, и т.д.; за заголовком может идти **NULL bitmap** (если в таблице есть nullable столбцы), затем **данные столбцов** по порядку. Порядок и размер полей зависят от версии PostgreSQL.
- Для столбца с **NULL** в теле строки **место под значение не выделяется** — в теле идут только **не-NULL** значения в порядке столбцов. Проверка «столбец IS NULL» сводится к чтению **одного бита** в NULL bitmap. Так мы не тратим, например, 8 байт на NULL в bigint — только 1 бит.

#### Пошагово: как хранится строка с NULL (на примере)

Таблица: id (bigint), name (text), phone (text, nullable). Строка: id=1, name='Ivan', phone=NULL.

1. **Заголовок:** размер, xmin, xmax, количество столбцов = 3, флаг «есть NULL».
2. **NULL bitmap:** три бита — по одному на столбец. Бит 1 (id): 0 (значение есть). Бит 2 (name): 0 (значение есть). Бит 3 (phone): **1** (NULL).
3. **Тело строки:** только значения **не-NULL** подряд. id = 1 (8 байт), name = 'Ivan' (длина + данные). **phone не хранится** — его NULL задаётся битом в bitmap. При чтении: смотрим бит phone — 1 → возвращаем NULL; не читаем из тела.

Итог: **NULL = бит в заголовке**, не байты в теле. Экономия места и быстрая проверка IS NULL.

#### Простыми словами

**Заголовок строки** — «паспорт» строки: кто создал (xmin), кто удалил/заменил (xmax), сколько столбцов, какие из них NULL. **NULL** не хранится как отдельное значение в теле — в заголовке стоит **бит** «этот столбец NULL». Так мы не тратим место на само значение NULL (например, не кладём «пустые» 8 байт для bigint).

#### Картинка в голове

Строка — это **анкета**. Вверху анкеты — **шапка** (заголовок): номер транзакции-создателя, номер транзакции-удалителя, и **галочки** «этот пункт не заполнен» (NULL bitmap). Внизу — только **заполненные** пункты (тело строки). Пункт «телефон» не заполнен — в шапке галочка «телефон = пусто», в теле ничего не пишем. Так экономим место.

#### Как запомнить

**Заголовок** = паспорт строки: xmin, xmax (MVCC), количество столбцов, **NULL bitmap**. **NULL** = один бит в bitmap, не байты в теле — экономим место и быстро проверяем IS NULL. Запомни: в теле строки идут только **не-NULL** значения по порядку; NULL «закодирован» битом в шапке.

#### Проверь себя (25.2)

1. Почему NULL хранится битом в заголовке, а не как «пустое значение» в теле строки? Что мы выигрываем?  
<details><summary>Ответ</summary> **Экономия места** и **однозначность**. Если бы NULL хранился как значение в теле, для каждого nullable столбца пришлось бы резервировать место (например, 8 байт для bigint) даже когда значение NULL — место пропадает. Бит в заголовке занимает 1 бит на столбец — много меньше. Кроме того, «пустое значение» для разных типов разное (0 для числа, '' для строки?), а **бит NULL** однозначно говорит: «значения нет». Проверка IS NULL сводится к чтению одного бита — быстро.</details>

2. В таблице 10 столбцов, из них 4 могут быть NULL. Сколько минимум бит в NULL bitmap занимают эти столбцы? Хранится ли в **теле** строки место под значение NULL для столбца, который равен NULL?  
<details><summary>Ответ</summary> **NULL bitmap** — по одному биту на столбец (в том числе на не-nullable, но обычно биты считают для всех столбцов или только для nullable). Для 10 столбцов — минимум 10 бит, округляют до байт (например, 2 байта). В **теле** строки место под значение **не выделяется**, если столбец = NULL: в теле идут только значения **не-NULL** по порядку. NULL «закодирован» только битом в bitmap; при чтении по биту понимаем «значения нет», из тела ничего не читаем для этого столбца.</details>

3. Для чего в заголовке строки нужны поля **xmin** и **xmax** (кратко)?  
<details><summary>Ответ</summary> **xmin** — идентификатор транзакции, **создавшей** строку; **xmax** — транзакции, **удаливой** или **обновившей** её (при UPDATE в PostgreSQL создаётся новая версия, у старой выставляется xmax). По ним и по снимку текущей транзакции определяется **видимость** строки: видна ли она текущей транзакции (MVCC). Нужны для изоляции и для того, чтобы разные транзакции видели согласованное состояние без блокировок чтения.</details>

#### Запомните

- **Заголовок строки** содержит метаданные: **xmin**, **xmax** (для MVCC), количество столбцов, **NULL bitmap**.
- **NULL** хранится **битом** в заголовке (битовая карта: 1 = NULL, 0 = значение есть), **а не** как значение в теле строки; тело содержит только не-NULL значения.

### 25.3. TOAST

**Цель раздела.**  
Понять механизм **TOAST** (The Oversized-Attribute Storage Technique) в PostgreSQL: вынос больших значений в отдельное хранилище.

#### Термины

- **TOAST** — в PostgreSQL: хранение значений, не помещающихся в основную строку (примерно > 2 KB с учётом заголовка и ограничений страницы), в отдельной TOAST-таблице. В основной строке остаётся только указатель (или часть значения при стратегии MAIN).
- **Стратегии TOAST** (PostgreSQL): **PLAIN** — без сжатия и без TOAST (не для varlena типов с большими значениями); **EXTENDED** — сжатие и вынос в TOAST (по умолчанию для text, bytea и т.д.); **EXTERNAL** — вынос без сжатия (быстрее доступ к подстрокам); **MAIN** — попытка хранить в строке, при нехватке — вынос, можно со сжатием.

#### Теория и правила

- Строка в PostgreSQL должна помещаться в одну страницу (обычно 8 KB); с учётом заголовка страницы и строки остаётся меньше 8 KB на строку. Если столбец (например, text, jsonb) может быть большим, для него задаётся стратегия TOAST; при превышении порога значение сжимается и/или выносится в TOAST-таблицу.
- TOAST-таблица у каждой таблицы своя (скрытая); в ней хранятся «куски» (chunks) больших значений. Размер одного куска ограничен (например, 2 KB). При чтении столбца СУБД по указателю в основной строке подгружает куски из TOAST.

#### Пример (PostgreSQL)

```sql
-- По умолчанию для text используется TOAST (EXTENDED)
CREATE TABLE docs (
  id bigint PRIMARY KEY,
  body text   -- большие значения уйдут в TOAST
);

-- Запретить TOAST для столбца (только для маленьких значений)
ALTER TABLE t ADD COLUMN small_fixed varchar(100) SET STORAGE PLAIN;
```

#### Пошагово: что происходит при вставке большого значения (TOAST)

1. Запрос вставляет строку с большим значением в столбец text (например, 10 KB).
2. СУБД проверяет: помещается ли вся строка (заголовок + данные) в **одну страницу** (обычно 8 KB)? С учётом заголовка страницы и строки на одну строку остаётся меньше 8 KB. 10 KB не влезает.
3. СУБД применяет стратегию TOAST для этого столбца (по умолчанию **EXTENDED**): значение **сжимается** (если возможно) и разбивается на **куски** (chunks, например по 2 KB). Куски записываются в **TOAST-таблицу** этой таблицы (скрытую; у каждой таблицы своя TOAST-таблица).
4. В **основной строке** в месте столбца остаётся только **указатель** (OID или идентификатор) на TOAST-запись — «где лежат куски». Сама строка теперь помещается в страницу (указатель маленький).
5. При **чтении** столбца СУБД по указателю находит TOAST-запись, подгружает **куски** из TOAST-таблицы, собирает значение (и при необходимости распаковывает) и отдаёт запросу.

Итог: **TOAST = вынос больших значений в отдельную таблицу; в строке — только указатель.** Строка помещается в одну страницу; при чтении по указателю достаём куски и собираем значение.

#### Простыми словами

**TOAST** — «полка для больших вещей». Если значение не влезает в одну строку на странице, оно сжимается и кладётся в отдельную «TOAST-таблицу», а в строке остаётся только «ссылку на полку». При чтении по ссылке достаём куски и собираем значение.

#### Картинка в голове

Строка — коробка на полке (странице). В коробке есть ячейка «текст документа». Документ огромный — не влезает в коробку. Мы кладём документ в **отдельный ящик** (TOAST-таблица), а в коробке оставляем **записку** «документ в ящике № 5». При чтении по записке идём в ящик № 5, достаём документ и отдаём. TOAST — это «ящик для больших документов»; в строке только записка (указатель).

#### Как запомнить

**TOAST** = вынос **больших** значений (> ~2 KB) в **отдельную TOAST-таблицу**; в основной строке остаётся только **указатель** на куски. Строка должна помещаться в одну страницу — без TOAST большие text/bytea не влезли бы. Стратегии: **EXTENDED** (сжатие + вынос, по умолчанию), **EXTERNAL** (вынос без сжатия), **MAIN**, **PLAIN**. Запомни: **большое значение → TOAST-таблица, в строке — указатель**.

#### Что будет, если запрос выбирает только столбцы без TOAST (SELECT id, name FROM t), а в таблице есть столбец body в TOAST?

Тогда **TOAST вообще не читается**. В основной строке лежат id, name и **указатель** на body (не само значение). Запросу body не нужен — СУБД не подгружает куски из TOAST-таблицы. Читается только **страница heap** с основной строкой — один I/O вместо нескольких (heap + TOAST). Поэтому в широких таблицах важно выбирать в SELECT только нужные столбцы: если не запрашивать большие TOAST-столбцы, их не читают с диска, запрос быстрее.

#### Проверь себя (25.3)

1. В таблице есть столбец body типа text. Вставляем строку с body длиной 5 KB. Куда физически попадёт значение 5 KB — в основную таблицу или куда-то ещё? Почему?  
<details><summary>Ответ</summary> **В TOAST-таблицу** (отдельное хранилище для больших значений). Строка в PostgreSQL должна помещаться в **одну страницу** (обычно 8 KB). С учётом заголовка страницы и строки, заголовка строки и других столбцов на одно значение body остаётся меньше 8 KB. 5 KB — уже больше порога (~2 KB), поэтому значение **выносится** в TOAST: разбивается на куски и записывается в TOAST-таблицу этой таблицы; в основной строке остаётся только **указатель** на TOAST-запись.</details>

2. Что происходит при **чтении** строки, в которой столбец body вынесен в TOAST? Откуда СУБД берёт значение 5 KB?  
<details><summary>Ответ</summary> В основной строке в месте столбца body лежит только **указатель** (OID/идентификатор) на TOAST-запись. При чтении столбца СУБД по этому указателю находит TOAST-запись, **загружает куски** из TOAST-таблицы (несколько страниц TOAST, если значение большое), **собирает** значение из кусков и при стратегии EXTENDED **распаковывает** (если было сжато). Полученное значение отдаётся запросу. То есть чтение TOAST-столбца — это дополнительный **I/O** к TOAST-таблице; для больших значений это несколько чтений страниц.</details>

3. Какие стратегии хранения TOAST (PLAIN, EXTENDED, EXTERNAL, MAIN) и чем они отличаются по сжатию и выносу?  
<details><summary>Ответ</summary> **PLAIN** — без сжатия и без выноса в TOAST (для небольших значений; для больших varlena не подходит). **EXTENDED** — сжатие и вынос в TOAST (по умолчанию для text, bytea и т.д.): сначала пытаемся сжать, при превышении порога выносим в TOAST. **EXTERNAL** — вынос в TOAST **без сжатия** (быстрее доступ к подстрокам, но больше места). **MAIN** — хранить в основной строке по возможности, при нехватке — вынос (можно со сжатием). Для типичного text/bytea обычно **EXTENDED**.</details>

#### Запомните

- **TOAST** — вынос **больших** значений (> ~2 KB) в **отдельное хранилище** (TOAST-таблица); в основной строке остаётся только **указатель**.
- Стратегии: **EXTENDED** (сжатие + вынос, по умолчанию для text/bytea), **EXTERNAL** (вынос без сжатия), **MAIN** (в строке по возможности), **PLAIN** (без TOAST).

### 25.4. HOT и fillfactor

**Цель раздела.**  
Понять **HOT (Heap Only Tuple)** — обновление без обновления индексов — и роль **fillfactor** (свободное место на странице для таких обновлений).

#### Термины

- **HOT (Heap Only Tuple)** — в PostgreSQL: обновление строки, при котором **новая версия строки помещается на ту же страницу** и **ни один из изменённых столбцов не входит в индекс**. Тогда индексы не нужно обновлять: все индексы по-прежнему указывают на старую версию (старый TID), а цепочка версий на странице ведёт к новой. Это сильно снижает стоимость UPDATE.
- **Fillfactor** — доля страницы (в процентах), которую занимают данные; остальное остаётся свободным для вставки новых версий строк (HOT) или новых строк. Например, fillfactor = 90 означает «заполнять страницу только на 90%, 10% оставить свободными». Задаётся на уровне таблицы (и иногда индекса).

#### Теория и правила

- При UPDATE в PostgreSQL создаётся новая версия строки (tuple). Если изменён хотя бы один столбец, входящий в какой-либо индекс, все такие индексы должны получить новую запись (новый TID). Если изменяются только столбцы, не входящие ни в один индекс, и на странице есть свободное место (благодаря fillfactor), новая версия может быть помещена на ту же страницу и связана с цепочкой версий — индексы не трогаем (HOT).
- Fillfactor задаётся при создании таблицы: `CREATE TABLE t (...) WITH (fillfactor = 80);` или `ALTER TABLE t SET (fillfactor = 80);`. VACUUM и обычные вставки будут заполнять только до этой доли, оставляя место для HOT.

#### Пример (PostgreSQL)

```sql
-- Таблица с частыми обновлениями счётчиков/флагов — оставляем 20% под HOT
CREATE TABLE events (
  id bigint PRIMARY KEY,
  payload jsonb,
  processed boolean DEFAULT false,
  updated_at timestamptz DEFAULT now()
) WITH (fillfactor = 80);

-- Индекс только по id (PK); столбцы processed, updated_at не в индексе
-- UPDATE events SET processed = true, updated_at = now() WHERE id = ? — может быть HOT
```

#### Пошагово: когда UPDATE будет HOT, а когда нет

**Условия для HOT:**
1. Изменяются **только** столбцы, которые **не входят ни в один индекс** этой таблицы. (Первичный ключ и уникальные индексы содержат свои столбцы; если ты меняешь столбец, входящий в индекс, этот индекс должен получить новую запись с новым TID.)
2. На **той же странице** heap есть **достаточно свободного места** для новой версии строки. Свободное место создаётся за счёт fillfactor (страницу заполняют не до конца) и за счёт мёртвых строк после VACUUM.
3. Новая версия строки **помещается** на эту страницу и связывается с цепочкой версий (старая версия указывает на новую на той же странице).

**Если хотя бы одно условие нарушено** — HOT не будет: новая версия пойдёт на другую страницу и/или придётся обновить индексы (добавить в каждый затронутый индекс новую запись с новым TID). Это дороже: и место в heap, и обновление всех индексов, в которые входят изменённые столбцы.

**Пример:** таблица (id PK, status, updated_at). Индекс только по id. UPDATE SET status = 'done', updated_at = now() WHERE id = 5. Столбцы status и updated_at **не в индексе** — индекс по id не нужно менять (id не изменился). Если на странице есть место — новая версия строки может быть помещена на ту же страницу → **HOT**.

**Контрпример:** есть индекс по (status). Тот же UPDATE меняет status → индекс по status должен получить новую запись (новый TID), потому что значение ключа изменилось. **HOT нет** — индексы обновляются.

#### Простыми словами

**HOT** — «обновили строку так, что новая версия поместилась на той же странице и мы не трогали ни один индекс». Тогда индексы по-прежнему указывают на старый адрес, а цепочка версий на странице ведёт к новой. **Fillfactor** — «заполняем страницу не до упора, а оставляем место для таких обновлений». Меньше fillfactor — больше места под HOT, но больше страниц на тот же объём данных.

#### Картинка в голове

Страница — полка. Обновление строки — «переписать карточку». HOT — новая карточка кладётся **на ту же полку** и связывается со старой (цепочка); справочник (индекс) по-прежнему указывает на старую карточку, но по цепочке мы доходим до новой. Fillfactor — «на полке оставляем пустое место специально под такие новые карточки».

#### Как запомнить

**HOT** = обновление **без** обновления индексов. Нужно **два условия**: 1) изменяются **только** столбцы, **не входящие ни в один индекс**; 2) на **той же странице** есть **свободное место** (fillfactor оставил место). Тогда новая версия строки на той же странице, индексы не трогаем — дешевле. **Fillfactor** = доля заполнения страницы (80% = 20% оставить свободными под HOT). Запомни: **HOT = та же страница + меняем только неиндексированные столбцы**.

#### Проверь себя (25.4)

1. В таблице есть индекс по столбцу status. Выполняется UPDATE SET status = 'closed' WHERE id = 1. Будет ли это обновление HOT? Почему?  
<details><summary>Ответ</summary> **Скорее всего нет.** Обновляется столбец **status**, который **входит в индекс**. Индекс по status должен отразить новое значение и новый TID (новая версия строки). Поэтому индексы придётся обновлять — условие «ни один из изменённых столбцов не входит в индекс» не выполняется. HOT не будет (если только новая версия не поместится на ту же страницу и индексы как-то не обновляются — в стандартной реализации PostgreSQL индексы обновляются).</details>

2. Как fillfactor влияет на вероятность HOT? Почему для таблицы с частыми обновлениями часто ставят fillfactor = 80?  
<details><summary>Ответ</summary> **Fillfactor** задаёт долю заполнения страницы (80% = заполняем только 80%, 20% остаётся **свободным**). Для **HOT** нужно, чтобы новая версия строки **поместилась на ту же страницу** — для этого на странице должно быть **свободное место**. Чем **меньше** fillfactor (например, 80 вместо 100), тем **больше** свободного места на странице и тем **чаще** новая версия строки поместится на той же странице → больше шансов на HOT. Для таблицы с частыми обновлениями fillfactor = 80 оставляет 20% места под новые версии строк — меньше шансов, что придётся класть новую версию на другую страницу и обновлять индексы.</details>

3. Можно ли получить HOT, если изменить только один столбец, но он входит в составной индекс (a, b)?  
<details><summary>Ответ</summary> **Нет.** Для HOT условие: **ни один из изменённых столбцов не входит ни в один индекс**. Если столбец входит в индекс (даже как часть составного ключа (a, b)), то при изменении этого столбца индекс должен быть обновлён — добавить новую запись с новым TID. Поэтому HOT не будет. HOT возможен только когда меняются столбцы, которые **не участвуют ни в одном индексе** (ни в ключе, ни в INCLUDE).</details>

#### Типичные ошибки

- **Ставить fillfactor = 100 на таблицах с частыми UPDATE**: страница заполняется «под завязку», свободного места нет — новая версия строки почти всегда уходит на другую страницу, индексы обновляются, **HOT почти не бывает**. Для таких таблиц лучше 80–90%, чтобы оставить место под новые версии на той же странице.
- **Ожидать HOT, когда изменяемый столбец входит в индекс**: если в таблице есть индекс по столбцу `status`, то UPDATE SET status = … **обязательно** обновляет этот индекс (новая запись с новым TID). HOT не будет, даже если на странице есть место. HOT возможен только когда меняются **только** столбцы, **не входящие ни в один индекс**.

#### Запомните

- **HOT** — обновление **без** обновления индексов: новая версия строки на **той же странице** heap, и **ни один из изменённых столбцов не входит ни в один индекс**.
- **Fillfactor** — доля заполнения страницы (например, 80%); остальное — свободное место для HOT и вставок; для таблиц с частыми обновлениями часто 80–90.

### 25.5. Размер строки и сжатие

**Цель раздела.**  
Понять ограничения на размер строки и варианты сжатия (страницы или строк) в разных СУБД.

#### Термины

- **Максимальный размер строки** — в PostgreSQL одна строка (без TOAST) ограничена размером страницы (обычно 8 KB); с TOAST теоретически до ~1.6 TB (ограничение формата). В InnoDB строка (включая overflow для больших полей) ограничена примерно половиной страницы (например, 8 KB при 16 KB странице) для одного row format.
- **Сжатие страниц** — в некоторых СУБД (например, InnoDB с Barracuda, таблицы с `ROW_FORMAT=COMPRESSED`) страницы сжимаются перед записью; при чтении распаковываются. Компромисс: меньше I/O и места, больше CPU.
- **Сжатие строк** — в колоночных хранилищах часто сжимают значения столбцов (например, по типам или run-length); в реляционных OLTP сжатие чаще на уровне страницы.

#### Теория и правила

- Оценка размера строки: заголовок + по каждому столбцу (тип, выравнивание); для varlena (text, bytea, jsonb) — длина + данные. NULL не занимает места в теле (только бит). При приближении к лимиту страницы большие столбцы уходят в TOAST (PostgreSQL).
- Сжатие уменьшает объём на диске и часто ускоряет последовательное чтение (меньше блоков читать), но увеличивает нагрузку на CPU при сжатии/распаковке.

#### Простыми словами

Строка не может быть «бесконечной»: в одной странице — ограничение (например, 8 KB); большие поля выносятся в TOAST. **Сжатие** — «ужимаем страницу или данные перед записью»: на диске меньше, при чтении распаковываем. Выгодно, когда диск или сеть — узкое место, а CPU есть.

#### Картинка в голове

**Размер строки:** коробка (строка) должна влезать в один «лист» (страница 8 KB). Если коробка огромная — основную часть выносим на отдельную полку (TOAST), в листе оставляем только «записку» (указатель). **Сжатие:** перед тем как положить лист в архив, **сжимаем** его (zip); в архиве занимает меньше места, но при чтении нужно распаковать — тратим время (CPU).

#### Как запомнить

**Размер строки** ограничен страницей (без TOAST ~8 KB); большие значения → TOAST. **Сжатие страниц** = перед записью сжимаем, при чтении распаковываем. **Компромисс:** меньше I/O и места на диске, но **больше CPU** (сжатие/распаковка). На медленном диске сжатие часто выгодно; на быстром SSD и при нехватке CPU — не всегда. Запомни: **сжатие = меньше диск, больше CPU**.

#### Проверь себя (25.5)

1. Почему сжатие страниц «уменьшает I/O, но увеличивает нагрузку на CPU»?  
<details><summary>Ответ</summary> Потому что при **записи** страница **сжимается** (алгоритм типа zlib) — это работа **CPU**. При **чтении** страница **распаковывается** — тоже работа CPU. Зато **на диске** лежит меньше данных (сжатый объём), поэтому мы **меньше читаем и пишем** байт (меньше I/O). Если узкое место — **диск**, сжатие выгодно (меньше I/O). Если узкое место — **CPU**, сжатие может замедлить (дополнительная нагрузка на сжатие/распаковку).</details>

2. В каком типе нагрузки сжатие страниц чаще всего **выгодно**: OLTP (много коротких запросов, случайные обновления) или аналитика (массовое чтение больших таблиц)? Почему?  
<details><summary>Ответ</summary> Чаще выгодно при **аналитике** (массовое чтение). При аналитике основная нагрузка — **чтение** больших объёмов с диска; сжатие уменьшает объём данных на диске → меньше I/O → быстрее. CPU на распаковку обычно не узкое место. При **OLTP** много **записей** (UPDATE/INSERT); каждая запись требует сжатия страницы (или блока) — нагрузка на CPU растёт, плюс фрагментация сжатых страниц при частых обновлениях. Не значит, что в OLTP сжатие всегда плохо, но компромисс «I/O vs CPU» чаще в пользу сжатия при read-heavy нагрузке.</details>

3. Что такое KEY_BLOCK_SIZE в InnoDB при ROW_FORMAT=COMPRESSED и на что он влияет?  
<details><summary>Ответ</summary> **KEY_BLOCK_SIZE** задаёт размер **сжатого** блока в байтах (например, 8, 4 KB). Страница данных (например, 16 KB) сжимается до одного или нескольких таких блоков на диске. Меньший KEY_BLOCK_SIZE — **сильнее сжатие** (меньше места на диске), но **больше накладных** (несколько блоков на страницу, больше операций при чтении/записи). Больший KEY_BLOCK_SIZE — слабее сжатие, но проще формат. Типичные значения 8 или 4 KB — компромисс между степенью сжатия и производительностью.</details>

#### Запомните

- Размер строки **ограничен** размером страницы (без TOAST, обычно 8 KB); с TOAST в PostgreSQL до ~1.6 TB.
- **Сжатие** (страниц или столбцов) уменьшает место на диске и I/O, но **увеличивает** нагрузку на CPU (сжатие при записи, распаковка при чтении).

### 25.6. Практические примеры

**Цель раздела.**  
Закрепить: оценка размера строки, выбор fillfactor, проверка TOAST.

#### Пример: оценка размера строки (PostgreSQL)

```sql
-- Примерная оценка: заголовок ~24 байта + NULL bitmap + данные
-- Для таблицы: id bigint (8), name varchar(100) (~100), created_at timestamptz (8)
-- Заголовок + 3 бита NULL + выравнивание + 8+100+8 ≈ 24+4+8+116 = 152 байта на строку
-- На странице 8192 байт: ~8192/152 ≈ 53 строки на страницу (без учёта точного FSM/заголовка страницы)
```

#### Пример: проверка использования TOAST

```sql
-- В PostgreSQL: атрибуты хранения столбцов
SELECT a.attname, a.attstorage
FROM pg_attribute a
JOIN pg_class c ON a.attrelid = c.oid
WHERE c.relname = 'docs' AND a.attnum > 0 AND NOT a.attisdropped;
-- 'x' = EXTENDED (TOAST), 'p' = PLAIN, 'e' = EXTERNAL, 'm' = MAIN
```

#### Пошагово: как пользоваться оценкой размера строки

1. Посчитай **заголовок** (~24 байта в PostgreSQL) + **NULL bitmap** (по одному биту на nullable столбец, округлённо до байт).
2. Для каждого столбца прибавь **размер типа**: bigint = 8, timestamp = 8, varchar(n) / text = длина в байтах (в среднем можно оценить), и т.д. NULL в теле не занимает места (только бит в bitmap).
3. Добавь **выравнивание** (padding до 8 байт) — обычно несколько байт на строку.
4. **Итого** — размер одной строки. Размер таблицы ≈ число_строк × размер_строки + заголовки страниц (например, 24 байта на страницу 8 KB). Число страниц ≈ размер_таблицы / 8192.

#### Простыми словами

**Оценка размера строки** — сложить заголовок, NULL bitmap, размер каждого столбца и выравнивание. По размеру строки прикидываем, сколько строк на страницу и какой объём займёт таблица. **Fillfactor** — сколько процентов страницы заполнять при вставке; остальное — под обновления (HOT). **TOAST** — большие значения (text, bytea) выносятся в отдельное хранилище; в основной строке остаётся указатель.

#### Картинка в голове

**Строка** — коробка: внутри заголовок (метаданные), битовая карта NULL, потом ячейки столбцов. Размер коробки = заголовок + карта + сумма ячеек + выравнивание. **Страница** — полка на 8 KB: сколько коробок поместится — 8192 / размер_коробки. **Fillfactor 80%** — заполняем полку только на 80%, остальное место «под обновления». **TOAST** — если в коробке огромный предмет (текст, картинка), кладём его в отдельный склад, в коробке оставляем **записку** (указатель).

#### Проверь себя (25.6)

1. Таблица из трёх столбцов: id bigint, name varchar(200), created_at timestamptz. Оценка размера одной строки без учёта точного выравнивания: заголовок ~24, NULL bitmap ~1 байт, id 8, name в среднем 50 байт, created_at 8. Сколько примерно байт на строку? Сколько строк поместится на одну страницу 8 KB (8192 байт), если не считать заголовок страницы?  
<details><summary>Ответ</summary> Размер строки ≈ 24 + 1 + 8 + 50 + 8 = **91 байт** (без точного выравнивания). На странице 8192 байт полезной площади меньше (заголовок страницы ~24 байта, плюс потери на выравнивание). Если грубо 8192 − 24 ≈ 8168 байт под данные, то 8168 / 91 ≈ **89 строк** на страницу (очень приближённо). Точное число зависит от выравнивания и формата.</details>

2. Почему «число строк на страницу» важно для оценки размера таблицы и для понимания полного сканирования?  
<details><summary>Ответ</summary> Потому что **размер таблицы на диске** ≈ (число строк / строк на страницу) × размер страницы — зная строк на страницу, переводим число строк в число страниц и в байты. Для **полного сканирования** мы читаем **страницами**; чем больше строк на страницу, тем меньше страниц при том же числе строк — меньше I/O. Оценка «строк на страницу» помогает прикинуть, сколько страниц затронет запрос и сколько памяти/диска займёт таблица.</details>

3. Как fillfactor влияет на «число строк на страницу» при оценке?  
<details><summary>Ответ</summary> **Fillfactor** задаёт долю страницы, которую заполняют данными (например, 80% — 20% остаётся свободным). При оценке «строк на страницу» полезную площадь страницы нужно считать с учётом fillfactor: не 8192 − 24 байта заголовка, а **(8192 − 24) × fillfactor / 100**. При fillfactor = 80 на одну страницу поместится **меньше** строк, чем при 100%. Итоговый размер таблицы при том же числе строк будет **больше** (больше страниц), но зато остаётся место под HOT-обновления.</details>

#### Запомните

- Оценка размера: **заголовок** + NULL bitmap + **данные по столбцам** с учётом выравнивания; для varlena (text, varchar) — средняя или максимальная длина.
- Fillfactor и отсутствие индексов на часто обновляемых столбцах способствуют **HOT**.
