## 196. Cross‑shard операции

### В этом разделе главное

- Cross‑shard операции — это любые запросы, которые **затрагивают более одного шарда**.
- Они почти всегда дороже локальных, потому что требуют **scatter‑gather, дополнительную сеть и координацию**.
- Часто архитектуру можно упростить, **перенесев часть агрегаций и JOIN‑ов ближе к данным** или спроектировав шард‑ключ иначе.

---

### Простыми словами

Если данные «нарезаны» на несколько шардов, то:

- запрос `SELECT SUM(amount) FROM orders` без фильтров:
  - **затрагивает все шарды** с `orders`;
- запрос `SELECT * FROM users WHERE id = 123`:
  - при хорошем шард‑ключе (`user_id`) затрагивает **ровно один шард**;
- запрос `SELECT * FROM orders WHERE amount > 1000`:
  - если `amount` не шард‑ключ — **все шарды должны быть просмотрены**.

Cross‑shard операция — это когда **нельзя ограничиться одним шаром**, приходится:

- отправлять подзапросы на несколько шардов;
- ждать ответы от всех;
- где‑то в координирующем компоненте **дособирать и досортировывать** результат.

---

### Теория и правила

Типичные виды cross‑shard операций:

- **Глобальные агрегаты**:
  - `COUNT(*)`, `SUM(amount)`, `MAX(created_at)` по всем данным;
  - выполняются как:
    - `SUM` по каждому шару → `SUM` сумм на координирующей ноде;
    - `MAX` по каждому шару → `MAX(MAX_i)` и т.п.
- **Сортировки**:
  - `ORDER BY created_at LIMIT 100`:
    - каждый шард локально сортирует `LIMIT K` (где `K > 100`);
    - координатор делает **merge sort** и берёт финальный `LIMIT 100`.
- **JOIN между шардами**:
  - если таблицы не co‑located:
    - координатор вынужден:
      - либо **броадкастить** одну таблицу на все шарды другой;
      - либо тянуть данные на себя и делать JOIN там;
      - либо делать сложные многошаговые планы.

Стоимость:

- растёт по мере:
  - увеличения числа шардов;
  - объёма данных, передаваемых по сети;
  - сложности агрегаций и сортировок.

---

### Пошагово/Практика (как думать о запросе)

Когда смотришь на запрос в шардированной системе, задай себе:

1. **Могу ли я ограничить запрос одним шардом?**
   - Да → отлично, это локальная операция.
   - Нет → это cross‑shard.
2. **Если это агрегат**, могу ли я:
   - считать **промежуточные агрегаты** на шардом;
   - а затем **сливать** их на координаторе?
3. **Если это сортировка**, могу ли я:
   - сортировать меньше данных на шардом (`LIMIT` + хороший индекс);
   - уменьшить `LIMIT` (например, пагинация вместо «выгрузи всё»)?
4. **Если это JOIN**, могу ли я:
   - переопределить схему так, чтобы таблицы были **co‑located**;
   - использовать **денормализацию** (дублировать нужные поля);
   - сначала получить ключи на одном шарде, затем адресно сходить в другой?

---

### Подробные примеры cross‑shard операций

#### Пример 1. Глобальный COUNT

Есть шардированная таблица `orders` по `customer_id`, 8 шардов.  
Запрос:

```sql
SELECT COUNT(*) FROM orders;
```

Что происходит:

1. Координатор (Citus/Vitess/другой middleware) понимает:
   - таблица шардирована;
   - фильтра по шард‑ключу нет → **затронуты все шарды**.
2. Он отправляет на каждый шард подзапрос:

```sql
SELECT COUNT(*) FROM orders;
```

3. Каждый шард отвечает, скажем:
   - шард 1 → `1000`;
   - шард 2 → `850`;
   - …;
   - шард 8 → `1200`.
4. Координатор суммирует:

```text
1000 + 850 + ... + 1200 = общее число строк
```

5. Клиент получает **одно число**, но под капотом был **scatter‑gather на 8 шардов**.

#### Пример 2. Сортировка с LIMIT

Запрос:

```sql
SELECT * FROM orders
ORDER BY created_at DESC
LIMIT 100;
```

Если `created_at` не шард‑ключ, то:

1. Каждый шард:
   - делает локальную сортировку:

```sql
SELECT * FROM orders
ORDER BY created_at DESC
LIMIT K;
```

   - где `K` берут с запасом (например, 200 или 500), чтобы после слияния хватило строк для финального `LIMIT 100`.
2. Координатор:
   - получает до `K` строк с каждого шарда;
   - **сливает отсортированные списки** (merge sort);
   - берёт первые 100 строк по `created_at DESC`.

Чем больше шардов и чем больше `K`, тем:

- больше данных по сети;
- больше памяти и CPU на координаторе.

#### Пример 3. Cross‑shard JOIN

Таблицы:

- `users(user_id, name, ...)` — шардирована по `user_id`;
- `orders(order_id, user_id, amount, ...)` — шардирована по **другому ключу** (ошибка дизайна).

Запрос:

```sql
SELECT u.name, o.amount
FROM users u
JOIN orders o ON o.user_id = u.user_id
WHERE u.user_id = 123;
```

Проблема:

- строки `user_id = 123` в `users` и `orders` могут оказаться на **разных шардом**.

Возможные планы:

- Выкачать строку `u` (`user_id = 123`) на координатор, затем:
  - спросить все шарды `orders`, есть ли у них строки с `user_id = 123`;
  - собрать и сделать JOIN на координаторе.
- Или наоборот:
  - найти `orders` по `user_id = 123` (scatter‑запрос);
  - подтянуть `users`.

Оба варианта = **cross‑shard JOIN** с сетью и дополнительной логикой.

Вывод: лучше было бы изначально **co‑locate** таблицы по одному шард‑ключу.

---

### Ещё один способ думать: «дорогой ли этот запрос для кластера?»

Простой чек‑лист:

- Затрагивает ли запрос **много шардов**?
  - Да → уже подозрительно.
- Передаётся ли по сети **много строк**?
  - `SELECT *` по всем без фильтров/лимитов → почти всегда плохо.
- Есть ли `JOIN` между таблицами, которые:
  - шардированы **по разным ключам**;
  - или вообще не шардированы согласованно?
  - Да → cross‑shard JOIN, дорогой по определению.

По сути, ты всегда спрашиваешь:

> «Могу ли я сделать большую часть работы **локально на шардом**, а на координирующей ноде только собрать итог?»

Если ответ «нет» — это кандидат на пересмотр схемы или переписывание запроса.

---

### Типичные ошибки (расширенный список)

- Дизайн, при котором:
  - все важные отчёты и выборки **обязательно затрагивают все шарды**;
  - в итоге любой аналитический отчёт — это **n× дороже**, чем в однонодовой БД.
- Слепое использование `SELECT *` в распределённом кластере:
  - особенно вместе с широкими таблицами и без `LIMIT`;
  - приводит к тому, что по сети гуляют **гигабайты лишних данных**.
- Попытка делать сложные OLAP‑запросы (много джойнов, оконные функции) в чисто OLTP‑шардированном кластере:
  - проще часто выделить **отдельное хранилище** (DW) и грузить туда данные пачками.

---

### Проверь себя (дополнительно)

<details>
<summary>Почему запрос без фильтра по шард‑ключу почти всегда дороже запроса с таким фильтром?</summary>

Потому что без фильтра по шард‑ключу система **не знает, какой конкретно шард** нужен для ответа.  
Ей приходится:

- либо рассылать запрос **на все шарды**;
- либо иметь сложные индексы/метаданные для выбора подмножества шардов.

Фильтр по шард‑ключу позволяет **ограничиться одним шардом** (или малым числом) и выполнять работу локально.

</details>

---

### Картинка в голове

Cross‑shard операция — это как **собрать отчёт, когда данные лежат в разных офисах**:

- тебе нужно:
  - разослать запросы во все офисы;
  - дождаться, пока все пришлют свои сводки;
  - сложить/отсортировать всё у себя;
- если офисов мало и отчёт компактный — нормально;
- если офисов сотни и отчёты огромные — становится **медленно и дорого**.

---

### Типичные ошибки

- Проектировать схему так, что **большинство важных запросов становится cross‑shard**, и удивляться высоким задержкам.
- Делать `ORDER BY ... LIMIT 1000` по всему кластеру вместо того, чтобы **фильтровать и резать запросы по шард‑ключу**.
- Надеяться, что «оптимизатор как‑то разберётся», и не анализировать фактические планы на конкретной распределённой системе.

---

### Проверь себя

<details>
<summary>Почему cross‑shard JOIN дороже, чем локальный JOIN?</summary>

Потому что данные для JOIN‑а:

- либо нужно **пересылать по сети** между шардом;
- либо **собирать на координирующей ноде** и делать JOIN там;
- оба варианта добавляют **сетевой overhead, дополнительную память и координацию**.

Локальный JOIN работает в рамках одной ноды — без сетевых пересылок и внешней координации.

</details>
