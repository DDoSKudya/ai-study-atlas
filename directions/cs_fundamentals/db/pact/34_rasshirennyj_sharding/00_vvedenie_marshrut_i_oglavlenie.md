## Часть XXXIV. Расширенные темы шардирования

> Цель этой части — **перестать думать про шардирование как про «какой‑то модный кластер»** и начать понимать, **как конкретно режутся данные на шарды, как они переезжают при росте нагрузки, как выполняются cross‑shard операции и как всё это мониторить и отлаживать без потери данных**.

---

После этой части ты сможешь:

- **объяснить простыми словами, чем отличаются range, hash, directory‑based шардирование и consistent hashing**, в каких сценариях что лучше подходит;
- **понимать архитектуру Citus поверх PostgreSQL**: distributed tables, reference tables, colocation, rebalance, cross‑shard запросы;
- **разобраться, как Vitess шардирует MySQL**: keyspace, shard, vindex, resharding и VReplication;
- **объяснить, как работают cross‑shard операции**: scatter‑gather, агрегаты, сортировка и JOIN‑ы между шардами, какие тут ограничения и цена;
- **спланировать и выполнить resharding (изменение числа шардов) с минимальным простоем**, понимать разницу offline/online подходов;
- **настроить базовый мониторинг шардирования**: горячие ключи, перекос распределения, триггеры rebalance;
- **выбрать и обосновать стратегию роутинга запросов**: в приложении, через прокси или через middleware (Citus/Vitess‑подобные решения).

Материал опирается на:

- часть про **партиционирование и внутреннее устройство хранения**;

- общие части про **распределённые системы, CAP‑интуицию и консистентность**.

### Краткая шпаргалка (термины по части)

| Термин | Что это простыми словами |
|--------|---------------------------|
| **Shard (шард)** | Логический кусок данных (поднабор строк/ключей), который хранится и обрабатывается отдельной группой нод. |
| **Range sharding** | Разбиение по диапазону значений (например, по дате или ID), удобно для диапазонных запросов, но есть риск горячих границ. |
| **Hash sharding** | Разбиение по хешу ключа (например, `hash(user_id) % N`), даёт равномерную нагрузку, но ломает диапазонные запросы. |
| **Directory‑based sharding** | Таблица маршрутизации «ключ → шард», даёт гибкость в распределении, но добавляет отдельный слой для lookup. |
| **Consistent hashing** | Способ распределять ключи по нодам так, чтобы при добавлении/удалении ноды **переезжала только часть ключей**, а не все. |
| **Distributed table (Citus)** | Таблица в PostgreSQL, строки которой распределяются по worker‑нодам по шард‑ключу. |
| **Reference table (Citus)** | Небольшая таблица, полностью реплицируемая на все ноды, чтобы делать JOIN‑ы без лишних cross‑shard обращений. |
| **Colocation (Citus)** | Гарантия, что связанные данные из разных таблиц с одним шард‑ключом лежат на одних и тех же шардах. |
| **Keyspace (Vitess)** | Логическая «база» в Vitess, которая может быть расшардирована на множество шардов. |
| **Vindex (Vitess)** | Виртуальный индекс: функция/структура, которая говорит Vitess, как маппить значение ключа на конкретный шард. |
| **Scatter‑gather** | Паттерн, при котором запрос рассылается на все (или многие) шарды, а результаты собираются и агрегируются. |
| **Resharding** | Процесс изменения числа шардов (или изменения шард‑ключа) с переносом данных между ними. |
| **Hot key (горячий ключ)** | Ключ, по которому приходит непропорционально много запросов, перегружая один шард. |
| **Rebalance** | Перераспределение данных между шардами/нодами для выравнивания нагрузки и объёма. |
| **Application‑level routing** | Когда само приложение знает, в какой шард идти с данным ключом. |
| **Proxy / middleware routing** | Когда отдельный слой (прокси или middleware) сам решает, на какой шард отправить запрос. |

### Сквозной учебный кейс: SaaS вырос с 1 инстанса до 8 шардов

Представь SaaS‑сервис с сущностью `tenant_id`.

Сначала всё живёт в одном PostgreSQL‑инстансе. Потом:

1. Появляются крупные tenants, и один инстанс перестаёт держать запись.
2. Команда выбирает `tenant_id` как основной шард‑ключ, потому что:
   - большинство запросов и так tenant‑scoped;
   - изоляция по клиенту упрощает routing;
   - cross‑tenant аналитика редкая и может жить отдельно.
3. На первом шаге вводится **routing‑слой** и правило “любой запрос обязан знать `tenant_id`”.
4. Дальше маленькие справочники становятся **reference/broadcast‑таблицами**, чтобы не делать лишние cross‑shard JOIN.
5. Затем один из крупных tenants становится hot tenant:
   - по метрикам видно, что один шард стабильно даёт больше CPU, WAL и p99;
   - команда решает не “хешировать всё заново”, а выделить этого tenant в отдельный shard tier.
6. После этого появляется задача online resharding:
   - включают backfill;
   - проверяют dual‑write или managed‑механику движка;
   - измеряют lag миграции;
   - готовят cut‑over и rollback‑план.
7. На финальном шаге команда добавляет мониторинг:
   - skew по размеру и QPS на shard;
   - top hot keys / hot tenants;
   - долю scatter‑gather запросов;
   - время rebalance и lag backfill.

Что важно вынести из кейса:

- шардирование почти всегда начинается не с “кластера”, а с **контракта routing**;
- удачный шард‑ключ уменьшает число cross‑shard операций сильнее, чем любой тюнинг;
- resharding без наблюдаемости и rollback‑плана почти всегда заканчивается больно.
