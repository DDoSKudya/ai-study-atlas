## 193. Стратегии шардирования

### В этом разделе главное

- Шардирование — это **логическое разбиение данных по нескольким хранилищам**, чтобы распределить нагрузку и объём.
- Есть несколько базовых стратегий: **range sharding, hash sharding, directory‑based**, а также **consistent hashing** как улучшение hash‑подхода.
- **Выбор стратегии** зависит от **паттернов запросов** (точечные vs диапазонные), **темпов роста данных** и требований к **гибкости при решардинге**.
- Любая стратегия должна учитывать риск **горячих ключей** и **перекоса нагрузки**.

---

### Простыми словами

Представь, что у тебя есть **огромная таблица пользователей**, которая **не влезает на один сервер** по объёму и нагрузке.

Идея шардирования:

- вместо одного «сундука» с данными мы делаем **несколько сундуков (шардов)**;
- заранее договариваемся по **правилу, как раскладывать пользователей по сундукам**;
- приложение, глядя на `user_id` (или ещё что‑то), **понимает, в какой сундук залезть**.

Разные стратегии — это просто **разные правила «в какой сундук класть»**:

- **range**: «пользователи с `id` от 1 до 1 млн — сюда, от 1 млн до 2 млн — туда»;
- **hash**: «берём `hash(user_id)`, делим по модулю числа шардов и кладём по остатку»;
- **directory‑based**: «сначала смотрим в отдельную табличку, там написано `user_id → шард 5`, значит идём туда»;
- **consistent hashing**: «рисуем круг, расставляем ноды по кругу, ключи падают между ближайшими нодами — так при добавлении нод меньше перестановок».

---

### Теория и правила

#### Range sharding

Идея:

- делим пространство значений **на интервалы (диапазоны)**, каждый диапазон — отдельный шард;
- классический пример — **по времени** (месяц/год) или **по ID**.

Плюсы:

- отлично подходит для **диапазонных запросов**: `WHERE created_at BETWEEN ...`, `WHERE id BETWEEN ...`;
- легко делать **архивацию старых данных** (отдельный шард на старые периоды);
- логично для **аналитики по времени**.

Минусы:

- если значения растут монотонно (например, `id` по sequence или `created_at = now()`), **последний шард становится горячим**;
- добавление новых шардов часто требует **разрезать существующий диапазон и перенести часть данных**.

Когда использовать:

- когда **большинство запросов — по диапазонам** (время, гео‑диапазоны, ID‑диапазоны);
- когда можно контролировать **поступление данных** (например, заранее продумать распределение по времени).

#### Hash sharding

Идея:

- выбираем **шард‑ключ** (например, `user_id`);
- считаем `hash(user_id)` (любая стабильная хеш‑функция) и берём остаток по модулю числа шардов `N`;
- по значению остатка (`0..N-1`) выбираем шард.

Плюсы:

- при «нормальном» распределении ключей — **почти равномерное распределение данных и нагрузки**;
- нет проблемы «последнего горячего диапазона».

Минусы:

- **диапазонные запросы по ключу не работают напрямую**: `user_id BETWEEN 100 AND 200` размазывается по всем шардом через хеш;
- при изменении числа шардов `N` (например, с 4 до 5) **всё хеш‑разбиение ломается** — нужно глобальное перемещение данных.

Когда использовать:

- когда **основные запросы точечные** (по конкретному ключу: `WHERE user_id = ?`);
- когда важнее **равномерность нагрузки**, чем удобство диапазонных запросов.

#### Directory‑based sharding

Идея:

- вместо жёсткой формулы мы храним **таблицу соответствия**: «ключ → шард»;
- когда нужно прочитать/записать данные по ключу:
  - сначала идём в директорию (routing table),
  - находим, какой шард отвечает за этот ключ/группу,
  - идём на нужный шард.

Плюсы:

- **максимальная гибкость**: можно перемещать ключи между шардом, просто изменяя запись в директории;
- удобно для случаев, когда **распределение ключей сильно неравномерно**.

Минусы:

- появляется **дополнительный компонент** (таблица/сервис маршрутизации), который нужно **реплицировать и масштабировать**;
- при большом числе ключей директория сама становится **узким местом**.

Когда использовать:

- когда **ключи и нагрузка очень неравномерны** (нужна тонкая ручная балансировка);
- когда хочешь уметь **перекладывать отдельные range/группы без глобального пересчёта хеша**.

#### Consistent hashing

Идея (упрощённо):

- представляем все ноды (шарды) как точки на **круге (хеш‑кольце)**;
- каждый ключ тоже хешируем в число и проецируем на круг;
- ключ принадлежит **первой ноде по часовой стрелке** после своей точки на круге;
- при добавлении/удалении ноды:
  - меняется только ближайший сегмент круга,
  - **переезжает только часть ключей**, а не все.

Плюсы:

- при изменении числа нод переносим **пропорционально малую часть данных**;
- удобно для динамически масштабируемых систем (например, кэши/ключ‑значение).

Минусы:

- сложнее реализация, чем простой `hash(key) % N`;
- для хорошей равномерности часто используют **виртуальные ноды** (много точек на круге на одну физическую ноду).

Когда использовать:

- когда **часто добавляются/удаляются ноды** и хочется минимизировать перетасовку данных;
- когда шардирование нужно для **ключ‑значение‑хранилищ** (кэши, распределённые ключ‑value‑сервисы).

---

### Пошагово/Практика

Пример: у тебя есть сервис с пользователями, и ты хочешь расшардировать таблицу `users` по `user_id`.

1. **Определи паттерн запросов.**
   - Если 90% запросов — `SELECT * FROM users WHERE id = ?` → **hash sharding** кандидат.
   - Если много запросов `WHERE created_at BETWEEN ...` → смотри в сторону **range sharding по дате**.
2. **Выбери шард‑ключ.**
   - Обычно это **стабильный, неизменяемый идентификатор** (например, `user_id`).
   - Старайся не использовать поле, которое часто обновляется.
3. **При hash sharding**:
   - выбери стабильную хеш‑функцию (например, `crc32`, `murmur` и т.п.);
   - определи `N` — число шардов;
   - реализуй функцию `shard_for(user_id) = hash(user_id) mod N`.
4. **При range sharding**:
   - определи **схему диапазонов** (например, по году `2020`, `2021`, `2022` или по ID‑интервалам);
   - заведи **таблицу с диапазонами и шард‑ID** или храни эту информацию в конфигурации;
   - реализуй поиск диапазона по ключу.
5. **Подумай про решардинг заранее.**
   - При hash sharding — можно сразу строить **consistent hashing** или использовать **слой индирекции** (таблица `hash_slot → шард`);
   - При range sharding — закладывай возможность **делить диапазон пополам** и переносить часть данных.

---

### Картинка в голове

Шардирование — это **раскладывание книг по полкам**:

- range sharding — «на этой полке книги с фамилиями от А до К, на следующей — от Л до Я»;
- hash sharding — «берём номер книги, считаем остаток по числу полок и ставим на полку по номеру остатка»;
- directory‑based — «есть каталог, где для каждой книги записано, на какой полке она лежит»;
- consistent hashing — «полки стоят по кругу, а книги кладём на первую полку после её номера на круге».

---

### Типичные ошибки

- Выбрать **range sharding по монотонно растущему полю** (например, `created_at`) и получить **одну перегретую ноду**.
- Делать hash sharding без продуманного решардинга, а потом «вдруг» понимать, что **надо удвоить число шардов** и **переносить вообще все данные**.
- Игнорировать **горячие ключи** (`user_id = 1`, суперпопулярный аккаунт) и не иметь механизма разнести его по нескольким шардом.
- Хардкодить маппинг `hash(key) % N` **в десятках сервисов**, усложняя дальнейший переход на другую схему.

### Дополнительные вопросы по пунктам

<details>
<summary>Range sharding: как ты поймёшь, что последний шард стал «горячим»?</summary>

Признаки:

- почти все новые записи попадают в **последний диапазон** (по дате или ID);
- по метрикам видно, что:
  - этот шард по QPS/latency **сильно перегружен**;
  - остальные шарды почти не растут и мало нагружены.

</details>

<details>
<summary>Hash sharding: почему выбор «плохого» шард‑ключа (например, часто меняющегося поля) опасен?</summary>

Потому что:

- при изменении значения такого поля **меняется результат хеш‑функции**;
- строку придётся **перекладывать с одного шарда на другой**;
- это даёт лишнюю нагрузку, риск несогласованности и усложняет resharding.

Лучше выбирать **стабильный идентификатор** (`user_id`, `tenant_id`), который не меняется.

</details>

<details>
<summary>Directory‑based sharding: какую цену мы платим за максимальную гибкость?</summary>

Мы получаем:

- дополнительный компонент — **таблицу/сервис маршрутизации**;
- его нужно:
  - **реплицировать**, чтобы не потерять;
  - **масштабировать**, чтобы он не стал узким местом;
- при большом числе ключей сама директория может **стать бутылочным горлышком** по размеру и по нагрузке.

</details>

<details>
<summary>Consistent hashing: зачем нужны «виртуальные ноды» на кольце?</summary>

Если разместить по одной точке на кольце на каждую физическую ноду:

- распределение ключей может получиться **неровным** (одна нода получит сильно больше ключей);

Если вместо этого:

- добавить **несколько виртуальных точек** на кольце для каждой ноды;
- ключи распределяются **более равномерно**, и нагрузка выравнивается.

</details>

---

### Дополнительные вопросы по пунктам

<details>
<summary>Distributed tables: зачем вообще выносить таблицу из координатора на worker‑ноды?</summary>

Чтобы:

- **распараллелить хранение и обработку данных**;
- использовать несколько машин вместо одной;
- дать возможность:
  - увеличивать число worker‑нод;
  - переносить шарды между ними при росте нагрузки.

Если бы всё жило на координаторе, шардирование бы не работало — это был бы один сервер.

</details>

<details>
<summary>Reference tables: какие таблицы чаще всего превращают в reference и почему?</summary>

Обычно это:

- небольшие **справочники**:
  - страны;
  - планы подписки;
  - статусы и т.п.

Причина:

- их удобно **целиком реплицировать на все worker‑ноды**;
- тогда `JOIN` с ними можно делать **локально**, не отправляя запросы через координатора и не устраивая scatter‑gather.

</details>

<details>
<summary>Colocation: почему важно, чтобы у co‑located таблиц совпадало число шардов и стратегия шардирования?</summary>

Потому что:

- только тогда система может гарантировать:
  - что строки с одинаковым шард‑ключом (например, `tenant_id`) окажутся на **одном и том же шарде**;
- если число шардов или стратегия отличаются:
  - распределение может «разъехаться»;
  - `JOIN` придётся делать как cross‑shard, с сетью и дополнительной координацией.

</details>

<details>
<summary>Rebalance: какие риски есть, если запустить его в пиковое рабочее время?</summary>

Во время rebalance:

- активно **переносятся данные между нодами**;
- растут:
  - нагрузка на диски (чтение+запись);
  - нагрузка на сеть.

Если делать это в пик:

- увеличится latency боевых запросов;
- возможны таймауты;
- в худшем случае можно «положить» часть кластера.

Поэтому rebalance обычно планируют на **низкую нагрузку** и внимательно мониторят.

</details>

---

### Проверь себя

<details>
<summary>Чем hash sharding принципиально отличается от range sharding по возможности диапазонных запросов?</summary>

При **range sharding** данные отсортированы по ключу в рамках диапазонов, поэтому запросы вида `WHERE key BETWEEN ...` можно направить только на часть шардов (понятно, какие диапазоны затрагиваются).  
При **hash sharding** ключи случайно размазаны по шардом через хеш‑функцию, и диапазон `key BETWEEN A AND B` в общем случае попадает **во все шарды**, теряется локальность диапазона.

</details>

<details>
<summary>Зачем нужен consistent hashing, если уже есть hash sharding с mod N?</summary>

При `hash(key) % N` изменение `N` меняет результат почти для всех ключей, значит нужно **массово переносить данные**.  
Consistent hashing устроен так, что при добавлении/удалении ноды перемещается только **часть ключей**, то есть решардинг становится гораздо дешевле по данным и времени.

</details>

### Ещё примеры и упражнения по стратегиям шардирования

#### Пример 1. Интернет‑магазин

У тебя есть:

- таблица `users(id, email, created_at, country)`;
- таблица `orders(id, user_id, created_at, amount)`.

Нагрузку описываем:

- **80%** запросов:
  - «показать профиль пользователя» (`SELECT * FROM users WHERE id = ?`);
  - «показать заказы пользователя» (`SELECT * FROM orders WHERE user_id = ? ORDER BY created_at DESC LIMIT 20`);
- **20%** запросов:
  - аналитика по дате: «сколько заказов за вчера/за неделю»;
  - выборка пользователей по стране и дате регистрации.

Как думать:

- Основные критичные запросы — **по конкретному пользователю** → шард‑ключ **`user_id`**.
- Для таблицы `users`:
  - шард‑ключ = `id`.
- Для таблицы `orders`:
  - шард‑ключ = `user_id`.

Тогда:

- `SELECT * FROM users WHERE id = ?` → всегда **один шард** (hash или range по `id`).
- `SELECT * FROM orders WHERE user_id = ?` → тоже **один шард**.
- Аналитика по дате (`WHERE created_at BETWEEN ...`) всё равно будет **cross‑shard**, но:
  - она обычно реже;
  - её можно вынести в **отдельный аналитический контур** (warehouse, OLAP).

Вывод: **не надо пытаться одной схемой шардирования оптимизировать и OLTP, и OLAP**.  
Для онлайновых запросов — `user_id` как ключ; для аналитики — часто отдельный слой.

#### Пример 2. Логи и события

Есть сервис логов:

- таблица `events(id, user_id, created_at, type, payload_json)`;
- запросы:
  - 70% — «последние N событий пользователя»;
  - 20% — «события за последний час по типу»;
  - 10% — тяжёлая аналитика.

Варианты:

- **Range по дате**:
  - шард‑ключ — `created_at` (месячные/недельные диапазоны);
  - отлично для запросов «за последний час/день/неделю»;
  - но, чтобы найти «последние N событий пользователя», надо:
    - знать, на каких партициях/шардом они лежат;
    - иногда ходить сразу в несколько шардов (если история длинная).
- **Hash по `user_id`**:
  - идеально для «последние N событий пользователя»;
  - но выборка «все события за последний час» превращается в cross‑shard.

Реальное решение часто такое:

- **боевой (OLTP) контур**:
  - hash по `user_id` (быстро обслуживаем пользователей);
- **отдельное хранилище логов/аналитики**:
  - range по `created_at` (для глобальных запросов по времени).

#### Мини‑упражнение

Попробуй мысленно ответить:

1. Есть таблица транзакций банка: `transactions(id, account_id, created_at, amount)`.  
   Основные запросы:
   - показать историю по одному счёту (`WHERE account_id = ?`);
   - отчёты по дате (днём позже, часом позже).  
   Какие варианты шард‑ключа видишь? Как бы ты разделил OLTP и отчёты?

2. Сервис соц. сети: `posts(id, author_id, created_at, text)`.  
   Запросы:
   - показать ленту автора;
   - показать ленту друзей (несколько `author_id` сразу);
   - поиск по хэштегам.  
   Где здесь шард‑ключ, а где — отдельные индексы/сервисы?

Не переживай, если ответы не кажутся «идеальными». Главное — **привыкать явно проговаривать паттерны запросов и делать под них схему**.

### Частые вопросы по стратегиям шардирования

**Вопрос:** почему нельзя просто взять «что угодно» как шард‑ключ?  
**Ответ:** можно, но тогда:

- часть запросов будет почти всегда **cross‑shard** (медленнее и сложнее);
- решардинг может оказаться очень дорогим;
- логика в приложении станет странной («почему по email медленно, а по id быстро?»).

---

**Вопрос:** а можно шардировать сразу по двум полям (`user_id` и `created_at`)?  
**Ответ:** как правило, **один шард‑ключ на таблицу**. Можно:

- хранить **составной ключ** (например, `(user_id, created_at)`), но шардирование будет всё равно по какому‑то одному вычисляемому значению;
- либо делать разные таблицы/материализованные представления для разных паттернов (OLTP/OLAP).

---

**Вопрос:** что делать с очень «толстыми» пользователями/клиентами (tenants), у которых слишком много данных?  
**Ответ:** часто используют:

- **выделенный шард** или даже **отдельный кластер** для крупного клиента;
- или **дополнительный уровень шардирования внутри тенанта** (но это уже отдельная сложная тема).
