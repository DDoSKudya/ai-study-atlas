## 198. Мониторинг шардирования

### В этом разделе главное

- Даже самая красивая схема шардирования со временем **расбалансируется** — иначе не бывает.
- Нужно мониторить **распределение данных, нагрузку по шардом и горячие ключи**, а также иметь **триггеры на rebalance**.
- Без мониторинга легко получить **одну перегретую ноду** и «холодные» шарды.

---

### Простыми словами

Шардирование — это не «один раз настроил и забыл». Со временем:

- появляются новые крупные клиенты/тенанты;
- меняются паттерны трафика;
- какие‑то ключи становятся **сверхпопулярными**;
- данные растут неравномерно.

Если не смотреть на это глазами метрик, можно **очень поздно заметить**, что:

- один шард упёрся в CPU/Disk;
- latency запросов через этот шард ползёт вверх;
- на других шардом в это время скука.

---

### Теория и правила

Что мониторить:

- **Размер шардов**:
  - количество строк;
  - объём на диске;
  - скорость роста.
- **Нагрузка на шарды**:
  - QPS/throughput по шардом;
  - latency запросов;
  - CPU/I/O.
- **Горячие ключи**:
  - топ‑N ключей по числу запросов;
  - ключи с аномально высокой долей трафика.

Триггеры на rebalance:

- если размер/нагрузка самого тяжёлого шарда в **несколько раз** превышает средний;
- если есть ключи/тенанты, которые **забирают большую часть ресурсов**;
- если latency через конкретный шард стабильно выше остальных.

---

### Пошагово/Практика

1. **Собери базовые метрики по шардом.**
   - Для Citus/Vitess — есть свои представления/метрики.
   - Для self‑made шардирования:
     - логируй `shard_id` для каждого запроса;
     - агрегируй статистику в мониторинге.
2. **Построй дашборды.**
   - Гистограммы:
     - размера шардов;
     - QPS по шардом;
   - графики:
     - latency по шардом;
     - рост объёма данных.
3. **Отслеживай горячие ключи.**
   - логируй `key`/`tenant_id` в доступной форме;
    - периодически считай топ‑N по частоте.
4. **Определи процедуры rebalance.**
   - что считаем «перекосом»;
   - какая процедура запускается:
     - перенос части ключей/тенантов;
     - split шарда;
     - перераспределение шардов по нодам.

---

### Дополнительные примеры и приёмы мониторинга

#### Пример 1. Обнаружение горячего шарда

Представь:

- кластер из 8 шардов;
- ты строишь график:
  - QPS (запросов в секунду) по каждому шардe;
  - latency по каждому шардe.

Ты видишь:

- шарды 1–7:
  - QPS около 500;
  - p95 latency около 20–30 мс;
- шард 8:
  - QPS 3000;
  - p95 latency 150–200 мс.

Вывод:

- шард 8 **явно перегружен**;
- нужно разобраться:
  - там больше данных?
  - там больше активных ключей?

Дальше можно:

- посмотреть распределение размера шардов (объём, количество строк);
- посмотреть топ‑ключей/тенантов по этому шарду.

#### Пример 2. Обнаружение горячего ключа

Логируешь для каждого запроса:

- `shard_id`;
- `key` или `tenant_id` (в обезличенном/сжатом виде);
- тип операции (read/write).

Раз в N минут/час:

- агрегируешь:
  - сколько запросов по каждому ключу за окно;
  - сортируешь по убыванию.

Если видишь:

- ключ `tenant_42` даёт:
  - 30–50% всех запросов на одном шарде;
  - сильно больше, чем другие;
- это **горячий ключ/тенант**.

Решения:

- выделить его на отдельный шард/кластер;
- в особо экстремальных случаях — **дополнительно шардировать внутри этого тенанта** (сложно, но иногда нужно).

#### Пример 3. Понимание роста данных

Строишь графики:

- `disk_usage` по каждому шардe;
- `row_count` по каждой крупной таблице и шардe;
- скорость прироста (наклон графика).

Зачем:

- планировать:
  - когда нужно будет добавлять ноды;
  - когда потребуется resharding;
- не доводить до ситуации «внезапно кончился диск».

---

### Частые вопросы по мониторингу

**Вопрос:** достаточно ли просто смотреть на общий QPS и latency по кластеру?  
**Ответ:** нет:

- общий QPS и средняя latency **размазывают картину**;
- один шард может задыхаться, пока остальные скучают;
- так легко пропустить горячий шард до момента, когда он «упадёт».

Нужно смотреть **по шардом** и, по возможности, **по ключам/тенантам**.

---

**Вопрос:** нужно ли логировать каждый запрос с `shard_id`? Не будет ли это очень дорого?  
**Ответ:**

- логирование каждого запроса действительно может быть дорогим;
- часто делают:
  - **семплирование** (логировать, например, 1% запросов);
  - агрегацию на уровне приложения/прокси (считать счётчики в памяти, а в лог писать уже агрегаты).

Главное — иметь способ **связать симптомы (высокая latency) с конкретным шардом/ключом**.

---

**Вопрос:** как выбрать пороги для rebalance?  
**Ответ (примерный):**

- если шард:
  - по размеру > 1.5–2× среднего;
  - или по QPS/latency заметно хуже других;
- то это кандидат на:
  - перенос части данных;
  - split;
  - или вынос горячих тенантов.

Конкретные пороги зависят от системы, но важно, чтобы они **были явно определены и задокументированы**.

---

### Мини‑упражнение по мониторингу

Представь:

- у тебя есть дашборд, где:
  - по оси X — шард‑ID;
  - по оси Y — QPS и latency;
- видно, что один шард выбивается по обоим параметрам.

Подумай:

1. Какие следующие шаги ты предпримешь, чтобы понять причину?
2. Какие дополнительные метрики тебе потребуется снять (диск, CPU, размер, горячие ключи)?
3. Как ты решишь, делать ли немедленный rebalance/resharding или пока достаточно оптимизаций запросов/индексов?

---

### Картинка в голове

Мониторинг шардирования — как **регулярно смотреть на карту загрузки дорог в городе**:

- какие дороги забиты (горячие шарды);
- где есть свободные объезды (недогруженные шарды);
- когда пора **строить развязку** (resharding/rebalance).

---

### Типичные ошибки

- Не логировать `shard_id`/ключ и не иметь способа **быстро понять, какой шард тормозит**.
- Считать, что «если hash sharding, то распределение всегда ровное» и игнорировать **горячие ключи**.
- Делать rebalance «вручную и на глаз» без явных критериев и процедур.

---

### Проверь себя

<details>
<summary>Как распознать горячий шард или горячий ключ?</summary>

По метрикам:

- шард:
  - значительно больше по размеру или QPS, чем остальные;
  - у него стабильно выше latency и нагрузка (CPU/I/O);
- ключ:
  - попадает в топ‑N по числу запросов;
  - доля запросов по нему сильно выше средней.

</details>
