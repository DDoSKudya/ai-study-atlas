## §68. Полнотекстовый поиск в SQL

### 68.1. tsvector и tsquery

**Цель раздела.**  
Понять, как устроен **полнотекстовый поиск** в PostgreSQL: тип **tsvector** (лексемы с позициями), тип **tsquery** (запрос с AND/OR/NOT), конфигурации (**simple**, **english** и др.) и оператор соответствия **@@**. После раздела ты сможешь строить поиск по тексту без выноса в отдельный поисковый движок.

**В этом разделе главное (три строки).** 1) **tsvector** — нормализованное представление текста: список лексем (слов) с позициями; получают через **to_tsvector('config', text)**. 2) **tsquery** — запрос поиска: лексемы и операторы & (AND), | (OR), ! (NOT); получают через **to_tsquery**, **plainto_tsquery**, **websearch_to_tsquery**. 3) Оператор **@@** — «текст соответствует запросу»: `to_tsvector('english', body) @@ to_tsquery('english', 'word & search')`.

#### Термины (расшифровка)

- **tsvector** — тип «вектор лексем»: текст разбит на слова (лексемы), приведённые к нормальной форме (стемминг для английского и др.), с номерами позиций. Используется для индексации и поиска. Создание: **to_tsvector('config', text)** или **to_tsvector(text)** (конфигурация по умолчанию).
- **tsquery** — тип «запрос полнотекстового поиска»: лексемы и операторы & (AND), | (OR), ! (NOT), возможно в скобках. **to_tsquery('config', 'word1 & word2')** — строгий синтаксис (лексемы через &, |, !). **plainto_tsquery('config', 'phrase')** — фраза разбивается на слова и соединяется через & (все слова должны быть). **websearch_to_tsquery('config', 'phrase')** — как в поисковой строке: кавычки для фразы, минус для исключения.
- **Конфигурация (text search configuration)** — набор правил: какой словарь использовать, стемминг, стоп-слова. Встроенные: **simple** (без стемминга, без стоп-слов), **english** (английский стемминг и стоп-слова). Для русского нужен дополнительный словарь (например, расширение или кастомный).
- **Оператор @@** — левый операнд (tsvector) соответствует правому (tsquery). В WHERE: `WHERE to_tsvector('english', body) @@ to_tsquery('english', 'search & term');`.

#### Теория и правила (подробно)

- **Зачем конфигурация.** simple — быстрее, без стемминга; для точного совпадения слов. english (и др. языки) — стемминг (running → run), стоп-слова (the, a) отбрасываются — поиск по смыслу.
- **Индекс.** GIN по tsvector: `CREATE INDEX ON t USING GIN (to_tsvector('english', body));` или по отдельному столбцу типа tsvector, который заполняется триггером или выражением.

#### Пошагово: как сделать полнотекстовый поиск по столбцу

1. **Преобразовать текст в tsvector:** `to_tsvector('english', body)` — текст разбивается на слова (лексемы), для english применяется стемминг (jumps → jump), стоп-слова (the, over) отбрасываются. Результат — список лексем с позициями.
2. **Сформировать запрос tsquery:** `to_tsquery('english', 'quick & fox')` — оба слова должны быть (оператор &). Или **plainto_tsquery('english', 'quick brown fox')** — фраза разбивается на слова и соединяется через & (все слова должны быть).
3. **Проверить соответствие:** `to_tsvector('english', body) @@ to_tsquery('english', 'quick & fox')` — true, если в body есть оба слова (с учётом стемминга).
4. **Ускорить поиск:** создать GIN-индекс по выражению: `CREATE INDEX ON articles USING GIN (to_tsvector('english', body));` — тогда WHERE с @@ будет использовать индекс.

#### Примеры (SQL) с пояснениями

```sql
-- Текст превращается в список лексем с позициями (english: стемминг, стоп-слова убраны)
SELECT to_tsvector('english', 'The quick brown fox jumps over the lazy dog');
-- Результат: 'brown':3 'dog':9 'fox':4 'jump':5 'lazi':8 'quick':2
-- (jumps → jump, lazy → lazi, the/over отброшены)

-- Поиск: оба слова должны быть в тексте
SELECT * FROM articles
WHERE to_tsvector('english', body) @@ to_tsquery('english', 'quick & fox');

-- plainto_tsquery: фраза «quick brown fox» → запрос quick & brown & fox (все три слова)
SELECT plainto_tsquery('english', 'quick brown fox');

-- GIN-индекс по tsvector — быстрый поиск по @@
CREATE INDEX ON articles USING GIN (to_tsvector('english', body));
```

#### Разбор одного примера по шагам: полнотекстовый поиск по фразе

Разберём запрос: **найти статьи, где в body есть оба слова «quick» и «fox»**.

1. **Текст в таблице** — например, строка body: «The quick brown fox jumps over the lazy dog».
2. **to_tsvector('english', body)** разбирает этот текст: разбивает на слова, для конфигурации **english** отбрасывает стоп-слова (the, over) и приводит слова к основе — **jumps** → jump, **lazy** → lazi. Получается список лексем с позициями: 'brown':3 'dog':9 'fox':4 'jump':5 'lazi':8 'quick':2. Это значение типа **tsvector**.
3. **to_tsquery('english', 'quick & fox')** строит запрос: «должны быть лексемы quick И fox». Оператор **&** — логическое И. Получается значение типа **tsquery**.
4. **Оператор @@** проверяет: «tsvector слева соответствует tsquery справа?» В нашем tsvector есть и 'quick':2, и 'fox':4 — да, соответствие есть, условие истинно.
5. В **WHERE** мы пишем: `WHERE to_tsvector('english', body) @@ to_tsquery('english', 'quick & fox')`. Строки, для которых это true, попадают в результат. Чтобы не вызывать to_tsvector для каждой строки при каждом запросе, создают **GIN-индекс** по to_tsvector('english', body) — тогда поиск идёт по индексу.

**Итог:** текст → to_tsvector (список лексем), запрос → to_tsquery (слова и &, |, !), соответствие — **@@**. Для скорости — GIN по tsvector.

#### Что будет, если не создать GIN-индекс по tsvector

Без индекса условие `WHERE to_tsvector('english', body) @@ to_tsquery(...)` заставит СУБД для **каждой** строки вызывать to_tsvector и проверять @@ — полный скан таблицы. На больших таблицах это медленно. **GIN-индекс** по tsvector хранит «лексема → список строк»; поиск по запросу выполняется по индексу — в разы быстрее. Всегда создавай такой индекс, если полнотекстовый поиск по столбцу используется в WHERE.

#### Часто путают

**simple и english.** **simple** — без стемминга и без стоп-слов; слово хранится «как есть». **english** — стемминг (running → run), стоп-слова (the, a, is) отбрасываются. Для точного совпадения слов — simple; для поиска «по смыслу» (найти и run, и running) — english.

**to_tsquery и plainto_tsquery.** **to_tsquery** — строгий синтаксис: лексемы через &, |, ! (например, 'word1 & word2'). **plainto_tsquery** — передаёшь фразу «слово1 слово2»; она разбивается на слова и соединяется через & (все слова должны быть). Удобно для поиска по фразе от пользователя.

#### Простыми словами

**tsvector** — текст превращён в «список слов»: лишнее (the, a) убрано, слова приведены к одной форме (jumps → jump). **tsquery** — запрос: «найти документы, где есть слово1 И слово2» (&) или «слово1 ИЛИ слово2» (|). **@@** — «этот текст подходит под этот запрос?» Для быстрого поиска создай **GIN-индекс** по to_tsvector('english', body). Конфигурация **english** — стемминг и стоп-слова; **simple** — слова как есть, без стемминга.

#### Как запомнить

**Текст → to_tsvector**, **запрос → to_tsquery** (или plainto_tsquery). **Соответствие** — оператор **@@**. Для скорости — **GIN-индекс** по to_tsvector('config', column). Конфигурация **english** — стемминг и стоп-слова; **simple** — без них.

#### Картинка в голове

Текст — это мешок слов. **tsvector** — этот мешок разобран: каждое слово приведено к «нормальной» форме (jump вместо jumps), мусор (the, a) выброшен, осталось только то, по чему ищем. **tsquery** — «найти документы, где есть слово1 И слово2». **@@** — «этот мешок подходит под этот запрос?» Индекс GIN — как алфавитный указатель: по слову сразу находишь список документов, где оно есть.

#### Проверь себя

1. Что делает to_tsvector('english', body)?  
<details><summary>Ответ</summary> Преобразует текст **body** в **tsvector**: разбивает на слова (лексемы), для конфигурации **english** применяет **стемминг** (jumps → jump, running → run) и убирает **стоп-слова** (the, a, is). Результат — список лексем с позициями, по которому потом ищут через **@@** и **tsquery**.</details>

2. Зачем создавать GIN-индекс по to_tsvector('english', body)?  
<details><summary>Ответ</summary> Без индекса условие **WHERE to_tsvector('english', body) @@ query** заставляет для **каждой** строки вызывать to_tsvector и проверять @@ — **полный скан** таблицы. **GIN-индекс** по tsvector хранит «лексема → список строк»; поиск по запросу выполняется по индексу — в разы быстрее на больших таблицах.</details>

#### Если прочитал и не понял

**tsvector** — это «список слов» из текста (после стемминга и без стоп-слов). **tsquery** — запрос «найти документы, где есть такие-то слова» (через & — все, через | — любое). **@@** — «текст подходит под запрос?» Для быстрого поиска создай **GIN-индекс** по to_tsvector('english', body). Конфигурация **english** даёт стемминг (running → run); **simple** — слова как есть.

**Повторение одной фразой.** Текст → **to_tsvector**, запрос → **to_tsquery** (или plainto_tsquery); соответствие — **@@**; для скорости — **GIN-индекс** по tsvector; **english** — стемминг и стоп-слова, **simple** — без них.

**Ещё раз в трёх предложениях.** 1) **tsvector** — это список лексем (слов) из текста после стемминга и без стоп-слов; получают через **to_tsvector('english', body)**. 2) **tsquery** — запрос «найти документы, где есть такие-то слова» (операторы &, |, !); получают через **to_tsquery** или **plainto_tsquery**. 3) Оператор **@@** проверяет соответствие текста запросу; для быстрого поиска создай **GIN-индекс** по to_tsvector('english', body).

#### Запомните

- **tsvector** — лексемы с позициями; **to_tsvector('config', text)**; конфигурации simple, english.
- **tsquery** — запрос с &, |, !; **to_tsquery**, **plainto_tsquery**, **websearch_to_tsquery**.
- **@@** — соответствие; для быстрого поиска — **GIN-индекс** по tsvector.

### 68.2. Ранжирование и подсветка

**Цель раздела.**  
Использовать **ts_rank()** и **ts_rank_cd()** для ранжирования результатов полнотекстового поиска и **ts_headline()** для подсветки найденных фрагментов в тексте. После раздела ты сможешь выводить «релевантность» и выделять совпадения в сниппетах.

**В этом разделе главное (три строки).** 1) **ts_rank(tsvector, tsquery)** — оценка релевантности (частота и близость лексем); выше — «более релевантный». 2) **ts_rank_cd** — учитывает близость (cover density) совпадений. 3) **ts_headline('config', text, tsquery, options)** — возвращает фрагмент текста с выделением совпадений (по умолчанию <b>...</b>); опции: MaxWords, MinWords, StartSel, StopSel.

#### Термины (расшифровка)

- **ts_rank(tsvector, tsquery [, normalization])** — числовая оценка релевантности документа запросу. Учитывает частоту лексем и их позиции. Используется в ORDER BY для сортировки по релевантности: `ORDER BY ts_rank(to_tsvector('english', body), query) DESC`.
- **ts_rank_cd** — вариант, учитывающий «плотность покрытия» (cover density): насколько близко друг к другу совпадающие слова; часто даёт более естественный порядок.
- **ts_headline('config', text, tsquery [, options])** — извлекает короткий фрагмент текста с подсветкой совпадений. Опции: MaxWords, MinWords (размер фрагмента), StartSel, StopSel (теги выделения, по умолчанию <b> и </b>).

#### Примеры (SQL)

```sql
-- Сортировка по релевантности
SELECT id, body, ts_rank(to_tsvector('english', body), q) AS rank
FROM articles, to_tsquery('english', 'search & term') q
WHERE to_tsvector('english', body) @@ q
ORDER BY rank DESC;

-- Подсветка совпадений
SELECT ts_headline('english', body, to_tsquery('english', 'fox'), 'MaxWords=35, MinWords=15');
```

#### Простыми словами

**ts_rank** даёт число — «насколько документ подходит под запрос»: чем чаще и ближе совпадают слова, тем выше число. Сортируй по нему: **ORDER BY ts_rank(...) DESC** — сначала самые подходящие документы. **ts_headline** — «выдернуть кусок текста и выделить в нём найденные слова» (тегом <b> по умолчанию); это «сниппет» для пользователя — видно, где и как запрос встретился в тексте.

#### Картинка в голове

**@@** отбирает документы «есть совпадения». **ts_rank** — ставит каждому документу «оценку»: 0.5, 0.8, 0.3… Сортируешь по оценке — сверху самые релевантные. **ts_headline** — вырезает из документа короткий кусок и подсвечивает в нём найденные слова; как в поисковике: «…**quick** **fox** jumps…».

#### Как запомнить

**ts_rank** — «насколько документ релевантен запросу»; используй в ORDER BY для сортировки по релевантности. **ts_headline** — «выдернуть фрагмент текста с подсветкой найденных слов» (сниппет для пользователя).

#### Проверь себя

1. Зачем нужен ts_rank, если уже есть поиск через @@?  
<details><summary>Ответ</summary> **@@** только отбирает документы, где есть совпадения с запросом. **ts_rank** даёт **число** — насколько документ релевантен (частота и близость слов). Это число используют в **ORDER BY** — чтобы показать пользователю сначала самые подходящие документы, а не в произвольном порядке.</details>

2. Что возвращает ts_headline и когда это нужно?  
<details><summary>Ответ</summary> **ts_headline** возвращает **короткий фрагмент** текста документа с **подсветкой** найденных слов (по умолчанию тегом &lt;b&gt;). Нужно для вывода пользователю «сниппета» — куска текста вокруг совпадения, чтобы было видно, где и как запрос встретился в документе.</details>

#### Если прочитал и не понял

**ts_rank** даёт число — оценку релевантности документа запросу; чем выше — тем «подходящее» документ. Сортируй по нему: ORDER BY ts_rank(...) DESC. **ts_headline** возвращает короткий кусок текста с выделением найденных слов (по умолчанию тегом <b>); опции MaxWords, MinWords задают размер фрагмента.

**Повторение одной фразой.** **ts_rank** — оценка релевантности для ORDER BY; **ts_headline** — фрагмент текста с подсветкой совпадений (сниппет).

#### Запомните

- **ts_rank** / **ts_rank_cd** — релевантность для ORDER BY; **ts_headline** — фрагмент с подсветкой совпадений.
- **Словари** — стемминг и стоп-слова задаются конфигурацией; для русского — отдельный словарь/расширение.

### 68.3. Индексы и триграммы

**Цель раздела.**  
Закрепить создание **GIN-индекса по tsvector** для полнотекстового поиска и познакомиться с расширением **pg_trgm** для **нечёткого (триграммного) поиска**: **similarity()**, оператор **%**, индексы GIN по выражению для LIKE/ILIKE.

**В этом разделе главное (три строки).** 1) **GIN-индекс по tsvector** — быстрый полнотекстовый поиск по столбцу или выражению. 2) **pg_trgm** — разбивает текст на триграммы (тройки символов); **similarity(a, b)** — число от 0 до 1 (похожесть); **a % b** — true, если similarity выше порога. 3) **GIN-индекс по expression USING gin(col gin_trgm_ops)** позволяет ускорить **LIKE '%substr%'** и **ILIKE** для нечёткого поиска.

#### Термины (расшифровка)

- **GIN (Generalized Inverted Index)** по tsvector — индекс «лексема → список строк»; поиск по tsquery выполняется через пересечение/объединение списков. Создание: `CREATE INDEX ON t USING GIN (to_tsvector('english', col));`.
- **pg_trgm** — расширение для триграмм: текст представляется множеством триграмм (подстрок из 3 символов). **similarity(str1, str2)** — коэффициент от 0 до 1 (чем выше, тем похожее). **str1 % str2** — true, если similarity(str1, str2) выше текущего порога (параметр pg_trgm.similarity_threshold).
- **gin_trgm_ops** — класс оператора GIN для триграмм; индекс по выражению с **gin_trgm_ops** позволяет использовать **LIKE '%...%'**, **ILIKE** и **%** (similarity) с индексом.

#### Простыми словами

**Триграммы** — подстроки по 3 символа (например, «Joh» — из «John»). **similarity(a, b)** — число от 0 до 1: насколько строки похожи (учёт опечаток, «Jhon» и «John»). **a % b** — true, если похожесть выше порога (удобно в WHERE для нечёткого поиска). **pg_trgm** даёт эти функции; **GIN-индекс с gin_trgm_ops** по столбцу ускоряет **LIKE '%...%'** и нечёткий поиск — без индекса был бы полный скан.

#### Картинка в голове

Текст разбит на «тройки букв» (триграммы). Две строки похожи, если у них много общих троек. **similarity** — «процент совпадения» (0–1). **%** — «похоже ли достаточно?» (выше порога — true). GIN с **gin_trgm_ops** — каталог по этим тройкам: запрос LIKE '%substr%' или name % 'Jhon' ищется по индексу, а не по всей таблице.

#### Примеры (SQL)

```sql
-- Включение расширения
CREATE EXTENSION IF NOT EXISTS pg_trgm;

-- Нечёткий поиск по имени
SELECT * FROM users WHERE similarity(name, 'Jhon') > 0.3;
SELECT * FROM users WHERE name % 'Jhon';

-- GIN для LIKE/ILIKE (триграммы)
CREATE INDEX ON users USING GIN (name gin_trgm_ops);
```

#### Как запомнить

**GIN по tsvector** — быстрый полнотекстовый поиск по @@. **pg_trgm** — разбить текст на тройки символов; **similarity(a,b)** — похожесть 0–1; **a % b** — true, если похожесть выше порога. GIN с **gin_trgm_ops** ускоряет LIKE '%...%', ILIKE и нечёткий поиск.

#### Проверь себя

1. Чем триграммный поиск (pg_trgm) отличается от полнотекстового (tsvector)?  
<details><summary>Ответ</summary> **Полнотекстовый** (tsvector) — по **словам** (лексемам), с стеммингом и стоп-словами; подходит для поиска по смыслу («run» найдёт и «running»). **Триграммный** (pg_trgm) — по **подстрокам из 3 символов**; подходит для **нечёткого** поиска и опечаток (например, «Jhon» найдёт «John») и для ускорения **LIKE '%...%'**.</details>

2. Зачем создавать GIN-индекс с gin_trgm_ops на текстовый столбец?  
<details><summary>Ответ</summary> Без индекса запросы **LIKE '%substr%'** и **ILIKE** выполняют **полный скан** таблицы. GIN-индекс с **gin_trgm_ops** по столбцу позволяет использовать индекс для таких условий и для оператора **%** (similarity) — поиск становится значительно быстрее на больших таблицах.</details>

#### Если прочитал и не понял

**Триграммы** — подстроки по 3 символа. **similarity(str1, str2)** — число от 0 до 1 (насколько строки похожи). **str1 % str2** — true, если similarity выше порога (удобно для поиска с опечатками). Расширение **pg_trgm** даёт эти функции; GIN-индекс с **gin_trgm_ops** по столбцу ускоряет LIKE и нечёткий поиск.

**Повторение одной фразой.** **pg_trgm** — триграммы (тройки символов); **similarity(a,b)** — похожесть 0–1; **a % b** — true, если похожесть выше порога; GIN с **gin_trgm_ops** ускоряет LIKE/ILIKE и нечёткий поиск.

#### Запомните

- **GIN по tsvector** — полнотекстовый поиск; **pg_trgm** — триграммы, **similarity()**, **%**; GIN с **gin_trgm_ops** ускоряет LIKE/ILIKE и нечёткий поиск.
