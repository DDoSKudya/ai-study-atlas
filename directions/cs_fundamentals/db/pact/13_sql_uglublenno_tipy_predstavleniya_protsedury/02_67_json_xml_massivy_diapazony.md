## §67. JSON, XML, массивы, диапазоны

### 67.1. JSON и JSONB

**Цель раздела.**  
Научиться работать с **JSON и JSONB** в PostgreSQL: операторы извлечения (**->**, **->>**, **#>**, **#>>**), проверка вхождения (**@>**, **?**, **?|**, **?&**), индексы **GIN** и при необходимости **jsonpath**. После раздела ты сможешь хранить и запрашивать полуструктурированные данные без выноса в отдельную документную БД.

**В этом разделе главное (три строки).** 1) **JSONB** — бинарный формат JSON в PostgreSQL; поддерживает индексы (GIN) и быстрые операции; ключи хранятся без дубликатов и в нормализованном виде. 2) **->** возвращает значение **как JSON** (можно цепочкой), **->>** — **как текст**; **#**-операторы работают по пути (массив ключей). 3) **@>** — «левый JSON содержит правый»; **?** — «есть ключ»; для поиска по содержимому создают **GIN-индекс** по столбцу jsonb.

#### Термины (расшифровка)

- **JSON** — тип «как есть»: хранит исходный текст, при каждом обращении парсит. **JSONB** — бинарное представление: разобранная структура, дубликаты ключей убраны, порядок ключей не гарантируется; запросы и индексация эффективнее. Для большинства задач выбирают **JSONB**.
- **-> ключ** — извлечь значение по ключу **как jsonb** (объект или массив можно дальше обрабатывать). **->> ключ** — извлечь значение **как text** (для сравнения и вывода). Пример: `data->'user'->>'name'` — из вложенного объекта user поле name как текст.
- **#> путь** — путь задаётся массивом ключей: `data #> '{user,address,city}'`. **#>>** — то же, но результат как text.
- **@>** — оператор «содержит»: левый jsonb содержит правый (включая вложенность). Используется для поиска: `WHERE payload @> '{"status": "active"}';`.
- **? ключ** — «есть ли такой ключ на верхнем уровне». **?| array** — есть хотя бы один из ключей. **?& array** — есть все перечисленные ключи.
- **GIN-индекс по jsonb** — позволяет быстро искать по @>, ?, ?|, ?&. Создание: `CREATE INDEX ON t USING GIN (jsonb_column);` или по выражению (например, по конкретному ключу).
- **jsonpath** (SQL/JSON Path) — язык путей и выражений для выборки и фильтрации в JSON. В PostgreSQL: `jsonb_path_query(data, '$.items[*] ? (@.price > 10)')` и аналоги; поддерживается в индексах по выражению.

#### Теория и правила (подробно)

- **Когда использовать JSONB.** Гибкая схема (разные документы с разным набором полей), конфигурации, лог-события, API-ответы, хранимые «как есть». Когда структура жёсткая и запросы по многим полям — чаще выгоднее обычные столбцы.
- **Индексы.** GIN по всему столбцу — универсальный поиск по любому ключу/значению. Если часто фильтруют по одному ключу, можно создать выражение: `CREATE INDEX ON t USING GIN ((column->'key'));`.

#### Пошагово: как извлечь значение из JSONB (-> и ->>)

1. **Один уровень:** `data->'key'` — вернёт значение **как jsonb** (объект или массив можно дальше разбирать). `data->>'key'` — вернёт значение **как текст** (удобно для WHERE и вывода).
2. **Вложенность:** цепочка: `data->'user'->'address'->>'city'` — из объекта user берётся address, из него как текст — city.
3. **Путь массивом:** `data #> '{user,address,city}'` — путь задаётся массивом ключей; результат jsonb. `data #>> '{user,address,city}'` — то же, но результат text.
4. **Проверка «есть ключ»:** `data ? 'user_id'` — true, если на верхнем уровне есть ключ user_id.

#### Примеры (SQL) с пояснениями

```sql
-- -> возвращает jsonb, ->> возвращает text
SELECT payload->'user'->>'name' FROM events;   -- из payload.user берём name как текст
SELECT payload #>> '{user,address,city}' FROM events;  -- путь: user → address → city, результат text

-- Поиск: «документ содержит этот фрагмент» (включая вложенность)
SELECT * FROM events WHERE payload @> '{"type": "click"}';

-- Поиск: «есть ли такой ключ на верхнем уровне»
SELECT * FROM events WHERE payload ? 'user_id';

-- GIN-индекс ускоряет @>, ?, ?|, ?&
CREATE INDEX ON events USING GIN (payload);

-- jsonpath: «все элементы массива items, у которых qty > 0»; взять первый
SELECT jsonb_path_query_first(payload, '$.items[*] ? (@.qty > 0)') FROM orders;
```

#### Что будет, если использовать JSON вместо JSONB

Тип **JSON** в PostgreSQL хранит текст «как есть»; при каждом обращении к полю (->, ->>, @>) текст парсится заново. При частых запросах и фильтрации это медленнее. **JSONB** хранит разобранную структуру в бинарном виде; дубликаты ключей убираются, порядок ключей не гарантируется; запросы и индексы (GIN) работают быстрее. **Итог:** для запросов и индексации используй **JSONB**; тип JSON оставляй только если нужна точная сохранность формата (пробелы, порядок ключей).

#### Часто путают

**-> и ->>.** **->** даёт **jsonb** (или json) — можно продолжать цепочку: data->'a'->'b'. **->>** даёт **text** — для сравнения в WHERE и вывода; цепочку после ->> не продолжают (результат уже строка). Запомни: одна стрелка — «объект/массив», две стрелки — «текст».

**@> и ?.** **@>** — «левый документ **содержит** правый» (включая вложенные ключи и значения). **?** — «есть ли **ключ** с таким именем на верхнем уровне» (значение не проверяется).

#### Простыми словами

**JSONB** — это «коробка с подписями»: внутри лежат пары «ключ — значение» (и вложенные коробки). **->** — «достань по ключу и оставь коробкой» (можно снова достать вложенное). **->>** — «достань по ключу и отдай как текст» (для сравнения и вывода). **@>** — «в этой коробке есть такой кусок?» (поиск по содержимому). **?** — «есть ли ключ с таким именем?» Для быстрого поиска по содержимому создай **GIN-индекс** по столбцу с jsonb.

#### Картинка в голове

Документ JSON — как вложенные коробки с наклейками (ключами). **->** — открываешь коробку по наклейке и достаёшь ещё коробку (или массив). **->>** — открываешь и достаёшь уже «готовое» значение (текст, число). **@>** — «в этой коробке лежит такой же маленький кусочек?» (поиск). **?** — «есть ли наклейка с таким именем?» GIN-индекс — как каталог: по «содержимому» быстро находишь нужные документы.

#### Как запомнить

**->** = jsonb, **->>** = text. **@>** = «содержит» (по содержимому), **?** = «есть ключ». Для поиска по содержимому — **GIN-индекс** по столбцу jsonb.

#### Проверь себя

1. Чем -> отличается от ->> при извлечении из JSONB?  
<details><summary>Ответ</summary> **->** возвращает значение **как jsonb** — можно продолжать цепочку: data->'user'->'address'. **->>** возвращает значение **как text** — для сравнения в WHERE и вывода; цепочку после ->> не продолжают (результат уже строка). Одна стрелка — объект/массив, две стрелки — текст.</details>

2. Зачем создавать GIN-индекс по столбцу с JSONB?  
<details><summary>Ответ</summary> Условия **@>**, **?**, **?|**, **?&** без индекса требуют **полного скана** таблицы — каждая строка проверяется. **GIN-индекс** по jsonb хранит «содержимое → список строк»; поиск по @> и ? выполняется по индексу — в разы быстрее на больших таблицах.</details>

#### Если прочитал и не понял

**JSONB** — это JSON, но внутри БД хранится в разобранном виде (быстрее запросы). **->** — «достать поле как объект» (можно дальше .field). **->>** — «достать поле как текст». **@>** — «найти строки, где в документе есть такой кусок». **?** — «есть ли такой ключ». Для быстрого поиска по полям создай **GIN-индекс** на столбец с jsonb.

**Повторение одной фразой.** Для полуструктурированных данных в PostgreSQL используй **JSONB**; доставать поля — **->** (объект) и **->>** (текст); искать по содержимому — **@>** и **?**, с **GIN-индексом** по столбцу.

**Ещё раз в трёх предложениях.** 1) **JSONB** хранит JSON в разобранном виде; запросы и индексы быстрее, чем у типа JSON. 2) **->** достаёт значение как jsonb (можно цепочкой), **->>** — как текст; **@>** — «документ содержит этот фрагмент», **?** — «есть ключ». 3) Для быстрого поиска по содержимому создай **GIN-индекс** по столбцу с jsonb; без индекса условия @> и ? дают полный скан таблицы.

#### Запомните

- **JSONB** — бинарный JSON; операторы **->** (jsonb), **->>** (text), **#>, #>>** (путь); **@>** (содержит), **?** (есть ключ).
- Для поиска по содержимому — **GIN-индекс** по jsonb-столбцу (или по выражению).
- **jsonpath** — сложная выборка и фильтрация внутри JSON; поддерживается в функциях и индексах.

### 67.2. Массивы и диапазоны

**Цель раздела.**  
Использовать типы **массив (ARRAY)** и **диапазон (range)** в PostgreSQL: создание, операторы **&&** (пересечение), **@>** (содержит), **<@** (содержится в), индексы **GIN** для массивов и **GiST** для диапазонов; функция **unnest()** для разворачивания массива в строки.

**В этом разделе главное (три строки).** 1) **Массив** — столбец может хранить несколько значений одного типа: `integer[]`, `text[]`; литерал `ARRAY['a','b']` или `'{a,b}'`. 2) **Диапазон** — непрерывный интервал: **int4range**, **int8range**, **tstzrange**, **daterange** и др.; границы включающие/исключающие `[)` по умолчанию. 3) **&&** — пересекаются ли массивы/диапазоны; **@>** / **<@** — содержит / содержится в; для массивов — **GIN**, для диапазонов — **GiST**.

#### Термины (расшифровка)

- **Массив (ARRAY)** — тип «множество значений» одного типа. Определение столбца: `tags text[]`, `ids integer[]`. Литералы: `ARRAY['a','b']`, `'{1,2,3}'::int[]`. Обращение по индексу (с 1): `arr[1]`. Длина: `array_length(arr, 1)`. **unnest(arr)** — развернуть в набор строк (можно в FROM с LATERAL).
- **Операторы массивов:** **&&** — пересечение (есть общий элемент); **@>** — левый содержит правый; **<@** — левый содержится в правом; **@>** для одного элемента: `arr @> ARRAY['x']`. Индекс по элементу: `arr[1]` или GIN по всему массиву для @>, &&, <@.
- **Диапазон (range)** — тип «интервал»: **int4range**, **int8range**, **tstzrange**, **daterange**, **numrange** и др. Литерал: `'[2020-01-01, 2021-01-01)'::tstzrange` — от включительно до исключительно. **Операторы:** **&&** — пересекаются; **@>** — содержит (диапазон или точку); **<@** — содержится в; **isempty()**. Индекс: **GiST** по столбцу диапазона для быстрого поиска по пересечению и вложенности.

#### Пошагово: как искать по массиву и «развернуть» его в строки

1. **Проверить «есть ли элемент»:** `tags && ARRAY['sql']` — true, если в массиве tags есть хотя бы один из перечисленных элементов (пересечение непустое).
2. **Проверить «массив содержит другой»:** `tags @> ARRAY['sql','db']` — true, если в tags есть и 'sql', и 'db' (левый содержит правый).
3. **Развернуть массив в строки:** `SELECT unnest(tags) AS tag FROM posts` — каждая запись даёт несколько строк (по одной на элемент). Удобно для JOIN с другими таблицами или для подсчёта по тегам.
4. **Индекс для быстрого поиска:** `CREATE INDEX ON posts USING GIN (tags);` — тогда условия `tags && ARRAY['x']` и `tags @> ARRAY['x']` используют индекс.

#### Примеры (SQL) с пояснениями

```sql
-- Массивы: столбец tags хранит несколько тегов (строк)
CREATE TABLE posts (id serial PRIMARY KEY, tags text[]);
INSERT INTO posts (tags) VALUES (ARRAY['sql', 'db']);
-- Найти посты, у которых есть тег 'sql' (или любой из списка)
SELECT * FROM posts WHERE tags && ARRAY['sql'];
-- Развернуть теги в отдельные строки (каждый тег — строка)
SELECT id, unnest(tags) AS tag FROM posts;

-- Диапазоны: период «от и до» (границы [ включительно, ) исключительно по умолчанию)
CREATE TABLE bookings (id serial PRIMARY KEY, period tstzrange);
INSERT INTO bookings (period) VALUES ('[2024-06-01, 2024-06-07)'::tstzrange);
-- Найти бронирования, пересекающиеся с 5–10 июня
SELECT * FROM bookings WHERE period && tstzrange('2024-06-05', '2024-06-10');

-- Индексы: GIN для массивов, GiST для диапазонов
CREATE INDEX ON posts USING GIN (tags);
CREATE INDEX ON bookings USING GiST (period);
```

#### Что будет, если искать по массиву без GIN-индекса

При большом числе строк условие `WHERE tags && ARRAY['sql']` заставит СУБД перебрать все строки и проверить массив в каждой — полный скан таблицы. С **GIN-индексом** по столбцу tags поиск «есть ли элемент» выполняется по индексу — быстрее. Если часто фильтруешь по элементам массива или по пересечению — создай **GIN-индекс** на массив.

#### Часто путают

**&& для массивов и для диапазонов.** В обоих случаях **&&** значит «пересекаются»: для массивов — есть общий элемент; для диапазонов — интервалы пересекаются по времени или по числам. Синтаксис один, смысл похожий.

**Границы диапазона [ и ).** Квадратная скобка **]** — граница **включена**, круглая **)** — **исключена**. `[2024-06-01, 2024-06-07)` — с 1 июня 00:00 включительно по 7 июня **не** включительно (до 7 июня 00:00). Так обычно задают «день по 6 июня включительно».

#### Простыми словами

**Массив** — в одной ячейке лежит несколько значений (теги, список id). «Есть ли тег sql?» — `tags && ARRAY['sql']`. «Развернуть в отдельные строки» — **unnest(tags)**. **Диапазон** — «от и до»: один столбец хранит интервал (даты, числа). «Пересекается ли с другим периодом?» — **period && другой_период**. Для быстрого поиска по элементам массива — **GIN**, по диапазону — **GiST**.

#### Картинка в голове

**Массив** — полка с несколькими коробками в одной ячейке (тег1, тег2, тег3). **&&** — «есть ли на полке хотя бы одна из этих коробок?» **unnest** — выложить коробки в ряд (каждая — отдельная строка). **Диапазон** — одна «линейка» от точки A до точки B; **&&** — «эта линейка пересекается с той?» GIN по массиву — быстрый поиск «есть ли такой элемент»; GiST по диапазону — быстрый поиск «пересекается с этим интервалом».

#### Как запомнить

**Массив:** тип `type[]`, искать по элементу — **&&** или **@>**; развернуть в строки — **unnest()**; индекс — **GIN**. **Диапазон:** тип int4range, tstzrange и т.д.; пересечение — **&&**; индекс — **GiST**; границы **)** по умолчанию — верхняя граница исключена.

#### Проверь себя

1. Как найти строки, где в массиве tags есть тег 'sql'?  
<details><summary>Ответ</summary> **tags && ARRAY['sql']** — оператор **&&** для массивов значит «есть пересечение» (хотя бы один общий элемент). Или **tags @> ARRAY['sql']** — «левый массив содержит правый» (в данном случае один элемент). Для быстрого поиска создай **GIN-индекс** по столбцу tags.</details>

2. Что делает unnest(tags)?  
<details><summary>Ответ</summary> **unnest(tags)** разворачивает массив в **набор строк**: каждый элемент массива становится отдельной строкой в результате. Удобно для JOIN с другими таблицами или для подсчёта «сколько раз встречается каждый тег» (GROUP BY unnest(tags)).</details>

#### Если прочитал и не понял

**Массив** — в одном столбце лежит несколько значений (например, теги). Искать «есть тег sql» — `tags && ARRAY['sql']`. Чтобы получить каждое значение отдельной строкой — **unnest(tags)**. **Диапазон** — «от и до» (даты, числа); проверка «пересекается с другим периодом» — **period && другой_период**. Для быстрого поиска: по массиву — **GIN**, по диапазону — **GiST**.

**Повторение одной фразой.** Массив = несколько значений в одном столбце; искать по элементу — **&&** или **@>**; развернуть — **unnest()**; индекс **GIN**. Диапазон = «от и до»; пересечение — **&&**; индекс **GiST**; граница **)** — верхняя исключена.

**Разбор одного примера по шагам: поиск по тегу и unnest.** Допустим, в таблице posts есть столбец **tags** типа text[] (массив тегов). Как найти посты с тегом 'sql' и как «развернуть» теги в отдельные строки?

1. **Поиск «есть тег sql»:** условие **tags && ARRAY['sql']**. Оператор **&&** для массивов значит «есть пересечение» — хотя бы один общий элемент. Если в массиве tags есть 'sql', условие истинно. Альтернатива: **tags @> ARRAY['sql']** — «левый массив содержит правый» (в данном случае один элемент).
2. **Чтобы поиск был быстрым** на большой таблице, создай индекс: **CREATE INDEX ON posts USING GIN (tags);** — тогда условия tags && ARRAY['x'] и tags @> ARRAY['x'] будут использовать индекс.
3. **Развернуть массив в строки:** **SELECT id, unnest(tags) AS tag FROM posts**. Функция **unnest(tags)** возвращает **набор строк**: для каждой строки таблицы каждый элемент массива становится отдельной строкой результата. Например, строка с id=1 и tags = ARRAY['sql','db'] даст две строки: (1, 'sql') и (1, 'db'). Удобно для подсчёта «сколько постов у каждого тега» (GROUP BY tag) или для JOIN с таблицей тегов.

**Итог:** && и @> — поиск по элементам массива; unnest() — «выложить» элементы в отдельные строки; для быстрого поиска — GIN по массиву.

#### Запомните

- **Массивы:** тип `type[]`, операторы **&&**, **@>**, **<@**; **unnest()** для разворота; индекс **GIN** для поиска по элементам.
- **Диапазоны:** int4range, tstzrange и др.; границы `[)` по умолчанию; операторы **&&**, **@>**, **<@**; индекс **GiST**.

### 67.3. XML и сетевые типы

**Цель раздела.**  
Кратко познакомиться с типом **xml** и функциями **xpath()**, **xmlagg()**, а также с типами **inet** и **cidr** для IP-адресов и подсетей. В приложениях JSON используется чаще XML; inet/cidr полезны для сетевых полей (логи, доступ по IP, подсети).

**В этом разделе главное (три строки).** 1) Тип **xml** хранит верифицированный XML; **xpath()** извлекает узлы по XPath-выражению. 2) **inet** — один IP-адрес (IPv4 или IPv6), опционально с маской; **cidr** — блок адресов (подсеть): адрес + маска (например, 192.168.1.0/24). 3) Операторы **<<** (адрес входит в подсеть), **>>** (подсеть содержит адрес), **&&** (подсети пересекаются) — удобны для фильтрации логов и проверки доступа.

#### Термины (расшифровка)

- **xml** — тип для хранения XML-документа в PostgreSQL; документ проверяется на корректность. **xpath(xml, xpath_expr)** — извлечь узлы по XPath-выражению. В современных приложениях чаще используют **JSON/JSONB** — проще и быстрее для типичных задач.
- **inet** — тип «один IP-адрес» (IPv4 или IPv6). Можно хранить с маской (например, 192.168.1.5/32) или без. Используется для полей «IP клиента», «адрес сервера».
- **cidr** — тип «блок адресов» (подсеть): адрес + длина маски в битах. Например, 192.168.1.0/24 — адреса от 192.168.1.0 до 192.168.1.255. Удобно для проверки «входит ли этот IP в нашу сеть» или «принадлежит ли запрос диапазону офисных IP».
- **Операторы:** **inet << cidr** — «адрес входит в подсеть» (true/false). **cidr >> inet** — то же с другой стороны: «подсеть содержит адрес». **cidr && cidr** — «подсети пересекаются».

#### Пошагово: как проверить, что IP входит в подсеть

1. **Столбец с IP:** например, `ip inet` в таблице логов — хранишь адрес клиента.
2. **Проверка «IP в нашей подсети»:** `WHERE ip << '10.0.0.0/8'::cidr` — отобрать строки, где IP из диапазона 10.x.x.x (типичная «внутренняя» сеть). Оператор **<<** значит «левый inet входит в правый cidr».
3. **Несколько подсетей:** можно проверить `ip << ANY(ARRAY['10.0.0.0/8'::cidr, '192.168.0.0/16'::cidr])` — IP входит хотя бы в одну из подсетей.
4. **Индекс для быстрого поиска:** для условий по inet/cidr подходят обычный B-tree или GiST по столбцу inet/cidr (в зависимости от операторов).

#### Примеры (SQL) с пояснениями

```sql
-- inet: один адрес; cidr: подсеть (блок адресов)
SELECT '192.168.1.5'::inet << '192.168.1.0/24'::cidr;
-- true: адрес 192.168.1.5 входит в подсеть 192.168.1.0/24

-- Отобрать логи только из внутренней сети 10.x.x.x
SELECT * FROM logs WHERE ip << '10.0.0.0/8'::cidr;

-- Проверить пересечение двух подсетей
SELECT '192.168.0.0/24'::cidr && '192.168.1.0/24'::cidr;  -- false (не пересекаются)
SELECT '192.168.0.0/23'::cidr && '192.168.1.0/24'::cidr;  -- true (пересекаются)
```

#### Простыми словами

**inet** — это «один адрес» (как один дом). **cidr** — это «диапазон адресов» (улица или район). Оператор **<<** отвечает на вопрос: «этот адрес (inet) входит в этот диапазон (cidr)?» — удобно для фильтрации логов по «внутренним» или «офисным» IP. **xml** в приложениях сейчас редко нужен — чаще хранят структурированные данные в **JSONB**.

#### Картинка в голове

**inet** — один номер дома (192.168.1.5). **cidr** — целая улица или квартал (192.168.1.0/24 — все дома от .0 до .255 на этой «улице»). Вопрос «ip << подсеть» — «этот дом на этой улице?» Да — true, нет — false. Для логов и проверки доступа по IP это очень удобно.

#### Часто путают

**inet и cidr.** **inet** — один адрес (или адрес с маской, но семантика «один хост»). **cidr** — именно блок (подсеть): задаётся базовый адрес и длина маски (/24, /16 и т.д.). Для «IP клиента» используй **inet**; для «наша офисная подсеть» — **cidr**.

**<< и >>.** **inet << cidr** — «адрес входит в подсеть». **cidr >> inet** — то же самое, но записано наоборот: «подсеть содержит адрес». Результат один и тот же, просто левый и правый операнды поменяны местами.

#### Как запомнить

**inet** = один IP, **cidr** = подсеть (блок). **<<** = «входит в подсеть». Для логов и доступа по IP — типы inet/cidr и оператор **<<**. **xml** — по необходимости; чаще используют **JSONB**.

#### Проверь себя

1. Как отобрать строки логов, где IP клиента из подсети 192.168.0.0/16?

2. Чем inet отличается от cidr?  
<details><summary>Ответ</summary> **inet** — один IP-адрес (один хост). **cidr** — блок адресов (подсеть): базовый адрес + длина маски (/24, /16 и т.д.). Для «IP пользователя» — inet; для «наша сеть» или «диапазон офисных IP» — cidr.</details>

#### Если прочитал и не понял

**inet** — столбец для одного IP-адреса (например, IP пользователя). **cidr** — блок адресов (подсеть), например 10.0.0.0/8. Условие **ip << '10.0.0.0/8'::cidr** — «выбрать строки, где IP из этой подсети». **xml** в реальных проектах чаще заменяют на **JSONB**.

**Повторение одной фразой.** **inet** = один IP; **cidr** = подсеть (блок); **<<** = «адрес входит в подсеть»; для логов и доступа по IP — inet/cidr и оператор **<<**.

#### Запомните

- **xml** — тип для XML; **xpath()** для извлечения; в приложениях чаще используют JSON/JSONB.
- **inet** — IP-адрес; **cidr** — подсеть; операторы **<<**, **>>**, **&&** для проверки вхождения и пересечения.
