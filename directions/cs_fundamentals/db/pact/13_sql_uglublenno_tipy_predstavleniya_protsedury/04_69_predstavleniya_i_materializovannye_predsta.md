## §69. Представления и материализованные представления

### 69.1. Обычные представления

**Цель раздела.**  
Понять, что такое **VIEW** — виртуальная таблица, заданная запросом — и как сделать представление **обновляемым** с помощью **WITH CHECK OPTION** и ограничений (одна таблица, без агрегатов и т.д.). После раздела ты сможешь скрывать сложность запросов и ограничивать доступ к данным через представления.

**В этом разделе главное (три строки).** 1) **VIEW** — именованный запрос; при обращении к нему выполняется этот запрос, данные не хранятся отдельно. 2) **WITH CHECK OPTION** — при INSERT/UPDATE через представление проверяется, что новая строка «видна» в представлении (удовлетворяет условию WHERE); иначе ошибка. 3) **Обновляемые представления** — только простые (одна базовая таблица, без DISTINCT/GROUP BY/агрегатов и т.д.); для сложных — **INSTEAD OF** триггеры.

#### Термины (расшифровка)

- **VIEW (представление)** — объект схемы: имя + определение запроса (SELECT). При SELECT из представления выполняется этот запрос. Данные хранятся только в базовых таблицах; представление — «окно» в них.
- **WITH CHECK OPTION** — опция создания представления. При INSERT или UPDATE через представление СУБД проверяет, что новая/изменённая строка удовлетворяет условию представления (WHERE и т.д.). Если нет — ошибка. **LOCAL** — проверка только для этого представления; **CASCADED** (по умолчанию) — и для нижележащих представлений.
- **Обновляемое представление** — представление, через которое разрешены INSERT/UPDATE/DELETE. В стандарте SQL и в PostgreSQL это возможно только для «простых» представлений: одна базовая таблица в FROM, без DISTINCT, GROUP BY, агрегатов, оконных функций, UNION и т.д. Для сложных представлений обновление реализуют через **INSTEAD OF** триггеры.

#### Теория и правила (подробно)

- **Зачем представления.** Скрыть сложный запрос (JOIN, подзапросы) под простым именем; ограничить набор столбцов и строк (безопасность); единая точка изменения логики (меняем определение представления — все запросы к нему получают новую логику).
- **WITH CHECK OPTION** предотвращает «исчезновение» строки из представления после вставки/обновления (например, вставили с status='draft', а представление показывает только status='published' — без CHECK OPTION строка вставится в таблицу, но не будет видна в представлении; с CHECK OPTION такая вставка может быть запрещена или скорректирована в зависимости от реализации).

#### Пошагово: когда создать VIEW и зачем WITH CHECK OPTION

1. **Скрыть сложный запрос:** вместо того чтобы везде писать JOIN и подзапросы, создаёшь VIEW с именем (например, active_users) и везде обращаешься к нему. Меняешь логику в одном месте — все запросы к представлению получают новую логику.
2. **Ограничить доступ:** представление может показывать только часть столбцов и только строки по WHERE (например, только не удалённых пользователей). Пользователю даёшь права на представление, а не на таблицу — он не видит лишнего.
3. **WITH CHECK OPTION:** если представление обновляемое (INSERT/UPDATE через него), эта опция заставляет проверять: новая или изменённая строка должна «видеться» в представлении (удовлетворять WHERE). Иначе вставишь строку с status='draft', а представление показывает только status='published' — строка в таблице есть, но в представлении её не видно; с CHECK OPTION такая вставка может быть запрещена (зависит от СУБД).

#### Примеры (SQL) с пояснениями

```sql
-- Простое представление: «активные пользователи» — только не удалённые
CREATE VIEW active_users AS
SELECT id, email, name FROM users WHERE deleted_at IS NULL;
-- При SELECT * FROM active_users выполняется этот запрос; данные хранятся в users

-- С WITH CHECK OPTION: вставки/обновления через представление проверяются
CREATE VIEW published_posts AS
SELECT * FROM posts WHERE status = 'published'
WITH CHECK OPTION;
-- При INSERT через published_posts новая строка должна удовлетворять status = 'published'

-- Чтение из представления — как из таблицы
SELECT * FROM active_users;
```

#### Что будет, если вставить через представление без WITH CHECK OPTION

Представление показывает только строки с status='published'. Если в нём нет WITH CHECK OPTION, ты можешь выполнить INSERT через представление и указать status='draft'. Строка вставится в базовую таблицу posts, но в самом представлении published_posts её не будет видно (она не удовлетворяет WHERE). Получается «исчезновение» строки из представления после вставки — путаница. **WITH CHECK OPTION** заставляет СУБД проверять: новая строка должна удовлетворять условию представления; иначе вставка/обновление может быть отклонено.

#### Часто путают

**VIEW и таблица.** VIEW — это **не копия данных**. При каждом SELECT из представления выполняется запрос; данные лежат только в базовых таблицах. Таблица — реальное хранилище; представление — «окно» в него.

**Обновляемое представление и сложное.** Обновлять (INSERT/UPDATE/DELETE) через представление можно только если оно «простое»: одна базовая таблица в FROM, без DISTINCT, GROUP BY, агрегатов, UNION и т.д. Если представление строится из нескольких таблиц или с агрегатами — обновление через него «из коробки» недоступно; нужны **INSTEAD OF** триггеры, которые сами пишут в базовые таблицы.

#### Как запомнить

**VIEW** = именованный запрос; данные в таблицах, не копируются. **WITH CHECK OPTION** = при вставке/обновлении через представление новая строка должна «видеться» в нём. Сложные представления обновляют через **INSTEAD OF** триггеры.

#### Картинка в голове

Представление — это «слой» поверх таблицы: ты смотришь на таблицу через фильтр и через выбранные столбцы. Данные лежат в таблице; представление каждый раз «пересчитывает» вид при запросе. WITH CHECK OPTION — правило: «что бы ты ни вставил или ни обновил через это окно, результат должен быть виден в том же окне» (нельзя вставить то, что не пройдёт фильтр представления).

#### Если прочитал и не понял

**VIEW** — это сохранённый запрос с именем. Когда ты делаешь SELECT из представления, выполняется этот запрос — данные не хранятся отдельно, они в базовых таблицах. **WITH CHECK OPTION** — при INSERT/UPDATE через представление проверяется, что новая строка подходит под условие представления (иначе вставишь то, чего в представлении не видно). Обновлять через представление можно только «простые» (одна таблица, без агрегатов); для сложных пишут **INSTEAD OF** триггеры.

**Повторение одной фразой.** VIEW = именованный запрос; данные в таблицах, не копируются. WITH CHECK OPTION = при вставке/обновлении через представление новая строка должна «видеться» в нём. Сложные представления обновляют через **INSTEAD OF** триггеры.

**Ещё раз в трёх предложениях.** 1) VIEW — это сохранённый запрос с именем; при SELECT из него выполняется этот запрос, данные лежат в базовых таблицах. 2) WITH CHECK OPTION при INSERT/UPDATE через представление проверяет: новая строка должна удовлетворять условию представления (иначе вставишь то, чего в представлении не видно). 3) Обновлять через представление «из коробки» можно только простые (одна таблица); для сложных пишут INSTEAD OF триггеры.

#### Запомните

- **VIEW** — именованный запрос; данные в базовых таблицах; **WITH CHECK OPTION** — проверка при INSERT/UPDATE через представление.
- Обновляемые представления — только простые (одна таблица, без агрегатов и т.д.); сложные — через **INSTEAD OF** триггеры.

### 69.2. Материализованные представления

**Цель раздела.**  
Использовать **MATERIALIZED VIEW** — представление, результат которого **хранится на диске** — и обновлять его через **REFRESH** и **REFRESH CONCURRENTLY** (без блокировки чтения). После раздела ты сможешь кэшировать тяжёлые агрегаты и отчёты.

**В этом разделе главное (три строки).** 1) **MATERIALIZED VIEW** — результат запроса сохраняется в таблице; чтение — из этой таблицы, без пересчёта. 2) **REFRESH MATERIALIZED VIEW** — пересчитать данные (по умолчанию с блокировкой чтения). 3) **REFRESH MATERIALIZED VIEW ... CONCURRENTLY** — пересчёт без эксклюзивной блокировки (читатели видят старые данные до завершения); требует **UNIQUE INDEX** на материализованном представлении.

#### Термины (расшифровка)

- **MATERIALIZED VIEW** — объект: запрос + хранимая копия результата (как таблица). SELECT из материализованного представления читает сохранённые данные; они не обновляются автоматически при изменении базовых таблиц.
- **REFRESH MATERIALIZED VIEW name** — пересчитать результат запроса и записать в хранимую копию. По умолчанию блокирует чтение (эксклюзивная блокировка) на время пересчёта.
- **REFRESH MATERIALIZED VIEW name CONCURRENTLY** — пересчёт без блокировки чтения: создаётся новая копия, по завершении подменяется. Требуется **уникальный индекс** на материализованном представлении (чтобы можно было сравнивать старую и новую версии по ключу). В момент REFRESH читатели продолжают видеть старые данные.
- **WITH NO DATA** — при создании материализованного представления не заполнять его сразу; первый запрос будет пустым до первого REFRESH.

#### Примеры (SQL)

```sql
-- Создание материализованного представления
CREATE MATERIALIZED VIEW mv_orders_by_month AS
SELECT date_trunc('month', created_at) AS month, count(*), sum(total)
FROM orders GROUP BY 1;

-- Уникальный индекс для REFRESH CONCURRENTLY
CREATE UNIQUE INDEX ON mv_orders_by_month (month);

-- Обычное обновление (блокировка чтения)
REFRESH MATERIALIZED VIEW mv_orders_by_month;

-- Обновление без блокировки чтения
REFRESH MATERIALIZED VIEW CONCURRENTLY mv_orders_by_month;

-- Создание без данных (первый SELECT пустой до первого REFRESH)
CREATE MATERIALIZED VIEW mv_large AS SELECT ... FROM ... WITH NO DATA;
REFRESH MATERIALIZED VIEW mv_large;
```

#### Пошагово: когда использовать материализованное представление и как обновлять

1. **Когда нужен кэш тяжёлого запроса.** Запрос с агрегатами и JOIN выполняется долго; результат нужен часто. Создаёшь MATERIALIZED VIEW — результат запроса сохраняется в таблице; SELECT из представления читает эту таблицу, без пересчёта.
2. **Обычное обновление:** **REFRESH MATERIALIZED VIEW name** — пересчитать результат и записать в хранимую копию. На время REFRESH чтение **блокируется** (эксклюзивная блокировка).
3. **Без блокировки чтения:** **REFRESH MATERIALIZED VIEW name CONCURRENTLY** — пересчёт идёт «рядом»; читатели продолжают видеть старую версию до завершения. Для этого нужен **уникальный индекс** на материализованном представлении (чтобы можно было сопоставить старую и новую версии при подмене).
4. **WITH NO DATA** — при создании не заполнять представление; первый SELECT будет пустым до первого REFRESH. Удобно, если первый пересчёт очень тяжёлый и хочешь сделать его отдельно (например, ночью).

#### Что будет, если сделать REFRESH без CONCURRENTLY на большой таблице

**REFRESH MATERIALIZED VIEW** без CONCURRENTLY берёт **эксклюзивную блокировку** на материализованное представление на всё время пересчёта. Читатели не смогут выполнить SELECT из этого представления — будут ждать. Если пересчёт длится минуты, приложение «зависнет» на запросах к этому представлению. **REFRESH ... CONCURRENTLY** пересчитывает данные без такой блокировки — читатели продолжают видеть старые данные до завершения; после подмены увидят новые. Требуется **UNIQUE INDEX** на материализованном представлении.

#### Часто путают

**VIEW и MATERIALIZED VIEW.** **VIEW** — при каждом SELECT выполняется запрос; данные не хранятся отдельно. **MATERIALIZED VIEW** — результат запроса **сохранён** в таблице; SELECT читает эту таблицу; данные обновляются только при **REFRESH**. Для «тяжёлых» отчётов и агрегатов используют материализованное представление; для «живого» вида на таблицы — обычное VIEW.

**REFRESH и REFRESH CONCURRENTLY.** **REFRESH** — пересчёт с блокировкой чтения (быстрее, но читатели ждут). **REFRESH CONCURRENTLY** — пересчёт без блокировки чтения (читатели видят старую версию); требуется **UNIQUE INDEX** на материализованном представлении.

#### Как запомнить

**MATERIALIZED VIEW** = «кэш результата запроса на диске». Обновление — **REFRESH** (блокирует чтение) или **REFRESH CONCURRENTLY** (не блокирует; нужен UNIQUE INDEX). **WITH NO DATA** — создать пустым, заполнить позже REFRESH.

#### Если прочитал и не понял

**Материализованное представление** — это результат запроса, сохранённый в таблице. Чтение из него — чтение из этой таблицы, без пересчёта. Чтобы обновить данные — выполняешь **REFRESH**. **REFRESH CONCURRENTLY** обновляет без блокировки чтения (читатели видят старые данные до конца); для этого на представлении должен быть **уникальный индекс**.

**Повторение одной фразой.** MATERIALIZED VIEW = кэш результата запроса на диске. Обновление — **REFRESH** (блокирует чтение) или **REFRESH CONCURRENTLY** (не блокирует; нужен UNIQUE INDEX).

**Ещё раз в трёх предложениях.** 1) Материализованное представление хранит результат запроса на диске; SELECT читает эту «таблицу», а не пересчитывает запрос. 2) Чтобы обновить данные — выполняешь REFRESH; без CONCURRENTLY чтение блокируется на время пересчёта. 3) REFRESH CONCURRENTLY обновляет без блокировки чтения (читатели видят старую версию до конца); для этого нужен уникальный индекс на материализованном представлении.

#### Запомните

- **MATERIALIZED VIEW** — результат запроса хранится на диске; обновление — **REFRESH** (или **REFRESH CONCURRENTLY** при наличии UNIQUE INDEX).
- **REFRESH CONCURRENTLY** — без блокировки чтения; требуется уникальный индекс на материализованном представлении.

### 69.3. Правила и триггеры

**Цель раздела.**  
Понять разницу между **правилами (rules)** и **триггерами**: правила **переписывают запрос** (подставляют вместо обращения к представлению/таблице другой запрос), триггеры **выполняют код** при событии. После раздела ты будешь выбирать подходящий механизм.

**В этом разделе главное (три строки).** 1) **Правило (rule)** — при обращении к таблице/представлению (SELECT/INSERT/UPDATE/DELETE) запрос **переписывается**: вместо одной таблицы подставляется заданный запрос или дополнительные действия. 2) **Триггер** — при событии (INSERT/UPDATE/DELETE) вызывается **функция**; логика в коде (PL/pgSQL и т.д.). 3) Для обновления сложных представлений чаще используют **INSTEAD OF** триггеры (явная логика), а не правила (сложнее отлаживать и предсказывать).

#### Термины (расшифровка)

- **Правило (rule)** — объект в PostgreSQL: при обращении к таблице или представлению (SELECT, INSERT, UPDATE, DELETE) **исходный запрос подменяется** другим запросом или к нему добавляются действия. СУБД «переписывает» запрос до выполнения. Правила сложнее отлаживать: поведение зависит от того, как планировщик переписал запрос.
- **Триггер** — при событии (INSERT/UPDATE/DELETE) на таблице или представлении вызывается **твоя функция** (PL/pgSQL и т.д.). Логика полностью в коде: ты явно пишешь, что делать (записать в аудит, вставить в другую таблицу и т.д.). Предсказуемо и удобно для отладки.
- **INSTEAD OF триггер** — только для представлений; операция (INSERT/UPDATE/DELETE) **не выполняется** над представлением; вместо неё выполняется код триггера. Так делают обновляемые сложные представления (из нескольких таблиц, с JOIN): в триггере сам пишешь, какие INSERT/UPDATE/DELETE выполнить в базовых таблицах.

#### Пошагово: когда выбрать правило, а когда триггер

1. **Нужно «подменить» запрос** (например, при SELECT из представления подставить другой SELECT с дополнительными условиями) — в PostgreSQL для этого исторически использовали **правила**. Минус: правила трудно читать и отлаживать; порядок применения может быть неочевидным.
2. **Нужно выполнить код при INSERT/UPDATE/DELETE** (аудит, валидация, запись в другую таблицу) — используй **триггер**. Код в одном месте (функция), поведение предсказуемо.
3. **Нужно сделать обновляемым представление из нескольких таблиц** (например, JOIN users и orders) — используй **INSTEAD OF** триггер на представлении. В функции сам решаешь, в какие базовые таблицы писать и какие значения подставлять. Правила для такого тоже можно использовать, но триггеры проще понимать и менять.

#### Простыми словами

**Правило** — это «подмена запроса»: пользователь написал SELECT из таблицы X, а СУБД подставила другой запрос (например, SELECT из представления или с дополнительным WHERE). Ты не пишешь код — только говоришь «вместо этого запроса выполни тот». **Триггер** — это «при каждой вставке/обновлении/удалении вызови мою функцию»; вся логика в твоём коде. Для обновления сложных представлений обычно берут **INSTEAD OF триггер**: в коде явно прописываешь, что писать в базовые таблицы — так проще, чем настраивать правила.

#### Картинка в голове

**Правило** — как «переводчик у двери»: пользователь просит «дай данные из X», а переводчик подменяет запрос: «на самом деле выполни другой запрос». Что именно выполнится — задаётся правилом, а не твоим кодом. **Триггер** — как «охранник у двери»: после того как что-то записали или удалили, вызывается твоя функция (аудит, проверка, запись в другую таблицу). Для сложных представлений «охранник» (INSTEAD OF триггер) обычно удобнее: ты сам решаешь, что делать, в коде.

#### Как запомнить

**Правило** — переписывание запроса (подмена до выполнения). **Триггер** — выполнение кода при событии. Для обновления сложных представлений — **INSTEAD OF** триггеры, не правила.

Правило — «подмена запроса»; триггер — «выполнить мой код».</details>

2. Почему для обновляемого представления из нескольких таблиц чаще используют INSTEAD OF триггер, а не правило?  
<details><summary>Ответ</summary> В **INSTEAD OF** триггере ты явно пишешь в коде, какие INSERT/UPDATE/DELETE выполнить в базовых таблицах и с какими значениями — логику легко читать и менять. **Правила** переписывают запрос; поведение сложнее предсказать и отладить. Для сложных представлений триггер даёт явный контроль.</details>

#### Если прочитал и не понял

**Правила** — механизм «подменить запрос»: когда кто-то обращается к таблице или представлению, СУБД подставляет другой запрос (или добавляет действия). **Триггеры** — при INSERT/UPDATE/DELETE вызывается твоя функция. Для обновления представлений из нескольких таблиц обычно пишут **INSTEAD OF** триггеры, а не правила — так проще отлаживать и понимать логику.

#### Запомните

- **Правила** переписывают запрос; **триггеры** выполняют код при событии.
- Для обновления сложных представлений обычно используют **INSTEAD OF** триггеры.
