## §66. Типы данных углублённо

### 66.1. Точность числовых и DECIMAL

**Цель раздела.**  
Понять, почему для денег и точных расчётов используют **DECIMAL (NUMERIC)**, а не **float/real**, и как задаётся точность и масштаб: **p** (всего цифр) и **s** (цифр после запятой). После раздела ты будешь выбирать тип под задачу и избегать ошибок округления.

**В этом разделе главное (три строки).** 1) **DECIMAL(p,s)** — точный тип: **p** цифр всего, **s** после запятой; арифметика без двоичной погрешности. 2) **float/real** — приближённые типы; сумма 0.1 + 0.2 в float может быть не 0.3 из‑за представления в двоичном виде. 3) Для **денег, процентов, учёта** — всегда **DECIMAL**; для научных расчётов и графиков допустим float, если погрешность приемлема.

#### Термины (расшифровка)

- **DECIMAL(p,s)** (или **NUMERIC(p,s)** в PostgreSQL — синонимы) — тип **точной** десятичной арифметики. **p** — точность (precision): общее число значащих цифр. **s** — масштаб (scale): число цифр **после** десятичной запятой. Например, DECIMAL(10,2): до 10 цифр всего, из них 2 после запятой — под суммы до 99 999 999,99. Ограничения по p и s зависят от СУБД (в PostgreSQL p до 131072, s до 16383).
- **float** (REAL, DOUBLE PRECISION) — тип **приближённой** арифметики с плавающей точкой (IEEE 754). Числа представляются в двоичном виде; многие десятичные дроби (0.1, 0.2) не представимы точно, поэтому при сложении и накоплении возникают ошибки округления.
- **Округление при переполнении** — при записи в DECIMAL значения, превышающего заданную точность, СУБД может округлить (по правилам банковского округления) или вернуть ошибку — зависит от настроек (в PostgreSQL по умолчанию выдаётся ошибка при переполнении).

#### Теория и правила (подробно)

- **Почему float «ломает» деньги.** В двоичной системе нельзя точно представить, например, 0.1 или 0.2. Поэтому 0.1 + 0.2 в float даёт не 0.3, а значение, очень близкое к 0.3, но с «хвостом» (например, 0.30000000000000004). При накоплении сумм, налогах, скидках эти ошибки суммируются — в бухгалтерии и платёжках это недопустимо.
- **DECIMAL хранит число как строку цифр** (внутренне — упакованное десятичное представление), поэтому 0.1 + 0.2 = 0.3 точно. Операции медленнее, чем float, но предсказуемы.
- **Выбор p и s.** Для денег в одной валюте: обычно достаточно 2 знака после запятой — DECIMAL(12,2) или DECIMAL(15,2) для больших сумм. Для процентов с двумя знаками — DECIMAL(5,2). Для коэффициентов с большей точностью — увеличить s (например, DECIMAL(10,6)).

#### Пошагово: как выбрать точность (p) и масштаб (s) для денег

1. **Реши, сколько знаков после запятой тебе нужно.** Для рублей и копеек — 2. Для курсов валют или коэффициентов — 4, 6 и т.д.
2. **Оцени максимальную сумму «до запятой».** Например, заказы до 99 млн рублей — значит до запятой 8 цифр (99 999 999). Плюс 2 после запятой — всего 10 цифр. Берёшь с запасом: **p = 12, s = 2** → DECIMAL(12, 2). Так ты хранишь до 9 999 999 999,99.
3. **Проверь лимиты СУБД.** В PostgreSQL p может быть до 131072, s до 16383 — для денег этого более чем достаточно.
4. **Итог:** для цен и сумм в одной валюте с копейками почти всегда хватает **DECIMAL(12, 2)** или **DECIMAL(15, 2)** для очень больших сумм.

#### Примеры (SQL) с пояснениями к каждой строке

```sql
-- Создание столбца для денег: 12 цифр всего, 2 после запятой (рубли.копейки)
ALTER TABLE orders ADD COLUMN total DECIMAL(12, 2);

-- Ошибка округления с float (НЕ ДЕЛАЙ ТАК для денег!)
-- Компьютер хранит 0.1 и 0.2 в двоичном виде приближённо; сумма даёт "хвост"
SELECT 0.1::float + 0.2::float;  -- например 0.30000000000000004

-- Точный результат с DECIMAL: числа хранятся в десятичном виде
SELECT 0.1::numeric + 0.2::numeric;  -- ровно 0.3

-- Накопление 100 раз по 0.01 в float — набегает погрешность
SELECT sum(x) FROM generate_series(1, 100) AS t(x);  -- 5050 (целые — ок)
SELECT sum(0.01::float) FROM generate_series(1, 100);  -- может быть не ровно 1.0!
SELECT sum(0.01::numeric) FROM generate_series(1, 100);  -- ровно 1.00

-- Округление при вставке: в DECIMAL(12,2) после запятой только 2 знака
INSERT INTO orders (total) VALUES (99.999);  -- в PostgreSQL округлится до 100.00
```

#### Разбор одного примера по шагам (почему 0.1 + 0.2 в float ≠ 0.3)

1. **Что такое 0.1 в десятичной системе.** Ты записываешь 0.1 — это одна десятая. В десятичной системе это записывается точно.
2. **Как компьютер хранит числа в float.** Компьютер хранит числа в **двоичной** системе (только 0 и 1). В двоичной системе дробь «одна десятая» (0.1) **не представляется конечным числом** — получается бесконечная двоичная дробь, как 1/3 = 0,333… в десятичной. Поэтому float хранит **приближение** к 0.1, а не точное 0.1.
3. **То же для 0.2.** 0.2 в двоичном виде тоже приближение. Когда ты складываешь два приближения, маленькая погрешность остаётся — и получается не ровно 0.3, а 0.30000000000000004 (примерно).
4. **Почему DECIMAL даёт 0.3.** DECIMAL хранит число **как набор десятичных цифр** (внутри — упакованное десятичное представление). 0.1 хранится как «одна десятая», 0.2 — как «две десятых», сумма — ровно 0.3. Ошибки округления из двоичного представления здесь нет.
5. **Итог.** float — «приближённый калькулятор» для очень больших/маленьких чисел; для денег и точных сумм он даёт «копейки из ниоткуда». DECIMAL — «точный калькулятор» по десятичным цифрам; для денег нужен именно он.

#### Что будет, если использовать float для накопления сумм

Представь: в таблице 10 000 строк, в каждой сумма 19.99 (float). Ты делаешь `SELECT sum(price) FROM items`. Из‑за того что 19.99 в float хранится приближённо, при сложении 10 000 раз погрешность накапливается. Вместо 199 900.00 может получиться 199 899.98 или 199 900.02 — в отчёте «не сойдётся» с кассой. С DECIMAL(10,2) сумма будет ровно 199 900.00.

#### Типичная ошибка

Использовать **float/real** для цен, сумм заказов, налогов. В отчётах и при сравнении после многих операций появятся «копейки» из ниоткуда или пропадут. **Правильно:** для любых денежных и учётных величин — **DECIMAL(p,s)** с подходящими p и s.

#### Простыми словами

DECIMAL — это «калькуляторная» точность: как ты считаешь в уме или на калькуляторе, так и хранится. float — это «научный калькулятор»: очень большой диапазон чисел, но с микроошибками в десятичных дробях. Для денег микроошибки превращаются в реальные потери или переплаты — поэтому деньги только в DECIMAL.

#### Картинка в голове

Касса в магазине: каждая копейка должна сойтись. DECIMAL — это касса, где каждая сумма записана ровно теми цифрами, которые ты видишь: 19.99, 100.00, без «хвостов». float — это касса, где после сотен операций «набегает» несколько копеек из‑за того, как компьютер хранит дроби в двоичном виде; в конце дня сумма по чекам не совпадает с суммой в ящике — в бухгалтерии так нельзя. Поэтому для денег всегда «калькуляторная» точность (DECIMAL), а не «научный калькулятор» (float).

#### Часто путают

**DECIMAL и NUMERIC.** В PostgreSQL это **одно и то же**: NUMERIC — стандартное имя в SQL, DECIMAL — то же самое. Оба означают «точная десятичная арифметика». Пиши тот вариант, который привычнее команде.

**p и s в DECIMAL(p,s).** **p** — это **все** значащие цифры (и до запятой, и после). **s** — только цифры **после** запятой. Значит цифр **до** запятой максимум **p − s**. Пример: DECIMAL(10, 2) — до 8 цифр до запятой и 2 после → максимум 99 999 999,99.

#### Как запомнить

**Деньги = DECIMAL.** Одна фраза: «Если это деньги, проценты или учёт — тип DECIMAL(p,s), не float.» Число **p** — сколько всего цифр нужно (с запасом), **s** — сколько из них после запятой (для рублей — 2).

#### Проверь себя

1. Почему 0.1 + 0.2 в float не равно 0.3?  
<details><summary>Ответ</summary> В двоичной системе с плавающей точкой десятичные дроби 0.1 и 0.2 не представимы точно; при сложении получается значение, очень близкое к 0.3, но с остаточной погрешностью. DECIMAL хранит числа в десятичном виде, поэтому 0.1 + 0.2 = 0.3 точно.</details>

2. Какой тип выбрать для столбца «цена товара» в рублях с копейками?  
<details><summary>Ответ</summary> **DECIMAL(p, 2)**, например DECIMAL(10, 2) — точная арифметика, два знака после запятой; тип float/real для денег использовать не следует из‑за погрешностей округления.</details>

3. Что такое p и s в DECIMAL(p,s)? Приведи пример для сумм до 1 млн рублей с копейками.  
<details><summary>Ответ</summary> **p** — общее число значащих цифр (до и после запятой), **s** — число цифр **после** запятой. Для сумм до 1 000 000,00 нужно 6 цифр до запятой и 2 после — всего 8 цифр; с запасом: DECIMAL(10, 2) или DECIMAL(12, 2).</details>

#### Запомните

- **DECIMAL(p,s)** — точный тип: p цифр всего, s после запятой; для денег и точных расчётов.
- **float/real** — приближённые типы; возможны ошибки округления в десятичных дробях — не для денег.
- Для денег, процентов, учёта — **всегда DECIMAL** с подходящими p и s.

#### Если прочитал и не понял

Коротко: компьютер хранит дроби в float приближённо (в двоичном виде), поэтому 0.1 + 0.2 даёт не 0.3, а «хвост». Для денег это недопустимо. DECIMAL хранит числа в десятичном виде — как на калькуляторе — поэтому сложение точное. Запомни: **деньги → DECIMAL(p,2)** (или другой s по задаче), **не float**.

**Повторение одной фразой.** Деньги и точные расчёты — только DECIMAL(p,s); float даёт погрешности в десятичных дробях, для денег это ошибка.

**Ещё раз в трёх предложениях.** 1) **DECIMAL(p,s)** — точный тип: p цифр всего, s после запятой; арифметика без двоичных погрешностей. 2) **float** хранит дроби приближённо (в двоичном виде), поэтому 0.1 + 0.2 ≠ 0.3 и при накоплении сумм набегают «копейки из ниоткуда». 3) Для денег, процентов и учёта всегда используй **DECIMAL(p,s)** с подходящими p и s; для цен в рублях с копейками — например DECIMAL(12, 2).

### 66.2. Коллации и кодировки

**Цель раздела.**  
Понять, что такое **коллация (COLLATE)** — правило сортировки и сравнения строк — и как связаны **кодировка** (character set) и **длина строки в байтах vs в символах**. После раздела ты сможешь настраивать сортировку (в т.ч. без учёта регистра) и корректно работать с UTF-8.

**В этом разделе главное (три строки).** 1) **Коллация** задаёт порядок сортировки (ORDER BY) и сравнения (WHERE, LIKE): например, «A» и «a» считаются равными или разными в зависимости от коллации. 2) **Кодировка** (в PostgreSQL — по сути UTF-8 для БД) определяет, как символы кодируются в байты; один символ может занимать несколько байт. 3) **char_length()** — число символов, **octet_length()** — число байт; для лимитов по длине важно понимать разницу (например, ограничение в 255 «символов» в UTF-8 может быть до 255 байт в старых системах).

#### Термины (расшифровка)

- **Коллация (collation)** — набор правил: как сравнивать и сортировать строки. Включает: порядок букв (алфавит), учёт/игнорирование регистра (case-sensitive vs case-insensitive), учёт диакритики (акценты), возможно несколько символов за один (например, в некоторых языках). В PostgreSQL коллация задаётся при создании БД (lc_collate) или для выражения/столбца через **COLLATE "имя"**.
- **lc_collate** — параметр кластера/БД: коллация по умолчанию для сортировки (ORDER BY). **lc_ctype** — классификация символов (что буква, что цифра, верхний/нижний регистр) — влияет на LIKE, SIMILAR TO, верхний/нижний регистр. Устанавливаются при создании кластера/БД, для БД можно задать при CREATE DATABASE ... LC_COLLATE ... LC_CTYPE.
- **Кодировка (encoding)** — соответствие символов и последовательностей байт. В PostgreSQL кодировка БД задаётся при создании (например, UTF8). **client_encoding** — кодировка, в которой клиент отправляет/получает текст (обычно совпадает с кодировкой БД для UTF-8).
- **char_length(str)** (или length(str)) — число **символов** в строке. **octet_length(str)** — число **байт**. В UTF-8 один символ может занимать 1–4 байта; для кириллицы часто 2 байта на символ.

#### Теория и правила (подробно)

- **Зачем менять коллацию.** Чтобы сортировать «по-человечески» для данного языка (например, русский алфавит: а, б, в…) или сделать сравнение без учёта регистра (например, 'Email' = 'email') без вездесущего LOWER() в запросах.
- **COLLATE в запросе.** В SQL можно указать коллацию для выражения: `SELECT * FROM t ORDER BY name COLLATE "ru_RU.utf8";` или для столбца в определении таблицы: `name TEXT COLLATE "ru_RU.utf8"`. Список доступных: `SELECT * FROM pg_collation;`.
- **Case-insensitive сравнение.** В коллациях вида `..._ci` (case insensitive) символы разного регистра при сравнении равны. В PostgreSQL можно использовать расширение **citext** (case-insensitive text) или коллацию, если доступна (зависит от ОС и установленных локалей).

#### Пошагово: когда и как менять коллацию

1. **Проверь, как сейчас сортируется.** Выполни `SELECT * FROM t ORDER BY name LIMIT 5;` — порядок зависит от **lc_collate** базы (задаётся при создании БД).
2. **Если нужна сортировка для другого языка** (например, русский алфавит: а, б, в…), укажи коллацию в запросе: `ORDER BY name COLLATE "ru_RU.utf8"` (имя локали зависит от ОС и установленных пакетов).
3. **Список доступных коллаций:** `SELECT collname FROM pg_collation WHERE collname NOT LIKE 'pg_%' ORDER BY collname;` — выбери нужную локаль.
4. **Для столбца навсегда:** при создании таблицы: `name TEXT COLLATE "ru_RU.utf8"`. Тогда все ORDER BY по этому столбцу по умолчанию используют эту коллацию.

#### Примеры (SQL) с пояснениями

```sql
-- Сортировка по умолчанию (правило берётся из lc_collate БД)
SELECT * FROM users ORDER BY name;

-- Явная коллация для этого запроса (если локаль установлена в системе)
SELECT * FROM users ORDER BY name COLLATE "en_US.utf8";

-- Длина в символах и в байтах (UTF-8): "привет" — 6 букв, но в байтах больше
SELECT char_length('привет'), octet_length('привет');
-- Результат: 6 и 12 (каждая кириллическая буква в UTF-8 — 2 байта)

-- Один символ эмодзи может быть 4 байта
SELECT char_length('👍'), octet_length('👍');  -- 1 символ, 4 байта

-- Проверка кодировки БД и клиента (должны совпадать, обычно UTF8)
SHOW server_encoding;  -- UTF8
SHOW client_encoding;  -- UTF8
```

#### Разбор одного примера по шагам: char_length и octet_length для «привет»

Разберём, почему для строки **«привет»** char_length даёт 6, а octet_length — 12.

1. **Строка «привет»** состоит из шести букв: п, р, и, в, е, т. Это **символы** (графемы), то, что видит пользователь.
2. **char_length('привет')** считает **количество символов** — получается **6**. Это то, что нужно для лимита «не более 100 символов»: пользователь ввёл 6 букв.
3. **В кодировке UTF-8** каждая кириллическая буква кодируется **двумя байтами**. Поэтому «п» — 2 байта, «р» — 2 байта, и так все шесть букв: 6 × 2 = **12 байт**.
4. **octet_length('привет')** считает **количество байт** в памяти — получается **12**. Это то, что занимает строка на диске и в памяти.
5. **Итог:** если в задаче сказано «поле не более 100 символов», проверяй **char_length(name) <= 100**, а не «100 байт» — иначе при 100 кириллических буквах получится 200 байт, и лимит «100 байт» отрежет половину «символов».

**Ещё пример:** эмодзи 👍 — это **1 символ**, но в UTF-8 он занимает **4 байта**. char_length('👍') = 1, octet_length('👍') = 4. Поэтому «длина в символах» и «длина в байтах» — разные вещи; для лимитов по смыслу («N символов») используй **char_length**.

#### Что будет, если ограничить поле «255 символов» по байтам

Если в старом коде или в другой СУБД столбец объявлен как VARCHAR(255) и имеется в виду **255 байт**, то при UTF-8 туда поместится не 255 символов, а меньше: кириллица — 2 байта на букву → максимум 127 символов; эмодзи — 4 байта → ещё меньше. Пользователь введёт длинное имя — получит ошибку «превышена длина». **Правильно:** либо ограничивать по символам (проверка char_length в CHECK или тип с семантикой символов), либо закладывать запас (например, 1000 байт для «до 255 символов» в UTF-8).

#### Типичная ошибка

Ограничивать длину поля по байтам (например, 255 байт), думая, что это «255 символов». В UTF-8 255 символов могут занять до 1020 байт (4 байта на символ). **Правильно:** либо лимит в символах (проверять char_length или использовать тип с семантикой «символы»), либо закладывать запас по байтам.

#### Простыми словами

Коллация — это «правила алфавита» для сортировки и сравнения: какой символ за каким идёт (например, «а» перед «б») и считаем ли мы «А» и «а» одним и тем же при сравнении. Кодировка — как буквы превращаются в байты на диске; в UTF-8 одна буква может быть одним байтом (латиница), двумя (кириллица) или четырьмя (часть эмодзи), поэтому «длина в символах» и «длина в байтах» — разные вещи. Если в задаче сказано «не более 100 символов», проверяй **char_length**, а не количество байт.

#### Картинка в голове

Коллация — это как порядок букв в алфавите и правило «считаем ли мы большие и маленькие буквы одинаково». В одном алфавите «А» = «а», в другом — нет; в одном «ё» идёт после «е», в другом — отдельно. Кодировка — как буквы записаны в тетради: латинская «A» — одна клетка, русская «А» — одна клетка, но в байтах они занимают разное место (1 и 2 в UTF-8). Поэтому «255 клеток» (символов) и «255 байт» — не одно и то же.

#### Часто путают

**Коллация и кодировка.** Коллация — **как сравнивать и сортировать** строки (порядок, регистр). Кодировка — **как хранить** символы в байтах (UTF-8, WIN1251 и т.д.). Одна БД имеет одну кодировку (например, UTF8); коллаций может быть несколько (для разных языков).

**char_length и octet_length.** **char_length** (или length) — число **символов** (букв, цифр). **octet_length** — число **байт** в памяти. В UTF-8 для кириллицы char_length × 2 ≈ octet_length; для латиницы часто char_length = octet_length.

#### Как запомнить

**Коллация = порядок и сравнение, кодировка = байты.** Для лимита «N символов» смотри **char_length**; для размера на диске — **octet_length**. В UTF-8 один символ — от 1 до 4 байт.

#### Запомните

- **Коллация (COLLATE)** задаёт порядок сортировки и сравнение строк; lc_collate/lc_ctype задаются для БД.
- **Кодировка** — соответствие символов и байт; **char_length** — символы, **octet_length** — байты; в UTF-8 один символ может быть несколькими байтами.

#### Если прочитал и не понял

Коллация отвечает на вопрос: «В каком порядке сортировать и считать ли "А" и "а" одинаковыми?» Кодировка отвечает на вопрос: «Сколько байт занимает одна буква?» В UTF-8 буквы занимают 1–4 байта, поэтому «длина в символах» и «длина в байтах» не совпадают — для лимитов используй **char_length**.

**Повторение одной фразой.** Коллация = как сравнивать и сортировать строки; кодировка = как хранить символы в байтах. Для «N символов» — **char_length**, для размера на диске — **octet_length**.

#### Проверь себя

1. В чём разница между коллацией и кодировкой?  
<details><summary>Ответ</summary> **Коллация** задаёт **порядок сортировки** и **правила сравнения** строк (например, считать ли «А» и «а» одинаковыми, порядок букв по алфавиту). **Кодировка** задаёт, **как символы хранятся в байтах** (UTF-8, WIN1251 и т.д.). Одна — «как сравнивать», другая — «как хранить».</details>

2. Почему для лимита «не более 100 символов» нужно смотреть char_length, а не octet_length?  
<details><summary>Ответ</summary> **char_length** считает **символы** (буквы, цифры) — то, что видит пользователь. **octet_length** считает **байты** в памяти. В UTF-8 одна буква может занимать 1–4 байта (кириллица — чаще 2). Поэтому «100 байт» — это не «100 символов»; лимит по смыслу — «100 символов», значит проверяй **char_length**.</details>

### 66.3. Дата и время, таймзоны

**Цель раздела.**  
Научиться корректно хранить и обрабатывать **дату и время с учётом таймзон**: типы **TIMESTAMP WITH TIME ZONE** и **WITHOUT**, преобразование **AT TIME ZONE**, тип **INTERVAL**, функции **extract()** и **date_trunc()**. После раздела ты не будешь путать «локальное» время с UTC и сможешь считать интервалы и обрезать даты до часа/дня.

**В этом разделе главное (три строки).** 1) **TIMESTAMP WITH TIME ZONE** (timestamptz) хранит момент времени в **UTC**; при вводе и выводе используется текущая таймзона сессии или явная зона. 2) **INTERVAL** — длительность (дни, часы, минуты); с ним можно складывать/вычитать с датами. 3) **extract()** извлекает поле (год, месяц, день, час, минуту, день недели и т.д.), **date_trunc()** обрезает дату до заданной границы (секунда, минута, час, день, неделя, месяц, год).

#### Термины (расшифровка)

- **TIMESTAMP WITH TIME ZONE** (в PostgreSQL — **timestamptz**) — тип «момент времени в мире». Внутри хранится значение в **UTC** (универсальное координированное время). При вставке строка с таймзоной (или без неё — тогда подставляется таймзона сессии) приводится к UTC. При выборке значение может показываться в таймзоне сессии (или в любой другой при явном преобразовании). **Рекомендуется** для любых «моментов» (события, логи, заказы).
- **TIMESTAMP WITHOUT TIME ZONE** (timestamp) — хранит только «календарную дату и время» **без привязки к зоне**. Подходит для «местного» времени без перевода (например, «время открытия магазина 09:00» без указания, в какой зоне). Для событий в прошлом/будущем в разных зонах лучше timestamptz.
- **AT TIME ZONE** — преобразование: «интерпретировать это значение как время в зоне X» или «показать в зоне X». Например, `timestamp_column AT TIME ZONE 'Europe/Moscow'` даёт timestamptz (если исходный тип был timestamp) или timestamp (если исходный был timestamptz — «отбрасывает» зону и даёт локальное время в указанной зоне). Синтаксис: `value AT TIME ZONE 'Europe/Moscow'` или `'UTC'`, `'+03'` и т.д.
- **INTERVAL** — тип «длительность»: например, 1 day, 2 hours 30 minutes, -1 week. Используется в арифметике с датами: `now() + interval '7 days'`, `event_at - interval '1 hour'`. Можно извлекать поля через extract(epoch from interval) и т.п.
- **extract(field FROM date_or_interval)** — извлечь поле: year, month, day, hour, minute, second, dow (день недели 0–6), week, quarter и др. Возвращает число (double precision для second). Пример: `extract(year FROM created_at)`.
- **date_trunc('precision', timestamp)** — обрезать дату до заданной границы: 'second', 'minute', 'hour', 'day', 'week', 'month', 'quarter', 'year'. Например, `date_trunc('day', now())` даёт полночь текущего дня в той же таймзоне.

#### Теория и правила (подробно)

- **Почему хранить в UTC.** Один и тот же момент времени в разных точках Земли отображается разным «локальным» временем. Если хранить «как пришло» без зоны, при смене летнего времени или при показе пользователю в другой зоне возникнет путаница. Хранение в UTC даёт однозначный момент; вывод в нужную зону делается при отображении (AT TIME ZONE или настройка timezone в сессии).
- **Имена таймзон.** Предпочтительно использовать имена вида **Europe/Moscow**, **America/New_York** (база IANA), а не только смещения типа +03:00 — они учитывают историю изменений (летнее время и т.д.).
- **Таймзона сессии.** Параметр **TimeZone** (или переменная **timezone**) задаёт, в какой зоне показывать timestamptz при выводе и как интерпретировать «голый» timestamp при вставке. Установка: `SET timezone = 'Europe/Moscow';` или в connection string.

#### Пошагово: как сохранить «момент события» и показать его пользователю в его зоне

1. **В таблице храни только timestamptz.** Например, `created_at TIMESTAMP WITH TIME ZONE DEFAULT now()`. При вставке без явной зоны подставится таймзона сессии и значение сохранится в UTC.
2. **Таймзону сессии задай при подключении** (в connection string или после подключения): `SET timezone = 'Europe/Moscow';` — тогда при выводе timestamptz будет показываться в Москве.
3. **Для явного перевода в другую зону** в запросе используй AT TIME ZONE: `SELECT created_at AT TIME ZONE 'America/New_York' FROM orders;` — получишь время в Нью-Йорке (тип будет timestamp без зоны — «локальное время в той зоне»).
4. **Для расчётов «через N дней»** используй INTERVAL: `WHERE created_at > now() - interval '7 days';` — записи за последние 7 дней.

#### Примеры (SQL) с пояснениями

```sql
-- Текущий момент: внутри хранится UTC, на экран выводится в таймзоне сессии
SELECT now();  -- тип timestamptz

-- Показать этот же момент «как время в Москве» (результат — timestamp без зоны)
SELECT now() AT TIME ZONE 'Europe/Moscow';

-- Интервал: «через неделю от сейчас»
SELECT now() + interval '7 days';

-- Интервал с явным указанием: 2 дня 3 часа
SELECT now() + interval '2 days 3 hours';

-- Извлечь год и день недели (0=воскресенье, 1=понедельник, … 6=суббота)
SELECT extract(year FROM now()), extract(dow FROM now());

-- Обрезка до начала текущего дня (полночь в таймзоне сессии)
SELECT date_trunc('day', now());

-- Начало текущего месяца (первое число, 00:00:00)
SELECT date_trunc('month', now());

-- Разница между двумя моментами (тип interval)
SELECT now() - created_at FROM orders WHERE id = 1;
```

#### Разбор одного примера по шагам: AT TIME ZONE

Разберём, что происходит, когда ты пишешь **now() AT TIME ZONE 'Europe/Moscow'**.

1. **now()** возвращает текущий момент времени в типе **timestamptz**. Внутри PostgreSQL хранит его в UTC (например, 12:00 UTC).
2. **AT TIME ZONE 'Europe/Moscow'** говорит: «покажи этот момент **как локальное время в Москве**». Москва зимой — UTC+3, значит 12:00 UTC = 15:00 в Москве.
3. Результат выражения — тип **timestamp без зоны** (timestamp): это уже не «момент в мире», а «показание часов в Москве» — 15:00. То есть **тип меняется**: было timestamptz (с зоной), стало timestamp (без зоны).
4. Зачем это нужно: когда ты выводишь пользователю «время заказа в его городе», ты делаешь `created_at AT TIME ZONE 'Europe/Moscow'` и получаешь число «15:00» — то, что показывать на экране. В БД по-прежнему хранится один момент (UTC); зона нужна только при отображении.

**Итог:** AT TIME ZONE переводит «момент в мире» в «показание часов в указанной зоне» (результат — timestamp без зоны) или наоборот — «показание часов в зоне» трактует как момент и даёт timestamptz.

#### Что будет, если хранить «время заказа» в timestamp (без зоны)

Пользователь в Москве сделал заказ в 23:00. Ты сохранил в timestamp без зоны: 23:00. Пользователь в Калининграде открывает тот же заказ — у него тоже 23:00, хотя в Калининграде это уже другой момент (на час раньше по UTC). При смене летнего времени «23:00» в разные дни может означать разный момент в UTC. Путаница при отчётах и при сравнении «кто раньше». **Правильно:** хранить в **timestamptz**; при выводе переводить в зону пользователя (AT TIME ZONE или timezone сессии).

#### Типичная ошибка

Хранить «время события» в **TIMESTAMP WITHOUT TIME ZONE** и подставлять локальное время приложения. При смене летнего времени или при пользователях из разных зонах значения становятся неоднозначными. **Правильно:** хранить моменты в **timestamptz** (UTC), вывод в локальную зону делать при отображении.

#### Простыми словами

Момент времени в мире один (например, «когда ракета стартовала»), а на часах в Москве, Нью-Йорке и Токио он показывается по-разному. timestamptz хранит этот один момент в UTC; «на каких часах показывать» задаётся при выводе (таймзона сессии или AT TIME ZONE). INTERVAL — это «длительность»: 2 часа, 3 дня; её можно прибавлять к моменту или вычитать. extract — «вытащить» из даты год, месяц, день недели и т.д.; date_trunc — «обнулить» дату до границы (полночь дня, первое число месяца, начало года).

#### Картинка в голове

Глобальный момент — один (старт ракеты). В БД хранится именно он (в UTC). Когда пользователь смотрит «время заказа», ты показываешь ему те же часы, но переведённые в его город: Москва +3, Нью-Йорк −5 и т.д. timestamp без зоны — это как записать «23:00» без указания, в каком городе; при переезде или смене летнего времени непонятно, какой это момент. INTERVAL — «плюс неделя», «минус два часа»; date_trunc — «обрезать до понедельника» или «до первого числа месяца».

#### Часто путают

**timestamptz и timestamp.** **timestamptz** хранит **один момент в мире** (внутри — UTC); при выводе показывается в нужной зоне. **timestamp** хранит только «дату и время» без привязки к зоне — непонятно, в каком городе это 23:00. Для событий (заказ, лог, клик) — всегда **timestamptz**.

**AT TIME ZONE: что во что.** `timestamptz_value AT TIME ZONE 'Europe/Moscow'` даёт **timestamp** (локальное время в Москве, «без зоны»). `timestamp_value AT TIME ZONE 'Europe/Moscow'` трактует значение как «время в Москве» и даёт **timestamptz** (тот момент в UTC). Итого: после AT TIME ZONE тип «меняется» — с зоны на без зоны или наоборот.

#### Как запомнить

**События → timestamptz.** Одна фраза: «Любой момент в прошлом или будущем (заказ, лог, событие) храни в timestamptz». **extract** — вытащить поле (год, день недели); **date_trunc** — обрезать до границы (день, месяц). **INTERVAL** — прибавить или вычесть длительность.

#### Запомните

- **timestamptz** хранит момент в **UTC**; вывод и ввод зависят от таймзоны сессии; для событий используйте его.
- **AT TIME ZONE** — показать/интерпретировать время в указанной зоне; **INTERVAL** — длительность; **extract()** — поле из даты/интервала; **date_trunc()** — обрезка до границы (день, месяц, год).

#### Если прочитал и не понял

Коротко: храни **моменты времени** (когда что произошло) в типе **timestamptz** — так в БД лежит один и тот же момент для всей Земли (в UTC). Показывать пользователю можно в его зоне (SET timezone или AT TIME ZONE). **INTERVAL** — это «плюс 7 дней» или «минус 2 часа». **extract** — взять из даты год/месяц/день недели; **date_trunc** — обнулить до начала дня или месяца.

**Повторение одной фразой.** Любой момент события (заказ, лог, клик) храни в **timestamptz**; для «плюс N дней» используй **INTERVAL**; для «год/день недели» — **extract**, для «начало дня/месяца» — **date_trunc**.

#### Проверь себя

1. Почему «время заказа» лучше хранить в timestamptz, а не в timestamp без зоны?  
<details><summary>Ответ</summary> **timestamptz** хранит **один момент в мире** (внутри — UTC). Пользователи в разных городах увидят одно и то же событие в своём локальном времени. **timestamp** без зоны хранит только «дату и время» без привязки к зоне — при смене летнего времени или при пользователях из разных зон непонятно, какой это момент; возможна путаница в отчётах.</details>

2. Что делает date_trunc('month', created_at)?  
<details><summary>Ответ</summary> **date_trunc('month', ...)** обрезает дату/время до **начала месяца**: первое число этого месяца, 00:00:00. Например, для 2024-06-15 14:30:00 получится 2024-06-01 00:00:00. Удобно для группировки «по месяцам» или «за июнь».</details>

### 66.4. ENUM, домены, составные типы

**Цель раздела.**  
Понять пользовательские типы в PostgreSQL: **ENUM** (перечисление значений), **DOMAIN** (тип с ограничением, например CHECK), **составной тип** (набор полей как один тип). После раздела ты сможешь вводить ограниченные множества значений и переиспользуемые домены.

**В этом разделе главное (три строки).** 1) **ENUM** — тип с фиксированным списком значений; сравнение и сортировка по порядку объявления. 2) **DOMAIN** — тип на основе существующего (например, text) с добавлением CHECK; один раз описал — используешь в нескольких столбцах. 3) **Составной тип (composite type)** — тип «запись» из нескольких полей; можно использовать как столбец или в функциях.

#### Термины (расшифровка)

- **ENUM (перечисление)** — пользовательский тип, значения которого — заданный список строк. Создание: `CREATE TYPE status AS ENUM ('draft', 'published', 'archived');`. Столбец: `status status`. Вставка только перечисленных значений; сравнение и ORDER BY по порядку в объявлении. Добавить новое значение: `ALTER TYPE status ADD VALUE 'rejected';` (в PostgreSQL новое значение в конец).
- **Домен (DOMAIN)** — пользовательский тип на основе базового (integer, text, date и т.д.) с необязательными ограничениями (CHECK, DEFAULT). Пример: `CREATE DOMAIN positive_int AS integer CHECK (VALUE > 0);`. Использование: `quantity positive_int` — везде, где нужен «положительный целое», один домен вместо повторения CHECK в каждой таблице.
- **Составной тип (composite type)** — тип «структура» из нескольких полей. Создание: `CREATE TYPE point_2d AS (x double precision, y double precision);`. Использование: столбец типа point_2d или возврат функции. Обращение к полям: `(column_name).x`, `(column_name).y`.

#### Пошагово: когда выбрать ENUM, домен или составной тип

1. **ENUM** — когда у столбца **фиксированный небольшой набор значений** (статус заказа: new, paid, shipped; тип пользователя: admin, user, guest). Плюсы: нельзя вставить «опечатку», ORDER BY по статусу идёт по смыслу (по порядку в ENUM). Минус: добавить новое значение — ALTER TYPE (в PostgreSQL ADD VALUE).
2. **DOMAIN** — когда одно и то же **ограничение** нужно в нескольких столбцах (например, «положительное целое» или «email по формату»). Создаёшь домен один раз — используешь везде; меняешь проверку в одном месте.
3. **Составной тип** — когда логически **несколько полей — одна сущность** (имя и фамилия, координаты x,y, диапазон от-до). Удобно как столбец или как возврат функции; обращение к полям через (column).field.

#### Примеры (SQL) с пояснениями

```sql
-- ENUM: список значений задаётся при создании типа
CREATE TYPE order_status AS ENUM ('new', 'paid', 'shipped', 'done');
CREATE TABLE orders (id serial PRIMARY KEY, status order_status DEFAULT 'new');
INSERT INTO orders (status) VALUES ('paid');  -- только значения из ENUM
-- ORDER BY status сортирует по порядку в ENUM: new < paid < shipped < done

-- Добавить новое значение в ENUM (в конец)
ALTER TYPE order_status ADD VALUE 'cancelled';

-- Домен: тип + ограничение (CHECK); переиспользование
CREATE DOMAIN email_text AS text CHECK (value ~ '^[^@]+@[^@]+\.[^@]+$');
CREATE TABLE users (id serial PRIMARY KEY, email email_text);
INSERT INTO users (email) VALUES ('a@b.com');  -- ок
-- INSERT INTO users (email) VALUES ('not-email');  -- ошибка CHECK

-- Составной тип: несколько полей как один тип
CREATE TYPE full_name AS (first_name text, last_name text);
CREATE TABLE persons (id serial PRIMARY KEY, name full_name);
INSERT INTO persons (name) VALUES (('Ivan', 'Ivanov'));
-- Обращение к полям: скобки обязательны (name).first_name
SELECT (name).first_name, (name).last_name FROM persons;
```

#### Что будет, если вставить в ENUM-столбец значение не из списка

При INSERT или UPDATE с значением, которого нет в ENUM, PostgreSQL выдаст ошибку: тип не совпадает или значение не входит в перечисление. Так ENUM защищает от опечаток и «левых» статусов (например, 'paied' вместо 'paid'). Чтобы добавить новое допустимое значение, нужно выполнить **ALTER TYPE enum_name ADD VALUE 'new_value';** (в PostgreSQL новое значение добавляется в конец порядка сортировки).

#### Часто путают

**ENUM и CHECK по списку.** ENUM — отдельный **тип** с фиксированным списком; сортировка по порядку в ENUM. CHECK (col IN ('a','b','c')) — ограничение на столбце; порядок сортировки по типу столбца (например, текстовый). Для «статусов» с осмысленным порядком ENUM удобнее.

**Составной тип: скобки при обращении к полю.** Пиши **(column).field**, а не column.field — иначе PostgreSQL может трактовать точку иначе. Пример: `(name).first_name`, не `name.first_name`.

#### Простыми словами

**ENUM** — как «выпадающий список»: столбец может быть только new, paid или done; вставить «paied» нельзя — СУБД выдаст ошибку. **Домен** — «тип с правилом»: один раз описал «положительное целое» или «email» — используешь в разных таблицах; правило в одном месте. **Составной тип** — одна «коробка» из нескольких полей (имя и фамилия, x и y); достаёшь поле так: **(name).first_name** — скобки обязательны.

#### Картинка в голове

**ENUM** — светофор: только красный, жёлтый, зелёный; другого цвета не бывает. **Домен** — штамп «проверено»: один раз настроил (например, «только положительные числа») — ставишь на разные столбцы. **Составной тип** — чемодан с отделениями: один столбец — чемодан, поля — отделения; открываешь так: **(чемодан).отделение**.

#### Как запомнить

**ENUM** — «статус из списка»; **DOMAIN** — «тип + правило раз навсегда»; **составной тип** — «несколько полей как одна коробка». Обращение к полю составного типа — **(column).field**.

#### Проверь себя

1. Когда использовать ENUM, а когда CHECK (col IN ('a','b','c'))?  
<details><summary>Ответ</summary> **ENUM** — отдельный тип с фиксированным списком; **ORDER BY** по такому столбцу идёт **по порядку в ENUM** (осмысленная сортировка статусов). **CHECK (col IN (...))** — ограничение на столбце; сортировка по типу столбца (например, тексту). Если нужен **осмысленный порядок** статусов (new → paid → done) — удобнее **ENUM**.</details>

2. Как обратиться к полю составного типа (например, first_name в столбце name)?  
<details><summary>Ответ</summary> **Скобки обязательны:** **(name).first_name**. Пиши `(column).field`, а не `column.field` — иначе PostgreSQL может трактовать точку как обращение к таблице/схеме, а не к полю записи.</details>

#### Запомните

- **ENUM** — фиксированный список значений; порядок задаёт сортировку; новые значения — ALTER TYPE ... ADD VALUE.
- **DOMAIN** — базовый тип + CHECK (и опционально DEFAULT); переиспользование ограничений.
- **Составной тип** — тип «запись» из полей; обращение через **(column).field**.

#### Если прочитал и не понял

**ENUM** — столбец может принимать только значения из заранее заданного списка (например, new, paid, done). **Домен** — «тип с проверкой»: например, «текст, но только в формате email»; задаёшь один раз и используешь в разных таблицах. **Составной тип** — один столбец хранит «пару» или «структуру» (имя + фамилия); обращаешься к полям так: **(name).first_name**.

**Повторение одной фразой.** ENUM = статус из списка; DOMAIN = тип + правило (CHECK) раз навсегда; составной тип = несколько полей как одна коробка; обращение к полю — **(column).field**.
