## §70. Триггеры

### 70.1. BEFORE, AFTER, ROW, STATEMENT

**Цель раздела.**  
Понять модель триггеров в PostgreSQL: **BEFORE** / **AFTER** (момент срабатывания относительно операции), **FOR EACH ROW** / **FOR EACH STATEMENT** (на каждую строку или на операцию целиком), триггерная функция с **NEW** и **OLD** и возвратом **NEW** / **OLD** / **NULL**. После раздела ты сможешь писать триггеры для аудита, валидации и денормализации.

**В этом разделе главное (три строки).** 1) **BEFORE** — функция выполняется **до** применения изменения; может изменить **NEW** (для INSERT/UPDATE) или вернуть NULL (отменить вставку/обновление строки). **AFTER** — после применения; можно читать итоговое состояние строки. 2) **FOR EACH ROW** — триггер вызывается **для каждой строки**; в функции доступны **NEW** (новая строка) и **OLD** (старая строка при UPDATE/DELETE). **FOR EACH STATEMENT** — один раз на операцию; нет построчных NEW/OLD (в PostgreSQL в statement-триггере можно использовать переходные таблицы REFERENCING). 3) Триггерная функция для ROW: возвращает **NEW** (принять изменения), **OLD** или **NULL** (отменить операцию для этой строки в BEFORE).

#### Термины (расшифровка)

- **Триггер** — объект: при событии (INSERT/UPDATE/DELETE) на таблице (или представлении) вызывается **триггерная функция**. Событие задаётся: момент (BEFORE/AFTER/INSTEAD OF), уровень (FOR EACH ROW / FOR EACH STATEMENT), операция (INSERT/UPDATE/DELETE).
- **BEFORE** — выполнить функцию **до** того, как изменение будет применено. В ROW-триггере можно изменить **NEW** (значения, которые будут записаны) или вернуть NULL — тогда для этой строки операция не выполнится (INSERT/UPDATE отменяется для строки).
- **AFTER** — выполнить функцию **после** применения изменения. Изменять NEW уже нельзя; удобно для аудита, денормализации в другие таблицы, отправки событий.
- **FOR EACH ROW** — вызов для **каждой** затронутой строки. В функции доступны **NEW** (record новой строки при INSERT/UPDATE) и **OLD** (record старой строки при UPDATE/DELETE). Возврат: **NEW** (принять), **NULL** (отменить для этой строки в BEFORE).
- **FOR EACH STATEMENT** — один вызов на всю команду (не на каждую строку). В ROW-триггерах нет; в STATEMENT-триггерах в PostgreSQL можно использовать REFERENCING OLD TABLE / NEW TABLE для доступа к набору строк (в переходных таблицах).
- **Триггерная функция** — функция с возвратом **trigger**, без аргументов (или с аргументами через TG_ARGV). Должна возвращать: для BEFORE ROW — NEW, OLD или NULL; для AFTER ROW — обычно NULL (игнорируется).

#### Пошагово: как написать триггер (BEFORE ROW для валидации или AFTER ROW для аудита)

1. **Написать функцию с RETURNS TRIGGER.** Внутри доступны NEW (новая строка при INSERT/UPDATE), OLD (старая при UPDATE/DELETE), TG_TABLE_NAME, TG_OP (INSERT/UPDATE/DELETE). Для BEFORE ROW возвращай NEW (принять изменения) или NULL (отменить операцию для этой строки); для AFTER ROW обычно RETURN NEW или NULL (игнорируется).
2. **Создать триггер:** указать момент (BEFORE/AFTER), уровень (FOR EACH ROW или FOR EACH STATEMENT), операции (INSERT/UPDATE/DELETE), имя функции.
3. **BEFORE** — если нужно изменить данные перед записью или отменить вставку/обновление (валидация, подстановка значений). **AFTER** — если нужно среагировать на уже записанные данные (аудит, денормализация в другую таблицу, отправка события).

#### Примеры (SQL) с пояснениями

```sql
-- Функция: при любом изменении в orders записать в аудит-таблицу
CREATE OR REPLACE FUNCTION audit_trigger_fn()
RETURNS TRIGGER AS $$
BEGIN
  INSERT INTO audit_log (table_name, op, old_data, new_data, changed_at)
  VALUES (TG_TABLE_NAME, TG_OP, row_to_json(OLD), row_to_json(NEW), now());
  RETURN NEW;  -- для AFTER ROW возврат не влияет на операцию
END;
$$ LANGUAGE plpgsql;

-- Триггер: после каждой вставки/обновления/удаления строки в orders вызвать функцию
CREATE TRIGGER audit_orders
  AFTER INSERT OR UPDATE OR DELETE ON orders
  FOR EACH ROW EXECUTE FUNCTION audit_trigger_fn();
```

#### Что будет, если в BEFORE ROW триггере вернуть NULL

Для **INSERT** или **UPDATE**: если триггерная функция возвращает **NULL**, то для **этой строки** операция **не выполняется** — вставка или обновление отменяется. Строка не появится в таблице (или не изменится). Так можно «отфильтровать» строки по условию (валидация: если данные неверные — вернуть NULL и отменить запись). Для **DELETE** возврат NULL в BEFORE тоже отменяет удаление этой строки.

#### Часто путают

**BEFORE и AFTER.** **BEFORE** — функция выполняется **до** того, как строка записана (или удалена); можно изменить NEW (при INSERT/UPDATE) или вернуть NULL и отменить операцию. **AFTER** — функция выполняется **после** записи; изменить результат уже нельзя; удобно для аудита и побочных действий (запись в другую таблицу, отправка в очередь).

**FOR EACH ROW и FOR EACH STATEMENT.** **ROW** — триггер вызывается **для каждой затронутой строки**; в функции есть NEW и OLD для этой строки. **STATEMENT** — триггер вызывается **один раз на всю команду** (например, один раз на INSERT ... SELECT из 1000 строк); в PostgreSQL в statement-триггере можно использовать REFERENCING OLD TABLE / NEW TABLE для доступа к набору строк.

#### Как запомнить

**BEFORE** = до записи, можно изменить NEW или отменить (NULL). **AFTER** = после записи, только «среагировать». **FOR EACH ROW** = на каждую строку (NEW, OLD). **FOR EACH STATEMENT** = один раз на команду. Триггерная функция возвращает **trigger**; для BEFORE ROW — NEW или NULL.

#### Проверь себя

1. В чём разница между BEFORE и AFTER триггером по поводу того, можно ли отменить изменение строки?  
<details><summary>Ответ</summary> **BEFORE** выполняется **до** записи; в ROW-триггере можно изменить **NEW** или вернуть **NULL** и отменить операцию для этой строки (INSERT/UPDATE/DELETE не произойдёт). **AFTER** выполняется **после** записи; изменить результат уже нельзя — триггер подходит для аудита и побочных действий, но не для отмены самой операции.</details>

2. Чем FOR EACH ROW отличается от FOR EACH STATEMENT?  
<details><summary>Ответ</summary> **FOR EACH ROW** — триггер вызывается **для каждой затронутой строки**; в функции доступны **NEW** и **OLD** для этой строки. **FOR EACH STATEMENT** — триггер вызывается **один раз на всю команду** (например, INSERT ... SELECT из 1000 строк); в нём нет построчных NEW/OLD, но можно использовать переходные таблицы (REFERENCING OLD TABLE / NEW TABLE) для доступа к набору строк.</details>

#### Картинка в голове

Триггер — это «охранник у двери»: перед записью (BEFORE) он может проверить данные и либо пропустить (NEW), либо не пустить (NULL). После записи (AFTER) он только фиксирует факт (аудит) или делает что-то ещё (запись в другую таблицу). ROW — охранник проверяет каждую строку по отдельности; STATEMENT — один раз на всю «пачку» строк.

#### Если прочитал и не понял

**Триггер** — код, который выполняется при INSERT/UPDATE/DELETE на таблице. **BEFORE** — до записи (можно изменить данные или отменить запись, вернув NULL). **AFTER** — после записи (удобно для аудита). **FOR EACH ROW** — код вызывается для каждой строки; в коде доступны NEW (новая строка) и OLD (старая). Триггерная функция объявляется как **RETURNS TRIGGER** и вызывается автоматически при срабатывании триггера.

**Повторение одной фразой.** Триггер = функция при INSERT/UPDATE/DELETE; **BEFORE** — до записи (можно отменить NULL), **AFTER** — после (аудит); **FOR EACH ROW** — на каждую строку (NEW, OLD).

**Ещё раз в трёх предложениях.** 1) **BEFORE** — функция выполняется до того, как строка записана; можно изменить NEW или вернуть NULL и отменить операцию для этой строки. 2) **AFTER** — функция выполняется после записи; изменить результат нельзя; удобно для аудита и записи в другие таблицы. 3) **FOR EACH ROW** — триггер вызывается для каждой затронутой строки; в функции доступны NEW (новая строка) и OLD (старая при UPDATE/DELETE).

#### Запомните

- **BEFORE** — до изменения, можно изменить NEW или отменить (NULL); **AFTER** — после изменения.
- **FOR EACH ROW** — на каждую строку (NEW, OLD); **FOR EACH STATEMENT** — на операцию целиком.
- Триггерная функция возвращает **trigger**; для BEFORE ROW — NEW (принять) или NULL (отменить строку).

### 70.2. INSTEAD OF и условные триггеры

**Цель раздела.**  
Использовать **INSTEAD OF** триггеры для представлений (подмена INSERT/UPDATE/DELETE своей логикой) и **условные триггеры** (**WHEN (condition)**) и понимать **порядок** срабатывания нескольких триггеров. После раздела ты сможешь делать обновляемые сложные представления и запускать триггер только при нужном условии.

**В этом разделе главное (три строки).** 1) **INSTEAD OF** — только для представлений; операция (INSERT/UPDATE/DELETE) **не выполняется** над представлением как таковым; вместо неё выполняется код триггера (обычно вставка/обновление/удаление в базовых таблицах). 2) **WHEN (condition)** — триггер срабатывает только если условие истинно; условие может ссылаться на NEW/OLD. 3) Несколько триггеров на одно событие выполняются в порядке по **имени триггера** (алфавитный порядок); порядок можно задавать именованием (trig_01_before, trig_02_after и т.д.).

#### Термины (расшифровка)

- **INSTEAD OF** — тип триггера **только для представлений**. При INSERT/UPDATE/DELETE через представление сама операция над представлением не выполняется (представление — запрос, «таблицы» как таковой нет). Вместо неё выполняется функция триггера; в ней программист сам пишет, какие INSERT/UPDATE/DELETE выполнить в базовых таблицах. Обычно **FOR EACH ROW**.
- **WHEN (condition)** — условие срабатывания триггера. Условие вычисляется для каждой строки (для ROW) или один раз (для STATEMENT); если false, функция не вызывается. В условии можно использовать NEW/OLD (для ROW-триггеров).
- **Порядок триггеров** — если на одну таблицу и одно событие повешено несколько триггеров, они выполняются в порядке по **имени триггера** (сортировка по имени). Имена можно задавать так, чтобы порядок был предсказуемым (например, 01_validate, 02_audit).

#### Пошагово: как сделать обновляемое представление из нескольких таблиц (INSTEAD OF)

1. **Создай представление** из нескольких таблиц (например, JOIN users и orders). Обычный INSERT/UPDATE/DELETE через такое представление «из коробки» недоступен — СУБД не знает, в какую базовую таблицу писать.
2. **Напиши функцию с RETURNS TRIGGER.** В ней доступны NEW (новая строка при INSERT/UPDATE) и OLD (старая при UPDATE/DELETE). Ты сам решаешь: какие INSERT/UPDATE/DELETE выполнить в **базовых таблицах** и с какими значениями (из NEW/OLD).
3. **Создай триггер INSTEAD OF** на представлении: `CREATE TRIGGER ... INSTEAD OF INSERT OR UPDATE OR DELETE ON my_view FOR EACH ROW EXECUTE FUNCTION my_instead_of_fn();`. При INSERT/UPDATE/DELETE через представление сама операция над представлением **не выполняется** — вызывается только твоя функция.
4. **В функции** для INSERT: вставь строку в одну или несколько базовых таблиц (значения из NEW). Для UPDATE: обнови базовые таблицы (значения из NEW, ключи из OLD). Для DELETE: удали из базовых таблиц (ключи из OLD).

#### Пример (SQL): INSTEAD OF INSERT для представления

```sql
-- Представление: заказы с именем пользователя (JOIN orders и users)
CREATE VIEW orders_with_user AS
SELECT o.id, o.user_id, o.total, u.name AS user_name
FROM orders o JOIN users u ON o.user_id = u.id;

-- Функция: при INSERT через представление вставить только в orders (user_id и total из NEW)
CREATE OR REPLACE FUNCTION orders_with_user_insert_fn()
RETURNS TRIGGER AS $$
BEGIN
  INSERT INTO orders (user_id, total) VALUES (NEW.user_id, NEW.total);
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- Триггер INSTEAD OF: при INSERT через представление вызвать функцию
CREATE TRIGGER orders_with_user_insert
  INSTEAD OF INSERT ON orders_with_user
  FOR EACH ROW EXECUTE FUNCTION orders_with_user_insert_fn();
-- Теперь INSERT INTO orders_with_user (user_id, total) VALUES (1, 100); вставит строку в orders
```

#### Что будет, если повесить INSTEAD OF триггер на таблицу

**INSTEAD OF** триггеры в PostgreSQL разрешены **только для представлений (VIEW)**. На обычную таблицу создать INSTEAD OF триггер нельзя — СУБД выдаст ошибку. Логика такая: над таблицей операция INSERT/UPDATE/DELETE выполняется «как есть»; над представлением (которое — запрос) «как есть» выполнить нельзя, поэтому вместо неё выполняется код триггера.

#### Простыми словами

**INSTEAD OF** — «вместо того чтобы выполнять INSERT/UPDATE/DELETE над представлением, выполни мой код». Представление — это запрос, «таблицы» как таковой нет, поэтому СУБД не может сама решить, куда писать. Ты вешаешь INSTEAD OF триггер и в функции сам пишешь: вставь в таблицу orders так, обнови users так. **WHEN (condition)** — «вызывай этот триггер только если условие истинно» (например, только при NEW.total > 0). Несколько триггеров на одно событие выполняются **по имени** (алфавитный порядок) — можно назвать trig_01_validate, trig_02_audit.

#### Картинка в голове

Представление — «виртуальная таблица» (запрос). Когда пользователь делает INSERT в представление, СУБД не может сама «вставить в запрос» — его нет как хранилища. **INSTEAD OF** триггер — это «перехватчик»: «вместо этой операции выполни мой код». В коде ты сам решаешь: вставить в таблицу A, обновить таблицу B и т.д. **WHEN** — «вызывай меня только если условие выполняется».

#### Как запомнить

**INSTEAD OF** — только для представлений; операция INSERT/UPDATE/DELETE **не выполняется** над представлением; вместо неё выполняется код триггера (обычно INSERT/UPDATE/DELETE в базовых таблицах). **WHEN (condition)** — триггер срабатывает только если условие истинно (можно использовать NEW/OLD). Несколько триггеров на одно событие выполняются **по имени** (алфавитный порядок).

#### Проверь себя

1. На какой объект можно повесить INSTEAD OF триггер — на таблицу или на представление?  
<details><summary>Ответ</summary> **Только на представление (VIEW).** В PostgreSQL INSTEAD OF триггеры разрешены только для представлений. На обычную таблицу создать INSTEAD OF триггер нельзя — над таблицей операция выполняется «как есть».</details>

2. Зачем в INSTEAD OF триггере писать INSERT/UPDATE/DELETE в базовые таблицы?  
<details><summary>Ответ</summary> Представление — это запрос, у него нет «хранилища» для строк. При INSERT/UPDATE/DELETE через представление СУБД не знает сама, куда писать. В функции INSTEAD OF триггера ты **явно** выполняешь INSERT/UPDATE/DELETE в **базовых таблицах** (из которых построено представление), подставляя значения из NEW/OLD — так реализуется обновление «через» представление.</details>

#### Если прочитал и не понял

**INSTEAD OF** триггер вешается **только на представление**. Когда пользователь делает INSERT/UPDATE/DELETE через представление, сама операция над представлением не выполняется (представление — запрос, «таблицы» нет). Вместо неё вызывается твоя функция; в ней ты сам пишешь, какие INSERT/UPDATE/DELETE выполнить в базовых таблицах. **WHEN (condition)** — триггер срабатывает только при истинном условии; порядок нескольких триггеров — по имени.

**Повторение одной фразой.** INSTEAD OF — только для VIEW; вместо INSERT/UPDATE/DELETE выполняется твой код (обычно запись в базовые таблицы). WHEN — срабатывать только при условии; порядок триггеров — по имени.

#### Запомните

- **INSTEAD OF** — для представлений; подмена INSERT/UPDATE/DELETE своей логикой в базовых таблицах.
- **WHEN (condition)** — триггер срабатывает только при истинном условии; порядок нескольких триггеров — по имени.
