| Тема | Ключевые пункты |
|------|-----------------|
| Типы данных | DECIMAL для денег и точных расчётов; коллации и кодировки влияют на сравнение строк; `timestamptz` хранит момент времени, а не “локальные часы”. |

## Вопросы для самопроверки (часть XIII)

Ответы под спойлером: по 2–3 вопроса на этап.

### Этап 1. §66 — Типы данных углублённо

1. Почему для денег используют DECIMAL, а не float?  
<details><summary>Ответ</summary> В float десятичные дроби (0.1, 0.2) не представимы точно в двоичном виде; при сложении и накоплении возникают ошибки округления. DECIMAL хранит числа в десятичном виде — арифметика точная; для денег и учёта это обязательно.</details>

2. Чем timestamptz отличается от timestamp (without time zone)? Как хранится timestamptz?  
<details><summary>Ответ</summary> **timestamptz** хранит момент времени в **UTC**; при вводе/выводе используется таймзона сессии или явная зона (AT TIME ZONE). **timestamp** хранит только дату и время без привязки к зоне — один и тот же момент в разных зонах не различается. Для событий рекомендуется timestamptz.</details>

3. Что такое коллация и зачем указывать COLLATE?  
<details><summary>Ответ</summary> **Коллация** задаёт порядок сортировки и правила сравнения строк (алфавит, учёт регистра). COLLATE указывают, когда нужна сортировка для конкретного языка или сравнение без учёта регистра без LOWER() в каждом запросе.</details>

### Этап 2. §67 — JSON, массивы, диапазоны

4. Чем оператор -> отличается от ->> в JSONB?  
<details><summary>Ответ</summary> **->** возвращает значение **как jsonb** (можно продолжать цепочку: data->'user'->'name'). **->>** возвращает значение **как text** — для сравнения в WHERE и вывода.</details>

5. Какой индекс нужен для поиска по содержимому JSONB (@>) и по ключам (?)?  
<details><summary>Ответ</summary> **GIN-индекс** по столбцу jsonb: `CREATE INDEX ON t USING GIN (jsonb_column);` — ускоряет операторы @>, ?, ?|, ?&.</details>

6. Для чего используют unnest() при работе с массивами?  
<details><summary>Ответ</summary> **unnest(array)** разворачивает массив в набор строк (по одной строке на элемент). Используется для JOIN с другими таблицами, агрегации по элементам или вывода списка элементов в столбцах.</details>

### Этап 3. §68 — Полнотекстовый поиск

7. Что такое tsvector и tsquery? Как проверить соответствие текста запросу?  
<details><summary>Ответ</summary> **tsvector** — нормализованный текст (лексемы с позициями); **tsquery** — запрос с лексемами и &, |, !. Соответствие проверяется оператором **@@**: `to_tsvector('english', body) @@ to_tsquery('english', 'word1 & word2')`.</details>

8. Зачем нужны ts_rank и ts_headline?  
<details><summary>Ответ</summary> **ts_rank** (или ts_rank_cd) даёт оценку релевантности документа запросу — для сортировки результатов по релевантности (ORDER BY rank DESC). **ts_headline** возвращает короткий фрагмент текста с подсветкой найденных слов (сниппет для вывода пользователю).</details>

### Этап 4. §69 — Представления

9. Чем обычное представление (VIEW) отличается от материализованного?  
<details><summary>Ответ</summary> **VIEW** — виртуальная таблица: при каждом обращении выполняется запрос, данные не хранятся. **Материализованное представление** хранит результат запроса на диске; данные обновляются только при **REFRESH** (или REFRESH CONCURRENTLY).</details>

10. Зачем REFRESH MATERIALIZED VIEW ... CONCURRENTLY и какое условие для него нужно?  
<details><summary>Ответ</summary> **REFRESH CONCURRENTLY** пересчитывает данные без эксклюзивной блокировки — читатели продолжают видеть старую версию до завершения. Требуется **уникальный индекс** на материализованном представлении, чтобы можно было сопоставлять старую и новую версии при подмене.</details>

### Этап 5. §70 — Триггеры

11. В чём разница между BEFORE и AFTER триггером? Когда можно отменить изменение строки?  
<details><summary>Ответ</summary> **BEFORE** выполняется до применения изменения; можно изменить NEW или вернуть NULL — тогда операция для этой строки не выполнится (отмена). **AFTER** выполняется после применения; изменить результат операции уже нельзя; удобно для аудита и побочных действий.</details>

12. Для чего используют INSTEAD OF триггеры?  
<details><summary>Ответ</summary> **INSTEAD OF** используется **только для представлений**. Операция INSERT/UPDATE/DELETE над представлением не выполняется «как есть»; вместо неё выполняется код триггера, который сам решает, какие изменения сделать в базовых таблицах. Так реализуют обновление сложных представлений.</details>

### Этап 6. §71 — Процедуры и функции

13. Как влияют VOLATILE, STABLE и IMMUTABLE на выполнение функции?  
<details><summary>Ответ</summary> **VOLATILE** — результат может меняться, оптимизатор не кэширует и не переносит вызов. **STABLE** — в одной команде при тех же аргументах результат тот же; можно вычислить один раз в команде. **IMMUTABLE** — результат только от аргументов; можно кэшировать и использовать в индексах по выражению.</details>

14. Как безопасно подставлять имя таблицы в динамический SQL?  
<details><summary>Ответ</summary> Использовать **format()** с **%I** для идентификатора: `format('SELECT * FROM %I', tablename)`. **%I** экранирует идентификатор; не конкатенировать пользовательский ввод в строку запроса — только через %s (значения) и %I (имена) во избежание SQL-инъекций.</details>

### Этап 7. §72 — Последовательности

15. Почему после ROLLBACK в нумерации появляются «дыры»?  
<details><summary>Ответ</summary> **nextval()** выдаёт значение и «привязывает» его к транзакции; при ROLLBACK это значение не возвращается в последовательность — оно уже «использовано» с точки зрения последовательности. Это нормальное поведение; последовательности не гарантируют непрерывную нумерацию.</details>

16. Чем GENERATED ALWAYS AS IDENTITY отличается от BY DEFAULT AS IDENTITY?  
<details><summary>Ответ</summary> **ALWAYS** — столбец заполняется только из sequence; явная вставка своего значения по умолчанию не допускается (в стандарте можно через OVERRIDING). **BY DEFAULT** — при вставке можно явно указать значение; если не указано — подставляется nextval. SERIAL в PostgreSQL по смыслу ближе к BY DEFAULT.</details>

## Типичные ошибки по части XIII

- Хранить деньги и точные бизнес-значения во `float`, а потом бороться с “магическими копейками”.
- Использовать JSONB как оправдание для отсутствия модели данных и индексов.
- Ставить триггеры и `SECURITY DEFINER` функции без чёткой дисциплины ревью, документации и ограничений по правам.
- Ждать от materialized view “автоматической свежести”, забывая про refresh-стратегию и требования к конкурентности.
- Считать sequence источником “идеально непрерывных” номеров, хотя разрывы — нормальная часть её поведения.

## Резюме части XIII

- **Типы углублённо:** DECIMAL(p,s) для денег и точных расчётов; коллации (COLLATE) для сортировки и сравнения строк; timestamptz для моментов времени (хранение в UTC); INTERVAL, extract(), date_trunc(); ENUM, DOMAIN, составные типы.
- **JSON/XML/массивы/диапазоны:** JSONB с операторами ->, ->>, @>, ?, индексы GIN, jsonpath; массивы (ARRAY[], &&, @>, <@, unnest), GIN; диапазоны (int4range, tstzrange), GiST; xml, inet/cidr.
- **Полнотекст:** tsvector, tsquery, конфигурации (simple, english); оператор @@; ts_rank, ts_headline; GIN по tsvector; pg_trgm для нечёткого поиска (similarity, %).
- **Представления:** VIEW — виртуальная таблица, WITH CHECK OPTION; материализованные представления — REFRESH / REFRESH CONCURRENTLY (с UNIQUE INDEX); правила переписывают запрос, триггеры выполняют код.
- **Триггеры:** BEFORE/AFTER, FOR EACH ROW/STATEMENT; NEW, OLD; возврат NEW/NULL в BEFORE ROW; INSTEAD OF для представлений; WHEN (condition); порядок по имени.
- **Процедуры и функции:** PL/pgSQL; VOLATILE/STABLE/IMMUTABLE; параметры IN/OUT/INOUT, VARIADIC; курсоры; динамический SQL (EXECUTE format() с %s, %I); SECURITY DEFINER; PROCEDURE и CALL; PARALLEL SAFE/UNSAFE/RESTRICTED.
- **Последовательности:** nextval, currval, setval; CACHE и разрывы в нумерации; GENERATED ALWAYS/BY DEFAULT AS IDENTITY; SERIAL/BIGSERIAL в PostgreSQL.

**Связь с другими частями:** Часть II (SQL основы) даёт DDL/DML и типы; Часть VIII (реляционные СУБД) — контекст PostgreSQL и типы (JSONB, массивы). Часть XIV (партиционирование, WAL, репликация) опирается на знание представлений, процедур и последовательностей при настройке репликации и бэкапов.
