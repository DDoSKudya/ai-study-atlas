## §72. Последовательности и IDENTITY

### 72.1. SEQUENCE и nextval/currval/setval

**Цель раздела.**  
Управлять объектом **SEQUENCE**: **CREATE SEQUENCE**, **nextval()**, **currval()**, **setval()**; свойства **START**, **INCREMENT**, **MINVALUE**, **MAXVALUE**, **CYCLE**, **CACHE**. После раздела ты будешь использовать последовательности в DEFAULT и понимать, откуда берутся «дыры» в нумерации.

**В этом разделе главное (три строки).** 1) **nextval(seq)** — взять следующее значение последовательности и вернуть его; значение «привязывается» к текущей транзакции (при откате транзакции значение не возвращается в последовательность — будут разрывы). 2) **currval(seq)** — последнее значение, выданное **в этой сессии** для этой последовательности (без вызова nextval в сессии — ошибка). 3) **setval(seq, value)** — установить текущее значение последовательности; следующий nextval вернёт value + increment.

#### Термины (расшифровка)

- **SEQUENCE (последовательность)** — объект-счётчик: при вызове **nextval(seq)** возвращается следующее число и внутренний счётчик увеличивается. Создание: **CREATE SEQUENCE name [START n] [INCREMENT n] [MINVALUE n] [MAXVALUE n] [CYCLE]** **[CACHE n]**.
- **nextval(seq)** — получить следующее значение; оно «используется» (при откате транзакции не возвращается — разрыв в нумерации).
- **currval(seq)** — текущее значение для **этой сессии** (последнее, выданное nextval в этой сессии). Без предшествующего nextval в сессии — ошибка.
- **setval(seq, value [, is_called])** — установить значение; по умолчанию следующий nextval вернёт value + increment. **CACHE n** — сервер заранее резервирует n значений в памяти; при откате транзакции зарезервированные значения не возвращаются — возможны разрывы.

#### Пошагово: как использовать последовательность в DEFAULT и почему появляются «дыры»

1. **Создать последовательность:** `CREATE SEQUENCE seq_id START 1 INCREMENT 1;` — счётчик начнётся с 1 и будет увеличиваться на 1.
2. **Подключить к столбцу:** `id bigint DEFAULT nextval('seq_id'::regclass)` — при INSERT без указания id будет вызван nextval и подставлено следующее значение.
3. **nextval(seq)** — взять следующее число и вернуть его. Это число «привязывается» к транзакции: при **ROLLBACK** оно **не возвращается** в последовательность — будут разрывы (1, 2, 3, откат, следующий INSERT получит 4).
4. **currval(seq)** — вернуть последнее значение, выданное **в этой сессии** для этой последовательности. Без предшествующего nextval в этой сессии — ошибка.
5. **setval(seq, value)** — установить текущее значение последовательности; следующий nextval вернёт value + increment (по умолчанию).

#### Примеры (SQL) с пояснениями

```sql
-- Создание последовательности: старт 1, шаг 1
CREATE SEQUENCE seq_id START 1 INCREMENT 1;

-- Столбец id по умолчанию берёт следующее значение из последовательности
CREATE TABLE t (id bigint DEFAULT nextval('seq_id'::regclass) PRIMARY KEY);

-- Ручное получение следующего значения (и «занятие» его)
SELECT nextval('seq_id');

-- Текущее значение в этой сессии (последнее, выданное nextval в этой сессии)
SELECT currval('seq_id');  -- без nextval в сессии — ошибка

-- Установить значение: следующий nextval вернёт 101 (если increment = 1)
SELECT setval('seq_id', 100);
```

#### Что будет, если вызвать nextval и сделать ROLLBACK

**nextval** выдаёт значение и «занимает» его — с точки зрения последовательности это значение уже использовано. При **ROLLBACK** транзакции это значение **не возвращается** в последовательность. Следующий INSERT (в другой транзакции) получит следующее число — в нумерации будет «дыра». Это **нормальное** и ожидаемое поведение: последовательности не гарантируют непрерывную нумерацию. Если нужны строго подряд идущие номера без дыр — нужен другой механизм (например, отдельная таблица с блокировкой или центральный генератор).

#### Часто путают

**nextval и currval.** **nextval** — «выдать следующее значение и запомнить его»; каждый вызов даёт новое число. **currval** — «какое значение было выдано последним nextval **в этой сессии**»; без nextval в сессии currval даёт ошибку. currval не выдаёт новое значение, а только возвращает последнее выданное в этой сессии.

**CACHE и разрывы.** **CACHE n** — сервер заранее резервирует n значений в памяти (например, 10); при следующем nextval выдаётся из зарезервированного блока. При откате транзакции или краше сервера зарезервированные, но не использованные значения «пропадают» — в нумерации появятся разрывы. CACHE ускоряет выдачу (меньше обращений к общему счётчику), но увеличивает «дыры» при откатах.

#### Простыми словами

**Последовательность** — счётчик: каждый вызов **nextval** даёт следующее число и «занимает» его — даже если транзакция откатится, число обратно не вернётся (появятся «дыры»). **currval** — «какое число было выдано последним nextval в этой сессии»; без nextval в сессии — ошибка. **setval** — «переставить счётчик» (следующий nextval даст value + increment). Для автоинкремента столбца в DEFAULT укажи **nextval('seq_name'::regclass)**.

#### Как запомнить

**nextval** — взять следующее (при ROLLBACK не возвращается). **currval** — последнее в сессии (без nextval в сессии — ошибка). **setval** — установить текущее значение. **CACHE** — быстрее, но больше разрывов при откатах.

#### Проверь себя

1. Почему после ROLLBACK в нумерации id появляются «дыры» (например, 1, 2, 3, откат, следующий id = 5)?  
<details><summary>Ответ</summary> **nextval** выдаёт значение и «привязывает» его к транзакции; с точки зрения последовательности это значение уже **использовано**. При **ROLLBACK** оно **не возвращается** в последовательность — следующий nextval (в другой транзакции) выдаст следующее число. Разрывы в нумерации — нормальное поведение последовательностей.</details>

2. В чём разница между nextval и currval?  
<details><summary>Ответ</summary> **nextval(seq)** — **взять следующее** значение последовательности и вернуть его; каждый вызов даёт новое число. **currval(seq)** — вернуть **последнее** значение, выданное nextval **в этой сессии**; новый номер не выдаёт. Без предшествующего nextval в сессии currval даёт ошибку.</details>

#### Картинка в голове

Последовательность — это очередь номеров. **nextval** — «возьми следующий номер из очереди»; ты его занял, даже если потом откатил транзакцию — номер обратно не кладётся. **currval** — «какой номер ты взял последним в этой сессии» (не выдаёт новый). **setval** — «переставить указатель очереди» (следующий nextval выдаст value + 1). Разрывы в нумерации — норма: номера не обязаны идти подряд.

#### Если прочитал и не понял

**Последовательность** — объект-счётчик. **nextval(seq)** — получить следующее число; при ROLLBACK оно не возвращается — будут «дыры». **currval(seq)** — «последнее число, выданное nextval в этой сессии»; без nextval в сессии — ошибка. **setval(seq, n)** — установить текущее значение (следующий nextval даст n + increment). Для автоинкремента столбца в DEFAULT укажи **nextval('seq_name'::regclass)**.

**Повторение одной фразой.** Последовательность — счётчик; **nextval** берёт следующее (при ROLLBACK не возвращается → дыры); **currval** — последнее в сессии; **setval** — установить значение. Разрывы в нумерации нормальны.

**Ещё раз в трёх предложениях.** 1) **nextval(seq)** выдаёт следующее число и «занимает» его — при ROLLBACK оно не возвращается в последовательность, поэтому в нумерации появляются «дыры». 2) **currval(seq)** возвращает последнее значение, выданное nextval **в этой сессии** (новое не выдаёт); без nextval в сессии — ошибка. 3) **setval(seq, n)** устанавливает текущее значение последовательности; следующий nextval вернёт n + increment. Для автоинкремента столбца в DEFAULT укажи **nextval('seq_name'::regclass)**.

#### Запомните

- **nextval** — следующее значение (при откате не возвращается); **currval** — последнее в сессии; **setval** — установить значение.
- **CACHE** ускоряет выдачу, но при откатах и крашах даёт разрывы в нумерации.

### 72.2. IDENTITY и разрывы в нумерации

**Цель раздела.**  
Использовать **GENERATED ALWAYS AS IDENTITY** и **GENERATED BY DEFAULT AS IDENTITY** (стандартный автоинкремент) и понимать связь с **SERIAL**/ **BIGSERIAL** в PostgreSQL (обёртка над sequence). Понять причины **разрывов** в нумерации: CACHE, откат транзакции, репликация.

**В этом разделе главное (три строки).** 1) **GENERATED ALWAYS AS IDENTITY** — столбец только из последовательности; вставка своего значения по умолчанию запрещена (или через OVERRIDING). **GENERATED BY DEFAULT AS IDENTITY** — можно подставить своё значение при вставке; если не указано — берётся nextval. 2) **SERIAL** / **BIGSERIAL** в PostgreSQL — сокращение: создаётся sequence и столбец с DEFAULT nextval(seq); аналог BY DEFAULT. 3) **Разрывы** в нумерации нормальны: nextval коммитится «сразу» (значение не откатывается при ROLLBACK); CACHE резервирует блок значений — при откате они пропадают; при логической репликации sequence на реплике может отставать или задаваться вручную.

#### Термины (расшифровка)

- **GENERATED ALWAYS AS IDENTITY** — столбец заполняется только последовательностью; явная вставка значения по умолчанию не разрешена (в стандарте можно через OVERRIDING SYSTEM VALUE). **GENERATED BY DEFAULT AS IDENTITY** — при вставке можно указать значение явно; иначе — nextval.
- **SERIAL** — в PostgreSQL эквивалент: INTEGER с DEFAULT nextval('table_col_seq'::regclass); создаётся sequence с именем table_col_seq. **BIGSERIAL** — то же с BIGINT.
- **Разрывы в нумерации** — ожидаемое поведение: nextval выдаёт значение и не «возвращает» его при ROLLBACK; при CACHE > 1 выданные значения при откате не переиспользуются; при репликации sequence на реплике может быть другим. Для счётчиков «без дыр» нужны другие механизмы (например, отдельная таблица с блокировкой или центральный генератор).

#### Пошагово: когда использовать ALWAYS и когда BY DEFAULT AS IDENTITY

1. **GENERATED ALWAYS AS IDENTITY** — столбец заполняется **только** из последовательности. Явная вставка своего значения по умолчанию **не допускается** (в стандарте SQL можно через OVERRIDING SYSTEM VALUE). Удобно, когда важно, чтобы id всегда выдавался только последовательностью.
2. **GENERATED BY DEFAULT AS IDENTITY** — при INSERT можно **явно указать** значение (например, при переносе данных из другой БД); если значение не указано — подставляется nextval. Обычный сценарий для «автоинкремента» в приложении.
3. **SERIAL / BIGSERIAL** в PostgreSQL — сокращение: создаётся sequence и столбец с DEFAULT nextval(seq); по смыслу ближе к **BY DEFAULT** (можно вставить своё значение).

#### Что будет, если вставить своё значение в столбец GENERATED ALWAYS AS IDENTITY

При INSERT с явным указанием значения для столбца **GENERATED ALWAYS AS IDENTITY** (например, INSERT INTO t (id, name) VALUES (100, 'x')) СУБД выдаст ошибку: нельзя подставлять значение для столбца ALWAYS AS IDENTITY. Если нужно вставить своё значение (например, при миграции данных), используй **GENERATED BY DEFAULT AS IDENTITY** или в стандарте SQL — **OVERRIDING SYSTEM VALUE** в INSERT (в PostgreSQL поддерживается).

#### Часто путают

**ALWAYS и BY DEFAULT.** **ALWAYS** — столбец **только** из sequence; вставить своё значение нельзя (без OVERRIDING). **BY DEFAULT** — можно вставить своё значение; если не указано — подставляется nextval. Для «гибкого» автоинкремента (миграции, импорт) — BY DEFAULT.

**SERIAL и IDENTITY.** **SERIAL** в PostgreSQL — это создание sequence + столбец с DEFAULT nextval(seq); по поведению как **BY DEFAULT AS IDENTITY** (можно вставить своё значение). **IDENTITY** — стандарт SQL; ALWAYS/BY DEFAULT задаётся явно. Для новых таблиц предпочтительно **GENERATED BY DEFAULT AS IDENTITY** (стандарт).

#### Простыми словами

**IDENTITY** — стандартный способ «автоинкремента» столбца. **ALWAYS** — значение **только** из sequence; вставить своё нельзя (миграция данных — через OVERRIDING или BY DEFAULT). **BY DEFAULT** — можно вставить своё значение (при импорте, переносе); если не указано — подставляется nextval. **SERIAL** в PostgreSQL — то же по смыслу, что BY DEFAULT (создаётся sequence и DEFAULT nextval). «Дыры» в нумерации нормальны: nextval не возвращается при ROLLBACK, CACHE даёт разрывы.

#### Картинка в голове

**ALWAYS** — «билет только из кассы»: свой номер подставить нельзя. **BY DEFAULT** — «если не указал номер — выдадут из кассы; если указал свой (при импорте) — примут». **SERIAL** — та же «касса» (sequence) под капотом. Разрывы в нумерации — как пропущенные номера в очереди: кто-то взял номер и не пришёл — следующий получит следующий номер.

#### Как запомнить

**IDENTITY** — стандартный автоинкремент: **ALWAYS** = только из sequence; **BY DEFAULT** = можно подставить своё значение. **SERIAL** в PostgreSQL — обёртка над sequence, по смыслу как BY DEFAULT. Разрывы в нумерации нормальны (nextval при ROLLBACK не возвращается; CACHE; репликация).

#### Проверь себя

1. Когда использовать GENERATED ALWAYS AS IDENTITY, а когда BY DEFAULT AS IDENTITY?  
<details><summary>Ответ</summary> **ALWAYS** — когда важно, чтобы значение столбца **всегда** выдавалось только последовательностью (вставить своё нельзя). **BY DEFAULT** — когда при миграции или импорте нужно **подставить свои** значения; если значение не указано — подставляется nextval. В большинстве приложений достаточно **BY DEFAULT**.</details>

2. Почему в нумерации id бывают «дыры» даже без явного ROLLBACK?  
<details><summary>Ответ</summary> **CACHE n** резервирует блок значений в памяти; при откате транзакции или краше сервера зарезервированные, но не использованные значения **не возвращаются** в последовательность — в нумерации появятся разрывы. Кроме того, при логической репликации sequence на реплике может отставать или задаваться вручную — это тоже даёт отличия в нумерации.</details>

#### Если прочитал и не понял

**IDENTITY** — способ задать автоинкремент столбца в стандарте SQL. **ALWAYS** — значение только из sequence; вставить своё нельзя. **BY DEFAULT** — можно вставить своё значение; иначе подставляется nextval. **SERIAL** в PostgreSQL — то же по смыслу, что BY DEFAULT (sequence + DEFAULT nextval). «Дыры» в нумерации нормальны: nextval не возвращается при ROLLBACK, CACHE даёт разрывы при откатах.

**Повторение одной фразой.** IDENTITY = автоинкремент по стандарту: **ALWAYS** — только из sequence; **BY DEFAULT** — можно подставить своё. SERIAL = обёртка над sequence; разрывы в нумерации нормальны.

**Ещё раз в трёх предложениях.** 1) **GENERATED ALWAYS AS IDENTITY** — столбец заполняется только последовательностью; вставить своё значение нельзя (без OVERRIDING). 2) **GENERATED BY DEFAULT AS IDENTITY** — можно вставить своё значение при импорте/миграции; если не указано — подставляется nextval; в большинстве приложений используют именно BY DEFAULT. 3) **SERIAL/BIGSERIAL** в PostgreSQL — создаётся sequence и столбец с DEFAULT nextval(seq); по смыслу как BY DEFAULT; разрывы в нумерации нормальны (ROLLBACK, CACHE, репликация).

#### Запомните

- **IDENTITY** — стандартный автоинкремент: ALWAYS (только из sequence) или BY DEFAULT (можно подставить значение).
- **SERIAL/BIGSERIAL** в PostgreSQL — обёртка над sequence; разрывы в нумерации нормальны (ROLLBACK, CACHE, репликация).
