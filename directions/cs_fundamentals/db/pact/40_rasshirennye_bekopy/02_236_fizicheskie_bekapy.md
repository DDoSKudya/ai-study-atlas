## 236. Физические бэкапы

### Цель раздела

Понять, как устроены **физические бэкапы** (копирование файлов данных/кластера), какие инструменты есть в популярных СУБД, как обеспечить **консистентность** таких копий и когда физический бэкап предпочтительнее логического.

### В этом разделе главное

1. Физический бэкап — это **копия файлов**, а не SQL‑структур.
2. Главное слово — **консистентность**: нельзя просто скопировать “каталог данных” работающей СУБД.
3. Специализированные утилиты (`pg_basebackup`, XtraBackup, RMAN, `BACKUP DATABASE`) умеют делать **онлайн‑копии**.
4. Snapshot’ы на уровне диска/облака — удобный способ делать быстрые физические бэкапы.
5. Физический бэкап обычно **быстрее восстанавливается**, но хуже переносится между версиями/платформами.

### Термины

- **Консистентный бэкап** — копия данных, которая может быть корректно запущена/прочитана СУБД.
- **pg_basebackup** — утилита PostgreSQL для создания физической копии кластера.
- **XtraBackup** — утилита Percona для инкрементальных физических бэкапов MySQL/InnoDB.
- **RMAN** — инструмент Oracle для резервного копирования и восстановления.
- **Snapshot (снимок)** — консистентный снимок тома/диска на уровне ОС/облака.

### Теория и правила

#### 1) Почему нельзя просто `cp -R /var/lib/postgresql/...`

СУБД активно **пишет и читает** файлы данных:

- изменяет страницы внутри файлов;
- пишет журнал (WAL/binlog);
- делает чекпоинты;
- может держать изменённые страницы только в памяти.

Если в этот момент “на лету” копировать файлы:

- часть файлов будет в состоянии “до изменений”, часть — “после”;
- журнал может не соответствовать данным;
- при попытке запуска из такого бэкапа СУБД увидит “грязное” состояние и, в лучшем случае, откажется стартовать.

Поэтому нужны либо:

- **специализированные утилиты**, которые знают, как заморозить состояние, или
- **snapshot’ы**, делающие снимок **атомарно** на уровне тома.

#### 2) pg_basebackup (PostgreSQL)

`pg_basebackup`:

- подключается к серверу как реплика;
- читает данные последовательно;
- параллельно получает WAL, чтобы сделать бэкап консистентным;
- умеет создавать бэкапы в формате:
  - `plain` (копия каталога),
  - `tar` (tar‑архив).

Пример (упрощённо):

```bash
pg_basebackup -h primary-db -U replicator -D /backups/base/2026-03-11 -Fp -Xs -P
```

- `-Fp` — формат plain;
- `-Xs` — включить архивирование WAL (stream);
- `-P` — прогресс.

Для восстановления:

1. Останавливаем СУБД.
2. Разворачиваем содержимое бэкапа в `PGDATA`.
3. Кладём нужные WAL‑файлы (если используем PITR).
4. Настраиваем `postgresql.conf`/`postgresql.auto.conf` и запускаем.

#### 3) Физические бэкапы MySQL / SQL Server / Oracle

- **MySQL**:
  - Percona XtraBackup делает горячие физические бэкапы InnoDB;
  - поддерживает full и incremental;
  - работает с innodb‑файлами и журналом redo.
- **SQL Server**:
  - `BACKUP DATABASE` и `BACKUP LOG` создают бэкап на уровне файлов БД;
  - есть разные типы: full, differential, log;
  - восстановление происходит через `RESTORE DATABASE` с нужной комбинацией.
- **Oracle**:
  - RMAN делает физические копии datafiles, control files, архивных redo logs;
  - поддерживает хот‑бэкапы и интеграцию со storage snapshot’ами.

#### 4) Snapshot’ы (LVM, ZFS, EBS, Ceph, SAN)

Идея:

- том/файловая система поддерживает **моментальный снимок**;
- можно за доли секунды создать snapshot большого тома;
- после этого спокойно копировать данные со snapshot’а (не нагружая прод).

Важно:

- перед snapshot полезно **синхронизировать** состояние СУБД:
  - сделать чекпоинт,
  - при необходимости кратко остановить запись (для некоторых движков).
- нужно понимать, как **storage** гарантирует консистентность (crash‑consistent vs application‑consistent).

### Пошагово: физический бэкап PostgreSQL через pg_basebackup

1. Создать пользователя‑реплику с правами на `REPLICATION`.
2. Включить `wal_level = replica` (или выше) и настроить `max_wal_senders`.
3. Запустить `pg_basebackup` с нужными параметрами.
4. Убедиться, что бэкап **полностью скачался** и WAL‑сегменты получены.
5. Записать в систему бэкапов:
   - дату/время;
   - размер;
   - версию PostgreSQL;
   - путь к архиву WAL.
6. Периодически **проверять восстановление** из такого бэкапа (отдельный сервер).

### Простыми словами

Физический бэкап — это как **сделать копию всего жёсткого диска** с базой:

- если копию сделали правильно (в одном моменте времени),  
  можно просто “подключить” её на другом сервере, и всё заработает;
- если копию делали “по кусочкам”, пока диск активно менялся,  
  система может не понять, что у неё в файлах, и отказать.

### Картинка в голове

Вообрази:

- БД — это шкаф с папками (файлы данных).
- Журнал (WAL/binlog) — лента с записью всех изменений.
- Физический бэкап — это **фотография всего шкафа и ленты на момент времени**.  
  Если фотографировать шкаф, пока кто‑то перекладывает папки, — фото получится “размазано”.

### Как запомнить

- **Физический** бэкап = **быстрый запуск**, **жёсткая привязка** к версии/формату.
- **Логический** бэкап = **гибкость и переносимость**, но медленное восстановление.

### Примеры

#### PostgreSQL + snapshot EBS

1. На время snapshot:
   - отправляем серверу команду сделать чекпоинт;
   - ждём его завершения.
2. Запускаем snapshot тома EBS.
3. Монтируем snapshot как отдельный том на другой машине.
4. Копируем оттуда данные в архив.

#### MySQL + XtraBackup

```bash
innobackupex --user=backup --password=... /backups/mysql/2026-03-11
```

Потом `--apply-log` для подготовки бэкапа.

### Практика / реальные сценарии

- Крупный интернет‑магазин на PostgreSQL:
  - перешёл с ночных логических дампов на `pg_basebackup` + WAL‑архив;
  - время восстановления сократилось с часов до десятков минут.
- Банк на Oracle:
  - используют RMAN + snapshot’ы SAN‑хранилища;
  - тестируют восстановление на стендах, идентичных продакшну.

### Типичные ошибки

- Копирование каталога данных “на горячую” без специальных средств.
- Делание snapshot’а без учёта того, что БД **пишет** в момент снимка.
- Игнорирование версии СУБД: попытка восстановить физический бэкап на другой версии/архитектуре.

### Что будет, если…

- …сделать snapshot в момент активной записи без crash‑consistent гарантий?
  - БД может считать, что произошёл краш и попытаться recovery; иногда это получится, иногда — нет.
- …удалить архивы WAL, на которые ссылается физический бэкап?
  - Нельзя будет прокрутить изменения дальше точки бэкапа (сломается PITR/реплика).

### Проверь себя

1. Почему snapshot на уровне облачного тома можно считать физическим бэкапом?
<details><summary>Ответ</summary>
Потому что он сохраняет физическое состояние тома/файловой системы, на которой лежит БД. Это копия “сырых” блоков/страниц, а не SQL‑структур. При восстановлении мы подключаем этот снимок или копируем файлы в новое место — это типичный сценарий физического бэкапа.
</details>

2. В чём преимущество `pg_basebackup` перед ручным копированием каталога данных?
<details><summary>Ответ</summary>
`pg_basebackup` знает про внутренние детали PostgreSQL: координирует чтение файлов и WAL, обеспечивает консистентность, может стримить журнал и создавать бэкап онлайн без остановки сервера. Ручное копирование этого не делает и легко создаёт неконсистентную копию.
</details>

### Запомните

- Физический бэкап — основной инструмент **быстрого восстановления больших баз**.
- Консистентность важнее скорости: лучше медленный, но рабочий бэкап, чем быстрый, но битый.
