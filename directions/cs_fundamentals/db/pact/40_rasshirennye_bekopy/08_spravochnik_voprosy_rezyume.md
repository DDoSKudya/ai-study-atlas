### Типы бэкапов и их свойства

| Тип                    | Скорость бэкапа   | Скорость восстановления                      | Место                 | Переносимость  | Типичные СУБД/инструменты                                            |
| ---------------------- | ----------------- | -------------------------------------------- | --------------------- | -------------- | -------------------------------------------------------------------- |
| Полный                 | медленная/средняя | быстрая                                      | много                 | низкая/средняя | PostgreSQL `pg_basebackup`, SQL Server full backup, Oracle RMAN full |
| Инкрементальный        | быстрая           | медленная/средняя (зависит от длины цепочки) | экономично            | низкая/средняя | XtraBackup, RMAN incremental, SQL Server log/diff backups            |
| Дифференциальный       | средняя           | средняя/быстрая                              | растёт со временем    | низкая/средняя | SQL Server differential, RMAN incremental level 1 (cum)              |
| Логический             | медленная         | медленная                                    | много (но сжимаемо)   | высокая        | `pg_dump`, `mysqldump`, Data Pump, экспорт из SQLite                 |
| Snapshot (том/инстанс) | очень быстрая     | быстрая                                      | зависит от реализации | низкая         | LVM/ZFS snapshots, AWS EBS/RDS snapshots                             |

### СУБД → основные инструменты

| СУБД          | Физические бэкапы                      | Логические бэкапы        | Журнал/архив                                      | Облачные возможности                                              |
| ------------- | -------------------------------------- | ------------------------ | ------------------------------------------------- | ----------------------------------------------------------------- |
| PostgreSQL    | `pg_basebackup`, pgBackRest, snapshots | `pg_dump` / `pg_restore` | WAL + `archive_command`, сторонние WAL‑архиваторы | RDS, Cloud SQL, Azure Database (managed backups, snapshots, PITR) |
| MySQL/MariaDB | XtraBackup, snapshots                  | `mysqldump`              | binlog + архивирование                            | RDS, Cloud SQL, Azure Database for MySQL                          |

| SQLite        | файловые копии, snapshots              | `.dump`                  | – (журнал на уровне файла)                        | обычно самописные/облачные бэкапы                                 |

## Частые сценарии

### Сценарий 1. Случайное массовое удаление

- Используем:
  - физический/полный бэкап (до момента удаления),
  - журналы (WAL/binlog) для PITR до момента “до удаления”,
  - возможно, логический экспорт нужных таблиц для аккуратного возврата в прод.

### Сценарий 2. Падение диска/хоста

- Используем:
  - последний полный + инкрементальные бэкапы,
  - журналы до момента падения (если есть),
  - для managed‑БД — восстановление инстанса из snapshot + PITR.

### Сценарий 3. Миграция в другой регион/облако

- Используем:
  - логические бэкапы (для переносимости),
  - репликацию между кластерами (где возможно),
  - отдельные копии бэкапов в целевом регионе.

### Сценарий 4. Отладка сложного инцидента

- Поднимаем стенд через PITR на момент/окно инцидента;
- анализируем журналы/метрики, не трогая прод.

## Вопросы для самопроверки по части

1. Как связаны между собой RPO/RTO и выбор комбинации full/incremental/differential/WAL?
2. Когда физические бэкапы явно предпочтительнее логических, и наоборот?
3. Какие риски возникают при неверной настройке архивирования журнала?
4. Как использовать облачные возможности бэкапов, не попадая в vendor lock‑in?
5. Какой минимальный набор шагов должен входить в регулярную проверку бэкапов?

<details><summary>Ответы</summary>

1. RPO определяет **частоту бэкапов и необходимость журналов** (чем меньше RPO, тем чаще инкрементальные/журналы), RTO — **скорость восстановления** (физические/дифференциальные бэкапы быстро восстанавливаются, длинные цепочки инкрементальных замедляют). Выбор сочетания типов бэкапов и журналов должен обеспечивать достижение заданных RPO/RTO.
2. Физические бэкапы предпочтительнее для **больших прод‑БД**, где важна скорость восстановления и минимальный простой. Логические — для **миграций, частичных восстановлений, тестовых сред и переносимости** между версиями/СУБД. В реальности их часто комбинируют.
3. При неверной настройке архивирования журнала можно незаметно **потерять часть журнала**, сделать невозможным PITR, повредить цепочки репликации и переполнить диски. Это приводит к ухудшению RPO и, в худшем случае, к невосстанавливаемой потере данных после последнего бэкапа.
4. Нужно понимать, как работают automated backups и PITR, обязательно делать **дополнительные независимые копии** (логические/физические) и продумывать сценарий выхода в другой регион/провайдера. Не стоит полагаться только на один сервис и один аккаунт.
5. Минимум: выбор бэкапа и журналов → развёртывание на стенде → запуск БД → базовые проверки целостности → простые сценарии приложения → фиксация времени и проблем. Всё это должно быть задокументировано и повторяться регулярно.

</details>

## Типичные ошибки по части

- Отсутствие формальных RPO/RTO и, как следствие, случайный выбор частоты бэкапов.
- Ориентация только на один тип бэкапа (только full, только дампы, только snapshot’ы).
- Отсутствие архивирования журнала или его мониторинга.
- Слепая вера в облачные бэкапы без понимания ограничений.
- Отсутствие регулярных проверок восстановления и документации.

## Резюме части XL

- **Бэкапы — это не файлы, а стратегия**, которая начинается с RPO/RTO и заканчивается проверенными сценариями восстановления.
- **Физические и логические бэкапы дополняют друг друга**: первые дают скорость, вторые — гибкость и переносимость.
- **WAL/binlog‑архивирование и PITR** позволяют “перематывать время” и спасать от логических ошибок, но требуют дисциплины и мониторинга.
- **Облачные бэкапы сильны, но ограничены**: их нужно сочетать с собственными копиями и DR‑планами.
- **Настоящий бэкап — это тот, из которого вы уже успешно восстанавливались**. Всё остальное — только предположение.
