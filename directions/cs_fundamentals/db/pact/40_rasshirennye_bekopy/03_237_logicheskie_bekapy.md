## 237. Логические бэкапы

### Цель раздела

Разобраться, что такое **логические бэкапы**, когда их использовать, как они работают в разных СУБД и почему, несмотря на медленность, они незаменимы для миграций, частичных восстановлений и проверки схемы.

### В этом разделе главное

1. Логический бэкап — это **дамп структуры и данных** на уровне SQL или формата БД.
2. Он **переносимее** физического, но обычно **медленнее** и тяжелее для CPU/I/O.
3. Умеет делать **частичные бэкапы** (отдельные схемы/таблицы).
4. Идеален для **миграций между версиями/СУБД** и для **тестовых стендов**.
5. В больших системах логический бэкап редко бывает основным методом аварийного восстановления, но остаётся важным вспомогательным инструментом.

### Термины

- **Логический дамп** — файл (или набор файлов), содержащий DDL и данные (INSERT/COPY/json и т.п.).
- **pg_dump / pg_restore** — инструменты PostgreSQL для логических бэкапов и восстановления.
- **mysqldump** — утилита MySQL для логических дампов.
- **Data pump** (Oracle), **BCP** (SQL Server), экспорты MongoDB/NoSQL — аналоги логических дампов.

### Теория и правила

#### 1) Как устроен логический бэкап

Логический бэкап:

- читает **каталоги/метаданные** (структуру схемы, таблиц, индексов);
- затем **данные таблиц** и преобразует их в:
  - SQL‑операторы (`CREATE TABLE`, `INSERT`, `COPY` и т.д.), или
  - специализированный формат (binary, custom, directory), удобный для быстрого восстановления.

Восстановление:

1. создаёт структуру (DDL),
2. наполняет таблицы данными,
3. создаёт индексы/констрейнты,
4. включает триггеры и т.д.

#### 2) Примеры по СУБД

- **PostgreSQL**:
  - `pg_dump`:
    - форматы: `plain`, `custom`, `directory`, `tar`;
    - можно дампить:
      - всю БД,
      - отдельную схему/таблицы;
    - умеет параллельно (`-j`).
  - `pg_restore` восстанавливает из `custom`/`directory`.
- **MySQL**:
  - `mysqldump`:
    - генерирует SQL‑скрипт;
    - может дампить структуру отдельно (`--no-data`) или данные (`--no-create-info`).
- **SQL Server**:
  - экспорт/импорт данных, `bcp`, SSIS, DACPAC/BACPAC.
- **Oracle**:
  - Data Pump (`expdp`/`impdp`), legacy `exp`/`imp`.
- **SQLite**:
  - `sqlite3 db .dump` — простой текстовый дамп всей схемы и данных.

#### 3) Плюсы и минусы логических бэкапов

Плюсы:

- переносимость между:
  - версиями одной СУБД,
  - иногда даже разными СУБД (с маппингом типов и конструкций);
- возможность **частичного бэкапа**:
  - только схема,
  - только часть таблиц,
  - только справочники;
- пригодность для:
  - **миграций**,
  - **маскировки данных**,
  - **анонимизации** (через трансформации при импорте).

Минусы:

- долго:
  - чтение всех данных,
  - генерация SQL/формата,
  - возможные блокировки/нагрузка;
- восстановление может занимать часы/дни на больших объёмах;
- требует **достаточного свободного места** (дамп + временные файлы).

### Пошагово: логический бэкап PostgreSQL

1. Решить, **что именно** дампить:
   - всю БД (`-d dbname`);
   - отдельную схему (`-n schema`);
   - конкретные таблицы (`-t`).
2. Выбрать формат:

```bash
pg_dump -Fc -d mydb -f /backups/logical/mydb_2026-03-11.dump
```

3. Для больших БД — использовать **параллельный дамп**:

```bash
pg_dump -Fd -j 8 -d mydb -f /backups/logical/mydb_2026-03-11_dir
```

4. Для восстановления:

```bash
createdb mydb_restored
pg_restore -j 8 -d mydb_restored /backups/logical/mydb_2026-03-11_dir
```

### Простыми словами

Логический бэкап — это как **сделать сценарий, который умеет заново создать базу**:

- сначала сценарий создаёт пустые таблицы и индексы,
- потом “заливает” данные,
- потом включает ограничения и триггеры.

Такой сценарий можно запускать на других серверах/версиях (с ограничениями), менять по пути, маскировать данные и т.д.

### Картинка в голове

Представь, что у тебя есть:

- не просто “образы диска”, а **инструкции**: “создай таблицу A с такими полями, потом вставь туда такие строки, потом создай индекс B”.

Логический бэкап — это именно такие инструкции. Они дольше исполняются, но гораздо более гибкие.

### Как запомнить

> **Физический = образ диска** → быстро стартует, но не очень переносим.  
> **Логический = сценарий создания БД** → дольше выполняется, но легко менять и переносить.

### Примеры

- **Миграция PostgreSQL 13 → 16**:
  - для небольшой БД проще сделать `pg_dump | pg_restore`, получая чистую схему и данные;
  - для больших БД лучше репликация или logical replication, но дамп всё равно полезен для тестов.
- **Миграция MySQL → PostgreSQL**:
  - делаем `mysqldump` (строго структурированный),
  - прогоняем через инструменты/скрипты, переводящие типы/DDL,
  - заливаем в PostgreSQL.

### Практика / реальные сценарии

- Тестовые среды:
  - команда делает регулярные логические дампы части прод‑данных с маскировкой PII;
  - использует их для интеграционных тестов.
- Проверка схемы:
  - `pg_dump -s` (только схема) — удобная “истина” для сравнения с миграциями и документацией.

### Типичные ошибки

- Использование больших логических дампов **в час пик**, когда диск и так загружен.
- Хранение единственного логического дампа без физического бэкапа и без проверки восстановления.
- Ожидание, что логический дамп на сотни ГБ быстро восстановится.

### Что будет, если…

- …делать логический дамп без ограничения ресурсов?
  - Можно “убить” прод: выжрать I/O и CPU, получить тормоза и таймауты.
- …никогда не тестировать восстановление логического дампа?
  - Легко обнаружить в момент аварии, что дамп битый/неполный/несовместимый с новой версией.

### Проверь себя

1. Назови три ситуации, когда логический бэкап предпочтительнее физического.
<details><summary>Ответ</summary>
1. Миграция между версиями/СУБД. 2) Частичное восстановление (одной схемы/таблицы). 3) Создание тестовых сред с маскированием данных или изменённой схемой.
</details>

1. Почему логический бэкап хуже подходит как единственный инструмент аварийного восстановления больших БД?
<details><summary>Ответ</summary>
Потому что он медленно создаётся и особенно медленно восстанавливается, сильно грузит CPU/I/O, и восстановление может занимать часы/дни. Для больших БД обычно используют физические бэкапы + журнал, а логические — как вспомогательный механизм.
</details>

### Запомните

- Логические бэкапы — это “швейцарский нож” для миграций и частичных восстановлений.
- Для больших прод‑систем их нужно комбинировать с физическими бэкапами и WAL/binlog.
