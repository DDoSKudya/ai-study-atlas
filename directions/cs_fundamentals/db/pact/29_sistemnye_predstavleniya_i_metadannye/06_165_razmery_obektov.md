## 165. Размеры объектов

#### В этом разделе главное

- Зная только структуру и статистику использования, легко забыть о **физическом размере** таблиц и индексов.
- В PostgreSQL есть удобные функции (`pg_table_size`, `pg_indexes_size`, `pg_total_relation_size`, `pg_database_size`, `pg_size_pretty`), а в MySQL — поля `DATA_LENGTH`, `INDEX_LENGTH`.
- Размеры нужны для: **поиска «слонов»**, планирования ресурсов и оценки эффектов оптимизации.

#### Простыми словами

База данных — это не абстрактная схема, а **конкретные мегабайты и гигабайты на диске**.

- слишком большая таблица → медленные бэкапы, долгие сканы, проблемы с I/O;
- раздутые индексы → расход места и замедление записи;
- взрывной рост размера → риск внезапно упереться в дисковую квоту.

#### PostgreSQL: функции размеров

**`pg_table_size(regclass)`**

- Размер только таблицы (без индексов, с TOAST‑таблицей).

**`pg_indexes_size(regclass)`**

- Суммарный размер всех индексов таблицы.

**`pg_total_relation_size(regclass)`**

- Полный размер: таблица + индексы + TOAST.

**`pg_database_size(name)`**

- Размер всей базы.

**`pg_size_pretty(bigint)`**

- Преобразует размер в человекочитаемый формат (`kB`, `MB`, `GB`).

#### Примеры запросов

**Размеры всех таблиц схемы `public` по убыванию:**

```sql
SELECT
    schemaname,
    relname AS table_name,
    pg_size_pretty(pg_total_relation_size(relid)) AS total_size,
    pg_size_pretty(pg_table_size(relid))          AS table_size,
    pg_size_pretty(pg_indexes_size(relid))        AS indexes_size
FROM pg_catalog.pg_statio_user_tables
ORDER BY pg_total_relation_size(relid) DESC
LIMIT 50;
```

**Размер всей базы:**

```sql
SELECT
    pg_size_pretty(pg_database_size(current_database())) AS db_size;
```

**MySQL: размеры таблиц в базе `app_db`:**

```sql
SELECT
    table_name,
    table_rows,
    data_length,
    index_length,
    data_length + index_length                     AS total_bytes,
    ROUND((data_length + index_length) / 1024/1024, 2) AS total_mb
FROM INFORMATION_SCHEMA.TABLES
WHERE table_schema = 'app_db'
ORDER BY total_bytes DESC;
```

#### Картинка в голове

Представь, что у тебя есть **склад** (сервер БД):

- коробки‑таблицы;
- коробки‑индексы.

Функции размеров — это «весы и рулетка», которые:

- показывают, какая коробка сколько весит;
- помогают решить, что нужно перенести/разобрать/упаковать компактнее;
- позволяют заранее понять, когда склад забьётся.

#### Типичные ошибки

- «Главное — время запроса, размер не важен» — большие размеры влияют на время бэкапов, репликации, вакуума, аварийного восстановления.
- «Если таблица большая, значит, её надо сразу шардировать» — иногда достаточно архивирования старых данных или правильного партиционирования.
- «`pg_size_pretty` можно использовать в автоматизированных расчётах» — она возвращает строку, для машин лучше использовать «сырые» байты.

#### Как запомнить

- Комбо для PostgreSQL: `pg_table_size` (только таблица), `pg_indexes_size` (индексы), `pg_total_relation_size` (всё вместе) + `pg_size_pretty` для красоты.
- В MySQL — `DATA_LENGTH`, `INDEX_LENGTH` в `INFORMATION_SCHEMA.TABLES` + перевод в MB/GB руками.

#### Как учить, если считаешь себя «тупым»

1. **Представь конкретный жёсткий диск.**  
   Подумай не абстрактно, а так: «у нас диск на 100 ГБ, база занимает X ГБ, эта таблица — Y ГБ».  
   Тогда функции размеров оказываются не «про теорию», а про очень реальную вещь — «хватит ли места завтра».

2. **Сделай один отчёт по размерам.**

   - Запусти запрос по `pg_total_relation_size` (или `INFORMATION_SCHEMA.TABLES` в MySQL).
   - Отсортируй таблицы по размеру и выпиши себе топ‑5 **самых тяжёлых**.

3. **Спроси себя для каждой из этих таблиц: «почему она такая большая?»**

   - много строк?
   - большие JSON‑поля?
   - исторические данные, которые никто не читает?  
     Уже на этом этапе часто рождаются идеи: архивирование, партиционирование, чистка.

4. **Повтори упражнение через неделю/месяц.**  
   Сравни отчёты: какие таблицы выросли больше всего?  
   Так ты начнёшь **чувствовать динамику роста**, а не только статический размер.

5. **Не бойся цифр — бойся их отсутствия.**  
   Гораздо страшнее не знать, сколько весит база и что растёт, чем видеть большие числа.  
   Задача этого раздела — научить тебя получать эти числа быстро и просто.

#### Проверь себя

<details>
<summary>1. Зачем отдельно смотреть на размер индексов, если есть полный размер таблицы?</summary>

Потому что индексы:

- занимают значительную часть места;
- влияют на производительность записи (каждый `INSERT/UPDATE/DELETE` обновляет индексы).

Если индексы занимают в разы больше данных, чем сама таблица, это повод проверить, все ли они нужны и нет ли дублей.

</details>

<details>
<summary>2. Почему <code>pg_size_pretty</code> удобно для людей, но неудобно для программ?</summary>

Она возвращает **строку** вида `'123 MB'`, `'10 GB'`, а не чистое число в байтах.  
Глазами это читать удобно, но для автоматических расчётов (суммирование, сравнение) лучше использовать базовые функции размеров без форматирования.

</details>
