## 163. Метаданные схемы

#### В этом разделе главное

- Метаданные схемы позволяют **программно узнать, как выглядит БД**: какие таблицы, столбцы, индексы, ограничения, типы.
- Это фундамент для: генераторов кода, документации, проверки миграций, анализа качества схемы.
- Главное — **не смешивать уровень данных и уровень метаданных**: в запросах к метаданным тебе интересны названия и типы, а не содержимое таблиц.

#### Простыми словами

Когда ты открываешь таблицу в GUI и видишь список колонок/типов — инструмент просто делает запросы к системным представлениям.

Если ты научишься делать такие же запросы сам:

- тебе не нужны будут «магические» кнопки IDE;
- ты сможешь встроить эти запросы в свои скрипты, миграции, проверки и CI;
- документация и код будут **синхронизированы с реальной БД**, а не с устаревшими описаниями.

#### Практические паттерны запросов

**1. Список всех таблиц приложения**

PostgreSQL:

```sql
SELECT
    table_schema,
    table_name
FROM information_schema.tables
WHERE table_schema NOT IN ('information_schema', 'pg_catalog')
  AND table_type = 'BASE TABLE'
ORDER BY table_schema, table_name;
```

MySQL:

```sql
SELECT
    table_schema,
    table_name
FROM INFORMATION_SCHEMA.TABLES
WHERE table_schema = 'app_db'
  AND table_type = 'BASE TABLE'
ORDER BY table_name;
```

**2. Столбцы таблицы с ключевыми свойствами**

```sql
SELECT
    c.column_name,
    c.data_type,
    c.character_maximum_length,
    c.numeric_precision,
    c.numeric_scale,
    c.is_nullable,
    c.column_default
FROM information_schema.columns AS c
WHERE c.table_schema = 'public'
  AND c.table_name   = 'orders'
ORDER BY c.ordinal_position;
```

**3. Индексы таблицы (PostgreSQL, через `pg_indexes`)**

```sql
SELECT
    schemaname,
    tablename,
    indexname,
    indexdef
FROM pg_indexes
WHERE schemaname = 'public'
  AND tablename  = 'orders'
ORDER BY indexname;
```

**4. Проверка наличия PK и уникальных ограничений (переносимо через `information_schema`)**

```sql
SELECT
    tc.table_schema,
    tc.table_name,
    tc.constraint_type,
    kcu.column_name
FROM information_schema.table_constraints AS tc
JOIN information_schema.key_column_usage AS kcu
  ON tc.constraint_name = kcu.constraint_name
WHERE tc.table_schema = 'public'
  AND tc.table_name   = 'users'
  AND tc.constraint_type IN ('PRIMARY KEY','UNIQUE')
ORDER BY tc.constraint_type, kcu.ordinal_position;
```

#### Использование в генерации кода и документации

Имея эти запросы, ты можешь:

- **генерировать модели** (например, структуры в языке, ORM‑модели) по фактической схеме;
- **формировать HTML/Markdown‑документацию** по таблицам и полям;
- **проверять миграции**: сравнивать ожидаемую схему (из описания) с реальной (из метаданных).

Простой сценарий:

1. Скрипт достаёт из БД список таблиц и столбцов.
2. Формирует по ним Markdown‑таблицы (`table`, `column`, `type`, `nullable`, `default`).
3. Сохраняет в репозиторий как «актуальную схему» или сравнивает с предыдущей версией.

#### Пошаговый сценарий: как «прочитать» схему чужого проекта за вечер

Допустим, ты пришёл в новый проект, и тебе дали только:

- доступ к базе;
- минимальное слово, что «там есть таблицы про пользователей и заказы».

План:

1. **Собрать список всех таблиц.**

   - Выполни запрос «Список всех таблиц приложения» для PostgreSQL или MySQL (см. выше).
   - Скопируй результат (названия таблиц) себе в заметки.

2. **Для каждой важной таблицы собрать список колонок.**

   - Для `users`, `orders`, `payments` и т.п. запусти запрос «Столбцы таблицы с ключевыми свойствами».
   - По каждой таблице выпиши:
     - какие колонки относятся к «ключевым идентификаторам» (`id`, `user_id`, `order_id`);
     - какие — к деньгам/суммам;
     - какие — к статусам.

3. **Узнать, какие индексы уже есть.**

   - Для PostgreSQL используй `pg_indexes` (запрос №3).
   - Обрати внимание:
     - какие поля входят в индексы (например, `user_id`, `created_at`);
     - есть ли индексы на FK‑поля (`user_id`, `product_id`).
   - Это даст понимание: «по каким полям чаще всего ищут».

4. **Проверить, что PK и уникальные ограничения на месте.**

   - Запусти запрос №4 для таблиц `users` и `orders`.
   - Убедись, что:
     - у каждой таблицы есть PK;
     - уникальные ограничения соответствуют здравому смыслу (например, email уникален).

5. **Собрать мини‑документацию.**
   - По каждой из главных таблиц выпиши коротко:
     - о чём она (в терминах бизнеса);
     - какие у неё ключевые поля;
     - какие индексы есть.
   - Это уже будет лучше, чем большинство «официальных» описаний.

Повторив этот сценарий 2–3 раза на разных проектах, ты начнёшь видеть схемы БД **как знакомые карты местности**, а не как хаос.

#### Типы данных и mapping между СУБД

Когда ты смотришь на `information_schema.columns.data_type`, важно помнить две вещи:

1. Это **«обобщённый» тип** — СУБД пытается привести свои внутренние типы к более‑менее стандартному имени.
2. Разные СУБД хранят одни и те же понятия **по‑разному** (например, автоинкремент).

Мини‑табличка для ориентира (упрощённо):

| Понятие                   | PostgreSQL (тип/особенности)              | MySQL/MariaDB (тип/особенности)        | Что обычно видим в `data_type`                |
| ------------------------- | ----------------------------------------- | -------------------------------------- | --------------------------------------------- |
| Целое число (обычное)     | `integer` / `int4`                        | `INT` / `INTEGER`                      | `integer` / `int`                             |
| Большое целое             | `bigint` / `int8`                         | `BIGINT`                               | `bigint`                                      |
| Мелкое целое              | `smallint` / `int2`                       | `SMALLINT`                             | `smallint`                                    |
| Автоинкрементный ключ     | `serial` / `bigserial` (связь с sequence) | `INT AUTO_INCREMENT`                   | чаще всего просто `integer` / `bigint`        |
| Деньги/точные числа       | `numeric(p,s)` / `decimal(p,s)`           | `DECIMAL(p,s)`                         | `numeric` / `decimal`                         |
| Строка фикс./перем. длины | `char(n)` / `varchar(n)`                  | `CHAR(n)` / `VARCHAR(n)`               | `character` / `character varying` / `varchar` |
| Длинный текст             | `text`                                    | `TEXT`, `MEDIUMTEXT`, `LONGTEXT`       | `text`                                        |
| Булево                    | `boolean` (`true`/`false`)                | `TINYINT(1)` (часто 0/1) или `BOOLEAN` | `boolean` / иногда `tinyint`                  |
| Дата/время                | `timestamp [with/without time zone]`      | `DATETIME`, `TIMESTAMP`                | `timestamp` / `datetime`                      |
| JSON                      | `json`, `jsonb`                           | `JSON` (MySQL 5.7+)                    | `json`                                        |

Пара важных замечаний:

- В PostgreSQL `serial` и `bigserial` — это **синтаксический сахар**: создаётся обычный `integer`/`bigint` + sequence, а в `information_schema` ты увидишь просто `integer`/`bigint` и default вроде `nextval(...)`.
- В MySQL автоинкремент — это флаг на столбце (`AUTO_INCREMENT`), тип обычно `INT`/`BIGINT`, а в `information_schema` про автоинкремент можно увидеть в `EXTRA`/`COLUMN_DEFAULT`, а `data_type` будет просто `int`/`bigint`.
- Для булевых значений MySQL исторически использует `TINYINT(1)`, и в `data_type` ты можешь увидеть `tinyint`, хотя логически это «true/false».

Практический приём:

- когда ты видишь в метаданнах только `data_type`, но не понимаешь, как это выглядит физически, — смотри также на:
  - `column_type` (MySQL) — там часто хранится «сырой» тип с размерностью (`int(11)`, `varchar(255)`);
  - `column_default` — по нему можно понять, есть ли автоинкремент/sequence;
  - дополнительные поля (`numeric_precision`, `numeric_scale`, `character_maximum_length`).

Цель этого блока не в том, чтобы зазубрить все типы, а в том, чтобы у тебя в голове была **простая карта соответствий** и ты не пугался того, что одна и та же идея (например, «автоинкрементный PK») выглядит по‑разному в разных СУБД и в их метаданных.

#### Картинка в голове

Думай о метаданных схемы как о **карте местности**:

- таблицы — города;
- столбцы — улицы;
- индексы — скоростные магистрали;
- ограничения — мосты/ограждения.

Если у тебя есть живая карта (метаданные), ты можешь:

- строить маршруты (запросы) осознанно;
- видеть, где узкие места;
- автоматически подсвечивать проблемы (например, таблицы без PK).

#### Типичные ошибки

- «Схему можно хранить только в ORM‑моделях, метаданные БД не нужны» — тогда при ручных изменениях в БД ORM и реальность разъедутся.
- «Метаданные нужны только DBA» — разработчикам они не менее важны: для миграций, дебага, оптимизации.
- «Достаточно раз в год сделать диаграмму в Draw.io» — без автоматического получения схемы она устареет через неделю.

#### Как запомнить

- Любой вопрос вида «какие у нас есть X?» (таблицы, столбцы, индексы) чаще всего решается **одним запросом к метаданным**.
- Если ты открываешь GUI, чтобы «посмотреть, какие там колонки», — почти всегда это можно (и полезно) заменить SQL‑запросом к `information_schema` / `pg_catalog`.

#### Как учить, если считаешь себя «тупым»

1. **Выбери один конкретный сценарий.**  
   Например: «я хочу сгенерировать Markdown‑документацию по таблице `users`».

2. **Разбей задачу на микрошаги:**

   - запросить список колонок через `information_schema.columns`;
   - выписать из результата `column_name`, `data_type`, `is_nullable`, `column_default`;
   - руками/скриптом собрать табличку:
     | column | type | nullable | default |.

3. **Сделай этот сценарий до конца.**  
   Неважно, насколько «глупым» он кажется — важно, что ты **прошёл полный путь** от запроса к метаданным до полезного артефакта.

4. **Повтори с другой таблицей или БД.**  
   Тот же приём работает в любой схеме: API‑таблицы, аналитические, тестовые.  
   Каждый повтор укрепляет связь «метаданные → реальная польза».

5. **Сохрани результат в репозитории.**  
   Как только твой скрипт/документация окажутся рядом с кодом, ты начнёшь воспринимать метаданные как **часть нормального рабочего процесса**, а не «дополнительную сложность».

#### Проверь себя

<details>
<summary>1. Как получить список таблиц без первичного ключа в схеме <code>public</code> (идея, не обязательно точный код)?</summary>

Идея:

1. Взять все таблицы из `information_schema.tables` для `table_schema = 'public'` и `table_type = 'BASE TABLE'`.
2. Взять таблицы, у которых есть `PRIMARY KEY` в `information_schema.table_constraints`.
3. Вычесть вторые из первых (через `LEFT JOIN ... WHERE pk_table IS NULL`).

Так получишь список таблиц без PK.

</details>

<details>
<summary>2. Чем <code>pg_indexes</code> удобно отличается от низкоуровневой связки <code>pg_class</code> + <code>pg_index</code>?</summary>

`pg_indexes` уже собирает вместе:

- имя схемы;
- имя таблицы;
- имя индекса;
- текст определения индекса (`indexdef`).

То есть это более удобное «человекочитаемое» представление, тогда как `pg_class` + `pg_index` дают более низкоуровневые детали (OID, массивы `indkey` и т.п.).

</details>
