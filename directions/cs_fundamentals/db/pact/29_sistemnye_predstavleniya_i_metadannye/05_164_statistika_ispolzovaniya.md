## 164. Статистика использования

#### В этом разделе главное

- Одной структуры схемы мало — важно знать, **как объекты реально используются**.
- В PostgreSQL для этого есть `pg_stat_user_tables`, `pg_stat_user_indexes` и расширение `pg_stat_statements`.
- По статистике можно находить **горячие таблицы, неиспользуемые индексы и тяжёлые запросы**.

#### Простыми словами

Схема отвечает на вопрос «**что у нас есть**», а статистика — «**как это живёт в проде**».

Пример:

- таблица может быть «красивой», но если по ней каждый запрос сканирует миллионы строк — это видно в статистике, а не в схеме;
- индекс может существовать, но ни разу не использоваться — это тоже видно только по статистике.

#### PostgreSQL: ключевые представления

**`pg_stat_user_tables`**

- По каждой пользовательской таблице:
  - `n_tup_ins`, `n_tup_upd`, `n_tup_del` — сколько строк вставлено/обновлено/удалено;
  - `n_live_tup`, `n_dead_tup` — оценка живых и мёртвых строк;
  - `seq_scan`, `seq_tup_read` — сколько было последовательных сканирований и сколько строк прочитано;
  - `idx_scan`, `idx_tup_fetch` — сколько раз использовались индексы и сколько строк прочитали.

**`pg_stat_user_indexes`**

- По каждому индексу:
  - `idx_scan` — сколько раз по этому индексу обращались;
  - `idx_tup_read`, `idx_tup_fetch` — строки, прочитанные через индекс;
  - вместе с размером индекса (через `pg_indexes_size`) это даёт понимание «польза/стоимость».

**`pg_stat_statements`** (расширение)

- Агрегированная статистика по запросам:
  - `queryid`, `query` (нормализованный текст);
  - `calls` — сколько раз выполнялся;
  - `total_time`, `mean_time` — суммарное и среднее время;
  - `rows` — суммарное число возвращённых строк.

#### Примеры запросов

**Топ таблиц по числу последовательных сканов (возможно, не хватает индексов):**

```sql
SELECT
    relname,
    seq_scan,
    seq_tup_read,
    idx_scan
FROM pg_stat_user_tables
ORDER BY seq_scan DESC
LIMIT 20;
```

**Неиспользуемые или почти неиспользуемые индексы (кандидаты на удаление):**

```sql
SELECT
    s.schemaname,
    s.relname       AS table_name,
    s.indexrelname  AS index_name,
    s.idx_scan,
    pg_size_pretty(pg_relation_size(i.indexrelid)) AS index_size
FROM pg_stat_user_indexes AS s
JOIN pg_index AS i
  ON i.indexrelid = s.indexrelid
WHERE s.idx_scan < 50  -- порог подбери под свой проект
ORDER BY pg_relation_size(i.indexrelid) DESC;
```

**Топ самых «дорогих» запросов по суммарному времени (через `pg_stat_statements`):**

```sql
SELECT
    query,
    calls,
    total_time,
    mean_time,
    rows
FROM pg_stat_statements
ORDER BY total_time DESC
LIMIT 20;
```

> На проде часто обрезают/маскируют `query`, но `queryid` остаётся стабильным для анализа.

#### Пошаговый сценарий: как быстро понять, что нагружает базу

1. **Посмотреть на таблицы с большим количеством последовательных сканов.**

   - Выполни запрос про топ таблиц по `seq_scan`.
   - Для топ‑3 таблиц спроси себя:
     - ожидаешь ли ты, что по ним часто идут полные сканы;
     - нет ли там запросов, которые можно ускорить индексом.

2. **Найти индексы, которые «весят много, а работают мало».**

   - Запусти запрос про неиспользуемые индексы.
   - Обрати внимание на:
     - большие индексы (`index_size` в сотни МБ и больше);
     - очень маленький `idx_scan`.
   - Такие индексы — кандидаты на обсуждение: нужны ли они вообще.

3. **Определить несколько самых тяжёлых запросов.**

   - Включи/настрой `pg_stat_statements` (если ещё не включено).
   - Выполни запрос про «топ по `total_time`».
   - Для каждого запроса из топ‑5 посмотри:
     - `calls` — насколько он частый;
     - `mean_time` — насколько он медленный;
     - `rows` — сколько строк он возвращает.

4. **Сделать одну маленькую оптимизацию и проверить эффект.**

   - Выбери один конкретный запрос из топ‑5.
   - Попробуй:
     - добавить подходящий индекс;
     - или переписать запрос так, чтобы он читал меньше строк.
   - Через какое‑то время снова посмотри в `pg_stat_statements` — изменились ли `mean_time` и `total_time`.

5. **Зафиксировать выводы.**
   - Кратко запиши:
     - какие таблицы/индексы/запросы оказались проблемными;
     - что ты сделал и как это повлияло на статистику.
   - Эти заметки помогут в будущем быстрее вспоминать, как читать `pg_stat_*`.

#### Картинка в голове

Представь, что у каждой таблицы и индекса есть **«фитнес‑браслет»**, который считает:

- сколько шагов (запросов) они сделали;
- сколько «калорий» (строк) сожгли;
- сколько времени провели в активности.

По этим показателям можно понять:

- кто «перегрет» и требует оптимизации;
- кто «простаивает» и тратит место зря.

#### Типичные ошибки

- «Если индекс не используется за последние N часов, его можно сразу удалять» — осторожнее: есть редкие, но критичные сценарии (отчёты, end‑of‑month, бэкапы).
- «`n_dead_tup` большой — значит, надо немедленно VACUUM FULL» — сначала оцени влияние на прод и причины, часто достаточно обычного `VACUUM` + настройки автovacuum.
- «Достаточно один раз посмотреть на статистику» — она **накапливается и может сбрасываться**, важно понимать, за какой период ты смотришь.

#### Как запомнить

- Статистика отвечает на вопросы «как часто? насколько тяжело? какие топ‑N?».
- Минимум для PostgreSQL:
  - `pg_stat_user_tables` — понять, как живут таблицы;
  - `pg_stat_user_indexes` — понять пользу индексов;
  - `pg_stat_statements` — понять, какие запросы реально «едят» ресурсы.

#### Как учить, если считаешь себя «тупым»

1. **Сначала просто посмотри на цифры.**

   - Выполни по очереди запросы к `pg_stat_user_tables` и `pg_stat_user_indexes` из этого раздела.
   - Не пытайся всё интерпретировать, просто запомни: «у каждой таблицы и индекса есть счётчики».

2. **Сфокусируйся на одном вопросе.**  
   Например: «какие таблицы чаще всего читаются последовательным сканом?».

   - Смотри только на `seq_scan` и `seq_tup_read`.
   - Отметь себе топ‑3 таблицы и подумай, логично ли это для твоего проекта.

3. **Задай себе один конкретный «диагностический» вопрос.**  
   Например: «есть ли у нас индексы, которые почти никто не использует?».

   - Возьми готовый запрос про неиспользуемые индексы.
   - Посмотри, что он выдаёт, и обсуди с кем‑то из команды (или сам для себя), действительно ли эти индексы нужны.

4. **Повторяй небольшими порциями.**  
   Не нужно разом понимать все десятки полей в статистических представлениях.  
   Достаточно **выбирать 1–2 метрики за раз** и учиться их читать на реальных примерах.

5. **Связывай статистику с изменениями в коде.**  
   Сделал оптимизацию запроса — через некоторое время посмотри в `pg_stat_statements`, изменились ли `mean_time` и `total_time`.  
   Так статистика перестанет быть «табличкой цифр», а станет **обратной связью на твои действия**.

#### Проверь себя

<details>
<summary>1. Как по статистике понять, что индекс, возможно, не нужен?</summary>

Признаки:

- в `pg_stat_user_indexes.idx_scan` очень маленькое значение (или ноль) за разумный период;
- при этом размер индекса (через `pg_relation_size` / `pg_indexes_size`) заметен;
- нет редких, но важных отчётных задач, которые бы его использовали.

В таких случаях индекс может быть кандидатом на удаление или пересмотр.

</details>

<details>
<summary>2. Чем полезен <code>pg_stat_statements</code> по сравнению с логами запросов?</summary>

Логи дают **сырые данные** (каждый запрос отдельно), а `pg_stat_statements` — **агрегированные**:

- суммарное и среднее время;
- число вызовов;
- число возвращённых строк.

То есть не нужно вручную собирать статистику из логов — расширение делает это за тебя и даёт сразу «топ медленных/частых запросов».

</details>
