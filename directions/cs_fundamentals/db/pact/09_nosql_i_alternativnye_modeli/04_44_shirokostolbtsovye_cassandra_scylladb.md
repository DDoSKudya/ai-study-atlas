## §44. Широкостолбцовые (Cassandra, ScyllaDB)

### 44. Широкостолбцовые (Cassandra, ScyllaDB)

**Цель раздела.**  
Понять, что такое широкостолбцовая модель (на примере Cassandra/ScyllaDB): как устроены партиции и строки, как именно данные оказываются на разных узлах, что означают уровни консистентности (ONE, QUORUM, ALL и др.) и почему в таких БД **запросы диктуют схему**, а не наоборот.

**В этом разделе главное (три строки).**  
1) Широкостолбцовые БД хранят данные в виде **разреженных таблиц**: одна партиция (partition key) содержит много логически связанных строк, а набор столбцов может отличаться от строки к строке.  
2) Таблицы автоматически **распределяются по узлам кластера** по partition key; при чтении/записи клиент выбирает уровень консистентности (ONE, QUORUM, ALL…), который определяет баланс скорости и согласованности.  
3) Здесь **сначала проектируют запросы, потом схему**: CQL похож на SQL, но нет JOIN и произвольных WHERE; если запрос не подходит под схему — его просто нельзя эффективно выполнить.

---

#### Ещё проще: как «тетрадки по пользователям»

- Представь не одну гигантскую таблицу, а **много тетрадок**:
  - одна тетрадь — все события пользователя 1;
  - другая тетрадь — все события пользователя 2;
  - и так далее.
- Внутри каждой тетради страницы отсортированы по времени: новые события идут по порядку.
- Вопросы вида:
  - «покажи последние 100 событий пользователя 1» — **открыть тетрадь 1 и пролистать пару страниц** — быстро;
  - «покажи все события за вчера по всем пользователям» — **пролистать все тетради** — дорого.

Широкостолбцовая БД вроде Cassandra как раз и делает множество таких тетрадей (партиций) и умеет **очень быстро работать, если ты знаешь номер тетради (partition key)**.

#### Термины (расшифровка)

- **Partition key** — набор столбцов, по которому строка попадает в конкретную партицию и на определённые узлы кластера. Важно для распределения нагрузки.
- **Clustering columns** — столбцы, задающие **порядок строк внутри партиции** (по ним можно делать диапазонные запросы).
- **Партиция** — группа строк с одинаковым partition key. Внутри партиции строки хранятся упорядоченно по clustering columns.
- **Разреженная таблица (wide table)** — таблица, где у разных строк может быть разный набор столбцов; отсутствующие значения практически не занимают место.
- **CQL (Cassandra Query Language)** — SQL‑подобный язык запросов к Cassandra/ScyllaDB. Похож на SQL, но **жёстко ограничен**: нет полноценного JOIN, подзапросы сильно урезаны.
- **Consistency level** — настройка, которая говорит, **сколько реплик** должно подтвердить запись или чтение, чтобы операция считалась успешной (ONE, TWO, THREE, QUORUM, ALL, LOCAL_QUORUM и др.).

#### Мини‑история: логи игры на одном сервере и в кластере

Представь онлайн‑игру:

- миллионы игроков;
- каждый игрок каждую секунду посылает события (позиция, действия, покупки).

**Сначала** ты всё пишешь в одну таблицу в PostgreSQL:

```sql
CREATE TABLE game_events (
  player_id   uuid,
  ts          timestamptz,
  event_type  text,
  payload     jsonb
);
```

Пока игроков мало — работает. Потом:

- таблица растёт до сотен гигабайт;
- индексы по `player_id, ts` становятся очень тяжёлыми;
- один сервер и диск начинают «задыхаться».

**Переход на Cassandra**:

- создаёшь таблицу:

```sql
CREATE TABLE events_by_player (
  player_id   uuid,
  ts          timestamp,
  event_type  text,
  payload     text,
  PRIMARY KEY (player_id, ts)
);
```

- каждый `player_id` получает **свою партицию** (свою «тетрадку»);
- кластер Cassandra из многих узлов:
  - раскладывает партиции по узлам;
  - реплицирует их на несколько машин;
  - позволяет быстро писать/читать события по игроку в любом месте кластера.

Ключевой момент: ты **осознанно решил**, что главный запрос — «события по игроку за время», и спроектировал таблицу **под этот запрос**, а не наоборот.

#### Теория и правила (подробно)

- **Как устроена таблица в Cassandra.**

  На первый взгляд таблица в Cassandra выглядит как обычная таблица:

  ```sql
  CREATE TABLE events_by_user (
    user_id      uuid,
    event_time   timestamp,
    event_type   text,
    payload      text,
    PRIMARY KEY (user_id, event_time)
  );
  ```

  Но под капотом:

  - `user_id` — **partition key**: все события пользователя хранятся **в одной партиции**;
  - `event_time` — **clustering column**: внутри партиции события упорядочены по времени.

  То есть таблица на самом деле — это **множество больших «строк‑групп» (партиций)**, каждая из которых хранит историю событий по одному ключу (одну «тетрадь»).

- **Распределение данных по узлам.**

  - Кластер Cassandra состоит из многих узлов.
  - Partition key через хеш‑функцию преобразуется в токен, который попадает в определённый **диапазон токенов**, закреплённый за узлом.
  - Чтобы обеспечить отказоустойчивость, данные партиции реплицируются на несколько узлов (фактор репликации, например `RF=3`).

  В итоге: **одна партиция хранится на нескольких узлах**, но для клиента это выглядит как одна логическая таблица. Ты как будто открываешь «тетрадь игрока», а не думаешь, на каких полках и серверах она физически лежит.

- **Чтение и запись с уровнями консистентности.**

  Допустим, у нас `RF=3` — каждая партиция хранится на трёх узлах. Тогда:

  - `CONSISTENCY ONE` при записи означает: **достаточно, чтобы один из трёх узлов подтвердил запись**, остальные догонят асинхронно.
  - `CONSISTENCY QUORUM` означает: нужно подтверждение от **двух из трёх** (большинство).
  - `CONSISTENCY ALL` — только когда **все три** подтвердили, операция считается успешной.

  При чтении аналогично: `ONE` — достаточно одного узла, `QUORUM` — большинства, `ALL` — всех.

  Интуитивно:

  - `ONE` — **максимальная скорость и доступность**, но возможны чуть устаревшие данные;
  - `QUORUM` — баланс: при правильной настройке чтение `QUORUM` + запись `QUORUM` дают «почти как одна БД» по консистентности;
  - `ALL` — строго, но медленно и хрупко (если один из узлов недоступен, операция ломается).

- **Почему сначала запросы, потом схема.**

  Cassandra **не предназначена** для произвольных запросов типа:

  ```sql
  SELECT * FROM events_by_user
  WHERE event_type = 'click'
    AND event_time > '2025-01-01';
  ```

  если **ключом партиции** является только `user_id`. Такая выборка потребовала бы прохода по всем партициям — это дорого и в общем случае **запрещено/неэффективно**.

  Поэтому:

  1. Сначала ты описываешь, **какие именно запросы будут чаще всего**: по какому ключу, по какому диапазону, в каком порядке.
  2. Затем под каждый важный запрос **проектируешь таблицу**: выбираешь partition key и clustering columns так, чтобы запрос ложился на одну партицию (или минимум партиций).
  3. Если запросы сильно отличаются, создаёшь **несколько таблиц с денормализованными данными** под разные шаблоны запросов.

  Это противоположность классическому подходу SQL, где сначала нормализуют модель, а потом думают о запросах.

- **Типичные сценарии использования.**

  - Логи и события (клик‑стрим, телеметрия): быстро пишем миллионы событий, читаем по пользователю/устройству/времени.
  - Временные ряды с огромным объёмом (но не критичные по транзакционной логике).
  - Хранилище исторических данных, которые редко меняются, но должны быть доступны для чтения.

#### Микро‑диалог в голове про «сначала запрос, потом схема»

**Вопрос к себе 1.** «Почему нельзя просто взять аккуратную нормализованную схему и развернуть её в Cassandra?»  
**Ответ.** Потому что Cassandra умеет быстро отвечать только на запросы, которые **используют partition key и уважают порядок clustering columns**. Всё остальное требует обхода множества партиций по кластеру и либо будет запрещено, либо будет работать очень плохо. Поэтому здесь главная мысль: «какой конкретно запрос я хочу выполнять?» — и только потом: «как под него сделать таблицу».

**Вопрос к себе 2.** «Получается, мне придётся дублировать данные в нескольких таблицах?»  
**Ответ.** Да, и это нормально. В Cassandra **денормализация — это норма**: под разные частые запросы делают разные таблицы с разным partition key и clustering. Цена — больше места на диске, выигрыш — быстрые запросы без JOIN.

**Вопрос к себе 3.** «Не вырастет ли хаос из‑за дублирования?»  
**Ответ.** Может, если не думать. Поэтому:
  - обновления данных, которые размазаны по нескольким таблицам, стараются делать **в одном участке кода**;
  - используют уровни консистентности так, чтобы основные читатели видели «согласованную» картину;
  - выбирают Cassandra для задач, где **строгие инварианты не критичны** (логи, телеметрия, события), а не для учёта денег.

#### Пошагово: как спроектировать таблицу в Cassandra

1. **Определи главный запрос.**  
   Например: «хочу быстро получать последние события для конкретного пользователя за последний месяц».

2. **Выбери partition key.**  
   В нашем примере — `user_id`. Все события пользователя будут лежать в одной партиции.

3. **Выбери clustering columns и порядок.**  
   Пусть это будет `event_time DESC`, чтобы «последние события» были в начале партиции, и запросы `LIMIT 100` были быстрыми.

4. **Проверь размер партиции.**  
   Не должен быть гигантским (миллионы строк в одной партиции могут быть проблемой). Если данных очень много, добавь в partition key что‑то ещё (например, `month`), чтобы разделить по месяцам.

5. **Спроектируй вторичные таблицы, если нужны другие запросы.**  
   Например, если нужно искать события по типу `event_type`, придётся создать отдельную таблицу `events_by_type`, денормализовав данные.

#### Типичная ошибка

Пытаться использовать Cassandra **как обычную реляционную БД**:

- нормализовать схему как в SQL;
- потом писать произвольные запросы с фильтрацией по любым полям;
- ожидать, что БД «как‑нибудь оптимизирует».

Результат — медленные запросы, ошибки на стадии компиляции запросов и непредсказуемая нагрузка. В широкостолбцовых БД **так не делают**: здесь проектируют схему **строго под сценарии запросов**.

Ещё одна частая ошибка — делать **слишком большие партиции**:

- если в одной партиции миллионы строк (например, все события за годы для одного ключа), операции с ней становятся тяжёлыми;
- бэкапы и восстановление тоже усложняются.

Поэтому обычно партиции «режут»:

- добавляя в partition key ещё измерение (например, `user_id + месяц`);
- или организуя данные так, чтобы каждая партиция соответствовала разумному объёму (день, неделя, устройство и т.п.).

#### Простыми словами

Широкостолбцовая БД — это как **огромный шкаф с ящиками**, где:

- каждый ящик — это **партиция** (например, «все события пользователя 123»);
- внутри ящика лежат **карточки (строки)**, аккуратно отсортированные по времени или другому признаку;
- если ты знаешь номер ящика и примерное место карточки — достать данные очень быстро;
- но ты не можешь сказать: «найди мне все карточки по всему шкафу, где на полях написано `event_type = 'click'`» — это уже не тот инструмент.

#### Картинка в голове

Представь, что обычная SQL‑таблица — это **таблица в Excel**: все строки и колонки на одном листе, можно фильтровать по любому столбцу. Cassandra — это **огромный набор тетрадей**: одна тетрадь на пользователя, а в ней страницы — события по времени. Если хочешь найти события по пользователю и дате — открываешь одну тетрадь и листаешь в нужное место. Если хочешь найти события по какому‑то другому признаку — придётся завести отдельную тетрадь с другим порядком.

#### Проверь себя

1. Что такое partition key и зачем он нужен в Cassandra?  
<details><summary>Ответ</summary>Partition key определяет, к какой партиции относится строка и на какие узлы кластера она попадёт. Все строки с одинаковым partition key хранятся вместе; от выбора ключа зависит распределение нагрузки по узлам и возможность быстро выполнять запросы по этому ключу.</details>

2. Почему в Cassandra нельзя писать произвольные запросы, как в классическом SQL?  
<details><summary>Ответ</summary>Потому что данные распределены по узлам по partition key, и запросы, не использующие этот ключ (или нарушающие порядок clustering columns), требуют сканирования множества партиций по всему кластеру. Это дорого и в общем случае запрещено; схема должна быть спроектирована под конкретные шаблоны запросов.</details>

3. Какие типичные задачи хорошо решаются на Cassandra/ScyllaDB?  
<details><summary>Ответ</summary>Запись и хранение очень больших объёмов событий и логов, телеметрии, временных рядов, когда основной сценарий — «быстро писать новую строку и быстро читать по ключу (пользователь/устройство/диапазон времени)», а сложные транзакции и JOIN не нужны.</details>

#### Продвинуто: SSTable, memtable и compaction

**Как Cassandra хранит данные внутри.**

- Когда ты делаешь `INSERT`/`UPDATE`, запись попадает сначала в:
  - **коммит‑лог** (на диск, для надёжности);
  - **memtable** (в память, отсортированная по ключам структура).
- Когда memtable заполняется, он сбрасывается на диск в виде **SSTable (Sorted String Table)**.
- SSTable **не изменяются**: только читаются. Новые изменения пишутся в новые SSTable.

Со временем:

- одна и та же строка/ячейка может быть разбросана по нескольким SSTable (старые версии, новые версии, удаления);
- чтобы чтение не бегало по куче файлов, Cassandra периодически делает **compaction** — слияние SSTable:
  - объединяет несколько таблиц в одну;
  - выкидывает старые версии данных и tombstones (см. ниже);
  - уплотняет данные на диске.

**Зачем тебе это знать:**

- большое количество SSTable на узле ухудшает **latency чтения** (надо заглядывать в большее количество файлов);
- агрессивная compaction улучшает чтение, но может сильно грузить диск/CPU;
- при проектировании нагрузки важно понимать, что **массовые записи/удаления** приведут к волнам compaction.

---

#### Продвинуто: tombstones и почему про них говорят

В Cassandra удаление — это не немедленное стирание строки/ячейки, а запись **tombstone**:

- tombstone — это «могилка» со временем удаления;
- при чтении движок видит и данные, и tombstone и понимает, что эти данные **больше не актуальны**;
- физически данные убираются при последующих compaction’ах, когда tombstone «пережил» свой grace period.

**Проблемы от избытка tombstones:**

- если много строк часто помечаются и удаляются, но compaction/настройки не позволяют быстро их убирать:
  - чтения начинают «шерстить» кучу tombstones;
  - возрастает нагрузка на диск и задержка запросов.
- особенно болезненно при запросах, которые читают большие диапазоны (wide‑rows).

**Рекомендации:**

- не устраивать «частую массовую зачистку» по большим диапазонам;
- думать о шаблонах использования (например, TTL для «старых» данных, периодический drop/создание табличек по времени);
- следить за метриками compaction и количеством tombstones (есть специальные метрики в Cassandra).

---

#### Продвинуто: ScyllaDB как «быстрая Cassandra»

**ScyllaDB** — это реализация протокола Cassandra:

- совместима на уровне **CQL и модели данных**;
- переписана на C++ (и использует фреймворк Seastar);
- делает упор на:
  - многопоточность и асинхронность;
  - эффективное использование CPU и NUMA;
  - низкие задержки при больших нагрузках.

Если упростить:

- Cassandra исторически сильно завязана на JVM и может иметь overhead по GC;
- ScyllaDB избавляется от этого слоя, пытаясь выжать максимум из «железа».

Для тебя как пользователя:

- модель данных и «сначала запрос, потом схема» — те же;
- многие best practices по партициям, compaction и tombstones совпадают;
- ScyllaDB часто рассматривают, когда нужны **минимальные задержки и высокая пропускная способность**, но при этом устраивает Cassandra‑совместимая модель.

---

#### Запомните

- В широкостолбцовых БД **запросы важнее схемы**: сначала описывают запрос, потом под него проектируют таблицу (и иногда несколько таблиц).
- Внутри Cassandra данные живут в **SSTable** и периодически «убираются» через **compaction**; удаление — это tombstones, а не немедленное стирание.
- ScyllaDB — это Cassandra‑совместимая реализация с упором на производительность, но с теми же ключевыми идеями по модели данных.

#### Если прочитал и не понял

Коротко: Cassandra/ScyllaDB — это БД, где данные по ключу хранятся в большой «тетради‑партиции», а физически — в наборе файлов SSTable на диске. Новые записи и удаления просто добавляются сверху, а периодические compaction‑процессы наводят порядок и убирают старое. Запросы быстрые, когда ты приходишь с правильным ключом и читаешь «локально»; если идти без ключа или делать много массовых удалений, кладёшь лишнюю нагрузку на кластер.
