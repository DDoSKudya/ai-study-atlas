## §50. Документные БД (MongoDB и др.)

### 50. Документные БД (MongoDB и др.)

**Цель раздела.**  
Понять, как устроены документные базы данных (на примере MongoDB): что такое документ и коллекция, как работать с вложенными структурами и массивами, как устроен агрегирующий pipeline и индексы, что такое read/write concern, replica set и sharding, и в каких задачах документная модель особенно удобна.

**В этом разделе главное (три строки).**  
1) Документная БД хранит данные в виде документов (обычно JSON/BSON) с вложенными объектами и массивами; схема гибкая, но есть ограничения по размеру и структуре.  
2) MongoDB даёт мощный **агрегирующий pipeline** (`$match`, `$group`, `$project`, `$lookup`, `$unwind`, `$sort`, `$limit` и др.) и много типов индексов (составные, мультиключевые, TTL, гео и др.).  
3) Для отказоустойчивости и масштаба используются **replica set** и **sharding**, а консистентность и надёжность контролируются через **read/write concern**.

---

#### Ещё проще: «как в коде» и «как в БД»

- В обычной SQL‑БД ты часто делаешь так:
  - в коде у тебя есть **объект** `User` с полями `name`, `email`, массивом `phones`, списком `addresses`;
  - в БД это раскладывается на **несколько таблиц**: `users`, `phones`, `addresses`, с внешними ключами и JOIN.
- В документной БД:
  - **весь этот объект** можно хранить одним документом, очень похожим на JSON в коде;
  - читать/писать чаще всего будешь **целый документ**, без JOIN;
  - запросы выглядят как «поиск по полям документа», а не как «сборка из трёх таблиц».

На уровне интуиции документная БД — это как если бы ты мог **сохранить объект из кода почти как есть** и потом по нему нормально искать.

#### Термины (расшифровка)

- **Документ** — единица хранения в документной БД, чаще всего JSON‑подобный объект (в MongoDB — BSON). Может содержать вложенные объекты и массивы. Пример: профиль пользователя с адресами и списком заказов.
- **Коллекция** — набор документов, приблизительный аналог таблицы в реляционной БД. Схема не жёстко фиксирована: разные документы в коллекции могут иметь разные поля.
- **Dot notation** — обращение к вложенным полям по пути через точку: `address.city`, `items.0.price`, `profile.contacts.email`.
- **Агрегирующий pipeline** — последовательность стадий обработки данных, через которые «пропускаются» документы (`$match`, `$group`, `$project`, `$lookup`, `$unwind`, `$sort` и др.).
- **Мультиключевой индекс** — индекс по полю‑массиву: каждая запись массива участвует в индексе как отдельное значение.
- **Replica set** — группа MongoDB‑серверов (узлов), где один — **primary**, а остальные — **secondary**; данные реплицируются, при отказе primary выбирается новый.
- **Sharding** — горизонтальное разбиение данных по узлам (шардам) на основе **shard key**; используется для масштабирования по объёму и нагрузке.

#### Мини‑история: профиль пользователя в SQL и в документной БД

Допустим, у тебя есть приложение с пользователями. У каждого:

- имя, email;
- список телефонов;
- несколько адресов (домашний, рабочий);
- настройки (тема, язык, уведомления).

**В реляционной БД**:

- таблица `users` — базовая информация;
- таблица `phones` — один пользователь → много телефонов;
- таблица `addresses` — один пользователь → много адресов;
- таблица `settings` или JSON‑колонка в `users`.

Чтобы собрать **полный профиль**, тебе нужен SQL‑запрос с JOIN:

```sql
SELECT *
FROM users u
LEFT JOIN phones p ON p.user_id = u.id
LEFT JOIN addresses a ON a.user_id = u.id
LEFT JOIN settings s ON s.user_id = u.id
WHERE u.id = 42;
```

**В документной БД** ты можешь сделать **одну коллекцию `users`**, а документ будет выглядеть так:

```json
{
  "_id": ObjectId("..."),
  "name": "Иван",
  "email": "ivan@example.com",
  "phones": ["+7-900-123-45-67", "+7-999-111-22-33"],
  "addresses": [
    { "type": "home", "city": "Москва", "street": "Тверская", "house": "1" },
    { "type": "work", "city": "Москва", "street": "Ленинградский проспект", "house": "10" }
  ],
  "settings": {
    "theme": "dark",
    "language": "ru",
    "notifications": true
  }
}
```

Чтобы получить профиль, тебе достаточно:

```js
db.users.findOne({ _id: ObjectId("...") })
```

Никаких JOIN. Документ уже содержит всё, что тебе нужно показать на странице профиля.

#### Теория и правила (подробно)

- **Документ vs «строка в таблице».**  
  В реляционной БД сложный объект (например, пользователь + список адресов + список телефонов) часто раскладывается по нескольким таблицам (`users`, `addresses`, `phones`) и связывается через внешние ключи. В документной БД ты можешь **хранить всё это в одном документе**:

  ```json
  {
    "_id": ObjectId("..."),
    "name": "Иван",
    "email": "ivan@example.com",
    "addresses": [
      { "type": "home", "city": "Москва", "street": "Тверская", "house": "1" },
      { "type": "work", "city": "Москва", "street": "Ленинградский проспект", "house": "10" }
    ],
    "phones": ["+7-900-123-45-67", "+7-999-111-22-33"]
  }
  ```

  Это удобно, когда **большая часть операций читает/записывает «весь объект целиком»**.

- **Гибкая схема и её цена.**  
  В коллекции MongoDB разные документы могут иметь разные наборы полей — это удобно, когда структура эволюционирует. Но:
  - валидировать данные приходится на стороне приложения или через **валидацию схемы в самой БД**;
  - если сильно «разъезжаются» структуры, становится сложно писать запросы и поддерживать индексы.

- **Как «гибкая схема» сочетается с порядком.**  
  В реальных проектах обычно:
  - не дают схеме разрастаться хаотично «кто как написал»;
  - описывают **ожидаемую структуру документа** в документации или через JSON‑схемы;
  - на уровне БД включают **валидацию схемы**, чтобы отбрасывать «совсем левый» мусор.
  То есть гибкость — это не «делай что хочешь», а возможность **менять форму документов без жёстких миграций**, когда ты это изменения контролируешь.

- **Запросы и dot notation.**

  Примеры простых запросов в MongoDB:

  ```js
  // Найти всех пользователей из Москвы
  db.users.find({ "addresses.city": "Москва" })

  // Найти пользователей, у которых есть телефон с определённым кодом
  db.users.find({ "phones": /^\\+7-900/ })

  // Проекция: вернуть только имя и email
  db.users.find(
    { "addresses.city": "Москва" },
    { name: 1, email: 1, _id: 0 }
  )
  ```

- **Агрегирующий pipeline (очень кратко, пример).**

  Представим, что у нас есть коллекция `orders`:

  ```json
  {
    "_id": ObjectId("..."),
    "userId": ObjectId("..."),
    "items": [
      { "productId": 1, "price": 1000, "qty": 2 },
      { "productId": 2, "price": 500, "qty": 1 }
    ],
    "status": "paid",
    "createdAt": ISODate("2025-01-01T10:00:00Z")
  }
  ```

  Посчитаем выручку по пользователям:

  ```js
  db.orders.aggregate([
    { $match: { status: "paid" } },                         // фильтруем только оплаченные
    { $unwind: "$items" },                                  // разворачиваем массив позиций
    { $group: {
        _id: "$userId",
        revenue: { $sum: { $multiply: ["$items.price", "$items.qty"] } },
        ordersCount: { $sum: 1 }
      }
    },
    { $sort: { revenue: -1 } }                              // сортируем по выручке
  ])
  ```

  Это напоминает `SELECT ... FROM ... JOIN ... GROUP BY` в SQL, но выражено через последовательность шагов.

- **Индексы.**

  MongoDB поддерживает:

  - **одиночные** индексы: по одному полю;
  - **составные** индексы: по нескольким полям (`{ "userId": 1, "createdAt": -1 }`);
  - **мультиключевые** индексы по массивам (`"tags": 1`, `"items.productId": 1`);
  - **TTL‑индексы** (time‑to‑live) — документ автоматически удаляется по истечении времени;
  - **гео‑индексы** (2d, 2dsphere) для геозапросов;
  - **полнотекстовые** индексы.

  Как и в SQL‑БД, грамотно подобранные индексы критичны для производительности.

- **Консистентность: read concern и write concern.**

  В MongoDB можно управлять тем, **насколько надёжной** должна быть запись и насколько «свежие» данные мы хотим читать:

  - **write concern**:  
    - `w: 1` — запись считается успешной после подтверждения от primary;  
    - `w: "majority"` — запись считается успешной после того, как её применило большинство узлов replica set;  
    - можно включать/выключать журналирование (journal).
  - **read concern**:  
    - `local`, `available`, `majority`, `linearizable`, `snapshot` — разные уровни того, насколько «актуальными» и «устойчивыми» должны быть читаемые данные.

- **Replica set и чтение из реплик.**

  В типичной конфигурации:

  - есть один **primary**, куда идут все записи;
  - есть несколько **secondary**, куда реплицируются данные;
  - при отказе primary реплики проводят **выборы** и назначают нового primary.

  Клиент может выбирать **предпочтения чтения (read preference)**:

  - только с primary (строгая консистентность, выше нагрузка на primary);
  - с ближайшей реплики (лучше задержка, но возможны слегка устаревшие данные).

- **Sharding и выбор shard key.**

  Для масштабирования больших коллекций по нескольким узлам используется **sharding**:

  - **shard key** — поле или набор полей, по которому данные распределяются по шардам;
  - данные режутся на **chunks**, которые балансировщик перераспределяет между узлами;
  - запросы, которые фильтруют по shard key, могут обслуживаться **одним шардом**, остальные — всеми.

  Выбор shard key критичен:

  - слишком «горячий» ключ → один шард будет перегружен;
  - плохая кардинальность → неравномерное распределение и узкие места.

#### Микро‑диалог в голове про документы

**Вопрос к себе 1.** «Если в одном документе лежит всё — не опасно ли это? Он же может раздуться до гигантского размера.»  
**Ответ.** Опасно, если не думать. В MongoDB есть **жёсткое ограничение размера документа (16 МБ)**. Обычно документы для «профиля пользователя» и похожих сущностей занимают **килобайты**, максимум сотни килобайт. Если документ начинает раздуваться до мегабайт (например, туда пишут всё подряд, в том числе историю действий) — это сигнал, что часть данных лучше вынести в отдельную коллекцию.

**Вопрос к себе 2.** «Если схема гибкая, не будет ли бардака?»  
**Ответ.** Будет, если позволить всем писать документы «как попало». Поэтому:
  - на уровне команды договариваются о **формате документов**;
  - используют **валидацию схемы в БД** и/или типы в коде;
  - периодически проверяют реальные документы на соответствие ожиданиям.

**Вопрос к себе 3.** «Документная БД заменяет SQL‑таблицы?»  
**Ответ.** Нет. Там, где нужны **сложные связи и транзакции**, чаще всего удобнее реляционная модель. Документная модель — это инструмент, который **упрощает жизнь в задачах, где основная единица работы — один сложный объект**, и его удобно держать «как JSON».

#### Пошагово: когда документная БД удобнее

1. **Определи основной объект.**  
   Есть ли сущность, которую почти всегда читают/пишут целиком? Например: «профиль пользователя», «заказ с позициями», «страница блога с комментариями».

2. **Посмотри на вложенность.**  
   Есть ли у этой сущности естественно вложенные части (адреса, телефоны, элементы заказа), которые **логично хранить вместе**, а не размазывать по многим таблицам?

3. **Оцени размеры.**  
   Не превышает ли типичный документ ограничений (для MongoDB — 16 МБ)? Не будет ли один документ раздуваться до десятков мегабайт?

4. **Подумай о запросах.**  
   Большинство чтений — это «вытащить один документ по ключу» или «простая фильтрация по полям»? Сложных JOIN по нескольким сущностям почти нет?

5. **Реши вопрос с транзакциями.**  
   Нужны ли настоящие многодокументные транзакции? Если грань согласованности — один документ, а остальные действия можно сделать eventually, документная БД подходит.

#### Часто путают / типичные ошибки мышления

- **«Раз документная БД позволяет всё, можно ничего не проектировать».**  
  На практике это выливается в «зоопарк» из документов с разными полями и типами. Потом тяжело писать запросы и отчёты. Решение — **проектировать структуру документов так же внимательно**, как ты проектируешь таблицы.

- **«MongoDB — это просто JSON‑хранилище, положу туда всё подряд».**  
  На самом деле MongoDB — серьёзная БД: у неё есть индексы, репликация, транзакции, шардирование. Если относиться к ней как к «складированию JSON», легко получить медленную и трудно управляемую систему.

- **«Документ = одна строка в SQL, значит надо держать документы крошечными».**  
  Документ может быть **ощутимо больше строки**: он спокойно хранит массивы и вложенные объекты. Важна не «маленькость», а **разумный размер** (килобайты/сотни килобайт, но не мегабайты) и то, что содержимое документа логично относится к одной сущности.

#### Типичная ошибка

Хранить **огромные, бесконтрольные документы**, в которые постепенно добавляют всё новые и новые поля и вложенные структуры («монодокумент» на несколько мегабайт). Это:

- ухудшает производительность записи и чтения;
- повышает риск конфликтов при одновременных обновлениях;
- усложняет миграции схемы и индексацию.

Правильный путь — **проектировать документы осознанно**, разбивая их там, где граница согласованности и размер подсказывают, что сущности лучше разделить.

#### Простыми словами

Документная БД — это как **папка с отдельными файлами‑документами**, в каждом из которых лежит всё, что касается определённого объекта: профиль пользователя, заказ, статья. Ты не обязан заранее строго описывать содержание каждого файла, но если будешь складывать туда «всё подряд», то быстро перестанешь понимать, что где лежит. MongoDB добавляет к этому мощный **конвейер обработки** (pipeline), который позволяет фильтровать, группировать и склеивать документы почти как в SQL‑запросах.

#### Картинка в голове

Представь, что в реляционной БД информация о заказе разложена на три полки: «заказы», «строки заказа», «товары». Чтобы увидеть полный заказ, надо собрать его с трёх полок. В документной БД у тебя **одна папка «заказ»**, и в ней сразу всё: данные заказа, товары, суммы. Когда ты открываешь папку — ты видишь **целостную картинку** без дополнительных шагов.

#### Примеры запросов (MongoDB)

```js
// 1. Найти все заказы пользователя с суммой больше 1000
db.orders.find(
  {
    userId: ObjectId("..."),
    totalAmount: { $gt: 1000 }
  },
  { _id: 1, totalAmount: 1, createdAt: 1 }
)

// 2. Посчитать суммарную выручку по дням
db.orders.aggregate([
  { $match: { status: "paid" } },
  {
    $group: {
      _id: { $dateToString: { format: "%Y-%m-%d", date: "$createdAt" } },
      revenue: { $sum: "$totalAmount" },
      count: { $sum: 1 }
    }
  },
  { $sort: { _id: 1 } }
])

// 3. Присоединить данные о пользователе к заказам (аналог JOIN)
db.orders.aggregate([
  { $match: { status: "paid" } },
  {
    $lookup: {
      from: "users",
      localField: "userId",
      foreignField: "_id",
      as: "user"
    }
  },
  { $unwind: "$user" },
  { $project: { _id: 1, "user.name": 1, totalAmount: 1 } }
])
```

#### Продвинуто: выбор shard key и антипаттерны

**Задача shard key.**  
Shard key определяет, **как коллекция будет разрезана по шардам**. От него зависит:

- равномерность распределения данных и нагрузки;
- возможность обслуживать запросы **одним шардом** (астрономическая разница по производительности);
- насколько сложно будет делать миграции и расширять кластер.

**Основные антипаттерны:**

- **Монотонно растущий ключ (например, `_id` по умолчанию или `createdAt`).**
  - Все новые записи летят в один и тот же диапазон значений.
  - Если шардирование по диапазону (`range sharding`), один шард становится «горячим».
  - Даже при hashed‑шардинге надо думать, как ключ реально распределяется.

- **Низкая кардинальность (мало возможных значений).**
  - Shard key вида `country` или `status` распределяет данные по нескольким корзинам, но их всё равно мало.
  - Один шард может оказаться с большинством данных, остальные — полупустыми.

- **Shard key, по которому почти не фильтруют.**
  - Если большинство запросов не использует shard key в фильтрах, они превращаются в **scatter‑gather**: запрос рассылается на все шарды и потом агрегируется.
  - Это убивает масштабируемость: каждый запрос трогает весь кластер.

**Хороший shard key обычно:**

- имеет **высокую кардинальность** (много разных значений);
- хорошо распределяет данные (часто используют `hashed` вариант);
- входит в большинство важных запросов в фильтрах/джойнах;
- иногда составной: например, `{ userId: 1, createdAt: 1 }` или `{ tenantId: 1, orderId: 1 }`.

**Примеры:**

- Логи по пользователям: shard key `userId` (или `hashed userId`) — логично, потому что:
  - большинство запросов — «логи по пользователю»;
  - нагрузка распределяется по пользователям.
- Логи по микросервисам: shard key `{ serviceName, timestamp }`:
  - позволяет читать/агрегировать по сервису и времени;
  - timestamp может идти как clustering key внутри шарда.

**Практический чек перед выбором shard key:**

1. Выпиши **TOP‑3 реальных запросов** к коллекции.
2. Попробуй ответить на каждый из них, используя **только shard key (и, возможно, дополнительные поля индекса)**.
3. Если запрос почти всегда требует ходить на все шарды — shard key выбран плохо.

#### Продвинуто: change streams (отслеживание изменений)

**Что это такое.**  
`Change streams` в MongoDB позволяют **подписаться на поток изменений** в коллекции/БД/кластере:

- ты получаешь события «insert/update/delete» в реальном времени;
- не нужно вручную читать oplog и разбирать внутренний формат репликации;
- удобно для:
  - синхронизации с другими системами (CDC — change data capture);
  - обновления кэшей или поисковых индексов;
  - реактивных приложений (websocket‑уведомления и т.п.).

**Пример (упрощённо, на Node.js):**

```js
const pipeline = [
  { $match: { "operationType": { $in: ["insert", "update", "replace"] } } }
];

const changeStream = db.collection("orders").watch(pipeline);

changeStream.on("change", (change) => {
  console.log("Изменение в заказах:", change);
  // здесь можно:
  // - обновить кэш
  // - отправить событие в Kafka
  // - обновить индекс в Elasticsearch
});
```

**Типичный сценарий:**

- Есть основная коллекция `orders` в MongoDB.
- Отдельный сервис слушает change stream и:
  - складывает изменения в Kafka;
  - индексирует заказы в Elasticsearch;
  - обновляет денормализованные проекции.

Так MongoDB превращается в **источник событий**, а не только в хранилище.

#### Продвинуто: CouchDB, Couchbase, RavenDB (чем отличаются от MongoDB)

Все эти системы тоже документные, но с разным акцентом:

- **CouchDB**
  - Использует **multi‑master‑репликацию**: несколько узлов могут быть мастерами, принимать запись локально.
  - Репликация происходит по HTTP, часто с использованием **MVCC** (многоверсионности).
  - Конфликты **ожидаемы**: если два узла независимо изменили один и тот же документ, при репликации появляются две версии.
  - Задача приложения — **разруливать конфликты** (выбрать «победителя» или как‑то объединить).
  - Хорошо подходит для сценариев с периодически подключающимися устройствами (офлайн‑приложения, мобильные клиенты).

- **Couchbase**
  - Комбинирует **ключ–значение** + **документы** + **N1QL (SQL‑подобный язык)**.
  - Ближе к высокопроизводительному key–value с документами и встроенным квери‑движком.
  - Часто позиционируется как альтернатива MongoDB + Redis в одном продукте.

- **RavenDB**
  - Документная БД с сильной ориентацией на .NET‑экосистему.
  - Даёт удобный API для C#, автоматические индексы, репликацию и др.

**В отличие от MongoDB:**

- MongoDB исторически шла по пути **одного primary** с replica set (хотя с шардированием архитектура сложнее), а CouchDB — по пути **multi‑master** и «разруливай конфликты сам».
- MongoDB ближе к «централизованной БД с масштабированием»; CouchDB — к «набору узлов, которые синхронизируют документы между собой».

#### Проверь себя

1. В чём главное отличие документной БД от реляционной модели?  
<details><summary>Ответ</summary>Документная БД хранит данные в виде документов (обычно JSON/BSON) с вложенными структурами и массивами; один документ часто соответствует целому объекту предметной области, который в реляционной БД был бы разложен по нескольким таблицам и связан через внешние ключи.</details>

2. Для чего нужен агрегирующий pipeline в MongoDB?  
<details><summary>Ответ</summary>Это конвейер стадий обработки, который позволяет фильтровать, группировать, преобразовывать и соединять документы (через $match, $group, $project, $lookup, $unwind и др.), реализуя задачи, похожие на сложные SQL‑запросы с GROUP BY и JOIN.</details>

3. Что такое shard key и почему его выбор важен?  
<details><summary>Ответ</summary>Shard key — это поле (или набор полей), по которому документы распределяются по шардам в шардированной коллекции. От его выбора зависят равномерность нагрузки и возможность обслуживать запросы одним шардом; плохой shard key приводит к «горячим» шардам и неравномерной нагрузке.</details>

#### Запомните

- Документная БД удобна, когда основная единица работы — **целый объект** с вложенными частями, который чаще читается/пишется целиком.
- MongoDB даёт гибкую схему, мощный агрегирующий pipeline и много типов индексов, но требует осознанного проектирования документов и shard key.
- Replica set и sharding обеспечивают отказоустойчивость и масштаб, а read/write concern позволяют балансировать между скоростью и надёжностью.

#### Если прочитал и не понял

Коротко: документная БД хранит не строки в таблицах, а **документы**, похожие на JSON‑объекты. В одном документе ты можешь держать всё, что относится к сущности — профиль пользователя, заказ с товарами. Запросы к такой БД во многом похожи на SQL, но выражены через стадии конвейера. Для больших систем MongoDB масштабируется через replica set и sharding, а настройки read/write concern позволяют выбирать баланс между скоростью и надёжностью.
