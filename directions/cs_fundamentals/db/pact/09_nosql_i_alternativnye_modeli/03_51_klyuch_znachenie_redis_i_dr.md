## §51. Ключ–значение (Redis и др.)

### 51. Ключ–значение (Redis и др.)

**Цель раздела.**  
Понять, как работает модель «ключ–значение» на примере Redis: какие структуры данных есть, как использовать TTL, персистентность (RDB/AOF), транзакции (MULTI/EXEC) и Lua, чем Redis отличается от Memcached и когда key–value‑хранилище — мощный помощник, а не «главная БД на все случаи».

**В этом разделе главное (три строки).**  
1) Redis — это не просто «строковый кэш», а набор быстрых in‑memory структур данных (строки, списки, множества, хэши, sorted sets, streams) с богатым набором команд.  
2) TTL позволяет удобно строить кэши и временные данные, а RDB/AOF обеспечивают персистентность; однопоточность и атомарность команд упрощают мышление о конкурентных обновлениях.  
3) Redis отлично подходит для кэша, сессий, счётчиков, очередей и простых стримов, но не заменяет полноценную транзакционную БД для долговременного хранения критичных данных.

---

#### Ещё проще: образ Redis в голове

- Представь **очень быстрый словарь в памяти**:
  - ты даёшь **ключ** — строку (`"user:42"`, `"session:abc123"`);
  - получаешь или записываешь **значение** — строку или небольшую структуру.
- Главное, что Redis:
  - живёт **в оперативной памяти** → отвечает почти мгновенно;
  - умеет автоматически **забывать ключи** по времени (TTL);
  - поддерживает полезные структуры: списки (очереди), множества (уникальные элементы), сортированные множества (рейтинги).
- Думай о Redis как о **рабочем столе рядом с основной БД**: на нём лежит то, что нужно быстро, а не «вся бухгалтерия за 5 лет».

#### Термины (расшифровка)

- **Key–value (ключ–значение)** — модель хранилища, где каждое значение доступно по уникальному ключу. В Redis ключи — строки, значения могут быть строками или структурами данных.
- **Строка (string)** — базовый тип в Redis; может хранить текст или бинарные данные, а также использоваться как целое/вещественное число для инкремента.
- **Список (list)** — упорядоченная последовательность элементов; поддерживает операции push/pop с головы и хвоста (очереди, стеки).
- **Множество (set)** — неупорядоченное множество уникальных элементов.
- **Сортированное множество (sorted set, zset)** — множество элементов с числовым «баллом» (score), по которому можно сортировать и выбирать диапазоны.
- **Хэш (hash)** — ассоциативный массив поле → значение; удобен для хранения небольших объектов.
- **TTL (time‑to‑live)** — время жизни ключа; по его истечении ключ автоматически удаляется.
- **RDB / AOF** — механизмы персистентности в Redis: RDB — периодические снимки, AOF — журнал команд.
- **MULTI/EXEC** — механизм транзакций в Redis: пакет команд, который выполняется атомарно.

#### Мини‑история: сайт без Redis и с Redis

Представь, что у тебя есть сайт:

- пользователи логинятся, у них есть **сессии**;
- на главной странице выводятся **популярные товары**;
- у каждой страницы ты считаешь **количество просмотров**.

**Вариант «без Redis» (только SQL‑БД):**

- при каждом запросе:
  - идёшь в БД, проверяешь сессию;
  - считаешь популярные товары через тяжёлый запрос с агрегатами;
  - увеличиваешь счётчик просмотров в таблице.
- при высокой нагрузке БД начинает «задыхаться»: каждое мелкое действие — это запись/чтение на диск.

**Вариант «с Redis»:**

- сессию (`session:abc123 → {userId: 42}`) хранишь в Redis с TTL, проверка — одно быстрое чтение по ключу;
- популярные товары обновляешь, например, раз в минуту, а между обновлениями **читаешь только из Redis**;
- счётчик просмотров держишь как ключ `page:/home:views` и увеличиваешь через `INCR`.

Основная БД продолжает хранить **главные данные** (пользователи, товары, заказы), а Redis берёт на себя **частые и дешёвые операции**, разгружая диск.

#### Теория и правила (подробно)

- **Redis как «швейцарский нож» для быстрых операций.**

  В отличие от классической БД, Redis держит данные **в памяти** (с возможностью записи на диск для персистентности) и оптимизирован под:

  - очень быстрые операции `GET`/`SET` по ключу;
  - инкремент/декремент чисел (`INCR`, `INCRBY`, `DECRBY`);
  - очереди и стеки (`LPUSH`, `RPUSH`, `LPOP`, `RPOP`, блокирующие версии `BLPOP`/`BRPOP`);
  - множества (`SADD`, `SINTER`, `SUNION`, `SDIFF`);
  - сортированные множества (`ZADD`, `ZRANGE`, `ZREVRANGE`, `ZRANGEBYSCORE`);
  - хэши (`HGET`, `HSET`, `HGETALL`);
  - потоки (streams) для простого стриминга событий.

- **TTL и кэш с истечением.**

  Типичный сценарий — **кэширование**:

  ```bash
  SET user:42 '{"name":"Иван","age":30}' EX 300
  # Ключ user:42 живёт 300 секунд, потом Redis его удалит
  ```

  - Можно задать TTL при установке (`EX`, `PX`) или отдельной командой `EXPIRE key seconds`.
  - Команды `TTL` и `PTTL` показывают оставшееся время жизни.

- **Персистентность: RDB, AOF и их комбинация.**

  - **RDB (Redis Database)** — периодические снимки (snapshot) всей БД; быстрее восстанавливать, но можно потерять данные между снимками.
  - **AOF (Append‑Only File)** — журнал комманд; при включённом `fsync=always` почти не теряет данные, но медленнее; есть режим `everysec` (баланс).
  - Часто используют **гибридный подход**: RDB + AOF.

- **Однопоточность и атомарность.**

  Сердце Redis по умолчанию однопоточно обрабатывает команды, что даёт простую модель:

  - каждая команда выполняется **атомарно**;
  - нет классических блокировок; если ты выполняешь одну команду, никто не «влезет посередине»;
  - для нескольких команд подряд можно использовать `MULTI/EXEC` или Lua‑скрипты (`EVAL`).

- **Транзакции и Lua.**

  Пример: увеличить счётчик и, если он перешагнул порог, добавить пользователя в специальный список — всё атомарно:

  ```bash
  MULTI
    INCR user:42:counter
    ZADD leaderboard 0 user:42
  EXEC
  ```

  Или через Lua:

  ```bash
  EVAL "
    local new = redis.call('INCR', KEYS[1])
    if new > tonumber(ARGV[1]) then
      redis.call('ZADD', KEYS[2], new, ARGV[2])
    end
    return new
  " 2 user:42:counter leaderboard 100 user:42
  ```

- **Типичные use‑case‑ы Redis.**

  - **Кэш** (краткоживущие данные, результаты дорогих запросов).
  - **Сессии пользователей** (быстрый доступ, TTL по длительности сессии).
  - **Счётчики и лимиты** (просмотры, лайки, rate limiting).
  - **Очереди задач** (списки и блокирующие операции).
  - **Лёгкий pub/sub** и **streams** для внутри‑сервисной коммуникации.

#### Микро‑диалог в голове про Redis

**Вопрос к себе 1.** «Если Redis такой быстрый, почему бы не хранить в нём вообще всё?»  
**Ответ.** Потому что:
  - он живёт **в памяти**, а память дороже и ограниченнее диска;
  - при сбое/неправильной настройке персистентности часть последних данных можно потерять;
  - у него нет такой мощной транзакционной модели и инструментов отчётности, как у PostgreSQL.
  Поэтому важные и большие данные обычно живут **в реляционной БД**, а Redis — это ускоритель вокруг неё.

**Вопрос к себе 2.** «Если Redis in‑memory, значит, всё пропадёт при перезапуске?»  
**Ответ.** Не обязательно. С помощью RDB и/или AOF Redis может **писать снимки и журналы на диск**. Но:
  - это всё равно не заменяет полноценной БД с ACID‑гарантиями;
  - при агрессивных настройках можно выбрать баланс между «максимальная скорость» и «минимальная потеря данных».

**Вопрос к себе 3.** «Нужно ли использовать все структуры Redis?»  
**Ответ.** Нет. Часто достаточно:
  - **строк** (`GET`/`SET` + TTL) — для кэша и сессий;
  - **счётчиков** (`INCR`/`INCRBY`) — для простых метрик;
  - **списков** (`LPUSH`/`BRPOP`) — для очередей задач;
  - **sorted sets** — для рейтингов. Остальное можно осваивать постепенно, когда понадобится.

#### Пошагово: как построить кэш с Redis

1. **Определи ключи.**  
   Выбери, как будешь называть ключи. Принято использовать префиксы: `user:{id}`, `session:{id}`, `cache:product:{id}`.

2. **Реши, что кэшировать.**  
   Это могут быть:
   - результаты тяжёлых SQL‑запросов;
   - подготовленные HTML‑фрагменты;
   - агрегаты (счётчики, суммарные значения).

3. **Выбери TTL.**  
   Для разных типов данных — разный TTL: профиль пользователя можно кэшировать дольше, чем список популярного контента.

4. **Реализуй стратегию «read‑through».**  
   - При чтении: сначала смотри в Redis; если нет — читаешь из основной БД, кладёшь в Redis, возвращаешь клиенту.
   - При записи: либо сразу инвалидируешь кэш, либо обновляешь его.

5. **Соблюдай умеренность.**  
   Не кэшируй всё подряд. Измеряй hit‑ratio, объём памяти Redis и эффект на производительность.

#### Часто путают / типичные ошибки

#### Типичная ошибка

Использовать Redis как **основную долговременную БД** для критичных данных (деньги, заказы) «потому что он быстрый». Это опасно:

- Redis — in‑memory‑система, потеря памяти/диска или неправильная настройка персистентности может привести к потере данных;
*- транзакционная модель гораздо проще, чем у реляционных БД;*
- в случае сложных запросов и отчётов Redis не подходит по модели.

Правильный подход — использовать Redis как **слой кэша и быстрых структур данных** поверх надёжной основной БД.

К этому добавляются и другие ошибки:

- **Хранить слишком большие значения.**  
  Например, складывать в один ключ многомегабайтные JSON‑объекты. Это бьёт по памяти и сети. Лучше:
  - держать в Redis **только то, что нужно быстро** (идентификаторы, короткие данные, агрегаты);
  - большие объекты оставлять в основной БД и подгружать по необходимости.

- **Отсутствие политики очистки.**  
  Если никогда не задавать TTL и не чистить ключи, Redis может постепенно «забить» всю память. Нужен осознанный подход: что живёт долго, что — минуту, что — час.

- **Считать Redis «магическим ускорителем всего».**  
  Если исходные запросы к БД плохо спроектированы (нет индексов, лишние JOIN), Redis их не «починит», а только скроет часть проблем. Сначала стоит **починить БД**, а уже потом добавлять кэш.

#### Простыми словами

Redis — это **очень быстрый «записная книжка‑калькулятор» в памяти**, где ты можешь по ключу моментально получить значение, увеличить счётчик, положить элемент в очередь или добавить пользователя в рейтинг. Он идеален для временных и вторичных данных: кэш, сессии, счётчики, очереди. Но хранить в нём единственную копию важных данных — как хранить семейный архив только в оперативной памяти ноутбука без бэкапа.

#### Картинка в голове

Представь, что у тебя есть основная БД‑«архив» (PostgreSQL) и маленький **стол у рабочего места**, на который ты выкладываешь самые часто используемые документы и черновики. Redis — это этот стол: всё под рукой и быстро, но если в офисе случится пожар, **спасать будешь архив**, а не стол. Поэтому важные документы должны жить в архиве, а на столе — только их копии.

#### Примеры команд Redis

```bash
# Строки и TTL
SET session:abc123 '{"userId":42}' EX 1800   # сессия на 30 минут
GET session:abc123

# Счётчики
INCR page:home:views
INCRBY user:42:score 10

# Очередь задач
LPUSH queue:emails '{"to":"a@example.com","body":"..."}'
BRPOP queue:emails 0   # блокирующее чтение

# Рейтинг (leaderboard)
ZADD leaderboard 100 user:1
ZADD leaderboard 250 user:2
ZRANGE leaderboard 0 -1 WITHSCORES
```

#### Продвинуто: Redis Cluster (слоты, MOVED/ASK)

**Задача Redis Cluster.**  
Redis по умолчанию — один экземпляр. Redis Cluster позволяет:

- горизонтально **шардировать ключи по нескольким узлам**;
- при этом клиенту не нужно самому решать, на какой узел идти — кластер подскажет.

**Как это работает (упрощённо):**

- Пространство ключей делят на **16384 хеш‑слота**.
- Каждый ключ попадает в слот: `hash_slot = CRC16(key) mod 16384`.
- Каждый слот закреплён за одним мастером (и, возможно, репликами).
- Узлы кластера знают, какие слоты на каких нодах.

Когда клиент обращается к «неправильному» узлу:

- узел отвечает **MOVED** (ключ навсегда живёт на другом узле) или **ASK** (ключ временно мигрирует);
- клиент запоминает это и в следующий раз идёт уже на нужный узел.

**Практические выводы:**

- Все ключи кластера **равномерно распределяются** по слотам, если key‑space разнообразный.
- Если тебе нужно, чтобы несколько ключей лежали на одном узле (например, `user:42:profile` и `user:42:sessions`), используют **hash tags**:  
  `user:{42}:profile`, `user:{42}:sessions` → хеш считается только по `{42}` → оба ключа в один слот.

#### Продвинуто: Sentinel и отказоустойчивость

**Redis Sentinel** — отдельная компонента, которая:

- следит за мастером и репликами;
- при падении мастера запускает **автоматический failover**:
  - выбирает реплику;
  - делает её новым мастером;
  - уведомляет клиентов.

Схема:

- у тебя есть `redis-master` и несколько `redis-replica-*`;
- несколько процессов Sentinel опрашивают их и голосуют, жив ли мастер;
- если мастер недоступен, при кворуме Sentinels принимают решение о переключении.

Важно:

- клиент должен уметь **общаться с Sentinel’ами**, чтобы узнать, кто сейчас мастер;
- Sentinel — это не кластер по слотам, а именно система управления мастером/репликами.

#### Продвинуто: Streams и consumer groups

**Streams** в Redis — это:

- лог упорядоченных сообщений (`XADD`, `XREAD`, `XREADGROUP`);
- каждое сообщение имеет ID вида `timestamp-sequence`;
- данные хранятся на диске (при включённой персистентности) и могут быть перечитаны.

**Зачем это нужно:**

- простые очереди задач;
- ленты событий между сервисами;
- реализация «почти Kafka» для небольших нагрузок.

**Consumer groups:**

- Позволяют нескольким потребителям **делить работу** по одному стриму:
  - стрим — общий;
  - consumer group — логическая группа потребителей;
  - каждому сообщению назначается конкретный consumer в группе;
  - можно отслеживать непрочитанные/застрявшие сообщения.

Паттерн:

1. Продюсер пишет события: `XADD events * key1 val1 key2 val2`.
2. Группа потребителей (`XGROUP CREATE events mygroup ...`) читает батчами: `XREADGROUP GROUP mygroup consumer-1 STREAMS events >`.
3. При падении одного consumer’а сообщения могут быть перераспределены.

#### Продвинуто: семейство Redis (KeyDB, Dragonfly)

- **KeyDB**
  - форк Redis с поддержкой **многопоточности**;
  - совместим по протоколу, но использует несколько потоков для обработки, что даёт больший throughput на современных CPU;
  - цель — «скоростной Redis без изменения клиентского кода».

- **Dragonfly**
  - «современный Redis‑совместимый сервер», написан с упором на производительность;
  - многопоточность, улучшенное использование памяти, высокая пропускная способность;
  - цель — заменить Redis там, где нужна максимальная производительность и при этом подходит совместимость по протоколу.

Оба продукта:

- ориентированы на те же use‑case’ы, что и Redis;
- обычно рассматриваются как **drop‑in‑замена** для части сценариев, когда упираешься в пределы одного инстанса Redis по CPU.

#### Проверь себя

1. Почему Redis называют in‑memory key–value‑хранилищем?  
<details><summary>Ответ</summary>Потому что основные данные он хранит в оперативной памяти (RAM), а доступ организован по ключу; это даёт очень быстрые операции чтения/записи, а запись на диск (RDB/AOF) используется для персистентности, но не как основное место работы.</details>

2. Для чего использовать TTL в Redis?  
<details><summary>Ответ</summary>TTL позволяет автоматически удалять ключи по истечении заданного времени; это удобно для кэшей, сессий, временных токенов и любых данных, которые должны жить ограниченное время.</details>

3. Почему Redis плохо подходит в роли единственной БД для критичных данных?  
<details><summary>Ответ</summary>Потому что он в первую очередь in‑memory‑кэш с упрощённой транзакционной моделью; неправильная настройка персистентности или отказ узла может привести к потере последних данных, а сложные транзакции и запросы в нём не поддерживаются так же богато, как в реляционных БД.</details>

#### Запомните

- Redis — быстрый in‑memory key–value с несколькими структурами данных; он идеален как кэш, хранилище сессий, счётчики и очереди.
- TTL, RDB и AOF позволяют настраивать баланс между скоростью, экономией памяти и устойчивостью к потерям данных.
- Redis **дополняет**, а не заменяет реляционную БД: важные данные храним в надёжной СУБД, Redis используем как ускоритель.

#### Если прочитал и не понял

Коротко: Redis — это «сверхбыстрый словарь в памяти», который умеет не только строки, но и списки, множества, хэши и рейтинги. Он идеально подходит для кэшей, сессий, счётчиков и очередей задач, где главное — скорость, а не сложные транзакции и отчётность. Для долгосрочного, критичного хранения данных всё равно нужна реляционная (или другая надёжная) БД; Redis строится вокруг неё как ускоритель.
