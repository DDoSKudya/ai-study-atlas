## 6.4. Тестовые дубли (Test Doubles)

### Цель раздела

Понять, какие бывают типы тестовых дублей — **Stub, Mock, Fake, Spy, Dummy** — чем они отличаются, когда какой использовать, и какие проблемы возникают при злоупотреблении моками и слишком тесной привязке тестов к внутренней реализации.

### В этом разделе главное

- **Stub** — возвращает заготовленные ответы.
- **Mock** — кроме ответов, позволяет проверять взаимодействия (вызовы, аргументы).
- **Fake** — упрощённая рабочая реализация (например, in‑memory БД).
- **Spy** — записывает факты вызовов для последующей проверки.
- **Dummy** — просто «затычка» для параметра, который не используется.
- Избыточное использование моков может сделать тесты **хрупкими и привязанными к реализации**.

### Термины

| Термин | Определение |
|--------|-------------|
| **Test Double** | Любой объект, который подменяет реальную зависимость в тесте (по аналогии с дублёром актёра) |
| **Stub** | Дубль, возвращающий заранее подготовленные значения, но не проверяющий взаимодействий |
| **Mock** | Дубль, который и возвращает значения, и позволяет проверять, как его вызывали |
| **Fake** | Упрощённая, но рабочая реализация, подходящая для тестов (например, in‑memory) |
| **Spy** | Объект, который «подглядывает» за вызовами: записывает их для последующей проверки |
| **Dummy** | Объект‑заглушка для параметра, который не используется в логике теста |

### Теория и правила

**Stub — возвращаем значения.**

- Используется, когда нам **нужны данные от зависимости**, но не важно, как именно её вызывают.
- Пример: репозиторий пользователей, который всегда возвращает заранее подготовленный список.

**Mock — проверяем взаимодействия.**

- Полезен, когда важно не только **что вернулось**, но и **как вызывалось**:
  - сколько раз;
  - с какими аргументами;
  - в каком порядке.
- Пример: проверка, что был отправлен e‑mail конкретному пользователю.

**Fake — рабочая, но упрощённая реализация.**

- Часто in‑memory версия компонента:
  - in‑memory репозиторий вместо БД;
  - локальный HTTP‑сервер вместо реального.
- Хорошо подходит для интеграционных и высокоуровневых тестов, где хочется реального поведения без тяжёлой инфраструктуры.

**Spy — наблюдаем за вызовами.**

- Похож на mock, но акцент на **фиксации фактов**:
  - записываем аргументы и количество вызовов;
  - потом проверяем в `Then`.

**Dummy — просто затычка.**

- Объект, передаваемый только для удовлетворения сигнатуры, но не используемый в логике.
- Пример: фиктивный логгер, который ничего не делает.

### Пошагово: выбираем тип дубля

При выборе:

1. Спроси: **Нужен ли мне возврат данных от зависимости?**
   - Да → stub или fake.
2. Спроси: **Важно ли, как именно вызывается зависимость?**
   - Да → mock или spy.
3. Спроси: **Нужна ли максимально реалистичная, но лёгкая реализация?**
   - Да → fake.
4. Если зависимость вообще не используется в сценарии, но требуется по типу:
   - dummy.

**Проверь себя (выбор дубля)**  

1. Почему не стоит всегда использовать mocks «на всякий случай»?
   <details><summary>Ответ</summary>
   Моки привязывают тест к деталям взаимодействия (порядок и количество вызовов, точные аргументы). При малейшем рефакторинге внутренней реализации (изменении способа взаимодействия при том же поведении) тесты могут падать без изменения контракта. Это делает тесты хрупкими и мешает рефакторингу.
   </details>

2. Когда fake (например, in‑memory реализация) предпочтительнее mock?
   <details><summary>Ответ</summary>
   Когда важно проверять **реальное поведение** компонента (например, как он хранит и возвращает данные) в более естественном стиле, не выписывая ожидания взаимодействий. Fake позволяет писать тесты, близкие к продакшен‑сценариям, но без затрат на настоящую инфраструктуру.
   </details>

### Простыми словами

Тестовый дубль — это **актёр‑заместитель**:

- stub — актёр, который просто читает заранее выданный текст;
- mock/spy — актёр, за которым режиссёр внимательно следит: сколько раз вышел на сцену, что сказал;
- fake — актёр, который умеет полностью играть роль, но на «репетиционной» сцене;
- dummy — актёр массовки, который просто должен присутствовать.

### Картинка в голове

Представь съёмку трюков в кино:

- дублёр заменяет настоящего актёра на опасных сценах;
- иногда важно только, **что он делает** (stub/fake);
- иногда — **как он взаимодействует** с другими (mock/spy).

В тестах мы делаем то же самое с зависимостями.

### Как запомнить

- **Stub** — просто «заткнуть» зависимость данными.
- **Mock/Spy** — проверить **как** вызвали зависимость.
- **Fake** — почти как настоящая зависимость, но проще/быстрее.
- **Dummy** — «для галочки», не используется.

### Примеры

**Stub:**

```python
class StubUserRepository:
    def __init__(self, existing_users):
        self._users = existing_users

    def find_by_email(self, email):
        return self._users.get(email)


def test_auth_success_when_user_exists():
    # Given
    repo = StubUserRepository(
        {"alice@example.com": User("alice@example.com", password_hash="hash")}
    )
    auth = AuthService(user_repo=repo)

    # When
    result = auth.login("alice@example.com", "password")

    # Then
    assert result.is_success
```

**Mock (на примере pytest + monkeypatch или библиотеки моков):**

```python
def test_sends_email_on_order_confirmation(mocker):
    # Given
    email_sender = mocker.Mock()
    service = OrderService(email_sender=email_sender)
    order = Order(...)

    # When
    service.confirm(order)

    # Then
    email_sender.send.assert_called_once_with(order.customer_email, ANY)
```

**Fake (in‑memory репозиторий):**

```python
class InMemoryOrderRepository:
    def __init__(self):
        self._orders = {}

    def save(self, order):
        self._orders[order.id] = order

    def get(self, order_id):
        return self._orders[order_id]
```

### Практический сценарий

В проекте почти все тесты используют мощную библиотеку моков и выглядят так:

```python
def test_service_does_x(mocker):
    repo = mocker.Mock()
    payment = mocker.Mock()
    email = mocker.Mock()
    # десятки настроек mocks
    ...
```

При любом рефакторинге сервиса тесты падают из‑за изменившегося количества вызовов или аргументов, даже если внешнее поведение осталось тем же. Команда говорит: «рефакторить невозможно, всё завязано на тесты».

Решение:

- там, где важен **результат**, а не детали взаимодействия, заменить mocks на **fake/in‑memory** реализации;
- проверять поведение по **возвращаемым значениям и состоянию**, а не по количеству вызовов.

### Типичные ошибки

- Мокать **всё подряд**, включая простые объекты данных.
- Использовать mocks/spy для проверки чисто **внутренних деталей** (например, вызов приватного метода), а не публичного контракта.
- Путать stub и mock: настраивать поведение, но при этом ещё и проверять взаимодействия, где это не нужно.

### Что будет, если…

Если злоупотреблять моками и проверять каждую деталь взаимодействия:

- тесты станут **очень хрупкими** — малейший рефакторинг реализации потребует переписывать множество тестов;
- тесты перестанут играть роль **защиты контракта** и станут защищать внутренности (которые по определению должны уметь меняться);
- команда начнёт бояться менять код и писать новые тесты.

### Проверь себя (6.4)

1. Чем отличается stub от mock с точки зрения того, что именно они проверяют?
   <details><summary>Ответ</summary>
   Stub только **возвращает данные** (подставляет ответы) и не участвует в проверке взаимодействий. Mock, помимо возвращения данных, участвует в **проверке того, как его вызвали** (количество вызовов, аргументы, порядок).
   </details>

2. В каком случае fake‑реализация предпочтительнее набора моков и почему?
   <details><summary>Ответ</summary>
   Когда мы хотим проверить поведение компонента в сценарии, близком к реальному использованию (например, работа сервиса с репозиторием и несколькими сущностями), но не хотим поднимать настоящую инфраструктуру. Fake (in‑memory реализация) позволяет писать тесты, которые выглядят «как прод», но выполняются быстро и изолированно.
   </details>

3. Зачем нужны dummy‑объекты и почему важно, чтобы они не влияли на поведение теста?
   <details><summary>Ответ</summary>
   Dummy заполняют обязательные параметры, которые в данном сценарии не используются (например, логгер или незначащий ID). Важно, чтобы они **не участвовали** в логике теста, иначе тест станет зависеть от значений, которые никто явно не проверяет и не настраивает.
   </details>

### Запомните

- Тестовые дубли — мощный инструмент **изоляции** юнитов, но ими нужно пользоваться **осознанно**.
- Не всё нужно мокать: где можно, лучше использовать простые объекты и in‑memory реализации.
- Тест должен проверять **контракт**, а не детали внутренней кухни.
