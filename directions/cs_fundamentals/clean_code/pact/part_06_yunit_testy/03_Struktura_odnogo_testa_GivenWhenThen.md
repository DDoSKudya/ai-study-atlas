## 6.3. Структура одного теста: Given‑When‑Then

### Цель раздела

Научиться писать тесты так, чтобы они читались как **короткая история**: «дано (Given)… когда (When)… тогда (Then)…». Отделить подготовку данных от действия и проверки, уменьшить дублирование и сделать каждый тест понятным «с двух прочтений».

### В этом разделе главное

- Один тест должен проверять **один концепт**.
- Структура **Given‑When‑Then** (или Arrange‑Act‑Assert) делает тесты читаемыми.
- Важно визуально **разделять** части теста (пустыми строками, комментариями‑якорями или хелперами).
- Слишком много `assert` в одном тесте — сигнал к разбиению.

### Термины

| Термин        | Определение |
|--------------|-------------|
| **Given**    | Подготовка условий и данных для теста |
| **When**     | Действие: вызов тестируемой функции/метода |
| **Then**     | Проверка результата и побочных эффектов |
| **Arrange‑Act‑Assert** | Альтернативное название той же структуры (AAA) |

### Теория и правила

**Один тест — один концепт.**

- Хороший тест отвечает на один вопрос: «что должно происходить в этом сценарии?».
- Если тест проверяет и «валидный ввод», и «невалидный», и «пограничный случай», и «взаимодействие с БД» — его сложно понять и поддерживать.

**Структура Given‑When‑Then.**

Типичный тест можно записать так:

1. **Given** — подготовка:
   - входные данные;
   - окружение;
   - тестовые двойники.
2. **When** — действие:
   - вызов функции/метода;
   - отправка запроса в API.
3. **Then** — проверка:
   - `assert` по результату;
   - проверка вызовов моков/спаев;
   - проверки состояния.

**Визуальное разделение.**

- Используй **пустые строки** для отделения блоков.
- При необходимости — короткие комментарии‑якоря `# Given`, `# When`, `# Then` (особенно в больших тестах).

### Пошагово: переписываем тест в Given‑When‑Then

Допустим, у нас есть:

```python
def test_create_user():
    user = User("alice@example.com")
    repo = InMemoryUserRepository()
    repo.save(user)
    assert repo.get_by_email("alice@example.com") is not None
```

Структура смешана. Перепишем:

```python
def test_create_user_persists_in_repository():
    # Given
    user = User("alice@example.com")
    repo = InMemoryUserRepository()

    # When
    repo.save(user)

    # Then
    assert repo.get_by_email("alice@example.com") is not None
```

**Проверь себя (Given‑When‑Then)**  

1. Почему важно явно отделять подготовку (Given) от действия (When) и проверки (Then)?
   <details><summary>Ответ</summary>
   Это делает тест читаемым как сценарий: сначала видно, с какими данными мы работаем, затем — какое действие совершаем, потом — что ожидаем. Читателю не нужно «выковыривать» эти части из одной кучи кода. Разделение также облегчает рефакторинг: например, подготовку можно вынести в хелпер или фикстуру.
   </details>

2. Что означает «один тест — один концепт» в контексте Given‑When‑Then?
   <details><summary>Ответ</summary>
   Что внутри `When`/`Then` мы проверяем один конкретный аспект поведения: один сценарий успеха или один сценарий ошибки. Если в одном тесте после одного `When` идёт множество логически независимых проверок, возможно, это несколько тестов в одном и их стоит разделить.
   </details>

### Простыми словами

Тест — это **рассказ**:

- **Дано**: вот какие у нас исходные условия.
- **Когда**: вот что делает пользователь/система.
- **Тогда**: вот что должно произойти.

Если этот рассказ невозможно быстро прочитать из кода, тесту нужно помочь — структурой и именами.

### Картинка в голове

Представь судебное дело:

- «При данных обстоятельствах (Given)… когда подсудимый сделал X (When)… то наступили такие‑то последствия (Then)…».  

Юнит‑тест формулирует такое же **«если‑то»** для кода.

### Как запомнить

- Всегда думай: «Что здесь **дано**, что **когда**, что **тогда**?».
- Если не можешь легко выделить эти части, тест, вероятно, проверяет **слишком много** или написан **слишком вперемешку**.

### Примеры

**Плохо (смешение всего):**

```python
def test_add_to_cart():
    cart = Cart()
    cart.add(Product("A", 100), quantity=2)
    cart.add(Product("B", 50), quantity=1)
    assert cart.total == 250
    assert len(cart.items) == 2
```

Два разных аспекта (сумма и количество) в одном тесте.

**Лучше — два теста:**

```python
def test_add_to_cart_accumulates_total():
    # Given
    cart = Cart()
    cart.add(Product("A", 100), quantity=2)

    # When
    cart.add(Product("B", 50), quantity=1)

    # Then
    assert cart.total == 250


def test_add_to_cart_tracks_number_of_distinct_products():
    # Given
    cart = Cart()

    # When
    cart.add(Product("A", 100), quantity=2)
    cart.add(Product("B", 50), quantity=1)

    # Then
    assert len(cart.items) == 2
```

Каждый тест проверяет **одну мысль**.

### Практический сценарий

В legacy‑проекте ты видишь тест:

```python
def test_process_order():
    # огромный тест, который:
    # - создаёт заказ
    # - проверяет валидацию
    # - проверяет списание средств
    # - проверяет отправку письма
    # - проверяет запись в лог
    ...
```

Он падает при любом изменения в заказах, платеже или почте, и никто не понимает, что именно сломалось. Переписав его в несколько тестов вида:

- `test_process_order_valid_order_charges_customer()`;
- `test_process_order_invalid_order_raises_validation_error()`;
- `test_process_order_sends_confirmation_email()` —

ты:

- упрощаешь отладку (падает конкретный сценарий);
- делаешь тесты **документацией поведения**, а не свалкой проверок.

### Типичные ошибки

- Собирать «всё про функцию» в один тест.
- Не отделять логически разные проверки внутри теста (множество `assert` вперемешку).
- Использовать магические данные, не позволяющие понять, что именно проверяется (неочевидные числа и строки).

### Что будет, если…

Если игнорировать структуру Given‑When‑Then и писать тесты «как получится», то:

- тесты будут тяжело читаться и отлаживаться;
- новые члены команды будут бояться трогать тесты;
- с ростом проекта тестовый набор превратится в такой же непонятный legacy, как и код без чистоты.

### Проверь себя (6.3)

1. Почему наличие нескольких `assert` в тесте может быть проблемой?
   <details><summary>Ответ</summary>
   Если `assert` проверяют **разные аспекты поведения**, падение одного из них делает неочевидным, что именно сломалось. Тест становится «пакетом» проверок, и при изменении поведения нужно менять сразу много ожиданий. Лучше иметь один логический концепт на тест (иногда несколько связанных `assert` вокруг одного объекта допустимы).
   </details>

2. Как структура Given‑When‑Then помогает при код‑ревью тестов?
   <details><summary>Ответ</summary>
   Ревьюер быстро видит подготовку (контекст), действие и ожидание. Это упрощает поиск ошибок в самом тесте (например, некорректная подготовка данных) и позволяет сосредоточиться на вопросе «правильно ли мы формулируем поведение».
   </details>

3. Чем Given‑When‑Then отличается от Arrange‑Act‑Assert и почему это не принципиально?
   <details><summary>Ответ</summary>
   Это по сути одно и то же: Arrange ≈ Given, Act ≈ When, Assert ≈ Then. Различие лишь в терминологии и культуре вокруг: Given‑When‑Then часто используют в BDD‑подходах и сценариях на естественном языке. Для структуры тестов смысл одинаков: подготовка → действие → проверка.
   </details>

### Запомните

- Один тест — **один сценарий**, с чётко выделенными Given, When, Then.
- Структура теста должна быть **видна глазами**: данные, действие, проверка.
- Хорошо структурированные тесты становятся **живой документацией** поведения системы.
