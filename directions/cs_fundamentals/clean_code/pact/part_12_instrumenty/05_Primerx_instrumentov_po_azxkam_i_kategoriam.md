## 12.5. Примеры инструментов по языкам и категориям

### Цель раздела

Собрать **карту конкретных инструментов** по основным языкам и категориям (форматтеры, линтеры, статический анализ, безопасность, pre‑commit), чтобы можно было **быстро подобрать стартовый набор** для своего стека.

### В этом разделе главное

- В каждой экосистеме есть **де‑факто стандартные инструменты**.
- Не нужно изобретать свой линтер/форматтер — достаточно:
  - выбрать популярный;
  - включить рекомендованный пресет;
  - адаптировать под свой проект.
- Иногда один инструмент закрывает **сразу несколько задач** (линтинг + форматирование + метрики).

### Термины

| Термин | Определение |
|--------|-------------|
| **Toolchain** | Набор инструментов для работы с кодом в конкретном стеке |
| **SAST** | Static Application Security Testing — статический анализ безопасности |

### Теория и правила

1. **Выбираем инструменты по категориям.**

   - Форматтеры;
   - Линтеры;
   - Типовые статические анализаторы;
   - Инструменты метрик и качества;
   - Инструменты безопасности;
   - Pre‑commit фреймворки.

2. **Смотрим на экосистемные стандарты.**

   - Там, где есть общепринятый инструмент (gofmt, rustfmt, Black), **лучше взять его**, чем экзотику.

3. **Оцениваем интеграцию.**

   - Поддержка IDE;
   - наличие плагинов для pre‑commit;
   - простота запуска в CI.

### Примеры по языкам (схематично)

> Ниже приведены **ориентиры**, а не исчерпывающий список. Для конкретного проекта важно учитывать версии, требования и ограничения.

#### JavaScript / TypeScript

- **Форматтеры:**
  - Prettier.
- **Линтеры:**
  - ESLint (+ плагины для React, TypeScript и др.).
- **Статический анализ / типы:**
  - TypeScript compiler (`tsc`);
  - дополнительные правила через ESLint.
- **Инструменты качества:**
  - Jest (тесты + покрытие);
  - SonarQube / CodeClimate при необходимости.
- **Pre‑commit:**
  - Husky / lint-staged;
  - или `pre-commit` (кросс‑языковой).

#### Python

- **Форматтеры:**
  - Black (или альтернативы вроде `yapf`, `autopep8`, но Black — де‑факто стандарт).
- **Линтеры:**
  - Ruff (быстрый и многофункциональный);
  - Pylint / Flake8 (в старых кодовых базах).
- **Статический анализ / типы:**
  - mypy / pyright.
- **Инструменты качества:**
  - pytest (+ `coverage.py`);
  - SonarQube / CodeClimate.
- **Pre‑commit:**
  - фреймворк `pre-commit` (очень популярен в Python‑мире).

#### Java / Kotlin

- **Форматтеры:**
  - Google Java Format;
  - Spotless (как оболочка для форматирования в Gradle).
- **Линтеры:**
  - Checkstyle, PMD;
  - SpotBugs для поиска потенциальных багов.
- **Статический анализ и качество:**
  - SonarQube/ SonarCloud;
  - IntelliJ IDEA inspections.

#### C# / .NET

- **Форматтеры и стиль:**
  - встроенный форматтер Visual Studio / Rider;
  - EditorConfig.
- **Анализ и линтинг:**
  - Roslyn Analyzers;
  - ReSharper.
- **Качество и метрики:**
  - SonarQube;
  - встроенные средства в IDE.

#### Go

- **Форматтеры:**
  - gofmt (стандарт);
  - goimports (форматирование + сортировка импортов).
- **Линтеры:**
  - golangci-lint (объединяет множество линтеров).
- **Статический анализ:**
  - встроенные инструменты `go vet` и др.

#### Rust

- **Форматтеры:**
  - rustfmt.
- **Линтеры и анализ:**
  - Clippy (мощный линтер, встроен в экосистему).

#### Инструменты безопасности (перекрывают многие языки)

- **SAST / security‑сканеры:**
  - Semgrep;
  - Bandit (Python);
  - CodeQL (GitHub).

##### Как использовать security‑инструменты на практике

- **Semgrep**:
  - ставится как обычный CLI‑инструмент;
  - запускается по коду с выбранными правилами (например, набором «для Python» или «для веб‑приложений»);
  - хорошо подходит для **локального запуска** и интеграции в CI.

  Пример шага в GitHub Actions:

```yaml
- uses: returntocorp/semgrep-action@v1
  with:
    config: "p/ci"
```

- **Bandit (Python)**:
  - анализирует Python‑код на известные шаблоны уязвимостей (использование небезопасных API, слабые генераторы случайных чисел и т.п.);
  - удобно запускать в pre‑commit или отдельной CI‑джобе.

  Пример hook’а в `pre-commit`:

```yaml
- repo: https://github.com/PyCQA/bandit
  rev: 1.7.9
  hooks:
    - id: bandit
      args: ["-r", "."]
```

- **CodeQL (GitHub)**:
  - интегрируется через стандартный GitHub‑workflow;
  - подходит для регулярного анализа основного репозитория;
  - использует готовые или кастомные запросы к базе кода.

  Пример (упрощённый) фрагмент workflow:

```yaml
jobs:
  codeql:
    uses: github/codeql-action/.github/workflows/codeql.yml@v3
```

Во всех случаях важно:

- сначала настроить **минимальный набор правил**, чтобы не утонуть в ложноположительных срабатываниях;
- договориться в команде, **как triage’ить найденные уязвимости** и кто за это отвечает;
- помнить, что security‑сканеры **дополняют**, а не заменяют ручные проверки и здравый смысл в дизайне.

#### Pre‑commit фреймворки (мульти-языковые)

- `pre-commit` (Python‑проект, но подходит для любых языков);
- Lefthook;
- Husky (для JS/TS‑экосистемы).

### Практика / реальные сценарии

- В новом JS‑проекте вы:
  - выбираете ESLint + Prettier;
  - добавляете Jest для тестов;
  - настраиваете Husky или `pre-commit` для запуска форматтера и линтера.

- В Python‑проекте вы:
  - включаете Black и Ruff;
  - добавляете mypy для типовой проверки;
  - настраиваете `pre-commit` для всего этого.

### Типичные ошибки

- Выбирать **экзотические инструменты**, вместо того чтобы взять широко принятые в экосистеме.
- Пытаться «одним инструментом закрыть всё», когда он явно для этого не предназначен.
- Игнорировать поддержку в IDE и CI:
  - инструмент трудно запускать;
  - команда не использует его на практике.

### Проверь себя

1. Почему полезно ориентироваться на **де‑факто стандарты** инструментов в экосистеме, а не придумывать свои?

   <details><summary>Ответ</summary>
   Стандартные инструменты:
   - лучше документированы;
   - поддерживаются сообществом;
   - интегрируются в IDE и CI;
   - привычны новым членам команды.  
   Использование экзотики повышает порог входа и усложняет поддержку.
   </details>

2. Какой минимальный набор инструментов вы бы выбрали для нового проекта на Python или JS/TS?

   <details><summary>Ответ</summary>
   Пример для Python: Black (форматтер) + Ruff (линтер) + pytest (тесты) + `pre-commit`.  
   Пример для JS/TS: Prettier (форматтер) + ESLint (линтер) + Jest (тесты) + Husky/`pre-commit` для хуков.
   </details>

3. Почему важно учитывать **поддержку в IDE** при выборе инструмента?

   <details><summary>Ответ</summary>
   Если инструмент плохо интегрирован в IDE, разработчики будут запускать его реже или вообще забывать о нём. Хорошая интеграция (подсветка, автоисправление, запуск по клику) делает использование инструмента естественной частью работы, а не дополнительной нагрузкой.
   </details>

### Запомните

- В каждой экосистеме есть **набор проверенных инструментов** — начинайте с них.
- Минимальный набор: **форматтер + линтер + тесты + pre‑commit**.
- Удобство интеграции в IDE и CI — **не менее важно**, чем мощность самого инструмента.
