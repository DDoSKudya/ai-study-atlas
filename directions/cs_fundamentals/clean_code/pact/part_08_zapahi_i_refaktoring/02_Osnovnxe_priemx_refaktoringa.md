## 8.2. Основные приёмы рефакторинга

### Цель раздела

Освоить базовые **микро‑операции рефакторинга**, из которых строятся большие улучшения: извлечение метода/класса, перемещение метода/поля, введение параметра‑объекта, замена условной логики полиморфизмом, split phase, замена магических чисел константами и др.

### В этом разделе главное

- Рефакторинг — это не «героический рывок», а **серия маленьких безопасных шагов**.
- Для каждого типового запаха есть 1–3 особенно полезных приёма.
- Важно знать **как именно** выполняется приём шаг за шагом и **какие проверки** делать по пути.

### Термины

| Термин | Определение |
|--------|-------------|
| **Extract Method** | Извлечение фрагмента кода в отдельную функцию с говорящим именем |
| **Extract Class** | Выделение части полей и методов класса в новый класс |
| **Move Method / Move Field** | Перемещение метода/поля в класс, к которому он логически относится |
| **Introduce Parameter Object** | Объединение группы параметров в один объект/структуру |
| **Replace Conditional with Polymorphism** | Замена большого `if/elif/else` по типу и состоянию на иерархию классов с разными реализациями |
| **Split Phase** | Разделение функции на последовательные фазы (парсинг, подготовка данных, расчёт) |
| **Replace Magic Number with Constant** | Замена «голых» чисел и строк на именованные константы |
| **Inline Method / Inline Variable** | Обратные приёмы: убрать лишнюю абстракцию, вернув код на место |

### Теория и правила

#### Extract Method (извлечение метода)

**Когда применять:**

- длинный метод;
- повторяющийся фрагмент кода;
- фрагмент с естественным осмысленным именем («посчитать скидку», «проверить права»).

**Шаги (упрощённо):**

1. Выбери непрерывный фрагмент кода.
2. Убедись, что нужные переменные:
   - либо локальны,
   - либо могут быть переданы как параметры/возврат.
3. Создай новую функцию:
   - с осмысленным именем,
   - с параметрами, соответствующими использованным данным.
4. Перенеси код, заполни параметры и возвращаемое значение.
5. Замени исходный фрагмент вызовом новой функции.
6. Запусти тесты (или хотя бы ручной сценарий), чтобы убедиться, что поведение не изменилось.

#### Extract Class (извлечение класса)

**Когда применять:**

- класс слишком большой (много полей и методов);
- поля и методы естественно делятся на «кластеры», почти не пересекающиеся;
- часть методов работает только с подмножеством полей.

**Идея:** выделить новый класс, отвечающий за конкретную подсистему/аспект (например, «формирование отчётов» внутри `ApplicationManager`).

**Шаги (очень кратко):**

1. Найди группу полей и методов, тесно связанных между собой.
2. Создай новый класс с осмысленным именем.
3. Перенеси туда поля и методы; при необходимости передавай ссылку на исходный класс.
4. Обнови использование: методы исходного класса начинают делегировать в новый.
5. Запусти тесты.

#### Move Method / Move Field

**Когда применять:**

- метод класса А постоянно использует данные класса B;
- изменение данных в B влечёт необходимость менять метод в А.

**Шаги:**

1. Определи, к какому классу **по сути принадлежит** поведение (где «живёт» бизнес‑смысл).
2. Перенеси метод в этот класс:
   - обнови параметры (возможно, `self`/`this` поменяется);
   - обнови вызовы по проекту.
3. При необходимости оставь в старом классе «тонкий делегат» (метод, просто вызывающий новый).

#### Introduce Parameter Object

**Когда применять:**

- 3–5 аргументов, которые **всегда ходят вместе**;
- добавление нового поля этой группы требует правки множества сигнатур.

**Шаги:**

1. Создай класс или структуру с полями этой группы.
2. Обнови сигнатуру функций: вместо множества параметров передаётся один объект.
3. Поэтапно обновляй вызовы, используя IDE‑рефакторинг.

#### Replace Conditional with Polymorphism

**Когда применять:**

- большой `if/elif/else` или `switch` по типу/статусу;
- добавление нового типа требует добавлять новую ветку в каждый такой `if`.

**Идея:** вместо «если тип А — делай это, если тип B — делай то» ввести **иерархию классов** с разными реализациями метода:

- `ShippingCostCalculator` → подклассы `StandardShipping`, `ExpressShipping`, `FreeShipping`.

**Шаги (упрощённо):**

1. Выдели место, где принимается решение по типу.
2. Введи базовый тип (класс/интерфейс) с нужным методом.
3. Создай подклассы/реализации под каждую ветку.
4. Перенеси код из `if` в соответствующие реализации.
5. Там, где раньше выбирался `if`, теперь создавай нужный объект (фабрика/мэппинг).

#### Split Phase (разделение фаз)

**Когда применять:**

- функция делает несколько **последовательных фаз**:
  - парсит вход,
  - преобразует структуру,
  - считает результат.

**Шаги:**

1. Чётко выдели фазы в комментариях/мысленно.
2. Вынеси каждую фазу в отдельную функцию (или даже класс), передавая результаты между фазами.
3. Идеал — отдельные фазы можно тестировать по отдельности.

#### Replace Magic Number with Constant

**Когда применять:**

- встречаешь число/строку, смысл которого неочевиден из контекста:
  - `if (status == 3)`,
  - `sleep(86400)`.

**Шаги:**

1. Придумай хорошее имя константе.
2. Введи константу на уровне модуля/класса.
3. Замени «голое» значение на использование константы.

### Пошагово: как выполнять маленький рефакторинг

1. **Убедись, что есть хотя бы базовые тесты** вокруг изменяемого кода (юнит, интеграционный тест или ручной сценарий).
2. Выбери **один** маленький приём (например, Extract Method).
3. Применяй приём **по шагам**, не перескакивая.
4. После каждого шага **запускай тесты** (или сценарий).
5. Только когда код стабилен, переходи к следующему приёму.

**Проверь себя (микро‑рефакторинги)**

1. Почему важно после каждого небольшого шага рефакторинга запускать тесты?
   <details><summary>Ответ</summary>
   Маленькие шаги легко контролировать, но человек всё равно ошибается: опечатки, не тот параметр, забыли обновить вызов. Частый запуск тестов позволяет быстро поймать ошибку и точно знать, на каком шаге она возникла.
   </details>

2. В чём разница между Extract Method и Extract Class по эффекту на дизайн?
   <details><summary>Ответ</summary>
   Extract Method уменьшает сложность и размер конкретной функции, делая код более модульным внутри одного класса. Extract Class меняет архитектуру уровня классов: выносит ответственность в новый тип, улучшая связность и уменьшая связанность. Первый работает на «локальном» уровне, второй — на уровне структуры системы.
   </details>

### Простыми словами

Рефакторинг — это набор **маленьких движений**, а не одна большая операционная. Из кода «как получилось» мы шаг за шагом строим аккуратный, понятный модуль. Каждый приём — как отдельный инструмент в ящике: один для откручивания винтов (Extract Method), другой для распиливания больших деталей (Extract Class), третий — для наведения порядка в проводах (Replace Conditional with Polymorphism).

### Картинка в голове

Представь **набор инструментов**:

- отвёртка (Extract Method),
- пила (Extract Class),
- пассатижи (Move Method),
- изолента (Replace Magic Number with Constant),
- набор переходников (Introduce Parameter Object).

Когда ты умеешь ими пользоваться, ремонт (рефакторинг) перестаёт быть страданием и превращается в ремесло: понятно, какой инструмент под какую проблему.

### Как запомнить

- **Запах → приём:** дублирование → Extract Method / Extract Class; длинный метод → Extract Method / Split Phase; длинный список параметров → Introduce Parameter Object; feature envy → Move Method; большой `if` по типу → Replace Conditional with Polymorphism.
- **Один шаг → один приём → тесты.**

### Примеры

**Extract Method + Replace Magic Number with Constant:**

```python
# До
def calculate_price(order):
    total = 0
    for item in order.items:
        total += item.price * item.quantity
    tax = total * 0.2
    return total + tax


# После
TAX_RATE = 0.2


def calculate_price(order):
    total = calculate_items_total(order)
    tax = total * TAX_RATE
    return total + tax


def calculate_items_total(order):
    return sum(item.price * item.quantity for item in order.items)
```

**Introduce Parameter Object:**

```python
# До
def create_user(first_name, last_name, email, phone, street, city, zip_code, country):
    ...


# После
class ContactInfo:
    def __init__(self, first_name, last_name, email, phone):
        self.first_name = first_name
        self.last_name = last_name
        self.email = email
        self.phone = phone


class Address:
    def __init__(self, street, city, zip_code, country):
        self.street = street
        self.city = city
        self.zip_code = zip_code
        self.country = country


def create_user(contact: ContactInfo, address: Address):
    ...
```

### Практический сценарий

Ты видишь длинный метод `generate_invoice`, который:

- считает суммы;
- считает налоги;
- форматирует строки;
- пишет PDF;
- отправляет письмо.

План:

1. Сначала Extract Method:
   - `calculate_totals`,
   - `format_invoice`,
   - `render_pdf`,
   - `send_invoice_email`.
2. Затем, возможно, Extract Class:
   - `InvoiceCalculator`,
   - `InvoiceRenderer`,
   - `InvoiceSender`.
3. На каждом шаге — тесты (или ручные сценарии).

Через 10–20 таких шагов код становится ясным, а не «простынёй».

### Типичные ошибки

- Пытаться «сделать всё за один рефакторинг» вместо маленьких шагов.
- Переименовывать, извлекать и перемещать **вручную**, игнорируя возможности IDE (увеличивает риск ошибок).
- Не запускать тесты между шагами → трудно понять, когда именно что‑то сломалось.

### Что будет, если…

**Если применять приёмы хаотично и большими порциями:**

- легко сломать поведение;
- тяжело отследить, какие изменения что повлияли;
- ревью превращается в просмотр «огромной простыни» diff’а, где трудно отличить рефакторинг от случайных изменений.

### Проверь себя (8.2)

1. Какой приём рефакторинга ты применишь в первую очередь к длинной функции, где можно выделить логически цельный блок?
   <details><summary>Ответ</summary>
   Extract Method: выделить фрагмент кода в отдельную функцию с говорящим именем, чтобы уменьшить размер и повысить читаемость исходной функции.
   </details>

2. Когда вместо Extract Method стоит задуматься об Extract Class?
   <details><summary>Ответ</summary>
   Когда множество методов класса работают с одной и той же группой полей, образуя «подсистему» внутри класса, и когда ответственность этой подсистемы можно описать отдельно (например, «расчёт цен», «управление уведомлениями»). Тогда новый класс с этой ответственностью улучшит связность и уменьшит связанность.
   </details>

3. Как понять, что пора вводить объект параметров?
   <details><summary>Ответ</summary>
   Когда 3–5 аргументов почти всегда передаются вместе, логически представляют собой одну сущность (адрес, диапазон, контактные данные) и при добавлении нового поля приходится менять множество сигнатур. Это сигнал к Introduce Parameter Object.
   </details>

### Запомните

- Рефакторинг строится из **маленьких, чётко определённых приёмов**.
- У каждого запаха есть **типовой набор приёмов**, которые особенно хорошо работают.
- Основная защита при рефакторинге — **тесты и маленькие шаги**.
