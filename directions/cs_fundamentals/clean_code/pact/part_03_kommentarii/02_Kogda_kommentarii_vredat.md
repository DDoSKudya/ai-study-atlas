## 3.2. Когда комментарии вредят

### Цель раздела

Научиться распознавать и устранять вредные комментарии: закомментированный код, шумовые и устаревшие комментарии, а также комментарии, которые используются вместо рефакторинга и ухудшают качество кода.

### В этом разделе главное

- **Закомментированный код** почти всегда должен быть удалён.
- **Шумовые комментарии** (повтор кода, очевидные вещи) загружают мозг читателя.
- **Устаревшие комментарии** хуже отсутствующих: они дезинформируют.
- Комментарии **не должны заменять** улучшение имён, структуры и архитектуры.

### Термины

| Термин                    | Определение |
|---------------------------|-------------|
| **Шумовой комментарий**  | Комментарий, который не добавляет информации по сравнению с кодом |
| **Устаревший комментарий** | Комментарий, не соответствующий текущему поведению кода |
| **Комментарий‑костыль**  | Комментарий, который прикрывает плохой код вместо его улучшения |

### Теория и правила

**1. Закомментированный код**

Пример:

```python
def calculate_total(items):
    total = 0
    for item in items:
        total += item.price * item.quantity
    return total

    # Старый вариант:
    # total = sum(map(lambda x: x.price * x.quantity, items))
    # return total
```

Проблемы:

- трудно понять, **какой код реально используется**;
- возникает вопрос: «старый код уже не нужен или его оставили на всякий случай?»;
- при чтении и ревью приходится тратить время на неиспользуемый фрагмент;
- закомментированный код **никто не поддерживает**: он устаревает и ещё сильнее сбивает с толку.

В системе контроля версий весь старый код и так сохранён. Поэтому **почти всегда лучше удалить** закомментированный код и при необходимости сослаться на коммит.

**2. Шумовые комментарии («очевидное очевидно»)**

Пример:

```python
total = 0  # устанавливаем total в ноль
for item in items:  # перебираем элементы
    total += item.price * item.quantity  # добавляем к общей сумме
``+

Такие комментарии:

- не несут дополнительной информации;
- мешают быстро просканировать код;
- удорожают сопровождение: при изменении кода комментарии нужно переписывать или удалять.

Хороший критерий: если комментарий **можно удалить без потери понимания**, он, скорее всего, не нужен.

**3. Устаревшие комментарии**

Пример:

```python
def get_discounts():
    # Возвращает список всех скидок пользователя
    return discount_repository.get_active_discounts_for_today()
```

Комментарий говорит «все скидки», код — только «активные на сегодня». Через время поведение может ещё поменяться, а комментарий останется. Читатель, доверяя тексту, будет делать неверные выводы.

**Устаревший комментарий хуже его отсутствия**, потому что:

- создаёт **ложное чувство понимания**;
- ведёт к неверным решениям и багам;
- усложняет ревью (нужно сверять код и комментарий).

**4. Комментарии вместо рефакторинга**

Пример:

```python
# Здесь мы валидируем заказ и сохраняем его в базу.
# Не трогать без крайней необходимости!
def process_order(order):
    # 100 строк логики...
    ...
```

Или:

```python
# Эта функция делает очень много, но переписывать страшно.
def do_everything(...):
    ...
```

Такие комментарии:

- признают, что код **сложен и хрупок**, но ничего с этим не делают;
- передают страх, а не знание;
- закрепляют «грязь», вместо того чтобы планировать рефакторинг.

Правильная реакция:

- зафиксировать техдолг (тикет, ADR);
- постепенно разбирать код под тестами (часть 8 и 11);
- оставить комментарий только как **временное пояснение**, если рефакторинг невозможен прямо сейчас.

**5. Декоративные комментарии и «заборы»**

Пример:

```python
##########################
## Работа с пользователями
##########################
```

Вместо того чтобы использовать:

- заголовки функций/классов;
- логичную структуру модулей (часть 4).

Такие разделители:

- часто ломают форматирование;
- плохо переживают рефакторинг (название блока меняется, комментарий — нет);
- не нужны при нормальной структуре файлов.

### Пошагово: как чистить вредные комментарии

1. При чтении кода **отмечай**:
   - закомментированный код;
   - комментарии, повторяющие код;
   - комментарии, противоречащие коду.
2. Для каждого случая спроси:
   - Можно ли этот фрагмент **удалить** без потери смысла?
   - Можно ли заменить комментарий на **лучшее имя/структуру**?
   - Нужен ли **новый комментарий**, описывающий реальное поведение?
3. Примени одно из действий:
   - удалить закомментированный код (оставив при необходимости ссылку на коммит);
   - удалить шумовой комментарий;
   - обновить устаревший комментарий или, лучше, переписать код так, чтобы комментарий не был нужен.

**Проверь себя (чистка комментариев)**

1. Почему удаление закомментированного кода редко является рискованной операцией?
   <details><summary>Ответ</summary>
   Потому что весь старый код хранится в системе контроля версий, и его всегда можно восстановить по хэшу коммита или по истории изменений. Закомментированный код в рабочем файле не даёт никаких преимуществ, кроме визуального шума и путаницы.
   </details>

2. В каких случаях устаревший комментарий может быть опаснее отсутствия комментариев?
   <details><summary>Ответ</summary>
   Когда читатель доверяет тексту комментария и на его основе принимает решения (например, предполагает, что функция возвращает все записи, а на самом деле — только активные). Это может привести к неправильной бизнес‑логике или некорректным изменениям кода.
   </details>

### Простыми словами

Вредные комментарии — это те, которые **не помогают понять код** или, ещё хуже, **запутывают**. Закомментированный код, повторения очевидного и устаревшие пояснения создают иллюзию порядка, но в реальности мешают. Гораздо полезнее их убрать и сделать сам код чище.

### Картинка в голове

Представь старую книжную полку, где между книгами лежат **старые газеты**: когда‑то они были актуальны, но теперь только мешают найти нужное. Закомментированный и устаревший код — те же газеты: место занимают, информацию не несут, а выбросить «жалко».

### Как запомнить

- **Закомментированный код → удалить**, полагаясь на историю в git.
- **Шумовой комментарий** = комментарий, который можно удалить без потери понимания.
- **Устаревший комментарий опаснее его отсутствия.**

### Примеры

**Шумовой комментарий (плохо):**

```python
count = 0  # устанавливаем счётчик в ноль
```

**Лучше без комментария:**

```python
item_count = 0
```

или ещё лучше:

```python
items_processed = 0
```

**Устаревший комментарий (плохо):**

```python
def get_active_users():
    # Возвращает всех пользователей
    return user_repo.get_active_users(limit=100)
```

**Исправление:**

```python
def get_active_users(limit: int = 100):
    """
    Вернуть активных пользователей, ограничив количество.

    :param limit: максимальное число пользователей в выдаче.
    """
    return user_repo.get_active_users(limit=limit)
```

### Практический сценарий

На ревью ты видишь:

```python
def handle_request(request):
    # Обрабатываем запрос
    ...
```

Комментарий **ничего не добавляет**. Вместо него:

- переименуй функцию в `handle_order_creation_request` или `handle_auth_request`, если это отражает суть;
- внутри раздели на маленькие функции с говорящими именами;
- убери комментарий, потому что код и так «говорит сам за себя».

### Типичные ошибки

- Оставлять закомментированный код «на всякий случай».
- Писать комментарии ради «галочки», не думая, что именно хочет узнать читатель.
- Не обновлять или не удалять комментарии при изменении кода.

### Что будет, если…

**…не чистить вредные комментарии?**

База комментариев превращается в «свалку»: читателю становится трудно отличить полезное от мусора. Это снижает доверие к комментариям в целом: разработчики перестают на них опираться, а значит, даже хорошие комментарии теряют ценность.

### Проверь себя (3.2)

1. Почему закомментированный код — почти всегда кандидат на удаление?
   <details><summary>Ответ</summary>
   Потому что система контроля версий хранит полную историю, а закомментированный код только загромождает файл и сбивает с толку. Если старый код понадобится, его можно восстановить из истории, а не держать в комментариях.
   </details>

2. Как определить, что комментарий является «шумовым»?
   <details><summary>Ответ</summary>
   Если удалить комментарий и при этом понимание кода не ухудшится, он, скорее всего, шумовой. Особенно если он просто пересказывает очевидные действия кода («увеличиваем счётчик», «перебираем список»).
   </details>

3. Почему комментарии не должны использоваться вместо рефакторинга?
   <details><summary>Ответ</summary>
   Комментарий не убирает сложность кода, а только описывает её. Такой код остаётся трудным для понимания и изменения, а комментарий может устареть. Рефакторинг снижает сложность, а комментарий лишь добавляет слова поверх неё.
   </details>

### Запомните

- **Закомментированный код, шум и устаревшие комментарии — это долги**, а не актив.
- Комментарии **не призваны маскировать плохой код**: при возможности нужно улучшать сам код.
- **Доверие к комментариям** формируется тем, насколько они точны и полезны; шум это доверие разрушает.
