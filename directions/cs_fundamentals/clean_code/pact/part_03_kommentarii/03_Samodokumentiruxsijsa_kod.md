## 3.3. Самодокументирующийся код

### Цель раздела

Понять, как сделать так, чтобы код **максимально ясно «рассказывал историю» без комментариев**: за счёт имён, структуры, функций и форматирования; и при этом знать, когда всё‑таки нужно добавить комментарий поверх даже хорошо написанного кода.

### В этом разделе главное

- **Самодокументирующийся код** — это код, где структура и имена достаточны для понимания основных идей.
- Комментарии нужны **как дополнение**, а не как основное средство объяснения.
- Меньше комментариев ≠ меньше понимания: **смысл переносится в код**.

### Термины

| Термин                      | Определение |
|-----------------------------|-------------|
| **Самодокументирующийся код** | Код, который ясно описывает своё поведение за счёт имён и структуры, минимально нуждаясь в комментариях |
| **Код‑история**            | Код, который читается сверху вниз как последовательность шагов сценария |

### Теория и правила

**1. Имена как встроенные комментарии**

Из части 1:

- имена **раскрывают намерение**;
- функции как шаги сценария;
- классы и модули как «вывески» ответственности.

Если вместо:

```python
def process(d):
    ...
```

ты пишешь:

```python
def charge_customer_for_confirmed_orders(orders, payment_gateway):
    ...
```

— половина работы по «документации» уже сделана.

**2. Структура функций и модулей**

Из части 2:

- маленькие функции;
- один уровень абстракции;
- step‑down (сверху сценарий, ниже детали).

Если верхнеуровневая функция выглядит как:

```python
def process_order(order, user):
    validate_order(order)
    discount = calculate_discount(order, user)
    apply_discount(order, discount)
    reserve_items(order)
    save_order(order)
    notify_user(order, user)
```

— это уже **живая документация** сценария работы с заказом.

**3. Форматирование как навигация**

Из части 4 (анонс):

- отступы и пустые строки разделяют логические блоки;
- порядок функций и классов помогает увидеть поток выполнения.

Самодокументирующийся код часто не требует комментариев‑разделителей — навигация обеспечивается структурой.

**4. Когда комментарий всё же нужен в самодокументирующемся коде**

Даже если код:

- хорошо назван;
- грамотно структурирован;
- разделён по слоям,

комментарий может быть нужен, когда:

- речь идёт о **внешнем ограничении** (регуляция, внешний API, баг сторонней библиотеки);
- нужно пояснить **нетривиальное бизнес‑решение**;
- нужно отметить **особый граничный случай**, неочевидный из кода.

То есть комментарии остаются, но их роль смещается с «пояснить, что мы написали» на «пояснить, почему мир устроен так».

### Пошагово: превращаем комментарии в самодокументирующийся код

1. Найди участок кода с комментарием вида:
   - `# проверяем, что заказ валиден`;
   - `# применяем скидку`;
   - `# отправляем уведомление`.
2. Спроси: можно ли заменить этот комментарий на:
   - функцию с говорящим именем (`validate_order`, `apply_discount`, `notify_user`);
   - лучшее имя переменной (`is_valid`, `discount_amount`);
   - более явную структуру (guard‑клозы, отдельные функции).
3. Реализуй:
   - извлеки кусок кода в функцию с именем из комментария;
   - удали комментарий, если он стал лишним;
   - оставь комментарий только там, где **дополняешь код внешним контекстом**.

**Проверь себя (перенос смысла в код)**

1. Почему имя функции часто лучше комментария перед блоком кода?
   <details><summary>Ответ</summary>
   Имя функции travel вместе с кодом, и его нельзя «забыть обновить» отдельно, как комментарий. Замена комментария на вызов функции с говорящим именем делает намерение частью структуры кода, а не отдельным текстом, который может устареть.
   </details>

2. Почему самодокументирующийся код не означает «никаких комментариев вообще»?
   <details><summary>Ответ</summary>
   Потому что есть вещи, которые код выразить не может или не должен: юридические требования, внешние ограничения, ссылки на документацию, нетривиальные бизнес‑решения. Самодокументирующийся код минимизирует комментарии, но не отменяет их там, где нужна дополнительная информация.
   </details>

### Простыми словами

Самодокументирующийся код — это **код, который сам объясняет себя**. Ты читаешь имена функций и переменных, и тебе не нужны комментарии, чтобы понять, что происходит. Комментарии остаются для редких случаев, когда нужно рассказать что‑то о бизнесе или внешнем мире, а не о самом коде.

### Картинка в голове

Представь две инструкции:

- первая — с плохими заголовками, но с кучей сносок внизу страницы, объясняющих каждый шаг;
- вторая — с хорошими заголовками и структурой, где сноски нужны только для редких исключений.

Самодокументирующийся код — это вторая инструкция.

### Как запомнить

- **Имя функции = встроенный комментарий.**
- **Структура модуля** и порядок функций — это оглавление и план.
- Комментарии нужны для **мира вокруг кода**, а не для повторения того, что уже видно.

### Примеры

**До (комментарии объясняют код):**

```python
def process(order, user):
    # проверяем, что заказ валиден
    if not order.items:
        raise ValueError("Empty order")
    if order.total <= 0:
        raise ValueError("Invalid total")

    # применяем скидку
    discount = 0
    if user.is_vip:
        discount = order.total * 0.1

    # отправляем уведомление
    email.send(user.email, "Order confirmed")
```

**После (код говорит сам за себя):**

```python
def process_order(order, user):
    validate_order(order)
    apply_discount_for(user, order)
    notify_user_about_confirmation(user, order)
```

### Практический сценарий

При ревью ты видишь множество комментариев вида:

```python
# проверяем права
if not user.is_admin:
    raise PermissionError()

# сохраняем заказ
db.save(order)
```

Рефакторинг:

- выносишь проверку прав в `ensure_admin(user)`;
- заменяешь комментарии на вызовы этих функций;
- код становится короче и яснее, а комментарии остаются только там, где нужен дополнительный контекст (например, почему именно admin).

### Типичные ошибки

- Пытаться полностью избавиться от комментариев, даже там, где они оправданы.
- Считать, что «самодокументирующийся код» — это просто длинные имена, а не хорошая структура.
- Оставлять комментарии, которые уже можно заменить на нормальные имена и функции.

### Что будет, если…

**…не стремиться к самодокументирующемуся коду?**

Проект обрастёт слоями комментариев, которые пытаются объяснить всё: от того, что делает цикл, до там, почему мы вызываем ту или иную функцию. С ростом кода комментарии перестанут обновляться, и вместо помощи станут мешать. Каждый новый разработчик будет тратить время на чтение текста вокруг кода вместо того, чтобы понимать сам код.

### Проверь себя (3.3)

1. Какой первый шаг ты сделаешь, увидев комментарий «# проверяем, что заказ валиден»?
   <details><summary>Ответ</summary>
   Попробую вынести код проверки в функцию с именем `validate_order(order)` и заменить комментарий вызовом этой функции. Если после этого комментарий становится лишним, его можно удалить.
   </details>

2. Почему самодокументирующийся код особенно важен в больших проектах?
   <details><summary>Ответ</summary>
   Потому что количество кода и участников команды велико; полагаться на комментарии везде не масштабируется — они устаревают и конфликтуют. Самодокументирующийся код делает понимание более надёжным и менее зависящим от текстовых пояснений.
   </details>

3. Может ли быть самодокументирующимся код с плохими именами, но хорошими комментариями?
   <details><summary>Ответ</summary>
   Нет: самодокументирующийся код по определению опирается на имена и структуру. Хорошие комментарии могут временно компенсировать плохие имена, но это не делает код самодокументирующимся, а лишь создаёт зависимость от текста, который легко устаревает.
   </details>

### Запомните

- **Самодокументирующийся код** — это результат хороших имён, функций, структуры и форматирования.
- Комментарии используются **по минимуму** и там, где код не может выразить нужную информацию.
- При виде комментария **сначала подумай, можно ли перенести смысл в код**.
