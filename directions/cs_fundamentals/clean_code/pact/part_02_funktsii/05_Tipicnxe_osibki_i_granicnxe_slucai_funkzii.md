## 2.5. Типичные ошибки и граничные случаи (функции)

### Цель раздела

Собрать воедино основные ошибки при работе с функциями и понять, в каких ситуациях правила (маленькие функции, мало аргументов и т.д.) можно осознанно ослабить.

### В этом разделе главное

- **Длинные функции** и **длинные списки параметров** — ключевые запахи, которые нужно уметь распознавать и исправлять.
- **Смешение уровней абстракции** и побочных эффектов делает функции хрупкими.
- Есть **граничные случаи**, когда «большая» функция или отсутствие разбиения могут быть приемлемы, но это должно быть осознанным решением.

### Термины

| Термин | Определение |
|--------|-------------|
| **Длинная функция** | Функция, трудно читаемая целиком и выполняющая много логических шагов |
| **Граничный случай** | Ситуация, когда общее правило (например, «всегда дели») можно ослабить по понятным причинам |

### Теория и правила

**Длинная функция.**  
Признаки:

- занимает больше одного экрана;
- трудно описать одним предложением;
- содержит и валидацию, и бизнес‑логику, и работу с инфраструктурой.

Решения:

- разбиение по шагам (как в 2.1);
- выделение **частей в отдельные функции** и, при необходимости, **классы** (часть 7);
- вынесение работы с внешними ресурсами в отдельные слои (часть 9).

**Длинный список параметров.**  
Обсуждалось в 2.2: 4+ параметров часто указывают на проблемы дизайна. Рефакторинг:

- объект параметров;
- несколько функций с меньшим количеством параметров;
- использование сущностей вместо набора примитивов.

**Выходные параметры вместо возвращаемого значения.**  
В ряде языков (C, C#, C++) можно передавать параметры «по ссылке» (`out`, `ref`) и изменять их внутри функции, словно это дополнительные выходные значения. Это затрудняет чтение:

- по сигнатуре не всегда очевидно, какие параметры меняются;
- у вызывающего кода появляется «скрытый» канал передачи данных.

Лучше:

- возвращать значение через `return` (или специальный объект результата);
- если нужно несколько выходных значений — использовать структуру/класс/кортеж с понятными именами полей;
- оставлять `out`/`ref` только там, где это действительно оправдано (низкоуровневые API, производительность) и хорошо задокументировано.

**Пример (до/после):**

```csharp
// До: два выходных параметра, непонятно из имени, кто меняется
bool TryParseUser(string raw, out string name, out int age)
{
    // внутри меняем name и age
}

// После: явный объект результата
public record ParsedUser(string Name, int Age);

ParsedUser? TryParseUser(string raw)
{
    // возвращаем либо ParsedUser, либо null/Result
}
```

Во втором варианте:

- по сигнатуре видно, что функция ничего не меняет «снаружи» и возвращает один объект;
- проще добавить новые поля (например, Email), не меняя десятки вызовов с `out`;
- тесты становятся короче и понятнее: мы сравниваем один возвращаемый объект, а не проверяем набор изменённых переменных.

**Смешение уровней и побочных эффектов.**  
Частая ошибка — в одной функции:

- принимать HTTP‑запрос;
- валидировать тело;
- выполнять бизнес‑логику;
- управлять транзакцией БД;
- формировать HTTP‑ответ;
- логировать.

Это нарушает:

- принцип одной задачи;
- разделение команд и запросов;
- границы слоёв (часть 9).

### Граничные случаи

**1. Скрипты «на выброс» и одноразовые утилиты.**  
Если код объективно одноразовый (маленький миграционный скрипт, разовая конвертация данных), можно позволить себе больше вольностей. Но нужно честно ответить: *правда ли он одноразовый?* Скрипты часто живут дольше, чем предполагалось.

**2. Простые функции без очевидных точек разбиения.**  
Иногда большая функция — это просто:

- один `switch`/`match` по enum;
- или длинная линейная последовательность простых операций (например, парсинг формата).

Если разбиение не улучшает читаемость, а только увеличивает количество переходов — можно оставить функцию крупнее, но:

- дать ей хорошее имя;
- добавить тесты;
- документировать решение.

**3. Производительность в горячем участке.**  
Иногда разбиение на множество маленьких функций действительно вредит производительности в **hot path** (критичный участок, вызывающийся миллионы раз). В таких случаях:

- сначала измерь (профилируй);
- потом, если действительно это bottleneck, можно пожертвовать частью читаемости ради скорости, но:
  - ограничь жертву только горячим участком;
  - зафиксируй решение (комментарий/ADR);
  - напиши тесты.

### Простыми словами

Правила про функции — это **инструменты**, а не религия. В большинстве случаев они помогают сделать код понятнее и тестируемее. Но иногда есть причины их ослабить: одноразовый скрипт, горячий участок, линейная функция без естественных точек разбиения. Важно не «нарушить правило», а **осознанно его применить** или ослабить, понимая последствия.

### Картинка в голове

**Правила дорожного движения.**  
Обычно скорость 60 км/ч в городе — разумное ограничение. Но есть объездные дороги, где стоит разрешённые 80. Главное — не устраивать гонки там, где это опасно. Так же и с функциями: общее правило «делай маленькими» работает почти везде, но иногда можно чуть‑чуть отклониться, понимая, зачем.

### Как запомнить

- **Длинные функции и списки параметров** — сигналы, а не приговор: их нужно обследовать и часто — лечить.
- **Граничные случаи** допустимы, если:
  - ты понимаешь, почему отступаешь от правила;
  - ты готов зафиксировать это решение;
  - ты знаешь, чем заплатишь.

### Примеры

**Граничный случай — линейный парсер:**

```python
def parse_simple_format(line: str) -> ParsedLine:
    # длинная, но линейная последовательность шагов,
    # где разбиение не улучшает читаемость
    ...
```

Если каждый шаг тривиален, а чтение в одном месте удобнее, чем «прыгать» по функциям, это может быть приемлемо — при условии, что есть тесты и хорошее описание.

### Типичные ошибки

- Механически применять правило «всегда дели функции», раздробляя простые вещи на микрофункции без выигрыша в читаемости.
- Игнорировать запахи «длинная функция» и «длинный список параметров», прикрываясь «оптимизацией» без измерений.

### Что будет, если…

**Если относиться к правилам бездумно (либо всё делить, либо ничего не делить):**  
Можно либо перегрузить код абстракциями, либо оставить его громоздким. В обоих случаях страдает читаемость. Баланс и осознанность — ключ.

### Проверь себя (2.5)

1. В каких ситуациях допустимо оставить функцию длинной?
   <details><summary>Ответ</summary>
   Когда она представляет собой линейную последовательность простых операций без естественных точек разбиения, а разбиение только затрудняет чтение; когда есть тесты и хорошее имя; когда это не критично для команды и не нарушает читаемость.
   </details>

2. Что важно сделать, если ты осознанно нарушаешь правило (например, не делишь функцию ради производительности)?
   <details><summary>Ответ</summary>
   Зафиксировать решение (комментарий, ADR), покрыть код тестами, ограничить отклонение только критичным участком и регулярно перепроверять допущение при изменении контекста (нагрузка, инфраструктура).
   </details>

3. Почему правило «одна функция — одна задача» нельзя применять механически?
   <details><summary>Ответ</summary>
   Механическое применение может привести к раздроблению кода на микрофункции типа `doStep1`, `doStep2`, которые не повышают понятность. Важно, чтобы выделенная функция имела осмысленное имя и цель, а разбиение улучшало, а не ухудшало читаемость.
   </details>

### Запомните

- **Запахи** (длинные функции, длинные списки параметров, смешение уровней, скрытые эффекты) — повод для анализа и рефакторинга.
- **Граничные случаи** допустимы при осознанном решении и понимании последствий.
