## 2.4. Вложенность, guard‑клозы и структура

### Цель раздела

Научиться управлять вложенностью в функциях, использовать guard‑клозы для упрощения структуры, писать функции, которые читаются линейно и не превращаются в «лес из `if` и `for`».

### В этом разделе главное

- Глубокая вложенность (`if` внутри `if` и ещё цикл) затрудняет чтение и повышает риск ошибок.
- **Ранние возвращения (guard‑клозы)** позволяют обрабатывать исключительные случаи в начале и упрощать основной путь.
- Структура функции должна быть **максимально плоской**, когда это возможно.

### Термины

| Термин | Определение |
|--------|-------------|
| **Вложенность** | Количество уровней `if`/`for`/`while` и других блоков внутри друг друга |
| **Guard clause (охранное условие)** | Ранний возврат из функции при выполнении простого или исключительного условия |

### Теория и правила

**Почему глубокая вложенность вредна.**  
Когда в функции много уровней вложенности:

- сложно отслеживать, в какой ветке кода ты сейчас находишься;
- легко забыть закрывающую скобку или неправильный отступ;
- трудно увидеть «основной путь» исполнения.

**Guard‑клозы.**  
Вместо того чтобы писать:

```python
def process(order):
    if order is not None:
        if order.items:
            # длинный код
            ...
```

Лучше:

```python
def process(order):
    if order is None:
        return
    if not order.items:
        return
    # основной путь
    ...
```

Охранные условия:

- **обрабатывают особые случаи сразу**;
- уменьшают уровень вложенности для основного сценария;
- делают функцию линейной.

**Лимит вложенности.**  
Жёсткого числа нет, но ориентир:

- 1–2 уровня вложенности — обычно нормально;
- 3 — на грани;
- 4+ — почти всегда признак того, что нужно рефакторить (выносить части в отдельные функции, использовать guard‑клозы, упрощать условия).

### Пошагово: как уменьшить вложенность

1. Найди самый глубокий участок кода (4+ уровней вложенности).
2. Посмотри, можно ли **заменить внутренний `if` на ранний `return`**.
3. Попробуй **вынести внутренний блок** в отдельную функцию с понятным именем.
4. Проверь, можно ли **объединить условия** или переписать их в более простую форму.

**Проверь себя (уменьшение вложенности)**

1. Почему guard‑клозы часто делают код короче и понятнее, даже если добавляют несколько `return`?
   <details><summary>Ответ</summary>
   Они выносят обработку исключительных или простых случаев наверх и позволяют основному потоку идти без дополнительной вложенности. В результате уменьшается «лес» скобок и отступов, а основная логика становится более линейной и очевидной.
   </details>

2. Есть ли риск злоупотребить guard‑клозами?
   <details><summary>Ответ</summary>
   Если охранные условия становятся слишком многочисленными и разбросанными, функция может превратиться в серию ранних выходов без ясного основного пути. Важно использовать guard‑клозы для явных «ранних выходов» и при этом сохранять понятный основной блок логики.
   </details>

### Простыми словами

Каждый уровень вложенности — как ещё один уровень отступа в книге: цитата внутри цитаты внутри комментария. Через пару уровней ты забываешь, кто кому отвечает. Guard‑клозы помогают «выкинуть лишних персонажей» сразу и оставить на сцене только главную линию сюжета.

### Картинка в голове

Представь **дерево с ветками**, на концах которых висят ещё несколько веточек, и так несколько уровней. Пройтись по такому дереву сложно. Если обрезать ненужные ветки в самом начале, остаётся аккуратный ствол с небольшим количеством ответвлений — по нему легко проходить.

### Как запомнить

- **Глубокая вложенность = сложное чтение.** Используй guard‑клозы и извлечение функций.
- **Особые случаи — наверх, основной сценарий — ниже.**

### Примеры

**До (глубокая вложенность):**

```python
def process_orders(orders):
    for order in orders:
        if order is not None:
            if order.items:
                if not order.cancelled:
                    # длинная логика
                    handle_order(order)
```

**После (guard‑клозы и отдельная функция):**

```python
def process_orders(orders):
    for order in orders:
        if not should_process(order):
            continue
        handle_order(order)


def should_process(order):
    if order is None:
        return False
    if not order.items:
        return False
    if order.cancelled:
        return False
    return True
```

### Практический сценарий

В legacy‑коде часто встречаются функции с кучей вложенных условий:

```python
if user is not None:
    if user.is_active:
        if user.role == "admin":
            # что‑то делаем
```

После рефакторинга:

```python
if not can_perform_action(user):
    return
# основной путь
```

Функция `can_perform_action` становится явным местом для проверки прав и состояния, её легко тестировать отдельно.

### Типичные ошибки

- Бояться множества `return` и всегда тянуть к одному выходу, в результате получая огромный `if/else` с вложенностями.
- Пихать всё внутрь одного большого цикла с кучей условий вместо того, чтобы вынести проверки.

### Что будет, если…

**Если не ограничивать вложенность:**  
Функции превращаются в «лес» из `if`/`for`, в котором тяжело ориентироваться. Любая правка в середине этого леса рискованна, а тесты покрывают не все ветки.

### Проверь себя (2.4)

1. Почему множество ранних `return` в функции не обязательно плохо?
   <details><summary>Ответ</summary>
   Если они используются для обработки простых и исключительных случаев в начале функции, а основной путь остаётся линейным, множество ранних `return` улучшает читаемость. Плохо, когда они разбросаны хаотично и мешают видеть основной поток.
   </details>

2. Как определить, что вложенность уже «слишком глубокая»?
   <details><summary>Ответ</summary>
   Если при чтении функции приходится считать уровни отступов, а IDE сложно показать весь блок без прокрутки, вложенность, скорее всего, чрезмерна. Практически это 3–4 и более вложенных уровней, особенно если внутри есть существенная логика.
   </details>

3. В каких случаях `continue` в цикле играет роль guard‑клозы?
   <details><summary>Ответ</summary>
   Когда он используется в начале итерации, чтобы пропустить неинтересные элементы (`if not valid: continue`) и тем самым оставить основной код цикла без дополнительного уровня вложенности. Это аналог раннего `return`, но для одной итерации цикла.
   </details>

### Запомните

- Ограничивай **глубину вложенности**; используй **guard‑клозы** и **извлечение функций**, чтобы выпрямлять код.
- Особые случаи — в начале, основной сценарий — ниже, с минимальной вложенностью.
