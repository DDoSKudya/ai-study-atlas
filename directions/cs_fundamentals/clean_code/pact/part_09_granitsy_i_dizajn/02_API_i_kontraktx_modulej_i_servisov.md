## 9.2. API и контракты модулей и сервисов

### Цель раздела

Научиться смотреть на свой код как на **API для других** (модулей, сервисов, команд в проекте), формулировать чёткие контракты (типы, инварианты, ошибки) и использовать контрактное тестирование, чтобы эти обещания не нарушались при изменениях.

### В этом разделе главное

- Любой публичный класс/функция/сервис — это **API** с контрактом.
- Контракт включает:
  - **типы параметров и возвращаемых значений**;
  - **инварианты** (что всегда верно);
  - **ошибки и исключения**;
  - иногда — **ограничения производительности** или прочие нефункциональные свойства.
- Контракты можно и нужно **фиксировать и проверять**:
  - документацией (OpenAPI, комментарии, README);
  - **контрактными тестами** (в т.ч. consumer‑driven).

### Термины

| Термин | Определение |
|--------|-------------|
| **API** | Набор доступных операций и структур данных, предоставляемых модулем/сервисом |
| **Контракт** | Совокупность обещаний API о входах, выходах и ошибках |
| **Инвариант** | Условие, которое всегда должно быть истинным для определённого состояния/объекта |
| **Consumer‑driven contracts** | Подход, при котором потребители API формулируют свои ожидания в виде тестов/спецификаций, а провайдер обязан им соответствовать |

### Теория и правила

1. **Публичное → контракт.**

   Всё, что помечено как `public`/экспортируется из модуля, становится контрактом:
   - клиенты зависят от **сигнатур**;
   - от **поведения** (семантики, инвариантов);
   - от **ошибок**, которые можно ожидать.

   Меняя это без согласования, мы ломаем клиентов.

2. **Из чего состоит контракт.**

   - **Сигнатура:**
     - имена методов;
     - типы аргументов;
     - типы возвращаемых значений.
   - **Инварианты и пред-/пост‑условия:**
     - «метод никогда не возвращает `null`»;
     - «список всегда отсортирован»;
     - «сумма всегда ≥ 0».
   - **Ошибки:**
     - какие исключения/коды/Result‑типы могут вернуться;
     - в каких ситуациях.
   - **Контекст:**
     - идемпотентность (можно ли вызывать повторно без изменения результата);
     - требования к порядку вызовов.

3. **Документация контрактов.**

   - Для **HTTP API**:
     - OpenAPI/Swagger, JSON Schema, AsyncAPI;
     - примеры запросов/ответов;
     - коды ответов, ошибки.
   - Для **библиотек и модулей**:
     - docstring’и и Javadoc‑подобные комментарии;
     - README с разделом «Public API»;
     - описания инвариантов и ошибок.

4. **Контрактные тесты.**

   - **Consumer‑driven**:
     - потребитель формулирует сценарии и ожидаемые ответы в виде спецификаций;
     - провайдер должен проходить эти тесты;
     - пример инструментов: Pact, Spring Cloud Contract.
   - **Стабильность контракта**:
     - прежде чем менять контракт, нужно:
       - обновить спецификацию;
       - синхронизироваться с потребителями;
       - при необходимости ввести новую версию.

### Пошагово: как сформулировать контракт для модуля

Допустим, у тебя есть модуль `BillingService` с методом:

```python
def charge(user_id: str, amount: Decimal) -> bool:
    ...
```

**Шаг 1. Проясни поведение.**

- Что значит `True`/`False`?
- Выбрасывает ли метод исключения?
- Какие случаи считаются «нормальными», а какие — ошибками?

**Шаг 2. Перепроектируй сигнатуру под контракт.**

Например, вместо булевого вернуть Result‑тип:

```python
class ChargeError(Enum):
    INSUFFICIENT_FUNDS = "insufficient_funds"
    CARD_EXPIRED = "card_expired"
    TECHNICAL_ERROR = "technical_error"


@dataclass
class ChargeResult:
    success: bool
    error: ChargeError | None = None
```

Тогда контракт явно говорит:

```python
def charge(user_id: str, amount: Decimal) -> ChargeResult:
    ...
```

**Шаг 3. Задокументируй инварианты и ошибки.**

- `amount` > 0;
- при `success=True` деньги действительно списаны;
- при `error=TECHNICAL_ERROR` можно повторить операцию позже.

**Шаг 4. Покрой контракт тестами.**

- тесты на успех;
- тесты на разные ошибки;
- тесты на граничные случаи.

### Практический сценарий

Представь, что ты проектируешь внутренний HTTP‑сервис `billing-service`, к которому будут обращаться другие команды. Наивный вариант — просто «повесить» контроллер поверх текущего кода и вернуть, скажем, `200` или `500` в зависимости от того, кинула ли исключение библиотека провайдера. На практике такой API неудобен:

- клиенты не понимают, **почему** списание не прошло;
- нет явной разницы между «денег не хватило» и «упал внешний провайдер»;
- любая смена библиотеки платёжного провайдера «просачивается» в HTTP‑слой.

Более чистый подход:

1. Сначала сформулировать **доменный контракт**: `ChargeResult` (успех/ошибка с типом ошибки).
2. На основе доменного контракта спроектировать HTTP‑API:
   - `200 OK` с телом `{ "success": true }` для успешного списания;
   - `400 Bad Request` с телом `{ "error": "insufficient_funds" }` для бизнес‑ошибок пользователя;
   - `502 Bad Gateway` или `503 Service Unavailable` с `{ "error": "technical_error" }` для проблем на стороне провайдера.
3. Зафиксировать это в OpenAPI и в тестах:
   - контрактные тесты от имени потребителей (например, отдельный тестовый проект);
   - интеграционные тесты самого `billing-service`, проверяющие соответствие реализации OpenAPI.

Так ты отделяешь:

- **внутренние детали** (какой именно SDK использован) — скрыты внутри адаптеров;
- **доменный контракт** (`ChargeResult`);
- **внешний HTTP‑контракт**, на который завязаны клиенты.

**Проверь себя (контракт мышлением)**

1. Почему `bool` как результат часто плохой контракт для сложной операции?
   <details><summary>Ответ</summary>
   Булевый результат не говорит, **почему** операция не удалась. Клиент не знает, что делать дальше (повторить? показать конкретное сообщение? перейти в другой сценарий?). Лучше явный Result‑тип или исключения с понятным смыслом.
   </details>

2. Как понять, какие инварианты стоит явно описать в контракте?
   <details><summary>Ответ</summary>
   Смотри на предположения, которые делает вызывающий код: «метод никогда не вернёт пустой список», «ID всегда уникален», «объект всегда в статусе X после вызова». Всё, что важно для правильной работы клиента, стоит зафиксировать в контракте и, по возможности, проверить в тестах.
   </details>

3. Чем контрактные тесты отличаются от обычных интеграционных?
   <details><summary>Ответ</summary>
   Интеграционный тест — это проверка работы нескольких компонентов вместе под конкретным сценарием. Контрактный тест фокусируется **на соответствию заявленному контракту** API/сервиса: он ближе к «спецификации в виде тестов», которую должны соблюдать все реализации или версии.
   </details>

### Простыми словами

Контракт — это как «договор аренды»: там не только написано, сколько платить, но и:

- кто за что отвечает;
- что будет при нарушении условий;
- какие вещи никогда не допускаются.

Если ты меняешь договор в одностороннем порядке (например, вдруг повышаешь аренду в два раза без уведомления), арендатору будет неприятно. С API то же самое: **клиенты привязаны к твоему контракту**, и любое изменение без согласования — «ломание людей».

### Картинка в голове

Представь, что твой модуль — это **магазин**:

- Витрина и касса — это API.
- Условия возврата, гарантия, время работы — это контракт.

Если сегодня ты работаешь до 20:00, а завтра **вдруг** закрываешься в 18:00 без предупреждения — клиенты придут и уткнутся в закрытые двери. Так же ощущается breaking change в API без смены версии и без документации.

### Как запомнить

- Всё, что `public`/экспортируется, — **твой API**, а не «деталь реализации».
- Контракт = **сигнатура + инварианты + ошибки + контекст (идемпотентность и порядок вызовов)**.
- Контракты нужно **явно фиксировать** (документация, схемы, тесты), а не держать в голове.

### Типичные ошибки

- Считать, что «контракт — это только типы, остальное понятно из кода».
- Менять возвращаемые значения или поведение без обновления документации и координации с потребителями.
- Использовать слишком общие исключения или `bool`/`null` там, где нужен явный результат.

### Что будет, если…

**Если не относиться к своему коду как к API:**

- внешние и внутренние клиенты будут угадывать поведение;
- любое изменение может сломать пол‑системы неожиданным образом;
- обсуждения сводятся к «ну я же не обещал» вместо «мы нарушили контракт, нужно аккуратное изменение».

### Проверь себя (9.2)

1. Что входит в контракт метода, помимо типов аргументов и возвращаемого значения?
   <details><summary>Ответ</summary>
   Инварианты (что всегда верно до/после вызова), список возможных ошибок/исключений и их семантика, требования к порядку вызовов и идемпотентность, а также иногда нефункциональные свойства (например, «не блокирует больше N секунд»).
   </details>

2. Чем плоха ситуация, когда контракт API существует только «в головах» разработчиков?
   <details><summary>Ответ</summary>
   Новые люди не могут быстро понять правила, разные члены команды помнят разные детали, а изменения легко нарушают негласные ожидания. Ошибки выявляются в проде, а споры в стиле «мы так не договаривались» становятся нормой.
   </details>

3. Почему полезно описывать контракты не только для HTTP‑API, но и для внутренних библиотек/модулей?
   <details><summary>Ответ</summary>
   Внутренние модули тоже имеют потребителей (другие команды, сервисы, будущие разработчики). Чёткие контракты уменьшают количество случайных зависимостей, делают рефакторинг менее рискованным и помогают использовать модуль правильно без чтения всей реализации.
   </details>

### Запомните

- **API без контракта** — это лотерея.
- Контракт — это не только типы, но и **обещания о поведении и ошибках**.
- Контракты можно и нужно **проверять тестами**, а не только словами и документацией.
