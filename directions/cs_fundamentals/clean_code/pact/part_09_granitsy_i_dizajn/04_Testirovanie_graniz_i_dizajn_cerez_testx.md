## 9.4. Тестирование границ и дизайн через тесты

### Цель раздела

Показать, как **тесты на границах** (интеграционные, контрактные, TDD‑сценарии) помогают формировать дизайн API и адаптеров, и как использовать их как «страховку» при изменении внешних зависимостей.

### В этом разделе главное

- Юнит‑тесты проверяют логику **внутри**; для границ нужны **интеграционные и контрактные тесты**.
- TDD и consumer‑driven подход позволяют **отрепетировать интерфейсы** до окончательной реализации.
- Хорошие тесты на границах:
  - изолируют изменчивость внешнего мира;
  - чётко фиксируют контракт.

### Термины

| Термин | Определение |
|--------|-------------|
| **Интеграционный тест** | Тест, проверяющий взаимодействие нескольких компонентов, часто с реальными внешними зависимостями (БД, API) |
| **Contract test** | Тест, проверяющий соответствие реализации заявленному контракту (спецификации) |
| **TDD (Test‑Driven Development)** | Подход «сначала тест, потом реализация, потом рефакторинг» |
| **Seam (шов)** | Место, куда можно «вшить» подмену зависимостей для тестирования |

### Теория и правила

1. **Роль тестов на границах.**

   - Юнит‑тесты проверяют, что логика **внутри** модуля корректна.
   - Но они не отвечают на вопросы:
     - правильно ли мы **формируем запросы** к внешнему API;
     - корректно ли **переводим ошибки** стороннего кода в наши исключения/Result‑типы;
     - соблюдаем ли мы **контракт версий**.

   Это зона интеграционных и контрактных тестов.

2. **Интеграционные тесты адаптеров.**

   - Поднимаем (или используем) **реальный сервис/тестовый стенд**.
   - Запускаем наш адаптер:
     - с реальным HTTP‑клиентом/драйвером БД;
     - на тестовых данных.
   - Проверяем:
     - формат запросов;
     - интерпретацию ответов и ошибок;
     - корректность обработки граничных случаев (таймауты, 4xx/5xx).

3. **Contract tests/consumer‑driven подход.**

   - Потребитель API описывает свои ожидания:
     - «при таком запросе должен быть такой ответ»;
     - «это поле обязательно», «это опционально».
   - Эти ожидания превращаются в **машиночитаемую спецификацию** или набор тестов.
   - Провайдер обязан:
     - проходить эти тесты;
     - не ломать их при обновлениях.

4. **TDD для границ.**

   - Сначала пишем тест, который использует **наш желаемый интерфейс адаптера/сервиса**:

```python
def test_can_fetch_user_balance_by_id(user_client: UserClient):
    balance = user_client.get_balance("user-123")

    assert balance.currency == "USD"
    assert balance.amount > 0
```

   - Тест формулирует **какой интерфейс нам нужен**:
     - метод `get_balance`;
     - тип `Balance` с `currency` и `amount`.
   - Только потом мы реализуем адаптер и внутреннюю работу с внешним API.

### Пошагово: как написать тест на границу

Пример: адаптер к REST‑API пользователей.

1. Выбери **одну операцию**, например «получить пользователя по id».
2. Открой документацию API (OpenAPI/Swagger) и **зафиксируй пример**:
   - URL;
   - тело ответа;
   - коды ошибок.
3. Напиши интеграционный тест:

```python
def test_user_client_returns_domain_user(http_user_client: HttpUserClient, test_server):
    # Arrange: на тестовом сервере есть пользователь user-123
    user_id = "user-123"

    # Act
    user = http_user_client.get_user(user_id)

    # Assert
    assert user.id == user_id
    assert user.email.endswith("@example.com")
```

4. Добавь тест на ошибку:

```python
def test_user_client_raises_not_found_for_unknown_user(http_user_client: HttpUserClient, test_server):
    with pytest.raises(UserNotFoundError):
        http_user_client.get_user("non-existent")
```

5. Теперь можешь смело рефакторить адаптер или обновлять библиотеку HTTP, пока эти тесты зелёные.

**Проверь себя (тесты на границах)**

1. Почему важно иметь хотя бы несколько интеграционных тестов строго против адаптеров, даже если есть end‑to‑end тесты?
   <details><summary>Ответ</summary>
   End‑to‑end тесты проверяют всё приложение целиком и часто медленные и хрупкие. Тесты против адаптеров дают более **точный сигнал**, что именно сломалось в интеграции с внешним API/БД, и проще отлаживаются. Они дешевле по времени и ресурсу и лучше подходят для частого запуска.
   </details>

2. Чем seam‑подход помогает в тестировании границ?
   <details><summary>Ответ</summary>
   Seam — это место, куда можно «подшить» фейковую реализацию (фейковый адаптер, in‑memory БД, мок‑HTTP‑клиент). Благодаря seams мы можем тестировать доменный код **без реальных внешних зависимостей**, а адаптеры — отдельными интеграционными тестами, сохраняя чистоту дизайна.
   </details>

3. Как TDD‑подход помогает улучшать дизайн API адаптеров?
   <details><summary>Ответ</summary>
   Пишем тест, использующий адаптер как **клиент** хотел бы его видеть. Это заставляет сразу думать о удобстве и ясности интерфейса (методы, типы, имена), а реализацию подстраивать под этот интерфейс, а не наоборот. В итоге получается API, удобный для потребителей, а не для библиотеки.
   </details>

### Простыми словами

Тесты на границах — это как **тест‑драйв для дверей и окон дома**: ты проверяешь, что дверь открывается/закрывается, не заклинивает, не сквозит, что ключ подходит. Юнит‑тесты — это проверка качества комнаты (ровные стены, розетки), а интеграционные и контрактные тесты — проверка того, как дом стыкуется с улицей.

### Картинка в голове

Представь, что у тебя есть **мост через реку** (граница между берегами). Юнит‑тесты проверяют, что бетон прочный, балки не ржавые. Интеграционные/контрактные тесты проверяют, что:

- мост действительно соединяет нужные берега;
- выдерживает нужный вес;
- имеет нужное количество полос.

Нельзя ограничиться только проверкой бетона, игнорируя тот факт, что мост может упираться не в тот берег.

### Как запомнить

- Границы без тестов → сложно и страшно их менять.
- Интеграционные тесты + seams → устойчивый дизайн, который легче эволюционирует.
- TDD на границах → интерфейсы, удобные для потребителя, а не для технической реализации.

### Типичные ошибки

- Полагаться только на end‑to‑end тесты для проверки интеграции (медленно, хрупко, трудная локализация проблем).
- Писать юнит‑тесты только на внутренние функции и игнорировать тесты против адаптеров.
- Не иметь швов (seams): жёстко зашивать реальные клиенты в доменный код, усложняя подмену в тестах.

### Что будет, если…

**Если не тестировать границы явно:**

- любые изменения библиотек и протоколов будут давать «случайные» баги в проде;
- будет сложно понять, сломался ли внешний сервис или наш адаптер;
- команда начнёт бояться обновлений зависимостей и сторонних API.

### Проверь себя (9.4)

1. Чем отличается интеграционный тест адаптера от юнит‑теста доменной логики?
   <details><summary>Ответ</summary>
   Интеграционный тест адаптера использует реальные внешние зависимости (БД, HTTP‑сервер, очередь) и проверяет, что адаптер правильно формирует запросы и обрабатывает ответы/ошибки. Юнит‑тест доменной логики работает с фейковыми/мок‑адаптерами и проверяет, что внутренняя бизнес‑логика правильно реагирует на данные, не трогая внешние системы.
   </details>

2. Зачем нужны contract tests, если уже есть OpenAPI/Swagger?
   <details><summary>Ответ</summary>
   Спецификация — это **документ‑обещание**. Contract tests — это **исполняемая проверка** соответствия кода этой спецификации. Они помогают убедиться, что и провайдер, и потребители действительно реализуют и используют API в соответствии с документом, а не просто «считают, что так».
   </details>

3. Почему без seam‑подхода тесты часто становятся медленными и хрупкими?
   <details><summary>Ответ</summary>
   Если везде зашиты реальные клиенты/соединения, каждый тест влечёт за собой реальные HTTP‑запросы, соединения с БД и внешними сервисами. Такие тесты:
   - медленные;
   - зависят от окружения;
   - часто флапают (дают разные результаты).
   Seam‑подход позволяет подменять зависимости, делая большинство тестов быстрыми и устойчивыми.
   </details>

### Запомните

- Хороший дизайн границ **обязан** сопровождаться тестами на этих границах.
- Юнит‑тесты ≠ интеграционные ≠ контрактные — у каждого свой уровень и цель.
- TDD на уровне адаптеров помогает сразу строить удобные и тестируемые интерфейсы.
