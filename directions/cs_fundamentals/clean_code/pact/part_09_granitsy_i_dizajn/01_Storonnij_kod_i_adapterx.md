## 9.1. Сторонний код и адаптеры

### Цель раздела

Понять, почему прямое использование сторонних библиотек и внешних API по всему проекту делает код хрупким, как оборачивать внешний код в собственные интерфейсы (адаптеры, фасады) и как тестировать такие границы.

### В этом разделе главное

- Любая внешняя библиотека или сервис — это **нестабильность**, которую полезно изолировать.
- **Адаптеры и фасады** позволяют:
  - спрятать «уродливый» или шумный API;
  - сконцентрировать зависимость от стороннего кода в одном месте;
  - облегчить тестирование (можно подменить адаптер фейком или мок‑объектом).
- Граница должна иметь **чёткий контракт**: какие входы, какие выходы, какие ошибки.

### Термины

| Термин | Определение |
|--------|-------------|
| **Boundary (граница)** | Место, где наш код взаимодействует с внешней системой или библиотекой |
| **Adapter** | Класс, который «подстраивает» внешний интерфейс под ожидаемый нашим кодом |
| **Facade** | Упрощённый интерфейс к сложной подсистеме или набору библиотек |
| **Port/Adapter (Hexagonal Architecture)** | Порт — наш абстрактный интерфейс; адаптер — конкретная реализация под технологию |

### Теория и правила

1. **Сторонний код нестабилен по определению.**  
   Мы не контролируем:
   - сроки и содержание обновлений;
   - поведение в редких ситуациях (ошибки сети, таймауты, странные ответы);
   - качество документации и обратную совместимость.

   Если такой код «растекается» по проекту, любое изменение (например, переход c одной библиотеки HTTP на другую) требует массовых правок.

2. **Зачем оборачивать библиотеку в адаптер.**

   - Скрыть неудобные детали:
     - странные типы исключений;
     - низкоуровневый протокол (например, сырой JSON);
     - лишние параметры и флаги.
   - Связать внешний API с **нашими доменными понятиями**:
     - вместо `requests.get(url)` — `UserApiClient.get_user(user_id)`;
     - вместо «сырого» SQL — `UserRepository.find_by_email(email)`.
   - Получить **одну точку входа** для всех внешних вызовов, где:
     - настраиваются таймауты, ретраи, логирование;
     - конвертируются ошибки в доменные исключения/Result‑типы.

3. **Граница как слой.**

   Удобный паттерн — **порт/адаптер**:

   - **Порт (интерфейс)** — наш абстрактный контракт, к которому привязан доменный код:
     - `UserRepository` с методами `get_by_id`, `save`, `find_by_email`.
   - **Адаптер (реализация)** — конкретный способ обращения к внешнему миру:
     - `PostgresUserRepository`, `HttpUserRepository`, `InMemoryUserRepository`.

   В домене мы работаем только с портом, реализация подсовывается через DI/конфигурацию.

4. **Тесты на границе.**

   - Для DOMЕНА:
     - подменяем адаптер фейком или мок‑объектом;
     - тесты не зависят от конкретной библиотеки/БД.
   - Для ГРАНИЦЫ:
     - отдельно пишем интеграционные тесты, которые:
       - вызывают адаптер;
       - общаются с реальной БД или тестовым стендом API;
       - проверяют, что мы правильно используем сторонний API.

### Пошагово: как выделить адаптер вокруг библиотеки

Представь, что в проекте везде используется прямой вызов `requests`:

```python
import requests

def get_user_name(user_id: str) -> str:
    response = requests.get(f"https://example.com/users/{user_id}")
    response.raise_for_status()
    return response.json()["name"]
```

И таких вызовов — десятки по всему коду.

**Шаги рефакторинга:**

1. **Создай интерфейс (порт)** — наш абстрактный клиент:

```python
class UserClient:
    def get_user_name(self, user_id: str) -> str:
        raise NotImplementedError
```

2. **Создай конкретный адаптер**, использующий библиотеку:

```python
import requests


class HttpUserClient(UserClient):
    BASE_URL = "https://example.com"

    def get_user_name(self, user_id: str) -> str:
        response = requests.get(f"{self.BASE_URL}/users/{user_id}", timeout=5)
        response.raise_for_status()
        data = response.json()
        return data["name"]
```

3. **Заменяй прямые вызовы** на использование порта:

```python
def greet_user(user_client: UserClient, user_id: str) -> str:
    name = user_client.get_user_name(user_id)
    return f"Hello, {name}!"
```

4. В **тестах домена** используй фейк:

```python
class FakeUserClient(UserClient):
    def __init__(self, names_by_id: dict[str, str]):
        self._names_by_id = names_by_id

    def get_user_name(self, user_id: str) -> str:
        return self._names_by_id[user_id]
```

5. Отдельно напиши **интеграционный тест** для `HttpUserClient`, который ходит на тестовый API или проверяет работу с реальным сервером.

**Проверь себя (адаптеры)**

1. Почему в доменном коде лучше работать с `UserClient`, а не с `requests` напрямую?
   <details><summary>Ответ</summary>
   `UserClient` отражает **доменные операции** (получить пользователя по id) и не раскрывает деталей HTTP‑протокола, URL, формата JSON и библиотеки. Благодаря этому доменный код легче читать, тестировать (можно подменить реализацию), а при смене библиотеки или протокола менять нужно только адаптер.
   </details>

2. В чём преимущество тестов против адаптера по сравнению с тестами «поверх всего приложения»?
   <details><summary>Ответ</summary>
   Тесты против адаптера проверяют **конкретную границу**: что мы правильно формируем запрос, обрабатываем ошибки и парсим ответ. Они меньше, быстрее и локальнее, чем end‑to‑end тесты, и дают более точный сигнал, где именно что‑то сломалось (на границе или в домене).
   </details>

3. Чем фасад отличается от адаптера?
   <details><summary>Ответ</summary>
   Адаптер обычно «переводит» интерфейс одной конкретной внешней системы в наш интерфейс. Фасад может объединять несколько подсистем (несколько библиотек, модулей) и предоставлять поверх них **упрощённый** интерфейс. Адаптер — про «совместимость интерфейсов», фасад — про «упрощение и сокрытие сложности».
   </details>

### Простыми словами

Сторонняя библиотека — как розетка другого стандарта: она может работать, но штекер не подходит, напряжение и контакты другие. Адаптер — это «переходник», который позволяет аккуратно подключиться, не меняя всю проводку в квартире. Если ты везде режешь проводку и скручиваешь напрямую — каждый ремонт становится кошмаром; с переходником меняешь один элемент — и всё.

### Картинка в голове

Представь, что у тебя дома **несколько разных зарядок**: USB‑C, micro‑USB, старый разъём ноутбука. Можно:

- каждый раз обрезать провод и припаивать новый штекер (аналог — менять код везде, где используется библиотека);
- один раз купить **универсальный переходник** и всегда подключаться через него (аналог — адаптер/фасад).

Долгосрочно второй вариант всегда дешевле и безопаснее.

### Как запомнить

- **Сторонний код = нестабильность → изолировать.**
- **Адаптер = переходник между нашим доменом и внешним API.**
- **Домен видит порт (интерфейс), а не конкретную библиотеку.**

### Типичные ошибки

- Подключать библиотеку напрямую в десятках мест без обёртки.
- Протаскивать типы/исключения библиотеки до самого UI (например, `HttpException` в доменном слое).
- Не иметь единого места, где настраиваются таймауты/ретраи, и копипастить их по всему коду.

### Что будет, если…

**Если не изолировать сторонний код:**

- смена библиотеки превратится в «mini‑migration» по всей кодовой базе;
- будет сложно писать тесты без реальной БД/сервиса;
- любые странности поведения библиотеки (таймауты, нестабильные ответы) начнут «просачиваться» в доменную логику.

### Проверь себя (9.1)

1. Зачем выносить работу со сторонней библиотекой в отдельный адаптер, если её API и так «вроде бы удобный»?
   <details><summary>Ответ</summary>
   Сегодня API может казаться удобным, но завтра меняется версия, добавляются обязательные параметры, меняется формат ошибок. Адаптер даёт **точку изоляции**: даже если библиотека меняется, доменный код и остальные слои остаются нетронутыми. Плюс адаптер позволяет говорить на **языке домена**, а не на языке библиотеки.
   </details>

2. Как адаптер помогает при тестировании?
   <details><summary>Ответ</summary>
   Мы можем подменить адаптер на фейк/мок в тестах доменного слоя, не поднимая реальные внешние сервисы. Это делает тесты быстрее, стабильнее и независимее. Для проверки правильности интеграции нужен только небольшой набор интеграционных тестов против самого адаптера.
   </details>

3. Чем плоха ситуация, когда `requests`/`axios`/`psycopg` напрямую используются внутри доменных сущностей?
   <details><summary>Ответ</summary>
   Доменные сущности начинают знать о технических деталях (HTTP, SQL), что:
   - нарушает разделение слоёв;
   - усложняет тестирование (нужен реальный HTTP/БД);
   - увеличивает связанность кода с конкретной библиотекой и делает замену/обновление дорогостоящими.
   </details>

### Запомните

- Любой внешний API/библиотека должны быть **ограничены адаптерами**; домен с ними напрямую не общается.
- На границе мы:
  - конвертируем типы и ошибки;
  - настраиваем политику таймаутов/ретраев/логирования;
  - формируем **свой** контракт, удобный для остальной системы.
