## 9.3. Эмерджентный дизайн и YAGNI

### Цель раздела

Понять, как дизайн системы может **постепенно «вырастать»** из кода под тестами и рефакторинга, а не быть полностью спроектированным заранее, и как применять принципы YAGNI и правило трёх на уровне дизайна модулей и границ.

### В этом разделе главное

- «Идеальную архитектуру» редко удаётся спроектировать заранее — требования меняются.
- Эмерджентный дизайн опирается на:
  - **тесты** (в т.ч. на границах);
  - **устранение дублирования**;
  - **маленькие функции и классы**;
  - **рефакторинг под правило трёх**.
- YAGNI (You Aren’t Gonna Need It) помогает **не строить лишних абстракций и слоёв** там, где ещё нет реальной вариативности.

### Термины

| Термин | Определение |
|--------|-------------|
| **Эмерджентный дизайн** | Дизайн, который формируется постепенно через итерации «тест → реализация → рефакторинг» |
| **YAGNI** | Принцип «тебе это не понадобится» — не реализовывать функциональность и абстракции, пока они реально не нужны |
| **Rule of Three (правило трёх)** | Решение обобщать и выносить абстракцию после третьего повторения |

### Теория и правила

1. **Почему «большой дизайн заранее» часто не работает.**

   - Требования меняются;
   - новые технические ограничения появляются по ходу;
   - «на бумаге» легко спроектировать избыточную сложность.

   В итоге команда тратит много времени на архитектуру, которую потом всё равно приходится переделывать.

2. **Контур эмерджентного дизайна.**

   Типичный цикл:

   1. Пишем **тест** (или хотя бы прогоняем сценарий в голове).
   2. Делаем **минимальную реализацию**, чтобы тест прошёл.
   3. Видим **дублирование, длинные функции, смешение абстракций**.
   4. Делаем **рефакторинг**:
      - Extract Method/Class;
      - введение порта/адаптера;
      - упрощение контрактов.
   5. Повторяем, пока дизайн не станет достаточно чистым для текущих задач.

3. **YAGNI и правило трёх на уровне дизайна.**

   - Не вводи абстракцию «на будущее», если:
     - сейчас есть только один потребитель;
     - неясны реальные варианты использования.
   - При втором похожем случае можно временно продублировать решение.
   - При **третьем**:
     - выделяешь общий интерфейс/базовый класс;
     - выносишь общую реализацию;
     - оформляешь это как часть дизайна.

4. **Связь с границами.**

   - Не надо сразу строить «идеальную» архитектуру с десятком уровней портов и адаптеров.
   - Достаточно:
     - сначала аккуратно обернуть самый опасный внешний API;
     - потом, по мере появления новых потребителей/интеграций, выделять дополнительные уровни абстракции.

### Пошагово: пример эмерджентного дизайна границы

Сценарий: сначала у нас только HTTP‑клиент к платежному провайдеру.

1. **Шаг 1: простой клиент.**

```python
class PaymentClient:
    def charge(self, user_id: str, amount: Decimal) -> ChargeResult:
        ...
```

2. **Шаг 2: появляется второй провайдер.**

Мы начали копипастить код, понимаем, что добавляется второй способ оплатить.

3. **Шаг 3: правило трёх → абстракция.**

Вводим интерфейс:

```python
class PaymentProvider(Protocol):
    def charge(self, user_id: str, amount: Decimal) -> ChargeResult:
        ...
```

И реализации:

```python
class StripePaymentProvider(PaymentProvider):
    ...


class PaypalPaymentProvider(PaymentProvider):
    ...
```

Доменный код видит только `PaymentProvider`, а конкретную реализацию подсовывает конфигурация.

**Проверь себя (эмерджентный дизайн)**

1. Почему вводить интерфейс `PaymentProvider` сразу, ещё до появления второго провайдера, — не всегда хорошая идея?
   <details><summary>Ответ</summary>
   Пока есть только один провайдер, мы не знаем, какие **общие части** у разных реализаций и какие требования появятся позже. Легко придумать абстракцию, которая плохо подходит для второго провайдера, и тогда придётся менять и абстракцию, и первую реализацию. Лучше сначала сделать простой код, а при появлении второго реального кейса — спроектировать абстракцию «по фактам».
   </details>

2. Как тесты помогают эмерджентному дизайну?
   <details><summary>Ответ</summary>
   Тесты фиксируют поведение и дают «страховку» при изменении структуры. Это позволяет смело вытаскивать обёртки, интерфейсы и новые классы, не боясь сломать функциональность. Кроме того, при TDD сам тест как первый потребитель формирует требования к API и дизайну.
   </details>

3. Чем отличается «эмерджентный дизайн» от «хаотичных изменений на глаз»?
   <details><summary>Ответ</summary>
   Эмерджентный дизайн опирается на **принципы** (тесты, устранение дублирования, маленькие функции/классы, SRP, правило трёх) и систематический рефакторинг. Хаотичные изменения — это правки «как придётся» без тестов и без явных критериев, когда код со временем становится всё грязнее.
   </details>

### Простыми словами

Эмерджентный дизайн — это как **наращивание мышц**: ты не можешь разом стать сильным, но можешь регулярно тренироваться и немного улучшаться. Каждая новая фича и каждый рефакторинг — это микро‑шаг. Через месяц‑два структура становится гораздо аккуратнее, чем если бы ты пытался «один раз сделать идеальный дизайн», а потом не трогал.

### Картинка в голове

Представь, что ты строишь **сад**:

- сначала сажаешь несколько деревьев и дорожку;
- смотришь, где люди реально ходят, где нужна тень;
- потом добавляешь скамейки, фонари, новые дорожки.

Если сразу проложить все дорожки «по линейке», но люди начнут ходить по другим траекториям, ты получишь вытоптанные тропинки мимо официальных дорожек. Эмерджентный дизайн — это когда ты **подстраиваешь дорожки под реальные траектории**, а не наоборот.

### Как запомнить

- **Сначала простое решение + тесты.**
- **Повторился третий раз → обобщаем.**
- **Архитектура должна расти вместе с требованиями**, а не быть монолитным планом «навсегда».

### Типичные ошибки

- Пытаться заранее спроектировать все границы и абстракции, не имея реальных сценариев.
- Бесконечно откладывать рефакторинг «на потом», в результате дизайн никогда не улучшится.
- При первой же необходимости второго варианта «ломать» всю архитектуру вместо осознанного выделения интерфейсов.

### Что будет, если…

**Если игнорировать эмерджентный подход и пытаться всё решить upfront‑архитектурой:**

- значительная часть продуманной архитектуры окажется неиспользуемой;
- появятся лишние слои и абстракции, которые никто не понимает и не использует;
- изменения требований будут больно бить по архитектуре, к которой команда эмоционально привязалась.

### Проверь себя (9.3)

1. Как правило трёх помогает на уровне дизайна, а не только на уровне функций?
   <details><summary>Ответ</summary>
   Оно говорит: не создавай интерфейсы/базовые классы/слои заранее, при первом использовании. Но когда реальный сценарий повторился 3 раза (3 похожих класса, 3 похожих интеграции), пора вынести общее в абстракцию. Это удерживает баланс между дублированием и излишней сложностью.
   </details>

2. Почему эмерджентный дизайн особенно хорошо работает в связке с TDD?
   <details><summary>Ответ</summary>
   В TDD тесты задают форму API ещё до реализации. После каждого «зелёного» теста мы рефакторим. Это естественный цикл «появилось поведение → вытащили из него структуру», что и есть эмерджентный дизайн.
   </details>

3. Приведи пример решения, которое часто делают «на будущее», а потом почти не используют.
   <details><summary>Ответ</summary>
   Например, очень общая фабрика с кучей типов, generics и конфигураций при наличии всего одной конкретной реализации; или сложный механизм плагинов, когда сейчас нужен только один источник данных; или API‑шлюз со сложной системой роутинга, когда у проекта один backend‑сервис.
   </details>

### Запомните

- Эмерджентный дизайн = **тесты + маленькие шаги + рефакторинг + правило трёх**.
- YAGNI защищает от избыточных абстракций и лишних слоёв.
- Хороший дизайн появляется **в процессе**, а не только в начале проекта.
