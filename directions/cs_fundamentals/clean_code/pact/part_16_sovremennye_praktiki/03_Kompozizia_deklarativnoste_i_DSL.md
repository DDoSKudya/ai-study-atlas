## 16.3. Композиция, декларативность и DSL

### Цель раздела

Научиться строить код вокруг **композиции компонентов и функций**, а не глубокого наследования, и использовать **декларативный стиль и простые DSL/fluent‑API**, чтобы бизнес‑логика читалась как описание правил, а не как набор технических шагов.

### В этом разделе главное

- Композиция уменьшает связанность и делает поведение **настраиваемым через составление**.
- Декларативный стиль помогает:
  - отделить **что** от **как**;
  - упростить обсуждение бизнес‑логики.
- DSL и fluent‑API могут сделать код читаемым, но важно не утонуть в избыточной абстракции.

### Термины

- **Композиция объектов** — паттерн «has‑a»: объект содержит другие объекты и делегирует им части поведения.
- **Композиция функций** — соединение функций в цепочку, как в 16.1.
- **DSL (domain‑specific language)** — мини‑язык для конкретной области (например, конфигурация пайплайна обработки платежей).
- **Fluent‑API** — интерфейс, в котором методы возвращают `self` или строителя, что позволяет писать цепочки вида `builder.setA().setB().build()`.

### Теория и правила

1. **Композиция вместо наследования**
   - Наследование:
     - жёстко связывает классы;
     - выносит поведение в базу, из‑за чего дочерние классы **зависят от деталей реализации**.
   - Композиция:
     - объект **держит ссылки** на другие объекты;
     - может свободно их подменять, комбинировать и тестировать по отдельности.

2. **Декларативный стиль**
   - Императивно: «сделай А, потом B, потом C, если X, то D…».
   - Декларативно: «мне нужно, чтобы выполнялись такие‑то правила/условия».
   - Примеры:
     - SQL вместо ручного обхода коллекций;
     - декларативные описания роутинга, UI, пайплайнов.

3. **Когда уместен DSL**
   - Когда область предметная и **часто повторяется**:
     - правила ценообразования;
     - конфигурация пайплайнов;
     - сценарии нотификаций.
   - Когда нужно, чтобы **не‑программисты могли читать/редактировать** правила (частично).

### Пошагово: от императивного к декларативному через композицию

Допустим, у нас есть код начисления бонусов:

```python
def calc_bonus(order):
    bonus = 0
    if order.total > 1000:
        bonus += 100
    if order.customer.is_vip:
        bonus += 200
    if order.created_at.is_weekend():
        bonus += 50
    # ещё несколько условий...
    return bonus
```

Можно выделить **правила в объекты/функции**, а затем составлять их:

```python
def total_over_1000(order):
    return 100 if order.total > 1000 else 0

def vip_customer(order):
    return 200 if order.customer.is_vip else 0

def weekend_bonus(order):
    return 50 if order.created_at.is_weekend() else 0

BONUS_RULES = [
    total_over_1000,
    vip_customer,
    weekend_bonus,
]

def calc_bonus(order):
    return sum(rule(order) for rule in BONUS_RULES)
```

Мы получили:

- **композицию правил** (список функций);
- декларативное описание:
  - «бонус = сумма результатов всех правил».

В дальнейшем:

- легко добавить новое правило;
- можно конфигурировать `BONUS_RULES` из вне (например, из настроек).

### Простыми словами

Композиция и декларативность — это как **набор строительных блоков**:

- вместо того чтобы каждый раз объяснять: «поставь кирпич сюда, этот поверни так, этот на него…»;
- ты говоришь: «построй **такую‑то стену** из вот таких блоков».  
В коде:

- ты описываешь **какие правила и шаги должны примениться**;
- реализация деталей «как именно» спрятана внутри блоков (функций, объектов).

### Картинка в голове

Представь:

- Императивный код — это **подробный рецепт**: «возьми нож, сделай такой‑то надрез, поверни сковородку под таким‑то углом…».
- Декларативный код — это **меню ресторана**: «стейк medium rare с таким‑то гарниром».  
Клиенту (и читателю кода) в большинстве случаев важнее **что** он получит, а не каждое техническое действие по пути.

### Как запомнить

- **Композиция**: «собираем поведение из блоков» → легко тестировать и заменять блоки.
- **Декларативность**: «описываем результат и правила, а не шаги исполнения».
- Если код можно оформить как список правил / шагов, которые применяются в одном и том же порядке, — **кандидат на декларативный стиль**.

### Примеры и практика / реальные сценарии

- **Роутинг HTTP‑запросов**
  - Императивно:
    - большой `if/else` по URL и методам.
  - Декларативно:
    - таблица/список роутов: `[(method, path, handler), …]`.

- **Конфигурация пайплайна обработки данных**
  - Список шагов:
    - парсинг;
    - валидация;
    - обогащение;
    - сохранение.  
  Описывается как **набор компонентов**, которые вызываются по очереди.

- **Правила доступа (ACL/permissions)**
  - Вместо разбросанных `if user.is_admin` по коду — декларативная конфигурация:
    - роль → список прав;
    - ресурс → правила доступа.

### Типичные ошибки и что будет, если…

- **Ошибка:** создавать сложный DSL «на будущее», когда сейчас достаточно пары функций или простой конфигурации.
  - **Последствие:** больше кода, сложнее дебажить; команда тратит больше времени на поддержку, чем на решение реальных задач.
  - **Лучше:** начинать с **простого композиционного решения**; выносить в DSL только то, что реально повторяется и меняется.

- **Ошибка:** использовать fluent‑API, который скрывает важные обязательные шаги.
  - **Последствие:** можно забыть вызвать критичный метод (например, `validate()`), потому что интерфейс не делает это явно.
  - **Лучше:** проектировать fluent‑API так, чтобы **нельзя было собрать некорректный объект** (Builder с обязательными шагами или проверкой).

### Проверь себя (16.3)

1. Чем композиция объектов отличается от паттерна «God‑объект»?

   <details><summary>Ответ</summary>
   При композиции:
   - объект делегирует работу **нескольким узкоспециализированным компонентам**;
   - каждый компонент можно протестировать и заменить отдельно.  
   God‑объект:
   - содержит слишком много ответственности;
   - часто сам реализует всё поведение, а не делегирует его.  
   В композиции фокус на **разделении обязанностей**, а не на концентрации всей логики в одном месте.
   </details>

2. Почему декларативный стиль хорошо сочетается с тестированием?

   <details><summary>Ответ</summary>
   Декларативный код часто выражает:
   - **правила и связи** (например, правила бонусов, матрицу прав доступа);  
   Эти структуры легко:
   - проверять в тестах;
   - изменять, не трогая реализацию;  
   Тесты могут работать на уровне «данные конфигурации → ожидаемый результат», не лезя в императивные шаги.
   </details>

3. Приведи пример простого DSL или fluent‑API, который можно было бы ввести в твоём проекте.

   <details><summary>Ответ</summary>
   Возможные примеры:
   - описание валидации формы как цепочки:
     - `validator.field("email").required().email().max_length(255)`;  
   - конфигурация оповещений:
     - `rule.when(order_total > 1000).notify("sales-team").via_email()`;  
   В обоих случаях мы получаем код, который **читается как описание правил**, а не как набор технических шагов.
   </details>

4. В чём риск слишком общего или «гибкого» DSL для бизнес‑правил?

   <details><summary>Ответ</summary>
   Слишком общий DSL:
   - часто превращается в «язык в языке», который сложнее самой бизнес‑логики;
   - требует отдельного онбординга и инструментов;
   - затрудняет статический анализ и рефакторинг.  
   В результате команда тратит больше сил на поддержку DSL, чем на решение предметных задач, а читаемость падает.
   </details>

5. Как композиция правил (список функций, применяемых к объекту) помогает упорядочить и документировать сложные бизнес‑политики?

   <details><summary>Ответ</summary>
   Когда правила собраны в список/таблицу:
   - видно, **какие именно правила вообще существуют**;
   - легко добавить/изменить правило, не трогая остальные;
   - порядок применения фиксирован и прозрачен;  
   Такой подход превращает бизнес‑логику в **явную структуру данных**, которую можно:
   - проверять тестами;
   - визуализировать;
   - обсуждать с аналитиками и заказчиками.
   </details>

### Запомните (16.3)

- Композиция и декларативность помогают **сделать бизнес‑логику явной** и отделённой от технических деталей.
- DSL и fluent‑API полезны, когда они **упрощают чтение и изменение правил**, а не ради красоты.
- Наследование остаётся инструментом, но по умолчанию стоит предпочитать **композицию и явные зависимости**.
