## 16.1. Функциональный стиль и чистые функции

### Цель раздела

Научиться **видеть** и **писать** участки кода в функциональном стиле: выделять чистые функции, использовать функции высшего порядка (например, `map`, `filter`) и строить простые композиции вместо длинных императивных циклов с состоянием.

### В этом разделе главное

- Функциональный стиль — это **про то, как мы думаем о решении**: как о последовательности преобразований данных.
- Чистые функции упрощают:
  - тестирование (нет скрытого состояния);
  - рефакторинг (меньше неожиданностей);
  - конкурентность (меньше точек гонок).
- Функции высшего порядка и композиция позволяют строить **понятные пайплайны**, но важно не превращать их в ребусы.

### Термины

- **Функция высшего порядка** — функция, которая принимает другие функции как аргументы или возвращает их.
- **Пайплайн обработки данных** — последовательность шагов `данные → трансформация 1 → трансформация 2 → … → результат`.
- **Лямбда / анонимная функция** — короткая функция без имени, обычно передаётся в другую функцию.

### Теория и правила

1. **Чистая функция — «идеальный кирпичик»**
   - Не читает глобальные переменные (кроме констант).
   - Не изменяет внешние данные (аргументы по ссылке, глобальные объекты).
   - Не делает побочных эффектов (нет логов, запросов, записи в БД и файлов).
   - Явно возвращает результат.

2. **Где держать побочные эффекты**
   - В слое «на границе»: контроллеры, хендлеры, адаптеры, инфраструктура.
   - Основная бизнес‑логика — по возможности **чистые функции**, которые получают данные и возвращают новые.
   - Там, где эффект неизбежен (лог, метрика, событие) — делать его **явным и локализованным**.

3. **Функции высшего порядка и коллекции**
   - Вместо цикла с ручной мутацией:
     - `for item in items: if условие: что‑то сделать`  
     использовать `filter` + `map`/`reduce`/другие операции.
   - Правило: если операция **естественно описывается как «возьми каждый элемент и преобразуй»**, функция высшего порядка обычно читабельнее.

4. **Композиция функций**
   - В простом виде: `result = f(g(h(data)))`.
   - В языках с пайплайнами: `data |> h |> g |> f`.
   - Для читаемости полезно давать промежуточным функциям **говорящие имена**, а не писать всё в одну анонимную лямбду.

### Пошагово: превращаем императивный код в функциональный

Предположим, есть код (на псевдоязыке, близком к Python/TypeScript):

```python
valid_orders = []
for order in orders:
    if not order.is_active:
        continue
    if order.total < 1000:
        continue
    discount = calculate_discount(order)
    valid_orders.append((order.id, discount))
```

Пошаговое преобразование:

1. **Выделить проверку в функцию/лямбду**

   ```python
   def is_interesting(order):
       return order.is_active and order.total >= 1000
   ```

2. **Использовать `filter`**

   ```python
   filtered = filter(is_interesting, orders)
   ```

3. **Выделить логику расчёта результата в отдельную функцию**

   ```python
   def to_result(order):
       discount = calculate_discount(order)
       return (order.id, discount)
   ```

4. **Использовать `map`**

   ```python
   valid_orders = list(map(to_result, filtered))
   ```

В итоге у нас:

- две чистые функции: `is_interesting` и `to_result`;
- основной код читается как **сценарий**: «отфильтровать интересные заказы → преобразовать в результат».

### Простыми словами

Функциональный стиль — это когда ты представляешь свой код **как конвейер**:

1. На входе у тебя есть данные.
2. Ты проходишь через цепочку **маленьких станций‑функций**, каждая из которых:
   - ничего не знает о предыдущих станциях;
   - просто получает вход и выдаёт выход.
3. В конце конвейера получается результат.

Если станции **ничего не портят по дороге** (не ломают детали на соседних станциях, не меняют конвейер), код легче проверять, отлаживать и изменять.

### Картинка в голове

Представь, что твой код — это **кухня ресторана**:

- Императивный стиль с кучей состояний — это один повар, который:
  - делает всё сам;
  - помнит в голове, в каком состоянии какое блюдо;
  - периодически что‑то забывает и путает.
- Функциональный стиль — это конвейер:
  - отдельный человек (функция) нарезает овощи;
  - другой жарит;
  - третий собирает блюдо.  
  Каждый делает **одно понятное действие**, и если что‑то идёт не так, легче понять, **на каком шаге** ошибка.

### Как запомнить

- **Чистая функция**: «без побочных эффектов и сюрпризов» — зависит только от аргументов.
- **Функциональный стиль**: «преобразуем данные через конвейер маленьких функций».
- **Правило применения**: «если можно выразить через `для каждого` или `из этих взять только те, которые…`, — подумай про `map/filter`».

### Примеры и практика / реальные сценарии

- **Валидация входящих данных**
  - Императивно: цепочка `if` и накопление флага `is_valid`.
  - Функционально: список правил‑функций, которые принимают объект и возвращают либо `Ok`, либо `Error`, а затем:
    - последовательное применение правил;
    - сбор всех ошибок в список.

- **Обработка HTTP‑запроса**
  - Внутренняя бизнес‑логика: чистые функции, которые принимают DTO и возвращают результат.
  - Побочные эффекты (логирование, доступ к БД, вызов других сервисов) — в отдельных, явно помеченных слоях.

- **Трансформация отчётных данных**
  - Вместо длинного цикла, где ты по шагам:
    - фильтруешь;
    - агрегируешь;
    - форматируешь;  
  делай цепочку:
  - `filter по датам` → `group by` → `map к DTO` → `sort`.

### Функциональный стиль и обработка ошибок (Result/Option)

Функциональный стиль хорошо сочетается с **явными типами ошибок и отсутствия значения**:

- вместо `null`:
  - `Option/Maybe<T>`: либо `Some(T)`, либо `None`;
- вместо выброса исключений в обычных сценариях:
  - `Result<T, E>` / `Either<E, T>`: либо `Ok(T)`, либо `Error(E)`.

Преимущества:

- ошибки и «нет значения» **видны в типе**: их нельзя «забыть обработать»;
- можно строить **композиции**:
  - последовательность шагов, каждый из которых возвращает `Result`;
  - при ошибке цепочка прерывается автоматически.

Пример на псевдо‑TypeScript:

```ts
type Result<T> =
  | { type: "ok"; value: T }
  | { type: "error"; message: string };

function parse(json: string): Result<any> { /* ... */ }
function validateUser(raw: any): Result<User> { /* ... */ }
function enrich(user: User): Result<EnrichedUser> { /* ... */ }

function loadUser(json: string): Result<EnrichedUser> {
  return pipe(
    parse(json),
    andThen(validateUser),
    andThen(enrich),
  );
}
```

Здесь:

- `pipe` и `andThen` реализуют **композицию по Result**:
  - если шаг вернул `ok`, следующий получает его значение;
  - если шаг вернул `error`, остальные шаги пропускаются, ошибка «просачивается» наверх.

Такой подход:

- делает обработку ошибок **частью конвейера**;
- уменьшает количество вложенных `if/try` и ранних `return` в каждом шаге;
- позволяет централизованно решать, как отображать/логировать ошибки.

### Современные идиомы в разных языках

- **JavaScript / TypeScript**
  - Работа с массивами через `map`, `filter`, `reduce`, `some`, `every`, `find`.
  - Асинхронность через `async/await` вместо «пирамиды колбэков»:
    - читаемый «линейный» код поверх промисов;
    - важно **не смешивать** `then`‑цепочки и `async/await` в одном фрагменте без необходимости.
- **Python**
  - Списковые включения (`[f(x) for x in xs if cond(x)]`) и генераторы.
  - Функции `map`, `filter`, `any`, `all`, `sum` для агрегирования.
  - `itertools` для сложных конвейеров (но важно не превращать их в головоломку).
- **C# / .NET**
  - LINQ (`Where`, `Select`, `GroupBy`, `Aggregate`) как декларативный способ работы с коллекциями.
  - `async/await` и `IAsyncEnumerable` для асинхронных потоков данных.
- **Kotlin / Java**
  - Kotlin‑расширения для коллекций (`map`, `filter`, `flatMap`, цепочки вызовов).
  - Java Streams (`stream().filter(...).map(...).collect(...)`) — но важно следить за читаемостью и не вкладывать слишком много логики в одну лямбду.
- **SQL и ORM**
  - Описание запросов **декларативно**: «что выбрать и как связать», а не «как обходить строки вручную».

Общий принцип: **использовать идиомы языка так, как принято в экосистеме**, не изобретая свой велосипед и не усложняя выражения ради «функциональности».

### Типичные ошибки и что будет, если…

- **Ошибка:** переписывать любой цикл в `map/filter` даже там, где логика сложная и с большим количеством условий.
  - **Последствие:** код становится **меньше строк, но хуже читается**; команда тратит больше времени на понимание.
  - **Лучше:** использовать функциональный стиль **там, где он делает код проще**, а не короче.

- **Ошибка:** пытаться сделать все функции чистыми, включая те, которые должны логировать или ходить в сеть.
  - **Последствие:** искусственное усложнение, лишние абстракции и обёртки.
  - **Лучше:** разделять:
    - чистую бизнес‑логику;
    - слой, который реализует побочные эффекты.

- **Ошибка:** полностью избегать императивного стиля даже там, где он естественен (простые циклы, временные локальные счётчики).
  - **Последствие:** код становится «функциональным ради функциональности», хуже читается новичками и разработчиками без сильного функционального бэкграунда.
  - **Лучше:** выбирать стиль **по читаемости для команды**: иногда ясный цикл с парой хорошо названных шагов лучше цепочки из нескольких функций высшего порядка.

### Проверь себя (16.1)

1. Приведи пример функции из твоего кода, которая могла бы быть чистой, но сейчас имеет побочный эффект.

   <details><summary>Ответ</summary>
   Возможный пример: функция `calculateTotal(order)` внутри себя:
   - логирует промежуточный результат;
   - меняет поле `order.discountApplied = True`;  
   В чистом варианте:
   - она бы только считала итог и возвращала его;
   - логирование и изменение `order` происходили бы снаружи.
   </details>

2. В каких случаях императивный цикл может быть понятнее, чем композиция `map/filter`?

   <details><summary>Ответ</summary>
   Когда логика слишком **ветвистая**:
   - множество разных веток;
   - промежуточные состояния;
   - зависимость от порядка побочных эффектов.  
   В таких случаях попытка «втиснуть» всё в `map/filter` может сделать код менее очевидным. Иногда **явный цикл с хорошо названными шагами** — лучший выбор.
   </details>

3. Как функциональный стиль влияет на тесты?

   <details><summary>Ответ</summary>
   Чистые функции:
   - проще тестировать: достаточно передать аргументы и проверить результат;
   - не требуют сложной подготовки окружения;
   - не зависят от порядка запуска и внешнего состояния.  
   В итоге тесты получаются **быстрее, надёжнее и проще**. Побочные эффекты остаются в меньшем числе мест, для которых пишутся целевые интеграционные тесты.
   </details>

4. В чём практическая польза Result/Option‑подхода по сравнению с `null` и выбрасыванием исключений в каждом шаге конвейера?

   <details><summary>Ответ</summary>
   Result/Option:
   - делает **отсутствие значения и ошибки явными в типе**, поэтому их сложнее забыть обработать;
   - позволяет строить конвейеры (`pipe`, `andThen`), где обработка ошибок встроена в композицию: при `error` дальнейшие шаги не вызываются;
   - уменьшает количество дублирующих `if value is None` / `try/except` в каждом шаге.  
   В итоге конвейер читается как «последовательность преобразований», а не как смесь логики и проверки ошибок на каждом шаге.
   </details>

5. Как использование идиоматических средств языка (LINQ, списковые включения, streams и т.п.) связано с идеей читаемости в функциональном стиле?

   <details><summary>Ответ</summary>
   Идиоматические средства:
   - дают **общий, привычный для экосистемы способ** описывать операции над данными;
   - позволяют опытному разработчику «считать» смысл выражения почти мгновенно (например, `Where → Select → GroupBy` в LINQ или списковое включение в Python);  
   Если не перегибать с вложенностью и сложными лямбдами, такой код **читается лучше**, чем произвольные циклы, потому что опирается на знакомые паттерны мышления.
   </details>

### Запомните (16.1)

- Функциональный стиль в прикладном коде — это **осознанное использование чистых функций и композиции**, а не религиозный запрет на побочные эффекты.
- Чем больше ключевой бизнес‑логики выражено через чистые функции, тем легче **тестировать и рефакторить** систему.
- Используй функции высшего порядка **для ясности сценария**, а не ради модных конструкций.
