## 5.1. Исключения вместо кодов возврата

### Цель раздела

Понять, почему **коды возврата** (магические числа, статусы) часто делают код хрупким и нечитаемым; увидеть, как **исключения** (или `Result`‑типы) помогают отделить нормальный поток от ошибочного, и научиться добавлять к исключениям **контекст**, не теряя исходную причину.

### В этом разделе главное

- Коды возврата легко **забыть проверить**, они размывают обработку ошибок по всему коду.
- Исключения отделяют **нормальный путь выполнения** от **обработки ошибок** и концентрируют обработку в понятных местах.
- У исключения должен быть **контекст**: сообщение, тип, при необходимости — вложенная причина (`cause`).
- При перевыбросе ошибки важно **не терять исходную причину**, а оборачивать её и сохранять цепочку.

### Термины

| Термин               | Определение |
|----------------------|-------------|
| **Код возврата**     | Специальное значение (обычно число или enum), которым функция сигнализирует об успехе или типе ошибки |
| **Исключение**       | Объект/событие, указывающее на ошибочную ситуацию и передаваемое по стеку до обработчика |
| **Контекст исключения** | Дополнительная информация (сообщение, данные), помогающая понять, что именно произошло |
| **Цепочка причин (cause chain)** | Последовательность исключений при оборачивании низкоуровневых ошибок в более высокоуровневые |

### Теория и правила

#### Почему коды возврата — слабое место

Классический стиль:

```c
int rc = do_something();
if (rc != 0) {
    // обработка ошибки
}
```

Проблемы:

- **Легко забыть проверить** код: кто‑то вызовет `do_something();` и не посмотрит на результат.
- Часто используются **магические значения**: `-1`, `42`, `9999`, которые ничего не говорят о сути ошибки.
- Логика проверки **размазана** по коду: возле каждого вызова — `if rc != OK`, и в каждом месте — чуть разная реакция.
- Код нормального потока **перемешивается** с проверками, что ухудшает читаемость.

В чистом коде цель — иметь **линейный нормальный поток** и явную, хорошо локализованную обработку ошибок.

#### Исключения: отделение нормального и ошибочного потока

Исключения меняют модель:

- нормальный код выглядит как **линейная последовательность** вызовов;
- ошибки «всплывают» вверх до ближайшего осмысленного обработчика;
- мы можем обработать ошибки **на границе слоя**, а не в каждом методе;
- IDE и язык (в ряде случаев) помогают не забыть про возможные исключения.

Пример (Python‑стиль):

```python
def process_order(order_id: str) -> None:
    order = load_order(order_id)          # может бросить OrderNotFoundError
    validate_order(order)                 # может бросить ValidationError
    charge_customer(order)                # может бросить PaymentError
    save_order(order)
```

Нормальный поток **не засорён** проверками. Обработка ошибок — в другом месте:

```python
def handle_order_request(request):
    try:
        process_order(request.order_id)
    except OrderNotFoundError:
        return http_not_found("Order not found")
    except ValidationError as exc:
        return http_bad_request(str(exc))
    except PaymentError as exc:
        log_error("Payment failed", exc)
        return http_bad_gateway("Payment provider error")
```

Читается:

- сверху — сценарий;
- ниже — явная карта ошибок и реакций.

#### Контекст и сообщения исключений

Исключение без сообщения:

```python
raise ValueError()
```

— почти бесполезно. В логах и стектрейсах не будет понятно:

- для чего именно это `ValueError`;
- какие аргументы привели к ошибке;
- что ожидалось.

Лучше:

```python
raise ValueError(f"Order {order_id}: total must be positive, got {order.total}")
```

Или (ещё лучше) использовать **свой тип исключения**:

```python
class InvalidOrderTotalError(ValueError):
    pass
```

и бросать его с осмысленным сообщением. Тип исключения уже несёт часть смысла, сообщение — остальное.

#### Цепочка причин (cause chain) и перевыброс

Часто мы оборачиваем низкоуровневое исключение в более высокоуровневое:

- библиотека БД бросила `DatabaseTimeoutError`;
- на уровне домена мы хотим `PaymentProviderUnavailableError`.

Важно **сохранить исходную причину**:

```python
try:
    db.charge_card(card, amount)
except DatabaseTimeoutError as exc:
    raise PaymentProviderUnavailableError("Payment DB timeout") from exc
```

Теперь в стектрейсе и логах:

- доменное исключение содержит сообщение о бизнес‑контексте;
- цепочка причин (`__cause__`/`innerException` и т.п.) хранит исходное техническое событие.

### Пошагово: миграция от кодов возврата к исключениям

Представь, что у тебя есть функция:

```python
def save_user_to_db(user) -> int:
    # 0 = успех, 1 = ошибка подключения, 2 = конфликт
    ...
```

1. **Явно опиши значения кодов** (enum, константы).
2. Найди все места, где эта функция вызывается:
   - посмотри, как там обрабатываются коды;
   - заметишь дублирование и пропуски.
3. Спроектируй **типы исключений**:
   - `DatabaseConnectionError`;
   - `UserConflictError`.
4. Измени функцию:

   ```python
   def save_user_to_db(user) -> None:
       ...
       raise DatabaseConnectionError("...")  # вместо кода 1
       ...
   ```

5. На вызывающей стороне:
   - убери `if rc != 0`;
   - добавь `try/except` на границе слоя (контроллер, сервис);
   - свяжи исключения с реакцией (лог, ответ пользователю).

**Проверь себя (миграция)**  

1. Почему полезно сначала выписать все значения кодов возврата и их смысл, прежде чем переходить к исключениям?
   <details><summary>Ответ</summary>
   Это помогает понять, какие **типы ошибок** вообще моделируются в системе и где есть путаница. Часто коды возвращаются непоследовательно, некоторые значения не используются или используются по‑разному. Переход к исключениям без этой инвентаризации может перенести хаос «как есть», вместо того чтобы его упорядочить.
   </details>

2. Почему исключения обычно ловят на границе слоя, а не рядом с каждым вызовом функции?
   <details><summary>Ответ</summary>
   Ловля ошибок в каждом месте вызова ведёт к дублированию и кода, и логики; сложно обеспечить единообразную реакцию. На границе слоя мы можем централизованно решить, как реагировать на разные типы ошибок (лог, ретрай, ответ API), оставив внутренним функциям ответственность только за «поднять» правильное исключение.
   </details>

### Простыми словами

Коды возврата — это как «моргать фонариком» в надежде, что кто‑то заметит: если забыли посмотреть — беда. Исключения — это как **громкий сигнал**, который невозможно игнорировать: либо его поймают и обработают, либо программа упадёт, и мы об этом узнаем. Чистый код выбирает второй вариант и учится делать этот сигнал информативным и правильно размещённым.

### Картинка в голове

Представь, что в поезде есть датчик перегрева:

- Вариант с кодом возврата: датчик тихо меняет значение в журнале, надеясь, что машинист туда заглянет.
- Вариант с исключением: датчик посылает тревогу в кабину, автоматически включает сигнал и требует реакции.

И то, и другое — информация, но во втором случае **сложнее проигнорировать проблему**.

### Как запомнить

- **Код возврата легко забыть проверить**, исключение игнорировать сложнее.
- **Исключения = разделение потоков**: нормальный сценарий и ошибочный.
- Исключение без контекста — почти так же плохо, как код `-1` без описания.

### Примеры

**Коды возврата (плохо):**

```python
RC_OK = 0
RC_CONNECTION_ERROR = 1


def send_payment(order) -> int:
    # ...
    if not connection_ok:
        return RC_CONNECTION_ERROR
    # ...
    return RC_OK


rc = send_payment(order)
if rc != RC_OK:
    logger.error("payment failed")
    # забыли различить типы ошибок, что делать дальше — непонятно
```

**Исключения с контекстом (лучше):**

```python
class PaymentError(Exception):
    pass


class PaymentConnectionError(PaymentError):
    pass


def send_payment(order) -> None:
    if not connection_ok:
        raise PaymentConnectionError(f"Payment connection failed for order {order.id}")
    ...
```

А где‑то выше:

```python
try:
    send_payment(order)
except PaymentConnectionError as exc:
    logger.error("Payment connection error: %s", exc)
    notify_ops_team(exc)
    return http_bad_gateway("Payment provider unavailable. Try again later.")
```

### Практический сценарий

В legacy‑коде ты видишь:

```python
status = repo.save(order)
if status == -1:
    log_error("db error")
    return False
return True
```

В другом месте:

```python
if not repo.save(order):
    # просто игнорируем
    pass
```

Статус `-1` трактуется по‑разному, а где‑то вообще игнорируется. Это создаёт неявные точки отказа. После перехода на исключения (`DatabaseError`) и обработку на уровне слоя сервисов ты получаешь:

- единый способ реагировать на сбой БД;
- меньше дублирования;
- меньше шансов «случайно» проглотить ошибку.

### Типичные ошибки

- Смешивать коды возврата и исключения в одном модуле без явных правил («иногда бросаем, иногда возвращаем код»).
- Использовать «общий» код ошибки `-1` или «общий» тип исключения вроде `Exception`/`Error` без уточнения.
- Бросать исключения без сообщений или с бессмысленными сообщениями (`"Error"`, `"fail"`).
- Перевбрасывать исключение, **теряя исходную причину** (логировать и бросать новое без `cause`/`innerException`).

### Что будет, если…

Если оставить коды возврата как есть и не переходить к явной модели ошибок, то:

- часть ошибок будет **теряться**, потому что кто‑то забудет проверить код;
- часть будет **обрабатываться по‑разному** в разных местах;
- добавление новых типов ошибок приведёт к ещё большему хаосу;
- поиск причины сбоя будет сводиться к охоте за «минус‑первым» по всей кодовой базе.

### Проверь себя (5.1)

1. Почему исключения помогают отделить нормальный поток от обработчика ошибок?
   <details><summary>Ответ</summary>
   Нормальный код пишет только то, что нужно для успешного сценария, без постоянных проверок статусов после каждого вызова. Ошибки всплывают до `catch`/`except`, где сосредоточена логика обработки. В результате основной путь линейный, а обработка ошибок — в одном месте, а не размазана по всему коду.
   </details>

2. Зачем добавлять к исключению контекст (сообщение, идентификаторы, параметры)?
   <details><summary>Ответ</summary>
   Без контекста в логах и стектрейсах будет непонятно, что именно пошло не так и с какими данными. Контекст позволяет быстрее диагностировать проблему, воспроизвести её и понять, относится ли ошибка к одной сущности или к системной проблеме. Это особенно важно в продакшене, где доступа к пошаговому дебагу нет.
   </details>

3. Чем полезна цепочка причин (cause chain) при оборачивании исключений?
   <details><summary>Ответ</summary>
   Она позволяет видеть «дорогу» ошибки от низкоуровневого события (например, таймаут в БД) до высокого уровня (например, ошибка платежа). Это помогает и разработчикам, и операторам лучше понять, где действительно случилась проблема и как её исправлять — на уровне бизнес‑логики, инфраструктуры или внешнего поставщика.
   </details>

### Запомните

- **Коды возврата** легко пропустить, смешивают ошибочный и нормальный потоки и часто используют магические значения.
- **Исключения** (или `Result`‑типы) позволяют отделить обработку ошибок, собрать её в одном месте и сделать поведение предсказуемым.
- У исключения должен быть **контекст и сохранённая причина**; «голые» и «немые» исключения мало полезны для диагностики.
