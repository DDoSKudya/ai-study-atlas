## 5.2. Границы и контракты

### Цель раздела

Научиться решать, **где именно** в системе должны ловиться и обрабатываться ошибки, как проектировать **контракты функций и API** так, чтобы `null` и «тихий» провал не ломали код, и как использовать `Optional/Option` и `Result`‑типы для явного выражения «значения может не быть» и «операция может завершиться ошибкой».

### В этом разделе главное

- Ошибки лучше ловить **на границах слоёв**, а не в каждом низкоуровневом методе.
- **Пустые `catch`** и «молчаливое» игнорирование ошибок — один из главных анти‑паттернов.
- `null`/**`None`** как «нет значения» без явного контракта — источник скрытых сбоев.
- `Optional/Option/Maybe` и `Result`‑типы позволяют **явно моделировать** отсутствие и ошибки в сигнатурах функций.
- Важно различать, что является **нормальным потоком**, а что — **ошибкой**, и не использовать исключения для обычной логики.

### Термины

| Термин                 | Определение |
|------------------------|-------------|
| **Граница слоя**       | Место, где один слой системы вызывает другой (UI → сервис, сервис → репозиторий, сервис → внешнее API) |
| **Доменная ошибка**    | Ошибка бизнес‑уровня (недостаточно средств, недопустимое состояние заказа) |
| **Техническая ошибка** | Ошибка инфраструктуры (нет сети, таймаут, ошибка БД) |
| **`Optional` / `Option`** | Тип, явно представляющий «есть значение» или «нет значения» |
| **`Result`‑тип**       | Тип, представляющий «успех (с результатом)» или «ошибка (с описанием)» |

### Теория и правила

#### Где ловить ошибки: границы слоёв

Типичная страсть новичков — оборачивать **каждый вызов** в `try/except`:

```python
def process():
    try:
        do_a()
    except Exception:
        log_error("A failed")
    try:
        do_b()
    except Exception:
        log_error("B failed")
```

Проблемы:

- обработка ошибок **дублируется**;
- трудно понять, какая ошибка считается фатальной, а какая — нет;
- «общий» `except Exception` может скрыть баги в логике, не только ожидаемые сбои.

Лучше:

- внутри слоёв бросать **конкретные доменные/технические исключения**;
- ловить их **на границе**:
  - HTTP‑обработчик;
  - CLI‑команда;
  - фоновой воркер;
  - точка входа в сервис.

Там мы:

- решаем, какой **ответ** дать пользователю/клиенту;
- логируем и, при необходимости, **алертим**;
- решаем, делать ли **ретрай** или деградировать.

#### Не пропускать исключения молча

Анти‑паттерн:

```python
try:
    do_something()
except Exception:
    pass  # «тихо провалились»
```

или:

```python
except Exception as exc:
    logger.error("Something failed")
    # без исключения/контекста
```

Проблемы:

- ошибка **происходит**, но система делает вид, что всё нормально;
- сложно потом понять, почему «что‑то не сработало», в логах информации мало;
- тесты могут не заметить, что часть функциональности фактически не работает.

Принцип:

- либо **обрабатываем** ошибку (например, возвращаем альтернативный результат, даём понятное сообщение пользователю, делаем ретрай и т.п.);
- либо **пробрасываем/оборачиваем** её выше;
- либо **завершаем** операцию/процесс, явно сигнализируя о сбое;
- но не делаем вид, что ничего не случилось.

#### Не возвращать и не передавать `null` без контракта

`null`/**`None`** часто используют как:

- «нет значения»;
- «ошибка» / «специальный случай»;
- «ещё не инициализировано».

Если это **не записано в контракте функции**, то:

- вызывающий не знает, **должен ли он проверять на `null`**;
- ошибки превращаются в `NullPointerException`/`AttributeError`/`NoneType has no attribute`, всплывающие в произвольном месте;
- при передаче `null` в параметры функция вынуждена либо «защищаться» кучей проверок, либо молча падать в глубине стека.

Чистый код предпочитает:

- **пустую коллекцию** вместо `null` для «нет элементов»;
- **специальный тип** (`Optional`, `Option`, `Maybe`) вместо «то значение, то `null`»;
- **`Result`‑тип** вместо «успех или `null`/магический код»;
- явную документацию: в сигнатуре, комментариях API, docstring/типах.

Примеры:

```python
# Плохо: может вернуть None, но это не видно в сигнатуре
def find_user(user_id: str) -> "User | None":
    ...

# Лучше: явно моделируем «может не быть»
from typing import Optional


def find_user(user_id: str) -> Optional["User"]:
    ...
```

То же самое относится и к **входным параметрам**: если функция не ожидает `None`, это стоит отражать в типах и явно запрещать в контракте, а не «надеяться на лучшее». В противном случае придётся либо добавлять проверки `if x is None` в каждом методе, либо отлавливать падения в произвольных местах.

Пример:

```python
def send_email(to: str, body: str) -> None:
    """
    :param to: e-mail получателя, не None и не пустая строка.
    """
    if not to:
        raise ValueError("Recipient e-mail must be provided")
    ...
```

Здесь контракт явно запрещает `None`/пустую строку и сразу превращает некорректный ввод в понятную ошибку, а не в «AttributeError где‑то внутри SMTP‑клиента».

Или в функциональном стиле:

```kotlin
fun findUser(id: UserId): Option<User> { ... }  // Option.empty() вместо null
```

#### `Result`‑подход: успех или ошибка как значение

Вместо «сделал или бросил исключение» часто удобно возвращать **результат с двумя ветками**:

- `Ok(value)` — успех;
- `Err(error)` — ошибка.

Пример (псевдокод, Rust‑подобный стиль):

```rust
fn charge_customer(order: Order) -> Result<Receipt, PaymentError> {
    ...
}

match charge_customer(order) {
    Ok(receipt) => save_receipt(receipt),
    Err(error) => log_payment_error(error),
}
```

Плюсы:

- компилятор/типизация **заставляет** учесть обе ветки;
- ошибки «не взрываются» в неожиданных местах, а обрабатываются явно;
- удобнее тестировать: нет необходимости проверять выброс исключений, достаточно работать с обычными значениями.

Минусы/ограничения:

- больше **шумового кода** (обработка `Err` везде);
- если не использовать средства языка (pattern matching, монады), код может стать громоздким;
- в некоторых ситуациях (глубокий стек вызовов) исключения читаются проще.

Важно: `Result`‑подход не отменяет идею **границ**: мы так же можем оборачивать внутренние ошибки в более высокоуровневые, централизовать обработку и т.п.

#### Нормальный поток vs ошибка

Ошибкой стоит считать то, что:

- **выходит за рамки нормального сценария** (сбой инфраструктуры, нарушение инварианта);
- обычно ожидается **редко** и требует особого внимания.

Не всегда отсутствие результата — это ошибка:

- запрос «найди пользователя по id» может **нормально** возвращать «нет такого пользователя»;
- поиск в коллекции может возвращать «ничего не найдено»;
- пагинация может возвращать пустую страницу, если дальше элементов нет.

Опасный паттерн:

```python
try:
    user = repo.get_user(id)
except UserNotFoundError:
    # нормально, просто нет пользователя
    user = None
```

Если отсутствие пользователя — **обычный** сценарий, лучше:

- не бросать исключение, а возвращать `Optional[User]` или `None` в рамках контракта;
- или использовать `Result`/`Either`‑подобные конструкции.

### Пошагово: проектируем контракт функции с учётом ошибок

Допустим, нужно спроектировать функцию `find_user_by_email`.

1. Спроси, **что является нормальным сценарием**:
   - email может не существовать? (скорее всего — да);
   - такое отсутствие — ошибка или нормальный «нет»?
2. Реши, как это лучше выразить:
   - `Optional[User]` / `Option<User>`;
   - пустая коллекция;
   - `Result[User, UserNotFound]`.
3. Если это **ручка API**, определи:
   - когда возвращать 404, а когда 200 с пустым телом;
   - где и как логировать странные ситуации (например, дубли email).
4. Определи, какие **технические ошибки** могут возникнуть (БД, сеть) и как они отображаются на доменный контракт.

**Проверь себя (контракты)**  

1. Почему важно решить, считается ли отсутствие объекта (например, пользователя) нормальным сценарием или ошибкой?
   <details><summary>Ответ</summary>
   От этого зависит, каким образом мы выражаем ситуацию в коде: исключением, пустым результатом, `Optional` или `Result`. Если отсутствие — норма (поиск, фильтрация), использовать исключение избыточно и делает код тяжёлым. Если это нарушение инварианта (например, «заказ, который должен существовать»), тогда отсутствие — ошибка, и исключение уместно.
   </details>

2. Чем полезны `Optional`/`Option` по сравнению с «возвратом `null` по договорённости»?
   <details><summary>Ответ</summary>
   `Optional`/`Option` заставляют вызывающего явно обработать оба случая (`Some` и `None`), а типизация/инструменты подсказывают, что значение может отсутствовать. Возврат `null` по «договорённости» легко забыть учесть, и ошибка проявится как `NullPointerException`/`AttributeError` в неожиданном месте.
   </details>

### Простыми словами

Ошибки живут **на стыках**: между слоями приложения и между нашим кодом и внешним миром. Если расставить ловушки и обработчики везде подряд, код превращается в «минное поле» с кучей разрозненных проверок. Если же осознанно разместить обработку на границах и выразить в сигнатурах, где значение может отсутствовать, а где может случиться ошибка, код станет предсказуемым и честным.

### Картинка в голове

Представь, что у твоего дома есть три двери:

- входная (пользователь/клиент),
- дверь в подсобку (инфраструктура),
- аварийный выход (ошибки).

Ловить и разруливать все проблемы **посреди коридора** — странно. Логичнее:

- проверять документы и пропуска на входе;
- контролировать доступ к подсобке на её двери;
- иметь понятные указатели на аварийный выход.

В коде границы слоёв — это те же двери.

### Как запомнить

- Обрабатывать ошибки **на границах**, а внутри — бросать/возвращать осмысленные ошибки.
- `null` как «нет значения» без контракта — **скрытая мина**.
- `Optional`/`Option` и `Result` делают отсутствие и ошибки **частью типа**, а не «договорённостью на словах».

### Примеры

**`null` без контракта (плохо):**

```python
def get_user(user_id: str):
    # возвращает None, если не найдено
    ...


user = get_user("123")
send_email(user.email, "Hi!")  # AttributeError: 'NoneType' ...
```

**`Optional` в контракте (лучше):**

```python
from typing import Optional


def find_user(user_id: str) -> Optional["User"]:
    ...


user = find_user("123")
if user is None:
    return http_not_found("User not found")

send_email(user.email, "Hi!")
```

**Result‑подход (псевдокод):**

```python
class UserNotFound: ...
class DbError: ...


ResultUser = Result["User", Union[UserNotFound, DbError]]


def find_user(user_id: str) -> ResultUser:
    ...
```

### Практический сценарий

У тебя есть репозиторий:

```python
def get_user(user_id: str) -> User:
    # бросает UserNotFoundError, если пользователя нет
    ...
```

В одном сервисе это нормально — отсутствие пользователя действительно ошибка. В другом — нужен поиск (например, «если есть, то…»). Попытка использовать один и тот же метод для двух сценариев приводит к `try/except UserNotFoundError` в куче мест. Решение:

- оставить `get_user` для случая, когда пользователь **обязан быть**;
- добавить `find_user` (`Optional[User]`), когда «нет» — нормальный сценарий;
- в каждом месте кода использовать соответствующую функцию.

### Типичные ошибки

- Ловить `Exception`/`Throwable`/`BaseException` повсюду, не различая типы ошибок.
- Проглатывать исключения (`pass` или пустой `catch`), теряя информацию.
- Возвращать `null`/магический код вместо явного типа результата.
- Использовать исключения для нормального потока (`try/except` как «if/else»), особенно в часто вызываемом коде.

#### Обработка ошибок, безопасность и надёжность

Обработка ошибок тесно связана с **безопасностью** и **надёжностью** системы.

- При безопасности важно:
  - **не раскрывать внутренние детали** через ошибки пользователю (стектрейсы, реальные сообщения СУБД, пути к файлам);
  - не «маскировать» ошибки так, чтобы защитные проверки перестали работать;
  - корректно обрабатывать ошибки аутентификации/авторизации, не оставляя «тихих» провалов.
- При надёжности важно:
  - различать **временные сбои** (таймаут внешнего сервиса) и **постоянные** (невалидные данные);
  - иметь предсказуемое поведение при ошибках: ретраи, деградация, фоллбэки, алерты.

Примеры типичных проблем:

- Вывод пользователю полного текста исключения `SQLException`, содержащего структуру таблиц и названия БД — утечка внутренней информации, полезной атакующему.
- Пустой `except` вокруг проверки токена авторизации: при любой ошибке обработки токена пользователь «проходит» как анонимный или, наоборот, как авторизованный, что ломает модель безопасности.

Лучшие практики:

- На границах (HTTP, CLI) мапить **внутренние исключения на «безопасные» ответы**:
  - логировать полное исключение;
  - пользователю отдавать код и короткое сообщение без деталей реализации.
- Для временных технических ошибок использовать **ограниченные ретраи** и алерты, а не бесконечные циклы.
- Для доменных ошибок — чётко определять, какие из них стоит показывать пользователю, а какие — только логировать.

**Проверь себя (безопасность и надёжность)**  

1. Почему стектрейсы и тексты низкоуровневых исключений не стоит показывать конечному пользователю?
   <details><summary>Ответ</summary>
   Они могут содержать внутренние детали системы (пути к файлам, имена таблиц, версию фреймворка), которые облегчают атаку и не помогают пользователю решить свою задачу. Лучше логировать эти детали во внутренние журналы, а пользователю возвращать нейтральное сообщение и код ошибки.
   </details>

2. Как различие между доменной и технической ошибкой влияет на выбор стратегии ретрая?
   <details><summary>Ответ</summary>
   Доменная ошибка (например, «недостаточно средств») не исправится от повторов — ретрай бессмысленен и может выглядеть как баг. Техническая ошибка (таймаут сети, кратковременная недоступность сервиса) часто временная, и ограниченные ретраи могут повысить надёжность. Путаница ведёт либо к лишним ретраям, либо к преждевременному отказу.
   </details>

### Что будет, если…

Если продолжать возвращать `null` и коды возврата без явных контрактов, то:

- часть проверок будет забыта;
- часть ошибок будет проявляться в совершенно других местах (там, где используют результат);
- при изменении реализации функции (например, добавлении нового типа ошибки) легко сломать десятки мест, которые полагались на старое «по умолчанию» поведение.

### Проверь себя (5.2)

1. Почему пустой `catch`/`except` почти всегда анти‑паттерн?
   <details><summary>Ответ</summary>
   Потому что он скрывает факт ошибки: система делает вид, что всё нормально, хотя что‑то пошло не так. Это затрудняет диагностику проблем, может ломать бизнес‑процесс и делает поведение непредсказуемым. Исключение должно быть либо обработано (с осмысленным действием), либо проброшено/обёрнуто, но не проигнорировано.
   </details>

2. В чём разница между доменной и технической ошибкой, и почему это важно на границах?
   <details><summary>Ответ</summary>
   Доменная ошибка — нарушение бизнес‑правила (например, недостаточно средств), её часто показывают пользователю или учитывают в бизнес‑логике. Техническая — сбой инфраструктуры (нет сети, БД недоступна) и обычно означает, что операцию нужно повторить, деградировать или заалертить поддержку. На границах нужно решать, какое представление этих ошибок отдавать наружу (коды HTTP, тексты сообщений, ретраи).
   </details>

3. Почему возвращать пустую коллекцию лучше, чем `null`, если «ничего не найдено»?
   <details><summary>Ответ</summary>
   Пустая коллекция — всё ещё коллекция, с которой можно безопасно работать (итерироваться, измерять длину). `null` требует отдельной проверки и легко приводит к `NullPointerException`/`AttributeError`. Пустая коллекция делает контракт проще: «функция всегда возвращает список, просто иногда он пустой».
   </details>

### Запомните

- Ошибки ловим **на границах**, а не у каждого вызова; внутри слоёв — бросаем/возвращаем осмысленные типы ошибок.
- `null`/магические коды без явного контракта — анти‑паттерн; лучше `Optional`/`Result` и пустые коллекции.
- Различие между **домеными** и **техническими** ошибками важно для правильной реакции и пользовательского опыта.
