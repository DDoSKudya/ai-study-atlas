## 5.4. Альтернативы и граничные случаи (ошибки)

### Цель раздела

Разобраться, какие существуют **альтернативы** исключениям (checked/unchecked, `Result`‑подход, коды возврата в C и системном коде), в каких случаях они уместны, а также рассмотреть **граничные случаи** и типичные анти‑паттерны: «исключения для нормального потока», `null` vs пустые коллекции, иерархии исключений, которые трудно поддерживать.

### В этом разделе главное

- В языках с checked‑исключениями важно не «оборачиваться от всего», а проектировать контракты.
- В функциональном стиле `Result`/`Either` могут быть удобнее исключений.
- В C и других низкоуровневых API коды возврата неизбежны, но их нужно **изолировать**.
- Пустая коллекция почти всегда лучше `null` для «ничего нет».
- Исключения не должны использоваться для нормативного потока (особенно в горячих участках).

### Термины

| Термин                | Определение |
|-----------------------|-------------|
| **Checked‑исключения** | Исключения, которые компилятор требует явно объявлять/обрабатывать (например, Java `IOException`) |
| **Unchecked‑исключения** | Исключения, которые можно не объявлять в сигнатуре и которые не требуют обязательной обработки |
| **Legacy‑API**        | Старый API (часто на C), использующий коды возврата и глобальные состояния |

### Теория и правила

#### Checked vs unchecked

В Java (и похожих языках) есть:

- checked‑исключения (объявляются в сигнатуре, компилятор требует `throws` или `try/catch`);
- unchecked‑исключения (наследники `RuntimeException`, можно не объявлять).

**Плюсы checked:**

- заставляют **не забыть** про возможные ошибки;
- делают контракт ошибок явным.

**Минусы:**

- при неудачном дизайне API заставляют прокидывать «простыню `throws`» по всему коду;
- склоняют к анти‑паттерну «ловим, логируем и оборачиваем в `RuntimeException`», чтобы «заткнуть» компилятор.

Подход чистого кода:

- checked‑исключения уместны **на границах** (например, файловые операции, сеть), где ошибки действительно часть контракта;
- внутри домена часто лучше использовать unchecked или `Result`‑подобные типы;
- не объявлять десятки checked‑исключений, если вызывающим в реальности всё равно, **какой именно технический сбой** произошёл (например, любая `IOException` может мапиться в один доменный тип «невозможно прочитать конфигурацию»).

#### `Result`/`Either` и похожие подходы

Мы уже говорили о `Result`: это особенно удобно:

- в функциональных языках;
- когда хочется **избежать «квеста» по стектрейсу** и иметь все ветки прямо в коде;
- когда ошибки — **нормальная часть протокола** взаимодействия, а не исключения (например, валидация).

Но:

- слишком глубокая вложенность `match`/`if` по `Result` может сделать код тяжёлым;
- нужна дисциплина (или синтаксический сахар/монады), чтобы **не превратить код в лес `if is_err()`**.

#### Коды возврата в C и legacy

В низкоуровневом коде (C, системные библиотеки):

- исключений может не быть;
- коды возврата — норма (`0` — ок, неноль — ошибка).

Чистый код здесь:

- не пытается «задним числом добавить исключения» везде внутри;
- а **изолирует** такой код на границе:

  - создаёт обёртку, которая превращает коды в исключения или `Result`‑тип;
  - внутри обёртки есть единое место перевода `errno`/код → осмысленное доменное/техническое представление.

Пример (псевдокод):

```c
// C-API
int read_file(const char* path, char* buffer, size_t len); // 0 = ok, !=0 = error
```

Обёртка на более высоком уровне:

```python
def read_file_safe(path: str) -> str:
    rc, data = c_api.read_file(path)
    if rc == 0:
        return data
    if rc == ERR_NOT_FOUND:
        raise FileNotFoundError(path)
    raise IOError(f"Error {rc} reading file {path}")
```

#### Иерархия исключений в проекте

В средних и больших проектах полезно иметь **осмысленную иерархию ошибок**, а не случайный набор `FooError`, `BarException` без структуры.

Подход:

- Ввести **корневой тип** ошибок проекта:
  - `AppError` / `ProjectError` — базовый класс для всех исключений приложения.
- Разделить ошибки по крупным ветвям:
  - **доменные** (`DomainError`, `OrderError`, `PaymentError` и подтипы);
  - **технические** (`InfrastructureError`, `DatabaseError`, `HttpClientError` и т.п.).
- Внутри доменных семейств вводить подтипы **только там, где реально нужны разные реакции**:
  - `InsufficientFundsError`, `OrderAlreadyShippedError`, `OrderNotCancelableError`;
  - если обработка одинакова, подтип не нужен — достаточно одного типа с разными сообщениями.

Именование:

- суффикс **`Error`** или **`Exception`** по конвенции языка/проекта;
- имя включает **домен или подсистему**:
  - `UserNotFoundError`, `InvalidOrderStateError`, `PaymentProviderUnavailableError`;
  - а не просто `NotFoundError`, `InvalidStateError` без контекста.

Преимущества:

- можно **ловить по ветке**:
  - `except PaymentError:` — обработать все ошибки платежей;
  - `except DomainError:` — общая обработка бизнес‑ошибок;
  - `except InfrastructureError:` — логирование и ретраи.
- легче ориентироваться при логировании и метриках (по типу/ветке ошибки).

**Проверь себя (иерархия исключений)**  

1. Зачем выносить общий базовый тип ошибок проекта (`AppError`/`ProjectError`), если есть стандартный `Exception`?
   <details><summary>Ответ</summary>
   Собственный базовый тип позволяет отделить «наши» ошибки от ошибок платформы и библиотек, ловить их одним `except AppError` в нужных местах и при необходимости обрабатывать по‑особому (логировать, мапить в коды ответа). Это помогает не путать системные сбои (например, баги фреймворка) с ожидаемыми ошибками приложения.
   </details>

2. Когда имеет смысл вводить отдельный подтип ошибки, а когда достаточно одного общего типа с разными сообщениями?
   <details><summary>Ответ</summary>
   Подтип нужен, когда для разных ошибок требуется **разная реакция** (другой HTTP‑код, другой текст для пользователя, другая стратегия ретрая) или когда эти ошибки нужно по‑разному считать/фильтровать в логах и метриках. Если реакция одинакова, отдельный подтип даёт мало пользы и только усложняет иерархию.
   </details>

#### Пустая коллекция vs `null` vs `Optional`

Частые варианты при «ничего не найдено»:

- вернуть `null`/`None`;
- вернуть пустую коллекцию;
- вернуть `Optional`/`Option`.

Рекомендации:

- для **коллекций** — почти всегда лучше **пустой список/множество**, чем `null`:
  - можно итерироваться без проверок;
  - код проще (`for item in items` вместо `if items is not None: ...`).
- для **одиночных значений** — `Optional/Option` или доменный тип «нет значения».
- `null` — только там, где:
  - язык/библиотека обязывают;
  - это тщательно документировано;
  - и есть слой‑обёртка, который как можно раньше переводит `null` в более безопасное представление.

#### Исключения для нормального потока

Иногда разработчики используют исключения как «обычный способ ветвления»:

```python
try:
    value = cache.get(key)  # кидает KeyError, если нет
except KeyError:
    value = fetch_from_db(key)
```

Иногда это приемлемо (если библиотека именно так спроектирована), но важно понимать:

- исключения приносят **накладные расходы** (по производительности и по сложности стектрейсов);
- для часто вызываемого кода («горячих путей») лучше иметь **явную проверку** или метод, который не бросает.

### Пошагово: выбираем подход к ошибкам для конкретного случая

Для каждой операции спроси:

1. **Уровень:** это доменный код, инфраструктура или «склейка»?
2. **Частота ошибки:** это ожидаемый сценарий (часто) или редкий сбой?
3. **Требуется ли типизация ошибок?** Нужны ли разные реакции на разные сбои?
4. **Средства языка:** есть ли удобный `Result`/`Option`, есть ли checked‑исключения?

Далее:

- если это **редкий сбой**, который должен «пробиться наверх» → **исключение**;
- если это **обычный сценарий «не найдено» / «не подходит»** → `Optional`/пустая коллекция/`Result` без исключений;
- если это **низкоуровневый С‑API** → коды возврата, но **обёртка** на границе;
- если нужен **полностью контролируемый протокол ошибок** (например, в чистой функции) → `Result`/`Either`.

**Проверь себя (выбор подхода)**  

1. Почему использование исключений для управления нормальным потоком (частые ветвления) считается плохой практикой?
   <details><summary>Ответ</summary>
   Исключения дороже по производительности, усложняют стектрейсы и затрудняют понимание кода: обычный `if/else` читается проще и ожидаемее. Когда исключения используются повсеместно, трудно отличить действительно исключительные ситуации от обычных веток логики, а оптимизация и тестирование становятся сложнее.
   </details>

2. Зачем в legacy с кодами возврата изолировать их на границе, а не менять все вызовы сразу?
   <details><summary>Ответ</summary>
   Мгновенная массовая замена по всему проекту рискованна: легко допустить ошибку и сломать поведение. Обёртка на границе позволяет постепенно «вытеснить» коды возврата, заменяя их более чистым представлением (исключения, `Result`) в новом коде, не ломая существующий API и поведения сразу.
   </details>

### Простыми словами

Нет «серебряной пули» для обработки ошибок. Исключения, `Result`, коды возврата — это разные инструменты. Важно понимать их **сильные и слабые стороны** и выбирать подход, который делает код явным, предсказуемым и тестируемым в данном контексте, а не слепо следовать одному шаблону на все случаи.

### Картинка в голове

Представь ящик с инструментами:

- молоток (исключения) — отлично забивает гвозди (редкие ошибки), но не подходит для ювелирной работы;
- отвёртка (Result) — позволяет аккуратно разбирать и собирать (деликатные ветвления логики);
- пассатижи (коды возврата и низкоуровневые API) — иногда нужны, когда других инструментов нет.

Строить дом одним только молотком — странно.

### Как запомнить

- Checked‑исключения — хороши на границах, но плохо, когда их «слишком много и везде».
- `Result`/`Option` полезны, когда ошибки и отсутствие — **часть обычного протокола**.
- В legacy с кодами возврата важна **изоляция** и постепенная миграция, а не переписывание всего сразу.

### Примеры

**Checked‑исключения (Java):**

```java
public String readFile(String path) throws IOException {
    ...
}
```

Вызов:

```java
try {
    String content = readFile("config.yaml");
} catch (IOException e) {
    // обработка или проброс
}
```

**Result‑подход (Kotlin, псевдокод):**

```kotlin
sealed class PaymentResult {
    data class Success(val receipt: Receipt) : PaymentResult()
    data class Failed(val error: PaymentError) : PaymentResult()
}

fun charge(order: Order): PaymentResult { ... }
```

### Практический сценарий

Ты работаешь с внешним HTTP‑API, которое возвращает:

- 200 с данными;
- 404, если ресурса нет;
- 500, если внутренний сбой.

Можно:

- бросать исключения и для 404, и для 500;
- или:
  - для 404 — возвращать `Optional`/`None` (нормальный «нет ресурса»);
  - для 500 — бросать исключение `RemoteServiceError` (сбой на стороне сервиса).

Так контракт становится яснее: **нет ресурса** — нормальное «пусто», **500** — ошибка.

### Типичные ошибки

- Считать, что один подход (только исключения / только Result) решает все задачи.
- Не различать «редкие» и «частые» ошибки по частоте и важности.
- Не изолировать коды возврата из legacy‑API, размазывая проверки по всему коду.

### Что будет, если…

Если бездумно выбирать один инструмент (например, только исключения) на всё:

- часть кода станет перегружена `try/catch` и стектрейсами;
- местами появится соблазн «заткнуть компилятор» вместо осмысленной обработки;
- другие подходы (Option/Result) так и не будут использованы там, где они упрощают жизнь.

### Проверь себя (5.4)

1. Чем `Result`‑подход отличается от исключений с точки зрения читаемости и тестируемости?
   <details><summary>Ответ</summary>
   `Result` делает все ветки (успех/ошибка) **явными в коде** и позволяет обрабатывать их без механизмов исключений, что упрощает тестирование (обычные проверки значений) и reasoning. Исключения лучше отделяют «редкие» ошибки, но могут скрывать детали в стектрейсах и требуют внимания к месту обработки.
   </details>

2. Почему пустая коллекция предпочтительнее `null` для обозначения «ничего не найдено»?
   <details><summary>Ответ</summary>
   Потому что пустая коллекция — валидный объект, с которым можно безопасно работать (`for`, `len`, фильтрации). `null` требует дополнительной проверки и легко ведёт к runtime‑ошибкам, если её забыть.
   </details>

3. Как бы ты изолировал коды возврата С‑библиотеки в приложении на высокоуровневом языке?
   <details><summary>Ответ</summary>
   Создал бы модуль‑обёртку, который вызывает С‑библиотеку, интерпретирует её коды возврата и переводит их в исключения или `Result`‑тип с осмысленными доменными/техническими именами ошибок. Всё остальное приложение работало бы уже с этими типами, не зная о низкоуровневых кодах.
   </details>

### Запомните

- Checked/unchecked, Result и коды возврата — **инструменты**, а не идеология.
- Для разных уровней (домен, инфраструктура, legacy) уместны разные сочетания этих инструментов.
- Важнее всего **явность и предсказуемость**: ошибки должны быть видны в контрактах и коде, а не прятаться в `null` или «магических» значениях.
