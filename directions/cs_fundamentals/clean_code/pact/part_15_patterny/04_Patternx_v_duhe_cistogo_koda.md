## 15.4. Паттерны в духе чистого кода

### Цель раздела

Показать на практических примерах, **как несколько ключевых паттернов помогают писать чистый код**: Strategy/State, Factory/Builder, Repository/Adapter, Observer/Decorator. Понять, какие запахи они смягчают, как влияют на тестируемость и когда кажутся уместными, но на деле лишние.

### В этом разделе главное

- Strategy/State → вместо ветвлений по типу/статусу.
- Factory/Builder → вместо громоздких конструкторов и ручного создания сложных объектов.
- Repository/Adapter → вместо прямой зависимости бизнес‑кода от БД/HTTP‑клиентов/SDK.
- Observer/Decorator → вместо «запихивания» дополнительного поведения в один класс.

### Термины

| Термин        | Определение |
|--------------|-------------|
| **Context**  | Объект, который использует стратегию/состояние для выполнения операции |
| **Concrete strategy/state** | Конкретная реализация вариативного поведения |
| **Creator / Factory** | Компонент, отвечающий за создание объектов по контракту |
| **Subscriber / Listener** | Подписчик в паттерне Observer |

### 15.4.1 Strategy и State

#### Цель подраздела

Понять, как Strategy и State помогают **разделить вариативное поведение** и устранить длинные ветвления, сохранив при этом читаемость и тестируемость.

#### Теория и правила (коротко)

- **Strategy** — когда есть несколько алгоритмов/подходов:
  - выбор стратегии часто зависит от конфигурации, пользовательского выбора, контекста.
- **State** — когда объект ведёт себя по‑разному в зависимости от **текущего состояния**, и переходы между состояниями — часть логики.

#### Пример Strategy (обработчик платёжных провайдеров)

До:

```python
def charge(payment_method: str, amount: Decimal, payload: dict) -> None:
    if payment_method == "stripe":
        # логика Stripe
        ...
    elif payment_method == "paypal":
        # логика PayPal
        ...
    elif payment_method == "offline":
        # ручная обработка
        ...
    else:
        raise ValueError("Unknown payment method")
```

После:

```python
class PaymentGateway(Protocol):
    def charge(self, amount: Decimal, payload: dict) -> None: ...


class StripeGateway(PaymentGateway):
    def charge(self, amount: Decimal, payload: dict) -> None:
        ...


class PaypalGateway(PaymentGateway):
    def charge(self, amount: Decimal, payload: dict) -> None:
        ...


class OfflineGateway(PaymentGateway):
    def charge(self, amount: Decimal, payload: dict) -> None:
        ...


def charge(gateway: PaymentGateway, amount: Decimal, payload: dict) -> None:
    gateway.charge(amount, payload)
```

Выбор стратегии (gateway) можно делать один раз на уровне конфигурации/фабрики, а бизнес‑логика работает с интерфейсом.

#### Пример State (жизненный цикл заказа)

Если в коде появляется множество проверок вида:

```python
if order.status == "NEW":
    ...
elif order.status == "PAID":
    ...
elif order.status == "SHIPPED":
    ...
```

и таких блоков много, State позволяет:

- вынести поведение в отдельные классы состояний;
- сделать переходы между состояниями явными.

#### Картинка в голове

Strategy — как **сменные насадки на шуруповёрте**: сам шуруповёрт (контекст) один, но биты (алгоритмы) разные.  
State — как **режимы работы устройства**: в режиме «ожидание» и «работа» кнопки ведут себя по‑разному, но режим задаётся явно.

#### Проверь себя

1. В чём практическая разница между Strategy и State?

   <details><summary>Ответ</summary>
   Strategy — про **независимые алгоритмы**, которые можно выбирать и комбинировать (например, разные способы вычисления скидки). State — про **эволюцию состояния одного объекта** во времени: переходы между состояниями и различия поведения зависят от того, в каком состоянии он сейчас находится. Технически они похожи (семейство объектов с общим интерфейсом), но мотивация другая.
   </details>

### 15.4.2 Factory и Builder

#### Цель подраздела

Понять, когда создание объекта становится достаточно сложным, чтобы **вынести логику создания в отдельный компонент** (фабрику/билдер), и как это влияет на чистоту кода.

#### Теория и правила (коротко)

- Фабрика нужна, когда:
  - создание объекта зависит от **условий/конфигурации**;
  - требуется **полиморфный результат** (один интерфейс, разные реализации);
  - мы хотим **скрыть детали создания** от вызывающего кода.
- Builder нужен, когда:
  - у объекта много опций и комбинаций параметров;
  - классический конструктор становится плохо читаемым;
  - хотим пошаговое, читаемое построение.

#### Пример Factory (выбор реализации по окружению)

```python
class Storage(Protocol):
    def save(self, key: str, data: bytes) -> None: ...


class LocalStorage(Storage):
    ...


class S3Storage(Storage):
    ...


def create_storage(env: str) -> Storage:
    if env == "dev":
        return LocalStorage(path="/tmp/app")
    if env == "prod":
        return S3Storage(bucket="app-prod")
    raise ValueError("Unknown env")
```

Вызывающий код работает с интерфейсом `Storage`, а фабрика принимает решение, какую реализацию вернуть.

#### Пример Builder (конфиг отчёта)

Без Builder:

```python
report = Report(
    title="Sales",
    include_vat=True,
    group_by="region",
    with_summary=True,
    currency="USD",
    timezone="Europe/Berlin",
)
```

С Builder:

```python
report = (
    ReportBuilder()
    .with_title("Sales")
    .include_vat()
    .group_by_region()
    .with_summary()
    .with_currency("USD")
    .with_timezone("Europe/Berlin")
    .build()
)
```

Читаемость улучшается, особенно если аргументов становится больше.

#### Проверь себя

1. Почему нельзя слепо заменять любой конструктор с 4+ параметрами на Builder?

   <details><summary>Ответ</summary>
   Иногда эти 4 параметра:
   - обязательны;
   - логически связаны;
   - редко меняются.  
   В таком случае конструктор остаётся понятным, а Builder добавит лишний слой. Builder выигрывает, когда есть **много опций, часть из которых не обязательна**, и мы хотим сделать последовательность настроек читаемой и гибкой.
   </details>

### 15.4.3 Repository и Adapter

#### Цель подраздела

Показать, как Repository и Adapter помогают **изолировать доменную логику от деталей инфраструктуры**, и почему это критично для чистого кода в реальных проектах.

#### Теория и правила (коротко)

- Repository:
  - говорит «я храню и выдаю сущности домена»;
  - скрывает SQL/ORM/HTTP‑детали;
  - даёт тестируемый контракт.
- Adapter:
  - приводит неудобный или чужой интерфейс к нашему;
  - уменьшает количество мест в коде, где мы напрямую зависим от сторонней библиотеки.

#### Картинка в голове

Repository — как **витрина магазина**, а БД — как **склад**. Клиент кода ходит на витрину, а не вглубь склада.  
Adapter — как **переходник для розетки**: внутри — другой стандарт, но для нас интерфейс единый.

### 15.4.4 Observer и Decorator

#### Цель подраздела

Понять, как Observer и Decorator позволяют **расширять поведение** без изменения исходного класса и при этом сохранять читаемость и контролируемость потока событий.

#### Кратко

- **Observer**:
  - полезен, когда есть источник событий и несколько независимых обработчиков;
  - помогает избежать жёсткой связки между источником и слушателями.
- **Decorator**:
  - позволяет оборачивать объект дополнительным поведением (логирование, кеширование, метрики);
  - сохраняет тот же интерфейс, что и исходный объект.

#### Пример Decorator (кеширующий репозиторий)

```python
class CachingUserRepository(UserRepository):
    def __init__(self, inner: UserRepository, cache: Cache):
        self._inner = inner
        self._cache = cache

    def get_balance(self, user_id: int) -> Decimal:
        key = f"user_balance:{user_id}"
        cached = self._cache.get(key)
        if cached is not None:
            return cached
        balance = self._inner.get_balance(user_id)
        self._cache.set(key, balance)
        return balance
```

Бизнес‑код по‑прежнему работает с `UserRepository`, а кеш — отдельный оборачивающий слой.

#### Пример Observer (доменные события)

```python
class DomainEvent(Protocol):
    ...


class EventBus:
    def __init__(self) -> None:
        self._subscribers: dict[type[DomainEvent], list[Callable[[DomainEvent], None]]] = {}

    def subscribe(self, event_type: type[DomainEvent], handler: Callable[[DomainEvent], None]) -> None:
        self._subscribers.setdefault(event_type, []).append(handler)

    def publish(self, event: DomainEvent) -> None:
        for handler in self._subscribers.get(type(event), []):
            handler(event)
```

Источник события (`OrderService`) просто публикует `OrderPaid`, а различные обработчики (отправка письма, начисление бонусов, логирование) подписываются через `EventBus`. Источник не знает, кто именно отреагирует.

#### Практика / реальные сценарии

- **Decorator**:
  - добавить логирование времени выполнения к любому сервису;
  - добавить кеш к «дорогому» репозиторию;
  - добавить подсчёт метрик (количество вызовов, ошибки) к HTTP‑клиенту.

- **Observer**:
  - реакция на доменные события (оплата заказа → отправить чек, начислить бонусы, обновить отчёт);
  - событийная интеграция между bounded‑context’ами (DDD).

#### Типичные ошибки

- Для Decorator:
  - строить **глубокие цепочки обёрток**, где трудно понять, в каком порядке применяются эффекты;
  - смешивать в одном декораторе много разных ответственностей (и кеш, и логирование, и метрики).

- Для Observer:
  - использовать его там, где есть **один жёсткий потребитель** — обычный вызов проще;
  - не контролировать ошибки в обработчиках (один упавший слушатель «роняет» весь поток событий);
  - превращать события в «скрытые зависимости», о которых трудно догадаться, читая источник события.

#### Проверь себя

1. Когда Observer оправдан, а когда обычный вызов метода лучше?

   <details><summary>Ответ</summary>
   Observer оправдан, когда:
   - есть **источник событий** (например, изменения сущностей, доменные события);
   - количество обработчиков может меняться;
   - обработчики не должны знать друг о друге.  
   Если есть один фиксированный обработчик, обычный вызов или композиция часто проще и чище, чем полноценный механизм подписки/отписки.
   </details>

### Проверь себя (паттерны в целом)

1. Какой из перечисленных паттернов ты бы рассмотрел для:
   - а) замены растущего `if` по типу события;
   - б) добавления кеширования поверх существующего сервиса;
   - в) изоляции HTTP‑клиента стороннего сервиса?

   <details><summary>Ответ</summary>
   - а) Strategy или State (в зависимости от того, про варианты алгоритма или про состояния объекта);  
   - б) Decorator (обёртка, добавляющая кеширование);  
   - в) Adapter (обёртка над клиентом, приводящая его к нашему доменному интерфейсу), возможно, в сочетании с Repository, если это доступ к данным.
   </details>

2. В каких случаях разумно предпочесть простую функцию или класс без паттерна, даже если формально можно было бы «узнать» Strategy/Factory/Observer?

   <details><summary>Ответ</summary>
   Когда:
   - вариативность поведения **отсутствует или минимальна** (один сценарий, нет реальных требований к расширяемости);
   - код легко читается и тестируется в текущем виде;
   - добавление паттерна увеличит количество сущностей и уровней косвенности, не давая существенной выгоды.  
   В духе чистого кода мы предпочитаем **минимально достаточную абстракцию** и дорастаем до паттерна только тогда, когда простое решение перестаёт быть достаточным.
   </details>

3. Как связаны части 7, 8, 9 и 15 глобального плана с темой паттернов, если смотреть на них как на единый маршрут?

   <details><summary>Ответ</summary>
   - Часть 7 («Классы и организация кода») даёт базу: SRP, связность, связанность, композиция vs наследование.  
   - Часть 8 («Запахи и рефакторинг») учит видеть симптомы (длинные методы, дублирование, god‑объекты) и приёмы улучшения.  
   - Часть 9 («Границы и дизайн») показывает, как изолировать внешние зависимости и строить эмерджентный дизайн.  
   - Часть 15 («Паттерны и чистый код») даёт **язык и шаблоны**, которыми можно описать и систематизировать найденные решения (Strategy вместо `if`, Repository на границе и т.д.).  
   Вместе это маршрут: сначала учимся писать и организовывать код, затем замечать запахи, потом работать с границами, и только после этого используем паттерны как осмысленные инструменты, а не как цель сами по себе.
   </details>
