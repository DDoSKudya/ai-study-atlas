## 15.2. Когда паттерны усложняют

### Цель раздела

Научиться узнавать ситуации, когда применение паттерна **делает код хуже**: менее понятным, более хрупким и трудным для онбординга. Понять, почему «паттерн ради паттерна» и «на будущее» — частые источники грязного кода.

### В этом разделе главное

- Паттерн — это **дополнительный слой абстракции**, который нужно обслуживать.
- Паттерны особенно опасны, когда:
  - добавляются **преждевременно** («вдруг пригодится ещё один вариант»);
  - появляются **там, где нет реальной вариативности**;
  - используются **как самоцель** («давайте всё сделаем на паттернах»).
- Чистый код предпочитает **простое решение до тех пор, пока оно работает и остаётся понятным**.

### Термины

| Термин                   | Определение |
|--------------------------|-------------|
| **Overengineering**      | Переусложнение решения по сравнению с реальными требованиями |
| **Speculative generality** | «Спекулятивная обобщённость»: абстракции «на будущее», без реальной потребности |
| **Паттерн‑фетишизм**    | Склонность применять паттерны ради самого факта их применения |

### Теория и правила

1. **Каждый паттерн — это стоимость.**

   - Больше классов/файлов → сложнее навигировать.
   - Больше уровней косвенности → сложнее отследить поток данных и вызовов.
   - Больше абстракций → выше порог входа для новичка.

2. **Преждевременное введение паттерна = новый долг.**

   - Когда вариантов поведения ещё один, но уже есть Strategy, добавляется «лишний слой».
   - Когда есть одна реализация интерфейса Repository/Adapter и в обозримом будущем не планируется другая, возможно, прямая реализация проще.

3. **Паттерн должен опираться на факты, а не предположения.**

   - Уже есть **минимум два‑три места повторяющейся логики**.
   - Уже понятно, что **варианты поведения будут расти**.
   - Уже есть **неудовлетворённость текущим кодом** (запахи, сложность тестов).

4. **Когда лучше не вводить паттерн (или отложить):**

   - Прототип/одноразовый скрипт, который не планируется поддерживать.
   - Маленький модуль с одной простой responsibility, который читается отлично без абстракций.
   - Команда не знакома с паттерном и не готова его поддерживать (риски неправильного использования выше выигрыша).

### Пошагово: как проверить, не переусложняешь ли ты

1. Спроси: **сколько вариантов поведения прямо сейчас существует?**
2. Спроси: **есть ли реальные, а не гипотетические требования, которые потребуют новых вариантов?**
3. Подумай: **станет ли код проще для новичка после введения паттерна?**
4. Если ответ:
   - «вариантов один»,
   - «новых пока не планируется»,
   - «структура и так понятна» — скорее всего, паттерн пока не нужен.

### Простыми словами

Переусложнение паттернами похоже на ситуацию, когда ты:

- вместо простой кружки покупаешь **туристическую систему с фильтром, подогревом и складной ручкой**, чтобы пить воду дома;
- технически это «круче», но в быту **только мешает**.

Так же и в коде: если простой класс с одним методом отлично решает задачу, не нужно сразу заворачивать его в три интерфейса и фабрику.

### Картинка в голове

Представь **дом из LEGO**:

- Минималистичный дом из пары крупных блоков легко переносить, перестраивать, объяснять ребёнку.
- Дом, где каждая стенка собрана из десятков микродеталей с хитрыми креплениями, впечатляет, но:
  - его сложнее чинить;
  - одно неверное движение — и рушится пол‑конструкции;
  - новому человеку трудно понять, «что тут от чего зависит».

Паттерны — это как специальные детали LEGO. Если использовать их везде, дом превратится в сложный конструктор без особой нужды.

### Как запомнить

> **Лучший паттерн — тот, который нужен сейчас, а не «когда‑нибудь»**  
> **Если без паттерна код проще читать — паттерн рано**

### Примеры

**Пример 1. Лишний Strategy.**

Есть функция форматирования даты:

```python
def format_date(dt: datetime) -> str:
    return dt.strftime("%Y-%m-%d")
```

Разработчик делает так:

```python
class DateFormatter(Protocol):
    def format(self, dt: datetime) -> str: ...


class IsoDateFormatter(DateFormatter):
    def format(self, dt: datetime) -> str:
        return dt.strftime("%Y-%m-%d")


def format_date(dt: datetime, formatter: DateFormatter | None = None) -> str:
    formatter = formatter or IsoDateFormatter()
    return formatter.format(dt)
```

Если в системе **нет реальной потребности** в других форматах, это — лишний уровень. Читателю придётся разбираться в интерфейсах и классах, вместо того чтобы увидеть одну строку `strftime`.

**Пример 2. Фабрика «на будущее».**

До:

```python
class EmailNotifier:
    ...


notifier = EmailNotifier(smtp_config)
```

После:

```python
class NotifierFactory:
    def create_email_notifier(self) -> EmailNotifier:
        return EmailNotifier(load_smtp_config())


factory = NotifierFactory()
notifier = factory.create_email_notifier()
```

Если есть **только EmailNotifier**, нет планов других каналов и DI‑контейнера, такая фабрика:

- добавляет уровень косвенности;
- не даёт реальной выгоды.

### Практика / реальные сценарии

- Найди в своём коде место, где уже используется паттерн. Спроси:
  - **что именно стало проще** после его введения?
  - можно ли объяснить это коллеге в одном абзаце?
  - если нет — возможно, паттерн применён преждевременно или избыточен.

- На ревью кода:
  - когда видишь новую абстракцию/паттерн, попроси автора объяснить **конкретную боль**, которую он решает;
  - если объяснение сводится к «так красивее / по книжке» — повод обсудить упрощение.

### Типичные ошибки

- Вводить паттерны **раньше тестов и простого рефакторинга**:
  - сначала нужно сделать код простым, понятным, покрытым тестами;
  - только потом поднимать уровень абстракции паттернами.
- Воспринимать отсутствие паттерна как «признак неопытности», даже если код читается прекрасно.
- Учить новичков «думать паттернами» вместо того, чтобы **думать задачами и простыми решениями**.

### Что будет, если…

- …**систематически переусложнять код паттернами**?

  - Онбординг замедлится;
  - ревью превратится в разбор абстракций вместо обсуждения бизнес‑логики;
  - тесты будут сложнее настраивать (больше слоёв для мока/фейка);
  - любой рефакторинг потребует касаться множества файлов.

### Проверь себя

1. Как отличить «подготовку к расширяемости» от speculative generality?

   <details><summary>Ответ</summary>
   Подготовка к расширяемости опирается на **конкретные факты**:
   - уже есть несколько вариантов;
   - есть чёткие планы/требования новых вариантов;
   - текущая структура уже начинает мешать.  
   Speculative generality — это абстракции «на всякий случай», без реальной потребности и сроков. Если аргументы сводятся к «вдруг потом понадобится» — это спекуляция.
   </details>

2. Почему «фабрика на одну реализацию» часто признак переусложнения?

   <details><summary>Ответ</summary>
   Если фабрика всегда возвращает **один и тот же класс**, без условий и полиморфизма, она:
   - добавляет уровень косвенности;
   - не даёт реальной гибкости;
   - создаёт лишний артефакт, который нужно поддерживать.  
   До тех пор, пока нет нескольких реализаций или сложной логики создания, прямой `new` (или вызов конструктора/функции) обычно проще и чище.
   </details>

### Запомните

- Паттерны — мощный инструмент, но **злоупотребление ими превращает код в лабиринт**.
- Прежде чем вводить паттерн, спроси: «станет ли код проще объяснять новичку?».
- В сомнительных случаях **лучше начать с более простого решения** и дорасти до паттерна, когда потребность станет очевидной.
