## 15.1. Когда паттерны помогают

### Цель раздела

Понять, **в каких конкретных ситуациях паттерны реально упрощают жизнь**: уменьшают связанность, снимают дублирующиеся `if`/`switch`, делают систему расширяемой и улучшают читаемость за счёт общего языка.

### В этом разделе главное

- Паттерн — это **ответ на повторяющуюся проблему**, а не украшение.
- Паттерны помогают, когда:
  - **есть вариативность** поведения / алгоритмов;
  - **нужна расширяемость** без изменения старого кода;
  - **надо изолировать нестабильные части** (границы, сторонние API).
- Правильное применение паттерна обычно **уменьшает связанность и повышает локализацию изменений**.

### Термины

| Термин              | Определение |
|---------------------|-------------|
| **Вариативность**   | Наличие нескольких вариантов поведения или алгоритмов, которые должны уметь подменять друг друга |
| **Расширяемость**   | Способность системы добавлять новые возможности **без изменения** существующего кода (Open/Closed) |
| **Локализация изменений** | Свойство системы, при котором изменение требования затрагивает **минимальное число мест** в коде |

### Теория и правила

1. **Паттерн полезен, когда решает конкретную боль.**

   Типичные боли:

   - Разрастающиеся `if/else` или `switch` по типу/статусу/режиму.
   - Повторяющиеся куски кода, отличающиеся только деталями алгоритма.
   - «Залитый» в бизнес‑код внешний API/ORM/SDK, который хочется спрятать за доменной абстракцией.
   - Возрастающее количество опций при создании объектов (конструктор с десятком параметров).

2. **Паттерны помогают реализовать принцип Open/Closed (OCP).**

   - Strategy/State позволяют **добавлять новые варианты поведения** через новые классы.
   - Factory/Builder — добавлять новые способы создания объектов.
   - Observer/Decorator — подключать новое поведение без изменения «ядра».

3. **Паттерны снижают связанность между модулями.**

   - Repository/Adapter изолируют код от деталей хранилища/библиотеки.
   - Фабрики/DI избавляют бизнес‑код от прямых `new` и знания о конкретных реализациях.

4. **Паттерн даёт общий словарь.**

   - Говоря «тут у нас Repository», ты сразу обозначаешь:
     - здесь абстракция доступа к данным;
     - бизнес‑код не должен знать про SQL/REST‑детали;
     - тесты могут использовать фейковую реализацию.

### Пошагово: как понять, нужен ли паттерн

1. Спроси себя: **какая именно боль здесь есть?**
   - Много условий?
   - Сложно тестировать?
   - Сильно зависит от внешней библиотеки?
2. Определи, **повторяется ли эта боль** в нескольких местах.
3. Посмотри, **какой паттерн уже описывает подобную ситуацию**:
   - вариативное поведение → Strategy/State;
   - абстракция данных → Repository;
   - оборачивание стороннего API → Adapter/Facade;
   - много опций создания → Builder/Factory.
4. Оцени: **сделает ли паттерн код проще читать?**
   - исчезнут ли дублирующиеся `if`?
   - станет ли легче добавить новый вариант?
   - упростится ли тестирование?
5. Если «да» по нескольким пунктам — паттерн, вероятно, уместен.

### Простыми словами

Паттерн помогает, когда у тебя **одна и та же головная боль всплывает снова и снова**. Например:

- Уже третий раз ты добавляешь в `if/elif/else` новый способ расчёта скидки → время подумать о Strategy.
- Ты в каждом сервисе пишешь `http_client.get("https://...")` и парсишь JSON вручную → пора сделать Adapter/клиент с нормальным интерфейсом.
- Конструктор класса разросся до 12 параметров, и большая часть из них опциональна → Builder может сделать код читаемым.

Если боли нет или она единична, паттерн может быть избыточен.

### Картинка в голове

Представь **ящик с инструментами**:

- Гаечный ключ, отвёртка, пассатижи — это твои паттерны.
- Если тебе нужно **один раз забить маленький гвоздь**, можно использовать подручный предмет.
- Если ты **регулярно собираешь мебель**, выгодно иметь **набор подходящих инструментов** и уметь быстро выбрать нужный.

Паттерн‑инструмент нужен там, где операция **повторяется** и от её качества зависит надёжность конструкции.

### Как запомнить

Формула:

> **Нужен паттерн = повторяющаяся боль + вариативность + желание локализовать изменения**

Если одного из трёх нет — возможно, ещё рано.

### Примеры

**Пример 1. Замена цепочки `if` на Strategy (скидки).**

До:

```python
def calculate_discount(order, user):
    if user.is_vip:
        return order.total * Decimal("0.1")
    if user.has_coupon:
        return order.total * Decimal("0.05")
    if order.total > 10_000:
        return order.total * Decimal("0.02")
    return Decimal("0")
```

Через некоторое время появляются ещё 3 вида скидок, `if` разрастается, его сложно тестировать и расширять.

После (с Patten Strategy):

```python
class DiscountStrategy(Protocol):
    def calculate(self, order, user) -> Decimal: ...


class VipDiscount(DiscountStrategy):
    def calculate(self, order, user) -> Decimal:
        return order.total * Decimal("0.1") if user.is_vip else Decimal("0")


class CouponDiscount(DiscountStrategy):
    def calculate(self, order, user) -> Decimal:
        return order.total * Decimal("0.05") if user.has_coupon else Decimal("0")


class VolumeDiscount(DiscountStrategy):
    def calculate(self, order, user) -> Decimal:
        return order.total * Decimal("0.02") if order.total > 10_000 else Decimal("0")


class DiscountCalculator:
    def __init__(self, strategies: list[DiscountStrategy]):
        self._strategies = strategies

    def calculate(self, order, user) -> Decimal:
        return sum(strategy.calculate(order, user) for strategy in self._strategies)
```

Теперь:

- Добавление новой скидки = новый класс + регистрация в списке стратегий.
- Логику каждой скидки легко тестировать отдельно.

**Пример 2. Repository для изоляции БД.**

До:

```python
def get_user_balance(user_id: int) -> Decimal:
    row = db.execute("SELECT balance FROM users WHERE id = %s", [user_id])
    return row["balance"]
```

SQL‑строка размазана по коду, тестировать сложно, смена БД больно.

После:

```python
class UserRepository(Protocol):
    def get_balance(self, user_id: int) -> Decimal: ...


class SqlUserRepository(UserRepository):
    def __init__(self, db):
        self._db = db

    def get_balance(self, user_id: int) -> Decimal:
        row = self._db.execute("SELECT balance FROM users WHERE id = %s", [user_id])
        return row["balance"]
```

В бизнес‑коде:

```python
def can_place_order(repo: UserRepository, user_id: int, order_total: Decimal) -> bool:
    return repo.get_balance(user_id) >= order_total
```

Здесь Repository:

- изолирует SQL;
- даёт явный контракт;
- упрощает тесты (можно подставить фейковый репозиторий).

### Практика / реальные сценарии

- В твоём проекте есть длинный `switch` по типу операции или статусу заказа.  
  Подумай:
  - можно ли превратить его в семействo Strategy/State;
  - какие классы понадобятся;
  - станет ли читателю проще понять, какие бывают варианты.

- В нескольких местах кода работа с одной и той же сторонней библиотекой (например, платежный API) повторяется похожим образом.  
  Вопрос:
  - можно ли выделить Adapter/клиент и сделать бизнес‑код независимым от деталей конкретной библиотеки?

### Типичные ошибки

- «Натягивать» паттерн на место, где реально **нет вариативности**: один сценарий — один паттерн — лишний слой.
- Вводить Strategy/State отложенно (когда уже пять разных веток с особыми случаями и тесты не покрывают всё).
- Пытаться решить **неструктурные проблемы** (плохие имена, отсутствие тестов) внедрением паттерна вместо упрощения.

### Что будет, если…

- …**игнорировать подходящие паттерны** и держать всё на `if` и глобальных вызовах?

  - Увеличится **цикломатическая сложность**;
  - тесты будут громоздкими и хрупкими;
  - каждый новый вариант поведения будет требовать правок в нескольких местах (shotgun surgery).

- …**подходить к паттернам как к чек‑листу («надо где‑то применить Strategy, где‑то Observer»)**?

  - Появятся лишние абстракции;
  - новички будут тратить больше времени на разбор архитектуры;
  - реальные проблемы читаемости могут не решиться.

### Проверь себя

1. Какой признак кода чаще всего говорит, что пора подумать о Strategy/State?

   <details><summary>Ответ</summary>
   Разрастающиеся `if/elif/else` или `switch` по типу/статусу/режиму, которые:
   - появляются в нескольких местах;
   - часто меняются;
   - становятся трудно покрываемыми тестами.  
   Это сигнал, что поведение можно вынести в отдельные объекты (стратегии/состояния) и расширять через добавление новых классов.
   </details>

2. Почему Repository считается хорошим союзником чистого кода?

   <details><summary>Ответ</summary>
   Repository:
   - **отделяет доменную логику от инфраструктуры** (SQL, HTTP, ORM‑детали);
   - даёт явный интерфейс доступа к данным, основанный на терминах домена;
   - упрощает тестирование (можно подменять реализацию);
   - локализует изменения, связанные с БД или внешним API.  
   Всё это повышает читаемость, тестируемость и снижает связанность.
   </details>

### Запомните

- Паттерн полезен только тогда, когда **решает конкретную повторяющуюся проблему**.
- Главное в паттерне — **уменьшить связанность, локализовать изменения и улучшить читаемость**, а не «соответствовать книжке».
- Хорошее применение паттерна обычно **делает код короче и понятнее** на уровне сценариев, пусть количество классов формально и растёт.
