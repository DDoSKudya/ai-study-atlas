## 15.3. Антипаттерны применения паттернов

### Цель раздела

Разобрать **распространённые ошибки** при использовании паттернов: когда формально паттерн вроде бы есть, но его применение противоречит целям чистого кода. Научиться узнавать такие случаи и предлагать более простые и чистые альтернативы.

### В этом разделе главное

- Singleton почти всегда = **глобальное состояние**.
- Наследование из «любви к паттернам» часто хуже композиции.
- «Паттерн на каждый случай» делает архитектуру тяжёлой и негибкой.
- Чистый код предпочитает **минимально достаточную абстракцию**, а не «максимальное количество паттернов».

### Термины

| Термин                | Определение |
|-----------------------|-------------|
| **Global state**      | Глобально доступное изменяемое состояние, влияющее на поведение в разных частях программы |
| **God object**        | Класс, который «знает и делает всё» (слишком много ответственности и зависимостей) |
| **Иерархия ради иерархии** | Наследование, введённое без реальной потребности в полиморфизме и общем контракте |

### Теория и правила

1. **Singleton → глобальное состояние и трудности тестирования.**

   - Трудно контролировать порядок создания/очистки.
   - Трудно подменять в тестах (особенно в многопоточном контексте).
   - Зависимости от Singleton размазываются по коду, что затрудняет рефакторинг.

2. **Наследование как средство переиспользования — антипаттерн.**

   - Когда класс наследуется не по принципу «является», а ради заимствования кода, это почти всегда должен быть **композит/делегат**, а не базовый класс.
   - Глубокое наследование усложняет чтение, отладку и изменение поведения.

3. **«Паттерн‑суп» — когда каждый модуль соответствует какому‑то паттерну.**

   - Обилие «Manager‑ов», «Controller‑ов», «Factory‑ов» без ясных ролей ведёт к god‑объектам и размытым ответственностям.

4. **Золотое правило: узнаваемый паттерн ≠ автоматически хороший дизайн.**

   - Важно **как именно он применён**:
     - есть ли чёткий контракт;
     - понятны ли границы;
     - не скрывается ли за ним глобальное состояние или лишняя сложность.

### Примеры

**Пример 1. Singleton как глобальный логгер.**

```python
class Logger:
    _instance: "Logger | None" = None

    def __init__(self):
        self._file = open("log.txt", "a")

    @classmethod
    def instance(cls) -> "Logger":
        if cls._instance is None:
            cls._instance = Logger()
        return cls._instance

    def log(self, message: str) -> None:
        self._file.write(message + "\n")
        self._file.flush()
```

По коду:

```python
Logger.instance().log("Something happened")
```

Проблемы:

- глобальное состояние файла;
- трудно заменить логгер на другой в тестах;
- трудно контролировать время закрытия файла.

Чище:

- передавать логгер через зависимости (конструктор/параметры);
- использовать DI‑контейнер или настройку окружения вместо Singleton.

**Пример 2. Наследование ради кода.**

```python
class BaseService:
    def log(self, message: str):
        print(message)


class UserService(BaseService):
    def create_user(self, ...):
        self.log("Creating user")
        ...
```

Через пару месяцев:

- появляется `OrderService`, тоже наследуется от `BaseService`;
- в `BaseService` появляется куча общих методов:
  - работа с конфигом, кешем, логами и т.д.

Получается god‑база, которую меняют все, и одна правка может сломать несколько сервисов. Лучше:

- иметь `Logger`/`Config` как зависимости;
- передавать их через конструктор или DI.

### Практика / реальные сценарии

- Просканируй проект на наличие Singleton‑ов. Для каждого спроси:
  - действительно ли нужен **один глобальный экземпляр**?
  - можно ли сделать зависимость явной (через параметры/конструктор)?
  - как это повлияет на тесты и изоляцию?

- Посмотри на иерархии наследования:
  - есть ли классы, которые наследуются только ради пары методов?
  - можно ли заменить наследование композицией?

### Типичные ошибки

- «Всё должно быть Singleton‑ом, чтобы не создавать много объектов»:
  - в современных языках создание объектов дешёвое;
  - гораздо дороже обходиться с глобальным состоянием и сложностью тестов.
- Раздувание базовых абстрактных классов, чтобы «всем было удобно от них наследоваться».
- Создание «паттерн‑зоопарка»:
  - `UserManagerFactoryControllerService`, в котором трудно понять реальную роль.

### Что будет, если…

- …**широко использовать Singleton и глобальные сингл‑объекты**?

  - Тесты станут зависеть от глобального состояния и друг от друга;
  - параллельный запуск тестов и многопоточность станут источником трудноуловимых багов;
  - рефакторинг и замена зависимостей будут крайне болезненными.

### Проверь себя

1. Какой простой вопрос помогает понять, что Singleton используется как глобальное состояние?

   <details><summary>Ответ</summary>
   «Что произойдёт, если два теста/два потока одновременно изменят состояние Singleton‑объекта?»  
   Если ответ — «может быть конфликт», «зависит от порядка», «тесты начнут влиять друг на друга» — это признак глобального состояния, а не безвредной «одиночной» сущности.
   </details>

2. Почему наследование ради переиспользования кода (без отношения «является») считается антипаттерном?

   <details><summary>Ответ</summary>
   Наследование жёстко связывает классы:
   - изменения в базовом классе затрагивают всех наследников;
   - сложно контролировать, какие именно методы и поля реально нужны каждому наследнику;  
   при этом повторное использование кода можно решить **композицией**: выделить отдельный класс с нужными методами и использовать его как зависимость. Композиция даёт больше гибкости и меньше неожиданных эффектов.
   </details>

### Запомните

- Singleton, глубокие иерархии наследования и «паттерн‑суп» — частые источники грязного кода.
- Используйте паттерны **осознанно и точечно**, а не как обязательный слой поверх любого кода.
- В сомнительных случаях задавай вопрос: «Это улучшает читаемость и тестируемость или делает хуже?» — ответ важнее формального «соответствия паттерну».
