## 11.1. Пример полного рефакторинга

### Цель раздела

Разобрать реалистичный пример «грязного» модуля и пройти **цепочку маленьких улучшений**: от длинной функции и плохих имён до более чистой структуры с функциями/классами и тестами, не меняя наблюдаемого поведения.

### В этом разделе главное

- Рефакторинг — это **последовательность маленьких шагов**, а не «магия одним коммитом».
- Каждый шаг:
  - **сохраняет поведение** (тесты должны оставаться зелёными);
  - решает **одну конкретную проблему** (имя, дублирование, выделение фазы…).
- Важно уметь:
  - видеть **запах**;
  - выбирать **конкретный приём**;
  - выполнять его аккуратно, **не расползаясь** на всё вокруг.

### Термины

| Термин | Определение |
|--------|-------------|
| **Исходный грязный модуль** | Код, в котором намеренно собраны типичные проблемы: длинные функции, дублирование, смешение уровней |
| **Шаг рефакторинга** | Маленькое изменение структуры без изменения поведения (по возможности проверяемое тестами) |
| **Инвариант поведения** | Обещание «до и после шага все тесты проходят, внешние сценарии работают одинаково» |

### Теория и правила

1. **Начни с работающего кода.**  
   Рефакторинг имеет смысл, когда:
   - код **работает сейчас** (есть хоть какая‑то версия, проходящая тесты или реальные сценарии);
   - ты хочешь сделать его **проще для понимания и изменения**.

2. **Сначала тесты, потом изменения.**
   - Если тестов нет — добавь хотя бы **минимальный набор**:
     - один–два ключевых сценария;
     - несколько граничных случаев.
   - Тесты должны «запомнить» текущее поведение, чтобы можно было смело менять структуру.

3. **Один шаг — один приём.**
   - Не делай «всё сразу»:
     - сначала переименуй;
     - потом вынеси метод;
     - потом введи класс;
     - потом поменяй контракт и т.д.
   - Один шаг = один логический коммит → проще откатить/понять/ревьюить.

4. **Верх‑вниз: сначала читаем, потом режем.**
   - Прочитай функцию/модуль **как рассказ**:
     - где нарушена последовательность «от общего к частному»;
     - где дублируется код;
     - где уровни абстракции перепутаны.
   - Принимай решение, **что именно улучшить первым** (часто — самое больное место по читаемости).

### Пошагово: упрощённый сценарий рефакторинга

Возьмём упрощённый пример (на псевдо‑Python), но идеи применимы к любому языку.

#### Исходный код (с запахами)

```python
def process_order(order, user, db, logger):
    # логируем
    logger.info(f"Start processing order {order.id} for user {user.id}")

    # считаем сумму
    total = 0
    for item in order.items:
        if item.type == "product":
            total += item.price * item.quantity
        elif item.type == "delivery":
            if item.region == "remote":
                total += item.price * 1.5
            else:
                total += item.price
        else:
            logger.warning(f"Unknown item type {item.type} in order {order.id}")

    # скидка
    if user.vip:
        total = total * 0.9

    # сохранение в БД
    try:
        db.execute(
            "INSERT INTO orders (id, user_id, total) VALUES (%s, %s, %s)",
            (order.id, user.id, total),
        )
        db.commit()
        logger.info(f"Order {order.id} processed with total {total}")
        return True
    except Exception as e:
        logger.error(f"Failed to process order {order.id}: {e}")
        db.rollback()
        return False
```

Запахи:

- длинная функция, делающая **всё сразу**: логирование, расчёт, скидки, работа с БД;
- смешение уровней абстракции (бизнес‑логика, SQL, логирование вперемешку);
- «магические» строки (`"product"`, `"delivery"`, `"remote"`);
- обработка ошибок вперемешку с бизнес‑логикой.

#### Шаг 0. Добавляем минимальные тесты

Мы не будем писать здесь весь тестовый модуль, важна идея:

- тест на успешный кейс;
- тест на неизвестный тип позиции;
- тест на ошибку БД.

Главное: **запомнить текущее поведение** функции `process_order`.

#### Шаг 1. Выделяем расчёт суммы

Цель шага — отделить **бизнес‑расчёт** от логирования и работы с БД.

```python
def calculate_order_total(order, user):
    total = 0
    for item in order.items:
        if item.type == "product":
            total += item.price * item.quantity
        elif item.type == "delivery":
            if item.region == "remote":
                total += item.price * 1.5
            else:
                total += item.price
        else:
            # пока только логируем, не меняем поведение
            # (обработка ошибки останется в process_order)
            continue

    if user.vip:
        total = total * 0.9

    return total
```

`process_order` теперь использует эту функцию:

```python
def process_order(order, user, db, logger):
    logger.info(f"Start processing order {order.id} for user {user.id}")

    total = calculate_order_total(order, user)

    try:
        db.execute(
            "INSERT INTO orders (id, user_id, total) VALUES (%s, %s, %s)",
            (order.id, user.id, total),
        )
        db.commit()
        logger.info(f"Order {order.id} processed with total {total}")
        return True
    except Exception as e:
        logger.error(f"Failed to process order {order.id}: {e}")
        db.rollback()
        return False
```

Инвариант: **тесты должны пройти без изменений**.

#### Шаг 2. Улучшаем расчёт: один уровень абстракции

Сейчас `calculate_order_total` всё ещё смешивает:

- логику по типам позиций;
- скидку для VIP.

Разобьём на две функции:

```python
def calculate_items_total(order):
    total = 0
    for item in order.items:
        if item.type == "product":
            total += item.price * item.quantity
        elif item.type == "delivery":
            total += calculate_delivery_price(item)
        else:
            # пока игнорируем неизвестные типы — как и раньше
            continue
    return total


def calculate_delivery_price(item):
    if item.region == "remote":
        return item.price * 1.5
    return item.price


def apply_user_discount(total, user):
    if user.vip:
        return total * 0.9
    return total


def calculate_order_total(order, user):
    items_total = calculate_items_total(order)
    return apply_user_discount(items_total, user)
```

Теперь каждая функция делает **одну вещь** на своём уровне.

#### Шаг 3. Выносим работу с БД в отдельный компонент

SQL‑логика смешана с обработкой результата.

Вводим `OrderRepository`:

```python
class OrderRepository:
    def __init__(self, db):
        self._db = db

    def save_order(self, order_id, user_id, total):
        self._db.execute(
            "INSERT INTO orders (id, user_id, total) VALUES (%s, %s, %s)",
            (order_id, user_id, total),
        )
        self._db.commit()
```

И переписываем `process_order`:

```python
def process_order(order, user, order_repository, logger):
    logger.info(f"Start processing order {order.id} for user {user.id}")

    total = calculate_order_total(order, user)

    try:
        order_repository.save_order(order.id, user.id, total)
        logger.info(f"Order {order.id} processed with total {total}")
        return True
    except Exception as e:
        logger.error(f"Failed to process order {order.id}: {e}")
        # откат теперь внутри репозитория или внешней транзакции —
        # в примере опустим детали, поведение тестов должно сохраниться
        return False
```

Мы **сузили** ответственность функции `process_order`:

- «посчитать → сохранить → залогировать».

#### Шаг 4. Анализируем результат и решаем, когда остановиться

После этих шагов:

- расчёт вынесен в отдельные маленькие функции;
- работа с БД спрятана за репозиторием;
- `process_order` стала короче и описывает **сценарий высокого уровня**.

Можно продолжать:

- сделать отдельный сервис `OrderService`;
- улучшить обработку ошибок (доменные исключения вместо `Exception`);
- вынести магические строки в константы/enum.

Но уже сейчас код **значительно чище**, и каждый следующий шаг можно делать по мере необходимости.

#### Мини‑чек‑лист пошагового улучшения

Используй этот короткий список как «рутинный сценарий» при любом рефакторинге:

1. **Понять поведение.**  
   Прочитай код, прогоняя в голове реальные входные данные или ручной сценарий. Убедись, что понимаешь, что сейчас делает этот участок и какие есть граничные случаи.
2. **Зафиксировать поведение тестами.**  
   Добавь/обнови хотя бы 2–3 теста: нормальный сценарий, один–два граничных случая и нештатный сценарий с ошибкой. Это твой инвариант поведения.
3. **Выбрать один приём.**  
   Определи, что даст наибольший выигрыш за один шаг:
   - `extract method` / `extract function`;
   - `rename` (имя класса/функции/переменной);
   - `introduce class` / `repository` / `service`;
   - устранение дублирования по правилу трёх.
4. **Сделать маленький шаг.**  
   Применить выбранный приём только там, где планировал; не «подхватывать» соседние места в этом же шаге.
5. **Прогнать тесты.**  
   Все тесты должны оставаться зелёными. Если что‑то сломалось — либо шаг слишком большой, либо инвариант поведения был понят не до конца.
6. **Оценить результат.**  
   Стал ли код:
   - короче;
   - понятнее;
   - легче для изменения?  
   Если да — можно либо остановиться, либо наметить **ещё один маленький шаг**; если нет — вернись и пересмотри изменение.

**Проверь себя (чек‑лист)**

1. На каком шаге мини‑чек‑листа чаще всего «экономят» начинающие разработчики и к чему это приводит?
   <details><summary>Ответ</summary>
   Часто пропускают шаги 2 и 5: не добавляют/обновляют тесты перед изменением и не прогоняют их после. В результате рефакторинг превращается в скрытую перепись поведения: код становится внешне чище, но появляются незаметные регрессии, которые потом всплывают в проде или на стыке с другими модулями.
   </details>

2. Почему важно заранее ограничить себя одним приёмом на шаг (extract, rename и т.д.), а не пытаться «применить весь чек‑лист сразу»?
   <details><summary>Ответ</summary>
   Один приём на шаг упрощает контроль над изменениями: легче понять, что именно повлияло на поведение и читаемость, проще откатить правку и объяснить её на ревью. Когда в одном шаге смешиваются несколько приёмов, любые ошибки расплываются по большому диффу и их сложнее обнаружить и исправить.
   </details>

### Простыми словами

Пошаговое улучшение — это как **уборка комнаты**:

- сначала убираешь самые крупные и мешающие вещи (мусор, разбросанные вещи);
- потом раскладываешь по местам;
- потом уже можно заняться деталями (вытереть пыль, разложить по цветам).

Если пытаться сделать всё идеально сразу, легко **устать и бросить на полпути**. Маленькие шаги дают результат на каждом этапе и позволяют **останавливаться в разумный момент**.

### Картинка в голове

Представь старую фотографию в низком качестве:

- сначала ты применяешь грубый фильтр «сделать резче» (как первый шаг рефакторинга);
- потом понемногу правишь контраст, яркость, баланс белого (последующие шаги);
- на каком‑то этапе фото уже **достаточно хорошее для печати**, и дальнейшая правка даёт всё меньше эффекта.

Код — такая же фотография: **улучшения уменьшают боль**, но после некоторого момента отдача снижается, и можно остановиться.

### Как запомнить

- **Сначала сделай, чтобы работало, потом — чтобы было понятнее.**
- **Один шаг — один приём, один инвариант поведения.**
- **Тесты до и после — гарантия, что «улучшение» не сломало систему.**

### Практика / реальные сценарии

- В проекте есть «монструозный» метод контроллера:
  - сначала выносим расчёты в отдельные функции;
  - затем выносим обращение к БД в репозитории;
  - затем добавляем слой сервиса;
  - на каждом шаге гоняем тесты.
- В legacy‑коде слой UI напрямую ходит в БД:
  - шагами вводим слой доменных сервисов и репозиториев;
  - UI начинает говорить только с сервисом;
  - старые вызовы к БД постепенно выносятся в один модуль.

### Типичные ошибки

- Пытаться «отрефакторить всё» одним гигантским коммитом:
  - сложно ревьюить;
  - сложно откатывать;
  - высок риск ошибок.
- Рефакторить **без тестов**:
  - нет уверенности, что поведение не изменилось;
  - любые улучшения превращаются в риск.
- Одновременно:
  - менять структуру;
  - добавлять фичи;
  - чинить баги — всё в одном шаге.

### Что будет, если…

- **Если всегда откладывать рефакторинг «на потом»?**  
  Код будет постепенно зарастать сложностью, и в какой‑то момент любое изменение станет слишком дорогим и рискованным.
- **Если рефакторить без ясного плана и маленьких шагов?**  
  Легко сломать поведение, потратить много времени и не получить ощутимого улучшения, а команда начнёт «бояться» любых изменений структуры.

### Проверь себя (11.1)

1. Почему важно начинать рефакторинг с добавления тестов, даже если хочется сразу «просто переписать всё красиво»?
   <details><summary>Ответ</summary>
   Тесты фиксируют текущее поведение и дают «сигнал тревоги», если при переписывании что‑то сломается. Без них любое изменение превращается в риск: ты можешь сделать код красивее внешне, но незаметно поменять его смысл. Тогда не будет ни уверенности, ни аргументов, что улучшение действительно безопасно.
   </details>

2. Зачем ограничивать один шаг рефакторинга одним приёмом (например, только «extract method» или только «rename»)?
   <details><summary>Ответ</summary>
   Один приём в одном шаге упрощает: (1) откат изменений, если что‑то пошло не так; (2) код‑ревью (понятно, что именно изменилось и зачем); (3) поиск источника регрессии. Когда много разных изменений смешаны, трудно понять, где именно возникла ошибка.
   </details>

3. Почему полезно сначала выделять чистые функции (например, расчёт суммы), а уже потом двигаться к классам и слоям?
   <details><summary>Ответ</summary>
   Чистые функции легче тестировать и переиспользовать. Они уменьшают связность и делают поведение более прозрачным. Когда расчёты вынесены, дальнейшее выделение классов и слоёв опирается на уже разделённые по смыслу куски логики, а не на сплошной монолитный блок.
   </details>

### Запомните

- **Рефакторинг — это не «переписать всё», а «пошагово улучшить без изменения поведения».**
- **Каждый шаг должен быть маленьким, понятным и проверяемым.**
- **Лучший индикатор успешного шага — код стал понятнее, а тесты всё ещё зелёные.**
