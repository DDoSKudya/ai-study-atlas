## 7.5. Типичные ошибки и граничные случаи (классы)

### Цель раздела

Научиться распознавать **часто встречающиеся ошибки** в дизайне классов (God Object, круговые зависимости, неправильное наследование и др.), понимать их последствия и знать, когда большие или «толстые» классы всё же допустимы.

### В этом разделе главное

- **God Object** — симптом того, что множество обязанностей слиплись в одном классе.
- **Круговые зависимости** усложняют инициализацию, тестирование и понимание системы.
- Наследование, применённое ради переиспользования кода без отношения «является», создаёт хрупкую иерархию.
- Иногда «большой» класс (например, DTO) допустим — важно понимать, **чем он отличается** от God Object.

### Термины

| Термин                | Определение |
|-----------------------|------------|
| **God Object**        | Класс, который «знает и делает всё» в системе |
| **Круговая зависимость** | Ситуация, когда два (или более) класса зависят друг от друга напрямую или через цепочку |
| **DTO (Data Transfer Object)** | Объект для переноса данных без поведения (или с минимальным) |

### Теория и правила

**God Object.**  
Признаки:

- много (десятки) полей;
- множество методов, практически не пересекающихся по используемым полям;
- класс фигурирует почти во всех фичах и багфиксах;
- тесты для него громоздки и покрывают совершенно разные сценарии.

Последствия:

- любое изменение становится рискованным;
- тесты либо огромные и хрупкие, либо отсутствуют;
- команда боится трогать этот класс.

**Круговые зависимости.**  
Когда `A` зависит от `B`, а `B` зависит от `A` (напрямую или через цепочку классов), возникают проблемы:

- сложно инициализировать объекты (кто создаётся первым?);
- в некоторых языках возникают проблемы на уровне модулей (импорты);
- при тестировании приходится поднимать «полмира» ради проверки одной функции.

Обычно круг разрывается:

- введением **абстракции** (интерфейса);
- выделением общей зависимости, от которой оба класса зависят;
- изменением направления зависимости (сверху вниз по слоям).

**Граничный случай: большие DTO.**  
Иногда существуют объекты с множеством полей, но **минимальным поведением** (например, `CustomerProfile` с десятками атрибутов). Это:

- не обязательно God Object, если:
  - объект не содержит разнообразного поведения;
  - изменения в бизнес‑логике происходят **в сервисах**, а не внутри DTO;
  - поля логично относятся к одной доменной сущности.

**Утилитарные классы (utility).**  
Часто в коде встречаются классы вида `StringUtils`, `DateUtils`, `MathHelper` с набором статических методов. Они могут быть уместны, если:

- действительно инкапсулируют **общие операции над одним типом/доменом**;
- не тянут за собой состояние и тяжёлые зависимости;
- не превращаются в свалку «всего, что некуда положить».

Если в таком классе появляются:

- методы, не связанные с основной темой (`StringUtils` вдруг содержит функции работы с файлами);
- зависимости от БД, сетевых клиентов, очередей и т.п.;

— это сигнал, что утилитарный класс стал God Object‑ом «по умолчанию» и ответственность пора разделять (выносить отдельные сервисы или модули).

**Feature Envy (зависть к чужим данным).**  
Запах проявляется, когда метод одного класса:

- почти не использует собственные поля;
- при этом активно дергает поля и методы другого объекта (`other.getX()`, `other.getY()` и т.п.).

Пример:

```python
class Invoice:
    def __init__(self, customer, amount):
        self.customer = customer
        self.amount = amount


class InvoicePrinter:
    def print_invoice(self, invoice):
        # метод явно «завидует» данным Invoice и Customer
        print(f"Customer: {invoice.customer.name}")
        print(f"Address: {invoice.customer.address}")
        print(f"Amount: {invoice.amount}")
```

Лучше переместить ответственность ближе к данным:

```python
class Invoice:
    def __init__(self, customer, amount):
        self.customer = customer
        self.amount = amount

    def as_printable_lines(self):
        return [
            f"Customer: {self.customer.name}",
            f"Address: {self.customer.address}",
            f"Amount: {self.amount}",
        ]
```

Теперь «знание о том, как печатать счёт», живёт там, где находятся данные, и меньше зависит от внешних утилит.

**Inappropriate Intimacy (неподобающая близость).**  
Запах возникает, когда:

- классы лезут к приватным деталям друг друга (часто через `friend`‑механизмы, protected‑наследование, «дырявые» геттеры);
- множество методов существует только ради обхода инкапсуляции (`getInternal*`, «полу‑приватные» API для «особого» друга).

Пример:

```python
class Account:
    def __init__(self, balance):
        self._balance = balance

    def _set_balance_directly_for_migration(self, new_balance):
        self._balance = new_balance


class AccountMigrator:
    def migrate(self, account, legacy_balance):
        account._set_balance_directly_for_migration(legacy_balance)
```

Здесь `AccountMigrator` зависит от внутреннего механизма изменения баланса; любое изменение инвариантов в `Account` легко обойдётся этим методом. Лучше ввести **явный контракт**:

```python
class Account:
    def restore_from_legacy(self, legacy_balance):
        # здесь можно проверить инварианты и зафиксировать правила
        self._balance = legacy_balance


class AccountMigrator:
    def migrate(self, account, legacy_balance):
        account.restore_from_legacy(legacy_balance)
```

Теперь мигратор опирается на публичное (или ограниченно публичное) API, а инварианты контролирует сам класс `Account`.

### Пошагово: как разруливать God Object

1. Собери **список обязанностей** класса: какие сценарии он обслуживает.
2. Сгруппируй поля и методы по обязанностям (как в 7.2 и 7.3).
3. Для каждой группы придумай **отдельное имя класса**.
4. Пошагово выделяй новые классы, оставляя старый как фасад или полностью упраздняя.
5. При каждом шаге запускай тесты (см. часть 6 и 8).

#### Мини‑пример: разбор «толстого» контроллера

Исходный класс:

```python
class OrdersController:
    def __init__(self, db, http_client, tax_service, logger):
        self._db = db
        self._http_client = http_client
        self._tax_service = tax_service
        self._logger = logger

    def post_order(self, request):
        # HTTP-детали
        user_id = request.headers.get("X-User-Id")
        if not user_id:
            return {"status": 400, "error": "Missing user id"}

        # парсинг и валидация
        payload = request.json()
        # ... длинная ручная валидация ...

        # доменная логика
        items = payload["items"]
        subtotal = sum(item["price"] * item["qty"] for item in items)
        tax = self._tax_service.calculate_tax(subtotal, payload["country"])
        total = subtotal + tax

        # сохранение
        order_id = self._db.insert_order(user_id, items, total)

        # вызов внешнего сервиса доставки
        self._http_client.post("https://shipping/api", json={"order_id": order_id})

        # логирование
        self._logger.info("Created order %s", order_id)

        return {"status": 201, "id": order_id}
```

Здесь в одном классе и одном методе перемешаны:

- HTTP‑детали (заголовки, коды статусов, структура ответа);
- парсинг и валидация входных данных;
- доменная логика цен и налогов;
- доступ к БД;
- интеграция с внешним API;
- логирование.

Пошаговое разбиение:

1. Вынести доменную логику и сохранение в `OrderService` (доменный слой).
2. Вынести работу с БД в `OrderRepository`.
3. Вынести интеграцию с внешним API в `ShippingGateway`.
4. Оставить в `OrdersController` только:
   - чтение HTTP‑запроса;
   - вызов доменного сервиса с уже провалидированными данными;
   - формирование HTTP‑ответа.

В итоге:

- контроллер простой, легко тестируется как слой адаптера (см. часть 9 о границах);
- доменная логика собрана в одном месте, её проще развивать и покрывать юнит‑тестами;
- зависимости по слоям становятся понятнее: **контроллер → сервис → репозиторий/шлюз**, а не «всё во всём».

### Простыми словами

God Object — это «монолит в миниатюре»: одно место, куда привыкли сваливать всё. Круговые зависимости — как два человека, которые могут двигаться только держась за руки: ни один не может шагнуть, пока другой не пошевелится.

### Картинка в голове

- **God Object:** огромный комбайн, который и пашет, и сеет, и убирает, и сам себя ремонтирует — но ломается от любого прикосновения.
- **Круговая зависимость:** два дома, которые опираются только друг на друга мостиками, без фундамента; если один качнётся, второй тоже.

### Как запомнить

- Если класс часто фигурирует во всех изменениях и багах — **проверь его на God Object**.
- Если два модуля никак не могут «развязаться» при рефакторинге — ищи **круговую зависимость**.
- Большой DTO без поведения — не проблема сам по себе; проблема — когда в него начинают добавлять разнородное поведение.

### Примеры

**God Object (упрощённый):**

```python
class ApplicationManager:
    def __init__(self, db, mailer, logger, payment_gateway, cache):
        self.db = db
        self.mailer = mailer
        self.logger = logger
        self.payment_gateway = payment_gateway
        self.cache = cache

    def register_user(self, data): ...
    def send_newsletter(self): ...
    def process_payment(self, payment): ...
    def generate_reports(self): ...
    def clear_cache(self): ...
    # и ещё десятки методов
```

**DTO (большой, но допустимый):**

```python
class CustomerProfile:
    def __init__(self, id, name, email, phone, address, birthdate, preferences, ...):
        self.id = id
        self.name = name
        self.email = email
        self.phone = phone
        self.address = address
        self.birthdate = birthdate
        self.preferences = preferences
        # десятки полей, но почти нет поведения
```

DTO хранит данные одной доменной сущности; бизнес‑логика живёт в сервисах, использующих этот объект.

### Практика / реальные сценарии

- Найди в проекте «самый большой класс» и оцени, God Object это или допустимый DTO:
  - много ли в нём разнородного поведения?
  - кто его главный актор?
- Поиск круговых зависимостей:
  - с помощью инструментов (IDE, `depgraph`, плагины);
  - вручную по импортам модулей и зависимостям классов.

### Типичные ошибки

- Объявлять любой большой класс God Object без разбора причин его размера.
- Снимать круговые зависимости через «магические» глобальные синглтоны, вместо осмысленного разрыва зависимостей.
- Добавлять поведение в DTO «потому что удобнее здесь», превращая его в God Object.

### Что будет, если…

- **Если не разруливать God Object:** он станет «бутылочным горлышком» для всей разработки. Любая фича упирается в него, риск регрессии растёт, а мотивация рефакторить падает.
- **Если допускать круговые зависимости:** код становится хрупким, трудно тестируемым и плохо поддающимся анализу инструментами; некоторые изменения становятся почти невозможными без серьёзной перестройки.

### Проверь себя

1. Как отличить большой DTO от God Object?
   <details><summary>Ответ</summary>
   DTO в основном хранит данные одной доменной сущности и имеет минимум поведения (конвертация, валидация). God Object, напротив, содержит множество разнородных методов, работающих с непересекающимися наборами полей, и обслуживает разные сценарии и акторов. В DTO изменения обычно связаны с данными сущности; в God Object — со всем подряд.
   </details>

2. Почему «синглтон‑менеджер» часто становится God Object?
   <details><summary>Ответ</summary>
   Синглтон доступен отовсюду, и в него удобно «докидывать» новую функциональность. Со временем туда тащат любые удобные методы: от кэшей до отправки писем. В итоге класс обрастает обязанностями и зависимостями, превращаясь в God Object и создавая жёсткую связанность по всему приложению.
   </details>

3. Как по коду заметить запах Feature Envy?
   <details><summary>Ответ</summary>
   Признаки:
   - метод почти не использует поля собственного класса, но активно обращается к полям/методам другого объекта (`invoice.customer.name`, `invoice.customer.address` и т.п.);
   - перенос метода ближе к данным (в класс `Invoice` или `Customer`) делает код короче и естественнее;
   - после переноса уменьшается количество параметров и глубина цепочек вызовов. В таком случае логика «завидовала» чужим данным и жила не в том классе.
   </details>

4. Чем опасна Inappropriate Intimacy между классами?
   <details><summary>Ответ</summary>
   Классы начинают опираться на внутренние детали друг друга: приватные поля, специальные «дырявые» методы, protected‑наследование. В результате:
   - любые изменения инвариантов или структуры одного класса ломают «друга»;
   - становится сложно понять, где именно контролируются правила и ограничения;
   - тестирование и рефакторинг требуют синхронного изменения нескольких плотно сцепленных классов. Явные, узкие контракты и методы‑фасады безопаснее, чем доступ к «внутренностям».
   </details>

### Запомните

- God Object и круговые зависимости — одни из самых болезненных запахов структуры классов.
- Большой размер класса сам по себе не преступление; важна **природа ответственности** и поведение.
- DTO стоит защищать от «наращивания» логики; для поведения лучше создавать отдельные сервисы.
