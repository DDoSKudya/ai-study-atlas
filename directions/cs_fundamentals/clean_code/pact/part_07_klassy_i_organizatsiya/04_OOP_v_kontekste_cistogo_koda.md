## 7.4. ООП в контексте чистого кода

### Цель раздела

Понять, как использовать ключевые идеи ООП — абстракции, полиморфизм, композицию и наследование — так, чтобы они **упрощали код**, а не создавали избыточные уровни иерархий и магию, которую никто не понимает.

### В этом разделе главное

- **Абстракции** вводятся ради понятного контракта, а не ради «красоты ради красоты».
- **Полиморфизм** помогает избавиться от сложных условных конструкций и упростить добавление новых вариантов поведения.
- **Композиция** чаще предпочтительнее наследования для переиспользования кода.
- **Наследование** уместно, когда есть настоящее отношение «является» и нужен полиморфизм.
- **Закон Деметры** ограничивает «болтовню» объектов и уменьшает связанность.

### Термины

| Термин              | Определение |
|---------------------|------------|
| **Абстракция**      | Обобщённый контракт, скрывающий детали реализации |
| **Полиморфизм**     | Способность разных реализаций иметь один интерфейс и вести себя по‑разному |
| **Композиция**      | Построение поведения за счёт включения других объектов как полей |
| **Наследование**    | Отношение «является» между классами, при котором подкласс расширяет поведение базового |
| **Закон Деметры**   | Принцип «не разговаривай с чужими друзьями» — ограничение цепочек вызовов |

### Теория и правила

**Абстракции и интерфейсы.**  
Абстракция вводится, когда:

- есть **несколько реализаций** (или ожидается появление);
- клиенту важно **поведение**, а не конкретная реализация;
- нужно **тестировать в изоляции**, подменяя реализации.

Плохой признак — интерфейс, для которого есть одна реализация и не видно реальной потребности в расширяемости: это может быть преждевременной абстракцией.

**Полиморфизм vs условная логика.**  
Когда в коде появляется множество `if` / `switch` по типу или коду статуса, это кандидат на полиморфизм:

```python
if user.role == "admin":
    ...
elif user.role == "manager":
    ...
elif user.role == "guest":
    ...
```

Вместо этого можно ввести иерархию ролей с методом, определяющим поведение:

```python
class Role:
    def can_approve(self, request):
        raise NotImplementedError


class Admin(Role):
    def can_approve(self, request):
        return True


class Manager(Role):
    def can_approve(self, request):
        return request.amount <= 1000
```

Так добавление новой роли не требует править старый `if` — добавляется новый класс.

**Композиция vs наследование.**  
Композиция:

- даёт больше гибкости (можно подменять включённые объекты);
- лучше изолирует ответственность;
- не влечёт за собой жёсткую иерархию.

Наследование:

- полезно, когда есть общее поведение и **истинное отношение «является»**;
- опасно, когда используется только ради переиспользования кода (см. 7.5).

**Закон Деметры.**  
Формулировка:

> Объект должен общаться только со своими «друзьями»: собой, своими полями, параметрами методов и объектами, которые он сам создаёт.

На практике это означает избегать длинных цепочек вида:

```python
order.customer.address.city.name
```

Вместо этого стоит предусмотреть в API более прямые методы (`order.city_name()`), скрывающие внутреннюю структуру.

Пример переписывания по закону Деметры:

```python
class Address:
    def __init__(self, city_name):
        self.city_name = city_name


class Customer:
    def __init__(self, address):
        self._address = address

    def city_name(self):
        return self._address.city_name


class Order:
    def __init__(self, customer):
        self._customer = customer

    def customer_city_name(self):
        return self._customer.city_name()
```

Код, который раньше делал `order.customer.address.city_name`, теперь делает `order.customer_city_name()`. Наружу виден только метод заказа; внутренняя структура клиента и адреса может меняться без затрагивания остального кода.

### Пошагово: когда вводить абстракцию

1. Заметь повторяющийся паттерн использования («у нас несколько способов сохранять данные», «несколько способов считать цену»).
2. Сформулируй **контракт**: какие операции нужны клиентскому коду.
3. Выдели интерфейс/абстрактный класс и реализуй его существующей логикой.
4. Добавь альтернативные реализации при необходимости (например, для тестов или другой БД).

### Простыми словами

ООП даёт инструменты — абстракции, полиморфизм, композицию, наследование. Чистый код говорит: **используй их, когда они упрощают чтение и изменение кода**, а не ради того, чтобы «код казался архитектурным». Хорошая абстракция делает код прямее и понятнее, а не сложнее.

### Картинка в голове

Абстракция — как розетка: пользователь видит только два отверстия и знает контракт («даёт 220В»). Внутри может быть разная проводка, автоматы и т.д., но это скрыто. Если же каждый прибор нужно было бы подключать напрямую к голым проводам — это аналог отсутствия абстракций.

### Как запомнить

- **Абстракция ради поведения, не ради моды.**
- **Композиция по умолчанию, наследование по необходимости.**
- **Полиморфизм вместо копипасты `if` по типу.**

### Примеры

**Плохое наследование ради переиспользования:**

```python
class SqlOrderRepository(DatabaseConnection):
    def save(self, order):
        self.execute("INSERT ...")  # использует методы базового класса
```

Здесь `SqlOrderRepository` «является» подключением к БД только ради доступа к методам.

**Лучше с композицией:**

```python
class SqlOrderRepository:
    def __init__(self, connection):
        self._connection = connection

    def save(self, order):
        self._connection.execute("INSERT ...")
```

Теперь ответственность разделена: один класс — про подключение, другой — про заказы.

### Практика / реальные сценарии

- Найди в коде место с большим `if` по типу или значению и попробуй мысленно переписать его в полиморфизм.
- Посмотри на классы, использующие наследование. Убедись, что у них действительно отношение «является», а не просто «мне нужны его методы».

### Типичные ошибки

- Вводить интерфейсы и абстрактные классы при одной реализации «на будущее».
- Строить глубокие иерархии наследования там, где достаточно композиции.
- Нарушать закон Деметры длинными цепочками вызовов (`a.getB().getC().getD()`).

### Что будет, если…

- **Если злоупотреблять наследованием:** любая правка в базовом классе влияет на всех наследников; понять реальное поведение становится сложно, приходится смотреть по всей иерархии.
- **Если не использовать абстракции там, где они напрашиваются:** условные конструкции разрастаются, добавление новых вариантов поведения требует править множество мест сразу.

### Проверь себя

1. Как понять, что использование наследования в конкретном месте оправдано?
   <details><summary>Ответ</summary>
   Спроси себя: «Объект подкласса действительно является частным случаем базового? Можно ли заменить базовый экземпляр подклассом везде, где он ожидается?» Если да, и если общий код и интерфейс реально используются, наследование оправдано. Если нет — вероятно, нужна композиция.
   </details>

2. Почему закон Деметры помогает снижать связанность?
   <details><summary>Ответ</summary>
   Он запрещает объекту «знать слишком много» о внутренней структуре других объектов (друзей друзей). Вместо длинных цепочек мы вводим методы‑обёртки там, где это логично, и уменьшаем вероятность того, что изменение глубокой структуры сломает внешний код.
   </details>

3. В чём практическая польза переписывания `order.customer.address.city_name` в вызов `order.customer_city_name()`?
   <details><summary>Ответ</summary>
   Такой шаг:
   - прячет внутреннюю структуру связанных объектов (`Customer`, `Address`) за методом заказа;
   - уменьшает количество мест, которые нужно менять при изменении структуры домена (добавление новой сущности, переименование полей);
   - делает вызов более «доменным» (мы спрашиваем у заказа, в каком он городе), а не техническим обходом графа объектов. Это снижает связанность и облегчает последующий рефакторинг.
   </details>

### Запомните

- Абстракции и полиморфизм — инструменты **упрощения кода**, а не цель сами по себе.
- **Композиция чаще безопаснее**, чем наследование.
- Закон Деметры — простой фильтр против чрезмерной связанности.
