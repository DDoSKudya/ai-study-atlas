## 7.3. Связность и связанность

### Цель раздела

Научиться оценивать, насколько класс «собран вокруг одной идеи» (связность) и насколько он «залип» на других классах (связанность), а также понимать, как повышать связность и снижать опасную связанность.

### В этом разделе главное

- Высокая **связность (cohesion)** — когда методы класса работают с одним и тем же набором данных и одной концепцией.
- **Низкая связность** проявляется в том, что разные методы используют разные подмножества полей.
- **Связанность (coupling)** бывает полезной (зависимость от абстракции) и опасной (жёсткая привязка к деталям).
- Цель — **высокая связность внутри** класса и **разумная, слабая связанность** между классами.

### Термины

| Термин           | Определение |
|------------------|------------|
| **Связность (cohesion)**   | Степень, с которой элементы класса (поля, методы) относятся к одной задаче |
| **Связанность (coupling)** | Сила и количество зависимостей между классами/модулями                       |
| **God Object**             | Класс с низкой связностью и высокой связанностью: «всё в одном»             |

### Теория и правила

**Связность.**  
Высокая связность означает, что:

- большинство методов класса работают с **одними и теми же полями**;
- в описании класса легко выделить одну центральную концепцию;
- нет «подсемейства» методов, которые используют только подмножество полей.

Низкая связность часто выглядит так:

- часть методов работает только с полями `a, b`;
- другая часть — только с `x, y`;
- третья — с `config` и `logger`.

Это сигнал, что внутри класса прячутся **несколько разных классов**.

Инструменты статического анализа (см. часть 14) часто дают численные оценки связности, например, метрику **LCOM (Lack of Cohesion of Methods)**. Высокий LCOM — сигнал, что методы мало связаны общими данными. Важно помнить: метрика — **повод посмотреть на код внимательнее**, а не автоматический приговор. Читай число вместе с реальной структурой методов и полей, как описано выше.

**Связанность.**  
Связанность — это:

- от скольких других классов наш класс зависит;
- насколько он зависит от **конкретных реализаций** vs от абстракций (интерфейсов, протоколов);
- насколько изменение в зависимом классе затронет наш.

Желательно:

- зависеть от **абстракций** (интерфейсов), а не от конкретных реализаций;
- минимизировать число «жёстких связок»;
- передавать зависимости **через конструктор** или DI‑контейнер, а не создавать их внутри (см. «новые»).

### Пошагово: как выявить проблемы со связностью

1. Нарисуй таблицу: строки — методы, столбцы — поля класса. Отметь, какие поля используются в каждом методе.
2. Посмотри на **кластеры**: если часть методов совсем не пересекается по полям с другой, это потенциальные «скрытые классы».
3. Попробуй описать для каждого кластера отдельную ответственность и имя класса.
4. Оцени, сколько зависимостей новый класс потребует — иногда разбиение ещё и уменьшает связанность.

### Пошагово: как уменьшать опасную связанность

1. Выяви места, где класс **создаёт зависимости сам** (`new SomeService()` внутри).
2. Вынеси эти зависимости в **конструктор или фабрику** и передавай их снаружи.
3. Заменяй зависимости на **интерфейсы** там, где нужна гибкость и тестируемость.
4. Со временем введи более крупные абстракции (фасады, адаптеры) на границах модулей.

### Простыми словами

Связность — это «насколько вещи в одном классе вообще про одно и то же». Связанность — «насколько этот класс прицеплен к другим». Мы хотим, чтобы **внутри было плотно**, а **снаружи — свободно и по контракту**.

### Картинка в голове

Представь комнаты в квартире:

- **Высокая связность:** на кухне — всё для готовки; в спальне — всё для сна.
- **Низкая связность:** на кухне хранится половина одежды, в спальне — кастрюли.

Связанность — это:

- насколько комнаты зависят друг от друга (например, если выключить свет в кухне, пропадёт и в спальне).

Хороший дизайн — каждая комната отвечает за своё, но есть понятные, не чрезмерные связи (коридор, выключатели).

### Как запомнить

- **Внутри — вместе, снаружи — по контракту.**
- Если методы работают с разными подмножествами полей, **ищи скрытые классы**.
- Если класс создаёт кучу `new` внутри и напрямую пользуется «толстой» реализацией — это высокая связанность.

### Примеры

**Класс с низкой связностью:**

```python
class UserManager:
    def __init__(self, db, mailer, logger):
        self._db = db
        self._mailer = mailer
        self._logger = logger
        self._cache = {}

    def create_user(self, user_data):
        user = self._db.save_user(user_data)
        self._cache[user.id] = user
        return user

    def send_newsletter(self, subject, body):
        users = self._db.load_all_users()
        for user in users:
            self._mailer.send(user.email, subject, body)

    def log_failed_login(self, user_id):
        self._logger.warn(f"Failed login for {user_id}")
```

Здесь:

- создание/кэширование пользователей,
- рассылка новостей,
- логирование неудачных логинов

— это три разных кластера поведения. Связность низкая, связанность высокая (зависимость от трёх разных сервисов).

**Разделение по связности:**

```python
class UserRepository:
    def __init__(self, db):
        self._db = db
        self._cache = {}

    def create_user(self, user_data):
        user = self._db.save_user(user_data)
        self._cache[user.id] = user
        return user


class NewsletterService:
    def __init__(self, mailer):
        self._mailer = mailer

    def send_newsletter(self, users, subject, body):
        for user in users:
            self._mailer.send(user.email, subject, body)


class SecurityLogger:
    def __init__(self, logger):
        self._logger = logger

    def log_failed_login(self, user_id):
        self._logger.warn(f"Failed login for {user_id}")
```

Теперь у каждого класса:

- связность выше (методы работают вокруг одной задачи);
- связанность более локальна (каждый зависит от одного сервиса).

### Практика / реальные сценарии

- Построй для одного из своих классов «матрицу методов и полей» и поищи кластеры.
- Посмотри, не завязан ли какой‑то класс напрямую на множество других реализаций; попробуй заменить хотя бы одну на интерфейс и инъекцию через конструктор.

### Типичные ошибки

- Пытаться «на глаз» оценивать связность, не глядя на использование полей.
- Создавать сервисы, которые знают «всех и вся» (зависимость от десятка других классов).
- Прятать God Object за красивым именем (`ApplicationService`, `Manager`, `Processor`) и считать, что проблема решена.

### Что будет, если…

- **Если связность низкая:** изменения в одной части класса легко затронут другую, тесты станут сложнее, а понимание — медленнее. В какой‑то момент класс становится кандидатом на «невозможный рефакторинг».
- **Если связанность чрезмерна:** любое изменение в зависимом классе вызывает каскад изменений. Тестировать в изоляции сложно, потому что нужно поднимать много реальных зависимостей.

### Проверь себя

1. Как по тестам можно заметить низкую связность класса?
   <details><summary>Ответ</summary>
   Если тесты для одного класса выглядят как тесты совершенно разных подсистем (отчетность, логирование, отправка почты), и при этом каждый тест использует разные подмножества полей/зависимостей, это признак низкой связности: в одном классе живут несколько «мини‑подсистем».
   </details>

2. Почему зависимость от интерфейса обычно лучше, чем зависимость от конкретного класса?
   <details><summary>Ответ</summary>
   Интерфейс задаёт контракт без конкретной реализации, что позволяет подменять реализацию (в тестах или при развитии системы) без изменения зависимого кода. Это снижает связанность и упрощает эволюцию дизайна.
   </details>

### Запомните

- **Высокая связность** внутри и **слабая связанность** между — цель при проектировании классов.
- Кластеры «методы ↔ поля» — подсказка, где спрятаны новые классы.
- Зависеть лучше от **контрактов**, чем от конкретных реализаций.
