## 7.2. Единственная ответственность (SRP)

### Цель раздела

Научиться видеть, когда класс взял на себя слишком много обязанностей, и уметь разделять их по **причинам для изменения** так, чтобы каждый класс обслуживал одну область требований или одного «актора».

### В этом разделе главное

- Принцип SRP формулируется как **«у класса должна быть одна причина для изменения»**.
- «Причина для изменения» — это **актёр** (роль или подсистема), которая предъявляет требования к классу.
- Много методов ≠ нарушение SRP; нарушение — это много различных причин меняться.
- Разделение по SRP уменьшает **область влияния изменений** и упрощает тестирование.

### Термины

| Термин        | Определение |
|--------------|------------|
| **SRP**      | Single Responsibility Principle — принцип единственной ответственности |
| **Актор**    | Лицо или подсистема, инициирующая изменение требований к коду |
| **Причина для изменения** | Изменение требований, из-за которого нужно править данный класс |

### Теория и правила

**Интуиция SRP.**  
Класс не должен быть «местом, где мы правим всё подряд». Если к одному и тому же классу приходят запросы:

- от бухгалтерии («нужна новая форма отчёта»),
- от отдела безопасности («нужно логировать это событие»),
- от команд, работающих с БД («мы меняем схему таблиц»),

— это значит, что в одном классе смешаны обязанности перед разными акторами.

**Формулировка SRP.**  
> Класс должен иметь **одну причину для изменения**.

Это значит:

- он обслуживает **одну зону требований**;
- одним взглядом на класс можно понять, «кто его главный клиент».

**Как искать нарушения SRP.**

- Пройди по методам класса и для каждого задай вопрос: **«кто попросил об этом поведении?»**.
- Если ответы разные (бухгалтерия, безопасность, интеграция с внешним API), — высокий риск нарушения SRP.
- Если методы можно разбить на несколько групп по акторам, это кандидаты на новые классы.

### Пошагово: как разделить класс по SRP

1. **Составь список методов** класса и для каждого подпиши «кто заинтересован» (доменный актор, подсистема, внешний API).
2. **Сгруппируй методы по акторам.**  
   Часто увидишь 2–3 группы, которые почти не пересекаются по данным.
3. **Определи для каждой группы имя ответственности.**  
   Например, «отчётность», «логирование аудита», «синхронизация с CRM».
4. **Создай отдельные классы** под каждую ответственность с соответствующим именем.
5. **Перенеси методы и данные** в новые классы, оставив в исходном классе только то, что действительно относится к его изначальной задаче (или превратив его в фасад).

### Простыми словами

SRP говорит: «Не заставляй один класс отвечать за всех сразу». Пусть у каждого класса будет **один «шеф»** — один тип требований. Тогда, когда этот шеф что‑то меняет, вы правите один класс, а не десяток.

### Картинка в голове

Представь отдел в компании. Хорошо, когда есть:

- отдел продаж,
- отдел бухгалтерии,
- отдел поддержки.

Плохо, когда есть один отдел «делаем всё» — от приёма звонков до закрытия месяца и настройки серверов. Такой отдел постоянно завален задачами, и любое изменение процесса бьёт по всем сразу. Класс‑«комбайн» — это такой отдел.

### Как запомнить

- **Один актор — один класс.**
- **Одна причина для изменения** — простая проверка на ревью.
- Если при обсуждении изменений часто говорите «когда меняем это, всегда трогаем ещё и тот же класс» — там, вероятно, нарушение SRP.

### Примеры

**Класс, нарушающий SRP:**

```python
class ReportService:
    def __init__(self, db, mailer, audit_logger):
        self._db = db
        self._mailer = mailer
        self._audit_logger = audit_logger

    def generate_monthly_report(self, month):
        data = self._db.load_for_month(month)
        # ... формируем отчёт ...
        return data

    def send_report_to_management(self, report):
        self._mailer.send("ceo@company.com", "Monthly report", report)

    def log_report_access(self, user_id, report_id):
        self._audit_logger.log("REPORT_ACCESS", user_id=user_id, report_id=report_id)
```

Здесь:

- отчётность (бизнес),
- рассылка писем (коммуникации),
- аудит (безопасность)

— все свалены в один класс.

**Разделение по SRP:**

```python
class ReportGenerator:
    def __init__(self, db):
        self._db = db

    def generate_monthly_report(self, month):
        data = self._db.load_for_month(month)
        # ... формируем отчёт ...
        return data


class ReportSender:
    def __init__(self, mailer):
        self._mailer = mailer

    def send_to_management(self, report):
        self._mailer.send("ceo@company.com", "Monthly report", report)


class ReportAudit:
    def __init__(self, audit_logger):
        self._audit_logger = audit_logger

    def log_access(self, user_id, report_id):
        self._audit_logger.log("REPORT_ACCESS", user_id=user_id, report_id=report_id)
```

Теперь:

- отчёты меняются независимо от способов отправки и аудита;
- тесты для каждой ответственности проще и яснее.

### Практика / реальные сценарии

- Найди в своём проекте класс, который:
  - зависит и от БД, и от почтового сервиса, и от логгера, и от внешнего API;
  - часто фигурирует в багфиксах и новых фичах.
- Попробуй расписать, какие акторы/подсистемы предъявляют к нему требования, и предложи возможное разбиение.

### Типичные ошибки

- Интерпретировать SRP как «у класса должен быть один метод» — это не так; важна не длина, а **число причин для изменения**.
- Считать, что «класс с десятком методов» обязательно нарушает SRP (иногда это широкий, но однородный интерфейс).
- Делить классы механически («по три метода на класс»), не думая об акторах и ответственности.

### Что будет, если…

- **Если игнорировать SRP:** каждый новый вид изменений будет добавлять методы и поля в уже перегруженный класс. Со временем он превратится в God Object, и любое изменение станет рискованным — затрагивается слишком много кода.
- **Если применять SRP бездумно:** можно раздробить простой модуль на множество крошечных классов, где каждый «про мелочь», но вместе они создают переусложнение. Важно искать **естественные границы ответственности**, а не делить ради деления.

### Проверь себя

1. Как понять, что класс нарушает SRP, по истории коммитов?
   <details><summary>Ответ</summary>
   Если в истории видно, что любые изменения, связанные с разными областями (отчёты, логирование, интеграции), постоянно правят один и тот же класс, это признак, что он служит нескольким акторам. Грубо: «по любому поводу мы лезем в этот класс».
   </details>

2. Почему класс‑фасад, который только делегирует вызовы другим классам, может при этом **не** нарушать SRP?
   <details><summary>Ответ</summary>
   Потому что его ответственность может заключаться ровно в одном: предоставить упрощённый интерфейс для определённого сценария (например, «оформление заказа»), скрывая детали внутренних сервисов. Его «шеф» — один сценарий или один актор (клиентский код), и он не берёт на себя другие обязанности.
   </details>

### Запомните

- SRP — про **причины для изменения**, а не про количество методов.
- У каждого класса должен быть **один главный актор**.
- История коммитов и багов — хороший индикатор нарушений SRP.
