## 7.1. Организация класса

### Цель раздела

Понять, как выглядит **хорошо организованный класс**: что такое инкапсуляция, как упорядочивать поля и методы, почему маленькие классы полезнее «комбайнов», и как структура файла помогает читать и сопровождать код.

### В этом разделе главное

- Класс должен **скрывать детали реализации** и показывать только минимально необходимый публичный интерфейс.
- **Порядок членов** в классе помогает читателю быстро ориентироваться.
- **Маленькие классы** с чёткой ответственностью проще тестировать, рефакторить и переиспользовать.
- Структура файла с классом — это тоже **часть чистого кода**.

### Термины

| Термин           | Определение |
|------------------|------------|
| **Инкапсуляция** | Сокрытие внутренних деталей реализации класса за ясным публичным интерфейсом |
| **Публичный интерфейс** | Набор доступных извне методов и свойств, через которые класс используется |
| **Маленький класс** | Класс, чью ответственность можно описать одним предложением и который не разрастается в «комбайн» |

### Теория и правила

**Инкапсуляция.**  
Инкапсуляция означает, что **внутреннее состояние и детали реализации скрыты**, а снаружи виден только аккуратный интерфейс. Это даёт:

- свободу менять реализацию без влияния на вызывающий код;
- понятные «точки входа» в функциональность;
- защиту от неконтролируемого изменения состояния извне.

Признаки плохой инкапсуляции:

- множество публичных полей;
- «сетеры для всего» без инвариантов;
- методы, раскрывающие внутреннюю структуру (`getInternalMap()` и т.п.).

**Порядок членов.**  
Удобно придерживаться **стабильного порядка** элементов класса. Например:

1. константы и статические поля;
2. публичные поля/свойства;
3. конструкторы и фабрики;
4. публичные методы;
5. защищённые методы;
6. приватные методы и вспомогательные функции.

Такой порядок помогает:

- быстро найти **точку входа** (конструктор, фабрики);
- увидеть публичный API **до деталей реализации**;
- не прыгать по файлу, отыскивая важные части.

**Маленькие классы.**  
Как и функции, классы лучше держать **маленькими по смыслу**. Это не столько про количество строк, сколько про:

- возможность описать класс одной фразой без «и ещё»;
- небольшое количество полей, которые логически связаны;
- отсутствие разнородных методов, работающих с непересекающимися наборами полей.

Если по описанию получается «класс отвечает и за загрузку данных, и за их валидацию, и за отчёты, и за логирование» — это сигнал к разбиению.

### Пошагово: как привести класс в порядок

1. **Опиши класс одним предложением.**  
   Если в предложении есть несколько союзов «и», «а также», это намёк на смешение обязанностей.
2. **Выяви публичный интерфейс.**  
   Посмотри, какие методы и свойства действительно используются снаружи.
3. **Спрячь лишнее.**  
   Всё, что не должно вылезать наружу, сделай приватным/protected (или убери экспорт в модуле).
4. **Упорядочь элементы.**  
   Расставь поля и методы в согласованном порядке.
5. **Посмотри на размер и связность.**  
   Если внутри есть группы методов, почти не пересекающихся по данным, — это задел на разделение (см. 7.2 и 7.3).

### Простыми словами

Класс — это как раздел в книге. Хороший раздел:

- посвящён **одной теме**;
- начинается с заголовка и краткого введения;
- затем идут подразделы в логичном порядке.

Плохой раздел — свалка из всего подряд: немного истории, немного рецептов, немного биографии автора. То же и с классом: порядок и фокус делают его читабельным.

### Картинка в голове

Представь **ящик с инструментами**. Хороший ящик:

- подписан («электрика», «плотницкий»);
- внутри инструменты одного типа, разложены по отсеку.

Плохой ящик — всё подряд: отвертки, вилки, шурупы, лекарства. Класс — это такой ящик: чем понятнее его подпись и содержимое, тем проще с ним работать.

### Как запомнить

- **Сначала интерфейс, потом детали.** Публичные методы и конструкторы — выше, приватные детали — ниже.
- **Описываем класс одним предложением.** Если не получается — он, скорее всего, делает слишком много.
- **Инкапсуляция по умолчанию.** Всё приватно, пока нет явной причины сделать публичным.

### Примеры

**Неудачно организованный класс (смешанный порядок и открытые детали):**

```python
class OrderService:
    def __init__(self, db, notifier):
        self.db = db
        self.notifier = notifier
        self.orders_cache = {}  # публичное внутреннее состояние

    def save(self, order):
        self.db.save(order)

    def _calculate_discount(self, order):
        ...

    def send_email(self, order):
        self.notifier.send(order.user_email, "Спасибо за заказ!")

    def load_all(self):
        if self.orders_cache:
            return self.orders_cache.values()
        orders = self.db.load_all()
        self.orders_cache = {o.id: o for o in orders}
        return orders
```

Здесь:

- внутренний кэш ничем не защищён;
- порядок методов хаотичен;
- сервис и сохраняет, и шлёт письма, и кэширует.

**Лучше (минимальный пример разделения обязанностей и порядка):**

```python
class OrderRepository:
    def __init__(self, db):
        self._db = db
        self._cache = {}

    def load_all(self):
        if self._cache:
            return list(self._cache.values())
        orders = self._db.load_all()
        self._cache = {o.id: o for o in orders}
        return orders

    def save(self, order):
        self._db.save(order)
        self._cache[order.id] = order


class OrderNotifier:
    def __init__(self, notifier):
        self._notifier = notifier

    def send_order_confirmation(self, order):
        self._notifier.send(order.user_email, "Спасибо за заказ!")
```

Ответственности разделены, поля инкапсулированы, порядок очевиден.

### Практика / реальные сценарии

- При ревью смотри на класс и задавай вопрос: **«Если бы этот класс был единственным, что я вижу, смог бы я понять, за что он отвечает?»**.
- Возьми класс из своего проекта и:
  - опиши его одним предложением;
  - спрячь лишние поля и методы;
  - упорядочи элементы;
  - подумай, не пора ли разбить его на 2–3 меньших класса.

### Типичные ошибки

- Публичные поля «для удобства», которые ломают инварианты.
- Методы, не относящиеся к основной ответственности класса, но добавленные «потому что удобно».
- Произвольный порядок методов, зависящий от истории редактирования, а не от логики.

### Что будет, если…

- **Если инкапсуляция нарушена:** любое изменение внутренней структуры класса может сломать код в десятках мест, которые напрямую обращаются к полям и внутренним коллекциям. Тесты сложнее писать и поддерживать: нужно воссоздавать сложное состояние вручную.
- **Если порядок членов хаотичен:** чтение файла превращается в постоянные прыжки: сначала конструктор, потом приватный метод, потом снова константа. Это замедляет понимание и увеличивает шанс пропустить важную деталь.

### Проверь себя

1. Как по коду класса понять, что инкапсуляция нарушена?
   <details><summary>Ответ</summary>
   Признаки: много публичных полей; методы возвращают внутренние изменяемые коллекции; вызывающий код регулярно изменяет поля напрямую; инварианты (ограничения на состояние) не соблюдаются, потому что данные можно изменить в обход методов класса.
   </details>

2. Почему «одна фраза про класс» полезна не только при проектировании, но и при ревью?
   <details><summary>Ответ</summary>
   Формулируя одну фразу, ревьюер проверяет, не «расползлась» ли ответственность класса со времени создания. Если сейчас в описание приходится добавлять новые «и ещё», это сигнал, что класс за время жизни оброс обязанностями и требует пересмотра.
   </details>

### Запомните

- Класс **скрывает детали** и показывает минимальный, ясный интерфейс.
- **Интерфейс сверху, детали снизу** — правило организации файла.
- Если класс нельзя описать одной фразой без «и ещё», он, вероятно, делает слишком много.
