## 13.2. Стратегии работы с legacy‑кодом

### Цель раздела

Разобрать практические стратегии, которые позволяют **безопасно менять legacy‑код**: добавлять тесты, локализовать изменения, работать через границы и швы, чтобы минимизировать риск регрессий.

### В этом разделе главное

- **Никогда не менять legacy‑код без попытки зафиксировать текущее поведение.**
- Стратегия «сначала тесты, потом рефакторинг, потом функциональные изменения» — базовый шаблон.
- Изоляция изменений — ключ к тому, чтобы не «потрогать всё сразу».

### Термины

| Термин | Определение |
|--------|-------------|
| **Characterization test** | Тест, описывающий текущее (включая странное) поведение системы |
| **Точка входа** | Место, через которое внешний мир вызывает рассматриваемый код (API, метод сервиса, CLI и т.п.) |
| **Seam / seam point** | Место, где можно подменить зависимость или вставить обёртку без изменения остального кода |

### Теория и правила

1. **Сначала зафиксируй текущее поведение.**  
   Перед изменением legacy‑кода нужно:
   - найти **точку входа** для нужного сценария;
   - написать один или несколько characterization tests, которые:
     - проходят на текущем коде;
     - описывают как «нормальное», так и странное существующее поведение.

2. **Менять по минимальной поверхности.**
   - Не затрагивать части, не относящиеся к текущей задаче.
   - По возможности **не менять интерфейсы** (подписи методов, форматы JSON), если это не часть задачи.

3. **Использовать швы для внедрения тестов и новых реализаций.**
   - Внедрить интерфейс или абстракцию (**порт**) и подменять реализацию (**адаптер**) в тестах.
   - Использовать DI/фабрики, чтобы подставить фейк или мок вместо реальной зависимости.

4. **Комбинировать юнит‑, интеграционные и end‑to‑end тесты.**
   - В legacy часто начать проще с **интеграционных и e2e** тестов на важный сценарий.
   - Далее, по мере рефакторинга, можно выделять меньшие юнит‑тестируемые части.

### Seam‑точки: конструктор, setter, фабрика

В legacy не всегда возможно сразу внедрить полноценный DI‑контейнер, но почти всегда есть возможность добавить **хотя бы один шов**:

- **Конструктор:**  
  Если класс создаётся где‑то в коде напрямую (`new Service()` или аналог), мы можем:
  - добавить параметр конструктора для зависимости (`Repository`, `Client`);
  - в production передавать реальную реализацию;
  - в тестах — фейк/мок.

- **Setter / init‑метод:**  
  Если конструктор менять трудно (сложно найти все места создания или сигнатуры жёстко зашиты), можно:
  - добавить `setRepository(...)` / `initDependencies(...)`;
  - в тестах вызывать этот метод, подменяя зависимость.

- **Фабрика:**  
  Для особенно жёсткого legacy можно ввести **фабрику**:
  - создать `ServiceFactory`, которая знает, как «правильно» собрать объект;
  - в production использовать стандартную фабрику;
  - в тестах подменять фабрику или её конфигурацию, подсовывая другие реализации.

Каждый такой шов — возможность:

- написать локальные тесты;
- подменить тяжёлые зависимости (БД, сеть, файловая система);
- постепенно вытягивать логику из «бетона» монолита в тестируемые куски.

### Пошагово: классический сценарий изменения legacy

1. **Сформулировать задачу.**  
   Что именно нужно изменить: бизнес‑правило, формат ответа, алгоритм?
2. **Найти точку входа.**  
   Откуда сценарий начинается: контроллер, обработчик очереди, CLI‑команда?
3. **Написать characterization тесты.**  
   - На текущем поведении;
   - С нужными граничными случаями.
4. **Запустить тесты и убедиться, что они зелёные.**
5. **Сделать минимальный рефакторинг под тестами.**  
   Например:
   - извлечь метод;
   - ввести интерфейс для внешней зависимости;
   - разрезать гигантский метод на несколько шагов.
6. **Внести изменение в логику.**
7. **Запустить тесты.**  
   - При необходимости обновить их, если изменилась спецификация поведения.

### Простыми словами

Работа с legacy‑кодом похожа на **ремонт старого дома, в котором уже живут люди**:

- вы сначала **делаете опорные конструкции безопасными** (тесты, швы),
- только потом **перекладываете проводку и трубы** (рефакторинг),
- и уже затем **ставите новую технику** (новая функциональность).
 
### Примеры (высокоуровневые)

- Изменение логики скидок:
  - точка входа — метод `calculateDiscount(order)` в сервисе;
  - пишем characterization тесты на текущие кейсы (без скидки, со скидкой по купону, максимальная скидка и т.п.);
  - под ними разбиваем длинный метод на несколько шагов (`calculateBase`, `applyCoupon`, `capDiscount`);
  - добавляем новый тип скидки только после этого.

- Переход с одной библиотеки логирования на другую:
  - создаём свой интерфейс `Logger`;
  - текущую библиотеку прячем за адаптером `ExistingLibLogger`;
  - везде используем `Logger`;
  - затем спокойно подменяем реализацию.

### Пример characterization test (псевдокод)

Предположим, у нас есть «грязный» метод расчёта комиссии:

```python
def calculate_commission(order, user):
    # 300 строк странной логики
    ...
```

Мы хотим изменить правило для VIP‑пользователей. Прежде чем лезть в реализацию, пишем characterization test, который описывает **текущее** поведение:

```python
def test_calculate_commission_keeps_legacy_behavior_for_vip():
    # Given
    order = Order(amount=1000)
    user = User(type="VIP", region="EU")

    # When
    commission = calculate_commission(order, user)

    # Then
    # Это значение мы «засняли» с продакшена или ручного расчёта
    assert commission == Decimal("12.34")
```

Пока мы рефакторим `calculate_commission`, этот тест гарантирует, что мы **случайно не изменили** старое поведение для VIP‑пользователей. Когда бизнес официально меняет правило, мы **сначала меняем тест** (новое ожидаемое значение), а затем код.

### Пример seam‑точки через интерфейс

Пусть legacy‑код напрямую использует глобальный клиент БД:

```python
def get_user_balance(user_id: int) -> Decimal:
    row = global_db.execute("SELECT balance FROM users WHERE id = %s", [user_id])
    return row["balance"]
```

В тестах такой код трудно изолировать. Мы вводим seam‑точку — интерфейс доступа к данным:

```python
class UserRepository(Protocol):
    def get_balance(self, user_id: int) -> Decimal: ...


class DbUserRepository(UserRepository):
    def __init__(self, db):
        self._db = db

    def get_balance(self, user_id: int) -> Decimal:
        row = self._db.execute("SELECT balance FROM users WHERE id = %s", [user_id])
        return row["balance"]


def get_user_balance(repo: UserRepository, user_id: int) -> Decimal:
    return repo.get_balance(user_id)
```

В production мы передаём `DbUserRepository`, а в тестах — простой фейк:

```python
class FakeUserRepository(UserRepository):
    def __init__(self, balances):
        self._balances = balances

    def get_balance(self, user_id: int) -> Decimal:
        return self._balances[user_id]
```

Такой интерфейс и есть seam‑точка: мы можем тестировать логику, не трогая реальную БД, и со временем вынести работу с БД в отдельный, более чистый модуль.

### Как запомнить

- Любое изменение в legacy начинай с вопроса: **«Как я это проверю?»** — сначала тесты, потом код.
- Ищи, где можно вставить **шов** (интерфейс, конструкторный параметр, фабрика), а не ломать всё сразу.
- Минимальный шаг, который сохраняет поведение под тестами, почти всегда лучше «идеального», но рискованного переписывания.

### Типичные ошибки

- Менять код без тестов и «проверять на проде».
- Пытаться **сразу сделать «идеально чисто»**, переписывая всю функцию/класс.
- Не изолировать изменения: поправили одно место, задели пять других.

### Что будет, если…

- **Сделать большую переразбивку кода без предварительных тестов?**  
  Высокий риск скрытых регрессий, которые проявятся через недели или месяцы в самых неожиданных местах.

- **Ограничиться только e2e‑тестами без локальных швов?**  
  Любое изменение придётся проверять только через полный прогон всей системы, что:
  - долго;
  - плохо локализует источник проблем.

### Проверь себя

1. Зачем нужны characterization tests, если они фиксируют даже «кривое» поведение?

   <details><summary>Ответ</summary>
   Сейчас «кривое» поведение — это **фактическая спецификация** системы: пользователи и другие сервисы опираются именно на него. Characterization tests позволяют:
   - зафиксировать это поведение;
   - убедиться, что мы не сломали его случайно при рефакторинге;  
   а затем уже осознанно менять спецификацию и обновлять тесты.
   </details>

2. Почему в legacy‑коде особенно важно минимизировать поверхность изменений?

   <details><summary>Ответ</summary>
   Поскольку мы **плохо понимаем все зависимости и сценарии использования**, любое изменение может иметь неожиданные последствия. Чем меньше кода мы трогаем:
   - тем проще локализовать возможные баги;
   - тем легче откатиться;
   - тем меньше риск зацепить критичную функциональность.
   </details>

3. Как связаны понятия «точка входа» и «characterization test» при изменении legacy‑кода?

   <details><summary>Ответ</summary>
   Точка входа — это место, откуда начинается интересующий нас сценарий (эндпоинт, CLI‑команда, публичный метод сервиса). Characterization test обычно пишется **против этой точки входа**: он воспроизводит сценарий так, как его видит внешний потребитель. Благодаря этому:
   - тест фиксирует текущее наблюдаемое поведение;
   - остаётся устойчивым к внутренним перестройкам (рефакторингу) до тех пор, пока контракт точки входа не меняется.
   </details>

### Запомните

- Базовый шаблон: **тесты → рефакторинг → новые фичи**.
- Characterization tests — ваш **страховочный трос** при работе с legacy.
- Изоляция изменений и работа через швы резко снижают риски.
