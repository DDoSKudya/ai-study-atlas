## 13.3. Внедрение чистого кода в legacy‑проект

### Цель раздела

Показать, как **реалистично** внедрять стандарты чистого кода, линтеры, форматтеры и правило бойскаута в существующем проекте, не ломая команду и не тормозя бизнес.

### В этом разделе главное

- Новый код должен быть **лучше старого** — иначе долг только растёт.
- Стандарты нужно вводить **поэтапно** и с учётом legacy.
- «Островки чистого кода» и правило бойскаута — практические инструменты ежедневного улучшения.

### Термины

| Термин | Определение |
|--------|-------------|
| **Островок чистого кода** | Новый или отрефакторенный модуль, который следует стандартам чистого кода и служит примером |
| **Baseline (для линтера/анализа)** | Снимок текущих нарушений, которые временно допускаются, чтобы не блокировать сборку |
| **Правило бойскаута** | Принцип «оставь место чуть чище, чем нашёл» |

### Теория и правила

1. **Стандарты — сначала для нового кода.**
   - Не пытайтесь сразу «починить всё».
   - Фокус: **всё новое и изменяемое** должно соответствовать стандартам (имена, форматирование, тесты).

2. **Инструменты — с baseline и по зонам.**
   - Включать линтер с baseline: новые нарушения запрещены, старые — помечены как долг.
   - Для сильно проблемных директорий можно временно ослабить правила и ужесточать по мере рефакторинга.

3. **Островки чистого кода.**
   - Новые сервисы/модули — сразу по правилам (чистые интерфейсы, тесты).
   - Старый код по возможности изолируется за фасадами и постепенно вытесняется.

4. **Правило бойскаута.**
   - Любой коммит, который трогает legacy‑код, должен **чуть улучшать его**:
     - переименование переменной;
     - извлечение метода;
     - добавление одного теста;
     - удаление мёртвого кода.

### Пошагово: поэтапное внедрение стандартов

1. **Определить минимальный стандарт.**
   - Обязательные пункты (например: форматтер, базовый линтер, тесты для нового кода).
2. **Настроить инструменты.**
   - Форматтер в репозитории;
   - линтер с baseline;
   - быстрые тесты в pre‑commit / CI.
3. **Зафиксировать правила.**
   - Документ (README/CONTRIBUTING/стандарты кода);
   - короткий чек‑лист для ревью.
4. **Внедрять правило бойскаута.**
   - На каждом ревью напоминать: «мы этот файл трогаем — давайте сделаем его чуть лучше».
5. **Постепенно ужесточать правила.**
   - Убирать исключения из конфига;
   - переносить часть «warning» в «error» по мере улучшения базы.

### Простыми словами

Внедрение чистого кода в legacy‑проект — это **не революция, а эволюция**. Вы:

- не сносите весь дом;
- а **начинаете с комнат, в которых живёте чаще всего**, делая их удобными и безопасными;
- постепенно переделываете остальные, когда до них доходит дело.

### Примеры

- В проекте на Python:
  - добавили `black` и `ruff` с baseline;
  - договорились: любой новый файл проходит форматтер и линтер без предупреждений;
  - при касании старого файла — приводим хотя бы затронутые фрагменты к стандарту.

- В монолите на Java:
  - новые микросервисы пишутся по правилам (чистые границы, тесты, метрики);
  - старый монолит постепенно «разгружается» через вынос функционала в новые сервисы.

### Типичные ошибки

- Включить линтер «на максимум» для всего проекта и утонуть в тысячах предупреждений.
- Навязывать стандарты без объяснения «зачем», вызывая сопротивление команды.
- Не закреплять изменения процессом (ревью, CI) — всё возвращается к хаосу.

### Проверь себя

1. Зачем нужен baseline при внедрении линтера в legacy‑проект?

   <details><summary>Ответ</summary>
   Baseline позволяет:
   - **зафиксировать текущие нарушения** как известный долг;
   - запретить появление новых нарушений;
   - постепенно исправлять старые по мере касания кода.  
   Без baseline команда столкнётся с заваленной в красный цвет кодовой базой и потеряет мотивацию что‑то исправлять.
   </details>

2. В чём практический смысл правила бойскаута в legacy‑проекте?

   <details><summary>Ответ</summary>
   Оно превращает улучшение кода в **часть ежедневной рутины**, а не в отдельный «когда‑нибудь» проект. Каждый маленький шаг:
   - улучшает конкретное место;
   - снижает накопление нового долга;
   - даёт команде ощущение прогресса.  
   В сумме это приводит к заметному повышению качества без огромных «рефакторинг‑спринтов».
   </details>

3. Почему важно зафиксировать минимальный стандарт качества перед внедрением линтера и форматтера?

   <details><summary>Ответ</summary>
   Минимальный стандарт:
   - задаёт **чёткую планку**, что считается приемлемым для нового кода;
   - помогает избежать ситуации, когда разные разработчики по‑разному трактуют «чистоту»;
   - делает конфигурацию линтера и форматтера продолжением договорённостей, а не их заменой.  
   Это снижает споры и упрощает ревью: можно опираться на единый набор правил, а не на вкусы отдельных людей.
   </details>

### Запомните

- Новый код должен быть **примером того, как мы хотим писать**.
- Инструменты и стандарты внедряем **поэтапно**, с baseline.
- Правило бойскаута — мощный, но простой механизм постоянного улучшения.
