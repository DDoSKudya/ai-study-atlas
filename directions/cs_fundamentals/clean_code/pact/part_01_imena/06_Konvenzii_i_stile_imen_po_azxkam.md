## 1.6. Конвенции и стиль имён по языкам

### Цель раздела

Ориентироваться в конвенциях именования по языкам и проектам: camelCase, snake_case, PascalCase; префиксы интерфейсов и приватных членов; константы и enum; следовать принятому стилю и не смешивать конвенции без причины.

### В этом разделе главное

- **camelCase** (переменные, функции/методы), **PascalCase** (классы, типы), **snake_case** (Python, константы в части языков) — следовать конвенции языка и проекта.
- **Префиксы** интерфейсов (I), суффиксы (-Service, -Repository) — по принятому в экосистеме.
- **Константы и enum**: UPPER_SNAKE_CASE или PascalCase в зависимости от языка.
- **Приватные/внутренние** члены: подчёркивание, префикс _ или конвенция языка.

### Термины

| Термин | Определение |
|--------|-------------|
| **camelCase** | Первая буква слова с маленькой, остальные слова с большой: `getUserById`. |
| **PascalCase** | Каждое слово с большой: `UserRepository`, `OrderStatus`. |
| **snake_case** | Слова через подчёркивание: `user_id`, `get_user_by_id`. |
| **UPPER_SNAKE_CASE** | snake_case в верхнем регистре: `MAX_RETRY_COUNT`, `API_KEY`. |

### Теория и правила

**Переменные и функции/методы.** В Java, C#, JavaScript/TypeScript: переменные и методы — camelCase (`userName`, `getOrderById`). В Python, Ruby: snake_case (`user_name`, `get_order_by_id`). В Go: camelCase для экспортируемых, могут использовать аббревиатуры типа `ID` в PascalCase. Следовать официальному гайду (PEP 8, Go naming) и стилю проекта.

**Классы и типы.** Обычно PascalCase: `User`, `OrderRepository`, `HttpClient`. **Интерфейсы и абстрактные типы** — по принятому в экосистеме: префикс `I` (C#/Java: `IUserRepository`), суффиксы **-Interface**, **-Service**, **-Repository** там, где это принято (например, `UserRepository` как контракт, `OrderService` как интерфейс сервиса). В других проектах — интерфейс без префикса, реализация с суффиксом или префиксом по смыслу (`PostgresOrderRepository`, `DefaultOrderService`).

**Константы.** Во многих языках UPPER_SNAKE_CASE: `MAX_RETRIES`, `DEFAULT_TIMEOUT`. В части языков (C#, Java enum) константы или enum-значения в PascalCase: `OrderStatus.Shipped`. Следовать гайду языка.

**Приватные и внутренние члены.** Python: один ведущий `_` для «внутреннего» использования (`_internal_helper`). В других языках префикс `_` или ключевое слово `private` без дублирования в имени (не `_privateValue` если уже private). Цель — визуально отличать публичный контракт от деталей реализации.

**Проверь себя (приватные члены и конвенции)**

1. Зачем визуально выделять приватные/внутренние члены (например, префикс _)?
   <details><summary>Ответ</summary>
   Читатель сразу видит границу между публичным API и деталями реализации; при использовании класса не нужно гадать, можно ли вызывать метод снаружи. Это снижает риск случайной зависимости от внутренней реализации и упрощает рефакторинг.
   </details>

2. Почему дублировать «private» в имени (например, _privateField) в языке с ключевым словом private — плохо?
   <details><summary>Ответ</summary>
   Это шум: модификатор уже задаёт видимость; дублирование в имени не добавляет информации и усложняет чтение. Достаточно _field или field по конвенции языка. Избыточность вредит при переименовании и поддержке.
   </details>

**Не смешивать конвенции.** В одном проекте один стиль: не camelCase в одном модуле и snake_case в другом без явной причины (например, разные языки в одном репозитории). Форматтер и линтер помогают зафиксировать правила (часть 12).

### Простыми словами

В каждом языке и проекте есть принятый способ писать имена: с большой/маленькой буквы, через подчёркивание или без. Важно следовать этому способу, чтобы код выглядел единообразно и соответствовал ожиданиям сообщества. Константы и приватные члены часто выделяют стилем (UPPER_SNAKE, _prefix).

### Картинка в голове

**Дресс-код офиса:** в одном месте носят галстуки, в другом — футболки. Важно соответствовать «офису» (языку и проекту), а не смешивать стили без правил. Конвенции имён — такой же дресс-код для кода.

### Как запомнить

- **Язык и проект задают стиль:** camelCase/snake_case/PascalCase по гайду.
- **Классы/типы** — обычно PascalCase; **константы** — UPPER_SNAKE или PascalCase по языку.
- **Приватное** — _ или private; не смешивать конвенции в одном проекте.

### Примеры

**Python (PEP 8):** переменные и функции snake_case, классы PascalCase, константы UPPER_SNAKE:

```python
def get_user_by_id(user_id: str) -> User: ...
class OrderRepository: ...
MAX_RETRY_COUNT = 3
_internal_cache = {}
```

**Java/C#:** методы и поля camelCase, классы и интерфейсы PascalCase, константы UPPER_SNAKE или PascalCase для enum:

```java
public User getUserById(String userId) { ... }
public class OrderRepository { ... }
public static final int MAX_RETRY_COUNT = 3;
```

### Типичные ошибки

- Смешивать стили в одном файле: часть переменных в camelCase, часть в snake_case (в Python везде snake_case для функций и переменных).
- Дублировать «private» в имени: `_privateField` в языке, где уже есть ключевое слово private — достаточно `_field` или `field` по конвенции.

### Что будет, если…

**Если в одном проекте смешивать конвенции имён (например, camelCase и snake_case для переменных):** код выглядит непоследовательно; при ревью возникают споры о стиле вместо содержания. Линтер и форматтер не могут быть настроены однозначно — часть правил конфликтует. Новый член команды не понимает, какому стилю следовать, и копирует то, что видит в ближайшем файле, закрепляя смешение.

### Проверь себя (1.6)

1. Почему в одном проекте важно не смешивать конвенции имён (например, camelCase и snake_case для переменных)?
   <details><summary>Ответ</summary>
   Единообразие снижает когнитивную нагрузку: читатель не гадает, «здесь так, там иначе». Соответствие гайду языка упроняет онбординг и код-ревью. Смешение без причины выглядит как отсутствие стандартов и затрудняет автоматическую проверку (линтер).
   </details>

2. В вашем языке как принято именовать константы и приватные члены?
   <details><summary>Ответ</summary>
   Зависит от языка: в Python константы UPPER_SNAKE, приватные — с ведущим _ . В Java/C# константы часто UPPER_SNAKE или PascalCase для enum; приватные — без префикса в имени, только модификатор private. Нужно свериться с гайдом (PEP 8, Oracle Java Code Conventions, Microsoft C#) и с правилами проекта.
   </details>

3. Когда допустимо отступить от общепринятой конвенции языка?
   <details><summary>Ответ</summary>
   Когда проект явно зафиксировал иное (например, legacy или корпоративный стандарт) и это документировано. Или при интеграции с внешним API, где имена заданы контрактом. В остальных случаях лучше следовать конвенции — отступление без причины усложняет поддержку и онбординг.
   </details>

### Запомните

- **camelCase / snake_case / PascalCase** — по конвенции языка и проекта; не смешивать без причины.
- **Константы** — UPPER_SNAKE или PascalCase по языку; **приватные** — _ или конвенция.
- Следовать **официальному гайду** и **стилю проекта**; исключения — минимальные и обоснованные.
