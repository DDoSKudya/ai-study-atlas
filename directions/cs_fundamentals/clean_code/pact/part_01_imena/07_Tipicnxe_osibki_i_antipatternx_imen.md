## 1.7. Типичные ошибки и антипаттерны имён

### Цель раздела

Научиться распознавать типичные плохие практики именования и исправлять их: слишком общие имена, дезинформация, шумовые префиксы, дублирование смысла; понимать, когда аббревиатуры и короткие имена вредят.

### В этом разделе главное

- **Слишком общие имена:** `data`, `result`, `handler`, `util`, `info`, `thing`, `item` без контекста — не несут намерения.
- **Дезинформация:** суффикс/префикс не соответствует типу или смыслу (List для не-списка); похожие имена для разных сущностей без разграничения.
- **Шумовые префиксы:** венгерская нотация и дублирование типа в имени там, где тип и так ясен.
- **Сокращения** вне глоссария: `amt`, `cust`, `usr` — непонятны новому читателю.
- Как **исправлять:** переименовать с учётом намерения, контекста и конвенций; при необходимости вынести в глоссарий.

### Термины

| Термин | Определение |
|--------|-------------|
| **Антипаттерн** | Типичная плохая практика, которая повторяется и ухудшает качество кода. |
| **Венгерская нотация** | Префикс типа в имени переменной (например, strName, iCount); в современных языках с сильной типизацией обычно избыточна. |

### Теория и правила

**Слишком общие имена.** Слова вроде `data`, `result`, `handler`, `util`, `info`, `thing`, `item` без контекста не отвечают на вопрос «зачем». Что за data? Результат чего? Какой handler? Заменить на конкретику: `userRegistrationData`, `validationResult`, `orderNotificationHandler`, `formatDateUtil` (или лучше — имя по действию, не «util»).

**Дезинформация.** Уже разобрано в 1.1: не использовать имена, противоречащие типу или смыслу. Дополнительно: не давать разным сущностям почти одинаковые имена (`user` и `userAccount`, `order` и `orderData`) без чёткого правила, когда что использовать — лучше явно развести по смыслу и именам.

**Шумовые префиксы.** Венгерская нотация (strName, iCount, arrItems) в языках с явными типами дублирует информацию и усложняет переименование при смене типа. В современном коде обычно избегают. То же с дублированием в имени того, что уже видно из типа: `userUser` или `orderOrder` — шум.

**Сокращения.** Сокращения допустимы, если они в глоссарии или общеприняты (ID, URL). Свои сокращения (`amt`, `cust`, `usr`) без договорённости в команде вредят: новый человек не поймёт. Лучше полное имя или зафиксировать в глоссарии.

**Как исправлять.** При ревью или рефакторинге: (1) выявить сущность с неясным именем; (2) сформулировать намерение («это список заказов для отчёта»); (3) подобрать имя по принципам 1.1–1.6 и конвенциям; (4) переименовать везде (IDE Refactor / поиск по проекту), а не только в одном месте — иначе появятся сломанные ссылки и устаревшие комментарии; (5) при доменном термине — при необходимости обновить глоссарий.

**Проверь себя (как исправлять имена)**

1. Почему переименовывать «только в одном месте» опасно?
   <details><summary>Ответ</summary>
   Остаются сломанные ссылки (другие файлы, комментарии, документация), устаревшие упоминания старого имени. Код и документация расходятся; при следующем чтении или рефакторинге легко опереться на старое имя и допустить ошибку. Переименование должно быть глобальным (поиск, IDE Refactor).
   </details>

2. В каком порядке логично действовать при исправлении плохого имени и почему шаг «сформулировать намерение» идёт до «подобрать имя»?
   <details><summary>Ответ</summary>
   Сначала выявить сущность, затем сформулировать намерение («что это за данные/действие»), потом подобрать имя по принципам и конвенциям. Намерение задаёт критерий для имени; без него легко подобрать новое, но столь же размытое имя (result → output). Имя должно отражать намерение.
   </details>

### Простыми словами

Плохие имена — те, по которым нельзя понять «зачем» или которые врут. Общие слова (data, handler), врущие суффиксы (List не для списка), лишние префиксы типа и непонятные сокращения — всё это антипаттерны. Исправление — переименовать по намерению и контексту, с учётом конвенций и глоссария.

### Картинка в голове

**Подписи на ящиках:** «Разное», «Другое», «Данные» — бесполезные подписи. «Канцелярия», «Данные за март», «Обработчик уведомлений» — полезные. Антипаттерны имён — это «разное» и «данные» в коде; исправление — заменить на осмысленную подпись.

### Как запомнить

- **Общие имена (data, result, handler)** — заменить на конкретику по намерению.
- **Дезинформация и похожие имена** — не врать типу/смыслу; развести имена по смыслу.
- **Шум (венгерская нотация, дубли типа)** — убрать; **сокращения** — только по глоссарию или общепринятые.

### Примеры

**До (антипаттерны):**

```python
def process(data):
    result = []
    for item in data:
        if item['f']:
            result.append(item['v'])
    return result
```

**После (намерение и контекст):**

```python
def collect_confirmed_order_totals(orders: list[Order]) -> list[Decimal]:
    totals = []
    for order in orders:
        if order.is_confirmed:
            totals.append(order.total)
    return totals
```

**Дезинформация:** `accountList` при типе `set` → `accounts` или `accountSet` по конвенции.

**Шум:** `strUserName` в Java → `userName` (тип и так виден).

### Практика: упражнение на переименование

Переименуйте сущности в фрагменте ниже так, чтобы имена раскрывали намерение и не дезинформировали. Проверьте: один ли концепт у каждой переменной, нет ли магических значений, соответствуют ли имена конвенциям (например, snake_case для Python).

Исходный фрагмент:

```python
def proc(d, flg):
    r = []
    for x in d:
        if x['s'] == 1 and flg:
            r.append(x['v'] * 0.2)
    return sum(r)
```

<details><summary>Вариант решения (переименование и константа)</summary>

Один из возможных вариантов — явно выразить домен (например, заказы и скидка):

```python
DISCOUNT_RATE = 0.2

def total_discount_amount(orders, apply_discount):
    amounts_with_discount = []
    for order in orders:
        if order['status'] == OrderStatus.CONFIRMED and apply_discount:
            amounts_with_discount.append(order['total'] * DISCOUNT_RATE)
    return sum(amounts_with_discount)
```

Что сделано: `proc` → глагольная фраза по действию; `d` → `orders` (намерение); `flg` → `apply_discount` (булево намерение); `r` → `amounts_with_discount`; `x` → `order`; `'s' == 1` → статус с константой/enum; `0.2` → именованная константа; убрана дезинформация, если `r` не список в смысле типа — здесь список, имя это отражает. В вашем домене имена могут быть другими (например, `line_items`, `include_tax`), главное — раскрыть намерение и не оставлять общих имён.
</details>

**Проверь себя (упражнение на переименование)**

1. Какие принципы из 1.1–1.6 вы применили бы в первую очередь к фрагменту с `proc(d, flg)` и однобуквенными именами?
   <details><summary>Ответ</summary>
   Раскрыть намерение (proc → глагольная фраза по действию; d, flg, r, x — заменить на осмысленные имена). Одно понятие на переменную. Булевы с префиксом (flg → is_.../has_.../apply_...). Магическое число вынести в константу. Соответствие конвенциям (snake_case для Python). При домене — термины из глоссария.
   </details>

### Типичные ошибки

- Оставлять «временно» общие имена (data, result) — они закрепляются и множатся.
- Переименовывать только в одном месте — сломать ссылки и комментарии; переименовывать везде (поиск, IDE Refactor).
- Вводить новые сокращения без фиксации в глоссарии — следующий разработчик не поймёт.

### Что будет, если…

**Если не исправлять антипаттерны имён:** общие имена (data, result) тиражируются в новом коде по образцу существующего. Дезинформирующие имена приводят к ошибкам при вызове методов или смене типа. Поиск и навигация по кодовой базе усложняются; рефакторинг становится страшным — непонятно, что за что отвечает. Технический долг накапливается: переименовать потом сложнее из-за множества ссылок и привычки команды к старым именам.

### Проверь себя (1.7)

1. Почему «data» и «result» считаются антипаттернами? Что с ними делать?
   <details><summary>Ответ</summary>
   Они не раскрывают намерение: какие данные, результат чего. Читатель вынужден смотреть тип и контекст использования. Делать: заменить на имена по намерению — что за данные (userInput, orderItems), какой результат (validatedEmails, totalAmount).
   </details>

2. В чём вред венгерской нотации в современном коде с явными типами?
   <details><summary>Ответ</summary>
   Тип уже виден из объявления или IDE; префикс дублирует информацию и при смене типа требует менять имя (strName → intName?), что создаёт лишний шум. Имя должно нести смысл и намерение, а не копировать тип.
   </details>

3. Вы нашли в коде переменную `cust` и функцию `getCustById`. Как поступить?
   <details><summary>Ответ</summary>
   Если в проекте есть глоссарий и «cust» зафиксирован как «customer» — можно оставить, но лучше полная форма в коде для читаемости. Если глоссария нет — переименовать в customer/getCustomerById и при необходимости зафиксировать термин в глоссарии. Цель — чтобы новый человек понял без расшифровки.
   </details>

4. Чем вредны «похожие имена для разных сущностей» (user и userAccount, order и orderData) без чёткого правила?
   <details><summary>Ответ</summary>
   Читатель не понимает, когда использовать какое имя; при рефакторинге легко подставить не ту переменную. Нужно явно развести по смыслу и именам: например, user — текущий пользователь сессии, userAccount — запись в БД; order — доменная сущность, orderDto — объект для передачи. Или зафиксировать правило в глоссарии/документации.
   </details>

### Запомните

- **Антипаттерны:** слишком общие имена (data, result, handler), дезинформация (List не для списка), шумовые префиксы (венгерская нотация), непонятные сокращения.
- **Исправление:** переименовать по намерению и контексту; переименовывать везде; доменные термины — в глоссарий при необходимости.
- Следовать принципам из 1.1–1.6 и конвенциям проекта.
