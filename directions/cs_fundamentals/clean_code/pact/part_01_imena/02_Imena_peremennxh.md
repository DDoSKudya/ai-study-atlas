## 1.2. Имена переменных

### Цель раздела

Научиться именовать переменные: существительные и существительные с прилагательными, булевы переменные с префиксами is/has/can/should, константы с полными именами; избегать магических чисел и общих слов без контекста; учитывать область видимости и место объявления.

### В этом разделе главное

- Переменные — **существительные** или существительные с прилагательными; **одно понятие на переменную** (не переиспользовать одну переменную для разных смыслов в разных частях кода).
- **Булевы**: префиксы `is`, `has`, `can`, `should` и т.п. для ясности.
- **Константы**: полные имена; при необходимости UPPER_SNAKE_CASE по конвенции языка.
- **Избегать**: магические числа и строки без имени; `data`, `info`, `thing`, `item` без контекста.
- Объявлять переменную **как можно ближе к первому использованию** (где язык и конвенция допускают).

### Термины

| Термин | Определение |
|--------|-------------|
| **Магическое число (магическая строка)** | Число или строка в коде без имени, смысл которого неочевиден из контекста. |
| **Константа** | Именованное неизменяемое значение; по конвенции языка может быть в UPPER_SNAKE_CASE. |

### Теория и правила

**Существительные.** Переменная хранит «что-то» — объект, значение, коллекцию. Имя — существительное: `user`, `order`, `totalAmount`. Если нужно уточнение — существительное с прилагательным или составное: `activeUser`, `pendingOrder`, `discountPercentage`. **Одно понятие на переменную:** не использовать одну переменную для двух разных смыслов в разных местах кода (например, в первой половине функции `result` — это список найденных id, во второй — флаг успеха). Разделить на две переменные с ясными именами (`foundIds`, `success`).

**Булевы переменные.** Имя должно читаться как утверждение да/нет. Префиксы:

- `is` — состояние: `isActive`, `isEmpty`, `isValid`;
- `has` — наличие: `hasPermission`, `hasChildren`;
- `can` — возможность: `canEdit`, `canSubmit`;
- `should` — должен ли: `shouldRetry`, `shouldNotify`.

Тогда условие читается естественно: `if (isValid)` или `if (user.hasPermission("admin"))`.

**Константы.** Значения, не меняющиеся в ходе выполнения, именуются полными именами. В многих языках константы — UPPER_SNAKE_CASE: `MAX_RETRY_COUNT`, `DEFAULT_TIMEOUT_SECONDS`, `API_BASE_URL`. Так они визуально отличаются от переменных и не «теряются» в коде. Имя константы должно раскрывать смысл: не просто `TIMEOUT`, а `CONNECTION_TIMEOUT_SECONDS` или `REQUEST_TIMEOUT_MS`.

**Избегать магических чисел и строк.** Вместо `if (status == 3)` — `if (status == OrderStatus.SHIPPED)` или константа `ORDER_STATUS_SHIPPED = 3`. Вместо `sleep(86400)` — `sleep(SECONDS_PER_DAY)`. Строки тоже: не `role == "A"`, а константа или enum `UserRole.ADMIN`.

**Область видимости и место объявления.** Переменную объявляют как можно ближе к первому использованию, чтобы **не растягивать время жизни переменной** и не заставлять читателя помнить её далеко выше по коду. Чем дольше переменная «живёт» в коде, тем выше риск переприсвоить её другому смыслу или забыть, где она используется. Исключения: конвенция языка (объявление в начале функции в C), цикловые переменные.

### Простыми словами

Переменная — «коробка с подписью». Подпись говорит, что в коробке: не «штука», а «сумма заказа со скидкой». Для да/нет — «это активно?», «есть ли дети?». Числа и строки с неочевидным смыслом получают своё имя (константу), чтобы не гадать, что значит 3 или "A".

### Картинка в голове

**Полки на складе:** на каждой полке подпись — что там лежит. «Коробки» и «разное» — плохие подписи. «Клиентские заказы за март», «флаг: заказ оплачен» — хорошие. Переменные в коде — такие же полки: подпись должна однозначно говорить о содержимом.

### Как запомнить

- **Переменная = существительное (с уточнением).** Булева = is/has/can/should + сущность.
- **Константа = полное имя**, часто UPPER_SNAKE; никаких голых чисел/строк с неочевидным смыслом.
- **Объявлять близко к использованию** — меньше «летучей» информации в голове.

### Примеры

**Плохо:**

```python
d = 10
flag = True
if x == 2:
    ...
```

**Лучше:**

```python
max_retries = 10
is_connection_established = True
if order.status == OrderStatus.CONFIRMED:
    ...
```

**Константы вместо магических значений:**

```python
# Плохо
timeout = 30
if role == "A":

# Лучше
DEFAULT_REQUEST_TIMEOUT_SECONDS = 30
if role == UserRole.ADMIN:
```

**Одно понятие на переменную (до и после):**

```python
# Плохо: одна переменная с разным смыслом
result = fetch_ids()      # result — список id
# ... 20 строк ...
result = len(result) > 0  # result — уже булев флаг!

# Лучше: два имени — два концепта
found_ids = fetch_ids()
has_results = len(found_ids) > 0
```

**Проверь себя (одно понятие на переменную)**

1. Как по коду понять, что переменная используется для двух разных смыслов? Что делать?
   <details><summary>Ответ</summary>
   Признаки: одна переменная переприсваивается в разных ветках или фазах разными по смыслу значениями (например, сначала список, потом булев флаг); имя при этом общее (result, data). Нужно разделить на две (или больше) переменных с ясными именами, каждую объявлять близко к месту использования и давать имя по намерению.
   </details>

2. Почему «одно понятие на переменную» упрощает отладку и рефакторинг?
   <details><summary>Ответ</summary>
   Одна переменная с одним смыслом имеет предсказуемое значение в каждой точке кода; при отладке не нужно отслеживать «в какой момент result из списка превратился во флаг». При рефакторинге переименование или перенос блока не создаёт путаницы между разными ролями одной переменной.
   </details>

### Типичные ошибки

- Булева переменная без префикса: `valid` вместо `is_valid` — в контексте `if valid` ещё читаемо, но `status` или `ready` без префикса могут быть неоднозначны.
- Константа с неясным именем: `MAX` — максимум чего? `MAX_ITEMS_PER_PAGE` яснее.
- Объявление переменной в начале функции с использованием только через 50 строк — по возможности перенести объявление ближе к месту использования.
- Одна переменная с разным смыслом в разных ветках или фазах кода — разделить на несколько переменных с разными именами.

### Что будет, если…

**Если оставить магические числа и строки:** при изменении значения (например, таймаут или код статуса) править придётся во многих местах — легко пропустить вхождение и получить неконсистентное поведение. Новый читатель не понимает смысл числа (что значит 3 или "A"?) и вынужден искать комментарий или контекст. При рефакторинге выше риск случайно изменить семантику.

**Если объявлять переменную далеко от использования:** читатель вынужден держать в голове «летучую» информацию через десятки строк; при длинной функции сложнее проследить, где переменная реально используется и где переопределяется. Сужается польза от ограничения области видимости (например, в блочном scope).

### Проверь себя (1.2)

1. Зачем для булевых переменных использовать префиксы is/has/can? Приведите пример, где без префикса читаемость страдает.
   <details><summary>Ответ</summary>
   Префикс превращает имя в утверждение да/нет и делает условие читаемым как фраза: `if is_empty` vs `if empty` (empty — что именно?). Пример: `if deleted` — удалён ли объект или это «список удалённых»? `if is_deleted` однозначно.
   </details>

2. Почему магическое число хуже, чем именованная константа, даже если число встречается один раз?
   <details><summary>Ответ</summary>
   Имя константы документирует смысл: «что это за число». При изменении значения правится одно место. При чтении кода не нужно гадать или искать комментарий. Даже одно вхождение: читатель видит `SECONDS_PER_DAY` и понимает намерение; `86400` требует знания или комментария.
   </details>

3. Что значит «объявлять переменную как можно ближе к первому использованию»? В чём выигрыш?
   <details><summary>Ответ</summary>
   Не объявлять переменную в начале функции, если она используется только в середине или в конце. Объявить её перед первым использованием. Выигрыш: меньше «летучей» информации — не нужно помнить переменную через десятки строк; область видимости сужается, код легче читать по блокам.
   </details>

4. Чем имя константы `MAX_ITEMS_PER_PAGE` лучше, чем просто `MAX`? Применимо ли это к другим константам?
   <details><summary>Ответ</summary>
   MAX не говорит, максимум чего: элементов, запросов, байт? MAX_ITEMS_PER_PAGE однозначно задаёт контекст (пагинация, лимит на страницу). К другим константам применимо так же: имя должно раскрывать не только «что за значение», но и «в каком контексте» (например, REQUEST_TIMEOUT_SECONDS, а не просто TIMEOUT).
   </details>

5. В каком случае булева переменная без префикса (например, `valid`) ещё допустима, а в каком уже нет?
   <details><summary>Ответ</summary>
   Допустима, когда контекст однозначен: например, внутри короткой функции, где `valid` — результат проверки одной сущности и сразу используется в `if valid`. Уже нет, когда имя может трактоваться по-разному (valid — «валидный ввод» или «валидация пройдена»?) или в широкой области видимости. Надёжнее всегда использовать is_valid, has_errors и т.п.
   </details>

### Запомните

- Переменные — **существительные** (с уточнением); булевы — **is/has/can/should**.
- Константы — **полные имена**, без магических чисел и строк в логике.
- Объявление переменной — **близко к первому использованию**, где это допускает язык и стиль проекта.
