## 10.5. Асинхронность и конкурентность (async/await)

### Цель раздела

Понять, как применять принципы чистого кода к современным асинхронным моделям: `async/await`, корутины, event loop; как избегать смешения стилей и как тестировать асинхронный код так, чтобы он оставался читаемым и надёжным.

### В этом разделе главное

- Асинхронный код — **тоже конкурентность**, просто в другой форме.
- Важно выбрать **единый стиль**: не смешивать callback‑ад, промисы и async/await без необходимости.
- Асинхронные функции всё так же должны быть **маленькими, с одной задачей и ясными контрактами**.
- Тестирование асинхронного кода требует **контроля времени и планировщика** (event loop, scheduler).

### Термины

| Термин         | Определение |
|----------------|-------------|
| **Async/await**| Синтаксический сахар над корутинами/промисами, упрощающий запись асинхронного кода |
| **Event loop** | Цикл, который планирует и выполняет асинхронные задачи/обработчики событий |
| **Корутина**   | Функция, выполнение которой можно приостанавливать и возобновлять позже |

### Теория и правила

1. **Асинхронность не отменяет проблему shared mutable state.**  
   Даже если у тебя один поток, но несколько корутин, разделяющих состояние, при `await` управление может передаваться другим задачам, которые изменят это состояние.

2. **Не смешивай стили без явной причины.**  
   - В одном модуле/слое: либо callbacks, либо промисы, либо async/await.
   - Смешивание увеличивает когнитивную нагрузку и риск ошибок.

3. **Асинхронный API — тоже контракт.**  
   - Ясно указывай, какие функции `async` и что они возвращают.
   - Документируй, какие исключения/ошибки могут выбрасываться асинхронно.

4. **Тестируй асинхронный код с контролем планировщика.**  
   - Используй средства фреймворка (виртуальный event loop, mock времени).
   - Избегай «магических sleep», полагаясь на события/сигналы.

### Простыми словами

`async/await` позволяет писать асинхронный код **почти как обычный линейный**, но под капотом остаётся та же конкурентность: между двумя `await` управление переходит к другим задачам, которые могут поменять мир.  
Тебе нужно думать о том же: **кто владеет состоянием**, какие инварианты должны держаться между await‑точками и как это тестировать.

### Картинка в голове

Представь официанта, который:

- принимает заказ (start корутины);
- отдаёт его на кухню и **ждёт** (await), пока еда приготовится;
- в это время обслуживает других клиентов (другие корутины);
- когда еда готова, возвращается к исходному клиенту (resumed корутина).

Официант один (один поток), но обслуживает много клиентов «параллельно» за счёт переключения задач. Если он будет **путать заказы** (общий mutable стейт без дисциплины), начнётся хаос.

### Примеры

**Асинхронный обработчик с чистой логикой**

```python
async def fetch_user_profile(client, user_id):
    resp = await client.get(f"/users/{user_id}")
    data = resp.json()
    return normalize_user_profile(data)  # чистая функция
```

- Сетевая часть — асинхронна.
- `normalize_user_profile` — обычная чистая функция, легко тестируемая без async.

**Тест асинхронной функции (псевдокод)**

```python
@pytest.mark.asyncio
async def test_fetch_user_profile_ok(mock_client):
    mock_client.get.return_value = make_response(json={"name": "Ada"})

    profile = await fetch_user_profile(mock_client, user_id=1)

    assert profile.name == "Ada"
```

### Практика / реальные сценарии

- HTTP‑сервисы на async‑фреймворках (FastAPI, aiohttp и др.).
- Чат‑боты и real‑time приложения с большим количеством одновременных подключений.
- Клиентские приложения, не блокирующие UI во время сетевых/IO операций.

### Типичные ошибки

- Смешение sync и async API без необходимости (цепочки `run_until_complete` везде).
- Забытые `await` (вызвали async‑функцию, но не дождались результата).
- Общие изменяемые структуры (словари, списки) между корутинами без дисциплины доступа.
- Использование голых `sleep` для ожидания асинхронных событий вместо явных сигналов/фьючерсов.

### Что будет, если…

- **В асинхронном коде не думать о владении состоянием?**  
  Те же гонки и неконсистентность, что и в многопоточности, только воспроизвести их может быть ещё сложнее.
- **Мешать несколько стилей асинхронности?**  
  Чтение и сопровождение кода станут сильно труднее: нужно держать в голове несколько моделей сразу, отлаживать разные механизмы ошибок и отмены.

### Проверь себя

1. Почему `async/await` не «решает» проблему гонок сам по себе?
   <details><summary>Ответ</summary>
   Потому что он лишь упрощает запись асинхронного кода, но не меняет факта: у тебя по‑прежнему есть несколько задач, которые могут менять общий стейт между await‑точками. Если эти задачи делят изменяемые данные без дисциплины, гонки и неконсистентность никуда не деваются.
   </details>

2. Зачем в тестах асинхронного кода обычно избегают «голых» `sleep`?
   <details><summary>Ответ</summary>
   `sleep` делает тесты зависимыми от скорости машины и планировщика: иногда событий ещё нет, иногда уже прошли. Лучше опираться на события (например, сигнал completion, mock времени, ожидание конкретного условия), чтобы тесты были повторяемыми и быстрыми.
   </details>

### Запомните

- Асинхронность — **форма конкурентности**, к которой применимы все принципы чистого кода.
- Держите **чистую логику отдельно от асинхронной оболочки**, тестируйте её синхронно.
- Выбирайте **один стиль асинхронности** на слой/модуль и придерживайтесь его.
