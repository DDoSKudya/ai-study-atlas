## 10.2. Принципы чистой конкурентности

### Цель раздела

Познакомиться с набором **практических принципов**, которые делают конкурентный код чище: уменьшают количество мест, где нужно думать о потоках, ограничивают общий стейт, поощряют неизменяемость и упрощают чтение и тестирование.

### В этом разделе главное

- **Минимизировать область конкуренции**: чем меньше кода «знает о потоках», тем лучше.
- **Предпочитать неизменяемые данные** и передачи сообщений общему изменяемому состоянию.
- **Явно описывать контракты потокобезопасности**: что можно вызывать параллельно, а что нет.
- **Разделять ответственность потоков/тасков**: «один поток — одна зона ответственности» где возможно.

### Термины

| Термин                   | Определение |
|--------------------------|-------------|
| **Область конкуренции** | Часть кода, где несколько потоков/тасков потенциально взаимодействуют с общим состоянием |
| **Потокобезопасность**  | Свойство объекта/функции корректно работать при одновременном доступе из нескольких потоков |
| **Actor‑модель**        | Подход, при котором состояние инкапсулировано в «акторах», общение — через сообщения |

### Теория и правила

1. **Сужай область конкуренции.**  
   Пусть 90–95% кода остаётся однопоточным по своей модели: все потоки/таски сводятся к:
   - постановке задач в очередь;
   - извлечению задач и исполнению их в однопоточном «рабочем» коде.

2. **Инкапсулируй общий стейт.**  
   Общий стейт не должен быть «просто глобальной переменной». Он должен:
   - жить внутри чётко определённого объекта/модуля;
   - управляться через небольшой, хорошо задокументированный API;
   - по возможности быть **единственной точкой записи**.

3. **Предпочитай неизменяемые структуры.**  
   Если данные не меняются после создания:
   - их безопасно кешировать и передавать между потоками;
   - можно не думать о гонках при чтении;
   - проще рассуждать о системе.

4. **Один поток — одна ответственность.**  
   Где возможно:
   - отдельный поток/таск отвечает за работу с БД;
   - другой — за приём запросов;
   - третий — за фоновую обработку и т.д.

   Это ближе к actor‑модели: каждый поток/актор «владеет» своим состоянием.

5. **Документируй контракт потокобезопасности.**  
   Для классов и функций:
   - «потокобезопасен» или «должен вызываться из одного потока (UI‑поток)»;
   - какие методы можно вызывать параллельно, какие — нет;
   - какие гарантии по видимости данных предоставляются.

### Пошагово (пример применения принципов)

Допустим, у тебя есть сервис кэша, который сейчас:

- хранит данные в обычном словаре;
- вызывается из разных потоков.

Шаги:

1. **Инкапсулируй словарь в отдельный класс `Cache`.**
2. **Определи API**: `get`, `put`, `contains`, `clear`.
3. **Ограничь контекст использования**:
   - либо явно синхронизируешь доступ внутри (`lock`/потокобезопасная коллекция);
   - либо документируешь: «вызывать только из worker‑потока; никакого параллельного доступа».
4. **Где возможно, храни значения как неизменяемые структуры**, чтобы потребители не могли их «тихо» менять.

### Простыми словами

Сделай так, чтобы **как можно меньшая часть кода «знала» про потоки**. Всю сложность доступа к общему состоянию спрячь за простым интерфейсом. Там, где это возможно, **не делись изменяемыми данными**, а делись копиями или сообщениями.

### Картинка в голове

Вместо того, чтобы всем разработчикам давать **прямой доступ к базе с правами суперпользователя**, ты создаёшь **API‑сервис**, который:

- чётко описывает, что можно и когда;
- валидирует входные данные;
- ведёт лог.

С потоками — то же самое: не давай «прямой доступ» к shared mutable state, организуй «сервис», который владеет этим состоянием и управляет им.

### Как запомнить

Три S:

- **Shrink** — сузить область конкуренции.
- **Separate** — разделить ответственность между потоками/акторами.
- **Share‑less** — меньше делиться изменяемыми данными.

### Примеры

**Плохой пример: общий список задач**

```python
tasks = []

def add_task(task):
    tasks.append(task)

def process_tasks():
    while tasks:
        task = tasks.pop(0)
        handle(task)
```

Если `add_task` и `process_tasks` вызываются из разных потоков, возможны гонки, IndexError, потеря задач.

**Лучше: очередь, инкапсулированная в worker**

```python
import queue

class TaskQueue:
    def __init__(self):
        self._q = queue.Queue()  # потокобезопасная очередь

    def add(self, task):
        self._q.put(task)

    def run_worker(self, handler):
        while True:
            task = self._q.get()
            handler(task)
            self._q.task_done()
```

Теперь:

- очередь инкапсулирована;
- есть ясный контракт: `add` можно вызывать из разных потоков, `run_worker` — из одного.

### Практика / реальные сценарии

- Сервис, принимающий HTTP‑запросы и обрабатывающий их в пуле потоков.
- Фоновый воркер, слушающий очередь сообщений и обновляющий БД.
- GUI‑приложение, где тяжёлые задачи выносятся в фоновые потоки, а результат возвращается в UI‑поток.

Во всех случаях полезно:

- изолировать тяжёлую/общую часть в отдельном модуле/потоке;
- чётко фиксировать, кто владеет каким состоянием;
- проводить линии «здесь многопоточность, здесь — обычный код».

### Типичные ошибки

- Делать «потокобезопасным» весь класс, добавляя лок вокруг каждого метода, вместо продумывания модели владения состоянием.
- Использовать глобальные локи «на всякий случай», превращая всю систему в последовательную.
- Не документировать контракт потокобезопасности, оставляя коллег в догадках.

### Что будет, если…

- **Не ограничивать область конкуренции?**  
  Код, в котором «всё может звать всё» из любых потоков, быстро становится неосмысляемым: невозможно понять, кто когда может изменять данные.
- **Не думать о неизменяемости?**  
  Простое «передали объект в другой поток» может означать, что оба потока теперь меняют одни и те же поля без синхронизации.

### Проверь себя

1. В чём практическая польза принципа «один поток — одна ответственность»?
   <details><summary>Ответ</summary>
   Он уменьшает число мест, где нужно думать о конкурентности. Если каждый поток/актор владеет своим состоянием и обрабатывает свой тип задач, рассуждать о коде легче, а риск гонок снижается: нет множества потоков, неконтролируемо меняющих одни и те же данные.
   </details>

2. Когда имеет смысл предпочесть неизменяемые структуры данных вместо мьютексов?
   <details><summary>Ответ</summary>
   Когда данные передаются между потоками больше, чем изменяются, и когда стоимость копирования приемлема. В таком случае проще создавать новые объекты при изменении, чем синхронизировать общий объект. Чтение неизменяемых структур не требует локов, и это упрощает рассуждения.
   </details>

### Запомните

- **Инкапсулируйте общий стейт** и уменьшайте количество мест, где к нему обращаются.
- **Неизменяемые данные и очереди сообщений** часто проще и надёжнее, чем мьютексы «повсюду».
- **Документированный контракт потокобезопасности** — часть чистого кода.
