## 10.4. Типичные ловушки конкурентности

### Цель раздела

Разобрать основные классы ошибок в конкурентном коде: **deadlock, гонки, livelock, starvation, shared mutable state**, понять, как они возникают, как их диагностировать и предотвращать в духе чистого кода.

### В этом разделе главное

- Deadlock, гонки и starvation — не «мистика», а следствие конкретных нарушенных правил.
- Многие ловушки можно предотвратить **простыми дисциплинами**: порядок захвата локов, избегание вложенных блокировок, чёткое владение ресурсами.
- «Shared mutable state» часто можно заменить **очередями, неизменяемостью и чётким владением состоянием**.

### Термины

| Термин           | Определение |
|------------------|-------------|
| **Deadlock**     | Ситуация, когда несколько потоков навсегда ждут освобождения ресурсов друг другом |
| **Starvation**   | Ситуация, когда один или несколько потоков не получают ресурса (CPU/лок) практически никогда |
| **Livelock**     | Потоки активны, продолжают «разруливать конфликт», но не делают полезной работы |

### Теория и правила

1. **Deadlock: четыре условия.**  
   Для deadlock обычно нужны:
   - взаимное исключение (ресурс можно держать только одному потоку);
   - удержание и ожидание (поток держит ресурс и ждёт другой);
   - отсутствие принудительного освобождения;
   - циклическое ожидание (цепочка «А ждёт Б, Б ждёт В, …, Z ждёт А»).

   Разрушив хотя бы одно условие, можно избежать deadlock.

2. **Гонки** возникают при:
   - отсутствии синхронизации при доступе к общему состоянию;
   - некорректном использовании примитивов (например, защита не всех связанных полей).

3. **Livelock и starvation** — следствие неверных стратегий планирования и обратной связи:
   - все потоки «вежливо уступают место» друг другу, но никто не входит в критическую секцию;
   - один агрессивный поток постоянно отбирает лок у других.

4. **Shared mutable state** без строгих инвариантов — питательная среда для всех этих проблем.

### Простыми словами

- **Deadlock** — как если бы два человека попытались разойтись в узком коридоре и каждый раз делали шаг в одну и ту же сторону, блокируя друг друга навсегда.
- **Livelock** — те же двое, которые всё время вежливо отпрыгивают, но в итоге никто не проходит.
- **Starvation** — третий человек, который стоит в стороне и никак не может войти в коридор, потому что другие всё время «занимают проход».

### Примеры

**Пример потенциального deadlock (псевдокод)**

```python
lock_a = Lock()
lock_b = Lock()

def task1():
    with lock_a:
        # ...
        with lock_b:
            # ...

def task2():
    with lock_b:
        # ...
        with lock_a:
            # ...
```

Если `task1` и `task2` выполняются параллельно, возможна ситуация:

- `task1` захватил `lock_a`, ждёт `lock_b`;
- `task2` захватил `lock_b`, ждёт `lock_a`.

**Простое правило предотвращения:**  
Всегда брать локи в **одном и том же порядке** во всех местах кода.

---

**Пример гонки при обновлении баланса (упрощённо)**

```python
balance = 0

def deposit(amount):
    global balance
    tmp = balance
    tmp += amount
    balance = tmp
```

Если два потока одновременно выполняют `deposit(100)` при начальном `balance = 0`, возможен результат `balance == 100` вместо `200`: оба потока прочитали 0, прибавили 100 и записали 100.

**Как исправлять:**

- вводить мьютекс/лок вокруг критической секции;
- использовать атомарные типы (где они есть в языке);
- изменить модель так, чтобы каждый поток работал со **своим локальным состоянием**, а агрегация происходила отдельно.

---

**Пример starvation в очереди задач (псевдокод)**

```python
high_priority_queue = Queue()
low_priority_queue = Queue()

def scheduler():
    while True:
        if not high_priority_queue.empty():
            run(high_priority_queue.get())
        else:
            run(low_priority_queue.get())
```

Если поток задач с высоким приоритетом **никогда не иссякает**, низкоприоритетные задачи могут **никогда не выполняться** — это starvation.

Варианты решения:

- вводить квоту («после N задач высокого приоритета обязательно выполнить одну низкую»);
- использовать справедливые планировщики/очереди, которые учитывают «возраст» задач.

### Практика / реальные сценарии

- Два сервиса, блокирующие записи в разных таблицах БД в разном порядке.
- Несколько потоков, работающих с несколькими ресурсами (файлы, сокеты, локи) без общего соглашения о порядке захвата.
- Несбалансированные семафоры/счётчики, которые «забывают» освобождать ресурс в части сценариев.
- Очереди и планировщики, где высокоприоритетные задачи полностью вытесняют низкоприоритетные.

### Типичные ошибки

- Захват нескольких локов в разном порядке в разных частях кода.
- Использование блокирующих операций без таймаутов (ожидание «навсегда»).
- Логика повторных попыток без ограничений (повторять до бесконечности).
- Отсутствие чётких инвариантов: какие данные должны изменяться только под каким локом.

### Что будет, если…

- **Не продумывать порядок захвата локов?**  
  Deadlock будет «бомбой замедленного действия», проявляясь только под нагрузкой или в редких сценариях.
- **Не ограничивать количество повторных попыток/ретраев?**  
  Можно получить livelock: система будет постоянно пытаться «исправиться», но не двигаться вперёд.

### Проверь себя

1. Как простое соглашение «порядок захвата локов всегда A → B» помогает избежать deadlock?
   <details><summary>Ответ</summary>
   Оно разрушает условие циклического ожидания: нельзя построить цикл «A ждёт B, B ждёт A», если все захватывают локи в одном и том же порядке. Поток либо возьмёт оба лока, либо не возьмёт ни одного и отступит, но не зависнет взаимно.
   </details>

2. В чём практическая разница между deadlock и livelock?
   <details><summary>Ответ</summary>
   При deadlock потоки «замерли» и больше не делают никаких действий — система как будто «замёрзла». При livelock потоки продолжают что‑то делать (например, пытаться пере‑захватить локи или отступать), но полезная работа не выполняется: система тратит ресурсы впустую.
   </details>

### Запомните

- **Deadlock, гонки, livelock, starvation** — результат **конкретных нарушенных дисциплин**, а не «случайной неудачи».
- Простые соглашения (порядок локов, ограничение ретраев, чёткий владелец состояния) резко снижают риск.
- Чистый конкурентный код делает такие соглашения **явными, кратко задокументированными и проверяемыми тестами**.
